Decision: Request Changes

Summary: The implementation demonstrates solid foundational awareness — atomic Lua scripts, heartbeat scoping, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a race condition, a silent failure path, and test coverage gaps that collectively make this unacceptable for a high-contention flash sale context where inventory correctness is a hard requirement. The issues range from a stock-read/write race that undermines the entire locking strategy to a heartbeat thread that can silently abandon a held lock without any upstream notification.

---

## Findings

### 🔴 Critical

---

**1. Stock read occurs outside the database transaction — lock does not protect against a stale read race**

```python
# services/inventory.py — lines inside _acquire_lock context

current_stock = self.db.get_stock(req.item_id)   # ← read is OUTSIDE transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write is INSIDE transaction
    self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` executes outside the database transaction boundary. If the underlying database uses `READ COMMITTED` isolation (the default for PostgreSQL, MySQL InnoDB, and most others), a concurrent writer that commits between your read and your `update_stock` write will produce a phantom stock value. The lock only helps if **every participant honors it**, but nothing prevents a non-locking path (admin tool, migration script, direct DB write, a future service that bypasses Redis) from writing stock between these two lines. The read and the write must be atomic at the database layer as well.

**Recommended fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) to hold the row lock for the duration of the transaction. The Redis lock is a useful first gate for reducing DB contention, but the DB transaction must be self-consistent independently.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Heartbeat thread silently swallows lock loss — caller proceeds with a dead lock**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread exits, but `yield token` in the context manager is still executing. The caller has no awareness that the lock is gone and continues reading and writing inventory under the false assumption of exclusive access. In a flash sale this is a direct path to overselling.

The thread needs a communication channel back to the main execution context. One approach is a shared `threading.Event` that the main path checks at a safe point, or raising into the main thread via a flag checked before the DB write. A simpler but still effective pattern is a `lock_lost` event:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling caller", lock_key)
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In reserve_item, before the DB write:
if lock_lost.is_set():
    raise LockAcquisitionError(f"Lock expired during operation for {lock_key}")
```

### 🟠 Major

---

**3. Early return inside lock context leaks the lock token from the `yield` but the `finally` block still releases — the real problem is the `ReservationResult` returns expose internal lock state indirectly via timing**

This is a subtler issue. When `reserve_item` returns early (item not found, insufficient stock), the `finally` block in `_acquire_lock` correctly releases the lock. That part is fine. However, the `ReservationResult` returned from inside the `with self._acquire_lock(lock_key):` block causes the context manager's `__exit__` to run, which calls `stop_event.set()` and then `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)`. On a hot path with high contention, a `t.join` with a 9-second ceiling timeout on every early-exit path (including "item not found") is a latency bomb. Under load, threads pile up waiting on join.

The join timeout should be much shorter — the heartbeat interval is 8 seconds, but the thread should respond to `stop_event.set()` almost immediately since `wait()` is interruptible. The `+ 1` padding is reasonable, but the base `HEARTBEAT_INTERVAL` as the join timeout means you could block for up to 9 seconds on cleanup. Consider a dedicated short join timeout (e.g., 1–2 seconds) since `stop_event.set()` will unblock `wait()` immediately.

```python
t.join(timeout=2)  # stop_event.set() unblocks wait() immediately; 2s is generous
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

**4. `update_stock` uses an absolute value write — no database-level guard against negative stock**

```python
new_stock = current_stock - req.quantity
self.db.update_stock(req.item_id, new_stock)
```

`update_stock` is called with a pre-computed absolute value. If the DB layer executes this as `UPDATE inventory SET stock = $1 WHERE item_id = $2`, there is no database-enforced guard. A race that slips through (see finding #1) or a bug in `new_stock` computation could write a negative stock value silently. The update should use a conditional decrement:

```sql
UPDATE inventory SET stock = stock - $quantity
WHERE item_id = $item_id AND stock >= $quantity
```

Check the affected row count. If it is 0, the reservation failed at the DB layer and should be treated as a conflict, not a success. This is defense-in-depth and the correct pattern for inventory systems.

### 🟡 Minor / Improvements

---

**5. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is documented as TTL/3 but is actually closer to TTL/3.75**

The comment says `# fires at TTL/3` but `30 / 8 = 3.75`. This is not a bug, but the comment is misleading. TTL/3 would be 10 seconds. At 8 seconds the heartbeat fires more frequently than documented, which is safe but the discrepancy will confuse future maintainers. Either update the comment or align the constant.

---

**6. Retry backoff ceiling is very low and jitter range is asymmetric**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45 seconds`. For a 30-second lock TTL this is fine, but the jitter range `[0, 0.05]` is only 50ms, which provides minimal thundering herd protection under high concurrency. Standard full-jitter applies `random.uniform(0, cap)` to the entire computed backoff, not just a fixed addend. This is low severity here given the retry count is only 3, but worth noting for when retry counts are tuned upward.

---

**7. No test for heartbeat lock-loss signal (finding #2)**

The test suite has no coverage for the scenario where the heartbeat detects `result == 0` (lock expired mid-operation). This is the most dangerous failure mode in the system and it is entirely untested. A test should mock `_extend_script` to return `0` and assert that the reservation fails rather than proceeding to write.

---

**8. `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args`**

```python
service._release_script.assert_called_once()
```

This assertion confirms the script was called but does not verify it was called with the correct lock key and token. A regression that passes the wrong key would not be caught. Use `assert_called_once_with(keys=[expected_key], args=[expected_token])` or at minimum inspect `call_args`.

---

**9. Daemon thread suppresses clean shutdown visibility**

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed abruptly on interpreter exit without running cleanup. In a graceful shutdown scenario (SIGTERM, gunicorn worker recycling), a daemon heartbeat thread holding an active lock extension will be killed without releasing the lock, relying entirely on TTL expiry. This is acceptable behavior given the `finally` block handles normal paths, but it should be explicitly documented. If the process is killed between `stop_event.set()` and `self._release_script(...)`, the lock will persist for up to `LOCK_TTL` seconds. Consider whether that TTL window is acceptable for your SLA.
