Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and largely correct. However, there are several layered flaws ranging from a critical TOCTOU race condition in the stock check, to a silent lock-loss scenario in the heartbeat, to a thread leak on DB exceptions, to test coverage gaps that would let these issues ship undetected. None of these are cosmetic; they represent real failure modes under the exact concurrent load this service is designed to handle.

---

## Findings

### 🔴 Critical — TOCTOU Race on Stock Check

```python
# Inside _acquire_lock context:
current_stock = self.db.get_stock(req.item_id)   # read
# ... logic ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write
```

The Redis lock serializes access at the *application layer*, but `get_stock` is executed **outside** the database transaction. If the database has any read-committed or weaker isolation, a concurrent process — one not going through this service, such as a bulk admin adjustment, a refund crediting stock back, or a different microservice — can modify `current_stock` between the read and the transactional write. The correct pattern is to move the stock read **inside** the transaction and use a pessimistic row-level lock (`SELECT ... FOR UPDATE`) or an optimistic CAS update with a version/etag:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

The Redis lock alone is not a substitute for database-level consistency guarantees.

### 🔴 Critical — Thread Leak on Database Exception

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block correctly sets `stop_event` and joins the thread. However, if `self.db.transaction().__exit__` raises (e.g., a deadlock or connection drop), the exception propagates out of `reserve_item` **without being caught**, which means it exits the `with self._acquire_lock(lock_key):` block via exception. The `finally` in `_acquire_lock` *will* run, so the thread itself is cleaned up — that part is fine. The real problem is that `_release_script` is called **after** `t.join()`, but `t.join()` has a bounded timeout of `HEARTBEAT_INTERVAL + 1` (9 seconds). If the heartbeat thread is somehow blocked (e.g., Redis latency spike), `join()` returns without confirming thread termination, and then `_release_script` fires. This is a narrow but real window where the heartbeat thread could extend the lock *after* the release script has already deleted it, allowing a subsequent holder to acquire it and then have it silently deleted by the stale heartbeat. The fix is to check `t.is_alive()` after join and log a hard warning, and to ensure the heartbeat thread checks `stop_event` before each extend attempt (it does check `stop_event.wait()` return value, but the race is between the `wait()` timeout firing and `stop_event.set()` being called in the finally block).

### 🟠 High — Silent Lock Loss in Heartbeat Is Not Propagated

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← lock was lost; silently exits
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or evicted). The heartbeat thread exits silently, but the main thread continues executing the database transaction with **no awareness that the lock is gone**. Another process can now acquire the lock and begin its own reservation for the same item. You now have two threads simultaneously inside what was supposed to be a mutually exclusive critical section.

The heartbeat needs a communication channel back to the main thread. A threading.Event or a shared flag should be set on lock loss, and the main thread must check it before committing:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            logger.error("Lock lost during operation key=%s", lock_key)
            return

# In the main thread, before committing:
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock expired during critical section")
```

### 🟠 High — `update_stock` Uses Absolute Value, Not Atomic Decrement

```python
new_stock = current_stock - req.quantity
self.db.update_stock(req.item_id, new_stock)
```

This is an absolute overwrite (`UPDATE inventory SET stock = 7 WHERE item_id = 'x'`). If any concurrent writer touches the row between the read and this write — even with the Redis lock in place, as noted in finding #1 — you silently overwrite their change. The correct form is a relative decrement with a guard:

```sql
UPDATE inventory SET stock = stock - :qty
WHERE item_id = :id AND stock >= :qty
```

Check the affected row count. If it is zero, the reservation must fail. This makes the DB operation itself idempotent and safe regardless of what the application layer does.

### 🟡 Medium — Retry Backoff Is Insufficient for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait before giving up is roughly `0.1 + 0.2 + 0.05_jitter ≈ 0.35 seconds`. Under a flash sale with thousands of concurrent requests, this will produce a thundering herd of `LockAcquisitionError` failures almost immediately. The jitter range of `0 to 0.05` is also too narrow to meaningfully spread load. Consider full jitter over a wider range (e.g., `random.uniform(0, min(cap, base * 2**attempt))`) and a higher default `retry_attempts` (5–7), or expose these as constructor parameters so callers can tune them per use case.

---

### 🟡 Medium — `LOCK_TTL` of 30 Seconds Is Dangerously Long for a Flash Sale

A 30-second lock TTL means a crashed process holds the lock for up to 30 seconds. During a flash sale, this is an eternity. The heartbeat interval of 8 seconds (TTL/3) is a reasonable ratio, but the base TTL should be much shorter — 5 to 10 seconds — with the heartbeat keeping it alive as needed. The current value suggests the DB operations are expected to be slow, which is itself a concern worth flagging.

---

### 🟡 Medium — `t.join()` Timeout Does Not Guarantee Thread Termination

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns regardless of whether the thread has actually stopped. There is no check of `t.is_alive()` afterward. If the Redis call inside `_beat` is hanging (network partition, Redis replica failover), the join times out, the release script fires, and the heartbeat thread is now a zombie that will eventually wake up and attempt to extend a lock that no longer belongs to this process. Add:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible Redis connectivity issue", lock_key)
```

---

### 🟡 Medium — No Redis Connection Error Handling

Neither `_acquire_lock` nor `reserve_item` catches `redis.RedisError` or `redis.ConnectionError`. A Redis outage during lock acquisition, heartbeat extension, or release will raise an unhandled exception that propagates to the caller with no structured `ReservationResult`. This should be caught and converted to a clean failure response, with appropriate alerting/logging.

---

### 🔵 Low — Test Suite Does Not Cover the Failure Modes Introduced Above

The tests are well-structured but test only the happy path and basic validation. Missing coverage:

| Scenario | Risk |
|---|---|
| Heartbeat detects lock loss (`result == 0`) | Silent double-reservation |
| DB transaction raises mid-flight | Verify lock is still released |
| `t.join()` timeout (thread still alive) | Zombie heartbeat |
| `redis.RedisError` during acquisition | Unhandled exception to caller |
| `update_stock` with concurrent stock modification | Oversell |

The `test_lock_release_uses_lua_script` test asserts `service._release_script.assert_called_once()`, but because `register_script` returns a single shared mock, this assertion would pass even if the extend script were called instead. The mock setup should differentiate the two registered scripts.

---

### 🔵 Low — `daemon=True` on Heartbeat Thread Masks Shutdown Behavior

Marking the thread as a daemon means it will be silently killed on interpreter exit. This is acceptable for a heartbeat, but it means that if the process is shutting down mid-reservation, the lock will not be explicitly released — it will expire after `LOCK_TTL`. This is a known tradeoff but should be documented explicitly.
