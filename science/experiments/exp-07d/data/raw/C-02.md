Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness bugs, architectural flaws, and missing failure-mode handlers that would make this dangerous to deploy under real flash-sale load. The issues range from a TOCTOU race on stock reads, to an unhandled exception path that silently holds the lock, to a heartbeat thread that can outlive its lock context.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Outside the Database Transaction

```python
# CURRENT — stock read is OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes access *across application nodes*, but it does not serialize against direct database writes (migrations, admin tools, other services). The stock read and the write are in two separate database operations with no transactional isolation between them. If the database row changes between `get_stock` and `update_stock`, you silently write a stale value.

**Required fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) so the read and write are atomic at the database level. The Redis lock is a performance optimization for cross-node coordination; it is not a substitute for database-level consistency.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...  # early return inside transaction rolls back cleanly
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Bare `except` Missing — Unhandled Exceptions Suppress Lock Release

```python
try:
    with self._acquire_lock(lock_key):
        current_stock = self.db.get_stock(req.item_id)
        ...
except LockAcquisitionError as e:
    ...
```

Only `LockAcquisitionError` is caught at the outer level. If `db.get_stock`, `db.update_stock`, or `db.create_order` raises *any* other exception (network timeout, ORM error, `AttributeError`), it propagates out of `reserve_item` as an unhandled exception. The `contextmanager` `finally` block *will* run, so the lock itself is released — but the caller receives a raw exception instead of a `ReservationResult`. This breaks the contract of the method signature and will cause the calling HTTP handler (or whatever invokes this) to return a 500 with no structured error.

**Required fix:** Add a broad exception handler that returns a structured failure result and logs the traceback.

```python
except LockAcquisitionError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 3. Heartbeat Thread Can Outlive the Lock on Join Timeout

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # = 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`t.join()` is called with a timeout. If the heartbeat thread is blocked (e.g., Redis is slow, the thread is stuck in a network call inside `_extend_script`), `join()` returns after 9 seconds and execution continues to `_release_script`. At this point the thread is still alive and still holds a reference to `lock_key` and `token`. The Lua release script deletes the key. The heartbeat thread then wakes up, calls `_extend_script`, and — because the token no longer matches — gets `result == 0` and exits. This is *mostly* safe, but there is a window where a new lock holder has acquired the key and the orphaned thread calls `expire` on it. The Lua script guards against this correctly (token mismatch returns 0), so the extend fails silently — but this is a latent correctness dependency that is not documented and will confuse future maintainers.

**Required fix:** Document this dependency explicitly, or restructure so that `_release_script` is only called after confirming `t.is_alive() == False`. If the thread is still alive after the join timeout, log a critical alert — this indicates Redis latency severe enough to threaten lock integrity.

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.critical(
            "Heartbeat thread did not terminate within timeout for key=%s. "
            "Redis latency may be compromising lock integrity.", lock_key
        )
    self._release_script(keys=[lock_key], args=[token])
```

## Significant Issues

### 4. Heartbeat Interval Is Too Close to TTL

```
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8   # comment says "TTL/3"
```

`30 / 3 = 10`, not `8`. The comment is wrong, and the actual value is more aggressive than stated. More importantly, the heartbeat fires at `t=8`, `t=16`, `t=24` — leaving only 6 seconds of margin before expiry on the third interval. Under GC pause, network jitter, or a slow Redis round-trip, this margin can evaporate. The Redlock paper and most production implementations recommend firing at `TTL / 2` or `TTL * 0.6` with the heartbeat interval set to `TTL / 3` as the *maximum* safe interval, not the actual one. Either fix the comment or increase the TTL.

---

### 5. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` constraint. A caller can pass `quantity=2_000_000_000`. This won't cause a crash, but it will result in `new_stock` going deeply negative if stock is somehow miscounted, and it opens a vector for abuse in a flash-sale context where quantities should be tightly bounded (typically 1–10 per reservation).

```python
quantity: int = Field(..., ge=1, le=10)
```

---

### 6. `item_id` Max Length Allows Redis Key Bloat

```python
item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
```

The lock key is `inventory:lock:{item_id}`. With 128-character item IDs, keys are 144 characters. This is not a security issue, but it is inconsistent with typical SKU/product ID schemas (usually 8–32 chars). If `item_id` comes from user input, 128 characters is generous enough to be suspicious. Tighten to 64 or document why 128 is required.

## Test Coverage Gaps

### 7. No Test for DB Exception Path

There is no test that verifies behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises an exception. Given Issue #2 above, this path currently propagates an unhandled exception — and there is no test catching it.

### 8. Heartbeat Thread Is Never Tested

The heartbeat renewal path (`_extend_script` being called after `HEARTBEAT_INTERVAL` seconds) has zero test coverage. The most critical failure mode — heartbeat detecting a lost lock (`result == 0`) — is untested. This should have at least one test using `threading.Event` or mocked time.

### 9. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Use `assert_called_once_with`.

### 10. No Test for Concurrent Reservation (the Core Use Case)

This is a *distributed inventory reservation* service. There is no test that spawns two threads simultaneously attempting to reserve the last unit and verifies exactly one succeeds. This is the primary correctness guarantee of the entire system and it is completely untested.

---

## Minor / Style

- The `_beat` function logs `"Lock TTL extended"` at DEBUG but logs nothing when it detects a lost lock (`result == 0`). A lost lock mid-operation is a serious event and should log at ERROR or CRITICAL.
- `HEARTBEAT_INTERVAL + 1` as a join timeout is a magic number. Extract it as a constant (`HEARTBEAT_JOIN_TIMEOUT`).
- `logger.info("Lock released key=%s")` fires even when the release script returns `0` (i.e., the lock was already expired or stolen). The log message is misleading in that case.
