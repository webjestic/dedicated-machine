Decision: Request Changes

Summary: The core distributed locking pattern is sound — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all implemented correctly. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a silent data-integrity gap (stock read outside the DB transaction) to an unhandled exception path that can leave callers with no response. The test suite also has a structural flaw that makes the lock-release assertion unreliable.

---

## Detailed Findings

### 🔴 Critical

**1. Stock read is outside the database transaction (TOCTOU within the lock)**

```python
# services/inventory.py — reserve_item()
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
        self.db.create_order(...)
```

The Redis lock serializes callers *at the application layer*, but the stock read and the stock write are in different database transactions. If the underlying DB uses any isolation level below SERIALIZABLE, a concurrent writer (a background job, a different service, a direct DB write) can modify stock between the `get_stock` call and the `update_stock` call without the Redis lock knowing about it. The fix is to move `get_stock` inside the `db.transaction()` block **and** use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) so the DB also serializes the read-modify-write:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is still valuable for reducing DB contention, but the DB must be the final authority on consistency.

---

**2. Unhandled exceptions inside the lock context propagate silently to the caller**

Any exception from `self.db.get_stock`, `self.db.transaction`, `self.db.update_stock`, or `self.db.create_order` is not caught. The `contextmanager` `finally` block *will* release the lock correctly, but the exception will bubble up through `reserve_item` with no `ReservationResult` returned. Any caller not wrapping this in a try/except gets an unhandled exception instead of a structured failure response. This is especially dangerous in an async or agentic pipeline where the caller may not expect raw exceptions from what is presented as a result-returning method.

```python
# Add inside reserve_item, wrapping the lock body:
except (DatabaseError, redis.RedisError) as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟠 High

**3. Heartbeat thread silently swallows a lost lock — no caller notification**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was stolen or expired; caller has no idea
```

When `result == 0`, the lock is gone. The business logic in `reserve_item` continues executing under the false assumption that it still holds the lock. Another process can acquire the lock and begin its own reservation while this one is still writing to the database. The heartbeat thread needs a signaling mechanism back to the main thread:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
```

The main thread should check `lock_lost_event` at a safe checkpoint (e.g., after the DB transaction) and raise/return an error if it is set.

---

**4. `quantity` has no upper bound — denial-of-service / inventory wipe vector**

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` upper bound. A malformed or malicious request with `quantity=999999999` passes validation, acquires the lock, reads stock, and — if stock is somehow that large — wipes it in one call. Even if stock is lower, it wastes a lock acquisition and a DB round-trip. Add a domain-appropriate ceiling:

```python
quantity: int = Field(..., ge=1, le=10_000)  # tune to business rules
```

### 🟡 Medium

**5. `item_id` is reflected into log messages and error strings without sanitization**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

`req.item_id` passes the regex `^[a-zA-Z0-9_\-]+$`, which does prevent injection into most contexts. However, the raw value is interpolated into user-facing message strings and log lines. For a government contracting context, log injection (newline characters are blocked by the regex, but multi-line log parsers can still be confused by crafted alphanumeric payloads) and information leakage in error messages are audit findings. Prefer opaque messages to end users and reserve the raw ID for internal logs only.

---

**6. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are module-level constants with no configuration path**

Hardcoding these as class attributes means they cannot be tuned per-deployment without a code change. For a distributed system under variable DB load, these values may need to differ between staging and production. Inject them via the constructor or a settings object:

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 8):
    self.LOCK_TTL = lock_ttl
    self.HEARTBEAT_INTERVAL = heartbeat_interval
```

### 🔵 Test Suite Issues

**7. `test_lock_release_uses_lua_script` asserts on the wrong mock object**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()  # ← this is the registered script mock
```

`service._release_script` is the return value of `redis_client.register_script(...)`, which in the fixture is a single shared `MagicMock`. Both `_release_script` and `_extend_script` point to the **same** mock object. `assert_called_once()` will fail if the heartbeat fires even once (calling `_extend_script` increments the same call count). The fixture must return distinct mocks per `register_script` call:

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.register_script.side_effect = [MagicMock(return_value=1), MagicMock(return_value=1)]
    return mock
```

**8. No test covers the lost-lock scenario (heartbeat returns 0)**

Given finding #3, there should be a test that verifies the system behaves safely when the heartbeat detects a lost lock mid-operation. This is a critical path with no coverage.

**9. No test covers DB exceptions inside the lock context**

Finding #2 has zero test coverage. Add a test where `db.update_stock` raises an exception and assert that `reserve_item` returns a structured `ReservationResult` rather than propagating the exception.
