Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several concrete failure modes ranging from a critical TOCTOU race condition in the stock check, to a thread leak on DB exceptions, to a heartbeat timing flaw that can cause premature lock expiry. These issues must be resolved before this code handles production flash-sale load.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and DB Write

This is the most serious flaw in the design. The stock check is performed **outside** the database transaction:

```python
# INSIDE the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← writes stale value
    self.db.create_order(...)
```

The Redis lock serializes callers **across processes/pods**, but it does not protect against the DB's own concurrency model. If the DB read uses `READ COMMITTED` isolation (the default for PostgreSQL and MySQL), the stock value read before the transaction can be stale by the time the write commits. More critically, `update_stock` is called with a **pre-computed absolute value** (`new_stock = current_stock - req.quantity`), not a relative decrement. This means two callers that both read `stock=10` will both write `stock=9`, silently losing one reservation.

**Required fix:** Move the read inside the transaction and use a pessimistic lock or an atomic conditional update:

```python
with self.db.transaction():
    # FOR UPDATE prevents concurrent reads of the same row within the transaction
    current_stock = self.db.get_stock_for_update(req.item_id)
    if current_stock is None or current_stock < req.quantity:
        ...  # return early, transaction rolls back automatically
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Or use an atomic conditional decrement at the DB layer:

```sql
UPDATE inventory SET stock = stock - %s
WHERE item_id = %s AND stock >= %s
RETURNING stock
```

The Redis lock alone is not sufficient as a correctness guarantee for the DB write.

### 2. Thread Leak on Database Exception

If `self.db.get_stock()`, `self.db.transaction()`, or any code inside the `with self._acquire_lock(lock_key):` block raises an **uncaught exception**, the `contextmanager` `finally` block will run correctly — **but only `LockAcquisitionError` is caught at the call site**. Any other exception (e.g., `redis.RedisError`, `psycopg2.OperationalError`, a generic `Exception`) will propagate up through `reserve_item` as an unhandled exception, bypassing the outer `try/except LockAcquisitionError`.

```python
try:
    with self._acquire_lock(lock_key):
        current_stock = self.db.get_stock(req.item_id)  # raises DBError
        ...
except LockAcquisitionError as e:
    ...
# DBError propagates to caller — no ReservationResult returned
```

The `finally` in `_acquire_lock` will still run, so the lock itself will be released. However, the caller receives a raw exception instead of a `ReservationResult`, breaking the contract of this method. Under high load this also means the heartbeat thread's `join()` will block the request thread for up to `HEARTBEAT_INTERVAL + 1` seconds (9 seconds) on every DB failure before the exception propagates.

**Required fix:**

```python
except LockAcquisitionError as e:
    logger.error(...)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except redis.RedisError as e:
    logger.error("Redis error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error.")
```

### 3. Heartbeat Timing Flaw — Lock Can Expire Before First Renewal

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # described as "TTL/3"
```

`30 / 8 = 3.75`, not `3`. The comment says "fires at TTL/3" but the math is off. More importantly, the heartbeat fires at `t=8s`, `t=16s`, `t=24s`. If the DB operation starts at `t=0` and takes longer than `30s` (e.g., a slow query, a lock wait, a network partition), the lock expires at `t=30s` and the heartbeat at `t=32s` will correctly detect `result == 0` — but by then another process has already acquired the lock and may be mid-transaction on the same item.

The heartbeat interval should be `LOCK_TTL / 3` computed, not hardcoded independently:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # 10s — fires at t=10, t=20, t=30 (before expiry)
```

Additionally, the heartbeat thread logs `result == 0` silently with just a `return`. There is no alerting or propagation of the "lock was lost" signal back to the main thread. If the lock is lost mid-transaction, the DB write will still commit, but another process may have already committed a conflicting write. This is a data integrity hole.

**Required fix:** Use a shared `threading.Event` or `threading.local` flag to signal lock loss back to the main thread, and check it before committing:

```python
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()
            logger.error("Lock lost during heartbeat key=%s — signaling main thread", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the main thread, before committing:
if lock_lost.is_set():
    raise LockAcquisitionError("Lock expired during transaction")
```

## Moderate Issues

### 4. `t.join(timeout=...)` Does Not Guarantee Thread Termination

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns after the timeout **whether or not the thread has exited**. If the heartbeat thread is blocked on a slow Redis call (e.g., Redis is under load), `join` returns, the lock is released via the Lua script, and then the heartbeat thread wakes up and calls `_extend_script` on a key that now belongs to a different owner. The Lua script protects against actually extending the wrong lock (token mismatch), but the extend call still happens and wastes a Redis round-trip. More importantly, this is a latent bug if the token UUID ever collides (astronomically unlikely but worth noting in a security-sensitive context).

**Required fix:** Check `t.is_alive()` after join and log a warning:

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate in time key=%s — possible Redis hang", lock_key)
self._release_script(keys=[lock_key], args=[token])
```

### 5. Retry Backoff Is Too Aggressive for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait before giving up is:
- Attempt 0 fails: sleep `0.1 + jitter ≈ 0.15s`
- Attempt 1 fails: sleep `0.2 + jitter ≈ 0.25s`
- Attempt 2 fails: raise

Total: ~`0.4s` before failing. Under flash-sale load with thousands of concurrent requests, a 30-second lock TTL means most callers will fail immediately. The retry window is far too short relative to the lock TTL. Either reduce the lock TTL significantly (to 1–2s for fast operations) or increase retry attempts with a cap, or implement a queue-based approach rather than spinning.

Also, `random.uniform(0, 0.05)` provides only 50ms of jitter, which is insufficient to spread load across thousands of simultaneous waiters. Consider `random.uniform(0, backoff * 0.5)` for proportional jitter.

---

### 6. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity constraint. A caller can request `quantity=1_000_000` and the only check is whether stock is sufficient. This is a business logic gap that could be exploited to drain all inventory in a single request or to cause integer overflow in downstream systems. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business rule is
```
