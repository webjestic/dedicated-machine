Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff, and Pydantic validation are all sound. However, there are several correctness, reliability, and security issues that need to be addressed before this merges, ranging from a silent data-integrity gap to thread-safety concerns and incomplete test coverage.

---

## Findings

### 🔴 Critical — Data Integrity

**1. Stock read occurs outside the database transaction (TOCTOU within the lock)**

```python
# Line ~90 — outside transaction()
current_stock = self.db.get_stock(req.item_id)

# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes stale value
    self.db.create_order(...)
```

The Redis lock serializes *this service's* callers, but it does not prevent a direct database write from another process, a migration script, an admin tool, or a future service that bypasses the lock. The read and write are not atomic at the database level. `update_stock` should use a conditional `UPDATE ... WHERE stock >= quantity` (optimistic or pessimistic DB-level lock) and the transaction should verify the affected row count. If zero rows were updated, the reservation must be rolled back and the caller should receive an "insufficient stock" response.

```python
# Preferred pattern inside the transaction:
rows_affected = self.db.decrement_stock_if_sufficient(req.item_id, req.quantity)
if rows_affected == 0:
    raise InsufficientStockError()
self.db.create_order(order_id, req.item_id, req.quantity)
```

### 🔴 Critical — Reliability

**2. Lock is released even when the heartbeat reports it was lost**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return   # ← heartbeat knows the lock is gone, but caller does not
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits silently. The `reserve_item` body continues executing under the false assumption that it still holds the lock, potentially writing to the database after another process has acquired the lock and begun its own reservation. The lock-lost condition must be surfaced to the main thread.

```python
# Suggested fix — use a shared flag
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the main flow, check before the DB write:
if lock_lost.is_set():
    raise LockAcquisitionError("Lock expired during operation")
```

### 🟠 Major — Correctness

**3. `t.join()` timeout does not guarantee the heartbeat thread has stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns regardless of whether the thread finished. If the heartbeat thread is blocked on a slow Redis call, the release script fires while the heartbeat may still be attempting an extension. This is a race condition. The code must check `t.is_alive()` after the join and handle the case where the thread is still running — at minimum log a critical error; ideally treat it as a lock-lost scenario.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical("Heartbeat thread did not terminate; lock state is uncertain key=%s", lock_key)
    # Do not release — the lock state is unknown
    return
self._release_script(keys=[lock_key], args=[token])
```

### 🟠 Major — Security / Information Disclosure

**4. Item ID reflected in error messages returned to callers**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"  # user-controlled string in response
)
```

Even though `item_id` is validated by the regex, embedding user-supplied input in API responses is a habit that leads to injection or enumeration issues as the codebase evolves. Use a static message and log the detail server-side.

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

---

### 🟡 Minor — Reliability

**5. No timeout on the Redis `get_stock` or `transaction` calls**

If the database hangs for longer than `LOCK_TTL` (30 s), the heartbeat will keep extending the lock indefinitely, starving all other callers for that item. Database calls should have explicit timeouts, and the heartbeat should have a maximum extension count or wall-clock deadline.

---

### 🟡 Minor — Test Coverage Gaps

**6. The lock-lost scenario (heartbeat `result == 0`) is not tested**

There is no test verifying that a mid-operation lock loss causes the reservation to fail and prevents the DB write. This is the most dangerous failure mode and should be the first test added once finding #2 is fixed.

**7. `test_lock_release_uses_lua_script` asserts on the wrong mock**

```python
service._release_script.assert_called_once()
```

`_release_script` is the registered script object returned by `register_script`. The test is asserting on the mock returned at fixture setup time, which will pass trivially regardless of whether the correct key/token arguments were passed. The assertion should verify `called_with(keys=[lock_key], args=[token])` using a captured token, or the test should verify that the underlying Lua logic is exercised correctly.

**8. Heartbeat thread is never exercised in tests**

All tests complete near-instantly, so the heartbeat timer never fires. A test using `unittest.mock.patch` on `HEARTBEAT_INTERVAL` (set to 0) and a slow mock DB would validate that the extension script is actually called during a long operation.
