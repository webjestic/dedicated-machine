Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several concrete failure modes ranging from a critical TOCTOU race condition in the stock check to a lock-release-after-expiry bug, an unhandled exception path that silently drops the lock, and a heartbeat thread that can outlive its usefulness without signaling the caller. These issues must be resolved before this code handles production flash sale traffic.

---

## Critical Issues

### 1. TOCTOU Race Between `get_stock` and `update_stock`

**File:** `services/inventory.py`, lines inside `reserve_item`

```python
current_stock = self.db.get_stock(req.item_id)
# ...
new_stock = current_stock - req.quantity

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but `get_stock` is executed **outside the database transaction**. If the database read is not repeatable (e.g., READ COMMITTED isolation), another process — one that bypasses this service, a migration script, a manual admin operation, or a future service that acquires the same lock on a different Redis node during a failover — can mutate stock between the read and the write. The stock decrement must be a conditional atomic operation at the database layer, not a read-modify-write split across two calls.

**Required fix:** Use a conditional update with an optimistic check:

```python
with self.db.transaction():
    # Atomic: UPDATE inventory SET stock = stock - %s
    #         WHERE item_id = %s AND stock >= %s
    rows_affected = self.db.decrement_stock_if_sufficient(
        req.item_id, req.quantity
    )
    if rows_affected == 0:
        # Rollback happens automatically on context exit without commit,
        # or raise to trigger __exit__ with exception
        raise InsufficientStockError()
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This makes the database the authoritative source of truth, not the Redis lock. The lock becomes a contention-reduction optimization, not a correctness guarantee.

### 2. Lock Released After It Has Already Expired

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The heartbeat thread logs `result == 0` and **silently returns** when the lock extension fails — it does not signal the caller. The main thread then proceeds to complete the database transaction and call `_release_script`, which will correctly return `0` (token mismatch), so it won't delete someone else's lock. That part is safe. But the real problem is: **the database transaction committed successfully while the lock was not held.** Another caller acquired the lock, read stale stock, and is now also committing. The Lua script's no-op release masks what is actually a correctness violation.

**Required fix:** The heartbeat must communicate lock loss back to the business logic so the transaction can be aborted:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling caller", lock_key)
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The `yield` site or the business logic must check `lock_lost` before committing. One clean pattern is to yield the event and let `reserve_item` check it before the transaction commits:

```python
with self._acquire_lock(lock_key) as (token, lock_lost):
    # ... stock check ...
    with self.db.transaction():
        if lock_lost.is_set():
            raise LockLostError("Lock expired before transaction commit")
        self.db.decrement_stock_if_sufficient(...)
        self.db.create_order(...)
```

### 3. Unhandled Exceptions Inside the Lock Context Silently Swallow Errors

**File:** `services/inventory.py`, `reserve_item`

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

Any exception raised by `db.get_stock`, `db.update_stock`, or `db.create_order` — network timeout, deadlock, constraint violation — propagates out of `reserve_item` entirely uncaught. The `_acquire_lock` context manager's `finally` block will still run and release the lock (correct), but the caller receives an unhandled exception rather than a `ReservationResult`. For a service returning structured results, this is a contract violation and will crash whatever is calling `reserve_item`.

**Required fix:** Wrap the lock body in a broad exception handler:

```python
except (DatabaseError, redis.RedisError) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

Do not use a bare `except Exception` — be explicit about what you're catching so programming errors surface during development.

## Significant Issues

### 4. Heartbeat Thread Join Timeout Is Too Long and Masks Hangs

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
```

If the heartbeat thread hangs (e.g., blocked on a Redis call that never returns due to a network partition), the main thread blocks for 9 seconds on every lock release. Under flash sale load this will exhaust the thread pool. The thread is `daemon=True`, so it will be killed on process exit, but during normal operation this is a latency bomb.

**Required fix:** Use a much shorter join timeout (e.g., 1–2 seconds) and log a warning if the thread did not terminate cleanly:

```python
t.join(timeout=2)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

### 5. Exponential Backoff Ceiling Is Unbounded

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff is `(2**2) * 0.1 + 0.05 = 0.45s`, which is acceptable for the default. But `retry_attempts` is a caller-controlled parameter with no upper bound. If a caller passes `retry_attempts=20`, the backoff reaches `(2**19) * 0.1 ≈ 52,428 seconds`. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

---

### 6. Redis Connection Errors During Lock Acquisition Are Not Caught

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.RedisError` and `redis.ConnectionError` are not caught here. They will propagate through `_acquire_lock` and out of `reserve_item` as unhandled exceptions. Given that this is a flash sale service — exactly the time Redis is under maximum load — this is a realistic failure path.

---

### 7. `get_stock` Result Is Trusted Without Type Validation

```python
current_stock = self.db.get_stock(req.item_id)

if current_stock is None:
    ...
if current_stock < req.quantity:
    ...
```

If `db.get_stock` returns a non-integer (e.g., a string from a misconfigured ORM, a `Decimal`, or a corrupted row), the `<` comparison will either raise a `TypeError` or produce incorrect results silently. Validate the return value:

```python
if not isinstance(current_stock, int) or current_stock < 0:
    logger.error("Invalid stock value for item_id=%s: %r", req.item_id, current_stock)
    return ReservationResult(success=False, message="Internal error.")
```

## Test Coverage Gaps

### 8. No Test for Heartbeat Lock-Loss Path

The most critical new behavior — the heartbeat detecting `result == 0` and signaling lock loss — has zero test coverage. This is the exact scenario that prevents overselling. A test must verify that when the extend script returns `0`, the reservation fails and no stock is decremented.

### 9. `test_lock_release_uses_lua_script` Is Fragile and Incomplete

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the script was called but does not verify it was called with the correct `keys` and `args`. A regression that passes the wrong token would not be caught. Assert the full call signature:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.approx_any_str()]  # or capture the token via mock
)
```
