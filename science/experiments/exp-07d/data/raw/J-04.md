Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, scoped heartbeat with stop_event, exponential backoff, and Pydantic validation are all solid choices. However, there are several correctness, safety, and architectural issues that must be addressed before this merges, particularly around exception handling gaps, a stock-read/lock-write ordering vulnerability, and test coverage blind spots that could mask real failures in production.

---

## Detailed Review

### 🔴 Critical Issues

**1. Stock read occurs inside the lock but outside the database transaction — TOCTOU window remains**

```python
# Current code
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # read outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # write inside tx
        self.db.create_order(...)
```

The Redis lock serializes callers, but `get_stock` is not inside the database transaction. If the database has any read-committed isolation or the lock is ever lost mid-operation (e.g., Redis failover, network partition), the stock read and the stock write are not atomic at the database level. A concurrent writer that bypasses Redis (direct DB access, a different service, a migration script) can corrupt inventory silently.

**Fix:** Move `get_stock` inside the `db.transaction()` block, or use a `SELECT ... FOR UPDATE` / optimistic version-check pattern at the DB layer so the database itself enforces the invariant, not just Redis.

---

**2. Unhandled exceptions inside the lock context leak the lock state silently**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `db.get_stock`, `db.update_stock`, or `db.create_order` raises any exception that is *not* `LockAcquisitionError` — a `DatabaseError`, `ConnectionError`, `TimeoutError`, etc. — the exception propagates out of `reserve_item` entirely uncaught. The `_acquire_lock` context manager's `finally` block *will* still release the lock, which is correct, but the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the method's contract and will cause 500-level errors in any HTTP handler wrapping this.

**Fix:** Add a broad exception handler inside the lock context:

```python
except (DatabaseError, Exception) as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

Be deliberate about which exceptions are caught — don't silently swallow `KeyboardInterrupt` or `SystemExit`.

---

**3. Heartbeat thread silently swallows lock loss — no signal propagates to the main thread**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # lock was lost — exits quietly
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits, but the main thread continues executing the database transaction with no awareness that the lock has been lost. Another process can now acquire the lock and begin its own reservation on the same item concurrently, defeating the entire purpose of the distributed lock.

**Fix:** Use a shared `threading.Event` (e.g., `lock_lost_event`) that the heartbeat sets on `result == 0`. The main thread should check this event at critical points, or the `_acquire_lock` context manager should raise on exit if the lock was lost mid-operation.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            return
```

Then in the finally block or yielded context, check `lock_lost_event` and raise appropriately.

### 🟡 Moderate Issues

**4. `retry_attempts` default of 3 with aggressive backoff may be too short for flash sale load**

With `retry_attempts=3` and backoff of `0.1 + 0.05 jitter`, `0.2 + jitter`, `0.4 + jitter`, total max wait is under a second. Under genuine flash sale concurrency (thousands of requests), the vast majority of callers will receive "Service temporarily unavailable" rather than queuing. This may be intentional for a flash sale (fail-fast), but it should be a configurable parameter, not a hardcoded default, and it should be documented as a deliberate policy decision.

---

**5. `LOCK_TTL = 30` seconds is very long for an inventory reservation**

A 30-second lock TTL means a crashed worker holds the lock for up to 30 seconds. For a flash sale, this is an eternity. The heartbeat interval of 8 seconds is reasonable, but the base TTL should be tuned to the expected p99 DB operation latency — likely 1–3 seconds — with the heartbeat keeping it alive only for legitimately slow operations. Consider making both values configurable via constructor parameters.

---

**6. `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` may not be sufficient**

If the heartbeat thread is blocked on a slow Redis call to `_extend_script` when `stop_event.set()` is called, the join timeout of 9 seconds could cause the `finally` block to proceed to lock release before the heartbeat thread has finished. The heartbeat could then extend the lock *after* the Lua release script has deleted it, allowing a new holder to acquire the lock and then have it silently extended by the zombie heartbeat. This is a race condition.

**Fix:** Ensure the heartbeat thread checks `stop_event` before executing the extend script, not just at the `wait()` call:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        if stop_event.is_set():  # double-check before network call
            return
        result = self._extend_script(...)
```

### 🟠 Test Coverage Gaps

**7. No test for unhandled DB exceptions inside the lock**

There is no test that verifies behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises. Given issue #2 above, this is a critical missing case.

---

**8. `test_lock_release_uses_lua_script` is a weak assertion**

```python
service._release_script.assert_called_once()
```

This only asserts the script was called once, not that it was called with the correct `keys` and `args`. A future refactor could call the script with wrong arguments and this test would still pass.

**Fix:**
```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, match any string
)
```

---

**9. No test for heartbeat lock-loss propagation (once issue #3 is fixed)**

Once the lock-lost signal is implemented, there must be a test that verifies a reservation fails cleanly when the heartbeat detects the lock was lost mid-operation.

---

### 🟢 Positive Notes

- Lua scripts for atomic release and extension are correct and well-commented.
- `stop_event.wait(timeout=...)` as a timer is the right pattern — avoids busy-waiting.
- Pydantic validation with regex pattern on `item_id` is appropriate for this context.
- `uuid4` for both token and order ID is correct.
- Daemon thread prevents process hang on unclean shutdown.
- Structured logging with consistent field names is good practice.
- The `finally` block ordering (set stop_event → join thread → release lock) is correct in intent.

---

## Required Before Approval

1. Move `get_stock` inside the database transaction or add DB-level locking
2. Add exception handling for non-`LockAcquisitionError` exceptions inside the lock context
3. Propagate lock-loss signal from heartbeat thread to main thread
4. Add tests for DB exception paths and heartbeat lock-loss
5. Strengthen `test_lock_release_uses_lua_script` argument assertions
