Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent failure mode that could cause inventory corruption under real load, and test coverage gaps that would not catch these issues in production. These must be resolved before this touches any flash sale workload.

---

## Critical Issues

### 1. Stock Read Occurs Outside the Database Transaction — TOCTOU Race

This is the most serious flaw. The `get_stock` call happens *before* `self.db.transaction()` is entered. The Redis lock serializes Python-level callers on this single service instance, but it provides **zero protection** against:

- A second service replica holding its own lock token (this is not Redlock — it's a single-node Redis lock)
- A database-level write from any other process, migration script, admin tool, or replica lag scenario

```python
# CURRENT — stock read is outside the transaction boundary
current_stock = self.db.get_stock(req.item_id)          # read here
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)         # write here
    self.db.create_order(...)
```

The stock value read on line N is stale by the time the transaction commits. The correct pattern is a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) *inside* the transaction, or a conditional `UPDATE` with an affected-rows check:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # row-locked read
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Without this, two concurrent reservations on separate replicas can both read `stock=1`, both pass the availability check, and both commit — resulting in `stock=-1` and an oversold item.

### 2. Single-Node Redis Lock Is Not a Distributed Lock

The PR title and summary claim "distributed" locking, but `redis.set(..., nx=True, ex=...)` against a single Redis instance is a **single point of failure and split-brain vector**, not a distributed lock. If Redis restarts, fails over to a replica (which may not have flushed the key), or becomes network-partitioned:

- The lock key may vanish, allowing two holders simultaneously
- A replica promoted to primary after a crash may not have the key, so a new acquisition succeeds while the original holder believes it still holds the lock

For a government-grade flash sale system with inventory integrity requirements, this needs either:
- **Redlock** across ≥3 independent Redis nodes (with full awareness of its known limitations under clock drift)
- A CP consensus store (etcd, ZooKeeper) for the lock primitive
- Or — most pragmatically — drop the Redis lock entirely and rely solely on the database row lock (`SELECT FOR UPDATE`) plus a unique constraint on the order, which is already ACID

The current architecture gives the *appearance* of distributed safety without the guarantees.

### 3. Heartbeat Thread Silently Swallows Lock Loss — No Caller Notification

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # <-- lock was lost; thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread exits, but the `reserve_item` business logic continues executing under the assumption that it still holds the lock. There is no mechanism to interrupt the caller. This means:

- The DB transaction may commit *after* another process has acquired the lock and begun its own read
- You now have two concurrent writers with no mutual exclusion

The fix requires a shared flag or event between the heartbeat and the main thread, with the main thread checking it before committing:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()  # signal the main thread
            return
        ...

# In reserve_item, before committing:
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

## Moderate Issues

### 4. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` Is Fragile Under GIL Contention

The comment says "fires at TTL/3" (30/3 = 10), but 8 ≠ 10 — minor, but the comment is wrong. More importantly, Python's GIL means the heartbeat thread may not be scheduled within the 8-second window under heavy CPU load. With a 30-second TTL and an 8-second heartbeat, there is a 22-second window in which a GIL-starved thread could miss its renewal. Consider:

- Reducing TTL or increasing heartbeat frequency ratio
- Documenting the GIL caveat explicitly
- Using a separate process-level heartbeat if this is truly latency-sensitive

### 5. `t.join(timeout=HEARTBEAT_INTERVAL + 1)` Does Not Verify Join Completion

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns `None` whether the thread exited or timed out. If the heartbeat thread is hung (e.g., blocked on a Redis call), the release script runs while the heartbeat thread is still alive and may subsequently call `_extend_script` on a key that has already been released and potentially re-acquired by another holder. The thread is `daemon=True` so it won't block process exit, but within a long-lived service process it's a latent hazard.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not proceed to release — or accept the risk explicitly
```

### 6. Exponential Backoff Ceiling Is Missing

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, max backoff is `(2**2) * 0.1 = 0.4s` — acceptable here, but the formula has no cap. If a caller ever passes `retry_attempts=20`, backoff reaches `~104 seconds` on the last attempt. Add `min(..., MAX_BACKOFF)` or document the parameter constraint.

## Test Coverage Gaps

### 7. The Most Critical Path Is Not Tested

There is no test for the TOCTOU scenario described in issue #1 — i.e., stock changing between the read and the transaction commit. This is the exact failure mode that will cause overselling. A test should mock `get_stock` to return different values on sequential calls and assert that the second reservation fails.

### 8. Heartbeat Lock-Loss Path Is Not Tested

The `result == 0` branch in `_beat` (lock lost during extension) has no test. Given that this is the silent-failure path identified in issue #3, it needs explicit coverage.

### 9. `test_lock_release_uses_lua_script` Is Structurally Weak

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Assert the full call signature:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, match by presence
)
```

### 10. No Test for Database Transaction Rollback Behavior

If `create_order` raises, does `update_stock` roll back? The test suite does not verify this. The mock context manager returns `False` from `__exit__` (suppresses no exceptions), but there's no test that injects an exception mid-transaction and asserts stock was not decremented.

## Minor / Style

- `HEARTBEAT_INTERVAL = 8` comment says "TTL/3" but 30/3 = 10, not 8. Fix the comment or the value.
- `reserve_item` logs at `INFO` on success but the lock release also logs at `INFO` — in high-throughput flash sales this will be extremely noisy. Consider `DEBUG` for the release log or make log levels configurable.
- The `quantity` field default in `reserve_item(self, item_id, quantity=1)` is fine, but the Pydantic model has no default — the two interfaces are slightly inconsistent in how they communicate the default. Document or unify.
- `order_id` is generated before the transaction opens. If the transaction is retried externally, a new `order_id` should be generated per attempt to avoid duplicate key collisions on the orders table.

---

## What Would Make This Approvable

1. Move `get_stock` inside the transaction with a row-locking read (`SELECT FOR UPDATE` or equivalent)
2. Add the `lock_lost_event` signaling mechanism so heartbeat loss aborts the commit path
3. Add `t.is_alive()` check after `join`
4. Either document the single-node Redis limitation explicitly with a known-risk acceptance, or upgrade to a proper distributed primitive
5. Add tests for lock-loss mid-operation, transaction rollback on error, and full argument assertion on the release script call
