Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, scoped heartbeat with stop_event, exponential backoff, and Pydantic validation are all solid choices. However, there are several correctness, safety, and architectural issues that must be addressed before this merges, ranging from a silent exception-swallowing bug to a race condition in the heartbeat and missing test coverage for critical failure paths.

---

## Issues

### 🔴 Critical

**1. Database exceptions silently escape the lock context and leave stock inconsistent**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception raised by `db.get_stock`, `db.update_stock`, or `db.create_order` propagates out of `reserve_item` as an unhandled exception — not a `ReservationResult`. The caller receives a raw exception instead of a structured response. The lock *will* still be released (the `finally` block handles that), but the caller contract is broken. Wrap the entire lock body in a broad `except Exception` and return a `ReservationResult(success=False, ...)` with the error logged. Do not let database errors surface as unhandled exceptions from a service method that promises to return a `ReservationResult`.

---

**2. Heartbeat thread can extend a lock after the `finally` block has already released it**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        ...
```

`stop_event.set()` and `t.join()` are sequential, but there is a window: `stop_event.wait()` times out and returns `False`, the thread enters `_extend_script`, and *concurrently* the `finally` block calls `stop_event.set()` then immediately calls `_release_script`. The extend and release can now race. The Lua token-check makes the extend a no-op after release, which is safe *in this case*, but only because the release script deletes the key first. If the extend wins the race, it re-sets the TTL on a key that is about to be deleted — harmless but indicative of a design gap. The safer fix is to check `stop_event.is_set()` at the top of the loop body before calling `_extend_script`, and to set `stop_event` *before* `t.join()` (which you already do), but also document this explicitly. Consider using a `threading.Lock` to serialize the extend and release operations.

### 🟠 High

**3. `get_stock` is called outside the database transaction**

```python
current_stock = self.db.get_stock(req.item_id)  # outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The Redis lock serializes *this service's* access, but `get_stock` is not inside the DB transaction. If the database has read-committed isolation (the default for PostgreSQL and MySQL), a concurrent writer outside this service — a manual admin correction, a returns processor, another service — can modify stock between the read and the write. The stock read and write must be inside the same transaction, ideally using a `SELECT ... FOR UPDATE` or equivalent to make the DB itself the final authority on stock levels, with the Redis lock as a performance optimization layer only.

---

**4. `HEARTBEAT_INTERVAL` is too close to `LOCK_TTL` for safety**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

The comment says "fires at TTL/3" which is correct (30/3 = 10, but the value is 8 — the comment is slightly wrong, it fires at TTL/3.75). More importantly, `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` means the `finally` block can block for up to 9 seconds waiting for the heartbeat thread. During that 9-second window, the lock is still held and the heartbeat may still be extending it. This is a latency hazard under load. The join timeout should be much shorter — a few hundred milliseconds — and if the thread does not exit within that window, log a warning and proceed with release anyway. The Lua release script is safe regardless.

---

**5. No test for database exception path**

The test suite has no case where `db.update_stock` or `db.create_order` raises an exception. Given issue #1 above, this is the most dangerous untested path. Add:

```python
def test_reserve_db_failure_returns_result_not_exception(service, db):
    db.update_stock.side_effect = Exception("DB connection lost")
    result = service.reserve_item("item_001", quantity=1)
    assert result.success is False
    assert result.order_id is None
```

### 🟡 Medium

**6. `retry_attempts` default is not configurable at the service level**

The retry count is hardcoded as a default argument to `_acquire_lock`. For a government-facing flash sale system, retry behavior should be configurable at construction time or via a settings object, not buried in a private method signature. This also makes it harder to tune per-environment without code changes.

---

**7. `remaining_stock` is leaked in the insufficient-stock response**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,
    message="Insufficient stock"
)
```

Returning the exact current stock count to the caller is an information disclosure issue. An adversary probing the endpoint can enumerate real-time inventory levels by submitting requests with varying quantities. Return only a boolean success and a generic message. Internal stock levels are operational data, not client-facing data.

---

**8. Lock key namespace collision risk**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared with other services (common in cost-constrained government deployments), a key collision with another service using the same namespace is possible. Prefix with a service-specific namespace, e.g., `obotix:inventory:lock:{req.item_id}`, and document the Redis keyspace policy.

---

### 🔵 Minor / Nits

**9. `test_lock_release_uses_lua_script` assertion is fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. The token and key should be verified. Use `assert_called_once_with(keys=[...], args=[...])` — though since the token is a UUID generated internally, you may need to capture it via a mock side effect. At minimum, assert `keys` contains the expected lock key.

---

**10. Daemon thread suppresses heartbeat errors silently**

```python
t = threading.Thread(target=_beat, daemon=True)
```

If `_extend_script` raises a Redis connection error, `_beat` exits silently. The lock will expire naturally, which may be acceptable, but the failure should be logged at `ERROR` level so operators know the heartbeat died. Wrap the extend call in a `try/except` inside `_beat`.

---

## What's Good

- Lua scripts for atomic release and extension are correct and well-scoped.
- `stop_event.wait(timeout=...)` as a timer (rather than `time.sleep`) is the right pattern — it allows clean shutdown.
- Pydantic validation with regex on `item_id` correctly blocks path traversal attempts.
- Exponential backoff with jitter is appropriate for lock contention.
- `nx=True, ex=self.LOCK_TTL` on the SET is the correct atomic acquisition pattern.
- Daemon thread prevents the heartbeat from blocking process shutdown.
- Test coverage for validation, not-found, and insufficient-stock cases is solid.
