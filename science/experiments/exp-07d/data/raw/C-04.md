Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and correctly motivated. However, there are several correctness flaws, architectural gaps, and missing failure-mode handlers that are unacceptable for a high-concurrency flash sale reservation system in a production government contracting context. The issues range from a TOCTOU race condition in the stock check, to unhandled exceptions that can silently corrupt inventory state, to thread lifecycle problems that will cause resource leaks under load.

---

## Detailed Findings

### 🔴 Critical

---

**1. TOCTOU Race: Stock Read Is Outside the Database Transaction**

```python
# Lock is held, but the read is OUTSIDE the transaction block
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    return ReservationResult(...)

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← writes stale value
    self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` executes outside the database transaction. If the database has any read-committed or snapshot isolation behavior, or if a non-locking path (admin tool, migration, another service) modifies stock between the read and the `update_stock` write, you will silently oversell. The read, the guard check, and the write **must** all occur inside a single atomic database transaction with a `SELECT FOR UPDATE` (or equivalent pessimistic lock) on the inventory row. The Redis lock alone is not sufficient — it only serializes threads that go through *this* code path.

**Required fix:**

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ...
    if current_stock < req.quantity:
        return ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Unhandled Exceptions Inside the Lock Context Will Silently Swallow Errors and Leak Locks**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ... no except for db errors
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `get_stock`, `update_stock`, `create_order`, or the transaction commit raises any exception (network timeout, DB connection drop, deadlock), it propagates out of `reserve_item` entirely uncaught. The `_acquire_lock` context manager's `finally` block will release the lock correctly, but the caller receives an unhandled exception rather than a `ReservationResult`. For a service API this means the caller gets a 500 with no structured error, and depending on transaction rollback behavior, you may have a partial write. All database operations inside the lock must be wrapped in a `try/except` that returns a structured `ReservationResult(success=False, ...)` and logs the exception with a traceback.

---

**3. Heartbeat Thread Has No Alerting Path When Lock Is Lost Mid-Operation**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← silent exit
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread silently returns. The main thread has no awareness of this and continues executing the database transaction under the assumption that it holds the lock — which it does not. Another process may have acquired the lock and begun its own stock read by the time the first thread commits. This is the exact scenario the lock is meant to prevent.

**Required fix:** Use a shared `threading.Event` or a flag to signal lock loss back to the main thread. The main thread should check this flag at a safe point (e.g., before committing the transaction) and abort if the lock was lost.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during operation key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In main thread, before committing:
if lock_lost_event.is_set():
    raise LockLostError(f"Lock expired during transaction for {lock_key}")
```

**4. `t.join(timeout=...)` Does Not Guarantee Thread Termination Before Lock Release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # ← timeout may expire
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns after N seconds regardless of whether the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call (e.g., network partition), `join` returns with the thread still alive. The lock is then released by the main thread. The heartbeat thread may subsequently wake up and call `_extend_script` — which will now fail silently (result == 0, correct), but the thread is now a zombie that will not terminate until the daemon exits. Under sustained load with lock contention, this will accumulate zombie heartbeat threads. After `join`, you must check `t.is_alive()` and log an error if the thread did not terminate.

---

**5. Backoff Values Are Too Aggressive for Flash Sale Load**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum wait is `(2^2 * 0.1) + 0.05 = 0.45s`. With a `LOCK_TTL` of 30 seconds and a heartbeat keeping the lock alive, a caller that arrives while a slow DB operation is in progress will exhaust all retries in under half a second and return "service unavailable" to the user. For a flash sale with a 30-second lock TTL, either the retry count needs to be dramatically higher, the backoff ceiling needs to be much larger, or — better — the lock TTL should be reduced to match the expected operation duration (sub-second for a well-indexed DB write). The current combination of a 30-second TTL and 3 retries over 0.45 seconds is incoherent.

---

**6. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity validation. A caller can submit `quantity=1_000_000` and the only guard is the stock check. For a flash sale system this is an obvious abuse vector — a single bad actor or misconfigured client can attempt to reserve the entire catalog in one request. Add a `le=` constraint appropriate to business rules (e.g., `le=10` for flash sales).

### 🟡 Minor / Improvements

---

**7. `item_id` Leaks Into Log Output Without Sanitization**

```python
logger.warning("Lock contention key=%s attempt=%d ...", lock_key, ...)
logger.info("Reserved item_id=%s ...", req.item_id, ...)
```

`item_id` passes Pydantic validation (alphanumeric + `_-`), so injection into log lines is low risk here. However, `lock_key` is constructed via f-string before validation in `_acquire_lock`. The lock key is built *after* validation in `reserve_item`, so this is fine in the current call path — but `_acquire_lock` accepts any arbitrary `lock_key` string from any future caller. Document this assumption or assert the format inside `_acquire_lock`.

---

**8. Test Coverage Does Not Exercise the Heartbeat or Lock-Loss Path**

`test_lock_release_uses_lua_script` verifies the Lua release is called, but no test exercises:
- The heartbeat thread detecting `result == 0` and setting `lock_lost_event`
- The main thread aborting when lock is lost mid-transaction
- The `t.join` timeout path (thread still alive after join)
- DB exceptions inside the lock context

These are the highest-risk paths in the entire implementation and they are completely untested.

---

**9. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants, Not Configurable**

Hardcoding these as class attributes makes it impossible to tune them per-deployment or per-item-category without subclassing. They should be constructor parameters with sensible defaults, allowing injection from config or environment variables.
