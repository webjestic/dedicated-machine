Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, architectural weaknesses, and test gaps that must be addressed before this is safe to merge into a high-concurrency flash sale context. The most critical issues are a TOCTOU race between the stock read and the DB write, and a lock release that can execute even when the lock was never held or was silently lost.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Outside the Database Transaction

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The Redis lock serializes callers *through this service*, but it provides zero protection against:

- Direct database writes from other services, admin tools, or batch jobs that bypass this lock entirely.
- A DB failover or read replica lag causing `get_stock` to return a stale value.
- Any future service that touches inventory without acquiring this exact lock key.

**Required fix:** Move `get_stock` inside the transaction and use a pessimistic row-level lock (e.g., `SELECT ... FOR UPDATE` in SQL). The Redis lock is a performance optimization for serializing *this service's* callers — it must not be the sole correctness guarantee.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # rollback and return early
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Lock Release Executes Unconditionally on Acquisition Failure Path

The `finally` block in `_acquire_lock` always calls `_release_script`, even when `acquired` is `False`. If the lock was never acquired, the token is a freshly generated UUID that was never written to Redis — the Lua script will correctly return `0` in that case, so no *data corruption* occurs. However, this is a latent correctness assumption that is not enforced structurally. If the control flow ever changes (e.g., partial acquisition in a retry scenario), this becomes a silent bug.

**Required fix:** Gate the release on `acquired`:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if acquired:
        self._release_script(keys=[lock_key], args=[token])
```

But more importantly: the `finally` block runs even when `LockAcquisitionError` is raised *before* `yield`. Currently the thread `t` is started only after a successful acquisition, so the `t.join()` in `finally` would block on a thread that was never started. This will raise `RuntimeError` or hang depending on Python version.

**Trace the failure path:**

```python
if not acquired:
    raise LockAcquisitionError(...)  # jumps out

# t is never created
t = threading.Thread(...)
t.start()

try:
    yield token
finally:
    stop_event.set()
    t.join(...)           # NameError or RuntimeError — t is not defined in this scope
    self._release_script(...)
```

This is a real bug. The `finally` block references `t` and `stop_event` which are only defined in the success path. **This will crash the process on lock contention.**

**Required fix:** Restructure so the thread is only started and joined when the lock is held, or initialize `t = None` / `stop_event = None` before the retry loop and guard the join.

### 3. Heartbeat Thread Silently Swallows Lock Loss

```python
if result == 0:
    return  # lock was lost — thread exits silently
```

When the heartbeat detects the lock has been lost (token mismatch or key expired), it exits silently. The main thread has no way to know this happened and will proceed to execute the DB transaction *without holding the lock*. This is the exact failure mode the heartbeat is supposed to prevent.

**Required fix:** Use a shared flag or event to signal lock loss back to the main thread, and check it before executing the critical section:

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then in `reserve_item`, check `lock_lost_event` before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock was lost during operation")
```

## Significant Issues

### 4. Retry Backoff Is Too Aggressive for Flash Sale Workloads

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum wait is `0.4s + 0.05s = 0.45s` total across all retries. Under a flash sale with thousands of concurrent requests, this will result in a thundering herd on every retry wave. The jitter window (`0.05s`) is too narrow relative to the base backoff to provide meaningful spread.

Additionally, `retry_attempts=3` is hardcoded as a default but not configurable at the service level — callers cannot tune this per-item or per-sale-tier.

**Recommendation:** Widen jitter to `random.uniform(0, backoff)` (full jitter pattern), make `retry_attempts` a constructor parameter, and consider a configurable max backoff cap.

---

### 5. No Exception Handling for Redis or DB Failures Inside the Lock

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        ...
```

If `get_stock`, `update_stock`, `create_order`, or any Redis call raises an unexpected exception, it propagates out of `reserve_item` as an unhandled exception. The caller gets a 500 rather than a `ReservationResult`. For a public-facing flash sale endpoint, this is unacceptable.

**Required fix:** Wrap the lock body in a broad exception handler and return a safe `ReservationResult` with a generic error message. Log the full traceback internally.

---

### 6. `LOCK_TTL = 30` Is Dangerously Long for a Flash Sale

A 30-second lock TTL means a crashed worker holds the lock for up to 30 seconds, blocking all other reservations for that item. In a flash sale, 30 seconds of unavailability per item is catastrophic.

The heartbeat interval of 8 seconds (TTL/3) is a reasonable ratio, but the absolute TTL should be tuned to the expected operation duration, not set to a round number. If the DB transaction takes 200ms in the normal case, a 5-second TTL with a 1.5-second heartbeat is far more appropriate.

**Required fix:** Make `LOCK_TTL` and `HEARTBEAT_INTERVAL` constructor parameters with documented defaults. Add a comment explaining the expected operation duration that informed the defaults.

## Test Coverage Gaps

### 7. No Test for the Lock-Loss Signal Path

There is no test verifying that if the heartbeat detects lock loss (`result == 0`), the reservation fails rather than proceeding. This is the most critical behavioral guarantee of the heartbeat system and it is completely untested.

### 8. No Test for DB Transaction Rollback on Failure

There is no test verifying that if `create_order` raises an exception, `update_stock` is rolled back. The mock transaction context manager is set up but never tested in a failure scenario.

### 9. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test.

**Required fix:**
```python
service._release_script.assert_called_once_with(
    keys=[expected_lock_key], args=[expected_token]
)
```

This requires exposing the token from the lock context, which is itself a design smell — consider whether the lock abstraction needs a testable interface.

### 10. Heartbeat Thread Timing Is Not Tested

All tests run synchronously against mocks. There is no test that exercises the heartbeat thread's renewal behavior under simulated time pressure. At minimum, a test should verify that the extend script is called when the heartbeat fires, and that `stop_event.set()` terminates the thread.

## Minor Issues

### 11. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. This will pass validation, acquire the lock, and attempt to decrement stock by 10 million. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=100)  # tune to business rules
```

### 12. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared with other services (common in cost-constrained environments), a key collision with another service's namespace is possible. Prefix with a service-specific namespace or use a dedicated Redis logical database.
