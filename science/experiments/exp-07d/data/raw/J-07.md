Decision: Request Changes

Summary: The core distributed locking logic is well-structured — the Lua-atomic release/extend scripts, scoped heartbeat with `stop_event`, exponential backoff with jitter, and Pydantic input validation are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges into a government-facing production system. The test suite also has meaningful coverage gaps that leave critical failure paths unverified.

---

## Issues

### 🔴 Critical

**1. TOCTOU race between stock read and DB transaction**

```python
# Lock is held, but stock is read OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)   # dirty read possible
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # assumes stock hasn't changed
    self.db.create_order(...)
```

The Redis lock serializes callers *through this service*, but it does not protect against:
- Direct DB writes from other services or admin tooling
- Replica lag if `get_stock` hits a read replica
- The DB transaction not using `SELECT ... FOR UPDATE`

The stock read must occur *inside* the transaction with a row-level lock. The current pattern creates a window where stock can go negative.

```python
# Preferred pattern:
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...  # early return or rollback
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

---

**2. Heartbeat thread silently swallows lock loss — caller is never notified**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; caller continues executing with no lock
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits quietly. The `yield token` in the context manager continues executing — the caller believes it holds the lock, but another process may have already acquired it and begun modifying inventory. This is a correctness failure that can cause double-reservation.

The heartbeat must signal lock loss back to the calling context. One approach:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()
            return

# After yield, in finally or via a wrapper:
if lock_lost.is_set():
    raise LockAcquisitionError("Lock expired during operation")
```

This still requires the business logic to check the flag at a safe point, which is non-trivial. The architecture should be documented with an explicit decision here.

### 🟠 High

**3. Unhandled exceptions inside the lock context leak the lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # what if this raises?
    ...
    with self.db.transaction():
        self.db.update_stock(...)   # what if this raises?
```

If `get_stock`, `update_stock`, or `create_order` raise an unexpected exception, it propagates through the `with` block. The `finally` in `_acquire_lock` will release the lock correctly — that part is fine. But `reserve_item` has no outer `except Exception` handler, so the exception will bubble to the caller completely unhandled. For a service layer this should return a structured `ReservationResult`, not raise.

```python
except Exception as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

---

**4. `quantity` is not validated to have a reasonable upper bound**

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` upper bound. A caller can pass `quantity=10_000_000`. Even if stock checks prevent the reservation, the value is logged and passed to DB calls. Define a domain-appropriate maximum (e.g., `le=1000`) and enforce it at the model layer.

---

**5. `remaining_stock` is exposed on lock-failure path**

```python
if current_stock < req.quantity:
    return ReservationResult(
        success=False,
        remaining_stock=current_stock,   # ← leaks exact inventory count
        message="Insufficient stock"
    )
```

Returning the exact remaining stock count on failure leaks inventory intelligence to callers. In a flash-sale context, this can be exploited to time attacks or infer competitor stock levels. Return a boolean or a bucketed value (`"low"`, `"out of stock"`) depending on business requirements. At minimum, this should be a documented and deliberate product decision — it should not be the default.

---

### 🟡 Medium

**6. Heartbeat thread is daemon but `t.join()` timeout is not checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within the join timeout (e.g., it is blocked on a Redis call that is hanging), `join()` returns silently and the lock release proceeds anyway. This is probably acceptable behavior, but it should be logged:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
```

---

**7. `LOCK_TTL` of 30 seconds is very long for a flash sale**

A 30-second lock TTL means a crashed worker holds the lock for up to 30 seconds, blocking all other reservations for that item during peak load. For a flash sale this is likely unacceptable. The heartbeat interval of 8 seconds is appropriate for a shorter TTL (e.g., 10–15 seconds). Revisit this value with the product team and document the rationale.

---

**8. Redis connection errors are not caught**

If `self.redis.set(...)` raises `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`, the exception propagates uncaught from `reserve_item`. These should be caught and returned as a structured failure, or at minimum documented as the caller's responsibility.

---

### 🔵 Test Coverage Gaps

**9. Lock loss during operation is not tested**

The scenario where the heartbeat detects `result == 0` (lock stolen/expired mid-operation) has no test. This is the most dangerous failure mode and must be covered.

**10. DB transaction failure / rollback is not tested**

There is no test for `db.update_stock` or `db.create_order` raising an exception. The test suite should verify that a DB failure does not result in a partial reservation being returned as successful.

**11. Heartbeat extension is not tested**

`test_lock_release_uses_lua_script` verifies the release script is called, but there is no test that the extend script is called when an operation runs longer than `HEARTBEAT_INTERVAL`. This is the primary correctness guarantee of the heartbeat design.

**12. `redis_client.set.return_value = False` does not simulate retry behavior**

`test_reserve_lock_contention` sets `set.return_value = False` for all calls, which exercises the exhausted-retry path. There is no test for the partial-retry success path (fails twice, succeeds on third attempt), which exercises the backoff logic.

---

## Minor Notes

- The `token` yielded by `_acquire_lock` is not used by the caller. Consider yielding nothing or yielding a lock handle object if future callers need it.
- Log lines in `reserve_item` log `req.item_id` after validation, which is correct. Ensure downstream log aggregators treat these as structured fields, not interpolated strings, to prevent log injection.
- Consider naming the module `services/inventory_service.py` to match the class name and avoid ambiguity with an `inventory` data model module if one is added later.
