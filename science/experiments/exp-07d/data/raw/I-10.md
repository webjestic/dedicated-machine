Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and largely correct. However, there are several correctness, safety, and architectural issues ranging from a critical TOCTOU race in the stock check, to silent exception swallowing, to a heartbeat thread that can outlive its usefulness without signaling upward, that must be resolved before this touches a production flash sale workload. The test suite also has meaningful coverage gaps that would let real bugs ship undetected.

---

## Findings

### 🔴 Critical

**1. TOCTOU Race: Stock Read Is Outside the Database Transaction**

```python
# CURRENT — stock read is OUTSIDE the transaction
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← read here
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← write here
        self.db.create_order(...)
```

The Redis lock serializes callers *at the application layer*, but `get_stock` is issued outside the database transaction. If the database has any read-committed or weaker isolation, another process that bypasses the Redis lock (a migration, a manual correction, a different service) can mutate stock between the read and the write. The stock read and the conditional write must be atomic at the database level — move `get_stock` inside `self.db.transaction()`, or use a `SELECT ... FOR UPDATE` / optimistic version-check pattern inside the transaction.

```python
# REQUIRED pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

**2. Silent Swallowing of All Non-Lock Exceptions**

```python
except LockAcquisitionError as e:
    ...
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `LockAcquisitionError` is caught. Any exception raised inside the `with self._acquire_lock(lock_key):` block — database errors, network timeouts, unexpected `None` dereferences — propagates out of `reserve_item` as an unhandled exception. The caller receives no `ReservationResult`; they receive a raw exception. For a public-facing service this is both a reliability and a potential information-disclosure issue. A broad `except Exception` with logging and a safe `ReservationResult(success=False, ...)` return must wrap the critical section, distinct from the lock-acquisition catch.

---

### 🟠 High

**3. Heartbeat Thread Silently Absorbs Lock Loss — No Signal to the Critical Section**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; thread exits, but caller never knows
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat correctly detects lock loss, but it silently returns. The `reserve_item` code continues executing — reading stock, writing orders — under the false belief it still holds the lock. Another node may have acquired the lock and begun its own reservation by this point. The heartbeat must signal lock loss back to the main thread. A shared `threading.Event` (e.g., `lock_lost_event`) should be set on `result == 0`, and the critical section must check it before performing writes.

```python
# Sketch
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()  # signal the owner
            return
        ...

# In reserve_item, before the DB write:
if lock_lost_event.is_set():
    raise LockLostError(f"Lock lost mid-operation for {lock_key}")
```

**4. `t.join(timeout=...)` Does Not Guarantee Thread Termination Before Lock Release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← timeout may expire
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns after N seconds *regardless of whether the thread has finished*. If the heartbeat thread is blocked on a slow Redis call, the lock release script fires while the heartbeat thread is still alive and potentially mid-extension. This creates a narrow but real window where the lock is released and then re-extended by the zombie heartbeat, briefly re-locking a key no longer owned by this process. Either check `t.is_alive()` after join and log a hard warning, or restructure so the heartbeat cannot call `_extend_script` after `stop_event` is set (the current `stop_event.wait()` logic actually handles this correctly *if* the thread isn't blocked in the Redis call itself — but that case is unguarded).

---

**5. Retry Backoff Ceiling Is Dangerously Low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~0.10s, attempt=1 → ~0.20s, attempt=2 → ~0.40s
```

With only 3 attempts and a max backoff of ~0.45 seconds against a 30-second lock TTL, this is effectively a fast-fail, not a retry strategy. Under a flash sale load spike, every contending caller will exhaust retries in under a second and return `"Service temporarily unavailable"` to the user. Either increase `retry_attempts` (10–15 is reasonable for a 30s TTL), raise the backoff multiplier, or make both configurable via constructor parameters so callers can tune for their SLA. The jitter ceiling of `0.05` is also too narrow to meaningfully spread thundering-herd retries.

### 🟡 Medium

**6. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class-Level Constants**

These values are baked into the class body. Different flash sale items may have very different expected transaction durations. These should be constructor parameters with sensible defaults, enabling per-instance or per-deployment tuning without subclassing.

---

**7. `_acquire_lock` Yields `token` But `reserve_item` Ignores It**

```python
with self._acquire_lock(lock_key):   # token discarded
```

The token is yielded for a reason — it is the proof-of-ownership used by the Lua scripts. Discarding it at the call site is not a bug today (the heartbeat captures it via closure), but it is a misleading API. Either stop yielding the token (yield `None` or nothing), or document explicitly that the token is for internal heartbeat use only. As written, a future developer may assume the yielded token is needed and introduce a second release path.

---

**8. No Redis Connection Error Handling**

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are never caught. A Redis outage during lock acquisition or heartbeat will raise unchecked exceptions. For a flash sale service, the failure mode on Redis unavailability should be a deliberate policy decision (fail open? fail closed?) documented and enforced in code, not an accidental unhandled exception.

---

**9. Lua Script Uses String Comparison for TTL Argument**

```python
args=[token, str(self.LOCK_TTL)]
```

```lua
return redis.call("expire", KEYS[1], ARGV[2])
```

Redis `EXPIRE` accepts an integer. Passing a string works because Redis coerces it, but this is an implicit dependency on Redis's type coercion and is not obvious to future maintainers. Pass the integer directly or add an explicit comment.

### 🟢 Low / Observations

**10. Test: `test_lock_release_uses_lua_script` Is Structurally Weak**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the Lua release script was called once, but it does not assert *what arguments* were passed. A future refactor that calls the script with the wrong key or a hardcoded token would pass this test. Assert `called_once_with` including the correct `keys` and `args` shape.

---

**11. No Test for Heartbeat Lock-Loss Scenario**

Finding #3 above — the heartbeat silently absorbing lock loss — has zero test coverage. Once the lock-lost signaling mechanism is added, a test must verify that `reserve_item` raises or returns a failure result when the heartbeat detects `result == 0` mid-operation.

---

**12. No Test for Database Transaction Rollback Behavior**

There is no test asserting that if `db.update_stock` or `db.create_order` raises, the transaction is rolled back and the lock is still released. This is the most catastrophic failure path (partial write + lock held) and it is entirely untested.

---
