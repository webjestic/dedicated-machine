Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, backoff/jitter, and Pydantic validation are all solid choices. However, there are several correctness, resilience, and security issues that must be addressed before this merges into a government-facing production system. The test suite in particular has meaningful gaps that would give false confidence under adversarial or failure conditions.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU race between `get_stock` and `update_stock`**

```python
current_stock = self.db.get_stock(req.item_id)
# ...
new_stock = current_stock - req.quantity
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The stock read happens *outside* the database transaction. The distributed lock serializes Redis-level access, but if the database itself is accessed by any other path (admin tool, migration script, another service node that bypasses Redis), the stock check is stale by the time the write commits. The read must be inside the transaction, and `update_stock` should use an optimistic or conditional write — e.g., `UPDATE inventory SET stock = %s WHERE item_id = %s AND stock >= %s` — so the database itself enforces the invariant. A lock that only lives in Redis is not a substitute for database-level consistency.

---

**2. Silent lock loss in the heartbeat thread**

```python
if result == 0:
    return  # lock was lost — exits silently
```

When the heartbeat detects that the lock token no longer matches (lock expired and was re-acquired by another worker), it exits quietly. The main thread has no way to know this happened and continues executing the database transaction against inventory it no longer holds the lock on. This is a correctness violation: two workers can now both believe they hold the reservation right for the same item simultaneously. The heartbeat must signal the main thread — via a shared `threading.Event` or a flag — so the main thread can abort the transaction before writing.

---

### 🟠 High

**3. Unhandled exceptions inside the `finally` block can swallow the original exception**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If `self._release_script(...)` raises (Redis connection lost, network partition), it will replace any exception already propagating from the `try` block. Use a bare `try/except` around the release call and log the failure without re-raising. Additionally, if `t.join()` times out, the thread is still running — this should at minimum be logged as an error, since a zombie heartbeat thread can extend a lock that the main thread has already abandoned.

**4. No timeout on the database operations**

The heartbeat extends the lock indefinitely as long as the DB call is in progress. A hung DB connection (deadlock, network partition) will hold the distributed lock forever, blocking all other workers for that `item_id`. Database calls should be wrapped with a timeout, and the heartbeat should have a maximum extension count or a wall-clock deadline after which it stops extending and the main thread is signaled to abort.

**5. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=2_000_000_000`. This passes validation and will be logged, stored, and potentially used to zero out stock in a single call. Add a reasonable `le=` ceiling consistent with your business rules (e.g., `le=1000`).

---

### 🟡 Medium

**6. `item_id` leaked into log output and error messages**

```python
message=f"Item {req.item_id} not found"
logger.error("Lock acquisition failed item_id=%s: %s", req.item_id, e)
```

`item_id` is user-supplied. Even though it passes the regex, embedding it verbatim into user-facing messages and structured logs can facilitate log injection or information disclosure in audit trails. Log the validated value only in structured fields; do not interpolate it into the human-readable `message` string returned to callers.

**7. Test for lock release uses the wrong assertion**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service._release_script.assert_called_once()
```

`service._release_script` is the mock returned by `register_script`, not the script object stored on the instance after `__init__`. This test passes trivially regardless of whether the release path actually executes. The assertion should be on `service._release_script` as accessed through the instance after the call, and should also assert the correct `keys` and `args` were passed.

**8. No test for lock-loss-during-transaction scenario**

There is no test that verifies behavior when the heartbeat detects `result == 0` mid-transaction. Given finding #2 above, this is the most dangerous code path and has zero coverage.

**9. No test for DB transaction failure / rollback**

If `self.db.update_stock` or `self.db.create_order` raises inside the transaction, the lock is still released (correctly), but there is no test confirming the `ReservationResult` reflects the failure and that no partial state is committed.

---

### 🔵 Low / Nits

**10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants, not configurable**

For a service deployed across environments (dev, staging, prod), these should be injectable via constructor parameters with the current values as defaults. Hardcoded values make tuning for different DB latency profiles impossible without a code change.

**11. Daemon thread is acceptable but worth documenting**

`daemon=True` means the heartbeat is killed if the process exits mid-operation. This is probably the right call, but it should be explicitly documented as a conscious decision — a non-daemon thread would prevent clean shutdown.

---

## Required Before Approval

1. Move `get_stock` inside the DB transaction and use a conditional `UPDATE` to enforce the stock invariant at the database layer.
2. Implement a mechanism for the heartbeat to signal the main thread on lock loss, and have the main thread abort the transaction on that signal.
3. Guard the `finally` release call against exceptions.
4. Add an upper bound to `quantity`.
5. Fix the lock-release test assertion and add tests for lock-loss and DB failure paths.
