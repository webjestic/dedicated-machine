Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a race condition, a silent failure path, and test gaps that would make this unsafe to deploy under real flash-sale concurrency. The issues range from a stock-read/write inconsistency that can corrupt inventory under certain failure modes, to a heartbeat that silently swallows lock loss without surfacing it to the caller.

---

## Detailed Findings

### 🔴 Critical

---

**1. Stock read occurs outside the database transaction — TOCTOU window**

```python
# Line ~90-91: read happens OUTSIDE the transaction block
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # write happens INSIDE
    self.db.create_order(...)
```

The Redis lock serializes concurrent *reservation attempts*, but it does not protect against the database itself. If `get_stock` reads from a replica with replication lag, or if the DB connection is routed differently than the transaction connection (common in connection-pooled environments), the stock value observed before the transaction may be stale by the time the `UPDATE` commits. The read and the conditional write must be atomic at the database layer — either move `get_stock` inside the transaction, or use a `SELECT ... FOR UPDATE` / optimistic version-check pattern so the DB itself enforces the invariant. The Redis lock alone is not sufficient here.

---

**2. Heartbeat thread silently absorbs lock loss — caller is never notified**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # <-- lock was lost; thread exits, but nothing tells reserve_item
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or evicted). The heartbeat thread exits cleanly, but the main thread continues executing the database transaction *without holding the lock*. This is the exact failure mode the lock is supposed to prevent. Another process can acquire the lock between the heartbeat's silent exit and the DB commit, and both processes will write to inventory concurrently. The heartbeat must signal the main thread — via a shared `threading.Event`, a queue, or by raising in the main thread via `ctypes` — and the main thread must check that signal before and ideally during the transaction.

---

### 🟠 High

---

**3. `t.join(timeout=...)` in `finally` does not guarantee the thread has stopped before lock release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # timeout = 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns after N seconds *regardless of whether the thread finished*. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis slowlog, etc.), `join` will time out and return `False`, and then `_release_script` will execute while `_beat` is still mid-execution and potentially calling `_extend_script` concurrently. You must check the return value of `join` or use a hard deadline with a fallback. At minimum:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # do not release — log and alert; releasing here is a race
```

---

**4. Retry backoff ceiling is extremely low under real contention**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~100ms, attempt=1 → ~200ms, attempt=2 → ~400ms
```

With only 3 attempts and a maximum backoff of ~405ms, this will produce a thundering herd under flash-sale load. Hundreds of goroutines hitting a 30-second lock will all fail within 700ms and return "service unavailable" to the user. The retry count and ceiling should be configurable and tuned against the expected lock hold time. A cap of at least `min(LOCK_TTL / 2, configured_max)` with a configurable attempt count would be more appropriate for a flash-sale scenario.

---

**5. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are hardcoded class-level constants**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8
```

These are not injectable, not per-item-type configurable, and not documented with rationale. For a government contracting context where audit trails and configuration management matter, magic numbers embedded in class bodies are a compliance concern. These should be constructor parameters with documented defaults, and the relationship `HEARTBEAT_INTERVAL < LOCK_TTL / 2` should be enforced with an assertion at construction time.

---

### 🟡 Medium

---

**6. `db` interface is completely untyped — no protocol or ABC**

The `db` parameter accepts anything. There is no `Protocol`, `ABC`, or type stub defining `get_stock`, `update_stock`, `create_order`, and `transaction`. This makes the contract invisible to static analysis, makes mocking in tests unverifiable, and makes it trivial to pass an incompatible object silently. Define a `InventoryRepository` protocol.

---

**7. Exception handling inside `reserve_item` is too narrow**

```python
except LockAcquisitionError as e:
    ...
```

Any exception raised by `self.db.get_stock`, `self.db.update_stock`, `self.db.create_order`, or `self.db.transaction` propagates uncaught out of `reserve_item`. In a flash-sale service, a DB timeout or connection error will bubble up as an unhandled exception to whatever is calling this — likely an HTTP handler that will return a 500 to the user. Either define and catch a `InventoryServiceError` base class, or at minimum wrap the DB calls in a `try/except` that returns a safe `ReservationResult`.

---

**8. `order_id` is generated before the transaction commits**

```python
order_id = str(uuid.uuid4())
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If the transaction rolls back, `order_id` has already been logged and potentially returned. This is a minor issue if the function never returns on rollback (exception propagates), but it becomes a real problem if the caller logs or caches the order ID before confirming the result. Generate `order_id` inside the transaction, or only surface it after commit confirmation.

---

### 🟢 Test Coverage Gaps

---

**9. No test for heartbeat lock-loss scenario (the most critical path)**

There is no test verifying that if `_extend_script` returns `0` (lock lost mid-operation), the reservation fails safely. This is the highest-risk code path and it has zero coverage.

---

**10. `test_lock_release_uses_lua_script` is structurally weak**

```python
service._release_script.assert_called_once()
```

This asserts the script was called, but not *what it was called with*. A test that passes `keys=[]` and `args=[]` would still pass. Assert the specific `keys` and `args` to verify the token is correctly threaded through.

---

**11. No test for DB transaction rollback behavior**

There is no test verifying that if `db.update_stock` or `db.create_order` raises, the transaction is rolled back and the function does not return `success=True`. Given finding #7, this is an untested failure mode.

---

**12. No test for concurrent reservation of the same item**

The entire point of this service is flash-sale concurrency. There is no test that spins up two threads calling `reserve_item` simultaneously and verifies exactly one succeeds when stock is 1. This is table stakes for a distributed locking implementation.

---
