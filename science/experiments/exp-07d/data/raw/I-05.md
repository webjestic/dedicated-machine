Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent data-integrity race, a resource-leak vector, and test gaps that collectively make this unfit for a high-concurrency flash sale workload in its current state. The issues range from a critical TOCTOU stock-read vulnerability to a heartbeat thread that can silently abandon a live lock without any caller notification.

---

## Critical Issues

### 1. TOCTOU Race on Stock Read — Data Integrity Violation

This is the most serious flaw in the submission.

```python
# Inside the lock — looks safe, but isn't
current_stock = self.db.get_stock(req.item_id)   # READ outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE inside transaction
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens **outside** the database transaction. The distributed lock serializes callers through Redis, but it provides zero protection against:

- A direct database write from another service, a migration script, an admin tool, or any process that bypasses this lock entirely.
- A database replica lag scenario where `get_stock` reads stale data from a follower.
- Any future code path that touches inventory without acquiring this exact lock key.

The read **must** be inside the transaction, and the update should use a conditional decrement (optimistic or pessimistic at the DB level) rather than a blind overwrite. The correct pattern is:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

`SELECT FOR UPDATE` (or equivalent) makes the database itself the authoritative serialization point for the stock row, which is what you actually want. The Redis lock then serves its correct role: rate-limiting the thundering herd before it hits the database, not as the sole correctness mechanism.

### 2. Silent Lock Loss in Heartbeat — No Caller Notification

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # <-- lock was lost; caller has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has expired or been stolen. The heartbeat thread exits silently. The main thread continues executing under the assumption that it still holds the lock — it does not. Any work done after this point is unprotected. In a flash sale context, this is a direct path to overselling.

The heartbeat needs a signaling mechanism back to the caller. A `threading.Event` for lock loss, or storing the loss state on the instance, is the minimum required:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during operation key=%s", lock_key)
            lock_lost_event.set()
            return
```

The `yield` block (or the caller) must then check `lock_lost_event` before committing any state, and raise an exception to abort the operation if the lock was lost.

## Significant Issues

### 3. Exception Handling Gap — Unhandled Exceptions Leak the Lock Thread

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block is correct in structure, but `self._release_script(...)` can itself raise (Redis connection failure, network timeout, etc.). If it does, the exception propagates out of the `finally` block and **swallows** any original exception that triggered the `finally`. This makes debugging extremely difficult in production.

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Lock released key=%s", lock_key)
    except redis.RedisError:
        logger.exception("Failed to release lock key=%s — TTL will expire naturally", lock_key)
```

### 4. `t.join()` Timeout Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked (e.g., on a slow Redis call), `join` returns after the timeout and execution continues — but the thread is still running. The `_release_script` call then races with the still-running heartbeat's `_extend_script`. This is a benign race in practice but is still undefined behavior and will produce confusing log output.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

### 5. Retry Backoff Is Functionally Broken for `attempt=0`

```python
for attempt in range(retry_attempts):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
    if acquired:
        break
    backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
    time.sleep(backoff)
```

On `attempt=0`, `backoff = 0.1 + jitter`. On `attempt=2` (the last attempt), the code sleeps **after** the final failed acquisition, then exits the loop and raises. That final sleep is wasted latency. The sleep should be conditioned on whether another attempt will follow:

```python
if attempt < retry_attempts - 1:
    time.sleep(backoff)
```

## Minor Issues

### 6. `HEARTBEAT_INTERVAL` Relative to `LOCK_TTL` Is Fragile

`HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` gives a comfortable ratio, but these are bare class-level constants with no enforced relationship. Nothing prevents a future change to `LOCK_TTL = 10` that makes the heartbeat fire at TTL*0.8, leaving almost no margin before expiry. The interval should be derived:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

Or better, enforce it in `__init__` with an assertion.

### 7. `new_stock` Can Go Negative Under Concurrent Negative Quantity

`quantity` is validated `ge=1`, so this is not currently exploitable through the public API. However, `update_stock` is called with a pre-computed `new_stock = current_stock - req.quantity` rather than a delta. If `update_stock` is ever called from another path with a raw value, there is no floor check at the DB layer visible here. A delta-based decrement (`UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`) is more robust and eliminates the TOCTOU issue simultaneously.

## Test Coverage Gaps

### 8. No Test for Heartbeat Lock-Loss Path

The most critical behavioral path — heartbeat detects `result == 0` — has zero test coverage. This needs a test that verifies the caller is notified and the operation is aborted.

### 9. No Test for `finally` Block on Exception

There is no test verifying that the lock is released when an exception is raised inside the `with self._acquire_lock(...)` block. This is a standard requirement for any context manager managing a resource.

### 10. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A call with the wrong token would pass this test. Use `assert_called_once_with(keys=[...], args=[...])` — though since the token is a UUID generated internally, you will need to capture it via a mock side effect or inspect the call args directly.
