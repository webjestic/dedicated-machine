Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, input validation, and transaction wrapping are all solid. However, there are several correctness, reliability, and security concerns that must be addressed before this merges into a government-facing production system. The issues range from a silent exception-swallowing gap in the critical path to a thread lifecycle race and a test that asserts the wrong object.

---

## Findings

### 🔴 Critical

**1. Unhandled exceptions inside the lock context leak the lock (or silently succeed)**

In `reserve_item`, only `LockAcquisitionError` is caught at the outer level. Any exception raised by `db.get_stock`, `db.update_stock`, `db.create_order`, or the transaction context manager — network errors, ORM exceptions, timeouts — will propagate out of the `with self._acquire_lock(lock_key):` block. The `finally` clause in `_acquire_lock` *will* release the lock, which is correct, but the caller of `reserve_item` receives an unhandled exception rather than a `ReservationResult`. For a service method that promises to return a `ReservationResult`, this is a broken contract and will cause 500-class failures upstream.

```python
# Required: wrap the lock body
try:
    with self._acquire_lock(lock_key):
        ...
except LockAcquisitionError as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

---

**2. TOCTOU gap between stock read and transaction write**

`get_stock` is called *outside* the database transaction. If the DB layer does not hold a row-level lock across that call, a second process that somehow bypasses the Redis lock (Redis failure, lock expiry under extreme GC pause, Redis restart) can read the same stock value and both will commit. The stock read must be inside the transaction and should use a `SELECT ... FOR UPDATE` or equivalent pessimistic lock at the DB layer. This is defense-in-depth and is especially important for a government inventory system where overselling has audit consequences.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

---

### 🟠 High

**3. Heartbeat thread can outlive the lock release window**

`t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` is `9` seconds. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), `join` returns after 9 seconds and the Lua release script fires anyway — but the heartbeat thread is still alive and will attempt another `_extend_script` call on a key that is now either deleted or owned by a different token. The Lua token check prevents a wrong-owner extension, which is good, but the thread continues running as a daemon until process exit. Under sustained load this accumulates zombie-ish threads. The heartbeat should also catch and log Redis exceptions internally rather than silently dying.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
            if result == 0:
                logger.warning("Lock lost during heartbeat key=%s", lock_key)
                return
            logger.debug("Lock TTL extended key=%s", lock_key)
        except redis.RedisError as exc:
            logger.error("Heartbeat Redis error key=%s: %s", lock_key, exc)
            return  # Do not retry blind; let the TTL expire naturally
```

---

**4. `retry_attempts=3` with exponential backoff produces misleading semantics**

With `attempt` values `0, 1, 2`, the backoffs are `~0.1s`, `~0.25s`, `~0.45s` — total wait under `1s`. For a 30-second lock TTL on a flash sale, this is almost certainly too short and will surface as false "service unavailable" responses under legitimate load spikes. The retry count and backoff ceiling should be configurable via constructor parameters, not hardcoded, so operators can tune without a code deploy.

---

### 🟡 Medium

**5. `remaining_stock` is exposed in the "item not found" path**

This is minor but worth noting: when `get_stock` returns `None`, the response omits `remaining_stock`, which is correct. However, when stock is insufficient, `remaining_stock=current_stock` is returned to the caller. Depending on the API surface, leaking exact inventory counts to end users is a business/security consideration. Consider returning a boolean or bucketed value (`"low"`, `"out"`) at the API boundary rather than raw counts.

---

**6. `lock_key` namespace is predictable and flat**

`inventory:lock:{item_id}` with `item_id` constrained to `[a-zA-Z0-9_\-]` is fine for injection, but there is no environment or tenant namespace. If this Redis instance is shared across staging and production (common in rapid-growth shops), a staging flash sale test will block a production reservation on the same `item_id`. Prefix with environment: `f"{env}:inventory:lock:{req.item_id}"`.

---

### 🔵 Test Issues

**7. `test_lock_release_uses_lua_script` asserts the wrong object**

```python
service._release_script.assert_called_once()
```

`service._release_script` is the return value of `redis_client.register_script(RELEASE_LOCK_SCRIPT)` — which in the fixture is a single shared `script_mock` returned for *all* `register_script` calls. Both `_release_script` and `_extend_script` point to the same mock object. This test will pass even if the extend script is called and release is never called, or if both are called. Each script registration should return a distinct mock.

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    release_mock = MagicMock(return_value=1)
    extend_mock = MagicMock(return_value=1)
    mock.register_script.side_effect = [release_mock, extend_mock]
    return mock
```

Then assert `service._release_script` and `service._extend_script` independently.

**8. No test for DB exception inside the lock context**

Given finding #1, there should be a test confirming that a `db.get_stock` exception returns a `ReservationResult(success=False)` rather than propagating. This is a regression guard for the fix.

**9. No test for heartbeat behavior under lock loss**

The heartbeat's `result == 0` branch (lock lost mid-operation) has no test coverage. This is the most dangerous silent failure path in the system.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | 🔴 | Catch all exceptions inside `reserve_item`, return `ReservationResult` |
| 2 | 🔴 | Move `get_stock` inside transaction with pessimistic DB lock |
| 3 | 🟠 | Add Redis error handling in heartbeat thread |
| 4 | 🟠 | Make retry count and backoff configurable |
| 5 | 🟡 | Evaluate `remaining_stock` exposure policy |
| 6 | 🟡 | Add environment/tenant namespace to lock key |
| 7 | 🔵 | Fix shared script mock in test fixture |
| 8 | 🔵 | Add test for DB exception path |
| 9 | 🔵 | Add test for heartbeat lock-loss path |

The architecture is sound and the author clearly understands distributed locking. Items 1 and 2 are blocking — they represent correctness failures under failure conditions that *will* occur in a flash sale load profile.
