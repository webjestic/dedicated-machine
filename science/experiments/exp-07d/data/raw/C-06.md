Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness bugs, architectural flaws, and test gaps that would make this dangerous to deploy in a high-concurrency flash sale environment where inventory correctness is a hard requirement. The issues range from a TOCTOU race inside the lock boundary to silent exception swallowing that could leave stock permanently decremented without an order record.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Outside the Database Transaction

**File:** `services/inventory.py`, `reserve_item()`

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes computed value
    self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` is called outside the database transaction. If the database supports read-committed or higher isolation, another process (a refund, an admin adjustment, a bulk import) can modify stock between the `get_stock` call and the `update_stock` call. The lock does not protect against out-of-band writers.

**Required fix:** Move `get_stock` inside the transaction boundary, and use a conditional update or SELECT FOR UPDATE at the database layer:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return inside transaction
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is a performance optimization for reducing DB contention — it is not a substitute for database-level consistency guarantees.

### 2. Silent Exception Swallowing: Partial Failure Leaves Inconsistent State

**File:** `services/inventory.py`, `reserve_item()`

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Any exception raised by `update_stock` or `create_order` (network timeout, constraint violation, deadlock) is not caught. The `with self.db.transaction()` block will presumably roll back, which is correct — but the exception then propagates up through `_acquire_lock`'s `finally` block, which releases the lock and logs "Lock released," and then exits `reserve_item` entirely with an unhandled exception. The caller receives an uncaught exception rather than a `ReservationResult`.

There is no `except Exception` guard around the transaction block, and no test covering this path.

**Required fix:**

```python
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error("Transaction failed item_id=%s order_id=%s: %s", req.item_id, order_id, e)
    return ReservationResult(success=False, message="Reservation failed due to an internal error.")
```

### 3. Heartbeat Thread Detects Lock Loss But Does Not Propagate It

**File:** `services/inventory.py`, `_beat()`

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # <-- silently exits
```

If the lock expires (Redis eviction under memory pressure, a clock skew event, a Redis restart), the heartbeat thread silently exits. The main thread continues executing the database transaction believing it holds the lock, but it does not. Another process can now acquire the lock and begin its own reservation concurrently — breaking the mutual exclusion guarantee entirely.

This is the most architecturally dangerous flaw. The heartbeat must signal the main thread to abort.

**Required fix:** Use the `stop_event` bidirectionally, or introduce a separate `lock_lost_event`:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling abort", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then inside `reserve_item`, check `lock_lost_event` before and after the transaction:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock was lost during operation")
```

## Significant Issues

### 4. `t.join()` Timeout Does Not Guarantee Thread Has Exited

**File:** `services/inventory.py`, `_acquire_lock()` finally block

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call, `join` times out and returns, and then `_release_script` runs. Now both the heartbeat thread and the main thread are issuing Redis commands concurrently. The Lua script is atomic per-call, so the release will succeed, but the heartbeat thread may then attempt an extension on a key that no longer exists — or worse, a key that has been re-acquired by another caller with a different token (which the Lua script handles correctly, but the log noise and timing window are real).

**Required fix:** Check the return value of `join` or log a warning when the thread does not exit cleanly:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly for key=%s — possible resource leak", lock_key)
```

### 5. Backoff Jitter Range Is Too Narrow Under High Concurrency

**File:** `services/inventory.py`, `_acquire_lock()`

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With 100 concurrent callers, the jitter window of 50ms is insufficient to spread retries. At attempt 0, all callers sleep in the range `[0.1, 0.15]` seconds and then stampede simultaneously. This is a thundering herd problem.

**Required fix:** Widen the jitter to a meaningful fraction of the backoff:

```python
backoff = (2 ** attempt) * 0.1
jitter = random.uniform(0, backoff)  # full jitter — industry standard
backoff = backoff + jitter
```

### 6. `quantity` Field Has No Upper Bound

**File:** `services/inventory.py`, `ReservationRequest`

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can submit `quantity=999999999`, which will pass validation, acquire the lock, and either drain all stock or produce a nonsensical negative stock value if the check is bypassed. For a flash sale context, this should have a hard cap.

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever business rule applies
```

## Test Coverage Gaps

### 7. No Test for Database Transaction Failure

There is no test covering the scenario where `update_stock` or `create_order` raises an exception. This is the partial-failure path identified in Issue #2. Without a test, the silent propagation behavior will go undetected.

### 8. No Test for Lock Loss Mid-Operation

There is no test simulating the heartbeat returning `0` (lock expiry). Given that this is the most dangerous failure mode, it must have explicit test coverage.

### 9. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Prefer:

```python
service._release_script.assert_called_once_with(keys=[expected_key], args=[expected_token])
```

This requires capturing the token, which may require a small test helper or mock injection.

### 10. Heartbeat Thread Is Not Tested at All

The heartbeat renewal path — the `_beat` function — has zero test coverage. A unit test should mock `time` or use `threading.Event` injection to verify that the extend script is called on interval and that the thread exits cleanly when `stop_event` is set.

## Minor Issues

### 11. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

These should be configurable at construction time to support different SLA environments (test, staging, production). Hardcoding them makes the class difficult to tune without modifying source.

### 12. Lock Key Namespace Is Flat

`inventory:lock:{item_id}` — if this Redis instance is shared with other services, there is no service-level namespace prefix. Consider `obotix:inventory:lock:{item_id}` or inject a configurable prefix.

### 13. `logger.info("Lock released ...")` Fires Even on Exception Paths

In the `finally` block, the release log fires unconditionally. If the lock was never actually held (which cannot happen given the current control flow, but could after refactoring), this log would be misleading. Minor, but worth noting.
