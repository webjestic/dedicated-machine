Decision: Request Changes

Summary: The core distributed locking mechanism is structurally sound — atomic Lua scripts, scoped heartbeat, token-matched release, and `SET NX EX` acquisition are all correctly implemented. However, there are several correctness, safety, and architectural issues that must be addressed before this is suitable for a high-stakes flash sale workload. The most critical is a TOCTOU race between the stock read and the database write, which can cause overselling even with the Redis lock held.

---

## Critical Issues

### 1. TOCTOU Race — Stock Read Outside the Database Transaction

```python
# Lock is held, but stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # Writes a value computed outside the tx
    self.db.create_order(...)
```

The stock is read outside the database transaction. The Redis lock serializes callers *through this service*, but it provides zero protection against:

- Direct database writes from other services, admin tooling, or batch jobs that do not participate in this lock protocol.
- A crash between the read and the write leaving stock in an inconsistent state.
- Any future multi-instance deployment where a second service instance does not use this exact lock key scheme.

The stock decrement must be atomic at the database level. The correct pattern is a conditional `UPDATE` inside the transaction:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id AND stock >= :quantity
RETURNING stock;
```

If the `UPDATE` affects 0 rows, the stock was insufficient or the item does not exist. The read-then-write pattern here is a classic oversell bug. This is a **blocking issue**.

---

### 2. Lock Release Proceeds Even After Heartbeat Reports Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # <-- silently exits; caller never knows
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When the heartbeat detects `result == 0` (lock expired or stolen), it returns silently. The main thread continues executing the database transaction and then calls `_release_script` — which will correctly no-op because the token no longer matches. But the database work has already been committed under a lock that was no longer held. Another process may have acquired the lock and begun its own transaction concurrently.

The heartbeat must have a signaling path back to the main thread. A `threading.Event` for lock loss, or a shared flag checked before the DB transaction is committed, is required. If the lock is lost, the transaction should be rolled back and the caller should receive an error, not a success result.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()  # Signal main thread
            return
```

The main thread must check `lock_lost_event` before committing.

### 3. Heartbeat Thread Leak on Unhandled Exceptions

```python
t = threading.Thread(target=_beat, daemon=True)
t.start()

try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block is correct in structure, but `t.join(timeout=...)` can return before the thread has actually stopped — `join` with a timeout does not raise if the timeout expires. After `join` returns, the code unconditionally calls `_release_script`. If the thread is still running (e.g., blocked on a slow Redis call), it may call `_extend_script` *after* the lock has been released, potentially extending a lock that has been legitimately acquired by another caller.

You must check `t.is_alive()` after `join` and log or raise if the thread did not terminate cleanly:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly for key=%s", lock_key)
    # Do not proceed silently
```

## Significant Issues

### 4. Exponential Backoff Ceiling Is Missing

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff is `(2^2) * 0.1 + 0.05 = 0.45s`. This is fine for the default, but `retry_attempts` is a caller-controlled parameter with no documented upper bound. If a caller passes `retry_attempts=20`, the backoff reaches `(2^19) * 0.1 ≈ 52,428 seconds`. A cap is required:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

---

### 5. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity. A single caller can reserve `quantity=999999` and drain the entire inventory in one request. For a flash sale context, this is an obvious abuse vector. A reasonable domain maximum should be enforced:

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business rule is
```

---

### 6. `item_id` Max Length Is Generous But the Pattern Is Correct

This is minor, but `max_length=128` is fine. No issue here — just confirming the regex `^[a-zA-Z0-9_\-]+$` correctly prevents path traversal and injection. The test for `../../admin` correctly validates this.

---

### 7. No Timeout on Database Operations

The heartbeat keeps the Redis lock alive, but there is no timeout on `self.db.get_stock()` or the transaction block. If the database hangs indefinitely, the heartbeat will extend the lock indefinitely, blocking all other callers for that item forever. Database calls must have an explicit timeout, and the lock context must have a maximum wall-clock TTL that causes it to abort.

---

## Test Coverage Gaps

### 8. Heartbeat Lock-Loss Path Is Not Tested

There is no test that simulates `_extend_script` returning `0` mid-operation. The most dangerous failure mode in this code — silent lock loss during a DB transaction — has zero test coverage.

### 9. `t.join` Non-Termination Is Not Tested

The thread leak scenario described in issue #3 is not covered.

### 10. Database Transaction Rollback Is Not Tested

There is no test for what happens when `self.db.update_stock` or `self.db.create_order` raises an exception inside the transaction. The `with self.db.transaction()` block should roll back, but this is not verified.

### 11. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A regression that passes the wrong token would not be caught:

```python
service._release_script.assert_called_once_with(keys=[expected_lock_key], args=[expected_token])
```

---

## Minor / Informational

### 12. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These should be configurable at instantiation time, not hardcoded as class attributes. Different item categories may have different expected transaction durations. Injecting them via `__init__` parameters with defaults is the correct pattern.

### 13. Structured Logging Is Good, But `lock_key` Leaks Into Logs at `INFO`

```python
logger.info("Lock released key=%s", lock_key)
```

`lock_key` contains `item_id`, which may be sensitive in some inventory contexts. This is low severity but worth noting for compliance review given the contracting context.

### 14. `daemon=True` on the Heartbeat Thread

Using `daemon=True` means if the main process exits while the heartbeat is running, the thread is killed without cleanup. This is acceptable here since the `finally` block handles release, but it should be documented as an intentional choice.
