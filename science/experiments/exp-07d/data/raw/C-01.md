Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several concrete failure modes that would cause data corruption, silent lock loss, or deadlock under production load, and the test suite does not catch them. This code should not be merged into a high-stakes flash sale system in its current form.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and DB Transaction

```python
# Lock is held here — good
current_stock = self.db.get_stock(req.item_id)
...
new_stock = current_stock - req.quantity

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes a computed value, not a delta
    self.db.create_order(...)
```

`get_stock()` runs **outside** the database transaction. If the DB transaction uses a different connection (common in pooled environments), the stock read and the stock write are not isolated at the DB level. The Redis lock serializes callers *of this service*, but it provides zero protection against:

- Direct DB writes from other services or admin tooling
- A crash between `get_stock` and `update_stock` that leaves stock inconsistent
- Any future refactor that bypasses this lock

The write should be a conditional decrement, not an assignment of a pre-computed value:

```sql
UPDATE inventory SET stock = stock - :qty
WHERE item_id = :item_id AND stock >= :qty
```

Check the affected row count. If it's 0, the reservation failed. This makes the DB the authoritative source of truth and eliminates the TOCTOU window entirely. The Redis lock becomes a performance optimization (reducing DB contention), not a correctness mechanism.

### 2. Heartbeat Thread Silently Swallows Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # <-- lock was lost; caller has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits silently. The main thread continues executing the DB transaction believing it holds the lock, when in fact another process may have acquired it and already begun its own reservation. This is a silent correctness violation — the exact scenario distributed locking is meant to prevent.

The heartbeat must have a signaling path back to the main thread. A `threading.Event` or a shared flag is sufficient:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
```

The main thread must check `lock_lost_event` before and after the DB transaction and abort if it is set.

### 3. Lock Released After DB Transaction Commits — Wrong Order

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The lock is released in the `finally` block of the context manager, which runs *after* the `with self._acquire_lock(lock_key):` block exits — meaning after the DB transaction has already committed. This ordering is correct. However, if `t.join()` times out (the heartbeat thread is hung on a slow Redis call), the code falls through and releases the lock anyway, with no log entry and no indication that the join failed. A hung heartbeat thread is a real scenario under Redis latency spikes.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Decide: raise, alert, or continue — but do not silently proceed
```

### 4. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum. A caller can submit `quantity=10_000_000` against an item with 10 units in stock, which will fail gracefully, but not before acquiring the lock, hitting the DB, and blocking all other reservations for that item during the lock hold time. In a flash sale context this is a trivial denial-of-service vector.

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business limit is
```

### 5. `item_id` Length Allows Lock Key Inflation

```python
item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
```

The lock key is `f"inventory:lock:{req.item_id}"` — up to 144 bytes. This is not dangerous on its own, but 128 characters is generous for a product ID. More importantly, there is no validation that the `item_id` actually exists in the catalog before the lock is acquired. An attacker can force lock creation for arbitrary nonexistent keys, consuming Redis memory and polluting the keyspace. The existence check should happen before lock acquisition, or the item lookup should be part of the locked section with a fast pre-check outside.

## Moderate Issues

### 6. Exponential Backoff Ceiling Is Missing

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff is `(2^2) * 0.1 = 0.4s` — fine for the default. But `retry_attempts` is a parameter with no upper bound. A caller passing `retry_attempts=20` produces a backoff of `~104 seconds` on the last attempt, blocking a thread for nearly two minutes. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1, 2.0) + random.uniform(0, 0.05)
```

---

### 7. Redis Connection Errors Are Not Handled

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not caught anywhere in `_acquire_lock` or `reserve_item`. A Redis blip will propagate an unhandled exception up to the caller, bypassing the `LockAcquisitionError` path and the structured error response. The caller gets a raw exception instead of a `ReservationResult`. These should be caught and converted:

```python
except redis.RedisError as e:
    logger.error("Redis error acquiring lock key=%s: %s", lock_key, e)
    raise LockAcquisitionError(f"Redis unavailable: {e}") from e
```

---

### 8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These are not configurable at instantiation time. In production you will want different TTLs for different item categories, or the ability to tune them without a code deploy. Move them to `__init__` parameters with defaults:

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 8):
    self.LOCK_TTL = lock_ttl
    self.HEARTBEAT_INTERVAL = heartbeat_interval
```

## Test Suite Issues

### 9. Tests Do Not Cover the Lock-Loss Scenario

There is no test for the case where the heartbeat detects `result == 0` (lock lost mid-transaction). This is the most dangerous failure mode in the system and it is completely untested. Given that the current implementation silently ignores it, the absence of a test means the bug would never be caught in CI.

### 10. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Use `assert_called_once_with`:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[ANY]  # token is internal; at minimum assert structure
)
```

### 11. No Test for Redis Errors

There is no test covering `redis.RedisError` during lock acquisition. Given issue #7 above, this path currently raises an unhandled exception — and the test suite would not catch it.

### 12. Heartbeat Thread Is Not Tested for Timing Behavior

All tests complete synchronously because the mock Redis client returns immediately. The heartbeat thread starts and is joined but never actually fires in any test. There is no test that verifies the heartbeat actually extends the lock, or that it correctly signals lock loss. Consider a slow-path integration test or a test that directly exercises `_beat` logic.
