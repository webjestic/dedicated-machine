Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a race condition, a silent failure path, and test gaps that would make this unsafe to merge into a production flash-sale system where inventory integrity is critical. The issues range from a TOCTOU stock-read pattern to an unhandled exception class that can silently corrupt inventory state.

---

## Detailed Findings

### 🔴 Critical

---

**1. TOCTOU: Stock read occurs outside the database transaction**

```python
# Line in reserve_item — inside the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...stock check logic...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes based on stale read
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The distributed lock serializes Redis-level access, but `get_stock` is a bare read with no transactional isolation guarantee. If the database has read-committed or lower isolation, a concurrent write from a non-lock-aware path (admin correction, refund, bulk import) can modify stock between the read and the `update_stock` write. The lock protects against other *lock-holders*, not against out-of-band writers.

**Required fix:** Move `get_stock` inside the transaction block, or use a `SELECT ... FOR UPDATE` / CAS pattern (`UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`) so the decrement is atomic at the database layer. The Redis lock should be a *latency optimization*, not the sole correctness mechanism.

---

**2. Unhandled exception escapes the lock context, leaving heartbeat thread in an ambiguous state**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # can raise
    # ...
    with self.db.transaction():
        self.db.update_stock(...)    # can raise
        self.db.create_order(...)    # can raise
```

Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate *through* `_acquire_lock`'s `finally` block, which is correct for lock release. However, `reserve_item` has no `except` clause for generic database exceptions. This means the caller receives an unhandled exception rather than a `ReservationResult`, breaking the contract the method signature implies. In an async framework or WSGI handler that swallows exceptions, this can result in a silent partial write if the transaction context manager itself has a bug.

**Required fix:** Add an explicit `except Exception` (or typed DB exception) block around the lock body in `reserve_item`, log it, and return a safe `ReservationResult(success=False, ...)`. Do not let DB exceptions become caller-visible unchecked exceptions from a method that promises a `ReservationResult`.

### 🟠 High

---

**3. Heartbeat thread silently swallows lock loss — no propagation to the critical section**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # <-- lock was lost; thread exits quietly
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread exits, but the main thread continues executing inside the critical section — reading stock, writing to the database — *without holding the lock*. This is the exact scenario the heartbeat was designed to prevent, and it silently fails to prevent it.

**Required fix:** Use a shared `threading.Event` (e.g., `lock_lost_event`) that the heartbeat sets when `result == 0`. The critical section should check this event at meaningful points, or the `_acquire_lock` context manager should raise inside the `yield` by interrupting via a flag checked at the top of the `with` body. At minimum, log this as an ERROR, not silently return.

---

**4. `t.join(timeout=...)` does not guarantee the thread has stopped before lock release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # timeout = 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns after N seconds *regardless of whether the thread has exited*. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), `join` will time out and the release script will execute while the heartbeat thread is still alive and may issue a competing `expire` call immediately after. This creates a narrow window where a lock that was just released gets its TTL re-extended by the zombie heartbeat, preventing another node from acquiring it for up to `LOCK_TTL` seconds.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()`. If the thread is still alive, log a critical error. Consider making the heartbeat check `stop_event` before issuing the extend call (it does check `stop_event.wait`, but the extend call itself is not guarded against a race with the release).

---

**5. Retry backoff ceiling is extremely low under real contention**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~0.10s, attempt=1 → ~0.20s, attempt=2 → ~0.40s
```

With `retry_attempts=3` and a `LOCK_TTL` of 30 seconds, total retry time is under 1 second. If the lock holder is performing a slow DB transaction (which the heartbeat is explicitly designed to support), all retries will exhaust before the lock is released. The caller gets a `LockAcquisitionError` on a perfectly healthy system under moderate load.

**Required fix:** Either increase `retry_attempts` (5–10 is common), increase the backoff multiplier, or add a maximum cap (e.g., `min(backoff, 2.0)`). The retry budget should be proportional to `LOCK_TTL`, not to an arbitrary small constant.

### 🟡 Medium

---

**6. `HEARTBEAT_INTERVAL` of 8 seconds is too close to `LOCK_TTL / 3` only by coincidence — no enforcement**

The comment says `fires at TTL/3` but the values are hardcoded independently. If `LOCK_TTL` is changed to 15 seconds, `HEARTBEAT_INTERVAL` of 8 seconds means the first heartbeat fires at TTL/1.875 — dangerously close to expiry with only one renewal window before the lock expires.

**Required fix:**

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

Or enforce the relationship in `__init__` with an assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, "Heartbeat interval too large relative to TTL"
```

---

**7. `new_stock` can go negative if `update_stock` is not constrained at the DB layer**

```python
new_stock = current_stock - req.quantity
```

This is computed in application memory. If the DB layer does not enforce `CHECK (stock >= 0)`, and if finding #1 (TOCTOU) is not fixed, a race can produce a negative stock value that gets committed. There is no guard here.

**Required fix:** Either enforce a DB-level constraint, or use a conditional update: `UPDATE ... SET stock = stock - quantity WHERE stock >= quantity` and check rows-affected == 1.

---

### 🔵 Test Coverage Gaps

---

**8. No test for heartbeat lock-loss path (finding #3)**

The most dangerous silent failure — heartbeat detects `result == 0` and exits without notifying the main thread — has zero test coverage.

---

**9. No test for DB exception propagation**

There is no test verifying behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises. Given finding #2, this is a critical missing case.

---

**10. `test_lock_release_uses_lua_script` is structurally weak**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the script was called once, but does not assert it was called with the correct `keys` and `args` (i.e., the token that was used to acquire the lock). A refactor that calls the script with wrong arguments would pass this test.
