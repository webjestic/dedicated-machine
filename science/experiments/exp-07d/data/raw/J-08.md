Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, backoff/jitter, and Pydantic validation are all solid. However, there are several correctness, reliability, and security issues that need to be addressed before this merges. The most critical is a TOCTOU race between the stock read and the database write, which undermines the entire purpose of the distributed lock.

---

## Issues

### 🔴 Critical

**1. TOCTOU Race — Stock Read Outside the Database Transaction**

```python
# INSIDE the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...then later:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The stock read and the conditional write are not atomic at the database level. If the DB has read-committed or higher isolation, another process (one that bypassed Redis, a background job, a direct DB write, an admin tool) can modify stock between your `get_stock` call and your `update_stock` call. The Redis lock only protects against other callers of *this service*, not the database itself.

**Fix:** Use an optimistic or pessimistic DB-level lock. The safest pattern is a conditional update with a row-level lock or a compare-and-swap:

```python
with self.db.transaction():
    # Pessimistic: SELECT ... FOR UPDATE inside the transaction
    current_stock = self.db.get_stock_for_update(req.item_id)
    if current_stock is None or current_stock < req.quantity:
        # handle and return inside the transaction
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Or use a CAS update: `UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s` and check rows affected.

**2. Heartbeat Thread Silently Swallows Lock Loss**

```python
if result == 0:
    return  # lock was lost — thread exits, but caller is never notified
logger.debug("Lock TTL extended key=%s", lock_key)
```

When the heartbeat detects the lock has been lost (result == 0), it exits silently. The main thread continues executing the DB transaction believing it holds the lock, when it does not. This is a correctness violation — another process may have acquired the lock and already started its own reservation.

**Fix:** Use a shared flag or event to signal lock loss back to the main thread, and check it before and after the DB transaction:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# After the DB transaction:
if lock_lost_event.is_set():
    # compensate/rollback and raise
```

### 🟠 High

**3. No Exception Handling Around DB Operations**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

Any DB exception (connection timeout, deadlock, constraint violation) propagates uncaught out of `reserve_item`. The lock *will* still be released by the `finally` block, but the caller receives an unhandled exception rather than a `ReservationResult`. For a service method with a defined return type contract, this is a reliability gap.

**Fix:**

```python
except (DatabaseError, SomeOtherExpectedException) as e:
    logger.error("DB error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to a server error.")
```

Be specific about which exceptions you catch — do not use a bare `except`.

---

**4. `t.join()` Timeout Does Not Guarantee Thread Termination**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns regardless of whether the thread has actually stopped. If the heartbeat thread is blocked (e.g., Redis is slow), the lock release script runs while the thread may still be attempting an extension. This creates a narrow window where the lock is released and then re-extended by the stale thread — which would extend a lock that now belongs to another holder.

**Fix:** Check `t.is_alive()` after the join and log a critical error. More importantly, the Lua extension script already protects against this by checking the token, so the stale thread's extension will fail harmlessly *if* the new holder uses a different token (which it will, since tokens are UUIDs). Document this explicitly and add a log line:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical(
        "Heartbeat thread did not terminate within timeout key=%s. "
        "Extension will fail safely due to token mismatch.", lock_key
    )
```

### 🟡 Medium

**5. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound on quantity. A caller can request `quantity=10_000_000`. This won't cause a security breach given the stock check, but it's an input validation gap that should be closed at the boundary. Define a reasonable business-rule maximum.

```python
quantity: int = Field(..., ge=1, le=10_000)
```

---

**6. `item_id` Length Allows Oversized Redis Keys**

`max_length=128` on `item_id` means the Redis key can be `inventory:lock:` + 128 chars = 144 chars. This is fine for Redis, but the lock key construction should be documented and the max length justified against your actual item ID scheme. If IDs are UUIDs, `max_length=36` is more appropriate and reduces the attack surface for any future key-space confusion bugs.

---

**7. Test for Lua Script Call Is Fragile**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the script is called exactly once, but `_release_script` is also the mock returned by `register_script`, which is shared with `_extend_script`. If the heartbeat fires during the test (timing-dependent), the mock call count will be wrong. The fixtures should use separate mocks for release and extend scripts, or the test should use `assert_called()` with argument inspection.

---

**8. Missing Test: DB Transaction Failure / Rollback**

There is no test covering what happens when `db.update_stock` or `db.create_order` raises. Given issue #3 above, this is currently an untested failure path that produces an unhandled exception.

---

### 🔵 Informational

- The daemon thread (`daemon=True`) is appropriate here — it won't block process shutdown. Good.
- Lua scripts registered at `__init__` time are a clean pattern. Good.
- The `stop_event.wait()` used as a timer is correct and avoids busy-waiting. Good.
- Consider whether `LOCK_TTL = 30` is appropriate for your actual DB SLA. If a DB call can legitimately take longer than 30 seconds under load, the heartbeat interval math changes.
- Log lines should avoid logging `order_id` at INFO level if order IDs are considered sensitive identifiers in your compliance context. Check your data classification policy.
