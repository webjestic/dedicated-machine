Decision: Request Changes

Summary: The core distributed locking pattern is sound — atomic Lua scripts for release/extension, scoped heartbeat with stop_event, and NX+EX acquisition are all correctly implemented. However, there are several correctness, resilience, and test-coverage issues that need to be addressed before this merges, ranging from a silent exception-swallowing bug in the critical path to a race condition in the heartbeat thread and gaps in the test suite that would let real failure modes go undetected.

---

## Detailed Findings

### 1. 🔴 Critical — Unhandled exceptions inside the lock context silently succeed

**File:** `services/inventory.py`, `reserve_item`

The `with self._acquire_lock(lock_key):` block does not catch general exceptions. If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unexpected exception, it will propagate up through the context manager's `finally` block (which correctly releases the lock), but then bubble out of `reserve_item` entirely — **uncaught**. The caller receives an exception instead of a `ReservationResult`, breaking the contract the return type implies.

```python
# Current — exception escapes to caller
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # raises → uncaught
    ...
```

```python
# Required — wrap the lock body
try:
    with self._acquire_lock(lock_key):
        ...
except LockAcquisitionError as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 2. 🔴 Critical — TOCTOU gap between stock read and DB transaction

**File:** `services/inventory.py`, `reserve_item`

`get_stock` is called **outside** the database transaction. The lock prevents concurrent Python processes from entering this block simultaneously, but it does not protect against:

- A direct DB write bypassing the lock (admin tool, migration, another service)
- The DB read returning a stale value from a read replica with replication lag

The stock check and decrement must be atomic at the database level. The `get_stock` call should be inside the `with self.db.transaction():` block, and ideally the update should use a conditional `UPDATE ... WHERE stock >= quantity RETURNING stock` rather than a read-then-write pattern.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 3. 🟠 High — Heartbeat thread race: lock may be extended after release

**File:** `services/inventory.py`, `_acquire_lock`

The `finally` block calls `stop_event.set()` then `t.join(...)`. However, there is a window where the heartbeat thread has already passed the `stop_event.wait()` check, is mid-execution of `_extend_script`, and the main thread concurrently calls `_release_script`. The sequence becomes:

```
heartbeat:   extend_script(lock_key, token)  → extends TTL
main:        release_script(lock_key, token) → deletes key
heartbeat:   (returns, sees result != 0, logs "extended") ← stale log, harmless here
```

In isolation this is benign because the Lua token-match guards both scripts. But if Redis is slow and the heartbeat `_extend_script` call has not yet returned when `_release_script` fires, the release wins (correct), but then the extend response comes back and the heartbeat logs a false success. More importantly, if the join timeout expires before the thread finishes, `_release_script` runs while `_extend_script` is in-flight. The join timeout should be validated:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

Add this check — a zombie heartbeat thread is a resource leak and a signal that something is wrong with Redis connectivity.

### 4. 🟠 High — `quantity` is not validated as a reasonable upper bound

**File:** `services/inventory.py`, `ReservationRequest`

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound. A caller can submit `quantity=10_000_000` and the code will attempt to reserve ten million units. For a flash sale context this is almost certainly a logic error or abuse. Add a reasonable ceiling:

```python
quantity: int = Field(..., ge=1, le=1000)
```

The exact ceiling is a business decision, but the absence of any ceiling is a validation gap.

### 5. 🟠 High — `item_id` is reflected into log messages without sanitization

**File:** `services/inventory.py`, multiple log calls

```python
logger.info("Reserved item_id=%s ...", req.item_id, ...)
```

`req.item_id` passes the regex `^[a-zA-Z0-9_\-]+$`, so log injection via newlines or ANSI escapes is already blocked by the pattern — this is fine. However, the log line in the `not found` branch uses an f-string directly in the `message` field of `ReservationResult`, which is returned to the caller:

```python
message=f"Item {req.item_id} not found"
```

If this message is ever serialized into a response body (likely, given it's a `ReservationResult`), the item_id is reflected back. The Pydantic pattern guard makes this safe today, but this is a fragile dependency. Use a static message and log the id separately:

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

### 6. 🟡 Medium — Lock retry backoff is too aggressive for flash sale load

**File:** `services/inventory.py`, `_acquire_lock`

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait before failure is:
- Attempt 0 → 0.1s + jitter
- Attempt 1 → 0.2s + jitter
- Attempt 2 → 0.4s + jitter
- **Total: ~0.75s**

For a flash sale with high concurrency, 3 attempts over 750ms will cause a large percentage of legitimate requests to fail with "Service temporarily unavailable" during the spike. Consider either increasing `retry_attempts` or making it a constructor parameter so it can be tuned per deployment without a code change.

### 7. 🟡 Medium — `_acquire_lock` yields `token` but callers don't use it

**File:** `services/inventory.py`

```python
with self._acquire_lock(lock_key):   # token is discarded
```

The `yield token` in the context manager is unused at the call site. This is not a bug, but it signals the API is inconsistent. Either remove the yield value or use it — leaving it dangling will confuse future maintainers.

### 8. 🟡 Medium — Test suite gaps

**File:** `tests/test_inventory.py`

The following scenarios are not covered and represent real failure modes:

| Missing Test | Risk |
|---|---|
| `db.get_stock()` raises an exception | Verifies the unhandled-exception bug in finding #1 |
| `db.transaction()` raises mid-write | Ensures partial writes don't corrupt stock |
| Heartbeat thread terminates when lock is lost (extend returns 0) | Core heartbeat correctness |
| `reserve_item` called with `quantity` exceeding stock by exactly 1 | Off-by-one boundary |
| `item_id` at max length (128 chars) | Boundary validation |
| `retry_attempts=0` | Edge case in lock acquisition loop |

The existing `test_lock_release_uses_lua_script` test asserts `service._release_script.assert_called_once()` but `_release_script` is set on the instance during `__init__` from `register_script`. Since `register_script` returns a single `script_mock`, both `_release_script` and `_extend_script` point to the same mock object, meaning the call count assertion is unreliable — it will count extension calls too if the heartbeat fires during the test.
