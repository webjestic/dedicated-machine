Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent data-integrity race, and test gaps that would be unacceptable in a high-contention flash sale context where every reservation must be exact. The issues range from a stock-read/lock-acquire ordering inversion to an unhandled DB exception path that can leave the lock orphaned in a degraded state.

---

## Findings

### 🔴 Critical

**1. Stock is read *inside* the lock but `get_stock` is not part of the database transaction — TOCTOU within the transaction boundary**

```python
# services/inventory.py  lines ~90-107
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← read outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← write inside tx
        self.db.create_order(...)
```

`get_stock` is executed outside the database transaction. If the underlying database uses `READ COMMITTED` (the default for PostgreSQL and MySQL), a concurrent writer that commits between `get_stock` and `update_stock` will silently produce a negative or incorrect stock value. The Redis lock serializes *this service's* callers, but it offers zero protection against direct DB writes from other services, admin tooling, or a second deployment of this service that hasn't yet acquired the lock. The read **must** be inside the transaction, and the `update_stock` call should use an optimistic or pessimistic row-level lock (`SELECT ... FOR UPDATE`) rather than a blind overwrite.

```python
# Correct pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

**2. Unhandled exception inside the lock context leaks the lock state silently**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)   # ← what if this raises?
        self.db.create_order(...)
```

If `update_stock` or `create_order` raises any exception (DB timeout, constraint violation, network partition), it propagates out of `reserve_item` as an **unhandled exception** — not a `ReservationResult`. The caller gets an uncaught exception, no order is returned, but depending on transaction isolation the partial write may or may not have rolled back. The `_acquire_lock` `finally` block *will* release the Redis lock, which is correct, but the caller has no structured error to handle. A `try/except Exception` around the DB work should return a `ReservationResult(success=False, message="Reservation failed due to internal error")` and log the traceback.

---

### 🟠 High

**3. Heartbeat thread silently swallows lock loss — no propagation to the main thread**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; main thread has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat exits silently. The main thread continues executing the DB transaction believing it holds the lock, when in fact another process may have acquired it and is also writing stock. This is the most dangerous failure mode in the entire file. The heartbeat must signal the main thread. A `threading.Event` (e.g., `lock_lost_event`) should be set on `result == 0`, and the main thread must check it before committing.

```python
# Sketch
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()
            return

# In reserve_item, before the DB commit:
if lock_lost.is_set():
    raise LockLostError("Lock expired before commit; aborting transaction")
```

---

**4. `t.join(timeout=...)` does not verify the thread actually stopped**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=N)` returns whether or not the thread has exited. If the heartbeat thread is blocked on a slow Redis call, `join` returns after the timeout and the release script fires while the heartbeat thread may still be mid-`_extend_script`. This creates a race where the extend fires *after* the release, re-setting a TTL on a key that now belongs to a different owner (or no one). Check `t.is_alive()` after the join and log a critical alert if it is still running.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical("Heartbeat thread did not terminate before lock release key=%s", lock_key)
```

### 🟡 Medium

**5. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is reasonable, but the extension resets to the full TTL unconditionally**

The Lua extension script calls `expire(key, LOCK_TTL)` — always 30 seconds from *now*. If a legitimate operation takes 45 seconds (slow DB under load), the heartbeat fires at t=8, t=16, t=24, t=32, t=40 — each time resetting to 30s. This means the lock can be held indefinitely. There is no maximum hold time enforced. For a flash sale this is a denial-of-service vector: one stuck thread starves all other reservations for that item indefinitely. A maximum wall-clock hold time should be enforced.

---

**6. Retry backoff ceiling is extremely low under real contention**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~100ms, attempt=1 → ~200ms, attempt=2 → ~400ms
```

With `retry_attempts=3` the total wait is at most ~700ms. Under a flash sale with thousands of concurrent requests, 3 attempts over 700ms will produce a flood of `LockAcquisitionError` responses. The jitter ceiling of 50ms is also too narrow to spread load effectively. Consider increasing `retry_attempts` to at least 5–7 and widening the jitter to `random.uniform(0, 0.1 * 2**attempt)` (full jitter). Alternatively, expose these as constructor parameters so callers can tune per use-case.

---

**7. `update_stock` performs a blind overwrite rather than a delta update**

```python
new_stock = current_stock - req.quantity
self.db.update_stock(req.item_id, new_stock)
```

This writes an absolute value. If the DB layer does `UPDATE inventory SET stock = $1 WHERE item_id = $2`, any concurrent write that slipped past the Redis lock (see finding #1) will be silently overwritten. Prefer a delta: `UPDATE inventory SET stock = stock - $quantity WHERE item_id = $item_id AND stock >= $quantity` with a row count check. This makes the DB itself the last line of defense.

---

### 🔵 Low / Test Coverage Gaps

**8. No test for heartbeat thread lock-loss path**

`test_lock_release_uses_lua_script` verifies the release script is called, but there is no test that simulates `_extend_script` returning `0` and verifies the main thread is notified (once finding #3 is resolved). This is the most dangerous code path and it has zero coverage.

**9. No test for DB exception inside the lock**

There is no test where `update_stock` or `create_order` raises an exception to verify the lock is still released and the caller receives a structured `ReservationResult` rather than a raw exception.

**10. `test_lock_release_uses_lua_script` asserts on a shared mock**

```python
service._release_script.assert_called_once()
```

`_release_script` is the same mock object used for both release *and* extend (both registered via `register_script` which returns the same `script_mock`). This test does not actually distinguish release from extend calls and would pass even if the release script were never called. The mocks for release and extend scripts should be separate.
