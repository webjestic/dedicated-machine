Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, exponential backoff, and Pydantic validation are all solid choices. However, there are several correctness, resilience, and security issues that need to be addressed before this is production-ready for a high-stakes flash sale context. The most critical is a TOCTOU race on stock reads and a silent swallowing of database exceptions that could corrupt inventory state.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU: Stock read is outside the database transaction**

```python
# CURRENT — read happens BEFORE the transaction
current_stock = self.db.get_stock(req.item_id)  # ← outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write inside transaction
```

The Redis lock serializes callers *in this service*, but `get_stock` is not inside the database transaction. If any other process or service (a refund, an admin adjustment, a different service instance that bypassed the lock) modifies stock between the read and the write, you will silently oversell. The read and the conditional write must be atomic at the database level.

```python
# RECOMMENDED — optimistic lock or conditional update inside the transaction
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is the single most important fix. The Redis lock is a performance optimization (reducing DB contention), not a substitute for database-level consistency.

---

**2. Database exceptions are not caught — lock is released but state is unknown**

```python
with self._acquire_lock(lock_key):
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `update_stock` or `create_order` raises (network partition, DB timeout, constraint violation), the exception propagates out of `_acquire_lock`, the lock is released correctly, but the caller of `reserve_item` receives an unhandled exception rather than a `ReservationResult`. This breaks the contract of the public API and will cause upstream 500s.

```python
# Add inside the lock context
except (DatabaseError, SomeOtherExpectedException) as e:
    logger.error("DB failure during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 🟠 High

**3. Heartbeat thread failure is silent**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return  # ← lock was lost, but caller has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits silently. The critical section continues executing under the belief that it holds the lock, but the lock has expired and may have been acquired by another caller. This is a correctness violation — you now have two holders of the "same" lock simultaneously.

The heartbeat needs a way to signal the main thread that the lock was lost. A shared `threading.Event` or a shared flag checked at yield boundaries would work. At minimum, log at ERROR level, not just return silently.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat lost lock key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost_event` before and/or after the DB transaction.

---

**4. `t.join(timeout=...)` does not guarantee the thread has stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns whether or not the thread has finished. If the heartbeat thread is blocked on a slow Redis call, `join` returns after the timeout and the release script runs anyway — but the heartbeat thread is still alive and may call `_extend_script` *after* the lock has been released and re-acquired by another caller. This is a race condition.

```python
stop_event.set()
t.join()  # no timeout — block until the thread actually exits
self._release_script(keys=[lock_key], args=[token])
```

If you must have a timeout, at least check `t.is_alive()` and log a critical error if the thread did not stop.

### 🟡 Medium

**5. `retry_attempts` is not validated**

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

A caller passing `retry_attempts=0` will skip the loop entirely and raise `LockAcquisitionError` even if the lock is free, because `range(0)` is empty and `acquired` remains `False`. This is a subtle bug. Add a guard:

```python
if retry_attempts < 1:
    raise ValueError("retry_attempts must be at least 1")
```

---

**6. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. While the stock check will reject it, there is no domain-level cap. For a flash sale system, this should be bounded (e.g., `le=100` or a configurable max per item). This also prevents abuse where a single request monopolizes the lock while checking a trivially impossible quantity.

---

**7. Lock key uses user-supplied input directly**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated by the regex `^[a-zA-Z0-9_\-]+$`, which is good. However, this validation happens *before* the lock key is constructed, so the regex is the only defense. This is currently safe, but the dependency is implicit. A comment explicitly noting that the lock key safety depends on the Pydantic validation would prevent a future refactor from breaking this silently.

---

### 🔵 Low / Test Coverage Gaps

**8. No test for heartbeat lock-loss signaling**

The heartbeat's `result == 0` path (lock expired mid-operation) has no test coverage. This is the most dangerous failure mode and should be explicitly tested.

**9. No test for database transaction failure**

There is no test verifying behavior when `update_stock` or `create_order` raises. Given finding #2, this is also a missing coverage gap for a critical path.

**10. `test_lock_release_uses_lua_script` is fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. A future change that calls the script with the wrong key or token would pass this test. Prefer:

```python
service._release_script.assert_called_once_with(keys=[expected_key], args=[ANY])
```
