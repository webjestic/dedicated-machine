Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent data-integrity hazard, and test gaps that would make this dangerous to deploy under real flash-sale load. The issues range from a race window that can corrupt stock counts to a thread leak on DB exceptions.

---

## Critical Issues

### 1. Stock Read Outside the Database Transaction — TOCTOU Race

This is the most serious flaw. The stock check happens *before* the transaction opens:

```python
# OUTSIDE transaction
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():          # transaction opens HERE
    self.db.update_stock(...)        # write happens here
```

The Redis lock serializes callers *at the application layer*, but the database is not aware of the lock. If the DB read and the DB write are not inside the same transaction with a `SELECT ... FOR UPDATE` (or equivalent pessimistic/optimistic locking), a concurrent writer that bypasses this service (a cron job, a manual correction, a second service instance that lost the Redis lock but still has a DB connection) can modify stock between your read and your write. The lock only helps if *every* writer in the entire system respects it — that is rarely a safe assumption in a government-grade system.

**Required fix:** Move `get_stock` inside the transaction and use a row-level lock:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None: ...
    if current_stock < req.quantity: ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

---

### 2. Heartbeat Thread Leak on Database Exception

If `self.db.get_stock(...)` or anything inside the `with self.db.transaction()` block raises an uncaught exception, the `finally` block in `_acquire_lock` *does* run — that part is correct. However, `reserve_item` does not catch general exceptions from inside the lock context:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # raises, e.g., DB connection error
    # ... exception propagates out of the `with` block
    # _acquire_lock's finally runs, good
    # BUT reserve_item has no except for this — exception bubbles to caller
```

The lock *will* be released (the `finally` is correct), but the caller receives an unhandled exception rather than a `ReservationResult`. For a service API this means an unstructured 500 rather than a graceful degraded response. In a government system, unhandled exceptions leaking stack traces to clients is a security concern.

**Required fix:**

```python
except (LockAcquisitionError, Exception) as e:
    # differentiate and return appropriate ReservationResult
    logger.error("Unexpected error during reservation item_id=%s", req.item_id, exc_info=True)
    return ReservationResult(success=False, message="Internal error during reservation")
```

---

### 3. Heartbeat Silently Swallows Lock Loss

When the heartbeat detects the lock was lost (`result == 0`), it simply returns:

```python
if result == 0:
    return   # ← silent exit
logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread has no way to know the lock was lost. It will continue executing the DB transaction *without holding the lock*, which is exactly the scenario the lock was designed to prevent. Another process may have acquired the lock and begun its own transaction. You now have two writers.

**Required fix:** Use a shared flag or event to signal lock loss to the main thread, and check it before committing:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then in the main thread, check `lock_lost_event.is_set()` before committing the transaction.

## Moderate Issues

### 4. `update_stock` Uses Absolute Value — No DB-Level Guard Against Negative Stock

```python
self.db.update_stock(req.item_id, new_stock)
```

This is a blind `UPDATE stock SET quantity = <value>`. If two reservations somehow both read `stock = 1` (e.g., due to the TOCTOU issue above, or a bug), both could write `stock = 0` — but the second write is still "valid" from the DB's perspective. A safer pattern is a conditional decrement:

```sql
UPDATE inventory SET stock = stock - :qty
WHERE item_id = :id AND stock >= :qty
```

Then check `rowcount == 1` to confirm the update succeeded. This makes the DB itself the final arbiter of correctness, not the application layer.

---

### 5. Retry Backoff Is Too Aggressive for Flash Sale Workloads

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, worst-case wait before giving up is `0.4s + 0.2s + 0.05s ≈ 0.65s`. Under a flash sale with thousands of concurrent requests, this means threads pile up waiting. The jitter range (`0–0.05s`) is also very narrow, which reduces its thundering-herd mitigation value. Consider full jitter (`random.uniform(0, cap)`) per the AWS backoff paper, and expose `retry_attempts` as a constructor parameter so it can be tuned per environment.

---

### 6. `LOCK_TTL = 30s` Is Dangerously Long for a Flash Sale

A 30-second lock on a hot inventory item means one slow or hung DB operation can block *all* other reservations for that item for up to 30 seconds. The heartbeat mitigates expiry during legitimate work, but 30 seconds is the *failure* TTL — the time a dead process holds the lock. This should be much shorter (3–5 seconds), with the heartbeat keeping it alive for legitimate long operations.

## Test Coverage Gaps

### 7. No Test for Lock-Loss Mid-Transaction

There is no test verifying behavior when the heartbeat detects `result == 0` (lock lost). This is the most dangerous failure mode and it has zero coverage.

### 8. No Test for DB Exception Inside Lock Context

No test covers `db.get_stock` or `db.update_stock` raising an exception. Given issue #2 above, this path is untested and broken.

### 9. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not verify it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Assert the full call signature.

### 10. Heartbeat Thread Is Never Exercised in Tests

All tests mock `redis.set` to return immediately. The heartbeat thread is spawned but the tests never simulate a long-running operation or verify the extend script is called. The heartbeat logic has zero functional test coverage.

---

## Minor Issues

### 11. `daemon=True` Thread Could Be Killed Mid-Extension

Marking the heartbeat thread as `daemon=True` means if the main process exits while the heartbeat is mid-Lua-script execution, the thread is killed without cleanup. This is acceptable for most cases but worth documenting explicitly, as it means the lock *may* not be extended on process shutdown even if the operation is still in flight.

### 12. Logger Name Should Be Module-Scoped (Already Correct) — But Log Level for Contention Is Wrong

```python
logger.warning("Lock contention key=%s attempt=%d ...", ...)
```

Lock contention during a flash sale is *expected and normal*. Logging it at `WARNING` will flood monitoring dashboards with false positives. This should be `DEBUG` or `INFO`.
