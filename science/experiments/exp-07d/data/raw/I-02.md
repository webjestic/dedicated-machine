Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a race condition, a silent failure path, and test gaps that would make this unsafe to deploy in a high-concurrency flash sale context. The issues range from a critical TOCTOU stock-read vulnerability to an unhandled exception that silently drops the lock without releasing it.

---

## Detailed Findings

### 🔴 Critical — TOCTOU: Stock Read Outside the Database Transaction

```python
# Line ~90 in reserve_item
current_stock = self.db.get_stock(req.item_id)  # ← outside transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are **not in the same database transaction**. The Redis lock serializes callers through this service, but it does not protect against:

- A separate service or admin process modifying stock between the read and the write.
- A Redis failover or lock expiry edge case that allows a second holder to enter before this write commits.
- Any future refactor that removes or bypasses the Redis lock.

The correct pattern is to move the read inside the transaction and use a database-level `SELECT ... FOR UPDATE` (or equivalent pessimistic lock), making the Redis lock a performance optimization rather than the sole correctness guarantee.

```python
# Recommended
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return/rollback
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 🔴 Critical — Unhandled Exceptions Silently Suppress Lock Release Failures

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])  # ← no exception handling
```

If `_release_script` raises (Redis connection drop, timeout, serialization error), the exception propagates out of the `finally` block and **masks the original exception** that triggered the `finally`. The caller receives a confusing error, and depending on the call stack, the original business logic error may be lost entirely.

Additionally, the return value of `_release_script` is never checked. A return value of `0` means the token did not match and the lock was **not released** — this should be logged as an error, not silently ignored.

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error("Lock release failed — token mismatch or lock expired key=%s", lock_key)
    except redis.RedisError:
        logger.exception("Redis error during lock release key=%s", lock_key)
```

### 🟠 High — Heartbeat Thread Leak on Join Timeout

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
# execution continues regardless of whether t has actually stopped
self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns after the timeout **whether or not the thread has exited**. If the heartbeat thread is blocked on a slow Redis call, it may still be running when `_release_script` executes. This creates a race:

1. `_release_script` deletes the key.
2. A new lock holder acquires the same key with a new token.
3. The still-running heartbeat thread calls `_extend_script` with the **old token** — this will correctly return `0` due to the Lua token check, but only because of a defense-in-depth property, not by design.

The deeper issue is that a leaked thread holding a reference to `token` and `lock_key` is a resource leak and a latent correctness hazard. The join timeout should be treated as a hard error:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — potential thread leak", lock_key)
    # Do not proceed silently; alert or raise
```

### 🟠 High — Heartbeat Failure Is Silently Swallowed

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← lock was lost; caller has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat silently exits. The main thread continues executing the database transaction under the assumption that it still holds the lock — but it does not. Another process may now acquire the lock and begin modifying the same inventory record concurrently.

The heartbeat needs a signaling mechanism back to the main thread:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return
```

And the main thread should check `lock_lost_event` before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock expired during operation for {lock_key}")
```

### 🟡 Medium — Retry Backoff Is Miscalibrated for Flash Sale Load

```python
for attempt in range(retry_attempts):  # default: 3 attempts
    ...
    backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait time before failure is approximately `0.1 + 0.2 + 0.05 ≈ 0.35 seconds` (ignoring jitter). Under a flash sale with hundreds of concurrent requests, this means the vast majority of requests will fail immediately with "Service temporarily unavailable" rather than queuing behind the lock holder.

The jitter range `uniform(0, 0.05)` is also very narrow — under high concurrency, many threads will compute nearly identical backoffs and create a retry thundering herd.

Consider a wider jitter range (`uniform(0, backoff * 0.5)`) and a configurable retry count with a longer ceiling, or a proper queue-based approach for flash sale scenarios.

### 🟡 Medium — `db` Interface Is Entirely Implicit

The `InventoryService.__init__` accepts `db` as an untyped parameter with no interface definition. The methods `get_stock`, `update_stock`, `create_order`, and `transaction` are called without any contract. This means:

- Type checkers cannot validate usage.
- Incorrect `db` implementations will fail at runtime inside a lock, potentially causing the lock to be held until TTL expiry.
- There is no documentation of what `transaction()` is expected to do on rollback.

Define a `Protocol` or `ABC`:

```python
from typing import Protocol, ContextManager

class InventoryDB(Protocol):
    def get_stock(self, item_id: str) -> Optional[int]: ...
    def get_stock_for_update(self, item_id: str) -> Optional[int]: ...
    def update_stock(self, item_id: str, new_stock: int) -> None: ...
    def create_order(self, order_id: str, item_id: str, quantity: int) -> None: ...
    def transaction(self) -> ContextManager[None]: ...
```

### 🟡 Medium — `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity constraint. A single request could attempt to reserve `2^63 - 1` units. While the stock check would catch it, this still executes a lock acquisition, a DB read, and a comparison before rejecting. In a flash sale context, this is a trivial denial-of-service vector.

```python
quantity: int = Field(..., ge=1, le=1000)  # or a configurable MAX_QUANTITY
```

### 🟡 Medium — `item_id` Leaked Into Error Messages Returned to Callers

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Embedding internal identifiers in API-facing messages is an information disclosure pattern. In a government contracting context this is particularly relevant — item IDs may correlate to procurement records, contract line items, or other sensitive identifiers. Return a generic message to the caller and log the detail server-side.

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```
