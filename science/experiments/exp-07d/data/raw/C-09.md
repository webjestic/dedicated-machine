Decision: Request Changes

Summary: The core locking primitive and heartbeat design are directionally sound — atomic Lua scripts, scoped stop events, and NX/EX acquisition are all correct. However, there are several concrete failure modes ranging from a silent data-integrity race to a thread-leak vector to missing operational safeguards that must be addressed before this merges into a flash-sale-grade system. Details below.

---

## Critical Issues

### 1. Stock read is outside the database transaction — TOCTOU race

This is the most serious flaw. The stock check and the write are not atomic at the database level:

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)   # READ
if current_stock < req.quantity:
    ...
new_stock = current_stock - req.quantity

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE
    self.db.create_order(...)
```

The Redis lock serializes callers *across processes*, but it does nothing to protect against:

- A direct DB write from a non-lock-aware code path (admin tool, migration, another service).
- A future refactor that moves the lock or bypasses it.
- The DB read returning a stale snapshot if the connection uses `READ COMMITTED` or weaker isolation and a concurrent transaction committed between the `get_stock` call and the `update_stock` call.

**Required fix:** Move `get_stock` inside the transaction, and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) so the read and write are atomic at the database level. The Redis lock is a performance optimization (coarse-grained early exit); it must not be the sole correctness guarantee.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ...
    if current_stock < req.quantity:
        return ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Heartbeat thread is leaked on `LockAcquisitionError` — not a real leak here, but the pattern is fragile

Actually, re-reading: the thread is only started *after* acquisition succeeds, so a `LockAcquisitionError` does not leak a thread. That part is fine. **However**, there is a real leak vector:

```python
t = threading.Thread(target=_beat, daemon=True)
t.start()

try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← timeout is 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If `t.join()` times out (e.g., the thread is blocked on a slow Redis call inside `_extend_script`), execution falls through to `_release_script` anyway — but the heartbeat thread is still running. It will then race against the release: the thread may call `_extend_script` *after* `_release_script` has deleted the key, which is benign in isolation, but if another caller has already re-acquired the lock with a new token, the extend script will correctly return 0 (token mismatch) and do nothing — so this is safe *only because* the Lua script checks the token. Still, a thread that outlives its context manager is a resource leak and a correctness smell. The join timeout should be documented, and a warning must be logged if it is exceeded:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate for key=%s; lock may be in inconsistent state", lock_key)
```

### 3. Unhandled exceptions inside the lock context silently release the lock without rolling back

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `db.get_stock`, `db.update_stock`, or `db.create_order` raises an unexpected exception (network error, ORM error, etc.), the `contextmanager` `finally` block releases the lock — correct — but the exception propagates out of `reserve_item` uncaught. The caller receives an unhandled exception rather than a `ReservationResult`. For a service method that promises to return a `ReservationResult`, this is a broken contract. Add a top-level `except Exception` with logging and a safe fallback return.

## Significant Issues

### 4. Backoff ceiling is missing — retry storm risk

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With only 3 attempts the values are `0.1s`, `0.2s`, `0.4s` — harmless today. But `retry_attempts` is a parameter. A caller passing `retry_attempts=20` produces a backoff of `~52 seconds` on the last attempt with no cap. Add `min(backoff, MAX_BACKOFF)` and document the parameter's safe range, or remove the parameter and hardcode 3.

---

### 5. `quantity` has no upper bound — inventory exhaustion / integer overflow

```python
quantity: int = Field(..., ge=1)
```

A caller can pass `quantity=2**31` and the stock check will fail gracefully, but the value is logged and stored in the order record. More importantly, there is no business-logic ceiling. For a flash-sale system, a per-request quantity cap (e.g., `le=10`) should be enforced at the model level, not left to the caller.

---

### 6. `item_id` length allows Redis key bloat

```python
item_id: str = Field(..., min_length=1, max_length=128, ...)
```

The lock key becomes `inventory:lock:<item_id>` — up to 144 bytes. This is not a security vulnerability, but 128 characters is unusually generous for a product identifier. Tighten to a realistic maximum (e.g., 64) and document the rationale.

---

### 7. Redis connection errors are not distinguished from lock contention

If `self.redis.set(...)` raises a `redis.RedisError` (connection refused, timeout, auth failure), the exception propagates out of `_acquire_lock` uncaught and surfaces as an unhandled exception in `reserve_item` rather than a `LockAcquisitionError`. The caller cannot distinguish "lock held by someone else" from "Redis is down." Wrap the `set` call:

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.RedisError as e:
    raise LockAcquisitionError(f"Redis error acquiring lock: {e}") from e
```

Similarly, `_extend_script` inside the heartbeat thread can raise — currently silently swallowed by the daemon thread dying, which means the lock expires and another node can acquire it mid-operation.

## Test Coverage Gaps

### 8. No test for the TOCTOU path (stock changes between read and write)

There is no test that verifies behavior when stock changes between `get_stock` and `update_stock`. This is the most important invariant to test and it is absent.

### 9. No test for Redis failure during lock acquisition

`test_reserve_lock_contention` only covers `set` returning `False`. There is no test for `redis.RedisError` being raised.

### 10. Heartbeat thread behavior is untested

The heartbeat is the most complex and failure-prone component. There are no tests verifying that:
- The extend script is called when the lock is held for longer than `HEARTBEAT_INTERVAL`.
- The heartbeat stops after `stop_event.set()`.
- A lost lock (extend returns 0) is detected and surfaced.

### 11. `test_lock_release_uses_lua_script` is testing the mock, not the behavior

```python
service._release_script.assert_called_once()
```

`_release_script` is the mock returned by `register_script`. This test passes trivially because the mock is always callable. It does not verify the key or token arguments passed to the script. At minimum assert `called_once_with(keys=[...], args=[...])`.

---

## Minor / Nits

### 12. `daemon=True` on the heartbeat thread

Daemon threads are killed abruptly on interpreter shutdown. If the process exits while a reservation is in-flight, the heartbeat dies, the lock expires, and another node can acquire it before the DB transaction commits. This is an inherent limitation of daemon threads for critical paths. Consider making it non-daemon and ensuring the join in `finally` is always reached, or document this known limitation explicitly.

### 13. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants, not configurable

For a deployed service these should be injectable (constructor params with defaults) so they can be tuned per environment without a code change.

### 14. Log message on lock release fires even if release script returns 0

```python
self._release_script(keys=[lock_key], args=[token])
logger.info("Lock released key=%s", lock_key)
```

If the lock was lost (token mismatch, TTL expired despite heartbeat), the script returns 0 and the log still says "Lock released." Check the return value and log a warning on mismatch.
