Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical TOCTOU race, architectural concerns around thread safety and failure handling, and test gaps that must be addressed before this is suitable for a high-concurrency flash sale context.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and Database Write

This is the most serious flaw in the design. The lock is held correctly, but the stock check and the write are not atomic at the database level:

```python
# Under the lock — correct
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

new_stock = current_stock - req.quantity

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # Overwrites whatever is in DB now
    self.db.create_order(order_id, req.item_id, req.quantity)
```

`update_stock` is called with a pre-computed absolute value (`new_stock`). If anything modifies stock between the read and the write — a manual admin correction, a different code path that bypasses this lock, a replication lag on a read replica, or a future developer adding a second write path — this will silently overwrite that change. The write should be a conditional decrement at the database level:

```sql
UPDATE inventory SET stock = stock - :qty
WHERE item_id = :item_id AND stock >= :qty
```

Then check affected rows. The Redis lock alone is insufficient as a correctness guarantee if the database is the system of record.

### 2. Heartbeat Thread Silently Swallows Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return  # Lock was lost — exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread exits, but the main thread has no way to know the lock was lost. The main thread continues executing the database transaction without holding the lock. This defeats the entire purpose of the locking mechanism. The lock loss must be communicated back to the main thread and the operation must be aborted.

A threading.Event or similar signaling mechanism is required:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            return
```

The main thread must check `lock_lost_event` before and ideally during the DB transaction, and abort with an appropriate error if it is set.

---

### 3. Lock Released After DB Transaction Regardless of Lock Loss

Related to the above: the `finally` block always calls `_release_script`. If the lock was lost (expired and potentially acquired by another process), this script will correctly do nothing (token mismatch), but the code has already committed a database transaction without holding the lock. The release call itself is fine; the problem is the absence of any abort path when lock loss is detected.

## Significant Issues

### 4. Heartbeat Interval Is Too Close to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

The comment says "fires at TTL/3" which is correct for the first beat. But the heartbeat resets the TTL to the full 30 seconds on each extension. This means the effective minimum lock lifetime is always `LOCK_TTL - HEARTBEAT_INTERVAL = 22s` at worst. This is acceptable in isolation, but the real concern is clock skew and Redis latency. If the Redis call to extend takes longer than expected (network partition, Redis slowdown), the lock may expire before the extension completes. The interval should be configurable and documented with explicit reasoning about the safety margin.

### 5. Thread Join Timeout Is a Silent Failure

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within the join timeout, execution continues and the lock is released anyway. There is no log, no exception, no indication that the thread is still running. A daemon thread that outlives the context manager and then attempts to extend a lock that has already been released (or re-acquired by another holder) is a correctness hazard. At minimum, log a warning. Consider raising if the thread fails to join.

### 6. Exponential Backoff Maximum Is Extremely Low

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`:
- Attempt 0: ~0.10s
- Attempt 1: ~0.20s
- Attempt 2: ~0.40s

Total maximum wait: ~0.75 seconds. For a 30-second lock TTL, this retry window is negligible. Under genuine flash sale load, callers will fail almost immediately if the lock is held. Either increase `retry_attempts`, cap the backoff at something reasonable (e.g., 2–5s), or make these parameters configurable per call site.

### 7. `db.get_stock` Is Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)  # Outside transaction

with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(...)
```

The read is not part of the same transaction as the write. Depending on the database isolation level, this read may be stale. Move the read inside the transaction, or use a `SELECT FOR UPDATE` pattern to ensure the read and write are consistent at the DB layer.

## Moderate Issues

### 8. No Exception Handling for Redis Failures

Neither `_acquire_lock` nor `reserve_item` handles `redis.RedisError`. A Redis connection failure during lock acquisition, heartbeat extension, or lock release will propagate as an unhandled exception. In a flash sale context this will surface as a 500 to the caller. Catch `redis.RedisError` explicitly and return an appropriate `ReservationResult` or re-raise as a domain exception.

### 9. No Exception Handling for Database Failures Inside the Lock

If `self.db.update_stock` or `self.db.create_order` raises, the exception propagates out of the `with self.db.transaction()` block and then out of the `with self._acquire_lock()` block. The `finally` clause will release the lock correctly, but the caller receives an unhandled exception rather than a `ReservationResult`. This should be caught and returned as a structured failure.

### 10. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum. A caller can request `quantity=10_000_000`. This will pass validation, acquire the lock, read stock, and attempt a write. Add a reasonable upper bound (`le=1000` or configurable) appropriate to the business domain.

## Test Gaps

### 11. Heartbeat Lock Loss Is Not Tested

There is no test that simulates `_extend_script` returning `0` and verifies that the main thread aborts the reservation. This is the most critical failure mode and has zero coverage.

### 12. Thread Join Timeout Path Is Not Tested

No test covers the scenario where the heartbeat thread fails to terminate within the join timeout.

### 13. Redis Failure Paths Are Not Tested

No tests cover `redis.RedisError` during lock acquisition, extension, or release.

### 14. Database Exception Inside Transaction Is Not Tested

No test covers `db.update_stock` or `db.create_order` raising an exception, and verifies the caller receives a structured result rather than an unhandled exception.

### 15. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert the arguments. A future refactor could call the script with wrong keys or a wrong token and this test would still pass. Assert the specific `keys` and `args` values.

---

## Minor Issues

- **No `__all__` or module-level exports defined.** For a service module in a larger codebase, explicit exports reduce accidental coupling.
- **`LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants.** For a government contracting context where operational tuning is expected, these should be constructor parameters with documented defaults, not hardcoded class attributes.
- **`logger = logging.getLogger(__name__)` at module level is correct**, but there is no guidance on log level configuration. In production, `DEBUG`-level heartbeat logs will be noisy under load.
- **`order_id` is generated before the transaction.** If the transaction fails and is retried, a new `order_id` will be generated on the next call. This is acceptable but should be documented as intentional.
