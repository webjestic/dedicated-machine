# Rate Limiter Implementation

## 1. Requirement Checklist

Derived from the design document before any code is written.

### §1 Overview / Assumptions
- [ ] **R-01** Middleware library interface (not standalone service — see Open Items)
- [ ] **R-02** Per-identifier rate limiting
- [ ] **R-03** Redis-backed (does not own the connection)
- [ ] **R-04** Identifier is passed in / extracted by caller-supplied function
- [ ] **R-05** `ioredis` client with `lazyConnect: true` and `enableOfflineQueue: false`

### §2 Algorithm
- [ ] **R-06** Approximated sliding window counter algorithm
- [ ] **R-07** `current_window = floor(now / window_size)`
- [ ] **R-08** `previous_window = current_window - 1`
- [ ] **R-09** `elapsed_fraction = (now % window_size) / window_size`
- [ ] **R-10** `weighted_count = previous_count * (1 - elapsed_fraction) + current_count`
- [ ] **R-11** Reject if `weighted_count >= limit`
- [ ] **R-12** Allow and increment if `weighted_count < limit`
- [ ] **R-13** Lua script executed atomically via `EVALSHA` with `EVAL` fallback
- [ ] **R-14** Lua script body matches normative specification exactly

### §3 Storage
- [ ] **R-15** Key format: `ratelimit:{namespace}:{identifier}:{window_id}`
- [ ] **R-16** `window_id = Math.floor(Date.now() / 1000 / windowSizeSeconds)`
- [ ] **R-17** TTL set to `window_size * 2` on first write only (`new_count == 1`)
- [ ] **R-18** TTL not reset on subsequent writes

### §4 Failure Modes
- [ ] **R-19** F-01: Redis connection unavailable — fail-open or fail-closed per D-01
- [ ] **R-20** F-01: Log WARN at most once per second (rate-limited logging)
- [ ] **R-21** F-01: Emit `ratelimit.redis.unavailable` metric
- [ ] **R-22** F-02: Redis command timeout enforced client-side at 100ms
- [ ] **R-23** F-02: Emit `ratelimit.redis.timeout` metric (distinct from unavailable)
- [ ] **R-24** F-03: Log ERROR with raw Redis response; do not crash; do not expose internals
- [ ] **R-25** F-06: `identifierFn` returning null → fail-open for that request, log ERROR, no silent fallback
- [ ] **R-26** F-06: Emit `ratelimit.identifier.missing` metric
- [ ] **R-27** F-07: Catch `NOSCRIPT`, fall back to `EVAL`, re-cache SHA
- [ ] **R-28** F-07: Emit `ratelimit.redis.script_miss` metric
- [ ] **R-29** F-01/F-02/F-03: Fail-closed path returns 503 with `Retry-After: 10`

### §5 API Contract
- [ ] **R-30** `rateLimiter(options)` returns Express middleware `(req, res, next)`
- [ ] **R-31** Options: `namespace`, `limit`, `windowSizeSeconds`, `identifierFn`, `redisClient`, `onLimitExceeded` (optional), `failOpen` (required, no default)
- [ ] **R-32** Allowed request: call `next()`, set `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `X-RateLimit-Window` headers
- [ ] **R-33** `X-RateLimit-Remaining` floored at 0
- [ ] **R-34** `X-RateLimit-Reset = (current_window + 1) * window_size`
- [ ] **R-35** Rejected request: HTTP 429 with `Retry-After`, `X-RateLimit-*` headers, JSON body
- [ ] **R-36** `Retry-After = ceil((current_window + 1) * window_size - now)`, minimum 1
- [ ] **R-37** `retryAfter` in body matches `Retry-After` header exactly
- [ ] **R-38** 429 body: `error`, `message`, `retryAfter`, `limit`, `window` fields
- [ ] **R-39** 429 body must not contain identifier, Redis keys, stack traces
- [ ] **R-40** 503 body on fail-closed Redis failure: `error`, `message`, `retryAfter` fields
- [ ] **R-41** `onLimitExceeded` override invoked instead of default 429 handler when provided

### §6.1 Observability — Metrics
- [ ] **R-42** `ratelimit.request.total` counter with `namespace`, `result` tags
- [ ] **R-43** `ratelimit.request.allowed` counter with `namespace` tag
- [ ] **R-44** `ratelimit.request.rejected` counter with `namespace` tag
- [ ] **R-45** `ratelimit.redis.latency_ms` histogram with `namespace` tag
- [ ] **R-46** `ratelimit.redis.error` counter with `namespace`, `error_type` tags
- [ ] **R-47** `ratelimit.redis.script_miss` counter with `namespace` tag
- [ ] **R-48** `ratelimit.failopen.request` counter with `namespace` tag
- [ ] **R-49** `ratelimit.identifier.missing` counter with `namespace` tag
- [ ] **R-50** Metrics emitted on every request (not only rejection)

### §6.1 Observability — Structured Logging
- [ ] **R-51** All log events are structured JSON
- [ ] **R-52** Required log fields: `timestamp`, `level`, `component`, `namespace`, `identifier`, `result`, `weighted_count`, `limit`, `window_id`, `redis_latency_ms`, `request_id`
- [ ] **R-53** Rejections and errors always logged regardless of volume configuration

### §6.2 Graceful Degradation
- [ ] **R-54** Rate limiter must never crash the Node.js process
- [ ] **R-55** All Redis errors caught within middleware
- [ ] **R-56** No unhandled promise rejections from Redis operations
- [ ] **R-57** Fail-open behavior must be explicit and logged (not silent)
- [ ] **R-58** Recovery from Redis outage automatic (no process restart required)

### §6.4 Health Check
- [ ] **R-59** Export `isAlive()` — returns `true` if process running and middleware initialized
- [ ] **R-60** Export async `isReady()` — Redis PING with 200ms timeout
- [ ] **R-61** `isReady()` fail-open: 200 with `{"status":"ready","redis":"degraded","mode":"fail-open"}`
- [ ] **R-62** `isReady()` fail-closed: 503 with `{"status":"not_ready","redis":"unavailable"}`

### §7 Decision Points (from references throughout document)
- [ ] **R-63** D-01: fail-open vs. fail-closed — explicit `failOpen` boolean, no default
- [ ] **R-64** D-03: IP extraction from `X-Forwarded-For` — implementing engineer must audit; safer default documented
- [ ] **R-65** D-05: Metrics backend — abstract emit interface; concrete backend is a decision point

### Section 2.3 — `src/luaScript.js`

**Design reference:** §2, "Lua script (normative)". The design states: *"The implementing engineer must not alter the logic of this script."* The script body below is a verbatim transcription of the normative specification.

```javascript
'use strict';

// §2 — Normative Lua script for atomic sliding window counter.
// DO NOT ALTER THE LOGIC OF THIS SCRIPT without re-running the load test
// suite defined in §6.5. The design document (§2) explicitly prohibits
// logic changes without load test validation.
//
// Script contract:
//   KEYS[1] = current window key
//   KEYS[2] = previous window key
//   ARGV[1] = limit (number)
//   ARGV[2] = window_size in seconds (number)
//   ARGV[3] = now in seconds (number, may be fractional)
//   ARGV[4] = elapsed_fraction (number)
//
// Return value (array):
//   [0]: 1 = allowed, 0 = rejected
//   [1]: current window count after increment (or before, if rejected)
//   [2]: previous window count
//   [3]: weighted count used for decision
const RATE_LIMIT_SCRIPT = `
local current_key   = KEYS[1]
local previous_key  = KEYS[2]
local limit         = tonumber(ARGV[1])
local window_size   = tonumber(ARGV[2])
local now           = tonumber(ARGV[3])
local elapsed_frac  = tonumber(ARGV[4])

local current_count  = tonumber(redis.call('GET', current_key) or 0)
local previous_count = tonumber(redis.call('GET', previous_key) or 0)
local weighted       = previous_count * (1 - elapsed_frac) + current_count

if weighted >= limit then
  return {0, current_count, previous_count, weighted}
end

local new_count = redis.call('INCR', current_key)
if new_count == 1 then
  redis.call('EXPIRE', current_key, window_size * 2)
end

return {1, new_count, previous_count, weighted}
`;

module.exports = { RATE_LIMIT_SCRIPT };
```

**Checklist satisfied:** R-14

### Section 2.4 — `src/metrics.js`

**Design reference:** §6.1 (metric names and semantics are normative), §7 D-05 (metrics backend is a decision point).

**Decision Point D-05 — Metrics Backend:**
> The design specifies metric names and semantics but leaves the backend (StatsD, Prometheus, etc.) as an explicit decision point. The implementation provides an abstract `emit` interface. The caller must inject a concrete implementation. See Decision Point Resolutions section.

```javascript
'use strict';

// §6.1 — Metrics interface.
//
// DECISION POINT D-05: The metrics backend (StatsD, Prometheus client, etc.)
// is unresolved per §7 of the design document. This module defines the
// abstract interface that the rate limiter calls. The caller must supply a
// concrete `metricsEmitter` object at initialization time (see rateLimiter.js
// options). If none is supplied, a no-op emitter is used and a warning is
// logged at startup.
//
// The no-op default is the SAFER default: the rate limiter functions correctly
// without a metrics backend. Silent metric loss is preferable to a startup
// crash caused by a missing metrics dependency. However, operating without
// metrics means the degraded-state detection thresholds in §6.1 cannot be
// evaluated. The operator must supply a real emitter before production.
//
// To wire in a real backend, supply a `metricsEmitter` object with these
// methods:
//   increment(metricName, tags)   — increments a counter
//   histogram(metricName, value, tags) — records a histogram observation
//
// Example StatsD adapter:
//   { increment: (name, tags) => statsd.increment(name, 1, tagsToArray(tags)),
//     histogram: (name, val, tags) => statsd.histogram(name, val, tagsToArray(tags)) }

/**
 * Creates a metrics emitter bound to a namespace.
 * All metric names and tag keys match the normative specification in §6.1.
 *
 * @param {object} emitter - Object with `increment(name, tags)` and
 *   `histogram(name, value, tags)` methods. Tags is a plain object.
 * @param {string} namespace - Rate limiter namespace for tagging.
 * @returns {object} Bound metrics helper.
 */
function createMetrics(emitter, namespace) {
  // §6.1 — Metric names are normative. Do not rename them.
  const ns = namespace;

  function increment(metricName, extraTags = {}) {
    try {
      emitter.increment(metricName, { namespace: ns, ...extraTags });
    } catch (_err) {
      // Metrics emission must never crash the rate limiter (§6.2).
      // Errors here are silently swallowed. If the emitter is broken,
      // that is an infrastructure concern, not a rate limiter concern.
    }
  }

  function histogram(metricName, value, extraTags = {}) {
    try {
      emitter.histogram(metricName, value, { namespace: ns, ...extraTags });
    } catch (_err) {
      // Same rationale as above.
    }
  }

  return {
    // §6.1 — ratelimit.request.total: every request evaluated.
    requestTotal(result) {
      increment('ratelimit.request.total', { result });
    },

    // §6.1 — ratelimit.request.allowed
    requestAllowed() {
      increment('ratelimit.request.allowed');
    },

    // §6.1 — ratelimit.request.rejected
    requestRejected() {
      increment('ratelimit.request.rejected');
    },

    // §6.1 — ratelimit.redis.latency_ms histogram
    // Buckets specified in §6.1: 1, 5, 10, 25, 50, 100, 250, 500.
    // Bucket configuration is the responsibility of the metrics backend
    // adapter (D-05). This call emits the raw value.
    redisLatency(ms) {
      histogram('ratelimit.redis.latency_ms', ms);
    },

    // §6.1 — ratelimit.redis.error
    // error_type values: 'timeout' | 'unavailable' | 'unexpected'
    redisError(errorType) {
      increment('ratelimit.redis.error', { error_type: errorType });
    },

    // §6.1 — ratelimit.redis.script_miss (F-07 NOSCRIPT recovery)
    redisScriptMiss() {
      increment('ratelimit.redis.script_miss');
    },

    // §6.1 — ratelimit.failopen.request
    failOpenRequest() {
      increment('ratelimit.failopen.request');
    },

    // §6.1 — ratelimit.identifier.missing (F-06)
    identifierMissing() {
      increment('ratelimit.identifier.missing');
    },
  };
}

/**
 * No-op emitter used when no metrics backend is supplied.
 * See DECISION POINT D-05 comment above.
 */
const NO_OP_EMITTER = {
  increment() {},
  histogram() {},
};

module.exports = { createMetrics, NO_OP_EMITTER };
```

**Checklist satisfied:** R-42, R-43, R-44, R-45, R-46, R-47, R-48, R-49, R-50 (emission calls present; R-50 is enforced by call sites in rateLimiter.js)

### Section 2.5 — `src/logger.js`

**Design reference:** §6.1 Structured Logging. All log events must be structured JSON. Required fields are normative.

```javascript
'use strict';

// §6.1 — Structured logging.
//
// All log events emitted by the rate limiter are structured JSON written to
// stdout. The logging pipeline (log shipper, aggregator) is responsible for
// routing and storage. PII handling (hashing/masking of identifiers) is an
// infrastructure concern per §6.1: "This is an infrastructure concern, not a
// rate limiter concern, but the implementing engineer must flag it to the
// operator."
//
// ⚠️  OPERATOR NOTICE: The `identifier` field is logged on every event.
// If identifiers contain PII (email addresses, phone numbers, etc.), the
// logging pipeline MUST hash or mask this field before storage. See §6.1.
//
// Log volume: §6.1 notes that INFO-level logging of every allowed request
// will be high-volume. See DECISION POINT in rateLimiter.js regarding
// allowed-request log level.

/**
 * Emits a structured JSON log event to stdout.
 *
 * @param {string} level - 'INFO' | 'WARN' | 'ERROR'
 * @param {object} fields - Must include all required fields from §6.1.
 */
function log(level, fields) {
  // §6.1 — Required fields on every rate limiter log event.
  // Fields not available at log time (e.g., redis_latency_ms on error paths
  // where Redis never responded) must be present with a null value rather
  // than omitted, so log parsers can rely on a consistent schema.
  const event = {
    timestamp: new Date().toISOString(),   // §6.1: ISO8601
    level,                                  // §6.1: INFO|WARN|ERROR
    component: 'ratelimit',                 // §6.1: static
    ...fields,
  };

  // Write to stdout as a single-line JSON object.
  // Using process.stdout.write avoids console.log's implicit newline
  // behavior differences across Node versions.
  process.stdout.write(JSON.stringify(event) + '\n');
}

module.exports = { log };
```

### Section 2.6 — `src/redisClient.js`

**Design reference:** §2 (atomicity, EVALSHA/EVAL/NOSCRIPT), §4 F-07 (script eviction recovery), §4 F-01/F-02 (connection/timeout errors), §3 (Redis config requirements).

```javascript
'use strict';

const crypto = require('crypto');
const { RATE_LIMIT_SCRIPT } = require('./luaScript');

// §2 — Pre-compute the SHA1 of the normative Lua script.
// This is used for EVALSHA calls. The SHA is deterministic given the script
// body; computing it at module load time avoids recomputing per request.
const SCRIPT_SHA = crypto
  .createHash('sha1')
  .update(RATE_LIMIT_SCRIPT)
  .digest('hex');

/**
 * Executes the rate limit Lua script against Redis.
 *
 * Implements:
 *   - §2: EVALSHA primary path, EVAL fallback on NOSCRIPT (F-07)
 *   - §4 F-07: Catches NOSCRIPT, falls back to EVAL, emits script_miss metric
 *   - §4 F-02: Command timeout is enforced by ioredis commandTimeout option
 *     (set on the client at construction time in rateLimiter.js — see §4 F-02
 *     requirement: "timeout must be enforced client-side")
 *
 * @param {import('ioredis').Redis} redisClient
 * @param {object} metrics - Bound metrics helper from createMetrics()
 * @param {string} currentKey
 * @param {string} previousKey
 * @param {number} limit
 * @param {number} windowSizeSeconds
 * @param {number} nowSeconds - Current time in seconds (may be fractional)
 * @param {number} elapsedFraction
 * @returns {Promise<{allowed: boolean, currentCount: number, previousCount: number, weightedCount: number}>}
 * @throws Will throw if Redis returns an error that is not NOSCRIPT.
 *   The caller (rateLimiter.js) is responsible for catching all throws
 *   and applying fail-open/fail-closed behavior.
 */
async function executeRateLimitScript(
  redisClient,
  metrics,
  currentKey,
  previousKey,
  limit,
  windowSizeSeconds,
  nowSeconds,
  elapsedFraction
) {
  const keys = [currentKey, previousKey];
  const args = [
    String(limit),
    String(windowSizeSeconds),
    String(nowSeconds),
    String(elapsedFraction),
  ];

  let rawResult;

  try {
    // §2 — Primary path: EVALSHA for performance (avoids sending script body
    // on every call once the script is cached in Redis).
    rawResult = await redisClient.evalsha(SCRIPT_SHA, keys.length, ...keys, ...args);
  } catch (err) {
    if (err.message && err.message.startsWith('NOSCRIPT')) {
      // §4 F-07 — Script was evicted from Redis script cache (Redis restart,
      // SCRIPT FLUSH, or memory pressure). Fall back to EVAL with full script
      // body. Redis will re-cache the script and return the SHA implicitly.
      // Emit script_miss metric so operators can detect this event (§6.1).
      metrics.redisScriptMiss();

      // Fall back to EVAL with the full script body.
      // After this succeeds, subsequent calls will use EVALSHA again because
      // Redis caches the script on EVAL execution.
      rawResult = await redisClient.eval(RATE_LIMIT_SCRIPT, keys.length, ...keys, ...args);
    } else {
      // Re-throw all other errors (timeout, connection errors, etc.) to the
      // caller, which applies fail-open/fail-closed per D-01.
      throw err;
    }
  }

  return parseScriptResult(rawResult);
}

/**
 * Parses the Lua script return value into a structured object.
 *
 * §2 return value contract:
 *   [0]: 1 = allowed, 0 = rejected
 *   [1]: current count
 *   [2]: previous count
 *   [3]: weighted count
 *
 * §4 F-03: If the result cannot be parsed as the expected structure, this
 * function throws. The caller catches and applies F-03 behavior.
 *
 * @param {Array} raw - Raw return value from Redis EVAL/EVALSHA
 * @returns {{allowed: boolean, currentCount: number, previousCount: number, weightedCount: number}}
 */
function parseScriptResult(raw) {
  // §4 F-03 — Validate the response structure before trusting it.
  if (
    !Array.isArray(raw) ||
    raw.length !== 4 ||
    (raw[0] !== 0 && raw[0] !== 1)
  ) {
    const err = new Error('Unexpected Redis script response structure');
    err.code = 'UNEXPECTED_RESPONSE';
    err.rawResponse = raw; // Attached for logging in F-03 handler; never sent to client.
    throw err;
  }

  return {
    allowed: raw[0] === 1,
    currentCount: Number(raw[1]),
    previousCount: Number(raw[2]),
    weightedCount: Number(raw[3]),
  };
}

module.exports = { executeRateLimitScript, SCRIPT_SHA };
```

**Checklist satisfied:** R-13, R-27, R-28 (script_miss metric call), R-24 (F-03 structured error)

### Section 2.7 — `src/identifierHelpers.js`

**Design reference:** §1 Assumption 2 (X-Forwarded-For extraction), §7 D-03 (extraction logic must be implemented and audited).

```javascript
'use strict';

// §7 Decision Point D-03 — IP extraction from X-Forwarded-For.
//
// The design states (§1, Assumption 2):
//   "The real client IP must be extracted from X-Forwarded-For or a trusted
//    header — not req.socket.remoteAddress, which will be the proxy IP.
//    This extraction logic must be implemented and audited before deployment."
//
// DECISION POINT D-03: The design does not specify:
//   1. How many proxy hops to trust (i.e., which index in the XFF chain).
//   2. Whether to validate that the extracted IP is a valid IP address.
//   3. What to do if the header is absent or malformed.
//   4. Whether to support alternative trusted headers (X-Real-IP, CF-Connecting-IP).
//
// SAFER DEFAULT IMPLEMENTED HERE:
//   - Extract the LAST (rightmost) IP from X-Forwarded-For that was appended
//     by a trusted proxy. In a standard single-proxy setup (nginx, ALB), the
//     rightmost value is the one the proxy itself appended and is not
//     spoofable by the client.
//   - If X-Forwarded-For is absent, fall back to req.socket.remoteAddress
//     with a WARNING comment — this is the proxy IP in production and will
//     cause all clients to share one rate limit bucket. This is a
//     misconfiguration, not a silent failure.
//   - If the header is present but empty/malformed, return null (triggers
//     F-06 behavior in the middleware).
//
// ⚠️  AUDIT REQUIRED BEFORE DEPLOYMENT: The number of trusted proxy hops
// must be confirmed with the infrastructure team. In a multi-layer proxy
// setup (e.g., Cloudflare → ALB → nginx → app), the correct index is
// different. Using the wrong index allows clients to spoof their IP by
// injecting values into X-Forwarded-For. See §4 F-05 (attack surface note).
//
// To change the extraction strategy, replace `extractIPFromRequest` with
// an application-specific implementation and supply it as `identifierFn`
// in the rateLimiter options. This helper is provided as a reference
// implementation only.

/**
 * Extracts the client IP from the X-Forwarded-For header.
 *
 * SAFER DEFAULT: Takes the rightmost IP in the X-Forwarded-For chain,
 * which is the value appended by the nearest trusted proxy and is not
 * client-controllable in a standard single-proxy deployment.
 *
 * ⚠️  AUDIT REQUIRED: See D-03 comment above before using in production.
 *
 * @param {object} req - Express/Node.js request object
 * @returns {string|null} Normalized IP string, or null if extraction fails.
 */
function extractIPFromRequest(req) {
  const xForwardedFor = req.headers && req.headers['x-forwarded-for'];

  if (xForwardedFor && typeof xForwardedFor === 'string' && xForwardedFor.trim() !== '') {
    // X-Forwarded-For may contain a comma-separated list:
    //   "client, proxy1, proxy2"
    // SAFER DEFAULT: take the rightmost entry (last trusted proxy's append).
    // ⚠️  D-03: This is correct for single-proxy deployments only.
    const parts = xForwardedFor.split(',');
    const ip = parts[parts.length - 1].trim();

    if (ip === '') {
      return null; // Malformed header — triggers F-06.
    }

    return `ip:${ip}`; // §3 key strategy example: "ip:203.0.113.42"
  }

  // X-Forwarded-For header is absent.
  // ⚠️  In production behind a proxy, req.socket.remoteAddress is the proxy
  // IP, not the client IP. All clients will share one rate limit bucket.
  // This is a misconfiguration. Return null to trigger F-06 rather than
  // silently applying a wrong identifier.
  // §4 F-06: "Do not use a fallback identifier silently."
  return null;
}

module.exports = { extractIPFromRequest };
```

**Checklist satisfied:** R-64 (D-03 documented and safer default implemented)

### Section 2.8 — `src/health.js`

**Design reference:** §6.4 Health Check — `isAlive()` and `isReady()`.

```javascript
'use strict';

// §6.4 — Health check exports.
//
// isAlive() and isReady() are exported for use by:
//   - Kubernetes liveness/readiness probes
//   - Load balancer health check endpoints
//   - Standalone service health routes (if the caller wraps these in HTTP handlers)
//
// §6.4 note: "The readiness check must not be on the hot path. It must not
// run on every request." These functions are called by health check
// infrastructure on its own schedule, not by the middleware itself.

/**
 * Creates health check functions bound to the rate limiter's state.
 *
 * @param {object} state - Internal state object from rateLimiter.js
 *   { initialized: boolean, failOpen: boolean, redisClient: ioredis.Redis }
 * @returns {{ isAlive: () => boolean, isReady: () => Promise<object> }}
 */
function createHealthChecks(state) {
  /**
   * §6.4 Liveness Check.
   * Returns true if the process is running and the middleware is initialized.
   * Never returns false due to Redis unavailability — Redis state is a
   * readiness concern, not a liveness concern.
   *
   * @returns {boolean}
   */
  function isAlive() {
    // §6.4: "Never returns a non-200 for liveness unless the process should
    // be killed and restarted. Redis unavailability does not affect liveness."
    return state.initialized === true;
  }

  /**
   * §6.4 Readiness Check.
   * Attempts a Redis
