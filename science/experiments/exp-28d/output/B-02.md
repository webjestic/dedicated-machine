# Production Rate Limiter in Node.js

Let me design this fully before writing a line of code.

---

## Phase 1: Requirements Design

### What algorithm should I use?

| Algorithm | Memory | Burst Handling | Accuracy | Distributed-Safe |
|---|---|---|---|---|
| Fixed Window | Low | Poor (boundary bursts) | Low | Easy |
| Sliding Window Log | High | Excellent | High | Medium |
| Sliding Window Counter | Low | Good | Medium | Easy |
| Token Bucket | Medium | Excellent | High | Hard |
| Leaky Bucket | Medium | None (smooths) | High | Hard |

**Decision: Sliding Window Counter** — best balance of accuracy, memory efficiency, and distributed correctness without the complexity of token bucket synchronization across nodes.

### What are the limiting dimensions?

A production rate limiter must answer: *limited by what, identified how?*

- **By IP** — unauthenticated traffic, abuse prevention
- **By API key / user ID** — authenticated traffic, fair use enforcement
- **By endpoint** — `/auth/login` needs tighter limits than `/api/data`
- **By method** — POST vs GET may have different costs
- **Composite keys** — `userId:endpoint` for fine-grained control

### What storage backend?

- **In-memory (single node):** Fast, no network latency, but state lost on restart and doesn't work across multiple instances
- **Redis:** Atomic operations via Lua scripts, TTL support, survives restarts, works across all nodes

**Decision: Redis with in-memory fallback.** If Redis is unavailable, fail open (allow traffic) with local memory as a degraded safety net. This is a deliberate tradeoff — failing closed during a Redis outage would take down your entire API.

### What should happen when a limit is hit?

- Return `429 Too Many Requests`
- Return `Retry-After` header (RFC 6585)
- Return `X-RateLimit-*` headers so clients can self-throttle
- Log the event for observability
- Never silently drop — always respond

### What headers should we return?

Following the draft IETF standard and common practice:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1704067200
Retry-After: 47
```

## Phase 2: Architecture

```
Incoming Request
      │
      ▼
┌─────────────────────────────────┐
│         Rate Limiter MW         │
│                                 │
│  1. Build composite key         │
│  2. Determine rule for route    │
│  3. Check Redis (Lua script)    │
│     │                           │
│     ├─ Redis healthy ──────────►│ Atomic sliding window check
│     │                           │
│     └─ Redis down ─────────────►│ In-memory fallback (fail open)
│                                 │
│  4. Attach headers              │
│  5. Allow or 429                │
└─────────────────────────────────┘
      │
      ▼
  Route Handler
```

### Sliding Window Counter — How It Works

```
Window size: 60 seconds
Current time: 75s

Previous window (0-60s):   80 requests
Current window (60-120s):  30 requests
Elapsed in current window: 15s → 25% through

Weighted count = (80 × 0.75) + (30 × 1.0) = 60 + 30 = 90

If limit is 100 → 10 remaining
```

This smooths the boundary spike problem of fixed windows without storing every individual request timestamp.

/**
 * Rate limit rules evaluated in order — first match wins.
 *
 * Each rule defines:
 *   match:     function(req) → boolean
 *   limit:     max requests allowed in the window
 *   window:    window size in seconds
 *   keyBy:     how to identify the caller
 *   name:      human-readable label for logs/metrics
 *
 * Design decision: rules are code, not config files.
 * This means changes require a deploy, but you get full
 * expressiveness and no YAML parsing surprises.
 */

const rules = [
  // Tightest limits first — auth endpoints are brute-force targets
  {
    name: 'auth-login',
    match: (req) => req.path === '/auth/login' && req.method === 'POST',
    limit: 5,
    window: 60,        // 5 attempts per minute
    keyBy: 'ip',
  },
  {
    name: 'auth-general',
    match: (req) => req.path.startsWith('/auth/'),
    limit: 20,
    window: 60,
    keyBy: 'ip',
  },

  // Authenticated API routes — key by user to prevent one user
  // from consuming shared IP quota (e.g., behind a corporate NAT)
  {
    name: 'api-authenticated',
    match: (req) => req.path.startsWith('/api/') && req.user?.id,
    limit: 1000,
    window: 3600,      // 1000 requests per hour per user
    keyBy: 'user',
  },

  // Public API routes — key by IP
  {
    name: 'api-public',
    match: (req) => req.path.startsWith('/api/'),
    limit: 100,
    window: 60,
    keyBy: 'ip',
  },

  // Global catch-all
  {
    name: 'global',
    match: () => true,
    limit: 200,
    window: 60,
    keyBy: 'ip',
  },
];

module.exports = { rules };

### `src/utils/keyBuilder.js`

```javascript
/**
 * Builds the Redis/memory key for a given request + rule.
 *
 * Key format: rl:{ruleName}:{identifier}:{windowSlot}
 *
 * The windowSlot buckets time so we only ever need to track
 * two keys per rule per caller: the current window and the
 * previous one. No cleanup job needed — Redis TTL handles it.
 *
 * Example: rl:api-public:ip:1.2.3.4:28411
 *   where 28411 = Math.floor(Date.now() / 1000 / 60)
 *   (the 60-second window slot we're currently in)
 */

'use strict';

/**
 * Extract the caller identifier from the request based on rule.keyBy.
 *
 * IP extraction handles the X-Forwarded-For header for requests
 * behind a load balancer or reverse proxy. We take the FIRST IP
 * in the chain — that's the original client. Be aware this can
 * be spoofed if your proxy doesn't strip/overwrite the header.
 * In production, configure your proxy to set a trusted header.
 */
function extractIdentifier(req, keyBy) {
  switch (keyBy) {
    case 'ip': {
      const forwarded = req.headers['x-forwarded-for'];
      if (forwarded) {
        // "client, proxy1, proxy2" — take the leftmost
        return forwarded.split(',')[0].trim();
      }
      return req.socket?.remoteAddress ?? 'unknown';
    }

    case 'user': {
      const userId = req.user?.id;
      if (!userId) {
        // Rule said keyBy user but no user exists.
        // Fall back to IP so we don't skip limiting entirely.
        console.warn('[rate-limiter] keyBy=user but req.user.id missing, falling back to IP');
        return extractIdentifier(req, 'ip');
      }
      return `user:${userId}`;
    }

    case 'apiKey': {
      const key = req.headers['x-api-key'] ?? req.query?.api_key;
      if (!key) {
        return extractIdentifier(req, 'ip');
      }
      // Don't log the actual key — hash it for the rate limit key
      // so raw keys never appear in Redis keyspace or logs
      const crypto = require('crypto');
      return `apikey:${crypto.createHash('sha256').update(key).digest('hex').slice(0, 16)}`;
    }

    default:
      throw new Error(`[rate-limiter] Unknown keyBy value: "${keyBy}"`);
  }
}

/**
 * Returns the two window slot keys needed for the sliding window calculation:
 *   currentKey  — the bucket we're writing into right now
 *   previousKey — the bucket immediately before it
 *
 * Both keys include a TTL hint (2 × window) so Redis can expire them
 * automatically without a cleanup job.
 */
function buildWindowKeys(ruleName, identifier, windowSeconds) {
  const nowSeconds = Math.floor(Date.now() / 1000);
  const currentSlot = Math.floor(nowSeconds / windowSeconds);
  const previousSlot = currentSlot - 1;

  const base = `rl:${ruleName}:${identifier}`;

  return {
    currentKey: `${base}:${currentSlot}`,
    previousKey: `${base}:${previousSlot}`,
    currentSlot,
    windowSeconds,
    nowSeconds,
  };
}

module.exports = { extractIdentifier, buildWindowKeys };
```

### `src/store/redisStore.js`

```javascript
/**
 * Redis-backed sliding window counter store.
 *
 * Uses a Lua script executed atomically on the Redis server.
 * Lua scripts in Redis are atomic — no other command runs
 * between our read and write. This is how we avoid race
 * conditions without MULTI/EXEC transactions.
 *
 * The script:
 *   1. Increment the current window counter
 *   2. Set TTL on the current key (2 windows, so previous is still readable)
 *   3. Read the previous window counter
 *   4. Return both counts so the caller can compute the weighted total
 */

'use strict';

const { createClient } = require('redis');

// ---------------------------------------------------------------------------
// Lua script — runs atomically on the Redis server
// ---------------------------------------------------------------------------
//
// KEYS[1] = current window key   e.g. rl:api-public:1.2.3.4:28411
// KEYS[2] = previous window key  e.g. rl:api-public:1.2.3.4:28410
// ARGV[1] = TTL in seconds (2 × window size)
//
// Returns: [currentCount, previousCount]
// ---------------------------------------------------------------------------
const SLIDING_WINDOW_SCRIPT = `
local current_key  = KEYS[1]
local previous_key = KEYS[2]
local ttl          = tonumber(ARGV[1])

-- Increment current window and get new value atomically
local current_count = redis.call('INCR', current_key)

-- Set TTL only on first write to this key (when count just became 1)
-- Avoids resetting TTL on every request, which would prevent expiry
if current_count == 1 then
  redis.call('EXPIRE', current_key, ttl)
end

-- Read previous window (may not exist — returns nil → 0)
local previous_count = tonumber(redis.call('GET', previous_key)) or 0

return {current_count, previous_count}
`;

class RedisStore {
  constructor(options = {}) {
    this.options = {
      url: options.url ?? process.env.REDIS_URL ?? 'redis://localhost:6379',
      socket: {
        connectTimeout: options.connectTimeout ?? 2000,  // 2s — don't block requests
        reconnectStrategy: (retries) => {
          // Exponential backoff capped at 10s
          // Returning false stops reconnection (we handle that via health check)
          if (retries > 10) return new Error('Redis max retries exceeded');
          return Math.min(retries * 100, 10000);
        },
      },
      ...options.redis,
    };

    this.client = null;
    this.healthy = false;
    this._scriptSha = null;  // SHA of loaded Lua script (EVALSHA is faster than EVAL)
  }

  async connect() {
    this.client = createClient(this.options);

    this.client.on('connect', () => {
      this.healthy = true;
      console.info('[rate-limiter:redis] Connected');
    });

    this.client.on('ready', async () => {
      // Pre-load the Lua script so we can use EVALSHA instead of EVAL
      // EVALSHA sends only the hash — saves bandwidth on hot paths
      try {
        this._scriptSha = await this.client.scriptLoad(SLIDING_WINDOW_SCRIPT);
        console.info('[rate-limiter:redis] Lua script loaded, SHA:', this._scriptSha);
      } catch (err) {
        console.error('[rate-limiter:redis] Failed to load Lua script:', err.message);
      }
    });

    this.client.on('error', (err) => {
      // Log but don't throw — the middleware will detect unhealthy
      // and fall back to in-memory
      console.error('[rate-limiter:redis] Error:', err.message);
      this.healthy = false;
    });

    this.client.on('reconnecting', () => {
      console.warn('[rate-limiter:redis] Reconnecting...');
      this.healthy = false;
    });

    await this.client.connect();
  }

  async disconnect() {
    if (this.client) {
      await this.client.quit();
      this.healthy = false;
    }
  }

  /**
   * Perform the sliding window check-and-increment.
   *
   * Returns: { currentCount, previousCount, allowed: boolean }
   *
   * Does NOT make the allow/deny decision itself — that belongs
   * in the middleware layer where we have rule context.
   */
  async increment(currentKey, previousKey, windowSeconds) {
    if (!this.healthy || !this.client) {
      throw new Error('Redis store is not healthy');
    }

    const ttl = windowSeconds * 2;  // Keep previous window readable

    let result;

    try {
      // Prefer EVALSHA (uses cached script) — fall back to EVAL if script was flushed
      if (this._scriptSha) {
        try {
          result = await this.client.evalSha(this._scriptSha, {
            keys: [currentKey, previousKey],
            arguments: [String(ttl)],
          });
        } catch (err) {
          if (err.message?.includes('NOSCRIPT')) {
            // Script was flushed (e.g., Redis restart) — reload and retry
            console.warn('[rate-limiter:redis] Script cache miss, reloading...');
            this._scriptSha = await this.client.scriptLoad(SLIDING_WINDOW_SCRIPT);
            result = await this.client.evalSha(this._scriptSha, {
              keys: [currentKey, previousKey],
              arguments: [String(ttl)],
            });
          } else {
            throw err;
          }
        }
      } else {
        // No SHA cached yet — use EVAL directly
        result = await this.client.eval(SLIDING_WINDOW_SCRIPT, {
          keys: [currentKey, previousKey],
          arguments: [String(ttl)],
        });
      }
    } catch (err) {
      this.healthy = false;
      throw err;
    }

    return {
      currentCount: Number(result[0]),
      previousCount: Number(result[1]),
    };
  }

  isHealthy() {
    return this.healthy;
  }
}

module.exports = { RedisStore };
```

### `src/store/memoryStore.js`

```javascript
/**
 * In-memory sliding window counter store.
 *
 * Used as a fallback when Redis is unavailable.
 *
 * Limitations vs Redis store (document these explicitly):
 *   - State is per-process — in a multi-instance deployment,
 *     each instance has its own counters. A client could make
 *     N × limit requests by round-robining across N instances.
 *   - Memory grows with unique keys. The cleanup interval
 *     prevents unbounded growth but adds a small GC cost.
 *   - Not suitable as a primary store for strict enforcement.
 *
 * This is a deliberate "fail open with degraded accuracy"
 * strategy. The alternative (fail closed) would take down
 * your API every time Redis has a hiccup.
 */

'use strict';

class MemoryStore {
  constructor(options = {}) {
    // Map of key → { count, expiresAt }
    this._windows = new Map();

    // Clean up expired entries every 60s
    // Without this, a long-running process would accumulate
    // entries for every unique IP/user ever seen
    this._cleanupInterval = setInterval(
      () => this._cleanup(),
      options.cleanupIntervalMs ?? 60_000
    );

    // Allow the process to exit even if this interval is running
    this._cleanupInterval.unref();
  }

  /**
   * Mirrors the RedisStore.increment interface exactly.
   * The middleware doesn't need to know which store it's talking to.
   */
  increment(currentKey, previousKey, windowSeconds) {
    const now = Date.now();
    const ttlMs = windowSeconds * 2 * 1000;

    // Current window
    const current = this._getOrCreate(currentKey, now, ttlMs);
    current.count += 1;

    // Previous window (read-only — don't reset its TTL)
    const previous = this._windows.get(previousKey);
    const previousCount = (previous && previous.expiresAt > now) ? previous.count : 0;

    return Promise.resolve({
      currentCount: current.count,
      previousCount,
    });
  }

  _getOrCreate(key, now, ttlMs) {
    let entry = this._windows.get(key);

    if (!entry || entry.expiresAt <= now) {
      // Key doesn't exist or has expired — start fresh
      entry = { count: 0, expiresAt: now + ttlMs };
      this._windows.set(key, entry);
    }

    return entry;
  }

  _cleanup() {
    const now = Date.now();
    let removed = 0;

    for (const [key, entry] of this._windows.entries()) {
      if (entry.expiresAt <= now) {
        this._windows.delete(key);
        removed++;
      }
    }

    if (removed > 0) {
      console.debug(`[rate-limiter:memory] Cleaned up ${removed} expired entries, ${this._windows.size} remaining`);
    }
  }

  destroy() {
    clearInterval(this._cleanupInterval);
    this._windows.clear();
  }

  // Expose size for monitoring
  size() {
    return this._windows.size;
  }
}

module.exports = { MemoryStore };
```

### `src/middleware/rateLimiter.js`

```javascript
/**
 * Rate limiter middleware factory.
 *
 * Usage:
 *   const limiter = createRateLimiter({ redisStore, memoryStore });
 *   app.use(limiter);
 *
 * This is the orchestration layer. It:
 *   1. Finds the matching rule for this request
 *   2. Builds the window keys
 *   3. Calls the appropriate store
 *   4. Computes the sliding window weighted count
 *   5. Attaches response headers
 *   6. Either calls next() or returns 429
 */

'use strict';

const { rules } = require('../config/rules');
const { extractIdentifier, buildWindowKeys } = require('../utils/keyBuilder');

/**
 * Compute the sliding window weighted request count.
 *
 * The core of the algorithm:
 *   - We have two fixed windows: previous and current
 *   - We weight the previous window by how much of the
 *     current window has NOT yet elapsed
 *   - This approximates a true sliding window without
 *     storing individual request timestamps
 *
 * Example:
 *   Window = 60s, now = 75s (15s into current window)
 *   previousCount = 80, currentCount = 30
 *   elapsedFraction = 15/60 = 0.25
 *   weight of previous = 1 - 0.25 = 0.75
 *   weightedCount = (80 × 0.75) + 30 = 90
 */
function computeWeightedCount(previousCount, currentCount, nowSeconds, windowSeconds) {
  const currentSlot = Math.floor(nowSeconds / windowSeconds);
  const windowStart = currentSlot * windowSeconds;
  const elapsedInWindow = nowSeconds - windowStart;
  const elapsedFraction = elapsedInWindow / windowSeconds;
  const previousWeight = 1 - elapsedFraction;

  return (previousCount * previousWeight) + currentCount;
}

/**
 * Build the standard rate limit response headers.
 * Clients that respect these can self-throttle before hitting 429.
 */
function buildHeaders(rule, weightedCount, resetAtSeconds) {
  const remaining = Math.max(0, rule.limit - Math.floor(weightedCount));

  return {
    'X-RateLimit-Limit': String(rule.limit),
    'X-RateLimit-Remaining': String(remaining),
    'X-RateLimit-Reset': String(resetAtSeconds),
    'X-RateLimit-Policy': `${rule.limit};w=${rule.window}`,  // Draft IETF format
  };
}

/**
 * Factory function — creates the middleware with injected stores.
 *
 * Dependency injection (rather than module-level singletons) makes
 * this testable: tests can pass mock stores without monkey-patching.
 */
function createRateLimiter({ redisStore, memoryStore, onLimitReached } = {}) {
  if (!memoryStore) {
    throw new Error('[rate-limiter] memoryStore is required (used as fallback)');
  }

  return async function rateLimiterMiddleware(req, res, next) {
    // ------------------------------------------------------------------
    // 1. Find the first matching rule
    // ------------------------------------------------------------------
    const rule = rules.find((r) => r.match(req));

    if (!rule) {
      // No rule matched — this shouldn't happen if you have a catch-all,
      // but if it does, fail open rather than blocking legitimate traffic
      console.warn('[rate-limiter] No rule matched for:', req.method, req.path);
      return next();
    }

    // ------------------------------------------------------------------
    // 2. Build the composite key and window slots
    // ------------------------------------------------------------------
    let identifier;
    try {
      identifier = extractIdentifier(req, rule.keyBy);
    } catch (err) {
      console.error('[rate-limiter] Failed to extract identifier:', err.message);
      return next();  // Fail open
    }

    const { currentKey, previousKey, nowSeconds, windowSeconds } =
      buildWindowKeys(rule.name, identifier, rule.window);

    // ------------------------------------------------------------------
    // 3. Increment counter in the appropriate store
    // ------------------------------------------------------------------
    let currentCount, previousCount;
    let usingFallback = false;

    const useRedis = redisStore?.isHealthy();

    try {
      if (useRedis) {
        ({ currentCount, previousCount } = await redisStore.increment(
          currentKey,
          previousKey,
          rule.window
        ));
      } else {
        throw new Error('Redis not healthy');
      }
    } catch (err) {
      // Redis failed — fall back to in-memory
      // Log this: it means your rate limiting is degraded
      if (!usingFallback) {
        console.warn('[rate-limiter] Falling back to memory store:', err.message);
        usingFallback = true;
      }

      try {
        ({ currentCount, previousCount } = await memoryStore.increment(
          currentKey,
          previousKey,
          rule.window
        ));
      } catch (fallbackErr) {
        // Both stores failed — fail open, log loudly
        console.error('[rate-limiter] Both stores failed, failing open:', fallbackErr.message);
        return next();
      }
    }

    // ------------------------------------------------------------------
    // 4. Compute sliding window weighted count
    // ------------------------------------------------------------------
    const weightedCount = computeWeightedCount(
      previousCount,
      currentCount,
      nowSeconds,
      windowSeconds
    );

    // When does the current window reset?
    const currentSlot = Math.floor(nowSeconds / windowSeconds);
    const resetAtSeconds = (currentSlot + 1) * windowSeconds;

    // ------------------------------------------------------------------
    // 5. Attach rate limit headers (always — even on 429)
    // ------------------------------------------------------------------
    const headers = buildHeaders(rule, weightedCount, resetAtSeconds);
    res.set(headers);

    // ------------------------------------------------------------------
    // 6. Allow or reject
    // ------------------------------------------------------------------
    const allowed = weightedCount <= rule.limit;

    if (!allowed) {
      const retryAfter = resetAtSeconds - nowSeconds;

      res.set('Retry-After', String(retryAfter));

      // Fire optional callback for metrics/alerting
      if (typeof onLimitReached === 'function') {
        onLimitReached({
          rule: rule.name,
          identifier,
          weightedCount,
          limit: rule.limit,
          usingFallback,
          req,
        });
      }

      console.warn('[rate-limiter] Limit reached', {
        rule: rule.name,
        identifier,
        weightedCount: weightedCount.toFixed(2),
        limit: rule.limit,
        usingFallback,
        path: req.path,
        method: req.method,
      });

      return res.status(429).json({
        error: 'Too Many Requests',
        message: `Rate limit exceeded. Try again in ${retryAfter} seconds.`,
        retryAfter,
      });
    }

    next();
  };
}

module.exports = { createRateLimiter, computeWeightedCount };
```

### `app.js`

```javascript
/**
 * Application entry point.
 *
 * Wires up the stores and middleware, then starts the server.
 * Demonstrates how to compose the rate limiter into an Express app.
 */

'use strict';

const express = require('express');
const { RedisStore } = require('./src/store/redisStore');
const { MemoryStore } = require('./src/store/memoryStore');
const { createRateLimiter } = require('./src/middleware/rateLimiter');

async function bootstrap() {
  const app = express();
  app.use(express.json());

  // ------------------------------------------------------------------
  // Initialize stores
  // ------------------------------------------------------------------
  const memoryStore = new MemoryStore();

  const redisStore = new RedisStore({
    url: process.env.REDIS_URL ?? 'redis://localhost:6379',
    connectTimeout: 2000,
  });

  // Attempt Redis connection — if it fails, we still start
  // The middleware will use memoryStore as fallback
  try {
    await redisStore.connect();
  } catch (err) {
    console.error('[app] Redis connection failed at startup, using memory fallback:', err.message);
  }

  // ------------------------------------------------------------------
  // Rate limiter middleware
  // ------------------------------------------------------------------
  const rateLimiter = createRateLimiter({
    redisStore,
    memoryStore,
    onLimitReached: ({ rule, identifier, weightedCount, limit }) => {
      // Hook this into your metrics system (Datadog, Prometheus, etc.)
      // metrics.increment('rate_limiter.limit_reached', { rule });
      console.info('[metrics] rate_limit_reached', { rule, identifier, weightedCount, limit });
    },
  });

  app.use(rateLimiter);

  // ------------------------------------------------------------------
  // Routes (minimal — this is infrastructure, not application code)
  // ------------------------------------------------------------------
  app.post('/auth/login', (req, res) => {
    res.json({ token: 'example-token' });
  });

  app.get('/api/data', (req, res) => {
    res.json({ data: 'example' });
  });

  app.get('/health', (req, res) => {
    res.json({
      status: 'ok',
      rateLimiter: {
        redis: redisStore.isHealthy(),
        memoryStoreSize: memoryStore.size(),
      },
    });
  });

  // ------------------------------------------------------------------
  // Graceful shutdown
  // ------------------------------------------------------------------
  const server = app.listen(process.env.PORT ?? 3000, () => {
    console.info(`[app] Listening on port ${process.env.PORT ?? 3000}`);
  });

  async function shutdown(signal) {
    console.info(`[app] ${signal} received, shutting down gracefully`);
    server.close(async () => {
      await redisStore.disconnect();
      memoryStore.destroy();
      console.info('[app] Shutdown complete');
      process.exit(0);
    });
  }

  process.on('SIGTERM', () => shutdown('SIGTERM'));
  process.on('SIGINT', () => shutdown('SIGINT'));
}

bootstrap().catch((err) => {
  console.error('[app] Fatal startup error:', err);
  process.exit(1);
});
```
