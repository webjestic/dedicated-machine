Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all sound choices. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a TOCTOU race between the stock read and the DB transaction to a silent exception swallow that could corrupt inventory state. The test suite also has coverage gaps around the most critical failure paths.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU Race Between Stock Read and DB Write**

```python
# Line ~90 in reserve_item
current_stock = self.db.get_stock(req.item_id)
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The stock is read *outside* the database transaction. If the DB supports it, the `get_stock` call should be inside the transaction with a `SELECT FOR UPDATE` (or equivalent pessimistic lock), otherwise a concurrent writer that bypasses Redis (e.g., a manual admin correction, a different code path, or a Redis failover) can cause a double-spend. The Redis lock protects against *this service's* concurrency, but not against external writers or Redis unavailability.

**Recommendation:** Move `get_stock` inside `self.db.transaction()` and use a row-level DB lock if your DB supports it. The Redis lock should be a performance optimization, not the sole correctness guarantee.

---

**2. Silent Exception Swallow in `reserve_item` — Inventory Corruption Risk**

```python
with self._acquire_lock(lock_key):
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
    # <-- if db.transaction().__exit__ raises, execution falls through here
    return ReservationResult(success=True, ...)
```

Any exception raised *after* the lock is acquired but *outside* the `LockAcquisitionError` catch — for example, a DB connection error, a network timeout, or an unexpected `RuntimeError` — will propagate up to the caller as an unhandled exception with no `ReservationResult` returned. The caller gets a raw exception instead of a structured response. Worse, if the transaction partially commits (depending on DB behavior), stock may be decremented without an order being created.

**Recommendation:** Add a broad `except Exception` handler inside the lock context that logs the error and returns a `ReservationResult(success=False, message="Internal error")`. Do not let raw exceptions escape a service boundary.

---

**3. Heartbeat Thread Leak on Unhandled Exception**

```python
t = threading.Thread(target=_beat, daemon=True)
t.start()
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block is correct *structurally*, but if `self._release_script(...)` itself raises (e.g., Redis is down), the exception propagates and the log line `"Lock released"` never fires. More importantly, if `t.join()` times out (the heartbeat thread is stuck), the lock release still proceeds — but you have no signal that the thread is still running. A daemon thread will be killed at process exit, but in a long-running service this is a latent resource concern.

**Recommendation:** Check `t.is_alive()` after `t.join()` and log a warning if the thread did not terminate. Wrap `self._release_script` in a `try/except` to prevent a Redis error from masking the original exception.

### 🟡 Significant

**4. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can pass `quantity=999999999`. This passes validation, will read as insufficient stock (probably), but the value is logged and stored. More importantly, there is no domain-level cap. For a flash sale service this is a business logic gap that could be exploited to probe stock levels or cause integer overflow in downstream systems.

**Recommendation:** Add a reasonable `le=` upper bound (e.g., `le=100` or make it configurable).

---

**5. `item_id` Logged Directly From User Input**

```python
logger.error("Lock acquisition failed item_id=%s: %s", req.item_id, e)
logger.info("Reserved item_id=%s quantity=%d order_id=%s remaining=%d", ...)
```

`item_id` is validated by the regex `^[a-zA-Z0-9_\-]+$`, which does prevent log injection via newlines or special characters. This is acceptable *because* the Pydantic validation runs before any logging. However, this dependency is fragile — if someone refactors the logging call to use the raw `item_id` parameter before validation, it becomes a log injection vector. 

**Recommendation:** Log only after validation (already the case), but add a comment explicitly noting this dependency so it survives refactoring.

---

**6. Lock TTL vs. Heartbeat Interval Is Hardcoded and Undocumented**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

The relationship between these two constants is a correctness invariant (`HEARTBEAT_INTERVAL < LOCK_TTL / 2`), but it is not enforced. If someone changes `LOCK_TTL` to `15` without updating `HEARTBEAT_INTERVAL`, the heartbeat fires correctly, but if they change `HEARTBEAT_INTERVAL` to `20`, the lock can expire before renewal.

**Recommendation:** Either derive `HEARTBEAT_INTERVAL` from `LOCK_TTL` (e.g., `LOCK_TTL // 3`) or add an assertion in `__init__` that enforces the invariant.

---

**7. `retry_attempts` Default Is Not Configurable at the Service Level**

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

The retry count is buried in the method signature with no exposure at the `InventoryService` constructor level. Under flash sale load, operators may need to tune this without modifying source code.

**Recommendation:** Accept `lock_retry_attempts` in `__init__` and pass it through, or read from a config object.

---

### 🟢 Test Coverage Gaps

**8. No Test for DB Exception During Transaction**

There is no test that verifies behavior when `db.update_stock` or `db.create_order` raises. This is the most dangerous failure path (potential partial commit) and it is completely untested.

**9. No Test for Heartbeat Thread Behavior**

The heartbeat is a non-trivial concurrent mechanism. There is no test verifying that the `stop_event` is set on normal exit, or that the heartbeat correctly detects a lost lock (`result == 0`). At minimum, a test should verify that `_extend_script` is called when the lock is held for longer than `HEARTBEAT_INTERVAL`.

**10. `test_lock_release_uses_lua_script` Is Fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test.

**Recommendation:**
```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # or capture the token
)
```

---

## What Is Done Well

- Lua scripts for atomic release and extension are correct and idiomatic.
- `uuid4` token prevents accidental lock release by a different holder.
- `SET NX EX` is the correct single-command atomic acquisition.
- Pydantic validation with regex on `item_id` prevents path traversal and injection at the model level.
- Daemon thread is appropriate here — it will not block process shutdown.
- Backoff with jitter is correctly implemented.
- `stop_event.wait(timeout=...)` as a timer is the right pattern (avoids busy-wait).
