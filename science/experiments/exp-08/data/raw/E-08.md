Decision: Request Changes

Summary: The implementation demonstrates solid surface-level discipline — atomic Lua scripts, heartbeat renewal, scoped stop events, Pydantic validation, and structured logging are all present and correctly wired. However, there is a critical architectural gap that the checklist does not name and the tests do not probe: **the heartbeat thread's detection of lock loss is silent and unobserved by the protected operation.** When the heartbeat detects `result == 0` (lock stolen or expired), it returns quietly — the `reserve_item` body continues executing, reads stock, and commits a database transaction under the belief that it holds exclusive access when it does not. The gap between lock loss and operation awareness is the exact failure mode this architecture is supposed to prevent, and it is not closed.

---

## Detailed Findings

### 1. Critical — Heartbeat Loss Is Not Propagated to the Protected Operation

This is the central flaw. Follow the execution path:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # <-- silent exit
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the thread returns. Nothing else happens. The `reserve_item` body — running concurrently on the main thread — has no mechanism to learn this. It will proceed to `db.get_stock()`, the stock comparison, and `db.transaction()` / `db.update_stock()` / `db.create_order()` with no exclusive coordination in place. A second process that acquired the lock in the interim will do the same. The database transaction does not substitute for the lock — it provides atomicity of writes, not mutual exclusion of the read-check-write cycle.

The checklist item reads: *"heartbeat detects token mismatch (result == 0)"* — detection is confirmed, but propagation is absent. Detection without propagation is not a safety mechanism. It is observability theater.

**What is required:** The heartbeat must communicate loss back to the protected scope. The canonical patterns are:

**Option A — Shared flag checked at critical boundaries:**
```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In reserve_item, before each phase boundary:
if lock_lost.is_set():
    raise LockLostError(f"Lock lost mid-operation for {lock_key}")
```

**Option B — Exception injection via a wrapper that raises on yield resumption.** More complex, but allows the context manager itself to raise on `__exit__` entry if the lock was lost during the body.

Neither option is implemented. Option A is the minimum acceptable fix. The check must occur at least before the database transaction is entered — the point of no safe return.

### 2. Significant — `t.join()` Timeout Is Silently Ignored

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise on timeout expiry. If the heartbeat thread is blocked — on a hung Redis call, a GIL contention spike, or a network partition — `join()` returns after `HEARTBEAT_INTERVAL + 1` seconds and execution falls through to `_release_script` regardless. The thread is still running. It may subsequently call `_extend_script` after the release script has already deleted the key, or after another process has acquired the lock with a new token. The Lua guard (`get` then `expire` only if token matches) prevents the *extension* from corrupting a new holder's lock, but the thread is now a dangling resource with no owner.

**Required fix:**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error(
            "Heartbeat thread did not terminate within timeout key=%s — "
            "possible Redis connectivity issue; proceeding with release attempt",
            lock_key
        )
    self._release_script(keys=[lock_key], args=[token])
```

This does not eliminate the dangling thread, but it surfaces the condition. A production system should also track thread liveness externally and alert on accumulation.

### 3. Significant — Stock Read Is Outside the Database Transaction

```python
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # outside transaction

    if current_stock < req.quantity:
        ...

    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
        self.db.create_order(...)
```

The read and the write are in separate database operations. If the database connection is lost and re-established between the read and the transaction open, or if the database has read-committed isolation and another path modifies stock outside the lock (a manual admin correction, a refund path, a different code branch that does not respect this lock), the stock value used for the check is stale at the time of the write.

The correct pattern is to bring the read inside the transaction and use a pessimistic row lock or a conditional update:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

The distributed lock provides inter-process mutual exclusion. The database transaction with a row lock provides intra-database consistency. Both layers are needed. Currently only one is present.

### 4. Moderate — `_release_script` Return Value Is Not Checked

```python
self._release_script(keys=[lock_key], args=[token])
```

The Lua release script returns `0` if the token did not match — meaning the lock had already expired and been acquired by another process before the release attempt. This is a meaningful event: it confirms that the protected operation ran past its coordination window. The return value is discarded. This should be logged at `ERROR` level and, combined with finding #1, should trigger a compensating action or at minimum an alert.

```python
released = self._release_script(keys=[lock_key], args=[token])
if not released:
    logger.error(
        "Lock release found no matching token key=%s — lock expired or was "
        "stolen during operation; data consistency may be compromised",
        lock_key
    )
```

---

### 5. Moderate — Heartbeat `LOCK_TTL` Is Passed as String; Type Contract Is Implicit

```python
args=[token, str(self.LOCK_TTL)]
```

The Lua `EXPIRE` command requires an integer argument. Redis will coerce a string representation of an integer correctly in most versions, but this is an implicit contract with the Redis version in use. The argument should be passed as an integer, or the Lua script should explicitly convert it:

```lua
return redis.call("expire", KEYS[1], tonumber(ARGV[2]))
```

This is low severity in practice but represents an unexamined assumption in a security-critical path.

---

### 6. Minor — Test Coverage Does Not Probe the Failure Gap

The test suite validates the happy path, insufficient stock, item not found, lock contention, input validation, and Lua release. No test simulates:

- Heartbeat detecting `result == 0` mid-operation and verifying the protected body does not proceed
- `_release_script` returning `0` and verifying the condition is logged
- `t.join()` timing out and verifying the thread liveness check fires

These are not style gaps. They are the exact scenarios the architecture claims to handle. A test suite that does not exercise the failure path of a safety mechanism does not verify the safety mechanism.

---

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | **Critical** | Heartbeat lock-loss must be propagated to the protected operation via shared event or equivalent; the body must not proceed after lock loss is detected |
| 2 | **Significant** | `t.join()` timeout expiry must be detected and logged; dangling thread must be surfaced |
| 3 | **Significant** | Stock read must be inside the database transaction with a row-level lock |
| 4 | **Moderate** | `_release_script` return value must be checked and logged on `0` |
| 5 | **Moderate** | Lua `EXPIRE` argument should use `tonumber()` or be passed as integer |
| 6 | **Minor** | Tests must cover heartbeat loss propagation and release token mismatch paths |

The architecture is sound in intent. The surface mechanics are correctly implemented. The flaw is in the gap between when coordination is lost and when the protected code learns it — and that gap is currently open.
