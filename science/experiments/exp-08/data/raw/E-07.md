# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Author:** dev-team
**Branch:** feature/flash-sale-inventory

---

## Structural Simulation Before Reading Implementation

Before evaluating any line of code, I ask the question the checklist cannot answer:

> *If the Redis lock expires — or is silently lost — while the database transaction is executing, does the protected code know? Or does it commit against state it no longer exclusively owns?*

The failure mode is not in acquisition. It is not in release. It lives in the gap between when coordination is lost and when the code learns it. That gap must be simulated, not read.

---

## Critical Finding — The Heartbeat Detects Loss But Does Not Propagate It

This is the central architectural defect in the submission.

The heartbeat thread correctly detects lock loss:

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # ← lock was lost; thread exits
```

When `result == 0`, the token no longer matches the key. Another process has acquired the lock. The heartbeat thread exits cleanly. This part is correct.

What happens next is the problem.

The main thread — currently inside `with self._acquire_lock(lock_key):` — does not know. There is no shared signal, no exception raised, no flag set. The main thread continues executing:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← executes without the lock
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This transaction commits inventory state that was read under a lock the process no longer holds. A second process has already acquired that lock, read the same stock value, and may be executing the same transaction concurrently. The distributed lock has failed to provide mutual exclusion at the only moment mutual exclusion matters — during the write.

The checklist entry `"heartbeat detects token mismatch (result == 0)"` is accurate. The detection exists. The propagation does not. Detection without propagation is not a safety mechanism — it is a log message that never gets written.

**Concrete failure sequence:**

1. Process A acquires lock, reads `stock = 1`, begins DB transaction
2. Redis suffers a latency spike; lock TTL expires before heartbeat fires
3. Process B acquires the same lock
4. Process A's heartbeat fires, detects `result == 0`, thread exits silently
5. Process A's main thread commits: `stock → 0`, order A created
6. Process B reads `stock = 1` (stale, pre-commit), commits: `stock → 0`, order B created
7. Two orders exist for one unit of stock

This is not a theoretical edge case. Under flash sale load — the exact scenario this service is designed for — Redis latency spikes and GC pauses are routine. This is the highest-probability failure mode in the system.

## Secondary Findings

### Lock Release After Confirmed Loss

In the `finally` block:

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

The Lua release script is token-gated, so it will not delete a lock held by another process. This is correct. However, the release is attempted unconditionally with no branch on whether the lock was actually held at release time. If the lock was lost mid-operation and the above critical defect is addressed by raising an exception, the finally block still attempts release — which is safe due to the Lua guard — but the code provides no log distinction between "released a lock we held" and "attempted release of a lock we had already lost." Observability is degraded at exactly the moment an incident is occurring.

### Heartbeat Interval Is Aggressive for the Stated TTL

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 8`. The checklist states heartbeat fires at TTL/3, which is `10`. The actual interval is `8`, meaning the heartbeat fires at TTL×0.267. This provides reasonable margin, but the discrepancy between the comment and the constant is a documentation defect that will mislead future maintainers tuning these values under operational pressure.

### `t.join(timeout=...)` Return Value Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join()` with a timeout returns `None` whether the thread exited or timed out. If the heartbeat thread hangs — due to a Redis network stall during the extend call — the main thread proceeds to release after the timeout without knowing the heartbeat is still running. The heartbeat thread is daemonized, so it will not block process exit, but it may execute a lock extension after the release script has already deleted the key, creating a window where a freshly acquired lock by another process is extended by a zombie thread from the previous holder. The token guard in the Lua script prevents incorrect extension, but the race is present and worth naming.

### Stock Read Occurs Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)  # ← outside transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are in separate transactional scopes. Under the assumption that the distributed lock holds, this is safe — no other writer can be between these two operations. But this assumption is precisely what the critical defect above invalidates. Even setting aside lock loss, if the database layer uses read-committed isolation, a concurrent writer that bypasses the lock (direct DB access, a bug in another service) can modify stock between the read and the write without detection. A compare-and-swap or optimistic lock at the DB layer (`UPDATE inventory SET stock = new_stock WHERE item_id = ? AND stock = current_stock`) would provide a defense-in-depth layer independent of Redis coordination.

### Test Coverage Does Not Exercise the Failure Path Being Reviewed

The test suite validates acquisition, release, insufficient stock, and input validation. There is no test that simulates the heartbeat detecting lock loss (`result == 0`) and verifies that the main thread is interrupted before the database write executes. The most critical behavioral guarantee in the system is entirely untested. A test that mocks `_extend_script` to return `0` mid-operation and asserts that `db.update_stock` is never called would be the minimum required coverage for this path.

---

## Summary of Required Changes

| Severity | Finding |
|---|---|
| **Critical** | Heartbeat detects lock loss but does not propagate it to the main thread; DB transaction executes without the lock |
| **High** | No DB-layer optimistic check; Redis coordination is the only safety layer |
| **Medium** | `t.join()` return value unchecked; zombie heartbeat thread can race against a new lock holder |
| **Medium** | No test coverage for the lock-loss-during-operation path |
| **Low** | Heartbeat interval (8s) does not match the documented TTL/3 (10s) |
| **Low** | Release path does not distinguish "released held lock" from "attempted release of lost lock" in logs |

---

## Decision: Request Changes

**Summary:** The heartbeat correctly detects lock loss via token mismatch but provides no mechanism to interrupt the main thread — the database transaction executes without mutual exclusion in exactly the failure scenario this service is designed to survive. This is not a latent edge case; under flash sale load, Redis latency spikes and GC pauses make mid-operation lock expiry a routine operational event. The fix requires a shared cancellation signal between the heartbeat thread and the main execution path, verified by a test that exercises the loss detection path end-to-end. The secondary finding regarding the stock read occurring outside the database transaction compounds this defect and should be addressed with a compare-and-swap write as a defense-in-depth layer independent of Redis coordination.
