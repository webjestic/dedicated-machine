## Code Review

### Critical Findings

#### 🔴 CRITICAL — Stock Read Outside the Database Transaction (TOCTOU)

```python
# Line executes BEFORE the transaction opens:
current_stock = self.db.get_stock(req.item_id)

# ... stock check logic ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes based on stale read
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers *at the Redis layer*, but `get_stock` executes outside the database transaction. This creates a gap:

1. The lock is acquired and stock is read as `N`.
2. The database transaction opens.
3. If the DB connection is on a different isolation level (READ COMMITTED), or if the DB itself has a replica lag, or if any other writer bypasses the Redis lock (direct DB access, a migration, an admin tool, another service node that didn't participate in the locking protocol), the `update_stock` call writes `N - quantity` against a value that may have already changed.

The correct pattern is to move `get_stock` **inside** the transaction, and ideally use a `SELECT ... FOR UPDATE` (pessimistic row lock) or an optimistic CAS update (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?`) so the database itself enforces the invariant atomically. The Redis lock alone does not make the read-modify-write atomic at the DB layer.

This is the exact failure mode my identity question targets: the lock *appears* to protect the operation, but the read and write are in different transactional scopes, so the invariant can be violated silently under concurrent or cross-system load.

---

#### 🔴 CRITICAL — Heartbeat Thread Silently Swallows Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← exits silently; caller has no idea the lock was lost
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` (lock was lost — expired or stolen), the heartbeat thread exits silently. The main thread continues executing the database transaction believing it still holds the lock. There is no signal back to the caller. The `stop_event` is not set, and no exception is raised. This means:

- The lock has expired.
- Another process has acquired it and may be reading/writing the same inventory row.
- The original thread proceeds to commit a transaction it no longer has exclusive rights to perform.

The heartbeat thread needs a communication channel back to the main thread (e.g., a shared flag, a `threading.Event`, or a `concurrent.futures.Future`) so the main thread can abort the operation when lock loss is detected.

---

#### 🔴 CRITICAL — Lock Release After Confirmed Lock Loss

In the `finally` block:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The Lua release script is token-gated, so it won't delete a lock held by another process — that part is correct. However, if the lock expired and was re-acquired by another process, the `join` waits up to `HEARTBEAT_INTERVAL + 1` seconds (9 seconds) before proceeding. During those 9 seconds, the new lock holder may be mid-operation. This delay is a correctness hazard in timing-sensitive scenarios and a latency hazard in all scenarios. The `join` timeout should be much shorter, or the thread should be designed to exit promptly on `stop_event`.

---

#### 🟡 HIGH — `t.join` Can Silently Time Out Without Detection

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If `t.join` times out (the heartbeat thread is stuck — e.g., Redis call hanging), the code proceeds to release the lock anyway. There is no check of `t.is_alive()` after the join. A stuck heartbeat thread is a daemon thread, so it won't block process exit, but the release call proceeds without knowing whether the heartbeat is still running and potentially mid-extend. This should at minimum log a warning when `t.is_alive()` is `True` after join.

---

#### 🟡 HIGH — Database Transaction Does Not Roll Back on Heartbeat Loss

Because the heartbeat loss is not communicated to the main thread (see second critical finding), there is no path where the database transaction is aborted when the lock is lost mid-operation. The `with self.db.transaction()` block will commit even if the Redis lock expired before the commit. The transaction manager has no awareness of the distributed lock state.

---

#### 🟡 MEDIUM — `EXTEND_LOCK_SCRIPT` Uses `expire` (Relative), Not `expireat` (Absolute)

```lua
return redis.call("expire", KEYS[1], ARGV[2])
```

`EXPIRE` sets a TTL relative to the moment of execution. If the heartbeat fires late (e.g., the Python process was GC-paused, the event loop was delayed, or the Redis call itself was slow), the effective lock lifetime extends beyond the intended `LOCK_TTL` window from the *original* acquisition. This is a minor drift issue but can compound under load. `EXPIREAT` with a computed absolute deadline anchored to acquisition time would be more precise.

---

#### 🟡 MEDIUM — No Upper Bound on `quantity`

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity validation. A caller can request `quantity=2_000_000_000`. This will pass validation, read stock, compute `new_stock = current_stock - 2_000_000_000` (a large negative number), and — depending on the DB schema — either fail at the DB layer with an error or, if the column is unsigned and wraps, corrupt inventory data. An `le` bound should be enforced at the model level.

---

#### 🟡 MEDIUM — `item_id` Not Normalized Before Lock Key Construction

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`req.item_id` is validated to `^[a-zA-Z0-9_\-]+$`, which is good. However, if the system ever relaxes this regex or if `item_id` values are case-sensitive in some paths and case-insensitive in others (e.g., `Item_001` vs `item_001`), two callers could acquire different locks for the same inventory row. This is a latent correctness risk. Explicit normalization (e.g., `.lower()`) should be applied before key construction.

---

#### 🟢 LOW — Test Coverage Gaps

The tests do not cover:

1. **Lock loss mid-operation** — the most critical failure path is untested.
2. **Heartbeat thread behavior** — no test verifies the extend script is called or that lock loss is detected.
3. **DB transaction rollback** — no test verifies that `update_stock` is not committed when the transaction raises.
4. **`t.join` timeout** — no test for a stuck heartbeat thread.
5. **Concurrent reservation** — `test_reserve_concurrent_quantity` is not actually concurrent; it's a sequential mock test.

The test for `test_lock_release_uses_lua_script` is structurally correct but trivially satisfied by the mock setup.

---

### What Is Done Well

- Atomic lock acquisition via `SET NX EX` is correct.
- Lua scripts for release and extension are correctly token-gated — no unconditional deletes.
- Exponential backoff with jitter is appropriate for contention scenarios.
- Pydantic validation with regex on `item_id` prevents injection into the lock key namespace.
- Daemon thread prevents heartbeat from blocking process shutdown.
- `stop_event` using `wait()` as a timer (rather than `sleep`) is the correct pattern for interruptible heartbeats.
- `ValidationError` is caught and converted to a typed result rather than propagated as an exception.

---

## Decision: Request Changes

**Summary:** The implementation contains two critical correctness flaws that would cause inventory corruption under real concurrent load: the stock read occurs outside the database transaction (making the Redis lock insufficient to prevent a TOCTOU race against any writer that bypasses the lock), and heartbeat thread lock-loss events are silently discarded with no signal to the main thread, allowing a transaction to commit after the distributed lock has expired. These must be resolved — specifically by moving `get_stock` inside the transaction with a DB-level row lock or atomic CAS update, and by establishing a communication channel from the heartbeat thread to the main execution context — before this can be approved.
