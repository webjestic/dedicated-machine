Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, heartbeat renewal, token-scoped release, and jitter backoff are all sound. However, there are several correctness, safety, and architectural issues that must be addressed before this is mergeable, ranging from a silent exception-swallowing bug in the heartbeat thread to a TOCTOU race between the stock read and the database write, to missing test coverage for critical failure paths.

---

## Findings

### 🔴 Critical

**1. Heartbeat thread silently swallows all exceptions**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return  # ← silent exit; caller never knows the lock was lost
        logger.debug("Lock TTL extended key=%s", lock_key)
```

If `_extend_script` raises (Redis connection drop, network partition, serialization error), the exception is swallowed, the thread exits, the lock expires, and `reserve_item` continues executing under the false belief it still holds the lock. This is a phantom-lock scenario — the most dangerous failure mode in distributed locking.

Additionally, when `result == 0` (token mismatch — lock was stolen or expired), the thread returns silently. The parent context has no mechanism to detect this and will proceed to commit a database transaction without holding the lock.

**Required fix:** The heartbeat must have a signaling path back to the main thread. Use a shared `threading.Event` for lock-loss notification, check it before the DB transaction, and surface the failure:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = self._extend_script(
                keys=[lock_key],
                args=[token, str(self.LOCK_TTL)]
            )
            if result == 0:
                logger.error("Lock lost during heartbeat key=%s", lock_key)
                lock_lost_event.set()
                return
        except Exception:
            logger.exception("Heartbeat extension failed key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then, before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost mid-operation for {lock_key}")
```

**2. TOCTOU race between stock read and database write**

```python
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
# ... logic ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
    self.db.create_order(...)
```

The stock read happens outside the database transaction. If the DB layer supports any read isolation below `SERIALIZABLE`, another process (one not respecting the Redis lock, a background job, a direct DB write, a migration) can modify stock between the read and the write. The Redis lock protects against concurrent callers of this service, but it does not protect against anything that touches the database directly.

**Required fix:** Move `get_stock` inside the transaction boundary, or use a `SELECT ... FOR UPDATE` / optimistic locking with a version column, or at minimum add a conditional update with a stock floor check:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # row-level lock
    if current_stock < req.quantity:
        # rollback and return
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

This is defense-in-depth. The Redis lock is your first line; the DB transaction with a row lock is your second. Neither alone is sufficient.

### 🟠 High

**3. `t.join()` timeout does not guarantee thread termination before lock release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← join can time out
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns regardless of whether the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call, `join` returns after the timeout and the Lua release script runs. Now both the heartbeat thread and the finally block are racing to interact with the same Redis key. The heartbeat may extend the TTL of a key that was just deleted, or worse, extend a key that has already been re-acquired by another process with a different token (the Lua script guards against this for release, but the extend script firing after release is still noise and a potential source of confusion in incident debugging).

**Required fix:** Check `t.is_alive()` after join and log a warning if the thread did not terminate. Consider whether a non-daemon thread with a hard timeout and a forced stop is more appropriate for your SLA.

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error(
            "Heartbeat thread did not terminate within timeout key=%s — "
            "proceeding with lock release; thread may still be running.",
            lock_key
        )
    self._release_script(keys=[lock_key], args=[token])
```

**4. Retry backoff is too aggressive for a flash sale workload**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3` and `LOCK_TTL=30s`, the maximum wait across all retries is approximately `0.1 + 0.2 + 0.4 = 0.7s` plus jitter. For a flash sale with hundreds of concurrent requests, this means most requests fail within under one second and return "service temporarily unavailable" to the user, even though the lock holder will release within milliseconds in the common case.

The jitter range (`0–0.05s`) is also very narrow. Under a thundering herd, all waiters will retry at nearly the same time, defeating the purpose of jitter. The standard recommendation (AWS, Google SRE) is full jitter: `random.uniform(0, cap)` where `cap` grows exponentially.

**Recommended fix:**

```python
cap = min(2.0, (2 ** attempt) * 0.1)
backoff = random.uniform(0, cap)  # full jitter
```

Also consider making `retry_attempts` a constructor parameter so it can be tuned per deployment environment.

### 🟡 Medium

**5. `quantity` field has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity constraint. A caller can request `quantity=999999999` and the only thing that stops them is the stock check. This is not a security vulnerability in isolation, but it is an input validation gap that should be closed at the schema layer. Define a business-rule maximum:

```python
quantity: int = Field(..., ge=1, le=1000)  # or whatever your business rule is
```

---

**6. `item_id` length limit may be insufficient for some ID schemes, and the pattern should be documented**

The `max_length=128` and `pattern=r'^[a-zA-Z0-9_\-]+$'` are reasonable, but the pattern is not documented as a business rule anywhere. If a downstream team adds a new item ID scheme (e.g., UUIDs with colons, or namespaced IDs like `sku:region:variant`), this will silently reject valid items. Add a comment explaining the constraint rationale.

---

**7. Lock key namespace collision risk**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If `item_id` values are not globally unique across all services sharing this Redis instance, lock keys will collide. This is not validated anywhere. If this Redis instance is shared (common in microservice deployments), consider prefixing with a service or environment identifier:

```python
lock_key = f"{self.key_prefix}:inventory:lock:{req.item_id}"
```

Where `key_prefix` is injected at construction time.

### 🔵 Test Coverage Gaps

**8. No test for heartbeat lock-loss signaling**

The most critical new behavior — the heartbeat detecting a lost lock and signaling the main thread — has zero test coverage. This is the exact scenario that would cause a silent double-reservation in production.

**9. No test for DB transaction failure / rollback**

There is no test verifying that a database exception inside `self.db.transaction()` does not result in a partial write (stock decremented but order not created, or vice versa). The transaction context manager should be tested with a raised exception to confirm rollback behavior.

**10. No test for heartbeat thread join timeout**

The case where `t.join()` times out (heartbeat thread stuck) is untested. This is a real failure mode under Redis latency spikes.

**11. `test_lock_release_uses_lua_script` is fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert the arguments. A future refactor could call the script with wrong keys or a wrong token and this test would still pass. Assert the full call:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID; verify it's a string if needed
)
```

- Lua scripts for atomic release and extension are correct and idiomatic.
- `SET NX EX` for atomic acquisition is the right primitive.
- Token-scoped lock (UUID per acquisition) correctly prevents a process from releasing a lock it no longer owns.
