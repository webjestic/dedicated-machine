Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all sound. However, there are several correctness and reliability issues that must be addressed before this merges: a race condition between the DB read and transaction, missing exception handling around DB and Redis operations, and a gap in the heartbeat's failure signaling. Test coverage also has meaningful blind spots that need to be closed.

---

## Findings

### 1. TOCTOU Race on Stock Read (High)

**File:** `services/inventory.py`, `reserve_item`

```python
# Inside the lock — looks safe, but read is outside the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens **outside** the database transaction. The distributed lock serializes Redis-side access, but if the DB has any replica lag, a separate non-locking read path exists, or the lock is ever lost mid-operation (Redis eviction under memory pressure, network partition), the stock check and the write are no longer atomic at the DB layer.

**Required fix:** Move `get_stock` inside the transaction, or use a `SELECT FOR UPDATE` / optimistic version check inside the transaction so the DB itself enforces the invariant.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Unhandled DB and Redis Exceptions (High)

**File:** `services/inventory.py`, `reserve_item` and `_acquire_lock`

Only `LockAcquisitionError` and `ValidationError` are caught. Any exception from `db.get_stock`, `db.update_stock`, `db.create_order`, or the Redis scripts propagates uncaught to the caller. In a flash-sale context under load, transient DB timeouts or Redis connection errors will surface as unhandled exceptions rather than graceful `ReservationResult(success=False, ...)` responses.

**Required fix:** Wrap the body of `reserve_item` (inside the lock) in a `try/except Exception` that logs and returns a safe failure result. Separately handle `redis.RedisError` in `_acquire_lock`.

```python
except (redis.RedisError, Exception) as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
```

Be specific with exception types — don't swallow everything silently. Log with `exc_info=True` so the stack trace is preserved.

### 3. Heartbeat Silently Swallows Lock Loss (Medium)

**File:** `services/inventory.py`, `_beat`

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # lock was lost — exits silently
```

When the heartbeat detects the lock is gone (`result == 0`), it exits the thread without signaling the main thread. The main thread continues executing the DB transaction against an inventory it no longer holds the lock on. This is the exact failure mode the heartbeat is supposed to prevent.

**Required fix:** Use a shared flag or event to signal the main thread that the lock was lost, and check it before and after the DB transaction.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat lost lock key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then in `reserve_item`, after the transaction:

```python
if lock_lost_event.is_set():
    # Attempt compensating rollback or flag the order as suspect
    raise LockAcquisitionError("Lock lost during transaction")
```

### 4. Early Return Inside Lock Leaks Cleanly, But Logging Is Misleading (Low)

**File:** `services/inventory.py`, `reserve_item`

```python
if current_stock is None:
    return ReservationResult(success=False, message=f"Item {req.item_id} not found")
```

Returning inside the `with self._acquire_lock(lock_key):` block is fine — the context manager's `finally` will run. However, the `logger.info("Lock released ...")` fires for every early return including not-found and insufficient-stock paths, which is noise in structured logs and makes log-based alerting harder to tune.

**Suggested fix:** Log lock release at `DEBUG` for non-error paths, or distinguish between "released after successful reservation" and "released after early exit."

---

### 5. `quantity` Has No Upper Bound (Low)

**File:** `services/inventory.py`, `ReservationRequest`

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound on `quantity`. A caller can submit `quantity=10_000_000` and the validation passes. For a flash-sale context this is almost certainly wrong and could be used to drain inventory in a single request or expose integer overflow behavior downstream.

**Required fix:**

```python
quantity: int = Field(..., ge=1, le=1000)  # tune ceiling to business rules
```

### 6. Test Coverage Gaps (Medium)

**File:** `tests/test_inventory.py`

The existing tests are well-structured, but these paths are untested:

| Missing Test | Risk |
|---|---|
| Heartbeat lock-loss path (`_extend_script` returns 0) | Core safety mechanism unverified |
| DB exception during `get_stock` or `update_stock` | Unhandled exception path |
| DB transaction rollback on `create_order` failure | Partial write scenario |
| `item_id` at max length boundary (128 chars) | Boundary validation |
| Retry exhaustion with partial success (lock acquired on attempt 2 or 3) | Backoff logic unverified |

The `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args` were passed. A refactor that passes the wrong token would not be caught.

```python
service._release_script.assert_called_once_with(keys=[lock_key], args=[token])
```

## What Is Working Well

- Lua scripts for atomic release and extension are correct and well-commented.
- `stop_event.wait(timeout=...)` as a timer (not `time.sleep`) is the right pattern — allows clean shutdown.
- Heartbeat interval at TTL/3 (8s of 30s) is a reasonable safety margin.
- `SET NX EX` is atomic — no separate `EXISTS` + `SET` split.
- Pydantic validation with regex on `item_id` closes injection surface.
- `daemon=True` on the heartbeat thread prevents it from blocking process shutdown.
- Exponential backoff with jitter is correct.
- `t.join()` before `_release_script` ensures the heartbeat cannot extend after intentional release.

## Required Before Approval

1. Move stock read inside the DB transaction (or use `SELECT FOR UPDATE`)
2. Add exception handling for DB and Redis errors in `reserve_item`
3. Signal main thread when heartbeat detects lock loss
4. Add upper bound to `quantity` field
5. Add tests for DB exception paths and heartbeat lock-loss path
