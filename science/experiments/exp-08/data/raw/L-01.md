## Code Review

### Critical Issues

**1. Stock read occurs outside the database transaction (TOCTOU within the lock)**

```python
# Inside the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

# Only THEN does the transaction open:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The Redis lock serializes *this service's* access. But if any other process, service, or admin tool can write to the database directly — or if the DB connection uses `READ COMMITTED` isolation — the stock value read before the transaction is not the stock value at the moment of write. The lock does not extend to the database layer. The correct pattern is to perform the read *inside* the transaction with a `SELECT FOR UPDATE` or equivalent pessimistic row lock, so the read and write are atomic at the database level. As written, this is a TOCTOU window between the `get_stock` call and the `update_stock` call, even with the Redis lock held.

**2. Heartbeat thread silently swallows lock loss — no propagation to the main thread**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← lock was lost; main thread has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread exits silently. The main thread continues executing — reading stock, opening a transaction, writing data — believing it holds a lock it no longer holds. Another process may have acquired the lock and begun its own transaction. The database transaction will proceed concurrently with another holder's transaction, defeating the entire purpose of the distributed lock.

The heartbeat must have a signaling path back to the main thread. A `threading.Event` for lock loss, a shared flag, or a queue is required. The main thread must check this signal at critical points (before and inside the transaction).

**3. Lock release after silent lock loss releases another process's lock**

This is a consequence of issue #2. If the heartbeat exits because `result == 0` (token mismatch — the lock was re-acquired by another process with a new token), the `finally` block still calls:

```python
self._release_script(keys=[lock_key], args=[token])
```

The Lua script correctly checks the token before deleting, so this specific call is safe — it will return 0 and not delete the new holder's lock. This is the one place the Lua token-check saves correctness. However, the main thread has still executed business logic without a valid lock, which is the real damage.

**4. `t.join(timeout=...)` does not confirm the thread actually stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=N)` returns after N seconds regardless of whether the thread has exited. If the heartbeat thread is blocked on a slow Redis call, the release script executes while the heartbeat thread may still be running and attempting to extend the lock. The code should check `t.is_alive()` after join and log a warning or raise if the thread did not terminate. In a production system this is a resource leak and a correctness concern.

**5. No upper bound on `quantity` — denial of service / inventory manipulation**

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound. A caller can submit `quantity=999999999` and the service will acquire a lock, read stock, find insufficient stock, and return — but at scale this is a valid DoS vector against the locking layer. More critically, if stock is ever seeded incorrectly and a large quantity passes the stock check, the resulting `new_stock` could go deeply negative. A `le=` bound appropriate to the business domain should be enforced at the validation layer.

### Moderate Issues

**6. Exponential backoff ceiling is extremely low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45s`. Under a genuine flash sale load spike, 3 attempts over ~0.55 seconds total is likely insufficient. The retry count and backoff ceiling should be configurable and tuned to the expected lock hold time (`LOCK_TTL = 30s`). Callers may want to configure retry behavior per call site.

**7. `db` interface is entirely implicit — no contract, no error handling**

`self.db.get_stock`, `self.db.update_stock`, `self.db.create_order`, and `self.db.transaction` are called with no interface definition, no type annotation, and no exception handling. If any of these raise, the exception propagates out of the `with self._acquire_lock(lock_key):` block, the `finally` executes, the lock is released — but the caller receives an unhandled exception rather than a `ReservationResult`. The public method `reserve_item` should catch database-layer exceptions and return a structured failure result, consistent with how `LockAcquisitionError` is handled.

**8. Redis connection errors are unhandled**

If `self.redis.set(...)` raises `redis.RedisError` (connection refused, timeout, cluster failover), the exception propagates uncaught through `_acquire_lock` and out of `reserve_item`. This should be caught and converted to a `ReservationResult(success=False, ...)` or a well-defined service exception, not a raw Redis exception surfaced to the caller.

### Test Coverage Gaps

**9. The most critical failure mode is not tested**

There is no test for the scenario where the heartbeat detects lock loss (`result == 0`) while the main thread is mid-transaction. This is the highest-severity bug in the implementation, and the test suite does not cover it at all — which means it would not catch a fix either.

**10. `test_lock_release_uses_lua_script` tests mock behavior, not correctness**

```python
service._release_script.assert_called_once()
```

This asserts that the mock registered script was called once. It does not verify the arguments (`keys=[lock_key]`, `args=[token]`), so it would pass even if the wrong key or a null token were passed. Use `assert_called_once_with(keys=[...], args=[...])`.

**11. No test for database exception during transaction**

There is no test verifying behavior when `db.update_stock` or `db.create_order` raises. The lock should still be released in the `finally` block, but this is untested.

**12. No test for Redis failure during lock acquisition**

`redis_client.set.side_effect = redis.RedisError(...)` is not tested. Given issue #8, this would currently surface as an unhandled exception.

---

### Minor Issues

- `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants with no configuration injection. For a service deployed across environments with different Redis latency profiles, these should be injectable via constructor parameters with documented defaults.
- The `_beat` thread is `daemon=True`, which means if the process exits abnormally, the lock may not be released. This is acceptable for a daemon thread, but the operational implication should be documented — lock TTL is the safety net, and 30 seconds is the worst-case stale lock window.
- `logger.info("Lock released key=%s", lock_key)` logs after the script call but does not log whether the release returned 0 (token mismatch) or 1 (successful delete). This makes post-incident log analysis harder.

## Decision

**Decision:** Request Changes

**Summary:** The implementation has two critical correctness flaws that would cause inventory corruption under concurrent flash sale load: the stock read occurs outside the database transaction, creating a TOCTOU window that the Redis lock does not close, and the heartbeat thread silently discards lock-loss signals without notifying the main thread, allowing business logic to execute without a valid lock. Additionally, database and Redis exceptions are unhandled at the public API boundary, and the test suite does not cover the highest-severity failure scenarios. All critical issues must be resolved before this can be approved.
