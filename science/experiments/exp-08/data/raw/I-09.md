Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic Lua scripts for lock release/extension, scoped heartbeat state, input validation, and structured logging are all well-executed. However, there are several correctness, safety, and architectural issues that must be addressed before this is suitable for a high-concurrency flash sale environment. The most critical is a TOCTOU race between the stock read and the database write, which the distributed lock does not fully protect against without a corresponding database-level constraint.

---

## Detailed Review

### 🔴 Critical Issues

---

#### 1. TOCTOU Race — Stock Read Outside the Database Transaction

```python
# Line executes OUTSIDE the transaction block
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes a stale value
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The distributed lock serializes Redis-level access, but `get_stock` is called outside the database transaction. If the DB read and the DB write are not atomic at the database level, another process that bypasses Redis (a migration, a manual correction, a different code path, a Redis failover event) can modify stock between the read and the write. The `update_stock` call then commits a value derived from stale data.

**Required fix:** Either move the read inside the transaction and use a `SELECT FOR UPDATE` (or equivalent pessimistic row lock), or use an optimistic CAS-style update:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle inside transaction
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Or a conditional update with affected-row checking:

```python
rows_affected = self.db.decrement_stock_if_sufficient(req.item_id, req.quantity)
if rows_affected == 0:
    raise InsufficientStockError()
```

This is the most dangerous flaw in the PR. Inventory overselling is a direct business and financial integrity failure.

#### 2. Heartbeat Thread Silently Swallows Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return  # ← exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread exits, but the main thread has no awareness of this. It continues executing under the assumption that it holds the lock — reads stock, writes to the database — with no exclusive access guarantee. This is the exact failure mode a heartbeat is supposed to surface.

**Required fix:** Signal the main thread on lock loss. A simple approach:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then in the main thread, check `lock_lost_event` before committing:

```python
if lock_lost_event.is_set():
    raise LockLostError(f"Lock was lost during operation for {lock_key}")
```

There is still an inherent race window between the check and the commit, which is why fix #1 (database-level locking) is non-negotiable regardless.

#### 3. Unhandled Exceptions Inside the Lock Context Leak State

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `db.get_stock`, `db.update_stock`, or `db.create_order` raises an unexpected exception (network timeout, DB connection drop, etc.), it propagates out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` will correctly release the lock, which is good. However, `reserve_item` has no outer `except Exception` handler, so the exception propagates to the caller as an unhandled crash rather than a graceful `ReservationResult`.

**Required fix:**

```python
except LockAcquisitionError as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal service error")
```

---

#### 4. `LOCK_TTL` of 30 Seconds Is Dangerously Long for a Flash Sale

A 30-second lock TTL means a crashed or hung process holds the lock for up to 30 seconds. In a flash sale context where thousands of requests may be queued, this is a denial-of-service against your own inventory system. The `HEARTBEAT_INTERVAL` of 8 seconds (TTL/3) is a reasonable ratio, but the base TTL should be tuned to the actual expected operation duration.

If the DB transaction is expected to complete in under 2 seconds under normal conditions, a TTL of 5–10 seconds with a heartbeat interval of 1–2 seconds is more appropriate. The current values suggest these constants were chosen arbitrarily rather than benchmarked.

**Required:** Document the basis for these values or make them configurable per-item-class.

---

#### 5. Retry Backoff Is Insufficient for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait time before failure is approximately:

- Attempt 0: 0ms (immediate)
- Attempt 1: ~105ms
- Attempt 2: ~205ms
- Total: ~310ms before `LockAcquisitionError`

Under flash sale load with hundreds of concurrent requests per item, nearly all of them will fail immediately. This is by design for a lock (you don't want unbounded queuing), but the caller receives a generic "unavailable" message with no retry hint. Consider returning a `Retry-After` style signal or using a short-lived queue/semaphore pattern upstream.

Also, the jitter range of `uniform(0, 0.05)` (50ms max) is negligible relative to the backoff. It provides almost no thundering-herd protection. Recommend `uniform(0, backoff * 0.5)` for proportional jitter.

### 🟡 Minor Issues

---

#### 6. Test for Lock Release Is Fragile and Tests Implementation Detail

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts on `_release_script` directly, which is a private implementation detail. It will break on any internal refactor. More importantly, it does not verify that the script was called with the correct `keys` and `args`. A call with the wrong token would pass this test.

**Recommended:**

```python
service._release_script.assert_called_once_with(keys=[expected_lock_key], args=[expected_token])
```

Though accessing the token from outside the context manager requires a small refactor to expose it for testing, or using a spy/capture approach.

---

#### 7. No Test for Lock-Lost-During-Operation Scenario

There is no test covering the case where the heartbeat detects lock loss mid-operation. This is the most dangerous runtime failure mode and should have explicit coverage once fix #2 is implemented.

---

#### 8. `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` — Join Timeout Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise if the thread is still alive after the timeout. If the heartbeat thread hangs (e.g., Redis call blocks), the join silently returns and the lock release proceeds. This is acceptable behavior, but it should be logged:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

#### 9. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=999999999`. This passes validation and will result in `new_stock` going deeply negative if the stock check is ever bypassed (e.g., due to the TOCTOU issue above). Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)
```

The appropriate ceiling is a business decision, but no ceiling at all is a validation gap.
