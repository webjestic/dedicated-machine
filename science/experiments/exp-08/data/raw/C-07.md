Decision: Request Changes

Summary: The implementation demonstrates genuine architectural awareness — scoped heartbeat state, Lua-atomic release and extension, token-bound ownership, and validated input are all present and correctly reasoned. However, there is one structural flaw that the test suite does not exercise and the checklist does not name: the heartbeat thread silently absorbs extension failures without surfacing that information to the protected operation, meaning the code inside the lock context can complete a database write against state it no longer owns. There are also secondary issues in the stock-read/write gap and the `t.join` placement that compound the primary concern.

---

## Detailed Findings

### 1. [Critical] Heartbeat Loss Is Not Communicated to the Protected Operation

This is the gap between the examples in the system prompt and what the code actually does.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # ← thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread exits. The lock is gone — either expired or stolen by another process. The protected operation in `reserve_item` does not know this. It continues, reads stock, opens a database transaction, writes `update_stock`, and creates an order, all against state it no longer exclusively owns.

The failure mode is not in the heartbeat's detection of loss. The detection is correct. The failure is in the gap between when the lock is lost and when the caller learns it. That gap is unbounded — it spans the entire remaining duration of the `with self._acquire_lock(lock_key):` block.

A second process can acquire the lock the moment the first process's TTL expires. Both processes can now be inside the stock-read/write path simultaneously. The database transaction provides atomicity for the write pair, but it does not protect against two processes each reading `current_stock = 10`, each computing `new_stock = 9`, and each committing. Stock goes to 9 twice; ten units were sold.

**Required fix:** The heartbeat must signal lock loss back to the protected operation. One correct pattern is a shared `threading.Event` that the heartbeat sets on `result == 0`, and the protected operation checks at meaningful boundaries (after `get_stock`, before committing). Alternatively, the heartbeat can set a flag that the context manager checks in the `finally` block and raises into the caller. Either way, silent exit is not acceptable.

```python
# Sketch — not a complete implementation
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()   # caller can observe this
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the context manager finally block or yielded token wrapper,
# check lock_lost before allowing the write path to proceed.
```

### 2. [Moderate] Stock Read Occurs Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

    # ...

    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The read and the write are not in the same transaction. If the database layer has any read-committed or snapshot isolation behavior, or if `get_stock` goes to a read replica, the value read may not reflect the value that `update_stock` will act on. The distributed lock is doing the coordination work here, but the read should be inside the transaction boundary to make the atomicity guarantee explicit and not dependent solely on the lock.

Move `get_stock` inside `self.db.transaction()`, or use a `SELECT FOR UPDATE` equivalent if the database supports it.

---

### 3. [Moderate] `t.join` Executes Before Lock Release, Blocking the Release Path

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)        # ← blocks up to 9 seconds
    self._release_script(keys=[lock_key], args=[token]) # ← release happens after join
```

`stop_event.set()` wakes the heartbeat thread immediately if it is sleeping in `wait()`. In the normal case, the join is fast. But if the heartbeat thread is mid-extension when `stop_event` is set, the join waits for that extension to complete before the release script runs. During that window, the lock is still held and the TTL may have just been extended. This is a minor latency concern in normal operation but becomes a correctness concern if the extension thread is blocked on a slow Redis call — the lock is held longer than intended with no protected operation running.

More importantly, if `t.join` times out (the heartbeat thread is hung), the release script still runs, which is correct. But the timeout value of `HEARTBEAT_INTERVAL + 1 = 9` seconds is long for a `finally` block in a web-facing service. Consider a shorter timeout and a warning log on timeout expiry.

---

### 4. [Minor] Heartbeat Extension Failure Is Only Logged at `DEBUG`

```python
if result == 0:
    return
logger.debug("Lock TTL extended key=%s", lock_key)
```

There is no log statement on `result == 0`. A silent return on lock loss means the only signal of this event is the absence of future debug logs. This should log at `ERROR` or `WARNING` with the lock key and token, so that operational monitoring can detect it.

---

### 5. [Minor] Test Coverage Does Not Exercise the Lock-Loss Path

The test suite covers acquisition failure, insufficient stock, not-found, input validation, and Lua release. It does not test the scenario where the heartbeat detects `result == 0` mid-operation. There is no test asserting that a reservation fails (or is rolled back) when the lock is lost during the database write. This is the most important failure mode in the system and it has zero test coverage, which is consistent with the fact that the code does not handle it.

---

## What Is Done Well

- Token-bound lock ownership via `uuid4` is correct.
- Lua-atomic release prevents accidental deletion of another process's lock.
- Lua-atomic extension with token check is the right mechanism — the detection logic in the heartbeat is sound; the problem is only in what happens after detection.
- `stop_event.wait(timeout=...)` as a timer rather than `time.sleep` is the correct pattern for interruptible heartbeat threads.
- Exponential backoff with jitter is appropriate for lock contention.
- Pydantic validation with pattern enforcement on `item_id` closes an injection surface.
- Daemon thread prevents the heartbeat from blocking process shutdown.
- The checklist is honest about what it claims — it does not claim lock-loss handling, which is accurate.

---

## Required Before Approval

1. Heartbeat loss must be communicated to the protected operation and must prevent the database write from completing.
2. `get_stock` must be inside the database transaction boundary.
3. Log at `WARNING` or `ERROR` when the heartbeat detects `result == 0`.
4. Add a test that simulates lock loss mid-operation and asserts the reservation does not commit.
