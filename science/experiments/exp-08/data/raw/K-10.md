Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all sound. However, there are several correctness and reliability issues that must be resolved before this merges: a race condition between the DB read and transaction, missing exception handling around Redis and DB operations, and a gap in the heartbeat's failure signaling. Test coverage also has a structural flaw that masks the lock release assertion.

---

## Findings

### 1. TOCTOU Race on Stock Read (High)

```python
# Inside the lock — correct so far
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes pre-computed value
    self.db.create_order(...)
```

`get_stock` runs outside the transaction. If the DB supports concurrent writers (e.g., another process bypasses Redis), the stock value read before `transaction()` may be stale by the time the write commits. The read must occur **inside** the transaction, or the update must use a conditional decrement (e.g., `UPDATE ... WHERE stock >= quantity AND stock = :expected`), not a blind write of a pre-computed value.

```python
# Preferred pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle inside transaction
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

### 2. Unhandled Exceptions from Redis and DB (High)

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

No exception handling exists for:

- `redis.RedisError` during lock acquisition or heartbeat extension
- `redis.RedisError` during Lua script execution in `finally`
- Any DB exception raised inside or outside the transaction block

A DB exception inside `with self.db.transaction()` will propagate through `_acquire_lock`'s `finally`, which is correct for lock release, but `reserve_item` will raise an unhandled exception to the caller rather than returning a `ReservationResult`. This breaks the contract the method signature implies.

```python
# Minimum required in reserve_item
except (redis.RedisError, SomeDatabaseError) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

Also, the `_release_script` call in `finally` can itself raise — that must be wrapped:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
    except redis.RedisError:
        logger.error("Lock release failed key=%s — TTL will expire naturally", lock_key)
```

### 3. Heartbeat Failure Is Silent (Medium)

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # lock was lost — exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits but the main thread has no way to know the lock was lost. The operation continues under the assumption it still holds the lock, which is the exact guarantee a distributed lock is meant to provide. A threading.Event or similar signal should be used to notify the main thread.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the main thread, after yield or at a checkpoint:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock expired during operation: {lock_key}")
```

### 4. Heartbeat Thread Not Joined on Lock Acquisition Failure (Low)

```python
if not acquired:
    raise LockAcquisitionError(...)
```

This path is correct — no thread is started before this point. No issue here. However, if `acquired` is `True` and the `yield` raises before the `finally` block runs (which cannot happen given Python's context manager guarantees), the thread is covered. This is fine as written, but worth a comment for maintainability.

### 5. Test Structural Flaw — `_release_script` Assertion Is Unreliable (Medium)

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is assigned during `__init__` from `redis_client.register_script(...)`, which returns a `MagicMock`. The assertion `assert_called_once()` verifies the mock was called once total — but because the fixture shares the same mock across all registered scripts (`register_script` always returns the same `script_mock`), both `_release_script` and `_extend_script` point to the same object. A call to either will satisfy this assertion. Use `side_effect` or separate return values per `register_script` call to isolate them:

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    release_mock = MagicMock(return_value=1)
    extend_mock = MagicMock(return_value=1)
    mock.register_script.side_effect = [release_mock, extend_mock]
    return mock
```

### 6. Missing Test Cases (Low)

The following scenarios have no coverage:

| Scenario | Risk |
|---|---|
| `redis.RedisError` during `get_stock` or transaction | Unhandled exception surfaces to caller |
| Heartbeat detects lock loss (`result == 0`) | Silent failure, no caller notification |
| `quantity` at `int` boundary (e.g., `2**63`) | Pydantic passes it; DB layer may not |
| `db.transaction().__exit__` called with an exception | Rollback path untested |

## What Is Done Well

- Lua scripts for atomic release and extension are correct
- `SET NX EX` is the right primitive — no separate TTL set
- `stop_event.wait(timeout=N)` as a timer is correct; avoids busy-wait
- `t.join()` before `_release_script` in `finally` is the right ordering
- Pydantic validation with regex on `item_id` closes injection surface
- Exponential backoff with jitter is appropriate for lock contention
- `daemon=True` on the heartbeat thread prevents process hang on abnormal exit
- Structured logging with consistent field names throughout
