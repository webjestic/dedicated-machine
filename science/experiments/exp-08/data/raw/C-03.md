Decision: Request Changes

Summary: The implementation is thoughtful — the heartbeat design, Lua atomicity, and input validation are all sound. However, there is a silent correctness failure in the gap between lock loss and code awareness, and a secondary race in the release path, either of which can corrupt inventory state under real flash-sale load. These must be resolved before merge.

---

## Critical Issues

### 1. Heartbeat Loss Is Not Propagated to the Protected Operation

This is the central flaw. When the heartbeat thread calls `_extend_script` and receives `0`, it logs nothing, returns silently, and stops renewing. The lock has now expired in Redis. Another process can acquire it. But `reserve_item` has no awareness of this — it continues reading stock, computing `new_stock`, and writing both `update_stock` and `create_order` as if the lock still holds.

```python
# _beat():
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return   # <-- silent exit. Protected code is not notified.
```

```python
# reserve_item() — running concurrently with the above:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes against invalidated coordination
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The failure mode: two processes hold the lock simultaneously after the first loses it silently. Both read the same stock value. Both write. You get double-spend on inventory — exactly the invariant this service exists to protect.

**Required fix:** The heartbeat must signal failure back to the critical section. One correct pattern is a shared `threading.Event` (`lock_lost_event`) that the heartbeat sets on extension failure. The protected code checks it before committing, and the `finally` block skips release if the lock was already lost.

```python
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()  # signal the critical section
            logger.error("Lock lost during operation key=%s", lock_key)
            return

# In reserve_item, before the DB write:
if lock_lost.is_set():
    raise LockLostError(f"Distributed lock expired mid-operation for {lock_key}")
```

The check must happen *before* the transaction commits, not after.

---

### 2. Release-After-Expiry Is Not Guarded

In the `finally` block, `stop_event.set()` and `t.join()` happen before `_release_script`. If the heartbeat already set `lock_lost`, the lock in Redis either expired naturally or was acquired by another process. Calling `_release_script` at this point is safe only because the Lua script checks the token — but the caller has no way to distinguish "we released our own lock" from "lock was already gone." This is acceptable *if* issue 1 is fixed and the operation was aborted. If issue 1 is not fixed and the operation committed, this becomes a correctness hole: the write happened under no valid coordination, and the release call is cosmetic.

Fix is contingent on fix 1: abort the operation when `lock_lost` is set, then the release path is a no-op against an already-expired key, which is fine.

## Moderate Issues

### 3. `get_stock` Runs Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)   # outside transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The stock read and the stock write are not in the same transaction. If the DB layer uses read-committed isolation (common default), another writer could modify stock between the read and the write. The Redis lock serializes callers *through this service*, but it does not protect against writes from other services or direct DB access. The read should be inside the transaction, and `update_stock` should use a conditional write (`WHERE stock = current_stock` or `WHERE stock >= quantity`) rather than an unconditional set.

---

### 4. Heartbeat Thread Leak on `_extend_script` Exception

```python
def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)  # if this raises, thread exits without setting lock_lost
        if result == 0:
            return
```

If `_extend_script` raises (Redis connection error, serialization error), the exception propagates out of `_beat`, the thread terminates, and neither `stop_event` nor `lock_lost` is set. The `finally` block's `t.join()` will still complete (the thread is dead), but the protected operation has been running without heartbeat for an unknown duration. Wrap the script call in a try/except inside `_beat` and treat exceptions as lock loss.

---

## Minor Issues

### 5. Test Coverage Does Not Exercise the Heartbeat Failure Path

`test_lock_release_uses_lua_script` verifies the release mechanism but no test simulates `_extend_script` returning `0` mid-operation and asserts that the reservation is aborted and inventory is not written. Once issue 1 is fixed, this path must be covered.

### 6. `LOCK_TTL = 30` With No Documented Basis

Thirty seconds is a long TTL for a reservation operation. If the heartbeat fails silently (issue 1), a stale lock blocks all other callers for up to 30 seconds. Document the expected worst-case DB operation time that this TTL is sized against, or reduce it and rely on the heartbeat for long operations.

### 7. No Observability on Heartbeat Extension Failures

When `result == 0`, the current code returns silently. Even after fixing issue 1, this event should be logged at `ERROR` level with the lock key and token, as it represents an unexpected coordination failure in a production inventory system.

---

## What Is Done Well

The Lua scripts for atomic release and extension are correct. The `stop_event.wait()` pattern as a timer is the right mechanism — it avoids a busy loop and responds promptly to shutdown. The token-per-lock-context design prevents cross-context release. Pydantic validation with pattern enforcement on `item_id` is appropriate for this surface. Exponential backoff with jitter is correct. The `daemon=True` thread prevents process hang on unclean exit.
