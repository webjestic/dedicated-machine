Decision: Request Changes

Summary: The implementation is thoughtful — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and mostly correct. However, there are two silent failure modes that survive all existing tests: the heartbeat thread detects lock loss but has no channel to communicate that loss back to the protected operation, which then continues writing against invalidated state; and the stock read happens outside the database transaction, creating a TOCTOU window that the distributed lock alone cannot close if the DB layer has independent writers. These are not style concerns — they are correctness gaps that can produce double-reservations or oversells under real load.

---

## Detailed Findings

### 1. Lock Loss Is Detected But Not Acted Upon (Critical)

This is the central flaw. The heartbeat thread correctly identifies when the lock has been lost:

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # ← lock is gone. thread exits silently.
```

The thread exits. The protected operation in `reserve_item` does not know this happened. It continues:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← runs against invalidated state
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The lock expired. Another process acquired it. Both processes now believe they hold the lock. Both write. The heartbeat was supposed to prevent this, but it only *detects* the failure — it does not *stop* the operation.

The gap is not in acquisition. It is not in release. It lives between when the lock is lost and when the protected code learns it. The current code never closes that gap.

**What is needed:** A shared cancellation flag — set by the heartbeat thread on `result == 0`, checked by the protected operation before and ideally during the DB transaction. Something like:

```python
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()  # signal the main thread
            return
        ...

# In the protected block, before committing:
if lock_lost.is_set():
    raise LockLostError("Lock expired mid-operation; aborting transaction")
```

This check must happen *before* the commit, not after. Checking after means the write already happened.

No existing test covers this scenario. A test would need to simulate `_extend_script` returning `0` mid-operation and assert that the DB write was not committed.

### 2. Stock Read Is Outside the Database Transaction (High)

```python
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)  # ← outside transaction

    if current_stock < req.quantity:
        return ReservationResult(...)

    new_stock = current_stock - req.quantity

    with self.db.transaction():                      # ← transaction starts here
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(...)
```

The stock value is read, evaluated, and used to compute `new_stock` before the transaction opens. If anything modifies stock between the read and the `update_stock` call — a direct DB write, a migration, an admin correction, a bug in another service that bypasses Redis — the reservation proceeds on a stale value.

The distributed lock protects against concurrent callers *through this service*. It does not protect against writers that do not participate in the locking protocol.

**What is needed:** The read should be inside the transaction, and ideally the update should be conditional — a `UPDATE ... WHERE stock >= quantity RETURNING new_stock` pattern, or an equivalent optimistic check inside the transaction boundary. At minimum:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        # rollback / return
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

### 3. Heartbeat Thread Join Timeout Is Silently Ignored (Moderate)

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within the join timeout, execution falls through to the release script anyway. There is no check of `t.is_alive()` after the join. A hung heartbeat thread is silently ignored, and the lock is released while the thread may still be running — potentially calling `_extend_script` after release, which is harmless in this case but indicates the join is providing false assurance.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly for key=%s", lock_key)
    # decide: raise, metric, or continue — but do not silently proceed
```

---

### 4. Test Coverage Does Not Reach the Failure Modes (Moderate)

The tests validate the happy path, input validation, and lock contention. None of them simulate:

- Lock loss mid-operation (heartbeat returns `0` while DB write is in progress)
- Stock changing between the read and the transaction open
- DB transaction rollback behavior
- Heartbeat thread lifecycle under exception paths

The `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()` but does not assert the arguments — it would pass even if the wrong token or key were passed.

---

### Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is reasonable (fires at ~TTL/3.75), but the comment says TTL/3. Not a bug, but the comment is imprecise.
- Retry backoff caps at `(2^2 * 0.1) + jitter = ~0.25s` for 3 attempts. For a flash sale under heavy load this may be too aggressive — callers will surface errors quickly. This is a product decision, not a defect, but worth flagging.
- The `LockAcquisitionError` is defined but not exported in `__all__`. If other services need to catch it, they can still import it directly — low severity.

---

## What Approval Requires

1. Heartbeat thread communicates lock loss to the protected operation, and the operation aborts before committing when lock loss is detected.
2. Stock read moves inside the database transaction with appropriate isolation (SELECT FOR UPDATE or equivalent).
3. Thread join result is checked; a hung heartbeat thread is logged as an error.
4. Tests added for: lock-lost-mid-operation abort, and stock-changed-between-read-and-write.
