Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all sound. However, there are several correctness and reliability issues that must be addressed before this merges: a TOCTOU race between the stock read and the DB transaction, missing exception handling around Redis and DB operations, and test coverage gaps that would let real failure modes go undetected in production.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU: Stock read is outside the database transaction**

```python
# Current — race window exists here
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # read outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # write inside tx
```

The distributed lock serializes callers at the Redis layer, but `get_stock` runs outside the DB transaction. If the DB itself is accessed by any path that bypasses this lock (a migration, an admin tool, a second service), the stock read is stale by the time the write commits. The read must be inside the transaction, and the update should use a conditional write — either a `WHERE stock >= quantity` guard or a `SELECT FOR UPDATE`.

```python
# Recommended
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Unhandled exceptions from Redis and DB operations leave the lock unreleased or the thread orphaned**

```python
current_stock = self.db.get_stock(req.item_id)  # no except
```

If `get_stock`, `update_stock`, or `create_order` raises an unexpected exception, it propagates out of the `with self._acquire_lock(lock_key):` block. The `finally` in `_acquire_lock` will run, so the lock itself is released — that part is fine. But the caller gets an unhandled exception with no `ReservationResult`, breaking the contract of the method signature. Wrap the body in a broad `except Exception` and return a safe result, or let it propagate intentionally and document that the caller must handle it.

Similarly, if `self.redis.set(...)` or the Lua script calls raise a `redis.RedisError`, `_acquire_lock` propagates it uncaught. The caller only catches `LockAcquisitionError`, so a Redis connection failure surfaces as an unhandled exception.

```python
# In reserve_item, add:
except redis.RedisError as e:
    logger.error("Redis error item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except Exception as e:
    logger.exception("Unexpected error item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Internal error.")
```

### 🟡 Moderate

**3. Heartbeat thread silently swallows lock loss — no caller notification**

```python
if result == 0:
    return  # lock was lost — thread exits, caller never knows
```

When the heartbeat detects `result == 0` (lock expired and was taken by another process), it exits quietly. The calling thread continues executing the DB transaction believing it holds the lock, which it does not. This is a correctness violation: two processes can now both believe they own the reservation window.

The heartbeat should signal the main thread. A `threading.Event` or a shared flag works:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling caller", lock_key)
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost` before committing the transaction.

---

**4. `quantity` has no upper bound — denial of service / logic abuse vector**

```python
quantity: int = Field(..., ge=1)
```

A caller can submit `quantity=10_000_000`. This passes validation, acquires the lock, and if stock is somehow that large, commits a single order for the entire warehouse. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)
```

The ceiling value is a business decision, but the absence of any ceiling is a bug.

---

**5. `item_id` length limit may be insufficient for some SKU schemes, and the pattern should be reviewed**

The current pattern `r'^[a-zA-Z0-9_\-]+$'` is reasonable, but `max_length=128` should be confirmed against the actual SKU format in use. More importantly, the lock key is constructed as:

```python
lock_key = f"inventory:lock:{req.item_id}"
```

Because `item_id` is validated before this point, injection into the Redis key namespace is not possible. This is fine — just confirm the key prefix is consistent with your Redis keyspace conventions and that no other service writes to `inventory:lock:*`.

---

### 🟢 Minor / Observations

**6. Test for heartbeat lock-loss path is missing**

The test suite covers happy path, insufficient stock, not-found, lock contention, and input validation. It does not test the case where the heartbeat detects `result == 0` (lock stolen mid-operation). This is the most dangerous failure mode and should have an explicit test.

**7. `t.join(timeout=...)` result is not checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

If the thread does not exit within the timeout, `join` returns silently and the release script runs anyway. Since the thread is a daemon it will not block process exit, but a leaked heartbeat thread could extend a lock that has already been released by the Lua script, causing a brief phantom lock. Log a warning if `t.is_alive()` after join.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.warning("Heartbeat thread did not exit cleanly key=%s", lock_key)
```

**8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants — consider making them configurable**

For a government contracting context where deployment environments vary (latency, DB speed), these should be injectable at construction time with the current values as defaults. Hardcoded timing constants in a distributed system are a future operational headache.

| # | Severity | Item |
|---|----------|------|
| 1 | 🔴 Critical | Move stock read inside DB transaction; use `SELECT FOR UPDATE` |
| 2 | 🔴 Critical | Catch `RedisError` and bare `Exception` in `reserve_item` |
| 3 | 🟡 Moderate | Signal caller when heartbeat detects lock loss |
| 4 | 🟡 Moderate | Add upper bound to `quantity` field |
| 5 | 🟢 Minor | Check `t.is_alive()` after join; log warning if leaked |

The architecture is sound and the author clearly understands distributed locking. Items 1–3 are blocking; the rest should be addressed in the same PR given the sensitivity of inventory correctness in a flash sale context.
