Decision: Request Changes

Summary: The architecture is coherent and the surface mechanics are well-constructed — atomic Lua scripts, heartbeat scoping, input validation, and transaction wrapping are all present and correctly reasoned about. However, there is one structural flaw that the test suite does not exercise and the implementation does not handle: the gap between heartbeat failure and the protected operation's awareness of that failure. There are also secondary issues that compound the risk in a high-stakes flash sale context.

---

## Critical Finding: The Heartbeat Failure Gap

This is the flaw the checklist misses entirely.

The heartbeat thread detects lock loss — `result == 0` — and responds by returning silently:

```python
if result == 0:
    return
logger.debug("Lock TTL extended key=%s", lock_key)
```

The thread exits. The lock is gone. But `reserve_item` continues executing inside the `with self._acquire_lock(lock_key):` block with no awareness that the coordination guarantee has been invalidated.

Simulate the timeline:

```
t=0   Lock acquired, TTL=30s
t=8   Heartbeat fires, extends lock — OK
t=16  Heartbeat fires — Redis is unreachable or the lock was evicted
      result == 0, _beat() returns silently
t=17  reserve_item reads stock: 5
t=18  Another process acquires the same lock (lock is gone)
t=19  That process reads stock: 5
t=20  This process writes stock: 4, creates order A
t=21  That process writes stock: 4, creates order B
      Both orders committed. Stock decremented twice from 5. Actual stock: 3.
```

The heartbeat was designed to detect this exact condition. It detects it correctly. It then does nothing actionable with that information. The protected operation is not interrupted. The invariant the lock was protecting — exclusive access during the read-modify-write — is violated silently.

**What is required:** The heartbeat must have a channel back to the protected operation. Options include a shared `threading.Event` that the heartbeat sets on loss, which the protected code checks at the boundary between the read and the write, or an exception raised into the main thread via `ctypes` or a flag checked before the transaction is opened. The simplest correct approach is a `lock_lost_event` that the heartbeat sets, checked explicitly before `self.db.transaction()` is entered.

## Secondary Finding: Stock Read Outside the Transaction

```python
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens outside the transaction boundary. If the database supports it, the read and the conditional write should be inside the same transaction with an appropriate isolation level, or the `update_stock` call should be a conditional decrement — `UPDATE ... WHERE stock >= quantity RETURNING new_stock` — with the result checked inside the transaction. As written, a race exists between the read and the write at the database layer independent of the Redis lock, which matters if the Redis lock is ever lost (see above) or if the database is accessed by any path that does not go through this service.

---

## Secondary Finding: Heartbeat Thread Leak on Redis Failure at Release

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If `t.join()` times out — which can happen if the heartbeat thread is blocked on a slow Redis call — `_release_script` is called while the heartbeat thread may still be running and attempting to extend the lock. The join timeout is a safety valve, but the code proceeds to release regardless of whether the join succeeded. The thread is daemon=True so it will not prevent process exit, but within a request lifecycle this is a silent coordination failure. The join result should be checked and logged.

---

## Test Suite Gap

No test simulates heartbeat failure mid-operation. The test `test_reserve_lock_contention` tests acquisition failure, which is handled correctly. The untested path is: lock acquired, heartbeat fires, extension returns 0, operation continues, write commits. This is the highest-risk path in the entire service and it has zero coverage.

---

## Minor Observations

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` means the heartbeat fires at roughly TTL/3.75. The comment says TTL/3. This is not a bug but the comment is inaccurate.
- The `retry_attempts=3` default with exponential backoff produces a maximum wait of approximately `0.2 + 0.45 + jitter` seconds before failing. For a flash sale under high concurrency this may be appropriate, but it should be a configurable parameter, not a hardcoded default, so it can be tuned per deployment without a code change.
- `quantity` has no upper bound validation. A request for `quantity=10_000_000` passes Pydantic validation. A `le` bound should be defined appropriate to the business domain.

---

## What Approval Requires

1. The heartbeat loss signal must propagate to the protected operation. The operation must not commit after coordination has been lost.
2. The stock read should move inside the transaction boundary, or the write should be a guarded conditional update verified inside the transaction.
3. A test must be added that exercises the heartbeat-loss path and asserts the operation does not commit.
4. Join timeout handling should be explicit.
