## Code Review

### Critical Findings

**1. TOCTOU: Stock read occurs outside the database transaction (CRITICAL)**

```python
# Lock is held — good
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)  # READ — outside transaction

    if current_stock < req.quantity:
        return ReservationResult(...)

    new_stock = current_stock - req.quantity

    with self.db.transaction():                      # WRITE — inside transaction
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(...)
```

The stock read and the stock write are in **different transaction scopes**. The Redis lock serializes callers at the application layer, but it does not prevent a database-level anomaly: if the DB connection is using `READ COMMITTED` isolation (common default), another process — one that bypasses this service entirely, such as an admin tool, a refund flow, a batch job, or a second service instance that acquired the lock on a different Redis node during a partition — can modify stock between the `get_stock` call and the `update_stock` call. The lock provides mutual exclusion only among callers of this specific code path. The fix is to move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` or equivalent pessimistic read, or to use an optimistic `UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?` and check affected rows.

**2. Heartbeat thread silently swallows lock loss — no propagation to the main thread (HIGH)**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; main thread has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat detects that the lock token no longer matches — meaning the lock expired and was acquired by another holder. The heartbeat exits silently. The main thread continues executing, reads stock, and commits a write **while believing it holds the lock it no longer holds**. This is the exact failure mode distributed locks are designed to prevent. The heartbeat must signal the main thread. The correct pattern is to set a shared `threading.Event` (e.g., `lock_lost_event`) and have the main thread check it before committing, or raise into the main thread via a flag checked at the transaction boundary.

**3. Lock release after silent lock loss releases another holder's lock (HIGH)**

This is a consequence of finding #2. If the lock expires, Redis reassigns it to another waiter with a new token. When the original holder's `finally` block runs `_release_script`, the Lua script correctly checks the token and returns 0 (does not delete). That part is safe. However, the write has already been committed under no lock. The Lua release script is not the problem — the undetected lock loss before the write is.

**4. `t.join(timeout=...)` in `finally` does not verify the thread actually stopped (MEDIUM)**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns whether or not the thread has exited. If the heartbeat thread is blocked on a slow Redis call and does not return within the timeout, the main thread proceeds to release the lock while the heartbeat thread is still running and may attempt to extend the lock after it has been released and potentially re-acquired by another holder. The fix is to check `t.is_alive()` after join and log or handle the case where the thread did not terminate.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate before lock release key=%s", lock_key)
    # At minimum: do not release, or raise
```

**5. Database exceptions inside the transaction are not caught — lock is released but state is unknown (MEDIUM)**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `update_stock` or `create_order` raises, the exception propagates through `_acquire_lock`'s `finally`, which releases the lock and logs — but the caller of `reserve_item` receives an unhandled exception rather than a `ReservationResult`. The PR claims to return structured results; this is an incomplete contract. DB exceptions should be caught and returned as a failed `ReservationResult`, with the exception logged.

**6. Test suite does not test the critical failure modes it should (MEDIUM)**

- No test for lock loss mid-operation (heartbeat returns 0 while main thread is in the transaction).
- No test for DB exception during transaction.
- No test for the heartbeat thread's behavior under a slow Redis extend call.
- `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but because `register_script` returns a single shared mock, both `_release_script` and `_extend_script` are the same mock object — the assertion is not testing what it claims to test.

```python
# Both scripts are the same mock:
script_mock = MagicMock(return_value=1)
mock.register_script.return_value = script_mock
# service._release_script is script_mock
# service._extend_script is also script_mock
```

**7. Minor: `quantity` field has no upper bound (LOW)**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. For a flash sale service, an upper bound should be enforced at the model level.

**Decision:** Request Changes

**Summary:** The most critical flaw is a TOCTOU gap: the stock read occurs outside the database transaction, meaning the distributed lock only serializes application-layer callers and does not protect against writes from any other DB client. Compounding this, the heartbeat thread silently discards lock loss (`result == 0`) without notifying the main thread, allowing a database write to commit after the lock has expired and potentially been acquired by another holder — the exact scenario this architecture is designed to prevent. These two issues together mean the reservation system can produce double-reservations or negative stock under real production load, and neither failure is surfaced to the caller or observable in logs until inventory is already corrupted.
