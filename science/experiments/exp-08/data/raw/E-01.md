Decision: Request Changes

Summary: The implementation demonstrates genuine distributed systems literacy — atomic Lua scripts, scoped heartbeat state, token-matched release, and structured validation are all present and correctly reasoned. However, there is a critical flaw in the gap between lock coordination and database state that the test suite does not probe and the implementation does not close. The heartbeat silently abandons the operation when the lock is lost mid-flight, but the protected code has no awareness of this event and will continue writing to the database as if coordination still holds.

---

## Review

### What is being asked

This service acquires a Redis distributed lock, reads stock, and writes a decrement plus an order record inside a database transaction. A heartbeat thread extends the lock TTL at `TTL/3` intervals. The claim is that the heartbeat keeps the lock alive during slow DB operations.

That claim is the first thing to simulate, not read.

---

### Critical: The Heartbeat Failure Path Is Not Closed

The heartbeat thread calls `_extend_script`. When the lock has been lost — expired, evicted, or stolen — the script returns `0`. The thread then does:

```python
if result == 0:
    return
```

It exits. Silently. No signal is raised. No event is set. No flag is written.

The main thread, at that exact moment, may be anywhere inside:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

It does not know the lock is gone. It completes the write. The stock is decremented. The order is created. Another process, which legitimately acquired the lock after expiry, has already read the pre-decrement stock value and is preparing its own write. Both writes commit. Stock goes negative. Two orders are created for inventory that does not exist.

This is not a theoretical edge case. Under flash sale load — which is the stated deployment context — slow DB operations under high contention are the expected condition, not the exception. This is precisely the scenario the heartbeat was designed to protect against, and it is the scenario in which the heartbeat's failure is invisible to the code it is supposed to protect.

**The fix requires a coordination channel between the heartbeat and the main thread.** The heartbeat must set a shared flag or event when extension fails. The main thread must check that flag at a meaningful boundary — before committing, or before writing, depending on acceptable granularity. A simple approach:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()   # signal the main thread
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The caller then checks `lock_lost` before committing:

```python
if lock_lost.is_set():
    raise LockLostError(f"Lock expired mid-operation for {lock_key}")
```

Without this, the heartbeat is not a safety mechanism. It is a best-effort TTL extension with no observable failure mode for the code it protects.

---

### Significant: The Stock Read Is Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)   # outside transaction

# ... stock check ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # inside transaction
    self.db.create_order(...)
```

The read and the write are not atomic at the database level. The distributed lock is the only thing preventing a concurrent read between these two lines. That is a valid design choice, but it creates a hard dependency: the lock must be held continuously and exclusively from the read through the commit. If the lock is lost in that window — which is now possible given the silent heartbeat failure — the read value is stale and the write is unsafe.

This is a second reason the lock-lost signal matters. It is also worth making explicit in the design: the correctness of the stock check depends entirely on the lock. That assumption should be documented, and ideally enforced by moving the read inside the transaction or using a database-level check constraint or conditional update (`UPDATE ... WHERE stock >= quantity RETURNING stock`).

---

### Moderate: The `finally` Block Releases the Lock After a Lost-Lock Scenario

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The Lua release script is token-matched, so releasing a lock that has already been re-acquired by another process is safe — the script will return `0` and do nothing. This is correct. However, if `lock_lost` is introduced as a flag, the `finally` block should log when the release returns `0` unexpectedly, to make the lost-lock event observable in production.

---

### Moderate: The Thread Join Timeout Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`join` with a timeout returns `None` whether the thread exited cleanly or timed out. If the heartbeat thread is blocked — on a slow Redis call, for example — the join returns after 9 seconds and the release script runs while the heartbeat thread is still alive and may attempt another extension. The thread is daemonized, so it will not prevent process exit, but the race between a timed-out join and a subsequent extension attempt is a latent issue. The thread liveness should be checked:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
```

---

### Minor: The Test Suite Does Not Probe the Failure Gap

`test_lock_release_uses_lua_script` verifies that `redis.delete` is not called directly. That is a surface test. No test simulates the heartbeat returning `0` mid-operation and verifies that the database write is suppressed. No test simulates a slow DB call that outlasts the lock TTL. These are the conditions under which the implementation fails, and they are absent from coverage.

A minimal test for the critical path:

```python
def test_lock_lost_mid_operation_aborts_write(service, db, redis_client):
    # Simulate lock expiry: extend script returns 0 on first heartbeat call
    extend_mock = MagicMock(side_effect=[0])
    service._extend_script = extend_mock

    # DB operation is slow enough that heartbeat fires
    def slow_update(item_id, stock):
        time.sleep(service.HEARTBEAT_INTERVAL + 0.1)
    db.update_stock.side_effect = slow_update

    result = service.reserve_item("item_001", quantity=1)

    assert result.success is False
    db.create_order.assert_not_called()
```

This test does not pass against the current implementation. It should.

---

### What Is Done Well

The Lua scripts are correctly scoped — release and extension are both token-matched, preventing a process from releasing or extending a lock it no longer owns. The `SET NX EX` acquisition is atomic. The heartbeat interval at `TTL/3` is a reasonable choice. The stop event is correctly used as a timer via `wait(timeout=...)` rather than a sleep loop. The `finally` block correctly joins the thread before releasing. Input validation is present and the error path returns a typed result rather than raising. The exponential backoff with jitter is appropriate for contention scenarios.

The architecture is sound. The gap is specific and closeable.

---

### Required Before Approval

1. Heartbeat must signal the main thread when extension fails (`lock_lost` event or equivalent).
2. Main thread must check that signal before committing the database transaction and abort if the lock was lost.
3. Test coverage must include the mid-operation lock loss scenario and verify that the database write is suppressed.
