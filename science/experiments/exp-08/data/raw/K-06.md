Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, token-scoped heartbeat, NX+EX acquisition, and input validation are all sound. However, there are several correctness, resilience, and observability gaps that need to be addressed before this merges into a government-facing production pipeline. The issues range from a silent data-integrity race to missing Redis failure handling to test coverage gaps that would let real bugs pass undetected.

---

## Findings

### 1. CRITICAL — Stock Read Outside the Database Transaction (Data Integrity Race)

```python
# Current code — race window between read and write
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
        self.db.create_order(...)
```

The Redis lock serializes callers *at the application layer*, but `get_stock` runs outside the database transaction. If the database has any other writer path — a manual correction, a different service, a failed transaction that partially committed — the stock value read is stale by the time the transaction commits. The lock does not protect against that.

**Required fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic read lock) so the read and write are atomic at the database level. The Redis lock handles cross-process coordination; the DB transaction must handle consistency within the write.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return — transaction rolls back automatically
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. HIGH — Redis Errors Are Completely Unhandled

No `redis.RedisError` (or `redis.exceptions.ConnectionError`, `TimeoutError`) is caught anywhere — not in `_acquire_lock`, not in the heartbeat thread, not in the release call.

**Consequences:**
- An unhandled `RedisError` during acquisition propagates as an uncaught exception out of `reserve_item`, bypassing the `LockAcquisitionError` path and returning a 500 to the caller instead of a graceful degradation message.
- A Redis error during the heartbeat `_extend_script` call silently kills the thread. The code treats `result == 0` as a lost lock, but a connection exception is not `result == 0` — it is an unhandled raise inside a daemon thread, which is swallowed silently. The lock then expires naturally, which is the correct outcome, but the failure is invisible.
- A Redis error during `_release_script` in the `finally` block means the lock is never released for the remainder of its TTL. For a 30-second TTL this is tolerable, but it should be logged explicitly.

**Required fix:**

```python
# In _acquire_lock, wrap acquisition:
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.RedisError as e:
    raise LockAcquisitionError(f"Redis unavailable: {e}") from e

# In _beat, wrap extension:
try:
    result = self._extend_script(...)
except redis.RedisError:
    logger.error("Heartbeat Redis error key=%s — lock will expire naturally", lock_key)
    return

# In finally, wrap release:
try:
    self._release_script(keys=[lock_key], args=[token])
except redis.RedisError:
    logger.error("Lock release failed key=%s — will expire in TTL", lock_key)
```

### 3. HIGH — Heartbeat Thread Leak on Unhandled Exceptions

The `finally` block calls `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` but does not check whether the join actually completed:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # return value ignored
    self._release_script(...)
```

If the heartbeat thread is blocked (e.g., on a hung Redis call), `join` returns after the timeout and the thread continues running in the background — potentially extending a lock that has already been logically released. Because the thread is a daemon it will not prevent process exit, but it will continue making Redis calls against a key that may have been re-acquired by another process, and the token check in the Lua script is the only guard.

**Required fix:**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error(
            "Heartbeat thread did not terminate key=%s — possible Redis hang",
            lock_key
        )
    self._release_script(keys=[lock_key], args=[token])
```

### 4. MEDIUM — `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can submit `quantity=2_000_000_000`. This passes validation, hits the database, and `new_stock` becomes a large negative number if stock is insufficient — but the insufficient-stock check prevents the write. The deeper concern is that an unbounded integer is an input that should be explicitly bounded by business rules. For a flash sale service, a per-request quantity cap (e.g., `le=100` or a configurable constant) is standard practice and prevents abuse.

**Required fix:**

```python
MAX_RESERVATION_QUANTITY = 100  # or inject from config

quantity: int = Field(..., ge=1, le=MAX_RESERVATION_QUANTITY)
```

### 5. MEDIUM — `item_id` Exposed in Error Message

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Reflecting caller-supplied input into response messages is a low-grade information disclosure. In a government contracting context this is worth flagging: even though `item_id` is validated against `^[a-zA-Z0-9_\-]+$`, echoing it back confirms to an enumerating caller which item IDs exist versus do not exist. Use a generic message.

**Required fix:**

```python
message="Item not found"
```

### 6. MEDIUM — Test Coverage Gaps

The test suite covers the happy path and basic validation but misses several cases that correspond directly to the issues above:

| Missing Test | Why It Matters |
|---|---|
| `redis.RedisError` raised during `redis.set` | Verifies `LockAcquisitionError` is raised, not a raw Redis exception |
| `redis.RedisError` raised during `_release_script` | Verifies the error is logged, not silently swallowed |
| `db.transaction().__exit__` called with an exception | Verifies rollback path — currently the mock always returns `False` |
| `db.update_stock` raises mid-transaction | Verifies the order is not created on partial failure |
| Heartbeat extension returns 0 mid-operation | Verifies the thread exits cleanly without extending a stolen lock |

The existing `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args` arguments, so a regression that passes the wrong token would not be caught.

### 7. LOW — LOCK_TTL and HEARTBEAT_INTERVAL Are Not Validated Against Each Other

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

The comment says "fires at TTL/3" but this is a manual invariant with no enforcement. If someone changes `LOCK_TTL` to 10 and leaves `HEARTBEAT_INTERVAL` at 8, the heartbeat fires at TTL×0.8, which gives almost no margin before expiry. Add an assertion in `__init__`:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
    "HEARTBEAT_INTERVAL must be less than half LOCK_TTL"
)
```

---

### 8. LOW — Backoff Ceiling Is Missing

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3` the maximum backoff is `(2^2) * 0.1 + 0.05 = 0.45s`, which is fine at the default. But `retry_attempts` is a parameter with no documented or enforced upper bound. A caller passing `retry_attempts=20` produces a backoff of `(2^19) * 0.1 ≈ 52,428 seconds` on the last attempt. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

## What Is Done Well

- Lua scripts for atomic release and extension are correct and necessary — this is the right approach.
- `SET NX EX` is the correct single-command atomic acquisition; no SETNX+EXPIRE split.
- `stop_event.wait(timeout=...)` as a timer is correct — avoids busy-waiting and responds immediately to `set()`.
- Token-scoped heartbeat means a re-acquired lock by another process will not be extended by a stale thread.
- Pydantic validation with a regex allowlist on `item_id` is appropriate.
- Daemon thread prevents process hang on abnormal exit.
- Structured logging with key context fields is consistent throughout.
