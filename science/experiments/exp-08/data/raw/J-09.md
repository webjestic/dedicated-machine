Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, heartbeat scoping, and input validation are all solid. However, there are several correctness, resilience, and testability issues that need to be addressed before this is production-ready for a high-stakes flash sale context. The most critical is a TOCTOU race between the stock read and the database write that the lock alone does not fully close.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU: Stock read is outside the database transaction**

```python
# PROBLEM: stock is read before the transaction opens
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes based on stale read
    self.db.create_order(...)
```

The Redis lock serializes callers *at the application layer*, but if the DB connection is ever lost and reconnected mid-operation, or if any other writer bypasses the lock (a migration, a manual correction, another service), the stock read is stale by the time the write lands. The correct pattern is to move the read inside the transaction and use a database-level optimistic lock or a conditional update:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT ... FOR UPDATE
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
```

The Redis lock is a good first line of defense for throughput, but the DB transaction must be the authoritative safety net. Right now it is not.

---

**2. Silent exception swallowing inside the lock context**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        ...
```

Any unexpected exception raised inside `_acquire_lock` (DB timeout, network error, etc.) is not caught. The `finally` block in `_acquire_lock` *will* release the lock correctly, but `reserve_item` has no outer `except Exception` handler, so the exception propagates to the caller as an unhandled 500. For a service method this should be caught and returned as a structured error, or at minimum documented explicitly as a raising path. The checklist claims `LockAcquisitionError` is handled, but that is the only exception that is.

---

### 🟠 High

**3. Heartbeat thread has no way to signal lock loss back to the business logic**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return   # lock was lost — exits silently
```

If the lock is lost mid-operation (Redis restart, network partition, TTL misconfiguration), the heartbeat exits silently. The code inside `with self._acquire_lock(...)` continues executing and will commit a DB transaction against an item that another process may have already locked and modified. The heartbeat needs a shared flag or event that the main thread checks, or the lock context manager should raise on detected loss.

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            return
```

Then the business logic should check `lock_lost_event` before committing, or the context manager should raise after `yield` if the event is set.

---

**4. `t.join(timeout=...)` failure is not checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join()` with a timeout returns silently whether or not the thread actually stopped. If the heartbeat thread is hung (e.g., blocked on a slow Redis call), the lock release fires anyway, but the thread is still alive and will attempt to extend a lock that no longer belongs to this token. The Lua script protects against a successful extension, but the thread leaks. Add a check:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

### 🟡 Medium

**5. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is commented as TTL/3, but 30/3 = 10, not 8**

Minor but the comment is wrong and the value is inconsistent with the stated design intent. If the intent is TTL/3, use `HEARTBEAT_INTERVAL = LOCK_TTL // 3`. Hardcoded magic numbers that drift from their own documentation are a maintenance hazard.

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # fires at TTL/3 ≈ 10s
```

---

**6. `retry_attempts` default is not configurable at the class level**

`_acquire_lock` defaults `retry_attempts=3` but this is not exposed on `InventoryService.__init__` or as a class constant. Under extreme flash sale load you may want to tune this per deployment without modifying the method signature. Promote it to a class constant or constructor parameter alongside `LOCK_TTL`.

---

**7. `remaining_stock` leaks internal stock levels on failure**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,   # exact stock count returned to caller
    message="Insufficient stock"
)
```

Returning exact inventory counts to external callers is an information disclosure risk, particularly in a government contracting context. Consider returning a boolean `in_stock` flag or a bucketed value ("low", "out") rather than the raw integer.

---

### 🔵 Low / Test Coverage Gaps

**8. No test for the lock-lost-mid-operation scenario**

The heartbeat's silent exit on `result == 0` is the most dangerous code path and it has zero test coverage. There should be a test that simulates the extend script returning 0 mid-operation and asserts the expected behavior (currently: nothing; after fix: an error signal).

**9. `test_lock_release_uses_lua_script` is fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. A future refactor that calls the script with wrong keys or args would pass this test. Use `assert_called_once_with(keys=[...], args=[...])`.

**10. No test for DB exception inside the lock context**

There is no test covering what happens when `db.update_stock` raises. Given finding #2, this is a gap that would expose the unhandled-exception path in CI before production.
