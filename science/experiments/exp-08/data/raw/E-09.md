# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/flash-sale-inventory`

---

## Preliminary Orientation

Before reading a line of implementation, I ask the structural question this PR is built around:

> *If the distributed lock expires or is lost while the protected operation is still running, does the protected code know — or does it continue as if the lock still holds?*

The heartbeat is the answer this PR proposes. My job is to determine whether that answer is actually correct, or whether it only appears correct on the acquisition and release paths while leaving the critical gap unaddressed.

---

## Finding 1 — The Heartbeat Detects Lock Loss But Cannot Stop the Operation (Critical)

This is the central architectural flaw. It is not a code bug in the conventional sense. It is a gap between what the heartbeat *knows* and what it can *do* with that knowledge.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # ← heartbeat exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread knows the lock is gone. It logs nothing. It sets nothing. It returns. The main thread, which is somewhere inside the `with self._acquire_lock(lock_key):` block — potentially mid-transaction — has no idea. It continues executing as if it still holds the lock.

The checklist item reads:

> ✅ heartbeat detects token mismatch (result == 0)

Detection without signaling is not protection. Detection that silently exits is the worst possible outcome: it creates the illusion of a safety net while providing none.

**The failure scenario this enables:**

1. Lock acquired. Heartbeat starts.
2. DB operation is slow. Redis becomes briefly unreachable or the heartbeat fires late.
3. Lock TTL expires. Another process acquires the lock and begins its own reservation.
4. Heartbeat fires, gets `result == 0`, silently exits.
5. Original process completes its DB transaction — now two processes have committed against the same stock window.
6. Inventory goes negative. Orders are oversold.

**What is required:** The heartbeat must signal the main thread on lock loss. A `threading.Event` for lock validity, checked at meaningful points in the protected operation, or a cancellation mechanism that interrupts the DB transaction. At minimum, `result == 0` must set a shared flag that the main thread checks before committing.

## Finding 2 — The Stock Read Is Outside the Database Transaction (Critical)

```python
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

    if current_stock < req.quantity:
        return ReservationResult(...)

    new_stock = current_stock - req.quantity
    order_id = str(uuid.uuid4())

    with self.db.transaction():                        # ← transaction starts here
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The distributed lock provides mutual exclusion between processes competing for the *same item*, but it provides no protection against:

- A manual DB write or admin correction between the read and the transaction open
- A different code path that modifies stock without acquiring this lock
- Any future refactor that introduces a second write path

The read must occur inside the transaction, with the transaction isolation level set appropriately (at minimum `READ COMMITTED`, ideally `REPEATABLE READ` or `SERIALIZABLE` for this use case), or the write must use a conditional update (`UPDATE ... WHERE stock >= quantity`) that fails atomically if the stock has changed since the read.

The distributed lock reduces the probability of this race to near zero in the current codebase, but the architecture is one unlocked write path away from a silent correctness failure. For a flash sale system under adversarial load, that is not an acceptable margin.

## Finding 3 — Lock Release Executes After a Silently Failed Heartbeat (Compounding)

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat detected `result == 0` — meaning the lock was already lost and potentially re-acquired by another process — the release script still executes. The Lua script is token-matched, so it will correctly return 0 and not delete a lock it doesn't own. That part is safe.

However, this means the system has already committed a DB transaction under an expired lock and then silently attempted to release a lock it no longer held. The Lua script's correctness here masks the upstream failure rather than surfacing it. There is no log entry, no exception, no metric emitted when the release script returns 0 after a heartbeat failure. The failure is invisible in production.

---

## Finding 4 — Heartbeat Thread Join Has a Bounded Timeout That Can Be Exceeded

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
```

If the heartbeat thread is blocked on a slow Redis call — network partition, Redis replica failover, connection pool exhaustion — `t.join()` returns after 9 seconds regardless of whether the thread has actually exited. The release script then executes while the heartbeat thread may still be alive and holding a reference to `lock_key` and `token`. This is a minor resource concern in practice given the daemon flag, but it means the "heartbeat thread joined before lock release" checklist item is not strictly guaranteed.

## Finding 5 — Test Coverage Does Not Test the Failure Path That Matters

```python
def test_lock_release_uses_lua_script(service, redis_client):
    """Lock release is Lua-atomic: does not call redis.delete directly."""
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

The test suite covers:
- Successful reservation
- Insufficient stock
- Item not found
- Lock contention (acquisition failure)
- Input validation
- Lua release atomicity

The test suite does not cover:
- Heartbeat detecting `result == 0` and the main thread continuing anyway
- DB transaction committing after lock expiry
- `_extend_script` returning 0 mid-operation
- Any behavior when Redis is unavailable during heartbeat
- The `t.join()` timeout being reached with the thread still alive

The most important failure mode in this entire design — lock lost mid-operation — has zero test coverage.

---

## Finding 6 — Redis Exceptions Are Unhandled in the Heartbeat and Release Path

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
```

```python
self._release_script(keys=[lock_key], args=[token])
```

Neither call has exception handling. A `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` during the heartbeat will propagate up and kill the heartbeat thread — with the same silent failure mode as `result == 0`, but now without even the token-mismatch signal. The lock will expire naturally. The main thread continues unaware.

A `ConnectionError` during the release script in the `finally` block will propagate out of the context manager and potentially suppress or replace an exception that was already in flight from the protected block.

---

## Finding 7 — `LOCK_TTL` Is a Module-Level Constant Passed as a String to Lua

```python
args=[token, str(self.LOCK_TTL)]
```

This is a minor defensive concern: `LOCK_TTL` is an integer class attribute. Passing it as a string to the Lua `expire` command works because Redis coerces it, but it is worth noting that if `LOCK_TTL` were ever changed to a float (e.g., `30.5`), the Lua `expire` call would fail silently or raise a Redis error. `int(self.LOCK_TTL)` is more defensive.

| 4 | Thread join timeout does not guarantee thread exit before release | **Medium** |

---

## Decision: Request Changes

**Summary:** The heartbeat correctly detects lock loss but provides no mechanism to signal or interrupt the main thread, meaning the protected operation can complete against expired coordination state — the exact failure this design claims to prevent. Compounding this, the stock read occurs outside the database transaction, leaving a read-modify-write race that the distributed lock only probabilistically closes. These two findings together represent an architecture that appears safe on the acquisition and release paths but fails silently in the gap between lock expiry and operation completion, which is precisely the gap that matters under flash sale load.
