Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all sound choices. However, there are several correctness, resilience, and security issues that must be addressed before this merges. The most critical is a TOCTOU race between the stock read and the DB write that undermines the entire point of the distributed lock.

---

## Critical Issues

### 1. TOCTOU Race — Stock Read Outside the DB Transaction

This is the most serious flaw. The stock check happens *before* the database transaction opens:

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...later...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes a value computed above
    self.db.create_order(...)
```

The Redis lock serializes *this service's* access, but it provides zero protection against:

- Direct DB writes from other services or admin tooling
- DB replicas with replication lag if `get_stock` hits a replica
- Any future code path that bypasses the lock

The stock read **must** be inside the transaction, and the update should use an optimistic or pessimistic DB-level lock (e.g., `SELECT ... FOR UPDATE` in SQL). The pattern should be:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return — transaction rolls back cleanly
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is still valuable for reducing DB contention under flash-sale load, but it cannot be the *sole* correctness guarantee.

### 2. Silent Lock Loss in Heartbeat — No Caller Notification

When the heartbeat detects a lost lock (`result == 0`), it silently returns:

```python
if result == 0:
    return  # lock was lost — but the main thread has no idea
logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread continues executing the DB transaction believing it holds the lock. This is a correctness violation — another process may have acquired the lock and begun its own reservation. The heartbeat needs a signaling mechanism:

```python
# Use a shared Event or a thread-safe flag
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling caller", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread must check `lock_lost_event` before committing, or the DB transaction must be structured so that a DB-level lock (see issue #1) catches the conflict independently.

---

### 3. Exception Handling Gap — Unhandled DB Errors Leak State

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises, the exception propagates out of `reserve_item` entirely — no `ReservationResult` is returned, and the caller receives an unhandled exception. For a service under flash-sale load, this likely means a 500 to the end user with a stack trace that may expose internals.

```python
# Currently: only LockAcquisitionError is caught
except LockAcquisitionError as e:
    ...
```

Add a broad catch for DB/unexpected errors:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

## Moderate Issues

### 4. `retry_attempts` Default Is Not Configurable at the Service Level

`_acquire_lock` accepts `retry_attempts=3` but `reserve_item` calls it with no argument, hardcoding 3 retries. Under flash-sale load this should be tunable without a code change — expose it as a constructor parameter or config value.

### 5. Heartbeat Thread Join Timeout Is Too Generous

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # up to 9 seconds
```

If the heartbeat thread hangs (e.g., Redis connection stall), the lock release is delayed by up to 9 seconds. The join timeout should be short (1–2 seconds) and followed by a warning log if the thread is still alive. The lock release via Lua script should still execute regardless.

```python
t.join(timeout=2)
if t.is_alive():
    logger.warning("Heartbeat thread did not terminate cleanly key=%s", lock_key)
self._release_script(keys=[lock_key], args=[token])
```

### 6. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum. A caller can request `quantity=10_000_000`. Add a reasonable upper bound (`le=1000` or a configurable cap) to prevent abuse and to make the stock check meaningful.

---

## Minor Issues

### 7. Test Coverage Gaps

- No test for the heartbeat thread detecting a lost lock (`_extend_script` returns 0).
- No test for DB exceptions during `get_stock`, `update_stock`, or `create_order`.
- `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args` were passed — a refactor could break the atomic guarantee silently.
- No test verifying that `update_stock` is *not* called when the lock is lost mid-operation.

### 8. `new_stock` Computed Before Transaction

```python
new_stock = current_stock - req.quantity  # computed outside transaction
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

This is a minor style concern compounded by issue #1 — once the stock read moves inside the transaction, this naturally resolves. Flagging it for awareness.

---

## What Is Done Well

- Lua scripts for atomic release and extension are correct and idiomatic.
- `SET NX EX` is the right primitive — no separate TTL set.
- `stop_event.wait(timeout=...)` as a timer is cleaner than `time.sleep` and responds immediately to shutdown.
- Pydantic validation with `pattern` on `item_id` prevents injection-style inputs.
- `daemon=True` on the heartbeat thread ensures it doesn't block process shutdown.
- Structured logging with consistent key-value fields is production-ready.

---

## Required Before Approval

1. Move stock read inside the DB transaction with a DB-level row lock (`SELECT FOR UPDATE` or equivalent).
2. Signal the main thread when the heartbeat detects lock loss; abort the transaction if the signal is set before commit.
3. Catch and handle DB/unexpected exceptions, returning a `ReservationResult` rather than propagating.
4. Add an upper bound to `quantity`.
5. Tighten the heartbeat thread join timeout and handle the case where it does not terminate.
