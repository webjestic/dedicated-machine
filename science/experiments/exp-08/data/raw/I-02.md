Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, input validation, and structured logging are all present and largely correct. However, there are several correctness flaws, a silent failure mode in the heartbeat, a race condition in the lock release sequence, and test gaps that would allow real bugs to ship undetected. These issues are significant enough in a high-concurrency flash sale context that I cannot approve this as written.

---

## Critical Issues

### 1. Lock Release Happens After `stop_event.set()` + `t.join()` — Heartbeat Can Extend a Lock You're About to Release

```python
# Current finally block:
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # waits up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

The `t.join()` call blocks for up to `HEARTBEAT_INTERVAL + 1` (9 seconds) before the lock is released. During that window the lock is held but the critical section is already complete. Under high flash sale load this artificially serializes throughput. The join timeout should be short (e.g., 1–2 seconds) or the heartbeat thread should be designed to exit immediately on `stop_event.set()` without waiting a full interval. The `stop_event.wait(timeout=...)` in `_beat` will return `True` immediately once the event is set, so the thread *will* exit promptly — but the join timeout of `HEARTBEAT_INTERVAL + 1` is a misleading and unnecessarily long safety ceiling that masks the real behavior and could hide a hung thread silently.

### 2. Silent Heartbeat Failure — No Propagation to the Critical Section

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return   # <-- silently exits; caller has no idea the lock was lost
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When the lock is lost mid-operation (`result == 0`), the heartbeat thread silently returns. The main thread continues executing the database transaction under the assumption it still holds the lock. This is the most dangerous failure mode in the entire implementation — it defeats the purpose of the distributed lock entirely. Another process could acquire the lock and begin its own reservation while this thread is still writing to the database.

**Fix:** Use a shared flag or `threading.Event` to signal lock loss back to the main thread, and check it before and after the DB transaction.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost_event.set()
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the critical section, after the DB transaction:
if lock_lost_event.is_set():
    # compensate / rollback / raise
    raise LockLostError(f"Lock lost during reservation for {lock_key}")
```

### 3. Stock Read Is Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction

    if current_stock < req.quantity:
        ...

    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
        self.db.create_order(...)
```

The stock read occurs outside the database transaction. While the distributed lock provides mutual exclusion at the application layer, this pattern is fragile: if the database itself supports concurrent writers (e.g., a replica lag scenario, a direct DB write bypassing this service, or a future refactor that removes the lock), the read is not protected by the same atomicity guarantee as the write. The `get_stock` call should be inside the `db.transaction()` block, or the update should use a conditional decrement (`UPDATE ... WHERE stock >= quantity`) to enforce correctness at the DB layer as a second line of defense.

## Moderate Issues

### 4. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no `le` (less than or equal) upper bound on quantity. A caller can submit `quantity=10_000_000` and the service will attempt to reserve it. For a flash sale system this is both a business logic flaw and a potential abuse vector. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)
```

### 5. `item_id` Length Allows Arbitrarily Long Lock Keys in Redis

`max_length=128` for `item_id` means Redis lock keys can be up to `len("inventory:lock:") + 128 = 143` characters. This is not dangerous by itself, but combined with the fact that the lock key namespace is predictable and user-influenced, it's worth noting. Consider whether `item_id` should be validated against a known catalog rather than just a regex.

### 6. Heartbeat Extension Uses `str(self.LOCK_TTL)` Passed as Lua `ARGV`

```python
args=[token, str(self.LOCK_TTL)]
```

The Lua script receives `ARGV[2]` as a string and passes it to `redis.call("expire", KEYS[1], ARGV[2])`. Redis's `EXPIRE` command accepts integer seconds. Redis will coerce a string integer, but this is an implicit dependency on Redis's type coercion behavior. Pass the integer directly or document this explicitly.

## Test Gaps

### 7. The Most Critical Bug Is Not Tested — Heartbeat Lock Loss

There is no test that simulates `_extend_script` returning `0` mid-reservation and verifies that the main thread detects the lost lock and does not commit the transaction. This is the highest-risk scenario and it is completely uncovered.

### 8. `test_lock_release_uses_lua_script` Does Not Assert the Correct Key or Token

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not verify it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test.

```python
# Should be something like:
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.any_string()]  # or capture the token
)
```

### 9. No Test for Heartbeat Thread Lifecycle

There is no test verifying that the heartbeat thread is started, joined, and does not leak after `reserve_item` returns. A daemon thread leak under load is a real operational concern.

### 10. No Test for DB Transaction Failure / Rollback Path

If `self.db.update_stock` or `self.db.create_order` raises inside the transaction, there is no test verifying the lock is still released and the result is handled gracefully. The `finally` block in `_acquire_lock` should handle this, but it is untested.

## Minor Notes

- The backoff ceiling is very low (`(2^2) * 0.1 + 0.05 = 0.45s` max for 3 attempts). Under genuine flash sale load with hundreds of concurrent requests, 3 attempts with sub-second backoff will produce a high rejection rate. Consider making `retry_attempts` and backoff parameters configurable.
- `logger.info("Lock released key=%s", lock_key)` fires even if `_release_script` returns `0` (lock was already expired or stolen). Log the return value.
- The `HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` gives a ratio of ~3.75x, which is reasonable, but this should be enforced programmatically (e.g., assert `HEARTBEAT_INTERVAL < LOCK_TTL / 2`) rather than left as a magic constant relationship.
