## Code Review

### Critical Findings

#### 1. TOCTOU Race Between Stock Read and Database Transaction (High Severity)

```python
# Line executes OUTSIDE the database transaction:
current_stock = self.db.get_stock(req.item_id)

# ...validation logic...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes inside transaction
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but `get_stock` is called outside the database transaction. If the database has any other writer path — a manual correction, an admin tool, a different service, a future code path that bypasses the Redis lock — the stock value read before the transaction is stale by the time the write commits. The correct pattern is to read and write stock inside the same transaction, ideally using a `SELECT ... FOR UPDATE` or equivalent pessimistic lock at the database layer, or using an atomic `UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?` with a rows-affected check. The Redis lock alone is not sufficient if the database is the system of record.

#### 2. Lock Release Executes After Potential Exception Propagation Gap (Medium Severity)

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)       # blocks up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If `t.join()` times out — which can happen if the heartbeat thread is blocked on a slow Redis call or a GIL-contended operation — the code proceeds to `_release_script` without confirming the thread has exited. The thread may then call `_extend_script` concurrently with `_release_script`. The Lua scripts are individually atomic, but the sequence is not: extend could win the race, re-setting TTL on a key that was just deleted, then re-acquired by another caller, creating a ghost extension on a legitimately held lock. The thread should be checked after join and an alarm/log should fire if it did not terminate, and the thread must not call `_extend_script` after the stop event is set.

The current `_beat` implementation does handle this correctly via `stop_event.wait()` — the thread will not extend after stop is set — but the join timeout means you cannot guarantee the thread has observed the event before `_release_script` runs. This is a narrow but real window.

#### 3. Heartbeat Silent Failure Is Invisible to the Caller (Medium Severity)

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return  # lock was lost — thread exits silently
```

When the heartbeat detects `result == 0` (lock expired or stolen), it exits silently. The main thread continues executing the database transaction with no awareness that the distributed lock is no longer held. Another process may have acquired the lock and begun its own reservation. The database transaction then commits a stock decrement that races with the new lock holder's read. The lock was the entire serialization guarantee; losing it mid-operation must abort the operation, not be silently swallowed. The heartbeat needs a mechanism — a shared `threading.Event` or a flag — to signal the main thread that the lock was lost, and the main thread must check this flag before committing.

#### 4. `t.join()` Timeout Is Silently Ignored (Low-Medium Severity)

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
# No check: if t.is_alive() after join, this is a leaked thread
```

`threading.Thread.join(timeout=...)` does not raise on timeout. The thread may still be alive and holding a reference to `token` and `lock_key`. In a long-running service under load, this accumulates leaked threads. The code must check `t.is_alive()` after join and log an error at minimum. In a security-sensitive government contracting context, a leaked thread that holds a stale lock token and periodically calls Redis is a reliability and auditability concern.

#### 5. Database Exception Inside `with self.db.transaction()` Does Not Release the Lock Cleanly — By Design, But Untested (Low Severity)

If `self.db.update_stock` or `self.db.create_order` raises, the transaction context manager rolls back (assumed), the exception propagates out of `with self._acquire_lock(lock_key)`, hits the `finally` block, and the lock is released. This is structurally correct. However, there is no test for this path, and the `reserve_item` method has no `except` clause for general database exceptions — they will propagate to the caller as unhandled exceptions rather than returning a `ReservationResult`. For a service layer, this is an inconsistent contract: some failures return a result object, others raise.

#### 6. Test Coverage Does Not Test the Actual Failure Mode Being Guarded Against (Medium Severity)

```python
def test_reserve_concurrent_quantity(service, db):
    db.get_stock.return_value = 3
    result = service.reserve_item("item_001", quantity=3)
    assert result.success is True
    assert result.remaining_stock == 0
```

This test is named "concurrent" but tests no concurrency. It is a sequential single-call test. There is no test that:
- Simulates two threads calling `reserve_item` simultaneously
- Verifies that stock is not double-decremented
- Tests the heartbeat loss scenario (lock stolen mid-operation)
- Tests database exception rollback behavior

The mock-based tests verify the happy path wiring but provide no evidence that the distributed coordination is correct.

#### 7. Retry Backoff Is Applied After the Final Failed Attempt (Minor)

```python
for attempt in range(retry_attempts):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
    time.sleep(backoff)  # sleeps even on the last failed attempt
```

On the final attempt, if acquisition fails, the code sleeps before raising `LockAcquisitionError`. This adds unnecessary latency (up to ~0.45s + jitter) to every failed lock acquisition. The sleep should be skipped on the last attempt.

**Decision:** Request Changes

**Summary:** The core distributed locking mechanism is structurally sound — atomic acquisition, Lua-script release, and heartbeat renewal are all correctly implemented at the Redis layer — but the design contains a critical gap: when the heartbeat detects lock loss (`result == 0`), it exits silently while the main thread continues and commits a database transaction without a valid distributed lock, defeating the entire serialization guarantee. Additionally, the stock read occurs outside the database transaction, meaning the Redis lock is the sole guard against a read-modify-write race that the database itself does not enforce. Both issues must be resolved before this can be approved for a high-concurrency flash sale workload.
