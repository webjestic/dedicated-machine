Decision: Request Changes

Summary: The implementation demonstrates genuine architectural awareness — atomic Lua scripts, heartbeat-scoped stop events, token-matched release — but contains a silent failure mode in the heartbeat path that can cause the protected operation to continue against an expired lock with no awareness, and a stock-read/write race that the lock is supposed to prevent but doesn't fully close at the database layer. These are not style concerns; they are correctness flaws under the exact load profile this service is designed for.

---

## Critical Issues

### 1. Heartbeat Failure Is Silent — The Protected Code Never Learns the Lock Was Lost

This is the central flaw. When `_beat()` detects that the lock extension returned `0` (token mismatch — the lock expired and was taken by another process), it does this:

```python
if result == 0:
    return  # exits the thread silently
```

The thread exits. The stop event is never set. The calling code in `reserve_item` has no signal. It continues executing — reading stock, opening a database transaction, writing — while believing it holds a lock that no longer exists. Another process may have already acquired the lock and begun its own read-modify-write cycle on the same item.

The gap is not in acquisition. It is not in release. It lives between the moment the lock is lost and the moment the protected code learns it. Here, the protected code never learns it.

**What is required:** The heartbeat must have a channel back to the protected operation. Options include a shared `threading.Event` that the caller checks at safe points before committing, a `threading.local` flag, or raising into the main thread via a stored exception. The database transaction commit is the minimum gate — it must not proceed if the heartbeat has signaled loss.

A concrete sketch:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost_event.set()  # signal the protected operation
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the protected section, before committing:
if lock_lost_event.is_set():
    raise LockLostError(f"Lock expired mid-operation for {lock_key}")
```

The caller must check this before `self.db.transaction()` commits, not after.

### 2. Stock Read Occurs Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
        self.db.create_order(...)
```

The Redis lock serializes access across application instances, but the read and the write are not atomic at the database layer. If the database has read-committed or weaker isolation, a concurrent write (from a process that bypasses this service, a migration, an admin tool, a second service instance that acquired the lock after a crash) can modify stock between the read and the transactional write. The correct pattern is to move the read inside the transaction and use a database-level row lock (`SELECT ... FOR UPDATE`) so the read-modify-write is atomic at both layers.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The Redis lock is still valuable for reducing database contention under flash-sale load. But it should not be the sole correctness guarantee.

## Moderate Issues

### 3. Heartbeat Thread Join Timeout Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join()` with a timeout returns whether or not the thread has actually stopped. If the thread is blocked (e.g., Redis is slow, network partition), the join returns after the timeout and the release script executes anyway — but the heartbeat thread is still alive, holding a reference to `token` and `lock_key`, and may call `_extend_script` after release. A new process could acquire the lock, and the orphaned heartbeat thread would immediately evict it.

**Fix:** Check `t.is_alive()` after join and log a critical warning. Consider whether to proceed with release or raise.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical(
        "Heartbeat thread did not terminate before lock release key=%s — "
        "potential lock ownership conflict", lock_key
    )
```

### 4. Unhandled Exceptions from `db.get_stock` and the Transaction Block

If `self.db.get_stock()` raises (connection error, timeout), the exception propagates out of the `with self._acquire_lock(lock_key)` block. The `finally` in `_acquire_lock` will run and release the lock correctly — that part is fine. But `reserve_item` has no handler for database exceptions, so the exception surfaces to the caller as an unhandled error rather than a `ReservationResult`. For a service boundary, all failure modes should return a typed result or a well-defined exception.

## Test Coverage Gaps

### 5. The Heartbeat Failure Path Has No Test

There is no test that simulates `_extend_script` returning `0` mid-operation and verifies that the protected code does not commit. This is the most important failure mode in the system and it is completely untested. Once the signaling mechanism from issue #1 is implemented, a test must verify that a transaction is not committed when the lock is lost before the commit gate.

### 6. `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not that it was called with the correct token and key. A call with wrong arguments would pass this test. The assertion should be:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"], args=[ANY]
)
```

### 7. No Test for Heartbeat Thread Lifecycle

There is no test verifying that the heartbeat thread is started on lock acquisition and confirmed stopped after the context exits. A daemon thread that outlives the lock context (issue #3) would not be caught by any existing test.

## Minor Notes

- The backoff cap is missing. At `retry_attempts=10`, attempt 9 produces a base delay of `2^9 * 0.1 = 51.2s`. For a flash sale service this will cause request pile-up. Cap the backoff (e.g., `min(backoff, 2.0)`).
- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is a reasonable ratio, but the TTL should be configurable at construction time for environments with different DB latency profiles. Hardcoded class-level constants will cause problems when this service is deployed against a slower database tier.
- The `remaining_stock` field leaks internal inventory state on failed reservations. Depending on classification requirements for this deployment context, exposing exact stock counts to callers on failure may be undesirable.
