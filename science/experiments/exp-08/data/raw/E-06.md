# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Author:** dev-team
**Branch:** feature/flash-sale-inventory

---

## Structural Analysis

Before reading a single line of implementation, I ask the question that defines whether a distributed lock is actually providing safety or merely the appearance of it:

> *If the lock expires or is lost while the protected operation is still running, does the protected code know — or does it continue writing as if the lock still holds?*

The heartbeat is the author's answer to that question. So the second question becomes:

> *If the heartbeat detects that the lock was lost — what happens to the operation that is already in flight?*

That gap is where I begin.

---

## Finding 1 — Critical: Heartbeat Loss Is Silent; The Operation Continues Unguarded

This is the central flaw. It is not in the acquisition path. It is not in the release path. It lives in the gap between when the lock is lost and when the protected code learns about it.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # <-- heartbeat thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread returns. The lock is gone — expired, evicted, or stolen. The thread exits cleanly. And then:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # <-- executes without a lock
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The database transaction continues. The stock decrement commits. The order is created. A second process, which acquired the lock after the first lost it, may be doing the exact same thing simultaneously. The lock provided no protection at the moment it mattered most.

The heartbeat detects the loss. It does not communicate it. There is no shared flag, no exception raised into the main thread, no cancellation mechanism. The checklist item "heartbeat detects token mismatch (result == 0)" is technically true and operationally meaningless. Detection without propagation is not a safety mechanism.

**The concrete failure sequence:**

1. Process A acquires lock, reads stock = 5, decides to reserve 5 units.
2. Lock TTL expires before the heartbeat fires (e.g., slow DB, GC pause, network hiccup).
3. Process B acquires the same lock, reads stock = 5, decides to reserve 5 units.
4. Process A's heartbeat fires, gets `result == 0`, exits silently.
5. Process A commits: stock → 0, order A created.
6. Process B commits: stock → -5 (or 0 depending on DB constraints), order B created.
7. Inventory is now oversold. Both orders exist. The lock accomplished nothing.

**What a correct implementation requires:**

The heartbeat must write to a shared flag that the main thread checks at a defined synchronization point before committing. A threading.Event or similar primitive, checked explicitly before the transaction is entered, and ideally inside the transaction boundary:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()   # signal the main thread
            return

# In the main thread, before committing:
if lock_lost.is_set():
    raise LockLostError("Lock expired during operation — aborting transaction")
```

Even this is not fully safe. The check and the commit are not atomic. A rigorous implementation checks the flag, and the DB transaction itself enforces an invariant (e.g., a conditional update with a version column or a CHECK constraint) so that a commit racing against another writer fails at the database level rather than silently succeeding. The distributed lock is a performance optimization to reduce DB contention — the DB must remain the authority on correctness.

## Finding 2 — Significant: `t.join()` Does Not Block the Release Script

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call, `join` returns after the timeout and execution falls through to `_release_script`. The release script runs. The lock is deleted. The heartbeat thread unblocks and calls `_extend_script` on a key that no longer exists — or worse, on a key that a new process has already acquired with a different token. The Lua script protects against the latter case (token mismatch), but the sequencing is not guaranteed by the join.

The join return value is never checked. This should be:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # handle or at minimum alert
```

## Finding 3 — Significant: Stock Read Occurs Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction

    # ... decision logic ...

    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The read and the write are in different transactional scopes. The distributed lock is the only thing preventing another writer from modifying stock between the read and the write. As established in Finding 1, that lock can be lost silently. Even if Finding 1 were corrected, the read should be inside the transaction boundary, and the update should be a conditional write (`UPDATE ... WHERE stock = :expected_stock`) rather than an unconditional set. This makes the DB the final arbiter of correctness, not the lock.

## Finding 4 — Moderate: `LOCK_TTL = 30` Is Untethered From Operational Reality

The TTL is a hardcoded constant with no relationship to the actual latency characteristics of `db.get_stock`, `db.update_stock`, or `db.create_order`. Under load — which is the exact condition this code is designed for — DB operations can take seconds. The heartbeat fires at TTL/3 (10 seconds), which is reasonable under normal conditions. Under pathological conditions (DB failover, network partition, GC pause), the lock may expire before the first heartbeat fires.

There is no mechanism to detect this case. The first heartbeat fires at 8 seconds. If the DB operation takes 35 seconds, the lock expired at 30 seconds, another process acquired it at 31 seconds, and the first heartbeat at 38 seconds detects the loss — but as established in Finding 1, that detection is silent.

## Finding 5 — Moderate: Test Suite Does Not Test the Failure Mode This Code Is Designed to Handle

```python
def test_lock_release_uses_lua_script(service, redis_client):
    """Lock release is Lua-atomic: does not call redis.delete directly."""
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

The test suite verifies acquisition, release, insufficient stock, and input validation. It does not test:

- What happens when `_extend_script` returns 0 mid-operation.
- Whether the database transaction is aborted when the lock is lost.
- Whether two concurrent calls with the same item produce one successful reservation or two.
- Whether `t.join()` timeout is handled.

The most critical behavior — the behavior under lock loss — has zero test coverage. The checklist says "tests pass." That is true. It is also not the relevant question.

---

## Finding 6 — Minor: Daemon Thread Masks Unclean Shutdown

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed without cleanup when the main process exits. If the process is terminated while a heartbeat thread holds an extended lock, the lock will expire naturally, which is the correct behavior. This is acceptable. However, it also means that if the process exits during the `finally` block after `stop_event.set()` but before `_release_script`, the lock will not be released until TTL expiry. This is a known tradeoff with TTL-based locks, but it should be documented.

---

## Summary of Findings

| # | Severity | Description |
|---|----------|-------------|
| 1 | **Critical** | Heartbeat detects lock loss but does not propagate it — protected operation continues unguarded |
| 2 | Significant | `t.join()` timeout return value unchecked — release script may race with heartbeat thread |
| 3 | Significant | Stock read is outside the DB transaction — lock is the sole correctness mechanism, and it is unreliable |
| 4 | Moderate | `LOCK_TTL` is a magic constant with no relationship to observed operation latency |
| 5 | Moderate | Test suite has zero coverage of the lock-loss-during-operation path |
| 6 | Minor | Daemon thread behavior on unclean shutdown is undocumented |

---

**Decision:** Request Changes

**Summary:** The heartbeat correctly detects lock loss via token mismatch but does not communicate that loss to the main thread — the database transaction proceeds and commits against unprotected state, which is the exact failure mode this architecture is designed to prevent. Compounding this, the stock read occurs outside the transaction boundary, making the distributed lock the sole correctness mechanism at the moment it is demonstrably unreliable. The test suite validates the happy path and input rejection but contains no coverage of mid-operation lock loss, which is the only scenario that distinguishes this implementation from one with no lock at all.
