Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all sound. However, there are several correctness and resilience issues that must be addressed before this merges: a TOCTOU race between the stock read and the DB transaction, missing exception handling around Redis and DB operations, and a heartbeat failure that is silent rather than surfaced. Test coverage also has gaps that would let real failure modes go undetected.

---

## Detailed Findings

### 1. TOCTOU Race — Stock Read Outside the DB Transaction (High)

```python
# CURRENT — read is outside the transaction
current_stock = self.db.get_stock(req.item_id)          # ← snapshot taken here
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)         # ← write happens here
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but it does **not** protect against:

- A direct DB write from another process or service that bypasses this code path.
- A Redis failover or network partition that causes the lock to be silently lost between the read and the write (the heartbeat detects this, but only on the *next* interval — up to 8 seconds later).

**Required fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic read) so the DB engine itself enforces the invariant:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # row-level lock
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock still has value for reducing DB contention under flash-sale load, but correctness must not depend on it alone.

### 2. Unhandled Exceptions from Redis and DB Operations (High)

The `with self._acquire_lock(lock_key):` block catches `LockAcquisitionError` but nothing else. If `redis.set`, `self._extend_script`, `self._release_script`, `db.get_stock`, `db.update_stock`, or `db.create_order` raise, the exception propagates uncaught to the caller with no structured error response and no guarantee the lock is released cleanly.

The `finally` block in `_acquire_lock` does call `_release_script`, so the lock *will* be released on an exception path — that part is correct. But `_release_script` itself can raise a `redis.RedisError`, which would then suppress the original exception and leave the caller with an opaque traceback.

**Required fix:**

```python
try:
    with self._acquire_lock(lock_key):
        ...
except LockAcquisitionError as e:
    logger.error(...)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
except redis.RedisError as e:
    logger.error("Redis error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="An unexpected error occurred.")
```

Also wrap `_release_script` in the `finally` block:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
    except redis.RedisError:
        logger.error("Failed to release lock key=%s — TTL will expire naturally", lock_key)
```

### 3. Silent Heartbeat Failure Does Not Abort the Operation (Medium)

```python
if result == 0:
    return   # ← lock was lost; _beat exits quietly
```

When the heartbeat detects `result == 0`, it means the lock token no longer matches — the lock expired or was taken by another process. The heartbeat thread exits, but `reserve_item` continues executing, reads stock, and writes to the DB **without holding the lock**. This defeats the entire purpose of the distributed lock.

**Required fix:** Signal the main thread that the lock was lost. One clean approach:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost_event` at the critical point before the DB write:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock expired during operation for {lock_key}")
```

### 4. `quantity` Has No Upper Bound (Low-Medium)

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` constraint. A caller can pass `quantity=10_000_000`, which passes validation, acquires the lock, and issues a DB write. For a flash sale service this is almost certainly unintentional and could be used to drain all stock in a single request.

**Required fix:** Add a business-rule ceiling, e.g. `le=100`, or make it configurable via a constant.

---

### 5. Test Coverage Gaps (Medium)

The existing tests are well-structured, but the following cases are untested:

| Missing Test | Why It Matters |
|---|---|
| DB transaction raises an exception | Verifies lock is still released and a structured error is returned |
| `_extend_script` returns 0 (lock lost mid-operation) | Core safety invariant — currently untested |
| `redis.set` raises `RedisError` | Verifies the caller gets a `ReservationResult`, not a raw exception |
| `quantity` above any upper bound | Validates the business rule once added |
| Heartbeat thread is joined before release | `test_lock_release_uses_lua_script` checks the script was called but does not verify join ordering |

---

### 6. Minor: `item_id` Leaked in Error Message (Low)

```python
message=f"Item {req.item_id} not found"
```

In a government contracting context, leaking internal identifiers in API-visible error messages is a low-severity information disclosure. Prefer a generic message and log the `item_id` at the `WARNING` level instead.

---

## What Is Done Well

- Lua scripts for atomic release and extension are correct and registered at construction time (not per-call).
- `SET NX EX` is a single atomic operation — no separate TTL set.
- `stop_event.wait(timeout=...)` used as the heartbeat timer is correct; avoids `time.sleep` in a tight loop.
- Heartbeat thread joined before `_release_script` is called — correct ordering.
- Pydantic validation with a strict regex on `item_id` is appropriate.
- Exponential backoff with jitter is the right pattern for lock contention.
- `daemon=True` on the heartbeat thread prevents it from blocking process shutdown.
- Structured logging is consistent and useful.
