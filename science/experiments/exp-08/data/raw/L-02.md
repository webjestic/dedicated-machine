## Code Review

### Critical Defect — Race Condition Between Stock Read and Transaction Write

This is the most serious flaw in the submission.

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # READ (outside transaction)

# ... validation logic ...

new_stock = current_stock - req.quantity

with self.db.transaction():                        # WRITE (inside transaction)
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes *Redis-aware* callers. It does not prevent a database-level concurrent writer — a background job, a different service, a direct DB write, or any caller that does not participate in this lock protocol — from modifying stock between the `get_stock` call and the `update_stock` call. The `get_stock` executes **outside** the database transaction. If the DB supports `SELECT FOR UPDATE` or equivalent, it is not used here. The correct pattern is to read stock **inside** the transaction with a row-level lock, so the read and write are atomic at the database level, not just at the Redis level. As written, this is a classic TOCTOU (Time-of-Check to Time-of-Use) defect. In a flash sale scenario under high concurrency, this will produce oversell.

**The fix:** Move `get_stock` inside `self.db.transaction()`, and if the DB supports it, use a locking read (`SELECT ... FOR UPDATE`).

### Significant Defect — Heartbeat Thread Silently Swallows Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← silent exit
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When the lock is lost (e.g., Redis eviction under memory pressure, network partition, Redis restart), `result == 0` causes the heartbeat to silently return. The main thread has no awareness of this. It continues executing the database transaction believing it holds the lock — but it does not. Another process may have acquired the lock and be executing concurrently. The lock loss is not propagated to the caller. This defeats the entire purpose of the heartbeat.

**The fix:** The heartbeat must signal lock loss back to the main thread — via a shared `threading.Event`, a queue, or a flag — and the main thread must check that signal before and ideally during the DB transaction. This is architecturally non-trivial and is a design gap, not just a code gap.

### Significant Defect — `t.join()` Called After `_release_script` on Some Paths

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)      # waits up to 9s
    self._release_script(keys=[lock_key], args=[token])
```

The ordering here is correct — join before release — which is good. However, `t.join()` has a timeout of `HEARTBEAT_INTERVAL + 1 = 9s`. If the heartbeat thread is blocked (e.g., Redis call hanging), `join()` returns after 9 seconds **without confirmation that the thread has stopped**, and then `_release_script` runs. There is no check of `t.is_alive()` after the join. The thread may still be alive and attempt to extend the lock after it has been released and potentially re-acquired by another caller. The token check in the Lua script mitigates this for extension, but the thread leak itself is a resource concern under sustained load.

### Moderate Defect — `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound on `quantity`. A caller can request `quantity=10_000_000`. This passes validation, acquires the lock, and proceeds. In a flash sale context this is a business logic vulnerability — a single caller can drain all inventory or cause integer underflow in stock calculations if `new_stock` goes deeply negative (though the stock check prevents negative writes, it does not prevent absurdly large legitimate-looking requests from holding the lock and blocking all other callers for the full lock duration).

### Moderate Defect — DB Exception Inside Lock Context Is Unhandled

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

Any exception from `get_stock`, `update_stock`, or `create_order` propagates out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` will still release the lock — that part is correct. However, the exception propagates all the way to the caller of `reserve_item`, which has no `except` for DB errors. The caller receives an unhandled exception rather than a `ReservationResult`. The function's return type contract is broken. This is especially dangerous in production where DB timeouts are common under flash sale load.

### Minor Defect — Test Coverage Does Not Cover the Critical Path

`test_reserve_concurrent_quantity` does not test actual concurrency — it tests sequential execution with a mocked stock value. There is no test that:

- Simulates lock loss mid-heartbeat and verifies the main thread detects it
- Verifies that a DB exception inside the lock returns a `ReservationResult` rather than raising
- Exercises the retry/backoff path with partial lock acquisition (e.g., fails twice, succeeds on third attempt)

The test suite validates the happy path and basic input validation well, but the tests do not cover the failure modes that matter most in this domain.

### Minor Observation — Lua Script `expire` Returns 1/0, Not TTL

```lua
return redis.call("expire", KEYS[1], ARGV[2])
```

`EXPIRE` returns `1` if the timeout was set, `0` if the key does not exist. The heartbeat checks `result == 0` to detect lock loss. This is correct, but the comment says "returns 0 if lock was lost" — the actual condition is "key does not exist or token mismatch." If the key exists but holds a different token, `GET` returns a non-matching value, the `if` branch is skipped, and `0` is returned. This is correct behavior, but the semantic is "token mismatch or key gone," not just "lock was lost." Low severity, but worth documenting accurately.

## Decision: Request Changes

**Summary:** This PR contains a TOCTOU race condition where stock is read outside the database transaction, meaning the Redis lock does not actually prevent oversell from non-lock-aware writers or from the window between read and transactional write — a critical correctness defect for a flash sale reservation system. The heartbeat thread silently discards lock-loss events without notifying the main thread, meaning the system can proceed with database writes after the distributed lock has been lost. These two defects together undermine the core correctness guarantee the PR claims to provide and must be resolved before this code is approved.
