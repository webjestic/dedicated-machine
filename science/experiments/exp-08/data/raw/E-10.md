# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/flash-sale-inventory`

---

## Preliminary Posture

Before reading a line of implementation, I ask the structural question:

> If the lock expires or is lost while the protected operation is still running, does the protected code know — or does it continue as if the lock still holds?

The heartbeat is the answer the author gives to this question. So my first job is not to verify that the heartbeat *runs* — it is to verify what happens when the heartbeat *fails*, and whether that failure propagates to the code that depends on the lock being valid.

---

## Finding 1 — Critical: Heartbeat Failure Is Silent; Protected Code Is Never Notified

This is the central architectural flaw. It is not subtle.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # ← lock was lost. thread exits. nothing else happens.
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread silently exits. The main thread is inside `with self._acquire_lock(lock_key):` and has no knowledge this happened. It continues executing:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← runs without a valid lock
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is the exact failure mode the heartbeat was designed to prevent — and the heartbeat's failure path *reintroduces it*. The checklist entry "heartbeat detects token mismatch (result == 0)" is accurate. Detection is not the same as notification.

**What this means in a flash sale context:** Another process acquires the lock after the TTL expires. Both processes now hold what they believe to be exclusive access. Both read stock, both decrement, both write orders. Inventory goes negative. The distributed lock provides no mutual exclusion guarantee under this failure path.

**Required fix:** The heartbeat must have a channel back to the main thread. A `threading.Event` set on lock loss, checked at critical boundaries, or a shared flag that causes the protected block to raise before committing, are both viable. The main thread must not be permitted to commit a database transaction after lock loss is detected.

## Finding 2 — Significant: Stock Read Is Outside the Database Transaction

```python
# Outside transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    return ReservationResult(...)

new_stock = current_stock - req.quantity

# Inside transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The stock read and the stock write are not atomic at the database level. The Redis lock provides serialization *across processes using this service*, but it provides no protection against:

- Direct database writes from other services or administrative tooling not participating in the lock protocol
- Any future code path that modifies stock without acquiring this lock

The correct pattern is a conditional update inside the transaction:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id AND stock >= :quantity
RETURNING stock
```

If the update affects zero rows, the reservation fails. This makes the stock check and decrement atomic at the database layer, independent of the locking layer. Defense in depth is not optional for inventory systems.

## Finding 3 — Significant: `t.join()` Does Not Guarantee Heartbeat Has Exited Before Release

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis slowdown), `join` times out and returns `False`. The release script then executes. Now there are two concurrent Redis operations against the same key: the release script and the still-running extend script.

The Lua scripts are individually atomic, but their *ordering* is now a race. If the extend script wins, it resets the TTL on a key the release script just deleted — or worse, on a key another process has already acquired with a new token. The token check in the extend script protects against the latter case, but only if the new owner's token is different, which it will be. So the extend script will return 0 in that case. This is partially mitigated, but the race window is real and the `join` timeout gives a false sense of synchronization.

**Required fix:** Check the return value of `t.join()` or use `t.is_alive()` after the join. If the thread is still alive, log a critical error. Do not proceed with release silently.

## Finding 4 — Moderate: Retry Backoff Is Miscalibrated for the Lock TTL

```python
LOCK_TTL = 30
retry_attempts: int = 3

backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0: ~0.10–0.15s
# attempt=1: ~0.20–0.25s
# attempt=2: ~0.40–0.45s
```

Total maximum wait before failure: approximately 0.85 seconds. The lock TTL is 30 seconds. A legitimate holder can hold the lock for up to 30 seconds (plus heartbeat extensions, making it unbounded). A caller that fails to acquire after 0.85 seconds and returns "service temporarily unavailable" during a flash sale is not a retry strategy — it is a fast-fail with misleading framing.

This is a product decision, but the mismatch between the TTL and the retry window should be explicit and intentional, not accidental. If fast-fail is the intent, document it. If retry-until-available is the intent, the backoff ceiling and attempt count need to be reconsidered.

## Finding 5 — Moderate: Heartbeat Extension Uses String Conversion for TTL Argument

```python
result = self._extend_script(
    keys=[lock_key],
    args=[token, str(self.LOCK_TTL)]   # ← string
)
```

The Lua script passes this to `redis.call("expire", KEYS[1], ARGV[2])`. Redis's `EXPIRE` command accepts an integer. The `redis-py` client serializes all args as byte strings, and Redis coerces the argument. This works in current Redis versions but is relying on implicit type coercion inside a Lua script. The release script does not have this issue. Consistency and explicitness matter in Lua scripts that run atomically on the server.

## Finding 6 — Moderate: No Test Covers the Heartbeat Loss Path

The test suite covers:
- Successful reservation
- Insufficient stock
- Item not found
- Lock contention (acquisition failure)
- Input validation
- Lua script usage

No test covers the case where the heartbeat detects lock loss mid-operation. This is the most critical failure path in the system. Its absence from the test suite is consistent with Finding 1 — the path exists in the heartbeat thread but has no observable effect on the main thread, so there is nothing to assert against. The missing test is a symptom of the architectural flaw, not an independent issue.

---

## Finding 7 — Minor: `daemon=True` Thread Behavior on Process Exit

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed without cleanup on interpreter exit. If the process exits while a heartbeat thread holds a conceptual lock extension in flight, the lock will expire naturally, which is the correct behavior. This is acceptable. However, in a WSGI/ASGI context where worker processes are recycled (SIGTERM, graceful shutdown), the daemon thread may be killed mid-extension without the lock being explicitly released. The lock will expire after TTL, which is tolerable, but worth noting in operational documentation.

---

## Summary Assessment

The implementation demonstrates correct understanding of lock acquisition mechanics, atomic Lua scripting, and heartbeat architecture. The checklist is thorough and the code is well-structured. However, the heartbeat's failure path — the only path that matters for correctness under adverse conditions — does not propagate lock loss back to the protected operation. The main thread will commit database writes after the distributed lock has been lost and potentially re-acquired by another process. In a flash sale inventory context, this is an oversell vector. The stock read outside the database transaction compounds this by removing the last line of defense at the persistence layer.

---

**Decision:** Request Changes

**Summary:** The heartbeat correctly detects lock loss but does not communicate it to the main thread, allowing database writes to proceed without a valid lock — the exact failure mode the heartbeat exists to prevent. The stock read occurring outside the database transaction removes defense-in-depth at the persistence layer, leaving the system dependent solely on the Redis lock for correctness. Both issues must be resolved before this can be approved for a flash sale workload.
