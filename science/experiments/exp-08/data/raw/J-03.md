Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, input validation, and backoff/jitter are all solid patterns. However, there are several correctness, resilience, and security issues that must be addressed before this merges: a silent exception swallowing problem inside the lock context, a race condition between stock read and the DB transaction, a missing heartbeat failure signal path, and test gaps that would let real bugs through undetected.

---

## Detailed Findings

### 🔴 Critical

**1. Unhandled exceptions inside `_acquire_lock` silently release the lock and return success-adjacent behavior**

```python
# services/inventory.py
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `db.get_stock()`, `db.update_stock()`, or `db.create_order()` raises an unexpected exception, it propagates out of the `with self._acquire_lock(...)` block — which is correct for lock release — but `reserve_item` has **no outer catch** for generic exceptions. The exception will bubble up to the caller as an unhandled crash rather than a graceful `ReservationResult(success=False, ...)`. In a web handler context this likely produces a 500 with a stack trace. Wrap the entire lock body in a broad exception handler:

```python
except (LockAcquisitionError, Exception) as e:
    # differentiate and return appropriate ReservationResult
```

Or add a dedicated `except Exception` block after `LockAcquisitionError`.

---

**2. Stock read is outside the DB transaction — TOCTOU window remains**

```python
current_stock = self.db.get_stock(req.item_id)   # read OUTSIDE transaction
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write INSIDE transaction
```

The Redis lock serializes callers *on this service instance*, but it does **not** protect against:
- Direct DB writes from other services or admin tooling
- Multi-region deployments where the Redis lock is not shared
- Any future code path that bypasses this service

The stock read must be inside the transaction, and `update_stock` should use an optimistic or pessimistic DB-level guard (e.g., `UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`). If the affected row count is 0, the reservation must fail. As written, you can oversell.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

### 🟠 High

**3. Heartbeat failure is logged but not surfaced — operation continues on a potentially expired lock**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; main thread has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat exits silently. The main thread continues executing the DB transaction believing it holds the lock, but it does not. Another process may have acquired the lock and started its own reservation. This is the exact scenario the lock is meant to prevent.

The heartbeat must signal the main thread. One clean pattern:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat lost lock key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
```

Then check `lock_lost_event` at a safe point before committing the transaction, and abort if set.

---

**4. `t.join(timeout=...)` in `finally` does not verify the thread actually stopped**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns `None` whether the thread stopped or timed out. If the heartbeat thread is blocked on a slow Redis call, `_release_script` fires while the heartbeat is still potentially mid-extend. Check `t.is_alive()` after join and log a warning if it is still running.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not stop cleanly for key=%s", lock_key)
```

---

**5. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is tight for slow DB operations**

At TTL/3 the heartbeat fires once before the lock expires if the first extension fails. For a government contracting context where DB operations under load may spike, consider `LOCK_TTL = 60` and `HEARTBEAT_INTERVAL = 15`, or make both configurable via constructor parameters so they can be tuned per deployment without a code change.

### 🟡 Medium

**6. Test for lock release uses `assert_called_once()` on a shared script mock**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is the same mock object returned by `register_script`. If the heartbeat fires during the test (unlikely but possible given threading), the extend script and release script are the **same mock** (`register_script` returns one mock for all calls). The test does not distinguish between the release call and a potential extend call. Register separate mocks per script or assert on call arguments:

```python
service._release_script.assert_called_once_with(keys=[f"inventory:lock:item_001"], args=[ANY])
```

---

**7. No test for the DB exception path**

There is no test covering what happens when `db.update_stock` or `db.create_order` raises. Given finding #1, this path currently crashes. A test should be added:

```python
def test_db_exception_returns_failure_result(service, db):
    db.update_stock.side_effect = Exception("DB connection lost")
    result = service.reserve_item("item_001")
    assert result.success is False
```

---

**8. No test for heartbeat lock-loss scenario**

The most critical failure mode — heartbeat detects lock loss mid-transaction — has zero test coverage. This should be tested with a mock that returns `0` from the extend script after one call.

---

### 🔵 Low / Informational

**9. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. Add a reasonable `le=` bound appropriate to your business domain. This prevents both accidental and intentional stock exhaustion in a single call.

**10. `item_id` is logged directly in structured logs**

`item_id` passes through Pydantic validation, so it is safe against injection, but consider whether item IDs are considered sensitive identifiers in your agency contracts. If so, hash or truncate them in log output.

**11. `daemon=True` on the heartbeat thread**

Daemon threads are killed abruptly on interpreter shutdown. If the process receives SIGTERM mid-reservation, the heartbeat dies without the `finally` block running, leaving the lock to expire naturally (acceptable) but the DB transaction may be left open depending on the DB driver. This is worth documenting as a known limitation.

---

## Required Before Approval

1. Catch generic exceptions in `reserve_item` and return `ReservationResult(success=False, ...)`
2. Move stock read inside the DB transaction with a DB-level guard against oversell
3. Signal the main thread when the heartbeat detects lock loss
4. Check `t.is_alive()` after `join` and log appropriately
5. Add tests for DB exception path and heartbeat lock-loss path
6. Add upper bound to `quantity` field
