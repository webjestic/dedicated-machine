Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges, ranging from a TOCTOU race in the stock check to silent exception swallowing in the critical path. The test suite also has gaps that would let real bugs ship undetected.

---

## Issues Found

### 🔴 Critical

**1. TOCTOU Race Between Stock Read and DB Transaction**

```python
# Line ~90 in reserve_item
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
```

The Redis lock serializes callers at the application layer, but `get_stock` runs *outside* the database transaction. If the DB has any other writer (a refund process, admin correction, another service node that bypassed the lock), the stock value read can be stale by the time `update_stock` fires. The read must be inside the transaction, and `update_stock` should use a conditional decrement (e.g., `WHERE stock >= quantity`) with row-level locking, not a blind set of a pre-computed value.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

**2. Unhandled Exceptions in the Critical Path Leak the Lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception from `get_stock`, `update_stock`, or `create_order` (network timeout, DB error, etc.) propagates out of `reserve_item` as an unhandled exception. The `_acquire_lock` `finally` block *will* release the lock, which is correct, but the caller receives an uncaught exception rather than a `ReservationResult`. The contract of this method is to always return a `ReservationResult`. Add a broad catch inside the lock context:

```python
except (LockAcquisitionError, ...) as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟠 High

**3. Heartbeat Thread Silently Abandons on Lock Loss — Caller Is Never Notified**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return   # ← lock was stolen/expired; just exits
```

When `result == 0`, the heartbeat exits quietly. The main thread continues executing the DB transaction believing it holds the lock, but it does not. Another process may now also hold the lock and be modifying the same stock record simultaneously. This is the exact race condition the lock is meant to prevent.

The heartbeat must signal the main thread. Use a shared flag or a second event:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost_event` at appropriate points in the critical section, or raise inside the context manager.

---

**4. `LOCK_TTL = 30` Is Hardcoded and Untested Against Real Operation Duration**

30 seconds is an assumption. If a DB transaction takes longer (slow replica, deadlock retry), the heartbeat extends it — but the heartbeat interval of 8 seconds means the lock *could* expire in the window between the last extension and the next beat if Redis latency spikes. These values should be configurable via constructor parameters, not module-level constants, so they can be tuned per deployment and tested with different values.

### 🟡 Medium

**5. `t.join(timeout=HEARTBEAT_INTERVAL + 1)` Does Not Verify the Thread Actually Stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns whether or not the thread has exited. If the heartbeat thread is blocked on a slow Redis call, the join times out silently and the release script fires while the heartbeat thread may still be executing `_extend_script`. This is a benign race in practice, but it should be explicit:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.warning("Heartbeat thread did not exit cleanly key=%s", lock_key)
```

---

**6. `retry_attempts` Parameter Has No Upper Bound**

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

A caller can pass `retry_attempts=10000`. Under high contention this creates a thread that sleeps for `(2^9999) * 0.1` seconds (effectively forever). Cap it:

```python
retry_attempts: int = Field(default=3, ge=1, le=10)
```

Or at minimum add a guard at the top of the method.

---

**7. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

A request for `quantity=999999999` is valid per the model. This should have a reasonable domain cap (e.g., `le=1000`) to prevent abuse or accidental bulk wipes of inventory.

---

### 🔵 Test Coverage Gaps

**8. Lock-Lost Mid-Operation Is Not Tested**

There is no test that simulates `_extend_script` returning `0` mid-reservation and verifies the system behaves safely. This is the most dangerous failure mode and has zero coverage.

**9. DB Transaction Failure Is Not Tested**

No test covers `db.update_stock` or `db.create_order` raising an exception. The expected behavior (rollback, lock release, error result returned) is untested.

**10. `test_lock_release_uses_lua_script` Is Fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. A future refactor that calls the script with wrong keys/args would pass this test. Assert the arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"], args=[ANY]
)
```

## What Is Done Well

- Lua-atomic lock acquire (`SET NX EX`), release, and extension are correct.
- `stop_event.wait(timeout=N)` as a timer is the right pattern — avoids busy-loop.
- Heartbeat thread is daemon and joined before release — correct ordering.
- Pydantic validation with `pattern=` on `item_id` closes path traversal and injection vectors.
- `ValidationError` is caught and returned as a typed result, not re-raised.
- Structured logging with consistent key=value fields throughout.
- `uuid4` for both token and order ID — no predictability concern.
