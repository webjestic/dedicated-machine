Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, backoff/jitter, and Pydantic validation are all solid choices. However, there are several correctness, safety, and architectural issues that must be addressed before this merges, ranging from a silent data-integrity gap in the stock-read/write boundary to thread-safety concerns and incomplete test coverage of failure paths.

---

## Findings

### 🔴 Critical — Data Integrity

**1. Stock read occurs outside the database transaction**

```python
# Line executes BEFORE the transaction block
current_stock = self.db.get_stock(req.item_id)

# ...later...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The distributed lock serializes Redis-level access, but `get_stock` runs outside the DB transaction. If the DB and Redis are ever even slightly out of sync — a Redis failover, a lock that expired silently before the heartbeat fired, or a direct DB write from another service — the stock read is stale by the time `update_stock` commits. The read and the conditional write must be inside the same transaction, or you need a compare-and-swap / optimistic lock at the DB layer (e.g., `UPDATE ... WHERE stock >= quantity AND stock = :expected_stock`). Relying solely on the Redis lock to guard a DB read that sits outside the DB transaction is a layered assumption that will eventually break.

---

### 🔴 Critical — Thread Safety

**2. Heartbeat thread silently swallows lock-loss without signaling the caller**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # <-- lock lost; caller has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits quietly. The main thread continues executing the DB transaction believing it holds the lock, when it does not. Another process may have acquired the lock and begun its own reservation against the same item. This is the exact race condition the lock is meant to prevent. The heartbeat must signal the main thread — via a shared `threading.Event`, a queue, or a flag — so the operation can be aborted.

---

### 🟠 High — Reliability

**3. `t.join()` timeout does not verify the thread actually stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns whether or not the thread has finished. If the heartbeat thread is blocked on a slow Redis call and the join times out, the release script runs while the heartbeat thread is still alive and may still call `_extend_script`. In pathological cases this could re-extend a lock that was just released and re-acquired by another holder. Check `t.is_alive()` after the join and log a warning (or raise) if the thread did not terminate cleanly.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # consider raising or at minimum alerting
```

---

### 🟠 High — Reliability

**4. Unhandled exceptions inside the lock context leak the lock**

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unexpected exception, the `contextmanager` `finally` block will run and release the lock correctly — that part is fine. But the exception propagates out of `reserve_item` entirely uncaught, returning nothing to the caller rather than a `ReservationResult`. The public API contract of this method is to always return a `ReservationResult`. Wrap the body in a broad `except Exception` at the outermost level, log it, and return a safe failure result.

---

### 🟡 Medium — Architecture

**5. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is aggressive for a government-grade system**

At TTL/3, you get roughly three heartbeat windows before expiry. That is reasonable in theory, but 30 seconds is a long time to hold a distributed lock on a hot inventory item during a flash sale. Consider whether the DB operations actually need that window, and document the rationale. More importantly, the TTL should be configurable via constructor argument or environment variable rather than a hardcoded class constant — this is a deployment concern, not a code concern.

---

### 🟡 Medium — Architecture

**6. No Redis connection error handling**

Any `redis.RedisError` raised during lock acquisition, heartbeat extension, or lock release is unhandled and will propagate as an untyped exception. At minimum, Redis connectivity failures during acquisition should be caught and converted to `LockAcquisitionError`. Failures during release should be logged as errors (the lock will expire naturally, but the operator needs to know).

---

### 🟡 Medium — Test Coverage Gaps

**7. Missing tests for the critical failure paths**

The following scenarios have no test coverage:

- Heartbeat detects lock loss (`_extend_script` returns 0) — what does the caller receive?
- `db.get_stock()` raises an exception mid-lock — does the method return a `ReservationResult` or propagate?
- `db.transaction().__exit__` receives an exception (rollback path) — is stock left in a consistent state?
- Lock acquired but `db.update_stock` raises — verifies the lock is still released in `finally`.
- `redis.RedisError` raised during `self.redis.set` — verifies graceful degradation.

The existing `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()`, but because `register_script` returns a single shared mock, this assertion would pass even if the extend script were called instead. Tighten the mock setup to distinguish the two registered scripts.

---

### 🟢 Minor — Style / Observability

**8. Log the lock-loss event in the heartbeat**

```python
if result == 0:
    return  # silent
```

This should log at `ERROR` level. A lost lock during an active reservation is a significant operational event.

**9. `order_id` is generated before the transaction**

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(order_id, ...)
```

This is fine functionally, but if the transaction rolls back and the caller retries, a new `order_id` will be generated. Idempotency keys should be considered if the upstream caller may retry on timeout.

---

## Required Before Approval

1. Move `get_stock` inside the DB transaction, or add a DB-level optimistic lock / conditional update.
2. Signal the main thread when the heartbeat detects lock loss, and abort the operation.
3. Check `t.is_alive()` after `join` and handle the case where the thread did not stop.
4. Catch broad exceptions inside `reserve_item` and always return a `ReservationResult`.
5. Handle `redis.RedisError` explicitly in lock acquisition and release.
6. Add tests for the missing failure paths listed in finding #7.
