Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, heartbeat scoping, and input validation are all solid. However, there are several correctness and reliability issues that must be addressed before this merges: a race condition between stock read and DB transaction, missing exception handling around the database path, and test coverage gaps that would let real failure modes go undetected. None of these are cosmetic; each represents a realistic failure scenario under production load.

---

## Detailed Findings

### 🔴 Critical

**1. Stock read occurs outside the database transaction (TOCTOU within the lock)**

```python
# Current — read is outside the transaction block
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The Redis lock serializes callers at the application layer, but `get_stock` runs outside the DB transaction. If the DB supports any read-committed isolation or if a non-lock-using path (admin tool, migration, direct DB write) modifies stock between the read and the `update_stock` call, you silently oversell. The read must be inside the transaction, and `update_stock` should use a conditional decrement — not a blind set — to make the DB itself the final authority.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    rows_affected = self.db.decrement_stock_if_sufficient(req.item_id, req.quantity)
    if rows_affected == 0:
        return ReservationResult(success=False, message="Stock changed during reservation")
    self.db.create_order(order_id, req.item_id, req.quantity)
```

---

**2. No exception handling around the database operations**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # can throw
    ...
    with self.db.transaction():                        # can throw
        self.db.update_stock(...)                      # can throw
        self.db.create_order(...)                      # can throw
```

Any uncaught exception here propagates out of the `with self._acquire_lock(...)` block. The `finally` clause in `_acquire_lock` will still release the lock correctly — that part is fine — but the exception surfaces to the caller as an unhandled error rather than a structured `ReservationResult`. Under a flash sale load, DB timeouts and connection pool exhaustion are not edge cases. Add a `except Exception` catch inside the lock context that logs the error and returns a safe `ReservationResult`.

---

### 🟡 Significant

**3. Heartbeat thread logs nothing on lock loss — silent failure**

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # lock was lost — exits silently
```

If the heartbeat detects that the lock was lost (another process stole it, Redis evicted it under memory pressure), the thread exits with no log at ERROR or WARNING level. The main thread continues executing the DB transaction believing it holds the lock. This should log at ERROR and ideally set a shared flag the main thread can check before committing.

```python
if result == 0:
    logger.error("Heartbeat lost lock key=%s — lock was not renewed", lock_key)
    lost_event.set()  # signal main thread
    return
```

**4. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A request for `quantity=999999999` passes validation. Add a reasonable `le=` ceiling (e.g., `le=1000` or whatever the business rule is). Without it, a single malformed or malicious request can drain all stock in one call, bypassing any rate-limiting intent of the flash sale design.

---

### 🟠 Moderate

**5. `t.join()` timeout does not verify the thread actually stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join()` with a timeout returns silently whether or not the thread stopped. If the heartbeat thread is blocked on a slow Redis call, the lock release fires while the heartbeat thread may still be running and attempting an extension. Check `t.is_alive()` after the join and log a warning if the thread did not stop cleanly.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not stop cleanly for key=%s", lock_key)
```

**6. Redis connection errors are not handled in `_acquire_lock`**

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

A `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` here will propagate as an unhandled exception rather than a `LockAcquisitionError`. Wrap the retry loop body and map Redis exceptions to `LockAcquisitionError` so the caller always gets a consistent error type.

---

### 🔵 Test Coverage Gaps

**7. No test for DB exception inside the lock**

There is no test verifying that a DB failure (e.g., `db.update_stock` raises) results in a safe `ReservationResult` rather than an unhandled exception. This is the most important missing test given finding #2.

**8. No test for heartbeat lock-loss path**

The `result == 0` branch in `_beat` is untested. Given that this is the silent-failure path identified in finding #3, it needs an explicit test.

**9. `test_lock_release_uses_lua_script` asserts `assert_called_once()` without verifying arguments**

```python
service._release_script.assert_called_once()
```

This confirms the script was called but not that it was called with the correct token. A regression that passes the wrong token would not be caught. Use `assert_called_once_with(keys=[...], args=[...])` — or at minimum verify the `args` contain a valid UUID.

---

## What Is Working Well

- Lua-based atomic release and extension are correctly implemented and the token-match pattern is right
- `SET NX EX` is the correct single-command atomic acquisition
- Heartbeat interval at TTL/3 is a sound choice
- `stop_event.wait(timeout=...)` as a timer (rather than `time.sleep`) is the correct pattern — the thread wakes immediately on shutdown signal
- Pydantic validation with the regex pattern on `item_id` is appropriate for this surface
- Exponential backoff with jitter is correct
- Structured logging fields are consistent
