Decision: Request Changes

Summary: The core distributed locking pattern is sound — atomic Lua scripts, heartbeat renewal, exponential backoff, and Pydantic validation are all well-implemented. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a race condition in the stock-check flow to silent exception swallowing and missing test coverage for critical failure paths.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU race between `get_stock` and `update_stock`**

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)
# ...
new_stock = current_stock - req.quantity

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # Overwrites absolute value
    self.db.create_order(...)
```

The stock read happens outside the database transaction. If the DB has its own concurrent writers (e.g., a separate refund or admin adjustment path that bypasses this Redis lock), the `get_stock` value can be stale by the time `update_stock` runs. The write should be a conditional decrement (`UPDATE ... WHERE stock >= quantity AND stock = :expected`) or the read must be inside the transaction with a row-level lock (`SELECT FOR UPDATE`). Writing an absolute value computed from a pre-transaction read is a classic lost-update bug.

```python
# Preferred pattern:
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

**2. Unhandled exceptions inside the lock context leak the lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # Can raise
    # ...
    with self.db.transaction():                      # Can raise
        self.db.update_stock(...)                    # Can raise
```

Any uncaught exception from the DB layer propagates out of `reserve_item` entirely — it is not caught, not logged with context, and not returned as a `ReservationResult`. The `finally` block in `_acquire_lock` will still release the lock, which is correct, but the caller receives an unhandled exception rather than a graceful failure response. This is especially dangerous in a web handler context where it could surface as a 500 with internal detail.

```python
# Add inside reserve_item:
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟠 High

**3. Heartbeat thread failure is completely silent**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # Lock was lost — silently exits
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` (lock stolen or expired), the heartbeat exits silently. The main thread continues executing the DB transaction believing it holds the lock, but it does not. This is the most dangerous scenario in a flash sale context — another process may have acquired the lock and already committed a reservation for the same stock. The main thread should be notified.

```python
# Use an event to signal lock loss back to the main thread:
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat lost lock key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
```

The main thread should check `lock_lost_event` before and after the DB transaction, and abort with an error if it is set.

---

**4. `quantity` has no upper bound — denial of service / abuse vector**

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity constraint. A caller can submit `quantity=999999999`, which passes validation, acquires the lock, and causes a stock check that will always fail — but still holds the lock for the duration. In a flash sale scenario this is a trivial denial-of-service against the lock key. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)  # tune to business rules
```

---

**5. `item_id` is logged directly without sanitization**

```python
logger.info(
    "Reserved item_id=%s quantity=%d order_id=%s remaining=%d",
    req.item_id, req.quantity, order_id, new_stock
)
```

`item_id` passes the regex `^[a-zA-Z0-9_\-]+$`, which does prevent log injection from most payloads. However, the same `item_id` is also interpolated into the `lock_key` string and passed to Redis. This is safe given the regex, but the defense-in-depth review should confirm the regex is applied *before* any logging or Redis call — which it is via Pydantic at the top of `reserve_item`. This is acceptable, but worth a comment.

### 🟡 Medium

**6. `t.join(timeout=...)` result is not checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not terminate within the timeout (e.g., blocked on a Redis call), `join` returns silently and the release script runs anyway. This is probably fine in practice, but the thread state should be checked:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

**7. Test for Lua release atomicity is shallow**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This only asserts the script was called once — it does not assert the correct `keys` and `args` were passed. A refactor that calls the script with the wrong token would still pass this test.

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[ANY]  # token is a uuid, use unittest.mock.ANY
)
```

---

**8. No test for heartbeat lock-loss scenario (finding #3)**

There is no test that simulates the extend script returning `0` mid-operation and verifies the main thread is notified or aborts safely. This is the highest-risk failure mode and should have explicit coverage.

---

**9. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` — document the ratio**

The comment says `TTL/3` but `8` is not `30/3`. It is `30/3.75`. This is not a bug, but in a government contracting context where this code will be audited, magic numbers with incorrect comments create unnecessary review friction. Either correct the comment or derive the interval:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # Renew at 1/3 of TTL
```
