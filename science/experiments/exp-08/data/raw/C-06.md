Decision: Request Changes

Summary: The implementation demonstrates solid surface-level discipline — atomic Lua scripts, heartbeat renewal, Pydantic validation, structured logging — but contains a critical architectural flaw in the gap between lock expiry detection and protected operation continuity, a silent data-integrity hole in the stock-read/write path, and a test suite that validates mechanics without simulating the failure modes the code was written to prevent. None of these are style concerns; they are correctness and safety defects.

---

## Finding 1 — Critical: Heartbeat Detects Lock Loss, But the Protected Operation Does Not

This is the central flaw. The `_beat` thread calls `_extend_script` and checks `result == 0`. When the lock is lost (token mismatch, Redis eviction under memory pressure, network partition), the heartbeat thread **returns silently**. The main thread — currently executing `db.update_stock` or `db.create_order` — **has no knowledge of this**. It continues to completion, commits the transaction, and returns `ReservationResult(success=True)`.

The lock existed to serialize access to inventory state. Once the lock is gone, another process may have already acquired it, read the same stock value, and begun its own write. Both transactions commit. Stock is oversold.

The heartbeat's `result == 0` branch is not a safety mechanism. It is a thread exit. The protected region is not interrupted, not notified, and not rolled back.

**What is required:** The heartbeat must communicate loss back to the main thread. A `threading.Event` (e.g., `lock_lost_event`) set on `result == 0` is the minimal fix. The main thread must check this event at meaningful points within the protected region — or the context manager must expose it so callers can check it. The database transaction must not commit if the event is set.

```python
# Sketch — not a complete fix
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()   # signal the main thread
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In reserve_item, before committing:
if lock_lost.is_set():
    raise LockLostError(f"Lock lost mid-operation for {lock_key}")
```

This still has a race window (lock lost between the check and the commit), but it closes the wide-open case. A more robust design yields the `lock_lost` event from `_acquire_lock` so the caller can gate every meaningful step.

## Finding 2 — Critical: Stock Read Is Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)   # outside transaction

# ... validation ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # inside transaction
    self.db.create_order(...)
```

The stock read and the stock write are not atomic at the database level. The Redis lock serializes callers of *this service*, but it provides no protection against:

- A direct database write from another service, admin tool, or migration script
- A Redis failover or partition that causes two instances to both believe they hold the lock (split-brain on Redis Sentinel/Cluster without fencing)
- Any future code path that bypasses this lock

The correct pattern is to move `get_stock` inside the transaction and use a database-level locking read (`SELECT ... FOR UPDATE` in SQL) so the read and write are atomic at the storage layer regardless of what happens above it. The Redis lock is a performance optimization (avoiding DB contention); the DB transaction with a locking read is the correctness guarantee.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

## Finding 3 — Moderate: Lock Release Executes After `stop_event.set()` With No Loss Check

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat already detected `result == 0` (lock lost), the release script runs against a key that either no longer exists or belongs to another holder. The Lua script handles this correctly (it won't delete a foreign token), but the calling code has no awareness that the lock was lost. The `finally` block should check `lock_lost` (from Finding 1) and log or raise accordingly rather than silently proceeding as if release succeeded.

## Finding 4 — Moderate: `t.join(timeout=...)` Timeout Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(...)
```

`Thread.join(timeout=N)` returns whether or not the thread has exited. If the heartbeat thread is blocked (e.g., Redis call hanging), the join returns after the timeout and the release script runs while the heartbeat thread is still alive and may still call `_extend_script`. The thread is daemon, so it won't block process exit, but within the lifetime of a request this is a live race.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
    # do not proceed silently
```

## Finding 5 — Moderate: Test Suite Validates Mechanics, Not Failure Modes

The tests confirm the happy path, input rejection, and that `redis.delete` is not called directly. They do not test:

- **Lock lost mid-operation**: heartbeat returns `0` while `db.update_stock` is in progress. Expected behavior: transaction does not commit, result is not `success=True`. There is currently no test for this, and the code would fail it.
- **DB transaction failure after stock read**: `db.update_stock` raises. Is the order record created? Is the stock left in a consistent state?
- **`t.join` timeout**: heartbeat thread hangs. Does the release still execute? Does the caller get a correct result?
- **Redis extension failure on the first heartbeat beat vs. mid-operation**: timing matters.

These are the scenarios the distributed lock was built to handle. The test suite should be the proof that it handles them.

## Finding 6 — Minor: Retry Backoff Ceiling Is Absent

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff is `(2^2) * 0.1 + 0.05 = 0.45s`, which is fine at the default. But `retry_attempts` is a caller-controlled parameter with no documented maximum. A caller passing `retry_attempts=20` produces a backoff of over 100 seconds on the last attempt, blocking the thread. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Heartbeat loss (`result == 0`) must signal the main thread; protected operation must not commit if lock was lost |
| 2 | Critical | `get_stock` must move inside the DB transaction with a locking read (`SELECT FOR UPDATE`) |
| 3 | Moderate | `finally` block must inspect lock-lost state before treating release as nominal |
| 4 | Moderate | `t.join()` return must be checked; unexited heartbeat thread must be logged and handled |
| 5 | Moderate | Tests must cover lock-lost-mid-operation and DB failure paths |
| 6 | Minor | Backoff must be capped regardless of `retry_attempts` value |

The architecture is sound in intent. The gap between "lock coordination is lost" and "the code that depends on it learns this" is where the defects live, and that gap is not closed by any current mechanism.
