# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/flash-sale-inventory`

---

## Structural Analysis

Before reading a single line of implementation, I ask the question this PR's checklist never asks:

> *If the heartbeat fails to extend the lock — Redis drops it, another process acquires it — does the protected operation know? Or does it continue writing to the database as if the lock still holds?*

The answer here is: **it continues writing.** That is the load-bearing flaw in this design, and everything else is downstream of it.

---

## Finding 1 — Critical: Heartbeat Failure Is Silent to the Protected Operation

**Severity: Critical**
**Location:** `_acquire_lock` / `_beat` / `reserve_item`

The heartbeat thread detects lock loss correctly:

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # ← lock was lost. Thread exits.
```

The thread exits. Nothing else happens. The main thread, currently inside `reserve_item`, has no idea. It continues executing:

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← writes against lost lock
    self.db.create_order(order_id, req.item_id, req.quantity)
```

By the time the heartbeat detects the extension failure, another process may have already acquired the lock, read the stock, and begun its own transaction. Now two processes are writing concurrently against state that was supposed to be exclusively owned. The lock exists precisely to prevent this. The heartbeat detection is wired to nothing.

**The gap:** Lock loss is detected in the heartbeat thread. The protected operation lives in the main thread. There is no communication channel between them. The checklist item "heartbeat detects token mismatch (result == 0)" is true and meaningless — detection without signaling is not protection.

**Required fix:** The heartbeat must signal the main thread on lock loss. A `threading.Event` (e.g., `lock_lost_event`) set by `_beat` on `result == 0`, checked by the main thread at critical boundaries, or used to interrupt the operation via exception injection. The `yield` in the context manager must surface this condition. Alternatively, the context manager itself must expose a validity predicate that the caller is required to check before committing.

## Finding 2 — Critical: Stock Read Occurs Outside the Database Transaction

**Severity: Critical**
**Location:** `reserve_item`, lines reading stock then opening transaction

```python
current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

# ... stock check logic ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
```

The read and the write are not atomic at the database level. The Redis lock is the only thing preventing a concurrent read between these two lines. That is acceptable *only if the lock is guaranteed to hold for the entire window.* Finding 1 demonstrates it is not. If the lock expires between the read and the write, another process reads the same stock value, and both processes write a decrement computed from the same baseline. This is a classic TOCTOU race producing inventory oversell.

**Required fix:** Move `get_stock` inside the transaction boundary, or use a database-level `SELECT FOR UPDATE` / compare-and-swap on the stock column so the write is conditional on the value not having changed since the read. The distributed lock should be a second layer of defense, not the only one.

## Finding 3 — Significant: Lock TTL Is Not Calibrated to Actual Operation Duration

**Severity: Significant**
**Location:** `LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 8`

30 seconds is an arbitrary constant. The actual operation involves a network round-trip to Redis (stock read), a database transaction (two writes, potentially with replication lag), and any retry logic inside `db.transaction()`. Under load — slow DB, connection pool exhaustion, network partition — these operations can exceed 30 seconds. The heartbeat at 8-second intervals provides renewal, but:

- If the DB operation blocks the GIL or the thread scheduler delays the heartbeat thread, renewal may be late.
- Python's GIL means the heartbeat thread is not truly concurrent with CPU-bound work. For I/O-bound DB operations this is typically acceptable, but it is an assumption that is not validated anywhere.
- There is no maximum operation timeout. A hung DB call holds the lock indefinitely via heartbeat renewal, starving all other waiters for the duration.

**Required fix:** Define an explicit maximum operation deadline. The heartbeat should renew up to a bounded number of times or a wall-clock ceiling, after which it stops renewing and signals the operation to abort. This prevents a single slow operation from monopolizing the lock indefinitely.

## Finding 4 — Significant: `t.join()` Timeout Does Not Guarantee Thread Termination

**Severity: Significant**
**Location:** `_acquire_lock` finally block

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # 9 seconds
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked — on a Redis call that is hanging due to network partition — `join` returns after 9 seconds and the release script executes anyway. The thread is still running. It will eventually unblock, call `_extend_script`, and attempt to extend a lock that has already been released and potentially re-acquired by another process. The Lua script's token check prevents a foreign lock from being extended, but this is a latent race that should not exist.

**Required fix:** Check `t.is_alive()` after `join`. Log a critical-level alert if the thread did not terminate. Consider whether the release should be conditional or whether the architecture should be redesigned so the heartbeat thread cannot outlive the lock context.

## Finding 5 — Moderate: Retry Backoff Window Is Insufficient for Flash Sale Load

**Severity: Moderate**
**Location:** `_acquire_lock`, retry loop

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait before failure is approximately:
- Attempt 0 fails: sleep ~0.15s
- Attempt 1 fails: sleep ~0.25s
- Attempt 2 fails: raise

Total: ~0.4 seconds. Under flash sale load with hundreds of concurrent waiters, a 30-second lock TTL means most callers will fail immediately and return "service unavailable" to the user. The retry window is not calibrated to the lock hold time. The jitter range (`0.05`) is also very narrow, providing minimal thundering-herd protection.

This is not a correctness flaw but it is an operational one — the service will degrade visibly under exactly the load it was designed for.

## Finding 6 — Moderate: Test Suite Does Not Test the Failure Path That Matters

**Severity: Moderate**
**Location:** `tests/test_inventory.py`

The test suite covers: successful reservation, insufficient stock, item not found, lock contention (acquisition failure), input validation, and Lua script usage for release. These are all acquisition-path and release-path tests.

There is no test for:
- Heartbeat extension failure mid-operation (lock lost while DB transaction is in flight)
- Concurrent reservation producing oversell (two threads racing through the stock check)
- DB transaction failure after lock acquisition (partial failure cleanup)
- `join` timeout on heartbeat thread not terminating

The most dangerous failure mode — Finding 1 — has zero test coverage. A test suite that does not cover the scenario the architecture was designed to prevent is not a safety net.

---

## Finding 7 — Minor: Unhandled Exceptions Inside Lock Context Leak Lock State

**Severity: Minor**
**Location:** `reserve_item`, inside `with self._acquire_lock(lock_key):`

If `self.db.get_stock()` raises an unexpected exception (connection error, timeout), it propagates out of the `with` block. The `finally` clause in `_acquire_lock` will execute and release the lock correctly. However, `reserve_item` does not catch this exception — it propagates to the caller. The caller receives an unhandled exception rather than a `ReservationResult`. The public interface contract (always returns `ReservationResult`) is broken under DB failure.

**Required fix:** Wrap the lock body in a broad exception handler that returns a `ReservationResult(success=False, message="Internal error")` and logs the exception at error level with a stack trace.

## Summary of Findings

| # | Severity | Finding |
|---|----------|---------|
| 1 | **Critical** | Heartbeat detects lock loss but does not signal the main thread — protected operation continues writing against an expired, potentially re-acquired lock |
| 2 | **Critical** | Stock read is outside the DB transaction — TOCTOU race if the lock expires between read and write |
| 3 | Significant | No maximum operation deadline — a hung DB call holds the lock indefinitely via heartbeat |
| 4 | Significant | `t.join(timeout=...)` does not guarantee thread termination — zombie heartbeat thread can attempt to extend a released lock |
| 5 | Moderate | Retry backoff window (~0.4s total) is not calibrated to lock TTL (30s) — flash sale load will produce mass failure |
| 6 | Moderate | Test suite covers only acquisition and release paths — the mid-operation lock loss scenario has zero coverage |
| 7 | Minor | Unhandled DB exceptions inside the lock context break the `ReservationResult` return contract |

---

**Decision: Request Changes**

**Summary:** The heartbeat correctly detects lock loss but is wired to nothing — the main thread continues executing database writes after the lock has expired and may have been acquired by a competing process, directly defeating the purpose of the distributed lock. Compounding this, the stock read occurs outside the database transaction boundary, creating a TOCTOU race that the lock was the only defense against. These two findings together mean the system can oversell inventory under the exact load conditions it was designed for; both must be resolved with architectural changes before this
