Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a silent data-integrity gap in the stock-read/transaction boundary to a thread leak on Redis extension failure and missing exception handling around the DB layer.

---

## Detailed Findings

### 🔴 Critical

**1. Stock read occurs outside the database transaction (TOCTOU on the DB side)**

```python
# Line ~90 — stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes inside tx
```

The Redis lock serializes callers at the application layer, but `get_stock` runs outside the DB transaction. If the DB supports read-committed or higher isolation, a concurrent write from a non-lock-aware path (admin tool, migration, another service) can modify stock between the read and the `update_stock` call. The read must be inside the transaction, and ideally `update_stock` should use an optimistic or conditional update (e.g., `UPDATE ... WHERE stock = current_stock AND stock >= quantity`) so the DB itself enforces the invariant:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

---

**2. Heartbeat thread leaks silently on Redis extension failure**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← exits thread, but caller has no idea the lock expired
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat exits quietly. The `reserve_item` code continues executing under the assumption that it still holds the lock — it does not. A second process can acquire the lock and begin modifying the same stock record concurrently. The lock expiry must be communicated back to the main thread. A `threading.Event` or a shared flag should signal the worker, which should then abort:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
```

And the main path should check `lock_lost_event` before and after the DB transaction.

### 🟠 High

**3. No exception handling around DB operations**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Any DB exception here propagates uncaught out of `reserve_item`. The `finally` block in `_acquire_lock` will still release the lock correctly, but the caller receives an unhandled exception rather than a `ReservationResult`. For a service boundary this is a contract violation. Wrap DB operations and return a structured failure:

```python
except (DatabaseError, Exception) as e:
    logger.error("DB error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

Be careful not to leak internal error details (table names, query text) into the message returned to callers.

---

**4. `retry_attempts` is not configurable and the backoff ceiling is very low**

With `retry_attempts=3` and a base of `0.1s`, the maximum wait before giving up is roughly `0.1 + 0.2 + 0.4 = 0.7s` plus jitter. Under a flash sale, lock hold time can approach `LOCK_TTL` (30 s). Callers will almost always fail on contention. Expose `retry_attempts` and `retry_delay_base` as constructor parameters with sensible defaults, and consider a configurable ceiling on the backoff.

---

**5. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` constraint. A caller can request `quantity=2_000_000_000`. This won't crash the lock logic, but it will always produce "Insufficient stock" and still acquire the lock, blocking other legitimate reservations for up to 30 seconds. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)
```

Tune the ceiling to your business rules, but it must exist.

---

### 🟡 Medium

**6. Heartbeat thread is `daemon=True` — acceptable but worth documenting**

Daemon threads are killed abruptly on interpreter shutdown. The `finally` block in `_acquire_lock` calls `t.join(timeout=...)`, which mitigates this in normal operation, but if the process is `SIGKILL`ed the lock will expire naturally via Redis TTL. This is the correct behavior for a distributed lock, but it should be explicitly documented so future maintainers don't change `daemon=False` and introduce shutdown hangs.

---

**7. Test for lock release does not verify the correct token is passed**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test confirms the Lua script is called but does not assert that the `token` argument passed to the script matches the one used to acquire the lock. A regression that passes a hardcoded or empty token would pass this test. Assert on `call_args`:

```python
call_args = service._release_script.call_args
assert call_args.kwargs["args"][0] == <captured_token>
```

Capturing the token requires a small test helper or spy on `uuid.uuid4`.

---

**8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants, not instance parameters**

These values are tightly coupled to expected DB latency. In test environments or under different load profiles, they cannot be tuned without subclassing. Move them to `__init__` with defaults.

---

### 🔵 Minor / Nits

- `logger = logging.getLogger(__name__)` is correct, but the service never logs the Redis host/port on startup. A single `logger.info` in `__init__` aids operational debugging.
- The `EXTEND_LOCK_SCRIPT` passes `ARGV[2]` as a string (`str(self.LOCK_TTL)`). Redis `expire` accepts this, but it should be documented that the TTL is always in seconds.
- Consider registering Lua scripts lazily or verifying connectivity in `__init__` to fail fast on misconfiguration rather than at first reservation.

| 3 | 🟠 High | Wrap DB operations in `try/except`; return structured `ReservationResult` on failure |
| 4 | 🟠 High | Make retry count and backoff configurable; raise the practical ceiling |
| 5 | 🟠 High | Add upper bound to `quantity` field |
| 6 | 🟡 Medium | Document daemon thread behavior explicitly |
| 7 | 🟡 Medium | Strengthen lock-release test to assert correct token is passed |

Items 1 and 2 are data-integrity issues that can cause double-reservation under realistic concurrent load. They are blockers.
