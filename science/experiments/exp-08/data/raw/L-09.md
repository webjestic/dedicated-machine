## Code Review

### Critical Finding 1 — The Heartbeat Cannot Prevent the Race It Was Designed to Prevent (Architectural Defect)

This is the most serious flaw and it is invisible to every test in the suite.

The heartbeat fires at `HEARTBEAT_INTERVAL = 8` seconds and extends the lock by `LOCK_TTL = 30` seconds. The intent is correct. The implementation has a fatal gap:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← silent return; no signal propagated to the main thread
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost — expired and acquired by another process. The heartbeat thread silently exits. The main thread has **no knowledge of this**. It continues executing, reads stock, opens a database transaction, decrements stock, and creates an order — all while another process holds the lock and is doing the same thing. The distributed lock now provides zero mutual exclusion at the exact moment of the write. This is a silent double-spend path.

The fix requires a shared cancellation signal — a `threading.Event` or similar — that the heartbeat sets on lock loss, and the main thread must check before and ideally during the DB transaction. The current architecture has no such channel.

```python
# What is needed — sketch only
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()  # signal the main thread
            return
        ...

# In reserve_item, before the DB transaction:
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

Even with that check, there is a window between the check and the transaction commit. A fully correct implementation would require the DB transaction itself to validate a lock-ownership sentinel, or accept that this window exists and bound its impact.

### Critical Finding 2 — Stock Read Outside the Database Transaction (TOCTOU)

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

    if current_stock < req.quantity:
        return ReservationResult(...)

    new_stock = current_stock - req.quantity
    order_id = str(uuid.uuid4())

    with self.db.transaction():                        # ← transaction begins here
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read is outside the database transaction. If the DB supports read-committed or higher isolation, the value read may be stale by the time the write executes. The distributed lock is supposed to serialize this, but as shown in Finding 1, that serialization is not guaranteed. Even if it were, best practice for correctness-critical inventory operations is to move the read inside the transaction and use a `SELECT FOR UPDATE` or equivalent to hold a row-level lock. The current structure relies entirely on Redis for correctness, with no DB-level safety net.

### Finding 3 — Lock Release Executes After `stop_event.set()` But `t.join()` Has a Bounded Timeout

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is blocked (e.g., Redis is slow), `t.join()` returns after 9 seconds without confirming the thread has exited. The release script then executes. This is acceptable — the Lua script is atomic and token-matched, so a concurrent extend from a zombie heartbeat thread will correctly fail. However, the thread is not joined to completion; it is abandoned. In a high-throughput service this accumulates zombie threads. The thread should be daemon=True (it is), which limits the damage, but the condition should be logged.

### Finding 4 — Unhandled Exceptions Inside the Lock Context Leak No Information

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

Any exception from `get_stock`, `update_stock`, or `create_order` (network error, DB timeout, etc.) propagates out of `reserve_item` entirely unhandled — not caught, not converted to a `ReservationResult`. The caller receives a raw exception rather than a structured error response. For a service boundary this is a contract violation. A `try/except Exception` block inside the lock context with appropriate logging and a structured failure result is needed.

### Finding 5 — Test Suite Validates the Mock, Not the System

The tests mock `register_script` to return a `MagicMock(return_value=1)`. This means:

- The Lua scripts are never executed
- The heartbeat extension path is never exercised
- The lock-loss scenario (Finding 1) is never tested
- `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` — this only verifies the mock was called, not that the Lua script would behave correctly against a real Redis instance

There are no integration tests against a real or embedded Redis. For a distributed locking implementation in a flash sale context — where correctness under concurrency is the entire value proposition — this test suite provides false confidence. At minimum, a test using `fakeredis` or a real Redis instance in CI is required to validate the Lua scripts and the heartbeat timing.

### Minor Finding 6 — Quantity Upper Bound Not Enforced

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound on quantity. A request for `quantity=10_000_000` passes validation, reads stock, and if stock is sufficient, executes. This is likely unintended for a flash sale context and should have a domain-appropriate `le=` constraint.

---

## Decision: Request Changes

**Summary:** The heartbeat thread silently exits on lock loss with no signal to the main thread, meaning the distributed lock provides no mutual exclusion guarantee during the database write in the exact failure scenario it was designed to prevent — this is a silent double-spend path. The stock read outside the database transaction removes the DB-level safety net that would otherwise partially compensate. The test suite mocks away Redis entirely, so none of the concurrency properties are actually validated; the tests confirm the mock behaves as configured, not that the system is correct.
