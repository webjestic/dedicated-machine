Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all solid. However, there are several correctness, reliability, and security issues that must be addressed before this merges into a production inventory path, particularly the TOCTOU race between the stock read and the DB transaction, the silent swallowing of DB exceptions, and the heartbeat's inability to signal the caller when the lock is lost mid-operation.

---

## Detailed Review

### 🔴 Critical

**1. TOCTOU Race — Stock Read Outside the DB Transaction**

```python
# CURRENT — stock read is outside the transaction
current_stock = self.db.get_stock(req.item_id)   # <-- dirty read possible
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # <-- based on stale value
    self.db.create_order(...)
```

The Redis lock serializes *this service's* writers, but it does not protect against:
- Direct DB writes from other services or admin tooling that bypass the lock.
- Read-committed isolation gaps if the DB connection sees a snapshot from before the transaction opened.

The stock read must happen *inside* the transaction with a pessimistic row lock (`SELECT ... FOR UPDATE` in Postgres/MySQL). The Redis lock is a performance gate; the DB transaction is the correctness guarantee. Both are needed.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

---

**2. DB Exceptions Are Silently Unhandled — Lock Is Still Released Cleanly, But the Caller Gets No Signal**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
# If either of the above raises, the exception propagates out of _acquire_lock's
# try/yield, the lock IS correctly released (finally block runs), but the
# exception is completely unhandled by reserve_item — it will bubble up as
# an untyped exception to the caller rather than a ReservationResult.
```

The method signature promises `ReservationResult` in all cases. Add a broad exception handler:

```python
except (DatabaseError, Exception) as e:
    logger.error("DB error during reservation item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

Be specific about DB exception types where your ORM/driver exposes them.

---

**3. Heartbeat Lock Loss Is Silently Ignored — Operation Continues on an Expired Lock**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # <-- thread exits, but the main thread has NO idea
```

If the lock expires and another process steals it, the heartbeat exits quietly. The main thread continues executing the DB transaction believing it holds the lock. This defeats the entire purpose of the distributed lock for the oversell-prevention guarantee.

The heartbeat needs a signaling path back to the main thread:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Distributed lock lost — signaling main thread key=%s", lock_key)
            lock_lost_event.set()
            return
```

Then check `lock_lost_event` at the critical boundary before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost during operation for {lock_key}")
```

This is not a complete fix for all interleaving scenarios, but it prevents the most dangerous case: committing a transaction after lock expiry.

### 🟡 Significant

**4. Heartbeat Thread Leak on Redis Extension Failure**

If `self._extend_script(...)` itself raises (Redis connection drop, timeout), the exception is unhandled inside `_beat`, the thread dies without setting `lock_lost_event`, and the main thread has no visibility. Wrap the extension call:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        except redis.RedisError as e:
            logger.error("Heartbeat Redis error key=%s: %s", lock_key, e)
            lock_lost_event.set()
            return
        if result == 0:
            lock_lost_event.set()
            return
```

---

**5. `t.join(timeout=...)` Failure Is Not Checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join()` with a timeout returns `None` whether the thread stopped or timed out. If the heartbeat thread is stuck (e.g., blocked on a Redis call), the release script runs anyway — which is correct behavior — but a stuck daemon thread is a resource leak and an operational blind spot. Log a warning if the thread is still alive after join:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.warning("Heartbeat thread did not stop cleanly key=%s", lock_key)
```

---

**6. `retry_attempts=3` With Backoff Caps at ~0.45s Total Wait**

At 3 attempts with `2^attempt * 0.1` backoff: 0.1s, 0.2s, 0.4s. For a 30-second lock TTL under a genuine flash sale load spike, this will produce a high rate of `LockAcquisitionError` returns to end users. Consider making `retry_attempts` and max backoff configurable at the service level, or document explicitly that callers are expected to retry at the application layer.

---

### 🟠 Test Coverage Gaps

**7. No Test for Lock-Lost Mid-Operation**

The most dangerous scenario — lock expires between stock read and DB commit — has zero test coverage. Once the lock-lost signaling mechanism is added (see Critical #3), a test must verify that `reserve_item` returns a failure result (not a successful reservation) when the heartbeat signals lock loss before the transaction commits.

**8. `test_lock_release_uses_lua_script` Is Fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. A future refactor could call the script with wrong keys/args and this test would still pass. Assert the arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, use ANY or capture it
)
```

**9. No Test for DB Transaction Rollback Behavior**

There is no test verifying that if `db.update_stock` or `db.create_order` raises, `update_stock` is not committed (i.e., the transaction context manager's `__exit__` is called with the exception). This is especially important given Critical #2 above.

---

### 🔵 Minor / Informational

**10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants**

These should be constructor parameters with defaults, or at minimum pulled from a config object. Hardcoded class constants make it impossible to tune per-deployment without modifying source.

**11. `order_id` Generated Before Transaction**

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    ...
    self.db.create_order(order_id, ...)
```

This is fine functionally, but if the transaction rolls back and is retried externally, the same `order_id` will not be reused — a new one will be generated. Ensure the order table treats `order_id` as the idempotency key and that upstream callers understand a failed call does not mean a failed order (the DB may have committed before the connection dropped).

**12. Log Line Exposes `order_id` Before Confirming Commit**

```python
logger.info("Reserved item_id=%s ... order_id=%s ...", ...)
```

This log fires after the `with self.db.transaction()` block exits. If the transaction context manager raises on `__exit__` (e.g., commit failure), the log has already recorded a successful reservation that did not persist. Move the success log to after confirmed return, or accept this as a known operational caveat and document it.
