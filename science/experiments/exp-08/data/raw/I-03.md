Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, input validation, and structured logging are all present and correctly reasoned about. However, there are several correctness defects, a silent data-integrity gap, and test coverage holes that would be unacceptable in a high-contention flash-sale context where every race condition translates directly to oversold inventory or phantom orders.

---

## Critical Issues

### 1. Stock read occurs *outside* the database transaction — TOCTOU within the lock

```python
# CURRENT — read is outside the transaction block
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← dirty read possible
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← based on stale read
        self.db.create_order(...)
```

The distributed lock serializes *Redis-level* access, but `get_stock` executes outside the database transaction. If the database uses `READ COMMITTED` isolation (the default in PostgreSQL and MySQL), another concurrent writer — one that bypassed the Redis lock entirely, such as an admin tool, a migration script, a direct DB write, or a second service instance that lost its lock mid-operation — can commit a stock change between your read and your write. The stock decrement is then computed against a stale value, and you silently oversell.

**Required fix:** Move `get_stock` inside the transaction, or use a `SELECT ... FOR UPDATE` / optimistic version-check pattern so the database itself enforces the invariant:

```python
with self._acquire_lock(lock_key):
    with self.db.transaction():
        current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
        if current_stock is None: ...
        if current_stock < req.quantity: ...
        self.db.update_stock(req.item_id, current_stock - req.quantity)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is a *performance* optimization (reduces DB contention); the database transaction with a row-level lock is the *correctness* guarantee. Both layers must be present.

### 2. Heartbeat thread silently swallows lock loss — no propagation path to the business logic

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; thread exits, but reserve_item() has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat exits quietly. The `reserve_item` caller continues executing the database transaction believing it still holds the lock. Another process can acquire the lock and begin its own transaction concurrently. You now have two writers operating simultaneously on the same inventory row — exactly the race condition the lock was designed to prevent.

**Required fix:** Signal lock loss back to the main thread. One clean pattern:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost_event` at critical points inside the lock context, or raise from the context manager's `finally` block if the event is set before the release script runs.

### 3. Exception in `db.transaction()` block leaves the lock held until TTL expiry in the worst case — and the `finally` block may silently fail

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If `self._release_script(...)` raises (Redis connection error, network partition), the exception propagates out of `finally` and *suppresses* any exception that was already propagating from the `yield` block. This is a standard Python footgun with exceptions in `finally`. The lock key remains in Redis until TTL expiry, blocking all other reservations for up to 30 seconds.

**Required fix:**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Lock released key=%s", lock_key)
    except Exception:
        logger.exception("Failed to release lock key=%s — will expire at TTL", lock_key)
```

## Moderate Issues

### 4. `t.join()` timeout is silently ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(...)   # proceeds regardless
```

`Thread.join(timeout=...)` does not raise if the thread is still alive after the timeout — it simply returns. If the heartbeat thread is blocked (e.g., on a hung Redis call), the main thread proceeds to release the lock while the heartbeat thread is still running and may later call `_extend_script` on a key that has already been released and potentially re-acquired by another holder.

**Required fix:**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not release — the thread may still be mid-extend on a re-acquired lock.
    # Log and let TTL handle cleanup, or raise to surface the anomaly.
```

### 5. Retry backoff ceiling is extremely low under real contention

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~0.10s, attempt=1 → ~0.20s, attempt=2 → ~0.40s
```

With `retry_attempts=3`, the maximum total wait before giving up is under one second. Under a flash sale with hundreds of concurrent requests, the lock holder may legitimately hold the lock for several seconds (DB transaction latency). Nearly all contending callers will exhaust retries and return "service unavailable" to the end user. The jitter range (`0–0.05s`) is also too narrow to meaningfully spread thundering-herd retries.

**Suggested fix:** Increase `retry_attempts` to at least 5–10, widen the jitter window to `random.uniform(0, backoff)` (full jitter), and cap the ceiling:

```python
backoff = min((2 ** attempt) * 0.1, 2.0)
jitter = random.uniform(0, backoff)
time.sleep(backoff + jitter)
```

---

### 6. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are module-level constants with no runtime override

These are hardcoded class attributes. In a government/high-assurance context, operational parameters that affect correctness under load (lock duration, heartbeat frequency) must be configurable at deployment time — not require a code change and redeploy. Inject them via constructor parameters with documented defaults.

## Test Coverage Gaps

### 7. No test for heartbeat lock-loss propagation (Issue #2 above)

The most dangerous failure mode — heartbeat detects `result == 0` but business logic continues — has zero test coverage. This scenario is entirely plausible under Redis eviction pressure or a network partition.

### 8. No test for database transaction failure / rollback

There is no test asserting that a `db.update_stock` or `db.create_order` exception causes the transaction to roll back and the lock to be released. Given that the lock release is in a `finally` block that can itself raise (Issue #3), this interaction must be explicitly tested.

### 9. `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert arguments

```python
service._release_script.assert_called_once()
```

This passes even if the script is called with the wrong key or a mismatched token. The token-matching is the entire security property of the Lua release script. Assert `keys` and `args` explicitly.

### 10. No test for `t.join()` timeout / zombie heartbeat thread (Issue #4)

## Minor / Style

- `logger.warning` during normal retry is arguably too noisy at `WARNING` level for a flash sale with expected contention; consider `INFO` with a counter, escalating to `WARNING` only after a threshold.
- The `ReservationResult` model exposes `remaining_stock` on failure paths (e.g., insufficient stock). Verify this is intentional and not an inadvertent inventory disclosure to callers who should not know current stock levels.
- `order_id` is generated before the transaction begins. If the transaction rolls back and is retried externally, a new `order_id` will be generated, which is correct — but this should be documented explicitly so future maintainers do not move the `uuid4()` call inside a retry loop expecting idempotency.
