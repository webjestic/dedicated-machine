# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/flash-sale-inventory`

---

## Structural Analysis

Before reading a single line of implementation, I simulate the failure topology. The central claim of this PR is that a heartbeat thread prevents lock expiry during legitimate operations. The question I ask before reading is:

> *If the heartbeat fails silently — or succeeds but the lock is lost through a mechanism the heartbeat cannot observe — does the protected operation know? Or does it continue writing against state it no longer owns?*

The answer to that question determines whether this architecture is sound. Reading the code with that question active reveals several issues the checklist does not name.

---

## Findings

### 🔴 CRITICAL — Heartbeat Failure Is Silent; Protected Operation Is Never Notified

This is the central architectural flaw. The heartbeat thread detects lock loss — `result == 0` from the extend script — and responds by returning silently:

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
if result == 0:
    return  # ← thread exits. nothing else happens.
```

The `reserve_item` body — currently executing inside `with self._acquire_lock(lock_key):` — receives no signal. No event is set. No exception is raised. No flag is checked. The database transaction proceeds, stock is decremented, and an order is created against inventory state that is no longer exclusively owned by this process.

A second process may have acquired the lock in the interval between TTL expiry and the heartbeat's next fire. Both processes now read the same stock value, both decrement it, and both commit. The lock existed to prevent exactly this. The heartbeat's silent exit means the lock's failure is invisible to the code the lock was protecting.

The checklist entry — *"heartbeat detects token mismatch (result == 0)"* — is accurate as a description of detection. It does not describe what happens after detection. What happens after detection is nothing. Detection without propagation is not a safety mechanism.

**What is required:** The heartbeat must propagate lock loss into the critical section. One correct pattern is a shared `lock_lost_event` that the heartbeat sets on `result == 0`, and that the protected code checks at meaningful boundaries — before and after the database transaction. Another is raising into the main thread via a registered callback. The specific mechanism is negotiable; the propagation is not.

### 🔴 CRITICAL — Stock Read Occurs Outside the Database Transaction

```python
# Outside transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

new_stock = current_stock - req.quantity

# Inside transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The Redis lock serializes callers *through this service*, but it does not prevent a write from a different service, a migration script, an admin tool, a replica lag scenario, or any other actor that does not participate in this lock protocol. The read is a snapshot. The write is committed against that snapshot. The gap between them is unguarded at the storage layer.

The correct pattern is to move `get_stock` inside the transaction boundary and use a database-level locking primitive — `SELECT FOR UPDATE` or equivalent — so that the read and write are atomic at the layer that owns the data. The Redis lock is a coordination layer; the database transaction is the durability layer. Both are necessary. Neither substitutes for the other.

### 🟡 SIGNIFICANT — Lock Release Executes After `stop_event.set()` But Before `t.join()` Completes; Race Window Exists

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` is not a guaranteed join. If the heartbeat thread does not exit within `HEARTBEAT_INTERVAL + 1` seconds — due to a slow Redis call, a blocked network operation, or any other delay — `join` returns and the release script executes while the heartbeat thread is still alive. The heartbeat thread may then call `_extend_script` after the lock has been released, and after another process has acquired it. The extend script is token-gated, so in the normal case this is a no-op. But the window exists, and under network partition or Redis slowness it becomes a real race.

The join timeout should be bounded tightly and a warning should be logged if it is exceeded. More importantly, the heartbeat thread should be designed so that its Redis call is itself bounded — a timeout on the Redis operation — so that the join is reliable.

### 🟡 SIGNIFICANT — `db.transaction()` Exception Handling Is Absent; Lock Is Released on DB Failure

If `self.db.update_stock` or `self.db.create_order` raises, the context manager's `finally` block releases the lock and logs success. The exception propagates up through `reserve_item` with no catch, surfaces as an unhandled exception to the caller, and the lock is released. This is the correct lock behavior — but the caller receives an unhandled exception rather than a `ReservationResult`, which violates the function's declared return contract. The function signature says it returns `ReservationResult`. Under database failure it does not.

---

### 🟡 SIGNIFICANT — Heartbeat Thread Is Daemon; Daemon Threads Are Not a Reliability Guarantee

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed when the main thread exits, without running `finally` blocks. In a long-running service this is generally acceptable. But if the process exits during a reservation — due to a signal, an OOM kill, or a crash — the lock is not released, the heartbeat is killed, and the lock holds until TTL expiry. This is expected behavior for a crash scenario and is not itself a flaw. It should be documented, because the TTL becomes the recovery window and operators need to know it.

The more subtle issue: if this code runs inside a framework that manages thread pools or greenlets, daemon thread semantics may interact unexpectedly with the framework's lifecycle. This is an integration concern, not a code defect, but it warrants documentation.

---

### 🟡 SIGNIFICANT — Backoff Ceiling Is Unbounded Relative to Lock TTL

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45s`. This is small relative to the 30-second TTL and is not itself dangerous. However, `retry_attempts` is a caller-controlled parameter with no upper bound. A caller passing `retry_attempts=20` produces a backoff of `(2^19) * 0.1 ≈ 52,428 seconds` on the final attempt. The backoff should be capped.

---

### 🟠 MODERATE — Test Suite Does Not Test the Failure Path That Matters

The tests verify acquisition, release, insufficient stock, item not found, input validation, and lock contention. None of them test what happens when the heartbeat detects lock loss mid-operation. This is the most critical failure path in the design, and it has zero test coverage. The checklist marks heartbeat behavior as tested. It is not tested.

A test for this path would: acquire the lock, simulate `_extend_script` returning 0 mid-operation, and assert that the protected operation does not complete a write. This test does not exist and could not pass against the current implementation, because the current implementation does not implement that behavior.

---

### 🟠 MODERATE — `token` Is Yielded but Unused; Yield Value Creates a Misleading Interface

```python
yield token
```

The token is yielded, but the caller — `reserve_item` — ignores it:

```python
with self._acquire_lock(lock_key):
    ...
```

Yielding the token implies it is part of the public interface of the context manager. If a future caller uses it to perform manual lock operations, they will be operating on the lock's internal credential. The token should not be yielded unless there is a documented use case for it. If it is yielded for testability, that is a design smell — testability should not drive exposure of internal state.

---

### 🟢 NOTED — Lua Scripts Are Correct

The release and extension scripts are correctly token-gated. The release script deletes only if the token matches. The extension script extends only if the token matches. Both are registered at construction time. These are correct.

---

### 🟢 NOTED — Input Validation Is Correct

Pydantic validation with pattern enforcement on `item_id` and `ge=1` on `quantity` is correct. `ValidationError` is caught and returned as a `ReservationResult`. The path traversal test case (`../../admin`) correctly fails validation.

---

### 🟢 NOTED — Lock Acquisition Is Correct

`SET NX EX` is atomic. The token is a UUID generated per acquisition. Exponential backoff with jitter is present. These are correct.
