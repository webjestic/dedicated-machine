Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, heartbeat scoping, backoff/jitter, and Pydantic validation are all solid. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a silent data-integrity gap to a thread-leak vector and incomplete test coverage of critical failure paths.

---

## Findings

### 🔴 Critical — Data Integrity

**1. Stock read is outside the database transaction (TOCTOU within the lock)**

```python
# Line ~90 in reserve_item
current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
    self.db.create_order(...)
```

The distributed lock prevents *other processes* from racing, but it does **not** protect against the database itself returning a stale read if the connection uses `READ COMMITTED` or if the lock is ever lost between the read and the write. The stock read and the conditional write must both occur inside the transaction. Use a `SELECT ... FOR UPDATE` (or equivalent) inside the transaction to make the check-and-decrement atomic at the DB layer as well. The Redis lock is a performance optimization — it should not be the sole correctness guarantee.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...  # early return / rollback
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

### 🔴 Critical — Resilience / Thread Safety

**2. Unhandled exception inside `_acquire_lock` leaks the heartbeat thread**

If `self.db.get_stock(...)` or any code inside the `with self._acquire_lock(lock_key):` block raises an unexpected exception (e.g., a DB connection error), the `finally` block in `_acquire_lock` *does* run — good. But the lock release script call is **not guarded**:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])  # ← can raise, swallows context
```

If `_release_script` raises (Redis connection dropped, network partition), the exception propagates out of `finally` and **masks the original exception** from the `with` block. Wrap the release in a `try/except` with logging:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
    except Exception:
        logger.exception("Failed to release lock key=%s — TTL will expire naturally", lock_key)
```

**3. `t.join()` timeout is silently ignored**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

If the heartbeat thread hangs (e.g., Redis call blocks), `join` returns after the timeout but the thread is still running. There is no check and no log. Since the thread is `daemon=True` it won't prevent process exit, but it *will* hold a reference to `token` and continue attempting to extend a lock that has already been released. Add a liveness check:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate for key=%s", lock_key)
```

### 🟡 Moderate — Security

**4. Item ID is reflected into log messages without sanitization**

```python
logger.info("Reserved item_id=%s ...", req.item_id, ...)
logger.error("Lock acquisition failed item_id=%s: %s", req.item_id, e)
```

Even though `item_id` is validated by the regex `^[a-zA-Z0-9_\-]+$`, the validated value is still user-supplied and is being written to structured logs. In a government contracting context, logs are often shipped to SIEMs or audit systems. Confirm that the logging backend escapes or sanitizes structured fields, and consider explicitly marking user-supplied values in the log record (e.g., a `user_input` key) so downstream tooling can apply appropriate handling.

**5. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can pass `quantity=2_000_000_000`. While the stock check prevents actual over-reservation, an unbounded integer still passes validation and reaches the DB layer. Add a reasonable domain ceiling:

```python
quantity: int = Field(..., ge=1, le=10_000)
```

### 🟡 Moderate — Correctness

**6. Heartbeat logs extension success but not failure**

```python
if result == 0:
    logger.debug("Lock TTL extended key=%s", lock_key)  # ← wrong branch
    return
```

Wait — re-reading the code: `logger.debug` fires on the *else* branch (result != 0, i.e., success). When `result == 0` (lock lost), the thread silently returns. This is a silent failure during an active reservation. Log it at `ERROR` level:

```python
if result == 0:
    logger.error("Heartbeat: lock lost for key=%s — reservation may be compromised", lock_key)
    return
logger.debug("Lock TTL extended key=%s", lock_key)
```

### 🟡 Moderate — Test Coverage Gaps

**7. No test for DB exception inside the lock**

There is no test verifying behavior when `db.update_stock` or `db.create_order` raises. This is the path most likely to expose the exception-masking issue in finding #2.

**8. No test for heartbeat thread cleanup**

`test_lock_release_uses_lua_script` confirms the Lua script is called, but there is no assertion that the heartbeat thread terminates (i.e., `t.is_alive() == False` after `reserve_item` returns). This is testable by subclassing or injecting a spy on `threading.Thread`.

**9. `test_reserve_lock_contention` does not verify retry count**

The test sets `redis_client.set.return_value = False` but does not assert that `set` was called exactly `retry_attempts` (3) times. A regression that short-circuits retries would not be caught.

```python
assert redis_client.set.call_count == 3
```

---

### 🟢 Minor — Style / Maintainability

**10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants but not configurable**

For a service deployed across environments (dev, staging, prod), these should be injectable via constructor parameters with the class constants as defaults. This also makes them testable without monkey-patching.

**11. `reserve_item` returns early from inside the lock context without logging**

The "Item not found" and "Insufficient stock" early returns do not log at any level. Under load, silent no-ops make debugging flash sale incidents difficult.
