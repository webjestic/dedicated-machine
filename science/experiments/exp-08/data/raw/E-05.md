# Pull Request Review

**PR:** Add distributed inventory reservation for flash sale items
**Author:** dev-team
**Branch:** feature/flash-sale-inventory

---

## Structural Analysis

Before reading a line of implementation, I simulate the failure modes that live *between* the documented paths. The checklist is well-formed. That is precisely why I read it last.

---

### Issue 1 — CRITICAL: The Heartbeat Detects Lock Loss But Cannot Stop the Protected Operation

This is the central architectural flaw. The author has correctly identified that the heartbeat must detect token mismatch and return early:

```python
if result == 0:
    return  # heartbeat exits
```

But the heartbeat thread exiting is not the same as the protected operation stopping. The two threads are not coordinated on the failure path. When the lock expires mid-operation and the heartbeat detects it, the main thread continues executing as if the lock still holds:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # <-- lock may be gone here
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # <-- writes against invalidated state
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The gap is concrete. Consider this sequence:

1. Lock acquired. Heartbeat thread starts.
2. DB operation stalls. Lock TTL expires before the next heartbeat fires.
3. A second process acquires the same lock. It reads stock, begins its own reservation.
4. Heartbeat fires. `EXTEND_LOCK_SCRIPT` returns `0`. Heartbeat thread exits quietly.
5. Original thread resumes. It commits `update_stock` and `create_order` — against stock state that a second holder now also believes it owns.

The result is a double-spend of inventory. The heartbeat detection is wired to a dead end. There is no cancellation signal back to the main thread. The checklist item "heartbeat detects token mismatch (result == 0)" is accurate but incomplete — detection without propagation is not protection.

**What is required:** A shared cancellation event between the heartbeat thread and the main execution context. When the heartbeat detects `result == 0`, it must set a flag the main thread checks at meaningful boundaries — before the DB read, before the DB write, and ideally as a guard inside the transaction. The main thread must treat that signal as a hard abort.

### Issue 2 — CRITICAL: Lock Release After Detected Loss Attempts a No-Op Delete, But the Sequence Is Still Wrong

In the `finally` block:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If the lock expired and was acquired by another process, the Lua release script correctly returns `0` and does not delete the foreign lock. That part is safe. However, the `finally` block has no awareness of whether the protected operation completed against valid or invalid lock state. It releases cleanly regardless. The log line `"Lock released key=%s"` will appear in both the legitimate and the corrupted path, making post-incident forensics misleading.

Additionally: `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` — if the heartbeat thread is blocked on a slow Redis call, the join can time out and the main thread proceeds to release before the heartbeat has finished its last extension attempt. This creates a narrow race where the heartbeat extends the lock *after* the main thread has already released it, briefly re-extending a key that should be gone. The Lua token check prevents a foreign lock from being extended, but if the release completed and the key was deleted, a re-extension on a now-absent key will silently create a new TTL-bearing key under the same name with no holder. This is a ghost lock.

### Issue 3 — SIGNIFICANT: Stock Read Occurs Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)  # outside transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The distributed lock is intended to serialize access, and under normal operation it does. But this design creates a layered dependency: database consistency is contingent on Redis correctness. If the lock fails for any reason — expiry, Redis partition, heartbeat race — the database has no independent protection. The read-modify-write must occur within the transaction, or the stock update must use a conditional decrement (`UPDATE ... WHERE stock >= quantity`) that enforces the invariant at the database layer regardless of lock state. Defense in depth is not optional in a flash sale context.

---

### Issue 4 — MODERATE: Heartbeat Join Timeout Is a Silent Failure

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

If the join times out, execution continues silently. There is no log, no metric, no exception. A hung heartbeat thread is an operational signal — it likely indicates a Redis connectivity problem. Silently proceeding past a failed join means the operator has no visibility into a condition that may be actively corrupting lock state. At minimum, log a warning. At best, treat a join timeout as a lock integrity failure and abort the operation.

---

### Issue 5 — MODERATE: Retry Backoff Does Not Reach Meaningful Contention Durations

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is approximately `0.4s + jitter`. The lock TTL is 30 seconds. Under genuine flash sale load, a lock held for several seconds will exhaust all retries in under a second and return `LockAcquisitionError`. The retry window is not calibrated to the lock duration. Either the retry count must be substantially higher, the backoff ceiling must be raised, or the caller must be given a configurable retry policy. As written, under realistic load, most callers will fail immediately.

---

### Issue 6 — MODERATE: `db` Abstraction Is Untested for Exception Behavior

The tests mock `db` with `MagicMock` and do not simulate DB exceptions inside the transaction. If `update_stock` raises, the `with self.db.transaction()` context manager's `__exit__` is responsible for rollback. The mock's `__exit__` returns `False`, which is correct for non-suppression, but the test never verifies that a DB failure inside the transaction does not leave a partial write. In production, if `create_order` raises after `update_stock` succeeds and the transaction does not roll back correctly, stock is decremented with no corresponding order. This is a test coverage gap against a real failure mode.

---

### Issue 7 — LOW: `daemon=True` on Heartbeat Thread Masks Ungraceful Shutdown

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed when the main process exits without waiting for them to complete. If the process is terminated mid-operation, the heartbeat dies without releasing the lock. The lock will eventually expire via TTL, which is acceptable. However, if the process is part of a graceful shutdown sequence that expects all threads to complete cleanly, daemon threads will not participate in that sequence. This is a deployment concern, not a correctness concern, but it should be documented.

---

## Test Coverage Assessment

The tests cover the happy path, insufficient stock, item not found, lock contention, input validation, and Lua release verification. These are surface-level correctness tests. None of the following are tested:

- Heartbeat detecting lock loss mid-operation (the critical path)
- DB exception inside transaction
- Heartbeat join timeout
- Lock expiry between stock read and stock write
- Concurrent callers racing after lock expiry

The test suite validates that the code does what it says. It does not validate that the code is safe when the environment does not cooperate.

---

## Summary of Findings

| Severity | Issue |
|---|---|
| CRITICAL | Heartbeat detects lock loss but has no mechanism to abort the protected operation — double-spend is possible |
| CRITICAL | Heartbeat join race can produce a ghost lock via post-release extension |
| SIGNIFICANT | Stock read is outside the DB transaction — database has no independent consistency guarantee |
| MODERATE | Join timeout is silent — no logging, no abort |
| MODERATE | Retry backoff window is too short relative to lock TTL for real flash sale load |
| MODERATE | No test coverage for DB exceptions inside transaction |
| LOW | Daemon thread behavior during graceful shutdown is undocumented |

---

**Decision:** Request Changes

**Summary:** The heartbeat correctly detects lock loss via token mismatch but has no path to communicate that loss back to the executing operation — the main thread will complete a DB write against invalidated lock state, producing a double-spend under expiry conditions. Compounding this, the stock read occurs outside the database transaction, meaning database consistency is entirely dependent on Redis correctness with no independent enforcement layer. These two issues together represent a correctness failure in the exact scenario this service is designed to protect against; the remaining issues must also be addressed before this is suitable for a flash sale workload.
