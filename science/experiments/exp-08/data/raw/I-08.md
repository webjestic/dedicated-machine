Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent failure mode in the heartbeat, a TOCTOU race in the stock check, and test gaps that would leave this brittle under real flash-sale load. None of these are cosmetic; they represent genuine failure paths in a concurrent, high-stakes reservation system.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Outside the Database Transaction

```python
# CURRENT — stock read is OUTSIDE the transaction
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← dirty read possible
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← writes inside
        self.db.create_order(...)
```

The Redis lock serializes access across application nodes, but `get_stock` executes outside the database transaction. If the database itself supports concurrent writers (e.g., a second process bypassing the Redis lock, a background job, a DBA script), the stock value read can be stale by the time `update_stock` commits. The read and write must be atomic at the database level too.

```python
# REQUIRED — read and write inside the same transaction
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

`SELECT FOR UPDATE` (or equivalent) inside the transaction is the correct defense. The Redis lock is a performance optimization to reduce DB contention — it is not a substitute for database-level isolation.

### 2. Silent Heartbeat Failure — Lock Loss Is Undetectable by the Caller

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; caller has NO idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat thread silently exits. The main thread continues executing the database transaction believing it holds the lock — it does not. Another process may have already acquired the lock and begun its own reservation against the same item. The stock decrement will still commit, potentially creating an oversell condition or a corrupted order.

The heartbeat must communicate failure back to the main thread. A threading.Event is the right primitive here:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s — signaling caller", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread must check `lock_lost_event` before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock expired during operation for {lock_key}")
```

Ideally this check happens immediately before the DB transaction commits, not just at yield time.

### 3. Heartbeat Interval Is Too Aggressive Relative to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8   # comment says "TTL/3" but 30/3 = 10, not 8
```

The comment is wrong and the math is slightly off, which is a minor issue on its own. The real concern is that with an 8-second interval and a 30-second TTL, there is a 22-second window after acquisition before the first renewal fires. Under a slow DB operation or GC pause, this is survivable but tighter than it needs to be. TTL/3 (10s) is the conventional Redlock guidance. Fix the constant or fix the comment — having them disagree is a maintenance hazard.

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # self-documenting and always consistent
```

## Moderate Issues

### 4. Unhandled Exceptions Inside `reserve_item` Leak Stack Traces to Callers

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception from `get_stock`, `update_stock`, `create_order`, or the transaction context manager propagates uncaught out of `reserve_item`. The lock context manager's `finally` block will still release the lock (good), but the caller receives a raw exception rather than a `ReservationResult`. The API contract implied by the return type annotation is broken. At minimum, catch broad DB exceptions and return a structured failure result, and log the full traceback:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

---

### 5. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `^[a-zA-Z0-9_\-]+$`, which is good. But if this Redis instance is shared across services or environments (common in cost-constrained deployments), there is no environment or service prefix. A staging deployment can stomp on a production lock. Require a configurable namespace prefix injected at construction time:

```python
def __init__(self, redis_client, db, lock_namespace: str = "inventory"):
    self._lock_namespace = lock_namespace

# usage
lock_key = f"{self._lock_namespace}:lock:{req.item_id}"
```

---

### 6. `t.join(timeout=...)` Failure Is Ignored

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within the join timeout (e.g., it is blocked on a Redis call that is itself hung), `join` returns silently and the code proceeds to release the lock anyway. This is probably the right behavior, but it should be logged:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not exit cleanly key=%s — proceeding with release", lock_key)
    self._release_script(keys=[lock_key], args=[token])
```

## Test Coverage Gaps

### 7. No Test for Heartbeat Lock-Loss Signal

The most important new failure mode introduced by this PR — the heartbeat detecting a lost lock — has zero test coverage. There must be a test that simulates `_extend_script` returning `0` mid-operation and asserts that `reserve_item` returns a failure result rather than committing the transaction.

### 8. `test_lock_release_uses_lua_script` Is Fragile and Incomplete

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test. Assert the actual arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[ANY]  # token is a uuid, use unittest.mock.ANY
)
```

### 9. No Test for DB Exception During Transaction

There is no test asserting behavior when `db.update_stock` or `db.create_order` raises. Given issue #4 above, this is currently an uncaught exception path. The test should be written now to define the expected contract, which will also drive the fix.

### 10. Heartbeat Thread Races in Tests

The tests run synchronously against a mock Redis client, but the heartbeat thread is a real `threading.Thread` that is actually started. In fast unit tests this is usually harmless, but it introduces non-determinism. The heartbeat should be injectable or the thread start should be mockable to keep unit tests fully deterministic.

## Minor / Nitpick

- `EXTEND_LOCK_SCRIPT` uses `expire` (seconds), which is correct for Redis >= 1.0, but `pexpire` (milliseconds) gives finer-grained control if sub-second precision ever matters. Not a blocker.
- `retry_attempts=3` with exponential backoff produces a maximum wait of roughly `0.1 + 0.2 + jitter ≈ 0.35s`. Under genuine flash-sale load this may be too short. Consider making this configurable at construction time rather than a per-call parameter with a default.
- The `quantity` field has `ge=1` but no upper bound. A request for `quantity=10_000_000` is technically valid. Consider a reasonable `le` ceiling.
