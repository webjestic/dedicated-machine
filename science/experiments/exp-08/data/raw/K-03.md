Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, scoped heartbeat with proper stop_event semantics, input validation, and structured logging are all solid. However, there are several correctness, resilience, and security issues that must be addressed before this merges into a government-facing production pipeline.

---

## Issues Requiring Resolution

### 1. TOCTOU Race Between Stock Read and DB Transaction — Critical

```python
# Lock is held, but stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)  # read outside transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
```

The stock read happens outside the database transaction. If the DB supports concurrent readers (and most do), another process could read the same stock value between `get_stock` and `update_stock`, even with the Redis lock held — for example, if any code path reads stock without acquiring the lock first, or if the Redis lock is lost and reacquired by another node mid-operation. The read must occur inside the transaction, and the update should use a conditional decrement (optimistic locking or `WHERE stock >= quantity`) rather than a blind write:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

### 2. Lock Loss During DB Operation Is Silently Ignored — High

The heartbeat thread logs a warning and exits if `_extend_script` returns 0 (lock lost), but the main thread has no way to detect this. The DB transaction proceeds and commits successfully even though the lock is gone — meaning another node may have acquired the lock and is now also writing stock. There is no cancellation path.

```python
# _beat() exits silently on lock loss — main thread never knows
if result == 0:
    return  # lock lost; main thread continues unaware
```

The heartbeat needs a shared signal back to the main thread:

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost_event.set()  # signal main thread
            return
```

The main thread should check `lock_lost_event` before committing, and abort the transaction if the lock was lost.

---

### 3. Unhandled Exceptions Leak State — High

No exception handling wraps the DB operations inside the lock context. If `db.get_stock`, `db.update_stock`, or `db.create_order` raises, the exception propagates out of `reserve_item` entirely — no `ReservationResult` is returned, the caller gets an unhandled exception, and depending on the framework, this may expose stack traces or internal DB structure to the caller.

```python
# If this raises, exception escapes reserve_item entirely
current_stock = self.db.get_stock(req.item_id)
```

Wrap the locked section:

```python
except (DatabaseError, redis.RedisError, Exception) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Reservation failed due to internal error")
```

Be specific with exception types — catching bare `Exception` is acceptable as a last-resort boundary here, but the DB-specific exceptions should be caught first and handled with appropriate rollback guarantees.

---

### 4. Redis Connection Failures Are Not Handled — High

`self.redis.set(...)`, `self._extend_script(...)`, and `self._release_script(...)` can all raise `redis.RedisError` or `redis.ConnectionError`. None of these are caught. A Redis outage during a flash sale would cause unhandled exceptions to propagate to callers rather than a clean degraded response.

---

### 5. Heartbeat Thread Join Timeout Is Too Tight — Medium

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
```

If the heartbeat thread is blocked on a slow Redis call when `stop_event.set()` is called, the join may time out and return before the thread has exited. The lock release then executes while the heartbeat thread may still be attempting an extension — a benign race in most cases, but the thread is daemon=True and will be killed on process exit regardless. The more significant issue: if `join` times out, the code silently continues. Add a log warning if the join times out:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.warning("Heartbeat thread did not exit cleanly for key=%s", lock_key)
```

---

### 6. Test Coverage Gaps — Medium

The test suite covers the happy path and basic validation, but is missing:

- **Lock lost mid-operation**: verify that a lock loss event aborts the transaction (once issue #2 is fixed, this test becomes mandatory)
- **DB exception during transaction**: verify the exception is caught and a `ReservationResult` is returned rather than propagating
- **Redis failure during lock acquisition**: `redis.RedisError` raised from `self.redis.set`
- **Heartbeat extension failure detection**: verify `lock_lost_event` is set when `_extend_script` returns 0
- **Concurrent reservation of the same item**: integration-level test confirming only one reservation succeeds when two requests race

The existing `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()` but does not verify the arguments — it should confirm the correct `lock_key` and `token` are passed.

---

### 7. `quantity` Field Has No Upper Bound — Low / Security

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity. A caller can submit `quantity=999999999` and, if stock is sufficient, reserve the entire warehouse in a single call. Add a domain-appropriate ceiling:

```python
quantity: int = Field(..., ge=1, le=1000)  # tune to business rules
```

---

### 8. Lock Key Namespace Collision Risk — Low

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared with other services (common in cost-constrained government deployments), a key collision with another service using the same prefix is possible. Use a service-specific prefix or configure a Redis key prefix at the client level. Also confirm Redis is not shared with session storage or cache — a misconfigured flush or eviction policy on a shared instance would silently drop locks.

---

## What Is Done Well

- Lua scripts for atomic release and extension are correct and the right approach
- `SET NX EX` is the correct single-command atomic acquisition — no separate TTL set
- Exponential backoff with jitter is properly implemented
- `stop_event.wait(timeout=...)` used as the heartbeat timer rather than `time.sleep` — correct
- Pydantic validation with regex on `item_id` prevents injection via that field
- `daemon=True` on the heartbeat thread is appropriate
- `uuid4` for both token and order ID is correct
- Structured logging with key/value pairs is consistent throughout
