Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and Pydantic validation are all sound. However, there are several correctness and resilience issues that need to be addressed before this is production-ready for a high-contention flash sale environment. The most critical is a TOCTOU race between the stock read and the DB write, and a missing rollback path on DB transaction failure.

---

## Review

### 🔴 Critical

**1. TOCTOU: Stock read is outside the database transaction**

```python
# This read happens BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write is inside
    self.db.create_order(...)
```

The Redis lock serializes access across application instances, but `get_stock` is not inside the `db.transaction()` block. If the DB read uses a different connection or isolation level than the write, another process (a refund, an admin adjustment, a different code path that doesn't respect the Redis lock) can mutate stock between the read and the write. The read must be inside the transaction, and the update should use a conditional write or a `SELECT FOR UPDATE` pattern:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # row-level lock
    if current_stock is None or current_stock < req.quantity:
        # handle and return
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

---

**2. DB transaction failure has no rollback signal and leaks state**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `create_order` raises, the context manager presumably rolls back — but the code outside the `with` block has no exception handler. The exception will propagate up through `_acquire_lock`'s `finally`, the lock will be released, and the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the stated return contract of the method. Wrap the transaction block:

```python
try:
    with self.db.transaction():
        ...
except Exception as e:
    logger.error("Transaction failed item_id=%s order_id=%s: %s", req.item_id, order_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 🟡 Significant

**3. Heartbeat thread failure is silent**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            return  # lock was lost — exits silently
```

When the heartbeat detects `result == 0` (lock expired or stolen), it exits, but the main thread has no way to know this happened. The operation continues under the assumption it holds the lock, but it does not. This is a correctness hazard: the stock write could proceed after another process has acquired the lock and begun its own reservation.

The heartbeat should signal back to the main thread:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat lost lock key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then check `lock_lost_event` before the DB write.

---

**4. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` constraint. A caller can pass `quantity=10_000_000` and the validation passes. For a flash sale context this should have a domain-appropriate ceiling (e.g., `le=100` or a configurable max). This is also a mild DoS vector if the value is echoed into logs or downstream systems.

---

**5. Heartbeat thread join timeout is too generous and masks hangs**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # = 9 seconds
```

If the heartbeat thread is blocked (e.g., Redis is unresponsive), the main thread will wait up to 9 seconds before proceeding to release the lock. The join result is never checked. After join, if the thread is still alive, the Lua release executes anyway — which is correct — but the hang is invisible. Check `t.is_alive()` after join and log a warning.

---

**6. Retry backoff ceiling is very low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0: ~0.10s, attempt=1: ~0.20s, attempt=2: ~0.40s
```

With 3 attempts and a 30-second lock TTL, a caller that hits contention will exhaust retries in under a second and return "service unavailable" even though the lock holder may finish in 1–2 seconds. Either increase `retry_attempts` or add a configurable `max_backoff` cap with a longer window. This is especially important for a flash sale where contention is expected and brief.

### 🟢 Minor / Observations

**7. Test for lock release checks the wrong thing**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is the same mock object returned by `register_script` for both the release and extend scripts. This assertion does not distinguish between a release call and an extend call. Use separate mocks per registered script, or assert on the `keys` and `args` passed to the call.

**8. No test for heartbeat lock-loss path**

The `result == 0` branch in `_beat` is untested. Given that this is the primary safety mechanism against operating without a lock, it warrants an explicit test.

**9. `item_id` appears in log output but is user-supplied**

The pattern validation on `item_id` mitigates injection risk, but it is worth confirming that the logging backend does not interpret format characters. Since structured logging with `%s` is used (not f-strings), this is acceptable — flagging for awareness only.
