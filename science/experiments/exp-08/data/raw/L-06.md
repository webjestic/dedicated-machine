## Code Review

### Critical Defects

**1. TOCTOU Race: Stock Read Outside the Database Transaction**

This is the most serious flaw. The stock check and the write are not atomic at the *database* level:

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    return ReservationResult(...)

new_stock = current_stock - req.quantity
order_id = str(uuid.uuid4())

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The Redis lock serializes *this service's* access, but it does not protect against:

- **Direct database writes** from other services, admin tools, batch jobs, or a second deployment of this service that doesn't use the same lock key scheme.
- **Lock expiry + re-acquisition by another process** between the read and the transaction commit. If the heartbeat fails silently (Redis connection blip, thread scheduling starvation), the lock expires. Another process acquires it, reads the same stock value, and both commit a decrement against the same baseline. Stock goes negative.
- **Database read replicas with replication lag**: if `get_stock` hits a replica and `update_stock` hits the primary, the read can be stale even within the same lock window.

The correct pattern is to move the read *inside* the transaction and use a `SELECT ... FOR UPDATE` (pessimistic lock at the DB layer) or a conditional `UPDATE ... WHERE stock >= quantity RETURNING new_stock` (optimistic). The Redis lock alone is insufficient as the sole concurrency guard for a durable state mutation.

**2. Heartbeat Thread Failure Is Silent and Unrecoverable**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← lock was lost; caller is never notified
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread silently exits. The main thread continues executing the database transaction believing it holds the lock. It does not. Another process can now acquire the lock and begin its own read-check-write cycle *concurrently* with the ongoing transaction. The `finally` block will then call `_release_script` with a token that no longer matches — the release is a no-op (by design of the Lua script), but the damage is done: the lock protecting another process's critical section has been silently abandoned by the thread that lost it.

There is no mechanism to signal the main thread to abort. This needs a shared flag (`lock_lost_event`) that the main thread checks before committing, and ideally raises an exception to abort the transaction.

**3. `t.join(timeout=...)` Does Not Guarantee the Thread Has Exited**

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=N)` returns after N seconds *whether or not the thread has exited*. If the heartbeat thread is blocked on a slow Redis call (e.g., network partition), it can still be alive and attempting to call `_extend_script` *after* `_release_script` has already deleted the key and another process has acquired it with a new token. The extend script would then fail (result == 0) harmlessly, but the thread is a zombie holding a reference to the service. Under sustained Redis latency, this accumulates.

The fix is to check `t.is_alive()` after join and log a warning, or to use a shorter Redis socket timeout so the extend call cannot block longer than the join window.

**4. Lock TTL vs. Heartbeat Interval Is Fragile Under Thread Starvation**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

TTL/3 is a reasonable ratio, but the heartbeat fires *after* `HEARTBEAT_INTERVAL` seconds of sleep. If the Python GIL or OS scheduler starves the heartbeat thread (possible under CPU-bound load or GC pressure), the first beat fires at `8 + ε` seconds. The second at `16 + 2ε`. The third at `24 + 3ε`. With sufficient jitter, the lock can expire before the third extension fires. This is not theoretical on overloaded systems. The interval should be TTL/4 or lower, and the extension should reset to full TTL, not extend by a fixed amount from the current remaining TTL (which the current Lua script does correctly via `expire`).

### Moderate Defects

**5. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. The service will attempt to reserve it. There is no domain-level cap. For a flash sale context, this is a business logic gap that can be exploited to drain all stock in a single request. Add `le=<max_per_order>`.

**6. Exception Handling Gap: Unhandled Exceptions Inside the Lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception from `get_stock`, `update_stock`, `create_order`, or the transaction context manager propagates out of `reserve_item` as an unhandled exception. The `finally` block in `_acquire_lock` will still release the lock (correct), but the caller receives an uncaught exception rather than a `ReservationResult`. The method's return type annotation and docstring imply it always returns a `ReservationResult`. Either catch `Exception` inside the lock and return a failure result, or document the exception contract explicitly.

**7. `order_id` Generated Before Transaction Commits**

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)

return ReservationResult(success=True, order_id=order_id, ...)
```

If the transaction rolls back (DB error), `order_id` is returned in the success result to the caller before the commit is confirmed — but the transaction failure would raise an exception (see defect 6), so this is only dangerous in combination with defect 6 if the exception is swallowed upstream. Still, the `ReservationResult` should only be constructed after a confirmed commit.

### Test Coverage Gaps

**8. No Test for Heartbeat Lock-Loss Scenario**

The most critical runtime failure path — heartbeat detects `result == 0` and exits silently while the main thread continues — has zero test coverage. This is the scenario that causes silent data corruption.

**9. No Test for DB Transaction Rollback**

There is no test verifying behavior when `db.update_stock` or `db.create_order` raises an exception inside the transaction. Given defect 6, this is a gap that masks a real failure mode.

**10. `test_lock_release_uses_lua_script` Tests the Mock, Not the Behavior**

```python
service._release_script.assert_called_once()
```

`_release_script` is the mock returned by `mock.register_script.return_value`. This test asserts that the mock was called, which it will be as long as the code path reaches `finally`. It does not verify that the correct `keys` and `args` were passed. A regression that passes the wrong token would not be caught.

## Decision

**Decision:** Request Changes

**Summary:** The implementation contains a critical TOCTOU vulnerability: the stock read occurs outside the database transaction, meaning the Redis lock does not actually serialize the read-check-write sequence against non-Redis-aware writers or against lock expiry between read and commit. Compounding this, the heartbeat thread silently abandons the lock on extension failure with no mechanism to abort the in-progress database transaction, creating a window for concurrent writes against the same stock baseline. These two defects together can produce negative stock counts under realistic failure conditions and must be resolved — at minimum by moving the stock read inside a `SELECT FOR UPDATE` transaction — before this code is safe to merge into a flash sale path.
