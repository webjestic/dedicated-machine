Decision: Request Changes

Summary: The implementation demonstrates genuine care — atomic Lua scripts, scoped heartbeat state, token-matched release, Pydantic validation, and structured logging are all present and correctly reasoned. However, there is a silent correctness gap in the heartbeat failure path that the code does not handle, a stock-read/write race that the lock does not fully close against the database, and several secondary issues that collectively make this unfit for a high-stakes flash sale workload.

---

## Critical Issues

### 1. Heartbeat Loss Is Silent — The Protected Operation Continues Unaware

This is the most serious flaw. When `_beat()` detects that the lock was lost (`result == 0`), it logs nothing and returns quietly. The caller inside `with self._acquire_lock(lock_key):` has no idea the lock is gone and continues executing — reading stock, writing to the database — under the assumption that it still holds exclusive access.

```python
# _beat(), on loss:
if result == 0:
    return   # ← silent. caller is not notified.
```

A second process can acquire the lock the moment the TTL expires. Both processes now believe they hold the lock. The stock check and the database write proceed concurrently. This is the exact race condition the lock was designed to prevent.

**Required fix:** The heartbeat must signal the caller. The canonical pattern is a shared `threading.Event` (`lock_lost_event`) that `_beat()` sets on `result == 0`, and the protected code checks it at least once before committing — or the context manager raises on exit if the event is set, rolling back any in-flight transaction.

```python
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()
            logger.error("Lock lost during operation key=%s", lock_key)
            return

# In the context manager's finally block:
stop_event.set()
t.join(...)
if lock_lost.is_set():
    raise LockLostError(f"Lock expired mid-operation for {lock_key}")
self._release_script(...)
```

The caller must then handle `LockLostError` and ensure the database transaction is not committed.

### 2. Stock Read Is Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
        self.db.create_order(...)
```

The stock read and the stock write are not atomic at the database level. If the database supports read-committed or higher isolation, another writer that bypasses Redis (a direct DB write, a migration, an admin tool, a future service that doesn't use this lock) can modify stock between the read and the write. The Redis lock serializes callers of *this service*, but it does not protect against anything outside it.

**Required fix:** Move `get_stock` inside the transaction, and use a `SELECT ... FOR UPDATE` or equivalent pessimistic read at the database level. This makes the reservation correct even if the Redis lock is ever bypassed or if a second code path is introduced.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

## Significant Issues

### 3. `t.join()` Timeout Is Silently Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within the timeout, `join()` returns anyway and the lock is released. The thread is still running and will attempt to extend a lock that has just been released — potentially extending a lock now held by a different process with a different token. The Lua script's token check prevents the extension from succeeding in that case, but the thread leaks and the situation is not logged or alarmed.

**Required fix:** Check `t.is_alive()` after `join()` and log an error. Consider whether a leaked thread in this state is acceptable or whether it warrants a harder failure signal.

### 4. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum. A caller can request `quantity=10_000_000`. This passes validation, acquires the lock, reads stock, and writes a zero-stock result. For a flash sale service this is an obvious abuse vector and likely an application logic error that should be caught at the boundary.

**Required fix:** Add `le=` with a domain-appropriate maximum, or at minimum `le=current_stock` enforced post-read.

### 5. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which prevents injection. However, if this Redis instance is shared with other services (common in cost-constrained environments), there is no tenant or service prefix. Two services using the same item ID namespace will collide.

**Required fix:** Add a service-scoped prefix, e.g. `obotix:inventory:lock:{item_id}`, and document the key namespace in a central registry.

## Test Coverage Gaps

### 6. No Test for Heartbeat Loss Path

The most critical failure mode — lock lost mid-operation — has zero test coverage. There is no test that simulates `_extend_script` returning `0` and verifies that the reservation is not committed.

### 7. `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct token. A bug that passes the wrong token (or no token) would pass this test. Assert `called_once_with(keys=[...], args=[expected_token])`.

### 8. No Test for DB Transaction Rollback on Exception

There is no test verifying that if `db.update_stock` or `db.create_order` raises, the transaction is rolled back and the lock is still released. This is a liveness concern: a crash inside the transaction should not leave the lock permanently held (it won't, due to TTL, but the test suite should document the expected behavior).

---

## Minor Issues

### 9. `LOCK_TTL = 30` Is Undocumented as a Tuning Parameter

Thirty seconds is a long TTL for an inventory lock in a flash sale. If the heartbeat thread is killed (process crash, OOM), the lock is held for up to 30 seconds. This should be documented as a deliberate trade-off, and the value should be configurable via constructor parameter or environment variable rather than a class-level constant.

### 10. No Redis Connection Error Handling

`self.redis.set(...)` and both Lua scripts can raise `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`. These propagate as unhandled exceptions from `reserve_item`, bypassing the `LockAcquisitionError` handler and returning a 500 to the caller rather than a graceful degradation response.

---

## What Is Done Well

The token-matched atomic release and extension via Lua are correct. The heartbeat `wait()`-as-timer pattern is correctly implemented — it does not busy-loop. The `stop_event` is set before `join()`, which is the right ordering. Pydantic validation with `pattern=` on `item_id` closes the injection surface at the boundary. Exponential backoff with jitter is appropriate for lock contention. The daemon thread designation is correct. These are not trivial details and they reflect genuine understanding of the problem space.

The implementation is close. The heartbeat signaling gap and the stock-read transaction boundary are blocking issues that must be resolved before this merges.
