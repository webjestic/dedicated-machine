Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, scoped stop events, and Pydantic validation are all present and correctly reasoned about. However, there is a critical gap between when the heartbeat silently detects lock loss and when the protected operation learns about it, meaning a DB write can complete against an invalidated lock with no awareness that another process may have acquired it. Several secondary issues compound this: a stock read outside the DB transaction creates a TOCTOU window, the heartbeat failure is silent by design but has no recovery path, and the test suite validates mechanics without probing the failure modes the architecture was built to prevent.

---

## Detailed Findings

### Critical — Silent Lock Loss During DB Operation

This is the central flaw. When the heartbeat calls `_extend_script` and receives `0`, it logs nothing and returns:

```python
if result == 0:
    return  # heartbeat thread exits silently
```

The protected code in `reserve_item` has no awareness this happened. It continues executing — reads stock, opens a transaction, writes `update_stock`, writes `create_order` — all while the lock it believes it holds has already expired and may have been re-acquired by another process.

The failure mode is not in the acquisition path. It is not in the heartbeat detection path. It lives in the gap between when the heartbeat learns the lock is gone and when the DB write completes against that invalidated state. Two processes can now hold the same logical lock simultaneously, and both will write. This is the exact race condition the distributed lock was designed to prevent.

The heartbeat must have a signaling path back to the main thread. A `threading.Event` for lock loss, checked before and ideally during the DB transaction, is the minimum viable fix. A cleaner approach raises an exception from the heartbeat context that the `finally` block can inspect before releasing.

```python
# Minimal signal path — heartbeat sets this on loss
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during operation key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the protected section, before the DB write:
if lock_lost_event.is_set():
    raise LockLostError(f"Lock expired during operation for {lock_key}")
```

This still has a window — the check is a point-in-time sample, not a continuous guard — but it eliminates the case where loss is detected and completely ignored.

### High — Stock Read Occurs Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction

    # ... stock check ...

    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
        self.db.create_order(...)
```

The stock read and the stock write are not atomic at the DB level. If the DB supports read-committed or higher isolation, another committed write between the `get_stock` call and the `update_stock` call will be invisible to this transaction. The distributed lock serializes callers through Redis, but it does not protect against writes from processes that bypass Redis — migrations, admin tools, other services, or a Redis partition that caused two nodes to believe they hold the lock.

The read must move inside the transaction, and the update should use a conditional write (`UPDATE ... WHERE stock >= quantity AND stock = :expected_stock`) rather than a blind set. This makes the DB the final authority on consistency, not Redis.

---

### Medium — `t.join()` Timeout Allows Heartbeat to Outlive Lock Release

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is blocked on a slow Redis call, `join` times out and the main thread proceeds to release the lock anyway. The heartbeat thread is now running against a key that no longer belongs to this token. The Lua script protects against deleting someone else's lock, but the extend script will silently fail — which is acceptable. The real concern is that `join` returning does not guarantee the thread has stopped; it guarantees the caller stopped waiting. The daemon flag mitigates process-level leakage, but within a long-running service, thread accumulation under sustained Redis latency is a real operational risk. Log a warning when `join` times out.

---

### Medium — Heartbeat Logs Nothing on Lock Loss

When `result == 0` the heartbeat returns without logging. This is a silent failure of a critical coordination primitive. At minimum:

```python
if result == 0:
    logger.error("Heartbeat detected lock loss — token mismatch key=%s", lock_key)
    lock_lost_event.set()
    return
```

Operational visibility into lock loss events is essential for diagnosing flash sale incidents after the fact.

---

### Low — Test Suite Validates Mechanics, Not Failure Modes

The tests confirm the happy path, input rejection, and that the Lua script is called instead of `redis.delete`. They do not test:

- Lock loss mid-operation (heartbeat returns 0 while DB write is in flight)
- `join` timeout behavior
- DB transaction rollback on exception within the `with self.db.transaction()` block
- Behavior when `_extend_script` raises a Redis connection error rather than returning 0

The most important test missing is the one that simulates the critical flaw above: heartbeat fires, gets 0, sets `lock_lost_event`, and the test asserts the DB write was not committed. Without that test, fixing the signaling path has no regression guard.

---

### Low — No Exception Handling Around DB Operations

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

A DB exception here propagates uncaught through `_acquire_lock`'s `finally` block, which will still attempt lock release — that part is correct. But the caller receives an unhandled exception rather than a `ReservationResult`. For a service boundary this should return a structured failure. Wrap in a `try/except` and return `ReservationResult(success=False, message="Internal error")` with the exception logged at ERROR level.

---

## What Is Done Well

The Lua scripts for atomic release and extension are correct. The `stop_event.wait(timeout=N)` pattern as a non-blocking timer is the right mechanism. Token-scoped heartbeat state prevents cross-context interference. Pydantic validation with pattern enforcement on `item_id` is appropriate for a service that constructs Redis keys from user input. Exponential backoff with jitter is correct for contention handling. The `nx=True, ex=TTL` acquisition is atomic. These are not small things — the skeleton is sound.

The lock loss signaling path and the read-inside-transaction requirement are the two changes that must land before this merges.
