Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, scoped heartbeat with stop_event, NX+EX acquisition, and input validation are all sound. However, there are several correctness and resilience issues that need to be addressed before this is safe to merge into a high-concurrency flash sale path. The concerns range from a silent data-integrity gap to missing failure handling that could leave the system in a bad state under load.

---

## Findings

### 1. TOCTOU Between Stock Read and DB Transaction — Critical

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
    self.db.create_order(...)
```

The stock read happens **outside** the database transaction. If the DB layer uses read-committed isolation (the default for Postgres and MySQL), another process that commits between the `get_stock` call and the `update_stock` call will cause a phantom. The Redis lock protects against concurrent Python processes, but it does **not** protect against direct DB writes, migration scripts, admin corrections, or any other writer that bypasses the lock. The read must move inside the transaction, and `update_stock` should use an optimistic or conditional update (e.g., `UPDATE ... WHERE stock = :expected_stock`) so the write fails loudly rather than silently corrupting inventory.

**Recommended fix:**

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    order_id = str(uuid.uuid4())
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. No Exception Handling Around DB Operations — High

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception from `get_stock`, `update_stock`, or `create_order` propagates uncaught out of `reserve_item`. The `finally` block in `_acquire_lock` will still release the lock correctly, but the caller receives an unhandled exception instead of a `ReservationResult`. In a flash sale context under load, a transient DB timeout will surface as a 500 to the user and potentially break the calling pipeline. DB errors must be caught and mapped to a structured result.

**Recommended fix:**

```python
except (DatabaseError, redis.RedisError) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 3. Heartbeat Failure Is Silent — Medium

```python
if result == 0:
    return  # lock was lost — thread exits silently
logger.debug("Lock TTL extended key=%s", lock_key)
```

When the heartbeat detects a lost lock (`result == 0`), it logs nothing and returns. The main thread continues executing the DB transaction against a lock it no longer holds. This is the most dangerous failure mode: another process can now acquire the lock and read stale stock before this thread's transaction commits. At minimum this must log at `ERROR` level. Ideally, the heartbeat should signal the main thread to abort via a shared flag.

**Recommended fix:**

```python
if result == 0:
    logger.error("Heartbeat detected lost lock key=%s — lock expired mid-operation", lock_key)
    lock_lost_event.set()  # signal main thread
    return
```

And in `reserve_item`, check `lock_lost_event` before committing the transaction.

---

### 4. `quantity` Has No Upper Bound — Medium

```python
quantity: int = Field(..., ge=1)
```

There is no maximum. A caller can submit `quantity=10_000_000` and the service will attempt to reserve it. This is a business logic gap that also creates a vector for inventory exhaustion in a single request. Add a domain-appropriate ceiling.

**Recommended fix:**

```python
quantity: int = Field(..., ge=1, le=1000)
```

---

### 5. `item_id` Leaked Into Error Message — Low

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Even though `item_id` is validated against `^[a-zA-Z0-9_\-]+$`, reflecting user-supplied input into API responses is a habit that should be broken at a government contractor. If the validation regex ever relaxes, this becomes a reflected injection vector. Return a generic message; log the item ID server-side.

**Recommended fix:**

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

---

### 6. Test Coverage Gaps — Low

The test suite covers the happy path and basic validation, but is missing:

- **Heartbeat lock-loss path**: no test verifies behavior when `_extend_script` returns 0 mid-operation.
- **DB exception path**: no test verifies that a `DatabaseError` during `update_stock` returns a structured result rather than raising.
- **Retry exhaustion timing**: the backoff test only checks the final result, not that retries were actually attempted.
- **`lock_release_uses_lua_script`**: asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args` were passed, so a call with the wrong token would still pass.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Move stock read inside transaction; use `SELECT FOR UPDATE` or conditional update |
| 2 | High | Catch DB/Redis exceptions in `reserve_item`; return structured result |
| 3 | Medium | Log and signal main thread on heartbeat lock-loss |
| 4 | Medium | Add upper bound to `quantity` field |
| 5 | Low | Remove `item_id` from user-facing error message |
| 6 | Low | Expand test coverage for failure paths |

The locking architecture is solid and the author clearly understands the distributed systems problem. Items 1 and 2 are blocking — they represent real data integrity and reliability risks under the exact conditions this service is designed to handle.
