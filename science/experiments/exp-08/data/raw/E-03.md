Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, token-matched release, and input validation are all present and correctly reasoned. However, there is a critical gap between when the lock is silently lost and when the protected operation learns about it. The heartbeat thread detects expiry and exits quietly, but the business logic in `reserve_item` continues executing against what is now uncoordinated shared state. That gap is the failure mode this entire design exists to prevent, and it is not closed.

---

## Review

### Critical: Silent Lock Loss During Protected Operation

This is the primary concern. Trace the failure path:

1. Lock is acquired. Heartbeat thread starts.
2. Redis becomes unreachable or slow. The `_extend_script` call fails — either by exception or by returning `0` (token mismatch, meaning the lock expired and was re-acquired by another process).
3. The heartbeat thread detects this and returns early — **silently**.
4. The `reserve_item` body continues executing: it reads stock, computes `new_stock`, and commits the database transaction.
5. Another process has now acquired the same lock and is doing the same thing against the same stock record.

The heartbeat correctly detects lock loss. It does nothing with that information. There is no channel back to the main thread. The protected operation has no awareness that the coordination guarantee it was granted no longer exists.

```python
# _beat() on lock loss:
if result == 0:
    return   # ← exits silently. main thread is not notified.
```

```python
# reserve_item continues regardless:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← executes without a valid lock
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is a classic fencing problem. The heartbeat thread and the business logic thread share no cancellation contract. The lock is the coordination primitive, but the code that depends on it cannot observe its loss.

**Required fix:** The heartbeat must signal the main thread on lock loss. A `threading.Event` (`lock_lost_event`) set by `_beat()` on `result == 0` is one approach. The main thread must check this event at meaningful points — before the DB read, before the DB write, and ideally the DB layer should be capable of being interrupted. At minimum, check before committing:

```python
if lock_lost_event.is_set():
    raise LockLostError("Lock expired during operation; aborting transaction")
```

This does not fully close the window — there is still a TOCTOU gap between the check and the commit — but it eliminates the case where the code proceeds with full knowledge that coordination is gone. A fencing token passed into the DB transaction is the stronger solution if the DB layer supports it.

### Significant: Heartbeat Exception Handling

If `_extend_script` raises a network or Redis exception (not just returns `0`), `_beat()` propagates the exception, which kills the thread silently. The lock is now unextended and will expire. The main thread again has no signal.

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
```

No exception handling here. A `redis.RedisError` during extension is operationally equivalent to `result == 0` — the lock is lost — but the code does not treat it that way.

**Required fix:** Wrap the extension call. On exception, treat it as lock loss and signal accordingly. Consider whether a single failed extension should be treated as fatal or whether a short retry window is appropriate before declaring the lock lost.

---

### Significant: `t.join()` Blocks Release Path

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

The join timeout is `HEARTBEAT_INTERVAL + 1` = 9 seconds. If the heartbeat thread is blocked on a slow Redis call during extension, the main thread waits up to 9 seconds before releasing the lock. During that window, the lock is held by a process that has already finished its business logic. Other waiters are blocked.

Additionally, if `t.join()` times out and the thread is still running, `_release_script` executes while the heartbeat thread may still be attempting an extension. These two Redis operations are now racing on the same key with the same token. The Lua scripts are individually atomic, but their sequencing is not guaranteed.

**Required fix:** Reduce join timeout or restructure so the heartbeat thread cannot outlive the release. Verify that a timed-out join is logged and treated as an anomaly, not silently accepted.

### Moderate: `result == 0` Ambiguity in Extension Script

```python
# Atomic lock extension: extend TTL only if token matches; returns 0 if lock was lost
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

`redis.call("expire", ...)` returns `1` on success and `0` if the key does not exist. If the key disappears between the `get` and the `expire` (possible in a Redis Cluster failover or under memory pressure with `maxmemory-policy` set to a volatile eviction policy), the script returns `0` — which the heartbeat correctly interprets as lock loss. This is acceptable behavior, but it should be documented explicitly, because the `0` return from `expire` is semantically different from the `0` returned by the `else` branch, even though the handling is identical.

More importantly: if the Redis instance is configured with any volatile eviction policy and the lock key is set with `ex=`, it is eligible for eviction under memory pressure. This is a deployment constraint that must be documented and enforced. `maxmemory-policy noeviction` or `allkeys-lru` with explicit key-space monitoring should be required for any Redis instance used for locking.

### Moderate: Stock Read Outside Transaction

```python
current_stock = self.db.get_stock(req.item_id)
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens outside the database transaction. The write happens inside. If the DB layer does not hold a row-level lock from the read through the write, another process that bypasses the Redis lock — a direct DB write, a migration, an admin tool — can modify stock between these two points. The Redis lock coordinates application-layer processes, but it does not constrain the database itself.

**Required fix:** The `get_stock` call should occur inside the transaction with a `SELECT FOR UPDATE` or equivalent row-level lock, depending on the DB engine. This closes the gap between application-layer coordination and database-layer consistency.

---

### Minor: Daemon Thread Lifecycle

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed abruptly on interpreter shutdown without running `finally` blocks. If the process exits during a reservation — SIGTERM during a graceful shutdown, for example — the heartbeat dies without the lock being released. The lock will expire naturally after `LOCK_TTL`, which is acceptable, but if the DB transaction committed before shutdown and the lock has not yet been released, the next process to acquire the lock will find the stock already decremented. This is correct behavior, but it means the TTL must be treated as a worst-case unavailability window, and operators should be aware of it.

### Test Coverage Gap

The tests do not exercise the lock-loss-during-operation path. There is no test that simulates the heartbeat detecting `result == 0` and verifies that the main thread aborts the transaction. This is the most important failure mode in the system, and it has no test coverage. Until the signaling mechanism is implemented, this test cannot be written — which is itself evidence that the mechanism is missing.

```python
# No test exists for:
# - heartbeat detects lock loss → main thread aborts DB write
# - heartbeat raises exception → main thread is notified
# - join timeout exceeded → anomaly is logged
```

| Significant | Heartbeat exceptions kill the thread silently; main thread not notified |
| Significant | `t.join()` blocks release for up to 9 seconds; race between join timeout and release script |
| Moderate | Stock read outside DB transaction; row-level lock not held across read-modify-write |
| Moderate | Redis eviction policy is an undocumented deployment constraint for lock correctness |
| Minor | Daemon thread abrupt kill on shutdown is acceptable but must be documented as a known TTL-bounded unavailability window |
