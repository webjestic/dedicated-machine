Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, scoped stop events, Pydantic validation, and structured logging are all present and correctly reasoned about at the surface level. However, there is a critical race condition in the stock-read-to-write path that the distributed lock does not actually close, a silent failure mode when the heartbeat loses the lock mid-operation, and a database transaction boundary that is drawn in the wrong place. These are not style concerns — they are correctness and data integrity failures under realistic concurrent load.

---

## Findings

### 1. CRITICAL — Stock read occurs outside the database transaction

```python
# Lock is held — good.
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # <-- read is here

    # ... stock check ...

    with self.db.transaction():                        # <-- write is here
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer. But `get_stock` runs outside the database transaction. If the database uses any isolation level below SERIALIZABLE — which is the default for PostgreSQL (`READ COMMITTED`), MySQL InnoDB (`REPEATABLE READ` in some configurations), and most ORMs — the stock value read before the transaction opened is not protected by the transaction's snapshot.

The failure mode: a concurrent process that bypasses the Redis lock (a direct DB write, a migration, a different service, a lock that was lost and re-acquired) can modify stock between the `get_stock` call and the `update_stock` call. The transaction then commits a decrement based on a stale read. This is a classic TOCTOU (time-of-check to time-of-use) defect.

**Required fix:** Move `get_stock` inside the `with self.db.transaction()` block, and use a `SELECT FOR UPDATE` or equivalent pessimistic row lock on the stock record so the database itself serializes the read-modify-write. The Redis lock alone is insufficient unless every writer in every system is guaranteed to go through this code path — an assumption that cannot be enforced architecturally.

---

### 2. CRITICAL — Heartbeat lock loss is detected but the protected operation is not interrupted

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return   # <-- heartbeat exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the heartbeat thread knows the lock has been lost — either the TTL expired before the heartbeat fired, or another process has taken the key. The thread exits. But the main thread, which is executing `get_stock`, the stock comparison, and the database transaction, has no awareness of this. It continues to completion and commits.

This is the gap described in the review posture: the failure mode does not live in acquisition or release. It lives between when coordination is lost and when the code learns it. Here, the code never learns it. The heartbeat's knowledge of lock loss is not communicated back to the operation it is protecting.

**Required fix:** The heartbeat must signal the main thread when the lock is lost. A threading.Event (`lock_lost_event`) set by the heartbeat, checked by the main thread at critical points (before and after the DB transaction), is one approach. A more robust approach is to have the heartbeat set a shared flag that the context manager checks in the `finally` block and raises from, preventing the caller from treating a lock-lost execution as successful. The DB transaction should also be structured so that if the lock was lost, the commit does not proceed.

---

### 3. MODERATE — Lock release proceeds even when lock was lost

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The Lua release script is token-matched, so it will correctly return 0 and not delete a key held by another process. This is not a correctness failure in isolation. But the `finally` block has no awareness of whether the operation that just completed was running under a valid lock. If the lock was lost mid-operation and the DB transaction committed anyway (finding 2), the release script returning 0 is logged as a normal release. There is no signal to the caller that the reservation may have been made without valid coordination. The `ReservationResult` returned to the caller will say `success=True`.

---

### 4. LOW — Heartbeat thread join timeout is a best-effort, not a guarantee

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not exit within `HEARTBEAT_INTERVAL + 1` seconds (9 seconds), the join returns and the release script runs anyway. Because the thread is a daemon, it will not block process exit, but it may still be executing `_extend_script` concurrently with the release script. The Lua scripts are individually atomic, but the sequence — extend fires, release fires, extend fires again on the next loop iteration — is not. If the thread is mid-sleep when the join times out, this is benign. If it is mid-extension, the release may race. The `stop_event.set()` before the join should prevent the next loop iteration, but the join timeout means the guarantee is probabilistic, not structural.

---

### 5. LOW — Test coverage does not exercise the failure paths that matter most

The tests mock Redis and the DB at a coarse level. None of the following are tested:

- Heartbeat fires and successfully extends (the thread is never observed)
- Heartbeat detects lock loss (`result == 0`) and the main thread's behavior
- DB transaction rollback path (what does `reserve_item` return if `update_stock` raises?)
- Lock is acquired, DB raises mid-transaction, lock is still released correctly

`test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert the arguments passed to it, so it would pass even if the wrong token or key were used.

---

## What needs to happen before approval

1. Move `get_stock` inside the database transaction and use a row-level lock (`SELECT FOR UPDATE` or equivalent).
2. Implement a signaling mechanism from the heartbeat thread to the main thread so that lock loss aborts the operation before commit.
3. Ensure `ReservationResult(success=True)` is never returned when the lock was lost during the operation.
4. Add tests for the heartbeat-loss path and the DB exception path.
