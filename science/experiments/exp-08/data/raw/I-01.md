Decision: Request Changes

Summary: The core distributed locking architecture is sound — atomic Lua scripts, heartbeat renewal, NX+EX acquisition, and scoped stop events are all correctly implemented. However, there are several correctness, safety, and observability issues that must be addressed before this merges, ranging from a silent data-integrity gap in the stock-read/transaction boundary to a missing exception pathway that could leave locks orphaned in production.

---

## Critical Issues

### 1. Stock Read Outside the Database Transaction — TOCTOU Within the Lock

This is the most important flaw. The Redis lock serializes access correctly, but the stock read and the stock write are not atomic at the *database* level.

```python
# Current — stock read is OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)   # ← read here
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write here
    self.db.create_order(...)
```

If the database uses read-committed isolation (the default in PostgreSQL and MySQL), a concurrent writer that bypasses Redis (a direct DB admin script, a migration, a different service path, a future code path that forgets the lock) can commit between your read and your write. The Redis lock only protects processes that respect it. The database transaction should enforce the invariant independently.

**Required fix:** Move the read inside the transaction and use a pessimistic row lock (`SELECT ... FOR UPDATE`) or an optimistic CAS update (`UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`), then check affected rows.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is a performance optimization (avoiding DB lock contention at scale) — it must not be the *sole* correctness mechanism.

### 2. Unhandled Exceptions Inside the Lock Context Leak No Error Signal, But More Importantly — Broad Exception Classes Are Silently Swallowed

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `db.get_stock`, `db.update_stock`, or `db.create_order` raises *any* exception (network timeout, serialization failure, deadlock), it propagates out of `reserve_item` as an unhandled exception. The `finally` block in `_acquire_lock` will correctly release the lock, which is good — but the caller receives a raw exception rather than a `ReservationResult`. This breaks the contract the method signature implies and will cause unhandled 500s upstream.

**Required fix:** Catch database exceptions explicitly inside `reserve_item` and return a structured `ReservationResult`.

```python
except (DatabaseError, SomeOtherExpectedException) as e:
    logger.error("DB error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

Do not use a bare `except Exception` — be explicit about what you're catching so programming errors surface during testing.

### 3. Heartbeat Thread Silently Absorbs Lock Loss — No Caller Notification

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or evicted). The heartbeat thread exits, but the main thread continues executing the database transaction under the false assumption that it still holds the lock. This is a correctness violation: another process may have acquired the lock and begun its own transaction.

**Required fix:** Signal lock loss back to the main thread. One clean approach is a shared `lock_lost` event.

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s — signaling main thread", lock_key)
            lock_lost.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread (or the `finally` block) should check `lock_lost` and raise or abort accordingly. This is especially important before committing the database transaction.

## Moderate Issues

### 4. `LOCK_TTL = 30s` With `HEARTBEAT_INTERVAL = 8s` Is Reasonable, But the Ratio Should Be Documented and Enforced

The comment says "fires at TTL/3" but `8` is not `30/3`. This is a minor inaccuracy but in a distributed locking context, the ratio matters. If someone changes `LOCK_TTL` without updating `HEARTBEAT_INTERVAL`, the heartbeat could fire *after* expiry. Consider deriving the interval programmatically:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # enforce the invariant in code, not a comment
```

---

### 5. `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` — Join Timeout Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is somehow stuck (blocked on a Redis call that hangs), `join` returns after the timeout and the release script runs anyway — which is correct behavior. However, there is no log or signal that the join timed out. A stuck heartbeat thread is a signal of Redis connectivity problems and should be observable.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible Redis connectivity issue", lock_key)
```

---

### 6. Retry Backoff Ceiling Is Very Low Under High Contention

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45s`. Under a flash sale with thousands of concurrent requests, this means failed requests give up in under half a second. This may be intentional for UX reasons (fail fast), but it should be a documented decision, and `retry_attempts` and the backoff multiplier should be constructor parameters, not magic numbers, so callers can tune them per use case.

## Test Coverage Gaps

### 7. No Test for Heartbeat Lock-Loss Scenario

The most dangerous failure mode — lock lost during DB operation — has zero test coverage. A test should verify that if `_extend_script` returns `0`, the `lock_lost` event (once added) is set and the reservation either aborts or logs an error.

### 8. `test_lock_release_uses_lua_script` Is Fragile

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A future refactor could call the script with wrong arguments and this test would still pass.

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, use ANY or capture it
)
```

### 9. No Test for DB Exception Inside Lock

There is no test verifying behavior when `db.update_stock` or `db.create_order` raises an exception. Given issue #2 above, this is a required test once the exception handling is added.

### 10. No Test for Concurrent Reservation Attempting to Over-Reserve

The test suite has no integration-style test that fires two reservations for the same item with quantity equal to total stock and asserts only one succeeds. This is the primary invariant of the entire service.

## Minor / Nits

- `quantity: int = Field(..., ge=1)` — no upper bound on quantity. A request for `quantity=10_000_000` on an item with 10M stock would succeed. Consider a reasonable `le` ceiling or business-rule validation.
- The `message` field in `ReservationResult` for the "item not found" case leaks the internal `item_id` back to the caller: `f"Item {req.item_id} not found"`. In a government contracting context, enumerating valid item IDs via error messages is an information disclosure risk depending on whether this API is externally exposed.
- `logger.info("Lock released key=%s", lock_key)` fires even when the Lua script returns `0` (lock was already gone). The log message would be misleading. Check the return value of `_release_script` before logging.
