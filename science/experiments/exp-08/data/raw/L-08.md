## Code Review

### Critical Finding 1 — TOCTOU Race on Stock Read (High Severity)

The stock check and the database write are **not atomic with respect to the lock boundary**.

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # (1) read

if current_stock < req.quantity:                  # (2) check
    return ReservationResult(...)

new_stock = current_stock - req.quantity
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # (3) write
    self.db.create_order(...)
```

The Redis lock serializes callers at the Redis layer, which is correct. However, `db.get_stock()` is called **outside** the database transaction. If the database is also accessed by any path that does not go through this Redis lock — a manual adjustment, an admin tool, a different service, a rollback from a prior failed transaction — the stock value read at step (1) can be stale by step (3). The `update_stock` call writes an absolute value (`new_stock`) rather than a relative decrement, which means a concurrent out-of-band write between (1) and (3) is silently overwritten. A correct implementation would use a conditional update:

```sql
UPDATE inventory SET stock = stock - :qty
WHERE item_id = :id AND stock >= :qty
```

and check the affected row count, or perform the read inside the same transaction with a `SELECT FOR UPDATE`. The current design trusts the Redis lock as the sole concurrency gate, which is only safe if **every** stock mutation in the entire system goes through this exact lock path — an assumption that is not enforced and not documented.

### Critical Finding 2 — Heartbeat Thread Leak on Lock Extension Failure (High Severity)

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← exits thread silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has been lost (expired or stolen). The heartbeat thread exits. The calling code in `reserve_item` has **no awareness of this event**. It continues executing the database transaction believing it holds the lock, when in fact another process may have already acquired the lock and begun its own stock read. The `stop_event` is never set by the heartbeat thread on failure, so the `finally` block will still call `t.join()` and `_release_script`, but the release will correctly no-op (token mismatch). The real problem is the window between lock loss and transaction commit — two processes can both commit stock decrements for the same inventory. This is the exact race the lock is meant to prevent.

The heartbeat must have a signaling path back to the main thread. A `threading.Event` for lock loss, checked before the DB transaction, is the minimum fix:

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            return
```

And in `reserve_item`, before committing:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock expired during operation")
```

Even this has a narrow race window; the authoritative fix is Finding 1 (database-level conditional update).

### Critical Finding 3 — Heartbeat Thread Timing Race on `stop_event` (Medium Severity)

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`t.join()` has a timeout of `HEARTBEAT_INTERVAL + 1` = 9 seconds. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), `join()` returns after 9 seconds and `_release_script` is called anyway. This is the correct behavior — the release is atomic and token-matched. However, if the thread is mid-extension when the release fires, the sequence is:

1. Extension script runs → lock TTL reset to 30s, token unchanged
2. Release script runs → token matches → lock deleted ✓

This is safe. But if the thread has not yet started its extension call and the join times out:

1. Release script runs → lock deleted
2. Thread wakes, calls extension → key does not exist → returns 0 → thread exits

Also safe due to the token check. This is the one area the Lua scripts genuinely protect correctness. **No bug here**, but the 9-second blocking `join` in the `finally` block means lock release can be delayed by up to 9 seconds under Redis pressure, holding up all other callers waiting on the lock. The join timeout should be documented or made configurable.

### Finding 4 — `quantity` Field Has No Upper Bound (Medium Severity)

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can request `quantity=2_000_000_000`. This passes validation, proceeds through the lock, reads stock (e.g., 500), fails the stock check, and returns cleanly — but only after acquiring the distributed lock and hitting the database. Under adversarial or buggy input this is a resource amplification vector. A reasonable upper bound (e.g., `le=10_000` or a configurable constant) should be enforced at the model layer.

---

### Finding 5 — Test Suite Does Not Cover the Lock-Loss Scenario (Medium Severity)

The tests mock `register_script` to return a single `MagicMock(return_value=1)` for both the release and extend scripts. There is no test for the case where the extend script returns `0` (lock lost mid-operation). This is the most dangerous failure mode in the system (Finding 2) and it has zero test coverage. The checklist item "heartbeat detects token mismatch (result == 0)" is checked, but the test does not exist.

---

### Finding 6 — Absolute Stock Write Without Optimistic Concurrency (Low Severity, Reinforces Finding 1)

```python
self.db.update_stock(req.item_id, new_stock)
```

Writing an absolute value rather than a delta (`stock = stock - qty WHERE stock >= qty`) means any concurrent modification — even one that is itself correctly locked — that commits between the read and this write will be silently overwritten. This is a direct consequence of Finding 1 but warrants separate callout because it affects the database schema contract: the DB layer has no self-protection and is entirely dependent on the application layer for correctness.

---

### Finding 7 — `item_id` Logged Directly from User Input (Low Severity)

```python
logger.info("Reserved item_id=%s ...", req.item_id, ...)
```

`req.item_id` passes the regex `^[a-zA-Z0-9_\-]+$`, which is sufficient to prevent log injection in most structured logging pipelines. This is acceptable given the validation, but worth noting that the regex should be reviewed if the logging backend ever changes to one that interprets special characters differently.

**Decision:** Request Changes

**Summary:** Two critical correctness flaws block approval: the stock read occurs outside the database transaction with an absolute write, meaning any out-of-band stock mutation silently corrupts inventory regardless of lock correctness; and lock expiry during a DB operation is undetected by the main thread, allowing two processes to commit against the same inventory window. The heartbeat architecture is well-structured and the Lua scripts are correct, but the system's safety guarantees collapse if either the Redis lock expires under load or any code path touches stock outside this lock — both realistic production conditions that must be addressed before this merges.
