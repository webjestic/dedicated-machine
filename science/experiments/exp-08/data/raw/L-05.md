**Critical failure mode class for this domain:** Distributed systems with shared mutable state. The class of defect that is invisible to surface reading is **time-of-check to time-of-use (TOCTOU) races and lock lifecycle violations** — scenarios where state observed under a lock is invalidated by the time it is acted upon, or where the lock itself is silently lost or released at the wrong moment.

**My carried question:** "After I verify that the lock is held and stock is checked correctly, I ask: what temporal or state scenario could silently invalidate the correctness of that check — causing the system to act on stale stock data — while the code appears to hold the invariant?"

## Review

### Critical Flaw 1 — TOCTOU Between Stock Read and Transaction (Architectural)

This is the most serious defect. The stock check and the DB write are **not atomic with respect to the database**. The Redis lock serializes Redis-aware callers, but it does not prevent a concurrent actor — a background job, an admin tool, a different service, a DB-level trigger, or any caller that does not participate in the Redis lock protocol — from modifying stock between `get_stock()` and `update_stock()`.

```python
# Line A — stock read
current_stock = self.db.get_stock(req.item_id)

# ... time passes, heartbeat runs, other non-lock-aware writers can act here ...

# Line B — write happens based on stale read
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # new_stock computed from stale value
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The correct pattern is to perform a **conditional update inside the transaction** — either a `SELECT FOR UPDATE` (pessimistic) or a CAS update (`UPDATE inventory SET stock = stock - qty WHERE item_id = ? AND stock >= qty`) and check affected rows. The Redis lock alone cannot substitute for database-level atomicity. This is a silent correctness failure: the code looks correct, passes all tests, but can oversell inventory under real-world conditions.

### Critical Flaw 2 — Heartbeat Failure Is Silently Ignored (Data Integrity)

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return  # ← lock was lost; thread exits silently
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has expired and been lost — potentially acquired by another process. The heartbeat thread exits, but **the main thread continues executing the database transaction under the assumption it holds the lock**. There is no signaling mechanism back to the main thread. The stop event is not set. The main thread will complete the write, commit the transaction, and return success — all while another process may have acquired the lock and is performing its own write.

This is the exact scenario the lock exists to prevent. The fix requires a shared flag or event that the main thread checks before committing, or that the heartbeat sets to abort the operation.

### Critical Flaw 3 — `t.join()` Does Not Block Lock Release Reliably

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call, `join` times out and returns `False`, and the release script runs anyway. The thread is now a zombie that may call `_extend_script` after the lock has been released and potentially re-acquired by another holder — extending a lock it no longer owns. The return value of `join` is never checked.

### Critical Flaw 4 — `quantity` Field Has No Upper Bound (DoS / Business Logic)

```python
quantity: int = Field(..., ge=1)
```

There is no upper bound on `quantity`. A caller can request `quantity=2_000_000_000`. This passes validation, proceeds through lock acquisition, and executes a DB write. For a flash sale system at a government contractor, unbounded quantity is both a business logic flaw and a potential abuse vector.

### Flaw 5 — Test Suite Does Not Cover the Failure Modes That Matter

The tests mock the lock as always-succeeding and the DB as always-returning clean values. None of the following are tested:

- Heartbeat loss during a transaction (Flaw 2)
- DB transaction failure after lock acquisition (what happens to the lock? it is released correctly, but stock state should be verified)
- Non-lock-aware concurrent writer invalidating stock between read and write (Flaw 1)
- `t.join()` timeout scenario (Flaw 3)
- `quantity` at `INT_MAX` (Flaw 4)

`test_lock_release_uses_lua_script` asserts that `_release_script` is called once, but because both `_release_script` and `_extend_script` are the same `script_mock` object returned by `register_script`, this assertion is not actually verifying what it claims.

```python
script_mock = MagicMock(return_value=1)
mock.register_script.return_value = script_mock  # same object for both scripts
```

Both `self._release_script` and `self._extend_script` point to the same mock. The test cannot distinguish between them.

### Minor Issues

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` means the first heartbeat fires at 8s, leaving a 22s window before the first extension. For a 30s TTL this is acceptable, but the comment says "TTL/3" — 30/3 is 10, not 8. The comment is misleading.
- No Redis connection error handling. A `redis.exceptions.ConnectionError` during lock acquisition or the heartbeat will propagate uncaught or silently kill the heartbeat thread.
- The `_beat` thread is `daemon=True`, which is correct for preventing process hang, but combined with Flaw 3, a daemon thread extending a foreign lock is worse than a non-daemon thread that leaks.

---

## Decision

**Decision:** Request Changes

**Summary:** The implementation contains two architectural correctness failures that can silently produce oversold inventory under real load: the stock read and DB write are not atomic at the database level (a non-lock-aware writer can invalidate the stock check between read and commit), and heartbeat failure is not propagated to the main thread, allowing a transaction to commit after the distributed lock has been lost. A third flaw — `t.join()` timeout not being checked — can cause the heartbeat thread to extend a lock after it has been released and re-acquired by another holder. These flaws will not surface in the current test suite and require architectural changes before this can be approved.
