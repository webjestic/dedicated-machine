## Code Review

### Critical Defect — TOCTOU Between Lock Scope and Database Transaction

This is the most serious flaw in the submission. The stock check (`db.get_stock`) occurs **inside the lock**, which is correct. However, the database transaction is a **separate operation** that begins *after* the check. The lock protects the Redis-level mutual exclusion, but it does **not** guarantee that the database itself enforces the same invariant.

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # READ outside transaction

if current_stock < req.quantity:                  # CHECK outside transaction
    return ReservationResult(...)

with self.db.transaction():                       # WRITE inside transaction
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The stock read happens **outside the database transaction**. This means:

1. The Redis lock serializes callers *through this service*, but any other service, admin tool, migration job, or direct DB writer that does **not** acquire this Redis lock can modify stock between the `get_stock` call and the `update_stock` call.
2. Even within this service, if the Redis lock is ever lost and re-acquired by another instance (e.g., due to a Redis failover, clock skew, or a network partition causing the heartbeat to miss), the stock read is stale and the write proceeds on bad data.
3. The correct pattern is to perform the read **inside** the transaction and use a database-level pessimistic lock (`SELECT FOR UPDATE`) or an optimistic concurrency check (compare-and-swap on a version column) so the database itself rejects the write if the invariant was violated.

The fix requires moving `get_stock` inside the transaction and using a locking read:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # rollback and return early
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is still valuable as a first line of defense to reduce DB contention, but the database transaction must be the authoritative correctness boundary.

### Significant Defect — Heartbeat Thread Silently Swallows Lock Loss

When the heartbeat detects that the lock was lost (Lua script returns 0), it simply returns:

```python
if result == 0:
    return   # ← silent exit
logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread continues executing the database transaction with no awareness that the lock is gone. Another process may have already acquired the lock and begun its own read-check-write cycle. The two transactions now race with no mutual exclusion. The heartbeat thread should set a shared flag that the main thread checks before and ideally during the transaction, or it should raise into the main thread via a mechanism like `threading.excepthook` or a shared `Event`.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost_event.set()  # signal main thread
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

The main thread must check `lock_lost_event` before committing.

### Significant Defect — `t.join()` Called Before `_release_script` With No Error Handling on Release

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If `t.join()` times out (the heartbeat thread is hung, e.g., on a blocked Redis call), execution falls through and the release script is still called — but the heartbeat thread may still be alive and attempting to extend the lock concurrently with the release. This is a race between the release Lua script and the extend Lua script. The token match in both scripts makes this *mostly* safe, but the thread is now a zombie holding a Redis connection.

Additionally, if `_release_script` itself raises (Redis connection failure), the exception propagates out of `finally` and masks any exception from the `try` block. The release should be wrapped:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not terminate key=%s", lock_key)
    try:
        self._release_script(keys=[lock_key], args=[token])
    except Exception:
        logger.exception("Failed to release lock key=%s", lock_key)
```

### Moderate Defect — `new_stock` Computed Before Transaction, Passed as Absolute Value

```python
new_stock = current_stock - req.quantity
# ...
self.db.update_stock(req.item_id, new_stock)
```

`update_stock` appears to take an absolute value rather than a delta. If the DB layer uses `SET stock = new_stock` rather than `SET stock = stock - quantity WHERE stock >= quantity`, then any concurrent write between the read and the write (from outside this lock) will be silently overwritten. This is the same TOCTOU concern from the first defect, but it also signals an API design risk: the `update_stock` interface does not enforce atomicity at the DB level.

---

### Moderate Defect — No Timeout on Database Operations

The heartbeat is designed to keep the lock alive during slow DB operations, but there is no timeout on `db.get_stock`, `db.update_stock`, or `db.create_order`. A hung database call will hold the lock indefinitely (heartbeat keeps renewing), blocking all other reservations for that item. A maximum operation timeout should be enforced, and the lock should be released (and the transaction rolled back) if it is exceeded.

---

### Minor Defect — Retry Backoff Is Insufficient for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum wait before giving up is approximately `0.4 + 0.1 + 0.05 = ~0.55 seconds`. Under flash sale load with hundreds of concurrent requests, nearly all of them will exhaust retries and return "service unavailable" within one second. This is likely acceptable as a circuit breaker, but the jitter range (`0–0.05s`) is too narrow to meaningfully spread thundering herd retries. The jitter should scale with the backoff window, e.g., `random.uniform(0, 2 ** attempt * 0.1)`.

---

### Minor Defect — Test Coverage Does Not Cover the Defects That Matter

The tests mock `db.transaction()` as a context manager but never verify that `get_stock` is called **inside** the transaction. The tests cannot detect the TOCTOU defect because they mock at the wrong granularity. There are no tests for:

- Lock lost mid-operation (heartbeat returns 0 while main thread is in transaction)
- `_release_script` raising an exception
- `update_stock` raising inside the transaction (rollback path)
- Concurrent reservation of the same item from two service instances

The checklist item "Tests pass" is true but misleading — the tests validate the happy path and input validation only.

## Decision: Request Changes

**Summary:** The implementation contains a critical TOCTOU defect: the stock read occurs outside the database transaction, meaning the Redis lock provides no protection against writers that bypass it, and the stock check is not atomically coupled to the write. The heartbeat thread silently discards lock-loss events without signaling the main thread, allowing a transaction to commit after mutual exclusion has been lost. These two defects together mean the distributed lock provides a false sense of safety — the system can produce oversold inventory under realistic failure conditions. The remaining issues (exception masking in `finally`, absolute-value stock update API, missing DB timeouts) must also be addressed before this is suitable for a production flash sale workload.
