Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, scoped heartbeat, NX+EX acquisition, and input validation are all sound. However, there are several correctness and reliability issues that need to be resolved before this merges: a TOCTOU race between the stock read and the DB transaction, silent swallowing of DB exceptions inside the lock context, and a heartbeat thread that logs nothing and returns silently when it loses the lock mid-operation. The test suite also has meaningful gaps that would let regressions through undetected.

---

## Detailed Findings

### 1. TOCTOU Race — Stock Read Outside the Transaction (High)

```python
# Current — read is outside the transaction boundary
current_stock = self.db.get_stock(req.item_id)   # ← dirty read possible here

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers *through this service*, but it does not protect against:

- A separate process or service writing stock directly to the DB between the read and the transaction commit.
- A DB replica lag if `get_stock` hits a read replica.
- Any future code path that bypasses this service entirely.

The stock read and the conditional decrement must be inside the same transaction, and the update should use an optimistic or pessimistic row-level lock at the DB layer — not just a Redis lock.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. DB Exceptions Silently Escape the Lock Context (High)

```python
with self._acquire_lock(lock_key):
    # ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
    # If either raises, the exception propagates out of _acquire_lock
    # The finally block releases the lock — that part is fine —
    # but the caller receives an unhandled exception, not a ReservationResult
```

Any `DatabaseError`, `OperationalError`, or timeout raised inside the lock context will bubble up as an unhandled exception. The caller (likely an HTTP handler) will either crash or return a 500 with no structured response. This should be caught explicitly:

```python
except LockAcquisitionError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 3. Heartbeat Loses the Lock Silently (Medium)

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was lost; main thread has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat exits silently. The main thread continues executing the DB transaction believing it holds the lock — but it does not. Another process may now acquire the lock and begin its own reservation against the same item concurrently, defeating the entire purpose of the lock.

The heartbeat needs a way to signal the main thread that the lock was lost so the operation can be aborted:

```python
# Option: shared flag checked at yield point, or a threading.Event
# At minimum, log at ERROR level so the condition is observable
if result == 0:
    logger.error("Heartbeat lost lock key=%s — lock expired under load", lock_key)
    lock_lost_event.set()
    return
```

And the main thread should check `lock_lost_event` before committing the transaction.

### 4. `quantity` Has No Upper Bound (Medium)

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can pass `quantity=2_000_000_000`. This will not cause a crash here, but it will produce a nonsensical reservation and may cause integer overflow or unexpected behavior in downstream systems. Add a reasonable ceiling:

```python
quantity: int = Field(..., ge=1, le=10_000)
```

The exact ceiling is a business decision, but the absence of any upper bound is a validation gap.

---

### 5. `item_id` Leaked Into Log Output and Error Messages (Low-Medium)

```python
return ReservationResult(success=False, message=f"Item {req.item_id} not found")
```

`item_id` is user-supplied input. Even though it passes the regex, embedding it directly into user-facing messages and structured logs without a secondary review is a habit worth breaking in a government contracting context. Prefer a fixed message for the user and log the ID separately at the appropriate level:

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

### 6. Test Coverage Gaps (Medium)

The existing tests are a reasonable starting point but miss the following cases:

| Missing Case | Risk |
|---|---|
| DB exception raised inside the lock | Verifies exception handling path added in finding #2 |
| Heartbeat lock-loss path (`extend_script` returns 0) | The silent failure in finding #3 is untested |
| `quantity` above any future upper bound | Validates finding #4 |
| Partial transaction failure (order created, stock not updated) | Verifies rollback behavior |
| Lock retry exhaustion with partial successes (e.g., 2 of 3 fail) | Verifies backoff logic |

`test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert the arguments passed to it. A regression that calls the script with the wrong token would not be caught.

---

### 7. Minor: Heartbeat Thread Join Timeout Is Fragile (Low)

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), the join will time out and the main thread will proceed to call `_release_script` while the heartbeat thread may still be executing `_extend_script`. This is unlikely but worth noting — a shorter, configurable join timeout with a logged warning on timeout would be more observable:

```python
t.join(timeout=2)
if t.is_alive():
    logger.warning("Heartbeat thread did not exit cleanly for key=%s", lock_key)
```

---

## What Is Working Well

- Atomic `SET NX EX` acquisition is correct.
- Lua scripts for release and extension are correct — token comparison prevents foreign lock deletion.
- `stop_event.wait(timeout=...)` as a timer is the right pattern; avoids busy-wait.
- Daemon thread prevents process hang on unclean exit.
- Pydantic validation with regex on `item_id` is appropriate.
- Exponential backoff with jitter is correctly implemented.
- `LockAcquisitionError` is a clean, typed exception.

The architecture is sound. The issues above are fixable without a redesign. Resolve findings 1–3 before merge; the rest can be addressed concurrently or in a fast-follow.
