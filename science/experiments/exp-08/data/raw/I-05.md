Decision: Request Changes

Summary: The architecture is sound in its broad strokes — atomic Lua scripts for release/extension, heartbeat scoped to the lock context, Pydantic validation, and transactional DB writes are all correct choices. However, there are several correctness defects, a silent data-integrity hole, and test gaps that must be addressed before this merges into anything load-bearing.

---

## Detailed Findings

### 🔴 Critical — Data Integrity

**1. Stock read occurs outside the database transaction**

```python
# CURRENT — race-prone even under the Redis lock
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside tx
        self.db.create_order(...)
```

The Redis lock serializes callers *through this service*, but it provides zero protection against:

- A direct DB write from another service, a migration script, an admin tool, or a second deployment of this service that doesn't participate in the same Redis lock namespace.
- A crash between the read and the write leaving the stock in an inconsistent state.

The read must happen **inside the same database transaction** with a pessimistic row-level lock (e.g., `SELECT ... FOR UPDATE` in Postgres/MySQL). The correct pattern is:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None: ...
    if current_stock < req.quantity: ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

The Redis lock is then a *performance optimisation* (avoiding DB lock contention under hot paths), not the sole correctness mechanism. Right now it is being used as the sole correctness mechanism, which is insufficient.

### 🔴 Critical — Correctness

**2. Heartbeat thread silently swallows lock loss; caller is never notified**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock was stolen/expired; thread exits, caller never knows
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat thread exits quietly. The `yield token` in the context manager continues executing as if the lock is still held. The caller will then write to the database under the false belief that it holds the lock, producing a split-brain write. At minimum, a `threading.Event` should signal lock loss back to the caller, and the caller must check it before and after the DB transaction:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost_event.set()
            return
        ...

try:
    yield token
    if lock_lost_event.is_set():
        raise LockAcquisitionError("Lock lost during operation")
finally:
    ...
```

Even this is a best-effort check. The truly safe answer is the `SELECT FOR UPDATE` from finding #1, which makes the DB the authority.

### 🟠 High — Reliability

**3. `t.join()` timeout is silently ignored**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`threading.Thread.join(timeout=...)` does **not** raise on timeout — it returns silently whether or not the thread actually stopped. If the heartbeat thread is blocked (e.g., on a hung Redis call), the release script executes while the thread may still be mid-extension. The thread's state must be checked:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not release — the lock state is unknown. Alert and let TTL expire.
```

---

### 🟠 High — Reliability

**4. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is dangerously tight for slow DB operations**

The comment says "fires at TTL/3" (30/3 = 10), but 8 ≠ 10. More importantly, 30 seconds is a very short TTL for a service that wraps an arbitrary DB transaction. If the database is under load and a transaction takes >30 seconds (not unusual during a flash sale), the lock expires before the heartbeat fires a second time. The TTL and heartbeat interval should be configurable and documented with explicit reasoning, not hardcoded magic numbers.

---

### 🟠 High — Reliability

**5. Exponential backoff ceiling is missing**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3` the maximum backoff is `(2**2) * 0.1 + 0.05 = 0.45s`, which is fine today. But `retry_attempts` is a parameter — a caller passing `retry_attempts=20` produces a backoff of `~104 seconds + jitter` on the final attempt with no cap. Add a ceiling:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), MAX_BACKOFF)
```

### 🟡 Medium — Correctness

**6. `EXTEND_LOCK_SCRIPT` uses `expire` (relative), not `expireat` (absolute)**

```lua
return redis.call("expire", KEYS[1], ARGV[2])
```

`EXPIRE` resets the TTL to a fixed duration from *now*. If the heartbeat fires late (GC pause, thread scheduling delay, Redis latency spike), the effective lock window shrinks. Consider `PEXPIREAT` with an absolute deadline calculated at acquisition time, or at minimum document that the heartbeat must fire well within the TTL window.

---

### 🟡 Medium — Observability

**7. Lock loss in the heartbeat thread is only logged at `debug` level — it should be `error` or `critical`**

```python
if result == 0:
    return  # no log at all
```

There is actually *no log statement* on lock loss. A silent exit from the heartbeat thread during a live reservation is a critical operational event. It must be logged at `ERROR` or higher with the `lock_key` and `token` so operators can correlate it with any data anomalies.

---

### 🟡 Medium — Test Coverage

**8. The heartbeat thread is never exercised in tests**

All tests mock `redis.set` to return `True` immediately and `register_script` to return a mock that returns `1`. The heartbeat thread starts, waits 8 seconds, and is then stopped by `stop_event.set()` — but no test verifies:

- That the heartbeat actually calls `_extend_script` when the interval elapses.
- That lock loss (`result == 0`) is handled correctly.
- That the thread is joined before release.

These are the most failure-prone paths in the entire implementation and they have zero test coverage.

---

### 🟡 Medium — Test Correctness

**9. `test_lock_release_uses_lua_script` is a vacuous assertion**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is the *same mock object* returned by `register_script`. Because the fixture returns the same mock for both `_release_script` and `_extend_script`, this assertion does not distinguish between a release call and an extension call. The mock setup needs to return distinct objects per `register_script` call.

---

### 🟢 Minor — Style / Hygiene

**10. `yield token` — the token is yielded but never used by the caller**

```python
with self._acquire_lock(lock_key):   # token discarded
    ...
```

Either remove the `yield token` and yield `None`, or use `with self._acquire_lock(lock_key) as token:` in the caller. The current mismatch between what is yielded and what is consumed is a readability smell.
