Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent data-integrity gap, and test coverage holes that would be unacceptable in a high-contention flash sale context where every race condition translates directly to overselling or deadlock. The issues range from a read-outside-transaction stock check that undermines the entire locking strategy, to a heartbeat that silently swallows extension failures, to a thread leak on DB exceptions.

---

## Detailed Findings

### 🔴 Critical

---

**1. Stock read occurs outside the database transaction — lock is doing less work than it appears**

```python
# services/inventory.py  lines ~90–105
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # ← READ IS HERE

    ...

    with self.db.transaction():                        # ← WRITE IS HERE
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers correctly, but `get_stock` is issued before the DB transaction opens. If the underlying DB uses `READ COMMITTED` isolation (the default for PostgreSQL and MySQL), a concurrent writer that bypasses the Redis lock — a background job, a manual admin correction, a different service — can commit a stock change between the `get_stock` call and the `update_stock` call. The read and the write are not atomic at the DB layer.

**Required fix:** Move `get_stock` inside the transaction boundary, and either use `SELECT ... FOR UPDATE` (pessimistic) or an optimistic CAS update (`UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`) with a rowcount check. The Redis lock alone is not sufficient if any writer can touch the DB outside this code path.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Heartbeat thread is leaked on any exception thrown by `db.get_stock`, `db.transaction`, or `db.update_stock`**

```python
        try:
            yield token
        finally:
            stop_event.set()
            t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← 9 second block on exception path
            self._release_script(keys=[lock_key], args=[token])
```

The `finally` block does run, so the lock is eventually released — that part is correct. The problem is `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)`. On an exception path, the caller's stack is frozen for up to **9 seconds** waiting for the heartbeat thread to notice the stop event. Under high concurrency this turns a DB timeout into a thundering-herd stall. The join timeout should be a small multiple of the event check granularity, not the full heartbeat interval.

Additionally, if `t.join` times out silently (the thread is stuck in a Redis call), `_release_script` still fires — but the thread is still running and holds a reference to `token`. If the thread wakes up after the lock has been released and re-acquired by another caller, `_extend_script` will correctly return 0 (token mismatch), so there is no correctness violation here, but the zombie thread is a resource leak that accumulates under sustained error conditions.

**Required fix:**

```python
finally:
    stop_event.set()
    t.join(timeout=2)          # short — event check is near-instant once set
    if t.is_alive():
        logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
    self._release_script(keys=[lock_key], args=[token])
```

### 🟠 Significant

---

**3. Heartbeat failure is silent — lock expiry goes undetected and the operation continues**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            return          # ← thread exits, but reserve_item() has no idea
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0` the heartbeat thread exits quietly. The main thread continues executing `db.update_stock` and `db.create_order` without a lock. Another caller may have already acquired the lock and read the same stock value. This is precisely the race condition the lock was designed to prevent.

The heartbeat needs a signaling path back to the main thread. A simple approach is a shared `threading.Event` for lock loss:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the main thread, after yield, before DB work:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock expired mid-operation for {lock_key}")
```

This requires checking `lock_lost_event` at meaningful points inside `reserve_item` — at minimum before opening the DB transaction.

**4. `EXTEND_LOCK_SCRIPT` uses `expire` (relative TTL) which is correct, but the argument is passed as a raw string without bounds validation**

```python
result = self._extend_script(
    keys=[lock_key],
    args=[token, str(self.LOCK_TTL)]
)
```

`self.LOCK_TTL` is a class-level constant, so in practice this is fine. But the Lua script accepts `ARGV[2]` as an arbitrary string and passes it directly to `redis.call("expire", ...)`. If `LOCK_TTL` were ever sourced from configuration or an environment variable without validation, a non-integer or negative value would cause a Redis error that propagates as an exception inside the daemon thread — unhandled, silently swallowed. Add a type assertion or clamp at the call site.

---

**5. Exponential backoff ceiling is missing — retry delay is effectively unbounded for large `retry_attempts`**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3` the maximum backoff is `(2^2)*0.1 + 0.05 = 0.45s` — acceptable. But `retry_attempts` is a parameter with no upper bound, and the formula is uncapped. A caller passing `retry_attempts=20` produces a 52-second sleep on the last attempt. Add a cap:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

### 🟡 Minor / Housekeeping

---

**6. Test for lock release uses `assert_called_once()` on the shared script mock — the assertion is fragile**

```python
# tests/test_inventory.py
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is the same mock object returned by `register_script` for **both** `RELEASE_LOCK_SCRIPT` and `EXTEND_LOCK_SCRIPT` because `mock.register_script.return_value` is a single mock. This means `_release_script` and `_extend_script` are the same object. The assertion `assert_called_once()` will fail if the heartbeat fires even once during the test, and it conflates release calls with extension calls. Use `side_effect` or `return_value` per-script, and assert on `call_args` to verify the correct keys/args were passed.

```python
release_mock = MagicMock(return_value=1)
extend_mock  = MagicMock(return_value=1)
mock.register_script.side_effect = [release_mock, extend_mock]
```

---

**7. No test for the heartbeat extension path or lock-lost signaling**

The heartbeat is the most operationally critical and least-tested component. There is no test that:
- Verifies the extend script is called when the operation takes longer than `HEARTBEAT_INTERVAL`
- Verifies that a lock-lost signal (extend returns 0) causes the reservation to fail rather than silently succeed

These are not edge cases in a flash sale — they are the primary failure mode under load.

---

**8. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. This passes validation and will attempt to decrement stock by ten million. Add a reasonable `le=` constraint appropriate to the business domain.
