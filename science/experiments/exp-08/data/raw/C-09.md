Decision: Request Changes

Summary: The implementation demonstrates genuine architectural awareness — atomic Lua scripts, scoped heartbeat state, token-matched release, and validated input are all present and correctly reasoned about. However, there is a silent correctness gap in the stock-read path that can produce phantom reservations under concurrent load, and the heartbeat failure mode is detected but not surfaced to the caller. These are not style concerns; they are correctness and reliability failures that would manifest in production under the exact conditions this service is designed to handle.

---

## Critical Issues

### 1. Stock Read Is Outside the Database Transaction — Race Condition Survives the Lock

This is the primary defect. The Redis lock serializes callers correctly. The problem is what happens *inside* the lock.

```python
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # ← read outside transaction

    if current_stock < req.quantity:
        return ReservationResult(...)

    new_stock = current_stock - req.quantity

    with self.db.transaction():                        # ← write inside transaction
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(...)
```

The read and the write are in different transactional scopes. Between `get_stock` and `update_stock`, the database is not under a consistency guarantee. If the Redis lock is ever lost and re-acquired by another process — or if a non-Redis-aware writer touches the stock table — the read value is stale when the write executes. The stock check passed on a number that is no longer the committed value at write time.

The correct pattern is to pull `get_stock` inside the transaction and use a database-level conditional update or `SELECT FOR UPDATE` so the read and write are atomic at the database layer. The Redis lock handles distributed coordination; the database transaction must handle read-modify-write atomicity.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

Without this, the Redis lock is providing coordination theater while the actual invariant — stock never goes negative — is unenforceable.

### 2. Heartbeat Loss Is Detected but Silently Discarded

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← lock lost; thread exits; caller is never told
        logger.debug("Lock TTL extended key=%s", lock_key)
```

When `result == 0`, the lock has expired and been claimed by another process. The heartbeat thread exits. The calling code in `reserve_item` continues executing — reading stock, writing to the database — with no knowledge that the lock it believes it holds is gone. The protected region is no longer protected.

The checklist states "heartbeat detects token mismatch (result == 0)" as complete. Detection is not handling. The caller must be informed.

The standard resolution is a shared flag:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()   # signal the main thread
            return
```

And the protected body must check it at meaningful points, or the context manager must raise on exit if the flag is set. Silently continuing is the failure mode the heartbeat was designed to prevent.

## Moderate Issues

### 3. `t.join(timeout=...)` Does Not Verify the Thread Actually Stopped

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns regardless of whether the thread exited. If the heartbeat thread is blocked on a slow Redis call, the release script executes while the heartbeat thread may still be attempting an extension. This is a narrow window but it is a real one. The thread's liveness should be asserted:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
    # decide: raise, or proceed with release and accept the risk
```

---

### 4. `db` Abstraction Has No Specified Contract

The `db` parameter is untyped and undocumented. `get_stock`, `update_stock`, `create_order`, and `transaction` are called without any interface definition. This means:

- There is no enforcement that `transaction()` is actually a context manager.
- There is no enforcement that `get_stock` participates in the transaction if moved inside it.
- Mocking in tests passes regardless of whether the real implementation satisfies these assumptions.

A `Protocol` or abstract base class should define the required interface. This is not pedantry — the correctness of issue #1's fix depends entirely on whether the real `db.get_stock` can be called inside a `db.transaction()` scope and whether it will use the same connection.

---

### 5. Test Coverage Does Not Exercise the Failure Paths That Matter

The tests verify the happy path and the lock-not-acquired path. They do not test:

- Heartbeat loss mid-operation (`_extend_script` returns 0 while the body is executing).
- Database transaction failure after lock acquisition — does the lock still release?
- `get_stock` raising an exception — same question.
- Concurrent callers with real timing, even with `threading` in tests.

The `test_lock_release_uses_lua_script` test asserts `_release_script.assert_called_once()` but does not verify the arguments — it would pass even if the wrong token or key were passed to the script.

## Minor Notes

- The `LOCK_TTL = 30` / `HEARTBEAT_INTERVAL = 8` ratio is reasonable. No objection there.
- Exponential backoff with jitter is correctly implemented.
- Lua scripts for atomic release and extension are correctly structured.
- Input validation pattern is sound.
- The `daemon=True` on the heartbeat thread is appropriate — it will not block process shutdown.

---

## Required Before Approval

1. Move `get_stock` inside the database transaction and use a row-level lock (`SELECT FOR UPDATE` or equivalent) so the read-modify-write is atomic at the database layer.
2. Propagate heartbeat loss to the calling context — either via a shared event checked in the body, or by having the context manager raise if the lock was lost before release.
3. Assert thread liveness after `join`.
4. Define a typed `Protocol` for the `db` dependency.
5. Add tests for heartbeat loss and database failure paths.
