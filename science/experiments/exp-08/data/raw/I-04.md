Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped stop events are all directionally correct. However, there are several correctness flaws, a silent data-integrity gap, and test coverage holes that collectively make this unsafe to merge into a production flash-sale path. The issues range from a race condition that can corrupt inventory counts to a heartbeat that silently swallows lock-loss without surfacing it to the caller.

---

## Critical Issues

### 1. Stock Read Occurs Outside the Database Transaction — TOCTOU Race

This is the most serious flaw. The distributed lock serializes access correctly, but the stock check and the write are not atomic at the *database* level.

```python
# Current — read is OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)   # dirty read possible

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write is inside
    self.db.create_order(...)
```

If the database read uses `READ COMMITTED` isolation (the default in PostgreSQL and MySQL), another transaction that commits between the `get_stock` call and the `update_stock` call will produce a stale read. The Redis lock only protects against *this service's* concurrent callers. It does **not** protect against:

- Direct DB writes from another service or admin tool
- A second instance of this service running against a Redis replica that hasn't yet received the lock key (split-brain)
- Lock expiry edge cases (see issue #2)

**Required fix:** Move the read inside the transaction and use a pessimistic row lock or an optimistic CAS update:

```python
with self.db.transaction():
    # SELECT ... FOR UPDATE — holds a row-level lock for the duration
    current_stock = self.db.get_stock_for_update(req.item_id)
    if current_stock is None or current_stock < req.quantity:
        # handle and return — transaction rolls back automatically
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Alternatively, use an atomic decrement with a check constraint:

```sql
UPDATE inventory
SET stock = stock - %(qty)s
WHERE item_id = %(id)s AND stock >= %(qty)s
RETURNING stock;
```

If zero rows are affected, the reservation fails. This eliminates the need for the Redis lock entirely for correctness (though the lock still has value for rate-limiting contention).

### 2. Heartbeat Lock-Loss Is Silent — Caller Proceeds Unaware

When the heartbeat detects that the lock token no longer matches (`result == 0`), it simply returns. The main thread has no idea the lock was lost and continues executing the database transaction against inventory it no longer holds exclusive access to.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← silent. main thread is unaware.
        logger.debug("Lock TTL extended key=%s", lock_key)
```

This is a correctness violation. Two processes can now both believe they hold the lock and both proceed to write stock.

**Required fix:** Use a shared flag or event to signal lock loss to the main thread, then check it before and after the DB transaction:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            lock_lost.set()
            logger.error("Lock lost during heartbeat key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then in the main path, check `lock_lost` before committing:

```python
if lock_lost.is_set():
    raise LockLostError(f"Lost distributed lock for {lock_key} during operation")
```

### 3. Lock Release After Transaction Commit — Unlock Can Fail Silently

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])  # ← return value ignored
```

The return value of `_release_script` is never checked. If the lock had already expired and been acquired by another process, this call returns `0` (token mismatch), but the code logs a success message anyway. This is a debugging nightmare and masks split-brain scenarios.

```python
released = self._release_script(keys=[lock_key], args=[token])
if released == 0:
    logger.error("Lock release failed — token mismatch, lock may have been stolen key=%s", lock_key)
```

---

### 4. Heartbeat Thread Join Timeout Is Dangerously Long

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # = 9 seconds
```

If the heartbeat thread hangs (e.g., Redis connection stall), the `finally` block blocks for 9 seconds before proceeding to release. Under flash-sale load this will exhaust thread pools and cascade into timeouts. The join timeout should be short (1–2 seconds), and if the thread hasn't exited, it should be logged as an error rather than silently proceeding.

## Moderate Issues

### 5. `quantity` Has No Upper Bound — Denial of Service Vector

```python
quantity: int = Field(..., ge=1)
```

An authenticated caller can submit `quantity=10_000_000` and acquire the lock, forcing every other request to wait through the full retry backoff. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=1000)
```

The ceiling should be a named constant pulled from configuration, not a magic number.

---

### 6. `item_id` Maximum Length of 128 Is Unvalidated Against Your Schema

The pattern `^[a-zA-Z0-9_\-]+$` is correct for injection prevention, but 128 characters is long enough that a caller can craft a key that exceeds Redis's recommended key length conventions or your DB's indexed column width. Verify this matches your actual `item_id` column definition. If the column is `VARCHAR(64)`, a 128-character input will produce a DB error that is not caught here and will propagate as an unhandled exception, bypassing the `LockAcquisitionError` handler.

---

### 7. Unhandled Exceptions Inside the Lock Context Leak the Lock

The `finally` block does release the lock on exception, which is correct. However, if `self.db.transaction()` raises an unexpected exception (connection drop, deadlock), it propagates out of `reserve_item` entirely — past both the `LockAcquisitionError` handler and the caller. The caller receives an unhandled exception rather than a `ReservationResult`. This breaks the contract of the method signature.

```python
# Missing at the top level of reserve_item:
except Exception as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Internal service error")
```

Whether to swallow all exceptions here is a design decision, but the current behavior (unhandled propagation) is inconsistent with the method's declared return type.

## Test Coverage Gaps

### 8. No Test for Heartbeat Lock-Loss Path

There is no test that simulates `_extend_script` returning `0` mid-operation and verifies the main thread does not commit. This is the most dangerous code path and it has zero coverage.

### 9. No Test for DB Transaction Failure

No test covers `self.db.transaction().__exit__` raising an exception. The lock should still be released and the result should be a failed `ReservationResult`, not an unhandled exception.

### 10. `test_lock_release_uses_lua_script` Is Fragile and Incomplete

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Use `assert_called_once_with`:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.any_str()]  # token is uuid, verify format separately
)
```

### 11. No Test for Concurrent Reservation Correctness

The entire point of this service is to handle concurrent flash-sale load. There is no test that spins up two threads simultaneously and verifies only one succeeds. This is achievable with `threading.Barrier` in a unit test.

---

## Minor / Nits

### 12. `HEARTBEAT_INTERVAL = 8` With `LOCK_TTL = 30` Is Acceptable But Underdocumented

TTL/3 is a reasonable heuristic (borrowed from Redlock). Document *why* this ratio was chosen. If `LOCK_TTL` is changed by a future developer without updating `HEARTBEAT_INTERVAL`, the safety margin silently degrades. Consider deriving it:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

### 13. `logger.info("Lock released key=%s")` Fires Even When Release Returns 0

As noted in issue #3, the log message is unconditional and will report a successful release even when the Lua script returns `0` (token mismatch / lock already gone). This will produce misleading audit logs in a government-contracting context where log integrity matters.
