Decision: Request Changes

Summary: The core locking primitive and heartbeat design are solid — atomic Lua scripts, scoped stop events, token-matched release, and jitter backoff are all correct. However, there are several architectural and correctness issues that range from a silent data-integrity hazard (stock read outside the transaction) to a real availability risk (unbounded heartbeat thread leak on Redis failure) to a test suite that doesn't actually exercise the most critical concurrent scenario it claims to cover. None of these are cosmetic; they affect correctness under the exact load profile this service is designed for.

---

## Issue 1 — CRITICAL: Stock read occurs outside the database transaction (TOCTOU)

**File:** `services/inventory.py`, lines ~90–110

```python
# CURRENT — race window exists here
with self._acquire_lock(lock_key):

    current_stock = self.db.get_stock(req.item_id)   # ← read is OUTSIDE the tx

    if current_stock < req.quantity:
        ...

    with self.db.transaction():                        # ← write is INSIDE the tx
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(...)
```

The Redis lock serializes callers *across processes*, but it does nothing to protect against a concurrent DB-level write that happens between the `get_stock` call and the `update_stock` call — for example, an admin correction, a return/refund crediting stock, or a second reservation path that bypasses this service entirely. The read must be inside the same transaction boundary, and the update should use a conditional decrement (optimistic or pessimistic at the DB level) rather than a blind overwrite.

```python
# REQUIRED pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is a *coordination* layer, not a *consistency* layer. The DB transaction is where consistency is enforced. These two must overlap.

## Issue 2 — HIGH: Heartbeat thread leaks silently on Redis connection failure during extension

**File:** `services/inventory.py`, `_beat()` function

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            return   # ← token mismatch: exits cleanly
        logger.debug("Lock TTL extended key=%s", lock_key)
    # stop_event was set — exit cleanly
```

If `self._extend_script(...)` raises a `redis.RedisError` (connection drop, timeout, cluster failover), the exception propagates out of `_beat()` uncaught. Because the thread is `daemon=True`, the exception is silently swallowed by the threading runtime — the thread dies, the heartbeat stops, the lock expires, and the main thread continues executing under the false belief that it still holds the lock. This is a phantom-lock scenario.

The heartbeat must catch Redis exceptions, signal the main thread, and the main thread must check that signal before committing.

```python
# Minimal required structure
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = self._extend_script(...)
            if result == 0:
                lock_lost_event.set()
                return
        except redis.RedisError:
            logger.error("Heartbeat Redis error — lock may have expired key=%s", lock_key)
            lock_lost_event.set()
            return

# In the main path, before committing:
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

## Issue 3 — HIGH: `t.join()` timeout is silently ignored; lock release may race the heartbeat

**File:** `services/inventory.py`, `finally` block

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← timeout is not checked
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise on timeout — it returns silently whether or not the thread actually stopped. If the heartbeat thread is blocked on a slow Redis call when `stop_event.set()` fires, `join()` may return while `_beat()` is still mid-execution and about to call `_extend_script`. The `_release_script` then runs, deletes the key, and a moment later `_extend_script` in the still-running `_beat()` attempts to extend a key that no longer exists — or worse, a key that a *new* holder has already acquired.

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
        # Do not release — the lock will expire naturally via TTL.
        # Releasing here risks a race with the still-running heartbeat.
    else:
        self._release_script(keys=[lock_key], args=[token])
```

## Issue 4 — MEDIUM: `quantity` has no upper bound; denial-of-service vector

**File:** `services/inventory.py`, `ReservationRequest`

```python
quantity: int = Field(..., ge=1)   # ← no upper bound
```

An authenticated but malicious caller can submit `quantity=2_000_000_000`. This passes validation, acquires the lock (blocking all other callers for up to 30 seconds per retry cycle), reads stock, finds it insufficient, and returns — but the lock was held the entire time. Under flash-sale load this is an effective DoS against the reservation path. Add a domain-appropriate ceiling.

```python
quantity: int = Field(..., ge=1, le=1000)  # tune to business rules
```

## Issue 5 — MEDIUM: Retry backoff is too aggressive for flash-sale load profiles

**File:** `services/inventory.py`, `_acquire_lock`

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum cumulative wait before failure is:
- Attempt 0 fails → sleep 0.1 + jitter
- Attempt 1 fails → sleep 0.2 + jitter
- Attempt 2 fails → raise

That is at most ~0.35 seconds of retry before the caller gets a 503. Under a flash sale, the lock holder may be doing a DB transaction that legitimately takes 500ms–2s. Three attempts with sub-second backoff will produce a very high false-failure rate and a flood of "Service temporarily unavailable" responses to real users. Either increase `retry_attempts` and cap the backoff, or make both configurable at construction time so callers can tune per use-case.

## Issue 6 — LOW: The concurrency test does not test concurrency

**File:** `tests/test_inventory.py`, `test_reserve_concurrent_quantity`

```python
def test_reserve_concurrent_quantity(service, db):
    db.get_stock.return_value = 3
    result = service.reserve_item("item_001", quantity=3)
    assert result.success is True
    assert result.remaining_stock == 0
```

This test is named for concurrency but is entirely sequential. It does not spawn threads, does not simulate lock contention between two simultaneous callers, and does not verify that two concurrent requests for the last 3 units result in exactly one success and one failure. This is the most important invariant of the entire service and it is untested. Add a threading-based integration test that fires two simultaneous reservations and asserts the outcome is deterministic.

## Issue 7 — LOW: Lua extension script uses `expire` (relative) not `expireat` (absolute)

**File:** `services/inventory.py`, `EXTEND_LOCK_SCRIPT`

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
...
"""
```

`EXPIRE` resets the TTL to a full `LOCK_TTL` seconds from *now*, regardless of how much time has already elapsed. This is generally acceptable, but it means a heartbeat that fires slightly late (e.g., due to GIL pressure or system load) resets the clock fully rather than extending from the original expiry. This is not a correctness bug in isolation, but if the heartbeat fires at `HEARTBEAT_INTERVAL` (8s) and the lock TTL is 30s, the effective TTL never decreases — the lock can be held indefinitely if the process hangs without triggering the stop event. Consider whether indefinite lock hold is acceptable, and if not, add a maximum hold duration enforced outside the heartbeat.

| 4 | Medium | `quantity` needs an upper bound |
