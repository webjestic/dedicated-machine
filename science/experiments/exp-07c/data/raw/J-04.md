Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, input validation, and exponential backoff are all solid. However, there are several correctness, reliability, and security concerns that must be addressed before this merges, ranging from a TOCTOU race on stock reads to a missing Redis exception handler that could leave locks orphaned, to test coverage gaps on critical failure paths.

---

## Detailed Review

### 🔴 Critical — Must Fix

**1. TOCTOU Race: Stock Read Outside the Database Transaction**

```python
# Current — read is outside the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The Redis lock serializes callers *through this service*, but it does not protect against:

- A direct database write from another service or admin tool that bypasses Redis entirely.
- A future refactor that introduces a second code path.

The read must be inside the transaction, and the update should use a conditional write (optimistic or pessimistic at the DB layer) to enforce the invariant:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return inside transaction
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is a correctness flaw, not just a theoretical concern. Under any load where stock is also modified outside this service, overselling is possible.

**2. Unhandled `redis.RedisError` Can Orphan Locks or Silently Fail**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
```

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unhandled exception, the `finally` block in `_acquire_lock` will attempt to release the lock — good. But if `self._extend_script(...)` or `self._release_script(...)` raises a `redis.RedisError` (network partition, Redis failover), the exception propagates uncaught out of the heartbeat thread or the `finally` block, and the caller receives an unhandled 500 rather than a clean `ReservationResult`. The heartbeat thread is a daemon, so a crash there is silently swallowed — but a Redis error in the `finally` block is not.

```python
# In _acquire_lock finally block:
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    try:
        self._release_script(keys=[lock_key], args=[token])
    except redis.RedisError:
        logger.error("Failed to release lock key=%s — will expire naturally in %ds", lock_key, self.LOCK_TTL)
    logger.info("Lock released key=%s", lock_key)
```

Similarly, `reserve_item` should catch broad exceptions and return a safe `ReservationResult` rather than propagating to the caller:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟡 Significant — Should Fix Before Merge

**3. Heartbeat Thread Join Timeout Is a Silent Failure**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread does not terminate within the join timeout, execution continues silently and the lock is released anyway. There is no log line, no metric, and no assertion that the thread actually stopped. A stuck heartbeat thread holding a reference to `token` and continuing to call `_extend_script` after the lock has been released is a correctness bug — it could extend a lock now held by a different caller.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible lock extension after release", lock_key)
```

---

**4. `EXTEND_LOCK_SCRIPT` Passes TTL as String — Fragile**

```python
args=[token, str(self.LOCK_TTL)]
```

Redis `EXPIRE` accepts an integer. Passing a string works because Redis coerces it, but this is an implicit contract. If `LOCK_TTL` is ever changed to a float (e.g., `30.5`), the Lua call silently fails or raises a Redis error. Enforce the type explicitly:

```python
args=[token, self.LOCK_TTL]  # redis-py serializes int correctly
```

---

**5. Retry Backoff Ceiling Is Missing**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the max backoff is `(2^2) * 0.1 + 0.05 = 0.45s` — acceptable here. But `retry_attempts` is a caller-controlled parameter with no upper bound. A caller passing `retry_attempts=20` produces a backoff of over 52 seconds on the last attempt. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

### 🟠 Test Coverage Gaps

**6. No Test for Database Exception During Transaction**

There is no test verifying that a DB exception inside the transaction block results in a clean `ReservationResult` rather than an unhandled exception propagating to the caller. This is directly related to Critical Issue #2.

```python
def test_db_exception_during_transaction_returns_safe_result(service, db):
    db.update_stock.side_effect = Exception("DB connection lost")
    result = service.reserve_item("item_001", quantity=1)
    assert result.success is False
```

---

**7. No Test for Heartbeat Lock-Loss Detection**

The `result == 0` branch in `_beat` is untested. This is a meaningful code path — if the lock is lost mid-operation, the service should log a warning. A test that configures `_extend_script` to return `0` and asserts the warning is logged would cover this.

---

**8. `test_lock_release_uses_lua_script` Is Brittle**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test. Prefer:

```python
service._release_script.assert_called_once_with(keys=[ANY], args=[ANY])
# or capture the token and assert exact args
```

---

### 🟢 Positive Notes

- Lua scripts for atomic release and extension are correct and well-commented.
- `stop_event.wait(timeout=...)` as a timer rather than `time.sleep` is the right pattern — it allows clean shutdown without sleeping through the stop signal.
- Pydantic validation with the regex pattern on `item_id` is appropriate for this threat surface.
- `daemon=True` on the heartbeat thread is correct — it will not block process shutdown.
- Structured logging with key-value pairs is consistent and grep-friendly.
- `uuid4` for both `token` and `order_id` is appropriate.
