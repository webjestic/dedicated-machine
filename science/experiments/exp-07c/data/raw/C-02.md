Decision: Request Changes

Summary: The core locking primitives are well-constructed — atomic Lua scripts, fenced tokens, heartbeat scoped to context, and `SET NX EX` acquisition are all correct. However, there are several failure modes ranging from critical race conditions to missing operational safeguards that must be addressed before this touches production flash-sale traffic. The issues below are ordered by severity.

---

## Critical

### 1. Stock read occurs outside the database transaction — TOCTOU race

```python
# CURRENT — read is outside the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers *on this service instance*, but it does not protect against:

- A second service instance that acquired the same lock after this one released it seeing a stale read if the DB read was cached or replicated with lag.
- More importantly: if the DB itself is the source of truth, the `get_stock` read and the `update_stock` write must be atomic at the DB level. A concurrent write from a different code path (manual adjustment, admin tool, another microservice) can land between the read and the write, producing a negative stock balance.

**Required fix:** Perform a conditional `UPDATE` with a `WHERE stock >= quantity` guard and check the affected row count, or use `SELECT ... FOR UPDATE` inside the transaction to re-read and lock the row at the DB level. The Redis lock is a performance optimization (reduces DB contention), not a substitute for DB-level consistency.

---

### 2. Heartbeat thread silently swallows the lock-loss signal — no caller notification

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return   # ← exits thread; caller has no idea
```

When the heartbeat detects lock loss it logs and returns. The `yield token` in the context manager is still executing. The caller continues to believe it holds the lock, reads stock, and writes an order — all without the lock. This is the exact failure mode the heartbeat is supposed to prevent.

**Required fix:** Use a shared `threading.Event` (e.g., `lock_lost_event`) that the heartbeat sets on token mismatch. The `yield` body should check this event at a safe point, or the `finally` block should inspect it and raise before releasing:

```python
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Lock loss detected key=%s", lock_key)
            lock_lost.set()
            return
```

Then in `reserve_item`, after the `with self._acquire_lock(lock_key) as token:` block, or by passing the event out of the context manager, raise a `LockLostError` if `lock_lost.is_set()`. This is a fundamental correctness requirement for a fenced locking system.

## High

### 3. `t.join(timeout=...)` does not verify the thread actually stopped

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout)` returns `None` whether the thread stopped or timed out. If the heartbeat thread is blocked on a slow Redis call, the lock is released while the heartbeat is still running. The heartbeat then wakes, calls `_extend_script`, and if a new holder has already acquired the lock with the same key (different token), the Lua script correctly returns 0 — but if by some collision the token matched, it would extend a lock the new holder believes is theirs.

**Required fix:**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not release — the lock state is uncertain. Raise or let it expire.
    raise RuntimeError(f"Heartbeat thread stuck for key={lock_key}")
self._release_script(keys=[lock_key], args=[token])
```

---

### 4. Redis connection errors are not caught — unhandled exceptions propagate

`self.redis.set(...)`, `self._extend_script(...)`, and `self._release_script(...)` can all raise `redis.exceptions.RedisError` (connection timeout, sentinel failover, etc.). None of these are caught. In the `finally` block especially, an exception during `_release_script` will suppress the original exception and leave the lock held until TTL expiry.

**Required fix:** Wrap Redis calls in `try/except redis.exceptions.RedisError` at minimum in the `finally` block. Log and allow TTL-based expiry as the fallback. In the acquisition loop, distinguish transient connection errors from contention.

---

### 5. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is dangerously tight for slow DB operations

A flash sale DB under load can easily exceed 8 seconds for a transaction. The heartbeat fires at TTL/3, which is reasonable, but 30 seconds is a long time to hold a per-item lock under high concurrency. More critically, if the DB transaction takes longer than 30 seconds (deadlock, slow replica, etc.), the lock expires before the transaction commits, another caller acquires the lock, reads the old stock (transaction not yet committed), and a double-reservation occurs.

**Required fix:** Either enforce a DB transaction timeout shorter than `LOCK_TTL`, or make `LOCK_TTL` configurable and document the required relationship: `LOCK_TTL > max_expected_db_transaction_time`, and `HEARTBEAT_INTERVAL < LOCK_TTL / 2`. Add an assertion or startup check.

---

## Medium

### 6. `retry_attempts=3` with exponential backoff has an unbounded worst-case wait

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

Attempt 0: ~0.10s, Attempt 1: ~0.20s, Attempt 2: ~0.40s. This is actually fine numerically, but the jitter ceiling of `0.05` is very small relative to the base — under a thundering herd all callers will retry at nearly the same intervals, defeating the purpose of jitter. The jitter should be proportional to the backoff:

```python
backoff = (2 ** attempt) * 0.1
jitter = random.uniform(0, backoff)
sleep_time = backoff + jitter
```

---

### 7. `new_stock` is computed before the transaction and passed as a literal value

```python
new_stock = current_stock - req.quantity
...
self.db.update_stock(req.item_id, new_stock)
```

This passes an absolute value to `update_stock`. If the DB layer does `SET stock = new_stock` rather than `SET stock = stock - quantity WHERE stock >= quantity`, any concurrent write between the read and the write will be silently overwritten. This compounds issue #1. The update should be expressed as a delta with a guard condition, not an absolute assignment.

---

### 8. Lock key namespace is predictable and unsegmented

```python
lock_key = f"inventory:lock:{req.item_id}"
```

There is no environment or tenant prefix. In a shared Redis instance (staging keys leaking to production, multi-tenant deployment), lock keys from different environments will collide. Add a configurable namespace prefix injected at construction time.

---

## Low / Test Coverage Gaps

### 9. No test for heartbeat lock-loss path

The most critical new behavior — heartbeat detecting token mismatch — has zero test coverage. There should be a test where `_extend_script` returns `0` and the test asserts that the reservation either fails or raises, not silently succeeds.

### 10. No test for Redis connection failure during lock acquisition or release

`redis_client.set.side_effect = redis.exceptions.ConnectionError` should be a test case.

### 11. `test_lock_release_uses_lua_script` is fragile

```python
service._release_script.assert_called_once()
```

This asserts call count but not arguments. A future refactor could call the script with wrong keys/args and this test would still pass. Assert the specific `keys` and `args` values.

### 12. The `db` fixture's transaction mock does not verify rollback behavior

There is no test asserting that if `create_order` raises, `update_stock` is not committed. This is the most important property of the transaction wrapper.
