Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent data-integrity gap, and test coverage holes that would make this unsafe to merge into a production flash-sale path. The issues range from a race condition in the heartbeat teardown to a missing exception handler around the database transaction that could leave inventory in an inconsistent state.

---

## Findings

### 🔴 Critical

---

**1. Database transaction exception is unhandled — silent inventory corruption**

```python
# services/inventory.py  ~line 112
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `update_stock` succeeds but `create_order` raises, the behavior depends entirely on what `self.db.transaction().__exit__` does with the exception. That contract is never enforced here, and there is no `except` block catching `Exception` around the transaction. If the underlying DB layer swallows the exception or auto-commits partial work, stock is decremented with no corresponding order record. The lock is then released and the unit of inventory is gone forever.

**Required fix:** Wrap the transaction block in a `try/except Exception`, log the failure, and return a failure `ReservationResult`. Do not rely on the DB abstraction's `__exit__` to be the only safety net.

```python
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error(
        "Transaction failed item_id=%s order_id=%s: %s",
        req.item_id, order_id, e
    )
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

**2. Heartbeat thread is joined *before* lock release — race window on lock loss**

```python
# services/inventory.py  ~line 82
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)          # ← blocks up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])  # ← release happens after join
```

The join timeout is `HEARTBEAT_INTERVAL + 1 = 9 seconds`. If the heartbeat thread is mid-sleep when `stop_event.set()` is called, `t.join()` will block for up to 9 seconds before the release script runs. During that window the lock is still held and no other caller can proceed. For a flash-sale service under load, this is a multi-second stall on every successful reservation.

The correct ordering is: release the lock first (the Lua script is safe regardless of thread state), then signal and join the thread.

```python
finally:
    stop_event.set()
    self._release_script(keys=[lock_key], args=[token])  # release immediately
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)          # then reap the thread
    logger.info("Lock released key=%s", lock_key)
```

### 🟠 Major

---

**3. `quantity` is not validated as an upper bound — stock can go negative**

```python
class ReservationRequest(BaseModel):
    quantity: int = Field(..., ge=1)
```

`ge=1` prevents zero or negative quantities but places no ceiling. A caller can pass `quantity=999999` and, if stock is 5, the insufficient-stock branch catches it. However, there is no defense against a caller passing a quantity that is technically satisfiable but abusive (e.g., bulk-draining an entire SKU in one call). For a flash-sale context this is a business logic control that must exist at the model layer.

```python
quantity: int = Field(..., ge=1, le=100)  # enforce per-reservation ceiling
```

The exact ceiling is a product decision, but the absence of any ceiling is a gap.

---

**4. `get_stock` result is trusted without a DB-level consistency guarantee**

```python
current_stock = self.db.get_stock(req.item_id)
# ...
new_stock = current_stock - req.quantity
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The stock is read outside the transaction and then written inside it. The Redis lock serializes callers *for this service instance*, but it does not protect against:

- A separate service or admin tool writing stock directly to the DB.
- A Redis failover or network partition causing the lock to be silently lost between `get_stock` and `update_stock`.

The `update_stock` call should use an optimistic or conditional write — e.g., `UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s` — and the transaction should verify that exactly one row was affected. If zero rows are affected, the reservation must be aborted.

```python
rows_affected = self.db.update_stock_conditional(
    req.item_id, new_stock, expected=current_stock
)
if rows_affected == 0:
    raise StockConflictError("Stock changed under lock — aborting reservation")
```

**5. Heartbeat thread silently swallows Redis errors**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

If `self._extend_script` raises a `redis.RedisError` (connection drop, timeout, cluster failover), the exception propagates out of `_beat`, the thread dies, the lock is no longer being renewed, and the main thread has no signal that this happened. The main thread will continue executing the DB transaction against a lock that is expiring.

```python
try:
    result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
except redis.RedisError as e:
    logger.error("Heartbeat extension failed key=%s: %s", lock_key, e)
    return  # or signal the main thread via an event
```

A more robust design would expose a `lock_lost_event` that the main thread checks before committing the transaction.

### 🟡 Minor / Observations

---

**6. `t.join(timeout=...)` return value is never checked**

`Thread.join(timeout=N)` returns `None` whether the thread exited or timed out. If the heartbeat thread is stuck (e.g., blocked in a Redis call), the join silently times out and the thread is abandoned as a daemon. This is survivable but should at minimum be logged.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s", lock_key)
```

---

**7. Test for lock release uses a brittle assertion**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

`service._release_script` is the mock returned by `register_script`, so this assertion passes trivially regardless of what arguments were passed. It does not verify that the correct `lock_key` and `token` were supplied to the script. Use `assert_called_once_with` with argument matchers, or capture the call args explicitly.

---

**8. No test for the heartbeat renewal path**

There is no test that verifies the heartbeat actually fires and calls `_extend_script`. Given that the heartbeat is a core correctness mechanism (it is the entire reason the lock does not expire under a slow DB), this path must be covered. A test can use a short `LOCK_TTL` / `HEARTBEAT_INTERVAL` override and a `time.sleep` to trigger at least one renewal cycle.

---

**9. No test for DB transaction failure**

There is no test simulating `db.update_stock` or `db.create_order` raising an exception inside the transaction. Once the exception handler from finding #1 is added, this path needs a corresponding test to confirm the `ReservationResult` is a failure and that no partial state is visible.
