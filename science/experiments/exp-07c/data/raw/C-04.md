Decision: Request Changes

Summary: The core distributed locking primitive is structurally sound — atomic Lua scripts, fenced tokens, heartbeat renewal, and exponential backoff with jitter are all present and correctly implemented. However, there are several meaningful correctness, reliability, and security issues that must be addressed before this merges, ranging from a race condition in the heartbeat shutdown path to missing exception handling around the database layer and incomplete test coverage of critical failure modes.

---

## Findings

### 🔴 Critical

**1. Heartbeat thread can extend a lock that has already been atomically released**

In `_acquire_lock`'s `finally` block:

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`stop_event.set()` signals the heartbeat to stop, but `t.join()` only *waits* — it does not guarantee the thread has exited before `_release_script` runs. If the heartbeat thread is mid-sleep when `stop_event.set()` fires, `wait()` will return `True` and the thread will exit cleanly. But if the thread is currently *executing* `_extend_script` (i.e., it woke up and is in the Redis round-trip), `t.join()` will block until that call completes, and then `_release_script` will run — which is correct.

The actual race is subtler: if `t.join(timeout=...)` **times out** (Redis is slow, network hiccup), the heartbeat thread is still alive and running when `_release_script` fires. The release succeeds (Lua script deletes the key), but the heartbeat thread then calls `_extend_script` on a now-deleted key — which will return `0` (token mismatch path), log a spurious "lock loss" warning, and exit. This is not catastrophic because the Lua guard prevents re-extending a key owned by another caller, but it produces misleading log noise and indicates the join timeout is not enforced as a hard guarantee. More importantly, if the join times out silently, you have a leaked daemon thread with no alerting.

**Fix:** After `t.join(timeout=...)`, assert or log explicitly if the thread is still alive:

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible Redis latency spike", lock_key)
self._release_script(keys=[lock_key], args=[token])
```

Consider also whether a leaked-thread scenario should raise rather than silently continue.

**2. Unhandled exceptions from the database layer will propagate through the lock context without cleanup**

Inside the `with self._acquire_lock(lock_key):` block:

```python
current_stock = self.db.get_stock(req.item_id)
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `get_stock`, `update_stock`, `create_order`, or the transaction context manager raises any exception (network error, ORM error, integrity constraint), it will propagate out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` *will* run and release the lock — that part is correct. However, `reserve_item` has no `except` clause for these cases, so the exception bubbles up to the caller as an unhandled exception rather than a `ReservationResult(success=False, ...)`. This breaks the contract of the method, which is documented to always return a `ReservationResult`.

**Fix:**

```python
except (DatabaseError, SomeOtherExpectedException) as e:
    logger.error("DB error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

At minimum, a broad `except Exception` guard should wrap the lock body if this is intended to be a resilient service boundary.

### 🟠 High

**3. Stock read is outside the database transaction — TOCTOU window within the lock**

```python
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
    self.db.create_order(...)
```

The Redis lock serializes access across service instances, so concurrent *reservation* calls are protected. However, `get_stock` is executed outside the DB transaction. If any other code path (admin adjustment, cancellation, manual correction) can write to the stock table without going through this lock, the value read by `get_stock` may be stale by the time `update_stock` commits. The correct pattern is to move the read inside the transaction and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic read) so the DB itself enforces consistency:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

This also makes the Redis lock a performance optimization (reducing DB lock contention) rather than the sole correctness mechanism, which is a much more defensible architecture.

**4. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is dangerously close for a government-grade service**

The heartbeat fires at TTL/3.75. Under normal conditions this is fine. But consider: the heartbeat fires, the `_extend_script` call is queued in Redis, Redis is under load and takes 7+ seconds to respond. The lock expires at second 30. The extend response arrives at second 31 — the lock is already gone. The heartbeat returns `0`, logs "lock loss," and exits. Meanwhile the main thread is still inside the DB transaction believing it holds the lock.

This is the classic Redlock fencing problem. For a flash sale inventory system (financial correctness required), the Redis lock alone is not sufficient if the DB does not also enforce the invariant. This reinforces finding #3: the DB transaction must be the final arbiter of correctness, not the Redis lock.

Additionally, consider documenting why these specific constants were chosen and whether they are configurable per-deployment.

### 🟡 Medium

**5. `quantity` has no upper bound — unbounded reservation is a business logic vulnerability**

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can submit `quantity=1_000_000` and, if stock is available, reserve the entire warehouse in one call. For a flash sale this is almost certainly unintentional and could be exploited.

**Fix:**

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business rule is
```

---

**6. `item_id` length is validated but the lock key is not length-bounded in Redis**

`item_id` is capped at 128 characters, and the lock key is `inventory:lock:{item_id}`. The resulting key is at most ~145 characters, which is within Redis limits, so this is not a hard bug — but it is worth noting that the Pydantic validation on `item_id` is the only thing preventing a pathologically long key. This is fine as long as validation always runs before `_acquire_lock` is called, which it does here. Just ensure this invariant is documented so future callers of `_acquire_lock` directly are aware.

---

**7. Retry backoff ceiling is very low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45s`. Under a flash sale with thousands of concurrent requests, this will result in a thundering herd of retries all converging within a half-second window. Consider increasing jitter range significantly or using a capped exponential:

```python
backoff = min((2 ** attempt) * 0.1, 2.0) + random.uniform(0, 0.5)
```

### 🔵 Test Coverage Gaps

**8. No test for DB exception propagation**

There is no test that verifies behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises an exception. Given finding #2, this is a correctness gap — the test suite would not catch the unhandled exception behavior.

**9. No test for heartbeat thread join timeout**

The `t.is_alive()` scenario (finding #1) has no test coverage. This is a hard scenario to test deterministically, but at minimum a test that injects a slow `_extend_script` and verifies the log output or thread state would be appropriate.

**10. `test_lock_release_uses_lua_script` is fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A future refactor could call the script with wrong arguments and this test would still pass. Use `assert_called_once_with(keys=[...], args=[...])` — though since the token is a UUID generated internally, you may need to capture it via a mock side effect.
