Decision: Request Changes

Summary: The core distributed locking pattern is sound — atomic Lua scripts, heartbeat renewal, scoped stop events, and Pydantic validation are all implemented correctly. However, there are several correctness, reliability, and security issues that need to be addressed before this can be approved for a production flash sale system. The most critical is a TOCTOU race between the stock read and the database write, which undermines the entire purpose of the distributed lock.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU Race: Stock Read Outside the Database Transaction**

```python
# services/inventory.py — lines ~110–125
current_stock = self.db.get_stock(req.item_id)   # ← read is OUTSIDE transaction

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write is INSIDE transaction
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but `get_stock` is issued outside the database transaction. If the database itself has any read-committed or higher isolation behavior, a crash or rollback between the read and the write leaves stock in an inconsistent state. More critically, if this service ever runs across multiple processes (which is the entire point of a *distributed* lock), a process restart between the read and the write will allow another process to read the same stale stock value before the first write commits.

The stock read must be inside the transaction, and the update should use an optimistic or pessimistic DB-level lock (e.g., `SELECT ... FOR UPDATE`) as a second line of defense. The Redis lock alone is not sufficient to guarantee consistency against DB-level races.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return — transaction rolls back cleanly
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Silent Data Loss on Database Exception**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

There is no `except` block around the transaction. If `update_stock` or `create_order` raises, the exception propagates up through the lock context manager's `finally` block, the lock is released, and the caller receives an unhandled exception rather than a `ReservationResult`. In a flash sale context this means the caller gets a 500 and may retry, potentially double-reserving if the transaction partially committed depending on the DB driver. The transaction block must be wrapped and the exception must be caught and returned as a failed `ReservationResult`.

```python
try:
    with self.db.transaction():
        ...
except Exception as e:
    logger.error("DB transaction failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 🟠 High

**3. Heartbeat Thread Leak on Lock Acquisition Failure Path**

```python
t = threading.Thread(target=_beat, daemon=True)
t.start()

try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block calls `t.join(timeout=...)` but does not check whether the join actually completed. If the heartbeat thread is blocked (e.g., Redis is slow), `join` returns after the timeout and the lock release script runs anyway — but the thread is still alive and will attempt to extend a lock that has already been released and potentially re-acquired by another caller. This is a correctness violation: the heartbeat could extend a lock it no longer owns (the Lua script guards against token mismatch, so it won't extend the wrong lock, but the thread itself leaks). Add a check and log a warning:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

**4. `remaining_stock` Leaks Internal State on Failure**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,  # ← exact stock count returned to caller
    message="Insufficient stock"
)
```

Returning the exact remaining stock count to an external caller is an information disclosure issue. In a flash sale context this allows a client to probe exact inventory levels, which is commercially sensitive and potentially exploitable for timing attacks (e.g., holding locks to starve other buyers). Return a boolean or a bucketed value (`"low"`, `"out_of_stock"`) instead, or omit `remaining_stock` on failure entirely.

---

### 🟡 Medium

**5. `retry_attempts` Is Not Validated**

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

`retry_attempts` is an internal parameter, but if a caller passes `0` or a negative value, the `for` loop exits immediately and `acquired` is `False`, raising `LockAcquisitionError` without ever attempting acquisition. This is a latent bug. Add a guard:

```python
retry_attempts = max(1, retry_attempts)
```

---

**6. Test Coverage Gaps**

The test suite covers the happy path and basic failure modes well, but is missing:

- **No test for the DB transaction exception path** (finding #2 above). This is the most important missing test given the severity of that bug.
- **No test verifying stock read is inside the transaction** — this is the architectural guarantee the whole service is built on.
- **`test_lock_release_uses_lua_script` is fragile**: it asserts `_release_script.assert_called_once()` but the mock returns `1` for all script calls, meaning it cannot distinguish between the release script and the extend script being called. Use `assert_called_once_with` with explicit `keys` and `args` arguments.
- **No test for `retry_attempts=0`** edge case.
- **No concurrency/integration test** — acceptable for a unit test file, but a note should be added that integration tests under load are required before production deployment.

---

**7. Lock Key Namespace Collision Risk**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which is good. However, if this Redis instance is shared with other services (common in government contracting environments where infrastructure is shared across contract boundaries), there is no tenant or service namespace prefix. Add an application-level prefix or enforce Redis key namespacing via a dedicated DB index.

---

### 🔵 Low / Nits

- `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 8` are module-level constants but should be constructor parameters (or at minimum environment-variable-driven) to allow tuning without a code deploy. Flash sale load profiles vary significantly.
- The `_beat` thread is `daemon=True`, which is correct for preventing process hang on shutdown, but this should be documented in a comment since it's a deliberate choice with implications.
- `logger.info("Lock released key=%s", lock_key)` fires even if the Lua release script returns `0` (lock was already lost). The log message is misleading in that case. Check the return value of `_release_script` and log accordingly.

---

## Required Before Approval

1. Move `get_stock` inside the database transaction and use a DB-level lock (`SELECT FOR UPDATE` or equivalent).
2. Wrap the transaction block in a `try/except` and return a `ReservationResult` on failure.
3. Remove exact stock count from failure responses.
4. Add tests for the DB exception path and fix the fragile release script assertion.
