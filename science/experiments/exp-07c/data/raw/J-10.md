Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic Lua scripts for lock release/extension, Pydantic validation, scoped heartbeat state, and a reasonable test suite. However, there are several correctness, resilience, and security issues that must be addressed before this merges into a government-facing production system. The most critical is a TOCTOU race between the stock read and the database write, but there are additional concerns around exception handling, lock semantics, and test coverage gaps.

---

## Issues

### 🔴 Critical

**1. TOCTOU Race — Stock Read Outside the Database Transaction**

```python
# CURRENT — read is outside the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes stale value
    self.db.create_order(...)
```

The stock is read *before* the transaction opens. If the database supports it, `get_stock` must be called *inside* the transaction with a pessimistic row lock (e.g., `SELECT ... FOR UPDATE`). The Redis lock serializes callers *through this service*, but it does not protect against writes from other services, direct DB access, or a Redis failover that causes two nodes to both believe they hold the lock. The stock decrement must be atomic at the database layer as well.

```python
# REQUIRED pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None: ...
    if current_stock < req.quantity: ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

**2. Unhandled Exceptions Inside the Lock Context Silently Hold the Lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # raises → propagates out
```

Any unexpected exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate through `reserve_item` uncaught. The `finally` block in `_acquire_lock` will release the lock correctly, but the *caller* receives an unhandled exception rather than a graceful `ReservationResult`. For a service endpoint this means a 500 with a potential stack trace leak. Catch broad exceptions inside the lock body and return a safe result.

```python
except Exception as e:
    logger.error("Unexpected error during reservation item_id=%s", req.item_id, exc_info=True)
    return ReservationResult(success=False, message="Internal service error")
```

### 🟠 High

**3. Redis Connection Errors Are Not Handled**

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.exceptions.ConnectionError`, `TimeoutError`, and `ResponseError` are not caught anywhere. A Redis outage will raise through `_acquire_lock`, bypass the `LockAcquisitionError` handler in `reserve_item`, and surface as an unhandled exception. These must be caught explicitly and either retried or converted to a `LockAcquisitionError` so the caller gets a clean degraded response.

---

**4. Heartbeat Thread Failure Is Silent**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return
```

When the heartbeat detects lock loss (`result == 0`), it logs a warning and exits — but the main thread continues executing the database transaction under the assumption it still holds the lock. This is a correctness violation. The heartbeat must signal the main thread to abort. Use a shared `lock_lost_event = threading.Event()` and check it at safe points, or raise into the main thread via a flag checked before the DB write.

---

**5. `LOCK_TTL = 30` With `HEARTBEAT_INTERVAL = 8` Is Fragile Under Load**

30 seconds is an unusually long TTL for a flash sale reservation. Under high concurrency this creates a 30-second blast radius if a process dies after acquiring the lock but before releasing it. Consider 5–10 seconds with a proportionally shorter heartbeat. Additionally, document the reasoning for these values — magic numbers in a distributed system are a maintenance hazard.

---

### 🟡 Medium

**6. `remaining_stock` Leaks Internal State on Failure Paths**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,   # exact stock count returned to caller
    message="Insufficient stock"
)
```

Returning the exact remaining stock count to an external caller is an information disclosure risk. An adversary can use repeated requests to monitor inventory levels in real time. Return a boolean or a bucketed value (e.g., `"low"`, `"out_of_stock"`) for failure responses. The success path may reasonably return remaining stock for UX purposes, but this should be a deliberate product decision documented here.

**7. `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` Does Not Check Join Result**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the thread does not exit within the timeout, `join` returns silently and the release script runs anyway — but the thread is still alive and may attempt a lock extension *after* the lock has been released and potentially re-acquired by another caller. Check `t.is_alive()` after join and log a critical-level alert if the thread is still running.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical("Heartbeat thread did not terminate key=%s — potential lock corruption", lock_key)
```

---

### 🔵 Low / Test Coverage Gaps

**8. No Test for Heartbeat Lock-Loss Detection**

The heartbeat's `result == 0` branch (lock lost mid-operation) has zero test coverage. This is the most operationally dangerous code path and must be tested.

**9. No Test for Redis Connection Failure**

There is no test verifying that a `redis.exceptions.ConnectionError` during lock acquisition produces a graceful `ReservationResult` rather than an unhandled exception.

**10. `test_lock_release_uses_lua_script` Is Brittle**

```python
service._release_script.assert_called_once()
```

This asserts the script was called but not *what it was called with*. Assert the keys and args explicitly to prevent a regression where the script is called with the wrong token.

```python
service._release_script.assert_called_once_with(keys=[expected_lock_key], args=[ANY])
```

---

## What Is Done Well

- Lua scripts for atomic release and extension are correctly implemented and registered at construction time.
- `SET NX EX` is the correct atomic acquisition primitive — no separate `EXPIRE` call.
- `stop_event.wait(timeout=...)` as a timer rather than `time.sleep` is the right pattern.
- Pydantic validation with a strict regex on `item_id` prevents injection via the key path.
- Exponential backoff with jitter is correctly implemented.
- The daemon thread flag is appropriate.
- Structured logging is consistent and includes relevant context fields.
