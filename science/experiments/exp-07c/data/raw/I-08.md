Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness, safety, and architectural issues that must be resolved before this can be approved for a high-concurrency flash sale context. The most critical concern is a TOCTOU race between the stock read and the database write that the distributed lock does not fully protect against, compounded by a heartbeat thread design that silently swallows lock loss without propagating failure to the caller.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

**1. TOCTOU: Stock read is outside the database transaction**

```python
# Lock is held, but this read is NOT inside the transaction block
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # Writes a value derived from a stale read
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock is read before the transaction opens. If the database supports any level of read isolation weaker than serializable — or if the Redis lock is ever lost between the read and the write (see issue #2) — another process can modify stock between `get_stock` and `update_stock`. The write then commits a value derived from a stale snapshot.

**Required fix:** Move `get_stock` inside the transaction boundary, and use a conditional update (optimistic locking or a `WHERE stock >= quantity` guard) at the database layer to make the decrement atomic:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

The distributed lock reduces contention; the database transaction with a row-level lock is what actually guarantees correctness.

---

**2. Heartbeat thread detects lock loss but cannot propagate it to the caller**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # ← silently exits; caller continues executing under no lock
```

When the heartbeat detects a token mismatch it logs a warning and returns. The main thread is completely unaware. It continues reading and writing inventory data under the false assumption that it still holds the lock. This is the exact failure mode a heartbeat is supposed to prevent.

**Required fix:** Use a shared flag or event to communicate lock loss back to the critical section, and check it before committing:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return

# Inside the critical section, before the DB write:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost during operation for {lock_key}")
```

**3. `t.join()` timeout is silently ignored — lock may be released before heartbeat exits**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # ← join can time out
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise on timeout. If the heartbeat thread is blocked on a slow Redis call and the join times out, `_release_script` executes while `_beat` is still running and potentially still calling `_extend_script`. This creates a race where the lock is released and then immediately re-extended by the zombie heartbeat, which could extend a lock now held by a different caller.

**Required fix:**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not exit cleanly for key=%s — lock release deferred", lock_key)
        # Policy decision: raise, alert, or accept the risk explicitly
    self._release_script(keys=[lock_key], args=[token])
```

At minimum the condition must be detected and logged at ERROR severity. Ideally this raises so the anomaly surfaces in monitoring.

### 🟡 Significant — Should Fix Before Merge

---

**4. `LOCK_TTL = 30s` with `HEARTBEAT_INTERVAL = 8s` is dangerously tight for a flash sale**

The comment says "fires at TTL/3" but `8 * 3 = 24`, not `30`. The first heartbeat fires at `t=8s`, which leaves a `22s` window before expiry on the first interval — acceptable. But the real risk is that a slow database transaction (index contention, replication lag, lock wait) can easily exceed 30 seconds under flash sale load. When it does, the lock expires, a second caller acquires it, and both callers are now inside the critical section simultaneously.

The heartbeat mitigates this only if it fires successfully. Under Redis cluster failover or network partition, it may not. Consider:

- Making `LOCK_TTL` and `HEARTBEAT_INTERVAL` constructor parameters so they can be tuned per deployment
- Documenting the maximum tolerated DB operation latency explicitly
- Adding a circuit breaker or timeout on the DB operations themselves

---

**5. Retry backoff ceiling is extremely low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → ~0.10s, attempt=1 → ~0.20s, attempt=2 → ~0.40s
```

With `retry_attempts=3` the total maximum wait is under one second. Under a flash sale with hundreds of concurrent requests, this will produce a thundering herd: all waiters will exhaust retries nearly simultaneously and return `503` to users. The jitter range (`0–0.05s`) is too narrow to spread load meaningfully.

**Recommended fix:**

```python
MAX_BACKOFF = 2.0  # seconds
jitter = random.uniform(0, min(0.5, backoff))
backoff = min((2 ** attempt) * 0.2 + jitter, MAX_BACKOFF)
```

Also consider making `retry_attempts` a parameter rather than a hardcoded default.

---

**6. `db` abstraction has no defined interface — contract is implicit**

`InventoryService` calls `self.db.get_stock`, `self.db.transaction`, `self.db.update_stock`, and `self.db.create_order` with no protocol, abstract base class, or type annotation enforcing these methods exist. The tests mock all of them, which means a real `db` implementation that spells one method differently will fail at runtime, not at test time.

**Required fix:** Define an explicit interface:

```python
from typing import Protocol, ContextManager

class InventoryDB(Protocol):
    def get_stock(self, item_id: str) -> Optional[int]: ...
    def get_stock_for_update(self, item_id: str) -> Optional[int]: ...
    def update_stock(self, item_id: str, new_stock: int) -> None: ...
    def create_order(self, order_id: str, item_id: str, quantity: int) -> None: ...
    def transaction(self) -> ContextManager: ...
```

### 🔵 Minor — Address in Follow-up or as Noted

---

**7. Test for lock release uses `assert_called_once()` without verifying arguments**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test verifies the script was called but not that it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test. Use `assert_called_once_with(keys=[...], args=[...])` — or at minimum capture and inspect the call args.

---

**8. No test for heartbeat thread behavior under lock loss**

The heartbeat's lock-loss detection path (`result == 0`) has zero test coverage. Given that this is the primary safety mechanism for long-running operations, it needs an explicit test that mocks `_extend_script` to return `0` and asserts the expected behavior (once issue #2 is fixed, this should assert that the DB write does not proceed).

---

**9. `quantity` has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=999999999`. Add a reasonable upper bound (`le=`) appropriate to your business domain. This also prevents a class of abuse where a single request attempts to drain all inventory.

---

**10. Redis connection errors are not caught**

Any `redis.RedisError` (connection refused, timeout, cluster failover) raised inside `_acquire_lock` or `_beat` will propagate as an unhandled exception through `reserve_item`. The caller receives a 500-class error with no structured `ReservationResult`. Catch `redis.RedisError` at the `reserve_item` boundary and return a structured failure result, consistent with how `LockAcquisitionError` is handled.
