Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a critical race condition, missing fault-tolerance paths, and test gaps that make this unfit for a high-availability flash sale system. The issues range from a stock-read/lock-order inversion that undermines the entire locking guarantee, to an unhandled DB exception path that will leak locks and corrupt inventory state under failure.

---

## Findings

### 🔴 Critical

---

**1. Stock read occurs outside the lock — the lock protects nothing**

```python
# _acquire_lock is a context manager; the lock is acquired here
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # ✅ inside lock — this is fine
```

Wait — re-reading the code, the `get_stock` call *is* inside the `with self._acquire_lock(lock_key):` block. The lock is acquired before the read. This is structurally correct.

However, the real critical issue is below.

---

**1. (Actual Critical) Unhandled DB exception leaks the lock and leaves inventory in an undefined state**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # can throw
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # can throw mid-transaction
        self.db.create_order(order_id, req.item_id, req.quantity)  # can throw
```

If `get_stock`, `update_stock`, or `create_order` raises an exception, it propagates out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` *will* run, so the lock itself is released — that part is safe. **But the database transaction behavior is undefined.** If `update_stock` succeeds and `create_order` throws, the transaction context manager's `__exit__` is responsible for rollback. There is zero guarantee that the injected `db` object handles this correctly, and the code makes no attempt to catch, log, or surface DB-layer exceptions. A partial write here means stock is decremented with no corresponding order — inventory corruption.

The service must catch DB exceptions explicitly, ensure rollback semantics are enforced, and return a safe `ReservationResult` rather than propagating a raw exception to the caller.

```python
# Required pattern
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error("DB write failed during reservation item_id=%s order_id=%s: %s",
                 req.item_id, order_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

**2. Heartbeat thread silently swallows lock loss — caller is never notified**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # ← exits thread silently
```

When the heartbeat detects token mismatch (lock expired and stolen), it logs a warning and exits. The main thread has **no idea this happened**. It continues executing the DB transaction believing it holds the lock — it does not. This is the exact failure mode distributed locking is meant to prevent. Another process has now acquired the lock and may be reading/writing the same inventory record concurrently.

The heartbeat must signal the main thread. The canonical pattern is to set a shared flag or event that the main thread checks at critical points, or to use a `threading.Event` that the main thread can observe before committing.

```python
# Minimal fix: expose lock_lost_event to the context manager scope
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock loss detected key=%s — signaling main thread", lock_key)
            lock_lost.set()
            return

# In reserve_item, before committing:
if lock_lost.is_set():
    logger.error("Aborting reservation — lock was lost during operation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Reservation aborted due to lock loss.")
```

### 🟠 Major

---

**3. `t.join(timeout=...)` does not verify the thread actually stopped**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=N)` returns after N seconds regardless of whether the thread has exited. If the heartbeat thread is blocked on a slow Redis call, `join` returns and the lock is released while the heartbeat thread is still alive and may call `_extend_script` moments later on a key that now belongs to a different holder.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible Redis hang", lock_key)
    # Do not release — or release and accept the race. Document the decision.
```

This is a hard problem; the right answer depends on your Redis client timeout configuration. At minimum, the failure must be detected and logged. Silently proceeding is not acceptable.

---

**4. Retry backoff is too short and the jitter range is too narrow for a flash sale**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0: 0.10–0.15s
# attempt=1: 0.20–0.25s
# attempt=2: 0.40–0.45s
```

With `retry_attempts=3` and a 30-second lock TTL, a caller gives up after ~0.75 seconds maximum. Under a flash sale with hundreds of concurrent requests, the thundering herd will cause mass `LockAcquisitionError` failures within the first second. The jitter ceiling of 0.05s provides almost no desynchronization across concurrent waiters.

Recommended: increase jitter to `random.uniform(0, backoff)` (full jitter), increase `retry_attempts` to at least 5–10, and make both configurable at construction time rather than hardcoded.

---

**5. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants, not instance-configurable**

These values are appropriate for some workloads and wrong for others. A slow DB host, a large transaction, or a network partition changes the calculus entirely. These must be constructor parameters with documented defaults, not magic numbers baked into the class definition.

---

### 🟡 Moderate

---

**6. `_extend_script` passes TTL as a string — type safety is implicit and fragile**

```python
args=[token, str(self.LOCK_TTL)]
```

The Lua script receives `ARGV[2]` as a string and passes it directly to `redis.call("expire", ...)`. Redis will coerce this, but this is an undocumented implicit contract. The TTL should be validated as a positive integer before being passed, and a comment should document the expected type.

---

**7. No Redis connection error handling anywhere in the lock path**

If Redis is unavailable, `self.redis.set(...)`, `self._extend_script(...)`, and `self._release_script(...)` will all raise `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`. None of these are caught. A Redis outage during a flash sale will propagate raw exceptions to callers rather than returning a graceful `ReservationResult`. At minimum, Redis errors in the lock acquisition path should be caught and converted to `LockAcquisitionError`.

---

**8. `db.get_stock` result is trusted without type validation**

```python
current_stock = self.db.get_stock(req.item_id)
if current_stock is None: ...
if current_stock < req.quantity: ...
```

If `db.get_stock` returns a non-integer (e.g., a float, a string from a misconfigured ORM, or a negative value due to a prior bug), the comparison `current_stock < req.quantity` will either raise a `TypeError` or silently produce wrong behavior. Stock values should be validated as non-negative integers before use.

---

### 🟢 Test Coverage Gaps

---

**9. No test for heartbeat lock-loss signaling**

The most critical behavioral path — heartbeat detects token mismatch and signals the main thread — has zero test coverage. This is the core safety guarantee of the design and it is completely untested.

---

**10. No test for DB exception during transaction**

There is no test verifying that a `db.update_stock` or `db.create_order` exception results in a safe `ReservationResult(success=False)` rather than an unhandled exception propagating to the caller.

---

**11. `test_lock_release_uses_lua_script` asserts on the wrong mock object**

```python
service._release_script.assert_called_once()
```

`_release_script` is assigned during `__init__` as the return value of `mock.register_script(...)`. The test is asserting on the mock returned by `register_script`, which is the same object used for both `_release_script` and `_extend_script` (since `register_script` is a single mock returning the same value for all calls). This test may pass vacuously. The fixture should configure distinct return values per script registration and assert on the correct one
