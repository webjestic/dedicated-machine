Decision: Request Changes

Summary: The core distributed locking pattern is well-constructed — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, security, and architectural issues that must be addressed before this merges, ranging from a TOCTOU race between the stock read and the DB write, to missing error handling around the database transaction that could leave inventory in a corrupted state. The test suite also has structural gaps that would let real bugs pass undetected.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

**1. TOCTOU Race: Stock Read Outside the Database Transaction**

```python
# Current — stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but the stock read and the stock write are not atomic at the database layer. If the DB connection is routed through a read replica, or if the transaction isolation level is READ COMMITTED, the value of `current_stock` may already be stale by the time the `UPDATE` executes. A rollback or replica lag between the read and the write can silently oversell inventory.

The stock read must happen **inside** the transaction, and the `UPDATE` should use a conditional clause to make it atomic at the DB level as a second line of defense:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT ... FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Unhandled Exception Inside the Lock Context Leaks the Lock**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `db.get_stock`, `db.update_stock`, `db.create_order`, or the transaction context manager raises an unexpected exception (network timeout, DB constraint violation, etc.), it propagates out of `_acquire_lock`'s `try/yield` block, which does correctly release the lock via `finally`. However, the `reserve_item` method has **no outer `except` clause** for these cases. The exception will bubble up to the caller as an unhandled 500-class error with no `ReservationResult` returned, breaking the contract of this method.

```python
# Missing entirely:
except Exception as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="An internal error occurred.")
```

---

**3. Heartbeat Thread Silently Swallows Lock Loss — No Propagation Path**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # <-- exits the thread, but the main thread has no idea
```

When the heartbeat detects a lost lock (result == 0), it logs a warning and exits. The main thread continues executing the DB transaction believing it holds the lock — it does not. This is the exact scenario the heartbeat is supposed to guard against, but it provides no protection because there is no signaling back to the main thread.

At minimum, a shared flag must be checked at the yield point or the main thread must be interrupted:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s — flagging for abort", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Then inside the lock context, before the DB write:

```python
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock was lost during operation for {lock_key}")
```

### 🟡 Significant — Should Fix Before Merge

---

**4. `t.join(timeout=...)` Does Not Guarantee the Thread Has Stopped**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=N)` returns after N seconds **regardless of whether the thread finished**. If the heartbeat thread is blocked on a slow Redis call, the release script fires while `_beat` may still be mid-execution, creating a race between the release and a concurrent extend. The join should either be unbounded or followed by a liveness check:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly for key=%s", lock_key)
```

---

**5. `item_id` Leaked Into Error Message Returned to Caller**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Even though `item_id` is validated by the regex, embedding user-supplied input into response messages is a bad habit in a government contracting context. An enumerator can use this to confirm which item IDs exist vs. do not exist. Return a generic message and log the detail server-side:

```python
logger.info("Stock lookup returned None for item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

---

**6. No Redis Connection Error Handling**

Every Redis call (`self.redis.set`, `self._extend_script`, `self._release_script`) can raise `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`. None of these are caught. A Redis outage will raise an unhandled exception rather than returning a graceful `ReservationResult`. At minimum, the `_acquire_lock` context manager should catch and re-raise as `LockAcquisitionError`, and `reserve_item` should catch it.

---

### 🔵 Test Coverage Gaps

---

**7. No Test for DB Exception Inside the Lock**

There is no test verifying that a `db.update_stock` exception results in a clean `ReservationResult` rather than an unhandled exception. This is directly related to Critical Issue #2.

---

**8. No Test for Heartbeat Lock-Loss Path**

The `result == 0` branch in `_beat` is completely untested. Given that this is the primary safety mechanism of the heartbeat, it needs a test that mocks `_extend_script` to return 0 and verifies the main thread aborts the operation (once Issue #3 is fixed).

---

**9. `test_lock_release_uses_lua_script` Is Fragile and Incomplete**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with the wrong token would pass this test. Assert the call arguments explicitly:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.approx_any_string]  # or capture the token via mock
)
```

### ⚪ Minor / Nits

- `HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` means the first heartbeat fires at 8s, leaving 22s of buffer. This is fine, but a comment explaining the TTL/3 rationale in the constant definition would help future maintainers avoid accidentally setting `HEARTBEAT_INTERVAL >= LOCK_TTL`.
- The `retry_attempts` parameter on `_acquire_lock` is not exposed through `reserve_item`. If callers ever need to tune retry behavior per item type (e.g., high-demand vs. low-demand SKUs), this will require an interface change. Consider threading it through now.
- `daemon=True` on the heartbeat thread is correct — it prevents the thread from blocking interpreter shutdown — but worth a comment since it's a non-obvious choice.
