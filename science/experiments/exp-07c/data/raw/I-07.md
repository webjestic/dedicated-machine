Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a silent failure mode that could cause inventory corruption under real concurrent load, and test gaps that would let those flaws ship undetected. The issues range from a race condition between stock read and DB write, to a heartbeat that can silently lose the lock without surfacing the failure to the caller, to missing exception handling around the database transaction itself.

---

## Detailed Findings

### 🔴 Critical — Silent Inventory Corruption on DB Transaction Failure

```python
# services/inventory.py  lines ~100-103
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

There is no `try/except` around the transaction block. If `update_stock` or `create_order` raises — network blip, constraint violation, deadlock — the exception propagates out of `reserve_item` entirely uncaught. The caller receives an unhandled exception rather than a `ReservationResult`. Worse, depending on the DB driver's rollback semantics and whether the context manager is implemented correctly on the injected `db`, partial writes are possible. This needs an explicit `except` that rolls back and returns a safe `ReservationResult(success=False, ...)`.

```python
# Suggested shape
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error("Transaction failed item_id=%s order_id=%s: %s", req.item_id, order_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 🔴 Critical — Heartbeat Lock Loss Is Silently Swallowed

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return   # <-- thread exits, but the main thread has no idea
```

When the heartbeat detects a lost lock (result == 0), it logs a warning and exits. The main thread continues executing the DB transaction believing it holds the lock. Another process may have already acquired the lock and begun its own read-stock/write cycle. This is the exact race condition the lock is meant to prevent, and the heartbeat's detection of it is doing nothing actionable.

The heartbeat needs a shared signal back to the main thread. A simple approach is a `threading.Event` named `lock_lost_event` that the heartbeat sets, and the main thread checks before committing:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return

# In the main thread, before committing:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost mid-operation for {lock_key}")
```

### 🟠 Major — Stock Read Outside the DB Transaction Creates a TOCTOU Window

```python
current_stock = self.db.get_stock(req.item_id)   # read is OUTSIDE the transaction

if current_stock < req.quantity:
    ...

with self.db.transaction():                        # write is INSIDE
    self.db.update_stock(req.item_id, new_stock)
```

The Redis lock serializes access at the application layer, which is the right instinct. But the stock read happens before the DB transaction opens. If the DB itself is accessed by any path that bypasses this service (admin tooling, a migration, another service), the stock value read here may be stale by the time the transaction commits. The read should be inside the transaction boundary and, ideally, use a `SELECT ... FOR UPDATE` or equivalent pessimistic row lock at the DB layer to make the guarantee durable:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # row-locked read
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 🟠 Major — `t.join()` Timeout Silently Ignored; Lock Released Anyway

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` does not raise if the thread fails to stop within the timeout — it returns silently. If the heartbeat thread is hung (e.g., blocked on a Redis call), the lock is released while the thread is still running. That thread may then call `_extend_script` after release, potentially extending a lock that has already been re-acquired by another caller. The return value of `join` should be checked:

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly for key=%s", lock_key)
    # Do NOT release — the thread's state is undefined. Raise or alert.
```

Whether to release or not in this state is a policy decision, but silently proceeding is not acceptable.

### 🟡 Moderate — Retry Backoff Is Too Aggressive for a Flash Sale Workload

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum wait before giving up is approximately `0.1 + 0.2 + jitter ≈ 0.35s`. Under genuine flash sale load — hundreds of concurrent requests — three attempts with sub-second backoff will result in a thundering herd on attempt exhaustion. Consider either increasing `retry_attempts` with a cap, or exposing `retry_attempts` as a constructor parameter so callers can tune it per use case. The jitter range of `0–0.05s` is also too narrow to meaningfully desynchronize competing threads; `random.uniform(0, 0.1 * (2 ** attempt))` would spread them more effectively.

### 🟡 Moderate — `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no `le=` constraint. A caller can pass `quantity=2_000_000_000`. This will pass validation, read stock, compute `new_stock` as a large negative number if stock is lower, and — if the stock check logic ever has an off-by-one — write a corrupted value to the database. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=10_000)  # tune to business requirements
```

### 🟡 Moderate — `item_id` Leaked Into Error Message Returned to Caller

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

In a government contracting context, returning internal identifiers in API responses is an information disclosure concern. The item ID here is user-supplied and already known to the caller, so the risk is low — but the pattern is worth flagging. If `item_id` ever becomes an internal surrogate key rather than a user-facing slug, this becomes a real finding. Prefer a generic message or ensure the calling layer strips internal identifiers before surfacing responses externally.

### 🟡 Moderate — Redis Connection Errors Are Completely Unhandled

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not caught anywhere in `_acquire_lock` or `reserve_item`. A Redis outage causes an unhandled exception to propagate to the caller. This should be caught and converted to a `LockAcquisitionError` or a safe `ReservationResult`, with appropriate alerting:

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.RedisError as e:
    logger.error("Redis error during lock acquisition key=%s: %s", lock_key, e)
    raise LockAcquisitionError(f"Redis unavailable for {lock_key}") from e
```

### 🔵 Minor — Test Suite Does Not Cover the Critical Paths Identified Above

The tests cover the happy path and basic validation, which is good. But none of the following are tested:

| Missing Test | Why It Matters |
|---|---|
| DB transaction raises mid-write | Verifies rollback behavior and safe return value |
| Heartbeat detects lock loss (`_extend_script` returns 0) | Verifies the main thread is notified and aborts |
| Redis raises `ConnectionError` during lock acquisition | Verifies graceful degradation |
| `t.join()` timeout exceeded (thread still alive) | Verifies the hung-thread path is handled |
| `quantity` at boundary values (e.g., `le` max) | Verifies upper bound once added |
