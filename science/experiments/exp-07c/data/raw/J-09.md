Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges, ranging from a TOCTOU race between the stock read and the DB transaction, to an unhandled exception path that can silently leave stock in an inconsistent state, to test gaps that would miss these failure modes in CI.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU between stock read and DB write (race condition)**

```python
# Inside the lock — looks safe, but isn't if the DB enforces no constraint
current_stock = self.db.get_stock(req.item_id)   # read
# ... heartbeat fires, lock is held, but:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # blind write
    self.db.create_order(...)
```

`update_stock` performs a blind write of a pre-computed value. If the DB layer does not enforce a `WHERE stock >= quantity` guard, a crash-and-retry scenario or a second service instance that acquired the lock after a Redis failover (see finding #2) can drive stock negative. The fix is a conditional update at the DB layer:

```python
# Preferred: push the guard into the DB
rows_affected = self.db.update_stock_if_sufficient(
    req.item_id, req.quantity  # UPDATE ... SET stock = stock - %s WHERE stock >= %s
)
if rows_affected == 0:
    raise InsufficientStockError(...)
```

This makes the DB the authoritative source of truth and the Redis lock a contention-reducer, not the sole safety gate.

---

**2. No handling of unhandled exceptions inside the lock context**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # can raise
    ...
    with self.db.transaction():
        self.db.update_stock(...)   # can raise
        self.db.create_order(...)   # can raise
```

Any exception from the DB calls propagates out of `reserve_item` entirely uncaught. The `_acquire_lock` `finally` block will release the lock correctly, but the caller receives an unhandled exception rather than a `ReservationResult`. For a service method this breaks the contract implied by the return type and will cause 500s upstream. Add a broad catch inside `reserve_item`:

```python
except (LockAcquisitionError) as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

---

**3. Redis failover / split-brain is not acknowledged (Redlock not used)**

A single-node Redis lock is vulnerable to the following scenario: the Redis primary fails after `SET NX EX` succeeds, the replica is promoted, and a second caller acquires the same lock on the new primary before the first caller's TTL would have expired. Both callers now hold the lock simultaneously. For a government-grade inventory system this is a material risk. Either:

- Document explicitly that this service requires a Redis Sentinel/Cluster setup with `wait` configured, **and** that the DB-level conditional update (finding #1) is the true safety net, or
- Implement Redlock across ≥3 independent Redis nodes if the DB cannot provide a conditional update.

---

### 🟡 Major

**4. Heartbeat thread silently swallows lock loss — caller is not notified**

```python
if result == 0:
    logger.warning("Heartbeat detected lock loss key=%s", lock_key)
    return   # thread exits, but the main thread keeps executing inside the lock context
```

When the heartbeat detects token mismatch it logs and exits, but the main thread continues executing the DB transaction believing it still holds the lock. This is the exact scenario the heartbeat was designed to catch. The thread needs a way to signal the main thread to abort. A simple approach:

```python
# Use a shared Event
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return
        ...

# In the main thread, check before the DB write
if lock_lost_event.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

---

**5. `t.join(timeout=...)` result is not checked — silent thread leak**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is blocked (e.g., on a slow Redis call) and does not exit within the timeout, `join` returns silently and the lock is released while the thread is still running. The thread then holds a reference to `token` and `lock_key` and may attempt to extend a lock that has already been released and potentially re-acquired. Check the return:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly key=%s — possible thread leak", lock_key)
    # Do not suppress; alert monitoring
```

---

**6. `item_id` leaked into error message returned to caller**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Even though `item_id` is validated by the regex, reflecting user-supplied input into response messages is a habit that has caused issues in similar services. Use a generic message or an opaque code:

```python
message="Item not found"
```

### 🟠 Minor / Test Gaps

**7. No test for DB exception path**

There is no test that verifies behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises. Given finding #2, this is a required test before merge:

```python
def test_db_exception_returns_error_result(service, db):
    db.get_stock.side_effect = Exception("DB connection lost")
    result = service.reserve_item("item_001")
    assert result.success is False
    assert "Internal error" in result.message
```

**8. No test for heartbeat lock-loss signaling (finding #4)**

Once finding #4 is fixed, a test should verify that if the extend script returns `0` during an active reservation, the operation is aborted rather than completing.

**9. `test_lock_release_uses_lua_script` is fragile**

```python
service._release_script.assert_called_once()
```

This asserts call count but not arguments. If the script is called with the wrong key or token, the test still passes. Prefer:

```python
service._release_script.assert_called_once_with(
    keys=[pytest.approx(...)], args=[ANY]
)
```

Or at minimum assert `keys` contains the expected lock key pattern.

---

**10. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` — document the reasoning**

The comment says `TTL/3` but `8` is closer to `TTL/3.75`. This is not a bug, but for a codebase that will be audited, the constants should have an explicit comment explaining the safety margin (e.g., "fires at TTL/3 to allow two missed heartbeats before expiry"). This is especially important for government audit trails.
