Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat scoping, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, reliability, and security concerns that must be addressed before this merges. The test suite covers happy paths and basic edge cases but has meaningful gaps around the race conditions this code is specifically designed to prevent.

---

## Issues

### 🔴 Critical

**1. TOCTOU race between `get_stock` and `update_stock`**

The stock read happens *outside* the database transaction:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside tx
```

The distributed lock serializes callers *through this service*, but it provides no protection against:
- Direct DB writes from other services or admin tooling
- A second service instance that doesn't participate in the same Redis lock namespace
- Any future code path that bypasses this service

The stock read must be inside the transaction, and `update_stock` should use a conditional decrement (optimistic or pessimistic at the DB level) rather than a blind write of a pre-computed value:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
```

Without this, overselling is possible.

---

**2. Heartbeat thread silently swallows lock loss — no rollback signal**

When the heartbeat detects a lost lock (`result == 0`), it logs a warning and returns. But the main thread has no way to know this happened. It continues executing the DB transaction against a lock it no longer holds, potentially writing stock decrements that are now racing with another holder of the same lock:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # ← main thread is never notified
```

The heartbeat needs a shared flag or event that the main thread checks at critical points, or the lock context manager itself should surface this as an exception. A `threading.Event` named `lock_lost` that the main thread checks before committing the transaction would be the minimum fix.

### 🟠 High

**3. `t.join(timeout=...)` does not guarantee the thread has stopped before lock release**

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If `t.join` times out (the heartbeat thread is hung on a slow Redis call), the release script runs while the heartbeat thread may still be mid-execution of `_extend_script`. This is a narrow window but a real one. You should check `t.is_alive()` after the join and log a critical-level alert if the thread did not stop cleanly. In a government-grade system this should also increment a metric or trigger an alert.

---

**4. `remaining_stock` leaks internal inventory state to callers on failure**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,  # ← exact stock count returned
    message="Insufficient stock"
)
```

Returning the exact remaining stock on a failed reservation exposes inventory levels to any caller who can probe the API. In a flash sale context this is an enumeration vector — a bad actor can binary-search exact stock counts. Return a boolean or a bucketed value (`"low"`, `"out"`) instead, or omit `remaining_stock` on failure entirely.

---

**5. Lock key namespace is flat and predictable**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which is good. But there is no tenant, environment, or service prefix. If this Redis instance is shared with other services (common in cost-constrained government deployments), a key collision with an unrelated service using the same naming pattern is possible. Prefix with a service-scoped namespace: `obotix:inventory:lock:{item_id}`.

---

### 🟡 Medium

**6. No Redis connection error handling**

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not caught anywhere. A Redis outage will propagate an unhandled exception out of `reserve_item`, bypassing the `ReservationResult` contract the caller expects. Wrap Redis calls and return a `ReservationResult(success=False, message="Service temporarily unavailable")` on connection failure.

---

**7. `LOCK_TTL = 30` is arbitrary and undocumented**

30 seconds is a long TTL for a lock in a flash sale system where throughput matters. There is no comment explaining how this value was chosen relative to expected DB transaction latency. If a DB operation hangs for 29 seconds, the lock expires, another caller acquires it, and both proceed. Document the expected p99 DB latency this is sized against, and consider making it a constructor parameter so it can be tuned per deployment without a code change.

---

**8. Test for lock release uses `assert_called_once()` without verifying arguments**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test does not verify that the release script was called with the correct `keys` and `args` (i.e., the right lock key and the matching token). A bug that calls the script with wrong arguments would pass this test. Assert on the call arguments.

---

**9. No test for lock-lost-during-transaction scenario**

Given that issue #2 above is the most dangerous correctness bug, there is no test that simulates the heartbeat detecting lock loss while the DB transaction is in progress. This is the exact failure mode this architecture is meant to handle, and it has zero test coverage.

---

**10. Daemon thread suppresses clean shutdown visibility**

```python
t = threading.Thread(target=_beat, daemon=True)
```

Marking the heartbeat as a daemon thread means it will be killed silently on interpreter shutdown without releasing the lock or logging. In a long-running service this is acceptable, but it should be a conscious documented decision, not a default. Add a comment explaining the tradeoff.

---

## What's Done Well

- Lua scripts for atomic release and extension are correct and the right approach
- `stop_event.wait(timeout=...)` as a timer is the correct pattern (avoids busy-wait)
- Pydantic validation with regex on `item_id` prevents injection into the lock key
- `SET NX EX` is atomic — no separate `EXISTS` + `SET` race
- `ValidationError` is caught and surfaced cleanly without leaking stack traces
- Exponential backoff with jitter is properly implemented
- Structured logging with contextual fields throughout
