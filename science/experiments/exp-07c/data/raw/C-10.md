Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, architectural gaps, and test coverage blind spots that would make this unsafe to deploy in a high-stakes flash sale context. The issues range from a race condition in the heartbeat shutdown sequence to a missing DB rollback path and unhandled Redis exceptions that could silently corrupt inventory state.

---

## Critical Issues

### 1. Race Condition: Heartbeat Can Extend Lock After Release

This is the most serious flaw. In the `finally` block:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`stop_event.set()` signals the heartbeat to stop, but `t.join()` has a timeout of `HEARTBEAT_INTERVAL + 1` (9 seconds). If the heartbeat thread is mid-execution when `stop_event.set()` is called — specifically inside `_extend_script(...)` — the join will wait, but there is no guarantee the thread completes the extension *before* the release script fires. More critically: if `t.join()` times out (which it can, since it's bounded), the release script runs while the heartbeat thread is still alive and may call `_extend_script` after the lock has been released and potentially re-acquired by another process. The Lua token check is the only guard, but the window is real and the join timeout silently swallows the failure.

**Fix:** Reverse the order — release the lock first, then signal stop and join. Or use an internal flag that the heartbeat checks before extending, set atomically before the release call.

---

### 2. No Rollback on Database Transaction Failure

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `create_order` raises after `update_stock` succeeds, the behavior depends entirely on whether `db.transaction()`'s `__exit__` rolls back on exception. This is not enforced anywhere in this service. If the DB layer silently commits partial state, stock is decremented without an order record — inventory is permanently lost. There is no explicit rollback call, no exception catch around the transaction block, and no test for this failure mode.

**Fix:** Catch exceptions from the transaction block explicitly, log them, and return a failure `ReservationResult`. Do not assume the DB abstraction handles rollback correctly without testing it.

---

### 3. Unhandled Redis Exceptions Propagate Uncaught

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are never caught anywhere in `_acquire_lock` or `reserve_item`. A Redis connectivity blip during `self.redis.set(...)`, `_extend_script(...)`, or `_release_script(...)` will raise an unhandled exception that bubbles up to the caller with no `ReservationResult` returned — violating the method's own return type contract and potentially crashing the calling thread.

**Fix:** Wrap Redis operations in targeted exception handlers. At minimum, catch `redis.RedisError` in `reserve_item` and return a safe failure result.

## Significant Issues

### 4. `t.join(timeout=...)` Failure Is Silently Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join()` with a timeout does not raise if the thread is still alive after the timeout expires. The code proceeds to release the lock regardless. There is no `t.is_alive()` check after the join. In production under load, this means the heartbeat thread can outlive the lock context, and you will never know.

**Fix:**
```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

### 5. Stock Read Outside the Database Transaction

```python
current_stock = self.db.get_stock(req.item_id)
# ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The stock read happens outside the transaction. While the Redis lock serializes access for this service, this design is fragile. If there are other writers to the DB (direct SQL, another service, migrations, admin tooling) that do not respect the Redis lock, the read-check-write sequence is not atomic at the DB level. For a government contracting context where correctness guarantees must be provable, the stock read should occur inside the transaction with a `SELECT FOR UPDATE` or equivalent pessimistic lock.

---

### 6. Heartbeat Thread Leak on Lock Acquisition Failure Path

In `_acquire_lock`, if `acquired` is False, a `LockAcquisitionError` is raised before the thread is ever started — that path is fine. However, if `self.redis.set(...)` raises mid-retry loop (see issue #3), the function exits without ever entering the `try/finally` block that manages the thread. The thread hasn't started yet in that case, so there's no leak there — but the exception propagates completely unhandled to `reserve_item`, which has no handler for it either. This is a gap in the exception contract.

## Minor Issues

### 7. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

For a service deployed across environments (dev, staging, prod), these should be configurable via constructor parameters or environment variables. A 30-second lock TTL that is appropriate for a slow staging DB may be dangerously long in production, holding locks during cascading failures.

---

### 8. Test Coverage Gaps

The tests mock away all the interesting concurrency behavior:

- **No test for DB transaction failure** — the partial-write scenario (issue #2) has zero coverage.
- **No test for Redis exception during lock acquisition or release** — issue #3 is completely untested.
- **No test that verifies heartbeat thread terminates** — `t.is_alive()` is never asserted.
- **`test_lock_release_uses_lua_script`** asserts `_release_script.assert_called_once()` but the mock is `register_script`'s return value, which is shared across both `_release_script` and `_extend_script`. This test may pass for the wrong reason depending on mock call routing.
- **No integration-level test** against a real or fake Redis (e.g., `fakeredis`) to validate the Lua scripts actually execute correctly.

---

### 9. Logger Name Will Be `services.inventory` — Confirm This Is Intentional

```python
logger = logging.getLogger(__name__)
```

This is fine, but confirm the logging configuration in the deployment environment routes this namespace correctly. No issue per se, just worth a callout for ops visibility.
