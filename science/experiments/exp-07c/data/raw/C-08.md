Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all appropriate choices for this problem domain. However, there are several correctness defects, a meaningful race condition, missing error handling paths, and test gaps that must be addressed before this is production-ready for a high-stakes flash sale context.

---

## Critical Issues

### 1. Race Condition: Stock Read Outside the Database Transaction

**File:** `services/inventory.py`, `reserve_item()`, lines reading `current_stock` and the `with self.db.transaction()` block.

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read is performed outside the database transaction. Even with the Redis lock held, if the Redis lock is ever lost and re-acquired (e.g., due to a GC pause on this node, network partition, or clock skew), another process could modify stock between the read and the write. More practically: the Redis lock and the database are two separate consistency domains. The database transaction should be the authoritative guard. The correct pattern is a **compare-and-swap or SELECT FOR UPDATE inside the transaction**, so the database itself enforces the invariant atomically. As written, the stock check is advisory only.

**Required fix:** Move `get_stock` inside the transaction and use a locking read (e.g., `SELECT ... FOR UPDATE`) if the DB supports it, or use an optimistic update with a conditional `WHERE stock >= quantity` and check affected rows.

### 2. Unhandled Exception Paths Leave Lock Unreleased (Partially) and Swallow DB Errors

**File:** `services/inventory.py`, `reserve_item()`

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unexpected exception (network error, DB timeout, ORM error), it propagates out of `reserve_item()` entirely — not caught, not returned as a `ReservationResult`. The `_acquire_lock` context manager's `finally` block will release the Redis lock correctly, but the **caller receives an unhandled exception** rather than a structured result. For a service layer in a government-grade system, all exception paths must return structured results or be explicitly re-raised with proper typing.

**Required fix:** Wrap the lock body in a broad `except Exception` (or specific DB exception types) and return a `ReservationResult(success=False, message="Internal error")` with appropriate logging, or declare the exception contract explicitly in the interface.

---

### 3. Heartbeat Thread Silently Loses the Lock Without Notifying the Caller

**File:** `services/inventory.py`, `_beat()` inner function

```python
if result == 0:
    logger.warning("Heartbeat detected lock loss key=%s", lock_key)
    return  # <-- thread exits, but the main thread has no idea
```

When the heartbeat detects lock loss (`result == 0`), it logs and exits. The main thread continues executing — it proceeds to write to the database believing it holds the lock, when it does not. This is the classic **fencing token problem**. Without a monotonic fencing token passed to the database layer, or without the heartbeat signaling the main thread to abort, this is a correctness hole: two processes can simultaneously believe they hold the reservation and both commit.

**Required fix:** The heartbeat must signal the main thread on lock loss. One approach: set a shared `threading.Event` (e.g., `lock_lost_event`) that the main thread checks between the stock read and the DB write, and raises an exception if set. Alternatively, use a fencing token (e.g., Redis `INCR`-based version) passed into DB writes so the database can reject stale writers.

## Moderate Issues

### 4. `t.join(timeout=...)` Does Not Guarantee Thread Termination

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call (e.g., network stall during `_extend_script`), `join` returns after the timeout and execution proceeds to `_release_script`. The release script may then race with an in-flight extend call. The thread is `daemon=True` so it won't block process exit, but the race is real within a single request lifecycle.

**Required fix:** Check `t.is_alive()` after `join` and log a warning if the thread did not terminate. Consider adding a timeout to the Redis extend call itself.

### 5. Exponential Backoff Ceiling Is Missing

**File:** `services/inventory.py`, `_acquire_lock`

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With only 3 attempts this is bounded at `~0.45s` maximum, which is acceptable here. However, if `retry_attempts` is ever increased (it's a parameter), there is no cap. With `retry_attempts=10`, the final backoff is `~51.2s`, which will cause request timeouts and thread exhaustion in a flash sale scenario. This is a latent misconfiguration hazard.

**Required fix:** Add `min(backoff, MAX_BACKOFF)` with a defined constant (e.g., `MAX_BACKOFF = 2.0`).

---

### 6. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Not Validated Against Each Other

**File:** `services/inventory.py`, class constants

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

The comment says "fires at TTL/3" but this is a hardcoded assumption. If someone changes `LOCK_TTL` to 10 without updating `HEARTBEAT_INTERVAL`, the heartbeat fires at TTL/1.25 — barely ahead of expiry, with no margin for Redis latency. There is no enforcement of the relationship.

**Required fix:** Either derive `HEARTBEAT_INTERVAL` from `LOCK_TTL` (e.g., `HEARTBEAT_INTERVAL = LOCK_TTL // 3`), or add an `__init__` assertion: `assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2`.

## Minor Issues

### 7. Test for Lock Release Is Fragile and Incomplete

**File:** `tests/test_inventory.py`, `test_lock_release_uses_lua_script`

```python
service._release_script.assert_called_once()
```

This asserts the script was called once, but does not assert it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments (wrong token, wrong key) would pass this test. Given this is the safety-critical release path, the assertion should verify the arguments.

**Required fix:**
```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"], args=[pytest.ANY]
)
```

---

### 8. No Test for Heartbeat Lock-Loss Path

The scenario in Issue #3 — heartbeat detects `result == 0` and exits — has zero test coverage. This is the most dangerous failure mode in the implementation and it is completely untested.

**Required fix:** Add a test that mocks `_extend_script` to return `0` mid-operation and verifies the system behaves safely (does not commit, returns an error result).

---

### 9. No Test for DB Exception During Transaction

No test covers `db.update_stock` or `db.create_order` raising an exception. Given Issue #2 above, this is both a code defect and a test gap.

---

### 10. `quantity` Field Has No Upper Bound

**File:** `services/inventory.py`, `ReservationRequest`

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity. A caller can request `quantity=10_000_000` and the validation passes. For a flash sale system, a per-request quantity cap should be enforced at the validation layer.
