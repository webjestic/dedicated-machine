Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness bugs, a critical race condition in the heartbeat/release sequencing, missing exception handling around DB and Redis failures, and test coverage gaps that would allow silent data corruption under production load. This cannot be approved for a high-concurrency flash sale context without the issues below addressed.

---

## Critical Issues

### 1. Race Condition: Lock Released Before Heartbeat Thread Fully Exits

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # waits up to 9s
    self._release_script(keys=[lock_key], args=[token])  # released after join
```

This ordering looks correct at first glance, but `t.join()` has a timeout. If the heartbeat thread does **not** exit within `HEARTBEAT_INTERVAL + 1` seconds (e.g., due to Redis latency in `_extend_script`), execution falls through and the lock is released while the heartbeat thread is still alive and may still call `_extend_script`. The thread could then extend a lock that has already been re-acquired by another process (token mismatch protects against *deletion*, but the extend script could theoretically race against a new token being set in a narrow window if Redis is slow).

**More importantly:** there is no check on whether `t.join()` actually succeeded. Silent timeout is a hidden failure mode.

```python
# Required fix
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — lock release proceeding anyway", lock_key)
    # Proceed, but this is an observable anomaly that warrants alerting
self._release_script(keys=[lock_key], args=[token])
```

### 2. No Exception Handling Around Database Operations Inside the Lock

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `db.get_stock()`, `db.update_stock()`, or `db.create_order()` raises **any exception**, it propagates out of `_acquire_lock`'s `try/yield` block, hits the `finally`, and the lock is released — which is correct. However, the caller of `reserve_item()` receives an **unhandled exception** rather than a `ReservationResult`. The method signature promises to return a `ReservationResult` in all cases, but it does not. This will cause 500-level failures to surface to callers in production.

```python
# Required fix: wrap DB operations
try:
    with self._acquire_lock(lock_key):
        ...
except LockAcquisitionError as e:
    ...
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 3. Stock Read Outside the Database Transaction — TOCTOU Within the Lock

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # read outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # write inside transaction
```

The stock read and the stock write are not in the same database transaction. The Redis lock serializes access from this service, but it does **not** protect against:

- A direct database write from another service or admin tool
- A database replica lag if `get_stock` reads from a replica
- Any scenario where the DB transaction is rolled back after the read already occurred

The read must be inside the transaction, and ideally the update should use an optimistic or conditional write (e.g., `UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?` with row count validation) rather than a read-compute-write pattern. The current pattern is fragile even with the Redis lock in place.

```python
# Recommended fix
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ...
    if current_stock < req.quantity:
        return ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 4. No Exception Handling Around Redis Operations in the Heartbeat Thread

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return
```

If `_extend_script` raises a `redis.RedisError` (connection drop, timeout, cluster failover), the exception is **unhandled inside a daemon thread**. The thread dies silently, the lock is no longer being renewed, and the operation continues under the assumption that the lock is held. The lock will expire at `LOCK_TTL` (30s), potentially allowing another process to acquire it while the current operation is still running.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        except redis.RedisError:
            logger.exception("Heartbeat Redis error key=%s — lock renewal failed", lock_key)
            return  # or signal the main thread via an event
        if result == 0:
            ...
```

## Significant Issues

### 5. `LOCK_TTL` of 30 Seconds Is Dangerously Long for a Flash Sale

A 30-second lock TTL means a crashed or hung process can block all inventory operations for up to 30 seconds. For a flash sale with high concurrency, this is an unacceptable blast radius. The heartbeat pattern is specifically designed to allow a **short** TTL (e.g., 5–10 seconds) with renewal. The TTL should be reduced to match the heartbeat interval more tightly, and the heartbeat interval should be `TTL / 3` or less — the comment says this but the values don't reflect it (`HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` is `TTL / 3.75`, which is acceptable, but the TTL itself is still too long).

---

### 6. `retry_attempts=3` With Exponential Backoff Has Unbounded Worst-Case Wait

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

- Attempt 0: ~0.10s
- Attempt 1: ~0.20s
- Attempt 2: ~0.40s

Total worst-case blocking time before raising `LockAcquisitionError`: ~0.75s. This is acceptable, but the backoff cap should be made explicit. If `retry_attempts` is ever increased (e.g., passed as a parameter), the backoff grows unboundedly. Add a `max_backoff` cap:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 1.0)
```

### 7. Heartbeat Thread Communicates Lock Loss But Cannot Cancel the Main Thread

When the heartbeat detects `result == 0` (lock lost), it logs a warning and returns. The main thread continues executing the database transaction **without a valid lock**. This defeats the entire purpose of the distributed lock. The heartbeat needs a mechanism to signal the main thread that the lock has been lost, and the main thread must check for this condition.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost key=%s — signaling main thread", lock_key)
            lock_lost_event.set()
            return

# In the main thread, after DB operations or at a checkpoint:
if lock_lost_event.is_set():
    raise LockLostError(f"Distributed lock was lost during operation for {lock_key}")
```

This is a fundamental correctness issue for a system that relies on the lock for inventory integrity.

## Minor Issues

### 8. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity validation. A single request could reserve `quantity=999999` items. Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business rule is
```

### 9. Test Coverage Gaps

The test suite is missing coverage for the following scenarios:

- **Heartbeat thread behavior**: No test verifies that `_extend_script` is called during a long-running operation, or that lock loss is detected.
- **DB exception propagation**: No test for `db.get_stock()` or `db.transaction()` raising an exception — this would currently surface as an unhandled exception to the caller (see Issue #2).
- **Transaction rollback**: No test verifying that if `create_order` fails, `update_stock` is also rolled back.
