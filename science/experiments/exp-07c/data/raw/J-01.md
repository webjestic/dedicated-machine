Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic input validation are all sound. However, there is a meaningful TOCTOU race between the stock read and the database write, the heartbeat thread is not joined before the lock release script fires in all code paths, and the test suite has gaps that would let these bugs ship silently. The issues below range from correctness bugs to minor hardening concerns; none are catastrophic, but several need to be resolved before this merges into a high-concurrency flash-sale path.

---

## Findings

### 1. TOCTOU: Stock Read Outside the Database Transaction (High)

```python
# Current — read is outside the transaction
current_stock = self.db.get_stock(req.item_id)   # ← dirty read possible

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but `get_stock` is issued before the database transaction opens. If the underlying DB engine allows dirty or non-repeatable reads (READ COMMITTED or weaker), another process that bypasses the Redis lock (admin tool, migration script, a second app node that lost its lock silently) can decrement stock between the read and the write. The fix is to move the read inside the transaction and use a `SELECT … FOR UPDATE` (or equivalent) so the DB itself also serializes the row:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Early returns inside a `with self.db.transaction()` block need to be verified as safe for your DB abstraction (i.e., that `__exit__` rolls back cleanly on a non-exception path), but the read must be inside the transaction boundary.

### 2. Lock Release Can Fire Before `t.join()` Completes (Medium)

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← join can time out
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call (network hiccup, Redis replica failover), `join` times out and the release script fires while the heartbeat thread is still alive and holding a reference to `token`. The heartbeat then calls `_extend_script` after the lock has been released — and potentially after another caller has acquired it with a different token. Because the extend script checks the token, the extend will correctly return 0 in that case, but the window is real and the log warning will be confusing. The safer pattern is to check the join result and log a hard warning if it timed out:

```python
finally:
    stop_event.set()
    joined = t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not stop within timeout for key=%s — lock release proceeding anyway", lock_key)
    self._release_script(keys=[lock_key], args=[token])
```

This doesn't fully eliminate the race but makes it observable and auditable.

### 3. Unhandled Exceptions Inside the Lock Context Leak the Lock State (Medium)

If `self.db.get_stock()` or `self.db.transaction()` raises an unexpected exception, the `finally` block in `_acquire_lock` will correctly release the lock. However, `reserve_item` has no outer `except Exception` handler, so the exception propagates to the caller as an unhandled exception rather than a `ReservationResult`. For a service method that promises to return a `ReservationResult`, this is a broken contract. Add a broad catch at the bottom of `reserve_item`:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal service error")
```

### 4. `quantity` Has No Upper Bound (Low-Medium)

```python
quantity: int = Field(..., ge=1)
```

There is a lower bound but no upper bound. A caller can submit `quantity=10_000_000` and the service will happily attempt to reserve ten million units. Add a reasonable ceiling:

```python
quantity: int = Field(..., ge=1, le=1000)
```

Adjust the ceiling to match business rules, but it must exist.

---

### 5. `item_id` Length Allows Oversized Redis Keys (Low)

`max_length=128` on `item_id` means the Redis lock key can be up to `len("inventory:lock:") + 128 = 143` bytes. This is fine for Redis, but the pattern should be documented as a deliberate choice. More importantly, if `item_id` ever comes from an external API boundary rather than an internal catalog, 128 characters is generous enough to warrant a comment explaining why that bound was chosen.

---

### 6. Test Suite Gaps (Medium)

The existing tests are well-structured, but the following scenarios are not covered:

**a. DB exception inside the lock — lock must still be released.**
```python
def test_db_exception_releases_lock(service, db, redis_client):
    db.get_stock.side_effect = RuntimeError("DB connection lost")
    with pytest.raises(Exception):  # or assert ReservationResult if #3 is fixed
        service.reserve_item("item_001")
    # Lock release script must still have been called
    service._release_script.assert_called_once()
```

**b. Heartbeat thread actually extends the lock.**
The heartbeat is a daemon thread and the mock returns `1` by default, but no test verifies that `_extend_script` is called after `HEARTBEAT_INTERVAL` elapses. This can be tested by patching `HEARTBEAT_INTERVAL` to a very small value and sleeping past it.

**c. `t.join()` timeout path.**
No test covers the case where the heartbeat thread does not stop in time.

**d. `quantity` upper bound.**
Once the `le` constraint is added, add a test for `quantity=10_000_000`.

---

### 7. Minor: `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants (Low)

These are currently hardcoded. For a government contracting environment where deployment configurations differ between environments (dev, staging, production), these should be injectable via the constructor or drawn from a config object. This is not a blocker but is worth a follow-up ticket.
