Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and largely correct. However, there are several meaningful failure modes and architectural gaps that must be addressed before this is suitable for a production flash sale system under adversarial or high-concurrency conditions. The issues range from a silent data-integrity race in the critical path, to unhandled exceptions that can leave locks stranded, to test coverage that does not actually validate the most important behavioral guarantees.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and DB Write

This is the most serious flaw in the implementation.

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # READ outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE inside transaction
    self.db.create_order(...)
```

The stock read happens **outside** the database transaction. The Redis lock serializes callers *through this service*, but it provides zero protection against:

- A direct database write from another service, admin tool, or migration script that bypasses this lock entirely.
- A Redis failover or network partition between the read and the transaction commit, where the lock is silently lost and another instance proceeds concurrently.

The stock read **must** be inside the transaction boundary, and the `update_stock` call should use an optimistic or conditional update (e.g., `UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s`) so that a concurrent bypass is caught at the database level rather than silently corrupting inventory. The distributed lock is a performance optimization — the database must remain the authoritative source of truth with its own integrity enforcement.

```python
# Correct pattern:
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ...  # rollback is implicit on context exit without commit
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

### 2. Unhandled Exceptions Inside the Lock Context Leave the Lock Stranded

The `finally` block in `_acquire_lock` will execute on any exception, which is good — but consider what happens when `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unexpected exception (network timeout, DB connection drop, etc.):

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # raises DBConnectionError
    # ... exception propagates up through reserve_item
    # LockAcquisitionError is caught, but NOT generic exceptions
```

The `except LockAcquisitionError` block in `reserve_item` only catches lock failures. Any other exception propagates to the caller as an unhandled exception, **and the lock is released** (the `finally` runs), but the caller receives an unhandled exception rather than a `ReservationResult`. This is a broken API contract — the method signature promises to return `ReservationResult` but can raise in practice.

All exceptions inside the lock context must be caught and converted to a `ReservationResult`, or at minimum documented explicitly. For a service method returning a result type, silent raises are a contract violation.

---

### 3. Heartbeat Thread Silently Swallows Lock Loss — No Propagation Mechanism

```python
if result == 0:
    logger.warning("Heartbeat detected lock loss key=%s", lock_key)
    return  # <-- thread exits, main thread has no idea
```

The heartbeat detects that the lock was lost (expired and re-acquired by another caller), logs a warning, and exits. The main thread continues executing the database transaction **without holding the lock**, potentially writing stale data concurrently with another holder. This defeats the entire purpose of the lock.

The lock loss event must be communicated back to the main thread. A simple approach:

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Lock lost during operation key=%s", lock_key)
            lock_lost_event.set()
            return
        ...

# In the main thread, after yield, check before committing:
if lock_lost_event.is_set():
    raise LockLostError(f"Lock lost during operation for {lock_key}")
```

Alternatively, check `lock_lost_event` before the DB transaction commit. The exact mechanism matters less than the guarantee that a lost lock aborts the operation.

## Significant Issues

### 4. `t.join(timeout=...)` Does Not Guarantee Thread Termination

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=...)` returns whether or not the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call, the join times out silently and `_release_script` is called while the heartbeat thread may still be running and about to call `_extend_script`. This creates a race: the release script deletes the key, another caller acquires it, and then the zombie heartbeat extends *their* lock with the old token — which will fail due to token mismatch, but it's a gratuitous race condition.

The join result must be checked:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # proceed with release anyway, but log the anomaly
```

### 5. Retry Backoff Is Miscalibrated for a Flash Sale Workload

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3` and `LOCK_TTL=30s`, the maximum total wait time before giving up is approximately `0.1 + 0.2 + small jitter ≈ 0.35 seconds`. Under a flash sale with hundreds of concurrent requests, most callers will exhaust retries in under half a second and return "service unavailable" while the lock holder is still in a 30-second window.

Either the retry count needs to be much higher, the backoff ceiling needs to be capped at a reasonable value (e.g., 500ms max with full jitter), or the `LOCK_TTL` needs to be dramatically reduced to match the expected operation duration. A 30-second lock TTL is appropriate only if DB operations genuinely take that long, which would be a separate architectural concern. For a typical inventory check + write, 2–5 seconds is more appropriate, which would also make the retry window more useful.

---

### 6. `quantity` Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity validation. A caller can request `quantity=999999999`, which will pass validation, read stock, compute `new_stock = stock - 999999999` (a large negative number if stock is smaller, which is caught), but the absence of a business-rule ceiling is a gap. For a flash sale, a per-request quantity cap (e.g., `le=10`) should be enforced at the model level.

## Test Coverage Gaps

### 7. Heartbeat Behavior Is Not Tested

There are no tests for:
- Lock extension being called when the heartbeat fires.
- Lock loss detection (`result == 0` from `_extend_script`) and its effect on the operation.
- The heartbeat thread being joined before release.

The test `test_lock_release_uses_lua_script` verifies that `redis.delete` is not called directly, which is good, but it does not verify that `_release_script` is called with the correct token, nor that the token used for release matches the token used for acquisition.

### 8. No Test for DB Exception Propagation

There is no test verifying behavior when `db.get_stock()`, `db.update_stock()`, or `db.create_order()` raises an exception. Given issue #2 above, this is a critical missing case.

### 9. `test_reserve_lock_contention` Does Not Test Retry Logic

The test sets `redis_client.set.return_value = False` unconditionally, which tests total lock failure. There is no test for partial contention — e.g., lock acquired on the second or third attempt — which would verify that the retry loop and backoff logic actually function.

## Minor Issues

### 10. Logger Is Module-Level but Service Has No Instance Identifier

In a multi-instance deployment, log lines like `key=inventory:lock:item_001` will appear from multiple pods with no way to correlate them to a specific service instance. Consider including a service instance ID (e.g., hostname or UUID assigned at `__init__`) in structured log fields.

### 11. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These should be constructor parameters with defaults, not hardcoded class attributes. Different deployment environments (test, staging, production) may need different values, and hardcoding them makes the class harder to configure without subclassing.
