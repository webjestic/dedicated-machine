Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness, safety, and architectural issues that must be addressed before this touches production flash sale traffic. The most critical is a TOCTOU race between the stock read and the database write, compounded by a transaction scope that does not include the stock check, meaning the distributed lock is doing work the database should be reinforcing.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

**1. TOCTOU: Stock check is outside the database transaction**

```python
# stock is read HERE, outside the transaction
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # write is HERE
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The distributed lock serializes callers *at the Redis layer*, but if the lock is ever lost mid-operation (network partition, Redis restart, lock expiry under a slow DB), a second caller can acquire the lock, read the same `current_stock`, and both writers will commit. The stock check and the conditional write must be atomic at the database layer — either via a `SELECT ... FOR UPDATE` inside the transaction, or a conditional `UPDATE` with a `WHERE stock >= quantity` guard and a row-level check of affected rows. The Redis lock should be a *performance optimization* (reducing DB contention), not the sole correctness mechanism.

```python
# Safer pattern — move everything inside the transaction
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ...  # or raise, let the context manager rollback
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

---

**2. Heartbeat thread has no way to abort the critical section on lock loss**

```python
if result == 0:
    logger.warning("Heartbeat detected lock loss key=%s", lock_key)
    return   # <-- thread exits, but the main thread keeps executing
```

When the heartbeat detects token mismatch, it logs and exits. The main thread is still inside the `with self._acquire_lock(lock_key):` block, happily proceeding with the DB write — now without holding the lock. This defeats the entire purpose of the heartbeat. You need a shared cancellation signal back to the main thread.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s — signaling abort", lock_key)
            lock_lost_event.set()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In the main thread, check before the DB write:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost during operation for {lock_key}")
```

This check should occur at minimum before entering the database transaction.

---

**3. `t.join()` blocks the release script if the heartbeat is mid-extension**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # can block up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`stop_event.set()` interrupts the `wait()` sleep, so in the normal case the thread wakes immediately and exits. However, if the thread is currently *inside* `self._extend_script(...)` when `stop_event.set()` is called, it will finish that network call before returning — which is fine. The real problem is the `join(timeout=HEARTBEAT_INTERVAL + 1)` — if the thread hangs (e.g., Redis is slow), the release script is delayed by up to 9 seconds, holding the lock open. After `stop_event.set()`, the join timeout should be short (1–2 seconds), and the release should proceed regardless of whether the join completed, since the Lua release script is token-guarded.

### 🟡 Significant — Should Fix Before Merge

---

**4. No handling of Redis exceptions in the heartbeat or lock path**

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not caught anywhere in `_acquire_lock` or `_beat`. A Redis blip during lock acquisition will propagate as an unhandled exception through `reserve_item`, bypassing the `LockAcquisitionError` handler and returning a 500 to the caller. Wrap Redis calls explicitly and map them to `LockAcquisitionError` or a dedicated `ServiceUnavailableError`.

---

**5. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants with no configurability**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8
```

These are baked in at class definition time. For a flash sale service that may run against varying DB latency profiles, these need to be injectable at construction time. A 30-second lock TTL is also aggressive — if the DB transaction takes 25 seconds something is already wrong and the lock should probably be shorter with a faster heartbeat, or the DB operation needs a timeout.

---

**6. `db.transaction()` failure is not handled**

If `self.db.transaction().__exit__` is called with an exception, there is no logging or mapping to a meaningful return value. The exception will propagate up through `reserve_item` as an unhandled error. Add an explicit `except` for database errors and return a `ReservationResult(success=False, message="Reservation failed due to internal error")` rather than letting it surface as a 500.

---

**7. `new_stock` can go negative if `update_stock` is not atomic at the DB layer**

```python
new_stock = current_stock - req.quantity
self.db.update_stock(req.item_id, new_stock)
```

This is a blind write. If the DB layer does not enforce `stock >= 0` as a constraint, and the TOCTOU issue in point #1 manifests, stock can be written as a negative integer. Add a `CHECK (stock >= 0)` constraint at the schema level and/or use a conditional update:

```sql
UPDATE inventory SET stock = stock - %s
WHERE item_id = %s AND stock >= %s
```

Then assert `affected_rows == 1`.

### 🟠 Test Coverage Gaps

---

**8. No test for heartbeat lock-loss signaling**

The most critical new behavior — the heartbeat detecting token mismatch and aborting the main thread — has zero test coverage. Once the `lock_lost_event` mechanism is added, it must be tested: mock `_extend_script` to return `0` and assert the reservation fails safely without writing to the DB.

---

**9. `test_lock_release_uses_lua_script` is testing mock wiring, not behavior**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts that `register_script`'s return value was called once, which is trivially true given the mock setup. It does not verify that the script was called with the correct `keys` and `args`. At minimum:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.approx(ANY)]  # token is dynamic, but key must match
)
```

---

**10. No test for Redis exceptions propagating through `reserve_item`**

Once Redis error handling is added (point #4), there should be a test that mocks `redis_client.set` to raise `redis.exceptions.ConnectionError` and asserts the result is a graceful `ReservationResult(success=False, ...)`.

---

### 🔵 Minor / Nits

---

**11. `retry_attempts` default is 3 but not exposed on `reserve_item`**

Callers have no way to tune retry behavior without subclassing. Consider exposing it or making it a constructor parameter.

---

**12. Log message on lock release fires even if `_release_script` raises**

```python
self._release_script(keys=[lock_key], args=[token])
logger.info("Lock released key=%s", lock_key)
```

The log should be conditional on the script not raising, or moved inside a try/except.

---

**13. `order_id` is generated before the transaction**

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    ...
    self.db.create_order(order_id, ...)
```

This is fine functionally, but if the transaction rolls back and is retried, a new `order_id` should be generated per attempt to avoid any idempotency confusion downstream.
