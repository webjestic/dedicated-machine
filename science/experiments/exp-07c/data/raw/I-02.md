Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, input validation, and structured logging are all present and directionally correct. However, there are several correctness, safety, and architectural issues that must be resolved before this can be approved for a high-concurrency flash sale context. The most critical are a TOCTOU race between the stock read and the DB write, a silent exception swallow in the heartbeat thread, and a lock release that fires even when the lock was lost mid-operation.

---

## Detailed Review

### 🔴 Critical

---

**1. TOCTOU Race: Stock Read Outside the Database Transaction**

```python
# Lock is held, but the read is OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes a value derived from the stale read
    self.db.create_order(...)
```

The stock is read under the Redis lock, but the lock is a *coordination* primitive — it does not make the database read and write atomic at the database level. If any other code path (a non-locking admin adjustment, a bulk cancel, a restock job) touches `update_stock` between the `get_stock` call and the transaction commit, you will silently write a corrupted stock value. The stock read **must** be inside the transaction, and the update must be a conditional decrement, not a blind overwrite.

**Required fix:** Move `get_stock` inside the transaction and use an atomic conditional decrement at the DB layer, e.g.:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

`SELECT FOR UPDATE` (or equivalent) is the correct primitive here. The Redis lock is still valuable for cross-service coordination and reducing DB contention, but it cannot replace row-level locking for correctness.

---

**2. Lock Released Even When the Heartbeat Detected Lock Loss**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])  # always runs
```

The heartbeat thread sets a local flag (`return`) when it detects token mismatch, but that state is never communicated back to the `finally` block. The release script is safe in isolation (it won't delete a foreign token), but the operation that ran *after* lock loss has already written to the database under a lock it no longer held. The system has no way to know this happened, and the caller gets a success response for a write that may have raced with another holder.

**Required fix:** The heartbeat must signal lock loss back to the critical section. Use a shared `threading.Event` (e.g., `lock_lost_event`) that the heartbeat sets on token mismatch, and check it before committing the transaction:

```python
if lock_lost_event.is_set():
    raise LockLostError("Lock expired during operation; transaction rolled back")
```

---

**3. Unhandled Exceptions in the Heartbeat Thread Are Silently Swallowed**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        # What if this raises redis.RedisError, ConnectionError, etc.?
```

Any exception from `_extend_script` (network partition, Redis failover, serialization error) will silently terminate the heartbeat thread. The lock will then expire naturally, and the main thread will continue operating under a lock it no longer holds — with no indication anything went wrong. Daemon threads do not propagate exceptions.

**Required fix:**

```python
def _beat() -> None:
    try:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            try:
                result = self._extend_script(...)
                if result == 0:
                    lock_lost_event.set()
                    return
            except redis.RedisError as e:
                logger.error("Heartbeat extension failed key=%s: %s", lock_key, e)
                lock_lost_event.set()
                return
    except Exception as e:
        logger.critical("Heartbeat thread crashed key=%s: %s", lock_key, e)
        lock_lost_event.set()
```

### 🟠 Significant

---

**4. `t.join(timeout=...)` Does Not Guarantee the Thread Has Stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join(timeout=N)` returns after N seconds *regardless* of whether the thread has exited. If the heartbeat thread is blocked on a slow Redis call, the release script fires while the heartbeat thread is still alive and may attempt an extension concurrently. This is a race between release and extension on the same key.

**Required fix:** Check `t.is_alive()` after join and log a critical warning. Consider a hard timeout with a lock-lost assumption:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical("Heartbeat thread did not terminate cleanly key=%s — lock state unknown", lock_key)
```

---

**5. Exponential Backoff Ceiling Is Missing**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With only 3 attempts this is bounded in practice, but the formula has no cap. If `retry_attempts` is ever increased (configuration drift, subclass override), backoff grows unboundedly. This is a latent operational hazard.

**Required fix:**

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), MAX_BACKOFF)
```

Define `MAX_BACKOFF` as a class constant.

---

**6. `quantity` Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=10_000_000`. This passes validation, proceeds through lock acquisition, and reaches the DB. There is no business-logic ceiling on reservation size. For a flash sale context this is an obvious abuse vector.

**Required fix:** Add a configurable `le=` constraint or a service-level constant:

```python
quantity: int = Field(..., ge=1, le=MAX_RESERVATION_QUANTITY)
```

---

**7. `item_id` Appears in Log Output Without Sanitization**

```python
logger.info("Reserved item_id=%s ...", req.item_id, ...)
```

`item_id` is validated by regex, so injection into the log format string is not possible here. However, the raw value is emitted into structured logs. If downstream log aggregators parse these fields and `item_id` contains something like `AAAA-1111` that collides with an internal identifier scheme, it could cause log correlation issues. This is low severity given the regex, but worth noting for a government contracting context where log integrity is often audited.

---

### 🟡 Minor / Test Coverage Gaps

---

**8. No Test for Heartbeat Lock-Loss Path**

`test_lock_release_uses_lua_script` verifies the happy path, but there is no test that simulates the heartbeat detecting a token mismatch (`result == 0` from `_extend_script`) and verifies that the operation is correctly aborted or flagged. This is the most operationally dangerous failure mode and has zero coverage.

---

**9. No Test for DB Transaction Failure / Rollback**

If `update_stock` or `create_order` raises inside the transaction, the lock is released (correctly), but there is no test verifying the `ReservationResult` in that scenario. The exception will propagate uncaught out of `reserve_item` since there is no `except` clause for DB errors.

**Required fix:** Add a `try/except` around the transaction block and a corresponding test:

```python
except Exception as e:
    logger.error("DB transaction failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error")
```

---

**10. `LOCK_TTL = 30` With No Documented Justification**

30 seconds is a long lock TTL. For a flash sale with high concurrency, a holder that crashes after acquiring the lock will block all other reservations for that item for 30 seconds. The heartbeat makes this less likely in the happy path, but the TTL should be documented with reasoning and ideally configurable at construction time.
