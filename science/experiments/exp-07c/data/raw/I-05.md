Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness, safety, and architectural issues that must be resolved before this can be approved for a high-concurrency flash sale context. The most critical is a TOCTOU race between the stock read and the database write, which the distributed lock does not fully protect against without a corresponding database-level guard.

---

## Detailed Findings

### 🔴 Critical

---

**1. TOCTOU: Stock check is not atomic with the database write**

```python
# Inside the lock:
current_stock = self.db.get_stock(req.item_id)   # READ
# ... heartbeat can lose the lock here under extreme latency ...
new_stock = current_stock - req.quantity
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE (unconditional)
    self.db.create_order(...)
```

The Redis lock serializes *Redis-level* access, but `update_stock` writes an absolute value (`new_stock`) computed from a stale read. If the heartbeat silently loses the lock (logs a warning and returns, but the main thread is never notified), a second caller can acquire the lock, read the same stock value, and both writes will race to commit the same decrement independently — resulting in stock going negative or orders being double-created.

The database write must be a conditional decrement, not an unconditional set:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id
  AND stock >= :quantity
-- then check rows_affected == 1
```

This is your true safety net. The distributed lock is an optimization to reduce DB contention, not a correctness guarantee on its own.

---

**2. Heartbeat lock loss is silent to the main thread**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return   # <-- exits quietly; main thread continues executing under no lock
```

When the heartbeat detects token mismatch (lock expired and re-acquired), it logs and exits — but the main thread is completely unaware. It will continue executing the database transaction believing it holds the lock. This defeats the entire purpose of the heartbeat.

The heartbeat must signal the main thread to abort. A shared flag or a second event is required:

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return

# In the main thread, check before/after the DB transaction:
if lock_lost_event.is_set():
    raise LockAcquisitionError(f"Lock lost during operation for {lock_key}")
```

Ideally the DB transaction itself should be aborted, not just checked before commit.

### 🟠 High Severity

---

**3. `t.join(timeout=...)` does not guarantee the thread has stopped before lock release**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # may time out
    self._release_script(keys=[lock_key], args=[token])
```

If `t.join()` times out (e.g., the heartbeat thread is blocked on a slow Redis call), the release script executes while the heartbeat thread is still alive and may still call `_extend_script`. This creates a window where the lock is released and then re-extended by the zombie heartbeat thread — extending a lock that now belongs to another caller.

The join timeout should be short (e.g., 1–2 seconds), and if it elapses, that must be logged as an error condition, not silently ignored. The daemon flag prevents process hang, but does not prevent the race within the process lifetime.

---

**4. No upper bound on `quantity`**

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can submit `quantity=10_000_000` against an item with 10 units in stock. While this results in a graceful "Insufficient stock" response, it still acquires the distributed lock, hits the database, and serializes all other legitimate requests behind it. In a flash sale context this is a trivial denial-of-service vector.

Add a reasonable upper bound:

```python
quantity: int = Field(..., ge=1, le=100)  # tune to business requirements
```

---

**5. `item_id` is returned in error messages and log output without sanitization**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Even though `item_id` passes the regex, embedding user-supplied input directly into structured response messages is a habit that should be broken at this layer. If this message is ever surfaced to a frontend or an audit log aggregator, it creates a log injection surface. Use a fixed message template and keep the `item_id` in structured log fields only.

---

### 🟡 Medium Severity

---

**6. Retry backoff ceiling is extremely low**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0: ~0.10s, attempt=1: ~0.20s, attempt=2: ~0.40s
```

With 3 attempts and a 30-second lock TTL, a caller that fails all retries in under a second will surface "service unavailable" to the user while the lock holder may still be mid-transaction. Either increase `retry_attempts` or add a cap with a longer ceiling (e.g., up to 2–5 seconds per attempt) to give legitimate lock holders time to complete and release.

---

**7. `db.get_stock` is called outside the database transaction**

```python
current_stock = self.db.get_stock(req.item_id)   # outside transaction
# ...
with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(...)
```

The stock read and the stock write are in separate transaction scopes. Even with the Redis lock, if the database uses read-committed isolation (the default for most RDBMS), a concurrent write from a different code path (e.g., an admin adjustment, a cancellation refund) can slip in between the read and the write. Move `get_stock` inside the transaction, or use `SELECT ... FOR UPDATE` to hold a row-level lock.

---

**8. No Redis connection error handling**

Neither `_acquire_lock` nor the heartbeat catches `redis.RedisError`. A transient Redis failure during `set()` will propagate as an unhandled exception from `reserve_item`, bypassing the `LockAcquisitionError` path and potentially surfacing a raw stack trace to the caller. Wrap Redis calls and map them to `LockAcquisitionError` or a dedicated `ServiceUnavailableError`.

---

### 🔵 Low Severity / Test Coverage Gaps

---

**9. No test for heartbeat lock-loss signaling**

`test_reserve_lock_contention` only tests failed initial acquisition. There is no test that simulates the heartbeat detecting `result == 0` mid-operation and verifying the main thread aborts. Once finding #2 is fixed, this test must exist.

---

**10. `test_lock_release_uses_lua_script` is testing mock wiring, not behavior**

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts that the mock was called, which only confirms the mock fixture is wired correctly. It does not verify the Lua script receives the correct `keys` and `args`. Assert on the call arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.approx_any_str]  # or capture the token
)
```

---

**11. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are module-level constants, not configurable**

These values are hardcoded on the class. In a production deployment, lock TTL should be tunable via environment or configuration injection to allow adjustment without a code change. A 30-second TTL may be appropriate for some DB backends and catastrophic for others.
