Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a meaningful race condition, missing failure handling, and test gaps that would make this unsafe to deploy against a high-concurrency flash sale workload. The issues range from a silent data-integrity hole in the DB transaction to an unhandled exception path that could leave the lock permanently orphaned.

---

## Critical Issues

### 1. Stock read occurs outside the database transaction — TOCTOU race

```python
# CURRENT — stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes a stale value
    self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` is issued outside the DB transaction. If the DB read and the DB write land in different transaction snapshots (which they will under any standard isolation level), a concurrent writer that bypasses the Redis lock — a direct DB write, a migration, a manual correction, a second service instance that lost its lock silently — can produce a negative stock balance. The read and the conditional write must be atomic at the database level.

**Required fix:** Move `get_stock` inside the `with self.db.transaction()` block, and use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) so the check-and-decrement is serialized at the DB layer as well. The Redis lock is a performance optimization to reduce DB contention; it must not be the sole correctness guarantee.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Unhandled exceptions inside the lock context will orphan the lock until TTL expiry

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

The `finally` block in `_acquire_lock` does call `_release_script`, so the lock *will* be released on an exception — that part is correct. The problem is that `reserve_item` catches only `LockAcquisitionError`. Any other exception (DB connection failure, unexpected `RuntimeError`, etc.) propagates to the caller completely unhandled and unlogged at this layer. The caller receives a raw exception instead of a `ReservationResult`, which breaks the stated contract of the method. For a public-facing service method this is a contract violation.

**Required fix:**

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 3. Heartbeat thread join timeout is too short and the join result is never checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # 9 seconds
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns whether or not the thread has actually stopped. The code proceeds to release the lock regardless. If the heartbeat thread is blocked (e.g., on a slow Redis call), the release script fires while the thread is still running and holding a reference to `token`. The thread then wakes up, calls `_extend_script` with the now-deleted (or re-acquired) key, and silently succeeds or fails — either outcome is wrong.

**Required fix:** Check `t.is_alive()` after the join and log a warning. The join timeout itself should be short (e.g., 1–2 seconds) since the stop event is set before joining; the thread should exit at its next `wait()` return, which is near-immediate.

```python
stop_event.set()
t.join(timeout=2)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
self._release_script(keys=[lock_key], args=[token])
```

## Significant Issues

### 4. `db.transaction()` failure is silently swallowed

If `self.db.transaction().__exit__` raises (i.e., the transaction fails to commit), the exception propagates out of the `with self._acquire_lock` block. As noted in issue #2, this is unhandled. More specifically, `update_stock` and `create_order` may have been called but the commit never landed — the caller would receive an unhandled exception with no `ReservationResult`, and there is no rollback signal logged. The DB transaction context manager should handle rollback internally, but the service layer must still handle the propagated exception gracefully.

---

### 5. Retry backoff is too aggressive for a flash sale scenario

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the worst-case cumulative sleep before giving up is:

- Attempt 0 → 0.1s + jitter
- Attempt 1 → 0.2s + jitter
- Attempt 2 → 0.4s + jitter

Total: ~0.7s before raising `LockAcquisitionError`. Under a flash sale with hundreds of concurrent requests, this means each waiting request holds a thread for up to 700ms. With a thread pool of any reasonable size this will cause thread exhaustion. The jitter range (`0.05`) is also too small relative to the base delay to meaningfully spread thundering herd.

**Recommendation:** Reduce base delay, increase jitter range, and make `retry_attempts` configurable at construction time. Consider whether a non-blocking "fail fast" mode is more appropriate for flash sale UX.

---

### 6. Lock key namespace collision risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which is safe. However, there is no Redis key prefix scoping for environment (dev/staging/prod). If this Redis instance is shared across environments or tenants, lock keys will collide. The Redis client should be initialized with a key prefix or the lock key should include an environment/tenant discriminator.

## Test Gaps

### 7. Heartbeat thread behavior is completely untested

The heartbeat is the most operationally complex part of this PR. There are zero tests for:

- Lock extension being called after `HEARTBEAT_INTERVAL` elapses
- Heartbeat detecting lock loss (`result == 0`) and logging the warning
- Thread cleanup on normal exit
- Thread cleanup on exception inside the lock context

The existing mock always returns `1` from the script mock, so the `result == 0` branch is never exercised.

---

### 8. `test_lock_release_uses_lua_script` is testing mock wiring, not behavior

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts that `redis.delete` was not called (trivially true since nothing in the code calls it) and that the script mock was invoked. It does not verify the script was called with the correct `keys` and `args`, does not verify the token matches what was set, and does not test the failure path (script returns 0). This test provides false confidence.

---

### 9. No test for DB transaction failure / rollback path

There is no test covering `self.db.transaction().__exit__` raising an exception. This is the most likely real-world failure mode under load.

---

## Minor Issues

### 10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants, not configurable

These should be constructor parameters with defaults. Different item categories may have legitimately different lock durations, and hardcoded class constants make this impossible to tune without code changes.

### 11. `quantity` field has no upper bound

```python
quantity: int = Field(..., ge=1)
```

A caller can request `quantity=999999`. There should be a reasonable `le=` upper bound appropriate to business constraints.

### 12. Daemon thread swallows Redis errors silently in the heartbeat

```python
result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
```

If `_extend_script` raises a `redis.RedisError` (network partition, Redis failover), the exception propagates out of `_beat`, the thread dies silently, and the lock will expire without renewal. This should be wrapped in a `try/except redis.RedisError` with a logged warning and a retry or graceful degradation path.
