Decision: Request Changes

Summary: The core locking primitives are well-constructed — atomic Lua scripts, scoped heartbeat, NX+EX acquisition, and input validation are all solid. However, there are several correctness, reliability, and security issues that must be resolved before this touches production inventory data. The most critical is a TOCTOU race between the in-memory stock read and the database write, which can cause overselling under concurrent load regardless of the distributed lock.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU Race / Overselling Risk (`reserve_item`)**

The stock check and the write are not atomic with respect to the database. The sequence is:

```python
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
# ... check ...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
```

The Redis lock serializes callers *through this service*, but it does not protect against:
- A separate process or service writing stock directly to the DB.
- A Redis failover or network partition that causes the lock to be silently lost between the read and the write (the heartbeat detects this *eventually*, not instantly).
- Any future code path that bypasses the lock.

The read **must** occur inside the transaction with a pessimistic row-level lock (e.g., `SELECT ... FOR UPDATE` in SQL), or the update must use a conditional decrement (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?`) and check affected rows. The current design gives a false sense of safety.

---

**2. Silent Data Corruption on Heartbeat Lock Loss**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return   # ← exits the thread, but the main flow continues unaware
```

When the heartbeat detects lock loss, it logs and returns — but the main thread is never notified. The `reserve_item` logic will continue executing the DB transaction **after the lock has been lost and potentially re-acquired by another caller**. This is the exact scenario the lock is meant to prevent, and the code silently proceeds through it. A `threading.Event` or similar signal must be checked at the yield point, or the heartbeat must raise into the main thread via a flag checked before the DB write.

---

### 🟠 High

**3. Unhandled Redis Exceptions Propagate as 500s**

`redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and script execution errors are not caught anywhere. A Redis blip during `self.redis.set(...)` or during the Lua script calls will raise an unhandled exception that bypasses the `ReservationResult` return contract entirely. At minimum, a broad `redis.RedisError` catch should be added with a safe fallback result and logging.

**4. DB Exceptions Inside the Lock Are Not Caught**

```python
with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(...)
```

Any exception here propagates out of the `with self._acquire_lock(lock_key):` block, through the `finally` clause (which does correctly release the lock — good), but then surfaces as an unhandled exception to the caller. The method's documented return type is `ReservationResult`; callers depending on that contract will get an exception instead. Wrap DB operations in a `try/except` and return a failure `ReservationResult`.

**5. Lock Key Namespace Collision Risk**

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which is good, but there is no tenant, environment, or service namespace prefix. If this Redis instance is shared with other services (common in government/multi-tenant deployments), a key like `inventory:lock:admin` from another service could collide. Add a service-scoped prefix, e.g., `obotix:inventory:lock:{item_id}`.

---

### 🟡 Medium

**6. Heartbeat Thread Join Timeout Is Too Tight**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
```

After `stop_event.set()`, the heartbeat thread will exit on its next `wait()` call, which returns immediately when the event is set — so in the normal case this is fine. However, if the thread is blocked inside `self._extend_script(...)` (e.g., Redis is slow), the join will time out and the main thread will proceed to release the lock while the heartbeat thread is still mid-script execution. This is a benign race in practice but should be documented, or the extend call should be given an explicit socket timeout.

**7. `retry_attempts` Default Is Not Configurable at the Service Level**

The retry count is hardcoded as a parameter default (`retry_attempts: int = 3`) on a private method. For a flash sale scenario — which is the entire stated purpose of this service — 3 attempts with exponential backoff tops out at roughly 0.35 seconds of wait time. Under genuine flash sale load this will produce a flood of `LockAcquisitionError` failures. This should be configurable via the constructor or a named constant, and the product team should be consulted on acceptable wait budgets.

**8. `remaining_stock` Leaked on Item-Not-Found Path**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

This is fine. But the insufficient-stock path returns `remaining_stock=current_stock`, which leaks live inventory levels to the caller. Depending on the API surface this is exposed through, this may be an information disclosure issue. Confirm with the security team whether callers should receive exact stock counts.

---

### 🟢 Minor / Positive Notes

- Lua scripts for atomic release and extension are correctly implemented.
- `uuid4` token prevents accidental release by a different caller.
- Pydantic validation with regex pattern is appropriate and the path traversal test (`../../admin`) is a good catch.
- `daemon=True` on the heartbeat thread is correct — prevents the thread from blocking interpreter shutdown.
- The test for Lua-atomic release (`redis_client.delete.assert_not_called()`) is a nice explicit contract test.
- `stop_event.wait()` used as a timer rather than `time.sleep()` is the right pattern.

---

## Required Before Approval

1. Move `get_stock` inside the transaction with a row-level lock, **or** replace the update with a conditional atomic decrement checked by affected rows.
2. Signal the main thread when the heartbeat detects lock loss and abort the DB write.
3. Catch `redis.RedisError` and DB exceptions; return `ReservationResult(success=False, ...)` rather than propagating.
4. Namespace the lock key with a service/environment prefix.
