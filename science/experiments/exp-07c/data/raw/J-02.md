Decision: Request Changes

Summary: The core distributed locking logic is well-constructed — atomic Lua scripts, heartbeat scoping, NX+EX acquisition, and input validation are all sound. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a TOCTOU race on stock reads to unhandled exceptions that could silently corrupt inventory state.

---

## Detailed Review

### 🔴 Critical — Must Fix

**1. TOCTOU Race Between Stock Read and DB Transaction**

```python
# Line ~90-100 in reserve_item
current_stock = self.db.get_stock(req.item_id)  # read outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
```

The stock read happens *outside* the database transaction. If the DB supports concurrent writers (or if the lock is ever lost between the read and the transaction commit), a stale read can cause inventory to go negative. The read must occur *inside* the transaction, and the update should use an optimistic or conditional write (e.g., `UPDATE ... WHERE stock >= quantity AND stock = :expected`), not a blind set.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

**2. Unhandled Exceptions Inside the Lock Context Leak State**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

Any exception from `get_stock`, `update_stock`, or `create_order` (network error, DB timeout, etc.) propagates out of `reserve_item` entirely uncaught. The lock *will* be released by the `finally` block — that part is fine — but the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the contract of the method and could expose internal stack traces to upstream callers.

```python
# Add inside the lock context:
except (DatabaseError, redis.RedisError) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please retry.")
```

---

**3. Heartbeat Thread Silently Swallows Lock Loss — No Propagation Mechanism**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # ← exits thread, but the main thread has no idea
```

When the heartbeat detects lock loss (`result == 0`), it logs a warning and exits — but the main thread continues executing the DB transaction under the false assumption it still holds the lock. This is a correctness violation: another process may have acquired the lock and begun its own reservation. You need a shared flag (e.g., a `threading.Event`) that the main thread checks after acquiring the lock context, or the heartbeat should set a `lock_lost_event` that the main thread polls at critical points.

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost.set()
            return

# In reserve_item, after get_stock:
if lock_lost.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

### 🟡 Moderate — Should Fix

**4. `quantity` Field Has No Upper Bound**

```python
quantity: int = Field(..., ge=1)
```

There is no `le` constraint. A caller can submit `quantity=999999999`, which passes validation, acquires the lock, and attempts to decrement stock by an arbitrarily large number. This is both a logic concern and a potential abuse vector in a flash sale context.

```python
quantity: int = Field(..., ge=1, le=100)  # or whatever the business rule is
```

---

**5. `item_id` Leaked in Error Message Returned to Caller**

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Even though `item_id` is validated, reflecting internal identifiers in API-facing error messages is an information disclosure pattern. Use a generic message; log the specific ID server-side.

```python
logger.warning("Item not found item_id=%s", req.item_id)
return ReservationResult(success=False, message="Item not found")
```

---

**6. Heartbeat Thread Join Timeout Is Too Short and Silent on Failure**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # = 9 seconds
```

If the heartbeat thread is blocked (e.g., Redis is slow), `join()` returns silently after the timeout and the lock release proceeds anyway. There is no check on whether the thread actually terminated. This is a minor resource leak but should be logged.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

### 🔵 Minor — Recommended

**7. Test Coverage Gaps**

- No test for the heartbeat lock-loss path (`result == 0` from extend script).
- No test verifying behavior when `db.transaction()` raises an exception mid-flight.
- No test for the `quantity` upper-bound once added.
- `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert the correct `keys` and `args` were passed — the assertion is too weak to catch a regression.

**8. Magic Numbers Should Be Named Constants or Config**

`retry_attempts=3` is hardcoded in the method signature and not configurable at the service level. Consider making it a constructor parameter alongside `LOCK_TTL` and `HEARTBEAT_INTERVAL` for testability and operational tuning.
