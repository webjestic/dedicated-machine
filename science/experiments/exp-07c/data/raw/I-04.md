Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness, safety, and architectural issues that range from a silent data integrity hole to a thread lifecycle race that could cause lock theft in production. None of these are cosmetic; they need to be resolved before this touches a flash sale workload.

---

## Critical Issues

### 1. Stock Read Occurs Outside the Database Transaction — TOCTOU Race

This is the most serious flaw in the implementation. The Redis lock serializes callers, but the stock read and the write are not atomic with respect to the database itself.

```python
# CURRENT — stock read is OUTSIDE the transaction
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← read here
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← write here
        self.db.create_order(...)
```

If the Redis lock is ever lost between the read and the write — due to heartbeat failure, Redis restart, network partition, or a clock skew eviction — a second process can acquire the lock, read the same stale stock value, and both processes will decrement from the same baseline. You will oversell.

The stock read must be inside the transaction, and the update should use a conditional decrement (optimistic or pessimistic DB-level lock) rather than a blind `update_stock(item_id, new_stock)`.

```python
# REQUIRED pattern — read and write are atomic at the DB level
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock is a performance optimization (reduces DB contention under flash sale load). The database transaction with a row-level lock is the correctness guarantee. Right now you have the optimization but not the guarantee.

### 2. Heartbeat Thread Is Joined Before Lock Release — Exposes a Lock Theft Window

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # blocks up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` does not raise if the thread fails to terminate within the timeout. If the heartbeat thread is blocked (network stall to Redis, GIL contention, anything), the join silently returns after 9 seconds. The lock's TTL is 30 seconds. During those 9 seconds the lock is still held by this process, but the code has not yet released it. That is acceptable.

The actual bug is the inverse scenario: if the heartbeat thread detected a lock loss (`result == 0`) and returned early, the lock has already been re-acquired by another process. The `finally` block then unconditionally calls `_release_script`, which is safe only because the Lua script checks the token. But the join still waits up to 9 seconds for a thread that may have exited immediately, adding unnecessary latency to every lock release on a contested key.

More importantly: **there is no signal back to the main thread that the lock was lost mid-operation.** The heartbeat silently logs a warning and returns. The `reserve_item` code continues executing the database transaction against a lock it no longer holds, then returns `success=True` with an order that may conflict with another concurrent reservation.

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return   # ← main thread has no idea this happened
```

You need a shared flag or event that the main thread checks after the critical section, or the heartbeat should set an event that the context manager raises on exit.

```python
lock_lost_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost_event.set()
            return

# In the finally block:
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if lock_lost_event.is_set():
        raise LockAcquisitionError(f"Lock was lost mid-operation for {lock_key}")
    self._release_script(keys=[lock_key], args=[token])
```

## Significant Issues

### 3. Retry Backoff Is Miscalibrated for Flash Sale Workloads

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → 0.10–0.15s
# attempt=1 → 0.20–0.25s
# attempt=2 → 0.40–0.45s
```

With `retry_attempts=3`, a caller can wait up to ~0.85 seconds before failing. Under a flash sale with thousands of concurrent requests, every caller that doesn't win the lock on the first attempt is sleeping. The jitter ceiling of 0.05s is too narrow — it provides almost no thundering herd protection when hundreds of waiters all backed off for the same duration and wake within a 50ms window.

Widen the jitter to at least `random.uniform(0, backoff)` (full jitter), and consider whether 3 attempts is the right default for this use case or whether that should be caller-configurable.

```python
backoff = (2 ** attempt) * 0.1
jitter = random.uniform(0, backoff)  # full jitter
time.sleep(backoff + jitter)
```

### 4. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Module-Level Magic Numbers With No Enforcement of Their Relationship

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # seconds — fires at TTL/3
```

The comment says "fires at TTL/3" but `30 / 3 = 10`, not `8`. This is a minor documentation inconsistency, but it points to a deeper problem: if someone changes `LOCK_TTL` without updating `HEARTBEAT_INTERVAL`, the heartbeat may fire too infrequently and allow the lock to expire. This relationship should be enforced in code.

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # enforced relationship
```

Or make them constructor parameters with a validation assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, \
    "Heartbeat interval must be less than half the lock TTL"
```

---

### 5. `db.update_stock` Accepts an Absolute Value — Dangerous API

```python
self.db.update_stock(req.item_id, new_stock)
```

Passing an absolute stock value rather than a delta (`-quantity`) means any race at the DB layer (even outside this service) will silently overwrite whatever stock value was set by a concurrent writer. A delta-based update (`UPDATE inventory SET stock = stock - %s WHERE item_id = %s AND stock >= %s`) is atomic at the SQL level and eliminates the need for the application to compute `new_stock` at all. If `db` is an abstraction you control, change its interface. If it is external, document this constraint explicitly and add a comment explaining why this is only safe under the Redis lock.

## Minor Issues

### 6. Test for Lua Atomic Release Is Structurally Weak

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts that `_release_script` was called once, but it does not assert the arguments. A refactor that calls the script with the wrong key or a hardcoded token would pass this test. Assert the key structure at minimum:

```python
call_args = service._release_script.call_args
assert call_args.kwargs["keys"][0].startswith("inventory:lock:")
```

### 7. No Test Covering Lock-Lost-Mid-Operation Scenario

Once Issue #2 is fixed (lock loss propagation), there is no test that verifies the service returns a failure result (or raises) when the heartbeat detects token mismatch during the DB transaction. This is the most dangerous failure mode and it needs a test.

### 8. `t.join(timeout=...)` Return Value Is Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join()` returns `None` whether the thread exited cleanly or the timeout elapsed. You cannot tell the difference. Use `t.is_alive()` after the join to detect a stuck heartbeat thread and log an error.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```
