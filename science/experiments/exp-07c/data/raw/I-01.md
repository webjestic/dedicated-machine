Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, input validation, and structured logging are all present and largely correct. However, there are several correctness, safety, and architectural issues that must be addressed before this can be approved for a government-grade flash sale system operating under concurrent load. The most critical concerns involve a TOCTOU race on stock reads, an unhandled exception path that silently holds the lock, and a heartbeat thread that can outlive the lock release in a failure scenario.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Outside the Database Transaction

```python
# CURRENT — stock read is outside the transaction boundary
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← read here
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← write here
        self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` is issued outside the database transaction. If the database has any read-committed or weaker isolation, a concurrent process that bypasses this service (a batch job, admin tool, direct DB write, another service instance that lost its lock silently) can mutate stock between your read and your write. The stock read and the conditional write must be atomic at the database level, not just at the Redis level. Move `get_stock` inside `self.db.transaction()`, or use a `SELECT ... FOR UPDATE` / optimistic version check inside the transaction.

```python
# RECOMMENDED
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock should be treated as a performance optimization to reduce DB contention, not as the sole correctness guarantee.

### 2. Bare `except` Gap — Unhandled Exceptions Suppress the Lock Release Path Silently

```python
try:
    with self._acquire_lock(lock_key):
        ...
        return ReservationResult(...)

except LockAcquisitionError as e:
    ...
    return ReservationResult(success=False, ...)
```

Only `LockAcquisitionError` is caught at the outer level. Any other exception raised inside the `with self._acquire_lock(lock_key):` block — a `redis.RedisError` from the heartbeat's extend call, a DB connection failure, an unexpected `AttributeError` — will propagate up through the `contextmanager`'s `finally` block, which is correct for the lock release itself. However, the caller of `reserve_item` receives an unhandled exception rather than a `ReservationResult`. For a service API this means the caller must handle raw exceptions, which breaks the contract implied by the return type annotation. Add a broad catch at the outer level:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal service error")
```

This is not just ergonomics — in a flash sale context, an unhandled exception that propagates through a web framework may expose stack traces or internal state.

### 3. Heartbeat Thread Join Timeout Is Insufficient Under Lock Loss

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

The `join` with a timeout is correct in principle, but there is no check of `t.is_alive()` after the join. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis replica failover), `join` returns after the timeout and `_release_script` is called regardless — which is the right behavior. But the thread is still running in the background and will eventually call `_extend_script` after the lock has already been released and potentially re-acquired by another caller. Since the Lua script guards on token match this won't corrupt the new owner's lock, but it does mean you have a zombie thread holding a reference to `self` and making Redis calls. You should log a warning when `t.is_alive()` is still `True` after join:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error(
            "Heartbeat thread did not terminate in time key=%s — possible Redis connectivity issue",
            lock_key
        )
    self._release_script(keys=[lock_key], args=[token])
```

## Significant Issues

### 4. `LOCK_TTL = 30` Is Dangerously Long for a Flash Sale

A 30-second lock TTL means a crashed process holds the lock for up to 30 seconds before it expires. In a flash sale, that is an eternity. The heartbeat interval of 8 seconds (`TTL / 3`) is a reasonable ratio, but the absolute TTL needs to be justified against your actual p99 DB transaction latency. If your DB transactions complete in under 500ms, a TTL of 5–10 seconds with a heartbeat of 1–2 seconds is far more appropriate. Document the rationale for 30 seconds or reduce it.

---

### 5. Retry Backoff Is Too Aggressive for Flash Sale Workloads

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
# attempt=0 → 0.10–0.15s
# attempt=1 → 0.20–0.25s
# attempt=2 → 0.40–0.45s
```

With only 3 attempts and a maximum total wait of ~0.85 seconds, a request that arrives during a 30-second lock hold will fail immediately. This is probably intentional for flash sales (fail fast), but the jitter range of `0–0.05s` is too narrow to prevent thundering herd behavior when hundreds of goroutines are retrying simultaneously. Widen the jitter to at least `0–0.1 * (2 ** attempt)` (full jitter pattern). Also, `retry_attempts=3` is hardcoded in the signature default but not configurable at the service level — expose it as a constructor parameter.

---

### 6. `db.get_stock` Return Value Contract Is Unverified

```python
current_stock = self.db.get_stock(req.item_id)

if current_stock is None:
    ...
if current_stock < req.quantity:
    ...
```

There is no guard against `current_stock` being a negative integer (a pre-existing data integrity issue) or a non-integer type. If `get_stock` returns a `Decimal` from an ORM or a string from a misconfigured query, the comparison `current_stock < req.quantity` will either raise a `TypeError` or produce incorrect results. Add an explicit type and range assertion:

```python
if not isinstance(current_stock, int) or current_stock < 0:
    logger.error("Invalid stock value for item_id=%s: %r", req.item_id, current_stock)
    return ReservationResult(success=False, message="Internal data error")
```

## Test Coverage Gaps

### 7. Heartbeat Behavior Is Not Tested

The heartbeat thread is the most complex and failure-prone component of this implementation. There are zero tests for:

- Lock extension being called after `HEARTBEAT_INTERVAL` elapses
- Heartbeat detecting token mismatch (`result == 0`) and logging the warning
- Thread termination on `stop_event.set()`
- Zombie thread behavior when `join` times out

These require either real `threading` + `time` (with mocked Redis) or injecting a fake clock. The current test suite mocks everything at a level that makes the heartbeat thread a no-op, which means the most critical correctness property of this service is completely untested.

### 8. `test_lock_release_uses_lua_script` Is Fragile

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test asserts `_release_script` is called exactly once, but `_release_script` is also the mock returned by `register_script`, which is shared between both `_release_script` and `_extend_script` on the service instance. If the heartbeat fires during this test (unlikely but possible given thread scheduling), the assert will fail non-deterministically. Separate the mocks for release and extend scripts.

### 9. No Test for DB Transaction Rollback on Failure

There is no test verifying that if `db.create_order` raises an exception, `db.update_stock` is rolled back. This is the entire point of the `with self.db.transaction()` block and it is untested.

## Minor Issues

### 10. `lock_key` Logged at `INFO` on Release But Not on Acquisition

Acquisition is logged only on contention (`WARNING`). Successful acquisition should also be logged at `DEBUG` for operational traceability, especially given this is a security-sensitive inventory operation.

### 11. `order_id` Generated Before Transaction Commit

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is acceptable, but if the transaction rolls back and is retried externally, the same `order_id` will not be reused — a new one will be generated on the next call.
