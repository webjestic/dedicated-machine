Decision: Request Changes
Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness bugs, a meaningful race condition, a silent failure mode that could corrupt inventory state, and test coverage gaps that would not catch these issues in production. These must be resolved before this code is approved for a high-concurrency flash sale context.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and DB Write

This is the most serious flaw. The lock is correctly acquired, but the stock decrement is computed in application memory and then written back as an absolute value:

```python
current_stock = self.db.get_stock(req.item_id)   # read
new_stock = current_stock - req.quantity           # compute in app
self.db.update_stock(req.item_id, new_stock)       # write absolute value
```

If `update_stock` is not itself a conditional atomic operation (e.g., `UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s`), then any failure in the Redis lock — clock skew causing premature expiry, a Redis failover, a network partition — allows two processes to read the same stock value, both compute a decrement, and both write back, resulting in **silent inventory corruption with no error surfaced to the caller**.

The database write must be a conditional decrement:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id AND stock >= :quantity
```

And the return value of that update must be checked. The application-layer read is fine for the early-exit stock check, but the authoritative decrement must happen atomically inside the database. The lock is a performance optimization to reduce contention, not a correctness guarantee on its own.

---

### 2. Lock Release After Heartbeat-Detected Loss Is Incorrect

In `_acquire_lock`, when the heartbeat detects a token mismatch (`result == 0`), it logs a warning and returns — but the `finally` block still unconditionally calls `_release_script`:

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])  # always runs
```

The Lua release script is token-gated, so it won't delete a foreign lock — that part is safe. However, the code continues executing the business logic inside the `with` block after lock loss has been detected, with no mechanism to abort the in-progress reservation. The heartbeat thread returning early is a silent signal that goes completely unobserved by the caller. This needs a shared flag or event that the business logic can observe, or the heartbeat loss must propagate as a raised exception into the `with` block.

---

### 3. `t.join()` Timeout Silently Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join()` with a timeout does not raise if the thread is still alive after the timeout expires. If the heartbeat thread is blocked on a slow Redis call, the release script will execute while the heartbeat is potentially mid-extension — a narrow but real race. The join result must be checked:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

## Significant Issues

### 4. Heartbeat Interval Is Too Aggressive Relative to TTL

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 8`. The first heartbeat fires at 8 seconds, which is TTL/3.75 — acceptable. But the comment says "fires at TTL/3," which would be 10 seconds. This is a documentation inconsistency that signals the values were not derived from a principled formula. More importantly, there is no enforcement that `HEARTBEAT_INTERVAL < LOCK_TTL / 2`. If someone changes `LOCK_TTL` to 10 and leaves `HEARTBEAT_INTERVAL` at 8, the lock will expire before the first heartbeat fires. Add a constructor assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
    "HEARTBEAT_INTERVAL must be less than half of LOCK_TTL"
)
```

---

### 5. Retry Backoff Is Insufficient for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum total wait before failing is approximately `0.1 + 0.2 + 0.05 = 0.35` seconds plus jitter. Under genuine flash sale load — hundreds of concurrent requests — this will produce a thundering herd on the first retry wave, and most callers will fail immediately. The jitter ceiling of `0.05s` is negligible. Either increase `retry_attempts` and jitter range, or expose them as constructor parameters so they can be tuned per deployment. The current defaults will cause mass `LockAcquisitionError` failures under the exact load this service is designed for.

---

### 6. DB Transaction Does Not Cover the Stock Read

The stock read happens outside the transaction:

```python
current_stock = self.db.get_stock(req.item_id)  # outside transaction
# ...
with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(...)
```

If the database uses read-committed isolation (the default in PostgreSQL and MySQL), the stock read may observe a committed value that becomes stale by the time the transaction opens. The read should occur inside the transaction with a `SELECT ... FOR UPDATE` or equivalent to ensure the snapshot is consistent with the write.

---

## Minor Issues

### 7. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity. A caller can request `quantity=999999` and the only check is whether stock is sufficient. For a flash sale context, a per-request quantity cap (e.g., `le=10`) is standard business logic and should be enforced at the model layer, not left to the caller.

---

### 8. Test for Lock Release Does Not Verify Script Arguments

```python
def test_lock_release_uses_lua_script(service, redis_client):
    service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
    service._release_script.assert_called_once()
```

This test verifies the script was called but not *what it was called with*. A regression that passes the wrong token or wrong key would not be caught. It should assert:

```python
service._release_script.assert_called_once_with(
    keys=["inventory:lock:item_001"],
    args=[pytest.approx(ANY)]  # token is dynamic, but key and structure must be correct
)
```

---

### 9. No Test for Heartbeat Lock-Loss Path

The heartbeat detecting `result == 0` and returning early is a critical code path with no test coverage. Given that this is the safety net for lock expiry during slow DB operations, it must be tested explicitly — including verifying that the business operation is aborted or flagged appropriately once that path is resolved per issue #2.

---

### 10. No Redis Connection Error Handling

If `self.redis.set(...)` raises a `redis.RedisError` (connection refused, timeout, etc.), it propagates uncaught out of `reserve_item` as an unhandled exception. Given that Redis is a hard dependency, connection failures should be caught and returned as a `ReservationResult(success=False, ...)` with appropriate logging, consistent with how `LockAcquisitionError` is handled.

---
