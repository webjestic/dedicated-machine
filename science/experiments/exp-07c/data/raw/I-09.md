Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all present and largely correct. However, there are several correctness, safety, and architectural concerns that must be addressed before this touches production flash-sale traffic, where the blast radius of a bug is immediate and measurable. The issues range from a silent data-integrity gap in the stock-read/transaction boundary to thread-safety problems and incomplete test coverage of failure modes.

---

## Findings

### 🔴 Critical

**1. Stock read occurs outside the database transaction — TOCTOU window**

```python
# Line ~90 — stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():          # transaction opens here
    self.db.update_stock(...)        # write happens here
    self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` executes outside the database transaction. If the database itself is accessed by any other path — a backoffice tool, a migration, a second service node that ignores the Redis lock, or a Redis failover that causes the lock to be silently lost — the stock value read is stale by the time the write commits. The correct pattern is to move `get_stock` inside the `with self.db.transaction()` block and, critically, use a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) so the database itself enforces the invariant. The Redis lock is a performance optimization, not a substitute for database-level consistency.

```python
# Recommended structure
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None: ...
    if current_stock < req.quantity: ...
    self.db.update_stock(...)
    self.db.create_order(...)
```

**2. Heartbeat thread has no mechanism to abort the in-flight operation on lock loss**

```python
if result == 0:
    logger.warning("Heartbeat detected lock loss key=%s", lock_key)
    return   # ← thread exits silently
```

When the heartbeat detects token mismatch it logs and exits, but the main thread continues executing — it will complete the database transaction and commit a stock decrement under a lock it no longer holds. Another process has already re-acquired the lock and may be reading stock concurrently. This is the exact race condition the lock is meant to prevent. The heartbeat must have a shared cancellation signal back to the main thread (e.g., a second `threading.Event` — `lock_lost_event`) that the main thread checks before committing, or raises an exception into the main thread via a flag checked at the transaction boundary.

```python
# Sketch
lock_lost = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        ...

# In reserve_item, before committing:
if lock_lost.is_set():
    raise LockLostError(f"Lock lost mid-operation for {lock_key}")
```

### 🟠 Major

**3. `t.join(timeout=...)` result is not checked — lock may be released with heartbeat still running**

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # ← return value ignored
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns `None` regardless of whether the thread actually stopped. If the heartbeat thread is blocked on a slow Redis call and does not exit within the timeout, `_release_script` fires while the heartbeat thread is still alive and holding a reference to `token`. The heartbeat could then attempt an extend on a key that has just been deleted (or re-acquired by another caller). Check `t.is_alive()` after the join and log a critical alert if the thread did not terminate cleanly.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.critical("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do NOT proceed silently — this is a sentinel for a deeper problem
```

---

**4. Exponential backoff ceiling is missing — retry storm risk under sustained contention**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With only 3 attempts the maximum backoff is `0.4s + jitter`, which is fine at this retry count. However, `retry_attempts` is a parameter with no enforced upper bound. A caller passing `retry_attempts=20` produces a backoff of `~52 seconds` on the final attempt — effectively a hung request. Cap the backoff:

```python
backoff = min((2 ** attempt) * 0.1 + random.uniform(0, 0.05), 2.0)
```

---

**5. `db` abstraction is entirely opaque — no interface contract enforced**

The `InventoryService` constructor accepts any `db` object and calls `.get_stock()`, `.transaction()`, `.update_stock()`, `.create_order()` on it with no type annotation, no Protocol/ABC, and no documented contract. In a government-grade system this is a maintenance and auditability liability. Define a `typing.Protocol` or ABC:

```python
from typing import Protocol, ContextManager

class InventoryDB(Protocol):
    def get_stock(self, item_id: str) -> Optional[int]: ...
    def get_stock_for_update(self, item_id: str) -> Optional[int]: ...
    def update_stock(self, item_id: str, new_stock: int) -> None: ...
    def create_order(self, order_id: str, item_id: str, quantity: int) -> None: ...
    def transaction(self) -> ContextManager: ...
```

### 🟡 Minor / Improvements

**6. Tests mock away the exact behavior under scrutiny**

`test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but the mock returns `1` unconditionally, meaning the test never exercises what happens when release returns `0` (lock already expired at release time). There is also no test for:
- Lock lost mid-operation (heartbeat detects token mismatch while DB transaction is in flight)
- `db.transaction()` raising an exception — does the lock still release cleanly? (It should, via `finally`, but this should be explicitly verified)
- Heartbeat thread extend being called the correct number of times for a long-running operation

**7. `quantity` field has no upper bound**

```python
quantity: int = Field(..., ge=1)
```

A request for `quantity=2_000_000_000` passes validation. For a flash sale context this is almost certainly unintentional and could mask a client bug or abuse attempt. Add a `le=` bound appropriate to business constraints.

**8. Lock key namespace is predictable and flat**

`inventory:lock:{item_id}` is fine for single-environment deployments, but there is no environment or tenant namespace. If this Redis instance is shared across staging and production (common in early-stage government deployments), a staging lock will block a production reservation for the same item ID. Recommend: `inventory:lock:{env}:{item_id}` injected at construction time.

**9. `daemon=True` on the heartbeat thread is correct but deserves a comment**

Daemon threads are silently killed on interpreter shutdown. This is the right choice here — you do not want a heartbeat thread to keep the process alive after a SIGTERM — but it should be documented explicitly so a future maintainer does not change it thinking it is an oversight.

Items 1 and 2 are blocking. The current code can silently double-decrement stock or commit under a lost lock, which in a flash sale context means either overselling or phantom inventory loss — both are unacceptable outcomes.
