Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all appropriate choices for this problem domain. However, there are several correctness flaws, a critical race condition, missing error handling paths, and test gaps that must be addressed before this is suitable for a production flash sale system where inventory errors have direct financial consequences.

---

## Critical Issues

### 1. Race Condition: Stock Read Outside the Database Transaction

This is the most serious flaw in the implementation.

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

new_stock = current_stock - req.quantity

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers at the application layer, but the stock read and the stock write are not atomic at the database layer. If the database transaction fails and is retried, or if a separate process modifies stock outside this lock (admin tooling, bulk imports, refunds, cancellations), the `new_stock` value computed from the stale read will be written unconditionally. This can produce negative inventory.

**Required fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` or equivalent pessimistic row lock, or use a conditional `UPDATE` with a `WHERE stock >= quantity` guard and check affected rows. The Redis lock alone is not sufficient if anything else can touch the stock row.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Lock Released Even When Heartbeat Detected Lock Loss

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # <-- heartbeat exits, but lock_lost is never communicated
        ...

# In finally:
stop_event.set()
t.join(...)
self._release_script(keys=[lock_key], args=[token])  # <-- always runs
```

When the heartbeat detects token mismatch (lock expired and re-acquired by another caller), the `finally` block still calls `_release_script`. The Lua script will correctly return `0` because the token no longer matches — so it will not delete the new owner's lock. This part is safe.

However, the critical problem is that **the main thread has no idea the lock was lost**. The code inside the `with self._acquire_lock(lock_key):` block continues executing and will commit a database transaction under the false assumption that it still holds the lock. This defeats the entire purpose of the distributed lock.

**Required fix:** Use a shared flag or event to communicate lock loss back to the main thread, and raise an exception from within the context manager body.

```python
lock_lost = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            lock_lost.set()
            return
        ...

# After yield, or checked at critical points:
if lock_lost.is_set():
    raise LockAcquisitionError(f"Lock expired during operation for {lock_key}")
```

This is non-trivial to implement cleanly with a `contextmanager` since the yield has already returned control to the caller. The most robust approach is to check `lock_lost` explicitly after the DB transaction, or to restructure so the heartbeat can interrupt via a callback or exception injection pattern.

## Significant Issues

### 3. No Exception Handling for Database Failures Inside the Lock

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `get_stock`, `update_stock`, `create_order`, or the transaction commit raises any exception, it propagates out of `_acquire_lock`'s `try/yield` block, hits the `finally`, and the lock is released. That part is correct. But the exception then propagates all the way out of `reserve_item` as an unhandled exception — the caller receives a raw database exception rather than a `ReservationResult`.

**Required fix:** Catch database/unexpected exceptions inside `reserve_item` and return a structured error result. Log the full traceback.

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 4. Heartbeat Thread Join Timeout Is Silently Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise if the timeout expires — it returns silently. If the heartbeat thread is somehow stuck (blocked on a slow Redis call), the join returns after 9 seconds and the release script runs anyway, with the thread still alive and potentially calling `_extend_script` after release. This is a thread leak and a logical race.

**Required fix:** Check `t.is_alive()` after join and log a warning. Consider whether the heartbeat's Redis call itself needs a timeout.

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s — possible thread leak", lock_key)
```

### 5. HEARTBEAT_INTERVAL Relative to LOCK_TTL Is Fragile as a Class Constant

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # seconds — fires at TTL/3
```

The comment says "fires at TTL/3" but `30/3 = 10`, not `8`. This is a documentation error, but more importantly, these two values are independent constants with an implicit coupling. If someone changes `LOCK_TTL` without updating `HEARTBEAT_INTERVAL`, the heartbeat fires too infrequently and the lock expires. This is a latent maintenance hazard.

**Required fix:** Derive `HEARTBEAT_INTERVAL` from `LOCK_TTL`.

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # Must be well under TTL
```

---

### 6. `quantity` Field Has No Upper Bound

```python
quantity: int = Field(..., ge=1)
```

There is no maximum quantity validation. A caller can request `quantity=10_000_000` against a stock of `10_000_001` and the reservation will succeed. For a flash sale system, this is an obvious abuse vector and likely a business logic error.

**Required fix:** Add a configurable `le` constraint appropriate to the business domain, or validate against a per-item maximum purchase limit.

## Minor Issues

### 7. Retry Backoff Ceiling Is Very Low

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum backoff before the third attempt is `(2^2) * 0.1 + 0.05 = 0.45s`. Under a flash sale with thousands of concurrent requests, this is reasonable for low latency, but the jitter range (`0–0.05s`) is very narrow relative to the base delay, which reduces its effectiveness at spreading thundering herd. Consider widening the jitter to `random.uniform(0, backoff * 0.5)` or similar.

### 8. `item_id` Leaked in Error Message

```python
return ReservationResult(
    success=False,
    message=f"Item {req.item_id} not found"
)
```

Returning the item ID in an error message is low risk here since the caller provided it, but in a government contracting context, consider whether API responses should echo back user-supplied identifiers. At minimum, be consistent — other error messages do not echo inputs.

### 9. `order_id` Generated Before Transaction Commits

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is fine functionally, but if the transaction rolls back and is retried externally, a new `order_id` will not be generated. This is not a bug given the current code structure, but it is worth noting if retry logic is ever added.

## Test Gaps

### 10. No Test for Lock-Loss-During-Operation Scenario

There is no test verifying that if the heartbeat detects lock loss, the reservation does not complete. Once issue #2 is fixed, this case must be tested.

### 11. No Test for Database Transaction Failure

No test covers the case where `update_stock` or `create_order` raises an exception. The test suite should verify that the lock is still released and no partial state is committed.
