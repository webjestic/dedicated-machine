Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a critical race condition, missing error handling paths, and test gaps that make this unsafe to merge into a production inventory system, particularly one under flash-sale load. The issues range from a silent data corruption window to thread lifecycle problems that will surface under real concurrency.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and DB Transaction

This is the most serious flaw. The stock check and the write are not atomic with respect to the database:

```python
# Lock held here — good
current_stock = self.db.get_stock(req.item_id)   # READ outside transaction

# ... logic ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE inside transaction
    self.db.create_order(...)
```

The `get_stock` call happens **outside** the database transaction. If the DB read uses `READ COMMITTED` isolation (the default in PostgreSQL and MySQL), another process could commit a stock change between the read and the `update_stock` write. The Redis lock protects against concurrent Python processes, but it does **not** protect against:

- Direct DB writes from other services not participating in the Redis lock protocol
- Redis/network partitions that caused the lock to silently expire before the heartbeat caught it
- Any future code path that bypasses this service

**Fix:** Move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` (pessimistic locking) or a conditional `UPDATE ... WHERE stock >= quantity RETURNING new_stock` (optimistic). The Redis lock is a performance optimization; the DB must be the authoritative guard.

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(...)
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

### 2. Heartbeat Thread Can Outlive the Lock Release

```python
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # = 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

`t.join()` has a timeout of 9 seconds, but it is **not checked for success**. If the heartbeat thread is blocked (e.g., on a slow Redis call inside `_extend_script`), `join()` returns after 9 seconds and execution continues to `_release_script`. At this point the heartbeat thread is still alive and will attempt to call `_extend_script` on a key that has just been deleted. In a worst case, the key has been re-acquired by another caller and the heartbeat extends a lock it no longer owns.

**Fix:**

```python
finally:
    stop_event.set()
    joined = t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
        # Do NOT release — the lock state is unknown. Raise or let it expire.
        raise RuntimeError(f"Lock cleanup failed for {lock_key}")
    self._release_script(keys=[lock_key], args=[token])
```

### 3. Unhandled Exceptions Inside the Lock Context Leak State

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(...)
        self.db.create_order(...)
```

If `db.get_stock`, `db.update_stock`, `db.create_order`, or the transaction commit raises an unexpected exception (network error, DB timeout, etc.), the `contextmanager` `finally` block will still release the lock — that part is correct. However, `reserve_item` will propagate the exception to the caller with no `ReservationResult` returned, which breaks the function's declared return contract. Any caller not wrapping this in a try/except will crash.

**Fix:** Catch broad exceptions at the `reserve_item` level:

```python
except (LockAcquisitionError) as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error during reservation")
```

## Significant Issues

### 4. `LOCK_TTL = 30s` With `HEARTBEAT_INTERVAL = 8s` Is Reasonable, But the Ratio Is Undocumented and Fragile

The comment says "fires at TTL/3" but `8` is not `30/3`. This is a minor inaccuracy but more importantly, these two constants are not enforced to maintain their relationship. If someone changes `LOCK_TTL` to `15` without updating `HEARTBEAT_INTERVAL`, the heartbeat fires after the lock has already expired.

**Fix:** Derive one from the other:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # Enforced relationship
```

---

### 5. Retry Backoff Is Too Aggressive for Flash Sale Load

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

With `retry_attempts=3`, the maximum wait before giving up is roughly `0.1 + 0.2 + jitter ≈ 0.35s`. Under flash sale load with hundreds of concurrent requests, the lock holder may legitimately hold the lock for several seconds (DB transaction under load). Three retries with sub-second backoff means the vast majority of requests will fail with "service unavailable" almost immediately.

This is a product/architecture decision, but it should be explicitly documented or made configurable. Hardcoding `retry_attempts=3` with no caller-facing knob is a hidden capacity ceiling.

---

### 6. `db.transaction()` Failure Is Not Distinguished From Success

If the transaction context manager's `__exit__` swallows exceptions (which some ORM implementations do), a failed commit could return `ReservationResult(success=True)` with no actual DB write. The code has no way to detect this.

**Fix:** Verify the transaction result or ensure the DB abstraction raises on failure. At minimum, add a comment that the `db` contract requires exceptions to propagate on commit failure.

## Test Gaps

### 7. Heartbeat Behavior Is Not Tested

There is no test that verifies the heartbeat actually fires and calls `_extend_script`. The `test_lock_release_uses_lua_script` test checks that `_release_script` is called, but the heartbeat path is entirely untested. A regression could silently remove the heartbeat and the test suite would still pass.

### 8. DB Exception During Transaction Is Not Tested

No test covers what happens when `db.update_stock` or `db.create_order` raises. This is the most common failure mode in production.

### 9. `test_reserve_lock_contention` Does Not Verify Retry Count

```python
redis_client.set.return_value = False
```

This test verifies the failure path but does not assert that `redis_client.set` was called exactly `retry_attempts` (3) times. A future change that removes retries entirely would not be caught.

```python
assert redis_client.set.call_count == 3
```

### 10. No Test for Concurrent Quantity Boundary (Stock = Quantity - 1)

`test_reserve_concurrent_quantity` tests `quantity=3, stock=3` (exact match). There is no test for `quantity=3, stock=2` (off-by-one boundary), which is the most common source of inventory oversell bugs.

## Minor Issues

### 11. `order_id` Generated Before Transaction

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(order_id, ...)
```

If the transaction fails and is retried externally, a new `order_id` will be generated on the retry, which is correct. But if the caller retries `reserve_item` after a timeout, they get a different `order_id` each time with no idempotency key. For a flash sale system, idempotency on the reservation call (client-supplied `idempotency_key`) should be considered.

### 12. Lock Key Namespace Collision Risk

`f"inventory:lock:{req.item_id}"` — if `item_id` values are shared with other Redis key namespaces in the same instance, collisions are possible. This is low risk given the Pydantic validation, but the namespace should be documented.

### 13. `daemon=True` on Heartbeat Thread

Daemon threads are killed abruptly on interpreter shutdown. If the process exits while holding a lock, the lock will not be released until TTL expiry. This is acceptable behavior for a distributed lock (TTL is the safety net), but it should be explicitly documented as intentional.
