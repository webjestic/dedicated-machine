Decision: Request Changes

Summary: The core distributed locking logic is well-structured — atomic Lua scripts, heartbeat renewal, exponential backoff with jitter, and Pydantic validation are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges, ranging from a TOCTOU race on stock reads to missing exception handling around the database transaction and an incomplete test suite that doesn't actually validate the heartbeat behavior or transaction rollback paths.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU Race Between `get_stock` and `update_stock`**

```python
# services/inventory.py, inside _acquire_lock context
current_stock = self.db.get_stock(req.item_id)
...
new_stock = current_stock - req.quantity
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
```

The Redis lock serializes callers at the application layer, but `get_stock` reads *outside* the database transaction. If the DB is also accessed by any other process, replica lag, or a future code path that bypasses this service, the read is stale by the time the write commits. The stock read must occur *inside* the transaction and must use a pessimistic row lock (e.g., `SELECT ... FOR UPDATE`). This is the most critical issue — it is a correctness flaw that can produce oversells under specific failure modes.

```python
# Required pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        ...  # early return inside transaction is fine
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

---

**2. Unhandled Exceptions Inside the Lock Context Will Silently Swallow Errors**

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `update_stock` or `create_order` raises (DB connection drop, constraint violation, etc.), the exception propagates up through `_acquire_lock`'s `finally` block, which correctly releases the lock — but `reserve_item` has no `except` clause for generic DB errors. The exception will bubble to the caller completely unhandled. For a service layer this needs an explicit catch:

```python
except Exception as e:
    logger.error("Unexpected error during reservation item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟠 High

**3. Heartbeat Thread Leak on Lock Loss**

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat detected lock loss key=%s", lock_key)
            return  # ← thread exits, but the main flow continues unaware
```

When the heartbeat detects lock loss (`result == 0`), it logs and exits silently. The main thread continues executing the DB transaction believing it holds the lock — but it does not. Another process may have acquired the lock and begun its own reservation. This can produce a double-reservation. The heartbeat must signal the main thread on lock loss, and the main thread must abort:

```python
# Add a lock_lost_event: threading.Event()
if result == 0:
    logger.error("Lock lost during operation key=%s — aborting", lock_key)
    lock_lost_event.set()
    return

# In reserve_item, check after the transaction:
if lock_lost_event.is_set():
    # attempt compensating rollback / raise
```

---

**4. `t.join(timeout=...)` Does Not Guarantee Thread Termination**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`join` with a timeout returns whether or not the thread has stopped. If the heartbeat thread is blocked on a slow Redis call, `join` returns after the timeout and the release script fires anyway. The thread is now a zombie that may call `_extend_script` *after* the lock has been released and potentially re-acquired by another caller, corrupting the new owner's TTL. Either use `join()` without a timeout (and ensure `_beat` cannot block indefinitely), or check `t.is_alive()` after join and log a hard error.

---

**5. Redis Exceptions Are Not Caught**

Neither `_acquire_lock` nor `reserve_item` catches `redis.exceptions.RedisError`. A Redis connection failure during `self.redis.set(...)`, `_extend_script`, or `_release_script` will raise an unhandled exception. Given this is a flash sale service under high load, Redis timeouts are a realistic failure mode. Wrap Redis calls and surface them as `LockAcquisitionError` or a new `ServiceUnavailableError`.

---

### 🟡 Medium

**6. `LOCK_TTL = 30` Is Dangerously Long for a Flash Sale**

A 30-second lock TTL means a crashed process holds the lock for up to 30 seconds, blocking all reservations for that item during peak load. The heartbeat interval of 8 seconds is reasonable, but the initial TTL should be shorter (5–10 seconds) with the heartbeat keeping it alive for legitimate long operations. This is a configuration concern but has real availability impact during a flash sale event.

**7. `retry_attempts=3` With Exponential Backoff Has Unbounded Worst-Case Latency**

```python
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

At attempt 2, backoff is ~0.4s + jitter. With 3 attempts the worst-case wait before failure is ~0.5s of sleep plus Redis RTT. That is acceptable, but this is hardcoded. For a flash sale context, this should be configurable at construction time and documented with SLA implications.

**8. `remaining_stock` Leaks Internal Inventory State on Failure**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,  # ← exposed on insufficient stock
    message="Insufficient stock"
)
```

Returning exact remaining stock to the caller on failure is an information disclosure. An adversary can use repeated requests to poll exact inventory levels. Return a boolean or bucketed value ("low stock", "out of stock") instead.

---

### 🔵 Low / Test Coverage Gaps

**9. No Test for DB Transaction Failure / Rollback Path**

There is no test asserting that if `update_stock` raises, `create_order` is not called and the caller receives a failure result. This is the most important error path for a financial-adjacent reservation system.

**10. Heartbeat Behavior Is Not Tested**

The heartbeat thread is a core correctness mechanism. There is no test that verifies the lock is extended during a long operation, or that lock loss is detected. These should be integration-level tests using a real or fake Redis with TTL control.

**11. `test_lock_release_uses_lua_script` Is Fragile**

```python
service._release_script.assert_called_once()
```

This asserts the script was called once but does not assert it was called with the correct `keys` and `args`. A future refactor could call it with wrong arguments and this test would still pass.

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"], args=[ANY]
)
```

---

## Required Before Approval

1. Move `get_stock` inside the DB transaction with `SELECT FOR UPDATE`
2. Add exception handling for DB errors in `reserve_item`
3. Implement lock-loss signaling from heartbeat to main thread with abort logic
4. Verify `t.is_alive()` after `join` and handle the zombie thread case
5. Wrap Redis calls in `redis.exceptions.RedisError` handlers
6. Remove `remaining_stock` from failure responses
7. Add tests for DB failure/rollback and heartbeat lock-loss paths
