Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for a high-concurrency flash sale environment. The most severe issues are a non-reentrant, instance-level heartbeat state that creates race conditions under concurrent use, a non-atomic lock release that introduces a TOCTOU vulnerability, and a missing atomicity guarantee between the stock update and order creation database operations. These issues must be resolved before this code is anywhere near a production workload.

---

## Review

### 🔴 Critical Issues

---

#### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most dangerous flaw in the submission. `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`.

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the **same service instance** — which is the entire point of this service — the second call to `_start_heartbeat` will overwrite `self._heartbeat_active` and `self._heartbeat_thread`. The first lock's heartbeat thread is now orphaned with no way to be stopped, and `_stop_heartbeat` from either context manager exit will stop **both** heartbeats simultaneously, potentially killing the renewal for a lock that is still legitimately held.

The heartbeat state must be scoped to the lock acquisition context, not the instance. A clean fix is to pass a `threading.Event` into the heartbeat thread and close over it locally:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to THIS acquisition
    acquired = False
    ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        ...

def _start_heartbeat(self, lock_key, token, stop_event):
    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    threading.Thread(target=_beat, daemon=True).start()
```

Note also that `time.sleep(self.HEARTBEAT_INTERVAL)` followed by an `if self._heartbeat_active` check is a poor pattern regardless — `threading.Event.wait(timeout=...)` is the correct primitive here because it responds immediately to the stop signal rather than sleeping through a full interval before noticing it should exit.

#### 2. Non-Atomic Lock Release — TOCTOU Race Condition

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # <-- conditional delete
```

There is a window between the `GET` and the `DELETE` where another process can acquire the lock after the TTL expires but before the `DELETE` executes. You would then delete a lock you do not own, corrupting the mutual exclusion guarantee for whoever acquired it legitimately.

The canonical solution is a Lua script, which Redis executes atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

This is not optional. Any distributed lock implementation that does not use an atomic compare-and-delete is broken by definition.

#### 3. No Atomicity Between `update_stock` and `create_order`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock has been decremented but no order record exists. The item is effectively vaporized. The distributed lock protects against concurrent access, but it does not protect against partial failure within the lock boundary. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager. If it does not, that is a separate architectural gap that needs to be addressed before this service can be considered safe.

### 🟡 Significant Issues

---

#### 4. No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `ReservationResult` is an **output** model. There is no validation on the inputs to `reserve_item`. `item_id` is an unvalidated string that flows directly into a Redis key and a database query. A caller passing `item_id = "../../admin"` or an empty string, or `quantity = -5` or `quantity = 0`, will produce undefined behavior.

At minimum:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(strip_whitespace=True, min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(default=1, ge=1, le=1000)
```

Validate at the entry point of `reserve_item` before any lock is acquired.

---

#### 5. Heartbeat Interval Is Too Close to Lock TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T+10, T+20, T+30. At T+30 the lock expires. Due to scheduling jitter, GIL contention, or a slow Redis round-trip, the T+30 renewal may arrive after the key has already expired. The general guidance from the Redlock paper and practical Redisson implementations is that the renewal interval should be no more than `TTL / 3`, and the renewal should reset the TTL to the full value, not add to it. Here `HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` is exactly at that boundary with zero margin. Set `HEARTBEAT_INTERVAL = 5` or lower, or increase `LOCK_TTL`.

---

#### 6. Bare `RuntimeError` for Lock Failure Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a broad exception type. A `RuntimeError` thrown from inside the `with self._acquire_lock(lock_key):` block — say, from `self.db.get_stock` — would be silently swallowed and returned to the caller as "service temporarily unavailable," masking a real bug. Define a specific `LockAcquisitionError` exception and catch only that:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) instead of RuntimeError in _acquire_lock
```

### 🟠 Test Coverage Gaps

---

#### 7. Heartbeat Is Mocked Away Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is one of the most complex and failure-prone components in this PR, and it has zero test coverage. There should be at least one integration-level test that verifies the heartbeat actually calls `redis.expire` with the correct key and token, and that it stops after `_stop_heartbeat` is called. Mocking it away entirely means the heartbeat thread has never been exercised under test.

---

#### 8. The TOCTOU Test Validates the Wrong Behavior

```python
def test_lock_not_released_if_expired(service, redis_client):
    """Lock release is skipped if token no longer matches (lock expired or stolen)."""
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test is asserting correct behavior for a broken implementation. Once the lock release is replaced with an atomic Lua script, this test needs to be rewritten to verify that `redis.eval` is called with the correct script, key, and token — not that `delete` is conditionally skipped.

---

#### 9. No Test for Partial DB Failure

There is no test covering the scenario where `update_stock` succeeds but `create_order` raises. This is the exact failure mode described in issue #3 above. Once transaction support is added, a test must verify that a failure in `create_order` rolls back the stock decrement.

---

### 🔵 Minor Notes

- The `db` parameter has no type annotation or interface definition. For a service of this criticality, a `Protocol` or abstract base class should define the expected contract for `get_stock`, `update_stock`, and `create_order` so that implementors and the type checker both know what is expected.
- `logger = logging.getLogger(__name__)` is correct, but there is no guidance on how the caller should configure the logger. For a library-style module this is fine, but document it.
- The checklist item "Input validation via Pydantic" should be removed or corrected until actual input validation is implemented.
