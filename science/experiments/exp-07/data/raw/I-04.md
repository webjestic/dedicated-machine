Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — lock ownership verification, heartbeat renewal, and backoff retry are all present and structurally sound. However, there are several critical architectural and correctness flaws that must be addressed before this can be approved, particularly around instance-level heartbeat state, non-atomic lock release, database transaction safety, and missing input validation that the checklist falsely claims is present.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the entire implementation.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as **instance variables**, not as locals scoped to the lock context. If a single `InventoryService` instance is shared across threads — which is the standard deployment pattern for a service object — two concurrent calls to `reserve_item` will race on `_heartbeat_active`. The second call's `_stop_heartbeat()` will set the flag to `False`, silently killing the first call's heartbeat thread while its lock is still held. This can cause the first caller's lock to expire mid-operation, defeating the entire purpose of the heartbeat.

**Required fix:** Heartbeat state must be scoped to the lock acquisition context, not the instance. Use a local `Event` object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note also that `_stop_heartbeat` currently does not `join()` the thread. The heartbeat thread can continue executing for up to `HEARTBEAT_INTERVAL` seconds after the lock has already been released, potentially extending a lock that now belongs to a different owner.

### 2. Non-Atomic Lock Release — TOCTOU Race Condition

```python
# In _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The `GET` and `DELETE` are two separate round trips. Between the `GET` returning your token and the `DELETE` executing, another process could acquire the lock. You would then delete a lock you no longer own. This is a well-documented failure mode and the canonical solution is a Lua script for atomic compare-and-delete:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def __init__(self, redis_client, db):
    ...
    self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a similar Lua script.

### 3. No Database Transaction — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two sequential database operations with no transactional boundary. If `update_stock` succeeds and `create_order` raises an exception, stock has been decremented but no order record exists. The system is now in an inconsistent state — inventory is gone, but the customer has no order. The distributed lock protects against concurrent access, but it does not protect against partial failure within the lock boundary.

**Required fix:**

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose transactional semantics, and the tests must verify rollback behavior on `create_order` failure.

### 4. Checklist Claims Input Validation — It Does Not Exist

The PR checklist marks "Input validation via Pydantic" as complete. `ReservationResult` is a Pydantic **output** model. There is zero validation on the inputs to `reserve_item`.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing validates item_id or quantity before use
    lock_key = f"inventory:lock:{item_id}"
```

Consider what happens with:
- `item_id = ""` — acquires lock on `inventory:lock:`, a degenerate key
- `item_id = "a" * 10_000` — unbounded Redis key length
- `quantity = 0` — succeeds, creates a zero-unit order
- `quantity = -5` — decrements stock by negative value, **increasing** inventory
- `quantity = 999999` — likely to pass the stock check if stock is high enough

**Required fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

Validate at the entry point of `reserve_item` before any Redis or DB interaction occurs.

## Moderate Issues

### 5. Heartbeat Sleep Is Not Interruptible

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 seconds
        if self._heartbeat_active:
            ...
```

`time.sleep(10)` cannot be interrupted. When `_stop_heartbeat()` is called, the thread will continue sleeping for up to 10 seconds before checking the flag. During that window, the thread may extend a lock that has already been released and re-acquired by another process. The `threading.Event.wait(timeout=...)` pattern shown above resolves this cleanly.

---

### 6. Bare `RuntimeError` for Lock Failure Is Too Broad

```python
raise RuntimeError(
    f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
)
```

Catching `RuntimeError` in `reserve_item` will also silently swallow any other `RuntimeError` raised inside the lock body, including bugs in `db.get_stock`, `db.update_stock`, or `db.create_order`. Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass
```

---

### 7. `quantity` Parameter Not Validated Against Reasonable Business Bounds at DB Layer

Even with Pydantic input validation added, the service should confirm that `quantity` is a positive integer before constructing the lock key. Defense in depth matters here given the financial nature of the operation.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is mocked in every test. This means the most complex and dangerous component in the service — the background thread — has zero test coverage. At minimum, an integration-style test should verify that the heartbeat actually extends the TTL and that it stops after lock release.

### 9. No Test for Partial DB Failure

There is no test covering the scenario where `update_stock` succeeds but `create_order` raises. This is the most dangerous failure mode in the service and it is completely untested.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test patches `redis.get` to return a different token globally, which means `db.get_stock` is never reached either — the test is asserting behavior about a code path it has not actually exercised correctly. The mock setup conflates the heartbeat's `get` call with the release `get` call. Once the Lua script fix is applied, this test will need to be rewritten entirely.

---

## Minor Issues

- **No maximum retry backoff cap.** With `(2 ** attempt) * 0.1` and a configurable `retry_attempts`, callers could pass large values and produce extremely long waits. Cap the backoff at a reasonable ceiling (e.g., 2 seconds).
- **`LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants** rather than constructor parameters, making them impossible to tune per-deployment without subclassing.
- **No Redis connection error handling.** A `redis.RedisError` during lock acquisition or release will propagate as an unhandled exception rather than a graceful `ReservationResult(success=False, ...)`.
