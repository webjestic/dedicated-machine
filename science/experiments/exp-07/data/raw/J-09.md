Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning a single `InventoryService` instance cannot safely handle concurrent calls to `reserve_item` without one call's heartbeat stomping on another's. Beyond that, the heartbeat's check-then-extend sequence is a non-atomic TOCTOU race, and the tests are structured in a way that masks several of the real failure modes rather than exercising them. The issues below need to be addressed before this is merged.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Not Concurrency-Safe

This is the most serious problem. `_heartbeat_active` and `_heartbeat_thread` are shared across all calls on the same instance:

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same service instance, the second call will overwrite `_heartbeat_active`, killing the first call's heartbeat. The first lock then silently expires while the first call is still inside the critical section. This is a silent correctness failure — no exception is raised, but the distributed lock guarantee is gone.

**Fix:** Scope heartbeat state to the lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff,
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
            logger.info("Lock released key=%s", lock_key)

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning(
                    "Heartbeat: lock no longer owned, stopping key=%s", lock_key
                )
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 2. TOCTOU Race in Heartbeat Renewal

The current heartbeat does a GET then an EXPIRE as two separate commands:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the GET and the EXPIRE, another process could acquire the lock. You would then extend a lock you no longer own. This should be a single atomic operation using a Lua script:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at construction time
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In the heartbeat:
result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not result:
    logger.warning("Heartbeat: lost lock ownership key=%s", lock_key)
    break
```

The same TOCTOU issue exists in the lock release in `_acquire_lock`. That should also use a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts a raw `str` and `int` with no validation at all. `quantity` can be zero or negative, and `item_id` is completely unchecked. A caller passing `quantity=0` would succeed and create a zero-unit order record.

```python
# Add a request model
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id, req.quantity below
```

### 4. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction wrapping:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. This is a data integrity hole regardless of the distributed lock. These must execute inside a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should document and enforce this.

## Moderate Issues

### 5. Heartbeat Thread Is Not Joined on Exit

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The thread is set as `daemon=True` and the flag is cleared, but there is no `join()`. The thread may still be running when the lock is released. If the thread wakes up between the flag clear and the lock delete, it can extend a lock that is about to be deleted, or in a worst case re-extend a lock that has already been handed to another process. After the fix in issue #1, `stop_event.set()` followed by `heartbeat_thread.join()` handles this correctly.

### 6. `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

In an async or thread-pool context this will block an entire worker thread for up to `0.3s` per call. For a flash sale service under high concurrency this will degrade throughput significantly. If this service will ever be called from an async context, the retry loop should be async-aware or the blocking should be documented explicitly as a constraint.

## Test Issues

### 7. Tests Bypass the Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking these out means the tests never exercise the heartbeat lifecycle, the lock release path, or the `_stop_heartbeat` / `join` interaction. The tests pass but they do not verify the most failure-prone parts of the implementation. At minimum, add integration-style tests that let the heartbeat run with a mock Redis and verify the extend script is called.

### 8. `test_lock_not_released_if_expired` Does Not Actually Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

`redis_client.get` is used both inside `reserve_item` (for the lock release check) and inside the heartbeat. Because the heartbeat is mocked out, this test only verifies the release-path branch. It does not test the scenario where the lock actually expires mid-operation, which is the scenario the heartbeat exists to prevent. Rename this test to reflect what it actually covers, and add a separate test for mid-operation expiry.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but document explicitly that `HEARTBEAT_INTERVAL` must always be less than `LOCK_TTL / 2`.
- The `RuntimeError` catch in `reserve_item` is too broad. Consider a dedicated `LockAcquisitionError` so other unexpected `RuntimeError`s from the DB layer are not silently swallowed and returned as "service unavailable."
- No metrics or alerting hooks. For a flash sale service, lock contention rate and reservation failure rate should be observable. Even a simple counter increment point would be appropriate here.
