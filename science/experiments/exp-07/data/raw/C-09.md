Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The most severe issues are a non-atomic lock release (TOCTOU race), a singleton heartbeat design that breaks under concurrent calls, and missing database transaction atomicity. These are not minor style concerns — they are correctness failures that would cause inventory corruption, overselling, or silent lock theft under real load.

---

## Critical Issues

### 1. Non-Atomic Lock Release — TOCTOU Race Condition

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
# BROKEN — two separate round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock. You will then delete a lock you no longer own. This is a textbook TOCTOU (time-of-check/time-of-use) race condition.

**Required fix:** Use the canonical Lua script for atomic compare-and-delete:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU problem exists in the heartbeat:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # another process can steal here
```

The heartbeat renewal must also be atomic. Use a Lua script or `SET key token XX EX ttl` with ownership verification in a single operation.

### 2. Singleton Heartbeat — Breaks Entirely Under Concurrent Use

**File:** `services/inventory.py`, instance variables

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields, not lock-scoped. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern — a shared service singleton), the following happens:

1. Thread A acquires lock for `item_001`, starts heartbeat, sets `_heartbeat_active = True`
2. Thread B acquires lock for `item_002`, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
3. Thread A finishes, calls `_stop_heartbeat`, sets `_heartbeat_active = False`
4. Thread B's lock now has **no active heartbeat** and will expire mid-operation

This is a silent failure. No exception is raised. Inventory operations proceed while the lock has expired, destroying the mutual exclusion guarantee entirely.

**Required fix:** Heartbeat state must be scoped per lock acquisition, not per instance. Use a local flag passed by reference or a dedicated object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = threading.Thread(
        target=self._heartbeat_worker,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock_atomic(lock_key, token)
```

### 3. Database Operations Are Not Transactional

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The item is effectively lost. The Redis lock provides no protection against this — it only serializes access, it does not provide rollback.

**Required fix:** These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager. If it does not, that is a design gap that must be resolved before this merges.

### 4. Input Validation Is Absent Despite Checklist Claiming Otherwise

**File:** `services/inventory.py`, `reserve_item`

The checklist marks "Input validation via Pydantic" as complete. This is false. `item_id` and `quantity` are raw, unvalidated parameters. There is no Pydantic model gating the inputs to `reserve_item`. Pydantic is only used for the *output* `ReservationResult`.

Concretely:

- `quantity=0` will succeed and create a zero-unit order
- `quantity=-5` will *increase* stock (`new_stock = 10 - (-5) = 15`) — this is an inventory inflation bug
- `item_id=""` or `item_id=None` will produce a lock key of `inventory:lock:` or crash

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

Validate at the entry point of `reserve_item` before any lock is acquired.

## Significant Issues

### 5. Heartbeat Thread Is Never Joined on Stop

**File:** `services/inventory.py`, `_stop_heartbeat` and `_acquire_lock` finally block

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has stopped. The thread may be mid-sleep (`time.sleep(HEARTBEAT_INTERVAL)`) when the flag is set. It will wake up 10 seconds later, check the flag, and exit — but by then the `finally` block has already run, the lock has been deleted, and the thread is operating on stale state. In the worst case, if a new lock with the same key has been acquired by another process, the orphaned thread could attempt to extend or delete it.

The `finally` block must join the thread with a timeout before releasing the lock.

### 6. Exponential Backoff Has No Jitter

**File:** `services/inventory.py`, `_acquire_lock`

```python
backoff = (2 ** attempt) * 0.1
```

Under a flash sale, hundreds of threads will contend on the same lock simultaneously. Without jitter, all threads back off for identical durations and retry in synchronized waves — a thundering herd. This defeats the purpose of backoff.

```python
import random
backoff = (2 ** attempt) * 0.1 * (0.5 + random.random())
```

### 7. Redis Connection Errors Are Silently Swallowed

**File:** `services/inventory.py`, `reserve_item`

Only `RuntimeError` is caught. `redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not handled. These will propagate as unhandled exceptions to the caller rather than returning a clean `ReservationResult`. Depending on the framework, this could expose internal stack traces.

```python
except (RuntimeError, redis.exceptions.RedisError) as e:
    logger.error(...)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Test Coverage Gaps

### 8. Tests Do Not Validate the Core Concurrency Invariant

There is no test that verifies two concurrent calls cannot both succeed when only one unit of stock remains. This is the single most important property of this entire service. A test using `threading.Thread` or `concurrent.futures.ThreadPoolExecutor` with a mock that simulates a race window should be present.

### 9. The TOCTOU Race in Lock Release Is Not Tested

`test_lock_not_released_if_expired` patches `redis.get` to return a different token, but this does not test the race — it tests the happy-path branch. The actual race (token matches at GET time, another process acquires between GET and DELETE) cannot be tested without the atomic Lua script fix in place first.

### 10. Negative and Zero Quantity Are Not Tested

Given the inventory inflation bug in issue #4, there must be explicit tests:

```python
def test_reserve_zero_quantity_rejected(service):
    result = service.reserve_item("item_001", quantity=0)
    assert result.success is False

def test_reserve_negative_quantity_rejected(service):
    result = service.reserve_item("item_001", quantity=-1)
    assert result.success is False
```

## Minor Issues

### 11. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

These should be constructor parameters with defaults, not class-level constants. Different item categories may require different TTLs. Hardcoding them makes the service untestable with shorter intervals without monkey-patching.

### 12. `decode()` Called Without Encoding Specification

```python
current.decode()
```

This assumes UTF-8. It will work in practice but should be explicit: `current.decode("utf-8")`. More importantly, once the Lua script fix is applied, this call disappears entirely from the release path.
