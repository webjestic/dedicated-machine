Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — atomic lock acquisition via `SET NX EX`, token-verified release, heartbeat renewal, and exponential backoff are all present and directionally correct. However, there are several critical architectural flaws, a race condition in the heartbeat, a non-atomic lock release vulnerability, and meaningful gaps in test coverage that collectively make this unsafe to merge for a high-throughput flash sale context. The issues range from correctness bugs that can cause double-reservations under failure conditions to instance-level state that breaks concurrent request handling entirely.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Concurrency Correctness Failure

This is the most severe bug in the submission. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on `InventoryService`, which is almost certainly a singleton or long-lived service object.

```python
# Current — catastrophically broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests call `reserve_item` concurrently:

- Request A acquires lock, starts heartbeat, sets `self._heartbeat_active = True`
- Request B acquires a *different* lock, calls `_start_heartbeat`, overwrites `self._heartbeat_thread`
- Request A's `finally` block calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`
- **Both heartbeats die.** Request B's lock now expires unattended mid-operation

The heartbeat state must be scoped to the individual lock acquisition, not the instance. The fix is to encapsulate this in a dedicated object or pass state through closures local to `_acquire_lock`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()
    # set the event to signal "keep running"
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock was lost, stop silently

    # ... acquire lock ...

    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

Note the `join()` call — addressed further below.

### 2. Non-Atomic Lock Release — TOCTOU Race Condition

```python
# Current — check-then-act, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock TTL can expire and be re-acquired by another process. You would then delete a lock you do not own, allowing two processes to operate simultaneously on the same inventory item. This is a textbook TOCTOU (Time-of-Check-Time-of-Use) race.

The canonical fix is a Lua script that makes the check-and-delete atomic at the Redis server level:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

This is not optional for a correctness-critical reservation system. The heartbeat mitigates the window, but does not eliminate it — especially during GC pauses, network hiccups, or thread starvation.

### 3. Heartbeat Thread Is Not Joined on Exit

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # No join — thread may still be alive
```

After `_stop_heartbeat()` returns, the heartbeat thread may still be in the middle of a `time.sleep(10)` or, worse, in the middle of executing `redis.expire(...)`. The `finally` block then proceeds to delete the lock. There is a narrow but real window where:

1. `_stop_heartbeat()` sets the flag to `False`
2. Lock is deleted via `redis.delete()`
3. Heartbeat thread wakes from sleep, reads the flag, sees `False`, exits — fine in this path
4. **But** if the thread woke *just before* the flag was set, it may call `redis.expire()` on a key that now belongs to someone else

Using `threading.Event` with `wait(timeout=...)` instead of `time.sleep()` allows the thread to wake immediately on signal, and `join()` provides the happens-before guarantee you need before touching the lock key in `finally`.

### 4. Database Operations Are Not Atomic — Partial Failure Leaves Corrupt State

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock has been decremented but no order exists. The reservation is lost. The distributed lock does not protect against this — it only serializes access, it does not provide atomicity across the two DB operations.

These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the DB layer, but the requirement is non-negotiable for a financial/inventory operation.

## Significant Issues

### 5. Input Validation Is Claimed But Not Implemented

The checklist asserts "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` — a caller passing `quantity=0` or `quantity=-5` would "succeed" and corrupt stock
- `item_id` being an empty string, whitespace, or a path-traversal-style string that could affect lock key construction
- `quantity` being unreasonably large (no upper bound)

A Pydantic model is defined in the file but never used for input. Either define a `ReservationRequest` model and validate through it, or add explicit guards at the top of `reserve_item`:

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError("quantity must be a positive integer")
if quantity > MAX_RESERVATION_QUANTITY:
    raise ValueError(f"quantity exceeds maximum of {MAX_RESERVATION_QUANTITY}")
```

A negative quantity reservation is a potential inventory manipulation vector.

---

### 6. Lock Key Namespace Is Injectable

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is not sanitized and contains colons or Redis-special characters, an attacker or buggy caller could construct a key that collides with other namespaces in the Redis keyspace (e.g., `item_id = "foo:lock:other_item"` produces `inventory:lock:foo:lock:other_item` which is merely ugly, but `item_id` values sourced from user input with insufficient upstream validation could be more problematic depending on Redis ACL configuration). Validate and sanitize `item_id` before constructing the key.

---

### 7. Bare `RuntimeError` Is Too Broad in the `except` Clause

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a broad base class. If `db.get_stock`, `db.update_stock`, or `db.create_order` raise a `RuntimeError` (not uncommon in ORM layers), the caller receives a misleading "service unavailable" message, the error is silently swallowed, and the operation appears to have simply failed on lock contention. Define a specific `LockAcquisitionError` exception and catch only that.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out — Core Behavior Is Untested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test stubs out the heartbeat. There is zero test coverage for:
- TTL extension actually being called
- Heartbeat stopping when the lock is released
- Heartbeat correctly aborting when the token no longer matches

At minimum, an integration-style test using a real or fake Redis (e.g., `fakeredis`) should verify that `redis.expire` is called with the correct key and TTL during a simulated long operation.

### 9. Partial DB Failure Is Not Tested

There is no test for the case where `db.update_stock` succeeds but `db.create_order` raises. This is the corrupt-state scenario from Issue 4 and it needs a regression test once the transaction fix is in place.
