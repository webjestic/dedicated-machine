Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will stomp each other's heartbeat state, potentially killing an active heartbeat or allowing a rogue thread to extend a lock it no longer owns. There are also meaningful gaps in input validation, atomicity guarantees, and test coverage that need to be addressed before this is safe to merge into a production flash-sale path. Detailed findings below.

---

## Critical Issues

### 1. Heartbeat State Is Not Thread-Safe (Instance-Level Variables)

This is the most serious bug. `_heartbeat_active` and `_heartbeat_thread` are shared across all concurrent invocations of `reserve_item` on the same service instance.

**Scenario:**
- Request A acquires lock, starts heartbeat → `_heartbeat_active = True`
- Request B acquires a different lock, calls `_start_heartbeat` → overwrites `_heartbeat_thread`
- Request A finishes, calls `_stop_heartbeat` → sets `_heartbeat_active = False`
- Request B's heartbeat is now silently dead. Its lock will expire mid-operation.

The inverse is also dangerous: Request B's `_stop_heartbeat` kills Request A's still-running heartbeat.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning("Lost lock ownership key=%s, stopping heartbeat", lock_key)
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note the use of `stop_event.wait(timeout=...)` instead of `time.sleep` — this allows the heartbeat to exit promptly when the event is set, rather than sleeping through its interval after the lock is released.

### 2. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used for the *output* model.

- `item_id` could be an empty string, whitespace, a path traversal fragment, or an arbitrarily long string that becomes a Redis key.
- `quantity` could be zero or negative, which would silently "succeed" and *increase* stock (`new_stock = 10 - (-5) = 15`).

**Fix:** Add a request model and validate at the entry point:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValueError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")

    lock_key = f"inventory:lock:{req.item_id}"
    # ... rest of method uses req.item_id, req.quantity
```

### 3. DB Operations Are Not Atomic

`update_stock` and `create_order` are two separate DB calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. This is a silent data integrity failure.

```python
# This is not atomic:
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

**Fix:** These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock serializes access, but it does not substitute for transactional integrity at the persistence layer.

## Moderate Issues

### 4. Lock Release Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock. The correct pattern is an atomic Lua script:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if not released:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

This same TOCTOU issue exists in the heartbeat's `GET` + `expire` sequence and should be fixed there too.

---

### 5. Heartbeat Thread Is Never Joined

The current `_stop_heartbeat` just sets a flag. The thread may still be alive and extending a lock that has already been released and re-acquired by another process. The fix in item #1 above addresses this with `thread.join(timeout=...)`.

---

### 6. `time.sleep` in Retry Loop Blocks the Caller's Thread

For a service expected to handle flash-sale traffic, blocking a thread for up to `0.1 + 0.2 + 0.4 = 0.7s` per failed reservation is a throughput problem. Consider whether this service should be async (`asyncio` + `aioredis`) or whether the retry should be offloaded. At minimum, document the threading model assumption explicitly.

## Test Coverage Gaps

### 7. The Core Concurrency Bug Is Not Tested

There is no test that instantiates a single `InventoryService` and calls `reserve_item` concurrently from two threads. This is exactly the scenario that would expose the instance-level heartbeat state collision.

```python
import concurrent.futures

def test_concurrent_reservations_do_not_corrupt_heartbeat_state(db):
    # Use a real threading scenario, not mocked heartbeat
    redis_mock = MagicMock()
    redis_mock.set.return_value = True
    redis_mock.get.return_value = None  # simulate expired lock on release check

    svc = InventoryService(redis_mock, db)
    db.get_stock.return_value = 100

    with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
        futures = [executor.submit(svc.reserve_item, f"item_{i}") for i in range(5)]
        results = [f.result() for f in futures]

    # All should complete without corrupting each other's state
    assert all(r is not None for r in results)
```

### 8. Negative and Zero Quantity Are Not Tested

```python
def test_reserve_zero_quantity_rejected(service):
    result = service.reserve_item("item_001", quantity=0)
    assert result.success is False

def test_reserve_negative_quantity_rejected(service):
    result = service.reserve_item("item_001", quantity=-5)
    assert result.success is False
```

### 9. Atomic Lock Release Script Is Not Tested

Once the Lua script is added, test that the release is skipped when the token does not match, and that the script is called with the correct arguments.

## Minor Notes

- The `test_lock_not_released_if_expired` test passes `b"different-token"` from `redis.get`, but the token generated inside `_acquire_lock` is a fresh UUID — the mock token `b"test-token"` set in the fixture will never equal it either. The test is accidentally correct but for the wrong reason. Pin the token via a `uuid.uuid4` patch to make the assertion meaningful.
- Consider a maximum `item_id` length cap on the Redis key to prevent oversized key injection regardless of upstream validation.
- Log the `quantity` in the insufficient-stock path to aid debugging under load.
