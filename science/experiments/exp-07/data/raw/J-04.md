Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields, causing one call's heartbeat to clobber another's or be prematurely stopped. Beyond that, the DB operations lack atomicity guarantees and there is no input validation on the public method boundary despite the checklist claiming otherwise. The test suite also has a structural gap that masks the concurrency bug entirely.

---

### Critical Issues

#### 1. Instance-Level Heartbeat State Is Not Thread-Safe

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are shared across all concurrent invocations of `reserve_item` on the same service instance.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests arrive simultaneously:
- Request A sets `_heartbeat_active = True` and starts its thread.
- Request B sets `_heartbeat_active = True` and starts a *second* thread, overwriting `_heartbeat_thread`.
- Request A finishes and calls `_stop_heartbeat()`, setting `_heartbeat_active = False`.
- Request B's heartbeat thread is now dead even though B's lock is still held.
- Request B's lock expires mid-operation. Another process steals the lock. Data is now corrupted.

The fix is to scope heartbeat state to each lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat():
        while not active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        active.set()          # signal the thread to stop
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note also that `_stop_heartbeat` currently just sets a flag — it does not `join()` the thread. The heartbeat thread can still be mid-execution (between the `get` and `expire` calls) when the lock is deleted, causing a spurious TTL extension on a key that now belongs to another holder.

#### 2. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record. There is no rollback. This is a silent data integrity failure — inventory is consumed but the sale is lost. The lock does not protect you from your own partial failure.

At minimum:

```python
try:
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
except Exception:
    # attempt compensating write or re-raise for caller to handle
    logger.exception("DB write failed, attempting stock rollback item_id=%s", item_id)
    self.db.update_stock(item_id, current_stock)  # best-effort
    raise
```

The real fix is a database transaction wrapping both writes. The lock is a coordination primitive, not a substitute for transactional integrity.

#### 3. No Input Validation on `reserve_item`

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is nothing preventing `item_id=""`, `item_id="../../etc/passwd"` (depending on DB driver), or `quantity=0` / `quantity=-5`. A negative quantity would pass the `current_stock < quantity` check and *increase* stock. This is a correctness bug and a potential abuse vector.

```python
if not item_id or not isinstance(item_id, str):
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")
```

### Moderate Issues

#### 4. `time.sleep` in Lock Retry Blocks the Calling Thread

Under high flash-sale load, every blocked caller sleeps in-thread. If this service runs in a synchronous WSGI context (Gunicorn, uWSGI), this will exhaust the worker pool quickly. This should be called out in documentation at minimum, and an async-compatible implementation (e.g., `asyncio` with `aioredis`) should be considered for the flash sale use case specifically.

#### 5. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
```

`RuntimeError` is also raised by Python internals and third-party libraries. Define a specific exception class:

```python
class LockAcquisitionError(RuntimeError):
    pass
```

### Test Suite Issues

#### 6. Tests Do Not Cover the Concurrency Bug

The heartbeat is mocked out in every test that uses the `service` fixture. The test that would expose the instance-variable race condition (`test_reserve_concurrent_quantity`) does not actually run two threads. No test exercises concurrent calls to `reserve_item`. At minimum, add:

```python
def test_concurrent_reservations_do_not_corrupt_heartbeat_state():
    import threading
    # Two threads calling reserve_item simultaneously on the same instance
    # should each maintain independent heartbeat state.
    ...
```

#### 7. Token Mismatch Test Is Misleading

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test passes because the mock returns `b"different-token"` for *all* `get` calls, including the one inside `_acquire_lock`'s `finally` block. But the `uuid4()` token generated at runtime will never equal `"different-token"`, so `delete` is correctly skipped. The test is valid but the fixture setup is coincidental rather than intentional — the token comparison works only because the mock returns a static string that will never match a real UUID. Add a comment explaining this or restructure to be explicit.
