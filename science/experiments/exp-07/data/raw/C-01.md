Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that make it unsafe for production use in a high-concurrency flash sale context. The heartbeat design introduces a race condition that can cause lock theft, the service is not safe for concurrent use across multiple reservations, and the lock release is not atomic — all of which are unacceptable in a distributed inventory system where correctness is non-negotiable. The test suite also masks rather than exercises the most dangerous failure modes.

---

## Critical Issues

### 1. Non-Atomic Lock Release (Race Condition / Security Vulnerability)

**File:** `services/inventory.py`, lines in `_acquire_lock` finally block

```python
# BROKEN — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock after the TTL expires. You then delete a lock you no longer own. This is a textbook TOCTOU (Time-of-Check to Time-of-Use) race condition.

**Required fix:** Use a Lua script to make the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

The same race exists in the heartbeat — `GET` then `EXPIRE` is also non-atomic. Use a Lua script there too.

### 2. Instance-Level Heartbeat State — Not Safe for Concurrent Calls

**File:** `services/inventory.py`, `InventoryService.__init__` and heartbeat methods

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal case for a shared service object), two concurrent calls will clobber each other's heartbeat state. Call A's `_stop_heartbeat()` will kill Call B's heartbeat thread, causing Call B's lock to expire while still held.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass state through a local object or closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()
    ...
    thread = self._start_heartbeat(lock_key, token, active)
    try:
        yield token
    finally:
        active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

### 3. Heartbeat Thread Is Never Joined — Silent Lock Abandonment

**File:** `services/inventory.py`, `_stop_heartbeat`

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The heartbeat thread could be mid-sleep when the flag is set, then wake up and attempt to extend a lock that has already been released and re-acquired by another process. This extends a lock the service no longer owns, silently corrupting the distributed lock state.

**Required fix:** Join the thread with a timeout after clearing the flag:

```python
active.clear()
if thread and thread.is_alive():
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 4. Database Operations Are Not Transactional

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `update_stock` succeeds and `create_order` fails (network blip, DB error, process crash), stock is decremented but no order exists. The inventory is now permanently corrupted — stock is gone, but no order was created. The distributed lock only protects against concurrent access; it does not protect against partial writes.

**Required fix:** These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction doesn't support transactions, that interface must be redesigned before this service can be considered safe.

### 5. No Input Validation on Public Entry Point

**File:** `services/inventory.py`, `reserve_item`

The PR checklist claims "Input validation via Pydantic," but there is no validation on `item_id` or `quantity` at the entry point. `quantity` could be zero, negative, or a float coerced to int. `item_id` could be an empty string, which would produce a lock key of `inventory:lock:` and match any other empty-string item.

```python
# Nothing stops this:
service.reserve_item("", quantity=-5)
service.reserve_item("item_001", quantity=0)
```

**Required fix:**

```python
from pydantic import BaseModel, Field, validator

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
    ...
```

The `item_id` pattern constraint also prevents lock key injection attacks where a crafted `item_id` could target an unrelated lock key.

## Significant Issues

### 6. Bare `RuntimeError` Is Too Broad in Exception Handling

**File:** `services/inventory.py`, `reserve_item`

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is caught here to handle lock contention, but `RuntimeError` is also raised by Python internals and potentially by `self.db` calls. A database error mid-reservation will be silently swallowed and reported as "lock unavailable" — masking data corruption events. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

### 7. Retry Backoff Is Blocking — Unsuitable for Async or High-Throughput Contexts

**File:** `services/inventory.py`, `_acquire_lock`

```python
time.sleep(backoff)
```

Blocking sleep inside a lock acquisition loop will stall the thread pool under high concurrency. With 3 retries and exponential backoff, a single call can block for up to 0.1 + 0.2 + 0.4 = 0.7 seconds. At scale, this will exhaust thread pools. If this service is ever used in an async context, `time.sleep` will block the event loop entirely.

At minimum, add jitter to prevent thundering herd:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

And document clearly that this service is synchronous-only, or provide an async variant.

---

### 8. Lock Key Namespace Collision Risk

**File:** `services/inventory.py`

```python
lock_key = f"inventory:lock:{item_id}"
```

If the Redis instance is shared with other services (common in cost-constrained environments), there is no tenant or service namespace prefix. Another service using the key pattern `inventory:lock:*` for a different purpose will collide. Use a configurable prefix:

```python
def __init__(self, redis_client, db, key_prefix: str = "obotix:inventory"):
    self._key_prefix = key_prefix

lock_key = f"{self._key_prefix}:lock:{item_id}"
```

## Test Suite Issues

### 9. Tests Disable the Heartbeat Rather Than Testing It

**File:** `tests/test_inventory.py`

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is one of the most complex and failure-prone components in this service. Mocking it out entirely means the thread lifecycle, the token ownership check, and the extend-then-expire race are completely untested. At minimum, there must be integration-level tests that run the real heartbeat against a real or fake Redis (e.g., `fakeredis`) and verify:

- The lock TTL is extended before expiry
- The heartbeat stops cleanly after lock release
- The heartbeat does not extend a lock it no longer owns

### 10. `test_lock_not_released_if_expired` Does Not Test the Real Race

**File:** `tests/test_inventory.py`, `test_lock_not_released_if_expired`

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test patches `redis.get` to return a different token globally, which means the lock is never considered acquired
