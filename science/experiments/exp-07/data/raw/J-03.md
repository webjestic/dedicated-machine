Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields, causing one call's heartbeat to clobber another's or be stopped prematurely. Beyond that, the DB operations lack atomicity guarantees and there is no input validation on the public method's arguments despite the checklist claiming otherwise. The test suite also has a structural gap that masks the concurrency bug entirely.

---

### Detailed Findings

#### 🔴 Critical — Heartbeat State Is Not Thread-Safe

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields. If two requests call `reserve_item` concurrently on the same `InventoryService` instance (the normal deployment model), the following race occurs:

- Request A acquires lock, starts heartbeat → `_heartbeat_active = True`
- Request B acquires a different item's lock, calls `_start_heartbeat` → overwrites `_heartbeat_thread`
- Request A finishes, calls `_stop_heartbeat` → sets `_heartbeat_active = False`, killing **both** heartbeats
- Request B's lock now expires mid-operation, defeating the entire purpose of the heartbeat

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass a `threading.Event` into `_start_heartbeat` and return it, keeping state local to the `_acquire_lock` context.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...

def _start_heartbeat(self, lock_key, token, stop_event):
    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note the secondary fix here: `time.sleep` in the original heartbeat does not respond to the stop signal until after the sleep completes. Using `stop_event.wait(timeout=...)` makes shutdown immediate.

#### 🔴 Critical — No Atomicity Between DB Operations

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. The distributed lock does not protect against this — it only serializes access, it does not roll back. For a government contracting context, this is an audit and financial integrity failure.

**Fix:** Wrap both operations in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

#### 🔴 High — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no guards. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the method will silently produce nonsensical results (e.g., stock *increases* on negative quantity).

```python
# Current — no validation
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

**Fix:** Validate at the boundary.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="item_id must not be empty")
if quantity < 1:
    return ReservationResult(success=False, message="quantity must be a positive integer")
```

Or introduce a Pydantic request model and validate before entering the lock.

#### 🟡 Medium — Lock Release Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock. This is a classic check-then-act race. The standard fix is a Lua script for atomic compare-and-delete, which Redis guarantees executes atomically.

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

#### 🟡 Medium — Tests Do Not Cover the Critical Concurrency Path

The test suite mocks out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the thread-safety bug described above is completely invisible to CI. There should be at least one integration-style test that runs two threads against a real (or fakeredis) instance.

```python
import fakeredis
import concurrent.futures

def test_concurrent_reservations_are_serialized():
    r = fakeredis.FakeRedis()
    # ... set up real db mock with shared state ...
    svc = InventoryService(r, db)
    with concurrent.futures.ThreadPoolExecutor(max_workers=5) as pool:
        futures = [pool.submit(svc.reserve_item, "item_001", 1) for _ in range(5)]
        results = [f.result() for f in futures]
    successful = [r for r in results if r.success]
    assert len(successful) == 3  # only 3 units in stock
```

---

#### 🟢 Low — Daemon Thread Leak on Process Shutdown

Daemon threads are silently killed on interpreter exit, which is acceptable, but if `InventoryService` is used in a context where the process stays alive (e.g., a long-running worker), a heartbeat thread that loses its stop signal due to the race condition above will run indefinitely, polling Redis. After the thread-safety fix, `join` with a timeout in the `finally` block addresses this.
