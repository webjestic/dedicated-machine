Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The most severe issues are a non-atomic lock release (TOCTOU race in the finally block), instance-level heartbeat state that breaks under concurrent calls, and missing database transaction atomicity. These are not minor concerns — under real flash sale load, they represent exploitable race conditions and potential inventory corruption.

---

## Critical Issues

### 1. Non-Atomic Lock Release — TOCTOU Race Condition

**Severity: Critical**

```python
# services/inventory.py — _acquire_lock finally block
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # <-- delete (not atomic with read)
```

Between `GET` and `DELETE`, another process can acquire the lock. You would then delete a lock you no longer own, causing a cascading failure where two processes believe they hold the lock simultaneously.

**Required fix:** Use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register on __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

This is the canonical pattern from the Redis documentation and is non-negotiable for a correct distributed lock.

### 2. Instance-Level Heartbeat State Breaks Under Concurrency

**Severity: Critical**

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If `reserve_item` is called concurrently from multiple threads sharing the same `InventoryService` instance (the normal deployment pattern), one call's `_stop_heartbeat()` will kill the heartbeat belonging to a different concurrent call. The lock that was supposed to be kept alive will expire, and the operation it was protecting will proceed without a valid lock.

**Required fix:** Heartbeat state must be scoped to each lock acquisition, not to the instance. Pass state through a local object or closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not active.is_set():
                break
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    # ... acquire lock ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script
```

Note also that the current implementation never calls `thread.join()`, meaning the heartbeat thread can outlive the lock context and attempt to extend a lock that has already been released or re-acquired by another owner.

### 3. Database Operations Are Not Atomic

**Severity: Critical**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists. The item is effectively lost. The distributed lock protects against concurrent Redis access, but it does not substitute for database-level atomicity.

**Required fix:**

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract must expose a transaction context manager, and this must be enforced at the service layer.

## Significant Issues

### 4. Quantity Is Not Validated

**Severity: High**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that `quantity` is a positive integer. A caller passing `quantity=0` would succeed and create a zero-unit order. A caller passing `quantity=-5` would *increase* stock (`new_stock = current_stock - (-5)`). This is a potential inventory manipulation vector.

**Required fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

The PR checklist claims "Input validation via Pydantic" but the `ReservationRequest` model does not exist. `ReservationResult` is an output model only. This checklist item is false.

---

### 5. `item_id` Is Not Sanitized Before Use as a Redis Key

**Severity: High**

```python
lock_key = f"inventory:lock:{item_id}"
```

`item_id` is interpolated directly into a Redis key with no validation. A caller supplying `item_id = "../../admin"` or `item_id = "x:lock:other_item"` can target arbitrary keys in the Redis keyspace. Combined with the missing quantity validation, this represents a meaningful attack surface.

**Required fix:** Validate `item_id` against a strict allowlist pattern (alphanumeric, hyphens, underscores, bounded length) before constructing any Redis key.

### 6. Heartbeat Has a Silent Race at Startup

**Severity: Medium**

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
# ...
self._start_heartbeat(lock_key, token)
# heartbeat thread sleeps HEARTBEAT_INTERVAL before first extend
```

The heartbeat sleeps *first*, then extends. With `LOCK_TTL=30` and `HEARTBEAT_INTERVAL=10`, the first extension happens at T+10s, which is fine under normal conditions. However, if the system clock skews or Redis is under memory pressure and evicts keys early, there is no protection. More importantly, the heartbeat should extend *before* sleeping, or use a tighter ratio. The current `HEARTBEAT_INTERVAL / LOCK_TTL` ratio of 1/3 is acceptable but should be documented as a deliberate choice, not an accident.

---

### 7. Bare `RuntimeError` Is Too Broad in the Exception Handler

**Severity: Medium**

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is raised for lock contention, but it is also the base class for many other runtime failures. A programming error inside `_acquire_lock` (e.g., a Redis connection error that gets wrapped, or a future code change) will be silently swallowed and returned as "service unavailable," masking real bugs. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

Raise and catch only that. Let unexpected exceptions propagate so they surface in monitoring.

## Test Coverage Issues

### 8. Tests Mock Away the Exact Behavior Being Tested

**Severity: High**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat entirely disabled
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most complex and failure-prone component in this PR. Mocking it out entirely means there is zero test coverage for the thread lifecycle, the TOCTOU race in lock release, or the concurrent-call corruption described in Issue #2. These are precisely the scenarios that need integration-level tests.

### 9. The Lock Release Test Tests the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test validates the current broken behavior (a non-atomic GET then conditional DELETE) rather than the correct behavior (an atomic Lua CAS). Once the lock release is fixed with a Lua script, this test will need to be rewritten to assert that the script was called with the correct arguments. As written, it encodes the flaw as a passing test.
