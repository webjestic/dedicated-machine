Decision: Request Changes

Summary: The implementation demonstrates a reasonable understanding of distributed locking primitives and the heartbeat pattern is a thoughtful inclusion, but there are several architectural flaws that range from subtle race conditions to outright correctness violations that would be dangerous in a high-concurrency flash sale environment. The test suite, while structured well, contains a critical mock misconfiguration that causes it to silently validate incorrect behavior. None of these are acceptable for production use in a government contracting context where correctness and auditability are non-negotiable.

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the submission.

```python
# InventoryService is presumably a singleton or shared service instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as **instance variables**, not scoped to the lock acquisition context. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — the following race is trivially reproducible:

- Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a different item's lock, starts its heartbeat (overwrites `_heartbeat_thread`)
- Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- **Thread B's heartbeat is now silently dead.** Thread B's lock will expire mid-operation.

This is a silent correctness failure. The lock will expire, another process can acquire it, and you now have two concurrent writers to the same inventory record. In a flash sale context, this is an oversell vector.

**Fix:** Heartbeat state must be scoped to the lock context, not the instance. Pass a threading `Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        stop_event.set()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note also that `_stop_heartbeat` does not join the thread. The heartbeat thread can continue executing for up to `HEARTBEAT_INTERVAL` seconds after the lock is released, potentially re-extending a lock that was intentionally released or already acquired by another process.

### 2. Non-Atomic Lock Release — TOCTOU Race

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # <-- conditional delete
```

This is a textbook **check-then-act** race condition. Between the `GET` and the `DELETE`, the lock TTL could expire and be acquired by another process. You would then delete a lock you no longer own.

This is a well-known problem with Redis locking. The canonical solution is a **Lua script** that performs the compare-and-delete atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same atomicity concern applies to the heartbeat's `GET` + `EXPIRE` sequence, though the consequence there is less severe (failing to extend is safer than deleting someone else's lock).

### 3. Database Operations Are Not Atomic — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises an exception, you have decremented inventory with no corresponding order record. The item is effectively lost. The distributed lock protects against concurrent access but does nothing to protect against partial writes within the critical section.

These operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the database layer, but the absence of any transactional boundary here is a data integrity defect, not a style issue.

### 4. Missing Input Validation

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw, unvalidated arguments:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that:
- `item_id` is non-empty or matches an expected format (e.g., alphanumeric, bounded length). An empty string or a string containing Redis glob characters could be problematic in the lock key.
- `quantity` is a positive integer. `quantity=0` would succeed and create a zero-unit order. `quantity=-5` would *increase* stock silently.

```python
if not item_id or not item_id.isalnum():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

The Pydantic model (`ReservationResult`) is used only for the *output*, not the input. The checklist item is misleading.

## Moderate Issues

### 5. Test Suite Has a Silent Mock Misconfiguration

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.get.return_value = b"test-token"   # <-- hardcoded bytes
```

The lock token is generated via `uuid.uuid4()` at runtime. The mock always returns `b"test-token"`, which will never equal the generated token. This means the lock release path (`redis.delete`) is **never exercised** in any test that uses this fixture. The test `test_lock_not_released_if_expired` is intended to test the "token mismatch" case, but every test is silently hitting that same path.

`test_reserve_success` is asserting that the reservation succeeds but is not actually verifying that the lock was released. This is a coverage gap masquerading as a passing test.

To fix this properly, the token needs to be injectable or the mock needs to be configured to return whatever was stored via `set`:

```python
stored = {}

def mock_set(key, value, nx=False, ex=None):
    if nx and key in stored:
        return False
    stored[key] = value
    return True

def mock_get(key):
    return stored.get(key, None)

redis_client.set.side_effect = mock_set
redis_client.get.side_effect = mock_get
```

### 6. Heartbeat Interval Is Too Close to Lock TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T+10 and T+20. At T+20, after the extension, the lock is valid until T+50. This is acceptable in isolation, but the heartbeat uses `time.sleep()` which is not precise under GIL contention or system load. Under a heavily loaded Python process, the sleep could drift significantly. The safety margin should be larger, or the heartbeat should use a monotonic clock reference rather than relying on sleep precision.

### 7. No Redis Connection Error Handling

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

`redis.exceptions.ConnectionError` and `redis.exceptions.TimeoutError` are not caught anywhere. A Redis blip during a flash sale will produce an unhandled exception that propagates out of `reserve_item` entirely, bypassing the `RuntimeError` handler and returning nothing to the caller. At minimum, Redis exceptions should be caught and mapped to a `ReservationResult(success=False, ...)`.

## Minor Issues

- The `_beat` function sleeps *before* the first extension check. If a DB operation takes longer than `LOCK_TTL - HEARTBEAT_INTERVAL` seconds before the first heartbeat fires, the lock can still expire. Consider extending immediately upon acquisition or reducing the initial sleep.
- `daemon=True` on the heartbeat thread means it will be killed abruptly on interpreter shutdown without releasing the lock. This is probably acceptable behavior but should be documented.
- No maximum bound on `quantity` is enforced. A caller could reserve the entire stock in one call, which may or may not be intended for flash sale semantics.
