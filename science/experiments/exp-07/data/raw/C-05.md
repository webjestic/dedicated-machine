Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The heartbeat design has a race condition that can cause lock theft, the service is not safe for concurrent use across multiple callers within the same process, and the database operations lack atomicity guarantees. These issues must be resolved before this code is merged.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the submission. The heartbeat state is stored on the instance, not scoped to the lock acquisition:

```python
# These are instance variables, not call-scoped variables
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` concurrently on the **same** `InventoryService` instance — which is the normal deployment pattern for a shared service object — the following sequence occurs:

1. Thread A acquires lock, starts heartbeat, sets `self._heartbeat_active = True`
2. Thread B acquires a different lock, calls `_start_heartbeat`, overwrites `self._heartbeat_thread`
3. Thread A's `finally` block calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`
4. Thread B's heartbeat is now silently dead — its lock will expire mid-operation
5. Thread A's `_heartbeat_thread` reference is now pointing at Thread B's thread object

The heartbeat state must be encapsulated per lock acquisition, not per instance. A dataclass or a simple closure-scoped flag should be used:

```python
# Correct pattern: scope the flag to the lock context
heartbeat_active = threading.Event()
heartbeat_active.set()

def _beat():
    while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...
        pass
```

Additionally, `threading.Event` should replace the bare boolean flag. A raw boolean is not guaranteed to be visible across threads without a memory barrier on all Python implementations, even though CPython's GIL makes this practically safe today. Writing correct concurrent code means not relying on implementation-specific behavior.

### 2. Heartbeat Race Window — Lock Can Be Stolen Before Extension

Inside `_beat()`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a classic TOCTOU (time-of-check/time-of-use) race. Between the `GET` and the `EXPIRE`, the lock TTL can expire and be acquired by another process. You are then extending a lock you no longer own, silently evicting the new legitimate holder.

This **must** be an atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same TOCTOU problem exists in the lock release path:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a well-known pattern that Redis documentation explicitly warns against. The canonical fix is the same Lua script approach, which is also what `redis-py`'s own `Lock` implementation uses. There is no excuse for reimplementing this incorrectly when the correct pattern is documented and available.

### 3. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is killed, or the database throws after `update_stock` but before `create_order`, you have decremented stock with no corresponding order record. This is silent inventory loss.

The Redis lock provides mutual exclusion at the application layer, but it does not provide durability or atomicity at the persistence layer. These calls must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager, or this code is not safe for production regardless of how correct the locking is.

### 4. No Input Validation Despite Checklist Claiming Otherwise

The checklist marks "Input validation via Pydantic" as complete. This is false. `ReservationResult` is a Pydantic output model. There is no validation on the inputs to `reserve_item`:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is not validated for length, character set, or format. An attacker can pass a crafted string that becomes a malicious Redis key, e.g., `inventory:lock:../../admin` or an extremely long string causing memory pressure.
- `quantity` has no lower bound check. `quantity=0` will succeed and create a zero-unit order. `quantity=-5` will **increase** stock silently: `new_stock = 10 - (-5) = 15`.

```python
if current_stock < quantity:  # 10 < -5 is False — passes the check
```

This is a serious logic error. A Pydantic input model should be defined and used:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Significant Issues

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The thread is set as `daemon=True`, which means it will be killed when the main process exits, but it will continue running until the next `HEARTBEAT_INTERVAL` sleep cycle completes. In a test or short-lived context this leaks threads. More importantly, there is no guarantee the heartbeat has actually stopped before the lock is released. The thread could wake up and attempt to extend a lock that has already been deleted and re-acquired by another process.

After setting the flag, the thread must be joined with a timeout:

```python
def _stop_heartbeat(self, thread: threading.Thread, event: threading.Event) -> None:
    event.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 6. `RuntimeError` Catch Scope Is Too Broad

```python
try:
    with self._acquire_lock(lock_key):
        ...
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is caught across the entire block, including the database operations. If `db.update_stock` raises a `RuntimeError` after stock has been read but before the order is created, the caller receives a "service unavailable" message while the stock may or may not have been modified depending on where exactly the failure occurred. Database exceptions should be caught separately and explicitly, and should trigger a rollback, not a generic unavailability message.

---

### 7. Lock Key Namespace Is User-Controlled Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

Without the input validation described in issue 4, `item_id` is passed directly into the Redis key. A caller passing `item_id = "foo:bar"` produces key `inventory:lock:foo:bar`, which may collide with other key namespaces. A caller passing a very long string causes an oversized key. Sanitize or validate before constructing the key.

## Test Coverage Issues

### 8. Tests Mock Away the Exact Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking out the heartbeat entirely means the tests provide zero coverage of the most complex and failure-prone part of this implementation. At minimum there should be:

- A test that verifies the heartbeat actually extends the TTL
- A test that verifies the heartbeat stops after the context manager exits
- A test that verifies the TOCTOU-safe release behavior under a simulated expiry

### 9. `test_lock_not_released_if_expired` Tests the Wrong Behavior

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test asserts that `delete` is never called when the token does not match. But because the heartbeat is mocked out and the token generated inside `_acquire_lock` is a fresh UUID that will never equal `b"different-token"`, this test is asserting the correct behavior for the wrong reason. The mock token is hardcoded in the fixture as `b"test-token"`, but the actual token used during lock acquisition is a UUID generated at runtime. This test would pass even if the ownership check were completely removed, because `redis_client.get` returns `b"different-token"` which will never match any UUID. The test is not actually validating the ownership check logic.
