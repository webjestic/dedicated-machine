Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The most severe issues involve non-atomic lock release, instance-level heartbeat state that breaks under concurrent calls, and missing database transaction atomicity — any of which could cause inventory corruption or overselling. These must be resolved before this code is merged.

---

## Critical Issues

### 1. Non-Atomic Lock Release (Race Condition / Security Vulnerability)

```python
# CURRENT — broken, two separate round-trips
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock after the TTL expires. You then delete a lock you no longer own. This is a textbook TOCTOU race condition.

**Required fix:** Use the canonical Lua script for atomic compare-and-delete, which Redis guarantees executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

The same non-atomic pattern appears in the heartbeat:

```python
# Also broken
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This must also be replaced with a Lua script that atomically checks ownership before extending TTL.

### 2. Instance-Level Heartbeat State Breaks Under Concurrency

```python
# InventoryService instance fields
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the service instance, not scoped to an individual lock acquisition. If two threads call `reserve_item()` concurrently on the same `InventoryService` instance — which is the entire point of this service — the following sequence destroys both heartbeats:

1. Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
2. Thread B acquires a different lock, starts heartbeat, overwrites `_heartbeat_thread`
3. Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
4. Thread B's heartbeat is now dead. Its lock will expire. Inventory corruption follows.

**Required fix:** Heartbeat state must be scoped to the lock context, not the instance. Pass a threading `Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquisition logic ...

    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)

def _run_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        self._extend_lock(lock_key, token)
```

### 3. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. The item is silently lost. The distributed lock only serializes access — it does not provide atomicity across DB operations.

**Required fix:** These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the atomicity requirement is non-negotiable for financial/inventory operations.

### 4. Input Validation Is Absent Despite Checklist Claiming Otherwise

The checklist marks "Input validation via Pydantic" as complete, but there is no validation on the inputs to `reserve_item`. `item_id` and `quantity` are used directly without any sanitization or constraint checking.


## Significant Issues

### 5. Heartbeat Thread Is Joined Nowhere — Leaked Threads on Failure

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is never joined
```

The heartbeat thread is set as `daemon=True`, which means it will be killed when the main process exits, but it will continue running until then. Under the fixed design (Issue #2), the thread must be explicitly joined after the stop event is set to ensure it has actually terminated before the lock is released. Failing to join means the heartbeat could extend a lock after the `finally` block has already decided to release it.

---

### 6. Retry Backoff Has No Jitter — Thundering Herd Under Load

```python
backoff = (2 ** attempt) * 0.1
time.sleep(backoff)
```

During a flash sale, hundreds of threads will fail to acquire the lock simultaneously. They will all retry at the same intervals (0.1s, 0.2s, 0.4s), reconverging on the lock at the same moment on every retry cycle. This is the thundering herd problem and will cause Redis to be hammered in synchronized bursts.

**Required fix:** Add random jitter:

```python
import random
backoff = (2 ** attempt) * 0.1
jitter = random.uniform(0, backoff * 0.5)
time.sleep(backoff + jitter)
```

---

### 7. `time.sleep()` in Lock Acquisition Blocks the Calling Thread

For a service expected to handle high-concurrency flash sales, blocking threads with `time.sleep()` during retry is a serious throughput concern. If this service runs in a thread pool, sleeping threads consume worker slots. This is an architectural concern that should be addressed — either by using async I/O (`asyncio`) or by documenting explicitly that the caller must account for blocked threads in pool sizing.

## Test Coverage Issues

### 8. Tests Mock Away the Exact Code That Needs Testing

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat and lock release are the most failure-prone parts of this system. Mocking them out entirely means the tests provide no confidence in the behavior that matters most. There must be at minimum:

- A test that verifies the Lua release script is called, not a GET+DELETE pair
- A test that verifies two concurrent `reserve_item` calls on the same instance do not corrupt each other's heartbeat state
- A test that verifies a DB failure in `create_order` does not result in a committed `update_stock`

### 9. `test_lock_not_released_if_expired` Tests the Wrong Behavior

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test validates the current broken GET+DELETE pattern. Once the Lua script is implemented, this test needs to be rewritten to verify the Lua script receives the correct token and that a non-matching token results in a no-op delete, not that Python-side logic skips the call.
