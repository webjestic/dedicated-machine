Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic NX lock acquisition, token-based ownership verification, and heartbeat renewal are all correct instincts. However, there are several architectural flaws ranging from critical race conditions to instance-level state corruption that would make this unsafe under the concurrent load it was explicitly designed to handle. These issues must be resolved before this can be approved for a government-facing flash sale system where inventory integrity is non-negotiable.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the submission. The heartbeat state is stored on the instance, not scoped to the lock context:

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item()` concurrently on the **same** `InventoryService` instance — which is the exact scenario this service was built for — the following happens:

- Thread A acquires a lock, starts heartbeat, sets `self._heartbeat_active = True`
- Thread B acquires a different lock, starts its own heartbeat, also sets `self._heartbeat_active = True`
- Thread A finishes, calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`
- Thread B's heartbeat loop now silently dies, leaving Thread B's lock unrenewed and vulnerable to expiry mid-operation

The heartbeat state must be scoped to the lock context, not the instance. The fix is to encapsulate the heartbeat lifecycle into a local closure or a dedicated lock-context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local scope, not instance state

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 2.0)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_active.set()

    def _beat():
        while heartbeat_active.is_set():
            heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL)
            if not heartbeat_active.is_set():
                break
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

### 2. Lock Release Is Not Atomic — TOCTOU Race Condition

The lock release performs a read-then-delete as two separate Redis commands:

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # step 2: delete
```

Between step 1 and step 2, the lock TTL could expire and be re-acquired by another process. This service would then delete a lock it no longer owns, corrupting the mutual exclusion guarantee entirely. This is a textbook TOCTOU (time-of-check to time-of-use) race.

The canonical fix is a Lua script executed atomically on the Redis server:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

This is not optional. The entire correctness guarantee of the lock depends on atomic release.

### 3. No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation whatsoever:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` — a caller passing `quantity=0` or `quantity=-5` would succeed, decrementing stock by a negative number and **increasing** inventory
- `item_id` being an empty string, which would produce a lock key of `inventory:lock:` and a nonsensical DB lookup
- Excessively large quantities that could drain all stock in a single call

Pydantic is imported but unused in the service logic. Either introduce a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

Or add explicit guards at the top of `reserve_item`. The negative quantity case in particular is a potential inventory manipulation vulnerability.

## Significant Issues

### 4. Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The thread is set to `daemon=True`, which means it will be killed abruptly on process exit, but during normal operation it continues running for up to `HEARTBEAT_INTERVAL` seconds after the lock is released. In the current broken instance-state design, this means a stopped heartbeat thread could race with a newly started one on the next call. Even after fixing issue #1, the thread should be joined with a bounded timeout to ensure clean teardown.

---

### 5. Heartbeat Uses `time.sleep` — Not Interruptible

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 full seconds
        if self._heartbeat_active:
            ...
```

The thread sleeps for the full interval before checking the stop flag. After `_stop_heartbeat()` is called, the thread can linger for up to 10 seconds. This compounds the thread lifecycle issue above. Use `threading.Event.wait(timeout=...)` instead, which can be interrupted immediately when the event is cleared.

---

### 6. Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock has been decremented but no order record exists. The item is effectively lost. This is a partial-write failure with no recovery path.

These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock protects Redis-level concurrency, but it does not protect against database-level partial failures. These are orthogonal concerns and both must be addressed.

---

### 7. Bare `RuntimeError` for Lock Failure Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a generic exception that could be raised by the database layer, the Redis client, or any number of internal failures — all of which would be silently swallowed and returned as "service unavailable." Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass
```

Raise it explicitly in `_acquire_lock` and catch only that. Let unexpected exceptions propagate so they surface in monitoring.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is mocked in every test, meaning the thread lifecycle — the most operationally dangerous part of this implementation — has zero test coverage. There should be at least one integration-style test that exercises the actual heartbeat path, verifying that `redis.expire` is called and that the thread terminates cleanly after the context exits.

### 9. Atomic Release Is Not Tested

The test `test_lock_not_released_if_expired` verifies the conditional logic but does so against a non-atomic implementation. Once the Lua script is introduced, this test needs to verify the script is called, not the individual `get`/`delete` sequence.

### 10. No Test for Negative or Zero Quantity

Once input validation is added, there must be tests confirming that `quantity=0` and `quantity=-1` are rejected before any lock is acquired or DB call is made.

## Minor Notes

- The backoff cap is missing. `(2 ** attempt) * 0.1` with `retry_attempts=3` only reaches 0.4s maximum, which is acceptable here, but if `retry_attempts` is ever increased by a future developer the backoff becomes unbounded. Add `min(..., MAX_BACKOFF)`.
