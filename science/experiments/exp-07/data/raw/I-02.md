Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — atomic SET NX, token-verified release, heartbeat renewal, and backoff retry are all directionally correct. However, there are several architectural flaws, race conditions, and missing safeguards that make this code unsuitable for a high-concurrency flash sale environment where correctness is non-negotiable. The issues range from a critical instance-level concurrency bug to absent atomicity guarantees on lock release and unvalidated inputs that could corrupt inventory state.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe defect in the submission.

```python
# On the InventoryService instance:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are **instance-level attributes**, not scoped to an individual lock acquisition. If a single `InventoryService` instance handles concurrent calls — which is the entire point of a flash sale service — two simultaneous `reserve_item` calls will share this state. The sequence below is not hypothetical; it is nearly guaranteed under load:

```
Thread A: acquires lock_A → sets _heartbeat_active = True → starts heartbeat_A
Thread B: acquires lock_B → sets _heartbeat_active = True (no-op, already True)
Thread A: finishes → calls _stop_heartbeat() → sets _heartbeat_active = False
Thread B: heartbeat_B now exits silently — lock_B will expire unrenewed
Thread B: DB operation still running → lock expires → another process steals lock_B
Thread B: writes stock decrement to DB while another process holds the lock
          → DOUBLE RESERVATION, INVENTORY CORRUPTION
```

The heartbeat must be encapsulated per-acquisition, not per-instance. The fix is to pass a local `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to THIS acquisition
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 5.0)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

### 2. Non-Atomic Lock Release — TOCTOU Race

```python
# In _acquire_lock finally block:
current = self.redis.get(lock_key)          # ← read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← write — NOT atomic with the read
```

Between the `GET` and the `DELETE`, another process can acquire the lock if it expired in that window. You then delete a lock you no longer own. This is a well-documented Redis distributed lock pitfall. The canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU problem exists in the heartbeat's renewal check:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # non-atomic
```

This should also be a Lua script or use `SET key token XX EX ttl` which is atomic.

### 3. No Input Validation on `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation whatsoever:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:

- `quantity <= 0` — a caller passes `quantity=0` and successfully "reserves" nothing while holding a distributed lock, or passes `quantity=-5` and **increases stock**
- `quantity` being a float that slips past the type hint at runtime
- `item_id` being an empty string, which would produce the lock key `inventory:lock:` — a collision vector

Minimum required:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError(f"quantity must be a positive integer, got {quantity!r}")
```

A negative quantity decrementing stock in the wrong direction under a held lock in a flash sale is a serious financial and integrity concern.

## Significant Issues

### 4. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The thread is marked `daemon=True`, so it won't block process exit, but it will continue running for up to `HEARTBEAT_INTERVAL` (10 seconds) after the lock is released. During that window, if a new lock is acquired on the same key by another process, the orphaned heartbeat could extend that new process's lock with a stale token check — which would fail silently but represents unnecessary Redis traffic and a latent confusion risk. After fixing issue #1 with a scoped `stop_event`, call `thread.join(timeout=...)` in the `finally` block.

---

### 5. DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock has been decremented but no order exists. The lock is released, the reservation is effectively lost, and the item is gone from inventory with no record. This requires either:

- A database transaction wrapping both operations, or
- A compensating rollback in an `except` block that restores stock before re-raising

This is not a Redis concern — it is a DB consistency concern that the distributed lock does not protect against.

---

### 6. `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

In an async or thread-pool-based web server, blocking the calling thread during retry sleep is expensive. The backoff ceiling is also unbounded — at `retry_attempts=3`, the maximum sleep is `(2^2) * 0.1 = 0.4s`, which is acceptable, but if `retry_attempts` is ever increased without a cap, this becomes a problem. Add `min(..., MAX_BACKOFF)`:

```python
MAX_BACKOFF = 2.0  # seconds
backoff = min((2 ** attempt) * 0.1, MAX_BACKOFF)
```

## Test Coverage Gaps

### 7. The Most Critical Bug Has No Test

There is no test exercising concurrent access through a **single shared `InventoryService` instance**. Every test creates an isolated fixture. The instance-level heartbeat corruption described in issue #1 would not be caught by any test here. A test using `concurrent.futures.ThreadPoolExecutor` with a shared instance is required.

### 8. Heartbeat Is Silently Mocked Away

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat entirely means the heartbeat logic has zero test coverage. At minimum, the heartbeat's token-check and renewal behavior should be tested in isolation with a real (or fakeredis) client.

### 9. Lock Release Atomicity Is Not Tested

`test_lock_not_released_if_expired` sets a different token and asserts `delete` is not called — but this tests the non-atomic `GET`+`DELETE` path, which is itself the bug. Once the release is fixed to use a Lua script, this test needs to be rewritten to verify the script is invoked correctly.

## Minor Notes

- `logger = logging.getLogger(__name__)` at module level is correct, but the service should document that callers are responsible for configuring handlers. No issue, just worth noting in internal docs.
- The `LOCK_TTL = 30` comment says "generous TTL for slow DB operations" — 30 seconds is reasonable, but this should be configurable via constructor parameter rather than a class-level constant, since different deployment environments will have different DB latency profiles.
- `ReservationResult` using Pydantic is good. Consider adding `model_config = ConfigDict(frozen=True)` to make results immutable after creation.
