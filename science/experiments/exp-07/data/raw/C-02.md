Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The most severe issues involve non-atomic lock release, instance-level heartbeat state that breaks under concurrency, and a complete absence of database transaction guarantees. These are not minor concerns — they are failure modes that will manifest under exactly the load conditions this service is designed to handle.

---

## Critical Issues

### 1. Non-Atomic Lock Release (Race Condition — CVE-Class Bug)

```python
# DANGEROUS — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and `DELETE`, another process can acquire the lock after the TTL expires. You then delete a lock you no longer own. This is a well-documented failure mode. The canonical fix is a Lua script executed atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

In Python:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same non-atomic pattern exists in the heartbeat:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two lines
```

This also needs a Lua script or use of `SET key value XX EX ttl` with token validation.

### 2. Instance-Level Heartbeat State Breaks Under Any Concurrency

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the instance, not scoped to a lock acquisition. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — the following happens:

- Thread A acquires a lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a different lock, starts heartbeat (overwrites `_heartbeat_thread`)
- Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now silently dead, and its lock will expire

This is a silent failure. The lock appears held, the heartbeat appears running, but the lock will expire mid-operation. The fix is to scope heartbeat state to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # atomic extend via Lua
            ...

    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

### 3. No Database Transaction — Partial Write Is Possible

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order exists. This is an inventory leak. The distributed lock protects against concurrent access, but it does not protect against partial writes within a single acquisition. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The PR makes no mention of this, and the tests do not cover it.

---

### 4. Bare `RuntimeError` Catch Masks Unrelated Exceptions

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is too broad. If `db.update_stock` raises a `RuntimeError` (database connection failure, ORM error, etc.), it will be silently swallowed and returned as "service unavailable" — with the stock already decremented. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError in reserve_item
```

## Significant Issues

### 5. Input Validation Is Claimed But Not Present

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is never sanitized — if it's used in a Redis key or passed to the DB without escaping, this is a potential injection vector. `quantity` has no lower bound check; `quantity=0` or `quantity=-5` would pass silently and produce nonsensical results. A proper input model should be used:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The flag is set to `False` but the thread is never joined. The thread will continue sleeping through its current `time.sleep(HEARTBEAT_INTERVAL)` interval before it checks the flag. In a test or shutdown scenario this is a resource leak. More critically, in the concurrent scenario described in issue #2, the orphaned thread may attempt to extend a lock it no longer owns after a new acquisition has started.

---

### 7. HEARTBEAT_INTERVAL Is More Than Half of LOCK_TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

While the ratio here is acceptable (10s interval, 30s TTL), this is a class-level constant with no enforcement of the invariant `HEARTBEAT_INTERVAL < LOCK_TTL / 2`. If someone changes `LOCK_TTL` to 15 without adjusting the interval, the lock will expire before the heartbeat fires. Add a constructor assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
    "HEARTBEAT_INTERVAL must be less than half of LOCK_TTL"
)
```

## Test Coverage Gaps

### 8. Tests Mock Away the Exact Code Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat is acceptable for unit tests of reservation logic, but there are zero tests for the heartbeat itself — no test verifies that `redis.expire` is called with the correct token, no test verifies that the heartbeat stops when the lock is released, and no test covers the concurrent instance scenario described in issue #2.

### 9. The Lock Release Test Is Incorrect

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test patches `redis.get` to return a different token globally, which means the token check inside `_acquire_lock` also sees the wrong token — but the lock was already "acquired" via `redis.set` returning `True`. The test is asserting behavior of the release path while inadvertently corrupting the acquire path. The test passes for the wrong reasons and would pass even if the release logic were completely removed.

### 10. No Test for Partial DB Failure

There is no test covering `update_stock` succeeding followed by `create_order` raising an exception. This is the most financially damaging failure mode in the service and it is entirely untested.

---

## Minor Issues

- `time.sleep()` is called inside the context manager during retry backoff, which blocks the calling thread. Under high flash sale load this compounds latency. Consider whether async or a queue-based approach is more appropriate for this use case.
- The `db` parameter has no type annotation or interface definition. There is no `Protocol` or abstract base class defining the expected contract for `get_stock`, `update_stock`, and `create_order`. This makes the dependency opaque and untestable against a real interface.
- `logger.warning` is emitted on the first retry attempt (attempt 0), which will produce noise in normal operation. First retry should arguably be `DEBUG`, with `WARNING` reserved for repeated contention.
