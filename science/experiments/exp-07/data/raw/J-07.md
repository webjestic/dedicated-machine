Decision: Request Changes

Summary: The implementation has several meaningful flaws that need to be addressed before merging. The most critical is a thread-safety issue with the heartbeat mechanism — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields, potentially stopping a sibling reservation's heartbeat or orphaning threads entirely. Beyond that, the lock release in the `finally` block is not atomic — there is a TOCTOU window between the `GET` and `DELETE` where another process could steal the lock and have it deleted out from under them; this should be a Lua script or a `redis.delete` via a pipeline with a watch. There are also secondary concerns around input validation, error handling scope, and test coverage gaps that need to be resolved.

---

### Critical Issues

**1. Instance-level heartbeat state is not concurrency-safe**

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` concurrently on the same instance (which is the normal case for a service object), the second call overwrites `_heartbeat_active` and `_heartbeat_thread`. The first reservation's heartbeat is silently killed, and its lock can expire mid-operation — exactly the failure mode this feature is supposed to prevent.

```python
# Fix — scope heartbeat state to the individual lock acquisition
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                break  # lock was lost; stop beating

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

Note the `thread.join()` — without it, the daemon thread can fire one more extension *after* the lock has been deleted, re-creating the key with a new TTL and leaving a ghost lock. The `Event`-based sleep also means the thread wakes immediately on shutdown rather than waiting out the full interval.

**2. Lock release is not atomic (TOCTOU race)**

```python
# Current — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock. Your `DELETE` then removes their lock, not yours. This is a well-known Redis distributed lock pitfall and must be fixed with a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at class or module level
_release_lock_script = None  # initialized lazily or in __init__

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

Then in the `finally` block:

```python
finally:
    heartbeat_active.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    released = self._release_lock(lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### Significant Issues

**3. No input validation on `reserve_item`**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, whitespace, or a path-traversal-style string that becomes a malformed Redis key. `quantity` could be zero or negative, which would silently pass the stock check and create a nonsense order.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id and req.quantity from here
```

**4. Bare `except` scope is too narrow — DB exceptions are unhandled**

```python
try:
    with self._acquire_lock(lock_key):
        current_stock = self.db.get_stock(item_id)
        ...
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except RuntimeError as e:
    ...
```

Only `RuntimeError` (lock failure) is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` — network timeout, ORM error, constraint violation — propagates uncaught to the caller. Worse, if `update_stock` succeeds but `create_order` raises, stock is decremented with no order record. There is no transaction or compensating action.

```python
# Minimum fix — catch and log DB exceptions
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

The real fix is wrapping `update_stock` and `create_order` in a DB transaction so they commit or roll back together. That is an architectural requirement, not optional.

### Minor Issues

**5. `test_lock_not_released_if_expired` tests the wrong token**

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

Because `_start_heartbeat` is mocked out, `redis.get` is only called during lock release. The token generated inside `_acquire_lock` is a fresh `uuid4`, but the mock always returns `b"different-token"`, so the check fails and `delete` is skipped — the assertion passes. However, this test does not actually exercise the Lua-script path once that fix is applied. Update the test to use the scripted release and verify the return value.

**6. Heartbeat interval is more than half the TTL**

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 10`. The heartbeat fires at 10 s, which is fine. But if the heartbeat thread is delayed by GIL pressure or system load, the next fire is at 20 s, leaving only 10 s of margin. Consider `HEARTBEAT_INTERVAL = LOCK_TTL // 3` as a named relationship so the margin is explicit and maintained if TTL changes.

**7. No test for concurrent access**

Given that concurrency is the entire point of this service, there should be at least one test that spins up two threads calling `reserve_item` simultaneously and asserts that stock is not double-decremented. This is the scenario the instance-level heartbeat bug above would have caught.
