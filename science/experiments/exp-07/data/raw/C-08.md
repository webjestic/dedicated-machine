Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The heartbeat design has a race condition that can cause split-brain lock ownership, the service is not safe for concurrent use across multiple reservations, and the database operations lack atomicity guarantees. These issues must be resolved before this code is merged.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Not Concurrency Safe

This is the most severe flaw in the implementation.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes. If two coroutines or threads share the same `InventoryService` instance and call `reserve_item` concurrently, the second call to `_start_heartbeat` will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently orphaning the first heartbeat thread. Worse, when `_stop_heartbeat` is called by either path, it sets `_heartbeat_active = False`, which kills *both* heartbeats simultaneously.

The heartbeat state must be scoped to each individual lock acquisition, not to the service instance. The fix is to pass a mutable container (e.g., a threading.Event or a local dict) into the heartbeat closure.

```python
# Correct pattern — heartbeat state is local to each lock acquisition
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to THIS acquisition
    acquired = False
    ...
    def _beat():
        while not stop_event.wait(self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 2. Heartbeat Race Condition — Lock Can Be Extended After Release

Inside `_beat()`:

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps FIRST
        if self._heartbeat_active:            # checks AFTER sleep
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The thread sleeps for 10 seconds, then wakes up and calls `redis.expire`. Between `_stop_heartbeat()` setting `_heartbeat_active = False` and the thread actually checking it, there is a window where the thread can still issue `redis.expire` on a lock that has already been deleted and potentially re-acquired by another process. This extends a lock you no longer own.

The `stop_event.wait()` pattern above resolves this. Additionally, the get-then-expire sequence is not atomic. Another process could delete the key between your `GET` and your `EXPIRE`. This should be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 3. Lock Release Is Also Not Atomic

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # DELETE — not atomic with GET
```

Between the `GET` and the `DELETE`, the lock TTL could expire and be acquired by another process. You then delete a lock you no longer own, corrupting the mutual exclusion guarantee. This is a well-known failure mode and must use a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is not a theoretical concern — under the load profile of a flash sale, this window will be hit.

### 4. Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `create_order` raises an exception after `update_stock` succeeds, stock has been decremented but no order exists. The item is lost. There is no try/except, no rollback, and no compensating transaction.

This must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock protects against concurrent Redis-level races, but it does not protect against partial database writes.

### 5. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative units would decrement stock incorrectly or pass the stock check trivially)
- `item_id` being an empty string
- `quantity` being an absurdly large integer

Pydantic is imported but only used for the output model. A request model should validate inputs:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

A `quantity=0` call right now would pass the stock check (`10 < 0` is False) and create a zero-unit order. A `quantity=-1` call would *increase* stock.

## Significant Issues

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The thread is set as `daemon=True`, which means it will be killed when the main process exits, but during normal operation it is never joined. If the thread is in the middle of a Redis call when the lock is released, there is no coordination. The `finally` block in `_acquire_lock` should join the thread with a timeout before proceeding to the lock release check.

---

### 7. `reserve_item` Catches `RuntimeError` Too Broadly

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is also raised by Python internals in various failure conditions. If `self.db.update_stock` raises a `RuntimeError` (e.g., a database driver wrapping a connection error), it will be silently swallowed and returned as "Service temporarily unavailable" with no stock rollback and no re-raise. Use a specific exception type for lock failure, or restructure the try/except to only wrap the lock acquisition call.

---

### 8. No Redis Connection Error Handling

There is no handling of `redis.exceptions.RedisError` or `redis.exceptions.ConnectionError`. A Redis outage during a flash sale will propagate an unhandled exception through the lock context manager, bypassing the `ReservationResult` return contract entirely and likely surfacing a 500 to the caller with no graceful degradation path documented.

## Test Coverage Gaps

### 9. Tests Mock Away the Exact Code Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and failure-prone component in this PR. Mocking it entirely means none of the critical heartbeat race conditions described above are exercised by the test suite. The checklist claiming tests pass is accurate but misleading — the tests do not cover the code paths that matter most.

### 10. No Test for the Non-Atomic Lock Release Bug

`test_lock_not_released_if_expired` tests the happy path of the token mismatch check, but does not test the race condition where the token matches at `GET` time but the lock expires before `DELETE`. This is untestable with the current implementation because it requires atomicity, which reinforces why the Lua script fix is necessary.

### 11. No Test for Partial Database Failure

There is no test covering the scenario where `update_stock` succeeds and `create_order` raises. This is the data integrity scenario that would cause real financial harm in production.

## Minor Issues

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at T+10, leaving a 20-second window at the start where no extension has occurred. If the DB operation takes longer than 30 seconds before the first heartbeat, the lock expires. The heartbeat should fire at `TTL / 3` intervals, not `TTL / 3` offset from zero.
- The lock key `f"inventory:lock:{item_id}"` has no namespace isolation. If `item_id` is user-controlled input (which it appears to be), a malformed ID like `inventory:lock:foo:bar` could collide with other key patterns. Sanitize or hash the key component.
- `logger = logging.getLogger(__name__)` at module level is correct, but there is no log entry when `reserve_item` is entered, making distributed tracing across services difficult. A correlation ID or request ID should be threaded through.
