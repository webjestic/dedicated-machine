Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that make it unsafe for production use in a high-concurrency flash sale environment. The heartbeat mechanism introduces a race condition that can corrupt inventory state, the service is not safe for concurrent use across multiple callers within the same process, and the database operations lack transactional atomicity guarantees. These issues must be resolved before this code is deployed anywhere near a production workload.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the entire PR. The heartbeat state is stored on the instance, not scoped to the lock context:

```python
# These are instance variables — shared across ALL callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern — one service instance, many request threads), the following sequence is possible:

- Thread A acquires lock for `item_001`, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires lock for `item_002`, calls `_start_heartbeat`, sets `_heartbeat_active = True` (no-op, already True), but overwrites `_heartbeat_thread`
- Thread A finishes, calls `_stop_heartbeat`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now dead. Thread B's lock expires mid-operation.
- A third process acquires the lock for `item_002` while Thread B is still writing to the database.
- **Inventory is now double-decremented. Stock goes negative. Orders are created for items that don't exist.**

This is not a theoretical edge case. Under flash sale load, this is the expected execution pattern. The heartbeat state must be local to each lock acquisition context, not stored on `self`.

**Fix:** Encapsulate heartbeat state in a local object or pass it through closures scoped to the `_acquire_lock` context manager.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread, not self._heartbeat_active
```

### 2. Non-Atomic Lock Release — TOCTOU Race

The lock release in the `finally` block has a check-then-act race condition:

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # Step 1: read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # Step 2: delete
```

Between Step 1 and Step 2, another process can acquire the lock. You then delete a lock you no longer own. This is a well-known failure mode and the canonical fix has existed for years.

**Fix:** Use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    script(keys=[lock_key], args=[token])
```

This is not optional. The current implementation can delete another process's lock under load.

### 3. No Input Validation on Public Interface

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with zero validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is never validated. An empty string, a string containing Redis key injection characters (`:`, `*`, `?`, `[`), or a multi-kilobyte string all pass through directly into `f"inventory:lock:{item_id}"`.
- `quantity` is never checked for being positive. `quantity=0` silently succeeds and creates an order for zero units. `quantity=-5` would increase stock.

**Fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

Validate at the entry point of `reserve_item`, not downstream.

### 4. Database Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping them. If `create_order` raises an exception after `update_stock` succeeds, stock has been decremented but no order exists. The item is gone from inventory with no corresponding order record. This is a data integrity failure that will be invisible until someone audits the books.

The Redis lock provides mutual exclusion, but it does not provide atomicity for the database operations inside it. These are different guarantees.

**Fix:** These calls must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract must be documented to require transaction support.

### 5. Heartbeat Interval Is Dangerously Close to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T=10 and T=20. At T=20, it extends the TTL back to 30 seconds from that moment (T=50). This is technically functional but the margin is thin. More critically, the heartbeat thread calls `time.sleep(self.HEARTBEAT_INTERVAL)` before the first extension. If the database operation takes 11 seconds, the lock has 19 seconds remaining when the heartbeat fires — fine. But if the operation takes 29 seconds, the lock has 1 second remaining when the heartbeat fires, and a GIL pause or thread scheduling delay on a loaded system could mean the lock expires before the extension executes.

The standard Redlock guidance is to renew at `TTL / 3`. With a 30s TTL that means renewing every 10s, which matches, but the first renewal should happen before `HEARTBEAT_INTERVAL` elapses if the lock was just acquired. Consider renewing immediately on acquisition or reducing the interval to `TTL / 5`.

## Significant Issues

### 6. Bare `RuntimeError` for Lock Failure

```python
raise RuntimeError(
    f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
)
```

Using a bare `RuntimeError` means callers cannot distinguish a lock contention failure from a Redis connection failure from a programming error. Define a domain-specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

This also prevents the catch block in `reserve_item` from accidentally swallowing unrelated `RuntimeError` exceptions raised by `db.get_stock` or `db.update_stock`.

---

### 7. Redis Connection Failures Are Silently Swallowed

The `except RuntimeError` block in `reserve_item` catches lock acquisition failures, but `redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not caught anywhere. If Redis is unavailable, the exception propagates uncaught to the caller with no structured error response. Given that this is a flash sale service, Redis availability is a known failure mode that must be handled explicitly.

---

### 8. Retry Backoff Does Not Include Jitter

```python
backoff = (2 ** attempt) * 0.1
# attempt=0 → 0.1s, attempt=1 → 0.2s, attempt=2 → 0.4s
```

Under flash sale load, hundreds of threads will hit lock contention simultaneously. Without jitter, they all wake up at the same time and create a thundering herd against Redis. Add randomized jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 * (0.5 + random.random())
```

## Test Coverage Issues

### 9. Tests Do Not Cover the Concurrency Bug

The test suite mocks out the heartbeat entirely:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the most critical failure mode — concurrent calls corrupting `_heartbeat_active` — is completely untested. There should be at minimum one test that calls `reserve_item` from two threads simultaneously and asserts that both operations complete correctly with consistent stock levels.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts that `delete` is not called when the token doesn't match. But because `_start_heartbeat` is mocked, `redis.get` is also called during the lock acquisition check in the `finally` block — the test is actually verifying that the mocked `get` returns `b"different-token"` and the code skips deletion. This is testing the mock behavior, not the actual lock ownership logic. With the Lua script fix in place, this test would need to be rewritten entirely to verify the script is called with the correct arguments.

### 11. No Test for `quantity <= 0`

Given the missing input validation identified in Issue 3, there is no test asserting that `reserve_item("item_001", quantity=0)` or `reserve_item("item_001", quantity=-1)` returns a failure result. These would currently succeed and corrupt inventory.
