Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The most severe issues involve non-atomic lock release, instance-level heartbeat state that breaks under concurrent calls, and missing database transaction atomicity — any of which could cause inventory corruption or overselling under real load. The test suite also has structural gaps that mask these failure modes rather than expose them.

---

## Critical Issues

### 1. Non-Atomic Lock Release (Race Condition — CVE-Class Bug)

```python
# UNSAFE — two separate round-trips to Redis
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and `DELETE`, another process can acquire the lock after your lock expires. You then delete a lock you no longer own. This is a textbook TOCTOU race condition.

**Required fix:** Use the canonical Lua script for atomic check-and-delete:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

The same race exists in the heartbeat's `GET` + `expire` sequence and must be fixed identically.

### 2. Instance-Level Heartbeat State Breaks Concurrency

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the expected pattern in any web framework — the following happens:

- Call A sets `_heartbeat_active = True`, starts thread A
- Call B sets `_heartbeat_active = True`, starts thread B
- Call A finishes, sets `_heartbeat_active = False`
- Thread B is now silently killed while Call B is still holding its lock

This will cause lock expiry mid-operation under concurrent load, which is precisely the failure mode the heartbeat was designed to prevent. The heartbeat state must be scoped per-lock-acquisition, not per-instance.

**Required fix:** Encapsulate heartbeat state in a dedicated object or pass state through a `threading.Event`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

### 3. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists. The item is silently lost. This is a data integrity failure, not just an edge case.

**Required fix:** Wrap in a transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract must enforce this. If the injected `db` object does not support transactions, that is an interface design deficiency that must be addressed before this merges.

---

### 4. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting a flag does not wait for the thread to finish. The thread may still be mid-execution when the lock is released. Specifically:

1. `_stop_heartbeat()` sets flag to `False`
2. Lock is released via `DELETE`
3. Heartbeat thread wakes, checks flag... but the `GET` + `expire` already ran before the flag was checked
4. Thread extends TTL on a lock that has already been released or acquired by another process

**Required fix:** Call `self._heartbeat_thread.join(timeout=...)` after setting the stop flag, before releasing the lock.

## Significant Issues

### 5. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (negative reservation silently increases stock)
- `item_id` being an empty string (creates a lock key `inventory:lock:`)
- `item_id` containing characters that could be meaningful in your key namespace

**Required fix:**

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
```

Consider a Pydantic input model if the checklist claim is to be accurate.

### 6. Bare `RuntimeError` Catch Masks Real Failures

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` raised inside the `with` block — including from `db.update_stock` or `db.create_order` — will be silently swallowed and returned as "service unavailable." A partially-committed database state (stock decremented, order not created) will be invisible to the caller and to monitoring.

**Required fix:** Only catch the specific lock acquisition failure. Use a dedicated exception class:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) instead of RuntimeError in _acquire_lock
```

Then catch only `LockAcquisitionError` at the call site.

---

### 7. Heartbeat Interval Relative to TTL Is Dangerously Tight

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10s intervals and extends by 30s. This means if the heartbeat fires at T=10 and the next fire is T=20, the lock expires at T=40. That is fine in isolation. However, `time.sleep` is not precise under GIL contention or system load. Under a loaded Python process, the actual sleep may be significantly longer than 10 seconds, leaving a window where the lock expires before renewal.

**Recommendation:** Reduce `HEARTBEAT_INTERVAL` to `LOCK_TTL / 3` (10s for a 30s TTL is at the boundary), or better, renew for `LOCK_TTL` seconds from the point of renewal, and fire the heartbeat at `LOCK_TTL * 0.4` to give two renewal opportunities before expiry.

## Test Suite Issues

### 8. Tests Mock Away the Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This is acceptable for unit isolation, but the test named `test_lock_not_released_if_expired` is specifically testing lock release behavior while the lock release path contains the TOCTOU race described in Issue 1. The test passes today only because it is testing the broken non-atomic implementation. Once the Lua script fix is applied, this test must be rewritten.

### 9. No Test for Partial DB Failure

There is no test covering the scenario where `update_stock` succeeds and `create_order` raises. This is the exact failure mode from Issue 3. A test like the following must be added:

```python
def test_partial_db_failure_does_not_corrupt_stock(service, db):
    db.update_stock.return_value = True
    db.create_order.side_effect = Exception("DB write failed")

    with pytest.raises(Exception):
        service.reserve_item("item_001", quantity=1)

    # Stock must not be permanently decremented if order was never created
    # This requires transaction support — test will fail until Issue 3 is fixed
```

### 10. No Concurrency Tests

For a service explicitly designed for high-concurrency flash sales, there are zero tests exercising concurrent access. At minimum, a `threading`-based test should verify that two concurrent calls to `reserve_item` with quantity equal to total stock result in exactly one success and one failure.

## Minor Issues

- `logger.warning` on lock contention during retry is noisy; `logger.debug` is more appropriate for expected retry behavior
- The `db` parameter has no type annotation or interface definition; dependency inversion should be formalized with a `Protocol` or ABC
- `daemon=True` on the heartbeat thread means it will be killed abruptly on process shutdown, potentially mid-renewal; this is acceptable but should be documented
- `token.encode()` should be used when comparing Redis bytes responses rather than `.decode()` on the Redis value, to avoid assuming UTF-8 encoding consistency
