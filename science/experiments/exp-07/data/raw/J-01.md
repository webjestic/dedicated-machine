Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields, causing one heartbeat to stomp another or prematurely stop a live lock. Beyond that, there are several architectural and correctness issues that need to be addressed before this is safe to merge into a production flash-sale path.

---

## Detailed Findings

### 🔴 Critical — Thread Safety on Heartbeat State

```python
# Current — instance-level, shared across all concurrent callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests call `reserve_item` simultaneously on the same service instance (which is the normal case under a flash sale), they share `_heartbeat_active`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat thread mid-operation, potentially allowing the first lock to expire while the DB write is still in flight.

**Fix:** Scope heartbeat state to the lock acquisition itself, not the instance. Pass a `threading.Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        stop_event.set()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note also that `stop_event.wait(timeout=...)` is preferable to `time.sleep` — it wakes up immediately when signaled rather than sleeping through the full interval on shutdown.

### 🔴 Critical — Missing Input Validation

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `item_id: str` and `quantity: int` with zero validation. There is no Pydantic model gating the public interface.

- `item_id` can be empty string, whitespace, or a path-traversal-style value — all of which will produce a valid Redis key like `inventory:lock:` or `inventory:lock:../../etc`.
- `quantity` can be zero or negative. A reservation of `quantity=0` will always succeed and create a phantom order record. A negative quantity would *increase* stock.

```python
# quantity <= 0 passes the current check
if current_stock < quantity:   # 10 < -5 → False → proceeds
    ...
new_stock = current_stock - quantity  # 10 - (-5) = 15 — stock increases
```

**Fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        return ReservationResult(success=False, message=f"Invalid input: {e}")
    ...
```

### 🔴 Critical — Non-Atomic DB Operations Outside Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost with no compensating action and no alert. The Redis lock does not protect against DB-level partial failure.

**Fix:** These must execute inside a single DB transaction, or a saga/outbox pattern must be used with explicit rollback logic.

### 🟡 Major — Lock Release Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The `GET` + conditional `DELETE` is not atomic. Between the `GET` and the `DELETE`, the lock can expire and be acquired by another process. You then delete their lock. This is a well-known Redis distributed lock pitfall.

**Fix:** Use a Lua script for atomic check-and-delete, which is the standard approach:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The same race exists in the heartbeat's `GET` + `expire` sequence and should be made atomic as well.

### 🟡 Major — Test Coverage Gaps

The tests mock away the heartbeat entirely (`svc._start_heartbeat = MagicMock()`), which means the threading bug described above is completely invisible to the test suite. There are no tests for:

- Negative or zero `quantity`
- Empty or malformed `item_id`
- `db.update_stock` raising an exception (partial failure path)
- `db.create_order` raising an exception after stock is already decremented
- The Lua release script behavior (once added)

The `test_lock_not_released_if_expired` test is also misleading — it patches `redis.get` to return `b"different-token"` globally, but the token generated inside `_acquire_lock` is a fresh UUID, so this test is only accidentally correct. It should explicitly inject a known token to make the assertion meaningful.

---

### 🟡 Major — Heartbeat Interval vs. TTL Ratio

`LOCK_TTL = 30s`, `HEARTBEAT_INTERVAL = 10s`. The heartbeat fires at 10s, but the lock expires at 30s. This means the heartbeat has three windows to fire before expiry, which is fine in isolation. However, the heartbeat checks ownership with a non-atomic `GET` before calling `expire`. Under network partition or Redis failover, the heartbeat can silently fail to extend the lock with no error surfaced to the caller. The operation will then proceed past the lock boundary with no lock held.

At minimum, log a warning (or raise) if `expire` returns 0 (key not found).

---

### 🟢 Minor — Daemon Thread Leak on Process Shutdown

Daemon threads are killed abruptly on interpreter exit, which is acceptable, but if the service is running inside a framework that does graceful shutdown (e.g., gunicorn with `--graceful-timeout`), a daemon thread holding a Redis connection open during shutdown can produce confusing errors. Consider documenting this or providing a `close()` method.

---

### 🟢 Minor — `quantity` Not Validated as Integer at Boundary

If this service is ever called from a web layer that deserializes JSON, `quantity` could arrive as a float (e.g., `1.0` or `1.5`). Python will not reject `1.5 < 10` — it will proceed and set `new_stock = 8.5`. The Pydantic fix above handles this, but it needs to be enforced at the entry point.
