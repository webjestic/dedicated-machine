Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for a high-concurrency flash sale environment. The most severe issues involve instance-level heartbeat state creating race conditions under concurrent use, a non-atomic lock release sequence that opens a TOCTOU vulnerability, and missing input validation that the checklist falsely claims is present. These issues must be resolved before this code is anywhere near a production deployment.

### 🔴 Critical — Must Fix

#### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object. If two threads call `reserve_item` concurrently on the same service instance — which is the entire premise of a flash sale — they will race on these fields.

**Concrete failure scenario:**

- Thread A acquires lock for `item_001`, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires lock for `item_002`, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Thread A's `finally` block calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now silently dead. Its lock will expire mid-operation. Stock may be double-sold.

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass a threading `Event` or a local mutable container into the heartbeat closure.

```python
# Suggested pattern
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

#### 2. Non-Atomic Lock Release Is a TOCTOU Vulnerability

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # ← read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # ← conditional delete
```

There are two network round-trips here. Between the `GET` and the `DELETE`, another process can acquire the lock. You would then delete a lock you no longer own, corrupting the next holder's critical section entirely.

This is a well-known problem with Redis locking. The canonical solution is a Lua script that makes the check-and-delete atomic:

```lua
-- release.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same atomicity concern applies inside the heartbeat thread:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← not atomic
```

Use a Lua script here as well, or use `SET key token XX EX ttl` which is atomic by nature.

#### 3. The Checklist Claims Input Validation Exists — It Does Not

The checklist marks "Input validation via Pydantic" as complete. `ReservationResult` is a response model, not a validator. `reserve_item` accepts arbitrary `item_id` strings and `quantity` integers with zero validation.

**Missing guards:**

- `item_id` is passed directly into `f"inventory:lock:{item_id}"` — a malformed or adversarially crafted ID (e.g., `"item:lock:*"`, excessively long strings, or injection-style values) could cause unintended behavior depending on the Redis client and downstream DB layer.
- `quantity` has no lower-bound check. `quantity=0` would succeed and create a zero-unit order. `quantity=-5` would *increase* stock silently (`new_stock = 10 - (-5) = 15`).

```python
# Minimum required
if not isinstance(item_id, str) or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if not quantity or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

A proper fix would use a Pydantic request model for the method inputs, which is what the checklist implies but does not deliver.

#### 4. Database Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls. If `update_stock` succeeds and `create_order` throws, you have decremented stock with no corresponding order record. The lock does not protect you from this — the lock is a Redis construct and has no knowledge of your database transaction state.

These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is itself a blocking architectural concern that needs to be surfaced.

#### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has stopped. The thread could be mid-sleep when the flag is cleared, and the `finally` block proceeds to delete the lock while the heartbeat thread wakes up and attempts to extend it. You have a race between your cleanup path and your background thread.

```python
def _stop_heartbeat(self, stop_event: threading.Event, thread: threading.Thread) -> None:
    stop_event.set()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if thread.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly")
```

---

#### 6. Redis Errors Are Silently Swallowed

Only `RuntimeError` from lock acquisition failure is caught. Any `redis.RedisError` thrown during `get_stock`, `update_stock`, `create_order`, or within the heartbeat thread is completely unhandled. Under a Redis partition or timeout, this will surface as an uncontrolled exception propagating up through the call stack with no structured error response.

```python
except (RuntimeError, redis.RedisError) as e:
    logger.error(...)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

### 🔵 Observations — Low Severity

---

#### 7. Test Coverage Has a Structural Gap

The test suite mocks out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the TOCTOU race in the release path (Issue #2) and the concurrent state corruption (Issue #1) are completely invisible to the test suite. The lock release logic is only tested via `test_lock_not_released_if_expired`, but that test uses a hardcoded `b"different-token"` mock rather than exercising the actual race condition.

At minimum, an integration test against a real or embedded Redis instance (e.g., `fakeredis`) should be added to verify the atomic release behavior.

#### 8. `LOCK_TTL = 30` Is Undocumented as a Tunable

Thirty seconds is a very long TTL for an inventory lock in a flash sale. Under high load this will serialize all requests for a given item into a 30-second queue. This value should be configurable via constructor parameter or environment variable, not a class-level constant, with documented guidance on how to tune it for expected DB latency.
