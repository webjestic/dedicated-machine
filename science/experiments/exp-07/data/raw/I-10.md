Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for a high-concurrency flash sale environment. The most severe issues involve non-atomic lock release, instance-level heartbeat state that breaks under concurrent use, missing input validation on public entry points, and a non-atomic DB update pattern that can produce inconsistent state under failure. These issues must be resolved before this code is anywhere near production.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

#### 1. Non-Atomic Lock Release (Race Condition)

```python
# Current — BROKEN
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a textbook check-then-act race. Between the `GET` and the `DELETE`, another process can acquire the lock after yours expires. You will then delete a lock you no longer own. This is the exact scenario the ownership token is supposed to prevent, and it doesn't.

**Required fix:** Use a Lua script to make the check-and-delete atomic.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat:

```python
# Also broken — non-atomic check-then-extend
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Fix with a Lua script or use `SET key value XX EX ttl` with the token as value and a conditional.

#### 2. Instance-Level Heartbeat State — Completely Broken Under Concurrency

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level attributes**, not call-scoped. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the entire point of this service), the following happens:

- Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a different item lock, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Thread A finishes, calls `_stop_heartbeat`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now silently dead — its lock will expire mid-operation

This is a silent correctness failure. The heartbeat gives a false sense of safety.

**Required fix:** Heartbeat state must be scoped to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # Lua-based atomic extend here
            ...

    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release
```

#### 3. No Input Validation on the Public Method

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw, unvalidated arguments. Pydantic is only used on the *output* model.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing validates item_id or quantity before use
    lock_key = f"inventory:lock:{item_id}"
```

An empty string, a negative quantity, a 10,000-unit reservation, or a maliciously crafted `item_id` (e.g., containing `:` or newline characters that could manipulate Redis key structure) all pass through silently.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    ...
```

#### 4. Non-Atomic DB Operations Without Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary and no rollback. If `update_stock` succeeds and `create_order` throws, you have decremented inventory with no corresponding order record. Stock is silently lost. Under flash sale load, this is a real failure path.

**Required fix:** These must execute within a single database transaction. The DB abstraction layer needs to expose a transactional interface, or this method needs to be redesigned around one.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Significant — Should Fix Before Merge

---

#### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread sleeps for `HEARTBEAT_INTERVAL` (10s) before checking the flag. After `_stop_heartbeat` returns, the thread is still alive and may attempt one final `expire` call after the lock has already been released or acquired by another process. The `finally` block needs to join the thread with a timeout.

---

#### 6. Bare `RuntimeError` Used as Control Flow

```python
raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")
```

Using `RuntimeError` for an expected, recoverable condition (lock contention) is poor design and makes it impossible for callers to distinguish between a lock contention failure and an unexpected programming error. Define a domain-specific exception.

```python
class LockAcquisitionError(Exception):
    pass
```

---

#### 7. `retry_attempts=3` Is Hardcoded and Misconfigured

With 3 attempts and backoffs of 0.1s, 0.2s, 0.4s, the maximum wait before giving up is **0.7 seconds**. During a flash sale with high contention, this will produce a flood of failures. This parameter should be configurable at the service level and documented with reasoning. Also consider total wait budget rather than attempt count as the primary control.

### 🔵 Test Coverage Gaps

---

#### 8. Tests Mock Away the Exact Code Under Scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is one of the most complex and failure-prone parts of this implementation. Mocking it entirely means the tests provide zero coverage for the actual threading behavior, the token check, or the TTL extension. At minimum, integration-level tests against a real or embedded Redis instance (e.g., `fakeredis`) should be added for the lock lifecycle.

#### 9. The Lock Release Race Is Not Tested Correctly

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test validates the broken non-atomic implementation. Once the release is fixed with a Lua script, this test needs to be rewritten to verify the script is called with the correct arguments, not that `delete` is skipped.

#### 10. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the exact data-integrity failure path described in issue #4 above.
