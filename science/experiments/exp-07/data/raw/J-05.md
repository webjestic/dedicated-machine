Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields, causing one heartbeat to silently cancel another's lock renewal or orphan threads entirely. Beyond that, the DB operations lack transactional guarantees and the input validation claimed in the checklist is not actually present. The test suite also has a structural problem that masks the concurrency bug entirely by mocking the heartbeat out of existence.

---

### Detailed Findings

#### 🔴 Critical — Instance-Level Heartbeat State (Thread Safety)

```python
# Current — breaks under concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or long-lived object. Two simultaneous reservations on different items will share `_heartbeat_active`. The second caller's `_stop_heartbeat()` will set it to `False`, killing the first caller's renewal thread mid-operation, potentially letting that lock expire and allowing a competing process to acquire it — a silent inventory double-spend.

The fix is to make heartbeat state local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, then:
    thread.start()
    try:
        yield token
    finally:
        active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which is harder to reason about and slower to respond to shutdown.

#### 🔴 Critical — No Transactional Integrity on DB Operations

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost with no compensating action and no alert. For a flash sale context this is a direct revenue and audit risk.

```python
# Minimum acceptable pattern
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer doesn't support context-managed transactions, that needs to be resolved before this merges.

---

#### 🟠 High — Input Validation Checklist Item Is False

The checklist claims "Input validation via Pydantic" but `reserve_item` performs zero validation on its inputs:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing validates item_id is non-empty, non-whitespace,
    # or that quantity is a positive integer
    lock_key = f"inventory:lock:{item_id}"
```

An empty `item_id` produces the lock key `"inventory:lock:"`, which is a valid Redis key that would serialize all reservations for every item with an empty ID. A `quantity` of `0` or `-5` would pass the stock check and silently corrupt inventory. Add explicit guards:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1)
    quantity: int = Field(ge=1)

    @field_validator("item_id")
    @classmethod
    def item_id_not_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("item_id must not be blank")
        return v
```

#### 🟠 High — Heartbeat Has a Race Window at Startup

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
# ...
self._start_heartbeat(lock_key, token)  # thread starts here
try:
    yield token  # caller begins slow DB work here
```

There is a window between lock acquisition and the heartbeat thread actually scheduling its first iteration. Under a loaded system, if the DB operation begins immediately and takes longer than `LOCK_TTL` before the heartbeat fires, the lock can expire. The heartbeat interval of 10s against a TTL of 30s is reasonable, but the first renewal should fire sooner, or the lock should be re-extended immediately after the thread starts:

```python
# Immediately confirm ownership after thread launch
self.redis.expire(lock_key, self.LOCK_TTL)
```

#### 🟡 Medium — Tests Mock Away the Bug They Should Catch

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` at the instance level means the concurrency bug described above is completely invisible to the test suite. Unit tests for lock behavior are fine, but there must be at least one integration-level test that runs real threads:

```python
def test_concurrent_reservations_do_not_corrupt_stock():
    # Use a real Redis (testcontainers or fakeredis) and a real DB stub
    # Fire N threads simultaneously, assert total reserved == initial stock
```

`fakeredis` is the appropriate tool here and avoids any external dependency in CI.

---

#### 🟡 Medium — `RuntimeError` Is Too Broad an Exception Boundary

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` will also catch unexpected programming errors thrown inside the `with` block (e.g., a bug in `update_stock` that raises `RuntimeError`). Those should propagate, not be swallowed as "service unavailable." Use a dedicated exception type:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError specifically in reserve_item
```

---

#### 🟢 Minor — `thread.join()` Is Missing on Heartbeat Stop

`_stop_heartbeat` sets the flag but does not join the thread. The thread may still be mid-`expire` call when the lock is deleted, causing a redundant TTL extension on a key that either no longer exists or has been re-acquired by another process. Joining with a short timeout before the delete closes this window cleanly.

---

### What Is Done Well

The lock ownership verification via token comparison before release is correct and important — the `test_lock_not_released_if_expired` test validates real behavior there. The exponential backoff structure is sound. Structured logging with key/value pairs is the right pattern. Pydantic for the result type is appropriate.
