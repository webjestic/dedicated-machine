Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — atomic lock acquisition via `SET NX EX`, token-based ownership verification, and heartbeat renewal are all directionally correct. However, there are several architectural flaws, race conditions, and missing safeguards that would make this dangerous to run under real flash sale concurrency pressure. None of these are cosmetic — they represent genuine failure modes that could cause overselling, deadlocks, or silent data corruption in production.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Under Concurrency

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the **instance**, not scoped to the lock context. If two callers share the same `InventoryService` instance (which is the overwhelmingly common deployment pattern — singleton service injected into a request handler), concurrent calls to `reserve_item` will race on these fields.

**Failure scenario:**
- Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a *different* lock, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Thread A's `finally` block calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now silently dead — its lock will expire mid-operation
- Oversell occurs

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    # ... acquire lock ...

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 2. Non-Atomic Lock Release — TOCTOU Race

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a textbook **check-then-act** race. Between the `GET` and the `DELETE`, another process can:
1. Let the lock expire naturally
2. Acquire the lock with a new token
3. Your `DELETE` fires and releases *their* lock

This is precisely the attack vector that Lua scripts on Redis were designed to close. This must be atomic:

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's `GET` + `EXPIRE` sequence — that should also be a Lua script or use `GETEX`.

### 3. Database Operations Are Not Atomic — No Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `update_stock` succeeds and `create_order` raises (DB timeout, constraint violation, network partition), you have decremented stock with no corresponding order record. The item is effectively destroyed — neither reserved nor available.

The distributed lock protects against concurrent *reads*, but it does not substitute for database-level atomicity. This needs an explicit transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose transactional semantics. If it doesn't, that's a blocking architectural gap.

### 4. No Input Validation on Public Interface

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with zero validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is injected directly into a Redis key: `f"inventory:lock:{item_id}"`. A caller passing `item_id = "foo:bar:../../admin"` or an arbitrarily long string can manipulate key structure or cause memory pressure.
- `quantity` has no lower bound check. `quantity=0` passes the stock check and creates a zero-unit order. `quantity=-5` would *increase* stock silently.

Minimum required:

```python
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    raise ValueError("Invalid item_id")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

Or define a proper request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1)
```

## Significant Issues

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread sleeps for `HEARTBEAT_INTERVAL` (10s) and only checks the flag after waking. In the window between `_stop_heartbeat()` and the thread observing the flag, the thread can still fire a `redis.expire()` call — potentially extending a lock that the `finally` block is simultaneously trying to release. Use `threading.Event` with `wait(timeout=...)` and join the thread before releasing the lock.

---

### 6. Bare `RuntimeError` for Lock Failure Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

`RuntimeError` is a base Python exception that many libraries raise for unrelated reasons. A Redis connection failure, a DB driver error, or a bug in your own code will all be silently swallowed here and returned to the caller as "temporarily unavailable." Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError specifically in reserve_item
```

### 7. Test Suite Has a Structural Flaw That Masks the Core Bug

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking out `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the heartbeat lifecycle at all. The concurrency bug in issue #1 is completely invisible to this test suite. You need at minimum:

- A test that instantiates a real `InventoryService` and verifies the heartbeat thread starts and stops correctly
- A concurrency integration test using `threading.Barrier` to force two threads into `reserve_item` simultaneously and assert exactly one succeeds
- A test that verifies the Lua release script is called, not `GET` + `DELETE`

The `test_lock_not_released_if_expired` test is also subtly broken — it mocks `redis.get` to return `b"different-token"` globally, which means the lock *acquisition* path (`SET NX`) still returns `True` but the token comparison in release uses a hardcoded fixture token that will never match the dynamically generated `uuid4()` token from inside `_acquire_lock`. The test is passing for the wrong reasons.

## Minor Issues

### 8. `LOCK_TTL = 30` With `HEARTBEAT_INTERVAL = 10` Is Marginal

With a 10-second heartbeat and a 30-second TTL, you have a 20-second window before the first renewal. If the heartbeat thread is delayed by GIL contention or system scheduling, you are cutting it close. A common convention is `HEARTBEAT_INTERVAL = TTL / 3`. At minimum, document the reasoning and add a constructor assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, "Heartbeat must fire well before TTL expiry"
```

### 9. No Observability on Heartbeat Failures

If `redis.expire` fails inside the heartbeat (Redis unreachable, connection dropped), the failure is silently swallowed. The lock will expire, another process will acquire it, and you will have two processes inside the critical section simultaneously. The heartbeat should signal the main thread on failure so the operation can be aborted:

```python
# Use threading.Event to signal failure back to the main thread
```
