Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several critical flaws that would cause data corruption and race conditions under real concurrent load. The instance-level heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) makes this service non-reentrant and completely unsafe for any concurrent use, including simple async frameworks or thread pools. Additionally, the DB operations lack transactional atomicity, and the input validation claimed in the checklist is not actually implemented.

---

## Detailed Review

### 🔴 Critical — Must Fix

**1. Non-reentrant instance state (concurrency data corruption)**

```python
# Current — instance-level fields shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests hit the same `InventoryService` instance simultaneously (which is the entire point of a flash sale scenario), the second call's `_stop_heartbeat()` will kill the first call's heartbeat thread. The first lock will then expire mid-operation, another process can steal it, and you now have two concurrent writers past the lock boundary.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

**2. No input validation despite checklist claiming it**

The checklist marks "Input validation via Pydantic" but there is zero validation on `item_id` or `quantity` in `reserve_item`. Pydantic is only used for the *output* model. A caller can pass `quantity=-5` and silently *increase* stock, or pass an empty string as `item_id`.

```python
# Add this — or use FastAPI's request model upstream
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        raise ValueError("item_id must be a non-empty string")
    if quantity < 1:
        raise ValueError(f"quantity must be >= 1, got {quantity}")
```

**3. DB operations are not atomic with respect to each other**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect you from partial DB failure. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The DB abstraction needs to expose a transaction context manager. This is a design requirement, not optional.

---

**4. Heartbeat `time.sleep()` is not interruptible**

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 full seconds
```

When `_stop_heartbeat()` sets the flag to `False`, the thread still sleeps for up to 10 seconds before checking. During shutdown or rapid sequential calls this leaks threads. Use `threading.Event` with `wait(timeout=...)` as shown above — it wakes immediately when cleared.

### 🟡 Significant — Should Fix

**5. Lock release has a TOCTOU race**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can steal the key between GET and DEL
```

This is a classic check-then-act race. The standard solution is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

---

**6. Bare `RuntimeError` is too broad**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

This silently swallows any `RuntimeError` from the DB layer, masking real bugs as "try again" messages. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError in reserve_item
```

### 🔵 Test Coverage Gaps

**7. The concurrency bug is completely untested**

There is no test exercising two concurrent calls on the same service instance, which is precisely the failure mode introduced by the instance-level heartbeat state. A test like this should be required:

```python
def test_concurrent_reservations_do_not_corrupt_state():
    # Two threads calling reserve_item simultaneously
    # Verify neither heartbeat stomps the other
```

**8. `test_lock_not_released_if_expired` tests the wrong token**

The test patches `redis_client.get` to return `b"different-token"`, but the lock context internally generates a `uuid4` token that will never equal `b"different-token"`. The test passes vacuously — `delete` is never called regardless of the fix, because the token never matched in the first place. The test needs to be rewritten to actually inject the token or use a spy on the Lua release path.
