Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — ownership-verified lock release, heartbeat extension, and exponential backoff are all directionally correct. However, there are several critical architectural flaws and at least one race condition that would make this dangerous to deploy under real flash sale concurrency, and the test suite has a structural defect that masks the most important failure mode entirely. These issues must be resolved before this can be approved.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe issue in the PR.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as **instance variables**, not scoped to the lock context. If a single `InventoryService` instance handles two concurrent calls — which is the entire point of a flash sale scenario — the second call's `_stop_heartbeat()` will set `_heartbeat_active = False` and silently kill the **first call's** heartbeat thread. The first lock then expires mid-operation, another process steals it, and you have a double-reservation.

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. A clean fix is to pass a `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquisition logic ...

    def _beat(stop: threading.Event) -> None:
        while not stop.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, args=(stop_event,), daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

Note also that `_stop_heartbeat` currently does not `join()` the thread. There is a window between setting the flag and the thread actually stopping where the thread could extend a lock that the `finally` block is simultaneously trying to delete. The `join()` call above closes that window.

### 2. Non-Atomic Lock Release — TOCTOU Race Condition

```python
# In _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic check-then-act race. Between `GET` and `DELETE`, another process can acquire the lock. You would then delete a lock you do not own, causing a double-reservation or corrupted state. This is a well-documented problem with Redis locking and the canonical fix is a Lua script for atomic compare-and-delete:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

This must be registered as a cached script (`redis.register_script`) in production to avoid re-parsing overhead on every release.

### 3. Heartbeat Check Has the Same TOCTOU Problem

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` + `EXPIRE` sequence is also non-atomic. A lock that expires between the `GET` and `EXPIRE` can be re-acquired by another node, and your heartbeat will then extend a lock owned by someone else. Use a Lua script here as well, or use `SET key token XX EX ttl` with a token re-check baked into the conditional.

### 4. No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation whatsoever. Pydantic is only used on the **output** model `ReservationResult`. There is no guard against:

- `quantity <= 0` (reserve zero or negative units — this would silently *increase* stock)
- `item_id` containing Redis key injection characters (e.g., a crafted `item_id` like `"foo:lock:admin"` could collide with other lock namespaces)
- Excessively long `item_id` strings

At minimum:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

A `quantity` of zero or negative is particularly dangerous because `current_stock < quantity` would be `False` for any non-negative stock, the reservation would succeed, and `new_stock = current_stock - (-5)` would **add** stock.

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent access but does not protect against partial writes. These must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should be documented or enforced to require transaction support.

## Moderate Issues

### 6. Silent Exception Swallowing on DB Errors

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` is caught. Any exception from `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` — including `redis.RedisError` during lock operations — will propagate uncaught to the caller with a raw stack trace that may leak internal implementation details. Either broaden the exception handling with appropriate logging, or let it propagate intentionally and document that contract. Silently swallowing DB exceptions while the lock is still held is particularly dangerous because the lock will be released in `finally` but the caller has no way to distinguish "lock contention" from "DB write failed halfway."

### 7. `time.sleep()` in Retry Loop Blocks the Calling Thread

```python
time.sleep(backoff)
```

With `retry_attempts=3`, the worst-case blocking time is `0.1 + 0.2 + 0.4 = 0.7s` per call. Under flash sale load this will exhaust thread pools quickly. Consider whether this service should be async-native or whether the retry should be delegated to the caller via a raised exception with retry metadata.

## Test Suite Issues

### 8. Tests Mock Away the Exact Code Under Scrutiny

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat and stop methods are mocked out in every test except `test_lock_not_released_if_expired`. This means the instance-level state collision bug described in issue #1 is **completely invisible** to the test suite. The tests would pass even if the heartbeat thread was actively corrupting state. There must be at least one integration-level test that exercises real thread behavior, even if it uses a fake Redis (e.g., `fakeredis`).

### 9. `test_lock_not_released_if_expired` Tests the Wrong Token

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test uses the `service` fixture which mocks `_start_heartbeat` and `_stop_heartbeat`. The `redis_client.get` is also used by the heartbeat during a real run, but since the heartbeat is mocked, this test only validates the release path in isolation. More importantly, it does not test that the TOCTOU race described in issue #2 is actually safe — it just tests the happy-path token mismatch. Once the Lua script fix is applied, this test needs to be rewritten to verify the atomic script is called with the correct arguments.

## Minor Issues

- **No TTL on the lock key in the heartbeat path.** If `LOCK_TTL` is ever changed to a value less than `HEARTBEAT_INTERVAL`, the lock will expire before the first heartbeat fires. Assert `LOCK_TTL > HEARTBEAT_INTERVAL * 2` at construction time.
- **`daemon=True` on the heartbeat thread** means it will be killed abruptly on interpreter shutdown. This is probably acceptable here, but it should be a conscious documented decision given that an abruptly killed heartbeat during a reservation could leave a lock held until TTL expiry.
