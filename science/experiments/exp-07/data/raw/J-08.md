Decision: Request Changes

Summary: The implementation has solid bones — the lock ownership check, heartbeat concept, and structured logging are all reasonable starting points. However, there are several correctness, safety, and architectural issues that must be addressed before this can be merged into a production system handling flash sale inventory, where race conditions and data inconsistency have direct financial impact.

---

## Critical Issues

### 1. Heartbeat Race Condition — Lock Can Be Released While Heartbeat Is Still Running

`_stop_heartbeat()` sets `self._heartbeat_active = False` but does **not join the thread**. The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10s) between iterations. After `_stop_heartbeat()` returns, the `finally` block immediately checks and deletes the lock — but the heartbeat thread may wake up moments later and call `redis.expire()` on a key that now belongs to a different owner.

```python
# current finally block — race window exists
finally:
    self._stop_heartbeat()          # sets flag, does NOT wait for thread
    current = self.redis.get(lock_key)
    if current and current.decode() == token:
        self.redis.delete(lock_key) # lock deleted
    # heartbeat thread wakes up 0–10s later, may extend a recycled key
```

**Fix:** Join the thread with a timeout before releasing the lock.

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread is not None:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._heartbeat_thread = None
```

---

### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes. If `InventoryService` is used as a shared singleton (the typical pattern in a web service), two concurrent calls to `reserve_item()` will clobber each other's heartbeat state. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, potentially letting the first lock expire mid-operation.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass state via closure or encapsulate in a small `_LockHandle` dataclass.

```python
@dataclass
class _LockHandle:
    key: str
    token: str
    active: bool = True
    thread: Optional[threading.Thread] = None
```

---

### 3. Lock Release Is Not Atomic — TOCTOU Vulnerability

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock. This is a textbook time-of-check/time-of-use (TOCTOU) race. In a flash sale context this could cause a legitimate lock to be deleted by a previous owner whose lock had just expired and been re-acquired.

**Fix:** Use a Lua script for atomic check-and-delete. This is the standard Redis pattern.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 4. No Input Validation on Public Method

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input side. A negative `quantity`, an empty `item_id`, or a zero `quantity` will silently pass through to the database layer.

```python
# These all reach db.update_stock unchecked:
service.reserve_item("", quantity=-5)
service.reserve_item("item_001", quantity=0)
```

**Fix:** Add an input model and validate at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Database Operations Are Not Atomic — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order record exists. The distributed lock does not protect against this — the lock only prevents concurrent access, not partial writes within a single operation.

**Fix:** These two operations must execute within a single database transaction. The `db` abstraction should expose a transactional interface, or the service should be restructured to use one.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 6. Bare `RuntimeError` Is Too Broad in the Exception Handler

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a broad base class. If `db.update_stock` or `db.create_order` raises a `RuntimeError` (e.g., a connection pool error), it will be silently swallowed and returned as "service unavailable" with no re-raise, no alerting path, and a misleading message. Introduce a dedicated `LockAcquisitionError` and only catch that.

---

### 7. Test Token Mismatch Makes Lock Release Test Invalid

```python
# fixture
mock.get.return_value = b"test-token"

# service generates its own uuid token internally
token = str(uuid.uuid4())  # never equals "test-token"
```

The `redis.get` mock always returns `b"test-token"`, but the service generates a fresh UUID. This means the lock release check (`current.decode() == token`) will **always fail** in tests, so `redis.delete` is never called in the success path. The tests pass, but they are not testing the actual release logic. `test_lock_not_released_if_expired` is testing the wrong thing for the wrong reason.

**Fix:** Capture the token used during `SET` and return it from the `GET` mock, or use `patch('uuid.uuid4', return_value='test-token')`.

---

### 8. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)
```

In an async or thread-pool-based web server, sleeping the calling thread during lock contention under flash sale load will exhaust the thread pool. This should be documented as a known limitation at minimum, and ideally replaced with async-compatible locking (`asyncio.sleep` in an async context, or a queue-based approach).

---

## Minor Issues

- `LOCK_TTL = 30` seconds is described as "generous for slow DB operations." Thirty seconds is extremely long for what should be a sub-second database read-modify-write. This should be tuned down significantly (2–5s) with a comment explaining the rationale, or the slow DB operations should be investigated.
- No metrics or observability hooks (lock wait time, reservation success rate). For a flash sale service this is operationally critical.
- The `db` parameter has no interface definition (ABC or Protocol). This makes the contract implicit and the mock in tests potentially diverge from real implementations silently.
