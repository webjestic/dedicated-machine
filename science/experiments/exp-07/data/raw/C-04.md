Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale environment. The heartbeat mechanism has a race condition that can cause split-brain lock ownership, the service is not safe for concurrent use across multiple reservations, and the database operations lack atomicity guarantees. These issues must be resolved before this code is merged.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Fatal Concurrency Bug

This is the most severe flaw in the submission. The heartbeat state is stored on the instance, not scoped to the lock context:

```python
# These are instance-level fields, not call-scoped
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of a flash sale service — the following happens:

- Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a different item's lock, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Thread A's `finally` block calls `_stop_heartbeat()`, setting `_heartbeat_active = False`
- **Thread B's heartbeat is now silently dead**, and Thread B's lock will expire while the operation is still running

This is a silent data corruption path. The lock expires, another process acquires it, and two processes now believe they hold the lock simultaneously. For a government contracting system handling inventory, this is unacceptable.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # Pass heartbeat_active into the thread closure, not self
```

### 2. Heartbeat Race Condition — Check-Then-Act on Lock Ownership

The heartbeat thread performs a non-atomic check-then-extend:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You then extend a lock you no longer own. This is the exact split-brain scenario distributed locks are meant to prevent.

**Fix:** Use a Lua script to make ownership verification and extension atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same issue exists in the lock release path — the `GET` + `DELETE` is also non-atomic and must use a Lua script.

---

### 3. Lock Release Is Also Non-Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # Another process can acquire between GET and DELETE
```

This is a well-documented anti-pattern. The canonical fix is a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is not a theoretical concern. Under the load profile of a flash sale, this race window will be hit.

### 4. Database Operations Are Not Atomic — Inventory Can Go Negative

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `update_stock` succeeds and `create_order` raises an exception, you have decremented inventory with no corresponding order record. The item is effectively lost. The reverse failure (order created, stock not decremented) is also possible depending on execution order.

**Fix:** These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The current code also has no exception handling around the DB calls inside the lock. An unhandled DB exception will propagate out of the `with self._acquire_lock(...)` block, bypassing the `ReservationResult` return path and surfacing a raw exception to the caller.

### 5. No Input Validation Despite Checklist Claiming Otherwise

The checklist marks input validation as complete, but `reserve_item` accepts arbitrary input:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `quantity=0` is accepted and will succeed, creating a zero-unit order
- `quantity=-5` is accepted and will **increase** stock (`new_stock = current_stock - (-5)`)
- `item_id` with path-traversal or injection characters is passed directly to `db.get_stock` and used to construct `lock_key` with no sanitization

The Pydantic model is only used for the *output* (`ReservationResult`). There is no Pydantic model or guard on the *input*. The checklist item is misleading.

**Fix:**

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

Or define a `ReservationRequest` Pydantic model and validate at the boundary.

## Significant Issues

### 6. HEARTBEAT_INTERVAL Is Too Close to LOCK_TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T+10, T+20, T+30. At T+30 the lock expires. The heartbeat at T+30 is a race. Under any scheduling jitter, GC pause, or thread starvation, the heartbeat fires *after* expiry. The ratio should be at minimum 3:1 with margin. A common production pattern is to renew at 1/3 of TTL, meaning `HEARTBEAT_INTERVAL` should be closer to 8–9 seconds with a `LOCK_TTL` of 30, but more importantly the heartbeat should renew to a *fresh* TTL from the point of renewal, not just re-assert the original. This is already the behavior of `EXPIRE`, but the timing margin is dangerously thin.

---

### 7. Heartbeat Thread Swallows Errors Silently

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

If Redis is unreachable during a heartbeat cycle, the exception is swallowed, the lock expires, and the operation continues believing it holds the lock. There is no mechanism to signal the main thread that lock ownership has been lost.

**Fix:** The heartbeat thread must communicate failure back to the main thread. Use a `threading.Event` or a shared flag that the main thread checks at critical points.

---

### 8. `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has stopped. The thread may be mid-sleep when the flag is set and will not check it for up to `HEARTBEAT_INTERVAL` seconds. In the worst case, the thread wakes up after the lock has been released and re-acquired by another process, and extends a lock it no longer owns — even with the Lua fix, this is a logic error that should be closed.

**Fix:**

```python
def _stop_heartbeat(self, thread: threading.Thread, event: threading.Event) -> None:
    event.set()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

Use `threading.Event` with `event.wait(timeout)` instead of `time.sleep` in the heartbeat loop so it wakes immediately on stop.

---

### 9. Lock Key Constructed From Unsanitized User Input

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` contains `:` characters or Redis glob patterns, this can collide with other key namespaces. For a service operating in a shared Redis instance (common in government infrastructure with cost constraints), this is a namespace pollution risk. At minimum, hash or encode the `item_id`:

```python
import hashlib
safe_id = hashlib.sha256(item_id.encode()).hexdigest()
lock_key = f"inventory:lock:{safe_id}"
```

## Test Coverage Issues

### 10. Tests Mock Away the Exact Code Under Scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and failure-prone component in this PR. Mocking it out entirely means the tests provide no coverage for the most critical path. There should be at minimum:

- A test that verifies the heartbeat actually extends the TTL
- A test that verifies the heartbeat stops after lock release
- A test that verifies the heartbeat does not extend a lock with a mismatched token
