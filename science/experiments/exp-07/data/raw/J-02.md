Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several critical flaws that make this unsafe for a high-concurrency flash sale environment. The instance-level heartbeat state creates a race condition that will cause silent lock mismanagement under concurrent calls, and the DB operations lack atomicity guarantees that the lock is meant to protect. The test suite also has a structural defect that causes it to silently pass without actually exercising the lock release path.

---

## Detailed Review

### 🔴 Critical — Instance-Level Heartbeat State (Concurrency Bug)

This is the most serious issue. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on `InventoryService`, which is almost certainly a singleton or long-lived object.

```python
# These are shared across ALL concurrent calls to reserve_item
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests call `reserve_item` concurrently:

1. Request A acquires lock, starts heartbeat → `_heartbeat_active = True`
2. Request B acquires a *different* item's lock, calls `_start_heartbeat` → overwrites `_heartbeat_thread`
3. Request A calls `_stop_heartbeat` → sets `_heartbeat_active = False`, killing **both** heartbeats
4. Request B's lock now expires silently while its DB operations are still running

This is a data integrity failure. The heartbeat state must be scoped per-lock, not per-instance. The fix is to pass the active flag into the thread closure or use a per-call `threading.Event`.

```python
# Correct pattern — per-call event
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller calls stop_event.set() to stop
```

### 🔴 Critical — Non-Atomic DB Operations Inside Lock Boundary

The lock protects the read-modify-write cycle, but the two DB writes are not atomic:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, the Redis lock expires, or the network drops between these two calls, you get decremented stock with no corresponding order record — inventory is silently lost. For a government contracting context this is an auditability and financial integrity failure.

These must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction should expose a transaction context manager, and the tests should verify rollback behavior.

---

### 🔴 Critical — Input Validation Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but there is no validation of `item_id` or `quantity` anywhere in the service layer. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar"` or `item_id = ""` or `quantity = -5` will produce undefined behavior or potentially manipulate the key namespace. The Pydantic model `ReservationResult` is an *output* DTO, not input validation. This checklist item is inaccurate.

Minimum required:

```python
if not item_id or not isinstance(item_id, str) or ":" in item_id:
    raise ValueError(f"Invalid item_id: {item_id!r}")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError(f"quantity must be a positive integer, got {quantity!r}")
```

Or introduce a proper input model validated at the call site.

### 🟡 High — Test Suite Has a Structural Defect

The `test_lock_not_released_if_expired` test mocks the heartbeat at the fixture level but then creates a fresh `InventoryService` instance without the mock:

```python
# In the fixture:
svc._start_heartbeat = MagicMock()  # applied to fixture instance

# In the test — a NEW instance, heartbeat NOT mocked:
def test_reserve_lock_contention(redis_client, db):
    redis_client.set.return_value = False
    svc = InventoryService(redis_client, db)         # fresh instance
    svc._start_heartbeat = MagicMock()               # re-applied manually — OK here
```

More critically, `test_lock_not_released_if_expired` uses the `service` fixture which mocks `_start_heartbeat`. This means the token generated inside `_acquire_lock` is a real `uuid4()`, but `redis_client.get` returns `b"different-token"`. The test passes, but it never actually exercises the lock expiry path in a realistic way because the heartbeat is fully mocked out. The test gives false confidence. It should use `patch` on `uuid.uuid4` to control the token and verify the full flow.

---

### 🟡 High — Heartbeat Sleep Pattern Is Imprecise

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)  # sleeps THEN checks
    if self._heartbeat_active:
        ...
```

With `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10`, the heartbeat extends the lock at T+10, T+20. But if the lock is acquired at T=0 and the heartbeat thread sleeps first, the first extension doesn't happen until T+10. Under a slow DB operation that starts at T=0, the lock could expire at T=30 before the second extension at T+20 if the first sleep is delayed by OS scheduling. Use `threading.Event.wait(timeout=...)` as shown above, which also makes the stop responsive rather than waiting up to 10 seconds for the thread to notice `_heartbeat_active = False`.

---

### 🟡 Medium — Lock Retry Backoff Blocks the Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In a web server context (WSGI/ASGI), blocking the thread for up to 700ms during lock contention will exhaust the thread pool under flash sale load — which is exactly the scenario this code is designed for. Consider making `reserve_item` async or documenting clearly that this service must not be called from an async context without running in a thread executor. At minimum, add a comment warning about this.

---

### 🟢 What Is Done Well

- Lock ownership verification via token before release is correct and important
- `nx=True, ex=self.LOCK_TTL` in a single atomic `SET` is the right Redis primitive
- Structured logging with key/value pairs is clean
- `ReservationResult` as a typed return value is good API design
- Daemon thread usage is appropriate so threads don't block process shutdown
