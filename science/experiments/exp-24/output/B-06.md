Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed locks are supposed to provide. A lock with an auto-renewing TTL is not a bounded lock; it can be held indefinitely by a hung or slow process, which defeats the purpose of the TTL entirely and reintroduces the split-brain / stale-write scenario the PR claims to prevent. There are also several concurrency, correctness, and test-quality issues that need to be addressed before this is mergeable.

---

## Detailed Findings

### 🔴 Critical — Heartbeat Renewal Undermines Lock Safety

This is the core issue. The PR summary explicitly acknowledges stop-the-world GC pauses and argues the heartbeat solves them. It does not. It makes the problem worse.

**The scenario the PR claims to handle:**

1. Process A acquires lock, heartbeat starts renewing it.
2. Process A hits a long GC pause (or slow DB call).
3. Heartbeat keeps renewing the lock on behalf of a paused process.
4. Process A resumes, believes it still owns the lock, and writes stale data.

The heartbeat does not know whether the application thread is in a valid state. It only knows the token still matches. A process that is paused, deadlocked, or running arbitrarily slowly will have its lock continuously renewed. This is not a safety improvement — it removes the only self-correcting mechanism (TTL expiry) that would allow another process to take over.

**The correct mental model:** A distributed lock TTL is a *fencing* mechanism, not a *lease extension* mechanism. If your operation cannot complete within the TTL, the correct response is to make the TTL longer at configuration time, or to accept that the lock may be lost and handle that case explicitly (e.g., via a fencing token / optimistic version check on the DB write).

```python
# What this code does (wrong):
# "If my operation is slow, keep renewing so I can finish."

# What a distributed lock should do (correct):
# "If my operation is slow, let the lock expire so another node can proceed,
#  and use a fencing token or DB-level version check to reject my stale write."
```

**Recommendation:** Remove the heartbeat entirely. Either:
- Increase `LOCK_TTL` to a value that conservatively covers the worst-case DB operation time, and document that bound explicitly.
- Or implement a fencing token pattern: pass a monotonically increasing token (e.g., from Redis `INCR`) to the DB write, and have the DB layer reject writes with a token older than the last committed one.

### 🔴 Critical — Heartbeat Has a Race Condition (Check-Then-Act)

Even if the heartbeat pattern were acceptable, the implementation is not atomic:

```python
# services/inventory.py — _start_heartbeat
current = self.redis.get(lock_key)        # Step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2: extend
```

Between step 1 and step 2, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is a TOCTOU (time-of-check/time-of-use) race. The correct approach is a Lua script that atomically checks ownership and extends in a single Redis operation:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In heartbeat:
extended = self.redis.eval(EXTEND_LOCK_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.warning("Heartbeat: lock no longer owned, stopping. key=%s", lock_key)
    self._heartbeat_active = False
```

The same issue exists in the lock release path — `get` then `delete` is not atomic. This is already a known pattern and should use the standard Lua script for atomic release.

### 🔴 Critical — Lock Release Is Not Atomic

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same TOCTOU race as above. Between `get` and `delete`, the lock can expire and be re-acquired by another process, which you then delete. This is a well-documented failure mode. Use the standard atomic Lua release script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 🟠 High — Shared Mutable State Makes the Service Non-Reentrant and Non-Thread-Safe

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal case in any web framework with a shared service layer), two concurrent calls will overwrite each other's `_heartbeat_active` and `_heartbeat_thread`. One call stopping its heartbeat will stop the other's.

**Fix:** Move heartbeat state into the `_acquire_lock` context manager as local variables, scoped to the lock acquisition, not the service instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread closure, not self
    ...
```

### 🟠 High — No Input Validation on Public Method

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is passed directly into a Redis key (`inventory:lock:{item_id}`) and a DB query with no sanitization. A caller passing `item_id = "foo:bar"` or `item_id = ""` or a very long string could cause unexpected behavior.
- `quantity` has no lower-bound check. `quantity=0` or `quantity=-5` would pass the stock check and silently create a nonsense order or increase stock.

```python
# Minimum required guards:
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be at least 1")
```

### 🟡 Medium — Heartbeat Thread Leak on Lock Acquisition Failure

```python
# _acquire_lock
if not acquired:
    raise RuntimeError(...)

self._start_heartbeat(lock_key, token)  # only reached if acquired
```

This path is fine. However, if `_start_heartbeat` itself raises (e.g., thread creation fails), `_stop_heartbeat` is never called because the `try` block hasn't been entered yet. This is a minor but real resource management gap.

---

### 🟡 Medium — `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` fails, stock is decremented with no order record. The distributed lock does not protect against this — it only serializes access. This needs to be wrapped in a DB transaction, or the DB layer needs to expose a combined atomic operation.

### 🟡 Medium — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat mocked out
    svc._stop_heartbeat = MagicMock()   # ← stop mocked out
    return svc
```

The heartbeat is the novel and risky component of this PR. Mocking it out means zero test coverage of the behavior being introduced. There is no test that:
- Verifies the heartbeat actually extends the TTL.
- Verifies the heartbeat stops after the lock is released.
- Verifies the race condition in the heartbeat (which, as noted above, exists).
- Verifies concurrent calls don't corrupt shared heartbeat state.

`test_lock_not_released_if_expired` also does not test what its docstring claims — it tests the mock fixture behavior, not the actual lock expiry path, because `_stop_heartbeat` is mocked.

---

### 🟢 Low — Minor Style / Robustness Notes

- `current.decode()` will raise if Redis returns non-UTF-8 bytes. Prefer `current.decode("utf-8", errors="replace")` or store/compare as bytes consistently.
- `LOCK_TTL` and `HEARTBEAT_INTERVAL` as class-level constants make them hard to tune per-deployment. Consider accepting them as constructor parameters with defaults.
- The `RuntimeError` catch in `reserve_item` is overly broad — it will also swallow `RuntimeError` raised by the DB layer, masking real errors as "service temporarily unavailable."
