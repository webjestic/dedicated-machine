Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly undermines the safety guarantee that distributed lock TTLs are designed to provide. Beyond that, there are several concurrency correctness issues and test coverage gaps that need to be resolved before this is production-ready for a high-demand flash sale context.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of Lock TTL (Architecture Flaw)

The PR summary frames heartbeat renewal as a *feature* that handles GC pauses and slow DB operations. This is precisely backwards. The TTL on a distributed lock exists as a **safety valve** — it guarantees that a lock held by a dead, paused, or partitioned process will eventually expire, allowing other nodes to make progress.

When a process is paused (GC, OS scheduling, network partition), the heartbeat thread is also paused. The lock *will* expire naturally in that case, which is correct behavior. The dangerous scenario this pattern actually creates is:

1. Process A acquires the lock and begins a slow DB write.
2. The lock TTL expires *before* the heartbeat fires (race condition — see Issue 2).
3. Process B acquires the lock and begins its own write.
4. Process A's heartbeat fires, re-extends the lock, and A's write completes.
5. **Both A and B now believe they hold the lock simultaneously.** This is the exact split-brain / stale write scenario the lock was supposed to prevent.

The heartbeat does not prevent this — it makes it harder to reason about and introduces a false sense of safety. The correct fix is to ensure DB operations complete well within the TTL, or to use fencing tokens (Redlock with monotonic version numbers) to detect and reject stale writes at the DB layer.

**Reference:** Martin Kleppmann's analysis of Redlock is directly applicable here. A lock that can be extended by a thread that may be running concurrently with another lock holder is not a safe distributed lock.

### 2. TOCTOU Race in Heartbeat Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are not atomic. Between the check and the `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This must be a single atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the release in `_acquire_lock`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is also a TOCTOU race. Use the standard Lua script for conditional delete:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 3. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two callers invoke `reserve_item` concurrently on the same `InventoryService` instance, they will race on `_heartbeat_active` and `_heartbeat_thread`. One caller's `_stop_heartbeat()` will kill the other caller's heartbeat. This is a silent correctness failure with no error raised.

Heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass state through a local object or closure, not `self`.

### 4. Input Validation Is Incomplete

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` with no validation:

- `item_id` is passed directly into a Redis key (`inventory:lock:{item_id}`) and a DB query with no sanitization. A malformed or adversarially crafted `item_id` (e.g., containing `:`, `*`, or newline characters) could cause key collisions in Redis or injection issues at the DB layer depending on the driver.
- `quantity` has no lower-bound check. `quantity=0` or `quantity=-5` would pass the `current_stock < quantity` check and produce nonsensical results (negative reservation, stock increase).

Minimum fix:

```python
if not isinstance(item_id, str) or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be at least 1")
```

Better fix: define a `ReservationRequest` Pydantic model and validate at the boundary.

## Moderate Issues

### 5. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` fails, stock is decremented with no corresponding order record. This is a partial write that leaves inventory in an inconsistent state. These two operations must be wrapped in a single DB transaction. The distributed lock serializes *access*, but it does not provide atomicity for the DB operations themselves.

---

### 6. Heartbeat Thread Leak on Lock Acquisition Failure Path

In `_acquire_lock`, if `_start_heartbeat` raises (e.g., thread creation fails), the `finally` block will call `_stop_heartbeat`, but there is no guarantee the thread was started. More importantly, if the `yield` raises an exception that is *not* caught by the caller, `_stop_heartbeat` is called but the thread may still be running if `_heartbeat_active` is checked on a delay. The daemon flag mitigates process-level leaks, but within a long-running service process, thread accumulation under error conditions should be explicitly handled with `thread.join(timeout=...)` in `_stop_heartbeat`.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and risky component in this PR, and it has zero test coverage. There are no tests verifying:

- That the heartbeat actually extends the TTL.
- That the heartbeat stops after lock release.
- That the heartbeat correctly skips extension when the token no longer matches.
- The TOCTOU race scenario described in Issue 2.

### 8. Concurrent Access Is Not Tested

For a service explicitly designed for high-concurrency flash sales, there are no tests using `threading.Thread` or `concurrent.futures` to simulate two goroutines racing on the same `item_id`. A test that spawns N threads all calling `reserve_item` on the same item with stock=1 and asserts exactly one success is a minimum bar for this use case.

### 9. Partial DB Write Failure Is Not Tested

No test covers the case where `update_stock` succeeds but `create_order` raises. Given Issue 5, this is a known failure mode with no test and no mitigation.
