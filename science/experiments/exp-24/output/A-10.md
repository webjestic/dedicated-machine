Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed TTL locks are designed to provide. Beyond that, there are several concurrency, correctness, and input-validation issues that need to be resolved before this is production-ready for a high-throughput flash sale context.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of a TTL Lock

This is the most serious design flaw, and the PR summary actually acknowledges the exact scenario that makes it dangerous.

The PR states the heartbeat exists to handle **stop-the-world GC pauses**. This is precisely the scenario where lock renewal is **unsafe**. The entire point of a TTL on a distributed lock is to ensure that if a process is paused, crashed, or partitioned, the lock expires and another process can make progress. If you renew the lock from a heartbeat thread while the main thread is paused, you are extending a lock that the main thread may no longer be able to complete work under safely.

The canonical reference here is Martin Kleppmann's analysis of Redlock. A process that is paused does not know it is paused. When it resumes, it may believe it still holds a valid lock and proceed to write stale data — and the heartbeat has ensured the lock is still technically valid, removing the one safeguard that would have prevented the stale write.

```
Timeline:
T=0   Main thread acquires lock, heartbeat starts
T=5   Main thread reads stock=10 from DB
T=6   GC pause begins — all threads frozen including heartbeat
T=15  GC pause ends
T=16  Heartbeat resumes, extends lock (lock was still valid, no other holder)
T=17  Main thread resumes, writes new_stock=9 — but DB may have been
      modified by another process that acquired the lock during the pause
      window if the heartbeat had NOT extended it
```

The heartbeat makes this window **larger and less predictable**, not smaller. The correct mitigations are:

- Use **fencing tokens** (monotonically increasing lock version numbers) passed to the DB layer, which rejects writes with stale tokens
- Use **optimistic locking** at the DB layer (compare-and-swap on a version column)
- Accept that the TTL is the safety boundary and size operations to complete well within it

### 2. Heartbeat Has a TOCTOU Race on Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # non-atomic
```

The `GET` and `EXPIRE` are two separate operations. Between them, the lock can expire and be acquired by another process. You would then extend a lock you no longer own. This must be a single atomic operation via a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the lock release in `_acquire_lock`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # non-atomic — lock can be stolen between GET and DELETE
```

This should also be a Lua script. This is a well-known requirement for correct Redis lock release and should not have passed internal review.

## Significant Issues

### 3. Shared Mutable Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the expected use case for a service object), two concurrent calls will overwrite each other's `_heartbeat_active` flag. One call stopping its heartbeat will silently stop the other's. This is a straightforward data race.

The heartbeat state must be scoped to each lock acquisition, not to the instance. Pass state through a local object or closure, not instance variables.

### 4. No Input Validation on `item_id` or `quantity`

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model gating the inputs to this method.

- `item_id` is interpolated directly into a Redis key: `f"inventory:lock:{item_id}"`. A caller passing `item_id = "foo:bar:baz"` or an empty string produces a malformed or colliding key.
- `quantity` has no lower bound check. `quantity=0` or `quantity=-5` would pass the stock check and write a nonsensical or corrupted stock value.

At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

### 5. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access, it does not provide atomicity. These two operations must be wrapped in a database transaction.

### 6. Exceptions From DB Layer Are Not Caught

If `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` raise any exception, it will propagate out of the `with self._acquire_lock(lock_key)` block. The `finally` clause will still run and release the lock correctly, but the caller receives an unhandled exception rather than a structured `ReservationResult`. Given this is a service method returning a result type, all DB exceptions should be caught and returned as a failure result with appropriate logging.

## Minor Issues

### 7. `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the heartbeat thread has stopped before the lock is released. The thread may be mid-sleep at `time.sleep(HEARTBEAT_INTERVAL)` and will wake up after the lock has been released and potentially re-acquired by another process, then attempt a `GET`/`EXPIRE` on a lock it no longer owns. The Lua script fix in issue #2 prevents the actual extension, but the thread should still be joined with a short timeout for correctness and to avoid accumulating zombie threads.

### 8. Test Coverage Gaps

- No test exercises the actual heartbeat thread behavior — it is mocked out entirely in the fixture, meaning the TOCTOU race in issue #2 is untestable as written
- No test covers the partial-failure scenario where `update_stock` succeeds but `create_order` raises
- No test covers concurrent calls on the same service instance, which would expose issue #3
- `test_lock_not_released_if_expired` verifies `delete` is not called but does not assert the reservation still returned a result, making it an incomplete assertion
