Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly undermines the safety guarantee that Redis distributed locks are designed to provide. Beyond that, there are several concurrency, validation, and transactional correctness issues that need to be resolved before this is anywhere near production-ready for a high-stakes flash sale scenario.

---

## Critical Issues

### 1. The Heartbeat Pattern Is Fundamentally Unsafe (Architecture)

The PR summary acknowledges GC pauses and then argues the heartbeat *solves* them. It does not — it inverts the safety contract.

The entire point of a TTL on a distributed lock is that if the process holding the lock dies, stalls, or is partitioned, the lock eventually releases and another node can make progress. The heartbeat defeats this. Consider:

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   Process A's main thread enters a stop-the-world GC pause (or kernel swap, or network partition)
T=10  Heartbeat thread is NOT paused (it's a separate OS thread) — it renews the TTL
T=25  Process A's main thread resumes, reads stale stock value from before the pause
T=25  Process B has been blocked the entire time, waiting on a lock that A "owns" but isn't doing useful work
T=25  Process A writes a stale decrement based on data that may now be wrong
```

The heartbeat thread and the main application thread can diverge. The heartbeat keeps the lock alive while the main thread is in an inconsistent state. This is not a theoretical concern — it is the exact failure mode Martin Kleppmann documented when critiquing Redlock. The correct mitigations are:

- **Fencing tokens**: The lock server issues a monotonically increasing token; the storage backend rejects writes with a lower token than the last seen. Redis alone cannot provide this.
- **Optimistic locking / CAS at the DB layer**: Use `UPDATE stock SET quantity = quantity - %s WHERE item_id = %s AND quantity >= %s` and check rows affected. This is the correct primitive for inventory.
- **Accept that distributed locks are advisory**, not a hard safety guarantee, and design the DB writes to be idempotent and self-validating.

The heartbeat should be removed. It provides false confidence.

### 2. `InventoryService` Is Not Thread-Safe (Concurrency Bug)

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If a single `InventoryService` instance is shared across threads (standard in any web framework — FastAPI, Flask with a thread pool, Django, etc.), two concurrent calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. One call's `_stop_heartbeat()` will kill the other call's heartbeat.

```
Thread 1: _start_heartbeat() → sets self._heartbeat_active = True
Thread 2: _start_heartbeat() → sets self._heartbeat_active = True (overwrites Thread 1's thread ref)
Thread 1: _stop_heartbeat() → sets self._heartbeat_active = False
Thread 2's heartbeat: now dead, lock will expire mid-operation
```

If the heartbeat pattern is kept (it shouldn't be), the active flag and thread reference must be local to each lock acquisition, not instance state. Pass them into the heartbeat closure or use a local object.

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is passed directly into `f"inventory:lock:{item_id}"` and then to `self.db.get_stock(item_id)`. A malicious or malformed `item_id` containing `:`, `*`, newlines, or Redis glob characters could cause lock key collisions or unexpected behavior.
- `quantity` has no lower bound check. `quantity=0` or `quantity=-5` will pass the `current_stock < quantity` check and silently *increase* stock (`new_stock = 10 - (-5) = 15`).

```python
# This succeeds silently and corrupts inventory:
service.reserve_item("item_001", quantity=-100)
# remaining_stock = 110, order created for -100 units
```

Minimum fix:

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
# Sanitize or allowlist item_id characters
```

### 4. Non-Atomic DB Operations Without Transactional Guarantee

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `update_stock` succeeds and `create_order` fails (DB timeout, exception, crash), stock is decremented but no order exists. This is a silent inventory leak. The distributed lock does not protect against this — the lock is a Redis primitive and has no knowledge of DB transaction state.

These must be wrapped in a single atomic DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 5. Heartbeat Has a TOCTOU Race on Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are not atomic. Between the `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends *the other process's lock*. This must be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same issue exists in the lock release in `_acquire_lock` — the `GET` + `DELETE` must also be atomic (this is a well-known pattern; use the standard Lua script for it).

---

### 6. `retry_attempts=3` With Backoff Is Inadequate and Blocks the Event Loop

With 3 attempts and exponential backoff of `0.1, 0.2, 0.4` seconds, the maximum wait is 0.7 seconds. Under real flash sale load (hundreds of concurrent requests), most will fail immediately. More importantly, `time.sleep()` in a thread pool blocks that worker thread entirely. For async frameworks this is a hard blocker; even for sync frameworks it exhausts the thread pool under load.

Consider:
- Configurable retry count and backoff parameters injected at construction time, not hardcoded.
- If using an async framework, use `asyncio.sleep` and an async Redis client.

## Test Issues

### 7. Tests Mock Away the Exact Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is mocked in every test. This means the tests provide zero coverage of the actual heartbeat behavior, the thread lifecycle, or the TOCTOU race in the token check. The tests validate the happy-path business logic only, which is fine, but the checklist claim of coverage for the heartbeat mechanism is false.

### 8. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

With `_start_heartbeat` mocked, the `redis.get` in the `finally` block will return `b"different-token"`, which won't match the generated UUID token, so `delete` is skipped. But the test is asserting behavior of the release path while the heartbeat (which could have changed the token state) is completely mocked. This test would pass even if the lock release logic were entirely deleted.

## Summary of Required Changes

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Remove heartbeat pattern; use DB-layer fencing or optimistic locking |
| 2 | Critical | Move heartbeat state to lock-local scope, not instance fields |
| 3 | Critical | Validate `quantity > 0` and sanitize `item_id` |
| 4 | Critical | Wrap DB operations in a single atomic transaction |
| 5 | High | Make heartbeat extend and lock release atomic via Lua scripts |
| 6 | Moderate | Make retry parameters configurable; audit for async compatibility |
| 7 | Low | Add real heartbeat thread tests; fix misleading checklist claim |
| 8 | Low | Rewrite `test_lock_not_released_if_expired` to test actual behavior |
