Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here: it directly undermines the safety guarantee that Redis distributed locks are designed to provide. Beyond that core issue, there are several concurrency, correctness, and test-coverage problems that must be resolved before this is mergeable.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of Lock TTL

This is the most serious problem and it is not a minor concern — it is a fundamental design error.

The TTL on a Redis lock exists precisely to bound the damage when a process hangs, crashes, or experiences a stop-the-world GC pause. The PR summary actually acknowledges GC pauses as a motivation for the heartbeat, but this reasoning is backwards.

Consider the scenario the PR is trying to protect against:

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   Process A enters a stop-the-world GC pause (all threads suspended,
      including the heartbeat thread)
T=30  Lock TTL expires — Redis correctly evicts the lock
T=31  Process B acquires the lock, reads stock=10, begins reservation
T=35  Process A resumes from GC pause, still believes it holds the lock
T=36  Process A writes new_stock=9, creates order
T=36  Process B writes new_stock=9, creates order
      → Two reservations issued for the same unit of stock
```

The heartbeat thread is also suspended during a GC pause. When the process resumes, the heartbeat cannot have renewed the lock during the pause. The lock may have already been acquired by another process. The token check in the release path catches this for the *delete*, but the *write operations* (`update_stock`, `create_order`) have already executed against a lock the process no longer holds. The heartbeat does not fix this — it only extends the window of false confidence.

This is the exact problem Martin Kleppmann documented in his critique of Redlock. The correct mitigation is a **fencing token** (a monotonically increasing integer returned by the lock server) passed through to the database write, where the DB rejects writes carrying a stale token. Redis `SET ... NX` with a UUID does not provide this.

```python
# What is actually needed: fencing at the storage layer
# The lock server issues a monotonic fence token
fence_token = self.redis.incr("inventory:fence:{item_id}")
# DB write must include and enforce this token
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
# DB rejects: UPDATE stock SET qty=? WHERE item_id=? AND last_fence_token < ?
```

If fencing is not feasible in this stack, the correct posture is to accept that the lock provides only a performance optimization (reducing conflicts), not a safety guarantee, and to enforce correctness at the database layer with optimistic concurrency control instead.

### 2. Shared Heartbeat State Breaks Under Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to a single lock acquisition. If two callers share an `InventoryService` instance (which is the normal pattern in a web framework with dependency injection or a singleton service), concurrent calls will corrupt each other's heartbeat state.

```python
# Thread 1 acquires lock for item_001, starts heartbeat, _heartbeat_active = True
# Thread 2 acquires lock for item_002, starts heartbeat, overwrites _heartbeat_thread
# Thread 1 finishes, calls _stop_heartbeat(), sets _heartbeat_active = False
# Thread 2's heartbeat is now silently dead — lock for item_002 will expire
```

The fix is to scope heartbeat state to the lock context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure, not self
```

### 3. Non-Atomic Check-Then-Act on Lock Release

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a TOCTOU race. Between `GET` and `DELETE`, another process could acquire the lock. The correct implementation is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The same race exists in the heartbeat renewal:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # another process may own it now
```

This also needs to be a Lua script or use `SET key token XX EX ttl` (SET with XX flag only sets if key exists, but still does not verify token atomically — Lua is required).

## Moderate Issues

### 4. No Input Validation on Public Method

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. There is no guard against:

```python
service.reserve_item("", quantity=0)
service.reserve_item("item_001", quantity=-5)
service.reserve_item("a" * 10000, quantity=1)
```

If Pydantic is already a dependency, use it:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Database Exceptions Are Not Handled

If `self.db.update_stock` raises, the lock is released correctly (the `finally` block runs), but the stock may be in a partially updated state if `update_stock` succeeded and `create_order` raised. There is no transaction boundary or compensating action. At minimum, this should be documented and ideally the two writes should be wrapped in a DB transaction.

### 6. `retry_attempts` Hardcoded at Call Site

`reserve_item` calls `_acquire_lock(lock_key)` without exposing `retry_attempts` to callers. This makes it impossible to tune retry behavior per call site or via configuration without modifying the internals.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out in Every Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and most dangerous part of this PR. Mocking it out entirely means the tests provide no coverage of the behavior being advertised. There should be at minimum:

- A test verifying the heartbeat actually extends TTL while the lock is held
- A test verifying the heartbeat stops after lock release
- A test verifying concurrent calls do not corrupt each other's heartbeat state (which would catch issue #2 above)

### 8. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is a data-integrity scenario that should be explicitly tested and handled.

### 9. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test asserts the lock is not deleted when the token does not match, which is correct. But it does not assert that the *write operations were also skipped*. In the current implementation they are not skipped — the reservation proceeds and writes to the DB even though the lock is no longer held. This test is passing for the wrong reason and is masking the core safety violation.
