Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed locks are supposed to provide. There are also thread-safety issues with the shared heartbeat state, missing input validation on public method parameters, and test gaps that mask real behavior. None of these are cosmetic; they affect correctness under the exact high-concurrency conditions this service is designed for.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of Lock TTL

This is the most serious problem, and it is explicitly endorsed in the PR description as a feature.

The PR states:

> "The 30-second LOCK_TTL combined with 10-second heartbeat renewal ensures the lock remains valid throughout any realistic process pause."

This reasoning is backwards. The TTL on a distributed lock is not a timeout to work around — it is the **safety boundary**. Its entire purpose is to ensure that if a process pauses (GC, network partition, kernel scheduling, etc.), the lock eventually expires so other nodes can make progress. A heartbeat that continuously renews the TTL removes that guarantee entirely.

The failure scenario this creates:

1. Process A acquires the lock and starts a reservation.
2. Process A experiences a stop-the-world GC pause of, say, 40 seconds.
3. The heartbeat thread is also paused (it runs in the same JVM/process — all threads stop during STW GC).
4. The lock expires. Process B acquires the lock and completes a reservation.
5. Process A resumes. Its heartbeat thread wakes up, checks the token — **but the token comparison is not atomic with the renewal**. Under a race, it may renew a lock it no longer owns.
6. Both processes now believe they hold the lock. Inventory is double-decremented.

The heartbeat does not solve the GC pause problem. It makes it worse by creating a false sense of safety. The correct mitigations are fencing tokens, optimistic locking at the database layer, or idempotency keys — not TTL extension.

```python
# The check-then-extend is a non-atomic TOCTOU race
current = self.redis.get(lock_key)          # check
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # extend  ← not atomic
```

Between `GET` and `EXPIRE`, another process can acquire the lock. This should use a Lua script if renewal is kept at all:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

But again — the correct fix is to remove the heartbeat and design for lock expiry being a normal event.

### 2. Shared Mutable Heartbeat State Is Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not per-lock-acquisition attributes.

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two callers invoke `reserve_item` concurrently on the same `InventoryService` instance (which is the normal case for a shared service object), they will race on these fields. The second caller's `_stop_heartbeat()` call sets `_heartbeat_active = False`, which stops the first caller's heartbeat thread. The first caller's lock then expires while it is still in use.

The fix is to scope heartbeat state to the lock acquisition, not the instance:

```python
# Pass a threading.Event or a local flag into the thread closure
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # renew...

# To stop:
stop_event.set()
```

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is only used on the return type.

The following inputs are not handled:

```python
service.reserve_item("", quantity=1)       # empty item_id → lock key becomes "inventory:lock:"
service.reserve_item("item_001", quantity=0)   # zero quantity passes the stock check
service.reserve_item("item_001", quantity=-5)  # negative quantity increases stock
```

A negative quantity would pass `current_stock < quantity` (e.g., `10 < -5` is `False`), call `update_stock(item_id, 15)`, and silently add inventory. This is a logic vulnerability.

```python
# Minimum required guard
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

## Moderate Issues

### 4. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `create_order` fails, stock has already been decremented and there is no compensating rollback. This leaves inventory in an inconsistent state. Both calls must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 5. `RuntimeError` Is Too Broad to Catch

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a base class that can be raised by many unrelated internal failures. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

And raise/catch only that. As written, a `RuntimeError` from inside the DB layer would be silently swallowed and returned as "Service temporarily unavailable," hiding real errors.

---

### 6. Heartbeat Thread Leaks on Unhandled Exceptions

If `self.db.get_stock()` raises an unexpected exception (not `RuntimeError`), it propagates out of the `with self._acquire_lock(lock_key):` block. The `finally` clause in `_acquire_lock` calls `_stop_heartbeat()`, which sets the flag to `False` — but the heartbeat thread is already sleeping and will not check the flag for up to `HEARTBEAT_INTERVAL` seconds. During that window, the lock may be renewed after it should have been released. Using `threading.Event` with `stop_event.set()` and `stop_event.wait(timeout=...)` eliminates this delay.

## Test Issues

### 7. Tests Mock Away the Behavior Under Test

The fixture stubs out both heartbeat methods:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means no test exercises the actual lock acquisition, heartbeat lifecycle, or lock release path. The tests are validating the business logic in isolation, which has value, but the distributed locking behavior — the primary feature of this PR — has zero real test coverage.

At minimum, add integration-style tests that use a real or fake Redis (e.g., `fakeredis`) and do not stub the lock machinery.

### 8. `test_lock_not_released_if_expired` Does Not Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

Because `_start_heartbeat` and `_stop_heartbeat` are mocked on the `service` fixture, and the token generated inside `_acquire_lock` will never match `b"different-token"`, this test does confirm `delete` is not called — but it does so by coincidence of the mock setup, not by exercising the actual ownership-check logic in the `finally` block. The test would pass even if the ownership check were deleted entirely, as long as the token never matches. This gives false confidence.

| # | Severity | Issue |
|---|----------|-------|

| 2 | Critical | Scope heartbeat state per-acquisition, not per-instance |
| 3 | High | Add input validation for `item_id` and `quantity` |
| 4 | High | Wrap DB calls in a transaction |
| 5 | Moderate | Replace broad `RuntimeError` catch with a specific exception type |
| 6 | Moderate | Use `threading.Event` to stop heartbeat immediately |
