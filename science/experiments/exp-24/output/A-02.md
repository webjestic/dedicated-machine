Decision: Request Changes

Summary: The PR demonstrates solid intent — atomic lock acquisition via `SET NX EX`, ownership-verified release, exponential backoff, and structured logging are all good foundations. However, there are several correctness, safety, and architecture issues that must be addressed before this is production-ready, the most critical being a race condition in the heartbeat's check-then-extend sequence and a shared-state concurrency bug that makes the service unsafe under concurrent use. The DB operations also lack transactional guarantees, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory state.

## Detailed Review

### 🔴 Critical — Must Fix

**1. Heartbeat has a TOCTOU race condition (check-then-act is not atomic)**

```python
# services/inventory.py — _start_heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are two separate round-trips. Between them, the lock can expire and be acquired by another process. You then extend a lock you no longer own, silently invalidating the new holder's mutual exclusion guarantee. This is the exact failure mode the PR claims to prevent.

**Fix:** Use a Lua script to make the check-and-extend atomic.

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning(
                    "Heartbeat lost lock ownership key=%s — signaling caller",
                    lock_key
                )
                self._heartbeat_active = False
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same TOCTOU problem exists in the lock release path and must receive the same treatment:

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

```python
# Fix — atomic Lua release
RELEASE_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""
```

**2. `InventoryService` instance is not safe for concurrent use**

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

Heartbeat state is stored as instance variables. If two threads call `reserve_item` on the same service instance simultaneously, they share `_heartbeat_active` and `_heartbeat_thread`. One thread stopping its heartbeat will kill the other's, and the thread reference will be overwritten. This is a silent data race.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through a local object or closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            result = extend_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            if result == 0:
                logger.warning(
                    "Heartbeat lost lock ownership key=%s", lock_key
                )
                heartbeat_active.clear()
                break

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        release_script = self.redis.register_script(RELEASE_LOCK_SCRIPT)
        release_script(keys=[lock_key], args=[token])
        logger.info("Lock released key=%s", lock_key)
```

Note also that `_stop_heartbeat` currently just sets a flag and returns immediately. The heartbeat thread may still be mid-execution when the lock is deleted, creating a window where a dead thread extends a lock it should not touch. The `thread.join()` above closes that window.

**3. Non-atomic DB operations will corrupt inventory on partial failure**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, loses its DB connection, or receives `SIGKILL` between these two calls, stock is decremented but no order exists. This is a silent inventory leak. The distributed lock provides no protection here because the failure is within the lock boundary.

**Fix:** These two writes must be wrapped in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If your DB layer does not support this, that is a prerequisite gap that must be resolved before this service ships.

**4. No input validation on `item_id` or `quantity`**

The checklist claims "Input validation via Pydantic," but there is none. `item_id` is used directly in a Redis key (`inventory:lock:{item_id}`) and passed to DB calls with no sanitization. A caller passing `item_id = "foo:bar"` will silently corrupt the lock key namespace. A negative or zero `quantity` will pass the stock check and decrement stock incorrectly.

```python
# Minimum viable validation
if not item_id or not isinstance(item_id, str) or ":" in item_id:
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")
```

Or introduce a proper request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]+$", min_length=1, max_length=64)
    quantity: int = Field(..., ge=1, le=1000)
```

---

**5. Heartbeat thread is daemon but never joined on normal exit**

Daemon threads are silently killed when the main thread exits, which is acceptable for true background services. Here the heartbeat thread holds a logical resource (the Redis lock TTL). If the process exits cleanly while the thread is sleeping, the lock will eventually expire naturally, which is fine. However, if the thread is in the middle of an `EXPIRE` call during shutdown, the operation may be left in an undefined state. The `join()` added in fix #2 resolves this.

---

**6. Backoff ceiling is missing**

```python
backoff = (2 ** attempt) * 0.1
```

With `retry_attempts=3` the maximum backoff is `0.4s`, which is fine today. But `retry_attempts` is a caller-controlled parameter. A caller passing `retry_attempts=20` produces a `52-second` blocking sleep in a single thread. Cap the backoff.

```python
backoff = min((2 ** attempt) * 0.1, 2.0)
```
