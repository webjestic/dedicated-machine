Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock with token ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and architecture issues that must be resolved before this is suitable for a production government contracting environment where inventory integrity is a hard requirement.

---

## Critical Issues

### 1. Race Condition in Heartbeat — Non-Atomic Check-and-Extend

```python
# CURRENT — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and `EXPIRE`, another process could acquire the lock. You are extending a lock you no longer own. This must be a single atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_lua = self.redis.register_script(HEARTBEAT_SCRIPT)

# In heartbeat thread
self._heartbeat_lua(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same race condition exists in `_acquire_lock`'s `finally` block and must receive the same treatment.

---

### 2. Instance-Level Heartbeat State Breaks Concurrent Usage

```python
# These are instance variables — shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously, they will clobber each other's `_heartbeat_active` flag. The heartbeat state must be scoped to the lock context, not the instance. Pass state through a local object or use a `threading.Event`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=2)
        # atomic delete via Lua
```

### 3. Database Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record. This is a data integrity failure. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract must expose a transaction context manager. This is non-negotiable for financial/inventory operations.

### 4. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" but there is no validation on `reserve_item`'s inputs. `item_id` could be an empty string, a path traversal attempt, or absurdly long. `quantity` could be zero or negative. Pydantic is only used on the *output* model.

## Moderate Issues

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread may still be mid-execution, attempting a Redis call after the lock has been released. Add a join with a bounded timeout:

```python
stop_event.set()
if heartbeat_thread and heartbeat_thread.is_alive():
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error(...)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` from anywhere in the call stack — including inside `db.get_stock` or `db.update_stock` — will be silently swallowed and reported as a lock failure. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

Raise and catch only that. Let unexpected errors propagate so they surface in monitoring.

---

### 7. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

Under a flash sale, many threads will be sleeping here simultaneously. For a service expected to handle high concurrency, consider async-compatible locking or at minimum document that callers must use a thread pool with bounded size. Sleeping inside a synchronous web request handler will exhaust your worker pool quickly.

---

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Away Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is one of the most complex and failure-prone parts of this implementation and has zero test coverage. At minimum, add:
- A test verifying the Lua script is called during heartbeat
- A test verifying the thread stops after `_stop_heartbeat`
- A test verifying the lock is not extended when the token no longer matches

### 9. `test_lock_not_released_if_expired` Passes for the Wrong Reason

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

Because `_start_heartbeat` is mocked, the token used internally is a real `uuid4()` which will never equal `b"different-token"`. The test passes, but it is not actually testing the ownership verification path through real code. This test gives false confidence.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but document explicitly that `HEARTBEAT_INTERVAL` must always be less than `LOCK_TTL / 2`.
- Consider whether `db` should be typed against a Protocol/ABC so the interface contract is explicit and mockable without surprises.
- Log the `quantity` in the insufficient stock path to aid debugging.

---

The lock ownership verification pattern and the overall structure are on the right track. Resolve the atomicity issues (items 1 and 3 especially) and the instance-level concurrency bug (item 2) before re-submitting. Those three together represent a realistic data corruption path under production load.
