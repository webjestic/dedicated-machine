Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and concurrency issues that must be resolved before this can be approved for a government-facing production system. The test suite also has meaningful gaps that mask the bugs present in the implementation.

---

## Critical Issues

### 1. Heartbeat Race Condition (Check-Then-Act on Token)

The heartbeat's token check is not atomic:

```python
# services/inventory.py — _beat() inner function
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock can expire or be stolen between these two lines
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. This extends a lock you no longer own, which is a correctness violation that can cause double-reservation — a direct inventory integrity failure.

**Fix:** Use a Lua script to atomically check ownership and extend TTL:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    script = self.redis.register_script(HEARTBEAT_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if not result:
                    logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
                    self._heartbeat_active = False

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

### 2. Lock Release Is Also Non-Atomic (Same Race)

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can acquire between GET and DELETE
```

This is the canonical distributed lock mistake. If the lock expires between `GET` and `DELETE`, you delete a lock owned by another process, corrupting the mutual exclusion guarantee.

**Fix:** Use the standard Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. `InventoryService` Is Not Safe for Concurrent Use

The heartbeat state is stored as instance variables:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of this service — they will corrupt each other's heartbeat state. `_stop_heartbeat()` on one reservation will kill the heartbeat for the other.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through a local object or threading event:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    stop_event = threading.Event()

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        released = self._release_script(keys=[lock_key], args=[token])
        if released:
            logger.info("Lock released key=%s", lock_key)
        else:
            logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 4. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts arbitrary `item_id` strings and `quantity` integers with no validation whatsoever. A negative quantity, zero quantity, or a malformed `item_id` (e.g., containing Redis key injection characters like `:` or `*`) will pass through silently.

**Fix:** Add a request model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        request = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValueError as e:
        logger.warning("Invalid reservation request item_id=%s: %s", item_id, e)
        return ReservationResult(success=False, message="Invalid request parameters")

    # proceed with request.item_id and request.quantity
```

The `pattern` constraint on `item_id` is particularly important — it prevents a malformed ID from constructing a malicious Redis key like `inventory:lock:*` or `inventory:lock:item:lock:admin`.

## Significant Issues

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

Setting the flag and returning immediately means the heartbeat thread may still be running when the lock is released. In the worst case, the thread wakes up after `_stop_heartbeat()` returns but before `redis.delete()` executes, sees the token still matches, and extends the TTL — preventing the delete from being the final operation. Join the thread before releasing:

```python
stop_event.set()
heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 6. Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The item is effectively lost. This is a data integrity failure that the distributed lock does not protect against.

**Fix:** Wrap in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architectural gap that needs to be addressed before this service can be considered production-ready.

---

### 7. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

This catches any `RuntimeError` raised inside the lock context — including ones thrown by `db.update_stock` or `db.create_order`. A database failure mid-reservation would be silently swallowed and reported as a lock failure, making incidents very difficult to diagnose.

**Fix:** Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) instead of RuntimeError
# catch only LockAcquisitionError at the call site
```

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Away in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test mocks out the heartbeat entirely. The atomic token-check bugs in the heartbeat and release logic are completely untested. At minimum, add integration-level tests that use a real or fake Redis (e.g., `fakeredis`) to exercise the actual lock lifecycle.

### 9. Concurrency Is Not Tested

The entire value proposition of this service is safe concurrent reservation. There are no tests that simulate two threads attempting to reserve the same item simultaneously. This is the highest-risk
