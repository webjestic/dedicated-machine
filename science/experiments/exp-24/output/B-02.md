Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here: it explicitly defeats the safety guarantee that distributed lock TTLs are designed to provide. If a process pauses (GC, network partition, host suspension), the heartbeat also pauses — but the PR's stated goal is to handle exactly that case by keeping the lock alive *before* the pause, meaning a resumed process with a stale world-view will confidently proceed with writes it no longer has exclusive authority to make. Beyond that core issue, there are several additional correctness, safety, and design problems that must be addressed before this is mergeable.

---

## Detailed Findings

### 🔴 Critical — Architecture

**1. Heartbeat renewal inverts the safety model of distributed locks**

The PR summary explicitly states the heartbeat is meant to survive stop-the-world GC pauses. This is precisely the wrong behavior. A distributed lock TTL exists so that *if a process freezes*, the lock expires and another node can safely take over. A heartbeat that pre-extends the TTL before a pause means:

1. Process A holds the lock, heartbeat extends it to T+30.
2. Process A freezes for 25 seconds (GC, swap, VM suspension).
3. No other node can acquire the lock — availability is degraded.
4. Process A resumes, believes it still owns the lock, and writes — but its read of `current_stock` is now 25 seconds stale.
5. Another process may have legitimately acquired the lock after the *original* TTL would have expired, already decremented stock, and committed — Process A now double-decrements.

The heartbeat does not solve the stale-read problem; it conceals it while making the race window larger. The correct approach is to treat lock expiry as a signal that the operation's preconditions are invalid and abort, not to fight the clock.

```python
# What should happen instead: validate the lock is still held
# immediately before committing writes, and abort if it is not.
# Redlock / fencing tokens are the standard solution here.
```

**2. No fencing token / monotonic version guard on DB writes**

Even without the heartbeat issue, `db.update_stock` and `db.create_order` are called with no version assertion. If the Redis lock expires between the read and the write (a real possibility under load), two processes can both pass the stock check and both write. The DB layer must enforce a conditional update:

```python
# Correct pattern: optimistic concurrency at the DB layer
rows_affected = self.db.update_stock_if_version(item_id, new_stock, expected_version)
if rows_affected == 0:
    raise StaleDataError("Concurrent modification detected — aborting reservation")
```

**3. `update_stock` and `create_order` are not atomic**

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists — inventory is silently lost. These two operations must be wrapped in a single DB transaction.

```python
# Required
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🔴 Critical — Correctness

**4. Shared heartbeat state breaks concurrent use**

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (the normal case in a web server), one call's `_stop_heartbeat()` will kill the other call's heartbeat mid-operation, or the second call will overwrite `_heartbeat_thread` before the first has stopped.

```python
# The state must be scoped to the lock acquisition, not the instance.
# Pass a threading.Event into _start_heartbeat and cancel that specific event.
def _start_heartbeat(self, lock_key, token):
    stop_event = threading.Event()
    def _beat():
        while not stop_event.wait(self.HEARTBEAT_INTERVAL):
            ...
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event, t  # caller holds the cancellation handle
```

**5. Heartbeat check-then-extend is a TOCTOU race**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock can expire between these two calls
```

This is a classic check-then-act race. The lock can expire between `GET` and `EXPIRE`. This must be an atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

**6. Lock release is also a TOCTOU race**

The same pattern in `finally`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process could acquire between GET and DELETE
```

This must also be an atomic Lua script (this is the standard Redlock release pattern):

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 🟡 Major — Input Validation

**7. `item_id` and `quantity` are not validated**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input path. An empty string, a negative quantity, or a zero quantity will silently pass through.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

**8. `quantity` can be zero or negative**

`current_stock < quantity` passes when `quantity` is 0 or negative, resulting in stock being *increased* via `new_stock = current_stock - quantity`.

---

### 🟡 Major — Observability & Operations

**9. Lock acquisition failure swallows the attempt count**

The `RuntimeError` message says "after N attempts" but the caller only sees "Service temporarily unavailable." Structured logging should record the item, attempt count, and final backoff duration so on-call engineers can distinguish a brief spike from a sustained deadlock.

**10. No metrics / alerting hooks**

For a flash sale service — which by definition operates under extreme load — there are no counters for lock contentions, reservation failures, or heartbeat extension failures. This will make the post-mortem for the first production incident much harder than it needs to be.

---

### 🟡 Major — Testing

**11. Tests mock away the exact behavior under review**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture stubs out the heartbeat. This means the TOCTOU races in the heartbeat (findings 5 and 6), the shared-state concurrency bug (finding 4), and the lock-release race are entirely untested. At minimum there should be integration-level tests using a real or fake Redis (e.g., `fakeredis`) that exercise the full lock lifecycle.

**12. No test for the partial-write failure case**

There is no test covering `update_stock` succeeding and `create_order` raising. Given the missing transaction (finding 3), this is the most important failure mode to cover.

---

### 🟢 Minor

- `logger.debug` inside the heartbeat thread will be noisy at scale; consider rate-limiting or only logging on extension failure.
- `LOCK_TTL` and `HEARTBEAT_INTERVAL` as class-level constants make them hard to tune per-environment; prefer constructor injection or a config object.
- Daemon thread teardown on process exit is implicit; document that in-flight reservations may not complete cleanly on SIGTERM.
