Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed locks are supposed to provide. Beyond that, there are several concurrency, correctness, and robustness issues that need to be resolved before this is production-ready for a high-demand flash sale scenario.

---

## Critical Issues

### 1. The Heartbeat Defeats the Purpose of a TTL Lock

This is the most serious problem, and it is architectural, not cosmetic.

The PR summary explicitly acknowledges stop-the-world GC pauses and frames the heartbeat as the solution. This reasoning is backwards. The TTL exists precisely to handle the case where a process is paused, crashes, or becomes unresponsive. If a process is paused long enough for the lock to expire, that expiry is the *correct and intended behavior* — it allows another node to take over and make progress.

The heartbeat renewal pattern creates the exact race condition it claims to prevent:

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   Process A enters STW GC pause (all threads suspended, including heartbeat)
T=30  Redis expires the lock (correct behavior)
T=31  Process B acquires the lock, begins reading stock
T=32  Process A resumes from GC pause, continues writing stock
T=32  Both A and B now hold the "lock" simultaneously
```

The heartbeat thread is suspended during a GC pause just like every other thread. It cannot extend the lock during the pause. When the process resumes, the lock may have already been acquired by another node, but Process A has no mechanism to detect this before it proceeds with its database writes. The token check in the `finally` block only protects the *delete* operation, not the critical section itself.

**The correct approach** is to accept that a process paused long enough for its lock to expire must treat its reservation as failed when it resumes. This is enforced by making the lock TTL a hard deadline, not a renewable lease. If operations genuinely require more than 30 seconds, the architecture needs to be reconsidered — not the lock TTL extended.

If lock renewal is truly required for long-running operations, the correct pattern is a fencing token (a monotonically increasing integer from Redis), passed through to the database layer, where the DB rejects any write with a stale fence value. This is the approach described in the Redlock analysis by Martin Kleppmann. The current implementation has no fencing.

### 2. Shared Heartbeat State Breaks Concurrent Usage

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` on the same `InventoryService` instance concurrently — which is the entire point of a flash sale service — they will race on `_heartbeat_active` and `_heartbeat_thread`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, and vice versa. The heartbeat state must be local to each lock acquisition, not stored on the instance.

```python
# Broken: shared state across concurrent callers
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False  # kills every active heartbeat
```

A minimal fix would be to pass a `threading.Event` into `_start_heartbeat` and return it, keeping all state local to the `_acquire_lock` context. But given Issue 1, the heartbeat should be removed entirely.

### 3. Check-Then-Act Race in the Heartbeat

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are not atomic. Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat would then extend a lock it no longer owns. This must use a Lua script to be correct:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same issue exists in the `finally` block for the delete, though that one is less dangerous since it only affects cleanup.

## Significant Issues

### 4. No Input Validation on Public Entry Points

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is no guard against:

- Empty or whitespace-only `item_id` (which would produce a valid but meaningless lock key)
- Negative `quantity`
- Zero `quantity`
- Excessively long `item_id` strings that could affect Redis key sizing

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # No validation before lock acquisition
```

A Pydantic model should be used as the method signature, or explicit validation should occur before any Redis or DB interaction.

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `update_stock` succeeds and `create_order` fails, stock is decremented but no order exists. This is a data integrity issue independent of the locking strategy. These must execute within a single database transaction, and any exception from either call must roll back both.

### 6. Broad Exception Suppression Hides Real Failures

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` is caught, but `redis.exceptions.RedisError`, `redis.exceptions.ConnectionError`, and any exception from the DB layer will propagate uncaught. Conversely, a `RuntimeError` raised from inside the critical section (e.g., from `db.update_stock`) will be caught here and silently reported as a lock failure, which is incorrect and will mislead on-call engineers. The exception handling scope needs to be tightened and differentiated.

## Test Coverage Gaps

### 7. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — the primary novel mechanism in this PR — is never exercised by any test. There is no test for the concurrent instance-state collision described in Issue 2, no test for the non-atomic GET/EXPIRE race in Issue 3, and no test verifying that a resumed-after-pause process detects lock loss before writing. The test suite validates the happy path business logic but provides no coverage for the distributed systems behavior this PR is specifically designed to handle.

### 8. `test_lock_not_released_if_expired` Does Not Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    """Lock release is skipped if token no longer matches (lock expired or stolen)."""
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test verifies that `delete` is not called when the token does not match. It does not test that the *write operations* (`update_stock`, `create_order`) are skipped or rolled back in this scenario. The writes happen before the token check, so this test is asserting correct cleanup behavior while the actual data corruption path goes untested.

## Minor Issues

- `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants with no ability to override per-environment. These should be constructor parameters with sensible defaults.
- The `_acquire_lock` context manager yields `token` but `reserve_item` ignores it (`with self._acquire_lock(lock_key):`). If fencing tokens are introduced later, this yield is the right place, but currently it is unused.
- No timeout is set on the Redis client itself. A Redis connection hang will block the calling thread indefinitely, holding no lock but consuming a thread.
- The `retry_attempts` default of 3 with backoffs of 0.1s, 0.2s, 0.4s means a maximum wait of 0.7 seconds before failure. Under genuine flash sale load this will produce a high failure rate. The retry strategy should be configurable and documented against expected lock hold times.

---

## Recommended Path Forward

1. Remove the heartbeat renewal mechanism entirely. Accept lock expiry as the correct failure signal.
2. If long-running operations are unavoidable, implement fencing tokens at the database layer.
3. Move heartbeat state (if retained) to be local to each lock acquisition context.
4. Make the token check and extend atomic via a Lua script.
5. Wrap the two DB calls in a single transaction.
6. Add input validation before lock acquisition.
7. Narrow and differentiate exception handling.
8. Write tests that exercise the distributed failure modes, not just the business logic happy path.
