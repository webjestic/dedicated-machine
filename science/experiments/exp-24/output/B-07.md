Decision: Request Changes

Summary: The heartbeat renewal pattern fundamentally undermines the safety guarantee that distributed lock TTLs are designed to provide. A process that has experienced a stop-the-world GC pause, network partition, or any other form of suspension *should* lose its lock — that is the entire point of the TTL — and a heartbeat that fights to keep the lock alive during exactly those scenarios can cause two processes to believe they hold the lock simultaneously, producing the phantom concurrency and stale writes this PR claims to prevent. Beyond that architectural flaw, there are several additional correctness and safety issues detailed below.

---

## Critical Issues

### 1. Heartbeat Pattern Inverts the Safety Contract of Distributed Locks

The PR summary explicitly states the heartbeat is intended to keep the lock alive *through* stop-the-world GC pauses. This is the wrong design goal.

```
Timeline of the failure mode this code enables:

T=0   Process A acquires lock, heartbeat starts
T=5   Process A enters STW GC pause — all threads suspended, including heartbeat
T=15  Redis TTL expires, lock is gone
T=16  Process B acquires the lock legitimately
T=20  Process B reads stock=10, begins reservation
T=22  Process A resumes from GC pause
T=22  Process A's heartbeat fires, checks token...
      BUT: if A's token is still in Redis (race), it extends the TTL
T=22  Both A and B now operate under the assumption they hold the lock
T=23  Double-spend / inventory corruption
```

The heartbeat check (`current.decode() == token`) is not atomic with the `expire` call. Between the `GET` and the `EXPIRE`, another client can acquire the lock. This is a TOCTOU race:

```python
# NOT ATOMIC — vulnerable to TOCTOU
current = self.redis.get(lock_key)          # GET
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # EXPIRE — separate round trip
```

The correct pattern for extending a lock atomically requires a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

Even with that fix, the heartbeat-renewal approach should be reconsidered entirely. If the operation genuinely takes longer than the TTL, the correct response is to design the operation to be shorter, or to accept that the lock was lost and abort with a conflict error rather than silently continuing.

### 2. The Lock Release Has the Same TOCTOU Race

```python
# Also NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # Another client could acquire between GET and DELETE
```

This is a well-known problem with naive Redis locking. The canonical fix is a Lua script, which Redis executes atomically:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    script(keys=[lock_key], args=[token])
```

### 3. Instance-Level Heartbeat State Breaks Concurrent Usage

```python
# Stored on the instance, not scoped to the lock context
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal case in any web framework with a shared service layer), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat or causing the stop flag to bleed across contexts. These must be local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into _start_heartbeat, keep it local
    ...
```

## Moderate Issues

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no `ReservationRequest` model being validated. A negative `quantity`, an empty string `item_id`, or a pathologically long string all pass through unchecked:

```python
# These all reach the database layer unvalidated
service.reserve_item("", quantity=-1)
service.reserve_item("../../etc/passwd", quantity=0)
service.reserve_item("a" * 10_000, quantity=1)
```

Add a request model and validate at the boundary:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Database Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against this — it only serializes access. These two operations must be wrapped in a database transaction. The current code has no exception handling around the DB calls inside the lock, so any DB exception will propagate out of the `with` block, trigger `_stop_heartbeat`, and leave the inventory in a partially updated state.

### 6. Daemon Heartbeat Thread Is Silently Swallowed on Errors

The heartbeat thread is a daemon thread with no error reporting. If it throws an unhandled exception (e.g., Redis connection failure), it dies silently and the lock will eventually expire with no warning. At minimum, wrap the loop body in a try/except and log the failure, and consider setting a flag that causes the main thread to abort the operation.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The most dangerous code in this PR — the heartbeat thread — has zero test coverage. There are no tests verifying that the heartbeat actually extends the TTL, that it stops when the lock is released, or that it handles a Redis failure gracefully.

### 8. The TOCTOU Races Are Not Tested

There is no test simulating the condition where the lock token changes between the `GET` and the `DELETE` or between the `GET` and the `EXPIRE`. The test `test_lock_not_released_if_expired` only checks the happy-path of the ownership check, not the race itself.

### 9. DB Transaction Failure Is Not Tested

No test covers the case where `create_order` raises after `update_stock` succeeds. This is the partial-update scenario that would cause inventory corruption in production.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Remove or fundamentally redesign the heartbeat; if kept, the extend operation must be atomic via Lua |
| 2 | Critical | Lock release must be atomic via Lua script |
| 3 | Critical | Heartbeat state must be local to each lock context, not instance-level |
| 4 | Moderate | Add and enforce a `ReservationRequest` Pydantic model with field constraints |
| 5 | Moderate | Wrap DB operations in a transaction; handle partial failure |
| 6 | Moderate | Heartbeat thread must log and signal on error, not die silently |
| 7 | Moderate | Add real tests for heartbeat behavior and failure modes |
| 8 | Low | Add tests for TOCTOU race conditions |
| 9 | Low | Add test for partial DB failure scenario |
