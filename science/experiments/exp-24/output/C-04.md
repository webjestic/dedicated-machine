Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and concurrency issues that must be resolved before this can be approved for a government-facing production system.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Thread Safety / Concurrency Correctness

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`. If a single service instance handles concurrent calls (e.g., via a thread pool or async framework), two simultaneous `reserve_item` calls will stomp on each other's heartbeat state.

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass a mutable container or use a `threading.Event`:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return stop_event  # caller signals stop via stop_event.set()
```

This also eliminates the `time.sleep` + double-check pattern, which has a race window.

### 2. Non-Atomic Lock Release — TOCTOU Race

The lock release is a read-then-delete, not atomic:

```python
# Current — race condition between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock after expiry and you would delete their lock. This must be a single atomic operation via a Lua script, which is the industry standard for this exact problem:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a similar Lua script.

---

### 3. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is no guard against:

- Empty or whitespace-only `item_id` (would create a valid lock key `inventory:lock:`)
- Negative `quantity` (would *increase* stock silently)
- Zero `quantity` (meaningless reservation succeeds)
- Excessively long `item_id` strings used as Redis keys

```python
# Minimum required
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item ID")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

A Pydantic request model at the call boundary would be the correct fix to match the checklist claim.

## Moderate Issues

### 4. Database Operations Outside Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction. If `create_order` fails after `update_stock` succeeds, stock is decremented with no corresponding order — inventory is silently lost. These must execute within a single atomic database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. At minimum, the PR description must acknowledge this risk if the DB layer handles it externally.

---

### 5. Heartbeat Can Extend a Lock Indefinitely

There is no maximum lock hold time. A deadlocked or infinitely slow DB operation will keep renewing the lock forever because the heartbeat has no ceiling. Add a `MAX_LOCK_HOLD` constant and enforce it:

```python
MAX_LOCK_HOLD = 120  # seconds — hard ceiling regardless of heartbeat

# In _beat loop
if time.monotonic() - start_time > self.MAX_LOCK_HOLD:
    logger.error("Max lock hold time exceeded, stopping heartbeat key=%s", lock_key)
    stop_event.set()
    break
```

---

### 6. Bare `RuntimeError` for Lock Failure Is Too Broad

Catching `RuntimeError` in `reserve_item` will silently swallow unrelated runtime errors from the DB layer, masking real bugs as "service unavailable." Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat logic — the most novel and failure-prone part of this PR — has zero test coverage. At minimum, add:

- A test that the heartbeat calls `expire` while active
- A test that the heartbeat stops calling `expire` after `_stop_heartbeat`
- A test that the heartbeat does not extend a lock it no longer owns (token mismatch)

### 8. No Test for the Negative Quantity / Zero Quantity Edge Cases

Once input validation is added, these paths need explicit test coverage.

### 9. `test_reserve_lock_contention` Does Not Test Retry Behavior

The test confirms failure after contention but does not verify that retry with backoff was actually attempted the correct number of times (`redis.set` should be called `retry_attempts` times).

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at 10s with 20s remaining — acceptable margin, but document the reasoning explicitly so future engineers don't tighten the interval without understanding the dependency.
- Consider whether `daemon=True` on the heartbeat thread is appropriate. Daemon threads are killed abruptly on interpreter shutdown, which could leave locks unreleased in edge cases. This is a known tradeoff worth a comment.
- The `db` parameter type is untyped. Define a `Protocol` or abstract base class for the DB interface to make the contract explicit and testable.
