Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed TTL locks are designed to provide. When a process pauses (GC, network stall, OS scheduling), the heartbeat thread pauses with it, meaning the TTL expires as intended and another node correctly acquires the lock; when the paused process resumes and its heartbeat fires, it races to extend a lock that may now be legitimately owned elsewhere, creating a split-brain window where two processes believe they hold the lock simultaneously. Beyond that core issue, there are several additional concerns that need to be addressed before this can be approved.

---

**Critical — Distributed Safety Violation**

The PR summary explicitly states the heartbeat solves stop-the-world GC pauses, but this reasoning is backwards and dangerous. A GC pause suspends *all* threads, including the heartbeat thread. The TTL expiring during that pause is the *correct* behavior — it is the system's only mechanism for detecting that a lock holder is unresponsive. When the process resumes, it should not assume it still holds the lock. The heartbeat does not solve this problem; it masks it while introducing a new one.

```
Timeline illustrating the race:

T=0   Process A acquires lock (TTL=30s)
T=15  Process A enters GC pause — all threads suspended, heartbeat stops
T=30  Lock TTL expires — Redis correctly evicts it
T=31  Process B acquires the lock legitimately
T=32  Process A resumes from GC, heartbeat fires, sees its token... wait.

Actually the token check saves you here — BUT only if B has already
written its token. There is a narrow window between T=30 and T=31
where A resumes, heartbeat fires, re-extends the now-expired key
with A's token (Redis SET NX is not used on expire, just EXPIRE),
and B has not yet acquired it. A then proceeds with a lock it
should not hold, and B acquires it moments later.

Result: concurrent writes to db.update_stock and db.create_order.
```

The token check in the heartbeat (`current == token`) does not close this window because `EXPIRE` does not verify ownership atomically — it extends whatever key exists at that moment.

**Required fix:** Remove the heartbeat entirely. Accept that if an operation takes longer than `LOCK_TTL`, the lock expires and the operation must be considered failed. Design the DB operations to complete well within the TTL, or use a fencing token pattern with a monotonic counter written to the DB layer so stale writes are rejected at the storage level regardless of lock state.

---

**Critical — Non-Atomic Heartbeat Extension**

Even setting aside GC pauses, the heartbeat's check-then-extend sequence is a TOCTOU race:

```python
# NOT atomic — another process can acquire the lock between these two calls
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

This must be a single atomic operation. The correct implementation uses a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The lock release in `_acquire_lock` has the same TOCTOU problem between `GET` and `DELETE` and should use the same Lua pattern.

---

**Critical — Instance-Level Heartbeat State**

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`, meaning a single `InventoryService` instance cannot safely handle concurrent calls:

```python
# Thread 1 calls reserve_item("item_A") — sets self._heartbeat_active = True
# Thread 2 calls reserve_item("item_B") — sets self._heartbeat_active = True
# Thread 1 finishes — calls self._stop_heartbeat(), sets self._heartbeat_active = False
# Thread 2's heartbeat is now silently dead while its lock is still held
```

Heartbeat state must be scoped to the lock acquisition context, not the service instance.

---

**Moderate — Input Validation Not Enforced**

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is no guard against:

```python
service.reserve_item("", quantity=0)       # empty item_id, zero quantity
service.reserve_item("../etc/passwd", quantity=-5)  # path traversal in lock key
service.reserve_item("item:" + "A"*10000)  # oversized Redis key
```

A negative `quantity` would *increase* stock. Pydantic should be used on a request model at the method boundary, or explicit guards added:

```python
if not item_id or not item_id.isalnum():
    raise ValueError("item_id must be a non-empty alphanumeric string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

---

**Moderate — DB Operations Outside Transaction**

`update_stock` and `create_order` are two separate DB calls with no transaction wrapping:

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # fails — stock is now wrong
```

A failure between these two calls leaves stock decremented with no corresponding order. These must execute in a single atomic transaction, and the distributed lock does not substitute for DB-level atomicity.

---

**Minor — Tests Mock Away the Code Under Test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The tests validate happy-path logic but do not exercise the lock lifecycle, heartbeat behavior, or the TOCTOU scenarios described above. The `test_lock_not_released_if_expired` test name implies coverage of expiry behavior, but the heartbeat is mocked out so no actual expiry path is tested. Integration tests against a real or fake Redis instance (e.g., `fakeredis`) are needed.
