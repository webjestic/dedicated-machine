Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, exponential backoff, and structured logging are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real flash sale load in a government-adjacent production environment.

---

## Critical Issues

### 1. Race Condition in Heartbeat: Non-Atomic Check-and-Extend

```python
# CURRENT — two separate commands, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be a single atomic operation using a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In heartbeat:
self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same issue exists in the lock release path, though `redis-py`'s recommended Lua pattern should be applied there too for consistency and safety.

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance variables**, not per-invocation. If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool or async worker), two calls will share `_heartbeat_active`. The first call's `_stop_heartbeat()` will kill the second call's heartbeat thread.

Fix: scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = threading.Thread(
        target=self._beat,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        # optionally join with short timeout
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no guard against:

- Empty string `item_id` → produces lock key `inventory:lock:` 
- Negative or zero `quantity`
- Excessively long `item_id` strings (potential key abuse)

Add a request model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 4. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record. This is a silent data integrity failure — inventory is consumed but no order exists. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your `db` abstraction, but the transactional boundary is non-negotiable for an inventory system.

## Significant Issues

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread could still be mid-execution (inside `redis.expire`) when the lock is released. Add a join with a bounded timeout:

```python
heartbeat_active.clear()
heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if heartbeat_thread.is_alive():
    logger.warning("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

---

### 6. Bare `RuntimeError` for Lock Failure Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
```

`RuntimeError` is a built-in that many libraries raise for unrelated reasons. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

Raise and catch that instead. This prevents accidentally swallowing unrelated runtime errors and returning a misleading "service unavailable" message.

---

### 7. Test Suite Mocks Out the Entire Heartbeat

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the race condition in point #1 — is completely untested. There should be at least one integration-style test that exercises the real heartbeat path, including verifying that the Lua script is called and that the lock is not extended after it has been released.

---

### 8. `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

In any async or thread-pool context, this blocks the worker thread for up to `0.1 + 0.2 = 0.3` seconds per call. For a flash sale with high concurrency, this will exhaust thread pools quickly. If the service runs under an async framework (FastAPI, etc.), this should use `asyncio.sleep` or the retry should be offloaded. At minimum, document the threading model assumption.

---

## Minor Issues

- **No maximum `quantity` guard in the lock key** — `item_id` is user-supplied and goes directly into the Redis key. Validate and sanitize before constructing the key.
- **`LOCK_TTL = 30` is a class constant** — this should be configurable at instantiation time for different deployment environments (test vs. production).
- **`logger.error` on lock contention** — lock contention during a flash sale is expected behavior, not an error. The log level should be `warning` at most, or it will flood error dashboards.
- **No Redis connection error handling** — if `redis.set` raises a `ConnectionError`, it propagates uncaught. Redis failures should be caught and surfaced as a clean service error, not an unhandled exception.
