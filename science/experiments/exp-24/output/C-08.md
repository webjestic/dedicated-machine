Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architectural issues that must be addressed before this handles real flash sale load in a government contracting context where reliability and auditability are non-negotiable.

---

## Critical Issues

### 1. Race Condition in Heartbeat — Non-Atomic Token Check + Extend

```python
# CURRENT — two separate commands, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could be stolen between these two lines
```

Between `GET` and `EXPIRE`, another process could acquire the lock. You'd then extend a lock you no longer own. This must be a single atomic operation via a Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same race exists in `_acquire_lock`'s finally block, but there it's less dangerous since you're releasing — the existing Lua-based conditional delete pattern (Redlock spec) should be used there too.

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables. If two threads call `reserve_item` on the **same `InventoryService` instance** simultaneously (very likely under flash sale load), one call's `_stop_heartbeat()` will kill the other call's heartbeat thread. This is a silent correctness failure — no exception, no log, just a dropped lock.

**Fix:** Scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Lost lock during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                break

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire, start thread, yield, finally clear event and join thread
```

Using `threading.Event` also eliminates the `time.sleep` polling anti-pattern and makes shutdown cleaner.

### 3. No Input Validation on `item_id` and `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with zero validation. `ReservationResult` is a response model, not an input validator.

```python
# item_id="../../../../etc/passwd" or item_id="" will pass through today
# quantity=-1 will silently "add" stock
```

Add an explicit input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

A negative quantity is particularly dangerous — `current_stock < quantity` would be `10 < -1` which is `False`, so the reservation would succeed and `new_stock` would be `10 - (-1) = 11`, **silently increasing stock**.

### 4. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` throws after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent reads, but it does not provide rollback semantics. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction doesn't support transactions, that's an architecture gap that needs to be resolved before this merges.

## Moderate Issues

### 5. Bare `RuntimeError` Is Too Broad in the Except Clause

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` can be raised by many things other than lock acquisition failure — including bugs in your own DB layer. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined — no guarantee it has stopped before lock is released
```

The heartbeat thread could still be running (sleeping in `time.sleep`) when the finally block checks the token and deletes the lock. In the worst case, the thread wakes up after deletion and extends a lock now owned by someone else (if the key was re-acquired extremely quickly). Join with a timeout:

```python
self._heartbeat_active = False
if self._heartbeat_thread:
    self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

With the `threading.Event` fix from Issue 2, this becomes `event.clear(); thread.join(timeout=...)` which is cleaner.

---

### 7. Test Coverage Gaps

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic has **zero test coverage** despite being the most complex and failure-prone part of the implementation. At minimum add:

- A test that verifies the heartbeat calls `expire` when the token matches
- A test that verifies the heartbeat stops when the token no longer matches
- A test for the negative quantity edge case (currently would pass and corrupt stock)
- A test verifying DB transaction rollback on `create_order` failure

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at 10s, leaving a 20s window before expiry. That's fine, but document the invariant explicitly: `HEARTBEAT_INTERVAL must be < LOCK_TTL / 2`.
- Log the `quantity` in the lock contention warning — it's useful for debugging flash sale patterns.
- Consider whether `RuntimeError` from the DB layer (e.g., connection timeout) should surface as a 503 or be retried. Right now it would propagate uncaught out of `reserve_item`.
