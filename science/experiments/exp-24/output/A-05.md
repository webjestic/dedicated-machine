Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed locks are supposed to provide. Beyond that, the service carries several concurrency, atomicity, and input validation gaps that would make it unsafe to deploy under real flash-sale load. The test suite mocks away the heartbeat entirely, meaning the most dangerous code path has zero coverage.

## Detailed Review

### Critical: Heartbeat Renewal Defeats the Purpose of Lock TTL

This is the most serious issue and it needs to be understood clearly before anything else.

The PR summary frames the heartbeat as a feature: *"prevents stale writes when the process resumes."* It is actually the opposite — it **enables** stale writes.

The TTL on a distributed lock is not a convenience timeout. It is a **fencing mechanism**. If a process holding a lock pauses (GC, network partition, OS scheduling), the TTL allows the cluster to declare that process dead and let another node proceed. When the paused process resumes, it finds its lock gone and must abort. That is the correct behavior.

The heartbeat subverts this entirely:

```python
# The heartbeat thread is also suspended during a stop-the-world GC pause.
# When the process resumes, the heartbeat resumes too — and extends the lock
# even though another node may have already acquired it and begun writing.
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The token check (`current.decode() == token`) does not protect against this. The token is a UUID stored in Redis. If the original lock expired and a second process acquired it with the same key (different token), the check correctly blocks renewal. But if the lock has **not yet expired** when the paused process resumes — which is exactly the scenario the heartbeat is designed to create by extending the TTL — the original process will find its token still present, extend the lock, and continue writing concurrently with the second process that acquired the lock during the pause window.

The PR summary explicitly acknowledges GC pauses as the motivation. This is precisely the scenario described in the Redlock literature as unsafe. The heartbeat does not solve the GC pause problem; it widens the race window.

**The correct mitigations are fencing tokens (monotonic counters from a lock server, passed to the database layer so stale writes are rejected at the storage level) or accepting that long-pausing operations should not hold distributed locks and redesigning the critical section to be short.**

### Critical: Non-Atomic Check-Then-Act Without Fencing

```python
current_stock = self.db.get_stock(item_id)
# ... gap here ...
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

The lock is supposed to serialize this sequence, but as described above, the lock guarantee is broken by the heartbeat. Even setting that aside, `update_stock` and `create_order` are two separate database calls with no transaction boundary shown. If `update_stock` succeeds and `create_order` fails, stock is decremented with no corresponding order. There is no rollback, no compensation, and no mention of transactional semantics in the DB interface.

This needs either a database transaction wrapping both calls, or a saga/outbox pattern with explicit compensation logic.

### High: Shared Mutable State Makes the Service Non-Reentrant

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the normal case in any web framework with a shared service object), they will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa.

The heartbeat state must be scoped to the lock acquisition context, not the instance. A local variable or a per-call object is required.

```python
# Safer pattern — keep state local to the context manager
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread, not self
```

### High: Lock Release Has a TOCTOU Race

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # delete — separate round trip
```

Between the `GET` and the `DELETE`, another process can acquire the lock. This is a classic TOCTOU. The standard fix is a Lua script that makes the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In release:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### Medium: No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `item_id` is non-empty, no maximum length, no check that `quantity` is a positive integer. A caller passing `quantity=0` or `quantity=-5` would decrement stock in the wrong direction or produce a nonsensical result. A Pydantic model is used for the *output* (`ReservationResult`) but not the input. The checklist item is misleading.

```python
# Minimum required
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="item_id is required")
if quantity < 1:
    return ReservationResult(success=False, message="quantity must be at least 1")
```

### Medium: Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread may still be mid-execution (inside `redis.expire`) when the lock is released. The `finally` block proceeds to check and delete the lock while the heartbeat thread may simultaneously be extending it. Add a join with a short timeout:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread and self._heartbeat_thread.is_alive():
        self._heartbeat_thread.join(timeout=2.0)
```

### Medium: Tests Mock Away the Code Under Scrutiny

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- the dangerous code is never executed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture bypasses the heartbeat entirely. The concurrency behavior, the shared-state race, the thread lifecycle — none of it is tested. There are no integration tests, no tests under simulated concurrent load, and no test that exercises the actual `_beat` loop. The test suite provides false confidence.

---

### Low: Retry Backoff Timing Is Misaligned With Lock TTL

With `retry_attempts=3` and backoff of 0.1s, 0.2s, 0.4s, the total wait before giving up is 0.7 seconds. The lock TTL is 30 seconds. A caller will fail fast even though the lock holder has up to 30 seconds remaining. This is probably not the intended behavior for a flash sale where the lock holder is doing DB work. Either the retry count, the backoff multiplier, or the TTL needs to be reconsidered in relation to the others.

---

### Low: `RuntimeError` Is Too Broad for Lock Failure

Catching `RuntimeError` in `reserve_item` will also swallow unrelated runtime errors from the DB layer or Redis client. Use a dedicated exception class:

```python
class LockAcquisitionError(Exception):
    pass
```
