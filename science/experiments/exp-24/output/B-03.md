Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed locks are supposed to provide. A lock with an auto-renewing TTL can survive indefinitely through process pauses, network partitions, or silent hangs, meaning a second node can never safely acquire the lock even after the first node has genuinely stalled, which is exactly the split-brain scenario Redis TTLs exist to prevent. Beyond that, the service has a thread-safety problem where a single `InventoryService` instance cannot safely serve concurrent requests because `_heartbeat_active` and `_heartbeat_thread` are instance-level fields rather than per-invocation state, so two simultaneous calls to `reserve_item` will race on those fields and one heartbeat will silently stop the other's renewal. There are also several secondary issues that need to be addressed before this is production-ready.

---

### Critical Issues

**1. Heartbeat renewal defeats the purpose of the TTL**

The PR summary explicitly acknowledges stop-the-world GC pauses but frames the heartbeat as a *solution* to them. It is not — it is the problem made worse. The correct mental model is:

- The TTL is a *safety fuse*. If the lock holder pauses, crashes, or partitions, the TTL ensures the lock is eventually released so another node can make progress.
- A heartbeat that keeps renewing the TTL while the holder is paused (the heartbeat thread is still running even when the main thread is GC-paused) means the lock never expires, and the second node is blocked indefinitely.
- If the process crashes entirely, the heartbeat stops and the TTL fires correctly — but that is the easy case. The dangerous case is a live-but-stalled process, and the heartbeat makes that case *worse*, not better.

The correct approach for long-running critical sections is to make the DB operations faster, use optimistic locking with a version/CAS column, or accept that the TTL is a hard upper bound and design operations to complete within it. Redlock does not endorse heartbeat renewal for this reason.

**2. Instance-level heartbeat state is not thread-safe**

```python
# Both concurrent calls write to the same fields:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests call `reserve_item` concurrently on the same service instance, the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first call's `_stop_heartbeat` sets `_heartbeat_active = False`, killing the second call's renewal. The fix is to make heartbeat state local to each lock acquisition, not stored on `self`.

**3. Check-then-act race on lock release is not atomic**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock. This is a classic TOCTOU bug. The standard fix is a Lua script that makes the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This same race exists in the heartbeat's `expire` call — it checks ownership and then extends, but those two operations are not atomic either.

### Moderate Issues

**4. No input validation on `reserve_item`**

The checklist claims input validation via Pydantic, but `item_id` and `quantity` are raw parameters with no validation. `quantity` can be zero or negative, and `item_id` is an unchecked string. A request with `quantity=-5` would *increase* stock. Add explicit guards:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

**5. DB operations are not transactional**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. These two writes must be wrapped in a database transaction. The distributed lock serializes access but does not substitute for DB-level atomicity.

**6. `RuntimeError` is too broad to catch**

```python
except RuntimeError as e:
```

`RuntimeError` can be raised by many things other than lock acquisition failure — including bugs inside the lock body. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

And raise/catch only that.

---

### Minor Issues

**7. Tests mock away the exact behavior under test**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test in the suite mocks out the heartbeat, so there is zero test coverage of the heartbeat logic itself, the atomic release path, or concurrent behavior. At minimum, add an integration-style test with a real or fakeredis instance. The `test_lock_not_released_if_expired` test is particularly misleading — it asserts behavior of the TOCTOU-vulnerable release path and treats it as correct.

**8. Daemon thread lifecycle is not explicitly managed**

Using `daemon=True` means the heartbeat thread is silently killed when the main process exits. That is acceptable, but if the `finally` block in `_acquire_lock` raises before `_stop_heartbeat` is called, the heartbeat can outlive the lock context. Ensure `_stop_heartbeat` is the very first operation in `finally`.

**9. `time.sleep` in the lock retry blocks the event loop**

If this service is ever called from an async context, the blocking `time.sleep` in the retry loop will stall the thread. This is not a blocker for a purely synchronous service, but it should be documented or an async variant should be planned.
