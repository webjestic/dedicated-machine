Decision: Request Changes

Summary: The heartbeat renewal pattern is the core architectural flaw here — it directly defeats the safety guarantee that distributed lock TTLs are designed to provide. If a process pauses (GC, network partition, host suspension), the heartbeat also pauses, meaning the TTL *already* handles that case correctly by expiring the lock; if the process is *running*, it should not need 30 seconds to complete a stock decrement and order insert, and if it does, that is a database performance problem to fix, not a locking problem to paper over. Beyond that, there are several additional correctness, concurrency, and security issues detailed below.

---

## Critical Issues

### 1. Heartbeat Pattern Inverts the Safety Model of Distributed Locks

This is the most serious architectural concern and the stated justification in the PR summary makes it worse, not better.

The PR argues the heartbeat is needed to survive stop-the-world GC pauses. But consider what actually happens:

- Process A holds the lock and enters a GC pause.
- The heartbeat thread is **also paused** — it is a thread in the same process.
- The TTL expires. Redis correctly evicts the lock.
- Process B acquires the lock and begins its critical section.
- Process A resumes from GC, believes it still holds the lock (its `token` variable is unchanged), and proceeds to write.

You now have **two processes simultaneously inside the critical section**. The heartbeat does not solve the GC problem; it creates a false sense of safety while leaving the actual race condition intact. The correct mitigation for this class of problem is fencing tokens (e.g., Redlock with monotonic version numbers written to the DB row, rejected if stale).

If the heartbeat *is* running (no pause), it is extending a lock that should not need extending. A stock read + decrement + order insert should complete in well under a second. A 30-second TTL with renewal is a signal that the critical section is too large or the DB is too slow.

```python
# What the PR claims protects against GC pause:
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # also paused during GC
        ...
        self.redis.expire(lock_key, self.LOCK_TTL)  # never reached during pause

# Result: lock expires anyway, but now two processes think they own it
# after the paused process resumes with a stale token in memory
```

### 2. Non-Atomic Lock Release (TOCTOU Race)

The lock release is a two-step operation: `GET` then `DELETE`. Between those two calls, the lock can expire and be acquired by another process, which you then delete.

```python
# Current code — NOT atomic:
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # step 2: delete — another owner by now?
```

This must be a single atomic operation. The standard solution is a Lua script evaluated atomically by Redis:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

### 3. Shared Mutable State on the Instance (`_heartbeat_active`, `_heartbeat_thread`)

`InventoryService` is almost certainly instantiated once and shared across request threads (typical for a service object). `_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, meaning concurrent calls to `reserve_item` will race on them.

- Thread A sets `_heartbeat_active = True` and starts its heartbeat.
- Thread B calls `reserve_item`, sets `_heartbeat_active = True` (fine), then later sets it to `False` when *its* operation completes.
- Thread A's heartbeat loop reads `_heartbeat_active == False` and exits, abandoning A's lock mid-operation.

These must be scoped to the individual lock acquisition, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure, not self
    ...
```

## Significant Issues

### 4. No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no guard against:

- Empty string `item_id` — produces a valid Redis key `inventory:lock:` and a DB query with an empty string
- `item_id` containing Redis key separators or injection characters
- `quantity <= 0` — a reservation of zero or negative quantity would succeed and corrupt stock

```python
# Minimum necessary:
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

### 5. DB Operations Outside Transaction Boundary

`update_stock` and `create_order` are two separate DB calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented with no corresponding order — inventory is silently lost.

```python
# Current:
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # raises — stock gone, no order
```

These must be wrapped in a DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Heartbeat Does Not Handle Redis Connectivity Failures

The heartbeat thread silently swallows Redis errors. If Redis becomes unavailable mid-operation, the lock will expire, another process will acquire it, and the original process will continue unaware:

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)  # can raise redis.RedisError
            # exception propagates into daemon thread, silently dies
```

At minimum, log the failure and signal the owning thread to abort the critical section.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out in All Tests

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # bypassed entirely
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the primary novel mechanism in this PR and it has zero test coverage. There are no tests verifying:
- That the heartbeat actually extends the TTL
- That it stops after lock release
- That it does not extend a lock it no longer owns
- Concurrent `reserve_item` calls on the same instance (which would expose issue #3)

### 8. TOCTOU in Release Is Not Tested Under Real Conditions

`test_lock_not_released_if_expired` mocks `redis.get` to return a different token, which tests the conditional path but does not test the atomicity gap between `GET` and `DELETE`. The test passes today but would also pass with the broken two-step implementation, giving false confidence.

## Minor Issues

- `time.sleep()` inside `_acquire_lock` blocks the calling thread. For a high-concurrency flash sale service, consider async Redis clients (`aioredis`) and `asyncio.sleep`.
- `RuntimeError` is too broad for a lock acquisition failure. A domain-specific exception (`LockAcquisitionError`) would allow callers to distinguish this from unexpected runtime errors.
- The `retry_attempts=3` default with backoffs of 0.1s, 0.2s, 0.4s means a caller waits up to 0.7 seconds before failure. Under flash sale load this should be configurable and documented.
