Decision: Request Changes

Summary: The core locking logic is well-structured and the intent is sound, but there are several meaningful flaws — a thread-safety race condition in the heartbeat, a shared-state concurrency bug that makes the service non-reentrant, and a non-atomic lock release — that would cause real failures under the exact load conditions this service is designed to handle. The test suite also mocks away the heartbeat entirely, leaving the most complex and failure-prone subsystem completely untested. These issues need to be resolved before this can be approved for a production flash-sale context.

---

## Detailed Review

### 1. Critical: Shared Heartbeat State Makes the Service Non-Reentrant

**File:** `services/inventory.py`

```python
# Current — instance-level state
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance, not scoped to the lock acquisition. If two threads call `reserve_item` on the same `InventoryService` instance concurrently — which is the entire point of this service — one thread calling `_stop_heartbeat()` will set `_heartbeat_active = False` and silently kill the other thread's heartbeat. The second reservation will then lose its lock mid-operation with no error raised.

**Fix:** Scope heartbeat state to each lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning("Lost lock ownership key=%s — stopping heartbeat", lock_key)
                heartbeat_active.clear()

    # ... acquire lock, start thread with _beat, yield, then:
    finally:
        heartbeat_active.clear()
        # release lock
```

Using a `threading.Event` per acquisition also eliminates the race condition in item 2 below.

### 2. Race Condition Between Heartbeat Check and `time.sleep`

**File:** `services/inventory.py`

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # <-- sleeps BEFORE checking flag
        if self._heartbeat_active:
            ...
```

`_stop_heartbeat()` sets the flag to `False`, but the thread may already be inside `time.sleep(10)`. It will sleep for up to 10 seconds after the lock has been deleted before it wakes up and exits. In a tight loop scenario this is benign, but it means the thread lingers well past the end of the critical section. With `threading.Event.wait(timeout=...)` as shown above, the thread wakes immediately when the event is cleared.

### 3. Non-Atomic Lock Release (Check-Then-Delete)

**File:** `services/inventory.py`

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # step 1
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # step 2
```

Between `GET` and `DELETE`, another process could acquire the lock. This is a classic TOCTOU bug. The correct pattern is a Lua script that makes the ownership check and deletion atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

finally:
    heartbeat_active.clear()
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

This is the standard Redlock pattern and is well-documented in the Redis documentation.

### 4. Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts arbitrary `item_id: str` and `quantity: int` with no validation. Pydantic is imported and used for the *output* model only. A negative quantity, an empty string, or a zero quantity would pass through silently.

### 5. Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access. These two operations must be wrapped in a database transaction. The current code will silently lose inventory on any DB error between the two calls.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Heartbeat Is Mocked Out in All Tests

**File:** `tests/test_inventory.py`

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- completely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test uses this fixture. The heartbeat thread — the most operationally complex and failure-prone part of this PR — has zero test coverage. At minimum, add:

- A test that verifies `expire` is called after `HEARTBEAT_INTERVAL` elapses
- A test that verifies the heartbeat stops when the token no longer matches
- A test that verifies the heartbeat thread does not extend a lock it no longer owns

The `test_lock_not_released_if_expired` test is a good instinct but it only tests the release path, not the heartbeat renewal path.

---

### 7. Minor: `retry_attempts` Defaults Are Not Configurable at Construction Time

Hardcoding `retry_attempts=3` in `_acquire_lock` is fine for now, but `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants while retry behavior is a parameter. For consistency and testability, consider moving all tuning parameters to `__init__`.
