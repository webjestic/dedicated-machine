Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly undermines the safety guarantee that distributed TTL locks are designed to provide. Beyond that, there are several concurrency correctness issues, missing input validation, and test gaps that need to be addressed before this is production-ready.

---

## Detailed Review

### 🔴 Critical — Architecture

#### 1. The Heartbeat Defeats the Purpose of the TTL Lock

The PR summary acknowledges stop-the-world GC pauses and frames the heartbeat as a *solution*. It is not. It is the exact failure mode the lock TTL exists to prevent.

The safety model of a distributed TTL lock is:

> If the lock holder dies, stalls, or becomes unresponsive, the TTL ensures the lock is eventually released and another process can make progress.

A heartbeat thread running in the *same process* as the paused application thread does not help here. During a stop-the-world pause, **all threads are suspended**, including the heartbeat. The heartbeat cannot extend the lock while the process is paused.

What this pattern actually introduces is the opposite problem: if the process *appears* healthy (heartbeat is running) but the critical section is executing slowly due to a GC pause or slow DB call, the lock TTL is extended indefinitely. If the process then crashes or is killed *after* the heartbeat has extended the lock but *before* the operation completes, the lock is held for up to another full `LOCK_TTL` window (30 seconds) before another process can acquire it. You have traded a bounded worst-case for an unbounded one.

The correct mitigations for slow operations under a distributed lock are:

- **Fencing tokens** — a monotonically increasing token issued by the lock service; the database rejects writes with a stale token. This is the only mechanism that provides a true safety guarantee.
- **Reduce critical section duration** — move slow I/O outside the lock boundary where possible.
- **Accept the TTL expiry** — detect it, roll back, and retry rather than trying to prevent it.

The heartbeat pattern is appropriate in systems like ZooKeeper or etcd session leases where the lock service itself tracks liveness via a session. It is not appropriate here.

### 🔴 Critical — Concurrency Correctness

#### 2. Shared Heartbeat State Breaks Under Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal case for a service object — two concurrent calls will overwrite each other's `_heartbeat_active` and `_heartbeat_thread`. Calling `_stop_heartbeat` from one request will stop the heartbeat for a different concurrent request's lock.

This needs to be scoped to the lock acquisition context, not the instance.

#### 3. Check-Then-Act Race in Heartbeat and Lock Release

```python
# heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

```python
# release
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Both of these are non-atomic check-then-act sequences. Between the `GET` and the `EXPIRE` or `DELETE`, another process could acquire the lock. The standard fix is a Lua script that performs the check and action atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is well-established practice and should not be omitted.

#### 4. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. This is a data integrity bug independent of the locking strategy. These must execute within a single database transaction.

### 🟡 Significant — Input Validation

#### 5. `quantity` and `item_id` Are Not Validated

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitive arguments with no validation. There is no `ReservationRequest` model. Specifically:

- `quantity=0` or `quantity=-1` will pass the `current_stock < quantity` check and silently create a zero-quantity or negative-quantity order.
- `item_id` is an unvalidated string — no length check, no format enforcement.

A minimal fix:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 🟡 Significant — Test Coverage Gaps

#### 6. Heartbeat Is Mocked Away in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test mocks out the heartbeat entirely. There is no test for the actual heartbeat behavior: does it extend the lock? Does it stop when the context exits? Does it handle a missing or stolen lock correctly? The tests provide false confidence.

#### 7. DB Transaction Failure Is Not Tested

There is no test covering the case where `update_stock` succeeds and `create_order` raises. Given the missing transaction boundary noted above, this is the most important failure path to cover.

#### 8. Concurrent Access Is Not Tested

There are no tests exercising two threads calling `reserve_item` on the same instance simultaneously. Given the shared instance-level heartbeat state, this is a necessary test.

---

### 🟢 Acknowledged — What Is Done Well

- `nx=True, ex=self.LOCK_TTL` on the `SET` is correct atomic lock acquisition.
- Exponential backoff on contention is appropriate.
- Daemon thread prevents the heartbeat from blocking process shutdown.
- `ReservationResult` as a typed return value is clean.
- Structured logging with contextual fields is good practice.
- The lock release does check token ownership before deleting — the atomicity fix is needed, but the intent is correct.
