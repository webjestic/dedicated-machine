Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly contradicts the safety guarantee that distributed TTL locks are designed to provide. Beyond that, there are several concurrency, correctness, and validation issues that must be resolved before this is suitable for a production inventory system.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of the TTL Lock

This is the most serious problem in the PR, and the PR summary actually acknowledges the scenario while drawing the wrong conclusion.

The PR states the design handles "stop-the-world GC pauses that temporarily suspend all application threads." In Python (CPython), the GIL means you won't have true STW GC pauses the way you would in JVM languages, but the broader problem is real and the proposed solution makes it *worse*, not better.

The entire point of a TTL on a distributed lock is to guarantee that a crashed, hung, or partitioned process cannot hold a lock indefinitely. If the process is paused, network-partitioned, or enters an infinite loop *after* acquiring the lock but *before* completing the critical section, the heartbeat thread will keep renewing the lock. This means:

- A hung process holds the lock forever.
- Other nodes are permanently blocked.
- The safety property of the lock is eliminated.

The correct approach for operations that may exceed the TTL is to either (a) design the critical section to complete well within the TTL, or (b) use a fencing token / monotonic version number written to the database so that a stale write from a process that lost its lock can be detected and rejected at the storage layer. Redis's own documentation and Martin Kleppmann's analysis of Redlock both address this directly.

```python
# The heartbeat thread should be removed entirely.
# If LOCK_TTL = 30s is not enough, the DB operations need to be optimized,
# or the architecture needs a fencing token approach.
```

### 2. Instance-Level Heartbeat State Is Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are instance variables on a shared `InventoryService` object. If two concurrent callers invoke `reserve_item` simultaneously on the same service instance, they will race on these fields.

```python
# Caller A sets _heartbeat_active = True, starts thread A
# Caller B sets _heartbeat_active = True, starts thread B
# Caller A finishes, sets _heartbeat_active = False
# Thread B is now also stopped, but Caller B is still in its critical section
# Caller B's lock will expire undefended (under the heartbeat model)
```

If the heartbeat pattern were kept (it should not be), these fields would need to be local to each lock acquisition context, not stored on `self`.

### 3. Check-Then-Act Race on Lock Release Is Not Atomic

```python
# In _acquire_lock finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic TOCTOU race. Between the `GET` and the `DELETE`, another process could acquire the lock (if it expired between those two calls). You would then delete a lock you do not own. This must be an atomic operation using a Lua script:

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 4. No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is only used on the *output* model `ReservationResult`. There is no guard against:

- `quantity <= 0` (reserving zero or negative items would silently succeed or corrupt stock)
- `item_id` being an empty string or containing injection-relevant characters depending on the DB layer
- `quantity` being an arbitrarily large integer

```python
# Minimum required:
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

A proper fix would introduce a `ReservationRequest` Pydantic model as the input.

## Moderate Issues

### 5. `db.update_stock` and `db.create_order` Are Not Atomic

If `update_stock` succeeds but `create_order` raises an exception, stock has been decremented but no order record exists. This is a partial write that leaves the system in an inconsistent state. These two operations must be wrapped in a database transaction.

```python
# Pseudocode — exact API depends on your DB layer:
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Swallowing All Exceptions From the Critical Section

Only `RuntimeError` from lock acquisition is caught. Any exception raised by `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate out of the `with self._acquire_lock(lock_key)` block uncaught, bypassing the caller's error handling and potentially surfacing raw stack traces. The `finally` block in `_acquire_lock` will still run (good), but the caller of `reserve_item` receives an unhandled exception rather than a `ReservationResult`. This should be caught and returned as a structured failure.

### 7. Single Redis Node Is Not a Safe Distributed Lock

The PR uses a single `redis.Redis` client. A single-node Redis lock does not provide the safety guarantees of a distributed lock — if that node fails or restarts, all locks are lost and concurrent processes may proceed simultaneously. For a government contracting context where correctness matters, this should either use Redis Sentinel/Cluster with appropriate acknowledgment of the limitations, or the codebase should include a comment explicitly acknowledging this is an availability-optimized single-node lock, not a safety-critical distributed lock.

## Test Issues

### 8. Tests Mock Out the Exact Behavior Being Tested

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- mocked
    svc._stop_heartbeat = MagicMock()   # <-- mocked
    return svc
```

The heartbeat is mocked in most tests, so the tests do not cover the actual lock lifecycle. `test_lock_not_released_if_expired` is the only test that exercises the release path, but it does not test the TOCTOU race described in Issue 3 because the mock bypasses the Lua script requirement entirely. Integration tests against a real Redis instance (e.g., using `pytest-redis` or a Docker fixture) are required for the lock behavior to be meaningfully tested.
