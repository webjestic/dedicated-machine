Decision: Request Changes

Summary: The PR demonstrates solid intent — atomic lock acquisition via `SET NX EX`, token-verified release, exponential backoff, and structured logging are all good foundations. However, there are several correctness, safety, and architectural issues that must be addressed before this is suitable for a high-concurrency flash sale environment. The most critical flaw directly contradicts the PR's own safety rationale: the heartbeat mechanism it relies on to survive GC pauses cannot itself survive a GC pause, meaning the core threat model is unresolved.

---

## Critical Issues

### 1. The Heartbeat Cannot Survive the Problem It Claims to Solve

This is the central architectural contradiction in the PR summary.

> *"stop-the-world GC pauses that temporarily suspend all application threads"*

The heartbeat renewal thread **is an application thread**. A stop-the-world GC pause suspends it along with everything else. If the pause is long enough for the lock to expire, the heartbeat was also paused and could not renew it. The PR's own justification for the heartbeat is invalid.

This is not a theoretical concern — it is the exact failure mode described in Martin Kleppmann's analysis of Redlock and distributed locking in general. The correct mitigation is a **fencing token** (a monotonically increasing value returned at lock acquisition time, passed to the database layer, which rejects writes from stale tokens). The database, not the lock, becomes the authority on write validity.

```python
# What the DB layer should enforce — not shown anywhere in this PR
def update_stock(item_id: str, new_stock: int, fence: int) -> bool:
    # Reject if fence <= last_seen_fence for this item
    ...
```

Without fencing, a process that resumes after a long pause can still commit a stale write even if the lock has expired and been re-acquired by another process.

### 2. Race Condition in Heartbeat: Check-Then-Act Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by a different process. The heartbeat then extends a lock it no longer owns. This must be a single atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern already used conceptually for release should be applied here consistently.

---

### 3. Lock Release Is Also Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same TOCTOU race as above. This is a well-known pattern that must use a Lua script. The Redis documentation explicitly calls this out:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 4. `InventoryService` Is Not Safe for Concurrent Use — Instance-Level Heartbeat State

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-level locals. If `reserve_item` is called concurrently on the same service instance (the normal case in a web framework with a shared service object), two concurrent calls will clobber each other's `_heartbeat_active` and `_heartbeat_thread`. One call stopping its heartbeat will stop the other's.

The heartbeat state must be scoped to the individual lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread closure, not self
    ...
```

### 5. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is used only for the *output* model `ReservationResult`. There is no guard against:

- `quantity <= 0` (reserving zero or negative items)
- Empty or malformed `item_id` strings that could produce malformed Redis keys or SQL injection depending on the DB layer
- Excessively large `quantity` values

At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

## Moderate Issues

### 6. `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` fails, stock is decremented with no corresponding order record. This is a data integrity issue independent of the locking strategy. These two operations must be wrapped in a database transaction. The PR gives no indication this is handled at the `db` layer, and the tests do not cover this failure path.

### 7. Heartbeat Thread Leak on Unhandled Exceptions

If `self.db.get_stock(item_id)` raises an unexpected exception (network error, ORM error), it propagates through the `with` block. The `finally` clause in `_acquire_lock` calls `_stop_heartbeat()`, which sets `_heartbeat_active = False`, but the heartbeat thread is sleeping and will not observe this until it wakes. With a 10-second interval, the thread lingers up to 10 seconds. In a high-throughput environment under failure conditions, this accumulates. The thread should use a `threading.Event` with `wait(timeout)` instead of `time.sleep` so it can be woken immediately:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # renew lock
        ...

# To stop:
stop_event.set()
```

### 8. Retry Logic Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In an async web framework (FastAPI, Starlette), this blocks the event loop thread. Even in a threaded framework, holding a thread during backoff under flash sale load is wasteful. This should be documented as a known limitation, or the service should be made async-native.

## Test Coverage Gaps

### 9. Heartbeat Is Mocked Away Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — the most novel and risk-bearing component of this PR — has zero test coverage. There is no test verifying that the heartbeat actually extends the TTL, that it stops when the lock is released, or that it correctly skips renewal when the token no longer matches.

### 10. No Test for Partial DB Failure

There is no test covering `update_stock` succeeding followed by `create_order` raising an exception. Given the non-transactional concern in issue #6, this is a required test case.

### 11. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test patches the mock *after* fixture construction and relies on the heartbeat being mocked out. It does not test the actual race condition scenario and gives false confidence. The token comparison in the real code uses a dynamically generated UUID, so this test only works because `_start_heartbeat` is mocked and the token never reaches Redis in a meaningful way.
