Decision: Request Changes

Summary: The heartbeat renewal pattern is architecturally unsound for a distributed lock and introduces a class of safety violation that the PR explicitly acknowledges but misframes as solved. There are also concurrency correctness issues in the heartbeat implementation itself, input validation gaps, and non-thread-safe instance state that will cause silent failures under the exact concurrent load this service is designed to handle. The tests mock away the heartbeat entirely, meaning the most critical and dangerous code path has zero meaningful test coverage.

## Detailed Review

### 🔴 Critical — Architectural Flaw: Heartbeat Renewal Defeats the Purpose of Lock TTL

This is the most serious issue and it is a well-documented distributed systems failure mode.

The PR summary states:

> *"The 30-second LOCK_TTL combined with 10-second heartbeat renewal ensures the lock remains valid throughout any realistic process pause, preventing stale writes when the process resumes."*

This reasoning is **backwards**. The TTL on a distributed lock is not a convenience timeout — it is the **safety boundary**. It exists precisely to handle the scenario where a process pauses (GC, network partition, OS scheduling) and *cannot* be trusted to know its own state. The canonical failure sequence is:

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   Process A enters stop-the-world GC pause
        — ALL threads suspended, including heartbeat thread
T=15  Redis TTL would have expired (safety kicks in)
        — heartbeat is also paused, so lock expires correctly
T=25  Process A resumes from GC
        — heartbeat thread wakes, checks token, token still matches
        — heartbeat extends the lock for another 30 seconds
        — Process A believes it still holds the lock
T=26  Process B acquired the lock at T=15 (correct behavior)
        — Process B is now mid-transaction
T=27  Process A resumes its write, also believing it holds the lock
        — BOTH processes are writing concurrently
```

The heartbeat does not prevent this. It makes it worse. After a sufficiently long pause, the process wakes up, the heartbeat re-extends a lock that may have already been acquired by another process, and two writers proceed simultaneously. This is a **phantom lock** scenario. The token check in the heartbeat (`current == token`) only confirms the key still holds the original value — it cannot detect that the lock expired and was re-acquired with the same token by coincidence, and more importantly it cannot detect the window between expiry and re-acquisition.

The correct approach for long-running operations under a distributed lock is one of:

1. **Do not hold the lock across slow operations.** Restructure the critical section to be as short as possible. Read stock and write the reservation in a single atomic operation (Redis transaction or Lua script).
2. **Use optimistic locking / compare-and-swap at the database layer** so the lock is not needed to guarantee correctness.
3. **Accept that the TTL is a hard deadline** and design the operation to complete within it, with the TTL set conservatively short.

The heartbeat pattern is used in systems like etcd-backed leader election where the tradeoffs are explicitly understood and the lease model is designed for it. It is not appropriate here.

### 🔴 Critical — Non-Thread-Safe Instance State

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped locals. If two threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the entire use case), they will share and clobber each other's `_heartbeat_active` flag.

**Failure scenario:**

```
Thread A: _heartbeat_active = True   (acquires lock for item_001)
Thread B: _heartbeat_active = True   (acquires lock for item_002)
Thread A: _stop_heartbeat() → _heartbeat_active = False
Thread B: heartbeat thread checks _heartbeat_active → False → stops
Thread B: lock for item_002 is no longer being renewed
Thread B: lock expires mid-transaction
```

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. This could be a dataclass or a simple object passed into the heartbeat closure.

### 🔴 Critical — Race Condition in Lock Release (Check-Then-Act)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic TOCTOU (time-of-check/time-of-use) race. Between the `GET` and the `DELETE`, another process can acquire the lock. This is a well-known problem with Redis distributed locks and the standard solution is a **Lua script** that makes the check and delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The same race exists in the heartbeat's `GET` + `EXPIRE` sequence.

### 🔴 Critical — No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation whatsoever. Pydantic is only used on the *output* model `ReservationResult`.

Missing validations:

- `item_id` is used directly in a Redis key: `f"inventory:lock:{item_id}"`. A caller passing `item_id = "foo:bar:baz"` or `item_id = ""` or a path-traversal-style string could manipulate the key namespace.
- `quantity` has no lower bound check. `quantity=0` or `quantity=-5` would pass the stock check and silently write a nonsensical order.
- No maximum quantity guard against bulk reservation abuse.

Minimum fix:

```python
if not item_id or not isinstance(item_id, str) or ":" in item_id:
    raise ValueError(f"Invalid item_id: {item_id!r}")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError(f"quantity must be a positive integer, got {quantity!r}")
```

### 🟡 Moderate — Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The lock provides mutual exclusion but does not provide atomicity across the DB operations. These must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Moderate — Tests Mock Away the Only Dangerous Code

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← entire heartbeat mocked out
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat thread is the most complex and most dangerous component in this PR. Every test that uses the `service` fixture is testing a version of the code that does not run the heartbeat. There are no tests for:

- Heartbeat extending a lock it still owns
- Heartbeat correctly stopping when `_heartbeat_active` is set to False
- Concurrent calls to `reserve_item` on the same instance
- The TOCTOU behavior in lock release
- Any behavior involving actual Redis TTL semantics

---

### 🟡 Moderate — Bare `RuntimeError` for Lock Failure

```python
raise RuntimeError(
    f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
)
```

`RuntimeError` is too broad and will catch unrelated runtime errors in the `except` block. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

This also prevents accidentally swallowing legitimate programming errors that happen to raise `RuntimeError` inside the lock acquisition path.

### 🟢 Positive Observations

- The `nx=True, ex=self.LOCK_TTL` pattern for atomic lock acquisition is correct.
- The token-based ownership model is the right approach conceptually (implementation has the TOCTOU flaw noted above).
- Structured logging with key/value pairs is clean and consistent.
- Exponential backoff on retry is appropriate.
- `ReservationResult` as a typed return value is a good interface contract.
