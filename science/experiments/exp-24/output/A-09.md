Decision: Request Changes

Summary: The heartbeat renewal mechanism contains a critical TOCTOU race condition — the `GET` and `EXPIRE` operations are non-atomic, meaning the lock can expire and be acquired by another process between the two calls, after which this instance will extend a lock it no longer owns, directly enabling the stale-write scenario the PR claims to prevent. Beyond that flaw, the service carries two additional architectural problems: the `InventoryService` instance is not safe for concurrent use because `_heartbeat_active` and `_heartbeat_thread` are instance-level fields rather than per-invocation state, and the database `update_stock`/`create_order` pair is not wrapped in a transaction, leaving stock and order records inconsistently diverged on any mid-sequence failure. The test suite mocks out the heartbeat entirely, so none of these failure modes are exercised.

---

### Detailed Findings

#### 1. Non-Atomic Heartbeat Renewal (Critical — Security / Correctness)

```python
# _start_heartbeat — as written
current = self.redis.get(lock_key)          # Step A: read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step B: extend TTL
```

Between Step A and Step B, the TTL can reach zero, Redis can delete the key, and a competing process can call `SET … NX` and acquire the lock. This instance then calls `EXPIRE` on a key it no longer owns, silently extending the *new owner's* lock identity — or, worse, extending a key that was re-created with a different token, giving two processes simultaneous belief that they hold the lock. This is the canonical Redlock check-then-act hazard.

**Required fix:** Replace the GET+EXPIRE pair with a single atomic Lua script:

```lua
-- extend.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def __init__(self, redis_client, db):
    ...
    self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# Inside _beat():
extended = self._extend_script(
    keys=[lock_key],
    args=[token, self.LOCK_TTL]
)
if not extended:
    logger.warning("Heartbeat: lock lost, stopping renewal key=%s", lock_key)
    self._heartbeat_active = False
```

The same non-atomic flaw exists in the lock release path in `_acquire_lock` and must receive the same Lua treatment:

```lua
-- release.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

#### 2. Shared Mutable State Makes the Service Unsafe for Concurrent Callers (High)

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False        # shared across all calls
        self._heartbeat_thread = None         # overwritten on every call
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the normal case in any WSGI/ASGI worker — the second call overwrites `_heartbeat_thread` and calling `_stop_heartbeat` sets `_heartbeat_active = False` for *both* ongoing locks. One lock will silently stop renewing.

**Required fix:** Move all per-invocation state into a local dataclass or a dedicated lock-context object rather than instance fields.

```python
from dataclasses import dataclass, field

@dataclass
class _LockContext:
    key: str
    token: str
    active: bool = True
    thread: Optional[threading.Thread] = None
```

Pass this context object into `_start_heartbeat` and `_stop_heartbeat` instead of touching `self`.

#### 3. Non-Transactional Database Writes (High)

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. The item is effectively lost. The distributed lock provides no protection here because the inconsistency is in the database layer, not in Redis.

**Required fix:** Wrap both writes in a single database transaction. The exact API depends on the ORM/driver in use, but the pattern must be:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

#### 4. Input Validation Is Incomplete (Medium)

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation at the call site. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string. The Pydantic model is only used for the *output* type.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

#### 5. Test Coverage Does Not Exercise the Heartbeat (Medium)

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The most dangerous code path — the heartbeat renewal loop — is completely bypassed in every test. The critical race condition in Finding 1 would not be caught by this suite even after the fix is applied.

**Required additions:**
- A test that verifies the Lua script is invoked with the correct key and token.
- A test that verifies `_heartbeat_active` is set to `False` when the script returns `0` (lock lost).
- An integration-level test using `fakeredis` to exercise the full lock lifecycle without mocking Redis internals.
