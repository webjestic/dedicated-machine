Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic NX lock acquisition, token-verified release, heartbeat renewal, and structured logging are all sound patterns. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a government-grade system. The most critical flaw is a race condition in the heartbeat that can cause lock theft, but there are also concurrency safety problems with the service being used as a shared instance.

---

## Critical Issues

### 1. Heartbeat Race Condition — Lock Theft Vector

The heartbeat performs a non-atomic check-then-extend:

```python
current = self.redis.get(lock_key)       # step 1
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2
```

Between step 1 and step 2, the lock can expire and be acquired by another process. You then extend a lock you no longer own, effectively stealing it from a legitimate holder. This is the exact failure mode distributed locks are designed to prevent.

**Fix:** Use a Lua script to make the check-and-extend atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same atomic pattern must also be applied to the lock release in `_acquire_lock`. The current `get` + `delete` sequence has the identical race.

---

### 2. Instance-Level Heartbeat State — Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is a shared singleton (the standard deployment pattern), two concurrent `reserve_item` calls will clobber each other's heartbeat state. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat thread.

**Fix:** Heartbeat state must be scoped to the lock context, not the instance. Pass state through a local closure or a dedicated lock object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure, not self
```

---

### 3. Input Validation Is Declared But Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation whatsoever. There is nothing preventing:

- `item_id = "../../etc/passwd"` or an empty string
- `quantity = 0`, `-1`, or `999999`

Pydantic is imported and `ReservationResult` is defined, but it is only used as an output model. A request model is completely absent.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Significant Issues

### 4. `db.update_stock` and `db.create_order` Are Not Atomic

The Redis lock protects concurrent access, but the two database writes are not wrapped in a transaction:

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # fails
```

A crash between these two lines produces phantom stock decrements with no corresponding order — inventory is silently lost. This is a data integrity failure that will be very difficult to detect or recover from in production.

**Fix:** These operations must execute within a single database transaction. The `db` abstraction should expose a transactional context manager, or both writes should be composed into a single atomic call.

---

### 5. Heartbeat Can Extend a Lock After It Has Been Intentionally Released

`_stop_heartbeat()` sets `_heartbeat_active = False`, but the thread may already be sleeping in `time.sleep(HEARTBEAT_INTERVAL)`. After the lock is deleted in `finally`, the thread wakes up, reads the flag, and if there is a scheduling delay between the flag check and the `expire` call, it can re-extend a lock that was just released or is now owned by someone else.

**Fix:** Use `threading.Event` with a timeout instead of `time.sleep` so the thread wakes immediately on stop:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # atomic extend via Lua
        ...

# to stop:
stop_event.set()
thread.join(timeout=2)  # ensure thread has exited before releasing lock
```

The `join()` before lock deletion is critical — it eliminates the window where a waking heartbeat thread extends a lock that is being released.

## Minor Issues

### 6. Test Coverage Gaps

- `test_lock_not_released_if_expired` mocks `_start_heartbeat` and `_stop_heartbeat`, which means it does not test the actual heartbeat behavior at all. The most dangerous code path has no real test coverage.
- There is no test for the non-atomic DB write failure scenario (item 4 above).
- There is no test for concurrent calls on a shared instance (item 2 above).
- The lock contention test (`test_reserve_lock_contention`) does not verify that retry attempts were actually made — it only checks the final result.

### 7. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

These should be configurable at construction time. Flash sale load profiles vary significantly, and hardcoded values make tuning require a code change and redeployment.

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 10):
```

### 8. Daemon Thread Lifecycle Is Not Documented or Bounded

Daemon threads are silently killed on interpreter shutdown. This is acceptable here, but it should be explicitly documented. More importantly, there is no upper bound on how long the heartbeat thread can hold a lock open. A deadlocked or hung DB operation will hold the lock indefinitely. Consider a maximum lock hold time as a safety ceiling.
