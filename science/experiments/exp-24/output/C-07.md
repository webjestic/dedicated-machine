Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this handles real flash sale load in a government contracting context where auditability and correctness are non-negotiable.

---

## Critical Issues

### 1. Race Condition in Heartbeat Token Check (TOCTOU)

The heartbeat performs a non-atomic GET + EXPIRE sequence:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could be stolen here
```

Between the `GET` and `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be an atomic operation.

**Fix:** Use a Lua script to atomically verify ownership and extend TTL:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._heartbeat_sha = self.redis.script_load(HEARTBEAT_SCRIPT)

# In heartbeat:
self.redis.evalsha(self._heartbeat_sha, 1, lock_key, token, self.LOCK_TTL)
```

The same issue exists in the lock release — the GET + DELETE is also non-atomic. Use the canonical Lua pattern there too:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 2. Instance-Level Heartbeat State Breaks Concurrency

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — one call's `_stop_heartbeat()` will cancel the other's heartbeat, potentially letting the other's lock expire mid-operation. This is a silent data integrity failure.

**Fix:** Make heartbeat state local to each lock acquisition, not shared on the instance. Pass state via a mutable container or a small dataclass:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = {"running": False}
    # pass heartbeat_active into _start_heartbeat, stop via heartbeat_active["running"] = False
```

### 3. Input Validation Is Claimed But Not Enforced

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used for the *output* model. There is no guard against:

- Empty string `item_id`
- Negative or zero `quantity`
- Excessively long `item_id` strings (potential key injection into Redis)

**Fix:** Add an input model and validate at the boundary:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

The `item_id` pattern constraint is especially important — without it, a caller could inject arbitrary strings into Redis key names (`inventory:lock:../../admin` style abuse).

---

### 4. No Database Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. This is a silent inventory leak with no compensating mechanism.

**Fix:** Wrap in a transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

Or use a single atomic stored procedure/upsert depending on the DB layer.

## Moderate Issues

### 5. Heartbeat Interval Is Dangerously Close to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10s, but the thread sleeps *first* before checking. If a DB operation takes 25 seconds to start and the heartbeat fires at t=10 and t=20, the lock expires at t=30 before the next beat at t=30. There is a race at the boundary.

**Fix:** Either reduce the interval to `LOCK_TTL / 4` (7s here), or fire the first heartbeat proactively before sleeping, or sleep in shorter increments and check elapsed time.

---

### 6. Test Coverage Gaps

- No test for the TOCTOU heartbeat race (acknowledged above as a bug)
- No test for concurrent calls on the same instance (the shared state bug)
- No test for DB transaction failure between `update_stock` and `create_order`
- `test_lock_not_released_if_expired` mocks `_start_heartbeat` and `_stop_heartbeat`, meaning the test never exercises the actual lock release path with a real token mismatch — it only tests the `reserve_item` return value, not the Redis interaction

---

### 7. Daemon Thread Leak on Service Shutdown

Heartbeat threads are `daemon=True`, which means they die with the process — acceptable. However, if `_stop_heartbeat` is never called (e.g., an unhandled exception type other than `RuntimeError` propagates out of `reserve_item`), the thread runs until the lock TTL expires naturally. This is bounded but worth making explicit.

The `finally` block in `_acquire_lock` does call `_stop_heartbeat`, so this is mostly covered — but exceptions raised *before* `yield` (e.g., during lock acquisition) would still start a heartbeat that never gets stopped if the logic order were ever changed. Consider restructuring so the heartbeat only starts inside the `try` block after `yield` is reached, or assert the ordering in a comment.
