Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, exponential backoff, and structured logging are all present and correctly motivated. However, there are several correctness, safety, and architectural issues that must be addressed before this is suitable for a production inventory system, particularly one under flash-sale concurrency pressure.

---

## Critical Issues

### 1. Race Condition in Heartbeat — Non-Atomic Check-and-Extend

```python
# CURRENT — two separate round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be a single atomic operation using a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In heartbeat:
self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same issue applies to lock release — the current `GET` + conditional `DELETE` is also non-atomic and must use the canonical Lua pattern.

---

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the entire point of this service), two concurrent calls will stomp each other's `_heartbeat_active` flag. Stopping one reservation's heartbeat will kill the other's. The heartbeat state must be scoped to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # extend lock ...

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # release lock ...
```

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string, and the code will happily acquire a distributed lock and attempt a DB write. At minimum:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

Pydantic should be used for a request model here, not just the response.

---

### 4. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. This is a data integrity failure. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock protects against concurrent Redis-level races, but it does not substitute for database-level atomicity.

## Moderate Issues

### 5. Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread could still be mid-execution (e.g., in the middle of `redis.eval`) when the lock release proceeds. Use `thread.join()` with a timeout after clearing the flag to ensure the heartbeat has fully stopped before the lock is deleted.

---

### 6. Broad Exception Handling Silently Swallows DB Errors

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught, bypassing the `finally` block's lock release... actually, `finally` will still run, but the caller receives an unhandled exception rather than a structured `ReservationResult`. Add a broader catch inside the lock context for DB-layer errors and return a structured failure.

---

### 7. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

Under flash-sale load, many threads will pile up here. For an async-capable service this should use `asyncio.sleep`, or the service should be redesigned with an async interface. For a synchronous service, this is acceptable but should be documented explicitly as a threading concern, and the retry count/backoff ceiling should be configurable.

---

## Test Issues

### 8. Tests Mock Away the Exact Behavior Being Tested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and failure-prone part of this implementation. Mocking it out entirely means the tests provide no coverage for the race conditions identified above. At least one integration-style test should exercise the real heartbeat path against a mock Redis that validates atomic Lua script calls.

### 9. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test verifies that `delete` is not called when the token doesn't match — which is correct behavior — but it does so while the reservation *succeeds* (`result.success` is not asserted). A mismatched token during release means the lock was stolen mid-operation, which is a serious correctness failure that should likely surface as an error, not a silent success. The expected behavior here needs a design decision and explicit assertion.
