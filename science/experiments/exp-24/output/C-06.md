Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real flash sale load in a government contracting context where auditability and correctness are non-negotiable.

---

## Critical Issues

### 1. Race Condition in Heartbeat — Non-Atomic Token Check + Expire

```python
# CURRENT — two separate commands, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be a single atomic operation via a Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat
extended = self._extend_lock_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same atomic-script pattern already used by `redis-py`'s own `Lock` implementation should be followed here.

---

### 2. Same Race Condition in Lock Release

```python
# CURRENT — also non-atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Use the canonical Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

This is a well-known pattern. Skipping it is a correctness bug, not a style preference.

---

### 3. `InventoryService` Is Not Safe for Concurrent Use — Instance-Level Heartbeat State

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), one call's `_stop_heartbeat()` will kill the other's heartbeat thread. This is a silent correctness failure under the exact load this service is designed for.

**Fix:** Move heartbeat state into the lock context itself, scoped per-call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... atomic release ...
```

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` could be an empty string, a path traversal fragment, or absurdly long. `quantity` could be zero or negative.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not isinstance(item_id, str):
        return ReservationResult(success=False, message="Invalid item_id")
    if not item_id.replace("-", "").replace("_", "").isalnum():
        return ReservationResult(success=False, message="Invalid item_id format")
    if quantity < 1 or quantity > MAX_RESERVATION_QUANTITY:
        return ReservationResult(success=False, message="Invalid quantity")
```

Pydantic is imported but only used for the response model. Either validate inputs with a Pydantic model or add explicit guards. The checklist box should not be checked as-is.

## Significant Issues

### 5. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. This is a partial write. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The Redis lock protects against concurrent reservation but does not protect against partial DB writes on failure.

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread could still be mid-`expire` call when the lock is released. Add a join with a bounded timeout:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread and self._heartbeat_thread.is_alive():
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Unhandled Redis Exceptions Propagate as Untyped Errors

Only `RuntimeError` from lock acquisition is caught. A `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` during `get_stock`, `update_stock`, or `create_order` will propagate uncaught and likely surface as a 500 with an internal stack trace. These should be caught and returned as structured `ReservationResult` failures with appropriate logging.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test mocks the heartbeat, meaning the actual heartbeat logic — which contains the race condition identified in Issue #1 — has zero test coverage. There should be at least one integration-style test that exercises the real heartbeat path.

### 9. DB Transaction Failure Is Not Tested

There is no test for the case where `create_order` raises after `update_stock` succeeds. Once Issue #5 is fixed, this scenario must be covered.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test verifies that `delete` is not called when the token doesn't match, but it does not verify that the reservation itself failed or that the order was not created. A stolen lock mid-operation is a data integrity scenario, not just a cleanup scenario. The test should assert `result.success is False` or that `db.create_order` was not called in that state.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is reasonable, but both should be configurable via constructor parameters or environment config rather than class-level constants. Flash sale profiles vary.
- Consider using `redis-py`'s built-in `redis.lock.Lock` which already implements atomic extend and release correctly. Rolling a custom distributed lock is high-risk and requires the atomic scripting above to be correct.
- The `daemon=True` on the heartbeat thread is appropriate — it won't block process shutdown — but document this explicitly since daemon threads can be surprising to maintainers.
