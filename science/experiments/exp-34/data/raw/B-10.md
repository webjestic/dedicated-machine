**Findings**

---

**1. Silent Heartbeat Exit — Main Thread Has No Visibility Into Lock Loss**

**Failure path:**

Process A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. A GC pause or slow external call in `_dispatch()` stalls the process for >60s. The lock expires naturally. Process B runs `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job. Process A's heartbeat fires at the next `RENEWAL_INTERVAL` boundary. `RENEW_CLAIM_SCRIPT` executes `GET job:claim:{job_id}` → returns `T2`, which does not match `T1` → script returns `0`. The `_renew()` function hits `if result == 0: return` and the thread exits silently.

The main thread is inside `with self._claim_job(req.job_id):` — it yielded at `yield token` and is now somewhere in:

```python
existing = self.db.get_execution(req.job_id)   # may already be past this
result = self._dispatch(req.job_type, req.payload)
with self.db.transaction():
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

Nothing propagates the heartbeat thread's exit to the main thread. `stop_event` is only written by the main thread (in `finally:`); it is never set by `_renew()`. The main thread has no `.is_alive()` check, no shared flag, no exception channel. It continues executing, reaches `self.db.transaction()`, and commits. Process B does the same. The job executes twice and both writes land.

**Required fix — two independent layers, both necessary:**

*Layer 1 (detection):* Have `_renew()` set a shared `threading.Event` (e.g., `lock_lost_event`) on `result == 0` instead of silently returning. The main thread must poll this event at each major phase boundary — after `_dispatch()` returns, before entering `db.transaction()`. If the event is set, abort and raise. This closes the silent-exit gap and is straightforward to implement. **However, this alone is not sufficient** — see Finding 2.

---

**2. No Fencing Token at the Database Write Boundary — Zombie Write Is Not Prevented**

**Failure path (continuation of Finding 1, and independent of whether threading.Event is added):**

Even with `lock_lost_event` polling, there is a race window: the main thread checks the event, sees it unset (lock still held), proceeds into `db.transaction()`, and then a GC pause occurs *inside* the transaction — after the check, before the commit. The lock expires during that pause. The event is set by the heartbeat, but the main thread is already past the check point and inside the commit. The zombie write lands.

More fundamentally: the database write carries no proof that the lock was still held at commit time. `record_execution` and `mark_job_complete` are unconditional writes. The database has no way to reject a write from a worker whose lock has already been superseded by another worker.

**Required fix:**

The database write boundary must enforce mutual exclusion independently of the Redis lock. The correct mechanism is **optimistic concurrency control / a fencing token at the DB layer**:

- When Process B acquires the lock, it receives or generates a monotonically increasing **fencing token** (e.g., a Redis `INCR` on a per-job sequence key at claim time, or a lock version stored alongside the claim). This token is passed through to the DB write.
- `mark_job_complete` is rewritten as a conditional update: `UPDATE jobs SET status='complete', completed_by=:execution_id WHERE job_id=:job_id AND status='pending' AND lock_version < :fencing_token` (exact predicate depends on schema).
- If Process A's write arrives after Process B has already written with a higher token, the `WHERE` clause fails, the update affects 0 rows, and the application detects this and aborts.

Alternatively, `mark_job_complete` can use a **compare-and-swap** on job status: `UPDATE jobs SET status='complete' WHERE job_id=:job_id AND status='pending'` with the application checking `rowcount == 1`. This is weaker than a fencing token (it doesn't order by lock generation) but it does prevent double-completion if `record_execution` is also guarded by a unique constraint on `job_id`. A unique constraint on `(job_id)` in the executions table would cause the second `record_execution` to raise an integrity error, which is a viable backstop — but only if the DB enforces it and the application surfaces the error rather than swallowing it.

The key architectural point: **the Redis lock is a coordination hint, not a write gate**. Any process that can reach `db.transaction()` can write, regardless of whether its lock is still valid. The only layer that can enforce "exactly one write succeeds" is the database itself, via constraints or conditional updates. Threading signals operate in-process and cannot protect against the window between the signal check and the commit.

---

**3. Idempotency Check Is Not Atomic With the Write — TOCTOU on `get_execution`**

**Failure path:**

Both Process A and Process B pass the Redis claim sequentially (A holds it, B waits, A's lock expires, B acquires). But consider two workers that both pass `get_execution` returning `None` before either has written — possible if the DB read is dirty or if the check-then-act window is wide. More concretely: `get_execution` reads, returns `None`, then `_dispatch()` runs (potentially slow), then the write happens. Another worker can pass the same `get_execution` check during that window. The idempotency check is not inside the same transaction as the write, so it provides no atomicity guarantee.

**Required fix:**

The idempotency check must be inside `db.transaction()`, or the write itself must be the idempotency enforcement via a unique constraint on `job_id` in the executions table (with the application catching the integrity error). Reading outside the transaction and writing inside it is a TOCTOU gap.

---

**4. Test Suite Does Not Cover the Failure Mode**

The tests mock `register_script` to return a mock that always returns `1`. There is no test for `result == 0` from the renewal script, no test that verifies main thread behavior when the heartbeat exits, and no test for the DB write boundary under lock loss. The test for `test_claim_release_uses_lua_script` verifies that `redis.delete` is not called (correct) but does not verify that a stale-lock scenario is detected or rejected. The test suite would pass with the zombie-write vulnerability fully present.

**Required fix:**

Add a test that simulates `result == 0` from the renewal script mid-execution and asserts that the main thread does not proceed to `db.record_execution`. Add a test that simulates the DB write racing with lock loss and asserts the write is rejected.

---

**Verdict:** NOT APPROVED

The primary blocker is Finding 2: there is no fencing token or conditional write at the database boundary. A process that loses its lock can still commit. Finding 1 (silent heartbeat exit) is the trigger mechanism; Finding 2 is why the trigger produces a correctness violation that cannot be stopped in-process. Both must be fixed. Finding 3 is a secondary correctness issue. Finding 4 means the gap would survive CI. This does not ship.
