**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Possible**

**Issue name:** Undetected lock loss / zombie write

**Failure path, step by step:**

1. Process A acquires `job:claim:{job_id}` with token `T_A`, TTL = 60 s. Heartbeat thread starts.
2. Process A enters `_dispatch()`. A GC pause, slow external call, or network stall holds the process for > 60 s.
3. The lock TTL expires in Redis. The key is gone.
4. Process B calls `SET job:claim:{job_id} T_B NX EX 60` — succeeds. Process B begins executing the same job.
5. Process A's heartbeat fires (RENEWAL_INTERVAL = 15 s). `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T_B` ≠ `T_A`. Script returns `0`. The `_renew` function hits `if result == 0: return` and exits. **The thread dies silently.**
6. `stop_event` is never set by the heartbeat thread. The main thread in Process A has no join, no callback, no shared flag checked between `_dispatch()` and `self.db.transaction()`. It does not know the heartbeat exited.
7. Process A resumes after the stall. `_dispatch()` returns a result. Process A enters `with self.db.transaction()` and calls `self.db.record_execution(...)` and `self.db.mark_job_complete(...)`.
8. Process B does the same.
9. Both processes write a completion record. The job executes twice. For a payment charge, alert fire, or outbound notification, this is a correctness violation.

**Why the existing idempotency check does not save you:**

The `get_execution` check at the top of the `with self._claim_job(...)` block runs *before* either process writes. At the moment Process A checks, no execution record exists. At the moment Process B checks (after acquiring the lock while A is stalled), no execution record exists yet either. Both pass the check and proceed to `_dispatch`.

**Why `threading.Event` signaling is insufficient as a fix:**

Adding `stop_event.set()` in `_renew` on `result == 0`, and checking `stop_event.is_set()` in the main thread before the DB write, is a detection mechanism — not a prevention mechanism. It closes the window under normal timing but does not close it architecturally. The main thread can still read `is_set() == False` in the nanoseconds between the check and the DB write while the heartbeat thread is in the process of setting it. More importantly, it relies on in-process signaling to enforce a distributed invariant. Any scenario that kills the thread without setting the event (SIGKILL, interpreter crash, OS scheduling) bypasses it entirely. A signal is not a fence.

**Required fix — fencing token at the database write boundary:**

The architectural fix must live at the layer where the duplicate write would occur: the database transaction.

Concretely:

- When Process A acquires the lock, it receives `token = T_A`. This token is the fencing token.
- Before writing, Process A must verify that `T_A` is still the current lock holder. This check must be atomic with the write — either as a single database transaction that reads the current token from a `job_claims` table and writes only if it matches, or as an optimistic concurrency check using a version/sequence column on the job row.
- The simplest correct form: store `(job_id, token)` in a `job_claims` table when the lock is acquired (written by the Redis SET path or by a DB insert). At write time, execute:

```sql
INSERT INTO job_executions (...)
WHERE NOT EXISTS (
    SELECT 1 FROM job_executions WHERE job_id = $job_id
)
AND EXISTS (
    SELECT 1 FROM job_claims WHERE job_id = $job_id AND token = $token
)
```

or use a `SELECT ... FOR UPDATE` on the job row with an expected-token column, inside the same transaction. If the token no longer matches (because Process B has updated it), the write fails with a constraint violation or zero rows affected, and Process A aborts.

- Alternatively, use a monotonic sequence number as the fencing token (Redis `INCR` at lock acquisition time). Pass it through to the DB write as an optimistic version check: `UPDATE jobs SET status='complete', lock_version=$seq WHERE job_id=$id AND lock_version=$seq - 1`. If Process B has already written with a higher sequence, Process A's update affects 0 rows and must abort.

The key property: the DB write is conditional on the token/version being valid *at write time*, evaluated atomically inside the DB transaction. No amount of in-process signaling achieves this — only the database can make the write and the validity check atomic.

**Required fix — heartbeat exit must set a shared flag (defense in depth, not the primary fix):**

The `_renew` function should set a shared `threading.Event` (e.g., `lock_lost_event`) when `result == 0`, and the main thread should check it before entering the DB transaction. This is defense-in-depth — it catches the common case and avoids unnecessary DB round-trips — but it is explicitly not the primary correctness guarantee. Document it as such.

```python
# In _renew:
if result == 0:
    lock_lost_event.set()
    return

# In execute, before db.transaction():
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting write")
```

This check + the DB-level fencing token together form a correct defense: the event catches most cases cheaply; the DB fence catches the race that the event misses.

**Finding 2: Test Suite Does Not Cover the Failure Mode**

**Issue name:** No test for lock-loss-during-execution / zombie write path

**Failure path:** The test `test_execute_success` mocks `register_script` to return a script mock with `return_value=1` unconditionally. There is no test where the renew script returns `0` mid-execution and the test asserts that `record_execution` is *not* called. The heartbeat behavior under lock loss is entirely untested. A reviewer cannot confirm from the test suite that lock loss is handled correctly, because the test suite does not exercise that path.

**Required fix:** Add a test that simulates `result == 0` from the renew script (lock lost mid-execution) and asserts:
- `db.record_execution` is not called
- `db.mark_job_complete` is not called
- The result indicates failure or the call raises, depending on the chosen abort mechanism

This test will fail until Finding 1 is fixed, which is the correct behavior — the test encodes the invariant.

**Finding 3: Release Script Runs After Lock May Have Expired and Been Re-Acquired**

**Issue name:** Spurious release attempt on expired token (minor — Lua script is safe, but log is misleading)

**Failure path:** In the `finally` block of `_claim_job`, after `stop_event.set()` and `t.join()`, `_release_script` runs unconditionally. If the lock expired and Process B holds it, the Lua script correctly returns `0` (token mismatch) and does not delete Process B's lock. The script itself is safe. However, the `logger.info("Claim released job_id=%s")` fires unconditionally after the script call regardless of the return value. An operator reading logs will see "Claim released" even when the release was a no-op against an already-expired lock, making post-incident log analysis misleading.

**Required fix:** Check the return value of `_release_script` and log distinctly:

```python
released = self._release_script(keys=[claim_key], args=[token])
if released:
    logger.info("Claim released job_id=%s", job_id)
else:
    logger.warning("Claim release no-op — token mismatch, lock had already expired job_id=%s", job_id)
```

This is a logging/observability issue, not a correctness issue given the Lua guard, but in a post-incident investigation the distinction between "we released cleanly" and "our lock had already expired when we tried to release" is operationally significant.

---

**Verdict: NOT APPROVED**

Finding 1 is a correctness violation that will produce duplicate job execution under GC pause or slow external call conditions in a multi-worker deployment. The heartbeat thread exits silently on lock loss; the main thread has no visibility into this; the database write boundary has no fencing token or optimistic concurrency check to reject a write from a process that no longer holds the lock. Both processes complete the job. The fix must be implemented at the database write boundary before this ships.
