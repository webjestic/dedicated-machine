## Findings

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Primary Correctness Risk**

**Issue name:** Zombie write via undetected lock loss

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token T, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters a GC pause, a slow external call inside `_dispatch`, or any other stall that exceeds 60 seconds without a renewal firing in time. The key expires in Redis.
3. Worker B calls `SET job:claim:{job_id} <token_B> NX EX 60` — it succeeds. Worker B now holds the lock and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET` returns `<token_B>`, which does not match `T`. The script returns `0`.
5. `_renew()` sees `result == 0` and executes `return`. The heartbeat thread exits silently. No exception is raised, no event is set, nothing is written to a shared variable the main thread observes.
6. Worker A's main thread is still inside `with self._claim_job(...)`. It has no mechanism to detect that the heartbeat exited. It continues: checks `db.get_execution()` (returns None, because B hasn't committed yet), calls `_dispatch()`, enters `db.transaction()`, writes `record_execution` and `mark_job_complete`.
7. Worker B does the same. Both workers complete. The job executes twice.

This is the classical zombie-write pattern: a process that has lost its lock continues writing as if it still holds it, because the only thread that knew about the loss had no channel back to the execution thread.

**Why the existing structure cannot catch this:**

The `stop_event` is a one-way signal — the main thread uses it to tell the heartbeat to stop. There is no reverse channel. The heartbeat thread's `return` on `result == 0` is invisible to the main thread. The `finally` block in `_claim_job` runs `stop_event.set()` and `t.join()` — but by the time the job is done, the damage is already done. Joining a thread that already exited is a no-op; it does not surface the reason the thread exited.

**Required fix — two layers, both necessary:**

*Layer 1 — Immediate detection (necessary but not sufficient):*

Add a reverse signal. Replace the bare `return` in `_renew` with:

```python
if result == 0:
    lost_event.set()   # threading.Event shared with the main thread
    return
```

In `_claim_job`, poll `lost_event` at checkpoints, or have the `yield` block wrap execution in a way that checks the event before proceeding past `_dispatch`. A clean approach:

```python
# After yield returns (i.e., after the with-block body completes):
if lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting commit")
```

This prevents the `db.transaction()` commit from proceeding after confirmed lock loss. However, this check is only at the boundary of the `with` block body — it does not interrupt `_dispatch` mid-execution.

*Layer 2 — Architectural fix (required for correctness guarantee):*

Threading events are a mitigation, not a solution. The fundamental problem is that the database write has no proof of lock ownership at commit time. The fix is a **fencing token** (monotonic counter) or **optimistic concurrency control**:

- When acquiring the lock, use a Redis-side monotonic counter (`INCR job:fence:{job_id}`) and record the fence value alongside the token.
- Pass the fence token into `record_execution` and `mark_job_complete`.
- The database write includes a `WHERE fence_token = <expected>` condition (or equivalent conditional insert). If Worker A's fence token has been superseded by Worker B's, Worker A's write fails at the database layer — regardless of what the Redis lock state is, regardless of thread timing.

This is the only boundary that provides a hard correctness guarantee: the database commit itself rejects stale writers. Redis lock state is advisory; the database write is the authoritative gate.

Without Layer 2, any window between the last heartbeat check and the database commit is a potential zombie-write window. The threading.Event approach narrows the window but does not close it.

**Finding 2: Prior Review's Atomicity Concern — Dismissed**

**Claim under review:** `RENEW_CLAIM_SCRIPT` executes `GET` then `EXPIRE` as sequential Lua operations; under high Redis connection saturation these may not execute atomically.

**Assessment: This is not a real failure path.**

Redis executes Lua scripts atomically by design. From the Redis documentation and implementation: a Lua script is executed as a single atomic unit — no other Redis command from any client can interleave between the `GET` and the `EXPIRE` within the same script invocation. Redis is single-threaded with respect to command execution; the event loop does not yield between Lua opcodes. Connection saturation affects when a script is admitted to the execution queue, not what happens once it begins executing. Two sequential `redis.call()` invocations inside a single `EVAL` are atomic with respect to all other Redis clients.

The concern conflates network-layer queuing (connections waiting to send commands) with execution-layer interleaving (commands from different clients running between each other). The former is real; the latter cannot happen inside a Lua script on a standard Redis instance.

The prior review's atomicity concern is a false positive. The Lua script is correct. The real atomicity risk in this codebase is not inside the renewal script — it is the gap between the renewal script returning 0 and the main thread's execution continuing, which is Finding 1.

**Finding 3: Release After Potential Lock Loss Releases Another Worker's Lock**

**Issue name:** Spurious lock release on foreign token

**Failure path:**

This is partially mitigated by the token-verified release script — if Worker B holds the lock when Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[token])`, the Lua script's `GET` returns Worker B's token, the comparison fails, and the `DEL` is not executed. The script correctly returns 0.

However, this only holds if the lock has not expired and been re-acquired with the *same* token value — which cannot happen with UUID4 tokens in practice. So the release script is safe as written.

The residual concern is that Worker A's `finally` block runs at all after lock loss, which it does. The release attempt is harmless (token mismatch prevents deletion), but it is a sign that the finally block does not know it is in a degraded state. This reinforces Finding 1: the finally block should check `lost_event` and log a clear alarm rather than silently attempting a release that will no-op.

---

**Finding 4: No Test Covers the Lock-Loss / Zombie-Write Scenario**

The test suite covers contention at acquisition time (`redis_client.set.return_value = False`) but has no test for the scenario where the lock is lost *during* execution — heartbeat returns 0, main thread continues, commit proceeds. This is the highest-severity failure mode and it is entirely uncovered. A test should simulate `_renew_script` returning 0 mid-execution and assert that `db.record_execution` is never called.

---

## Verdict: NOT APPROVED

The primary correctness risk is **Finding 1**: the heartbeat thread exits silently on lock loss with no signal to the main thread, which continues executing and commits to the database. This is a confirmed zombie-write path. It requires both a threading.Event reverse signal (to narrow the window) and a fencing token at the database write layer (to close it). Neither is present.

The prior review's Lua atomicity concern is dismissed — it is not a real failure mode. The actual risk is not inside the script; it is in what the caller does when the script returns 0.
