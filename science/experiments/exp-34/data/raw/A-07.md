## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token T1, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s — longer than `CLAIM_TTL`. No renewal fires in time (or the renewal fires but the network call to Redis itself stalls), and the lock expires naturally.
3. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B passes the idempotency check (`get_execution` returns None, job not yet recorded), enters `_dispatch()`, and begins executing the job.
4. Worker A's main thread resumes. It is now past the stall. Its heartbeat fires: `RENEW_CLAIM_SCRIPT` runs `GET job:claim:{job_id}` → returns T2, which ≠ T1. The script returns 0. The heartbeat thread hits `if result == 0: return` and exits silently. **No signal is sent to the main thread.**
5. Worker A's main thread has no awareness of this. It continues executing — `_dispatch()` completes, the `db.transaction()` block runs, `record_execution` and `mark_job_complete` are called.
6. Worker B also completes the same path.
7. **Both workers execute the job to completion.** For a payment charge, alert dispatch, or external API call inside `_dispatch()`, this is a duplicate operation.

**The structural gap:** `_renew()` returns on `result == 0` but the `stop_event` is never set, and no exception is raised into the main thread. The `yield token` in `_claim_job` continues uninterrupted. The heartbeat thread's knowledge that the lock is gone is not reachable from the execution path.

**Required fix — two layers, both required:**

*Layer 1 — Signal the main thread on lock loss (necessary but not sufficient):*
Set `stop_event` on `result == 0` is insufficient alone because `stop_event` is only checked by the heartbeat thread itself. A separate `threading.Event` (e.g., `lock_lost_event`) must be set by the heartbeat on `result == 0`. The main thread must check this event at each meaningful boundary — after `_dispatch()` returns, before entering the `db.transaction()` block — and abort if the lock was lost. This closes the gap for the common case but does not close it architecturally.

*Layer 2 — Fencing token / optimistic concurrency at the write boundary (required to close the zombie-write scenario completely):*
Thread signaling only helps if the main thread is alive and checking. It does not protect against a process that is paused (GC, SIGSTOP, swap) across the entire execution window and resumes after the lock has expired and been re-acquired by another worker. The only architectural boundary that prevents the zombie write is a **fencing token** (monotonic counter or generation number) embedded in the lock value and propagated into the database write.

Concretely: Redis can be replaced with a lock service that issues a monotonic sequence number on each acquisition (Redlock does not provide this; a single authoritative Redis instance with `INCR` on each acquisition does, or a purpose-built lock service). The token passed to `record_execution` and `mark_job_complete` must be the fencing token. The database write must include a conditional: `UPDATE jobs SET status='complete', fencing_token=N WHERE job_id=X AND (fencing_token IS NULL OR fencing_token < N)`. If Worker A's write arrives after Worker B has already written with a higher token, Worker A's write is rejected. The application must treat a rejected write as a correctness signal, not a silent no-op.

Without Layer 2, any sufficiently long process pause produces the duplicate regardless of how correctly the threading is handled.

**Finding 2: Idempotency Check Is Outside the Lock's Causal Guarantee**

**Issue name:** Check-then-act race on idempotency guard

**Failure path:**

1. Workers A and B both attempt to claim `job:claim:{job_id}`. Worker A wins the `SET NX`. Worker B fails and raises `ClaimError` — correct so far.
2. But consider: Worker A holds the lock, completes execution, calls `mark_job_complete`, then releases the lock. Worker C now acquires the lock. Worker C calls `db.get_execution()` — this correctly returns the existing record and Worker C exits early. This path is fine.
3. The dangerous path: the idempotency check (`get_execution`) and the subsequent `_dispatch` + `record_execution` are not atomic with respect to the Redis lock. If the lock expires between `get_execution` returning None and `record_execution` being called (the zombie-write scenario from Finding 1), two workers can both pass the idempotency check and both write. The idempotency check is only meaningful if the lock is guaranteed to be held for its entire duration — which it is not, given Finding 1.

**Required fix:** The idempotency check becomes reliable only when combined with the fencing token write described in Finding 1, Layer 2. The database `INSERT` for `record_execution` should be made conditional (unique constraint on `job_id`, or `INSERT ... WHERE NOT EXISTS`, or the fencing token condition). The database itself must be the final arbiter of uniqueness, not the Redis lock. Redis provides the coordination layer to reduce contention; the database provides the correctness guarantee.

---

**Finding 3: Release Script Executes After Lock May Already Be Held by Another Worker**

**Issue name:** Phantom release — token-verified release is correct but the timing window is unguarded

**Failure path:**

1. Worker A's lock expires. Worker B acquires the lock with token T2.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join()`, then `self._release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET job:claim:{job_id}` → T2 ≠ T1 → returns 0, does not delete. This is correct behavior — the script itself is safe.
4. However, Worker A's main thread has already executed `_dispatch` and `db.transaction()` before reaching `finally`. The release script's correctness does not retroactively prevent the duplicate execution that already occurred.

**Assessment:** The release script is correctly written and does not introduce an additional failure mode beyond Finding 1. It does not make things worse. But it also provides no protection against the zombie write — it only prevents Worker A from deleting Worker B's lock after the fact, which is the right behavior. This finding is noted to confirm the release path is not an independent source of incorrectness.

---

**Finding 4: `t.join(timeout=...)` Does Not Confirm Thread Completion**

**Issue name:** Unconfirmed thread join — release may proceed with heartbeat still in flight

**Failure path:**

1. `stop_event.set()` is called. `t.join(timeout=RENEWAL_INTERVAL + 1)` = 16s timeout.
2. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis latency spike), `join` returns after 16s regardless of whether the thread has actually exited.
3. `_release_script` is then called. If the heartbeat thread is still mid-execution of `_renew_script`, there is a brief window where both the release and a renewal are in flight concurrently against the same key. Because both are Lua scripts and Redis is single-threaded, they will serialize — but the ordering is non-deterministic. A renewal that lands after the release re-extends a key that was intentionally deleted, leaving a ghost lock that blocks the next legitimate acquisition for up to `CLAIM_TTL` seconds.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()`. If the thread is still alive, log an error and do not call the release script — or accept that the TTL will expire the key naturally. Alternatively, reduce the join timeout and document the trade-off explicitly.

---

**Zombie-Write Scenario Confirmation**

The zombie-write scenario **can occur** under the following conditions, all of which are reachable in production:

- Worker A holds the lock and enters a GC pause or slow external call in `_dispatch()` exceeding `CLAIM_TTL` (60s). This is not exotic — a JVM-style full GC, a slow payment processor call, or a large database query can exceed this.
- Worker B acquires the lock, passes the idempotency check, and begins executing.
- Worker A resumes, its heartbeat exits silently on `result == 0`, its main thread continues with no signal, and it completes execution.
- Both workers call the job handler and commit to the database.

**What is missing to close it:** A fencing token issued monotonically on each lock acquisition, propagated into the database write, with a conditional write at the database layer that rejects stale tokens. This is the only boundary that holds under arbitrary process pauses. Thread signaling (`lock_lost_event`) is a useful defense-in-depth measure for the non-paused case but is not sufficient as the sole architectural control.

---

## Verdict: NOT APPROVED

The heartbeat thread exits silently on lock loss with no signal to the main thread, and there is no fencing token or conditional write at the database layer. Under a GC pause or slow external call exceeding `CLAIM_TTL`, two workers execute the same job concurrently and both commit. For jobs that charge payments, send notifications, or trigger external actions, this is a correctness violation. The code must not go to production in this state.
