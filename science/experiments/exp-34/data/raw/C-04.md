## Findings

---

**Finding 1 — Primary Correctness Risk: Heartbeat Thread Exits Silently; Main Thread Continues Executing Without Lock**

**Issue name:** Silent lock loss — zombie write via undetected heartbeat failure

**Failure path:**

1. Process A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Process A enters `_dispatch()` — a slow external call, a GC pause, or a long DB query holds the process for >60s without the heartbeat firing in time, *or* the heartbeat fires but the Redis connection is saturated and the renewal fails silently, or the process simply stalls longer than one renewal interval allows recovery.
3. The lock TTL expires. Process B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes `GET job:claim:{job_id}` → returns `T2`. Token mismatch. Script returns `0`.
5. `_renew()` checks `if result == 0: return`. The thread exits. **No signal is sent to the main thread. No exception is raised. No event is set.**
6. Process A's main thread is still inside `with self._claim_job(...)`. It has no visibility into the heartbeat thread's exit. It continues through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution()` and `mark_job_complete()`.
7. Process B does the same. Both processes complete the job. For a payment charge, alert dispatch, or external event trigger, this is a duplicate execution — a correctness violation.

The `finally` block in `_claim_job` will eventually call `self._release_script(...)` with token `T1`. Since the key now holds `T2`, the Lua release script correctly returns `0` and does not delete B's lock. That part is safe. But the damage — duplicate execution — has already occurred.

**Why the existing idempotency check does not save you:**

The `db.get_execution()` check runs *before* `_dispatch()`. At the moment Process A checks, Process B has not yet written a completion record (it is also mid-execution). Both processes pass the idempotency check and proceed to dispatch. The check is a necessary layer but cannot close this gap because it is not re-evaluated after the lock is lost.

**Required fix — two layers, both necessary:**

*Layer 1 (immediate, necessary but not sufficient): Signal the main thread on lock loss.*

`_renew()` must communicate lock loss back to the main thread. The minimal mechanical fix is a shared `threading.Event` (`lock_lost_event`). When `result == 0`, set it. The main thread must check this event at a meaningful boundary — ideally before and after `_dispatch()`, and before entering the DB transaction. If the event is set, abort and raise.

```python
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key], args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()
            logger.error("Lock lost during renewal job_id=%s — signaling main thread", job_id)
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# In the main execution path, before the DB transaction:
if lock_lost_event.is_set():
    raise ClaimError(f"Lock lost during execution of job {job_id}")
```

This is Tier 0.5: correct diagnosis, incomplete architectural fix. It reduces the window but does not eliminate it. The main thread can still be mid-`_dispatch()` when the event is set, and the check only fires at the next polling point. Any work already done by `_dispatch()` with side effects (external API calls, charges) has already executed.

*Layer 2 (architectural, required for correctness): Fencing token / monotonic counter passed through to side-effecting operations.*

The root problem is that the lock token is not threaded through to the operations the lock is supposed to protect. The lock and the work it guards are decoupled after acquisition.

The fix is to use a Redis-based monotonic counter (e.g., `INCR job:fence:{job_id}`) at claim acquisition time, returning a fencing token — a monotonically increasing integer. This token is passed into `_dispatch()` and into the DB transaction. Every side-effecting operation — the external call, the DB write — must include the fencing token as a precondition. The DB `record_execution` write should be a conditional write: `INSERT ... WHERE fence_token = $1 AND NOT EXISTS (SELECT 1 FROM executions WHERE job_id = $2 AND fence_token >= $1)`. If Process B has already written with a higher fence token, Process A's write fails. The DB becomes the final arbiter of which execution is authoritative.

This is the only fix that closes the gap for side effects that have already fired: it makes the *recording* of the execution — and therefore its observability and downstream consequences — conditional on still holding the most recent claim. External calls (HTTP, email, payment APIs) remain a harder problem and require idempotency keys scoped to the fencing token passed to the downstream system.

**Why this fix must live at this layer:** The heartbeat thread is the only component with visibility into lock loss. The main thread has no other channel. The DB transaction is the last atomic boundary before the execution becomes durable. Both layers are required because signaling alone does not protect already-in-flight side effects, and fencing alone does not prevent redundant external calls.

**Finding 2 — Prior Review's Atomicity Concern: Lua Script Sequential Operations**

**Issue name:** Claimed atomicity gap in `RENEW_CLAIM_SCRIPT` — dismissed as stated, real concern is different**

**Assessment: The prior review's specific concern is incorrect. The stated failure path does not exist.**

The concern was: under high Redis connection saturation, the `GET` and `EXPIRE` inside `RENEW_CLAIM_SCRIPT` may not execute atomically, creating a window where renewal appears to succeed after another worker has claimed the key.

This is wrong. Redis Lua scripts execute atomically with respect to all other Redis commands. From the Redis documentation and implementation: the entire script runs in a single-threaded execution context; no other client command is processed between the `GET` and the `EXPIRE`. Redis connection saturation affects *queueing* of commands — how long a client waits before its script starts executing — but once the script begins, it runs to completion without interleaving. There is no window between the `GET` and `EXPIRE` inside the Lua script where another client can modify the key. The prior review's concern conflates network/connection latency with Redis execution atomicity. These are separate concerns. The script is atomic. The concern is dismissed.

**However, there is a real timing concern in the renewal mechanism that the prior review missed:**

The window that matters is not *inside* the Lua script. It is the gap between the lock expiring and the heartbeat firing. The heartbeat renews every 15 seconds. The TTL is 60 seconds. If the process stalls for 60 seconds, the lock expires. The heartbeat fires at second 60+ε, the key is gone (or held by another worker), and `result == 0`. The script's internal atomicity is irrelevant to this scenario. The real atomicity concern is between lock expiry and the main thread's continued execution — and that is Finding 1.

---

**Finding 3 — Secondary Risk: `_dispatch()` Executes Outside Any Atomic Boundary**

**Issue name:** Side effects fire before DB transaction; no rollback possible on external calls

**Failure path:**

`_dispatch()` is called before `self.db.transaction()`. If `_dispatch()` triggers an external side effect (sends a notification, charges a payment, fires a webhook) and then the DB transaction fails, the external action has already occurred but no execution record exists. On retry, the job will pass the idempotency check (no existing execution record) and dispatch again. This is a duplicate execution triggered by DB failure rather than lock loss.

**Required fix:** Either move `_dispatch()` inside the DB transaction (if the handler is DB-safe), or require all handlers to accept an idempotency key derived from `execution_id` and pass it through. The `execution_id` is generated before `_dispatch()` is called — it should be passed into the handler so the handler can use it as an idempotency key for external calls. The DB transaction should then record that `execution_id` as the canonical record, making retries detectable.

---

## Verdict: NOT APPROVED

The primary correctness risk is **Finding 1**: the heartbeat thread exits silently on lock loss and the main thread continues executing, producing duplicate job execution in any scenario where the process stalls beyond `CLAIM_TTL`. This is a live zombie-write path in a multi-worker deployment.

The prior review's Lua atomicity concern is **dismissed**: Redis Lua scripts are atomic by specification; the `GET`/`EXPIRE` sequence inside `RENEW_CLAIM_SCRIPT` cannot be interleaved by another client. The real timing risk is the lock-expiry-to-heartbeat gap, which is the mechanism behind Finding 1, not a script internals issue.

The code cannot be approved until: (a) `_renew()` signals the main thread on `result == 0`, (b) the main thread checks that signal before entering the DB transaction, and (c) a fencing token strategy is defined for side-effecting operations in `_dispatch()` so that the DB write — and downstream systems — can reject stale executions. Finding 3 requires resolution before any payment or notification handler is registered against this executor.
