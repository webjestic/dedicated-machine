## Findings

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie-write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()` — a slow external call, GC pause, or I/O stall holds the process for >60s without the heartbeat firing in time (or the heartbeat fires but the Redis call itself is slow/blocked, missing the renewal window).
3. The lock expires. Worker B races in, acquires `job:claim:{job_id}` with token `T2`, passes the `get_execution` idempotency check (nothing recorded yet), and begins executing the same job.
4. Worker A's heartbeat eventually fires. `RENEW_CLAIM_SCRIPT` executes `GET job:claim:{job_id}` → returns `T2`, which does not match `T1`. The script returns `0`. The heartbeat thread hits `if result == 0: return` and exits silently.
5. Worker A's main thread has no reference to the heartbeat thread's state. `stop_event` is never set by the heartbeat thread. The main thread continues executing — `_dispatch()` completes, the `db.transaction()` block runs, `record_execution` and `mark_job_complete` are called.
6. Worker B also completes the same path.
7. **Outcome:** The job executes twice. Both workers write completion records. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, this is a duplicate operation.

**The structural gap:** `_renew()` returns on `result == 0` but has no mechanism to signal the main thread. The `stop_event` is write-only from the heartbeat's perspective — the heartbeat reads it but never sets it. There is no shared flag, no exception raised into the main thread, no cancellation path. The main thread's `yield token` inside the `try` block proceeds unconditionally.

**Required fix — two layers:**

*Layer 1 (necessary but insufficient): threading.Event signaling.* Add a `lock_lost_event = threading.Event()` that `_renew()` sets before returning on `result == 0`. After `yield token` returns (i.e., after `_dispatch` and the DB transaction complete), check `lock_lost_event.is_set()` and raise. Better: check it *before* the DB write. This prevents the completion record from being written after lock loss, but does not prevent `_dispatch()` from having already executed the side-effectful work — the external call already fired.

*Layer 2 (architecturally required): Fencing token / optimistic concurrency at the side-effect boundary.* The Redis lock is a *advisory* mechanism. Any process that survives a pause longer than TTL will have its lock silently stolen. The only way to prevent the zombie write from mattering is to make the side-effect itself reject stale writers.

Concretely:
- Use a **monotonic generation counter** stored in Redis (or the DB) per job. When Worker A acquires the lock, it reads generation `N` and holds it as its fencing token. Worker B, acquiring after expiry, increments to generation `N+1`.
- Pass the fencing token into `_dispatch()` and into the DB write. The DB write becomes a conditional update: `UPDATE jobs SET status='complete', execution_id=? WHERE job_id=? AND generation=N`. Worker A's write with generation `N` fails because the DB now holds `N+1`. Worker B's write with `N+1` succeeds.
- This makes the DB the authoritative fencing boundary. The Redis lock becomes a performance optimization (reduces contention), not a correctness guarantee.

Without Layer 2, any GC pause or network stall longer than CLAIM_TTL produces a window where two workers execute the job and both writes succeed.

**Finding 2: Idempotency Check Is Not Atomic With Execution**

**Issue name:** TOCTOU race on `get_execution` / `record_execution`

**Failure path:**

1. Workers A and B both acquire the lock sequentially (not concurrently) — but consider a variant: the lock is released and re-acquired between the check and the write due to a crash-and-restart scenario.
2. More directly: Worker A checks `get_execution` → `None`, proceeds to `_dispatch()`. Worker A crashes after `_dispatch()` but before `record_execution`. Worker B acquires the lock, checks `get_execution` → still `None` (nothing was recorded), and executes the job again.
3. **Outcome:** The job executes twice. The idempotency check only guards against *recorded* completions; it does not guard against in-progress or partially-completed executions.

**Required fix:** Record an "in-progress" or "claimed" state in the DB *before* calling `_dispatch()`, inside a transaction. The completion record then becomes an update from `in-progress` → `complete`. A second worker checking `get_execution` will find the in-progress record and bail. This requires `get_execution` to treat in-progress as "do not execute," not just completed.

**Finding 3: Lock Release Executes After Lock May Already Be Lost**

**Issue name:** Release script runs against a key owned by another worker

**Failure path:**

1. Worker A's lock expires. Worker B acquires the same key with token `T2`.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join()`, then `_release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET job:claim:{job_id}` → `T2 ≠ T1` → returns `0`, does not delete. This is **correct behavior** — the script is token-verified and safe.
4. No incorrect outcome from the release itself.

**Assessment:** This is handled correctly by the Lua script. No fix required here, but it underscores that the release script's token check is the *only* fencing in the current design — and it only protects the key deletion, not the work already done.

**Finding 4: Heartbeat Interval Does Not Guarantee Renewal Before Expiry Under Timing Pressure**

**Issue name:** Renewal window too narrow under load

**Failure path:**

1. CLAIM_TTL=60s, RENEWAL_INTERVAL=15s. Under normal conditions, the lock is renewed at ~15s, ~30s, ~45s — well within TTL.
2. Under GC pressure, thread starvation, or Redis latency spikes, the renewal thread's `stop_event.wait(timeout=15)` wakes on time but the subsequent `_renew_script` Redis call blocks for >45s (remaining TTL). The lock expires before the renewal completes.
3. This is the same trigger as Finding 1 but via the renewal path rather than the main thread path.

**Required fix:** The ratio of RENEWAL_INTERVAL to CLAIM_TTL is reasonable (15/60 = 0.25), but the renewal thread should log a warning if the renewal call itself takes longer than a threshold, and the lock_lost_event mechanism from Finding 1 must cover this path as well. The architectural fix (fencing token) covers this case identically.

---

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** No test for heartbeat-death / continued execution path

**Failure path:** The test `test_claim_release_uses_lua_script` verifies the release script is called. No test simulates `_renew_script` returning `0` and verifies that the main thread halts execution. The gap is untested and will not be caught by CI.

**Required fix:** Add a test that configures `_renew_script` to return `0`, injects a `lock_lost_event` check, and asserts that `record_execution` is *not* called after lock loss. This test will fail against the current implementation, which is the correct outcome — it documents the bug.

## Zombie-Write Scenario — Explicit Statement

**Can it occur?** Yes, unambiguously.

**Trigger:** Worker A stalls (GC pause, slow `_dispatch()` call, thread starvation) for longer than CLAIM_TTL=60s.

**Sequence:** Lock expires → Worker B acquires → Worker B executes job → Worker A's heartbeat detects token mismatch, returns 0, exits silently → Worker A's main thread has no signal, continues → Worker A completes `_dispatch()` and writes to DB → both workers have executed the job.

**What closes it:** A fencing token (monotonic generation counter) passed through to the DB write, making `record_execution` a conditional update that rejects the stale writer. The Redis lock alone cannot provide this guarantee because TTL-based locks have no way to prevent a paused process from continuing after expiry. The authoritative rejection must live at the resource boundary — the database write — not at the lock acquisition boundary.

The `threading.Event` signal from heartbeat to main thread is a necessary improvement (it stops the *next* job from being dispatched and reduces the duplicate-write window) but is not sufficient: `_dispatch()` may have already executed the external side effect before the signal could be checked.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed reproducible. The heartbeat thread exits silently on lock loss with no signal path to the main thread, and there is no fencing token at the DB write boundary to reject stale writers. Two workers can execute the same job concurrently and both writes succeed. This must not go to production in a deployment where job execution has external side effects.
