## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write on Lock Loss**

**Issue name:** Undetected lock loss / zombie write

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. The heartbeat thread starts.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the process for >60s.
3. The lock TTL expires. Redis deletes the key.
4. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B begins executing the same job: idempotency check passes (DB record not yet written), dispatch runs, transaction commits.
5. Meanwhile, Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes `GET` → value is `T2`, not `T1`. The script returns `0`.
6. `_renew()` sees `result == 0` and executes `return`. The thread exits. **No signal is sent to the main thread. No exception is raised. No flag is set.**
7. Worker A's main thread is still inside `_dispatch()` or has returned from it. It has no visibility into the heartbeat's exit. It proceeds to `self.db.transaction()`, calls `record_execution` and `mark_job_complete`, and returns `JobResult(claimed=True, ...)`.
8. Both workers complete the job. The job executes twice.

This is the classical zombie-write: a process that has lost its lock continues writing as though it still holds it. The heartbeat correctly detects the loss but discards the information.

**The specific code gap:**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return          # ← exits silently; main thread never knows
        logger.debug("Claim renewed job_id=%s", job_id)
```

The `return` statement terminates the thread. The main thread, blocked in `_dispatch()` or executing the DB transaction, has no mechanism to observe this. `stop_event` is only written by the `finally` block — it is never written by `_renew()`.

**Why `threading.Event` signaling is insufficient as a complete fix:**

Adding `stop_event.set()` in `_renew()` on lock loss, and checking that event in the main thread between `_dispatch()` and the DB transaction, would prevent the duplicate *commit* in many cases. But it does not prevent the duplicate *side effect*. If the external action (payment, notification, alert) happens inside `_dispatch()`, both workers have already executed it before either checks any flag. The event signal cannot un-ring that bell. It also introduces a TOCTOU gap: the check must happen at every point where irreversible work occurs, and the main thread can be paused again between the check and the next write.

**Required fix — fencing token / optimistic concurrency at the persistence layer:**

The architectural boundary that actually prevents the zombie write is the database transaction, not the lock check. The fix must make the DB write conditional on the lock still being held at commit time.

Concretely:

- Use a monotonic fencing token (a Redis counter incremented on each acquisition, or a Lua script that returns the token value on SET) rather than a random UUID.
- Pass that token into the DB transaction as an optimistic concurrency condition: `UPDATE jobs SET status='complete', last_token=? WHERE job_id=? AND last_token IS NULL` (or equivalent compare-and-swap on the execution record).
- If Worker A's token has been superseded by Worker B's token, Worker A's transaction either fails the conditional write or is rejected by a unique constraint on `(job_id, token)`.
- The DB write, not the lock check, becomes the serialization point. The lock is demoted to a performance optimization (reducing contention) rather than a correctness guarantee.

This fix must live at the persistence layer because that is the only layer that is not subject to process pauses between the check and the write. A Redis check followed by a DB write is always a two-phase operation with a gap; a conditional DB write is atomic with respect to the state it guards.

**Finding 2: Dismissal of the Prior Review's Atomicity Concern re: `RENEW_CLAIM_SCRIPT`**

**The concern:** The prior review flagged that `GET` followed by `EXPIRE` in the Lua script might not be atomic under high Redis connection saturation.

**Verdict: This concern is not a real failure path.**

Redis executes Lua scripts atomically with respect to all other Redis commands. From the Redis documentation and implementation: the entire script runs in a single-threaded execution step; no other client command can be interleaved between `redis.call("get", ...)` and `redis.call("expire", ...)` within the same script invocation. Redis's single-threaded command processing model is the atomicity boundary here. Connection saturation affects *queuing* of script invocations — a script may wait longer to begin execution — but once execution begins, it runs to completion without interleaving.

The scenario the prior review imagined — another worker claims the key between the `GET` and the `EXPIRE` within a single Lua invocation — cannot occur. If Worker B's `SET NX` is queued behind Worker A's `RENEW_CLAIM_SCRIPT`, it executes after the script completes. If it is queued before, the script's `GET` will see Worker B's token and return `0` without extending. There is no interleaving window.

**The prior review's concern is dismissed.** The Lua script is correctly atomic. It is not the primary correctness risk.

**The primary correctness risk is Finding 1:** the heartbeat correctly detects lock loss and returns `0`, but that information is silently discarded, leaving the main thread to execute and commit as a zombie.

---

## Verdict: NOT APPROVED

The Lua renewal script is correctly atomic — the prior review's concern is dismissed. The primary correctness risk is structural: the heartbeat thread detects lock loss and exits silently, with no mechanism to halt or invalidate the main thread's continued execution. Under a GC pause or slow external call exceeding `LOCK_TTL`, two workers will both complete the job. For jobs with external side effects — charges, notifications, outbound events — this is a duplicate-execution correctness violation. The fix requires a fencing token propagated into a conditional DB write, making the persistence layer the serialization point rather than relying on the lock check alone.
