## Findings

---

**Finding 1: Heartbeat Silent Exit Does Not Signal Main Thread — Zombie Write Scenario**

**Issue name:** Undetected lock loss / zombie write

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s. The heartbeat fires during the stall but the renewal interval (15s) means up to one full 60s TTL window can elapse between a successful renewal and the next attempt — or the process stalls hard enough that the heartbeat thread itself is not scheduled.
3. The lock expires. Worker B races in, acquires `job:claim:{job_id}` with token `T2`, passes the idempotency check (no completed execution recorded yet), and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` reads the key, finds `T2 ≠ T1`, returns `0`.
5. The heartbeat thread hits `if result == 0: return` — **it exits silently**. No exception is raised. No `threading.Event` is set. No flag is written. The main thread has zero visibility into this exit.
6. Worker A's main thread resumes from the stall, completes `_dispatch()`, enters `self.db.transaction()`, records execution, and marks the job complete.
7. Worker B does the same. Both complete. The job executes twice.

**The structural gap:** `_renew()` treats `result == 0` as a terminal condition for the *thread* but not for the *caller*. The context manager `yield token` has already handed control to the job body. There is no shared signal between the heartbeat thread and the main thread. The `stop_event` only flows *from* main thread *to* heartbeat — never the reverse.

**Required fix — two layers:**

*Layer 1 (necessary, insufficient alone):* Add a `lock_lost_event = threading.Event()` that `_renew()` sets before returning on `result == 0`. The main thread must check this event at each logical checkpoint (after `_dispatch()`, before entering the DB transaction). If set, abort and raise. This closes the intra-process signaling gap but does not close the architectural gap.

*Layer 2 (necessary for correctness):* The DB write must be gated by a **fencing token / optimistic concurrency check** at commit time. The token acquired at lock claim time (or a monotonic counter from Redis, e.g. `INCR` on a version key at acquisition) must be presented to the DB transaction as a precondition. The DB layer rejects the write if a higher token has already been recorded for this job. This means even if the main thread races past a stale lock and reaches the DB write, the write is rejected rather than committed. This is the only boundary that survives a scenario where the process stalls so hard that the heartbeat never fires at all and the main thread resumes after lock expiry with no in-process signal ever having been generated.

Without Layer 2, any scenario where the process stalls longer than CLAIM_TTL *before* the heartbeat fires produces a zombie write that no in-process check can catch.

**Finding 2: Renewal Interval vs. TTL Ratio Leaves an Undetected Expiry Window**

**Issue name:** Renewal gap — lock expires between heartbeat ticks

**Failure path:**

CLAIM_TTL=60s, RENEWAL_INTERVAL=15s. The heartbeat renews at t=0, t=15, t=30, t=45. If the process stalls at t=16 for 45s (stall ends at t=61), the lock expired at t=60. The heartbeat at t=61 fires, gets `result == 0`, exits silently. The main thread resumes at t=61 with an expired lock and no signal. Worker B has had up to 1 second to acquire the lock and may already be mid-execution.

The 4:1 TTL-to-interval ratio is reasonable for network jitter but provides no protection against process-level pauses. The ratio is not the root cause — the root cause is Finding 1 — but it means the window for undetected expiry is up to `RENEWAL_INTERVAL` seconds (15s) even in the best case where the heartbeat is running normally.

**Required fix:** Addressed by the fencing token fix in Finding 1, Layer 2. No ratio adjustment closes this without architectural change.

**Finding 3: Idempotency Check Is Not Atomic With Lock Acquisition**

**Issue name:** TOCTOU on idempotency check

**Failure path:**

1. Worker A acquires lock, reads `db.get_execution(job_id)` → `None`.
2. Worker A stalls. Lock expires. Worker B acquires lock, reads `db.get_execution(job_id)` → `None` (A has not written yet).
3. Both workers proceed past the idempotency check. Both execute. Both attempt to write.
4. The DB transaction in step 3 of Finding 1 is the only remaining gate — and without a fencing token precondition, both writes succeed if they hit different DB replicas or if the transaction isolation level permits phantom reads.

The idempotency check as written is a best-effort optimization, not a correctness guarantee. It correctly prevents re-execution of *already-completed* jobs but provides no protection against concurrent execution of an *in-progress* job where neither worker has written yet.

**Required fix:** The DB `record_execution` / `mark_job_complete` transaction must use an INSERT with a unique constraint on `job_id` (or equivalent upsert-with-conflict-detection), so the second writer receives a constraint violation rather than a silent success. This makes the DB the authoritative correctness boundary, which it must be regardless of what the lock layer does.

**Finding 4: Release After Lock Loss Silently Succeeds on Wrong Key State**

**Issue name:** Phantom release on expired lock

**Failure path:**

Worker A's lock expires. Worker B acquires the same key with token `T2`. Worker A's `finally` block calls `_release_script(keys=[claim_key], args=[T1])`. The Lua script checks `get(key) == T1` → false → returns `0`. The release is correctly a no-op. **This part is fine.**

However: if Worker B has already completed and released the key, and Worker A's release fires into an empty key, the Lua script also returns `0` (key does not exist, `get` returns nil, nil ≠ T1). This is also a no-op. **Also fine.**

The release script itself is correct. This finding is informational: the Lua guard on release is functioning as intended and does not introduce a new failure path. No fix required here.

---

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** Heartbeat silent exit not tested; zombie write scenario has no regression coverage

**Failure path:** The test `test_claim_release_uses_lua_script` verifies the release script is called. No test simulates `_renew_script` returning `0` and asserts that the main thread is notified or that execution is aborted. No test simulates concurrent execution after lock expiry. The gap in Finding 1 has no regression test, meaning it can be reintroduced silently.

**Required fix:** Add tests that:
- Mock `_renew_script` to return `0` on first call and assert that `execute()` raises or returns a non-success result rather than completing the job.
- Simulate two executor instances racing on the same job_id after lock expiry and assert exactly one `record_execution` call occurs (requires the DB unique constraint from Finding 3).

## Zombie-Write Scenario — Explicit Statement

**The zombie-write scenario can occur under current code.**

Trigger: Worker A's process stalls (GC, I/O, OS scheduling) beyond CLAIM_TTL. Worker B acquires the lock. Worker A's heartbeat detects token mismatch and exits — silently, with no signal to the main thread. Worker A resumes, completes `_dispatch()`, and commits to the database. Worker B also commits. The job executes twice.

**What must be added to close it:**

1. A `lock_lost_event` flowing from heartbeat thread to main thread, checked before the DB write.
2. A fencing token passed into the DB transaction as an optimistic concurrency precondition, so a write from a worker that has lost its lock is rejected at the DB layer even if the in-process signal is missed.
3. A unique constraint on `job_id` in the execution table so concurrent writes produce a conflict rather than duplicate rows.

Items 2 and 3 together form the architectural boundary. Item 1 is a defense-in-depth improvement. Without 2 and 3, the system has no hard correctness guarantee regardless of what the locking layer does.
