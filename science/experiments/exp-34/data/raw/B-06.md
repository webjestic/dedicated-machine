**Findings**

---

**Finding 1: Silent Heartbeat Exit Does Not Stop Main Thread — Zombie Write Is Possible**

**Issue name:** Undetected lock loss / zombie write at DB boundary

**Failure path, step by step:**

1. Process A acquires `job:claim:{job_id}` with token `T_A`, TTL = 60s. Heartbeat thread starts, renewing every 15s.
2. Process A enters `_dispatch()`. A GC pause, slow external HTTP call, or I/O stall holds the process for > 60s. No renewal fires during the stall (the heartbeat thread is also paused, or fires but the lock has already expired before it can renew).
3. The Redis TTL expires. The key is gone.
4. Process B calls `redis.set(claim_key, token, nx=True, ex=60)` — succeeds. Process B now holds the lock with token `T_B` and begins executing the same job.
5. Process A's stall ends. Its heartbeat fires: `RENEW_CLAIM_SCRIPT` runs `GET job:claim:{job_id}` → returns `T_B` ≠ `T_A` → returns `0`. The `_renew` function hits `if result == 0: return` and the thread exits silently.
6. **Critical gap:** The main thread in Process A is inside the `with self._claim_job(req.job_id):` block. `stop_event` has not been set — it is only set in the `finally` block after `yield` returns. The main thread has zero visibility into the heartbeat thread's exit. There is no shared flag, no exception raised into the main thread, no `threading.Event` checked by the main thread between steps.
7. Process A continues: `self.db.get_execution(req.job_id)` — if Process B has not yet committed, this returns `None`. Process A calls `_dispatch()`, then enters `self.db.transaction()` and calls `record_execution` + `mark_job_complete`.
8. Process B does the same, independently, because it also saw `get_execution` return `None` before either committed.
9. Both processes commit. The job executes twice. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the duplicate is already out the door before the DB write even happens — `_dispatch()` is called *before* the transaction, so the side effect is not transactional at all.

**What the code does have and why it is insufficient:**

- The Lua renewal script correctly detects token mismatch and returns 0. This is necessary but not sufficient — it only informs the heartbeat thread.
- The `stop_event` pattern correctly stops the heartbeat on normal exit. It does nothing for the lock-loss case because the heartbeat exits without setting it.
- The DB transaction wraps `record_execution` + `mark_job_complete` atomically. This prevents a torn write *within* a single process but does not prevent two processes from both reaching that transaction.
- The idempotency check (`get_execution` before dispatch) is a read, not a write with a uniqueness constraint. Under concurrent execution it is a TOCTOU race: both processes can read `None` before either writes.

---

**Finding 2: Side-Effecting `_dispatch()` Executes Outside Any Concurrency Guard**

**Issue name:** External side effect before DB transaction — duplicate action is not preventable by DB-level idempotency

**Failure path:**

Even if Finding 1 were partially mitigated by a `threading.Event` that the main thread polls, the structure of `execute()` is:

```
_dispatch(req.job_type, req.payload)   # ← external side effect here
with self.db.transaction():            # ← DB write here
    record_execution(...)
    mark_job_complete(...)
```

`_dispatch` runs the job handler — sends the notification, charges the payment, fires the event — *before* any durable record exists. If two workers reach `_dispatch` concurrently (the zombie-write scenario above), both external actions fire. The DB transaction that follows cannot undo an already-sent notification or already-submitted charge. The idempotency check at the top of the block does not help because both workers passed it before either committed.

The DB transaction also cannot enforce uniqueness between the two workers unless the schema has a unique constraint on `job_id` in the executions table. Nothing in this code establishes or relies on such a constraint.

---

**Finding 3: No Fencing Token or Optimistic Concurrency at the DB Write Boundary**

**Issue name:** Missing write-boundary guard — threading signal is not an architectural fence

**What is absent:**

The code has no mechanism at the database write boundary that makes a zombie write *structurally impossible*. Specifically:

- No unique constraint on `job_id` in the executions/completion table that would cause one of the two concurrent `record_execution` calls to fail with a constraint violation.
- No optimistic concurrency check (e.g., `UPDATE jobs SET status='complete', version=N+1 WHERE job_id=X AND status='pending' AND version=N`) that would cause exactly one writer to succeed and the other to get zero rows affected.
- No fencing token passed from the lock acquisition into the DB write (e.g., a monotonic lock version stored in Redis and written as a condition into the DB update).

A `threading.Event` checked by the main thread would be a Tier 0.5 fix: it reduces the window by detecting lock loss and aborting *before* the write in the common case. It does not close the window. The main thread can be scheduled between the event check and the DB write. The event can be set after the check passes. The heartbeat can exit between two lines of Python. Only a constraint enforced by the DB itself — evaluated atomically at write time — makes the duplicate write impossible regardless of scheduling.

---

**Required Fixes**

**Fix for Finding 1 + 3 (the architectural fix — must live at the DB layer):**

Add a unique constraint on `job_id` in the job completions table (or executions table, whichever is written by `mark_job_complete` / `record_execution`). The `record_execution` or `mark_job_complete` call must be structured so that a duplicate write from a zombie worker raises a constraint violation, which propagates as an exception and prevents the result from being returned as success. This is the only boundary that is atomic with respect to both workers simultaneously and does not depend on Python scheduling, thread visibility, or network timing.

Concretely: `mark_job_complete` should execute something equivalent to `INSERT INTO job_completions (job_id, execution_id) VALUES (?, ?) ON CONFLICT (job_id) DO NOTHING` returning rows affected, or `UPDATE jobs SET status='complete' WHERE job_id=? AND status='pending'` checking rows affected = 1. If rows affected = 0, the worker must treat the job as already completed and not return a success result.

**Fix for Finding 2 (structural — dispatch must move inside the transaction or be idempotent):**

For jobs with external side effects, either: (a) the side effect must be idempotent and the `execution_id` must be passed to the external system as an idempotency key so a duplicate call is a no-op, or (b) `_dispatch` must be moved to after the DB write that establishes exclusive ownership, so that only the worker that wins the DB constraint race proceeds to the external call. The current ordering — dispatch then record — means the external action is unguarded.

**Fix for Finding 1 (defense in depth — signal layer, not a replacement for DB fix):**

The heartbeat thread should set a shared `threading.Event` (e.g., `lock_lost_event`) when `result == 0`. The main thread should check this event at the boundary between `_dispatch` and the DB transaction. This reduces the duplicate-dispatch window in practice but is explicitly not a substitute for the DB-layer constraint — it must be documented as defense-in-depth only.

---

**Verdict:** NOT APPROVED

The zombie-write scenario is live in this code as submitted. A GC pause or slow external call beyond LOCK_TTL produces two workers executing the same job with no architectural boundary that prevents both from completing. The heartbeat correctly detects lock loss but has no channel to the main thread. The DB write boundary has no uniqueness enforcement. The external side effect in `_dispatch` runs before any durable ownership record exists. None of the checklist items — atomic Lua scripts, Pydantic validation, exponential backoff, transaction wrapping — address this failure path. The fix must be a write-time uniqueness constraint at the database layer.
