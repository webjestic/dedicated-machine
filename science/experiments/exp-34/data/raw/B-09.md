**Findings**

**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Possible**

**Issue name:** Undetected lock loss / zombie write at DB boundary

**Failure path, step by step:**

1. Process A acquires `job:claim:{job_id}` with token `T_A`, TTL = 60s. Heartbeat thread starts, renewing every 15s.
2. Process A enters `_dispatch()`. A GC pause, slow external HTTP call, or I/O stall holds the process for >60s without the heartbeat firing in time — or the heartbeat fires but the network call to Redis itself stalls, so the renewal does not land before TTL expiry.
3. The Redis key expires. The lock is now vacant.
4. Process B runs `SET NX EX`, acquires the same key with token `T_B`, and begins executing the same job.
5. Process A's heartbeat eventually fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T_B` ≠ `T_A`, so the script returns `0`.
6. In `_renew()`:
   ```python
   if result == 0:
       return          # thread exits silently
   ```
   The thread exits. No exception is raised. No event is set. The `stop_event` is never set by the heartbeat thread — it is only ever set by the `finally` block in the main thread.
7. Process A's main thread is still executing. It has **zero visibility** into the heartbeat's exit. There is no shared flag checked between `_dispatch()` and the `db.transaction()` block. The `with self._claim_job(...)` context manager yields control to the body and does not re-check lock ownership at any point after yield.
8. Process A completes `_dispatch()`, enters `with self.db.transaction()`, calls `db.record_execution(...)` and `db.mark_job_complete(...)`.
9. Process B does the same.
10. Both writes land. The job executes twice. For a payment charge, alert fire, or notification send that happened inside `_dispatch()`, the external action has already been taken twice before either DB write.

**Why `threading.Event` signaling alone does not fix this:**

Even if `_renew()` called `stop_event.set()` on `result == 0`, that signal is advisory. The main thread would need to poll it — at minimum between `_dispatch()` and the transaction, and ideally inside `_dispatch()` at every external call boundary. That still leaves a window: the stall that caused lock expiry may have already concluded before the heartbeat fires, meaning the main thread is past the check point before the signal arrives. More fundamentally, setting an event does not make the DB write conditional on lock ownership. It is a detection mechanism, not a prevention mechanism.

**Required fix — fencing token at the DB write boundary:**

The architectural fix must live at the database layer, not the threading layer, because that is the boundary where the duplicate write actually occurs.

Concretely:

- When Process A acquires the lock, the lock value (token `T_A`) or a monotonically increasing version counter associated with the job must be passed through to the DB write.
- The DB write must be conditional: `UPDATE jobs SET status='complete', locked_by=? WHERE job_id=? AND (locked_by IS NULL OR locked_by=?)` — or equivalently, an optimistic concurrency check on a `version` column incremented atomically.
- If Process B has already written with `T_B`, Process A's conditional write fails (0 rows affected). Process A detects this, does not return a success result, and the duplicate is prevented at the only boundary that matters.

This is the same pattern as Redlock's recommendation to use fencing tokens with storage backends: the token acquired from the lock service is presented to the storage system, which rejects writes from stale token holders. The Redis lock coordinates *who tries to write*; the DB conditional write enforces *who succeeds*.

A secondary defense (not a substitute) is to have `_renew()` set a dedicated `_lock_lost_event` on `result == 0`, and have the main thread check it before entering the transaction:

```python
if self._lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution")
```

This closes the window for the common case but does not handle the race where the main thread passes the check and then stalls again before the write commits. The DB-level fencing token closes that window unconditionally.

**Finding 2: Heartbeat Can Miss Renewal Window Under Thread Starvation or Redis Latency**

**Issue name:** Renewal interval provides insufficient margin; no missed-renewal detection

**Failure path:**

- `CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. Under normal conditions, four renewal opportunities exist per TTL window.
- `stop_event.wait(timeout=15)` is a blocking call. If the Python process is under GC pressure or the OS scheduler deprioritizes the daemon thread, the wait may return late.
- The renewal call itself is a synchronous Redis round-trip. If Redis is slow (network blip, replication lag, connection pool exhaustion), the call blocks. During this block the heartbeat thread is not renewing.
- There is no detection of a missed renewal cycle — no "I expected to renew at T+15 but it is now T+45" check.

**Required fix:**

- Reduce `RENEWAL_INTERVAL` to `CLAIM_TTL / 3` or less (here: ≤20s, preferably ≤15s with a tighter TTL).
- Record the wall-clock time of the last successful renewal. If the heartbeat thread detects that `now - last_renewal > CLAIM_TTL * 0.8`, treat the lock as lost and signal accordingly.
- Use a non-blocking Redis client or set an explicit socket timeout so a hung Redis call does not silently consume an entire renewal window.

**Finding 3: Release Script Executes After Lock May Have Expired — Releases Another Worker's Lock**

**Issue name:** Stale release deletes a valid lock held by a successor worker

**Failure path:**

1. Process A's lock expires. Process B acquires the same key with token `T_B`.
2. Process A's `finally` block runs: `stop_event.set()`, `t.join(...)`, then `self._release_script(keys=[claim_key], args=[token_A])`.
3. The Lua script checks `GET job:claim:{job_id} == T_A`. At this point the key holds `T_B`, so the script returns `0` and does not delete — **this part is correct**.
4. However: if Process A's lock had *just* expired and Process B has not yet acquired it (the window between expiry and B's SET NX), and Process A's release script fires in that window, it correctly does nothing (key is gone). This case is safe.
5. The dangerous case is not in the release script itself but in the combination with Finding 1: Process A believes it still holds the lock (main thread has no signal), completes work, and the release script's no-op return is silently swallowed — no exception, no log at WARNING or ERROR level indicating that the release found a foreign token. The operator has no signal that a zombie execution occurred.

**Required fix:**

- The release script returning `0` must be treated as an ERROR-level event, not silently ignored. Log it at ERROR with job_id and token. This does not prevent the duplicate but makes it detectable in on-call.
- Structurally, this is a symptom of Finding 1; the fencing token fix at the DB layer is the prevention mechanism.

**Finding 4: Test Suite Does Not Cover the Failure Mode**

**Issue name:** Heartbeat-exit / zombie-write path is untested

**Failure path:**

- All tests mock `register_script` to return a mock with `return_value=1`. The `result == 0` branch in `_renew()` is never exercised.
- There is no test that simulates lock expiry mid-execution (script returns 0 mid-run) and verifies that the main thread does not proceed to the DB write.
- `test_claim_release_uses_lua_script` verifies the release script is called but does not verify behavior when the script returns 0.

**Required fix:**

- Add a test: mock `_renew_script` to return `0` on first call; simulate a slow `_dispatch` (or just let the heartbeat fire); assert that `db.record_execution` is **not** called.
- This test will currently fail, which is the correct outcome — it documents the gap and gates the fix.

---

**Verdict:** NOT APPROVED

The primary defect is a structural zombie-write vulnerability: the heartbeat thread exits silently on lock loss, the main thread has no visibility into this exit, and there is no fencing token or optimistic concurrency check at the database write boundary to prevent a stale process from completing a duplicate write. The checklist item "Atomic claim renewal via Lua script (token-verified)" describes the renewal script correctly but does not acknowledge that a `result == 0` from that script — the signal that the lock has been lost — is silently discarded. The DB transaction boundary has no awareness of lock state. This is the exact failure pattern that produces duplicate charges and double-fired events in production. The fix must be implemented at the DB write boundary (fencing token / conditional update), with the threading signal as a secondary defense.
