**Findings**

---

**1. Silent Heartbeat Exit — Main Thread Has No Visibility Into Lock Loss**

**Failure path:**

Process A acquires `job:claim:{job_id}` with token `T1`, TTL = 60s. A GC pause or slow external call (e.g., `_dispatch` calling a slow downstream service) holds Process A for longer than 60 seconds. The lock expires in Redis. Process B calls `SET job:claim:{job_id} T2 NX EX 60` — it succeeds. Process B now holds the lock and begins executing the same job.

Process A's heartbeat fires at the next `RENEWAL_INTERVAL` boundary. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2`, not `T1`. The script returns `0`. The `_renew` function sees `result == 0` and executes `return` — the thread exits cleanly with no exception, no signal, no side effect visible outside the thread.

Process A's main thread is blocked in `_dispatch` or has returned from it and is now entering `self.db.transaction()`. It has no join, no poll, no shared flag checked at any point in `execute()`. The `with self._claim_job(...)` block does not expose the heartbeat's exit state to the `yield` site. The main thread proceeds to `self.db.record_execution(...)` and `self.db.mark_job_complete(...)` — writing a completion record for a job that Process B is also executing.

Both processes complete the job. Both write to the database. For a job that sends a notification, charges a payment, or fires an external event, this is a duplicate execution.

**What the `stop_event` does and does not do:**

`stop_event` is set in the `finally` block of `_claim_job` after `yield` returns — i.e., after the main thread has already finished executing. It is a shutdown signal sent *to* the heartbeat thread *after* execution completes. It provides no signal *from* the heartbeat thread *to* the main thread. The communication channel is one-directional and too late. The checklist item "Background renewal thread scoped to claim context — stop_event as timer" accurately describes what it does; it does not describe what it cannot do.

**Required fix — two layers, both necessary:**

*Layer 1 — Threading signal (necessary but not sufficient):* Add a `lock_lost` event set by the heartbeat on `result == 0`. Check this event at the boundary between `_dispatch` returning and the database write:

```python
if lock_lost.is_set():
    raise ClaimError("Lock lost during execution — aborting before DB write")
```

This closes the window between heartbeat exit and the DB write for the common case. It is a best-effort defense. It does not close the window if the main thread is already inside `db.transaction()` when the heartbeat fires, or if the check-then-write is interrupted by another pause.

*Layer 2 — Fencing token at the database write boundary (necessary and sufficient):* The `token` acquired at lock time is a per-execution UUID. Pass it as a fencing token to the database write. The database must enforce uniqueness or conditional update on this token — either a unique constraint on `(job_id, claim_token)`, or an optimistic concurrency check that rejects the write if a completion record for this `job_id` already exists with a different token. This must be enforced inside `db.transaction()`, not before it.

The reason this must live at the database layer: the database write is the operation with external consequence. Any check performed before the write — including `lock_lost.is_set()` — is subject to a TOCTOU race. Only a constraint enforced atomically *at* the write boundary, by the system that owns the state, can provide a hard guarantee. Redis lock loss detection tells you something went wrong; the database constraint is what actually prevents the duplicate write.

The current `db.get_execution()` idempotency check is also subject to this race: both Process A and Process B can pass the `existing is None` check before either writes, then both write. This is a separate but related TOCTOU gap that the same fencing token constraint closes.

**2. Idempotency Check Is a Non-Atomic Read-Then-Write**

**Failure path:**

Process A and Process B both acquire the lock in sequence (not simultaneously — this does not require simultaneous acquisition). Process A acquires, calls `db.get_execution(job_id)` → `None`, proceeds to `_dispatch`. Process A's lock expires. Process B acquires, calls `db.get_execution(job_id)` → `None` (Process A has not yet written), proceeds to `_dispatch`. Both processes now pass the idempotency check and both proceed to write. This is a distinct path from Finding 1 and can occur even if a `lock_lost` signal is added, if the signal fires after the idempotency check but before the write.

**Required fix:**

The `record_execution` write must be conditional — either an `INSERT ... WHERE NOT EXISTS` or a unique constraint on `job_id` that causes the second write to fail with a constraint violation. The transaction must treat a constraint violation as a signal to abort and return "already completed," not as an unhandled exception. This is the same database-layer fencing described in Finding 1; it closes both races simultaneously.

---

**3. Test Suite Does Not Cover the Failure Mode**

The tests mock `register_script` to return a mock with `return_value=1` unconditionally. No test simulates `result == 0` from the renewal script and verifies that the main thread aborts before the DB write. No test simulates a lock expiry mid-execution. The test suite validates the happy path and the "already claimed" path; it does not validate the invariant that matters: that a process which loses its lock does not write to the database.

**Required fix:**

Add a test that sets the renewal script mock to return `0` after the first call, allows `_dispatch` to proceed, and asserts that `db.record_execution` and `db.mark_job_complete` are never called. This test will fail against the current implementation, correctly documenting the gap. After the architectural fix is in place, this test becomes the regression guard.

---

**Verdict:** NOT APPROVED

The service has correct lock *acquisition* mechanics — `SET NX EX`, token-verified Lua release, exponential backoff. The structural gap is at the lock *loss* boundary: the heartbeat thread exits on token mismatch with no signal to the main thread, and the database write boundary has no fencing mechanism to reject a write from a process that no longer holds the lock. Until a fencing token or conditional write constraint is enforced atomically at `db.transaction()`, a process pause beyond CLAIM_TTL produces a duplicate execution. The threading signal in Finding 1 is a defense-in-depth measure; the database constraint is the required architectural control.
