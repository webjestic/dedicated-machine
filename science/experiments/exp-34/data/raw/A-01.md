## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie-write via undetected lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s without yielding to the heartbeat interval in a way that saves it — or the heartbeat fires but Redis is momentarily unreachable, the renewal fails, and the TTL expires naturally.
3. More precisely: the main thread stalls for >60s. The heartbeat fires at t=15, t=30, t=45 and succeeds. At t=60 the main thread is still stalled. The heartbeat fires at t=60 but the lock has already expired at exactly t=60 (race). Or: the main thread stalls across a renewal window — the heartbeat fires, Redis is slow, the TTL lapses before `EXPIRE` lands.
4. Lock expires. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET` returns `T2`, not `T1`. Script returns `0`. The `_renew` function hits `if result == 0: return` — thread exits silently. No exception is raised. No flag is set. No signal is sent to the main thread.
6. Worker A's main thread resumes from its stall. It has no awareness the heartbeat thread exited. It continues through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
7. Worker B is also executing the job. Both workers complete `_dispatch()` and attempt to commit. Depending on DB isolation and timing, both commits may succeed. The job executes twice.

**For a job that sends a payment, fires an alert, or triggers an external webhook, this is a correctness violation.**

The `stop_event` is only used to signal the heartbeat to stop — it is never written *by* the heartbeat to signal the main thread. The channel is unidirectional and the failure direction is unmonitored.

**Required fix — two layers:**

*Layer 1 (necessary but not sufficient): Thread signaling*
Add a `lock_lost_event = threading.Event()`. In `_renew`, on `result == 0`, set `lock_lost_event` before returning. In `_claim_job`, after `yield` returns (i.e., after the job body executes), check `lock_lost_event.is_set()` and raise if set. More robustly: check it at the yield boundary so the `finally` block can abort before release. This closes the silent-exit gap but does not close the window where the main thread is mid-execution when the event is set.

*Layer 2 (architecturally required): Fencing token / optimistic concurrency at the effect boundary*
Thread signaling only catches lock loss if the main thread checks before committing. It cannot interrupt a thread mid-`_dispatch()` that is already calling an external API. The correct architectural fix is a **monotonic fencing token** (e.g., Redis `INCR` on acquisition, or a Lua script that returns the lock version). This token is passed through to every side-effecting operation. The DB write uses an optimistic concurrency check: `UPDATE jobs SET status='complete' WHERE job_id=? AND expected_token=?` — if the row was already written by Worker B with a higher token, Worker A's write fails and is discarded. External calls (payment APIs, notification services) must be idempotent with the token as the idempotency key. Without this layer, even with the threading fix, a worker that is mid-external-call when it loses the lock cannot be stopped.

**Finding 2: Idempotency Check Is Outside the Lock's Causal Guarantee**

**Issue name:** TOCTOU on completion check

**Failure path:**

1. Worker A and Worker B both attempt to claim the same job. Worker A wins the Redis lock.
2. Worker A calls `self.db.get_execution(req.job_id)` — returns `None`. Job has not been completed.
3. Worker A stalls (GC, I/O). Lock expires. Worker B acquires the lock.
4. Worker B calls `self.db.get_execution(req.job_id)` — still returns `None` (Worker A hasn't committed yet).
5. Both workers proceed past the idempotency check. Both call `_dispatch()`. Both commit.

This is a secondary path to duplicate execution. The idempotency check is a read followed by a conditional write — a classic TOCTOU. The DB transaction wraps only `record_execution` + `mark_job_complete`, not the check.

**Required fix:**
The idempotency check must be inside the DB transaction as a conditional write, not a pre-flight read. Use `INSERT ... WHERE NOT EXISTS` or an equivalent upsert with a unique constraint on `job_id` in the executions table. The unique constraint makes the second commit fail at the DB level regardless of what Redis did. This is the backstop that makes the system correct even when the locking layer fails.

**Finding 3: Release Script Executes After Lock May Have Been Reassigned**

**Issue name:** Benign-but-confirming zombie release

**Failure path:**

In the scenario from Finding 1, Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[token])`. At this point Worker B holds the lock with token `T2`. The Lua script does `GET` → `T2 ≠ T1` → returns `0`, no delete. This is correct behavior — the script is token-verified and does not delete Worker B's lock.

This is not itself a bug, but it confirms the zombie-write window: Worker A executed the job body, committed to the DB, and only then did the release script harmlessly no-op. The lock mechanism correctly refused to delete the wrong token, but the damage was already done in the execution layer.

**No fix needed at this layer** — the Lua release script is correctly written. The fix must live in Finding 1 and Finding 2.

**Finding 4: Heartbeat Cannot Interrupt an In-Progress External Call**

**Issue name:** Non-preemptive execution — thread signaling is insufficient for external side effects

**Failure path:**

Even if Finding 1's thread signaling fix is applied, consider: Worker A is inside `_dispatch()`, which calls an external payment API. The HTTP call takes 45 seconds. The lock expires at second 30 of that call. The heartbeat sets `lock_lost_event`. Worker B acquires the lock and also calls the payment API. Worker A's HTTP call completes. Worker A checks `lock_lost_event` — it is set — and raises before the DB commit. Worker A does not double-write the DB. But the external payment API was called twice. The DB is consistent; the payment processor is not.

**Required fix:**
External side effects must use the fencing token as an idempotency key passed to the external system. Payment APIs, notification services, and webhook endpoints must be called with a stable, lock-version-scoped idempotency key so that the second call is a no-op at the destination. This is an architectural requirement that cannot be solved inside `job_executor.py` alone — it must be a contract with every handler registered in `get_handlers()`.

---

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** Missing adversarial test — heartbeat death with main thread continuation

The test `test_claim_release_uses_lua_script` verifies the release path. No test simulates `_renew_script` returning `0` and verifies that the main thread detects this and aborts execution. The test suite will pass green while the zombie-write scenario is fully exploitable.

**Required fix:**
Add a test that configures `_renew_script` to return `0`, triggers a renewal cycle, and asserts that `record_execution` is never called. This test will fail against the current code (correctly) and must pass after the threading fix is applied. Add a second test that verifies the fencing token is propagated to the DB write call.

---

## Zombie-Write Scenario Confirmation

**The zombie-write scenario can occur.** The exact conditions:

- Worker A acquires the lock and enters a slow `_dispatch()` call
- The lock TTL elapses (GC pause, slow external dependency, network stall to Redis during a renewal window)
- Worker B acquires the lock and begins executing the same job
- Worker A's heartbeat detects token mismatch and exits silently
- Worker A's main thread has no signal and continues to completion
- Both workers execute the job and attempt to commit

**What is absent that would prevent it:**

1. No `lock_lost_event` or equivalent signal from heartbeat thread to main thread
2. No fencing token propagated to the DB write (no optimistic concurrency check)
3. No unique constraint backstop on the executions table enforced at commit time
4. No idempotency key passed to external handlers

Any one of layers 2 or 3 would contain the DB-level duplicate. Neither is present. Layer 4 is required to contain external side effects regardless of DB correctness.

---

**Verdict: NOT APPROVED**

The heartbeat thread exits silently on lock loss with no signal to the main thread. The main thread continues executing and committing as if the lock is held. The idempotency check is a non-atomic pre-flight read with no DB-level enforcement. Under a GC pause or slow external call exceeding LOCK\_TTL, two workers will execute the same job concurrently and both will commit. Required before approval: (1) `lock_lost_event` signaling from heartbeat to main thread, (2) fencing token with optimistic concurrency check at the DB write boundary, (3) unique constraint on job executions table as a hard backstop, (4) idempotency key contract for external handler calls, (5) adversarial test coverage for the lock-loss path.
