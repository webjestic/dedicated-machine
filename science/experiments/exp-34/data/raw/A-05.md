## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Scenario**

**Issue name:** Heartbeat-death / zombie write

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL = 60s.
2. Worker A enters `_dispatch()`. A GC pause, slow DB call, or external HTTP timeout stalls the main thread for > 60s.
3. The lock TTL expires. Redis deletes `job:claim:{job_id}`.
4. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B now holds the lock and begins executing the same job.
5. Worker A's heartbeat thread fires at the next `RENEWAL_INTERVAL` boundary. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2`, which does not equal `T1`. The script returns `0`.
6. The `_renew` function sees `result == 0` and executes `return`. The thread exits. **No signal is sent to the main thread. No exception is raised. No flag is set.**
7. Worker A's main thread is unaware the heartbeat exited. It resumes after the stall, completes `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
8. Worker B independently completes the same sequence.
9. Both workers have now fully executed the job. For a job that sends a payment, fires an alert, or triggers an external action, this is a correctness violation.

**The `stop_event` does not help here.** `stop_event` is only set in the `finally` block of `_claim_job`, which runs *after* the job body completes. It is a shutdown signal from main→heartbeat, not a failure signal from heartbeat→main. The communication channel in the failure direction does not exist.

**Required fix and layer:**

The heartbeat thread must communicate lock loss back to the main thread in a way that causes the main thread to abort. There are two layers of fix, and both are required for a complete solution:

*Layer 1 — Thread signaling (necessary but not sufficient):*
Add a shared `threading.Event` (e.g., `lock_lost_event`) that the heartbeat sets when `result == 0`. After every yield point in the job body — or via a checked wrapper — the main thread must observe this event and raise. The simplest structural approach: check `lock_lost_event` at the start of the `finally` block and raise before attempting release, but this is too late. The check must occur *before* side-effectful work completes. The practical place is immediately after `_dispatch` returns and before the DB transaction commits:

```python
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting to prevent duplicate")
```

This reduces the window but does not eliminate it: the main thread can lose the lock, the heartbeat can exit, and the main thread can complete the DB transaction before it checks the event — particularly if `_dispatch` itself is the slow call and the check comes after.

*Layer 2 — Fencing token / optimistic concurrency at the DB write (necessary for correctness):*
The only architectural boundary that actually prevents the duplicate write is making the DB commit conditional on the worker still being the legitimate lock holder at commit time. This requires a **fencing token** (monotonic counter or lock generation number) issued at lock acquisition time and verified at write time.

Concretely: when Worker A acquired the lock, it received generation `N`. The DB write includes `WHERE lock_generation = N AND job_id = ?`. When Worker B acquired the lock after expiry, it received generation `N+1`. Worker A's commit either fails the conditional write (if the DB enforces the generation check) or is rejected by a compare-and-swap on the execution record. Redis does not natively issue monotonic fencing tokens; this requires either a Redis counter incremented atomically at acquisition time, or Redlock with a fencing token extension, or delegating the authority check to the DB layer with an optimistic lock column.

Without Layer 2, a sufficiently unlucky timing — stall ends, heartbeat hasn't fired yet, main thread commits before the next heartbeat interval — produces the duplicate regardless of Layer 1.

**Finding 2: Lock Release After Lock Loss Executes Unconditionally in Finally Block**

**Issue name:** Post-expiry release attempt on already-lost lock (minor, but confirms the gap)

**Failure path:**

In the scenario above, after Worker A's main thread completes, the `finally` block runs `stop_event.set()`, joins the heartbeat thread (which has already exited), then calls `self._release_script(keys=[claim_key], args=[token])`. At this point `job:claim:{job_id}` holds `T2` (Worker B's token). The Lua script correctly does nothing (token mismatch → returns 0). This part is safe. However, this also means the code has no awareness that it just completed a job it no longer held the lock for. The release returning 0 is silently discarded — same structural problem as the heartbeat: a meaningful signal (0 = you didn't own the lock) is returned and ignored.

**Required fix:** The release result should be checked. If it returns 0, log at ERROR level and emit a metric. This does not prevent the duplicate but makes the incident detectable.

**Finding 3: Idempotency Check Is Not Atomic With Execution — TOCTOU Race**

**Issue name:** Check-then-act on job completion state

**Failure path:**

1. Workers A and B both acquire the claim in sequence (B after A's lock expires, per Finding 1), or in a variant: both workers pass the `SET NX` race within the retry window due to a Redis failover that loses the key.
2. Both workers call `self.db.get_execution(req.job_id)` and both receive `None` — the job has not yet been recorded as complete.
3. Both workers proceed to `_dispatch()`.
4. Both workers enter `self.db.transaction()` and attempt to write.

The idempotency check and the execution record are not in the same atomic unit. A unique constraint on `(job_id)` in the executions table would cause one of the two `record_execution` calls to fail with an integrity error, but this is not shown in the code and not enforced here. Without a DB-level unique constraint, both writes succeed.

**Required fix:** The DB layer must enforce a unique constraint on `job_id` in the executions table (or equivalent). The `record_execution` + `mark_job_complete` transaction must be conditioned on the job not already being complete — i.e., the idempotency check and the write must be in the same transaction with a `SELECT FOR UPDATE` or equivalent row lock, not a separate read followed by a write.

**Finding 4: Renewal Interval Allows Lock Expiry Between Heartbeats**

**Issue name:** Renewal window exceeds safe margin

**Failure path:**

`CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. The heartbeat renews every 15s, resetting to 60s. Under normal operation this is fine. However, `stop_event.wait(timeout=self.RENEWAL_INTERVAL)` means the first renewal fires 15s after thread start, not 15s after acquisition. If the thread starts 1s after acquisition, the first renewal is at T+16s from acquisition — still well within 60s. The margin is adequate under normal scheduling.

The risk is not the interval arithmetic — it is that a single missed renewal (thread descheduled, Redis timeout) drops the TTL to whatever remains. With a 60s TTL and 15s interval there are three renewal opportunities before expiry, which is reasonable. This is not a blocking issue but the values should be documented as a ratio constraint: `RENEWAL_INTERVAL < CLAIM_TTL / 3`.

**Required fix:** Document the invariant. No code change required for this finding alone.

## Zombie-Write Scenario Confirmation

**The zombie-write scenario can and does occur under this implementation.**

**Trigger:** Any process stall (GC pause, slow external call, thread descheduling) that causes the main thread to be inactive for longer than `CLAIM_TTL` (60 seconds).

**Mechanism:** The heartbeat thread detects lock loss via the Lua script returning 0 and exits silently. The main thread has no visibility into this exit. It continues executing, completes the job, and commits the result — after a second worker has already acquired the lock and begun or completed the same job.

**What is missing:** A fencing token or monotonic generation counter that is verified at the point of the DB write, making the commit conditional on the worker still being the authoritative lock holder. Thread signaling alone (threading.Event from heartbeat to main) reduces the window but cannot close it, because the check can be bypassed by timing: the main thread can complete its side-effectful work in the interval between lock expiry and the next heartbeat firing.

---

## Verdict: NOT APPROVED

The implementation contains a zombie-write vulnerability. Two workers can execute the same job concurrently and both commit results. The required fix is a fencing token issued at lock acquisition time and verified atomically at DB write time, combined with a DB-level unique constraint on job execution records and heartbeat-to-main-thread signaling as a defense-in-depth measure. None of these are present.
