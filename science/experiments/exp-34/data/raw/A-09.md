## Findings

**Finding 1: Silent Heartbeat Death — Main Thread Continues Executing After Lock Loss**

**Issue name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()` — a slow external call, GC pause, or I/O stall holds the process for >60s. The lock expires naturally in Redis.
3. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B passes the idempotency check (`get_execution` returns None, job not yet recorded). Worker B begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`. The `_renew` function hits `if result == 0: return` and exits silently. The `threading.Event` `stop_event` is never set; the main thread has no signal, no exception, no shared flag.
5. Worker A's main thread resumes from `_dispatch()`. It proceeds to `self.db.transaction()`, calls `record_execution` and `mark_job_complete`. Worker B does the same.
6. Both workers complete the job. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the action has already executed twice before either worker reaches the DB transaction. The DB transaction's uniqueness constraint (if any) may prevent double-write to the database, but the external side effect is already duplicated.

**The exact gap:** `_renew` returns on `result == 0` with no mechanism to propagate that signal to the `with self._claim_job(...)` block. The `stop_event` is a one-way signal from main→heartbeat only. There is no reverse channel.

**Required fix and layer:**

*Layer: heartbeat thread + claim context manager, with an architectural backstop at the execution boundary.*

Minimum fix — add a shared flag so lock loss is observable by the main thread:

```python
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

And in the `finally` / post-yield path, check before proceeding:

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    if lock_lost.is_set():
        raise ClaimError(f"Lock lost during execution of job {job_id}")
    self._release_script(keys=[claim_key], args=[token])
```

**Why this is necessary but not sufficient:** The `yield token` block runs `_dispatch` and the DB transaction *before* the `finally` block checks `lock_lost`. If lock loss happens mid-`_dispatch`, the external side effect (payment, notification, API call) has already fired before the flag is checked. The `finally` check prevents the DB write from completing, which is useful for idempotency on retry, but it does not un-ring the bell on the external action.

**Required architectural fix — fencing token / optimistic concurrency at the execution boundary:**

The only correct fix for the zombie-write class is to make the external action itself conditional on lock ownership at the moment of execution. This requires one of:

- **Fencing token passed through to side-effect boundary:** The token acquired at lock time is passed to `_dispatch` and to any external system that can reject a stale token (monotonically increasing counter preferred over UUID, since UUIDs provide no ordering guarantee). The external system rejects writes from a token it has already superseded.
- **Optimistic concurrency on the DB record:** Before committing `record_execution`, assert inside the same DB transaction that no execution record exists AND that the Redis lock is still held (or use a DB-level advisory lock with the same token). If the assertion fails, roll back and raise.
- **Monotonic lock version in Redis:** Replace the UUID token with a Redis-incremented counter (`INCR job:claim:version:{job_id}`). Pass the version number as the fencing token. Any downstream write includes the version; the DB enforces `WHERE version = N` on insert, failing atomically if a higher version has already written.

The `threading.Event` signal alone is Tier 0.5: it closes the double-DB-write path but does not prevent the double side-effect inside `_dispatch`.

**Finding 2: Idempotency Check Is Outside the Lock's Validity Window**

**Issue name:** TOCTOU on idempotency guard

**Failure path:**

1. Worker A acquires lock, checks `get_execution` → None, begins `_dispatch`.
2. Lock expires (as above). Worker B acquires lock, checks `get_execution` → still None (A hasn't written yet), begins `_dispatch`.
3. Both workers pass the idempotency check. Both execute.

**Note:** This is a consequence of Finding 1, not an independent bug — but it means the idempotency check provides no protection against the zombie-write scenario. It only protects against re-queuing a *completed* job, not a *concurrently executing* one.

**Required fix:** The idempotency check must be inside a DB transaction that also writes an "in-progress" marker with the lock token, and the final commit must assert that marker still matches. This collapses the TOCTOU window to the DB transaction boundary.

**Finding 3: Release Script Executes After Lock Has Already Expired and Been Reacquired**

**Issue name:** Stale release deletes a live lock owned by another worker

**Failure path:**

1. Worker A's lock expires. Worker B acquires the lock with token `T2`.
2. Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET job:claim:{job_id}` → `T2` ≠ `T1` → returns `0`, does not delete. ✓

**Assessment:** The Lua release script correctly handles this case. No bug here — the token check prevents the stale release. This is the one thing the implementation gets right atomically.

**Finding 4: Heartbeat Join Timeout Does Not Prevent Release on Slow Renewal**

**Issue name:** Release races with in-flight renewal

**Failure path:**

1. `stop_event.set()` fires. Heartbeat is mid-execution of `_renew_script` (network call to Redis).
2. `t.join(timeout=RENEWAL_INTERVAL + 1)` = 16s. If the Redis call hangs (network partition), `join` times out and returns without confirming thread exit.
3. `_release_script` executes. Then the hung `_renew_script` call returns and attempts to extend the TTL — but since the key is already deleted (or reacquired by another worker), the Lua script's token check handles it correctly.

**Assessment:** The token-verified renewal script prevents the renewal from extending a lock that has been released and reacquired. Low severity given the Lua guard, but the `join` timeout silently swallowing a hung thread is worth noting for observability.

**Finding 5: Test Suite Does Not Cover the Zombie-Write Scenario**

**Issue name:** Missing test for lock-loss-during-execution

The test `test_execute_success` mocks `_renew_script` to always return `1`. There is no test that simulates `_renew_script` returning `0` mid-execution and asserts that the main thread does not proceed to `_dispatch` or `record_execution`. The checklist item "Atomic claim renewal via Lua script (token-verified)" is checked, but the behavioral consequence of a `0` return is untested.

**Required fix:** Add a test that configures `_renew_script` to return `0` after the first call, injects a delay in `_dispatch`, and asserts that `record_execution` is not called and a `ClaimError` is raised.

---

## Zombie-Write Scenario Determination

**The zombie-write scenario can occur.** The exact conditions:

- Worker A holds the lock and enters a slow operation (GC pause, slow HTTP call, slow DB query) exceeding `CLAIM_TTL` (60s)
- The lock expires; Worker B acquires it and begins executing the same job
- Worker A's heartbeat detects token mismatch, returns silently, sets no flag
- Worker A's main thread resumes, completes `_dispatch`, writes to DB
- Both workers execute the job's side effects

The architectural boundary that would prevent it — a fencing token propagated to the side-effect boundary, or optimistic concurrency on the DB write conditioned on lock ownership — is absent. The `threading.Event` signal path from heartbeat to main thread does not exist in the current code.

---

## Verdict: NOT APPROVED

The implementation has correct atomicity on acquisition (`SET NX EX`), correct token-verified release (Lua), and correct token-verified renewal (Lua). The structural gap is that the renewal script's `0` return on token mismatch is a dead signal — it exits the heartbeat thread with no observable effect on the main execution path. Under a GC pause or slow external call exceeding the 60s TTL, two workers will execute the same job concurrently. For any job with external side effects, this is a correctness violation.

**Required before approval:**

1. `threading.Event lock_lost` reverse-signal from heartbeat to main thread, checked in `finally` before DB write — closes the double-DB-write path
2. Fencing token or optimistic concurrency at the `_dispatch` / side-effect boundary — closes the double-execution path
3. Test coverage for the lock-loss-during-execution scenario
