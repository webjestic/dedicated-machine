## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write is Live**

**Issue name:** Undetected lock loss — zombie write via process pause

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. The heartbeat thread starts.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the process for >60 seconds.
3. During the stall, the heartbeat fires at t=15s, t=30s, t=45s — all renewals succeed. At t=60s the stall is still in progress. The heartbeat fires at t=60s but the key has already expired (or expires between the `GET` and `EXPIRE` in the Lua script — see Finding 2 for why this is a minor secondary concern). The renewal returns `0`. The heartbeat thread executes `return` and exits silently. No exception is raised. No event is set. No flag is written.
4. Worker B, observing the expired key, acquires `job:claim:{job_id}` with token `T2` and begins executing the same job.
5. Worker A's stall ends. Its main thread resumes inside the `with self._claim_job(...)` block. It has no mechanism to detect that the heartbeat thread has exited. `stop_event` is never set by the heartbeat — it is only set in the `finally` block of the context manager, which has not yet been reached. The main thread proceeds: `db.get_execution()` returns `None` (Worker B has not yet committed), `_dispatch()` runs, the database transaction commits.
6. Worker B completes the same path independently.
7. **Result:** the job executes twice. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, this is a correctness violation.

The idempotency check (`db.get_execution`) does not close this window. It is a read before the write. Both workers can pass through it before either commits. The database transaction protects record consistency but does not prevent `_dispatch()` — which is where the external side effect lives — from running twice.

**Required fix — two layers, both necessary:**

*Layer 1 — thread coordination (necessary but not sufficient):*
The heartbeat must communicate lock loss to the main thread. A `threading.Event` (`lock_lost_event`) set on `result == 0` is the correct intra-process signal. The main thread must check this event at a safe yield point — minimally before `_dispatch()`, and ideally the context manager itself should raise on exit if lock loss was detected. This is the correct fix for the coordination gap, but it only catches cases where the heartbeat fires and detects the loss before the main thread reaches the critical section. It does not protect against a stall that ends *after* the lock has expired but *before* the next heartbeat interval fires.

*Layer 2 — fencing token / optimistic concurrency at the effect boundary (required for full correctness):*
The only architectural boundary that fully closes the zombie-write scenario is a monotonic fencing token carried through to the resource being mutated. The Redis lock token (`uuid4`) is not monotonic and is not presented to the downstream system. The correct pattern: use a monotonic counter (e.g., Redis `INCR` on a version key, or a database sequence) as the lock token. Pass that token as a precondition to the database write — `UPDATE jobs SET status='complete', version=N+1 WHERE job_id=X AND version=N`. If Worker A's write arrives after Worker B has already incremented the version, the precondition fails, the write is rejected, and the duplicate is detected at the effect boundary rather than relying solely on the lock. This is the only fix that holds when the process stall exceeds the heartbeat interval and the lock has already been re-acquired by another worker before the heartbeat detects it.

The `threading.Event` fix alone is Tier 0.5: it reduces the window but does not eliminate it. The fencing token fix closes it structurally.

**Finding 2: Prior Review's Atomicity Concern — Dismissed as Primary Risk, Valid as Secondary**

**Issue name:** Lua script atomicity — GET + EXPIRE sequencing

**Assessment:**

The prior review's concern is technically misstated. Lua scripts in Redis execute atomically with respect to other Redis commands — the Redis event loop is single-threaded, and no other command can interleave between the `GET` and `EXPIRE` within a single `EVAL` call. "High Redis connection saturation" does not break Lua atomicity; it may delay when the script is scheduled, but once execution begins, it is uninterrupted. The concern as stated — that saturation creates an interleaving window inside the script — is not a real failure path. **The prior atomicity concern is dismissed.**

However, there is a legitimate edge case adjacent to this script: if the key expires at the OS/Redis level between the `GET` returning the matching token and the `EXPIRE` executing, the `EXPIRE` will return `0` (key does not exist), which is indistinguishable in the script's return value from a token mismatch. In both cases `result == 0` is returned. This means the heartbeat correctly exits in this case — the failure mode is not a false-positive renewal, but a false-negative that is handled correctly. This is a minor semantic ambiguity (the caller cannot distinguish "token mismatch" from "key vanished") but it does not produce a zombie write on its own. It is a secondary concern, not the primary correctness risk.

**The primary correctness risk is Finding 1.** The Lua script is not the problem. The problem is that the main thread has no visibility into the heartbeat thread's exit.

**Finding 3: Release Races With Expired / Re-acquired Key**

**Issue name:** Post-stall release of a key owned by another worker

**Failure path:**

Worker A's lock expires. Worker B acquires the key with token `T2`. Worker A's main thread eventually reaches the `finally` block and calls `_release_script(keys=[claim_key], args=[T1])`. The Lua release script correctly checks the token — `T1 != T2` — and returns `0` without deleting. This is handled correctly by the existing code.

**Assessment:** The release script's token check prevents Worker A from evicting Worker B's lock. This is not a bug. It is noted here for completeness because it is the one place the existing design correctly handles the post-stall scenario at the Redis layer. It does not compensate for the fact that Worker A has already executed the job.

---

**Finding 4: Test Suite Does Not Cover the Failure Mode**

**Issue name:** No test for lock-loss / heartbeat-exit / main-thread-continuation scenario

The test `test_execute_claim_contention` tests Redis returning `False` on `SET NX` — clean contention at acquisition time. No test simulates the heartbeat returning `0` mid-execution and verifies that the main thread halts. This means the gap in Finding 1 is not regression-tested and will not be caught by CI if a fix is applied incorrectly.

**Required fix:** Add a test that configures the renew script mock to return `0` after the claim is acquired, injects a delay between claim acquisition and `_dispatch()`, and asserts that `_dispatch()` is not called (or that a lock-loss exception is raised). This test will fail against the current code and pass only after Finding 1 is correctly fixed.

---

## Verdict: NOT APPROVED

The zombie-write scenario is live and reachable in production. A GC pause or slow external call exceeding `CLAIM_TTL` causes the heartbeat to exit silently while the main thread continues executing. The main thread has no signal path from the heartbeat thread. Both the stalled worker and the worker that re-acquired the lock will complete `_dispatch()` and commit. For any job with an external side effect this is a duplicate operation.

The prior review's Lua atomicity concern is dismissed — it misidentifies the failure layer. The real gap is the missing coordination between the heartbeat thread and the main execution thread, and the absence of a fencing token at the effect boundary.

The fix requires both a `threading.Event` lock-loss signal (so the main thread can halt when the heartbeat detects loss) and a monotonic fencing token passed as a precondition to the database write (so that a stall ending before the next heartbeat interval still produces a detectable conflict at commit time). Neither fix alone is sufficient. The test suite must be extended to cover the lock-loss-mid-execution path before this merges.
