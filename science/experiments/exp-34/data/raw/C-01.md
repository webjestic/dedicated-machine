## Findings

---

**Finding 1: Heartbeat Thread Exits Silently on Lock Loss — Main Thread Continues Executing (PRIMARY CORRECTNESS RISK)**

**Name:** Zombie write via undetected lock loss

**Failure path — traced exactly:**

1. Process A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewal interval=15s.
2. Process A enters `_dispatch()`. A GC pause, slow external HTTP call, or database stall holds the process for >60 seconds. The lock expires naturally in Redis — no explicit release, just TTL exhaustion.
3. Process B calls `redis.set(claim_key, token, nx=True, ex=60)`. The key is gone. B acquires the lock with token `T2` and begins executing the same job.
4. Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2`, which does not match `T1`. The script returns `0`. The `_renew` function hits `if result == 0: return` and the thread exits.
5. **The thread exit is invisible to the main thread.** There is no `threading.Event`, no exception raised, no shared flag set, no channel back to the `_claim_job` context manager. The `stop_event` is only ever set by the `finally` block — it is a one-way signal from main thread to heartbeat, never the reverse.
6. Process A's main thread continues: `_dispatch()` completes, `self.db.transaction()` commits `record_execution` and `mark_job_complete`.
7. Process B, also holding a valid lock, completes the same path.
8. Both processes execute the job. For a payment charge, alert fire, or external notification, this is a duplicate operation.

**Why the idempotency check does not save you:**

The `existing = self.db.get_execution(req.job_id)` check runs *before* dispatch, under the lock. At that point, neither A nor B has written a completion record. The check passes for both workers. The race is not at the read-before-dispatch boundary — it is at the lock-loss-during-execution boundary.

**Why the database transaction does not save you:**

The transaction wraps `record_execution` + `mark_job_complete`. If both workers reach this point, one will commit first. The second may hit a uniqueness constraint (if the schema enforces it on `job_id`) — but this code does not show that constraint, does not catch that exception, and does not treat a constraint violation as a safe no-op. Even if the DB layer has a unique index, the external side effect (`_dispatch` — the payment, the notification, the alert) has already fired before the transaction. The DB transaction is not a fencing mechanism for the external action.

**Required fix — and why it must live at this layer:**

The heartbeat thread must have a bidirectional signal path to the main execution thread. When `result == 0`, the thread must not silently return — it must set a shared `threading.Event` (e.g., `lock_lost_event`) that the main thread checks at defined yield points, or it must raise into the main thread via a mechanism the main thread is actually monitoring.

Concretely:

```python
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must check `lock_lost` before and after any non-atomic operation — critically, before committing the DB transaction and before any external side effect. If `lock_lost` is set, execution must abort and the result must not be committed.

**However:** this is necessary but not sufficient. The fundamental architectural gap is that a process-pause long enough to expire the lock can occur *between* the `lock_lost` check and the external side effect. The only complete fix is a **fencing token** (monotonic counter or version number) propagated through to the external action and to the DB write, so that a stale writer's commit is rejected at the resource boundary regardless of what the writer believes about its lock state. Redis's `SET NX EX` does not provide a fencing token. The lock token (`uuid4`) is a mutual-exclusion credential, not a monotonic fence — it cannot be used to order writes. The DB layer must enforce `INSERT ... WHERE NOT EXISTS` or an optimistic-concurrency version check that makes a stale write structurally impossible to commit, not merely unlikely.

**Finding 2: Prior Review's Atomicity Concern — Dismissed**

**Name:** Alleged non-atomicity of `RENEW_CLAIM_SCRIPT` GET+EXPIRE

**The concern:** That under high Redis connection saturation, the `GET` and `EXPIRE` in the Lua script execute non-atomically, creating a window where another worker claims the key between the two operations.

**Why this is not a real failure path:**

Redis executes Lua scripts atomically with respect to all other Redis commands. This is a documented guarantee of the Redis execution model: while a Lua script is running, no other command from any client can interleave. The script is not two round-trips — it is a single atomic unit dispatched to the Redis interpreter. Connection saturation affects *queuing* of commands before execution begins; it does not cause interleaving *within* a script. A second worker's `SET NX` cannot execute between the `GET` and `EXPIRE` inside the script. The prior review's concern conflates network-layer queuing with Redis's single-threaded command execution model.

**The Lua script is correct for what it does.** The atomicity boundary holds. This is not the primary correctness risk.

**What the script correctly handles:** If Process A's lock has expired and Process B has acquired it with token `T2`, Process A's renewal attempt will see `T2 ≠ T1` and return `0` without extending B's lock. The script does not corrupt the lock state of the new owner. This is correct behavior — the problem is solely that the `0` return is not propagated to Process A's main thread.

---

**Finding 3: Release Attempt on an Unowned Lock**

**Name:** Spurious release attempt after lock loss

**Failure path:**

In the `finally` block of `_claim_job`, after `stop_event.set()` and `t.join()`, the code unconditionally calls `self._release_script(keys=[claim_key], args=[token])`. If the lock was lost (TTL expired, Process B acquired it with `T2`), the Lua release script correctly no-ops because `T1 ≠ T2`. This is safe in isolation.

However, if the main thread is not aborted on lock loss (Finding 1 is not fixed), this release call executes *after* Process A has already completed the job under a stale lock assumption. The release is a no-op, but the damage (duplicate execution) has already occurred. This is a dependent failure, not an independent one — it does not cause additional harm beyond Finding 1, but it confirms that the `finally` block has no awareness of whether the lock was legitimately held through to completion.

**Required fix:** Contingent on Finding 1's fix — if `lock_lost` is set, the `finally` block should log the lock-loss event explicitly rather than silently skipping the release, to make post-incident forensics tractable.

---

## Verdict: **NOT APPROVED**

The primary correctness risk is **Finding 1**: the heartbeat thread's silent exit on lock loss is not observable by the main execution thread. Under a GC pause or slow external call exceeding `CLAIM_TTL`, two workers will execute the same job concurrently and both will commit. The prior review's atomicity concern about the Lua renewal script is **dismissed** — Lua scripts in Redis are atomic by specification, and the concern reflects a misunderstanding of where the atomicity boundary lies.

This cannot go to production in a multi-worker deployment handling payments, notifications, or any non-idempotent external action without at minimum bidirectional lock-loss signaling, and properly with a fencing token enforced at the DB and external-action layer.
