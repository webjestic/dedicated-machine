## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token T1, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()` — a slow external call (payment gateway, alert webhook) or a GC pause holds the thread for >60s.
3. The heartbeat fires at t=15s, t=30s, t=45s — all succeed. At t=60s the lock has not yet expired because renewals kept it alive. But if the stall begins at, say, t=50s and lasts 70s, the lock expires at t=110s (50s into the stall, 10s after the last renewal at t=45s extended it to t=105s). The exact timing depends on stall onset, but the scenario is: **a stall longer than RENEWAL_INTERVAL (15s) that begins just after a renewal causes the lock to expire before the next renewal fires.**
4. Lock expires. Worker B calls `SET NX EX` on the same key, acquires it with token T2, begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns T2 ≠ T1. Script returns 0. Heartbeat thread hits `if result == 0: return` — **exits silently. No exception. No signal. No flag set.**
6. Worker A's main thread is unblocked from `_dispatch()`. It has no knowledge the heartbeat exited. It proceeds: `db.get_execution()` — returns None (B hasn't committed yet). It calls `db.record_execution()`, `db.mark_job_complete()` inside a transaction. Commits.
7. Worker B completes the same path independently. Both commit. **Job executes twice.**

**The structural gap:** `_renew()` returns on `result == 0` but sets nothing the main thread can observe. `stop_event` is write-only from the main thread's perspective — the heartbeat never writes to it. The `yield token` in `_claim_job` continues regardless.

**Required fix — two layers required:**

*Layer 1 (necessary but insufficient): Thread signaling.* `_renew()` must signal the main thread on lock loss. A second `threading.Event` (`lost_event`) written by the heartbeat and checked by the main thread is the minimum. However, this is a TOCTOU patch — it reduces the window but does not close it. The main thread can still be between the last check and a side-effecting operation when the lock is lost.

*Layer 2 (required for correctness): Fencing token / optimistic concurrency at the effect boundary.* The database write that records execution must be conditional on the lock still being held by this worker. This requires one of:

- **Fencing token passed through to the DB write:** The Redis lock version (a monotonic counter, e.g., via `INCR` at acquisition time) is stored alongside the execution record. The DB `INSERT` includes a `WHERE NOT EXISTS` guard or a conditional update that rejects a write if a higher-versioned execution for the same job_id already exists.
- **Optimistic concurrency on the job row:** The job row carries a `lock_version` column. Acquisition writes version N. The final `mark_job_complete` executes as `UPDATE jobs SET status='complete' WHERE job_id=? AND lock_version=N` — if Worker B has since written version N+1, Worker A's update affects 0 rows, and the transaction is rolled back.
- **Redis-side guard at commit time:** Before the DB transaction commits, re-verify the lock token in Redis. This is weaker (still a race between check and commit) but better than nothing if DB schema changes are not immediately feasible.

The UUID token as currently implemented is not a fencing token — it is not monotonically ordered, so the DB layer cannot distinguish "older write" from "newer write."

**Finding 2: Idempotency Check Is Not Atomic With Execution — TOCTOU on DB State**

**Issue name:** Check-then-act race on `get_execution` / `record_execution`

**Failure path:**

1. Workers A and B both acquire the claim in rapid succession (B acquires after A releases, or in the concurrent scenario above).
2. Both call `db.get_execution(job_id)` before either has committed. Both see `None`.
3. Both proceed to `_dispatch()` and then `db.record_execution()`.
4. Both commit. Duplicate execution.

This is a secondary path to duplication, independent of Finding 1. The idempotency check provides defense-in-depth only if the DB write itself is made conditional — e.g., `INSERT INTO executions ... WHERE NOT EXISTS (SELECT 1 FROM executions WHERE job_id=?)` with a unique constraint on `job_id`, causing one transaction to fail with a constraint violation.

**Required fix:** Unique constraint on `(job_id)` in the executions table. `record_execution` must use an insert that fails loudly on duplicate rather than a blind write. The executor must catch the constraint violation and treat it as "already executed" — not as an unhandled error.

**Finding 3: Release Attempt After Lock Has Already Expired — Harmless But Misleading**

**Issue name:** Phantom release on expired/stolen lock

**Failure path:**

In the zombie-write scenario, Worker A's `finally` block calls `_release_script` with token T1. By this time the key holds T2 (Worker B's token). The Lua script correctly returns 0 and does not delete the key. Worker B's lock is not destroyed.

This is not a correctness violation on its own, but the `logger.info("Claim released job_id=%s")` fires unconditionally — it logs a successful release even when the release was a no-op because the lock was lost. This will produce misleading operational signals during incident investigation.

**Required fix:** Check the return value of `_release_script`. Log distinctly: `"Claim released"` vs `"Release no-op — lock already expired or stolen"`. This is operational hygiene, not a correctness fix, but it matters at 3am.

**Finding 4: Heartbeat Join Timeout Does Not Prevent Release Race**

**Issue name:** Release proceeds despite uncertain heartbeat state

**Failure path:**

`t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), `join` returns after 16s with the thread still running. `_release_script` then executes. The thread may subsequently attempt a renewal after the release, against a key that is now either gone or held by another worker. The Lua script handles this correctly (returns 0), so there is no correctness violation here — but the join timeout creates a false guarantee that the heartbeat has stopped before release.

**Required fix:** After `join(timeout=...)`, check `t.is_alive()`. If true, log a warning — the release is proceeding with a potentially still-running renewal thread. This is defense-in-depth logging, not a blocking issue given the Lua guard.

---

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** Zombie-write scenario has no test coverage

The test mocks `register_script` to return a mock that always returns 1. There is no test that simulates `result == 0` from the renewal script and verifies the main thread's behavior. The test `test_claim_release_uses_lua_script` verifies the release script is called — it does not verify what happens when the lock is lost mid-execution.

**Required fix:** Add tests for:
- Renewal script returns 0 mid-execution → assert job does not complete (once Finding 1 is fixed)
- Concurrent execution simulation where `get_execution` returns None for both workers → assert unique constraint prevents double-record
- Heartbeat thread exit does not propagate to main thread (documents the current broken behavior as a failing test, driving the fix)

## Zombie-Write Verdict

The zombie-write scenario **can occur** and is not closed by any boundary in this code.

**Trigger:** GC pause or slow external call in `_dispatch()` exceeding `CLAIM_TTL` minus the time since last renewal (worst case: 15s stall causes lock expiry if it begins 45s into a 60s TTL with renewals at 15s intervals — but a stall of any duration >15s that begins just after a renewal will cause expiry before the next renewal).

**Path to duplicate execution:** Lock expires → Worker B acquires → Worker A's heartbeat detects token mismatch → heartbeat exits silently → Worker A continues with no signal → both workers complete the job.

**What closes it:** A fencing token (monotonic counter) threaded from lock acquisition through to the database write, with the DB enforcing that only the highest-versioned write for a given job_id commits. This makes the DB the authoritative boundary, not the Redis lock — the lock becomes a performance optimization (reduces contention) rather than a correctness guarantee.
