**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Scenario**

**Issue name:** Undetected lock loss / zombie write

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts with `RENEWAL_INTERVAL=15s`.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s.
3. The lock expires in Redis. Worker B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2`, which does not match `T1`. The script returns `0`.
5. The `_renew()` function hits `if result == 0: return` and exits. **No exception is raised. No event is set. No flag is written.**
6. Worker A's main thread is entirely unaware. `stop_event` is never set by the heartbeat thread — it is only set *by the main thread* in the `finally` block. The main thread continues executing `_dispatch()`, proceeds to `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
7. Worker B does the same.
8. Both workers complete the job. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the action fires twice.

**The structural gap:** `stop_event` is a one-directional signal — main thread → heartbeat. There is no reverse channel. The heartbeat thread's detection of lock loss is a silent return into the void. The main thread has no mechanism to observe it.

**Required fix and layer:**

The fix must live at the heartbeat-to-main-thread signaling boundary. Two components are required:

*Component 1 — Immediate: threading.Event reverse signal.*
Add a second event, e.g. `lock_lost_event = threading.Event()`. In `_renew()`, replace `return` with `lock_lost_event.set(); return`. In the main thread, check `lock_lost_event` at the earliest safe point before side effects — ideally as a guard immediately after `_dispatch()` returns and before the database transaction commits. If set, abort and raise.

```python
# In _renew():
if result == 0:
    lock_lost_event.set()
    return

# In execute(), after _dispatch() returns:
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting to prevent duplicate")
```

This closes the window between `_dispatch()` completion and `db.transaction()`. It does **not** close the window *inside* `_dispatch()` if the side effect (payment, alert) already fired before the check. That window requires the architectural fix below.

*Component 2 — Required for correctness: fencing token / optimistic concurrency at the side-effect boundary.*

The `threading.Event` signal is Tier 0.5. It reduces the duplicate window but cannot eliminate it, because the dangerous side effect lives inside `_dispatch()`, which runs before any post-dispatch check. A GC pause can occur at any point inside `_dispatch()` — including after the external call fires and before the function returns.

The correct architectural fix is a **fencing token** (monotonic counter or version number) propagated through to every external side effect:

- Replace the UUID token with a Redis-issued monotonic counter (`INCR job:fence:{job_id}`) at claim acquisition time. This becomes the fencing token.
- Pass the fencing token into `_dispatch()` and into every external call that has side effects (payment processor, notification service, event bus).
- Each downstream system accepts writes only if the presented token is ≥ the last token it recorded for that job. A write from a stale worker (lower token) is rejected.
- The database `record_execution` and `mark_job_complete` calls must also be conditioned on the fencing token: use an `UPDATE ... WHERE fence_token < $token` pattern, not an unconditional insert.

This is the only mechanism that provides a hard correctness boundary regardless of where in `_dispatch()` the pause occurs.

**Finding 2: Idempotency Check Is Outside the Lock's Validity Window**

**Issue name:** TOCTOU on completion check — idempotency guard is not re-verified at commit time

**Failure path:**

1. Worker A acquires the lock, calls `db.get_execution(job_id)` — returns `None`. Proceeds.
2. Worker A's lock expires mid-execution (same GC pause scenario). Worker B acquires the lock, also calls `db.get_execution(job_id)` — still returns `None` (A hasn't committed yet). Worker B proceeds.
3. Both workers pass the idempotency check. Both dispatch. Both commit.

Even without the GC pause: the check at the top of the `with` block and the commit at the bottom are not atomic with respect to the lock. The database transaction does not verify "I still hold the lock" at commit time. A `SELECT` then later `INSERT` with no re-validation at insert time is a classic TOCTOU.

**Required fix:**

The `mark_job_complete` write must use an **insert-with-unique-constraint** or **conditional update** pattern that fails if another worker has already written a completion record for this `job_id`. The database unique constraint on `job_id` in the executions table is the correct enforcement layer — it makes the commit itself idempotent and turns a race into a detectable error. The application must handle the unique constraint violation and treat it as "already completed." This is defense-in-depth that works even if the Redis lock fails.

**Finding 3: Release Script Executes After Lock May Already Be Expired and Re-Acquired**

**Issue name:** Stale release attempt — release fires against a lock held by another worker

**Failure path:**

1. Worker A's lock expires. Worker B acquires the same key with token `T2`.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join()`, then `_release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET job:claim:{job_id}` → returns `T2` ≠ `T1` → returns `0` and does not delete. **This part is correct.**
4. However, the log line `"Claim released job_id=%s"` fires unconditionally regardless of the script's return value. An operator reading logs sees a false confirmation of clean release.

**Required fix:**

Check the return value of `_release_script`. If it returns `0`, log a warning: `"Claim release skipped — token mismatch, lock was already re-acquired by another worker"`. This is an observability fix but it matters operationally: silent `0` returns here mask the exact scenario that produces duplicates, making post-incident diagnosis harder.

---

**Finding 4: Heartbeat Thread Join Timeout Is Silently Ignored**

**Issue name:** Unverified thread join — release proceeds even if heartbeat thread is still running

**Failure path:**

1. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds.
2. If the heartbeat thread is blocked (e.g., Redis call hanging), `join()` returns after 16s with the thread still alive.
3. `_release_script` fires. The thread is still running and may fire another `_renew_script` call after the release, against a key now held by another worker or re-created. The Lua script's token check prevents an incorrect extension, so this does not directly cause a duplicate — but it is a latent correctness assumption that should be explicit.

**Required fix:**

After `t.join(timeout=...)`, check `t.is_alive()`. If true, log an error. This surfaces the condition rather than silently proceeding.

---

**Zombie-Write Scenario Verdict:**

The zombie-write scenario **can occur** and is not closed by this implementation. The exact conditions: Worker A acquires the lock → GC/IO pause exceeds CLAIM_TTL → lock expires → Worker B acquires lock and begins execution → Worker A's heartbeat detects token mismatch and exits silently → Worker A's main thread continues with no awareness of lock loss → both workers complete `_dispatch()` and commit → job executes twice.

The `threading.Event` reverse signal (Finding 1, Component 1) reduces the window. The fencing token propagated into `_dispatch()` and downstream systems (Finding 1, Component 2) is the architectural boundary required to close it.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed and open. The heartbeat correctly detects lock loss; the result is correctly returned as `0`; the Lua scripts are correctly written. The gap is entirely in the signaling path: `result == 0` causes a silent return with no reverse notification to the main thread. The main thread proceeds through `_dispatch()` and commits against a lock it no longer holds. Closing this requires both the `threading.Event` reverse signal and a fencing token at the side-effect boundary. Neither is present.
