## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write is Live**

**Issue name:** Undetected lock loss — main thread continues executing after claim expiry

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external HTTP call, or database stall holds the main thread for >60s without yielding control back to the heartbeat loop. The heartbeat *does* fire — but `stop_event.wait(timeout=15)` is wall-clock time, so the heartbeat thread is alive and running independently of the main thread's stall. However, if the stall exceeds 60s *between* renewal windows (e.g., stall begins just after a renewal, lasts 61s), the TTL expires before the next heartbeat fires.
3. More precisely: the stall need only exceed `CLAIM_TTL` (60s) minus the time since the last renewal. Worst case: stall begins immediately after a renewal → Worker A has a full 60s window before expiry. But `_dispatch()` can block arbitrarily.
4. Lock expires. Worker B calls `SET NX EX` on the same key, acquires it with token `T2`, passes the idempotency check (`db.get_execution()` returns None — Worker A hasn't written yet), and begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`. Heartbeat thread hits `if result == 0: return` and **exits silently**.
6. Worker A's main thread is never notified. `stop_event` is never set by the heartbeat. The main thread completes `_dispatch()`, enters `db.transaction()`, calls `record_execution()` and `mark_job_complete()`.
7. Worker B does the same.
8. **Both workers complete the job. The job executes twice.**

The structural gap is in `_renew()`:
```python
if result == 0:
    return   # ← thread dies here; main thread has no idea
```
And in `_claim_job`, the `yield token` has no mechanism to be interrupted:
```python
try:
    yield token   # ← main thread runs here; heartbeat exit is invisible
finally:
    ...
```
The `stop_event` is only ever set *by the main thread* in the `finally` block. The heartbeat thread cannot signal back. This is a one-way channel.

**Required fix — two layers, both necessary:**

*Layer 1 (immediate, necessary but not sufficient):* Wire the heartbeat exit back to the main thread using a shared `threading.Event` (`lock_lost_event`). Set it in `_renew()` when `result == 0`. Check it at yield boundaries — but this is insufficient alone because `_dispatch()` is a black box that may not check any event. You cannot retrofit cooperative cancellation into arbitrary job handlers reliably.

*Layer 2 (architectural, sufficient):* Implement a **fencing token / monotonic counter** scheme. Replace the UUID token with a Redis-sourced monotonic counter (e.g., `INCR job:fence:{job_id}` at acquisition time). Pass the fence token into `_dispatch()` and require all side-effecting operations (DB writes, external calls) to include the fence token. The database write uses an optimistic concurrency check: `INSERT ... WHERE fence_token = $1` or equivalent. If Worker A's fence token is stale (Worker B has a higher one), the DB write fails. This makes the *effect* conditional on lock validity at the moment of commit, not just at the moment of acquisition.

The fencing token approach is the correct architectural boundary because it moves the safety guarantee from "did I hold the lock continuously?" (unprovable from inside the process) to "is my write the authoritative one?" (enforceable at the database layer, which is the actual consistency boundary).

Without Layer 2, any solution that relies solely on the process detecting its own lock loss is vulnerable to the scenario where detection happens *after* the side effect is already in flight.

**Finding 2: Dismissal of Prior Review's Atomicity Concern re: `RENEW_CLAIM_SCRIPT`**

**Issue name:** Lua atomicity concern — not a real failure path

**Analysis:**

The prior review's concern was that `GET` followed by `EXPIRE` in the Lua script might not be atomic under high Redis connection saturation.

This is incorrect. Redis executes Lua scripts atomically with respect to all other Redis commands. From the Redis documentation and implementation: the entire script runs in a single-threaded execution context; no other client command can interleave between `redis.call("get", ...)` and `redis.call("expire", ...)` within the same script invocation. Redis's single-threaded command processing model is the atomicity boundary here. Connection saturation affects *queuing* of script invocations — a script may wait longer to begin — but once it begins executing, it runs to completion without interleaving.

The concern would be valid if the GET and EXPIRE were two separate round-trip commands from the application layer (classic TOCTOU). But inside a Lua script, they are not. The script is the atomic unit.

**Verdict on prior review:** The atomicity concern is **dismissed**. The Lua script is correctly atomic. This is not a real failure path and is not the primary correctness risk.

The primary correctness risk is Finding 1: the heartbeat exits silently on lock loss and the main thread has no mechanism to detect or respond to that exit before completing side-effecting operations.

---

**Finding 3: Idempotency Check Does Not Close the Duplicate-Execution Window**

**Issue name:** Idempotency check is pre-execution, not post-effect — race window remains

**Failure path:**

The `db.get_execution()` check occurs *after* lock acquisition but *before* `_dispatch()`. This correctly prevents re-execution when a prior run completed and committed. However, it does not protect against the concurrent duplicate described in Finding 1, because in that scenario both Worker A and Worker B pass the idempotency check — neither has committed yet when the other checks. The check is not atomic with the dispatch. This is a secondary issue that reinforces why the fencing token at the DB write layer (Finding 1, Layer 2) is the correct fix: the uniqueness constraint must live at the commit boundary, not the pre-flight check.

**Required fix:** The `db.record_execution()` call must enforce uniqueness at the database level — a unique constraint on `job_id` in the executions table, or an `INSERT ... WHERE NOT EXISTS` / optimistic lock check. The pre-flight check is a useful optimization to avoid unnecessary work, but it cannot be the correctness guarantee.

---

## Verdict: **NOT APPROVED**

The primary correctness risk is the silent heartbeat exit with no signal path to the main thread (Finding 1). The zombie-write scenario is live: a process pause exceeding the lock TTL causes lock expiry, a second worker acquires the lock and begins execution, the first worker's heartbeat detects token mismatch and exits silently, and the first worker's main thread completes execution regardless. Both workers commit. The job runs twice.

The prior review's atomicity concern about `RENEW_CLAIM_SCRIPT` is dismissed — Lua scripts are atomic in Redis by design and connection saturation does not create interleaving within a script execution.

This PR is not approved until Finding 1 is resolved with a fencing token / optimistic concurrency check at the database write boundary, and Finding 3's uniqueness guarantee is enforced at the commit layer.
