## Findings

---

**Finding 1: Heartbeat Thread Exits Silently on Lock Loss — Main Thread Continues Executing (Primary Correctness Risk)**

**Issue name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Main thread enters `_claim_job`, yields, begins executing the job pipeline: idempotency check passes (no prior execution), `_dispatch` is called.

2. Worker A stalls — GC pause, slow external HTTP call inside `_dispatch`, kernel scheduling delay. The stall exceeds 60 seconds. Redis expires `job:claim:{job_id}` without any notification to Worker A.

3. Worker B calls `redis.set(claim_key, token, nx=True, ex=CLAIM_TTL)` — the key is gone, so the SET succeeds. Worker B acquires the lock with token `T2`, passes the idempotency check (Worker A hasn't written completion yet), and begins executing the same job.

4. Worker A's heartbeat fires at `t=75s`. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`, so the script returns `0`. The `_renew` thread sees `result == 0` and executes `return`. The thread exits.

5. Worker A's main thread is still inside the `with self._claim_job(...)` block. **There is no signaling path from `_renew` back to the main thread.** `stop_event` is only written by the main thread's `finally` block; `_renew` reads it but never sets it. The main thread has no way to observe that `_renew` exited abnormally.

6. Worker A's `_dispatch` completes. Worker A executes `self.db.transaction()`, calls `record_execution` and `mark_job_complete`. Worker B, running concurrently, does the same.

7. Both workers complete the job. The job executes twice. For a payment charge, alert dispatch, or external webhook, this is a correctness violation.

**Why the idempotency check does not save you:**

The idempotency check (`db.get_execution`) runs *before* `_dispatch`. Both workers pass it because neither has written a completion record yet. The check is not re-evaluated after dispatch. The database transaction at the end is not conditioned on the Redis lock still being held — it commits unconditionally. The lock and the database write are not in the same atomic boundary.

**Required fix — two layers, both necessary:**

*Layer 1 — Signal the main thread on lock loss (necessary but not sufficient):*

`_renew` must communicate lock loss back to the main thread. The minimal mechanism is a shared `threading.Event` (`lock_lost_event`) that `_renew` sets when `result == 0`. The main thread must check this event at a safe cancellation point — either by polling between pipeline stages or by having `_dispatch` itself be interruptible. The `_claim_job` context manager should raise on exit if `lock_lost_event` is set, preventing the `db.transaction()` from executing.

```python
# In _claim_job:
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            lock_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# After yield, in finally or via a wrapper:
if lock_lost.is_set():
    raise ClaimError(f"Lock lost during execution of job {job_id}")
```

This prevents the database write *if the stall ended before dispatch completed and the heartbeat fired in time*. It does not prevent the write if the stall occurs entirely within `_dispatch` and the heartbeat hasn't fired yet when dispatch returns.

*Layer 2 — Fencing token on the database write (necessary for correctness guarantee):*

The only hard guarantee against duplicate writes is making the database commit conditional on the worker still holding the authoritative claim. This requires a **fencing token** — a monotonic counter or version number that Redis increments on each successful acquisition (e.g., using a Redis counter alongside the lock, or using Redlock's index). The token is passed through to the database write as an optimistic concurrency condition:

```python
# Acquisition returns (token, fence):
fence = self.redis.incr(f"job:claim:fence:{job_id}")
# Database write uses conditional update:
self.db.record_execution(execution_id, job_id, job_type, result, fence=fence)
# record_execution issues: INSERT ... WHERE NOT EXISTS (SELECT 1 FROM executions WHERE job_id=? AND fence >= ?)
# or uses a unique constraint on (job_id, fence) with INSERT OR IGNORE semantics
```

With a fencing token, even if both Worker A and Worker B reach the database write, only the one with the higher fence (Worker B, the legitimate current holder) succeeds. Worker A's write is rejected at the database layer regardless of what Redis state was at the time of the write. This is the architectural boundary that closes the window — the correctness invariant is enforced at the data layer, not the coordination layer.

**Why the fix must live at both layers:** The `threading.Event` signal is a latency-dependent mitigation — it catches the failure only if the heartbeat fires before the main thread reaches the commit. The fencing token is the invariant — it catches the failure unconditionally, including the case where the process stalls for exactly one heartbeat interval and resumes just before the next renewal would have detected loss. Defense in depth requires both.

**Finding 2: Prior Review's Atomicity Concern — Dismissed**

**Claim under review:** `RENEW_CLAIM_SCRIPT` executes `GET` then `EXPIRE` as sequential Lua operations; under high Redis connection saturation these may not execute atomically, creating a window where renewal appears to succeed after another worker has claimed the key.

**Analysis:**

Redis executes Lua scripts atomically with respect to all other Redis commands. This is guaranteed by the Redis execution model: the server is single-threaded for command processing, and a Lua script runs to completion without interleaving any other client commands. No other client can issue a `SET`, `DEL`, or `EXPIRE` on `KEYS[1]` between the `GET` and the `EXPIRE` within the script. This is not a property of connection count or saturation — it is a property of the Redis execution model documented since Redis 2.6.

Connection saturation affects *queuing* — commands from multiple clients queue before execution. Once the script begins executing, it is atomic. A saturated connection pool means the script may be delayed in starting, but once started, the `GET`→`EXPIRE` sequence is uninterruptible.

**The prior review's concern is not a valid failure path.** The Lua script is correctly atomic. The `GET`→`EXPIRE` sequence cannot be split by another client's write.

The renewal script is not the primary correctness risk. The primary risk is Finding 1: the main thread has no visibility into heartbeat thread exit, and the database write is not conditioned on continued lock ownership.

---

## Verdict: NOT APPROVED

The Lua renewal script is correctly atomic — dismiss the prior review's concern. The primary correctness risk is the missing signal path from the heartbeat thread to the main thread, compounded by the absence of a fencing token on the database write. A process stall beyond `CLAIM_TTL` produces a window where two workers execute the same job and both commit. The fix requires `threading.Event` signaling for early detection and a monotonic fencing token enforced at the database write for a hard correctness guarantee. Neither is present. This goes to production as written and the duplicate-execution scenario is reachable under normal GC and I/O latency conditions.
