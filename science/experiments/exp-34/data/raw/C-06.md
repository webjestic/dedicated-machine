## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write on Lock Loss**

**Issue name:** Undetected lock loss / zombie write

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token T, TTL=60s. The heartbeat thread starts.
2. Worker A enters `_dispatch()`. A GC pause, slow DB call, or network stall holds the process for >60s.
3. The lock TTL expires. Redis evicts the key.
4. Worker B calls `SET job:claim:{job_id} <token_B> NX EX 60` — succeeds. Worker B begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes `GET` → key now holds token_B, not T → returns `0`.
6. The heartbeat thread hits `if result == 0: return` and exits. **No signal is sent to the main thread. No exception is raised. No flag is set.**
7. Worker A's main thread is still inside `with self._claim_job(...)`. It has no visibility into the heartbeat's exit. It continues: `_dispatch()` completes, `db.record_execution()` runs, `db.mark_job_complete()` runs.
8. Worker B completes the same sequence independently.
9. Both workers complete the job. The job executes twice.

**Why the existing structure does not contain this:**

`stop_event` is a one-way signal from the main thread *to* the heartbeat. There is no reverse channel. The heartbeat thread's return value is never checked — `t.join()` in the `finally` block joins after the work is already done, and `Thread.join()` does not propagate exceptions or return values. The main thread has no mechanism to observe that the heartbeat detected lock loss.

**Required fix — two layers, both necessary:**

*Layer 1 — Signal propagation (necessary but not sufficient):*
The heartbeat must communicate lock loss back to the main thread. The minimal mechanical fix is a shared `threading.Event` (`lock_lost_event`) that the heartbeat sets before returning on `result == 0`. The main thread must check this event at a safe yield point — either via a wrapper that polls between logical phases, or by having `_dispatch` be interruptible. Without this, the main thread cannot react.

*Layer 2 — Fencing token / optimistic concurrency on the write side (required for correctness):*

Signaling alone is not sufficient. Consider: Worker A's heartbeat fires at t=61s and sets `lock_lost_event`. But Worker A's `_dispatch()` already completed at t=60.5s, and `db.record_execution()` is already in flight. The signal arrives too late to prevent the duplicate write.

The correct architectural fix is to make the **terminal write unforgeable without current lock ownership**. This requires a fencing token — a monotonic counter (e.g., Redis `INCR` on a version key at claim acquisition time, or a Redlock-style fence) that is passed through to the database write. The DB write includes a conditional: `INSERT ... WHERE NOT EXISTS (SELECT 1 FROM executions WHERE job_id = ? AND fence >= ?)` or equivalent optimistic-concurrency check. A worker whose lock expired cannot produce a fence value that is ≥ the current holder's, so its write is rejected at the DB layer even if it arrives.

The DB-level idempotency check (`get_execution` before dispatch) is not a substitute: it runs *before* dispatch, not at commit time, and is not evaluated under the lock held by the writing transaction. Worker B can pass the idempotency check in the window before Worker A's write commits.

**Fix location rationale:** The signal must live in `_claim_job` / `_renew` because that is where lock state is observable. The fencing token must live at the DB write boundary because that is the only layer that can make the write conditional on lock validity at commit time — Redis and the DB are not in the same transaction domain, so Redis lock state cannot be atomically coupled to the DB write by any other means.

**Finding 2: Dismissal of Prior Review's Atomicity Concern re: `RENEW_CLAIM_SCRIPT`**

**Issue name:** Lua script atomicity — prior review concern is invalid

**Analysis:**

The prior review flagged that `GET` followed by `EXPIRE` inside `RENEW_CLAIM_SCRIPT` might not execute atomically under high Redis connection saturation.

This concern is incorrect. Redis executes Lua scripts atomically by design. From the Redis documentation and source: the entire script runs as a single atomic unit on the Redis event loop. No other client command can be interleaved between `redis.call("get", ...)` and `redis.call("expire", ...)` within the same script invocation. Redis is single-threaded with respect to command execution; while a Lua script is running, no other command executes. Connection saturation affects *queuing* of commands before execution begins, not interleaving during execution.

The `RENEW_CLAIM_SCRIPT` is not the atomicity problem. The GET+EXPIRE sequence is safe.

**The prior review's concern is dismissed.** It identified the wrong risk surface. The Lua script is correct. The failure mode is not inside the script — it is in the gap between what the script reports to the heartbeat thread and what the main thread does with that information.

---

**Finding 3: Release Races With Expired Lock — Minor Correctness Note**

**Issue name:** Post-expiry release attempts to delete a key owned by another worker

**Failure path:**

In the scenario from Finding 1, when Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[token])`, the key now holds Worker B's token. The Lua release script correctly returns `0` without deleting — this part is safe. However, this also means Worker A's release is silently a no-op, and Worker B's lock remains intact. This is the correct behavior from the release script's perspective.

This is not an independent bug — it is a consequence of Finding 1 — but it confirms that the release script's token-check is functioning as intended and does not introduce an additional failure path.

---

## Verdict: **NOT APPROVED**

The primary correctness risk is the missing reverse signal from the heartbeat thread to the main execution thread on lock loss, compounded by the absence of a fencing token at the DB write boundary. A GC pause or slow external call longer than `CLAIM_TTL` produces a zombie write with no architectural boundary to stop it. The prior review's atomicity concern about `RENEW_CLAIM_SCRIPT` is dismissed — Lua scripts in Redis are atomic by specification and the GET+EXPIRE sequence is not the failure point.
