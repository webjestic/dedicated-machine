## Findings

---

**Finding 1: Heartbeat Exit Is Invisible to the Main Thread — Zombie Write Is Possible**

**Issue name:** Silent heartbeat exit / zombie write at DB boundary

**Failure path — traced in full:**

1. Process A acquires `job:claim:job-001` with token `T1`, TTL=60s. Heartbeat thread starts.
2. Process A enters `_dispatch()`. A GC pause, slow external call, or network stall holds the process for >60s.
3. The lock TTL expires. Redis evicts `job:claim:job-001`.
4. Process B acquires `job:claim:job-001` with token `T2`. Process B passes the `get_execution` idempotency check (nothing recorded yet), dispatches, and begins writing.
5. Process A's heartbeat fires at the next `RENEWAL_INTERVAL` boundary. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:job-001` returns `T2`, not `T1`. The script returns `0`. The heartbeat thread hits `if result == 0: return` and exits silently. No exception is raised. No shared state is set.
6. Process A's main thread is in `execute()`, inside `with self._claim_job(req.job_id):`. The context manager `yield` has already returned `token`. The main thread has no reference to `stop_event`, no reference to the thread object, and no mechanism that propagates the heartbeat's exit into the main thread's execution context. `threading.Thread` daemon exit does not raise in the spawning thread.
7. Process A completes `_dispatch()`, enters `with self.db.transaction():`, calls `self.db.record_execution(...)` and `self.db.mark_job_complete(...)`.
8. Both Process A and Process B have now written a completed execution record for the same job. The job ran twice.

**Why the existing idempotency check does not save you:**

The `get_execution` check at step entry is a read, not a write with a uniqueness constraint. Between Process A's read (returns `None`) and Process A's write, Process B also reads `None` and also writes. The check is a TOCTOU race. It closes the window for the common case (job already done before either worker starts) but does nothing for the concurrent-execution window opened by lock expiry.

**Why `stop_event` does not save you:**

`stop_event` is set in the `finally` block of `_claim_job` — *after* the `yield` returns, i.e., after `execute()` has already finished the DB write. The heartbeat thread sets `stop_event` only from the main thread side. The reverse direction — heartbeat signaling the main thread — does not exist in this code. There is no `lost_lock_event` that the main thread polls. Even if one were added, a signal detects loss; it does not prevent the write that has already begun or is executing concurrently on the other process.

**Required fix — and why it must live at the DB layer, not the threading layer:**

The architectural boundary that prevents a zombie write is **optimistic concurrency control at the database write boundary**, not a threading signal. A threading signal is best-effort and subject to the same pause that caused the problem: if the process is paused, it cannot check the signal before writing.

The correct fix is one of the following, in order of strength:

- **Fencing token written to the DB row.** The Redis lock acquisition sequence number (or a monotonic counter stored in Redis alongside the token) is passed through to the DB write. The `record_execution` and `mark_job_complete` calls include a `WHERE fencing_token = <expected>` clause (or equivalent). The DB enforces that only the write carrying the highest/current token succeeds. Process A's write, carrying a stale token, is rejected by the DB engine — not by a Python thread check.

- **Unique constraint + conditional insert.** `record_execution` uses an `INSERT ... WHERE NOT EXISTS` or a unique index on `job_id` in the executions table, combined with the transaction. The first writer wins at the DB level; the second writer gets a constraint violation, which is caught and treated as "already executed." This is weaker than a fencing token (it does not distinguish *which* lock holder wrote) but it does prevent double-write if the DB enforces uniqueness atomically.

- **Optimistic version field.** `mark_job_complete` issues `UPDATE jobs SET status='complete', version=version+1 WHERE job_id=? AND status='pending'`. Zero rows affected means another writer already completed it; the transaction is rolled back and the result is treated as a duplicate.

Any of these must live at the DB layer because that is the only layer that is:
(a) outside the paused process, and therefore not subject to the same stall, and
(b) capable of atomic conditional writes that enforce mutual exclusion on the state transition.

**Finding 2: TOCTOU on the Idempotency Check**

**Issue name:** Read-then-write idempotency gap

**Failure path:**

Even without a GC pause, two workers can race through the `get_execution` check simultaneously. Both read `None`. Both proceed to `_dispatch()`. Both write. The Redis lock serializes claim acquisition, but if the lock is not held for the full duration of the read-check-dispatch-write sequence (which it is not, given Finding 1), or if two workers acquire the lock in sequence after a crash-without-release, the check provides no protection.

This is a secondary consequence of Finding 1, but it also exists independently: the idempotency check is advisory, not enforced. It will not catch the concurrent-write scenario.

**Required fix:**

Same DB-layer fix as Finding 1. A unique constraint on `(job_id)` in the executions table, or a conditional update on job status, makes the idempotency check enforceable rather than advisory.

**Finding 3: Heartbeat Thread Joined with Finite Timeout — Release May Proceed Against Unverified Thread State**

**Issue name:** Bounded join does not confirm thread exit before release

**Failure path:**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)  # 16 seconds
self._release_script(keys=[claim_key], args=[token])
```

If the heartbeat thread is blocked (e.g., Redis call hanging), `join` returns after 16s regardless of whether the thread has exited. The release script then executes. The thread may subsequently wake, find the key gone or re-owned, and return `0` — which is handled. However, there is no assertion or log that the thread failed to exit within the timeout. In a production incident, this silent continuation makes post-hoc diagnosis harder. Minor issue relative to Finding 1, but worth noting.

**Required fix:**

After `t.join(timeout=...)`, check `t.is_alive()` and log a warning (or raise) if the thread did not exit. This does not fix Finding 1 but improves observability.

---

**Finding 4: Test Suite Does Not Cover the Failure Mode**

**Issue name:** No test for lock-loss-during-execution path

**Failure path:**

The test `test_execute_success` mocks `register_script` to return a mock that always returns `1`. There is no test that simulates `RENEW_CLAIM_SCRIPT` returning `0` mid-execution and verifies that the main thread is halted before the DB write. This means the gap in Finding 1 is not caught by CI.

**Required fix:**

Add a test that configures the renew script mock to return `0` on first call, triggers a simulated execution, and asserts that `db.record_execution` is **not** called. This test will fail against the current implementation, which is the correct outcome — it encodes the invariant that lock loss must prevent the DB write.

---

## Verdict: **NOT APPROVED**

The code contains a confirmed zombie-write path. A GC pause or slow external call beyond `CLAIM_TTL` causes lock expiry, heartbeat silent exit, and main thread continuation into the DB write — with no architectural boundary preventing the duplicate write. The threading layer (`stop_event`, daemon thread) detects lock loss but cannot prevent a write that is already executing or that executes while the process is paused. The fix must be a conditional write or fencing token at the DB layer. This is not mergeable until that boundary exists.
