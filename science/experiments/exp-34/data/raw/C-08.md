## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write Is Live**

**Issue name:** Undetected lock loss — zombie write via GC/stall pause

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL = 60s. The background `_renew` thread starts.
2. Worker A enters `_dispatch()`. A GC pause, slow DB call, or external HTTP timeout holds the main thread for > 60s.
3. The renewal thread fires at t=15s, t=30s, t=45s — all succeed. At t=60s the lock has not yet expired because renewals kept it alive. But if the stall begins at, say, t=50s and lasts 25s, the next renewal fires at t=65s — after the TTL has lapsed.
4. At t=60s the key expires. Worker B calls `SET NX EX`, acquires the key with token `T2`, and begins executing the same job.
5. At t=65s Worker A's renewal thread fires. `RENEW_CLAIM_SCRIPT` does `GET job:claim:{job_id}` → returns `T2` ≠ `T1`. The script returns `0`. The `_renew` function hits `if result == 0: return` and **exits silently**.
6. Worker A's main thread is in `_dispatch()` or the DB transaction. It has zero visibility into the heartbeat thread's state. `stop_event` is never set by `_renew`; the main thread never checks it. Worker A continues to completion.
7. Both workers call `self.db.record_execution(...)` and `self.db.mark_job_complete(...)`. The job executes twice.

**Why the existing mitigations do not close this:**

- The idempotency check (`db.get_execution`) runs *before* `_dispatch`. By the time Worker B checks, Worker A has not yet written its completion record. Both workers see `None` and proceed.
- The DB transaction wraps record + mark atomically, but it does not consult the Redis lock. It provides no fencing against a second worker that already passed the idempotency check.
- The `stop_event` is only ever set in the `finally` block of `_claim_job` — by the main thread, after the work is done. It is never set by `_renew` on lock loss. The main thread has no channel to learn that `_renew` exited early.

**Required fix — and why it must live at this layer:**

The heartbeat thread must communicate lock loss back to the main thread so the main thread can abort before side effects are committed. A `threading.Event` (e.g., `lock_lost_event`) set by `_renew` on `result == 0` is the minimum necessary signal. The main thread must check this event at each major boundary: after `_dispatch` returns and before the DB transaction commits. If `lock_lost_event` is set, raise an exception that unwinds the `with self._claim_job(...)` block without writing.

However, this is a Tier 0.5 fix. It reduces the window but does not eliminate it. The fundamental problem is that the Redis lock and the DB write are in different systems with no shared atomicity boundary. The correct architectural fix is a **fencing token / monotonic version counter**:

- The lock acquisition returns a monotonic token (e.g., a Redis `INCR`-based counter, or a Redlock-style fence token).
- The DB write includes a `WHERE lock_version = :acquired_version` predicate (optimistic concurrency / conditional write).
- If Worker A's token has been superseded by Worker B's acquisition, Worker A's DB write fails the condition and is rejected — regardless of whether the heartbeat thread communicated anything.

This fix must live at the DB write layer because that is the only layer that can provide a single serializable decision point. Redis and the DB are not in the same transaction domain; the only way to make the DB write conditional on lock ownership is to carry the lock's identity into the write itself.

**Finding 2: Dismissal of the Prior Review's Atomicity Concern re: `RENEW_CLAIM_SCRIPT`**

**Issue name:** Lua atomicity concern — not a real failure path

**The prior concern:** Under high Redis connection saturation, the `GET` and `EXPIRE` inside `RENEW_CLAIM_SCRIPT` may not execute atomically, creating a window where renewal appears to succeed after another worker has claimed the key.

**Verdict: Dismissed. This is not a real failure path.**

Redis executes Lua scripts atomically with respect to all other Redis commands. From the Redis documentation and implementation: a Lua script is executed as a single atomic unit — no other command from any client can interleave between the `GET` and the `EXPIRE` within the script. This is not a property that degrades under connection saturation. Connection saturation affects *queuing* of commands at the network/socket layer; once a script begins execution on the Redis server, it runs to completion without preemption. The single-threaded command processing model of Redis is the architectural boundary that enforces this.

The concern would be valid if the renewal were implemented as two separate round-trip commands (`GET` then `EXPIRE` in Python). It is not. The Lua script boundary is the correct and sufficient atomicity guarantee here.

**The prior review identified the right layer (the Lua script) but the wrong failure mode.** The script is not the primary correctness risk. The primary risk is Finding 1: the script correctly detects lock loss and returns `0` — but the caller discards that signal without aborting the main thread.

---

**Finding 3: Release After Lock Loss Attempts to Delete a Key Owned by Another Worker**

**Issue name:** Spurious release on foreign lock

**Failure path:** In the scenario from Finding 1, Worker A's `finally` block calls `self._release_script(keys=[claim_key], args=[token])`. At this point the key holds Worker B's token `T2`. The Lua script correctly checks `GET == ARGV[1]` → `T2 ≠ T1` → returns `0` without deleting. This is safe by itself.

**However:** if Worker B's lock has since expired and a third worker C has acquired it, the same no-op applies. The release is always a no-op after lock loss, which is correct behavior. This is not an independent bug — it is a consequence of Finding 1 — but it confirms that the Lua release script is functioning as designed and does not introduce additional risk.

---

## Verdict: **NOT APPROVED**

The primary correctness risk is Finding 1: the heartbeat thread exits silently on lock loss and the main thread has no mechanism to detect this, allowing a stalled worker to complete job execution and commit DB writes after its lock has expired and been acquired by a competing worker. This is a live zombie-write path that produces duplicate job execution.

The prior review's atomicity concern about `RENEW_CLAIM_SCRIPT` is dismissed — Lua scripts are atomic on the Redis server and this property does not degrade under connection pressure.

Minimum required before approval: (1) heartbeat thread sets a `lock_lost_event` on `result == 0`; (2) main thread checks that event before committing the DB transaction and aborts if set; (3) architectural path to fencing token / conditional DB write is documented as a follow-on requirement for the cases where the event check itself races (stall begins after `_dispatch` returns but before the check executes).
