**Findings**

---

**Finding 1: Silent Heartbeat Exit — Main Thread Has No Visibility Into Lock Loss**

*Issue name:* Undetected lock loss / zombie execution

*Failure path, step by step:*

1. Process A acquires `job:claim:{job_id}` with token `T1`, TTL=60s.
2. A GC pause, slow `_dispatch` call, or slow external I/O holds Process A for >60s.
3. The lock expires in Redis. Process B runs `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Process A's heartbeat thread wakes at `RENEWAL_INTERVAL=15s`, executes `RENEW_CLAIM_SCRIPT`. Redis now holds `T2`; the token comparison fails; the script returns `0`.
5. The `_renew` function hits `if result == 0: return` — it exits silently. No exception is raised. No flag is set. The `stop_event` is never set by the heartbeat thread.
6. Process A's main thread is inside `with self._claim_job(...)`. It has no join on the heartbeat thread, no polling of any shared state, and no mechanism to observe that `_renew` exited early. It continues executing as if the lock is still held.
7. Both Process A and Process B reach `self.db.transaction()` → `record_execution` → `mark_job_complete`. The job executes twice.

*Why `threading.Event` signaling is insufficient as a fix:*

Setting a shared flag (e.g., `lock_lost = threading.Event()`) and checking it in the main thread before the DB write is a detection mechanism, not a prevention mechanism. It is a TOCTOU boundary: the flag can be checked clean, then the process stalls again between the check and the DB write, and the write proceeds anyway. The flag also requires the author to remember to check it at every write site — a convention, not an enforcement boundary.

*Required fix — fencing token at the database write boundary:*

The architectural fix must live at the DB layer because that is the only boundary that serializes both Process A and Process B. The mechanism:

- When the lock is acquired, the token (or a monotonic counter issued by Redis via `INCR`) is the fencing token.
- The DB schema carries a `claim_token` column on the job row (or a conditional-write predicate).
- The `record_execution` / `mark_job_complete` writes are issued as conditional writes: `UPDATE jobs SET status='complete', claim_token=:token WHERE job_id=:job_id AND claim_token=:token AND status='pending'` (or equivalent `INSERT ... WHERE NOT EXISTS` with token).
- If Process A's token `T1` is no longer current (Process B wrote `T2`), Process A's conditional write matches zero rows and raises an error. The DB rejects the zombie write structurally, regardless of what any thread-local flag says.

This is the only boundary that works because: (a) it is enforced by the DB engine, not by application convention; (b) it covers the case where the main thread never polls the flag; (c) it covers the case where the flag is checked clean but the process stalls again before the write; (d) it is the same boundary both workers contend on, so one of them must lose.

**Finding 2: Heartbeat Exit Does Not Propagate — `_renew` Return Value Is Discarded**

*Issue name:* Swallowed thread termination / no crash-up path

*Failure path:*

The `_renew` function returns on `result == 0` with no side effect. The thread object `t` terminates. Because `t` is a `daemon=True` thread, the runtime makes no complaint. The `finally` block in `_claim_job` calls `stop_event.set()` and `t.join()` — but by the time `finally` runs, the job has already executed. There is no mechanism by which the early return of `_renew` interrupts or cancels the `with` block body. The gap between heartbeat exit and `finally` execution is unbounded — it spans the entire remaining execution of `_dispatch` and the DB transaction.

*Required fix:*

Independent of Finding 1, the heartbeat thread should write to a shared `threading.Event` (e.g., `lock_lost_event`) on `result == 0` before returning. The main thread should check this event at each major phase boundary (before `_dispatch`, before the DB transaction). This is a defense-in-depth measure — it reduces the window and catches cases where the DB-layer fencing token is misconfigured or absent. It does not replace Finding 1's fix; it complements it. Alone, it is Tier 0.5; combined with the DB-layer fencing token, it is correct defense in depth.

**Finding 3: Idempotency Check Is Not Atomic With Execution — TOCTOU on `get_execution`**

*Issue name:* Check-then-act race on job completion state

*Failure path:*

```python
existing = self.db.get_execution(req.job_id)   # READ
if existing is not None:
    return ...                                  # guard
# ... time passes, other worker writes here ...
self._dispatch(...)                             # ACT
self.db.record_execution(...)                   # WRITE
```

Two workers can both pass the `get_execution` check before either has written the completion record. The Redis lock is the intended serialization point, but as shown in Finding 1, that serialization can be lost. Even with a correct lock, the idempotency check provides no additional safety unless it is inside the same atomic DB transaction as the write.

*Required fix:*

The idempotency guard must be enforced by a unique constraint at the DB layer (e.g., `UNIQUE(job_id)` on the executions table, or a conditional insert). The application-level `get_execution` check is a performance optimization (avoid dispatching work that is already done), not a correctness boundary. The DB unique constraint is the correctness boundary — it makes the write itself idempotent regardless of how many workers reach it.

**Finding 4: Test Suite Does Not Cover the Failure Mode**

*Issue name:* Missing test for lock-loss / zombie-write path

*Observation:*

Every test mocks `register_script` to return a script mock with `return_value=1`. No test sets `return_value=0` on the renew script to simulate lock loss and then asserts that the main thread does not proceed to `record_execution`. The test suite validates the happy path and the claim-contention path, but provides no regression coverage for the exact failure mode this PR is supposed to prevent.

*Required fix:*

Add a test that: (a) simulates `_renew_script` returning `0` mid-execution, (b) asserts that `db.record_execution` is not called (once Finding 1's fix is in place). Without this test, the architectural fix in Finding 1 has no regression harness.

---

**Verdict: NOT APPROVED**

The dominant issue is Finding 1. The heartbeat thread exits silently on lock loss; the main thread has no visibility into this exit; there is no fencing token or conditional write at the database boundary. Under a GC pause or slow external call exceeding `CLAIM_TTL=60s`, two workers will both complete the job and both write to the database. For a job that sends a payment, fires an alert, or triggers an external action, this is a correctness violation. The fix must be a conditional write at the DB layer keyed on the fencing token — `threading.Event` signaling detects the loss but does not close the zombie-write window at the boundary that matters.
