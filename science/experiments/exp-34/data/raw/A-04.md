## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s. The heartbeat fires but the renewal interval (15s) means the lock can expire between heartbeat cycles if the process is fully paused — or the heartbeat itself is paused by the same GC event.
3. The lock expires. Worker B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Worker A's main thread resumes. Worker A's next heartbeat fires: `RENEW_CLAIM_SCRIPT` compares the stored value (`T2`) against `T1` — mismatch, returns `0`.
5. The `_renew` function sees `result == 0` and executes `return` — the thread exits silently. No exception is raised. No event is set. No flag is written. The main thread is never notified.
6. Worker A's main thread continues executing: `_dispatch()` completes, `db.transaction()` commits `record_execution` and `mark_job_complete`.
7. Worker B also completes the same path.
8. **Both workers have executed the job.** For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the operation fires twice.

**The exact gap in code:**

```python
# _renew():
if result == 0:
    return          # ← thread exits, nothing signals the main thread
```

```python
# main thread in execute():
with self._claim_job(req.job_id):
    # ... no mechanism to detect that _renew exited
    result = self._dispatch(req.job_type, req.payload)   # ← proceeds regardless
    with self.db.transaction():
        self.db.record_execution(...)
        self.db.mark_job_complete(...)
```

There is no shared state between the heartbeat thread and the main thread that would cause the `with self._claim_job(...)` block to abort when lock ownership is lost.

**Required fix — two layers, both necessary:**

*Layer 1 (necessary but insufficient alone): Thread signaling*

The heartbeat must communicate loss back to the main thread:

```python
# In _claim_job, pass a loss event into the thread:
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key], args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()   # signal main thread
            return

# Main thread must poll or check before each phase boundary:
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution")
```

This closes the silent-exit gap but is still not sufficient for correctness — see Finding 2.

*Layer 2 (required for correctness): Fencing token / optimistic concurrency at the side-effect boundary*

Thread signaling only helps if the main thread checks the event before the side effect. A GC pause can occur between the check and the write. The only architectural boundary that actually prevents duplicate side effects is moving the lock ownership proof into the side-effect transaction itself.

The database write must be conditioned on lock ownership, not just preceded by a check:

**Option A — Fencing token written into the DB row:**
The lock token (`T1`) is passed into `record_execution`. The DB schema has a unique constraint on `(job_id, claim_token)` or the insert is a conditional upsert. If Worker B has already written with `T2`, Worker A's insert either fails or is rejected by the constraint. The transaction becomes the serialization point.

**Option B — Monotonic counter / version field:**
Redis lock value is a monotonic counter (INCR-based) rather than a UUID. The counter value is stored in the DB row. Worker A's write includes `WHERE claim_version = <A's version>`. If Worker B has already incremented and written, Worker A's UPDATE affects 0 rows — detectable as a concurrency failure.

**Option C — Idempotency key tied to the lock token:**
`execution_id` is derived from or includes the lock token. The DB has a unique index on `execution_id`. Worker A's insert conflicts with Worker B's and raises an integrity error, which the executor catches and treats as "already executed by another worker."

Without one of these options, even with thread signaling, a sufficiently timed pause between the `lock_lost_event.is_set()` check and the `db.transaction()` commit leaves the zombie-write window open.

**Finding 2: Idempotency Check Is Not Inside the Lock's Serialization Boundary**

**Issue name:** TOCTOU on idempotency check — check and write are not atomic relative to the database

**Failure path:**

1. Worker A and Worker B both attempt to claim the job. Worker A wins the Redis lock.
2. Worker A calls `self.db.get_execution(req.job_id)` — returns `None`.
3. Worker A enters `_dispatch()` and stalls (slow handler, network call).
4. (In the zombie-write scenario from Finding 1) Worker B acquires the lock, calls `self.db.get_execution(req.job_id)` — also returns `None` because Worker A has not yet committed.
5. Both workers proceed to write.

The `get_execution` check is only meaningful if the lock is held continuously and exclusively through the write commit. Once the lock can be lost and re-acquired (as in Finding 1), the check provides no protection. Even without lock loss, if the DB transaction is not serializable, two workers that somehow both hold the lock could both read `None` and both write.

**Required fix:** The idempotency check must be enforced at the database layer with a unique constraint on `job_id` in the executions table, not only as an application-level read-before-write. The `record_execution` / `mark_job_complete` writes must fail with a constraint violation if another worker has already written. The application must catch that violation and treat it as "already executed" rather than an unhandled error.

**Finding 3: Renewal Interval Does Not Guarantee Lock Is Held Continuously**

**Issue name:** TTL expiry window between renewal cycles

**Failure path:**

`CLAIM_TTL = 60s`, `RENEWAL_INTERVAL = 15s`. Under normal operation the lock is renewed at t=15, t=30, t=45 — well within TTL. However, if the Python process is paused (GC, OS scheduling, container CPU throttle) for longer than 60s at any point, the lock expires before the next renewal fires. The renewal interval being 4× shorter than TTL is reasonable under normal conditions but provides no guarantee under pause scenarios, which are exactly the failure mode this locking is meant to handle.

This is not independently fixable at the renewal-interval layer — no renewal interval eliminates the pause window. It reinforces that the fencing token at the DB write boundary (Finding 1, Layer 2) is the only correct architectural control.

**Required fix:** Document explicitly that the renewal interval does not bound pause duration and that correctness under pauses depends on the fencing token at the write boundary, not on the heartbeat.

**Finding 4: Release Script Executes After `stop_event.set()` Without Verifying Lock Is Still Held**

**Issue name:** Unconditional release attempt on a lock that may have already expired and been re-acquired

**Failure path:**

1. Worker A loses the lock (TTL expiry). Worker B acquires it with token `T2`.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join()`, then `self._release_script(keys=[claim_key], args=[token])`.
3. The Lua script correctly checks `get(key) == T1` — this is `False` because the value is now `T2`. The script returns `0` and does not delete the key. **This part is correct.**
4. However, the `finally` block does not check the return value of `_release_script`. A `0` return (meaning "I no longer owned this lock") is silently ignored.

The release script itself is safe (token-verified). The gap is that the caller discards the signal that lock ownership had already been lost before release was attempted. Combined with Finding 1, this means the main thread has no post-hoc confirmation that it held the lock through the critical section.

**Required fix:** Check the return value of `_release_script`. If it returns `0`, log a warning at ERROR level indicating that the lock was lost before explicit release — this is a signal that the zombie-write scenario may have occurred and should trigger an alert or compensating check.

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** Heartbeat-exit / lock-loss scenario has no test coverage

The test `test_claim_release_uses_lua_script` verifies the release path. No test simulates `_renew_script` returning `0` and verifies that the main thread aborts execution. The mock fixture hardcodes `script_mock = MagicMock(return_value=1)` — always-successful renewal — making the failure path structurally untestable with the current fixture design.

**Required fix:** Add a test that configures `_renew_script` to return `0` on the first call and asserts that `record_execution` is never called and a `ClaimError` (or equivalent lock-loss exception) is raised. This test will fail against the current implementation, correctly exposing the gap.

## Zombie-Write Scenario Confirmation

**Can two workers execute the same job concurrently despite the locking mechanism?**

Yes. The exact conditions:

1. Worker A acquires the lock and enters `_dispatch()`.
2. A GC pause or slow external call stalls Worker A's process for longer than `CLAIM_TTL` (60s).
3. The lock expires. Worker B acquires the lock and begins executing the same job.
4. Worker A resumes. Its heartbeat fires, sees token mismatch, returns `0`, and **exits silently without notifying the main thread**.
5. Both workers complete `_dispatch()` and commit to the database.
6. The job executes twice.

**What is missing to close it:**

- The heartbeat must signal the main thread on lock loss (`threading.Event` — necessary, not sufficient).
- The main thread must check that signal at each phase boundary before side effects.
- The database write must be conditioned on lock ownership via a fencing token, monotonic counter, or unique constraint that makes the second write fail atomically — this is the only boundary that actually closes the window.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed open. The heartbeat thread exits silently on lock loss; the main thread has no mechanism to detect this and continues to execute and commit. Two workers can and will execute the same job under a GC pause or process stall exceeding the lock TTL. The fix requires both thread-level signaling and a fencing token at the database write boundary. Neither is present.
