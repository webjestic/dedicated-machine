## Findings

---

**Finding 1: Heartbeat Exit Is Invisible to the Main Thread — Zombie Write Is Possible**

**Issue name:** Silent heartbeat exit / zombie write at DB boundary

**Failure path — traced exactly:**

1. Process A acquires `job:claim:job-001` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. A GC pause, slow `_dispatch()` call, or slow external I/O stalls Process A for >60s. The lock expires in Redis.
3. Process B calls `redis.set(claim_key, token, nx=True, ex=60)` — succeeds. Process B now holds the lock with token `T2` and begins executing the same job.
4. Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `redis.call("get", KEYS[1])` returns `T2`, not `T1`. The script returns `0`.
5. In `_renew()`:
   ```python
   if result == 0:
       return
   ```
   The thread returns. No exception is raised. No event is set. No flag is written. The thread exits silently.
6. Process A's main thread is executing inside `with self._claim_job(req.job_id):`. It has **zero visibility** into the heartbeat's exit. There is no shared flag, no exception propagated across the thread boundary, no check between `_dispatch()` and the DB write.
7. Process A resumes after its stall. `_dispatch()` completes. Execution reaches:
   ```python
   with self.db.transaction():
       self.db.record_execution(execution_id, req.job_id, req.job_type, result)
       self.db.mark_job_complete(req.job_id)
   ```
   This write commits. Process B's identical write also commits (or has already committed).
8. The job has now executed twice. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the duplicate has already happened before the DB write — `_dispatch()` is called with no lock-loss check at all.

**Why the existing idempotency check does not save you:**

The `get_execution` check at entry is a read-before-execute guard, not a write-time guard. By the time Process A reaches the DB write, Process B may not yet have committed (race), or the check was already passed by both workers before either committed. The check is not inside the same atomic transaction as the write, and it is not re-evaluated after `_dispatch()`. It provides no protection against the scenario above.

**Why `threading.Event` signaling is insufficient as a fix:**

Adding `stop_event.set()` in `_renew()` on `result == 0`, then checking `stop_event.is_set()` in the main thread between `_dispatch()` and the DB write, is a partial mitigation only. It narrows the window but does not close it:
- The stall may occur *inside* `_dispatch()` after the event check — the main thread cannot poll continuously through arbitrary external calls.
- More critically, `_dispatch()` itself may have already executed the side-effectful operation (sent the notification, charged the card) before any check is possible. The duplicate is the external call, not the DB write.
- Signal-based detection is a best-effort race condition, not an architectural guarantee.

**Required fix — fencing token at the DB write boundary:**

The only fix that provides a hard guarantee is optimistic concurrency / a fencing token enforced at the layer that must not duplicate: the database write.

Concretely:

**Option A — Optimistic concurrency on job state:**
The `jobs` table carries a `lock_token` column written at claim time. The `mark_job_complete` update includes a `WHERE job_id = ? AND lock_token = ? AND status = 'running'` predicate. If Process A's token has been superseded by Process B's token, the update matches zero rows. The application checks `rowcount == 0` and aborts. This is a hard barrier — it does not matter what the main thread believes about lock state.

```sql
UPDATE jobs
   SET status = 'complete', execution_id = ?
 WHERE job_id = ?
   AND lock_token = ?        -- fencing predicate
   AND status = 'running';   -- idempotency predicate
```

If `rowcount == 0`: raise, do not commit `record_execution`.

**Option B — Monotonic fencing token from Redis (Redlock-style):**
Replace the UUID token with a monotonically increasing counter (e.g., Redis `INCR` on a per-job sequence key). Pass this token through to the DB write predicate. A higher token from Process B will cause Process A's predicate to fail. This requires the DB schema to store the current token and the write to be conditional on it.

**Why it must live at the DB layer:**
The database transaction is the only synchronization point that is atomic, durable, and shared across all processes. Redis lock state is advisory — it can expire, be lost, or be superseded between the moment the main thread last "knew" it held the lock and the moment it writes. Only a conditional write at the DB boundary — one that will fail if the precondition (token ownership) is no longer true — provides a correctness guarantee. Threading signals operate within a single process and cannot enforce anything about what another process has written or will write.

**Finding 2: `_dispatch()` Executes External Side Effects Outside Any Fencing Boundary**

**Issue name:** Side-effectful dispatch before any lock-loss detection point

**Failure path:**

`_dispatch()` is called before the DB transaction:
```python
result = self._dispatch(req.job_type, req.payload)   # external call here

with self.db.transaction():                           # fencing would live here
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

If the job handler sends a notification, charges a payment, or fires an external event, that action occurs with no lock-loss check at any point. Even if a fencing token is added to the DB write, the external side effect has already been duplicated. The DB write fencing prevents duplicate DB records but not duplicate real-world actions.

**Required fix:**

For jobs with external side effects, the handler must be idempotent at the external system boundary (e.g., idempotency keys on payment APIs, deduplication tokens on notification services). This is a contract that must be documented and enforced per job type. Additionally, the lock-loss check (whether event-based or token-based) should be evaluated immediately before `_dispatch()` is called, not only at the DB write. This does not eliminate the need for the DB-layer fencing token — it reduces (but cannot eliminate) the window for duplicate external calls.

**Finding 3: Heartbeat Thread Join in `finally` Does Not Bound Claim Release Correctly Under Stall**

**Issue name:** Claim release races with stall recovery

**Failure path:**

In the `finally` block:
```python
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)   # waits up to 16s
self._release_script(keys=[claim_key], args=[token])
```

If the lock has already expired and been acquired by Process B with token `T2`, the release script runs:
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then  -- T2 != T1, false
    return redis.call("del", KEYS[1])
else
    return 0                                     -- no-op, correct
end
```

The Lua release script is correctly token-gated and will not delete Process B's lock. This part is safe. However, the `t.join(timeout=16)` in `finally` means the main thread blocks for up to 16 additional seconds after the stall before releasing — during which Process B is already executing. This is a latency concern, not a correctness issue given the token-gated release, but it is worth noting that the join timeout should be short (1–2s) since the heartbeat thread either exits promptly on `stop_event.set()` or is already dead.

---

## Verdict: NOT APPROVED

The code is well-structured and demonstrates correct instincts — token-gated Lua scripts for release and renewal, daemon thread scoping, exponential backoff, transactional DB writes. These are not cosmetic. But the structural gap is a correctness violation in a multi-worker deployment:

**The heartbeat thread exits silently on lock loss. The main thread has no visibility into this exit. No fencing token or conditional write predicate exists at the DB boundary. A process stall beyond LOCK_TTL produces two workers executing the same job to completion.**

The fix is not a threading signal. It is a conditional write at the database layer — `WHERE job_id = ? AND lock_token = ? AND status != 'complete'` — that fails atomically if token ownership has been superseded. Until that predicate exists and is verified to return `rowcount == 0` on mismatch with a hard abort, this code will produce duplicate executions under GC pause or slow external call conditions. That scenario is not hypothetical; it is a normal operating condition in any JVM-adjacent or Python GC environment under load.
