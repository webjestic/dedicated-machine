**Findings**

---

**1. Silent Heartbeat Exit — Main Thread Has No Visibility Into Lock Loss**

**Name:** Zombie write via silent heartbeat exit on token mismatch

**Failure path, step by step:**

Process A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. The heartbeat thread starts, firing every 15s. Process A enters `_dispatch()` — a slow external call, or the JVM-equivalent GC pause in CPython (e.g., a long GC cycle under memory pressure, a slow DNS resolution, a blocking I/O call) stalls the process for >60s. The lock expires naturally in Redis.

Process B runs `SET NX EX` on the same key, succeeds, acquires the lock with token `T2`, and begins executing the same job. Process B passes the idempotency check (`get_execution` returns `None` — Process A hasn't written yet) and enters `_dispatch()`.

Process A's stall ends. Its heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2` ≠ `T1`, so the script returns `0`. The heartbeat thread hits `if result == 0: return` and exits. **The thread exits silently.** No exception is raised. No flag is set on any object visible to the main thread. `stop_event` is not set by the heartbeat — it is only ever set by the `finally` block in `_claim_job`, which runs on the main thread after `yield` returns.

Process A's main thread resumes from `_dispatch()` with a return value. It proceeds to:

```python
with self.db.transaction():
    self.db.record_execution(execution_id, req.job_id, req.job_type, result)
    self.db.mark_job_complete(req.job_id)
```

This write commits. Process B's main thread does the same. Both commits succeed. The job has executed twice. For a job that sends a payment, fires an alert, or triggers an external webhook, this is a correctness violation.

**Why `threading.Event` signaling is insufficient as a fix:**

Adding `stop_event.set()` inside the `if result == 0` branch of `_renew()` is a partial mitigation, not a fix. It creates a TOCTOU window: the heartbeat fires on its 15s interval. Between the moment the lock expires and the moment the heartbeat next fires, the main thread can complete `_dispatch()` and enter the DB transaction. The signal arrives after the damage is done. Even with prompt signaling, the main thread would need to poll `stop_event` between every meaningful operation — `_dispatch()`, and before the DB write — and that polling is not present. A signal detects lock loss; it does not prevent the write that has already begun.

**Required fix — fencing token at the database write boundary:**

The architectural fix must live at the database layer, not the threading layer, because the database is where the duplicate write actually occurs. The Redis lock is a coordination hint; the database is the authority.

**Option A — Optimistic concurrency / conditional write:**
The `mark_job_complete` call must be a conditional update:

```sql
UPDATE jobs
SET status = 'complete', execution_id = :execution_id
WHERE job_id = :job_id
  AND status = 'pending'
```

If Process B has already written `status = 'complete'`, Process A's update matches zero rows. The executor checks `rowcount == 0` and treats it as a duplicate — rolls back, does not record the execution. This works even if both processes reach the write concurrently, because the database serializes the two `UPDATE` statements under row-level locking. Only one can transition from `pending → complete`; the other sees zero rows affected.

**Option B — Monotonic fencing token stored in Redis:**
Use `INCR job:fence:{job_id}` at claim acquisition time. Store the returned counter value as the fencing token. Pass it through to the DB write as a precondition:

```sql
UPDATE jobs SET status = 'complete'
WHERE job_id = :job_id AND fence_token = :my_fence_token
```

Process B's claim increments the counter; Process A holds a stale token value. Process A's conditional write fails.

Either option closes the window unconditionally, regardless of timing, GC pause duration, or thread scheduling. The database enforces the invariant at the only boundary that matters.

**2. Idempotency Check Is Not Atomic With the Completion Write**

**Name:** Check-then-act race on `get_execution` / `mark_job_complete`

**Failure path:**

Even without a GC pause, two workers can pass the idempotency check concurrently. Both call `get_execution(job_id)`, both receive `None` (neither has written yet), both proceed to `_dispatch()`. This is a narrower window than Finding 1 but is structurally the same class of bug — the read and the write are not atomic. The Redis lock is supposed to serialize this, but as shown in Finding 1, the lock can be held by two processes simultaneously during the expiry window.

**Required fix:**

The conditional `UPDATE ... WHERE status = 'pending'` from Finding 1 also closes this window. The `get_execution` pre-check can remain as a fast-path optimization, but correctness must not depend on it. The database write must be the authoritative serialization point.

---

**3. Heartbeat Join in `finally` Does Not Bound Zombie Write Window**

**Name:** `t.join()` in `finally` executes after the DB write, not before

**Failure path:**

The `finally` block executes after `yield` returns — i.e., after `execute()` has already called `db.record_execution()` and `db.mark_job_complete()`. The join serves only to clean up the thread before releasing the Redis key. It provides no protection against the zombie write because the write has already happened by the time `finally` runs.

**Required fix:**

Thread lifecycle management is correct for resource cleanup. It is not a substitute for a write-boundary guard. No restructuring of the `finally` block addresses the core issue — see Finding 1.

---

**4. Test Suite Does Not Cover the Failure Mode**

**Name:** Heartbeat-exit / lock-loss scenario is untested

**Failure path:**

All tests mock `register_script` to return a script mock with `return_value=1` unconditionally. No test simulates `result == 0` from `_renew_script` and verifies that the main thread does not proceed to the DB write. The test `test_claim_release_uses_lua_script` verifies that the Lua script is called, not that lock loss is handled correctly. The gap is invisible in CI.

**Required fix:**

Add a test that configures `_renew_script` to return `0` mid-execution (simulating lock expiry), then asserts that `db.record_execution` and `db.mark_job_complete` are **not** called. This test will fail against the current implementation, correctly surfacing the bug. After the architectural fix in Finding 1 is applied, this test documents the invariant.

---

**Verdict:** NOT APPROVED

The code contains a zombie-write vulnerability. A GC pause or slow external call exceeding `CLAIM_TTL` causes the Redis lock to expire, a second worker to acquire it, and the original worker to complete a database write after its lock has lapsed — with no mechanism at the database boundary to reject the stale write. The heartbeat correctly detects token mismatch but exits without signaling the main thread, and even prompt signaling would not close the TOCTOU window between `_dispatch()` completion and the DB commit. A conditional database write (`UPDATE ... WHERE status = 'pending'`) is required before this can be approved.
