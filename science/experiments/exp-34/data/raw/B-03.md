**Findings**

---

**Finding 1: Silent Heartbeat Exit Does Not Propagate to Main Thread — Zombie Write Possible**

**Issue name:** Undetected lock loss / zombie write at DB boundary

**Failure path, step by step:**

1. Process A acquires `job:claim:{job_id}` with token `T_A`, TTL = 60s. Heartbeat thread starts, renewing every 15s.
2. Process A enters `_dispatch()`. A GC pause, slow external HTTP call, or I/O stall holds the process for >60s. No renewal fires during the stall (the heartbeat thread is also paused, or fires but the process wall-clock time exceeds TTL between renewals).
3. The Redis key expires. Lock is now vacant.
4. Process B calls `redis.set(claim_key, T_B, nx=True, ex=60)` — succeeds. Process B acquires the lock and begins executing the same job.
5. Process A's stall ends. Its heartbeat fires: `RENEW_CLAIM_SCRIPT` runs `GET job:claim:{job_id}` → returns `T_B` ≠ `T_A` → script returns `0`. The `_renew()` function hits `if result == 0: return` and the thread exits silently. No exception is raised, no event is set, no flag is written.
6. Process A's main thread is executing inside `with self._claim_job(req.job_id):`. It has **zero visibility** into the heartbeat thread's exit. `stop_event` is never set by the heartbeat thread on lock loss — it is only set in the `finally` block of `_claim_job`, which has not been reached. The main thread continues as if the lock is still held.
7. Process A completes `_dispatch()`, passes the idempotency check (Process B has not yet written `mark_job_complete`), and enters `with self.db.transaction(): self.db.record_execution(...); self.db.mark_job_complete(...)`.
8. Process B does the same.
9. Both writes commit. The job executes twice. For a payment charge, alert dispatch, or external event trigger, this is a correctness violation.

**Why `threading.Event` signaling alone does not fix this:**

Even if the heartbeat thread called `stop_event.set()` on `result == 0`, and even if the main thread polled `stop_event` between `_dispatch()` and the DB write, there is a TOCTOU window: the check and the write are not atomic. The main thread can observe "lock still held" and then lose the lock before the write commits. A signal is a detection mechanism, not a prevention mechanism. It narrows the window; it does not close it.

**Required fix — fencing token at the database write boundary:**

The architectural fix must live at the layer where the duplicate side-effect occurs: the database write.

The Redis lock must issue a monotonically increasing fencing token at acquisition time (e.g., via `INCR job:fence:{job_id}` inside the acquisition script, or using Redlock's index). That token is passed through to the DB write. The database enforces: "only write if the fencing token presented equals the current expected token for this job_id." This can be implemented as:

- A `job_claims` table with a `fence_token` column and a conditional UPDATE/INSERT: `UPDATE jobs SET ... WHERE job_id = ? AND fence_token = ?` — if 0 rows affected, abort.
- Or an optimistic concurrency check: a `version` column incremented at lock acquisition, with the write conditioned on `WHERE version = ?`.

Under this scheme, when Process A attempts `record_execution` / `mark_job_complete` with stale token `T_A` (fence value `N`), the DB write predicate fails because Process B has already written with fence value `N+1`. The write is rejected at the storage boundary regardless of what the main thread believes about its lock state. The zombie write cannot commit.

The current code passes no fencing token to `db.record_execution` or `db.mark_job_complete`. The DB transaction has no knowledge of the Redis lock state. These two layers are entirely decoupled, which is the root of the vulnerability.

**Concrete change required:**

```python
# In _claim_job: obtain a fencing token at acquisition time
fence_token = self.redis.incr(f"job:fence:{job_id}")
yield token, fence_token

# In execute: pass fence_token to every DB write
with self.db.transaction():
    self.db.record_execution(execution_id, req.job_id, req.job_type, result,
                             fence_token=fence_token)
    self.db.mark_job_complete(req.job_id, fence_token=fence_token)
# DB enforces: UPDATE ... WHERE job_id = ? AND current_fence = ?
# 0 rows affected → raise and abort
```

The `incr`-based fence must be part of the atomic acquisition script or executed immediately after `SET NX` in a pipeline — not a separate round-trip that can race.

**Finding 2: Heartbeat Thread Exit on Lock Loss Is Semantically Invisible — No Annotation, No Signal**

**Issue name:** Silent failure mode in `_renew()` — `result == 0` branch has no observable effect on caller

**Failure path:** As traced above, `result == 0` causes `return` with no logging, no event, no flag. An operator reading logs during an incident sees heartbeat renewal debug lines stop — and nothing else. There is no `logger.error` or `logger.warning` emitted when lock loss is detected. Incident diagnosis requires source-reading, not log-reading.

**Required fix:** This is secondary to Finding 1 but must be addressed independently:

```python
if result == 0:
    logger.error(
        "Claim lost — token mismatch during renewal, lock expired. "
        "job_id=%s Halting renewal thread. Main thread may continue executing "
        "without a valid claim.", job_id
    )
    return
```

Additionally, the heartbeat thread should set a `threading.Event` (`lock_lost_event`) on `result == 0`, and the main thread should check this event before the DB write as a defense-in-depth signal (not a replacement for the fencing token). This does not prevent the zombie write on its own, but it closes the easy cases where the stall is short enough that the main thread checks the event before proceeding.

---

**Finding 3: Test Suite Does Not Cover the Lock-Loss Scenario**

**Issue name:** Missing test for heartbeat exit path and post-loss DB write behavior

The test `test_execute_success` mocks `register_script` to always return `1`. There is no test that:
- Simulates `_renew_script` returning `0` mid-execution
- Verifies that `db.record_execution` is **not** called after lock loss
- Verifies that a second executor instance cannot write after the first loses its lock

Until Finding 1 is fixed architecturally, these tests cannot pass correctly. But they must be written as part of the fix to prevent regression.

---

**Verdict:** NOT APPROVED

The code contains a zombie-write vulnerability in the class that causes duplicate job execution in production. The heartbeat correctly detects lock loss via token mismatch but discards that information without notifying the main thread and without any fencing mechanism at the database write boundary. A GC pause or slow external call exceeding LOCK_TTL is a realistic production event — not a theoretical edge case. The fix requires a fencing token threaded from Redis lock acquisition through to a conditional database write predicate. Threading.Event signaling is insufficient as a standalone fix. This must not merge until the DB write boundary enforces lock ownership.
