## Findings

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Primary Correctness Risk**

**Name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token T, TTL=60s. The heartbeat thread starts with a 15s renewal interval.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s. The heartbeat fires at t=15s and renews successfully. The main thread is still stalled at t=30s, t=45s. At t=60s the lock has been renewed to t=75s, but the stall continues past t=75s without a renewal firing in time — or alternatively, a single stall between renewal intervals exceeds 60s outright.
3. The lock expires. Worker B executes `SET NX EX`, acquires the same key with token T′, and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns T′ ≠ T, so the script returns 0. The heartbeat thread hits `if result == 0: return` and exits. **No exception is raised. No event is set. No flag is written.**
5. Worker A's main thread is unaware. It is not joined to the heartbeat thread. It does not poll any shared state. It continues: `_dispatch()` completes, `self.db.transaction()` commits `record_execution` and `mark_job_complete`.
6. Worker B, having acquired the lock legitimately, also completes the same path.
7. Both workers commit. The job executes twice.

**Why the existing structure cannot contain this:**

The `stop_event` is a one-directional signal: main thread → heartbeat thread (stop). There is no reverse channel. The heartbeat's `return` on `result == 0` is a thread-local exit with no observable side effect on the main thread. The `finally` block in `_claim_job` calls `stop_event.set()` and `t.join()` — but by the time those execute, the damage is already done: `yield token` has already returned through the `with self._claim_job()` block, meaning `_dispatch()` and the DB transaction have already completed.

**Required fix — two layers, both necessary:**

*Layer 1 — Signal propagation (necessary but not sufficient):*
The heartbeat must write lock-loss back to the main thread. A `threading.Event` (`lock_lost_event`) set on `result == 0` is the minimum. The main thread must check this event at a meaningful boundary — ideally before `_dispatch()` and before the DB commit. This closes the silent-exit gap but does not close the race: Worker A could check the event, find it unset, stall, lose the lock, and then commit — the check and the commit are not atomic with respect to the Redis state.

*Layer 2 — Fencing at the write boundary (necessary to fully close the race):*
A `threading.Event` alone is Tier 0.5. It reduces the window but does not eliminate it. The correct fix is a **fencing token** (monotonic counter or generation number) embedded in the lock value and propagated to the database write. The DB `record_execution` / `mark_job_complete` must include a conditional write that verifies the fencing token matches the current lock generation — so a stale worker's commit is rejected at the persistence layer even if it passes the in-process event check. Redis's `SET` with a monotonic counter (e.g., via `INCR` on a generation key, or a Lua-managed counter) provides the token; the DB write becomes `UPDATE ... WHERE job_id = ? AND lock_generation = ?`. A worker that has lost the lock writes with a stale generation and the DB rejects it. This is the only boundary that is atomic with respect to both the lock state and the persistent side effect.

**Finding 2: Prior Review's Atomicity Concern — Dismissed**

**Name:** Alleged non-atomicity of `RENEW_CLAIM_SCRIPT`

**The prior concern:** That `GET` followed by `EXPIRE` in Lua may not execute atomically under high Redis connection saturation.

**Verdict: This is not a real failure path.**

Redis executes Lua scripts atomically by design. From the Redis documentation and source: a Lua script is executed as a single atomic unit — no other Redis command can execute between the `GET` and the `EXPIRE` within the same script invocation. Redis is single-threaded with respect to command execution; the event loop does not interleave another client's commands mid-script. Connection saturation affects *queuing* of script invocations, not the atomicity of an in-flight script. A saturated connection pool means the script may be delayed before it starts, but once it starts, it runs to completion without interleaving.

The `GET` + `EXPIRE` sequence inside `RENEW_CLAIM_SCRIPT` is atomic. The prior review's concern is technically unfounded.

**What the prior review may have been gesturing at, but named incorrectly:** The real gap is not within the script — it is the temporal gap *before* the script runs. If the main thread stalls past the TTL, the lock expires and is re-acquired by Worker B *before* the heartbeat fires. When the heartbeat eventually fires, the script correctly detects the mismatch and returns 0. The script's atomicity is not the problem; the problem is that the return value of 0 is silently discarded by the calling thread. That is Finding 1.

**Finding 3: Release After Lock Loss — Correctness Defect**

**Name:** Unconditional release attempt on a lock the worker no longer holds

**Failure path:**

In the scenario from Finding 1, after Worker A's heartbeat exits silently, the `finally` block still executes `self._release_script(keys=[claim_key], args=[token])`. At this point, `job:claim:{job_id}` holds Worker B's token T′. The Lua script correctly checks `GET` == T (Worker A's token) → false → returns 0, so Worker B's lock is not deleted. The Lua script's token check saves correctness here.

**However:** This is only safe because the release script is correctly token-verified. If it were a bare `DEL`, Worker B's lock would be destroyed mid-execution, opening the key for a third worker. The current implementation is safe on this specific path, but it is worth noting explicitly: the safety is entirely dependent on the Lua release script, and any future refactor that replaces it with a bare `DEL` (e.g., in an error handler or timeout path) would immediately reintroduce the vulnerability. This should be documented with an explicit comment.

**Required fix:** Add a comment on the `_release_script` call in `finally` stating that the token-verified Lua script is load-bearing for correctness in the lock-loss scenario, and that it must never be replaced with an unconditional delete.

---

**Finding 4: Test Suite Does Not Cover the Failure Mode**

**Name:** Missing test for heartbeat-exit / main-thread-continues scenario

**Failure path:** The test suite mocks `register_script` to return a mock that always returns 1. There is no test that simulates `_renew_script` returning 0 and verifies that the main thread detects lock loss and does not proceed to `record_execution`. `test_claim_release_uses_lua_script` verifies the release script is called but does not verify behavior under lock loss.

**Required fix:** Add a test that configures `_renew_script` to return 0 on first call, then asserts that `db.record_execution` is never called. This test will fail against the current implementation, correctly surfacing Finding 1 as a test failure rather than a production incident.

---

## Verdict: NOT APPROVED

The primary correctness risk is **Finding 1**: the heartbeat thread exits silently on lock loss and the main thread has no mechanism to detect this, allowing a worker that has lost its lock to complete job execution and commit to the database. This is a live zombie-write path that produces duplicate job execution.

The prior review's atomicity concern (Finding 2) is **dismissed**. Lua scripts in Redis are atomic by the Redis execution model. The `GET` + `EXPIRE` sequence cannot be interleaved. The prior review identified the right script but the wrong failure mechanism.

The fix requires both a signal path from heartbeat to main thread *and* a fencing token propagated to the database write layer. Either alone is insufficient: the event check is not atomic with the DB commit, and the DB write without a fencing token cannot distinguish a stale worker's commit from a legitimate one.
