**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external HTTP call, or database stall holds the main thread for >60s without the heartbeat firing in time — or the heartbeat fires but the Redis call itself stalls (network partition, Redis replica failover) long enough that the TTL expires before renewal lands.
3. The lock expires. Worker B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`.
5. The `_renew` function checks `if result == 0: return`. The heartbeat thread exits. **No exception is raised. No event is set. No flag is written.**
6. Worker A's main thread is executing inside `with self._claim_job(req.job_id):`. It has no reference to the heartbeat thread's state. It continues through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution()` and `mark_job_complete()`.
7. Worker B, having passed the `get_execution()` idempotency check before Worker A's write lands, also completes the same path.
8. Both workers write execution records and mark the job complete. The job executes twice. For a payment charge or outbound notification, this is a duplicate operation.

**The idempotency check does not close this window.** Both workers can pass `get_execution() is None` before either commits, because the check and the write are not atomic with respect to the lock state. The database transaction serializes the write, but if both workers reach the `record_execution` call, the outcome depends entirely on database-level uniqueness constraints — which are not present in this code.

**Required fix — two layers:**

*Layer 1 (necessary but not sufficient): Signal the main thread on heartbeat death.*

```python
# Add to _claim_job:
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must poll or check this event at safe points — before `_dispatch`, before the DB transaction, and ideally inside long-running dispatch operations. This closes the "main thread continues unaware" path but is still not sufficient alone, because the check is not atomic with the DB write.

*Layer 2 (architecturally required): Fencing token / optimistic concurrency at the DB write boundary.*

The `record_execution` and `mark_job_complete` calls must be conditioned on the lock still being held at write time. The correct pattern:

- Pass the lock token (or a monotonic version counter) into the DB write as a conditional: `UPDATE jobs SET status='complete', claimed_by=:token WHERE job_id=:job_id AND claimed_by=:token AND status='pending'`. If the row was claimed by a newer token (Worker B), this write affects 0 rows and must be treated as a failure.
- Alternatively: enforce a `UNIQUE` constraint on `(job_id)` in the execution record table, so the second writer gets a constraint violation rather than a silent duplicate insert.

Without Layer 2, even with the `lock_lost` event, a race remains: Worker A checks `lock_lost` (not set), proceeds to the DB write, but the lock expires between the check and the commit. The fencing token at the DB layer is the only boundary that is atomic with the state mutation.

**This is the zombie-write scenario. It is present in this code as submitted.**

**Finding 2: Renewal Interval Does Not Guarantee Coverage Against Arbitrary Pauses**

**Issue name:** TTL/renewal ratio insufficient against GC or I/O stalls

**Failure path:**

CLAIM_TTL=60s, RENEWAL_INTERVAL=15s. Under normal operation the heartbeat fires at t=15, t=30, t=45 — adequate. However: if the process experiences a stop-the-world GC pause of >60s (possible in Python with large heaps and certain GC configurations, or under memory pressure triggering OS-level swapping), or if the Redis connection stalls for >60s (network partition, proxy timeout), the lock expires before the next successful renewal. The heartbeat then exits per Finding 1, and the zombie-write path is open.

This is not fixable purely by tuning the ratio — any finite TTL can be exceeded by a sufficiently long pause. The architectural fix in Finding 1 Layer 2 (fencing token at DB write) is the correct boundary. The ratio tuning is defense-in-depth only.

**Required fix:** Reduce RENEWAL_INTERVAL to TTL/4 or less (already satisfied here at TTL/4), and document explicitly that the renewal thread is not a safety guarantee — the DB-layer fencing token is. No ratio eliminates the need for Layer 2.

**Finding 3: Idempotency Check Is Not Atomic With Execution**

**Issue name:** Check-then-act race on `get_execution`

**Failure path:**

1. Workers A and B both acquire the lock in sequence (not concurrently) — or both pass the lock check in the zombie scenario above.
2. Both call `self.db.get_execution(req.job_id)` and receive `None`.
3. Both proceed to `_dispatch()` and then to `self.db.transaction()`.
4. Both call `record_execution()`. Without a unique constraint on `job_id` in the executions table, both writes succeed silently.

The `get_execution` check is a read outside the write transaction. It is not a substitute for a database-level uniqueness constraint.

**Required fix:** The executions table must have a `UNIQUE` constraint on `job_id` (or `(job_id, status)` scoped appropriately). The `record_execution` call must be structured as an `INSERT ... WHERE NOT EXISTS` or rely on the constraint to produce a conflict error that is caught and handled as "already executed." This is the correct layer for idempotency enforcement — not a pre-check read.

**Finding 4: Release Executes After Lock May Already Be Expired or Re-Owned**

**Issue name:** Release call on zombie lock is a no-op but logs success

**Failure path:**

In the zombie scenario, when `_claim_job`'s `finally` block runs on Worker A, it calls `self._release_script(keys=[claim_key], args=[token])`. The key now holds Worker B's token. The Lua script correctly returns `0` and does not delete the key — this part is correct. However, the `logger.info("Claim released job_id=%s")` fires unconditionally, producing a log entry that says the claim was released when it was not. This misleads post-incident analysis.

**Required fix:**

```python
released = self._release_script(keys=[claim_key], args=[token])
if released:
    logger.info("Claim released job_id=%s", job_id)
else:
    logger.error(
        "Claim release failed — token mismatch, lock was lost or re-owned job_id=%s",
        job_id
    )
```

The error log here is also a signal that the zombie-write scenario may have occurred and should trigger an alert.

**Finding 5: Heartbeat Thread Join Timeout Does Not Verify Completion**

**Issue name:** Unverified thread join before release

**Failure path:**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)  # 16 seconds
self._release_script(keys=[claim_key], args=[token])
```

If the heartbeat thread is blocked on a Redis call (network stall), `t.join()` returns after 16s with the thread still running. The release script then executes. The thread may subsequently complete its `_renew_script` call — but since the lock is now released (or re-owned), this is a no-op due to the token check. This specific sequence is safe given the Lua guard. However, the code does not check `t.is_alive()` after the join, so a blocked thread is silently ignored. Under the zombie scenario, this compounds the observability problem.

**Required fix:** Check `t.is_alive()` after join and log an error if the thread did not terminate. This is observability, not a correctness fix, but it is required for incident diagnosis.

**Finding 6: Test Suite Does Not Cover the Zombie-Write Scenario**

**Issue name:** Missing test for lock-loss during execution

The test `test_claim_release_uses_lua_script` verifies the release path. No test simulates `_renew_script` returning `0` mid-execution and verifies that the main thread halts before writing. This is the exact failure mode. Without this test, the fix for Finding 1 can be reverted without a failing test catching it.

**Required fix:** Add a test that configures the renew script mock to return `0` after the first call, injects a `lock_lost` check into the execution path, and asserts that `record_execution` is never called when the lock is lost before the DB write.

## Zombie-Write Scenario Confirmation

**The zombie-write scenario can occur.** The exact conditions:

- Worker A holds the lock and enters a pause (GC, slow I/O, OS swap) exceeding CLAIM_TTL.
- The lock expires. Worker B acquires it and begins executing.
- Worker A's heartbeat detects token mismatch, returns 0, exits silently.
- Worker A's main thread has no signal. It completes `_dispatch()` and writes to the database.
- Worker B also completes. Both execution records are written. The job runs twice.

**What must be added to close it:**

1. `threading.Event` signaling from heartbeat to main thread — necessary to make lock loss visible, insufficient alone.
2. Fencing token or optimistic concurrency check at the database write boundary — the only atomic boundary that prevents the duplicate write. Specifically: the DB write must be conditional on the job not already being claimed by a different token, enforced by the database engine, not by a pre-check read.
3. Unique constraint on `job_id` in the executions table as a last-resort correctness backstop.
