**Findings**

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Name:** Zombie Write via Silent Heartbeat Exit

**Failure Path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow DB call, or external HTTP timeout holds the main thread for >60s without the heartbeat firing in time — or the heartbeat fires but the Redis connection is transiently unavailable, causing the renewal to fail silently (exception path — see Finding 2), or the process is simply paused long enough that the TTL expires between heartbeat intervals.
3. The lock TTL expires. Worker B calls `SET NX EX` on the same key, succeeds, acquires the lock with token `T2`, and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. The script returns `0`. The `_renew` function hits `if result == 0: return` and the thread exits.
5. **The `stop_event` is never set by the heartbeat thread. The main thread has no mechanism to observe the thread's exit.** Worker A's main thread continues executing past `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
6. Worker B does the same.
7. Both workers complete the job. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the action has already fired twice before either worker reaches the DB transaction. The DB idempotency check (`get_execution`) only runs *before* dispatch — it does not protect against concurrent execution that began while both workers held what they each believed was a valid lock.

**The zombie write is confirmed.** Worker A is the zombie: it has lost its lock, does not know it, and writes (and acts) anyway.

**Required Fix — Layer: Architectural (fencing token passed to all downstream operations)**

`threading.Event` signaling from heartbeat to main thread is a partial mitigation only. It closes the gap between heartbeat exit and the *next* operation the main thread attempts, but it is not sufficient because:

- The main thread may be mid-dispatch (inside a payment call, inside an HTTP request) when the heartbeat exits. Setting an event does not interrupt that in-flight operation.
- Even with the event, the main thread must *poll* it — there is no preemptive cancellation in Python threads. Any code path that does not check the event before a side-effecting operation is still a zombie.

The correct architectural fix is a **fencing token / monotonic counter**:

- Replace `uuid4()` with a Redis-issued monotonic counter (e.g., `INCR job:fence:{job_id}`) at claim acquisition time. This value is the fencing token.
- Pass the fencing token through to every downstream system that accepts writes: the database `record_execution` call must include a `WHERE fence_token = $N AND fence_token > last_seen_fence` predicate, or use optimistic concurrency control (OCC) with a version column.
- The DB write from Worker A (with fence token `N`) will be rejected if Worker B has already written with fence token `N+1`. The side-effecting external call inside `_dispatch` must also be made idempotent using the fencing token as an idempotency key passed to the external system.
- This is the only fix that holds at the layer where the violation occurs: the write itself, not the lock check.

The `threading.Event` signal from heartbeat to main thread should *also* be added as a defense-in-depth measure to abort before dispatch when lock loss is detected early, but it must not be the sole control.

**Finding 2: Heartbeat Thread Swallows Redis Exceptions — Lock Loss Is Undetectable on Network Error**

**Name:** Exception Masking in Renewal Loop

**Failure Path:**

1. Worker A holds the lock. A transient Redis connection error occurs during the `_renew_script` call inside `_renew()`.
2. The exception propagates out of `_renew_script(...)` and is not caught. The `while not stop_event.wait(...)` loop exits via unhandled exception. The thread dies.
3. The lock TTL is not renewed. After 60s, the lock expires.
4. Worker B acquires the lock. Worker A's main thread continues, unaware the heartbeat thread has died and the lock has expired.
5. Same zombie-write outcome as Finding 1.

Conversely, if the intent is to catch the exception and continue the loop, the renewal silently skips, the TTL is not extended, and the lock expires anyway.

**Required Fix — Layer: Heartbeat thread + main thread signaling**

The heartbeat thread must catch all exceptions from the renewal call, log them, and signal the main thread via a shared `threading.Event` (a separate `lock_lost_event`). The main thread must check `lock_lost_event` before every phase boundary (before dispatch, before DB write). This is defense-in-depth; the fencing token (Finding 1) remains the authoritative control.

---

**Finding 3: Idempotency Check Is Outside the Lock's Causal Scope for the Side Effect**

**Name:** Check-Then-Act Race on DB Completion State

**Failure Path:**

1. Workers A and B both attempt to claim the same job. Worker A wins the lock. Worker B is blocked in retry backoff.
2. Worker A checks `get_execution` → `None`. Proceeds to `_dispatch()`. The dispatch call is slow (external HTTP).
3. Worker A's lock expires mid-dispatch (Finding 1 trigger). Worker B acquires the lock.
4. Worker B checks `get_execution` → still `None` (Worker A has not yet written). Worker B proceeds to `_dispatch()`.
5. Both dispatch. Both write. Duplicate execution.

The `get_execution` check is correct as a fast-path guard against *already-completed* jobs, but it does not protect against two workers that are *concurrently executing* the same job for the first time. The DB transaction wrapping `record_execution` + `mark_job_complete` would catch the second writer *if* there is a unique constraint on `job_id` in the execution table — but this is not visible in the PR, and `_dispatch()` (the side-effecting call) has already fired before the transaction.

**Required Fix — Layer: Database schema + dispatch architecture**

- The `executions` table must have a `UNIQUE` constraint on `job_id`. The transaction will then fail with a constraint violation for the second writer, and that error must be caught and handled as a non-error outcome (job already executed by peer).
- Side-effecting operations inside `_dispatch()` must use the fencing token as an idempotency key passed to downstream systems, so that even if two workers reach the dispatch call, the downstream system deduplicates on the token.

---

**Finding 4: Lock Release in `finally` Block May Release a Lock Owned by Another Worker**

**Name:** Stale Release After Lock Expiry (Partial — Mitigated by Lua Script, but Conditional)**

**Failure Path:**

1. Worker A's lock expires. Worker B acquires the lock with token `T2`.
2. Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[token])` with `T1`.
3. The Lua script checks `GET claim_key == T1` → false (it's `T2`). Returns `0`. Does not delete.

This specific scenario is correctly handled by the token-verified Lua release script. **However**, if the `_release_script` call itself raises an exception (Redis unavailable), the lock is not released and will hold until TTL expiry. This is acceptable behavior (TTL is the backstop), but the exception must be caught and logged in the `finally` block rather than propagating and masking the original exception from the `try` body.

**Required Fix — Layer: `finally` block exception handling**

Wrap `_release_script(...)` in a `try/except` inside the `finally` block. Log failures. Do not let release errors suppress job execution errors.

---

**Finding 5: `t.join(timeout=...)` Does Not Confirm Thread Has Exited**

**Name:** Unconfirmed Thread Join Before Release**

**Failure Path:**

1. `stop_event.set()` is called. `t.join(timeout=RENEWAL_INTERVAL + 1)` is called (16s timeout).
2. If the heartbeat thread is blocked on a hung Redis call, `join` returns after 16s regardless of whether the thread has actually exited.
3. `_release_script` is called. The release succeeds (or fails as in Finding 4).
4. The heartbeat thread unblocks from its hung Redis call and calls `_renew_script` *after* the lock has been released — potentially after Worker B has acquired it. The renewal call will return `0` (token mismatch) and exit, so no lock extension occurs. This is benign *only* because the Lua script is token-verified.

This is a low-severity issue given the Lua guard, but the join timeout should be logged as a warning when `t.is_alive()` is true after join returns.

**Required Fix — Layer: `_claim_job` finally block**

After `t.join(timeout=...)`, check `t.is_alive()`. If true, log a warning. No functional change needed given the Lua token guard, but observability is required.

---

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can and will occur** under the following conditions:

- Worker A acquires the lock and begins `_dispatch()`.
- A process pause (GC, slow syscall, external call latency) causes the lock TTL to expire before the next heartbeat renewal.
- Worker B acquires the lock and begins executing the same job.
- Worker A's heartbeat detects the mismatch and exits silently.
- Worker A's main thread has no signal and completes execution.
- Both workers execute the job and write results.

**What is missing to close it:**

1. A **fencing token** (monotonic counter) issued at claim time, threaded through to all downstream writes and external calls, enforced at the write layer with optimistic concurrency control or unique constraints. This is the only control that holds at the layer where the violation occurs.
2. A **lock-lost event** signaled from the heartbeat thread to the main thread, checked at every phase boundary before side-effecting operations, as defense-in-depth.

Neither is present in this PR.

---

**Verdict: NOT APPROVED**

The zombie-write scenario is confirmed and open. The heartbeat thread exits silently on lock loss; the main thread has no visibility into that exit and no mechanism to abort in-flight execution. The fencing token pattern — the architectural boundary required to close this class of failure — is absent. This code will produce duplicate job executions under GC pause, slow external calls, or transient Redis unavailability in a multi-worker deployment.
