**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Name:** Zombie-write via silent heartbeat exit on lock expiry.

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()` — a slow external call, GC pause, or I/O stall holds it for >60s without the heartbeat firing in time (or the heartbeat fires but the renewal round-trip itself is delayed past expiry).
3. The Redis key expires. Worker B calls `SET NX EX`, acquires the same key with token `T2`, passes the idempotency check (`get_execution` returns None — job not yet recorded), and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`. The `_renew` thread hits `if result == 0: return` and exits silently. **No signal is sent to the main thread.**
5. Worker A's main thread continues executing. It has no awareness the heartbeat exited. It completes `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
6. Worker B independently completes the same sequence.
7. Both workers execute the job to completion. For a payment charge, alert dispatch, or external event trigger: **the operation fires twice.**

**The structural gap:** `_renew` returns on lock loss but has no channel back to the main thread. `stop_event` is write-only from the main thread's perspective — the heartbeat never sets it. The `yield token` in the context manager proceeds unconditionally regardless of heartbeat state.

**Required fix — two layers, both required:**

*Layer 1 — Signal the main thread on lock loss (necessary but not sufficient):*
Replace the bare `threading.Event` stop signal with a shared failure flag. In `_renew`, on `result == 0`, set a `lock_lost_event` before returning. In the `try` block wrapping `yield`, poll this event at checkpoints or use a wrapper that raises into the main thread. The cleanest form: after `yield` returns (i.e., the job body finished), check the flag before proceeding to the DB write. This closes the gap for the case where the job body completes after lock loss but before the DB commit.

```python
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

Then in the finally / post-yield path:
```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    if lock_lost.is_set():
        raise ClaimError(f"Lock lost during execution of job {job_id}")
    self._release_script(keys=[claim_key], args=[token])
```

This prevents the DB write from proceeding after confirmed lock loss — but it only catches loss detected before or during `t.join()`. If the job body finishes and the heartbeat hasn't fired yet, the race window is not fully closed.

*Layer 2 — Fencing token / optimistic concurrency at the DB write (required to close the window completely):*

The DB transaction must verify that this worker still holds the claim at commit time. The token acquired at lock time must be passed through to the DB layer and checked atomically with the write. One of:

- **Fencing token written to DB at claim time, verified at commit:** On acquisition, write `(job_id, token)` to a `job_claims` table. The `record_execution` transaction includes a conditional: `WHERE claim_token = :token`. If another worker has overwritten the claim (or the row is gone), the write affects 0 rows and the transaction aborts.
- **Monotonic counter / Redis fencing token:** Use `INCR` on a separate counter key at acquisition time. Pass the counter value as a fencing token through the execution path. The DB write includes this value; a constraint or check rejects a write from a worker whose counter is stale.
- **Conditional DB write:** `INSERT INTO executions ... WHERE NOT EXISTS (SELECT 1 FROM executions WHERE job_id = :job_id)` inside the transaction, with the unique constraint on `job_id` as the hard backstop. This is the minimum viable guard — it relies on DB uniqueness rather than token matching, but it does prevent double-write if both workers reach the commit.

The DB unique constraint on `job_id` in the executions table is the architectural backstop that Layer 1 alone cannot provide. Without it, two workers can both pass the `lock_lost` check (neither has detected loss yet) and both commit.

**Finding 2: Idempotency Check Is Outside the Lock's Safety Window**

**Name:** TOCTOU on idempotency check — check and dispatch are not atomic with respect to DB state.

**Failure path:**

1. Workers A and B both acquire the claim in sequence (B after A's lock expires, as above).
2. Worker A checks `get_execution` → None. Begins `_dispatch`.
3. Worker B checks `get_execution` → None (A hasn't committed yet). Begins `_dispatch`.
4. Both dispatch. Both commit. Duplicate execution.

This is the same scenario as Finding 1, viewed from the DB side. The idempotency check is correct in the single-worker case but provides no protection when the lock has been lost and a second worker has acquired it — because the first worker's execution hasn't been recorded yet when the second worker checks.

**Required fix:** The DB unique constraint on `(job_id)` in the executions table must be present and enforced. The `record_execution` call must be structured so that a duplicate insert raises a constraint violation, which propagates as a transaction rollback. This is the only layer that is atomic with respect to both workers' commits. The application must handle this exception and treat it as "job already executed by another worker" rather than an unhandled error.

**Finding 3: Renewal Interval Does Not Guarantee Coverage Against Arbitrary Pauses**

**Name:** Renewal window assumption — TTL=60s, interval=15s assumes bounded pause duration.

**Failure path:**

A GC pause, container CPU throttle, or blocking I/O call in `_dispatch` can stall the process for longer than CLAIM_TTL (60s). The heartbeat thread is a daemon thread in the same process — a stop-the-world GC pause stalls it too. The heartbeat cannot fire during a GC pause. If the pause exceeds 60s, the lock expires regardless of the renewal interval.

This is not a bug in the renewal logic itself — it is an inherent property of process-local heartbeats. No in-process heartbeat can protect against a pause that exceeds the TTL, because the heartbeat is paused alongside the main thread.

**Required fix:** This is the architectural argument for why Layer 2 (fencing token at DB commit) is not optional. The in-process heartbeat is a best-effort mechanism. The DB-layer fencing token is the correctness guarantee. Document this explicitly: the heartbeat reduces the window; the DB constraint closes it.

**Finding 4: `t.join()` Timeout in `finally` Can Mask Heartbeat Thread Hang**

**Name:** Heartbeat thread join timeout — silent continuation if thread does not exit.

**Failure path:**

`t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds. If the heartbeat thread is blocked on a Redis call (network partition, Redis latency spike), `join` returns after 16s with the thread still running. The `finally` block then calls `_release_script`. The release script runs. The heartbeat thread, still alive, may subsequently attempt a renewal against a key that has been released and potentially re-acquired by another worker. The renewal script's token check prevents it from extending the new worker's lock (correct), but the thread is now a leaked daemon thread with no cleanup path.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()` and log a warning if the thread did not exit. This does not fix the leak (daemon threads cannot be forcibly killed in Python) but makes it observable. The real fix is a Redis client with a configured socket timeout so the renewal call cannot block indefinitely.

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not exit cleanly job_id=%s — possible Redis hang", job_id)
```

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Name:** Missing test — lock loss during execution is not exercised.

The test `test_execute_success` mocks the renew script to return `1` unconditionally. There is no test where the renew script returns `0` mid-execution and the test verifies that the DB write is suppressed. The checklist item "Atomic claim renewal via Lua script (token-verified)" is checked, but the behavioral consequence of a `0` return — that execution must halt — is untested.

**Required fix:** Add a test that simulates lock loss mid-execution:

```python
def test_lock_loss_during_execution_prevents_db_write(redis_client, db):
    # Renew script returns 0 (lock lost) on first renewal call
    renew_mock = MagicMock(side_effect=[0])
    release_mock = MagicMock(return_value=1)
    redis_client.register_script.side_effect = [release_mock, renew_mock]
    
    ex = JobExecutor(redis_client, db)
    # With correct implementation, this should raise or return unclaimed
    result = ex.execute("job-001", "notify")
    
    assert result.claimed is False
    db.record_execution.assert_not_called()
```

This test will fail against the current implementation — which is the point. It codifies the required behavior before the fix is merged.

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can occur** under current implementation.

**Trigger:** Process A stalls (GC pause, slow external call) beyond CLAIM_TTL. Lock expires. Process B acquires the lock and begins executing the same job. Process A's heartbeat fires, detects token mismatch, exits silently. Process A's main thread continues with no awareness of lock loss. Both processes complete `_dispatch` and commit to the database.

**What prevents it — currently: nothing.** The Lua renewal script correctly detects the mismatch. The heartbeat correctly exits. The missing piece is the signal from the heartbeat to the main thread, and the DB-layer fencing that makes the commit conditional on still holding the claim.

**What must be added to close it:**
1. `lock_lost` event set by heartbeat on `result == 0`, checked by main thread before DB commit — raises `ClaimError` and aborts the write.
2. DB unique constraint on `job_id` in the executions table, with the application treating constraint violation as a non-error "already executed" outcome — this is the hard correctness boundary that holds even when the timing of lock loss detection is ambiguous.

Both are required. Layer 1 alone has a race window. Layer 2 alone requires the DB to be the coordination point (acceptable, but the heartbeat failure should still be surfaced). Together they close the zombie-write scenario.

---

**Verdict: NOT APPROVED**

The locking mechanism is structurally incomplete. The heartbeat correctly detects lock loss but has no path to halt execution in the main thread. A process pause exceeding CLAIM_TTL produces concurrent execution of the same job with no architectural boundary preventing both workers from committing. Required before merge: (1) `lock_lost` event signaling from heartbeat to main thread with abort on detection, (2) DB unique constraint on job executions with application-level handling of constraint violations as the fencing boundary, (3) test coverage for the lock-loss-during-execution path.
