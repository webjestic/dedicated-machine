**Findings**

---

**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s.
2. Worker A begins `_dispatch()` — an external HTTP call, a slow DB query, or a GC pause holds the thread for >60s.
3. The lock expires in Redis. Worker B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Worker A's heartbeat fires at t=75s. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2`, not `T1`. The script returns `0`.
5. `_beat()` sees `result == 0` and executes `return` — the thread exits silently. No exception is raised, no event is set, no signal reaches the main thread.
6. Worker A's main thread is still inside the `with self._claim_job(...)` block. It has no mechanism to observe the heartbeat thread's exit. It continues: `_dispatch()` completes, `db.transaction()` commits `record_execution` and `mark_job_complete`.
7. Worker B completes the same sequence independently.
8. Both workers have now fully executed the job. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the action has fired twice.

**The checklist entry "Atomic claim renewal via Lua script (token-verified)" is accurate but incomplete as a safety claim.** Token verification on renewal correctly detects lock loss. The gap is not in the detection — it is in the absence of any propagation path from detection to the main execution thread.

**Required fix and layer:**

The fix must live at two layers:

*Layer 1 — Signal propagation (necessary but not sufficient):*
`_renew()` must set a shared `threading.Event` (e.g., `lock_lost_event`) when `result == 0`, rather than returning silently. The main thread must poll this event at safe checkpoints — before `_dispatch()`, between `_dispatch()` and the DB transaction, and before committing. If the event is set, execution must abort. This closes the gap for the cooperative case.

*Layer 2 — Fencing token / optimistic concurrency at the side-effect boundary (required for correctness):*
`threading.Event` polling is not sufficient alone. The main thread can observe the event only at polling points; a pause that spans an entire polling interval still produces a zombie write. The architectural fix is a **fencing token** (monotonic counter) issued at lock acquisition time, passed through to every side-effect operation, and validated atomically at the point of commitment.

Concretely: Redis `INCR` on a global counter at claim time produces a monotonic token. The DB transaction that calls `record_execution` and `mark_job_complete` must include a conditional write that rejects the commit if a higher token has already been recorded for this `job_id`. Worker A's stale token is numerically lower than Worker B's; Worker A's commit is rejected at the database layer regardless of what the main thread believed about its lock state.

This is the only boundary that holds unconditionally — it does not depend on thread scheduling, polling frequency, or GC pause duration.

**Finding 2: Release Executes Against a Lock the Worker No Longer Holds**

**Issue name:** Phantom release attempt after lock expiry

**Failure path:**

In the `finally` block of `_claim_job`:
```python
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

If the lock has already expired and Worker B has acquired it with token `T2`, the release script runs `GET job:claim:{job_id}` → `T2 ≠ T1` → returns `0` and does nothing. This is correct behavior from the Lua script.

However, the return value of `self._release_script(...)` is not checked. A `0` return here is operationally significant — it means the worker is releasing a lock it no longer holds, which is a confirmation that the zombie-write window was open. Without checking this return value, the system has no log entry, no metric, and no alerting signal for this condition. Forensic investigation of a duplicate-execution incident has no artifact to find.

**Required fix:** Check the return value of `_release_script`. If it returns `0`, log at ERROR level with `job_id` and `token`, and — if the `lock_lost_event` mechanism from Finding 1 is implemented — assert that the event was already set. A `0` on release that was not preceded by a detected lock loss is evidence of an undetected zombie-write interval.

**Finding 3: `t.join(timeout=...)` Does Not Confirm Thread Termination Before Release**

**Issue name:** Non-blocking join allows release before heartbeat thread completes final renewal attempt

**Failure path:**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`t.join(timeout=16)` returns whether or not the thread has actually exited — it returns after the timeout regardless. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis replica failover), the join times out, and `_release_script` runs while the heartbeat thread is still alive and may attempt a concurrent `RENEW_CLAIM_SCRIPT` call. These two Redis operations — renew and release — are not coordinated. The release succeeds (deletes the key), then the renewal fires against a now-absent key, returns `0`, and the thread exits. Net effect: the release is correct, but there is a window where both operations are in-flight against Redis simultaneously with no ordering guarantee.

In a degraded-Redis scenario, the renewal could arrive after a new worker has acquired the lock, producing a spurious TTL extension on a lock the new worker believes is fresh.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()`. If the thread is still alive, log a warning and do not proceed with release until the thread is confirmed dead, or accept the risk explicitly and document it. The `stop_event` is already set before the join, so the thread will exit at its next `wait()` wakeup — the join timeout should be sufficient in the non-degraded case, but the liveness check makes the degraded case observable.

**Finding 4: Idempotency Check Is Not Inside the Lock's Atomic Boundary**

**Issue name:** TOCTOU on `get_execution` check

**Failure path:**

```python
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)
    if existing is not None:
        return JobResult(claimed=False, ...)
    # ... execute and record
```

The `get_execution` check and the `record_execution` write are not atomic with respect to each other at the database layer. Two workers that both pass the `existing is None` check before either has committed can both proceed to `_dispatch()`. This is a secondary defense that is only meaningful if the Redis lock is functioning correctly. If the Redis lock is functioning correctly, only one worker holds the claim at a time and this race cannot occur through the normal path.

However, this check provides zero protection in the zombie-write scenario from Finding 1, because both workers pass it in sequence (Worker B checks after Worker A has already committed, but Worker A is the zombie — it checks before Worker B has committed). The check is not a substitute for the fencing token.

**Required fix:** The `record_execution` insert must be constrained by a unique index on `job_id` at the database layer, so that the second writer receives a constraint violation rather than a silent success. This transforms the idempotency check from a best-effort read into an enforced write-once guarantee. The application must handle the constraint violation and treat it as a duplicate-execution signal.

---

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can occur** under the following conditions, all of which are reachable in production:

- Worker A holds the lock and enters a pause (GC, slow external call, network stall) longer than `CLAIM_TTL` (60s).
- Worker B acquires the lock during the pause.
- Worker A's heartbeat detects token mismatch and exits silently.
- Worker A's main thread has no visibility into the heartbeat exit and completes execution.
- Both workers commit side effects.

The architectural boundary that would prevent this — a fencing token validated atomically at the side-effect commit point — **is not present in this PR.**

`threading.Event` signaling from the heartbeat to the main thread is a partial mitigation that reduces the window but does not close it: a pause that begins after the last renewal and ends before the next polling check is undetectable by the event mechanism alone.

---

**Verdict: NOT APPROVED**

The PR is well-constructed — atomic acquisition, token-verified Lua scripts, daemon thread scoping, exponential backoff, and transactional DB writes are all present and correct. The gap is not in any of those components individually. It is in the missing propagation path between lock-loss detection (which works) and execution abort (which does not exist), and in the absence of a fencing token at the side-effect boundary that would make the guarantee unconditional regardless of pause duration or thread scheduling.

Required before approval:
1. Heartbeat sets a `lock_lost_event` on `result == 0`; main thread checks it before and after `_dispatch()`.
2. Fencing token (monotonic counter from Redis `INCR`) issued at claim time, validated in the DB commit via unique constraint or conditional write on `job_id`.
3. Unique index on `job_id` in the executions table to enforce write-once at the database layer.
4. Release return value checked and logged; `t.is_alive()` checked after join.
