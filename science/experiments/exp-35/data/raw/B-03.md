**Findings**

---

**Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie Write Scenario**

**Name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s.
3. The lock expires in Redis. Worker B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. The script returns `0`.
5. `_renew()` sees `result == 0` and executes `return` — the thread exits silently. No exception is raised, no event is set, nothing is written to shared state that the main thread observes.
6. Worker A's main thread resumes from the stall. It has no visibility into the heartbeat thread's exit. It continues: `_dispatch()` completes, the `db.transaction()` block runs, `record_execution` and `mark_job_complete` are called.
7. Worker B completes the same sequence independently.
8. Both workers have now executed the job. For a job that sends a payment, fires an alert, or triggers an external action, this is a correctness violation.

**The structural gap:** `_renew()` returns on lock loss but sets no shared signal. The main thread never checks whether the heartbeat is still alive or whether the lock is still held. The `stop_event` is write-only from the main thread's perspective — the heartbeat never writes to it. There is no channel from the heartbeat thread back to the main thread.

**Required fix — two layers:**

*Layer 1 (necessary, not sufficient): Thread signaling.*
Add a second `threading.Event` (e.g., `lock_lost_event`). In `_renew()`, before returning on `result == 0`, call `lock_lost_event.set()`. In the main thread, check `lock_lost_event.is_set()` at every yield point — before `_dispatch()`, before the DB transaction, before `record_execution`. Raise an exception that aborts the `with _claim_job()` block. This closes the intra-process race.

This is necessary but incomplete. It only helps if the main thread is schedulable to check the event. A true GC pause or OS-level stall can prevent the check from running until after the damage is done.

*Layer 2 (architecturally required): Fencing token / optimistic concurrency at the effect boundary.*
The job's side-effecting operation — the DB write that marks the job complete and records execution — must be conditioned on proof that the lock is still held at the moment of commit. The lock token (`uuid`) must be passed through to the DB transaction and checked there. Concretely:

- Store the current lock token in the DB when the job is first claimed (a `claims` table row: `job_id`, `token`, `claimed_at`).
- The `record_execution` + `mark_job_complete` transaction must include a predicate: `WHERE claim_token = :token AND status = 'claimed'`. If Worker A's token has been superseded by Worker B's token, Worker A's transaction matches zero rows and raises an error.
- This is a monotonic counter / fencing token pattern applied at the storage layer — the only layer that can provide a hard boundary when the lock has expired and a second worker has taken over.

Without Layer 2, even a perfectly signaled heartbeat cannot prevent the zombie write if the main thread stalls between the signal check and the DB commit.

**Finding 2: Lock Release After Lock Expiry May Delete Another Worker's Lock**

**Name:** Stale release deletes successor lock

**Failure path:**

1. Worker A's lock expires (same GC-pause scenario above).
2. Worker B acquires the lock with token `T2`.
3. Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[T1])`.
4. The Lua script does `GET job:claim:{job_id}` → `T2` ≠ `T1` → returns `0`. The key is not deleted.

This part is correctly handled by the token-verified Lua script. The release is safe. No issue here — noted explicitly because it is the one thing the code gets right under this scenario.

**Finding 3: Heartbeat Join Timeout Does Not Abort on Failure**

**Name:** Unchecked thread join masks heartbeat lifecycle

**Failure path:**

1. `stop_event.set()` is called in `finally`.
2. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds — is called.
3. If the heartbeat thread is blocked (e.g., Redis call hanging), `join` returns after the timeout with the thread still alive. No exception, no log, no indication that the thread did not terminate cleanly.
4. The `_release_script` call proceeds regardless.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()`. If true, log an error at minimum; in a strict implementation, treat this as an unclean shutdown and do not attempt the release (the token-verified script makes the release safe, but the operator needs visibility into this state).

---

**Finding 4: Idempotency Check Is Not Inside the Lock's Atomic Boundary**

**Name:** TOCTOU on idempotency check

**Failure path:**

1. Worker A acquires the lock. `db.get_execution(job_id)` returns `None`. Worker A proceeds.
2. Worker A's lock expires before it commits. Worker B acquires the lock. `db.get_execution(job_id)` returns `None` (Worker A has not committed yet). Worker B proceeds.
3. Both workers reach `record_execution`. Without the fencing token fix from Finding 1, both commits succeed.

The idempotency check is correct in the happy path but provides no protection when lock expiry allows two workers to pass through it concurrently. This is a consequence of Finding 1, not an independent bug, but it means the idempotency check cannot be cited as a safety net — it is not one under the failure scenario.

**Required fix:** Same as Finding 1, Layer 2. The DB transaction predicate is the only boundary that can enforce idempotency atomically.

---

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Name:** Heartbeat-exit / lock-loss path is untested

**Failure path:** The mock in `redis_client` fixture sets `register_script` to return a mock with `return_value=1` unconditionally. No test exercises `result == 0` from the renewal script. No test verifies that the main thread is interrupted when the heartbeat exits. The zombie-write scenario is not represented in the test suite at all.

**Required fix:** Add a test that configures the renew script mock to return `0` on the first call, then asserts that `execute()` raises or returns a non-claimed result — and that `record_execution` is not called. This test will fail against the current implementation, which is the correct outcome: it documents the bug and gates the fix.

---

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can occur** under the following conditions:

- Worker A holds the lock and stalls (GC pause, slow external I/O, OS scheduling) for longer than `CLAIM_TTL` (60s).
- Worker B acquires the lock and begins executing.
- Worker A's heartbeat detects lock loss (`result == 0`) and exits silently.
- Worker A's main thread resumes with no knowledge of lock loss and completes execution including the DB write.

The architectural boundary that would prevent this — a fencing token checked atomically at the DB commit boundary — **is absent**. `threading.Event` signaling closes the intra-process gap but cannot prevent the zombie write if the stall occurs between the last signal check and the commit. Only a predicated DB write conditioned on the current lock token provides a hard boundary.

---

**Verdict: NOT APPROVED**

The locking mechanism contains a confirmed zombie-write path. The heartbeat correctly detects lock loss but does not signal the main thread, and there is no fencing token at the DB commit boundary. Under a GC pause or slow external call exceeding 60s — a realistic production condition — two workers will execute the same job concurrently and both will commit. Required before approval: (1) `lock_lost_event` signaling from heartbeat to main thread with checked abort points, and (2) a fencing token predicate on the `record_execution` / `mark_job_complete` transaction that conditions the commit on the lock token still being current in the DB claims table.
