## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Continues Executing After Lock Loss**

**Issue name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the main thread for >60s. The lock expires in Redis.
3. Worker B calls `redis.set(claim_key, token, nx=True, ex=60)` — succeeds. Worker B acquires the lock with token `T2` and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`.
5. `_renew()` sees `result == 0` and executes `return` — the heartbeat thread exits silently. **No signal is sent to the main thread.**
6. Worker A's main thread resumes from the stall. It has no visibility into the heartbeat thread's exit. It continues through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
7. Worker B completes the same path concurrently.
8. **Both workers complete the job.** For a payment charge, alert dispatch, or external API call inside `_dispatch()`, the action fires twice.

**The structural gap:** `_renew()` returns on lock loss but owns no channel back to the main thread. `stop_event` is write-only from the main thread's perspective — the heartbeat never sets it. The `yield token` in `_claim_job` is uninterruptible from the background thread. There is no mechanism by which lock loss in the heartbeat causes the `with self._claim_job(...)` block to abort.

**Required fix — two layers, both required:**

*Layer 1 — Signal propagation (necessary but not sufficient):*
The heartbeat must set a shared flag on lock loss. A `threading.Event` (`lock_lost_event`) set by `_renew()` when `result == 0`, checked by the main thread at boundaries inside the `with` block (after `_dispatch()`, before the DB transaction), allows early abort. This prevents the DB write in the nominal case but does not close the window between the stall and the check.

*Layer 2 — Fencing token / optimistic concurrency at the write boundary (required to fully close the zombie-write):*
The DB write is the irreversible action. The token `T1` acquired at claim time must be passed through to the database write and validated atomically there. The DB transaction must include a conditional: "write this execution record only if `job:claim:{job_id}` still equals `T1`" — either by reading the Redis value inside the transaction and aborting on mismatch, or by using a monotonic fencing counter stored in Redis and passed as a precondition to the DB write (compare-and-write pattern). This ensures that even if Worker A's main thread reaches the DB write after lock loss, the write is rejected because the fencing token no longer matches the current lock holder.

Without Layer 2, the window between "lock expired" and "heartbeat fires" is unbounded — a stall longer than `RENEWAL_INTERVAL` (15s) but shorter than `CLAIM_TTL` (60s) means the heartbeat hasn't fired yet and `lock_lost_event` hasn't been set, yet the lock has already been acquired by Worker B. Worker A proceeds to the DB write with no signal to stop it.

**Finding 2: Release Executes After Lock Has Already Transferred**

**Issue name:** Spurious release of successor's lock

**Failure path:**

1. Worker A's lock expires. Worker B acquires the lock with token `T2`.
2. Worker A's `finally` block executes `_release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET claim_key == T1` — returns `T2` ≠ `T1`, so it correctly returns `0` without deleting.

This specific path is **handled correctly** by the token-verified Lua script. However, it is only safe because the release script is already token-gated. This is noted as correct; it does not independently require a fix but depends on the Lua script being correct, which it is.

---

**Finding 3: Idempotency Check Is Not Atomic With Execution**

**Issue name:** TOCTOU on completion check

**Failure path:**

1. Worker A and Worker B both fail to acquire the lock on first attempt (a third worker held it and crashed without releasing).
2. The lock TTL expires. Both Worker A and Worker B retry and race on `SET NX`.
3. Worker A wins the lock. Worker B loses and raises `ClaimError`. This path is fine.

However, consider: Worker A acquires the lock, checks `db.get_execution()` → `None`, begins `_dispatch()`. Worker A stalls. Lock expires. Worker B acquires the lock, checks `db.get_execution()` → still `None` (Worker A hasn't written yet), begins `_dispatch()`. Both proceed. The idempotency check is only meaningful if the DB write from a prior execution is already committed. It provides no protection against two concurrent in-flight executions of the same job — that protection is supposed to come entirely from the Redis lock, which is the mechanism that fails under Finding 1.

This is a dependent finding: the idempotency check is not a substitute for a correct locking boundary. It is correctly scoped as a post-lock guard against re-execution of already-completed jobs, but it must not be read as providing safety against concurrent execution. The PR's safety model collapses entirely to the Redis lock, which has the gap described in Finding 1.

---

**Finding 4: Heartbeat Join Timeout Does Not Abort on Failure**

**Issue name:** Silent join timeout — release proceeds regardless**

**Failure path:**

1. `stop_event.set()` is called.
2. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds — is called.
3. If the heartbeat thread is blocked (e.g., Redis call hanging), `join` returns after 16s with the thread still alive.
4. `_release_script` is called. No error is raised. No log is emitted indicating the join timed out.

The thread is `daemon=True`, so it will not prevent process exit, but a hung heartbeat thread that outlives the `finally` block can interact with a subsequent claim acquisition on the same key in the same process. Minor issue relative to Finding 1, but the join result should be checked and logged.

**Required fix:** Check `t.is_alive()` after `t.join(...)` and log a warning if the thread did not terminate cleanly.

---

**Conditions Under Which Two Workers Execute the Same Job Concurrently**

The sufficient condition is: **Worker A's main thread stalls for longer than `CLAIM_TTL` (60s) at any point after acquiring the lock.** This includes:

- JVM/Python GC stop-the-world pause (uncommon but not impossible under memory pressure)
- Blocking I/O inside `_dispatch()` — network call to slow external service, database query under lock contention
- OS-level scheduling starvation in a resource-constrained container
- Any `time.sleep` or synchronous wait inside a registered handler

Once the lock expires, Worker B can acquire it. Worker A's heartbeat will eventually detect this (within `RENEWAL_INTERVAL`, up to 15s after expiry), but the heartbeat's detection is not connected to the main thread. Worker A continues executing. Both workers reach `_dispatch()` and the DB write. The job executes twice.

The 15s renewal interval means there is also a window of up to 15s after lock expiry during which the heartbeat has not yet fired — during this window, Worker A has no signal at all that it has lost the lock, even with a `lock_lost_event` fix applied.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed and open. The heartbeat correctly detects lock loss (Lua token check is sound) but the detection is not connected to the main execution thread. The main thread has no mechanism to abort after lock loss. The irreversible action — `_dispatch()` and the DB write — can execute on two workers simultaneously after a process stall exceeds `CLAIM_TTL`.

Closing this requires a fencing token passed from lock acquisition through to the DB write, validated atomically at write time. Threading signal alone (`lock_lost_event`) is insufficient because it cannot cover the window between lock expiry and the next heartbeat interval, and cannot retroactively abort a `_dispatch()` call already in progress. The architectural fix must live at the write boundary.
