**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie-Write Scenario**

**Name:** Heartbeat-Loss / Zombie-Write

**Failure path, step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or network stall holds the main thread for >60s.
3. The lock expires in Redis. No renewal fires in time (or a renewal fires but the lock has already expired and been evicted — the renewal script finds no key and returns 0).
4. Worker B calls `redis.set(claim_key, token, nx=True, ex=60)` — it succeeds. Worker B acquires the lock with token `T2` and begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`.
6. `_renew()` sees `result == 0` and executes `return` — the thread exits. **No signal is sent to the main thread.**
7. Worker A's main thread resumes from the stall. It has no visibility into the heartbeat thread's exit. It proceeds: `db.get_execution()` → None (B hasn't committed yet), `_dispatch()` completes, `db.transaction()` commits.
8. Worker B also completes and commits.
9. Both workers have executed the same job. For a payment charge, alert fire, or notification send, this is a duplicate operation.

**Why the idempotency check does not save you:** The `db.get_execution()` check and `db.mark_job_complete()` are inside a database transaction, but they are not coupled to continued Redis lock ownership. Worker A reads `None` from the DB (B hasn't written yet, or B's write races), then both proceed. Even if B commits first, A's read already passed the gate. The DB transaction prevents a torn write within one worker; it does not prevent two workers from both passing the pre-execution check under concurrent execution.

**Required fix — two layers, both required:**

*Layer 1 — Thread signaling (necessary but not sufficient):*
`_renew()` must set a `threading.Event` (e.g., `lock_lost_event`) when `result == 0` instead of silently returning. The main thread must check this event at every yield point — before `_dispatch()`, before the DB transaction, and ideally inside `_dispatch()` at handler boundaries. This closes the gap for cooperative workloads but cannot interrupt a thread blocked in a syscall or native extension.

```python
# In _claim_job, add:
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            lock_lost_event.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# yield both:
yield token, lock_lost_event

# In execute(), after every major step:
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting")
```

*Layer 2 — Fencing token / optimistic concurrency at the effect boundary (required for correctness):*
Thread signaling cannot interrupt a thread already past the check point. The only architectural boundary that prevents the duplicate write is making the external effect conditional on still holding the lock at the moment of commit.

The DB transaction must include a lock-ownership assertion. Concretely: write the fencing token into the DB row as part of `record_execution`, and use a conditional write (e.g., `INSERT ... WHERE NOT EXISTS` or an optimistic version check) so that if two workers both reach the commit, only one can succeed. The second worker's transaction must fail and be rolled back.

For external effects that cannot be made conditional (HTTP calls, payment APIs, email sends): the job must be structured so the external call happens inside an idempotency envelope — either the external system is called with an idempotency key derived from `execution_id`, or the external call is deferred to after the DB commit and the DB record is the authoritative gate. The current code calls `_dispatch()` before the DB transaction, meaning the external effect fires before any durable record exists.

**Correct execution order:**
```
1. Acquire lock
2. Check DB for prior execution (idempotency gate)
3. BEGIN DB transaction
4. INSERT execution record (with fencing token, conditional on no prior record)
5. COMMIT — this is the serialization point
6. Execute external effect (using execution_id as idempotency key)
7. Release lock
```

The current code has `_dispatch()` at step 3, before the DB transaction. This must move.

**Finding 2: Renewal Race at Lock Expiry Boundary**

**Name:** Renewal-Miss / Expiry Race

**Failure path:**

The renewal interval is 15s and the TTL is 60s, giving four renewal windows. However, `stop_event.wait(timeout=15)` measures wall time from when the previous iteration completed, not from when the lock was acquired. If the main thread is slow to start (thread scheduling jitter, interpreter startup), or if a renewal call itself blocks on a Redis timeout, the effective renewal cadence can slip. More critically: if a renewal fires at t=59.9s and the lock expired at t=60.0s between the `GET` and the `EXPIRE` in the Lua script, the script returns 0 and the heartbeat exits — but the lock has already been acquired by another worker at t=60.0s. This is the same zombie-write path as Finding 1, triggered by timing rather than a long pause.

**Required fix:** The renewal interval should be ≤ TTL/4 (currently satisfied at 15s/60s), but the TTL should also be set conservatively large relative to expected job duration. More importantly, this is a symptom — the fix is the fencing token at the DB layer (Finding 1, Layer 2). No renewal timing can eliminate the race; it can only reduce its probability.

**Finding 3: `_dispatch()` Executes Before Durable Commit**

**Name:** Effect-Before-Record / Uncommitted External Action

**Failure path:**

```python
result = self._dispatch(req.job_type, req.payload)   # external effect fires here

with self.db.transaction():                           # durable record written here
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

If `_dispatch()` succeeds (payment charged, alert sent, notification delivered) and then `db.transaction()` raises (DB unavailable, constraint violation, network partition), the external effect has occurred but no completion record exists. On retry, `db.get_execution()` returns `None`, the idempotency check passes, and the job executes again. The DB transaction is not the serialization point for the external effect.

**Required fix:** Move `_dispatch()` inside the DB transaction, after the execution record is inserted, or restructure so the external effect is the last step and is idempotent via a stable idempotency key (`execution_id`). If the external system supports idempotency keys, pass `execution_id` to every external call so a retry is a no-op at the destination.

**Finding 4: `t.join()` Timeout Does Not Prevent Release on Hung Renewal**

**Name:** Join-Timeout / Release-Without-Renewal-Confirmation

**Failure path:**

```python
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)   # waits up to 16s
self._release_script(keys=[claim_key], args=[token])
```

If the renewal thread is blocked (Redis call hung, network partition), `join()` returns after 16s and the release script runs. This is correct behavior — you do not want to block forever. However, if the renewal thread is mid-execution of `_renew_script` when `join()` times out, the release script may race with the renewal script. Both are Lua and therefore atomic individually, but their sequencing is not guaranteed. In practice this is low-risk (release after a timed-out join is the right behavior), but the 16-second delay in the `finally` block means lock release is delayed by up to 16s after job completion, holding the lock longer than necessary.

**Required fix:** Reduce join timeout to 2–3s. The renewal thread should exit promptly once `stop_event` is set; a 16s join timeout suggests the thread may be blocked in a Redis call with no timeout configured. Ensure the Redis client has a socket timeout set (e.g., `socket_timeout=5`) so renewal calls do not block indefinitely.

**Finding 5: Test Suite Does Not Cover the Zombie-Write Path**

**Name:** Missing Adversarial Test Coverage

**Failure path:** All tests mock `register_script` to return a script mock with `return_value=1`. No test sets the renewal script mock to return `0` mid-execution and verifies that the main thread aborts. The zombie-write scenario is untested. A future refactor that "cleans up" the `result == 0` branch will have no test failure to catch it.

**Required fix:** Add a test that:
1. Starts a real or simulated `_claim_job` context.
2. Forces the renewal script to return `0` (simulating lock loss).
3. Asserts that the main thread raises `ClaimError` or equivalent before completing the job.
4. Asserts that `db.record_execution` is not called.

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can occur** under the current implementation.

**Trigger:** Worker A stalls (GC pause, slow external call, thread scheduling) beyond `CLAIM_TTL`. Lock expires. Worker B acquires the lock. Worker A's heartbeat detects token mismatch, returns 0, and exits silently. Worker A's main thread continues with no knowledge of lock loss. Both workers execute `_dispatch()` and attempt to commit.

**What prevents it:** Nothing in the current code. The heartbeat exit is silent. The main thread has no signal. The DB idempotency check is a read-before-write that is not serialized against concurrent workers at the DB layer.

**What must be added to close it:**
1. `lock_lost_event` signaling from heartbeat to main thread — closes the cooperative case.
2. Conditional write / fencing token at the DB transaction boundary — closes the case where the main thread is past the check point when lock loss occurs.
3. `_dispatch()` moved after the DB serialization point, or external calls made idempotent via `execution_id` — closes the effect-before-record gap.

All three are required. Any single one in isolation leaves a window.
