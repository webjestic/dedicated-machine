### Finding 1: Heartbeat Thread Exits Silently on Lock Loss — No Signal Path to Main Thread

**Issue name:** Undetected lock loss / zombie execution

**Failure path — step by step:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL = 60s. The heartbeat thread starts, firing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow HTTP call, or I/O stall holds the main thread for >60 seconds.
3. The Redis key expires. `job:claim:{job_id}` no longer exists.
4. Worker B calls `redis.set(claim_key, token_B, nx=True, ex=60)`. The key is gone — `NX` succeeds. Worker B now holds the lock with token `T2`.
5. Worker B's idempotency check (`db.get_execution`) returns `None` — the job has not been recorded yet. Worker B proceeds to `_dispatch()`.
6. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2`, which does not match `T1`. The script returns `0`.
7. The heartbeat thread sees `result == 0` and executes `return`. **The thread exits silently.** No exception is raised. No `threading.Event` is set. No flag is written. The main thread is not notified.
8. Worker A's main thread resumes from the stall. It has no visibility into the heartbeat thread's state. It continues executing as if the lock still holds — completing `_dispatch()`, entering `db.transaction()`, calling `db.record_execution()` and `db.mark_job_complete()`.
9. Worker B completes the same sequence independently.
10. Both workers commit. The job executes twice. For a payment charge, alert dispatch, or external event trigger, this is a correctness violation.

**The exact trigger condition:** Any process stall on Worker A lasting longer than `CLAIM_TTL` (60s) — GC pause, blocking I/O, slow downstream call, thread scheduling delay. This is not a rare edge case; it is a normal operational event in JVM-adjacent runtimes, containerized environments under memory pressure, or any worker making synchronous external calls.

**What the existing boundaries do and do not prevent:**

- The Lua renewal script correctly detects token mismatch and returns 0. ✓
- The heartbeat thread correctly exits on `result == 0`. ✓
- **There is no signal path from the heartbeat thread to the main thread.** The `stop_event` is written only by the main thread (in `finally`); the heartbeat thread never sets it. The main thread never reads the heartbeat thread's exit status. ✗
- The idempotency check (`db.get_execution`) is evaluated *before* execution, not at commit time. It does not prevent a second commit after Worker B has already committed, and it does not protect Worker A's stale write after lock loss. ✗
- The database transaction wraps `record_execution` + `mark_job_complete` atomically — but both workers reach this point, so both transactions land unless the database layer independently enforces uniqueness. ✗

### Finding 2: No Fencing Token at the Database Write Boundary

**Issue name:** Missing write-boundary guard — stale writer can commit

**Why `threading.Event` signaling is necessary but not sufficient:**

Adding `stop_event.set()` inside the heartbeat thread when `result == 0` would cause the main thread to observe lock loss *if* it is checking the event. But:

- The main thread never checks `stop_event` during execution. It would require polling at every yield point — `_dispatch()`, the transaction boundary, etc. — and `_dispatch()` is an arbitrary external call.
- More critically: even if the main thread checks the event and raises an exception, there is a race window. Worker A could pass the event check, then stall, then commit *after* Worker B has already committed. The event check and the database write are not atomic.
- A process that is GC-paused cannot check anything. The signal is only visible when the thread is scheduled. A 90-second GC pause followed by resumption means the main thread wakes up, the event is set, but the thread is already past the last check point.

**`threading.Event` lives at the wrong architectural layer.** It is an intra-process coordination primitive. It cannot make guarantees across process boundaries, and it cannot prevent a write that has already been issued to the database.

**The required fix — fencing token at the database write boundary:**

The database write must be conditioned on proof that the lock is still held at the moment of the write. The architectural pattern is:

**Option A — Optimistic concurrency / fencing token in the database:**

Introduce a monotonic `lock_version` or `claim_token` column on the job record. When Worker A acquires the lock, it writes its token into the job row (or reads the current version). At commit time, the `UPDATE` is conditioned on the token matching:

```sql
UPDATE jobs
SET status = 'complete', execution_id = :exec_id
WHERE job_id = :job_id
  AND claim_token = :token   -- fencing condition
  AND status != 'complete';
```

If Worker B has acquired the lock and written its own token, Worker A's `UPDATE` matches zero rows. The application checks `rowcount == 0` and treats the write as rejected. Worker B's write lands; Worker A's does not. **This is safe regardless of what Worker A believes about its lock state.**

**Option B — Redis token re-verification immediately before write, inside the transaction:**

Before opening the database transaction, re-read the Redis key and verify the token still matches. If it does not, abort. This narrows the race window significantly but does not close it entirely — there is still a gap between the Redis read and the database commit. Option A closes the window completely.

**Option C — Unique constraint on `(job_id, status='complete')`:**

A database-level unique constraint on `job_id` in the completions table means the second `INSERT` raises an integrity error. This prevents duplicate records but requires the application to handle the exception and distinguish "already completed by me" from "completed by another worker." It is a valid backstop but does not give the application clean control over which write wins.

**Why the fix cannot live at the threading layer:**

The threading layer is inside a single process. It has no authority over what another process writes to the database. A fencing token works because the database is the shared resource both workers are writing to — the guard must live at that boundary. The database evaluates the condition atomically within its own transaction isolation, which is the only layer that can make a cross-process serialization guarantee.

### Finding 3: Idempotency Check Is Not Atomic With Execution

**Issue name:** TOCTOU on idempotency guard

**Failure path:**

The check `db.get_execution(req.job_id)` and the subsequent `db.record_execution()` are not in the same transaction. Between the check and the write, another worker can complete the job. This is a secondary issue — the primary lock is supposed to prevent concurrent execution — but once the lock can be lost (Finding 1), this check provides no protection. Even without lock loss, if two workers both pass the `NX` check in rapid succession due to a Redis failover or replication anomaly, both will see `get_execution` return `None` and both will proceed.

**Required fix:** Move the idempotency check inside the database transaction, and enforce it via a unique constraint or a `SELECT FOR UPDATE` on the job row. The check-then-act must be atomic at the database level.

### Finding 4: Test Coverage Does Not Exercise the Failure Mode

**Issue name:** Missing test for lock-loss-during-execution scenario

The test suite mocks `register_script` to return a mock that always returns `1`. There is no test that:
- Simulates the heartbeat receiving `result == 0`
- Verifies whether the main thread is notified
- Verifies that the database write does not proceed after lock loss

The existing tests confirm the happy path and basic contention, but they cannot catch the zombie-write scenario because the mock never simulates token mismatch during execution. This means the structural gap is invisible to CI.

**Required fix:** Add a test that configures the renew script mock to return `0` on the first call (simulating lock expiry during execution), then asserts that `db.record_execution` is not called. This test will fail against the current implementation, which is the correct outcome — it should fail until the fix is in place.

## Summary of Required Fixes

| Layer | Fix |
|---|---|
| Database write boundary | Fencing token: condition `UPDATE`/`INSERT` on `claim_token` matching; check `rowcount`; reject stale writes |
| Database schema | Unique constraint on job completion to enforce single-execution invariant at the storage layer |
| Intra-process signaling | `stop_event.set()` in heartbeat on `result == 0`; main thread checks event before proceeding to write (necessary, not sufficient alone) |
| Idempotency check | Move inside the database transaction; make check-and-write atomic |
| Test coverage | Add test simulating `result == 0` from renew script during execution; assert no DB write occurs |

## Verdict: NOT APPROVED

The zombie-write scenario is real and reachable in production. The trigger is any process stall on Worker A exceeding `CLAIM_TTL` — a normal operational event. The event sequence: lock expires → Worker B acquires → Worker A's heartbeat exits silently → Worker A's main thread continues with no visibility into lock loss → both workers commit. The existing code has no signal path from the heartbeat thread to the main thread, and no guard at the database write boundary that would reject a stale write. For a job executor handling payments, notifications, or external events, this is a critical correctness violation. The fix requires a fencing token at the database write boundary — this cannot be implemented at the threading layer alone.
