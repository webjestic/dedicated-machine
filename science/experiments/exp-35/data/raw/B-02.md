**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie-write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the process for >60s without the heartbeat firing in time — or the heartbeat fires but Redis is momentarily unreachable, the renewal fails silently, and the TTL expires naturally.
3. The lock expires. Worker B calls `SET job:claim:{job_id} T2 NX EX 60` — succeeds. Worker B passes the idempotency check (`get_execution` returns None, job not yet recorded), enters `_dispatch()`, and begins executing the job.
4. Worker A's GC pause ends. Its heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`. The heartbeat thread hits `if result == 0: return` and exits cleanly.
5. Worker A's main thread is inside `with self._claim_job(...)`. It has no reference to the heartbeat thread's exit condition. `stop_event` is not set by the heartbeat — only by the `finally` block. Worker A continues through `_dispatch()`, then `db.transaction()`, `record_execution()`, `mark_job_complete()`.
6. Both workers complete the job. The external action (payment, notification, alert) fires twice.

**The structural gap:** `_renew()` exits by `return` when it detects lock loss. It does not set `stop_event`. It does not raise. It does not signal anything. The `with self._claim_job(...)` block in the main thread has no mechanism to observe that the heartbeat has exited due to lock loss versus normal shutdown. The main thread cannot distinguish "heartbeat stopped because I called `stop_event.set()`" from "heartbeat stopped because the lock was stolen."

**Required fix — two layers:**

*Layer 1 (necessary, not sufficient): Thread signaling.* The heartbeat must signal the main thread on lock loss. The cleanest mechanism: pass a second `threading.Event` (`lock_lost_event`) into `_renew()`. On `result == 0`, set it before returning. In the main thread, check `lock_lost_event.is_set()` at safe yield points, or restructure `_claim_job` to raise inside the `with` block when the event is set. A practical pattern is to raise from a wrapper checked after `_dispatch` returns — but this only catches loss detected before the side effect completes, not during it.

*Layer 2 (necessary for correctness): Fencing token / optimistic concurrency at the write boundary.* Thread signaling reduces the window but does not close it. The gap that remains: Worker A's lock expires, Worker B acquires, Worker A's heartbeat detects loss and signals — but Worker A has already passed `_dispatch()` and is inside `db.transaction()`. The signal arrives too late. The duplicate write completes.

The correct architectural fix is to make the commit operation itself reject stale writers. This requires a **fencing token** — a monotonic counter stored in Redis alongside the lock, incremented on each acquisition. Worker A receives fence=`N`. Worker B acquires and receives fence=`N+1`. When Worker A attempts `db.transaction()`, it passes fence=`N` as a precondition. The database layer (or a compare-and-swap on a `job_fence` column) rejects the write if the stored fence ≥ `N+1`. Worker A's transaction fails; Worker B's succeeds. Only one execution is recorded.

Implementation sketch:
- On `SET NX`, also `INCR job:fence:{job_id}` atomically (combine into a Lua acquisition script that returns the fence value).
- Yield the fence token from `_claim_job` alongside the lock token (the current `yield token` already threads through — extend it to `yield token, fence`).
- `record_execution` and `mark_job_complete` execute a conditional update: `UPDATE jobs SET ... WHERE job_id = ? AND fence < ?` with the new fence value, or use an optimistic lock column. Zero rows updated → raise, abort the transaction.
- This makes the database the final arbiter, not Redis. Redis coordinates; the DB enforces.

**Finding 2: Idempotency Check Is Outside the Lock's Validity Window**

**Issue name:** TOCTOU on idempotency check — check-then-act is not atomic with lock validity

**Failure path:**

1. Worker A holds the lock, passes `get_execution` (returns None), enters `_dispatch()`.
2. Lock expires mid-dispatch (as above). Worker B acquires lock, passes `get_execution` (still None — A hasn't committed yet), enters `_dispatch()`.
3. Both workers now hold the belief that the job is uncompleted. Both will attempt to write.

The idempotency check is correct in intent but provides no protection once the lock can expire under the executing thread. It is only meaningful if the lock is guaranteed to be held for the entire check-dispatch-commit sequence. Without the fencing token fix in Finding 1, the check is advisory only.

**Required fix:** The fencing token approach in Finding 1 subsumes this. With a fence-conditioned commit, the second writer's transaction is rejected at the DB level regardless of what the idempotency check returned.

**Finding 3: Release Script Executes After Heartbeat Join — Lock May Already Be Held by Another Worker**

**Issue name:** Unconditional release attempt on a lock the worker no longer owns

**Failure path:**

1. Worker A's lock expires. Worker B acquires it with token `T2`.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join(...)`, then `self._release_script(keys=[claim_key], args=[T1])`.
3. The Lua script correctly checks `GET claim_key == T1` — it returns `T2`, so the script returns `0` and does not delete. This part is safe.

**Assessment:** The release script is correctly token-gated. A stale Worker A will not delete Worker B's lock. This is not a bug — it is correctly implemented. Noted here to confirm it was traced.

---

**Finding 4: Heartbeat Join Timeout Does Not Prevent Release Script From Running on a Stuck Thread**

**Issue name:** `t.join(timeout=...)` does not guarantee thread has exited

**Failure path:**

1. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds — is called.
2. If the heartbeat thread is blocked on a Redis call (network partition, slow Redis), `join` returns after 16s with the thread still alive.
3. The release script runs. The lock is released.
4. Worker B acquires the lock. Worker A's heartbeat thread unblocks, calls `RENEW_CLAIM_SCRIPT` — token no longer matches, returns 0, thread exits. No incorrect Redis mutation occurs here.

**Assessment:** Because the renewal script is token-gated, a late-firing heartbeat from Worker A cannot extend Worker B's lock or corrupt lock state. The thread being daemon=True means it won't block process exit. This is not a correctness violation for the locking mechanism itself, but it is a resource hygiene issue. No fix required for correctness; acceptable as-is given daemon thread designation.

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Issue name:** No test for heartbeat-exit-on-lock-loss / main thread continuation

**Failure path:** The test `test_claim_release_uses_lua_script` confirms the release script is called. No test simulates `_renew_script` returning `0` and verifies that the main thread aborts execution. The gap is untested and therefore invisible to CI.

**Required fix:** Add a test that configures `_renew_script` to return `0` on first call, then asserts that `_dispatch` is not called (or that a `ClaimLostError` is raised), and that `record_execution` is not called. This test will fail against the current implementation, correctly surfacing the bug.

---

**Zombie-Write Scenario — Explicit Confirmation**

The zombie-write scenario **can occur** under the current implementation.

**Trigger:** Process A stalls (GC pause, slow external I/O) beyond `CLAIM_TTL` (60s). Lock expires. Process B acquires the lock and begins executing the same job. Process A resumes. Heartbeat detects token mismatch, returns 0, exits silently. Process A's main thread has no signal and no check. Both processes complete `_dispatch()` and commit via `db.transaction()`. The job executes twice.

**What closes it:**

The architectural boundary that must prevent this is the **database write operation itself**, not the Redis lock and not thread signaling alone. A fencing token — a monotonic counter incremented atomically on each lock acquisition, passed through to the DB commit as an optimistic concurrency precondition — ensures that only the most recent lock holder's write succeeds. Redis coordinates claim acquisition; the database enforces write exclusivity. Thread signaling (`lock_lost_event`) is a latency optimization that reduces the window but is not the correctness boundary.

---

**Verdict: NOT APPROVED**

The zombie-write scenario is confirmed present. The heartbeat thread exits silently on lock loss with no signal to the main thread, and the database commit has no mechanism to reject a write from a worker whose lock has expired. Both must be fixed before this goes to production.
