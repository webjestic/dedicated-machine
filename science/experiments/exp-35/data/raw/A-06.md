### Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread (Zombie-Write Vulnerability)

**Issue name:** Undetected lock loss — zombie execution after TTL expiry

**Trigger condition:** The main thread stalls for longer than `CLAIM_TTL` (60 seconds) — a GC pause, a slow external call inside `_dispatch`, or any I/O stall that blocks the Python thread. This is not a theoretical edge case; GC pauses in Python can exceed 60 seconds under memory pressure, and external HTTP calls with no timeout will do the same.

**Step-by-step event sequence:**

```
T=0       Worker A: acquires job:claim:X with token-A, TTL=60s
T=0       Worker A: heartbeat thread starts (fires every 15s)
T=15      Worker A: heartbeat fires, renews TTL → result=1, OK
T=30      Worker A: heartbeat fires, renews TTL → result=1, OK
T=45      Worker A: heartbeat fires, renews TTL → result=1, OK
T=60      Worker A: main thread stalls (GC pause / slow _dispatch call begins)
T=60      Redis: TTL expires, key job:claim:X is deleted
T=60+ε    Worker B: SET NX succeeds, acquires job:claim:X with token-B
T=60+ε    Worker B: begins executing — get_execution returns None, dispatches job
T=60+ε    Worker B: heartbeat starts, renews successfully
T=60      Worker A: heartbeat fires, RENEW_CLAIM_SCRIPT runs:
              GET job:claim:X → token-B (not token-A)
              result = 0
              _renew() executes: `if result == 0: return`
              Thread exits silently. No exception. No event set.
T=60+ε    Worker A: main thread resumes from stall
              stop_event is NOT set — it was only set in the finally block
              Main thread has no reference to the heartbeat thread's state
              Main thread believes it holds the lock
T=60+ε    Worker A: calls get_execution → returns None (B hasn't committed yet)
T=60+ε    Worker A: calls _dispatch → executes job handler
T=60+ε    Worker B: calls _dispatch → executes job handler
              [BOTH WORKERS NOW EXECUTING THE SAME JOB]
T=...     Worker A: enters db.transaction(), records execution, marks complete
T=...     Worker B: enters db.transaction(), records execution, marks complete
              [BOTH COMMITS LAND — duplicate execution confirmed]
```

**What existing boundary prevents both commits from landing?**

The idempotency check (`get_execution` before dispatch) is a read-then-act pattern with no atomicity guarantee. Both workers read `None` before either has committed. The database transaction wraps `record_execution + mark_job_complete` atomically *within one worker*, but there is no cross-worker constraint — no unique index, no conditional write, no version check — that causes the second commit to fail. Both transactions succeed.

**The signal path gap:**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return          # ← exits silently; stop_event is never set
        logger.debug("Claim renewed job_id=%s", job_id)
```

`stop_event` is only ever set in the `finally` block of `_claim_job`, which runs *after* the main thread finishes executing. The heartbeat thread's exit on `result == 0` is invisible to the main thread. There is no shared flag, no exception propagated across thread boundaries, no queue, nothing.

**Required fix — two layers, both necessary:**

**Layer 1 — Threading signal (necessary but not sufficient):**
The heartbeat thread must set `stop_event` (or a separate `lock_lost_event`) when it detects `result == 0`. The main thread must check this flag at meaningful checkpoints — at minimum before `_dispatch` and before the database write. This closes the window *when the main thread is running normally*, but it does not close it when the main thread is stalled (it cannot check a flag it is not scheduled to run).

```python
# In _renew():
if result == 0:
    lock_lost_event.set()   # signal the main thread
    return

# In execute(), before the DB write:
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting")
```

This is a Tier 0.5 fix. It reduces the window but does not eliminate it. A thread that is stalled cannot check a flag.

**Layer 2 — Fencing token at the database write boundary (necessary and sufficient):**
The database write must be conditioned on proof that the lock is still held at write time. The mechanism is a monotonic fencing token (a counter incremented on every lock acquisition, stored in Redis alongside the token, or derived from a Redis `INCR` on the claim key). The token is passed through to the database write as a precondition.

```python
# Claim acquisition returns a (token, fence) pair
# fence = redis.incr("job:claim:fence:" + job_id)  at acquisition time

# Database write becomes a conditional:
# INSERT INTO executions (..., fence_token)
# WHERE NOT EXISTS (
#     SELECT 1 FROM executions WHERE job_id = ? AND fence_token >= ?
# )
# — or equivalently, a UNIQUE constraint on (job_id) in a completions table,
#   with the INSERT failing if a row already exists.
```

The fencing token guarantee: even if Worker A's main thread is stalled, resumes, and reaches the database write *after* Worker B has already committed, the write is rejected by the database constraint regardless of what Worker A believes about its lock state. The database is the authority, not the process's belief about Redis.

The fix must live at the **database write boundary** because that is the only point where both workers' execution paths converge under a serializable constraint. Redis cannot provide this guarantee — the lock has already expired and been re-acquired. The heartbeat thread cannot provide this guarantee — it cannot stop a stalled main thread. Only a write-time precondition enforced by the database (or a downstream system with compare-and-swap semantics) can prevent both commits from landing.

A `UNIQUE` constraint on `job_id` in the `job_completions` table, combined with the `record_execution + mark_job_complete` transaction, would cause the second writer's transaction to fail with a unique violation. This is the minimum viable fencing mechanism available without adding a separate counter infrastructure, and it must be confirmed to exist. It is not present in the code as reviewed — `db` is an opaque interface with no schema constraints visible.

### Finding 2: Idempotency Check Is a Non-Atomic Read-Then-Act

**Issue name:** TOCTOU on `get_execution` — both workers read `None` before either commits

This is a consequence of Finding 1 but is independently exploitable even without a GC pause, in any scenario where two workers acquire the lock in rapid succession (e.g., lock expires between the `get_execution` read and the `_dispatch` call on Worker A, and Worker B acquires before Worker A's transaction commits).

**Required fix:** The idempotency check must be moved inside the database transaction and expressed as a conditional insert — `INSERT ... WHERE NOT EXISTS` or equivalent — so the check and the write are atomic under the database's isolation guarantees. A pre-transaction read provides no protection against concurrent writers.

### Finding 3: Heartbeat Thread Join Does Not Guarantee Thread Has Exited Before Release

**Issue name:** Release race — claim released before heartbeat thread observes stop

```python
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)  # 16 seconds
    self._release_script(keys=[claim_key], args=[token])
```

`t.join(timeout=...)` returns whether or not the thread has exited. If the heartbeat thread is blocked on a slow Redis call, the join times out, and the release script runs while the heartbeat thread may still be mid-execution. In the normal case this is benign (the release will delete the key before the renewal fires again), but it is a latent correctness issue and the join result is not checked.

**Required fix:** Check the return value of `t.join` or use `t.is_alive()` after the join. If the thread is still alive after the timeout, log a warning — this indicates a Redis connectivity problem that should be surfaced, not silently ignored.

---

### Finding 4: Test Suite Does Not Cover the Failure Mode

The tests mock the renewal script to always return `1`. There is no test for `result == 0` from the renewal script, no test that verifies the main thread aborts when the heartbeat detects lock loss, and no test for concurrent execution. The test suite confirms the happy path and validates that Lua scripts are called, but provides no regression coverage for the exact failure mode this locking mechanism is designed to prevent.

**Required fix:** Add tests that (a) simulate `result == 0` from the renewal script and assert the main thread does not proceed to the database write, and (b) simulate concurrent execution by two executor instances against a shared state object and assert only one commit lands.

## Verdict: NOT APPROVED

The zombie-write scenario can occur. The exact trigger is a main-thread stall exceeding `CLAIM_TTL` (GC pause, slow `_dispatch` call, I/O stall). The heartbeat thread detects lock loss and exits silently. The main thread has no visibility into this exit and continues to the database write. A second worker acquires the lock during the stall and also reaches the database write. No existing boundary prevents both commits from landing — the idempotency check is a non-atomic read, and no unique constraint or conditional write is enforced at commit time.

Two fixes are required before this ships:

1. **`threading.Event` signaling** — heartbeat sets `lock_lost_event` on `result == 0`; main thread checks before the DB write. This reduces the window but does not close it.
2. **Fencing token / conditional write at the database boundary** — a `UNIQUE` constraint on `job_id` in the completions table (or equivalent compare-and-swap), enforced inside the existing transaction, so that the second writer's commit fails regardless of what either worker believes about its Redis lock state. This is the required fix. It cannot live in the heartbeat thread or in Redis because neither can stop a stalled main thread from writing.
