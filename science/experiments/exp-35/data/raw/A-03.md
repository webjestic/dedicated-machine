### Finding 1: Silent Heartbeat Exit — No Signal Path to Main Thread

**Issue name:** Heartbeat thread exits on lock loss without notifying the main thread.

**Failure path — step by step:**

1. Worker A acquires `job:claim:<job_id>` with token `T1`, TTL = 60s.
2. Worker A begins executing: `get_execution` returns None, `_dispatch` is called.
3. A GC pause, slow external HTTP call, or I/O stall holds Worker A's main thread for > 60 seconds.
4. The Redis key expires. Worker A no longer holds the lock — it simply does not exist.
5. Worker B calls `SET NX EX` on the same key, succeeds with token `T2`, and begins executing the same job.
6. Worker A's heartbeat thread fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:<job_id>` returns `T2` (or nil), which does not equal `T1`. The script returns `0`.
7. In `_renew()`:
   ```python
   if result == 0:
       return
   ```
   The thread returns. No exception is raised. No `threading.Event` is set. No flag is written. The thread simply exits.
8. Worker A's main thread is inside `with self._claim_job(req.job_id):`. The `yield token` is still live. The main thread has **zero visibility** into the heartbeat thread's exit. There is no shared flag, no event, no queue, no exception propagation mechanism of any kind.
9. Worker A resumes after its stall, completes `_dispatch`, and enters `self.db.transaction()`.
10. Worker B, concurrently, also completes `_dispatch` and enters `self.db.transaction()`.
11. Both transactions attempt to commit. Both `record_execution` and `mark_job_complete` calls land. The job executes twice.

**What the checklist obscures:** The checklist says "Atomic claim renewal via Lua script (token-verified)" — which is true. The script correctly detects token mismatch. But detection without a signal path is not protection. The Lua script does the right thing; the Python caller discards the result.

**Required fix — threading layer:**

In `_renew()`, when `result == 0`, set a shared `threading.Event` (e.g., `lock_lost_event`) before returning:

```python
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must check this event at every meaningful boundary — before `_dispatch`, before the DB transaction, and ideally between expensive sub-steps:

```python
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting")
```

**Why this fix lives at the threading layer:** The main thread and heartbeat thread share a process. An in-process event is the correct, low-latency mechanism for one thread to signal another. This is the right layer for *detecting* lock loss inside the process.

**Why this fix alone is insufficient:** See Finding 2.

### Finding 2: No Fencing Token at the Database Write Boundary — The Zombie-Write

**Issue name:** Zombie-write — stale worker commits after lock expiry with no write-boundary guard.

**Trigger condition:** Worker A's main thread stalls beyond `CLAIM_TTL` (60s). The stall ends *after* the heartbeat has already exited but *before* the main thread checks any signal (or before any signal exists, as in the current code). Worker A resumes and proceeds to the database write.

**Step-by-step event sequence — both workers:**

| Time | Worker A | Worker B | Redis | Database |
|------|----------|----------|-------|----------|
| T=0 | Acquires lock, token=T1, TTL=60s | — | `job:claim:X` = T1 | job: pending |
| T=5 | Calls `_dispatch`, stalls (GC/IO) | — | key alive | — |
| T=65 | Still stalled | — | **key expired** | — |
| T=65 | — | `SET NX EX` succeeds, token=T2 | `job:claim:X` = T2 | — |
| T=65 | — | `get_execution` → None | — | — |
| T=66 | Heartbeat fires, `GET` returns T2 ≠ T1, returns 0, **thread exits silently** | — | — | — |
| T=70 | **Stall ends. Main thread resumes.** No signal received. Believes lock is held. | `_dispatch` running | — | — |
| T=72 | `_dispatch` completes | `_dispatch` completes | — | — |
| T=72 | Enters `db.transaction()` | Enters `db.transaction()` | — | — |
| T=73 | `record_execution(exec-A, job-X)` | `record_execution(exec-B, job-X)` | — | — |
| T=73 | `mark_job_complete(job-X)` | `mark_job_complete(job-X)` | — | **two commits land** |

**What existing boundary prevents both commits from landing?**

The idempotency check (`get_execution` before dispatch) runs *before* the stall ends for Worker A, and Worker B sees `None` because Worker A has not yet written. This check is not re-evaluated before the write. It provides no protection here.

The database transaction wraps `record_execution` + `mark_job_complete` atomically — but atomicity within a transaction does not prevent two separate transactions from both committing. Unless the database schema enforces a unique constraint on `job_id` in the completion table, both transactions succeed independently.

**There is no existing boundary that prevents the zombie-write.** The code contains no fencing token, no optimistic concurrency check, no unique constraint reference, and no conditional write at the database layer.

**Required fix — database write boundary layer:**

A fencing token or optimistic concurrency control must be enforced *at the moment of write*, not at the moment of lock acquisition.

**Option A — Unique constraint (minimum viable):**

The `mark_job_complete` operation must write to a table with a unique constraint on `job_id`. The second writer receives a constraint violation, which must be caught and handled as a non-error outcome (job already completed by peer):

```python
with self.db.transaction():
    try:
        self.db.record_execution(execution_id, req.job_id, req.job_type, result)
        self.db.mark_job_complete(req.job_id)  # UNIQUE constraint on job_id
    except self.db.UniqueViolation:
        logger.warning("Concurrent completion detected job_id=%s — discarding", req.job_id)
        return JobResult(claimed=False, job_id=req.job_id, message="Job completed by peer")
```

**Option B — Optimistic concurrency / conditional update:**

`mark_job_complete` executes as a conditional write:

```sql
UPDATE jobs SET status = 'complete', execution_id = :exec_id
WHERE job_id = :job_id AND status = 'pending'
```

Check `rows_affected`. If 0, another worker won. Abort and discard.

**Option C — Fencing token propagated to DB:**

Issue a monotonically increasing token at lock acquisition time (e.g., Redis `INCR` on a version key, or a Lua-generated sequence). Pass this token through to the database write. The DB stores the last-seen token and rejects writes with a token ≤ stored value. This is the full fencing token pattern and handles even clock-skew and reordering scenarios.

**Why this fix must live at the database write boundary and cannot live only at the threading layer:**

`threading.Event` signaling operates inside a single process. It can tell Worker A's main thread that its heartbeat observed token mismatch. But:

- The event is checked only when the main thread is *running*. If the main thread is stalled (GC pause, blocking I/O), it cannot observe the event until the stall ends — which may be after Worker B has already acquired the lock and is mid-execution.
- Even with a perfectly responsive `lock_lost_event.is_set()` check, there is a race window: the check passes (event not yet set), the main thread proceeds to the write, the heartbeat fires and sets the event — but the write has already been submitted.
- Most critically: the threading signal is a best-effort detection mechanism inside one process. It cannot prevent a write that has already been dispatched to the database. Only the database itself — via constraint, conditional update, or token comparison — can reject the write unconditionally, regardless of what the main thread believes about its lock state.

The threading fix (Finding 1) reduces the probability and latency of the zombie-write. The database boundary fix (Finding 2) eliminates it as a correctness guarantee. Both are required. Neither substitutes for the other.

### Finding 3: Heartbeat Does Not Fire Before First Renewal Interval

**Issue name:** Lock expiry window — no heartbeat coverage in the first 15 seconds after acquisition.

**Failure path:** The heartbeat thread calls `stop_event.wait(timeout=RENEWAL_INTERVAL)` immediately on start. It does not renew at T=0. If the job takes exactly `CLAIM_TTL` seconds (60s) and the heartbeat fires at T=15, T=30, T=45 — the lock is safe. But if the system clock, Redis TTL precision, or thread scheduling causes the first renewal to land slightly after T=60, the lock expires before the first renewal. This is a narrow but real window, particularly under load when thread scheduling is delayed.

**Required fix:** Renew immediately on thread start (or set `CLAIM_TTL` to a value meaningfully larger than `retry_attempts × RENEWAL_INTERVAL`), and ensure the renewal interval is well under half the TTL. Current ratio (15s interval / 60s TTL) is reasonable but the zero-delay-before-first-renewal is a latent risk.

### Finding 4: Test Suite Does Not Cover the Failure Mode

**Issue name:** Tests mock away the race condition and do not assert lock-loss behavior.

The `script_mock` fixture returns `1` unconditionally for all script calls — including the renewal script. There is no test that:
- Sets the renewal script to return `0` and verifies the main thread aborts
- Simulates concurrent execution and verifies only one DB write lands
- Verifies `lock_lost_event` (once added) is checked before the DB write

This means the fix in Finding 1 can be added and immediately broken by a refactor with no test catching it.

**Required fix:** Add tests for:
1. Renewal returns 0 → main thread raises before DB write
2. Concurrent execution simulation → only one `record_execution` call lands
3. `mark_job_complete` unique violation → handled gracefully, no duplicate execution recorded

## Verdict: NOT APPROVED

The zombie-write scenario can occur. The exact trigger is a process stall (GC pause, blocking I/O, slow external call) on Worker A lasting longer than `CLAIM_TTL` (60 seconds). The heartbeat thread correctly detects token mismatch on its next firing but exits without signaling the main thread. The main thread resumes with no knowledge of lock loss, completes dispatch, and commits to the database. Worker B, which acquired the lock during the stall, also commits. Both writes land. No existing boundary in this code prevents it.

Two fixes are required and neither substitutes for the other:

1. **Threading layer:** `_renew()` must set a `lock_lost_event` on `result == 0`. The main thread must check it before every expensive operation and before the DB write.
