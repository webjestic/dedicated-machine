### Finding 1: Silent Heartbeat Exit — No Signal Path to Main Thread

**Issue name:** Undetected lock loss / zombie execution

**Failure path, step by step:**

1. Worker A acquires `job:claim:<job_id>` with token `T1`, TTL = 60s. The heartbeat thread starts.
2. Worker A's main thread enters `_dispatch()`. A GC pause, slow external HTTP call, or I/O stall holds the main thread for > 60 seconds.
3. The Redis key expires. The lock is now vacant.
4. Worker B calls `redis.set(claim_key, token_B, nx=True, ex=60)` — it succeeds. Worker B now holds the lock and begins executing the same job.
5. Worker A's heartbeat thread fires at its next `RENEWAL_INTERVAL` (15s). `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:<job_id>` returns `token_B`, which does not equal `T1`. The script returns `0`.
6. The heartbeat's `if result == 0: return` branch executes. **The thread exits silently.** `stop_event` is never set. No exception is raised. No flag is written. The main thread has zero visibility into this event.
7. Worker A's main thread resumes after the stall. It proceeds through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`. It believes it still holds the lock — nothing has told it otherwise.
8. Worker B independently completes the same sequence and commits.
9. **Both commits land.** The job executes twice.

**What prevents this:** Nothing. The `stop_event` is only written by the `finally` block of `_claim_job`, which is controlled by the main thread. The heartbeat thread reads `stop_event` but never writes it. The signal path is strictly one-directional: main → heartbeat. There is no reverse channel.

**Required fix — two layers, both necessary:**

**Layer 1 (threading): Reverse signal path.**
The heartbeat thread must write to a shared flag when it detects lock loss. The main thread must check that flag at every meaningful boundary — before `_dispatch`, before entering the DB transaction, and before committing.

```python
# In _claim_job:
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost.set()   # signal the main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# In execute(), before each phase boundary:
if lock_lost.is_set():
    raise ClaimError("Lock lost during execution — aborting")
```

**Why this layer is necessary but not sufficient:** It closes the window when the main thread is actively running between heartbeat intervals. But if the main thread is stalled (GC pause, blocking I/O), it cannot check the flag. The flag is only observed when the main thread is scheduled. A sufficiently long stall means the main thread resumes, the flag is set, but the DB write has already happened before the check. The threading layer is detection, not prevention.

**Layer 2 (database write boundary): Fencing token / optimistic concurrency.**
The database write must be conditioned on proof that the lock is still held at write time. This is the only layer that provides a hard boundary regardless of what the main thread believes about its lock state.

The mechanism: pass the Redis token (or a monotonic lock version) into the DB transaction as a precondition. The `mark_job_complete` write must be a conditional update:

```sql
UPDATE jobs
SET status = 'complete', execution_id = :execution_id, lock_token = :token
WHERE job_id = :job_id
  AND status != 'complete'
  AND (lock_token IS NULL OR lock_token = :token)
```

If Worker A's token `T1` no longer matches (because Worker B wrote `T2`, or because the row was already committed by Worker B), Worker A's update affects 0 rows. The application checks `rowcount == 0` and raises, rolling back the transaction.

Alternatively: use a monotonic lock version counter. Redis stores an integer version alongside the token. Each acquisition increments it. The DB write includes `WHERE lock_version = :acquired_version`. Only the writer who holds the current version can commit.

**Why this fix must live at the DB write boundary and cannot live only at the threading layer:**

The threading `Event` is a cooperative, in-process signal. It only works when the main thread is scheduled and executing Python bytecode. It cannot interrupt a blocking syscall, a GC pause, or a native extension holding the GIL. A process that stalls for 61 seconds and then resumes will find `lock_lost.is_set() == True` — but only *after* it resumes. If the DB write executes in that same resumed burst before the check, the threading layer provides no protection. The fencing token at the DB boundary is enforced by the database engine, unconditionally, regardless of thread scheduling, GC behavior, or what the application believes about its lock state. These are different guarantees at different layers; neither substitutes for the other.

### Finding 2: Idempotency Check Is Not Atomic With the Write

**Issue name:** TOCTOU race on `get_execution` / `record_execution`

**Failure path:**

1. Workers A and B both fail to acquire the Redis lock on first attempt but both succeed on retry (e.g., a prior lock expired between retries).
2. Both workers pass the `get_execution` check — neither sees a completed record yet.
3. Both proceed to `_dispatch()` and then to `db.transaction()`.
4. Both attempt `record_execution` + `mark_job_complete`.

The idempotency check (`get_execution` returning `None`) and the write (`record_execution`) are not atomic. Without a unique constraint on `job_id` in the executions table (or an equivalent DB-level guard), both writes can succeed.

**Required fix:** The `jobs` table must have a unique constraint or a conditional insert that enforces exactly-once semantics at the DB level:

```sql
-- Option A: unique constraint on job_id in executions table
ALTER TABLE job_executions ADD CONSTRAINT uq_job_execution UNIQUE (job_id);

-- Option B: conditional insert
INSERT INTO job_executions (job_id, execution_id, ...)
SELECT :job_id, :execution_id, ...
WHERE NOT EXISTS (SELECT 1 FROM job_executions WHERE job_id = :job_id);
-- check affected rows == 1
```

The application-level `get_execution` check is a useful optimization (avoids unnecessary work) but cannot be the correctness boundary. The DB constraint is the correctness boundary.

### Finding 3: Heartbeat Thread Exit Is Not Logged

**Issue name:** Silent failure — operational invisibility

**Failure path:** When `result == 0`, the heartbeat thread returns with no log line. In production, lock loss is a high-severity event — it means a second worker may be executing the same job. The current code makes this invisible to on-call.

**Required fix:**

```python
if result == 0:
    logger.error(
        "Claim renewal failed — token mismatch, lock lost job_id=%s. "
        "Another worker may have acquired this job.",
        job_id
    )
    lock_lost.set()
    return
```

This is an `error`-level event, not `debug` or `warning`.

### Finding 4: Test Suite Does Not Cover the Failure Mode

**Issue name:** Missing test for lock-loss-during-execution

**Observation:** The test suite covers claim contention at acquisition time (`test_execute_claim_contention`) but has no test for the scenario where the lock is lost *after* acquisition — i.e., where `_renew_script` returns `0` mid-execution. The mock in the fixture sets `register_script.return_value` to a mock returning `1` unconditionally. There is no test asserting that the main thread aborts when the heartbeat detects token mismatch.

**Required fix:** Add a test that configures the renew script mock to return `0`, triggers execution, and asserts that the DB write does not occur (or that a `ClaimError` is raised):

```python
def test_execute_aborts_on_lock_loss(redis_client, db):
    renew_mock = MagicMock(return_value=0)   # simulates token mismatch
    release_mock = MagicMock(return_value=1)
    redis_client.register_script.side_effect = [release_mock, renew_mock]
    ex = JobExecutor(redis_client, db)

    # needs enough delay for heartbeat to fire — or inject lock_lost directly
    result = ex.execute("job-001", "notify")

    assert result.claimed is False
    db.record_execution.assert_not_called()
```

## Verdict: **NOT APPROVED**

The zombie-write scenario is real and reachable in production.

**Trigger condition:** Any stall on Worker A's main thread exceeding `CLAIM_TTL` (60s) — GC pause, blocking external call, slow DB query, network timeout.

**Event sequence:**
1. Worker A acquires lock, stalls > 60s → lock expires
2. Worker B acquires same lock, begins execution
3. Worker A's heartbeat detects token mismatch → exits silently
4. Worker A resumes, has no visibility into heartbeat exit
5. Both workers complete `_dispatch()` and enter `db.transaction()`
6. Both commits land → duplicate execution

**What prevents both commits from landing:** Nothing in the current code.

**What must be added:**
- **Threading layer:** `lock_lost` event written by heartbeat, checked by main thread before the DB write. Closes the cooperative detection gap.
- **DB write boundary:** Fencing token or unique constraint enforced by the database, conditioned on lock token or job state. This is the hard boundary that holds regardless of thread scheduling or GC behavior. It is the required fix for the zombie-write scenario. The threading layer alone is not sufficient.
