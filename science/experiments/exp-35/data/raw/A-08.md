### Finding 1: Silent Heartbeat Exit — No Signal Path to Main Thread

**Issue name:** Heartbeat-exits-silently-on-lock-loss

**Failure path, step by step:**

1. Worker A acquires `job:claim:<job_id>` with token T1, TTL=60s. The heartbeat thread starts.
2. Worker A's main thread enters `_dispatch()` — a slow external call, GC pause, or I/O stall holds it for >60 seconds.
3. The Redis key expires. Worker A no longer holds the lock.
4. Worker B calls `redis.set(claim_key, T2, nx=True, ex=60)` — it succeeds. Worker B now holds the lock legitimately.
5. Worker B passes the idempotency check (`get_execution` returns None — nothing has been committed yet), dispatches the job, and begins executing.
6. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:<job_id>` returns T2, not T1. The script returns 0.
7. The `_renew` function sees `result == 0` and executes `return` — the thread exits silently.
8. **There is no signal back to the main thread.** `stop_event` is never set by the heartbeat thread. The main thread has no `.join()` check, no shared flag, no exception raised into it.
9. Worker A's main thread resumes after the stall. It is still inside the `with self._claim_job(...)` block. It believes it holds the lock. It proceeds to `self.db.transaction()` and calls `record_execution` + `mark_job_complete`.
10. Worker B does the same.
11. Both commits land. The job executes twice.

**The exact structural gap:** `_renew` returns on `result == 0` but sets nothing. The main thread has no mechanism to observe this. The `stop_event` is write-only from the main thread's perspective — the heartbeat thread never writes to it.

**Required fix — two layers, both necessary:**

**Layer 1 (threading signal — necessary but not sufficient):** The heartbeat thread must signal the main thread on lock loss:

```python
# In _renew():
if result == 0:
    logger.error("Lock lost — token mismatch. Signaling main thread. job_id=%s", job_id)
    lost_event.set()   # a second Event, written by heartbeat, read by main thread
    return

# In the main thread (execute), after _dispatch returns and before the DB write):
if lost_event.is_set():
    raise ClaimError("Lock was lost during execution — aborting commit")
```

This closes the gap *when the main thread is running and checks the flag*. It does not close the gap when the main thread is mid-commit or when the check races with the write. It is Tier 0.5: correct diagnosis, incomplete fix.

**Layer 2 (fencing token at the DB write boundary — required for correctness):** The threading signal cannot protect against a commit that is already in flight, or a check that passes a microsecond before the flag is set. The only boundary that is guaranteed to be evaluated *at the moment of the write* is the database itself.

The fix is a **fencing token / optimistic concurrency check** at the `db.transaction()` boundary:

- Before acquiring the Redis lock, read a monotonic version counter for the job (e.g., `job_version = db.get_job_version(job_id)`).
- Pass that version into the transaction as a precondition: `UPDATE jobs SET status='complete', version=version+1 WHERE job_id=? AND version=?`
- If the `WHERE` clause matches zero rows, the transaction must raise and roll back.

Worker A and Worker B will both attempt to write with the version they read. Only one `UPDATE` can match the row — the second one finds `version` already incremented and fails. This is the only mechanism that prevents both commits from landing regardless of what either worker believes about its lock state.

**Why the fix must live at the DB layer and cannot live only at the threading layer:**

The threading signal is intra-process. It can only observe lock loss *if the heartbeat fires and the main thread checks the flag before committing*. It cannot observe:
- A commit already in progress when the flag is set
- A race where the flag is checked, passes, and the commit begins before the flag is set
- Any scenario where the main thread is paused (the same GC/IO stall that caused the original problem) between the check and the write

The database write boundary is the only point in the system where both workers' execution paths converge on a single serializable resource. A conditional write (`UPDATE ... WHERE version = N`) evaluated under the database's own concurrency control is the only guarantee that is immune to process-level timing.

### Finding 2: Idempotency Check Is Not Inside the Lock's Validity Window

**Issue name:** TOCTOU on idempotency check

**Failure path:**

Even setting aside the GC-pause scenario, the idempotency check (`get_execution`) is a read that happens *after* lock acquisition but the lock's validity is not re-confirmed before the write. If the lock expires between the check and the write (possible in any slow-dispatch scenario), the check's result is stale by the time the write executes. This is a narrower version of Finding 1 but is structurally distinct: the check-then-act is not atomic with respect to lock validity.

**Required fix:** Same fencing token as Finding 1. The conditional `UPDATE ... WHERE version = N` replaces the need for a separate idempotency read-then-check pattern, because the write itself is the atomic test-and-set.

### Finding 3: Heartbeat Join Does Not Verify Lock State Before Release Attempt

**Issue name:** Release-after-expiry is a no-op but creates false confidence

**Failure path:**

In the `finally` block:
```python
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

If the lock has expired and Worker B holds it, `RELEASE_CLAIM_SCRIPT` correctly returns 0 (token mismatch) and does not delete Worker B's key. This part is safe. However, the return value of `_release_script` is not checked. A caller reading this code sees "claim released" in the log unconditionally, regardless of whether the release was a no-op. This is a logging correctness issue that masks the failure mode during incident investigation.

**Required fix:** Check the return value and log distinctly:
```python
released = self._release_script(keys=[claim_key], args=[token])
if released:
    logger.info("Claim released job_id=%s", job_id)
else:
    logger.error("Claim release was a no-op — lock had already expired or been taken job_id=%s", job_id)
```

### Finding 4: Test Suite Does Not Cover the Failure Mode

**Issue name:** Missing test for lock-loss-during-execution

**Observation:** Every test mocks `register_script` to return a script mock with `return_value=1`. No test sets `return_value=0` on the renew script to simulate lock loss mid-execution and then verifies that the main thread does not proceed to commit. The test suite passes entirely because it never exercises the failure path. A test that simulates `result == 0` from the renewal script and asserts that `db.record_execution` is *not* called would have caught Finding 1 at review time.

**Required fix:** Add a test:
```python
def test_execute_aborts_if_lock_lost_during_dispatch(redis_client, db):
    # Simulate lock loss: renew returns 0
    renew_mock = MagicMock(return_value=0)
    release_mock = MagicMock(return_value=0)
    redis_client.register_script.side_effect = [release_mock, renew_mock]
    
    # Simulate slow dispatch so heartbeat fires
    def slow_handler(payload):
        time.sleep(JobExecutor.RENEWAL_INTERVAL + 1)
        return {"sent": True}
    
    db.get_handlers.return_value = {"notify": slow_handler}
    ex = JobExecutor(redis_client, db)
    result = ex.execute("job-001", "notify")
    
    assert result.claimed is False  # should have aborted
    db.record_execution.assert_not_called()
```

This test will fail against the current implementation, which is the correct outcome — it surfaces the bug.

## Verdict: **NOT APPROVED**

The zombie-write scenario is real and reachable in production.

**Trigger condition:** Any process stall on Worker A longer than `CLAIM_TTL` (60s) — GC pause, slow external HTTP call, DNS timeout, I/O stall.

**Event sequence:**
1. Worker A acquires lock, begins dispatch, stalls >60s
2. Redis key expires
3. Worker B acquires same key, passes idempotency check (nothing committed), begins dispatch
4. Worker A's heartbeat sees token mismatch, exits silently — no signal to main thread
5. Worker A resumes, proceeds to `db.transaction()`, commits
6. Worker B completes dispatch, proceeds to `db.transaction()`, commits
7. Both `record_execution` and `mark_job_complete` are called twice

**What prevents both commits from landing:** Nothing in the current implementation. The Lua release script is token-verified but that only protects the *release*, not the *write*. The idempotency check is a read that is stale by the time the second write executes.

**What must be added:**
1. A `lost_event` threading.Event written by the heartbeat thread on `result == 0`, checked by the main thread before the DB write — closes the common case
2. A fencing token / optimistic concurrency `UPDATE ... WHERE version = N` at the database write boundary — closes the race unconditionally, regardless of thread timing or process pause behavior

Both are required. The threading signal alone is insufficient. The fencing token at the DB boundary is the definitive fix.
