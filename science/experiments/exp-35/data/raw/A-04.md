## Findings

---

### Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie-Write Enabled

**Issue name:** Undetected lock loss / zombie-write

**Trigger condition:** The main thread stalls (GC pause, slow external call, blocking I/O, or any other cause) for longer than `CLAIM_TTL` (60 seconds) while holding the job lock.

**Step-by-step event sequence:**

1. **Worker A** acquires `job:claim:{job_id}` with token `T_A`, TTL = 60s. Heartbeat thread starts with `RENEWAL_INTERVAL = 15s`.
2. Worker A's main thread enters `_dispatch()` — or any blocking call — and stalls for > 60 seconds (GC pause, slow DB call, network wait, etc.).
3. The Redis key `job:claim:{job_id}` expires naturally. No process deleted it; Redis evicted it on TTL expiry.
4. **Worker B** calls `redis.set(claim_key, T_B, nx=True, ex=60)`. The key is absent. Worker B acquires the lock with token `T_B`. Worker B begins executing the same job: checks `db.get_execution()` (returns `None` — job not yet complete), calls `_dispatch()`, proceeds toward `db.transaction()`.
5. Worker A's heartbeat thread fires (at the next 15s boundary). `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T_B` (Worker B's token), which does not equal `T_A`. The script returns `0`.
6. The `_renew` function observes `result == 0` and **returns silently**. The thread exits. No exception is raised. `stop_event` is not set. No signal is sent to the main thread.
7. Worker A's main thread resumes from its stall. It has no reference to the heartbeat thread's state. `stop_event` is still clear from the main thread's perspective — the main thread never checks it. Worker A continues executing: calls `db.transaction()`, `db.record_execution()`, `db.mark_job_complete()`.
8. Worker B also completes and commits.
9. **Both workers commit.** The job executes twice. For a payment charge, alert dispatch, or external event trigger, this is a correctness violation.

**What existing boundary prevents both commits from landing?**

The `db.get_execution()` idempotency check runs *before* `_dispatch()`, not at commit time. It is a read-check-then-write pattern with no enforcement at the write boundary. If both workers pass the check before either commits (which is exactly what happens here — Worker B reads `None` before Worker A has written), both proceed. The database transaction wraps `record_execution` + `mark_job_complete` atomically *within one worker*, but provides no cross-worker exclusion at commit time. There is no constraint, unique index, or conditional write that causes the second commit to fail.

**Failure path summary:**

```
stall > CLAIM_TTL
  → Redis key expires
  → Worker B acquires lock
  → Worker B passes idempotency check (nothing written yet)
  → Worker A resumes, heartbeat already exited silently
  → Worker A passes idempotency check (nothing written yet, or races)
  → Both workers call _dispatch()
  → Both workers commit to DB
  → Duplicate execution
```

---

**Required Fix — two layers, both necessary, at different architectural boundaries:**

**Layer 1 (in-process signaling) — `threading.Event` between heartbeat and main thread:**

This is a necessary improvement but is *not sufficient alone*. The heartbeat must set `stop_event` (or a separate `lock_lost_event`) when `result == 0`, and the main thread must check it at safe checkpoints. This closes the window for a *cooperative* process that checks its own state. However, it cannot protect against a main thread that is stalled and not checking, and it cannot prevent a commit that races through before the signal is observed.

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must then check `lock_lost_event` before the DB write and raise if set. This is Tier 0.5: correct diagnosis, incomplete fix. A stalled main thread will not check the event until it resumes — by which time Worker B may have already committed.

**Layer 2 (write boundary) — fencing token / optimistic concurrency at the database write:**

This is the only fix that provides a hard correctness guarantee regardless of what the main thread believes about its lock state. The fix must live at the database write boundary because that is the only point where both workers' writes converge and where a conflict can be detected and rejected atomically.

The mechanism: use a monotonic generation counter (fencing token) stored alongside the job record. When Worker A acquires the lock, it reads the current generation `G`. When it attempts to commit, it writes conditionally: `UPDATE jobs SET status='complete', generation=G+1 WHERE job_id=? AND generation=G AND status='pending'`. If Worker B has already committed and incremented the generation to `G+1`, Worker A's conditional write matches zero rows and fails. The application layer checks affected row count and treats zero as a lost-lock abort.

This cannot live at the Redis layer because Redis has already expired the key — the lock is gone before the write occurs. It cannot live at the heartbeat layer because the heartbeat thread cannot interrupt a stalled main thread. It cannot live at the idempotency-check layer because that check is not atomic with the write. It must live at the write boundary, expressed as a conditional update that is atomic in the database.

```python
# At commit time, not just at the pre-check:
rows_affected = self.db.conditional_mark_complete(
    job_id=req.job_id,
    expected_generation=generation,   # read at claim time
)
if rows_affected == 0:
    raise StaleClaimError("Lock was lost before commit; aborting.")
```

**Why `threading.Event` alone is Tier 0.5:**

It detects lock loss *inside the process* when the heartbeat fires and the main thread is running and checking. It does not prevent a commit from a main thread that resumed from a stall after the heartbeat already exited, if the main thread does not check the event before writing. The fencing token at the DB write boundary rejects the stale write unconditionally — it does not depend on the main thread's awareness of its lock state.

**Both fixes together close the zombie-write scenario completely:**
- `threading.Event` provides fast in-process detection and early abort for the cooperative case.
- The fencing token at the DB write boundary provides a hard backstop that catches the case where the main thread was stalled and missed the signal.

### Finding 2: `stop_event` Is Not Exposed to the Main Thread for Lock-Loss Detection

**Issue name:** No signal path from heartbeat thread to main thread.

**Failure path:** As described in Finding 1, step 6–7. The heartbeat's `stop_event` is used only as a *shutdown* signal from main → heartbeat. There is no reverse signal path. The main thread has no way to know the heartbeat exited due to lock loss vs. normal shutdown.

**Required fix:** Introduce a separate `lock_lost_event = threading.Event()`. Set it in `_renew` when `result == 0`. Check it in `execute()` before the DB write. Raise a `ClaimError` (or `StaleClaimError`) if set. This is necessary but not sufficient — see Finding 1, Layer 2.

---

### Finding 3: Idempotency Check Is Not Atomic With the Write

**Issue name:** Read-check-then-write TOCTOU on job completion state.

**Failure path:** Two workers both pass `db.get_execution()` returning `None` before either has committed. Both proceed to `_dispatch()` and both attempt to write. The check provides no protection once two workers are past it concurrently.

**Required fix:** Replace the pre-check + unconditional write pattern with a conditional write (unique constraint on `job_id` in the executions table, or a `WHERE status='pending'` conditional update). The database must enforce uniqueness at write time, not just at read time. This is the same fencing-token layer described in Finding 1, Layer 2 — they are the same architectural fix.

---

### Finding 4: Test Suite Does Not Cover the Failure Mode

**Issue name:** Missing test for lock-loss-during-execution scenario.

**Failure path:** The test `test_execute_claim_contention` covers the case where `redis.set` returns `False` at acquisition time. No test simulates `_renew_script` returning `0` mid-execution and verifies that the main thread aborts rather than commits. The gap means the fix for Finding 1 can be omitted or regressed without a failing test.

**Required fix:** Add a test that configures `_renew_script` to return `0` on the first renewal call while `_dispatch` is running (use a threading barrier or side-effect mock), and asserts that `db.record_execution` is *not* called and the result indicates failure. This test will fail against the current code and pass only after the `lock_lost_event` signal path is implemented.

## Verdict: NOT APPROVED

The zombie-write scenario is real and reachable in production. A GC pause, slow external call, or any blocking operation exceeding 60 seconds causes the Redis lock to expire, allows a second worker to acquire the same lock and begin executing the same job, and the first worker's heartbeat exits silently with no signal to the main thread. Both workers commit. There is no existing boundary — not the Redis lock, not the idempotency check, not the DB transaction — that prevents both writes from landing.

Two fixes are required before approval:

1. **`threading.Event` reverse signal** from heartbeat to main thread on `result == 0`, with main thread checking before the DB write. (In-process detection layer.)
2. **Fencing token / conditional write** at the database write boundary — a `WHERE generation=N AND status='pending'` conditional update that causes the second committer's write to fail atomically. (Write boundary enforcement layer.)

Fix 2 is the load-bearing correctness guarantee. Fix 1 is defense-in-depth. Neither alone is sufficient. Both together close the zombie-write scenario.
