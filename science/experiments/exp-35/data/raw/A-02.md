### Finding 1 — Silent Heartbeat Exit / No Signal Path to Main Thread (Zombie-Write Enabler)

**Issue name:** Heartbeat thread exits silently on lock loss; main thread has no visibility and continues executing.

**Failure path, step by step:**

1. Worker A acquires `job:claim:<job_id>` with token `T_A`, TTL = 60s. Heartbeat thread starts with `RENEWAL_INTERVAL = 15s`.
2. Worker A's main thread enters `_dispatch()` — a slow external call, GC pause, or I/O stall holds the thread for >60 seconds.
3. The Redis key expires. `job:claim:<job_id>` no longer exists.
4. Worker B calls `redis.set(claim_key, token_B, nx=True, ex=60)` — the key is gone, so `SET NX` succeeds. Worker B now holds the lock with token `T_B`.
5. Worker A's heartbeat fires (at the 15s or 30s or 45s mark after expiry). `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:<job_id>` returns `T_B`, which does not equal `T_A`. The script returns `0`.
6. The heartbeat thread sees `result == 0` and executes `return` — it exits silently. No exception is raised. `stop_event` is never set. The main thread is not notified in any way.
7. Worker A's main thread resumes (the stall ends). It is still inside the `with self._claim_job(...)` block. It has no mechanism to observe that the heartbeat thread has exited. It proceeds: `_dispatch()` completes, `self.db.transaction()` opens, `record_execution` and `mark_job_complete` are called.
8. Worker B is executing the same job concurrently and will do the same.
9. Both workers complete the job. For a job that sends a notification, charges a payment, or fires an external event, this is a correctness violation.

**What exists that might prevent this:** The `db.get_execution()` idempotency check at the top of the `with` block. However, this check runs *before* `_dispatch()`, not after. If both workers pass the check before either commits, the check provides no protection. The check is a read, not a write with a uniqueness constraint — it is not atomic with the subsequent write.

**Required fix — two layers, both necessary:**

**Layer 1 (threading — necessary but insufficient):** `stop_event` must be set by the heartbeat thread when `result == 0`, and the main thread must check it before proceeding past any point of no return. A `threading.Event` named `lock_lost` should be set by `_renew()` on `result == 0`. The main thread should check this event before calling `_dispatch()` and before opening the database transaction. This closes the window *when the signal arrives in time* — but it cannot close the window when the main thread is stalled and the check never runs.

**Layer 2 (database write boundary — necessary and sufficient for correctness):** A fencing token or optimistic concurrency check must exist at the database write boundary. This is the only layer that can prevent both commits from landing regardless of what either worker believes about its lock state. Concretely:

- `record_execution` should be guarded by a `UNIQUE` constraint on `job_id` in the executions table. The first `INSERT` wins; the second raises a unique constraint violation, which the transaction catches and converts to a no-op or error.
- Alternatively, `mark_job_complete` should use an optimistic update: `UPDATE jobs SET status='complete', version=version+1 WHERE job_id=? AND status='pending' AND version=?` where the version was read before `_dispatch()`. If Worker B has already committed, Worker A's update affects 0 rows, which must be detected and cause the transaction to abort.

The fix **cannot live only at the threading layer** because a `threading.Event` check is conditional on the main thread being scheduled and running. A thread that is stalled in a blocking syscall or GC pause will not check the event. The database write boundary is unconditional — it fires regardless of thread scheduling, pause duration, or process state.

The fix **cannot live only at the Redis layer** because by the time the lock has expired and been re-acquired by Worker B, the Redis state is already consistent (Worker B holds it). The problem is that Worker A is still executing with a stale belief about its own lock ownership. Redis cannot retroactively prevent Worker A's database write.

### Finding 2 — Idempotency Check Is a Non-Atomic Read (TOCTOU)

**Issue name:** `get_execution()` check is not atomic with `record_execution()` write.

**Failure path:**

1. Workers A and B both acquire the lock at different times (see Finding 1), or in a scenario where the lock is not the coordination mechanism (e.g., lock key evicted under memory pressure before TTL).
2. Both workers call `self.db.get_execution(req.job_id)` and both observe `None`.
3. Both proceed to `_dispatch()` and both proceed to `self.db.transaction()`.
4. Both call `record_execution` and `mark_job_complete`. Without a database-level uniqueness constraint, both succeed.

**Required fix:** Same as Finding 1, Layer 2. The idempotency check is useful as a fast-path optimization (avoid dispatching a job that is already done) but it cannot be the correctness boundary. The correctness boundary must be a unique constraint or conditional update at write time, inside the transaction.

### Finding 3 — Heartbeat Thread Joined with Bounded Timeout; Release Script May Execute Against Wrong Owner

**Issue name:** `t.join(timeout=self.RENEWAL_INTERVAL + 1)` does not guarantee the thread has exited before `_release_script` runs; and if the lock has been lost, the release script correctly no-ops (returns 0) — but this is not checked, so the `finally` block silently succeeds even when the lock was never released by this worker because it was already gone.

**Failure path:** This is a secondary issue. The release script is token-verified, so it will not delete Worker B's lock. The no-op is correct behavior. However, the caller has no signal that the release was a no-op versus a successful release. Combined with the silent heartbeat exit, this means the `finally` block completes without error in both the success case and the lock-loss case, reinforcing the false impression that everything is fine.

**Required fix:** Check the return value of `_release_script`. If it returns `0`, log a warning at ERROR level indicating that the lock was lost before release — this is a signal that a zombie-write may have occurred and should trigger an alert or compensating action.

### Finding 4 — Test Suite Does Not Cover the Failure Mode

**Issue name:** No test exercises lock expiry during execution or heartbeat-detects-loss behavior.

The test `test_execute_claim_contention` only covers the case where `SET NX` fails at acquisition time. No test covers:
- Heartbeat receiving `result == 0` and exiting
- Main thread continuing after heartbeat exit
- Two workers both completing the same job

**Required fix:** Add tests that mock `_renew_script` to return `0` mid-execution and assert that the main thread either raises or aborts the transaction. Add a concurrent execution test (using threads or mocked time) that asserts `record_execution` is called at most once across two workers racing on the same job.

## Zombie-Write Scenario — Explicit Statement

**The zombie-write can occur.** The exact trigger condition is: Worker A's main thread stalls (GC pause, slow external call, blocking I/O) for longer than `CLAIM_TTL` (60 seconds) at any point between lock acquisition and database commit.

**Step-by-step event sequence:**

| Time | Worker A | Worker B | Redis | Database |
|------|----------|----------|-------|----------|
| T+0 | Acquires lock, token=T_A, TTL=60s | — | `job:claim:X = T_A` | job=pending |
| T+0 | Passes `get_execution()` check (returns None) | — | — | — |
| T+0 | Enters `_dispatch()`, stalls (GC/IO) | — | — | — |
| T+60 | Still stalled | — | Key expires | — |
| T+60 | Still stalled | Acquires lock, token=T_B, TTL=60s | `job:claim:X = T_B` | — |
| T+60 | Still stalled | Passes `get_execution()` check | — | — |
| T+65 | Heartbeat fires: GET returns T_B ≠ T_A, returns 0, thread exits silently | — | — | — |
| T+70 | Stall ends, `_dispatch()` returns | Worker B `_dispatch()` returns | — | — |
| T+70 | Opens transaction, `record_execution`, `mark_job_complete` | Opens transaction, `record_execution`, `mark_job_complete` | — | — |
| T+70 | Commits | Commits | — | **Two execution records, job marked complete twice** |

**What existing boundary prevents both commits from landing:** Nothing, unless the database schema has a `UNIQUE` constraint on `job_id` in the executions table or `mark_job_complete` uses a conditional update. Neither is present in this code — `record_execution` and `mark_job_complete` are opaque calls with no documented or enforced uniqueness semantics at the application layer.

---

## Verdict: NOT APPROVED

The code contains a confirmed zombie-write path. The heartbeat correctly detects lock loss (the Lua script returning `0` is sound) but does not signal the main thread, and there is no fencing token or unique constraint at the database write boundary. Under a process stall exceeding 60 seconds — a realistic condition in any JVM-adjacent environment, containerized workload under memory pressure, or service making slow external calls — two workers will execute the same job and both commits will land. For jobs that send notifications, charge payments, or fire external events, this is a production correctness violation. The fix requires changes at two layers: `threading.Event` signaling from heartbeat to main thread (necessary, not sufficient) and a unique constraint or conditional update at the database write boundary (necessary and sufficient).
