**Finding 1: Silent Heartbeat Death — Main Thread Has No Visibility Into Lock Loss**

**Issue name:** Zombie write via silent heartbeat exit on lock expiry

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or I/O stall holds the process for >60s without the heartbeat firing in time — or the heartbeat fires but the Redis connection is transiently unavailable, so the renewal silently fails and the key expires.
3. The lock expires. Worker B calls `SET NX EX` on the same key, succeeds with token `T2`, passes the idempotency check (`get_execution` returns None — Worker A hasn't written yet), and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:{job_id}` returns `T2` ≠ `T1`. Script returns `0`. `_renew()` hits `if result == 0: return` and exits the thread. **No signal is sent to the main thread.**
5. Worker A's main thread continues executing. Both workers complete `_dispatch()`, both enter `db.transaction()`, both call `record_execution` and `mark_job_complete`. The job executes twice.

**The structural gap:** `_renew()` treats `result == 0` as a terminal condition for the heartbeat thread but has no mechanism to communicate lock loss to the main thread. `stop_event` is write-only from the main thread's perspective — the heartbeat thread never sets it. The main thread proceeds through the entire execution pipeline with a stale assumption of lock ownership.

**Required fix — two layers, both necessary:**

*Layer 1 — Signal propagation (necessary but not sufficient):* The heartbeat thread must set a shared `threading.Event` (e.g., `lock_lost_event`) when `result == 0`. The main thread must check this event at each major execution boundary — after `_dispatch()` returns, before entering the database transaction. If the event is set, abort and raise. This closes the communication gap but does not prevent the race window.

*Layer 2 — Fencing token / optimistic concurrency at the write boundary (required for correctness):* Thread signaling only narrows the window; it cannot eliminate it. Worker A can lose the lock, fail to detect it in time, and still reach the database write. The database transaction must include a conditional write that fails if another worker has already written a completion record. Concretely: `record_execution` and `mark_job_complete` must use optimistic concurrency — either a conditional INSERT that rejects duplicate `job_id` (unique constraint + INSERT ... WHERE NOT EXISTS), or a compare-and-swap on a monotonic job state column (e.g., `UPDATE jobs SET state='complete', version=N+1 WHERE job_id=X AND state='pending' AND version=N`). If the conditional write returns 0 rows affected, the transaction must be rolled back and the execution treated as a duplicate. This is the architectural boundary that actually prevents the duplicate write — the database, not Redis, is the final arbiter of whether a job completed.

The fencing token pattern applied here: the execution record insertion is itself the atomic fence. Whichever worker's transaction commits first wins; the second worker's conditional write fails, the transaction rolls back, no duplicate is recorded, and no duplicate side effect should have been triggered. This requires that `_dispatch()` either be idempotent or that external side effects (payment charge, notification send) be deferred until after the conditional write commits — otherwise the external action fires twice even if the DB write is deduplicated.

**Finding 2: Idempotency Check Is Outside the Lock-Protected Critical Section Relative to DB Writes**

**Issue name:** Check-then-act gap between idempotency read and execution write

**Failure path:**

1. Worker A and Worker B both attempt to claim the same job. Only one can hold the Redis lock at a time — so this gap is partially mitigated by the lock. However, as established in Finding 1, the lock can expire mid-execution.
2. More precisely: the idempotency check (`db.get_execution`) and the write (`db.record_execution`) are not atomic with respect to each other at the database layer. If two workers reach the write concurrently (via the zombie-write path), both will have seen `get_execution` return None and both will attempt to write.
3. Without a unique constraint or conditional write on `job_id` in the database, both writes succeed.

**Required fix:** Same as Finding 1, Layer 2. The database must enforce uniqueness. The idempotency check is a useful fast-path optimization but cannot be the correctness guarantee. The write must be conditional.

**Finding 3: Heartbeat Renewal Failure Is Silently Swallowed on Redis Errors**

**Issue name:** Redis exception in renewal thread causes unlogged silent lock abandonment

**Failure path:**

1. Worker A holds the lock. A transient Redis connection error occurs during heartbeat renewal. `self._renew_script(...)` raises a `redis.RedisError`.
2. The exception propagates out of `_renew()` uncaught. The daemon thread dies with an unhandled exception. No log is emitted at ERROR level. No signal reaches the main thread.
3. The lock TTL runs out. Worker B acquires the lock. Worker A continues. Duplicate execution.

**Required fix:** Wrap the renewal call in a try/except. On `redis.RedisError`, log at ERROR level and either retry with backoff (for transient errors) or set the `lock_lost_event` from Finding 1 to abort the main thread. Do not let the renewal thread die silently.

**Finding 4: Release Script Executes After Stop Event But Before Join Completes — Token Released While Heartbeat May Still Be Running**

**Issue name:** Release-before-join race in finally block

**Failure path, examined:**

```python
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    self._release_script(keys=[claim_key], args=[token])
```

The join happens before the release. This ordering is correct — the heartbeat is stopped (or timed out) before the lock is released. This specific ordering does not introduce a race.

However: if `t.join()` times out (heartbeat thread is hung on a Redis call for >16s), the release script executes while the heartbeat thread is still live. The heartbeat thread could then attempt a renewal after the release, fail (key deleted), return 0, and exit — no harm done in this specific case since the key is already gone. This is a minor issue, not a correctness violation, but the join timeout should be logged as a warning since it indicates a hung renewal thread.

**Required fix:** Log a warning if `t.is_alive()` after `t.join()` returns. No correctness fix required here, but observability is missing.

**Finding 5: No Test Covers the Zombie-Write Scenario**

**Issue name:** Test suite does not exercise lock-loss-during-execution path

The test `test_execute_claim_contention` only covers failure to acquire the initial lock. No test simulates the renewal script returning 0 mid-execution and verifies that the main thread aborts. No test simulates two workers reaching the database write concurrently.

**Required fix:** Add tests that mock `_renew_script` to return 0 after acquisition and verify the main thread does not complete the job write. Add a test that simulates concurrent `record_execution` calls and verifies the unique constraint / conditional write prevents duplication.

## Zombie-Write Scenario — Explicit Confirmation

**The zombie-write scenario can and does occur under this implementation.**

Trigger: Worker A's process stalls (GC pause, slow external I/O) beyond `CLAIM_TTL`. Worker B acquires the lock. Worker A's heartbeat detects token mismatch, returns 0, exits silently. Worker A's main thread has no visibility into this and continues to completion. Both workers write execution records and trigger external side effects.

**What must be added to close it:**

1. `threading.Event` set by the heartbeat on `result == 0` or on Redis exception, checked by the main thread before the database write. (Narrows the window, does not close it.)
2. Conditional write at the database layer — unique constraint on `job_id` in the execution table plus INSERT-if-not-exists or UPDATE-WHERE semantics — so that only one worker's transaction can commit a completion record. This is the required architectural boundary. (Closes the window for the DB write.)
3. External side effects (payment, notification, event emission) must be deferred to after the conditional write commits, or must themselves be idempotent with deduplication keys, so that the DB fence also fences the external action.

Without all three, the system has a correctness violation that will manifest under production load whenever process stalls exceed the lock TTL.
