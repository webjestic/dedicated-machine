### Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread (Zombie-Write Vulnerability)

**Issue name:** Undetected lock loss — zombie execution after TTL expiry

**Trigger condition:** The main thread stalls for longer than `CLAIM_TTL` (60 seconds) — via GC pause, slow `_dispatch` call, blocking I/O, or any other stop-the-world event — before the heartbeat thread has had a chance to renew.

**Step-by-step event sequence:**

```
T=0s    Worker A: acquires job:claim:X with token-A, TTL=60s
        Worker A: starts heartbeat thread (fires at T=15, 30, 45...)
        Worker A: enters _dispatch() — begins slow external call

T=0s–75s  Worker A main thread: stalled inside _dispatch()
           [GC pause, slow HTTP call, blocking I/O — does not matter which]

T=60s   Redis: TTL expires — key job:claim:X is deleted automatically

T=60s+ε Worker B: SET NX EX succeeds — acquires job:claim:X with token-B
        Worker B: starts its own heartbeat thread
        Worker B: checks db.get_execution(X) → None (not yet complete)
        Worker B: calls _dispatch() → begins executing the same job

T=75s   Worker A heartbeat fires:
          RENEW_CLAIM_SCRIPT runs GET job:claim:X → returns token-B (not token-A)
          Script returns 0
          _renew() hits `if result == 0: return`  ← exits silently
          Thread terminates. No exception. No Event set. No signal to main thread.

T=75s+  Worker A main thread: _dispatch() returns (stall has ended)
        Worker A: checks nothing — stop_event is never set by the heartbeat
        Worker A: proceeds to db.get_execution(X) → still None
          (Worker B has not yet committed)
        Worker A: calls db.record_execution() + db.mark_job_complete()
          inside db.transaction() → COMMITS

T=80s   Worker B: _dispatch() completes
        Worker B: db.get_execution(X) → now sees Worker A's record
          ... or does not, depending on transaction isolation and timing
        Worker B: calls db.record_execution() + db.mark_job_complete() → COMMITS
```

Both workers complete `_dispatch()` and both can reach the commit path. The job executes twice.

**Why the existing code does not close this:**

The `stop_event` is a one-way signal: the main thread sets it to *stop* the heartbeat. There is no reverse path. The heartbeat thread discovering `result == 0` calls `return` — it does not call `stop_event.set()`, does not raise into the main thread, does not set any shared flag. The main thread has zero visibility into the heartbeat's exit reason.

The idempotency check (`db.get_execution`) is a read before the commit. Under concurrent execution it is a TOCTOU race: both workers can read `None` before either has committed, then both proceed to write.

**Required fix — two layers, both necessary:**

**Layer 1 — In-process signal path (necessary but not sufficient):**

The heartbeat must set a shared event when it detects lock loss, and the main thread must check that event at the boundary before writing:

```python
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()   # ← signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

And in the main thread, before the database write:

```python
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting write")
```

This closes the window where the main thread *knows* it has lost the lock and proceeds anyway. It does not close the window where the main thread stalls so long that the heartbeat has not yet fired after lock expiry — the main thread can still reach the write before the heartbeat detects the loss.

**Layer 2 — Fencing token at the database write boundary (necessary and sufficient):**

`threading.Event` signaling is a best-effort, in-process detection mechanism. It cannot protect against the race where both workers reach the write concurrently before either detects the other. The only boundary that is guaranteed to serialize both writes is the database itself.

The fix is a fencing token (optimistic concurrency / conditional write) at the point of `record_execution` and `mark_job_complete`:

Option A — Database unique constraint: add a `UNIQUE` constraint on `job_id` in the executions table (or the job completion table). The second writer gets a constraint violation, not a silent double-write. This is the minimal correct fix.

Option B — Conditional update with version counter: `mark_job_complete` issues `UPDATE jobs SET status='complete', version=version+1 WHERE job_id=X AND status='pending'`. Check rows-affected. If 0, another worker already committed — abort and roll back.

Option C — Explicit fencing token: pass the Redis token into the DB write and store it; use `WHERE claim_token = :token` in the update. If the token no longer matches what was stored at claim time (because the lock expired and was re-acquired), the update affects 0 rows.

**Why the fix cannot live only at the threading layer:**

`threading.Event` operates inside one process. It cannot observe what Worker B is doing. Even with perfect in-process signaling, two workers can both pass the `lock_lost_event.is_set()` check (both believe they hold the lock at that instant) and both proceed to write. The database write boundary is the only point where both execution paths converge under a single serializable authority.

**Why the fix cannot live only at the Redis layer:**

Redis TTL expiry is what enables Worker B to acquire the lock. Once Worker A's lock has expired and Worker B holds it, Redis correctly reflects the new state. The problem is that Worker A's main thread does not know this. Redis cannot retroactively prevent Worker A from writing to the database — it has no visibility into the DB transaction.

**Architectural layer:** The fencing token must live at the database write boundary. The `threading.Event` signal is a latency optimization (fail fast, avoid unnecessary work) but is not a correctness boundary.

### Finding 2: Heartbeat Join Timeout Does Not Prevent Premature Release

**Issue name:** Claim released before heartbeat thread confirms termination

In the `finally` block:

```python
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)   # 16 seconds
self._release_script(keys=[claim_key], args=[token])
```

If the heartbeat thread is blocked (e.g., Redis call hanging), `t.join()` returns after 16 seconds regardless of whether the thread has actually stopped. The release script then runs while the heartbeat thread may still be alive and holding a reference to `token`. This is a minor issue in practice (the release script is token-verified, so a subsequent renewal attempt would fail), but the join timeout should be documented as a best-effort bound, not a guarantee of thread termination.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()` and log a warning if the thread did not terminate cleanly. This does not affect correctness of the release (the Lua script is already token-verified) but makes the operational behavior explicit.

### Finding 3: Test Suite Does Not Cover the Failure Mode

**Issue name:** No test exercises lock-loss-during-execution path

All tests mock `register_script` to return a script mock that always returns `1`. There is no test that simulates `result == 0` from the renewal script mid-execution. The zombie-write scenario is therefore invisible to the test suite.

**Required fix:** Add a test that configures the renewal script mock to return `0` after the first call, verifies that `lock_lost_event` is set, and verifies that the main thread does not proceed to `record_execution`. This test will fail against the current code (confirming the bug) and pass only after the fix is applied.

## Verdict: NOT APPROVED

The zombie-write scenario can occur. The exact trigger is a main-thread stall exceeding `CLAIM_TTL` (GC pause, slow external call, blocking I/O). The event sequence is: lock expires → Worker B acquires lock → Worker A heartbeat detects token mismatch and exits silently → Worker A main thread continues with no awareness of lock loss → both workers execute `_dispatch()` and both can commit to the database. The existing idempotency check is a TOCTOU read that does not serialize concurrent writers.

Two fixes are required before this is production-safe:

1. **`threading.Event` reverse signal** — heartbeat sets `lock_lost_event` on `result == 0`; main thread checks it before the DB write. Closes the in-process detection gap. Tier 0.5 fix: necessary, not sufficient.

2. **Fencing token at the database write boundary** — unique constraint on `job_id` in the executions table, or a conditional `UPDATE ... WHERE status='pending'` with rows-affected check. This is the correctness boundary. No stale writer can commit regardless of what it believes about its lock state. Tier 1.0 fix: required for correctness.

Neither fix alone is sufficient. Both must be present.
