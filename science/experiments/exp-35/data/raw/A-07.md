## Findings

---

### Finding 1: Silent Heartbeat Exit Does Not Signal the Main Thread — Zombie-Write Scenario Is Live

**Issue name:** Undetected lock expiry / zombie-write

**Trigger condition:** The main thread stalls (GC pause, slow external HTTP call, blocking I/O, OS scheduling delay) for longer than `CLAIM_TTL` (60 seconds) before or during `_dispatch()` or the database transaction.

---

**Step-by-step event sequence:**

| Time | Worker A | Worker B | Redis |
|------|----------|----------|-------|
| T+0 | Acquires `job:claim:X`, token=`A1`. Heartbeat thread starts. | — | Key set, TTL=60s |
| T+0..15 | Main thread enters `_dispatch()`. Heartbeat fires at T+15, renews TTL. | — | TTL reset to 60s |
| T+15 | Main thread stalls (GC pause / slow external call). | — | TTL counting down |
| T+75 | Main thread still stalled. | — | TTL expires. Key deleted. |
| T+75 | Heartbeat fires at T+30 (relative to last renewal). `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:X` → nil → token mismatch → **returns 0**. `_renew()` hits `if result == 0: return`. Thread exits silently. **No signal to main thread.** `stop_event` is never set by the heartbeat thread. | — | Key gone |
| T+75 | Main thread is still inside `_dispatch()` or entering the DB transaction. It has no reference to the heartbeat thread's return value. `stop_event` is only checked by the heartbeat thread, not by the main thread. | Worker B calls `redis.set(claim_key, token_B, nx=True, ex=60)` → **succeeds**. Worker B acquires the lock. | Key set, TTL=60s, value=`B1` |
| T+76..N | Worker A resumes from stall. Completes `_dispatch()`. Calls `db.get_execution()` → returns `None` (B hasn't committed yet, or A's read precedes B's write). Enters `db.transaction()`. Calls `db.record_execution()` and `db.mark_job_complete()`. **Commits.** | Worker B independently checks `db.get_execution()` → `None`. Dispatches. Enters `db.transaction()`. Commits. | — |
| Final | Job has executed twice. Two execution records exist. Side effects (charge, notification, event) have fired twice. | | |

**What the `finally` block does — and does not — prevent:**

When Worker A eventually exits the `with self._claim_job(...)` block, the `finally` clause runs `_release_script(keys=[claim_key], args=[token_A])`. The Lua script checks `GET job:claim:X == A1`. At this point the key holds `B1` (Worker B's token), so the script returns 0 and does nothing. The release is correctly a no-op. **But this happens after both commits have already landed.** The token-verified release prevents Worker A from evicting Worker B's lock, but it does not undo the duplicate execution that already occurred.

**What the idempotency check does — and does not — prevent:**

`db.get_execution(req.job_id)` before dispatch is a read-before-write guard. It is not inside the database transaction that records completion. There is no unique constraint or conditional write at the commit boundary. If both workers read `None` before either commits (a classic TOCTOU race), both proceed. Even with a unique constraint on `job_id` in the execution table, only one commit would fail with an integrity error — but `_dispatch()` has already run for both workers by that point, meaning the external side effect has already fired twice.

---

**Root cause — architectural layer analysis:**

The heartbeat thread has information the main thread needs: the lock is gone. There is no signal path from the heartbeat thread to the main thread. `stop_event` is write-only from the main thread's perspective; the heartbeat thread only reads it. The thread returns silently. The main thread has no `.join()` with a result check, no shared flag, no exception propagation mechanism.

This is a **two-layer failure**:

**Layer 1 — In-process signaling (necessary but not sufficient):**
The heartbeat thread must write to a shared flag when it detects `result == 0`, and the main thread must check that flag at each stage boundary (after `_dispatch()`, before entering the DB transaction). A `threading.Event` named `lock_lost` serves this purpose:

```python
lock_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost.set()   # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread checks after dispatch and before committing:

```python
result = self._dispatch(req.job_type, req.payload)

if lock_lost.is_set():
    raise ClaimError("Lock expired during dispatch — aborting commit")

with self.db.transaction():
    ...
```

This closes the window in the common case. **However**, it does not close it completely. There is a residual race: the lock can expire and Worker B can acquire it in the interval between `lock_lost.is_set()` returning `False` and `db.transaction()` committing. The check and the commit are not atomic. A sufficiently unlucky pause between those two lines reproduces the same scenario. `threading.Event` signaling is an in-process detection mechanism; it cannot make the database write conditional on lock ownership at commit time.

**Layer 2 — Fencing token at the database write boundary (necessary and sufficient):**

The only guarantee that closes the zombie-write scenario completely is a **conditional write at the commit boundary** that fails atomically if the lock is no longer held by this worker. This requires a fencing token — a value that is verifiable at write time against an authoritative source.

The implementation requires:

1. **A monotonic claim counter in Redis.** When a worker acquires the lock, it also reads (or atomically increments) a per-job sequence number stored in Redis. This sequence number is the fencing token. Worker A gets token `N`; if the lock expires and Worker B acquires it, Worker B gets token `N+1`.

2. **The fencing token is passed into the database transaction.** The `record_execution` and `mark_job_complete` calls receive the token value.

3. **The database write is conditional on the token.** This can be implemented as:
   - A `WHERE claim_version = :expected_token` clause on the update to the job row, with the application checking `rowcount == 1` before committing, **or**
   - A unique constraint on `(job_id, claim_version)` that causes the second commit to fail with an integrity error.

   Either way, the database rejects the stale write regardless of what the main thread believes about its lock state.

This fix must live at the **database write boundary** because that is the only point where the decision to commit and the authority to commit can be made atomically. An in-process check (Layer 1) can be bypassed by a pause that occurs after the check but before the commit. The database is the single serialization point for the state that must not be duplicated; the guard must live there.

The two fixes are complementary, not alternatives:
- `threading.Event` signaling: fast, in-process detection; aborts early in the common case; reduces the window but does not eliminate it.
- Fencing token at DB write boundary: eliminates the residual race unconditionally; the correct architectural fix for a correctness guarantee.

### Finding 2: `db.get_execution()` Idempotency Check Is Outside the Transaction

**Issue name:** TOCTOU on idempotency guard

**Failure path:** The `get_execution` read and the `record_execution` + `mark_job_complete` writes are not in the same transaction and are not protected by a unique constraint. Two workers can both read `None`, both dispatch, and both commit. Even a single worker that retries after a transient failure can double-execute if the first attempt's side effect succeeded but the DB write failed.

**Required fix:** Move `db.get_execution()` inside `db.transaction()`, or — better — enforce a unique constraint on `job_id` in the execution table and treat the integrity error on the second insert as the idempotency signal. The constraint must be at the database layer because application-level read-before-write cannot be made atomic without it.

### Finding 3: Heartbeat Thread Joined with a Timeout That Can Silently Expire

**Issue name:** Unjoinable heartbeat thread masks execution state

**Failure path:** `t.join(timeout=self.RENEWAL_INTERVAL + 1)` (16 seconds) will return whether or not the thread has actually exited. If the heartbeat thread is blocked on a slow Redis call, `join` returns after 16 seconds and the `finally` block proceeds to call `_release_script` without knowing whether the heartbeat thread is still running. The release script executes while the heartbeat thread may still be alive, creating a brief window where the thread could attempt a renewal after the release.

**Required fix:** After `t.join(timeout=...)`, check `t.is_alive()` and log a warning (or raise) if the thread did not exit cleanly. The heartbeat thread should also check `stop_event` before each renewal attempt to ensure it does not renew a lock that the main thread has already released.

---

### Finding 4: Test Suite Does Not Cover the Lock-Expiry Path

**Issue name:** Missing test coverage for the critical failure mode

**Failure path:** No test simulates `_renew_script` returning 0 and verifies that the main thread aborts the job. The mock fixture sets `script_mock = MagicMock(return_value=1)` uniformly. The zombie-write scenario is entirely untested.

**Required fix:** Add a test that configures the renew script to return 0 after the first call, injects a delay in `_dispatch`, and asserts that `db.record_execution` is never called. This test will fail against the current implementation (correctly), and pass only after the `lock_lost` signaling fix is in place.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed live. The heartbeat thread exits silently on lock loss (`result == 0`) with no signal to the main thread. The main thread continues executing and commits to the database. A second worker can acquire the same lock and also commit. There is no fencing token at the database write boundary to prevent both commits from landing.

Two fixes are required before this can go to production:

1. **`threading.Event` lock-loss signaling** — heartbeat sets `lock_lost` on `result == 0`; main thread checks before committing. Closes the common case.
2. **Fencing token at the database write boundary** — a monotonic counter passed through to a conditional DB write. Closes the residual race unconditionally. This is the required architectural fix; the `threading.Event` alone is insufficient.
