## Findings

---

**Finding 1: Silent Heartbeat Death — Main Thread Continues Executing After Lock Loss**

**Issue name:** Zombie write via unobserved heartbeat thread exit

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts, renewing every 15s.
2. Worker A enters `_dispatch()` — an external HTTP call, a slow DB query, or a GC pause stalls the main thread for >60s without a renewal firing in time. The lock expires in Redis.
3. Worker B calls `redis.set(claim_key, token, nx=True, ex=60)` — it succeeds. Worker B acquires the lock with token `T2` and begins executing the same job.
4. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2`, which does not match `T1`. The script returns `0`.
5. `_renew()` sees `result == 0` and executes `return` — the thread exits silently. No exception is raised. No `threading.Event` is set. No flag is written. The thread simply terminates.
6. Worker A's main thread has no join, no poll, no shared state check on the heartbeat thread. It continues executing through `_dispatch()`, into `self.db.transaction()`, and calls `record_execution` and `mark_job_complete`.
7. Worker B does the same.
8. Both workers complete the job. The job executes twice. For a payment charge, alert dispatch, or external API call inside `_dispatch()`, this is an unrecoverable correctness violation.

The critical structural gap is in `_renew()`:

```python
if result == 0:
    return   # ← thread dies here; main thread never knows
```

And in `execute()`, nothing between `_dispatch()` and `db.transaction()` checks whether the lock is still held. The `with self._claim_job(req.job_id):` block provides no runtime invariant — it only guarantees acquisition at entry and release at exit. It cannot guarantee continuous ownership.

**Required fix — two layers, both required:**

*Layer 1 — Thread signaling (necessary, not sufficient):* The heartbeat must signal the main thread on lock loss. Add a `lock_lost_event = threading.Event()` passed into `_renew()`. On `result == 0`, set it before returning. The main thread must check this event at the boundary between `_dispatch()` completion and the DB write — or use a wrapper that raises into the main thread.

```python
if result == 0:
    lock_lost_event.set()
    return
```

```python
# In execute(), before the db.transaction() block:
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting write")
```

This closes the gap for the common case but is not sufficient alone. There is a window between the check and the DB write where the lock could be lost. This is a TOCTOU race.

*Layer 2 — Fencing token / optimistic concurrency at the DB write (required for correctness):* The `record_execution` and `mark_job_complete` calls must be conditioned on the lock still being held at write time. The architectural fix is a **fencing token**: a monotonic counter stored in Redis (via `INCR`) at lock acquisition time, passed through to the DB write, and checked there as a precondition.

```python
# At acquisition:
fence = self.redis.incr(f"job:fence:{job_id}")
# fence is passed into execute() and down to db.record_execution()

# DB write becomes:
self.db.record_execution_if_fence_matches(execution_id, job_id, job_type, result, fence)
# → UPDATE ... WHERE job_id = ? AND fence_token = ? AND status != 'complete'
# If 0 rows affected, abort — another worker has written with a higher fence
```

This makes the DB write itself the serialization point. Even if two workers reach the write concurrently, only the one whose fence token matches the current DB state succeeds. The second write is rejected by the database, not by a Redis check that may have already expired.

The `threading.Event` signal alone is Tier 0.5: it closes the obvious race but leaves the TOCTOU window between the event check and the DB commit. The fencing token closes that window. Both are required for a correct implementation.

**Finding 2: Lock Release After Lock Loss Deletes Another Worker's Lock**

**Issue name:** Spurious lock deletion on release after expiry

**Failure path:**

1. Worker A's lock expires. Worker B acquires the lock with token `T2`.
2. Worker A's `finally` block runs: `stop_event.set()`, `t.join()`, then `self._release_script(keys=[claim_key], args=[token])`.
3. `RELEASE_CLAIM_SCRIPT` correctly checks `GET key == ARGV[1]` — `T2 != T1` — so it returns `0` without deleting. This part is correct.

**Assessment:** This specific sub-scenario is handled correctly by the Lua release script. The token check prevents Worker A from deleting Worker B's lock. No fix required here — noting it explicitly because reviewers sometimes assume the release is the zombie-write vector. It is not. The zombie write occurs before the release, during execution.

**Finding 3: Idempotency Check Is Not Atomic With Execution — TOCTOU on DB State**

**Issue name:** Check-then-act race on `get_execution` / `record_execution`

**Failure path:**

1. Two workers both acquire the lock sequentially (Worker B after Worker A's lock expires, as in Finding 1). But consider a separate scenario: the lock is released normally by Worker A just before it crashes after `mark_job_complete` but before returning. Worker C retries the job. `get_execution` returns the existing record. Worker C correctly returns "already completed." This path is fine.
2. The dangerous path: Worker A and Worker B both hold the lock simultaneously (via the zombie-write scenario in Finding 1). Both call `get_execution` — both see `None` (job not yet complete). Both proceed to `_dispatch()`. Both proceed to `db.transaction()`. The DB transaction provides atomicity for the write pair, but without a unique constraint or conditional insert on `(job_id)` in the `mark_job_complete` table, both transactions can commit.

**Required fix:** The `mark_job_complete` write must be a conditional upsert with a unique constraint on `job_id`, structured as `INSERT ... WHERE NOT EXISTS` or equivalent, with the DB enforcing uniqueness. If the second insert is rejected, the transaction must roll back and the worker must treat this as a duplicate execution signal. This is a defense-in-depth layer — it does not replace the fencing token fix, but it catches races the fencing token does not cover (e.g., two workers with different fence values if the fence implementation has a bug).

---

**Finding 4: Heartbeat Renewal Interval Does Not Guarantee TTL Maintenance**

**Issue name:** Renewal timing allows expiry under thread scheduling jitter

**Failure path:**

CLAIM_TTL=60s, RENEWAL_INTERVAL=15s. Under normal operation the lock is renewed at 15s, 30s, 45s — well within TTL. However: `stop_event.wait(timeout=15)` is subject to OS thread scheduling. Under high load, a Python GIL-contended process, or a cgroup CPU throttle event, the renewal thread may not be scheduled for significantly longer than 15s. If the main thread is also stalled (the GC pause scenario), neither thread runs. The lock expires.

This is the trigger condition for Finding 1, not an independent bug — but the ratio of RENEWAL_INTERVAL to CLAIM_TTL (15s/60s = 25%) is the margin. A tighter ratio (e.g., RENEWAL_INTERVAL = CLAIM_TTL / 3 = 20s) provides less margin than it appears because the first renewal fires at T+15, meaning the lock must survive until T+15 without renewal. Under a 60s TTL this is adequate in practice, but the design has no documented safety margin analysis.

**Required fix:** Document the scheduling assumption. Consider RENEWAL_INTERVAL = CLAIM_TTL / 4 as a floor, not CLAIM_TTL / 4 as a ceiling. More importantly, this finding is subsumed by Finding 1 — if the fencing token fix is implemented, a missed renewal leading to lock expiry results in a rejected DB write rather than a duplicate execution.

## Zombie-Write Scenario — Explicit Statement

**The zombie-write scenario can and does occur under this implementation.**

**Trigger:** Worker A's main thread stalls beyond CLAIM_TTL (GC pause, slow external call, CPU throttle). The lock expires. Worker B acquires the lock. Worker A's heartbeat detects token mismatch, returns 0, and exits silently. Worker A's main thread resumes with no knowledge of lock loss and completes execution including the DB write. Worker B also completes execution. Both writes commit. The job executes twice.

**What prevents it (required, not present):**
1. Heartbeat must signal main thread on `result == 0` — closes the common case.
2. DB write must be conditioned on a fencing token (monotonic counter incremented at lock acquisition, stored as a precondition on the DB write) — closes the TOCTOU window between the signal check and the commit.
3. DB unique constraint on job completion — defense-in-depth layer.

None of these three controls are present in the submitted code.

---

## Verdict: NOT APPROVED

The zombie-write scenario is confirmed present. The heartbeat correctly detects lock loss but discards that information — the main thread executes and writes to the database with no awareness that the lock has expired and another worker is running the same job. The fix requires both thread-level signaling and a fencing token at the DB write boundary. Approving this code means approving duplicate job execution in any deployment where GC pauses, slow external calls, or CPU throttle events can stall a worker beyond CLAIM_TTL. That condition is not exotic — it is normal production behavior under load.
