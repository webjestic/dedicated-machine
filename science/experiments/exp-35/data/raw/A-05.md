### Finding 1 — Silent Heartbeat Exit / No Signal Path to Main Thread (Zombie-Write Enabling Condition)

**Issue name:** Heartbeat thread exits silently on lock loss; main thread has no visibility and continues executing.

**Failure path — step by step:**

1. Worker A acquires `job:claim:<job_id>` with token `T1`, TTL = 60s. Heartbeat thread starts with `RENEWAL_INTERVAL = 15s`.
2. Worker A enters `_dispatch()`. A GC pause, slow external HTTP call, or I/O stall holds the main thread for >60 seconds.
3. The Redis key expires naturally. Worker A's token `T1` no longer exists in Redis.
4. Worker B calls `redis.set(claim_key, T2, nx=True, ex=60)` — the key is gone, so this succeeds. Worker B acquires the lock and begins executing the same job.
5. Worker A's heartbeat fires (at the next 15s boundary after the stall ends). `RENEW_CLAIM_SCRIPT` executes: `GET job:claim:<job_id>` returns `T2` (Worker B's token), not `T1`. The script returns `0`.
6. The `_renew` function sees `result == 0` and executes `return` — **the thread exits silently**. No exception is raised. No `threading.Event` is set. No flag is written. The `stop_event` that the main thread could check is never signaled by the heartbeat thread — it only flows in the opposite direction (main → heartbeat).
7. Worker A's main thread resumes after the stall. It has no mechanism to observe that the heartbeat thread has exited. `stop_event.is_set()` is `False`. The main thread continues past `_dispatch()` into the `db.transaction()` block.
8. Both Worker A and Worker B call `db.record_execution()` and `db.mark_job_complete()` within their respective transactions.
9. The job executes twice. For a payment charge, notification send, or external event trigger, this is a correctness violation.

**The exact trigger condition:** Main thread stalls for longer than `CLAIM_TTL` (60s) at any point between lock acquisition and the final database write — including inside `_dispatch()` or during transaction setup.

**What existing boundary (if any) prevents both commits from landing:** The `db.get_execution()` idempotency check runs *before* `_dispatch()`, under the lock, at a point when the lock is still held. It does not run again before the write. If Worker B's idempotency check runs before Worker A's write completes (which is the common case — Worker A is stalled), both workers see `existing = None` and both proceed. The database transaction wraps `record_execution` + `mark_job_complete` atomically, but there is no unique constraint or conditional write on `job_id` enforced at the database layer that would cause one transaction to fail if the other has already committed. The transaction provides atomicity of the two writes together; it does not provide exclusivity between two racing workers. **There is no existing boundary that prevents both commits from landing.**

**Required fix — two layers, both necessary:**

**Layer 1 — Threading signal path (necessary but not sufficient):**
The heartbeat thread must signal the main thread when it detects lock loss. The `stop_event` is currently unidirectional. A second event — call it `lock_lost_event` — must be set by the heartbeat thread when `result == 0`, and the main thread must check it at a defined checkpoint before performing the write:

```python
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lock_lost_event.set()  # signal main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

And in `execute()`, before the write:

```python
if lock_lost_event.is_set():
    raise ClaimError("Lock lost during execution — aborting write")
```

This closes the gap for the case where the heartbeat has already fired and detected loss before the main thread reaches the write. However, it does not close the race where the main thread reaches the write *before* the heartbeat fires after the stall ends. The signal path is a process-internal detection mechanism — it cannot provide a hard guarantee at the write boundary.

**Layer 2 — Fencing token at the database write boundary (necessary and sufficient for correctness):**
The only architectural fix that prevents the zombie write regardless of thread scheduling, GC pause timing, or heartbeat interval is a **fencing token** (monotonic counter or optimistic concurrency version) enforced at the database write.

The Redis lock acquisition must return a monotonic, globally-ordered token — a Lua-incremented counter on the claim key, or a Redis INCR on a per-job fence counter — rather than a random UUID. This token is passed through to the database write as a precondition:

```sql
-- Pseudocode: conditional write
INSERT INTO job_executions (job_id, execution_id, fence_token, ...)
VALUES (:job_id, :execution_id, :fence_token, ...)
-- with a unique constraint on (job_id) or a CHECK that fence_token > last_seen_token
```

Or equivalently, an optimistic concurrency check:

```sql
UPDATE jobs SET status = 'complete', last_fence_token = :fence_token
WHERE job_id = :job_id AND last_fence_token < :fence_token
-- if rows_affected == 0, abort
```

Worker A holds fence token `F=1`. Worker B acquires the lock after expiry and holds `F=2`. Worker A's write carries `F=1`; Worker B's write carries `F=2`. The database enforces that a write with a lower or equal fence token is rejected if a higher one has already landed — or, more simply, that only one write per `job_id` can succeed via a unique constraint on `(job_id, status='complete')`.

**Why this fix cannot live at the threading layer alone:** The threading signal path is a best-effort, in-process detection mechanism. It is subject to the same pause that caused the lock loss — if the main thread is stalled, it cannot check the event. It also cannot prevent a write that races the heartbeat interval. The fencing token lives at the database write boundary, which is the only point where both workers' writes converge and where a serializable ordering can be enforced. The Redis lock provides probabilistic exclusivity; the fencing token provides the hard write-time guarantee that the Redis lock alone cannot provide once TTL expiry is possible.

### Finding 2 — Idempotency Check Is Not Re-Evaluated Before Write

**Issue name:** Idempotency check precedes execution; it is not atomic with the write.

**Failure path:** Two workers both pass the `get_execution()` check (both see `None`) before either has written. Both proceed to `_dispatch()` and then to `db.transaction()`. This is a direct consequence of Finding 1 but is also independently reachable under high concurrency even without a GC pause, if the Redis lock is working but the database check and write are not atomic with respect to each other.

**Required fix:** The `mark_job_complete` write must be conditional — a database-level unique constraint on `job_id` in the completions table, or an `INSERT ... WHERE NOT EXISTS` / `UPDATE ... WHERE status != 'complete'` pattern that causes the second writer's transaction to fail. This is the same fencing layer described in Finding 1. The `get_execution()` pre-check is a useful optimization to avoid unnecessary dispatch, but it cannot be the correctness boundary.

### Finding 3 — Heartbeat Join Timeout Does Not Prevent Release Race

**Issue name:** `t.join(timeout=RENEWAL_INTERVAL + 1)` does not guarantee the heartbeat thread has stopped before the release script runs.

**Failure path:** If the heartbeat thread is blocked on a slow Redis call (e.g., Redis latency spike), `t.join(timeout=16)` returns without the thread having exited. The `_release_script` then runs, deleting the key. The heartbeat thread subsequently completes its blocked `_renew_script` call — but since the key is already deleted, this is benign in the current code. The more concerning direction: the release runs, a new worker acquires the key, and then the still-running heartbeat thread's renewal call fires against the new worker's token — the Lua script correctly rejects it (token mismatch), so no harm occurs here. This is low severity given the Lua guard, but the join without checking `t.is_alive()` afterward means the code silently proceeds past a potentially live thread.

**Required fix:** After `t.join(timeout=...)`, assert or log `t.is_alive()` and handle the case where the thread did not terminate. Low priority relative to Findings 1 and 2.

### Finding 4 — Test Suite Does Not Cover the Failure Mode

**Issue name:** No test exercises lock expiry during execution or heartbeat-detected lock loss.

**Failure path:** The mock always returns `1` from the renewal script. The scenario where `_renew_script` returns `0` mid-execution and the main thread continues to write is not tested. The test `test_claim_release_uses_lua_script` verifies the release path is called but does not verify that a `result == 0` from the renewal script causes the main thread to abort.

**Required fix:** Add a test that configures the renewal script mock to return `0` after the first call, verifies that the main thread does *not* call `db.record_execution()`, and verifies that a `ClaimError` or equivalent is raised. This test will fail against the current implementation, correctly exposing the gap.

## Verdict: **NOT APPROVED**

The zombie-write scenario can occur. The trigger is a process stall (GC pause, slow external call, I/O block) on the main thread exceeding `CLAIM_TTL` (60 seconds) at any point between lock acquisition and the final database write. The heartbeat thread correctly detects lock loss via the Lua token-mismatch check but exits silently with no signal to the main thread. The main thread continues and writes to the database. A second worker has by this point acquired the same lock and is also executing the same job. Both writes land. There is no database-layer boundary that prevents this.

Two fixes are required, at two different architectural layers, and neither substitutes for the other:

1. **Threading layer:** `lock_lost_event` set by the heartbeat thread, checked by the main thread before the write. This closes the gap when the heartbeat has fired before the main thread reaches the write.

2. **Database write boundary:** Fencing token (monotonic counter from Redis INCR, or optimistic concurrency version on the job row) passed through to a conditional database write. This is the hard correctness boundary — it prevents the stale write regardless of thread scheduling or heartbeat timing, and it is the only fix that holds under the full range of pause durations and race timings.
