**Findings**

---

**Finding 1: Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Guaranteed**

**Name:** Zombie write via undetected lock loss

**Failure path:**

1. Worker A acquires `job:claim:{job_id}` with token `T1`, TTL=60s. Heartbeat thread starts.
2. Worker A enters `_dispatch()`. A GC pause, slow external call, or network stall holds the process for >60s.
3. The Redis TTL expires. The key is gone. Worker A holds no lock.
4. Worker B calls `redis.set(claim_key, token, nx=True, ex=60)` — succeeds. Worker B acquires the lock with token `T2` and begins executing the same job.
5. Worker A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2`, which does not equal `T1`. The script returns `0`.
6. `_renew()` checks `if result == 0: return`. The thread exits. **No signal is sent to the main thread. No exception is raised. No flag is set.**
7. Worker A's main thread is unaware. It completes `_dispatch()`, enters `self.db.transaction()`, calls `record_execution()` and `mark_job_complete()`.
8. Worker B does the same.
9. Both workers complete the job. For a job that sends a payment, fires an alert, or triggers an external action, the operation executes twice.

**The `stop_event` does not help here.** `stop_event` is set by the `finally` block in the main thread — it is a signal *from* main *to* heartbeat, not the reverse. The heartbeat has no channel back to main.

**Required fix — two layers:**

*Layer 1 (necessary, not sufficient): Thread signaling.*
Add a `lock_lost_event = threading.Event()` passed into `_renew`. When `result == 0`, set `lock_lost_event` before returning. In the `finally` block (or via a wrapper around `yield`), check `lock_lost_event.is_set()` and raise `ClaimError("Lock lost during execution")` before releasing or recording anything. This prevents the *current* execution from completing, but does not prevent the duplicate — Worker B is already running.

*Layer 2 (architecturally required): Fencing token / optimistic concurrency at the effect boundary.*
The database write is the effect that must be guarded. The lock token (`uuid4`) must be used as a fencing token passed through to the DB layer. `record_execution` and `mark_job_complete` must execute conditionally: only if the job's recorded token matches the token this worker holds. This is implemented as a compare-and-write at the DB layer — e.g., an `UPDATE jobs SET status='complete', token=? WHERE job_id=? AND token=?` or equivalent optimistic check. If Worker B has already written with `T2`, Worker A's write with `T1` fails the condition and is rejected. This is the only boundary that prevents both workers from recording completion. The Redis lock alone cannot provide this guarantee once the TTL has expired.

Without Layer 2, even with correct thread signaling, a sufficiently unlucky timing window (A's main thread past the lock expiry, B already executing) produces a duplicate that no amount of heartbeat logic can prevent.

**Finding 2: Idempotency Check Is Inside the Lock, But the Lock Is Not Durable**

**Name:** TOCTOU on idempotency guard under lock loss

**Failure path:**

1. Worker A acquires lock, calls `db.get_execution(job_id)` — returns `None`. Proceeds.
2. Lock expires mid-execution (as above). Worker B acquires lock, calls `db.get_execution(job_id)` — still returns `None` because Worker A has not yet written. Worker B proceeds.
3. Both workers pass the idempotency check. Both execute.

**Root cause:** The idempotency check is only meaningful if the lock is held for its entire duration. Once the lock can expire silently, the check provides no protection for the window between expiry and Worker A's write.

**Required fix:** The idempotency check must be enforced at the DB write boundary, not only at the read boundary. The `record_execution` insert must be on a unique constraint (`job_id` unique or `(job_id, status)` unique), so that the second writer receives a constraint violation rather than a silent success. This converts a race into a detectable error at the only layer that is authoritative.

**Finding 3: Release Script Executes After Lock May Have Been Reassigned**

**Name:** Benign-but-masked release of another worker's lock

**Failure path:**

1. Worker A's lock expires. Worker B acquires the lock with token `T2`.
2. Worker A's `finally` block runs `_release_script(keys=[claim_key], args=[T1])`.
3. The Lua script checks `GET claim_key == T1` — it is `T2`, so the script returns `0` and does not delete. **This part is correct.**
4. However: if Worker A's stall is shorter — lock expires, Worker B acquires, Worker A's heartbeat fires and exits, Worker A's main thread finishes *quickly*, and the `finally` runs while Worker B is mid-execution — Worker A's release attempt is a no-op (token mismatch). Worker B continues correctly.

**Assessment:** The release script itself is correctly token-gated and does not delete another worker's lock. This specific sub-path is not a bug. However, it only appears safe because the Lua guard works. The absence of any logging or error handling on `result == 0` from the release script means a lock-loss event during release is invisible in observability. **Add logging when release returns 0** — it indicates the worker executed without holding the lock for some portion of its run, which is a correctness signal worth alerting on.

**Finding 4: Heartbeat Renewal Interval Allows a Full TTL Expiry Window**

**Name:** Renewal interval does not guarantee TTL safety under thread scheduling jitter

**Failure path:**

- `CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. Under normal conditions, renewal fires at 15s, 30s, 45s — safe.
- Under thread starvation (GIL contention, CPU saturation, OS scheduling delays), the renewal thread may not wake at exactly 15s. If the main thread is in a CPU-bound or I/O-blocked section and the renewal thread is starved, the effective renewal gap can exceed 60s on a loaded system.
- Additionally, `stop_event.wait(timeout=15)` returns `True` (stop requested) or `False` (timeout). The loop condition `while not stop_event.wait(...)` exits when `stop_event` is set. This is correct. But the first renewal fires at t=15s, meaning the lock has already been held for 15s unrenewal. If the initial TTL is 60s and renewal fires at 15, 30, 45 — the lock is safe. But if the process stalls at t=14s and renewal fires at t=15s into a stalled process, the stall can extend past t=60s before the next renewal at t=30s is missed.

**Required fix:** Set `RENEWAL_INTERVAL` to `CLAIM_TTL / 3` or less (here, ≤20s is fine; 15s is acceptable). More importantly, the TTL should be extended on each renewal to the full `CLAIM_TTL` from *now* — which the current `EXPIRE key TTL` does correctly. The ratio is acceptable as written, but should be documented as an invariant: `RENEWAL_INTERVAL < CLAIM_TTL / 2`.

**Finding 5: Test Suite Does Not Cover the Failure Mode**

**Name:** Heartbeat silent-exit path is untested

**Failure path:** No test simulates `_renew_script` returning `0` and verifies that the main thread is interrupted or raises. The mock returns `1` (success) unconditionally. The gap identified in Finding 1 is invisible to the test suite as written.

**Required fix:** Add a test that configures `_renew_script` to return `0` on the first call and asserts that `execute()` raises or returns a failure result rather than completing successfully. Until this test exists and fails on the current code (then passes after the fix), the fix cannot be verified.

---

**Zombie-Write Scenario — Explicit Statement**

The zombie-write scenario **can and will occur** under the following conditions:

- Worker A acquires the lock and begins execution.
- A GC pause, slow syscall, or external call stalls Worker A beyond `CLAIM_TTL` (60s).
- Worker B acquires the lock and begins executing the same job.
- Worker A's heartbeat detects token mismatch, exits silently.
- Worker A's main thread completes execution and writes to the database.
- Worker B's main thread completes execution and writes to the database.
- The job executes twice.

**What closes it:** A fencing token carried from the Redis lock into the database write, enforced as a conditional update or unique constraint at the DB layer, such that only one writer's commit succeeds. Thread signaling (lock_lost_event) is a defense-in-depth measure that reduces the window but does not close the scenario — a sufficiently long stall that resumes after Worker B has already started cannot be stopped by signaling alone.

---

**Verdict: NOT APPROVED**

The zombie-write scenario is confirmed present and reachable in production. The heartbeat thread exits on lock loss without signaling the main thread, and there is no fencing token or optimistic concurrency check at the database write boundary. Under a GC pause or slow external call exceeding 60 seconds, two workers will execute the same job and both writes will succeed. For jobs that send payments, fire alerts, or trigger external actions, this is a live duplicate-execution bug. The code may not be approved until Finding 1 (both layers) and Finding 2 are resolved and covered by tests.
