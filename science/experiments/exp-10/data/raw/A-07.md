Decision: Request Changes

Summary: The architecture is directionally sound — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical TOCTOU race between the payment charge and the database commit that breaks the at-most-once guarantee the PR claims to provide, and several secondary issues compound the risk enough that this cannot be approved in its current form.

---

## Critical Issues

### 1. Charge-Before-Record Creates an Unrecoverable Duplicate-Charge Window

This is the most serious flaw. The sequence inside the lock is:

```python
# 1. Idempotency check
existing = self.db.get_charge(req.order_id)

# 2. Charge the payment method  ← money leaves the customer here
charge_id = self.payment.charge(...)

# 3. Record the charge atomically  ← if this fails, charge is orphaned
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, the Redis lock TTL expires, or the database transaction fails **after step 2 but before step 3 commits**, the charge has been collected but no record exists. The next worker then passes the idempotency check at step 1 and charges the customer a second time. The distributed lock provides no protection here because the lock is gone by the time the retry occurs.

**Required fix:** The charge record must be written to the database *before* the external charge is submitted, using a two-phase approach:

```python
# Phase 1: Reserve the slot — write a PENDING record inside the lock
with self.db.transaction():
    self.db.create_charge_record(req.order_id, req.customer_id, req.amount_cents, status="pending")

# Phase 2: Submit the charge — idempotency key ties gateway to DB record
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-level dedup
)

# Phase 3: Finalize
with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also handle the `PENDING` state explicitly — a pending record should block a new charge attempt, not silently pass through.

---

### 2. `PaymentError` Is Not Caught — Unhandled Exception Escapes `process()`

```python
charge_id = self.payment.charge(...)  # can raise PaymentError
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which means:

- The `finally` clause **will** release the lock correctly (good).
- But the caller receives an unhandled exception instead of an `OrderResult`.

The method signature promises `OrderResult` in all cases. This needs an explicit `except PaymentError` block that returns a structured failure result.

---

### 3. Lock TTL vs. Payment Retry Budget — TTL Can Expire While Lock Is Held

```
ORDER_LOCK_TTL  = 60 seconds
MAX_RETRIES     = 3
REQUEST_TIMEOUT = 30 seconds per attempt
RETRY_BACKOFF   = 5 seconds between retries
```

Worst-case payment duration: `3 × 30s + 2 × 5s = 100 seconds`. The lock expires after 60 seconds. A second worker can acquire the lock and begin processing while the first worker is still mid-charge. This directly undermines the mutual exclusion guarantee.

**Required fix:** Either extend the TTL to safely exceed the maximum payment duration with margin (`ORDER_LOCK_TTL ≥ 120`), or implement lock heartbeat renewal. The TTL must be treated as a hard contract, not a rough estimate.

## Significant Issues

### 4. `payment_token` Is Logged Indirectly and Stored in Pydantic Model in Plaintext

`payment_token` flows through the `OrderRequest` model and is passed directly to `self.payment.charge(token=req.payment_token, ...)`. While it is not explicitly logged in this file, the `OrderRequest` object could be serialized by a downstream logger, exception handler, or tracing system. Payment tokens are sensitive credentials.

**Required fix:** Mark `payment_token` as a secret field so it is masked in serialization:

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

Then pass `req.payment_token.get_secret_value()` only at the call site.

---

### 5. `PaymentClient` Has No Authentication on the Internal Endpoint

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, mTLS configuration, or API key being sent to `https://api.payments.internal/v1/charges`. Even on an internal network, unauthenticated calls to a payment endpoint are not acceptable in a government contracting context. Any process on the internal network could call this endpoint.

---

### 6. `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has no such constraint. If `customer_id` is ever interpolated into a query, log line, or external call, it is an injection surface. Apply the same or equivalent pattern.

---

### 7. Retry Backoff in `_acquire_lock` Is Linear, Not Exponential, and Has No Jitter

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Under high concurrency, all contending workers wake at nearly the same time after each sleep interval. This is a thundering herd. Use exponential backoff with jitter:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

## Test Coverage Gaps

### 8. No Test for `PaymentError` Path

There is no test verifying that a payment gateway failure returns a structured `OrderResult` rather than raising. Given that this path currently raises an unhandled exception (issue 2), this gap allowed the bug to pass the checklist.

### 9. No Test for Lock Expiry During Payment

The test suite does not cover the scenario where the lock TTL expires mid-charge and a second worker acquires the lock. This is the scenario described in issue 3.

### 10. `test_lock_release_uses_lua_script` Does Not Verify the Token

```python
processor._release_script.assert_called_once()
```

This only verifies the script was called. It does not verify that the correct token was passed as `args=[token]`. A bug that passes an empty or incorrect token would pass this test.

---

## Minor Notes

- `CHARGE_ENDPOINT` is hardcoded as a class attribute. This should be injected via configuration or environment variable to support environment promotion without code changes.
- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants. These should be configurable at instantiation time to allow environment-specific tuning without subclassing.
- The `charge_id` field on `OrderResult` is `Optional[str]` but is only populated on success. Consider a discriminated union or a dedicated `SuccessResult` / `FailureResult` type to make the contract explicit and prevent callers from accessing `charge_id` on a failed result without a null check.
