Decision: Request Changes

Summary: The core locking and idempotency architecture has a critical TOCTOU race condition that directly undermines the duplicate-charge guarantee this PR is designed to provide. There are also meaningful reliability gaps in the payment client and test coverage that need to be addressed before this is safe to merge.

---

## Critical Issues

### 1. TOCTOU Race Between Payment Charge and DB Record (Critical)

This is the most serious flaw. The charge happens **outside** the database transaction:

```python
# Charge the payment method
charge_id = self.payment.charge(...)   # ← money leaves here

# Record the charge and mark the order complete atomically
with self.db.transaction():            # ← recorded here, separately
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, is killed, or the DB transaction fails after `payment.charge()` returns but before `db.record_charge()` commits, the customer has been charged and there is **no record of it**. The next worker will pass the idempotency check (`get_charge` returns `None`) and charge again.

The Redis lock does not protect against this — the lock is released in `finally`, which runs after the DB transaction, but the window between the charge and the commit is the real danger zone.

**Required fix:** The payment gateway charge must be idempotent by design. Pass a deterministic idempotency key (e.g., derived from `order_id`) to the payment gateway so that a retry of the same charge returns the same `charge_id` rather than creating a new charge. Many payment APIs (Stripe, etc.) support this natively.

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,  # deterministic — safe to retry
)
```

Without this, retries after a crash are not safe regardless of the locking strategy.

---

### 2. PaymentError Is Silently Swallowed (Critical)

In `process()`, `PaymentError` is never caught. If `self.payment.charge()` raises, the exception propagates out of the `with self._acquire_lock(...)` block. The `finally` clause correctly releases the lock, but the caller receives an unhandled exception rather than a structured `OrderResult`. This means the caller has no way to distinguish "lock contention" from "payment declined" from "gateway down" without catching raw exceptions.

```python
# This is never caught:
charge_id = self.payment.charge(...)  # raises PaymentError → unhandled
```

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult`:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 3. Timeout Race on Lock TTL (High)

`ORDER_LOCK_TTL` is 60 seconds. `PaymentClient.REQUEST_TIMEOUT` is 30 seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds between retries. Worst case payment time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock expires at 60 seconds. A second worker can acquire the lock while the first is still waiting on a payment retry, defeating the entire mutual exclusion guarantee. The idempotency check will pass for the second worker because no charge has been recorded yet.

**Required fix:** Either reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` so the total payment time is well under `ORDER_LOCK_TTL`, or extend the TTL with a lock-refresh (keepalive) mechanism. The values must be chosen deliberately and documented with this constraint in mind.

## Moderate Issues

### 4. `payment_token` Logged or Exposed Risk

`payment_token` flows through `OrderRequest` and is passed directly to `payment.charge()`. Confirm it is never interpolated into log messages anywhere in the call chain. The current code appears clean, but the field has no redaction marker. Consider a custom Pydantic type that masks `__repr__` and `__str__` to prevent accidental logging downstream:

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

### 5. `response.json()["charge_id"]` Has No Error Handling

In `payment_client.py`:

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id`, non-JSON content type), this raises an unhandled `KeyError` or `JSONDecodeError`. These will not be caught by the `HTTPError` handler and will propagate as unexpected exceptions.

```python
try:
    data = response.json()
    charge_id = data["charge_id"]
except (ValueError, KeyError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

### 6. Retry Logic Retries on Timeout Without Knowing Charge State

When a `Timeout` occurs, the charge may or may not have been applied by the gateway. Retrying without an idempotency key (see Issue #1) can result in a double charge. This is a second reason the idempotency key fix is mandatory, not optional.

---

## Minor Issues

### 7. Test Coverage Gaps

- No test for `PaymentError` being raised — currently this would surface as an unhandled exception in tests, which should be caught.
- No test for the lock TTL/payment timeout race scenario.
- No test for malformed gateway response (`KeyError` on `charge_id`).
- `test_lock_release_uses_lua_script` is a reasonable structural test but does not verify the token passed to the script matches what was set in Redis — the token-verification guarantee is untested end-to-end.

### 8. `customer_id` Has No Pattern Validation

`order_id` enforces `^[a-zA-Z0-9_\-]+$` but `customer_id` does not. If `customer_id` is used in any downstream query construction, this is an inconsistency worth closing.

---

## What Is Done Well

- Lua script for atomic lock release is correct and the approach is sound.
- `SET NX EX` is the right primitive for distributed locking.
- Pydantic validation with pattern enforcement on `order_id` is good.
- Structured logging with consistent field naming is clean.
- The `contextmanager` lock abstraction is readable and the `finally` release is correctly placed.
- Retry with backoff in the payment client is a reasonable pattern.

---

The lock mechanism itself is architecturally sound, but the gap between charging and recording, combined with the TTL/timeout race, means the duplicate-charge guarantee stated in the PR description does not hold under realistic failure conditions. Address the critical items before re-review.
