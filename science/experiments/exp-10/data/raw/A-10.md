Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical atomicity gap between the payment charge and the database commit that creates a real-world double-charge vector, and several secondary issues compound the risk enough that this cannot be approved in its current form.

---

## Critical Issues

### 1. Charge-Before-Record Creates an Unrecoverable Double-Charge Window

This is the most serious flaw in the design. The sequence inside the lock is:

```python
# Step 1 — external side effect (money moves)
charge_id = self.payment.charge(...)

# Step 2 — record keeping (may never execute)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, the network drops, the DB is unavailable, or an unhandled exception fires between Step 1 and Step 2, the customer has been charged but `db.get_charge()` will return `None` on the next attempt. The idempotency check will pass, and the worker will charge again. The distributed lock does not protect against this — it only serializes concurrent workers, not crash-recovery re-entrancy.

**This is a textbook at-least-once payment problem being sold as at-most-once.**

The standard mitigation is a two-phase approach: persist a `PENDING` charge record with a deterministic idempotency key *before* calling the payment gateway, then update it to `COMPLETE` after. The idempotency check must gate on the existence of *any* record for the order, not just a completed one.

```python
# Correct ordering
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents, idempotency_key)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents, idempotency_key=idempotency_key)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check then becomes:

```python
existing = self.db.get_charge(req.order_id)  # catches PENDING and COMPLETE
if existing:
    # optionally: if existing.status == PENDING, attempt recovery
    return OrderResult(processed=False, message="Order already processed or in progress")
```

### 2. `PaymentError` Is Silently Swallowed Inside the Lock Context

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    ...
```

`PaymentError` and any other unexpected exception propagate out of the `with` block, which correctly releases the lock via `finally`. However, the outer `try/except` only catches `LockError`:

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
```

Any `PaymentError`, `requests.RequestException`, `DatabaseError`, or unhandled runtime exception will bubble up to the caller completely unhandled. The caller gets an unformatted exception rather than a structured `OrderResult`. This is inconsistent with the contract the method signature implies and will cause upstream failures in whatever is consuming this service.

Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, order_id=req.order_id, message="Internal processing error")
```

### 3. Lock TTL Is Dangerously Undersized for the Actual Critical Section Duration

```python
ORDER_LOCK_TTL = 60  # seconds
```

The actual critical section includes:

- `db.get_charge()` — DB round-trip
- `payment.charge()` — up to `30s × 3 retries + 5s × 2 backoffs = 100 seconds` worst case
- `db.transaction()` with two writes

The payment client alone can run for **100 seconds** at maximum retry depth. The lock TTL is 60 seconds. This means the lock *will* expire before the payment client exhausts its retries in a degraded gateway scenario, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge.

Either:
- Reduce `PaymentClient.MAX_RETRIES` and `RETRY_BACKOFF` so the worst-case fits inside the TTL with margin, or
- Implement lock extension (a background thread or heartbeat that refreshes the TTL while the critical section is active), or
- Document and enforce a hard constraint that `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) < ORDER_LOCK_TTL` and add a startup assertion.

The current values violate this constraint by a factor of ~1.67x.

## Secondary Issues

### 4. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some versions. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials. Sanitize the error before logging or log only field names and error types, not values.

```python
safe_errors = [{"field": e["loc"], "type": e["type"]} for e in exc.errors()]
logger.warning("Invalid order request fields=%s", safe_errors)
```

### 5. Lock Retry Backoff Is Fixed Linear, Not Jittered

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Under thundering herd conditions (many workers hitting the same order simultaneously), synchronized linear backoff causes retry storms at predictable intervals. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

### 6. `db.get_charge()` Is Outside the Database Transaction

The idempotency read and the subsequent write are not atomic at the database level. The distributed lock handles the concurrency concern here, but this is a latent correctness dependency — if the lock is ever removed or weakened, this becomes a TOCTOU race. Consider moving the idempotency check inside the DB transaction with a `SELECT FOR UPDATE` or equivalent, or at minimum add a unique constraint on `order_id` in the schema so the DB enforces idempotency as a hard invariant independent of application logic.

### 7. `PaymentClient` Has No Authentication Headers

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no `Authorization` headers, API keys, mTLS configuration, or any credential material on this request to an internal payment gateway. Even on an internal network, unauthenticated payment API calls are a significant security concern. This needs to be addressed before any production deployment.

### 8. Test Coverage Gaps

The tests do not cover:

- `PaymentError` raised by `payment.charge()` — what does the caller receive?
- Lock TTL expiry during processing — no test validates behavior when the lock expires mid-flight
- DB transaction failure after a successful charge — the core crash-recovery scenario from Issue 1
- `payment_token` appearing in logs on validation failure
- Retry behavior in `PaymentClient` (no tests exist for `payment_client.py` at all)

The `PaymentClient` has zero test coverage despite containing the retry logic that is central to the TTL violation in Issue 3.
