Decision: Request Changes

Summary: The core locking and idempotency logic contains a critical TOCTOU race condition that directly undermines the stated "at-most-once payment guarantee" — a charge can fire between the idempotency check and the DB write if the Redis lock expires mid-flight, and the payment client's retry logic can silently produce duplicate charges on timeout. Several secondary issues around error handling, observability, and test coverage also need to be addressed before this is production-ready.

---

## Detailed Review

### 🔴 Critical — Must Fix

#### 1. TOCTOU Race: Charge Fires Outside the Database Transaction

This is the most serious flaw. The sequence is:

```python
# Inside the lock — correct
existing = self.db.get_charge(req.order_id)   # (1) idempotency check

# Payment fires OUTSIDE the DB transaction
charge_id = self.payment.charge(...)           # (2) money moves here

# DB write happens separately, after the charge
with self.db.transaction():
    self.db.record_charge(...)                 # (3) recorded here
    self.db.mark_order_complete(...)
```

If the process crashes, the Redis lock expires, or the DB transaction fails between steps (2) and (3), the charge has been collected but is never recorded. A retry will then pass the idempotency check (no record exists) and charge again. The PR summary claims atomic behavior, but the charge and the record are not atomic.

**Fix:** The payment charge must be recorded in a way that is recoverable. A common pattern is to write a "charge pending" record with the payment token *before* calling the gateway, then update it to "complete" after. This gives you an audit trail for reconciliation and prevents silent double-charges on crash.

```python
with self.db.transaction():
    # Reserve the order slot — this is the idempotency anchor
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Now charge — if this crashes, the pending record exists for reconciliation
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

#### 2. Redis Lock TTL Does Not Cover the Payment Client's Retry Window

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5    # seconds between retries
REQUEST_TIMEOUT = 30 # seconds per attempt
```

Worst-case payment client runtime: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock, pass the idempotency check (nothing recorded yet), and issue a second charge while the first worker is still in its retry loop. These two constants are coupled and must be documented and enforced together, or the lock TTL must be derived from the payment client's parameters.

```python
# Minimum: make the relationship explicit and enforced
ORDER_LOCK_TTL = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
    + 30  # buffer
)
```

#### 3. Payment Timeout Retries Can Produce Duplicate Charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request *may have succeeded* on the gateway side — the response was simply lost in transit. Retrying unconditionally on timeout without an idempotency key sent to the gateway will charge the card a second time if the first request landed. This is a standard pitfall with payment APIs.

**Fix:** Generate a client-side idempotency key per logical charge attempt and pass it in the request headers. Most payment gateways support this (`Idempotency-Key`). Do not retry with a new key.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 🟡 Significant — Should Fix

#### 4. Unhandled `PaymentError` Propagates Through the Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # raises PaymentError
    # lock is released via finally — correct
    # but PaymentError is never caught in process()
```

`PaymentError` is not caught in `process()`. It will propagate as an unhandled exception to the caller. The caller has no way to distinguish a payment failure from a programming error. This should return a structured `OrderResult` with `processed=False` and a safe error message, not a raw exception.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

#### 5. Lock Retry Backoff Is Too Aggressive and Blocks the Worker Thread

```python
LOCK_RETRY_ATTEMPTS = 3
# backoff: 0.5s, 1.0s, 1.5s — total blocking time up to 2s before giving up
```

`time.sleep()` in a synchronous worker blocks the entire thread. With 3 attempts and linear backoff, a worker can be held for up to 2 seconds on a hot order. If this service runs under a thread pool or async framework, this will degrade throughput. Consider whether the retry belongs here at all — in most cases, lock contention on a single order means the caller should be told to retry, not that the worker should spin.

---

#### 6. `customer_id` Has No Pattern Validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary Unicode, including control characters and injection payloads. Apply the same or equivalent pattern.

---

#### 7. `payment_token` Is Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is not directly logged, which is good. However, confirm that `PaymentError` messages, debug-level request logging from the `requests` library, or any middleware do not echo the token. This field should be marked sensitive and excluded from any serialization used in logs or error reporting.

### 🔵 Minor — Worth Noting

#### 8. Test for Lock Release Does Not Verify the Token Argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()  # does not check args
```

This test verifies the script was called but not that the correct token was passed. A bug that passes an empty string or a hardcoded value would pass this test. Assert on `call_args` to verify the token round-trips correctly.

---

#### 9. No Test for the Crash-Between-Charge-and-Record Scenario

There is no test that simulates `db.record_charge` raising an exception after `payment.charge` succeeds. This is exactly the failure mode described in issue #1 and should be a required test case once the architecture is corrected.

---

#### 10. `CHARGE_ENDPOINT` Is Hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoints make environment promotion (staging → production) error-prone and untestable. This should be injected via configuration or environment variable.

The structural approach — distributed locking, Lua-script release, Pydantic validation, idempotency checks — is sound and the right direction. The implementation just has a gap between what the architecture promises and what the code actually guarantees under failure conditions. Fix the three critical items and this will be in good shape for re-review.
