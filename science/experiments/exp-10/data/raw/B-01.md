Decision: Request Changes

Summary: The implementation demonstrates solid foundational patterns — atomic lock acquisition with `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there is a critical race condition between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, and several additional issues ranging from security to reliability that must be addressed before this can be approved.

---

## Critical Issues

### 1. Charge-Before-Record Race Condition — Breaks At-Most-Once Guarantee

This is the most serious flaw. The current sequence is:

```python
# 1. Charge the payment method
charge_id = self.payment.charge(...)

# 2. Record the charge atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, is killed, or loses its Redis connection between step 1 and step 2, the customer has been charged but no record exists. On retry, `db.get_charge()` returns `None`, the idempotency check passes, and the customer is charged again. The distributed lock does not protect against this because it is a temporal gap, not a concurrency gap.

**Required fix:** The payment gateway must support idempotency keys. Pass a stable, deterministic idempotency key derived from `order_id` to the payment gateway on every attempt. This ensures that if the charge call is retried after a crash, the gateway returns the original `charge_id` rather than creating a new charge. This is the standard pattern used by Stripe, Braintree, and similar processors.

```python
# Derive a stable idempotency key — same order always maps to same key
idempotency_key = f"order-{req.order_id}"

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,
)
```

`PaymentClient.charge()` must be updated to forward this key as a header (e.g., `Idempotency-Key`) to the gateway. Without this, the at-most-once claim in the docstring is false.

---

### 2. Payment Timeout Retry Is Unsafe Without Idempotency Keys

Directly related to issue 1. `PaymentClient` retries on `requests.exceptions.Timeout`. A timeout does not mean the charge failed — the request may have reached the gateway and succeeded. Retrying a timed-out charge without an idempotency key will create duplicate charges.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ← Retrying here without an idempotency key is a double-charge risk
```

This must be fixed in conjunction with issue 1. Once idempotency keys are in place, timeout retries become safe.

## Significant Issues

### 3. Lock TTL Is Insufficient to Cover Payment Retries

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
MAX_RETRIES = 3
RETRY_BACKOFF = 5    # seconds between retries
REQUEST_TIMEOUT = 30 # seconds per attempt
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock expires in 60 seconds. A second worker can acquire the lock while the first is still mid-payment, defeating the entire concurrency protection.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed worst-case payment duration (e.g., 120–150 seconds with margin), or implement lock extension (e.g., a background thread that refreshes the TTL while the payment call is in progress). The TTL must be documented as a function of the payment client's configuration so the two values stay in sync.

---

### 4. Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # Can raise PaymentError
    # ← PaymentError is not caught here
```

`PaymentError` is not caught inside `process()`. It will propagate out of the `with` block, through `_acquire_lock`'s `finally` (which correctly releases the lock), and then surface as an unhandled exception to the caller. The caller receives a raw exception rather than an `OrderResult`. This should be caught and returned as a structured result.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 5. `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), the token value may appear in the log output. Payment tokens are sensitive credentials and must never appear in logs.

**Required fix:** Either log only `e.error_count()` and field names without values, or explicitly redact the `payment_token` field before logging.

---

## Minor Issues

### 6. Lock Retry Backoff Is Linear, Not Exponential

The PR description says "retry with backoff" but the implementation is linear:

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

This is not exponential backoff. For a distributed system under contention, exponential backoff with jitter is the correct pattern to avoid thundering herd. This is a minor point but the PR description is misleading.

---

### 7. Test Coverage Gaps

The test suite validates the happy path and basic contention but is missing coverage for the failure modes introduced by this code:

- No test for `PaymentError` being raised during `charge()` — currently this would cause an unhandled exception (see issue 4)
- No test for the lock TTL expiring mid-payment (lock expiry scenario)
- No test verifying that `payment_token` does not appear in log output on validation failure
- No test for the `db.transaction()` context manager raising an exception after a successful charge (the crash-between-charge-and-record scenario)

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Add idempotency keys to payment gateway calls |
| 2 | Critical | Timeout retries are unsafe without idempotency keys |
| 3 | High | Lock TTL must exceed worst-case payment duration |
| 4 | High | `PaymentError` must be caught and returned as `OrderResult` |
| 5 | High | `payment_token` may be written to logs via Pydantic error output |
| 6 | Low | Backoff is linear, not exponential as described |
| 7 | Low | Test coverage gaps on failure paths |

Issues 1 through 5 must be resolved. The architecture is sound and the locking primitives are correctly implemented — the primary concern is that the at-most-once guarantee claimed in the docstring is not actually delivered by the current code.
