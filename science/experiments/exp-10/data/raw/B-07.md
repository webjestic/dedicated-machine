Decision: Request Changes

Summary: The implementation has sound fundamentals — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there is a critical TOCTOU race that breaks the at-most-once guarantee, a charge-before-commit ordering flaw that can produce unrecoverable double-charges, and several secondary issues that must be resolved before this touches production.

---

## Critical Issues

### 1. Charge Happens Outside the Database Transaction — Unrecoverable Double-Charge Window

This is the most serious bug in the PR.

```python
# Charge the payment method
charge_id = self.payment.charge(...)          # ← charge fires HERE

# Record the charge and mark the order complete atomically
with self.db.transaction():
    self.db.record_charge(...)                # ← recorded HERE
    self.db.mark_order_complete(...)
```

The charge is issued to the payment gateway **before** it is recorded in the database. If the process crashes, the network drops, or the DB transaction rolls back after `payment.charge()` returns, the customer has been charged but no record exists. The next worker that picks up this order will pass the idempotency check (`get_charge` returns `None`) and charge again.

**Fix:** The charge must be recorded atomically with the DB commit, and the system must be designed to tolerate a "charge succeeded, record failed" state — typically via an outbox/saga pattern, or at minimum by persisting a "charge-in-flight" marker **before** calling the gateway, then confirming or voiding on recovery.

---

### 2. TOCTOU Race on the Idempotency Check

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← read
    if existing:
        return ...
    charge_id = self.payment.charge(...)           # ← act
```

The lock is correct for preventing two workers from entering this block simultaneously, but the idempotency check and the charge are not atomic with respect to the database. If the DB read uses `READ COMMITTED` isolation and another transaction committed between the lock acquisition and the read completing, the check can still be stale. More concretely, if the lock TTL expires mid-flight (60 seconds is tight for a payment round-trip with retries — `MAX_RETRIES=3` × `RETRY_BACKOFF=5s` × `REQUEST_TIMEOUT=30s` = up to 105 seconds worst case), a second worker can acquire the lock, pass the idempotency check, and charge again before the first worker's DB write commits.

**Fix:** The lock TTL must be strictly greater than the maximum possible processing time. With the current payment client, the minimum safe TTL is ~120 seconds. Alternatively, write a "processing" sentinel to the DB inside the lock before charging, so any concurrent worker sees it even if the lock has expired.

## Significant Issues

### 3. `payment_token` Logged or Leaked via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic's `ValidationError.__str__` includes field values in some versions. `payment_token` is a sensitive credential. If validation fails on an adjacent field, the token can appear in logs.

**Fix:** Sanitize the error before logging, or log only field names and error types, never values. Mark `payment_token` with `repr=False` and consider a `SecretStr` type.

```python
safe_errors = [{"field": e["loc"], "type": e["type"]} for e in exc.errors()]
logger.warning("Invalid order request: %s", safe_errors)
```

### 4. Lock Release Silently Swallows Failures

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script call raises (Redis connection lost, script evicted), the exception is swallowed by the `finally` block's implicit suppression and the log line claims the lock was released when it was not. The lock will hold until TTL expiry, which is acceptable for correctness but the silent failure is a diagnostic hazard.

**Fix:**

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error("Lock release returned 0 — token mismatch or TTL expired order_id=%s", order_id)
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s — will expire via TTL", order_id)
```

### 5. Payment Client Retries on Timeout Are Unsafe Without Idempotency Keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout does not mean the charge failed. The gateway may have processed it and the response was lost in transit. Retrying without an idempotency key sent to the gateway will issue a second charge. This is a financial correctness bug.

**Fix:** Generate a stable idempotency key (e.g., derived from `order_id`) and pass it as a header on every attempt:

```python
headers = {"Idempotency-Key": f"order-{order_id}"}
response = requests.post(..., headers=headers)
```

### 6. `requests.post` Has No Connection Timeout — Only a Read Timeout

```python
timeout=self.REQUEST_TIMEOUT
```

Passing a single integer to `requests` sets only the read timeout. A connection that hangs during the TCP handshake will block indefinitely.

**Fix:**

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

## Minor Issues

### 7. Test Coverage Gaps

- No test for the charge-succeeds-but-DB-fails scenario (the critical bug above).
- No test verifying that `payment_token` does not appear in log output on validation failure.
- No test for lock TTL expiry mid-processing.
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called but does not assert the script was called with the correct token, making it a weak correctness check.

### 8. `ORDER_LOCK_TTL` Is a Class-Level Magic Number

The TTL is not derived from or validated against the payment client's maximum possible latency. These two values will drift independently. They should be co-located or the processor should assert at construction time that `ORDER_LOCK_TTL > payment_max_latency`.
