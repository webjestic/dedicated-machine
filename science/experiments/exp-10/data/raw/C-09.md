Decision: Request Changes

Summary: The core locking and idempotency design has a critical TOCTOU race condition: the payment charge happens *outside* the database transaction, meaning a successful charge can occur without a corresponding record if the process crashes between the charge call and the `db.transaction()` block — this is a real double-charge risk in production. Additionally, unhandled exceptions from the payment client and database layer will propagate out of `process()` uncaught, leaving the lock held until TTL expiry and giving callers no structured error response. The test suite also has meaningful gaps that would have caught these issues.

---

## Detailed Findings

### 🔴 Critical — TOCTOU / Charge-Without-Record Window

```python
# services/order_processor.py

# Charge the payment method
charge_id = self.payment.charge(          # ← charge succeeds here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():               # ← crash/exception here = money taken, no record
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process dies, the DB connection drops, or `db.transaction()` raises *after* the charge succeeds, the customer is billed with no record. The idempotency check will then find nothing on retry and charge again.

**Required fix:** The charge must be recorded in a way that survives this window. Standard approaches:

1. **Outbox / two-phase commit:** Write a *pending charge intent* to the DB inside a transaction first, then charge, then mark it confirmed. On retry, detect the pending intent and reconcile against the gateway.
2. **Idempotency key on the gateway call:** Pass `order_id` as the idempotency key to the payment gateway so duplicate calls are safe, then always record the result.
3. At minimum, wrap the charge *and* the DB write in a compensating pattern with explicit reconciliation logic.

### 🔴 Critical — Unhandled Exceptions Escape `process()`

```python
# services/order_processor.py

with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # can raise — uncaught
    ...
    charge_id = self.payment.charge(...)           # raises PaymentError — uncaught
    with self.db.transaction():
        self.db.record_charge(...)                 # can raise — uncaught
        self.db.mark_order_complete(...)           # can raise — uncaught
```

`PaymentError`, database exceptions, and network errors all propagate to the caller as unhandled exceptions. The method signature promises an `OrderResult` return. Callers will receive a raw exception instead of a structured response, and upstream error handling is now the caller's responsibility with no contract for what exceptions to expect.

**Required fix:** Catch `PaymentError` and general DB/infrastructure exceptions explicitly and return structured `OrderResult` failures. Log the full exception with stack trace for observability.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 🟡 Medium — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)   # ← e may contain field values
```

Pydantic validation errors can include the submitted value in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in logs. Payment tokens are sensitive credentials.

**Required fix:**

```python
logger.warning("Invalid order request order_id=%s: %s", order_id, e.error_count())
```

Or sanitize the error before logging — log field names and error types only, never values.

### 🟡 Medium — `PaymentClient` Has No Authentication

```python
# services/payment_client.py

response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

No `Authorization` header, no mTLS configuration, no API key. Even on an internal network, this is a concern for a government contracting environment. The endpoint is hardcoded as a class attribute with no injection point, making it untestable without monkeypatching.

**Required fix:** Accept auth credentials and endpoint via constructor injection. Do not hardcode internal service URLs as class-level constants.

---

### 🟡 Medium — Retry Logic Retries on Timeout Without Idempotency Guarantee

```python
# services/payment_client.py

except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying a timed-out charge without an idempotency key on the gateway request can result in duplicate charges. This compounds the TOCTOU issue above.

**Required fix:** Pass a stable idempotency key (e.g., `order_id`) in the request headers to the payment gateway on every attempt so retries are safe.

---

### 🟡 Medium — Lock TTL May Be Insufficient Under Retry Backoff

```python
ORDER_LOCK_TTL = 60        # seconds

# PaymentClient
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Worst-case payment client execution time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. This exceeds the 60-second lock TTL. The lock will expire while a worker is still processing, allowing a second worker to acquire it and begin a concurrent charge on the same order.

**Required fix:** Either raise `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime (e.g., 120–150s), or implement lock extension (refresh the TTL mid-processing).

---

### 🟠 Minor — Test Suite Gaps

The tests pass but do not cover the failure modes that matter most:

| Missing Test | Why It Matters |
|---|---|
| `PaymentError` raised during charge | Verifies structured `OrderResult` returned, not raw exception |
| DB exception after successful charge | Validates the TOCTOU window behavior (currently broken) |
| Lock TTL expiry mid-process | Verifies behavior when lock expires before release |
| `payment_token` not present in logs | Validates the sensitive data logging concern |
| Retry exhaustion in `PaymentClient` | Verifies `PaymentError` raised after N attempts |
| Timeout retry behavior | Verifies idempotency key is sent on retried requests |

---

### 🟠 Minor — `charge_id` Missing from Idempotency Response

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id not returned
    )
```

If a caller receives this response, they have no way to retrieve the existing `charge_id` for reconciliation without a separate DB lookup. Return it from the existing record.

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| 🔴 Critical | Charge-without-record TOCTOU window — redesign required |
| 🔴 Critical | Unhandled exceptions escape `process()` — catch and return structured results |
| 🟡 Medium | Payment token may appear in validation error logs |
| 🟡 Medium | No auth on payment gateway requests |
| 🟡 Medium | Timeout retries without idempotency key — duplicate charge risk |
| 🟡 Medium | Lock TTL shorter than max payment client runtime |
| 🟠 Minor | Test coverage misses all critical failure paths |
| 🟠 Minor | Idempotency response omits existing `charge_id` |

The distributed locking mechanism itself (`SET NX EX` + Lua release script) is correctly implemented. The Pydantic validation and input pattern enforcement are solid. The structural issues are in the interaction between the lock, the charge, and the DB write — fix those and this is close to approvable.
