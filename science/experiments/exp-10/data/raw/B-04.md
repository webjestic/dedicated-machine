Decision: Request Changes

Summary: The implementation has a critical TOCTOU flaw — the payment charge happens *outside* the database transaction, meaning a successful charge can occur with no corresponding record if the DB write fails, producing an unrecoverable double-charge on retry. Additionally, the lock TTL of 60 seconds is dangerously short relative to the combined payment retry window (up to ~75 seconds with 3 attempts × 5s backoff + 30s timeout each), meaning the lock can expire mid-flight and a second worker can enter the critical section while the first is still charging. There are also several secondary issues detailed below that need to be addressed before this is mergeable.

---

## Critical Issues

### 1. Charge Occurs Outside the Database Transaction (TOCTOU / At-Most-Once Violation)

This is the most serious bug. The code charges the payment gateway, then opens a DB transaction to record it:

```python
# Charge the payment method
charge_id = self.payment.charge(...)          # ← charge happens HERE

# Record the charge and mark the order complete atomically
with self.db.transaction():                   # ← DB write happens HERE
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the DB transaction fails after the charge succeeds, the system has taken money with no record of it. On the next retry, the idempotency check (`db.get_charge`) finds nothing, so it charges again. This is a double-charge vulnerability. The payment gateway charge and the DB record are not in the same atomic unit, and there is no compensating rollback (refund) in the exception path.

**Required fix:** Either use an outbox/saga pattern, or at minimum catch exceptions after the charge and attempt a refund before re-raising. The idempotency key should be written to the DB *before* the charge is attempted (as a pending record), so that any retry path sees the in-progress state and can reconcile rather than re-charge.

---

### 2. Lock TTL Is Shorter Than the Maximum Payment Retry Window

```python
ORDER_LOCK_TTL = 60        # seconds
```

In `PaymentClient`:
```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock while the first is still in the payment retry loop, breaking the mutual exclusion guarantee entirely and enabling concurrent charges on the same order.

**Required fix:** Either extend `ORDER_LOCK_TTL` beyond the worst-case payment duration (minimum 120s with margin), or implement lock heartbeating/renewal. The TTL and payment client constants must be co-designed and enforced together, ideally derived from a shared constant.

## Significant Issues

### 3. Unhandled Exceptions Inside the Lock Context Silently Release the Lock Without Cleanup

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    with self.db.transaction():
        ...
```

If `self.payment.charge()` raises `PaymentError`, or if the DB transaction raises, the exception propagates out of the `with` block. The lock is released (correctly), but the exception is not caught in `process()`. It will bubble up to the caller as an unhandled exception rather than returning a structured `OrderResult`. This means callers get a raw exception instead of the documented return type, breaking the API contract.

**Required fix:** Add an explicit `except (PaymentError, Exception)` handler inside `process()` that returns a structured `OrderResult(processed=False, ...)` for each failure mode.

---

### 4. `payment_token` Is Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include the field value in their string representation. If `payment_token` fails validation (e.g., too long), its value may appear in the log output. Payment tokens are sensitive credentials and must never appear in logs.

**Required fix:** Scrub or selectively format the validation error to exclude the `payment_token` field value before logging.

---

### 5. Lock Retry Backoff Is Insufficient and Misaligned

```python
LOCK_RETRY_ATTEMPTS = 3
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Total wait before giving up: ~3 seconds. If the lock TTL is 60 seconds, a worker that fails to acquire the lock after 3 seconds gives up and returns an error to the caller, even though the lock will be held for up to another 57 seconds. The retry window is not proportional to the lock duration. For a 60-second lock, callers will receive contention errors that are misleading — the order is not being double-processed, it is simply still in flight.

**Required fix:** Either increase retry attempts/backoff to cover a meaningful portion of the lock TTL, or document clearly that callers are expected to retry at the application layer. Consider exponential backoff with jitter.

---

### 6. Test Coverage Gaps

The tests do not cover the most important failure scenarios:

- **No test for `PaymentError` propagation** — the unhandled exception path (issue #3) is not exercised.
- **No test for DB transaction failure after successful charge** — the critical TOCTOU scenario (issue #1) is not tested.
- **No test for lock expiry during payment** — the TTL race condition (issue #2) is not simulated.
- **`test_lock_release_uses_lua_script`** asserts `_release_script` is called but does not verify the correct `keys` and `args` are passed, so it would not catch a regression where the wrong token or key is used.

---

### 7. `payment_token` Field Validation Is Too Permissive

```python
payment_token: str = Field(..., min_length=1, max_length=256)
```

No pattern constraint is applied to `payment_token`. Depending on the payment gateway's token format, this could allow unexpected values through. At minimum, document why no pattern is applied; ideally add a format constraint consistent with what the gateway issues.

---

## Minor Issues

- `ORDER_LOCK_TTL` and `PaymentClient` timeout/retry constants are defined in separate classes with no enforced relationship. Consider centralizing them or adding an assertion at startup.
- The `CHARGE_ENDPOINT` is hardcoded as a class attribute string. It should be injected via configuration or environment variable to support different deployment environments.
- The `finally` block in `_acquire_lock` logs "lock released" even when the Lua script returns `0` (meaning the lock had already expired or was taken by another worker). This is a misleading log message in the TTL-expiry scenario.
