Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness and security flaws that must be resolved before this can be approved, the most severe being a charge-then-record race condition that directly undermines the at-most-once guarantee this PR claims to provide.

---

## Review

### 🔴 Critical — Charge-Before-Record Race Condition (At-Most-Once Guarantee is Broken)

This is the most serious flaw in the PR. The docstring explicitly claims an **at-most-once payment guarantee**, but the implementation does not deliver it.

```python
# Charge the payment method
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The charge is issued to the external payment gateway **before** it is recorded in the database. If the process crashes, is killed, loses network connectivity, or the database transaction fails after `payment.charge()` returns, the customer has been charged but no record exists. The next worker that picks up this order will pass the idempotency check (`get_charge` returns `None`), acquire the lock, and **charge again**.

This is a double-charge scenario. The distributed lock only protects against concurrent execution — it provides zero protection against this failure window.

**Required fix:** The database must be the system of record. Record the *intent* to charge before issuing the charge, and use a state machine (e.g., `PENDING → CHARGING → COMPLETE`) to handle crash recovery. Alternatively, if the payment gateway supports idempotency keys, pass a deterministic idempotency key derived from `order_id` to the gateway so that retries of the same logical charge are deduplicated at the gateway level. The idempotency key must be stored before the charge attempt.

```python
# Safer pattern sketch
with self.db.transaction():
    self.db.record_charge_intent(req.order_id, idempotency_key)

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,  # gateway-level dedup
)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

### 🔴 Critical — `PaymentError` is Silently Swallowed Inside the Lock

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    ...
```

`PaymentError` (and any other unexpected exception) is not caught inside the lock context. When `payment.charge()` raises, the exception propagates out of `_acquire_lock`, the lock is correctly released by `finally`, and then the exception continues to propagate out of `process()` entirely — **uncaught**. The caller receives an unhandled exception rather than an `OrderResult`.

The outer `except LockError` block only catches `LockError`. Any `PaymentError`, `requests.exceptions.ConnectionError`, database error, or unexpected `Exception` will bubble up as an unhandled exception.

```python
except LockError:
    return OrderResult(
        processed=False,
        message="Order currently being processed by another worker",
    )
# PaymentError, db errors, and all other exceptions: unhandled
```

**Required fix:** Add explicit handling for `PaymentError` and a catch-all for unexpected exceptions, returning structured `OrderResult` responses in all cases. Log the full exception with `exc_info=True` for unexpected errors.

### 🔴 Critical — Lock TTL Is Insufficient and Miscalibrated Against Retry Duration

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
# PaymentClient
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

The maximum possible execution time inside the lock is:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. The lock will **expire while the payment is still in flight** on a worst-case retry path. Another worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still retrying. This is another double-charge vector.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum possible critical section duration (at minimum 120s with margin), or implement lock extension (e.g., a background thread that refreshes the TTL while the critical section is active, similar to Redlock's watchdog pattern). The TTL and payment client timeout/retry parameters must be documented as coupled constants and validated together.

### 🟠 High — Redis Connection Failures Are Not Handled

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If the Redis client raises `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` during lock acquisition or release, these exceptions propagate uncaught. A Redis outage would cause all order processing to fail with unhandled exceptions rather than graceful degradation or a structured error response.

Additionally, if Redis fails during the `finally` block of `_acquire_lock`, the lock release silently fails and the exception from the release attempt may mask the original exception from the `try` block.

**Required fix:** Wrap Redis operations in explicit exception handling. Decide and document the failure policy: fail open (process without lock, risking concurrency) or fail closed (return error). For a payment processor, fail closed is almost certainly correct. Log Redis failures with high severity.

### 🟠 High — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., it exceeds `max_length=256`), the token value may be included in the log output. Payment tokens must never appear in logs.

**Required fix:** Do not log the raw `ValidationError`. Log only the field names and error types, not the values. Scrub or explicitly exclude sensitive fields from any logged validation output.

---

### 🟠 High — `payment_token` Stored in `OrderRequest` Model Without Sanitization Boundary

`payment_token` is carried through the `OrderRequest` Pydantic model and passed directly to the payment client. There is no explicit marking of this field as sensitive, no scrubbing in `__repr__` or `__str__`, and no documented handling policy. If `OrderRequest` instances are ever serialized, logged, or included in error messages (e.g., via a framework exception handler), the token will leak.

**Required fix:** Override `__repr__` and `__str__` on `OrderRequest` to mask `payment_token`, or use a dedicated `SecretStr` type from Pydantic which suppresses the value in all string representations.

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

### 🟡 Medium — Lock Retry Backoff Is Linear and Contention Window Is Narrow

```python
LOCK_RETRY_ATTEMPTS = 3
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Three attempts with linear backoff totaling 3 seconds is a very narrow contention window. Under high concurrency, workers that arrive while a lock is held will fail quickly and return an error to the caller. Whether this is appropriate depends on the expected lock hold time, but given that the lock can be held for up to 100 seconds (per the TTL issue above), a worker that arrives 2 seconds into that window will always fail. This may be intentional, but it needs to be explicitly documented as a design decision, not left as an implicit behavior.

### 🟡 Medium — `db.get_charge` Is Outside the Database Transaction

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)  # outside transaction
    if existing:
        return ...

    charge_id = self.payment.charge(...)

    with self.db.transaction():  # transaction starts here
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

The idempotency check reads from the database outside of a transaction. If the database has read-committed or lower isolation, a concurrent write could theoretically be invisible to this read. The distributed lock is the primary guard here, so this is lower severity, but the idempotency check should be inside the same transaction as the write, or the read should use `SELECT FOR UPDATE` semantics to be formally correct.

---

### 🟡 Medium — Test Suite Does Not Cover the Critical Failure Paths

The tests cover the happy path, already-processed, lock contention, and input validation. They do not cover:

- `payment.charge()` raising `PaymentError` — what does the caller receive?
- `payment.charge()` succeeding but `db.transaction()` raising — is the charge orphaned?
- Redis raising `ConnectionError` during lock acquisition
- Lock TTL expiry during payment (cannot be unit tested easily, but should be documented)
- `payment_token` not appearing in any log output (security regression test)

These are the exact scenarios where the critical bugs above manifest. Tests that only cover success and clean failure paths provide false confidence for a payment-critical service.
