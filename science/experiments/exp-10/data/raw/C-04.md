Decision: Request Changes

Summary: The core locking and idempotency architecture has a critical TOCTOU race condition that directly undermines the duplicate-charge guarantee this PR is designed to provide. There are also several secondary reliability and security issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. TOCTOU Race: Charge Happens Outside the Database Transaction

This is the most serious flaw. The payment charge and the database record are not atomic with respect to each other:

```python
# Lock is held here — good
charge_id = self.payment.charge(...)          # (A) Money leaves the customer

# If the process crashes between (A) and (B), money is gone but no record exists
with self.db.transaction():                   # (B) Record written
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the worker dies, loses its Redis connection, or the database transaction fails after `(A)`, the customer has been charged but no `charge_id` is ever recorded. On retry, `get_charge()` returns `None`, and the customer gets charged again. The Redis lock provides no protection here because it has already been released (or expired) by the time the retry occurs.

**The idempotency check is only as good as the record it reads.** If the record was never written, the check passes and a duplicate charge occurs.

**Recommended fix:** The `charge_id` must be persisted to a durable store *before or atomically with* the charge being considered "committed." A common pattern is a two-phase approach:

1. Write a `PENDING` charge record with a deterministic idempotency key *before* calling the payment gateway.
2. Call the gateway, passing that idempotency key to the gateway as well (most payment processors support this).
3. Update the record to `COMPLETE` on success.

On retry, the idempotency check finds the `PENDING` record and can either wait, recover, or surface the partial state — but it will never issue a second charge.

---

### 2. Lock Expiry Does Not Bound the Payment Call

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient.REQUEST_TIMEOUT = 30` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5`. Worst-case payment client runtime is approximately `3 × 30 + 2 × 5 = 100 seconds`, which exceeds the lock TTL by 40 seconds. A second worker can acquire the lock while the first is still mid-charge.

```python
ORDER_LOCK_TTL = 60        # Lock expires here
# ... payment client can run for ~100s ...
# Second worker acquires lock, passes idempotency check (no record yet),
# and issues a duplicate charge.
```

Either the TTL must exceed the maximum possible payment duration with margin, or the lock needs a heartbeat/renewal mechanism.

---

### 3. `PaymentError` Is Silently Swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` (and any other exception) propagates out of `process()` as an unhandled exception. The caller gets a raw exception instead of a structured `OrderResult`. This is inconsistent with the method's documented contract and will cause unhandled 500s at the API layer. Add an explicit `except PaymentError` (and likely a broad `except Exception` fallback with logging) that returns a structured failure result.

## Moderate Issues

### 4. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some configurations. If `payment_token` fails validation, its value (or a truncated form) may appear in log output. Payment tokens must never appear in logs. At minimum, scrub or summarize the error before logging, or log only field names.

### 5. Lock Release Silently Fails on Redis Error

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If the Redis call throws here (connection drop, timeout), the exception propagates out of the `finally` block and masks the original exception from the `try` body. Wrap the release in a `try/except` with logging so the original exception is preserved and the lock simply expires naturally via TTL.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
    except Exception:
        logger.exception("Failed to release lock for order_id=%s; will expire via TTL", order_id)
```

### 6. `customer_id` Has No Pattern Constraint

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. If `customer_id` is used in any downstream query or log aggregation, this is an inconsistency worth closing.

---

## Minor Issues

### 7. Test Coverage Gaps

- No test for `PaymentError` being raised by the payment client.
- No test for the lock expiry / TTL-vs-payment-duration scenario.
- No test verifying that `payment_token` does not appear in log output on validation failure.
- The `test_lock_release_uses_lua_script` test asserts the script was called but does not verify the correct `token` was passed, which is the entire point of the token-verified release.

### 8. `CHARGE_ENDPOINT` Hardcoded

`https://api.payments.internal/v1/charges` is hardcoded in the class body. This should come from configuration/environment so it can differ between environments without a code change.

---

## What's Done Well

The Lua script for atomic token-verified lock release is correct and the right approach. Input validation with the `order_id` pattern is solid and the path traversal test confirms it. The retry/backoff structure in `PaymentClient` is reasonable. Structured logging is consistent throughout.

The architecture is close — the fundamental shape is right — but the gap between the charge call and the durable record is a real-money bug that must be resolved before this merges.
