Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical charge-before-commit race condition that directly violates the at-most-once guarantee this PR claims to provide, and several secondary issues that must be addressed before this touches production infrastructure.

---

## Critical Issues

### 1. Charge Fires Outside the Database Transaction — At-Most-Once Guarantee is Broken

This is the most serious flaw in the PR. The docstring explicitly claims an *at-most-once payment guarantee*, but the implementation does not deliver it.

```python
# Charge the payment method  ← fires OUTSIDE the db.transaction() block
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():          # ← only the *recording* is atomic
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**The failure scenario is straightforward and reproducible:**

1. Worker A acquires the lock, passes the idempotency check (no existing charge), calls `payment.charge()` — the external charge succeeds and money moves.
2. Before `db.record_charge()` executes, the process crashes, the container is OOM-killed, the network drops, or the database throws.
3. The lock TTL expires (60 seconds).
4. Worker B acquires the lock, queries `db.get_charge()` — returns `None` because nothing was ever recorded.
5. Worker B calls `payment.charge()` again. **The customer is charged twice.**

The distributed lock only prevents *concurrent* double-charges. It provides zero protection against the charge-then-crash window, which is the failure mode that actually matters in payment systems.

**Required remediation — two viable paths:**

**Option A — Idempotency key on the payment gateway (preferred):**
Pass `order_id` as an idempotency key to the payment gateway on every attempt. Most payment processors (Stripe, Braintree, etc.) guarantee that retried requests with the same idempotency key return the original result rather than issuing a new charge. The idempotency check in the database then becomes a performance optimization rather than a correctness guarantee.

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # ← gateway deduplicates on its side
)
```

**Option B — Two-phase state machine in the database:**
Record a `PENDING` charge row *before* calling the payment gateway. On retry, detect the pending row, attempt to recover the charge status from the gateway, and only then transition to `COMPLETE`. This is significantly more complex but works when the gateway does not support idempotency keys.

Neither option is a minor tweak. This requires a design decision and must be resolved before merge.

## High Severity Issues

### 2. `PaymentError` is Silently Swallowed — Charges Can Succeed Without Being Recorded

```python
charge_id = self.payment.charge(...)

with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`PaymentError` is not caught anywhere in `process()`. If the payment gateway raises, the exception propagates out of the `with self._acquire_lock(...)` block, the lock is released cleanly, and the caller receives an unhandled exception. That is acceptable behavior *if the charge failed*. But if the charge *succeeded* and the gateway raised on the response parsing (e.g., malformed JSON, missing `charge_id` key), the charge is orphaned — money moved but nothing was recorded. This needs explicit handling:

```python
try:
    charge_id = self.payment.charge(...)
except PaymentError:
    logger.error("Payment failed for order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Payment failed")
```

Additionally, `response.json()["charge_id"]` in `PaymentClient.charge()` will raise a bare `KeyError` if the gateway returns a 200 with a malformed body. That exception is not caught and will surface as an unhandled error rather than a `PaymentError`.

```python
# Fragile
charge_id = response.json()["charge_id"]

# Required
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError("Gateway returned success but no charge_id in response")
```

### 3. Lock TTL Is Dangerously Mismatched With Payment Client Timeout

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `(30s timeout + 5s backoff) × 3 attempts = 105 seconds`.

The lock TTL is 60 seconds. A fully legitimate, slow-but-succeeding payment sequence will expire the lock before the charge completes. Another worker then acquires the lock, passes the idempotency check (still no recorded charge), and fires a second charge — while the first charge is still in-flight. This is a second, independent path to double-charging.

The lock TTL must be strictly greater than the maximum possible payment client execution time. Given the current retry configuration, `ORDER_LOCK_TTL` must be at minimum `120` seconds, and the relationship between these constants should be enforced programmatically, not left as two magic numbers in separate files:

```python
ORDER_LOCK_TTL = (
    PaymentClient.REQUEST_TIMEOUT + PaymentClient.RETRY_BACKOFF
) * PaymentClient.MAX_RETRIES + SAFETY_MARGIN_SECONDS
```

## Medium Severity Issues

### 4. `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., it is too long), the raw token value may appear in log output. Payment tokens are sensitive credentials. Sanitize the error before logging or log only field names and error types, never values.

### 5. Lock Retry Backoff Is Fixed Linear, Not Jittered

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Under high concurrency, all contending workers will sleep for identical durations and retry simultaneously, producing thundering herd behavior against Redis. This is a well-understood problem. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

### 6. `db.transaction()` Failure Leaves Lock Held Until TTL Expiry

If `self.db.transaction().__enter__` raises (database unavailable, connection pool exhausted), the exception propagates through `_acquire_lock`'s `finally` block, which correctly releases the lock. This is fine. However, if the database raises *inside* the transaction block after partial writes, the transaction should roll back — but whether it does depends entirely on the `db` abstraction, which is not defined in this PR. The contract of `db.transaction()` needs to be documented or enforced.

### 7. Test Coverage Gaps

The tests verify the happy path and a few contention scenarios but are missing coverage for the most important failure modes:

- Payment succeeds but `db.record_charge()` raises — verifies the orphaned charge scenario.
- Lock TTL expiry mid-payment — the core race condition identified in Issue 3.
- `PaymentError` propagation — currently swallowed silently.
- `response.json()["charge_id"]` `KeyError` path in `PaymentClient`.
- Lua script returning `0` (lock already expired before release) — the `finally` block does not check the return value; this should at minimum log a warning since it indicates the TTL was too short.

## Minor Issues

### 8. `customer_id` Has No Pattern Constraint

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary unicode strings up to 64 characters. If `customer_id` is used in any downstream query construction or logging context, this is a latent injection surface. Apply the same or a similarly restrictive pattern.

### 9. `CHARGE_ENDPOINT` Is a Hardcoded String Literal

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoints belong in configuration, not source code. This makes environment promotion (dev → staging → prod) impossible without code changes and creates a risk of accidentally pointing at the wrong environment.

---

## What Is Done Well

To be precise about what should be preserved:

- The Lua release script is correct. Token comparison before deletion is the right pattern and prevents a worker from releasing a lock it does not own.
- `SET NX EX` as a single atomic operation is correct. The old `SETNX` + `EXPIRE` two-command pattern is not used here, which is good.
- Pydantic validation with a strict regex on `order_id` is appropriate for an input that becomes a Redis key.
- The idempotency check pattern is structurally correct — the problem is purely that it cannot survive the charge-then-crash window without gateway-side idempotency keys.
- The `finally` block in `_acquire_lock` guarantees lock release even on exceptions, which is the correct pattern for a context manager.
