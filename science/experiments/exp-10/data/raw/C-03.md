Decision: Request Changes

Summary: The core locking and idempotency design has solid instincts — atomic SET NX EX, token-verified Lua release, and Pydantic input validation are all correct choices. However, there is a critical race condition between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, and several secondary issues compound the reliability and security risk. This cannot be approved until the charge-then-record gap is closed and the issues below are addressed.

---

## Critical Issues

### 1. Charge-Before-Record Race Condition (Correctness / Financial Risk)

This is the most serious flaw. The code charges the payment gateway **outside** the database transaction:

```python
# Charge the payment method
charge_id = self.payment.charge(...)          # ← money leaves here

# Record the charge and mark the order complete atomically
with self.db.transaction():                   # ← only then is it recorded
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, loses its DB connection, or the transaction rolls back **after** `payment.charge()` succeeds but **before** `db.record_charge()` commits, the customer has been charged with no record in the database. The idempotency check (`db.get_charge`) will find nothing on retry, and the customer will be charged again. This directly contradicts the PR's stated at-most-once guarantee.

The standard mitigation is a two-phase approach:

1. **Before charging:** Write a `PENDING` charge record inside a transaction, with the `order_id` and intended amount. This record acts as the idempotency anchor.
2. **Charge the gateway.**
3. **On success:** Update the record to `COMPLETED` with the returned `charge_id`.
4. **On failure:** Update the record to `FAILED` or leave it `PENDING` for reconciliation.

The idempotency check must then also detect `PENDING` records and treat them as "in-flight," not "safe to retry."

---

### 2. Unhandled `PaymentError` Propagates Through Lock Context Manager (Reliability)

`PaymentError` (and any other unexpected exception from `payment.charge()`) is not caught in `process()`. It will propagate up through `_acquire_lock`'s `finally` block — which correctly releases the lock — but then bubble out of `process()` as an unhandled exception. The caller receives a raw exception instead of an `OrderResult`. This breaks the contract of the method signature.

```python
# Nothing catches this:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
```

Add a `except PaymentError` (and a broad `except Exception` for unexpected failures) that returns a structured `OrderResult(processed=False, ...)`. Log the full exception with `exc_info=True` for the unexpected case.

---

### 3. Lock TTL Is Insufficient for Retry Duration (Reliability)

`ORDER_LOCK_TTL = 60` seconds, but the payment client can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The Redis lock will expire while the payment is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. The lock TTL must be set to comfortably exceed the maximum possible payment client execution time, or the payment client's timeout budget must be reduced to fit within the TTL with margin.

## Significant Issues

### 4. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is accepted as input and passed directly to the payment client. Confirm that no logging path (including any exception `__str__` that might include the request body) ever emits the raw token. The current code looks clean, but the `PaymentError` raised from `payment_client.py` includes the HTTP status, not the token — verify this holds if the requests library includes request bodies in its exception messages in your environment.

### 5. Lock Retry Backoff Is Blocking (Scalability)

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

In a synchronous worker this is acceptable but worth flagging: under high concurrency this blocks the thread for up to 3 seconds total per contention event. If this service is expected to move to async workers, `asyncio.sleep` will be required. Document the threading model assumption.

### 6. `db.transaction()` Exception Handling

If `db.transaction().__exit__` receives an exception, the behavior depends entirely on the `db` implementation. The code assumes the context manager handles rollback, but there is no explicit handling or logging if the transaction fails after a successful charge — which is exactly the crash scenario described in issue #1. This is partially subsumed by the fix for #1 but should be explicitly addressed.

## Test Coverage Gaps

### 7. No Test for `PaymentError` Path

There is no test covering what happens when `payment.charge()` raises `PaymentError`. Given that this is a financial service, this path must be tested explicitly.

### 8. No Test for Lock TTL vs. Payment Duration Mismatch

The test suite does not exercise the scenario where the lock expires mid-payment. This is hard to unit test but should at minimum be documented as a known risk with a comment in the code.

### 9. `test_lock_release_uses_lua_script` Is Fragile

```python
processor._release_script.assert_called_once()
```

This accesses a private attribute and asserts call count but not call arguments. It will not catch a regression where the script is called with the wrong key or the wrong token. Assert on `call_args` as well.

---

## Minor Notes

- `ORDER_LOCK_TTL` and `MAX_RETRIES` / `REQUEST_TIMEOUT` / `RETRY_BACKOFF` are defined in two separate classes with no shared configuration. A single config object or environment-variable-driven settings class would prevent them from drifting out of sync (as they already have in this PR).
- `customer_id` has no pattern validation, only length bounds. If it feeds into any query construction downstream, add a pattern consistent with your ID format.
- The `CHARGE_ENDPOINT` is hardcoded as a class attribute. This should come from environment configuration to support staging vs. production deployments without code changes.
