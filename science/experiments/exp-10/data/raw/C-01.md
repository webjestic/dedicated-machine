Decision: Request Changes

Summary: The core locking and idempotency design has a critical TOCTOU race that can cause double charges, and the payment client has a timeout ambiguity that could silently result in a charge with no recorded `charge_id`. These are not edge cases — they are realistic failure modes in a distributed system under load, and both need to be resolved before this merges.

---

## Critical Issues

### 1. TOCTOU Race Between Payment Charge and DB Record — Double-Charge Risk

This is the most serious bug in the PR. The sequence inside the lock is:

```python
# 1. Check for existing charge
existing = self.db.get_charge(req.order_id)
if existing:
    return ...  # already done

# 2. Charge the payment method (external side effect)
charge_id = self.payment.charge(...)

# 3. Record the charge in the DB
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, is killed, or the DB transaction fails **after step 2 but before step 3 completes**, the charge has been made but is never recorded. When the order is retried — by this worker or another — `get_charge` returns `None`, the idempotency check passes, and the customer is charged again.

The Redis lock does not protect against this. The lock has a 60-second TTL and is released in the `finally` block. A retry after a crash will acquire a new lock, see no DB record, and re-charge.

**Fix:** The payment gateway must support idempotency keys, and you must pass a stable, deterministic key (e.g., derived from `order_id`) on every attempt. This makes the charge operation itself idempotent at the gateway level, so a retry after a partial failure returns the original `charge_id` rather than creating a new charge. Without this, the at-most-once guarantee claimed in the docstring does not hold.

```python
# Example — pass a stable idempotency key to the gateway
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=f"order-{req.order_id}",  # stable, deterministic
)
```

### 2. Timeout Ambiguity in `PaymentClient` — Silent Charge With No `charge_id`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A `Timeout` from `requests` can fire **after the gateway has already processed the charge** — the request completed on the server side, but the response was not received in time. The current code treats this identically to a connection timeout and retries, which can result in a second charge being submitted with no way to correlate the two.

This is compounded by issue #1: if the retry also times out, you exhaust retries, raise `PaymentError`, record nothing, and the customer has been charged one or more times with no `charge_id` in your system.

**Fix:** Distinguish between connect timeouts and read timeouts. On a read timeout specifically, do not blindly retry — treat it as an unknown outcome and either surface it as a distinct error state or rely on the idempotency key (see issue #1) to make the retry safe.

```python
timeout=(5, 30)  # (connect_timeout, read_timeout) as a tuple
```

```python
except requests.exceptions.ReadTimeout:
    # Gateway may have processed this — do not retry without idempotency key
    raise PaymentError("Read timeout — charge outcome unknown") from e
except requests.exceptions.ConnectTimeout:
    last_error = "connection timeout"
    # Safe to retry — request never reached the gateway
```

## Moderate Issues

### 3. `PaymentError` Is Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # can raise PaymentError
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller gets a raw exception instead of an `OrderResult`. The PR claims structured error handling throughout, but this path is missing. Add a `except PaymentError` block and return an appropriate `OrderResult(processed=False, ...)`.

---

### 4. Lock TTL vs. Payment Retry Window

```
ORDER_LOCK_TTL = 60 seconds
MAX_RETRIES = 3
REQUEST_TIMEOUT = 30 seconds per attempt
RETRY_BACKOFF = 5 seconds between retries
```

Worst-case payment time: `3 × 30s + 2 × 5s = 100 seconds`. The lock TTL is 60 seconds. Under worst-case conditions the lock expires while a worker is still in the payment retry loop, another worker acquires the lock, passes the idempotency check (nothing recorded yet), and proceeds to charge. This is another path to a double charge.

The TTL needs to be longer than the maximum possible payment duration, or the payment timeout and retry parameters need to be reduced to fit within the TTL with margin. Pick one and document the constraint explicitly.

---

### 5. Lock Contention Retry Sleeps in a Synchronous Context

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

If this service runs under an async framework or in a thread pool with limited workers, blocking sleeps here will stall threads. This is not a correctness bug, but it is an operational concern. If the surrounding infrastructure is async, use `asyncio.sleep`. If it is threaded, document the blocking behavior explicitly so operators know to size the thread pool accordingly.

## Minor Issues

### 6. `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their output. If `payment_token` fails validation (e.g., too long), the raw token value may appear in logs. Sanitize or truncate the error message before logging, or log only field names and error types without values.

---

### 7. Test Coverage Gaps

The tests do not cover:
- `PaymentError` raised by `payment.charge()` — the unhandled exception path from issue #3
- DB transaction failure after a successful charge — the partial-failure path from issue #1
- Lock TTL expiry mid-processing
- `response.json()["charge_id"]` raising `KeyError` if the gateway returns a malformed response (no defensive access in `PaymentClient`)

These scenarios are exactly where the bugs above live. Tests should be added alongside the fixes.

---

## What Is Done Well

The Lua script for atomic lock release is correct and the token-verification approach is sound. Input validation with Pydantic and the `order_id` pattern constraint are good. The structured logging is consistent. The `SET NX EX` acquisition is the right primitive. The bones of this design are reasonable — the issues are in the failure paths, not the happy path.
