Decision: Request Changes

Summary: The implementation has solid foundational patterns — atomic SET NX EX acquisition, Lua-verified release, Pydantic validation, and idempotency intent are all correct in concept. However, there is a critical TOCTOU race that breaks the at-most-once guarantee, a charge-before-commit ordering flaw that can produce unrecoverable double charges, and several secondary issues that must be resolved before this handles real payment data.

---

## Critical Issues

### 1. Charge happens outside the database transaction — unrecoverable double-charge window

This is the most serious bug in the PR. The code charges the payment gateway *before* entering the database transaction, and the idempotency record is only written *inside* the transaction. If the process crashes, the database connection drops, or the transaction rolls back after `payment.charge()` succeeds, the charge is real and on the wire, but `db.get_charge()` will return `None` on the next attempt. The next worker will charge again.

```python
# CURRENT — charge is outside the transaction
charge_id = self.payment.charge(...)          # money leaves the customer here

with self.db.transaction():                   # if this fails, charge is orphaned
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The standard mitigation for this class of problem is to write a *pending* charge record to the database *before* calling the gateway, inside a transaction, so the idempotency check catches it on retry even if the process dies mid-flight. The flow should be:

```
BEGIN TRANSACTION
  INSERT charge_record (status=PENDING)
COMMIT

call payment.charge()   # if this crashes, the PENDING record is already visible

BEGIN TRANSACTION
  UPDATE charge_record (status=COMPLETE, charge_id=...)
  mark_order_complete()
COMMIT
```

On retry, `db.get_charge()` finds the PENDING record and the caller can decide to reconcile or reject rather than blindly re-charging.

---

### 2. TOCTOU race between idempotency check and charge — the lock does not fully protect you

The Redis lock serializes workers *at the Redis layer*, but it provides no durability guarantee. Redis is not a transactional store. Consider this sequence:

1. Worker A acquires the lock, passes the idempotency check (`get_charge` returns `None`).
2. Worker A calls `payment.charge()` — this takes up to 30 seconds (the gateway timeout).
3. The Redis lock TTL is 60 seconds. If the payment call takes longer than expected, or if there is clock skew, the lock can expire while the charge is in flight.
4. Worker B acquires the lock, passes the idempotency check (charge not yet recorded), and issues a second charge.

The lock TTL of 60 seconds and the payment timeout of 30 seconds (with up to 3 retries × 5-second backoff = 40+ seconds of sleep alone, not counting actual request time) means the total payment attempt window can exceed the lock TTL. This is a real scenario, not a theoretical one.

**Required fix:** The lock TTL must be set conservatively higher than the maximum possible payment client wall-clock time, *or* the payment client's total timeout budget must be strictly bounded below the lock TTL with margin. Currently `MAX_RETRIES=3`, `REQUEST_TIMEOUT=30`, `RETRY_BACKOFF=5` gives a worst-case of approximately `3×30 + 2×5 = 100 seconds` — well over the 60-second lock TTL.

Either reduce the payment client's total budget or implement lock extension (a watchdog that refreshes the TTL while the payment is in flight).

## Significant Issues

### 3. Unhandled `PaymentError` propagates through the lock context manager and is silently swallowed

`payment.charge()` raises `PaymentError` on permanent failure. This exception is not caught inside `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which is fine for lock release (the `finally` handles that), but the exception then bubbles up to the caller with no `OrderResult` returned. The caller gets an unhandled exception instead of a structured failure response.

```python
# PaymentError is not caught — caller receives an exception, not an OrderResult
charge_id = self.payment.charge(...)
```

Add an explicit `except PaymentError` block and return a structured `OrderResult(processed=False, ...)`.

### 4. Lock release silently ignores errors

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script call raises a `redis.RedisError` (network partition, Redis restart), the exception propagates out of `finally` and masks the original exception from the `try` block. Wrap the release in a try/except and log the failure rather than letting it escape.

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error("Lock token mismatch on release — lock may have expired: order_id=%s", order_id)
    except redis.RedisError:
        logger.error("Failed to release lock for order_id=%s", order_id, exc_info=True)
```

Also note: the current code does not check the return value of the Lua script. A return value of `0` means the token did not match (lock expired and was acquired by another worker). This should be logged as an error because it means the at-most-once guarantee may have already been violated.

### 5. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages can include field values in their output. If `payment_token` fails validation, its value (a payment credential) may appear in log output. Sanitize the error before logging or log only the field names, not the values.

## Minor Issues

### 6. Lock retry backoff is too short and not jittered

`backoff = 0.5 * (attempt + 1)` gives 0.5s, 1.0s, 1.5s. With multiple workers all retrying on the same schedule, they will collide repeatedly (thundering herd). Add random jitter: `backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.5)`.

### 7. Test coverage gaps

- No test for `PaymentError` being raised — this is the unhandled exception path identified above.
- No test for lock expiry during payment (TTL race).
- No test verifying that `record_charge` is *not* called when `payment.charge()` raises.
- The `test_lock_release_uses_lua_script` test asserts `_release_script` is called but does not assert it is called with the correct `lock_key` and token, which means the test would pass even if the wrong key were released.

### 8. `payment_token` field name collision

The `OrderRequest` model has a field named `payment_token`, and the `PaymentClient.charge()` parameter is also named `token`. This is fine internally, but the PR summary says "token-verified atomic lock operations via Lua script." The lock token (a UUID) and the payment token (a payment credential) are two entirely different things. The naming in the summary is ambiguous and could confuse future maintainers. Consider renaming the lock token variable to `lock_token` throughout.
