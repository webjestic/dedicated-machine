Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all directionally correct. However, there is a critical atomicity gap between the payment charge and the database commit that creates a real double-charge or silent-loss window, and several secondary issues compound the risk profile enough that this cannot be approved for a payment-critical path.

---

## Critical Issues

### 1. Charge-Before-Record Race: The At-Most-Once Guarantee Is Broken

This is the most serious flaw in the PR. The docstring claims an **at-most-once payment guarantee**, but the implementation does not deliver it.

```python
# Charge the payment method
charge_id = self.payment.charge(          # ← money leaves the customer here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():               # ← DB write happens AFTER
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Failure scenario — silent money loss / unrecorded charge:**

1. Worker A acquires the lock, passes the idempotency check (no prior charge), calls `payment.charge()` — the charge succeeds, money is captured.
2. Before `db.transaction()` commits, the worker process crashes (OOM kill, pod eviction, network partition, unhandled exception from `record_charge` itself).
3. The Redis lock TTL expires. Worker B acquires the lock, passes the idempotency check (still no DB record), and **charges the customer a second time**.

The idempotency check is gated on the DB record, but the DB record is written *after* the charge. These two operations are not atomic across the payment gateway and the database — they cannot be made so with this architecture — but the code makes no attempt to handle the gap.

**Minimum required mitigations (defense in depth, pick all that apply):**

- **Idempotency key on the payment gateway call.** Pass `order_id` as the gateway's idempotency key so that a retry of the same `order_id` returns the original `charge_id` rather than issuing a new charge. This is the single most important fix.
- **Outbox / two-phase record.** Write a `PENDING` charge record to the DB *before* calling the gateway, then update it to `COMPLETE` after. The idempotency check must detect `PENDING` records and either block or resume, not ignore them.
- **Compensating transaction on DB failure.** If `db.transaction()` raises after a successful charge, the code must attempt a gateway refund or void and surface the failure loudly — not silently drop the `charge_id`.

Currently, an unhandled exception from `db.record_charge` or `db.mark_order_complete` will propagate out of the `with self._acquire_lock` block, release the lock, and leave no trace that a charge occurred. The customer is charged; the system has no record of it.

### 2. Lock TTL vs. Payment Retry Budget: Timer Collision

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Worst-case payment client wall time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`.

The lock TTL is **60 seconds**. The payment client can run for **100 seconds**. If the gateway is slow, the lock expires while the charge is still in-flight. A second worker acquires the lock, passes the idempotency check (no DB record yet), and initiates a second charge — both potentially succeeding. This directly undermines the stated purpose of the lock.

**Fix:** Either reduce `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES-1) × RETRY_BACKOFF` to stay well under `ORDER_LOCK_TTL` with margin, or implement lock extension (Redlock-style `PEXPIRE` refresh) before the TTL elapses. The simpler path is to make the numbers coherent: e.g., `TTL=120`, `MAX_RETRIES=2`, `TIMEOUT=20`, `BACKOFF=5` → max 50s.

### 3. Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)   # raises PaymentError on permanent failure
    with self.db.transaction():
        ...
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of `_acquire_lock`, which will correctly release the lock — but the caller receives an unhandled exception rather than a structured `OrderResult`. The PR checklist claims structured error handling; this path is missing it.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

This must be added, and it must be distinct from the lock-contention path so callers can differentiate retriable from terminal failures.

## Significant Issues

### 4. `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials. The log line should be sanitized or the error should be reconstructed without field values.

---

### 5. Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep` blocks the worker thread. In an async or thread-pool deployment this is a concurrency hazard. Additionally, linear backoff with no jitter means all contending workers wake at the same time and re-collide. Standard practice is exponential backoff with jitter:

```python
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), max_backoff)
```

---

### 6. `customer_id` Has No Pattern Validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has no such constraint. If `customer_id` is used in DB queries or log correlation, this is an injection surface. Apply the same or equivalent pattern.

---

### 7. Test Suite Does Not Cover the Critical Failure Path

There is no test for:

- Payment succeeds → DB write fails → assert lock is released and `charge_id` is not silently dropped.
- Lock TTL expiry during payment (requires mocking time or TTL).
- `PaymentError` propagation (currently unhandled — test would expose the missing `except` block).
- `payment_token` not appearing in sanitized log output.

The existing tests cover the happy path and basic contention, but the dangerous edge cases — the ones that would cause a double charge or unrecorded charge in production — are entirely absent.

---

## Minor Notes

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants but not configurable at construction time. For a service that may run in environments with different Redis latency profiles, these should be injectable.
- The `CHARGE_ENDPOINT` hardcoded in `PaymentClient` should come from configuration or environment, not a class attribute, to avoid requiring a code change for environment promotion.
- `response.json()["charge_id"]` will raise `KeyError` if the gateway returns a 200 with an unexpected body. This should be wrapped with a descriptive `PaymentError`.
