Decision: Request Changes

Summary: The architecture demonstrates solid foundational intent — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all directionally correct. However, there is a critical TOCTOU race condition between the payment charge and the database transaction that breaks the at-most-once guarantee this PR claims to provide, and several secondary issues compound the risk profile enough that this cannot be approved in its current form.

---

## Critical Issues

### 1. Charge-Before-Record Race — Breaks At-Most-Once Guarantee

This is the most serious flaw in the design. The sequence inside the lock is:

```python
# 1. Idempotency check
existing = self.db.get_charge(req.order_id)

# 2. Charge the payment method  ← MONEY LEAVES HERE
charge_id = self.payment.charge(...)

# 3. Record the charge atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, is OOM-killed, loses its DB connection, or the DB transaction rolls back **after step 2 but before step 3 commits**, the customer has been charged and no record exists. The next worker that acquires the lock will pass the idempotency check (no charge found) and charge again. The distributed lock does not protect against this because the lock is gone — it expired or was released by the finally block before the crash was detected.

The canonical solution for at-most-once payment semantics is to **write intent before charging**:

```python
# Correct ordering
with self.db.transaction():
    self.db.insert_pending_charge(req.order_id, req.customer_id, req.amount_cents)
    # Commit intent first — crash here = no charge, no record, safe to retry

charge_id = self.payment.charge(...)  # Charge only after intent is durable

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also inspect the pending state so a retry after a crash between intent-commit and charge does not re-insert. The payment gateway itself should also be called with a deterministic idempotency key (e.g., `order_id`) if the gateway supports it — this is the only true defense against double-charges on network timeout retries.

### 2. Payment Timeout Retry Without Idempotency Key — Double-Charge Vector

In `payment_client.py`, the retry loop retries on `Timeout` exceptions:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request **may have succeeded** on the gateway side — the response was simply lost in transit. Retrying without a stable idempotency key passed to the gateway will issue a second charge for the same order. This is a well-known failure mode in payment systems.

```python
# Required: pass a deterministic key the gateway can use to deduplicate
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": order_id},
    headers={"Idempotency-Key": order_id},
    timeout=self.REQUEST_TIMEOUT,
)
```

The `order_id` must be threaded into `PaymentClient.charge()`. Currently the method signature is `charge(self, token, amount_cents)` — `order_id` is not passed at all.

## Significant Issues

### 3. Lock TTL Is Insufficient Under Retry Load

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire while the payment is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a concurrent charge. The TTL must exceed the maximum possible critical section duration with margin:

```python
ORDER_LOCK_TTL = 180  # Must exceed max payment client wall time
```

Or the payment client's worst-case time must be bounded below the TTL. Pick one, document the invariant explicitly, and enforce it with a test or assertion.

### 4. Lock Release in `finally` Swallows Exceptions Silently

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception is silently swallowed by the `finally` block if the `try` body already raised. The lock will eventually expire via TTL, but the caller gets no signal that release failed. At minimum, wrap this in a try/except and log at ERROR level. Consider whether a failed release should be surfaced to the caller in certain scenarios.

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error(
                "Lock release returned 0 — token mismatch or key expired order_id=%s",
                order_id,
            )
    except Exception:
        logger.exception("Failed to release order lock order_id=%s", order_id)
```

### 5. Lock Release Return Value Is Never Checked

The Lua script returns `0` when the token does not match (another worker holds the lock or the key expired). The current code discards this value entirely. A return of `0` after a successful charge but before DB commit is a meaningful signal — it means the TTL expired mid-operation, which is the exact scenario described in issue #3. This must be logged at ERROR level and likely trigger an alert.

### 6. `payment_token` Logged Indirectly — Verify It Never Reaches Log Sinks

`payment_token` is accepted as a field and passed to `payment.charge()`. Confirm at the integration layer that this value is never interpolated into any log message. The current code does not log it directly, but the `ValidationError` path logs `e` which may echo field values depending on Pydantic version and configuration. Consider redacting sensitive fields:

```python
class OrderRequest(BaseModel):
    payment_token: str = Field(..., min_length=1, max_length=256)

    model_config = {"json_schema_extra": {"sensitive": ["payment_token"]}}
```

Or use a custom `__repr__`/`__str__` that masks the token.

## Minor Issues

### 7. `LOCK_RETRY_ATTEMPTS = 3` With No Jitter — Thundering Herd

The retry backoff is deterministic: `0.5 * (attempt + 1)` = 0.5s, 1.0s, 1.5s. Under high concurrency all contending workers wake at the same instant and hammer Redis simultaneously. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

### 8. Test Coverage Gaps

The test suite does not cover:
- Lock TTL expiry mid-operation (the critical failure mode in issue #3)
- Payment timeout followed by successful DB write (the scenario where charge succeeded but timeout was raised)
- `_release_script` returning `0` (token mismatch)
- `db.transaction()` rollback after a successful charge (the core TOCTOU scenario)
- `PaymentError` propagation — currently unhandled in `process()`, will surface as an unhandled exception to the caller

This last point is worth calling out explicitly: `self.payment.charge()` can raise `PaymentError` and there is no `except PaymentError` block in `process()`. That exception will propagate out of the `with self._acquire_lock()` block, through the `finally` (releasing the lock), and up to the caller as an unhandled exception rather than a structured `OrderResult`. This is inconsistent with the method's documented contract.

| 4 | **High** | Handle and log `_release_script` exceptions in `finally` |
