Decision: Request Changes

Summary: The distributed locking design has a critical TOCTOU flaw — a successful payment charge occurs *outside* the database transaction, meaning a crash or exception between `payment.charge()` and `db.record_charge()` produces a real charge with no persisted record, violating the at-most-once guarantee the PR claims to provide. Additionally, the lock context manager silently swallows all exceptions from the processing body, and the payment client logs raw token values in error paths, which is a data exposure risk. These issues need to be resolved before this can merge.

---

## Detailed Findings

### 🔴 Critical — Charge/Record Race (TOCTOU, Data Integrity)

This is the most serious issue. The payment charge happens outside the database transaction:

```python
# Charge the payment method
charge_id = self.payment.charge(...)   # ← money leaves here

# Record the charge and mark the order complete atomically
with self.db.transaction():            # ← only then do we record it
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, the network drops, or any exception is raised between those two blocks, you have a charged customer with zero record in the database. The idempotency check (`db.get_charge`) will find nothing on retry, so the customer gets charged again. This directly contradicts the "at-most-once payment guarantee" stated in the docstring.

**The fix requires a two-phase approach.** Before charging, write a `PENDING` record to the database inside a transaction. After a successful charge, update that record to `COMPLETE` with the `charge_id`. On retry, detect the `PENDING` state and either confirm the charge status with the gateway or treat it as already-in-flight. This is a meaningful architecture change, not a one-liner.

### 🔴 Critical — Lock Context Manager Silently Swallows Exceptions

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    ...
    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Order lock released order_id=%s", order_id)
```

The `finally` block runs unconditionally, which is correct for lock release. However, the `process()` method only catches `LockError` — any other exception raised inside the `with self._acquire_lock(...)` block (e.g., `PaymentError`, a DB connection failure, an unexpected `KeyError` from `response.json()`) will propagate up to the caller completely unhandled and unlogged at the processor level. The caller receives an unhandled exception rather than a structured `OrderResult`. Either catch and handle these explicitly, or at minimum log them before re-raising.

```python
# What the caller sees today on PaymentError:
# Unhandled PaymentError: Payment failed after 3 attempts: gateway timeout
# Instead of: OrderResult(processed=False, message="Payment gateway unavailable")
```

### 🟡 High — Payment Token Logged in Error Path

In `payment_client.py`:

```python
except requests.exceptions.HTTPError as e:
    if e.response.status_code >= 500:
        ...
    else:
        raise PaymentError(f"Payment declined: {e.response.status_code}") from e
```

The `PaymentError` exception carries the original `HTTPError` as its cause via `from e`. Depending on how the requests library formats that exception and how your logging infrastructure captures unhandled exceptions, the raw request body — which contains `{"token": token, "amount": amount_cents}` — can appear in logs or error tracking systems. Payment tokens must be treated as secrets. Scrub or truncate them before they reach any log sink.

---

### 🟡 High — Lock TTL Is Dangerously Tight

```python
ORDER_LOCK_TTL = 60   # seconds
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment client execution time: `(30s timeout × 3 attempts) + (5s backoff × 2 gaps) = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock, pass the idempotency check (no record exists yet), and issue a second charge while the first worker is still waiting on attempt 2 or 3. The TTL must exceed the maximum possible execution time of everything inside the lock, with margin. Either increase `ORDER_LOCK_TTL` to at least 120–150 seconds, or reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` to fit within the budget. Document the math explicitly as a comment so this invariant doesn't silently break when someone tunes either value independently.

---

### 🟡 Medium — `response.json()["charge_id"]` Has No Error Handling

```python
response.raise_for_status()
charge_id = response.json()["charge_id"]
```

A malformed or unexpected response from the payment gateway (missing `charge_id` key, non-JSON body) raises a `KeyError` or `JSONDecodeError` that is not caught. This exception propagates out of the retry loop entirely, bypassing the retry logic and surfacing as an unhandled exception to the caller. Wrap this in explicit error handling and raise a `PaymentError` with a clear message.

---

### 🟢 Low — Lock Retry Backoff Is Linear, Not Exponential

The PR description says "retry with backoff" but the implementation is linear:

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

This is a minor naming issue rather than a correctness bug, but under high contention all workers will retry on the same cadence and continue to collide. True exponential backoff with jitter (`0.5 * 2^attempt + random jitter`) is the standard pattern for distributed lock contention and should be used here.

---

### 🟢 Low — Tests Do Not Cover the Charge/Record Race

There is no test asserting behavior when `payment.charge()` succeeds but `db.record_charge()` raises. Given that this is the central reliability claim of the service, it needs explicit test coverage. The current test suite would pass even if the transaction block were removed entirely.
