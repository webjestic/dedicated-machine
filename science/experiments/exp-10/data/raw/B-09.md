Decision: Request Changes

Summary: The implementation has sound architectural intent — atomic lock acquisition, token-verified Lua release, idempotency checks, and Pydantic validation are all correct patterns. However, there is a critical TOCTOU race condition between the payment charge and the database write that breaks the at-most-once guarantee the PR claims to provide, and several secondary issues compound the risk enough to block approval.

---

## Critical: Charge-Before-Record Creates an Unrecoverable Double-Charge Window

This is the most serious flaw. The code charges the payment gateway *outside* the database transaction:

```python
# Charge the payment method
charge_id = self.payment.charge(          # ← money leaves here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():               # ← only persisted here
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, the network drops, or the database transaction rolls back after `payment.charge()` returns but before `db.record_charge()` commits, the customer has been charged but no record exists. The next worker that acquires the lock will pass the idempotency check (`get_charge` returns `None`) and charge again. The distributed lock does not protect against this because the lock is gone by the time the retry happens.

**Required fix:** Persist a *pending* charge record inside the database transaction *before* calling the payment gateway, then mark it settled after the gateway confirms. This is the standard two-phase approach:

```python
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.settle_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also treat a *pending* record as "already in flight" so a retry worker does not re-charge while the first worker is mid-flight.

## High: Lock TTL Is Shorter Than the Maximum Payment Round-Trip

`ORDER_LOCK_TTL = 60` seconds, but the payment client can run for up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds**. The lock will expire while the payment is still in progress, allowing a second worker to acquire it, pass the idempotency check (no committed record yet), and issue a duplicate charge.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed the worst-case payment duration (≥ 120–150 s with margin), or implement lock renewal (a background thread or async task that extends the TTL while the critical section is still running).

---

## High: Unhandled `PaymentError` Propagates Through the Lock Context Manager

`payment.charge()` can raise `PaymentError`, which is not caught inside `process()`. It will propagate up through `_acquire_lock`'s `finally` block (which correctly releases the lock) and then escape `process()` entirely as an unhandled exception. The caller receives a raw exception rather than an `OrderResult`, violating the method's declared return contract.

```python
# Nothing catches PaymentError here
charge_id = self.payment.charge(...)
```

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult(processed=False, ...)`.

---

## Medium: `payment_token` Is Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's default `ValidationError` string representation includes field values in some versions. If `payment_token` fails validation, its value (or a truncated form) may appear in log output. Payment tokens must never appear in logs.

**Required fix:** Log only `e.errors()` filtered to field names, or construct a safe message that excludes the `payment_token` field value entirely.

---

## Medium: Retry Backoff in `_acquire_lock` Holds the Calling Thread

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

With three attempts and linear backoff, a worker can block for 1.5 seconds on lock contention before raising `LockError`. In a synchronous worker pool this is acceptable, but if this code is ever called from an async context (FastAPI, etc.) it will block the event loop. The backoff values should be documented as a known constraint, or the method should be made async-aware.

---

## Low: Test Suite Does Not Cover the Crash-Between-Charge-and-Record Scenario

There is no test that simulates `db.transaction().__exit__` raising an exception after `payment.charge()` succeeds. This is precisely the failure mode that causes double-charges, and it should be the most important test in the suite. Adding it now would also make the critical bug above immediately visible as a failing test.

---

## Low: `PaymentClient` Has No Instance-Level Configuration

`CHARGE_ENDPOINT`, `REQUEST_TIMEOUT`, `MAX_RETRIES`, and `RETRY_BACKOFF` are class-level constants. This makes it impossible to inject different values in tests without monkeypatching the class, and it prevents per-environment configuration (staging vs. production endpoints). These should be constructor parameters with sensible defaults.
