Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide: a charge can succeed while the subsequent database transaction fails, resulting in a real charge with no recorded evidence of it. Additionally, several secondary issues around exception handling, lock safety, and test coverage need to be addressed before this is production-ready.

---

## Critical Issue: Charge-Before-Record Creates an Unrecoverable Ghost Charge

This is the most serious bug in the PR. The current execution order is:

```python
# 1. Charge the payment method
charge_id = self.payment.charge(...)

# 2. Record the charge atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `self.payment.charge()` succeeds and returns a `charge_id`, but the database transaction fails (network partition, DB crash, constraint violation, anything), the customer has been charged and there is **no record of it anywhere**. The next retry will pass the idempotency check (`get_charge` returns `None`), acquire the lock, and charge the customer again. This directly contradicts the "at-most-once payment guarantee" stated in the docstring.

**Required fix:** Use a two-phase approach. Write a pending/in-flight charge record to the database *before* calling the payment gateway. The record should store enough state to detect and reconcile an ambiguous outcome. After the charge succeeds, update the record to confirmed. This pattern is sometimes called an "outbox" or "pending intent" record.

```python
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also be updated to treat a `pending` record as "do not charge again" — a pending record means a charge may already be in flight or completed, and the correct response is to reconcile, not to re-charge.

## Secondary Issue: Unhandled `PaymentError` Escapes the Lock Context

`self.payment.charge()` raises `PaymentError` on permanent failure. This exception is not caught inside `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which is fine for lock release (the `finally` clause handles that), but the caller receives an unhandled exception rather than an `OrderResult`. The PR's own return type annotation promises `OrderResult` in all cases.

```python
# PaymentError is not caught — caller gets a raw exception
charge_id = self.payment.charge(...)
```

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult`.

```python
from services.payment_client import PaymentError

try:
    charge_id = self.payment.charge(...)
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Secondary Issue: Lock TTL Is Insufficient Under Retry Conditions

`ORDER_LOCK_TTL` is 60 seconds. `PaymentClient` has `MAX_RETRIES = 3`, `REQUEST_TIMEOUT = 30` seconds per attempt, and `RETRY_BACKOFF = 5` seconds between retries. Worst-case payment duration is `(3 × 30) + (2 × 5) = 100 seconds`. The lock expires before the payment completes, allowing a second worker to acquire the lock, pass the idempotency check (no charge recorded yet), and issue a duplicate charge. The lock TTL must exceed the maximum possible payment client duration with margin.

**Required fix:** Either increase `ORDER_LOCK_TTL` to at minimum 120 seconds (preferably with a comment explaining the derivation), or reduce `PaymentClient` retry parameters so the math holds, or implement lock extension (GETEX) before the payment call.

---

## Secondary Issue: `payment_token` Is Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation, its value may appear in the log output. Payment tokens are sensitive credentials and must never appear in logs.

**Required fix:** Either sanitize the error before logging, or log only field names without values for payment-related fields.

```python
except ValidationError as e:
    safe_fields = [err["loc"] for err in e.errors()]
    logger.warning("Invalid order request fields: %s", safe_fields)
```

---

## Secondary Issue: Database Exceptions Are Not Caught

Neither `self.db.get_charge()` nor the `self.db.transaction()` block have exception handling. A transient database error will propagate as an unhandled exception. The lock will still be released (the `finally` clause is correct), but the caller receives a raw exception instead of an `OrderResult`, and no log entry is written at the `process()` level.

**Required fix:** Wrap database operations in a `try/except` that catches the appropriate DB exception types and returns a structured `OrderResult` with a safe error message.

---

## Secondary Issue: Test Coverage Gaps

The test suite does not cover:

1. **`PaymentError` propagation** — there is no test verifying behavior when `payment.charge()` raises. Given this is an unhandled exception path, this gap is significant.
2. **Database transaction failure** — no test covers the ghost charge scenario described above. Once the two-phase fix is applied, tests must verify that a pending record prevents re-charge.
3. **Lock TTL expiry during payment** — the race condition where the lock expires mid-payment is not tested.
4. **`payment_token` does not appear in logs** — given the logging concern above, a test asserting log output does not contain the token value would be appropriate.

## Minor Notes

- The `LOCK_RETRY_ATTEMPTS = 3` with backoff of `0.5 * (attempt + 1)` means a maximum wait of `0.5 + 1.0 + 1.5 = 3 seconds` before giving up. This is reasonable for interactive flows but should be documented as a deliberate choice, since it means under sustained contention the order will be rejected rather than queued.
- `requests.exceptions.ConnectionError` is not caught in `PaymentClient.charge()`. A DNS failure or TCP reset will propagate as an unhandled exception rather than triggering the retry logic. Add it alongside `Timeout`.
- The Lua script and atomic `SET NX EX` are correctly implemented. No issues there.
