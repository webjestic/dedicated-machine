Decision: Request Changes

Summary: The implementation has a critical architectural flaw that directly undermines the stated at-most-once payment guarantee: the payment gateway charge occurs *outside* the database transaction, creating a window where a charge can succeed but the database record fails, leaving the system in an inconsistent state with no recovery path. Additionally, the lock TTL of 60 seconds is dangerously short relative to the cumulative payment retry window (up to 75 seconds: 3 attempts × 30s timeout + 2 × 5s backoff), meaning the lock can expire and be acquired by a competing worker while a payment attempt is still in flight. These two issues together can produce exactly the double-charge scenario the service is designed to prevent.

---

## Detailed Findings

### 1. [Critical] Charge-then-record is not atomic — no recovery on DB failure

```python
# services/order_processor.py, lines ~90–100

charge_id = self.payment.charge(          # ← money leaves the customer here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

with self.db.transaction():               # ← if this fails, charge is orphaned
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment is executed before the database transaction opens. If `db.transaction()`, `record_charge()`, or `mark_order_complete()` raises any exception — network partition, deadlock, disk full, application crash — the charge has already been submitted to the gateway. On the next retry the idempotency check (`db.get_charge`) will find nothing and charge again.

**Required fix:** The gateway must support idempotency keys. Pass a deterministic idempotency key (e.g., `order_id`) with every charge request. On retry, the gateway will return the original `charge_id` rather than creating a new charge. This is the standard pattern for at-most-once payment semantics and is the only correct solution here; you cannot make a network call and a database write atomic without it.

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway deduplicates on this
)
```

### 2. [Critical] Lock TTL shorter than maximum payment retry window

```python
# services/order_processor.py
ORDER_LOCK_TTL = 60        # seconds

# services/payment_client.py
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment duration: `3 × 30s + 2 × 5s = 100 seconds`. The lock expires after 60 seconds. A second worker can acquire the lock, pass the idempotency check (nothing recorded yet), and issue a second charge while the first worker's third retry is still pending.

**Required fix:** Either extend `ORDER_LOCK_TTL` to comfortably exceed the maximum possible payment duration with margin (e.g., 150s), or implement lock extension (Redlock-style `pexpire` refresh) during the payment call. The payment client's timeout and retry budget must be explicitly bounded and documented alongside the lock TTL so they are maintained together.

---

### 3. [Moderate] `PaymentError` is silently swallowed in `process()`

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed ...")
```

`PaymentError` and any other exception from `self.payment.charge()` or the DB layer propagate uncaught out of `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. This is inconsistent with the method's documented contract and will surface as a 500 in any HTTP layer wrapping this.

**Required fix:** Add explicit handling for `PaymentError` and a catch-all for unexpected exceptions:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 4. [Moderate] Lock retry backoff is linear and blocks the worker thread

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
    time.sleep(backoff)
```

`time.sleep()` blocks the thread entirely. In any async or thread-pool worker environment this degrades throughput under contention. The backoff also has no jitter, so multiple workers retrying simultaneously will collide on the same intervals (thundering herd).

**Required fix:** Add random jitter to the backoff. If the service is async, use `asyncio.sleep`. Consider whether 3 attempts with 0.5–1.5s waits is actually the right policy for a 60-second lock — a locked order likely won't be released in 1.5 seconds.

---

### 5. [Minor] `payment_token` logged indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its (possibly partial) value may appear in log output. Payment tokens must never appear in logs.

**Required fix:** Sanitize or filter the validation error before logging, or log only field names and error types, not values.

---

### 6. [Minor] Test suite does not cover the charge-succeeds/DB-fails scenario

Given finding #1, the most important failure mode has no test. The test suite should include a case where `payment.charge()` returns successfully but `db.record_charge()` raises, and assert that the system behaves predictably (e.g., the charge is recoverable via idempotency key on retry).
