Decision: Request Changes

Summary: The core locking and idempotency architecture has a critical TOCTOU race condition: the payment charge happens *outside* the database transaction, meaning a successful charge can occur without a corresponding record if the DB write fails — resulting in a real-money charge with no audit trail and no idempotency protection on retry. There are also meaningful gaps in error handling, observability, and test coverage that need to be addressed before this touches production. Detailed findings below.

---

## Critical Issues

### 1. Charge-Before-Record Race (Data Loss / Double-Charge Risk)

This is the most serious flaw in the PR.

```python
# Charge the payment method
charge_id = self.payment.charge(...)          # ← money leaves the customer

# Record the charge and mark the order complete atomically
with self.db.transaction():
    self.db.record_charge(...)                # ← what if this throws?
    self.db.mark_order_complete(...)
```

If `self.db.transaction()` or anything inside it raises an exception, the charge has already been collected but `record_charge` never commits. On the next retry:

- `get_charge()` returns `None` (no record exists)
- The idempotency check passes
- The customer is charged a second time

The PR claims "at-most-once payment guarantee" in the docstring. That claim is false under this failure mode. The correct pattern is to record a *pending* charge intent before calling the gateway, then confirm or void it after. Alternatively, the payment gateway's own idempotency key (keyed to `order_id`) must be used so retries are safe — but that requires the payment client to accept and forward an idempotency key, which it currently does not.

**Required fix:** Pass `order_id` as an idempotency key to the payment gateway, and/or persist a `PENDING` charge record before calling `payment.charge()` so that a crash mid-flight is recoverable.

---

### 2. Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)   # can raise PaymentError
    with self.db.transaction():
        ...
```

`PaymentError` is not caught in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which is fine for lock release (the `finally` handles that), but the caller receives an unhandled exception rather than an `OrderResult`. The PR's own return type annotation says this method returns `OrderResult`. This is a broken contract and will cause 500-level errors at the call site.

```python
except LockError:
    return OrderResult(...)
# PaymentError, DB errors, network errors → all uncaught
```

**Required fix:** Catch `PaymentError` and DB exceptions explicitly and return structured `OrderResult` failures. Do not let implementation exceptions leak through a public interface that promises a value type.

---

### 3. Lock TTL Is Dangerously Tight Given Retry Window

```python
ORDER_LOCK_TTL = 60   # seconds

# PaymentClient
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment client runtime: `(30s timeout + 5s backoff) × 3 attempts = 105 seconds`. The lock TTL is 60 seconds. A slow-but-legitimate payment attempt will cause the lock to expire mid-flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime (≥ 120s with margin), or implement lock extension (a background heartbeat that refreshes the TTL while the payment is in-flight). The values across these two files must be designed together, not independently.

## Significant Issues

### 4. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some configurations. If `payment_token` fails validation (e.g., too long), the raw token value may appear in logs. Payment tokens are sensitive credentials. Validate and log field names only, never values.

**Required fix:** Sanitize or suppress field values from validation error log output. Log only which fields failed, not their contents.

---

### 5. No Timeout on Redis Operations

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If the Redis connection hangs, this call blocks indefinitely. There is no socket timeout configured on the client, and no timeout passed to the operation. In a distributed system, a hung Redis connection will stall a worker thread permanently.

**Required fix:** Ensure `redis.Redis` is instantiated with `socket_timeout` and `socket_connect_timeout`. Document this as a construction requirement or enforce it in `__init__`.

---

### 6. `requests.post` Has No Authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

The internal payment gateway endpoint receives no authentication header, API key, or mTLS configuration. Even on an internal network, unauthenticated calls to a payment endpoint are a security gap — especially relevant given this firm's compliance posture.

**Required fix:** Accept credentials in `PaymentClient.__init__` and attach them to every request (Bearer token, API key header, or session with client cert).

---

### 7. Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoding infrastructure URLs as class-level constants makes environment promotion (dev → staging → prod) impossible without code changes. This is a configuration management violation.

**Required fix:** Accept endpoint URL via constructor parameter or environment variable. Remove the hardcoded constant.

---

## Test Coverage Gaps

### 8. No Test for the Critical Failure Path (Charge Succeeds, DB Fails)

There is zero test coverage for the scenario identified in finding #1: payment succeeds, DB write throws. This is the most important test case for a payment processor and it's absent.

### 9. `PaymentError` Path Is Untested

No test verifies behavior when `payment.charge()` raises `PaymentError`. Given that the exception is currently unhandled (finding #2), this gap allowed the bug to ship undetected.

### 10. Lock TTL Expiry During Payment Is Untested

No test simulates the lock expiring before the payment completes. This is directly related to finding #3.

### 11. Retry Backoff in `_acquire_lock` Uses `time.sleep` Without Mocking

```python
time.sleep(backoff)
```

The lock contention test (`test_process_lock_contention`) will actually sleep for `0.5 + 1.0 + 1.5 = 3 seconds` in CI. Patch `time.sleep` in tests.

---

## Minor Notes

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as class-level constants are fine for now, but should eventually be constructor parameters to support environment-specific tuning without subclassing.
- The Lua script for atomic release is correct. Token-verified release is the right approach.
- `SET NX EX` for atomic acquisition is correct.
- Pydantic model structure and `pattern=` enforcement on `order_id` is good.
- Structured logging is consistent and appropriate.
