Decision: Request Changes

Summary: The core locking logic contains a critical TOCTOU race condition: the payment gateway is charged *outside* the database transaction, meaning a successful charge can occur without a corresponding record if the DB transaction fails — resulting in a real money movement with no audit trail and no idempotency key to prevent a retry from double-charging. Additionally, the lock TTL is dangerously short relative to the combined payment retry window, meaning the lock can expire mid-flight and allow a second worker to enter the critical section while the first is still executing. Several secondary issues compound these problems and are detailed below.

---

## Critical Issues

### 1. Charge-Before-Record: Money Movement Outside the Transaction

```python
# Charge the payment method
charge_id = self.payment.charge(          # ← real money moves here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():               # ← DB write happens here, separately
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious bug in the PR. The payment gateway is called before the database transaction opens. If anything fails between the `charge()` call and the successful commit of `record_charge` — a DB connection drop, an exception, a process crash, a network partition — the money has moved but no record exists. On the next retry, `get_charge()` returns `None`, the idempotency check passes, and the customer is charged again.

The payment gateway call cannot be made atomic with the DB write, but the standard mitigation is a two-phase approach: persist a `PENDING` charge record with the `charge_id` *before* calling the gateway, then update it to `COMPLETE` after. This gives you a recoverable audit trail and a durable idempotency key regardless of where the failure occurs. As written, the idempotency check is only reliable if nothing ever crashes between those two lines.

---

### 2. Lock TTL Expires Before Payment Retries Complete

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
# payment_client.py
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment execution time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock at second 61 while the first worker is still waiting on attempt 3 of the payment call. The idempotency check will pass for the second worker because no charge has been recorded yet, and both workers will call `payment.charge()` concurrently.

The TTL must be set to a value that is strictly greater than the maximum possible execution time of the critical section, with margin. At minimum: `(MAX_RETRIES × REQUEST_TIMEOUT) + ((MAX_RETRIES - 1) × RETRY_BACKOFF) + DB_overhead + safety_margin`. With the current payment client settings that floor is ~110–120 seconds. The TTL and payment client timeout budget need to be derived from a shared constant, not set independently.

## Significant Issues

### 3. Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)   # raises PaymentError
    ...
```

`PaymentError` is not caught inside `process()`. It will propagate out of the `with` block, which will correctly release the lock via `finally`, but the caller receives an unhandled exception rather than an `OrderResult`. The method signature promises to return `OrderResult` in all cases. This should be caught and returned as a structured failure result.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

### 4. Lock Release Silently Swallows Errors

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection failure, script error), the exception is suppressed by the `finally` block and the log line claiming the lock was released will never execute — but the caller won't know the release failed either. The lock will then hold until TTL expiry. This is acceptable behavior for a distributed lock, but the error should be explicitly caught and logged as a warning, not silently lost.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Order lock released order_id=%s", order_id)
    except Exception as e:
        logger.warning("Failed to release lock order_id=%s: %s — will expire via TTL", order_id, e)
```

---

### 5. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in the log output. Payment tokens are sensitive credentials. Validation error logging should either redact field values or log only field names and error types.

---

### 6. `payment_token` Appears in Plain Text in Structured Logs

The `payment_token` field flows through the system without any masking. If the `OrderRequest` object or its fields are ever serialized to logs (e.g., during a debug session, an unhandled exception traceback, or a future logging enhancement), the token is exposed. Consider storing a masked or hashed representation internally after validation.

## Minor Issues

### 7. Lock Retry Backoff Is Linear, Not Exponential

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

The PR description says "retry with backoff" but this is linear. Under high contention with many workers, linear backoff produces thundering herd behavior. Standard practice is exponential backoff with jitter: `min(cap, base * 2**attempt) + random_jitter`. This is a minor issue but worth correcting before this pattern is copied elsewhere.

---

### 8. Test Coverage Gaps

The test suite does not cover:

- **Lock expiry during payment**: No test verifies behavior when the lock TTL expires mid-execution.
- **`PaymentError` propagation**: No test for what happens when `payment.charge()` raises — this would currently surface as an unhandled exception (see issue 3).
- **DB transaction failure after successful charge**: The most dangerous failure mode has no test.
- **Lock release failure**: No test for Redis unavailability during lock release.
- **`payment_token` in validation error logs**: No test that sensitive fields are not logged.

The existing tests pass because the mocks are wired for the happy path and the lock contention path. They do not exercise the failure modes that matter most for a payment system.

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Charge-before-record: persist `PENDING` state before calling gateway |
| 2 | Critical | Lock TTL shorter than maximum payment retry window |
| 3 | High | `PaymentError` unhandled — breaks return type contract |
| 4 | Medium | Lock release errors silently swallowed |
| 5 | Medium | Sensitive token value may appear in validation error logs |
| 6 | Medium | `payment_token` unmasked throughout internal flow |
| 7 | Low | Linear backoff should be exponential with jitter |
| 8 | Low | Test coverage missing for critical failure modes |

Issues 1 and 2 must be resolved before this can be approved. A payment processor that can double-charge customers and whose distributed lock can expire during the operation it is meant to protect does not meet the correctness bar required for this system.
