Decision: Request Changes

Summary: The implementation demonstrates solid foundational intent — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all directionally correct. However, there is a critical charge-before-commit race condition that directly violates the at-most-once guarantee this PR claims to provide, compounded by several secondary issues that collectively make this unsafe for production deployment in any payment-adjacent context.

---

## Critical Issues

### 1. Charge Fires Outside the Database Transaction — At-Most-Once Guarantee is Broken

This is the most severe defect in the PR. The docstring explicitly claims an *at-most-once payment guarantee*, but the implementation does not deliver it.

```python
# Charge the payment method  ← happens OUTSIDE the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():           # ← DB write happens AFTER charge
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**The failure scenario is straightforward:**

1. Worker A acquires the lock, passes the idempotency check (no existing charge), and calls `payment.charge()` — the customer's card is debited.
2. The process crashes, the DB transaction never commits, or the DB connection drops before `record_charge` executes.
3. On retry, the idempotency check finds no recorded charge and issues a second charge.

The customer is double-charged. This is not a theoretical edge case — it is a well-known failure mode in distributed payment systems and the exact problem this PR is supposed to prevent.

**The correct pattern is to record intent before charging:**

```python
with self.db.transaction():
    # Write a pending/in-flight record FIRST, inside the transaction.
    # If the process dies here, no charge was issued. Safe to retry.
    self.db.record_charge_pending(
        req.order_id, req.customer_id, req.amount_cents
    )

# Charge only after the pending record is durable.
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Update the record to confirmed/complete.
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This still requires idempotency handling for the case where the charge succeeds but the confirmation write fails, but it eliminates the silent double-charge path.

### 2. `PaymentError` is Silently Swallowed in `process()`

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        charge_id = self.payment.charge(...)  # can raise PaymentError
        ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` is not caught. It propagates out of `process()` as an unhandled exception. The caller receives no `OrderResult` — they get a raw exception. This means:

- The caller cannot distinguish a payment decline from a bug.
- The lock is released (correctly, via `finally`), but the order is left in an indeterminate state with no structured response.
- Any upstream handler that expects an `OrderResult` will fail.

`PaymentError` must be caught explicitly and returned as a structured `OrderResult`.

```python
except PaymentError as e:
    logger.error("Payment failed for order_id=%s: %s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

### 3. Lock TTL is Dangerously Short Relative to Payment Retry Window

```python
ORDER_LOCK_TTL = 60   # seconds
```

```python
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
REQUEST_TIMEOUT = 30  # seconds per attempt
```

Worst-case payment client execution time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock expires at 60 seconds. A second worker can acquire the lock while the first is still mid-charge on attempt 3. The idempotency check passes (no committed charge yet), and a second charge is issued. This is another double-charge path, independent of issue #1.

`ORDER_LOCK_TTL` must be set to a value strictly greater than the maximum possible payment client wall time. Given the current payment client parameters, the minimum safe TTL is approximately 120–150 seconds, with margin. Alternatively, the payment client's total retry budget must be constrained to fit within the TTL.

## Secondary Issues

### 4. Lock Release Silently Fails on Redis Errors

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script call raises a `redis.RedisError` (connection drop, cluster failover), the exception propagates out of the `finally` block, potentially masking the original exception from the `try` body. The lock also remains held until TTL expiry. This should be wrapped:

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error(
                "Lock release returned 0 — token mismatch or key expired order_id=%s",
                order_id,
            )
    except redis.RedisError:
        logger.error(
            "Failed to release lock for order_id=%s — will expire at TTL",
            order_id, exc_info=True,
        )
```

The return value of the Lua script (0 vs 1) is also never checked. A return of 0 means the token did not match or the key had already expired — both are operationally significant and should be logged as errors, not silently ignored.

### 5. `payment_token` Logged Indirectly via `payment_client.py` — Verify No Token Leakage

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

The `token` field is a raw payment method token. Confirm that the logging configuration does not capture request bodies (e.g., via a `requests` debug hook or middleware). If any exception path serializes `response.request.body` into logs, a payment token ends up in plaintext log storage. This is a PCI-DSS concern. Explicit confirmation or a scrubbing mechanism is required before this goes near a production log aggregator.

---

### 6. Test Suite Does Not Cover the Critical Failure Paths

The tests validate the happy path and lock contention, but they do not cover:

- **Crash between charge and DB commit** — the core scenario this PR is supposed to prevent.
- **`PaymentError` raised during `payment.charge()`** — currently causes an unhandled exception; no test catches this.
- **Lock TTL expiry mid-charge** — no test simulates a second worker acquiring the lock while the first is still processing.
- **Lua script returning 0** — no test verifies the behavior when the release token does not match.
- **Redis connection failure during lock release** — no test verifies the `finally` block's error handling.

Tests that only cover the happy path and one contention case are insufficient for a payment-critical service. The failure modes are where the bugs live.

---

### 7. `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has a strict alphanumeric pattern enforced. `customer_id` does not. If `customer_id` is used in any downstream query construction, logging format string, or external API call, the absence of sanitization is a latent injection surface. Apply the same or equivalent pattern constraint.

|---|----------|-------|
| 1 | **Critical** | Charge executes outside DB transaction — double-charge on crash |
| 2 | **Critical** | `PaymentError` unhandled in `process()` — unstructured exception escape |
| 3 | **Critical** | Lock TTL (60s) shorter than max payment retry window (100s) |
| 4 | Medium | Lock release errors silently swallowed; Lua return value unchecked |
| 5 | Medium | Payment token leakage risk via request body in exception logs |
| 6 | Medium | Test suite does not cover any of the critical failure paths |

The architectural intent is sound and the locking primitives are correctly chosen. The implementation is not ready for merge in its current state. Address the three critical items — particularly the charge/commit ordering — and resubmit.
