Decision: Request Changes

Summary: The architecture demonstrates solid instincts — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical TOCTOU race condition between the payment charge and the database transaction that breaks the at-most-once guarantee this PR claims to provide, along with several additional issues of varying severity that must be addressed before this touches production.

---

## Critical Issues

### 1. Charge-Before-Record Race Condition — Breaks At-Most-Once Guarantee

This is the most dangerous flaw in the PR. The sequence inside the lock is:

```python
# 1. Idempotency check
existing = self.db.get_charge(req.order_id)
if existing:
    ...

# 2. Charge the payment method  ← MONEY LEAVES HERE
charge_id = self.payment.charge(...)

# 3. Record the charge atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, is killed, loses network connectivity, or the database transaction fails **after step 2 but before step 3 commits**, the customer has been charged but no record exists. On any retry — by this worker or another — the idempotency check at step 1 finds nothing and charges again. The distributed lock provides no protection here because the lock is gone after the crash. This is a classic phantom charge scenario.

**Required fix:** The payment gateway must support idempotency keys, and the charge record must be written to the database *before or atomically with* the charge being considered durable. A common correct pattern:

```python
with self.db.transaction():
    # Reserve the slot — unique constraint on order_id prevents double-entry
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Charge with an idempotency key derived from order_id
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=f"order-{req.order_id}",  # gateway deduplicates on retry
)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This requires the payment gateway to support idempotency keys (most production gateways — Stripe, Braintree, etc. — do). If it does not, that is itself a blocking infrastructure concern.

### 2. `PaymentError` Is Silently Swallowed Inside the Lock Context

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    ...
```

`PaymentError` is never caught in `process()`. If the payment client raises `PaymentError` after exhausting retries, the exception propagates out of the `with self._acquire_lock(...)` block, which correctly releases the lock via `finally`, but the exception then propagates to the caller completely unhandled — no `OrderResult` is returned, the caller receives a raw exception, and any upstream HTTP handler will likely produce an unstructured 500.

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult`:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

### 3. Timeout Retry in `PaymentClient` Is Unsafe Without Idempotency Keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge did not happen. The request may have reached the gateway and been processed; the response simply did not return in time. Retrying without an idempotency key passed to the gateway can result in multiple successful charges for the same payment token. This compounds the issue raised in finding #1.

**Required fix:** Pass a stable idempotency key (derived from `order_id`, not generated per-attempt) in the request headers or body on every attempt. Do not retry blindly on timeout without it.

## Significant Issues

### 4. Lock TTL Is Dangerously Mismatched With Payment Retry Window

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client execution time: `(30 × 3) + (5 × 2) = 100 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock while the first worker is still inside it executing a payment retry. The idempotency check will pass (no record exists yet), and a second charge will be initiated. The distributed lock has been rendered ineffective by this miscalculation.

**Required fix:** `ORDER_LOCK_TTL` must exceed the maximum possible execution time of everything inside the lock, with margin. Set it to at least 120–150 seconds, or implement lock extension (Redlock-style keepalive). Document the relationship explicitly so future changes to either constant are not made in isolation.

---

### 5. Redis Is a Single Point of Failure With No Failover Consideration

The PR uses a plain `redis.Redis` client with no mention of Sentinel, Cluster, or Redlock across multiple independent Redis nodes. If the Redis instance becomes unavailable:

- `self.redis.set(...)` raises `redis.exceptions.ConnectionError`
- This propagates uncaught through `_acquire_lock` and then through `process()`
- No `OrderResult` is returned; the caller receives a raw exception

More subtly: if Redis fails *after* lock acquisition but *before* the lock is released, the lock will expire via TTL (the one correct behavior here), but any exception from the release script is also unhandled.

**Required fix at minimum:** Catch `redis.exceptions.RedisError` in `process()` and return a structured failure result. Document explicitly that this implementation provides lock *best-effort* under Redis failure, not a hard guarantee, and that Redlock or Redis Cluster should be evaluated for higher availability requirements.

---

### 6. `payment_token` Is Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation in some configurations and versions. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in logs. Payment tokens are sensitive credentials.

**Required fix:** Do not log the raw `ValidationError` object. Log only the field names and error types:

```python
logger.warning(
    "Invalid order request order_id=%s errors=%s",
    order_id,
    [{"field": e["loc"], "type": e["type"]} for e in exc.errors()],
)
```

## Minor Issues

### 7. Lock Retry Backoff Is Linear and Contention-Blind

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Linear backoff with no jitter means multiple contending workers will retry in near-lockstep, creating a thundering herd on the Redis key at predictable intervals. Use exponential backoff with jitter:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

---

### 8. Test Coverage Has Meaningful Gaps

The test suite does not cover:

- **`PaymentError` propagation** — no test verifies that a payment failure returns a structured `OrderResult` (because it currently does not; it raises)
- **Lock TTL expiry during payment** — no test for the window where the lock expires mid-execution
- **Redis connection failure** — no test for `RedisError` during lock acquisition or release
- **`payment_token` not appearing in logs** — no assertion that sensitive fields are scrubbed from log output
- **Timeout retry behavior** — no test verifying that a timeout triggers a retry rather than immediate failure

These gaps are not blocking on their own, but they will need to be addressed alongside the fixes above, since several of them exist precisely because the code paths they would test are currently broken.

---

### 9. `CHARGE_ENDPOINT` Hardcoded as a Class Attribute

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoint URLs in source code make environment promotion (dev → staging → prod) fragile and require code changes to redirect traffic. This should be injected via configuration or environment variable at construction time.
