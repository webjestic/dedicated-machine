Decision: Request Changes

Summary: The distributed locking logic contains a critical TOCTOU race condition — the payment charge occurs *outside* the database transaction, meaning a successful charge can be orphaned if the subsequent `record_charge` or `mark_order_complete` fails, resulting in a customer being billed with no record in the system. Additionally, the lock context manager silently swallows all exceptions from within the `with` block (anything other than `LockError`), so payment failures, database errors, and unexpected exceptions are never surfaced to the caller or logged. The test suite also has meaningful gaps that would have caught both of these issues.

---

## Detailed Findings

### 🔴 Critical — Charge-Before-Record Creates Orphaned Payments

```python
# services/order_processor.py

# Charge the payment method
charge_id = self.payment.charge(        # ← money leaves the customer here
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():             # ← if this fails, charge is lost forever
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious issue. If `self.db.transaction()`, `record_charge`, or `mark_order_complete` raises *any* exception after the payment gateway has already confirmed the charge, the customer has been billed but the system has no record of it. The next call will pass the idempotency check (`get_charge` returns `None`) and charge them again.

The standard pattern for this class of problem is to record a *pending* charge intent before calling the gateway, then confirm or void it after. At minimum, the charge ID must be durably persisted before the lock is released, with a compensating refund issued on DB failure.

```python
# Safer ordering sketch
with self.db.transaction():
    self.db.create_pending_order(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

try:
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Attempt compensating refund; alert ops if refund also fails
    self.payment.refund(charge_id)
    raise
```

### 🔴 Critical — Silent Exception Swallowing in Lock Context Manager

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    ...
    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

The `finally` block runs unconditionally, which is correct for lock release. The problem is in `process()`:

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        charge_id = self.payment.charge(...)   # raises PaymentError
        with self.db.transaction():            # raises DB exception
            ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

The outer `except` only catches `LockError`. Any `PaymentError`, `DatabaseError`, `requests.Timeout`, or unexpected exception propagates out of `process()` as an unhandled exception. The caller gets a raw exception instead of an `OrderResult`, breaking the contract the method signature promises. This will cause 500 errors in any HTTP layer wrapping this service.

```python
# Fix: add a broad catch with logging
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

### 🟡 Medium — Payment Client Timeout Is Dangerously Long Inside a Held Lock

```python
# services/payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment client execution time: `(30 × 3) + (5 × 2) = 100 seconds`. The Redis lock TTL is 60 seconds. A slow-but-not-failing gateway call will cause the lock to expire mid-processing, allowing a second worker to acquire the lock and begin a duplicate charge while the first worker is still waiting on the gateway. The lock TTL must exceed the maximum possible payment client execution time, or the payment client's timeout envelope must be reduced to fit within the lock TTL with margin.

```python
# ORDER_LOCK_TTL must be > (REQUEST_TIMEOUT * MAX_RETRIES) + (RETRY_BACKOFF * (MAX_RETRIES - 1))
# Currently: 60 < 100  ← lock expires before payment can complete
ORDER_LOCK_TTL = 120  # minimum safe value given current payment client config
```

### 🟡 Medium — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include the field value in their string representation when the field fails validation. If `payment_token` fails its length check, the raw token value may appear in logs. Payment tokens must never appear in log output. Scrub or replace the token field before logging, or log only field names and error types.

```python
safe_errors = [{"field": e["loc"], "type": e["type"]} for e in exc.errors()]
logger.warning("Invalid order request fields=%s", safe_errors)
```

### 🟡 Medium — `charge_id` Missing from Already-Processed Result

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id is absent
    )
```

The `existing` record presumably contains the original `charge_id`. Callers performing idempotent retries need the original charge ID to reconcile their records. Returning it here costs nothing and prevents callers from treating a duplicate as a failure.

### 🟡 Medium — Lock Release Errors Are Silently Ignored

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If the Lua script call raises a `redis.RedisError` (connection lost, Redis restart), the exception propagates out of the `finally` block and masks any exception already in flight from within the `yield`. The lock will eventually expire via TTL, but the original processing exception is lost, making debugging significantly harder.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
    except redis.RedisError:
        logger.error("Failed to release lock for order_id=%s; will expire via TTL", order_id)
```

### 🟢 Minor — Test Coverage Gaps

The test suite validates the happy path and a few branch conditions but misses the failure modes introduced by the bugs above:

- No test for `PaymentError` raised inside the lock — would immediately reveal the silent swallow bug.
- No test for DB transaction failure after a successful charge — would reveal the orphaned payment bug.
- No test asserting `charge_id` is returned on the already-processed path.
- No test for lock TTL expiry during a long payment call.
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, but does not assert the script was called with the correct token, making it a weak correctness check.
