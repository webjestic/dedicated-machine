Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical atomicity gap between the payment charge and the database transaction that creates a real double-charge window, and several secondary issues across the lock lifecycle, payment client, and test coverage that must be addressed before this can be approved for a government-grade billing system.

---

## Critical Issues

### 1. Charge-Before-Record Race Condition (At-Most-Once Guarantee is Broken)

This is the most serious flaw. The PR claims an "at-most-once payment guarantee," but the architecture does not deliver it.

```python
# Charge the payment method
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is made **outside and before** the database transaction. If the process crashes, is killed, loses network connectivity, or the database transaction rolls back after `charge()` succeeds, the customer has been charged with no record of it. The idempotency check (`db.get_charge`) will return `None` on the next attempt, and the customer will be charged again.

This is not a theoretical edge case. This is the exact failure mode distributed payment systems are designed to prevent.

**Required remediation:** Adopt a two-phase approach. Before calling the payment gateway, write a `PENDING` charge record to the database within a transaction. After a successful charge, update that record to `COMPLETED` with the `charge_id`. On startup or retry, detect `PENDING` records and reconcile them against the gateway before re-attempting. This is the industry-standard pattern for idempotent payment processing.

---

### 2. Lock TTL Does Not Cover the Full Critical Section

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

The comment says "typical" — but the lock must cover the **worst case**, not the typical case. The critical section includes:

- `db.get_charge()` — potentially slow under load
- `payment.charge()` — up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds` worst case
- `db.transaction()` with two writes

The payment client alone can run for **100 seconds** in the worst case. A 60-second TTL means the lock can expire while a charge is in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. This directly contradicts the PR's stated guarantee.

**Required remediation:** Either (a) set `ORDER_LOCK_TTL` to a value that provably exceeds the worst-case critical section duration with margin, or (b) implement lock extension (a watchdog/renewal pattern) for long-running operations. The TTL and the payment client's maximum runtime must be co-designed and documented together.

## Significant Issues

### 3. Unhandled `PaymentError` Propagates Through Lock Release But Leaves No Audit Trail

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    with self.db.transaction():
        ...
```

`PaymentError` is not caught in `process()`. It will propagate up through the `contextmanager`, the lock will be correctly released (good), but the caller receives an unhandled exception with no `OrderResult` returned. The method signature promises `OrderResult` in all cases. The caller has no way to distinguish a payment decline from a bug without catching the raw exception.

Additionally, a declined payment (4xx from gateway) raises `PaymentError` immediately without retry — correct behavior — but this is indistinguishable at the `process()` level from a gateway outage.

**Required remediation:** Catch `PaymentError` in `process()` and return a structured `OrderResult(processed=False, message=...)`. Consider differentiating decline vs. gateway failure in the result.

---

### 4. Lock Release Silently Swallows Failures

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception is not caught. This will suppress any exception currently propagating from the `try` block — the original error is lost. Furthermore, the log message says "released" unconditionally, even if the script returned `0` (meaning another worker's token was found, which should be impossible but indicates a serious state corruption if it occurs).

**Required remediation:**

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error(
                "Lock release token mismatch — possible TTL expiry during critical section order_id=%s",
                order_id,
            )
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock for order_id=%s — lock will expire via TTL", order_id)
```

A return value of `0` from the Lua script is a sentinel for a serious operational event and must be logged as an error, not silently ignored.

---

### 5. Payment Client Retries on Timeout Without Idempotency Key

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

On a timeout, the request may have been received and processed by the payment gateway — the response simply didn't arrive. Retrying without an idempotency key sent to the gateway means the gateway may process the charge twice. This is a well-known failure mode.

**Required remediation:** Generate a stable idempotency key per charge attempt (e.g., derived from `order_id`) and pass it as a header (e.g., `Idempotency-Key`) on every request to the payment gateway. This makes retries safe at the gateway level regardless of timeout behavior.

---

### 6. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This specific line is fine. However, `payment_token` is accepted as a `str` with `max_length=256` and flows into `payment.charge(token=req.payment_token, ...)`. Confirm that no logging path — including exception formatters, request logging middleware, or error reporting integrations — will serialize the full `OrderRequest` or `payment_token` value. Payment tokens are PCI-DSS regulated data. This must be audited across the entire call stack, not just the lines visible here.

## Minor Issues

### 7. Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

Linear backoff with no jitter means all contending workers retry at the same intervals, creating a thundering herd. With only 3 attempts, this is low severity, but exponential backoff with jitter is the correct pattern for distributed lock contention.

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.1)
```

---

### 8. Test Coverage Gaps

The test suite validates the happy path and basic contention, but is missing:

| Missing Test | Risk |
|---|---|
| `PaymentError` raised during `charge()` | Unhandled exception path — confirms issue #3 |
| DB transaction rollback after successful charge | Confirms the double-charge race in issue #1 |
| Lock TTL expiry mid-critical-section | Validates the TTL gap in issue #2 |
| Lock release returns `0` (token mismatch) | Validates issue #4 remediation |
| Payment timeout retry with idempotency key | Validates issue #5 remediation |
| `amount_cents` boundary values (0, 1, 1_000_000, 1_000_001) | Boundary validation |

The existing tests are well-structured and the mock patterns are clean — this is a coverage gap, not a quality issue with what's present.

---

### 9. `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` accepts arbitrary Unicode. If `customer_id` is used in any downstream query, log line, or key construction, this is a potential injection surface. Apply the same or equivalent pattern constraint.
