Decision: Request Changes

Summary: The core locking logic contains a critical TOCTOU race condition: a successful payment charge is issued *outside* the database transaction, meaning a crash or exception between `payment.charge()` and `self.db.transaction().__enter__()` will result in a real charge with no recorded transaction — money is taken but the order is never marked complete, and the idempotency check will not protect against a retry since no record exists. Additionally, the distributed lock provides weaker guarantees than advertised because Redis lock expiry (60s) can race against a slow payment gateway call (up to 90s total retry window: 3 attempts × 30s timeout), allowing the lock to expire and a second worker to acquire it while the first is still mid-charge. There are also several secondary issues detailed below that need to be addressed before this can be approved.

---

## Critical Issues

### 1. Charge-Before-Record TOCTOU — Money Lost on Crash

```python
# PROBLEM: charge happens outside the DB transaction
charge_id = self.payment.charge(...)          # real money moves here

with self.db.transaction():                   # crash between these two =
    self.db.record_charge(...)                # charge exists, no record,
    self.db.mark_order_complete(...)          # idempotency check misses it
```

The payment gateway is charged before the database transaction opens. If the process crashes, loses its DB connection, or the transaction rolls back after this point, the customer is charged but the system has no record of it. On retry, `db.get_charge()` returns `None`, so the idempotency check passes and the customer is charged again.

**Required fix:** The charge record must be written *before or atomically with* the external charge. The standard pattern is a two-phase approach: write a `PENDING` charge record inside a DB transaction first, then issue the external charge, then update the record to `COMPLETE`. The DB record acts as the idempotency anchor, not the charge itself.

```python
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also handle the `PENDING` state — a pending record means a prior attempt may have charged; do not re-charge without reconciliation logic.

### 2. Lock TTL Shorter Than Maximum Payment Retry Window

```python
ORDER_LOCK_TTL = 60        # lock expires after 60 seconds
# payment_client.py:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
# worst case: 3×30s + 2×5s = 100 seconds total
```

The payment client can take up to **100 seconds** in the worst case (3 attempts × 30s timeout + 2 × 5s backoff). The Redis lock TTL is 60 seconds. The lock will expire while Worker A is still waiting on a payment response. Worker B can then acquire the lock, pass the idempotency check (no record exists yet), and issue a second charge. This is exactly the double-charge scenario the lock is supposed to prevent.

**Required fix:** Either raise `ORDER_LOCK_TTL` to safely exceed the maximum payment window (≥120s with margin), or implement lock extension (a background thread/task that refreshes the TTL while the payment call is in-flight). The TTL and payment client timeout constants should be defined in one place and the lock TTL derived from them, not set independently.

```python
# Derive lock TTL from payment client constants so they stay in sync
MAX_PAYMENT_DURATION = (
    PaymentClient.REQUEST_TIMEOUT * PaymentClient.MAX_RETRIES
    + PaymentClient.RETRY_BACKOFF * (PaymentClient.MAX_RETRIES - 1)
)
ORDER_LOCK_TTL = MAX_PAYMENT_DURATION + 30  # 30s safety margin
```

## Significant Issues

### 3. Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)   # raises PaymentError
    # exception propagates — lock IS released via finally, good
    # BUT: caller gets an unhandled exception, not an OrderResult
```

`PaymentError` is not caught in `process()`. It will propagate as an unhandled exception to the caller. The method signature promises it returns `OrderResult`, and `LockError` is handled, but `PaymentError` is not. This is an inconsistent contract.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

### 4. Lock Retry Logic Has No Jitter — Thundering Herd Under Contention

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — deterministic
time.sleep(backoff)
```

All workers contending on the same order will retry at identical intervals. Under high concurrency this creates synchronized retry bursts against Redis. This is a minor concern for low-concurrency deployments but a real problem at scale.

**Required fix:** Add jitter.

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

### 5. `payment_token` Logged Indirectly — Verify It Never Reaches Log Sinks

The `payment_token` field is accepted as a plain string and passed directly to the payment client. While it is not explicitly logged in this code, if any exception is raised that includes the request object (e.g., a `ValidationError` repr or a debug log of `req`), the token will appear in log output. Payment tokens are sensitive credentials.

**Required fix:** Mark the field as a secret in Pydantic and ensure it is excluded from any serialization or logging:

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

Then pass `req.payment_token.get_secret_value()` only at the point of use.

---

### 6. Test Suite Does Not Cover the Critical Failure Path

There is no test for the crash-between-charge-and-record scenario, no test for lock expiry during payment, and no test for `PaymentError` handling. The test suite validates the happy path and basic contention but does not exercise the failure modes that matter most for a payment processor.

**Required fix:** Add tests for:
- `payment.charge()` raises `PaymentError` → `OrderResult.processed` is `False`, no DB write occurs
- `db.transaction()` raises after a successful charge → verify behavior and what state is left
- Lock TTL relationship to payment timeout (can be a unit/config assertion)

## Minor Notes

- `customer_id` has no pattern constraint unlike `order_id`. If it is used in any query or key construction downstream, it should be validated similarly.
- The `CHARGE_ENDPOINT` is hardcoded in `PaymentClient`. This should come from configuration/environment to support different deployment environments without code changes.
- `time.sleep()` in the lock retry path will block the thread. If this is ever used in an async context this will block the event loop. Document the threading assumption or provide an async variant.
