Decision: Request Changes

Summary: The architecture demonstrates solid fundamentals — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic input validation, and idempotency gating are all correctly implemented. However, there is a critical atomicity gap between the payment charge and the database commit that creates a real-world double-charge vector, and several secondary issues compound the risk enough that this cannot be approved in its current form.

---

## Critical Issues

### 1. Charge-Before-Record Race Condition — Double Charge Vector

This is the most serious flaw. The sequence inside the lock is:

```python
charge_id = self.payment.charge(...)       # money leaves the customer's account

with self.db.transaction():                # only NOW do we persist anything
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the process crashes, the container is OOM-killed, the DB connection drops, or the `db.transaction()` raises **after** `payment.charge()` returns, the charge has settled externally but nothing is written to the database. On any subsequent retry the idempotency check (`db.get_charge`) finds nothing and **charges again**. The distributed lock does not protect against this because the lock is gone — it expired or was released — by the time the retry arrives.

The canonical fix is a two-phase approach:

1. Write a `PENDING` charge record to the database **before** calling the payment gateway, inside a transaction, with a unique idempotency key derived from `order_id`.
2. Call the payment gateway.
3. Update the record to `COMPLETED` (or `FAILED`) in a second transaction.

The idempotency check must then also catch `PENDING` records and either surface them as "in progress" or, with a sufficiently old timestamp, treat them as candidates for reconciliation. Many payment gateways also accept a caller-supplied idempotency key — that should be passed through.

---

### 2. Payment Timeout Retry is Unsafe Without Gateway-Side Idempotency

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` means the request may or may not have been processed by the gateway. Retrying unconditionally on timeout without sending a stable idempotency key to the gateway can result in duplicate charges at the network level, independent of the application-level lock. The `charge()` call must include a stable, order-scoped idempotency key in the request headers (e.g., `Idempotency-Key: order-001`) so the gateway deduplicates on its side.

## Significant Issues

### 3. Lock TTL vs. Payment Timeout Math is Broken

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still mid-charge, invalidating the entire mutual exclusion guarantee. Either raise `ORDER_LOCK_TTL` well above the worst-case payment path (≥ 120s with margin), or implement lock extension (e.g., a background thread calling `EXPIRE` periodically while the payment is in flight). The values must be documented as coupled constraints and enforced together.

### 4. `PaymentError` is Swallowed Silently

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    with self.db.transaction():
        ...
```

`PaymentError` and any unexpected exception raised inside the lock context are not caught in `process()`. They will propagate up through the `contextmanager`, which will correctly release the lock via `finally`, but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the stated contract of the method and will cause 500-level failures at the API layer with no structured error response. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 5. Lock Retry Backoff is Linear and Contention-Blind

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

With only 3 attempts and a 60-second TTL, a worker that loses the lock will exhaust retries in under 3 seconds and return a contention failure, even though the lock will be held for up to a minute. The retry window is far too narrow relative to the lock lifetime. Either increase `LOCK_RETRY_ATTEMPTS` substantially, use exponential backoff with jitter to avoid thundering herd, or surface the contention result as a retriable status code to the caller rather than a terminal failure.

## Minor Issues

### 6. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value (a sensitive credential) may appear in logs. Sanitize the error output or log only field names and error types, not values.

### 7. Test Coverage Gaps

The test suite validates the happy path and basic contention, but is missing:

- **Crash-after-charge, before DB write** — the core double-charge scenario described in issue #1.
- **Lock TTL expiry mid-payment** — a second processor acquiring the lock while the first is still in `payment.charge()`.
- **`PaymentError` propagation** — currently unhandled; no test would catch a regression here.
- **Timeout retry with idempotency key** — verifying the same key is sent on each retry attempt.
- **DB transaction rollback** — verifying behavior when `record_charge` or `mark_order_complete` raises.

The `test_lock_release_uses_lua_script` test is structurally correct but only verifies the mock was called, not that the correct token was passed. That test should assert `args=[<token>]`.

## Summary of Required Changes Before Approval

| Priority | Item |
|---|---|
| 🔴 Critical | Pre-record `PENDING` charge before calling payment gateway |
| 🔴 Critical | Pass stable idempotency key to payment gateway on all attempts including retries |
| 🟠 High | Align `ORDER_LOCK_TTL` with worst-case payment duration (currently under by ~40s) |
| 🟠 High | Catch and structure `PaymentError` and unexpected exceptions in `process()` |
| 🟡 Medium | Widen lock retry window or document contention as a retriable caller condition |
| 🟡 Medium | Sanitize Pydantic validation error logging to exclude sensitive field values |
| 🟢 Low | Expand test coverage to include crash-after-charge and token assertion in lock release test |
