Decision: Request Changes

Summary: The overall architecture is sound — distributed locking with token-verified Lua release, idempotency checks, Pydantic validation, and transactional DB writes are all the right ideas. However, there is a critical atomicity gap between the payment charge and the database transaction that creates a real double-charge risk, plus several secondary issues around error handling, secret exposure, and test coverage gaps that need to be addressed before this merges.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — double-charge window exists

This is the most serious bug in the PR. The charge happens *before* the DB transaction, and if `record_charge` or `mark_order_complete` fails, the charge has already been captured with no record of it. On retry, the idempotency check finds nothing in the DB and charges again.

```python
# CURRENT — charge is outside the transaction
charge_id = self.payment.charge(...)          # money leaves customer account here

with self.db.transaction():                   # if this fails or throws...
    self.db.record_charge(...)                # ...this never commits
    self.db.mark_order_complete(...)          # ...and we have a ghost charge
```

The standard pattern for this class of problem is to record a *pending* charge in the DB first, inside the transaction, then attempt the external charge, then mark it complete. If the process dies between the charge and the commit, a reconciliation job can detect pending-but-unconfirmed charges and resolve them against the gateway. There is no clean way to make an external HTTP call atomic with a DB write — the architecture needs to account for that explicitly.

```python
# RECOMMENDED APPROACH
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(...)  # attempt external charge

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

A reconciliation job should periodically inspect rows stuck in `pending` state and query the gateway to determine their true outcome.

---

### 2. `PaymentError` is silently swallowed in `process()`

The `except` block in `process()` only catches `LockError`. If `self.payment.charge()` raises `PaymentError`, it propagates uncaught, bypasses the `OrderResult` return contract, and surfaces as an unhandled exception to the caller. This breaks the stated interface and will cause upstream failures.

```python
# In process(), add:
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 3. Payment token logged or passed through without sanitization

`payment_token` is accepted as a raw string and passed directly to the payment client. Confirm that this token is never written to logs. The current code does not log it directly, but the `ValidationError` path calls `logger.warning("Invalid order request: %s", e)` — Pydantic validation errors include field values in their output, which means a malformed token could appear in log output. Mask or truncate sensitive fields before logging.

## Moderate Issues

### 4. Lock TTL of 60 seconds is dangerously close to the payment client's maximum execution time

`ORDER_LOCK_TTL = 60` seconds. The payment client allows up to `MAX_RETRIES=3` attempts with `REQUEST_TIMEOUT=30` seconds each and `RETRY_BACKOFF=5` seconds between them. Worst case: `3 × 30 + 2 × 5 = 100 seconds`. The lock expires before the payment completes, another worker acquires the lock, passes the idempotency check (nothing recorded yet), and issues a second charge. Either extend the TTL significantly (e.g., 180 seconds) or reduce the payment client's maximum execution time to fit well within the lock window.

### 5. Lock retry backoff is too short and does not include jitter

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Three workers hitting the same order simultaneously will all retry on nearly identical schedules, causing a thundering herd. Add randomized jitter:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.3)
```

### 6. `payment_token` field has no format validation

`customer_id` and `payment_token` accept any string up to their max length with no pattern constraint. At minimum, `payment_token` should be validated against the expected token format from your payment gateway (e.g., `^tok_[a-zA-Z0-9]+$`) to catch misrouted or malformed tokens before they hit the network.

## Minor Issues

### 7. Test coverage gaps

The test suite is missing coverage for the following cases, all of which represent real failure modes:

- `PaymentError` raised by `payment.charge()` — currently would cause an unhandled exception (see issue #2)
- DB transaction failure after a successful charge — the ghost charge scenario
- Lock TTL expiry mid-processing (requires mocking time or TTL behavior)
- `requests.ConnectionError` in `PaymentClient.charge()` — currently uncaught and not retried

### 8. `PaymentClient` does not handle `ConnectionError`

`requests.exceptions.ConnectionError` is not caught in the retry loop. A DNS failure or network blip raises immediately without retrying.

```python
except (requests.exceptions.Timeout, requests.exceptions.ConnectionError):
    last_error = "connection error"
    logger.warning("Payment connection error attempt=%d", attempt + 1)
```

### 9. `charge_id` extraction has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed body or missing `charge_id`, this raises `KeyError` or `JSONDecodeError` uncaught. Wrap this in a try/except and raise `PaymentError` with a descriptive message.

---

## What Is Done Well

The Lua script for atomic lock release is correct and the token-verification pattern is the right approach. Pydantic validation with the `order_id` pattern constraint is solid and the path-traversal test confirms it. The `SET NX EX` acquisition is atomic and correct. Structured logging with key-value pairs is consistent. The test fixture design is clean and the lock release test correctly asserts that `redis.delete` is never called directly.
