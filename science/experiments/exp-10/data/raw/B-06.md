Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical architectural flaw that breaks the at-most-once payment guarantee the PR claims to provide: a charge can succeed at the payment gateway while the subsequent database transaction fails, leaving the customer charged with no record in the system and no recovery path. Several additional bugs and gaps compound this concern enough to block merge.

---

## Critical Bug: Charge-Before-Record Creates Unrecoverable Double-Charge Window

This is the most serious issue. The code charges the payment gateway *outside* the database transaction:

```python
# Charge the payment method  ← happens OUTSIDE the db.transaction() block
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the charge and mark the order complete atomically
with self.db.transaction():          # ← only the RECORD is transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Failure scenario:**

1. Worker A acquires lock, passes idempotency check (no existing charge), calls `payment.charge()` — gateway succeeds, returns `charge_id`.
2. The database transaction fails (network blip, constraint violation, deadlock, anything).
3. The exception propagates up through `_acquire_lock`'s `finally` block, lock is released.
4. Worker B (or a retry) acquires the lock, passes idempotency check again (still nothing in DB), and charges the customer a second time.

The customer is now double-charged. There is no compensation, no dead-letter queue, no outbox pattern, nothing. The PR's core claim — "prevents duplicate charges under concurrent worker deployments" — is false under this failure mode.

**Required fix:** Adopt an outbox/saga pattern or at minimum persist a `charge_pending` record *before* calling the gateway, so the idempotency check catches retries even when the gateway call succeeded but the final commit did not. A minimal version:

```python
with self.db.transaction():
    self.db.create_charge_record(req.order_id, req.customer_id, req.amount_cents, status="pending")

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id, status="complete")
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must then treat `pending` records as already-in-progress, not as absent.

## High: `PaymentError` Is Silently Swallowed in `process()`

`process()` only catches `LockError`. Any `PaymentError` raised by `payment.charge()` — including permanent declines (4xx) — propagates uncaught to the caller. There is no `OrderResult(processed=False, ...)` returned; the caller receives a raw exception. This breaks the stated contract of the method, which promises to return an `OrderResult`.

```python
# Nothing catches this:
charge_id = self.payment.charge(...)  # raises PaymentError on decline or exhausted retries
```

**Required fix:** Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## High: Lock TTL Does Not Account for Payment Client's Maximum Execution Time

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock expires before the payment client finishes. Another worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first is still in flight.

**Required fix:** Either raise `ORDER_LOCK_TTL` well above 105 seconds with margin, or implement lock extension (a background thread that refreshes the TTL while the critical section is active, similar to Redlock's watchdog pattern).

---

## Medium: `payment_token` Is Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), its value — a sensitive credential — may appear in log output. This is a data exposure risk.

**Required fix:** Log only the field names and error types, not the raw error object:

```python
logger.warning("Invalid order request fields: %s", [e["loc"] for e in exc.errors()])
```

---

## Medium: No Test Coverage for the Critical Failure Path

The test suite has no test for the scenario where `payment.charge()` succeeds but `db.transaction()` raises. This is exactly the failure mode that causes double-charges. The absence of this test is part of why the bug survived to review.

**Required addition:**

```python
def test_charge_succeeds_but_db_fails_does_not_leave_ghost_charge(processor, db, payment):
    db.transaction.return_value.__exit__.side_effect = Exception("db commit failed")
    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    # After fix: a second call must NOT re-charge
    payment.charge.assert_called_once()
```

---

## Low: `requests.post` Has No Authentication Header

`PaymentClient.charge()` posts to an internal payment gateway with no authentication:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

Even on an internal network, service-to-service calls should carry credentials (mTLS, bearer token, API key). An unauthenticated internal endpoint is a lateral movement risk if any host on the network segment is compromised.

---

## Low: Retry Backoff in `_acquire_lock` Blocks the Thread Unnecessarily Long

With `LOCK_RETRY_ATTEMPTS = 3` and `backoff = 0.5 * (attempt + 1)`, total blocking time before raising `LockError` is 0.5 + 1.0 = **1.5 seconds** of `time.sleep` in a synchronous worker. If this runs in a thread pool or async context this will stall other work. Consider jitter (`random.uniform(0, backoff)`) and document the threading model assumption.
