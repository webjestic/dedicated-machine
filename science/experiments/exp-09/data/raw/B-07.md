Decision: Request Changes

Summary: The distributed locking design is solid — atomic Lua scripts, token-verified release, heartbeat renewal, and jitter backoff are all well-implemented. However, there are several correctness and reliability issues that need to be addressed before this is safe to merge into a production system: an unhandled exception path that leaks the Redis claim, a test that asserts the wrong behavior, a silent failure mode in the renewal thread, and a concurrency gap in the idempotency check.

---

## Detailed Findings

### 🔴 Critical

**1. `_dispatch` exceptions escape the claim context manager cleanly, but the DB state is left inconsistent — and the test asserts the wrong thing**

In `test_execute_unknown_job_type`, the test expects a `ValueError` to propagate out of `execute()`. But `execute()` has no catch for `ValueError`, so it *will* propagate — which means the job is neither marked complete nor marked failed in the database. On retry, the idempotency check (`get_execution`) will find nothing and re-execute, defeating the entire point of the executor.

The claim *will* be released (the `finally` block in `_claim_job` handles that), but the job state is left in limbo. The test should instead assert that `execute()` returns a `JobResult` with `claimed=False` and an appropriate error message, and the implementation should catch dispatch errors and record a failure state.

```python
# Current behavior — ValueError propagates, no DB record, job retryable forever
def _dispatch(self, job_type: str, payload: dict) -> dict:
    ...
    raise ValueError(f"No handler registered for job type: {job_type}")

# execute() has no except ValueError — it surfaces to the caller
```

**Fix:** Catch `ValueError` (and general handler exceptions) in `execute()`, record a failure state in the DB, and return a structured `JobResult(claimed=False, message="...")`.

---

**2. Idempotency check has a TOCTOU gap that the Redis lock does not fully close**

The sequence is:

1. Acquire Redis claim ✅
2. `db.get_execution(job_id)` — reads DB
3. `_dispatch(...)` — runs handler
4. `db.transaction()` — writes DB

The Redis lock prevents two workers from being in this window simultaneously, which is correct. However, if the DB transaction in step 4 fails (network blip, DB crash), the Redis claim is released in `finally`, and a retry will find no execution record and re-run the handler. This means the handler can execute more than once if the DB write fails after dispatch.

For a government contracting context where jobs may have side effects (notifications sent, records written to external systems), this is a real concern. The fix is to record a "started" or "in-progress" state in the DB *before* dispatching, inside the claim window, so a crashed worker leaves a detectable trace.

```python
# Recommended order:
with self.db.transaction():
    self.db.mark_job_started(execution_id, req.job_id)  # write before dispatch

result = self._dispatch(req.job_type, req.payload)      # now idempotency is durable

with self.db.transaction():
    self.db.record_execution(execution_id, req.job_id, req.job_type, result)
    self.db.mark_job_complete(req.job_id)
```

### 🟡 Moderate

**3. Renewal thread silently swallows all exceptions**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # claim was stolen — thread exits silently
        logger.debug("Claim renewed job_id=%s", job_id)
```

If `self._renew_script(...)` raises (Redis connection dropped, serialization error), the exception propagates out of `_renew()`, the thread dies, and the claim will expire after `CLAIM_TTL` seconds with no warning. The executing job continues running, unaware it has lost its lock. Another worker can then claim the job and begin executing it concurrently.

Additionally, when `result == 0` (claim stolen), the thread exits but the main thread is never notified. The job continues executing under a claim it no longer holds.

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        except redis.RedisError as e:
            logger.error("Claim renewal failed job_id=%s: %s — aborting", job_id, e)
            stop_event.set()  # signal main thread
            return
        if result == 0:
            logger.error("Claim lost job_id=%s — another worker may have taken over", job_id)
            stop_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread should check `stop_event` after dispatch returns (or use a shared flag) to detect a lost claim and abort the DB write.

**4. `job_type` is not validated against a known allowlist before dispatch**

`job_type` passes the Pydantic `max_length=64` check but has no pattern constraint. It is passed directly to `handlers.get(job_type)`. If handler keys are ever constructed dynamically or logged in a structured way that gets parsed downstream, an attacker-controlled `job_type` with special characters could cause issues. This is low severity given the dict lookup, but `job_type` should carry the same pattern constraint as `job_id`.

```python
job_type: str = Field(..., min_length=1, max_length=64, pattern=r'^[a-zA-Z0-9_\-]+$')
```

### 🟢 Minor / Nits

**5. `CLAIM_TTL = 60` with `RENEWAL_INTERVAL = 15` is a reasonable ratio, but should be documented as a constraint**

If a worker is paused (GC, swap, slow network) for more than 60 seconds between renewals, the claim expires. The ratio of `TTL / RENEWAL_INTERVAL = 4` gives three missed renewals before expiry, which is fine — but this relationship should be enforced in `__init__` with an assertion or documented explicitly so a future config change doesn't silently break the safety margin.

```python
assert self.CLAIM_TTL > self.RENEWAL_INTERVAL * 2, (
    "CLAIM_TTL must be significantly larger than RENEWAL_INTERVAL"
)
```

**6. Thread join timeout may be too tight**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)  # = 16 seconds
```

If the renewal thread is blocked on a slow Redis call when `stop_event.set()` is called, the join may time out and the claim release script runs before the renewal thread has exited. This is unlikely to cause data corruption (the Lua release script is atomic), but it means the thread is now orphaned and may attempt a renewal after the claim has been released. Add a log warning if `t.is_alive()` after the join.

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.warning("Renewal thread did not exit cleanly for job_id=%s", job_id)
```

**7. `payload: dict` type hint is unparameterized**

Minor, but use `dict[str, object]` or `Dict[str, Any]` for clarity and linter compatibility.
