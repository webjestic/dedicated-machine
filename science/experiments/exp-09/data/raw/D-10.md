Decision: Request Changes

Summary: The surface mechanics are well-constructed — atomic Lua scripts, token-verified release, heartbeat renewal, and idempotency checks are all present and correctly wired. However, there is a critical silent failure mode living in the gap between claim acquisition and claim release that the protected code is completely blind to, and the test suite does not probe it. If the renewal thread loses the claim mid-execution — whether due to a Redis eviction, a network partition, or a TTL miscalculation — `_dispatch` and the database transaction continue executing under the false assumption that the claim is still held. The coordination layer and the execution layer are decoupled with no feedback channel between them.

---

## Detailed Findings

### 🔴 Critical — Silent Claim Loss During Execution (Gap Failure)

This is the core issue. Walk through the adverse timing scenario:

```
t=0s   Worker A acquires claim. Token written to Redis. TTL=60s.
t=15s  Renewal thread fires. Redis responds. TTL reset to 60s.
t=30s  Redis experiences memory pressure. Key evicted (maxmemory-policy=allkeys-lru).
       OR: network partition isolates worker from Redis for >60s.
       OR: renewal thread throws an unhandled exception and silently exits.
t=31s  Worker B acquires the same claim key. It is now the legitimate holder.
t=32s  Worker A's _dispatch() is still running. It has no idea.
t=45s  Worker A completes _dispatch(). Enters db.transaction().
t=45s  Worker B also completes _dispatch(). Enters db.transaction().
t=46s  Both workers call db.record_execution() and db.mark_job_complete().
       Duplicate execution has occurred. The Redis lock protected nothing.
```

The protected code — `_dispatch` and the `db.transaction()` block — has zero awareness of claim state after acquisition. The renewal thread returns `0` when the key is gone, but that signal goes nowhere actionable:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return  # ← exits silently. nobody is listening.
        logger.debug("Claim renewed job_id=%s", job_id)
```

The thread exits. `stop_event` is never set. The `yield token` in the context manager is already past. The main thread is mid-execution and will never know the claim was lost. There is no feedback path.

**Required fix:** The renewal thread must have a mechanism to signal claim loss back to the executing context. One correct approach is a shared `claim_valid` event or flag that the main thread checks at a safe checkpoint — ideally before the database transaction commits:

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            logger.error("Claim lost during execution job_id=%s", job_id)
            claim_lost.set()  # signal the main thread
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

Then in the `execute` path, before committing:

```python
if claim_lost.is_set():
    raise ClaimError(f"Claim lost mid-execution for job {job_id}. Aborting commit.")

with self.db.transaction():
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

This does not make the system perfectly safe — there is still a race window between the check and the commit — but it closes the silent failure case and forces the design to acknowledge that the gap exists. For stronger guarantees, the database transaction itself should validate claim ownership, e.g., a conditional write that checks a `claimed_by` column matches the current execution token before marking complete.

### 🔴 Critical — Renewal Thread Exception Silently Kills Heartbeat

The `_renew` function has no exception handling. Any transient Redis error — connection reset, timeout, serialization fault — will raise an unhandled exception, kill the thread, and leave the claim to expire naturally. The main thread will again continue executing with no knowledge that heartbeating has stopped.

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(...)  # ← unguarded. throws on Redis error.
```

**Required fix:** Wrap the renewal call in a try/except. Distinguish between transient errors (log and retry) and persistent failures (set `claim_lost`, exit). A consecutive-failure counter with a threshold is a reasonable pattern here.

---

### 🔴 Critical — CLAIM_TTL / RENEWAL_INTERVAL Ratio Is Unsafe Under Load

`CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. That gives four renewal windows before expiry, which sounds comfortable. But `stop_event.wait(timeout=self.RENEWAL_INTERVAL)` is a wall-clock sleep. Under GIL contention, thread starvation, or system load, the renewal thread can be delayed significantly past its scheduled interval. There is no compensation for drift. If two consecutive renewals are delayed, the claim can expire before the third fires.

The safe design is to set `RENEWAL_INTERVAL` to no more than `CLAIM_TTL / 3`, and to record the last successful renewal timestamp so that the main thread can detect a stale heartbeat independently.

### 🟡 Moderate — `_dispatch` Executes Outside the Database Transaction

```python
result = self._dispatch(req.job_type, req.payload)   # ← runs here

with self.db.transaction():                           # ← committed here
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

If `_dispatch` succeeds but the transaction fails, the job has been executed with no record. On retry, `get_execution` returns `None`, and the job executes again. Whether this is acceptable depends on handler idempotency guarantees, but the architecture does not enforce or document that assumption. This should be explicitly addressed — either handlers must be idempotent (document and enforce it), or the execution record must be written before dispatch (write-ahead pattern).

---

### 🟡 Moderate — `execute` Lets `ValueError` From `_dispatch` Escape Unhandled

```python
with self._claim_job(req.job_id):
    ...
    result = self._dispatch(req.job_type, req.payload)
```

`_dispatch` raises `ValueError` for unknown job types. This propagates out of `execute` uncaught. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)` — meaning the test is validating the broken behavior rather than the correct one. A public `execute` method on a service boundary should not leak internal exceptions. This should be caught and returned as a structured `JobResult`.

---

### 🟡 Moderate — Idempotency Check Is Outside the Database Transaction

```python
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)   # ← read
    if existing is not None:
        return ...

    # ... time passes, _dispatch runs ...

    with self.db.transaction():                    # ← write
        self.db.record_execution(...)
        self.db.mark_job_complete(...)
```

The read and the write are not atomic at the database level. Under the silent claim loss scenario described above, two workers can both pass the `get_execution` check before either has committed. The database transaction should include a conditional insert or a unique constraint on `job_id` that makes double-execution structurally impossible, not just probabilistically unlikely.

---

### 🟡 Moderate — `t.join(timeout=...)` Result Is Not Checked

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`Thread.join()` with a timeout returns `None` whether the thread exited or timed out. If the renewal thread is hung, `_release_script` is called anyway with no indication that the thread state is unknown. The `claim_lost` signal mechanism suggested above partially addresses this, but the join result should also be explicitly checked via `t.is_alive()`.

---

### 🟢 Test Suite Gap — No Adverse Timing Coverage

The test suite validates the happy path, contention at acquisition, and already-completed idempotency. It does not test:

- Renewal returning `0` mid-execution (claim lost in the gap)
- Renewal thread throwing an exception
- `_dispatch` succeeding followed by transaction failure
- The `claim_lost` signal being set before the transaction commits

These are the exact scenarios that matter most for a distributed coordination primitive. Tests that only exercise acquisition and release boundaries give false confidence about gap behavior.

---

## Summary of Required Changes

| Severity | Issue |
|---|---|
| 🔴 | Renewal thread has no feedback path to executing context on claim loss |
| 🔴 | Renewal thread exceptions silently kill heartbeat with no alerting |
| 🔴 | TTL/interval ratio has no drift compensation under thread starvation |
| 🟡 | `_dispatch` runs outside the transaction — no write-ahead protection |
| 🟡 | `ValueError` from `_dispatch` escapes `execute` unhandled |
| 🟡 | Idempotency check is not atomic with the commit |
| 🟡 | `t.join()` timeout result is not checked |
| 🟢 | Test suite does not cover gap failure scenarios |

The architecture is not approvable in its current form. The gap between claim acquisition and claim release is where distributed systems fail, and this implementation has no instrumentation, no signaling, and no defensive logic covering that interval. The checklist items are all satisfied at the boundary level. The failure lives in the middle.
