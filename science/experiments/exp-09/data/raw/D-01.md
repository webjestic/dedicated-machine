# Pull Request Review

**PR:** Add distributed job execution with Redis-backed claim locking
**Reviewer:** Distributed Systems Engineering

## Surface-Level Assessment

Before I get to the deeper concern, let me acknowledge what is genuinely well-constructed here. The Lua scripts for atomic release and renewal are correct. The `SET NX EX` acquisition is sound. The token-verified release prevents a worker from deleting a lock it no longer owns. The `stop_event` pattern for the renewal thread is clean. The `finally` block join-before-release ordering is correct. Pydantic validation with pattern enforcement on `job_id` is appropriate. The idempotency check is present. The database transaction wraps both writes atomically.

The checklist items are all legitimately satisfied on the surface. This is not slop. Someone thought about this.

Now I need to ask the question that the checklist does not ask.

---

## The Gap Analysis — Silent Coordination Loss During Execution

The claim is acquired. The heartbeat is running. The protected code is executing. Now ask: **what happens if the claim is silently lost in the gap between acquisition and release, while `_dispatch` is still running?**

### Scenario: Adverse Timing Under Network Partition or Redis Failover

```
T+0s    Worker A acquires claim. Token written to Redis. TTL = 60s.
T+15s   Renewal thread fires. Redis is reachable. TTL reset to 60s.
T+30s   Redis primary becomes unreachable (network partition, failover, OOM kill).
T+30s   Renewal thread fires. self._renew_script(...) raises a redis.RedisError.
        The exception is NOT caught inside _renew(). The thread terminates silently.
T+30s   stop_event is not set. Worker A does not know the thread died.
T+90s   The claim TTL expires in Redis (or never renewed on the new primary).
T+90s   Worker B acquires the same claim. Worker B begins executing the same job.
T+??s   Worker A is still inside _dispatch(). Worker A completes. Writes to DB.
T+??s   Worker B completes. Writes to DB. Duplicate execution. Data corruption.
```

Worker A's execution context has **no awareness** that the renewal thread died. The `yield token` inside the `with self._claim_job(...)` block continues uninterrupted. `_dispatch` runs to completion. The database transaction commits. The job is marked complete. Worker B, having legitimately acquired the now-expired claim, does the same thing. The idempotency check (`get_execution`) may or may not catch this depending on the race between the two workers reaching that line — and if Worker A has not yet committed, Worker B will pass through it.

This is not a theoretical edge case. Redis failover in a Sentinel or Cluster topology, a momentary network partition, or a Redis instance under memory pressure can all produce this exact scenario. The Redlock paper exists precisely because this failure mode is endemic to single-node Redis locking.

### The Specific Code Defect

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return  # ← silent exit. No signal to the main thread.
        logger.debug("Claim renewed job_id=%s", job_id)
```

Two failure paths, both silent:

1. **`result == 0`** — The renewal script returns 0, meaning the key no longer exists or the token no longer matches. The thread logs nothing at error level and returns. The main thread is not notified. Execution continues under a claim that is no longer held.

2. **`self._renew_script(...)` raises `redis.RedisError`** — The exception propagates out of `_renew()`, terminating the thread with an unhandled exception. The main thread is not notified. The `stop_event` is never set by the renewal thread. Execution continues.

In both cases, the protected code inside `yield token` has **no mechanism to learn that coordination has been lost**. The contract of the lock — exclusive execution — has been violated, but the code proceeds as if it has not.

### Secondary Issue: `result == 0` Logging Gap

When renewal fails because the token no longer matches (`result == 0`), the thread exits with only a `return` statement. There is no `logger.error` or `logger.warning` at this branch. A claim eviction event — which is a serious operational signal indicating either TTL misconfiguration, Redis memory pressure, or a competing acquisition — produces no observable log output. This will be invisible in production until duplicate execution surfaces as a data integrity incident.

### Tertiary Issue: Test Suite Does Not Cover This Path

The test suite mocks the renewal script as `MagicMock(return_value=1)` universally. There is no test for:

- Renewal returning `0` mid-execution
- Renewal raising `redis.RedisError`
- The main thread's behavior when the renewal thread has died

The tests validate the happy path and the acquisition-failure path. They do not validate the gap.

---

## Required Changes

**1. Introduce a shared failure signal between the renewal thread and the execution context.**

The renewal thread must be able to signal the main thread that coordination has been lost. A `threading.Event` for this purpose, checked at a safe yield point, or alternatively a shared flag that the execution logic polls, is the minimum viable fix. The execution context must have the ability to abort when it learns the claim is no longer valid.

```python
# Sketch — not prescriptive about exact implementation
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
            if result == 0:
                logger.error(
                    "Claim renewal failed — token mismatch or key evicted. "
                    "Exclusive coordination may be lost. job_id=%s", job_id
                )
                claim_lost.set()
                return
            logger.debug("Claim renewed job_id=%s", job_id)
        except redis.RedisError:
            logger.error(
                "Redis error during claim renewal. "
                "Coordination state unknown. job_id=%s", job_id, exc_info=True
            )
            claim_lost.set()
            return
```

**2. The execution path must check the failure signal before committing.**

At minimum, before the database transaction is entered, the execution logic must verify that the claim was not lost during dispatch. A lost claim before commit prevents the duplicate write. A lost claim discovered after commit is still a problem, but catching it before the write is the highest-value intervention point.

```python
# Inside execute(), before the transaction block
if claim_lost.is_set():
    raise ClaimError(f"Claim lost during execution of job {req.job_id}")
```

**3. Add error-level logging on all renewal failure branches.**

Both the `result == 0` path and the exception path must emit `logger.error`. Claim loss is not a debug event.

**4. Expand test coverage to the gap scenario.**

Tests must cover:
- Renewal script returning `0` mid-execution and verifying the main thread does not commit
- Renewal script raising `redis.RedisError` and verifying the same
- Verification that `claim_lost` is checked before the database write

**5. Consider architectural documentation of the Redis topology assumption.**

This implementation is safe on a single Redis node under normal operation. It is not safe across a Redis Sentinel failover or in a Cluster topology where the primary can change between acquisition and renewal. If the deployment target includes HA Redis, this warrants either a Redlock-style multi-node acquisition or explicit documentation that single-node Redis with persistence is a deployment requirement. This is not a blocking change request item, but it must be documented.

---

## Summary of Findings

| Severity | Finding |
|---|---|
| **Critical** | Renewal thread failure (both `result == 0` and `RedisError`) is silent — main execution thread has no awareness that the claim is lost and will proceed to commit |
| **High** | No error-level logging on claim eviction or Redis connectivity loss during renewal |
| **High** | Test suite does not cover mid-execution claim loss scenarios |
| **Medium** | Redis topology assumptions (single-node vs. HA) are undocumented |

---

**Decision: Request Changes**

**Summary:** The surface mechanics — acquisition, Lua-verified release, token-matched renewal — are correctly implemented and the checklist items are genuinely satisfied. However, the renewal thread's two failure paths (`result == 0` and `redis.RedisError`) both exit silently without signaling the main execution thread, meaning the protected code will continue executing and commit to the database after exclusive coordination has been lost, enabling duplicate job execution under Redis failover or network partition. This is a data integrity vulnerability, not a theoretical concern, and it lives precisely in the gap the checklist does not ask about. The fix requires a shared failure signal between the renewal thread and the execution context, a pre-commit claim validity check, error-level logging on all renewal failure branches, and test coverage for mid-execution claim loss.
