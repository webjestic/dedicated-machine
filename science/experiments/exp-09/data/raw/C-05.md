Decision: Request Changes

Summary: The acquisition, renewal, and release mechanics are individually correct — atomic Lua scripts, token-verified operations, daemon thread scoped to the context manager, proper join before release. But there is a critical silent failure mode living in the gap between renewal and execution that the protected code never detects, and the test suite does not cover it. This is not a minor edge case; in a distributed government workload context this is a correctness defect that can cause duplicate execution under adverse timing.

---

## The Gap Failure — Full Lifecycle Simulation

Here is the scenario the checklist does not ask about.

**Setup:** `CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. Worker A acquires the claim and begins executing a job. The renewal thread is running.

**Adverse timing sequence:**

1. `t=0` — Worker A acquires claim. Redis key exists with Worker A's token. TTL = 60s.
2. `t=15` — Renewal thread fires. `RENEW_CLAIM_SCRIPT` succeeds. Returns `1`. TTL reset to 60s.
3. `t=30` — Renewal thread fires again. **Redis is momentarily unreachable** (network partition, Redis restart, eviction under memory pressure). The script call raises a `redis.exceptions.ConnectionError` or `redis.exceptions.ResponseError`.
4. **The renewal thread catches nothing.** Look at `_renew`:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return          # ← only handles "someone else owns it"
        logger.debug("Claim renewed job_id=%s", job_id)
```

A Redis exception propagates out of `_renew_script(...)`, is caught by nothing, and **silently kills the renewal thread**. No signal is sent to the main thread. No flag is set. The daemon thread simply dies.

5. `t=30` onward — The claim TTL is now counting down from wherever it was at `t=30`. No renewal is happening. The main thread does not know.
6. `t=90` (or wherever TTL expires) — **The Redis key expires. The claim is gone.** Worker A's token no longer owns anything.
7. `t=90+ε` — Worker B arrives, calls `SET NX EX`, acquires the claim legitimately. Worker B passes the idempotency check (`db.get_execution` returns `None` — Worker A has not committed yet). Worker B begins executing the same job.
8. Worker A and Worker B are now both executing the same job concurrently. Worker A will eventually commit. Worker B will eventually commit. Depending on the DB transaction semantics, one or both records land. The "mark_job_complete" race is now a data integrity problem.

**The protected code never learned the claim was lost.** The main thread is still inside `with self._claim_job(req.job_id):`, happily executing `_dispatch`, with no awareness that its coordination guarantee evaporated at step 4.

## Additional Issues

**1. Renewal thread exception handling is absent**

```python
# current — exception kills thread silently
result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
```

The thread needs to catch Redis exceptions, set a shared failure flag, and the main thread needs to check that flag at meaningful checkpoints. A `threading.Event` for `claim_lost` is the right primitive here.

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
            if result == 0:
                logger.error("Claim lost — token no longer matches job_id=%s", job_id)
                claim_lost.set()
                return
            logger.debug("Claim renewed job_id=%s", job_id)
        except redis.RedisError as e:
            logger.error("Renewal failed, claim integrity unknown job_id=%s: %s", job_id, e)
            claim_lost.set()
            return
```

**2. The main thread has no mechanism to act on claim loss**

Even with the flag set, the current architecture gives the main thread no way to abort `_dispatch` mid-execution. This is a harder problem. The minimum viable fix is checking `claim_lost` before the database write:

```python
with self._claim_job(req.job_id) as (token, claim_lost):
    existing = self.db.get_execution(req.job_id)
    if existing is not None:
        return JobResult(claimed=False, job_id=req.job_id, message="Job already completed")

    execution_id = str(uuid.uuid4())
    result = self._dispatch(req.job_type, req.payload)

    # Check claim integrity before committing — last safe checkpoint
    if claim_lost.is_set():
        logger.error(
            "Claim lost during execution, aborting commit job_id=%s execution_id=%s",
            req.job_id, execution_id
        )
        raise ClaimError(f"Claim lost during execution of job {req.job_id}")

    with self.db.transaction():
        self.db.record_execution(execution_id, req.job_id, req.job_type, result)
        self.db.mark_job_complete(req.job_id)
```

This does not prevent duplicate execution of `_dispatch` — that requires a cooperative cancellation mechanism or a timeout on dispatch itself — but it does prevent double-commit, which is the harder data integrity problem.

**3. `result == 0` in the renewal thread is currently a silent no-op**

```python
if result == 0:
    return   # thread exits, main thread never knows
```

This branch handles the case where another worker has taken the key. It correctly exits the renewal loop but does nothing to notify the main thread. Before this fix, it was just a quiet exit. It needs to set `claim_lost`.

**4. The test for unknown job type is wrong**

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}
    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

`execute()` catches no `ValueError`. `_dispatch` raises it, it propagates through `with self._claim_job`, through `execute`, and out to the caller uncaught. The test is asserting the wrong behavior — the real question is whether this should return a structured `JobResult` or raise. Either is defensible, but the current code raises unhandled from a public method, which is inconsistent with the rest of the error handling design that returns `JobResult` for all other failure modes. Pick one contract and enforce it.

**5. `_dispatch` calls `self.db.get_handlers()` on every invocation**

This is a minor architecture note. Handler registration should not be a per-call DB lookup. This belongs in `__init__` or as a registered mapping on the executor itself. Under load this is unnecessary latency and a failure point.

**6. Test suite does not cover the gap**

There is no test for renewal thread failure. There is no test for the `claim_lost` signal path. There is no test asserting that a commit does not happen when the claim was lost mid-execution. These are the tests that matter most for a distributed coordination primitive.

---

## What Needs to Change Before Approval

1. Renewal thread must catch `redis.RedisError` and signal a shared `claim_lost` event rather than dying silently.
2. `result == 0` in the renewal thread must also set `claim_lost`.
3. The main thread must check `claim_lost` before the database commit and abort if set.
4. The context manager signature needs to yield both the token and the `claim_lost` event.
5. The unknown job type test needs to be corrected to match the intended contract, and that contract needs to be consistent with the rest of the method.
6. Tests covering renewal failure, claim loss detection, and commit abort must be added.
7. Handler resolution should be moved out of the per-call dispatch path.

The bones of this implementation are solid. The Lua scripts are correct, the token verification is correct, the backoff is reasonable, the input validation is appropriate. The gap failure is a known class of distributed locking bug and fixing it here is straightforward once the signal path is wired in.
