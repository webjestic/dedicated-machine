Decision: Request Changes

Summary: The distributed locking design is solid — atomic Lua scripts, token-verified release, heartbeat renewal, and jitter backoff are all well-executed. However, there are several correctness and reliability issues that need to be addressed before this merges: an unhandled exception path that leaks the claim, a race condition in the idempotency check, and a test that asserts the wrong behavior. These aren't cosmetic — they're the kind of bugs that surface under production load or adversarial conditions.

---

## Findings

### 🔴 Critical

**1. Unhandled exceptions escape `execute()` without returning `JobResult`**

`_dispatch()` raises `ValueError` for unknown job types. That exception propagates out of `execute()` entirely — the caller gets an unhandled exception instead of a `JobResult`. The test `test_execute_unknown_job_type` actually *asserts this broken behavior* with `pytest.raises(ValueError)`, which means the test is validating the bug, not catching it.

The public contract of `execute()` is `-> JobResult`. It should never raise. Any internal failure should be caught and returned as a structured result.

```python
# Current behavior — breaks the contract
def _dispatch(self, job_type: str, payload: dict) -> dict:
    ...
    raise ValueError(f"No handler registered for job type: {job_type}")

# Fix: catch in execute()
except (ValueError, Exception) as e:
    logger.error("Job execution failed job_id=%s: %s", req.job_id, e)
    return JobResult(claimed=True, job_id=req.job_id, message=f"Execution failed: {e}")
```

The test needs to be corrected to assert a `JobResult` with a failure message, not a raised exception.

---

**2. Idempotency check is outside the database transaction — race condition**

```python
# Current code
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)   # read
    if existing is not None:
        return ...

    execution_id = str(uuid.uuid4())
    result = self._dispatch(...)                    # work

    with self.db.transaction():                     # write — too late
        self.db.record_execution(...)
        self.db.mark_job_complete(...)
```

The Redis claim prevents concurrent workers from racing here in the normal case, but the idempotency check and the write are not atomic at the database level. If the Redis claim is lost mid-execution (e.g., Redis failover, network partition) and another worker acquires the claim before `record_execution` commits, both workers can pass the `get_execution` check and both will write. The check should be inside the transaction, or the database write should use an upsert/insert-if-not-exists with a unique constraint as the true idempotency guard.

---

### 🟡 Moderate

**3. Claim is silently released if the renewal thread detects token loss**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return   # ← silently exits; job continues executing without a claim
```

If renewal fails (another worker stole the claim, Redis evicted the key), the thread exits quietly while `_dispatch()` continues running. The job will execute and attempt to write to the database with no claim, potentially duplicating work. The renewal thread should signal the main thread to abort — either via the `stop_event`, a shared flag, or a `threading.Event` that `execute()` checks after `_dispatch()` returns.

**4. `CLAIM_TTL = 60` is hardcoded with no relationship to actual job duration**

A 60-second TTL with a 15-second renewal interval is reasonable for short jobs, but there's no mechanism to configure this per job type or warn when a job is approaching the TTL boundary. For a government contracting context where job types may vary widely in duration, this should be at minimum configurable at the `JobExecutor` constructor level.

---

### 🟢 Minor / Suggestions

**5. `job_type` has no pattern validation**

`job_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `job_type` is validated only for length. Since `job_type` is used as a handler dispatch key and potentially logged, it should have a similar pattern constraint to prevent unexpected characters from entering the system.

**6. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — join result is not checked**

If the renewal thread doesn't exit within the timeout, `join()` returns silently and the claim is released anyway. The thread becomes a background leak. The result of `join()` should be checked and logged as a warning if the thread is still alive.

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.warning("Renewal thread did not exit cleanly for job_id=%s", job_id)
```

**7. `payload: dict` type hint is unparameterized**

Minor, but `dict` should be `dict[str, Any]` for clarity and linter compatibility, consistent with the Pydantic model's intent.
