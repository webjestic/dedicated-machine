Decision: Request Changes

Summary: The distributed locking architecture is sound — atomic Lua scripts, token-verified release, heartbeat renewal, and exponential backoff are all implemented correctly. However, there are several reliability and correctness issues that need to be resolved before this merges: unhandled exceptions inside the claim context can silently swallow errors, the test suite has a behavioral gap that masks a real bug, and there are a few smaller concerns worth addressing.

---

## Issues

### 1. `_dispatch` exceptions are unhandled inside the claim context — job gets silently dropped

**Severity: High**

```python
with self._claim_job(req.job_id):
    ...
    result = self._dispatch(req.job_type, req.payload)  # raises ValueError or anything else

    with self.db.transaction():
        self.db.record_execution(...)
        self.db.mark_job_complete(req.job_id)
```

If `_dispatch` raises — or if the DB transaction fails — the exception propagates out of the `with self._claim_job(...)` block. The claim is released correctly (the `finally` block handles that), but the exception is **not caught** in `execute()`. The caller receives an unhandled exception rather than a structured `JobResult`. The job is also left in an ambiguous state: the claim is gone, but `mark_job_complete` was never called, so the next worker will re-attempt it. That may be the desired behavior for transient errors, but it needs to be an explicit design decision with logging, not an accidental fall-through.

**Fix:** Catch dispatch and DB exceptions explicitly, log them, and return a `JobResult(claimed=True, message="Job failed: ...")` or re-raise with clear intent. Also consider whether a failed job should be marked as failed in the DB rather than left as pending.

### 2. The test for unknown job type asserts the wrong behavior

**Severity: Medium — masks a real bug**

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

This test asserts that `execute()` raises `ValueError` to the caller. But looking at the implementation, `_dispatch` raises `ValueError`, which is not caught anywhere in `execute()`. That means the exception leaks through the public API — callers get a raw exception instead of a `JobResult`. The test is *confirming a bug* rather than asserting correct behavior. The correct test should assert `result.claimed is False` and an appropriate error message, once the exception handling from issue #1 is addressed.

### 3. Renewal thread silently exits on Redis failure — claim expires without warning

**Severity: Medium**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # <-- silent exit
        logger.debug("Claim renewed job_id=%s", job_id)
```

If the renewal script returns `0` — meaning the claim was lost, possibly due to Redis eviction or a network partition — the thread exits silently. The main thread continues executing the job with no awareness that the claim is gone. Another worker could acquire the claim and begin processing the same job concurrently, defeating the entire purpose of the lock.

**Fix:** When renewal fails, signal the main thread (e.g., via a shared flag or `stop_event` variant) so it can abort execution and return a failure result. At minimum, log at `ERROR` level, not silently return.

### 4. `CLAIM_TTL = 60` seconds is fragile for "slow jobs" without documented bounds

**Severity: Low-Medium**

The comment says "generous for slow jobs," but 60 seconds with a 15-second renewal interval means the heartbeat has to succeed at least once every 60 seconds or the claim expires. If a job legitimately takes several minutes and Redis has a hiccup, the claim can be lost. This is related to issue #3 but is also a configuration concern — these values should either be constructor parameters or at least documented with the expected job duration bounds.

---

### 5. `db.transaction()` context manager mock is fragile

**Severity: Low**

```python
mock.transaction.return_value.__enter__ = MagicMock(return_value=None)
mock.transaction.return_value.__exit__ = MagicMock(return_value=False)
```

This only works because `db.transaction()` is called once. If it were called a second time, `return_value` would return the same mock and the `__enter__`/`__exit__` would still work by coincidence. Use `MagicMock(spec=...)` or a proper context manager mock (`unittest.mock.MagicMock` with `__enter__`/`__exit__` set on a `contextmanager`-decorated helper) to make this robust.

---

### 6. Minor: `job_type` has no pattern validation

**Severity: Low**

`job_id` correctly enforces `^[a-zA-Z0-9_\-]+$`, but `job_type` only has `min_length`/`max_length`. Since `job_type` is used to look up handlers, an unexpected value won't cause injection issues here, but consistency and defense-in-depth suggest applying a similar alphanumeric pattern.

---

## What's Done Well

- Lua scripts for atomic acquire, renew, and release are correct and token-verified — no unconditional deletes.
- `SET NX EX` is the right primitive; no SETNX + EXPIRE race.
- Heartbeat thread is properly scoped to the claim context with `stop_event` and `t.join()` before release.
- Pydantic validation with regex on `job_id` correctly blocks path traversal attempts.
- Idempotency check before dispatch is in the right place (inside the claim).
- DB transaction wrapping both `record_execution` and `mark_job_complete` is correct.

---

## Required Before Approval

1. Handle exceptions from `_dispatch` and `db.transaction()` inside `execute()` — return structured `JobResult` on failure.
2. Fix `test_execute_unknown_job_type` to assert correct API behavior, not exception leakage.
3. Add signal mechanism from renewal thread to main thread on claim loss.
