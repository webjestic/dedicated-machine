Decision: Request Changes

Summary: The distributed coordination mechanics are well-structured on the surface — atomic acquisition, token-verified renewal and release, heartbeat scoping, and Lua scripts are all present and correctly wired. However, there is a critical silent failure mode in the gap between renewal and execution: if the renewal thread detects claim loss (`result == 0`) it simply returns, but the main execution thread has no way to learn this has happened and will continue running, commit to the database, and return a success result — all without holding the claim it believes it holds. Additionally, the test suite contains a defect that masks a real exception path. Both issues need to be resolved before this can merge.

---

## Review

### What is correct

The fundamentals are solid. `SET NX EX` is the right primitive for atomic acquisition. Both the release and renewal scripts are token-verified, which prevents a worker from releasing or extending a claim it no longer owns. The renewal thread is scoped to the `_claim_job` context manager and joined before release. Pydantic validation with a pattern constraint on `job_id` is appropriate. The idempotency check before dispatch and the database transaction wrapping both writes are correct. Exponential backoff with jitter on contention is the right approach.

---

### Critical: Silent claim loss during execution

This is the failure the system prompt asks you to simulate. Walk through the adverse timing scenario:

1. Worker A acquires the claim. `CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`.
2. Worker A begins executing `_dispatch(...)`, which runs for longer than expected — say, a transient network stall, a slow handler, or a GIL-contended moment.
3. At `t=15s`, the renewal thread wakes and calls `RENEW_CLAIM_SCRIPT`. Redis is momentarily unreachable (network partition, Redis restart, connection pool exhaustion). The script raises an exception. The thread has no exception handler, so it crashes silently. `stop_event` is never set by the thread, but the thread is dead.
4. Alternatively: Redis is reachable but the key has already expired (e.g., the renewal fired late due to OS scheduling jitter, or a prior renewal attempt failed silently). `result == 0` is returned. The thread logs nothing and returns. The main thread is not notified.
5. Worker B, observing the expired key, acquires the claim and begins executing the same job.
6. Worker A's `_dispatch(...)` completes. Worker A enters the database transaction and commits. Worker B does the same. Both workers have now executed the job and recorded completion.

The protected code never learns the claim was lost. There is no mechanism for the main thread to observe the renewal thread's state. The `stop_event` is only ever set by the `finally` block — it is never set by the renewal thread to signal failure upward.

**The fix requires a shared failure channel between the renewal thread and the main execution thread.** One correct approach:

```python
# In _claim_job, before starting the thread:
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
        except Exception as e:
            logger.error("Renewal script raised during job_id=%s: %s", job_id, e)
            claim_lost.set()
            return
        if result == 0:
            logger.error("Claim lost during execution job_id=%s", job_id)
            claim_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must then check this event at a meaningful point — either by polling between logical phases of `_dispatch`, or by having `_dispatch` itself be interruptible. At minimum, the check must occur before the database transaction is entered:

```python
# In execute(), before entering the db.transaction() block:
if claim_lost.is_set():
    raise ClaimError(f"Claim lost during execution of job {job_id}")
```

This does not fully close the window — there is still a race between the final renewal check and the commit — but it eliminates the silent failure for the common case and makes the failure mode visible. If the system requires a stronger guarantee, the claim key should be checked inside the database transaction using a two-phase commit or a compare-and-swap against the token value before the write is finalized.

Also note: the renewal thread currently swallows all Redis exceptions because there is no `try/except` around the script call. A Redis connection error during renewal is indistinguishable from a clean exit. This must be fixed regardless of the signaling mechanism.

### Moderate: `test_execute_unknown_job_type` asserts the wrong behavior

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

`execute()` does not propagate `ValueError`. `_dispatch` raises it, but `execute()` has no handler for it, so it will bubble up through the `with self._claim_job(...)` block, through the `finally` clause (which correctly releases the claim), and then propagate to the caller as an unhandled exception. The test is asserting that this exception escapes, which is technically true — but the behavior is undocumented, unstructured, and inconsistent with every other error path in `execute()`, all of which return a `JobResult`. A caller integrating against this interface will receive a `JobResult` for validation errors, claim errors, and duplicate jobs, but a raw `ValueError` for unknown job types. This should either be caught and returned as a `JobResult(claimed=False, message=...)`, or explicitly documented as a programming error that is intentionally not caught. The test should reflect whichever decision is made.

---

### Minor: Renewal thread timing is not bounded against TTL

`RENEWAL_INTERVAL = 15` and `CLAIM_TTL = 60`. The ratio is 4:1, which is reasonable. However, this relationship is implicit and not enforced. If someone adjusts `CLAIM_TTL` downward (e.g., to 20 seconds for a tighter system), the renewal interval of 15 seconds leaves only one renewal window before expiry, and any single missed renewal loses the claim. Consider asserting or documenting the required relationship: `RENEWAL_INTERVAL` must be significantly less than `CLAIM_TTL / 2` to tolerate at least one missed renewal cycle.

---

### Minor: `_dispatch` fetches handlers from the database on every call

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
```

This is a database call inside the claim window on every execution. If `get_handlers()` is expensive or can fail, it adds latency and a failure mode inside the protected section. Handler registration should be done at construction time or cached, not fetched per-dispatch.

---

### Summary of required changes

| Severity | Issue |
|---|---|
| Critical | Renewal thread failure (both `result == 0` and exceptions) is not signaled to the main thread; execution continues without a valid claim |
| Critical | Renewal thread has no exception handler around the Redis script call; connection errors are silently swallowed |
| Moderate | `ValueError` from unknown job type escapes `execute()` unhandled, inconsistent with all other error paths; test asserts the broken behavior |
| Minor | `RENEWAL_INTERVAL` / `CLAIM_TTL` relationship is implicit and fragile |
| Minor | Handler registry fetched from DB on every dispatch inside the claim window |
