Decision: Request Changes

Summary: The distributed locking logic and renewal thread are well-structured, and the atomic Lua scripts for claim release and renewal are correct. However, there are several reliability and correctness issues — most critically, unhandled exceptions inside `_dispatch` will escape the `with self._claim_job()` context without recording failure state, and the test suite has a flawed assertion that will pass vacuously rather than catching a real bug. These need to be resolved before this is safe to merge into a production system.

---

## Findings

### 🔴 Critical

**1. Unhandled exceptions from `_dispatch` escape silently with no job state update**

In `execute()`, if `_dispatch` raises (e.g., `ValueError` for unknown job type, or any runtime exception from a handler), the exception propagates out of the `with self._claim_job()` block. The claim is correctly released by the context manager, but the job is left in an ambiguous state — not marked failed, not marked complete. On the next attempt, `get_execution()` returns `None`, so the job re-executes. Depending on the handler, this could cause duplicate side effects.

```python
# Current — _dispatch exception propagates uncaught
result = self._dispatch(req.job_type, req.payload)

with self.db.transaction():
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

At minimum, a failed dispatch should be recorded. A `try/except` around `_dispatch` should catch exceptions, record a failure state, and either re-raise or return a structured `JobResult(claimed=True, message="Job failed")`. Leaving the job invisible to the failure-tracking system is a reliability gap.

---

**2. Test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)` — but the code catches it nowhere, meaning this test validates the wrong behavior**

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

The `execute()` method has no `except ValueError` handler, so yes, the `ValueError` from `_dispatch` will propagate — and the test will pass. But this means the test is asserting that a public API method throws a raw `ValueError` to its caller, which is almost certainly not the intended contract. The caller has no way to distinguish this from a programming error. This should either return a structured `JobResult(claimed=False, message="Unknown job type")` and the test should assert on that, or the exception type should be a domain-specific one. As written, this is a latent API design bug that the test is accidentally validating as correct behavior.

---

### 🟡 Significant

**3. Renewal thread failure is silently ignored**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(...)
        if result == 0:
            return  # ← claim was lost, but execute() has no idea
        logger.debug("Claim renewed job_id=%s", job_id)
```

If the renewal script returns `0` (claim was stolen or expired), the thread exits quietly. The main thread continues executing the job and will attempt to write results — potentially after another worker has already claimed and completed the same job. This is the exact race condition the locking is meant to prevent.

The renewal thread should signal the main thread (e.g., via a shared `threading.Event` or by setting a flag checked before the DB write) so execution can be aborted when the claim is lost.

---

**4. Redis connection errors are not handled**

`self.redis.set(...)`, `self._renew_script(...)`, and `self._release_script(...)` can all raise `redis.RedisError`. None of these are caught. A transient Redis outage will surface as an unhandled exception to the caller of `execute()`. Given this is a government-facing reliability system, Redis errors should be caught and surfaced as structured results or logged with appropriate severity, not allowed to crash the worker.

---

**5. `_dispatch` calls `self.db.get_handlers()` on every execution**

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
    handler = handlers.get(job_type)
```

Handler registration is presumably static or near-static. Hitting the database on every dispatch is unnecessary overhead and introduces a failure point inside the claim window. Handlers should be registered at construction time or cached, not fetched per-execution.

---

### 🟠 Minor / Hardening

**6. `t.join(timeout=...)` in `finally` block does not verify the thread actually stopped**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`Thread.join(timeout=...)` returns `None` whether the thread stopped or timed out. If the renewal thread is stuck (e.g., blocked on a Redis call), the release script fires anyway, but the thread continues running in the background against a now-released key. The return value of `t.is_alive()` should be checked after the join and logged as a warning if the thread did not stop cleanly.

**7. `CLAIM_TTL = 60` with `RENEWAL_INTERVAL = 15` is a reasonable ratio, but is not documented as a constraint**

If someone changes `RENEWAL_INTERVAL` to be greater than `CLAIM_TTL / 2`, the claim can expire between renewals. This relationship should be enforced with an assertion in `__init__` and documented explicitly.

```python
assert self.RENEWAL_INTERVAL < self.CLAIM_TTL / 2, (
    "RENEWAL_INTERVAL must be less than half of CLAIM_TTL to prevent claim expiry between renewals"
)
```

**8. No test covers renewal thread behavior or claim-lost mid-execution scenario**

The test suite covers the happy path, contention, idempotency, and validation — all good. But there is no test for the case where the renewal script returns `0` mid-execution, which is the most dangerous failure mode this system is designed to handle. This should be tested.
