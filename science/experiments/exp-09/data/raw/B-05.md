Decision: Request Changes

Summary: The distributed locking design is solid — atomic Lua scripts, token-verified release, heartbeat renewal, and jitter backoff are all correct. However, there are several correctness and reliability issues that need to be addressed before this merges: an unhandled exception path that leaks the claim, a test asserting the wrong behavior for unhandled errors, a silent data loss risk when `_dispatch` raises inside the claim window, and a thread-safety gap in the renewal thread's exit path.

---

## Detailed Findings

### 🔴 Critical — Unhandled exceptions inside `execute()` silently leak the claim

**File:** `services/job_executor.py`, `execute()` method

The `with self._claim_job(req.job_id):` block correctly releases the claim in its `finally` clause — but only if the exception propagates *out of the context manager*. The problem is that `_dispatch()` can raise `ValueError` for unknown job types, and that exception is **not caught** inside `execute()`. It will propagate out of the `with` block (claim gets released — good), but then it bubbles up to the caller with no `JobResult` returned. The caller gets a raw exception instead of a structured response.

The test `test_execute_unknown_job_type` actually asserts `pytest.raises(ValueError)` — meaning the test is **validating the broken behavior**, not catching a bug. This is a test that should be failing the PR, not passing it.

```python
# Current behavior — leaks a raw exception to the caller
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}
    with pytest.raises(ValueError):          # ← this is wrong; callers expect JobResult
        executor.execute("job-001", "unknown_type")
```

**Fix:** Catch `ValueError` (or a more specific `HandlerNotFoundError`) inside `execute()` and return a structured `JobResult`.

```python
except ValueError as e:
    logger.error("Dispatch failed job_id=%s: %s", req.job_id, e)
    return JobResult(claimed=False, job_id=req.job_id, message="Unknown job type")
```

And update the test to assert `result.claimed is False`.

### 🔴 Critical — No exception handling around `_dispatch()` means a failed job permanently holds the "claimed" state in Redis until TTL expiry

**File:** `services/job_executor.py`, `execute()` method

If `_dispatch()` raises any exception other than `ValueError` (network error, handler bug, timeout), the exception propagates out of the `with self._claim_job` block. The `finally` in `_claim_job` *does* release the Redis lock — that part is fine. But `db.mark_job_complete()` is never called, so the job stays in an incomplete state. On retry, `db.get_execution()` returns `None`, so the job will be re-dispatched. That's actually the *correct* behavior for retryable failures — but only if it's intentional and documented. Right now it's accidental and undocumented, and the distinction between "retryable failure" and "permanent failure" is not modeled at all.

**Recommendation:** Add explicit error handling with a `db.mark_job_failed()` path, or at minimum add a docstring clarifying the retry contract.

### 🟡 Medium — Renewal thread can silently exit without the main thread knowing

**File:** `services/job_executor.py`, `_claim_job()` context manager

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return   # ← exits silently; main thread has no idea the claim expired
        logger.debug("Claim renewed job_id=%s", job_id)
```

If `_renew_script` returns `0` (another worker stole the token, or Redis evicted the key), the renewal thread exits quietly. The main thread continues executing the job and writing to the database, believing it still holds the claim — but it doesn't. This is a classic TOCTOU window in distributed locking.

**Fix:** Use a shared flag or `Event` to signal the main thread that the claim was lost, and raise `ClaimError` (or a new `ClaimLostError`) in the main thread before writing results.

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(...)
        if result == 0:
            claim_lost.set()
            return
        ...

# In the yield block, check before writing:
if claim_lost.is_set():
    raise ClaimError(f"Claim lost during execution for job {job_id}")
```

### 🟡 Medium — `job_type` has no pattern constraint, only length

**File:** `services/job_executor.py`, `JobRequest`

```python
job_type: str = Field(..., min_length=1, max_length=64)
```

`job_id` has a strict alphanumeric pattern, but `job_type` does not. Since `job_type` is used as a dispatch key into `db.get_handlers()`, an attacker or misconfigured caller could pass in arbitrary strings including path traversal sequences or injection payloads depending on how `get_handlers()` is implemented. Apply the same pattern discipline here.

```python
job_type: str = Field(..., min_length=1, max_length=64, pattern=r'^[a-zA-Z0-9_\-]+$')
```

---

### 🟡 Medium — `db.get_handlers()` is called on every dispatch

**File:** `services/job_executor.py`, `_dispatch()`

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
    handler = handlers.get(job_type)
```

This hits the database on every single job execution to retrieve the handler registry. Handler registries are almost always static or infrequently changing. This is a latency and load concern at scale. Cache the handler map at construction time or with a short TTL. At minimum, document why a live DB lookup is required here.

### 🟢 Minor — `t.join(timeout=...)` result is not checked

**File:** `services/job_executor.py`, `_claim_job()`

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`Thread.join(timeout=...)` does not raise if the thread is still alive after the timeout — it returns silently. If the renewal thread is stuck (e.g., blocked on a Redis call), the main thread proceeds to release the claim anyway. This is probably acceptable behavior, but it should be logged.

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.warning("Renewal thread did not exit cleanly for job_id=%s", job_id)
```

---

### 🟢 Minor — `payload` type annotation uses bare `dict`

**File:** `services/job_executor.py`, `JobRequest`

```python
payload: dict = Field(default_factory=dict)
```

Prefer `dict[str, Any]` (Python 3.9+) or `Dict[str, Any]` for explicitness and to enable downstream type checking. Bare `dict` passes through anything.

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| 🔴 | `ValueError` from `_dispatch` leaks to caller; test validates wrong behavior |
| 🔴 | No failure state modeled for dispatch errors — retry contract is implicit |
| 🟡 | Renewal thread exits silently on claim loss; main thread continues unaware |
| 🟡 | `job_type` missing pattern validation |
| 🟡 | `db.get_handlers()` called per-dispatch with no caching |
| 🟢 | `t.join()` timeout not checked |
| 🟢 | Bare `dict` annotation on `payload` |

The bones of this implementation are good. The Lua scripts, the `SET NX EX` acquisition, the token-verified release, and the heartbeat design are all correct. Fix the exception handling contract and the silent claim-loss gap and this is approvable.
