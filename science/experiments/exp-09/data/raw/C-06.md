Decision: Request Changes

Summary: The acquisition, renewal, and release mechanics are individually sound — atomic Lua scripts, token-verified operations, daemon thread scoped to the context manager, clean backoff on contention. But there is a critical undetected failure mode living in the gap between renewal and execution: if the renewal thread loses the claim silently (Redis eviction, network partition, TTL expiry before the first renewal fires), the protected code continues executing with no awareness that exclusivity has been lost. A second worker can acquire the claim and begin the same job while the first worker is still running `_dispatch` or committing the transaction. The system has no mechanism to detect or react to this split-brain window.

---

## Detailed Findings

### 1. Silent Claim Loss During Execution — Critical

This is the core flaw. Walk through the adverse timing scenario:

```
t=0s   Worker A acquires claim, TTL=60s
t=1s   Worker A enters _dispatch() — long-running handler begins
t=15s  Renewal thread fires, calls RENEW_CLAIM_SCRIPT
       Redis is momentarily unreachable — script returns 0 (or throws)
       Renewal thread sees result == 0, logs nothing, returns — thread exits silently
t=60s  Claim TTL expires naturally — key is gone from Redis
t=61s  Worker B calls SET NX EX — succeeds, acquires claim legitimately
t=61s  Worker B passes idempotency check (db.get_execution returns None — A hasn't committed yet)
t=61s  Worker B enters _dispatch() — duplicate execution begins
t=90s  Worker A finishes _dispatch(), enters db.transaction()
t=90s  Worker B finishes _dispatch(), enters db.transaction()
       Both workers now race to commit
```

The protected code — `_dispatch` and the transaction block — has no knowledge that the claim was lost at t=15s. There is no cancellation signal, no poison flag, no re-verification before committing. The renewal thread returning early is the only signal, and nothing reads it.

The renewal thread's early-return path is:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return          # ← silent exit, caller never knows
        logger.debug("Claim renewed job_id=%s", job_id)
```

`result == 0` means the claim is already gone. The thread exits. The main thread is still inside the `yield` block. Nothing connects these two facts.

A Redis exception during renewal is equally silent — it propagates up and kills the thread with no notification to the main thread.

---

### 2. No Pre-Commit Claim Verification — Critical (Consequence of #1)

Even if claim loss detection were added, the window between detection and the database commit is still a race. The correct pattern is to verify the claim is still held immediately before committing, not just at acquisition. This is the distributed analog of optimistic locking:

```python
# Before committing, verify the claim is still ours
current = self.redis.get(claim_key)
if current != token.encode():
    raise ClaimError("Claim lost before commit — aborting")
```

This does not fully close the race (there is still a gap between the check and the commit), but it dramatically narrows it and makes the failure detectable rather than silent.

---

### 3. Renewal Thread Exception Handling — High

If `self._renew_script(...)` raises (Redis connection error, timeout, serialization error), the exception propagates out of the `while` loop, the thread dies, and the claim will expire. The main thread is not notified. This needs a broad exception catch with a signal back to the caller:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
            if result == 0:
                claim_lost_event.set()   # signal the main thread
                return
        except Exception:
            logger.exception("Renewal failed for job_id=%s", job_id)
            claim_lost_event.set()
            return
```

---

### 4. No Claim-Loss Propagation Into Execution Context — High

The fix requires a shared signal between the renewal thread and the execution path. A `threading.Event` is the right primitive here. The context manager should expose it or check it at yield boundaries:

```python
claim_lost = threading.Event()

# In _renew: set claim_lost on failure
# In execute: check claim_lost before and after _dispatch

if claim_lost.is_set():
    raise ClaimError("Claim lost during execution — aborting before commit")
```

Without this, the `finally` block in the context manager will still call `_release_script` with a token that no longer matches the live key — which will correctly return 0 and not delete another worker's claim (the Lua script handles this correctly) — but the damage is already done because the job ran twice.

---

### 5. `_dispatch` Raises, But Test Expects It to Propagate Uncaught — Medium

`test_execute_unknown_job_type` asserts `pytest.raises(ValueError)`. But `execute()` does not catch `ValueError`. This means an unknown job type will propagate as an unhandled exception through the claim context manager, which will still correctly release the claim in `finally`. The behavior is technically correct but the test is asserting on an undocumented contract. If someone later wraps `_dispatch` in a broad exception handler (reasonable), this test will silently start passing for the wrong reason. The executor should explicitly handle `ValueError` from dispatch and return a structured `JobResult` with an error message.

---

### 6. TTL Arithmetic Creates an Expiry Window — Low

`CLAIM_TTL = 60` and `RENEWAL_INTERVAL = 15`. The first renewal fires at t=15s, which is well within the 60s TTL. This is fine under normal conditions. But if the renewal thread is delayed (GIL contention, GC pause, system load), the effective renewal window shrinks. Consider setting `RENEWAL_INTERVAL` to `CLAIM_TTL // 4` enforced by assertion, not by convention, so a future config change cannot accidentally invert the ratio.

---

## What Needs to Change Before Approval

1. **Renewal thread must signal claim loss** — use a `threading.Event` set on `result == 0` or on any exception.
2. **Execution path must check that signal** — at minimum before the database commit, ideally also after `_dispatch` returns.
3. **Renewal thread must catch exceptions** — bare `except Exception` with logging and event signaling, not silent thread death.
4. **Pre-commit claim re-verification** — a `GET` on the claim key before entering the transaction, compared against the local token.
5. **`ValueError` from dispatch should be caught and returned as `JobResult`** — not leaked as an unhandled exception.

The architecture is sound. The locking primitives are correctly chosen. The failure mode is not in the design, it is in the gap between the renewal thread's knowledge and the execution thread's knowledge. Close that gap and this is approvable.
