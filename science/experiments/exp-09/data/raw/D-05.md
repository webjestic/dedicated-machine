Decision: Request Changes

Summary: The acquisition, renewal, and release mechanics are each individually correct — atomic via Lua, token-verified, and properly scoped. However, there is a critical silent failure mode living in the gap between renewal and execution: when the renewal thread detects that its claim has been lost (result == 0), it returns quietly without signaling the executing thread, meaning the protected business logic continues running without coordination, with no awareness that another worker may now hold the same claim. This is not a locking bug at the boundary — it is a locking bug in the interior, and the test suite does not cover it at all.

---

## Full Review

### What the PR Gets Right

I want to be precise here because the surface mechanics are genuinely solid and I don't want that obscured by what follows.

- `SET NX EX` is correctly atomic. There is no SETNX/EXPIRE split that would create a window for TTL to never be set.
- The Lua scripts for both renewal and release are correct. They are token-verified and atomic. No unconditional deletes. No TOCTOU on the key.
- The `stop_event.wait(timeout=...)` pattern is the right way to drive a renewal loop. It avoids busy-waiting and responds promptly to shutdown.
- `t.join()` before `_release_script` is called is the correct ordering. The thread is stopped before the lock is released, which prevents a race where the thread renews a key that the main thread has already released and another worker has re-acquired.
- The idempotency check against the database before dispatch is correct in intent.
- Pydantic validation with the `job_id` pattern is appropriate for the injection surface.
- The exponential backoff with jitter is correctly structured.

The checklist items are not lies. Each one of them is individually true. The problem is what they collectively fail to guarantee.

---

### The Silent Failure in the Gap

Here is the lifecycle the PR describes:

```
[acquire claim] → [yield into execute()] → [heartbeat renews every 15s] → [release claim]
```

Now simulate this adverse scenario:

1. Worker A acquires the claim. Token is set in Redis with a 60-second TTL.
2. Worker A's renewal thread fires at t=15s. Redis is momentarily unreachable or the key has been evicted (memory pressure, Redis restart, operator error). The Lua script returns `0`.
3. The renewal thread sees `result == 0` and executes `return`. It exits. No exception. No event. No signal to the main thread.
4. The claim TTL was 60 seconds. At t=60s, the key expires naturally.
5. At t=61s, Worker B acquires the same claim. It passes the idempotency check — the job is not yet marked complete. Worker B begins executing.
6. Worker A is still inside `_dispatch()`. It has been running since t=0. It completes at t=75s and enters `self.db.transaction()`.
7. Both workers are now inside the handler or the commit path concurrently. The database transaction provides some protection, but the job handler itself — `_dispatch` — has already executed twice.

The protected region is not just the database write. It is the entire block from `_dispatch` through `db.transaction()`. The claim exists to protect that entire region. The renewal thread is the mechanism that keeps the claim valid across that region. When the renewal thread silently exits, the claim protection has ended, but the code does not know.

The specific line is here:

```python
if result == 0:
    return
```

This is the failure. A renewal failure is treated as a terminal condition for the thread but not for the execution. The thread exits. The job continues. The gap opens.

---

### Why the Test Suite Does Not Catch This

The test suite mocks `register_script` to return a `MagicMock` with `return_value=1`. This means:

- The renewal script always reports success.
- The release script always reports success.
- There is no test where `result == 0` is returned mid-execution.
- There is no test that verifies the main thread is interrupted when the renewal thread exits.

`test_claim_release_uses_lua_script` verifies that the Lua path is taken instead of a raw delete. That is correct and worth testing. But it says nothing about what happens when that script returns failure during renewal.

The test suite is testing the happy path and the acquisition-failure path. The interior failure path — claim lost during execution — is entirely untested.

### Additional Issues

**1. `ValueError` from `_dispatch` is unhandled and leaks through `execute()`**

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

The test asserts that `ValueError` propagates. But `execute()` is a public method returning `JobResult`. Callers expecting a `JobResult` will receive an unhandled exception instead. The `ValueError` should be caught and returned as a `JobResult(claimed=False, message=...)`. The claim will still be released via `finally`, so that part is safe — but the interface contract is broken.

**2. Renewal thread failure is not observable**

When `result == 0` causes the thread to return, there is no log at WARNING or ERROR level. There is no metric. There is no way for an operator to know that a claim was lost mid-execution without instrumenting the Redis key directly. At minimum this should be a `logger.error()` call. Ideally it raises into a structured alert path.

**3. `CLAIM_TTL` of 60 seconds is load-bearing and undocumented as such**

The TTL must be strictly greater than `RENEWAL_INTERVAL` to guarantee that a single missed renewal does not expire the claim. 60 > 15, so the math works. But this relationship is not enforced in code and not documented in the class. If someone changes `RENEWAL_INTERVAL` to 45 or `CLAIM_TTL` to 30 during a future refactor, the guarantee silently breaks. An assertion or a validator in `__init__` would catch this:

```python
assert self.CLAIM_TTL > self.RENEWAL_INTERVAL * 2, (
    "CLAIM_TTL must be more than 2x RENEWAL_INTERVAL to survive a missed renewal"
)
```

**4. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` does not verify join completion**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`join()` with a timeout returns whether or not the thread has actually stopped. If the thread is blocked — on a hung Redis call, for example — `join()` returns after the timeout and the release script runs anyway. The thread is now live and will attempt to renew a key that may have been re-acquired by another worker. This should be:

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.error("Renewal thread did not stop cleanly for job_id=%s — potential claim conflict", job_id)
```

At minimum log it. Ideally treat it as a fault condition.

**5. `db.get_handlers()` is called per dispatch**

This is a minor operational concern rather than a correctness issue, but calling `get_handlers()` on every dispatch means a database round-trip per job execution to retrieve the handler registry. If this is backed by a real database query, it should be cached or the handler registry should be injected at construction time. As written it is also untestable for handler registration errors independently of job execution.

### What a Correct Fix Looks Like for the Core Issue

The renewal thread needs a way to signal the main thread that the claim has been lost. The simplest correct approach uses a shared `threading.Event`:

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            logger.error(
                "Claim renewal failed — claim lost during execution job_id=%s", job_id
            )
            claim_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The protected block then needs to check this event at meaningful points, or the dispatch layer needs to be interrupted. For long-running jobs this may require a cooperative cancellation mechanism passed into the handler. The exact shape depends on what `_dispatch` does, but the invariant is: **if `claim_lost` is set, the handler must not commit and must not be treated as a successful execution.**

A minimal guard before the database write:

```python
if claim_lost.is_set():
    raise ClaimError(f"Claim was lost during execution of job {job_id}")
```

This does not protect against the case where the claim is lost after this check and before the commit, but it closes the most common window and makes the failure visible rather than silent.
