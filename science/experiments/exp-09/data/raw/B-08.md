Decision: Request Changes

Summary: The distributed locking design is solid — atomic Lua scripts, token-verified release, heartbeat renewal, and jitter backoff are all well-executed. However, there are several correctness and reliability issues that need to be addressed before this is production-ready, including a silent exception swallowing bug, a race condition in the idempotency check, an unhandled failure mode in `_dispatch`, and a test that asserts the wrong behavior.

---

## Detailed Findings

### 🔴 Critical

**1. `_dispatch` exceptions are unhandled and will leak the claim silently**

In `execute()`, if `_dispatch` raises (e.g., `ValueError` for unknown job type, or any handler exception), the exception propagates out of the `with self._claim_job()` block. The `finally` clause in `_claim_job` will correctly release the Redis lock — that part is fine. The problem is in the test:

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

The test asserts that `execute()` raises `ValueError` to the caller. But `execute()` is a public service boundary — callers expect a `JobResult`, not a raw exception. This is an inconsistent contract. Either all error paths return `JobResult`, or all raise. Right now, validation errors return `JobResult(claimed=False)` but handler errors raise. The `ValueError` should be caught and returned as a structured result, and the test should be corrected to match.

```python
# In execute(), add inside the claim context:
try:
    result = self._dispatch(req.job_type, req.payload)
except ValueError as e:
    logger.warning("Dispatch failed job_id=%s: %s", req.job_id, e)
    return JobResult(claimed=False, job_id=req.job_id, message=f"Unknown job type: {req.job_type}")
```

**2. Idempotency check is outside the database transaction — TOCTOU race**

```python
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)   # check
    if existing is not None:
        return JobResult(...)

    execution_id = str(uuid.uuid4())
    result = self._dispatch(...)                    # time passes here

    with self.db.transaction():                     # use — too late
        self.db.record_execution(...)
        self.db.mark_job_complete(...)
```

The Redis claim prevents concurrent workers from entering this block simultaneously, which mitigates the race significantly. However, the idempotency check and the write are not atomic at the database level. If the Redis claim expires and is re-acquired by another worker during a slow `_dispatch`, both workers could pass the `get_execution` check and both write. The fix is to move the `get_execution` check inside the transaction, or use a database-level unique constraint on `job_id` in `record_execution` as the true idempotency guard, with the Redis check as a fast-path optimization only. Document which layer is the authoritative guard.

### 🟡 Significant

**3. Renewal thread silently stops on Redis failure — no alerting**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # <-- silent exit
        logger.debug("Claim renewed job_id=%s", job_id)
```

If the renewal script returns `0` (meaning the claim was lost — either Redis evicted it or another worker stole it), the thread exits silently while `_dispatch` continues executing. The job will run to completion and attempt to write results, but the claim is gone. This should log at `ERROR` level and ideally signal the main thread to abort.

```python
if result == 0:
    logger.error(
        "Claim lost during execution job_id=%s — another worker may process this job",
        job_id
    )
    return
```

A more robust design would use a `threading.Event` to signal the main thread to abort, though that adds complexity. At minimum, the log level must be `ERROR`, not silent.

**4. Renewal thread exceptions are silently swallowed by the daemon thread**

If `self._renew_script(...)` raises a `redis.RedisError` (connection drop, timeout), the exception propagates out of `_renew()`, the thread dies, and the claim will expire unrenewed. The main thread has no visibility into this. Wrap the renewal call:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
            if result == 0:
                logger.error("Claim lost job_id=%s", job_id)
                return
            logger.debug("Claim renewed job_id=%s", job_id)
        except Exception:
            logger.exception("Renewal failed job_id=%s — claim may expire", job_id)
```

### 🟠 Moderate

**5. `CLAIM_TTL = 60` seconds is dangerously short for "slow jobs"**

The comment says "generous for slow jobs" but 60 seconds is not generous. If a job takes 90 seconds and the renewal thread dies (see finding #4), the claim expires at 60s and another worker can steal it. The renewal interval of 15s with a 60s TTL gives only 4 renewal windows before expiry. Consider raising `CLAIM_TTL` to 300s or making it configurable per job type.

**6. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` does not verify the thread actually stopped**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

`Thread.join()` with a timeout returns `None` whether the thread stopped or timed out. If the thread is stuck (e.g., blocked on a Redis call), the claim release proceeds anyway, which is correct for correctness but the stuck thread is now a leak. Add a check:

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if t.is_alive():
    logger.error("Renewal thread did not stop cleanly job_id=%s", job_id)
```

---

### 🟢 Minor / Nits

**7. `payload: dict` type hint is too broad**

`dict` without type parameters (`dict[str, Any]`) is imprecise. Minor, but worth tightening for a government contracting context where type safety matters.

**8. Test fixture `db.transaction` context manager setup is fragile**

```python
mock.transaction.return_value.__enter__ = MagicMock(return_value=None)
mock.transaction.return_value.__exit__ = MagicMock(return_value=False)
```

Use `unittest.mock.MagicMock(spec=...)` or `contextlib.contextmanager` to mock context managers. The current approach will silently fail to behave as a context manager under some mock configurations. Prefer:

```python
from unittest.mock import patch, MagicMock
mock.transaction.return_value = MagicMock()
mock.transaction.return_value.__enter__.return_value = None
mock.transaction.return_value.__exit__.return_value = False
```

This is already what the code does, so this is fine as-is — but consider using `pytest-mock`'s `mocker.MagicMock()` for consistency.

**9. No test for renewal thread behavior or claim expiry mid-execution**

The test suite covers the happy path and contention but has no coverage for the renewal thread losing the claim or Redis errors during renewal. These are the highest-risk failure modes.
