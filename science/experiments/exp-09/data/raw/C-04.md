**Decision: Request Changes**

**Summary:** The surface mechanics are well-constructed — atomic acquisition, token-verified Lua release, heartbeat renewal, and idempotency checks are all present and correctly wired. However, there is a critical silent failure mode in the gap between claim loss and execution awareness, and a secondary issue in the test suite that masks it. The code cannot be approved in its current form.

---

## The Gap Failure — The Core Issue

The prompt I apply to every distributed coordination review: *if coordination is lost silently during execution — not at acquisition, not at release, but in the gap between — would the protected code know?*

Walk through the adverse timing scenario:

1. Worker A acquires the claim. Redis sets `job:claim:{job_id}` with token `T1`, TTL 60s.
2. Worker A's renewal thread fires at t=15s. `RENEW_CLAIM_SCRIPT` runs, gets `result == 1`. Claim extended.
3. At t=29s, the Redis connection drops, or Redis restarts, or a network partition isolates Worker A. The key expires or is lost.
4. Worker B acquires the same claim at t=31s with token `T2`.
5. Worker A's renewal thread fires at t=30s. `RENEW_CLAIM_SCRIPT` returns `0` — the token no longer matches.

Now look at what happens in `_renew()`:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return  # <-- thread exits silently
        logger.debug("Claim renewed job_id=%s", job_id)
```

The thread returns. It sets nothing. It signals nothing. `stop_event` is never set. The main thread, executing `_dispatch()` or writing to the database, has no idea the claim is gone. Worker A continues executing, Worker B continues executing, and both proceed to `self.db.transaction()`. The database transaction provides some protection if the DB enforces uniqueness on `execution_id` or `job_id`, but that is not guaranteed by this code, and the idempotency check (`get_execution`) has already passed for both workers by this point.

**The fix:** When renewal fails, the worker must know. The renewal thread needs a shared signal — a `threading.Event` or similar — that the main execution thread polls or blocks on at a safe checkpoint. Something like:

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            logger.error("Claim lost during execution job_id=%s", job_id)
            claim_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

And the `yield` in `_claim_job` should yield this event alongside the token, or the context manager should check it before allowing execution to continue past the dispatch boundary. At minimum, the `finally` block should check `claim_lost` and raise rather than silently releasing a claim the worker no longer holds.

## Secondary Issues

### 1. Unhandled exception propagation from `execute()`

`_dispatch()` raises `ValueError` for unknown job types. This is not caught in `execute()`. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)`, confirming the exception escapes the public interface. A method returning `JobResult` should not raise — it should return a structured failure result. Either catch it and return `JobResult(claimed=False, message=...)`, or document the exception contract explicitly and update the return type. The current behavior is inconsistent with every other failure path in this method.

### 2. The renewal thread is joined with a timeout, but the join result is never checked

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

If the thread does not exit within the timeout, `join()` returns silently and the release proceeds anyway. This is unlikely to cause a real problem in practice, but it means a hung renewal thread goes undetected. A `t.is_alive()` check after the join with a warning log costs nothing.

### 3. Test suite does not exercise the gap failure at all

The mock renewal script always returns `1`. There is no test for `result == 0` from the renewal path, no test for what happens when the claim is lost mid-execution, and no test verifying that the main thread is notified. This is the most important failure mode in the system and it has zero test coverage. The tests that do exist are well-structured, but they validate the happy path and the acquisition failure path only.

### 4. `_dispatch` fetches handlers from the database on every call

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
```

This is a minor design concern, not a blocker, but fetching the handler registry from the database inside every dispatch call is wasteful and adds a DB round-trip inside the critical section. Handler registration is presumably static or slow-changing. This should be cached at construction time or injected as a dependency.

---

## What Needs to Change Before Approval

| Priority | Issue |
|---|---|
| **Critical** | Renewal failure must signal the executing thread — silent `return` in `_renew` is the gap failure |
| **High** | `ValueError` from `_dispatch` escapes the public interface inconsistently |
| **High** | Test coverage for claim-lost-mid-execution scenario is entirely absent |
| **Low** | `t.join()` result unchecked |
| **Low** | Handler registry fetched from DB on every dispatch |

The architectural skeleton here is sound and the author clearly understands distributed locking. The gap failure is a correctness bug, not a style issue — under adverse timing it permits exactly the duplicate execution this system is designed to prevent.
