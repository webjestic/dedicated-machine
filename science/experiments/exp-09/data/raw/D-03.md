Decision: Request Changes

Summary: The surface mechanics are well-constructed — atomic acquisition, token-verified Lua release, heartbeat renewal, and idempotency checks are all present and correctly wired. However, a critical silent failure mode exists in the gap between claim loss and execution awareness: if the renewal thread detects that the claim has been lost (returns 0) it simply exits silently, leaving the protected execution code running with no coordination and no knowledge that another worker may now hold the claim. Additionally, the test suite does not probe this failure path at all, and several secondary issues compound the risk.

---

## Review

### The Gap Failure — Silent Claim Loss During Execution

This is the central flaw. Walk through the adverse timing scenario:

```
T+0s    Worker A acquires claim, token written to Redis, TTL=60s
T+15s   Renewal thread fires, renew_script returns 1 — OK
T+30s   Redis experiences a transient partition or eviction event
        The key is gone. Worker A's claim no longer exists in Redis.
T+45s   Worker B calls redis.set(claim_key, token_B, nx=True, ex=60)
        NX succeeds — key is absent. Worker B now holds the claim.
T+45s   Worker A's renewal thread fires, renew_script returns 0.
        Thread logs nothing, sets no flag, raises no exception.
        Thread exits its while loop and returns.
T+45s   Worker A is still inside the `with self._claim_job(...)` block.
        _dispatch() is running or the db.transaction() is in flight.
        Worker A has no idea the claim is gone.
T+60s   Worker B passes the idempotency check (db.get_execution returns None
        because Worker A has not committed yet), dispatches the job,
        enters its own db.transaction().
        Duplicate execution. Both workers commit.
```

The renewal thread's return value is checked, but the result is discarded into a void. The executing thread — the one inside `yield token` — is never notified.

```python
# Current code — the failure is silent
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            return          # <-- thread exits. nobody is told. execution continues.
        logger.debug("Claim renewed job_id=%s", job_id)
```

The fix requires a shared signal between the renewal thread and the executing thread. The renewal thread must be able to interrupt execution when the claim is lost. One correct approach:

```python
# Introduce a claim-loss event alongside the stop event
claim_lost_event = threading.Event()
stop_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            logger.error(
                "Claim lost during execution — signaling abort job_id=%s", job_id
            )
            claim_lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

But setting an event is not enough on its own. The executing thread must actually check it. A passive event does nothing if the executing code never polls it. The protected execution path — `_dispatch` and the database transaction — must be made claim-loss-aware. One viable pattern is to raise an exception from the renewal thread into the main thread using `ctypes` thread interruption, or to structure the execution as a cooperative loop that checks the event at safe boundaries. The simplest auditable approach is a checked exception model:

```python
class ClaimLostError(Exception):
    """Raised when the distributed claim is lost during execution."""
    pass

# In the finally block and at yield, check before proceeding:
try:
    yield token, claim_lost_event   # caller receives the event
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    if not claim_lost_event.is_set():
        self._release_script(keys=[claim_key], args=[token])
```

And in `execute`:

```python
with self._claim_job(req.job_id) as (token, claim_lost_event):
    existing = self.db.get_execution(req.job_id)
    if existing is not None:
        return JobResult(claimed=False, job_id=req.job_id, message="Job already completed")

    if claim_lost_event.is_set():
        raise ClaimLostError("Claim lost before dispatch")

    execution_id = str(uuid.uuid4())
    result = self._dispatch(req.job_type, req.payload)

    if claim_lost_event.is_set():
        raise ClaimLostError("Claim lost after dispatch, before commit")

    with self.db.transaction():
        self.db.record_execution(execution_id, req.job_id, req.job_type, result)
        self.db.mark_job_complete(req.job_id)
```

Note that even this is not fully airtight — there is still a TOCTOU window between the final `claim_lost_event.is_set()` check and the database commit. A fully hardened design would use a database-level idempotency guard (unique constraint on `job_id` in the executions table) as the final arbiter, making the Redis claim a performance optimization rather than a correctness guarantee. That layered approach is the correct architecture for this class of problem.

### Secondary Issues

**1. Unhandled exception propagation from `_dispatch` exits the claim context without structured handling**

```python
result = self._dispatch(req.job_type, req.payload)
```

If `_dispatch` raises `ValueError` (unknown handler), it propagates out of the `with self._claim_job` block, through the `finally` clause, and is not caught in `execute`. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)` — meaning the caller receives a raw unhandled exception rather than a structured `JobResult`. This is inconsistent with every other failure path in the method, which returns `JobResult`. The claim is released correctly via `finally`, but the caller contract is broken.

```python
# Should be caught and returned as a structured result
except ValueError as e:
    logger.error("No handler for job_type=%s job_id=%s: %s", req.job_type, req.job_id, e)
    return JobResult(claimed=False, job_id=req.job_id, message=f"Unknown job type: {req.job_type}")
```

**2. Renewal thread joined with a fixed timeout that may be insufficient**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
```

`RENEWAL_INTERVAL` is 15 seconds. The join timeout is 16 seconds. If the renewal thread is blocked on a slow Redis call (network stall, not a clean timeout), `t.join` returns after 16 seconds with the thread still running. The release script then executes. The thread may subsequently wake and attempt a renewal against a key that has already been released — or against a key now held by another worker. The thread is daemonized so it will not block process exit, but the interleaving is a correctness hazard. The renewal script should use a Redis client configured with an explicit socket timeout shorter than `RENEWAL_INTERVAL`.

**3. The idempotency check is not atomic with claim acquisition**

```python
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)
    if existing is not None:
        return JobResult(...)
```

The claim is acquired, then the database is queried. This is correct sequencing — the claim gates the check. However, if the database read fails (connection error, timeout), the exception propagates out of the `with` block, the claim is released, and the job is left in an indeterminate state with no record. This should be caught explicitly.

**4. `CLAIM_TTL` is a class-level constant with no relationship to actual job duration**

60 seconds is described as "generous for slow jobs." If a job legitimately runs longer than 60 seconds and the renewal thread fails to renew for any reason (Redis unavailability lasting longer than one renewal window), the key expires and another worker can acquire the claim. The TTL and renewal interval relationship is correct in structure (`RENEWAL_INTERVAL < CLAIM_TTL`), but there is no upper bound enforcement on job execution time, and no alerting when renewal approaches the TTL boundary.

**5. Test suite does not cover the gap failure mode at all**

None of the tests simulate a renewal returning 0 during active execution. The most important failure path — claim loss mid-execution — has zero test coverage. A test should mock `_renew_script` to return 0 on the second call and assert that execution is aborted and no duplicate record is written.

```python
def test_claim_lost_during_execution_aborts_job(redis_client, db):
    renew_mock = MagicMock(side_effect=[1, 0])  # first renewal ok, second lost
    redis_client.register_script.return_value = renew_mock
    # Assert: ClaimLostError raised or JobResult reflects abort
    # Assert: db.record_execution not called (or called exactly once)
```

**6. `_dispatch` calls `self.db.get_handlers()` on every invocation**

This is a database call inside the claim-held critical section on every job execution. If handler registration is static or infrequently changing, this is unnecessary latency inside the protected window. Handlers should be registered at construction time or cached, not fetched per-dispatch.
