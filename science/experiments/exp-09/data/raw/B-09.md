Decision: Request Changes

Summary: The core distributed locking architecture is sound — atomic Lua scripts, token-verified release, heartbeat renewal, and Pydantic validation are all well-implemented. However, there are several correctness and reliability issues that need to be addressed before this merges: an unhandled exception path that leaks the Redis claim, a test that asserts the wrong behavior for an unhandled exception, a subtle race condition in the renewal thread, and a missing error handling gap in `_dispatch`. These aren't cosmetic — they're the kind of bugs that surface under load or failure conditions in production.

---

## Issues

### 1. `_dispatch` exceptions escape the claim context without being caught — claim is released, but job is left in a broken state

**Severity: High**

In `execute()`, if `_dispatch` raises (e.g., `ValueError` for unknown job type, or any exception thrown by a handler), the exception propagates out of the `with self._claim_job(...)` block. The `finally` clause in `_claim_job` *will* release the Redis lock correctly — that part is fine. The problem is that the exception then propagates out of `execute()` entirely, giving the caller a raw exception instead of a structured `JobResult`. The contract of `execute()` is to return a `JobResult`, and it breaks that contract on handler failure.

```python
# Current: ValueError escapes to caller
result = self._dispatch(req.job_type, req.payload)

# Fix: catch inside the claim context and return structured result
try:
    result = self._dispatch(req.job_type, req.payload)
except ValueError as e:
    logger.warning("Unknown job type job_id=%s type=%s: %s", req.job_id, req.job_type, e)
    return JobResult(claimed=True, job_id=req.job_id, message=f"Unknown job type: {req.job_type}")
except Exception as e:
    logger.error("Handler failed job_id=%s type=%s: %s", req.job_id, req.job_type, e, exc_info=True)
    return JobResult(claimed=True, job_id=req.job_id, message="Job execution failed")
```

Note `claimed=True` is appropriate here — the worker *did* claim the job, it just failed to execute it. This distinction matters for the caller deciding whether to retry.

### 2. The test for unknown job type asserts the wrong behavior

**Severity: High** (test is validating a bug, not correct behavior)

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):  # This is wrong
        executor.execute("job-001", "unknown_type")
```

This test asserts that `execute()` raises `ValueError`, but `execute()` is supposed to return a `JobResult`. This test is passing *because* the bug in issue #1 exists. Once issue #1 is fixed, this test should be updated to assert a structured failure result:

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    result = executor.execute("job-001", "unknown_type")

    assert result.claimed is False  # or True depending on your chosen semantics
    assert "Unknown job type" in result.message
    db.record_execution.assert_not_called()
```

### 3. Renewal thread has a race condition on stop — can renew after release

**Severity: Medium**

The renewal thread uses `stop_event.wait(timeout=RENEWAL_INTERVAL)` as its sleep mechanism. When `stop_event.set()` is called in `finally`, the thread wakes up, checks `not stop_event.wait(...)` which returns `True` (event is set), and exits the loop — correct so far. But there's a window: if the renewal thread is mid-execution of `_renew_script` when `stop_event.set()` is called, the thread will complete the renewal *after* the claim has already been released by `_release_script`. This means you're briefly re-extending a TTL on a key that was just deleted, which could allow a stale claim to reappear if the delete raced with the renewal.

The fix is to check `stop_event` inside `_renew` before calling the script:

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        if stop_event.is_set():  # double-check before renewing
            return
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            logger.warning("Claim lost during renewal job_id=%s", job_id)
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

Also note: when `result == 0` from the renewal script, the current code silently returns. This should be a warning log — it means another worker has taken the claim or the key expired, which is a serious operational event.

### 4. `db.get_handlers()` is called on every dispatch — potential performance and consistency issue

**Severity: Low-Medium**

Calling `self.db.get_handlers()` inside `_dispatch` on every job execution means a database round-trip per job just to look up the handler registry. If handler registration is dynamic, this could also introduce a consistency window where a handler is registered between the idempotency check and dispatch. Consider caching handlers at construction time or accepting them as a constructor argument, and only refreshing on explicit invalidation:

```python
def __init__(self, redis_client: redis.Redis, db, handlers: dict = None):
    ...
    self._handlers = handlers or {}
```

---

### 5. Minor: `job_id` missing from `JobResult` on claim contention

**Severity: Low**

In the `ClaimError` handler, `job_id` is not included in the returned `JobResult`:

```python
return JobResult(
    claimed=False,
    message="Job currently claimed by another worker"
    # job_id is missing — caller can't correlate this result to the job
)
```

The `job_id` is available as `req.job_id` at that point. Include it for consistency with other return paths.

## What's Done Well

- Lua scripts for atomic release and renewal are correct and well-scoped
- `SET NX EX` is the right primitive — no TOCTOU on acquisition
- Heartbeat thread is properly daemonized and joined before release
- Exponential backoff with jitter is textbook correct
- Pydantic pattern enforcement on `job_id` blocks path traversal and injection
- `db.transaction()` wrapping both writes is the right approach
- Structured logging with consistent field names throughout
