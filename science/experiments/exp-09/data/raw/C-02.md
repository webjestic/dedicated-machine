**Decision: Request Changes**

**Summary:** The surface mechanics are well-constructed — atomic acquisition, token-verified Lua release, heartbeat renewal, idempotency check, and structured retry all look correct. However, there is a critical silent failure mode in the gap between claim acquisition and execution: if the renewal thread loses the claim (Redis eviction, network partition, TTL misconfiguration), the protected code continues executing with no awareness that coordination has been lost, and a second worker can acquire the claim and run the same job concurrently. Additionally, there is a test that asserts the wrong behavior for an unhandled exception path, and the renewal thread's loss signal is discarded rather than acted upon.

---

## Detailed Findings

### 1. [Critical] Silent Claim Loss During Execution — The Gap Problem

This is the central flaw. Walk through the adverse timing scenario:

```
t=0:    Worker A acquires claim, starts executing
t=15:   Renewal thread fires, Redis returns 1 — claim extended
t=30:   Renewal thread fires — Redis is unreachable (network partition)
        _renew_script raises an exception
        The thread catches nothing, exits silently via daemon thread death
t=45:   CLAIM_TTL expires — Redis evicts the key
t=46:   Worker B calls SET NX EX — acquires the claim legitimately
t=47:   Worker A is still inside _dispatch(), has no idea
t=60:   Both workers call db.transaction() — duplicate execution recorded
```

The renewal thread's contract with the main thread is broken. When `_renew_script` returns `0` (someone else holds the key) or raises (Redis unreachable), the thread calls `return` — but `yield token` in the context manager is still executing. There is no cancellation signal, no exception injection, no shared flag the main thread polls. The protected code runs to completion under a claim it no longer holds.

```python
# Current behavior — loss is silently swallowed
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # ← thread exits, main thread never knows
        logger.debug("Claim renewed job_id=%s", job_id)
```

The fix requires a shared cancellation flag that the main thread checks at meaningful checkpoints, and the renewal thread must set it on loss:

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
            if result == 0:
                logger.error("Claim lost during execution job_id=%s", job_id)
                claim_lost.set()
                return
        except Exception:
            logger.exception("Renewal failed job_id=%s — treating as claim loss", job_id)
            claim_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must then check this flag at the boundary between `_dispatch` and the database write — the exact point where a duplicate execution becomes dangerous:

```python
result = self._dispatch(req.job_type, req.payload)

if claim_lost.is_set():
    raise ClaimError(f"Claim lost during execution of job {job_id}")

with self.db.transaction():
    ...
```

This does not make the system perfectly safe — `_dispatch` itself may have had side effects — but it prevents the database from recording a completion that a second worker is also about to record, and it gives the caller a signal that something went wrong rather than returning a false success.

### 2. [Moderate] Renewal Thread Exceptions Are Unhandled

As shown above, if `_renew_script` raises (connection error, timeout, serialization failure), the exception propagates up through `_renew()` uncaught, kills the daemon thread, and the claim silently expires. This must be wrapped with exception handling that sets `claim_lost` rather than dying quietly.

### 3. [Low] `CLAIM_TTL` and `RENEWAL_INTERVAL` Ratio Has No Enforcement

`CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15` — this is a 4x ratio, which is reasonable. But these are bare class attributes with no validation. If someone changes `RENEWAL_INTERVAL` to 65 in a subclass or configuration override, the claim will expire before the first renewal attempt. A constructor assertion would catch this class of misconfiguration:

```python
def __init__(self, redis_client, db):
    assert self.RENEWAL_INTERVAL < self.CLAIM_TTL / 2, (
        "RENEWAL_INTERVAL must be less than half CLAIM_TTL to ensure safe renewal"
    )
```

### 4. [Low] Test Asserts Wrong Behavior for Unhandled Exception

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

`execute()` does not catch `ValueError`. It propagates out of `_dispatch()`, through the `with self._claim_job()` block, through the `finally` clause (which correctly releases the claim), and then out of `execute()` unhandled. The test is technically correct that `ValueError` is raised — but the behavior is unintentional. The caller receives an uncaught exception from a public method that otherwise returns `JobResult`. This should either be caught and returned as a structured `JobResult(claimed=False, message=...)`, or the test should be documenting a known gap with a comment. As written, it looks like the test is validating correct behavior when it is actually validating an unhandled path.

### 5. [Low] `execute()` Returns `claimed=True` Inside a Claim That May Already Be Lost

Even without the renewal thread gap, the return value of `JobResult(claimed=True, ...)` is constructed after the `with self._claim_job()` block exits — meaning after the claim has been released. The semantics of `claimed=True` in the result should mean "we held the claim and completed the job," which is accurate, but if claim loss is added as a failure mode, the result must reflect that. This is a documentation/semantic issue now, but becomes a correctness issue once claim loss handling is added.

---

## Required Before Approval

1. `claim_lost` event must be set by the renewal thread on both `result == 0` and on exception.
2. The main thread must check `claim_lost` between `_dispatch` and the database write, and raise `ClaimError` if set.
3. Renewal thread must wrap `_renew_script` in a try/except.
4. `test_execute_unknown_job_type` must be resolved — either catch `ValueError` in `execute()` or explicitly document the behavior as intentional with a structured decision.
