Decision: Request Changes

Summary: The architecture is thoughtful — atomic Lua scripts, token-verified release, heartbeat renewal, and Pydantic validation are all solid choices. However, there are several correctness flaws, one silent data-loss risk, and a test gap that collectively make this not ready to merge into a production distributed system. The issues range from a race condition between the idempotency check and dispatch, to an unhandled exception path that leaks the claim without recording failure, to a test that asserts the wrong behavior for an unhandled exception type.

---

## Detailed Findings

### 🔴 Critical — Silent Data Loss / Unhandled Exception Escape

**File:** `services/job_executor.py`, `execute()` method

`_dispatch()` can raise `ValueError` for unknown job types. This exception is **not caught** in `execute()`. It propagates out of the `with self._claim_job(...)` block, which means:

1. The claim is correctly released (the context manager `finally` block handles that).
2. But the caller receives an **unhandled exception** instead of a structured `JobResult`.
3. The test `test_execute_unknown_job_type` actually **asserts this broken behavior** with `pytest.raises(ValueError)` — it is testing that the bug exists, not that it is handled.

Any other exception raised inside `_dispatch()` (network error, handler crash, etc.) has the same problem. The public contract of `execute()` should be to always return a `JobResult`. The method signature and docstring imply this, but the implementation does not guarantee it.

```python
# Current — exception escapes
result = self._dispatch(req.job_type, req.payload)

# Should be something like
try:
    result = self._dispatch(req.job_type, req.payload)
except Exception as e:
    logger.error("Dispatch failed job_id=%s: %s", req.job_id, e)
    return JobResult(claimed=True, job_id=req.job_id, message=f"Execution failed: {e}")
```

The test must be updated to assert a failed `JobResult` is returned, not that `ValueError` propagates.

### 🔴 Critical — TOCTOU Race on Idempotency Check

**File:** `services/job_executor.py`, `execute()` method

```python
with self._claim_job(req.job_id):
    existing = self.db.get_execution(req.job_id)   # ← check
    if existing is not None:
        return JobResult(...)                        # ← early return releases claim

    result = self._dispatch(...)                    # ← use
    with self.db.transaction():
        self.db.record_execution(...)
        self.db.mark_job_complete(...)
```

The early return on `existing is not None` exits the `with self._claim_job(...)` block, which is correct. However, the idempotency check and the `record_execution` write are **not inside the same database transaction**. Between the `get_execution` read and the `record_execution` write, another worker that lost the Redis race could — under a Redis failure/restart scenario where the lock is lost — acquire the claim and pass the same idempotency check before either worker commits. The idempotency check should be a `SELECT FOR UPDATE` or equivalent inside the same transaction as the write, not a pre-flight read outside of it.

```python
# Safer pattern
with self.db.transaction():
    existing = self.db.get_execution_for_update(req.job_id)  # locked read
    if existing is not None:
        return JobResult(claimed=False, ...)
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

### 🟠 Major — Renewal Thread Silently Stops on Transient Redis Error

**File:** `services/job_executor.py`, `_renew()` inner function

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return   # ← silently exits
        logger.debug("Claim renewed job_id=%s", job_id)
```

If `self._renew_script(...)` raises a `redis.RedisError` (connection drop, timeout), the exception propagates out of `_renew()`, the daemon thread dies silently, and the claim TTL begins counting down. For a 60-second TTL with a 15-second renewal interval, the job has at most 60 seconds before another worker can steal it — with no warning to the executing worker.

Additionally, `result == 0` means the token no longer matches — i.e., the claim was already lost. Silently returning here means the job continues executing without a valid claim. This should at minimum log a critical-level warning, and ideally set a shared flag so `execute()` can abort or at least be aware.

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        except redis.RedisError as e:
            logger.error("Claim renewal failed job_id=%s: %s", job_id, e)
            continue  # or set a lost_claim event
        if result == 0:
            logger.critical("Claim lost during execution job_id=%s — another worker may execute", job_id)
            lost_claim_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

### 🟠 Major — `_dispatch` Fetches Handlers Per Invocation

**File:** `services/job_executor.py`, `_dispatch()` method

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
    handler = handlers.get(job_type)
```

`self.db.get_handlers()` is called on every job execution. If this is a database round-trip (which the abstraction implies), this is unnecessary latency on every job and a potential failure point inside an already-claimed execution. Handlers should be registered at construction time or cached, not re-fetched per dispatch. This also makes the handler registry a runtime database dependency, which complicates failure modes.

### 🟡 Moderate — `job_type` Has No Pattern Validation

**File:** `services/job_executor.py`, `JobRequest`

```python
job_type: str = Field(..., min_length=1, max_length=64)
```

`job_id` has a strict alphanumeric pattern enforced via regex. `job_type` does not. If `job_type` is used as a lookup key into a handler registry (and potentially logged or stored), it should have equivalent sanitization. An attacker or misconfigured caller could inject whitespace, control characters, or path-like strings.

---

### 🟡 Moderate — Thread Join Timeout Is Dangerously Short

**File:** `services/job_executor.py`, `_claim_job()` context manager

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)  # = 16 seconds
```

If the renewal thread is blocked on a slow Redis call (e.g., a connection that is hanging rather than failing fast), `t.join()` will block the `finally` block for up to 16 seconds before proceeding to release the claim. This means the claim release is delayed by up to 16 seconds after job completion, holding the lock unnecessarily. The Redis client should have its own socket timeout configured, and the join timeout should reflect that, not the renewal interval.

---

### 🟡 Moderate — No Test for Renewal Thread or Claim Loss Scenario

**File:** `tests/test_job_executor.py`

There is no test that verifies:
- The renewal script is called during a long-running job.
- The renewal thread terminates cleanly after `stop_event` is set.
- Behavior when the renewal script returns `0` (claim lost mid-execution).

The heartbeat mechanism is one of the most failure-prone parts of this design and has zero test coverage. Given that this is the core correctness guarantee of the distributed lock, this is a significant gap.

---

### 🟢 Minor — `payload: dict` Type Hint Is Too Broad

**File:** `services/job_executor.py`, `JobRequest`

```python
payload: dict = Field(default_factory=dict)
```

`dict` without type parameters (`dict[str, Any]`) is acceptable but imprecise. More importantly, there is no depth or size limit on the payload. A deeply nested or very large payload dict could cause issues downstream. Consider `max_length` on serialized size or explicit schema validation if payloads are security-relevant.

---

### 🟢 Minor — Daemon Thread Swallows Exceptions Without Visibility

The renewal thread is a `daemon=True` thread. If it raises an unhandled exception (e.g., from the Redis script call), the thread dies silently. Python daemon threads do not propagate exceptions to the parent. Combined with the missing `try/except` in `_renew()`, failures here are completely invisible unless you happen to be watching thread state.

---

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| 🔴 | `ValueError` from `_dispatch` escapes `execute()` — fix exception handling and fix the test that validates the broken behavior |
| 🔴 | Idempotency check is outside the DB transaction — move to a locked read inside the transaction |
| 🟠 | Renewal thread dies silently on Redis error or claim loss — add error handling and claim-lost signaling |
| 🟠 | `get_handlers()` called per dispatch — cache at construction or inject the registry |
| 🟡 | `job_type` has no pattern validation |
| 🟡 | Thread join timeout tied to renewal interval, not Redis socket timeout |
| 🟡 | No tests for renewal thread behavior or claim-loss scenario |
