Decision: Request Changes

Summary: The distributed locking logic and claim lifecycle are well-structured — atomic Lua scripts, token-verified release, heartbeat renewal, and jitter backoff are all solid choices. However, there are several correctness and reliability issues that need to be addressed before this merges: an unhandled exception path that leaks the claim, a test asserting the wrong behavior for an unhandled exception, a silent data loss risk when `_dispatch` raises inside the claim window, and a renewal thread timing issue that can cause a spurious missed-renewal log on shutdown.

---

## Detailed Findings

### 1. `_dispatch` exceptions escape the claim context manager cleanly, but the job is left in a bad state silently

**Severity: High**

```python
# execute()
result = self._dispatch(req.job_type, req.payload)

with self.db.transaction():
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

If `_dispatch` raises (e.g., `ValueError` for an unknown handler, or any runtime error from the handler itself), the exception propagates out of the `with self._claim_job(...)` block. The context manager's `finally` clause will correctly release the Redis claim. However:

- The exception is **not caught in `execute()`**, so it bubbles up to the caller completely unhandled.
- No `JobResult` is returned — the caller gets a raw exception with no structured error.
- The job is left in a **limbo state**: not marked complete, not marked failed. A retry will re-enter the claim and re-attempt, which may or may not be safe depending on handler idempotency.

At minimum, `_dispatch` failures should be caught, logged, and returned as a structured `JobResult(claimed=True, message="Execution failed: ...")`. If the job should be retryable, the DB state needs a `failed` marker, not just silence.

```python
# Suggested shape
try:
    result = self._dispatch(req.job_type, req.payload)
except Exception as e:
    logger.error("Dispatch failed job_id=%s type=%s: %s", req.job_id, req.job_type, e)
    # optionally: self.db.mark_job_failed(req.job_id)
    return JobResult(claimed=True, job_id=req.job_id, message=f"Execution failed: {e}")
```

### 2. The test for unknown job type asserts the wrong behavior

**Severity: Medium — test is validating a bug, not correct behavior**

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

This test asserts that `execute()` raises `ValueError` to the caller. But `execute()` is a **public service method** — callers should never receive a raw `ValueError`. This test is encoding the broken behavior described in finding #1. Once that is fixed, this test should be updated to assert a structured `JobResult` with an appropriate failure message.

### 3. Renewal thread `stop_event.wait(timeout=...)` fires once before the first renewal interval

**Severity: Low — behavioral correctness issue**

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(...)
        if result == 0:
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

`stop_event.wait(timeout=N)` returns `False` after `N` seconds if the event was not set — this is correct for the renewal loop. However, when `stop_event.set()` is called in `finally`, the thread will wake immediately and `wait()` returns `True`, exiting the loop cleanly. This part is fine.

The subtle issue: if the job completes in under `RENEWAL_INTERVAL` seconds (the common case), the thread is sleeping when `stop_event.set()` fires. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` will then block for up to 1 second past the renewal interval waiting for the thread to wake and exit. For a `RENEWAL_INTERVAL` of 15 seconds, `join` will return almost immediately because `wait()` is interrupted by the event. This is actually fine — `wait()` is interruptible by the event. No change needed here, but the `join` timeout of `RENEWAL_INTERVAL + 1` is misleadingly large; a value of `2` would be more than sufficient and would make hung-thread detection faster.

### 4. `job_type` has no pattern validation — potential injection vector

**Severity: Medium**

```python
job_type: str = Field(..., min_length=1, max_length=64)
```

`job_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `job_type` has no such constraint. While `job_type` is used only as a dict key for handler lookup and does not appear to be used in Redis keys or SQL directly, the inconsistency is a gap. If `job_type` ever gets logged into structured systems or used in key construction downstream, it becomes a risk. Apply the same or similar pattern constraint.

---

### 5. `db.get_handlers()` is called on every dispatch — potential performance and consistency issue

**Severity: Low**

```python
def _dispatch(self, job_type: str, payload: dict) -> dict:
    handlers = self.db.get_handlers()
```

Fetching the handler registry from the database on every job execution is expensive and introduces a DB round-trip inside the claim window (extending the time the lock is held). Handler registries are typically static or change infrequently. Consider loading handlers at `JobExecutor` construction time or caching with a TTL. This also means the handler registry could change between the idempotency check and dispatch, which is a subtle consistency concern.

---

### 6. Minor: `execute()` does not pass `job_id` in the `ClaimError` result

```python
return JobResult(
    claimed=False,
    message="Job currently claimed by another worker"
)
```

`job_id` is available as `req.job_id` here and should be included for traceability, consistent with the other return paths.
