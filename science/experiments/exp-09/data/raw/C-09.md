**Decision: Request Changes**

**Summary:** The acquisition, renewal, and release mechanics are individually correct — atomic NX acquisition, token-verified Lua renewal, token-verified Lua release, and a scoped heartbeat thread are all sound on the surface. However, there is a critical silent failure mode in the gap between renewal and execution: if the renewal thread detects claim loss (returns 0) it simply exits, but the main execution thread has no way to learn this and continues running the job as if it still holds the exclusive claim. Additionally, the test suite has a structural flaw that masks a real exception path. These two issues together mean the system can silently double-execute jobs under adverse timing.

---

## Detailed Findings

### 1. [Critical] Silent Claim Loss — The Gap Between Renewal and Execution

This is the core failure. Walk through the adverse lifecycle:

```
t=0    Worker A acquires claim (TTL=60s)
t=15   Renewal thread fires → renew_script returns 1 (OK)
t=30   Renewal thread fires → Redis blip, renew_script returns 0
         → _renew() logs nothing, returns silently
         → stop_event is NOT set
         → Main thread has no signal
t=31   Worker B acquires the now-expired claim (TTL elapsed or Redis evicted it)
t=45   Worker A is still inside _dispatch(), believes it holds the claim
t=46   Worker A and Worker B both call db.record_execution() concurrently
```

The renewal thread's contract when it detects loss is to `return` — but it returns into the void. The main thread is blocked inside `_dispatch()` or the `db.transaction()` block with no awareness that the exclusive guarantee it was given at `yield token` no longer holds.

```python
# Current — renewal thread silently exits on loss
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # ← exits thread; main thread never knows
        logger.debug("Claim renewed job_id=%s", job_id)
```

The fix requires a shared signal that the main thread actively checks, or that interrupts it. One approach:

```python
claim_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
        except redis.RedisError as e:
            logger.error("Renewal Redis error job_id=%s: %s", job_id, e)
            claim_lost_event.set()
            return
        if result == 0:
            logger.error(
                "Claim lost during execution job_id=%s — another worker may proceed",
                job_id
            )
            claim_lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

But setting the event is not enough on its own. The main thread must check it at a meaningful boundary — after `_dispatch()` returns and before committing to the database:

```python
try:
    yield token
    # Caller must check claim_lost_event before committing
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    self._release_script(keys=[claim_key], args=[token])
```

A cleaner interface surfaces this as an exception. The context manager could raise `ClaimLostError` on exit if the event is set, giving the caller a structured way to abort the transaction rather than commit under a lost claim. The `execute()` method should treat `ClaimLostError` as a hard abort — do not record, do not mark complete, log loudly.

### 2. [Moderate] Renewal Thread Does Not Handle Redis Errors

If the Redis connection drops during a renewal call, `_renew_script()` will raise `redis.RedisError`. That exception propagates out of the `while` loop, terminates the thread with an unhandled exception, and the main thread again has no signal. This is a second path into the same silent-loss failure mode and should be caught explicitly in the renewal thread.

---

### 3. [Moderate] `_dispatch` Exceptions Are Unhandled in `execute()`

If `_dispatch()` raises `ValueError` (unknown job type) or any handler exception, it propagates out of the `with self._claim_job()` block. The `finally` clause correctly releases the claim, but `execute()` has no `except` for this — it re-raises to the caller. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)`, which confirms this is the intended behavior, but it is inconsistent with the rest of the method which returns structured `JobResult` objects. A caller expecting a `JobResult` will get an unhandled exception instead. Either document and enforce the exception contract explicitly, or catch it and return a `JobResult(claimed=False, message="Dispatch failed")`.

---

### 4. [Minor] Renewal Thread Joined With Insufficient Timeout

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)  # 16 seconds
```

If the renewal thread is blocked on a slow Redis call at the moment `stop_event.set()` is called, the join may time out and the release script executes while the renewal thread is still in-flight. This is unlikely to cause correctness issues since the release is token-verified, but the thread becomes a leak. Use a tighter Redis socket timeout on the client, or set a deadline on the script call itself.

---

### 5. [Minor] Test Suite Does Not Exercise the Claim-Loss Path

The test `test_execute_claim_contention` only tests acquisition failure. There is no test that simulates the renewal script returning 0 mid-execution and verifies that the job does not get committed. This is the highest-risk path and it has zero coverage. Once the claim-loss signaling is implemented, a test like this is required:

```python
def test_claim_lost_during_execution_does_not_commit(redis_client, db):
    # Renewal script returns 0 on second call (claim lost mid-execution)
    renew_mock = MagicMock(side_effect=[1, 0])
    redis_client.register_script.return_value = renew_mock
    # ... assert db.record_execution is not called
```

---

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Renewal thread silently exits on claim loss; main thread has no signal and may commit under a stolen claim |
| 2 | Moderate | Redis errors in renewal thread are unhandled and produce the same silent-loss outcome |
| 3 | Moderate | `_dispatch` exceptions escape `execute()` inconsistently with the rest of the method's contract |
| 4 | Minor | Thread join timeout may allow renewal thread to outlive the release call |
| 5 | Minor | No test coverage for mid-execution claim loss — the highest-risk path |

The architecture is sound and the surface-level locking primitives are correct. The single change needed to make this safe is a reliable signal from the renewal thread to the execution path, checked before any state is committed. That gap is the failure mode.
