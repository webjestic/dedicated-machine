Decision: Request Changes

Summary: The distributed locking architecture is sound — atomic Lua scripts, token-verified release, heartbeat renewal, and Pydantic validation are all implemented correctly. However, there are several reliability and correctness issues that need to be addressed before this merges: an unhandled exception path that leaks the claim, a silent failure mode in the renewal thread, and a test that asserts the wrong behavior for an unhandled exception. These aren't edge cases — they're reachable in normal operation.

---

## Detailed Findings

### 🔴 Critical — Claim Leaked on `_dispatch` Exception

```python
# execute() — inside with self._claim_job(req.job_id):
result = self._dispatch(req.job_type, req.payload)
```

`_dispatch` raises `ValueError` for unknown job types. That exception propagates out of the `with self._claim_job(...)` block, which correctly triggers the `finally` clause and releases the Redis lock — **but the exception is never caught in `execute()`**. It bubbles up to the caller as an unhandled exception.

The consequence: the job is neither marked complete nor returned as a clean `JobResult`. The caller gets a raw `ValueError` instead of a structured response. This is inconsistent with every other failure mode in this method, which all return `JobResult`.

**Fix:** Catch `ValueError` (or a more specific `DispatchError`) inside `execute()` and return a `JobResult(claimed=False, ...)`.

```python
except ValueError as e:
    logger.error("Dispatch failed job_id=%s: %s", req.job_id, e)
    return JobResult(claimed=False, job_id=req.job_id, message="Unknown job type")
```

### 🔴 Critical — Test Asserts Wrong Behavior for the Above Bug

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}

    with pytest.raises(ValueError):
        executor.execute("job-001", "unknown_type")
```

This test **encodes the broken behavior as expected behavior**. It asserts that `execute()` raises `ValueError`, but `execute()` is a public API method that should return a `JobResult` in all cases. If the bug above is fixed, this test will fail — which is the right outcome, but the test needs to be rewritten now, not after the fix.

**Fix:** Rewrite the test to assert a structured failure response:

```python
def test_execute_unknown_job_type(executor, db):
    db.get_handlers.return_value = {}
    result = executor.execute("job-001", "unknown_type")
    assert result.claimed is False
    assert result.message  # some informative message
    db.record_execution.assert_not_called()
```

### 🟡 Medium — Renewal Thread Failure Is Silent

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
        if result == 0:
            return  # ← exits silently, no signal to the main thread
        logger.debug("Claim renewed job_id=%s", job_id)
```

If the renewal script returns `0` — meaning another process has taken the key, or it expired — the thread exits quietly. The main thread continues executing the job with no knowledge that the claim has been lost. Another worker could claim and begin executing the same job concurrently.

This is a known hard problem in distributed locking (it's the same class of issue Redlock was designed to address), but the current code doesn't even log a warning or attempt to surface the condition.

**Minimum fix:** Log at `ERROR` level when renewal fails so it's observable in production.

**Better fix:** Set a shared flag (`claim_lost = threading.Event()`) that the main thread checks at a safe yield point, or after `_dispatch` returns, before writing to the database.

```python
claim_lost = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(...)
        if result == 0:
            logger.error("Claim renewal failed — lock lost job_id=%s", job_id)
            claim_lost.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# After _dispatch returns, before writing:
if claim_lost.is_set():
    raise ClaimError(f"Lost claim on job {job_id} during execution")
```

### 🟡 Medium — Database Exceptions Inside the Claim Are Unhandled

```python
existing = self.db.get_execution(req.job_id)
# ...
with self.db.transaction():
    self.db.record_execution(...)
    self.db.mark_job_complete(...)
```

Any database exception here propagates out of `execute()` as an unhandled exception. The claim will be released (the `finally` block runs), but the caller again receives a raw exception instead of a `JobResult`. This should be caught and returned as a structured error, with appropriate logging.

---

### 🟡 Medium — `job_type` Has No Pattern Validation

```python
job_type: str = Field(..., min_length=1, max_length=64)
```

`job_id` has a strict alphanumeric pattern enforced. `job_type` does not. Since `job_type` is used as a dispatch key into `db.get_handlers()`, and potentially logged or stored, it should have equivalent constraints. At minimum, add a pattern that excludes control characters and path separators.

---

### 🟢 Minor — `t.join()` Timeout Is Imprecise

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
```

If the thread is mid-sleep when `stop_event.set()` is called, `stop_event.wait(timeout=RENEWAL_INTERVAL)` will return promptly because the event is set. The `+ 1` buffer is fine in practice, but the comment or a constant would make the intent clearer. Not a blocker.

---

### 🟢 Minor — `payload` Type Annotation Could Be Tighter

```python
payload: dict = Field(default_factory=dict)
```

`dict` accepts any keys and values, including nested structures that could be very large. Consider `Dict[str, Any]` with a max-size validator if payload size is a concern in your deployment context. Low priority but worth noting for a government workload.
