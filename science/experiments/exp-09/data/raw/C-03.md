**Decision: Request Changes**

**Summary:** The surface mechanics are well-constructed — atomic acquisition, token-verified release, Lua-guarded renewal, and a scoped heartbeat thread are all present and correctly wired. However, there is a critical silent failure mode in the gap between renewal and execution: if the renewal thread loses the claim (returns 0) while `_dispatch` or the database transaction is still running, the protected code has no way to learn this and will continue executing to completion, potentially racing with another worker who has since acquired the claim. Additionally, there are several secondary issues in error handling, test coverage, and operational hygiene that need to be addressed before this is production-ready.

---

## Primary Issue: Silent Claim Loss During Execution

This is the core flaw. Walk through the adverse timing scenario:

1. Worker A acquires the claim. `_dispatch` begins — it's slow, maybe 45 seconds.
2. At `t=15s`, the renewal thread fires. Redis is momentarily unreachable (network partition, Redis restart, eviction under `maxmemory` policy). The renewal call raises an exception or returns `0`.
3. The renewal thread currently handles `result == 0` by silently returning — `return` with no signal to the main thread.
4. At `t=60s`, the TTL expires. The claim key is gone.
5. Worker B acquires the claim at `t=61s` and begins executing the same job.
6. Worker A's `_dispatch` finishes at `t=75s`. Worker A enters `db.transaction()` and commits. Worker B does the same shortly after.

The protected code never knew the claim was lost. The idempotency check (`get_execution`) only runs at acquisition time, not before the commit. You now have a double-execution and a race on the database write.

**The renewal thread must signal the main thread when the claim is lost.** The `stop_event` is currently one-directional (main → thread). You need a second event, or a shared flag, so the thread can signal back. The main thread must check this signal at a meaningful point — ideally before committing — and abort if the claim is no longer held.

```python
# Sketch of the required change
claim_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
        except Exception as e:
            logger.error("Renewal script raised during job_id=%s: %s", job_id, e)
            claim_lost_event.set()
            return
        if result == 0:
            logger.error("Claim lost during execution job_id=%s", job_id)
            claim_lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

Then in the `execute` path, before committing:

```python
if claim_lost_event.is_set():
    raise ClaimError(f"Claim was lost during execution of job {job_id}")

with self.db.transaction():
    ...
```

This doesn't fully close the race window (there's still a gap between the check and the commit), but it eliminates the silent failure. A truly airtight solution would require the database commit to be conditioned on Redis state, which is a distributed systems problem — at minimum, document this residual window explicitly.

## Secondary Issues

**1. Renewal exceptions are swallowed entirely**

```python
result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
```

If `redis` raises `ConnectionError`, `TimeoutError`, or any other exception here, the thread dies silently. There is no `try/except` around the renewal call. This is exactly the scenario where the claim loss goes undetected. Wrap it.

**2. `_dispatch` exceptions are unhandled in `execute`**

```python
result = self._dispatch(req.job_type, req.payload)
```

If `_dispatch` raises `ValueError` (unknown job type) or any handler exception, it propagates out of the `with self._claim_job(...)` block, which correctly releases the claim. But the caller gets an unhandled exception rather than a structured `JobResult`. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)` — this means the API contract is inconsistent. Either all failure modes return `JobResult`, or none do. Pick one and enforce it.

**3. The `CLAIM_TTL` of 60 seconds is dangerously short for "slow jobs"**

The comment says "generous for slow jobs," but 60 seconds is not generous. A job that takes 90 seconds will require two successful renewals. If either fails, the claim expires mid-execution. Consider making this configurable at construction time, or at minimum raising it to 300 seconds with a proportionally adjusted renewal interval.

**4. `RENEWAL_INTERVAL` creates a worst-case exposure window of 15 seconds**

If the claim is lost at `t=1s` after a renewal, the thread won't detect it for another 14 seconds. During that window, another worker can acquire and begin executing. This is acceptable if documented, but the `claim_lost_event` mechanism above still won't help if `_dispatch` completes in that window. Document this explicitly.

**5. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` can silently time out**

```python
t.join(timeout=self.RENEWAL_INTERVAL + 1)
self._release_script(keys=[claim_key], args=[token])
```

If the join times out, the thread is still running when `_release_script` fires. The thread could then attempt a renewal after the release, on a key that either no longer exists or has been re-acquired by another worker. Check `t.is_alive()` after the join and log a warning. The daemon flag prevents hard leaks, but the log signal matters for operations.

**6. Test coverage does not cover the gap**

None of the tests simulate claim loss during execution. `test_execute_success` mocks `_renew_script` to return `1` unconditionally. There is no test for:
- Renewal returning `0` mid-execution
- Renewal raising a `ConnectionError`
- The `claim_lost_event` (once added) being checked before commit

These are the exact scenarios this PR is supposed to protect against. The tests validate the happy path and the acquisition-time failure, but not the gap.

**7. Minor: `payload: dict` type hint is too broad**

`JobRequest.payload` is typed as `dict` without key/value constraints. For a government-context system handling arbitrary job parameters, consider `dict[str, Any]` at minimum, with a note that payload validation is the handler's responsibility. Not a blocker, but worth noting.

---

## What Needs to Change Before Approval

1. Implement bidirectional signaling between the renewal thread and the main execution path. The main thread must be able to detect and respond to claim loss before committing.
2. Wrap the renewal script call in a `try/except` and treat exceptions as claim loss.
3. Resolve the inconsistent exception vs. `JobResult` contract for `_dispatch` failures.
4. Add tests that simulate renewal failure during execution and verify the system does not commit.
5. Check and log `t.is_alive()` after the join timeout.

The architecture is sound and the author clearly understands distributed locking. The gap scenario is a known hard problem and the solution is close — it just needs the feedback channel from the renewal thread to the execution path wired up and tested.
