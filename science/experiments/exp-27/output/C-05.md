## Pre-Implementation Audit: What the Requirements Don't Say

Before building, here are the gaps a production deployment requires that your request doesn't mention. I'll address some and explicitly flag others.

**Addressed in this implementation:**
- Persistent storage (not just in-memory chain)
- P2P networking with peer discovery
- Mempool with fee-based transaction ordering
- Fork resolution (longest valid chain wins)
- Wallet key management (ECDSA secp256k1)
- Transaction signature verification
- UTXO model vs account model (I'll use account model with nonces to prevent replay attacks)
- Mining difficulty adjustment
- API for external interaction
- Input validation and error handling throughout

**Flagged for requester — NOT addressed (would require product decisions):**
- **Finality model**: This uses Nakamoto consensus (probabilistic finality). If you need deterministic finality, you need BFT/PBFT/Tendermint-style consensus — different architecture entirely.
- **Sybil resistance in production**: The P2P layer here has no peer authentication. Production requires peer identity, reputation, and connection limits.
- **Key storage security**: Private keys are written to disk unencrypted. Production requires HSM integration or at minimum AES-256 encrypted keystores with KDF (scrypt/argon2).
- **DoS protection**: No rate limiting on API or P2P message handling.
- **Checkpoint/snapshot system**: Long chain sync is O(n) block validation with no snapshots.
- **MEV/transaction ordering fairness**: Mempool is fee-ordered; this enables front-running.
- **Regulatory compliance**: No built-in transaction monitoring, AML hooks, or audit logging.
- **Smart contract execution**: Not included — would require a VM (EVM-compatible or custom).

  const n = parseInt(value ?? defaultVal, 10);

## src/crypto/hash.js

```javascript
'use strict';

const crypto = require('crypto');

/**
 * All hashing goes through this module.
 *
 * We use SHA-256 double-hash (SHA-256d) for block and transaction hashes,
 * matching Bitcoin's approach. The double hash provides protection against
 * length-extension attacks. For a production system you'd evaluate whether
 * SHA-3 or BLAKE3 better fits your threat model and performance requirements.
 */

/**
 * Compute SHA-256d (double SHA-256) of a string or Buffer.
 * Returns a hex string.
 */
function sha256d(data) {
  if (typeof data !== 'string' && !Buffer.isBuffer(data)) {
    throw new TypeError(`sha256d expects string or Buffer, got ${typeof data}`);
  }
  const first = crypto.createHash('sha256').update(data).digest();
  return crypto.createHash('sha256').update(first).digest('hex');
}

/**
 * Compute a single SHA-256 of arbitrary data.
 * Used for signatures where double-hash is not required.
 */
function sha256(data) {
  return crypto.createHash('sha256').update(data).digest('hex');
}

/**
 * Deterministic JSON serialization for hashing.
 * Standard JSON.stringify does not guarantee key ordering.
 * We sort keys recursively so the same logical object always produces
 * the same hash regardless of insertion order.
 */
function deterministicStringify(obj) {
  if (obj === null || typeof obj !== 'object' || Array.isArray(obj)) {
    return JSON.stringify(obj);
  }
  const sortedKeys = Object.keys(obj).sort();
  const parts = sortedKeys.map(k => {
    return `${JSON.stringify(k)}:${deterministicStringify(obj[k])}`;
  });
  return `{${parts.join(',')}}`;
}

module.exports = { sha256d, sha256, deterministicStringify };
```

## src/crypto/wallet.js

```javascript
'use strict';

const fs = require('fs');
const path = require('path');
const { ec: EC } = require('elliptic');
const { sha256, sha256d, deterministicStringify } = require('./hash');
const config = require('../../config');

/**
 * Wallet implementation using ECDSA over secp256k1 — the same curve used
 * by Bitcoin and Ethereum. This gives us compatibility with existing tooling
 * and a well-audited implementation via the `elliptic` library.
 *
 * PRODUCTION GAP (flagged in header): Private keys are stored as plaintext JSON.
 * Production requires encrypted keystores. The standard approach is:
 *   1. Derive an encryption key from a passphrase using scrypt or argon2id
 *   2. Encrypt the private key with AES-256-GCM
 *   3. Store: { salt, iv, ciphertext, mac } — never the raw key
 * We stub the interface so this can be swapped in without changing callers.
 */

const ec = new EC('secp256k1');

class Wallet {
  constructor(keyPair, address) {
    this._keyPair = keyPair;       // elliptic KeyPair — never expose directly
    this.address = address;        // public hex string used as account identifier
    this.publicKey = keyPair.getPublic('hex');
  }

  /**
   * Sign arbitrary data. Returns DER-encoded signature as hex.
   * We hash the data before signing so signature size is fixed regardless
   * of input size, and to match standard ECDSA usage.
   */
  sign(data) {
    const hash = sha256(
      typeof data === 'string' ? data : deterministicStringify(data)
    );
    const signature = this._keyPair.sign(hash);
    return signature.toDER('hex');
  }

  toJSON() {
    return {
      address: this.address,
      publicKey: this.publicKey,
      // PRODUCTION GAP: this should be encrypted
      privateKey: this._keyPair.getPrivate('hex'),
    };
  }

  /**
   * Generate a new wallet with a fresh secp256k1 key pair.
   * Address is derived as the last 40 hex chars of SHA-256(publicKey),
   * prefixed with '0x'. Production would use a proper address derivation
   * scheme (e.g., Keccak-256 of public key, checksum encoding per EIP-55).
   */
  static generate() {
    const keyPair = ec.genKeyPair();
    const publicKey = keyPair.getPublic('hex');
    const address = '0x' + sha256(publicKey).slice(-40);
    return new Wallet(keyPair, address);
  }

  /**
   * Restore a wallet from a hex private key string.
   */
  static fromPrivateKey(privateKeyHex) {
    if (typeof privateKeyHex !== 'string' || !/^[0-9a-fA-F]{64}$/.test(privateKeyHex)) {
      throw new Error('Invalid private key: must be 64 hex characters');
    }
    const keyPair = ec.keyFromPrivate(privateKeyHex, 'hex');
    const publicKey = keyPair.getPublic('hex');
    const address = '0x' + sha256(publicKey).slice(-40);
    return new Wallet(keyPair, address);
  }

  /**
   * Save wallet to disk. See PRODUCTION GAP above — this needs encryption.
   */
  save() {
    if (!fs.existsSync(config.WALLET_DIR)) {
      fs.mkdirSync(config.WALLET_DIR, { recursive: true });
    }
    const filePath = path.join(config.WALLET_DIR, `${this.address}.json`);
    fs.writeFileSync(filePath, JSON.stringify(this.toJSON(), null, 2));
    return filePath;
  }

  /**
   * Load a wallet from disk by address.
   */
  static load(address) {
    const filePath = path.join(config.WALLET_DIR, `${address}.json`);
    if (!fs.existsSync(filePath)) {
      throw new Error(`Wallet not found: ${address}`);
    }
    const data = JSON.parse(fs.readFileSync(filePath, 'utf8'));
    return Wallet.fromPrivateKey(data.privateKey);
  }

  /**
   * List all wallet addresses stored on disk.
   */
  static listAddresses() {
    if (!fs.existsSync(config.WALLET_DIR)) return [];
    return fs.readdirSync(config.WALLET_DIR)
      .filter(f => f.endsWith('.json'))
      .map(f => f.replace('.json', ''));
  }
}

/**
 * Verify a signature against a public key and data.
 * This is used during transaction validation and is called on every node —
 * it must never throw; return false for any invalid input.
 */
function verifySignature(publicKeyHex, data, signatureHex) {
  try {
    if (!publicKeyHex || !data || !signatureHex) return false;
    const key = ec.keyFromPublic(publicKeyHex, 'hex');
    const hash = sha256(
      typeof data === 'string' ? data : deterministicStringify(data)
    );
    return key.verify(hash, signatureHex);
  } catch {
    // Any malformed input returns false — never propagate crypto exceptions
    return false;
  }
}

module.exports = { Wallet, verifySignature };
```

 *

 * - Simpler to reason about balances

 * signed transaction could be rebroadcast by anyone. The nonce must equal
 * the sender's current transaction count on-chain. Nodes reject transactions
