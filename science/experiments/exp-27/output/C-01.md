# Production Blockchain Implementation in Node.js

Let me start by identifying what a production deployment actually requires that typical "build me a blockchain" requirements omit.

## Gap Analysis: Unstated Production Requirements

Before writing a line of code, here is what production blockchain infrastructure requires that you did not specify:

**Persistence & Recovery**
- Chain state must survive process restarts (LevelDB/RocksDB, not in-memory)
- WAL (write-ahead log) for crash recovery mid-block
- Snapshot + replay architecture for fast node sync

**Network Layer**
- Peer discovery (DNS seeds, mDNS, DHT, or static peers)
- Gossip protocol for block and transaction propagation
- NAT traversal (UPNP, hole punching)
- Eclipse attack resistance (peer diversity enforcement)

**Mempool**
- Fee-based transaction prioritization
- Replace-by-fee (RBF)
- Mempool eviction under memory pressure
- DoS protection (rate limiting per peer/address)

**Consensus Safety**
- Fork choice rule (longest chain / most work)
- Finality mechanism or explicit probabilistic finality guarantees
- Nothing-at-stake / long-range attack mitigations (if PoS)

**Cryptographic Infrastructure**
- Key derivation (BIP32/BIP39 or equivalent)
- Signature malleability protection
- Replay attack protection (chain ID in signatures)

**Operational**
- Structured logging with log levels
- Prometheus metrics endpoint
- Health check endpoint
- Graceful shutdown with state flush
- Configuration management (env + file, no hardcoded secrets)

**Security**
- Input validation on every external boundary
- RPC authentication
- Rate limiting on all public endpoints
- Memory limits on untrusted data structures

**What I will build and what I will flag:**
I will implement a complete, working system with persistence, P2P networking, mempool, PoW consensus, wallet/signing, and an RPC API. I will explicitly flag the gaps I do not implement inline.

### `src/crypto/hash.js`

```javascript
// src/crypto/hash.js
// All hashing goes through this module. Centralizing here means we can
// swap SHA-256 for a different primitive in one place if needed.

'use strict';

const crypto = require('crypto');

/**
 * SHA-256 hash of arbitrary data.
 * Always returns a hex string.
 * @param {string|Buffer} data
 * @returns {string} hex-encoded hash
 */
function sha256(data) {
  return crypto
    .createHash('sha256')
    .update(data)
    .digest('hex');
}

/**
 * Double SHA-256 (SHA-256d) - used for block and transaction IDs,
 * matching Bitcoin's approach to prevent length-extension attacks.
 * @param {string|Buffer} data
 * @returns {string} hex-encoded hash
 */
function sha256d(data) {
  const first = crypto.createHash('sha256').update(data).digest();
  return crypto.createHash('sha256').update(first).digest('hex');
}

/**
 * RIPEMD-160(SHA-256(data)) - used for address derivation.
 * @param {Buffer} data
 * @returns {string} hex-encoded hash
 */
function hash160(data) {
  const sha = crypto.createHash('sha256').update(data).digest();
  return crypto.createHash('ripemd160').update(sha).digest('hex');
}

/**
 * Compute a Merkle root from an array of transaction IDs.
 * Returns the zero hash for an empty array.
 * Uses SHA-256d at each level, matching Bitcoin's Merkle tree construction.
 *
 * PRODUCTION NOTE: This implementation is vulnerable to the CVE-2012-2459
 * duplicate-transaction Merkle tree attack if callers do not deduplicate txids
 * before calling. The blockchain.js layer enforces uniqueness before calling here.
 *
 * @param {string[]} txids - array of hex transaction IDs
 * @returns {string} hex Merkle root
 */
function merkleRoot(txids) {
  if (txids.length === 0) {
    return '0'.repeat(64);
  }

  let level = [...txids];

  while (level.length > 1) {
    const next = [];
    for (let i = 0; i < level.length; i += 2) {
      // If odd number of elements, duplicate the last one (standard Merkle behavior)
      const left = level[i];
      const right = level[i + 1] || level[i];
      next.push(sha256d(left + right));
    }
    level = next;
  }

  return level[0];
}

module.exports = { sha256, sha256d, hash160, merkleRoot };
```

### `src/crypto/keys.js`

```javascript
// src/crypto/keys.js
// Elliptic curve key generation, signing, and verification.
// Uses secp256k1 - the same curve as Bitcoin and Ethereum.
//
// PRODUCTION GAP: This does not implement BIP32 HD wallets or BIP39 mnemonics.
// A production wallet needs hierarchical deterministic key derivation so users
// can back up a single seed phrase and recover all keys.
//
// PRODUCTION GAP: Private keys are held in process memory as hex strings.
// A production system should use hardware security modules (HSM) or at minimum
// encrypted key files with the encryption key derived from a passphrase (PBKDF2/Argon2).

'use strict';

const EC = require('elliptic').ec;
const { hash160 } = require('./hash');

const ec = new EC('secp256k1');

/**
 * Generate a new secp256k1 key pair.
 * @returns {{ privateKey: string, publicKey: string, address: string }}
 */
function generateKeyPair() {
  const keyPair = ec.genKeyPair();
  const privateKey = keyPair.getPrivate('hex');
  const publicKey = keyPair.getPublic(true, 'hex'); // compressed
  const address = publicKeyToAddress(publicKey);
  return { privateKey, publicKey, address };
}

/**
 * Derive a public key from a private key.
 * @param {string} privateKeyHex
 * @returns {string} compressed public key hex
 */
function privateKeyToPublicKey(privateKeyHex) {
  const keyPair = ec.keyFromPrivate(privateKeyHex, 'hex');
  return keyPair.getPublic(true, 'hex');
}

/**
 * Derive a blockchain address from a compressed public key.
 * Address = "0x" + RIPEMD160(SHA256(publicKey)) in hex.
 * This matches Bitcoin's P2PKH address derivation (without Base58Check encoding).
 *
 * PRODUCTION GAP: A production system should use Base58Check or Bech32 encoding
 * to make addresses human-readable and include a checksum to catch typos.
 *
 * @param {string} publicKeyHex - compressed public key
 * @returns {string} address string
 */
function publicKeyToAddress(publicKeyHex) {
  const pubKeyBuffer = Buffer.from(publicKeyHex, 'hex');
  return '0x' + hash160(pubKeyBuffer);
}

/**
 * Sign arbitrary data with a private key.
 * The data is hashed with SHA-256 before signing (sign the hash, not the data).
 *
 * Returns a DER-encoded signature as hex.
 *
 * PRODUCTION NOTE: Raw ECDSA signatures are malleable - a third party can produce
 * a second valid signature for the same message without knowing the private key.
 * Bitcoin fixed this with BIP66 (strict DER encoding) and BIP62 (low-S normalization).
 * We enforce low-S here to prevent malleability.
 *
 * @param {string} dataHex - hex string of data to sign (typically a tx hash)
 * @param {string} privateKeyHex
 * @returns {string} DER signature hex
 */
function sign(dataHex, privateKeyHex) {
  const keyPair = ec.keyFromPrivate(privateKeyHex, 'hex');
  const msgBuffer = Buffer.from(dataHex, 'hex');

  const signature = keyPair.sign(msgBuffer, { canonical: true }); // canonical = low-S
  return Buffer.from(signature.toDER()).toString('hex');
}

/**
 * Verify a DER signature against data and a public key.
 * @param {string} dataHex - the original data that was signed (hex)
 * @param {string} signatureHex - DER signature hex
 * @param {string} publicKeyHex - compressed public key hex
 * @returns {boolean}
 */
function verify(dataHex, signatureHex, publicKeyHex) {
  try {
    const keyPair = ec.keyFromPublic(publicKeyHex, 'hex');
    const msgBuffer = Buffer.from(dataHex, 'hex');
    const sigBuffer = Buffer.from(signatureHex, 'hex');
    return keyPair.verify(msgBuffer, sigBuffer);
  } catch {
    // Any malformed input returns false rather than throwing
    return false;
  }
}

module.exports = {
  generateKeyPair,
  privateKeyToPublicKey,
  publicKeyToAddress,
  sign,
  verify,
};
```

// Transaction model:

//   - Each input references a previous output by (txid, outputIndex)
