**Decision: Request Changes**

**Summary:** This PR uses `requests.post()` — a synchronous, blocking HTTP call — inside an `async def` FastAPI route. Under load, every concurrent request to this endpoint will stall the event loop for the full duration of the external HTTP call (up to the 5-second timeout), meaning a notification service slowdown becomes an application-wide freeze across all routes, not just this one. Replace `requests` with `httpx.AsyncClient` (or `aiohttp`) and `await` the call. The tests pass because `TestClient` runs synchronously and will never surface this failure mode — the bug is invisible until production load hits it.

**Required change:**

```python
# Replace this import
import requests

# With this
import httpx

# Replace the blocking call
response = requests.post("https://notifications.internal/send", json={...}, timeout=5)

# With an awaited async call
async with httpx.AsyncClient() as client:
    response = await client.post(
        "https://notifications.internal/send",
        json={...},
        timeout=5,
    )
```

**Secondary issues to address before approval:**

1. **Exception handling is too broad in the test.** `test_send_notification_service_unavailable` patches with a bare `Exception`, but the production handler only catches `requests.RequestException` (or with httpx, `httpx.HTTPError`). An uncaught generic exception would return a 500, not a 503 — the test is asserting behavior the code doesn't actually guarantee. Patch with the correct exception type.

2. **No input sanitization on `message`.** The `message` field is passed directly into an outbound request payload with no length cap or content constraints. At minimum, add a `max_length` constraint in the Pydantic model to prevent oversized payloads being forwarded to the internal service.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=1000)
```

3. **`httpx.AsyncClient` should be a lifespan-managed singleton**, not instantiated per-request. Per-request client creation skips connection pooling entirely, which adds latency and risks connection exhaustion. Wire it through FastAPI's lifespan context or dependency injection.

The checklist says tests pass — they do, and that's precisely the problem. Passing tests are not evidence of correctness when the test harness is synchronous and the failure mode is concurrency-dependent.
