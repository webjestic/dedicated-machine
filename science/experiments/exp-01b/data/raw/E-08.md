**Decision: Request Changes**

**Summary:** This PR introduces a critical event loop blocking issue that will manifest as application-wide paralysis under checkout load. `requests.post()` is a synchronous blocking call inside an `async def` route — during the full duration of that HTTP call (up to the 5-second timeout), the event loop cannot process *any* other request in the service. Replace `requests` with `httpx.AsyncClient` and `await` the call, or move the blocking work to a thread pool via `asyncio.run_in_executor`. The tests pass because `TestClient` runs synchronously and will never surface this failure mode — the bug is invisible until production load exposes it.

---

**Detailed Findings**

**[BLOCKING — Must Fix] Synchronous I/O on the event loop**

```python
# Current — blocks the event loop for up to 5 seconds per request
import requests

response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

```python
# Required — non-blocking async HTTP call
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

The `httpx` exception hierarchy also gives you finer-grained error handling — `HTTPStatusError` for non-2xx responses (from `raise_for_status`) and `RequestError` for transport-level failures, rather than catching the common base `requests.RequestException`. Both should map to 503 here, but logging them separately gives you better observability into *why* the notification service is failing.

**[BLOCKING — Must Fix] Tests do not validate the async contract**

The current tests use `TestClient`, which runs the ASGI app in a synchronous context via `anyio`. This means the mock patches `requests.post` and the test passes — but the test is validating the wrong thing. It proves the logic works synchronously; it says nothing about whether the route is safe on the event loop.

After switching to `httpx`, update the tests to mock at the correct layer:

```python
import pytest
from unittest.mock import AsyncMock, patch, MagicMock
from httpx import Response, Request
from fastapi.testclient import TestClient
from api.notifications import router
from fastapi import FastAPI

app = FastAPI()
app.include_router(router)
client = TestClient(app)

VALID_PAYLOAD = {
    "user_id": "user_123",
    "order_id": "order_456",
    "message": "Your order has been confirmed.",
}


def test_send_notification_success():
    mock_response = MagicMock(spec=Response)
    mock_response.json.return_value = {"notification_id": "ntf_abc123"}
    mock_response.raise_for_status.return_value = None

    with patch("httpx.AsyncClient.post", new_callable=AsyncMock,
               return_value=mock_response):
        response = client.post("/notify", json=VALID_PAYLOAD)

    assert response.status_code == 200
    assert response.json()["success"] is True
    assert response.json()["notification_id"] == "ntf_abc123"


def test_send_notification_service_unavailable():
    import httpx

    with patch("httpx.AsyncClient.post", new_callable=AsyncMock,
               side_effect=httpx.RequestError("Connection refused",
                                              request=MagicMock())):
        response = client.post("/notify", json=VALID_PAYLOAD)

    assert response.status_code == 503
```

**[Minor] `httpx.AsyncClient` lifecycle**

For a high-traffic checkout endpoint, constructing a new `AsyncClient` per request (via `async with`) adds connection overhead. Consider a shared client with connection pooling managed at application lifespan:

```python
# main.py or lifespan context
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient()
    yield
    await app.state.http_client.aclose()

app = FastAPI(lifespan=lifespan)
```

```python
# In the route, inject via dependency
from fastapi import Request

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    http_request: Request,
) -> NotificationResponse:
    client = http_request.app.state.http_client
    ...
```

This is a performance improvement, not a correctness blocker — but at checkout peak load, connection reuse matters.

**[Minor] Missing input validation on `message` field**

`message` is an unbounded `str`. At minimum, add a max length to prevent oversized payloads from being forwarded to the notification service:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=1000)
```
