## Review

**Decision:** Request Changes

**Summary:** The endpoint is declared `async` but uses `requests.post`, a synchronous blocking call, directly on the event loop. Under checkout load, every concurrent request across the entire application will be starved for up to 5 seconds per invocation — exactly the blast radius described. This must be replaced with `httpx.AsyncClient` (or equivalent) with `await`, or the handler must be converted to a plain `def` function so FastAPI dispatches it to a thread pool. There are also secondary issues that need to be addressed before this is mergeable.

---

### Critical — Blocking I/O on the Event Loop

```python
# CURRENT — blocks the entire event loop
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)   # synchronous, blocks all concurrent requests
```

```python
# REQUIRED — async I/O with httpx
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
```

Alternatively, drop `async` and use a plain `def` handler — FastAPI will run it in a thread pool executor — but the `httpx` approach is preferred for a service at this traffic level.

---

### Secondary Issues

**1. Unvalidated fields allow unbounded input**

`user_id`, `order_id`, and `message` have no length or format constraints. A caller can send multi-megabyte strings that get forwarded to the downstream service. Add explicit constraints:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

**2. `notification_id` is accessed without a guard**

If the downstream service returns 200 but omits `notification_id`, this raises an unhandled `KeyError` that leaks as a 500:

```python
# CURRENT
notification_id=data["notification_id"]

# SAFER
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned no notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

**3. Exception catch in the test is too broad**

`test_send_notification_service_unavailable` patches with a bare `Exception`, but the handler only catches `requests.RequestException` (or, after the fix, `httpx.HTTPError`). The test passes today by accident — the `TestClient` swallows the uncaught exception and returns 500, which the assertion doesn't check. The test should use the correct exception type and assert the exact status code:

```python
# After migrating to httpx:
with patch("api.notifications.httpx.AsyncClient.post",
           side_effect=httpx.RequestError("Connection refused")):
    response = client.post("/notify", json={...})

assert response.status_code == 503
```

**4. No test for malformed downstream response**

Add a test covering the case where the downstream service returns 200 with a body that is missing `notification_id`. This is the scenario that would currently produce an unhandled 500.

### Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | **Critical** | Replace `requests.post` with `await httpx.AsyncClient.post` |
| 2 | Medium | Add field-level validation constraints to `NotificationRequest` |
| 3 | Medium | Guard against missing `notification_id` in downstream response |
| 4 | Low | Fix test exception type and add missing coverage for bad downstream response |
