**Decision: Request Changes**

**Summary:** This PR introduces a critical concurrency defect that directly contradicts the operational constraints of this service. The `async` endpoint calls `requests.post` — a synchronous, blocking I/O call — which blocks the entire event loop for up to 5 seconds per request under the exact high-load checkout conditions described. This must be replaced with an `httpx.AsyncClient` call using `await` before this can merge. There are also secondary issues: the `message` field accepts unbounded, unvalidated user-supplied content that is forwarded to an internal service, which is a correctness and potential security concern; and the test suite patches `requests.post` rather than the async client, meaning the tests will need to be rewritten alongside the fix.

---

**Detailed notes for the author:**

**[Blocking — Must Fix] Synchronous I/O on the async event loop**

```python
# Current — blocks the event loop
async def send_notification(...):
    response = requests.post(...)  # synchronous, holds the loop for up to 5s
```

```python
# Required — non-blocking
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={...},
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"]
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service HTTP error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service connection error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

For production, prefer a shared `httpx.AsyncClient` instance (e.g., created at app startup via lifespan) rather than constructing one per request, to benefit from connection pooling.

---

**[Blocking — Must Fix] Unvalidated `message` field forwarded to internal service**

`message: str` has no length or content constraints. User-supplied content is forwarded verbatim to `notifications.internal`. At minimum, add a length cap. Depending on what the downstream service does with this field (renders it in email/SMS, logs it, etc.), this warrants a closer look at injection risk.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

---

**[Must Fix] Tests validate the wrong thing**

The test patches `requests.post`, which won't exist in the corrected implementation. After switching to `httpx`, update the tests to patch `httpx.AsyncClient` or use `httpx`'s built-in `MockTransport` / `respx` for cleaner async mocking. Also note that the error-path test patches with a bare `Exception` rather than a `requests.RequestException` — the current handler would actually catch this, but it's testing the wrong contract.

---

**[Minor] `notification_id` key access is unguarded**

```python
notification_id=data["notification_id"]  # KeyError if upstream omits this field
```

Use `.get()` with a fallback or handle the `KeyError` explicitly to avoid a 500 leaking to the client.
