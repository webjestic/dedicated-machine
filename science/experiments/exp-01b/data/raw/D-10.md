Decision: Request Changes

Summary: The implementation has a critical correctness issue — it uses the synchronous `requests` library inside an `async` endpoint, which will block the event loop and degrade performance under load. There are also meaningful gaps in input validation, error handling, and test coverage that need to be addressed before this is ready to merge.

---

## Detailed Findings

### 1. Performance — Blocking I/O in Async Context (Critical)

**File:** `api/notifications.py`

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

`requests.post()` is synchronous. Calling it inside an `async` def without offloading it will block the entire event loop for the duration of the network call, serializing all concurrent requests. Replace with `httpx` (async-native) or `aiohttp`:

```python
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
```

### 2. Error Handling — Unguarded Key Access (High)

**File:** `api/notifications.py`

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if key is absent
)
```

If the external service returns a 200 but omits `notification_id` (malformed response, API change, etc.), this raises an unhandled `KeyError` that will surface as a 500. Handle it explicitly:

```python
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(
        status_code=502,
        detail="Notification service returned an unexpected response",
    )
```

Note the status code distinction: **503** = service unreachable, **502** = service reachable but misbehaving. Using them correctly helps operators triage incidents faster.

---

### 3. Input Validation — No Constraints on Fields (Medium)

**File:** `api/notifications.py`

The Pydantic model accepts any string for all three fields, including empty strings and arbitrarily long payloads. At minimum:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

Without length limits, a caller can send a multi-megabyte `message` field that gets forwarded to the external service, which is both a potential abuse vector and a source of unexpected downstream errors.

---

### 4. Security — Unvalidated Message Content Forwarded to External Service (Medium)

The `message` field is accepted from the caller and forwarded verbatim to the external notification service. Depending on how that service renders the message (email HTML, SMS, push notification), this could be a vector for injection (HTML injection, homograph attacks, etc.). Consider:

- Restricting `message` to a predefined set of template keys rather than free-form text, or
- Sanitizing/escaping the field before forwarding if free-form input is a requirement.

At minimum, document the trust assumption explicitly so future maintainers don't widen the surface unknowingly.

---

### 5. Test Coverage — Gaps (Medium)

**File:** `tests/test_notifications.py`

Several cases are untested:

| Missing Test Case | Why It Matters |
|---|---|
| External service returns 200 but missing `notification_id` | Exercises the unguarded key access bug above |
| External service returns a 4xx/5xx HTTP status | `raise_for_status()` path needs explicit coverage |
| Empty or invalid request body (e.g., `user_id: ""`) | Validates Pydantic rejection behavior |
| `requests.RequestException` subclass (not bare `Exception`) | The current failure test patches `Exception`, not `requests.RequestException`, so it doesn't accurately reflect the actual except clause |

On that last point — the failure test is technically wrong:

```python
# Current — patches with a bare Exception, which the except clause doesn't catch
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The handler only catches `requests.RequestException`. A bare `Exception` would propagate as a 500, not a 503. The test passes today only because `TestClient` may be masking this. Fix:

```python
import requests as requests_lib

with patch("api.notifications.requests.post",
           side_effect=requests_lib.RequestException("Connection refused")):
```

---

### 6. Correctness — Hard-Coded Internal URL (Low)

```python
"https://notifications.internal/send"
```

Hard-coding a service URL makes this untestable in isolation without mocking and undeployable across environments without code changes. Move it to configuration:

```python
# config.py or environment variable
NOTIFICATION_SERVICE_URL = os.getenv("NOTIFICATION_SERVICE_URL", "https://notifications.internal/send")
```

---

### 7. Documentation — Acceptable

The docstring is minimal but sufficient for a straightforward endpoint. No changes required here, though noting the trust assumption on `message` (per point 4) would be worthwhile.

---

## Summary of Required Changes

| # | Severity | Issue |
|---|---|---|
| 1 | Critical | Replace `requests` with `httpx` async client |
| 2 | High | Guard against missing `notification_id` in response |
| 3 | Medium | Add field length constraints to `NotificationRequest` |
| 4 | Medium | Document or restrict free-form `message` forwarding |
| 5 | Medium | Fix incorrect exception type in failure test; add missing test cases |
| 6 | Low | Externalize the notification service URL |
