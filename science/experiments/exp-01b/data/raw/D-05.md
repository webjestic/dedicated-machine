Decision: Request Changes

Summary: The implementation has a critical correctness issue — it uses the synchronous `requests` library inside an `async` endpoint, which will block the event loop and degrade performance under load. There are also meaningful gaps in input validation, error handling, and test coverage that need to be addressed before this is safe to merge into a checkout flow.

---

## Detailed Findings

### 1. Correctness

**🔴 Blocking I/O in async endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

Using `requests` inside an `async` def stalls the entire event loop for the duration of the HTTP call. Under any meaningful concurrency, this will cause latency spikes across all endpoints, not just this one.

```python
# Recommended — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

**🟡 Missing key access guard on `data["notification_id"]`**

If the external service returns a 200 but omits `notification_id`, this raises an unhandled `KeyError`, which will surface as a 500 with no useful log message.

```python
# Safer
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

### 2. Security

**🔴 `message` field is unbounded and unvalidated**

The `message` field is passed directly to an external service with no length limit or content constraints. This is a potential vector for payload injection or abuse.

```python
# Add constraints via Pydantic
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

**🟡 Hardcoded internal service URL**

```python
"https://notifications.internal/send"  # hardcoded
```

This should come from configuration (e.g., `settings.NOTIFICATION_SERVICE_URL`). Hardcoded URLs make environment promotion fragile and prevent secret rotation without a code change.

**🟡 No authentication on the outbound request**

If `notifications.internal` is a real internal service, the call should include an auth header (API key, service token, etc.). Sending unauthenticated requests to internal services is a common lateral-movement risk.

---

### 3. Error Handling

**🟡 Overly broad exception catch in tests**

```python
# Test patches with a generic Exception, not requests.RequestException
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`. The test should use the same exception type to actually validate the handler, not just any exception. As written, the test would pass even if the `except` clause were accidentally removed, since `Exception` is broader and would still be caught by FastAPI's default handler.

```python
import requests
side_effect=requests.exceptions.ConnectionError("Connection refused")
```

**🟡 No handling for non-503 HTTP errors from the upstream**

`raise_for_status()` will raise on any 4xx or 5xx. A 400 from the notification service (bad request) should probably not be surfaced to the caller as a 503 (service unavailable) — that's misleading. Consider distinguishing upstream client errors from upstream server errors.

---

### 4. Test Coverage

**🟡 Missing test cases**

The following scenarios have no coverage:

| Scenario | Why it matters |
|---|---|
| Missing `notification_id` in response | Guards the `KeyError` risk above |
| Pydantic validation rejection (e.g., empty `user_id`) | Confirms input constraints work |
| Upstream returns 4xx (e.g., 400 Bad Request) | Validates error discrimination |
| Upstream returns malformed JSON | `response.json()` can raise |

---

### 5. Performance

**🔴 (Covered above)** — Blocking `requests` call in async context is the primary concern.

**🟡 No connection pooling**

If `httpx` is adopted, instantiate the client at module level or use dependency injection rather than creating a new `AsyncClient` per request. Creating a new client per call skips connection reuse.

```python
# Module-level client (remember to close on shutdown)
_http_client = httpx.AsyncClient()

@router.on_event("shutdown")
async def shutdown():
    await _http_client.aclose()
```

---

### 6. Naming / Documentation

**✅** Names are clear and consistent. The docstring is minimal but adequate for a simple endpoint.

**🟡 Minor:** `request` shadows the FastAPI `Request` type name. Not a bug here since `Request` isn't imported, but consider `notification_request` or `payload` to avoid future confusion.
