**Decision:** Request Changes

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` FastAPI route, which will block the event loop for the full duration of every external HTTP call — up to 5 seconds per request under the configured timeout. Under checkout load, this will starve every other in-flight request in the application simultaneously. The fix is straightforward, but it is a hard blocker. There are also secondary issues in the test suite and error handling that need to be addressed before this merges.

---

## Detailed Review

### 1. 🚨 Critical — Synchronous I/O Inside Async Route (Correctness / Performance)

```python
# api/notifications.py

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(   # ← blocks the event loop
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

`requests` is a synchronous library. When this line executes, it seizes the thread running the event loop and holds it — no `await`, no yield, no opportunity for the loop to service any other coroutine — for up to 5 seconds. This is the canonical event loop starvation pattern. It passes every test (including the ones in this PR, because `TestClient` runs synchronously), ships cleanly, and then degrades the entire application under real concurrent load.

**Required fix:** Replace `requests` with `httpx` in async mode. The change is minimal:

```python
# api/notifications.py

import httpx  # replaces requests
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(   # ← yields control back to the event loop
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

> **Note on client lifecycle:** Creating a new `AsyncClient` per request is acceptable for correctness and is the right starting point. If profiling later shows connection overhead is significant under load, the client can be lifted to application lifespan scope using FastAPI's `lifespan` parameter and shared via dependency injection. Do not do that prematurely — but do not let it block this fix either.

### 2. ⚠️ Error Handling — Overly Broad Exception in Tests, Silent Data Failure in Production (Error Handling)

**In the test:**

```python
# tests/test_notifications.py

with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # ← too broad
```

The production code catches `requests.RequestException`. This test raises a bare `Exception`, which is not a subclass of `RequestException`. The test is asserting the right status code (503), but it is doing so because the uncaught `Exception` propagates up and FastAPI returns a 500 — not because the application's error handling path executed correctly. The test is passing for the wrong reason. With the `httpx` migration, use `httpx.RequestError` as the side effect.

**In the production code:**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]   # ← unguarded key access
)
```

If the external service returns a 200 but omits `notification_id` from the response body, this raises an unhandled `KeyError`, which FastAPI will surface as a 500. The contract with the external service should be validated explicitly:

```python
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service response missing notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

Note the status code distinction: 503 means the service is unavailable; 502 means the service responded but with something unexpected. Using the right code here makes downstream debugging significantly easier.

### 3. ⚠️ Test Coverage — Happy Path Only, No Input Validation Tests (Test Coverage)

The test suite covers one success case and one failure case. Missing coverage:

| Scenario | Covered |
|---|---|
| Successful notification | ✅ |
| Service connection failure | ✅ (but broken — see above) |
| Service returns non-2xx (e.g. 400, 500) | ❌ |
| Service returns 200 with missing `notification_id` | ❌ |
| Missing required request fields (`user_id`, `order_id`, `message`) | ❌ |
| Empty string field values | ❌ |

The Pydantic model validates that fields are present and are strings, but it does not validate that they are non-empty. `user_id: ""` will pass validation today. Whether that matters depends on what the notification service does with it, but it is worth a deliberate decision either way — add `min_length=1` constraints or add a test that documents the current behavior as intentional.

### 4. Minor — `message` Field Has No Length Constraint (Security / Correctness)

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str   # ← unbounded
```

`message` is forwarded directly to an external service with no length limit. An caller can send an arbitrarily large payload. Add a reasonable upper bound:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1024)
```

---

### 5. Minor — Logging Does Not Include Structured Context (Documentation / Observability)

```python
logger.error("Notification service error: %s", e)
```

When this fires in production, the log entry will not tell you which user or order was affected. Include the request identifiers:

```python
logger.error(
    "Notification service error for user_id=%s order_id=%s: %s",
    request.user_id,
    request.order_id,
    e,
)
```

## Summary of Required Changes

| Priority | Issue |
|---|---|
| 🚨 Blocker | Replace `requests` with `httpx.AsyncClient` and `await` the call |
| ⚠️ Required | Fix the test to raise the correct exception type and verify the actual error path |
| ⚠️ Required | Guard against missing `notification_id` in the response body |
| Minor | Add `Field` constraints to all request model fields |
| Minor | Add structured context to error log calls |
| Minor | Expand test coverage to non-2xx responses and invalid response bodies |

The core logic and structure are sound. The `async def` / `requests` mismatch is the only hard blocker, and the fix is a small, well-contained change. Happy to re-review once these are addressed.
