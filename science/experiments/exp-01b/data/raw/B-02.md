**Decision:** Request Changes

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` FastAPI route, which will block the event loop for the full duration of every external HTTP call — up to 5 seconds per request under the configured timeout. Under checkout load, this will stall every other in-flight request in the application simultaneously. The test suite also has a gap that masks a real failure mode. Both issues must be resolved before this merges.

---

## Detailed Findings

### 🔴 Critical — Synchronous I/O Inside Async Route

**File:** `api/notifications.py`, line 24

```python
# Current — blocks the event loop
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. When this line executes, the OS thread running the event loop sits idle waiting for the network response. No other coroutine can run during that wait. With a 5-second timeout, a modest traffic spike — say, 20 concurrent checkouts — means requests are queueing for up to 100 seconds of cumulative blocking time on a single-threaded event loop.

**Required fix — replace with `httpx` (async-native):**

```python
# In requirements.txt / pyproject.toml
# httpx>=0.27.0

import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    Raises HTTP 503 if the notification service is unreachable or returns an error.
    """
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error(
            "Notification service returned error status: %s %s",
            e.response.status_code,
            e.response.text,
        )
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service unreachable: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

If migrating to `httpx` is blocked by a dependency constraint, the acceptable fallback is `asyncio.to_thread`:

```python
import asyncio
import requests

# Acceptable only as a temporary measure — document the debt explicitly
response = await asyncio.to_thread(
    requests.post,
    "https://notifications.internal/send",
    json={...},
    timeout=5,
)
```

`asyncio.to_thread` offloads the blocking call to a thread pool, keeping the event loop free. It is not the preferred solution — thread pool exhaustion is a different failure mode — but it is categorically safer than blocking the event loop directly.

### 🔴 Critical — Test Suite Masks Real Failure Mode

**File:** `tests/test_notifications.py`, line 32

```python
# Patches the wrong exception type
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The production handler catches `requests.RequestException`. The test raises a bare `Exception`, which is a superclass of `RequestException` — so the test passes, but it is not testing the actual exception handling path. If someone later tightens the handler to catch only specific subclasses (e.g., `requests.ConnectionError`), this test will continue to pass while the real behavior breaks.

**Required fix:**

```python
import requests as requests_lib  # avoid name collision with the local variable

def test_send_notification_service_unavailable():
    with patch(
        "api.notifications.requests.post",
        side_effect=requests_lib.ConnectionError("Connection refused"),
    ):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503
    assert response.json()["detail"] == "Notification service unavailable"
```

### 🟡 Moderate — Missing Test Coverage

The following cases are untested and represent real failure modes in a checkout flow:

| Missing Case | Why It Matters |
|---|---|
| Upstream returns 4xx/5xx (e.g., 400, 500) | `raise_for_status()` throws — is the 503 mapping correct? |
| Response body missing `notification_id` | `data["notification_id"]` raises `KeyError` — currently unhandled, returns 500 |
| Malformed JSON response from upstream | `response.json()` raises — currently unhandled, returns 500 |
| Request validation rejection (missing fields) | Pydantic should reject, but worth asserting the 422 shape |

The `KeyError` case is worth calling out specifically:

```python
# Current code — will raise KeyError → unhandled → 500
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]
)

# Safer
notification_id = data.get("notification_id")
if not notification_id:
    logger.error(
        "Notification service response missing notification_id: %s", data
    )
    raise HTTPException(
        status_code=502,
        detail="Notification service returned an unexpected response",
    )
return NotificationResponse(success=True, notification_id=notification_id)
```

### 🟡 Moderate — No Input Sanitization on `message`

**File:** `api/notifications.py`, `NotificationRequest`

The `message` field is a free-form string passed directly into the JSON body of an outbound request to an internal service. There are no length constraints or content restrictions.

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded
```

This is a low-severity concern in an internal service, but it is worth constraining:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

### 🟢 Minor — `httpx.AsyncClient` Lifecycle

Once the fix to `httpx` is applied, consider whether a per-request `AsyncClient` is the right pattern. For a high-throughput checkout endpoint, a shared client with connection pooling — initialized at application startup via FastAPI's lifespan — will perform better than creating and tearing down a client on every request:

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
import httpx

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient(timeout=5.0)
    yield
    await app.state.http_client.aclose()

app = FastAPI(lifespan=lifespan)
```

Then inject via `Request`:

```python
from fastapi import Request

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    http_request: Request,
) -> NotificationResponse:
    client = http_request.app.state.http_client
    ...
```

This is a performance recommendation, not a blocker, but it is the right pattern for production.
