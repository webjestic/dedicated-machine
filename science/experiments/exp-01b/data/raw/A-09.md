**Decision: Request Changes**

**Summary:** This PR introduces a critical async correctness violation: `requests.post()` is a synchronous blocking call inside an `async def` route. Under real load, every concurrent request to this endpoint — and every *other* endpoint in the application — will stall while the event loop waits on that network call, which has a worst-case duration of 5 seconds per the configured timeout. The tests pass because `TestClient` runs synchronously and the mock never actually blocks, which is exactly the failure mode that makes this bug dangerous — it is invisible until production load exposes it.

Replace `requests` with `httpx` and use its async client:

```python
# Before — blocks the event loop
import requests

response = requests.post("https://notifications.internal/send", json={...}, timeout=5)

# After — yields control back to the event loop during I/O
import httpx

async with httpx.AsyncClient() as client:
    response = await client.post("https://notifications.internal/send", json={...}, timeout=5)
```

Two secondary issues also need to be addressed before this can merge:

**1. Exception handling is too broad in the test.** `test_send_notification_service_unavailable` patches with a bare `Exception`, but the route only catches `requests.RequestException`. This means the test is not actually exercising the error handling path as written — it is catching the exception at the framework level and returning 503 for the wrong reason. With `httpx`, update the test to raise `httpx.RequestError` to correctly validate the handler.

**2. The `message` field is unbounded and unvalidated.** Any string of arbitrary length is accepted and forwarded to the downstream service. At minimum, add a `max_length` constraint:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

Without this, the endpoint is a vector for oversized payloads being proxied downstream, and depending on how the notification service handles them, potentially a denial-of-service risk against that internal dependency.

The structure of the PR is otherwise solid — Pydantic validation, structured error handling, logging, and response modeling are all in the right places. The blocking I/O issue is the blocking issue.
