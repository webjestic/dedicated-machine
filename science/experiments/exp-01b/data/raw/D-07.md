Decision: Request Changes

Summary: The implementation has a critical correctness issue — it uses the synchronous `requests` library inside an `async` endpoint, which will block the event loop and degrade performance under load. There are also meaningful gaps in security (no authentication on the endpoint, unvalidated `message` field), error handling (missing key access on `data["notification_id"]` is unguarded, and the test mocks the wrong exception type), and test coverage (no validation failure cases, no malformed upstream response cases). These issues collectively make the PR not ready to merge into a checkout-critical path.

---

## Detailed Findings

### 1. Correctness

**🔴 Blocking I/O in async context**

```python
# Current — blocks the event loop
response = requests.post("https://notifications.internal/send", ...)
```

`requests` is synchronous. Calling it inside an `async def` endpoint does not make it non-blocking — it will block the entire event loop for the duration of the HTTP call (up to the 5-second timeout). Under any meaningful load this will cause latency spikes across all concurrent requests.

```python
# Recommended — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
        ...
```

**🟡 Unguarded key access on upstream response**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the upstream service returns a 2xx but with an unexpected payload shape, this raises an unhandled `KeyError`, which FastAPI will surface as a 500. This should be explicitly handled:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Upstream response missing notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

### 2. Test Coverage

**🔴 Wrong exception type mocked in failure test**

```python
# Current — mocks a generic Exception
with patch("api.notifications.requests.post", side_effect=Exception("Connection refused")):
```

The production code catches `requests.RequestException`, not bare `Exception`. This test would pass even if the `except` clause were removed entirely, because FastAPI would catch the unhandled `Exception` and return a 500 — which is *not* a 503. The test is asserting the wrong behavior and masking a real gap.

```python
# Correct
import requests as requests_lib

with patch(
    "api.notifications.requests.post",
    side_effect=requests_lib.RequestException("Connection refused"),
):
```

**🟡 Missing test cases**

| Scenario | Covered? |
|---|---|
| Successful notification | ✅ |
| External service network error | ⚠️ Wrong exception type |
| External service returns 4xx/5xx | ❌ |
| Upstream response missing `notification_id` | ❌ |
| Invalid request body (missing fields) | ❌ |
| Oversized or malicious `message` field | ❌ |

### 3. Security

**🔴 No authentication on the endpoint**

The `/notify` endpoint is unauthenticated. Any caller who can reach the service can trigger notifications to arbitrary `user_id` values with arbitrary `message` content. In a checkout service this is a significant abuse vector (notification spam, phishing-style messages to users).

At minimum, this endpoint should require an internal service token or be protected by the same auth middleware applied to other checkout endpoints:

```python
from fastapi import Depends
from api.auth import require_internal_token  # whatever your existing pattern is

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_token),
) -> NotificationResponse:
```

**🟡 `message` field is unbounded and unvalidated**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no length constraint, no content validation
```

An unbounded free-text field forwarded verbatim to an external service is a risk. Apply a reasonable length cap and consider whether callers should be able to supply arbitrary message content at all (vs. the service constructing the message from a template):

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

**🟡 Hardcoded internal service URL**

```python
"https://notifications.internal/send"
```

This should be pulled from configuration (environment variable or settings object) rather than hardcoded. Hardcoded URLs make environment promotion error-prone and prevent testing against a stub service URL.

```python
from api.config import settings

response = await client.post(settings.notification_service_url, ...)
```

### 4. Error Handling

- The `KeyError` on `data["notification_id"]` is addressed above.
- Consider distinguishing between 4xx and 5xx responses from the upstream service. A 4xx likely indicates a bug in the request being sent (worth a different log level and error code) vs. a 5xx indicating upstream unavailability.

```python
if response.status_code == 400:
    logger.error("Notification service rejected request: %s", response.text)
    raise HTTPException(status_code=500, detail="Internal notification configuration error")
response.raise_for_status()
```

---

### 5. Performance

The blocking `requests` issue (noted under Correctness) is the primary concern. No other performance regressions are introduced, and the 5-second timeout is a reasonable guard.

---

### 6. Naming Conventions

No issues. `NotificationRequest`, `NotificationResponse`, `send_notification`, and `router` are all consistent with FastAPI conventions and the apparent codebase style.

---

### 7. Documentation

The docstring is present but minimal. Given that this endpoint sits in a checkout-critical path, it would be worth documenting the expected caller, the upstream dependency, and the failure behavior — especially since a 503 here likely surfaces to the end user during checkout.

"""
Send an order confirmation notification to the user via the internal notification service.

Called by the checkout flow after successful order creation. Returns a notification_id
