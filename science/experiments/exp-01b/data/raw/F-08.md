**Decision: Request Changes**

**Summary:** This PR introduces a critical blocking I/O bug: `requests.post(...)` is a synchronous call made inside an `async` endpoint, which blocks the entire event loop for up to 5 seconds per request under the exact high-load conditions this service is designed to handle. This must be replaced with an async HTTP client (e.g., `httpx.AsyncClient` with `await`) before this can merge. Additionally, the `message` field accepts unbounded, unvalidated user-supplied string content that is forwarded directly to an internal service, which is a correctness and security concern that warrants at minimum a length constraint and ideally an allowlist of message types.

---

### Blocking Issues

#### 1. Synchronous `requests` call inside `async` endpoint

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous: freezes all concurrent requests
```

```python
# Required — non-blocking
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
```

The `async def` declaration is not sufficient — it only yields control at `await` points. `requests.post` has no `await` point and will hold the event loop for the full duration of the network call, including the 5-second timeout on failure. Under checkout load, this is application-wide paralysis, not a single-user delay.

---

#### 2. Unvalidated `message` field forwarded to internal service

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded, arbitrary content accepted and forwarded
```

The `message` field accepts any string of any length from the caller and forwards it verbatim to an internal service. At minimum:

```python
from pydantic import BaseModel, Field, constr

ALLOWED_MESSAGES = {
    "Your order has been confirmed.",
    # extend as needed
}

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=500)
    
    @field_validator("message")
    @classmethod
    def message_must_be_allowed(cls, v: str) -> str:
        if v not in ALLOWED_MESSAGES:
            raise ValueError("message not in allowed set")
        return v
```

If the message content is truly dynamic, enforce a strict length cap and sanitize before forwarding. Passing arbitrary caller-controlled strings to a downstream internal service is a vector for injection or abuse depending on how that service handles the payload.

### Secondary Issues

#### 3. Test suite does not catch the core bug

The tests use `TestClient`, which runs the ASGI app synchronously in a test thread and will pass even with a blocking `requests` call. The tests verify behavior but provide no signal about event loop blocking. This is not a reason to block the PR on its own, but the test coverage gives false confidence here.

#### 4. `data["notification_id"]` is accessed without validation

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

A malformed but HTTP-200 response from the notification service will raise an unhandled `KeyError`, bypassing the `except requests.RequestException` block and returning a 500 with no logging. Add a guard:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```
