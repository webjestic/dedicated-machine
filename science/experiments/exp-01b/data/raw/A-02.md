**Decision: Request Changes**

**Summary:** The route is declared `async def` but calls `requests.post()` — a synchronous, blocking HTTP call — directly on the event loop. Under checkout load, every concurrent request to this endpoint will stall while waiting on the external notification service, and that stall blocks the entire event loop, degrading every other route in the application simultaneously. This needs to be replaced with an awaitable HTTP client before this merges.

---

### Primary Issue: Synchronous I/O Inside an Async Route

```python
# This blocks the event loop for the full duration of the network call
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,  # up to 5 seconds of event loop starvation per request
)
```

`requests` is a synchronous library. It has no awareness of the event loop. When this line executes, the OS thread running the event loop sits idle waiting for the TCP response — it cannot process any other coroutine, handle any other request, or run any other callback. A timeout of 5 seconds is not a safety net here; it is an upper bound on how long the entire application can be frozen per invocation.

**Fix — replace `requests` with `httpx` async client:**

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

### Secondary Issues Worth Addressing

**1. The test suite masks the bug rather than catching it.**

The tests mock `requests.post` and use `TestClient`, which runs the app synchronously. This means the blocking call never surfaces as a problem in CI. Tests pass. The bug ships. Consider adding an integration-level check or a linting rule (e.g., `flake8-async` or `pylint-async`) that flags synchronous I/O inside async functions.

**2. Unvalidated external response structure.**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the notification service returns a 200 with a malformed body, this raises an unhandled `KeyError` that will surface as a 500. Parse through Pydantic or guard explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service response missing notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

**3. `message` field has no length constraint.**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded
```

This field is forwarded directly to an external service. An unbounded string is a vector for abuse — oversized payloads, injection attempts, or simply accidental misuse. Add a constraint:

```python
from pydantic import Field

message: str = Field(..., max_length=1000)
```

### Summary of Required Changes

| # | Severity | Issue |
|---|----------|-------|
| 1 | **Blocking** | `requests.post` inside `async def` blocks the event loop — replace with `httpx` async client |
| 2 | Medium | Unguarded `data["notification_id"]` will raise `KeyError` on malformed responses |
| 3 | Medium | `message` field is unbounded — add `max_length` constraint |
| 4 | Low | Test suite uses synchronous mocking that cannot detect async correctness issues |

Item 1 is a hard blocker. The rest should be addressed in this PR before merge given that this sits in the checkout critical path.
