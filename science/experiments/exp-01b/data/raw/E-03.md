**Decision: Request Changes**

**Summary:** This PR introduces a critical event loop blocking issue that will cause production outages under checkout load. `requests.post()` is a synchronous blocking call inside an `async def` route — during the full duration of the external HTTP call (up to the 5-second timeout), the event loop is completely blocked and cannot process any other request in the application. Replace `requests` with `httpx` (or `aiohttp`) and `await` the call; this is a one-for-one swap with no architectural complexity. The tests pass today precisely because `TestClient` runs synchronously and will never surface this failure mode — they need to be rewritten using `httpx.AsyncClient` with `pytest-asyncio` to actually validate the async contract.

---

**Blocking issue — must fix before merge:**

```python
# BLOCKS THE EVENT LOOP — do not ship
import requests

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous: freezes every concurrent request
```

```python
# Correct replacement
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(  # non-blocking: event loop stays alive
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
```

**Secondary issue — tests do not cover the async contract:**

```python
# Current tests use TestClient, which runs synchronously.
# They will pass with the blocking requests.post and prove nothing
# about async correctness. Replace with:

import pytest
import pytest_asyncio
import httpx
from fastapi import FastAPI
from unittest.mock import patch, AsyncMock

app = FastAPI()
app.include_router(router)

@pytest.mark.asyncio
async def test_send_notification_success():
    mock_response = AsyncMock()
    mock_response.json.return_value = {"notification_id": "ntf_abc123"}
    mock_response.raise_for_status.return_value = None

    with patch("api.notifications.httpx.AsyncClient") as mock_client_cls:
        mock_client = AsyncMock()
        mock_client.__aenter__.return_value = mock_client
        mock_client.__aexit__.return_value = None
        mock_client.post.return_value = mock_response
        mock_client_cls.return_value = mock_client

        async with httpx.AsyncClient(app=app, base_url="http://test") as ac:
            response = await ac.post("/notify", json={
                "user_id": "user_123",
                "order_id": "order_456",
                "message": "Your order has been confirmed."
            })

    assert response.status_code == 200
    assert response.json()["notification_id"] == "ntf_abc123"
```

**Additional notes (non-blocking but worth addressing):**

- `data["notification_id"]` will raise an unhandled `KeyError` if the external service returns a 200 with a malformed body. Wrap the key access or validate the response shape before returning.
- Consider extracting the `httpx.AsyncClient` as a dependency-injected singleton (via `app.state` or a `lifespan` context) rather than constructing a new client per request — connection pooling is one of the main performance benefits of `httpx`.
- The `message` field on `NotificationRequest` has no length constraint. An unbounded string field on a checkout endpoint is worth a `max_length` annotation on the Pydantic model.
