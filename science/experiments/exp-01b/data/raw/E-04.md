**Decision: Request Changes**

**Summary:** This PR introduces a critical event loop blocking issue: `requests.post()` is a synchronous I/O call inside an `async def` route. Under checkout load, every invocation holds the event loop hostage for up to the full 5-second timeout, freezing all concurrent requests across the entire application — not just this one. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if `requests` must be retained, offload via `asyncio.get_event_loop().run_in_executor()`. The tests pass precisely because `TestClient` runs synchronously and will never surface this failure mode; the bug ships cleanly and detonates under real production concurrency.

The core defect, the secondary issues, and the required fixes:

### 🔴 Blocking I/O on the Event Loop (Must Fix)

```python
# Current — synchronous, blocks the event loop
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is built on `urllib3`, which uses blocking socket I/O. When this line executes inside `async def send_notification`, the OS thread running the event loop is blocked. No other coroutine can be scheduled for the duration of the network call — up to 5 seconds per the configured timeout. At checkout peak load, this is not a per-user degradation. It is full application paralysis, per request, for every request in flight simultaneously.

**Fix — preferred:** Replace with `httpx`, which provides an async-native client with an API nearly identical to `requests`:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

**Fix — acceptable fallback** if `requests` cannot be replaced (e.g., a dependency constraint):

```python
import asyncio
from functools import partial

loop = asyncio.get_event_loop()
response = await loop.run_in_executor(
    None,  # uses the default ThreadPoolExecutor
    partial(
        requests.post,
        "https://notifications.internal/send",
        json={...},
        timeout=5,
    )
)
```

This moves the blocking call off the event loop thread. It is less clean than the `httpx` path and still consumes a thread pool slot per in-flight request, but it does not block the event loop.

### 🟡 Overly Broad Exception Handling

```python
# Current
except requests.RequestException as e:
```

This catches every `requests` exception uniformly and returns 503. A timeout (the caller is slow or unreachable) and a 4xx response (the request was malformed) are different failure modes and should not be treated identically. With `httpx`, separate `httpx.HTTPStatusError` from `httpx.RequestError` as shown above. Log enough context to distinguish them in production.

---

### 🟡 No Input Validation Constraints on `message`

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str
```

`message` is an unbounded string. This field is passed directly into an outbound HTTP request body. Add a `max_length` constraint to prevent oversized payloads:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1024)
```

### 🟡 Tests Do Not Cover the Async Failure Mode

The test suite uses `TestClient`, which runs the ASGI app synchronously via `anyio` in a single thread. It will never surface event loop starvation. This is not a reason to reject the tests — they are correct for what they test — but the team should be aware that passing tests here provide no signal about the production concurrency behavior. Once the `httpx` fix is in place, update the tests to patch `httpx.AsyncClient.post` rather than `requests.post`:

```python
import pytest
from unittest.mock import AsyncMock, patch, MagicMock
from httpx import Response

@pytest.mark.anyio
async def test_send_notification_success():
    mock_response = MagicMock(spec=Response)
    mock_response.json.return_value = {"notification_id": "ntf_abc123"}
    mock_response.raise_for_status.return_value = None

    with patch("httpx.AsyncClient.post", new_callable=AsyncMock, return_value=mock_response):
        # test via async client or AsyncClient directly
        ...
```

### Summary of Required Changes

| Severity | Issue | Action |
|---|---|---|
| 🔴 Blocker | `requests.post` inside `async def` blocks event loop | Replace with `httpx.AsyncClient` + `await` |
| 🟡 Recommended | Broad exception handling loses failure context | Separate timeout vs. HTTP error vs. connection error |
| 🟡 Recommended | Unbounded `message` field | Add `Field(max_length=...)` constraints |
| 🟡 Recommended | Tests patch the wrong library after fix | Update mocks to target `httpx` |

The rest of the structure — Pydantic models, router setup, logging, docstring, error response shape — is solid. This is one targeted fix away from being approvable.
