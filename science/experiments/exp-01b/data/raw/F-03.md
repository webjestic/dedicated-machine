## Review

**Decision: Request Changes**

**Summary:** This PR introduces a critical concurrency defect that directly matches the failure mode described in the service context: `requests.post` is a synchronous, blocking call made inside an `async` endpoint, which will block the entire event loop for up to 5 seconds per request under the declared timeout. At checkout-level traffic, this is a full-service stall, not a single-user degradation. Replace `requests` with `httpx.AsyncClient` and `await` the call, or offload to a thread pool via `asyncio.run_in_executor` as a stopgap. Additionally, the `message` field accepts unbounded, unvalidated user-supplied string content that is forwarded directly to an internal service — add length constraints and consider sanitization. The test suite also does not cover the happy path for the async behavior correctly (TestClient masks the blocking issue), and `requests.RequestException` won't catch the bare `Exception` used in the failure test, meaning the error-handling test is not actually validating the real exception path.

---

### Specific Issues

#### 1. Blocking I/O on the Event Loop — Critical

```python
# Current — blocks the event loop for up to 5 seconds
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

```python
# Required fix — non-blocking async HTTP
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
```

If `httpx` cannot be added as a dependency immediately, the minimum acceptable stopgap is:

```python
import asyncio
loop = asyncio.get_event_loop()
response = await loop.run_in_executor(None, lambda: requests.post(..., timeout=5))
```

This is not preferred — it still consumes a thread — but it does not block the event loop.

#### 2. Unbounded, Unvalidated Input Forwarded to Internal Service — Security

The `message` field is forwarded verbatim to an internal service with no length limit or content constraints. A caller can send an arbitrarily large payload.

```python
# Current
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str
```

```python
# Required fix
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=1000)
```

#### 3. `data["notification_id"]` — Unguarded Key Access

If the external service returns a malformed or unexpected response body, this raises an unhandled `KeyError` that will produce a 500, not a 503, and will not be logged.

```python
# Current
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if key absent
)
```

```python
# Required fix
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=503, detail="Notification service unavailable")
return NotificationResponse(success=True, notification_id=notification_id)
```

#### 4. Test Exception Type Mismatch

The failure test patches with a bare `Exception`, but the handler catches `requests.RequestException`. These are not the same hierarchy. The test passes only because `TestClient` has its own exception handling behavior — it does not validate that your application code is actually catching the right exception type.

```python
# Current test — does not exercise the real handler
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

```python
# Correct test
import requests as requests_lib

with patch("api.notifications.requests.post",
           side_effect=requests_lib.ConnectionError("Connection refused")):
```

After switching to `httpx`, update accordingly to `httpx.RequestError`.
