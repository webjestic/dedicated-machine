Decision: Request Changes

Summary: The implementation has a critical architectural flaw — using the synchronous `requests` library inside an `async` endpoint will block the event loop under load, which is a significant performance regression for a FastAPI service. Beyond that, there are security concerns around unvalidated user-controlled input being forwarded to an internal service, and the test suite has meaningful gaps that leave failure modes untested.

---

## Detailed Findings

### 1. Correctness

**🔴 Blocking HTTP client in async context**

```python
# Current — blocks the event loop on every external call
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

`requests` is synchronous. Calling it inside an `async` def ties up the event loop for the duration of the network call, eliminating FastAPI's concurrency benefits. Under any meaningful load this will cause latency to cascade across all endpoints in the service.

```python
# Fix — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

Alternatively, if keeping `requests` is a hard constraint, wrap the call with `asyncio.to_thread()` — but migrating to `httpx` is the cleaner path and is already the FastAPI-ecosystem convention.

**🟡 Missing key access guard on external response**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a malformed or unexpected payload, this raises an unhandled `KeyError` that will surface as a 500. It should be caught explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Unexpected response from notification service")
```

### 2. Security

**🔴 Unvalidated `message` field forwarded to internal service**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no length limit, no content constraints
```

The `message` field is caller-controlled and is forwarded verbatim to an internal service. This is a prompt injection / payload injection risk depending on how `notifications.internal` processes messages (e.g., if it renders HTML, sends SMS, or passes content to a template engine). At minimum:

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=500, strip_whitespace=True)
```

Also consider whether `message` should even be caller-supplied at all. For order confirmation, the message content could be generated server-side from `order_id`, removing the injection surface entirely.

**🟡 Hardcoded internal service URL**

```python
"https://notifications.internal/send"
```

This should be an environment variable or injected configuration, not a string literal. Hardcoded URLs make environment promotion (staging → production) error-prone and prevent the URL from being rotated without a code change.

```python
import os
NOTIFICATION_SERVICE_URL = os.environ.get(
    "NOTIFICATION_SERVICE_URL", "https://notifications.internal/send"
)
```

**🟡 No authentication on the outbound request**

There is no API key, bearer token, or mTLS configuration on the call to the internal service. Even internal services should require authentication. The mechanism will depend on your infrastructure, but it should be present.

### 3. Error Handling

**🟡 Overly broad exception catch in tests, inconsistency in production code**

In the test:
```python
side_effect=Exception("Connection refused")  # catches everything
```

But in production:
```python
except requests.RequestException as e:  # only catches requests errors
```

A bare `Exception` in the test masks the fact that a `KeyError` from `data["notification_id"]` would *not* be caught by the production handler and would produce a 500, not a 503. The test should use `requests.RequestException` to match the actual handler, and a separate test should cover the malformed-response case.

**🟡 No timeout on connection establishment vs. read**

```python
timeout=5
```

A scalar timeout applies to the entire request. Consider splitting it to avoid a slow-to-connect service holding the connection open:

```python
timeout=httpx.Timeout(connect=2.0, read=5.0)
```

### 4. Test Coverage

**🔴 Missing test cases**

| Scenario | Covered? |
|---|---|
| Success path | ✅ |
| External service unreachable | ✅ (but wrong exception type — see above) |
| External service returns 4xx/5xx | ❌ |
| External service returns valid JSON but missing `notification_id` | ❌ |
| Invalid request body (missing fields) | ❌ |
| `message` field exceeds max length | ❌ (once validation is added) |

The 4xx/5xx case is particularly important — `raise_for_status()` will throw, but the test suite never exercises this path:

```python
def test_send_notification_upstream_http_error():
    mock_response = MagicMock()
    mock_response.raise_for_status.side_effect = requests.HTTPError(response=MagicMock(status_code=500))

    with patch("api.notifications.requests.post", return_value=mock_response):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503
```

---

### 5. Performance

The blocking `requests` issue (covered under Correctness) is the primary concern. No other regressions are visible, but the client lifecycle also warrants attention: creating a new HTTP client per request forgoes connection pooling. With `httpx`, prefer a shared `AsyncClient` instance (e.g., managed via FastAPI's lifespan context) rather than instantiating one per call.

---

### 6. Naming & Documentation

These are minor and non-blocking:

- Names are clear and consistent. `NotificationRequest` / `NotificationResponse` follow the existing pattern.
- The docstring on `send_notification` is minimal but acceptable for a simple endpoint.
- Consider adding a note in the docstring or inline comment that `message` is caller-supplied and subject to the length constraint, once that is added.

---

## Required Before Approval

1. Replace `requests` with `httpx` (async) or wrap with `asyncio.to_thread`.
2. Add a guard for missing `notification_id` in the external response.
3. Add `max_length` validation to the `message` field.
4. Move the service URL to environment configuration.
5. Add authentication to the outbound request.
6. Fix the test exception type and add the missing test cases listed above.
