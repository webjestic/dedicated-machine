**Decision:** Request Changes

**Summary:** This PR introduces a critical async correctness violation: `requests.post()` is a synchronous, blocking call inside an `async def` route. Under real load, every concurrent request to this endpoint — and every other route in the application — will stall while the event loop waits for the external HTTP call to complete. The test suite passes because `TestClient` runs synchronously and mocking eliminates the actual I/O, meaning this bug is completely invisible until it hits production. Beyond the blocking I/O issue, there are secondary concerns around input validation, error handling robustness, and test coverage that need to be addressed before this can merge.

---

## Issue 1 — Blocking I/O Inside an Async Route (Critical, Must Fix)

**File:** `api/notifications.py`, line 27

```python
# Current — blocks the event loop on every call
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. When this line executes, it holds the event loop thread for the full duration of the network call — up to 5 seconds by the configured timeout. During that window, FastAPI cannot process any other request. This is not a theoretical concern: under checkout load, where many users hit this endpoint simultaneously, you will see cascading latency across the entire application, not just this route.

The fix is to replace `requests` with `httpx` in async mode, which is the standard drop-in replacement for this exact scenario:

```python
# requirements.txt or pyproject.toml
# remove: requests
# add:    httpx

import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, HttpUrl, field_validator
import logging

logger = logging.getLogger(__name__)
router = APIRouter()

# Reuse a single client across requests — avoids per-request connection
# overhead and allows connection pooling. Instantiated at module level so
# it is shared for the lifetime of the application process.
_http_client = httpx.AsyncClient(timeout=5.0)

NOTIFICATION_SERVICE_URL = "https://notifications.internal/send"


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user via the internal
    notification service. Raises HTTP 503 if the service is unreachable
    or returns a non-2xx response.
    """
    try:
        response = await _http_client.post(
            NOTIFICATION_SERVICE_URL,
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
    except httpx.HTTPStatusError as e:
        logger.error(
            "Notification service returned error status: %s %s",
            e.response.status_code,
            e.response.text,
        )
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

Note the shared `_http_client` instance. Creating a new `httpx.AsyncClient` per request defeats connection pooling and adds measurable overhead on a high-traffic checkout path. The client should be created once and reused. If your application uses lifespan events, registering `_http_client.aclose()` there is the cleanest teardown path:

```python
# In your application factory, e.g. main.py
from contextlib import asynccontextmanager
from api.notifications import _http_client

@asynccontextmanager
async def lifespan(app: FastAPI):
    yield
    await _http_client.aclose()

app = FastAPI(lifespan=lifespan)
```

## Issue 2 — Error Handling Does Not Distinguish Connection Failures from Service Errors (Moderate)

**File:** `api/notifications.py`

The current `except requests.RequestException` block catches everything from a DNS failure to a 500 response from the notification service. These are meaningfully different failure modes and collapsing them loses signal in your logs and makes alerting harder.

The `httpx` version above already separates these into `HTTPStatusError` (the service responded with an error code) and `RequestError` (the service was not reachable at all). Keep them separate. If you ever want to retry on connection errors but not on 4xx responses, you will need that distinction.

Additionally, the current code does not handle the case where the notification service returns a 2xx response but the JSON body does not contain `notification_id`. This will raise an unhandled `KeyError` that becomes a 500 with no useful log message:

```python
# Current — will raise KeyError if notification_id is absent
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"],
)
```

Add an explicit guard:

```python
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error(
        "Notification service response missing notification_id: %s", data
    )
    raise HTTPException(
        status_code=502,
        detail="Notification service returned an unexpected response",
    )
return NotificationResponse(success=True, notification_id=notification_id)
```

## Issue 3 — Input Validation Is Incomplete (Moderate)

**File:** `api/notifications.py`, `NotificationRequest`

Pydantic is present but doing almost no validation. All three fields accept any non-empty string. At minimum:

**`message`** has no length bound. A caller can send a multi-megabyte string that gets forwarded directly to the notification service. Add a `max_length` constraint:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1024)
```

**`user_id` and `order_id`** are passed directly into the JSON payload sent to an internal service. If those fields contain unexpected characters or are structurally malformed, the notification service receives garbage. Consider validating their format against the expected pattern (e.g., `user_\d+`, `order_\d+`) using a `field_validator` if you have a known schema.

## Issue 4 — Test Coverage Has Gaps (Moderate)

**File:** `tests/test_notifications.py`

The tests cover the happy path and a generic exception. Several important cases are missing:

**The failure test patches the wrong exception type.** The service error test raises a bare `Exception`, but the handler catches `requests.RequestException`. After switching to `httpx`, make sure the test raises `httpx.RequestError` or `httpx.HTTPStatusError` specifically, not a generic exception. A generic `Exception` bypassing the handler and still returning 503 would be a false pass — verify the test is actually exercising the except branch you think it is.

**Missing test cases that should be added:**

```python
# 1. Notification service returns 2xx but body lacks notification_id
def test_send_notification_missing_notification_id():
    mock_response = MagicMock()
    mock_response.json.return_value = {}  # no notification_id
    mock_response.raise_for_status.return_value = None

    with patch("api.notifications._http_client.post", ...):
        response = client.post("/notify", json={...})

    assert response.status_code == 502


# 2. Notification service returns a 4xx/5xx status
def test_send_notification_upstream_http_error():
    ...
    assert response.status_code == 503


# 3. Input validation — message too long
def test_send_notification_message_too_long():
    response = client.post("/notify", json={
        "user_id": "user_123",
        "order_id": "order_456",
        "message": "x" * 2000,
    })
    assert response.status_code == 422


# 4. Input validation — missing required fields
def test_send_notification_missing_fields():
    response = client.post("/notify", json={"user_id": "user_123"})
    assert response.status_code == 422
```

Also note: `TestClient` runs the ASGI app synchronously, which means the blocking `requests.post` bug does not surface in tests even without mocking. After switching to `httpx`, use `AsyncClient` from `httpx` with `pytest-anyio` or `anyio` for tests that need to exercise the async path end-to-end.

## Issue 5 — Hardcoded Service URL (Minor)

**File:** `api/notifications.py`, line 27

```python
"https://notifications.internal/send",
```

This URL is hardcoded. It cannot be overridden per environment without a code change, which means staging and production point at the same service, and local development has no clean override path. Move it to configuration:

```python
# config.py or via pydantic-settings
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str = "https://notifications.internal/send"

settings = Settings()
```

```python
# api/notifications.py
from config import settings

response = await _http_client.post(settings.notification_service_url, ...)
```

This also makes the URL injectable in tests without patching internals.
