**Decision: Request Changes**

**Summary:** This PR uses `requests.post()` — a synchronous, blocking HTTP call — inside an `async def` route. Under load, every concurrent request to this endpoint will stall on the event loop while waiting for the external notification service to respond (up to the 5-second timeout), starving every other handler in the application simultaneously. Replace `requests` with `httpx.AsyncClient` (or `aiohttp`) and `await` the call. The tests pass today because `TestClient` runs synchronously and the mock never actually blocks — this is exactly the class of bug that ships cleanly and surfaces only in production.

The fix is straightforward:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

Two secondary notes worth addressing in the same pass:

1. **`httpx.AsyncClient` lifecycle** — instantiating a new client per request works but is suboptimal. For a checkout-path endpoint under real load, inject a shared client via FastAPI's lifespan context so connection pooling is actually used.

2. **Test coverage gap** — the `service_unavailable` test patches with a bare `Exception`, which bypasses the `requests.RequestException` catch block in the original code entirely. It only passes because `TestClient` re-raises unhandled exceptions as 500s, not 503s — check the actual assertion. Update the test to raise `httpx.RequestError` (or `httpx.HTTPStatusError`) so the error-handling path is genuinely exercised.
