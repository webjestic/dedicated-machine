**Decision:** Request Changes

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` route, which will block the event loop for the full duration of every external HTTP call — up to 5 seconds per request at the configured timeout. Under checkout load, this will stall every other in-flight request in the application simultaneously. Beyond that critical defect, the test suite has a gap in error-type coverage and the error handling has a subtle data-safety issue. None of these are merge-blocking on their own except the first one, but all of them should be resolved before this goes near production.

---

## Detailed Findings

### 1. 🚨 Blocking I/O Inside Async Route (Critical — Correctness / Performance)

**File:** `api/notifications.py`, line 24

```python
# Current — blocks the event loop
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. When this line executes, the OS thread running the event loop is parked waiting on a TCP socket. No other coroutine can run during that wait. With a 5-second timeout and any meaningful checkout traffic, this becomes a denial-of-service against your own application.

The fix is to replace `requests` with `httpx` in async mode, which is a near-drop-in replacement for this usage pattern:

```python
# requirements.txt / pyproject.toml
# httpx>=0.27

import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    Raises 503 if the notification service is unreachable or returns an error.
    """
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error(
                "Notification service returned error status %s for order %s: %s",
                e.response.status_code,
                request.order_id,
                e,
            )
            raise HTTPException(
                status_code=503, detail="Notification service unavailable"
            )
        except httpx.RequestError as e:
            logger.error(
                "Notification service connection error for order %s: %s",
                request.order_id,
                e,
            )
            raise HTTPException(
                status_code=503, detail="Notification service unavailable"
            )
```

If `httpx` is not acceptable for some reason, the fallback is `asyncio.get_event_loop().run_in_executor(None, ...)` to push the blocking call onto a thread pool — but that is a workaround, not a solution, and `httpx` is the right tool here.

### 2. Error Handling: Bare `Exception` Catch in Tests, and Missing Key Access Guard (Moderate — Correctness / Error Handling)

**Issue A — test gap:** `test_send_notification_service_unavailable` patches with a bare `Exception`, but the production handler only catches `requests.RequestException` (or, after the fix, `httpx.RequestError` / `httpx.HTTPStatusError`). A bare `Exception` that is *not* a `RequestException` will propagate as an unhandled 500, not a clean 503. The test passes today because `TestClient` happens to catch it, but it is not testing the right contract.

```python
# test: should use the actual exception type the handler catches
import httpx

def test_send_notification_connection_error():
    with patch(
        "api.notifications.httpx.AsyncClient.post",
        side_effect=httpx.ConnectError("Connection refused"),
    ):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })
    assert response.status_code == 503
    assert response.json()["detail"] == "Notification service unavailable"


def test_send_notification_http_error():
    """Notification service returns a 500."""
    with patch(
        "api.notifications.httpx.AsyncClient.post",
        side_effect=httpx.HTTPStatusError(
            "Internal Server Error",
            request=MagicMock(),
            response=MagicMock(status_code=500),
        ),
    ):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })
    assert response.status_code == 503
```

**Issue B — unguarded key access:** In the success path, `data["notification_id"]` will raise a bare `KeyError` if the notification service returns a 200 with a malformed body. That becomes an unhandled 500 with a stack trace that may leak internal details. Add an explicit check:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error(
        "Notification service returned success but missing notification_id "
        "for order %s. Response body: %s",
        request.order_id,
        data,
    )
    raise HTTPException(
        status_code=502, detail="Notification service returned unexpected response"
    )
return NotificationResponse(success=True, notification_id=notification_id)
```

### 3. Security: Message Field Is Unbounded (Minor — Security)

**File:** `api/notifications.py`, `NotificationRequest.message`

The `message` field has no length constraint. This field is passed directly into the outbound JSON payload to an internal service. An oversized payload could cause issues at the downstream service or be used to probe internal behavior. Add a reasonable bound:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

This also catches empty-string inputs that would currently pass validation and produce silent failures downstream.

### 4. Resource Management: `AsyncClient` Lifecycle (Minor — Performance)

The corrected code above already addresses this, but worth calling out explicitly: creating a new `httpx.AsyncClient` per request is functional but suboptimal. Each instantiation opens and tears down a connection pool. For a checkout-path endpoint under load, a shared client with connection pooling is meaningfully better:

```python
# api/notifications.py — module level, or injected via lifespan
from contextlib import asynccontextmanager
import httpx

_notification_client: httpx.AsyncClient | None = None


@asynccontextmanager
async def lifespan(app):
    global _notification_client
    _notification_client = httpx.AsyncClient(timeout=5.0)
    yield
    await _notification_client.aclose()
```

This is a follow-up improvement rather than a blocker, but it should be tracked. The per-request client is acceptable for an initial merge once the blocking-I/O issue is resolved.

---

### 5. Logging: Order ID Missing from Error Context (Minor — Observability)

The current error log is:

```python
logger.error("Notification service error: %s", e)
```

When this fires in production, you will have no way to correlate the log line back to a specific order without digging through request logs separately. The corrected example above includes `request.order_id` in all log calls — that pattern should be consistent throughout.
