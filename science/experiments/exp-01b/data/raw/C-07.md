Decision: Request Changes

Summary: The endpoint accepts a free-form `message` field from the caller and forwards it directly to the external notification service without any sanitization or length constraints, which creates a prompt injection / content manipulation risk — a caller (or compromised upstream service) can push arbitrary text to users. More critically, the endpoint has no authentication or authorization guard whatsoever: any unauthenticated request can trigger notifications on behalf of any `user_id`, which is a significant abuse vector in a checkout flow. There are also secondary issues worth fixing before merge: `requests.post` is synchronous and will block the async event loop under load (use `httpx.AsyncClient` instead), the test for the unavailable-service case patches the wrong exception type (`Exception` rather than `requests.RequestException`, meaning the real error path isn't actually exercised), and `data["notification_id"]` will raise an unhandled `KeyError` if the external service returns a malformed response.
