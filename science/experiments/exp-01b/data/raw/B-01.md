**Decision: Request Changes**

**Summary:** This PR contains a critical async correctness violation: `requests.post()` is a synchronous blocking call inside an `async def` route, which will block the event loop for the full duration of every outbound HTTP request. Under checkout load, this will stall every concurrent request in the application. The fix is a drop-in replacement with `httpx.AsyncClient`, but the tests also need to be updated to cover failure modes more precisely.

---

## Review

### 1. Correctness — ❌ Critical

**`requests` inside `async def` blocks the event loop.**

```python
# api/notifications.py — line 24
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(   # <-- synchronous. blocks the entire event loop.
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

`requests` is a synchronous library. When this line executes, the OS thread running the event loop is parked waiting on a TCP socket for up to 5 seconds. During that window, FastAPI cannot process *any* other request — not other notifications, not cart updates, not health checks. This is not a theoretical concern: checkout flows are high-concurrency by nature, and a 5-second external timeout under load will produce cascading failures that look like the entire service is down.

**Fix:** Replace `requests` with `httpx.AsyncClient`, which is the idiomatic async-compatible HTTP client for FastAPI services.

```python
# api/notifications.py — corrected
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

> **Note on `httpx.AsyncClient` lifecycle:** For a high-traffic endpoint, consider managing a single shared `AsyncClient` instance at application startup (via FastAPI's lifespan context) rather than constructing one per request. Per-request client construction works correctly and is acceptable to ship initially, but connection pooling will not be effective. File a follow-up ticket if this endpoint will see significant volume.

### 2. Test Coverage — ⚠️ Needs Improvement

**The failure-mode test is too broad.**

```python
# tests/test_notifications.py — line 24
def test_send_notification_service_unavailable():
    with patch("api.notifications.requests.post",
               side_effect=Exception("Connection refused")):  # <-- bare Exception
```

This test patches a bare `Exception`, which means it will pass even if the error handling is broken in ways that accidentally catch unrelated exceptions. After migrating to `httpx`, use `httpx.RequestError` to match what the handler actually catches. You should also add a test for the HTTP error path (4xx/5xx response from the upstream service) since `raise_for_status()` raises `httpx.HTTPStatusError`, which is a distinct exception type from a connection failure.

**Missing test cases:**

| Scenario | Covered? |
|---|---|
| Successful notification | ✅ |
| Connection/network failure | ✅ (but wrong exception type) |
| Upstream returns 4xx/5xx | ❌ |
| Response body missing `notification_id` | ❌ |
| Input validation (missing fields) | ❌ |

### 3. Naming Conventions — ✅

No issues. `NotificationRequest`, `NotificationResponse`, `send_notification` are clear and consistent with FastAPI conventions. The local variable `response` shadowing the outer `response` (FastAPI response vs. httpx response) is a minor readability nit but not worth blocking on.

### 4. Error Handling — ⚠️ Minor

**`KeyError` on missing `notification_id` is unhandled.**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if key is absent
)
```

If the upstream service returns a 200 with a malformed body, this raises an unhandled `KeyError` that will surface as a 500 to the client with no log context. Use `.get()` with an explicit fallback and raise a meaningful error:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service response missing notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

Note the status code distinction: 503 is appropriate for *unavailability*, 502 is appropriate for a *bad upstream response*. Using them correctly makes on-call triage faster.

---

### 5. Performance — ⚠️ (Subsumed by the async issue above)

The blocking `requests` call is the dominant performance issue and is already covered. After the `httpx` migration, the per-request `AsyncClient` construction is a secondary concern worth a follow-up ticket but not a blocker.

---

### 6. Security — ✅ No new surface introduced

The endpoint does not expose internal error details to the caller. The external URL is hardcoded (not user-controlled), so there is no SSRF risk from this implementation. Input fields are validated by Pydantic. No credentials are handled here.
