Decision: Request Changes

Summary: The implementation has a critical architectural flaw — using the synchronous `requests` library inside an `async` endpoint will block the event loop under load, which is a meaningful performance regression for a checkout service. Beyond that, there are several correctness, security, and test coverage gaps that need to be addressed before this is mergeable.

---

## Detailed Findings

### 1. Performance — Blocking I/O in Async Context (Critical)

```python
# Current — blocks the event loop for the duration of the HTTP call
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocks all other requests
```

`requests` is a synchronous library. Calling it inside an `async` def without running it in a thread pool means every in-flight notification ties up the entire FastAPI event loop. Under checkout load this will cause latency spikes across all endpoints, not just this one.

**Fix:** Replace with `httpx` (async-native) or run the blocking call in a thread pool.

```python
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
        )
        response.raise_for_status()
        ...
```

---

### 2. Security — Hardcoded Internal Service URL (High)

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

Hardcoding the URL couples the code to a specific environment and makes it impossible to rotate or override the endpoint without a code change. It also means the URL ends up in version history permanently.

**Fix:** Read from configuration or environment variables.

```python
from api.config import settings  # or os.environ / pydantic BaseSettings

response = await client.post(settings.notification_service_url, ...)
```

---

### 3. Security — Unvalidated `message` Field

The `message` field is a free-form `str` with no length or content constraints. If this value is passed through to an external service (or ever rendered in a UI), it is a potential injection surface.

**Fix:** Add a `max_length` constraint and consider allowlisting content if the message is templated.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

The same applies to `user_id` and `order_id` — both accept empty strings today.

---

### 4. Correctness — Unhandled `KeyError` on Response Parsing

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # raises KeyError if key is absent
)
```

If the external service returns a 2xx response but omits `notification_id` (malformed payload, API version mismatch, etc.), this raises an unhandled `KeyError` that will surface as a 500 with no useful log context.

**Fix:**

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Unexpected response from notification service")
```

---

### 5. Test Coverage — Wrong Exception Type in Failure Test

```python
# test_send_notification_service_unavailable
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # too broad
```

The production code catches `requests.RequestException`, not bare `Exception`. This test passes today but for the wrong reason — it would also pass if the code caught nothing at all, since `Exception` is a supertype. The test should raise a `requests.RequestException` subclass to actually validate the error handling path.

```python
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.ConnectionError("Connection refused")):
```

---

### 6. Test Coverage — Missing Cases

The following scenarios are not tested and represent realistic failure modes in a checkout flow:

| Missing Test | Why It Matters |
|---|---|
| External service returns 4xx/5xx (non-exception path) | `raise_for_status()` raises `HTTPStatusError`, not `RequestException` in httpx; behavior should be explicit |
| Response body missing `notification_id` | Correctness issue identified above |
| Empty or invalid `user_id` / `order_id` / `message` | Validates Pydantic constraints are actually enforced |
| Timeout specifically | Timeout is configured — worth asserting it triggers the 503 path |

---

### 7. Documentation — Docstring Is Incomplete

The existing docstring describes intent but omits raises and response contract, which matters for a shared service endpoint.

```python
"""
Send an order confirmation notification to the user via the external
notification service.

Args:
    request: NotificationRequest containing user_id, order_id, and message.

Returns:
    NotificationResponse with success=True and the upstream notification_id.

Raises:
    HTTPException 503: If the notification service is unreachable or returns an error.
    HTTPException 502: If the notification service returns an unexpected payload.
"""
```

| Priority | Issue |
|---|---|

| High | Handle missing `notification_id` in response |
| Medium | Fix failure test to use correct exception type |
| Medium | Add missing test cases |
