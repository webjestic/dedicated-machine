Decision: Request Changes

Summary: The implementation has a critical correctness issue — it uses the synchronous `requests` library inside an `async` endpoint, which will block the event loop and degrade performance under load. There are also meaningful gaps in input validation, error handling, and test coverage that need to be addressed before this is ready to merge.

---

## Detailed Findings

### 1. Performance — Blocking I/O in Async Context (High Severity)

**File:** `api/notifications.py`

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

`requests.post` is synchronous. Calling it inside an `async` def without `run_in_executor` or replacing it with an async HTTP client will block the entire event loop for the duration of the network call. Under any meaningful concurrency, this will serialize all requests through this endpoint.

**Fix:** Replace `requests` with `httpx` (async-native, already common in FastAPI projects):

```python
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
```

### 2. Error Handling — Unguarded Key Access (Medium Severity)

**File:** `api/notifications.py`

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if key is absent
)
```

If the external service returns a 2xx response but omits `notification_id` (malformed response, API change, etc.), this raises an unhandled `KeyError` that will surface as a 500 with no useful logging.

**Fix:**

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

Note the status code distinction: `503` is appropriate when the service is unreachable; `502` is more accurate when the upstream responded but with a bad payload.

---

### 3. Error Handling — Overly Broad Exception Catch in Tests (Medium Severity)

**File:** `tests/test_notifications.py`

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # too broad
```

The production code catches `requests.RequestException`, but the test raises a bare `Exception`. These are different types. The test passes today only because `Exception` is a superclass of `requests.RequestException` — but it's testing the wrong thing. If the production handler were accidentally narrowed or the exception type changed, this test would still pass while the real behavior broke.

**Fix:**

```python
import requests as requests_lib

with patch("api.notifications.requests.post",
           side_effect=requests_lib.RequestException("Connection refused")):
```

### 4. Input Validation — No Constraints on `message` Field (Medium Severity)

**File:** `api/notifications.py`

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded
```

`message` has no length constraint. An unbounded string field on a public-facing endpoint that gets forwarded to an external service is a risk — it can be used to send arbitrarily large payloads or, depending on how the downstream service renders the message, could contribute to injection issues.

**Fix:**

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

---

### 5. Security — Hardcoded Internal Service URL (Low-Medium Severity)

**File:** `api/notifications.py`

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

Hardcoding the service URL makes it impossible to vary across environments (dev, staging, prod) without code changes, and it couples the service URL to the codebase. This should come from configuration.

**Fix:**

```python
from api.config import settings  # or however your app manages config

response = await client.post(
    settings.notification_service_url,
    ...
)
```

---

### 6. Test Coverage — Missing Cases

**File:** `tests/test_notifications.py`

The following cases are not covered:

| Missing Test Case | Why It Matters |
|---|---|
| 4xx response from upstream (e.g., 400, 401) | `raise_for_status()` raises on these too; behavior should be verified |
| Missing `notification_id` in response | Once the key-access fix is applied, this path needs a test |
| Empty or invalid input fields | Validates that Pydantic constraints are enforced (once added) |
| Malformed JSON response from upstream | `response.json()` can raise; should be handled and tested |

---

### 7. Minor — Logger Name Convention

```python
logger = logging.getLogger(__name__)
```

This is correct and idiomatic. No change needed.
