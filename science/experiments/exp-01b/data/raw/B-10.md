**Decision: Request Changes**

**Summary:** This PR introduces a critical async correctness violation: `requests.post()` is a synchronous blocking call inside an `async def` route, which will block the event loop for the full duration of every external HTTP call and degrade every concurrent request in the application. This must be replaced with an awaitable HTTP client before merge. There are also secondary issues with input validation, error handling robustness, and test coverage that need to be addressed.

---

## Review

### 1. Correctness — ❌ BLOCKING I/O IN ASYNC ROUTE (Must Fix)

```python
# api/notifications.py

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(   # ← synchronous blocking call
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

This is the primary blocker. `requests` is a synchronous library. When this line executes, it calls into the OS network stack and **blocks the thread** — which means it blocks the event loop — for up to 5 seconds per request. During that window, FastAPI cannot process any other request. Under checkout load this becomes a cascading stall: concurrent users queue up waiting for the single thread to free.

The fix is to replace `requests` with `httpx` in async mode, which is already the idiomatic choice for FastAPI services:

```python
# api/notifications.py

import httpx  # replaces requests
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, field_validator
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    async with httpx.AsyncClient(timeout=5.0) as client:
        try:
            response = await client.post(   # ← now awaitable; event loop is free
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service returned error status: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service connection error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

> **Note on client lifecycle:** For production, prefer a shared `httpx.AsyncClient` instantiated at application startup (via a lifespan handler or dependency) rather than creating a new client per request. A per-request `AsyncClient` works correctly but incurs connection overhead on every call. The example above is the minimal correct fix; the shared-client pattern is the production-grade follow-up.

### 2. Error Handling — ⚠️ Two Issues

**2a. Overly broad exception catch in tests (minor in prod code, critical in tests):**

```python
# tests/test_notifications.py

with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # ← bare Exception
```

The production handler catches `requests.RequestException`. The test raises a bare `Exception`, which is a different type. This test passes today only because `Exception` is a superclass of `requests.RequestException` — but it is testing the wrong thing. It should raise the specific exception type the handler is written to catch. After migrating to `httpx`, use `httpx.RequestError`.

**2b. Missing `KeyError` guard on `data["notification_id"]`:**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # ← unguarded key access
)
```

If the external service returns a 200 with a malformed or unexpected body, this raises an unhandled `KeyError`, which FastAPI will surface as a 500. The contract with the notification service should be validated explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Notification service returned invalid response")
```

Note the status code distinction: `503` is appropriate when the service is unreachable; `502` is more accurate when the service responds but with a bad payload.

### 3. Input Validation — ⚠️ Empty Strings Accepted

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str
```

Pydantic will accept `""` for all three fields. An empty `user_id` or `order_id` will produce a silent no-op notification or a confusing downstream error. Add `min_length` constraints:

```python
from pydantic import BaseModel, field_validator

class NotificationRequest(BaseModel):
    user_id: str = Field(min_length=1)
    order_id: str = Field(min_length=1)
    message: str = Field(min_length=1)
```

### 4. Test Coverage — ⚠️ Gaps

The existing two tests cover the happy path and a generic failure. Missing cases:

| Missing Test | Why It Matters |
|---|---|
| HTTP 4xx from notification service (e.g. 400, 401) | `raise_for_status()` raises on these; should verify 503 is returned |
| Malformed response body (missing `notification_id`) | Exercises the `KeyError` / `502` path added above |
| Empty field validation (e.g. `user_id: ""`) | Verifies Pydantic constraints are enforced |
| Timeout specifically | `requests.Timeout` / `httpx.TimeoutException` is a distinct failure mode worth asserting on |

Also, after migrating to `httpx`, the patch target changes:

```python
# Before (requests)
with patch("api.notifications.requests.post", ...):

# After (httpx)
with patch("api.notifications.httpx.AsyncClient.post", ...):
# or, more robustly, use httpx's built-in mock transport:
transport = httpx.MockTransport(handler=your_handler)
```

Consider using `httpx`'s `MockTransport` or `respx` for cleaner async HTTP mocking rather than `unittest.mock`.

### 5. Security — ✅ No New Surface

The endpoint does not expose internal data, does not reflect user input into headers or URLs, and the external URL is hardcoded. No injection surface is introduced. Pydantic validation is present. No concerns here.

## Required Changes Before Merge

1. **Replace `requests` with `httpx` (async).** This is a correctness issue, not a style preference. The route is `async def`; all I/O inside it must be awaitable.
2. **Guard `data["notification_id"]`** against missing keys and return `502` on bad payloads.
3. **Add `min_length` validation** to `NotificationRequest` fields.
4. **Fix the test exception type** to match the specific exception the handler catches.
5. **Add missing test cases** for 4xx responses, malformed payloads, and empty field validation.
