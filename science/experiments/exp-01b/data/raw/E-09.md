**Decision: Request Changes**

**Summary:** This PR introduces a critical async correctness defect: `requests.post()` is a synchronous blocking call inside an `async def` route. Under checkout load, every invocation of this endpoint blocks the event loop for the full duration of the external HTTP call — up to the 5-second timeout — starving every other concurrent request in the application for that entire window. The fix is to replace `requests` with `httpx` (or `aiohttp`) and `await` the call; `httpx.AsyncClient` is a drop-in replacement that makes this correct with minimal refactoring. The tests also pass today precisely *because* `TestClient` runs synchronously and will never catch this class of defect — async integration tests with a real event loop (e.g. `pytest-asyncio` with `httpx.AsyncClient` as the test client) should be added to prevent this from regressing.

---

**Blocking issue — synchronous I/O on the event loop:**

```python
# Current: blocks the event loop for up to 5 seconds per request
import requests

response = requests.post(...)  # synchronous — event loop is frozen here
```

```python
# Required: yields control back to the event loop during I/O
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(       # non-blocking
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service unreachable: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

**Secondary issues to address before merge:**

1. **`AsyncClient` lifecycle** — if request volume justifies it, instantiate `httpx.AsyncClient` once at application startup (via FastAPI lifespan) rather than per-request to avoid connection overhead on every checkout.

2. **Exception handling is too broad in the test** — `test_send_notification_service_unavailable` patches with a bare `Exception`, but the production handler only catches `requests.RequestException`. After the `httpx` migration, verify the test patches `httpx.RequestError` specifically so the test actually validates the handler path rather than accidentally passing through a different code path.

3. **No validation on `data["notification_id"]`** — if the external service returns a malformed response (missing key, wrong type), this raises an unhandled `KeyError` or `ValidationError` that will surface as a 500. Add a guard or use `.get()` with an explicit fallback and error path.

4. **`message` field is unbounded** — Pydantic accepts any-length string. If this value is passed through to the external service or logged, add a `max_length` constraint to prevent abuse.
