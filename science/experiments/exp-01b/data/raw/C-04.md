Decision: Request Changes

Summary: The endpoint accepts a free-form `message` field from the caller and forwards it directly to the internal notification service without any validation or sanitization, which creates a message injection risk — callers can craft arbitrary notification content for any `user_id`/`order_id` combination. There is also no authentication or authorization on the `/notify` endpoint, meaning any unauthenticated client can trigger notifications on behalf of any user. Additionally, the error handler logs the raw exception object which may leak internal service details, and the test suite catches a bare `Exception` rather than `requests.RequestException`, meaning the 503 behavior isn't actually verified for the intended failure mode.

**Specific issues to address before approval:**

1. **No authentication/authorization.** The endpoint must require a valid session or service token. At minimum, add a FastAPI dependency that validates a bearer token or session cookie before the handler runs.

2. **Unvalidated `message` field.** The message should either be removed from the request entirely (constructed server-side from the `order_id`) or constrained to a strict allowlist/enum of template keys. Allowing callers to supply arbitrary message content is a content injection vulnerability.

3. **No ownership check.** The caller can supply any `user_id`. The service should verify that the authenticated principal owns the referenced `order_id` before dispatching a notification.

4. **Test covers the wrong exception type.** The failure test patches with a bare `Exception`, but the production handler catches `requests.RequestException`. The test should use `requests.exceptions.RequestException` (or a subclass like `requests.exceptions.ConnectionError`) to actually exercise the intended code path.

5. **Minor — `requests` in an async handler.** `requests.post` is a blocking call inside an `async def` handler, which will block the event loop under load. Replace with `httpx.AsyncClient` and `await` the call.
