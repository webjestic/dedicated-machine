**Decision: Request Changes**

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` FastAPI route — a critical defect that will block the event loop for the full duration of every external HTTP call, including the configured 5-second timeout. Under checkout load, this means every concurrent request across the entire application is frozen while one user's notification is in-flight; this is not a performance concern, it is a correctness failure under the actual runtime model. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if `requests` must be retained for some reason, offload the call via `asyncio.get_event_loop().run_in_executor` — but the `httpx` path is the right fix here.

---

### Blocking Issue (must fix before merge)

**Synchronous I/O on the event loop — `api/notifications.py`, line 24**

```python
# What's here — blocks the event loop entirely
response = requests.post(...)
```

```python
# What it must be
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service unreachable: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

The tests pass today because `TestClient` runs the ASGI app synchronously and the mock never touches a real socket. That is exactly why this class of bug survives CI — the test environment does not reproduce the event loop contention that kills production.

### Secondary Issues (fix in the same pass)

**1. `httpx` client lifecycle**

If this endpoint is called at checkout volume, constructing and tearing down an `AsyncClient` per request adds overhead and risks connection pool exhaustion. Prefer a shared client, initialized at application startup:

```python
# lifespan or module-level setup
http_client: httpx.AsyncClient | None = None

@asynccontextmanager
async def lifespan(app: FastAPI):
    global http_client
    http_client = httpx.AsyncClient()
    yield
    await http_client.aclose()
```

**2. Unvalidated external response field access**

```python
data["notification_id"]  # KeyError if the external service changes its contract
```

This will produce an unhandled `500` instead of a controlled error. Guard it:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Unexpected response from notification service")
```

**3. Test covers `Exception`, not `httpx.RequestError`**

After switching to `httpx`, the error-path test should raise the correct exception type so the test actually validates the handler behavior rather than the bare `except` fallback:

```python
with patch.object(client, "post", side_effect=httpx.ConnectError("Connection refused")):
    ...
```

### What the PR Gets Right

The Pydantic model, structured logging, `raise_for_status`, and the overall route shape are all correct. The error handling intent is right — the exception type just needs to track the library change. This is a single-library swap away from being mergeable.
