**Decision:** Request Changes

**Summary:** This PR introduces a critical event loop blocking issue: `requests.post()` is a synchronous I/O call inside an `async def` route. Under checkout load, every call to this endpoint will block the entire event loop for up to 5 seconds (the configured timeout), starving all concurrent requests across the application simultaneously. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if keeping synchronous I/O is unavoidable, convert the route to a plain `def` so FastAPI dispatches it to a thread pool. Additionally, the test suite uses `TestClient`, which runs synchronous and will not catch this class of bug — it passes today and will fail silently in production.

---

### Detailed Findings

#### 🔴 Critical — Synchronous I/O on the Event Loop

```python
# Current — blocks the event loop on every call
import requests

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(          # ← synchronous, blocks event loop
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

`requests` is built on `urllib3`, which performs blocking socket I/O. Calling it inside `async def` does not make it async — it runs synchronously on the event loop thread, blocking every other coroutine for the full duration of the network call. With a 5-second timeout configured, the worst-case blast radius is 5 seconds of total application paralysis per in-flight request.

**Fix — preferred:** Replace with `httpx` async client.

```python
# Correct — yields control to the event loop during I/O
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(       # ← non-blocking
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

**Alternative fix** — if `requests` must be retained for dependency reasons, remove the `async` keyword so FastAPI routes the handler through `anyio`'s thread pool executor, which isolates the blocking call from the event loop:

```python
# Acceptable fallback — FastAPI runs sync def routes in a threadpool
@router.post("/notify", response_model=NotificationResponse)
def send_notification(request: NotificationRequest) -> NotificationResponse:
    ...
```

This is strictly a fallback. The `httpx` path is correct; this path is merely safe.

#### 🟡 Moderate — Tests Will Not Catch This Bug

```python
# TestClient is synchronous — it does not exercise the event loop
client = TestClient(app)
```

`TestClient` wraps the ASGI app in a synchronous adapter. It will happily call a blocking `requests.post` inside an `async def` route and report a passing test. The bug is invisible to this test suite by construction.

After switching to `httpx`, update the tests to use `AsyncClient` with `pytest-anyio` or `pytest-asyncio`:

```python
import pytest
import httpx
from unittest.mock import patch, AsyncMock, MagicMock
from fastapi import FastAPI
from api.notifications import router

app = FastAPI()
app.include_router(router)


@pytest.mark.anyio
async def test_send_notification_success():
    mock_response = MagicMock()
    mock_response.json.return_value = {"notification_id": "ntf_abc123"}
    mock_response.raise_for_status.return_value = None

    with patch("httpx.AsyncClient.post", new_callable=AsyncMock, return_value=mock_response):
        async with httpx.AsyncClient(app=app, base_url="http://test") as ac:
            response = await ac.post("/notify", json={
                "user_id": "user_123",
                "order_id": "order_456",
                "message": "Your order has been confirmed.",
            })

    assert response.status_code == 200
    assert response.json()["success"] is True
    assert response.json()["notification_id"] == "ntf_abc123"
```

---

#### 🟡 Minor — Exception Handling Loses Specificity

The current `except requests.RequestException` block catches both connection errors and HTTP error responses uniformly. The `httpx` rewrite above separates `HTTPStatusError` (the server responded with 4xx/5xx) from `RequestError` (the connection failed entirely). This distinction is worth preserving in logs — a 500 from the notification service is a different operational signal than a connection timeout.

---

### Summary of Required Changes

| # | Severity | Issue | Action |
|---|----------|-------|--------|
| 1 | 🔴 Critical | `requests.post` blocks event loop inside `async def` | Replace with `await httpx.AsyncClient.post(...)` or change to `def` |
| 2 | 🟡 Moderate | `TestClient` cannot detect event loop blocking | Migrate to `httpx.AsyncClient` + `pytest-anyio` |
| 3 | 🟡 Minor | Exception handling conflates connection errors and HTTP errors | Split into `HTTPStatusError` / `RequestError` handlers |

The structure of this PR is solid — validation, error handling, logging, and test coverage are all present. The single change needed is replacing the synchronous HTTP client with an async one. That is a small diff with a large correctness impact.
