**Decision:** Request Changes

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` FastAPI route, which will block the event loop for the full duration of every external HTTP call — under checkout load, that means every concurrent request in the application stalls while one notification waits on a network round-trip. Beyond the blocking I/O issue, the test suite has a meaningful gap that would let a real failure mode pass silently. Both must be addressed before this merges.

---

## Detailed Findings

### 🔴 Critical — Blocking I/O Inside Async Route

**File:** `api/notifications.py`, line 24

```python
# Current — blocks the event loop
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. When this line executes inside `async def send_notification`, it does not yield control to the event loop — it parks the OS thread for up to 5 seconds. FastAPI runs on a single-threaded event loop (uvicorn/asyncio). During those 5 seconds, **every other request in the process is frozen**. This is not a theoretical concern: a notification service that is slow or degraded will take down the entire checkout flow, not just the notification path.

The fix is to replace `requests` with `httpx` in async mode:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()

# Reuse a single client across requests — avoids per-request connection overhead
# and respects connection pool limits.
_http_client = httpx.AsyncClient(timeout=5.0)


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        response = await _http_client.post(  # yields control; event loop stays live
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

**Why a module-level client?** Constructing a new `httpx.AsyncClient` per request creates and tears down a connection pool on every call. A shared client reuses connections (HTTP keep-alive) and enforces a sane concurrency ceiling via its built-in connection limits. If the application lifecycle needs explicit teardown, register it with FastAPI's `lifespan` handler.

### 🔴 Critical — Missing Input Validation on `message` Field

**File:** `api/notifications.py`, `NotificationRequest`

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded, no sanitization
```

`message` is forwarded directly into a JSON payload sent to an internal service. There is no length cap and no content constraint. A caller can send an arbitrarily large string, causing:

- Oversized payloads to the downstream service (potential DoS or rejection)
- Log bloat if the message is echoed in error paths

At minimum, add a length constraint:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1024)
```

`user_id` and `order_id` also currently accept empty strings, which would produce a malformed downstream request that is harder to debug than a clean 422 at the boundary.

### 🟡 Moderate — Test Catches the Wrong Exception Type

**File:** `tests/test_notifications.py`, `test_send_notification_service_unavailable`

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # too broad
```

The handler in `notifications.py` catches `requests.RequestException`, not bare `Exception`. This test passes today only because `Exception` is a supertype — but it is testing the wrong contract. If someone later narrows the `except` clause (which would be correct), this test would still pass while the real failure path breaks.

After switching to `httpx`, the test should raise the specific exception the handler actually catches:

```python
import httpx

def test_send_notification_service_unavailable():
    with patch.object(
        _http_client,  # or patch "api.notifications._http_client.post"
        "post",
        side_effect=httpx.ConnectError("Connection refused"),
    ):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503
    assert response.json()["detail"] == "Notification service unavailable"
```

Also add a test for `httpx.HTTPStatusError` (e.g., the downstream service returns 500) — that is a distinct code path and currently untested.

### 🟡 Moderate — Unhandled `KeyError` on Successful Response

**File:** `api/notifications.py`, line 33

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the notification service returns HTTP 200 but omits `notification_id` from the body, this raises an unhandled `KeyError`, which FastAPI will surface as a 500. That is the correct status code, but the error will be logged as an unhandled exception rather than a controlled failure, and the response body will be a raw traceback in debug mode.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error(
        "Notification service returned 200 but missing notification_id. "
        "order_id=%s",
        request.order_id,
    )
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

502 is more semantically accurate here than 503 — the upstream service responded but with a malformed payload.

The docstring describes what the endpoint does. Given the function is short and the Pydantic models are self-documenting, this is acceptable. If this service generates OpenAPI docs for consumers, consider adding `summary` and `description` to the `@router.post` decorator for richer schema output.
