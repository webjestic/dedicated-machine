Decision: Request Changes

Summary: The implementation has a critical correctness issue—using the synchronous `requests` library inside an `async` FastAPI handler will block the event loop under load, which is a meaningful performance regression for a checkout-critical path. Beyond that, there are security concerns around unvalidated user-controlled input being forwarded to an internal service, missing authentication on the endpoint, and gaps in test coverage that leave important failure modes untested.

---

## Detailed Findings

### 1. Correctness

**🔴 Blocking HTTP call in async handler**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, will block
```

`requests.post` is synchronous. Calling it inside an `async` def without running it in a thread pool means every in-flight request blocks the entire event loop until the external call completes (up to the 5-second timeout). Under checkout load this will cause cascading latency across all endpoints.

```python
# Option A — switch to httpx with native async support (preferred)
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
        )
        response.raise_for_status()
        ...

# Option B — if requests must be kept, offload to a thread pool
import asyncio
from functools import partial

loop = asyncio.get_event_loop()
response = await loop.run_in_executor(
    None,
    partial(requests.post, url, json={...}, timeout=5)
)
```

**🟡 Missing key error handling on response payload**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a 2xx but omits `notification_id`, this raises an unhandled `KeyError` that will surface as a 500. Use `.get()` with a fallback or validate the response shape explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

### 2. Security

**🔴 No authentication or authorization on the endpoint**

The `/notify` endpoint accepts arbitrary `user_id`, `order_id`, and `message` values from any caller with no authentication. In a checkout service this should require at minimum a verified session or service-to-service token. Without it, any unauthenticated actor can spam the notification service or send arbitrary messages to users.

```python
from fastapi import Depends
from api.auth import require_internal_token  # your existing auth dependency

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_token),
) -> NotificationResponse:
    ...
```

**🟡 User-controlled `message` field forwarded without sanitization**

The `message` field is passed verbatim to the internal notification service. Depending on how that service renders messages (email HTML, SMS, push), this could be a vector for injection. At minimum, add a length constraint and document the trust boundary:

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=500, description="Plain-text notification body")
```

**🟡 Hardcoded internal service URL**

```python
"https://notifications.internal/send"
```

Service URLs belong in configuration, not source code. This makes environment promotion (staging vs. production) error-prone and prevents rotation without a code change:

```python
# config.py
from pydantic import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

settings = Settings()

# notifications.py
from api.config import settings

response = await client.post(f"{settings.notification_service_url}/send", ...)
```

### 3. Test Coverage

**🔴 Failure mode test patches the wrong exception type**

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The handler catches `requests.RequestException`, not bare `Exception`. This test passes today only because an unhandled `Exception` also produces a 500-level response—but it's not testing the intended code path. The test should verify the specific catch:

```python
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.RequestException("Connection refused")):
    response = client.post("/notify", json={...})

assert response.status_code == 503
assert response.json()["detail"] == "Notification service unavailable"
```

**🟡 Missing test cases**

| Scenario | Covered? |
|---|---|
| External service returns 4xx/5xx | ❌ |
| External service returns 2xx with missing `notification_id` | ❌ |
| Request with missing required fields | ❌ |
| Timeout specifically (`requests.Timeout`) | ❌ |
| Unauthenticated request (once auth is added) | ❌ |

### 4. Performance

The blocking `requests` issue above is the primary concern (covered in Correctness). Additionally, consider whether the notification call should be fire-and-forget via a background task rather than blocking the checkout response:

```python
from fastapi import BackgroundTasks

@router.post("/notify", status_code=202)
async def send_notification(
    request: NotificationRequest,
    background_tasks: BackgroundTasks,
) -> dict:
    background_tasks.add_task(_dispatch_notification, request)
    return {"accepted": True}
```

This is an architectural decision worth discussing with the team—it depends on whether the caller needs a synchronous confirmation.

---

### 5. Naming / Documentation

These are minor and non-blocking:

- `request` shadows FastAPI's own `Request` type in some contexts; `notification_request` or `payload` is clearer.
- The docstring is accurate but could note the 503 failure mode for API consumers.
- `NotificationResponse.success` is always `True` on the happy path; consider removing it or making it meaningful (e.g., set it based on the external service's own success field).
