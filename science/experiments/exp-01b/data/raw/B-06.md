**Decision:** Request Changes

**Summary:** This PR ships a synchronous `requests.post` call inside an `async def` FastAPI route, which will block the event loop for the full duration of every external HTTP call — under checkout load, that means every concurrent request in the application stalls while one notification waits on a network response. Beyond that critical flaw, the test suite has a gap that masks a real failure mode, and there is a latent security surface in the message field that needs to be addressed before this touches a checkout flow.

---

## Detailed Findings

### 1. 🔴 Blocking I/O Inside Async Route (Correctness / Performance)

**File:** `api/notifications.py`, line 24

```python
# Current — blocks the event loop
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. When this line executes inside `async def send_notification`, it does not yield control back to the event loop. It occupies the single thread running the event loop for up to 5 seconds per call. Every other request — including unrelated routes — queues behind it.

This is the exact failure mode that looks correct in isolation, passes all tests, and then causes cascading latency spikes under production load. The `TestClient` used in the test suite runs the app synchronously, so the tests will never catch this.

**Required fix — replace `requests` with `httpx` and use its async client:**

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, Field
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=1000)


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    Uses async HTTP to avoid blocking the event loop.
    """
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error(
            "Notification service returned error status: %s %s",
            e.response.status_code,
            e.response.text,
        )
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

If migrating to `httpx` is not immediately possible, the acceptable fallback is to offload the blocking call to a thread pool using `asyncio.to_thread` — but that is a workaround, not a solution, and should be treated as temporary:

```python
import asyncio

# Acceptable only as a short-term bridge
response = await asyncio.to_thread(
    requests.post,
    "https://notifications.internal/send",
    json={...},
    timeout=5,
)
```

### 2. 🔴 Test Gap: Wrong Exception Type Tested (Test Coverage)

**File:** `tests/test_notifications.py`, line 28

```python
# Current — patches with a bare Exception, not a requests.RequestException
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The route catches `requests.RequestException`. A bare `Exception` is also caught by that handler incidentally, but this test does not verify the actual contract. More importantly: if the `requests` import is later swapped for `httpx` (as it should be), this test will continue to pass even if the new exception types are not handled correctly, because `Exception` catches everything.

**Required fix — test the specific exception type the handler is written to catch, and add a test for HTTP error status responses:**

```python
import pytest
from unittest.mock import patch, MagicMock, AsyncMock
import httpx
from fastapi.testclient import TestClient
from api.notifications import router
from fastapi import FastAPI

app = FastAPI()
app.include_router(router)
client = TestClient(app)


def test_send_notification_success():
    mock_response = MagicMock()
    mock_response.json.return_value = {"notification_id": "ntf_abc123"}
    mock_response.raise_for_status.return_value = None
    mock_response.status_code = 200

    with patch("api.notifications.httpx.AsyncClient") as mock_client_cls:
        mock_client = AsyncMock()
        mock_client.__aenter__ = AsyncMock(return_value=mock_client)
        mock_client.__aexit__ = AsyncMock(return_value=None)
        mock_client.post = AsyncMock(return_value=mock_response)
        mock_client_cls.return_value = mock_client

        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 200
    assert response.json()["success"] is True
    assert response.json()["notification_id"] == "ntf_abc123"


def test_send_notification_connection_error():
    """Verifies that network-level errors return 503."""
    with patch("api.notifications.httpx.AsyncClient") as mock_client_cls:
        mock_client = AsyncMock()
        mock_client.__aenter__ = AsyncMock(return_value=mock_client)
        mock_client.__aexit__ = AsyncMock(return_value=None)
        mock_client.post = AsyncMock(
            side_effect=httpx.RequestError("Connection refused")
        )
        mock_client_cls.return_value = mock_client

        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503


def test_send_notification_upstream_5xx():
    """Verifies that a 5xx from the notification service returns 503."""
    mock_response = MagicMock()
    mock_response.status_code = 500
    mock_response.raise_for_status.side_effect = httpx.HTTPStatusError(
        "Server Error", request=MagicMock(), response=mock_response
    )

    with patch("api.notifications.httpx.AsyncClient") as mock_client_cls:
        mock_client = AsyncMock()
        mock_client.__aenter__ = AsyncMock(return_value=mock_client)
        mock_client.__aexit__ = AsyncMock(return_value=None)
        mock_client.post = AsyncMock(return_value=mock_response)
        mock_client_cls.return_value = mock_client

        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503
```

### 3. 🟡 Missing Input Bounds on `message` Field (Security)

**File:** `api/notifications.py`, `NotificationRequest`

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # unbounded
```

`message` has no length constraint. In a checkout flow, this field is caller-controlled. Without a bound, a caller can send arbitrarily large payloads that get forwarded verbatim to the internal notification service. This is a vector for downstream abuse of `notifications.internal` and a potential DoS amplifier.

**Required fix:**

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

The exact limits should be aligned with whatever the notification service documents as its own constraints. If those are unknown, conservative defaults are better than none.

### 4. 🟡 `notification_id` Not Validated Before Use (Error Handling)

**File:** `api/notifications.py`, lines 35–38

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the notification service returns a 200 with a body that does not include `notification_id`, this raises an unhandled `KeyError`, which FastAPI will surface as a 500. That is the wrong status code for a malformed upstream response, and it leaks an uncontrolled error to the caller.

**Required fix:**

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error(
        "Notification service returned success but missing notification_id: %s",
        data,
    )
    raise HTTPException(
        status_code=502,
        detail="Notification service returned an unexpected response",
    )
return NotificationResponse(success=True, notification_id=notification_id)
```
