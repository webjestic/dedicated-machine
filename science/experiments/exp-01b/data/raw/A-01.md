**Decision:** Request Changes

**Summary:** The route is declared `async def` but calls `requests.post()` — a synchronous, blocking HTTP call that will hold the event loop for the full duration of the network round-trip (up to the 5-second timeout). Under checkout load, every concurrent request in the application stalls while this call completes; this is a production correctness issue, not a style preference. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if `requests` must be retained for some reason, offload it to a thread pool via `asyncio.run_in_executor` — but the async-native path with `httpx` is the right fix here.

---

**Detailed findings:**

**[Blocker] Synchronous I/O inside async route**

```python
# Current — blocks the event loop on every request
import requests

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # holds the loop for up to 5 seconds
```

```python
# Required — async-native HTTP call
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service unreachable: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

The `async with httpx.AsyncClient()` per-request pattern is acceptable here, but if this endpoint will be called at volume, a shared client instance (created at startup, closed at shutdown via FastAPI lifespan) is preferable to avoid connection overhead on every request.

---

**[Minor] Test suite masks the core defect**

The tests mock `requests.post` at the module level, which means they pass regardless of whether the call is synchronous or asynchronous. `TestClient` runs the app in a synchronous context via `anyio`, so blocking behavior is invisible in tests — exactly why this class of bug ships. After the fix, the tests should mock `httpx.AsyncClient` instead (or use `respx` for async HTTP mocking), and at least one test should verify the happy path with a realistic async mock rather than a `MagicMock`.

---

**[Minor] Unvalidated external response field access**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a 200 with a malformed body, this raises an unhandled `KeyError` that becomes a 500. Use `.get()` with a fallback or parse through a Pydantic model, and handle the missing-field case explicitly.

---

**[Minor] `message` field has no length constraint**

The `message` field on `NotificationRequest` is an unbounded string. Since this value is forwarded directly to an external service, add a `max_length` constraint to prevent oversized payloads from being proxied outward.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=500)
```
