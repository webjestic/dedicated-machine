**Decision: Request Changes**

**Summary:** The route is declared `async def` but calls `requests.post()` — a synchronous, blocking HTTP call — directly on the event loop. Under checkout load, every concurrent request to this endpoint will stall while waiting on the external notification service, and that stall blocks the *entire* event loop, degrading every other route in the application simultaneously. This is the primary blocker and must be resolved before merge.

---

### Blocking Issue: Synchronous I/O Inside an Async Route

```python
# This blocks the event loop for the full duration of the HTTP call (up to 5s)
response = requests.post(
    "https://notifications.internal/send",
    ...
    timeout=5,
)
```

`requests` is a synchronous library. Calling it inside `async def` does not make it async — it runs on the event loop thread and holds it until the call completes. The `async` keyword here is not protective; it is misleading.

**Fix: Replace `requests` with `httpx.AsyncClient`**

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

> **Note on `httpx.AsyncClient` lifecycle:** For a high-traffic checkout endpoint, consider instantiating the client once at application startup (e.g., via FastAPI's lifespan context) rather than creating and tearing down a new client per request. Connection pooling is lost when the client is recreated on every call.

### Secondary Issue: Overly Broad Exception Handling in Tests

```python
# This catches everything, including programming errors — it will mask bugs
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`. The test should mirror that specificity. With the `httpx` fix, use `httpx.RequestError`:

```python
with patch(
    "api.notifications.httpx.AsyncClient.post",
    side_effect=httpx.ConnectError("Connection refused"),
):
```

This ensures the test is actually validating the error path the code handles, not accidentally passing because an unrelated exception bubbled up and happened to produce a 503.

---

### What Is Done Well

- Pydantic models for input validation are correct and complete
- `raise_for_status()` is the right pattern for surfacing HTTP-level errors
- Logging at the error site with the exception included is correct
- The 503 response code is semantically appropriate for a downstream service failure

### Required Before Merge

1. Replace `requests` with `httpx` (or `aiohttp`) and `await` the call
2. Update tests to use `pytest-anyio` or `httpx`'s async test utilities and patch at the correct level with appropriately typed exceptions
