Decision: Request Changes

Summary: The implementation has a critical architectural flaw — a synchronous `requests.post` call inside an `async` endpoint will block the event loop, degrading performance under any meaningful load. Beyond that, there are several correctness, security, and test coverage gaps that need to be addressed before this is merge-ready.

---

## Detailed Findings

### 1. Performance — Blocking I/O in Async Context (Critical)

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

`requests` is a synchronous library. Calling it inside an `async` def ties up the event loop for the entire duration of the network call, serializing all concurrent requests. This is a correctness issue for an async service, not just a style concern.

**Fix:** Replace with `httpx` (async-native) or run the blocking call in a thread pool.

```python
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

---

### 2. Security — Hardcoded Internal URL (High)

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

Hardcoding service URLs creates several problems: it makes the URL impossible to rotate without a code change, it leaks internal topology into version history, and it prevents environment-specific configuration (dev/staging/prod). It also makes the service fragile if the URL ever changes.

**Fix:** Load from configuration or environment variable with a clear, validated default.

```python
import os
from pydantic import AnyHttpUrl
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: AnyHttpUrl = "https://notifications.internal/send"

settings = Settings()
```

---

### 3. Security — No Authentication on the Endpoint (High)

The `/notify` endpoint has no authentication or authorization guard. Any caller — internal or external — can trigger notifications for arbitrary `user_id`/`order_id` pairs. In a checkout flow, this endpoint should at minimum require a valid session token or be restricted to internal service-to-service calls.

**Fix:** Add a dependency for authentication appropriate to your service's auth model (e.g., OAuth2, API key, internal service token).

```python
from fastapi import Depends
from api.auth import require_internal_service  # your existing auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_service),
) -> NotificationResponse:
```

---

### 4. Security — `message` Field Is Unbounded (Medium)

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no length constraint
```

An unbounded `message` field can be used to send arbitrarily large payloads to the downstream notification service, potentially enabling abuse or amplification attacks.

**Fix:** Add field constraints.

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)
```

---

### 5. Error Handling — Swallowing Non-`RequestException` Errors (Medium)

```python
except requests.RequestException as e:
    logger.error("Notification service error: %s", e)
    raise HTTPException(status_code=503, ...)
```

Only `requests.RequestException` is caught. If `data["notification_id"]` is missing from the response, or `response.json()` raises a `ValueError`/`KeyError`, the exception propagates as an unhandled 500. The test suite actually catches this gap — `test_send_notification_service_unavailable` patches with a bare `Exception`, which would *not* be caught by the current handler.

**Fix:** Handle the response parsing separately with a meaningful error.

```python
except requests.RequestException as e:
    logger.error("Notification service error: %s", e)
    raise HTTPException(status_code=503, detail="Notification service unavailable")
except (KeyError, ValueError) as e:
    logger.error("Unexpected response from notification service: %s", e)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

### 6. Test Coverage — Gaps (Medium)

The test for the failure path patches with a bare `Exception`, which does not match the actual `requests.RequestException` handler. This means the test is not actually exercising the code path it claims to test. Additional cases are also missing entirely.

| Scenario | Covered |
|---|---|
| Successful notification | ✅ |
| `requests.RequestException` (correct type) | ❌ |
| HTTP 4xx/5xx from downstream (via `raise_for_status`) | ❌ |
| Missing `notification_id` in response | ❌ |
| Invalid/missing request fields (Pydantic validation) | ❌ |
| Timeout specifically | ❌ |

**Fix the existing failure test:**

```python
import requests as requests_lib

def test_send_notification_service_unavailable():
    with patch(
        "api.notifications.requests.post",
        side_effect=requests_lib.RequestException("Connection refused"),
    ):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })

    assert response.status_code == 503
    assert response.json()["detail"] == "Notification service unavailable"
```

---

### 7. Correctness — No Retry or Circuit Breaker Logic (Low / Worth Noting)

For a checkout-critical notification path, a single attempt with a 5-second timeout and an immediate 503 is fragile. A transient blip in the notification service will surface directly to the user. Consider whether a retry with exponential backoff (e.g., via `tenacity`) or an async task queue is more appropriate for this use case.

---

### 8. Documentation — Log Message Leaks Internal Detail (Low)

```python
logger.error("Notification service error: %s", e)
```

Depending on what `e` contains, this may log internal service URLs or stack details to a log aggregator. Ensure your logging configuration scrubs or redacts sensitive fields before this reaches production.
