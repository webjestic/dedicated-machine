## Review

### Critical Issue: Synchronous I/O on the Async Event Loop

The core problem is immediately visible:

```python
# async def + blocking requests.post = event loop starvation
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(  # <-- synchronous, blocking
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

`requests.post` is synchronous. Calling it inside an `async def` route handler does **not** run it in a thread pool — it blocks the event loop directly. The `async` keyword here provides no protection; it makes the problem worse by creating a false sense of safety. Every other in-flight request across the entire application waits, frozen, for up to 5 seconds per invocation. Under checkout load this compounds immediately.

### Required Fix

Replace `requests` with `httpx` (async-native) and `await` the call:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    try:
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
    except httpx.HTTPStatusError as e:
        logger.error("Notification service returned error status: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
    except httpx.RequestError as e:
        logger.error("Notification service connection error: %s", e)
        raise HTTPException(status_code=503, detail="Notification service unavailable")
```

An acceptable alternative is to make the handler a plain `def` (not `async def`), which causes FastAPI to dispatch it to a thread pool automatically — but the `httpx` approach is preferable for a service at this traffic level.

### Secondary Issues

**Unvalidated external data written into the response.** `data["notification_id"]` is taken directly from the external service response and returned to the client with no validation. If the key is missing this raises an unhandled `KeyError`; if the value is malicious it passes through unchecked. Validate with Pydantic or at minimum guard the key access:

```python
notification_id = data.get("notification_id")
if not notification_id or not isinstance(notification_id, str):
    logger.error("Unexpected response shape from notification service: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

**`message` field is unbounded and unvalidated.** Any string of any length and content is accepted and forwarded to an internal service. Add constraints:

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    message: str = Field(..., min_length=1, max_length=1000)
```

**Test suite does not cover the blocking I/O problem.** `TestClient` runs synchronously, so the tests pass even with the broken async handler. The tests also mock at the wrong layer for the fix — once `httpx` is adopted, mocks need to target `httpx` (e.g., `respx` or `pytest-httpx`). The error test patches `Exception` rather than `requests.RequestException`, which means it would pass even if the except clause were removed entirely.

---

**Decision:** Request Changes

**Summary:** The handler uses `requests.post` — a synchronous blocking call — inside an `async def` route, which will block the entire event loop for up to 5 seconds per request under the stated checkout load. This must be replaced with `httpx.AsyncClient` and `await` before this can merge. Two additional issues need addressing: the `message` field accepts arbitrary unbounded input that is forwarded to an internal service without sanitization, and the external service response is written back to the client without validation or key-existence checks.
