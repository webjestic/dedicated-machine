Decision: Request Changes

Summary: The endpoint accepts a free-form `message` field from the caller with no length cap, format constraint, or server-side content policy, which creates a prompt/content injection risk if that field is ever relayed to users or downstream systems without sanitization. More critically, the endpoint has no authentication or authorization guard — any unauthenticated caller can trigger notifications on behalf of arbitrary `user_id`/`order_id` pairs, which is a meaningful abuse vector in a checkout flow. The test suite also patches the base `Exception` class rather than `requests.RequestException`, meaning the 503 error path isn't actually exercising the handler that's in production code.

**Specific issues to address before approval:**

1. **No authentication/authorization.** The `/notify` route has no dependency on an auth scheme (e.g., `Depends(get_current_user)`). Any anonymous request can trigger notifications for any user. Add an auth dependency and verify the requesting principal is authorized to act on the given `user_id`.

2. **Unbounded, unvalidated `message` field.** `message: str` with no constraints is risky. At minimum add `max_length` via Pydantic's `Field(max_length=...)`. If the message content is ever rendered downstream, establish an allowlist or sanitization step.

3. **`user_id` and `order_id` have no format validation.** These should be validated against an expected format (e.g., a regex or UUID type) to prevent unexpected values from reaching the external service.

4. **Synchronous `requests` call inside an async handler.** `requests.post` is blocking and will stall the event loop under load. Replace with `httpx.AsyncClient` and `await` the call.

5. **External service URL is hardcoded.** `"https://notifications.internal/send"` should come from configuration (e.g., `settings.notification_service_url`) so it can vary per environment and be rotated without a code change.

6. **Test patches wrong exception type.** `side_effect=Exception(...)` bypasses the `except requests.RequestException` handler; the 503 response in the test is actually coming from FastAPI's unhandled exception path, not your handler. Patch with `requests.RequestException` to test the actual code path.
