**Decision: Request Changes**

**Summary:** This PR introduces a critical event loop violation: `requests.post()` is a synchronous blocking call inside an `async def` route. Under real checkout load, every concurrent request will stall while this call waits on the external notification service — a single slow or unresponsive upstream can freeze the entire application. Replace `requests` with `httpx.AsyncClient` (or `aiohttp`) and `await` the call. The tests also mask this bug entirely because `TestClient` runs synchronously and `patch` replaces the call before it can block — green tests here are not a signal of correctness under the actual async runtime.

---

**Required fix — replace the blocking call:**

```python
# Before (blocks the event loop)
import requests

response = requests.post("https://notifications.internal/send", json={...}, timeout=5)

# After (non-blocking, correct under async runtime)
import httpx

async with httpx.AsyncClient() as client:
    response = await client.post(
        "https://notifications.internal/send",
        json={...},
        timeout=5.0,
    )
```

**Update the exception handling to match:**

```python
# Before
except requests.RequestException as e:

# After
except httpx.HTTPError as e:
```

---

**Update the tests to reflect the correct async client:**

```python
# Before — patches the wrong library and hides the blocking call
with patch("api.notifications.requests.post", return_value=mock_response):

# After — patches httpx correctly
with patch("httpx.AsyncClient.post", new_callable=AsyncMock, return_value=mock_response):
```

Also note: `test_send_notification_service_unavailable` patches with a bare `Exception`, which bypasses the `httpx.HTTPError` catch block. That test is not actually validating the error handling path — it should raise `httpx.RequestError` or `httpx.HTTPStatusError` to match what the handler catches.

---

**Secondary concern — input not sanitized before forwarding:**

The `message` field is passed directly to the external service without any length cap or content constraints. At minimum, add a `max_length` constraint on the Pydantic model to prevent oversized payloads being proxied upstream:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

The blocking `requests` call is a hard blocker. Everything else is clean — the structure, error logging, and response model are all solid. Fix the async violation and this is close to mergeable.
