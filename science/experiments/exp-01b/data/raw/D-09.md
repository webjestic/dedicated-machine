Decision: Request Changes

Summary: The implementation has a critical correctness issue — it uses the synchronous `requests` library inside an `async` endpoint, which will block the event loop and degrade performance under load. There are also several security, correctness, and test coverage gaps that need to be addressed before this is mergeable.

---

## Detailed Findings

### 1. Performance — Blocking I/O in Async Context (High Severity)

**File:** `api/notifications.py`

```python
# Current — blocks the event loop
response = requests.post(...)
```

Using `requests` (synchronous) inside an `async def` endpoint blocks the entire event loop for the duration of the HTTP call. Under concurrent load this will serialize all requests through this endpoint.

```python
# Fix — use httpx with async support
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

If migrating to `httpx` is out of scope, the synchronous call must be offloaded via `asyncio.to_thread()` at minimum.

---

### 2. Security — Unvalidated `message` Field (High Severity)

**File:** `api/notifications.py`

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # No length or content constraints
```

The `message` field is forwarded directly to an external service with no length cap or sanitization. This creates a vector for payload inflation attacks and potentially injection into the downstream service.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[\w\-]+$")
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[\w\-]+$")
    message: str = Field(..., min_length=1, max_length=1000)
```

`user_id` and `order_id` should also be constrained — accepting arbitrary strings and forwarding them to an internal service is risky.

---

### 3. Security — Hardcoded Internal Service URL (Medium Severity)

**File:** `api/notifications.py`

```python
"https://notifications.internal/send",
```

Hardcoded URLs make environment-specific deployment impossible and prevent rotating or overriding the endpoint without a code change. This should come from configuration.

```python
from api.config import settings  # or however your app manages config

response = await client.post(
    settings.notification_service_url,
    ...
)
```

---

### 4. Correctness — `KeyError` on Missing `notification_id` (Medium Severity)

**File:** `api/notifications.py`

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # unguarded dict access
)
```

If the external service returns a 2xx response with a malformed or unexpected body, this raises an unhandled `KeyError`, which will surface as a 500 with no useful context. The `raise_for_status()` call does not protect against this.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

### 5. Test Coverage — Gaps (Medium Severity)

**File:** `tests/test_notifications.py`

Several important cases are not covered:

| Missing Test Case | Why It Matters |
|---|---|
| HTTP 4xx/5xx from external service | `raise_for_status()` path is untested |
| Malformed JSON response (missing `notification_id`) | The `KeyError` bug above is invisible without this |
| Invalid input (empty strings, oversized `message`) | Pydantic validation is claimed but untested |
| `requests.Timeout` specifically | Timeout handling should be verified explicitly |

Additionally, the failure test patches `Exception` rather than `requests.RequestException`, which means it doesn't actually test the handler's `except` clause — it works by accident because FastAPI catches unhandled exceptions too.

```python
# Current — tests the wrong thing
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):

# Correct
from requests.exceptions import RequestException
with patch("api.notifications.requests.post",
           side_effect=RequestException("Connection refused")):
```

---

### 6. Documentation — Incomplete Docstring (Low Severity)

**File:** `api/notifications.py`

The docstring describes the happy path only. Non-obvious behavior (503 on any `RequestException`, the timeout value) should be noted, especially since this is called during checkout.

```python
"""
Send an order confirmation notification to the user via the internal
notification service.

Raises:
    HTTPException 503: If the notification service is unreachable or
        returns a non-2xx response.
    HTTPException 502: If the service response is missing expected fields.

Note:
    Uses a 5-second timeout. Failures do not roll back the order.
"""
```
