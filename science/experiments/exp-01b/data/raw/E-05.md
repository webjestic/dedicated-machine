**Decision:** Request Changes

**Summary:** The route is declared `async def` but calls `requests.post()` — a synchronous, blocking HTTP call — directly on the event loop. Under checkout load, every concurrent request across the entire application is frozen for up to the full 5-second timeout duration while this call blocks; the tests pass only because `TestClient` runs synchronously and the mock never actually blocks. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if `requests` must be retained, move the call into a thread pool via `await asyncio.get_event_loop().run_in_executor()` — but the former is the correct fix for a service at this traffic level.

---

### Detailed Findings

#### 🔴 Critical — Synchronous I/O on the Event Loop

```python
# Current — blocks the event loop
import requests

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous: freezes every concurrent request
```

```python
# Required fix — non-blocking async HTTP
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"]
        )
```

The blast radius here is not limited to this endpoint. The event loop is shared across the entire application process. One request hitting a 5-second `requests.post()` timeout means 5 seconds of total paralysis for every other in-flight request — checkout confirmations, inventory checks, payment callbacks — everything. Under a sales event with concurrent checkout load, this compounds: multiple simultaneous calls to this endpoint each independently block the loop, and the failures cascade.

The tests do not catch this because `TestClient` is synchronous by design and the mock returns instantly. The bug is invisible in CI and catastrophic in production.

---

#### 🟡 Secondary — Exception Handling Scope Is Too Broad in Tests

```python
# test_send_notification_service_unavailable
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # catches everything
```

The test raises a bare `Exception`, but the handler catches `requests.RequestException`. After the `httpx` migration, this test should raise `httpx.RequestError` to actually validate the error handling path. A test that passes by catching a broader exception than the handler targets is not testing the handler.

---

#### 🟡 Secondary — No Input Sanitization Before Forwarding

```python
json={
    "user_id": request.user_id,
    "order_id": request.order_id,
    "message": request.message,
}
```

Pydantic validates that these fields are strings and are present — it does not constrain their content. `message` in particular is forwarded verbatim to an internal service with no length limit or content validation. At minimum, add a `max_length` constraint on `message` and consider whether `user_id` and `order_id` should be validated against an expected format (e.g., `pattern=r'^[a-zA-Z0-9_-]+$'`). This is a lower-priority concern than the blocking I/O, but it is worth addressing before this ships.

---

#### ✅ What Is Done Well

- Pydantic models for request/response are correct and appropriately typed
- `raise_for_status()` is called before accessing response data
- Errors are logged with the exception value before re-raising as `HTTPException`
- The `503` status code is semantically correct for a downstream service failure
- The `timeout=5` on the HTTP call is present (though it does not mitigate the blocking problem, it at least bounds the worst case)

---

### Required Before Approval

1. Replace `requests` with `httpx` (or equivalent async HTTP client) and `await` the call
2. Update exception handling to catch `httpx.HTTPStatusError` and `httpx.RequestError`
3. Update tests to raise the correct exception type for the error path
