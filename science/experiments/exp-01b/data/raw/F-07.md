## Code Review

### Critical Issue: Synchronous I/O on the Async Event Loop

The core defect is on this line:

```python
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    ...
    response = requests.post(...)  # BLOCKS THE EVENT LOOP
```

`requests.post` is a synchronous, blocking call. Placing it inside an `async def` handler does **not** make it non-blocking — it runs directly on the event loop thread and blocks it entirely for the duration of the call. With a `timeout=5`, every concurrent request across the application stalls for up to 5 seconds per checkout. This is not a theoretical concern; under sale-event load this will cause cascading latency and timeouts service-wide.

**Required fix:** Replace `requests` with `httpx` (async-native) and await the call:

```python
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"]
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service returned error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service connection error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

An acceptable alternative is to keep `requests` but move the handler to a thread pool via `run_in_executor` or by declaring the route as a plain `def` (FastAPI will dispatch sync routes to a threadpool automatically). However, the `httpx` approach is idiomatic and preferred.

### Secondary Issues

**1. Unvalidated external response structure**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

A missing or renamed `notification_id` field raises an unhandled `KeyError`, which will produce a 500. Validate the response shape explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Unexpected response from notification service")
```

**2. Input fields are unbounded strings**

`user_id`, `order_id`, and `message` accept strings of arbitrary length. At minimum, add `max_length` constraints:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1024)
```

**3. Test coverage gap**

`test_send_notification_service_unavailable` patches with a bare `Exception`, but the production handler only catches `requests.RequestException`. These are inconsistent — the test passes for the wrong reason. The test should raise `requests.RequestException` (or `httpx.RequestError` after the fix) to actually exercise the error path being tested.

---

### Summary

**Decision: Request Changes**

**Summary:** The endpoint uses `requests.post` inside an `async` handler, which blocks the event loop for the full duration of the external call — up to 5 seconds per request — starving every concurrent connection in the service. This must be replaced with an async HTTP client (`httpx`) before merge. Two secondary issues also need addressing: the external response payload is accessed without validation (risking unhandled `KeyError` → 500), and the error-path test patches the wrong exception type, giving false confidence in coverage.
