**Decision:** Request Changes

**Summary:** This PR introduces a critical async correctness violation: `requests.post()` is a synchronous blocking call inside an `async def` route, which will block the entire event loop for the duration of every external HTTP call — starving all concurrent requests application-wide. This is the primary blocker, but there are several additional issues across correctness, security, and test coverage that must be addressed before this is mergeable.

---

## Review

### 1. Correctness — BLOCKING CALL IN ASYNC CONTEXT ❌

This is the primary blocker.

```python
# api/notifications.py

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(   # ← synchronous, blocking
        "https://notifications.internal/send",
        ...
        timeout=5,
    )
```

`requests` is a synchronous library. When this line executes, it holds the event loop thread for up to 5 seconds (the configured timeout). During that time, **every other request to the entire application is frozen** — not slowed down, frozen. This is not a theoretical concern; it is a production failure mode that appears only under real concurrency and is invisible in tests.

The checklist says "Tests pass." Of course they do. `TestClient` runs synchronously. It will never surface this bug.

**Required fix:** Replace `requests` with `httpx` in async mode:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
import logging

logger = logging.getLogger(__name__)
router = APIRouter()


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user.
    """
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
                timeout=5.0,
            )
            response.raise_for_status()
            data = response.json()
            return NotificationResponse(
                success=True,
                notification_id=data["notification_id"],
            )
        except httpx.HTTPStatusError as e:
            logger.error("Notification service returned error status: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service connection error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

> **Note on the `AsyncClient` lifecycle:** For a high-traffic checkout endpoint, constructing a new `AsyncClient` per request adds connection overhead. Consider injecting a shared, application-lifespan client via FastAPI's dependency injection and lifespan context manager. For this PR's scope, per-request is acceptable as a starting point, but flag it for follow-up.

### 2. Security — UNVALIDATED RESPONSE DATA ❌

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]   # ← bare dict access, no validation
)
```

Two problems here:

**a) `KeyError` on missing field.** If the external service returns a 200 with a body that does not contain `notification_id`, this raises an unhandled `KeyError`. That exception is not caught by the `requests.RequestException` handler, so FastAPI will return a raw 500 with a stack trace potentially visible to the caller.

**b) No response schema validation.** The response from `notifications.internal` is trusted and unpacked without any validation. If the external service is compromised or misbehaves, it could return unexpected types or values that propagate into your system's response.

**Required fix:** Validate the external response explicitly before accessing fields:

```python
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected response shape: %s", data)
    raise HTTPException(status_code=502, detail="Notification service returned invalid response")
return NotificationResponse(success=True, notification_id=notification_id)
```

Alternatively, define a Pydantic model for the external response and parse through it — this is the more robust approach and gives you type safety on the deserialized data.

### 3. Security — UNVALIDATED MESSAGE FIELD ❌

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str    # ← unbounded, unvalidated string
```

The `message` field accepts an arbitrary string of arbitrary length and passes it directly to an external service. This creates two risks:

- **No length bound:** A caller can send a megabyte-long message string. Add `Field(max_length=...)` appropriate to the notification service's constraints.
- **No content validation:** Depending on how `notifications.internal` renders messages (email, SMS, push), unvalidated content could be a vector for injection. At minimum, document the trust boundary. If this is a public-facing endpoint, enforce stricter constraints.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., max_length=128)
    order_id: str = Field(..., max_length=128)
    message: str = Field(..., max_length=1000)
```

### 4. Error Handling — EXCEPTION CATCH IS TOO BROAD IN TESTS ❌

In the test file:

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The production code catches `requests.RequestException`, not bare `Exception`. This test is patching with the wrong exception type — it happens to pass because `Exception` propagates uncaught through the `requests.RequestException` handler, hits FastAPI's default handler, and returns 500, not 503. But the test asserts 503, which means... this test should be failing. Unless `TestClient` is swallowing it differently.

**Verify this test actually passes as written.** If it does, it is passing for the wrong reason. The correct test is:

```python
import requests as requests_lib  # or httpx after the fix

with patch("api.notifications.requests.post",
           side_effect=requests_lib.RequestException("Connection refused")):
```

After the `httpx` migration, use `httpx.RequestError`.

### 5. Test Coverage — MISSING CASES ⚠️

The test suite covers the happy path and a vague error path. Missing:

| Scenario | Why it matters |
|---|---|
| External service returns 4xx (e.g. 400, 422) | `raise_for_status()` will raise; should map to 503 or a distinct code |
| External service returns 200 with missing `notification_id` | Currently causes unhandled `KeyError` (see §2) |
| Invalid request body (missing fields, wrong types) | Validates Pydantic rejection behavior; should return 422 |
| `notification_id` present but empty string | Edge case in response validation |

---

### 6. Performance — HARDCODED TIMEOUT ⚠️

```python
timeout=5,
```

A 5-second timeout on a checkout-path endpoint is very long. Under load, if the notification service degrades, you will accumulate requests waiting up to 5 seconds each. This is a latency and resource exhaustion risk on the critical checkout path.

Consider:
- Making the timeout configurable via settings (not hardcoded)
- Using a shorter timeout appropriate to checkout SLA requirements
- Evaluating whether notification delivery should be decoupled from the checkout response entirely (fire-and-forget via a queue), so a slow notification service cannot delay order confirmation

---

### 7. Naming / Documentation — MINOR ℹ️

The docstring is present but minimal. Given this is a checkout-critical endpoint, document:
- What triggers a 503 vs. other error codes
- The expected latency contract (i.e., that this is a synchronous call on the checkout path)
- Any retry behavior (currently: none)
