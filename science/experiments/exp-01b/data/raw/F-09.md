## Review

**Decision: Request Changes**

**Summary:** The endpoint is declared `async` but calls `requests.post()` — a synchronous, blocking I/O call — directly on the event loop. Under the checkout load described, every in-flight request across the entire application will be frozen for up to 5 seconds per notification call; this is not a performance concern, it is a correctness defect that will cause cascading failures during any high-volume event. The fix is straightforward: replace `requests` with `httpx` and use `await client.post(...)` via `httpx.AsyncClient`, or offload the synchronous call to a thread pool via `asyncio.to_thread()` as a stopgap.

---

### Detailed Findings

#### 🔴 Critical — Blocking I/O on the Async Event Loop

```python
# Current — blocks the event loop for up to 5 seconds
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)   # synchronous, holds the event loop
```

```python
# Required fix — non-blocking
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={
                "user_id": request.user_id,
                "order_id": request.order_id,
                "message": request.message,
            },
            timeout=5.0,
        )
        response.raise_for_status()
        data = response.json()
        return NotificationResponse(
            success=True,
            notification_id=data["notification_id"],
        )
```

If `httpx` cannot be introduced immediately, the minimum acceptable stopgap is:

```python
import asyncio

response = await asyncio.to_thread(
    requests.post,
    "https://notifications.internal/send",
    json={...},
    timeout=5,
)
```

This at least releases the event loop, though `httpx` is the correct long-term solution.

---

#### 🔴 Critical — Missing Input Validation / Injection Risk

`user_id`, `order_id`, and `message` are accepted as arbitrary strings and forwarded verbatim to the downstream service. There are no length limits, no format constraints, and no sanitization. A caller can send a `message` of arbitrary length or content.

```python
# Add constraints to the Pydantic model
from pydantic import BaseModel, Field
import re

class NotificationRequest(BaseModel):
    user_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    order_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    message: str = Field(..., min_length=1, max_length=500)
```

Without this, the endpoint is a vector for downstream injection and resource exhaustion.

---

#### 🟡 Moderate — Unhandled `KeyError` on `data["notification_id"]`

If the external service returns a 2xx response with a malformed or missing `notification_id` field, this raises an unhandled `KeyError` that will surface as a 500 with a raw traceback rather than a controlled error response.

```python
# Fragile
notification_id=data["notification_id"]

# Defensive
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Unexpected response from notification service")
```

---

#### 🟡 Moderate — Test Suite Does Not Catch the Core Defect

The tests mock `requests.post` and pass, which means they provide false confidence. They do not verify async correctness and will not catch the blocking behavior. Once the code is migrated to `httpx`, the tests must be updated accordingly:

```python
# After migrating to httpx
with patch("httpx.AsyncClient.post", new_callable=AsyncMock) as mock_post:
    mock_post.return_value = mock_response
    ...
```

The error-path test also catches `Exception` broadly rather than `requests.RequestException`, which means it would pass even if the exception handling were removed entirely — it is not testing the right contract.

---

#### 🟢 Minor — No Timeout on the `httpx.AsyncClient`

After migration, ensure the timeout is applied at the client or request level explicitly. A missing timeout on an async HTTP client in a checkout path is the same category of risk as the blocking call itself.

```python
async with httpx.AsyncClient(timeout=5.0) as client:
    ...
```

---

### Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | 🔴 Critical | Replace `requests.post` with `await httpx.AsyncClient.post` |
| 2 | 🔴 Critical | Add field-level validation (`pattern`, `max_length`) to `NotificationRequest` |
| 3 | 🟡 Moderate | Guard against missing `notification_id` in response payload |
| 4 | 🟡 Moderate | Update tests to correctly mock the async HTTP client |

Items 1 and 2 are blockers. This must not merge in its current state.
