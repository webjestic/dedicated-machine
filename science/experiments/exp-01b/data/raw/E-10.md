**Decision: Request Changes**

**Summary:** This PR introduces a critical event loop blocking issue: `requests.post()` is a synchronous I/O call inside an `async def` route. Under checkout load, every call to this endpoint will block the entire event loop for up to the full 5-second timeout duration, starving every concurrent request across the application simultaneously. Replace `requests` with `httpx.AsyncClient` and `await` the call, or if synchronous execution is genuinely required, offload via `asyncio.run_in_executor` — but the former is the correct fix here. The tests pass precisely because `TestClient` runs synchronously and will never surface this class of failure; the bug is invisible until it hits production load.

---

**Required changes before approval:**

**1. Blocking I/O on the event loop — must fix**

```python
# Current — blocks the event loop on every call
import requests
response = requests.post(...)

# Required — non-blocking, awaitable
import httpx
async with httpx.AsyncClient() as client:
    response = await client.post(
        "https://notifications.internal/send",
        json={...},
        timeout=5.0,
    )
```

Update the exception handler accordingly — `requests.RequestException` becomes `httpx.HTTPError` (or catch `httpx.RequestError` and `httpx.HTTPStatusError` separately for finer-grained handling).

---

**2. Test suite masks the production failure mode**

`TestClient` wraps the async app in a synchronous WSGI adapter. It will never detect event loop starvation. Add an async test using `httpx.AsyncClient` with `ASGITransport` to validate the route under its actual runtime model:

```python
import pytest
import httpx
from fastapi import FastAPI
from api.notifications import router

app = FastAPI()
app.include_router(router)

@pytest.mark.anyio
async def test_send_notification_success_async():
    transport = httpx.ASGITransport(app=app)
    async with httpx.AsyncClient(transport=transport, base_url="http://test") as ac:
        # patch httpx inside the module, not requests
        ...
```

Also note: `test_send_notification_service_unavailable` patches with a bare `Exception`, which bypasses the `requests.RequestException` catch block in the current code — it only returns 503 because `Exception` is the parent class. This should be tightened to the specific exception type being used (`httpx.RequestError` after the fix) to ensure the error handling contract is actually tested.

---

**3. Minor — input validation gap**

`message` accepts an unbounded string. For a field that gets forwarded to an external service, consider adding a `max_length` constraint:

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str = Field(..., max_length=1000)
```

This is not a blocker but is worth addressing before this endpoint is exposed to production traffic.

---

The structure of this PR is otherwise solid — Pydantic validation, error handling, logging, and docstrings are all present. The blocking I/O issue is the single change required for approval.
