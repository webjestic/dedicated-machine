**Decision:** Request Changes

**Summary:** The core defect here is a synchronous `requests.post()` call inside an `async def` route — this will block the event loop for the full duration of every external HTTP call (up to 5 seconds per the configured timeout), starving every other concurrent request in the application. This is not a style issue; under checkout load this will manifest as cascading latency across the entire service. Beyond the blocking I/O problem, there are secondary issues with input validation, error handling robustness, and test coverage that also need to be addressed before this merges.

---

## Detailed Review

### 1. Correctness — ❌ Critical: Blocking I/O Inside Async Route

```python
# CURRENT — blocks the event loop
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous — freezes the event loop
```

`requests` is a synchronous library. When this line executes, the OS thread running the event loop is parked waiting on network I/O. No other coroutine can run during that wait. With a 5-second timeout and any meaningful checkout traffic, this becomes a denial-of-service against your own application.

The fix is to replace `requests` with `httpx` (or `aiohttp`) and `await` the call:

```python
import httpx
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, Field, field_validator
import logging

logger = logging.getLogger(__name__)
router = APIRouter()

NOTIFICATION_SERVICE_URL = "https://notifications.internal/send"
NOTIFICATION_TIMEOUT_SECONDS = 5.0


class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str


class NotificationResponse(BaseModel):
    success: bool
    notification_id: str


@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    """
    Send an order confirmation notification to the user via the internal
    notification service. Raises 503 if the service is unreachable or returns
    a non-2xx response.
    """
    async with httpx.AsyncClient(timeout=NOTIFICATION_TIMEOUT_SECONDS) as client:
        try:
            response = await client.post(
                NOTIFICATION_SERVICE_URL,
                json={
                    "user_id": request.user_id,
                    "order_id": request.order_id,
                    "message": request.message,
                },
            )
            response.raise_for_status()
        except httpx.TimeoutException as e:
            logger.error("Notification service timed out: %s", e)
            raise HTTPException(status_code=503, detail="Notification service timed out")
        except httpx.HTTPStatusError as e:
            logger.error(
                "Notification service returned %s: %s",
                e.response.status_code,
                e,
            )
            raise HTTPException(status_code=503, detail="Notification service error")
        except httpx.RequestError as e:
            logger.error("Notification service unreachable: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")

    data = response.json()

    # Guard against malformed responses from the upstream service
    if "notification_id" not in data:
        logger.error("Notification service response missing notification_id: %s", data)
        raise HTTPException(status_code=502, detail="Invalid response from notification service")

    return NotificationResponse(success=True, notification_id=data["notification_id"])
```

If for some reason `httpx` cannot be added as a dependency right now, the fallback is `asyncio.get_event_loop().run_in_executor(None, ...)` to push the blocking call to a thread pool — but that is a workaround, not a solution. Prefer `httpx`.

### 2. Input Validation — ⚠️ Incomplete

Pydantic validates types but the current schema accepts empty strings and arbitrarily long messages, both of which will be forwarded to the upstream service without sanitization.

```python
# CURRENT — accepts "", "   ", or a 10MB string
message: str
```

Add constraints:

```python
from pydantic import BaseModel, Field, field_validator

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128)
    order_id: str = Field(..., min_length=1, max_length=128)
    message: str = Field(..., min_length=1, max_length=1000)

    @field_validator("user_id", "order_id", "message")
    @classmethod
    def must_not_be_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("Field must not be blank")
        return v
```

### 3. Error Handling — ⚠️ Two Issues

**Issue A — Bare `KeyError` on upstream response:**

```python
# CURRENT — unhandled KeyError if upstream omits notification_id
data = response.json()
return NotificationResponse(success=True, notification_id=data["notification_id"])
```

If the notification service returns `{}` or a malformed payload, this raises an unhandled `KeyError` that becomes a 500. The corrected version above adds an explicit guard.

**Issue B — Exception catch is too broad in the test:**

```python
# In the test — catches generic Exception, not requests.RequestException
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`. The test uses a plain `Exception`, which means the test would pass even if the except clause were removed entirely — it is testing the wrong code path. After switching to `httpx`, the test should raise `httpx.RequestError` specifically.

### 4. Test Coverage — ⚠️ Gaps

| Scenario | Covered |
|---|---|
| Successful notification | ✅ |
| Service unreachable | ✅ (but wrong exception type — see above) |
| HTTP 4xx/5xx from upstream | ❌ |
| Timeout | ❌ |
| Missing `notification_id` in response | ❌ |
| Empty/blank input fields | ❌ |
| `message` exceeding max length | ❌ |

At minimum, add tests for timeout, upstream HTTP errors, and malformed upstream responses. Example:

```python
import pytest
import httpx
from unittest.mock import AsyncMock, patch

@pytest.mark.anyio
async def test_send_notification_timeout():
    with patch("httpx.AsyncClient.post", side_effect=httpx.TimeoutException("timed out")):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })
    assert response.status_code == 503
    assert "timed out" in response.json()["detail"]


def test_send_notification_missing_notification_id():
    mock_response = MagicMock()
    mock_response.json.return_value = {}  # upstream omits notification_id
    mock_response.raise_for_status.return_value = None

    with patch("httpx.AsyncClient.post", return_value=mock_response):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed.",
        })
    assert response.status_code == 502


def test_send_notification_rejects_blank_message():
    response = client.post("/notify", json={
        "user_id": "user_123",
        "order_id": "order_456",
        "message": "   ",
    })
    assert response.status_code == 422
```

### 5. Performance — ⚠️ Client Instantiation Per Request

The corrected example above uses `async with httpx.AsyncClient()` inside the handler, which creates and tears down a client on every request. This works correctly but is inefficient under load — connection pooling is discarded between requests.

The preferred pattern is a shared client with application lifespan management:

```python
# main.py or app startup
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient(timeout=5.0)
    yield
    await app.state.http_client.aclose()

app = FastAPI(lifespan=lifespan)
```

```python
# In the route — inject via dependency
from fastapi import Request

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    http_request: Request,
) -> NotificationResponse:
    client: httpx.AsyncClient = http_request.app.state.http_client
    ...
```

This is a recommended follow-up, not a blocker, but worth doing before this endpoint sees production traffic.

### 6. Security — ⚠️ Hardcoded Internal URL

```python
# CURRENT
"https://notifications.internal/send"
```

Internal service URLs should come from environment configuration, not be hardcoded in source. This makes the service non-portable across environments (dev/staging/prod) and leaks topology information in the repository.

```python
import os

NOTIFICATION_SERVICE_URL = os.environ.get(
    "NOTIFICATION_SERVICE_URL",
    "https://notifications.internal/send",  # safe default only if acceptable
)
```

Prefer a settings object (e.g., `pydantic-settings`) if the project already uses one.
