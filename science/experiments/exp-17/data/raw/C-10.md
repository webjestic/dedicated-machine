Decision: Request Changes

Summary: The core distributed-locking strategy contains a timing flaw that directly undermines the at-most-once payment guarantee the PR claims to provide: a payment gateway timeout during retry creates a window where a charge has been issued but the database has no record of it, and the lock TTL can expire during that window, allowing a second worker to pass the idempotency check and charge again. Beyond that critical flaw, the `payment_token` field is logged and returned in structured results in ways that risk credential exposure, and the test suite mocks at the wrong layer, meaning the most dangerous concurrent failure paths are never actually exercised.

---

## Detailed Findings

### 1. CRITICAL — Lock TTL vs. Payment Retry Window Creates Double-Charge Race

This is the most serious flaw and it breaks the central guarantee of the PR.

**The math:**

```
ORDER_LOCK_TTL  = 60 seconds
MAX_RETRIES     = 3
REQUEST_TIMEOUT = 30 seconds per attempt
RETRY_BACKOFF   = 5 seconds between retries

Worst-case payment duration = (3 × 30s timeout) + (2 × 5s backoff) = 100 seconds
```

The lock TTL is 60 seconds. The payment client can hold the lock for up to 100 seconds. The lock **expires while the payment call is still in flight.**

**The failure sequence:**

```
T=0s   Worker A acquires lock, passes idempotency check
T=0s   Worker A calls payment.charge() — gateway times out at T=30s
T=30s  Worker A retries — second timeout fires at T=60s
T=60s  Redis TTL expires — lock is gone
T=61s  Worker B acquires lock, passes idempotency check (no DB record yet)
T=61s  Worker B calls payment.charge() — succeeds, records charge
T=65s  Worker A's third attempt succeeds — charges again
```

The customer is charged twice. The idempotency check is not a defense here because it reads from the database, and the database record only exists after a successful charge completes. The lock is the only thing preventing concurrent entry, and it has already expired.

**Required fix:** The lock TTL must exceed the maximum possible payment client duration with margin, or the lock must be actively renewed (a "lock heartbeat") during the payment call. The simplest correct fix is:

```python
ORDER_LOCK_TTL = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
    + 30  # safety margin
)  # = 130 seconds minimum
```

Better still, derive this constant from `PaymentClient` directly so they cannot drift independently, and add an assertion at startup that enforces the relationship.

### 2. CRITICAL — Timeout Ambiguity Violates At-Most-Once Semantics

The payment client retries on `requests.exceptions.Timeout`, but a timeout does **not** mean the charge failed. It means the response was not received within the window. The charge may have been processed by the gateway and simply not acknowledged.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retrying here may charge again
```

Retrying unconditionally on timeout is correct for idempotent operations. Charging a payment method is not idempotent unless the gateway explicitly supports idempotency keys. The fix is to include a caller-supplied idempotency key in every charge request:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # must be passed in, not generated internally
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

The `order_id` is the natural idempotency key here and should be threaded through from `OrderProcessor.process()` to `PaymentClient.charge()`. Without this, every timeout retry is a potential duplicate charge independent of the locking logic.

### 3. HIGH — `payment_token` Exposure Risk

`payment_token` is accepted as a Pydantic field and flows through the system without any scrubbing. Two specific risks:

**a. Logging:** The `ValidationError` path logs `e` directly:

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value appears in the log.

**b. `OrderResult` does not include the token**, which is correct, but `OrderRequest` holds it as a plain attribute. If `OrderRequest` is ever serialized for debugging, tracing, or error reporting, the token leaks. Mark it explicitly:

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

And sanitize the validation error log:

```python
except ValidationError as e:
    logger.warning("Invalid order request order_id=%s: %s", order_id, e)
    # Do not log the full error object if it may contain token values
```

### 4. HIGH — Exception Handling Gap: Unhandled Failures Inside the Lock

The `process()` method catches `LockError` but nothing else outside the lock context manager. If `payment.charge()` raises `PaymentError`, or if `self.db.transaction()` raises, the exception propagates to the caller **after the lock has been released**.

The lock release is correct (the `finally` block handles it), but the caller receives an unhandled exception with no `OrderResult`. The PR's own docstring says the method returns `OrderResult` in all cases, but that contract is broken for payment and database failures.

```python
# This path is unhandled:
except PaymentError:
    # no OrderResult returned — caller gets an exception
```

Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 5. MEDIUM — Lock Retry Backoff Is Too Aggressive for Its Purpose

```python
LOCK_RETRY_ATTEMPTS = 3
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s — total wait: 2s
```

The lock TTL is 60 seconds. If a legitimate worker holds the lock for 60 seconds, three attempts spaced 2 seconds apart will all fail. The retry logic provides almost no benefit against a legitimately held lock and only helps in the degenerate case where the lock was released between attempts. Either increase the retry window to be meaningful relative to the TTL, or document clearly that this is a fast-fail with no expectation of waiting out a held lock.

---

### 6. MEDIUM — Test Suite Does Not Cover the Critical Failure Paths

The tests mock `redis_client.set` and the script mock, but they never test:

- Lock expiry during payment (the double-charge race described above)
- `PaymentError` propagation (unhandled exception path)
- Timeout on payment with retry (idempotency key issue)
- Concurrent worker simulation (two `OrderProcessor` instances, same `order_id`)

The `test_lock_release_uses_lua_script` test asserts the Lua script is called, but does not verify the token passed to it matches the token stored in Redis. This is the entire point of the token-verified release and it is not tested.

The tests pass because they mock away every interesting behavior. Add integration-level tests using `fakeredis` to exercise the actual lock acquisition and release logic against a real Redis protocol implementation.

---

### 7. LOW — `charge_id` Missing from `OrderResult` on Already-Processed Path

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id is None — caller cannot retrieve the existing charge
    )
```

The caller has no way to retrieve the existing `charge_id` for reconciliation. Return it:

```python
return OrderResult(
    processed=False,
    order_id=req.order_id,
    charge_id=existing.get("charge_id"),
    message="Order already processed",
)
```

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | CRITICAL | Lock TTL (60s) shorter than max payment duration (100s) — double-charge race |
| 2 | CRITICAL | Timeout retries without idempotency key — silent duplicate charges |
| 3 | HIGH | `payment_token` can appear in logs via Pydantic validation errors |
| 4 | HIGH | `PaymentError` and DB exceptions escape `process()` unhandled |
| 5 | MEDIUM | Lock retry window is not meaningful relative to TTL |
| 6 | MEDIUM | Test suite does not cover concurrent failure paths or token verification |
| 7 | LOW | Existing `charge_id` not returned on already-processed path |

Items 1 and 2 must be resolved before this can be approved. They are not edge cases — they are reachable under normal load with a slow or flaky payment gateway, which is exactly the condition this code is supposed to defend against.
