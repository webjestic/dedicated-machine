## Lock Lifecycle Trace

Before rendering a verdict, I'm walking the full execution path.

---

### Acquisition

`SET order:lock:{order_id} {token} NX EX 60` — atomic, correct. Token is a UUID4, unguessable per worker. No issues here.

---

### The TTL Boundary — This Is Where It Breaks

**Critical Flaw: The lock TTL does not cover the critical section.**

Trace the timeline:

```
t=0      Worker A acquires lock (TTL=60s)
t=1      db.get_charge() → None
t=2      db.transaction().__enter__()
t=2      payment.charge() called  ← network I/O begins
         [PaymentClient: up to 3 attempts × 30s timeout + 5s backoff each]
         Worst case: 3 × 30s + 2 × 5s = 100 seconds of blocking I/O
t=62     ORDER LOCK EXPIRES IN REDIS
t=63     Worker B acquires the same lock (TTL resets)
t=63     Worker B: db.get_charge() → None  (charge not yet recorded)
t=63     Worker B: payment.charge() called ← DOUBLE CHARGE
t=100    Worker A: payment.charge() returns charge_id
t=100    Worker A: db.record_charge(), mark_order_complete()
t=100+ε  Worker B: payment.charge() returns charge_id_2
t=100+ε  Worker B: db.record_charge(), mark_order_complete()
```

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient.REQUEST_TIMEOUT = 30` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5`. Worst-case payment duration is **100 seconds**. The lock expires before the payment completes. The idempotency check (`db.get_charge`) runs under the lock but the charge record is not written until after the payment returns — so a second worker that acquires the lock after expiry sees no existing charge and fires a second payment. The entire double-charge prevention guarantee collapses under the exact failure mode this system was designed to prevent.

There is no lock heartbeat or renewal mechanism anywhere in this codebase.

### Additional Issues

**1. Payment inside `db.transaction()` — non-transactional side effect**

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`payment.charge()` is an external HTTP call to a payment gateway. It is not part of the database transaction. If `record_charge` raises after `payment.charge()` succeeds, the DB transaction rolls back, but the charge has already been executed against the payment gateway. The money is gone and there is no record. The comment in the PR — *"crash between charge and record cannot produce an unrecorded charge"* — is factually incorrect. This is a well-known distributed systems hazard: you cannot wrap a non-transactional resource inside a DB transaction and claim atomicity.

The correct pattern is to record a `PENDING` charge row *before* calling the payment gateway (within the transaction), then update it to `COMPLETED` after. This way a crash always leaves a recoverable record.

**2. Timeout on payment call is not bounded at the lock layer**

Even ignoring worst-case retries, a single `REQUEST_TIMEOUT = 30` second call is half the lock TTL. There is no assertion or enforcement that `PaymentClient` timeouts fit within `ORDER_LOCK_TTL`. These two constants live in separate files with no documented relationship. A future developer increasing `MAX_RETRIES` or `REQUEST_TIMEOUT` will not know they are invalidating the locking guarantee.

**3. `db.get_charge()` idempotency check is not inside the transaction**

```python
existing = self.db.get_charge(req.order_id)   # outside transaction
if existing:
    return ...

with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

The read and the write are not in the same transaction. Under read-committed isolation, two workers that both pass the `get_charge` check before either writes can both proceed to charge. The distributed lock is supposed to prevent this, but as shown above, the lock can expire. Even if the lock held, this pattern is fragile — the idempotency read should be inside the transaction with a `SELECT FOR UPDATE` or equivalent to be safe.

**4. `PaymentError` is not caught in `process()`**

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError on failure
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller receives a raw exception rather than an `OrderResult`. This breaks the stated contract of the method, which says it returns `OrderResult`. The test suite does not cover this path.

**5. Lock release on exception paths**

The `finally` block in `_acquire_lock` correctly releases the lock. However, if `self._release_script(...)` itself raises (Redis connection lost, script evicted), the exception will surface and potentially mask the original exception from the `try` block. The release should be wrapped in its own try/except to log and swallow errors, since a TTL-expiring lock is acceptable but a masked application exception is not.

**6. Test coverage gaps**

- No test for `PaymentError` propagation.
- No test for lock expiry mid-flight (the core failure mode).
- No test for `record_charge` failure after successful `payment.charge()`.
- `test_lock_release_uses_lua_script` verifies the script is called but does not verify the correct token is passed — a token mismatch would silently leave the lock held.

---

### Summary of Risk Severity

| Issue | Severity |
|---|---|
| Lock TTL shorter than max payment duration → double charge | Critical |
| Payment outside DB transaction falsely claimed atomic | Critical |
| `PaymentError` unhandled, breaks return contract | High |
| Idempotency read outside transaction | Medium |
| Lock release can mask exceptions | Medium |
| Missing test coverage for failure paths | Medium |

---

**Decision:** Request Changes

**Summary:** The lock TTL (60 s) is shorter than the worst-case payment client execution time (100 s across retries and backoffs), meaning the exclusive lock can expire while a charge is in-flight, allowing a second worker to pass the idempotency check and issue a duplicate charge — the exact failure mode this system claims to prevent. Additionally, `payment.charge()` is called inside `db.transaction()` but is not part of that transaction; a database failure after a successful charge will roll back the record while leaving the charge executed, producing an unrecorded payment. Both issues must be resolved — at minimum by adding a lock heartbeat/renewal loop and restructuring the charge flow to write a `PENDING` record before calling the gateway — before this can be approved.
