Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several correctness and reliability flaws that would make this unsafe to deploy in a payment-critical environment: a critical race condition between the payment gateway call and the database transaction, a lock TTL that is dangerously short relative to the retry budget, unhandled exceptions that silently drop lock release errors, and a timeout vulnerability in the payment client that can produce unrecorded charges. These must be resolved before approval.

---

## Detailed Findings

### 1. CRITICAL — Charge Occurs Inside the Database Transaction (Double-Spend / Unrecorded Charge Risk)

**File:** `services/order_processor.py`, lines inside `with self.db.transaction():`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side-effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is a non-transactional, external network operation placed inside a database transaction. This is architecturally incorrect and dangerous in two directions:

**Direction A — Charge succeeds, DB transaction rolls back.** If `record_charge` or `mark_order_complete` raises (deadlock, constraint violation, connection drop), the DB transaction rolls back but the charge has already been submitted to the payment gateway. The money is taken; the system has no record of it. The idempotency check will not catch it on retry because `get_charge` returns nothing. The next worker will charge again.

**Direction B — DB transaction holds open during network I/O.** A 30-second payment timeout (×3 retries = up to 90 seconds) holds a database connection and transaction open, exhausting connection pool resources under load.

**Required fix:** Charge outside the transaction. Record the result inside it. Use a two-phase approach with a "pending charge" record to handle the crash-between-charge-and-record window:

```python
# Outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB writes inside the transaction
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

To handle the crash between charge and record, the idempotency check should also query the payment gateway for a charge matching this order's idempotency key, or a "pending" pre-charge record should be written before the gateway call.

### 2. CRITICAL — Lock TTL Shorter Than Maximum Retry Window

**File:** `services/order_processor.py` + `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. If the payment gateway is slow, the lock expires while the first worker is still in `payment.charge()`. A second worker can now acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge. This is the exact double-charge scenario the lock is supposed to prevent.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed the maximum payment client wall time (≥120s with margin), or implement lock heartbeat renewal during the payment call. The simpler fix:

```python
ORDER_LOCK_TTL = 120  # must exceed max payment client wall time
```

And reduce payment client total budget to stay well under that ceiling:
```python
REQUEST_TIMEOUT = 15
MAX_RETRIES = 3
RETRY_BACKOFF = 3
# Max wall time: 3×15 + 2×3 = 51s — safely under 120s TTL
```

### 3. HIGH — Lock Release Errors Are Silently Swallowed

**File:** `services/order_processor.py`, `_acquire_lock` finally block

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection failure, script error), the exception is suppressed by the `finally` block's implicit swallowing — the log line after it won't even execute. More importantly, the caller never learns the lock may not have been released, and no alert fires. The lock will hold until TTL expiry, blocking all workers for up to 60 seconds.

**Required fix:** Catch and log explicitly; do not let release failures propagate and mask the original exception, but do ensure they are observable:

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error(
                "Lock release returned 0 — token mismatch or already expired order_id=%s",
                order_id,
            )
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception(
            "Failed to release order lock order_id=%s — will expire at TTL",
            order_id,
        )
```

### 4. HIGH — `PaymentError` and Unexpected Exceptions Not Caught in `process()`

**File:** `services/order_processor.py`, `process()` method

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        with self.db.transaction():
            charge_id = self.payment.charge(...)
            ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Only `LockError` is caught. `PaymentError`, `requests.exceptions.RequestException`, database exceptions, and any other runtime errors propagate uncaught to the caller. In a worker pool this means an unhandled exception crashes the worker process or surfaces a raw stack trace to the API layer, potentially leaking internal details. The lock will still be released via `finally`, but the caller receives no structured `OrderResult`.

**Required fix:** Add explicit exception handling for known failure modes:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 5. MEDIUM — Timeout Retry Logic Retries on Ambiguous Gateway State

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A `Timeout` on a POST to a payment gateway does not mean the charge was not processed. The request may have reached the gateway and been executed; only the response was lost. Retrying unconditionally on timeout can produce duplicate charges at the gateway level. This is only safe if the payment gateway supports idempotency keys.

**Required fix:** Pass a stable idempotency key (e.g., derived from `order_id`) in the request headers so the gateway deduplicates retries:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge()` method signature should accept and forward this key. Without it, retry-on-timeout is unsafe.

### 6. MEDIUM — `charge_id` Key Access Without Validation

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id`, wrong type, non-JSON content-type), this raises `KeyError` or `JSONDecodeError`, which propagates as an untyped exception rather than a `PaymentError`. This makes error handling at the call site fragile.

**Required fix:**

```python
try:
    body = response.json()
    charge_id = body["charge_id"]
    if not isinstance(charge_id, str) or not charge_id:
        raise PaymentError(f"Invalid charge_id in gateway response: {body!r}")
except (ValueError, KeyError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

### 7. LOW — Tests Do Not Cover the Lock-Expiry-During-Payment Race

**File:** `tests/test_order_processor.py`

The test suite covers happy path, idempotency, contention, and validation. It does not include a test simulating lock TTL expiry during a slow payment call, nor does it test the behavior when `_release_script` returns `0` (token mismatch after expiry). These are the exact failure modes that matter most in production.

**Recommended additions:**
- Test that a second processor can acquire the lock after TTL expiry (verifying the race condition is understood and guarded against once the TTL fix is applied).
- Test that `_release_script` returning `0` logs an error rather than silently succeeding.
- Test `PaymentError` propagation through `process()` once exception handling is added.
