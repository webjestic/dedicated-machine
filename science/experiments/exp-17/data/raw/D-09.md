Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow that undermines the at-most-once guarantee the PR claims to provide, plus several reliability and correctness issues that need to be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs inside a DB transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. A database transaction provides atomicity over database operations only. `self.payment.charge(...)` is an external HTTP call. If the process crashes, the DB connection drops, or the transaction rolls back after `charge()` returns but before `record_charge()` commits, you have a real charge with no record of it. This is a silent double-charge vector on retry.

```python
# Current — payment.charge() is NOT covered by DB atomicity
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect, not transactional
    self.db.record_charge(...)
```

The correct pattern is to move the external call outside and before the transaction, then make the DB write idempotent on `charge_id`:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Log charge_id for manual reconciliation — we cannot safely void without
    # knowing whether the void will succeed, and voiding is itself a side-effect
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but DB record failed "
        "order_id=%s charge_id=%s customer_id=%s amount_cents=%d",
        req.order_id, req.customer_id, charge_id, req.amount_cents,
    )
    raise
```

This does not fully solve the problem — distributed payment systems cannot achieve true atomicity across an external gateway and a local DB without a saga or outbox pattern — but it is strictly safer than the current code and the comment should be removed or corrected.

### 2. Timeout on payment retry is indistinguishable from a successful charge

In `payment_client.py`, a `Timeout` exception is retried. A timeout means the request may have reached the gateway and the charge may have already been processed. Retrying blindly on timeout can produce duplicate charges at the gateway level, regardless of what the Redis lock does.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # BUG: charge may have succeeded — retrying without idempotency key
    # can result in a second charge at the gateway
```

The fix is to generate a per-attempt idempotency key before the retry loop and pass it with every request to the gateway. Most payment APIs (Stripe, etc.) support this. If the gateway does not, timeouts must not be retried without explicit confirmation of failure.

```python
idempotency_key = str(uuid.uuid4())  # generated once, before the loop

response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 3. Lock TTL does not account for retry duration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire before the payment client gives up. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge.

Either increase the TTL to safely exceed the maximum payment duration with margin, or enforce that the payment timeout budget fits within the lock TTL. These two constants need to be defined together with an explicit invariant:

```python
# Invariant: ORDER_LOCK_TTL must exceed maximum possible payment duration
MAX_PAYMENT_DURATION = PaymentClient.MAX_RETRIES * (
    PaymentClient.REQUEST_TIMEOUT + PaymentClient.RETRY_BACKOFF
)
assert ORDER_LOCK_TTL > MAX_PAYMENT_DURATION, (
    f"Lock TTL {ORDER_LOCK_TTL}s must exceed max payment duration {MAX_PAYMENT_DURATION}s"
)
```

### 4. Unhandled exceptions inside the lock context manager silently release the lock

If `self.db.get_charge()`, `self.payment.charge()`, or the DB transaction raises an unexpected exception, the `finally` block releases the lock and the exception propagates up — but the caller receives an unhandled exception, not an `OrderResult`. The `process()` method has no catch-all handler, so this will surface as a 500 to whatever is calling this service. The lock release is correct behavior, but the exception handling contract should be explicit and documented.

### 5. `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value (a payment credential) may appear in logs. Sanitize or suppress field values from validation error logs, or log only field names and error types.

## Minor Issues

### 6. `requests.post` has no connection timeout, only a read timeout

`timeout=self.REQUEST_TIMEOUT` as a scalar sets the read timeout. A hung DNS resolution or TCP handshake will block indefinitely. Use a tuple:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

### 7. Test for lock release does not verify the token passed to the script

`test_lock_release_uses_lua_script` only asserts the script was called once. It does not verify that the token passed as `args` matches what was set via `redis.set`. This is the entire point of the token-verified release — the test should assert on the argument.

### 8. No test for the payment timeout / retry path

There is no test covering the case where `payment.charge` raises a `PaymentError` after exhausting retries. The `process()` method will raise an unhandled exception in that case (see issue 4), and there is no test that would catch this.
