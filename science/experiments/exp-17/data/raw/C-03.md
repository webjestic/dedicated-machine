Decision: Request Changes

Summary: The core distributed-lock guarantee is broken in at least two ways that directly contradict the "at-most-once payment" claim in the docstring. There are also secondary issues in the payment client and test suite that need to be resolved before this is safe to merge into any production path.

---

## Critical Flaws

### 1. The payment call is outside the database transaction — the "atomic" claim is false

This is the most dangerous defect in the PR.

```python
# What the code actually does:
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← network call to external gateway
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — that is incorrect. A database transaction wraps database operations. It does **not** make an external HTTP call atomic with those writes. The `payment.charge()` call goes out over the network to an external payment gateway. If the gateway succeeds and returns a `charge_id`, and then the process crashes before `record_charge` commits, you have charged the customer with no record of it. The database transaction rollback does nothing to reverse a completed charge at the payment gateway.

The correct pattern here is:
- Record a **pending** charge row *before* calling the gateway (within a transaction), so a crash leaves a recoverable pending state.
- Call the gateway.
- Update the row to confirmed within a second transaction.
- The idempotency check must also cover the pending state to prevent re-entry.

Alternatively, use an outbox pattern. What is not acceptable is the current structure, which provides no crash-safety guarantee while the comment claims it does.

---

### 2. Lock TTL vs. total payment retry duration — the lock can expire while the payment is in-flight

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. The payment client can hold the lock for up to 100 seconds. After 60 seconds Redis expires the key. A second worker can now acquire the lock, pass the idempotency check (the charge is not yet recorded because the first worker is still retrying), and issue a second charge. The lock TTL must be strictly greater than the maximum possible time spent inside the lock, with margin. Either raise the TTL to at least 120–150 seconds, or — better — reduce the payment client's maximum total duration to a value that fits inside the TTL with a documented margin and enforce that relationship with an assertion or a shared constant.

---

### 3. Timeout exceptions are silently retried — this is a double-charge vector

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the client did not receive a response. It does **not** mean the charge did not happen. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout will issue a second charge for the same token and amount. This is a textbook double-charge scenario.

The correct behavior is to treat a timeout as an ambiguous outcome and either:
- Query the gateway for the status of the previous attempt using an idempotency key before retrying, or
- Pass a client-generated idempotency key on every request so the gateway deduplicates on its side.

The current code does neither. The `json` payload has no idempotency key field, and there is no status-check step before retry.

## Significant Issues

### 4. `PaymentError` is not caught in `process()` — unhandled exception escapes

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # can raise PaymentError
```

`PaymentError` is not imported into `order_processor.py` and is not caught anywhere in `process()`. If the payment gateway returns a permanent 4xx or exhausts retries, the exception propagates out of `process()` uncaught. The lock is released correctly via `finally`, but the caller receives a raw exception instead of an `OrderResult`. The method signature promises `OrderResult` in all cases. Catch `PaymentError` and return an appropriate `OrderResult(processed=False, ...)`.

---

### 5. `customer_id` has no pattern constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` is correctly constrained to `[a-zA-Z0-9_\-]`. `customer_id` accepts arbitrary Unicode including control characters and SQL metacharacters. Apply the same or equivalent pattern constraint.

---

### 6. `payment_token` is logged indirectly through `charge_id` — verify no token leakage

The `payment_token` field is passed directly to the payment client. Confirm that no logging path in `payment_client.py` or any upstream middleware serializes the request body, which would write a live payment token to log files. The current code does not appear to log the token directly, but this should be explicitly audited and documented, given the sensitivity of the field.

---

## Test Suite Issues

### 7. Tests do not cover the crash-between-charge-and-record scenario

There is no test that verifies behavior when `record_charge` raises after `payment.charge` succeeds. This is the exact failure mode that the comment claims is handled. A test should exist that confirms the charge is recoverable (or at minimum, that the system does not silently lose it).

### 8. Lock expiry during payment is not tested

No test simulates the lock TTL expiring while the payment call is in progress. Given that this is the primary concurrency guarantee of the service, there should be a test that mocks time or Redis TTL behavior to verify the system detects or prevents this condition.

### 9. Timeout retry behavior is not tested

No test verifies what happens when `payment.charge` times out and retries. Given the double-charge risk identified above, this path needs explicit test coverage.
