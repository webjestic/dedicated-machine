Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic input validation, and idempotency checks are all correctly structured. However, there are several correctness and reliability defects that would make this unsafe to deploy in a payment-critical environment: a critical charge-outside-transaction flaw that breaks the atomicity guarantee the PR claims to provide, a missing heartbeat mechanism that allows TTL expiry to silently invalidate the lock mid-operation, unhandled exceptions that can leak lock state, and a timeout misconfiguration in the payment client that makes the retry budget incoherent. Each issue is detailed below.

---

## Critical Issues

### 1. `payment.charge()` Is Called Outside the Database Transaction — Atomicity Guarantee Is False

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

The PR summary states: *"charges the payment method and records the transaction within a single atomic database transaction."* This claim is incorrect and the comment in the code reinforces the false guarantee:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(          # ← external network call to payment gateway
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a third-party payment gateway. Wrapping it in `with self.db.transaction()` does not make it atomic with the database writes. The failure modes this creates:

- **Charge succeeds, process crashes before `record_charge`:** The payment gateway has captured funds. The database has no record. The idempotency check on retry finds nothing and charges again. The customer is double-charged.
- **Charge succeeds, `record_charge` raises an exception:** The transaction rolls back. Same outcome as above.
- **Gateway returns a timeout but the charge was actually processed:** The retry logic in `PaymentClient` will attempt the charge again. Without idempotency keys on the gateway request, this is a duplicate charge.

**Required fix:** The external charge call must happen *before* opening the database transaction. The database transaction should only wrap the idempotency check and the record writes. A two-phase pattern is required:

```python
# 1. Attempt charge (outside DB transaction)
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-level deduplication
)

# 2. Record atomically — if this fails, the lock is still held and
#    the idempotency check on retry will find the charge record
with self.db.transaction():
    # Re-check inside transaction to close the TOCTOU window
    if self.db.get_charge(req.order_id):
        # charge already recorded by a concurrent path; void if necessary
        ...
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Additionally, the payment client's `charge()` method should accept and forward an idempotency key to the gateway so that retries on timeout are safe.

### 2. No Lock Heartbeat — TTL Expiry Silently Invalidates the Lock Mid-Operation

**File:** `services/order_processor.py`, `ORDER_LOCK_TTL = 60`

The lock TTL is 60 seconds. The payment client has `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds per retry and `REQUEST_TIMEOUT = 30` seconds per attempt. Worst-case payment client execution time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100 seconds
```

The payment client alone can exceed the lock TTL by 40 seconds. When the TTL expires, Redis drops the key. A second worker can then acquire the lock, pass the idempotency check (the first worker hasn't recorded yet), and issue a second charge. The first worker then continues executing, unaware its lock has expired, and records its charge. Both charges are captured.

The lock provides no protection once its TTL expires. There is no heartbeat renewal, no expiry detection, and no abort path when the lock is lost.

**Required fix:** Either implement a background heartbeat thread that extends the lock TTL while the critical section is executing, or reduce the payment client's maximum execution time to be safely below the lock TTL with margin. The simpler near-term fix is to reduce `REQUEST_TIMEOUT` and `RETRY_BACKOFF` so that `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF` is well under `ORDER_LOCK_TTL`, and document this constraint explicitly so future changes to either constant don't silently break the invariant. A heartbeat is the more robust long-term solution.

## Significant Issues

### 3. Unhandled Exceptions Inside the Lock Context Leak Meaningful Failure Information and May Mask Lock State

**File:** `services/order_processor.py`, `process()` method

The `with self._acquire_lock(...)` block catches `LockError` but does not catch `PaymentError`, database exceptions, or any other runtime exception. If `self.payment.charge()` raises `PaymentError`, or if `self.db.transaction()` raises, the exception propagates out of the `with` block. The `finally` clause in `_acquire_lock` will run and release the lock correctly — that part is fine. However, the exception then propagates to the caller as an unhandled exception rather than returning a structured `OrderResult`.

For a service that returns `OrderResult` as its contract, this is a broken interface. The caller has no way to distinguish a payment decline from an unhandled crash. It also means the caller cannot safely retry without knowing whether a charge was attempted.

**Required fix:** Add explicit exception handling for `PaymentError` and database exceptions inside the lock context, returning appropriate `OrderResult` values with `processed=False` and meaningful messages. Log the full exception with `logger.exception()` for observability.

### 4. Payment Client Timeout Budget Exceeds Lock TTL — Constants Are Incoherent

**File:** `services/payment_client.py`

As calculated above, the maximum payment client execution time (100s) exceeds `ORDER_LOCK_TTL` (60s). These two constants are defined in separate files with no documented relationship and no enforcement. This is a latent correctness bug waiting to be triggered by a slow gateway or introduced by an unrelated change to either constant.

**Required fix:** Document the constraint `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF < ORDER_LOCK_TTL` explicitly. Either enforce it programmatically (an assertion at startup, or a single computed constant both values derive from) or reduce the payment client budget. With current values, reducing `REQUEST_TIMEOUT` to 15s gives `3×15 + 2×5 = 55s < 60s`, which is marginal. A more conservative budget with a heartbeat is preferred.

## Minor Issues

### 5. Idempotency Check Has a TOCTOU Race Between Check and Charge

**File:** `services/order_processor.py`

```python
existing = self.db.get_charge(req.order_id)   # check
if existing:
    return ...
# ... time passes, lock could expire here ...
charge_id = self.payment.charge(...)           # use
```

The distributed lock is intended to serialize this check-then-act sequence. If the lock TTL expires between the check and the charge (see Issue 2), a second worker can pass the same check concurrently. The idempotency check must be re-evaluated inside the database transaction after the charge, not only before it, to close this window. This is partially addressed by fixing Issue 1 and Issue 2 together.

---

### 6. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Appears in Log Output

**File:** `services/order_processor.py` and `services/payment_client.py`

`payment_token` is passed through to the payment client and included in the JSON body of the HTTP request. Confirm that the `requests` library is not configured with debug-level HTTP logging in any environment, as that would log the full request body including the token. This is a pre-deployment configuration check, not a code defect, but worth flagging explicitly given the sensitivity of the field.

---

### 7. Test Coverage Gaps

**File:** `tests/test_order_processor.py`

The following failure paths are not covered:

- Lock TTL expiry mid-operation (the most dangerous failure mode)
- `PaymentError` raised during charge — verifies the lock is released and an `OrderResult` is returned rather than an exception propagating
- Database transaction rollback after a successful charge — verifies the double-charge scenario is detectable
- Retry exhaustion in `PaymentClient` — verifies `PaymentError` is raised after three attempts
- Gateway timeout followed by success on retry — verifies the retry logic functions correctly

The existing tests are well-structured and the mock patterns are clean, but they only cover the happy path and the most basic contention case.
