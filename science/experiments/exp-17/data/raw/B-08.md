Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic input validation, and idempotency checks are all correctly structured. However, there are several correctness and reliability flaws that would make this unsafe to deploy in a payment-critical environment: a critical lock expiry race that can produce double charges, a timeout misconfiguration that guarantees lock expiry during payment, an unhandled exception path that silently swallows payment errors, and a retry strategy on timeouts that is unsafe without idempotency guarantees from the gateway. These must be resolved before approval.

---

## Detailed Findings

### 1. CRITICAL — Lock TTL Can Expire During Payment, Enabling Double Charge

**File:** `services/order_processor.py` / `services/payment_client.py`

**The problem:**

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

The maximum wall-clock time the payment client can consume before raising is:

```
(30s timeout × 3 attempts) + (5s backoff × 2 gaps) = 100 seconds
```

The lock TTL is 60 seconds. Worker A can hold the lock, begin charging, have the lock expire at t=60 while the payment client is still retrying, and Worker B can then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge. The lock is the sole concurrency barrier, and the payment client is designed to outlive it.

**Required fix:** Either (a) implement lock heartbeat renewal that extends the TTL while the payment is in-flight, or (b) ensure `ORDER_LOCK_TTL` strictly exceeds the maximum possible payment client duration with margin. Option (b) is simpler but fragile; option (a) is the correct distributed systems approach. At minimum, the TTL must be `> (REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES - 1))` plus a safety margin.

### 2. CRITICAL — Timeout Responses Are Retried Without Gateway Idempotency Keys

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to retry
```

A timeout does not mean the charge failed — it means the client did not receive a response within the window. The gateway may have processed the charge successfully. Retrying without an idempotency key will issue a second charge for the same payment token and amount. This is a classic at-most-once violation.

**Required fix:** Generate a per-order idempotency key (e.g., derived from `order_id`) and pass it as a request header (e.g., `Idempotency-Key`) on every attempt. The gateway must support this header; if it does not, timeout retries must be disabled and the timeout treated as a terminal failure requiring manual reconciliation.

### 3. HIGH — `PaymentError` and Unexpected Exceptions Are Not Caught in `process()`

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    # ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # can raise PaymentError
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

`PaymentError` is not imported into `order_processor.py` and is not caught anywhere in `process()`. If the payment gateway raises `PaymentError` after a successful charge (e.g., on response parse failure), the database transaction rolls back, the charge is not recorded, and the next call passes the idempotency check and charges again. The lock is released cleanly, but the system is now in an inconsistent state.

More broadly, any unexpected exception propagates out of `process()` as an unhandled exception rather than a structured `OrderResult`, breaking the contract the method signature implies.

**Required fix:**

```python
from services.payment_client import PaymentError

# inside process(), within the lock:
try:
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, order_id=req.order_id, message="Internal error")
```

Note that a `PaymentError` raised *after* a successful gateway charge but *before* `record_charge` commits is still a dangerous window — this is the broader idempotency key problem from finding #2.

### 4. HIGH — `payment_token` Logged or Exposed in Validation Error Path

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value will appear in log output. Payment tokens must never be logged.

**Required fix:** Sanitize the error before logging, or log only field names and error types without values:

```python
except ValidationError as e:
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

### 5. MEDIUM — Lock Retry Backoff Is Linear and Blocking

**File:** `services/order_processor.py`

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep()` blocks the worker thread entirely. In an async or thread-pool deployment this stalls a worker for up to 1.5 seconds of cumulative sleep per contended order. Additionally, linear backoff without jitter causes retry storms when multiple workers contend on the same order simultaneously — all workers wake at the same interval and collide again.

**Required fix:** Use exponential backoff with jitter:

```python
import random
backoff = min(0.1 * (2 ** attempt) + random.uniform(0, 0.1), 2.0)
```

If the service is async, use `asyncio.sleep()` instead of `time.sleep()`.

### 6. MEDIUM — `charge_id` Extraction Has No Error Handling

**File:** `services/payment_client.py`

```python
response.raise_for_status()
charge_id = response.json()["charge_id"]
```

If the gateway returns HTTP 200 with a malformed body (missing `charge_id`, invalid JSON), this raises `KeyError` or `JSONDecodeError`. Neither is caught or retried. The exception propagates as an unexpected type, bypassing the `PaymentError` handling path and leaving the charge state ambiguous.

**Required fix:**

```python
try:
    data = response.json()
    charge_id = data["charge_id"]
except (ValueError, KeyError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

### 7. LOW — Tests Do Not Cover the Lock Expiry / Double-Charge Scenario

**File:** `tests/test_order_processor.py`

The test suite validates the happy path, idempotency check, lock contention, and input validation, but there is no test for the scenario where the lock expires mid-processing and a second worker acquires it. This is the highest-severity failure mode and should have an explicit regression test once the heartbeat or TTL fix is in place.

---

### 8. LOW — `ORDER_LOCK_TTL` and Payment Client Constants Are Not Centrally Validated

There is no assertion or startup check that `ORDER_LOCK_TTL > (REQUEST_TIMEOUT * MAX_RETRIES) + (RETRY_BACKOFF * (MAX_RETRIES - 1))`. These values live in separate files and can drift independently. A future developer increasing `MAX_RETRIES` in `payment_client.py` will not be warned that they have broken the lock safety invariant.

**Recommended fix:** Add a module-level or `__init__`-time assertion in `OrderProcessor`:

```python
from services.payment_client import PaymentClient

_MAX_PAYMENT_DURATION = (
    PaymentClient.REQUEST_TIMEOUT * PaymentClient.MAX_RETRIES
    + PaymentClient.RETRY_BACKOFF * (PaymentClient.MAX_RETRIES - 1)
)
assert OrderProcessor.ORDER_LOCK_TTL > _MAX_PAYMENT_DURATION, (
    f"ORDER_LOCK_TTL ({OrderProcessor.ORDER_LOCK_TTL}s) must exceed "
    f"max payment duration ({_MAX_PAYMENT_DURATION}s)"
)
```
