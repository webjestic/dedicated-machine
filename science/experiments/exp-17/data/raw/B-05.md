Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several correctness and reliability flaws that would make this unsafe to deploy in a payment-critical environment: a critical race window between payment execution and database recording, a missing heartbeat mechanism that allows TTL expiry under slow gateways, and a timeout misconfiguration that can produce silent double charges. These must be resolved before approval.

---

## Critical Issues

### 1. Payment Executes Outside the Database Transaction — Charge Can Be Lost

This is the most serious flaw. The comment in the code claims "crash between charge and record cannot produce an unrecorded charge," but that claim is false as written.

```python
# services/order_processor.py, lines ~85–92
with self.db.transaction():
    charge_id = self.payment.charge(          # ← network call to external gateway
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is a network call to an external payment gateway. Placing it inside `self.db.transaction()` does **not** make it atomic with the database writes. A database transaction provides atomicity only over database operations. The payment gateway has no knowledge of your database transaction, and your database has no knowledge of the gateway's state.

**Failure scenario:** The gateway charges the card successfully and returns `charge_id`. Before `record_charge` executes, the process crashes (OOM kill, pod eviction, network partition). The database transaction rolls back. The charge is now real money taken from the customer with no record in your system. The idempotency check on retry will find no existing charge and attempt to charge again.

**Required fix:** Adopt an outbox or two-phase commit pattern. The standard approach for this architecture:

1. Within a single DB transaction: persist a `pending_charge` record with the order details and a generated idempotency key.
2. Outside the transaction: call the payment gateway, passing your idempotency key so the gateway deduplicates retries.
3. Within a second DB transaction: update the record from `pending` to `complete` with the returned `charge_id`.

On crash-and-retry, step 1 finds the existing `pending_charge`, skips re-insertion, re-attempts the gateway call (idempotent via the key), and completes step 3. This is the only way to achieve at-most-once charging with an external system.

### 2. Lock TTL Can Expire While the Worker Still Holds It

```python
ORDER_LOCK_TTL = 60   # seconds
```

```python
# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Maximum payment client wall time: `3 × 30s + 2 × 5s = 100 seconds`.

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. If the gateway is slow, the lock expires while the original worker is still executing. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and begins charging. Both workers are now charging concurrently. This is precisely the double-charge scenario the lock is supposed to prevent.

**Required fix:** Implement a lock heartbeat. A background thread (or async task) must periodically call `EXPIRE order:lock:{order_id} 60` while the worker holds the lock, stopping when the lock is released. Alternatively, reduce `REQUEST_TIMEOUT × MAX_RETRIES + total_backoff` to a value that is provably less than `ORDER_LOCK_TTL` with margin, and document that constraint explicitly with an assertion or CI check that enforces it. The former is more robust.

### 3. Timeout Misconfiguration Masks Gateway Ambiguity — Silent Double Charge Risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry loop
```

When a `requests.Timeout` fires, it means the client gave up waiting. It does **not** mean the gateway did not process the charge. The gateway may have received the request, charged the card, and its response was lost in transit. Retrying without an idempotency key passed to the gateway will attempt a second charge on the same card.

**Required fix:** Generate a stable idempotency key per order (e.g., derived from `order_id`, not a random UUID per attempt) and pass it in the request headers to the payment gateway on every attempt. Most payment APIs (`Stripe`, `Braintree`, etc.) honor `Idempotency-Key` headers and will return the original result for duplicate requests. This makes retries safe regardless of timeout ambiguity.

## Moderate Issues

### 4. Lock Release Silently Swallows Errors

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script returns `0` (token mismatch — meaning the lock expired and was acquired by another worker), the code logs a success message anyway. The caller has no signal that the lock was lost mid-execution, which is exactly the condition that may have allowed a concurrent worker to proceed. This should at minimum log a `CRITICAL` or `ERROR` when the script returns `0`, and ideally propagate a signal so the caller can determine whether its work should be trusted.

```python
finally:
    released = self._release_script(keys=[lock_key], args=[token])
    if not released:
        logger.critical(
            "Lock expiry detected during execution — possible concurrent processing order_id=%s",
            order_id,
        )
    else:
        logger.info("Order lock released order_id=%s", order_id)
```

### 5. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Payment tokens are sensitive credentials. Validate the log output of `ValidationError` for this model, or explicitly redact sensitive fields before logging.

---

### 6. `db.transaction()` Contract Is Implicit and Untested Under Failure

The code assumes `self.db.transaction()` is a context manager that rolls back on exception. This contract is never verified in tests — the mock's `__exit__` always returns `False` (no suppression) but never simulates a rollback. Add a test that raises inside the transaction block and asserts that `record_charge` and `mark_order_complete` effects are not committed. Without this, a DB layer that silently auto-commits on exception would go undetected.

---

### 7. `payment_token` Field Validation Is Insufficient

```python
payment_token: str = Field(..., min_length=1, max_length=256)
```

`payment_token` has no pattern constraint. A malformed or structurally invalid token will pass validation and be forwarded to the payment gateway, where it will fail with a 4xx and raise `PaymentError`. This is acceptable behavior, but consider whether a structural pattern (e.g., `tok_[a-zA-Z0-9]+`) can be enforced at the boundary to fail faster and avoid unnecessary gateway round-trips.

## Test Coverage Gaps

| Missing Test | Risk |
|---|---|
| Lock TTL expiry mid-execution (script returns `0`) | Silent concurrent processing |
| `PaymentError` raised inside `db.transaction()` — verify rollback | Unrecorded charge |
| Gateway timeout followed by successful retry | Verify single charge recorded |
| `db.record_charge` raises after `payment.charge` succeeds | Charge taken, not recorded |
| Lock retry exhaustion with partial backoff timing | Verify retry count and sleep calls |

The existing tests cover the happy path and basic contention well, but none of the failure-mode tests that matter most for a payment system are present.

## Summary of Required Changes Before Approval

1. **[Blocking]** Restructure to use an outbox/idempotency-key pattern so a crash between charge and record does not produce an unrecorded charge.
2. **[Blocking]** Implement lock heartbeat renewal, or enforce and document that total payment client timeout is strictly less than `ORDER_LOCK_TTL`.
3. **[Blocking]** Pass a stable per-order idempotency key to the payment gateway on all retry attempts to prevent double charges on timeout.
4. **[Required]** Log `CRITICAL` when lock release script returns `0`.
5. **[Required]** Audit `ValidationError` log output for payment token leakage.
6. **[Required]** Add failure-path tests covering the scenarios in the table above.
