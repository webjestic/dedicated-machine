## Lock Lifecycle Trace

Before I render a verdict, I'm walking every execution path end to end.

---

### Path 1: Happy path

```
acquire_lock → SET NX EX 60 ✓
  → db.get_charge() → None ✓
  → db.transaction().__enter__() ✓
    → payment.charge() → charge_id ✓
    → db.record_charge() ✓
    → db.mark_order_complete() ✓
  → db.transaction().__exit__() ✓
  → yield returns, finally block fires
  → _release_script(keys=[lock_key], args=[token]) ✓
```

Clean. Moving on.

---

### Path 2: `payment.charge()` raises an unhandled exception

```
acquire_lock → SET NX EX 60 ✓
  → db.get_charge() → None ✓
  → db.transaction().__enter__() ✓
    → payment.charge() → raises PaymentError / Exception
  → db.transaction().__exit__() fires → rolls back DB ✓
  → exception propagates out of `with self._acquire_lock(...)` block
  → _acquire_lock finally block fires → lock released ✓
  → exception propagates out of process() entirely — UNCAUGHT
```

**Finding 1 — Unhandled `PaymentError` propagates to caller.** `PaymentError` and generic `Exception` from the payment client are not caught in `process()`. The caller receives a raw exception rather than an `OrderResult`. Given this is a service method returning a structured result type, that contract is broken. The payment client can raise on 4xx, on timeout exhaustion, or on a malformed gateway response — all of these escape silently. This needs a `except (PaymentError, Exception)` guard that returns `OrderResult(processed=False, ...)`.

---

### Path 3: TTL expiry during payment round-trip

```
acquire_lock → SET NX EX 60
  → payment.charge() — gateway is slow, takes >60s
  → TTL expires → Redis deletes the key
  → Worker B acquires the same lock on the same order_id
  → Worker B passes idempotency check (charge not yet recorded)
  → Worker B calls payment.charge() → second charge issued
  → Worker A's payment.charge() returns
  → Worker A calls db.record_charge()
  → Worker B calls db.record_charge()
  → Double charge
```

**Finding 2 — No lock heartbeat / renewal mechanism.** `ORDER_LOCK_TTL = 60` with `REQUEST_TIMEOUT = 30` and `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` gives a worst-case payment client execution time of `3 × 30 + 2 × 5 = 100 seconds` — well beyond the 60-second TTL. The lock can expire while the payment is in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet written), and issue a duplicate charge. This is the most serious flaw in the design. The PR summary claims "prevents duplicate charges under concurrent worker deployments" — this claim is false under slow gateway conditions.

Mitigation options: (a) a background heartbeat thread that calls `EXPIRE` on the lock key while the payment is in flight, (b) reduce `MAX_RETRIES × REQUEST_TIMEOUT` to be strictly less than `ORDER_LOCK_TTL` with documented margin, or (c) use a fencing token / database-level unique constraint on `order_id` as the true idempotency barrier so that even if the lock expires, the DB transaction will reject the second write.

---

### Path 4: `db.transaction().__exit__()` raises (DB commit failure)

```
  → payment.charge() → charge_id returned (money moved)
  → db.record_charge() ✓
  → db.mark_order_complete() ✓
  → db.transaction().__exit__() raises (connection lost, deadlock, etc.)
  → exception propagates — charge_id is never returned to caller
  → lock is released (finally block fires) ✓
  → next worker acquires lock, idempotency check finds no record, charges again
```

**Finding 3 — DB commit failure after successful charge produces an unrecorded charge and enables double-charge on retry.** The charge has settled at the gateway but the DB record was never committed. On the next attempt (by this worker or another), `db.get_charge()` returns `None`, the idempotency check passes, and a second charge is issued. This is a known hard problem in distributed payments, but the PR makes no mention of it and the checklist claims the design prevents double charges. At minimum, the design needs a compensating mechanism: either the payment gateway must support idempotency keys (passed as a stable per-order value, not a per-attempt UUID), or there must be a pre-charge reservation record written to the DB before the charge call, so that a DB failure before `record_charge` is detectable on retry.

---

### Path 5: Redis unavailable at lock acquisition

```
  → self.redis.set(...) → raises redis.exceptions.ConnectionError
  → not caught → propagates out of process() — UNCAUGHT
```

**Finding 4 — Redis connectivity failures are unhandled.** Same broken contract as Finding 1. A Redis outage causes an unhandled exception rather than a graceful `OrderResult(processed=False, ...)`.

---

### Path 6: `_release_script` fails silently

```
  → _release_script(keys=[lock_key], args=[token])
  → raises redis.exceptions.ConnectionError (Redis went away)
  → exception propagates out of finally block
  → original exception (if any) is suppressed by the finally exception
  → lock is never released — it will hold until TTL expiry
```

**Finding 5 — Exception in `finally` block can suppress the original exception and leaves lock unreleased until TTL.** The `finally` block has no error handling around the release script call. If Redis is unavailable at release time, the exception from the release attempt replaces any in-flight exception from the `try` block, making debugging harder. The lock will self-release at TTL expiry, but combined with Finding 2, that window is already known to be exploitable. The release call should be wrapped in a `try/except` with a logged warning.

---

### Additional Findings

**Finding 6 — `payment_token` logged indirectly.** The `payment_token` field passes through `OrderRequest` and is referenced in log lines via `req.order_id` and `charge_id`, but the raw `payment_token` value is present in the object throughout the call stack. If any upstream exception handler or debug logger serializes the `req` object, the token is exposed. The `payment_token` field should be marked with a custom `__repr__`/`__str__` that masks the value, or excluded from any serialized log output explicitly.

**Finding 7 — `customer_id` has no pattern constraint.** `order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` has only length bounds. If `customer_id` is used in any downstream query or log aggregation, it accepts arbitrary Unicode including control characters and injection-relevant sequences. Apply the same or equivalent pattern constraint.

**Finding 8 — `payment_token` has no format constraint.** `payment_token` accepts any 1–256 character string with no pattern. Depending on what the payment gateway does with this value, a malformed or adversarially crafted token could produce unexpected gateway behavior. At minimum, document the expected format; ideally enforce it with a pattern.

**Finding 9 — Test suite does not cover the TTL expiry / double-charge scenario.** The most critical failure mode (Finding 2) has zero test coverage. The checklist marks tests as passing, but the tests do not exercise the race condition the lock is designed to prevent.

**Finding 10 — `response.json()["charge_id"]` has no error handling.** If the gateway returns a 200 with a malformed body (missing `charge_id` key, non-JSON content type), this raises `KeyError` or `JSONDecodeError`, both of which propagate uncaught through `process()`.

---

## Decision: Request Changes

**Summary:** The lock design contains a critical correctness flaw: the worst-case payment client execution time (up to 100 seconds across retries) exceeds the lock TTL (60 seconds), meaning the lock can expire while a charge is in flight, allowing a second worker to pass the idempotency check and issue a duplicate charge — directly contradicting the PR's stated guarantee. Additionally, `PaymentError`, Redis connectivity failures, and DB commit exceptions all propagate as unhandled exceptions out of `process()`, breaking the structured `OrderResult` return contract and leaving callers with no recoverable signal. These two issues must be resolved before merge; the remaining findings (silent release failure, missing token masking, incomplete test coverage of the race condition, and missing `customer_id` pattern validation) should be addressed in the same pass.
