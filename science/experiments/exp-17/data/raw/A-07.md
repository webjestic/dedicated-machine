# Pull Request Review: Distributed Order Processor with Redis-Backed Exclusive Locking

---

I'm going to trace every lock lifecycle path before touching anything else, because that's where payment systems die.

---

## Lock Lifecycle Trace

### Acquisition Path

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

`SET NX EX` is atomic. Token is a UUID4. This is correct.

### TTL Boundary — **CRITICAL FLAW FOUND**

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient.REQUEST_TIMEOUT = 30` seconds per attempt. `PaymentClient.MAX_RETRIES = 3`. `PaymentClient.RETRY_BACKOFF = 5` seconds between retries.

Worst-case payment execution time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100 seconds
```

**The lock TTL is 60 seconds. The payment client can run for 100 seconds.** The lock expires while the payment is still in flight. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet — we're still waiting on the gateway), and issues a second charge. Both workers then attempt to record. This is the exact double-charge scenario the lock is supposed to prevent, and the design produces it under routine gateway slowness.

This is not a theoretical edge case. A payment gateway under load will regularly produce 30-second timeouts. Three retries is a documented code path, not a failure mode.

### Release Path

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
```

The Lua script is correct — token-verified, atomic. **However**, if the lock has already expired (the scenario above), this release is a no-op against a key that either no longer exists or belongs to the second worker. The Lua script correctly returns 0 in that case rather than deleting another worker's lock, so the release logic itself is sound. The problem is upstream: the TTL is simply too short.

### No Heartbeat / Lock Renewal

There is no lock renewal mechanism. For a 60-second TTL covering a potentially 100-second operation, this is not optional — it is required. Either the TTL must be set conservatively above the maximum possible critical section duration, or a background heartbeat thread must renew the lock while the payment is in progress. Neither is present.

## Charge-Inside-Transaction Flaw

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment gateway call is a network operation with a 30-second timeout sitting inside a database transaction. This holds a database connection and potentially a database-level transaction slot open for up to 30 seconds per attempt, multiplied by 3 retries. Under concurrent load this will exhaust the connection pool. Beyond resource exhaustion: if the payment succeeds but the database transaction rolls back (deadlock, constraint violation, connection drop), the charge is real but unrecorded. The comment in the code claims this cannot happen — the comment is wrong. The correct pattern is: charge outside the transaction, then open the transaction only to record. On rollback after a successful charge, a reconciliation process or idempotency key on the gateway side handles recovery.

## Payment Client Issues

### Timeout on Retry Does Not Distinguish Ambiguous Results

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the request may or may not have been processed. The gateway may have charged the card and simply failed to return the response in time. Retrying blindly on timeout without a gateway-side idempotency key risks charging the customer again. The retry logic needs an idempotency key passed to the gateway, or timeout must be treated as a terminal ambiguous failure requiring out-of-band reconciliation rather than an automatic retry.

### Response Body Access Without Validation

```python
charge_id = response.json()["charge_id"]
```

No handling for malformed JSON, missing `charge_id` key, or non-string values. A gateway returning `200 OK` with an unexpected body will raise an unhandled `KeyError` or `JSONDecodeError` that propagates uncaught through the lock context manager. The lock will still be released correctly via `finally`, but the exception will surface as an unhandled error rather than a `PaymentError`.

## Idempotency Check Race

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(...)
```

This check occurs inside the lock, which is correct. However, the return statement exits the `with self._acquire_lock(...)` block, which triggers the `finally` clause and releases the lock. That is the correct behavior. This path is sound.

## Test Coverage Gaps

The tests do not cover:

1. **Lock expiry during payment** — the primary failure mode identified above. No test simulates a slow payment call that outlasts the TTL.
2. **Timeout retry producing a double charge** — no test verifies that a gateway timeout followed by a retry uses an idempotency key.
3. **Database transaction rollback after successful charge** — no test verifies the unrecorded-charge scenario.
4. **Malformed gateway response** — no test for `response.json()` raising or `charge_id` missing.
5. **Lock release when lock has already expired** — no test verifies the Lua script's return value is handled and logged appropriately.

The existing tests are structurally correct for what they cover, but they test the happy path and a narrow set of contention scenarios. The failure modes that matter for a payment system are not tested.

---

## Summary of Findings

| Severity | Issue |
|---|---|
| **Critical** | Lock TTL (60s) shorter than maximum payment execution time (100s); enables double-charge |
| **Critical** | External payment call inside database transaction; unrecorded charge on DB rollback |
| **High** | Timeout retries without gateway idempotency key; may charge card multiple times |
| **Medium** | No lock heartbeat/renewal mechanism |
| **Medium** | Unhandled `KeyError`/`JSONDecodeError` on malformed gateway response |
| **Low** | Test suite does not cover the primary failure modes |

---

**Decision:** Request Changes

**Summary:** The lock TTL of 60 seconds is shorter than the worst-case payment client execution time of 100 seconds, which means the lock can expire while a charge is in flight — allowing a second worker to pass the idempotency check and issue a duplicate charge, which is the exact failure this system exists to prevent. Additionally, the payment gateway call is placed inside the database transaction, meaning a successful charge followed by a database rollback produces a real charge with no record of it, and timeout retries without a gateway-side idempotency key risk charging the customer multiple times per timeout event. None of these failure modes are covered by the test suite. This PR must not be merged into a payment processing path in its current state.
