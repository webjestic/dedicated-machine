Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, which means a successful external charge followed by any DB failure will produce an unrecorded, unrecoverable charge against the customer with no way to reconcile it from within this service. Additionally, there are several meaningful gaps in error handling and test coverage that need to be addressed before this is production-ready.

---

## Critical Issue

### 1. External side-effect inside database transaction — unrecoverable charge on DB failure

```python
# Current — DANGEROUS
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external, non-rollbackable
    self.db.record_charge(...)             # if this raises, charge is lost
    self.db.mark_order_complete(...)
```

The comment in the PR summary claims "crash between charge and record cannot produce an unrecorded charge" — **this is incorrect.** A DB transaction rollback does not undo an already-executed HTTP charge. If `record_charge` or `mark_order_complete` raises, the transaction rolls back but the money has already moved. The correct pattern is to charge first, then open the transaction only for the DB writes:

```python
# Correct — charge outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# DB writes are the only thing inside the transaction
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Charge succeeded but record failed — log with full context for manual reconciliation
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but DB record failed "
        "order_id=%s charge_id=%s amount_cents=%s customer_id=%s",
        req.order_id, charge_id, req.amount_cents, req.customer_id,
    )
    raise
```

This is a fundamental distributed systems constraint: you cannot atomically span an external payment API and a local database. The idempotency lock reduces the window but does not eliminate the need for a reconciliation path.

## Significant Issues

### 2. `PaymentError` is silently swallowed — treated as an unhandled exception

`process()` catches `LockError` but has no `except` block for `PaymentError` or any other exception. If `payment.charge()` raises, the exception propagates out of the `with self._acquire_lock()` context manager, the lock is correctly released via `finally`, but the caller receives an unhandled exception rather than a structured `OrderResult`. All expected failure modes should return a result, not raise:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 3. Lock TTL of 60 seconds is dangerously tight given retry configuration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire and be acquirable by another worker while the first worker is still mid-charge. This directly undermines the duplicate-charge guarantee the PR is designed to provide. Either increase the TTL to safely exceed the maximum payment client runtime, or pass a configurable TTL so it can be kept in sync:

```python
ORDER_LOCK_TTL = 120  # must exceed PaymentClient max runtime: 3*(30+5) = 105s
```

### 4. `payment_token` is logged at INFO level on success in `PaymentClient`

```python
# payment_client.py
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine as written, but `token` is passed in the JSON body and any future logging of the request payload would expose it. More immediately, `OrderProcessor` logs `payment_token` is part of the `OrderRequest` — confirm that Pydantic model is never serialized to logs anywhere in the pipeline. Consider marking `payment_token` as a `SecretStr` in the Pydantic model as a defensive measure:

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

## Minor Issues

### 5. Missing test coverage for critical failure paths

The test suite covers the happy path and lock contention but is missing coverage for the most important failure scenarios:

- `PaymentError` raised by `payment.charge()` — currently causes an unhandled exception (see issue 2)
- DB failure after a successful charge — the reconciliation path
- Lock TTL expiry mid-processing
- `payment.charge()` timeout behavior

These are the cases most likely to cause production incidents and should have explicit tests.

### 6. `charge_id` key access on payment response is unguarded

```python
charge_id = response.json()["charge_id"]
```

A malformed but HTTP 200 response from the gateway (missing `charge_id`, unexpected content type) will raise an unhandled `KeyError` or `JSONDecodeError`. This should be wrapped:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

### 7. Lock retry backoff sleeps on the final attempt

```python
if attempt < self.MAX_RETRIES - 1:
    time.sleep(self.RETRY_BACKOFF)
```

`PaymentClient` correctly skips the sleep on the last attempt. `_acquire_lock` does not — it sleeps after the final failed attempt before raising `LockError`. Minor, but adds unnecessary latency on contention:

```python
if acquired:
    break
if attempt < self.LOCK_RETRY_ATTEMPTS - 1:
    time.sleep(backoff)
```
