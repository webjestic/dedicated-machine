Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide, plus several reliability and correctness issues that need to be addressed before this can be approved.

---

## Critical Issues

### 1. Payment charge happens inside the DB transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* but this is incorrect. The DB transaction only governs database operations. `self.payment.charge(...)` is an external HTTP call — it is not enrolled in the DB transaction.

**What actually happens on a crash:**

```
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← HTTP call succeeds, money moves
    self.db.record_charge(...)             # ← process crashes here
    self.db.mark_order_complete(...)       # ← never reached
```

If the process dies after `payment.charge()` returns but before `record_charge()` commits, you have charged the customer with no record. The DB transaction rollback does nothing to reverse a completed external charge.

**Fix:** Move the payment call *outside and before* the DB transaction. Record the returned `charge_id` first, then open the transaction to persist it. Accept that you need a reconciliation/recovery path for the window between charge success and DB commit — that window cannot be made zero without a two-phase protocol with the payment gateway.

```python
# Charge first, outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record atomically in the DB
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This doesn't eliminate the crash window entirely, but it stops the false claim and makes the failure mode explicit. A background reconciliation job can detect charged-but-unrecorded orders via the gateway's API.

### 2. `PaymentError` is silently swallowed — unhandled exception escapes `process()`

`process()` only catches `LockError`. If `self.payment.charge()` raises `PaymentError`, or if `self.db.record_charge()` raises any DB exception, the exception propagates uncaught to the caller with no `OrderResult` returned. The method signature promises `OrderResult` but can raise.

```python
except LockError:
    return OrderResult(processed=False, message="...")
# PaymentError, DB errors, network errors → uncaught
```

**Fix:** Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

---

### 3. Lock TTL of 60 seconds is too short given the retry configuration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The Redis lock will expire while a worker is still mid-payment, allowing a second worker to acquire the lock and attempt a duplicate charge on the same order.

**Fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime (e.g., 120–150 seconds), or implement lock extension (Redlock-style keepalive). At minimum, document the dependency and add an assertion or configuration validation that enforces `ORDER_LOCK_TTL > max_payment_runtime`.

## Moderate Issues

### 4. `payment_token` is logged indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some versions. If `payment_token` fails validation, its value may appear in logs. Payment tokens are sensitive credentials.

**Fix:** Log only the field names that failed, not the full error object. Or explicitly redact the token field in the model's `__repr__`/`model_config`.

### 5. `PaymentClient` has no authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, API key, mTLS, or any credential passed to the internal payment gateway. Even on an internal network, unauthenticated calls to a payment endpoint are a security gap.

### 6. Retry-on-timeout in `PaymentClient` can cause duplicate charges

On a gateway timeout, the charge may have succeeded server-side but the response was lost in transit. Retrying blindly on timeout can result in a double charge. This is a known problem with payment APIs.

**Fix:** The payment gateway should support idempotency keys. Pass a stable idempotency key (e.g., derived from `order_id`) on every attempt so the gateway deduplicates retries:

```python
headers={"Idempotency-Key": order_id}
```

## Minor Issues

### 7. Test for lock release is fragile

`test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` and `processor._release_script.assert_called_once()`. This tests implementation detail rather than behavior. If the Lua script path changes the test breaks for the wrong reason. Consider testing the observable outcome (lock is gone after processing) rather than the mechanism.

### 8. No test for `PaymentError` handling

Given that `PaymentError` is currently unhandled (issue #2), there is no test that would catch this regression. A test case for payment failure is missing from the suite.

### 9. `CHARGE_ENDPOINT` and `RETRY_BACKOFF` are hardcoded

These should be injected via configuration or environment variables, not hardcoded in the class body. This makes the client untestable without monkey-patching and inflexible across environments.
