Decision: Request Changes

Summary: The core distributed-lock guarantee is broken in at least two ways that directly contradict the "at-most-once payment" claim in the docstring. There are also secondary issues in the payment client and test suite that need to be addressed before this touches production.

---

## Critical: Lock TTL vs. Payment Retry Window Creates a Double-Charge Race

This is the most dangerous flaw in the PR. Trace the execution path:

```
ORDER_LOCK_TTL  = 60 seconds
REQUEST_TIMEOUT = 30 seconds per attempt
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5 seconds between retries
```

Worst-case payment duration:

```
3 attempts × 30s timeout = 90s
+ 2 gaps   ×  5s backoff = 10s
─────────────────────────────
Total possible             = 100 seconds
```

The lock TTL is 60 seconds. The payment call can run for 100 seconds. The lock expires while the payment is still in flight. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet because the first worker hasn't returned), and issue a second charge. Both workers then record their charges. The customer is double-charged. The entire point of the lock is defeated.

**Required fix:** Either enforce `ORDER_LOCK_TTL > MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` with a documented constant relationship, or implement lock extension (a watchdog/heartbeat thread that refreshes the TTL while the payment is in progress), or reduce the payment client's maximum wall-clock time to a hard ceiling well below the TTL. Whichever approach is chosen, the relationship must be enforced in code, not just in comments, because these constants will drift independently over time.

## Critical: Payment Happens Outside the Database Transaction — Charge Can Be Lost

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)          # network call inside tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says "crash between charge and record cannot produce an unrecorded charge." That is incorrect. `self.payment.charge()` is a network call to an external system. If the process crashes, the database connection drops, or the transaction rolls back for any reason after `charge()` returns a `charge_id` but before `record_charge()` commits, the charge exists at the payment gateway and does not exist in the database. The lock is also released in the `finally` block, so a retry will pass the idempotency check and charge again.

Placing a non-idempotent external network call inside a database transaction does not make it atomic with the database write. These are two separate systems with no two-phase commit.

**Required fix:** The standard pattern for this class of problem is to write an intent record to the database first (in its own committed transaction), then call the payment gateway, then update the record with the result. On retry, the intent record is detected and the gateway is queried for the outcome of the prior attempt before issuing a new charge. The payment gateway also needs to be called with a client-generated idempotency key tied to the `order_id` so the gateway itself can deduplicate.

## Serious: Timeout Retry Silently Produces Ambiguous Charge State

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry
```

A timeout does not mean the charge failed. It means the response was not received within the window. The gateway may have processed the charge successfully. Retrying unconditionally on timeout without an idempotency key sent to the gateway will produce duplicate charges at the payment processor level, independent of the Redis lock. This is a standard pitfall and it is not handled here.

**Required fix:** Generate a per-order idempotency key (e.g., derived from `order_id`) and include it in every request to the payment gateway. On timeout, retry with the same key. The gateway deduplicates and returns the original result.

## Serious: Lock Release Swallows All Exceptions

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `self._release_script` raises (Redis connection lost, script error), the exception propagates out of the `finally` block and suppresses any exception that was already being handled from the `try` body. The caller receives a Redis error instead of the original business logic error. Additionally, there is no logging or handling of the case where the script returns `0`, meaning the lock had already expired and was potentially held by another worker at release time — which is itself a signal that the TTL/duration invariant was violated.

**Required fix:** Wrap the release call in its own try/except, log failures, and check the return value. A return of `0` should be logged as a warning indicating the lock expired before release.

## Moderate: `payment_token` Echoed Into Logs

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is present in the `OrderRequest` model and will appear in any debug-level serialization of that object (e.g., `logger.debug("Processing request: %s", req)`). Payment tokens are sensitive credentials. The `payment_token` field should be excluded from any `__str__`, `__repr__`, or model serialization. Pydantic v2 supports `model_config` with field-level exclusion or a custom `__repr__`.

## Moderate: Test Suite Does Not Cover the Failure Paths That Matter

The tests verify the happy path and basic contention, but none of the critical failure modes are tested:

- No test for payment timeout followed by retry (ambiguous charge state).
- No test for process crash after `payment.charge()` returns but before `db.record_charge()` commits.
- No test for lock TTL expiry during payment processing.
- No test for `_release_script` raising an exception.
- No test verifying that a `PaymentError` from the client does not leave a dangling charge.

The existing `test_lock_release_uses_lua_script` test asserts `processor._release_script.assert_called_once()` but the fixture sets `mock.register_script.return_value = script_mock` and then the processor stores `self._release_script = redis_client.register_script(RELEASE_LOCK_SCRIPT)`. The test is asserting on `processor._release_script` which is the same mock object — this works, but it is testing the mock wiring, not the Lua script logic. The actual script should be tested against a real or emulated Redis instance (e.g., `fakeredis`) to verify the token-comparison branch.

## Minor: `LOCK_RETRY_ATTEMPTS = 3` with No Total Timeout Bound

Three attempts at 0.5s, 1.0s, 1.5s backoff means a worker can block for up to 3 seconds waiting for a lock. Under high concurrency this is acceptable, but the retry count and backoff are not documented relative to expected lock hold time. If the lock holder is slow (approaching the TTL), retrying three times and failing is correct behavior. If the lock holder crashed and the TTL has not yet expired, the caller will fail and the order will not be retried. This is a design decision that should be explicitly documented, not left implicit.
