Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured and the Lua-based token-verified release is correct, but there is a critical atomicity flaw: the payment charge call lives *inside* the database transaction, meaning a successful charge that precedes a DB commit failure will produce an unrecorded, unrecoverable charge — directly contradicting the comment claiming crash safety. There are also several secondary reliability and security issues that need to be addressed before this is mergeable.

---

## Critical Issue

### 1. Payment charge inside DB transaction violates the stated at-most-once guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect
    self.db.record_charge(...)             # if this or commit fails, charge is lost
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. The `payment.charge()` call is an external HTTP side-effect that cannot be rolled back. If the DB commit fails after a successful charge, you have charged the customer with no record of it. The comment in the code ("crash between charge and record cannot produce an unrecorded charge") is factually incorrect and dangerous.

**Required fix — two-phase approach:**

```python
# Step 1: persist intent before charging
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Step 2: charge externally (outside DB transaction)
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Step 3: confirm the record atomically
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This makes the pending record the source of truth. A recovery process can detect `pending` rows and reconcile them against the payment gateway. This is the standard pattern for at-most-once external payment calls.

## High Severity Issues

### 2. Redis connection failure is silently fatal with no structured handling

Any `redis.RedisError` (connection refused, timeout, cluster failover) will propagate as an unhandled exception out of `process()`. The caller gets a raw exception instead of an `OrderResult`. Given that Redis is a hard dependency for the lock, this needs explicit handling.

```python
# Add to process():
except redis.RedisError as e:
    logger.error("Redis failure during order processing order_id=%s: %s", order_id, e)
    return OrderResult(processed=False, message="Temporary system error, please retry")
```

Similarly, `_acquire_lock` can raise `redis.RedisError` on the `set()` call before `acquired` is ever set to `True`, leaving the lock state undefined.

### 3. Payment timeout is ambiguous and not retried safely

In `payment_client.py`, a `Timeout` exception means the request may or may not have reached the gateway and may or may not have been processed. Retrying blindly on timeout can cause duplicate charges if the first request succeeded but the response was lost.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # Retrying here is unsafe — the charge may have already been processed
```

Timeout retries require idempotency keys sent to the gateway so the provider can deduplicate. Either add an idempotency key to the request headers or do not retry on timeout.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be stable across retries
    timeout=self.REQUEST_TIMEOUT,
)
```

### 4. Lock TTL may expire before the payment round-trip completes

`ORDER_LOCK_TTL = 60` seconds and `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds means the worst-case payment client runtime is `3 × 30 + 2 × 5 = 100 seconds`, which exceeds the lock TTL by 40 seconds. A second worker can acquire the lock and begin processing while the first is still in a retry loop.

Either increase the TTL to safely exceed the worst-case payment duration, or implement lock extension (Redlock-style keepalive). At minimum, document the constraint and enforce it with a configuration assertion.

```python
assert self.ORDER_LOCK_TTL > (
    self.payment.REQUEST_TIMEOUT * self.payment.MAX_RETRIES +
    self.payment.RETRY_BACKOFF * (self.payment.MAX_RETRIES - 1)
), "Lock TTL must exceed worst-case payment duration"
```

## Medium Severity Issues

### 5. `payment_token` is logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some versions. If `payment_token` fails validation, its value may appear in log output. Payment tokens must never appear in logs.

```python
except ValidationError as e:
    logger.warning("Invalid order request fields: %s", [err["loc"] for err in e.errors()])
    return OrderResult(processed=False, message="Invalid order parameters")
```

### 6. `charge_id` not included in `OrderResult` for already-processed orders

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id is missing — callers cannot retrieve it for idempotent responses
    )
```

For idempotent retry scenarios, the caller needs the original `charge_id`. Return `existing.get("charge_id")` here.

### 7. No `PaymentError` handling in `process()`

`self.payment.charge()` raises `PaymentError` on permanent failure, but `process()` has no handler for it. This will propagate as an unhandled exception rather than returning a structured `OrderResult`.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Low Severity / Code Quality

### 8. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute was called, which is brittle and couples the test to implementation. A more meaningful test would verify that a second call to `process()` with the same `order_id` can acquire the lock after the first completes, i.e., test the observable behavior.

### 9. No test for Redis failure or PaymentError propagation

The test suite has no coverage for Redis exceptions or `PaymentError`. These are the two most likely production failure modes and should be explicitly tested.

### 10. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This should be injected via configuration or environment variable. Hardcoded internal endpoints make environment promotion (staging vs. production) error-prone.
