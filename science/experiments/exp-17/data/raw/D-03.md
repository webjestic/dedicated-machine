Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that is followed by a DB commit failure will result in a real charge with no recorded transaction (the comment in the code claims the opposite guarantee, which makes this worse). Additionally, there are several secondary issues around exception handling gaps and test coverage that need to be addressed before this is merge-ready.

---

## Critical Issue

### 1. Payment call inside DB transaction violates the stated atomicity guarantee

```python
# services/order_processor.py, lines inside with self.db.transaction():
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect inside TX
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The inline comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is **incorrect**. A DB transaction only makes DB operations atomic. If `payment.charge()` succeeds but the DB commit fails (deadlock, connection drop, serialization failure), you have charged the customer with no record of it. The external call is not rolled back when the DB transaction rolls back.

**Required fix:** Charge outside and before the transaction, then record inside it. Use a two-phase approach with a pre-commit idempotency record:

```python
# 1. Write a "charge pending" sentinel inside a transaction BEFORE charging
with self.db.transaction():
    self.db.reserve_order(req.order_id)  # idempotency sentinel; unique constraint prevents races

# 2. Charge externally — if this crashes, the sentinel prevents re-entry on retry
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 3. Record the completed charge
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This pattern is safe: the unique constraint on `reserve_order` is the true idempotency gate, and the Redis lock only prevents redundant concurrent attempts during the window before that record exists.

## Secondary Issues

### 2. `PaymentError` is silently swallowed

```python
# services/order_processor.py
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # raises PaymentError on failure
```

`PaymentError` (and any other unexpected exception) is not caught in `process()`. It will propagate out of the lock context manager, which does correctly release the lock via `finally`, but the caller receives an unhandled exception rather than a structured `OrderResult`. This is inconsistent with the method's contract and will cause unhandled 500s at the API layer.

```python
# Add inside the lock block:
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 3. Timeout on payment gateway creates an ambiguous charge state

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retries immediately
```

On a timeout, the charge may or may not have been applied. Retrying with the same `token` and `amount_cents` without an idempotency key passed to the gateway risks a double charge. The payment client should include a stable per-attempt or per-order idempotency key in the request header (e.g., `Idempotency-Key: <order_id>-<attempt>`), which most payment gateways honor.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # pass through from caller
    timeout=self.REQUEST_TIMEOUT,
)
```

### 4. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises `KeyError` or `JSONDecodeError`, neither of which is caught. These will propagate as unhandled exceptions. Add explicit handling:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

### 5. Lock TTL is too short relative to retry window

`ORDER_LOCK_TTL = 60` seconds, but `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds per retry and `REQUEST_TIMEOUT = 30` seconds per attempt. Worst-case payment time is `3 * (30 + 5) = 105` seconds, which exceeds the lock TTL. Another worker could acquire the lock while the first is still in a retry loop, defeating the duplicate-charge protection entirely. Either increase `ORDER_LOCK_TTL` to at least 120 seconds or reduce the payment client's worst-case ceiling.

### 6. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()  # ← does not check args
```

This test verifies the script was called but not that the correct token was passed. A regression that passes a hardcoded or empty token would go undetected. Assert on `call_args`:

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] is not None  # or capture and compare the token
```

### 7. No test for `PaymentError` propagation or DB commit failure

Given that the critical issue above involves the DB commit path, there is no test that exercises a `PaymentError` being raised or a DB transaction failure. These are exactly the failure modes that matter for a payment system and must be covered before merge.
