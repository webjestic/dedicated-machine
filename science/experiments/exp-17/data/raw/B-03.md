Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic input validation, and idempotency checks are all correctly structured. However, there are several correctness and reliability defects that must be resolved before this can be approved: a critical charge-outside-transaction flaw that breaks the atomicity guarantee the PR claims to provide, a missing heartbeat mechanism that makes the 60-second TTL a hard failure boundary under slow payment gateways, and a timeout/retry interaction in the payment client that can produce undetected duplicate charges. These are not style concerns — they are failure-mode bugs in a payment-critical path.

---

## Detailed Findings

### 🔴 Critical — Charge Occurs Outside the Database Transaction

**File:** `services/order_processor.py`, lines inside `process()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← network call to external gateway
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary states: *"charges the payment method and records the transaction within a single atomic database transaction."* This is incorrect. `self.payment.charge()` is an HTTP call to an external payment gateway. It is not part of the database transaction. The database transaction only wraps the *recording* of the charge, not the charge itself — which is the correct and only possible design, since you cannot enlist an external HTTP call in a local DB transaction.

The actual failure scenario this creates:

1. `payment.charge()` succeeds and money moves.
2. The process crashes before `db.record_charge()` commits.
3. On retry, `db.get_charge()` returns `None` (nothing was recorded).
4. The idempotency check passes.
5. The customer is charged a second time.

The comment *"crash between charge and record cannot produce an unrecorded charge"* is factually wrong and will mislead future maintainers into trusting a guarantee that does not exist.

**Required fix:** The charge must be made idempotent at the gateway level. Pass a stable idempotency key (e.g., `order_id`) to the payment gateway so that a retry of the same `order_id` returns the original `charge_id` rather than creating a new charge. The idempotency check in the DB then becomes a performance optimization (avoiding a redundant gateway call) rather than the sole safety barrier.

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway deduplicates on this
)
```

The misleading comment must also be corrected or removed.

### 🔴 Critical — Lock TTL Expiry Under Slow Payment Gateway Creates Double-Charge Window

**File:** `services/order_processor.py`

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

**File:** `services/payment_client.py`

```python
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5    # seconds between retries
```

Worst-case payment client wall time: `(30s × 3 attempts) + (5s × 2 gaps) = 100 seconds`.

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. There is no heartbeat renewal. This means:

1. Worker A acquires the lock (TTL = 60s).
2. The payment gateway is slow; Worker A is still waiting at t = 61s.
3. The lock expires. Worker B acquires the lock on the same order.
4. Worker B passes the idempotency check (no DB record yet) and charges the customer.
5. Worker A's charge eventually succeeds. Worker A records the charge.
6. Worker B's charge also succeeds. Double charge.

**Required fix (choose one or both):**

- **Option A — Heartbeat renewal:** Implement a background thread or async task that renews the lock TTL (e.g., `EXPIRE order:lock:{order_id} 60`) every 20–30 seconds while the worker holds it.
- **Option B — Align TTL with maximum client runtime:** Set `ORDER_LOCK_TTL` to at least `MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF + safety_margin`, which is `≥ 120s` given current payment client settings. This is a weaker fix because it does not handle indefinite hangs, but it closes the current gap.

The TTL and payment client timeout budget must be explicitly co-owned. If either value changes independently, the invariant silently breaks. Document the dependency or enforce it programmatically:

```python
assert self.ORDER_LOCK_TTL > (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
), "Lock TTL must exceed maximum payment client runtime"
```

### 🔴 Critical — Timeout Ambiguity Enables Silent Duplicate Charges

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` exception on a POST to a payment gateway means the request may have been received and processed by the gateway — the timeout occurred waiting for the *response*, not necessarily during transmission. Retrying unconditionally after a timeout can charge the customer multiple times if the gateway processed the first request.

**Required fix:** Pass a stable idempotency key in the request header (e.g., `Idempotency-Key: {order_id}` or a per-attempt UUID that is held constant across retries of the same logical charge). This is the same fix as the one required above and reinforces why gateway-level idempotency is non-negotiable in this design.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

### 🟡 Moderate — Lock Release Silently Swallows Errors

**File:** `services/order_processor.py`

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script returns `0` (meaning the lock was not held by this worker — e.g., it expired and was acquired by another), the release silently logs a success message. This is misleading and masks the exact condition that the TTL/heartbeat issue above produces.

**Required fix:**

```python
finally:
    released = self._release_script(keys=[lock_key], args=[token])
    if released:
        logger.info("Order lock released order_id=%s", order_id)
    else:
        logger.error(
            "Order lock was not held at release time — TTL may have expired "
            "while processing order_id=%s; potential concurrent execution",
            order_id,
        )
```

This turns a silent failure into an observable signal in your monitoring pipeline.

### 🟡 Moderate — `payment_token` Logged Indirectly via Pydantic Validation Errors

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the truncated or full token value may appear in the log output. Payment tokens must never appear in logs.

**Required fix:** Either suppress the value from the error output explicitly, or log only field names without values:

```python
except ValidationError as e:
    field_names = [err["loc"] for err in e.errors()]
    logger.warning("Invalid order request fields=%s", field_names)
```

### 🟡 Moderate — No `ConnectionError` Handling in Payment Client

**File:** `services/payment_client.py`

`requests.exceptions.ConnectionError` (DNS failure, TCP reset, network partition) is not caught. It will propagate as an unhandled exception through `process()`, bypass the `LockError` handler, and surface as an unstructured 500 to the caller. It should be caught and treated identically to `Timeout` — retryable, with the same idempotency-key caveat.

---

### 🟢 Minor — Test Coverage Gaps

The test suite covers the happy path and basic contention, but is missing:

- **Lock expiry during processing:** Verify the `released == 0` branch in the lock release path is observable.
- **Payment timeout followed by DB commit:** Simulate a timeout on attempt 1 and success on attempt 2 to confirm idempotency key prevents double charge (once the idempotency key is added).
- **`ConnectionError` from payment gateway:** Confirm it is handled gracefully after the fix above.
- **Transaction rollback on `record_charge` failure:** Confirm the payment charge is not re-attempted when the DB write fails (this is related to the gateway idempotency fix).
