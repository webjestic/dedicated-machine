Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful external charge followed by any DB failure will result in a real charge with no recorded transaction. The payment call must happen outside (and before) the DB transaction, with the resulting `charge_id` then persisted atomically. Additionally, there are several secondary issues that need to be addressed before this merges.

---

## Critical

### 1. Payment call inside DB transaction — unrecoverable charge loss on DB failure

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side-effect
    self.db.record_charge(...)                # if this or mark_order_complete raises,
    self.db.mark_order_complete(...)          # the DB rolls back but the charge already happened
```

The comment in the PR even says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. A DB rollback after a successful gateway call is exactly that scenario. The fix is to charge first, then record in the transaction, and handle the edge case explicitly:

```python
# Correct pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Charge succeeded but record failed.
    # Log with full context for manual reconciliation — do NOT silently swallow.
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but DB record failed "
        "order_id=%s charge_id=%s customer_id=%s amount_cents=%d",
        req.order_id, charge_id, req.customer_id, req.amount_cents,
    )
    raise
```

This doesn't make the problem disappear — distributed systems can't fully eliminate this gap without two-phase commit or an outbox pattern — but it makes the failure mode visible and recoverable rather than silently lost.

## High

### 2. Timeout on payment gateway is ambiguous — retry can double-charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded on the gateway side. Retrying a timed-out charge without an idempotency key can result in multiple charges for the same order. The gateway call must include a stable idempotency key derived from `order_id`:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # pass order_id or a hash of it
    timeout=self.REQUEST_TIMEOUT,
)
```

`PaymentClient.charge()` needs to accept and forward an idempotency key. The caller (`OrderProcessor`) should derive it from `req.order_id`.

### 3. Unhandled exceptions inside the lock context leak the lock early on Redis failure

If `self.db.get_charge()` or the transaction block raises an unexpected exception, it propagates out of the `with self._acquire_lock(...)` block. The `finally` clause will attempt to release the lock, which is correct — but if the Redis client itself is unavailable at that moment, `_release_script()` will raise, masking the original exception. Wrap the release:

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
    except Exception:
        logger.error("Failed to release lock for order_id=%s — TTL will expire it", order_id)
```

### 4. `PaymentError` is not caught in `process()`

If the payment gateway returns a permanent 4xx or exhausts retries, `PaymentError` propagates uncaught out of `process()`, which is documented to return `OrderResult`. This will surface as an unhandled exception to callers. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Medium

### 5. `payment_token` logged indirectly — verify it never reaches log output

`payment_token` is stored in `req` and passed to `payment.charge()`. Confirm that `PaymentClient` never logs the token value. The current code does not appear to, but this should be explicit — add a note or a scrubbing wrapper, since accidental token logging in a future change would be a PCI concern.

### 6. Lock TTL of 60s may be too short under retry backoff

`MAX_RETRIES=3` with `RETRY_BACKOFF=5s` per attempt means the payment client can take up to ~90 seconds in the worst case (3 attempts × 30s timeout + 2 × 5s backoff = 100s). The lock TTL of 60s can expire before the payment completes, allowing a second worker to acquire the lock and potentially charge again. Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client duration, or implement lock extension (e.g., a background thread that refreshes the TTL while the payment is in-flight).

### 7. `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary Unicode up to 64 characters. If `customer_id` is used in log formatting, DB queries, or downstream calls, this is an injection surface. Apply the same or a comparable pattern constraint.

---

## Low / Test Coverage Gaps

### 8. No test for `PaymentError` propagation

There is no test covering the case where `payment.charge()` raises `PaymentError`. Once issue #4 is fixed, a test should assert that `process()` returns a non-processed `OrderResult` rather than raising.

### 9. No test for lock TTL expiry during payment

The scenario in issue #6 (lock expires mid-payment) has no coverage. A test simulating a slow payment call relative to TTL would catch regressions here.

### 10. `response.json()["charge_id"]` has no error handling

If the gateway returns a 200 with a malformed or missing `charge_id` key, this raises an unhandled `KeyError`. Add explicit handling:

```python
data = response.json()
if "charge_id" not in data:
    raise PaymentError(f"Gateway returned success but no charge_id: {data}")
charge_id = data["charge_id"]
```
