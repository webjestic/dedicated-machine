## Lock Lifecycle Trace

Before I render a verdict, I'm walking every execution path.

---

### Path 1: Happy Path

```
acquire(SET NX EX 60) → yield token → db.get_charge() → db.transaction() →
payment.charge() → db.record_charge() → db.mark_order_complete() →
finally: release_script(key, token)
```

Lock TTL is 60 seconds. `payment.charge()` has a 30-second timeout × 3 attempts + 5-second backoff × 2 gaps = **up to 100 seconds of payment client wall time**. The lock expires at second 60. The payment call can still be in-flight when the TTL boundary is crossed, the key evicts, a second worker acquires a fresh lock, passes the idempotency check (charge not yet recorded), and issues a second charge. The database transaction hasn't committed yet. **This is a double-charge window.** The lock TTL does not cover the worst-case payment round-trip, and there is no heartbeat renewal to extend it.

---

### Path 2: Payment Succeeds, DB Transaction Crashes

```
payment.charge() → returns charge_id → db.record_charge() raises → transaction rolls back
```

The charge is real and settled at the gateway. The database has no record of it. On retry, `db.get_charge()` returns `None`, the idempotency check passes, and a second charge is issued. The comment in the code claims *"crash between charge and record cannot produce an unrecorded charge"* — this claim is **false**. The payment call is outside the transactional boundary of the database. A DB transaction cannot roll back a network call to an external payment gateway. The comment is actively misleading and will cause the next engineer to trust a guarantee that does not exist.

---

### Path 3: Lock TTL Expires Mid-Execution, Worker Continues

The `finally` block calls `_release_script(keys=[lock_key], args=[token])`. If the TTL has expired and a second worker has acquired the lock with its own token, the Lua script correctly returns `0` and does not delete the second worker's lock. That part is sound. However, Worker 1 is still executing past the TTL boundary — the lock has stopped protecting it. The release being safe does not undo the window described in Path 1.

---

### Path 4: Timeout on Payment, Retry Loop

A `requests.exceptions.Timeout` is caught and retried. A timeout means the gateway **may or may not have processed the charge** — the response was simply not received. Retrying on timeout without first checking for an existing charge at the gateway is a classic double-charge vector at the payment client layer, independent of the Redis lock. There is no idempotency key passed to the gateway in the POST body.

---

### Path 5: `payment_token` in Logs

`payment_token` is accepted as a field on `OrderRequest` and flows into `payment.charge(token=req.payment_token, ...)`. The structured logging logs `order_id` and `charge_id` but not the token directly — that part is acceptable. However, if any exception path logs `req` or the full request object (e.g., a future debug log), the token leaks. The `payment_token` field has no `repr=False` or `exclude` annotation on the Pydantic model to prevent accidental serialization. Minor, but worth hardening at this layer given the sensitivity of the field.

---

### Path 6: `db.get_charge()` Inside the Lock, Race on Read

The idempotency check reads from the database inside the lock. If `db.get_charge()` reads from a replica with replication lag, a charge recorded by a worker whose DB write hasn't propagated yet will be invisible. The idempotency check will pass and a second charge will be issued. This is a known hazard when the DB layer is not explicitly directed to read from primary inside the critical section. The code makes no such guarantee and the DB abstraction is opaque.

---

### Additional Observations

**`PaymentError` is not caught in `process()`.**  
If `payment.charge()` raises `PaymentError`, it propagates out of `process()` entirely, bypassing the `LockError` handler and surfacing as an unhandled exception to the caller. The lock is still released correctly via `finally`, but the caller receives an exception rather than an `OrderResult`. The method signature promises `OrderResult` — this is a contract violation.

**Test coverage gaps.**  
- No test for lock TTL expiry mid-execution.  
- No test for `PaymentError` propagation — the unhandled exception path is untested.  
- No test for `db.record_charge()` raising after `payment.charge()` succeeds (the false atomicity claim).  
- No test for timeout retry behavior in `PaymentClient`.  
- The `test_lock_release_uses_lua_script` test asserts `redis_client.delete` is not called, but this only confirms the mock path — it does not verify the Lua script receives the correct token.

**`RETRY_BACKOFF` is a class-level constant, not configurable at construction.**  
In a test environment this forces real `time.sleep(5)` calls unless patched. The payment client has no dependency injection point for the sleep function, making deterministic testing of retry behavior difficult.

---

## Decision: Request Changes

**Summary:** The lock TTL (60 s) is shorter than the maximum payment client execution time (≈100 s), creating a concrete double-charge window when payment retries exhaust near the TTL boundary — this is the primary blocker. The claim that placing `payment.charge()` inside `db.transaction()` provides atomicity is architecturally false and the misleading comment must be corrected; the correct fix is an idempotency key submitted to the gateway so that gateway-side deduplication covers the timeout-retry path. `PaymentError` is also unhandled in `process()`, breaking the `OrderResult` return contract on payment failure.
