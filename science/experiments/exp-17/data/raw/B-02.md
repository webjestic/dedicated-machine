Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several correctness and reliability flaws that would make this unsafe to deploy in a payment-critical environment: the most serious is that the payment call sits *inside* a database transaction, creating a window where a gateway timeout on retry produces an unrecorded charge despite the comment claiming otherwise, and the lock has no heartbeat mechanism meaning any payment round-trip approaching the 60-second TTL boundary can expire the lock mid-execution and allow a second worker to enter. Additional issues are catalogued below.

---

## Detailed Findings

### Critical

**1. Payment call inside `db.transaction()` violates the stated at-most-once guarantee**

```python
# services/order_processor.py  lines 107–113
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external I/O inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment reads *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. If `payment.charge()` succeeds at the gateway but the response is lost (network partition, worker OOM, process kill), the database transaction is never committed. The charge exists at the gateway; it does not exist in the database. On the next attempt the idempotency check finds no record and charges again.

The correct pattern is to record a *pending* charge row **before** calling the gateway, then confirm or void it after:

```python
# Sketch of correct ordering
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

A reconciliation job should sweep `pending` rows older than a threshold and void or confirm them via the gateway's idempotency key endpoint.

---

**2. No lock heartbeat — TTL expiry during payment I/O breaks mutual exclusion**

`ORDER_LOCK_TTL = 60` seconds is a fixed wall-clock budget. `PaymentClient` can consume up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The lock expires before the payment completes. A second worker then acquires the lock, passes the idempotency check (no record yet), and issues a second charge.

A heartbeat thread or a periodic `EXPIRE` renewal is required:

```python
import threading

def _heartbeat(self, lock_key: str, token: str, stop_event: threading.Event):
    interval = self.ORDER_LOCK_TTL // 2
    while not stop_event.wait(interval):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.ORDER_LOCK_TTL)
        else:
            break  # lock was lost; stop renewing
```

The heartbeat must be stopped and joined in the `finally` block of `_acquire_lock`.

### High

**3. `payment_token` logged or passed through without scrubbing**

`payment_token` flows through `OrderRequest`, is passed directly to `payment.charge()`, and the `PaymentClient` posts it in a JSON body. If structured logging is ever configured to serialize the full request object, or if an exception causes the payload to appear in a traceback, raw payment tokens reach log aggregators. Payment tokens are not card numbers but they are still sensitive bearer credentials.

At minimum, the token should be excluded from any `__repr__` or log serialization:

```python
class OrderRequest(BaseModel):
    payment_token: str = Field(..., min_length=1, max_length=256)

    model_config = {"json_schema_extra": {}}  # not sufficient alone

    def __repr__(self):
        return f"OrderRequest(order_id={self.order_id!r}, customer_id={self.customer_id!r}, amount_cents={self.amount_cents!r}, payment_token='[REDACTED]')"
```

**4. Timeout on `payment.charge()` is retried without gateway-side idempotency key**

```python
# services/payment_client.py  lines 28–30
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
```

On a `Timeout`, the request may have been received and processed by the gateway. Retrying without an idempotency key sends a second, independent charge request. The fix is to generate a stable idempotency key per order (e.g., derived from `order_id`) and include it as a header on every attempt:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### Medium

**5. `LockError` and `PaymentError` are silently swallowed at the `process()` boundary**

`PaymentError` raised inside `_acquire_lock` context propagates out of `process()` as an unhandled exception — the caller receives a raw exception rather than an `OrderResult`. The `except LockError` block does not catch `PaymentError`, `requests.exceptions.ConnectionError`, or any database exception. These should either be caught and mapped to structured `OrderResult` returns, or explicitly re-raised with a documented contract.

---

**6. Lock retry backoff is linear and blocks the worker thread**

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

Three attempts with linear backoff and no jitter means all workers under contention wake at the same time and produce a thundering herd against Redis. Use exponential backoff with jitter:

```python
import random
backoff = min(0.1 * (2 ** attempt) + random.uniform(0, 0.1), 2.0)
```

---

**7. `CHARGE_ENDPOINT` is a hardcoded string constant**

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Endpoint configuration must come from environment variables or a secrets manager, not source code. This prevents environment-specific deployment and makes rotation require a code change.

---

### Test Coverage Gaps

**8. No test for lock expiry during payment (the critical TTL race)**
The test suite does not exercise the scenario where the lock TTL expires while `payment.charge()` is in flight. This is the most dangerous failure mode and should have an explicit test.

**9. No test for `PaymentError` propagation**
There is no test verifying that a `PaymentError` raised by `payment.charge()` is handled gracefully and does not leave a dangling lock or uncommitted state.

**10. `test_lock_release_uses_lua_script` does not verify the token argument**

```python
processor._release_script.assert_called_once()
```

This asserts the script was called but does not verify that the correct token was passed as `args=[token]`. A regression that passes the wrong token would not be caught.
