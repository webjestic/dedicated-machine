Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured and the Lua-based token-verified release is correct, but there is a critical atomicity flaw that breaks the at-most-once guarantee the PR claims to provide. There are also several secondary issues around error handling, timeout exposure, and test coverage gaps that need to be addressed before this is safe to merge.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* but this is incorrect. `self.payment.charge()` is a network call to an external payment gateway. It is **not** a database operation and cannot participate in the database transaction. The database transaction only wraps the DB writes. If the process crashes after `payment.charge()` returns a `charge_id` but before `db.record_charge()` commits, the charge is real and unrecorded. The comment is actively misleading.

```python
# What the code claims:
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call, NOT transactional
    self.db.record_charge(...)             # ← only this is in the transaction
```

The correct pattern for at-most-once semantics is:

1. Write a `PENDING` charge record to the DB **before** calling the gateway (within a transaction).
2. Call the gateway.
3. Update the record to `COMPLETED` with the returned `charge_id`.
4. On restart/retry, detect the `PENDING` record and either reconcile with the gateway or surface it for manual review.

The Redis lock alone does not close this gap — it only prevents concurrent workers, not a crash mid-flight.

---

### 2. `PaymentError` and general exceptions are silently swallowed in `process()`

The `process()` method catches `LockError` but has no handler for `PaymentError` or any other exception. If `payment.charge()` raises `PaymentError` or if a DB call throws, the exception propagates out of the `with self._acquire_lock(...)` block. The `finally` clause in `_acquire_lock` will still release the lock, which is correct, but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract of the method signature.

```python
# PaymentError, DB exceptions, network errors — all unhandled
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # can raise PaymentError
```

Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

## Significant Issues

### 3. Total timeout exposure is dangerously high

With `MAX_RETRIES=3`, `REQUEST_TIMEOUT=30s`, and `RETRY_BACKOFF=5s`, a single `payment.charge()` call can block for up to **100 seconds** `(3 × 30) + (2 × 5)`. The Redis lock TTL is 60 seconds. This means the lock can expire while a payment attempt is still in flight, allowing a second worker to acquire the lock and attempt a duplicate charge — directly undermining the guarantee the lock is supposed to provide.

Either:
- Reduce `REQUEST_TIMEOUT + (MAX_RETRIES × RETRY_BACKOFF)` to be well under `ORDER_LOCK_TTL`, or
- Extend the lock TTL to safely exceed the maximum payment client duration, or
- Implement lock refresh (heartbeat) during long operations.

The numbers as written are inconsistent and will cause intermittent duplicate charges under load.

---

### 4. `payment_token` is logged indirectly through `charge_id` but the token itself could appear in exception messages

In `PaymentClient`, the `token` is passed in the request body. If `requests` raises a connection error or the response body contains the token in an error message, it could appear in logs. Payment tokens must be treated as secrets. Ensure log statements never include `token` and that exception messages from the HTTP layer are sanitized before logging.

---

### 5. No authentication or authorization on the payment endpoint call

`PaymentClient` posts to `https://api.payments.internal/v1/charges` with no authentication headers. Even on an internal network, this should use mTLS or a bearer token. Unauthenticated internal service calls are a common lateral movement vector.

## Minor Issues

### 6. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute was called, which is a brittle implementation test. It will break on any internal refactor. More importantly, there is no test that verifies the lock is released when an exception is raised mid-processing — the most important behavioral guarantee of the `finally` block.

Add a test like:

```python
def test_lock_released_on_payment_failure(processor, db, payment):
    payment.charge.side_effect = PaymentError("declined")
    with pytest.raises(PaymentError):  # or check OrderResult once handled
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    processor._release_script.assert_called_once()
```

### 7. `response.json()["charge_id"]` will raise `KeyError` if the gateway returns unexpected JSON

Wrap this in a try/except or use `.get()` with a fallback, and raise a clear `PaymentError` rather than letting a `KeyError` propagate.

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned no charge_id: {data}")
```
