Decision: Request Changes

Summary: The core locking architecture contains a critical atomicity flaw: a payment charge is initiated *inside* a database transaction but *outside* any rollback boundary that could undo it, meaning a crash or exception after `payment.charge()` succeeds but before `db.transaction().__exit__` commits will produce an unrecorded charge that the idempotency check will never detect — directly contradicting the PR's own safety claim. Additionally, the Redis lock TTL is dangerously mismatched with the cumulative payment retry window, and the lock is silently swallowed on release failure with no observable signal. These are not edge cases; they are the exact failure modes this service is supposed to prevent.

---

## Detailed Findings

### 1. CRITICAL — Charge Occurs Inside a Transaction That Cannot Roll It Back

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # external side effect — NOT transactional
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR description states: *"crash between charge and record cannot produce an unrecorded charge."* This is false. A database transaction provides atomicity over database operations only. `self.payment.charge()` is an HTTP call to an external payment gateway. If the process crashes, the network drops, or the DB commit fails *after* the gateway has already debited the customer, the charge exists in the payment system but `db.record_charge` was never committed. On the next attempt, `db.get_charge()` returns `None`, the idempotency check passes, and the customer is charged again.

The correct pattern here is one of:

- **Outbox/saga pattern**: Write a pending charge intent to the DB first, process it asynchronously, and reconcile against the gateway's idempotency key.
- **Gateway-side idempotency key**: Pass a stable, order-scoped idempotency key to `payment.charge()` so that retrying the same key returns the original charge rather than creating a new one. The gateway then becomes the source of truth for "did this charge happen," and you query it on retry rather than relying solely on your local DB record.

Neither is implemented here.

### 2. CRITICAL — Lock TTL Is Shorter Than Maximum Execution Time Under Retry

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case execution time inside the lock:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100s
```

The lock TTL is 60 seconds. A second worker can acquire the lock at t=60 while the first worker is still executing its third payment attempt at up to t=100. Both workers now hold the lock simultaneously — the entire mutual exclusion guarantee collapses. The idempotency check is then a race between two concurrent `db.get_charge()` reads, both of which return `None` before either has committed.

The fix requires either:
- `ORDER_LOCK_TTL` ≥ `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` + DB overhead margin, or
- Lock renewal (heartbeat) during long operations, or
- Reducing the payment client's cumulative timeout to fit within the TTL with margin.

### 3. HIGH — Lock Release Failure Is Silently Swallowed

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `self._release_script(...)` raises (Redis connection lost, script error, etc.), the exception propagates out of the `finally` block and suppresses any exception already in flight from the `try` block. More importantly, the log line says "released" regardless of whether the script returned `0` (token mismatch — lock was not released) or raised entirely. A return value of `0` means another worker's lock was left in place or this worker's lock was already expired and re-acquired by someone else — both are silent operational failures.

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error(
                "Lock release returned 0 — token mismatch or TTL expiry order_id=%s",
                order_id,
            )
    except Exception:
        logger.exception("Failed to release lock order_id=%s", order_id)
```

### 4. MEDIUM — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError.__str__` includes field values in its output. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are credentials. They must never appear in logs. At minimum, scrub or summarize the error before logging:

```python
logger.warning(
    "Invalid order request fields=%s",
    [e["loc"] for e in exc.errors()]   # field names only, no values
)
```

---

### 5. MEDIUM — `PaymentClient` Retries on Timeout Without Knowing Whether the Charge Succeeded

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have been received and processed by the gateway — the response was simply lost in transit. Retrying without a gateway-side idempotency key will submit a second charge attempt that the gateway treats as a new, independent charge. This is the same double-charge risk as finding 1 above, but at the HTTP layer. This is only safely retriable if the gateway endpoint is idempotent by key.

---

### 6. LOW — Test Coverage Does Not Exercise the Failure Paths That Matter Most

The tests mock the happy path and basic contention, but the following scenarios are entirely untested:

| Scenario | Risk |
|---|---|
| `payment.charge()` succeeds, DB commit raises | Unrecorded charge — the critical flaw |
| Lock TTL expires mid-payment, second worker acquires | Concurrent execution under the same lock key |
| `_release_script` returns `0` | Silent lock state corruption |
| `payment.charge()` times out on attempt 1, succeeds on attempt 2 | Potential double charge if no idempotency key |
| Redis connection drops during lock acquisition | Unclear failure mode |

Tests that only exercise mocked happy paths against the exact code paths that contain the bugs do not provide meaningful safety coverage for a payment processing service.

---

### Summary of Required Changes Before Approval

| # | Severity | Item |
|---|---|---|
| 1 | Critical | Implement gateway-side idempotency key on all charge calls; query gateway on retry rather than relying on local DB record |
| 2 | Critical | Align `ORDER_LOCK_TTL` with worst-case payment execution time, or implement lock renewal |
| 3 | High | Wrap lock release in error handling; treat return value of `0` as an observable error condition |
| 4 | Medium | Sanitize Pydantic validation errors before logging — field names only, no values |
| 5 | Medium | Document or enforce that `PaymentClient.charge()` is only retried when the gateway guarantees idempotency |
| 6 | Low | Add tests for charge-succeeds/commit-fails, TTL expiry, and release script failure paths |
