Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic SET NX EX acquisition, token-verified Lua release, Pydantic input validation, and idempotency checking are all correctly structured. However, there are several correctness and safety issues that must be resolved before this handles real payment traffic: a critical charge-outside-transaction flaw that breaks the atomicity guarantee the PR claims to provide, a missing heartbeat mechanism that makes the 60-second TTL a hard ceiling rather than a safety net, unhandled failure modes that can leave the system in inconsistent states, and a payment client that retries on timeout without idempotency protection — a direct double-charge risk.

---

## Detailed Findings

### Critical — Must Fix Before Merge

**1. `payment.charge()` is called inside `db.transaction()` but is not transactional**

```python
# services/order_processor.py, lines within process()
with self.db.transaction():
    charge_id = self.payment.charge(          # ← external HTTP call inside DB tx
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary states: *"crash between charge and record cannot produce an unrecorded charge."* This is false. The database transaction wraps both the HTTP charge call and the DB writes, but the payment gateway call is not part of the database transaction. If `payment.charge()` succeeds and then the DB transaction rolls back (network partition, DB crash, constraint violation), the charge is real and recorded nowhere. The lock releases, the idempotency check finds nothing, and the next worker charges again.

The correct pattern is charge-then-record with the external call *outside* the transaction boundary:

```python
# Charge outside the transaction — gateway call is not transactional
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically only after charge succeeds
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Charge succeeded but record failed — log with charge_id for manual reconciliation
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but record failed "
        "order_id=%s charge_id=%s", req.order_id, charge_id
    )
    raise
```

This does not eliminate the gap entirely — nothing can, because two independent systems are involved — but it makes the failure mode observable and reconcilable rather than silent and duplicative.

---

**2. No lock heartbeat — TTL expiry silently invalidates the lock while the worker continues**

```python
ORDER_LOCK_TTL = 60   # seconds
```

The lock has a 60-second TTL with no renewal mechanism. The payment client alone can consume up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** under worst-case retry conditions. The lock expires at second 60, a second worker acquires it, passes the idempotency check (nothing recorded yet), and proceeds to charge — while the first worker is still running. The Lua release script correctly prevents the second worker's lock from being deleted by the first worker, but the damage is already done.

Two options, pick one:

- **Heartbeat thread:** A background thread renews the lock TTL periodically while the critical section is executing.
- **Conservative TTL:** Set `ORDER_LOCK_TTL` to a value strictly greater than the maximum possible execution time of the critical section, and document that bound explicitly. Given the numbers above, 120–150 seconds minimum. This is simpler but means contending orders wait longer on a crashed worker.

Either way, the relationship between `ORDER_LOCK_TTL`, `REQUEST_TIMEOUT`, and `MAX_RETRIES` must be an explicit invariant with an assertion or a comment that is enforced in code, not just implied.

**3. Payment client retries on timeout without idempotency protection**

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry loop
```

A timeout means the request may or may not have been processed by the gateway. Retrying without an idempotency key sends a second charge request that the gateway cannot distinguish from the first. This is a direct double-charge path independent of the distributed lock.

The fix is to generate a per-call idempotency key (the `order_id` is the natural choice) and include it in every request to the gateway:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

Pass `req.order_id` as the idempotency key from `OrderProcessor.process()`. This requires the gateway to support idempotency keys — if it does not, retrying on timeout must be removed entirely and timeouts must be treated as permanent failures requiring manual resolution.

### Significant — Should Fix Before Merge

**4. `PaymentError` is not caught in `process()` — unhandled exception escapes the public API**

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError — not caught
```

`LockError` is caught and converted to a structured `OrderResult`. `PaymentError` is not. It propagates to the caller as an unhandled exception, bypassing the `OrderResult` contract the method signature promises. The lock is still released correctly by the `finally` block, but callers receive an exception instead of a result object. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, str(e))
    return OrderResult(processed=False, message="Payment failed")
```

---

**5. `db.get_charge()` and `db.transaction()` exceptions are not handled**

Database errors during the idempotency check or the record phase are also unhandled. A transient DB error during `get_charge()` will raise an exception that bypasses `OrderResult`, and the lock will be released, allowing a retry that may or may not find the prior charge depending on whether the first attempt partially committed. At minimum, wrap the locked section in a broad exception handler that logs and returns a structured failure result.

---

**6. Lock retry backoff is linear and very short**

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Three attempts with 0.5–1.5 second backoff is aggressive for a lock whose TTL is 60 seconds. If a worker holding the lock is processing normally, these retries will all fail and the caller gets a hard rejection. Consider either increasing the retry window or making the retry count and backoff configurable so callers can tune for their latency tolerance.

---

### Minor — Low Risk but Worth Addressing

**7. `payment_token` is logged indirectly via `ValidationError`**

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions. If `payment_token` fails validation, its value (or a truncated form) may appear in logs. Confirm that the log output does not include raw token values, or explicitly scrub sensitive fields before logging.

---

**8. Test for lock release verifies call count but not arguments**

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This confirms the Lua script was called but does not verify that the correct `lock_key` and `token` were passed. A regression that passes the wrong key would not be caught. Assert on `call_args` to verify the key and token are correctly threaded through.

---

**9. No test for TTL boundary or lock expiry behavior**

There is no test covering the scenario where the lock expires mid-execution. Given that this is the primary failure mode identified above, a test that simulates TTL expiry (mock Redis returning a miss on the token check at release time) would document the expected behavior and prevent regressions.
