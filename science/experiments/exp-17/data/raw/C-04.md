Decision: Request Changes

Summary: The core distributed-lock guarantee is broken in a way that directly enables the double-charge scenario this PR claims to prevent. There are also secondary issues in the payment client and test suite that need to be addressed before this is production-safe.

---

## Critical: The Lock Does Not Protect Against the Failure It Claims To

### 1. Payment charge happens *outside* the database transaction — the comment is wrong and the guarantee is false

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)          # ← This is a network call to an external service
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says "crash between charge and record cannot produce an unrecorded charge." That is incorrect. `self.payment.charge()` is a network call to an external payment gateway. It is not part of the database transaction. A database transaction cannot roll back a charge that has already been accepted by the payment processor.

The actual failure sequence:

1. `payment.charge()` succeeds — money is taken from the customer
2. Process crashes before `db.record_charge()` commits, or the DB write fails
3. Redis lock expires (60s TTL) or the process never releases it cleanly
4. A retry worker acquires the lock, finds no `get_charge()` record, and charges again

The fix is to move the payment call *before* the transaction, record the charge ID immediately, and treat an already-recorded charge ID as idempotent. Or use an outbox/saga pattern. The current structure gives a false sense of atomicity.

---

### 2. Lock TTL is shorter than the maximum possible operation duration

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client wall time: `3 × 30s (timeouts) + 2 × 5s (backoffs) = 100 seconds`.

The lock TTL is 60 seconds. A second worker can acquire the lock while the first worker is still inside `payment.charge()` on its second or third retry attempt. The idempotency check then passes (no DB record yet), and both workers charge.

Either `ORDER_LOCK_TTL` must exceed the maximum possible payment duration with margin, or the payment client's total budget must be capped below the TTL. Right now they are inconsistent and the payment client wins.

---

### 3. Lock release swallows all exceptions silently

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection dropped, script error), the exception is silently swallowed by the `finally` block's implicit fall-through into the caller. The log line after it also won't execute. The lock will then hold until TTL expiry. This is acceptable behavior for a lock, but the silent failure is not — it should at minimum log a warning so operators know the lock was not explicitly released.

---

### 4. Timeout on `payment.charge()` does not guarantee the charge did not go through

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retries...
```

A timeout means the client did not receive a response — it does not mean the server did not process the request. The payment gateway may have accepted the charge and the response was lost in transit. Retrying on timeout without an idempotency key sent to the gateway will submit duplicate charge attempts to the payment processor. This is a standard problem and the standard solution is to generate a per-attempt idempotency key (or a per-order idempotency key) and pass it in the request header so the gateway deduplicates on its side.

## Moderate

### 5. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation, its value may appear in the log output. Payment tokens are sensitive. The log message should either suppress field values or explicitly redact the token field before logging.

---

### 6. `customer_id` has no pattern constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` does not. If `customer_id` is used in database queries or log correlation, it should have equivalent sanitization.

---

## Test Suite Issues

### 7. Tests do not cover the TTL race condition at all

There is no test that simulates lock expiry mid-operation. The most important failure mode — lock expires while payment is in-flight — has zero coverage.

### 8. `test_lock_release_uses_lua_script` does not assert the correct arguments

```python
processor._release_script.assert_called_once()
```

This only asserts the script was called. It does not assert that the correct `lock_key` and `token` were passed. A refactor that calls the script with wrong arguments would pass this test.

### 9. No test for `PaymentError` propagation

There is no test verifying what `process()` returns when `payment.charge()` raises `PaymentError`. The current code does not catch `PaymentError` at all — it would propagate up as an unhandled exception rather than returning a structured `OrderResult`. That is likely a bug and definitely untested.
