Decision: Request Changes

Summary: The core locking architecture contains a critical atomicity flaw: a payment charge is initiated *inside* a database transaction but *outside* any rollback-safe boundary, meaning a successful charge followed by a database failure produces an unrecorded, unrecoverable charge — directly contradicting the PR's own "at-most-once" guarantee claim. Additionally, the payment client's retry loop is unsafe under timeout conditions, and the lock TTL is dangerously mismatched against the actual worst-case execution time the code can produce. These are not style concerns; they are correctness and financial integrity failures.

---

## Critical Flaws

### 1. Charge-Inside-Transaction Is Not Atomic — It's a Liability

The PR comment reads:

> *"Crash between charge and record cannot produce an unrecorded charge"*

This claim is false. Here is what actually happens:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external HTTP call, OUTSIDE DB atomicity
    self.db.record_charge(...)                # ← if this raises, DB rolls back
    self.db.mark_order_complete(...)          # ← but the charge already happened
```

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a payment gateway. If `record_charge` or `mark_order_complete` raises after `charge()` returns successfully, the database rolls back cleanly — but the customer has already been charged. The money is gone. There is no record of it. The idempotency check on the next attempt will find nothing and charge again.

The correct pattern for this class of problem is the **outbox pattern** or **two-phase commit with compensating transactions**:

1. Record *intent* to charge within the DB transaction first (status: `pending`).
2. Commit that record.
3. Perform the external charge.
4. Update the record to `completed` with the `charge_id`.

On retry, the idempotency check finds the `pending` record and can determine whether to re-attempt or reconcile with the gateway. The charge never precedes a durable record of intent.

### 2. Lock TTL Is Structurally Insufficient

```python
ORDER_LOCK_TTL = 60  # seconds
```

Trace the worst-case wall-clock time the lock must cover:

```
payment_client.py:
  REQUEST_TIMEOUT = 30   # per attempt
  MAX_RETRIES     = 3
  RETRY_BACKOFF   = 5    # between retries
```

Worst-case payment duration: `(30 × 3) + (5 × 2) = 100 seconds`

The lock TTL is 60 seconds. The payment client can run for 100 seconds. This means the lock *expires while the payment is still in flight*. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and initiates a second charge. Both workers then attempt to record. The result is a double charge — the exact failure mode this lock exists to prevent.

This must be resolved either by:
- Setting `ORDER_LOCK_TTL` to at minimum `(REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES - 1)) + safety_margin` (≥120s), or
- Implementing lock extension (heartbeat renewal) during the payment call, or
- Reducing `MAX_RETRIES × REQUEST_TIMEOUT` to fit within the TTL with margin.

The constants must be co-defined with an explicit assertion or comment that enforces their relationship. Right now they live in separate files with no coupling, and any future edit to either breaks the guarantee silently.

### 3. Retry on Timeout Violates At-Most-Once Semantics

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` means the client did not receive a response — it does *not* mean the charge did not execute. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout can produce a second charge for the same payment token.

Safe handling requires one of:
- **Idempotency keys**: Pass a stable, per-order idempotency key in the request header (e.g., `Idempotency-Key: {order_id}`). The gateway deduplicates on its side, making retries safe.
- **Query-before-retry**: After a timeout, query the gateway for the charge status by order reference before re-attempting.

Without one of these, `MAX_RETRIES = 3` on timeout means up to 3 charges may be submitted to the gateway for a single order.

## Significant Issues

### 4. `payment_token` Logged or Exposed in Error Paths

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds 256 chars), the raw token value appears in the log. Payment tokens are sensitive credentials. The log call should sanitize or explicitly exclude the `payment_token` field from the error output.

---

### 5. Lock Retry Sleep Blocks the Worker Thread

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

In any async or thread-pool worker context, `time.sleep` blocks the OS thread. Under high contention this is a latency and throughput problem. This is acceptable only in a purely synchronous, single-threaded-per-process deployment — which is not stated anywhere. If this runs under `asyncio`, `uvicorn`, `gunicorn` with async workers, or a thread pool, this must be `asyncio.sleep` or moved to a thread executor. The deployment context must be documented or the sleep must be made context-aware.

---

### 6. Test Suite Does Not Cover the Critical Failure Paths

The tests validate the happy path and lock contention, but are entirely absent for:

- **Charge succeeds, `record_charge` raises**: The most important failure mode given flaw #1. No test.
- **Lock expires mid-payment**: Not testable without the TTL fix, but no test even attempts to assert TTL adequacy.
- **Timeout retry behavior**: No test verifies that a timeout does or does not produce multiple charge attempts.
- **`PaymentError` propagation**: No test verifies that a `PaymentError` from the payment client is handled gracefully in `process()` — currently it would propagate as an unhandled exception, returning a 500 to the caller rather than a structured `OrderResult`.

Note that `PaymentError` is never caught in `process()`. If `self.payment.charge()` raises `PaymentError`, it escapes the `with self._acquire_lock()` block (the lock is still correctly released by `finally`), but the caller receives an unhandled exception rather than an `OrderResult`. This is inconsistent with the method's stated contract.

---

### 7. `customer_id` Has No Pattern Validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has no pattern constraint. If `customer_id` is used in database queries or log correlation, it is an injection surface. Apply the same or equivalent pattern.
