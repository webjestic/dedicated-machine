Decision: Request Changes

Summary: The core distributed-lock guarantee is broken in a way that directly enables the double-charge scenario this PR exists to prevent. There are also several secondary issues across the payment client and test suite that need to be resolved before this is safe to merge.

---

## Critical: The Lock Does Not Actually Protect the Payment

### 1. `payment.charge()` is called **outside** the database transaction's atomicity boundary — but **inside** a boundary that does not exist

The PR comment says:

> *"Crash between charge and record cannot produce an unrecorded charge"*

That claim is false. Trace the execution path:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call, not transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is a network call to an external payment gateway. A database transaction provides atomicity over **database operations only**. If the process crashes, the network is interrupted, or the gateway times out after successfully processing the charge but before returning the `charge_id`, you have:

- A real charge on the customer's card
- No `charge_id` in your database
- The DB transaction rolled back
- The lock released
- The next worker will pass the idempotency check (`get_charge` returns `None`) and charge again

This is the exact double-charge scenario the architecture claims to prevent. The comment in the code makes this worse because it creates false confidence for the next engineer reading it.

**Required fix:** The payment call must happen **before** the DB transaction opens. Record the `charge_id` immediately in a durable, pre-commit idempotency table (even as a "pending" state) before calling the gateway, or use an outbox pattern. At minimum, the misleading comment must be removed.

---

### 2. Timeout race: Lock TTL can expire while the payment client is still retrying

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Worst-case payment client runtime: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. A second worker can acquire the lock at T+60 while the first worker is still mid-retry on the payment call. Both workers will now be inside the `_acquire_lock` context simultaneously. The idempotency check will pass for the second worker because the first worker has not yet written to the DB. Double charge.

**Required fix:** Either reduce `MAX_RETRIES × REQUEST_TIMEOUT + RETRY_BACKOFF` to be well under `ORDER_LOCK_TTL` with margin, or implement lock extension (refresh the TTL while the lock is still held). The numbers must be documented as a constraint and enforced, not left as independent constants in two separate files.

---

### 3. Timeout ambiguity on payment retry creates an undetected duplicate charge risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A `Timeout` from `requests` can fire **after** the gateway has already processed the charge and is in the process of sending the response. The charge succeeded; you just did not receive the confirmation. The retry loop will then attempt to charge again with the same `token`. Whether this produces a duplicate charge depends entirely on whether the payment gateway is idempotent on `token` reuse — which is not documented, not tested, and not enforced here.

**Required fix:** Distinguish network-send timeouts from response-receive timeouts where the library allows it. Pass an idempotency key (separate from `token`) to the payment gateway on every call so retries are safe. Document the gateway's idempotency contract explicitly.

## Significant: Unhandled Exceptions Silently Release the Lock and Leave State Inconsistent

```python
try:
    yield token
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If `self.payment.charge()` raises `PaymentError`, or if `self.db.record_charge()` raises any exception, the `finally` block releases the lock and the exception propagates out of `_acquire_lock`. The `process()` method has no `except` clause for these cases. The exception will bubble up to the caller as an unhandled exception rather than returning a structured `OrderResult`. This breaks the stated contract of the method signature.

**Required fix:** Add explicit exception handling in `process()` for `PaymentError` and general DB exceptions, returning appropriate `OrderResult` values. Add tests for these paths.

## Moderate: `payment_token` Echoed Back in Structured Logs

```python
logger.info(
    "Order processed order_id=%s charge_id=%s",
    req.order_id, charge_id,
)
```

This specific line is fine, but `req.payment_token` is present on the validated model and could easily be added to a log line during debugging. More concretely, the `ValidationError` path logs `e` directly:

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If a near-valid `payment_token` is submitted, it may appear in the log output. Payment tokens must never appear in logs.

**Required fix:** Scrub sensitive fields from `OrderRequest` before logging, or use a custom `__repr__` that excludes `payment_token`. Confirm Pydantic's error serialization does not leak token values.

---

## Minor: Test Suite Does Not Cover the Failure Paths That Matter Most

The tests cover the happy path and lock contention, but none of the following are tested:

- Lock TTL expiry mid-payment (the critical race described above)
- `PaymentError` raised inside the lock — does `process()` return a structured result or raise?
- DB exception after a successful charge — is the charge orphaned?
- Timeout on payment followed by retry — is the idempotency key passed correctly?
- Lock release failure (Lua script returns 0 or Redis is unreachable in `finally`)

`test_lock_release_uses_lua_script` asserts the Lua script was called but does not assert it was called with the correct token, which is the entire point of the token-verification design.
