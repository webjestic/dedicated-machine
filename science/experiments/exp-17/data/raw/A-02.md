Decision: Request Changes

Summary: The distributed lock implementation contains a critical gap: there is no heartbeat or lock renewal mechanism, meaning any payment round-trip that approaches or exceeds `ORDER_LOCK_TTL` (60 seconds) will cause the lock to expire while the worker still holds it, allowing a second worker to acquire the lock, pass the idempotency check, and issue a duplicate charge — the exact failure mode this design is meant to prevent. Beyond that core flaw, the `payment.charge()` call sits inside `db.transaction()`, which means a gateway timeout can hold a database transaction open for up to 90 seconds (30s × 3 attempts), and the retry logic on timeouts makes the double-charge window worse, not better, because a timed-out charge may have already settled at the gateway before the retry fires. These issues together undermine the at-most-once guarantee the PR claims to provide.

## Detailed Findings

### 1. Lock Expiry Without Heartbeat — Critical

**File:** `services/order_processor.py`, `_acquire_lock` / `process`

`ORDER_LOCK_TTL` is 60 seconds. The payment client can spend up to 90 seconds on a single order (`REQUEST_TIMEOUT=30` × `MAX_RETRIES=3`). There is no renewal path.

```
t=0s   Worker A acquires lock, TTL=60s
t=60s  Lock expires in Redis
t=61s  Worker B acquires the same lock, passes idempotency check (charge not yet recorded)
t=75s  Worker A's payment.charge() returns, records charge
t=76s  Worker B's payment.charge() fires — duplicate charge issued
```

The idempotency check is a read against the database. It only protects if the first worker has already committed `record_charge`. If the lock expires before that commit, the check is invisible to the race.

**Required fix:** Implement a background heartbeat thread that calls `redis.expire(lock_key, ORDER_LOCK_TTL)` on a period of `ORDER_LOCK_TTL / 3` or less, and cancels on lock release. The heartbeat must be stopped and the lock considered poisoned if the renewal itself fails (Redis unreachable), triggering an abort of the in-flight transaction.

```python
import threading

@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    acquired = False

    for attempt in range(self.LOCK_RETRY_ATTEMPTS):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
        if acquired:
            break
        time.sleep(0.5 * (attempt + 1))

    if not acquired:
        raise LockError(f"Could not acquire lock for order {order_id}")

    stop_event = threading.Event()
    renewal_failed = threading.Event()

    def _heartbeat():
        interval = self.ORDER_LOCK_TTL / 3
        while not stop_event.wait(interval):
            try:
                refreshed = self._renew_script(keys=[lock_key], args=[token, self.ORDER_LOCK_TTL])
                if not refreshed:
                    # Lock was lost — another worker may now hold it
                    renewal_failed.set()
                    return
            except Exception:
                renewal_failed.set()
                return

    t = threading.Thread(target=_heartbeat, daemon=True)
    t.start()

    try:
        yield token, renewal_failed   # caller must check renewal_failed before committing
    finally:
        stop_event.set()
        t.join(timeout=2)
        self._release_script(keys=[lock_key], args=[token])
```

A companion Lua script for renewal:

```lua
-- RENEW_LOCK_SCRIPT
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

And `process()` must check `renewal_failed` before calling `db.transaction()`:

```python
with self._acquire_lock(req.order_id) as (token, renewal_failed):
    if renewal_failed.is_set():
        raise LockError("Lock expired before processing could begin")
    existing = self.db.get_charge(req.order_id)
    if existing:
        ...
    if renewal_failed.is_set():
        raise LockError("Lock expired during idempotency check")
    with self.db.transaction():
        ...
```

### 2. Payment Call Inside Database Transaction — Serious

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external I/O inside open transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

An open database transaction holds a connection and potentially row locks for the entire duration of the payment call. With `REQUEST_TIMEOUT=30` and `MAX_RETRIES=3`, that is up to 90 seconds. Under load this will exhaust your connection pool and cause cascading failures across unrelated orders.

**Required fix:** Charge outside the transaction, then open the transaction only for the database writes. Guard against the charge-succeeded / record-failed split with the idempotency check — if `record_charge` fails and the worker retries, the idempotency check will catch the existing charge on re-entry (provided you also persist a pending charge record before calling the gateway, or use a two-phase approach).

```python
with self._acquire_lock(req.order_id) as (token, renewal_failed):
    existing = self.db.get_charge(req.order_id)
    if existing:
        return OrderResult(processed=False, order_id=req.order_id,
                           message="Order already processed")

    # Charge OUTSIDE the database transaction
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )

    # Record atomically — if this fails, the lock is still held and
    # the idempotency check will not yet see the charge, so the caller
    # must handle PaymentError vs. record failure distinctly.
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id,
                              charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
```

Note that this restructuring introduces a narrow window between `charge()` returning and `record_charge()` committing. That window is the canonical reason to write a "pending charge" record *before* calling the gateway, then update it to "settled" after — but that is a design discussion for the next iteration. At minimum, remove the external call from inside the transaction.

### 3. Timeout Retry Amplifies Double-Charge Risk — Serious

**File:** `services/payment_client.py`, `charge()`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` from `requests` means the *client* gave up waiting. It does not mean the gateway did not process the charge. Retrying unconditionally after a timeout can issue the same charge twice if the first attempt settled after the timeout fired.

**Required fix:** The gateway API must support idempotency keys. Pass a stable, per-order idempotency key on every attempt so the gateway deduplicates on its side.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
        ...
```

The `order_id` is a natural idempotency key here. Pass it through from `process()`.

### 4. `PaymentError` Not Caught in `process()` — Moderate

**File:** `services/order_processor.py`, `process()`

`payment.charge()` raises `PaymentError` on permanent failure or retry exhaustion. `process()` catches only `LockError`. An unhandled `PaymentError` will propagate to the caller as an unhandled exception rather than returning a structured `OrderResult`.

```python
except LockError:
    return OrderResult(processed=False,
                       message="Order currently being processed by another worker")
# PaymentError propagates uncaught
```

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id,
                       message="Payment processing failed")
except LockError:
    return OrderResult(processed=False,
                       message="Order currently being processed by another worker")
```

### 5. Lock Retry Backoff Is Linear and Blocks the Thread — Minor

**File:** `services/order_processor.py`, `_acquire_lock()`

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — linear
time.sleep(backoff)
```

Three attempts with linear backoff gives a maximum wait of 3 seconds before raising `LockError`. Under high contention this is reasonable, but `time.sleep` in a synchronous worker blocks the thread entirely. If this runs in an async context (FastAPI, etc.) it will block the event loop. Use `asyncio.sleep` if the service is async, or document the synchronous-only constraint explicitly.
