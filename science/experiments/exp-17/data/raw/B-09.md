Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several correctness and reliability flaws that would make this unsafe to deploy in a production payment environment: the most critical is that the payment call sits *inside* a database transaction, creating a window where a gateway timeout on retry could produce an unrecorded charge that the idempotency check will never catch. Additional issues around lock TTL, retry semantics, and missing exception handling compound the risk.

---

## Critical Issues

### 1. Payment Charge Inside Database Transaction — Unrecorded Charge on Retry

```python
# services/order_processor.py, lines inside process()
with self.db.transaction():
    charge_id = self.payment.charge(...)        # ← network call inside DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most dangerous flaw in the PR. The comment claims *"crash between charge and record cannot produce an unrecorded charge"* — that claim is incorrect.

**The failure scenario:**

1. Worker A acquires the lock.
2. `self.payment.charge()` is called. The gateway receives the request and processes the charge successfully, but the response is lost due to a network partition or timeout.
3. `PaymentClient.charge()` retries (up to `MAX_RETRIES=3`). Each retry hits the gateway again. If the gateway is not idempotent on the caller's side, the customer is charged multiple times. If it is idempotent but the timeout persists, `PaymentError` is raised.
4. `PaymentError` propagates out of the `with self.db.transaction()` block. The transaction rolls back. `record_charge` is never committed.
5. The lock is released. Worker B acquires the lock. `db.get_charge()` returns `None` — the idempotency check passes. Worker B charges again.

**The database transaction provides zero protection here.** A network call to an external service cannot participate in a local ACID transaction. The rollback only undoes the DB writes; it cannot un-charge the customer.

**Required fix:** Separate the charge from the DB write. Record a *pending* charge row before calling the gateway, then update it to *completed* after. On startup or lock acquisition, check for pending rows and reconcile against the gateway before proceeding.

```python
# Correct pattern (sketch):
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also be updated to treat a *pending* row as "already attempted" and trigger reconciliation rather than silently skipping.

### 2. Lock TTL Does Not Account for Retry Duration

```python
ORDER_LOCK_TTL = 60   # seconds
# PaymentClient:
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
REQUEST_TIMEOUT = 30  # seconds per attempt
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. The lock will expire while a worker is still holding it and actively processing. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. The token-verified Lua release does not help here — the TTL expiry is the failure mode, not a competing release.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed worst-case processing time with margin (e.g., 180s), or implement lock heartbeat renewal during the payment call. The TTL must be a deliberate architectural decision documented with the formula above.

## Significant Issues

### 3. `PaymentError` Is Not Caught in `process()`

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        with self.db.transaction():
            charge_id = self.payment.charge(...)   # can raise PaymentError
            ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` is not caught. It will propagate to the caller as an unhandled exception. The caller receives no `OrderResult`, no structured error, and the lock is released (correctly, via `finally`) — but the caller has no way to distinguish a payment failure from a programming error without catching the raw exception.

**Required fix:** Add an explicit `except PaymentError` handler that returns a structured `OrderResult(processed=False, message="Payment failed: ...")`.

### 4. Retry on Timeout Retries Without Idempotency Key

```python
# payment_client.py
for attempt in range(self.MAX_RETRIES):
    response = requests.post(
        self.CHARGE_ENDPOINT,
        json={"token": token, "amount": amount_cents},
        ...
    )
```

On a timeout, the gateway may have already processed the charge. Retrying the same request without an idempotency key (e.g., a stable `order_id`-derived key passed in the request body or as a header) can result in duplicate charges at the gateway level.

**Required fix:** Pass a stable idempotency key derived from `order_id` to the payment gateway on every attempt:

```python
json={"token": token, "amount": amount_cents, "idempotency_key": order_id}
# or via header: headers={"Idempotency-Key": order_id}
```

This requires `order_id` to be threaded into `PaymentClient.charge()`.

## Minor Issues

### 5. Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep` blocks the worker thread. In an async or thread-pool context this is a latency hazard. The backoff is also linear — under high contention, exponential backoff with jitter is more effective at reducing thundering herd. Not a blocking issue for this PR, but should be noted.

### 6. Test for Lua Script Release Is Shallow

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test verifies that the script was called, but not that it was called with the correct `keys` and `args` (i.e., the right lock key and the token that was actually set). A regression that passes the wrong token would not be caught.

**Suggested improvement:**

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["keys"] == ["order:lock:order-001"]
# token is internal, but at minimum verify the key
```

### 7. No Test for Lock TTL Expiry / Duplicate Charge Scenario

There is no test that exercises the race condition where the lock expires mid-processing. Given that this is the primary safety guarantee of the service, a test simulating TTL expiry (mock `time.time` or inject a clock) should be present.
