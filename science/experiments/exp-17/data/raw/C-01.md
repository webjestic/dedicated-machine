Decision: Request Changes

Summary: The PR's core safety claim — that the distributed lock prevents double-charging — does not hold under the exact failure mode it was designed to prevent. There are also secondary issues in the payment client retry logic and test coverage that need to be addressed before this touches production payment flows.

---

## Critical: The Lock Does Not Actually Prevent Double-Charging

The comment in the code reads:

> "Crash between charge and record cannot produce an unrecorded charge"

That claim is false, and the reasoning behind it is the most important thing to trace carefully here.

### The Fatal Race: Payment Retry + Lock TTL Expiry

Walk through this execution path:

1. Worker A acquires the lock on `order-001`.
2. Worker A calls `self.payment.charge(...)`.
3. The payment gateway is slow. The request takes longer than `ORDER_LOCK_TTL` (60 seconds).
4. The Redis lock expires mid-flight. The key is gone.
5. Worker B calls `process("order-001", ...)`, acquires the lock (it's now free), passes the idempotency check (nothing recorded yet), and calls `self.payment.charge(...)` — issuing a second charge.
6. Worker A's request eventually returns with a `charge_id`. Worker A proceeds to record it.
7. Worker B's request also returns. Worker B records a second charge.

The customer has now been charged twice. The lock TTL of 60 seconds sounds generous, but `PaymentClient` has `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds and `REQUEST_TIMEOUT = 30` seconds per attempt. Worst-case total time inside `payment.charge()` is `3 × 30 + 2 × 5 = 100 seconds` — well beyond the 60-second TTL. This is not a theoretical edge case; it is a reachable path under normal transient gateway degradation.

The `db.transaction()` block does not help here because the payment call happens **inside** the transaction but **before** any database record exists. The database transaction provides atomicity between `record_charge` and `mark_order_complete`, but it does not prevent a second worker from charging before either of those records exist.

### What Needs to Change

The standard pattern for this class of problem is to **record intent before charging**, not after:

1. Inside the lock, write a `PENDING` charge record to the database (this is the idempotency anchor).
2. Release the lock or let it expire — it no longer matters, because the next worker will find the `PENDING` record and skip.
3. Call the payment gateway.
4. Update the record to `COMPLETED` with the `charge_id`.

The idempotency check must also handle the `PENDING` state explicitly — a second worker finding a `PENDING` record should either wait or return "already being processed," not proceed to charge.

Alternatively, if the intent-record approach is not feasible, the lock TTL must be set to a value that is provably greater than the maximum possible time spent inside the lock. Given the current payment client, that minimum safe TTL is `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF + margin`, which is over 100 seconds — and that number needs to be enforced as a derived constant, not two independent magic numbers that can drift apart independently.

## Serious: Payment Retry Logic Retries on Timeout Without Idempotency

In `PaymentClient.charge()`, a `requests.exceptions.Timeout` causes a retry. But a timeout does not mean the charge did not go through — the gateway may have processed the charge and the response was lost in transit. Retrying on timeout without an idempotency key sent to the gateway can produce duplicate charges at the gateway level, independent of the Redis lock issue above.

The fix is to generate an idempotency key (e.g., a UUID derived from `order_id`) before the retry loop and send it as a header (e.g., `Idempotency-Key`) on every attempt. Most payment gateways support this. If the gateway does not support idempotency keys, timeout must not be retried — it must be treated as an unknown outcome requiring manual reconciliation.

## Moderate: `payment_token` Logged or Exposed in Error Paths

`payment_token` flows through `OrderRequest` and is passed directly to `payment.charge(token=req.payment_token, ...)`. The Pydantic model does not mark this field as a secret, and nothing in the logging configuration prevents it from appearing in serialized validation error output or debug log lines. In a government contracting context this is a compliance concern. The field should use Pydantic's `SecretStr` type, and the payment client should never log the token value.

## Moderate: Lock Release Swallows Exceptions Silently

In `_acquire_lock`'s `finally` block:

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception propagates out of the `finally` block and masks whatever exception was already being handled inside the `with` block. The caller sees a Redis error instead of, for example, a `PaymentError`. Wrap the release in a try/except that logs the failure and continues — a failed release is bad (the lock holds until TTL expiry) but it should not corrupt the exception chain.

---

## Minor: Test Coverage Gaps

The tests do not cover:

- **Lock expiry during payment**: No test simulates the scenario where the lock TTL expires before `payment.charge()` returns. This is the exact failure mode described above and should be a test.
- **Timeout retry in PaymentClient**: No test verifies that a timeout followed by a success on retry returns the correct `charge_id`, or that a timeout on the final attempt raises `PaymentError`.
- **`PENDING` state idempotency** (once that pattern is implemented): A second worker arriving while the first is mid-charge needs explicit test coverage.
- **Lock release failure**: No test covers what happens when the Lua script call raises during `finally`.

The existing tests are structurally sound and the mock setup is correct, but they only cover the happy path and the most obvious contention case.

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| Critical | Lock TTL can expire during payment retry window, enabling double-charge |
| Critical | No intent-record before charge; idempotency anchor is missing |
| Serious | Timeout retries without gateway idempotency key |
| Moderate | `payment_token` not protected as `SecretStr` |
| Moderate | Lock release exception in `finally` masks upstream exceptions |
| Minor | Test coverage missing for the critical failure paths |

The architecture is on the right track — the Lua script for atomic release is correct, the `SET NX EX` acquisition is correct, and the Pydantic validation is solid. The fundamental problem is that the lock's TTL and the payment client's worst-case execution time are not coupled, and the idempotency anchor is placed after the side effect it is supposed to prevent. Fix those two things and the design becomes sound.
