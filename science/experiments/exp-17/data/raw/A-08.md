Decision: Request Changes

Summary: The locking design contains a critical flaw that breaks the at-most-once payment guarantee it claims to provide: the lock TTL can expire while a slow payment gateway call is in-flight, allowing a second worker to acquire the lock, pass the idempotency check, and issue a duplicate charge before the first worker records its charge — the exact race condition this architecture was built to prevent. Beyond that core failure, the payment client's retry-on-timeout behavior compounds the problem by potentially issuing multiple charges against the gateway while the caller has no way to distinguish a successful charge from a lost response. Several secondary issues in error handling, observability, and test coverage also need to be addressed before this is production-safe.

---

## Detailed Findings

### 1. CRITICAL — Lock TTL Expiry During Payment Gateway Call (Double-Charge Risk)

This is the central architectural flaw. Trace the execution path:

```
Worker A acquires lock (t=0, TTL=60s)
Worker A calls self.payment.charge() (t=1s)
  → gateway is slow, or Worker A is GC-paused, or network is degraded
  → payment.charge() takes >60s (PaymentClient.REQUEST_TIMEOUT=30s × MAX_RETRIES=3
     plus RETRY_BACKOFF=5s × 2 = up to 100s of wall time)
Lock expires at t=60s — key is deleted from Redis
Worker B acquires lock (t=61s)
Worker B calls db.get_charge() → returns None (Worker A hasn't recorded yet)
Worker B calls self.payment.charge() → second charge issued
Worker A's charge() eventually returns charge_id
Worker A calls db.record_charge() — one charge is recorded
Worker B calls db.record_charge() — second charge is recorded
Customer is billed twice.
```

The maximum wall time inside the lock is `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES - 1)` = `30×3 + 5×2` = **100 seconds**, which is 67% over the 60-second TTL. This is not a theoretical edge case — it is a near-certainty under any gateway degradation event.

**Required fix:** Implement lock heartbeat renewal. A background thread or async task must periodically extend the lock TTL while the critical section is executing. The lock must not be allowed to expire while the holder is still alive and working.

```python
import threading

@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    acquired = False

    for attempt in range(self.LOCK_RETRY_ATTEMPTS):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
        if acquired:
            break
        time.sleep(0.5 * (attempt + 1))

    if not acquired:
        raise LockError(f"Could not acquire lock for order {order_id}")

    stop_event = threading.Event()

    def _renew():
        # Renew at half the TTL interval so we never get close to expiry
        renewal_interval = self.ORDER_LOCK_TTL / 2
        while not stop_event.wait(timeout=renewal_interval):
            renewed = self.redis.expire(lock_key, self.ORDER_LOCK_TTL)
            if not renewed:
                # Key is gone — lock was lost; nothing we can do but log
                logger.error(
                    "Lock renewal failed — key missing order_id=%s token=%s",
                    order_id, token,
                )
                return

    renewer = threading.Thread(target=_renew, daemon=True)
    renewer.start()

    try:
        yield token
    finally:
        stop_event.set()
        renewer.join(timeout=2)
        self._release_script(keys=[lock_key], args=[token])
```

Note: even with heartbeat renewal, if the worker process dies mid-charge, the lock will eventually expire and a second worker can proceed. That is the correct behavior for a dead worker. The heartbeat only keeps a *live* worker's lock alive.

### 2. CRITICAL — Payment Retry on Timeout Creates Undetectable Duplicate Charges

In `payment_client.py`:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry loop
```

A `Timeout` means the request may have reached the gateway and the charge may have succeeded — the response was simply not received in time. Retrying unconditionally issues a second charge attempt with no idempotency key. If the gateway processed the first request, the customer is charged twice at the gateway level, and neither charge will be caught by the database idempotency check because neither has been recorded yet.

**Required fix:** All charge requests must include a stable idempotency key derived from `order_id`. The gateway must deduplicate on that key. The client must pass it on every attempt for the same order:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `order_id` is the natural idempotency key and must flow through from `OrderProcessor.process()` to `PaymentClient.charge()`.

### 3. HIGH — Unhandled Exceptions Inside the Lock Leave It Held Until TTL Expiry

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)  # can raise
    with self.db.transaction():
        charge_id = self.payment.charge(...)      # PaymentError propagates out
        self.db.record_charge(...)                # can raise
```

If `db.get_charge()`, `db.record_charge()`, `db.mark_order_complete()`, or `self.payment.charge()` raises an unexpected exception, it propagates through the `with` block. The `finally` clause in `_acquire_lock` does execute, so the lock *is* released — that part is correct. However, `process()` has no outer `except` for these cases, meaning the exception propagates to the caller with no `OrderResult` returned, violating the method's declared return type and leaving the caller with an unhandled exception rather than a structured failure response.

**Required fix:**

```python
except (PaymentError, DatabaseError) as e:
    logger.error("Order processing failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Processing failed — please retry")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 4. HIGH — Charge Issued Outside the Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external side effect inside tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment charge is an external, non-transactional side effect. Placing it inside `db.transaction()` does not make it atomic with the database writes — it only means the database transaction is held open for the entire duration of the network call to the payment gateway. This has two consequences:

1. **Database connection exhaustion:** Holding a DB transaction open for up to 100 seconds (gateway timeout × retries) under concurrent load will exhaust the connection pool.
2. **False atomicity:** If `record_charge` fails after `payment.charge` succeeds, the DB transaction rolls back but the charge has already been issued. The lock will eventually be released, the idempotency check will find no record, and a second worker can charge again.

**Required fix:** Issue the charge *before* opening the database transaction. Record the result inside the transaction. Accept that a crash between charge and record requires a reconciliation process (check gateway for charges with your idempotency key on startup), not a DB transaction wrapper.

```python
# Outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,
)

# Now record the already-completed charge atomically
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 5. MEDIUM — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions and configurations. If `payment_token` fails validation (e.g., too long), its value may appear in the log output. Payment tokens must never appear in logs.

**Required fix:** Log only the field names and error types, not the values:

```python
except ValidationError as e:
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

### 6. MEDIUM — Lock Key Namespace Collision Risk

```python
lock_key = f"order:lock:{order_id}"
```

`order_id` is validated to `[a-zA-Z0-9_\-]+` which is safe, but the namespace `order:lock:` is not configurable and will collide if this Redis instance is shared with other services or if a second lock type is added for orders. The namespace should be injected at construction time or drawn from a constants module so it is not silently duplicated.
