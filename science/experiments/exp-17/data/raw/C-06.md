# Pull Request Review: Distributed Order Processor with Redis-Backed Exclusive Locking

I traced every execution path through this code before writing a single word of this review. There are real problems here that block approval.

---

## Critical Flaw 1: Charge Occurs Inside a Database Transaction — Atomicity Guarantee Is False

The PR summary claims:

> "Crash between charge and record cannot produce an unrecorded charge"

That claim is **wrong**, and the architecture inverts the correct operation order.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)        # External network call INSIDE transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Path A — The unrecorded charge:**
1. `payment.charge()` succeeds. Money leaves the customer's account. `charge_id` is returned.
2. `self.db.record_charge()` raises an exception (deadlock, constraint violation, connection drop).
3. The database transaction rolls back.
4. The charge is real. The record does not exist. The lock is released.
5. The idempotency check on retry finds nothing. The customer is charged again.

The database transaction provides atomicity only over database operations. It provides **zero rollback capability** over the external payment call. Placing the payment call inside the transaction creates a false sense of safety while doing nothing to address the actual failure mode.

**The correct pattern** is:
1. Write a `PENDING` record to the database first, within a transaction, before touching the payment gateway.
2. Call the payment gateway outside the database transaction.
3. Update the record to `COMPLETE` with the `charge_id`.
4. The idempotency check must detect `PENDING` records and handle them (either surface them as in-progress or resume them), not only `COMPLETE` ones.

This is the standard outbox/saga pattern for exactly-once external side effects. The current code does not implement it.

## Critical Flaw 2: Payment Client Retries on Timeout — Guaranteed Double-Charge Path

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# ... falls through to retry
```

A `Timeout` exception means the client did not receive a response within 30 seconds. It does **not** mean the charge did not execute. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout will submit the charge a second (and third) time against the same `payment_token` with no idempotency key.

The fix requires two things working together:
1. Generate a stable, per-order idempotency key (e.g., derived from `order_id`) and send it as a header on every attempt to the payment gateway.
2. On timeout, the retry must re-query or rely on the gateway's idempotency enforcement to deduplicate — not assume the first attempt failed.

Without an idempotency key on the outbound request, retrying on timeout is a double-charge waiting to happen.

## Critical Flaw 3: Lock TTL vs. Payment Retry Window — The Lock Expires Mid-Processing

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client wall time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100 seconds
```

The lock TTL is 60 seconds. The payment client can run for 100 seconds. At second 61, the lock expires in Redis. Another worker acquires it, passes the idempotency check (charge not yet recorded — still in flight), and begins charging. Two charges are now in flight simultaneously against the same order.

The lock TTL must be strictly greater than the maximum possible time spent inside the lock. Either:
- Raise `ORDER_LOCK_TTL` to at minimum 120 seconds with margin, or
- Implement lock extension (re-`SET NX EX` or a Lua-based refresh) before expiry, or
- Reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` so the math closes.

## Significant Flaw 4: `payment_token` Logged at `INFO` Level

```python
logger.info(
    "Order processed order_id=%s charge_id=%s",
    req.order_id, charge_id,
)
```

This line is fine. But `payment_token` passes through `OrderRequest` and is present in the object throughout processing. Depending on how the logging framework is configured, structured loggers that serialize entire request objects will emit the token. More importantly, `logger.warning("Invalid order request: %s", e)` on `ValidationError` will emit Pydantic's error output, which in some versions includes field values. Payment tokens are PCI-DSS regulated data. They must never appear in logs. The token field should be masked or excluded from any serialization path.

## Minor Issues

**`customer_id` has no pattern constraint.** `order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary Unicode up to 64 characters. If it is interpolated into queries anywhere downstream, this is a vector. Apply the same pattern constraint.

**`db.get_charge()` return value is not typed or validated.** The idempotency check does `if existing:` on whatever the database layer returns. A falsy-but-non-None return (empty dict `{}`, `0`, empty string) would bypass the check silently. The check should be explicit: `if existing is not None:`.

**Test for lock release does not verify the correct token is passed.** `test_lock_release_uses_lua_script` asserts `_release_script.assert_called_once()` but does not assert that the `args` parameter contains the token that was set. A bug that passes a hardcoded or empty token would pass this test.

**`PaymentError` is not caught in `process()`.** If the payment gateway returns a permanent 4xx or exhausts retries, `PaymentError` propagates uncaught out of `process()`. The caller receives an exception rather than an `OrderResult`. The method signature promises `OrderResult` in all cases.

---

## Decision: Request Changes

**Summary:** This PR contains three critical correctness flaws that each independently produce duplicate charges under realistic failure conditions: the payment call inside the database transaction provides no atomicity guarantee over the external side effect; the payment client retries unconditionally on timeout without an idempotency key, guaranteeing double-charges on slow gateways; and the lock TTL is shorter than the maximum payment retry window, allowing a second worker to acquire the lock and charge while the first is still retrying. None of these are edge cases — they are the primary failure modes this code claims to prevent. The PR must not be merged until the outbox/pending-record pattern is implemented, an idempotency key is threaded through payment requests, and the TTL math is corrected.
