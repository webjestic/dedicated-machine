Decision: Request Changes

Summary: The architecture is sound and the security fundamentals are solid — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issue: Payment Call Inside `db.transaction()` Breaks the Guarantee

This is the core problem. The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is **false** as written.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external network call INSIDE db tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**What actually happens on a crash or timeout:**

1. The DB transaction opens.
2. `payment.charge()` succeeds — money leaves the customer's account.
3. The process crashes, the DB connection drops, or the gateway response is lost in transit.
4. The DB transaction is rolled back automatically.
5. `record_charge` was never committed.
6. On retry, `db.get_charge()` returns `None` — the idempotency check passes.
7. The customer is charged a second time.

The payment call **must happen before the transaction opens**, and the charge ID must be persisted in the first write inside the transaction. The correct pattern is:

```python
# 1. Charge externally — outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Persist atomically — if this crashes, charge_id is known and recoverable
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This does not fully eliminate the crash window (nothing can without two-phase commit or an outbox pattern), but it correctly narrows it: a crash after charge but before `record_charge` commits leaves a charge with no DB record, which is a reconciliation problem — not a double-charge problem. That is the correct trade-off for this architecture, and it is recoverable via the payment gateway's own charge history.

## Secondary Issues

### 1. `payment_token` Logged Indirectly via `charge_id` — Verify Gateway Response Handling

`payment_token` is passed directly to the external gateway. Confirm that `PaymentClient` never logs the token value. Currently it does not, but this should be called out explicitly in code review since `payment_token` in the wrong log line would be a PCI-DSS violation.

### 2. `PaymentClient` Has No Authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, mTLS configuration, or API key being sent. Even for an internal endpoint, this needs to be addressed. If authentication is handled at the infrastructure layer (e.g., service mesh), that assumption must be documented explicitly.

### 3. Lock TTL Is Insufficient Under Retry Logic

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire before the payment client gives up, allowing a second worker to acquire the lock and attempt a duplicate charge while the first worker is still mid-retry. Either increase the TTL to at least 120 seconds with margin, or cap the payment client's total runtime to well under 60 seconds.

### 4. `LockError` Is Not Caught in the Retry Loop

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If `self.redis.set(...)` raises a `redis.RedisError` (connection failure, timeout), it propagates uncaught through `process()` and becomes an unhandled exception at the call site. Redis connectivity errors should be caught and either wrapped in `LockError` or surfaced as a distinct error type so callers can handle them appropriately.

### 5. Test Coverage Gap — No Test for Payment Success Followed by DB Commit Failure

There is no test verifying behavior when `payment.charge()` succeeds but `db.record_charge()` raises. This is exactly the crash window that matters for the at-most-once guarantee and should be explicitly tested to document the expected behavior (and to catch regressions if the transaction boundary moves again).

### 6. Minor: `OrderResult.order_id` Is `None` on Lock Contention

```python
return OrderResult(
    processed=False,
    message="Order currently being processed by another worker",
)
```

`order_id` is not populated here, which makes it harder for callers to correlate the response back to the original request. This is minor but worth fixing for consistency with the other failure paths.

## What Is Done Well

- Lua script for atomic lock release is correct and properly registered.
- `SET NX EX` is the right primitive — no TOCTOU on lock acquisition.
- Pydantic validation with pattern enforcement on `order_id` is solid.
- `payment_token` is not echoed back in any response field.
- Retry backoff in the payment client correctly skips the sleep on the last attempt.
- Test for path traversal injection on `order_id` is a good inclusion.

Fix the atomicity issue and the TTL mismatch before merge. The rest can be addressed in follow-up if tracked.
