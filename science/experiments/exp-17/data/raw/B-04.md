Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several fault-tolerance and correctness flaws that would make this unsafe to deploy in a payment-critical environment: most critically, the lock has no heartbeat mechanism, the payment client retries on timeout without idempotency guarantees, and the charge occurs inside a database transaction boundary in a way that creates an unrecoverable inconsistency window. These issues are detailed below.

---

## Critical Issues

### 1. No Lock Heartbeat — TTL Expiry During Long Payment Round-Trips

**File:** `services/order_processor.py`, `OrderProcessor._acquire_lock`

The lock TTL is hardcoded at 60 seconds. The payment client has a 30-second per-attempt timeout and up to 3 retries with 5-second backoff between each, yielding a worst-case payment duration of:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

This exceeds the 60-second lock TTL. If the payment gateway is slow, the lock expires while the worker is still inside the critical section. A second worker then acquires the lock, passes the idempotency check (charge not yet recorded), and issues a duplicate charge. This is the exact double-charge scenario the lock is meant to prevent.

**Required fix:** Implement a background heartbeat thread that renews the lock TTL periodically while the critical section is executing, and tear it down atomically on exit. Alternatively, reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` such that the worst-case payment duration is provably bounded well below `ORDER_LOCK_TTL`, and enforce this with a hard timeout on the entire critical section.

```python
# Sketch — heartbeat renewal
import threading

@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    # ... acquisition logic ...

    stop_event = threading.Event()

    def _heartbeat():
        while not stop_event.wait(timeout=self.ORDER_LOCK_TTL / 3):
            # Only renew if this worker still holds the lock
            self._renew_script(keys=[lock_key], args=[token, self.ORDER_LOCK_TTL])

    heartbeat_thread = threading.Thread(target=_heartbeat, daemon=True)
    heartbeat_thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join()
        self._release_script(keys=[lock_key], args=[token])
```

The renewal script itself must be token-verified (only extend if the key still holds this worker's token) to avoid renewing a lock that has already expired and been re-acquired by another worker.

### 2. Payment Retry on Timeout Without Idempotency Key — Guaranteed Duplicate Charges

**File:** `services/payment_client.py`, `PaymentClient.charge`

On a `requests.exceptions.Timeout`, the client retries the charge unconditionally. A network timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without an idempotency key will issue a second charge against the same payment method.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to retry — charge may have already succeeded
```

**Required fix:** Generate a stable idempotency key per logical charge attempt (e.g., derived from `order_id`) and pass it as a request header on every attempt. The gateway deduplicates on this key, making retries safe.

```python
def charge(self, token: str, amount_cents: int, order_id: str) -> str:
    idempotency_key = f"charge-{order_id}"  # stable across retries
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 3. External I/O (Payment Charge) Inside Database Transaction Boundary

**File:** `services/order_processor.py`, `OrderProcessor.process`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Wrapping an external HTTP call inside a database transaction holds an open DB connection and transaction slot for the entire duration of the payment round-trip (up to 100 seconds worst-case as calculated above). This will exhaust connection pools under any meaningful concurrency and cause cascading failures across the service. Additionally, if the payment succeeds but `record_charge` raises, the transaction rolls back — but the charge has already been captured externally and cannot be rolled back. The atomicity guarantee claimed in the PR description does not hold.

**Required fix:** Perform the payment charge outside the transaction, then open a short-lived transaction only for the database writes.

```python
# Charge outside transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Short, bounded DB transaction for record-keeping only
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

To handle the crash-between-charge-and-record window that this opens, the idempotency key approach in issue #2 is essential: a recovery process can safely re-query the gateway using the idempotency key to determine whether a charge succeeded before retrying.

## Moderate Issues

### 4. `PaymentError` Not Caught in `process()` — Unhandled Exception Escapes

**File:** `services/order_processor.py`, `OrderProcessor.process`

`LockError` is caught and converted to a structured `OrderResult`, but `PaymentError` (and any other exception from `self.payment.charge` or the DB calls) propagates uncaught to the caller. This is an inconsistent error contract — callers cannot rely on always receiving an `OrderResult`.

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
# PaymentError, db errors, network errors — all propagate as raw exceptions
```

**Required fix:** Add explicit handling for `PaymentError` and a catch-all for unexpected exceptions, both returning structured `OrderResult` responses with appropriate logging.

---

### 5. Lock Retry Backoff Is Linear and Holds the Thread

**File:** `services/order_processor.py`, `OrderProcessor._acquire_lock`

The retry uses `time.sleep` with a linear backoff (0.5s, 1.0s, 1.5s). In an async or thread-pool worker environment, sleeping the thread degrades throughput. The total sleep time before giving up is 3 seconds, which may be too short for legitimate contention on slow payment paths and too long for a synchronous thread to block.

**Recommendation:** Use exponential backoff with jitter to reduce thundering herd behavior, and document the expected concurrency model (sync threads vs. async workers) so the sleep strategy can be evaluated in context.

---

### 6. Test Coverage Gaps

**File:** `tests/test_order_processor.py`

The following scenarios are not covered and represent real failure modes:

- **Lock TTL expiry during processing** — no test verifies behavior when the lock expires mid-execution.
- **`PaymentError` raised by `payment.charge`** — no test verifies the error surface to the caller.
- **DB transaction rollback after successful charge** — the most dangerous inconsistency window has no test.
- **Heartbeat thread lifecycle** — once implemented, the heartbeat must be verified to stop cleanly on both success and exception paths.
- **`test_lock_release_uses_lua_script`** — this test asserts `_release_script` was called once but does not verify the correct `keys` and `args` were passed, so it would not catch a regression where the wrong token is used.

## Minor Issues

### 7. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage in Error Paths

**File:** `services/payment_client.py`

The `PaymentError` message includes the HTTP status code, which is safe. However, confirm that `response.json()` is never logged on error paths — payment gateway error bodies sometimes echo back partial token data.

### 8. `ORDER_LOCK_TTL` and `REQUEST_TIMEOUT` Are Not Co-Validated

The relationship between lock TTL and maximum payment duration is a correctness invariant, but it is enforced nowhere in the code. A future developer increasing `MAX_RETRIES` or `REQUEST_TIMEOUT` would silently break the TTL bound. Consider adding an assertion or a class-level validator that enforces `ORDER_LOCK_TTL > MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF` with margin.
