Decision: Request Changes

Summary: The locking design contains a critical flaw — the payment call is made *inside* the database transaction but *outside* any lock-extension mechanism, meaning the 60-second TTL can expire mid-charge and allow a second worker to acquire the lock and pass the idempotency check before the first worker's `record_charge` commits. Additionally, a timeout on the payment gateway call creates an unresolvable ambiguity: the charge may have succeeded at the gateway while the caller receives an exception, and the current retry logic will re-attempt that charge without any idempotency key, risking a double charge. A third concern is that unhandled exceptions from the payment client or database layer propagate out of the `with self._acquire_lock(...)` block, which does correctly release the lock via `finally`, but the caller receives an unstructured exception rather than an `OrderResult`, breaking the contract the method signature promises.

---

## Detailed Findings

### Critical — Lock TTL Expiry During Payment (Double-Charge Risk)

This is the most serious issue in the PR.

```python
# ORDER_LOCK_TTL = 60 seconds
# PaymentClient.REQUEST_TIMEOUT = 30 seconds, MAX_RETRIES = 3, RETRY_BACKOFF = 5 seconds
# Worst-case payment duration: (30 + 5) * 3 = 105 seconds

with self._acquire_lock(req.order_id):        # TTL starts: T+0
    existing = self.db.get_charge(req.order_id)  # passes at T+0
    with self.db.transaction():
        charge_id = self.payment.charge(...)     # can block up to 105 seconds
        self.db.record_charge(...)               # may not commit until T+105
```

At `T+60` the Redis key expires. Worker B acquires the lock, calls `db.get_charge`, finds nothing (Worker A's transaction has not committed), and proceeds to charge. Worker A's transaction then commits. The result is two charges on the same order. The lock TTL and the maximum payment client execution time are not coordinated and the design provides no heartbeat renewal.

**Required fix:** Either (a) implement lock heartbeat renewal — a background thread or async task that calls `EXPIRE order:lock:<id> 60` at a regular interval while the lock is held, stopping when the context manager exits; or (b) enforce that `ORDER_LOCK_TTL` is strictly greater than the maximum possible payment duration and document that constraint explicitly with a startup assertion. Option (a) is strongly preferred for production use.

### Critical — Payment Timeout Ambiguity Enables Double Charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request reached the gateway and the response was lost in transit — the charge may already have been applied. Retrying without an idempotency key sends a second, independent charge request. This is a well-known failure mode in payment systems.

**Required fix:** Generate a stable, per-order idempotency key (e.g., `sha256(order_id + customer_id)` or a UUID stored before the first attempt) and pass it as a header on every attempt to the same logical charge operation. The gateway must support idempotency keys — confirm this with the internal payments team. Until this is confirmed and implemented, retrying on timeout must be removed.

---

### High — Unhandled Exceptions Break `OrderResult` Contract

```python
def process(...) -> OrderResult:
    ...
    with self._acquire_lock(req.order_id):
        with self.db.transaction():
            charge_id = self.payment.charge(...)   # raises PaymentError
            self.db.record_charge(...)             # raises arbitrary DB exceptions
```

`PaymentError`, database exceptions, and `requests` exceptions are not caught. They propagate to the caller as raw exceptions despite the return type annotation of `OrderResult`. Callers that follow the type contract will not handle these and will produce unstructured 500 errors.

**Required fix:** Wrap the lock body in a `try/except` that catches `PaymentError` and database exceptions, logs them with appropriate severity, and returns a structured `OrderResult(processed=False, message=...)`. Add explicit exception types to the `OrderResult` model if the caller needs to distinguish failure modes.

---

### High — `db.get_charge` Idempotency Check Is Not Atomic With Lock Acquisition

The idempotency check reads from the database after acquiring the lock, which is correct. However, there is no guarantee that `db.get_charge` and the subsequent `db.transaction()` block are isolated at the database level. If the database uses `READ COMMITTED` isolation, a concurrent transaction that committed between `get_charge` and `record_charge` would be invisible to the check. This is partially mitigated by the Redis lock, but only as long as the lock TTL issue above is resolved. Document this dependency explicitly.

---

### Medium — Lock Retry Backoff Is Linear and Blocks the Worker Thread

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s — linear, not exponential
time.sleep(backoff)
```

Linear backoff with only three attempts means a worker under heavy contention gives up after 3 seconds total. More importantly, `time.sleep` blocks the worker thread entirely. If this runs in a threaded web server (gunicorn, uvicorn with sync workers), this degrades throughput under contention.

**Recommended fix:** Use exponential backoff with jitter: `backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), max_backoff)`. If the service is async, replace `time.sleep` with `asyncio.sleep`.

---

### Medium — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some versions and configurations. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials.

**Required fix:** Either sanitize the validation error before logging (exclude the `payment_token` field from the error representation) or log only the field names that failed, not their values.

---

### Low — `RELEASE_LOCK_SCRIPT` Registered Once at `__init__` Time

```python
self._release_script = redis_client.register_script(RELEASE_LOCK_SCRIPT)
```

`register_script` in `redis-py` returns a callable that re-evaluates the script on each call using `EVALSHA` with a fallback to `EVAL` on cache miss. This is correct behavior. However, if the `redis_client` is replaced or reconnected (e.g., connection pool failover), the registered script object retains a reference to the original client. Document that `OrderProcessor` instances must not outlive their `redis_client` connection context.

---

### Test Coverage Gap — TTL Expiry and Heartbeat Path Not Tested

The test suite covers the happy path, idempotency, lock contention, and input validation, which is a reasonable baseline. However, there are no tests for:

- Lock expiry mid-payment (the critical scenario described above)
- `PaymentError` propagation and its effect on the caller
- Database transaction rollback when `record_charge` raises
- Lock release when an exception is raised inside the lock body

These are exactly the failure paths that matter in production. They should be added before merge.

---

### Minor — `ORDER_LOCK_TTL` and `MAX_RETRIES` Are Class-Level Constants With No Validation

If these are ever moved to configuration, there is no enforcement that `ORDER_LOCK_TTL > max_payment_duration`. Add a startup assertion or a `__post_init__` / `__init__` check once the heartbeat issue is resolved.
