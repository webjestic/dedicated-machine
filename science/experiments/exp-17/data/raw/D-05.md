Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all solid patterns. However, there is a critical atomicity flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is safe to merge into a payment-handling service.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — at-most-once guarantee is broken

This is the most serious bug. The PR description claims:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But the code does this:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← External network call INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment gateway call is a **non-transactional external side effect**. Wrapping it in a DB transaction does not make it atomic with the DB writes. The real failure scenario this creates:

1. `payment.charge()` succeeds — money is taken from the customer
2. `self.db.record_charge()` raises an exception or the process crashes
3. The DB transaction rolls back
4. The order is now in a state where money was charged but no record exists
5. The idempotency check (`db.get_charge`) will find nothing, so the **next retry will charge again**

The correct pattern is:

```python
# Charge BEFORE opening the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record the already-completed charge atomically
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This way, if the DB write fails, the charge exists at the gateway and can be reconciled or refunded — but it will never be silently re-charged on retry because the idempotency check will not find a DB record. You still need a reconciliation process for the orphaned charge, but you have eliminated the double-charge vector. Additionally, consider storing a `pending` charge record *before* calling the gateway so a crash mid-flight is detectable.

### 2. Redis lock TTL is dangerously short relative to payment retry window

```python
ORDER_LOCK_TTL = 60   # seconds
# PaymentClient:
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
REQUEST_TIMEOUT = 30  # seconds per attempt
```

Worst-case payment client runtime: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`.

The lock expires after 60 seconds. A second worker can acquire the lock while the first worker is still in the middle of its third payment retry. This directly re-introduces the double-charge race condition the lock is supposed to prevent. The TTL must be set to at least `(MAX_RETRIES * REQUEST_TIMEOUT) + (MAX_RETRIES * RETRY_BACKOFF) + DB_overhead` with a safety margin. Either increase `ORDER_LOCK_TTL` to 150+ seconds or reduce the payment client's maximum runtime to fit safely within the TTL.

## Significant Issues

### 3. `PaymentError` is silently swallowed — failed charges are invisible

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of the `with self._acquire_lock` block, which means:

- The lock **is** correctly released (the `finally` block handles that — this part is fine)
- But the caller receives an **unhandled exception** rather than an `OrderResult`

The method signature promises to return `OrderResult`. A `PaymentError` escaping to the caller breaks that contract and will likely produce an unhandled 500 at the API layer with no structured error information. Catch it explicitly:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 4. `payment_token` is logged at INFO level in `payment_client.py`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This specific line is fine, but review the full logging surface. If any upstream caller or middleware logs the full request object or `OrderRequest`, `payment_token` will appear in logs. The `payment_token` field should be treated as a secret credential. Consider masking it in `__repr__`/`__str__` on `OrderRequest`, or excluding it from any serialization used in log statements. Given that this is a government contracting environment, token leakage into log aggregation pipelines is a compliance concern.

### 5. `charge_id` key access on payment response has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id` key, non-JSON content type), this raises a `KeyError` or `JSONDecodeError` that is not caught. This will bubble up as an unhandled exception. Add explicit validation:

```python
data = response.json()
if "charge_id" not in data:
    raise PaymentError(f"Unexpected gateway response: missing charge_id")
charge_id = data["charge_id"]
```

## Minor Issues

### 6. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that `redis.delete` was not called and the script mock was called — but it is asserting implementation detail rather than observable behavior. More importantly, it accesses `processor._release_script` which is a private attribute. A more meaningful test would verify the lock is released even when an exception occurs mid-processing (i.e., the `finally` block is exercised on failure paths). That is the safety property worth testing.

### 7. No test coverage for `PaymentError` path or lock TTL expiry scenario

There are no tests for:
- Payment gateway failure after lock acquisition (verifies lock is still released)
- The idempotency check racing with lock expiry
- `PaymentError` propagation behavior (which, per issue #3 above, is currently broken)
