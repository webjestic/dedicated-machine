# Pull Request Review: Distributed Order Processor with Redis-Backed Exclusive Locking

---

## Lock Lifecycle Trace

Before reaching a verdict, I'm tracing every execution path through the lock lifecycle, because that's where distributed payment systems fail silently.

### Path 1: Happy Path

```
acquire(SET NX EX 60) → yield token → idempotency check → db.transaction() →
payment.charge() → db.record_charge() → db.mark_order_complete() →
finally: release_script(token) ✓
```

Lock is acquired, held for the duration of payment + DB write, released atomically. This path is structurally sound.

---

### Path 2: Payment Gateway Timeout (Critical Flaw Found)

```
acquire(SET NX EX 60) → yield token → idempotency check →
db.transaction() → payment.charge() → [TIMEOUT at attempt 1] →
time.sleep(5) → [TIMEOUT at attempt 2] → time.sleep(5) →
[TIMEOUT at attempt 3] → raise PaymentError
```

**The payment client's total retry wall time is:**

```
attempt_1(30s timeout) + sleep(5s) + attempt_2(30s timeout) + sleep(5s) + attempt_3(30s timeout)
= 30 + 5 + 30 + 5 + 30 = 100 seconds maximum
```

**The lock TTL is 60 seconds.**

The lock expires at second 60. The payment client is still retrying until second 100. During the window between second 60 and second 100, another worker can acquire the lock, pass the idempotency check (no charge recorded yet — the first worker's charge hasn't succeeded), and issue a second charge. If the first worker's third attempt then succeeds, you have two charges against the same order. This is exactly the double-charge scenario the lock was designed to prevent.

The comment in the code even acknowledges the intent: *"covers typical payment round-trip."* But "typical" is not "maximum under retry," and the retry budget was never reconciled against the TTL.

---

### Path 3: Lock Expiry During DB Transaction

Even if the payment succeeds quickly, consider:

```
acquire(SET NX EX 60) → payment.charge() [succeeds at second 58] →
db.record_charge() [slow DB, lock expires at second 60] →
Worker B acquires lock → idempotency check passes (record not committed yet) →
Worker B issues second charge
```

This is a narrower window than Path 2, but it's real under DB load. The lock TTL provides no safety margin for slow commit paths.

---

### Path 4: `PaymentError` Propagation Through `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # raises PaymentError
    self.db.record_charge(...)             # never reached
    self.db.mark_order_complete(...)       # never reached
```

`PaymentError` is not caught inside `process()`. It propagates out of `db.transaction()`, which should trigger a rollback — that part is fine. It then propagates out of `_acquire_lock`'s `try` block, hitting the `finally` clause, which releases the lock. The lock is released, no charge was recorded, and the caller receives an unhandled exception.

The problem: the caller gets a raw `PaymentError` stack trace rather than a structured `OrderResult`. The `process()` method's contract is to return `OrderResult`, but it silently breaks that contract on payment failure. This is an API contract violation that will cause upstream callers to crash if they're not defensively catching arbitrary exceptions.

---

### Path 5: Timeout During First Attempt — Ambiguous Charge State

```
payment.charge() → Timeout exception raised
```

A `requests.Timeout` means the request may or may not have reached the gateway. The gateway may have processed the charge and the response was lost in transit. The retry then issues a second charge request with the same token. Whether this is safe depends entirely on whether the payment gateway is idempotent on `token` — and that is **not documented, not enforced, and not tested here.** If the gateway treats each POST as a new charge, retrying on timeout is a billing error waiting to happen.

---

### Path 6: `db.get_charge()` Inside the Lock — Race Condition on Read

The idempotency check reads from the database inside the lock, which is correct. However, `db.get_charge()` is not wrapped in the `db.transaction()` block. If the DB read and the subsequent write are not in the same transaction, there is a TOCTOU window:

```
Worker A: get_charge() → None (no existing charge)
Worker A: enters db.transaction()
[DB replica lag — Worker A is reading a stale replica]
Worker B: [different order, unrelated] commits a record that causes A's read to be stale
```

This is lower severity than Paths 2 and 3, but the idempotency check should be inside the transaction boundary, or the `get_charge` read should be a `SELECT FOR UPDATE` equivalent to guarantee consistency.

---

### Path 7: Lock Retry Backoff vs. Lock TTL

```
LOCK_RETRY_ATTEMPTS = 3
backoff: 0.5s, 1.0s, 1.5s → total wait: 3 seconds before giving up
ORDER_LOCK_TTL = 60 seconds
```

A worker that fails to acquire the lock gives up after 3 seconds total. The lock holder may be mid-payment (legitimately holding the lock for up to 100 seconds per Path 2). The retry window is so short relative to the lock TTL that contention will almost always result in a `LockError` returned to the caller, who must then decide whether to retry at a higher level. This isn't a correctness flaw, but it means the retry logic here provides almost no practical benefit — the caller will see failures under any real contention.

---

### Path 8: `payment_token` Logged Indirectly

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Payment tokens are sensitive credentials. This needs explicit scrubbing or a custom error serializer that excludes the `payment_token` field value.

---

### Path 9: `requests` Without SSL Verification Confirmation

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

No explicit `verify=True` is set. While `True` is the default, for a payment endpoint carrying tokens and charge amounts, this should be explicit and enforced. Additionally, there is no certificate pinning or mTLS configuration, which for an internal payment gateway is a significant omission in a government contracting context.

## Summary of Findings

| Severity | Finding |
|---|---|
| **Critical** | Payment retry budget (100s max) exceeds lock TTL (60s), enabling double-charge under gateway timeout |
| **Critical** | `PaymentError` not caught in `process()`, breaking `OrderResult` return contract for callers |
| **High** | Gateway idempotency on timeout retry is assumed but not documented or enforced |
| **High** | `payment_token` may appear in logs via Pydantic validation error serialization |
| **Medium** | Idempotency check outside transaction boundary creates TOCTOU window under replica lag |
| **Medium** | SSL verification not explicit on payment endpoint; no mTLS for internal gateway |
| **Low** | Lock retry window (3s) is practically useless against a 60s TTL holder |

---

**Decision:** Request Changes

**Summary:** The payment client's maximum retry duration (100 seconds across three attempts with backoff) exceeds the Redis lock TTL (60 seconds), creating a concrete double-charge window under gateway timeout conditions — the exact failure mode this lock was designed to prevent. Additionally, `PaymentError` is not caught within `process()`, silently breaking the method's `OrderResult` return contract and crashing any caller that does not defensively handle arbitrary exceptions. Both issues must be resolved before this can be approved; at minimum, the lock TTL must exceed the payment client's maximum possible wall time, and all payment failure paths must return structured `OrderResult` objects.
