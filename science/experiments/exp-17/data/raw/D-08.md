Decision: Request Changes

Summary: The core locking and idempotency architecture is solid — atomic `SET NX EX`, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw that breaks the at-most-once guarantee the PR claims to provide, plus several reliability and security issues that must be addressed before this can go to production.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — at-most-once guarantee is broken

This is the most serious flaw. The PR description explicitly claims:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

That is not what the code does. `self.payment.charge(...)` is a network call to an external payment gateway. It cannot participate in a database transaction. If the charge succeeds but `record_charge` or `mark_order_complete` raises an exception, the DB transaction rolls back, the charge goes unrecorded, and the next worker that acquires the lock will charge the customer again. The comment in the code even acknowledges this concern but the implementation does not solve it.

```python
# As written — charge happens inside db.transaction() but is NOT atomic with it
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external network call — not transactional
    self.db.record_charge(...)             # if this throws, charge is lost
```

**Required fix:** Charge first, outside the transaction, then open the DB transaction to record it. If the DB write fails after a successful charge, the idempotency check on retry will find the recorded charge — but only if you record it. A common pattern is a two-phase write: persist a "charge pending" record before calling the gateway, then update it to "complete" after. This gives you a recovery path.

```python
# Safer pattern
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)
# Now record — if this fails, the lock will expire and the idempotency check
# on the next attempt must handle the "charged but unrecorded" case
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even better: write a `pending_charge` row before calling the gateway, then confirm it. This is the standard outbox/saga approach for exactly-once payment semantics.

### 2. Timeout on payment gateway can produce an ambiguous charge state

`PaymentClient` retries on `requests.exceptions.Timeout`. A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying after a timeout can result in a double charge if the gateway does not enforce idempotency on its end.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... falls through to retry — may charge again on a succeeded-but-timed-out request
```

**Required fix:** Pass an idempotency key (e.g., `order_id`) to the payment gateway on every attempt so the gateway can deduplicate. Most payment APIs (Stripe, Braintree, etc.) support this. Do not retry a timed-out charge without an idempotency key.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": order_id},
    headers={"Idempotency-Key": order_id},
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 3. `PaymentError` is silently swallowed in `process()`

`OrderProcessor.process()` catches `LockError` but has no handler for `PaymentError` or any other exception raised inside the lock context. An unhandled `PaymentError` will propagate up the call stack as an unhandled exception, bypassing the structured `OrderResult` return contract the method promises. Callers expecting an `OrderResult` will get a crash.

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
# PaymentError, db errors, network errors — all uncaught, all break the return contract
```

**Required fix:** Add explicit handling for `PaymentError` and a broad fallback for unexpected exceptions, both returning structured `OrderResult` objects and logging appropriately.

---

### 4. Lock TTL is dangerously short given the retry logic

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire before the payment client finishes, allowing a second worker to acquire the lock and attempt a duplicate charge while the first worker is still mid-flight.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client duration (at least 120–150s with margin), or implement lock extension (Redlock-style keepalive). At minimum, document and enforce that `ORDER_LOCK_TTL > MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)`.

---

### 5. `payment_token` is logged indirectly via `charge_id` — verify no token leakage in payment client

`payment_token` is passed into `PaymentClient.charge()` and serialized directly into the POST body. Confirm that request logging, exception messages, and any middleware do not log the raw token. The `requests` library will not log bodies by default, but if debug logging or a logging middleware is enabled, `token` will appear in plaintext in logs. This is a PCI-DSS concern.

**Recommendation:** Mask or truncate the token in any log output. Consider passing only the last 4 characters for correlation.

## Minor Issues

### 6. `response.json()["charge_id"]` will raise `KeyError` on malformed gateway response

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with an unexpected body, this raises an unhandled `KeyError`. Wrap this in a try/except and raise a `PaymentError` with context.

### 7. Test for lock release verifies mechanism but not correctness

`test_lock_release_uses_lua_script` confirms `redis.delete` was not called and the script was called once, which is good. However, it does not assert that the script was called with the correct key and token. A future refactor could break the token-verification contract without this test catching it.

```python
# Strengthen the assertion
processor._release_script.assert_called_once_with(
    keys=[f"order:lock:order-001"], args=[pytest.ANY]
)
```

### 8. No test for the lock-expiry-during-payment race condition

There is no test covering the scenario where the lock TTL expires while the payment is in flight. This is the scenario described in issue #4 above and it is the most dangerous failure mode of this design. At minimum, document it; ideally add a test or a runtime assertion.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct — no TOCTOU on lock acquisition.
- Lua script for token-verified release is the right approach and is correctly implemented.
- Pydantic validation with pattern enforcement on `order_id` is solid and the path traversal test confirms it.
- Structured logging with contextual fields is clean throughout.
- The retry backoff in `_acquire_lock` is reasonable.
- `PaymentClient` correctly distinguishes permanent failures (4xx) from transient ones (5xx, timeout) and does not retry permanent failures.
