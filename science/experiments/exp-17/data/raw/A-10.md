## Lock Lifecycle Trace — Full Concurrent Execution Analysis

Before I render a verdict I need to walk every path where the lock, the payment call, and the database transaction interact under failure. This is where payment infrastructure either holds or doesn't.

---

### Critical Flaw #1: Payment Call Lives Outside the Database Transaction — Timeout Creates an Unrecorded Charge

The PR summary explicitly claims:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

That claim is false. Trace this execution path:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← network call INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is a 30-second blocking HTTP request sitting inside an open database transaction. This is a compound failure on two axes:

**Axis 1 — Connection starvation.** The database transaction holds a connection (and likely a row-level lock on the order record) for the entire duration of the payment gateway round-trip, up to 30 seconds, multiplied by up to 3 retry attempts with 5-second sleeps between them. Under load, this exhausts the connection pool. Every other order hitting the same DB is now blocked or erroring. This is a classic "slow external call inside a transaction" anti-pattern that has taken down payment systems in production.

**Axis 2 — The atomicity claim is inverted.** The intent was to prevent a charged-but-not-recorded state. But a database transaction cannot roll back a charge that already settled at the payment gateway. If `payment.charge()` succeeds and returns a `charge_id`, and then `record_charge()` raises an exception, the DB transaction rolls back, the charge_id is lost, and the customer has been charged with no record in the system. The lock releases. The next worker acquires the lock, finds no existing charge, and charges again. This is a double-charge scenario, which is the exact failure mode this PR claims to prevent.

**The correct architecture** separates these concerns:

```python
# 1. Idempotency check (inside lock, outside transaction)
existing = self.db.get_charge(req.order_id)
if existing:
    return already_processed

# 2. Charge the payment gateway (outside the DB transaction)
charge_id = self.payment.charge(token=..., amount_cents=...)

# 3. Record the result atomically (short, fast DB transaction)
with self.db.transaction():
    self.db.record_charge(order_id, customer_id, charge_id, amount_cents)
    self.db.mark_order_complete(order_id)
```

Step 2 can still fail after charge succeeds, but that is handled by the idempotency check on retry — which only works if the record was written. The real fix requires an outbox pattern or a pre-write of a "charge_pending" record before calling the gateway, so a crash between charge and record is recoverable. This PR does neither.

### Critical Flaw #2: Lock TTL Does Not Cover the Actual Critical Section Duration

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment client execution time: `(30s × 3 attempts) + (5s × 2 sleeps)` = **100 seconds**.

The lock TTL is 60 seconds. At second 61, Redis expires the lock. A second worker acquires it, passes the idempotency check (no charge recorded yet — the first worker is still on attempt 2), and initiates a second charge. When the first worker's attempt 3 eventually succeeds, both workers record a charge. The lock has provided zero protection for the scenario it was designed to prevent.

There is no heartbeat renewal mechanism. The lock is set-and-forget. For a critical section whose duration is bounded by a retrying network call, a static TTL without renewal is not a safe design.

### Critical Flaw #3: `PaymentError` Is Not Caught — Unhandled Exception Leaks Through the Lock Context

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError on permanent failure
        ...
```

`PaymentError` is defined in `payment_client.py` but is never caught in `order_processor.py`. When the payment gateway returns a 4xx or exhausts retries, `PaymentError` propagates up through `_acquire_lock`'s `finally` block (lock is correctly released — that part works), then exits `process()` as an unhandled exception. The caller receives a raw exception instead of an `OrderResult`. The method signature promises `OrderResult` but does not deliver it under this path. Any upstream handler expecting a structured result will crash or mishandle the error.

---

### Critical Flaw #4: Timeout Exception Does Not Guarantee No Charge Was Made

In `PaymentClient.charge()`:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the client did not receive a response — it does not mean the gateway did not process the charge. The gateway may have charged the card and the response was lost in transit. The code retries unconditionally on timeout. Without idempotency keys sent to the payment gateway on each attempt, each retry is a fresh charge attempt. Three timeouts could produce three charges at the gateway with zero recorded in the database.

The fix is to generate a stable idempotency key per order (e.g., derived from `order_id`) and pass it as a header on every attempt to the same gateway endpoint, so the gateway deduplicates on its side.

---

### Moderate Issue #5: `db.get_charge()` Idempotency Check Has No Isolation Guarantee

The idempotency check and the charge happen in two separate steps with no database-level lock between them. Two workers can both pass `get_charge()` returning `None` before either has written a record. The Redis lock is supposed to prevent this, but as shown in Flaw #2, the lock can expire mid-execution. Even with a correct TTL, the idempotency check needs a database-level unique constraint on `order_id` in the charges table as a hard backstop. If that constraint exists, it is not visible in this PR. If it does not exist, the idempotency check is advisory only.

---

### Minor Issue #6: Test Coverage Does Not Exercise the Failure Paths That Matter

The tests verify the happy path, the already-processed path, and lock contention. They do not test:

- `PaymentError` propagation (the unhandled exception path)
- Lock expiry mid-execution (TTL boundary crossing)
- `payment.charge()` succeeding followed by `db.record_charge()` raising (the double-charge scenario)
- Timeout retry behavior and whether idempotency keys are passed

The tests that do exist mock at too high a level to catch the structural issues above.

---

### Summary of Findings

| Severity | Issue |
|---|---|
| Critical | Payment call inside DB transaction: connection starvation + inverted atomicity guarantee |
| Critical | Lock TTL (60s) shorter than worst-case critical section (100s), no heartbeat renewal |
| Critical | `PaymentError` uncaught — method contract broken on payment failure |
| Critical | Timeout retry without gateway idempotency keys — potential multiple charges |
| Moderate | No DB-level unique constraint visible as backstop for idempotency check |
| Minor | Test suite does not cover the failure paths that matter |

---

**Decision:** Request Changes

**Summary:** The PR's central safety claim — that charge and record are atomic and double-charging is prevented — is broken by four independent critical defects: the payment call sits inside the database transaction inverting the atomicity guarantee, the lock TTL is shorter than the worst-case payment retry duration with no heartbeat renewal, `PaymentError` is never caught leaving the method's return contract broken, and timeout retries are issued without gateway idempotency keys. All four must be resolved before this can be approved.
