Decision: Request Changes

Summary: The distributed lock design contains a critical gap: there is no heartbeat or lock renewal mechanism, meaning any payment round-trip that approaches or exceeds the 60-second TTL will cause the lock to expire silently while the worker continues executing, allowing a second worker to acquire the lock, pass the idempotency check, and issue a duplicate charge before the first worker's database transaction commits. Additionally, placing `self.payment.charge()` inside `self.db.transaction()` creates a non-atomic boundary between an external side-effect and the database commit — a crash or rollback after the charge succeeds but before `record_charge` commits leaves an unrecorded real charge with no recovery path. These two issues together directly undermine the at-most-once guarantee the PR claims to provide.

## Full Review

### Critical — Must Fix Before Merge

#### 1. No Lock Heartbeat: TTL Expiry Silently Invalidates the Lock Mid-Execution

This is the most serious flaw. The lock is acquired with a hard 60-second TTL and never renewed.

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

The comment says "covers typical payment round-trip," but the payment client is configured with:

```python
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. That exceeds the lock TTL by 40 seconds. Even under normal conditions, a slow gateway response plus database latency can push past 60 seconds.

When the TTL expires, Redis silently drops the key. A second worker then calls `SET NX EX`, succeeds, passes the idempotency check (because `record_charge` hasn't committed yet), and proceeds to charge. Worker 1 eventually commits. You now have two charges and one record, or two charges and two records depending on timing. The Lua release script does not protect against this — it correctly refuses to release a lock the worker doesn't hold, but that's cold comfort after the duplicate charge has already been issued.

**Required fix:** Implement a background heartbeat thread that calls `EXPIRE order:lock:{order_id} 60` at a regular interval (e.g., every 20 seconds) while the lock is held. The heartbeat must be stopped and the result checked before any critical section decision is made. Alternatively, reduce the payment client's maximum possible duration well below the TTL and enforce that contract with a hard timeout wrapper.

A sketch of what the heartbeat context manager needs to do:

```python
# On entry: acquire lock, start daemon thread issuing EXPIRE every interval
# On exit: signal heartbeat to stop, join it, then run Lua release
# The heartbeat thread must record whether renewal ever failed so the
# main thread can abort rather than commit after a lapsed lock
```

#### 2. External Side-Effect Inside Database Transaction Creates Unrecoverable Inconsistency

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations. It provides nothing over the external payment call. The failure modes here are severe:

- **Charge succeeds, `record_charge` raises:** The transaction rolls back. The idempotency check will now pass again on retry because no record exists, and the customer gets charged twice.
- **Charge succeeds, database connection drops before commit:** Same outcome.
- **`db.transaction().__exit__` raises during commit after charge:** Caller sees an exception, retries, charges again.

The comment in the PR says "crash between charge and record cannot produce an unrecorded charge" — this is incorrect. The database transaction does not wrap the payment call in any meaningful way.

**Required fix:** Separate the charge from the transaction. The standard pattern for at-most-once external calls with durable recording is:

1. Within a transaction, write a `charge_pending` record with a generated idempotency key before calling the gateway.
2. Call the gateway using that idempotency key so the gateway itself deduplicates retries.
3. Within a second transaction, update the record to `charge_complete` with the returned `charge_id`.

If the process crashes between steps 2 and 3, a reconciliation job can query the gateway using the stored idempotency key to determine the outcome. This is the only design that is actually safe here.

### High — Must Fix Before Merge

#### 3. Lock Expiry Is Not Detected at Release Time

The Lua release script correctly handles the case where another worker holds the lock:

```python
RELEASE_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
```

But the return value is never checked:

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

A return value of `0` means the lock had already expired and was potentially held by another worker during the critical section. This is a silent failure. The log line says "released" regardless of whether the lock was actually held at release time. If a heartbeat is added (fix for issue 1), this check becomes the mechanism by which the main thread learns the lock lapsed.

```python
released = self._release_script(keys=[lock_key], args=[token])
if not released:
    logger.error(
        "Lock was not held at release — possible TTL expiry during processing order_id=%s",
        order_id,
    )
    # Raise or record a critical alert; do not silently continue
```

#### 4. Payment Client Retry on Timeout Is Unsafe Without Gateway-Side Idempotency

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have been received and processed by the gateway — the response was simply lost in transit. Retrying without an idempotency key will issue a second charge attempt that the gateway has no way to deduplicate. This must be paired with a stable, per-order idempotency key sent as a request header (e.g., `Idempotency-Key: {order_id}`) so the gateway can safely deduplicate retries.

### Medium — Address Before Merge

#### 5. `payment_token` Echoed Into Logs on Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds max length), the raw token value may appear in log output. Payment tokens are sensitive credentials. Scrub or suppress field values from validation error logs, or explicitly construct a safe error message listing only field names.

#### 6. No Handling of Unexpected Exceptions Inside the Lock

If `self.db.get_charge()`, `self.payment.charge()`, or any database call raises an unexpected exception, it propagates out of the `with self._acquire_lock()` block, which correctly triggers the `finally` and releases the lock. That part is fine. What is missing is any logging or structured error response for the caller. The `process()` method will raise rather than returning an `OrderResult`, which breaks the return type contract and will surface as an unhandled exception to whatever calls `process()`.

```python
# Wrap the lock body in a broad except and return a structured failure result
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### Low — Cleanup

#### 7. Test Coverage Does Not Exercise the Failure Paths That Matter Most

The tests mock away the lock, the database, and the payment client entirely, which means none of the critical failure scenarios are tested:

- Lock TTL expiry during processing (not testable without a heartbeat to test against)
- Charge succeeds but `record_charge` raises
- Payment timeout followed by retry
- `_release_script` returning `0`

These are the exact scenarios this code is supposed to handle. The test suite as written confirms the happy path and basic contention detection, but provides no confidence in the failure modes. Integration-level tests against a real Redis instance (e.g., using `fakeredis`) and a mock HTTP server should be added.

#### 8. `LOCK_RETRY_ATTEMPTS = 3` With No Jitter

The retry backoff is deterministic: `0.5s, 1.0s, 1.5s`. Under high concurrency, multiple workers that failed to acquire the lock at the same time will retry in lockstep, creating a thundering herd at each backoff boundary. Add jitter: `backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.5)`.
