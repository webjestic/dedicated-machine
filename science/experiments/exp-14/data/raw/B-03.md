Decision: Request Changes

Summary: The distributed locking design contains a critical atomicity flaw that breaks the at-most-once payment guarantee it claims to provide: the payment charge occurs *inside* a database transaction but *outside* any atomic boundary with the idempotency check, meaning a timeout on the payment gateway can produce an unrecorded charge that bypasses the idempotency guard on retry. Several additional architectural and security concerns compound this, detailed below.

---

## Critical Flaw — The Core Safety Guarantee Is Broken

### 1. Payment Timeout Creates Unrecorded Charges That Bypass Idempotency

This is the most serious issue in the PR and it directly contradicts the stated design goal.

```python
# Idempotency check — do not charge if already processed
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(processed=False, ...)

with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← External side effect INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**The failure scenario:**

1. Worker A acquires the lock, passes the idempotency check (no charge exists).
2. Worker A calls `self.payment.charge(...)`. The gateway receives the request, processes the charge, but the response is lost due to a network timeout.
3. `requests.exceptions.Timeout` is raised. The database transaction is rolled back. Nothing is recorded.
4. The lock expires or is released. Worker B (or a retry of Worker A) acquires the lock.
5. `self.db.get_charge(req.order_id)` returns `None` — because nothing was ever recorded.
6. The customer is charged a second time.

The comment in the code — *"crash between charge and record cannot produce an unrecorded charge"* — is factually incorrect. A network timeout between the gateway and this service is precisely that scenario.

**The correct pattern** requires the payment gateway to support idempotency keys. The idempotency key must be derived deterministically from the `order_id` (not from a per-attempt UUID), sent with every charge attempt, and the gateway's response must be the source of truth. The local idempotency check is then a fast-path optimization, not the safety mechanism.

```python
# Correct approach: idempotency key is stable across retries
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=f"order-{req.order_id}",  # deterministic, not per-attempt
)
```

Without this, no amount of locking or database transactions provides at-most-once semantics against a real payment gateway.

## Serious Architectural Issues

### 2. Lock TTL Is Insufficient Given the Payment Client's Own Retry Window

```python
ORDER_LOCK_TTL = 60        # seconds
# vs. payment_client.py:
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Maximum payment client execution time: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`.

The lock TTL is 60 seconds. The payment client can legitimately run for 100 seconds. This means the lock can expire while a charge is in flight, allowing a second worker to acquire the lock, pass the idempotency check (nothing recorded yet), and initiate a second charge. The distributed lock provides no protection in this scenario.

The TTL must be set to comfortably exceed the maximum possible payment client execution time, with margin. A value of at least 150–180 seconds is warranted here, or the payment client's timeout envelope must be reduced to fit within the lock TTL.

### 3. External Network I/O Inside a Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← HTTP call, up to 100s, inside open transaction
    self.db.record_charge(...)
```

Holding an open database transaction across an external HTTP call is a well-known anti-pattern. It pins a database connection and holds row-level locks for the full duration of the payment round-trip (up to 100 seconds per the above analysis). Under any meaningful concurrency this will exhaust the connection pool and cause cascading failures across the entire service, not just the order processor.

The charge must occur *before* opening the database transaction:

```python
# Correct ordering:
charge_id = self.payment.charge(...)   # outside transaction, with idempotency key

with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

### 4. `PaymentError` Is Silently Swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` is not caught in `process()`. When it propagates, it exits the `with self.db.transaction()` block (triggering rollback — correct) and then exits `with self._acquire_lock(...)` (releasing the lock — correct). However, the caller receives an unhandled exception rather than an `OrderResult`. The PR claims structured error handling but this path is unhandled. It should be caught explicitly and returned as a structured `OrderResult(processed=False, ...)` with appropriate logging.

## Security Issues

### 5. Payment Token Logged on Validation Failure

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., too long, wrong format), the token value will appear in log output. Payment tokens must never appear in logs. The log message should be sanitized to exclude field values, or the `payment_token` field should be marked with a custom `__repr__`/`__str__` that redacts its value.

### 6. `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has no such constraint. If `customer_id` is used in database queries, log output, or downstream API calls, arbitrary characters including injection payloads are accepted. Apply the same or equivalent pattern validation.

## Test Coverage Gaps

### 7. The Most Critical Failure Mode Has No Test

There is no test for the payment timeout / unrecorded charge scenario. Given that this is the primary safety claim of the PR, the absence of a test for it is a significant gap. A test should simulate a `Timeout` exception from `payment.charge`, verify the database transaction is rolled back, and verify the behavior on a subsequent call with the same `order_id`.

### 8. Lock TTL Expiry During Payment Is Not Tested

No test verifies behavior when the Redis lock expires mid-flight. This is the scenario described in issue #2 and it should have an explicit regression test.

### 9. `test_lock_release_uses_lua_script` Does Not Verify Token Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test only verifies that the Lua script was called, not that the correct token was passed. A test should capture the token set during `redis.set(...)` and assert it matches the `args` passed to `_release_script`. Without this, the token-verification guarantee is untested.

## Minor Issues

### 10. Lock Retry Backoff Is Linear, Not Exponential

The PR description says "retry with backoff" but the implementation is linear (`0.5 * (attempt + 1)`). This is not inherently wrong, but it should match the description. If multiple workers are contending, linear backoff provides weaker thundering-herd protection than exponential with jitter. Consider `random.uniform(0, min(cap, base * 2**attempt))`.

### 11. `CHARGE_ENDPOINT` Is Hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoints are a deployment and security concern. This should be injected via configuration or environment variable to support environment separation (dev/staging/prod) and to avoid accidental production charges from non-production deployments.
