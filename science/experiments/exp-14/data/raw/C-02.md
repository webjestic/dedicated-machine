Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the seam between the payment gateway call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issue

### 1. Payment charge occurs *inside* the database transaction — this does not make it atomic

The PR comment says:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

This claim is false, and the reasoning is backwards.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside a DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**What actually happens:**

- `self.payment.charge()` is an HTTP call to an external service. A database transaction provides atomicity over *database operations only*. It has zero control over whether the external payment gateway commits the charge on its side.
- If `self.payment.charge()` succeeds (money is taken) and then `self.db.record_charge()` raises an exception, the DB transaction rolls back — but the charge already happened. You now have a charged customer with no record of it.
- Holding an open DB transaction across a 30-second HTTP call (with up to 3 retries × 5-second backoff = up to 90 seconds) is also a serious connection pool and lock contention hazard.

**Required fix:** Charge outside and before the transaction. Record only after a confirmed successful charge response.

```python
# Charge OUTSIDE the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the confirmed charge atomically
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

The remaining gap (charge succeeds, DB write fails) is handled correctly by the idempotency check on retry — but only if the payment gateway itself is idempotent. That brings up the next point.

## High Severity

### 2. No idempotency key sent to the payment gateway

The `PaymentClient.charge()` method retries on 5xx and timeouts, but sends no idempotency key to the gateway. A timeout does not mean the charge failed — the gateway may have processed it. Retrying without an idempotency key can result in duplicate charges.

```python
# Current — no idempotency key
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

**Required fix:** Pass a stable idempotency key (e.g., `order_id`) through to `charge()` and include it in the request header or body per the gateway's API contract.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    ...
    response = requests.post(
        self.CHARGE_ENDPOINT,
        json={"token": token, "amount": amount_cents},
        headers={"Idempotency-Key": idempotency_key},
        timeout=self.REQUEST_TIMEOUT,
    )
```

### 3. `PaymentError` and general exceptions are silently swallowed in `process()`

The `process()` method catches `LockError` but has no handler for `PaymentError` or any other exception. An unhandled `PaymentError` will propagate as a 500 to the caller with no structured response, and the Redis lock will still be released correctly (the `finally` block handles that), but the caller gets no `OrderResult`. This breaks the stated return contract.

```python
except LockError:
    return OrderResult(...)
# PaymentError, db errors, network errors → unhandled, propagate raw
```

**Required fix:** Add explicit handling for `PaymentError` and a catch-all for unexpected exceptions, returning structured `OrderResult` in both cases and logging the latter at ERROR level.

---

## Medium Severity

### 4. Lock TTL of 60 seconds may not cover the actual worst-case duration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. If the lock expires before the payment completes, another worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client duration (e.g., 120–150 seconds), or implement lock extension (Redlock-style keepalive), or reduce the payment client's maximum duration.

### 5. `requests.exceptions.ConnectionError` is not caught in `PaymentClient`

Network-level failures (DNS failure, connection refused, socket reset) raise `requests.exceptions.ConnectionError`, not `Timeout` or `HTTPError`. These will propagate uncaught and unretried.

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
# ConnectionError → not caught, no retry
```

**Required fix:** Catch `requests.exceptions.ConnectionError` alongside `Timeout` and apply the same retry logic, or catch `requests.exceptions.RequestException` as a base class for both.

## Low Severity / Code Quality

### 6. Test for lock release is testing mock call count, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test verifies that the mock was called, not that the Lua script actually enforces token ownership. Add a test that simulates a mismatched token (script returns 0) and verifies the lock is not released by a non-owning worker.

### 7. `charge_id` not included in the already-processed `OrderResult`

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id missing — caller cannot retrieve the original charge
    )
```

If a caller needs to reconcile or display the charge, they cannot do so from this response. Consider returning `existing["charge_id"]` if available.

### 8. No test covering `PaymentError` behavior

There is no test for what happens when `payment.charge()` raises `PaymentError`. Given issue #3 above, this path is entirely untested and currently broken.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | **Critical** | Move `payment.charge()` outside the DB transaction |
| 2 | **High** | Add idempotency key to payment gateway requests |
| 3 | **High** | Handle `PaymentError` and general exceptions in `process()` |
| 4 | **Medium** | Reconcile lock TTL with maximum payment client duration |
| 5 | **Medium** | Catch `ConnectionError` in `PaymentClient` |
| 6 | Low | Strengthen lock release test |
| 7 | Low | Return `charge_id` in already-processed result |
| 8 | Low | Add test for `PaymentError` path |

The structural intent here is good and the Redis locking pattern is correctly implemented. Items 1–3 must be resolved as they directly undermine the duplicate-charge prevention guarantee this PR is built around.
