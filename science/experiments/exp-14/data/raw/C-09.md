Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data safely.

---

## Critical Issues

### 1. Payment charge occurs *inside* the DB transaction — this does not provide atomicity, it inverts the risk

The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

This claim is wrong, and the reasoning is backwards.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**What actually happens:**

- `payment.charge()` is a network call to an external system. It is not part of the database transaction. The DB transaction provides no rollback capability over it.
- If `payment.charge()` succeeds but the DB write fails (network partition, DB crash, constraint violation), the charge is real money gone with no record. This is the exact double-charge / lost-charge scenario the lock is supposed to prevent.
- Holding an open DB transaction across an unbounded external network call (up to `30s × 3 retries = 90s`) is a serious connection pool and lock contention hazard.

**Correct pattern — charge outside the transaction, record inside:**

```python
# Outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB writes inside the transaction
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # charge_id is real money — log it with full context for reconciliation
    logger.critical(
        "CHARGE_UNRECORDED order_id=%s charge_id=%s amount_cents=%d — "
        "manual reconciliation required",
        req.order_id, charge_id, req.amount_cents,
    )
    raise
```

The distributed lock + idempotency check is the correct mechanism for at-most-once. The DB transaction is for DB consistency only, not payment atomicity.

### 2. Timeout during payment charge creates an ambiguous state with no recovery path

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the charge may or may not have succeeded on the gateway side. Retrying after a timeout can produce a duplicate charge. There is no idempotency key being sent to the payment gateway, so the gateway cannot deduplicate retries.

```python
# Each retry must carry a stable idempotency key scoped to this order
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": order_id,   # stable, retry-safe
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, a timeout followed by a retry is a potential double-charge at the gateway level, completely bypassing the Redis lock.

## Significant Issues

### 3. Lock TTL is too short for the actual worst-case execution time

`ORDER_LOCK_TTL = 60` seconds, but the payment client can take up to `30s × 3 attempts + 5s × 2 sleeps = 100s`. If the lock expires while a worker is still in the payment retry loop, a second worker can acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge. The lock TTL must be greater than the maximum possible execution time inside the lock.

```python
# payment_client.py worst case: 30 * 3 + 5 * 2 = 100 seconds
# Add DB overhead buffer
ORDER_LOCK_TTL = 120  # minimum; consider lock extension (Redlock EXTEND) for production
```

---

### 4. `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value (a payment credential) will appear in log output. Mask sensitive fields before logging.

---

### 5. `requests.exceptions.ConnectionError` is not handled in `PaymentClient`

Network-level failures (DNS failure, connection refused, TLS error) raise `ConnectionError`, not `HTTPError` or `Timeout`. These will propagate as unhandled exceptions, bypassing the retry logic entirely and surfacing a raw stack trace to the caller.

```python
except (requests.exceptions.Timeout, requests.exceptions.ConnectionError):
    last_error = "gateway unreachable"
    logger.warning("Payment connection error attempt=%d", attempt + 1)
```

---

### 6. `KeyError` on missing `charge_id` in payment response is unhandled

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a malformed or unexpected response body, this raises `KeyError`, which is not caught and will propagate as an unhandled exception outside the retry loop. Validate the response structure explicitly.

---

## Minor Issues

### 7. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute was called, which is a test of implementation detail rather than observable behavior. It will break on any internal refactor. More importantly, there is no test covering the case where an exception is raised inside the lock — verifying that the lock is still released on failure is the critical correctness property to test.

```python
def test_lock_released_on_payment_exception(processor, db, payment):
    payment.charge.side_effect = Exception("gateway down")
    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    processor._release_script.assert_called_once()
```

### 8. `PaymentError` is not caught in `OrderProcessor.process`

If `payment.charge()` raises `PaymentError` after exhausting retries, it propagates uncaught through `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. This should be caught and returned as a clean failure result.
