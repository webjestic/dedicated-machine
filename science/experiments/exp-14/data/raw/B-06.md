# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Charge Outside Database Transaction Atomicity Boundary

This is the most severe flaw in the PR, and it directly contradicts the comment in the code itself.

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(        # ← External network call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims a crash between charge and record is impossible, but that guarantee cannot hold. A database transaction provides atomicity over *database operations only*. The payment gateway call is an external network operation that is not part of the database transaction. The following failure modes are real and unaddressed:

- **Unrecorded charge:** The payment gateway succeeds and returns a `charge_id`, then a crash, timeout, or exception occurs before `record_charge` commits. The customer is charged. The database has no record. The lock is released. The next worker sees no existing charge and charges again.
- **Holding a database connection open during a network call:** `self.payment.charge()` has a 30-second timeout with up to 3 retries at 5-second backoff, meaning the database transaction can be held open for up to ~105 seconds. This will exhaust connection pools under any meaningful load.
- **Lock TTL vs. payment retry duration:** `ORDER_LOCK_TTL = 60` seconds, but the payment client can take up to ~105 seconds across retries. The lock expires before the payment completes, allowing a second worker to acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge.

The correct pattern is to perform the charge *before* opening the database transaction, then record the result:

```python
# Charge first — outside the db transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the result atomically — if this fails, charge_id is known
# and can be reconciled via the payment gateway's idempotency key
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This still leaves a narrow window where the charge succeeds but the DB write fails, which is why the payment gateway call should also use a **deterministic idempotency key** (e.g., derived from `order_id`) so the charge can be safely retried or reconciled. The current implementation passes no idempotency key to the gateway whatsoever.

### 🔴 Critical: Lock TTL Shorter Than Maximum Payment Duration

As noted above, `ORDER_LOCK_TTL = 60` seconds is shorter than the maximum payment client execution time:

```
MAX_RETRIES = 3, RETRY_BACKOFF = 5, REQUEST_TIMEOUT = 30
Worst case: 30 + 5 + 30 + 5 + 30 = 100 seconds
```

The lock expires while the payment is still in flight. A second worker acquires the lock, passes the idempotency check, and initiates a duplicate charge. This is the exact race condition the lock is supposed to prevent.

---

### 🔴 Critical: No Idempotency Key Passed to Payment Gateway

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # ← No idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

The payment client retries on 5xx and timeout, but with no idempotency key, each retry is a new charge attempt from the gateway's perspective. If the gateway processed the charge but returned a 5xx or timed out on the response, the retry will issue a second charge. The `order_id` must be passed as an idempotency key on every attempt.

---

### 🟡 Significant: Unhandled `PaymentError` Propagates Through Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # PaymentError raised here
        self.db.record_charge(...)
```

A `PaymentError` is not caught in `process()`. It will propagate up through the `_acquire_lock` context manager (which correctly releases the lock in `finally`), but the caller receives an unhandled exception rather than an `OrderResult`. The method signature promises `OrderResult` in all cases. This should be caught and returned as a structured failure result.

---

### 🟡 Significant: `get_charge` Race Condition on Idempotency Check

The idempotency check reads from the database *after* acquiring the lock, which is correct. However, the guarantee only holds if `get_charge` reads from the same database node that `record_charge` writes to. Under a replicated database with eventual consistency or read replicas, a recently committed charge may not be visible to the read, allowing the check to pass on a replica that has not yet received the write. This assumption must be documented and enforced at the infrastructure level.

---

### 🟡 Significant: Lock Retry Backoff Is Linear, Not Jittered

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Under high contention with many workers, synchronized linear backoff produces retry storms at predictable intervals. Standard practice for distributed systems is exponential backoff with random jitter. With only 3 attempts, the window is narrow, but this should be corrected before this pattern is reused elsewhere.

---

### 🟡 Significant: `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is accepted as input and passed to the payment gateway. Confirm that no logging path — including exception tracebacks from `requests`, Pydantic validation errors, or debug-level logging in the HTTP layer — can emit the raw token value. The `ValidationError` catch logs `e` directly; if the token fails validation, the invalid value may appear in the log output.

## `services/payment_client.py`

### 🟡 Significant: Retry on Timeout Without Idempotency Key Is Unsafe

As noted above, retrying a timed-out POST with no idempotency key is not a safe retry — it is a new charge. This is a fundamental correctness issue independent of the lock problem.

---

### 🟠 Minor: `response.json()["charge_id"]` — No KeyError Handling

```python
charge_id = response.json()["charge_id"]
```

A malformed or unexpected gateway response (missing key, non-JSON body) will raise an unhandled `KeyError` or `JSONDecodeError`. This should be caught and raised as a `PaymentError` with context.

---

## `tests/test_order_processor.py`

### 🟡 Significant: Tests Do Not Cover the Critical Failure Modes

The test suite validates the happy path and basic contention, but none of the following are tested:

- Lock expiry during payment processing (TTL race)
- `PaymentError` propagation and return value
- Gateway timeout followed by retry producing duplicate charge
- Database transaction failure after successful charge
- Idempotency check behavior under replica lag

Tests passing against a mock that does not model these failure modes provides false assurance. The checklist item "Tests pass" should not be treated as a correctness signal for the scenarios that matter most here.

**Decision:** Request Changes

**Summary:** This PR contains three critical correctness flaws that directly undermine its stated guarantee of at-most-once payment processing: the payment charge is incorrectly placed inside a database transaction boundary (providing no atomicity guarantee over the external call), the lock TTL is shorter than the maximum payment client execution time (allowing a second worker to acquire the lock and charge while the first is still in flight), and no idempotency key is passed to the payment gateway (making retries on timeout unsafe). These issues must be resolved before this code is merged, as the failure modes are not theoretical — they are reachable under normal operating conditions and will produce duplicate customer charges.
