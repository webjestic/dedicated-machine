# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Outside Database Transaction — At-Most-Once Guarantee is False

The PR description claims *"crash between charge and record cannot produce an unrecorded charge"* and an *"at-most-once payment guarantee."* Both claims are incorrect as written.

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External side effect INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment gateway call is a **network I/O operation against an external system**. Wrapping it in a database transaction does not make it atomic with respect to the payment gateway. The database transaction boundary controls only the local DB operations. The following failure modes remain entirely unaddressed:

- **Charge succeeds, DB write fails** — the database transaction rolls back, `record_charge` never persists, but the customer has been billed. The lock is then released. On retry, `get_charge` returns `None`, so the idempotency check passes and the customer is **charged again**.
- **Charge succeeds, process crashes before `record_charge`** — same outcome as above.
- **Gateway timeout on retry** — `PaymentClient` retries up to 3 times on timeout. The gateway may have processed the first attempt. Retrying can result in multiple charges if the gateway is not itself idempotent. There is no idempotency key passed to the gateway.

The correct architecture requires one of:
1. **Outbox pattern** — write a pending charge record to the DB *before* calling the gateway, within a transaction. A separate reconciliation process confirms or voids.
2. **Gateway-side idempotency key** — pass a stable, deterministic key (e.g., `order_id`) with every charge attempt so the gateway deduplicates retries.
3. **Two-phase approach** — record intent, charge, confirm. Never retry a charge without first checking gateway-side status.

### 🔴 Critical: Redis Lock Does Not Prevent the Race It Claims To

The lock provides mutual exclusion among workers *that respect the lock*, but the idempotency check and the charge are not atomic with respect to the lock's TTL.

```python
ORDER_LOCK_TTL = 60  # seconds
```

If the payment gateway call takes longer than 60 seconds (the `PaymentClient` timeout is `30s × 3 attempts + 5s × 2 backoffs = 100s` worst case), the lock **expires while the worker is still holding it**. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The lock TTL is structurally shorter than the maximum payment client execution time.

```
Max PaymentClient runtime = REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES - 1)
                          = 30 × 3 + 5 × 2
                          = 100 seconds

ORDER_LOCK_TTL            = 60 seconds
```

This is a direct contradiction. Either the TTL must exceed the worst-case payment duration, or lock extension (e.g., Redlock watchdog / `pexpire` renewal) must be implemented.

### 🟡 Significant: `PaymentError` is Silently Swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

`PaymentError` and any `requests` exception that escapes `PaymentClient` are not caught in `process()`. They will propagate up as unhandled exceptions to the caller with no `OrderResult` returned. The caller receives an exception rather than a structured failure response, breaking the stated return contract. This should be caught and returned as a structured `OrderResult(processed=False, ...)`.

---

### 🟡 Significant: Lock Released on Unhandled Exception — Exposes Re-entrancy Window

The `finally` block in `_acquire_lock` unconditionally releases the lock:

```python
try:
    yield token
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If an unhandled exception (e.g., `PaymentError`, DB connection failure) propagates out of the `with self._acquire_lock(...)` block, the lock is released immediately. A second worker can then acquire the lock and attempt to process the same order before any compensating action or reconciliation has occurred. For a payment system, intentional lock retention on certain failure classes (pending human or automated review) may be the safer default.

---

### 🟡 Significant: No Logging of Payment or DB Exceptions

If `self.payment.charge(...)` or any DB call raises, the exception propagates silently with no log entry at the `process()` level. Operational visibility into payment failures is absent. Structured error logging with `order_id` context should be present at the point of failure.

---

### 🟠 Minor: `customer_id` Has No Pattern Validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` enforces `r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. This is inconsistent and potentially exploitable depending on how `customer_id` is used downstream (SQL interpolation, log injection, external API passthrough).

---

### 🟠 Minor: Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

This is synchronous blocking sleep in what is presumably a worker thread or async context. Under high contention this degrades throughput. Jitter should be added to prevent thundering herd across multiple workers contending on the same order. If this runs in an async framework, `time.sleep` is a blocking call that will stall the event loop.

## `services/payment_client.py`

### 🔴 Critical: No Idempotency Key on Retry

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

On timeout, the client retries with the same payload. There is no idempotency key. If the first request was received and processed by the gateway but the response was lost in transit, the retry issues a second charge. This is the most common cause of duplicate charges in payment systems.

The fix is to generate a stable idempotency key per charge attempt (derived from `order_id`, not regenerated per retry) and pass it as a header or body field per the gateway's API contract.

---

### 🟡 Significant: `response.json()["charge_id"]` — No Key Guard

```python
charge_id = response.json()["charge_id"]
```

A `KeyError` here will propagate as an unhandled exception. A malformed but `2xx` response from the gateway will crash the worker. This should be guarded with `.get("charge_id")` and an explicit check, raising a `PaymentError` on missing data.

---

### 🟠 Minor: Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoints in source are an operational and security concern. This should be injected via configuration or environment variable to support environment promotion (dev/staging/prod) and secret management pipelines.

## `tests/test_order_processor.py`

### 🟡 Significant: Critical Failure Paths Not Tested

The test suite validates the happy path and basic contention, but the most dangerous scenarios are entirely absent:

- No test for `PaymentError` raised mid-transaction
- No test for DB failure after successful charge (the double-charge scenario)
- No test for lock TTL expiry during payment (requires mock time or TTL injection)
- No test verifying that a second worker cannot process the same order after the first worker's lock expires mid-flight
- No test verifying behavior when `record_charge` raises after `charge` succeeds

These are not edge cases — they are the primary failure modes this system is designed to prevent.

---

### 🟠 Minor: `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms the Lua script was called but does not verify that the correct token was passed as `args`. The token-verification property of the Lua script — the primary security guarantee of the lock — is untested.

**Decision:** Request Changes

**Summary:** This PR contains multiple critical correctness flaws that directly contradict its stated guarantees. The lock TTL (60s) is structurally shorter than the worst-case payment client execution time (100s), invalidating the mutual exclusion guarantee entirely. The charge-then-record pattern inside a DB transaction does not prevent double charges on gateway success with DB failure, and the payment client retries without an idempotency key, making duplicate charges possible on timeout. These are not implementation details — they are fundamental architectural flaws in a payment-critical path
