Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the boundary between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data safely.

---

## Critical Issues

### 1. Payment charge is inside the DB transaction — but the DB cannot roll it back

This is the most serious bug. The comment in the code even acknowledges the goal:

```python
# Crash between charge and record cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The claim is false. If `self.payment.charge()` succeeds but the DB transaction rolls back (deadlock, connection drop, constraint violation, application crash), you have charged the customer with no record of it. The charge is a non-transactional external side effect — wrapping it in a DB transaction provides zero rollback protection for it.

The correct pattern for at-most-once semantics is:

1. Write an **idempotency record** to the DB first, within a transaction, before touching the payment gateway.
2. Charge the gateway using the idempotency key so the gateway itself is idempotent on retry.
3. Update the record to `COMPLETED` in a second transaction.

This way a crash before step 1 means nothing happened, and a crash between steps 2 and 3 is recoverable by a reconciliation job that checks for `PENDING` records and queries the gateway.

---

### 2. Timeout on payment gateway can produce an undetected charge

In `payment_client.py`, a `Timeout` exception is retried up to `MAX_RETRIES` times:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge failed — the request may have been received and processed by the gateway. Retrying without an idempotency key sent to the gateway can result in multiple charges for the same order. The gateway must be called with a stable, per-order idempotency key (e.g., the `order_id`) on every attempt so duplicate requests are deduplicated server-side.

## Significant Issues

### 3. Lock TTL does not account for retry duration

```python
ORDER_LOCK_TTL = 60   # seconds
# payment client: MAX_RETRIES=3, RETRY_BACKOFF=5, REQUEST_TIMEOUT=30
# worst case: 3 * (30 + 5) = 105 seconds
```

The payment client can take up to ~105 seconds in the worst case. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still mid-charge, defeating the entire concurrency protection. Either raise the TTL to safely exceed the maximum payment duration, or implement lock extension (Redlock-style keepalive).

### 4. `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their output. If `payment_token` fails validation, its value may appear in logs. Payment tokens are sensitive — at minimum, scrub or redact the `payment_token` field from the error before logging, or log only field names and error types.

### 5. `charge_id` missing from the already-processed idempotency response

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id not returned
    )
```

Callers retrying a previously completed order have no way to retrieve the `charge_id` without a separate lookup. Return `existing["charge_id"]` (or equivalent) in the idempotent response so callers can confirm what was charged.

## Minor Issues

### 6. No exception handling for unexpected errors inside the lock

If `self.db.get_charge()`, `self.payment.charge()`, or the DB transaction raises an unexpected exception, it propagates out of `_acquire_lock`'s `try` block and the `finally` clause correctly releases the lock — that part is fine. However, the caller gets an unhandled exception rather than an `OrderResult`. Add a broad `except Exception` at the `process()` level to return a structured failure result and log the traceback.

### 7. `PaymentClient` has no instance state but is not a static class or module-level function

Minor style issue — `PaymentClient` holds no instance state. Either accept injected configuration (base URL, timeout, credentials) via `__init__` to make it properly configurable and testable, or make the methods static. As written, the hardcoded `CHARGE_ENDPOINT` makes environment-specific deployment (staging vs. production) awkward.

### 8. Tests do not cover the payment timeout/retry path or lock TTL expiry

The test for lock contention is present and correct. Missing coverage:
- Payment gateway timeout triggers retry and eventually raises `PaymentError`
- `PaymentError` from the payment client is handled gracefully by `process()` (currently it would be an unhandled exception — see issue 6)
- Idempotency response includes the existing `charge_id`

---

## What Is Done Well

- `SET NX EX` is the correct atomic acquisition primitive — no race between SET and EXPIRE.
- The Lua script for token-verified release is correct and necessary.
- Pydantic pattern enforcement on `order_id` is appropriate.
- The retry backoff in the lock acquisition path is reasonable.
- Structured logging with key=value pairs throughout is good practice.
- The test for Lua script usage (`redis_client.delete.assert_not_called()`) is a nice explicit guard.
