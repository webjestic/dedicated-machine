Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this touches production.

---

## Critical Issues

### 1. Payment charge occurs inside a DB transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect and the claim is dangerous.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an HTTP call to an external payment gateway. It is **not** enrolled in the database transaction. If the process crashes, is killed, or loses network after the gateway returns a successful `charge_id` but before `record_charge` commits, you have a real charge with no record of it. The DB transaction only protects the two DB writes against each other — it provides zero rollback capability for the external charge.

The correct pattern here is a two-phase approach:
1. Charge the payment gateway and persist the `charge_id` to a durable "pending charges" table **before** marking the order complete.
2. Use an idempotency key on the gateway call so retries do not double-charge.
3. A reconciliation/recovery job handles the gap between a recorded pending charge and a missing completion record.

Wrapping an external side effect in a DB transaction gives false safety and will mislead future maintainers.

---

### 2. Timeout race on the Redis lock creates a TOCTOU window

```python
ORDER_LOCK_TTL = 60  # seconds
```

If the payment gateway is slow (the client allows up to `30s * 3 attempts + 5s * 2 backoffs = 100s`), the Redis lock TTL of 60 seconds can expire while a worker is still mid-charge. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. This is exactly the double-charge scenario the lock is supposed to prevent.

The lock TTL must be greater than the maximum possible time spent inside the lock, with margin. Either:
- Increase `ORDER_LOCK_TTL` to exceed the worst-case payment client runtime (e.g., 120–150s), or
- Implement lock extension (Redlock-style `PEXPIRE` refresh) from within the locked section, or
- Reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` on the payment client so the math works.

As written, the two constants are in different files with no documented relationship, and the unsafe combination will not be caught at review time in the future.

## Significant Issues

### 3. Unhandled exceptions inside the lock silently swallow errors

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, DB exceptions, and any unexpected exception raised inside the lock are not caught. They will propagate up through `_acquire_lock`'s `finally` block (which correctly releases the lock), but then bubble out of `process()` entirely — the caller receives an unhandled exception rather than an `OrderResult`. The method signature promises `OrderResult` but does not deliver it on all paths.

```python
# Add at minimum:
except (PaymentError, SomeDatabaseError) as e:
    logger.error("Order processing failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Processing failed, please retry")
```

---

### 4. `payment_token` is logged indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value (a payment credential) may appear in log output. Sanitize the error before logging or log only field names.

---

### 5. `requests` in `PaymentClient` has no connection pooling or session reuse

```python
response = requests.post(self.CHARGE_ENDPOINT, ...)
```

Using the module-level `requests.post` creates a new TCP connection on every call, including retries. For a service under load this is wasteful and adds latency. Use a `requests.Session` instance, ideally injected or created once at construction time, so connections are reused and retry logic can be centralized with `urllib3.Retry` if desired.

---

### 6. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

A hardcoded internal URL in source is an ops and portability problem. This should come from configuration (environment variable, config file, or injected dependency) so it can differ between environments without a code change.

## Test Coverage Gaps

- **No test for the lock-expiry / timeout race** described in issue #2.
- **No test for `PaymentError` propagation** — the current test suite would pass even if `PaymentError` crashes the service.
- **No test for the crash-between-charge-and-record scenario** — this is the hardest case to test but a comment explaining the known limitation would be acceptable in the interim.
- `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` but this is testing implementation detail of the mock, not behavior. It would pass even if the Lua script was never registered. The assertion on `_release_script` is better but should also verify the correct `keys` and `args` were passed.

## Minor Notes

- `LOCK_RETRY_ATTEMPTS = 3` with `backoff = 0.5 * (attempt + 1)` gives a maximum wait of 3 seconds before giving up. For a 60-second lock TTL this is very aggressive — a worker that acquired the lock and is mid-payment will hold it for up to 100 seconds, so retrying for 3 seconds and failing is nearly always going to fail. Consider whether the retry loop adds value at all here, or whether it should be removed in favor of a single attempt with a clear error.
- The `order_id` field in `OrderResult` is `Optional[str]` but is omitted on the `already processed` path — callers cannot identify which order was already processed without it. This appears to be an oversight.
