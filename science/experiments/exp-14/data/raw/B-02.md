Decision: Request Changes

Summary: The distributed lock design contains a critical flaw that directly undermines the at-most-once payment guarantee it claims to provide: the payment charge call is made *inside* the database transaction but *outside* any rollback-safe boundary, meaning a timeout or crash after `payment.charge()` succeeds but before `db.record_charge()` commits will produce an unrecorded, unrecoverable charge ŌĆö exactly the double-charge scenario the architecture is supposed to prevent. Additionally, the lock TTL of 60 seconds is dangerously short relative to the combined worst-case latency of `MAX_RETRIES ├Ś (REQUEST_TIMEOUT + RETRY_BACKOFF)` in the payment client (up to 105 seconds), meaning the lock can expire and be acquired by a second worker while the first is still mid-charge. Several secondary issues compound these risks and must be addressed before this is safe to merge.

---

## Critical Issues

### 1. Lock TTL Is Shorter Than Maximum Payment Client Latency

This is the most severe flaw. The lock TTL and the payment client's worst-case execution time are mathematically incompatible.

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30   # per attempt
MAX_RETRIES    = 3
RETRY_BACKOFF  = 5     # between retries
```

Worst-case payment duration:

```
3 attempts ├Ś 30s timeout = 90s
2 gaps     ├Ś 5s backoff  = 10s
Total                    = 100s
```

The lock expires at **60 seconds**. A second worker can acquire the lock at **60s + ╬Ą**, pass the idempotency check (the charge hasn't been recorded yet), and issue a second charge while the first worker is still on attempt 2 or 3. The entire distributed lock guarantee collapses under this condition. The TTL must be set to a value strictly greater than the maximum possible payment client execution time, with margin. Either raise `ORDER_LOCK_TTL` to at minimum `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + safety_margin`, or enforce that the payment client's ceiling is always below the lock TTL at the type/config level so they cannot drift apart silently.

---

### 2. Payment Charge Occurs Inside the Database Transaction ŌĆö Atomicity Is Illusory

The PR comment claims:

```python
# Charge and record atomically ŌĆö crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ŌåÉ external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is incorrect. A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a third-party payment gateway. If the charge succeeds and the network call returns `charge_id`, but the process crashes, the DB connection drops, or the transaction rolls back for any reason before `record_charge` commits, the charge has been collected from the customer with no record in the system. This is an unrecoverable financial inconsistency.

The correct pattern for this class of problem is an **outbox/saga pattern** or **two-phase approach**:

1. Write a `pending_charge` record to the DB *before* calling the payment gateway (within a transaction).
2. Call the payment gateway.
3. Update the record to `completed` with the `charge_id` (within a second transaction).
4. A reconciliation job detects `pending` records older than a threshold and resolves them against the gateway.

Alternatively, if the payment gateway supports idempotency keys, pass a deterministic idempotency key (e.g., derived from `order_id`) so that retrying a timed-out charge returns the original `charge_id` rather than issuing a new charge, and then the DB write can be retried safely.

### 3. `PaymentError` Is Not Caught in `process()` ŌĆö Unhandled Exception Leaks

```python
# order_processor.py ŌĆö process()
with self.db.transaction():
    charge_id = self.payment.charge(...)  # can raise PaymentError
    ...
```

`PaymentError` is defined in `payment_client.py` but is never caught in `order_processor.py`. A payment failure after retries exhausted will raise an unhandled exception that propagates out of `process()`. The caller receives a raw exception instead of an `OrderResult`. The method signature promises `OrderResult` in all cases. This must be caught and returned as a structured failure result.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 4. Lock Release Silently Swallows Errors in `finally`

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception is propagated out of the `finally` block, which will **suppress any exception already in flight** from the `try` block. This means a payment failure exception could be silently replaced by a Redis connection error, making the failure invisible and undiagnosable.

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.warning("Lock token mismatch or already expired order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s ŌĆö lock will expire naturally", order_id)
```

Also note: the return value of `_release_script` is currently discarded. A return value of `0` means the token didn't match ŌĆö this is a detectable anomaly (possible lock theft or TTL expiry mid-flight) that should at minimum be logged as a warning.

## Significant Issues

### 5. Retry Backoff in Lock Acquisition Is Blocking

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

In any async or thread-pool-based deployment, `time.sleep()` in the lock retry loop blocks the worker thread entirely. If this runs in an async framework (FastAPI, etc.), this must use `asyncio.sleep()`. Even in a synchronous context, the blocking sleep should be documented as a known constraint. The current code gives no indication of its threading model.

---

### 6. Payment Token Logged or Exposed in Tracebacks

```python
class OrderRequest(BaseModel):
    payment_token: str = Field(..., min_length=1, max_length=256)
```

`payment_token` is stored on the `OrderRequest` model with no masking. If a `ValidationError` or unhandled exception causes the model to appear in a traceback or log, the raw payment token will be emitted to the log sink. Pydantic v2 supports `__repr__` overrides or `Secret` types. At minimum, `payment_token` should use `pydantic.SecretStr` or have its `__repr__` suppressed.

---

### 7. `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has a strict alphanumeric pattern enforced. `customer_id` does not. If `customer_id` is used in any DB query, log line, or downstream system, it is an injection surface. Apply the same or equivalent pattern constraint.

---

### 8. Test Coverage Gaps

The test suite does not cover:

- **Lock TTL expiry mid-payment**: No test simulates the lock expiring while the payment is in flight, which is the primary failure mode identified above.
- **`PaymentError` propagation**: No test verifies behavior when `payment.charge()` raises `PaymentError`. Given the unhandled exception issue in point 3, this is a required test.
- **DB transaction rollback**: No test verifies behavior when `record_charge` or `mark_order_complete` raises after a successful charge.
- **Lock release return value `0`**: No test covers the case where the Lua script returns `0` (token mismatch / expired lock).
- **Retry exhaustion path in `PaymentClient`**: The retry loop in `PaymentClient` is not tested at all.

---

### 9. `CHARGE_ENDPOINT` Is Hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoint in source code is an operational and security concern. This must be injected via configuration or environment variable. It cannot be changed without a code deployment, and it will differ across environments (staging, production, DR).
