Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw between the payment charge and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge happens inside a DB transaction — this does not make it atomic

The comment says *"crash between charge and record cannot produce an unrecorded charge"* but that claim is false. The `payment.charge()` call is a **network call to an external system** inside a `db.transaction()` block. A DB transaction provides atomicity over database operations only. It has no control over what the payment gateway has already done.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side-effect, NOT rolled back on DB failure
    self.db.record_charge(...)             # ← if this raises, charge_id is lost forever
    self.db.mark_order_complete(...)
```

Failure scenarios that produce an unrecorded charge:
- `record_charge` raises a DB exception → transaction rolls back, but the charge already settled
- Process crashes after `payment.charge()` returns but before the transaction commits
- DB connection drops mid-transaction

**Required fix:** Charge first, capture the `charge_id`, then open the DB transaction only for the record-keeping. The idempotency check + Redis lock already prevent double-charging. The DB write is the only thing that needs to be atomic with itself:

```python
# Outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Now persist the result — this is the only thing the DB transaction needs to cover
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even with this fix, you need a reconciliation job or a "pending charge" record written *before* calling the gateway, so that a crash after charge but before DB commit can be detected and resolved. That is a broader design concern but should be acknowledged in the architecture docs.

---

### 2. Timeout on payment retry can exceed the Redis lock TTL

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire while the payment is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge.

```
Worker A: acquires lock (t=0), calls payment.charge() → timeout at t=30
           retries → timeout at t=65 (lock expired at t=60)
Worker B: acquires lock (t=61), idempotency check passes (no record yet), charges again
Worker A: payment succeeds at t=95, records charge_id_A
Worker B: payment succeeds at t=100, records charge_id_B  ← double charge
```

**Required fix:** Either extend `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime (at least 120s with margin), or implement lock extension (Redlock-style keepalive), or reduce `PaymentClient` timeouts to fit within the TTL budget. The values must be documented as a coupled constraint.

## Moderate Issues

### 3. `payment_token` is logged at INFO level in `payment_client.py`

The `charge_id` is logged on success, which is fine. But if you add any debug logging or exception messages that include the `response` body or the request payload, the raw payment token will appear in logs. The token is also passed through as a plain string with no masking anywhere in the call chain. Recommend adding a `__repr__`/`__str__` guard on sensitive fields or at minimum a note in the logging policy. This is a PCI-DSS concern.

### 4. `LockError` is silently swallowed — no observability

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

There is no log line here. Lock contention at scale is a meaningful operational signal. The warning inside `_acquire_lock` fires per attempt, but the final failure is silent at the `process()` level. Add a `logger.warning` with `order_id` when returning the contention result.

### 5. `payment_token` field max length of 256 is not validated against the actual gateway's token format

The Pydantic model validates `order_id` with a strict regex but `payment_token` only has a length bound. If the gateway has a known token format (e.g., `tok_[a-zA-Z0-9]+`), add a pattern. If it does not, document why the validation is intentionally loose.

## Minor Issues

### 6. Test for lock release does not verify the correct key and token are passed

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This only asserts the script was called. It does not verify `keys=["order:lock:order-001"]` or that the `args` token matches what was set. A refactor that changes the key format or passes the wrong token would not be caught.

### 7. No test for the payment failure / `PaymentError` path

There is no test covering what happens when `payment.charge()` raises `PaymentError`. The processor will propagate an unhandled exception to the caller. If that is intentional, document it. If not, add handling and a test.

### 8. `requests` is used without a session in `PaymentClient`

Each call to `charge()` creates a new TCP connection. For a service under load this is wasteful. Use a `requests.Session` with connection pooling, and configure it at construction time so it can be injected in tests.
