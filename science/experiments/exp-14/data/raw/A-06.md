Decision: Request Changes

Summary: The architecture is directionally sound — the Lua-based token-verified lock release, Pydantic input validation, and idempotency check are all correct in principle. However, there are several non-trivial correctness and security flaws that must be resolved before this can be merged, the most critical being a charge-before-commit race that breaks the atomicity guarantee the PR explicitly claims to provide, and a timeout-retry hazard in the payment client that can produce double charges. Additional concerns around lock TTL sizing, missing exception handling, and test fidelity are documented below.

---

## Critical Flaws

### 1. Payment Charge Occurs Inside the Database Transaction — Atomicity Claim Is False

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary states:

> *"crash between charge and record cannot produce an unrecorded charge"*

This claim is incorrect. A database transaction provides atomicity over database operations only. `self.payment.charge(...)` is an external HTTP call to a payment gateway. If the charge succeeds but the subsequent `db.record_charge` or `db.mark_order_complete` raises an exception, the database transaction rolls back — but the charge has already been captured by the payment gateway. The money is gone with no local record. This is precisely the double-charge / lost-charge scenario the lock is supposed to prevent.

**Required fix:** Charge must happen *outside* and *after* the database transaction, or — the standard pattern for this class of problem — use an outbox/saga pattern:

1. Within the DB transaction, write a `pending_charge` record (idempotency key = `order_id`).
2. Commit.
3. Call `payment.charge(idempotency_key=order_id, ...)`.
4. Within a second DB transaction, promote the record to `complete`.

Alternatively, at minimum, restructure so the charge is issued *after* a successful pre-commit of intent, and reconciliation handles the gap. The current structure provides a false sense of safety.

### 2. Payment Client Retries on Timeout — Guaranteed Double-Charge Vector

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` on a POST to a payment gateway does **not** mean the charge did not occur. The request may have been received and processed by the gateway; the response simply did not arrive within the window. Retrying unconditionally on timeout will submit the same charge a second (or third) time. This is a textbook double-charge bug.

**Required fix:** The payment gateway API must support idempotency keys. Pass a stable, per-order idempotency key (e.g., derived from `order_id`) in the request headers or body on every attempt:

```python
requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": idempotency_key},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, retrying on timeout is unsafe regardless of any locking strategy.

## Significant Architectural Concerns

### 3. Lock TTL Is Structurally Undersized for Its Own Retry + Payment Timeout Budget

**File:** `services/order_processor.py`

```python
ORDER_LOCK_TTL = 60  # seconds
```

**File:** `services/payment_client.py`

```python
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment client duration:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. The payment call alone can outlive the lock by 40 seconds. A second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still mid-flight. The distributed lock provides no protection in this scenario.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed the maximum payment client duration (≥ 120s with margin), or implement lock extension (Redlock-style heartbeat) for long-held locks. The TTL and payment client timeout budget must be explicitly co-designed and documented.

---

### 4. Unhandled Exceptions Escape `process()` — No Catch-All

**File:** `services/order_processor.py`

`PaymentError`, `db` exceptions (connection loss, constraint violation, deadlock), and Redis exceptions during lock acquisition are all unhandled. They will propagate as uncaught exceptions to whatever is calling `process()`. For a service boundary function that returns `OrderResult`, this is an inconsistent contract. Callers expecting a structured result will get a raw exception instead.

**Required fix:** Add a catch-all handler (or at minimum catch `PaymentError` and database exceptions explicitly) and return a structured `OrderResult(processed=False, message=...)` with appropriate logging. Distinguish between transient errors (retriable by the caller) and permanent errors in the result model if needed.

---

### 5. Redis Unavailability Raises Unhandled Exception — Fails Open or Crashes

**File:** `services/order_processor.py`, `_acquire_lock`

If `self.redis.set(...)` raises `redis.exceptions.ConnectionError` or `redis.exceptions.RedisError`, the exception propagates uncaught through `_acquire_lock` and then through `process()`. Depending on the caller's error handling, this could either crash the worker or — if the caller silently swallows exceptions — fail open and process the order without a lock.

**Required fix:** Catch Redis exceptions explicitly in `_acquire_lock`. Treat Redis unavailability as a `LockError` (fail closed — do not process without a lock):

```python
except redis.exceptions.RedisError as e:
    raise LockError(f"Redis unavailable: {e}") from e
```

Failing open on a locking failure in a payment processor is a critical correctness violation.

## Moderate Issues

### 6. `payment_token` Logged Indirectly via Pydantic Validation Error

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Payment tokens are sensitive credentials.

**Required fix:** Either sanitize the validation error before logging (strip field values), or log only field names and error types, not values.

---

### 7. Lock Retry Backoff Is Linear and Blocks the Worker Thread

**File:** `services/order_processor.py`

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep` blocks the worker thread. In a synchronous worker pool this reduces throughput under contention. The backoff is also linear, not exponential, and has no jitter — multiple workers contending on the same lock will retry in lockstep.

**Recommended fix:** Use exponential backoff with jitter:

```python
backoff = min(0.1 * (2 ** attempt) + random.uniform(0, 0.1), 2.0)
```

This is not a blocking issue for merge but should be addressed before production load.

## Test Coverage Gaps

### 8. No Test for Lock TTL Expiry During Payment

There is no test verifying behavior when the lock expires mid-payment (i.e., TTL elapses before `charge()` returns). This is the scenario identified in issue #3 above. A test should simulate a slow payment call and assert that the system does not silently produce an unprotected charge.

### 9. No Test for Redis Failure During Lock Acquisition

There is no test for `redis.exceptions.ConnectionError` during `set()`. Given the fail-open risk identified in issue #5, this path must be explicitly tested to confirm it returns a safe `OrderResult` rather than raising or silently bypassing the lock.

### 10. `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

```python
processor._release_script.assert_called_once()
```

This test confirms the script was called but does not assert that the correct token was passed as `ARGV[1]`. A regression that passes a hardcoded or empty token would pass this test. Assert on `call_args`:

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] == <captured_token>
```

## Minor Notes

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants but not configurable via constructor or environment. For a production service these should be injectable.
- `CHARGE_ENDPOINT` is hardcoded in `PaymentClient`. Should be injected or pulled from configuration, not a class attribute.
- No structured log correlation between `order_id` and `charge_id` at the point of payment initiation — only at completion. A log entry at charge attempt start with `order_id` aids incident investigation.
- The `OrderResult.order_id` field is `Optional` but is omitted on several failure paths where it is known. Populate it consistently for observability.
