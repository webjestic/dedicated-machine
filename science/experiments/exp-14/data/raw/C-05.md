Decision: Request Changes

Summary: The core locking and idempotency architecture is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check pattern are all sound. However, there is a critical atomicity flaw that breaks the at-most-once payment guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this touches production.

---

## Critical Issue

### Payment call sits outside the database transaction — the guarantee is false

The PR description states: *"charges the payment method and records the transaction within a single atomic database transaction."* The code does not do this. The `payment.charge()` call is **inside** `with self.db.transaction()`, but a database transaction cannot roll back a network call to an external payment gateway. These are two independent systems with no two-phase commit.

The real failure scenario this creates:

1. Worker acquires lock.
2. `payment.charge()` succeeds — money is taken.
3. `self.db.record_charge()` raises an exception (DB timeout, constraint violation, anything).
4. The `with self.db.transaction()` block rolls back.
5. The charge is now unrecorded. The idempotency check on retry finds nothing and charges again.

```python
# Current code — the comment is misleading and the guarantee does not hold
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect, NOT rollback-safe
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The correct pattern is to record a **pending** state before charging, then confirm or void after:

```python
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = None
try:
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
except PaymentError:
    with self.db.transaction():
        self.db.mark_charge_failed(req.order_id)
    raise

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This way the idempotency check finds the pending record on retry and can either wait, void, or surface it for reconciliation — it never silently double-charges.

## High Severity

### Redis failure silently drops the lock release

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If the Lua script call raises (Redis connection lost, READONLY replica failover, etc.), the exception is swallowed by `finally` and the lock just holds until TTL expiry. That is acceptable for the lock itself, but the exception is also silently lost, which means the caller gets back a successful `OrderResult` with no indication that cleanup failed. At minimum, log the exception. Better: catch and re-raise after logging so the caller knows the state is uncertain.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s — will expire in %ds",
                         order_id, self.ORDER_LOCK_TTL)
```

### `payment_token` is logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some versions. If `payment_token` fails validation, its value (a raw payment credential) may appear in log output. Scrub or summarize validation errors before logging, or log only field names and error types.

### `PaymentClient` has no authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, mTLS configuration, or API key. Even for an internal endpoint, unauthenticated payment calls are not acceptable in a government contracting context. The client needs a credential injection mechanism (constructor parameter, environment variable, secrets manager lookup) and the credential must be present in every request.

## Medium Severity

### Lock retry backoff is synchronous and blocks the worker thread

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

In a threaded or async worker pool this blocks the entire thread for up to 3 seconds. Use async sleep if the surrounding framework is async, or document clearly that this is synchronous-only and unsuitable for async deployment.

### `ORDER_LOCK_TTL = 60` does not account for retry duration

With `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5`, the payment client can take up to ~90 seconds (3 attempts × 30s timeout + 2 × 5s backoff). The lock TTL of 60 seconds can expire before the payment completes, allowing a second worker to acquire the lock while the first is still mid-charge. Either extend the TTL to cover the worst-case payment duration, or implement lock renewal (a periodic background heartbeat that resets the TTL while the payment is in flight).

### `payment_token` field length is insufficient for some token formats

`max_length=256` may be too short depending on the payment gateway. Some gateways issue JWTs or opaque tokens that exceed 256 characters. Verify against the actual gateway's token format spec before merging.

## Low Severity / Nits

### Tests do not cover the critical failure path

There is no test for the scenario where `db.record_charge` raises after `payment.charge` succeeds. This is exactly the failure mode described above and it must have test coverage.

### `charge_id` is referenced outside the transaction block without a guard

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)
    ...

logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

If the transaction block raises after `charge_id` is assigned, the variable is still in scope and the log line may or may not execute depending on where the exception propagates. Minor, but worth cleaning up once the atomicity issue is resolved.

### No structured exception handling around the idempotency DB read

```python
existing = self.db.get_charge(req.order_id)
```

A database error here raises an unhandled exception that escapes `process()`. The caller receives an uncaught exception rather than an `OrderResult`. All DB calls should be wrapped and return a structured result.
