Decision: Request Changes

Summary: The architecture demonstrates solid fundamentals — atomic `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical flaw that undermines the entire at-most-once guarantee: `payment.charge()` is called *inside* the database transaction, meaning a successful charge that precedes a database commit failure will produce an unrecorded, unrecoverable charge with no compensation path. Several secondary issues compound this concern and must be resolved before this code handles real payment flows.

---

## Critical Issues

### 1. Payment charge inside DB transaction violates at-most-once semantics

This is the most dangerous flaw in the PR. The comment in the code claims the opposite of what is true:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a payment gateway. If the charge succeeds but the subsequent `db.record_charge()` or `db.mark_order_complete()` raises, or the DB commit fails, the transaction rolls back — but the charge has already been submitted and cannot be rolled back. The customer has been charged with no record in your system.

The comment `"crash between charge and record cannot produce an unrecorded charge"` is factually incorrect and dangerous. This is a classic two-phase commit problem across a non-transactional external resource.

**Required fix:** Restructure to a pre-commit staging pattern:

```python
# Stage intent before charging
self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Charge externally — outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Confirm atomically only after external success
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

The pending record also serves as a reconciliation artifact: any `PENDING` row without a corresponding `charge_id` after TTL expiry indicates a charge that needs investigation. This is the industry-standard approach (Stripe's own idempotency key design follows this pattern).

### 2. Lock TTL does not account for payment retry duration

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment client execution time: `(30s × 3) + (5s × 2) = 100 seconds`. The lock TTL is 60 seconds. Under worst-case conditions the lock expires while the payment is still in-flight, a second worker acquires the lock, passes the idempotency check (no charge recorded yet), and issues a duplicate charge. This silently defeats the entire purpose of the lock.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at minimum `120`–`150` seconds with a documented derivation, or implement lock extension (e.g., Redlock's `pexpire` refresh) before the TTL lapses during a retry cycle. The TTL value must be derived from the maximum observable payment client latency, not chosen arbitrarily.

## Significant Issues

### 3. `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

`PaymentError` and all other unexpected exceptions propagate out of `process()` entirely unhandled. Only `LockError` is caught. A `PaymentError`, `requests.ConnectionError`, or any DB exception will bubble up as an unhandled exception to the caller with no `OrderResult` returned, breaking the stated contract of the method. The caller has no structured way to distinguish a payment decline from a network partition from a DB failure.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 4. Timeout on charge creates an ambiguous charge state

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A gateway timeout does not mean the charge failed — it means the response was not received within the timeout window. The charge may have been processed. Retrying after a timeout without a gateway-side idempotency key risks issuing a duplicate charge at the payment provider level.

**Required fix:** Pass a stable, deterministic idempotency key derived from `order_id` to the payment gateway on every attempt:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": f"order-{order_id}"},
    timeout=self.REQUEST_TIMEOUT,
)
```

This ensures the gateway deduplicates retries server-side regardless of whether the original request was received.

### 5. `payment_token` echoed in `OrderResult` — not present here, but `charge_id` exposure

`charge_id` is returned directly to the caller in `OrderResult`. Depending on the payment provider, `charge_id` values can be sensitive. Confirm that returning `charge_id` to the API surface is intentional and that downstream logging does not capture it in plaintext. The `payment_token` field in `OrderRequest` should be explicitly excluded from any serialization or logging path.

## Minor Issues

### 6. Lock retry backoff is linear and too short

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Linear backoff with no jitter causes retry storms under high concurrency. Use truncated exponential backoff with jitter:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

### 7. Test suite does not cover the critical failure path

There is no test for the scenario where `payment.charge()` succeeds but `db.record_charge()` raises. This is precisely the scenario that exposes the critical flaw in issue #1. A test like the following must be present before merge:

```python
def test_charge_success_db_failure_does_not_lose_charge(processor, db, payment):
    db.record_charge.side_effect = Exception("DB unavailable")
    # Must assert that the pending charge record exists for reconciliation
    # and that the result is deterministic
    result = processor.process("order-001", "cust-001", 5000, "tok_visa")
    assert result.processed is False
    # Verify reconciliation artifact exists
    db.create_pending_charge.assert_called_once()
```

### 8. `requests` used without session or connection pooling

Each call to `payment.charge()` constructs a new TCP connection. Under load this is inefficient and increases latency variance. Use a `requests.Session` initialized in `__init__` with appropriate connection pool sizing.

| # | Severity | Issue |
|---|----------|-------|
| 1 | **Critical** | External charge inside DB transaction — unrecoverable unrecorded charges possible |
| 2 | **Critical** | Lock TTL shorter than max payment retry window — duplicate charge window exists |
| 3 | **Significant** | `PaymentError` and general exceptions unhandled — broken return contract |
| 4 | **Significant** | Timeout retries without idempotency key — gateway-level duplicate charge risk |
| 5 | **Minor** | Confirm `charge_id` exposure policy; protect `payment_token` from log serialization |
| 6 | **Minor** | Linear backoff without jitter — retry storm risk under concurrency |
| 7 | **Minor** | Missing test for charge-success/DB-failure path |
| 8 | **Minor** | No HTTP session reuse in payment client |

The structural intent here is sound and the author clearly understands distributed locking primitives. Issues 1 and 2 are not edge cases — they are reachable under normal operating conditions and will produce real financial harm. Do not merge until both are resolved with accompanying tests.
