Decision: Request Changes

Summary: The architecture demonstrates solid foundational instincts — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there is a critical atomicity violation at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, along with several secondary issues that must be resolved before this touches production.

---

## Critical Issues

### 1. Payment charge occurs *inside* the database transaction — this is the core flaw

```python
# services/order_processor.py, lines within process()
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary claims "crash between charge and record cannot produce an unrecorded charge," but this is precisely backwards from what the code achieves. Wrapping an external HTTP call inside a database transaction does **not** make them atomic. What it actually does is:

- Holds an open database transaction (and likely a connection from your pool) for the entire duration of the payment gateway round-trip — up to `REQUEST_TIMEOUT = 30s` per attempt, times `MAX_RETRIES = 3`, plus `RETRY_BACKOFF = 5s` between attempts. Under any meaningful concurrency this will exhaust your connection pool.
- If the payment gateway call succeeds but the subsequent `record_charge` or `mark_order_complete` raises (DB constraint violation, network partition to DB, etc.), the transaction rolls back. The charge has been captured at the gateway. You have charged the customer and have no record of it. This is the exact failure mode the design claims to prevent.
- The database transaction provides no rollback capability over an already-submitted payment gateway charge. Rolling back the DB transaction does not uncapture the charge.

**Required fix:** Charge outside and before the transaction. Only open the DB transaction after you have a `charge_id` in hand.

```python
# Correct structure
existing = self.db.get_charge(req.order_id)
if existing:
    return ...

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only now open the DB transaction — charge_id is known, no network I/O inside
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This still leaves the window where the charge succeeds but the DB write fails, which is an inherent distributed systems problem. That window must be handled by a reconciliation path (see item 2).

---

### 2. No reconciliation path for the charge-succeeded / record-failed window

Even with the corrected structure above, a crash or DB failure after `payment.charge()` returns but before `db.record_charge()` commits leaves a captured charge with no local record. The idempotency check will find nothing on retry and charge again.

This is not a theoretical edge case — it is a guaranteed eventual occurrence at scale.

**Required:** At minimum, a reconciliation job or a pre-insert "pending charge" record written before the gateway call, so that any recovery path can detect the in-flight state and either confirm or void the charge. The payment gateway should also be queried for idempotency keys. This is a design-level gap that needs an explicit decision and implementation before this goes to production.

---

### 3. Timeout budget arithmetic is broken relative to the lock TTL

```python
# order_processor.py
ORDER_LOCK_TTL = 60        # seconds

# payment_client.py
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client execution time: `(30 × 3) + (5 × 2) = 100 seconds`.

The lock TTL is 60 seconds. The payment client can run for 100 seconds. The lock will expire while the payment is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (no record exists yet), and issue a second charge. The distributed lock provides no protection in this scenario.

**Required:** Either reduce the payment client's maximum execution time to well under 60 seconds, increase the lock TTL with margin, or implement lock extension (Redlock-style keepalive). The TTL must be strictly greater than the maximum possible critical section duration, with margin.

## Significant Issues

### 4. `payment_token` is logged at INFO level on success

```python
# payment_client.py
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is safe, but examine the request construction:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

Depending on your logging configuration and any request-level debug logging enabled on the `requests` library or any middleware, the full request body — including `payment_token` — may be emitted to logs. Additionally, `payment_token` flows through `OrderRequest` and into `process()` without any scrubbing. Confirm that no logging path captures this value. PCI-DSS scope requires that payment tokens never appear in logs. This needs an explicit audit and a note in the PR.

---

### 5. Lock retry backoff is linear and blocking — it will cascade under load

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
    time.sleep(backoff)
```

`time.sleep()` blocks the worker thread. Under contention with many workers, this produces a thundering herd on lock retry. Jitter must be added. If this service is async or runs under a thread pool, sleeping the thread is also a resource concern. Recommend `random.uniform(0, backoff)` at minimum.

---

### 6. `PaymentError` is not caught in `process()`

```python
# order_processor.py — process() exception handling
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` and any other exception from `self.payment.charge()` or the DB calls propagate uncaught out of `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract the return type implies and may expose internal error detail to callers depending on how exceptions are surfaced upstream.

---

### 7. Redis connection failure is not handled

If `self.redis.set()` raises (connection refused, timeout, cluster failover), the exception propagates uncaught. Given that Redis is the gating mechanism for all order processing, a Redis outage silently takes down the entire order pipeline with no structured error response. At minimum, catch `redis.RedisError` and return a structured result or raise a domain-specific exception.

---

## Test Coverage Gaps

### 8. The critical failure mode is not tested

There is no test for the scenario where `payment.charge()` succeeds but `db.record_charge()` raises. This is the most important failure mode in the entire system and the test suite does not cover it.

### 9. Lock TTL expiry during processing is not tested

No test simulates the lock expiring mid-execution. Given that this is identified as a real risk above, it should be a documented test case even if the current implementation cannot fully prevent it.

### 10. `PaymentError` propagation is not tested

No test verifies behavior when the payment client raises. Given that `PaymentError` is currently uncaught, this test would immediately surface issue 6.

---

## Minor Issues

**11. `ORDER_LOCK_TTL` and payment client constants are not co-located or cross-referenced.** They are defined in separate files with no comment linking them. The next developer to adjust `REQUEST_TIMEOUT` will not see the lock TTL constraint. These should be explicitly documented as coupled.

**12. `response.json()["charge_id"]` has no error handling.** If the gateway returns a 200 with a malformed or missing `charge_id`, this raises `KeyError` with no context. Add explicit key access with a fallback error.

**13. The `finally` block in `_acquire_lock` runs even when the lock was never acquired.** If `acquired` is `False` and `LockError` is raised before `yield`, the `finally` still calls `_release_script`. In the current flow this is harmless because the exception is raised before `yield`, meaning the `finally` is never reached from inside the `with` block — but this is subtle and worth a comment or an explicit guard (`if acquired`) to make the intent clear to future maintainers.
