Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical atomicity flaw at the boundary between the external payment call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issue

### 1. Payment call is inside `db.transaction()` — this does not make it atomic

```python
# Current code
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is false. A DB transaction cannot wrap an external HTTP call. If `payment.charge()` succeeds but the process crashes before `record_charge()` commits, you have charged the customer with no record. Worse, holding an open DB transaction for the full duration of a payment gateway round-trip (up to `REQUEST_TIMEOUT * MAX_RETRIES = 90s`) will exhaust your connection pool under any real load.

**Required fix:** Charge outside the transaction, then record inside it. Use the idempotency check as your recovery path.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

with self.db.transaction():
    # Re-check inside transaction to guard against a race on recovery
    if self.db.get_charge(req.order_id):
        # Charge succeeded on a prior attempt; refund this duplicate or return existing
        self.payment.refund(charge_id)
        return OrderResult(processed=False, order_id=req.order_id, message="Order already processed")
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

The real at-most-once guarantee comes from the DB unique constraint on `order_id` in `record_charge`, not from the Redis lock alone. The lock reduces contention; the DB constraint is the hard safety net. Make sure that constraint exists and is documented.

## Significant Issues

### 2. `PaymentError` is silently swallowed

`PaymentError` (and any other unexpected exception) raised inside `_acquire_lock` is not caught in `process()`. It will propagate as an unhandled exception to the caller with the lock already released, which is fine for the lock, but the caller gets a raw exception instead of an `OrderResult`. The PR claims structured error returns throughout — this is inconsistent.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

Also consider: if `payment.charge()` raises after a successful charge on the gateway side (e.g., response parsing error), you have the same unrecorded-charge problem described above. This is another reason the charge must be outside the transaction and recoverable.

### 3. Timeout on payment call is not enforced at the retry layer

`PaymentClient` retries up to 3 times with a 5-second backoff and a 30-second per-attempt timeout. Worst case: ~105 seconds total. `ORDER_LOCK_TTL` is 60 seconds. The lock can expire while a payment attempt is still in flight, allowing a second worker to acquire the lock and begin the idempotency check before the first worker has recorded its charge. The lock TTL must exceed the maximum possible payment client duration.

```python
ORDER_LOCK_TTL = (PaymentClient.REQUEST_TIMEOUT * PaymentClient.MAX_RETRIES) + (PaymentClient.RETRY_BACKOFF * (PaymentClient.MAX_RETRIES - 1)) + SAFETY_MARGIN
# = (30 * 3) + (5 * 2) + margin = 100 + margin seconds minimum
```

### 4. Timeout on payment call does not account for retry in `requests.post`

`requests` does not retry on timeout by default — that is handled by the loop — but the `Timeout` exception path does not distinguish between a connect timeout and a read timeout. A read timeout means the server *may have received and processed the request*. Retrying on read timeout can cause duplicate charges at the gateway level even if your DB record is clean. Consider using separate connect/read timeouts and only retrying on connect timeout or confirmed 5xx, not on ambiguous read timeouts.

## Minor Issues

### 5. `payment_token` is logged indirectly via `charge_id` — verify no token leakage

The token itself is not logged, which is correct. Confirm that `PaymentClient` does not log the raw `token` field in any error path (it currently does not, but worth a note in a code review for future contributors).

### 6. Test for lock release verifies mechanism, not correctness

`test_lock_release_uses_lua_script` asserts the Lua script was called but does not assert it was called with the correct key and the same token that was set. A future refactor could call the script with wrong arguments and this test would still pass.

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["keys"] == [f"order:lock:order-001"]
# token is internal, but you can at least assert args[0] is a valid UUID string
```

### 7. No test for `PaymentError` propagation

There is no test covering what happens when `payment.charge()` raises. Given the unhandled exception issue in point 2, this gap is hiding a real defect.

### 8. `customer_id` has no pattern constraint

`order_id` has a strict alphanumeric pattern, but `customer_id` only has length bounds. If `customer_id` flows into any query or log aggregation downstream, it should have the same treatment. Apply a consistent validation policy.
