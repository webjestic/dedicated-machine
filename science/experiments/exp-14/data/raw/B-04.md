Decision: Request Changes

Summary: The architecture demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness flaws that must be resolved before this can be approved: a charge-before-commit race that breaks the at-most-once guarantee, a timeout retry window that can produce duplicate charges, and a lock TTL that is dangerously short relative to the actual worst-case execution time. These are not edge cases — they are the exact failure modes this system claims to prevent.

---

## Critical Issues

### 1. `payment.charge()` Is Called Inside `db.transaction()` — This Breaks the At-Most-Once Guarantee

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment reads *"crash between charge and record cannot produce an unrecorded charge"* — this is **incorrect**. A database transaction does not wrap an external HTTP call. The payment gateway is charged the moment `payment.charge()` returns successfully. If the database transaction subsequently fails to commit (deadlock, connection drop, disk error), the charge has already been applied at the gateway but `record_charge` was never persisted. On retry, the idempotency check finds no record and charges again.

**The correct pattern is charge-then-commit, not charge-within-transaction:**

```python
# Charge OUTSIDE the transaction — the gateway call is not transactional
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only AFTER a successful charge response, persist atomically
try:
    with self.db.transaction():
        # Re-check idempotency inside the transaction under a DB-level lock
        # to guard against the window between the outer check and this commit
        if self.db.get_charge_for_update(req.order_id):
            # Charge succeeded but a concurrent worker already recorded it.
            # Refund the charge we just made and return already-processed.
            self.payment.refund(charge_id)
            return OrderResult(processed=False, message="Order already processed")
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
except Exception:
    # DB commit failed after a successful charge — this is the unrecorded charge scenario.
    # Log with full context for manual reconciliation. Do NOT silently swallow.
    logger.critical(
        "UNRECORDED CHARGE: charge_id=%s order_id=%s — manual reconciliation required",
        charge_id, req.order_id,
    )
    raise
```

The unrecorded charge path must also be surfaced to an alerting/reconciliation system, not just logged.

### 2. Timeout Retry in `PaymentClient` Can Produce Duplicate Charges

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ← retries unconditionally
```

A timeout means the request may have been received and processed by the gateway — the response was simply lost in transit. Retrying on timeout without a gateway-side idempotency key will submit a second charge attempt that the gateway may fulfill independently.

**Required fix:** Pass an idempotency key on every charge request, and pass the **same key** on retries for the same logical charge:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},  # same key on every retry
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `idempotency_key` should be derived from `order_id` (e.g., `f"charge:{order_id}"`) and passed through from `OrderProcessor.process()`. This is the standard mechanism for safe payment retries and is supported by Stripe, Braintree, and most modern payment gateways.

### 3. Lock TTL Is Dangerously Short and Not Validated Against Actual Execution Time

**File:** `services/order_processor.py`

```python
ORDER_LOCK_TTL = 60  # seconds
```

The worst-case execution path inside the lock is:

- `db.get_charge()` — DB round-trip
- `payment.charge()` — up to `REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1)` = `30 * 3 + 5 * 2` = **100 seconds**
- `db.transaction()` commit — DB round-trip

The maximum time inside the lock is **well over 100 seconds**. With a 60-second TTL, Redis will expire the lock while the first worker is still executing. A second worker then acquires the lock, passes the idempotency check (no record yet), and proceeds to charge. This is a double-charge scenario introduced by the lock mechanism itself.

**Required fixes:**
- Set `ORDER_LOCK_TTL` to at minimum `REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1) + safety_margin` — at least 120–150 seconds given current payment client constants.
- Add a startup assertion or configuration validation that enforces `ORDER_LOCK_TTL > payment_client_max_execution_time`:

```python
def __init__(self, redis_client, db, payment_client):
    max_payment_time = (
        payment_client.REQUEST_TIMEOUT * payment_client.MAX_RETRIES
        + payment_client.RETRY_BACKOFF * (payment_client.MAX_RETRIES - 1)
    )
    assert self.ORDER_LOCK_TTL > max_payment_time, (
        f"ORDER_LOCK_TTL ({self.ORDER_LOCK_TTL}s) must exceed max payment "
        f"execution time ({max_payment_time}s)"
    )
```

## Moderate Issues

### 4. Idempotency Check Is Outside the Database Transaction — TOCTOU Window

```python
existing = self.db.get_charge(req.order_id)   # ← read
if existing:
    return ...

with self.db.transaction():                    # ← write happens later
    charge_id = self.payment.charge(...)
```

The Redis lock reduces this window significantly, but does not eliminate it entirely. If the Redis lock expires mid-execution (see Issue 3) and two workers are now both inside the critical section, both can read `existing = None` before either writes. The idempotency check must be re-evaluated **inside the transaction** using a `SELECT ... FOR UPDATE` or equivalent pessimistic DB lock to be fully correct. The outer check is acceptable as a fast-path optimization only if the inner check is also present.

### 5. `PaymentError` Is Not Caught in `OrderProcessor.process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # ← can raise PaymentError
```

`PaymentError` is not caught anywhere in `process()`. It will propagate as an unhandled exception to the caller. The PR claims structured error handling, but the payment failure path returns no `OrderResult` — it raises. This is inconsistent with the method's contract and will cause 500-level errors at the API layer unless the caller explicitly handles `PaymentError`, which is not documented.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

### 6. Lock Release Swallows All Exceptions Silently

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception is suppressed by the `finally` block's implicit swallowing — actually in this case it would propagate, but there is no handling. More critically, there is no check on the return value. A return value of `0` from the Lua script means the token did not match — the lock was either expired and re-acquired by another worker, or something corrupted the key. This is a sentinel event that should be logged as a critical warning, not silently treated as a successful release.

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.critical(
                "Lock token mismatch on release — lock may have expired and been "
                "re-acquired. Possible double-processing. order_id=%s", order_id
            )
    except redis.RedisError as e:
        logger.error("Failed to release lock order_id=%s error=%s", order_id, e)
```

## Minor Issues

### 7. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is never directly logged, which is correct. Confirm that `payment_client.charge()` does not log the raw token at any log level, and that the `requests` library debug logging is disabled in production (it will log full request bodies including the token).
