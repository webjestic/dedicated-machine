# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Outside Transaction Boundary — At-Most-Once Guarantee Is False

This is the most significant flaw in the PR, and it directly contradicts the documented guarantee.

```python
# The comment claims crash-safety, but the code does not deliver it:
with self.db.transaction():
    charge_id = self.payment.charge(      # ← External side effect INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is a network operation against an external payment gateway. Wrapping it in `self.db.transaction()` does **nothing** to make it atomic with the database writes. A database transaction provides atomicity over *database* operations only. The external HTTP call is not enrolled in the transaction coordinator.

The actual failure modes this architecture silently introduces:

**Scenario A — Charge succeeds, DB commit fails:**
The payment gateway has debited the customer. The `db.transaction()` rolls back. `record_charge` and `mark_order_complete` are never persisted. On retry, `get_charge` returns `None`, the idempotency check passes, and the customer is charged a second time. The "at-most-once" guarantee is violated.

**Scenario B — Charge succeeds, process crashes before DB commit:**
Same outcome as Scenario A. The lock TTL expires, another worker acquires the lock, idempotency check finds no record, double charge occurs.

**Scenario C — Gateway timeout, charge actually processed:**
`PaymentClient` retries on timeout. The gateway may have processed the first attempt. The retry issues a second charge. There is no idempotency key passed to the gateway, so the gateway has no way to deduplicate. This is compounded by the fact that `payment_token` is reused across retries without any idempotency token.

The correct architecture requires one of:
- **Outbox pattern**: Write a pending charge record to the DB *first* (in a transaction), then process it asynchronously. Only mark complete after confirmed gateway response.
- **Gateway-side idempotency key**: Generate a stable, deterministic idempotency key (e.g., `HMAC(order_id + attempt)`) and pass it to the gateway on every attempt so the gateway deduplicates.
- **Two-phase approach**: Record intent before charging, charge, then confirm — with reconciliation for ambiguous states.

The PR summary claims *"crash between charge and record cannot produce an unrecorded charge"* — this is factually incorrect and the comment should not be allowed to persist in the codebase.

### 🔴 Critical: Redis Lock Does Not Protect Against the Core Race It Claims To

The lock protects against *concurrent* workers. It does not protect against the failure window described above. More specifically:

```python
existing = self.db.get_charge(req.order_id)   # Check
if existing:
    return ...                                  # Guard

with self.db.transaction():
    charge_id = self.payment.charge(...)        # Charge — external, unprotected
```

If Worker A holds the lock, charges successfully, then crashes before committing, the lock TTL expires. Worker B acquires the lock. `get_charge` returns `None`. Worker B charges again. The lock provided a false sense of safety.

### 🟡 Significant: `PaymentClient` Retries Timeouts Without Idempotency Key

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request *may have been processed*. Retrying without a stable idempotency key passed to the gateway can result in duplicate charges at the gateway level, independent of the application-level idempotency check. The `order_id` itself should be sent as an idempotency key in the request headers or body on every attempt.

```python
# Suggested:
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # stable per order
    timeout=self.REQUEST_TIMEOUT,
)
```

### 🟡 Significant: `charge_id` Leaks Out of Transaction Scope Incorrectly

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)

# charge_id used here, outside transaction
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
return OrderResult(processed=True, ..., charge_id=charge_id, ...)
```

If the transaction rolls back after `payment.charge()` succeeds, `charge_id` is a live gateway charge ID with no corresponding DB record. It is then returned to the caller as if processing succeeded. The caller may believe the order is complete when the DB has no record of it.

---

### 🟡 Significant: Lock TTL Is Insufficient for Retry-Inclusive Payment Round-Trip

```python
ORDER_LOCK_TTL = 60   # seconds
```

```python
# PaymentClient:
REQUEST_TIMEOUT = 30   # per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # between retries
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock TTL of 60 seconds will expire before the payment client exhausts its retries. A second worker can acquire the lock while the first is still mid-payment, defeating the mutual exclusion entirely. Either the TTL must exceed the maximum possible payment duration with margin, or the payment client's retry budget must be capped below the lock TTL.

---

### 🟡 Significant: Unhandled Exceptions Leak Lock State Ambiguity

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError
        self.db.record_charge(...)
```

If `payment.charge()` raises `PaymentError`, it propagates out of `self.db.transaction()` (triggering rollback — correct) and then out of `_acquire_lock` (triggering lock release via `finally` — correct). However, the caller receives an unhandled exception rather than an `OrderResult`. The `process()` method has no `except PaymentError` handler. The caller gets a raw exception instead of a structured `OrderResult(processed=False, ...)`. This breaks the documented return contract of the method.

---

### 🟠 Minor: `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages may include field values in some configurations. If `payment_token` fails validation, its value could appear in log output. Payment tokens are sensitive credentials. The log message should be sanitized or the error serialized with field names only.

---

### 🟠 Minor: Lock Retry Backoff Is Linear, Not Exponential

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

The PR description claims "retry with backoff" but this is linear backoff. Under high contention with many workers, linear backoff provides poor thundering-herd mitigation. Exponential backoff with jitter is the standard for distributed lock retry. This is a minor issue but the description is misleading.

---

### 🟠 Minor: `order_id` Not Returned on Lock Contention

```python
except LockError:
    return OrderResult(
        processed=False,
        message="Order currently being processed by another worker",
        # order_id omitted
    )
```

All other failure paths return `order_id` in the result. Omitting it on lock contention makes caller-side correlation harder. Consistent inclusion is preferable.

## `tests/test_order_processor.py`

### 🔴 Critical: Tests Do Not Cover the Core Failure Scenarios

The test suite validates the happy path and basic contention, but does not test any of the critical failure modes:

- No test for `PaymentError` propagation (unhandled exception path)
- No test for charge-succeeds-then-DB-rollback (the double-charge scenario)
- No test for lock TTL expiry mid-payment
- No test verifying idempotency key is passed to the payment gateway

The tests pass because they mock away all the complexity. The mock-based approach here validates wiring, not correctness under failure.

---

### 🟡 Significant: `test_lock_release_uses_lua_script` Is a Tautology

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that the mock script was called. Because `_release_script` is itself a mock returned by `register_script`, this test only verifies that the code calls whatever `register_script` returned. It does not verify the Lua script content, the keys passed, or the token argument. A refactor that passes the wrong token would not be caught.
