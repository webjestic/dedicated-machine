Decision: Request Changes

Summary: The core locking and idempotency architecture contains a critical race condition that directly undermines the at-most-once payment guarantee this PR claims to provide. Several additional issues compound the risk тАФ most notably a timeout window that can produce undetected duplicate charges, a missing exception handler that will surface raw payment errors to callers, and test coverage that validates the happy path while leaving the most dangerous failure modes completely untested. This code should not be merged in its current state.

---

## Critical Issues

### 1. Race Condition: Charge Occurs Inside a DB Transaction, But Outside Idempotency Enforcement

This is the most dangerous flaw in the design. The comment in the code says:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**, and the architecture actually inverts the problem rather than solving it.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External side effect тАФ NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge(...)` is a call to an **external payment gateway**. It is not part of the database transaction. If the process crashes, is killed, or the network drops **after** the charge succeeds but **before** `db.record_charge` commits, you have:

- A real charge on the customer's card
- No record in the database
- On retry, `get_charge` returns `None`, so the idempotency check passes
- The customer gets charged **twice**

Wrapping an external side effect in a `db.transaction()` block does not make it atomic with the database. This is a textbook example of a distributed systems partial failure. The comment actively misleads reviewers about the safety guarantee.

**Remediation path:** The standard approach here is the **outbox pattern** or **two-phase commit equivalent**:
1. Record a `PENDING` charge intent to the database *before* calling the payment gateway, within a transaction.
2. Call the payment gateway.
3. Update the record to `COMPLETED` with the `charge_id`.
4. The idempotency check must also catch `PENDING` records and either block or resume, not just check for completed charges.

This also requires a reconciliation/recovery job for charges that are stuck in `PENDING` due to crash.

### 2. Timeout Race: `ORDER_LOCK_TTL` Can Expire Mid-Execution, Enabling Concurrent Duplicate Processing

```python
ORDER_LOCK_TTL = 60  # seconds тАФ covers typical payment round-trip
```

```python
# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment client execution time: `(30s timeout + 5s backoff) ├Ч 3 attempts = 105 seconds`.

The lock TTL is 60 seconds. A slow or retrying payment call will **outlive the lock**. A second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge тАФ all while the first worker is still running. The lock provides no protection in this scenario.

**Remediation:** Either:
- Extend `ORDER_LOCK_TTL` to safely exceed worst-case execution time with margin (e.g., 180s), **or**
- Implement lock extension/heartbeat (Redlock-style keepalive), **or**
- Reduce `MAX_RETRIES ├Ч (REQUEST_TIMEOUT + RETRY_BACKOFF)` to be well under `ORDER_LOCK_TTL`

All three of these numbers must be reviewed together. They currently are not.

### 3. `PaymentError` Is Not Caught in `process()` тАФ Unhandled Exception Escapes

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # Can raise PaymentError
        ...
```

`PaymentError` is defined in `payment_client.py` but is **never caught** in `order_processor.py`. If the payment gateway returns a 4xx or exhausts retries, `PaymentError` propagates out of `process()` as an unhandled exception. The caller receives a raw exception rather than an `OrderResult`.

The method signature promises `-> OrderResult`. That contract is broken.

```python
# Missing entirely:
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

Note also: the lock **will** still be released via `finally`, which is correct тАФ but the caller has no structured way to distinguish a payment failure from a lock failure or a validation failure.

## Significant Issues

### 4. Idempotency Check Is Not Atomic With Lock Acquisition

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # Read
    if existing:
        return ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

The idempotency check reads from the database *after* acquiring the lock, which is correct. However, if `get_charge` and `record_charge` do not operate on the same database isolation level or connection, a committed charge from a prior run that completed *just before* the lock was acquired could be missed depending on your DB's read consistency. This is less likely to be a practical issue than items 1тАУ3, but the DB isolation level should be explicitly documented and enforced here.

### 5. `payment_token` Is Logged Indirectly Through Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in logs. Payment tokens are sensitive credentials. This should be sanitized before logging.

```python
# Safer:
logger.warning("Invalid order request for order_id=%s: %s", order_id, e)
# And ensure the ValidationError repr does not include payment_token value
```

At minimum, the logging of `ValidationError` should be reviewed against your data classification policy.

### 6. `requests.post` Has No Authentication to the Internal Payment Gateway

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no headers, no API key, no mTLS configuration, no service identity. Even on an internal network, calling a payment endpoint with no authentication is a security concern. This endpoint processes real charges. At minimum, an authorization header should be injected, and the mechanism for credential management (env var, secrets manager, etc.) should be present or stubbed.

## Minor Issues

### 7. Test Coverage Gaps on the Most Critical Paths

The tests validate the happy path and basic contention, but the following scenarios have **zero coverage**:

| Scenario | Covered? |
|---|---|
| Charge succeeds, DB record fails (partial failure) | тЭМ |
| Lock expires mid-payment (TTL race) | тЭМ |
| `PaymentError` raised тАФ does caller get `OrderResult`? | тЭМ |
| Payment gateway timeout on all retries | тЭМ |
| `payment_token` in validation error log | тЭМ |
| DB transaction rollback behavior | тЭМ |

`test_lock_release_uses_lua_script` is testing implementation detail rather than behavior. It will break on any internal refactor and provides little safety signal.

### 8. `LOCK_RETRY_ATTEMPTS = 3` With No Jitter

```python
backoff = 0.5 * (attempt + 1)
```

Linear backoff with no jitter under high concurrency will cause retry storms тАФ all workers that failed to acquire the lock will retry at the same intervals. This is a well-known problem. Add `random.uniform(0, backoff)` jitter.

### 9. Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs in source code make environment promotion (dev/staging/prod) impossible without code changes. This should be injected via configuration or environment variable.
