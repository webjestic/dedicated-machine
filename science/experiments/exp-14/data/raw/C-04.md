Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all sound patterns. However, there is a critical atomicity flaw in the payment flow that undermines the stated at-most-once guarantee, plus several reliability and security issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge occurs inside the DB transaction — this does not provide atomicity, it breaks it

The PR comment claims "crash between charge and record cannot produce an unrecorded charge," but this is incorrect and the logic is inverted.

```python
# Current code — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Problems:**

- The payment gateway call is a network operation. Holding a DB transaction open across an external HTTP call (up to 30s × 3 retries = 90s per `PaymentClient`) will exhaust your DB connection pool under any real load.
- If `payment.charge()` succeeds but the DB transaction rolls back (network blip, DB error, process crash), the charge is real but unrecorded — exactly the failure mode the design claims to prevent. The DB transaction provides no rollback capability over an external payment API.
- If `payment.charge()` raises mid-transaction, you have an open transaction holding locks while the retry backoff sleeps.

**Correct pattern:** Charge outside the transaction, then record inside it. Use the idempotency check and the Redis lock as your duplicate-charge guard, not the DB transaction boundary.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only record after charge succeeds
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 2. Unhandled exceptions inside the lock context will silently swallow errors

```python
with self._acquire_lock(req.order_id):
    # ... processing ...
```

Any exception other than `LockError` (e.g., `PaymentError`, DB errors, `KeyError` on `response.json()["charge_id"]`) propagates out of `process()` uncaught. The caller gets an unhandled exception instead of an `OrderResult`. The lock is released correctly via `finally`, but the caller contract is broken. Add explicit exception handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

## Significant Issues

### 3. Redis connection failure is not handled

If `self.redis.set(...)` raises a `redis.RedisError` (connection refused, timeout), it propagates uncaught through `_acquire_lock` and then through `process()`. Redis being unavailable should degrade gracefully, not crash the worker.

### 4. `payment_token` is logged indirectly via `charge_id` — verify no token leakage in payment client

In `payment_client.py`, the `token` field is sent in the request body. Confirm that `requests` debug logging is not enabled in any environment, as it will log the full request body including the payment token. This should be explicitly documented or guarded.

### 5. `PaymentClient` has no authentication on the internal gateway call

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, mTLS configuration, or API key. Even for an internal endpoint, this is a missing control. If this is intentional (handled at the network layer), it needs a comment stating that explicitly.

### 6. Lock TTL may be insufficient under retry load

`ORDER_LOCK_TTL = 60s` but `PaymentClient` can take up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The lock will expire before the payment completes, allowing a second worker to acquire the lock and potentially double-charge. Either increase the TTL to at least 120s with margin, or reduce the payment client's total retry budget.

## Minor Issues

### 7. Test for lock release verifies mechanism but not correctness

`test_lock_release_uses_lua_script` confirms the Lua script was called but does not assert it was called with the correct `keys` and `args` (i.e., the matching token). A bug that passes the wrong token would not be caught.

```python
# Add assertion like:
call_kwargs = processor._release_script.call_args
assert call_kwargs.kwargs["keys"] == [f"order:lock:order-001"]
# token verification is harder with uuid but at minimum check keys
```

### 8. No test for payment failure within the lock

There is no test covering `PaymentError` being raised by `payment.charge()`. Given this is the most likely production failure mode, it should be explicitly tested to verify the lock is released and the return value is a well-formed `OrderResult`.

### 9. `charge_id` missing from `OrderResult` on already-processed path

When returning "Order already processed," the existing `charge_id` from `existing` is available but not returned. Callers may need it for idempotent response handling.

---

## What Is Done Well

- Lua script for atomic lock release is correct and the token-verification pattern is solid.
- `SET NX EX` is the right primitive — no separate TTL call, no SETNX race.
- Pydantic validation with regex on `order_id` correctly blocks path traversal attempts.
- Retry backoff in the lock acquisition loop is reasonable.
- Structured logging with contextual fields throughout.
- Test fixture design is clean and the mock setup is straightforward.
