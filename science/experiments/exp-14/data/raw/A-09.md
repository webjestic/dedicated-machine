# Pull Request Review

**PR Title:** Add distributed order processor with Redis-backed exclusive locking
**Author:** dev-team
**Branch:** feature/order-processor

---

## Detailed Analysis

### Critical Defects

#### 1. **CRITICAL — Charge Outside Database Transaction Creates Unrecorded Charge Window**

This is the most dangerous flaw in the entire PR. The PR summary explicitly claims:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**. Look at what is actually inside the `db.transaction()` block:

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External side effect INSIDE the transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over **database operations only**. The external HTTP call to the payment gateway is not a transactional resource. If `self.payment.charge()` succeeds but the process crashes before `record_charge()` commits, or if the database transaction rolls back for any reason after the charge succeeds, you have:

- **Real money debited from the customer**
- **No record of it in the database**
- **The idempotency check will not find it on retry**
- **The customer gets charged again on the next attempt**

The correct pattern is to perform the charge **before** opening the transaction, capture the `charge_id`, and then open the transaction only for the database writes. The idempotency check and the lock together prevent re-entry. The payment client already has retry logic, so the charge call should be outside the `db.transaction()` scope:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This does not eliminate all failure modes — a crash after charge but before commit still leaves an unrecorded charge — but that is an inherent property of two-phase operations across heterogeneous systems. The solution there is an outbox/reconciliation pattern, not wrapping an HTTP call in a DB transaction. What the current code does is **actively worse** because it creates the false impression of atomicity while providing none, and it holds a database connection open for the entire duration of a potentially 90-second HTTP call (3 attempts × 30s timeout).

#### 2. **CRITICAL — Lock TTL Does Not Cover Payment Retry Duration**

```python
ORDER_LOCK_TTL = 60        # seconds — covers typical payment round-trip
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client duration: `(30s × 3) + (5s × 2) = 100 seconds`

The lock TTL is **60 seconds**. Under worst-case transient gateway failures, the lock expires while the payment is still in flight. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge. This is the exact double-charge scenario this architecture is supposed to prevent.

The comment `# seconds — covers typical payment round-trip` is dangerously misleading. The TTL must be set to cover the **worst-case** duration, not the typical one. Either:

- Raise `ORDER_LOCK_TTL` to at least `120` (with margin), or
- Implement lock extension (Redlock-style keepalive), or
- Reduce `MAX_RETRIES × REQUEST_TIMEOUT + RETRY_BACKOFF` to fit within the TTL

These two constants living in separate files with no documented relationship to each other is itself an architectural smell.

#### 3. **HIGH — Timeout Exception Does Not Guarantee Non-Charge**

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A `Timeout` from `requests` can fire after the request was fully transmitted and the charge was processed by the gateway — the timeout occurred waiting for the **response**. The payment may have succeeded. Retrying on timeout without idempotency keys sent to the gateway can result in duplicate charges at the gateway level, entirely independent of the Redis lock.

The payment client must send a stable idempotency key (e.g., `order_id`) in the request headers:

```python
headers={"Idempotency-Key": order_id}
```

This is a standard capability of payment gateways (Stripe, etc.) and is the correct defense against retry-on-timeout double charges. Without it, the retry logic is unsafe.

### Significant Defects

#### 4. **HIGH — `payment_token` Logged or Exposed in Validation Error Path**

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic `ValidationError` string representations include field values in some versions. If `payment_token` fails validation (e.g., too long), the raw token value may appear in logs. Payment tokens are sensitive credentials. The log statement should either redact the payment_token field or log only field names and error types, not values.

---

#### 5. **MEDIUM — Redis Unavailability Is Not Handled — Silent Failure Path**

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If the Redis connection is down, `redis.set()` raises `redis.exceptions.ConnectionError` or `redis.exceptions.RedisError`. This exception is **not caught** in `process()`. It will propagate up as an unhandled exception. The caller receives no `OrderResult` — they receive a raw exception. Depending on the calling context this could:

- Crash a worker
- Be swallowed by a framework and silently drop the order
- Expose internal stack traces to an upstream caller

Redis failures should be caught and returned as a structured `OrderResult(processed=False, message="Service temporarily unavailable")` with appropriate alerting. The same applies to `self._release_script()` in the `finally` block — a Redis failure there would suppress the original exception.

---

#### 6. **MEDIUM — Idempotency Check Has a Race Condition Without Read-Your-Writes Guarantee**

```python
existing = self.db.get_charge(req.order_id)
```

If the database uses a read replica for `get_charge()`, replication lag means a recently committed charge may not be visible. Two workers could both pass the idempotency check before either commits. The Redis lock mitigates this for the normal case, but if the lock expires (see issue #2) or if there is a Redis failover, this check is the last line of defense — and it may be reading stale data. The idempotency check must read from the primary/leader node.

---

#### 7. **MEDIUM — `PaymentError` Not Caught in `process()`**

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # Can raise PaymentError
```

`PaymentError` is not caught in `process()`. It will propagate through the `db.transaction()` context manager (triggering a rollback, which is correct) and then through `_acquire_lock`'s `finally` block (releasing the lock, which is also correct), but it will then surface as an unhandled exception to the caller. The caller receives no `OrderResult`. This should be caught and returned as a structured result.

---

#### 8. **LOW — Lock Retry Backoff Strategy Is Weak**

```python
backoff = 0.5 * (attempt + 1)
```

This is linear backoff: 0.5s, 1.0s, 1.5s. Under high contention with many workers, linear backoff with no jitter causes retry storms — all waiting workers wake up at approximately the same time and hammer the lock simultaneously. Standard practice is exponential backoff with jitter.

---

#### 9. **LOW — Test Does Not Verify Charge Is Outside Transaction**

```python
def test_charge_recorded_within_transaction(processor, db):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    db.transaction.assert_called_once()
    db.record_charge.assert_called_once()
```

The test is named `test_charge_recorded_within_transaction` but it does not assert call ordering. It would pass even if `record_charge` were called outside the transaction. More critically, no test covers the scenario where `payment.charge()` succeeds but `record_charge()` raises — verifying whether the charge is orphaned or correctly handled. The test suite validates the happy path well but provides false confidence on the failure modes that matter most for a payment system.

---

### Minor Observations

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants but are not configurable at construction time, making it impossible to tune them per deployment environment without subclassing.
- `CHARGE_ENDPOINT` hardcoded as a class attribute in `PaymentClient` should be injected via constructor or configuration — hardcoded internal URLs are a deployment and testability smell.
- The `OrderResult.order_id` field is `Optional[str]` but is not populated in the `LockError` return path, making it harder for callers to correlate the failure back to the original order without re-parsing the message string.
