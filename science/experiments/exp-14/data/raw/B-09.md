# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Outside Database Transaction — At-Most-Once Guarantee Is False

The PR summary and inline comment both claim:

> *"Crash between charge and record cannot produce an unrecorded charge"*

This is incorrect. The `payment.charge()` call is **inside** `with self.db.transaction()`, but a database transaction provides atomicity over *database operations only*. The payment gateway call is a network side effect that cannot be rolled back.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect, NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `payment.charge()` succeeds but `record_charge()` or `mark_order_complete()` raises, the database transaction rolls back, but the charge has already been captured at the gateway. The order now has no recorded charge. On the next attempt, `get_charge()` returns `None`, the idempotency check passes, and the customer is charged a second time. The lock does not protect against this because it has been released by the time the retry occurs.

**Correct pattern:** charge outside the transaction, then open the transaction only for the database writes. If the DB write fails after a successful charge, a reconciliation/recovery path (or idempotency key at the gateway level) must handle it — not a DB rollback.

```python
# Correct structure
charge_id = self.payment.charge(...)          # outside DB transaction
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Additionally, the `PaymentClient` has no idempotency key passed to the gateway. On a timeout retry, the gateway may have already processed the first attempt. Without a stable idempotency key per order, retries can produce duplicate charges at the gateway level independent of the Redis lock.

---

### 🔴 Critical: Redis Lock Does Not Guarantee At-Most-Once Under All Failure Modes

The lock TTL is 60 seconds. If a worker stalls (GC pause, slow DB, network partition) for longer than 60 seconds while holding the lock, the lock expires. A second worker acquires the lock, passes the idempotency check (charge not yet recorded), and issues a second charge. Worker one then resumes and also completes. The Lua release script prevents the wrong worker from releasing the lock, but it does not prevent the first worker from continuing to execute after its lock has expired.

This is the classic **fencing token** problem. The lock expiry must be detectable by the holder. Options include:

- Fencing tokens (monotonic version written to DB, rejected if stale)
- Lock heartbeat / extension before critical operations
- Checking lock validity immediately before the charge

### 🟡 Significant: `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

`payment_token` is accepted as input and passed directly to `payment.charge()`. Confirm that no logging path (framework request logging, exception tracebacks, Pydantic validation errors) serializes the full `OrderRequest` including `payment_token`. The current code does not explicitly log it, but the `ValidationError` catch logs `e` directly — if the token is malformed and triggers a validation error, the raw value may appear in the error output depending on Pydantic's error formatting.

---

### 🟡 Significant: Unhandled Exceptions Inside Lock Context Leave Lock Held Until TTL

`PaymentError`, database exceptions, and any unexpected exception raised inside `_acquire_lock` are not caught in `process()`. The `finally` block in `_acquire_lock` will release the lock correctly in these cases — this part is fine. However, these exceptions propagate out of `process()` entirely unhandled, producing an unstructured exception rather than an `OrderResult`. The caller receives no structured response and the error is invisible unless the caller wraps it. A bare `except Exception` with logging and a failure `OrderResult` return should guard the inner block.

---

### 🟡 Significant: `get_charge()` Idempotency Check Has a Read Visibility Gap

`get_charge()` reads from the database. If the database uses read replicas with replication lag, a recently committed charge may not be visible on the replica being read. Two workers on different replicas could both see `None` and both proceed past the idempotency check. This is only safe if `get_charge()` is guaranteed to read from the primary. This should be documented or enforced.

---

### 🟡 Significant: Lock Retry Backoff Is Blocking (`time.sleep`)

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

In any async or threaded worker pool, blocking sleep on a worker thread stalls the thread for the entire backoff duration. If this runs in an async context (FastAPI, etc.), this must be `await asyncio.sleep()`. Even in a sync context, blocking the thread for lock contention is a throughput concern under load.

## `services/payment_client.py`

### 🟡 Significant: Timeout Retry Does Not Verify Whether the Charge Succeeded

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retries unconditionally
```

A timeout means the response was not received — it does not mean the charge did not occur. The gateway may have processed the charge and the response was lost in transit. Retrying without an idempotency key will submit a second charge request. This is a known and common source of duplicate charges in payment integrations. Every POST to the charge endpoint must include a stable, order-scoped idempotency key in the request headers.

---

### 🟠 Minor: `response.json()["charge_id"]` Has No Error Handling

If the gateway returns a 200 with a malformed or unexpected JSON body, this raises `KeyError` or `json.JSONDecodeError`, which propagates as an unhandled exception rather than a `PaymentError`. The response body should be validated before use.

## `tests/test_order_processor.py`

### 🟠 Minor: Tests Do Not Cover the Critical Failure Mode

There is no test for the scenario where `payment.charge()` succeeds but `db.record_charge()` raises. This is precisely the double-charge scenario described above, and it is the most important case to cover. Its absence means the broken guarantee went undetected.

### 🟠 Minor: `test_lock_release_uses_lua_script` Does Not Verify Token Correctness

The test confirms the Lua script was called but does not assert that the correct token was passed as `args[0]`. A future refactor could break token verification silently.

**Decision:** Request Changes

**Summary:** The PR contains two critical correctness flaws that directly contradict its stated at-most-once guarantee: the payment call is placed inside a database transaction that can roll back after a successful charge, and the Redis lock provides no fencing against a worker that continues executing after its TTL has expired. Additionally, the payment client retries on timeout without a gateway-level idempotency key, which is an independent path to duplicate charges. These issues must be resolved before this code handles real payment operations.
