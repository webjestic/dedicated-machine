# Pull Request Review

**PR Title:** Add distributed order processor with Redis-backed exclusive locking
**Author:** dev-team
**Branch:** feature/order-processor

---

## Detailed Analysis

### Critical Findings

#### 1. **CRITICAL — Charge Occurs Inside DB Transaction: External Side Effect Cannot Be Rolled Back**

This is the most dangerous architectural flaw in the entire PR. The payment charge is issued *inside* the database transaction:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External side effect — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `record_charge` or `mark_order_complete` raises after `payment.charge()` succeeds, the DB transaction rolls back — but the charge **has already been submitted to the payment gateway and cannot be rolled back with it**. The system will have charged the customer with zero record of it. This is the exact double-charge/phantom-charge scenario the PR claims to prevent. The correct pattern is:

```python
charge_id = self.payment.charge(...)           # Issue charge OUTSIDE the transaction
with self.db.transaction():                    # Then atomically record the result
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is a textbook violation of the rule: **never issue irreversible external side effects inside a database transaction.**

---

#### 2. **CRITICAL — Timeout on Payment Gateway Creates Unresolvable Ambiguity Under Lock**

In `payment_client.py`, a `Timeout` exception is caught and retried:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout does **not** mean the charge failed. The gateway may have processed the charge and the response was simply lost in transit. Retrying on timeout without first querying the gateway for idempotency status means the same customer can be charged multiple times within a single lock window. This is especially dangerous because the Redis lock TTL is 60 seconds and `MAX_RETRIES=3` with `RETRY_BACKOFF=5` and `REQUEST_TIMEOUT=30` means a worst-case execution of `3 * (30 + 5) = 105 seconds` — **exceeding the lock TTL by 45 seconds**. The lock will expire mid-execution, a second worker can acquire it, pass the idempotency check (charge not yet recorded), and issue a second charge concurrently.

The payment client must either:
- Use gateway-side idempotency keys (passed per-attempt, not per-retry), or
- Query the gateway for charge status before retrying a timed-out request.

---

#### 3. **CRITICAL — Lock TTL Can Expire During Normal Execution**

As computed above, the maximum execution time of the payment client alone (`3 * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 105s`) exceeds `ORDER_LOCK_TTL = 60s`. The lock will silently expire. Another worker will acquire it, pass the idempotency check, and issue a duplicate charge. The lock TTL must be set with a hard guarantee that it exceeds the maximum possible execution time of everything inside the lock, or a lock extension/watchdog mechanism must be employed.

### High Severity Findings

#### 4. **HIGH — `payment_token` Logged Indirectly via Pydantic ValidationError**

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors can include field values in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials. The log statement should be sanitized or the error message should be constructed to exclude field values.

---

#### 5. **HIGH — Unhandled Exceptions Inside Lock Context Leak Silently**

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, database exceptions, and any unexpected runtime errors raised inside the lock are **not caught**. They will propagate out of `process()` as unhandled exceptions, returning nothing to the caller. The caller has no structured `OrderResult` to inspect. The PR's own return type contract (`OrderResult`) is violated. These must be caught and returned as structured failure results, and the behavior must be tested.

---

#### 6. **HIGH — Idempotency Check is Not Atomic With Charge Issuance**

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return ...

with self.db.transaction():
    charge_id = self.payment.charge(...)
```

The check-then-act between `get_charge` and `payment.charge` is not atomic at the database level. The Redis lock serializes workers on this order, which mitigates the race *in the normal case*, but if the lock expires (see finding #3) or Redis experiences a failover, two workers can simultaneously pass the idempotency check and both issue charges. The idempotency check must be enforced at the database layer with a unique constraint on `order_id` in the charges table, not solely in application logic.

---

### Medium Severity Findings

#### 7. **MEDIUM — `payment_token` Appears in Serialized `OrderRequest` Object**

`OrderRequest` is a Pydantic model containing `payment_token`. If this model is ever serialized (e.g., `.model_dump()`, `.json()`, debug logging, exception tracebacks), the raw token will be exposed. The field should be marked with `repr=False` and excluded from serialization by default:

```python
payment_token: str = Field(..., min_length=1, max_length=256, repr=False, exclude=True)
```

---

#### 8. **MEDIUM — Lock Retry Backoff Is Linear, Not Exponential, and Has No Jitter**

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

This is linear backoff. Under high concurrency, multiple workers will retry on the same schedule and collide repeatedly (thundering herd). Standard practice for distributed lock retry is **exponential backoff with random jitter**. With only 3 attempts at these intervals, the effective contention window is extremely narrow. This should be revisited.

---

#### 9. **MEDIUM — `requests.post` Has No Connection Timeout, Only Read Timeout**

```python
timeout=self.REQUEST_TIMEOUT
```

When a scalar is passed to `requests`, it sets the read timeout only. A connection to a slow or unreachable host can hang indefinitely at the TCP handshake stage. The correct form is:

```python
timeout=(CONNECT_TIMEOUT, self.REQUEST_TIMEOUT)
```

---

#### 10. **MEDIUM — Hardcoded Internal Payment Endpoint**

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded service endpoints are an operational and security concern. This should be injected via configuration or environment variable to support environment parity (dev/staging/prod), secret rotation, and to prevent accidental charges against production from non-production environments.

---

#### 11. **MEDIUM — `response.json()["charge_id"]` Has No Error Handling**

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected response body, this raises `KeyError` or `JSONDecodeError`, which are not caught and will propagate as unhandled exceptions. The response schema must be validated before accessing fields.

---

### Low Severity / Observations

#### 12. **LOW — Test for Lock Release Verifies Mechanism, Not Correctness**

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms that `redis.delete` is not called and the script is invoked, but it does not verify that the correct `lock_key` and `token` are passed to the script. A future refactor could call the script with wrong arguments and this test would still pass.

---

#### 13. **LOW — No Test Coverage for PaymentError or DB Exception Paths**

There are no tests for `PaymentError` being raised inside the lock, or for database exceptions during `record_charge` or `mark_order_complete`. Given finding #5, these are exactly the paths most likely to violate the return type contract.

---

#### 14. **LOW — `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as Class-Level Constants**

These are defined as class attributes with no mechanism for per-instance or per-environment override. For a service that will operate across different deployment environments with different latency profiles, these should be injectable at construction time.

---

## Summary of Critical Path

The most dangerous failure scenario enabled by this code:

1. Worker A acquires lock, issues charge (succeeds at gateway), `record_charge` raises a DB exception.
2. DB transaction rolls back. Lock is released. No record of charge exists.
3. Worker B acquires lock, passes idempotency check (no record found), issues a second charge.
4. Customer is charged twice. No audit trail of the first charge exists.

This scenario is not hypothetical — it is a direct consequence of findings #1 and #5 together.

---

## Decision: Request Changes

**Summary:** This PR contains at least three critical architectural flaws that directly undermine its stated guarantee of at-most-once payment processing: the payment charge is issued inside a database transaction making it non-rollback-safe, the maximum payment client execution time (105s) exceeds the Redis lock TTL (60s) enabling concurrent duplicate charges, and unhandled exceptions inside the lock silently violate the structured return contract while leaving the system in an ambiguous charge state. No amount of test coverage compensates for these design-level failures — the charge issuance must be moved outside the DB transaction, the lock TTL must be reconciled with the maximum execution window,
