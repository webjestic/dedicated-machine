Decision: Request Changes

Summary: The architecture and security fundamentals here are solid — atomic lock acquisition, token-verified Lua release, idempotency check, Pydantic validation, and structured retry logic are all implemented correctly. However, there is a critical atomicity flaw in the payment flow that undermines the core at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this can be approved.

---

## Critical Issue

### Payment charge occurs inside a database transaction — this is not safe

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this claim is incorrect. A database transaction does not wrap an external HTTP call. If `payment.charge()` succeeds but the process crashes before `db.record_charge()` commits, you have charged the customer with no record of it. The database transaction only protects the two DB writes against each other. It provides no atomicity guarantee over the payment gateway call.

The correct pattern is:

```python
# 1. Charge externally — outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record atomically in DB — now the external call is already done
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This does not fully eliminate the crash window, but it is the correct ordering. The idempotency check on re-entry handles the case where the lock expires and a second worker retries — which is why the idempotency check must query by `order_id` and also by `charge_id` from the gateway if the gateway supports idempotency keys. ## High Priority Issues

### No idempotency key sent to the payment gateway

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

The retry loop in `PaymentClient.charge()` will re-submit the charge on timeout with no idempotency key. A gateway timeout does not mean the charge failed — it means you do not know. Retrying without an idempotency key risks a double charge. The `order_id` should be passed through and sent as an idempotency key header or field on every attempt.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### Lock expiry is not handled after the lock is acquired

The `ORDER_LOCK_TTL` is 60 seconds and `PaymentClient.REQUEST_TIMEOUT` is 30 seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. In the worst case the payment client runs for approximately `3 * 30 + 2 * 5 = 100 seconds`, which exceeds the lock TTL. A second worker can acquire the lock while the first is still mid-charge. The idempotency check provides a partial backstop, but only if the first worker's DB write has already committed — which it has not yet at that point. Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime, or check lock validity before committing.

## Medium Priority Issues

### Unhandled exceptions from the payment client and database are swallowed silently

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

`PaymentError`, database exceptions, and any unexpected exceptions propagate out of `process()` as unhandled exceptions. The caller gets no `OrderResult` — they get a raw exception. Either catch these explicitly and return structured `OrderResult` failures, or document clearly that callers must handle exceptions. The current docstring implies `OrderResult` is always returned, which is misleading.

### `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages can include the field value that failed validation. If `payment_token` fails validation, its value may appear in logs. Sanitize or suppress field values from validation error output before logging.

## Minor Issues

### Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This only verifies the script was called. It does not verify that the correct token was passed as `args=[token]`. A regression that passes the wrong token or no token would not be caught.

### `PaymentClient` has no dependency injection for the endpoint URL

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This is hardcoded as a class attribute. It cannot be overridden in tests or across environments without subclassing or monkeypatching. Accept it as a constructor parameter or read it from configuration.

### `charge_id` missing from the already-processed result

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
    )
```

The existing charge record is available but `charge_id` is not returned to the caller. This makes idempotent retries from the client side harder to handle correctly since they cannot confirm which charge corresponds to their order.
