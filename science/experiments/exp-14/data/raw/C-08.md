Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, which means a successful charge that is followed by a DB commit failure will result in a real charge with no recorded transaction — exactly the double-charge/lost-charge scenario this service is supposed to prevent. Additionally, there are several secondary issues around exception handling gaps, timeout exposure, and test coverage holes that need to be addressed before this merges.

---

## Critical Issue

### 1. Payment call inside DB transaction breaks atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment in the PR says *"crash between charge and record cannot produce an unrecorded charge"* — but that is only true if the payment call is **outside** the transaction. If `record_charge` or `mark_order_complete` raises after `payment.charge()` succeeds, the DB transaction rolls back and you have a real charge with zero record of it. The Redis lock also expires independently, so a retry will hit the idempotency check... which finds nothing, and charges again.

**Required fix:** Charge outside the transaction, then record inside it. If the DB write fails after a successful charge, you need a reconciliation path (e.g., a pending-charge table, an outbox, or at minimum an alert), not a silent rollback.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# DB failure here is recoverable via reconciliation — charge_id is known
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

## Significant Issues

### 2. `PaymentError` and general exceptions escape `process()` silently

`process()` catches `LockError` and `ValidationError` but nothing else. A `PaymentError`, a DB exception, or a Redis connection error will propagate as an unhandled exception to the caller. Given this is a payment service, callers need a structured `OrderResult` back, not a raw exception. At minimum, catch `Exception` as a fallback and return a safe error result, while logging the full traceback.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 3. Timeout on gateway is ambiguous — retry is dangerous without idempotency key

In `PaymentClient.charge()`, a `Timeout` exception is retried. A timeout does **not** mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying a timed-out charge without an idempotency key sent to the gateway risks a double charge at the gateway level, independent of the Redis lock.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": some_stable_key,  # required
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

The idempotency key should be derived from `order_id` (passed down from `OrderProcessor`) so it is stable across retries.

### 4. Lock TTL can expire before the payment round-trip completes

`ORDER_LOCK_TTL = 60` seconds and `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst case: 3 × 30s timeouts + 2 × 5s backoffs = **100 seconds** — well past the 60-second lock TTL. A second worker can acquire the lock while the first is still in a retry loop, leading to concurrent execution against the same order.

Either increase the TTL to safely exceed the maximum payment client wall time, or implement lock extension (Redlock-style keepalive). At minimum, document the constraint and enforce it with a constant:

```python
MAX_PAYMENT_WALL_TIME = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
assert ORDER_LOCK_TTL > MAX_PAYMENT_WALL_TIME, "Lock TTL too short for payment retry budget"
```

## Minor Issues

### 5. `payment_token` logged indirectly via `charge_id` — verify no token leakage

`payment_token` is accepted as a field and passed to `payment.charge()`. Confirm that `PaymentClient` never logs the raw token. The current code does not, but this should be called out explicitly in a code comment or enforced via a custom Pydantic type that masks `__repr__`.

### 6. Test coverage gaps

- No test for `PaymentError` being raised — what does `process()` return? (Currently: unhandled exception.)
- No test for DB transaction rollback after a successful charge.
- No test for lock TTL expiry mid-payment (the race described in issue 4).
- `test_lock_release_uses_lua_script` asserts `_release_script` was called but does not assert it was called with the correct `keys` and `args` — a regression could pass the wrong token and the test would not catch it.

### 7. `customer_id` has no pattern constraint

`order_id` has `pattern=r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. If `customer_id` is used in DB queries or log output, it should have equivalent sanitization.
