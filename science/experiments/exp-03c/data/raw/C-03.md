Decision: Request Changes

Summary: This PR introduces two compounding critical vulnerabilities that must be resolved before merge. The `update()` helper is an unrestricted `setattr` loop over unvalidated user input — any field on the `User` model is writable via the JSON body, including `role`, `is_admin`, `password_hash`, and `email`. The endpoint also performs no ownership check, so any authenticated user can send a `PATCH` to `/api/users/<id>/profile` for any `user_id` and hit both issues simultaneously: arbitrary field writes on arbitrary accounts. `@login_required` confirms identity; it does not scope the operation to the requesting user. The three new tests cover only `display_name`, `bio`, and an empty body — the attack surface is completely untested.

---

## Findings

### 1. Mass Assignment — Critical

**Location:** `api/models.py` — `User.update()`, called from `api/users.py:update_user_profile`

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on User is now a write target
```

The method docstring says "apply field updates from a dict of key-value pairs." That is accurate. It applies *all* key-value pairs with no filtering. The caller passes `**request.get_json()` directly. There is no allowlist, no denylist, no schema validation between the HTTP body and the ORM object.

A request like this is valid under the current code:

```http
PATCH /api/users/1/profile HTTP/1.1
Content-Type: application/json

{"role": "admin"}
```

So is this:

```http
PATCH /api/users/1/profile HTTP/1.1
Content-Type: application/json

{"password_hash": "$2b$12$attacker_controlled_hash"}
```

The comment "the model handles which fields are relevant" in the reviewer notes is precisely the wrong mental model. The model handles *persistence*. It has no knowledge of which fields are safe to accept from an unauthenticated external source. That boundary belongs at the API layer, enforced explicitly.

**Required fix:** Define an explicit allowlist of mutable fields. Do not make the `update()` helper generic — that generality is the vulnerability.

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    unknown = kwargs.keys() - PROFILE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Alternatively, validate at the endpoint level with a schema library (Marshmallow, Pydantic) before the data touches the model at all. Either way, the allowlist must be explicit and version-controlled.

### 2. Missing Ownership Check — IDOR — Critical

**Location:** `api/users.py:update_user_profile`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches by URL param, no ownership check
    data = request.get_json() or {}
    user.update(**data)
```

`@login_required` establishes that `current_user` is authenticated. It says nothing about whether `current_user.id == user_id`. Any authenticated user can target any other user's record by changing the `user_id` in the URL. Combined with finding #1, this means any authenticated user can escalate any account to admin, overwrite any user's password hash, or corrupt any user's data.

These two vulnerabilities are multiplicative, not additive.

**Required fix:**

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

The ownership check must come before the database fetch. Returning 404 instead of 403 for cross-user attempts is an acceptable hardening choice to avoid confirming whether an account exists.

### 3. Test Coverage Does Not Cover the Attack Surface

The three new tests verify:
- `display_name` can be updated
- `bio` can be updated
- empty body returns 400

None of the following are tested:

| Scenario | Status |
|---|---|
| User updates their own profile | ✅ (implicitly, if `user_headers` authenticates as user 1) |
| User attempts to update another user's profile | ❌ Not tested |
| User attempts to set `role` | ❌ Not tested |
| User attempts to set `is_admin` | ❌ Not tested |
| User attempts to set `password_hash` | ❌ Not tested |
| Unknown field in body is rejected | ❌ Not tested |

After the fixes above are applied, the following tests must be added at minimum:

```python
def test_update_profile_rejects_role_escalation(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 400  # or 422, depending on validation layer

def test_update_profile_cannot_target_other_user(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_update_profile_unknown_field_rejected(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'overwritten'},
        headers=user_headers
    )
    assert rv.status_code == 400
```

---

### 4. Minor — `get_or_404` Before Authorization Check

Currently the endpoint fetches the user record before checking ownership. This leaks existence information: a 404 response confirms the user does not exist, a 403 confirms they do. Depending on your threat model this may be acceptable, but the ownership check should structurally precede the DB fetch regardless.

---

## What Needs to Change Before Merge

1. Replace the generic `update(**kwargs)` method with a profile-specific method that enforces an explicit field allowlist. The generic helper should not exist on the model in its current form — if it is used elsewhere, those call sites need the same audit.
2. Add `current_user.id != user_id` ownership check at the top of the endpoint, before the DB fetch.
3. Add tests for cross-user IDOR attempts and for privilege field injection. CI passing on happy-path tests is not a signal that this surface is safe.

The `update()` helper as written is not a convenience — it is the vulnerability. The reviewer note framing it as the model "handling which fields are relevant" is the exact reasoning that produces postmortems.
