Decision: Request Changes

Summary: This PR contains two critical security vulnerabilities that must be resolved before merge. The `update()` helper is an unrestricted mass assignment vector — it calls `setattr` over every key in the request body with no allowlist, meaning any authenticated user can send `{"role": "admin", "password_hash": "..."}` and it will be written directly to the database. Compounding this, the endpoint never verifies that `current_user.id == user_id`, so any logged-in user can send that payload against any account in the system — including accounts they don't own.

---

## Findings

### [CRITICAL] Mass Assignment — No Field Allowlist

**File:** `api/models.py`, `update()` method

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on the model is writable
```

The reviewer notes say "the model handles which fields are relevant." It does not. There is no filtering here. `setattr` does not know what is sensitive. It will write whatever key-value pair arrives in `kwargs` to whatever attribute exists on the model under that name.

A request body of:

```json
{"role": "admin"}
```

or

```json
{"is_admin": true, "password_hash": "$2b$12$attacker_controlled_hash"}
```

will pass through `update(**data)` and be committed to the database without any check. The `update()` method is not a safe abstraction — it is a direct path from the network to the ORM with nothing in between.

**Fix:** Define an explicit allowlist of fields that users are permitted to modify, and enforce it before any `setattr` call.

```python
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

def update_profile(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_UPDATABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist should live as a constant, not be inferred at runtime. If a new sensitive field is added to the model, the allowlist must not silently expand to include it.

### [CRITICAL] IDOR — No Ownership Check

**File:** `api/users.py`, `update_user_profile()`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)
    # current_user is never consulted again
    data = request.get_json() or {}
    user.update(**data)
    db.session.commit()
```

`@login_required` confirms that a session exists. It says nothing about whose record is being modified. `user_id` comes from the URL. Any authenticated user can substitute any other user's ID and the request will succeed.

Combined with the mass assignment issue above, the full attack surface is: any authenticated user can escalate any account — including accounts they do not own — to admin. These two vulnerabilities do not add; they multiply.

**Fix:** Enforce ownership at the start of the handler, before any data is read or written. Admin bypass should be explicit and separate.

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

### [MODERATE] Tests Do Not Cover the Attack Surface

The three new tests verify `display_name`, `bio`, and the empty-body 400 response. These are happy-path tests. None of them assert that the endpoint rejects unknown fields, and none of them attempt a cross-user request. Passing CI on these tests provides no signal about whether the endpoint is safe.

**Required tests before merge:**

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # either 400, or 200 with role unchanged — field must not be written
    data = rv.get_json()
    assert data.get('role') != 'admin'


def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403
```

---

### [MINOR] Naming

`update()` on the model is too generic a name for a method with side effects this significant. A name like `update_profile_fields()` or `apply_profile_patch()` signals intent and makes it harder to accidentally call this method from a context where the allowlist assumption does not hold.

---

## What Needs to Happen Before Merge

1. Add a field allowlist to the model method. The method must reject any key not on that list.
2. Add an ownership check in the endpoint. The check must happen before any data is read from the request body.
3. Add tests that assert rejection of sensitive fields and rejection of cross-user requests.

The CI passing and the lint being clean are not relevant to these findings. These are logic vulnerabilities. No static analysis tool flagged them because the code is syntactically correct. The problem is what it does.
