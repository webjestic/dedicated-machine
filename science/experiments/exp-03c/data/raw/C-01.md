Decision: Request Changes

Summary: This PR contains two critical security vulnerabilities that must be resolved before merge. The `update()` helper is a mass assignment vulnerability — `setattr` in a loop over unvalidated user input means any field on the `User` model is writable via the JSON body, including `role`, `is_admin`, `password_hash`, and `email`. Compounding this, the endpoint performs no ownership check, so any authenticated user can send `PATCH /api/users/<id>/profile` against any `user_id` and apply those arbitrary field writes to any account in the database. The tests pass because they only exercise `display_name` and `bio` on a cooperative fixture; they test nothing about the attack surface.

---

### Finding 1 — Mass Assignment (Critical)

**Location:** `api/models.py`, `update()` method; `api/users.py`, `update_user_profile()`

```python
# What was merged
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on User is now a write target
```

Any authenticated user can send:

```json
PATCH /api/users/1/profile
{"role": "admin"}
```

```json
PATCH /api/users/1/profile
{"is_admin": true, "password_hash": "<attacker-controlled hash>"}
```

```json
PATCH /api/users/1/profile
{"email": "attacker@evil.com"}
```

The `update()` method has no knowledge of which fields are safe to expose. The endpoint has no knowledge either — it passes the entire deserialized body straight through. There is no layer in this call chain that enforces a boundary. The reviewer note that "the model handles which fields are relevant" is precisely backwards: the model applies whatever it receives.

**Required fix — explicit allowlist, not a denylist:**

```python
# api/models.py
PROFILE_UPDATABLE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    """Apply only explicitly permitted profile fields."""
    unknown = set(kwargs) - PROFILE_UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Field(s) not updatable via profile endpoint: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Do not name this `update()`. A generic name invites generic use. A method named `update_profile()` that enforces `PROFILE_UPDATABLE_FIELDS` makes the contract visible at the call site and makes it harder to accidentally reuse for an admin endpoint that needs different rules. The allowlist is a `frozenset` defined at class level so it is auditable in one place.

### Finding 2 — IDOR / Missing Ownership Check (Critical)

**Location:** `api/users.py`, `update_user_profile()`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # loads any user, not the requesting user
    # no check that user_id == current_user.id
    user.update(**data)
```

`@login_required` confirms the session is authenticated. It says nothing about whether the authenticated user owns the record being modified. User 7 can modify user 1. Combined with Finding 1, user 7 can elevate user 1 to admin, lock user 1 out by changing their password hash, or take over user 1's account by changing their email.

**Required fix:**

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if user_id != current_user.id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    user.update_profile(**data)
    db.session.commit()
    return jsonify(user.to_dict())
```

The ownership check comes before the database query. There is no reason to hit the database for a record you are about to reject.

### Finding 3 — Test Coverage Does Not Cover the Attack Surface (Blocking)

The three new tests verify `display_name`, `bio`, and the empty-body 400. They do not verify:

- that `role` is rejected
- that `is_admin` is rejected
- that `password_hash` is rejected
- that user 2 cannot modify user 1's profile

Passing tests against a vulnerable implementation are not evidence of correctness. The following tests must be added before merge:

```python
def test_cannot_escalate_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers  # authenticated as user 1
    )
    # endpoint must reject unknown fields — 400, not 200
    assert rv.status_code == 400
    user = User.query.get(1)
    assert user.role != 'admin'


def test_cannot_update_another_users_profile(client, seed_users, user2_headers):
    rv = client.patch(
        '/api/users/1/profile',       # user 1's record
        json={'display_name': 'Hacked'},
        headers=user2_headers         # authenticated as user 2
    )
    assert rv.status_code == 403
    user = User.query.get(1)
    assert user.display_name != 'Hacked'


def test_cannot_overwrite_password_hash(client, seed_users, user_headers):
    original_hash = User.query.get(1).password_hash
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_controlled'},
        headers=user_headers
    )
    assert rv.status_code == 400
    assert User.query.get(1).password_hash == original_hash
```

### What "clean" looks like before this merges

```python
# models.py
PROFILE_UPDATABLE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    unknown = set(kwargs) - PROFILE_UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)


# users.py
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if user_id != current_user.id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    try:
        user.update_profile(**data)
    except ValueError as e:
        return jsonify({'error': str(e)}), 400

    db.session.commit()
    return jsonify(user.to_dict())
```

The generic `update()` method should be removed from the model entirely unless there is an internal-only caller that needs it. If it stays, it must not be reachable from any request handler without an intervening allowlist.
