Decision: Request Changes

Summary: This PR introduces two compounding critical vulnerabilities and cannot be merged in its current state. The `update()` helper is an unrestricted `setattr` loop over unvalidated user input — any field on the `User` model is writable via the JSON body, including `role`, `is_admin`, `password_hash`, and `email`. Layered on top of that, the endpoint performs no ownership check: `@login_required` confirms only that a session exists, not that `current_user.id == user_id`, so any authenticated user can send this payload against any other user's `user_id`. The tests cover `display_name` and `bio` on the happy path and do not touch either attack surface.

---

## Findings

### 1. Mass Assignment — Critical

**Location:** `api/models.py`, `update()` method

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

This is not a helper. This is an open door to every column on the `User` model. There is no allowlist, no type coercion, no field validation of any kind. The loop will `setattr` whatever the caller sends.

A request body of:

```json
{"role": "admin"}
```

promotes an arbitrary user to admin. A body of:

```json
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

overwrites the password without knowing the current one. A body of:

```json
{"email": "victim@example.com"}
```

takes over an account by redirecting password reset flow.

I have written the postmortem for exactly this pattern. The fact that the endpoint is described as "for the account settings page" does not constrain what a client actually sends. The model does not know it is being called from a settings page. The `update()` method will accept `role` just as readily as `bio`.

**Required fix:** Replace the `setattr` loop with an explicit allowlist.

```python
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_UPDATABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist lives on the model, not in the caller, so every future call site inherits the constraint. If you add a new field to `PROFILE_UPDATABLE_FIELDS` you make a conscious, reviewable decision. If you add a new column to the model, it is not automatically exposed.

### 2. IDOR — Critical

**Location:** `api/users.py`, `update_user_profile()`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    ...
    user.update(**data)
```

`@login_required` answers one question: is there a valid session? It does not answer: does this session belong to the user being modified? There is no check that `current_user.id == user_id` before the update is applied.

Any authenticated user can iterate `user_id` values and modify other users' profiles. Combined with finding #1, any authenticated user can escalate any account — including accounts they do not own — to admin.

**Required fix:** Enforce ownership before touching the record. Admins may have a legitimate reason to edit other users' profiles; if so, that path should be explicit and separately gated.

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

Note the ownership check comes before the database query. There is no reason to hit the database for a record you are about to reject.

### 3. Tests Do Not Cover the Attack Surface

**Location:** `tests/test_users.py`

The three new tests verify:
- `display_name` can be updated
- `bio` can be updated
- an empty body returns 400

None of them test:
- that sending `{"role": "admin"}` is rejected
- that sending `{"password_hash": "x"}` is rejected
- that user A cannot modify user B's profile

Passing CI on these tests is not evidence of security. It is evidence that the happy path works. These are different things.

**Required additions — these should be blocking before merge:**

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # Must not silently succeed
    assert rv.status_code in (400, 422)
    user = User.query.get(1)
    assert user.role != 'admin'


def test_cannot_update_another_users_profile(client, seed_users,
                                              user_headers, other_user_id):
    rv = client.patch(
        f'/api/users/{other_user_id}/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers  # authenticated as a different user
    )
    assert rv.status_code == 403


def test_cannot_overwrite_password_hash(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_hash'},
        headers=user_headers
    )
    assert rv.status_code in (400, 422)
```

### 4. Minor — `admin_required` Comparison

This is pre-existing and outside the scope of this PR, but worth flagging while the file is open:

```python
if not current_user.role == 'admin':
```

This should be:

```python
if current_user.role != 'admin':
```

The current form is not a vulnerability — operator precedence evaluates it correctly — but it reads ambiguously under review. A future reader might misparse it as `not current_user.role` evaluated first. Clarity in auth checks costs nothing.
