Decision: **Request Changes**

Summary: This PR ships two compounding critical vulnerabilities: mass assignment via an allowlist-free `setattr` loop, and IDOR because the endpoint never verifies that `current_user.id == user_id`. Any authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate privileges on any account in the system. The three tests cover only the happy path on a single field each and provide zero coverage of the attack surface — CI passing here is meaningless.

---

## Findings

### 1. Mass Assignment — Critical

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every column on the User model is now a write endpoint
```

The `update()` helper is not a safe abstraction. It is a `setattr` loop with no allowlist, which means the "which fields are relevant" claim in the reviewer notes is false — the model does not filter anything. Every column the ORM maps is writable by anyone who can reach this endpoint. On a typical `User` model that includes:

- `role`
- `is_admin` / `is_staff`
- `password_hash`
- `email`
- `verified` / `email_confirmed`
- any foreign key or internal flag you have added since the model was created

A request body of `{"role": "admin"}` is all it takes. I have written the postmortem for exactly this. It took fourteen hours to detect. The fix is not to be more careful about what you send — the fix is to make the dangerous operation structurally impossible.

**Required fix:** Replace the loop with an explicit allowlist before any `setattr` is called.

```python
PROFILE_UPDATABLE_FIELDS = {"display_name", "bio", "avatar_url"}

def update_profile(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_UPDATABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist lives at the model layer so it cannot be bypassed by a second caller. The generic `update()` method should be removed or never exposed to request data under any circumstances.

### 2. IDOR — Critical

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches whoever the caller names
    # current_user is never consulted again
    user.update(**data)
```

`@login_required` confirms the caller is authenticated. It says nothing about whether the caller is the owner of `user_id`. Any authenticated user can target any other user's record. Combined with finding 1, this means a regular user can escalate any account, not just their own.

**Required fix:**

```python
if user_id != current_user.id:
    return jsonify({'error': 'Forbidden'}), 403
```

This check must come before the `get_or_404` call, or at minimum before any mutation. If admin users need to edit other accounts, that path should go through a separate endpoint protected by `@admin_required`, not through a settings-page endpoint with a user-ID parameter.

---

### 3. Tests Do Not Cover the Attack Surface

All three new tests authenticate as the same user, target user ID 1, and send only `display_name` or `bio`. They confirm the happy path works. They do not test:

- A user sending `{"role": "admin"}` — should be rejected, currently succeeds
- A user sending `{"password_hash": "..."}` — same
- User 2 patching user 1's record — should be 403, currently 200
- An unauthenticated request — should be 401

Passing CI on this test suite is not a signal of safety. It is a signal that the tests were written to match the implementation rather than to probe its boundaries.

**Required additions before merge:**

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch('/api/users/1/profile', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400  # or 422 — must not be 200

def test_cannot_update_another_users_profile(client, seed_users, other_user_headers):
    rv = client.patch('/api/users/1/profile', json={'bio': 'x'}, headers=other_user_headers)
    assert rv.status_code == 403

def test_unauthenticated_request_is_rejected(client, seed_users):
    rv = client.patch('/api/users/1/profile', json={'bio': 'x'})
    assert rv.status_code == 401
```

---

## What Needs to Happen Before This Merges

1. Remove the generic `update(**kwargs)` method from the model, or gate it so it is never reachable from request data
2. Add `update_profile()` with an explicit `PROFILE_UPDATABLE_FIELDS` allowlist
3. Add the `current_user.id == user_id` ownership check at the top of the endpoint
4. Add tests that assert rejection of privileged fields, cross-user modification, and unauthenticated access

The endpoint logic is four lines. The vulnerabilities are in three of them. The reviewer note that "the model handles which fields are relevant" is the exact reasoning that produced the incident I documented. That framing needs to be corrected, not merged.
