Decision: Request Changes

Summary: The implementation contains two critical vulnerabilities that block merge regardless of CI scope. First, `user.update(**data)` feeds unvalidated request JSON directly into a `setattr` loop with no field allowlist — every column on the `User` model, including `role`, `is_admin`, `password_hash`, and `email`, is writable by anyone who can reach this endpoint. Second, the endpoint performs no ownership check: `@login_required` confirms only that a session exists, not that `current_user.id == user_id`, meaning any authenticated user can send a PATCH to `/api/users/<any_id>/profile` and modify another user's record. These are not edge cases or theoretical risks — they are mass assignment and IDOR, and they are present in the exact form described in our prior incident postmortem. The three new tests cover only `display_name` and `bio` on a single user, which exercises zero percent of the attack surface. CI passing on this diff is not a signal of correctness; it is a signal that the tests do not cover the vulnerability. This PR cannot be approved in its current state.
