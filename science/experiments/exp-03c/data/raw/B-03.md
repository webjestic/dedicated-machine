Decision: Request Changes

Summary: The implementation does not match the PR description's implied scope — the `update()` model method applies `setattr` against arbitrary keyword arguments with no field allowlist, meaning any column on the User model (e.g., `role`, `password_hash`, `id`) can be written through this endpoint. The three new tests only exercise `display_name` and `bio`, leaving the actual behavior of the `update()` helper undertested relative to what it permits. Before this can merge, the `update()` method (or the endpoint itself) needs an explicit allowlist of patchable fields, and tests should confirm that disallowed fields are rejected or ignored.
