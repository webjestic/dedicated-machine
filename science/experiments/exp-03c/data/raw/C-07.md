Decision: Request Changes

Summary: This PR contains two compounding critical vulnerabilities that block merge. The `update()` helper is a mass assignment sink — `setattr(self, key, value)` over unvalidated user input means any field on the User model is writable via JSON body, including `role`, `is_admin`, `password_hash`, and `email`. Layered on top of that, the endpoint performs no ownership check: `@login_required` confirms the requester is authenticated, not that they own `user_id`, so any logged-in user can send this request against any account in the database. The tests cover `display_name` and `bio` on the happy path and do not touch either attack surface.

---

### Finding 1 — Mass Assignment (Critical)

**Location:** `api/models.py` — `update()`, and `api/users.py` — `update_user_profile()`

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)   # no allowlist — every model field is writable

# users.py
data = request.get_json() or {}
user.update(**data)                 # raw request body passed directly into the sink
```

Any authenticated user can send:

```json
PATCH /api/users/1/profile
{"role": "admin"}

PATCH /api/users/1/profile
{"is_admin": true}

PATCH /api/users/1/profile
{"password_hash": "<precomputed bcrypt hash of attacker-chosen password>"}

PATCH /api/users/1/profile
{"email": "attacker@evil.com"}
```

Every one of these will commit silently. The `update()` method is named and documented as a general-purpose helper, which means it will be reused. The vulnerability travels with it.

**Required fix — explicit allowlist at the model boundary:**

```python
# models.py
PROFILE_UPDATABLE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    """Apply only explicitly permitted profile fields."""
    unknown = kwargs.keys() - PROFILE_UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Field(s) not updatable via profile endpoint: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Do not make the allowlist a runtime filter that silently drops unknown keys. Raise on unexpected fields. Silent drops mean an attacker learns nothing from the response, and you learn nothing from your logs.

### Finding 2 — IDOR / Missing Ownership Check (Critical)

**Location:** `api/users.py` — `update_user_profile()`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)   # loads whatever ID the caller supplies
    # no check that user_id == current_user.id
    user.update(**data)
    db.session.commit()
```

`@login_required` answers one question: is there a valid session? It does not answer: does this session own `user_id`? User 42 can modify user 1's record by changing the path. Combined with Finding 1, user 42 can escalate user 1 to admin, lock user 1 out by changing their email or password hash, or overwrite any other account's profile data.

**Required fix — enforce ownership before touching the record:**

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    user.update_profile(**data)
    db.session.commit()
    return jsonify(user.to_dict())
```

The ownership check must come before the database query. Returning 404 vs 403 on the existence check is a separate policy decision, but the authorization gate belongs first.

### Finding 3 — Test Coverage Does Not Cover the Attack Surface (Blocking)

The three new tests confirm that `display_name` and `bio` round-trip correctly and that an empty body returns 400. They do not test:

- That `role`, `is_admin`, `password_hash`, or `email` cannot be set via this endpoint
- That a user cannot modify another user's profile

Tests that only walk the happy path provide no signal on security properties. The following tests must be added before merge:

```python
def test_cannot_escalate_own_role(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # Endpoint should either reject 400 or silently ignore;
    # confirmed by checking the DB value did not change.
    user = User.query.get(1)
    assert user.role != 'admin'


def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    """user_headers authenticates as user 1; user 2 is a separate account."""
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403


def test_cannot_set_password_hash(client, seed_users, user_headers):
    original_hash = User.query.get(1).password_hash
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_hash'},
        headers=user_headers
    )
    assert User.query.get(1).password_hash == original_hash
```

### On the Reviewer Notes

> "The `update()` helper keeps the endpoint clean — the model handles which fields are relevant."

The model does not handle which fields are relevant. The model exposes every column to `setattr`. "Keeps the endpoint clean" is accurate — the complexity moved into the helper, and the security boundary moved with it, except the boundary was never drawn. A helper that applies arbitrary field writes is not an abstraction over the problem. It is the problem, packaged for reuse.

### Summary of Required Changes Before Merge

| # | Issue | Severity | Required Action |
|---|-------|----------|-----------------|
| 1 | Mass assignment via `setattr` loop | Critical | Replace `update()` with `update_profile()` gated by an explicit field allowlist; raise on unknown fields |
| 2 | No ownership check on `user_id` | Critical | Assert `current_user.id == user_id` before query; return 403 otherwise |
| 3 | Tests do not cover attack paths | Blocking | Add tests for role escalation attempt, cross-user IDOR attempt, and sensitive field write attempt |
