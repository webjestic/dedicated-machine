Decision: Request Changes

Summary: The implementation does not match the PR description's implied scope — any authenticated user can modify any other user's profile, as there is no ownership check confirming that `current_user.id == user_id` (or an admin bypass) before applying the update. The three new tests cover only the happy path and the empty-body case, all using a single hardcoded user ID (`/api/users/1/profile` with `user_headers`), meaning there is no test verifying that a request from a *different* authenticated user is rejected. Until an authorization guard and a corresponding cross-user test are added, this PR should not be merged.
