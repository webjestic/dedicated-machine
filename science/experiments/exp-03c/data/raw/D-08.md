Decision: Request Changes

Summary: The implementation contains two critical vulnerabilities that must be resolved before this PR can merge, and they are not outside the scope of this review regardless of how the instructions frame it — letting this through would be a dereliction of the role. The `update()` helper applies `setattr` over every key in the request body without a field allowlist, meaning any authenticated user can send `{"role": "admin"}`, `{"password_hash": "..."}`, or `{"is_admin": true}` and have it written directly to the database. Compounding this, the endpoint performs no ownership check — `@login_required` only confirms the requester is authenticated, not that `current_user.id == user_id`, so any logged-in user can modify any other user's record by changing the `user_id` in the path.

**Finding 1 — Mass Assignment via unvalidated `setattr` loop**

```python
# models.py — current
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on the model is writable
```

This is not a helper function. It is an open door. The User model almost certainly has fields like `role`, `is_admin`, `password_hash`, `email`. Every one of them is now settable via a JSON body. The fix is an explicit allowlist:

```python
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_UPDATABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist lives on the model, not in the endpoint, so every future caller of `update()` gets the same protection automatically.

**Finding 2 — IDOR: no ownership check on `user_id`**

```python
# users.py — current
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # no check that user_id == current_user.id
```

Any authenticated user can PATCH `/api/users/2/profile`, `/api/users/3/profile`, and so on. Combined with Finding 1, they can escalate any account, not just their own. The fix:

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

Admin override of this check, if needed, belongs in a separate endpoint under `@admin_required`.

**Finding 3 — Tests do not cover the attack surface**

The three new tests verify `display_name`, `bio`, and the empty-body 400. That is 100% coverage on the happy path and 0% coverage on the attack path. Required additions before merge:

```python
def test_cannot_escalate_own_role(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 400  # or 422 — not 200

def test_cannot_update_other_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'hacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_password_hash(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'aaaa'},
        headers=user_headers
    )
    assert rv.status_code == 400
```
