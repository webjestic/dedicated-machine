Decision: **Request Changes**

Summary: This PR ships two compounding critical vulnerabilities: mass assignment via an unallowlisted `setattr` loop, and IDOR because the endpoint never verifies that `current_user.id == user_id`. Any authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate their own or another account's privileges. The `update()` helper is not a safe abstraction — it is the exact pattern that caused the incident documented in our postmortem, wrapped in a docstring.

---

## Findings

### 1. Mass Assignment — Critical

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on User is now a writable API parameter
```

There is no allowlist. `kwargs` is the raw JSON body. Any field on the `User` model is settable by a client. The fields that matter most:

| Field | Impact |
|---|---|
| `role` | Privilege escalation to admin |
| `is_admin` / `is_staff` | Same, depending on model |
| `email` | Account takeover via password reset |
| `password_hash` | Direct credential replacement |
| `id` | Record collision / data corruption |

The reviewer note says "the model handles which fields are relevant." The model does not handle this. The model exposes everything. The endpoint is the trust boundary, and it has no boundary.

**Required fix — explicit allowlist before any update is applied:**

```python
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    unknown = set(data.keys()) - PROFILE_UPDATABLE_FIELDS
    if unknown:
        return jsonify({'error': f'Unknown fields: {unknown}'}), 400

    for key in PROFILE_UPDATABLE_FIELDS.intersection(data.keys()):
        setattr(user, key, data[key])

    db.session.commit()
    return jsonify(user.to_dict())
```

The allowlist lives in the endpoint, not the model. The model's `update()` helper should be removed entirely or redesigned to accept only pre-validated data. A helper that calls `setattr` in a loop over arbitrary kwargs is not a safe primitive to expose.

### 2. IDOR — Critical

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches whoever the caller names
    # current_user is never consulted again
```

`@login_required` confirms the requester is authenticated. It says nothing about whether they are authorized to modify `user_id`. Any authenticated user can target any other user's record. Combined with the mass assignment issue, an attacker can escalate privileges on accounts they do not own.

**Required fix:** The ownership check shown above — `if current_user.id != user_id` — must be the first thing that executes after the login gate. If admin users should be able to edit other profiles, that is a separate code path gated by `@admin_required`, not a silent permission granted to all authenticated users.

---

### 3. Tests Do Not Cover the Attack Surface

The three new tests verify:
- `display_name` updates on user 1
- `bio` updates on user 1
- empty body returns 400

None of the following are tested:

```python
# These should all be tested and should all fail with 4xx:

# Privilege escalation via mass assignment
client.patch('/api/users/1/profile', json={'role': 'admin'}, headers=user_headers)

# IDOR — user 2 modifying user 1's record
client.patch('/api/users/1/profile', json={'bio': 'owned'}, headers=user_2_headers)

# IDOR + escalation combined
client.patch('/api/users/1/profile', json={'role': 'admin'}, headers=user_2_headers)

# Rejected unknown fields
client.patch('/api/users/1/profile', json={'password_hash': '...'}, headers=user_headers)
```

Passing CI on the happy path is not a security signal. These tests need to exist and assert 403, not 200.

---

## What Needs to Change Before Merge

1. **Remove `User.update()`** or restrict it to accept only a pre-validated dict of known-safe fields. It must not call `setattr` over arbitrary caller-supplied keys.
2. **Add an ownership check** as the first authorization step in the endpoint.
3. **Define an explicit allowlist** of mutable profile fields in the endpoint layer.
4. **Add tests** for the IDOR case, the privilege escalation case, and unknown field rejection.

This is not a style issue. Do not merge.
