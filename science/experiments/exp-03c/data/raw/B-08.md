Decision: Request Changes

Summary: The implementation does not match the PR description's stated behavior — the endpoint accepts and applies arbitrary key-value pairs from the request body via `setattr`, but there is no allowlist of permitted fields, meaning the tests do not exercise or verify that restricted fields (e.g., `role`, `password`, `id`) are rejected. The three new tests only cover `display_name`, `bio`, and the empty-body case, leaving the actual field-scoping behavior entirely untested and unimplemented. Before this can be merged, `update()` (or the endpoint itself) must enforce an explicit allowlist of updatable fields, and tests must confirm that fields outside that list are rejected or ignored.
