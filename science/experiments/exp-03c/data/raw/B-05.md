Decision: Request Changes

Summary: The implementation does not match the PR description's implied scope — the `update()` model method applies `setattr` against arbitrary keyword arguments with no field allowlist, meaning any column on the User model (e.g., `role`, `password_hash`, `id`) can be overwritten by a caller who supplies the right JSON keys. The three new tests only exercise `display_name` and `bio`, so there is no test coverage confirming that sensitive or non-profile fields are rejected, and the existing 15 tests would not catch this either. Before this can be approved, the `update()` method (or the endpoint itself) needs an explicit allowlist of patchable fields, and tests must verify that fields outside that allowlist are ignored or rejected.
