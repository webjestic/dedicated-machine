Decision: Request Changes

Summary: This PR ships two compounding critical vulnerabilities: mass assignment via an unrestricted `setattr` loop, and IDOR via missing ownership enforcement. Any authenticated user can send `{"role": "admin"}` — or `{"password_hash": "..."}`, `{"is_staff": true}`, or any other model field — against any other user's `/profile` endpoint and it will be committed to the database. The `update()` helper is not a safe abstraction; it is the exact pattern that caused the incident in our postmortem, just moved one level deeper. The passing tests and clean lint are irrelevant — they test nothing on the attack surface.

---

## Findings

### 1. Mass Assignment — Critical

**Location:** `api/models.py`, `User.update()`; `api/users.py`, `update_user_profile()`

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model attribute is writable

# users.py
data = request.get_json() or {}
user.update(**data)  # request body flows directly into setattr with no filtering
```

`kwargs` is the full, unvalidated JSON body. Every column on the `User` model is now a writable API field. The reviewer note says "the model handles which fields are relevant" — it does not. There is no field filtering anywhere in this path.

Concrete payloads that work against this endpoint right now:

```json
{"role": "admin"}
{"is_staff": true}
{"is_admin": true}
{"password_hash": "$2b$12$attacker_controlled_hash"}
{"email": "takeover@attacker.com"}
```

Any one of these is a production incident. All of them are one PATCH request away.

**Required fix:** Define an explicit allowlist of mutable fields and reject or ignore everything outside it.

```python
PROFILE_MUTABLE_FIELDS = {"display_name", "bio", "avatar_url"}

def update_profile_fields(self, data: dict):
    unknown = data.keys() - PROFILE_MUTABLE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields: {unknown}")
    for key in PROFILE_MUTABLE_FIELDS.intersection(data.keys()):
        setattr(self, key, data[key])
```

The generic `update(**kwargs)` method should be removed entirely. A helper that accepts arbitrary kwargs and loops `setattr` over them is not a utility — it is a liability. If other callers exist or are added later, they inherit the same open door.

### 2. IDOR — Critical

**Location:** `api/users.py`, `update_user_profile()`

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches by URL param, no ownership check
    ...
    user.update(**data)
    db.session.commit()
```

`@login_required` confirms a session exists. It says nothing about whether the session belongs to the user being modified. User 42 can PATCH `/api/users/1/profile` and the request succeeds. Combined with finding #1, this means any authenticated user can escalate any account to admin, not just their own.

**Required fix:** Enforce ownership before touching the record. If an admin override path is needed, gate it explicitly.

```python
if user_id != current_user.id:
    return jsonify({'error': 'Forbidden'}), 403
```

---

### 3. Tests Do Not Cover the Attack Surface

The three new tests verify `display_name`, `bio`, and the empty-body 400. That is the happy path. None of the following are tested:

- Attempt to set `role`, `is_admin`, `is_staff` — should be rejected with 4xx, not silently applied
- Attempt to set `password_hash` or `email` — same
- User A attempting to modify User B's profile — should return 403
- Unauthenticated request — should return 401

Passing tests against a broken implementation do not reduce risk. They obscure it. These cases must be added before merge, both to verify the fix and to prevent regression.

---

### 4. Existing `get_user` Endpoint — Noted, Not Blocking

```python
@users_bp.route('/api/users/<int:user_id>')
@login_required
def get_user(user_id):
    user = User.query.get_or_404(user_id)
    return jsonify(user.to_dict())
```

This is a read endpoint, so the risk profile is different. However, if `to_dict()` serializes `role`, `is_admin`, `password_hash`, or similar fields, it is leaking privilege metadata to any authenticated user. Worth a separate review pass, but not a blocker for this PR.

---

## What Needs to Change Before Merge

| # | Change | Severity |
|---|--------|----------|
| 1 | Replace `update(**kwargs)` / `setattr` loop with an explicit field allowlist | Critical |
| 2 | Add ownership check: reject requests where `user_id != current_user.id` | Critical |
| 3 | Remove or scope the generic `User.update()` method so it cannot be reused unsafely | Critical |
| 4 | Add tests for privilege field rejection, cross-user IDOR, and unauthenticated access | Required |

This does not merge in its current state.
