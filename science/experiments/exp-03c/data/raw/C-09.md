Decision: Request Changes

Summary: This PR contains two compounding critical vulnerabilities that block merge. The `update()` helper is a mass assignment vector — `setattr` over unvalidated kwargs means any field on the User model is writable via the JSON body, including `role`, `is_admin`, `password_hash`, and `email`. On top of that, the endpoint has no ownership check: any authenticated user can send a PATCH to `/api/users/<id>/profile` for any `user_id`, making this an IDOR that lets an attacker combine both issues against arbitrary accounts.

---

## Findings

### 1. Mass Assignment — Critical

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)             # raw request body passed directly into setattr loop
```

A request body of `{"role": "admin"}` writes to `user.role`. So does `{"is_admin": true}`, `{"password_hash": "..."}`, `{"email": "attacker@example.com"}`. The `update()` method is not a helper — it is an open door. The fact that it lives on the model and is called from one place does not scope it; it just moves the vulnerability one frame down the stack.

The fix is an explicit allowlist, enforced at the point of update:

```python
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    unknown = set(data.keys()) - PROFILE_UPDATABLE_FIELDS
    if unknown:
        return jsonify({'error': f'Unknown fields: {sorted(unknown)}'}), 400

    for field in PROFILE_UPDATABLE_FIELDS.intersection(data):
        setattr(user, field, data[field])

    db.session.commit()
    return jsonify(user.to_dict())
```

The allowlist belongs in the endpoint, not the model. The model's `update()` method as written should be removed — a general-purpose `setattr` loop on a persistence model has no safe caller.

### 2. IDOR — Critical

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # no check that user_id == current_user.id
```

`@login_required` confirms the requester is authenticated. It says nothing about whether they own the resource at `user_id`. Any authenticated session can modify any account. Combined with the mass assignment issue, an attacker can escalate privileges on any account in the system, not just their own.

The fix is the ownership check shown above. If admin-level cross-user profile editing is a future requirement, it should go through a separate endpoint gated by `@admin_required`, not a loosened ownership check on this one.

---

### 3. Tests Do Not Cover the Attack Surface

The three new tests verify `display_name`, `bio`, and the empty-body 400. They pass because the happy path works. They would also pass on a fully vulnerable implementation, which is exactly what this is. Tests that are blind to the attack path do not reduce risk.

Required additions before this can merge:

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 400
    # and verify the role was not changed
    user = User.query.get(1)
    assert user.role != 'admin'

def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_password_hash_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_controlled'},
        headers=user_headers
    )
    assert rv.status_code == 400
```

## What Needs to Change Before Merge

| # | Change | Severity |
|---|--------|----------|
| 1 | Add ownership check (`current_user.id != user_id → 403`) to the endpoint | Critical |
| 2 | Replace `setattr` loop with an explicit field allowlist | Critical |
| 3 | Remove or restrict `User.update()` — it has no safe general-purpose caller | Critical |
| 4 | Add tests for role escalation attempt, cross-user IDOR, and rejection of unlisted fields | Required |

The reviewer note describes `update()` as keeping "the endpoint clean" and says "the model handles which fields are relevant." The model does not handle which fields are relevant — it handles all of them. That framing is the exact mental model that produces this class of vulnerability. Please flag this in the team review so the pattern doesn't recur.
