Decision: Request Changes

Summary: This PR contains two critical security vulnerabilities that must be resolved before merge. The `update()` helper is a mass assignment vector — `setattr` in a loop over unvalidated user input means any field on the `User` model is writable via the JSON body, including `role`, `is_admin`, `password_hash`, and `email`. Compounding this, the endpoint performs no ownership check, so any authenticated user can send `PATCH /api/users/<id>/profile` against any other user's record — that is a textbook IDOR. The tests cover only `display_name` and `bio` on a single happy path, which provides zero coverage of either attack surface.

---

### Finding 1 — Mass Assignment (Critical)

**Location:** `api/models.py` — `update()`, and `api/users.py` — `update_user_profile()`

```python
# What ships in this PR:
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on User is now a writable API parameter
```

Any authenticated user can send:

```json
PATCH /api/users/1/profile
{"role": "admin"}
```

```json
PATCH /api/users/1/profile
{"password_hash": "<bcrypt hash of attacker's chosen password>"}
```

```json
PATCH /api/users/1/profile
{"is_staff": true, "email": "attacker@evil.com"}
```

The `update()` method has no knowledge of which fields are safe to expose. The endpoint has no knowledge of which fields `update()` will touch. Neither layer enforces a boundary. This is the same class of vulnerability as the incident in 2023 — a `setattr` loop over unvalidated input with no allowlist.

**Required fix — define the allowlist at the model layer and enforce it:**

```python
# api/models.py

PROFILE_UPDATABLE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})
# Enumerate exactly what the settings page is allowed to touch.
# Every other field on User is implicitly off-limits.

def update_profile(self, **kwargs):
    """Apply profile field updates. Only fields in PROFILE_UPDATABLE_FIELDS
    are accepted. Unknown or sensitive fields raise ValueError rather than
    silently applying or silently ignoring — silent ignore masks misconfiguration."""
    unknown = kwargs.keys() - PROFILE_UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields in request: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Rename the method so it cannot be confused with a general-purpose setter. If a future developer adds a field to `User` and forgets to think about this endpoint, the allowlist means the new field is protected by default. An unbounded `setattr` loop means the new field is exposed by default. Default-safe is the only acceptable posture.

### Finding 2 — Insecure Direct Object Reference (Critical)

**Location:** `api/users.py` — `update_user_profile()`

```python
# What ships in this PR:
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # no check that user_id == current_user.id
    user.update(**data)
```

`@login_required` confirms the request is authenticated. It says nothing about whether the authenticated user owns the record they are modifying. User 42 can modify user 1. User 1 can modify user 42. Every account in the system is writable by every other account. Combined with Finding 1, a regular user can escalate any account, not just their own.

**Required fix — scope the operation to the requesting user:**

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if user_id != current_user.id:
        # Do not 404 here — 404 leaks whether the user_id exists.
        # 403 is correct: the resource exists, the caller is not authorized.
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    try:
        current_user.update_profile(**data)
    except ValueError as e:
        return jsonify({'error': str(e)}), 422

    db.session.commit()
    return jsonify(current_user.to_dict())
```

Note: once the ownership check is in place, `User.query.get_or_404(user_id)` is unnecessary — `current_user` is already the correct object. Removing the query also eliminates the information leak from a 404 on a non-existent `user_id` reached via the IDOR path.

### Finding 3 — Test Coverage Does Not Cover the Attack Surface (Blocking)

The three new tests confirm that `display_name` and `bio` round-trip correctly and that an empty body returns 400. They do not test:

- That `role` is rejected
- That `is_admin` / `is_staff` are rejected
- That `password_hash` is rejected
- That user 2 cannot modify user 1's profile

Tests that only exercise the happy path give a false signal. CI is green. Coverage is 89%. Neither number means the vulnerability is absent — it means the vulnerability was not looked for. The following tests must be added before merge:

```python
@pytest.mark.parametrize("field,value", [
    ("role", "admin"),
    ("is_admin", True),
    ("is_staff", True),
    ("password_hash", "$2b$12$fakehash"),
    ("email", "hijacked@evil.com"),
])
def test_sensitive_fields_are_rejected(client, seed_users, user_headers, field, value):
    """Allowlist enforcement: sensitive fields must not be accepted."""
    rv = client.patch(
        '/api/users/1/profile',
        json={field: value},
        headers=user_headers
    )
    assert rv.status_code == 422


def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    """IDOR: authenticated user must not be able to modify a different user's record."""
    # user_headers authenticates as user 1; target is user 2
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Injected'},
        headers=user_headers
    )
    assert rv.status_code == 403


def test_cross_user_role_escalation_blocked(client, seed_users, user_headers):
    """Combined IDOR + mass assignment: must not be able to escalate another account."""
    rv = client.patch(
        '/api/users/2/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 403
```

### What Needs to Happen Before This Merges

| # | Item | File |
|---|------|------|
| 1 | Replace unbounded `setattr` loop with an explicit allowlist in a renamed method | `api/models.py` |
| 2 | Add ownership check (`user_id != current_user.id → 403`) before any data access | `api/users.py` |
| 3 | Remove the redundant `get_or_404` query after the ownership check is in place | `api/users.py` |
| 4 | Add tests for sensitive field rejection and cross-user modification | `tests/test_users.py` |

The reviewer note describes `update()` as keeping "the endpoint clean" because "the model handles which fields are relevant." The model does not handle which fields are relevant — it applies every field it receives. That framing is the conceptual error that produces this class of bug. The allowlist is not an implementation detail to be left to the model's discretion. It is a security boundary, and it has to be explicit, reviewed, and tested.
