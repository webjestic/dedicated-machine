Decision: Request Changes

Summary: The senior engineer's flag is the critical issue here — and the implementation does not resolve it. The heartbeat thread is suspended alongside all other threads during a stop-the-world GC pause or OS preemption event, meaning lock expiry is not prevented in that scenario, and the process can resume and execute `db.update_stock()` after another process has legitimately acquired the lock and begun its own write. The fix is not in the locking layer at all — it requires a database-level fencing mechanism. Beyond that core flaw, there are secondary architectural and correctness issues that also need to be addressed before this can be approved.

## Detailed Review

### Critical: Stale Write After Lock Expiry Is Not Prevented

The reviewer note identifies this precisely, and the answer to the question posed is: **no, this implementation does not prevent the stale write.**

Here is the failure sequence:

1. Process A acquires the lock and starts the heartbeat thread.
2. The OS preempts the entire process (GC pause, scheduler decision, etc.). All threads — including the heartbeat — are suspended simultaneously. The heartbeat cannot extend the TTL.
3. The lock TTL (`LOCK_TTL = 30s`) expires during the pause.
4. Process B acquires the lock, reads stock, and begins its own write sequence.
5. Process A resumes. It has no awareness that the lock expired. The `finally` block token check in `_acquire_lock` will correctly skip the `redis.delete()` call — but that check happens *after* `yield`, meaning `db.update_stock()` and `db.create_order()` have already executed inside the `with` block.
6. Both Process A and Process B have now written to the database. The lock provided no protection.

The heartbeat is a useful mitigation for *slow but running* operations, but it is not a solution for process suspension. This is a well-known limitation of distributed locks — Martin Kleppmann's fencing token critique of Redlock applies directly here.

**Required fix:** The database write layer must enforce a fencing token or optimistic concurrency check. Two viable approaches:

**Option 1 — Fencing token via monotonic lock version:**
Use a Redis counter or a monotonic token as a fencing value. Pass it into the DB write. The database rejects any write carrying a token older than the last committed token.

```python
# Acquire a monotonically increasing fence token
fence_token = self.redis.incr("inventory:fence:{item_id}")

# Pass it to the DB layer
self.db.update_stock(item_id, new_stock, fence_token=fence_token)

# DB layer enforces:
# UPDATE inventory SET stock = %s, last_fence = %s
# WHERE item_id = %s AND last_fence < %s
# -- reject (raise) if rows_affected == 0
```

**Option 2 — Optimistic locking with a version column:**
Read a `version` field from the DB alongside stock. Include it in the `WHERE` clause of the update. If the row was modified by another writer, the update affects zero rows and the operation fails explicitly.

```python
stock, version = self.db.get_stock_versioned(item_id)
# ...
rows_updated = self.db.update_stock(item_id, new_stock, expected_version=version)
if rows_updated == 0:
    raise StaleWriteError(f"Concurrent modification detected for item {item_id}")
```

Either approach ensures the database itself is the final arbiter of write validity, regardless of what the distributed lock layer believes.

### Significant: `InventoryService` Is Not Safe for Concurrent Use

The heartbeat state is stored as instance variables (`_heartbeat_active`, `_heartbeat_thread`). If two coroutines or threads call `reserve_item()` on the same `InventoryService` instance concurrently, they will overwrite each other's heartbeat state. `_stop_heartbeat()` sets `_heartbeat_active = False`, which will terminate the heartbeat for *both* concurrent operations.

```python
# Thread 1 acquires lock, starts heartbeat → _heartbeat_active = True
# Thread 2 acquires lock, starts heartbeat → _heartbeat_active = True (same flag)
# Thread 1 finishes → _stop_heartbeat() sets _heartbeat_active = False
# Thread 2's heartbeat thread now exits silently — lock will expire
```

**Fix:** Scope heartbeat state to the lock context, not the instance. Use a local `threading.Event` per lock acquisition.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ...
    heartbeat_thread = threading.Thread(
        target=self._beat, args=(lock_key, token, stop_event), daemon=True
    )
    heartbeat_thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        # release lock ...
```

### Moderate: Heartbeat Race Condition on Lock Release

In the `finally` block of `_acquire_lock`:

```python
self._stop_heartbeat()
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

There is a TOCTOU race between `GET` and `DELETE`. Another process could acquire the lock in the window between these two calls. The correct pattern is an atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

release = self.redis.register_script(RELEASE_SCRIPT)
release(keys=[lock_key], args=[token])
```

### Moderate: `quantity` Has No Validation

`quantity` is accepted as a plain `int` with no bounds checking. A caller passing `quantity=0` or `quantity=-5` would pass the stock check and produce a nonsensical reservation.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

### Minor: `item_id` Has No Validation

An empty string or a value containing Redis key separator characters (`:`) would produce a malformed lock key like `inventory:lock:`. At minimum:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item ID")
```

---

### Minor: `db.update_stock()` and `db.create_order()` Are Not Atomic

If `update_stock()` succeeds and `create_order()` raises, stock is decremented but no order record exists. These two operations should be wrapped in a database transaction. The DB interface should expose a transactional boundary, or both calls should be composed into a single transactional operation.

---

### Test Coverage Gap

The tests mock out `_start_heartbeat` and `_stop_heartbeat` at the fixture level, which means the heartbeat logic itself is never exercised. There is no test for:

- The stale write scenario (which is expected given the current implementation doesn't handle it, but a test should exist to document the expected behavior once fixed)
- Concurrent instance-level heartbeat state corruption
- The `quantity <= 0` edge case
- `db.create_order()` failure after `db.update_stock()` succeeds

`test_lock_not_released_if_expired` is a good test, but it only validates the Redis `delete` is skipped — it does not assert that the DB write was also skipped, which is the actual safety property that needs to be guaranteed.

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| Critical | DB write layer must enforce fencing token or optimistic locking to prevent stale writes after lock expiry |
| Significant | Heartbeat state must be scoped per lock acquisition, not per instance |
| Moderate | Lock release must use atomic Lua script (GET+DEL) |
| Moderate | `quantity` must be validated as a positive integer |
| Minor | `item_id` must be validated as non-empty |
| Minor | `update_stock` + `create_order` must execute within a DB transaction |
| Test | Add tests for the above scenarios; remove blanket heartbeat mocking |
