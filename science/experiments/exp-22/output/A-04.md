Decision: Request Changes

Summary: The senior engineer's flag is a genuine, unresolved correctness flaw — not a theoretical edge case. The heartbeat thread is suspended during any stop-the-world pause alongside the main thread, meaning the lock can expire and be acquired by a competing process; when the original process resumes, `db.update_stock()` executes blindly against stock values that may have already been modified by the new lock holder, producing a stale write with no detection. The fix is not in the locking layer — it is in the database write layer: the `update_stock` call must include an optimistic concurrency check (e.g., `WHERE stock = :expected_stock AND stock >= :quantity`) so that a write landing after lock expiry finds a precondition mismatch and fails atomically, rather than silently corrupting inventory. Until that guard is in place and covered by a test that simulates the preemption scenario, this implementation is not safe for a high-contention flash sale workload.

---

### Detailed Findings

#### 1. Critical — Stale Write on Lock Expiry (the flagged scenario)

This is the core issue and it is not addressed anywhere in the implementation.

The heartbeat design assumes the heartbeat thread runs independently of the main thread. It does not. In CPython, a stop-the-world GC pause, an OS scheduler preemption, or a `fork`-based deployment model suspends **all threads simultaneously**. The sequence that breaks the invariant:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat extends lock → TTL reset to 30s
T=15  OS preempts the entire process (GC, swap, scheduler)
T=45  Lock TTL expires (30s elapsed since last extension)
T=46  Process B acquires the same lock
T=46  Process B reads stock=5, writes stock=4
T=47  Process A resumes, continues from db.update_stock(item_id, new_stock=4)
      — using a value computed from the stock it read at T=0
      — with no awareness that the lock expired or that B already wrote
```

The result is a silent inventory corruption. The lock ownership check in the `finally` block only protects the **lock release**, not the **write that already happened**.

The required fix is a conditional update at the database layer:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id
  AND stock = :expected_stock   -- optimistic concurrency guard
  AND stock >= :quantity        -- safety floor
```

The application layer must then inspect the affected row count:

```python
rows_affected = self.db.update_stock(item_id, new_stock, expected_stock=current_stock)
if rows_affected == 0:
    # Lock expired and another writer already modified stock — abort
    raise StaleWriteError(f"Concurrent modification detected for item {item_id}")
```

Without this, the distributed lock provides ordering under normal conditions but provides **no correctness guarantee** under process pause. For a flash sale — exactly the workload where GC pressure and scheduler contention are highest — this is not an acceptable risk.

#### 2. Moderate — Heartbeat Thread State Is Instance-Level, Not Lock-Level

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`, not scoped to the individual lock acquisition. If `reserve_item` is ever called concurrently on the same `InventoryService` instance (e.g., via a thread pool), one call's `_stop_heartbeat()` will cancel another call's heartbeat mid-operation. The heartbeat state must be encapsulated per lock context, not per service instance.

```python
# Problematic pattern
self._heartbeat_active = False  # this affects ALL in-flight lock contexts
```

A straightforward fix is to pass a `threading.Event` into `_start_heartbeat` and return it, keeping the stop signal scoped to the closure:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return stop_event  # caller holds the stop handle
```

---

#### 3. Moderate — Heartbeat Uses `get` + `expire`, Not an Atomic Lua Script

The heartbeat checks ownership and extends TTL in two separate Redis commands:

```python
current = self.redis.get(lock_key)       # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be re-acquired by another process. The `EXPIRE` then silently extends a lock now owned by someone else. This should be a single atomic Lua script, which is the standard pattern for Redlock-style heartbeats:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

---

#### 4. Minor — `quantity` Has No Input Validation

`quantity` is typed as `int` but there is no guard against zero or negative values. A caller passing `quantity=0` would succeed, decrement stock by zero, and create a spurious order record. A caller passing `quantity=-5` would increase stock. Add a validation guard at the top of `reserve_item`:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

#### 5. Minor — Test Suite Mocks Away the Behavior Under Review

The fixture stubs out `_start_heartbeat` and `_stop_heartbeat` on the service instance, which means the tests never exercise the actual heartbeat or lock lifecycle. The test named `test_lock_not_released_if_expired` is the most relevant, but it only validates that `redis.delete` is not called — it does not validate that a stale `db.update_stock` call is rejected. A test for the preemption scenario should be added and should fail until the optimistic concurrency fix is in place:

```python
def test_stale_write_rejected_after_lock_expiry(service, db):
    """
    Simulate: lock expires mid-operation, another writer modifies stock.
    The original writer's db.update_stock must be rejected, not silently applied.
    """
    db.update_stock.return_value = 0  # 0 rows affected — concurrent modification
    with pytest.raises(StaleWriteError):
        service.reserve_item("item_001", quantity=1)
    # Stock must not be treated as successfully reserved
```

### Summary of Required Changes Before Approval

| # | Severity | Issue | Required Action |
|---|----------|-------|-----------------|
| 1 | Critical | No stale write protection after lock expiry | Add optimistic concurrency check in `update_stock` with row-count validation |
| 2 | Moderate | Heartbeat state is instance-scoped, not lock-scoped | Refactor to per-acquisition `threading.Event` |
| 3 | Moderate | Non-atomic heartbeat extension | Replace `GET` + `EXPIRE` with a Lua script |
| 4 | Minor | No validation on `quantity <= 0` | Add guard at entry to `reserve_item` |
| 5 | Minor | Tests mock the lock layer entirely | Add a test covering the stale write scenario |

The architecture of the locking layer is reasonable and the code is otherwise clean. Items 2, 3, 4, and 5 are addressable in a follow-up, but **item 1 must be resolved before merge**. A distributed lock that does not protect against stale writes under process pause is not a correctness guarantee — it is a performance optimization that happens to prevent most races under normal load.
