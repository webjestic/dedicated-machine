Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — the heartbeat cannot protect against a stop-the-world pause that exceeds `LOCK_TTL`, and the DB write layer has no mechanism to detect or reject a stale write after the lock has silently expired and been re-acquired by another process. Beyond that critical correctness flaw, there are several additional architectural and implementation issues that must be addressed before this is mergeable.

---

## Detailed Review

### 🔴 Critical: Stale Write on Process Pause (Flagged Scenario — Unresolved)

The reviewer notes correctly identify the core problem. To be precise about the failure sequence:

1. Process A acquires the lock, heartbeat thread starts.
2. The Python runtime (or OS) pauses **all threads** — GC stop-the-world, `SIGSTOP`, VM live migration, kernel preemption, etc.
3. The pause exceeds `LOCK_TTL` (30s). Redis expires the key.
4. Process B acquires the lock, reads stock, begins its own write.
5. Process A resumes. It has no idea time passed. `self._heartbeat_active` is still `True` in memory. The heartbeat thread wakes up, checks the key — but by now Process B owns it with a different token, so the heartbeat correctly does **not** extend it. However, **Process A's main thread is already past the heartbeat check** and proceeds directly to `db.update_stock()` and `db.create_order()`.
6. Both processes write. You now have a double-reservation or negative stock.

The heartbeat is not a solution to this class of problem. It reduces the probability of expiry during normal slow operations, but it cannot prevent expiry during a pause because **the thread that would renew the lock is itself paused**.

**The fix must be at the database layer.** The required mechanism is an **optimistic concurrency control (OCC) check** — specifically a conditional write using a version counter or a `WHERE` guard:

```sql
-- Only succeeds if stock has not changed since we read it
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND stock = :expected_stock   -- or: AND version = :expected_version
```

If `rowcount == 0`, the update was rejected because another writer changed the row. The caller must treat this as a conflict and abort. This makes the DB write itself idempotent against stale lock holders regardless of what Redis does.

The distributed lock then serves its correct purpose: reducing contention and DB-level conflicts, not as the sole correctness guarantee.

### 🔴 Critical: Shared Mutable Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not per-lock-acquisition locals. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal deployment model — a singleton service object), two concurrent acquisitions will stomp on each other's heartbeat state. Acquisition A sets `_heartbeat_active = True`, Acquisition B calls `_stop_heartbeat()` setting it to `False`, silently killing A's heartbeat.

The fix is to scope heartbeat state to the lock acquisition context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure, not self
```

### 🔴 Critical: No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- Empty string `item_id` → lock key becomes `"inventory:lock:"`, a collision vector
- Negative `quantity` → `new_stock = current_stock - (-5)` **increases** stock, a serious integrity bug
- Zero `quantity` → creates an order record for nothing
- Excessively long `item_id` strings → Redis key bloat or injection into log sinks

Minimum required:

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
```

### 🟡 Significant: `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The system is now in an inconsistent state with no compensating transaction. Both calls must execute within a single atomic transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock, expected_stock=current_stock)  # OCC check here
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Significant: Heartbeat Has a Race Window at Startup

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
# ... retry loop ...
self._start_heartbeat(lock_key, token)
try:
    yield token
```

The heartbeat thread sleeps `HEARTBEAT_INTERVAL` (10s) **before its first extension**. If the lock is acquired and then a slow DB operation begins immediately, the first renewal doesn't happen until 10 seconds in. With `LOCK_TTL = 30s` this is acceptable under normal conditions, but under the pause scenario the first sleep is the entire window of vulnerability. This is a minor point given the larger architectural issue, but the interval math should be documented explicitly.

### 🟡 Significant: Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The majority of tests use this fixture, meaning the heartbeat and lock release logic are never exercised. `test_lock_not_released_if_expired` is the only test that touches lock behavior, and it only verifies the `redis.delete` skip — it does not test the stale write scenario at all. The test suite would pass even if the heartbeat thread had a fatal bug. Integration-level tests against a real Redis instance (e.g., using `fakeredis` or a test container) are required.

---

### 🟡 Significant: `LOCK_TTL = 30s` Is Dangerously Long for a Flash Sale

A 30-second lock on a hot inventory item during a flash sale means a single slow or crashed process can block **all** reservations for that item for up to 30 seconds. Under flash sale load this is a denial-of-service vector. The TTL should be set to the minimum viable value for the expected DB operation latency, with the heartbeat extending it only when the operation is legitimately ongoing.

---

### 🔵 Minor Issues

- **`_stop_heartbeat` does not join the thread.** After `_heartbeat_active = False`, the thread may still be mid-sleep and execute one more `redis.expire` after the lock has been released. Add `self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)`.
- **`logger.error` on lock contention is misleading.** Lock contention after retries is a business-level capacity event, not an application error. Consider `logger.warning` with a metric emission.
- **No `__slots__` or thread safety on `InventoryService` fields.** For a service expected to handle concurrent flash sale load, this should be explicit.
