Decision: Request Changes

Summary: The senior engineer's flag is the critical blocker here. The heartbeat thread and the application thread are both hosted in the same OS process. A stop-the-world GC pause, a long OS scheduler preemption, or any other process-level suspension freezes *all* threads simultaneously — including the heartbeat. If that pause exceeds `LOCK_TTL`, the lock expires on Redis's side, another process acquires it, and when the original process resumes it proceeds to call `db.update_stock()` with no awareness that it has lost lock ownership. The heartbeat provides no protection against this class of failure because the protection mechanism lives in the same failure domain as the code it is meant to protect. This is a well-documented limitation of process-local lock renewal and is not a theoretical edge case in a flash sale workload where GC pressure is elevated.

## Critical Flaw — Stale Write on Process Pause

The scenario the senior engineer described is a real, unmitigated data integrity risk in this implementation. The heartbeat does not close the window; it only narrows it under normal operating conditions.

### Why the heartbeat cannot protect against this

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   OS pauses Process A (GC, scheduler, swap, etc.)
        → ALL threads in Process A are suspended
        → heartbeat thread is also suspended
T=35  Lock TTL expires on Redis (LOCK_TTL = 30s)
T=36  Process B acquires the same lock, reads stock=10, begins its write
T=40  OS resumes Process A
        → A has no idea it lost the lock
        → A calls db.update_stock(item_id, 9)  ← stale write, races with B
T=41  Process B calls db.update_stock(item_id, 9)
        → both writes succeed, stock decremented twice from the same baseline
```

The heartbeat thread and the critical section share a process boundary. Any event that suspends the process suspends both. Redis has no way to distinguish "process is paused" from "process is dead," so it correctly expires the lock. The application has no mechanism to detect that expiry before issuing the write.

---

## Required Fix — Optimistic Locking at the DB Write Layer

The only correct mitigation is a **fencing token** or **optimistic concurrency check** enforced at the database, not in application memory. Redis's Redlock documentation calls this a *fencing token*; Martin Kleppmann's analysis of distributed locks describes exactly this failure mode.

### Option A — Monotonic fencing token from Redis (recommended for this architecture)

Redis `INCR` returns a monotonically increasing integer each time a lock is acquired. Pass that token through to the DB write. The DB rejects any write whose token is not strictly greater than the last committed token for that item.

```python
# On lock acquisition
fence_key = f"inventory:fence:{item_id}"
fencing_token = self.redis.incr(fence_key)  # monotonically increasing

# Pass token into the DB write
self.db.update_stock(item_id, new_stock, fencing_token=fencing_token)
```

```sql
-- DB enforces the ordering constraint
UPDATE inventory
SET    stock = :new_stock,
       last_fence_token = :fencing_token
WHERE  item_id = :item_id
  AND  last_fence_token < :fencing_token   -- rejects stale writers
```

If Process A resumes after its lock expired and Process B has already committed with a higher token, A's write is rejected at the database layer. The application must check the affected row count and treat zero rows updated as a conflict.

```python
rows_updated = self.db.update_stock(item_id, new_stock, fencing_token=fencing_token)
if rows_updated == 0:
    raise StaleWriteError(f"Fencing token rejected for item_id={item_id}")
```

### Option B — Optimistic locking via version column (acceptable alternative)

If a fencing token is not feasible, a version column with a `WHERE version = :expected_version` predicate on the UPDATE provides equivalent protection, at the cost of not being strictly ordered across processes.

```sql
UPDATE inventory
SET    stock = :new_stock,
       version = version + 1
WHERE  item_id = :item_id
  AND  version = :expected_version
  AND  stock >= :quantity   -- guard against concurrent over-decrement
```

Zero rows updated means another writer committed first; the caller must retry or return a conflict response.

## Secondary Issues

### 1. Instance-level heartbeat state is not concurrency-safe

```python
# Current implementation — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool), two concurrent calls will stomp on each other's `_heartbeat_active` and `_heartbeat_thread` references. The heartbeat for one reservation can be cancelled by another reservation completing. This state must be local to each lock acquisition, not stored on `self`.

```python
# Correct — heartbeat state scoped to the lock context
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

Using a `threading.Event` also replaces the `time.sleep` polling loop, which is a cleaner shutdown pattern and avoids the thread lingering for up to `HEARTBEAT_INTERVAL` seconds after the lock is released.

### 2. `db.update_stock` and `db.create_order` are not atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes between these two calls, stock is decremented but no order record exists. These two operations must be wrapped in a single database transaction. The distributed lock serializes access across processes but provides no atomicity guarantee within the database itself.

### 3. No input validation on `item_id` or `quantity`

The Pydantic model is on the *output* (`ReservationResult`). The inputs `item_id: str` and `quantity: int` arrive as raw Python primitives with no validation. A negative `quantity` would silently increment stock. An `item_id` with special characters could interact badly with the Redis key format. A Pydantic input model or explicit guards are needed.

```python
if not item_id or not item_id.isalnum():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

### 4. Test coverage gaps

- No test exercises the stale write / fencing token rejection path (which is the highest-risk scenario).
- No test verifies concurrent calls on the same service instance expose the shared heartbeat state bug.
- `test_lock_not_released_if_expired` is the right instinct but does not assert that `update_stock` was still called in that scenario, which is the actual danger being tested.
