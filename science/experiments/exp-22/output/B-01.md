Decision: Request Changes

Summary: The reviewer note identifies a real and unresolved TOCTOU race — a stop-the-world GC pause or OS preemption can suspend all threads (including the heartbeat) long enough for the lock TTL to expire, after which another process acquires the lock and begins its own write, and then the original process resumes and issues a stale `db.update_stock()` with no awareness that lock ownership was lost. The heartbeat architecture cannot prevent this class of failure because the mechanism protecting the lock lives in the same process that can be paused. The DB write layer has no guard against the stale write, which is the precise gap the senior engineer flagged and which this PR does not close. Additional structural issues compound the problem and are detailed below.

---

## Critical — Stale Write on Lock Expiry (Unresolved Flagged Concern)

This is the core issue the reviewer note raises, and the implementation does not address it.

**The failure sequence:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
heartbeat running
  ← GC pause / OS preemption →
  [all threads suspended, including heartbeat]
  [LOCK_TTL elapses]
  [Redis expires the key]
                                   acquire lock (token=B)
                                   get_stock() → 1
                                   update_stock(item, 0)   ← legitimate write
                                   create_order(...)
                                   release lock
  ← process A resumes →
  heartbeat sees token mismatch, stops extending
  BUT execution continues in reserve_item()
  update_stock(item, -1)           ← STALE WRITE, stock goes negative
  create_order(...)                ← DUPLICATE ORDER
```

The heartbeat's token check only prevents *extending* an expired lock. It does not interrupt the executing thread. By the time the heartbeat detects the mismatch, the DB calls have already been made or are in flight.

**What is required at the DB write layer:**

The only correct fix is an optimistic concurrency check at the database, not at the Redis layer. Redis is a coordination hint, not a transaction boundary. Two patterns close this gap:

**Option 1 — Optimistic locking with a version/generation counter:**

```python
# Schema adds: stock_version INTEGER NOT NULL DEFAULT 0
# Read phase
row = self.db.get_stock_with_version(item_id)
# row = {"stock": 5, "version": 42}

# Write phase — conditional update
rows_affected = self.db.update_stock_if_version(
    item_id,
    new_stock=row["stock"] - quantity,
    expected_version=row["version"],   # CAS: only write if version still matches
)
if rows_affected == 0:
    # Another writer committed first — abort and surface a conflict error
    raise StaleWriteError(f"Concurrent modification detected for item {item_id}")
```

The SQL form of this is:

```sql
UPDATE inventory
SET stock = :new_stock, stock_version = stock_version + 1
WHERE item_id = :item_id
  AND stock_version = :expected_version
  AND stock >= :quantity;   -- also guards against negative stock at the DB layer
```

If `rows_affected == 0`, the write was rejected by the database itself — no stale update is possible regardless of what happened to the Redis lock.

**Option 2 — Encode the lock token in the DB write:**

```python
# Write carries the token that was valid when the read was taken
rows_affected = self.db.update_stock_if_lock_token(
    item_id, new_stock, lock_token=token
)
# DB checks a `current_lock_token` column; rejects if it no longer matches
```

This is less standard but achieves the same effect: the database rejects any write whose authorization token has been superseded.

Either option makes the DB write *self-validating* — the correctness guarantee no longer depends on the Redis lock still being held at write time.

## High — Shared Heartbeat State Breaks Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not scoped to a single lock acquisition. If two callers use the same `InventoryService` instance concurrently (likely in any threaded or async server), one call's `_stop_heartbeat()` will cancel the other call's heartbeat.

```python
# Current — instance-level, shared across all concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix — scope heartbeat state to the lock context:**

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                active.clear()  # lock was lost; stop extending
                break

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()
    try:
        yield token
    finally:
        active.clear()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # conditional delete unchanged
        ...
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern and allows clean, prompt shutdown.

## High — Heartbeat Thread Is Never Joined

`_stop_heartbeat()` sets the flag but does not join the thread. The thread continues sleeping for up to `HEARTBEAT_INTERVAL` seconds after the lock is released. During that window it can wake, find a *different* process's token in Redis (which happens to equal the old token by UUID collision — astronomically unlikely but architecturally unsound), and extend a lock it no longer owns. More practically, under high load this leaks threads.

```python
# Current — flag set, thread abandoned
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
# Thread wakes up to HEARTBEAT_INTERVAL seconds later
```

The fix is shown above: `heartbeat.join(timeout=...)` in the `finally` block.

## Medium — No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input path. `quantity=0` or `quantity=-5` passes all checks and results in a stock *increase*, which is a business logic error exploitable by any caller.

```python
# quantity=-1 passes the stock check (current_stock < -1 is False when stock >= 0)
# new_stock = current_stock - (-1) = current_stock + 1
# Stock is incremented, not decremented
```

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

## Medium — `item_id` Is Not Sanitized Before Use as a Redis Key

`item_id` is interpolated directly into the Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "abc:unlock:admin"` produces `inventory:lock:abc:unlock:admin`, which is a valid Redis key that could collide with or shadow other keys depending on key-space conventions. With the Pydantic fix above (pattern constraint), this is resolved, but it is worth calling out explicitly.

## Low — Race Between `GET` and `EXPIRE` in Heartbeat

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

These are two separate commands. Between `GET` and `EXPIRE`, the key can expire and be re-acquired by another process. The heartbeat then extends a lock it no longer owns. This should be a single atomic operation:

```lua
-- Lua script, executed atomically on the Redis server
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""
extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    active.clear()  # lock was lost; stop heartbeat
```

The same issue exists in the lock release path and should use the standard Redlock release script there as well.
