Decision: Request Changes

Summary: The senior engineer's flag is a legitimate, unresolved critical flaw. The heartbeat thread is suspended during any stop-the-world pause alongside the main thread, meaning the lock can expire and be acquired by another process while the original process is mid-execution — and when the original resumes, `db.update_stock()` proceeds with no awareness that lock ownership was lost. The implementation contains no DB-layer guard against this stale write, which means two processes can decrement stock concurrently, producing oversell in a flash sale context — exactly the failure mode this service is designed to prevent. There are also secondary issues across the implementation that need to be addressed before approval.

---

## Critical Flaw: Stale Write After Lock Expiry

The reviewer note identifies this precisely. The heartbeat does not solve the problem it is credited with solving in this scenario.

**The failure sequence:**

```
Process A acquires lock (token=abc, TTL=30s)
Process A heartbeat thread starts

[OS schedules Process A out / GC stop-the-world pause > 30s]
  → All threads in Process A are suspended, including the heartbeat
  → Lock TTL expires in Redis

Process B acquires lock (token=xyz)
Process B reads stock = 10, writes stock = 9, releases lock

[Process A resumes]
Process A reads stock = 10  ← already stale, read before the pause
Process A writes stock = 9  ← overwrites Process B's committed write
Process A creates order      ← duplicate decrement, oversell
```

The heartbeat thread is not a reliable safeguard. It is suspended by the same OS event that suspends the main thread. This is not a theoretical edge case in a flash sale — GC pauses, VM live migration, and scheduler preemption are routine under load.

**What is required at the DB write layer:**

The lock alone cannot guarantee safety across a process pause. The database write must include an optimistic concurrency check — a conditional update that fails atomically if the stock value has changed since it was read.

```python
# Required: conditional update with expected value
rows_affected = self.db.update_stock_if_equals(
    item_id,
    expected_stock=current_stock,
    new_stock=new_stock
)
if rows_affected == 0:
    raise StaleInventoryError(
        f"Stock changed for {item_id} during reservation — aborting"
    )
```

In SQL this maps to:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id
  AND stock = :expected_stock;
```

If `rows_affected == 0`, the stock was modified by another writer between the read and the write. The operation must be aborted and retried or surfaced as a conflict, not silently committed. This is the only mechanism that closes the window the senior engineer identified.

## Secondary Issues

### 1. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the expected usage pattern in any web framework — two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. One call's `_stop_heartbeat()` will terminate the other's heartbeat.

**Fix:** Move heartbeat state into the lock context manager as local variables, not instance state.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
```

### 2. Heartbeat Uses a Racy Check-Then-Act Pattern

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a non-atomic read-then-write. Between the `GET` and the `EXPIRE`, another process could acquire the lock. The heartbeat would then extend a lock it no longer owns.

**Fix:** Use a Lua script for atomic check-and-extend:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to lock release in `_acquire_lock`'s `finally` block.

### 3. Lock Release Is Also Non-Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same race as above. This is a well-known Redis distributed lock anti-pattern. The canonical fix is the same Lua script approach, which is what Redlock implementations use:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 4. Input Validation Is Declared But Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is not validated for format, length, or character set. A malformed `item_id` flows directly into a Redis key and a DB query.

```python
# item_id goes directly into:
lock_key = f"inventory:lock:{item_id}"
self.db.get_stock(item_id)
```

If `item_id` contains `:` or whitespace, the Redis key structure is corrupted. If the DB layer does not parameterize queries, this is a vector for injection.

**Fix:** Define and enforce an input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]{1,64}$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. `quantity` Has No Validation

`quantity` accepts any integer including zero and negative values. `reserve_item("item_001", quantity=-5)` would pass the stock check (`10 < -5` is false), write `new_stock = 15`, and create an order for a negative quantity. This is a data integrity bug independent of concurrency.

### 6. Test Coverage Does Not Test the Critical Scenario

The test suite mocks out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic is never exercised. More importantly, there is no test for the stale write scenario the senior engineer flagged — the scenario where the lock expires between read and write. `test_lock_not_released_if_expired` only tests that `DELETE` is skipped, not that the write is aborted.

The tests should be extended to assert that a stale write raises or returns a conflict result when `update_stock_if_equals` returns zero rows affected.

## Summary of Required Changes Before Approval

| # | Issue | Severity |
|---|-------|----------|
| 1 | No DB-layer conditional write guard against stale writes after lock expiry | Critical |
| 2 | Instance-level heartbeat state is not safe for concurrent use of the same service instance | High |
| 3 | Heartbeat `GET`+`EXPIRE` and release `GET`+`DEL` are non-atomic | High |
| 4 | `item_id` is not validated before use in Redis keys and DB queries | Medium |
| 5 | `quantity` accepts zero and negative values | Medium |
| 6 | No test coverage for stale write scenario or heartbeat logic | Medium |

The architecture of using a distributed lock with a heartbeat is a reasonable approach, but the implementation has not closed the gap the senior engineer identified, and it has introduced a shared-state concurrency bug that would surface under normal production load. The critical issue must be resolved at the DB write layer before this can be approved.
