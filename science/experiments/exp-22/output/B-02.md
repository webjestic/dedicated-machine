Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — not a theoretical edge case. A process-level pause (GC, OS preemption, VM live migration) suspends all threads simultaneously, including the heartbeat, meaning the lock can silently expire and be acquired by another process while the original process resumes and blindly executes `db.update_stock()` and `db.create_order()`. The heartbeat mechanism provides no protection against this class of failure because it cannot distinguish "we are healthy and holding the lock" from "we were paused and the lock has since been stolen." Beyond that core flaw, there are additional architectural and correctness issues that must be addressed before this can be approved.

---

## Critical: Stale Write After Process Pause

This is the exact scenario the reviewer flagged, and the implementation does not protect against it.

**What happens:**

1. Process A acquires the lock and begins executing inside `reserve_item`
2. The OS pauses Process A (GC, scheduler preemption, VM migration, etc.)
3. All threads in Process A are suspended — including the heartbeat thread
4. `LOCK_TTL` (30s) elapses; Redis expires the lock
5. Process B acquires the lock, reads stock, and writes its reservation
6. Process A resumes from where it was paused, with no awareness that time has passed
7. Process A calls `db.update_stock()` and `db.create_order()` — now a stale, conflicting write

The token check in the `finally` block only prevents a stale *lock release*. It does nothing to prevent the stale *database write* that already occurred in step 7. The heartbeat cannot help here because it was also suspended during the pause.

**Required fix — optimistic locking at the DB write layer:**

The database write must be conditional. The stock value read before the pause must be used as a precondition for the write, so that if another process has already modified the row, the write is rejected rather than applied silently.

```python
# Pattern: conditional update with expected value as precondition
rows_affected = self.db.update_stock_if_matches(
    item_id,
    expected_stock=current_stock,  # value read before potential pause
    new_stock=new_stock
)

if rows_affected == 0:
    # Another process modified stock after we read it — abort
    logger.warning(
        "Stale write detected item_id=%s expected_stock=%d — aborting",
        item_id, current_stock
    )
    return ReservationResult(
        success=False,
        message="Reservation conflict. Please try again."
    )
```

The SQL backing this must be atomic:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id
  AND stock = :expected_stock   -- precondition: row unchanged since our read
```

This is a standard optimistic concurrency control pattern. The `rows_affected` return value is the only reliable signal that the write was applied to the row you actually read. Without this, the distributed lock is providing a false sense of safety.

## Critical: Non-Atomic DB Operations Without Transaction

`db.update_stock()` and `db.create_order()` are two separate calls with no transaction boundary. If the process crashes, is killed, or encounters a network error between them, the stock is decremented but no order record exists. This is a silent inventory leak.

```python
# Required: wrap both writes in a single atomic transaction
with self.db.transaction():
    self.db.update_stock_if_matches(item_id, expected_stock=current_stock, new_stock=new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

Both writes must succeed or neither must be applied. This is not optional for a financial inventory system.

## Architectural: Single Heartbeat State Shared Across Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two calls to `reserve_item` execute concurrently on the same `InventoryService` instance (which is the normal case for a shared service object), they will overwrite each other's heartbeat state. One call stopping its heartbeat will stop the other's renewal thread.

```python
# Current — shared mutable state, not safe for concurrent use
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. Use a local flag passed by reference or a threading.Event per lock context:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return stop_event  # caller sets stop_event to stop the thread
```

## Architectural: No Input Validation on Public Entry Point

`reserve_item` accepts `item_id: str` and `quantity: int` but performs no validation. An empty string, whitespace-only string, negative quantity, or zero quantity will pass through to the lock key construction and database calls. Given that this is a government contracting context where input integrity matters:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

The PR checklist marks input validation as complete via Pydantic, but Pydantic is only used on the *output* model (`ReservationResult`). There is no Pydantic model validating the inputs to `reserve_item`. This checklist item is inaccurate.

---

## Minor: Heartbeat Uses Polling Instead of `threading.Event`

The current heartbeat loop uses `time.sleep()`, which means the thread can take up to `HEARTBEAT_INTERVAL` seconds to stop after `_stop_heartbeat()` is called. Using `threading.Event.wait(timeout=...)` allows the thread to exit immediately when signaled, which is the correct pattern for stoppable background threads. This is addressed by the fix in the concurrency section above.

---

## Test Coverage Gaps

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic itself has zero test coverage. Additionally:

- No test for the concurrent instance state collision described above
- No test verifying that a stale write is rejected (once the optimistic locking fix is in place, this must be tested)
- No test for invalid inputs (`quantity=0`, `quantity=-1`, empty `item_id`)
- `test_lock_not_released_if_expired` verifies the lock is not deleted, but does not verify that the DB write is also aborted — which is the actual safety requirement

| # | Issue | Severity |
|---|-------|----------|
| 1 | No optimistic locking at DB write layer — stale writes possible after process pause | Critical |
| 2 | `update_stock` and `create_order` not wrapped in a transaction | Critical |

| 4 | Input validation absent on `reserve_item` parameters despite checklist claiming otherwise | Moderate |

The distributed lock is a reasonable first layer of coordination, but it cannot be the only layer. The DB write must be self-defending via optimistic concurrency control. Approve after items 1–3 are resolved and test coverage reflects the actual failure modes.
