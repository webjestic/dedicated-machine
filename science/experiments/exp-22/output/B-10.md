Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — a process pause (GC stop-the-world, OS preemption, VM live migration) suspends all threads simultaneously, including the heartbeat, allowing the lock TTL to expire and be acquired by another process; when the original process resumes, it proceeds with `db.update_stock()` blind to the fact it no longer holds the lock, producing a stale write and potential oversell. The heartbeat pattern cannot protect against this class of failure because the mechanism defending the lock lives in the same process that can be paused. Beyond this core flaw, there are additional architectural and correctness issues that must be addressed before approval.

---

## Detailed Findings

### Critical — Stale Write on Process Pause (Raised by Senior Engineer, Unresolved)

The reviewer note identifies this precisely. The heartbeat thread is a best-effort mitigation for *slow but running* operations. It provides no protection against a scenario where the entire process is suspended by the OS or runtime.

**Attack path:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
[GC pause — all threads frozen]
  lock TTL expires
                                   acquire lock (token=B)
                                   read stock = 10
                                   write stock = 9  ✓
[GC resumes]
db.update_stock(item, 9)  ← stale write, no guard
db.create_order(...)      ← phantom order
```

The result is a negative or incorrect stock count and a fulfilled order that should have been rejected. In a flash sale context this is a direct financial and inventory integrity failure.

**What is required at the DB write layer:**

The lock alone cannot be the sole correctness guard. The database write must be made *conditional* — it should only succeed if the data has not changed since it was read. Two standard mechanisms:

**Option 1 — Optimistic locking with a version/fencing token:**

```python
# Read phase — DB returns stock AND a version counter or fencing token
current_stock, version = self.db.get_stock_with_version(item_id)

# Write phase — DB atomically checks version before applying update
# UPDATE inventory SET stock = %s, version = version + 1
#   WHERE item_id = %s AND version = %s
rows_affected = self.db.update_stock_if_version(item_id, new_stock, expected_version=version)

if rows_affected == 0:
    # Another writer got here first — abort, do not create order
    raise StaleWriteError(f"Concurrent modification detected for {item_id}")
```

This is the correct pattern. The DB write becomes a compare-and-swap. If Process A resumes after a pause and attempts the write, the version will have already been incremented by Process B, `rows_affected` will be 0, and the write is rejected.

**Option 2 — Atomic DB operation (preferred for simple decrement):**

```sql
UPDATE inventory
   SET stock = stock - :quantity
 WHERE item_id = :item_id
   AND stock >= :quantity
RETURNING stock;
```

If no row is returned, the reservation fails. This eliminates the read-modify-write race entirely at the cost of moving the stock-check logic into the DB layer.

The Redis lock should be retained for coarse-grained coordination (reducing DB contention), but it **must not be the only correctness mechanism**.

### High — Shared Heartbeat State Breaks Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two calls to `reserve_item` overlap on the same `InventoryService` instance (e.g., in a thread pool), the second call overwrites `_heartbeat_thread`, and `_stop_heartbeat()` from the first call sets `_heartbeat_active = False`, killing the second call's heartbeat prematurely.

```python
# Current — broken under concurrency
self._heartbeat_active = False        # instance attribute
self._heartbeat_thread: Optional[threading.Thread] = None  # instance attribute
```

Heartbeat state must be scoped to the lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()
    # pass `active` into the heartbeat closure, not self._heartbeat_active
```

### High — `db.update_stock` and `db.create_order` Are Not Atomic

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The item is silently lost.

```python
# Current — not atomic
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

Both operations must execute inside a single DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### Medium — Input Validation Is Declared but Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. `item_id` could be an empty string, a path traversal fragment, or an arbitrarily long value. `quantity` could be zero or negative.

```python
# Missing — should be present
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        raise ValueError("item_id must be a non-empty string")
    if quantity < 1:
        raise ValueError("quantity must be a positive integer")
```

Pydantic is imported and `ReservationResult` is modeled, but it is only used for the *output*. A `ReservationRequest` input model should be added and validated at the entry point.

---

### Medium — Lock Key Constructed from Unvalidated User Input

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is not sanitized, a caller can supply a value like `"../../admin"` or an extremely long string, potentially colliding with unrelated Redis keys or causing operational issues. Validate and constrain `item_id` format before constructing the key.

### Medium — Heartbeat Race on Lock Release

In `_acquire_lock`'s `finally` block:

```python
self._stop_heartbeat()
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

There is a TOCTOU window between `GET` and `DELETE`. The lock could expire and be acquired by another process between those two calls. The correct pattern is a Lua script that makes the check-and-delete atomic:

```lua
-- release.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

---

### Low — Test Suite Does Not Cover the Core Failure Mode

`test_lock_not_released_if_expired` verifies that `delete` is not called when the token mismatches, but there is no test asserting that `update_stock` is *also not called* (or is rolled back) when the lock has expired. The most important invariant — no stale write — has no test coverage.

A test should be added that simulates the process-pause scenario: lock expires mid-operation, a second reserver succeeds, and the first reserver's write is rejected.

---

### Low — `LOCK_TTL = 30` Is Undocumented and Untested Against Real DB Latency

A 30-second TTL is generous but the comment "generous TTL for slow DB operations" is not backed by any measurement or SLA. For a flash sale service under load, this should be derived from observed p99 DB latency with headroom, and the value should be configurable via environment/config rather than hardcoded.
