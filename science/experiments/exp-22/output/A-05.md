Decision: Request Changes

Summary: The senior engineer's flag is the correct call — this implementation contains a fundamental correctness flaw that the heartbeat thread does not and cannot solve. The DB write layer has no fencing mechanism, meaning a process that resumes after a GC pause or OS preemption will execute `db.update_stock()` against stale state with no awareness that it lost the lock. There are also secondary architectural concerns around instance-level heartbeat state that make this unsafe under concurrent use.

## Detailed Review

### Critical Flaw — No Fencing at the Write Layer

The reviewer note describes the exact failure mode, and the answer to the question posed is: **no, this implementation does not prevent the stale write**.

Here is the failure sequence written out precisely:

```
Process A acquires lock, token = "abc-123"
Process A reads stock = 10
  *** OS preempts Process A for > LOCK_TTL seconds ***
  *** All threads in Process A are suspended, including the heartbeat thread ***
  *** Lock TTL expires in Redis ***
Process B acquires lock, token = "xyz-789"
Process B reads stock = 10, writes stock = 9, creates order
  *** OS resumes Process A ***
Process A has no idea the lock expired
Process A calls db.update_stock(item_id, 9)  ← stale write, stock is now wrong
Process A calls db.create_order(...)          ← duplicate reservation
```

The heartbeat thread is suspended alongside every other thread during a stop-the-world GC pause or OS preemption. It cannot protect against this. This is not a theoretical edge case — it is a well-documented failure mode of distributed locks, described in detail by Martin Kleppmann in his critique of Redlock. The heartbeat provides a false sense of safety here.

**What is actually required** is a fencing token at the DB write layer. The standard approach is:

```
1. The lock service issues a monotonically increasing fencing token on acquisition
   (Redis can provide this via INCR on a separate key, or a version counter)
2. The token is passed with every write to the DB layer
3. The DB rejects any write whose token is less than or equal to the
   last committed token for that resource
```

A lightweight alternative for SQL-backed inventory is optimistic locking with a version column:

```python
# DB layer enforces: UPDATE stock SET quantity = %s, version = version + 1
#                    WHERE item_id = %s AND version = %s AND quantity >= %s
# Rows affected == 0 means the write must be rejected, not silently swallowed
rows_affected = self.db.update_stock_versioned(item_id, new_stock, expected_version)
if rows_affected == 0:
    raise StaleWriteError(f"Concurrent modification detected for item {item_id}")
```

Without one of these two mechanisms, the Redis lock provides ordering guarantees only under cooperative, non-preempted conditions. It does not provide safety under the failure modes that matter most.

### Secondary Flaw — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables on a shared `InventoryService` object. If two threads call `reserve_item` concurrently on the same service instance, their heartbeat state will collide. The second call to `_start_heartbeat` will overwrite `_heartbeat_thread`, and `_stop_heartbeat` from either call will set `_heartbeat_active = False`, stopping the other thread's heartbeat mid-operation.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure, not self
```

### Secondary Flaw — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or is killed between these two calls, stock is decremented but no order record exists. These two operations must be wrapped in a single DB transaction. The Redis lock does not substitute for transactional atomicity at the persistence layer.

### Minor Issues

**`quantity` parameter has no validation guard in the service method.** Pydantic is used on the result model but not on inputs. A caller passing `quantity=0` or `quantity=-5` will pass the stock check and write a nonsense record.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

**The heartbeat thread checks token ownership before extending TTL, which is correct**, but there is a TOCTOU window between the `GET` and the `EXPIRE`. This should be a single atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

**The lock release in `finally` has the same TOCTOU issue** between `GET` and `DELETE`. The standard fix is a Lua script here as well, which is the canonical Redlock release pattern.

---

### Test Coverage Gap

`test_lock_not_released_if_expired` is a good test, but there is no test covering the stale-write scenario described in the reviewer note — specifically, no test that verifies the DB write is rejected when the lock has been acquired by another process before the write completes. This test cannot be written until the fencing mechanism exists, which is further confirmation the mechanism is missing.
