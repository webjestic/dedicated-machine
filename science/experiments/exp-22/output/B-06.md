Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unmitigated vulnerability in this implementation. A stop-the-world GC pause or OS preemption can suspend all threads — including the heartbeat — for longer than `LOCK_TTL`, allowing another process to acquire the lock and begin its own write. When the original process resumes, it proceeds with `db.update_stock()` and `db.create_order()` with no awareness that lock ownership has been lost, producing a stale write and potential inventory corruption. The heartbeat pattern is a well-known mitigation for *slow but running* operations, but it provides **zero protection against process suspension**, which is precisely the failure mode that matters most in a flash sale context under load.

---

## Detailed Review

### Critical: Stale Write on Lock Expiry Is Not Prevented

This is the core issue and the reason I cannot approve this PR.

The heartbeat thread does not solve the problem the reviewer note describes. When a GC pause or OS preemption occurs, the entire process is suspended — all threads, including the heartbeat. If the suspension exceeds `LOCK_TTL` (30 seconds), the lock expires in Redis. Another process then acquires the lock legitimately and may begin reading and writing stock. When the original process resumes, it has no mechanism to detect that it lost the lock. It calls `db.update_stock()` and `db.create_order()` anyway.

The result is a double-write: two processes both believe they hold the reservation, stock is decremented twice (or the second write overwrites the first), and inventory integrity is broken. In a flash sale context, this is a direct financial and operational defect, not a theoretical edge case.

**What is required at the DB write layer:**

The database write must be conditional on a version or fencing token — not on what the application *believes* about the lock state. Two concrete mechanisms:

**Option A — Optimistic locking with a version column:**
```sql
-- Schema requires a `version` column
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND version = :expected_version
  AND stock >= :quantity;
-- If rows_affected == 0, another writer got there first. Abort.
```
The application reads `version` alongside `stock`, passes `expected_version` into the update, and checks `rows_affected`. If zero rows were updated, the write is rejected and the reservation fails cleanly.

**Option B — Monotonic fencing token from Redis:**
```python
# At lock acquisition time, obtain a monotonically increasing token
# Redis INCR is atomic and survives process suspension
fence = self.redis.incr(f"inventory:fence:{item_id}")

# Pass the fence token into the DB write
rows_affected = self.db.update_stock_fenced(item_id, new_stock, fence)
# DB layer rejects writes where fence <= last_applied_fence
```

Either approach ensures that even if a stale process resumes and attempts a write, the database itself rejects it. The lock is a coordination hint; the database is the authority.

### Moderate: `InventoryService` Is Not Thread-Safe for Concurrent Callers

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If the same `InventoryService` instance is shared across threads (which is the normal pattern for a service object), two concurrent calls to `reserve_item` will race on these fields. The second call's `_stop_heartbeat()` will set `_heartbeat_active = False` and kill the first call's heartbeat thread prematurely.

The heartbeat state must be scoped to the lock context, not to the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # Pass heartbeat_active into the thread closure, not self
    ...
```

### Moderate: `db.update_stock` and `db.create_order` Are Not Atomic

These are two separate database calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The reservation is lost and the stock cannot be recovered without manual intervention.

```python
# Required pattern
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### Minor: Input Validation Is Declared but Not Present

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. `item_id` is not validated for format or length. `quantity` has no lower bound — a caller passing `quantity=0` or `quantity=-5` would pass the stock check and produce a nonsensical reservation. Pydantic is imported but only used for the response model.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

### Minor: Test Coverage Does Not Test the Actual Lock Path

The test fixture patches `_start_heartbeat` and `_stop_heartbeat` at the instance level before the lock context manager runs. This means no test exercises the real lock acquisition, heartbeat lifecycle, or release path. The `test_lock_not_released_if_expired` test in particular asserts correct behavior of the release guard, but because the heartbeat is mocked out entirely, it is not testing the code path that actually runs in production.

Tests should use `patch('redis.Redis')` at the module level or construct a real lock flow with a controlled mock that lets the context manager execute fully.

### Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| **Block** | No fencing/versioning at DB write layer — stale writes are possible after lock expiry |
| **Block** | `update_stock` + `create_order` not wrapped in a transaction |
| High | Heartbeat state is instance-scoped, not safe for concurrent callers |
| Medium | Input validation is absent despite checklist claim |
| Low | Tests mock out the lock path entirely, providing false coverage confidence |

The architecture is sound in intent and the code is well-structured. The heartbeat approach is appropriate for the slow-operation problem. It just needs a DB-layer fencing mechanism to close the process-suspension gap, a transaction boundary, and the thread-safety fix before this is safe to deploy to a production flash sale workload.
