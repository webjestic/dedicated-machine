Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and concurrency issues that must be resolved before this can be approved for a production government contracting environment. The test suite also has meaningful gaps that mask real failure modes.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrent Safety

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, not scoped to the lock context.

```python
# Current — instance-level state
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the **same `InventoryService` instance** simultaneously (which is the entire point of this service under flash sale load), one thread's `_stop_heartbeat()` call will cancel the other thread's heartbeat. This can cause premature lock expiry mid-operation, which is exactly the failure mode the heartbeat is designed to prevent.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release via Lua script (see issue #2)
        self._release_lock(lock_key, token)
```

### 2. Non-Atomic Lock Release — TOCTOU Race Condition

The lock release is a two-step read-then-delete, which is not atomic:

```python
# Current — race condition window between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and `DELETE`, the lock can expire and be acquired by another process. You would then delete a lock you no longer own. This is a well-known Redis distributed lock pitfall. The fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    if result:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock release skipped — token mismatch or already expired key=%s",
            lock_key
        )
    return bool(result)
```

The same race condition exists in the heartbeat's check-then-extend:

```python
# Current — also non-atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Apply the same Lua pattern for heartbeat renewal.

### 3. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is never joined
```

Setting the flag does not guarantee the thread has stopped. The thread may still be mid-execution when the lock is released, creating a window where a thread you believe is stopped extends a lock you've already released (or that another process now owns). Always join the thread before releasing the lock.

### 4. Missing Input Validation

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserve zero or negative items)
- Empty or whitespace-only `item_id`
- Excessively large `quantity` values
- `item_id` values that could be used for lock key injection (e.g., containing `:` characters that corrupt the key namespace)

Pydantic is imported but only used for the result model. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = 1

    @validator("item_id")
    def item_id_must_be_valid(cls, v):
        v = v.strip()
        if not v:
            raise ValueError("item_id cannot be empty")
        if ":" in v:
            raise ValueError("item_id contains invalid characters")
        return v

    @validator("quantity")
    def quantity_must_be_positive(cls, v):
        if v <= 0:
            raise ValueError("quantity must be greater than zero")
        return v
```

## Moderate Issues

### 5. DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises an exception, stock is decremented with no corresponding order record. This is a data integrity failure. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction doesn't support transactions, that is an architectural gap that needs to be addressed before this service can be safely deployed.

---

### 6. Heartbeat Interval Relative to TTL Is Unsafe Under Slow Redis

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10s, which gives three renewal windows within the 30s TTL. This looks fine on paper, but the heartbeat sleeps *first*, then checks. If the heartbeat thread is delayed by GIL contention, OS scheduling, or a slow Redis round-trip, the effective renewal window shrinks. A safer ratio is to renew at `TTL / 3` or less, and the heartbeat should sleep *after* the renewal, not before. The current `sleep → check → extend` order means the first extension doesn't happen until 10 seconds in.

---

### 7. `RuntimeError` Is Too Broad for Lock Failure

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a generic Python exception. Any `RuntimeError` raised inside the `with` block — including from `db.get_stock`, `db.update_stock`, or `db.create_order` — will be silently swallowed and returned as "service unavailable." Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

And catch only that.

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Away in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the TOCTOU race in the renewal and the thread lifecycle — is never tested. At minimum, add an integration-style test that uses a real (or fakeredis) Redis instance and verifies the lock is actually extended.

### 9. DB Transaction Failure Is Not Tested

There is no test covering the case where `create_order` raises after `update_stock` succeeds. Once the transaction boundary is added, this path must be tested to confirm rollback behavior.

### 10. Concurrent Access Is Not Tested

The entire purpose of this service is safe behavior under concurrency. There is no test using `threading.Thread` or similar to verify that two simultaneous `reserve_item` calls on the same item produce exactly one success and one failure (or correct combined deduction). This is a required test for a distributed locking service.

---

## Minor Notes

- `lock_key` uses `f"inventory:lock:{item_id}"` — once input validation is added, ensure the sanitized `item_id` is used in key construction, not the raw input.
- Consider whether `LOCK_TTL` and `HEARTBEAT_INTERVAL` should be configurable at construction time rather than class-level constants, to support different deployment environments.
- The `token` yielded from `_acquire_lock` is not used by the caller. Yielding it exposes internal lock state unnecessarily; consider yielding `None` or nothing.
