Decision: Request Changes

Summary: The core locking logic is well-structured and the intent is sound, but there are several correctness, safety, and architectural issues that must be addressed before this merges. The most critical problems are a race condition in the heartbeat, a shared-state concurrency bug that makes the service unsafe under concurrent use, and a missing atomicity guarantee on lock release. Secondary concerns around input validation, error handling, and test fidelity also need resolution.

---

## Critical Issues

### 1. `InventoryService` Stores Heartbeat State as Instance Variables — Concurrency Bug

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored on `self`, meaning a single `InventoryService` instance **cannot safely handle concurrent calls**. If two threads call `reserve_item` simultaneously:

- Thread A sets `self._heartbeat_active = True`
- Thread B calls `_stop_heartbeat()`, setting it to `False`
- Thread A's heartbeat silently dies while Thread A still holds the lock

This is a silent correctness failure — the lock expires mid-operation with no indication.

**Fix:** Heartbeat state must be scoped to each lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

### 2. Lock Release Is Not Atomic — TOCTOU Race Condition

The release sequence is:

```python
current = self.redis.get(lock_key)       # step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)          # step 2
```

Between step 1 and step 2, the lock can expire and be acquired by another process. You would then delete a lock you no longer own. This is a classic TOCTOU bug.

**Fix:** Use the standard Lua script for atomic check-and-delete, which is the industry-standard pattern for this exact problem:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. Heartbeat Has a Race Condition on Stop

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps first
        if self._heartbeat_active:            # then checks again
            ...
            self.redis.expire(lock_key, self.LOCK_TTL)
```

The thread sleeps *before* doing anything. If `LOCK_TTL=30` and `HEARTBEAT_INTERVAL=10`, there is a 10-second window at startup where the lock is not being renewed. More importantly, after `_stop_heartbeat()` is called, the thread can still execute `redis.expire` once more after the lock has been released, silently extending a key that may now belong to another owner.

**Fix:** Use `threading.Event.wait()` with a timeout so the thread responds to stop signals promptly and does not extend after release:

```python
def _start_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

## Significant Issues

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but there is none. `item_id` is passed directly into a Redis key (`inventory:lock:{item_id}`) and into database calls with no sanitization. A caller passing `item_id = "foo:bar"` or `item_id = ""` or `quantity = -5` will produce silent misbehavior or potentially manipulate the lock key namespace.

**Fix:** Validate at the entry point of `reserve_item`:

```python
import re

_ITEM_ID_RE = re.compile(r'^[a-zA-Z0-9_-]{1,64}$')

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not isinstance(item_id, str) or not _ITEM_ID_RE.match(item_id):
        return ReservationResult(success=False, message="Invalid item_id")
    if not isinstance(quantity, int) or quantity < 1 or quantity > 1000:
        return ReservationResult(success=False, message="Invalid quantity")
    ...
```

### 5. Database Operations Are Not Atomic — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against this — it only serializes access, it does not provide rollback.

**Fix:** These two operations must execute within a database transaction. The implementation detail depends on the DB layer, but the contract must be enforced:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support transactions, that is a separate architectural gap that must be resolved before this service can be considered production-safe.

---

### 6. Bare `Exception` Not Caught — Unhandled Errors Propagate Unexpectedly

Only `RuntimeError` is caught. If `self.db.get_stock` raises a connection error, a timeout, or any other exception, it will propagate out of `reserve_item` uncaught, bypassing the `finally` block in `_acquire_lock`... actually, `finally` will still run, but the caller gets a raw exception instead of a `ReservationResult`. The public contract of this method implies it returns a `ReservationResult`.

**Fix:** Catch broadly inside `reserve_item` and return a structured failure:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Test Issues

### 7. Tests Mock Out the Exact Behavior Being Tested

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- mocked out
    svc._stop_heartbeat = MagicMock()    # <-- mocked out
    return svc
```

The heartbeat and lock release paths are the highest-risk code in this PR. Mocking them means the tests provide no coverage for the bugs identified in issues 1–3 above. The tests validate business logic only, which is fine, but they should not be presented as validating the distributed locking behavior.

**Recommendation:** Add a separate integration test suite (even using `fakeredis`) that exercises the actual lock lifecycle, heartbeat renewal, and concurrent reservation attempts.

### 8. `test_lock_not_released_if_expired` Does Not Actually Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

Because `_start_heartbeat` and `_stop_heartbeat` are mocked on the `service` fixture, and the lock release logic reads `self.redis.get()` which returns `b"different-token"`, this test does exercise the ownership check. However, it does not test the atomic Lua release path (which should be the fix), and the assertion `delete.assert_not_called()` will break once the Lua script approach is adopted. Flag this test for revision alongside the fix for issue 2.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first renewal happens at T+10s, leaving a 20-second window before first renewal. This is fine mathematically but should be documented explicitly.
- The `daemon=True` on the heartbeat thread is correct — daemon threads will not block process shutdown.
- Structured logging format is consistent and appropriate for a production service.
- Pydantic `ReservationResult` model is a good pattern; consider adding `item_id
