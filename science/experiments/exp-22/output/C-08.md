Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and concurrency issues that must be resolved before this can be approved for a government-facing production system. The test suite also has meaningful gaps that mask real failure modes.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If a single `InventoryService` instance handles concurrent requests (which is the normal deployment model), two concurrent calls to `reserve_item` will race on these fields. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, potentially allowing the first lock to expire while still held.

**Fix:** Scope heartbeat state to the lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff,
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic check-and-delete via Lua script (see issue #2)
        self._release_lock(lock_key, token)

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 2. Lock Release Is Not Atomic (TOCTOU Race)

```python
# Current — non-atomic check-then-delete
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock. This is a classic TOCTOU race. The standard solution is a Lua script, which Redis executes atomically.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock not released — already expired or stolen key=%s", lock_key
        )
```

### 3. Database Operations Are Not Atomic (No Transaction Boundary)

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record. This leaves the system in an inconsistent state that cannot be recovered without manual intervention.

**Fix:** Wrap both operations in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager. If it does not, that interface needs to be defined before this PR merges.

### 4. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts arbitrary `item_id` strings and `quantity` integers with no validation whatsoever. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a string long enough to cause issues at the Redis key layer.

```python
# item_id="" produces lock key "inventory:lock:" — ambiguous
# quantity=-1 would increase stock silently
```

**Fix:** Validate at the entry point.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValueError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")

    lock_key = f"inventory:lock:{req.item_id}"
    # ... proceed with req.item_id and req.quantity
```

## Moderate Issues

### 5. Heartbeat Uses `time.sleep` — Not Responsive to Stop Signal

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 seconds
        ...
```

The thread sleeps for the full interval before checking the stop condition. On shutdown or after the lock is released, the thread lingers for up to 10 seconds. At scale, this accumulates into a significant number of zombie threads. The `threading.Event.wait(timeout=...)` pattern in the fix above resolves this.

---

### 6. Unhandled Redis Exceptions Propagate as Untyped Errors

```python
# No handling for redis.RedisError, ConnectionError, TimeoutError
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

A Redis connection failure will raise an unhandled exception that bypasses the `RuntimeError` catch in `reserve_item` and surfaces as a 500 to the caller with no structured response.

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.RedisError as e:
    logger.error("Redis error during lock acquisition key=%s: %s", lock_key, e)
    raise RuntimeError(f"Lock backend unavailable for {lock_key}") from e
```

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test stubs the heartbeat. There is zero test coverage of the actual heartbeat behavior — the most novel and failure-prone part of this implementation. At minimum, add:

- A test that verifies `expire` is called with the correct key and TTL when the token still matches
- A test that verifies `expire` is **not** called when the token no longer matches
- A test that verifies the heartbeat stops cleanly after `stop_event.set()`

### 8. The Atomic Release Test Is Incorrect

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test is asserting behavior of the non-atomic GET/DELETE pattern that should be replaced with the Lua script. Once the Lua script is in place, this test needs to be rewritten to assert that `eval` is called with the correct script and that `delete` is never called directly.

### 9. No Test for the DB Transaction Rollback Path

There is no test verifying that a failure in `create_order` rolls back the `update_stock` call. This is the most dangerous failure mode in the service and it has no coverage.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but document the reasoning explicitly: the heartbeat must fire at least twice before the lock expires to tolerate one missed beat.
- Consider whether `retry_attempts=3` should be configurable at the service level rather than per-call, to allow environment-specific tuning without code changes.
- The `daemon=True` on the heartbeat thread is correct — it prevents the thread from blocking interpreter shutdown — but add a comment explaining the intent so a future maintainer does not remove it thinking it is an oversight.
