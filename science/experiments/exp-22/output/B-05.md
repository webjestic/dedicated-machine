Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — a process pause (GC stop-the-world, OS preemption, VM live migration) suspends all threads simultaneously, including the heartbeat, meaning the lock can expire and be acquired by another process while the original process is mid-execution. When the original process resumes, it proceeds with `db.update_stock()` and `db.create_order()` with no awareness that it has lost lock ownership, producing a stale write that silently corrupts inventory. The heartbeat mechanism does not protect against this class of failure; it only addresses the case where the process is alive and running normally but the operation takes longer than the TTL. This is a well-documented limitation of distributed locking with Redis, and the Redlock paper and Martin Kleppmann's critique of it both address exactly this scenario.

The fix is not in the locking layer — it is in the database write layer. The implementation needs a fencing token mechanism. Below is a precise description of what is required and why.

---

## Root Cause

The lock provides *mutual exclusion under normal execution*, but it cannot provide a *guarantee of exclusive write authority* in the presence of process pauses. The sequence that breaks this implementation:

```
Process A acquires lock, token = "abc", lock TTL = 30s
Process A pauses (GC / preemption) for 35s
  → heartbeat thread is also paused
  → lock expires at Redis
Process B acquires lock, token = "xyz"
Process B reads stock = 10, writes stock = 9, commits
Process A resumes
Process A reads stock = 10 (stale read from before pause, or re-reads 9)
Process A writes stock = 9 (or 8), commits — STALE WRITE, oversells inventory
```

The result is two processes both believing they hold exclusive access and both writing, with no conflict detection at the storage layer.

---

## Required Fix: Fencing Token at the DB Write Layer

The canonical solution is a **monotonically increasing fencing token** issued at lock acquisition time, passed through to the database write, and enforced by the database with a conditional write. Redis can provide this via `INCR` on a per-resource counter at lock acquisition time. The token is then passed into every database mutation, and the database rejects any write carrying a token older than the last committed token.

### 1. Issue a fencing token at lock acquisition

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    fence_key = f"{lock_key}:fence"
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    # Monotonically increasing token — survives lock expiry and re-acquisition
    fencing_token = self.redis.incr(fence_key)

    self._start_heartbeat(lock_key, token)

    try:
        yield token, fencing_token
    finally:
        self._stop_heartbeat()
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
            logger.info("Lock released key=%s", lock_key)
```

### 2. Pass the fencing token into every database write

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"

    try:
        with self._acquire_lock(lock_key) as (token, fencing_token):

            current_stock = self.db.get_stock(item_id)

            if current_stock is None:
                return ReservationResult(
                    success=False,
                    message=f"Item {item_id} not found"
                )

            if current_stock < quantity:
                return ReservationResult(
                    success=False,
                    remaining_stock=current_stock,
                    message="Insufficient stock"
                )

            new_stock = current_stock - quantity
            order_id = str(uuid.uuid4())

            # DB layer must reject this write if fencing_token < last_seen_token
            updated = self.db.update_stock(
                item_id, new_stock, fencing_token=fencing_token
            )
            if not updated:
                logger.warning(
                    "Stale write rejected item_id=%s fencing_token=%d — "
                    "process was paused and lock was re-acquired by another process",
                    item_id, fencing_token
                )
                return ReservationResult(
                    success=False,
                    message="Reservation conflict. Please retry."
                )

            self.db.create_order(order_id, item_id, quantity, fencing_token=fencing_token)

            logger.info(
                "Reserved item_id=%s quantity=%d order_id=%s remaining=%d fence=%d",
                item_id, quantity, order_id, new_stock, fencing_token
            )

            return ReservationResult(
                success=True,
                order_id=order_id,
                remaining_stock=new_stock,
                message="Reservation successful"
            )

    except RuntimeError as e:
        logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
        return ReservationResult(
            success=False,
            message="Service temporarily unavailable. Please try again."
        )
```

### 3. Enforce the fencing token inside the database write (SQL example)

The database must persist the last accepted fencing token per item and reject writes from older tokens. This is the only layer that can enforce this guarantee, because it is the authoritative source of truth and is not subject to the process pause problem.

```sql
-- Schema addition required
ALTER TABLE inventory ADD COLUMN last_fence_token BIGINT NOT NULL DEFAULT 0;

-- Conditional update — rejects stale writes atomically
UPDATE inventory
SET stock = :new_stock,
    last_fence_token = :fencing_token
WHERE item_id = :item_id
  AND last_fence_token < :fencing_token;

-- Application checks rows_affected == 1
-- If rows_affected == 0, the write was rejected as stale
```

This is an atomic compare-and-write at the database level. Even if two processes both attempt the write concurrently after a pause scenario, only the one with the higher fencing token succeeds. The database serializes this correctly because the `WHERE` clause and the `SET` execute atomically within the row lock.

## Additional Issues That Must Be Addressed

**1. The `InventoryService` instance is not safe for concurrent use across threads.**

`_heartbeat_active` and `_heartbeat_thread` are instance-level fields with no synchronization. If a single `InventoryService` instance is shared across threads (which is the common pattern in web frameworks), two concurrent calls to `reserve_item` will race on these fields. The heartbeat state must be scoped to the lock acquisition context, not the instance.

```python
# Wrong — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None

# Correct — local to each lock acquisition
active_flag = threading.Event()
```

Refactor the heartbeat to use a `threading.Event` passed into the heartbeat function, scoped entirely within `_acquire_lock`, with no instance-level state.

**2. The `db.update_stock` and `db.create_order` calls are not atomic.**

If the process crashes or is killed between these two calls, the stock is decremented but no order record is created. These two operations must execute within a single database transaction. The current design does not enforce this.

**3. The test for lock expiry does not test the actual vulnerability.**

`test_lock_not_released_if_expired` verifies that `redis.delete` is not called when the token has changed. This is correct behavior for lock release, but it does not test the stale write scenario — the scenario where the DB write proceeds after the lock has expired. A test must be added that verifies `db.update_stock` is *not called*, or returns a rejected result, when a stale fencing token is detected.

**4. Input validation is incomplete.**

`quantity` accepts any integer. A caller passing `quantity=0` or `quantity=-5` would pass the stock check (`current_stock >= 0` is always true for a non-negative stock value) and write a nonsensical or exploitable stock value. Add explicit validation:

```python
if quantity <= 0:
    return ReservationResult(
        success=False,
        message="Quantity must be a positive integer"
    )
```

## What Must Change Before Approval

| # | Issue | Severity |
|---|-------|----------|
| 1 | No fencing token — stale writes possible after process pause | **Critical** |
| 2 | Heartbeat state is instance-level, not thread-safe for concurrent callers | **High** |
| 3 | `update_stock` and `create_order` not wrapped in a single transaction | **High** |
| 4 | Test suite does not cover the stale write scenario | **Medium** |
| 5 | No validation that `quantity > 0` | **Medium** |

The architecture of the locking approach is sound for the cases it does handle. The heartbeat, exponential backoff, token-verified release, and structured logging are all implemented correctly. The gap is specifically in what happens when the process is paused for longer than the TTL, which is a real operational condition in any containerized or virtualized environment, and which cannot be solved at the Redis layer alone.
