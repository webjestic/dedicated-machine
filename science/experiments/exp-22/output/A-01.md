Decision: Request Changes

Summary: The senior engineer's flag is a genuine, unresolved correctness flaw — not a theoretical edge case. The heartbeat thread is suspended during any stop-the-world pause alongside the main thread, meaning the lock can expire and be acquired by another process while the original process is mid-execution; when the original resumes, `db.update_stock()` proceeds without any awareness that lock ownership was lost, producing a stale write with no detection. The DB write layer requires an optimistic concurrency control mechanism — a version/ETag column with a conditional `UPDATE ... WHERE version = :expected_version AND stock >= :quantity` — so the database itself rejects the write if the precondition is violated, making the lock a performance optimization rather than the sole correctness guarantee. Additionally, the `InventoryService` instance holds heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`), which means concurrent calls to `reserve_item` on the same service instance will race on those fields and corrupt each other's heartbeat lifecycle; the heartbeat state must be scoped to each lock acquisition, not to the service object.

### Critical — Stale Write on Process Pause (Correctness Flaw)

This is the issue the senior engineer raised, and the implementation does not address it.

**The failure sequence:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
  heartbeat thread starts
  read stock → 5
  [OS pauses entire process]
  [LOCK_TTL elapses]
                                   acquire lock (token=B)
                                   read stock → 5
                                   write stock → 4  ✓
  [OS resumes process A]
  heartbeat thread resumes
  (lock now owned by B)
  db.update_stock(item, 4)  ← stale write, no error raised
  db.create_order(...)      ← phantom order created
```

The heartbeat cannot prevent this. Both the main thread and the heartbeat thread are OS threads scheduled by the same process. A stop-the-world GC pause, `SIGSTOP`, or sufficiently long OS preemption suspends all threads simultaneously. The heartbeat provides no protection against pauses that exceed `LOCK_TTL`.

**Required fix — optimistic concurrency at the DB layer:**

The database write must be conditional. The stock row needs a version counter or an ETag. The `UPDATE` must assert the precondition:

```python
# Schema addition (conceptual)
# stock_items: (item_id, quantity, version)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"

    try:
        with self._acquire_lock(lock_key):
            row = self.db.get_stock_with_version(item_id)

            if row is None:
                return ReservationResult(success=False, message=f"Item {item_id} not found")

            current_stock, version = row.quantity, row.version

            if current_stock < quantity:
                return ReservationResult(
                    success=False,
                    remaining_stock=current_stock,
                    message="Insufficient stock"
                )

            new_stock = current_stock - quantity
            order_id = str(uuid.uuid4())

            # Conditional write — rejected if version has changed since we read
            rows_affected = self.db.update_stock_conditional(
                item_id,
                new_stock,
                expected_version=version
            )

            if rows_affected == 0:
                # Another process wrote between our read and our write
                logger.warning(
                    "Concurrent modification detected item_id=%s version=%d",
                    item_id, version
                )
                return ReservationResult(
                    success=False,
                    message="Concurrent modification. Please retry."
                )

            self.db.create_order(order_id, item_id, quantity)
            ...
    ...
```

The corresponding DB operation must be atomic:

```sql
UPDATE stock_items
   SET quantity = :new_stock,
       version  = version + 1
 WHERE item_id  = :item_id
   AND version  = :expected_version
   AND quantity >= :quantity;
-- rows_affected == 0 means the precondition failed
```

With this in place, a stale write from a resumed-after-pause process is rejected by the database itself. The distributed lock becomes a contention-reduction optimization, not the sole safety boundary.

### Critical — Heartbeat State Is Instance-Scoped, Not Call-Scoped

```python
# Current — instance variables shared across concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance concurrently, the second call overwrites `_heartbeat_active` and `_heartbeat_thread` while the first call's heartbeat is still running. `_stop_heartbeat()` in the first call's `finally` block then stops the second call's heartbeat, and vice versa.

**Required fix — scope heartbeat state to the lock acquisition:**

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local to this acquisition

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_active.set()
    heartbeat_thread = threading.Thread(
        target=self._beat,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
            logger.info("Lock released key=%s", lock_key)

def _beat(self, lock_key: str, token: str, active: threading.Event) -> None:
    while active.is_set():
        time.sleep(self.HEARTBEAT_INTERVAL)
        if active.is_set():
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
```

### Moderate — Heartbeat Has a Check-Then-Act Race

```python
# In _beat():
current = self.redis.get(lock_key)          # 1. read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # 2. extend — not atomic with step 1
```

Between step 1 and step 2, another process could acquire the lock with a new token. The `expire` call then extends a lock the heartbeat no longer owns. This must be a single atomic operation using a Lua script:

```lua
-- extend_if_owner.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In _beat():
extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.warning("Heartbeat: lock no longer owned, stopping key=%s", lock_key)
    active.clear()
```

The same atomicity requirement applies to the release in `_acquire_lock`'s `finally` block — the existing `GET`-then-`DELETE` pattern has the same race. It should also use a Lua script:

```lua
-- release_if_owner.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### Minor — No Input Validation on `item_id` or `quantity`

The PR checklist marks "Input validation via Pydantic" but `reserve_item` accepts a raw `str` and `int` with no validation. An empty string, a negative quantity, or an oversized string reaches the lock key and DB layer unchecked.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)
```
