Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this is suitable for a production inventory system where double-reservations or silent data loss would be unacceptable.

---

## Critical Issues

### 1. Race Condition in Heartbeat Lock Extension

The heartbeat performs a non-atomic check-then-extend:

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # extend — NOT atomic
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You would then extend a lock you no longer own. This is a TOCTOU race. The fix is a Lua script that atomically checks ownership and extends in a single round-trip:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same race exists in the lock release in `_acquire_lock`. That also needs to be an atomic Lua script (this is the canonical Redlock release pattern).

---

### 2. Shared Mutable State Makes the Service Non-Reentrant and Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — one call's `_stop_heartbeat()` will kill the other call's heartbeat thread. This is a silent correctness failure under the exact load conditions this code is designed to handle. Heartbeat state must be scoped to the lock context, not the instance. Pass a mutable flag object into the thread closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = {"running": True}  # local to this acquisition
    ...
    thread = self._start_heartbeat(lock_key, token, active)
    try:
        yield token
    finally:
        active["running"] = False
        ...
```

### 3. Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The reservation is silently lost. These two operations must be wrapped in a database transaction. If the `db` abstraction does not support transactions, that is an architecture gap that needs to be addressed before this merges:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 4. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts arbitrary `item_id` strings and `quantity` integers with no validation. A negative quantity, zero quantity, or a malformed `item_id` (e.g., one containing Redis key injection characters like `:` or `*`) will pass through silently. Add a Pydantic input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

The `lock_key` is constructed via f-string from unvalidated user input. Without the pattern constraint above, a crafted `item_id` could target arbitrary Redis keys.

## Moderate Issues

### 5. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a base class that could be raised by the database layer, the Redis client, or internal Python machinery for reasons entirely unrelated to lock acquisition. Introduce a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

Raise and catch that instead. Swallowing unexpected `RuntimeError`s from the DB layer as "try again later" is a silent failure mode.

---

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat` sets the flag to `False` but does not join the thread. The thread sleeps for `HEARTBEAT_INTERVAL` (10 seconds) before checking the flag. Under high throughput, you will accumulate many sleeping daemon threads. Add a join with a timeout in the `finally` block:

```python
active["running"] = False
if thread is not None:
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Test Coverage Gaps

- The heartbeat is mocked out in every test. There is no test verifying the heartbeat actually extends the lock or correctly skips extension when the token has changed. The most important behavioral guarantee of this class is untested.
- There is no test for the non-atomic DB failure scenario (stock decremented, order creation raises). Even if a transaction wrapper is added, the failure path should be tested.
- `test_reserve_lock_contention` does not verify the number of retry attempts made against Redis. The retry loop could be broken and this test would still pass.

---

## Minor Issues

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at t=10 and extends to t=40. This is fine, but it should be documented explicitly. The comment says "extend lock before it expires" but the math should be stated: heartbeat interval must be strictly less than TTL/2 to be safe under any single GC pause or network hiccup.
- The `token` yielded from `_acquire_lock` is not used by the caller (`with self._acquire_lock(lock_key):` discards it via `_`). Either remove the yield value or use it explicitly to make the ownership model clearer to future readers.
