Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — a process pause (GC stop-the-world, OS preemption, VM live migration) suspends all threads simultaneously, including the heartbeat, allowing the lock to expire and be acquired by another process; when the original process resumes, it executes `db.update_stock()` with no awareness that it has lost lock ownership, producing a stale write and a double-reservation. The heartbeat pattern cannot defend against this class of failure because the mechanism protecting the lock lives in the same process that can be paused. Beyond that critical flaw, there are additional architectural and correctness issues that must be addressed before this is mergeable.

## Critical: Stale Write on Process Pause (Flagged Scenario — Unresolved)

This is the core issue the reviewer notes surface, and the implementation does not address it.

**The failure sequence:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
  heartbeat thread starts
  [OS pauses entire process A]
                                   [lock TTL expires]
                                   acquire lock (token=B)
                                   read stock = 10
                                   write stock = 9  ✓
  [OS resumes process A]
  read stock = 10  (stale read, already committed by B)
  write stock = 9  ✗  (double-reservation, silent data corruption)
```

The heartbeat thread is suspended along with the main thread. There is no OS-level guarantee that a Python thread will be scheduled within any bounded wall-clock window. The heartbeat is a best-effort mechanism under normal load, not a safety guarantee under adversarial or resource-constrained conditions.

**What is actually required — a fencing token enforced at the DB write layer:**

The lock must issue a monotonically increasing fencing token (e.g., a Redis `INCR` counter at lock acquisition time), and the database write must be conditional on that token still being the highest seen for that resource. This is the standard solution described by Kleppmann in *Designing Data-Intensive Applications* (§8, "The Truth Is Defined by the Majority").

```python
# At lock acquisition time
fence = self.redis.incr(f"inventory:fence:{item_id}")

# At DB write time — the DB layer must enforce this
self.db.update_stock(item_id, new_stock, fence_token=fence)
```

The DB write becomes a conditional update:

```sql
UPDATE inventory
SET stock = :new_stock, last_fence = :fence
WHERE item_id = :item_id
  AND last_fence < :fence   -- rejects any write from a process that lost the lock
```

Without this, the distributed lock provides ordering under normal conditions only. It does not provide safety under pause/preemption scenarios. For a flash sale inventory system — where a double-reservation is a direct financial and operational liability — this is not an acceptable risk posture.

## High: Non-Atomic Heartbeat Check-Then-Extend

```python
# Current implementation — two separate round trips
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. The extension then silently resets a lock this process no longer owns. This must be a single atomic operation via a Lua script:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(EXTEND_LOCK_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    self._heartbeat_active = False  # signal main thread: lock was lost
    logger.error("Heartbeat detected lock loss key=%s", lock_key)
```

The same atomicity problem exists in the lock release path (`GET` then `DELETE`), which should also use a Lua script — this is the standard Redlock release pattern.

## High: No Atomicity Between `update_stock` and `create_order`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If the process crashes, is killed, or encounters a network partition between these two calls, stock is decremented but no order record exists. This is a silent inventory leak.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transactional context manager, and this code must use it.

## High: Shared Heartbeat State Prevents Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the expected usage pattern for a service object), two concurrent reservations will share and corrupt each other's heartbeat state. The heartbeat state must be scoped to the lock context, not the service instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread closure
    ...
```

## Medium: Input Validation Is Declared But Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. `item_id` is interpolated directly into a Redis key without sanitization. A caller passing `item_id = "foo:bar"` or an empty string will silently produce a malformed or colliding lock key.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

`reserve_item` should accept a `ReservationRequest` and validate at the boundary.

---

## Medium: Heartbeat Does Not Signal Lock Loss to the Main Thread

If the heartbeat detects that the token no longer matches (lock was lost or expired), it currently just stops extending. The main thread has no way to know this has happened and will proceed to execute the DB write against a lock it no longer holds.

The heartbeat must set a shared flag or event that the main thread checks before executing writes:

```python
if not lock_still_valid.is_set():
    raise LockLostError(f"Lock lost during operation for key={lock_key}")
```
