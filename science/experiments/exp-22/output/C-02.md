Decision: Request Changes

Summary: The core locking logic and heartbeat concept are sound, but there are several concurrency and correctness flaws that would be dangerous in a high-throughput flash sale context. The most critical issues are a race condition in the heartbeat's check-then-extend sequence, an instance-level heartbeat state design that breaks under concurrent calls, and a missing atomicity guarantee on lock release. These must be resolved before this code touches production inventory.

---

## Review

### 🔴 Critical — Must Fix

**1. Heartbeat state is instance-level, not call-level**

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern — a single service object shared across a thread pool), one call's `_stop_heartbeat()` will kill the other call's heartbeat thread.

```python
# Current — shared across all concurrent callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

Fix: Pass heartbeat state through a local `threading.Event` scoped to each lock acquisition.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        ...

def _start_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    threading.Thread(target=_beat, daemon=True).start()
```

---

**2. Lock release is not atomic — TOCTOU race**

The release sequence is:

```python
current = self.redis.get(lock_key)       # Step 1: read
if current and current.decode() == token:
    self.redis.delete(lock_key)          # Step 2: delete
```

Between steps 1 and 2, the lock can expire and be acquired by another process. You would then delete a lock you no longer own, corrupting another caller's critical section. This is a well-known Redis distributed lock pitfall.

Fix: Use a Lua script to make the check-and-delete atomic, which is the standard pattern per the Redis documentation.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

---

**3. Heartbeat has the same TOCTOU race**

```python
current = self.redis.get(lock_key)           # Step 1
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2
```

Same problem as above. The lock can expire between the GET and the EXPIRE, causing the heartbeat to extend a lock now owned by someone else. Use a Lua script here as well.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 🟡 Significant — Should Fix

**4. No input validation on `reserve_item`**

The checklist claims input validation via Pydantic, but `item_id` and `quantity` are raw, unvalidated parameters. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string, all of which would silently produce nonsensical results (negative stock, empty lock keys, etc.).

```python
# quantity=0 passes the stock check and creates a zero-quantity order
# quantity=-1 would *increase* stock
if current_stock < quantity:  # -1 < 10 is False — this passes
```

Fix: Validate at the entry point.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

**5. `db.update_stock` and `db.create_order` are not transactional**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record — inventory is silently lost. The Redis lock does not protect against application-layer partial failure.

Fix: Wrap both operations in a database transaction, or implement a compensating rollback.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

**6. Bare `RuntimeError` catch masks unexpected failures**

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is also raised by Python internals for unrelated reasons (e.g., generator issues, thread state errors). Use a specific custom exception so lock failures are distinguished from programming errors.

```python
class LockAcquisitionError(Exception):
    pass

# In _acquire_lock:
raise LockAcquisitionError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

# In reserve_item:
except LockAcquisitionError as e:
    ...
```

### 🔵 Minor — Recommended

**7. Tests mock out the exact code under test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat logic — which contains two of the critical race conditions above — is completely bypassed in every test. Add at least one integration-style test that runs the real heartbeat against a mock Redis to verify the extend script is called with the correct token.

**8. `time.sleep` in the retry loop blocks the calling thread**

Under a flash sale load spike, hundreds of threads could be sleeping simultaneously waiting for lock retry. Consider whether async primitives or a queue-based approach is more appropriate for the expected concurrency profile.

**9. `LOCK_TTL = 30` is undocumented as a tuning parameter**

Thirty seconds is generous. If a DB operation genuinely takes that long, the system has larger problems. Document the expected p99 DB latency this value is sized for, and consider surfacing it as a constructor parameter so it can be tuned per deployment without a code change.
