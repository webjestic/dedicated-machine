Decision: Request Changes

Summary: The senior engineer's flag is the critical blocker here. The heartbeat thread and the Python GIL/OS scheduler operate in fundamentally different domains — a stop-the-world GC pause or OS preemption suspends *all* threads simultaneously, meaning the heartbeat cannot extend a lock that has already expired before the pause ends. When the original process resumes, it proceeds to call `db.update_stock()` with no awareness that it has lost lock ownership, and potentially another process has already acquired the lock and begun its own write. This is a classic stale write / phantom lock scenario, and the implementation as written provides no protection against it at the database layer.

---

## Detailed Findings

### Critical — Stale Write on Lock Expiry After Process Pause

This is the core issue the senior engineer flagged, and it is not addressed.

**The failure sequence:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
  heartbeat thread running...
  [OS preemption / GC pause — ALL threads suspended]
                                   [lock TTL expires]
                                   acquire lock (token=B)
                                   read stock = 10
                                   write stock = 9  ✓
  [Process A resumes]
  heartbeat never fired — lock already expired
  db.update_stock(item_id, 9)  ← STALE WRITE, no guard
  db.create_order(...)         ← PHANTOM ORDER
```

The result is a double-decrement on stock and a phantom order created outside any valid lock window. For a flash sale inventory system this is a direct business integrity failure — overselling being the obvious consequence.

**What the heartbeat does and does not protect against:**

The heartbeat correctly handles the *slow but running* case — a DB operation that takes 25 seconds on a 30-second TTL. It does not and cannot protect against a pause that outlasts the TTL, because the renewal thread is also paused. This is not a theoretical edge case; GC pauses in CPython under memory pressure, and OS scheduler preemption under CPU contention (exactly the conditions a flash sale creates), make this a realistic production scenario.

**Required fix — optimistic fencing at the DB write layer:**

The standard mitigation is a *fencing token* (Kleppmann, "Designing Data-Intensive Applications", §8). The lock service issues a monotonically increasing token on each acquisition. The DB write layer rejects any write carrying a token lower than the highest it has already accepted.

```python
# Conceptual shape — DB layer must enforce the fence
acquired, fence_token = self.redis.acquire_lock_with_fence(lock_key, self.LOCK_TTL)

# ...

# DB write includes the fence token; DB rejects if a higher token has already written
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
self.db.create_order(order_id, item_id, quantity, fence_token=fence_token)
```

The DB layer (or a transactional wrapper around it) must then enforce:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id
  AND last_fence_token < :fence_token;  -- reject stale writers
```

If `ROW_COUNT == 0`, the write was rejected as stale and the caller must surface a failure, not a success.

An alternative for simpler DB layers is a **compare-and-swap on a version column**:

```sql
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND version = :expected_version;  -- read version must still match
```

Either approach closes the window. The Redis lock alone cannot.

### Moderate — Heartbeat Thread Leaks on Unhandled Exception Paths

`_stop_heartbeat()` sets `_heartbeat_active = False` but does not join the thread. If an exception propagates out of the `finally` block (e.g., `redis.get()` raises a `ConnectionError` during lock release), the thread may outlive the service instance in long-running processes. The thread is daemonized so it will not block shutdown, but it will continue issuing `redis.expire` calls against a key it no longer owns until the next `sleep` cycle completes.

```python
finally:
    self._stop_heartbeat()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # ... release lock
```

---

### Moderate — Instance-Level Heartbeat State Prevents Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance attributes, not scoped to the lock context. If `reserve_item` is ever called concurrently on the same `InventoryService` instance (e.g., from a thread pool), two concurrent acquisitions will share and clobber the same heartbeat state. The heartbeat management should be encapsulated within the context manager's local scope, not on `self`.

```python
# Preferred: pass state through closure, not instance attributes
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()
    def _beat():
        while not stop_event.wait(self.HEARTBEAT_INTERVAL):
            ...
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event, t
```

### Minor — Input Validation Is Declared but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. `ReservationResult` is a Pydantic output model, not an input validator. There is no guard against `quantity <= 0`, empty `item_id`, or excessively large quantities. A `ReservationRequest` input model should be introduced.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

---

### Minor — Test Coverage Gaps

- No test exercises the stale-write scenario (the most important correctness property).
- `test_lock_not_released_if_expired` mocks `redis.get` to return a mismatched token but does not assert that `db.update_stock` was still called — it does not verify whether the *write* was blocked, only whether the *lock release* was skipped. These are separate concerns and the test title is misleading.
- No test covers the `redis.set` raising a `ConnectionError`, which is a realistic failure mode under flash sale load.

## Summary of Required Changes Before Approval

| Priority | Item |
|---|---|
| **Blocker** | Add fencing token or optimistic version check at DB write layer to prevent stale writes after lock expiry |
| Moderate | Scope heartbeat state to lock context, not instance, to allow safe concurrent use |
| Moderate | Join heartbeat thread in `finally` block |
| Minor | Add `ReservationRequest` Pydantic input model with `quantity >= 1` guard |
| Minor | Add test asserting DB write is rejected when lock has expired |

The architecture of the locking and heartbeat approach is sound for the *slow-but-running* case. The gap is specifically at the DB write layer, which must be the last line of defense against a process that resumes after losing its lock. That fix is required before this can be approved.
