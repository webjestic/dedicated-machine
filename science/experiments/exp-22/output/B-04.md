Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — a process pause (GC, OS preemption, VM migration) suspends all threads simultaneously, including the heartbeat, allowing the lock TTL to expire and be acquired by another process; when the original process resumes, it executes `db.update_stock()` and `db.create_order()` with no awareness that it has lost the lock, producing a stale write and a potential double-reservation. The heartbeat mechanism cannot protect against this class of failure because it shares the same process fate as the main thread. Beyond that critical flaw, there are additional architectural and correctness issues that must be addressed before this is suitable for a production inventory system.

---

## Critical Issue: Stale Write on Process Pause (Flagged Scenario)

This is the core problem the reviewer flagged, and the implementation does not solve it.

### What the heartbeat actually protects against

The heartbeat defends against one thing: a legitimately slow operation that predictably consumes more wall-clock time than `LOCK_TTL`. If the DB write takes 25 seconds and `LOCK_TTL` is 30, the heartbeat extends the TTL before expiry. That is valid and correct.

### What the heartbeat cannot protect against

A stop-the-world pause — GC, OS scheduler preemption, hypervisor CPU steal, `SIGSTOP`, live migration — suspends the **entire process**. Every thread stops. The heartbeat thread stops. The wall clock continues. If the pause exceeds `LOCK_TTL`, the lock expires in Redis. Another process acquires it and begins its own reservation. When the original process resumes, it has no signal that anything happened. It proceeds to:

```python
self.db.update_stock(item_id, new_stock)   # stale write
self.db.create_order(order_id, item_id, quantity)  # phantom order
```

At this point two processes believe they hold the reservation. Stock can go negative. Two orders can be created for the same unit. This is a correctness failure in an inventory system, which in a government contracting context could mean fraudulent fulfillment records.

### What is required at the DB write layer

The heartbeat operates at the Redis layer. The fix must operate at the **database layer**, because that is where the write actually lands. Two viable approaches:

**Option A — Optimistic locking with a version/fence token**

Pass the lock's `token` (or a monotonic fencing token) into the DB write as a precondition. The DB write becomes a conditional update:

```sql
UPDATE inventory
SET stock = :new_stock, lock_token = :token
WHERE item_id = :item_id
  AND lock_token = :expected_token   -- or a version column
  AND stock = :expected_stock;       -- belt-and-suspenders
```

If the row was already written by the process that stole the lock, `expected_token` will not match and the update affects 0 rows. The application checks `rowcount == 0` and treats it as a conflict. This is the standard fencing token pattern described by Martin Kleppmann in the critique of Redlock.

**Option B — Serializable / SELECT FOR UPDATE transaction**

Wrap `get_stock` and `update_stock` in a single database transaction with `SELECT ... FOR UPDATE` (pessimistic row lock). The distributed Redis lock becomes a coarse admission gate; the DB transaction provides the true atomicity guarantee. A stale writer will either block on the row lock or fail on commit if the row version has changed.

**Option C — Atomic compare-and-swap at the DB layer**

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id AND stock >= :quantity;
```

Check affected rows. If 0, the reservation failed. No separate read required. This eliminates the read-modify-write race entirely.

Any of these options must be present. The Redis lock alone is insufficient.

## Additional Issues

### 1. Non-atomic heartbeat check-then-extend (TOCTOU)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The `EXPIRE` then extends a lock owned by someone else. This must be a single atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 2. Non-atomic lock release (same TOCTOU)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same race: between `GET` and `DELETE`, the lock can expire and be re-acquired. The `DELETE` then releases another process's lock. The canonical fix is a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is well-established practice and not optional for a correct distributed lock.

### 3. Instance-level heartbeat state is not re-entrant

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool), two calls will share and corrupt `_heartbeat_active`. The heartbeat state must be scoped to the lock acquisition context, not the service instance.

### 4. `db.update_stock` and `db.create_order` are not atomic

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The reservation is silently lost. These must execute within a single database transaction with rollback on failure.

### 5. No input validation on `item_id` or `quantity`

The PR claims "input validation via Pydantic" in the checklist, but `reserve_item` accepts a raw `str` and `int` with no validation. There is no Pydantic model on the input path. `quantity` can be zero, negative, or arbitrarily large. `item_id` is passed directly into a Redis key and a DB query with no sanitization. At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError(...)
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

### 6. Test suite mocks out the components under test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — the primary concurrency mechanism this PR introduces — is mocked in every test. The tests verify the happy-path business logic but provide zero coverage of the locking behavior, the TOCTOU race in release, or the stale-write scenario. The test for `test_lock_not_released_if_expired` is structurally correct in intent but relies on the mock bypassing the actual lock context manager, so it does not exercise the real code path.

### 7. `retry_attempts=3` produces misleading backoff logging

On the final attempt, the code logs a retry warning and sleeps before the loop exits and raises. The sleep on the last attempt is wasted latency and the log message implies a retry that will not happen. The backoff sleep should be conditional on whether another attempt will follow.
