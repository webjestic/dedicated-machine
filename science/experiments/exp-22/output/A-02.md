Summary: The senior engineer's flag is the critical issue here — this implementation does not prevent stale writes after a process pause exceeding `LOCK_TTL`, and the heartbeat architecture cannot solve that by design. Beyond that core flaw, there are several additional concurrency, correctness, and test-coverage issues that must be addressed before this is production-safe for a high-demand flash sale context.

## Review

### Critical: Stale Write After Process Pause (Raised by Senior Engineer)

The reviewer's scenario is valid and this implementation **does not protect against it**.

Here is the exact failure sequence:

1. Process A acquires the lock, heartbeat thread starts
2. Process A is paused by the OS scheduler or a GC stop-the-world event
3. All threads in the process are suspended simultaneously — including the heartbeat thread
4. `LOCK_TTL` (30s) elapses; Redis expires the lock
5. Process B acquires the lock, reads stock, begins its write
6. Process A resumes, has no awareness that it lost the lock, and proceeds to call `db.update_stock()` and `db.create_order()`
7. Both processes have now written to the database — this is a **phantom reservation / double-spend**

The heartbeat cannot prevent this. A heartbeat is a liveness signal, and a paused process cannot emit one. This is a fundamental limitation of process-external TTL-based locks.

**Required fix at the DB write layer:** The database write must include a fencing token or optimistic concurrency check so that a stale writer is rejected at the storage layer, independent of what Redis believes.

Two viable approaches:

**Option A — Optimistic locking with a version/generation counter:**
```python
# DB schema carries a `version` column (integer, monotonically increasing)
# Read the version inside the lock
current_stock, version = self.db.get_stock_with_version(item_id)

# Write only succeeds if the version has not changed since we read it
rows_affected = self.db.update_stock_if_version(
    item_id, new_stock, expected_version=version
)
if rows_affected == 0:
    # Another writer committed between our read and write — abort
    raise ConcurrentModificationError(item_id)
```
The SQL equivalent is:
```sql
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id AND version = :expected_version;
```
If `rows_affected == 0`, the lock was lost and another writer already committed. The stale writer is safely rejected.

**Option B — Monotonic fencing token passed through to the DB:**
Redis's `WAIT` or a Zookeeper-style fencing token (a monotonically increasing integer issued at lock acquisition time) can be written alongside the update. The DB rejects any write carrying a token older than the last committed token. This is the approach Martin Kleppmann describes in his critique of Redlock. This requires more infrastructure but is strictly stronger.

Option A is the pragmatic choice here given the existing stack.

### High: Shared Heartbeat State Breaks Concurrent `reserve_item` Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not scoped to a single lock acquisition. If two calls to `reserve_item` overlap on the same `InventoryService` instance (which is the normal case under load), the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first call's `_stop_heartbeat` sets `_heartbeat_active = False`, killing the second call's heartbeat prematurely.

```python
# Both calls share this state — this is a race condition
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance. Pass a `threading.Event` into the heartbeat thread:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

```python
def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### High: Lock Release Is Not Atomic

The release sequence in `finally` is a read-then-delete, which is a TOCTOU race:

```python
current = self.redis.get(lock_key)          # step 1
if current and current.decode() == token:   # step 2
    self.redis.delete(lock_key)             # step 3 — not atomic with steps 1-2
```

Between steps 1 and 3, another process can acquire the lock (if it expired) and set a new token. The `get` returns the old token match, but `delete` then removes the new owner's lock.

**Fix:** Use the standard Lua script for atomic check-and-delete, which is the Redis-documented correct approach:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### Medium: `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order record exists. This is a partial write — inventory is consumed but the reservation is invisible.

**Fix:** Both operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### Medium: Input Validation Is Declared But Not Implemented

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is imported and used only for the response model.

At minimum:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

Without this, `item_id` values like `"inventory:lock:other_item"` could be used to craft a lock key that collides with another item's lock.

### Low: Heartbeat Uses `time.sleep` — Imprecise and Uninterruptible

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)  # cannot be interrupted
    if self._heartbeat_active:
        ...
```

`time.sleep` cannot be interrupted by a flag change. The thread will sleep the full `HEARTBEAT_INTERVAL` after `_stop_heartbeat()` is called before it exits. With the `threading.Event` fix above, `stop_event.wait(timeout=...)` is both precise and immediately interruptible.

---

### Low: Test Suite Does Not Cover the Failure Scenarios That Matter Most

The tests mock out the heartbeat entirely (`svc._start_heartbeat = MagicMock()`), which means no concurrency behavior is tested at all. The following cases are missing:

| Missing Test | Why It Matters |
|---|---|
| Lock expires mid-operation (stale write scenario) | The core correctness concern flagged in the PR |
| `update_stock` succeeds but `create_order` raises | Partial write / data inconsistency |
| Heartbeat extends TTL correctly | The heartbeat is the primary availability mechanism |
| Concurrent calls on the same instance | Validates the shared-state race described above |
| Malformed `item_id` input | Input validation boundary |

`test_lock_not_released_if_expired` is a good test but it only checks that `delete` is not called — it does not assert that the DB write was also aborted, which is the actual safety requirement.
