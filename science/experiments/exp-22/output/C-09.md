Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw in the heartbeat design — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields, causing one call's heartbeat to clobber another's, or the stop signal to bleed across calls. Beyond that, the lock release in the `finally` block contains a TOCTOU race (GET then DELETE is not atomic), and there are several architectural concerns that need to be addressed before this is safe to merge into a production reservation path.

---

### Critical Issues

#### 1. Instance-level heartbeat state is not thread-safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same service instance (which is the entire point of this service), they share `_heartbeat_active`. Thread A calling `_stop_heartbeat()` sets `_heartbeat_active = False`, which also kills Thread B's heartbeat. This is a silent correctness failure — no exception is raised, but B's lock will expire and be stolen mid-operation.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

#### 2. Lock release is a TOCTOU race (non-atomic GET + DELETE)

```python
# Current — two separate round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process. You would then delete a lock you no longer own. This is a well-known Redis locking pitfall. The canonical fix is a Lua script, which Redis executes atomically.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally
released = self._release_script(keys=[lock_key], args=[token])
if not released:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

#### 3. Database operations are not atomic with respect to each other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against partial DB failure. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architectural gap that must be resolved before this merges.

### Moderate Issues

#### 4. No input validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. An empty string, a negative quantity, or a SQL-injectable value passes through unchecked. Pydantic is imported but not used at the entry point.

from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):

    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:

#### 5. Heartbeat does not join before lock release

In the original code, `_stop_heartbeat()` sets a flag but does not wait for the thread to finish. The heartbeat thread could fire one final `expire` call *after* the lock has been deleted, resurrecting a lock that should be gone.

The fix in Issue 1 above addresses this with `t.join()`.

---

#### 6. Bare `RuntimeError` is too broad in the exception handler

```python
except RuntimeError as e:
    ...
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

Any `RuntimeError` from inside the lock body (e.g., from `db.update_stock`) will be silently swallowed and returned as "temporarily unavailable," masking real errors. Introduce a dedicated exception type.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError here
```

### Test Coverage Gaps

#### 7. Heartbeat is fully mocked out in every test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the thread-safety bug in Issue 1 — is never exercised. At minimum, one integration-style test should run the real heartbeat path against a mock Redis to verify TTL extension and clean shutdown.

#### 8. The TOCTOU scenario is not tested

`test_lock_not_released_if_expired` checks that `delete` is skipped when the token mismatches, but it does not test the window between GET and DELETE where a race can occur. Once the Lua script is in place, a test should verify the script is called with the correct keys and args rather than testing GET/DELETE separately.

#### 9. No test for concurrent access

Given this is a concurrency primitive, there should be at least one test using `threading.Thread` or `concurrent.futures.ThreadPoolExecutor` to verify that two simultaneous reservations on the same item do not both succeed when only one unit of stock remains.

---

### Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but both should be configurable at construction time rather than class-level constants, to allow environment-specific tuning without subclassing.
- The `db` parameter has no type annotation or interface contract (ABC or Protocol). For a government contracting context where auditability matters, the expected interface should be explicit.
- Consider `redlock` (multi-node Redis) if this will run against a Redis cluster; single-node Redlock has known failure modes under network partition.
