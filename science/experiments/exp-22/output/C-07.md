Decision: Request Changes

Summary: The implementation has a critical concurrency flaw that undermines the entire purpose of a distributed lock: the `InventoryService` instance stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance variables, meaning a single service instance cannot safely handle concurrent `reserve_item` calls — a second call will stomp the heartbeat state of the first, potentially killing an active lock's renewal mid-operation. Beyond that, the lock release in the `finally` block contains a TOCTOU race (get-then-delete is not atomic), and the test suite mocks out the heartbeat entirely, meaning the most dangerous code path is never actually exercised. These issues together represent a correctness and reliability risk unsuitable for a high-contention flash sale scenario.

---

### Detailed Findings

#### 🔴 Critical — Shared Heartbeat State (Concurrency Correctness)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment model — one service singleton), the second call's `_start_heartbeat` will overwrite `_heartbeat_active` and `_heartbeat_thread`. When the first call finishes and invokes `_stop_heartbeat`, it sets `_heartbeat_active = False`, which also kills the second call's heartbeat thread. The second lock can then expire while the DB operation is still in flight, defeating the entire purpose of the heartbeat.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance. Pass a mutable container (e.g., a `threading.Event` or a small dataclass) into the heartbeat thread and manage it locally within `_acquire_lock`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... retry logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... atomic release ...
```

#### 🔴 Critical — Non-Atomic Lock Release (TOCTOU Race)

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # Step 1: read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # Step 2: delete
```

Between step 1 and step 2, another process could acquire the lock (if it expired in the gap) and set a new token. You then delete a lock you no longer own. This is a well-known Redis locking hazard. The canonical fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

#### 🔴 Critical — Heartbeat Has the Same TOCTOU Problem

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)       # read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # extend
```

Same non-atomic check-then-act. Use a Lua script here too:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

#### 🟡 Major — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread could still be mid-sleep when the lock is deleted, wake up, and attempt to extend a lock that either no longer exists or has been acquired by another process. Always join the thread before releasing the lock:

```python
stop_event.set()
if heartbeat_thread.is_alive():
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

#### 🟡 Major — No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model wrapping the inputs. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a path-traversal-style string as `item_id`. At minimum:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(strip_whitespace=True, min_length=1, max_length=128)
    quantity: int = Field(default=1, ge=1, le=1000)
```

---

#### 🟡 Major — Tests Mock Out All Dangerous Code Paths

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

The heartbeat thread — the most novel and risky piece of this PR — is completely bypassed in every test except `test_lock_not_released_if_expired`, and even that test mocks the heartbeat. The tests give a false sense of coverage. Add at least one integration-style test that lets the real heartbeat run with a short TTL and verifies the lock is actually extended.

---

#### 🟠 Minor — `db.update_stock` and `db.create_order` Are Not Transactional

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record — inventory is silently lost. This should either be wrapped in a DB transaction or the order should be created first with a "pending" status before stock is decremented.

---

#### 🟠 Minor — `LOCK_TTL = 30` Is Undocumented as a Tuning Risk

A 30-second lock on a flash sale endpoint is extremely long. Under a thundering herd, all competing requests will be blocked for up to 30 seconds. This value should be documented as environment-configurable and the implications called out explicitly.
