Decision: Request Changes

Summary: The senior engineer's flag is the critical issue here — the heartbeat thread and the application-layer lock provide **no protection** against stale writes following a stop-the-world pause, OS preemption, or network partition. When the process resumes after a pause longer than `LOCK_TTL`, it has silently lost the lock but has no mechanism to detect this before executing `db.update_stock()` and `db.create_order()`. The DB write layer has zero fencing, meaning a second process that legitimately acquired the lock during the pause will have its invariants violated by the resumed stale writer. There are also secondary concerns around thread safety, instance-level heartbeat state, and missing input validation that must be addressed before this is production-ready.

---

## Detailed Findings

### Critical — Stale Write / Missing Fencing Token

This is the issue the senior engineer flagged, and the implementation does not solve it.

**The scenario:**

```
T=0   Process A acquires lock, token = "abc", TTL = 30s
T=5   Process A begins db.get_stock() — returns 10
T=6   OS pauses Process A (GC, scheduler preemption, VM snapshot, etc.)
      All threads suspended — including the heartbeat thread
T=36  Lock TTL expires (30s elapsed during pause)
T=37  Process B acquires lock, reads stock = 10, writes stock = 9, commits
T=38  Process A resumes from exactly where it was suspended
      _heartbeat_active is still True in its memory
      It proceeds to db.update_stock(item_id, 9)  ← stale write, no error
      It proceeds to db.create_order(...)          ← phantom order created
```

The heartbeat thread is suspended alongside every other thread during a stop-the-world pause. It cannot extend the lock it cannot run. The token check in `_acquire_lock`'s `finally` block only fires at lock *release*, not before each DB operation. By the time the stale writer checks the token, the damage is already done.

**The required fix — fencing tokens at the DB write layer:**

A fencing token is a monotonically increasing integer issued by the lock service at acquisition time. The DB layer must reject any write carrying a token older than the last committed token. Redis alone cannot provide this guarantee; it requires cooperation from the storage layer.

```python
# Conceptual sketch — the DB layer must enforce the fence

# At lock acquisition, obtain a monotonic fence from a reliable source
# Redis INCR on a dedicated counter works for single-instance Redis;
# a proper implementation uses Redlock or a consensus system (etcd, ZooKeeper)
fence_token = self.redis.incr("inventory:fence:counter")

# Pass the fence token through every DB operation
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
self.db.create_order(order_id, item_id, quantity, fence_token=fence_token)

# DB layer (pseudocode) — must be enforced atomically, e.g. in a stored procedure
# or via optimistic locking with a version column
def update_stock(self, item_id, new_stock, fence_token):
    rows_affected = execute("""
        UPDATE inventory
        SET stock = :new_stock, last_fence = :fence_token
        WHERE item_id = :item_id
          AND last_fence < :fence_token   -- reject stale writers
    """, ...)
    if rows_affected == 0:
        raise StaleWriteError("Fencing token rejected — lock was lost")
```

Without this, the distributed lock is providing a false sense of safety. It reduces the *probability* of a race but does not eliminate it.

### High — Instance-Level Heartbeat State Is Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object. If two concurrent calls to `reserve_item` arrive on the same instance (which is the normal case for a singleton service), they will race on these fields.

```python
# Two concurrent callers, same service instance:
# Caller A sets self._heartbeat_active = True, starts thread A
# Caller B sets self._heartbeat_active = True, starts thread B
# Caller A finishes, sets self._heartbeat_active = False
# Thread B is now silently stopped — Caller B's lock will expire
```

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass it through a closure or a small dataclass local to `_acquire_lock`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### High — No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used on the *output* model `ReservationResult`.

Minimum required:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(strip_whitespace=True, min_length=1, max_length=128,
                    pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(default=1, ge=1, le=1000)
```

Without this, `item_id` values like `"inventory:lock:other_item"` could be used to craft a lock key that collides with another item's lock key, or inject unexpected characters into Redis key space.

### Medium — `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is not joined
```

The heartbeat thread may still be executing a `redis.expire()` call when the lock is released and deleted. The sequence becomes:

```
finally block: _stop_heartbeat() called
finally block: redis.get() → token matches → redis.delete(lock_key)
heartbeat thread: redis.expire(lock_key, 30)  ← re-creates the key with no value owner
```

This is a real race. The thread must be joined before the lock release check.

---

### Medium — `db.update_stock` and `db.create_order` Are Not Atomic

These are two separate DB calls with no transaction boundary shown. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The caller receives an exception, not a `ReservationResult`, so the error handling in `reserve_item` will not catch it (it only catches `RuntimeError`).

```python
# Minimum fix — wrap in a transaction
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### Low — Test Suite Mocks Out the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat logic never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat thread — the mechanism the PR claims prevents lock expiry — is mocked away in every test except `test_lock_not_released_if_expired`. There are no integration-level tests that exercise the actual heartbeat renewal path, the thread safety concern above, or the join-before-release race.

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| Critical | Add fencing token enforcement at the DB write layer |
| High | Scope heartbeat state to the lock context, not the service instance |
| High | Add and enforce input validation on `item_id` and `quantity` |
| Medium | Join the heartbeat thread before releasing the lock |
| Medium | Wrap `update_stock` + `create_order` in a single DB transaction |
| Low | Add tests that exercise the heartbeat thread directly |
