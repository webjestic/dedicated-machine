Decision: Request Changes

Summary: The senior engineer's flag is a genuine, unresolved correctness flaw — not a theoretical edge case. The heartbeat thread is suspended during any stop-the-world pause alongside the main thread, meaning the lock can expire and be acquired by another process while the original process is mid-execution; when the original process resumes, it proceeds with `db.update_stock()` and `db.create_order()` with no awareness that it has lost lock ownership, producing a stale write and a potential double-reservation. The Redis distributed lock provides mutual exclusion only at the *acquisition* boundary, not at the *write* boundary, and the heartbeat does not close this gap. The DB write layer requires an optimistic concurrency check — a version column or CAS predicate — so that a write from a process that has lost its lock window is rejected by the database itself, not by application logic that may never re-execute.

## Detailed Review

### Critical Flaw: Stale Write on Process Pause (Blocking)

This is the core issue the senior engineer flagged, and the implementation does not address it.

**The failure sequence:**

```
Process A                          Process B
─────────────────────────────────────────────────────────
acquire lock (token=A)
read stock → 5
                                   [lock expires — A paused > LOCK_TTL]
                                   acquire lock (token=B)
                                   read stock → 5
                                   write stock → 4  ✓ legitimate
                                   release lock
[A resumes from pause]
write stock → 4  ✗ stale — A
  overwrites B's committed write
  with a value derived from a
  stale read
```

The heartbeat thread does not prevent this. Both the main thread and the heartbeat thread are OS-level threads within the same Python process. A stop-the-world GC pause, an OS scheduler preemption, or a VM live migration suspends the entire process — all threads simultaneously. The heartbeat cannot extend the lock because it is also suspended. This is not a Python-specific problem; it applies to any runtime with GC or any process subject to OS scheduling.

**What is required at the DB write layer:**

The write must be conditional on the data not having changed since it was read. Two standard mechanisms accomplish this:

**Option 1 — Optimistic locking with a version column:**

```python
# Schema addition
# ALTER TABLE inventory ADD COLUMN version INTEGER NOT NULL DEFAULT 0;

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"
    try:
        with self._acquire_lock(lock_key):
            row = self.db.get_stock_with_version(item_id)
            # returns {"stock": int, "version": int} or None

            if row is None:
                return ReservationResult(success=False, message=f"Item {item_id} not found")

            if row["stock"] < quantity:
                return ReservationResult(
                    success=False,
                    remaining_stock=row["stock"],
                    message="Insufficient stock"
                )

            new_stock = row["stock"] - quantity
            order_id = str(uuid.uuid4())

            # Write is conditional — rejected if version has advanced
            rows_affected = self.db.update_stock_if_version(
                item_id,
                new_stock,
                expected_version=row["version"]
            )

            if rows_affected == 0:
                # Another process wrote between our read and our write
                logger.warning(
                    "Optimistic lock conflict item_id=%s version=%d",
                    item_id, row["version"]
                )
                return ReservationResult(
                    success=False,
                    message="Concurrent modification detected. Please retry."
                )

            self.db.create_order(order_id, item_id, quantity)
            return ReservationResult(
                success=True,
                order_id=order_id,
                remaining_stock=new_stock,
                message="Reservation successful"
            )
    except RuntimeError as e:
        logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
        return ReservationResult(success=False, message="Service temporarily unavailable.")
```

```python
# DB layer — the predicate is what makes this safe
def update_stock_if_version(self, item_id: str, new_stock: int, expected_version: int) -> int:
    result = self.conn.execute(
        """
        UPDATE inventory
           SET stock   = :new_stock,
               version = version + 1
         WHERE item_id = :item_id
           AND version = :expected_version
        """,
        {"new_stock": new_stock, "item_id": item_id, "expected_version": expected_version}
    )
    return result.rowcount  # 0 means the predicate failed
```

The predicate `AND version = :expected_version` is the safety boundary. If Process A was paused and Process B has already incremented the version, Process A's write is rejected by the database engine atomically. No application-level re-check of the Redis lock is needed or sufficient.

**Option 2 — CAS (check-and-set) with a stock floor predicate:**

A simpler variant that does not require a version column, but is less general:

```sql
UPDATE inventory
   SET stock = :new_stock
 WHERE item_id = :item_id
   AND stock   = :expected_stock
   AND stock  >= :quantity
```

This is weaker than versioning because two writes that happen to restore the same stock value would not be caught, but for a strictly decrementing reservation workload it is functionally equivalent.

### Secondary Issues

**1. Heartbeat thread is not joined on stop**

`_stop_heartbeat` sets a flag but does not call `self._heartbeat_thread.join()`. The thread sleeps for `HEARTBEAT_INTERVAL` (10s) before checking the flag. Under high throughput, many threads can accumulate in the sleep phase before they observe the stop signal.

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread and self._heartbeat_thread.is_alive():
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

**2. Instance-level heartbeat state prevents concurrent use**

`_heartbeat_active` and `_heartbeat_thread` are instance attributes, not scoped to a lock acquisition. If `reserve_item` is ever called concurrently on the same `InventoryService` instance (e.g., from a thread pool), one call's `_stop_heartbeat` will cancel another call's heartbeat. The heartbeat state must be scoped per-acquisition, not per-instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition
    acquired = False
    ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        ...
```

**3. `db.update_stock` and `db.create_order` are not atomic**

If `update_stock` succeeds and `create_order` raises, stock is decremented with no order record. These two operations must be wrapped in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock_if_version(item_id, new_stock, expected_version=row["version"])
    self.db.create_order(order_id, item_id, quantity)
```

**4. Input validation is declared but not implemented**

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts `item_id: str` and `quantity: int` with no validation. `quantity=0` or `quantity=-5` will pass silently. A Pydantic model should gate the public method:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

**5. Test suite mocks out the mechanism under test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat is never exercised
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` at the fixture level means the lock lifecycle is never tested. `test_lock_not_released_if_expired` asserts the correct behavior but does so against a mock that bypasses the actual lock context manager path. At minimum, one integration-style test should exercise the real lock acquisition and release against a local Redis instance (or `fakeredis`).
