Decision: Request Changes

Summary: The senior engineer's flagged scenario is a real, unresolved vulnerability in this implementation — a process pause (GC stop-the-world, OS preemption, VM live migration) suspends all threads simultaneously, including the heartbeat, meaning the lock can expire and be acquired by another process while the original process is mid-execution. When the original process resumes, `db.update_stock()` proceeds without any awareness that lock ownership was lost, producing a stale write that silently corrupts inventory. The heartbeat pattern addresses slow-but-running operations, not suspended ones, and no mechanism at the DB write layer exists here to catch this case. A fencing token passed through to the DB write (e.g., a monotonically increasing lock generation counter stored alongside the lock, then validated as a precondition inside the DB transaction) is required to make this safe — the DB rejects any write carrying a stale generation number, regardless of what the application layer believes about lock state.

---

### Detailed Findings

#### Critical — Stale Write on Process Suspension (Flagged Scenario, Unresolved)

The reviewer note describes this precisely. The implementation does not resolve it. Here is the exact failure sequence:

```
Process A acquires lock (token=X, generation=1)
Process A is suspended by OS scheduler / GC for > LOCK_TTL seconds
  → All threads suspended, including heartbeat thread
  → Lock expires in Redis
Process B acquires lock (token=Y, generation=2)
Process B reads stock=10, writes stock=9 ✓
Process A resumes
Process A calls db.update_stock(item_id, new_stock)  ← stale write, no guard
  → stock is now 8, but one reservation was never made
  → inventory is silently corrupted
```

The token check in the `finally` block only guards the Redis `DELETE`. It does not guard the `db.update_stock()` call that already executed on the line above it. By the time the `finally` block runs, the damage is done.

**Required fix — fencing token at the DB write layer:**

```python
# Redis must return a monotonically increasing generation counter with the lock.
# Redlock / a INCR-based scheme can provide this.

acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
generation = self.redis.get(f"{lock_key}:gen")  # incremented atomically on each acquisition

# Pass the generation (fencing token) into every DB write as a precondition.
# The DB layer rejects the write if its stored generation != the presented one.

self.db.update_stock(item_id, new_stock, fence=generation)
self.db.create_order(order_id, item_id, quantity, fence=generation)
```

```sql
-- DB write layer enforces the fence
UPDATE inventory
SET stock = :new_stock, lock_generation = :fence
WHERE item_id = :item_id
  AND lock_generation < :fence   -- rejects stale writes atomically
```

Without this, the distributed lock provides ordering guarantees only under normal execution. It provides no safety guarantee under suspension.

#### Moderate — Heartbeat Race on Lock Expiry

Even during normal (non-suspended) execution, there is a TOCTOU window in the heartbeat:

```python
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the lock can expire and be re-acquired by another process. The heartbeat then extends a lock it no longer owns. This should be a single atomic Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

---

#### Moderate — Non-Atomic Release in `finally`

The same TOCTOU issue exists in lock release:

```python
current = self.redis.get(lock_key)           # lock could expire here
if current and current.decode() == token:
    self.redis.delete(lock_key)              # deletes a lock we may no longer own
```

This is a well-known pattern that must be replaced with the standard atomic Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

#### Moderate — Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item()` on the same `InventoryService` instance concurrently (plausible in a web worker), `_stop_heartbeat()` from one call will cancel the heartbeat for the other. Heartbeat state must be scoped to each lock acquisition, not to the service instance.

```python
# Pass a threading.Event into _start_heartbeat; store nothing on self
stop_event = threading.Event()
self._start_heartbeat(lock_key, token, stop_event)
# ...
finally:
    stop_event.set()
```

---

#### Low — No Input Validation on `item_id` or `quantity`

The PR checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string. The Pydantic model is only used for the return type. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

#### Low — Tests Mock Away the Components Under Scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most operationally sensitive component in this PR and it is mocked out in every test. `test_lock_not_released_if_expired` is named as though it tests the expiry scenario but it only tests the token mismatch branch in the `finally` block — it does not test the stale write scenario at all. Integration-level tests against a real Redis instance (e.g., `fakeredis`) should cover the actual lock lifecycle.
