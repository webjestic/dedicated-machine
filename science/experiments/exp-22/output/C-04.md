Decision: Request Changes

Summary: The core locking logic contains a critical race condition in the heartbeat and release path — the token ownership check (`GET` then `DEL`) is non-atomic, meaning another process could acquire the lock between the two operations and have it incorrectly deleted. Additionally, the `InventoryService` instance stores heartbeat state as instance-level attributes (`_heartbeat_active`, `_heartbeat_thread`), which makes the service unsafe for concurrent use across multiple threads or requests. The database operations (`update_stock` + `create_order`) are not wrapped in a transaction, creating a window for partial failure that could corrupt inventory state.

---

### Detailed Findings

#### 🔴 Critical — Non-Atomic Lock Release (Race Condition)

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)          # ← Step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← Step 2
```

Between `GET` and `DEL`, another process can acquire the lock. This service would then delete a lock it no longer owns, causing a **silent inventory corruption window** under flash sale load — exactly the scenario this PR is meant to protect against.

**Fix:** Use a Lua script to make the check-and-delete atomic.

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# Register once at __init__
self._release_lock = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_lock(keys=[lock_key], args=[token])
```

The same race applies in the heartbeat's `GET` + `EXPIRE` sequence and needs the same treatment.

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

#### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the **instance**, not scoped to a single `reserve_item` call. If two requests hit the same `InventoryService` instance concurrently (e.g., in a thread pool or async framework), one call's `_stop_heartbeat()` will cancel the other's heartbeat, potentially allowing a live lock to expire mid-operation.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            self._extend_lock(lock_key, token)  # atomic Lua script

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(keys=[lock_key], args=[token])
```

Note also that `_stop_heartbeat` sets a flag but does not `join()` the thread. The heartbeat thread can still fire one more `GET`/`EXPIRE` after the lock has been released, which is a correctness problem.

#### 🔴 Critical — No Database Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented with no corresponding order record. There is no rollback. Under flash sale volume this will silently lose inventory.

**Fix:** Wrap both operations in a single transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not expose transactions, that interface needs to be extended before this PR merges.

#### 🟡 Major — Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, excessively large quantities, or non-integer input. `ReservationResult` is a Pydantic output model, not input validation.

**Fix:** Add an explicit input model or inline guards.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

#### 🟡 Major — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat thread — the most operationally risky component — is never exercised. The non-atomic release race is also not testable with this fixture. Tests pass because the hard parts are mocked out.

**Fix:** Add integration-level tests using `fakeredis` to exercise the actual locking path, including concurrent reservation attempts.

```python
import fakeredis

def test_concurrent_reservations_do_not_oversell():
    r = fakeredis.FakeRedis()
    db = RealOrFakeDb(initial_stock=1)
    svc = InventoryService(r, db)

    results = []
    def attempt():
        results.append(svc.reserve_item("item_001", quantity=1))

    threads = [threading.Thread(target=attempt) for _ in range(10)]
    for t in threads: t.start()
    for t in threads: t.join()

    successes = [r for r in results if r.success]
    assert len(successes) == 1  # exactly one reservation, no oversell
```

#### 🟠 Minor — Heartbeat Interval Allows a Dangerous Expiry Window

With `LOCK_TTL = 30s` and `HEARTBEAT_INTERVAL = 10s`, the heartbeat sleeps first, then extends. If the process is paused (GC, I/O stall) for longer than 20 seconds after the last extension, the lock expires before the next heartbeat fires. The remaining TTL at worst case is `30 - 10 = 20s`, which may be acceptable, but this should be documented explicitly and the interval should be derived from the TTL rather than hardcoded independently.

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # always derived, never drifts
```
