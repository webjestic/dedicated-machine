Decision: Request Changes

Summary: The core locking logic has a critical race condition in the lock-release path — the GET and DELETE operations are not atomic, meaning another process could acquire the lock between the ownership check and the delete, causing a legitimate lock to be stolen. Additionally, the heartbeat implementation uses shared instance state (`_heartbeat_active`, `_heartbeat_thread`) rather than call-scoped state, making the service non-reentrant and unsafe under any concurrent usage pattern. Several secondary concerns around input validation, error handling, and test fidelity also need to be addressed before this is production-ready.

---

## Detailed Review

### 🔴 Critical — Non-Atomic Lock Release (Race Condition)

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)          # <-- another thread can acquire lock HERE
if current and current.decode() == token:
    self.redis.delete(lock_key)             # <-- we delete a lock we no longer own
```

This is a textbook TOCTOU (time-of-check/time-of-use) race. Between the `GET` and the `DELETE`, a concurrent process can acquire the lock after the original TTL expires. Your `DELETE` then destroys their valid lock.

The canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block, replace the GET+DELETE with:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same race exists in the heartbeat's `GET` + `expire` sequence and must be fixed there too:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 🔴 Critical — Heartbeat Uses Shared Instance State (Non-Reentrant)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — the second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa. Under a flash sale load, this is virtually guaranteed to happen.

Heartbeat state must be scoped to each lock acquisition, not to the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            released = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not released:
                logger.warning("Heartbeat: lock lost key=%s", lock_key)
                break

    # ... acquire lock ...

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script here
```

Note also that the current heartbeat uses `time.sleep()` which is not interruptible. Using `threading.Event.wait()` allows clean, prompt shutdown.

### 🟠 High — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation whatsoever. `item_id` could be an empty string, a path-traversal attempt, or a multi-megabyte payload. `quantity` could be zero or negative, which would silently *increase* stock.

```python
# This currently succeeds and increases stock by 1:
service.reserve_item("item_001", quantity=-1)
# new_stock = 10 - (-1) = 11
```

Add a Pydantic input model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id, req.quantity from here
```

### 🟠 High — Database Operations Not Atomic; No Rollback on Partial Failure

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The item is effectively lost. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The current `except RuntimeError` handler also only catches lock failures. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught to the caller.

---

### 🟡 Medium — Heartbeat Interval Unsafe Relative to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T=10s and extends to T=40s. That is fine in isolation, but the heartbeat uses `time.sleep()`, which can be delayed arbitrarily by the OS scheduler, GC pauses, or system load. Under pressure, the sleep could run long enough that the lock expires before the extension fires. A safer ratio is heartbeat at ≤ TTL/3, and the heartbeat should log a warning if it detects the lock has already expired when it fires.

### 🟡 Medium — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # <-- stop never runs
    return svc
```

Mocking out `_start_heartbeat` and `_stop_heartbeat` means the tests provide zero coverage of the heartbeat logic, the lock release path, or the interaction between them. The critical race condition identified above would never be caught by this suite. Tests should mock at the Redis boundary, not at internal methods:

```python
@pytest.fixture
def service(redis_client, db):
    # Do NOT mock internal methods — test them
    return InventoryService(redis_client, db)
```

Add explicit tests for:
- Negative and zero quantity inputs
- The Lua release script being called (not raw GET+DELETE)
- Heartbeat extending a lock it owns
- Heartbeat *not* extending a lock it no longer owns
- Partial DB failure triggering rollback

---

### 🟢 Minor — Logging Leaks Internal Item IDs Without Sanitization

```python
logger.info("Reserved item_id=%s quantity=%d order_id=%s remaining=%d", ...)
```

For a government contracting context, consider whether `item_id` values could contain sensitive identifiers. At minimum, document the assumption that `item_id` is safe to log, or run it through a sanitizer before emission.
