Decision: Request Changes

Summary: The implementation has a critical concurrency flaw — the `InventoryService` instance stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance-level attributes, meaning concurrent calls to `reserve_item` on the same service instance will race on those fields and corrupt each other's lock lifecycle. Beyond that, the lock release in `_acquire_lock` contains a TOCTOU race between `GET` and `DELETE`, and the test suite mocks out the heartbeat entirely, leaving the most dangerous code path completely untested.

---

## Detailed Findings

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None

# In _start_heartbeat:
self._heartbeat_active = True

# In _stop_heartbeat:
self._heartbeat_active = False
```

`InventoryService` is almost certainly instantiated once and shared across requests (e.g., via dependency injection in a Flask/FastAPI app). When two concurrent requests call `reserve_item` simultaneously, they share the same `_heartbeat_active` flag. Request A stopping its heartbeat will also kill Request B's heartbeat thread, potentially allowing B's lock to expire mid-operation — the exact failure mode this code was designed to prevent.

**Fix:** Heartbeat state must be scoped to the individual lock acquisition, not the instance. Pass a mutable container or use a local `threading.Event` per call.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ...
    heartbeat_thread = threading.Thread(
        target=self._beat, args=(lock_key, token, stop_event), daemon=True
    )
    heartbeat_thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # then release lock
```

### 🔴 Critical — TOCTOU Race in Lock Release

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # ← GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # ← DELETE (non-atomic)
```

Between `GET` and `DELETE`, another process could acquire the lock. This is a classic check-then-act race. The standard solution is a Lua script that atomically checks ownership and deletes in a single round-trip.

```lua
-- release.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def __init__(self, redis_client, db):
    # ...
    self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists inside the heartbeat's `_beat` function — a `GET` followed by a separate `EXPIRE`. That should also be a Lua script or replaced with `SET ... XX EX` semantics.

### 🟠 High — Input Validation Is Claimed But Not Implemented

The checklist says "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation whatsoever. There is no Pydantic model wrapping the inputs. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will either silently succeed or behave incorrectly.

```python
# quantity=0 passes the stock check and creates an order for nothing
# quantity=-1 would *increase* stock
if current_stock < quantity:  # -1 < 10 is False — stock gets incremented
```

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 🟠 High — Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — the lock is a concurrency primitive, not a transaction boundary. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10s). After `_stop_heartbeat()` is called, the thread may still be alive for up to 10 seconds. In the current design, the lock is deleted immediately after, and then the thread wakes up, checks the token (which no longer exists in Redis), and silently exits. This is benign today but fragile — it relies on the token-check guard as an implicit cleanup mechanism rather than explicit lifecycle management. Under the proposed fix above, `stop_event.set()` + `join()` makes this deterministic.

### 🟡 Medium — Tests Mock Out the Exact Code Under Scrutiny

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

Every test that uses the `service` fixture bypasses the heartbeat entirely. The heartbeat thread is the most operationally risky component in this PR, and it has zero test coverage. At minimum, add an integration-style test that lets the heartbeat run and verifies TTL extension behavior.

---

### 🟡 Medium — `retry_attempts` Hardcoded, Not Configurable at Construction

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

The default is buried in a private method signature. Operational teams tuning this service for flash sale load will have no obvious place to configure retry behavior. This should be a constructor parameter.

---

### 🔵 Low — `RuntimeError` Is Too Broad for Lock Failure

Catching `RuntimeError` is a wide net. Define a specific exception class so callers can distinguish lock failures from programming errors.

```python
class LockAcquisitionError(Exception):
    pass
```

## Summary of Required Changes Before Approval

| Severity | Issue |
|----------|-------|
| 🔴 Critical | Heartbeat state is instance-scoped — breaks under concurrency |

| 🟠 High | No DB transaction — partial failure leaves data inconsistent |
| 🟡 Medium | Heartbeat thread never joined — implicit lifecycle dependency |
| 🟡 Medium | Heartbeat entirely mocked in tests — core risk path untested |
