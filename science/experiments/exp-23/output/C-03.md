Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `nx=True`, ownership-verified release, heartbeat renewal, and exponential backoff are all sound patterns. However, there are several correctness, safety, and architectural issues that must be addressed before this touches production inventory, particularly given the high-stakes nature of flash sale concurrency. The most critical flaw is a race condition in the heartbeat's lock extension logic and a shared-state concurrency bug that makes this service unsafe for concurrent use.

---

## Critical Issues

### 1. Heartbeat Race Condition (Check-Then-Act)

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

The `GET` + `EXPIRE` sequence is not atomic. The lock can expire (and be acquired by another process) between the `GET` and the `EXPIRE`, causing the heartbeat to silently extend a lock it no longer owns. This is the exact scenario the ownership check is meant to prevent.

**Fix:** Use a Lua script to make this atomic:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat failed to extend lock — lock lost key=%s", lock_key)
    self._heartbeat_active = False
```

The same atomic Lua pattern should also be applied to the lock release in `_acquire_lock`.

### 2. Shared Instance State Makes This Unsafe for Concurrent Use

```python
# These are instance-level attributes
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `InventoryService` is used as a singleton (the standard pattern in a web service), two concurrent calls to `reserve_item` will clobber each other's `_heartbeat_active` and `_heartbeat_thread`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat thread.

**Fix:** Move heartbeat state into the lock context itself, scoped per-call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

---

### 3. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is never joined
```

Setting the flag does not guarantee the thread has stopped before the lock is released. There is a window where the heartbeat thread wakes up, passes the `if self._heartbeat_active` check on a stale read, and extends the TTL of a lock that the main thread has already deleted — potentially extending a lock now owned by a different process.

**Fix:** Join the thread with a bounded timeout in the `finally` block before releasing the lock.

## Moderate Issues

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used on the *output* model. An empty string, negative quantity, or excessively long `item_id` will pass through unchecked and construct a Redis key directly.

```python
# item_id="" produces lock key "inventory:lock:" — ambiguous
# quantity=-5 would increase stock silently
```

**Fix:** Add an input model or explicit guards:

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

### 5. DB Operations Are Not Atomic Relative to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record — an inventory leak. The Redis lock protects against concurrent access but does not provide transactional rollback.

**Fix:** Wrap both DB calls in a database transaction, or use a compensating action in an exception handler:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB transaction failed, reservation rolled back item_id=%s", item_id)
    raise
```

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed ...")
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

Any `RuntimeError` from inside the lock body (e.g., a bug in `db.update_stock`) will be silently swallowed and returned as "service unavailable," masking real errors. Introduce a dedicated exception class:

```python
class LockAcquisitionError(Exception):
    pass

# Raise this specifically in _acquire_lock
# Catch only this in reserve_item
```

## Minor Issues

### 7. Test Coverage Gaps

- No test verifies the heartbeat actually extends the TTL (the mock stubs it out entirely).
- No test covers the partial-failure case where `update_stock` succeeds but `create_order` raises.
- `test_lock_not_released_if_expired` constructs a new `InventoryService` but then stubs `_start_heartbeat` and `_stop_heartbeat` on it — this test does not actually exercise the real heartbeat path and provides false confidence.

### 8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Should Be Configurable

Hardcoded class-level constants make tuning for different deployment environments (staging vs. production under load) require a code change. Pass these through `__init__` with sensible defaults.
