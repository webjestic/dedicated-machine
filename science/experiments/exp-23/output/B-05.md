Decision: Request Changes

Summary: This implementation contains a classic **Redlock / fencing token vulnerability** — specifically, the heartbeat renewal pattern creates a false sense of safety against the exact failure mode it claims to prevent. A stop-the-world GC pause (or any process suspension: VM migration, OS scheduling stall, debugger attach) can suspend *all* threads simultaneously, including the heartbeat thread, allowing the lock TTL to expire and be acquired by another process; when the original process resumes, it proceeds to execute `db.update_stock()` and `db.create_order()` with no re-validation of lock ownership at the database write layer, producing a double-reservation or inventory corruption. The test suite mocks out the heartbeat entirely and does not exercise this race condition at all.

---

## Detailed Findings

### Critical — Fencing Token Not Enforced at Write Layer

This is the core defect. The heartbeat pattern is architecturally insufficient as the sole guard against lock expiry.

```python
# The problem: lock ownership is only checked at Redis level,
# never at the point of the actual writes.

self.db.update_stock(item_id, new_stock)   # ← no fencing here
self.db.create_order(order_id, item_id, quantity)  # ← or here
```

**The failure sequence:**

```
Process A                          Redis                  Process B
─────────────────────────────────────────────────────────────────────
Acquires lock (token=A)
Reads stock = 10
[STW GC pause — all threads
 suspended, incl. heartbeat]
                                   TTL expires
                                                          Acquires lock (token=B)
                                                          Reads stock = 10
                                                          Writes stock = 9  ✓
[GC resumes]
Heartbeat wakes, checks token:
  Redis.get() → token B ≠ A
  Does NOT extend. Correct.
BUT: execution continues past
the `with` block yield point.
Writes stock = 9  ✗ (duplicate)
```

Wait — actually it's worse than that. The `yield` is *inside* the `with` block. The writes happen *before* the `finally` clause checks the token. The token check only governs lock *release*, not write *authorization*. Process A will complete both writes before it ever discovers the lock expired.

The correct mitigation is a **monotonic fencing token** (e.g., a Redis `INCR`-based counter) passed through to the database layer, which rejects writes carrying a stale token:

```python
# Sketch of correct approach
def update_stock(self, item_id: str, new_stock: int, fence: int) -> bool:
    # DB-side: UPDATE inventory SET stock = %s
    #          WHERE item_id = %s AND lock_fence >= %s
    #          -- or use optimistic locking / CAS at DB level
    ...
```

Without enforcement at the storage layer, the distributed lock is providing *performance isolation* (reducing contention), not *correctness guarantees*.

### High — Heartbeat Race on Token Check

The heartbeat's ownership check is non-atomic:

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)       # ← read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. This should be a single atomic Lua script:

```lua
-- atomic heartbeat renewal
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
RENEW_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    renew = self.redis.register_script(RENEW_SCRIPT)
    self._heartbeat_active = True

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                renewed = renew(keys=[lock_key], args=[token, self.LOCK_TTL])
                if not renewed:
                    logger.error("Heartbeat lost lock ownership key=%s — aborting", lock_key)
                    self._heartbeat_active = False
                    # signal main thread to abort
```

Note also: if the heartbeat detects it has lost the lock, it currently has no mechanism to signal the main thread to abort the operation. This needs an `threading.Event` or similar abort flag checked between DB operations.

### High — Lock Release Is Also Non-Atomic

Same TOCTOU issue in the `finally` block:

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # ← read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # ← write (non-atomic)
```

The canonical atomic release uses the same Lua pattern:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is well-documented in the Redis distributed locks page and should be considered table-stakes for any Redis locking implementation.

### Medium — Instance-Level Heartbeat State Prevents Concurrency

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables. A single `InventoryService` instance cannot safely serve concurrent calls to `reserve_item` — the second caller will stomp `_heartbeat_active` and `_heartbeat_thread`, silently corrupting the first caller's lock management. In a thread-pool or async web server context this is a latent bug.

Fix: scope heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass event into _beat closure, not instance state
    ...
```

### Medium — No Input Validation on Public Method

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # item_id could be empty string, whitespace, SQL injection payload
    # quantity could be 0, negative, or absurdly large
```

Pydantic is imported and `ReservationResult` is modeled, but it's only used for *output*. Add an input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### Low — Test Suite Mocks Out the Mechanism Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

The tests pass because they mock away the exact component with the defects. `test_lock_not_released_if_expired` is named as if it tests expiry behavior, but it only tests the token-mismatch branch of the `finally` clause — it does not test that writes are blocked after expiry, because they aren't blocked. This gives false confidence. Integration tests against a real Redis instance with injected pauses are needed to meaningfully validate this code path.

## Summary of Required Changes Before Approval

| Severity | Issue | Required Fix |
|---|---|---|
| Critical | No fencing at write layer | Enforce monotonic fence token at DB write operations |
| High | Non-atomic heartbeat renewal | Replace GET+EXPIRE with Lua script |
| High | Non-atomic lock release | Replace GET+DEL with Lua script |
| High | No abort signal from heartbeat to main thread | Add `threading.Event` abort mechanism |
| Medium | Instance-level heartbeat state | Scope state to lock context, not instance |
| Medium | No input validation on `reserve_item` | Add Pydantic input model |
| Low | Tests mock the component under test | Add integration tests against real Redis |
