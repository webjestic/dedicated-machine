Decision: Request Changes

Summary: The implementation demonstrates solid intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, concurrency safety, and resilience issues that must be addressed before this is suitable for a production flash sale workload. The test suite also has meaningful gaps that mask real failure modes.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrent Usage

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the **same `InventoryService` instance** concurrently (which is the entire point of this service), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, causing the first heartbeat to be orphaned or prematurely stopped. The lock boundary is meaningless if the heartbeat protecting it can be killed by a sibling request.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...

def _start_heartbeat(
    self, lock_key: str, token: str, active: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not active.is_set():
                break
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 2. Non-Atomic Lock Release (TOCTOU Race)

```python
# Current — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock. You would then delete a lock you no longer own. This is the canonical mistake in Redis locking — the Redis documentation explicitly requires a Lua script for atomic release.

**Fix:**

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def __init__(self, redis_client: redis.Redis, db):
    self.redis = redis_client
    self.db = db
    self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU issue exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a similar Lua script.

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation whatsoever. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will happily acquire a distributed lock and attempt a DB write.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    
    lock_key = f"inventory:lock:{req.item_id}"
    # ... proceed with req.item_id and req.quantity
```

Without the pattern constraint on `item_id`, a caller could pass `item_id = "../../admin"` or a value that manipulates the Redis key namespace.

### 4. Database Operations Have No Atomicity Guarantee

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record — a silent inventory leak. The distributed lock protects against concurrent access but does nothing to protect against partial writes.

**Fix:** These two operations must execute within a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 5. Bare `Exception` Not Caught — Redis Failures Crash the Caller

Only `RuntimeError` (lock contention) is caught. Any `redis.RedisError`, `ConnectionError`, or database exception propagates as an unhandled exception to the caller. For a flash sale endpoint, this likely means a 500 with no graceful degradation.

```python
except RuntimeError as e:
    # Only this is caught
```

**Fix:**

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except redis.RedisError as e:
    logger.error("Redis error during reservation item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    raise  # Re-raise truly unexpected errors after logging
```

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread may still be mid-execution (e.g., in the middle of a `redis.expire` call) when the lock is released. With the fix in Issue #1, `join()` with a timeout resolves this.

---

### 7. `time.sleep` in Retry Loop Blocks the Thread

```python
time.sleep(backoff)
```

For a high-concurrency flash sale service, blocking a thread for up to `(2^2) * 0.1 = 0.4s` per retry is acceptable, but the maximum total wait time of `~0.7s` across three attempts with no jitter will cause thundering herd behavior when hundreds of requests are released simultaneously.

**Fix:** Add jitter.

```python
import random
jitter = random.uniform(0, 0.05)
time.sleep(backoff + jitter)
```

## Test Issues

### 8. Heartbeat Is Mocked Away — Core Concurrency Logic Is Untested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test mocks out the heartbeat. This means the actual threading behavior, the `_heartbeat_active` flag lifecycle, and the `GET`/`EXPIRE` sequence are completely untested. At minimum, one integration-style test should exercise the real heartbeat path with a mock Redis client that records calls.

### 9. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test asserts that `delete` is not called when the token doesn't match — which is correct behavior — but it does so while the heartbeat is mocked, meaning it's testing the `finally` block in isolation from the actual lock lifecycle. It also doesn't verify that a warning is logged when the lock is found to be stolen, which is an important operational signal.

### 10. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the inventory leak scenario described in Issue #4, and it must be tested explicitly once the transaction fix is in place.

---

## Minor Notes

- `LOCK_TTL = 30` is generous. Consider making it configurable via constructor parameter so it can be tuned per deployment environment without a code change.
- The `db` interface is entirely implicit. A Protocol or ABC defining `get_stock`, `update_stock`, and `create_order` would make the contract explicit and catch integration mismatches earlier.
- Consider whether `ReservationResult.message` should be an enum or structured type rather than a free-form string, since callers may be branching on its content.
