Decision: Request Changes

Summary: This implementation contains a classic **Redlock / fencing token vulnerability** — specifically, the heartbeat renewal pattern creates a false sense of safety against the exact failure mode it claims to prevent. A stop-the-world GC pause (or any process suspension: VM live migration, OS scheduling starvation, debugger attach) can suspend *all* threads simultaneously, including the heartbeat thread, allowing the lock TTL to expire mid-operation. When the process resumes, `db.update_stock()` and `db.create_order()` execute against a lock the process no longer owns, with no re-validation at the write layer. A second process may have acquired the lock and already written during the gap, meaning two processes execute the critical section concurrently — the exact scenario the lock is meant to prevent. The heartbeat thread does not and cannot solve this; it is suspended along with everything else.

---

## Detailed Findings

### Critical — Race Condition via Process Pause (TOCTOU at Write Layer)

This is the core flaw. The sequence of events that breaks the invariant:

```
Process A                          Process B
─────────────────────────────────────────────────────────────
acquire lock (token=A)
read stock → 5
                                   [waiting on lock]
[STW GC pause — all threads
 suspended, including heartbeat]
                                   lock TTL expires
                                   acquire lock (token=B)
                                   read stock → 5
                                   write stock → 4  ✓ legitimate
                                   create order     ✓ legitimate
[GC resumes]
write stock → 4  ← DUPLICATE WRITE, no re-validation
create order     ← DUPLICATE ORDER, oversell
```

The heartbeat thread is a daemon thread running inside the same process. It cannot extend the lock if the process is paused. The check in `_beat()` — `if current and current.decode() == token` — only runs *after* the pause ends, at which point the damage is done. The writes in `reserve_item` happen with no fencing check at the DB layer.

```python
# This executes after resumption with zero guarantee the lock is still held:
self.db.update_stock(item_id, new_stock)   # ← no re-validation
self.db.create_order(order_id, item_id, quantity)  # ← no re-validation
```

---

### Critical — Shared Heartbeat State Breaks Concurrent Usage

The heartbeat state is stored on the instance (`self._heartbeat_active`, `self._heartbeat_thread`). If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the expected use case in any web framework — concurrent calls will stomp each other's heartbeat state. One call's `_stop_heartbeat()` will kill another call's renewal thread.

```python
# Instance-level state, not call-scoped:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to the individual lock acquisition, not the service instance.

---

### High — Heartbeat Check is Non-Atomic (TOCTOU in Renewal)

The `GET` + `EXPIRE` in the heartbeat is a two-operation sequence with no atomicity guarantee:

```python
current = self.redis.get(lock_key)          # step 1
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2 — lock may have changed between 1 and 2
```

Between `GET` and `EXPIRE`, another process could acquire the lock (if TTL expired in the gap). The `EXPIRE` then extends a lock owned by a different process. This must be a single atomic Lua script:

```lua
-- atomic_extend.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

---

### High — Lock Release is Non-Atomic

The same issue applies to release:

```python
current = self.redis.get(lock_key)           # step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)              # step 2 — not atomic
```

This is a well-known pattern that must use a Lua script. This is documented in the Redis `SET NX` documentation itself:

```lua
-- atomic_release.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

---

### Medium — No Input Validation on `item_id` or `quantity`

The PR claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model gating the public method. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a path-traversal-style string as `item_id`. The lock key becomes `inventory:lock:` with an attacker-controlled suffix.

```python
# quantity <= 0 should be rejected before acquiring a distributed lock
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

---

### Medium — No Fencing Token Propagated to DB Layer

Even if the process-pause issue were mitigated at the Redis layer, the database layer has no way to reject a stale write. The correct architecture for this class of problem is a monotonically increasing fencing token passed through to the storage layer, which rejects writes carrying a token lower than the last accepted one. This is described in detail in Martin Kleppmann's analysis of distributed locks. Without this, Redis-layer correctness cannot provide end-to-end safety guarantees.

---

### Low — `quantity` Parameter Has No Default Validation

`quantity=1` as a default is fine, but the parameter is not validated to be a positive integer before use. This should be enforced at the entry point.

---

### Low — Test Suite Mocks Away the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the primary novel mechanism in this PR. Mocking it out means the tests provide no coverage of the mechanism's correctness, thread safety, or interaction with the lock lifecycle. The test for `test_lock_not_released_if_expired` is the right instinct but does not test the process-pause scenario at all.

---

## Required Changes Before Approval

1. **Introduce a fencing token** propagated to `db.update_stock()` and `db.create_order()`, with the DB layer rejecting writes on token mismatch. This is the only reliable mitigation for process-pause scenarios.
2. **Make heartbeat state call-scoped**, not instance-scoped. Pass lock state into the heartbeat closure explicitly.
3. **Replace GET+EXPIRE and GET+DEL with atomic Lua scripts.**
4. **Add input validation** (`item_id` non-empty, `quantity >= 1`) before lock acquisition.
5. **Add integration-level tests** for the heartbeat and lock expiry paths that do not mock the mechanism under test.
