Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical correctness flaw that is directly relevant to my area of concern: a stop-the-world GC pause (or any OS-level suspension — SIGSTOP, VM live migration, container freeze) can suspend all threads simultaneously, including the heartbeat thread, allowing the Redis TTL to expire and the lock to be acquired by another process. When the original process resumes, it believes it still holds the lock and proceeds to write — producing a split-brain window where two processes are executing the critical section concurrently. Beyond that core issue, there are several additional architectural and correctness problems that need to be addressed before this is safe to merge.

---

## Detailed Findings

### 🔴 Critical — Heartbeat Does Not Protect Against Process Suspension

This is the fundamental flaw in the design premise.

```python
# The PR claims this prevents lock expiry "under load."
# It does not protect against:
#   - Stop-the-world GC pauses (CPython GIL notwithstanding, this applies to
#     PyPy, Jython, or any future runtime)
#   - SIGSTOP / SIGCONT sent to the process
#   - VM live migration or hypervisor-level pause
#   - Container runtime freeze (e.g., docker pause, cgroup freezer)
#   - OS scheduler starvation under extreme load
#   - Network partition between this host and Redis

def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # <-- entire process can be suspended here
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # <-- lock may already be gone
```

If the process is suspended for longer than `LOCK_TTL` (30 seconds), the lock expires. Another process acquires it. The original process resumes, the `get` returns the **new owner's token**, the `if` check correctly fails to renew — but the original process is already **past the lock acquisition point** and continues executing `db.update_stock` and `db.create_order` concurrently with the new lock holder. The token check on renewal does not protect the critical section that is already in flight.

The correct mental model: **the lock does not know the process is still alive; only the TTL does.** Heartbeats reduce the probability window but cannot eliminate it, and the code contains no defense for the resumption-after-expiry case.

**Required remediation:**

```python
# Option A: Fencing tokens (preferred for correctness)
# Redis WAIT or a monotonically increasing token (e.g., Redlock with fencing)
# passed through to the DB layer, which rejects writes with stale fence values.

# Option B: Post-lock ownership re-validation before any write
def reserve_item(self, item_id, quantity=1):
    with self._acquire_lock(lock_key) as token:
        # ... read stock ...

        # Re-validate ownership immediately before writes
        current = self.redis.get(lock_key)
        if not current or current.decode() != token:
            return ReservationResult(success=False, message="Lock expired during operation")

        self.db.update_stock(item_id, new_stock)   # still a TOCTOU gap, but narrowed
        self.db.create_order(order_id, item_id, quantity)
```

Note that even Option B is not fully safe — it narrows the window but does not eliminate it. A fencing token propagated to the database is the only correct solution for strong safety guarantees.

### 🔴 Critical — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared instance state
        self._heartbeat_thread = None           # shared instance state
```

If a single `InventoryService` instance handles concurrent requests (e.g., in a thread pool or async framework), two simultaneous calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will set `_heartbeat_active = False`, killing the first call's heartbeat. Both calls then proceed with an unprotected lock.

**Required remediation:** Heartbeat state must be scoped to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key, retry_attempts=3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into _start_heartbeat, never touch self._heartbeat_active
```

### 🔴 Critical — Non-Atomic Check-Then-Act on Lock Release

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # DEL  <-- not atomic with GET
```

Between `GET` and `DEL`, the lock can expire and be acquired by another process. The `DEL` then deletes the new owner's lock. This is a classic TOCTOU race. The standard fix has been known for years.

**Required remediation:** Use the canonical Lua script for atomic compare-and-delete.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

### 🟡 Major — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is killed, or raises between these two calls, stock is decremented but no order exists — inventory is silently lost. The distributed lock protects against concurrent access but does nothing for partial failure within the critical section.

**Required remediation:** These two operations must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Major — Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but there is no validation on `item_id` or `quantity` at the service boundary.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # No validation. quantity=-1 would increment stock.
    # item_id could be empty string, path traversal attempt, etc.
    lock_key = f"inventory:lock:{item_id}"   # unsanitized item_id injected into key
```

A negative `quantity` passes the `current_stock < quantity` check (e.g., `10 < -1` is `False`) and calls `db.update_stock(item_id, 11)` — silently inflating inventory. An empty or malformed `item_id` produces a malformed Redis key.

**Required remediation:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 🟡 Major — Tests Mock Away the Exact Behavior Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat is never tested
    svc._stop_heartbeat = MagicMock()    # <-- stop is never tested
    return svc
```

The heartbeat thread — the primary safety mechanism being introduced — is mocked out in every test except `test_lock_not_released_if_expired`, which also mocks it. There are no tests for the concurrent instance-state race, no tests for the non-atomic release, and no tests for negative quantity. The test suite provides false confidence.

### 🟠 Minor — Heartbeat Interval Is More Than Half the TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

`HEARTBEAT_INTERVAL` should be well under `LOCK_TTL / 2` to account for Redis round-trip latency, clock skew, and scheduling jitter. At 10 seconds with a 30-second TTL this is acceptable, but it should be documented as a deliberate ratio, not an arbitrary constant. Consider deriving it: `HEARTBEAT_INTERVAL = LOCK_TTL // 3`.

---

### 🟠 Minor — `RuntimeError` Is Too Broad for Lock Failure

```python
raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")
```

This conflates lock contention (an expected, recoverable condition) with programming errors. Define a domain exception.

```python
class LockAcquisitionError(Exception): ...
```
