Decision: Request Changes

Summary: The core locking logic and heartbeat concept are sound, but there are several architectural and correctness issues that must be addressed before this can be approved. Most critically, the heartbeat and lock release contain race conditions that could cause split-brain inventory corruption, and the service holds instance-level state that makes it unsafe for concurrent use. These are not cosmetic issues — in a high-concurrency flash sale scenario, they represent real paths to overselling.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton (injected once, shared across requests). Two concurrent calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, potentially allowing the first lock to expire mid-operation — the exact failure mode this code is designed to prevent.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through a local object or closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

### 2. Lock Release Is Not Atomic — TOCTOU Race Condition

```python
# Current — check then act: not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process. You would then delete a lock you do not own, causing inventory corruption. This is a well-known Redis distributed lock pitfall.

**Fix:** Use a Lua script to make the check-and-delete atomic. This is the standard pattern per the Redis documentation.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. Database Operations Are Not Atomic — No Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The item is effectively lost. The Redis lock does not protect against application-layer failures between these two calls.

**Fix:** These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is a gap in the interface contract that must be resolved before this service ships.

## Moderate Issues

### 4. Input Validation Is Claimed But Not Implemented

The checklist says "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will behave incorrectly or silently.

```python
# quantity=0 passes the stock check and creates a zero-quantity order
if current_stock < quantity:  # 10 < 0 is False — proceeds
```

**Fix:** Validate at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

---

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined
```

The thread sleeps for `HEARTBEAT_INTERVAL` (10s) before checking the flag. Under load, you will accumulate many threads in a sleep-wait state. In a flash sale scenario with thousands of requests, this is a thread leak.

**Fix:** Use `threading.Event` (shown above in fix #1) and call `thread.join()` in the `finally` block.

---

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

`RuntimeError` will also catch unexpected errors from `db.get_stock`, `db.update_stock`, and `db.create_order` if they raise it, silently swallowing real bugs as "service unavailable." Define a specific exception class.

```python
class LockAcquisitionError(Exception):
    pass

# Raise it explicitly in _acquire_lock, catch only it in reserve_item
```

## Test Coverage Gaps

### 7. Heartbeat and Atomic Release Are Not Tested

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, meaning the heartbeat logic has zero test coverage. The TOCTOU race in lock release is also not tested. These are the two highest-risk code paths.

### 8. `test_lock_not_released_if_expired` Tests the Wrong Behavior

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts the current (flawed) non-atomic behavior. Once the Lua script fix is applied, this test needs to be rewritten to verify the script is called with the correct arguments, not that `delete` is skipped.

### 9. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the data-loss scenario described in issue #3.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but document the reasoning explicitly: the heartbeat fires at `TTL / 3` to give two renewal windows before expiry.
- Consider whether `retry_attempts=3` with backoffs of `0.1s`, `0.2s`, `0.4s` is appropriate for your SLA. Under heavy flash sale load, callers may queue up and the total wait time should be bounded and documented.
- The `db` parameter has no interface contract (ABC or Protocol). Define one so implementors know what `get_stock`, `update_stock`, and `create_order` are expected to guarantee, especially around exceptions and transaction support.
