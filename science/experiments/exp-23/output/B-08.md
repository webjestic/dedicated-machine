Decision: Request Changes

Summary: This implementation contains a classic **Redlock/fencing token vulnerability** — specifically, the TOCTOU race between the heartbeat's ownership check and the `expire` call is not the primary concern; the deeper flaw is that the database write layer (`db.update_stock`, `db.create_order`) performs **no fencing validation whatsoever**, meaning a process that resumes after a stop-the-world GC pause (or any other suspension) can execute destructive writes against stale lock ownership with zero protection at the storage layer. The heartbeat mitigates *routine* expiry but provides no safety guarantee against arbitrary process pauses that exceed `LOCK_TTL`, and the architecture places unconditional trust in a Redis key check that may have already been invalidated by the time the DB calls execute. Additionally, the service has a structural concurrency defect where `_heartbeat_active` and `_heartbeat_thread` are instance-level fields rather than call-scoped, making the service non-reentrant and unsafe under concurrent use.

---

### Detailed Findings

#### 1. Missing Fencing at the Write Layer (Critical)

This is the core architectural flaw. The sequence inside `reserve_item` is:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # read
    ...
    self.db.update_stock(item_id, new_stock)      # write — no fence
    self.db.create_order(order_id, ...)           # write — no fence
```

Consider this scenario:

1. Process A acquires the lock, token = `abc-123`, reads stock = 1
2. Process A enters a stop-the-world GC pause for 35 seconds (exceeds `LOCK_TTL = 30`)
3. The heartbeat thread is also paused — it cannot renew
4. Redis expires the lock
5. Process B acquires the lock, token = `xyz-789`, reads stock = 1, writes stock = 0, creates order
6. Process A resumes, its in-memory state still says stock = 1, it calls `db.update_stock(item_id, 0)` and `db.create_order(...)` — **double-spend, inventory goes negative, two orders created for the last unit**

The lock check in `_acquire_lock`'s `finally` block only guards the Redis `delete` call, not the DB writes. By the time the writes execute, the process has no awareness that its lock tenure has lapsed.

The correct mitigation is a **monotonically increasing fencing token** (e.g., Redis `INCR` on lock acquisition, or a Lua-script-enforced version counter) passed through to the database layer, which rejects writes carrying a stale token. Without storage-layer enforcement, the distributed lock is not a safety mechanism — it is only a performance optimization.

```python
# Sketch of what's needed
fence_token = self.redis.incr(f"inventory:fence:{item_id}")

# ... later, at the DB write:
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
# DB layer: UPDATE inventory SET stock=? WHERE item_id=? AND fence_token < ?
```

#### 2. Non-Reentrant Service Due to Instance-Level Heartbeat State (High)

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on `self`. If two threads call `reserve_item` on the same `InventoryService` instance concurrently:

- Thread A sets `_heartbeat_active = True`, starts thread
- Thread B sets `_heartbeat_active = True`, starts a second thread, clobbering `_heartbeat_thread`
- Thread A's `_stop_heartbeat()` sets `_heartbeat_active = False`, killing **both** heartbeats
- Thread B's lock now goes unrenewed

Heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass state through a local object or closure:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller holds the stop handle
```

#### 3. Heartbeat Has a Non-Atomic Check-Then-Act Race (Medium)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, another process could acquire the lock. This should be a single atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

---

#### 4. Lock Release Also Non-Atomic (Medium)

The same TOCTOU issue exists in the `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a well-known pattern that must be replaced with the standard atomic Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

#### 5. Input Validation Not Enforced at the Service Boundary (Low)

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is only used on the *output* model `ReservationResult`. There is no guard against `quantity <= 0`, empty `item_id`, or pathologically large quantities. A request model should gate the method:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

---

#### 6. Test Suite Mocks Away the Behavior Under Test (Low)

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture stubs out the heartbeat. This means the tests provide zero coverage of the heartbeat's correctness, the non-reentrancy defect, or any timing-related behavior. At minimum, integration-level tests using a real or embedded Redis (e.g., `fakeredis`) should validate lock lifecycle behavior.
