Decision: Request Changes

Summary: This implementation contains a classic **Redlock/fencing token vulnerability** — specifically, the heartbeat approach does not protect against the scenario where the Python process experiences a stop-the-world GC pause (or equivalent OS-level suspension) long enough for the lock TTL to expire, after which another process acquires the lock, and then *this* process resumes and proceeds to execute `db.update_stock()` and `db.create_order()` without re-validating lock ownership at the database write layer. The heartbeat thread is also suspended during a STW pause, so it cannot renew the lock during the window that matters most. Additionally, there is a non-atomic TOCTOU race in the heartbeat itself (`GET` then `EXPIRE` are two separate commands, not a single atomic operation), and the service holds shared mutable heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance variables, making the class unsafe for concurrent use across multiple threads calling `reserve_item` simultaneously.

---

### Detailed Findings

#### 1. Missing Fencing Token / Write-Layer Validation (Critical)

This is the core flaw. The sequence of events that breaks correctness:

```
Process A acquires lock, token = "abc", TTL = 30s
Process A enters STW GC pause for 35 seconds
  └─ Heartbeat thread is also paused (it's in-process)
  └─ Lock expires in Redis
Process B acquires lock, token = "xyz"
Process B reads stock=10, writes stock=9, creates order
Process A resumes from GC pause
  └─ _heartbeat_active is still True from A's perspective
  └─ A calls db.update_stock(item_id, new_stock)   ← EXECUTES WITHOUT LOCK
  └─ A calls db.create_order(...)                  ← DUPLICATE ORDER CREATED
  └─ Stock is now corrupted (double-decremented)
```

The lock boundary in `reserve_item` only gates entry. Once inside the `with` block, there is no re-validation before the writes. The heartbeat cannot solve this because the heartbeat thread lives in the same process and is subject to the same pause.

The correct mitigation is a **fencing token** — a monotonically increasing value (e.g., from a Redis counter or Lua-script-issued sequence number) that is passed through to the database write operations, where the DB layer rejects any write carrying a token older than the last committed token. Without this, the distributed lock provides only a probabilistic, not a safety guarantee.

```python
# What is needed at the write layer — pseudocode
def update_stock(item_id, new_stock, fence_token):
    # DB must reject if fence_token <= last_seen_fence_token for this item
    ...
```

#### 2. Non-Atomic Heartbeat GET + EXPIRE (High)

```python
current = self.redis.get(lock_key)          # command 1
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # command 2
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The `EXPIRE` then extends a lock that is now owned by someone else. This must be a single atomic operation via a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_sha = self.redis.script_load(HEARTBEAT_SCRIPT)

# In heartbeat loop
self.redis.evalsha(self._heartbeat_sha, 1, lock_key, token, self.LOCK_TTL)
```

The same atomicity issue exists in the lock release in `_acquire_lock` (`GET` then `DELETE`), which should also be a Lua script — though this is a well-known pattern and a fix is straightforward.

#### 3. Shared Instance State Breaks Concurrent Callers (High)

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and either caller's `_stop_heartbeat` sets `_heartbeat_active = False` for both. This is a data race with no synchronization.

The heartbeat state must be scoped to the lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
    ...
```

#### 4. `quantity` Parameter Has No Validation (Medium)

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no lower-bound check. A caller passing `quantity=0` or `quantity=-5` would pass the stock check (`current_stock >= 0` is always true for non-negative stock) and either create a zero-quantity order or *increase* stock via `new_stock = current_stock - (-5)`. The `ReservationResult` model also does not validate this. Add an explicit guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

#### 5. Test Suite Mocks Away the Exact Code Under Review (Medium)

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

The tests mock out the heartbeat entirely, so none of the heartbeat logic (including the race condition in finding 2 above) is exercised. The tests also do not cover the concurrent-caller scenario that would expose finding 3. Tests should exercise the actual lock acquisition path with a real or fake Redis (e.g., `fakeredis`) rather than stubbing the lock internals.
