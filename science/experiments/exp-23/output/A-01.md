Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety violation: after a stop-the-world GC pause (or any OS-level suspension — OOM freeze, SIGSTOP, VM live migration, container CPU throttling), the process resumes with no awareness that wall-clock time elapsed while it was suspended. The heartbeat thread was also suspended, so it did not extend the TTL during the pause. If the pause exceeded the remaining TTL, the lock expired and may have been acquired by another process — but the resumed thread has no mechanism to detect this before it continues writing. It will proceed to call `db.update_stock` and `db.create_order` believing it still holds the lock, producing a classic post-pause unsafe write. The token check in the `finally` block only catches this at *release time*, not before the writes, which is too late. Additionally, the `InventoryService` instance carries mutable heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance variables rather than call-scoped locals, making it unsafe for concurrent use of the same service instance across requests — a realistic scenario in any threaded web server.

### Critical: Post-Pause Unsafe Write Window

The core problem is architectural. The heartbeat pattern is a reasonable defense against *slow operations under normal execution*, but it provides no protection against process suspension. The sequence of events that breaks this:

1. Lock acquired, heartbeat starts, TTL = 30s
2. Process suspended (GC pause, SIGSTOP, cgroup CPU throttle, hypervisor migration)
3. 30+ seconds elapse on the wall clock; Redis expires the lock
4. Another process acquires the lock and begins its own reservation
5. Original process resumes; heartbeat thread wakes up, checks token — token no longer matches, heartbeat silently stops
6. **Main thread has no idea.** It calls `db.update_stock` and `db.create_order` concurrently with the new lock holder
7. Both processes write; inventory is now corrupted

The token check before `redis.delete` in `finally` is the only guard, and it runs *after* the writes. This is the exact failure mode described in the Redlock analysis literature (Kleppmann, 2016) — the lock gives you a *window of safety*, not a *guarantee of exclusivity through the write*.

The correct mitigation requires a fencing token: a monotonically increasing value (e.g., from a Redis counter or a Lua-script-issued sequence number) that is passed through to the database write operations. The database layer must reject writes carrying a fencing token lower than the last observed token for that resource. Without this, no heartbeat scheme is safe against suspension.

```python
# What is needed — illustrative sketch only
# Redis issues a monotonic fence at lock acquisition time
fence = self.redis.incr(f"inventory:fence:{item_id}")

# The DB layer must enforce this
self.db.update_stock(item_id, new_stock, fence_token=fence)
# DB executes: UPDATE inventory SET stock=? WHERE item_id=? AND last_fence < ?
# If the condition fails, the write is rejected — not silently swallowed
```

Without a fencing token enforced at the storage layer, the distributed lock is providing false confidence.

### High: Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
# These are instance variables, shared across all concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

In a threaded web server (gunicorn sync workers, uWSGI threads, any threadpool executor), two simultaneous requests hitting `reserve_item` on the same `InventoryService` instance will race on `_heartbeat_active`. Request A's `_stop_heartbeat()` sets `_heartbeat_active = False`, which also kills Request B's heartbeat thread. Request B's lock then expires prematurely. This state must be call-scoped, not instance-scoped.

```python
# Correct approach: scope heartbeat state to the lock context
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start t, yield, finally: heartbeat_active.clear(), t.join(timeout=...)
```

Using `threading.Event` also gives you a clean join path, which the current implementation lacks entirely — the daemon thread is abandoned on `_stop_heartbeat`, not waited on.

### Medium: Heartbeat Check Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. This must be an atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same TOCTOU issue exists in the `finally` block's release logic (`GET` then `DELETE`). Both must be atomic. This is a well-known pattern and Redis documentation explicitly recommends the Lua approach for both operations.

### Medium: No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:baz"` or an empty string produces a malformed or unintended key. `quantity` has no lower bound check — a caller passing `quantity=0` or `quantity=-5` would pass the stock check and create a fraudulent order. Pydantic is imported but only used for the *output* model.

```python
# Minimum required
if not item_id or not item_id.isalnum():
    raise ValueError("item_id must be a non-empty alphanumeric string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```
