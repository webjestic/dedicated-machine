Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety violation under stop-the-world GC pauses or OS-level process suspension — the very scenario this architecture must defend against. If the process is suspended long enough for the lock TTL to expire and another node acquires the lock, the heartbeat thread will also be suspended; upon resumption it will check the token, find it no longer matches, and correctly skip renewal — but the *application thread* holding the original lock context will continue executing the write path (`update_stock`, `create_order`) against shared state it no longer exclusively owns, producing a double-reservation or inventory corruption. Additionally, there are two structural concurrency defects and one input validation gap that must be addressed before this is safe to merge.

---

## Detailed Findings

### 1. CRITICAL — Post-Expiry Write Safety (Stop-the-World / Process Suspension)

This is the core architectural flaw the heartbeat pattern was intended to solve, but it does not solve it.

**The failure sequence:**

```
T=0   Process A acquires lock, heartbeat starts, LOCK_TTL=30s
T=10  Heartbeat fires, extends TTL ✓
T=15  OS suspends process A (GC pause, SIGSTOP, VM live migration, etc.)
T=45  Lock expires in Redis (30s elapsed while process was suspended)
T=46  Process B acquires the same lock legitimately
T=47  Process A resumes — heartbeat fires, token mismatch, skips renewal ✓
      BUT: the application thread in reserve_item() continues:
        self.db.update_stock(item_id, new_stock)   ← executes under no lock
        self.db.create_order(order_id, ...)         ← executes under no lock
      Process B is simultaneously doing the same writes.
```

The heartbeat correctly detects the stolen lock and stops renewing, but it has **no mechanism to abort the application thread**. The write path proceeds regardless. This is a classic "fencing token" problem.

**Required remediation — fencing token pattern:**

The lock must carry a monotonically increasing fencing token (e.g., from a Redis `INCR` counter or Redlock's clock-based value), and every downstream write must present that token to the database, which rejects any write carrying a token lower than the last accepted one.

```python
# Acquire a monotonic fence alongside the lock
fence = self.redis.incr("inventory:fence:{item_id}")
self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)

# Pass fence to every write; DB layer enforces ordering
self.db.update_stock(item_id, new_stock, fence=fence)
self.db.create_order(order_id, item_id, quantity, fence=fence)
```

The database layer must then enforce:

```sql
UPDATE inventory
SET stock = %s, last_fence = %s
WHERE item_id = %s AND last_fence < %s   -- reject stale writes
```

Without this, the heartbeat is a false safety net. It detects the problem but cannot stop the damage.

### 2. HIGH — Shared Heartbeat State Breaks Concurrent Usage

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two callers invoke `reserve_item()` concurrently on the same `InventoryService` instance (which is the normal deployment model — a singleton service object), the second call's `_start_heartbeat` will overwrite the first call's thread reference, and `_stop_heartbeat` from either call will set `_heartbeat_active = False`, killing the other caller's heartbeat mid-operation.

```python
# Thread A: reserve_item("item_001") → _start_heartbeat sets _heartbeat_thread = Thread-1
# Thread B: reserve_item("item_002") → _start_heartbeat sets _heartbeat_thread = Thread-2
# Thread A: lock operation completes → _stop_heartbeat sets _heartbeat_active = False
# Thread-2 (B's heartbeat) now exits on next iteration — B's lock goes unrenewed
```

**Required remediation:** Heartbeat state must be scoped to the lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock ...
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 3. HIGH — Heartbeat Token Check is a TOCTOU Race

In `_beat()`:

```python
current = self.redis.get(lock_key)          # ← read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. The heartbeat then extends a lock it no longer owns. This must be an atomic operation.

**Required remediation — Lua script:**

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

self.redis.eval(HEARTBEAT_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same atomic pattern should already be applied to lock release (the existing `GET` + `DELETE` sequence in `_acquire_lock`'s `finally` block has the same race).

### 4. MEDIUM — No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. `quantity` has no lower bound — a caller can pass `quantity=0` or `quantity=-5`, which would silently *increase* stock (`new_stock = current_stock - (-5)`).

```python
reserve_item("item_001", quantity=-10)
# current_stock=5, new_stock=5-(-10)=15 → stock increases, order created
```

**Required remediation:**

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = Field(ge=1, le=1000)  # enforce positive, cap max

    @validator('item_id')
    def item_id_must_be_safe(cls, v):
        if not v.strip() or len(v) > 128:
            raise ValueError("item_id must be non-empty and ≤128 chars")
        return v

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. LOW — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

The heartbeat logic — the most operationally critical new code in this PR — has zero test coverage. The TOCTOU race in `_beat()`, the shared-state concurrency bug, and the post-expiry write problem are all invisible to the test suite as written. Tests should exercise the actual heartbeat thread using a real (or fakeredis) Redis instance with a short TTL, verifying both renewal and non-renewal behavior.

## Summary of Required Changes

| # | Severity | Issue | Resolution |
|---|----------|-------|------------|
| 1 | Critical | Writes proceed after lock expiry under process suspension | Implement fencing tokens at the DB write layer |
| 2 | High | Shared heartbeat state corrupts concurrent callers | Scope heartbeat state to lock context, not instance |
| 3 | High | TOCTOU race in heartbeat and lock release | Replace GET+EXPIRE and GET+DELETE with Lua atomics |
| 4 | Medium | No quantity/item_id validation; negative quantity increases stock | Add Pydantic `ReservationRequest` with `ge=1` constraint |
| 5 | Low | Heartbeat mocked out of all tests | Add integration-level tests against fakeredis with real TTLs |
