Decision: Request Changes

Summary: This implementation contains a classic **Redlock/fencing token vulnerability** — specifically, the heartbeat mechanism creates a false sense of safety against the exact failure mode it's designed to prevent. A stop-the-world GC pause (or any process suspension) will freeze the heartbeat thread alongside all application threads simultaneously, meaning the lock can still expire during a long DB operation, and critically, `db.update_stock()` and `db.create_order()` will execute after process resumption **without re-validating lock ownership at the write layer** — another process may have acquired the lock and already modified stock in the interim. The tests mock out the heartbeat entirely and do not cover this race condition, and there are additional secondary issues detailed below.

---

## Critical: TOCTOU Race — Process Pause Invalidates Lock Before Write

This is the core flaw. The heartbeat thread does **not** protect against process suspension.

```python
# The heartbeat thread and ALL application threads are frozen together
# during a stop-the-world GC pause or OS-level suspension.
# The lock TTL continues counting down in Redis during the pause.
# On resumption, the process believes it still holds the lock — but it may not.

with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # ← pause happens here
    # ... lock expires in Redis during pause ...
    # ... another process acquires the lock and decrements stock ...
    # ... this process resumes, unaware ...
    self.db.update_stock(item_id, new_stock)      # ← WRITES STALE DATA, no re-check
    self.db.create_order(order_id, item_id, quantity)  # ← ORDER CREATED ON PHANTOM STOCK
```

The heartbeat only helps with *slow but running* operations. It provides zero protection against suspension. The write operations must re-validate lock ownership at the moment of write, not just at lock acquisition time.

**Required fix — fencing token pattern:**

```python
# Option A: Re-validate lock ownership immediately before writes (minimum bar)
def _validate_lock_ownership(self, lock_key: str, token: str) -> bool:
    """Re-check ownership atomically before committing writes."""
    current = self.redis.get(lock_key)
    return current is not None and current.decode() == token

# Inside reserve_item, before any write:
with self._acquire_lock(lock_key) as token:
    current_stock = self.db.get_stock(item_id)
    # ... stock checks ...

    # Re-validate before writes — catches expired/stolen lock
    if not self._validate_lock_ownership(lock_key, token):
        logger.error(
            "Lock ownership lost before write item_id=%s — aborting reservation",
            item_id
        )
        return ReservationResult(
            success=False,
            message="Reservation could not be completed safely. Please retry."
        )

    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

```python
# Option B (preferred for correctness): Use monotonically increasing fencing tokens
# via Redis INCR, pass the token to the DB layer, and have the DB reject writes
# with stale tokens. This is the Martin Kleppmann recommendation and the only
# truly safe approach when the DB supports conditional writes.

fencing_token = self.redis.incr("inventory:fence:global")
self.db.update_stock(item_id, new_stock, min_fence=fencing_token)
```

## High: Heartbeat Race Condition on Lock Release

The heartbeat check-then-extend is a non-atomic read-modify sequence:

```python
# Thread A (heartbeat):
current = self.redis.get(lock_key)       # reads token — matches
# ← context switch here, lock expires, Process B acquires lock ←
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # extends Process B's lock!
```

This must be an atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if not result:
                    logger.error(
                        "Heartbeat lost lock ownership — lock expired or stolen key=%s",
                        lock_key
                    )
                    self._heartbeat_active = False
                    # Signal main thread that lock is lost
                    self._lock_lost.set()

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

Similarly, the lock release in `_acquire_lock` has the same non-atomic get-then-delete:

```python
# This is also a race — use a Lua script for atomic check-and-delete
RELEASE_LOCK_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

## Medium: Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance variables, not scoped to individual lock acquisitions. If `reserve_item` is ever called concurrently on the same `InventoryService` instance (e.g., in a thread pool), heartbeat state from one call will corrupt another:

```python
# Thread 1 acquires lock, starts heartbeat, _heartbeat_active = True
# Thread 2 acquires lock, calls _start_heartbeat, overwrites _heartbeat_thread
# Thread 1 calls _stop_heartbeat, sets _heartbeat_active = False
# Thread 2's heartbeat is now dead — lock will expire unrenewed
```

Fix: scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    lock_lost_event = threading.Event()
    # Pass event into heartbeat; heartbeat state is local to this context
    ...
```

## Medium: No Input Validation on Public Method

The PR claims Pydantic validation but `reserve_item` accepts raw primitives with no guards:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # item_id could be empty string, whitespace, or injection payload
    # quantity could be 0, negative, or absurdly large
```

Add explicit validation:

```python
from pydantic import BaseModel, Field, validator

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    ...
```

## Low: Tests Mock Out the Exact Behavior Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()   # ← stop bypassed
    return svc
```

The tests validate happy-path business logic but provide no coverage of the locking mechanism itself — the component carrying all the risk. Required additions:

- Test that lock expiry between read and write causes the reservation to abort
- Test that heartbeat loss is detected and surfaced
- Test concurrent `reserve_item` calls on the same instance don't corrupt heartbeat state
- Integration-level test against a real Redis instance (e.g., `fakeredis`) for the Lua scripts

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| Critical | Writes execute without re-validating lock ownership post-suspension |
| High | Heartbeat extend and lock release are non-atomic (TOCTOU) |
| Medium | Heartbeat state is instance-scoped, breaks concurrent use |
| Medium | No actual input validation on `reserve_item` parameters |
| Low | Tests mock out locking entirely; no coverage of failure modes |

The business logic and structure are sound. The distributed locking layer needs to be rebuilt with atomic Lua operations, per-context state scoping, and a mandatory ownership re-check immediately before any write. Do not merge in current form.
