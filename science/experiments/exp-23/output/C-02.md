Decision: Request Changes

Summary: The implementation demonstrates solid intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all sound patterns for this use case. However, there are several correctness, concurrency safety, and resilience issues that must be addressed before this is suitable for a production flash sale workload. The test suite also has meaningful gaps that mask some of these flaws.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the `InventoryService` instance. If this service is shared across threads (which is the normal deployment model for a web service), two concurrent calls to `reserve_item` will race on these fields. The second caller will overwrite `_heartbeat_active`, silently killing the first caller's heartbeat and potentially allowing its lock to expire mid-operation.

**Fix:** Heartbeat state must be local to each lock acquisition, not stored on `self`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread closure, not self
```

---

### 2. Check-Then-Act Race in the Heartbeat Thread

```python
# In _beat():
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The GET and EXPIRE are two separate commands. Between them, the lock can expire and be acquired by another process. You would then extend a lock you no longer own. This is a classic TOCTOU race.

**Fix:** Use a Lua script to make the check-and-extend atomic.

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat:
self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same issue exists in the lock release — GET then DELETE should also be a Lua script or use the standard Redlock release pattern.

---

### 3. The Same Lua Script Fix Applies to Lock Release

```python
# Current — also a TOCTOU race
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

**Fix:**

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

---

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation at all. There is no Pydantic model on the input path. A caller can pass:

- `quantity=0` or `quantity=-5` — would silently "reserve" zero or negative items and corrupt stock
- `item_id=""` — would attempt a lock on `inventory:lock:` and a DB lookup with an empty key
- `item_id` containing Redis key injection characters

**Fix:** Introduce a request model and validate at the boundary.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

---

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost. The distributed lock protects against concurrent reservation, but it does not protect against partial failure within a single reservation.

**Fix:** These must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread may still be mid-execution (e.g., in the middle of a Redis call) when the lock is released. In the worst case, the heartbeat extends the lock after the `finally` block has already deleted it — allowing another process to acquire the lock and then having it silently extended by the zombie thread.

**Fix:**

```python
def _stop_heartbeat(self, thread: threading.Thread, stop_event: threading.Event) -> None:
    stop_event.set()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

`RuntimeError` will also catch unexpected runtime errors from inside the `with` block (e.g., a bug in `db.update_stock` that raises `RuntimeError`). Those should not be silently swallowed and returned as "temporarily unavailable." Define a specific exception type.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) instead of RuntimeError
# catch only LockAcquisitionError at the call site
```

---

### 8. Redis Connection Errors Are Entirely Unhandled

If Redis is unavailable, `self.redis.set(...)` will raise `redis.exceptions.ConnectionError`. This propagates as an unhandled exception through `reserve_item`. For a flash sale service, Redis unavailability is a realistic failure mode and must be handled explicitly — either with a circuit breaker, a fallback, or at minimum a clean error response.

---

## Test Issues

### 9. Heartbeat Is Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the most complex and dangerous part of this code — the heartbeat thread — has zero test coverage. The TOCTOU race in the heartbeat, the zombie thread issue, and the instance-level state collision are all invisible to the test suite.

At minimum, add integration-level tests that let the heartbeat run against a real or fakeredis instance.

---

### 10. `test_lock_not_released_if_expired` Does Not Actually Test the Right Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

Because `_stop_heartbeat` is mocked, and the lock release reads `redis_client.get`, this test is asserting behavior of the mock, not of the real code path. It also does not verify that the reservation itself failed or succeeded correctly in this scenario — a stolen lock mid-operation should arguably be an error, not a silent success.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but both should be configurable via constructor parameters or environment config rather than class-level constants, since different deployment environments will have different Redis latency profiles.
- `logger = logging.getLogger(__name__)` is correct. No issues there.
- `daemon=True` on the heartbeat thread is appropriate — it will not block process shutdown.
