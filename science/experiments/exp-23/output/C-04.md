Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several concurrency and correctness flaws that could cause inventory corruption or silent failures under real flash-sale load. The test suite mocks away the heartbeat entirely and does not exercise the actual lock lifecycle, meaning the most critical paths are untested. These issues need to be resolved before this handles any production inventory.

---

## Detailed Review

### 🔴 Critical — Must Fix

#### 1. Shared Heartbeat State Breaks Concurrent Usage

The heartbeat uses instance-level fields (`self._heartbeat_active`, `self._heartbeat_thread`). If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — they will race on these fields. The second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, causing the first lock's heartbeat to be silently killed.

```python
# Current — instance-level state, not safe for concurrent callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass a `threading.Event` into the heartbeat thread and close over it locally.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()

    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        time.sleep((2 ** attempt) * 0.1)

    if not acquired:
        raise RuntimeError(...)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

#### 2. Heartbeat Check-Then-Act Is a Race Condition

The heartbeat does a `GET` followed by a separate `EXPIRE`. Between those two calls, the lock can expire and be acquired by another process. You would then extend a lock you no longer own.

```python
# Unsafe — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # may extend a stolen lock
```

**Fix:** Use a Lua script to make the check-and-extend atomic.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same atomic pattern should be applied to lock release in the `finally` block.

#### 3. `db.update_stock` and `db.create_order` Are Not Atomic

The Redis lock protects the read-modify-write on stock, but the two database writes are not wrapped in a transaction. If `create_order` fails after `update_stock` succeeds, stock is decremented with no corresponding order — inventory is silently lost.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # failure here = lost stock
```

**Fix:** These must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### 4. Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" but there is no validation on `item_id` or `quantity` in `reserve_item`. A caller can pass `quantity=0`, `quantity=-5`, or an empty string for `item_id` and the code will silently process it. Pydantic is imported but only used for the response model.

```python
# quantity=-1 passes all checks and "succeeds", increasing stock
if current_stock < quantity:  # 10 < -1 is False — proceeds
    ...
new_stock = current_stock - quantity  # 10 - (-1) = 11
```

**Fix:** Add a validated request model.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 🟡 Significant — Should Fix

#### 5. Test Suite Mocks Away the Code Under Review

Every test fixture patches out `_start_heartbeat` and `_stop_heartbeat`. This means the heartbeat thread, the atomic lock release, and the actual `finally` block behavior are entirely untested. The tests verify business logic but provide no confidence in the locking mechanism itself.

```python
# In fixture — the most important code path is never executed
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

At minimum, add integration-style tests that let the real lock lifecycle run against a mock Redis and verify: lock is released on success, lock is not released when token mismatches, heartbeat extends TTL, and heartbeat stops after context exit.

---

#### 6. `time.sleep` in Retry Loop Blocks the Calling Thread

The exponential backoff uses `time.sleep`, which blocks whatever thread is handling the request. Under flash-sale load with many goroutines/threads waiting, this will exhaust your thread pool.

Consider making the retry non-blocking (async) or at minimum document that callers must not invoke this from a request-handling thread without a timeout budget enforced at the caller level.

---

#### 7. No Maximum `quantity` Guard at the Service Layer

Even with a Pydantic model added, the service itself should enforce a sanity ceiling. A reservation for `quantity=999999` that passes validation should not silently drain all stock in one call without a business-rule check.

---

### 🔵 Minor / Informational

- **`LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10`:** The ratio is fine, but if the heartbeat thread is delayed by GIL contention or system load, you have a 20-second window before the lock expires without renewal. Consider tightening the interval to `TTL / 4`.
- **Daemon thread leak on process shutdown:** Daemon threads are acceptable here, but `heartbeat.join()` in the `finally` block (as shown in fix #1) ensures the thread does not outlive the lock context.
- **`logger.error` on lock contention is too noisy for expected behavior:** Lock contention under flash-sale load is expected. `logger.warning` is more appropriate; reserve `error` for genuinely unexpected failures.
