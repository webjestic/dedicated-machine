Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a classic stop-the-world vulnerability: if the Python process is paused (GC, OS scheduler starvation, container CPU throttling) for longer than the remaining TTL, the lock expires and another node acquires it — but when this process resumes, it believes it still holds the lock and proceeds to write, producing a split-brain write window. Beyond that core flaw, the implementation has a shared-state concurrency bug and a non-atomic release sequence that can corrupt lock ownership under concurrent callers. The tests mock away the heartbeat entirely, so none of these failure modes are exercised.

---

## Detailed Findings

### 1. Stop-the-World / Process Suspension — Split-Brain Write Window (Critical)

This is the central architectural flaw the heartbeat pattern cannot solve.

**The failure sequence:**

```
T=0   Process A acquires lock, heartbeat starts (TTL=30s)
T=10  Heartbeat fires, extends TTL to T=40
T=15  JVM/Python GC pause begins (or container CPU-throttled)
T=45  TTL expires — Redis evicts the lock
T=46  Process B acquires the lock, reads stock=5, begins write
T=47  Process A resumes from pause
      A's token check: redis.get(lock_key) → returns B's token
      ... but A already passed the stock check at T=14 with stock=5
      A calls db.update_stock() and db.create_order() concurrently with B
```

The token check in `finally` only protects the *delete* call. It does not protect the writes that already happened inside the `with` block. By the time the process resumes and reaches `finally`, the damage is done.

```python
# This check is too late. The writes below already executed.
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # read under (now-expired) lock
    ...
    self.db.update_stock(item_id, new_stock)      # write — no re-validation
    self.db.create_order(order_id, item_id, qty)  # write — no re-validation
```

**What is actually needed:** A fencing token (monotonic sequence number issued by the lock server) passed through to the database write, so the DB layer can reject writes from a stale lock holder. Redis's `WATCH`/`MULTI`/`EXEC` or a Redlock-with-fencing approach is required. Redisson (JVM) calls this a "fencing token"; the equivalent in Python requires explicit sequencing.

```python
# Sketch of the required pattern
fence = self.redis.incr("inventory:fence:{item_id}")  # monotonic, issued at acquire
...
# DB write must reject if stored fence > our fence
self.db.update_stock_fenced(item_id, new_stock, fence_token=fence)
```

Martin Kleppmann's analysis of this exact failure mode is the canonical reference: a heartbeat cannot distinguish "we are slow" from "we are dead and another node has taken over."

### 2. Shared Instance State Makes Concurrent Callers Unsafe (High)

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not per-call locals. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously:

```python
# Thread A calls _start_heartbeat → sets self._heartbeat_active = True
# Thread B calls _start_heartbeat → overwrites self._heartbeat_thread
# Thread A calls _stop_heartbeat  → sets self._heartbeat_active = False
#   → this also kills Thread B's heartbeat
# Thread B's lock now expires prematurely
```

This is a straightforward data race. The fix is to make heartbeat state local to each lock acquisition context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        active.clear()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 3. Non-Atomic Lock Release — TOCTOU on Delete (High)

```python
# _acquire_lock finally block:
current = self.redis.get(lock_key)          # (1) read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # (2) delete
```

Between (1) and (2), another process can acquire the lock. The delete then evicts a lock this process does not own. This is a well-known problem; the standard fix is a Lua script that makes the check-and-delete atomic:

```lua
-- release.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

release = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
release(keys=[lock_key], args=[token])
```

### 4. No Input Validation on `item_id` or `quantity` (Medium)

The Pydantic checklist item is misleading. `ReservationResult` is an *output* model. The *inputs* to `reserve_item` are raw strings and integers with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"  # item_id injected directly into key
```

A caller passing `item_id = "foo:bar"` or `item_id = ""` or `quantity = -1` or `quantity = 0` will produce incorrect behavior or a malformed Redis key. Add an input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. `db.update_stock` and `db.create_order` Are Not Atomic (Medium)

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes between these two calls, stock is decremented but no order exists — inventory is silently lost. These must execute in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Test Suite Does Not Cover the Failure Modes Being Protected Against (Medium)

The tests mock `_start_heartbeat` and `_stop_heartbeat` entirely, which means:

- The heartbeat thread is never exercised
- The TOCTOU race on lock release is never triggered
- The concurrent-caller shared-state bug cannot be detected
- There is no test simulating lock expiry mid-operation (the primary threat model)

At minimum, add:

```python
def test_write_blocked_after_lock_expiry():
    """Simulate process resuming after lock has expired and been acquired by another node."""
    ...

def test_concurrent_callers_independent_heartbeats():
    """Two threads using the same service instance must not interfere with each other's heartbeats."""
    ...

def test_lock_release_is_atomic():
    """Verify Lua script is used; a token mismatch must not delete the new holder's lock."""
    ...
```
