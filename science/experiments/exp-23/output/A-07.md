Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a classic stop-the-world safety gap: if the Python process is suspended (GC pause, OS scheduler stall, container CPU throttle, or a JVM-style STW event in a polyglot environment) for longer than `LOCK_TTL`, the lock expires on Redis's side, another process legitimately acquires it, and then *this* process resumes — unaware the lock is gone — and proceeds to write. The heartbeat thread is also suspended during that pause, so it cannot save you. Beyond that core flaw, there are several additional architectural and correctness issues that must be addressed before this is safe to merge.

---

## Detailed Findings

### 🔴 Critical — Stop-the-World / Process Suspension Safety Gap

This is the central flaw the heartbeat is meant to solve, but **cannot** solve.

```python
# LOCK_TTL = 30s, HEARTBEAT_INTERVAL = 10s
# Timeline of failure:
#
# t=0s   Process A acquires lock, heartbeat thread starts
# t=10s  Heartbeat extends TTL to t+30s (now expires at t=40s)
# t=15s  OS suspends entire process A (CPU steal, cgroup throttle,
#         swap storm, GC, kernel preemption — any cause)
# t=40s  Redis expires the lock — lock is now FREE
# t=41s  Process B acquires the same lock legitimately
# t=41s  Process B reads stock=10, begins its write path
# t=42s  Process A resumes — it never knew it was suspended
# t=42s  Process A's lock-ownership check:
#           current = self.redis.get(lock_key)  # returns Process B's token
#           current.decode() == token           # FALSE — so A won't delete it
#        BUT: A already passed the stock check at t=12s with stock=10
#        AND: A is about to call db.update_stock() and db.create_order()
#        RESULT: double reservation, inventory goes negative
```

The token check in the `finally` block only prevents *deleting* the lock. It does **not** prevent the already-in-progress write from completing. By the time the process resumes and reaches `db.update_stock()`, it is operating outside its lock window with stale data.

**Required fix:** The database layer must be the final safety net. Use a conditional update with a version/fence token, not just a Redis lock:

```python
# The DB write must be conditional — reject if stock changed since we read it
# Option A: Optimistic locking with a version column
def update_stock(item_id, expected_version, new_stock):
    rows_affected = db.execute(
        "UPDATE inventory SET stock = %s, version = version + 1 "
        "WHERE item_id = %s AND version = %s AND stock >= %s",
        (new_stock, item_id, expected_version, quantity)
    )
    if rows_affected == 0:
        raise StaleDataError("Stock changed under us — reservation rejected")

# Option B: Fencing token passed through to storage layer (Redlock-style)
# The lock itself issues a monotonically increasing token; storage rejects
# writes with a token older than the last accepted write.
```

The Redis lock should be treated as a *performance optimization* (reduces DB contention) not a *correctness guarantee*. Correctness must live in the storage layer.

### 🔴 Critical — Non-Atomic Heartbeat Check-Then-Extend

```python
# TOCTOU race in the heartbeat:
current = self.redis.get(lock_key)        # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. You are then extending *someone else's* lock. This must be a single atomic operation:

```lua
-- Lua script executed atomically in Redis
local val = redis.call('GET', KEYS[1])
if val == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
local val = redis.call('GET', KEYS[1])
if val == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
end
return 0
"""

# In heartbeat:
self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same TOCTOU issue exists in the lock release in `_acquire_lock`'s `finally` block and must be fixed with the equivalent atomic Lua script there too.

### 🔴 Critical — Shared Mutable Heartbeat State Breaks Concurrent Use

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False        # instance-level, not call-level
        self._heartbeat_thread = None         # instance-level, not call-level
```

If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal usage pattern — one service instance, many requests), two concurrent calls will stomp on each other's `_heartbeat_active` flag. Call A stopping its heartbeat will also stop Call B's heartbeat, leaving Call B's lock unrenewed.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        time.sleep((2 ** attempt) * 0.1)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key}")

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 🟡 High — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. This is a silent inventory leak. These two operations must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 High — No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is used only on the *output* model. An `item_id` of `"../../etc/passwd"`, an empty string, or a `quantity` of `-5` or `0` will pass straight through to the database layer.

```python
# Minimum required:
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    raise ValueError("Invalid item_id")
if not isinstance(quantity, int) or quantity < 1 or quantity > 10_000:
    raise ValueError(f"quantity must be between 1 and 10000, got {quantity}")
```

Or introduce a Pydantic input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=10_000)
```

### 🟡 High — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()    # ← stop entirely bypassed
    return svc
```

The heartbeat is the most complex and most failure-prone component in this PR, and the tests completely remove it. The tests do not cover:

- Concurrent calls on the same instance corrupting shared heartbeat state
- The TOCTOU race in the heartbeat's check-then-extend
- Behavior when the heartbeat thread raises an exception
- Lock expiry during a simulated process pause

These are not edge cases — they are the primary failure modes. Tests must exercise the real heartbeat path, using `threading` and time manipulation (e.g., `freezegun` or manual `Event` injection).

### 🟢 Low — Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The daemon thread will be abandoned. While `daemon=True` means it won't block process exit, it can still execute one more Redis call after `_heartbeat_active` is set to `False` (the `time.sleep` returns, the flag is checked, the flag is `False`, but there is a window). Use `threading.
