Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety flaw that is precisely the failure mode it claims to prevent: if the Python process undergoes a stop-the-world GC pause (or any OS-level suspension such as a VM live migration or SIGSTOP), all threads — including the heartbeat thread — are frozen, the lock TTL silently expires, another process acquires the lock and begins mutating inventory, and then *this* process resumes, believes it still holds the lock, and proceeds to execute `update_stock` and `create_order` concurrently with the new lock holder. The token check in the `finally` block does not protect the *critical section itself*, only the release step, so the double-write window is real and unmitigated. Additionally, the service instance is stateful (`_heartbeat_active`, `_heartbeat_thread`) in a way that is not concurrency-safe for multiple simultaneous callers, and the DB operations lack transactional atomicity guarantees.

---

## Detailed Findings

### 1. Stop-the-World Pause Invalidates the Heartbeat Safety Guarantee (Critical)

This is the central architectural flaw.

```
T=0   Process A acquires lock, heartbeat thread starts
T=5   GC pause begins — ALL threads suspended, including heartbeat
T=30  Redis TTL expires
T=31  Process B acquires the same lock, begins modifying stock
T=35  GC pause ends — Process A resumes inside the critical section
T=35  Process A calls db.update_stock() — CONCURRENT with Process B
```

The heartbeat pattern is borrowed from systems like Redlock and ZooKeeper session leases, but those systems acknowledge explicitly that a process that has been paused longer than its TTL **must not be trusted to continue writing** after resumption. The code has no mechanism to detect this condition. The token check only guards the lock *release*, not the *writes*:

```python
# This check happens AFTER update_stock and create_order have already executed.
# By this point the damage is done.
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)
    if current and current.decode() == token:  # too late
        self.redis.delete(lock_key)
```

The fix requires a **fencing token** (a monotonically increasing integer issued by Redis or a coordination service at lock acquisition time) that is passed through to the database layer. The DB must reject any write whose fencing token is not the highest it has seen for that resource. This is the only mechanism that survives process suspension.

```python
# Sketch of correct pattern
lock_key = f"inventory:lock:{item_id}"
fence_token = self.redis.incr(f"inventory:fence:{item_id}")  # monotonic
acquired = self.redis.set(lock_key, f"{token}:{fence_token}", nx=True, ex=self.LOCK_TTL)

# DB layer must enforce:
# UPDATE inventory SET stock = %s
# WHERE item_id = %s AND last_fence_token < %s
# and reject / raise if rows_affected == 0
```

Without this, the heartbeat is security theater against the exact threat it advertises.

### 2. Instance-Level Heartbeat State Is Not Safe for Concurrent Callers (High)

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped locals. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service under flash-sale load), the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first call's `_stop_heartbeat` sets `_heartbeat_active = False`, killing the second caller's heartbeat prematurely.

```python
# Caller A sets:          self._heartbeat_active = True
# Caller B sets:          self._heartbeat_active = True  (fine so far)
# Caller A finishes, sets: self._heartbeat_active = False  # kills B's heartbeat too
```

The heartbeat state must be encapsulated per lock acquisition, not per service instance:

```python
# Correct: pass a mutable cell scoped to the context manager
active = threading.Event()
active.set()
thread = threading.Thread(target=_beat, args=(active,), daemon=True)
```

### 3. DB Operations Are Not Atomic (High)

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If the process crashes, is killed, or receives an exception between these two lines, stock is decremented but no order record exists. Depending on the DB layer, this is either silent data loss or an inconsistency that requires manual reconciliation. Both calls must be wrapped in a single atomic transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 4. Input Validation Is Declared but Not Implemented (Medium)

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative units would pass the stock check and decrement stock)
- `item_id` being an empty string or containing injection-relevant characters
- `quantity` being a float passed where int is expected at runtime

A request model should be defined and validated at the boundary:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Heartbeat Has a TOCTOU Race on Token Check (Medium)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are not atomic. The lock can expire and be re-acquired by another process between these two lines, causing the heartbeat to extend a lock it no longer owns. This must be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same applies to the release in `_acquire_lock`'s `finally` block — this is a well-known pattern and the Lua script for it should be used there too.

---

### 6. Test Suite Mocks Away the Components Under Test (Low)

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture stubs out the heartbeat, meaning the tests provide zero coverage of the actual concurrency behavior. The tests verify the happy-path DB logic but would pass even if the heartbeat implementation were completely broken or absent. At minimum, one integration-style test should exercise the real heartbeat path against a local Redis instance (e.g., via `pytest-redis` or a testcontainer).

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Implement fencing tokens; heartbeat alone does not prevent post-pause writes |
| 2 | High | Move heartbeat state into call-scoped locals, not instance fields |
| 3 | High | Wrap `update_stock` + `create_order` in a single DB transaction |
| 4 | Medium | Add a validated `ReservationRequest` model with bounds on `quantity` and `item_id` |
| 5 | Medium | Replace GET+EXPIRE and GET+DELETE with atomic Lua scripts |
| 6 | Low | Add at least one test that exercises the real heartbeat and lock paths |
