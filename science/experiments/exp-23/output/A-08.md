Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical correctness flaw under stop-the-world GC pauses or OS-level process suspension — the very scenario this lock is meant to guard against. Beyond that, there are several structural and safety issues that must be resolved before this is suitable for a high-stakes inventory reservation path.

---

## Critical Issues

### 1. Heartbeat Does Not Solve the STW / Process Suspension Problem — It Obscures It

This is the most serious architectural flaw, and it is subtle enough that it could pass a casual review.

The heartbeat thread is a daemon thread running inside the same process. If the Python process is suspended — whether by a stop-the-world GC pause (relevant if this ever runs under a JVM-based Python, or adjacent to one), a kernel preemption, a SIGSTOP, a container freeze (e.g., `docker pause`, Kubernetes node pressure eviction staging, cgroup freezer), or a VM live migration — **both the application thread and the heartbeat thread are frozen simultaneously**. The Redis TTL clock keeps ticking. The lock expires. Another process acquires it. Then your process resumes, the heartbeat fires, sees the token still matches (because the new holder hasn't written yet, or the timing is tight), and **extends a lock it no longer legitimately owns**.

This is not a theoretical edge case. It is the exact failure mode that distributed lock TTLs are designed to expose, not paper over.

```python
# The race the heartbeat cannot prevent:
#
# T=0   Process A acquires lock, token = "abc", TTL = 30s
# T=10  Process A is frozen (container, GC, kernel, VM migration)
# T=30  Lock expires in Redis
# T=31  Process B acquires lock, token = "xyz"
# T=32  Process A resumes, heartbeat fires:
#         current = redis.get(lock_key)  → b"xyz"  (B's token, not A's)
#         "xyz" != "abc", so heartbeat correctly skips extension — OK here
#
# BUT consider this interleaving:
# T=31.0  Process B acquires lock, token = "xyz", begins write
# T=31.5  Process A resumes mid-heartbeat, reads key → b"xyz"
#          A's token "abc" != "xyz", skips — correct
#
# The REAL danger is not the heartbeat itself here, it's that:
# Process A's *application thread* also resumed and is still inside
# the `with self._acquire_lock(lock_key):` block, executing
# db.update_stock() and db.create_order() — WITHOUT holding the lock.
# The lock check only happens at release time, not during the write.
```

The heartbeat's token check on `expire` is correct. The problem is that **the application code continues executing writes after resumption, with no awareness that the lock was lost**. The lock boundary is not enforced at write time.

**Required fix:** The write operations must be fenced. After any operation that could have experienced a pause, and critically before committing writes, the service must verify it still holds the lock. A fencing token (monotonically increasing integer from Redis, e.g., via `INCR` at acquisition time) written as a precondition to the DB transaction is the correct mitigation. Alternatively, the DB writes must be conditional on a version/epoch that was captured under the lock.

```python
# Sketch of fencing token approach:
fence_token = self.redis.incr("inventory:fence")
# Pass fence_token into db.update_stock() and db.create_order()
# DB layer rejects writes if a higher fence_token has already been committed
```

### 2. Shared Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of a service object — two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will terminate the first call's heartbeat, and vice versa.

```python
# Concurrent call A sets _heartbeat_active = True, starts thread
# Concurrent call B sets _heartbeat_active = True, starts a second thread
# Call A finishes, sets _heartbeat_active = False
# Call B's heartbeat is now also dead — lock will expire
```

**Required fix:** Heartbeat state must be scoped to the lock acquisition context, not to the instance. Pass a `threading.Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 3. Heartbeat Check-Then-Act Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a TOCTOU race. Between the `GET` and the `EXPIRE`, another process can acquire the lock. This must be a single atomic operation.

**Required fix:** Use a Lua script.

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
# Register with redis.register_script() and call atomically
```

The same applies to lock release:

```python
# Current code — not atomic:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

**Required fix:** Standard Redlock-style atomic release script.

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

### 4. `db.update_stock` and `db.create_order` Are Not Atomic With Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` fails, stock is decremented with no corresponding order record. This is a phantom reservation — inventory is consumed but no order exists. There is no transaction boundary, no rollback, and no compensation logic.

**Required fix:** These two operations must execute within a single DB transaction, or the service must implement a saga with explicit compensation (`db.restore_stock` on `create_order` failure).

### 5. Input Validation Is Claimed But Not Present

The checklist marks "Input validation via Pydantic" as complete. `ReservationResult` is a Pydantic output model. There is no Pydantic model validating inputs to `reserve_item`. `item_id` is an unvalidated string passed directly into a Redis key and DB query. `quantity` has no minimum value check — a caller can pass `quantity=0` or `quantity=-5`.

```python
# quantity=-1 passes the stock check: current_stock(10) < -1 is False
# So it proceeds to: new_stock = 10 - (-1) = 11
# Stock is *increased* by a reservation call.
```

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat` sets the flag but does not join the thread. The daemon thread may fire one more time after the flag is set but before the thread exits, potentially touching Redis after the lock is released.

**Fix:** As shown in issue #2, `thread.join()` with a bounded timeout in the `finally` block.

### 7. `retry_attempts=3` With Backoff Is Hardcoded and Not Configurable

Maximum wait before giving up is `0.1 + 0.2 + 0.4 = 0.7s`. Under genuine flash sale load this may be appropriate, but it should be a constructor parameter, not a magic number buried in a method signature default.

### 8. No Redis Connection Error Handling

`redis.exceptions.ConnectionError` and `redis.exceptions.TimeoutError` are not caught. A Redis outage will propagate an unhandled exception through `reserve_item` rather than returning a clean `ReservationResult(success=False, ...)`. For a flash sale path this is a reliability gap.

---

## Test Coverage Gaps

- No test exercises concurrent calls on the same instance (the shared state bug in issue #2 would be caught by this).
- No test covers `update_stock` succeeding and `create_order` raising an exception (the partial-write bug in issue #4).
- No test covers `quantity <= 0` inputs (the negative quantity bug in issue #5).
- The heartbeat is fully mocked out in every test — the heartbeat's own logic (including the TOCTOU race in issue #3) has zero test coverage.
- No test covers Redis `ConnectionError` during lock acquisition or during the write path.
