Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this handles real flash sale load in a government-grade system. The most critical are a race condition in the heartbeat's token check, an instance-level concurrency flaw that breaks concurrent use, and a missing atomicity guarantee on the DB writes.

---

## Detailed Findings

### 🔴 Critical

**1. Heartbeat token check is not atomic (TOCTOU race)**

```python
# Current — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be a single atomic operation. Use a Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat
extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    self._heartbeat_active = False  # We lost the lock; stop beating
    logger.warning("Heartbeat: lock lost or expired key=%s", lock_key)
```

The same TOCTOU issue exists in the `finally` release block and must be fixed there too with a Lua `GET`+`DEL` script (this is the canonical Redlock release pattern).

---

**2. Instance-level heartbeat state breaks concurrent calls**

```python
# On the instance — not per-call
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (entirely normal in a web server), one call's `_stop_heartbeat()` will cancel the other's heartbeat mid-operation, silently dropping the lock. Heartbeat state must be scoped per lock acquisition, not per instance:

```python
@dataclass
class _LockContext:
    token: str
    active: threading.Event
    thread: threading.Thread

@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into heartbeat thread, not self._heartbeat_active
    ...
```

**3. DB operations are not atomic with respect to each other**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against this — the lock is a coordination primitive, not a transaction boundary. These two writes must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should require and document transaction support.

### 🟡 Significant

**4. Input validation is claimed but not implemented**

The checklist says "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` could be empty, a path-traversal string, or absurdly long. `quantity` could be zero or negative. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

A `quantity=0` reservation today silently succeeds (stock check passes, stock unchanged, order created for zero units) — that is a data integrity bug.

---

**5. Heartbeat can extend a lock after the context manager has exited**

The heartbeat thread is daemon, so it won't block shutdown, but the sequence:

1. `finally` block runs: `_stop_heartbeat()` sets `_heartbeat_active = False`
2. Thread is mid-sleep at `time.sleep(HEARTBEAT_INTERVAL)`
3. `finally` block: `redis.get` + `redis.delete` — lock released
4. Thread wakes, checks `_heartbeat_active` — it's `False`, so it stops

This is actually safe *in the current code* but only by coincidence of ordering. With the per-instance fix in point 2, use `threading.Event` with `wait(timeout)` instead of `time.sleep` so the thread wakes immediately on stop signal:

```python
def _beat(stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock ...
```

This also makes `_stop_heartbeat` instant rather than waiting up to `HEARTBEAT_INTERVAL` seconds.

---

**6. `retry_attempts=3` with backoff of 0.1s, 0.2s, 0.4s is likely too short for a flash sale**

Under genuine flash sale load, 0.7 seconds total wait before declaring "service unavailable" will produce a high false-failure rate. This should be configurable at construction time, not hardcoded, and the default values should be tuned against expected p99 DB operation latency.

### 🟠 Test Coverage Gaps

**7. Tests mock out the exact things being tested**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat and lock release logic — the most complex and failure-prone parts — have zero test coverage. At minimum add:

- A test that verifies the Lua release script is called, not a raw `delete`
- A test that the heartbeat stops when `stop_event` is set
- A test that concurrent calls on the same instance do not interfere (the bug in point 2)
- A test that a DB exception in `create_order` does not leave stock decremented (point 3)

**8. `test_lock_not_released_if_expired` does not test what it claims**

The test mocks `redis.get` to return `b"different-token"` but the `service` fixture already mocks `_start_heartbeat` and `_stop_heartbeat`, so the token generated inside `_acquire_lock` is `uuid4()`, not `"test-token"`. The assertion `delete.assert_not_called()` passes for the wrong reason. This test needs to be rewritten against the real lock release path.

---

### 🔵 Minor / Style

- `LOCK_TTL` and `HEARTBEAT_INTERVAL` as class-level constants means all instances share them. Move to constructor parameters with class-level defaults so they're testable and configurable per deployment environment.
- Log lines should include a correlation ID (e.g., `order_id` or a request trace ID) from the start of the operation, not only after success.
- The `db` parameter type is untyped (`db`). Define a `Protocol` or abstract base class so the interface contract is explicit and mockable without surprise.
