Decision: Request Changes

Summary: The implementation demonstrates solid intent — distributed locking with token ownership, heartbeat renewal, exponential backoff, and structured logging are all reasonable patterns for this problem. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a government-facing production system. The most critical issues are a race condition in the heartbeat, an instance-level concurrency bug, and a missing atomicity guarantee on lock release.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Not Thread-Safe for Concurrent Calls

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the **instance**, not scoped to the lock invocation. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), they will clobber each other's heartbeat state. One thread calling `_stop_heartbeat()` will silence the other thread's heartbeat, potentially allowing the other thread's lock to expire mid-operation.

**Required fix:** Heartbeat state must be local to each lock acquisition, not stored on `self`.

```python
# Suggested pattern — pass a threading.Event into the heartbeat closure
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller signals stop via stop_event.set()
```

### 2. Lock Release Is Not Atomic

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The `GET` and `DELETE` are two separate Redis commands. Between them, the lock could expire and be re-acquired by another process. You would then delete a lock you no longer own, evicting a legitimate holder. This is a classic TOCTOU race.

**Required fix:** Use a Lua script to make the check-and-delete atomic. This is the standard Redis pattern for safe lock release.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. Heartbeat Has Its Own TOCTOU Race

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` + `EXPIRE` in the heartbeat has the same atomicity problem as the release. Between the `GET` and the `EXPIRE`, the lock could expire and be re-acquired by another process. You would then extend a lock you no longer own.

**Required fix:** Use a Lua script here as well.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

## Significant Issues

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:../../admin"` or a zero/negative `quantity` will not be caught. Pydantic models are defined for the *result*, not the *input*.

**Required fix:** Validate inputs at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]{1,64}$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. Database Operations Outside a Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock protects against concurrent *Redis* access, but it does not protect against partial database writes.

**Required fix:** These must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. Bare `RuntimeError` Catch Masks Real Failures

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is too broad. A programming error inside the lock body (e.g., a bug in `db.update_stock`) that raises `RuntimeError` will be silently swallowed and returned as "service unavailable" instead of propagating. This will make production debugging extremely painful.

**Required fix:** Define a specific exception class.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

## Minor Issues

### 7. Tests Mock Away the Exact Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the heartbeat or lock release paths. `test_lock_not_released_if_expired` asserts `delete` is not called, but because the heartbeat is mocked, the token comparison in the `finally` block is what's actually being tested — and the mock `redis_client.get` returns `b"different-token"` globally, meaning the success-path tests are also operating with a mismatched token and the lock is never being "released" in those tests either.

The tests need to be restructured to test real behavior, or at minimum the mock boundaries need to be clearly documented as intentional.

### 8. `time.sleep` in Lock Retry Blocks the Event Loop

If this service is ever called from an async context (FastAPI, etc.), `time.sleep` in the retry loop will block the entire thread. This should be noted explicitly in the docstring, or the service should be designed with async compatibility in mind given the likely web-facing deployment.
