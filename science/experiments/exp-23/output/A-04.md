Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety flaw that is the central concern of this review: a stop-the-world GC pause (or any OS-level thread suspension) can silently invalidate the lock while the application believes it still holds it, and process resumption after that window means writes proceed without a valid lock — defeating the entire mutual exclusion guarantee. Beyond that core issue, there are several additional architectural and correctness problems that must be addressed before this is safe to merge.

---

## Detailed Findings

### 🔴 Critical — STW Pause / Thread Suspension Invalidates Lock Safety

This is the most serious issue and the reason the heartbeat pattern is fundamentally unsafe without fencing.

**The scenario:**

```
T=0   Lock acquired, TTL=30s, heartbeat thread starts
T=10  Heartbeat fires, extends TTL to T=40
T=12  GC pause begins (or OS suspends all threads — e.g., SIGSTOP, VM live migration)
T=42  Lock expires in Redis (TTL elapsed while process was suspended)
T=43  Another process acquires the same lock — it is now the rightful owner
T=45  GC pause ends, original process resumes
T=45  Heartbeat fires: redis.get() returns the NEW owner's token — mismatch, no extend
T=45  reserve_item() continues executing inside what it believes is a protected region
T=45  db.update_stock() and db.create_order() execute WITHOUT holding the lock
```

The heartbeat thread does check token ownership before extending, which is correct. But that check only protects the *extension* step. It does **not** protect the *critical section itself*. After resumption, the code in `reserve_item` has no awareness that the lock expired and was re-acquired by a competing process. Both processes now execute the critical section concurrently — the exact condition the lock was meant to prevent. For a flash sale inventory system, this means overselling is possible.

**The correct mitigation is a fencing token.** Redis's `WAIT` command, Redlock with monotonic fencing tokens, or a database-level optimistic lock (e.g., a `version` column with a conditional `UPDATE WHERE version = expected`) must be used so that even if the distributed lock is lost, the storage layer rejects the stale write.

```python
# Example: optimistic concurrency at the DB layer as a safety net
rows_affected = self.db.update_stock_if_version(item_id, new_stock, expected_version)
if rows_affected == 0:
    raise StaleWriteError("Stock version mismatch — concurrent modification detected")
```

Without this, the distributed lock provides only a *performance optimization* (reducing contention), not a *correctness guarantee*.

### 🔴 Critical — Shared Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to a single lock acquisition. If two coroutines or threads call `reserve_item` on the same `InventoryService` instance concurrently:

- One call's `_stop_heartbeat()` sets `_heartbeat_active = False`, killing the other call's heartbeat.
- `_heartbeat_thread` is overwritten, losing the reference to the first thread.

This is a race condition that can cause premature lock expiry under the exact concurrent load this service is designed to handle.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

### 🔴 Critical — Non-Atomic Check-Then-Act on Lock Release

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a TOCTOU race. Between `GET` and `DELETE`, the lock can expire and be acquired by another process. The `DELETE` then evicts the new owner's lock. This is a well-known Redis distributed lock anti-pattern.

**Fix:** Use the canonical atomic Lua script for conditional release, which is the industry standard for this exact reason.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

### 🔴 Critical — No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. There is no guard against:

- `item_id = ""` or `item_id = "../../../../etc/passwd"` (depending on DB layer)
- `quantity = 0` — succeeds, creates a zero-quantity order record
- `quantity = -5` — decrements stock by -5, *increasing* inventory
- `quantity = 999999999` — no upper bound

```python
# quantity=-1 path:
current_stock = 10
if current_stock < -1:  # False — passes the check
    ...
new_stock = 10 - (-1)  # = 11 — stock increases
```

**Fix:** Validate at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 🟡 Major — DB Operations Not Wrapped in a Transaction

`update_stock` and `create_order` are two separate DB calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. This is a silent data integrity failure with no compensation logic.

```python
# Current state — not atomic:
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # raises — stock is now wrong
```

**Fix:** Wrap in a transaction or use a saga with explicit rollback.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Major — Heartbeat Race on Startup

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # sleeps FIRST
        if self._heartbeat_active:
            ...
```

The heartbeat sleeps for 10 seconds before its first extension. With a 30-second TTL, this is acceptable in the normal case, but if the lock is acquired at T=0 and a slow DB operation runs from T=0 to T=25, the first heartbeat fires at T=10 (extending to T=40) — fine. But if `_stop_heartbeat` is called at T=9 (fast path), the thread wakes at T=10, checks `_heartbeat_active = False`, and exits cleanly. This is acceptable but should be documented. Using `threading.Event.wait(timeout)` instead of `time.sleep` would allow faster, cleaner shutdown.

### 🟡 Major — Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, meaning the heartbeat logic — the most operationally critical and complex part of this PR — has zero test coverage. The TOCTOU lock release bug, the shared state concurrency bug, and the STW resumption scenario are all untested. Tests should exercise the real code paths, with integration tests against a real or embedded Redis instance for the lock lifecycle.

---

### 🟢 Minor — `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Not Validated Relative to Each Other

If someone changes `HEARTBEAT_INTERVAL = 35` and `LOCK_TTL = 30`, the heartbeat never fires before the lock expires. This relationship should be enforced.

```python
def __init__(self, ...):
    assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
        "HEARTBEAT_INTERVAL must be less than half LOCK_TTL"
    )
```
