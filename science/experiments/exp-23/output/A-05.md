Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety flaw that is precisely the scenario this codebase should be hardened against: if the Python process experiences a stop-the-world GC pause, OS scheduling starvation, or any other form of thread suspension, the heartbeat thread silently stops extending the lock TTL, the lock expires, another process acquires it and begins writing, and then the original process resumes mid-critical-section with two writers now operating concurrently under the false assumption of mutual exclusion. Beyond that core architectural issue, there are several additional correctness and safety problems detailed below.

---

## Critical: Heartbeat Does Not Protect Against STW Suspension

This is the central flaw. The PR description claims the heartbeat thread "ensures the lock does not expire during slow DB operations," but this is only true when the process is healthy and scheduled. It is false in the exact failure modes that distributed locks are designed to defend against.

**The failure sequence:**

```
T=0s   Process A acquires lock, heartbeat thread starts
T=5s   GC pause begins (or OS suspends all threads — e.g., VM live migration,
       SIGSTOP, heavy memory pressure causing swap)
T=30s  Lock TTL expires — Redis deletes the key
T=31s  Process B acquires the same lock, begins decrementing stock
T=35s  Process A resumes — heartbeat fires, sees token mismatch, stops extending
       BUT reserve_item() is still executing inside the `with` block
T=35s  Process A and Process B are now both inside the critical section
       Double-spend / negative inventory is now possible
```

The token check in the heartbeat and in the `finally` block only prevents a *stale lock release*. It does **not** prevent a *stale write*. After resumption, `reserve_item()` has no mechanism to detect that the lock expired and was re-acquired by another process. It will proceed to call `db.update_stock()` and `db.create_order()` without any awareness that the invariant it relied on is gone.

**The correct mitigations are:**

1. **Fencing tokens (monotonic lock version numbers).** Redis with Redlock or a system like ZooKeeper can issue a fencing token — a monotonically increasing integer — at lock acquisition time. The database layer must reject any write whose fencing token is lower than the last committed token. This is the only mechanism that provides true write safety after a pause.

2. **Optimistic concurrency control at the DB layer.** The `update_stock` call should use a conditional write (e.g., `UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s`) so that a stale write from a resumed process fails rather than silently corrupting state.

3. **If the heartbeat pattern is retained at all**, the critical section must re-validate lock ownership *immediately before any write*, not just at release time, and that check must be atomic with the write (Lua script or a transaction). Even then, this does not fully close the window.

## High: Heartbeat State Is Instance-Level, Not Lock-Level

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the `InventoryService` instance, not scoped to an individual lock acquisition. If two callers share the same `InventoryService` instance (likely in any web framework with a shared service layer), concurrent calls to `reserve_item()` for different items will race on `_heartbeat_active` and `_heartbeat_thread`. One call stopping its heartbeat will silently kill the other's.

**Fix:** Heartbeat state must be local to the `_acquire_lock` context, passed by closure or encapsulated in a per-acquisition object.

```python
# Sketch of correct scoping
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

## High: Heartbeat Check-Then-Extend Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a TOCTOU race. Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The `EXPIRE` then extends a lock now owned by someone else, silently extending their TTL beyond what they set. This must be a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same applies to the release logic in the `finally` block — this is a well-known pattern and the fix is identical (Lua CAS delete, which the Redlock spec documents explicitly).

## Medium: No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is only used for the *output* model `ReservationResult`. There is no guard against:

- `quantity <= 0` (reserving zero or negative items would pass the stock check and corrupt inventory)
- `item_id` containing Redis key injection characters (`:`, `*`, `[`, etc.) — the lock key is constructed as `f"inventory:lock:{item_id}"` with no sanitization

At minimum:

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
```

And `item_id` should be validated against an allowlist pattern (alphanumeric + limited punctuation) before being interpolated into a Redis key.

---

## Medium: DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The lock is released and the item is effectively lost. These two operations must be wrapped in a database transaction, or the service must implement compensating logic. This is a data integrity issue independent of the locking concerns.

---

## Low: Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to observe it and exit. The thread may fire one more `EXPIRE` call after `_stop_heartbeat()` returns and before the `finally` block deletes the key, extending a lock that is about to be released. Use `threading.Event` with `wait(timeout=...)` and call `thread.join()` in the `finally` block.

---

## Low: Tests Mock Away the Exact Behavior Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture stubs out the heartbeat entirely. There are no tests for the heartbeat's actual behavior: does it extend correctly, does it stop on flag clear, does it handle a token mismatch? The most important test — behavior after lock expiry mid-critical-section — is entirely absent. Tests should be added that exercise the real heartbeat path with a fast-expiring lock and a mock Redis that simulates expiry.

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| Critical | No write safety after STW pause / lock expiry; fencing tokens or OCC required |
| High | Heartbeat state is instance-scoped, not acquisition-scoped; concurrent calls corrupt each other |
| High | Heartbeat extend and lock release are non-atomic TOCTOU races; require Lua scripts |
| Medium | No validation on `quantity` (allows ≤ 0) or `item_id` (allows key injection) |
| Medium | `update_stock` + `create_order` not wrapped in a DB transaction |
| Low | Heartbeat thread not joined before lock release |
| Low | Tests mock out heartbeat entirely; no coverage of actual heartbeat behavior |
