Decision: Request Changes

Summary: The core locking pattern and business logic are sound — atomic `SET NX EX`, token-verified release, and heartbeat renewal are all implemented correctly. However, there are several concurrency, correctness, and resilience issues that must be addressed before this is safe to merge into a high-throughput flash sale path. The test suite also has meaningful gaps that mask real failure modes.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrent Use

This is the most serious bug in the PR.

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the **instance**, not scoped to the lock acquisition. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the exact scenario this service is designed for — the following race occurs:

- Thread A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Thread B acquires a different lock, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Thread A's `finally` block calls `_stop_heartbeat`, sets `_heartbeat_active = False`
- Thread B's heartbeat is now silently dead; its lock will expire mid-operation

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. Use a local `threading.Event` or encapsulate the lock + heartbeat in a dedicated context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = threading.Thread(
        target=self._beat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 2. Lock Release Is Not Atomic

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The GET and DELETE are two separate round-trips. Between them, the lock can expire and be acquired by another process, which you then delete. This is a classic TOCTOU race. The canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's renewal check:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # another process could own it here
```

This should also be a Lua script or use `SET key token XX EX ttl` (set-if-exists-with-same-value is not a native command, so Lua is the right path).

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation at the entry point. Pydantic is only used for the **response** model. There is no guard against:

- `quantity <= 0` (reserving zero or negative stock would silently corrupt inventory)
- `item_id` being an empty string or containing injection-relevant characters if it ever flows into a non-parameterized query
- Extremely large `quantity` values

Add a request model and validate at the boundary:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Significant Issues

### 4. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread sleeps for `HEARTBEAT_INTERVAL` (10s) before checking the flag. If the lock is released and the thread wakes up just after, it will attempt a Redis GET on a key that may now be owned by another process. With the instance-level state bug from issue #1, this becomes a guaranteed problem under load. Always join the thread with a bounded timeout in the `finally` block.

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order record exists — inventory is silently lost. This must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the `db` abstraction, but the PR must define and enforce transactional semantics here.

### 6. Unhandled Redis Exceptions

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` is caught. `redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, and `redis.exceptions.RedisError` are not handled. A Redis outage will raise an uncaught exception that propagates to the caller with no structured response. Add explicit handling:

```python
except (redis.exceptions.RedisError, RuntimeError) as e:
    logger.error("Reservation failed item_id=%s error=%s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
```

## Test Suite Issues

### 7. Heartbeat Is Mocked Away in All Tests

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture bypasses the heartbeat entirely. There are no tests that exercise the actual heartbeat renewal logic, the thread lifecycle, or the stop behavior. Given that the heartbeat is the primary novel mechanism in this PR, it needs direct test coverage:

- Does the heartbeat extend the TTL when the token still matches?
- Does the heartbeat skip renewal when the token has changed?
- Does the thread terminate cleanly after `_stop_heartbeat`?

### 8. `test_lock_not_released_if_expired` Does Not Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

Because `_start_heartbeat` is mocked, `redis_client.get` is also called in the heartbeat path in real code — but not here. More critically, `redis_client.get` returning `b"different-token"` means the lock release check will correctly skip deletion, but the test is asserting on a mock that also controls the `_acquire_lock` token comparison. The test does not actually verify the TOCTOU scenario it describes; it only verifies that the mock returns the right value. Once the Lua script fix from issue #2 is applied, this test will need to be rewritten against the script invocation.

### 9. No Concurrency Tests

For a service explicitly designed to handle concurrent flash sale load, there are zero multi-threaded tests. At minimum, add a test that spawns N threads simultaneously calling `reserve_item` on the same item and asserts that the total reserved quantity never exceeds initial stock.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but consider making both configurable via constructor parameters for different deployment environments.
- The `token` yielded from `_acquire_lock` is not used by the caller (`with self._acquire_lock(lock_key):` discards it). Either remove the yield value or document its intended use.
- Log level for lock contention is `WARNING` on every retry attempt. Under flash sale load this will flood logs. Consider `DEBUG` for intermediate attempts and `WARNING` only on final failure.
