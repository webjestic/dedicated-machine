Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw that makes it unsafe for concurrent use — the heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) is stored as instance-level attributes rather than being scoped to each lock acquisition. Beyond that, the lock release in the `finally` block contains a TOCTOU race condition, and the test suite mocks away the heartbeat entirely, meaning the most dangerous code path is never actually exercised. These issues need to be resolved before this can be merged.

---

## Detailed Findings

### 🔴 Critical — Instance-Level Heartbeat State (Thread Safety)

```python
# Current — instance attributes shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or long-lived object. If two requests call `reserve_item` concurrently, they share `_heartbeat_active` and `_heartbeat_thread`. Request A's `_stop_heartbeat()` will kill Request B's heartbeat, potentially allowing B's lock to expire mid-operation. This is a silent data-integrity failure under load — exactly the scenario this code is supposed to protect against.

**Fix:** Scope heartbeat state to each lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local scope, not self.*

    def _beat():
        while not heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # wait() returns False on timeout, True on set() — invert logic
            pass  # see corrected version below

    # ... acquire lock ...

    heartbeat_active.set()  # signal stop on exit
```

A cleaner version:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.LOCK_TTL)

t = threading.Thread(target=_beat, daemon=True)
t.start()
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # ... release lock ...
```

### 🔴 Critical — TOCTOU Race in Lock Release

```python
# In finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can steal the lock between GET and DELETE
```

The GET and DELETE are not atomic. Between the `get` and `delete`, another process could acquire the lock (if it expired) and you would delete a lock you no longer own. This is a well-known Redis distributed lock pitfall. The canonical fix is a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 🟡 High — Tests Mock Away the Dangerous Code

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # stop never runs
    return svc
```

The heartbeat thread is the most complex and failure-prone part of this PR, and it is completely stubbed out in the majority of tests. The checklist claims "Tests pass" but the heartbeat logic has zero test coverage. A test for the concurrent instance-state collision described above would have caught the critical bug.

**Required additions:**
- A test that instantiates a real (non-mocked heartbeat) service and verifies TTL extension occurs
- A concurrency test using `threading.Thread` or `concurrent.futures.ThreadPoolExecutor` that fires two `reserve_item` calls simultaneously and asserts exactly one succeeds

### 🟡 Medium — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used on the *output* model. There is no guard against:

```python
service.reserve_item("", quantity=0)
service.reserve_item("item_001", quantity=-5)
service.reserve_item("../../../etc/passwd", quantity=1)  # lock key injection
```

The third case is particularly concerning — `item_id` is interpolated directly into a Redis key with no sanitization. A caller controlling `item_id` could craft a key that collides with unrelated Redis namespaces.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        return ReservationResult(success=False, message=f"Invalid input: {e}")
    # use req.item_id, req.quantity from here
```

### 🟡 Medium — Heartbeat Thread Not Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10s) between iterations. Setting the flag to `False` does not interrupt the sleep. The thread could extend the lock *after* `_stop_heartbeat()` is called but *before* the lock is released, extending a lock that should be freed. Use `threading.Event` with `wait(timeout=...)` as shown above, and call `t.join()` in the finally block.

---

### 🟢 Minor — Retry Backoff Blocks the Event Loop

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In an async or high-throughput WSGI context, blocking sleeps on a thread pool worker are acceptable but worth flagging. If this service is ever ported to `asyncio`, these must become `await asyncio.sleep(backoff)`. Consider adding a comment noting this assumption.
