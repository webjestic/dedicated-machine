Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real flash sale load in a government-grade system. The most critical is a race condition in the heartbeat's check-then-extend sequence and a concurrency flaw that makes the service unsafe for multi-threaded deployments.

---

## Critical Issues

### 1. Heartbeat Has a TOCTOU Race Condition

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. You are then extending a lock you no longer own. This must be an atomic operation.

**Fix — use a Lua script:**

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register on __init__
self._extend_lock = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same atomic pattern should also be applied to the release in `_acquire_lock`'s `finally` block, which has the identical TOCTOU flaw.

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
# These are instance attributes, not per-call
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of a flash sale service — the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat and leaving its lock unprotected.

**Fix — make heartbeat state local to each lock acquisition:**

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return
            logger.debug("Lock TTL extended key=%s", lock_key)

    # ... acquire lock ...

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

Note the use of `threading.Event` instead of a bare boolean — a plain `bool` flag is not guaranteed to be visible across threads without synchronization primitives.

### 3. `finally` Block Lock Release Has the Same TOCTOU Flaw

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

**Fix — use the same Lua pattern:**

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
self._release_lock = self.redis.register_script(RELEASE_SCRIPT)

# In finally
released = self._release_lock(keys=[lock_key], args=[token])
if not released:
    logger.warning("Lock already expired or stolen at release key=%s", lock_key)
```

## Moderate Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, a path traversal attempt, or absurdly long. `quantity` could be zero or negative.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id, req.quantity from here
```

### 5. `db.update_stock` and `db.create_order` Are Not Transactional

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record — inventory is silently lost. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction should expose a transaction context manager; if it does not, that interface needs to be updated as part of this PR.

### 6. Heartbeat Interval Is Too Close to Lock TTL

`HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` gives a 3x ratio. Under GIL contention, thread scheduling delays, or a Redis round-trip spike, the heartbeat could miss its window. A 5:1 ratio is the conventional minimum safe margin.

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 5  # extend at 5s, 10s, 20s — well within 30s TTL
```

## Minor Issues

### 7. Test Coverage Gaps

- No test for the heartbeat actually extending the lock TTL.
- No test for the `db.create_order` failure / partial write scenario (ties directly to issue #5).
- `test_lock_not_released_if_expired` mocks `redis.get` to return a different token, but the mock's `set` still returns `True`, meaning the test is not exercising a realistic expired-lock scenario end-to-end.
- No test for invalid `item_id` or non-positive `quantity` (ties to issue #4).

### 8. Bare `RuntimeError` Is Too Broad

```python
raise RuntimeError(f"Could not acquire lock...")
```

Define a domain-specific exception so callers can distinguish lock failure from other runtime errors without catching everything:

```python
class LockAcquisitionError(Exception):
    pass
```

The architecture is sound and the intent is right. Address the three critical items at minimum — the TOCTOU races and the shared heartbeat state — before this goes anywhere near a production flash sale.
