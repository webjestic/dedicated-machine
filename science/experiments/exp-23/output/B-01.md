Decision: Request Changes

Summary: This implementation contains a classic **Redlock/fencing token vulnerability** — specifically, the heartbeat mechanism creates a false sense of safety against the exact failure mode it claims to prevent. A stop-the-world GC pause (or any process suspension) can occur *after* the heartbeat confirms ownership but *before* `db.update_stock()` and `db.create_order()` execute; if the lock expires during that window and is acquired by another process, both processes will proceed to write against the database with no conflict detection at the storage layer. The heartbeat reduces the probability of this race but does not eliminate it, and the write operations carry no fencing token or version guard to catch it when it occurs.

---

## Detailed Review

### Critical — Race Condition: Process Pause After Heartbeat, Before Write

This is the core flaw and the reason I cannot approve this PR.

The sequence that breaks the guarantee:

```
Process A                          Process B
---------                          ---------
acquires lock (token=A)
heartbeat confirms ownership ✓
                                   
[STW GC pause — process A suspended for 35s]
                                   
                                   lock TTL expires (30s elapsed)
                                   acquires lock (token=B)
                                   reads stock = 10
                                   writes stock = 9  ✓
                                   
[Process A resumes]
reads stock = 10  ← stale read, already committed by B
writes stock = 9  ← double-decrement, oversell
```

The heartbeat thread is also suspended during a STW pause. It cannot extend the lock it cannot run. The comment `"generous TTL for slow DB operations"` actually makes this *worse* — a 30-second TTL means a 30-second pause window for exploitation, and GC pauses of that duration are not hypothetical in JVM-adjacent runtimes or under memory pressure.

The write operations `db.update_stock()` and `db.create_order()` have no mechanism to detect that lock ownership was lost between the read and the write. The lock protects the *entry* to the critical section but not the *writes within it*.

### Critical — Non-Atomic Check-Then-Act at the Database Layer

```python
current_stock = self.db.get_stock(item_id)
# ... lock can expire here ...
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are three separate database round-trips with no transactional atomicity enforced at the DB layer. Even if the distributed lock were perfectly reliable, a database failure between `update_stock` and `create_order` leaves the system in a partially committed state — stock decremented, no order record. There is no rollback path.

### Significant — Heartbeat Race on Lock Verification

The heartbeat's ownership check is itself a non-atomic read-then-extend:

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # extend
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This should be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same issue applies to the release path in `_acquire_lock`.

### Significant — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal deployment pattern — a singleton service), two concurrent calls will share and clobber `_heartbeat_active` and `_heartbeat_thread`. Call A's `_stop_heartbeat()` will kill Call B's heartbeat thread. This is a straightforward data race.

The fix is to scope heartbeat state to the lock context, not the service instance.

### Moderate — No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, or pathologically large quantities. A caller passing `quantity=-5` would *increase* stock. Pydantic is used only for the response model.

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

### Moderate — Test Suite Mocks Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # ← stop never runs
    return svc
```

The tests mock out the heartbeat entirely, so the concurrency behavior, the instance-level state collision, and the non-atomic heartbeat extension are all untested. The test `test_lock_not_released_if_expired` tests a property of the mock fixture, not of the actual implementation. There are no tests for concurrent invocations on the same instance.

## What a Correct Implementation Requires

1. **Fencing tokens at the write layer.** The database operations must reject writes if the expected version/stock value has changed since the read. An optimistic lock or a conditional update (`UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s`) makes the write self-validating regardless of what happened to the distributed lock.

2. **Atomic Lua scripts for heartbeat and release.** Both the TTL extension and the delete must be atomic GET-then-act operations executed server-side in Redis.

3. **Call-scoped heartbeat state.** Move `_heartbeat_active` and `_heartbeat_thread` into local variables within `_acquire_lock`, not instance variables.

4. **Database transaction wrapping the write pair.** `update_stock` and `create_order` must be a single atomic transaction with a rollback path.

5. **Input validation at the service boundary.** Validate `item_id` and `quantity` before acquiring any lock.
