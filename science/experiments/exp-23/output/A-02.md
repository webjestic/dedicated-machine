Decision: Request Changes

Summary: The heartbeat renewal pattern introduces a critical safety violation under stop-the-world GC pauses or any OS-level thread suspension: the application process can be frozen for longer than `LOCK_TTL` (30s), during which the heartbeat thread is also suspended, the Redis lock expires and can be acquired by another process, and then the original process resumes and proceeds to write — producing a split-brain write scenario with no fencing protection. Beyond that core architectural flaw, there are several additional correctness, safety, and structural issues that must be addressed before this can be approved.

---

## Detailed Findings

### Critical — Stop-the-World / Process Suspension Safety Violation

This is the primary rejection reason.

The heartbeat pattern assumes the renewal thread runs reliably on a wall-clock schedule. It does not. Any of the following can silently suspend **all** threads in the process — including the heartbeat — for longer than `LOCK_TTL`:

- JVM/CPython GC stop-the-world pause (less common in CPython but possible with certain GC extensions or PyPy)
- Linux kernel preemption, cgroup CPU throttling, or a noisy-neighbor VM steal
- Swap pressure causing the process to be paged out
- SIGSTOP from a debugger, container orchestrator, or OOM-adjacent behavior
- Live migration of the underlying VM

The sequence that breaks correctness:

```
T=0s   Process A acquires lock (TTL=30s), heartbeat starts
T=15s  Process A is suspended (GC, swap, throttle — any cause)
         Heartbeat thread is also suspended
T=31s  Redis expires the lock
T=32s  Process B acquires the same lock, reads stock=10, begins write
T=40s  Process A resumes, heartbeat fires, sees its token is gone — does NOT extend
         BUT reserve_item() is already past the stock read at T=14s
         Process A continues: update_stock(item_id, 9), create_order(...)
T=40s  Process B also calls update_stock(item_id, 9), create_order(...)
```

Stock is now decremented twice from the same baseline. Two orders are created for inventory that may not exist.

The token check in `_acquire_lock`'s `finally` block correctly skips the `delete`, but **it does not prevent the writes that already happened inside the `with` block**. The guard is on lock release, not on write safety.

**The heartbeat does not solve this problem — it only narrows the window. It cannot eliminate it.**

The correct mitigation is a **fencing token** (monotonic version/epoch from Redis or a database sequence) that is passed through to every write operation, combined with a conditional write at the database layer that rejects stale writes. Redlock is not sufficient here either; the fencing token pattern (as described by Kleppmann) is the appropriate solution for this class of problem.

```python
# Sketch of the required pattern
fence_token = redis.incr("inventory:fence")  # monotonically increasing

# DB write must be conditional:
db.update_stock_if_fence_valid(item_id, new_stock, fence_token)
# → SQL: UPDATE inventory SET stock=? WHERE item_id=? AND last_fence < ?
```

Without this, no amount of heartbeat tuning makes the system safe.

### High — Shared Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to an individual lock acquisition. If two callers use the same `InventoryService` instance concurrently (common in a thread-pool web server), one caller's `_stop_heartbeat()` will terminate the other caller's renewal thread.

```python
# Thread 1 acquires lock for item_001, starts heartbeat
# Thread 2 acquires lock for item_002, overwrites self._heartbeat_active = True
# Thread 1 calls _stop_heartbeat() → sets self._heartbeat_active = False
# Thread 2's heartbeat is now dead; its lock will expire
```

Heartbeat state must be local to each lock acquisition, not stored on `self`.

```python
# Corrected pattern — state is closure-local
def _start_heartbeat(self, lock_key, token):
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return active, t  # caller holds references, not self
```

### High — Non-Atomic Check-Then-Act on Lock Release

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a TOCTOU race. Between `GET` and `DELETE`, the lock can expire and be acquired by another process. The `DELETE` then releases a lock the current process no longer owns.

This must be an atomic Lua script, which is the standard Redis pattern:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
release = self.redis.register_script(RELEASE_SCRIPT)
release(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's `GET` + `expire` sequence and must be fixed identically.

### High — No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input path. An attacker or misconfigured caller can pass:

- `item_id = "inventory:lock:admin"` — crafted to collide with the lock keyspace
- `quantity = -5` — negative reservation decrements stock upward, creating phantom inventory
- `quantity = 0` — creates a no-op order record
- `item_id` of arbitrary length — no bound on Redis key size

Minimum required:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

The lock key must be constructed from a validated, sanitized `item_id`.

---

### Medium — Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction boundary. If the process crashes, is killed, or the DB connection drops between them, stock is decremented but no order exists — inventory is silently lost with no compensating mechanism.

```python
# Required
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### Medium — Heartbeat `time.sleep` Is Not Interruptible

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10s, ignores stop signal
    if self._heartbeat_active:
        ...
```

`_stop_heartbeat()` sets the flag but the thread sleeps for up to 10 seconds before checking it. On service shutdown or rapid lock cycling, this creates thread accumulation. Use `threading.Event.wait(timeout=...)` so the thread wakes immediately on stop.

---

### Medium — Test Suite Mocks Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally significant component of this PR and it has zero test coverage. There are also no tests for:

- Concurrent acquisition on the same `InventoryService` instance
- Behavior when the DB transaction fails mid-way
- The `GET`/`DELETE` race on lock release
- Negative or zero quantity inputs
- Oversized or injection-crafted `item_id` values

---

### Low — Logger Instantiated at Module Level Without Hierarchy

`logger = logging.getLogger(__name__)` is fine for a library but the service has no guidance on log level configuration. In a government contracting context, log output containing `order_id`, `item_id`, and stock levels may be subject to audit trail and data handling requirements. Ensure log destinations and retention are reviewed against applicable compliance controls (FedRAMP, NIST 800-53 AU controls) before this service is deployed.

---

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| Critical | Implement fencing tokens for write safety after process resumption |
| High | Scope heartbeat state to each lock acquisition, not to `self` |
| High | Replace GET+DELETE with atomic Lua script |
| High | Add and enforce input validation model on `reserve_item` |
| Medium | Wrap DB writes in a single transaction |
| Medium | Replace `time.sleep` in heartbeat with `Event.wait` |
| Medium | Add tests for heartbeat, concurrency, DB failure, and input edge cases |
