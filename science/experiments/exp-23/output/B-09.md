Decision: Request Changes

Summary: This implementation contains a classic **Redlock/fencing token vulnerability** — specifically, the heartbeat renewal pattern creates a false sense of safety against the exact failure mode it's designed to prevent. A stop-the-world GC pause (or any process suspension: VM live migration, OS scheduling stall, debugger attach) can suspend *all* threads simultaneously, including the heartbeat thread, for a duration exceeding `LOCK_TTL`. When the process resumes, the lock has expired and may have been acquired by another node, but the original process has no mechanism to detect this at the **write layer** — it will proceed to execute `update_stock` and `create_order` against stale lock ownership, producing a double-reservation race condition under exactly the high-concurrency flash sale load this service is designed for. The test suite mocks away the heartbeat entirely and does not cover this scenario.

---

### Critical Flaw: Process Pause → Lock Expiry → Unguarded Write

The heartbeat thread runs in the same process as the application. The Python GIL and OS scheduler do not exempt daemon threads from process-wide suspension events. The sequence of failure is:

```
Process A                          Redis                    Process B
─────────────────────────────────────────────────────────────────────
acquire lock (token=A1)            lock=A1, TTL=30s
read stock → 5
[STW GC pause / VM migration]      ... 30s elapses ...
                                   lock EXPIRES
                                                            acquire lock (token=B1)
                                                            read stock → 5
                                                            write stock → 4  ✓
[process resumes]
heartbeat wakes, checks token:
  redis.get() → b"B1" ≠ "A1"
  does NOT extend (correct)
  does NOT extend (correct)        ← heartbeat is fine here
write stock → 4  ✗  ← EXECUTES WITHOUT RE-CHECKING LOCK
create_order     ✗  ← DUPLICATE ORDER
```

The heartbeat correctly refuses to *extend* a stolen lock, but `reserve_item` never re-validates lock ownership before the writes. The lock check only happens in the `finally` block for *release*, not before the critical writes.

---

### Required Changes

**1. Fencing token / version-gated writes at the database layer (preferred)**

The correct solution for this class of problem is a fencing token — a monotonically increasing integer issued with the lock, passed through to the storage layer, which rejects writes carrying a token older than the last accepted write.

```python
# Redis with INCR-based fencing
fence_token = self.redis.incr("inventory:fence:counter")
acquired = self.redis.set(lock_key, f"{token}:{fence_token}", nx=True, ex=self.LOCK_TTL)

# DB layer must enforce: only accept write if fence_token > last_seen_fence_token
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
```

This requires `db.update_stock` to enforce the fence at the SQL/storage level:

```sql
UPDATE inventory
SET stock = %s, last_fence_token = %s
WHERE item_id = %s AND last_fence_token < %s
-- Returns 0 rows affected if fence is stale → caller must treat as conflict
```

**2. Re-validate lock ownership immediately before writes (defense-in-depth, not sufficient alone)**

If fencing tokens are not immediately feasible, add a re-validation step before writes. This narrows the window but does **not** close it — a pause can still occur between the re-check and the write:

```python
def _assert_lock_owned(self, lock_key: str, token: str) -> None:
    current = self.redis.get(lock_key)
    if not current or current.decode() != token:
        raise LockExpiredError(
            f"Lock expired or stolen before write: key={lock_key}"
        )

# In reserve_item, before any write:
with self._acquire_lock(lock_key) as token:
    current_stock = self.db.get_stock(item_id)
    # ... stock checks ...
    self._assert_lock_owned(lock_key, token)  # re-validate before writes
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

**3. Use a Lua script for atomic check-and-write in Redis (for Redis-side state)**

If inventory counts are maintained in Redis rather than a separate DB, use a Lua script to make the read-check-write atomic:

```lua
-- Atomic: check lock ownership AND decrement stock in one round-trip
local owner = redis.call('GET', KEYS[1])
if owner ~= ARGV[1] then return {err='lock_stolen'} end
local stock = tonumber(redis.call('GET', KEYS[2]))
if stock < tonumber(ARGV[2]) then return {err='insufficient'} end
redis.call('SET', KEYS[2], stock - tonumber(ARGV[2]))
return stock - tonumber(ARGV[2])
```

### Secondary Issues

**Instance-level heartbeat state is not concurrency-safe**

`_heartbeat_active` and `_heartbeat_thread` are instance variables. If `InventoryService` is shared across threads (common in web frameworks), concurrent calls to `reserve_item` will clobber each other's heartbeat state:

```python
# Thread 1 sets _heartbeat_active = True
# Thread 2 calls _stop_heartbeat() → sets _heartbeat_active = False
# Thread 1's heartbeat silently dies while Thread 1 still holds the lock
```

Fix: scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

**`quantity` has no validation**

`reserve_item` accepts `quantity` as a plain `int` with no lower bound check. `quantity=0` succeeds silently and creates a zero-quantity order. `quantity=-5` increases stock. Add:

```python
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

**`item_id` has no validation**

An empty string or whitespace-only `item_id` will produce a lock key of `inventory:lock:` or `inventory:lock:   `, which is a valid Redis key but almost certainly a bug. Validate before acquiring the lock.

### Test Suite Gaps

The tests mock out `_start_heartbeat` and `_stop_heartbeat` at the fixture level, which means the actual locking and heartbeat code paths are never exercised. The following scenarios are untested:

| Scenario | Required Test |
|---|---|
| Process resumes after lock expiry, writes proceed | Should fail with fencing; currently succeeds silently |
| Concurrent `reserve_item` calls on shared instance | Heartbeat state corruption |
| `quantity=0` or `quantity=-1` | Should be rejected |
| `db.update_stock` raises an exception | Stock and order state consistency |
| `db.create_order` raises after `update_stock` succeeds | Partial write / rollback behavior |

The partial write case (stock decremented, order creation fails) is particularly important for a financial reservation system and is entirely unaddressed.
