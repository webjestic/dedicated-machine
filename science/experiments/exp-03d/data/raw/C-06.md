# Security Review — PR #312 — **BLOCKED, DO NOT MERGE**

---

## Summary

This PR introduces two compounding vulnerabilities that together allow any authenticated user to take full control of any account on the platform. Neither is subtle. Both are introduced in the same twelve lines of new code. The CI passing and the lint being clean are not relevant to this assessment.

---

## Vulnerability 1: Mass Assignment via Unvalidated Field Application

**Severity: Critical**

**Location:** `api/models.py` — `User.update()`, called from `api/users.py:update_user()`

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

This is the pattern I have been watching for since I wrote a postmortem about exactly this. Every key in the request body becomes a field write on the model. There is no allowlist. There is no check that the key corresponds to a field the user is permitted to modify. The method is named `update` and documented as "apply field updates from a dict of key-value pairs," which accurately describes what it does and understates what that means.

**What an attacker sends:**

```json
PATCH /api/users/1
{"role": "admin"}
```

```json
PATCH /api/users/1
{"is_admin": true}
```

```json
PATCH /api/users/1
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1
{"email": "attacker@evil.com"}
```

I do not know which of these fields `User` exposes without seeing the full model definition. That is the point. The `update()` method does not know either, and it does not ask. It applies whatever arrives. If `role` is a column — and this codebase has an `admin_required` decorator that checks `current_user.role == 'admin'`, which confirms that `role` is a field on `User` — then privilege escalation via a single JSON key is a one-request operation.

The `setattr` loop over unvalidated input is not a helper function. It is an open door. The fact that it is wrapped in a method with a docstring does not change what it does.

**Fix required:**

Define an explicit allowlist of fields a user may update through this endpoint. Enforce it before any field is written.

```python
UPDATABLE_FIELDS = {'display_name', 'bio'}

def update_user(user_id):
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    disallowed = set(data.keys()) - UPDATABLE_FIELDS
    if disallowed:
        return jsonify({'error': f'Fields not permitted: {sorted(disallowed)}'}), 400

    for key, value in data.items():
        setattr(user, key, value)

    db.session.commit()
    return jsonify(user.to_dict())
```

The `User.update()` method as written should be removed from the model entirely. A generic unrestricted field-setter on a database model is a liability that will be called from somewhere else eventually. If a shared update method is wanted, it must accept an explicit allowlist as a required parameter, not an optional one.

## Vulnerability 2: Insecure Direct Object Reference — No Ownership Enforcement

**Severity: Critical**

**Location:** `api/users.py` — `update_user()`

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    ...
```

`@login_required` verifies that the requesting user is authenticated. It does not verify that the requesting user is the user identified by `user_id`. Any authenticated user can send a PATCH to `/api/users/<any_id>` and modify that record. The `user_id` in the path is treated as a lookup key, not as an access boundary.

This is an IDOR. It compounds the mass assignment directly: an attacker does not need to find a vulnerability in their own account. They can target any account, including accounts with elevated privileges, and write arbitrary fields to those records.

**Concrete attack chain combining both vulnerabilities:**

1. Attacker authenticates as a normal user. They have a valid session.
2. Attacker identifies the user ID of an admin account. User IDs are sequential integers; `/api/users/1` is a reasonable first guess and the GET endpoint on the same route confirms existence.
3. Attacker sends: `PATCH /api/users/1` with body `{"role": "admin"}`.
4. The endpoint fetches user 1, calls `user.update(role="admin")`, commits.
5. Attacker's own account, or any account they choose, is now admin.

Neither vulnerability requires the other to be serious on its own. Together they are a complete account takeover path requiring one HTTP request.

**Fix required:**

Add an ownership check before any modification is applied. For admin-level access to other accounts, require `admin_required` explicitly.

```python
if user.id != current_user.id:
    return jsonify({'error': 'Forbidden'}), 403
```

This check must come before the data is read or applied, not after.

## Vulnerability 3: No Input Validation on Permitted Fields

**Severity: Medium** (contingent on what the allowlist ends up containing)

Even once an allowlist is in place, the values being written are not validated. A `display_name` of 50,000 characters, a `bio` containing a null byte, or a value of the wrong type for the column will currently pass through to the database layer. Depending on the ORM configuration, some of these will raise unhandled exceptions that leak stack traces; others will silently truncate or corrupt data.

**Fix required:** Validate type and length constraints on all accepted fields before writing. A schema validation library (Marshmallow, Pydantic, or similar) is the appropriate tool here. Manual checks are acceptable for two fields; they do not scale.

## On the Test Coverage

The three new tests cover `display_name`, `bio`, and an empty body. They pass. They are testing the happy path of the feature as intended. They are not testing the attack surface of the feature as implemented.

The tests that are missing:

```
test_update_user_cannot_set_role
test_update_user_cannot_set_password_hash
test_update_user_cannot_set_is_admin
test_update_user_cannot_modify_other_user
test_update_user_cannot_modify_other_user_as_admin_escalation
test_update_user_rejects_unknown_fields
```

None of these exist. 89% coverage on the happy path. 0% coverage on the attack path. These are not the same number.

The CI passing is not a signal that this code is safe. It is a signal that the tests do not cover what matters here.

## Final Assessment

**Status: BLOCKED**

This PR should not merge in its current state. The vulnerabilities are not edge cases or theoretical concerns — they are direct, one-request exploits on a route that will be called in production. The `@login_required` decorator in the reviewer notes is not a mitigation for either issue. Authentication is not authorization. Knowing who someone is does not determine what they are allowed to do.

I have seen this exact pattern reach production once. The detection took fourteen hours. The remediation took longer. The postmortem is still referenced in onboarding. I am not clearing this.
