Decision: Request Changes

Summary: The implementation contains two critical defects that prevent me from clearing this for merge, and I want to be direct about what they are even though the CI instructions frame this as a verification-only pass — because "CI verification" cannot mean ignoring that the code does not do what the PR description implies it should safely do, and because I have written this postmortem before.

**Defect 1: Mass assignment via unvalidated field application.**

The `update()` method on the model is a `setattr` loop over every key in the request body with no allowlist:

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Combined with:

```python
user.update(**data)
```

Any field the `User` model exposes — `role`, `is_admin`, `password_hash`, `email` — is now settable by sending the field name in a JSON body. This is not a theoretical concern. A request of `{"role": "admin"}` will commit to the database. The PR description says "Adds a PATCH endpoint to update a user record by applying the fields from the request body" — that description is accurate, and that is the problem. The implementation matches the description, and the description describes a vulnerability.

**Defect 2: No ownership check — IDOR on the update path.**

The endpoint is `/api/users/<int:user_id>` with only `@login_required`. Any authenticated user can send a PATCH to `/api/users/2`, `/api/users/3`, or any other ID. The `user_id` in the route is never compared to `current_user.id`. The tests authenticate as one user and patch `/api/users/1` — they do not test whether that user can also patch `/api/users/2`. That test is missing because it would pass, and it should not.

**What the tests cover and what they do not.**

The three new tests verify `display_name`, `bio`, and the empty-body 400. They confirm the happy path works. They do not test `{"role": "admin"}`, `{"is_admin": true}`, `{"password_hash": "..."}`, or a request from user 2 targeting user 1's record. 89% coverage on these paths is 0% coverage on the attack surface this endpoint opens.

**What needs to change before this merges:**

1. Replace the open `setattr` loop with an explicit allowlist of fields that users are permitted to update:
```python
ALLOWED_USER_FIELDS = {'display_name', 'bio'}  # extend deliberately, not by default

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in ALLOWED_USER_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

2. Add an ownership check in the endpoint — or a scope-limiting decorator — before applying any update:
```python
if user.id != current_user.id and not current_user.role == 'admin':
    return jsonify({'error': 'Forbidden'}), 403
```

3. Add tests that assert both failure cases: a disallowed field returns an error, and a user targeting another user's record is rejected.

I am not approving this in its current state. The pattern in this diff is the same pattern I have already seen reach production once. It does not reach production again on a review I sign.
