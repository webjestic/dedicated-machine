**Reviewer:** Security Review

## What This PR Actually Ships

The reviewer notes say "`@login_required` is in place" as if that closes the security question. It does not. It means an attacker needs a valid session. Once they have one — and sessions are not hard to obtain on any platform with open registration — both vulnerabilities below are fully exploitable by any authenticated user.

---

## Critical Finding 1: Mass Assignment via Unvalidated Field Application

**Location:** `api/models.py` — `update()` method; `api/users.py` — `update_user()`  
**Severity:** Critical  
**CVSS equivalent:** 9.8

```python
# api/models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on the model. no exceptions.
```

```python
# api/users.py
data = request.get_json() or {}
user.update(**data)  # the entire request body, unsifted, passed directly in
```

This is the pattern. `request.get_json()` returns whatever the client sends. That dict is unpacked and passed to `update()`. `update()` calls `setattr` on every key without consulting any list of permitted fields. The User model has a `role` field — I can see that from `admin_required`, which checks `current_user.role == 'admin'`. That field is now settable by any authenticated user with a single request:

```bash
curl -X PATCH https://obotix.one/api/users/1 \
  -H "Content-Type: application/json" \
  -H "Cookie: session=<any valid session>" \
  -d '{"role": "admin"}'
```

I know what that looks like in production because I have seen it in production. It took fourteen hours to find. The postmortem I wrote after that incident is the reason I am writing this review now.

Other fields that are almost certainly on this model and are equally exposed: `password_hash`, `email`, `is_active`, `is_verified`, `created_at`. The `update()` method does not know or care. It will set any of them.

The docstring calls this "Apply field updates from a dict of key-value pairs." That is an accurate description of the problem, not a description of a safe helper function.

## Critical Finding 2: Insecure Direct Object Reference (IDOR)

**Location:** `api/users.py` — `update_user()`  
**Severity:** Critical

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    # current_user is never consulted again
    user.update(**data)
```

The endpoint accepts a `user_id` from the URL. It verifies the caller is logged in. It does not verify the caller *is* that user or has any relationship to that user. Any authenticated session can target any `user_id` in the database.

Combined with Finding 1, the full exploit is:

```bash
# Authenticated as user 47. Targeting user 1 (or any other account).
curl -X PATCH https://obotix.one/api/users/1 \
  -d '{"role": "admin", "email": "attacker@evil.com"}'
```

These two vulnerabilities compound each other. The IDOR means the attacker is not limited to their own account. The mass assignment means they are not limited to safe fields on any account they can reach. Together they constitute full account takeover of any user and privilege escalation to admin in a single unauthenticated-to-admin step.

## On the Test Coverage

The CI badge says ✅ 18 tests passed. That is not a signal of safety here. It is a signal of what was not tested.

```python
def test_update_user_display_name(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'display_name': 'Alice Smith'}, ...)
    assert rv.status_code == 200

def test_update_user_bio(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'bio': 'Backend engineer at obotix'}, ...)
    assert rv.status_code == 200
```

These tests confirm that the happy path works. They exercise exactly the fields that are safe to set and none of the fields that are not. They do not test:

- `PATCH /api/users/1` with `{"role": "admin"}` — should be rejected, is not
- `PATCH /api/users/2` authenticated as user 1 — should be rejected, is not
- `PATCH /api/users/1` with `{"password_hash": "..."}` — should be rejected, is not
- `PATCH /api/users/1` with `{"email": "attacker@x.com"}` — should be rejected, is not

100% coverage of the happy path is 0% coverage of the attack path. The three new tests tell me the implementation works when used correctly. They say nothing about what happens when it is used maliciously.

## Required Changes Before This Can Be Merged

### 1. Introduce a field allowlist — no exceptions

```python
# api/models.py

UPDATABLE_FIELDS = {'display_name', 'bio'}  # explicit, minimal, reviewed

def update(self, data: dict):
    """Apply only allowlisted fields from caller-supplied data."""
    for key in UPDATABLE_FIELDS:
        if key in data:
            setattr(self, key, data[key])
```

The allowlist lives in the model. It is the single authoritative statement of what is user-settable. Adding a field to the model does not automatically make it settable — a developer has to make a conscious decision to add it to `UPDATABLE_FIELDS`. That decision is visible in code review. The current implementation has no such checkpoint.

Do not make this allowlist a parameter. Do not accept it from the caller. It is not configurable at runtime.

### 2. Scope the operation to the requesting user

```python
# api/users.py

@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400
    user.update(data)
    db.session.commit()
    return jsonify(user.to_dict())
```

The authorization check happens before the database query. If the caller is not the target user, they get a 403 and the query never runs. Admin override paths, if they exist, go through `admin_required` on a separate endpoint with its own audit trail.

### 3. Write tests that cover the attack surface

The following tests are required, not optional:

```python
def test_update_role_is_rejected(client, seed_users, user_headers):
    """Mass assignment of privileged field must be silently ignored."""
    rv = client.patch(
        '/api/users/1',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 200
    assert rv.get_json()['role'] != 'admin'


def test_update_other_user_is_forbidden(client, seed_users, user_headers):
    """Authenticated user cannot modify another user's record."""
    rv = client.patch(
        '/api/users/2',  # user_headers authenticates as user 1
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403


def test_update_password_hash_is_rejected(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1',
        json={'password_hash': 'aaaa'},
        headers=user_headers
    )
    assert rv.status_code == 200
    # verify the hash was not changed by querying the db directly
    user = User.query.get(1)
    assert user.password_hash != 'aaaa'


def test_update_email_requires_separate_flow(client, seed_users, user_headers):
    """Email changes are sensitive and must not go through this endpoint."""
    rv = client.patch(
        '/api/users/1',
        json={'email': 'hijacked@evil.com'},
        headers=user_headers
    )
    assert rv.status_code == 200
    user = User.query.get(1)
    assert user.email != 'hijacked@evil.com'
```

If these tests pass after the fix, the fix is real. If they cannot be written because the framework makes it hard to assert that a field was *not* changed, that is a signal that the model layer needs better introspection, not that the tests should be skipped.

## What I Need From the Author

1. Replace `update(**kwargs)` with `update(data)` using an explicit `UPDATABLE_FIELDS` allowlist in the model
2. Add the ownership check in the endpoint before the query
3. Add the tests listed above — all of them
4. Confirm whether `email` changes are handled elsewhere with appropriate verification flow (confirmation email, re-auth, etc.) — if not, that is a separate ticket that needs to be opened before this merges

The implementation as written is not a draft with rough edges. It contains the exact pattern that has caused a real incident. The fix is not complex. The allowlist is four lines. The ownership check is three lines. The tests are straightforward. None of this is a large ask relative to the risk.

This does not merge until those changes are in and re-reviewed.
