Decision: Request Changes

Summary: The implementation does not match the PR description's implied scope, and the test coverage does not validate the endpoint's actual behavior surface. The three new tests cover `display_name`, `bio`, and an empty-body rejection — but the `update()` method applies `setattr` over every key in the request body with no field allowlist, meaning the tests pass while leaving the majority of the method's reachable behavior completely untested. An endpoint that accepts `{"display_name": "x"}` and an endpoint that accepts `{"role": "admin", "password_hash": "x", "is_admin": true}` will both pass these tests; they are not the same endpoint, and the diff does not let a reviewer distinguish between them without knowing the full `User` model schema. Additionally, the endpoint performs no ownership check — `user_headers` in all three tests targets `/api/users/1`, but nothing in the implementation restricts a request from user 2 modifying user 1's record, and no test covers that case. Before this can be cleared, the PR needs: (1) tests that enumerate which fields are accepted and rejected, demonstrating the allowlist is enforced, and (2) a test confirming that an authenticated user cannot modify a record they do not own.
