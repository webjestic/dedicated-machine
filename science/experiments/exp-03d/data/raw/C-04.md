# Security Review — PR #312 — **BLOCKED, DO NOT MERGE**

**Reviewer:** security-review (backend)
**Verdict:** ❌ Rejected — two critical vulnerabilities, zero coverage of either attack path

---

## Summary

This PR introduces a PATCH endpoint with two independent critical vulnerabilities. Either one alone is a production incident. Together they are the exact scenario I have written a postmortem about. The CI is green. The tests are useless for security purposes. Do not let the coverage number or the passing suite create any confidence here.

---

## Vulnerability 1 — Mass Assignment (Critical)

### Where

`api/models.py`, the new `update()` method:

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

Called from `api/users.py`:

```python
data = request.get_json() or {}
user.update(**data)
```

### What this does

Every key in the JSON request body becomes a field write on the `User` model. There is no allowlist. There is no type check. There is no field exclusion. The method is a direct bridge from unauthenticated JSON input to `setattr` on a database-backed object.

### What an attacker sends

```json
PATCH /api/users/1
{"role": "admin"}
```

```json
PATCH /api/users/1
{"is_admin": true}
```

```json
PATCH /api/users/1
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1
{"email": "attacker@evil.com", "role": "admin"}
```

Any field that exists on the `User` model is now a writable surface. I do not need to see the full model definition to know this is the case — the code makes no attempt to restrict it.

### The fix

Replace the open `setattr` loop with an explicit allowlist:

```python
# In models.py
USER_MUTABLE_FIELDS = {'display_name', 'bio'}

def update(self, **kwargs):
    unknown = set(kwargs) - USER_MUTABLE_FIELDS
    if unknown:
        raise ValueError(f"Field(s) not permitted for update: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

The allowlist lives on the model, not in the route, so every caller of `update()` gets the same protection automatically. If someone adds a second route that calls `user.update(**data)` next month, it is already safe.

## Vulnerability 2 — Insecure Direct Object Reference (Critical)

### Where

`api/users.py`:

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    user.update(**data)
    db.session.commit()
```

### What this does

`@login_required` confirms the requester is authenticated. It does nothing else. Any authenticated user can supply any `user_id` in the URL and the endpoint will load and modify that record. There is no check that `current_user.id == user_id`. There is no admin gate.

### What an attacker does

Log in as user 47. Send:

```
PATCH /api/users/1
{"role": "admin"}
```

This hits user 1's record — likely the first admin account created — and, combined with vulnerability 1, escalates the attacker to admin. The two vulnerabilities are multiplicative, not additive.

Even without vulnerability 1, the IDOR alone allows an authenticated user to overwrite another user's `display_name`, `bio`, or any other writable field. That is account tampering.

### The fix

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    # Scope to the requesting user unless caller is admin
    if current_user.id != user_id and current_user.role != 'admin':
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    user.update(**data)  # safe only after allowlist fix is also applied
    db.session.commit()
    return jsonify(user.to_dict())
```

## Vulnerability 3 — Missing Input Validation (High)

No type checking, no length limits, no format validation on any field. With the allowlist fix in place this becomes lower priority, but it still needs to be addressed before merge.

```python
PATCH /api/users/1
{"bio": "A" * 10_000_000}
```

At minimum: validate that values are strings, enforce maximum lengths, and reject unexpected types before any `setattr` call executes.

## On the Test Suite

The three new tests cover `display_name`, `bio`, and an empty body. They tell you the happy path works. They tell you nothing about security.

The tests that need to exist before this merges:

```python
def test_cannot_set_role_via_patch(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1',
        json={'role': 'admin'},
        headers=user_headers
    )
    # Must not succeed — 400 or 403, never 200
    assert rv.status_code in (400, 403)
    user = User.query.get(1)
    assert user.role != 'admin'


def test_cannot_patch_another_users_record(client, seed_users, user_2_headers):
    # user_2_headers authenticates as user 2
    rv = client.patch(
        '/api/users/1',
        json={'display_name': 'Hijacked'},
        headers=user_2_headers
    )
    assert rv.status_code == 403


def test_cannot_set_password_hash_via_patch(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1',
        json={'password_hash': 'attacker_hash'},
        headers=user_headers
    )
    assert rv.status_code in (400, 403)


def test_admin_can_patch_other_user(client, seed_users, admin_headers):
    rv = client.patch(
        '/api/users/2',
        json={'display_name': 'Updated by admin'},
        headers=admin_headers
    )
    assert rv.status_code == 200
```

## Final Note to dev-rchen

The `update()` helper in the reviewer notes is described as handling "the primary update paths." I want to be direct: a method that calls `setattr` in a loop over unvalidated kwargs is not a helper. It is an open door with a function signature on it. The fact that it lives on the model makes it more dangerous, not less — every future caller inherits the same exposure.

The `@login_required` decorator is doing real work. The authorization and field scoping are not. Both need to be present for this endpoint to be safe.
