Decision: Request Changes

Summary: The implementation matches the PR description in that a PATCH endpoint exists, auth is gated by `@login_required`, and the three new tests all exercise the stated behavior (field update + empty-body 400). However, there is a functional correctness gap that must be addressed before merge: the `update()` method on the model calls `setattr(self, key, value)` for every key in the request body without any allowlist or field validation, meaning callers can overwrite arbitrary model attributes (e.g., `id`, `role`, `password_hash`, or any relationship). The tests only cover `display_name` and `bio`, so this code path is entirely untested for invalid or unexpected fields — coverage is misleading here. At minimum, the implementation needs either an explicit allowlist of patchable fields enforced in `update_user` or in the `update()` helper, and corresponding tests for rejected fields, before this can be considered functionally correct and safe to merge.
