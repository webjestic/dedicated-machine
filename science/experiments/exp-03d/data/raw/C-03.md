## Critical Finding 1: Mass Assignment via Unvalidated Field Application

**Severity:** Critical
**Location:** `api/models.py` — `User.update()`, called from `api/users.py:update_user()`

```python
# This is not a helper. This is an open door.
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

This loop applies every key in the request body directly to the model object with no field allowlist, no type validation, and no awareness of what fields the `User` model exposes. The caller passes `request.get_json()` straight through:

```python
data = request.get_json() or {}
user.update(**data)
```

The path from the network to `setattr` is unobstructed. Whatever the `User` model has as attributes, a caller can set. Based on the codebase I can already see — `role` is a field. The `admin_required` decorator checks `current_user.role == 'admin'`. That field is now writable via a PATCH body.

**Proof of concept — privilege escalation:**
```http
PATCH /api/users/1 HTTP/1.1
Authorization: Bearer <any valid user token>
Content-Type: application/json

{"role": "admin"}
```

Response: `200 OK`. The user is now admin. Every admin-gated endpoint in this application is now accessible to them.

The likely additional fields exposed by a standard `User` model that are now writable:

| Field | Impact |
|---|---|
| `role` | Full privilege escalation to admin |
| `is_admin` / `is_staff` | Same, if these columns exist |
| `email` | Account takeover via password reset to attacker-controlled address |
| `password_hash` | Direct credential replacement |
| `is_active` / `is_verified` | Bypass email verification, lock out other accounts |
| `id` | Potential record collision or orphaned foreign keys |
| `created_at` | Data integrity |

I do not need to know which of these exist. The correct answer is not "audit which fields are dangerous." The correct answer is an explicit allowlist of fields that are safe to update. Anything not on that list is rejected before it reaches the model.

**Required fix:**

```python
# In User model or a dedicated schema layer
UPDATABLE_FIELDS = {'display_name', 'bio'}  # explicit, reviewed, minimal

def update(self, **kwargs):
    unknown = set(kwargs) - UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Fields not permitted for update: {unknown}")
    for key, value in kwargs.items():
        # UPDATABLE_FIELDS membership is already confirmed above
        setattr(self, key, value)
```

Or, preferably, use a validation schema (Marshmallow, Pydantic) that deserializes only declared fields and rejects everything else at the boundary, before the data ever reaches model code. The schema is the contract. The model should not be making trust decisions about field names.

## Critical Finding 2: Insecure Direct Object Reference — No Ownership Scoping

**Severity:** Critical
**Location:** `api/users.py` — `update_user()`

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    # user_id comes from the URL
    # current_user is never consulted again
    # any authenticated user reaches db.session.commit()
```

`@login_required` confirms a valid session exists. It does not confirm that the session belongs to the user being modified. After authentication passes, `user_id` is taken from the URL and the record is fetched and updated without any check that `user_id == current_user.id` or that `current_user` holds an admin role.

**Proof of concept — IDOR:**
```http
PATCH /api/users/47 HTTP/1.1
Authorization: Bearer <token for user id=12>
Content-Type: application/json

{"bio": "account hijacked"}
```

User 12 has just modified user 47's record. `@login_required` passed. No error was raised.

Combined with Finding 1, the full attack surface is: any authenticated user can set any field on any user record. The two vulnerabilities are multiplicative, not additive.

**Required fix:**

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    # Scope to the requesting user unless caller is admin
    if current_user.id != user_id and current_user.role != 'admin':
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    # ... validated field update follows
```

The ownership check must come before the database fetch. Returning 404 instead of 403 for records the user does not own is an acceptable hardening measure to avoid confirming whether a user ID exists, depending on your threat model.

## On the Test Coverage

The three new tests cover `display_name`, `bio`, and an empty body. They all pass. They tell you nothing about the attack surface.

```python
# Not tested. Should be.

def test_cannot_escalate_role(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400  # or 403 — not 200

def test_cannot_update_other_users_record(client, seed_users, user_headers):
    # user_headers authenticates user id=2
    rv = client.patch('/api/users/1', json={'bio': 'hijacked'}, headers=user_headers)
    assert rv.status_code == 403

def test_cannot_set_password_hash(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'password_hash': 'attacker_hash'}, headers=user_headers)
    assert rv.status_code == 400

def test_unknown_fields_are_rejected(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'nonexistent_field': 'x'}, headers=user_headers)
    assert rv.status_code == 400
```

18 tests passing and 89% coverage is not a signal that this code is safe. It is a signal that the tests were written to confirm the happy path works. The attack paths were not written and were not run. Coverage does not measure what you did not test.

## Summary

|---|---|---|
| Mass assignment — no field allowlist | Critical | ❌ Blocks merge |
| IDOR — no ownership check on `user_id` | Critical | ❌ Blocks merge |

This PR cannot be merged in its current state. The `update()` method as written should not exist in this codebase in any form that accepts arbitrary kwargs from request data. The endpoint as written should not exist without an ownership check.

Fix the allowlist. Fix the ownership scope. Add tests that assert the rejections, not just the acceptances. Then resubmit for review.
