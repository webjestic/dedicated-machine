# Security Review — PR #312 — REJECTED

**Reviewer:** Security Review (Backend)
**Verdict:** ❌ **Do not merge. Two critical vulnerabilities. No exceptions.**

## What This PR Actually Ships

The reviewer notes say "`@login_required` is in place" as if that closes the security question. It does not. It means an authenticated user can reach this endpoint. What they can do once they get there is the entire problem, and the PR does not address it.

### Vulnerability 1: Mass Assignment (Critical)

```python
# api/models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

```python
# api/users.py
data = request.get_json() or {}
user.update(**data)
```

This is a direct, unmediated path from JSON request body to `setattr` on the model. Every column on the `User` model is now a writable API parameter. There is no allowlist. There is no field filtering. There is nothing between the user's input and the database row.

A request like this will work:

```http
PATCH /api/users/1 HTTP/1.1
Content-Type: application/json

{"role": "admin"}
```

```http
PATCH /api/users/1 HTTP/1.1
Content-Type: application/json

{"is_admin": true, "is_staff": true}
```

```http
PATCH /api/users/1 HTTP/1.1
Content-Type: application/json

{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```http
PATCH /api/users/1 HTTP/1.1
Content-Type: application/json

{"email": "attacker@evil.com"}
```

I know what this looks like in production because I have seen it in production. It took fourteen hours to detect. The postmortem is on file. This is that pattern, verbatim.

The `update()` method is named and documented as a generic helper. That framing makes it feel like infrastructure. It is not. It is an open `setattr` loop over unvalidated caller input, and it will be reused elsewhere in this codebase because it looks like a utility function. That compounds the risk beyond this single endpoint.

### Vulnerability 2: Insecure Direct Object Reference (Critical)

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    # No check that current_user.id == user_id
    # No admin_required decorator
    data = request.get_json() or {}
    user.update(**data)
    db.session.commit()
```

Any authenticated user can patch any other user's record. The `user_id` in the URL is not validated against the requesting user's identity. `admin_required` exists in this file and is not applied here.

These two vulnerabilities compose. An authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>` — their own account, another user's account, or the primary admin account — and it will succeed. This is not a theoretical attack chain. It is four lines of curl.

## Why the Tests Do Not Catch This

```python
def test_update_user_display_name(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'display_name': 'Alice Smith'}, headers=user_headers)
    assert rv.status_code == 200

def test_update_user_bio(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'bio': 'Backend engineer at obotix'}, headers=user_headers)
    assert rv.status_code == 200
```

These tests confirm that the happy path works. They test two fields that happen to be safe. They do not test what happens when a caller sends `role`, `is_admin`, `password_hash`, or `email`. They do not test whether user A can modify user B's record. The CI is green because the tests were written to pass, not to probe the attack surface.

18 tests passing and 89% coverage means nothing here. Coverage measures which lines execute, not which inputs were tried. The attack paths are not in the test suite.

## Required Changes Before This Can Be Reviewed Again

### 1. Enforce field allowlist — no exceptions

```python
# In the endpoint, not in a shared model method
UPDATABLE_FIELDS = {'display_name', 'bio'}

@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    filtered = {k: v for k, v in data.items() if k in UPDATABLE_FIELDS}
    if not filtered:
        return jsonify({'error': 'No updatable fields provided'}), 400

    for key, value in filtered.items():
        setattr(user, key, value)

    db.session.commit()
    return jsonify(user.to_dict())
```

The allowlist lives at the endpoint level, not inside a reusable model method. A shared `update(**kwargs)` method on the model is the wrong abstraction because it cannot know its call context. Every caller of that method becomes a potential mass assignment vector.

### 2. Scope the operation to the requesting user

The ownership check goes in before the query. A 403 before a 404 leaks less information about whether the record exists, but the priority here is correctness: a user must not be able to modify records they do not own.

If admin users need to modify other users' records, that is a separate endpoint with `@admin_required`, a separate allowlist appropriate to admin operations, and a separate audit log entry.

### 3. Delete the `update(**kwargs)` model method

It should not exist. It will be found, it will be called with unfiltered input somewhere, and the next incident will reference this method in its postmortem. Remove it now while the surface is small.

### 4. Add tests that actually cover the attack surface

The following tests are required, not optional:

```python
def test_update_user_cannot_set_role(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400  # or 200 with role unchanged — pick one, enforce it
    user = User.query.get(1)
    assert user.role != 'admin'

def test_update_user_cannot_set_password_hash(client, seed_users, user_headers):
    original_hash = User.query.get(1).password_hash
    rv = client.patch('/api/users/1', json={'password_hash': 'attacker_hash'}, headers=user_headers)
    assert User.query.get(1).password_hash == original_hash

def test_update_user_cannot_modify_other_user(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch('/api/users/2', json={'display_name': 'Hijacked'}, headers=user_headers)
    assert rv.status_code == 403

def test_update_user_cannot_escalate_own_role(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'role': 'admin'}, headers=user_headers)
    assert User.query.get(1).role != 'admin'
```

This PR introduces two critical vulnerabilities and a reusable dangerous abstraction. The CI passing is not relevant. The lint being clean is not relevant. Do not merge this.
