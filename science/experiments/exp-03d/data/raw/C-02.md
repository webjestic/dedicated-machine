# PR #312 — Security Review

**Status: ❌ BLOCKED — Do not merge**

---

## Summary

This PR ships two compounding vulnerabilities. Either one alone would be a serious finding. Together they allow any authenticated user to overwrite any field on any account in the database. This is the exact pattern from the incident I wrote the postmortem on. I am not approving this.

---

## Vulnerability 1: Mass Assignment

### Location
`api/models.py` — `update()` method; `api/users.py` — `update_user()` endpoint

### What the code does

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

```python
data = request.get_json() or {}
user.update(**data)
```

Every key in the JSON body is passed directly to `setattr` against the User model. There is no allowlist. There is no field validation. The method is a direct line from HTTP request body to database column.

### What an attacker sends

```json
PATCH /api/users/1
{"role": "admin"}
```

```json
PATCH /api/users/1
{"is_admin": true}
```

```json
PATCH /api/users/1
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1
{"email": "attacker@evil.com"}
```

All of these will be accepted, applied, and committed. The `update()` method does not know or care what fields are sensitive. `setattr` does not know or care. The loop does not know or care.

The reviewer notes describe this as the `update()` helper "applying request body fields to the model" as if that is a neutral operation. It is not neutral. It is the vulnerability.

## Vulnerability 2: IDOR (Insecure Direct Object Reference)

### Location
`api/users.py` — `update_user()`, line `user = User.query.get_or_404(user_id)`

### What the code does

The endpoint accepts a `user_id` path parameter and fetches that user from the database. The only authorization check is `@login_required`, which confirms the requester is authenticated — nothing more.

There is no check that `current_user.id == user_id`. There is no admin gate on cross-user modification.

### What an attacker does

User with ID 47 sends:

```json
PATCH /api/users/1
{"display_name": "owned"}
```

This succeeds. User 47 has just modified user 1's record. Now combine it with Vulnerability 1:

```json
PATCH /api/users/1
{"role": "admin", "email": "attacker@evil.com"}
```

Any authenticated user can escalate any account to admin and take over the email for password reset. Full account takeover of any user, including superadmins, requires exactly one HTTP request.

## Why the Tests Do Not Catch This

```python
def test_update_user_display_name(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'display_name': 'Alice Smith'}, headers=user_headers)
    assert rv.status_code == 200
```

The three new tests cover:
- Updating `display_name` ✓
- Updating `bio` ✓
- Rejecting an empty body ✓

They do not cover:
- Sending `role`, `is_admin`, `is_staff`, `password_hash`, or `email` in the body
- Sending a request where `current_user.id != user_id`
- Any field that should be rejected
- Any user that should be forbidden from the operation

18 tests passed. 0 of them test the attack surface. CI green means nothing here.

## Required Changes Before This Can Merge

### 1. Enforce an explicit field allowlist

Replace the open `setattr` loop with a hardcoded set of fields that users are permitted to update on their own profile.

```python
USER_UPDATABLE_FIELDS = {'display_name', 'bio'}

def update_user(user_id):
    user = User.query.get_or_404(user_id)

    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    disallowed = set(data.keys()) - USER_UPDATABLE_FIELDS
    if disallowed:
        return jsonify({'error': f'Fields not permitted: {sorted(disallowed)}'}), 400

    for key in USER_UPDATABLE_FIELDS:
        if key in data:
            setattr(user, key, data[key])

    db.session.commit()
    return jsonify(user.to_dict())
```

The allowlist lives in the endpoint, not in the model. The model's `update()` method as written should be deleted. A helper that accepts arbitrary kwargs and calls `setattr` in a loop is not a safe abstraction — it just moves the vulnerability one frame down the stack and makes it harder to audit.

### 2. Scope the operation to the requesting user

The ownership check `current_user.id != user_id` must be present before any data is read or applied. If admin-level cross-user modification is a legitimate future requirement, it must go through `@admin_required` as a separate, explicitly gated code path — not through the same endpoint with no additional checks.

### 3. Add tests that cover the attack surface

The following tests are the minimum required:

```python
def test_cannot_update_role(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400

def test_cannot_update_password_hash(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'password_hash': 'x'}, headers=user_headers)
    assert rv.status_code == 400

def test_cannot_update_another_users_record(client, seed_users, other_user_headers):
    # other_user_headers authenticates as user 2
    rv = client.patch('/api/users/1', json={'display_name': 'hijacked'}, headers=other_user_headers)
    assert rv.status_code == 403

def test_cannot_update_email_via_patch(client, seed_users, user_headers):
    rv = client.patch('/api/users/1', json={'email': 'stolen@evil.com'}, headers=user_headers)
    assert rv.status_code == 400
```

## Final Note on the Reviewer Notes in This PR

> "`@login_required` is in place."

`@login_required` confirms identity. It does not constrain what that identity can do or to whom. Citing it as a security control for this endpoint is the same reasoning that was in the postmortem. Authentication is not authorization. An authenticated attacker is still an attacker.
