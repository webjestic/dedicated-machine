Decision: Request Changes

Summary: The implementation matches the PR description in that a PATCH endpoint is added with `@login_required` and tests cover the stated cases, but there are CI-relevant gaps that prevent approval. The three new tests only exercise `display_name` and `bio` updates against a hardcoded user ID (`/api/users/1`) and never assert that a requester cannot modify *another* user's record — meaning the authorization boundary described implicitly by the endpoint design is untested and the coverage number is misleading about what is actually exercised. Additionally, there is no test for a non-existent `user_id` (the 404 path from `get_or_404`) and no test confirming that an unauthenticated request is rejected, both of which are standard contract cases for this kind of endpoint and should be part of the 18-test suite before merge.
