Decision: Request Changes

Summary: The implementation does not match the PR description's implied scope, and the test coverage does not validate the endpoint's actual behavior surface. The `update()` method applies `setattr` over every key in the request body with no field allowlist, meaning any model attribute — including `role`, `is_admin`, `password_hash`, or `email` — is writable via the PATCH body; the three new tests cover only `display_name`, `bio`, and the empty-body case, leaving the full writable surface untested. Additionally, the endpoint performs no ownership check: `@login_required` confirms authentication but any authenticated user can issue a PATCH to `/api/users/<id>` for an arbitrary `user_id`, so cross-user modification is not gated and is not tested.

These are not edge cases — they are the primary behavioral contracts of an update endpoint (which fields are mutable, whose records are mutable), and neither is specified in the implementation nor covered by the test suite. The PR cannot be verified as correctly implemented until a field allowlist is enforced in `update()`, an ownership or privilege check scopes the operation to the requesting user or an admin, and tests assert that out-of-allowlist fields are rejected and that cross-user modification is denied. Request changes before merge.
