# Security Review — PR #312 — **REJECTED**

**Reviewer:** Security Review (Backend)
**Verdict:** ❌ Do not merge. Two critical vulnerabilities. No exceptions.

---

## Summary

This PR introduces a PATCH endpoint with two independent critical vulnerabilities, either of which is sufficient to reject. Together they are a complete account takeover primitive. The CI passing and the lint being clean are not relevant. The tests do not exercise the attack surface at all.

---

## Vulnerability 1 — Mass Assignment (Critical)

**Location:** `api/models.py` — `update()`, called from `api/users.py:update_user()`

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

This iterates over every key in the caller-supplied dict and writes it directly onto the model instance with no filtering. The caller is the HTTP request body. There is no allowlist. There is no field validation. Any field that exists on the `User` model is writable by sending the corresponding key in a JSON body.

The exploit is not theoretical. It is a single curl command:

```bash
curl -X PATCH https://obotix.one/api/users/1 \
  -H "Content-Type: application/json" \
  -H "Cookie: session=<any valid session>" \
  -d '{"role": "admin"}'
```

If the `User` model carries any of the following fields — and models like this typically carry most of them — every one is exposed:

| Field | Impact |
|---|---|
| `role` | Full privilege escalation to admin |
| `is_admin` / `is_staff` | Same |
| `email` | Account hijack via password reset |
| `password_hash` | Direct credential replacement |
| `is_active` / `is_verified` | Bypass email verification |
| `id` | Primary key collision |

The `update()` method is named and structured as a general-purpose helper. That makes it more dangerous, not less — it will be called again from other endpoints, carrying this same vulnerability forward.

**Required fix:** Replace the unconstrained `setattr` loop with an explicit allowlist. The method should not exist in its current form.

```python
# In User model
MUTABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

def update(self, **kwargs):
    unknown = set(kwargs) - self.MUTABLE_FIELDS
    if unknown:
        raise ValueError(f"Field(s) not updatable: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

The allowlist lives on the model. It is explicit. It is the only thing that gets written. Any key outside it raises before a `setattr` is ever called. When someone adds a new field to `User` in six months, it is not automatically writable via the API — they have to make a conscious decision to add it to `MUTABLE_FIELDS`.

## Vulnerability 2 — Insecure Direct Object Reference (Critical)

**Location:** `api/users.py:update_user()`

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    ...
```

`@login_required` confirms that a session exists. It does not confirm that the session belongs to the user being modified. The `user_id` in the URL path is taken at face value. Any authenticated user can send a PATCH request targeting any `user_id` and the endpoint will process it.

Combined with Vulnerability 1, the full attack surface is: any authenticated user can write any field on any account. That includes setting another user's `role` to `admin`, replacing their `password_hash`, or taking over their email address.

Even with Vulnerability 1 fully remediated, this IDOR remains a standalone critical issue. A user who can only update `display_name` and `bio` should still only be able to update their own `display_name` and `bio`.

**Required fix:** Scope the operation to the requesting user before touching the record. Admins get an explicit bypass.

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id and current_user.role != 'admin':
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    ...
```

The authorization check runs before the database query. There is no reason to fetch a record you are about to deny access to.

## On the Test Coverage

The three new tests cover `display_name`, `bio`, and an empty body. They all pass. They are testing the happy path on a compliant client. They do not constitute security coverage.

The tests that are missing and must be added before this merges:

```python
def test_update_user_cannot_escalate_role(client, seed_users, user_headers):
    """A regular user cannot set their own role to admin."""
    rv = client.patch('/api/users/1', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400  # or 403 — must not be 200

def test_update_user_cannot_write_password_hash(client, seed_users, user_headers):
    """A regular user cannot overwrite their own password_hash directly."""
    rv = client.patch('/api/users/1', json={'password_hash': 'x'}, headers=user_headers)
    assert rv.status_code == 400

def test_update_user_cannot_modify_other_user(client, seed_users, user_headers):
    """A regular user cannot modify a different user's record."""
    rv = client.patch('/api/users/2', json={'display_name': 'hacked'}, headers=user_headers)
    assert rv.status_code == 403

def test_admin_can_modify_other_user(client, seed_users, admin_headers):
    """An admin can modify another user's record."""
    rv = client.patch('/api/users/2', json={'display_name': 'Updated'}, headers=admin_headers)
    assert rv.status_code == 200
```

100% line coverage on the happy path is 0% coverage on the attack path. These are not the same metric.

## What Needs to Happen Before This Can Merge

1. **Replace `update()` with an allowlist implementation.** The current implementation must not be committed to any branch that touches production. The method name `update` with no qualifier will be called again by someone who does not read this review.

2. **Add the ownership check to the endpoint.** The authorization gate runs before the database query, not after.

3. **Add the four security test cases listed above.** They must pass with the correct status codes. A 200 on a role escalation attempt is a test failure regardless of what the response body says.

4. **Audit every other endpoint that calls `user.update()` or any equivalent pattern.** If this method already exists elsewhere under a different name, those call sites need the same review.

5. **Confirm the `User` model's field inventory.** I need to see which fields are on that model before I can sign off that the allowlist is complete. If `password_hash` or `role` are ORM columns and not in `MUTABLE_FIELDS`, the fix is correct. I want to verify that, not assume it.
