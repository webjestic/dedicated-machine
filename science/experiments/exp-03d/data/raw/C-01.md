# Security Review — PR #312 — **REJECTED**

**Reviewer:** Security Review (Backend)
**Verdict:** ❌ Do not merge. Two critical vulnerabilities. No exceptions.

---

## What This PR Actually Does

The description says "adds a PATCH endpoint with auth handled by `@login_required`." That framing is doing a lot of work to make this sound complete. It is not complete. `@login_required` confirms that a session exists. It does not confirm that the session belongs to the user being modified, and it does not constrain which fields that user can write. Both of those gaps are exploitable right now, independently, and they compound each other.

---

## Vulnerability 1 — Mass Assignment (Critical)

**Location:** `api/models.py`, `update()` method; `api/users.py`, `update_user()`

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)

# users.py
data = request.get_json() or {}
user.update(**data)
```

Every key in the JSON body becomes a `setattr` call on the `User` model. There is no allowlist. There is no field validation. The request body is the argument list.

A request like this:

```json
PATCH /api/users/1
{"role": "admin"}
```

will call `setattr(user, 'role', 'admin')` and commit it. The `admin_required` decorator exists in this codebase — which means `role` is a live authorization field on this model — and it is now writable by any authenticated user who knows a user ID.

This is not a theoretical edge case. This is the exact attack pattern from the incident I documented fourteen months ago. The field name was `role` then too. It took fourteen hours to detect because the attacker made themselves admin quietly and did nothing immediately. By the time we noticed, they had been in the admin panel for six hours.

Other fields that are almost certainly settable via this same path, depending on what the `User` model exposes:

- `email` — account takeover via email change, bypassing any verification flow
- `password_hash` — direct credential replacement if the column is named predictably
- `is_staff`, `is_verified`, `is_active` — any boolean privilege or status flag
- `created_at`, `id` — data integrity issues

The `update()` method in `models.py` is described in its docstring as a helper. It is not a helper. It is an open `setattr` loop with no constraints. The fact that it is a named method on the model makes it look safe. It is not safe. The name does not add validation.

**Fix required:**

```python
# models.py
MUTABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in MUTABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist belongs on the model, not in the route, because the model is what gets reused. If you put the filter only in the route handler, the next engineer who calls `user.update(**data)` from a different endpoint gets the same vulnerability with no warning.

## Vulnerability 2 — Broken Object-Level Authorization / IDOR (Critical)

**Location:** `api/users.py`, `update_user()`

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    user.update(**data)
    db.session.commit()
```

The endpoint accepts a `user_id` from the URL. It fetches that user. It applies the update. At no point does it check whether `current_user.id == user_id`. Any authenticated user can send a PATCH to `/api/users/<any_id>` and modify that record.

Combined with Vulnerability 1, the attack surface is:

- Any authenticated user can modify any other user's profile fields
- Any authenticated user can escalate any account — including accounts they do not own — to admin

These are not separate issues that partially overlap. They are fully composable. An attacker does not need to target their own account. They can target any account, including accounts with existing elevated privileges, and modify those.

**Fix required:**

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    user.update(**data)
    db.session.commit()
    return jsonify(user.to_dict())
```

Note: the authorization check should happen before the database query. Fail fast on the identity check. Do not fetch a record you are about to deny access to — it leaks timing information about whether the record exists.

## What the Tests Cover and What They Don't

The three new tests check:

1. `display_name` updates to user ID 1
2. `bio` updates to user ID 1
3. Empty body returns 400

These tests pass. They will always pass. They test the happy path for two fields that happen to be safe. They do not test:

- `{"role": "admin"}` — should be rejected, is not
- `{"is_admin": true}` — should be rejected, is not
- `{"email": "attacker@evil.com"}` — should be rejected, is not
- A request from user ID 2 targeting user ID 1 — should be 403, is 200
- A request from an unauthenticated client — covered by `@login_required`, but not tested

89% coverage on this file means nothing here. The attack paths are not in the coverage report because nobody wrote tests that walk them. That is the gap. Coverage measures which lines execute, not which inputs are dangerous.

**Tests that must exist before this merges:**

```python
def test_cannot_set_role_via_patch(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 400  # or 422 — not 200

def test_cannot_update_another_users_record(client, seed_users, other_user_headers):
    # other_user_headers is authenticated as user 2
    rv = client.patch(
        '/api/users/1',
        json={'display_name': 'Hijacked'},
        headers=other_user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_password_hash_via_patch(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1',
        json={'password_hash': 'aaaa'},
        headers=user_headers
    )
    assert rv.status_code == 400
```

## What Needs to Happen Before This Can Be Reconsidered

1. Add `MUTABLE_FIELDS` allowlist to `User.update()`. Raise on unknown fields. Do not silently ignore them — silent ignore means the caller never knows their input was dropped, which creates its own bugs.
2. Add ownership check in `update_user()` before the database query. Return 403 if `current_user.id != user_id`. If admin override of other accounts is a legitimate use case, that is a separate endpoint with `@admin_required`, not a conditional in this one.
3. Add tests for the attack paths listed above. These are not optional coverage improvements. They are the specification for what this endpoint is supposed to refuse.
4. Review every other call site of `User.update()` or any equivalent pattern in this codebase before this ships. If this method exists, it may already be called elsewhere.

The CI is green. The lint is clean. The PR is not safe to merge.
