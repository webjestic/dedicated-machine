# Security Review — PR #312 — BLOCKED, DO NOT MERGE

**Reviewer:** Security Review (Backend)
**Verdict:** ❌ **Rejected — Critical vulnerabilities. This PR cannot merge in its current state.**

---

## Summary

This PR introduces two critical security vulnerabilities in combination. Either one alone is serious. Together they allow any authenticated user to take over any account on the platform, including escalating themselves to admin. The CI passing and the 89% coverage number are not relevant to this assessment. The tests do not cover the attack surface.

---

## Vulnerability 1 — Mass Assignment (Critical)

**Location:** `api/models.py` — `update()` method; `api/users.py` — `update_user()`

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on the model is now a writable API parameter
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)  # unvalidated user input passed directly into setattr loop
```

This is the exact pattern I have seen go to production before. The `update()` method is not a helper — it is an open door to every column on the `User` model. There is no allowlist. There is no field filtering. Whatever the client sends in the JSON body, `setattr` will attempt to write to the model.

**Concrete attack requests, executable right now:**

```json
PATCH /api/users/1
{"role": "admin"}
```

```json
PATCH /api/users/1
{"is_admin": true}
```

```json
PATCH /api/users/1
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1
{"email": "attacker@evil.com"}
```

```json
PATCH /api/users/1
{"is_active": false}
```

The last one is a denial-of-service against a targeted account. The second-to-last hijacks account recovery. The first two are privilege escalation. None of these require any special tooling — `curl` is sufficient.

**The `update()` method must not exist in this form.** A generic `setattr` loop over unsanitized input is not a utility function. It is a vulnerability with a docstring.

## Vulnerability 2 — Insecure Direct Object Reference (Critical)

**Location:** `api/users.py` — `update_user()`

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)
    # current_user is never checked against user_id
    user.update(**data)
```

`@login_required` confirms that a session exists. It does not confirm that the session belongs to the user being modified. There is no ownership check anywhere in this function.

**Attack scenario:**

1. Attacker authenticates as user ID 47.
2. Attacker sends `PATCH /api/users/1` — targeting the admin account.
3. The endpoint fetches user 1, applies the attacker's payload, commits.

The attacker does not need to know anything about user 1 except their ID, which is enumerable from the URL structure.

**Combined with Vulnerability 1:** an attacker authenticates as any low-privilege account, sends `PATCH /api/users/<any_id>` with `{"role": "admin"}`, and owns the platform. This is not a theoretical attack chain. It is four lines of curl.

## Vulnerability 3 — No Input Validation on Field Values (High)

Even if an allowlist were in place, there is no type validation, length validation, or format validation on values. A `display_name` field accepting a 10MB string is a storage amplification vector. An `email` field accepting an arbitrary string without format validation breaks downstream assumptions. This needs to be addressed alongside the allowlist fix, not after.

## On the Tests

The three new tests cover `display_name`, `bio`, and an empty body. They confirm the happy path works. They do not constitute security coverage.

**Tests that are absent and must exist before this merges:**

```
test_update_user_cannot_set_role
test_update_user_cannot_set_is_admin
test_update_user_cannot_set_password_hash
test_update_user_cannot_set_email (if email is not an allowed update field)
test_update_user_cannot_modify_other_user
test_update_user_unauthenticated_returns_401
test_update_user_unknown_fields_are_ignored (not rejected with 500, ignored cleanly)
```

The absence of these tests is not a coverage gap — it is a signal that the attack surface was not considered during development. 89% line coverage on code that has a critical vulnerability in it is not a reassuring number.

## Required Changes Before Re-Review

### 1. Replace `update()` with an explicit allowlist

```python
# models.py

MUTABLE_FIELDS = {'display_name', 'bio'}  # explicit, reviewed, version-controlled

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in MUTABLE_FIELDS:
            raise ValueError(f"Field '{key}' is not updatable")
        setattr(self, key, value)
```

The allowlist must be defined at the model layer, not in the route. If it lives only in the route, the next engineer who adds a route calling `update()` will bypass it without realizing it.

### 2. Add an ownership check to the endpoint

```python
# users.py

@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

If admin users need to modify other users' records, that must be a separate endpoint decorated with `@admin_required`, with its own audit logging, and with its own restricted allowlist. It must not be the same code path.

### 3. Validate field values against expected types and lengths

Before `setattr` is called, values must be validated. Use a schema library (Marshmallow, Pydantic) or explicit per-field validation. Do not rely on the database to reject bad input — by the time the DB raises an error, the application may have already behaved incorrectly.

### 4. Write the missing security tests

Every item in the list above needs a test. These tests should fail against the current implementation and pass against the fixed one. That is how you know the fix is real.
