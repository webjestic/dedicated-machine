# Security Review — PR #312 — BLOCKED, DO NOT MERGE

**Reviewer:** Security Review  
**Verdict:** ❌ **Rejected**  
**Priority:** Critical — two independent vulnerabilities, either sufficient to block alone

---

## What This PR Does

Adds a PATCH endpoint that takes a JSON body and applies every field in it to a user record via `setattr` in a loop. Auth is present. Field validation is not. Ownership checks are not. The test suite covers two benign fields and an empty-body case.

I am going to be direct about what that means in practice before I get into the mechanics.

---

## Vulnerability 1 — Mass Assignment (Critical)

### The code

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every field on the model, no exceptions
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)  # caller controls kwargs entirely
```

This is the pattern. The `update()` method is named like a helper. It is not a helper. It is a direct pipe from the request body to `setattr` on the model with no allowlist, no denylist, no type checking, and no field awareness. The name makes it look safe. That is the danger.

### What an attacker sends

```json
PATCH /api/users/1
{"role": "admin"}
```

```json
PATCH /api/users/1
{"is_admin": true}
```

```json
PATCH /api/users/1
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1
{"email": "attacker@external.com"}
```

I don't know which of those fields your `User` model exposes without seeing the full model definition. That is the point. Neither does this PR. It applies all of them unconditionally.

The incident I wrote the postmortem on was exactly this shape. One field. `{"role": "admin"}`. Fourteen hours to detect because the user behaved normally after escalating — they were patient. The detection gap is not hypothetical.

### What the tests cover

```python
json={'display_name': 'Alice Smith'}
json={'bio': 'Backend engineer at obotix'}
json={}
```

Three tests. Zero coverage of the attack surface. 89% overall coverage is not relevant here. Coverage measures which lines execute, not which inputs are validated. The `setattr` line executes fine with `role` as the key. It will show as covered.

### Fix required

Define an explicit allowlist. Not a comment. Not a convention. Code.

```python
UPDATABLE_FIELDS = frozenset({'display_name', 'bio'})

def update_user(user_id):
    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    unknown = set(data.keys()) - UPDATABLE_FIELDS
    if unknown:
        return jsonify({'error': f'Fields not permitted: {sorted(unknown)}'}), 400

    for field in UPDATABLE_FIELDS:
        if field in data:
            setattr(user, field, data[field])

    db.session.commit()
    return jsonify(user.to_dict())
```

Returning a 400 on unknown fields rather than silently ignoring them is intentional. Silent ignoring means an attacker learns nothing from probing. Explicit rejection makes the boundary visible in logs and to legitimate callers.

The `update()` method on the model should be removed entirely or scoped to accept only the allowlisted fields. A generic `setattr` loop on a model is not a utility function that belongs in a shared location. Every callsite that uses it inherits the vulnerability.

## Vulnerability 2 — Insecure Direct Object Reference (Critical)

### The code

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    user = User.query.get_or_404(user_id)  # fetches by URL parameter
    # current_user is never consulted again
    user.update(**data)
```

`@login_required` confirms the request is authenticated. It does not confirm the request is authorized to modify this specific record. After the decorator passes, `current_user` is imported and then ignored. The endpoint fetches whatever `user_id` appears in the URL and updates it.

### What an attacker does

Log in as user 42. Send:

```
PATCH /api/users/1
{"display_name": "owned"}
```

User 1's record is updated. No error. No log entry distinguishing this from a legitimate self-update. Combined with Vulnerability 1, the attacker can update any field on any user record. The IDOR is bad alone. Alongside mass assignment it is a full account takeover primitive for any user ID the attacker can enumerate — and your `GET /api/users/<id>` endpoint, also gated only by `@login_required`, makes enumeration trivial.

### Fix required

Check ownership before applying any update. If admin-level cross-user updates are a legitimate use case, gate them explicitly.

```python
@users_bp.route('/api/users/<int:user_id>', methods=['PATCH'])
@login_required
def update_user(user_id):
    if current_user.id != user_id and current_user.role != 'admin':
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    # ... rest of handler
```

The 403 must come before the database fetch. Returning 404 for records the user is not permitted to access is also defensible — it avoids confirming that a user ID exists — but whichever you choose, the authorization check must happen.

## Secondary Issue — `admin_required` Logic

Not the focus of this PR, but it is in the diff and it has a bug:

```python
if not current_user.role == 'admin':
```

Operator precedence makes this `if (not current_user.role) == 'admin'`, which evaluates the boolean of `current_user.role` before comparing. If `role` is any truthy non-admin string, `not current_user.role` is `False`, and `False == 'admin'` is `False`, so the check passes — the condition reads as "if this comparison is falsy, block access," which is the correct behavior here, but only by accident. The intent is clearly:

```python
if current_user.role != 'admin':
```

Write what you mean. The current form will confuse the next reader and may behave unexpectedly if `role` is ever `None` or an empty string.
