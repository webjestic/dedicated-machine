## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, uncoordinated operations with no row-level lock between them.**

Here's the exact failure sequence:

```
Worker 1:  SELECT → finds task #42 (status = 'pending')
Worker 2:  SELECT → finds task #42 (status = 'pending')   ← both read before either writes
Worker 1:  UPDATE task #42 SET status = 'running'
Worker 2:  UPDATE task #42 SET status = 'running'         ← overwrites Worker 1, both proceed
Worker 1:  execute_task(#42) → complete_task(#42)
Worker 2:  execute_task(#42) → complete_task(#42)         ← duplicate completion
```

This was invisible at one worker. It only surfaces under concurrent load — which is exactly what the scale-out to 4 workers three days ago introduced. The tests pass because they run a single worker against an isolated DB with no concurrency.

---

## The Fix

The SELECT and UPDATE need to be a single atomic operation. The right approach depends on your database.

### Option A — SQLite (atomic UPDATE + RETURNING)

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically using UPDATE...RETURNING."""
    # Single statement: no window between read and write.
    # Subquery identifies the target row; UPDATE claims it in one shot.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

### Option B — PostgreSQL (SELECT FOR UPDATE SKIP LOCKED)

```python
def acquire_task(worker_id: int):
    """Claim the next pending task using a locked read."""
    # FOR UPDATE acquires a row lock.
    # SKIP LOCKED means other workers bypass rows already being claimed —
    # no blocking, no duplicates.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )
    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

### Option C — Database-agnostic optimistic lock (fallback)

```python
def acquire_task(worker_id: int):
    """Claim a task using a conditional UPDATE to avoid races."""
    # The WHERE status = 'pending' guard means only one writer wins.
    # Check rows_affected to confirm this worker actually claimed it.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard added here
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # Another worker beat us to it — don't proceed with this task.
    if rows_affected == 0:
        return None

    return dict(row)
```

Option C is the lowest-friction fix if you can't change the query shape right now. Option A or B is the correct long-term solution.
