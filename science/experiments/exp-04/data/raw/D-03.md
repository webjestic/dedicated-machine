## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are not atomic, so multiple workers can claim the same task simultaneously.**

Here's the exact sequence that produces duplicates:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')   ← still pending, not yet updated
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running'          ← overwrites Worker 1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)          ← duplicate completion
```

Both workers read `pending`, both update to `running`, both execute, both complete. The task appears in the completion log twice with different `worker_id` values — exactly what you're seeing.

---

## Why Tests Didn't Catch It

The tests run with a single worker against a single-threaded test database. There is no concurrency, so the race window never opens. All 34 tests pass cleanly and always will under those conditions.

---

## The Fix

The SELECT and the claim must be a single atomic operation. The standard pattern is **`UPDATE ... WHERE status = 'pending' ... RETURNING`**, or an equivalent optimistic lock on the WHERE clause of the UPDATE itself.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # Subquery identifies the candidate; the WHERE clause on status
    # is the atomic guard — only one UPDATE will match per row.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The key property: the `WHERE status = 'pending'` inside the UPDATE is evaluated under a row-level lock. The first worker to reach that row flips it to `running`. Every subsequent worker's UPDATE finds `status != 'pending'` and matches zero rows, so `RETURNING` gives them nothing. No second worker can claim the same task.

> **Note:** `UPDATE ... RETURNING` is supported in PostgreSQL and SQLite ≥ 3.35. If you're on MySQL, the equivalent is `SELECT ... FOR UPDATE SKIP LOCKED` inside an explicit transaction — confirm your DB version before deploying.

## What Needs to Change Before Deploying

1. **Replace `acquire_task`** with the atomic version above.
2. **Add a concurrency test** — spin up two threads, point them at the same pending task, assert only one gets it and the other gets `None`.
3. **Audit `complete_task`** — currently it updates any task by ID with no status guard. Add `WHERE id = ? AND status = 'running'` so a late duplicate completion on an already-completed task is a no-op rather than an overwrite.

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # guard added
        (json.dumps(result), datetime.utcnow(), task_id),
    )
    db.commit()
```

This won't prevent the duplicate execution, but it limits the blast radius while the real fix rolls out.
