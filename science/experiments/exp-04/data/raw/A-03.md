# Root Cause: Classic Check-Then-Act Race Condition in `acquire_task`

## The Short Version (for the client call)

Four workers are racing to claim the same task. The SELECT and the UPDATE are two separate operations with no lock between them. All four workers can read the same `pending` row before any of them writes `running`. The task executes four times.

---

## The Failure Sequence

Here is exactly what the logs are showing you:

```
T=0ms  Worker-1 SELECT → sees task #4471, status='pending'
T=0ms  Worker-2 SELECT → sees task #4471, status='pending'   ← same row, no lock
T=0ms  Worker-3 SELECT → sees task #4471, status='pending'   ← same row, no lock
T=1ms  Worker-1 UPDATE task #4471 SET status='running'
T=1ms  Worker-2 UPDATE task #4471 SET status='running'       ← overwrites, no guard
T=1ms  Worker-3 UPDATE task #4471 SET status='running'       ← overwrites, no guard
T=2ms  Worker-1 execute_task(#4471) → complete
T=2ms  Worker-2 execute_task(#4471) → complete               ← duplicate
T=2ms  Worker-3 execute_task(#4471) → complete               ← duplicate
```

The UPDATE has no condition that checks the current status. It will succeed regardless of whether another worker already claimed the row. Three workers write `running`, three workers execute, three completions land in the log.

---

## Why Tests Passed

The tests run a single worker against an in-process database. There is no concurrency. The race window does not exist in a single-threaded test. This is the exact failure mode where CI gives you a green check and production gives you an incident. The tests are not wrong — they are just not testing the thing that broke.

---

## The Broken Code

```python
def acquire_task(worker_id: int):
    # STEP 1: Read
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # ← THE WINDOW IS HERE
    # Any number of workers can be sitting at this line
    # having all read the same row above.
    # Nothing prevents it.

    # STEP 2: Act — no atomicity guarantee with Step 1
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",          # ← no status check in WHERE clause
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

Two problems compounding each other:

1. **SELECT and UPDATE are not atomic.** Any worker can read the row between another worker's SELECT and UPDATE.
2. **The UPDATE does not guard itself.** `WHERE id = ?` will match and succeed even if the row is already `running`. A worker that lost the race still claims the task.

---

## The Fix

### Option 1: Add `AND status = 'pending'` to the UPDATE and check rows affected

This is the minimum viable fix. Ship this now.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard added
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount

    db.commit()

    if rows_updated == 0:
        # Another worker claimed it between our SELECT and UPDATE.
        # Bail out. The next loop iteration will find a different task.
        return None

    return dict(row)
```

This closes the window. Only one UPDATE can transition a row from `pending` to `running`. The database enforces it. Every other worker gets `rowcount == 0` and moves on.

### Option 2: Atomic SELECT FOR UPDATE (preferred long-term)

If the database supports it, collapse the two operations into one:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

No SELECT-then-UPDATE split. No window. One round trip.

---

## What to Deploy Right Now

Option 1. It is a four-line change to a single function. It does not require a schema change or a database feature check. The `AND status = 'pending'` guard plus the `rowcount == 0` early return is enough to stop the bleeding immediately.

Option 2 is the right long-term shape but confirm `RETURNING` support on your database engine before you ship it under pressure.

---

## After the Incident

Two items for the postmortem:

1. **Add a concurrency test.** Spin up two workers against a shared in-memory database, insert one task, run both workers simultaneously, assert the completion log contains exactly one entry. This failure would have been caught before it reached production.
2. **Audit `complete_task` as well.** It also does an unconditional UPDATE with no status guard. If a task somehow gets completed twice, the second write silently overwrites the first result. It is not the cause of this incident but it is the same class of problem one step later in the pipeline.
