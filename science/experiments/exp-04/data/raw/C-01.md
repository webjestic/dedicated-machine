# Root Cause: Check-Then-Act Race Condition in `acquire_task`

## What's Happening

The incident started three days ago when you scaled from 1 to 4 workers. That's not a coincidence — that's the diagnosis. This bug was always there. One worker meant one reader, so the window never mattered. Four workers means four processes hitting `acquire_task` simultaneously, and now the window is getting hit under peak load.

The sequence that produces a duplicate completion:

```
Time →

Worker 1: SELECT ... WHERE status = 'pending' → gets task #42
Worker 2: SELECT ... WHERE status = 'pending' → gets task #42  ← same row, status still 'pending'
Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 2: UPDATE task_queue SET status = 'running' WHERE id = 42  ← overwrites worker 1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)
```

Both workers read the same pending row before either one commits the update. The SELECT and the UPDATE are two separate statements with no atomicity guarantee between them. That gap — between the check and the act — is the race condition.

---

## Why Tests Don't Catch It

The tests are single-threaded and sequential. `acquire_task` is called once, then `execute_task`, then `complete_task`. There is no concurrent reader. The test suite is correctly testing the behavior of each function in isolation. It is not testing the behavior of four processes running `acquire_task` at the same time against a shared database. This is the exact failure mode that only appears in production under load.

---

## The Fix

The SELECT and the UPDATE need to be a single atomic operation. The database has to do both or neither, with no window in between for another reader.

### Option 1: `UPDATE ... RETURNING` (PostgreSQL 9.5+, SQLite 3.35+)

This is the cleanest solution. One statement, no separate SELECT, no window.

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    row = db.query_one(
        """
        UPDATE task_queue
        SET status = 'running',
            worker_id = ?,
            claimed_at = ?
        WHERE id = (
            SELECT id FROM task_queue
            WHERE status = 'pending'
            ORDER BY priority DESC, created_at ASC
            LIMIT 1
        )
        RETURNING id, type, payload
        """,
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The subquery and the update execute as one atomic unit. If two workers race, one gets the row and the other gets zero rows back. No duplicate claims.

### Option 2: Optimistic Locking with `WHERE status = 'pending'` on the UPDATE

If `RETURNING` isn't available, use the status check as a guard on the UPDATE itself and verify that it actually claimed something.

```python
def acquire_task(worker_id: int):
    """Claim a task only if we can atomically transition it from pending to running."""
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard condition
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_affected == 0:
        # Another worker claimed this task between our SELECT and UPDATE.
        # Bail out and let the main loop retry.
        return None

    return dict(row)
```

The `AND status = 'pending'` on the UPDATE means the second worker's update hits zero rows because the first worker already flipped it to `'running'`. The `rowcount` check detects the loss and returns `None` so the loop retries cleanly. This is still two statements, but the UPDATE itself is the authoritative claim — the SELECT is just a hint.

---

## What to Do Right Now

**Immediate:** Deploy Option 1 or Option 2. Option 1 is preferable if your database supports `RETURNING`. Option 2 works everywhere.

**Before deploying:** Check the completion log for all tasks that have duplicate entries. Identify which downstream consumers already processed the duplicate payloads and what the remediation path is. The fix stops new duplicates — it does not undo the ones that already happened.

**After deploying:** Add a unique constraint or a database-level check if your schema allows it. Defense in depth. The application fix is the primary control; the constraint is the backstop.

---

## What to Note in the Postmortem

The tests passed because they tested correctness, not concurrency. This is a gap in the test strategy, not a gap in the test quality. The tests that exist are correct. What's missing is a test that spins up multiple threads or processes, calls `acquire_task` concurrently, and asserts that each task ID appears in exactly one result. That test would have caught this before it reached production. Write it as part of the remediation.

The scaling event was the trigger, not the cause. The cause was present on day one. Scaling just made the window wide enough to fall through.
