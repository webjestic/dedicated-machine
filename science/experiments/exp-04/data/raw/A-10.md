## Root Cause

**Classic check-then-act race condition in `acquire_task`.** There is no atomic claim. The SELECT and the UPDATE are two separate operations with no lock between them.

Here is the exact failure sequence:

```
Worker 1                              Worker 2
--------                              --------
SELECT → sees task #42 as 'pending'
                                      SELECT → sees task #42 as 'pending'
UPDATE task #42 → status='running'
                                      UPDATE task #42 → status='running'
execute_task(#42)
                                      execute_task(#42)
complete_task(#42)
                                      complete_task(#42)   ← second completion logged
```

Both workers read `status = 'pending'` before either one writes `status = 'running'`. The UPDATE has no guard — it will succeed for both workers because neither WHERE clause checks the current status. You get two workers executing the same task, two completion log entries, two downstream events.

The tests do not catch this because every test runs a single worker against an isolated database. There is no concurrency in the test suite. The CI green check means nothing here.

---

## What Needs to Change

The fix is to make the claim atomic. The UPDATE must be the ownership gate, not the SELECT.

**Option 1 — Atomic UPDATE with status guard (minimum viable fix):**

```python
def acquire_task(worker_id: int):
    # SELECT only to find a candidate ID — no ownership implied
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # The WHERE clause now requires status still be 'pending'
    # Only one worker wins this race — the other UPDATE matches zero rows
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- this is the gate
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker claimed it between our SELECT and this UPDATE
        return None

    return dict(row)
```

This works because the database serializes concurrent UPDATE statements on the same row. Exactly one of them will match `status = 'pending'`. The other gets `rows_affected = 0` and backs off. Your `db.execute` wrapper needs to return rowcount — confirm that before shipping.

**Option 2 — `SELECT FOR UPDATE` / `SKIP LOCKED` (preferred if your DB supports it):**

```sql
SELECT id, type, payload FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED
```

`FOR UPDATE` acquires a row-level lock. `SKIP LOCKED` tells other workers to skip any row already locked rather than queue behind it, which is exactly the behavior you want in a job queue. This is available in PostgreSQL 9.5+ and MySQL 8+. SQLite does not support it.

**Option 3 — Single atomic statement (PostgreSQL only):**

```sql
UPDATE task_queue
SET status = 'running', worker_id = $1, claimed_at = NOW()
WHERE id = (
    SELECT id FROM task_queue
    WHERE status = 'pending'
    ORDER BY priority DESC, created_at ASC
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING id, type, payload
```

One round trip, no application-level race window at all. This is the cleanest solution if you are on Postgres.

## What to Tell the Client Call in 90 Seconds

> Scaling from 1 to 4 workers exposed a race condition in how workers claim tasks. Two workers could read the same task as available at the same moment, both claim it, and both complete it. The fix is a one-line change to the claim query that makes the database enforce exclusive ownership. We are deploying it now. We will add a concurrency test to the suite so this cannot regress silently.

---

## Immediate Mitigation While Fix Is Staged

Scale back to 1 worker process. The race requires concurrent workers. One worker eliminates the window. Queue depth will grow but downstream consumers stop seeing duplicates immediately.
