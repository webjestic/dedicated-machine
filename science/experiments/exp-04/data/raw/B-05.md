## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

Here is the exact sequence that produces the duplicate completions:

```
Worker 1: SELECT → sees task 42, status = 'pending'
Worker 2: SELECT → sees task 42, status = 'pending'   ← same row, before W1 updates it
Worker 1: UPDATE task 42 SET status = 'running', worker_id = 1
Worker 2: UPDATE task 42 SET status = 'running', worker_id = 2   ← overwrites W1's claim
Worker 1: execute_task(42) → complete_task(42)
Worker 2: execute_task(42) → complete_task(42)
```

Both workers claimed the task. Both completed it. That is the duplicate in your log.

---

## Why the Tests Did Not Catch It

Every test runs a single worker against an in-process database. There is no concurrency. The window between SELECT and UPDATE is never contested. All 34 tests pass and will continue to pass forever against this bug. The test suite is not wrong — it is just not testing the thing that broke.

---

## Why Scaling Exposed It Now

At one worker, the race window exists but there is no second worker to enter it. At four workers under peak load, all four are polling simultaneously. The window is small — microseconds — but at high queue throughput it gets hit repeatedly. This is textbook: the bug was always there, concurrency made it probable.

---

## What Needs to Change

The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter of who owns the task, not the application layer.

### Option 1 — Atomic UPDATE with `WHERE status = 'pending'` (preferred, works in SQLite and Postgres)

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, claimed_at),
    )
    db.commit()

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )

    return dict(row) if row else None
```

The UPDATE is the claim. Only one worker's UPDATE wins the row. The subsequent SELECT is just reading back what this worker already owns — it is not a second opportunity for a race.

### Option 2 — `SELECT ... FOR UPDATE` (Postgres only)

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`SKIP LOCKED` means a worker skips any row another worker has locked rather than waiting on it. This is the standard pattern for concurrent job queues on Postgres. If you are on SQLite, use Option 1 — SQLite does not support `FOR UPDATE`.

---

## What to Tell the Junior Engineers

Fix `acquire_task`. Do not touch `execute_task`. The deduplication logic in `execute_task` and the O(n²) membership check in the summary loop are real issues, but they are not what is causing this incident. Fix one thing, the right thing, and ship it. Clean up the rest in a follow-up PR once the incident is closed.

The `complete_task` UPDATE also has no `WHERE status = 'running'` guard, which means it will overwrite a row even in a race. Add `AND status = 'running'` to that WHERE clause as a secondary defense, but that is not the root cause — it is just a guard rail.
