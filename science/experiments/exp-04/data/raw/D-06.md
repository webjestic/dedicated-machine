## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, uncoordinated operations with no row-level lock between them.**

Here's the exact failure sequence:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')   ← still pending, W1 hasn't updated yet
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running'          ← overwrites W1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)          ← duplicate completion
```

Both workers read the same row before either has written to it. The UPDATE has no guard — it will succeed for both workers because neither checks whether the row is still `pending` at write time.

**Why tests didn't catch it:** Every test runs with a single worker against a single-process in-memory database. There is no concurrency, so the race never occurs. All 34 tests pass cleanly and always will under those conditions.

**Why it appeared now:** With one worker process this race is impossible. Scaling to four workers under peak load maximizes the probability of two workers hitting the SELECT within the same ~50ms window.

---

## The Fix

The UPDATE itself needs to be the claim operation, and it needs to be conditional. The SELECT becomes a consequence of the UPDATE, not a prerequisite for it.

### Option A — Atomic UPDATE + conditional SELECT (works in SQLite, MySQL, Postgres)

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task using a conditional UPDATE."""

    # Step 1: Attempt to claim a specific row identified by a subquery.
    # The WHERE status = 'pending' in the outer UPDATE is the guard —
    # only one writer can win; the loser updates 0 rows.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Step 2: Read back only if we actually claimed something.
    if db.rows_affected() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The critical property: `WHERE status = 'pending'` is evaluated inside the same atomic write. The second worker's UPDATE matches zero rows and `rows_affected()` returns 0. No double-claim is possible.

### Option B — SELECT FOR UPDATE (Postgres / MySQL — preferred if your DB supports it)

```python
def acquire_task(worker_id: int):
    """Claim next task using a pessimistic row lock."""
    # SKIP LOCKED means competing workers skip rows already locked
    # rather than queuing behind them — critical for throughput.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`FOR UPDATE` acquires an exclusive row lock at SELECT time. `SKIP LOCKED` prevents the other workers from blocking — they move on to the next available row immediately. This is the idiomatic pattern for queue workers in Postgres.

## Recommended Defensive Addition (either option)

Add a `WHERE status = 'pending'` guard to `complete_task` as a secondary safeguard. It won't prevent the race but it will prevent a late-arriving duplicate completion from overwriting a legitimate result:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # ← guard added
        (json.dumps(result), datetime.utcnow(), task_id),
    )
    db.commit()
```

## Summary for the Call

| | |
|---|---|
| **Root cause** | Non-atomic read-then-write in `acquire_task` — SELECT and UPDATE are separate operations with no lock between them |
| **Trigger** | Scale-out from 1 → 4 workers created concurrent readers on the same queue rows |
| **Why tests missed it** | All tests are single-worker; the race requires concurrency to manifest |
| **Fix** | Make the claim atomic: either a conditional UPDATE with a subquery, or SELECT FOR UPDATE SKIP LOCKED |
| **Immediate mitigation** | Scale back to 1 worker to stop the bleeding while the fix is deployed |
