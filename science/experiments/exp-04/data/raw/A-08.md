# Root Cause: Unprotected Check-Then-Act on Task Acquisition

## The Race Condition

`acquire_task` is a two-statement sequence — a `SELECT` followed by a separate `UPDATE` — with no atomicity guarantee between them.

```python
# Statement 1: read
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# <-- every other worker can read the same row right here

# Statement 2: write
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
```

Between the `SELECT` and the `UPDATE`, the row is still `pending`. Any number of workers can read it, all get the same `id` back, and all proceed to claim and execute it. The `UPDATE` has no guard — it will succeed for every worker that reaches it because it only filters on `id`, not on the current status.

## Why It Didn't Show Up in Tests

The test suite runs a single worker against a single in-process database. There is no concurrency. The window between the `SELECT` and the `UPDATE` is never contested. All 34 tests pass cleanly and will continue to pass cleanly. They are not testing the thing that is broken.

## Why It Appeared Now

One worker, the race window exists but is never entered. Four workers under peak load, multiple workers are polling on the same 50ms sleep cycle, the queue is deep enough that the same pending row sits at the top for multiple consecutive reads, and the window gets hit repeatedly. Three days of stability followed by failure at scale is the fingerprint of this exact class of bug.

## The Fix

The claim must be atomic. The `UPDATE` itself needs to be the selection mechanism, and it needs to filter on `status = 'pending'` so only one writer can win.

**SQLite** (based on the codebase style):

```python
def acquire_task(worker_id: int):
    # Atomic claim: only the first writer to reach this row wins.
    # The WHERE clause on status = 'pending' is the guard.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Now read back only what this worker just claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at >= ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

The cleaner pattern if the database supports it is `UPDATE ... RETURNING`:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

**If the backing store is PostgreSQL**, use `FOR UPDATE SKIP LOCKED` which is purpose-built for this pattern:

```sql
SELECT id, type, payload
FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED
```

That runs inside a transaction with the subsequent `UPDATE`, and `SKIP LOCKED` means competing workers bypass rows already locked rather than queuing behind them.

## What to Verify After the Fix Ships

1. Confirm the completion log stops showing duplicate task IDs with different worker IDs.
2. Add a unique constraint or application-level assertion that a task ID appears exactly once in the completion log.
3. Add a concurrent test: two threads both call `acquire_task` against the same pending task and assert exactly one gets a non-None return.

## Immediate Mitigation if the Fix Isn't Ready

Scale back to one worker process. The race window still exists but it cannot be entered. It is not a fix — the bug is still there — but it stops the bleeding while the atomic claim is implemented and deployed.

**The visible bug in `execute_task`** — the O(n²) deduplication loop — is real and worth fixing, but it is not causing the incident. Do not touch it right now. Fix the acquisition window, deploy, confirm the duplicates stop, then address the rest in a follow-up.
