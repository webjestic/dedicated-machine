## Root Cause

**Classic check-then-act race condition in `acquire_task`.** There is no atomic claim on the row. Between the `SELECT` and the `UPDATE`, multiple workers can read the same pending task and all proceed to execute it.

The sequence that produces the duplicate completions:

```
Worker 1: SELECT → sees task 42, status = 'pending'
Worker 2: SELECT → sees task 42, status = 'pending'   ← both win the read
Worker 1: UPDATE task 42 SET status = 'running'
Worker 2: UPDATE task 42 SET status = 'running'        ← overwrites, no conflict
Worker 1: execute_task(42) → complete_task(42)
Worker 2: execute_task(42) → complete_task(42)         ← second completion logged
```

Both updates succeed because neither is conditional. The `WHERE id = ?` clause does not check the current status, so the second worker's `UPDATE` lands cleanly on a row that is already `running`. Both workers then execute and complete the task. The completion log gets two entries with different `worker_id` values. That is exactly what you are seeing.

This was invisible in tests because every test runs a single worker against an isolated database. The race only opens when two workers hit the queue simultaneously, which did not happen at one-process scale and only becomes probable under the peak load you hit this morning.

---

## The Fix

Make the claim atomic. The `UPDATE` must only succeed if the row is still `pending` at the moment of the write. Then check the row count to know whether this worker won the race.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task. Returns None if no task is available
    or if another worker claimed it between our SELECT and this UPDATE."""

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← the critical guard
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount

    db.commit()

    if rows_updated == 0:
        # Another worker claimed this task between our SELECT and this UPDATE.
        # Treat it as if no task was available and let the loop retry.
        return None

    return dict(row)
```

The only change that matters is `AND status = 'pending'` in the `WHERE` clause. If a second worker reaches the `UPDATE` after the first has already flipped the status to `running`, the condition is false, `rowcount` is zero, and that worker returns `None` and moves on. The task executes exactly once.

---

## What to Communicate on the Client Call

The duplicate completions are caused by a race condition introduced when the service was scaled to four workers. The code was written for single-process execution and has no mechanism to prevent two workers from claiming the same task simultaneously. The fix is a one-line change to the claim query — adding a status check to the `UPDATE` so that only one worker can win the claim. The fix is low-risk, requires no schema change, and does not affect any other code path.

---

## What to Watch After the Deploy

- Duplicate completion rate drops to zero immediately. If it does not, there is a second claim path you have not found yet.
- Watch for a transient increase in tasks staying `pending` longer than expected. That would mean the retry loop is not tight enough and workers are backing off too aggressively after a lost race. The `time.sleep(0.05)` should handle it, but confirm.
- The `execute_task` deduplication logic and the `complete_task` function are not the problem. Do not touch them in this deploy. One change, one risk surface.
