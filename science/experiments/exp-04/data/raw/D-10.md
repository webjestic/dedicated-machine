## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are not atomic, so multiple workers can claim the same task.**

Here's the exact sequence that produces duplicates:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')  ← still pending, not yet updated
Worker 1: UPDATE task #42 SET status = 'running', worker_id = 1
Worker 2: UPDATE task #42 SET status = 'running', worker_id = 2  ← overwrites Worker 1
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)  ← duplicate completion
```

Both workers read the row as `pending` before either has written `running`. Both proceed. Both complete. This is a classic **check-then-act** race.

---

## Why Tests Didn't Catch It

The tests run a single worker against an in-process database. There is no concurrency, so the race window never opens. The tests are correct for what they test — they just don't test the multi-worker scenario that was introduced three days ago when the service was scaled to 4 processes.

---

## The Fix

Use a single atomic statement with a conditional `WHERE` clause. The database's row-level locking guarantees only one writer wins.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task from the queue — atomically."""
    # Subquery identifies the candidate; WHERE status = 'pending' is the guard.
    # Only the one UPDATE that actually changes a row will return a row count > 0.
    # All other workers racing on the same task_id will match id but fail the
    # status check and update zero rows.
    row_id = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row_id:
        return None

    updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- the critical guard
        (worker_id, datetime.utcnow(), row_id["id"]),
    )
    db.commit()

    if updated.rowcount == 0:
        # Another worker claimed it between our SELECT and this UPDATE.
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row_id["id"],)
    )
```

The key change is `WHERE id = ? AND status = 'pending'` on the UPDATE. The database evaluates both conditions under a row lock. If a second worker reaches this UPDATE after the first has already flipped the status to `running`, it matches zero rows. `rowcount == 0` tells us we lost the race — return `None` and let the main loop move on.

## If You're on PostgreSQL or MySQL

Both support `SELECT ... FOR UPDATE SKIP LOCKED`, which is purpose-built for this pattern and performs better under high contention than the subquery approach above:

```sql
-- PostgreSQL / MySQL 8+
SELECT id, type, payload
FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

`FOR UPDATE` locks the selected row. `SKIP LOCKED` tells other workers to skip any row that's already locked rather than waiting — exactly the behavior you want in a job queue. Wrap the SELECT and UPDATE together in a transaction.
