## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

---

### The Failure Sequence

Here is exactly what is happening under load with four workers:

```
Time →

Worker 1: SELECT → gets task_id=42 (status='pending') ─────────────────────────────┐
Worker 2: SELECT → gets task_id=42 (status='pending') ──────────────────────┐       │
                                                                             │       │
Worker 2: UPDATE task_id=42 SET status='running', worker_id=2 ──────────────┘       │
Worker 1: UPDATE task_id=42 SET status='running', worker_id=1 ───────────────────────┘

Both workers now believe they own task_id=42.
Both call execute_task(). Both call complete_task().
Completion log shows task_id=42 twice with worker_id=1 and worker_id=2.
```

The window is small. Under low load with one worker, it never fires. Under peak load with four workers all polling the same queue, it fires regularly. This is why it ran clean for months and broke three days ago.

---

### Why the Tests Don't Catch It

Every test runs a single worker against an in-memory database in a single thread. There is no concurrency. The race window exists only when two processes execute the SELECT simultaneously, which a sequential test cannot produce. The tests are not wrong — they are testing the wrong thing for this failure mode. 34 green tests, production on fire. This is the exact shape of a concurrency bug.

---

### Why the Visible Bug Is Not the Root Cause

There is an O(n²) deduplication loop inside `execute_task`. Someone will see it. It is not causing the incident. Do not touch it yet. The duplicate completions are coming from two workers executing the same task, not from the summary logic. Fix the acquisition first.

---

### The Fix

The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter of ownership, not the application layer.

**Option 1 — Atomic UPDATE with RETURNING (PostgreSQL)**

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task. No separate SELECT."""
    row = db.query_one(
        """
        UPDATE task_queue
        SET    status = 'running',
               worker_id = ?,
               claimed_at = ?
        WHERE  id = (
            SELECT id FROM task_queue
            WHERE  status = 'pending'
            ORDER  BY priority DESC, created_at ASC
            LIMIT  1
            FOR UPDATE SKIP LOCKED   -- critical: skip rows locked by other workers
        )
        RETURNING id, type, payload
        """,
        (worker_id, datetime.utcnow()),
    )

    if not row:
        return None

    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is the key. When Worker 2 hits the subquery, any row Worker 1 is already updating is invisible to it. They cannot select the same row. This is the standard pattern for queue consumers in PostgreSQL and it is what this code needs.

**Option 2 — Optimistic locking with a WHERE guard (works on SQLite, MySQL)**

```python
def acquire_task(worker_id: int):
    """Claim a task only if it is still pending at the moment of update."""
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # guard condition
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, rows_affected == 0. Walk away.
    if rows_affected == 0:
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means the UPDATE only lands if the row has not been claimed since the SELECT. Exactly one worker will see `rows_affected == 1`. All others see zero and return None. This is safe but produces contention under high concurrency — multiple workers race, one wins, the rest burn a cycle. Option 1 is cleaner at scale.

---

### What to Tell the Junior Engineers

> The SELECT and the UPDATE in `acquire_task` are two separate database round-trips with no lock between them. Multiple workers read the same pending row before any of them write to it. All of them claim the task. All of them complete it. The fix is to make claim and select one atomic operation so the database enforces exclusive ownership. Use Option 1 if you are on Postgres. Use Option 2 if you are not. Do not ship either without a concurrent integration test that spins up four workers against a shared database and asserts that each task ID appears exactly once in the completion log.

That test would have caught this before the scale-up. Write it now so it catches the next one.
