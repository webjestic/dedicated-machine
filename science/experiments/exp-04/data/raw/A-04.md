# Root Cause

**Classic check-then-act race condition. The SELECT and UPDATE in `acquire_task` are not atomic.**

---

## What's Happening

```python
# Worker A and Worker B both execute this SELECT at nearly the same time
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"          # <-- both see status = 'pending'
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Both get back the same row — same task ID

# Worker A updates first
db.execute("UPDATE task_queue SET status = 'running' ... WHERE id = ?", ...)
db.commit()

# Worker B updates a fraction of a second later — this succeeds
# because there is no guard on the UPDATE itself
db.execute("UPDATE task_queue SET status = 'running' ... WHERE id = ?", ...)
db.commit()
```

Both workers now believe they own the task. Both call `execute_task`. Both call `complete_task`. The completion log gets two entries with different `worker_id` values. Downstream consumers see the payload twice.

**The window is the gap between the SELECT and the UPDATE.** At one worker, that window never mattered. At four workers under peak load, threads are hitting it constantly.

---

## Why Tests Didn't Catch It

Every test runs a single worker against an isolated database. There is no concurrent access. The race window exists only when two workers reach the SELECT within the same scheduling slice. Sequential tests cannot produce that condition. All 34 tests pass. The bug is invisible until you scale.

---

## What Needs to Change

The fix is to make the claim atomic — the check and the act must be a single database operation with no gap between them.

### Option 1 — `UPDATE ... WHERE status = 'pending'` with row count check (simplest, works on SQLite and Postgres)

```python
def acquire_task(worker_id: int):
    # Atomically claim the highest-priority pending task.
    # Only one writer can transition a given row from pending → running.
    # Any concurrent attempt hits the WHERE status = 'pending' guard and claims 0 rows.
    
    # First, identify the candidate task ID
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt to claim it. The WHERE clause is the guard.
    # If another worker already flipped this row to 'running', affected_rows = 0.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",   # <-- the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Lost the race. Another worker claimed it. Return None, loop will retry.
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],)
    )
```

The key line is `WHERE id = ? AND status = 'pending'`. The database enforces mutual exclusion. Exactly one UPDATE will match. Every other concurrent attempt sees `rowcount == 0` and backs off.

### Option 2 — `SELECT ... FOR UPDATE SKIP LOCKED` (Postgres, preferred at scale)

```sql
SELECT id, type, payload FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED
```

This locks the selected row at read time and skips rows already locked by other workers. No two workers can select the same row. Cleaner, but requires Postgres and an explicit transaction wrapping both the SELECT and the UPDATE.

---

## Immediate Recommendation

**Ship Option 1 now.** It works with whatever database is already in production, requires minimal change, and the `rowcount == 0` path is easy to test with two concurrent threads. Option 2 is the right long-term answer if you're on Postgres — put it in the next sprint.

One more thing: add a `claimed_at` timeout job that resets stale `running` tasks back to `pending`. If a worker dies mid-execution, that task is currently stuck forever. That's the next failure waiting to happen.
