## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

---

### The Failure Sequence

Here is exactly what is happening under four workers at peak load:

```
Time →

Worker 1: SELECT → sees task #42, status='pending'
Worker 2: SELECT → sees task #42, status='pending'   ← wins the same race
Worker 1:                                UPDATE task #42 SET status='running', worker_id=1
Worker 2:                                UPDATE task #42 SET status='running', worker_id=2
Worker 1: execute_task(#42) ...
Worker 2: execute_task(#42) ...
Worker 1: complete_task(#42) → writes result, status='complete'
Worker 2: complete_task(#42) → overwrites result, status='complete'
```

Both workers cleared the `status = 'pending'` filter on their SELECT. Neither UPDATE had a condition that would have rejected the second writer. The task runs twice, completes twice, and downstream consumers see both completion events.

This is why the completion log shows the same task ID with two different worker IDs. That is not a logging artifact. That is the exact timestamp window between the SELECT and the UPDATE, materialized.

---

### Why the Tests Did Not Catch This

Every test runs a single worker against an isolated database. There is no concurrency. The race window is never opened. All 34 tests pass and will continue to pass against this code indefinitely. The test suite is not wrong — it is testing the wrong threat model. It validates correctness under sequential access. The bug only exists under concurrent access.

---

### Why the Visible Bugs Are Not the Root Cause

There is an O(n²) deduplication loop in `execute_task`. There is no error handling around `execute_task`. Both are real problems. Neither one is why tasks are completing twice. Do not touch those yet. Fix the race first.

---

### The Fix

The SELECT and UPDATE must be atomic. The row must be claimed in a single operation that no other worker can interleave with.

**Option 1 — Atomic UPDATE with a guarded WHERE clause (preferred for most databases)**

```python
def acquire_task(worker_id: int):
    # Claim the task in one atomic statement.
    # The WHERE status = 'pending' acts as the guard —
    # only one UPDATE will match and modify the row.
    # Any concurrent worker hitting the same row gets 0 rows affected.
    
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker claimed it between our SELECT and UPDATE.
        # Bail out. We will pick up the next task on the next loop iteration.
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],),
    )
```

The `AND status = 'pending'` on the UPDATE is the critical change. If Worker 2 reaches its UPDATE after Worker 1 has already flipped the row to `'running'`, the WHERE clause matches zero rows. `rows_affected` is 0. Worker 2 returns None and moves on. The task runs exactly once.

**Option 2 — `SELECT ... FOR UPDATE` with a transaction (if your database supports it)**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED prevents queuing behind the lock
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE` takes an exclusive row-level lock at SELECT time. No other transaction can read or modify that row until this transaction commits. `SKIP LOCKED` means other workers skip locked rows instead of blocking on them, which is exactly the behavior you want in a queue — move to the next available task, do not queue up behind a lock.

---

### Which Option to Ship Right Now

If your database is PostgreSQL, ship Option 2. `SELECT FOR UPDATE SKIP LOCKED` is purpose-built for this pattern and is well-tested at scale.

If your database is SQLite (which the test setup suggests is possible given the `?` placeholders), SQLite's locking model is database-level, not row-level. In that case Option 1 is your path, and you should also evaluate whether SQLite is the right database for a multi-worker queue at all — it almost certainly is not.

---

### What to Tell the Junior Engineers

> The SELECT and the UPDATE in `acquire_task` are two separate database round-trips with no lock held between them. Multiple workers can SELECT the same pending row before any of them UPDATE it. Add `AND status = 'pending'` to the UPDATE's WHERE clause and check rows affected. If it is zero, return None. That closes the window. The task will only be claimed by the first worker whose UPDATE lands.

That is the change. One line added to the WHERE clause, one rows-affected check. Everything else is follow-up work.
