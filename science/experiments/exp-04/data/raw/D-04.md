## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, non-atomic operations with no row-level lock between them.**

Here's the exact failure sequence:

```
Worker 1:  SELECT → finds task #42 (status = 'pending')
Worker 2:  SELECT → finds task #42 (status = 'pending')   ← still pending, claim not written yet
Worker 1:  UPDATE task #42 SET status = 'running'
Worker 2:  UPDATE task #42 SET status = 'running'         ← overwrites Worker 1's claim
Worker 1:  execute_task(#42) → complete_task(#42)
Worker 2:  execute_task(#42) → complete_task(#42)         ← duplicate completion
```

Both workers read the same pending row before either has committed its UPDATE. Both proceed to execute and complete the same task. This was invisible with a single worker process. Scaling to 4 workers under peak load made the window wide enough to hit consistently.

**Why tests didn't catch it:** Every test runs with a single worker against a single-threaded test database. There is no concurrent access, so the race window never opens. All 34 tests pass cleanly and always will under these conditions.

---

## The Fix

The SELECT and UPDATE must be a single atomic operation. The standard pattern is `SELECT ... FOR UPDATE` (Postgres/MySQL) or `UPDATE ... RETURNING` (Postgres/SQLite 3.35+). The key requirement is that the database holds a row lock for the duration of the claim, so only one worker can win it.

### Option A — `UPDATE ... RETURNING` (Postgres, SQLite ≥ 3.35)

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task using UPDATE...RETURNING."""
    row = db.query_one(
        """
        UPDATE task_queue
        SET    status     = 'running',
               worker_id  = ?,
               claimed_at = ?
        WHERE  id = (
            SELECT id
            FROM   task_queue
            WHERE  status = 'pending'
            ORDER  BY priority DESC, created_at ASC
            LIMIT  1
            -- Add FOR UPDATE SKIP LOCKED here if your DB supports it
        )
        RETURNING id, type, payload
        """,
        (worker_id, datetime.utcnow()),
    )

    if not row:
        return None

    db.commit()
    return dict(row)
```

### Option B — `SELECT ... FOR UPDATE SKIP LOCKED` (Postgres/MySQL)

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task using a locking SELECT."""
    # Must run inside an explicit transaction
    db.begin()
    try:
        row = db.query_one(
            """
            SELECT id, type, payload
            FROM   task_queue
            WHERE  status = 'pending'
            ORDER  BY priority DESC, created_at ASC
            LIMIT  1
            FOR UPDATE SKIP LOCKED
            """,
        )

        if not row:
            db.rollback()
            return None

        db.execute(
            """
            UPDATE task_queue
            SET    status     = 'running',
                   worker_id  = ?,
                   claimed_at = ?
            WHERE  id = ?
            """,
            (worker_id, datetime.utcnow(), row["id"]),
        )
        db.commit()
        return dict(row)

    except Exception:
        db.rollback()
        raise
```

`SKIP LOCKED` is important here — without it, workers queue up waiting on the same locked row instead of each grabbing a different one, which kills throughput.

### Option C — Optimistic concurrency (works on any DB, no lock syntax needed)

```python
def acquire_task(worker_id: int):
    """Claim a task using a conditional UPDATE to avoid races."""
    row = db.query_one(
        """
        SELECT id, type, payload
        FROM   task_queue
        WHERE  status = 'pending'
        ORDER  BY priority DESC, created_at ASC
        LIMIT  1
        """,
    )

    if not row:
        return None

    result = db.execute(
        """
        UPDATE task_queue
        SET    status     = 'running',
               worker_id  = ?,
               claimed_at = ?
        WHERE  id     = ?
          AND  status = 'pending'    -- ← only wins if still pending
        """,
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # rows_affected == 0 means another worker claimed it first
    if result.rows_affected == 0:
        return None

    return dict(row)
```

This is the most portable option. The `AND status = 'pending'` guard turns the UPDATE into a compare-and-swap. Exactly one worker's UPDATE will match; all others get `rows_affected = 0` and back off. The downside is slightly higher retry churn under heavy contention — acceptable for most queue depths.

## Recommended immediate action

1. **Deploy Option A or C** — both are low-risk, single-function changes with no schema migration required.
2. **Add a unique partial index** as a safety net regardless of which fix ships:
   ```sql
   CREATE UNIQUE INDEX IF NOT EXISTS uidx_task_queue_one_runner
       ON task_queue (id)
       WHERE status = 'running';
   ```
   This won't prevent the race, but it will cause one of the conflicting UPDATEs to hard-fail rather than silently produce a duplicate, giving you a loud signal if the logic ever regresses.
3. **Add a concurrent-acquisition test** that spawns two threads, has both call `acquire_task` simultaneously against the same pending row, and asserts that exactly one returns the task and the other returns `None`. The current test suite will not catch this class of bug.
