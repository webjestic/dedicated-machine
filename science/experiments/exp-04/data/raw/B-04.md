## Root Cause

**Classic check-then-act race condition. The SELECT and UPDATE in `acquire_task` are not atomic. Multiple workers can read the same pending row before any of them writes the claim.**

---

### The Failure Sequence

Here is exactly what is happening under load with 4 workers:

```
Time →

Worker 1:  SELECT (sees task 42, status='pending') ─────────────────── UPDATE task 42 → 'running' → COMMIT
Worker 2:  SELECT (sees task 42, status='pending') ──── UPDATE task 42 → 'running' → COMMIT
                                                    ↑
                                              Both workers read 'pending'
                                              before either write lands.
                                              Both proceed. Both execute.
                                              Both complete. Task logged twice.
```

The window is the gap between the `query_one` and the `execute`. That gap is small. Under low concurrency with one worker it never mattered. Under peak load with four workers racing the same queue it hits constantly.

The UPDATE does not protect you here. It runs after the read. By the time Worker 2's UPDATE commits, Worker 1 has already passed its own read and is already executing the task. The second UPDATE just overwrites `worker_id` and `claimed_at`. Both workers finish. Both call `complete_task`. Both write to the completion log. Downstream sees the payload twice.

---

### Why the Tests Did Not Catch This

Every test runs a single worker against a single-process in-memory database. There is no concurrency. The check-then-act window exists in the code but is never exercised. All 34 tests pass. The bug ships.

This is the exact failure mode where a green test suite gives false confidence. The tests validate the logic of each function in isolation. They do not test the invariant that matters in production: **only one worker claims a given task.**

---

### What Needs to Change

The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter of the claim, not the application layer.

**Option 1 — Conditional UPDATE with row check (works on SQLite and most databases)**

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()

    # Subquery identifies the candidate. UPDATE claims it atomically.
    # If another worker claimed it first, affected_rows == 0.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, claimed_at),
    )
    db.commit()

    if affected.rowcount == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
    return dict(row) if row else None
```

There is no longer a window. The UPDATE is the claim. Either your worker's UPDATE wins the row or it does not. The database serializes concurrent writes to the same row. The application never sees a task it did not atomically own.

**Option 2 — `SELECT FOR UPDATE` with an explicit transaction (PostgreSQL, MySQL)**

If you migrate off SQLite this is the cleaner pattern:

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED drops contended rows immediately
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is purpose-built for this pattern. It locks the candidate row for the duration of the transaction and skips rows already locked by other workers. No contention, no waiting, no duplicate claims.

### The Test That Should Exist

After the fix, add this:

```python
def test_acquire_task_is_exclusive(db, seed_tasks):
    """Two concurrent workers must not claim the same task."""
    from concurrent.futures import ThreadPoolExecutor

    with ThreadPoolExecutor(max_workers=2) as pool:
        f1 = pool.submit(acquire_task, worker_id=1)
        f2 = pool.submit(acquire_task, worker_id=2)
        t1 = f1.result()
        t2 = f2.result()

    claimed_ids = [t["id"] for t in [t1, t2] if t is not None]
    assert len(claimed_ids) == len(set(claimed_ids)), (
        "Two workers claimed the same task"
    )
```

This test would have caught the bug before it shipped. Add it now so the next scale event does not produce another incident.

### Summary for the Team

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`. SELECT and UPDATE are separate statements with no lock between them. |
| **Trigger** | Scaling to 4 workers created concurrent readers. The race window that always existed became a near-certainty under peak load. |
| **Why tests missed it** | All tests are single-worker. Concurrency is never exercised. |
| **Fix** | Collapse the SELECT and UPDATE into one atomic statement. The database owns the claim decision, not the application. |
| **Immediate action** | Deploy Option 1 now. It works against your current database with no infrastructure change. |

The duplicate deduplication logic in `execute_task` and the inefficient linear scan in that same loop are real issues but they are not why you are paged right now. Do not touch them until this is deployed and the incident is closed.
