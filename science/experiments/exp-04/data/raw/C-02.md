## Root Cause

**Classic check-then-act race condition in `acquire_task`.** The SELECT and the UPDATE are two separate operations with no atomicity guarantee between them. Under a single worker, this never fires. Under four concurrent workers, it fires under load.

---

### The Failure Sequence

Here is what the completion log is showing you, reconstructed:

```
T=0ms  Worker-2: SELECT → finds task #847 (status='pending')
T=0ms  Worker-4: SELECT → finds task #847 (status='pending')   ← same row, same instant
T=1ms  Worker-2: UPDATE task #847 SET status='running', worker_id=2
T=1ms  Worker-4: UPDATE task #847 SET status='running', worker_id=4  ← overwrites
T=1ms  Worker-2: COMMIT
T=1ms  Worker-4: COMMIT
       ... both workers now execute and complete task #847 ...
T=800ms Worker-2: complete_task(847, result_A)
T=801ms Worker-4: complete_task(847, result_B)
```

Both workers read `status = 'pending'` before either write lands. Both proceed. The second UPDATE does not block the second worker — it just silently reassigns `worker_id`. Neither worker checks whether the claim actually succeeded. This is the unprotected window.

---

### Why the Tests Don't Catch It

The test suite runs one worker against an isolated in-process database. There is no concurrency. `test_acquire_task_marks_task_running` proves the UPDATE fires — it does not prove that two concurrent workers cannot both pass the SELECT before either UPDATE commits. The tests are correct for what they test. They test the wrong threat model.

---

### The Broken Code

```python
def acquire_task(worker_id: int):
    # WINDOW OPENS HERE
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # WINDOW CLOSES HERE — but four workers can all be inside it simultaneously
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)  # returned regardless of whether this worker won the race
```

The function returns the task unconditionally. It never asks: *did my UPDATE actually claim this row, or did someone else get there first?*

---

### The Fix

The solution is to make the claim atomic. The exact mechanism depends on your database, but the pattern is the same: **do not SELECT then UPDATE — UPDATE with a condition and check the row count.**

```python
def acquire_task(worker_id: int):
    """
    Claim a task atomically. The UPDATE itself is the claim — we only proceed
    if our UPDATE matched a row that was still 'pending' at the moment we wrote.
    """
    # Subquery finds the candidate; outer UPDATE claims it only if status is
    # still 'pending' at write time. One winner. All others get rowcount=0.
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",          # re-check inside the UPDATE
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if rows_affected == 0:
        return None

    # Only fetch the row we just claimed — keyed to this worker_id and
    # claimed_at to avoid any ambiguity.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The `AND status = 'pending'` predicate inside the UPDATE is the lock. The database evaluates it atomically at write time. If two workers race, one gets `rows_affected = 1` and proceeds. The other gets `rows_affected = 0` and returns `None`. The SELECT that follows only reads a row this worker already owns — it is not a second claim attempt.

**If your database supports `RETURNING`** (PostgreSQL, SQLite ≥ 3.35), the fetch after the UPDATE collapses into one round trip and removes even the theoretical gap between the UPDATE and the re-SELECT:

```python
row = db.query_one(
    "UPDATE task_queue"
    " SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ("
    "   SELECT id FROM task_queue"
    "   WHERE status = 'pending'"
    "   ORDER BY priority DESC, created_at ASC"
    "   LIMIT 1"
    " )"
    " AND status = 'pending'"
    " RETURNING id, type, payload",
    (worker_id, datetime.utcnow()),
)
db.commit()
return dict(row) if row else None
```

One statement. Either you get the row back or you get nothing. No window.

### Secondary Hardening

Once the acquisition is fixed, add a guard in `complete_task` so a late-arriving duplicate cannot overwrite a result that already landed:

```python
def complete_task(task_id: int, result: list) -> None:
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # only complete what we own
        (json.dumps(result), datetime.utcnow(), task_id),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker already completed this task. Log and do not re-emit.
        log.warning("complete_task: task %s already completed, discarding result", task_id)
```

This does not fix the root cause — it is a backstop. Defense in depth. The root cause fix is in `acquire_task`.

### What to Tell the Team Right Now

The immediate mitigation if you cannot ship the fix in the next hour: **scale back to one worker.** The race cannot fire with a single process. You lose throughput but you stop the duplicate completions while the fix is reviewed and deployed.

The visible things in this code — the O(n²) deduplication loop in `execute_task`, the missing null checks — are real and worth fixing. They are not why the pagers went off. The race condition was always there. It was invisible at one worker and inevitable at four.
