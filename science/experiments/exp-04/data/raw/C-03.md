# Root Cause: Check-Then-Act Race Condition in `acquire_task`

## What's Happening

The incident started three days ago when you scaled to 4 workers. The tests all pass. The code looks reasonable. Neither of those facts matter right now, because the bug isn't in the logic — it's in the **gap between the SELECT and the UPDATE**.

Here's the sequence that produces a duplicate completion:

```
Time  Worker 1                              Worker 2
────  ──────────────────────────────────    ──────────────────────────────────
T1    SELECT ... WHERE status = 'pending'
T2                                          SELECT ... WHERE status = 'pending'
T3    ← both read the same row, id=42 →
T4    UPDATE ... SET status = 'running'
      WHERE id = 42
T5                                          UPDATE ... SET status = 'running'
                                            WHERE id = 42
T6    execute_task(42) ...
T7                                          execute_task(42) ...
T8    complete_task(42, result_A)
T9                                          complete_task(42, result_B)
```

Both workers read `id=42` while it's still `pending`. Both issue the UPDATE. Both succeed — **there is nothing in the UPDATE clause that prevents this**. The WHERE condition is `WHERE id = ?`, not `WHERE id = ? AND status = 'pending'`. Even if you added that condition, you'd still have a window between T1 and T4 where a second worker can slip through.

This is a textbook **TOCTOU** (Time of Check to Time of Use) race. The check (SELECT) and the act (UPDATE) are two separate statements with no lock held between them.

---

## Why the Tests Don't Catch It

The tests run a single worker against an in-process database. There is no concurrency. The race window doesn't exist in that environment. **34 passing tests told you the logic is correct. They said nothing about what happens under concurrent access.** This is the gap.

---

## The Fix

You need to collapse the check and the act into a single atomic operation. The SELECT must not exist as a standalone step. The claim must happen in one statement that only succeeds if the row is still available.

### Option 1: Atomic UPDATE with status guard (SQLite / PostgreSQL)

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    # Find the candidate in a subquery, claim it in the same statement.
    # The WHERE status = 'pending' in the outer UPDATE is the guard —
    # only one writer can transition a row out of 'pending'.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This works because the database serializes concurrent writes to the same row. Only one UPDATE can transition `status` from `pending` to `running`. The second worker's UPDATE finds no matching row and returns nothing. No duplicate claim.

### Option 2: Optimistic locking with a version column

If your database or driver doesn't support `UPDATE ... RETURNING`, add a `version` column and use it as a compare-and-swap guard:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload, version FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    # The WHERE clause now includes version — if another worker
    # already claimed this row, affected_rows will be 0.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?,"
        "     version = version + 1"
        " WHERE id = ? AND status = 'pending' AND version = ?",
        (worker_id, datetime.utcnow(), row["id"], row["version"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Lost the race — another worker claimed it first. That's fine.
        return None

    return dict(row)
```

The `rowcount == 0` path is not an error. It's the race condition resolving correctly.

### Option 3: Advisory lock / SELECT FOR UPDATE (PostgreSQL)

If you're on PostgreSQL, `SELECT FOR UPDATE SKIP LOCKED` is purpose-built for this pattern:

```python
def acquire_task(worker_id: int):
    # SKIP LOCKED means: don't wait on rows another transaction has locked,
    # just move to the next available one. This is the canonical queue pattern.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )
    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

This requires both statements to run inside the same transaction, which your `db` abstraction needs to support.

## What to Do Right Now

**Immediate mitigation:** Scale back to 1 worker. The race window still exists with 1 worker but it cannot be triggered without concurrency. This stops the bleeding while you ship the fix.

**Before you ship the fix:** Add a test that actually exercises concurrent acquisition:

```python
def test_acquire_task_no_duplicate_claims(db, seed_tasks):
    """Two workers racing for the same task must not both claim it."""
    from concurrent.futures import ThreadPoolExecutor

    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(acquire_task, wid) for wid in (1, 2)]
        results = [f.result() for f in futures]

    claimed = [r for r in results if r is not None]
    # Exactly one worker should have claimed the task.
    assert len(claimed) == 1
```

This test will fail against the current code. It should pass after your fix. That's the regression test you want in CI before you scale back up.

## The Visible Bugs You Should Not Fix First

There's an O(n²) deduplication loop in `execute_task` and the `complete_task` function has no guard against writing a completion over a row that was already completed by a racing worker. Both are real problems. Neither is why downstream consumers are processing payloads twice right now. Fix the acquisition race first, then write the postmortem, then clean up the rest.
