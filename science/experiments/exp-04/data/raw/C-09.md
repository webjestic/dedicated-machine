# Root Cause Analysis

## The Short Answer

**There is no atomic claim.** Two workers can read the same pending task simultaneously, both see `status = 'pending'`, and both proceed to execute it. The SELECT and the UPDATE are two separate operations with nothing protecting the window between them.

---

## The Failure Sequence

Here is exactly what the completion log is showing you:

```
T+0ms   Worker 1: SELECT ... WHERE status = 'pending' → returns task_id=42
T+0ms   Worker 3: SELECT ... WHERE status = 'pending' → returns task_id=42
T+1ms   Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
T+1ms   Worker 3: UPDATE task_queue SET status = 'running' WHERE id = 42
T+800ms Worker 1: complete_task(42, result_A)
T+801ms Worker 3: complete_task(42, result_B)
```

Both workers passed the check. Both workers claimed the task. Both workers completed it. The completion log shows task 42 twice with different worker IDs because that is precisely what happened.

This is a classic **check-then-act** race condition. The check (`WHERE status = 'pending'`) and the act (`UPDATE ... SET status = 'running'`) are not atomic. Under a single worker, the loop is sequential and this window never matters. Under four workers running concurrently, it gets hit constantly at peak load. The timeline fits exactly: ran fine for months, broke three days after scaling, worsened this morning under load.

---

## Why the Tests Tell You Nothing

The tests pass because they run one worker at a time against an in-process database. There is no concurrency. The race window exists in time, and the test suite never opens it.

```python
# This test proves the happy path works sequentially.
# It says nothing about two workers calling this simultaneously.
def test_acquire_task_marks_task_running(db, seed_tasks):
    task = acquire_task(worker_id=1)
    row = db.query_one("SELECT status FROM task_queue WHERE id = ?", (task["id"],))
    assert row["status"] == "running"
```

34 green tests, production on fire. This is why you cannot test your way out of a concurrency bug with sequential tests.

---

## The Fix

The SELECT and UPDATE must become a single atomic operation. The database must do both under a lock, not your application code.

### Option 1: Atomic UPDATE with a subquery (preferred, no schema change)

```python
def acquire_task(worker_id: int):
    """Claim a task atomically. Only one worker can claim a given task."""
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only fetch the row this worker actually claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'running' AND worker_id = ?"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The database executes the UPDATE atomically. If two workers race, one of them updates zero rows. The other gets the task. There is no window.

### Option 2: Optimistic locking with a version column (if you want explicit conflict visibility)

```sql
ALTER TABLE task_queue ADD COLUMN version INTEGER NOT NULL DEFAULT 0;
```

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload, version FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?, version = version + 1"
        " WHERE id = ? AND version = ? AND status = 'pending'",
        (worker_id, datetime.utcnow(), row["id"], row["version"]),
    ).rowcount
    db.commit()

    if rows_updated == 0:
        # Another worker claimed it first. This is not an error, just contention.
        return None

    return dict(row)
```

The `AND version = ?` condition means the UPDATE only succeeds if nobody has touched the row since you read it. Zero rows updated means you lost the race. Return `None` and let the loop retry. This also gives you a metric: count the retries and you know exactly how much contention you have.

### Option 3: `SELECT FOR UPDATE` (if your database and connection pool support it)

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED prevents queue head blocking
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE` locks the row at read time. `SKIP LOCKED` means other workers skip rows already locked rather than waiting, which is exactly the behavior you want in a queue. Note: requires PostgreSQL 9.5+ or MySQL 8.0+. Check your database before reaching for this.

---

## What to Do Right Now

**Immediate mitigation:** Scale back to 1 worker process. This closes the race window without a code change. Downstream consumers need to deduplicate on task ID in the meantime — that is a symptom treatment, not a fix, but it stops the bleeding.

**Today:** Ship Option 1 or Option 2. Option 1 is the smallest diff. Option 2 gives you observability into contention going forward.

**After the incident:** Add a concurrency test. Spin up multiple threads, have them all call `acquire_task` simultaneously against a shared test database, assert that each task ID appears in exactly one result. That test would have caught this before it shipped.

---

## The Secondary Issue

While you are in this code: `execute_task` has an O(n²) deduplication loop — a linear scan through `summary` for every row appended. It will not cause duplicate completions, but it will cause slow completions under large payloads, which increases the time two workers spend executing the same task concurrently and makes the race window wider. Replace it with a set keyed on `(user_id, event_type)`. Fix the race first. Fix this second. Do not let this distract you from the actual incident.
