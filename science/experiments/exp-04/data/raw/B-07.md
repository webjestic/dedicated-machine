## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

Here is the exact failure sequence:

```
Worker 1: SELECT → sees task 42, status = 'pending'
Worker 2: SELECT → sees task 42, status = 'pending'   ← both read before either writes
Worker 1: UPDATE task 42 SET status = 'running', worker_id = 1
Worker 2: UPDATE task 42 SET status = 'running', worker_id = 2   ← overwrites worker 1
Worker 1: execute_task(42) → complete_task(42)
Worker 2: execute_task(42) → complete_task(42)
```

Both workers claimed the same task. Both completed it. That is your duplicate in the completion log with two different worker IDs. This was always in the code. It did not matter with one worker. With four workers under peak load, the window gets hit constantly.

---

## Why the Tests Did Not Catch It

Every test runs a single worker against an in-process database with no concurrency. The race window does not exist in that environment. All 34 tests pass. The bug is invisible to them by construction. This is not a test quality failure — it is a coverage gap for concurrent behavior that requires a different class of test entirely.

---

## What Needs to Change

The SELECT and the UPDATE must be collapsed into a single atomic operation. The database must be the arbiter of who owns the task, not the application layer.

### Fix for SQLite

```python
def acquire_task(worker_id: int):
    """
    Claim the next pending task atomically.
    The WHERE clause on status = 'pending' ensures only one writer wins.
    """
    now = datetime.utcnow()

    # First, find the candidate task ID
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt to claim it. The WHERE id = ? AND status = 'pending'
    # is the guard. If another worker already claimed it, affected
    # rows will be 0 and we walk away empty-handed.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",
        (worker_id, now, row["id"]),
    ).rowcount
    db.commit()

    if affected == 0:
        # Another worker won the race on this task. Return None
        # and let the polling loop try again on the next cycle.
        return None

    return dict(row)
```

The critical change is `WHERE id = ? AND status = 'pending'` on the UPDATE. The database executes this atomically under its row-level lock. Exactly one UPDATE will match and return `rowcount = 1`. Every other concurrent UPDATE against the same row will see `status = 'running'` and return `rowcount = 0`. The application then checks that count and discards the miss. The SELECT above it is now just a hint, not a claim.

### If You Are on PostgreSQL (preferred path)

Collapse it into one statement and eliminate the two-trip window entirely:

```sql
UPDATE task_queue
SET status = 'running', worker_id = $1, claimed_at = $2
WHERE id = (
    SELECT id FROM task_queue
    WHERE status = 'pending'
    ORDER BY priority DESC, created_at ASC
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING id, type, payload;
```

`FOR UPDATE SKIP LOCKED` is purpose-built for this pattern. Workers skip rows already locked by a peer rather than blocking on them. No application-level rowcount check needed. This is the correct long-term solution if you migrate off SQLite.

## What to Do Right Now

1. **Scale back to one worker immediately.** The race window is proportional to the number of concurrent workers. One worker closes it completely while the fix is prepared.

2. **Identify and deduplicate the already-affected tasks.** Query for any task with `status = 'complete'` that appears more than once in the completion log, or directly:

```sql
SELECT task_id, COUNT(*) as completions, array_agg(worker_id) as workers
FROM completion_log
GROUP BY task_id
HAVING COUNT(*) > 1;
```

Coordinate with the downstream consumers on which completion record is authoritative before they reprocess.

3. **Deploy the fix, then restore worker count.**

4. **Add a concurrency integration test** that spawns two workers simultaneously against a shared database and asserts each task ID appears exactly once in the completion log. The unit test suite will never catch this class of bug.

## The Invisible Bug in `execute_task`

While you are in this code: the deduplication loop in `execute_task` is O(n²) — for every event row it scans the entire summary list. It will not cause an incident today but it will cause a latency incident the first time a task has a large `user_ids` list. Replace the `any(...)` scan with a set of seen keys. Not the fire you are fighting right now, but put it in the backlog before you close this ticket.
