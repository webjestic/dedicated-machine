## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can SELECT the same row.**

---

### The Failure Sequence

Here is exactly what is happening under load with four workers:

```
Time →

Worker 1: SELECT → gets row id=42 (status='pending')
Worker 2: SELECT → gets row id=42 (status='pending')   ← same row, still pending
Worker 1:          UPDATE id=42 SET status='running'
Worker 2:          UPDATE id=42 SET status='running'   ← overwrites, no error raised
Worker 1:                           execute_task(42) → complete_task(42)
Worker 2:                           execute_task(42) → complete_task(42)
```

Both workers see `pending`, both claim it, both execute it, both write to the completion log. The UPDATE does not fail — it just silently overwrites. No exception is raised. No test catches it. The task appears twice in the completion log with two different `worker_id` values, which is exactly what you are seeing.

---

### Why the Tests Miss It

Every test runs with a single worker against a single-process test database. The race window does not exist in that environment. All 34 tests pass. The bug is invisible until you have concurrent workers sharing a real database connection — which is exactly what happened three days ago when you scaled to four processes.

The tests validate the *behavior* of each function in isolation. They do not validate *atomicity* of the claim operation under concurrency. That is the gap.

---

### Why the Visible Bugs Are Not the Root Cause

There is an O(n²) deduplication loop in `execute_task`. There is no timeout handling on stalled `running` tasks. Both are real problems. Neither one causes duplicate completions. Do not touch them yet. Fix the race first.

---

### What Needs to Change

The SELECT and the UPDATE must become a single atomic operation. The database must be the arbiter of which worker owns a task. There are two viable approaches depending on your database.

**Option 1 — `UPDATE ... WHERE status = 'pending'` with row-level locking (preferred)**

If you are on PostgreSQL:

```sql
UPDATE task_queue
SET status = 'running', worker_id = ?, claimed_at = ?
WHERE id = (
    SELECT id FROM task_queue
    WHERE status = 'pending'
    ORDER BY priority DESC, created_at ASC
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING id, type, payload
```

`FOR UPDATE SKIP LOCKED` is the key. It locks the candidate row during the subquery so no other worker can select it. `SKIP LOCKED` means competing workers move on to the next available row instead of blocking. This is the standard pattern for queue workers in Postgres.

The Python implementation becomes:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        "   FOR UPDATE SKIP LOCKED"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

One round trip. One atomic operation. The database guarantees exactly one worker gets each row.

**Option 2 — Optimistic locking with a status guard on the UPDATE (if you cannot use `FOR UPDATE`)**

If you are on SQLite or a database that does not support `FOR UPDATE SKIP LOCKED`, add a `WHERE status = 'pending'` guard to the UPDATE and check whether a row was actually modified:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",   # ← guard added
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if db.rowcount == 0:
        # Another worker claimed it between our SELECT and UPDATE
        return None

    return dict(row)
```

This is narrower than Option 1 but still has a residual window on SQLite without WAL mode and proper isolation. Option 1 is the correct fix if your database supports it.

---

### What to Tell the Junior Engineers

> The SELECT and UPDATE in `acquire_task` are two separate database calls with no lock between them. Four workers can all SELECT the same pending row before any of them Updates it. All four claim it. All four execute it. Fix: make the claim a single atomic operation using `UPDATE ... WHERE id = (SELECT ... FOR UPDATE SKIP LOCKED) RETURNING`. The database becomes the lock. No application-level coordination needed.

---

### After the Fix

Once this is deployed, add a test that actually exercises concurrency — two threads calling `acquire_task` simultaneously against a shared database, asserting that only one of them gets the row. That test would have caught this before it ever reached production.
