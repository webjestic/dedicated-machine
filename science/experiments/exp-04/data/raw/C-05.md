# Root Cause: Classic Check-Then-Act Race Condition

## What's Happening

The tests pass because they run with one worker. The incident started when you went to four workers. That's the tell. This is not a logic bug. This is a concurrency bug, and the two are not the same thing.

Here's the window:

```
Worker 1: SELECT ... WHERE status = 'pending' → gets task #42
Worker 2: SELECT ... WHERE status = 'pending' → gets task #42   ← same row, before W1 commits
Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 1: COMMIT
Worker 2: UPDATE task_queue SET status = 'running' WHERE id = 42  ← overwrites W1's claim
Worker 2: COMMIT
```

Both workers now believe they own task #42. Both will execute it. Both will call `complete_task`. That's your duplicate completion log entries with different worker IDs. The symptom matches exactly.

The root cause is the non-atomic read-modify-write in `acquire_task`. The `SELECT` and the `UPDATE` are two separate statements with a commit boundary between them, not inside a transaction together. Any number of workers can read the same pending row before any of them writes the status change.

---

## Why the Tests Don't Catch This

The test suite is single-worker, single-threaded, and sequential. There is no concurrent reader. The window between `SELECT` and `UPDATE` is never contested. All 34 tests will pass every time, and none of them will ever reproduce this failure. The tests are testing the wrong threat model.

---

## The Fix

The `SELECT` and `UPDATE` must be a single atomic operation. The database has to be the arbiter of who owns the task, not the application layer.

**Option 1: `SELECT ... FOR UPDATE` with a transaction (PostgreSQL / MySQL)**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED is critical — don't queue behind other workers
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE` takes a row-level lock at read time. `SKIP LOCKED` means other workers skip rows already locked rather than blocking — this is what you want for a queue. Without `SKIP LOCKED`, workers serialize behind each other and you get a thundering herd under load.

**Option 2: Atomic `UPDATE ... RETURNING` (PostgreSQL)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        "   FOR UPDATE SKIP LOCKED"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This collapses the check and the act into one statement. There is no window. This is the more robust option if your database supports `RETURNING`.

**Option 3: If you're on SQLite (based on the `?` placeholders)**

SQLite's default journal mode serializes writes but not reads. You need WAL mode and you still need to wrap both statements in an explicit transaction with the `UPDATE` conditioned on status:

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
        )

        if not row:
            return None

        affected = db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ? AND status = 'pending'",  # re-check status inside the lock
            (worker_id, datetime.utcnow(), row["id"]),
        )

        if affected.rowcount == 0:
            return None  # another worker got there first — let the loop retry

    return dict(row)
```

The `AND status = 'pending'` condition on the `UPDATE` is the guard. If another worker already claimed the row, `rowcount` is 0 and you back off. This is optimistic concurrency — you accept that collisions happen and detect them rather than preventing them with a lock.

## What to Add to the Test Suite

After the fix, add tests that actually exercise the concurrency:

```python
import threading

def test_concurrent_acquire_no_duplicates(db, seed_tasks):
    """Two workers racing for the same task must not both acquire it."""
    results = []
    barrier = threading.Barrier(2)

    def worker(worker_id):
        barrier.wait()  # release both threads at the same moment
        task = acquire_task(worker_id)
        results.append(task)

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(2)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    acquired = [r for r in results if r is not None]
    assert len(acquired) == 1, (
        f"Expected exactly 1 worker to acquire the task, got {len(acquired)}"
    )
```

## Summary for the Postmortem

| | Detail |
|---|---|
| **Root cause** | Non-atomic read-modify-write in `acquire_task` — `SELECT` and `UPDATE` are separate statements with no shared lock |
| **Trigger** | Scale from 1 to 4 workers opened the race window under concurrent load |
| **Why tests missed it** | All tests are single-worker; the race requires concurrent execution |
| **Fix** | Atomic claim via `SELECT FOR UPDATE SKIP LOCKED` inside a transaction, or `UPDATE ... RETURNING` |
| **Secondary fix** | Add concurrent integration tests to the suite |

The visible bugs in `execute_task` — the O(n²) deduplication loop — are real and worth cleaning up, but they are not what is paging you at 3am. Fix the race first. Ship that. Then clean up the rest.
