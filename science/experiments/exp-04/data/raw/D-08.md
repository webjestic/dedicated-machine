## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are not atomic, so multiple workers can claim the same task.**

Here's the sequence that produces duplicates:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')   ← still pending, not yet updated
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running'          ← overwrites Worker 1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)          ← duplicate completion
```

Both workers read `pending`, both write `running`, both execute, both complete. This was invisible with a single worker process. Scaling to 4 workers under peak load made the window wide enough to hit consistently.

---

## Why the Tests Don't Catch It

Every test runs with a single worker against an in-process test database. There is no concurrent access, so the race window never opens. The tests are correct for what they test — they just don't test concurrent acquisition.

---

## The Fix

Replace the non-atomic SELECT + UPDATE pair with a single statement that claims the row only if it is still `pending`. The exact syntax depends on the database engine.

### PostgreSQL — use `UPDATE ... RETURNING`

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        """
        UPDATE task_queue
           SET status     = 'running',
               worker_id  = %s,
               claimed_at = NOW()
         WHERE id = (
               SELECT id
                 FROM task_queue
                WHERE status = 'pending'
                ORDER BY priority DESC, created_at ASC
                LIMIT 1
                FOR UPDATE SKIP LOCKED   -- critical: skip rows locked by other workers
               )
        RETURNING id, type, payload
        """,
        (worker_id,),
    )
    db.commit()
    return dict(row) if row else None
```

`FOR UPDATE SKIP LOCKED` is the key addition. It means each worker skips any row another worker is already in the process of claiming, rather than blocking and then racing on the same row.

### SQLite (if that is the engine in use)

SQLite has no `SKIP LOCKED`. The correct approach is to serialize acquisition with an application-level lock or switch to a database that supports it. A minimal stop-gap:

```python
import threading

_acquire_lock = threading.Lock()

def acquire_task(worker_id: int):
    with _acquire_lock:
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
        )
        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ? AND status = 'pending'",  -- guard clause
            (worker_id, datetime.utcnow(), row["id"]),
        )
        db.commit()

        # If another worker claimed it between our SELECT and UPDATE,
        # rowcount will be 0. Bail out and let the loop retry.
        if db.rowcount == 0:
            return None

        return dict(row)
```

Note the `AND status = 'pending'` guard in the `WHERE` clause. Even with a lock, this makes the UPDATE a no-op if the row was already claimed, and `rowcount == 0` lets the caller detect the miss cleanly.

> When we scaled from 1 to 4 workers, we exposed a race condition in how workers claim tasks. Two workers could read the same task at the same moment, both mark it as theirs, and both complete it. The fix is a one-function change that makes claiming a task atomic at the database level. No data was lost — tasks were processed more than once rather than dropped.
