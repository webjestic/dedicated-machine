## Root Cause

**Classic check-then-act race condition. The SELECT and UPDATE in `acquire_task` are not atomic.**

---

### What's Happening

```python
# Worker A and Worker B both execute this SELECT at nearly the same time
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    ...
)

# Both get back the same row — status is still 'pending' for both reads

# Worker A writes 'running'
# Worker B writes 'running' 50ms later, overwriting Worker A's claim
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ? ..."
    " WHERE id = ?",
    (worker_id, row["id"]),
)
```

The window is between the `SELECT` and the `UPDATE`. With one worker, that window never mattered. With four workers under load, two workers can read the same `pending` row before either one has committed the `UPDATE` that would change its status. Both proceed. Both execute. Both complete. The task ID appears twice in the completion log with different worker IDs. That's exactly what you're seeing.

The `db.commit()` after the UPDATE does not close this window. By the time Worker A commits, Worker B has already read `pending` and is already past the check.

---

### Why Tests Don't Catch It

Every test runs a single worker against a single-process in-memory database. There is no concurrency. The race window physically cannot open. All 34 tests pass. The test suite is testing the happy path of the logic, not the safety of the claim mechanism. This is the gap.

---

### What Needs to Change

**The SELECT and UPDATE must be collapsed into a single atomic operation.** The claim must be made in the same statement that finds the row, with the `pending` guard inside the `WHERE` clause of the `UPDATE`.

**Option 1 — Atomic UPDATE with `RETURNING` (PostgreSQL / SQLite 3.35+)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The `WHERE status = 'pending'` is now inside the `UPDATE`. The database engine serializes concurrent writes to the same row. If two workers race, one gets the row and one gets zero rows affected. The loser returns `None` and moves on.

**Option 2 — Optimistic lock with `WHERE status = 'pending'` on the UPDATE**

If `RETURNING` is not available:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard added here
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_updated == 0:
        return None  # another worker claimed it first

    return dict(row)
```

The `AND status = 'pending'` on the `UPDATE` means the second writer's update lands on a row that is already `'running'` and affects zero rows. `rowcount == 0` tells you you lost the race. This is still two round trips, but the UPDATE is now the authoritative claim gate. Only one writer can transition a row from `pending` to `running`.

Option 1 is cleaner and preferred if your database supports it. Option 2 works on older SQLite.
