## Root Cause

**The `acquire_task` function has a classic check-then-act race condition.** The SELECT and the UPDATE are two separate statements with no atomicity guarantee between them. Under concurrent load with four workers, multiple workers can read the same `pending` row before any of them writes `running`. All of them then claim it, execute it, and complete it.

This is the window:

```
Worker 1: SELECT → sees task 42, status='pending'
Worker 2: SELECT → sees task 42, status='pending'   ← both read before either writes
Worker 1: UPDATE task 42 SET status='running'
Worker 2: UPDATE task 42 SET status='running'       ← overwrites, no conflict raised
Worker 1: execute_task(42) → complete_task(42)
Worker 2: execute_task(42) → complete_task(42)      ← duplicate
```

The UPDATE has no guard on the current status. It will succeed even if another worker already claimed the row. Neither worker sees an error. Both proceed. The completion log gets two entries for task 42 with different worker IDs, which is exactly what you're seeing.

## Why Tests Didn't Catch It

The tests run a single worker against an in-process database. There is no concurrency. The race window does not exist in that environment. All 34 tests pass because the failure mode requires two workers to interleave at the exact right moment — something the test suite never exercises. This is not a gap in test quality for the logic being tested. It is a gap in what the test suite is capable of observing.

## The Fix

Close the window by making the check and the claim a single atomic operation. The UPDATE itself becomes the claim. Add a `WHERE status = 'pending'` guard on the UPDATE and check whether any row was actually modified.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task using an atomic update."""
    
    # First, identify the candidate task ID.
    # This SELECT is now only advisory — the UPDATE below is the real claim.
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # The WHERE clause on status = 'pending' is the lock.
    # If another worker claimed this row between our SELECT and this UPDATE,
    # the status is already 'running' and this UPDATE touches zero rows.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # Check whether we actually won the claim.
    # rowcount == 0 means another worker got there first.
    if db.rowcount == 0:
        return None

    # Re-fetch the full row now that we own it.
    claimed = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],)
    )
    return dict(claimed)
```

The critical line is:

```sql
WHERE id = ? AND status = 'pending'
```

Only one worker can transition a row from `pending` to `running`. The database serializes concurrent UPDATEs to the same row. Whichever worker's UPDATE lands second sees `status = 'running'` in the WHERE clause, matches zero rows, and gets `rowcount == 0`. That worker discards the result and moves on. The task executes exactly once.

## What Else to Do Right Now

**Immediate mitigation.** Scale back to one worker process. The race still exists but the probability drops to near zero with a single worker. This stops the bleeding while the fix is prepared.

**Before deploying the fix.** Audit the completion log for the past three days. Any task ID that appears more than once has been double-processed. You need to know the blast radius before you can tell downstream consumers what to do about it.

**After deploying the fix.** Add a unique constraint or an idempotency check on the downstream consumer side if the business logic allows it. The queue should be the primary guard, but defense in depth matters here. A task queue that guarantees at-least-once delivery with a broken consumer is still a problem.

**Add a concurrency test.** The existing test suite cannot see this class of bug. Add a test that spawns two threads, has them both call `acquire_task` simultaneously against a shared database, and asserts that only one of them gets a non-None result. That test would have caught this before it shipped.

---

## The Pattern to Remember

Whenever you see a SELECT followed by an UPDATE in a claim or reservation flow, ask one question: *what happens if two callers execute the SELECT before either executes the UPDATE?* If the answer is "they both proceed," the code is wrong. The fix is always the same — move the condition into the UPDATE's WHERE clause and treat `rowcount` as the authoritative answer to whether you won the race.
