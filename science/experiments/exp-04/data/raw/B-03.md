## Root Cause

**Classic check-then-act race condition. No atomic claim. Multiple workers are reading the same pending row before any of them have written the update.**

---

### The Sequence That Produces the Duplicate

Here is exactly what is happening under load with four workers:

```
Time    Worker 1                            Worker 2
----    --------                            --------
T1      SELECT ... WHERE status='pending'
        → returns task #4712
T2                                          SELECT ... WHERE status='pending'
                                            → returns task #4712  ← same row
T3      UPDATE ... SET status='running'
        WHERE id = 4712
T4                                          UPDATE ... SET status='running'
                                            WHERE id = 4712  ← succeeds, no guard
T5      execute_task(#4712)
T6                                          execute_task(#4712)
T7      complete_task(#4712, result_A)
T8                                          complete_task(#4712, result_B)
```

Both updates at T3 and T4 succeed. The `WHERE` clause only filters on `id`. There is nothing that makes the second update fail. Both workers proceed to execute and complete the same task. That is your duplicate in the completion log with two different worker IDs.

---

### Why the Tests Did Not Catch It

Every test runs a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in that environment. All 34 tests pass and always will, right up until you scale to multiple processes. This is not a test quality problem you can fix by adding more assertions to the existing test structure — you need a concurrent integration test or you need to eliminate the window entirely.

---

### Why This Appeared Now

One worker, no race. The SELECT and UPDATE are effectively sequential. The moment you went to four workers under peak load, the queue depth created enough contention that multiple workers are hitting the SELECT at the same millisecond before any UPDATE has committed.

---

### What Needs to Change

**The fix is to make the claim atomic. The SELECT and UPDATE must be a single operation that only one worker can win.**

There are two standard approaches depending on what your database supports.

**Option 1 — `UPDATE ... RETURNING` (PostgreSQL, SQLite 3.35+)**

Replace the entire SELECT + UPDATE sequence with one statement:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

The subquery runs inside the UPDATE. The database engine serializes concurrent writes to the same row. Only one worker gets a returned row. The other gets nothing and loops. There is no window.

**Option 2 — Optimistic lock with `status` guard in the WHERE clause**

If `RETURNING` is not available:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard added
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        return None  # another worker claimed it first

    return dict(row)
```

The `AND status = 'pending'` turns the UPDATE into a compare-and-swap. The second worker to arrive finds `status = 'running'` and gets `rowcount = 0`. It returns `None` and moves on. This is weaker than Option 1 because there is still a window between the SELECT and UPDATE, but the guard ensures only one writer wins. Option 1 is cleaner — use it if your database version supports it.

---

### What to Tell the Junior Engineers

Do not touch `execute_task` or `complete_task`. The bug is entirely in `acquire_task`. The fix is one of the two above. Verify your SQLite or Postgres version before choosing. After the patch ships, add a concurrent integration test that spins up two workers against a shared database and asserts that each task ID appears exactly once in the completion log. That test will fail against the current code and pass against the fix. That is the regression test you want.
