## Root Cause

**The `acquire_task` function has a check-then-act race condition. The SELECT and the UPDATE are not atomic. Multiple workers can read the same pending row before any of them writes the claim.**

---

### The Failure Sequence

Here is exactly what is happening under load with four workers:

```
Time →

Worker 1: SELECT ... WHERE status = 'pending' → returns task_id=42
Worker 2: SELECT ... WHERE status = 'pending' → returns task_id=42   ← same row, status still 'pending'
Worker 1:                                         UPDATE task_id=42 SET status='running', worker_id=1
Worker 2:                                         UPDATE task_id=42 SET status='running', worker_id=2
Worker 1: execute_task(42) ...
Worker 2: execute_task(42) ...                    ← both are now running the same task
Worker 1: complete_task(42) → logged, downstream notified
Worker 2: complete_task(42) → logged again, downstream notified again
```

The SELECT sees `status = 'pending'`. That is the check. The UPDATE is the act. There is a window between them. With one worker, that window never mattered. With four workers under peak load, multiple workers are reading the queue simultaneously and all of them see the same unowned row before any of them has written the claim.

The UPDATE does not fail. It does not error. It just silently overwrites whatever the previous worker wrote. Both workers proceed. Both complete. Both emit to downstream. That is your duplicate.

---

### Why the Tests Did Not Catch It

Every test runs a single worker against an in-process database with no concurrency. The race window cannot open when there is only one thread and no scheduling between the SELECT and the UPDATE. The tests are testing the happy path of the logic, not the invariant that only one worker can claim a task. Those are different things. All 34 tests passing is not evidence the code is correct under concurrency. It is evidence the tests were not written for concurrency.

---

### Why the Visible Bugs Are Not the Problem

There is an O(n²) deduplication loop in `execute_task` and the `complete_task` UPDATE has no `WHERE status = 'running'` guard. Fix those if you want. They are not why downstream is processing payloads twice. Do not let the team ship a patch to `execute_task` and call the incident closed.

---

### What Needs to Change

The SELECT and the UPDATE must be collapsed into a single atomic statement. The database must do the claim, not the application.

**The fix — replace `acquire_task` with an atomic claim:**

```python
def acquire_task(worker_id: int):
    """
    Claim the next pending task atomically.
    
    The SELECT and UPDATE must be one operation from the database's
    perspective. Any approach that reads first and writes second in
    separate statements has a race window between them.
    """
    # Single statement: claim it or claim nothing.
    # The WHERE status = 'pending' inside the UPDATE is the guard.
    # Only one writer can win the row lock. The loser updates zero rows.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",   # ← the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker claimed it between our SELECT and this UPDATE.
        # Bail out. We will pick up the next task on the next loop tick.
        return None

    return dict(row)
```

Adding `AND status = 'pending'` to the UPDATE WHERE clause means the second writer's UPDATE matches zero rows. `rows_affected == 0` is the signal that we lost the race. We return `None` and move on. No duplicate execution.

**If your database supports it, use `UPDATE ... RETURNING` to eliminate the SELECT entirely:**

```python
def acquire_task(worker_id: int):
    """
    Fully atomic claim using UPDATE ... RETURNING.
    No SELECT. No window. One round trip.
    Supported by PostgreSQL and SQLite >= 3.35.
    """
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This is the correct solution. There is no SELECT followed by an UPDATE. There is one statement. The database serializes concurrent writers at the row lock. One worker gets the row. Every other concurrent attempt touches zero rows and returns nothing.

---

### Secondary Hardening

Once the claim is atomic, add a guard to `complete_task` so a late-arriving duplicate write cannot overwrite a legitimately completed task:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # ← do not stomp a completed task
        (json.dumps(result), datetime.utcnow(), task_id),
    )
    db.commit()
```

This is defense in depth, not the fix. The fix is the atomic claim.

---

### What to Tell the Junior Engineers

> The SELECT and UPDATE in `acquire_task` are two separate database round trips. Between them, any other worker can read the same row. The fix is to add `AND status = 'pending'` to the UPDATE and check `rows_affected`. If it is zero, we lost the race and we skip the task. If your database version supports `UPDATE ... RETURNING`, use that instead — it removes the SELECT entirely and there is no window at all. Do not touch `execute_task`. That is not the incident.
