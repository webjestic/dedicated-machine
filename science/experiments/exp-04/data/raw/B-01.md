## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and the UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

---

### What Is Actually Happening

Worker A and Worker B both execute the SELECT at nearly the same moment. Both see the same pending task. Both return the same `row["id"]`. Both then execute the UPDATE, claiming the task under their own `worker_id`. Both commit. Both call `execute_task` and `complete_task`. The task appears twice in the completion log with different worker IDs.

The UPDATE does not protect you here. It runs, it succeeds, it commits — but it does not know it is the second writer. Nothing in that UPDATE says "only do this if the row is still pending." It will overwrite whatever the first worker wrote. Both workers proceed.

This is the window:

```
Worker A: SELECT → sees task 42, status='pending'
                                                Worker B: SELECT → sees task 42, status='pending'
Worker A: UPDATE task 42 SET status='running', worker_id=1
Worker A: COMMIT
                                                Worker B: UPDATE task 42 SET status='running', worker_id=2
                                                Worker B: COMMIT
Worker A: execute_task(42) ...
                                                Worker B: execute_task(42) ...
Worker A: complete_task(42) ...
                                                Worker B: complete_task(42) ...
```

Both complete. Both log it. Downstream sees it twice.

---

### Why the Tests Did Not Catch It

Every test runs a single worker against an in-process database with no concurrency. The race window does not exist in that environment. All 34 tests pass because they are testing the single-worker case, which has never been broken. The tests are not wrong — they are just not testing the thing that is failing.

---

### Why It Appeared Now

One worker, no race. The SELECT and UPDATE are effectively serialized. You ran fine for months. You scaled to four workers three days ago. Under low load, the timing gap is rarely hit. Under peak load this morning, workers are hammering the queue simultaneously and the window is being hit repeatedly.

---

### The Fix

Close the window. The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter of who owns the task, not the application.

**Option 1 — Atomic UPDATE with a conditional WHERE clause (preferred for most databases):**

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    # First, find the candidate task ID
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt to claim it — only succeeds if it is STILL pending
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- this is the lock
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, affected rows will be 0
    if affected.rowcount == 0:
        return None

    return dict(row)
```

The `AND status = 'pending'` predicate is the critical change. The second writer's UPDATE matches zero rows. Its `rowcount` is 0. It returns `None`. Only one worker proceeds.

**Option 2 — `SELECT FOR UPDATE` with a transaction (PostgreSQL, MySQL):**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # locks the row, skips rows locked by other workers
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`SKIP LOCKED` is important here. Without it, Worker B blocks waiting for Worker A's lock to release, then acquires the same row after it is already claimed. With `SKIP LOCKED`, Worker B skips that row entirely and moves to the next pending task.

**Which option to use** depends on your database. If this is SQLite, use Option 1 — SQLite does not support `SELECT FOR UPDATE`. If this is Postgres or MySQL, Option 2 with `SKIP LOCKED` is cleaner and scales better under high contention.

---

### What to Tell the Junior Engineers

The fix is in `acquire_task` only. Do not touch `execute_task` or `complete_task`. Add `AND status = 'pending'` to the WHERE clause of the UPDATE and check `rowcount`. That is the entire change. The tests will still pass. Write a new test that runs two workers concurrently against the same pending task and asserts that only one of them gets a non-None return — that is the test that should have existed before the scale-out.

The inefficient deduplication loop in `execute_task` is real and worth fixing, but it is not causing this incident. Do not touch it right now. Fix the race, deploy, confirm the duplicate completions stop, then address the loop in a separate change.
