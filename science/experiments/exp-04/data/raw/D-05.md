## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are not atomic, so multiple workers can claim the same task.**

Here's the exact sequence that produces duplicates:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')   ← still pending, not yet updated
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running'          ← overwrites Worker 1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)          ← duplicate completion
```

Both workers read `pending`, both proceed, both complete. This was invisible at 1 worker. At 4 workers under peak load, it surfaces constantly.

---

## Why Tests Didn't Catch It

Every test runs a single worker against a single-process SQLite instance. There is no concurrency. The race condition requires two workers executing `acquire_task` simultaneously — a condition the test suite never creates. All 34 tests pass and always will in that setup.

---

## The Fix

The SELECT and UPDATE must be a single atomic operation. The standard pattern is **`UPDATE ... WHERE status = 'pending' RETURNING`**, or a `SELECT FOR UPDATE` inside a transaction. The goal is to make "find an unclaimed task and claim it" one indivisible step at the database level.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # Single atomic statement: only one worker can win the UPDATE for a given row.
    # The WHERE clause on status = 'pending' is the lock — if two workers race,
    # only one UPDATE will match and the other gets zero rows back.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )

    if not row:
        return None

    db.commit()
    return dict(row)
```

**Why this works:** The `WHERE id = (subquery)` with `status = 'pending'` inside the `UPDATE` means the status check and the claim happen in one statement. The database serializes concurrent writes to the same row. If two workers target the same `id`, only one `UPDATE` will find `status = 'pending'` — the other will match zero rows and get nothing back. No task can be claimed twice.

## What Needs to Happen Right Now

**Immediate mitigation** — roll back to 1 worker process. This eliminates the race condition without any code change and stops the duplicate completions immediately.

**Before re-scaling** — deploy the atomic `acquire_task` above. Also add a deduplication guard in `complete_task` so a second completion attempt on an already-complete task fails loudly rather than silently overwriting:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # guard: only complete what we own
        (json.dumps(result), datetime.utcnow(), task_id),
    )
    db.commit()
```

**After the fix** — add a concurrency test that spins up multiple threads calling `acquire_task` simultaneously and asserts each task ID appears in exactly one result. That test would have caught this before it reached production.
