## Root Cause

**Classic check-then-act race condition in `acquire_task`.** The SELECT and the UPDATE are two separate operations with no atomicity guarantee between them. Under a single worker, this never fires. Under four concurrent workers, it fires exactly when you'd expect: peak load, high queue depth, workers all polling at roughly the same interval.

Here is the sequence that produces the duplicate you're seeing in the completion log:

```
Time →

Worker 2:  SELECT → gets row 4471 (status='pending') ─────────────────────────────── UPDATE row 4471 → 'running' → execute → complete
Worker 3:  ────────────────── SELECT → gets row 4471 (status='pending') ─── UPDATE row 4471 → 'running' → execute → complete
                              ↑
                              This SELECT runs AFTER Worker 2's SELECT
                              but BEFORE Worker 2's UPDATE commits.
                              Row is still 'pending'. Both workers own it.
```

Both workers read `status = 'pending'`, both proceed to claim it, both execute it, both write a completion record. The last UPDATE wins on the row itself, but by then the downstream consumer has already received two payloads.

---

## Why the Tests Don't Catch It

The tests are sequential. One worker, one thread, one operation at a time. The window between SELECT and UPDATE is never contested. All 34 tests pass cleanly because the condition that causes the failure — **concurrent access** — is structurally absent from the test suite. This is not a test quality problem you can fix by adding assertions. You need a concurrent test or an integration test with multiple workers.

---

## What Needs to Change

The fix is to make the claim operation atomic. The SELECT and UPDATE must happen as a single indivisible unit from the database's perspective. There are two standard approaches:

### Option 1: `UPDATE ... RETURNING` (preferred, if your DB supports it)

Collapse the SELECT and UPDATE into one statement. There is no window because there is no separate read.

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The database evaluates the subquery and applies the UPDATE atomically under row-level locking. A second worker hitting this at the same instant will either get a different row or get nothing. It cannot get the same row in `pending` state.

**Supported by:** PostgreSQL, SQLite 3.35+, MariaDB 10.5+. Check your version before deploying.

---

### Option 2: Optimistic locking with a `WHERE` guard on status

If `RETURNING` is not available, add the status check to the UPDATE's WHERE clause and verify that exactly one row was affected.

```python
def acquire_task(worker_id: int):
    """Claim a task only if it is still pending at update time."""
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- the guard
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_affected == 0:
        # Another worker claimed this task between our SELECT and UPDATE.
        # Yield and let the next poll cycle find a free task.
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means the UPDATE is a no-op if another worker already claimed the row. `rowcount == 0` is the signal that you lost the race. You return `None` and the worker sleeps its 50ms and tries again. No duplicate execution.

This still has the SELECT/UPDATE gap, but the UPDATE itself acts as the authoritative claim. Only one UPDATE can transition a row from `pending` to `running`. The SELECT is now just a hint, not a commitment.

## Which Option to Ship

| | `UPDATE...RETURNING` | Optimistic lock |
|---|---|---|
| Atomicity | True — single statement | Near — two statements, guarded |
| DB support | Version-dependent | Universal |
| Complexity | Lower | Slightly higher |
| Risk of duplicate | None | None (rowcount check is the gate) |

If you are on PostgreSQL or SQLite 3.35+, ship Option 1. It is simpler and leaves no ambiguity. If you are on an older engine or cannot confirm the version right now during the incident, ship Option 2. It is safe on everything.

## What to Do Right Now

1. **Scale back to 1 worker** until the fix is deployed. The race only manifests under concurrency. One worker eliminates the symptom immediately.
2. **Audit the completion log** for all task IDs that appear more than once. Identify which downstream consumers received duplicate payloads and whether those operations are idempotent. If they are not, you have data to reconcile.
3. **Deploy the fix** against a staging environment with a concurrent load test — multiple workers hammering the same queue — before promoting to production.
4. **Add a unique constraint** on `(task_id, worker_id)` in the completion log table if one does not already exist. It will not prevent the race, but it will make the symptom loud and immediate rather than silent.
5. **Add a concurrent acquisition test** to the suite. Spin up two threads, have both call `acquire_task` simultaneously against the same seeded row, assert that exactly one returns a task and one returns `None`.

The tests passing is not evidence the code is correct. It is evidence the tests do not model the failure mode.
