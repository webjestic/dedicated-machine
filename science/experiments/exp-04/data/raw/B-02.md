## Root Cause

**Classic check-then-act race condition. The SELECT and the UPDATE in `acquire_task` are not atomic. Multiple workers can read the same pending row before any of them writes `status = 'running'`.**

---

### The Exact Failure Sequence

Here is what happens under load with 4 workers:

```
Time →

Worker 1: SELECT ... WHERE status = 'pending' → returns task #42
Worker 2: SELECT ... WHERE status = 'pending' → returns task #42   ← same row, status still 'pending'
Worker 1:                                         UPDATE task #42 SET status = 'running', worker_id = 1
Worker 2:                                         UPDATE task #42 SET status = 'running', worker_id = 2
```

Both workers got the row. Both workers committed their claim. Both workers execute and complete the task. The completion log shows task #42 twice with different worker IDs because that is exactly what happened.

This ran fine for months with one worker because there was no concurrent reader. The race window did not exist. You scaled to 4 workers and the window opened.

---

### Why the Tests Don't Catch It

Every test runs a single worker against a single-process in-memory database. There is no concurrency. The race window is never open. All 34 tests pass. The CI environment is structurally incapable of reproducing this failure.

---

### Why the Visible Bug Is Not the Bug

There is an inefficient deduplication loop in `execute_task` — the `any()` scan inside a nested loop. That is a performance problem. It is not this incident. Do not touch it right now. The duplicate completions exist before `execute_task` is ever called.

---

### The Fix

The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter of which worker wins. There are two standard approaches:

**Option 1 — `UPDATE ... WHERE status = 'pending'` with a conditional, then read back (works on SQLite, MySQL, Postgres)**

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()

    # Attempt to claim a task atomically.
    # The subquery selects the candidate; the WHERE clause on the outer UPDATE
    # ensures only one writer wins when multiple workers race on the same row.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",   # ← re-check inside the UPDATE
        (worker_id, claimed_at),
    )
    db.commit()

    if db.rows_affected() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
    return dict(row) if row else None
```

The key is `AND status = 'pending'` inside the UPDATE predicate. If two workers race, the database serializes the two writes. The second one finds `status = 'running'` and updates zero rows. `rows_affected() == 0` means this worker lost the race and moves on.

**Option 2 — `SELECT FOR UPDATE` (Postgres, MySQL with InnoDB)**

```python
def acquire_task(worker_id: int):
    # Row-level lock held until commit. Second worker blocks, not races.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"   # SKIP LOCKED avoids blocking; worker moves to next row
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`SKIP LOCKED` is important here. Without it, Worker 2 blocks on the locked row and claims it the moment Worker 1 commits. With it, Worker 2 skips that row and looks for the next available one, which is the correct behavior for a queue.

---

### Which Option to Ship

Check your database engine right now.

- **SQLite**: Use Option 1. SQLite does not support `SELECT FOR UPDATE`.
- **Postgres**: Use Option 2 with `SKIP LOCKED`. It is purpose-built for this pattern and the intent is explicit.
- **MySQL/InnoDB**: Either works. Option 2 with `SKIP LOCKED` is cleaner.

---

### What to Tell the Junior Engineers

> The SELECT and the UPDATE in `acquire_task` are two separate database operations with no lock between them. Under concurrent load, multiple workers read the same pending row before any of them marks it running. The fix makes the claim atomic at the database level so only one writer can win. Pick the option that matches the database engine, implement it in `acquire_task`, and add a concurrency test that spins up multiple threads racing on the same pending task and asserts the task is completed exactly once.

That last part matters. The test suite needs to be able to catch this class of bug before the next scale event.
