# Root Cause: Unprotected Check-Then-Act Race in `acquire_task`

## The Short Version (for the client call)

There is no atomicity guarantee between the SELECT and the UPDATE in `acquire_task`. Under 4 concurrent workers, two workers can read the same pending task row before either has written `status = 'running'`. Both claim it. Both complete it. That is your duplicate.

---

## The Failure Sequence

```
Time →

Worker 1:  SELECT (sees task 42, status='pending') ─────────────────► UPDATE task 42 → 'running' ► execute ► complete
                                                    ↑
Worker 2:                SELECT (sees task 42, status='pending') ────────────────────────────────► UPDATE task 42 → 'running' ► execute ► complete
```

The window is the gap between the SELECT returning and the UPDATE committing. Under a single worker this window is never contested. Under 4 workers at peak load it is hit constantly. The tests never caught it because every test runs a single worker against an isolated database — the race requires concurrency to materialize.

---

## Why the Tests Are Green and Useless Here

Every test calls `acquire_task` once, sequentially, against a test database with no other writers. The test suite has 34 passing tests that prove the function works correctly in isolation. None of them prove it is safe under concurrent access. This is not a test quality failure to fix right now — it is the reason you cannot trust green CI as a signal during this incident.

---

## The Exact Lines That Are Broken

```python
# acquire_task — this is two operations, not one
row = db.query_one(          # ← Worker A and Worker B both land here
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

if not row:
    return None

db.execute(                  # ← Both then land here with the same row["id"]
    "UPDATE task_queue"
    " SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
```

The UPDATE has no guard. It will succeed for both workers because neither checks whether `status` is still `'pending'` at write time. The row gets claimed twice.

---

## What Needs to Change

There are two viable fixes. Which one you reach for depends on your database.

### Fix 1 — Conditional UPDATE with re-check (works on any SQL database, fastest to ship)

Flip the logic. Do not SELECT first. UPDATE with the condition inline and check whether you actually claimed anything.

```python
def acquire_task(worker_id: int):
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only this worker's UPDATE will have set worker_id to worker_id
    # in the same transaction — fetch the row we just claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'running' AND worker_id = ?"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The UPDATE is a single atomic statement. The database serializes concurrent writes to the same row. Only one writer wins. The other UPDATE matches zero rows and returns nothing to claim.

### Fix 2 — `SELECT ... FOR UPDATE` with `SKIP LOCKED` (Postgres; cleaner, preferred long-term)

```python
def acquire_task(worker_id: int):
    # Run inside an explicit transaction
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )

    if not row:
        db.rollback()
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`FOR UPDATE` locks the row at read time. `SKIP LOCKED` means a concurrent worker skips that row entirely rather than blocking on it — which is exactly the queue-drain behavior you want. This requires Postgres and explicit transaction management.

## What to Do Right Now

**Immediate mitigation (next 5 minutes):** Scale `task-runner` back to 1 worker. This eliminates the concurrency and stops the bleeding. Downstream consumers need to deduplicate on `task_id` until the fix is deployed — confirm whether they have that guard.

**Fix to ship:** Fix 1 if you need to move fast and are not certain about the database driver's transaction handling. Fix 2 if you are on Postgres and can verify the transaction scope in the next deploy.

**Do not touch `execute_task`:** The deduplication logic in there is inefficient — O(n²) list scan instead of a set — but it is not causing this incident. Fix it in a follow-up. Do not let anyone conflate the two problems right now.

---

## One Line Summary for the Postmortem

> The SELECT and UPDATE in `acquire_task` are not atomic. Scaling to 4 workers created a concurrent window where multiple workers claim the same task. The fix is to make the claim a single atomic database operation.
