## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, uncoordinated operations with no row-level lock between them.**

Here's the exact failure sequence:

```
Worker 1:  SELECT → finds task #42 (status='pending')
Worker 2:  SELECT → finds task #42 (status='pending')   ← both read before either writes
Worker 1:  UPDATE task #42 SET status='running'
Worker 2:  UPDATE task #42 SET status='running'          ← overwrites Worker 1, no conflict
Worker 1:  execute_task(#42) → complete_task(#42)
Worker 2:  execute_task(#42) → complete_task(#42)        ← duplicate completion
```

Both workers see `status = 'pending'` before either has committed their UPDATE. The UPDATE has no guard — it will succeed for both workers because neither checks that the row is *still* pending at write time. This is a classic **check-then-act** race.

---

## Why Tests Didn't Catch It

Every test runs with a single worker against a single-process test database. The race only opens up under concurrent access. 34 tests passing is entirely consistent with this bug — it was invisible until the scale-out to 4 workers this morning.

## The Fix

The UPDATE itself needs to be the claim operation, and it needs to be conditional. There are two solid approaches:

### Option 1 — Atomic UPDATE + return (preferred for most databases)

Use a single statement that claims the row only if it is still pending. The SELECT becomes a consequence of the UPDATE, not a precondition.

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""

    # Single atomic operation: claim the row only if it is still 'pending'.
    # No other worker can claim the same row because the WHERE clause
    # acts as the guard — only one UPDATE will match and modify the row.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Now read back what we just claimed.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

> **Note:** The exact syntax for an UPDATE with a subquery varies by database. SQLite supports this form. PostgreSQL users should use `UPDATE ... WHERE id = (subquery) RETURNING *` to collapse this into one round-trip and eliminate the read-back race on `worker_id`.

### Option 2 — Optimistic lock with a WHERE guard on the UPDATE

If you need to keep the SELECT separate (e.g., for priority logic that is hard to express in a subquery), guard the UPDATE so it only fires if the row is still pending. Then check whether anything was actually updated.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard closes the window
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed this row first, rowcount will be 0.
    if db.rowcount == 0:
        return None  # Lost the race — caller will retry on next loop tick

    return dict(row)
```

This is simpler to implement but the race window still technically exists — it is just made safe by the `AND status = 'pending'` guard and the rowcount check. Option 1 is cleaner.
