# Root Cause: Check-Then-Act Race Condition in `acquire_task`

## What's Happening

The duplicate completions are not a mystery. Look at the sequence:

```
Worker 1: SELECT ... WHERE status = 'pending' → gets task #42
Worker 2: SELECT ... WHERE status = 'pending' → gets task #42   ← same row, same moment
Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 2: UPDATE task_queue SET status = 'running' WHERE id = 42
```

Both workers read the same pending row before either one has written the claim. Both proceed to execute and complete it. That's your duplicate completion log. That's why you see two different worker IDs on the same task ID.

The `SELECT` and the `UPDATE` are two separate statements with no atomicity guarantee between them. That gap — the window between the check and the act — is where four concurrent workers are colliding under peak load.

---

## Why Tests Pass and Production Burns

The tests run one worker against one database connection with no concurrency. The race condition is physically impossible to trigger in that environment. Every test is correct. The test suite is testing the wrong thing. This is the exact failure mode that only appears after you scale.

## The Fix

### Option 1: Atomic claim with `UPDATE ... WHERE status = 'pending'` and row count check

This is the right fix for most relational databases. Make the claim the only operation that matters — skip the `SELECT` entirely.

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    # First, find the candidate ID
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt the claim — only succeeds if status is still 'pending'
    # The WHERE clause is the lock. If another worker claimed it first,
    # affected_rows will be 0 and we walk away.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← this is the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Another worker claimed it between our SELECT and this UPDATE.
        # Return None and let the loop retry.
        return None

    return dict(row)
```

The `AND status = 'pending'` condition in the `UPDATE` is the atomic gate. The database engine serializes concurrent writes to the same row. Exactly one worker's `UPDATE` will match. Everyone else gets `rowcount == 0` and backs off. The check and the act are now one operation.

### Option 2: `SELECT FOR UPDATE` (if your database supports it)

If you're on PostgreSQL or MySQL with InnoDB, you can lock the row at read time:

```python
def acquire_task(worker_id: int):
    # Row is locked for the duration of this transaction.
    # Other workers block on the SELECT until we commit.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED avoids blocking, picks next available
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`SKIP LOCKED` is important here — without it, workers queue up behind the lock and you get a thundering herd. With it, each worker skips locked rows and grabs the next available one.

### Which to use

| | `UPDATE WHERE status = 'pending'` | `SELECT FOR UPDATE SKIP LOCKED` |
|---|---|---|
| Works on SQLite | ✅ | ❌ |
| Works on PostgreSQL | ✅ | ✅ (preferred) |
| Works on MySQL/InnoDB | ✅ | ✅ |
| Requires transaction awareness | Minimal | Yes |
| Throughput under contention | Good | Better |

If you're on SQLite, Option 1 is your only path. If you're on Postgres, Option 2 with `SKIP LOCKED` is the standard pattern for exactly this use case and will perform better at scale.

## What to Add to the Test Suite

The current tests cannot catch this. Add a concurrency test:

```python
import threading

def test_acquire_task_no_duplicate_claims(db, seed_tasks):
    """Two concurrent workers must not claim the same task."""
    claimed_ids = []
    lock = threading.Lock()

    def worker(worker_id):
        task = acquire_task(worker_id=worker_id)
        if task:
            with lock:
                claimed_ids.append(task["id"])

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(4)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # No task ID should appear more than once
    assert len(claimed_ids) == len(set(claimed_ids)), (
        f"Duplicate claims detected: {claimed_ids}"
    )
```

This test would have caught the bug before the scale-up shipped. Add it now so the next engineer doesn't find this the same way you did.

## Summary for the Postmortem

The root cause is a non-atomic check-then-act in `acquire_task`. The `SELECT` and `UPDATE` are two separate database round-trips with no isolation between them. Under single-worker deployment this window never mattered. Under four concurrent workers at peak load, multiple workers read the same pending row before any of them write the claim. The fix is to make the claim itself the only authoritative operation, using either `UPDATE WHERE status = 'pending'` with a `rowcount` check, or `SELECT FOR UPDATE SKIP LOCKED` inside a transaction. The test suite passes because it tests correctness under sequential execution, not safety under concurrency. A concurrent acquisition test needs to be added.
