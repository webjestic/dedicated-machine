## Root Cause

**Classic check-then-act race condition. No atomic claim on task acquisition.**

Here is the exact failure sequence:

```
Worker 1                              Worker 2
--------                              --------
SELECT ... WHERE status = 'pending'
                                      SELECT ... WHERE status = 'pending'
-- both read the same row --
-- both see status = 'pending' --

UPDATE ... SET status = 'running'
  WHERE id = 42
                                      UPDATE ... SET status = 'running'
                                        WHERE id = 42

-- both updates succeed --
-- both workers now own task 42 --
```

The `SELECT` and `UPDATE` in `acquire_task` are two separate statements with no protection between them. Under a single worker, this never fires. Under four workers at peak load, the window between the read and the write is wide enough to drive a truck through. Both workers see `status = 'pending'`, both update it to `'running'`, both execute, both complete. The task ID appears twice in the completion log with different worker IDs. That is exactly what you are seeing.

The tests pass because every test runs one worker against an isolated database. The race requires two concurrent readers. The test suite never produces that condition.

---

## What Needs to Change

The fix is to make the claim atomic. The `SELECT` and the `UPDATE` must be a single operation where the condition check and the state change cannot be separated by another writer.

**Option 1 — Atomic UPDATE with a WHERE guard (preferred, works on most databases):**

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    # Find the candidate first so we know which ID to target.
    # This read is still a race on its own, so the UPDATE below
    # must re-check status. Only one writer can win.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- re-check status atomically
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it between our SELECT and this UPDATE,
    # rows_affected will be 0. Bail out and let the loop retry.
    if rows_affected == 0:
        return None

    return dict(row)
```

The `AND status = 'pending'` in the `WHERE` clause is the critical change. The database engine evaluates the condition and applies the write as a single atomic operation. Exactly one worker will see `rows_affected = 1`. Every other worker that raced to the same row will see `rows_affected = 0` and return `None`. The loser retries on the next cycle and picks up a different task.

**Option 2 — `SELECT FOR UPDATE` with a transaction (if your database supports it):**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # lock the row, skip rows already locked
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is purpose-built for job queues. A worker locks the row for the duration of the transaction. Any other worker that hits the same row skips it and moves to the next candidate instead of blocking. This is the cleaner solution if you are on PostgreSQL. SQLite does not support it.

## What to Ship Right Now

**Option 1 is the immediate fix.** It requires no schema change, no new database features, and works regardless of which database is underneath this. The change is three tokens: `AND status = 'pending'` appended to the existing `WHERE` clause, plus checking `rows_affected`.

**Option 2 is the right long-term design** if you are on PostgreSQL. Put it in the follow-up ticket.

## What to Add to the Tests

The test suite needs a concurrent acquisition test. Without it, this class of bug is invisible to CI.

```python
def test_acquire_task_only_one_worker_claims_under_contention(db, seed_tasks):
    """Two workers racing for the same task — exactly one must win."""
    from concurrent.futures import ThreadPoolExecutor

    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(acquire_task, wid) for wid in (1, 2)]
        results = [f.result() for f in futures]

    claimed = [r for r in results if r is not None]
    assert len(claimed) == 1, (
        f"Expected exactly 1 worker to claim the task, got {len(claimed)}"
    )
```

This test would have caught the bug before the scale-out. Add it before closing the incident.

## Summary for the Client Call

> We scaled from 1 to 4 workers three days ago. The task acquisition logic reads a pending task and then updates it in two separate database operations. Under a single worker that is safe. Under multiple workers, two workers can read the same task before either has updated it, and both proceed to execute and complete it. The fix is a one-line change to the update condition that makes the claim atomic at the database level. We have the fix staged and are deploying now.
