## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, uncoordinated operations with no row-level lock between them.**

Here's the exact sequence that produces duplicates under 4 workers:

```
Worker 1: SELECT → sees task #42 as 'pending'
Worker 2: SELECT → sees task #42 as 'pending'   ← gap right here
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running' ← both claimed it
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42) ← duplicate completion
```

Both workers read the same row before either one writes. The UPDATE has no guard — it will succeed for both workers because by the time Worker 2's UPDATE runs, the row exists and the WHERE clause `WHERE id = ?` still matches regardless of current status.

---

## Why Tests Didn't Catch It

The tests run with a **single worker against a single-threaded test database**. There is no concurrency, so the race window never opens. All 34 tests pass cleanly and always will under those conditions. This is a classic case of tests validating correct single-process behavior while the actual failure mode only exists at the concurrency boundary.

---

## The Fix

The SELECT and UPDATE need to be **atomic**. The standard pattern is to add a `WHERE status = 'pending'` guard on the UPDATE itself and check whether any row was actually modified. Only the worker that wins the UPDATE proceeds.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task. Safe for concurrent workers."""
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard added
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount

    db.commit()

    if rows_updated == 0:
        return None  # another worker claimed it first — back off and retry

    return dict(row)
```

The key change is `AND status = 'pending'` on the UPDATE. Only one worker's UPDATE will match — the first one to execute it flips the status to `'running'`, and every subsequent worker's UPDATE matches zero rows and returns without claiming the task.

---

## Notes for the Fix

**Verify `db.execute()` exposes `rowcount`** (or equivalent) in your wrapper. If it doesn't, that needs to be added — it's the only way to know whether this worker won the race.

**This pattern (optimistic locking via a conditional UPDATE) is sufficient for most databases** including PostgreSQL and SQLite. If you're on PostgreSQL and want a stronger guarantee, `SELECT ... FOR UPDATE SKIP LOCKED` is the idiomatic alternative and eliminates the SELECT/UPDATE gap entirely:

```sql
-- PostgreSQL only
SELECT id, type, payload FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED
```

**The `complete_task` function has the same class of bug** — it updates `WHERE id = ?` with no status check. A task could theoretically be re-completed. Add `AND status = 'running'` there as well once the immediate fire is out.
