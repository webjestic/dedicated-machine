## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and UPDATE are two separate operations with no atomicity guarantee between them.**

Here is the exact failure sequence that produced your completion log:

```
Worker 1: SELECT → finds task #42 (status = 'pending')
Worker 2: SELECT → finds task #42 (status = 'pending')   ← both read before either writes
Worker 1: UPDATE task #42 SET status = 'running'
Worker 2: UPDATE task #42 SET status = 'running'          ← overwrites worker 1's claim
Worker 1: execute_task(#42) → complete_task(#42)
Worker 2: execute_task(#42) → complete_task(#42)          ← duplicate
```

Both workers see `pending`, both claim the row, both execute it. The UPDATE has no guard — it does not check that the row is still `pending` before writing. It will happily overwrite whatever status is already there.

---

## Why the Tests Did Not Catch This

Every test runs with a single worker against a single-process in-memory database. There is no concurrency. The race window does not exist in that environment. All 34 tests pass and will continue to pass. They are testing the wrong threat model.

The incident started three days after the scale-out, not immediately, because at low load the 50ms sleep was enough spacing that workers rarely collided. Peak load this morning compressed that window.

---

## What Needs to Change

The fix is to make the claim atomic. The SELECT and UPDATE must be a single operation that only succeeds if the row is still in the state you expect.

### Option 1: `UPDATE ... WHERE status = 'pending'` with row count check (simplest, works on most databases)

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # This is still two statements, so wrap in a transaction
    # and use SELECT FOR UPDATE if your DB supports it.
    # For SQLite, serialize with a transaction lock instead.
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
        )

        if not row:
            return None

        rows_affected = db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ? AND status = 'pending'",  # ← guard condition
            (worker_id, datetime.utcnow(), row["id"]),
        ).rowcount

        if rows_affected == 0:
            # Another worker claimed it between our SELECT and UPDATE.
            # Abort and let the next loop iteration try again.
            return None

    return dict(row)
```

The `AND status = 'pending'` guard is the critical change. If another worker already claimed the row, `rowcount` is 0 and this worker backs off. No double execution.

### Option 2: Single atomic statement (preferred if your database supports `RETURNING`)

```python
def acquire_task(worker_id: int):
    """Claim a task in a single atomic statement."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'"          -- guard against the race
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

One round trip. No window. If two workers run this simultaneously, the database serializes the writes and exactly one gets a row back.

### Option 3: `SELECT FOR UPDATE` (PostgreSQL, MySQL — not SQLite)

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"   # ← locks the row, skips rows locked by others
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`SKIP LOCKED` is specifically designed for queue consumers. Workers skip rows already locked by a peer rather than blocking on them, which gives you throughput without contention.

## Tests That Need to Be Added

The existing suite needs concurrency coverage. At minimum:

```python
import threading

def test_acquire_task_is_exclusive_under_concurrency(db, seed_tasks):
    """Only one worker should claim a given task."""
    claimed_ids = []
    lock = threading.Lock()

    def worker(worker_id):
        task = acquire_task(worker_id)
        if task:
            with lock:
                claimed_ids.append(task["id"])

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(4)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # Each task ID must appear exactly once across all workers
    assert len(claimed_ids) == len(set(claimed_ids)), (
        f"Duplicate claims detected: {claimed_ids}"
    )
```

This test would have failed before the fix and passes after it. That is the test that belongs in CI for a multi-worker queue.

## Summary

| | Before | After |
|---|---|---|
| **Claim operation** | SELECT, then UPDATE (two statements, no guard) | Atomic UPDATE with `status = 'pending'` guard, or `SELECT FOR UPDATE SKIP LOCKED` |
| **Race window** | Open between SELECT and UPDATE | Eliminated |
| **Concurrent workers** | Both can claim the same row | Exactly one claim succeeds |
| **Test coverage** | Single-worker only | Concurrency test added |

The visible bug in `execute_task` — the O(n²) deduplication loop — is real and should be replaced with a set, but it is not what is paging you right now. Fix the race first, then clean up the loop in a follow-on PR.
