## Root Cause

**The `acquire_task` function has a race condition. The SELECT and UPDATE are two separate, non-atomic operations with no row-level lock between them.**

Here's the exact failure sequence:

```
Worker 1:  SELECT → finds task #42 (status='pending')
Worker 2:  SELECT → finds task #42 (status='pending')   ← both read before either writes
Worker 1:  UPDATE task #42 SET status='running'
Worker 2:  UPDATE task #42 SET status='running'          ← second claim succeeds, no guard
Worker 1:  execute_task(#42) → complete_task(#42)
Worker 2:  execute_task(#42) → complete_task(#42)        ← duplicate completion
```

Both workers see `status = 'pending'` before either has committed the `UPDATE`. The `WHERE id = ?` clause on the update does nothing to prevent this — it matches on ID, not on the current status, so both updates succeed and both workers proceed to execute and complete the same task.

---

## Why Tests Didn't Catch It

Every test runs with a single worker against a single-threaded in-process database. There is no concurrency, so the race window never opens. The tests are correctly written for the logic they cover — they just don't model the multi-process environment that exists in production. This is a gap in the test suite, not a bug in the tests themselves.

---

## The Fix

The SELECT and UPDATE must be collapsed into a single atomic operation. The standard pattern is `UPDATE ... WHERE status = 'pending' RETURNING ...`, which lets the database's row locking guarantee only one writer wins.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

If the database doesn't support `RETURNING` (older MySQL, for example), the equivalent safe pattern is:

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",          # ← guard re-checked at write time
        (worker_id, claimed_at),
    )
    db.commit()

    if affected == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
```

The critical invariant either way: **status must be checked at write time inside the same statement, not read separately before the write.**

## What Also Needs to Change

**`complete_task` should guard its update the same way:**

```python
def complete_task(task_id: int, worker_id: int, result: list) -> bool:
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', result = ?, completed_at = ?"
        " WHERE id = ? AND worker_id = ? AND status = 'running'",
        (json.dumps(result), datetime.utcnow(), task_id, worker_id),
    )
    db.commit()
    return affected == 1   # caller can log/alert if False
```

Without the `AND worker_id = ? AND status = 'running'` guard, whichever worker loses the claim race can still overwrite the winning worker's completion. Pass `worker_id` through from `run_worker` so the completing worker can assert ownership.

---

## Immediate Mitigation (Before Deploy)

Scale back to 1 worker process. That eliminates the race entirely and stops the bleeding while the fix is tested and deployed. The queue will back up but completions will be correct.

## Summary for the Client Call

> A scaling change three days ago introduced a race condition in task claiming. When multiple workers check for available tasks simultaneously, more than one can claim the same task before any of them marks it as taken. The fix is a one-statement atomic claim operation — the database lock guarantees only one worker wins. We are deploying that fix now and have reduced worker concurrency in the interim to stop duplicate completions immediately.
