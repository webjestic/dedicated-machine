## Root Cause

**Classic check-then-act race condition in `acquire_task`. No atomic claim. Multiple workers are picking up the same task.**

---

### What's happening

`acquire_task` is two separate database operations with nothing protecting the window between them:

```python
# Step 1 — Worker A and Worker B both run this. Both get the same row.
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    ...
)

# Step 2 — Both workers now UPDATE the same id.
# The second UPDATE succeeds silently. No error. No conflict.
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ?, ..."
    " WHERE id = ?",
    (worker_id, ..., row["id"]),
)
```

Under a single worker this never fires. Under four workers at peak load, two workers can execute Step 1 concurrently, both read `status = 'pending'` for the same row, and both proceed to claim it. The second `UPDATE` overwrites the first — `worker_id` changes, but `status` is already `'running'` so nothing downstream notices. Both workers execute and complete the task. Two completion log entries, two downstream events, one task ID.

---

### Why the tests don't catch it

Every test calls `acquire_task` exactly once, in a single thread, against an empty-then-seeded database. There is no concurrent access. The race window is never open. All 34 tests pass. The tests are testing the happy path of a single-worker world.

---

### The fix

The SELECT and the UPDATE must be a single atomic operation. The claim only succeeds if the row is still `pending` at the moment of the write.

**Option 1 — Atomic UPDATE with a status guard (preferred, works on PostgreSQL and SQLite):**

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",   # guard re-checked at write time
        (worker_id, claimed_at),
    )
    db.commit()

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
    return dict(row) if row else None
```

The `AND status = 'pending'` on the `UPDATE` is the critical line. If a second worker already flipped the row to `'running'`, this `UPDATE` matches zero rows. The worker gets nothing back and moves on.

**Option 2 — `SELECT FOR UPDATE` (PostgreSQL):**

```python
# Inside an explicit transaction
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
    " FOR UPDATE SKIP LOCKED"
)
```

`SKIP LOCKED` is purpose-built for this pattern. Rows locked by another worker are skipped entirely rather than blocked on. This is the cleanest solution if you're on Postgres.

---

### What needs to change right now

1. **Deploy the atomic UPDATE fix.** Option 1 works on whatever database you're running. Option 2 is cleaner if you're on Postgres — confirm the DB and pick accordingly.

2. **Add a status guard to `complete_task`** as a secondary defense:

```python
db.execute(
    "UPDATE task_queue"
    " SET status = 'complete', result = ?, completed_at = ?"
    " WHERE id = ? AND status = 'running'",  # don't overwrite a duplicate completion
    (json.dumps(result), datetime.utcnow(), task_id),
)
```

3. **Add a concurrency test** that runs `acquire_task` from two threads simultaneously against the same pending row and asserts exactly one of them gets the task back. That test would have caught this before it shipped.

### Summary for the client call

> Scaling from 1 to 4 workers exposed a race condition in task claiming. The code reads a task and then claims it in two separate steps. Under concurrent load, two workers can read the same task before either has claimed it. Both proceed to execute and complete it. The fix is a single atomic database operation that only succeeds if the task is still unclaimed at write time. No data was lost — tasks were over-processed, not dropped. We are deploying the fix now.
