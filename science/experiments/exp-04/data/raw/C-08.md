## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. The SELECT and UPDATE are two separate, unprotected operations. Between them, any other worker can read the same row.**

This is not a logic bug. The code does exactly what it says. The problem is that "what it says" is not atomic.

---

## What Is Actually Happening

Under a single worker, this worked fine for months. Under four workers, this is the sequence that produces the duplicate completions you are seeing in the log:

```
T=0  Worker 1: SELECT → finds task #42, status='pending'
T=0  Worker 2: SELECT → finds task #42, status='pending'   ← same row, same moment
T=1  Worker 1: UPDATE task #42 SET status='running', worker_id=1
T=1  Worker 2: UPDATE task #42 SET status='running', worker_id=2  ← overwrites W1
T=2  Worker 1: execute_task(#42) → complete_task(#42)
T=2  Worker 2: execute_task(#42) → complete_task(#42)
```

Both workers read `status = 'pending'`. Both proceed past the `if not row` guard. Both claim the task. Both execute it. Both write a completion record. The completion log shows task #42 twice with different worker IDs because that is exactly what happened.

The UPDATE in Worker 2 silently overwrites Worker 1's claim. There is no error. There is no exception. The tests all pass because every test runs with a single caller. The race window does not exist in a single-threaded test suite.

---

## Why the Tests Did Not Catch This

The tests are testing the right things for a single worker. They are not wrong. They are incomplete. There is no test that calls `acquire_task` concurrently from two threads or processes and asserts that only one of them gets the task. That test would have failed immediately and pointed directly here.

The CI green light is not evidence the code is correct. It is evidence the code is correct when only one thing runs at a time.

---

## The Fix

The SELECT and the UPDATE must be a single atomic operation. The database must be the arbiter, not the application.

### Option 1 — Atomic UPDATE with a WHERE guard (preferred for most databases)

```python
def acquire_task(worker_id: int):
    """
    Claim a task atomically. The WHERE clause on status='pending' means only
    one worker can win the UPDATE. All others update zero rows and get nothing.
    """
    # First, find the candidate task ID with a subquery
    # Then UPDATE with the status guard in the same statement.
    # The database serializes concurrent UPDATEs on the same row.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",   # ← guard re-checked at UPDATE time
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if affected.rowcount == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE worker_id = ? AND status = 'running'",
        (worker_id,)
    )
```

The critical detail is `AND status = 'pending'` on the UPDATE itself, not just on the SELECT. If two workers race to this UPDATE, the database serializes them. The first one changes the status to `'running'`. The second one evaluates `status = 'pending'` — which is now false — and updates zero rows. `rowcount == 0` means you lost the race. Return `None` and loop.

### Option 2 — `SELECT FOR UPDATE` with an explicit transaction

If your database supports it (PostgreSQL does; SQLite does not):

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"   # lock the row, skip rows locked by others
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is purpose-built for job queues. It locks the selected row for the duration of the transaction and skips any rows already locked by another worker. No contention, no blocking, no duplicates.

### The test that should have existed

```python
import threading

def test_acquire_task_is_exclusive_under_concurrency(db, seed_tasks):
    """Only one worker should be able to claim a given task."""
    results = []

    def worker(wid):
        task = acquire_task(worker_id=wid)
        if task:
            results.append((wid, task["id"]))

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(4)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    task_ids = [r[1] for r in results]
    assert len(task_ids) == len(set(task_ids)), (
        f"Same task claimed by multiple workers: {results}"
    )
```

Add this test. It will fail against the current code. It will pass after the fix. That is the signal you want.

## Immediate Mitigation

While the fix is being deployed:

1. **Scale back to 1 worker process.** This closes the race window immediately. The queue will back up but downstream consumers will stop seeing duplicates.
2. **Do not add deduplication logic in the downstream consumers** as a permanent fix. That treats the symptom. The bug is here, in `acquire_task`.
3. **Audit the completion log** for the past three days — since the scale-up — and identify every task ID that appears more than once. Those downstream effects need to be assessed and potentially reversed.

## Summary

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: SELECT and UPDATE are separate statements with no lock between them |
| **Trigger** | Scaling from 1 to 4 workers opened the race window |
| **Why tests missed it** | All tests are single-caller; the race requires concurrent callers |
| **Fix** | Make the claim atomic: UPDATE with status guard, or SELECT FOR UPDATE SKIP LOCKED inside a transaction |
| **Mitigation now** | Scale back to 1 worker; audit duplicated task IDs |
