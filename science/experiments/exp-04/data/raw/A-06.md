## Root Cause

**Classic check-then-act race condition. No atomic claim on the queue row.**

Here is the exact failure sequence:

```
Worker 1                              Worker 2
--------                              --------
SELECT ... WHERE status = 'pending'
                                      SELECT ... WHERE status = 'pending'
-- both read the same row --
UPDATE ... SET status = 'running'
                                      UPDATE ... SET status = 'running'
execute_task(task)
                                      execute_task(task)
complete_task(task_id, result)
                                      complete_task(task_id, result)
```

The SELECT and the UPDATE are two separate statements with no lock held between them. Under a single worker this never surfaces. Under four workers at peak load, the window is wide enough to drive a truck through. Both workers read `status = 'pending'`, both issue the UPDATE, both proceed to execute and complete. That is exactly what the completion log is showing: same task ID, different worker IDs.

The tests pass because every test runs a single worker against an isolated database. The race requires concurrent readers. The test suite has no concurrency. It will never catch this.

---

## What Needs to Change

The SELECT and the UPDATE must become a single atomic operation. The database must do the claim, not the application.

**Option 1 — `UPDATE ... WHERE status = 'pending'` with `RETURNING` (PostgreSQL)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        """
        UPDATE task_queue
           SET status     = 'running',
               worker_id  = ?,
               claimed_at = ?
         WHERE id = (
               SELECT id FROM task_queue
                WHERE status = 'pending'
                ORDER BY priority DESC, created_at ASC
                LIMIT 1
                FOR UPDATE SKIP LOCKED   -- skip rows another worker already locked
         )
         RETURNING id, type, payload
        """,
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is the key. It tells the database: lock the candidate row for my UPDATE and skip any row already locked by another transaction. Two workers running this simultaneously will land on different rows. The claim is atomic. The window is gone.

**Option 2 — If your database does not support `RETURNING` or `FOR UPDATE SKIP LOCKED`**

Use an optimistic lock with a `claimed_by` token:

```python
def acquire_task(worker_id: int):
    import uuid
    claim_token = str(uuid.uuid4())

    db.execute(
        """
        UPDATE task_queue
           SET status      = 'running',
               worker_id   = ?,
               claim_token = ?,
               claimed_at  = ?
         WHERE id = (
               SELECT id FROM task_queue
                WHERE status = 'pending'
                ORDER BY priority DESC, created_at ASC
                LIMIT 1
         )
        """,
        (worker_id, claim_token, datetime.utcnow()),
    )
    db.commit()

    # Only this worker knows its claim_token. If we got the row, it's ours.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE claim_token = ?",
        (claim_token,),
    )
    return dict(row) if row else None
```

This is less clean and still has edge cases under certain isolation levels. Option 1 is strongly preferred.

## What to Tell the Client Call in 2 Minutes

> We scaled from 1 to 4 workers three days ago. The queue claim logic was not written to be safe under concurrent access. Multiple workers can read the same task in the same instant and both process it. We have identified the fix — a single atomic database operation that prevents two workers from ever claiming the same row. We are applying it now.

## What Not to Do

Do not add an application-level lock, a Redis mutex, or a retry-on-duplicate check. Those are layered complexity on top of a broken primitive. Fix the primitive. The database already has the machinery to make this safe. Use it.

The `execute_task` deduplication logic and the `complete_task` write are not the problem. Do not touch them in this patch. One change, one reason.
