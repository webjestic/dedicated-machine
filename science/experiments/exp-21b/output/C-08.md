Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `nx=True`, token-based ownership verification, and heartbeat renewal are all correctly conceived. However, there are several correctness, safety, and architectural issues that must be resolved before this handles real flash sale load in a government-grade system. The most critical flaw is a race condition in the heartbeat's check-then-extend sequence and a concurrency bug in the `InventoryService` instance model that would cause lock corruption under concurrent requests.

---

## Critical Issues

### 1. Heartbeat Race Condition (Check-Then-Extend is Non-Atomic)

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You are then extending a lock you no longer own. This must be a single atomic operation via a Lua script:

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In the heartbeat loop
result = self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not result:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
    self._heartbeat_active = False
    break
```

The same atomic-script pattern must also be applied to lock release in the `finally` block.

### 2. Instance-Level Heartbeat State Breaks Concurrent Requests

```python
# These are instance variables, not per-call variables
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly instantiated once and shared across requests (dependency injection, singleton, etc.). Two concurrent calls to `reserve_item` will clobber each other's `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat thread.

Refactor heartbeat state to be local to the lock context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    # ... acquire lock ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script
```

Using `threading.Event` is also cleaner than a boolean flag for cross-thread signaling.

---

### 3. Lock Release in `finally` is Also Non-Atomic

```python
# CURRENT — same GET/DELETE race as the heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Apply the same Lua script pattern:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
```

## Significant Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, whitespace, or a path-traversal-style string. `quantity` could be zero or negative, which would silently *increase* stock.

```python
# Negative quantity passes the stock check and corrupts inventory
if current_stock < -5:  # False — proceeds
    new_stock = 10 - (-5)  # = 15, stock increases
```

Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

### 5. `db.update_stock` and `db.create_order` Are Not Transactional

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. This is a silent data integrity failure — inventory is consumed with no corresponding order.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # if this throws, stock is gone
```

These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined — it may still be running
```

The daemon thread could still be mid-execution when the lock is released, creating a window where a released lock gets its TTL extended by a zombie heartbeat. Always join with a timeout:

```python
def _stop_heartbeat(self, thread: threading.Thread) -> None:
    heartbeat_active.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if thread.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly")
```

## Minor Issues

### 7. Test Coverage Gaps

- No test for the negative/zero quantity input case
- No test verifying the DB transaction rollback behavior when `create_order` fails
- `test_lock_not_released_if_expired` mocks `_start_heartbeat` globally, so it does not actually exercise the token-mismatch path in the real `finally` block — it tests a mock, not the implementation
- No test for the heartbeat's atomic ownership check

### 8. `LOCK_TTL = 30` With No Documented Basis

Thirty seconds is a long TTL for a lock in a flash sale context where you want high throughput. This value should be configurable via constructor parameter or environment variable, not a hardcoded class constant, so it can be tuned per deployment environment.

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 10):
```

### 9. `RuntimeError` Is Too Broad for Lock Failure

Catching `RuntimeError` will also swallow unrelated runtime errors from within the lock body. Use a dedicated exception class:

```python
class LockAcquisitionError(Exception):
    pass
```
