Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a government-grade production system. The most critical problems are a race condition in the heartbeat's lock extension, instance-level state that breaks concurrent use, and missing input validation despite the checklist claiming otherwise.

---

## Critical Issues

### 1. Race Condition in Heartbeat Lock Extension

The heartbeat performs a non-atomic check-then-extend:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← not atomic with the GET
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock (if the TTL expired in that window), and you would then extend a lock you no longer own. This is a TOCTOU race. The extension must be atomic. Use a Lua script:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init:
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat:
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat could not extend lock — lock lost key=%s", lock_key)
    self._heartbeat_active = False
```

The same atomic pattern should also be applied to the lock release in `_acquire_lock`. The current `GET` + `DELETE` is also a TOCTOU race. Use the standard Lua release script:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is used as a shared singleton (the expected deployment pattern), two concurrent calls to `reserve_item` will corrupt each other's heartbeat state. The first call's `_stop_heartbeat()` will kill the second call's thread, or vice versa.

Heartbeat state must be scoped to the lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Lock lost during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                break

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire, start thread, yield, stop thread
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which can delay shutdown by up to `HEARTBEAT_INTERVAL` seconds.

### 3. No Input Validation Despite Checklist Claiming Otherwise

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw, unvalidated parameters directly:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that `item_id` is non-empty, within expected length bounds, or free of injection-relevant characters. There is no validation that `quantity` is a positive integer. A caller passing `quantity=0`, `quantity=-5`, or `item_id=""` will silently produce incorrect behavior or a misleading lock key like `inventory:lock:`. Define a request model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

## Moderate Issues

### 4. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate calls with no transaction wrapping:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the DB connection drops between these two calls, stock is decremented but no order exists. This is a data integrity failure. These must execute within a single database transaction. The `db` abstraction should expose a transaction context manager, or this method should explicitly manage one.

### 5. Heartbeat Interval Is Too Close to Lock TTL

`LOCK_TTL = 30s` and `HEARTBEAT_INTERVAL = 10s`. The first heartbeat fires at T+10s, which is fine. But if the heartbeat thread is delayed by GIL contention, a slow Redis round-trip, or OS scheduling, the extension may not land before T+30s. A safer ratio is heartbeat at ≤ TTL/3, with TTL itself being more generous (e.g., 60s TTL, 15s heartbeat). Document the reasoning explicitly in the constants.

### 6. `db` Abstraction Has No Defined Interface

The `db` parameter is typed as a bare `db` with no protocol or abstract base class. This makes the contract invisible to callers, makes mocking fragile, and prevents static analysis from catching misuse. Define an explicit protocol:

```python
from typing import Protocol

class InventoryDB(Protocol):
    def get_stock(self, item_id: str) -> Optional[int]: ...
    def update_stock(self, item_id: str, new_stock: int) -> None: ...
    def create_order(self, order_id: str, item_id: str, quantity: int) -> None: ...
```

## Test Issues

### 7. Tests Mock Away the Exact Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the race condition identified above — is never exercised by any test. The tests verify the reservation logic in isolation, which has value, but the distributed locking behavior is entirely untested. Add integration-level tests (using `fakeredis`) that exercise the full lock lifecycle, including:

- Heartbeat actually extending TTL
- Lock not released when token mismatches
- Concurrent reservation attempts where only one succeeds

### 8. `test_lock_not_released_if_expired` Does Not Actually Test What It Claims

The test mocks `redis.get` to return `b"different-token"` and asserts `delete` is not called. But because `_start_heartbeat` is also mocked, the token used internally by `_acquire_lock` is a real `uuid4`, not `"test-token"`. The test passes for the right behavioral reason but for the wrong mechanical reason — it would pass even if the ownership check were removed, depending on mock wiring. This test needs to be rewritten against a `fakeredis` instance to be meaningful.
