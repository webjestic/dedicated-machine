Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes the heartbeat mechanism unsafe under concurrent use, and the lock release in the `finally` block contains a non-atomic check-and-delete race that undermines the ownership guarantee the code is trying to enforce. Several secondary issues compound the risk profile enough that this should not merge in its current form.

---

## Critical Issues

### 1. Shared Heartbeat State Breaks Concurrent Safety

`_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`. If two requests share the same service instance — which is the normal deployment pattern for a singleton service — concurrent calls to `reserve_item` will race on these fields.

```python
# Thread A calls reserve_item("item_001") → sets _heartbeat_active = True
# Thread B calls reserve_item("item_002") → sets _heartbeat_active = True
# Thread A's finally block calls _stop_heartbeat() → sets _heartbeat_active = False
# Thread B's heartbeat thread now silently stops extending its lock
```

The heartbeat is the entire defense against lock expiry under slow DB operations. Silently killing it mid-operation means the lock can expire, another process acquires it, and you now have two processes executing the stock decrement concurrently — the exact failure mode this code exists to prevent.

**Required fix:** Heartbeat state must be scoped to the lock acquisition context, not the instance. Use a local `threading.Event` per acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into the heartbeat thread
    # call stop_event.set() in finally
```

### 2. Non-Atomic Lock Release Is a TOCTOU Race

The `finally` block performs a read-then-delete as two separate Redis operations:

```python
current = self.redis.get(lock_key)          # operation 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # operation 2
```

Between `GET` and `DELETE`, the lock TTL can expire and a second process can acquire it with a new token. The `DELETE` then executes and destroys the second process's valid lock. This is a well-documented failure mode and the standard mitigation is a Lua script that makes the comparison and deletion atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

The same TOCTOU race exists in the heartbeat's `expire` call — it checks ownership then extends, but another process could acquire the lock between those two operations.

## Significant Issues

### 3. No Input Validation on `item_id` or `quantity`

The checklist marks "Input validation via Pydantic" but there is no validation on the actual entry point. `item_id` is passed directly into a Redis key and a DB query without sanitization. A caller supplying `item_id = "../../admin"` or an empty string, or `quantity = -5` or `quantity = 0`, reaches the DB layer unchecked. Pydantic is imported and a response model is defined, but no request model is validated at the boundary.

```python
# quantity = -5 passes the stock check: current_stock(10) < -5 is False
# so it proceeds to: new_stock = 10 - (-5) = 15
# stock silently increases on a "reservation"
```

This is a correctness and potential abuse vector. Validate at the method boundary:

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
```

### 4. Heartbeat Has an Off-by-One Timing Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat sleeps *first*, then checks. If the lock is acquired at T=0 and the heartbeat thread starts at T=0, the first extension attempt happens at T=10. That is fine. But `_stop_heartbeat()` sets the flag to `False` and returns immediately — it does not join the thread. The thread may be mid-sleep when the flag is cleared, and the `finally` block proceeds to release the lock. This is a minor race but worth noting: the thread should be joined after stopping.

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 5. DB Operations Are Not Transactional

`update_stock` and `create_order` are two separate DB calls with no transaction boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock has already been decremented and the order does not exist. The distributed lock protects against concurrent Redis races, but it does not protect against partial DB writes. These must execute within a single database transaction.

## Test Coverage Gaps

### 6. Heartbeat Is Mocked Away in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test stubs out the heartbeat entirely. The concurrent state bug in issue #1 will not be caught by this suite. There are no tests for:

- Two concurrent `reserve_item` calls on the same instance
- Heartbeat correctly extending TTL under a slow DB simulation
- Heartbeat stopping cleanly without joining
- The Lua-based atomic release path (once fixed)

### 7. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test verifies that `delete` is not called when the token does not match, which is correct behavior — but it does not test the scenario where the lock expires and is *re-acquired by another process* before the delete fires. That is the actual dangerous case and it requires the atomic Lua fix to be safe.
