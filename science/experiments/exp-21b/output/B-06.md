Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes it unsafe for concurrent use: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on shared heartbeat state, allowing one call's `_stop_heartbeat()` to silently kill another call's renewal thread mid-operation. There is also a non-atomic check-then-act race in the heartbeat's `get`/`expire` sequence, and the test suite mocks out the heartbeat entirely, providing zero coverage of the mechanism the PR's headline feature depends on. These issues collectively undermine the correctness guarantees the distributed lock is meant to provide.

---

## Detailed Findings

### Critical — Instance-Level Heartbeat State (Concurrency Unsafety)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` is a single boolean on the instance. If two threads call `reserve_item` concurrently on the same service object:

- Thread A acquires lock A, sets `_heartbeat_active = True`, starts heartbeat thread A.
- Thread B acquires lock B, sets `_heartbeat_active = True`, starts heartbeat thread B.
- Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`.
- Heartbeat thread B now exits its `while self._heartbeat_active` loop and stops renewing lock B.
- Lock B expires while Thread B is still inside the critical section performing DB writes.

This is a silent failure mode. The lock appears held from the application's perspective, but Redis has already expired it and may have granted it to a third party. The `InventoryService` is almost certainly intended to be a singleton or long-lived object, making this a near-certain production defect under flash sale load — exactly the scenario this code is designed for.

**Required fix:** Heartbeat state must be local to each lock acquisition, not shared instance state.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # scoped to this acquisition

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_active.set()
    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release via Lua script (see below)
        self._release_lock(lock_key, token)
```

### High — Non-Atomic Heartbeat Renewal (TOCTOU Race)

```python
# Current heartbeat implementation
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is the same class of bug the lock release already correctly defends against — but the defense was not applied consistently.

**Required fix:** Use a Lua script to make ownership check and renewal atomic.

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _run_heartbeat(
    self,
    lock_key: str,
    token: str,
    active: threading.Event
) -> None:
    script = self.redis.register_script(self.HEARTBEAT_SCRIPT)
    while active.wait(timeout=self.HEARTBEAT_INTERVAL):
        if not active.is_set():
            break
        result = script(keys=[lock_key], args=[token, self.LOCK_TTL])
        if result == 0:
            logger.error(
                "Heartbeat lost lock ownership key=%s — lock may have expired",
                lock_key
            )
            # Signal the owning context that the lock is gone
            active.clear()
            break
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Note the heartbeat should also signal back to the critical section when it detects lost ownership, so the application can abort the DB writes rather than proceeding without a lock.

### High — Lock Release is Also Non-Atomic

```python
# Current release logic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same TOCTOU issue. Between `GET` and `DELETE`, another process can acquire the lock, and this code will delete the new owner's lock. The standard fix is well-established.

**Required fix:** Atomic release via Lua script.

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(self.RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock release skipped — token mismatch or already expired key=%s",
            lock_key
        )
```

### Medium — Heartbeat Thread is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # _heartbeat_thread.join() is never called
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10s) between iterations. After `_stop_heartbeat()` sets the flag to `False`, the thread may still be mid-sleep and will not actually exit for up to 10 seconds. During that window, the thread holds a reference to `lock_key` and `token` and may attempt a renewal after the lock has been intentionally released and potentially re-acquired by another process.

Using `threading.Event` as shown above resolves this, since `event.wait(timeout=...)` wakes immediately when the event is cleared.

### Medium — Missing Input Validation on `reserve_item`

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is no guard against:

- `quantity <= 0` (reserving zero or negative units would silently corrupt stock)
- Empty or malformed `item_id` strings that could produce unexpected Redis key shapes

```python
# Minimum required guards
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not isinstance(item_id, str):
        return ReservationResult(success=False, message="Invalid item_id")
    if not isinstance(quantity, int) or quantity <= 0:
        return ReservationResult(success=False, message="Quantity must be a positive integer")
```

Pydantic is only used for the response model, not for request validation. The checklist item is misleading.

### Medium — Test Suite Mocks Out the Feature Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test except `test_lock_not_released_if_expired` uses this fixture. The heartbeat — the primary novel mechanism in this PR — has zero meaningful test coverage. The concurrency bug described above would not be caught by any test here. At minimum, the following should be covered with real (non-mocked) heartbeat execution:

- Heartbeat correctly extends TTL when token matches
- Heartbeat does not extend TTL when token has changed
- Heartbeat stops promptly when the context manager exits
- Concurrent `reserve_item` calls on the same instance do not interfere with each other's heartbeat state

---

### Low — Daemon Thread Lifecycle Under Gunicorn/uWSGI

`daemon=True` means the heartbeat thread will be killed if the main thread exits, but under multi-worker WSGI servers, worker processes are forked after the `InventoryService` is potentially initialized. Forked child processes inherit the parent's memory but not its threads. If the service is instantiated at module import time and then forked, the heartbeat thread will not exist in the child process. This is unlikely to be an issue if the service is instantiated per-request or per-worker, but it should be documented explicitly.
