Decision: Request Changes

Summary: The implementation contains a critical instance-level heartbeat state design that makes this service fundamentally unsafe for concurrent use — multiple simultaneous reservations on different items will corrupt each other's heartbeat lifecycle through shared `_heartbeat_active` and `_heartbeat_thread` fields. Beyond that architectural flaw, the lock release in the `finally` block contains a TOCTOU race between the `GET` and `DELETE` operations that can cause a process to delete a lock it no longer owns, and the test suite mocks away the heartbeat entirely rather than validating the behavior that is the primary justification for this design.

---

## Detailed Findings

### 1. Critical — Shared Heartbeat State Breaks Concurrent Reservation Calls

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped locals. If two coroutines or threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the expected production scenario for a flash sale — the following sequence is possible:

```
Thread A: _start_heartbeat("inventory:lock:item_001", token_A)
           sets self._heartbeat_active = True
Thread B: _start_heartbeat("inventory:lock:item_002", token_B)
           overwrites self._heartbeat_thread with B's thread
Thread A: _stop_heartbeat()
           sets self._heartbeat_active = False
           — now B's heartbeat thread also exits on its next iteration
           — lock for item_002 expires while Thread B is still writing
```

Thread B's lock silently expires mid-write. Stock is decremented, but the lock is gone. A third thread can now acquire the same lock and observe stale stock before Thread B's `create_order` commits. This is a **phantom inventory** / **double-spend** condition.

**Required fix:** Heartbeat state must be call-scoped, not instance-scoped.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning(
                    "Heartbeat: lock ownership lost key=%s — stopping beat",
                    lock_key
                )
                break

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release below — see finding #2
```

Note the use of `threading.Event` for clean signaling and `heartbeat.join()` in the `finally` block. Without `join()`, the daemon thread can attempt a Redis `expire` call **after** the lock has already been deleted, spuriously re-extending a key that now belongs to another owner if Redis key reuse occurs within the TTL window.

### 2. High — TOCTOU Race in Lock Release (GET then DELETE is Not Atomic)

```python
# Current release logic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock TTL can expire and be acquired by another process with a new token. The `DELETE` then fires and removes the **new owner's lock**, not the original one. This is a well-documented failure mode in Redis locking literature.

**Required fix:** Use a Lua script to make the compare-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init or module level
# self._release_script = self.redis.register_script(RELEASE_SCRIPT)

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    released = bool(result)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock release skipped — token mismatch or already expired key=%s",
            lock_key
        )
    return released
```

### 3. High — Input Validation Is Declared but Not Enforced

The checklist claims "Input validation via Pydantic," but `ReservationResult` is an **output** model. `reserve_item` accepts raw `item_id: str` and `quantity: int` with no validation at the entry point:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # item_id could be empty string, whitespace, path traversal fragment
    # quantity could be 0, negative, or absurdly large
```

A caller passing `quantity=0` will successfully "reserve" zero units and generate a real `order_id`. A caller passing `quantity=-5` will **increase** stock. Neither is caught.

**Required fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

    @field_validator("item_id")
    @classmethod
    def no_whitespace(cls, v: str) -> str:
        if v != v.strip():
            raise ValueError("item_id must not have leading or trailing whitespace")
        return v
```

The lock key is constructed directly from `item_id`:

```python
lock_key = f"inventory:lock:{item_id}"
```

Without the pattern constraint above, a crafted `item_id` such as `"../../admin"` or `"item:lock:other_item"` could produce a lock key that collides with or escapes the intended namespace. Enforce the alphanumeric pattern.

### 4. Medium — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is OOM-killed, or the DB connection drops between these two calls, stock is decremented but no order record exists. The item is effectively lost. The distributed lock does not protect against this — it only serializes access, it does not provide durability.

**Required fix:** These two operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the database layer, but the intent must be enforced at this level, not assumed from callers.

### 5. Medium — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # <-- stop never runs
    return svc
```

The heartbeat is the primary novel mechanism in this PR. Mocking it out means the tests provide **zero coverage** of the behavior that justifies the design complexity. There are no tests for:

- Heartbeat extending TTL before expiry
- Heartbeat stopping cleanly on lock release
- Heartbeat detecting token mismatch and halting
- Concurrent calls not corrupting each other's heartbeat state (which would have caught finding #1)

The TOCTOU release path (`test_lock_not_released_if_expired`) is also not meaningful because the mock bypasses the actual `GET`/`DELETE` sequence.

Tests should be restructured to use a real or fake Redis (e.g., `fakeredis`) so that lock mechanics execute as written.

### 6. Low — Heartbeat Sleep Is Not Interruptible

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 seconds
        if self._heartbeat_active:
            ...
```

With `time.sleep(10)`, stopping the heartbeat after a fast operation requires waiting up to 10 seconds for the sleep to return. During a graceful shutdown or rapid lock release, this delays thread cleanup and can cause the `finally` block to hang if a `join()` is added (as recommended in finding #1). The `threading.Event.wait(timeout=...)` pattern shown above resolves this.
