Decision: Request Changes

Summary: The core locking logic and heartbeat concept are sound, but there are several architectural flaws that make this unsafe for a concurrent production environment. Most critically, the heartbeat and lock state are stored as instance variables on a shared service object, meaning concurrent calls from multiple threads will race on `_heartbeat_active` and `_heartbeat_thread`, corrupting each other's lock state. Additionally, the lock release in the `finally` block contains a non-atomic check-then-delete race condition, and input validation is absent at the service boundary despite being listed as complete on the checklist.

---

## Detailed Findings

### 🔴 Critical — Shared Mutable State Breaks Concurrency (Architectural Flaw)

This is the most serious issue. `InventoryService` is almost certainly instantiated once and shared across threads (standard practice with dependency injection, Flask app context, FastAPI lifespan, etc.). The heartbeat state is stored on `self`:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently, Thread A and Thread B will both write to `self._heartbeat_active` and `self._heartbeat_thread`. Thread A's `_stop_heartbeat()` call sets `self._heartbeat_active = False`, which also kills Thread B's heartbeat. Thread B then loses its lock prematurely, which is exactly the failure mode this code was designed to prevent.

**Fix:** Heartbeat state must be local to each lock acquisition, not stored on the instance. Pass state through a shared mutable container or encapsulate it in a context-local object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = {"running": False}  # local to this call
    acquired = False

    # ... acquire lock ...

    heartbeat_active["running"] = True
    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)

    try:
        yield token
    finally:
        heartbeat_active["running"] = False
        # ... release lock ...
```

### 🔴 Critical — Non-Atomic Lock Release (TOCTOU Race)

The `finally` block performs a read-then-delete as two separate commands:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock (if it expired) and set a new token. The `GET` saw our token, but the `DELETE` now removes the new owner's lock. This is a well-known hazard documented in the Redis distributed locks guide.

**Fix:** Use the canonical atomic Lua script for safe release:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
```

### 🔴 Critical — Input Validation Absent Despite Checklist Claiming Otherwise

The checklist marks input validation via Pydantic as complete, but `reserve_item` accepts raw, unvalidated parameters:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that `item_id` is non-empty, within expected length bounds, or free of injection-relevant characters. There is no check that `quantity` is a positive integer. A caller passing `quantity=0` or `quantity=-5` would silently pass the stock check (`current_stock < quantity` is false for negatives) and write a corrupted order to the database.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 🟡 Major — `db.update_stock` and `db.create_order` Are Not Atomic

The two database writes are separate calls with no transaction boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the DB connection drops between these two calls, stock is decremented but no order is created. The item is effectively lost. The distributed lock protects against concurrent access but does nothing to protect against partial writes.

**Fix:** These two operations must be wrapped in a single database transaction. The `db` abstraction should expose a transactional interface:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Major — Heartbeat Has a Check-Then-Extend Race

The heartbeat itself repeats the same TOCTOU pattern as the lock release:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock could expire and be acquired by another process. The heartbeat then extends a lock it no longer owns.

**Fix:** Use a Lua script for atomic check-and-extend:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

---

### 🟡 Major — Tests Mock Away the Code Under Test

The test fixture patches out both `_start_heartbeat` and `_stop_heartbeat`:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the concurrency behavior — the primary risk surface of this entire service — is never tested. The tests only verify the happy path of the stock logic. There are no tests for the heartbeat renewal, the atomic release, concurrent reservation attempts, or the transaction integrity of the two DB writes.

---

### 🟢 Minor — `LOCK_TTL` and `HEARTBEAT_INTERVAL` Should Be Constructor Parameters

Hardcoded class-level constants make the service difficult to tune for different deployment environments without modifying source code.

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 10):
    self.lock_ttl = lock_ttl
    self.heartbeat_interval = heartbeat_interval
```

---

### 🟢 Minor — Bare `RuntimeError` Is Too Broad

Catching `RuntimeError` in `reserve_item` will silently swallow unexpected programming errors that happen to raise `RuntimeError` inside the lock context. Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass
```
