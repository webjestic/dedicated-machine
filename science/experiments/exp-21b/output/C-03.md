Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several correctness and safety flaws that could cause silent data corruption, race conditions, or lock mismanagement under real concurrent load — which is exactly the scenario this service is designed for. The issues below must be resolved before this can be merged.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton (shared across request threads). Two concurrent calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, potentially allowing the first lock to expire mid-operation.

**Fix:** Move heartbeat state into the lock context itself, scoped to the individual acquisition.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock was lost; stop extending

    # ... acquire lock, then:
    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock
```

### 2. Lock Release Is Not Atomic (TOCTOU Race)

```python
# Current — non-atomic check-then-delete
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock (if it expired) and have it deleted out from under them. This is a classic TOCTOU bug. The canonical fix for Redis lock release is a Lua script, which executes atomically on the Redis server.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if not released:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same non-atomic pattern exists in the heartbeat's check-then-extend. That should also use a Lua script or at minimum be understood as best-effort.

### 3. No Input Validation on Public Method

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation at the entry point.

```python
# item_id="" or quantity=-5 will silently pass through
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A negative quantity would decrement stock *upward*, which is a data integrity bug. An empty `item_id` would create a lock key of `inventory:lock:` and potentially collide across all items.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 4. Database Operations Are Outside Any Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. The distributed lock only protects against concurrent Redis access — it does not substitute for a database transaction. These two calls must be wrapped in a single atomic DB transaction, or the service must implement a compensating rollback.

## Moderate Issues

### 5. Heartbeat Can Silently Fail

The heartbeat thread has no exception handling. A Redis connection error inside `_beat()` will silently kill the thread, the lock will expire, and the operation will continue believing it holds the lock it no longer owns.

```python
def _beat():
    while ...:
        try:
            ...
        except redis.RedisError as e:
            logger.error("Heartbeat failed, lock may expire key=%s: %s", lock_key, e)
            break
```

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat()` sets a flag but does not join the thread. In the current (flawed) instance-level design, a stopped heartbeat thread could still be running when the next request starts a new one, leading to two threads extending the same key with potentially different tokens.

## Test Issues

### 7. Tests Mock Away the Exact Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- mocked out
    svc._stop_heartbeat = MagicMock()   # <-- mocked out
    return svc
```

The heartbeat is the most operationally risky part of this service and it is entirely untested. There should be at least one integration-style test that allows the real heartbeat to run and verifies TTL extension behavior.

### 8. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test verifies that `redis.delete` is not called when the token mismatches, but because `_start_heartbeat` is mocked, the token in the mock's `get` return value (`b"different-token"`) never matches the UUID generated internally. The test passes trivially and does not actually validate the atomic release path.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat state must be scoped per lock acquisition, not per instance |
| 2 | Critical | Lock release must use an atomic Lua script |
| 3 | Critical | Input validation must be enforced at the method boundary |
| 4 | Critical | DB operations must be wrapped in a transaction with rollback |
| 5 | Moderate | Heartbeat thread must handle Redis exceptions |
| 6 | Moderate | Heartbeat thread must be joined on stop |
