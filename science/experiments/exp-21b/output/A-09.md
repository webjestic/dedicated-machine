Decision: Request Changes

Summary: The implementation contains a critical instance-level concurrency flaw that makes it unsafe for any real concurrent workload: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields and corrupt each other's heartbeat lifecycle. Beyond that structural defect, there are several additional correctness and security issues detailed below that must be resolved before this can be approved.

---

## Critical Issues

### 1. Heartbeat State Is Instance-Level, Not Call-Level

This is the most serious flaw in the submission.

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or a shared service object. When two requests call `reserve_item` concurrently, they share `_heartbeat_active`. The sequence below is not hypothetical — it is the normal operating condition of a flash sale:

```
Thread A: _start_heartbeat() → sets self._heartbeat_active = True
Thread B: _start_heartbeat() → sets self._heartbeat_active = True (ok so far)
Thread A: _stop_heartbeat() → sets self._heartbeat_active = False
Thread B: heartbeat loop exits immediately — lock B is now unguarded
Thread B: slow DB write continues with no TTL renewal → lock expires mid-write
Thread C: acquires the same lock → two writers now hold the same lock simultaneously
```

The fix is to scope heartbeat state to the lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()   # call-scoped, not instance-scoped
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 5.0)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic compare-and-delete via Lua script (see issue #3)
        self._release_lock(lock_key, token)
```

### 2. Lock Release Is Not Atomic — TOCTOU Race

```python
# Current — two operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock TTL can expire and be acquired by another process. You then delete a lock you no longer own. This is a well-known failure mode documented in the Redis SETNX pattern literature. The fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    released = result == 1
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
    return released
```

The same TOCTOU race exists inside the heartbeat thread:

```python
# Also not atomic — same problem
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Replace with a Lua script or use `SET key token XX EX ttl` only if the value matches, which requires scripting since Redis has no native conditional EXPIRE.

### 3. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is not joined — no guarantee it has stopped
```

The thread may still be running when `_acquire_lock` exits its `finally` block and attempts lock release. If the heartbeat fires between `_stop_heartbeat()` and the `DELETE`, it will extend the TTL of a lock that is about to be deleted — a minor correctness issue. More importantly, if the thread holds a reference to a Redis connection from a pool, you may leak connections under load. Always join with a timeout.

### 4. `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`. A caller passing `quantity=0` will succeed and create a zero-quantity order record. A caller passing `quantity=-5` will *increase* stock (`new_stock = current_stock - (-5)`), which is a logic vulnerability that could be exploited to manufacture inventory. This must be validated before the lock is acquired:

```python
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")
if not item_id or not isinstance(item_id, str):
    return ReservationResult(success=False, message="Invalid item_id")
```

The checklist claims "Input validation via Pydantic" but Pydantic is only used for the *output* model. There is no Pydantic model or any other validation on the inputs to `reserve_item`.

---

### 5. DB Writes Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — the lock only serializes access, it does not provide atomicity across the DB operations. These two writes must be wrapped in a database transaction. The current code will silently lose inventory on any `create_order` failure.

## Test Coverage Gaps

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic — the most operationally complex part of this PR — has zero test coverage. The critical concurrency bug in issue #1 would not be caught by any test here.

Additional missing cases:

| Scenario | Covered |
|---|---|
| Concurrent calls on same instance racing on `_heartbeat_active` | No |
| `create_order` raises after `update_stock` succeeds | No |
| Heartbeat fires after lock is released (post-stop race) | No |
| `quantity=0` or `quantity=-1` | No |
| `item_id` is empty string or whitespace | No |
| Redis connection error mid-operation | No |

The test `test_lock_not_released_if_expired` is structurally correct in intent but does not actually test the TOCTOU race it claims to cover, because the mock bypasses the real lock release path entirely.

## Minor Issues

- **Backoff has no cap.** With `retry_attempts=3` the maximum sleep is `0.4s`, which is acceptable today, but if `retry_attempts` is ever increased the backoff is unbounded. Add `min(..., max_backoff)`.
- **`LOCK_TTL = 30` is undocumented as a tuning parameter.** For a flash sale service this value is load-dependent and should be configurable at construction time, not a class constant.
- **`RuntimeError` is too broad.** Define a specific `LockAcquisitionError` so callers can distinguish lock failure from programming errors.
