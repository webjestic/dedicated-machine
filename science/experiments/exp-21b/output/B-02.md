Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for production use in a high-concurrency flash sale scenario. The most serious issues are a shared-state concurrency bug that breaks the lock's isolation guarantees, a non-atomic lock release that introduces a TOCTOU race, and a heartbeat design that can silently extend a lock the service no longer owns. The test suite mocks away the heartbeat entirely and does not exercise any real concurrent behavior, so these bugs are invisible to the current test coverage.

---

## Critical Issues

### 1. Shared Heartbeat State Breaks Concurrent Safety

This is the most dangerous flaw in the PR. `_heartbeat_active`, `_heartbeat_thread` are instance-level attributes on `InventoryService`. If the service instance is shared across threads (the normal case for a service object), two concurrent calls to `reserve_item` will race on this shared state.

```python
# Thread A calls reserve_item("item_001") → sets self._heartbeat_active = True
# Thread B calls reserve_item("item_002") → sets self._heartbeat_active = True
# Thread A finishes → calls self._stop_heartbeat() → sets self._heartbeat_active = False
# Thread B's heartbeat thread now silently stops beating, lock for item_002 will expire
# Thread B continues writing, believing it holds the lock — it does not
```

The heartbeat state must be scoped to the lock acquisition, not the service instance. Each acquired lock needs its own independent stop-event.

```python
# Correct pattern: use a per-acquisition threading.Event
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()          # scoped to this acquisition
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=2)    # ensure beat does not fire after release
        # ... atomic release below
```

### 2. Non-Atomic Lock Release (TOCTOU Race)

The lock release is a GET followed by a DELETE. These are two separate Redis round-trips with no atomicity guarantee. Between the GET and the DELETE, the lock can expire and be acquired by another process, which you then delete out from under them.

```python
# Current code — NOT atomic:
current = self.redis.get(lock_key)          # round-trip 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # round-trip 2 — gap here is exploitable
```

This is a well-known problem with a well-known fix: use a Lua script to make the check-and-delete atomic. This is the canonical pattern documented by Redis for safe lock release.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

The same atomicity problem exists in the heartbeat's `get` + `expire` sequence and must be fixed there too.

### 3. Heartbeat Can Extend a Lock the Service No Longer Owns

The heartbeat's token check has the same non-atomic GET + EXPIRE problem. More critically, there is a scenario where the heartbeat extends a lock that was legitimately acquired by a different process after this process's lock expired:

```
t=0:  Process A acquires lock, token="aaa", TTL=30s
t=25: Heartbeat wakes, reads token="aaa" ✓
t=26: Lock expires (clock skew, GC pause, network delay)
t=26: Process B acquires lock, token="bbb"
t=26: Heartbeat calls EXPIRE(lock_key, 30) — extends Process B's lock under "aaa" token
      (EXPIRE does not check value, it just resets TTL on whatever is there)
```

The `expire` call must also be wrapped in a Lua script that validates the token before extending.

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 4. Database Operations Are Not Atomic With Respect to Each Other

The lock protects the read-modify-write cycle against concurrent *lock holders*, but `update_stock` and `create_order` are two separate database calls with no transaction boundary. If the process crashes, is killed, or the DB connection drops between these two calls, stock is decremented but no order is created. This is a silent partial write.

```python
# Current code — not transactional:
self.db.update_stock(item_id, new_stock)    # succeeds
self.db.create_order(order_id, item_id, quantity)  # crashes here → stock gone, no order
```

These must execute within a single database transaction. The exact mechanism depends on the DB layer, but the pattern should be:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 5. `quantity` Has No Input Validation

`quantity` is accepted as a plain `int` with no lower bound check. A caller passing `quantity=0` or `quantity=-5` will silently succeed, creating a zero-quantity or negative-quantity order and potentially *increasing* stock.

```python
# quantity=-1, current_stock=5:
# 5 < -1 is False → passes the stock check
# new_stock = 5 - (-1) = 6 → stock increases
```

Validate at the entry point:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="item_id is required")
    if quantity < 1:
        return ReservationResult(success=False, message="quantity must be at least 1")
```

---

### 6. Retry Loop Blocks the Calling Thread

`time.sleep()` inside `_acquire_lock` blocks the calling thread for up to `0.1 + 0.2 + 0.4 = 0.7s` before giving up. Under flash sale load with hundreds of concurrent requests, this will exhaust the thread pool. For a service at this concurrency level, consider async lock acquisition or at minimum document the blocking behavior and set a hard wall-clock timeout.

## Test Suite Issues

### 7. Heartbeat Is Mocked Away Entirely

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

The most complex and failure-prone component in this PR is never exercised. There are no tests for: heartbeat extending a lock it no longer owns, concurrent calls racing on shared heartbeat state, or heartbeat behavior across the lock release boundary.

### 8. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test sets `redis_client.get.return_value = b"different-token"` and asserts `delete` is not called. This only validates the GET+DELETE path. It does not test the atomic Lua release path (which should replace it), and it does not test what happens if the lock expires mid-operation and is acquired by a concurrent caller.
