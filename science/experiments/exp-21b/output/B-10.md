Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes it unsafe for concurrent use: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to individual lock acquisitions, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on shared heartbeat state and a stopped heartbeat in one call can silently kill the renewal thread for another. Beyond that architectural defect, there is a non-atomic check-then-act race in the lock release path and a TOCTOU window in the heartbeat renewal loop, both of which undermine the correctness guarantees the lock is supposed to provide. The test suite mocks away the heartbeat entirely and does not exercise any real concurrency, so these failure modes are invisible to the existing coverage.

---

## Detailed Findings

### 1. Critical ŌĆö Shared Mutable Heartbeat State (Instance-Level Race)

```python
# Current ŌĆö instance attributes shared across all concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly instantiated once and reused (dependency injection, singleton in a DI container, module-level object). Two concurrent calls to `reserve_item` on the same instance both write to `self._heartbeat_active`. The sequence below is not hypothetical under flash sale load:

```
Thread A: _start_heartbeat ŌåÆ sets self._heartbeat_active = True
Thread B: _start_heartbeat ŌåÆ sets self._heartbeat_active = True (overwrites Thread A's thread ref)
Thread A: _stop_heartbeat  ŌåÆ sets self._heartbeat_active = False  ŌåÉ kills Thread B's heartbeat too
Thread B: lock now expires mid-write because its heartbeat was murdered by Thread A's cleanup
```

The fix is to make heartbeat state local to each lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                        lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # Atomic Lua renewal ŌĆö see finding #2
            renewed = self._renew_lock(lock_key, token)
            if not renewed:
                logger.warning("Heartbeat lost ownership key=%s ŌĆö lock expired or stolen", lock_key)
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

### 2. High ŌĆö Non-Atomic Check-Then-Act on Lock Release and Heartbeat Renewal

Both the release path and the heartbeat loop perform a `GET` followed by a separate `EXPIRE` or `DELETE`. These are two round trips with a gap between them:

```python
# Release ŌĆö current (racy)
current = self.redis.get(lock_key)          # ŌåÉ another process can steal lock here
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ŌåÉ may delete a lock we no longer own

# Heartbeat ŌĆö current (racy)
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ŌåÉ same window
```

Between the `GET` and the `DELETE`/`EXPIRE`, the TTL can expire, another process acquires the lock with a new token, and the original process then deletes or extends a lock it no longer owns. This is the canonical Redlock mistake. Both operations must be atomic via Lua:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

RENEW_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    if result:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock release skipped ŌĆö token mismatch key=%s", lock_key)
    return bool(result)

def _renew_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RENEW_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
    if result:
        logger.debug("Lock TTL extended key=%s", lock_key)
    return bool(result)
```

### 3. High ŌĆö DB Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is OOM-killed, or the DB connection drops between these two calls, stock is decremented but no order exists. The distributed lock provides mutual exclusion but offers zero durability guarantees. These two writes must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If `db` does not expose a transaction context manager, that is a separate gap that needs to be addressed before this service goes to production.

### 4. Medium ŌĆö `quantity` Has No Validated Lower Bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (`10 < 0` is false), creates an order for zero units, and decrements stock by zero. `quantity=-5` creates a negative reservation that *increases* stock. Pydantic is listed on the checklist but is only used for the result model, not for input validation:

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = Field(default=1, ge=1, le=1000)  # enforce reasonable upper bound too
```

### 5. Medium ŌĆö `item_id` Is Not Sanitised Before Use in Lock Key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:` or Redis glob characters, it can produce unexpected key collisions or interfere with key-space notifications and monitoring. Validate or sanitise before interpolation:

```python
if not re.match(r'^[a-zA-Z0-9_\-]{1,128}$', item_id):
    return ReservationResult(success=False, message="Invalid item identifier")
```

### 6. Low ŌĆö Heartbeat `join()` Is Missing on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # no join ŌĆö thread may still be mid-sleep when lock is deleted
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10 s). After `_stop_heartbeat` returns, the `finally` block deletes the lock key. The thread wakes up, finds the token still matches (it hasn't been deleted yet due to a scheduler preemption), and re-extends a key that was supposed to be released. The local-state refactor in finding #1 addresses this via `heartbeat_active.clear()` + `t.join()`, but it is called out explicitly because the current code has no join at all.

### 7. Low ŌĆö Test Suite Mocks Away Everything It Should Be Testing

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ŌåÉ heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ŌåÉ stop never runs
    return svc
```

The tests confirm business logic (stock math, order creation) but provide no coverage of the locking machinery itself. There are no tests for:

- Concurrent acquisition on the same instance (the race in finding #1)
- Heartbeat renewal actually extending the TTL
- Atomic release behaviour when the token has changed between GET and DELETE
- Behaviour when `db.update_stock` raises after `db.get_stock` succeeds (partial failure)

At minimum, add an integration-style test using `fakeredis` that exercises the lock path without mocking it out.
