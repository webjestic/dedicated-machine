Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes the heartbeat mechanism unsafe for concurrent use, and the lock release in the `finally` block contains a non-atomic check-and-delete race that undermines the ownership guarantee the code is trying to enforce. There are also several secondary issues around input validation, DB transaction atomicity, and test fidelity that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Concurrency Corruption

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on the `InventoryService` object.

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the **same service instance** (which is the normal deployment pattern — a singleton service shared across a thread pool), the following sequence is possible:

```
Thread A: acquires lock_A, sets self._heartbeat_active = True, starts heartbeat
Thread B: acquires lock_B, sets self._heartbeat_active = True (no-op, already True)
Thread A: finishes, calls _stop_heartbeat() → self._heartbeat_active = False
Thread B: heartbeat thread for lock_B now exits silently
Thread B: lock_B expires unrenewed while DB writes are still in flight
```

Thread A's cleanup kills Thread B's heartbeat. This is a silent failure — no exception is raised, no log is emitted, and the lock simply expires. Under a flash sale load profile (exactly the scenario this code is designed for), this will produce phantom inventory oversells.

**Fix:** Heartbeat state must be scoped to the lock context, not the instance. Use a local `threading.Event` per acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        # atomic release below
```

### 2. Non-Atomic Lock Release — TOCTOU Race

The `finally` block performs a read-then-delete as two separate Redis commands:

```python
current = self.redis.get(lock_key)          # Step 1: read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # Step 2: delete
```

Between Step 1 and Step 2, the lock TTL can expire and be acquired by another process. You then delete a lock you no longer own, evicting a legitimate holder. This is a well-known failure mode and the Redis documentation explicitly addresses it.

**Fix:** Use the standard Lua script for atomic check-and-delete, which is the correct pattern here:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

The same non-atomic pattern exists in the heartbeat's ownership check:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock may have expired between these two lines
```

The heartbeat renewal should also use a Lua script that atomically checks ownership before extending TTL.

## Significant Issues

### 3. No Input Validation on `reserve_item`

The PR claims input validation via Pydantic, but `reserve_item` accepts raw primitives with no validation. Nothing prevents `quantity=0`, `quantity=-5`, or an empty `item_id` string from reaching the DB layer.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # quantity=-1 passes straight through to db.update_stock
```

A negative quantity would increment stock, which is an inventory integrity violation. Add explicit guards:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

### 4. Non-Atomic DB Operations — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order record exists. The distributed lock only protects against concurrent access — it does not provide atomicity across the DB operations themselves.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the intent must be that both writes commit or neither does.

---

### 5. Heartbeat Sleep-Then-Check Ordering

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # sleeps FIRST
        if self._heartbeat_active:
            ...
```

With `LOCK_TTL=30` and `HEARTBEAT_INTERVAL=10`, the first renewal happens at T+10s. That is fine. But with the `stop_event` fix, the thread should wake promptly on shutdown rather than sleeping a full interval after the lock is already released. Use `stop_event.wait(timeout=HEARTBEAT_INTERVAL)` instead of `time.sleep`, which also eliminates the redundant active check:

```python
def _run_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # atomic extend via Lua script
        self._extend_lock(lock_key, token)
```

## Test Coverage Gaps

### 6. Heartbeat Is Mocked Out Entirely

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # completely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally risky component in this PR and it has zero test coverage. There is no test verifying that:
- The heartbeat actually extends the TTL
- The heartbeat stops when the lock is released
- The heartbeat does not extend a lock it no longer owns

The concurrency corruption described in Issue 1 would not be caught by any test here.

### 7. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test verifies the current broken non-atomic behavior rather than the correct behavior. After the Lua script fix, this test needs to be rewritten to verify the script is called with the correct key and token, not that `delete` is skipped.

## Minor Notes

- `LOCK_TTL = 30` is described as "generous for slow DB operations." If DB operations routinely approach 30 seconds, that is a separate problem that should be addressed at the DB layer, not papered over with a longer lock TTL. Consider documenting an expected p99 DB latency and sizing the TTL relative to that with a documented safety margin.
- No maximum `quantity` bound is enforced. A caller could reserve the entire stock in a single call, which may or may not be intended behavior for a flash sale context.
- Redis connection failures are not distinguished from lock contention. A `redis.exceptions.ConnectionError` will propagate as an unhandled exception rather than being caught and returned as a structured `ReservationResult`.
