Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that breaks process isolation under concurrent load — `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to individual lock acquisitions, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on shared heartbeat state, allowing one call's `_stop_heartbeat()` to silence another call's renewal thread mid-operation. There is also a non-atomic check-then-act race in the heartbeat's `get`/`expire` sequence that can extend a lock the process no longer owns. The test suite mocks out the heartbeat entirely, which means none of the concurrency properties the PR claims to guarantee are actually exercised.

---

## Detailed Findings

### Critical — Instance-Level Heartbeat State (Process Isolation Violation)

This is the primary blocker.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance attributes**, not call-scoped locals. If two threads call `reserve_item` on the same service instance concurrently — which is the entire point of this service — the following sequence is possible:

```
Thread A: acquires lock_A, sets _heartbeat_active = True, starts thread_A
Thread B: acquires lock_B, sets _heartbeat_active = True, starts thread_B
Thread A: finishes, calls _stop_heartbeat() → sets _heartbeat_active = False
Thread B: heartbeat thread_B checks _heartbeat_active → False → exits silently
Thread B: lock_B now expires unrenewed during a slow DB write
Thread B: lock is acquired by Thread C → inventory double-spend
```

The heartbeat mechanism provides a false sense of safety. Under the exact load conditions this PR is designed for, it silently fails. The fix requires each lock acquisition to carry its own isolated cancellation flag:

```python
# Correct pattern — call-scoped state, not instance state
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

Note the `join()` call — without it, the heartbeat thread can extend a lock **after** the context manager has already deleted it, handing a renewed lock to a process that no longer holds it.

### High — Non-Atomic Heartbeat Extension (TOCTOU Race)

```python
# _beat() inner function
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. This is a classic TOCTOU. The extension must be atomic. The standard approach is a Lua script evaluated server-side:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(
    EXTEND_LOCK_SCRIPT, 1, lock_key, token, self.LOCK_TTL
)
if not extended:
    logger.error("Heartbeat lost lock ownership key=%s — aborting operation", lock_key)
    # signal the owning thread to abort
```

The same TOCTOU applies to the release path in `_acquire_lock`'s `finally` block, which has the same `get`/`delete` two-step. That should also be a Lua script or use `redis.delete` with a check script.

---

### High — No Heartbeat Loss Signal to the Owning Thread

Even if the heartbeat detects it no longer owns the lock, it currently does nothing but stop renewing. The DB writes in `reserve_item` continue executing against a lock that has been lost. There is no cancellation path from the heartbeat thread back to the business logic thread. At minimum, a shared `threading.Event` should be used so the owning thread can check `lock_lost_event.is_set()` between the `update_stock` and `create_order` calls.

---

### Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # no join
```

The thread is daemon=True, which prevents it from blocking process shutdown, but it does not prevent it from running past the lock release within the lifetime of the process. A thread that wakes up after `_stop_heartbeat()` but before the `redis.delete()` will attempt an `expire` on a key that is about to be deleted, or worse, on a key that has already been re-acquired by another process. The `join()` with a bounded timeout is required.

---

### Medium — Quantity Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is not validated before the lock is acquired. A caller passing `quantity=0` or `quantity=-5` will acquire the lock, pass the stock check (`current_stock >= 0` is always true for non-negative stock), and write a nonsense order. This should be validated at the entry point, before any I/O:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

The Pydantic checklist item is misleading — `ReservationResult` is an output model. There is no input model validating `item_id` or `quantity`.

---

### Low — Test Suite Does Not Test What It Claims

The fixture patches out the entire heartbeat:

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

This means:

- The concurrency bug described above is not caught
- `test_lock_not_released_if_expired` does not test lock expiry — it tests that `redis.delete` is skipped when the mock returns a different token, which is a trivial mock assertion, not a distributed systems property
- There are zero tests exercising actual thread interaction

Concurrent behavior should be tested with `threading.Barrier` or similar primitives to force interleaving, not by mocking away the mechanism under test.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat state must be call-scoped, not instance-scoped |
| 2 | High | Heartbeat extension and lock release must be atomic (Lua scripts) |
| 3 | High | Heartbeat loss must signal the owning thread to abort |
| 4 | Medium | Heartbeat thread must be joined before lock release |
| 5 | Medium | `quantity` must be validated before lock acquisition |
| 6 | Low | Tests must exercise actual concurrency, not mock it away |
