Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes the heartbeat mechanism unsafe for concurrent use, and the lock release sequence contains a TOCTOU race that undermines the ownership guarantee the code is explicitly trying to enforce. Additionally, the test suite mocks away the heartbeat entirely, meaning the most operationally complex and failure-prone subsystem has zero real coverage. These issues collectively represent correctness and reliability risks that are unacceptable in a high-contention flash sale context.

---

## Detailed Findings

### 1. Critical: Heartbeat State Is Instance-Level — Not Call-Level

**Severity: Critical**
**Location: `InventoryService.__init__`, `_start_heartbeat`, `_stop_heartbeat`**

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance, not scoped to the individual lock acquisition. If `InventoryService` is used as a shared singleton — which is the standard pattern for service objects — two concurrent calls to `reserve_item` will race on the same `_heartbeat_active` flag.

**The failure scenario:**

```
Thread A: acquires lock_A, sets _heartbeat_active = True, starts heartbeat
Thread B: acquires lock_B, sets _heartbeat_active = True (no-op, already True)
Thread A: finishes, calls _stop_heartbeat() → sets _heartbeat_active = False
Thread B: heartbeat for lock_B silently dies — lock_B now expires unguarded
Thread B: DB operations continue, believing the lock is held
```

The heartbeat for Thread B stops the moment Thread A exits. Lock B will expire on its natural TTL, and Thread B's DB writes proceed without the protection the lock was supposed to provide. This is a silent correctness failure — no exception is raised, no log is emitted.

**Fix:** Scope heartbeat state to the lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script (see finding #2)
        self._release_lock(lock_key, token)
```

### 2. High: TOCTOU Race in Lock Release

**Severity: High**
**Location: `_acquire_lock` finally block**

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # ← GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # ← DEL (non-atomic)
```

Between `GET` and `DEL`, the lock TTL can expire and be acquired by another process. The `DELETE` then removes a lock the current process no longer owns. This is the canonical mistake in DIY Redis locking — the Redis documentation explicitly calls it out and prescribes a Lua script as the solution.

**Fix:** Use an atomic Lua script for compare-and-delete:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same non-atomic pattern exists in `_start_heartbeat`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← also non-atomic
```

Apply the same Lua treatment here, or combine the check-and-extend into a single atomic operation.

### 3. High: Tests Mock Away the Only Complex Subsystem

**Severity: High**
**Location: `tests/test_inventory.py`, fixture `service`**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

Every test that uses the `service` fixture exercises zero heartbeat logic. The heartbeat thread is the most operationally risky part of this PR — it involves daemon threads, shared mutable state, and timing dependencies — and it has no test coverage whatsoever. The one test that appears to test lock release behavior (`test_lock_not_released_if_expired`) is also using this fixture, so it is not testing what its docstring claims.

**Required additions:**

- A test that verifies `redis.expire` is called by the heartbeat thread after `HEARTBEAT_INTERVAL` elapses, using `threading.Event` or `unittest.mock.patch` on `time.sleep`.
- A test that verifies the heartbeat stops after `_stop_heartbeat` (or `stop_event.set()` post-fix) is called.
- A test that verifies two concurrent `reserve_item` calls on different items do not interfere with each other's heartbeat lifecycle.
- A test that verifies the Lua release script is called, not a GET+DEL sequence.

### 4. Medium: `db.update_stock` and `db.create_order` Are Not Atomic

**Severity: Medium**
**Location: `reserve_item`**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The distributed lock provides mutual exclusion, but it does not provide atomicity across the two DB writes. This is a partial-write failure mode.

**Recommendation:** Wrap both writes in a database transaction. If the `db` abstraction does not support transactions, that interface contract needs to be defined and enforced here.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 5. Low: Input Validation Is Declared but Not Implemented

**Severity: Low**
**Location: `reserve_item` signature, PR checklist**

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` parameters with no Pydantic model involved. `ReservationResult` is a Pydantic output model, not input validation. There is no guard against `quantity <= 0`, negative values, or empty `item_id` strings.

```python
# Currently possible:
service.reserve_item("", quantity=-5)
```

Either introduce a `ReservationRequest` input model, or add explicit guards at the top of `reserve_item`.

---

### 6. Low: Heartbeat Thread Is Daemon but Never Joined

**Severity: Low**
**Location: `_stop_heartbeat`, `_acquire_lock` finally block**

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Setting `daemon=True` means the thread will be killed on process exit, which is acceptable. However, `_stop_heartbeat` only sets the flag — it does not join the thread. There is a window between `_heartbeat_active = False` and the thread actually stopping where a final `expire` call can still fire after the lock has been released and potentially re-acquired by another process. The fix in Finding #1 (using `threading.Event` and `join`) resolves this as well.
