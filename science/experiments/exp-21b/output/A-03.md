Decision: Request Changes

Summary: The implementation contains several correctness and safety issues that must be addressed before this can be approved. The most critical is a shared-instance heartbeat state design that makes the service unsafe for concurrent use, which is a fundamental contradiction for a component whose entire purpose is managing concurrent access. Additional issues around the lock release atomicity, input validation gaps, and test fidelity also need to be resolved.

---

## Critical Issues

### 1. Shared Heartbeat State — Race Condition Under Concurrency

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not scoped to individual lock acquisitions.

```python
# Current — instance-level state, not per-lock-acquisition state
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the expected production scenario), the following failure sequence is possible:

1. Request A acquires lock, starts heartbeat, sets `self._heartbeat_active = True`
2. Request B acquires a different lock, starts its own heartbeat thread
3. Request A finishes, calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`
4. Request B's heartbeat thread silently dies — its lock is now unguarded and will expire

This defeats the entire purpose of the heartbeat. The fix is to scope heartbeat state to each acquisition, not the instance. A dataclass or a simple closure capturing a local `active` flag and returning a stop callable is the right pattern:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller sets stop_event to stop the thread
```

Note also that `time.sleep(HEARTBEAT_INTERVAL)` followed by an `if self._heartbeat_active` check is not equivalent to `threading.Event.wait()`. The sleep is not interruptible, so `_stop_heartbeat()` can take up to `HEARTBEAT_INTERVAL` seconds to take effect. Use `Event.wait(timeout=...)` instead.

### 2. Non-Atomic Lock Release — TOCTOU Vulnerability

```python
# Current — get and delete are two separate operations
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

There is a time-of-check to time-of-use gap here. Between the `GET` and the `DELETE`, the lock can expire and be acquired by another process. You would then delete a lock you no longer own. This is a well-known failure mode in Redis locking and the canonical fix is a Lua script that makes the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The same TOCTOU issue exists in the heartbeat renewal — `GET` then `EXPIRE` should also be a single atomic script.

### 3. Missing Input Validation

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitive arguments with no validation at the call boundary:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative items would pass the stock check and corrupt inventory)
- `item_id` being an empty string or containing characters that could affect the Redis key namespace (e.g., colons, wildcards)
- `quantity` being unreasonably large

At minimum:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

A Pydantic input model for the method arguments would be the cleaner solution and would actually justify the checklist item.

## Moderate Issues

### 4. DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The distributed lock protects against concurrent access, but it does not protect against partial failure within the critical section. These two operations should be wrapped in a database transaction. The current design has no rollback path.

---

### 5. Heartbeat Can Extend a Lock After the Critical Section Has Logically Ended

The heartbeat is stopped by setting `_heartbeat_active = False`, but as noted above, the thread may still be sleeping. There is a window in the `finally` block where:

1. `_stop_heartbeat()` is called
2. The heartbeat thread wakes up and extends the TTL
3. The lock is then released via `DELETE`

This is a minor issue in practice since the lock is deleted immediately after, but it is an observable correctness gap and the `Event`-based approach eliminates it cleanly.

---

## Test Coverage Gaps

### 6. Heartbeat Is Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the shared-state race condition described above — has zero test coverage. There should be at least one integration-level test that exercises the real heartbeat path, verifying that TTL extension is called and that the thread terminates cleanly after lock release.

### 7. The TOCTOU Release Test Does Not Test the Right Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test validates the current broken behavior (two-step GET + DELETE) rather than the correct atomic behavior. Once the Lua script approach is adopted, this test needs to be rewritten to verify that `redis.eval` is called with the correct script, key, and token, and that it handles both the "owned" and "not owned" return values correctly.

## Minor Notes

- The `retry_attempts=3` default produces backoffs of 0.1s, 0.2s, 0.4s — a maximum wait of 0.7 seconds before failure. For a flash sale scenario this may be too aggressive. Consider making this configurable at construction time rather than hardcoded as a method parameter default.
- `LOCK_TTL = 30` is described as "generous for slow DB operations." If DB operations can actually approach 30 seconds, that is a separate problem that should be addressed at the infrastructure layer rather than accommodated in the lock TTL.
- The `_acquire_lock` context manager yields `token` but `reserve_item` ignores it (`with self._acquire_lock(lock_key):`). This is not a bug, but it is a leaky abstraction. If the token is not needed by callers, do not yield it.
