Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat thread that undermines the lock ownership guarantee the PR claims to provide, and the `InventoryService` instance is structured as a singleton-style stateful object that will corrupt lock state under concurrent requests. Additionally, the database operations lack transactional atomicity, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory without any recovery path. These are not minor issues — they represent fundamental correctness failures in a system designed for high-contention flash sale scenarios.

---

## Detailed Findings

### 🔴 Critical: Heartbeat Race Condition (TOCTOU on Lock Ownership)

```python
# _start_heartbeat internals:
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend ← NOT ATOMIC
```

Between steps (1) and (3), the lock can expire and be acquired by a different process with a different token. The heartbeat will then extend a lock it no longer owns. This is a classic TOCTOU flaw. The correct fix is a Lua script that atomically checks ownership and extends in a single round-trip:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat:
result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same TOCTOU flaw exists in the lock release path in `_acquire_lock`:

```python
# Also not atomic:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # ← another process may own this by now
```

This must also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 🔴 Critical: Instance-Level State Breaks Concurrent Request Isolation

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared mutable state
        self._heartbeat_thread: Optional[...] = None  # shared mutable state
```

`_heartbeat_active` and `_heartbeat_thread` are instance variables. If this service is instantiated once and shared across threads (the standard pattern for a service object), two concurrent calls to `reserve_item` will race on these fields. Request A's `_stop_heartbeat()` will set `_heartbeat_active = False`, killing Request B's heartbeat thread mid-operation.

The fix is to make heartbeat state local to each lock acquisition, not instance-level:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self._extend_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            if result == 0:
                logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return
            logger.debug("Lock TTL extended key=%s", lock_key)

    # ... acquire lock ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

Note also that `_stop_heartbeat` sets a flag but does not `join()` the thread. The heartbeat thread can still be running and extending a lock after the `finally` block has already deleted it, creating a window where a newly acquired lock by another process gets its TTL reset by a zombie thread from the previous holder.

### 🔴 Critical: Non-Atomic Database Operations with No Rollback

```python
self.db.update_stock(item_id, new_stock)   # (1) stock decremented
self.db.create_order(order_id, item_id, quantity)  # (2) order created
```

If the process crashes, is OOM-killed, or the DB connection drops between (1) and (2), stock is decremented with no corresponding order. This is a silent inventory leak. There is no compensation, no saga, and no transaction boundary.

At minimum this needs to be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction doesn't support transactions, that is a separate architectural problem that needs to be resolved before this service ships. An alternative is to use an outbox pattern or optimistic locking at the DB layer, but the current two-step write with no atomicity guarantee is not acceptable for inventory management.

### 🟡 Moderate: Heartbeat Interval vs. TTL Timing Is Unsafe Under Skew

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires every 10 seconds and resets to 30 seconds. Under normal conditions this is fine. However, `time.sleep()` is not precise — under GIL contention, thread scheduling delays, or system load, the actual sleep can exceed the interval. If a heartbeat fires at t=29s (one second before expiry), the lock may expire before the extension lands. The heartbeat interval should be no more than `TTL / 3`, and the implementation should use a shorter interval (e.g., 8s) with explicit logging if the extension fails:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # TTL / ~3.75, with margin
```

More importantly, if the Lua extension script returns 0 (lock lost), the heartbeat must signal the main thread that the lock is gone. Currently there is no such signal path, meaning the main thread will continue executing the protected section without actually holding the lock.

### 🟡 Moderate: Input Validation Is Incomplete

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `quantity > 0`. A caller passing `quantity=0` or `quantity=-5` will silently succeed, creating an order that increases stock. This should be validated at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 🟡 Moderate: Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat logic untested
    svc._stop_heartbeat = MagicMock()    # ← stop logic untested
    return svc
```

The heartbeat — which is the most complex and failure-prone part of this implementation — is entirely mocked out in every test except `test_lock_not_released_if_expired`. The TOCTOU race, the zombie thread issue, and the concurrent instance state corruption are all invisible to this test suite. Tests should exercise the actual heartbeat path with a real (or fakeredis) Redis instance and injected clock control.

### 🟢 Acknowledged Positives

- Lock acquisition uses `SET NX EX` atomically — correct.
- Lock release checks token ownership before deleting — correct intent, wrong implementation (needs Lua).
- Exponential backoff on contention is reasonable.
- Structured logging with contextual fields is good practice.
- `ReservationResult` as a typed return value is clean.
- Daemon thread prevents process hang on unclean shutdown.
