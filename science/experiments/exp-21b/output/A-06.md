Decision: Request Changes

Summary: The implementation contains a critical instance-level concurrency flaw that makes the heartbeat mechanism unsafe for any realistic deployment: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields and corrupt each other's heartbeat state. Beyond that architectural defect, the lock release in the `finally` block is not atomic, creating a TOCTOU window where the lock can be stolen between the `GET` and `DELETE`, and the tests mock away the heartbeat entirely rather than validating its correctness. These issues collectively undermine the safety guarantees the PR claims to provide.

---

### Detailed Findings

#### 1. Critical — Shared Heartbeat State Breaks Under Concurrency

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (the normal case in any web framework with a shared service layer), the following race occurs:

- Thread A acquires lock A, sets `_heartbeat_active = True`, starts heartbeat thread A.
- Thread B acquires lock B, sets `_heartbeat_active = True`, starts heartbeat thread B.
- Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`.
- **Heartbeat thread B is now silently dead.** Lock B will expire while Thread B is still inside the critical section, allowing a third process to acquire the same lock and corrupt inventory.

The heartbeat state must be **call-scoped**, not instance-scoped. Each lock acquisition needs its own independent stop signal. The standard pattern is a `threading.Event` per acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

#### 2. Critical — Non-Atomic Lock Release (TOCTOU)

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # DEL — separate round trip
```

Between `GET` and `DEL`, another process can acquire the lock. This is a well-known failure mode. The correct release is a **Lua script** that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    script = self.redis.register_script(RELEASE_SCRIPT)
    result = script(keys=[lock_key], args=[token])
    return result == 1
```

This is the canonical pattern from the Redis documentation and is non-negotiable for a correct distributed lock.

#### 3. High — Heartbeat Has the Same TOCTOU Problem

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)           # GET
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # EXPIRE — separate round trip
```

The same non-atomic check-then-act pattern appears in the heartbeat. The renewal should also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

If the renewal returns `0`, the heartbeat thread should log a critical alert and signal the caller that the lock was lost — the critical section should abort, not continue writing.

#### 4. High — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative units would corrupt stock)
- `item_id` being an empty string (would produce a lock key of `inventory:lock:`)
- `quantity` being an absurdly large integer

A request model should be introduced:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

#### 5. Medium — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock protects against concurrent access but does not protect against partial failure within the critical section. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

#### 6. Medium — Tests Mock Away the Exact Mechanism Being Validated

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- heartbeat is never exercised
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests provide zero coverage of the heartbeat lifecycle, the stop-event signaling, or the thread join behavior. The test `test_lock_not_released_if_expired` is the only test that exercises real lock release logic, and it only does so incidentally. Integration-level tests using a real or fake Redis (e.g., `fakeredis`) are required to validate the lock protocol end to end.

---

#### 7. Low — Daemon Thread Suppresses Heartbeat Errors Silently

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed without cleanup when the main process exits. Any exception inside `_beat` is silently swallowed. The thread should catch exceptions explicitly and log them at `CRITICAL` level, since a dead heartbeat is a silent correctness failure, not a routine warning.

### Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat state is instance-scoped; must be call-scoped via `threading.Event` |
| 2 | Critical | Lock release is non-atomic GET+DEL; must use Lua CAS script |
| 3 | High | Heartbeat renewal is non-atomic GET+EXPIRE; must use Lua CAS script |
| 4 | High | No input validation on `item_id` or `quantity` |
| 5 | Medium | DB writes are not wrapped in a transaction |
| 6 | Medium | Tests mock the heartbeat; no real lock lifecycle coverage |
| 7 | Low | Daemon thread silently drops heartbeat exceptions |
