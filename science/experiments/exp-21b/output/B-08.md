Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes this service unsafe for concurrent use: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, meaning a single `InventoryService` instance cannot safely handle overlapping reservations — a second call will corrupt the heartbeat state of the first. Beyond that architectural defect, there is a non-atomic check-then-delete in the lock release path and a race condition in the heartbeat renewal that together undermine the correctness guarantees the PR claims to provide. The test suite mocks out the heartbeat entirely and does not exercise the concurrency scenarios this service was built for, leaving the most dangerous failure modes completely untested.

---

## Detailed Findings

### 1. CRITICAL — Shared Mutable Heartbeat State (Instance-Level Concurrency Corruption)

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` is a single boolean on the instance. If two threads call `reserve_item` on the same `InventoryService` object concurrently — which is the entire point of this service — the following sequence is possible:

```
Thread A: acquires lock_A, sets _heartbeat_active = True, starts heartbeat
Thread B: acquires lock_B, sets _heartbeat_active = True (no-op, already True)
Thread A: finishes, calls _stop_heartbeat() → sets _heartbeat_active = False
Thread B: heartbeat thread for lock_B now exits silently
Thread B: lock_B expires unrenewed while Thread B is mid-write
```

This is a silent correctness failure. The lock expires, another process acquires it, and two writers are now operating on the same inventory record simultaneously — exactly the race condition this service exists to prevent.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Move it into a local closure or a dedicated context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.is_set():
            time.sleep(self.HEARTBEAT_INTERVAL)
            if heartbeat_active.is_set():
                # renewal logic
                ...

    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # release lock
```

Using `threading.Event` also gives you a clean signal primitive rather than a bare boolean read across threads without a memory barrier.

### 2. HIGH — Non-Atomic Lock Release (TOCTOU on Delete)

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # ← read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # ← write
```

Between the `GET` and the `DELETE`, the lock TTL could expire and be acquired by another process. You then delete a lock you no longer own, evicting a legitimate holder. This is a well-known failure mode in Redis locking and the reason Redlock specifies atomic release.

**Required fix:** Use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register on __init__:
self._release_lock_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_lock_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. HIGH — Heartbeat Renewal Has the Same TOCTOU Problem

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)       # ← read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

The same non-atomic pattern appears here. Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. You then extend a lock you no longer own, silently extending another process's lock or, worse, resurrecting an expired key if Redis timing is unfortunate.

**Required fix:** Use a Lua script for atomic check-and-extend:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 4. MEDIUM — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10 seconds) per cycle. After `_stop_heartbeat()` returns, the thread may still be alive for up to 10 seconds, holding a reference to `lock_key` and `token` from the enclosing scope. In a high-throughput service this accumulates threads. More critically, if the process is shutting down, daemon threads are killed abruptly — but in a long-running service, leaked threads are a resource and observability problem.

**Required fix:** Join the thread with a bounded timeout after clearing the event, as shown in finding #1.

### 5. MEDIUM — DB Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. The item is effectively lost. The distributed lock protects against concurrent writers but does nothing to protect against partial writes within a single holder.

**Required fix:** Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface should expose a transaction context manager, and this PR should define or document that contract.

### 6. LOW — Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is imported and used for the *output* model (`ReservationResult`) only. There is no guard against `quantity <= 0`, empty `item_id`, or excessively large quantities.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 7. LOW — Test Suite Does Not Test What It Claims To

The fixture mocks out the heartbeat entirely:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means no test exercises the actual heartbeat logic, the thread lifecycle, or the lock renewal path. `test_lock_not_released_if_expired` asserts that `delete` is not called when the token mismatches, but because the heartbeat is mocked, it does not test the scenario where the heartbeat itself causes a token mismatch. The concurrency bug in finding #1 is completely invisible to this test suite.

Tests needed at minimum:
- Concurrent calls on the same instance do not corrupt each other's heartbeat state
- Heartbeat renewal uses atomic Lua script and handles the case where the lock was stolen
- Partial DB failure (create_order raises) does not leave stock decremented
