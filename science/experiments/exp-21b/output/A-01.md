Decision: Request Changes

Summary: The implementation contains a critical shared-state concurrency defect that makes the service unsafe under the exact load conditions it is designed for. Beyond that, the heartbeat design has a race window that can corrupt inventory state, and the test suite mocks away the components under the most scrutiny. These issues must be resolved before this code is suitable for a flash-sale workload.

---

## Detailed Findings

### 1. CRITICAL — Instance-Level Heartbeat State Is Not Thread-Safe

This is the most serious defect in the submission.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, not as locals scoped to a single lock acquisition. Under concurrent load, two threads calling `reserve_item` on the **same service instance** will race on these fields.

**Failure scenario:**

```
Thread A: _start_heartbeat() → sets _heartbeat_active = True, spawns thread-A-beat
Thread B: _start_heartbeat() → sets _heartbeat_active = True, spawns thread-B-beat
Thread A: _stop_heartbeat()  → sets _heartbeat_active = False
           ↳ This also kills Thread B's heartbeat, which is still mid-operation.
           Thread B's lock now expires before its DB writes complete.
           A third thread can acquire the same lock and begin its own reservation
           while Thread B is still writing — the lock provides no protection.
```

The fix is to make heartbeat state local to each lock acquisition context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock was lost; stop extending

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Using a `threading.Event` also eliminates the `time.sleep` polling loop, which has its own race: `_heartbeat_active` can be set to `False` between the `while` check and the `time.sleep`, causing a full `HEARTBEAT_INTERVAL`-second delay before the thread notices it should stop.

### 2. HIGH — TOCTOU Race in Lock Release

```python
# In _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The GET and DELETE are two separate round-trips. Between them, the lock can expire and be acquired by another process, which is then evicted by this DELETE. This is a well-known failure mode; the Redis documentation explicitly prescribes a Lua script for atomic check-and-delete:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU problem exists in the heartbeat's check-then-extend sequence and should be addressed with a similar atomic script.

### 3. HIGH — No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation at the call boundary. Pydantic is only used on the **output** model `ReservationResult`.

A caller can pass:

```python
service.reserve_item("../../etc/passwd", quantity=-1)
service.reserve_item("item_001", quantity=0)
service.reserve_item("a" * 10_000, quantity=1)
```

Negative or zero quantities pass the `current_stock < quantity` check and will write a corrupted stock value to the database. Add a Pydantic input model or explicit guards:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 4. MEDIUM — Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. The distributed lock protects against concurrent readers, but it does not protect against a partial write leaving the database in an inconsistent state. These calls must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 5. MEDIUM — Tests Mock Away the Components Under Scrutiny

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()    # ← stop entirely bypassed
    return svc
```

The heartbeat is the most complex and failure-prone component in this PR. Mocking it out means the tests provide no coverage of the very behavior the PR claims to guarantee. The concurrent safety defect in finding #1 would not be caught by any test here.

At minimum, add:
- A test that instantiates two `InventoryService` instances sharing a mock Redis and calls `reserve_item` concurrently via `threading.Thread`, asserting that stock is decremented exactly once.
- A test that verifies the heartbeat thread terminates promptly after `_stop_heartbeat` is called, not after a full sleep interval.
- A test for the lock-expiry-during-write scenario (lock token changes mid-operation).

---

### 6. LOW — Heartbeat Extends a Potentially Lost Lock

In the current heartbeat loop, if `_heartbeat_active` is still `True` but the token no longer matches (lock was lost), the loop silently continues sleeping and checking. It should log a warning and exit immediately, since continuing to hold `_heartbeat_active = True` gives the caller a false sense of lock ownership.

---

### 7. LOW — `retry_attempts` Semantics Are Off-by-One

```python
for attempt in range(retry_attempts):  # 0, 1, 2 for retry_attempts=3
```

The parameter is named `retry_attempts` but the loop includes the first attempt. The docstring and error message say "3 attempts" and the behavior delivers 3 attempts, but the naming implies 3 *retries* after an initial try, which would be 4 total. Rename to `max_attempts` or adjust the loop to `range(1, retry_attempts + 1)` with consistent semantics.
