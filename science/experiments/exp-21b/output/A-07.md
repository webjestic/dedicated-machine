Decision: Request Changes

Summary: The implementation contains a critical instance-level concurrency flaw that makes the heartbeat mechanism unsafe for any realistic deployment: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields and corrupt each other's heartbeat state. Beyond that architectural defect, the lock release in the `finally` block has a TOCTOU race between the `GET` and `DELETE` operations, and the DB writes lack transactional atomicity, leaving the system exposed to partial-failure inventory corruption. The test suite mocks away the heartbeat entirely and does not exercise the concurrency paths that matter most.

---

## Detailed Findings

### 1. Critical — Shared Instance State for Heartbeat (Concurrency Corruption)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are scalar fields on the service instance. If two requests call `reserve_item` concurrently on the same service object — the normal case in any web framework with a shared service layer — the following race is deterministic:

- Request A sets `_heartbeat_active = True`, starts its thread.
- Request B sets `_heartbeat_active = True`, overwrites `_heartbeat_thread`.
- Request A's `finally` calls `_stop_heartbeat()`, setting `_heartbeat_active = False`.
- Request B's heartbeat thread exits immediately. Request B's lock now expires unguarded.
- Request A's reference to `_heartbeat_thread` is now Request B's thread; joining or inspecting it is meaningless.

The heartbeat, which is the stated justification for the entire design, silently stops working under the exact load conditions the feature is built for. This is not a theoretical edge case.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass state through a local closure or a small dataclass:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock already gone, stop beating

    # ... acquire lock ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

Note also that `_stop_heartbeat` does not join the thread. The heartbeat thread can execute one final `expire` call *after* the lock has been deleted and re-acquired by another process, silently extending a lock the current process no longer owns.

### 2. High — TOCTOU Race in Lock Release

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # DEL — not atomic with GET
```

Between the `GET` and the `DEL`, another process can acquire the lock. The correct pattern is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    script(keys=[lock_key], args=[token])
```

This is the canonical pattern from the Redis documentation for safe lock release and should be considered mandatory, not optional.

### 3. High — Non-Atomic DB Writes Inside the Lock

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is killed, or the DB connection drops between them, stock is decremented but no order exists. The distributed lock protects against concurrent reads of stale stock, but it provides zero protection against this partial-write failure. The lock is a coordination primitive, not a substitute for transactional integrity.

**Required fix:** Both writes must execute inside a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the DB layer, but the atomicity requirement is unconditional.

### 4. Medium — Input Validation Is Declared but Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives and performs no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `quantity >= 1`, no length or format constraint on `item_id`, and no guard against `quantity=0` or `quantity=-5`. A caller passing `quantity=0` will succeed and create an order for zero units. A caller passing `quantity=-1` will *increase* stock. Pydantic is imported but used only for the response model.

**Required fix:**

```python
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    return ReservationResult(success=False, message="Invalid item identifier")
```

Or define a proper request model and validate at the boundary.

### 5. Medium — Heartbeat Thread Is Fully Mocked in All Tests

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture bypasses the heartbeat entirely. The test named `test_lock_not_released_if_expired` claims to verify lock expiry behavior but also uses this fixture, so the heartbeat thread that would actually interact with Redis in that scenario is never exercised. No test validates that the heartbeat correctly extends the TTL, correctly stops when the lock is released, or correctly halts when the token no longer matches. The tests pass because they test a stripped version of the class that does not reflect production behavior.

---

### 6. Low — `retry_attempts` Hardcoded at Call Site, Not Configurable

```python
with self._acquire_lock(lock_key):  # retry_attempts defaults to 3, not exposed
```

Flash sale load profiles vary. The retry count and backoff parameters should be configurable at the service level, not buried as a default argument on a private method with no exposure to the caller or to configuration.

---

### 7. Low — No Observability on Partial Failures

There is no metric emission, no alert hook, and no structured log field distinguishing a lock-contention failure from a DB failure from a heartbeat anomaly. For a flash sale service under load, operators need to distinguish these failure modes in real time. At minimum, log fields should include `failure_reason` as a discrete enum value, not a free-form message string.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Heartbeat state must be local to the lock context, not instance fields |
| 2 | High | Lock release must use atomic Lua GET+DEL, not separate GET and DEL calls |
| 3 | High | DB writes must execute inside a transaction |
| 4 | Medium | Input validation must actually be enforced, not just declared |
| 5 | Medium | Tests must exercise the real heartbeat path, not mock it away |
| 6 | Low | Retry and backoff parameters should be service-level configuration |
| 7 | Low | Structured failure-reason logging for operational visibility |
