Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that breaks process isolation under concurrent load — `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on shared heartbeat state, allowing one call's `_stop_heartbeat()` to silently kill another call's renewal thread. There is also a non-atomic check-then-act race in the lock release path and a meaningful TOCTOU window in the heartbeat renewal logic, both of which undermine the correctness guarantees the lock is supposed to provide. The test suite mocks out the heartbeat entirely and does not exercise any real concurrency, so these failure modes are invisible to the current test coverage.

---

## Detailed Findings

### 1. CRITICAL — Shared Heartbeat State Breaks Concurrent Isolation

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** attributes. A single `InventoryService` instance is almost certainly shared across requests (injected as a singleton, stored on an app object, etc.). When two requests arrive concurrently:

```
Thread A: _heartbeat_active = True   (lock acquired for item_001)
Thread B: _heartbeat_active = True   (lock acquired for item_002)
Thread A: _stop_heartbeat()          → sets _heartbeat_active = False
Thread B: heartbeat thread checks _heartbeat_active → False → exits silently
          Thread B's lock for item_002 now expires mid-operation
```

Thread B's lock expires while it is between `update_stock` and `create_order`. Another process acquires the lock, reads stale stock, and double-sells the item. The distributed lock has provided a false safety guarantee.

The fix is to scope all heartbeat state to the lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread closure, never touch self
```

### 2. HIGH — Non-Atomic Lock Release (TOCTOU)

```python
# In the finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

There is a race between `GET` and `DELETE`. After the `GET` confirms ownership, the lock TTL could expire and be acquired by another process before `DELETE` executes. You then delete a lock you no longer own, evicting a legitimate holder mid-operation.

This is a well-known problem with a well-known fix — use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def __init__(self, ...):
    self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same atomicity problem exists in the heartbeat renewal — `GET` then `EXPIRE` is also non-atomic. Use a Lua script there too, or use `SET key value XX EX ttl` if the token is embedded in the value.

---

### 3. HIGH — Heartbeat Interval Allows Near-Expiry Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat sleeps *first*, then renews. In the worst case, renewal happens at T+10, T+20, T+30 — but the lock expires at T+30. Under GC pause, thread scheduling jitter, or a slow Redis round-trip, the renewal at T+30 races with expiry. The interval should be a fraction of the TTL with meaningful margin, and the thread should renew *before* sleeping, or use a tighter ratio:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # renew at ~27% remaining TTL, not 33%
```

More importantly, the heartbeat thread is a `daemon=True` thread. If the main process crashes mid-operation, the daemon thread dies with it, the lock expires naturally — that part is fine. But if the process is killed with SIGKILL between `update_stock` and `create_order`, neither the lock release nor the order creation completes. The PR description does not address this failure mode, and there is no compensating transaction or saga pattern visible in the diff.

---

### 4. MEDIUM — `quantity` Has No Validation Guard

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is typed as `int` but nothing prevents `quantity=0` or `quantity=-5`. A caller passing `quantity=-1` would pass the stock check (`current_stock < -1` is false for any non-negative stock), decrement to `current_stock + 1`, and silently *increase* inventory under the lock. Input validation is checked off in the PR checklist, but this case is not covered.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

### 5. MEDIUM — Test Suite Does Not Test What It Claims

The fixture patches out the heartbeat entirely:

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

`test_lock_not_released_if_expired` is described as testing that "lock release is skipped if token no longer matches," but it calls `service.reserve_item` through the full context manager with the heartbeat mocked — the token comparison in the `finally` block uses a freshly generated `uuid4()`, not `"test-token"`. The mock returns `b"different-token"`, so `delete` is not called, but this is testing the mock's behavior, not a realistic expiry scenario. No test exercises concurrent access, heartbeat renewal, or the Lua-script-less release race.
