Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes the heartbeat mechanism unsafe under concurrent use, a race condition in the lock release path that can cause silent data corruption, and a non-atomic heartbeat check-then-extend sequence that undermines the ownership guarantee the code claims to provide. These are not minor issues — in a flash sale context with high concurrency, they represent realistic paths to overselling inventory or corrupting stock counts.

---

## Critical Issues

### 1. Shared Heartbeat State Breaks Concurrent Correctness

```python
# InventoryService instance fields — shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` on the same `InventoryService` instance concurrently — which is the entire point of this service — the second caller's `_start_heartbeat` will overwrite `_heartbeat_thread`, and either caller's `_stop_heartbeat` will set `_heartbeat_active = False`, silently killing the other caller's heartbeat. The lock then expires mid-operation with no signal to the executing thread, and another process can acquire it while the first is still writing to the database.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the service instance. Use a local `threading.Event` per lock context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # then release lock
```

### 2. Non-Atomic Heartbeat Extension Is a TOCTOU Race

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)          # read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

The `GET` and `EXPIRE` are two separate round trips. Between them, the lock can expire and be acquired by another process. You then call `EXPIRE` on a key you no longer own, extending a lock held by a different process. This is exactly the failure mode the heartbeat is supposed to prevent, and it makes it worse.

**Required fix:** Use a Lua script to make the check-and-extend atomic:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(
    EXTEND_LOCK_SCRIPT, 1, lock_key, token, self.LOCK_TTL
)
if not extended:
    logger.error("Heartbeat lost lock ownership key=%s — aborting is required", lock_key)
    stop_event.set()  # signal the protected section to abort
```

Note that detecting this condition is not enough — the code holding the lock must be notified and must abort. There is currently no mechanism for that.

### 3. Lock Release Has the Same Non-Atomic Race

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)           # read
    if current and current.decode() == token:
        self.redis.delete(lock_key)              # write
```

Same TOCTOU problem as the heartbeat. Between `GET` and `DELETE`, the key can expire and be re-acquired by another holder. You then delete their lock. This is a well-known failure mode with a well-known fix.

**Required fix:** Use the canonical atomic release script from the Redis documentation:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""

released = self.redis.eval(RELEASE_LOCK_SCRIPT, 1, lock_key, token)
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen at release key=%s", lock_key)
```

## Significant Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is no guard against `quantity <= 0`, negative values, empty `item_id`, or excessively large quantities. A caller passing `quantity=0` produces a successful reservation of zero units. A caller passing `quantity=-5` increases stock.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
if quantity > MAX_RESERVATION_QUANTITY:
    return ReservationResult(success=False, message="Quantity exceeds limit")
```

### 5. Database Exceptions Are Not Caught Inside the Lock

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)
    # ...
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order exists. The lock is released, the inconsistency is permanent, and the exception propagates as an unhandled error to the caller. There is no transaction boundary and no compensating write.

At minimum, database exceptions must be caught and logged with full context. Ideally, `update_stock` and `create_order` execute within a database transaction so partial failure rolls back atomically.

---

### 6. Heartbeat Thread Is Not Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to observe it. The thread may still be mid-sleep when the lock is released and deleted. It then wakes up, checks the flag, finds it false, and exits — but during that window, the lock is gone and the thread holds a stale reference. With the Lua-based extension fix this becomes less dangerous, but the thread should still be joined before the lock is released to ensure clean shutdown ordering.

## Test Coverage Gaps

### 7. Heartbeat and Lua Scripts Are Not Tested

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic has zero test coverage. The atomic Lua scripts, once added, must also be tested with a real Redis instance or a mock that validates the script semantics.

### 8. Concurrent Access Is Not Tested

There are no tests exercising two threads calling `reserve_item` simultaneously on the same instance. Given that instance-level state is the root cause of Issue 1, this is the most important missing test case.

```python
def test_concurrent_reservations_do_not_corrupt_state():
    # Two threads must each maintain independent heartbeat state
    # and must not cancel each other's locks
    ...
```

The lock lifecycle correctness problems (1–3) must be resolved before this merges. The Redis documentation explicitly covers the atomic release pattern; the fact that it was not applied here, and that the same non-atomic pattern was replicated in the heartbeat, suggests the concurrency model was not reviewed against known failure modes before submission.
