Decision: Request Changes

Summary: The implementation demonstrates solid distributed systems thinking — the token-based lock ownership check, heartbeat renewal, and exponential backoff are all directionally correct. However, there are several correctness, safety, and concurrency bugs that would cause real failures under production load, particularly in a high-contention flash sale scenario. These need to be resolved before this can be approved.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Not Thread-Safe for Concurrent Calls

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If a single `InventoryService` instance handles concurrent requests (which it will — this is a service class), two simultaneous calls to `reserve_item` will race on these fields. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass a mutable container or use a local `threading.Event`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        # ... release lock
```

### 2. TOCTOU Race in Lock Release

```python
# Current — two separate Redis operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock. You would then delete a lock you no longer own. This is a well-known distributed lock pitfall. The release **must** be atomic.

**Fix:** Use a Lua script for atomic check-and-delete, which is the standard Redis pattern:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. TOCTOU Race in Heartbeat Renewal

```python
# Current — same GET/EXPIRE pattern in the heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Same problem. Between `GET` and `EXPIRE`, the lock could expire and be acquired by another process. You would then extend a lock you no longer own.

**Fix:** Use a Lua script here as well:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 4. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order record exists. The item is silently lost. The Redis lock provides no protection against application-layer partial failure.

**Fix:** These two operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. If the underlying DB layer doesn't support transactions, that is a separate architectural problem that needs to be surfaced now, not discovered in production.

## Moderate Issues

### 5. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but there is none on the public method signature. `item_id` accepts any string, including empty string, path traversal fragments, or a string long enough to cause Redis key bloat. `quantity` accepts zero or negative integers.

```python
# These will currently succeed silently or behave incorrectly
service.reserve_item("", quantity=0)
service.reserve_item("item_001", quantity=-5)
```

**Fix:** Validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

---

### 6. Heartbeat Thread Timing Creates a Reliability Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10s, 20s, 30s... but the lock expires at exactly 30s. If the heartbeat at t=20s is delayed by even a few milliseconds of thread scheduling jitter, the lock could expire before the t=30s renewal fires. The interval should be materially less than half the TTL.

**Fix:** A conservative rule is `HEARTBEAT_INTERVAL = LOCK_TTL // 3`, giving two renewal opportunities before expiry.

---

### 7. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` will also catch unexpected programming errors from inside the lock body (e.g., a `RuntimeError` raised by a misbehaving DB driver). These should propagate, not be silently swallowed as "service unavailable."

**Fix:** Define a specific exception type:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is mocked in every test, including the fixture. This means the actual heartbeat logic — the most operationally risky part of this PR — has zero test coverage. The TOCTOU bugs in the heartbeat and lock release would not be caught by this suite.

At minimum, add:
- A test that verifies the heartbeat calls `expire` only when the token still matches
- A test that verifies the heartbeat does **not** call `expire` when the token has changed
- A test that verifies the heartbeat stops after `_stop_heartbeat` is called

### 9. Lock Release Behavior Under DB Exception Is Untested

There is no test confirming that if `db.update_stock` raises, the lock is still released in the `finally` block. This is the most important invariant of the entire design.
