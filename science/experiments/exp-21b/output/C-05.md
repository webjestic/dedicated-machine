Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic `SET NX EX`, token-verified release, heartbeat renewal, and structured logging are all correct in principle. However, there are several architectural and correctness flaws that must be addressed before this can be merged into a production inventory system, particularly given the high-concurrency flash-sale context. The issues range from a critical race condition in the heartbeat to missing input validation that the checklist incorrectly claims is present.

---

## Critical Issues

### 1. Heartbeat Race Condition — `get` + `expire` Is Not Atomic

```python
# services/inventory.py — _start_heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← window between get and expire
```

Between the `GET` and the `EXPIRE` call, another process could acquire the lock (if the TTL expired in that window), and you would silently extend *their* lock. This is the exact class of bug Redlock was designed to address. The fix is a Lua script that performs the check-and-extend atomically:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In _beat:
extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat: lock lost key=%s — stopping heartbeat", lock_key)
    self._heartbeat_active = False
    break
```

The same atomic-script pattern already used by `redis-py`'s own `Lock` implementation should be followed here.

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped. If `InventoryService` is used as a singleton (the normal pattern with DI containers), two concurrent calls to `reserve_item` will share `_heartbeat_active`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat thread prematurely, or vice versa.

The heartbeat state must be scoped to the lock acquisition, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False
    # ... retry logic ...

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Lock lost during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                break

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release
        ...
```

### 3. Lock Release Is Also Not Atomic

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # ← same GET/DEL race
```

Same class of bug as the heartbeat. Use the canonical Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

This is well-documented in the Redis documentation and in `redis-py`'s own `Lock` class. There is no excuse for a non-atomic release in 2024.

## Significant Issues

### 4. No Input Validation — Checklist Item Is Incorrect

The checklist claims "Input validation via Pydantic" is complete, but `reserve_item` performs zero validation on its inputs:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # item_id is used directly in a Redis key with no sanitization
    lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "../../admin"` or `item_id = ""` or `quantity = -5` or `quantity = 0` will produce silent misbehavior or, in the key injection case, a potential lock bypass. Add a validated input model:

```python
from pydantic import BaseModel, Field, field_validator
import re

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128)
    quantity: int = Field(ge=1, le=1000)

    @field_validator("item_id")
    @classmethod
    def item_id_safe(cls, v: str) -> str:
        if not re.match(r'^[a-zA-Z0-9_\-]+$', v):
            raise ValueError("item_id contains invalid characters")
        return v
```

### 5. Database Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record — a silent inventory leak. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction needs to expose a transaction context manager, and the tests need to verify rollback behavior on partial failure.

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The daemon thread will continue executing for up to `HEARTBEAT_INTERVAL` seconds after the lock is released. In a high-throughput flash sale, this means hundreds of ghost threads sleeping and waking. Always join with a timeout:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread and self._heartbeat_thread.is_alive():
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

## Minor Issues

### 7. Test Coverage Gaps

- No test for the partial DB failure scenario (stock updated, order creation fails).
- No test verifying that `_start_heartbeat` and `_stop_heartbeat` are actually called in the correct order — the fixture mocks them out entirely, which means the heartbeat logic has zero test coverage.
- No test for invalid `item_id` or non-positive `quantity` once validation is added.
- `test_lock_not_released_if_expired` passes `b"different-token"` but the token comparison uses `uuid4` internally — the test is only correct by coincidence of mock setup, not by design.

### 8. `time.sleep` in Lock Retry Blocks the Calling Thread

Under a flash sale load with hundreds of concurrent requests, threads blocking in `time.sleep` during backoff will exhaust the thread pool. Consider whether this service should be async (`asyncio` + `aioredis`) or whether the retry should be offloaded. At minimum, document the threading model assumption.

### 9. `LOCK_TTL = 30` Is Generous to a Fault

Thirty seconds is an unusually long TTL for a lock protecting a stock read and two DB writes. If a process dies mid-operation, inventory is locked for up to 30 seconds during a flash sale. Profile your DB operation latency and set this to a value with a documented rationale, not a comment saying "generous."
