Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and concurrency issues that must be resolved before this handles real flash sale load in a government-grade system. The most critical problems are a race condition in the heartbeat's check-then-extend sequence, an instance-level heartbeat state design that breaks under concurrent use, and a missing atomicity guarantee on the Redis lock release.

---

## Critical Issues

### 1. Heartbeat Has a TOCTOU Race Condition

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You are then extending a lock you no longer own. This must be a single atomic operation using a Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event
```

### 2. Instance-Level Heartbeat State Breaks Under Concurrency

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), they will overwrite each other's `_heartbeat_active` flag. One thread stopping its heartbeat will kill the other's. The fix shown above — returning a `threading.Event` per call — scopes the stop signal to the individual lock acquisition and eliminates the shared state entirely.

```python
# In _acquire_lock:
stop_event = self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    stop_event.set()
    # ... atomic release below
```

### 3. Lock Release Is Not Atomic

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same TOCTOU problem as the heartbeat. Use a Lua script for atomic check-and-delete:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

This is the canonical pattern for safe distributed lock release and is non-negotiable for correctness.

## Significant Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will proceed to acquire a lock and query the database.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

### 5. `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # if this raises, stock is already decremented
```

If `create_order` fails after `update_stock` succeeds, you have decremented stock with no corresponding order record. This is a silent inventory leak. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager. If it does not, that is a separate PR requirement that must be satisfied before this merges.

---

### 6. Bare `Exception` Is Not Caught — But Swallowed State Is Still a Risk

The `except RuntimeError` block is appropriately scoped. However, if `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` raises any non-`RuntimeError` exception (e.g., a database connectivity error), it will propagate out of the `with self._acquire_lock` block. The `finally` clause will correctly release the lock, which is good. But the caller receives an unhandled exception rather than a `ReservationResult`. For a service boundary, consider catching broad exceptions at the top level and returning a structured failure result, while still logging the full traceback:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Minor Issues

### 7. Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat logic never tested
    svc._stop_heartbeat = MagicMock()   # stop logic never tested
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the heartbeat and lock lifecycle are never exercised by the test suite. The lock release path in `_acquire_lock`'s `finally` block is also bypassed because the mock Redis `get` returns a hardcoded `b"test-token"` that will never match the dynamically generated `uuid4()` token. Add integration-style tests using `fakeredis` to exercise the actual lock acquisition, renewal, and release paths.

### 8. `HEARTBEAT_INTERVAL` Is Too Close to `LOCK_TTL`

With `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10`, the heartbeat fires at 10s, 20s, and 30s. At exactly 30s the lock expires before the third heartbeat can fire, depending on scheduling jitter. A safer ratio is `HEARTBEAT_INTERVAL = LOCK_TTL / 3` or lower, and the heartbeat should log a warning and abort if it detects it has lost ownership (as shown in the Lua script fix above).
