Decision: Request Changes

Summary: The implementation contains a critical instance-level state management flaw that makes it unsafe for concurrent use: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on shared heartbeat state. Beyond that architectural defect, there is a non-atomic check-then-act in the lock release path and a TOCTOU window in the heartbeat renewal that could allow a stolen or expired lock to be silently extended. The test suite mocks out the heartbeat entirely, so none of these concurrency failure modes are exercised.

---

## Detailed Findings

### Critical — Shared Mutable Heartbeat State (Concurrency Correctness)

This is the most serious issue. `_heartbeat_active` and `_heartbeat_thread` are instance-level attributes:

```python
# As written — shared across all concurrent calls on this instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (the normal case for any web framework with a shared service object), the following race is deterministic:

1. Thread A acquires lock A, sets `self._heartbeat_active = True`, starts heartbeat.
2. Thread B acquires lock B, sets `self._heartbeat_active = True`, starts a second heartbeat thread.
3. Thread A finishes, calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`.
4. Thread B's heartbeat loop now exits immediately on its next iteration — lock B goes unrenewed and expires under load, exactly the failure mode this code was written to prevent.
5. The inverse is equally possible: Thread B's stop call leaves Thread A's heartbeat running after lock A is released, causing spurious TTL extensions on a key the process no longer owns.

The fix is to scope heartbeat state to each lock acquisition, not to the instance. A `threading.Event` passed by value into the closure is the idiomatic approach:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release via Lua script (see below)
        self._release_lock(lock_key, token)
```

```python
def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning(
                    "Heartbeat: lock no longer owned, stopping key=%s", lock_key
                )
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### High — Non-Atomic Lock Release (TOCTOU)

The release path performs a GET followed by a separate DELETE:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the GET and the DELETE, the lock TTL can expire and another process can acquire it. The DELETE then evicts the new owner's lock. This is a well-known failure mode documented in the Redis SETNX pattern literature. The canonical fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock release skipped — not owner or already expired key=%s", lock_key
        )
```

The same TOCTOU applies in the heartbeat's GET-then-EXPIRE sequence. Consider using a Lua script there as well, or accept that a spurious extend on a stolen lock is a lower-severity outcome than a spurious delete.

### Medium — Heartbeat Uses `time.sleep` Instead of `threading.Event.wait`

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for full interval
        if self._heartbeat_active:
            ...
```

`time.sleep` is not interruptible. When `_stop_heartbeat` is called, the thread sleeps for up to `HEARTBEAT_INTERVAL` (10 seconds) before it checks the flag and exits. During a graceful shutdown or rapid lock turnover this creates a 10-second tail on every lock release. `threading.Event.wait(timeout=...)` returns immediately when the event is set, which is the correct primitive here (shown in the fix above).

### Medium — No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- Empty or whitespace-only `item_id` (which would produce a valid but meaningless lock key `inventory:lock:`)
- `quantity <= 0` (reserving zero or negative units would succeed and corrupt stock)
- Excessively long `item_id` strings used as Redis keys

Pydantic is imported but only used for the response model. Add a request model or explicit guards:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="item_id must not be empty")
if quantity <= 0:
    return ReservationResult(success=False, message="quantity must be a positive integer")
```

### Low — Tests Mock Out the Entire Heartbeat

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # completely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture exercises zero heartbeat logic. The critical concurrency bug described above would not be caught by this suite even after the fix is applied. Add at minimum:

1. A test that calls `reserve_item` from two threads simultaneously on the same instance and asserts both operations complete with correct stock accounting.
2. A test that verifies the heartbeat thread exits promptly (within `HEARTBEAT_INTERVAL + 1s`) after lock release.
3. A test that verifies the Lua release script does not delete a key owned by a different token.

---

### Low — `db.update_stock` and `db.create_order` Are Not Transactional

This is an architectural observation rather than a blocking issue, but it is worth flagging. If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. The distributed lock protects against concurrent reservation races but does not protect against partial writes within a single acquisition. If the backing database supports transactions, these two writes should be wrapped in one. If it does not, a compensating rollback path is needed.
