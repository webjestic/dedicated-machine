Decision: Request Changes

Summary: The implementation contains a critical instance-level concurrency flaw that makes it unsafe for any real concurrent workload: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those shared fields — one call's `_stop_heartbeat()` will silence another call's heartbeat thread, leaving a live lock unrenewed and vulnerable to expiry mid-write. Beyond that architectural defect, there are several additional correctness and safety issues that must be resolved before this can be approved.

---

## Detailed Findings

### 1. Critical — Heartbeat State is Instance-Scoped, Not Lock-Scoped

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or a long-lived service object. When two threads call `reserve_item` concurrently:

- Thread A acquires its lock, starts its heartbeat, sets `self._heartbeat_active = True`
- Thread B acquires a different lock, calls `_start_heartbeat`, overwrites `self._heartbeat_thread`
- Thread A's `finally` block calls `_stop_heartbeat()`, sets `self._heartbeat_active = False`
- Thread B's heartbeat is now dead, its lock will expire, and the DB writes it is protecting may execute against a lock that another process has since acquired

The fix is to make heartbeat state local to each lock acquisition and pass it explicitly:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    stop_event = threading.Event()  # scoped to this acquisition

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 5.0)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=2)
        # atomic check-and-delete via Lua
        self._release_lock(lock_key, token)
```

### 2. High — Lock Release is Not Atomic (TOCTOU Race)

```python
# Non-atomic: two round trips, exploitable window between them
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock can expire and be acquired by another process. The `DELETE` then removes a lock you no longer own. This is a well-known failure mode and the canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

The same TOCTOU problem exists in the heartbeat's renewal check:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock may have expired between these two calls
```

The heartbeat renewal should also use a Lua script or `SET ... XX EX` with value comparison.

### 3. High — `item_id` and `quantity` Are Not Validated Before Use

The method signature accepts raw inputs but performs no validation before they reach the lock key construction and database calls:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"  # item_id is unsanitized here
```

A caller passing `item_id = "foo:bar"` or `item_id = ""` or `quantity = -5` will produce either a malformed lock key, a nonsensical reservation, or silent negative-stock corruption. The checklist claims Pydantic validation is present, but it is not applied to inputs — `ReservationResult` is an output model only.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    request = ReservationRequest(item_id=item_id, quantity=quantity)
    lock_key = f"inventory:lock:{request.item_id}"
```

### 4. Medium — `db.update_stock` and `db.create_order` Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access, it does not provide transactionality. These two operations must either execute within a single DB transaction or the failure path must explicitly compensate:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB transaction failed, reservation rolled back item_id=%s", item_id)
    raise
```

### 5. Medium — Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined — it may still be running when the lock is deleted
```

After `_stop_heartbeat()` returns, the heartbeat thread may be mid-sleep and will wake up, re-check `_heartbeat_active`, and exit — but the lock may have already been deleted by the `finally` block. In the worst case the thread wakes, sees the flag is still True from a race, and extends a lock that now belongs to someone else. Using a `threading.Event` and calling `thread.join()` in the `finally` block eliminates this window entirely.

---

### 6. Medium — Backoff Has No Cap

```python
backoff = (2 ** attempt) * 0.1
```

With `retry_attempts` left as a caller-controlled parameter and no upper bound on backoff, a high retry count produces unbounded wait times. Cap it:

```python
backoff = min((2 ** attempt) * 0.1, 5.0)
```

---

### 7. Low — Tests Mock Away the Exact Behaviors Being Tested

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # stop never runs
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the lock lifecycle at all. The test for "lock not released if expired" (`test_lock_not_released_if_expired`) is particularly misleading — it passes only because the mock bypasses the actual release logic. Integration-level tests using a real or embedded Redis (e.g., `fakeredis`) are necessary to validate the lock acquire/release/heartbeat path.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat state is instance-scoped; concurrent calls corrupt each other's heartbeat |
| 2 | High | Lock release and heartbeat renewal are non-atomic (TOCTOU); use Lua scripts |
| 3 | High | Input validation is claimed but not applied to `item_id` or `quantity` |
| 4 | Medium | Stock decrement and order creation are not transactional; no rollback on partial failure |
| 5 | Medium | Heartbeat thread is never joined; creates a race window after lock release |
| 6 | Medium | Exponential backoff has no ceiling |
| 7 | Low | Tests mock the lock lifecycle they are supposed to verify |
