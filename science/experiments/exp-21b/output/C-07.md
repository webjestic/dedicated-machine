Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architectural issues that must be addressed before this handles real flash sale traffic, particularly around race conditions in the heartbeat, instance-level state that breaks concurrent use, and a missing atomicity guarantee on lock release.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrent Requests

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored on `self`, meaning a single `InventoryService` instance cannot safely handle two concurrent calls to `reserve_item`.

```python
# If two threads call reserve_item simultaneously on the same instance:
self._heartbeat_active = True   # Thread A sets this
# ...Thread B also sets this...
self._stop_heartbeat()          # Thread A sets it False
# Thread B's heartbeat is now silently dead, but Thread B doesn't know
```

**Fix:** Move heartbeat state into the `_acquire_lock` context manager as local closure variables, scoped to each lock acquisition.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

### 2. Lock Release Is Not Atomic (TOCTOU Race)

The release sequence is a classic check-then-act race:

```python
current = self.redis.get(lock_key)          # Step 1: read
if current and current.decode() == token:   # Step 2: compare
    self.redis.delete(lock_key)             # Step 3: delete
```

Between steps 1 and 3, the lock can expire and be acquired by another process. You would then delete a lock you no longer own, corrupting another thread's critical section.

**Fix:** Use a Lua script for atomic compare-and-delete. This is the standard pattern:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

---

### 3. Heartbeat Race on Shutdown

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps 10s
        if self._heartbeat_active:            # re-checks after sleep
            self.redis.expire(...)
```

`_stop_heartbeat()` sets the flag to `False`, but the thread may be mid-sleep for up to 10 seconds. The double-check of `_heartbeat_active` does not help here — the thread is sleeping, not checking the flag. Using `threading.Event` with `wait(timeout=...)` as shown above resolves this cleanly and allows near-immediate shutdown.

## Moderate Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is no guard against:

- Empty or whitespace-only `item_id` (which would produce a valid but meaningless lock key)
- `quantity <= 0` (which would silently "succeed" reserving zero or negative items)
- Excessively long `item_id` strings used as Redis keys

```python
# Minimum required guards:
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

Consider a `ReservationRequest` Pydantic model as the actual entry point.

---

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against this — it only serializes access. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a broad base class. A `RuntimeError` from an unrelated internal failure (e.g., inside `db.update_stock`) would be silently swallowed and returned as "temporarily unavailable" with no stack trace logged. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

And log unexpected exceptions at `ERROR` with full traceback before re-raising or returning a distinct error.

## Minor Issues

### 7. Test Coverage Gaps

- **No test for the heartbeat thread behavior** — the heartbeat is mocked out in every test, so the actual `_start_heartbeat` / `_stop_heartbeat` logic has zero coverage.
- **No test for the DB transaction failure path** (once that is added).
- **`test_lock_not_released_if_expired`** tests the right behavior but relies on `redis_client.get` returning a hardcoded `b"different-token"` rather than simulating actual token mismatch logic. Once the Lua script is in place, this test needs to be rewritten against the `eval` call.
- **No test for invalid inputs** (`quantity=0`, empty `item_id`).

### 8. `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined
```

The `finally` block in `_acquire_lock` exits without waiting for the heartbeat thread to terminate. In a high-throughput scenario this can accumulate zombie threads. Always `join()` with a bounded timeout.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Instance-level heartbeat state is not concurrency-safe |
| 2 | Critical | Lock release is non-atomic (TOCTOU); use Lua eval |
| 3 | Critical | Heartbeat thread not stoppable promptly; use `threading.Event` |
| 4 | Moderate | No input validation on `reserve_item` parameters |
| 5 | Moderate | DB operations not wrapped in a transaction |
| 6 | Moderate | Overly broad `RuntimeError` catch masks real failures |
| 7 | Minor | Heartbeat logic has no test coverage |
| 8 | Minor | Heartbeat thread not joined on shutdown |

The core design intent is sound. Address the three critical items at minimum — particularly the instance-state concurrency issue and the non-atomic release — before this goes anywhere near a flash sale workload.
