Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several critical correctness and safety flaws that would cause data races, silent failures, and resource leaks under real concurrent load — exactly the conditions this service is designed to handle. The test suite mocks out the heartbeat entirely and does not exercise the actual lock release path, so these bugs are invisible to the current coverage. These issues must be resolved before this can be merged.

---

## Detailed Review

### 🔴 Critical — Correctness / Safety

#### 1. Instance-level heartbeat state is not thread-safe

```python
# InventoryService is shared across callers, but these are instance variables:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the **same** `InventoryService` instance, the second call overwrites `_heartbeat_active` and `_heartbeat_thread`. This means:

- The first reservation's heartbeat is silently killed mid-operation.
- The first lock can expire while the DB write is still in flight.
- A third party can steal the lock and decrement stock a second time — **oversell**.

**Fix:** Move heartbeat state into a local scope per call, or use a per-call object. A clean pattern is a small `_LockContext` dataclass passed into the heartbeat closure:

```python
# Per-acquisition state, not shared across calls
active_flag = threading.Event()
active_flag.set()

def _beat():
    while active_flag.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...

# To stop:
active_flag.clear()
```

---

#### 2. Heartbeat check-then-extend is a TOCTOU race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. The heartbeat then extends a lock it no longer owns, silently prolonging an adversary's hold.

**Fix:** Use a Lua script to make the check-and-extend atomic:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the release path in `_acquire_lock` (already a known best practice — worth calling out explicitly in the code with a comment referencing why).

---

#### 3. Lock release in `finally` block has the same TOCTOU race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same GET-then-DELETE window. Use the standard Lua release script here as well.

---

#### 4. `_stop_heartbeat` does not join the thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The heartbeat thread is still alive and sleeping when `_stop_heartbeat` returns. The `finally` block then immediately executes the lock release. If the thread wakes up between `_stop_heartbeat` and `redis.delete`, it will call `redis.expire` on a key that is about to be deleted — or on a key that has already been re-acquired by another caller.

**Fix:** Signal the thread and join it before releasing the lock:

```python
active_flag.clear()
self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

#### 5. Negative quantity is not validated

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 10 is False), creates a real order record, and decrements stock by zero — a phantom order. `quantity=-5` would **increase** stock. Neither is caught.

**Fix:**

```python
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

#### 6. `item_id` is not sanitized before use as a Redis key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:`, spaces, or newline characters, it can produce malformed or colliding key names. In a Redis Cluster environment, key-slot targeting could also be manipulated.

**Fix:** Validate `item_id` against an allowlist pattern (e.g., `^[a-zA-Z0-9_-]{1,64}$`) before constructing the key, or hash it.

### 🟡 Significant — Reliability / Architecture

#### 7. `db.update_stock` and `db.create_order` are not atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock has already been decremented with no corresponding order. The Redis lock does not protect against this — it only serializes access, not the DB transaction.

**Fix:** These two operations must be wrapped in a single DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### 8. Bare `RuntimeError` is too broad in the except clause

```python
except RuntimeError as e:
    ...
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

If `db.get_stock`, `db.update_stock`, or `db.create_order` raise a `RuntimeError` (e.g., a connection pool error), it will be silently swallowed and returned as a lock contention message. The caller has no way to distinguish a lock failure from a DB failure.

**Fix:** Define a specific `LockAcquisitionError` exception and catch only that.

---

### 🟠 Test Coverage Gaps

#### 9. Heartbeat is mocked out entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means zero test coverage for the most complex and failure-prone part of the service. At minimum, add:

- A test that verifies the heartbeat calls `redis.expire` with the correct token.
- A test that verifies the heartbeat does **not** extend a lock it no longer owns.
- A test that verifies the thread is stopped before lock release.

#### 10. No test for negative or zero quantity

There is no test asserting that `quantity=0` or `quantity=-1` is rejected.

---

### 🔵 Minor

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the heartbeat fires at 1/3 of TTL. This is fine, but the ratio should be documented as a deliberate choice (e.g., "fire at TTL/3 to tolerate one missed beat").
- The `daemon=True` on the heartbeat thread is correct for preventing process hang, but worth a comment explaining the tradeoff (daemon threads are killed abruptly on interpreter shutdown — acceptable here since the lock has a TTL anyway).
