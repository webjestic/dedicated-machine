Decision: Request Changes

Summary: The implementation contains a critical instance-level concurrency flaw that makes the heartbeat mechanism unsafe under concurrent load: `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields and one call's `_stop_heartbeat()` will silently kill another call's renewal thread. Beyond that architectural defect, the lock release in the `finally` block contains a TOCTOU race, the DB operations are not atomic with respect to each other, and the test suite mocks away the exact components that need exercising. These issues collectively undermine the correctness guarantees the PR claims to provide.

---

## Detailed Findings

### 1. Critical — Shared Heartbeat State (Instance-Level Race Condition)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` is a single boolean on the shared service instance. If two threads call `reserve_item` concurrently:

- Thread A sets `_heartbeat_active = True`, starts its beat thread.
- Thread B sets `_heartbeat_active = True`, starts a second beat thread.
- Thread A finishes, calls `_stop_heartbeat()` → sets `_heartbeat_active = False`.
- Thread B's beat thread now exits silently. Thread B's lock is no longer being renewed and will expire while Thread B is still inside the critical section.

The heartbeat state must be scoped to the individual lock acquisition, not the service instance. The fix is to encapsulate it in a per-call object or pass it through a closure-local flag.

```python
# Correct pattern: closure-local state, not instance state
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if heartbeat_active.is_set():
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    # ... acquire, start t, yield, finally: heartbeat_active.clear(); t.join(timeout=...)
```

Using `threading.Event` also eliminates the busy-sleep pattern and makes shutdown deterministic.

### 2. High — TOCTOU Race in Lock Release

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # (1) read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # (2) delete
```

Between the `GET` at (1) and the `DELETE` at (2), another process could acquire the lock. You then delete a lock you no longer own. This is a well-known failure mode; the correct fix is an atomic Lua script, which is the standard Redis pattern for safe lock release:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In finally block:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 3. High — Non-Atomic DB Operations Inside the Lock

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists — inventory is silently lost. The distributed lock protects against concurrent Redis races but does nothing to protect against partial DB failure.

```python
# Required pattern
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support transactions, that is an architectural gap that must be resolved before this service handles real money.

### 4. Medium — Input Validation Is Declared but Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, or excessively large quantities. A caller passing `quantity=0` would succeed and create a zero-quantity order. A Pydantic model is defined for the *output* (`ReservationResult`) but nothing validates the *input*. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The beat thread is set to `daemon=True` and the flag is cleared, but `_stop_heartbeat` never calls `self._heartbeat_thread.join()`. The thread may still be mid-execution (inside the `redis.expire` call) when the lock is released and deleted. In the worst case the thread extends the TTL of a lock key that has already been re-acquired by a different process. Always join with a bounded timeout:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 6. Low — Tests Mock the Components Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat thread — the most complex and failure-prone component in this PR — is mocked out in every test. The TOCTOU race in lock release is untestable with this setup. The tests verify the happy-path business logic but provide no coverage of the distributed coordination behavior. At minimum, integration tests against a real or embedded Redis instance (e.g., `fakeredis`) should cover: lock expiry during slow DB ops, concurrent reservation attempts, and lock release after token mismatch.
