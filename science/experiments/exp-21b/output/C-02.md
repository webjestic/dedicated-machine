Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw in the heartbeat design — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those fields, causing one reservation to cancel another's heartbeat or overwrite the thread reference entirely. Beyond that, there are several architectural and correctness issues that need to be resolved before this is safe to merge into a government-facing system.

---

## Critical Issues

### 1. Instance-Level Heartbeat State (Race Condition)

This is the most serious flaw. The heartbeat state is stored on `self`, not scoped to the individual lock acquisition:

```python
# BROKEN — shared across all concurrent callers on the same instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` simultaneously:
- Thread A sets `self._heartbeat_active = True` and starts its heartbeat
- Thread B sets `self._heartbeat_active = True` and **overwrites** `self._heartbeat_thread`
- Thread A's `_stop_heartbeat()` sets `self._heartbeat_active = False`, killing **Thread B's heartbeat too**
- Thread B's lock now expires mid-operation, silently corrupting inventory

**Fix:** Scope heartbeat state to the lock context, not the instance. Use a local `threading.Event` per acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    stop_event = threading.Event()
    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)

def _run_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.LOCK_TTL)
        else:
            break  # Lock was lost; no point continuing
```

---

### 2. Heartbeat Thread Is Never Joined

The current `_stop_heartbeat` only sets a flag. The thread is never joined, so:
- The thread may still be running when the lock is released and deleted
- The thread may attempt to `expire` a key that now belongs to a different owner
- Under load, you accumulate ghost threads with no visibility

The fix above addresses this with `heartbeat_thread.join(timeout=...)`.

### 3. Check-Then-Act Race on Lock Release

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # Another owner could acquire between GET and DELETE
```

This is a classic TOCTOU race. Between the `GET` and `DELETE`, another process could acquire the lock (if it expired between those two calls), and you'd delete their lock. This **must** be an atomic operation. The standard solution is a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

## Significant Issues

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input path. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will proceed into the lock acquisition.

```python
# Minimum required
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

Or introduce a `ReservationRequest` Pydantic model with `item_id: str` (non-empty, validated) and `quantity: int = Field(ge=1)`.

### 5. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock protects against concurrent access, but it does not protect against partial writes. These two operations must be wrapped in a database transaction. The `db` abstraction should expose a transaction context, or this method should call a single atomic `db.reserve(item_id, new_stock, order_id, quantity)`.

### 6. Unhandled Exceptions Leak Lock State

Only `RuntimeError` (from lock acquisition failure) is caught. If `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` raises any other exception (connection error, timeout, etc.), it will propagate through the `with self._acquire_lock(lock_key)` block. The `finally` clause will run, which is correct for lock release — but the caller receives an unhandled exception rather than a graceful `ReservationResult`. This is inconsistent with the method's contract of always returning a `ReservationResult`.

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Test Coverage Gaps

### 7. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — the most operationally complex and failure-prone component — has zero test coverage. There are no tests verifying:
- That the heartbeat actually extends the TTL
- That the heartbeat stops when the lock is released
- That the heartbeat correctly skips extension when the token no longer matches
- That the heartbeat thread terminates cleanly

### 8. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test passes because `_start_heartbeat` is mocked, but it also means the test is asserting behavior of the `finally` block in isolation from the heartbeat. More importantly, with the TOCTOU fix (Lua script), this test would need to be rewritten entirely to mock the script execution.

---

## Minor Issues

### 9. `time.sleep` in Retry Loop Blocks the Caller Thread

For a service expected to handle flash sale load, synchronous sleep-based backoff on the calling thread is a concern. This is acceptable if callers are already in worker threads, but it should be documented. If this is ever called from an async context, it will block the event loop.

### 10. Lock Key Namespace Could Collide

`f"inventory:lock:{item_id}"` — if `item_id` is user-supplied and not sanitized, a malformed ID like `lock:admin` would produce `inventory:lock:lock:admin`. Low risk given the validation fix in point 4, but worth noting.
