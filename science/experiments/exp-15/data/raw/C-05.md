Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic NX acquisition, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw between the payment charge and the database transaction that undermines the stated at-most-once guarantee, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge occurs inside the DB transaction — this does not make it atomic

The comment says *"crash between charge and record cannot produce an unrecorded charge"* but that claim is false. A DB transaction does not wrap a network call to an external payment gateway. These are two entirely independent systems.

```python
# Current code — the comment is misleading and the guarantee does not hold
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)             # Only this and below are transactional
    self.db.mark_order_complete(...)
```

**What actually happens on crash:** If the process dies after `payment.charge()` succeeds but before `record_charge()` commits, the customer is charged with no record in the database. The Redis lock has also expired by then, so a retry will pass the idempotency check and charge again.

**Fix:** Charge first, then record. Use an outbox/saga pattern or at minimum a pre-charge reservation record:

```python
# Safer pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# DB transaction only wraps DB writes — be honest about what it covers
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This does not fully solve the crash window, but it stops the misleading comment from hiding the real risk. A proper solution requires an idempotency key sent to the payment gateway so that a retry of the same logical charge returns the existing `charge_id` rather than creating a new charge. The payment client needs to accept and forward an idempotency key.

### 2. `PaymentClient` retries on timeout are unsafe without gateway-side idempotency

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # Falls through to retry
```

A timeout does not mean the charge failed. The gateway may have processed it and the response was lost in transit. Retrying without an idempotency key sent to the gateway will produce duplicate charges. This is a billing correctness issue.

**Fix:** Generate a stable idempotency key per logical charge attempt (e.g., derived from `order_id`) and include it in every request to the gateway. Most payment APIs support an `Idempotency-Key` header for exactly this reason.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

## Significant Issues

### 3. Lock TTL is too short and not validated against payment timeout

```python
ORDER_LOCK_TTL = 60   # seconds
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Maximum payment time: `3 × 30s + 2 × 5s = 100 seconds`. The lock TTL is 60 seconds. A slow payment gateway response will cause the lock to expire while the charge is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (no record yet), and issue a second charge. The TTL must exceed the maximum possible payment duration with margin.

```python
# Minimum safe TTL
ORDER_LOCK_TTL = (MAX_RETRIES * REQUEST_TIMEOUT) + ((MAX_RETRIES - 1) * RETRY_BACKOFF) + 30
# = 100 + 30 = 130 seconds minimum
```

These constants should be co-located or validated at startup so this relationship cannot silently drift.

---

### 4. `payment_token` is logged at INFO level in `payment_client.py`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine, but check your broader logging configuration. The `payment_token` field flows through `OrderRequest` and is logged in warning output on validation errors:

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. A malformed `payment_token` will appear in your logs. Scrub sensitive fields before logging or use a custom error formatter.

---

### 5. Lock retry backoff is linear and blocks the worker thread

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

Linear backoff with no jitter means all contending workers retry at the same intervals, creating a thundering herd. Use exponential backoff with jitter. Also, `time.sleep` in a web worker context blocks the thread for the full duration — this should be documented or made configurable for async environments.

```python
import random
backoff = min(0.1 * (2 ** attempt) + random.uniform(0, 0.1), 2.0)
```

## Minor Issues

### 6. `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary characters up to 64 bytes. If `customer_id` is used in any downstream query construction or logging, this is a latent injection surface. Apply the same pattern constraint.

### 7. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()   # Does not check args
```

This test confirms the script is called but does not verify that the correct token is passed. A regression that passes the wrong token would not be caught. Assert on `call_args`.

### 8. No test covering lock expiry during payment (the TTL race described above)

The test suite does not exercise the scenario where the lock expires mid-payment. This is the highest-risk failure mode and should have an explicit test.

### 9. `PaymentClient` has no authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

No API key, bearer token, or mTLS configuration is present. Even for an internal endpoint this should be explicit, not left to ambient network controls.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct
- Lua script for token-verified release is the right approach
- Pydantic validation with regex on `order_id` is good
- `raise_for_status()` with differentiated 4xx vs 5xx handling is correct
- Structured logging with consistent field names throughout
- Test fixture design is clean and the mock setup is reasonable

---

The two blocking items are the false atomicity claim around the payment charge and the unsafe retry behavior in `PaymentClient` without gateway idempotency keys. Both directly threaten billing correctness. Fix those and address the TTL math before merging.
