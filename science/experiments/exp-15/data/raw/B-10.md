# Pull Request Review

## Distributed Systems Analysis: `feature/order-processor`

---

## Critical Findings

### 🔴 CRITICAL-1 — Payment charge executes *inside* a database transaction (charge-then-record is not atomic across service boundaries)

**File:** `services/order_processor.py`, lines inside `with self.db.transaction():`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary claims: *"crash between charge and record cannot produce an unrecorded charge."* This is **false**, and the design inverts the correct pattern.

**What actually happens:**

| Failure Point | Outcome |
|---|---|
| `payment.charge()` succeeds, then DB crashes before commit | Charge is real, money is taken, no record exists — **silent double-charge on retry** |
| DB transaction holds open during the full payment gateway round-trip (up to 30s × 3 retries = 90s) | Long-held DB connection/lock under load |
| Payment gateway returns 200 but the response read times out | `charge_id` is unknown; retry fires a second charge |

**The correct pattern** is to perform the external side effect *after* the database has durably recorded intent:

```python
# 1. Record intent / reserve idempotency slot (DB commit)
with self.db.transaction():
    self.db.reserve_order(req.order_id, req.customer_id, req.amount_cents)

# 2. Execute external side effect
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Record outcome (DB commit)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

A background reconciliation job should detect rows in state `reserved` with no `charge_id` after a TTL and resolve them against the payment gateway's idempotency API. This is standard practice for payment systems.

### 🔴 CRITICAL-2 — Redis lock TTL is shorter than the maximum payment retry window

**File:** `services/order_processor.py` and `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py — per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall-clock time: `(30s timeout + 5s backoff) × 3 attempts = 105 seconds`

The lock TTL is **60 seconds**. A slow payment gateway can expire the lock while the original worker is still inside the critical section. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. The distributed lock provides **no mutual exclusion guarantee** under this condition.

**Fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed the maximum payment window (e.g., 180s with margin), or implement lock extension (Redlock-style `PEXPIRE` refresh) during the payment call. The TTL and payment timeout budget must be defined together and enforced as a single invariant, not as two independent constants in two files.

### 🔴 CRITICAL-3 — Payment client retries on timeout without gateway-side idempotency key

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... falls through to retry
```

A `Timeout` exception means the request *may have been received and processed* by the gateway — the response was simply lost in transit. Retrying without an idempotency key on the request will issue a **second charge** for the same payment token and amount. This is a textbook duplicate-charge vulnerability.

**Fix:** Generate a stable idempotency key (e.g., derived from `order_id`) before the retry loop and include it in every attempt:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `idempotency_key` should be the `order_id` or a deterministic hash of it, passed in from the caller, not generated inside `charge()`.

### 🔴 CRITICAL-4 — Single Redis node — no Redlock, no quorum

**File:** `services/order_processor.py`

```python
def __init__(self, redis_client: redis.Redis, ...):
    self.redis = redis_client
```

The lock is acquired against a single Redis instance. Per the [Redis documentation on distributed locks](https://redis.io/docs/manual/patterns/distributed-locks/): *"The above algorithm is safe only if you are working with a single, always-available instance."* Under a Redis failover (sentinel promotion, cluster primary election), a lock held on the old primary may not exist on the new primary. A second worker acquires the lock, and mutual exclusion is violated.

For a payment processing system at a government contractor, this is not an acceptable single point of failure. Either:
- Use the Redlock algorithm across ≥5 independent Redis nodes, or
- Use a CP-consistent coordination service (etcd, ZooKeeper, Postgres advisory locks) as the lock backend, or
- Document explicitly that this deployment assumes a single Redis instance with no failover and accept the risk window

## High Severity Findings

### 🟠 HIGH-1 — `payment_token` logged at INFO level on success path

**File:** `services/payment_client.py`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

While the token itself is not logged here, `payment_token` is present in the `OrderRequest` object and appears in `logger.warning("Invalid order request: %s", e)` — Pydantic validation errors include field values in their string representation by default. A validation error on `payment_token` (e.g., length violation) will emit the token value to the log.

**Fix:** Sanitize Pydantic error output before logging, or use a custom `__repr__` on `OrderRequest` that masks sensitive fields:

```python
except ValidationError as e:
    logger.warning("Invalid order request order_id=%s: validation failed", order_id)
    # Do NOT log `e` directly — contains field values
```

---

### 🟠 HIGH-2 — Lock release silently swallows failure

**File:** `services/order_processor.py`

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception propagates out of the `finally` block and **suppresses the original exception** from the `try` body. Additionally, a return value of `0` (token mismatch — lock was already expired and acquired by another worker) is silently ignored. The log line says "released" even when the lock was not held.

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error(
                "Lock release returned 0 — lock expired or stolen order_id=%s", order_id
            )
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s", order_id)
```

### 🟠 HIGH-3 — `db.get_charge()` idempotency check is outside the database transaction

**File:** `services/order_processor.py`

```python
existing = self.db.get_charge(req.order_id)   # read outside transaction
if existing:
    return ...

with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

The check and the write are not atomic at the database level. If two workers both pass the idempotency check before either commits (possible if the Redis lock fails per CRITICAL-4), both will proceed to charge. The idempotency check must be enforced by a **unique constraint on `order_id`** in the database schema, with the application catching the constraint violation as the authoritative duplicate signal. The application-level check is a performance optimization only, not a correctness guarantee.

## Medium Severity Findings

### 🟡 MED-1 — `customer_id` has no pattern validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary Unicode including control characters. If `customer_id` is used in log formatting, database queries, or downstream API calls, this is an injection surface. Apply the same or equivalent pattern constraint.

---

### 🟡 MED-2 — `payment_token` is stored and transmitted in plaintext

`payment_token` passes through Pydantic, into the `OrderRequest` object, into `db.record_charge()` (implied by the signature), and into the payment client. If the DB layer stores the raw token, this is a PCI-DSS violation. The token should be treated as a use-once credential: passed to the payment client and never persisted.

---
