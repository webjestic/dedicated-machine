Decision: Request Changes

Summary: The core locking and idempotency design is solid â€” atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issue â€” Payment Call Outside the Database Transaction

This is the most serious bug in the PR. The comment in the code actually *describes* the guarantee incorrectly:

```python
# Charge and record atomically â€” crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)       # <-- network call to external system
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**The payment charge is inside the `db.transaction()` block, but a database transaction provides no atomicity guarantee over an external network call.** The sequence of failure modes here is:

1. `payment.charge()` succeeds â€” money is taken from the customer
2. `self.db.record_charge()` raises an exception (deadlock, connection drop, disk full, etc.)
3. The `db.transaction()` rolls back
4. `get_charge()` will now return `None` on the next attempt
5. The customer gets charged again

The correct pattern is **charge first, then open the transaction to record**:

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only open the DB transaction after the external call succeeds
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This does not eliminate the crash-between-charge-and-record window, but it makes the DB write as tight as possible around a single commit, and the idempotency check on `get_charge()` will catch retries correctly. The Redis lock alone is not sufficient protection here â€” it only prevents *concurrent* double charges, not a *sequential* retry after a DB failure.

## High Priority Issues

### 1. Unhandled `PaymentError` Propagates Out of `process()`

`PaymentError` is never caught in `OrderProcessor.process()`. If the payment gateway returns a 4xx or exhausts retries, the exception bubbles up to the caller as an unhandled exception rather than returning a structured `OrderResult`. This is inconsistent with how `LockError` and `ValidationError` are handled and will likely cause 500 errors at the API layer.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

### 2. Lock TTL Is Dangerously Tight Given Retry Logic

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire before the payment client finishes, allowing a second worker to acquire the lock and potentially charge the customer again while the first worker is still mid-flight. Either increase the TTL to safely exceed the maximum payment client runtime, or implement lock extension (renewal) before expiry.

### 3. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation, its value could appear in logs. Payment tokens must never be logged. At minimum, sanitize the error output or log only field names, not values.

---

## Medium Priority Issues

### 4. No Test Coverage for `PaymentError`

The test suite has no case for payment gateway failure. Given that `PaymentError` is currently unhandled (issue #1 above), this gap allowed the bug to ship undetected. Add tests for both transient retry exhaustion and permanent 4xx decline.

### 5. `requests.post` Has No Authentication

`PaymentClient` posts to an internal endpoint with no authentication header. Even internal service-to-service calls should use mTLS or a bearer token. This is a security gap for a government contracting context.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
    # Missing: headers={"Authorization": f"Bearer {self._api_key}"}
)
```

### 6. `customer_id` Has No Pattern Validation

`order_id` correctly enforces `r'^[a-zA-Z0-9_\-]+'` but `customer_id` only enforces length. If `customer_id` is used in any downstream query construction, this is a latent injection surface. Apply consistent sanitization.

### 7. Lock Retry Backoff Is Blocking

`time.sleep()` in `_acquire_lock` blocks the worker thread. In any async or thread-pool context this will stall other work. If the service is or will be async, this needs `asyncio.sleep`. At minimum, document the synchronous assumption explicitly.

## Minor Issues

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants but should be configurable at construction time to allow environment-specific tuning without code changes.
- The `finally` block in `_acquire_lock` will attempt to release the lock even if it was never acquired â€” though `acquired` is checked before `yield`, a re-raise path after a failed acquire could theoretically reach `finally`. The current structure is safe because `yield` only runs after the `if not acquired: raise` guard, but this is worth a comment for the next reader.
- `charge_id` is returned directly from `response.json()["charge_id"]` with no `KeyError` guard. A malformed gateway response will raise an unhandled exception.

---

## What Is Done Well

The Lua script for atomic lock release is correct and the token-verification pattern is the right approach. Input validation with Pydantic and the `order_id` regex are good. The structured logging is consistent. The test for path traversal on `order_id` is a nice explicit safety check.
