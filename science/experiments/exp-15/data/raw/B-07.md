# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Charge Outside Database Transaction Atomicity Boundary

The most severe flaw in this PR is the comment claiming at-most-once safety:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)        # ← external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an **external HTTP call to a third-party payment gateway**. It is not a participant in `self.db.transaction()`. A database transaction provides atomicity over database operations only. It provides **zero rollback guarantee** over the payment gateway call.

The failure scenario the comment claims is impossible is, in fact, entirely possible:

1. `payment.charge()` succeeds — money is moved, gateway issues `charge_id`
2. Process crashes, network drops, or the DB transaction rolls back before `record_charge` commits
3. Result: charge occurred, no record exists, idempotency check on retry returns `None`, **worker charges again**

This is a double-charge vulnerability. The fix requires writing an idempotency/intent record to the database **before** calling the payment gateway, so that a crash-and-retry path detects the in-flight or completed charge regardless of whether the gateway response was received.

The correct pattern is roughly:

```python
# 1. Write intent record inside a DB transaction BEFORE calling gateway
with self.db.transaction():
    self.db.create_charge_intent(order_id, customer_id, amount_cents)

# 2. Call gateway OUTSIDE the DB transaction
charge_id = self.payment.charge(token=..., amount_cents=...)

# 3. Finalize the record
with self.db.transaction():
    self.db.finalize_charge(order_id, charge_id)
    self.db.mark_order_complete(order_id)
```

On retry, step 1 detects the existing intent record and the idempotency check blocks re-entry before the gateway is called again.

### 🔴 Critical: Redis Lock TTL Does Not Cover Payment Retry Duration

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

```python
# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. The lock **expires before the payment completes**. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still waiting on a gateway response. This directly undermines the duplicate-charge prevention guarantee the PR claims to provide.

The TTL must be set to a value strictly greater than the maximum possible payment client execution time, with margin. Either raise `ORDER_LOCK_TTL` to at least 120–150 seconds, or reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` such that their product plus backoff fits within the TTL. The two constants must be defined together with an explicit invariant enforced in code or configuration validation, not left as independent magic numbers in separate files.

### 🔴 Critical: `PaymentError` Unhandled in `process()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

`self.payment.charge()` raises `PaymentError` on permanent failure or retry exhaustion. `process()` does not catch `PaymentError`. The exception propagates out of `_acquire_lock`'s `try` block, hits the `finally`, releases the lock correctly, and then surfaces as an unhandled exception to the caller.

The caller receives an uncaught exception rather than an `OrderResult`. This breaks the stated return contract of the method and forces every caller to defensively catch an undocumented exception type that is not imported into `order_processor.py`. `PaymentError` must be caught and mapped to a structured `OrderResult(processed=False, ...)`.

---

### 🟡 Significant: Idempotency Check Is Not Inside the Lock's Consistent Read Window

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # read after lock
    if existing:
        return ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
```

The lock is acquired correctly before the read, which is the right structure. However, `get_charge` and the subsequent transaction are two separate database operations. If the database read uses a different connection or isolation level than the transaction, a concurrent commit from another process (one that somehow bypassed the Redis lock, e.g., due to lock expiry per the TTL issue above) could be invisible to this read. The idempotency check should use `SELECT ... FOR UPDATE` or equivalent within the same transaction to ensure the read and the subsequent write are serialized at the database level as a second line of defense. Redis alone should not be the only serialization mechanism for financial operations.

---

### 🟡 Significant: `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in the log output. Payment tokens are sensitive credentials. The log statement should either suppress field values entirely or explicitly reconstruct a safe error message listing only field names and error types.

---

### 🟡 Significant: Lock Retry Backoff Blocks the Worker Thread

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

This is a synchronous sleep on what is presumably a worker thread in a concurrent processing pool. Under sustained lock contention, worker threads accumulate in sleep, reducing throughput. This is an architectural concern if the service is expected to handle high concurrency. The appropriate fix depends on the deployment model (async workers, thread pool sizing), but the reviewer should flag that blocking sleep in a lock retry loop does not scale and should be replaced with async-compatible backoff if this runs under an async framework, or the thread pool must be sized to account for threads blocked in retry.

## `services/payment_client.py`

### 🟡 Significant: Timeout on Retry Does Not Account for Elapsed Time

```python
for attempt in range(self.MAX_RETRIES):
    response = requests.post(..., timeout=self.REQUEST_TIMEOUT)
    ...
    time.sleep(self.RETRY_BACKOFF)
```

Each retry resets the full `REQUEST_TIMEOUT`. There is no total deadline across all attempts. As noted above, worst-case wall time is 100 seconds. There is no circuit-breaker, no total budget, and no way for the caller to bound the maximum latency of a `charge()` call. For a financial operation inside a distributed lock, unbounded retry duration is a systemic risk.

---

### 🟠 Minor: `response.json()["charge_id"]` Has No Error Handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed body, missing `charge_id`, or non-JSON content type, this raises `KeyError` or `JSONDecodeError`. Neither is caught. Both propagate as unexpected exceptions rather than `PaymentError`. This should be wrapped with explicit error handling.

## `tests/test_order_processor.py`

### 🔴 Critical: No Test Covers the Double-Charge Scenario

There is no test for the scenario where `payment.charge()` succeeds but `db.record_charge()` raises an exception. This is precisely the failure mode that the PR's architecture comment claims is impossible but is not. A test should exist that verifies the system's behavior when the gateway succeeds and the DB write fails, and that behavior should be defined (currently it is undefined and dangerous).

### 🟡 Significant: Lock Expiry During Payment Is Not Tested

No test simulates the lock TTL expiring mid-payment. Given that this is the mechanism the PR relies on for safety, the absence of a test for lock expiry behavior is a gap.

### 🟠 Minor: `test_lock_release_uses_lua_script` Does Not Verify Token Matching

```python
processor._release_script.assert_called_once()
```

This asserts the script was called but does not verify that the token passed as `args=[token]` matches the token stored in Redis. The test does not validate the correctness of the token-verified release, only that the script was invoked.

---

**Decision: Request Changes**

**Summary:** This PR contains critical architectural flaws that directly contradict its stated safety guarantees. The payment gateway call is placed inside a database transaction boundary that provides no atomicity over external HTTP calls, creating a real double-charge path on crash-and-retry. Independently, the Redis lock TTL of 60 seconds is shorter than the maximum payment client execution time of 100 seconds, allowing a second worker to acquire the lock and pass the idempotency check while the first worker's gateway call is still in flight. These two issues together mean the duplicate-charge prevention guarantee claimed in the PR description does not hold under failure conditions that are routine in distributed deployments. No approval is possible until these are resolved.
