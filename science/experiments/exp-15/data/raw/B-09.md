# Pull Request Review

## Distributed Systems Analysis: `feature/order-processor`

---

## Critical Findings

### 🔴 CRITICAL — Charge Outside Transaction Boundary Creates Unrecoverable Duplicate Charge Window

This is the most severe flaw in the PR. The PR summary and inline comment both claim:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

**This claim is false.** The `payment.charge()` call is placed *inside* `with self.db.transaction()`, but a database transaction provides atomicity only over database operations. The external HTTP call to the payment gateway is not a database operation and cannot be rolled back.

```python
# services/order_processor.py, lines within process()
with self.db.transaction():
    charge_id = self.payment.charge(       # ← External HTTP call. NOT atomic with DB.
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)             # ← If this fails, charge_id is lost forever.
    self.db.mark_order_complete(...)
```

**Failure scenario:**

1. Worker A acquires lock, enters `db.transaction()`
2. `payment.charge()` succeeds — money is moved, `charge_id` is returned
3. `db.record_charge()` raises (DB connection drops, constraint violation, OOM, anything)
4. Transaction rolls back — `charge_id` is never persisted
5. Lock is released in `finally`
6. Worker B (or retry) acquires lock, idempotency check finds nothing, **charges again**

The customer is double-charged. The `charge_id` from step 2 is gone. There is no recovery path.

**Correct pattern — charge must happen outside and before the transaction:**

```python
# Charge FIRST, outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record atomically. If this fails, charge_id is known and can be reconciled.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # charge_id is known — log it for reconciliation, do not silently lose it
    logger.critical(
        "CHARGE_UNRECORDED order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

Even with this fix, a crash between `charge()` returning and `record_charge()` committing still creates an unrecorded charge. This is an inherent property of distributed systems involving external payment processors. The correct mitigation is an outbox/reconciliation pattern or a payment gateway that supports idempotency keys — not wrapping an HTTP call in a DB transaction.

### 🔴 CRITICAL — Payment Retry on Timeout Violates At-Most-Once Guarantee

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# loop continues → retries the charge
```

A `Timeout` exception means the client did not receive a response. It does **not** mean the charge failed. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout can charge the customer multiple times per invocation of `process()`.

This interacts lethally with the Redis lock: the lock is held for the entire retry loop, so the idempotency check cannot protect against retries happening *within the same lock acquisition*.

**Required fix:** The payment gateway must expose an idempotency key mechanism. Pass a deterministic idempotency key (e.g., `order_id` + attempt nonce) with each charge request so the gateway deduplicates retries server-side. Without this, retry-on-timeout is unsafe regardless of any client-side locking.

### 🔴 CRITICAL — Lock TTL Does Not Cover Payment Retry Duration

```python
ORDER_LOCK_TTL = 60   # seconds
```

```python
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment client execution time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. The payment client can run for 100 seconds. The lock expires while the worker is still inside the critical section. A second worker then acquires the lock, passes the idempotency check (nothing recorded yet), and issues a second charge.

The lock TTL must be strictly greater than the maximum possible execution time of everything inside the critical section, including all retries. Either increase `ORDER_LOCK_TTL` to ≥ 120 seconds with margin, or reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` such that the product is provably bounded below the TTL. These constants living in separate files with no enforced relationship is itself an architectural smell — this invariant should be asserted at startup.

## High Severity Findings

### 🟠 HIGH — Unhandled `PaymentError` Propagates Through Lock Release Without Result

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # raises PaymentError
        ...
# PaymentError propagates out of process() entirely
```

`PaymentError` is not caught in `process()`. It will propagate to the caller as an unhandled exception. The method signature promises `OrderResult` but can throw. The lock is released correctly via `finally`, but the caller receives an exception rather than a structured `OrderResult(processed=False, ...)`. This breaks the API contract and will cause unhandled exceptions in any caller that does not defensively wrap `process()`.

---

### 🟠 HIGH — Idempotency Check Is Outside the Transaction (TOCTOU)

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # read
    if existing:
        return ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)                 # write
```

The read (`get_charge`) and the write (`record_charge`) are not in the same transaction. Under normal single-worker operation with the Redis lock this is acceptable. However:

- If the Redis lock is lost (TTL expiry, Redis restart, network partition) between the check and the write, two workers can both pass the idempotency check simultaneously.
- The idempotency check should be inside the database transaction, ideally as a `INSERT ... WHERE NOT EXISTS` or with a unique constraint on `order_id` that will hard-fail on duplicate, providing a true atomic check-and-insert at the database layer.

The Redis lock is a performance optimization (avoid hitting the DB), not a correctness guarantee. The database must be the final arbiter of uniqueness.

---

### 🟠 HIGH — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in the log output. Payment tokens are sensitive credentials. Log the error type and field names only, never field values for sensitive fields.

## Medium Severity Findings

### 🟡 MEDIUM — `requests` Has No Connection Timeout, Only Read Timeout

```python
response = requests.post(..., timeout=self.REQUEST_TIMEOUT)
```

Passing a single scalar to `timeout` in `requests` sets only the read timeout. The connection phase is unbounded. Use `timeout=(connect_timeout, read_timeout)`:

```python
timeout=(5, self.REQUEST_TIMEOUT)
```

An unreachable gateway will hang the worker indefinitely at the TCP connection phase, holding the Redis lock until TTL expiry.

---

### 🟡 MEDIUM — Lock Retry Backoff Is Linear, Not Jittered

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Under high concurrency, all workers waiting on the same order will retry on the same schedule, producing synchronized thundering-herd retries against Redis. Use exponential backoff with jitter.

---

### 🟡 MEDIUM — No Observability on Lock Expiry

There is no mechanism to detect or alert when a lock expires before the critical section completes. This failure mode (TTL < execution time) is silent — the system will double-charge without any log entry indicating the lock was lost. A lock heartbeat or a post-hoc check that the lock token still matches before committing the DB transaction would surface this.

---

### 🟡 MEDIUM — Test Suite Does Not Cover the Critical Failure Modes

The tests cover the happy path and basic contention. None of the following are tested:

| Scenario | Tested |
|---|---|
| DB write fails after charge succeeds | ❌ |
| Payment timeout followed by retry succeeds (duplicate charge) | ❌ |
| Lock TTL expires mid-execution | ❌ |
| `PaymentError` propagation (unhandled exception from `process()`) | ❌ |
| Redis failure during lock acquisition | ❌ |
| Idempotency check race (lock lost between check and write) | ❌ |

The tests as written validate the happy path and provide false confidence that the at-most-once guarantee holds.
