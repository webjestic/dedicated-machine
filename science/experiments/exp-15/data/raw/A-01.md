**Decision: Request Changes**

**Summary:** The locking architecture has a critical atomicity gap: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that times out before the gateway returns a `charge_id` will be retried by `PaymentClient` — but the outer Redis lock has already been acquired and the idempotency check passed, so a second charge attempt on the same token can fire within the same lock window. The distributed lock also cannot protect against the crash-between-charge-and-record scenario it claims to, because the lock TTL expires independently of the database transaction boundary, and no dead-letter or reconciliation path exists for orphaned charges. Additionally, `PaymentClient` retries on timeout without any idempotency key sent to the gateway, making duplicate charges on transient network failures a near-certainty in production.

---

## Detailed Findings

### Critical — Charge-Inside-Transaction with No Idempotency Key

```python
# services/order_processor.py, lines inside process()
with self.db.transaction():
    charge_id = self.payment.charge(          # ← network call inside DB transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
```

A database transaction should never wrap an external network call. The consequences here are layered:

1. The DB connection is held open for up to `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES-1)` = `30×3 + 5×2 = 100 seconds` under worst-case gateway behavior. This will exhaust your DB connection pool under any real load.
2. If `payment.charge()` times out, `PaymentClient` retries — but the gateway may have already processed the first attempt. Without an idempotency key scoped to `order_id`, the gateway has no way to deduplicate, and you will double-charge.
3. The comment "crash between charge and record cannot produce an unrecorded charge" is incorrect. A process crash after the gateway responds but before `db.record_charge()` commits will produce exactly that: a real charge with no record.

**Required fix:** Generate a stable idempotency key (e.g., `f"order:{order_id}"`) and pass it to the gateway on every attempt. Move `payment.charge()` *outside* the database transaction. Only open the DB transaction after you have a confirmed `charge_id` in hand.

```python
# Correct structure
idempotency_key = f"order:{req.order_id}"
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,   # gateway deduplicates on its side
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

### Critical — Lock TTL Does Not Cover Actual Critical Section Duration

```python
ORDER_LOCK_TTL = 60   # seconds
# vs.
REQUEST_TIMEOUT = 30  # per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # between retries
# worst-case payment duration: 100 seconds > 60-second lock TTL
```

The lock expires before the payment client can finish under worst-case retry behavior. A second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still waiting on a gateway retry. Either raise `ORDER_LOCK_TTL` well above the maximum possible payment duration, or implement lock extension (Redlock-style keepalive) for long-running operations. The TTL must be a hard ceiling above `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF`.

---

### High — No Reconciliation Path for Orphaned Charges

If the process crashes after a successful charge but before `db.record_charge()` commits, the charge exists at the gateway but not in your database. The idempotency check (`db.get_charge()`) will return `None` on the next attempt, and if the gateway does not have an idempotency key to match against, it will charge again. There is no dead-letter queue, no charge reconciliation job, and no alerting path mentioned. This is a financial correctness requirement, not a nice-to-have.

---

### High — `PaymentClient` Retries Timeouts Without Idempotency Key

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry loop — same token, same amount, no dedup key
```

A timeout means the request *may* have been processed. Retrying an identical POST without an idempotency key is indistinguishable from a new charge request at the gateway. This is the most common source of production double-charges in payment integrations. The fix is the same as above: require and forward an idempotency key on every call.

---

### Medium — `customer_id` Validation Is Incomplete

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces an alphanumeric-plus-hyphen-underscore pattern. `customer_id` has no pattern constraint, which means arbitrary Unicode, control characters, or injection-relevant strings pass validation and reach `db.record_charge()`. Apply the same or a stricter pattern to `customer_id`.

---

### Medium — `payment_token` Logged at INFO Level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This line is safe, but `payment_token` is present in the `OrderRequest` object that is constructed and held in scope throughout. Confirm that no logging call, exception serializer, or Pydantic `.model_dump()` anywhere in the call stack emits `payment_token` to logs. Consider marking it with `SecretStr` in Pydantic to prevent accidental exposure:

```python
from pydantic import SecretStr
payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

---

### Low — Test Suite Does Not Cover the Critical Failure Modes

The tests cover the happy path, already-processed idempotency, lock contention, and input validation — all correct. What is absent:

- No test for gateway timeout followed by retry (the double-charge scenario).
- No test for process behavior when `ORDER_LOCK_TTL` expires mid-payment.
- No test asserting that `payment.charge()` is called with an idempotency key.
- No test for `db.transaction().__exit__` being called on payment failure (rollback path).

These are not cosmetic gaps; they are the exact scenarios this service is supposed to protect against.

---

### Low — `CHARGE_ENDPOINT` Hardcoded as Class Attribute

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoints in source are a deployment hygiene issue and make environment-specific configuration (staging vs. production) require code changes. Move this to an injected config or environment variable.
