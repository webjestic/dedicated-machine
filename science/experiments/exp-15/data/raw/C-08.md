Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* because both operations are inside `with self.db.transaction()`. This is incorrect and the claim is dangerous.

```python
# services/order_processor.py
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. The `payment.charge()` call is an external HTTP request to a third-party gateway. If the charge succeeds but the process crashes before `record_charge` commits, the money has moved but no record exists. The DB transaction rolling back does nothing to reverse the external charge.

The correct pattern is to record a *pending* charge intent before calling the gateway, then update its status after confirmation:

```python
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This way a crash after the gateway call but before finalization leaves a *pending* record that a reconciliation job can detect and resolve. A crash before the gateway call leaves a pending record that was never charged and can be safely voided. Neither case produces an invisible charge.

---

### 2. Lock TTL is too short relative to the payment client's maximum runtime

```python
ORDER_LOCK_TTL = 60   # order_processor.py
REQUEST_TIMEOUT = 30  # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock expires after 60 seconds. A second worker can acquire the lock while the first is still mid-charge on attempt 3, breaking the mutual exclusion guarantee entirely. Either extend the TTL to at least 150 seconds (with margin), or implement lock heartbeat renewal.

---

### 3. Timeout on a charge request is ambiguous — retrying is unsafe without idempotency keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request may or may not have succeeded at the gateway. Retrying a timed-out charge without a stable idempotency key sent to the gateway can result in a double charge. The `payment_token` field is not suitable for this — it identifies the payment method, not the specific charge attempt.

The fix is to generate a stable per-order idempotency key (e.g., derived from `order_id`) and include it in every retry attempt as a gateway idempotency header:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

## Moderate Issues

### 4. `payment_token` is logged indirectly and passed through Pydantic with no masking

The `payment_token` field flows through `OrderRequest` and is referenced in log calls nearby. Confirm that no log statement anywhere in the call chain emits the raw token. The Pydantic model should also mark this field sensitive:

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

This prevents accidental serialization of the token in error messages or debug output.

---

### 5. `PaymentClient` has no authentication on the internal gateway call

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, mTLS configuration, or API key. Even for an internal endpoint this is a gap — defense in depth requires authenticating service-to-service calls. This needs to be addressed before production deployment.

---

### 6. `charge_id` key access on gateway response is not guarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed body (missing `charge_id`), this raises an unhandled `KeyError` that propagates out of the lock context. It will be caught by nothing, the lock will still be released correctly (the `finally` block handles that), but the order will be left in an indeterminate state with no structured error returned to the caller. Use `.get()` and raise a `PaymentError` explicitly:

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError("Gateway returned success but no charge_id in response")
```

## Minor Issues

### 7. Test for lock release correctness is insufficient

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This confirms the Lua script was called but does not verify the token passed to it matches the token set in Redis. A test should capture the token written via `redis.set` and assert it is the same value passed to `_release_script`. Without this, the token-verification guarantee is untested.

### 8. No test for the lock expiry / TTL-shorter-than-payment-runtime scenario

Given issue #2 above, a test that simulates a slow payment gateway (exceeding TTL) and asserts the system behaves safely would be required once the TTL issue is resolved.

### 9. `customer_id` has no pattern constraint

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary unicode up to 64 characters. If `customer_id` is used in any downstream query or log aggregation key, apply a matching constraint.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct.
- Token-verified Lua release is the right approach and correctly implemented.
- Pydantic validation with pattern enforcement on `order_id` is good.
- Retry with backoff on 5xx is appropriate.
- Structured logging with key=value pairs is clean and consistent.
- The test fixture structure is reasonable and the happy-path coverage is present.

The architecture is on the right track but the atomicity claim in the PR description is incorrect as implemented, and the lock TTL / payment timeout mismatch is a real race condition under load. These need to be resolved before this touches production payment flows.
