Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical reliability flaw in the payment/transaction boundary that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this touches production.

---

## Critical Issue

### Payment call inside the database transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call inside TX
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious problem in the PR. The payment call is an external HTTP request (up to 90 seconds with retries) held open inside a database transaction. This will:

1. **Hold a DB connection and locks for the full payment round-trip**, causing connection pool exhaustion and lock contention under any real load.
2. **Create a phantom charge window** — if `payment.charge()` succeeds but the DB transaction rolls back (network blip, DB restart, application crash), you have charged the customer with no record. The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. A DB failure after a successful charge response absolutely can.
3. **Invert the failure modes** — the correct pattern is charge first, then open the shortest possible transaction to record it. If the record fails, you have a charge ID to reconcile against. If you never get a charge ID, you never charged.

The fix:

```python
# Outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Shortest possible transaction — only DB writes
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

You still need a reconciliation/recovery path for the case where `record_charge` fails after a successful charge, but at minimum the transaction must not wrap the network call.

## Secondary Issues

### Unhandled `PaymentError` propagates through the lock

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
```

`PaymentError` (and any other unexpected exception) is not caught in `process()`. It will propagate up through `_acquire_lock`'s `finally` block (lock is correctly released), but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract the return type implies and will cause unhandled 500s at the API layer.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

---

### Timeout on payment gateway is dangerously long relative to lock TTL

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The Redis lock TTL is 60 seconds. A slow-but-not-failing payment gateway will cause the lock to expire before the payment completes, allowing a second worker to acquire the lock and attempt a duplicate charge while the first is still in flight. The lock TTL must exceed the maximum possible payment client duration, or the payment client's maximum duration must be reduced to safely fit within the TTL.

---

### `payment_token` logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value in some versions. `payment_token` is a sensitive credential. At minimum, scrub or redact the token field before logging, or log only field names and error types, not values.

---

### `charge_id` key access on payment response is unguarded

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected JSON body, this raises a `KeyError` or `JSONDecodeError` that is not caught. This should be wrapped:

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError("Gateway returned success but no charge_id")
```

---

### Test coverage gaps

- No test for `PaymentError` raised during processing.
- No test asserting the lock TTL is longer than the maximum payment client duration (this relationship should be enforced, not just documented).
- No test for the DB transaction failing after a successful charge (the reconciliation path).
- `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` — this is a weak negative assertion. It does not verify the Lua script was called with the correct token, only that it was called at all. The token verification is the entire point of the Lua script.

---

## Minor Notes

- `customer_id` has no pattern constraint, unlike `order_id`. If it flows into queries or logs, it should have the same treatment.
- `CHARGE_ENDPOINT` is hardcoded. This should come from config/environment to support staging vs. production deployments.
- The lock retry backoff is linear (`0.5 * (attempt + 1)`). Exponential with jitter is standard for distributed systems to avoid thundering herd on lock release.
