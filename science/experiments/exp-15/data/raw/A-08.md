Decision: Request Changes

Summary: The distributed locking design contains a critical TOCTOU gap that directly undermines the at-most-once guarantee the PR claims to provide: the payment charge occurs *inside* the database transaction but *outside* any atomicity boundary with the idempotency check, meaning a timeout on the payment gateway can produce an unrecorded charge that the retry path will then charge again. Beyond that core flaw, the lock TTL is shorter than the maximum possible processing time, the payment client leaks sensitive token values into logs on failure paths, and several test fixtures mock away exactly the failure modes that matter most. Each of these is detailed below.

---

## Critical: TOCTOU / Double-Charge Window on Gateway Timeout

This is the most serious issue. The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is false, and the failure mode is subtle enough that it deserves a full trace.

```
Worker A                          Payment Gateway
--------                          ---------------
acquire lock ✓
idempotency check → no record
db.transaction().__enter__
  payment.charge(token, amount) ──────────────────→ [charge succeeds internally]
                                  ←── TCP timeout (30 s) ──
  requests.Timeout raised
  → PaymentError after 3 retries  [charge_id never returned]
db.transaction().__exit__ (rolls back — nothing was recorded)
lock released
```

Now the order has been charged at the gateway but has no record in the database. The next worker that picks up this order will:

1. Acquire the lock (it was released).
2. Run the idempotency check → finds **no record** (the transaction rolled back).
3. Call `payment.charge()` again → **second charge**.

The retry loop in `PaymentClient.charge()` compounds this: on a `Timeout`, it retries up to `MAX_RETRIES` times *within a single call*. Each retry may hit a gateway that already processed the previous attempt. The gateway's own idempotency behavior is not modeled, assumed, or tested here.

**Required fix:** The charge must be separated from the database transaction into a two-phase pattern:

1. **Pre-flight:** Write a `PENDING` record for `order_id` *before* calling the gateway (inside a transaction). This record acts as a reservation.
2. **Charge:** Call the gateway. On timeout/ambiguous failure, record the ambiguity and surface it for human or async reconciliation — do not silently swallow it.
3. **Confirm:** On a confirmed `charge_id`, update the record to `COMPLETE` in a second transaction.

This is the standard saga/outbox pattern for payment systems. Without it, the at-most-once guarantee does not hold.

## Critical: Lock TTL Shorter Than Maximum Processing Time

```python
ORDER_LOCK_TTL = 60        # seconds — covers typical payment round-trip
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum wall-clock time inside the lock for payment alone:

```
3 attempts × 30 s timeout + 2 gaps × 5 s backoff = 100 seconds
```

The lock TTL is 60 seconds. A slow-but-not-timing-out gateway response (e.g., 29 s per attempt) can expire the lock while Worker A is still inside `process()`. Worker B then acquires the lock, passes the idempotency check (nothing recorded yet), and begins charging. Worker A eventually succeeds and records. Worker B also succeeds and records. Two charges, one order.

**Required fix:** Either:

- Set `ORDER_LOCK_TTL` to at least `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + db_margin` with a documented derivation, **or**
- Implement lock extension (Redlock-style `PEXPIRE` heartbeat from a background thread while the lock is held), **or**
- Reduce `REQUEST_TIMEOUT` and `MAX_RETRIES` such that the math fits within the TTL with margin.

The TTL and the payment client timeout budget must be co-designed and the relationship must be enforced, not just commented.

## High: Sensitive Token Value Exposed in Log / Exception Path

```python
# payment_client.py
raise PaymentError(f"Payment declined: {e.response.status_code}") from e
```

The `PaymentError` is caught nowhere in `order_processor.py` — it propagates uncaught out of `process()`. Depending on the caller's exception handler and log aggregator, the full exception chain (including `e.response`) may be serialized. More directly:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

`requests` exceptions carry the `PreparedRequest` object, which contains the full request body including `token`. If this exception is logged at any level above `DEBUG` anywhere in the call stack, the payment token hits the log sink.

**Required fix:**

- Catch `PaymentError` explicitly in `order_processor.py` and return a structured `OrderResult` rather than letting it propagate.
- Sanitize or strip the request body from exceptions before raising or logging. At minimum, do not let `requests` exceptions propagate with their full request context attached.
- Add a test that asserts `payment_token` does not appear in any log output on failure.

## High: Idempotency Check Is Not Atomic With Lock Acquisition

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

The idempotency check (`get_charge`) runs *outside* the database transaction. If the database is under read-committed or lower isolation, a concurrent write that commits between `get_charge` and `record_charge` will not be seen. The Redis lock is supposed to prevent this, but the lock is a best-effort advisory mechanism — it does not compose with database transaction isolation. If Redis is unavailable or the lock is lost (see TTL issue above), the database is the last line of defense, and here it provides none.

**Required fix:** The idempotency check must be inside the same database transaction as the write, using `SELECT ... FOR UPDATE` or an equivalent pessimistic read, so the database engine enforces the uniqueness constraint even if the Redis lock fails.

## Medium: `PaymentError` Uncaught in `process()`

```python
except LockError:
    return OrderResult(
        processed=False,
        message="Order currently being processed by another worker",
    )
```

Only `LockError` and `ValidationError` are caught. `PaymentError`, `redis.RedisError`, any database exception, and any unexpected exception from `self.db.transaction().__exit__` all propagate to the caller as unhandled exceptions. For a service that is supposed to return structured `OrderResult` objects, this is an incomplete contract.

**Required fix:** Add explicit `except` blocks for `PaymentError` and `redis.RedisError` (and a catch-all for unexpected errors) that return structured `OrderResult` values with `processed=False` and appropriate messages, after sanitizing any sensitive fields.

## Medium: Tests Mock Away the Failure Modes That Matter

The tests verify the happy path and basic contention, but none of them test the scenarios that the architecture is actually designed to prevent:

| Scenario | Tested? |
|---|---|
| Gateway timeout followed by retry → duplicate charge | ✗ |
| Lock expires mid-processing (TTL < processing time) | ✗ |
| `PaymentError` propagation out of `process()` | ✗ |
| Redis unavailable during lock acquisition | ✗ |
| DB transaction rollback after successful charge | ✗ |
| `record_charge` raises after `payment.charge` succeeds | ✗ |

`test_lock_release_uses_lua_script` asserts that `redis.delete` was not called and that the Lua script was called once — this is a valid structural test, but it does not verify the token passed to the script matches the token stored in Redis, which is the entire security property of the token-verified release.

## Low: `RETRY_BACKOFF` Is a Flat Value, Not Exponential

The PR checklist says "retry with backoff" but `PaymentClient` uses a flat 5-second sleep between every attempt. The lock contention retry in `_acquire_lock` does use linear backoff (`0.5 * (attempt + 1)`). These should be consistent and documented. For a payment gateway, exponential backoff with jitter is the standard recommendation to avoid thundering-herd on a recovering gateway.

---

## Low: `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` Are Class-Level Constants

These values are not configurable at construction time. In any environment where the payment gateway SLA differs from the defaults (staging vs. production, different gateway vendors), the only way to change them is to subclass or monkey-patch. They should be constructor parameters with documented defaults.
