Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured and the Lua-based token-verified release is correct, but there is a critical atomicity flaw that undermines the entire at-most-once guarantee: the payment charge happens *inside* the database transaction, meaning a successful charge that is followed by a database rollback produces an unrecorded, unrecoverable charge against the customer. Additionally, the code silently swallows all exceptions that occur inside the lock (other than `LockError`), which can mask payment failures and leave the system in an ambiguous state. These two issues together represent a reliability and financial correctness risk that blocks approval.

---

## Detailed Findings

### 1. CRITICAL — Charge Inside DB Transaction Breaks At-Most-Once Guarantee

**File:** `services/order_processor.py`, lines inside `with self.db.transaction():`

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary claims: *"crash between charge and record cannot produce an unrecorded charge."* This is false. The payment gateway call is an **external, non-transactional side effect**. If `self.payment.charge()` succeeds but `record_charge()` or `mark_order_complete()` raises, the DB transaction rolls back. The charge has already been collected from the customer. On the next retry, the idempotency check finds no record and charges again.

The correct pattern is to charge *outside and before* the transaction, then record inside it. If the record step fails, you have a charge ID you can reconcile. If the process crashes after charging but before recording, your retry logic or a reconciliation job can detect the orphaned charge via the gateway's own idempotency key.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # See finding #2
)

with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

### 2. CRITICAL — No Idempotency Key Sent to Payment Gateway

**File:** `services/payment_client.py`

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

On a timeout, the client retries. But there is no idempotency key in the request body or headers. If the first request reached the gateway and succeeded but the response was lost in transit, the retry will issue a **second charge** for the same amount. This is the exact scenario the distributed lock is meant to prevent, but the lock does not protect against intra-request retries.

Fix: pass `order_id` through to `PaymentClient.charge()` and include it as an idempotency key per the gateway's API contract (commonly `Idempotency-Key` header or a body field).

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    ...
    response = requests.post(
        self.CHARGE_ENDPOINT,
        json={"token": token, "amount": amount_cents},
        headers={"Idempotency-Key": idempotency_key},
        timeout=self.REQUEST_TIMEOUT,
    )
```

### 3. HIGH — Broad Exception Suppression Inside the Lock

**File:** `services/order_processor.py`

```python
try:
    with self._acquire_lock(req.order_id):
        ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Only `LockError` is caught. Any other exception — `PaymentError`, a DB connection error, an unexpected `KeyError` from `response.json()` — propagates out of `process()` uncaught. The caller gets an unhandled exception with no `OrderResult`. This may be intentional (let the caller handle it), but it is inconsistent with the method's declared return type and the pattern used for `LockError` and `ValidationError`. At minimum, `PaymentError` should be caught and returned as a structured `OrderResult` so the lock is always cleanly released and the caller always gets a typed response.

The lock *is* released in the `finally` block, so there is no lock leak here — that part is correct. The issue is the inconsistent error contract.

---

### 4. MEDIUM — Lock TTL Can Expire During a Slow Payment Call

**File:** `services/order_processor.py` and `services/payment_client.py`

`ORDER_LOCK_TTL = 60` seconds. `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst-case payment duration: `3 × 30 + 2 × 5 = 100 seconds`. The lock expires before the payment completes. A second worker can then acquire the lock, pass the idempotency check (no record yet), and issue a duplicate charge.

Options: increase `ORDER_LOCK_TTL` to safely exceed worst-case payment time (e.g., 120–150s), or implement lock extension (Redlock-style `pexpire` renewal in a background thread). The simpler fix is adjusting the TTL with a documented calculation.

---

### 5. LOW — `payment_token` Logged Indirectly via Pydantic Validation Errors

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Sanitize or redact sensitive fields before logging, or log only field names and error types.

---

### 6. LOW — Test for Lock Release Does Not Assert Token Correctness

**File:** `tests/test_order_processor.py`

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This confirms the Lua script is called but does not verify that the correct `token` and `lock_key` are passed as `keys` and `args`. A future refactor could call the script with wrong arguments and this test would still pass. Assert on `call_args` to lock in the contract.
