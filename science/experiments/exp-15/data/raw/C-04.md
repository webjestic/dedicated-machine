Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and the idempotency check pattern are all correct. However, there is a critical atomicity flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this touches production.

---

## Critical Issue

### 1. Payment charge occurs inside the DB transaction — this does not make it atomic

The PR comment says:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

This claim is **false**, and the architecture is subtly wrong in a way that creates the opposite problem.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Problems:**

- A DB transaction does not wrap an external HTTP call. If `payment.charge()` succeeds but the DB write fails (deadlock, connection drop, rollback), the charge has already been sent to the payment gateway and money has moved — but no record exists. The next worker will pass the idempotency check and charge again. This is exactly the double-charge scenario the lock is supposed to prevent.
- Holding an open DB transaction for the duration of an HTTP call (up to `REQUEST_TIMEOUT = 30s`, up to 3 retries = 90s worst case) is a serious connection pool and lock contention hazard.

**Correct pattern:** charge first, then open the DB transaction only for the record writes. Use the idempotency check + the Redis lock together to guard the charge call, not the DB transaction.

```python
# Outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# DB transaction only wraps DB writes
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

If the DB write fails after a successful charge, the retry path will hit the idempotency check and return "already processed" — which is still wrong because the charge exists but is unrecorded. A proper recovery path (e.g., a reconciliation job or storing the `charge_id` in Redis before the DB write) is needed for that edge case, but it is far less dangerous than the current design.

## Significant Issues

### 2. Lock TTL is dangerously short relative to payment retry window

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient: MAX_RETRIES = 3, RETRY_BACKOFF = 5, REQUEST_TIMEOUT = 30
# Worst case: 3 × (30 + 5) = 105 seconds
```

The lock can expire while the payment client is still retrying. A second worker will then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge. The TTL must exceed the maximum possible payment client execution time. Set `ORDER_LOCK_TTL` to at least 120–150 seconds, or cap the payment client's total time budget to well under 60 seconds.

### 3. `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` is not caught in `process()`. It will propagate out of the context manager, which will correctly release the lock, but the caller receives an unhandled exception rather than an `OrderResult`. The PR claims structured error handling; this is missing. Catch `PaymentError` explicitly and return a structured `OrderResult(processed=False, ...)`.

### 4. No test coverage for the critical failure paths

The test suite covers the happy path and basic contention, but is missing:

- Payment gateway failure → `PaymentError` raised → verify lock is released and result is structured
- DB write failure after successful charge (the unrecorded charge scenario)
- Lock TTL expiry simulation
- `payment_token` is validated for length but not content — a test confirming sensitive token values are not logged would be appropriate given this handles payment credentials

### 5. `payment_token` appears in Pydantic model but has no format validation

`customer_id` has no pattern constraint. If this value is interpolated into any query or log downstream, it is an injection surface. Apply the same `pattern=r'^[a-zA-Z0-9_\-]+'` discipline used on `order_id`.

## Minor Issues

### 6. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

A malformed or unexpected gateway response will raise `KeyError` or `JSONDecodeError`, neither of which is caught. This will surface as an unhandled exception rather than a `PaymentError`. Wrap this in a try/except and raise `PaymentError` with context.

### 7. Lock retry backoff is linear and very short

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

For a 60-second lock TTL, this retry window (3 attempts over ~3 seconds) provides almost no benefit. Either use exponential backoff with jitter, or document that this is intentionally a fast-fail with the expectation that the caller retries at a higher level.

## What Is Done Well

- `SET NX EX` is the correct atomic acquisition primitive — no SETNX+EXPIRE race condition
- Lua script for token-verified release is correct and well-scoped
- Pydantic validation with regex on `order_id` is appropriate
- Structured logging without leaking sensitive values in the main flow
- The idempotency check placement (inside the lock) is correct
