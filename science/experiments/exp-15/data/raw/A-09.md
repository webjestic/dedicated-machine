# Pull Request Review

## `services/order_processor.py`

### Critical: Payment Outside Database Transaction — At-Most-Once Guarantee Is False

The PR description claims "crash between charge and record cannot produce an unrecorded charge." This is incorrect. The `payment.charge()` call is *inside* `with self.db.transaction()`, but a database transaction provides no rollback guarantee over an external HTTP call. If `charge()` succeeds and the process crashes before `db.record_charge()` commits, the charge exists in the payment gateway with no local record. On retry, the idempotency check finds no existing charge and charges again.

```python
# As written — charge happens inside db.transaction(), but that buys nothing
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect — not transactional
    self.db.record_charge(...)             # if this never commits, charge is orphaned
    self.db.mark_order_complete(...)
```

The correct pattern is to record *intent* before charging, then reconcile:

```python
# Safer ordering
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

With this pattern, a crash after `record_charge_pending` but before `payment.charge` leaves a pending record that a reconciliation job can inspect. A crash after `payment.charge` but before `finalize_charge` leaves a pending record with a known charge ID recoverable from the gateway. The idempotency check must also gate on the *pending* record, not just a completed one.

### Critical: Lock TTL Does Not Cover the Critical Section

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can consume up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The lock expires before the payment client gives up, allowing a second worker to acquire the lock, pass the idempotency check (no charge recorded yet), and issue a duplicate charge.

```python
# Worst-case payment client runtime
MAX_RETRIES = 3
REQUEST_TIMEOUT = 30   # per attempt
RETRY_BACKOFF = 5      # between attempts
# Total: 3*30 + 2*5 = 100s (two sleeps between three attempts)
# Lock TTL: 60s — expires during attempt 2 or 3
```

The TTL must be set to a value strictly greater than the maximum observable critical section duration, with margin. Either raise `ORDER_LOCK_TTL` to at least 120–150 seconds, or reduce `PaymentClient` timeouts and retries to fit within the TTL. Both values must be owned together and documented as a constraint.

### Moderate: `PaymentError` Is Silently Swallowed

`PaymentError` is not caught in `process()`. If the payment client raises it, the exception propagates out of the `with self._acquire_lock(...)` block, which correctly releases the lock via `finally`, but the caller receives an unhandled exception rather than an `OrderResult`. The contract of `process()` is to return `OrderResult` in all cases.

```python
# Missing handler
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

### Moderate: Idempotency Check Is Not Atomic With Lock Acquisition

The idempotency check (`db.get_charge`) runs after the lock is acquired, which is correct. However, the check reads from the database outside the transaction that would write the result. Under read-committed isolation, a concurrent transaction that committed between the lock acquisition and this read would be visible, which is the desired behavior. But if the database uses snapshot isolation or the connection is on a replica with replication lag, the check could return stale data. This should be documented as a deployment assumption, or the check should be moved inside the transaction with a `SELECT FOR UPDATE` or equivalent.

---

### Minor: `customer_id` Has No Pattern Constraint

`order_id` enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` has only length bounds. If `customer_id` is used in any downstream query construction or log aggregation, the absence of a pattern constraint is a latent injection surface. Apply the same or a similarly restrictive pattern.

---

### Minor: `payment_token` Logged at INFO on Success

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is safe, but `payment_token` is passed in the request body. Confirm no log line anywhere in the call stack emits `payment_token`. The Pydantic model should also mark it as a `SecretStr` to prevent accidental serialization.

## `services/payment_client.py`

### Moderate: Retry on Timeout Assumes Idempotency the Gateway May Not Provide

Retrying on `requests.exceptions.Timeout` assumes the gateway did not process the request. If the gateway processed the charge but the response was lost in transit, retrying issues a duplicate charge. This is only safe if the gateway endpoint is idempotent (e.g., accepts an idempotency key). No idempotency key is sent in the current implementation.

```python
# No idempotency key — retry on timeout is unsafe without it
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

The fix is to generate a stable idempotency key per logical charge attempt (e.g., derived from `order_id`) and include it as a header:

```python
headers={"Idempotency-Key": idempotency_key}
```

---

### Minor: `requests.exceptions.ConnectionError` Not Handled

Network-level failures (DNS failure, connection refused, TCP reset) raise `ConnectionError`, not `Timeout` or `HTTPError`. These are currently unhandled and will propagate as unretried exceptions despite being transient. Add a `ConnectionError` handler alongside `Timeout`.

## `tests/test_order_processor.py`

### Moderate: No Test for Lock Expiry During Payment

There is no test that verifies behavior when the lock TTL expires while payment is in flight. This is the most dangerous failure mode in the design and has no coverage.

### Moderate: No Test for `PaymentError` Propagation

Given that `PaymentError` is currently unhandled (see above), there is no test that would catch this gap. A test that mocks `payment.charge` to raise `PaymentError` and asserts `result.processed is False` would have caught the missing handler.

### Minor: Transaction Context Manager Mock Is Fragile

```python
mock.transaction.return_value.__enter__ = MagicMock(return_value=None)
mock.transaction.return_value.__exit__ = MagicMock(return_value=False)
```

This does not verify that `record_charge` and `mark_order_complete` are called *within* the transaction context. The test `test_charge_recorded_within_transaction` asserts both are called, but not that they are called between `__enter__` and `__exit__`. Use `unittest.mock.call` ordering assertions or a custom context manager mock that tracks call sequence.

**Decision: Request Changes**

**Summary:** Two critical flaws block approval: the lock TTL (60 s) is shorter than the maximum payment client execution time (100 s), creating a window for duplicate charges under retry; and placing `payment.charge()` inside a database transaction does not make it atomic — a crash after the external charge succeeds but before the DB commit produces an orphaned charge that will be duplicated on retry. Additionally, `PaymentError` is unhandled in `process()`, retrying on gateway timeout without an idempotency key risks duplicate charges at the network layer, and test coverage does not exercise the most dangerous failure paths. All critical and moderate issues must be resolved before this can be approved.
