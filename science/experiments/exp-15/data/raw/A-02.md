# Pull Request Review

## Distributed Order Processor with Redis-Backed Exclusive Locking

---

### Preliminary Assessment

The PR demonstrates solid foundational instincts: atomic `SET NX EX`, token-verified Lua release, Pydantic input validation, and idempotency checks are all present and structurally sound. However, tracing every lock acquisition boundary and TTL assumption reveals several correctness and security issues that must be resolved before this touches production.

---

## Critical Issues

### 1. Payment Charge Lives Outside the Database Transaction — At-Most-Once Guarantee is False

This is the most severe architectural flaw. The PR comment claims:

```python
# Crash between charge and record cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← NETWORK CALL TO EXTERNAL SYSTEM
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Wrapping `self.payment.charge()` inside `self.db.transaction()` does **nothing** to make the payment atomic with the database write. A database transaction controls the database connection — it has zero rollback authority over an already-committed external HTTP call to the payment gateway.

The actual failure scenario:

1. `payment.charge()` succeeds — money is taken from the customer
2. `db.record_charge()` raises an exception (deadlock, constraint violation, network partition to DB)
3. `db.transaction().__exit__` rolls back the DB write
4. The system has no record of the charge, but the charge exists at the gateway
5. On retry, `db.get_charge()` returns `None`, the idempotency check passes, and the customer is charged again

The comment `# Crash between charge and record cannot produce an unrecorded charge` is factually incorrect and dangerous. This is a **double-charge vector**.

**Required fix:** Adopt an outbox pattern or a two-phase approach:

```python
# Phase 1: Write a pending charge record BEFORE calling the gateway
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Phase 2: Call gateway — if this succeeds but DB write below fails,
# a reconciliation job can detect pending charges and resolve them
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# Phase 3: Confirm the charge in the DB
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

Alternatively, require that the payment gateway supports idempotency keys (most do), and pass `req.order_id` as the idempotency key so that a retry to the gateway returns the existing charge rather than creating a new one.

### 2. Redis Lock TTL Does Not Account for Payment Retry Duration

```python
ORDER_LOCK_TTL = 60        # seconds — covers typical payment round-trip
```

```python
REQUEST_TIMEOUT = 30      # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5         # seconds between retries on transient failure
```

Maximum payment client wall time: `(30s × 3 attempts) + (5s × 2 backoff intervals)` = **100 seconds**.

The lock TTL is 60 seconds. A slow but legitimate payment attempt will cause the lock to expire at ~60s while the worker is still in `payment.charge()` on attempt 2 or 3. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. This directly re-opens the double-charge race the lock was designed to prevent.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at minimum `(REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES - 1)) + safety_margin` (≥120s), or implement lock extension (Redlock-style `PEXPIRE` refresh) during long operations. These two constants must be defined with an explicit relationship, not independently.

### 3. Timeout Responses from Payment Gateway Are Retried Without Idempotency — Duplicate Charge Risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to retry
```

A `Timeout` exception means the HTTP response was not received — it does **not** mean the charge did not execute. The gateway may have processed the charge and the response was lost in transit. Retrying without an idempotency key will submit a second charge request that the gateway treats as a new, independent charge.

**Required fix:** Pass a stable idempotency key derived from `order_id` in every request:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be passed in
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, every timeout retry is a potential duplicate charge.

## Significant Issues

### 4. `payment_token` Reflected Into Log Output on Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic `ValidationError` string representations include field values in some versions and configurations. `payment_token` is a payment credential. If it appears in a validation error (e.g., it fails the `max_length=256` check), it will be written to log storage in plaintext. Log aggregation systems (Splunk, ELK, CloudWatch) are typically lower-security than payment systems and may be accessible to personnel who should not see payment tokens.

**Required fix:** Sanitize the validation error before logging, or log only field names and error types, never values:

```python
except ValidationError as e:
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

### 5. Lock Is Released After Returning Inside the `with` Block — Correct, But Fragile

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        return OrderResult(...)   # ← returns here; lock released in finally
```

The `finally` block in `_acquire_lock` does correctly fire on `return`. This is not a bug today, but the pattern is fragile: any future refactor that moves the `return` outside the `with` block, or that wraps the context manager in a way that suppresses the `finally`, will silently break lock release. This is worth a comment.

---

### 6. Lock Retry Backoff Is Blocking — Entire Worker Thread Stalls

```python
time.sleep(backoff)
```

With `LOCK_RETRY_ATTEMPTS = 3` and backoffs of 0.5s, 1.0s, the worst-case blocking time before raising `LockError` is 1.5 seconds of thread sleep. In a synchronous web worker (gunicorn sync workers, for example), this holds the worker thread and reduces throughput under contention. For an async framework this would be a blocking call on the event loop.

This is an acceptable tradeoff in some deployments, but it must be a conscious decision. At minimum, document it. If this runs in an async context, replace with `asyncio.sleep`.

---

### 7. `customer_id` Has No Pattern Validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has `pattern=r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. `customer_id` is written to the database via `db.record_charge()`. If the database layer uses string interpolation anywhere (not visible in this PR), this is a SQL injection surface. Even if parameterized queries are used throughout, consistency in input validation is a defense-in-depth requirement.

---

### 8. No Handling of Unhandled Exceptions Within the Lock

```python
with self._acquire_lock(req.order_id):
    # ... DB calls, payment calls ...
    # No except for unexpected exceptions
```

If `self.db.get_charge()` raises an unexpected exception (connection pool exhausted, unexpected schema error), it propagates out of `process()` entirely uncaught. The lock is released correctly by `finally`, but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract of `process()` returning `OrderResult` in all cases, and may expose internal stack traces to callers depending on the web framework's error handling.

## Test Coverage Gaps

### 9. No Test for Lock TTL Expiry During Payment

There is no test simulating the lock expiring mid-payment. This is the exact failure mode identified in Issue #2 and it is untested.

### 10. No Test for Payment Success Followed by DB Write Failure

The critical double-charge scenario (Issue #1) has no test coverage. A test should exist that verifies behavior when `payment.charge()` succeeds but `db.record_charge()` raises — and that test should currently fail, which would have surfaced the architectural flaw.

### 11. No Test for Timeout Retry Behavior in PaymentClient

`test_order_processor.py` does not test `PaymentClient` at all. The retry-on-timeout logic (Issue #3) has no test coverage.
