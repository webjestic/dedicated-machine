Decision: Request Changes

Summary: The distributed locking design contains a critical TOCTOU gap that breaks the at-most-once guarantee it claims to provide: the idempotency check (`db.get_charge`) runs *outside* the database transaction, meaning two workers can both read `None`, both pass the check, and both proceed to charge — the Redis lock is the only guard, and it has a failure mode that eliminates that guard entirely. Specifically, if the payment call takes longer than `ORDER_LOCK_TTL` (60 s) the lock expires, a second worker acquires it, passes the idempotency check (charge not yet recorded because the first worker is still in `payment.charge()`), and issues a duplicate charge before the first worker's `db.transaction()` commits. Beyond that race, the payment client's retry-on-timeout behavior compounds the problem: a `Timeout` exception does not mean the charge was not applied — the gateway may have processed it and the response was simply lost — so retrying on timeout can itself produce duplicate charges at the gateway level, independent of the locking layer. These two flaws together mean the system can double-charge a customer under realistic production conditions.

---

## Detailed Findings

### 1. Lock TTL / Long Payment Call Race (Critical — Correctness)

```python
# order_processor.py
ORDER_LOCK_TTL = 60   # seconds

with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← read outside transaction
    if existing:
        ...

    with self.db.transaction():
        charge_id = self.payment.charge(...)       # ← can exceed 60 s
        self.db.record_charge(...)
```

The payment client allows up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** of wall time. The lock TTL is 60 seconds. Worker A's lock expires during the payment call. Worker B acquires the lock, reads `get_charge` → `None` (nothing recorded yet), and enters `payment.charge()`. Both workers now hold an active charge attempt against the gateway simultaneously.

**Required fix:** Either (a) set `ORDER_LOCK_TTL` strictly greater than the maximum possible payment round-trip time and document that invariant with an assertion, or (b) move the idempotency check *inside* the database transaction with a `SELECT FOR UPDATE` on the order row so the database serializes the check-and-record regardless of lock state. Option (b) is strongly preferred because it makes correctness a database-level guarantee rather than a timing assumption.

```python
# Preferred pattern
with self.db.transaction():
    existing = self.db.get_charge_for_update(req.order_id)  # SELECT FOR UPDATE
    if existing:
        return OrderResult(processed=False, message="Order already processed")
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

### 2. Retry-on-Timeout Produces Gateway-Level Duplicates (Critical — Correctness)

```python
# payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to retry loop
```

A network timeout is *not* evidence that the charge was not applied. The gateway may have processed the charge and the response was dropped in transit. Retrying unconditionally produces a second charge at the gateway for the same payment token and amount. This is a well-known failure mode in payment integrations.

**Required fix:** The gateway must expose an idempotency key mechanism. Pass a stable, per-order idempotency key on every attempt so the gateway deduplicates retries server-side. If the gateway does not support idempotency keys, the only safe behavior on timeout is to *query* the charge status before retrying, not to issue a new charge blindly.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # stable per order_id
    timeout=self.REQUEST_TIMEOUT,
)
```

### 3. Idempotency Check Outside Transaction (High — Correctness)

Even setting aside the TTL race, the `get_charge` read and the `record_charge` write are not atomic with respect to each other. Under snapshot isolation or read-committed databases, two concurrent transactions can both observe `get_charge` → `None` before either commits. The idempotency check must be inside the transaction and must use a locking read or a unique constraint at the database level to be meaningful.

**Required fix:** Unique constraint on `(order_id)` in the charges table as a backstop, plus the `SELECT FOR UPDATE` pattern above. The constraint turns a logic race into a caught integrity error rather than a silent double-charge.

### 4. Lock Release Swallows Errors Silently (Medium — Observability / Correctness)

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis unavailable, network partition), the exception is propagated out of `__exit__` and will mask any exception already in flight from the `try` block. More importantly, a return value of `0` from the Lua script means the lock was already expired or stolen — the log line says "released" regardless, which is misleading and hides the TTL expiry scenario described in finding #1.

**Required fix:**

```python
finally:
    result = self._release_script(keys=[lock_key], args=[token])
    if result == 0:
        logger.error(
            "Lock release returned 0 — lock expired or stolen order_id=%s; "
            "possible duplicate processing window", order_id
        )
    else:
        logger.info("Order lock released order_id=%s", order_id)
```

### 5. `payment_token` Reflected Into Logs (Medium — Security)

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their default string representation. If `payment_token` fails validation, its value (a payment method token) may appear in log output. Payment tokens must never appear in logs.

**Required fix:** Sanitize or suppress field values in validation error logging, or catch the error per-field and log only field names.

```python
except ValidationError as e:
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
    return OrderResult(processed=False, message="Invalid order parameters")
```

---

### 6. `customer_id` Has No Pattern Constraint (Low — Defense in Depth)

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. If `customer_id` is used in any downstream query or log aggregation, unsanitized values are a latent injection surface.

**Required fix:** Apply the same or equivalent pattern constraint to `customer_id`.

---

### 7. Test Coverage Gaps (Low — Test Quality)

The test suite does not cover:

- Lock TTL expiry during payment (the primary failure mode identified above).
- Timeout retry behavior in `PaymentClient` — specifically that a timeout followed by a retry does not issue two charges.
- `record_charge` raising an integrity error after a successful `payment.charge()` call (charge issued, record fails — orphaned charge scenario).
- Lock release returning `0` (expired lock path).

These are the exact scenarios where the bugs above manifest. Tests that only cover the happy path and basic contention do not validate the safety properties the PR claims to provide.
