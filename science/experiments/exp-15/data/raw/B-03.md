Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several correctness-critical flaws that each independently can produce the exact failure mode this service is designed to prevent: a double charge or an unrecorded charge. None of these are edge cases; they are reachable under normal concurrent or failure conditions.

---

## Critical Flaws

### 1. Payment call is inside the database transaction — timeout window exceeds lock TTL

This is the most dangerous issue in the PR.

```python
# ORDER_LOCK_TTL = 60 seconds
# PaymentClient.REQUEST_TIMEOUT = 30 seconds
# PaymentClient.MAX_RETRIES = 3
# PaymentClient.RETRY_BACKOFF = 5 seconds
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. If the payment call takes longer than 60 seconds, the lock expires while the worker is still inside the critical section. A second worker can then acquire the lock, pass the idempotency check (the charge has not been recorded yet because the first worker's transaction is still open), and issue a second charge. This is a double-charge race condition that the lock was specifically designed to prevent.

**Fix:** Either:
- Reduce `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES-1) × RETRY_BACKOFF` to be strictly less than `ORDER_LOCK_TTL` with margin, or
- Extend the lock TTL to safely exceed worst-case payment duration, or
- Move the payment call outside the database transaction and use a two-phase approach: charge → record charge_id → commit, with a compensating refund path on DB failure.

The third option is architecturally correct regardless, because holding a DB transaction open across a network call to an external payment gateway is its own problem (connection pool exhaustion, long-held row locks).

---

### 2. Unhandled `PaymentError` propagates through the lock context manager without being caught

```python
with self._acquire_lock(req.order_id):
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError
        self.db.record_charge(...)
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which correctly releases the lock via `finally`. However, it then surfaces as an unhandled exception to the caller with no `OrderResult` returned. The caller receives a raw exception instead of a structured result, breaking the contract of the method signature (`-> OrderResult`).

**Fix:** Catch `PaymentError` inside `process()` and return a structured `OrderResult(processed=False, message=...)`.

---

### 3. Idempotency check is not durable — it reads from the database, not from a committed write

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(processed=False, ...)
```

If two workers both acquire the lock sequentially (first worker releases after lock TTL expiry per flaw #1), both will call `get_charge`. The second worker will only see the first worker's charge if the first worker's transaction committed. If the first worker's transaction is still in-flight or rolled back after a partial failure, the second worker proceeds to charge again.

This is a consequence of flaw #1, but it is worth stating explicitly: the idempotency check is only as strong as the guarantee that a prior charge is always committed before the lock is released. That guarantee does not hold here.

---

### 4. Lock release on exception swallows the release result silently

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (e.g., Redis connection lost during release), the exception is suppressed by the `finally` block completing normally, and the log line prints "released" even though the lock was not released. The lock then holds until TTL expiry. This is acceptable for correctness (TTL will eventually expire it), but the log message is actively misleading for incident response.

**Fix:** Wrap the release call, log the return value (0 means the token did not match — which should never happen but is worth alerting on), and log a warning on exception rather than an info-level "released" message.

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error("Lock release token mismatch — lock may have expired order_id=%s", order_id)
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s — will expire at TTL", order_id)
```

---

### 5. `payment_token` is logged indirectly via `payment_token` field presence in `OrderRequest`

Not a current log statement, but `OrderRequest` is a Pydantic model and if any calling code logs `req.dict()` or `str(req)` (common in debug paths), `payment_token` will appear in plaintext in log output. At a government contracting firm this is a PCI-scope concern.

**Fix:** Override `__repr__` and `__str__` on `OrderRequest`, or mark `payment_token` with a custom type that masks its value in serialization.

## Minor Issues

**Test coverage gap — lock TTL expiry race is not tested.** The test suite covers contention (lock not acquired) but not the case where the lock expires mid-processing. A test simulating TTL expiry and a concurrent second acquisition would have caught flaw #1.

**`RETRY_BACKOFF` is a flat constant, not exponential.** The payment client uses a flat 5-second sleep between all retries. Under sustained gateway degradation this produces a thundering herd from all workers retrying in lockstep. Jittered exponential backoff is standard practice.

**`requests.post` has no connection timeout, only a read timeout.** `timeout=30` in the requests library sets the read timeout. A hung connection at the TCP layer will block indefinitely. Pass `timeout=(connect_timeout, read_timeout)` as a tuple.

```python
timeout=(5, self.REQUEST_TIMEOUT)
```

**`charge_id = response.json()["charge_id"]`** — no guard against a malformed response. A `KeyError` here would propagate as an unhandled exception rather than a `PaymentError`.

---

## What Is Done Well

- `SET NX EX` is the correct atomic acquisition primitive; no TOCTOU on lock acquisition.
- The Lua script for token-verified release is correct and necessary.
- Pydantic pattern enforcement on `order_id` correctly blocks path traversal and injection attempts.
- `db.transaction()` as a context manager is the right shape for atomic charge recording.
- Structured logging with key=value pairs throughout.
- Test for Lua script usage (asserting `redis.delete` is not called) is a good negative assertion.
