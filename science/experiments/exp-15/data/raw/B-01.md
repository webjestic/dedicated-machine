# Pull Request Review

## Distributed Systems Analysis: `feature/order-processor`

I'll trace every lock boundary, TTL assumption, and cross-service dependency before rendering a verdict.

---

## Critical Findings

### 🔴 CRITICAL — Payment Inside DB Transaction Violates Atomicity Guarantee

This is the most serious flaw in the design. The PR summary claims:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**. The code does this:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External network call — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over **database operations only**. `self.payment.charge()` is an external HTTP call to a payment gateway. It is **outside the transaction boundary** by definition.

The failure modes this creates:

**Scenario A — Charge succeeds, DB write fails:**
The payment gateway has captured funds. The DB transaction rolls back. `record_charge` never commits. The order appears unprocessed. On retry, the idempotency check finds no existing charge and **charges the customer again**. This is a double-charge vulnerability.

**Scenario B — Charge succeeds, process crashes before DB commit:**
Same outcome as Scenario A. The Redis lock TTL expires, another worker acquires the lock, idempotency check returns clean, and the customer is charged twice.

**Scenario C — Timeout ambiguity:**
`PaymentClient` has a 30-second `REQUEST_TIMEOUT`. `ORDER_LOCK_TTL` is 60 seconds. A payment call that times out after 29 seconds, then retries twice with 5-second backoffs, consumes `29 + 5 + 29 + 5 + 29 = 97 seconds` — **exceeding the 60-second lock TTL**. The lock expires mid-retry. A second worker acquires the lock, passes the idempotency check (no charge recorded yet), and initiates a second charge while the first worker's final retry may still be in-flight.

**The correct pattern** is charge-then-record with idempotency anchored to the payment gateway's own idempotency key, not just a DB lookup:

```python
# Correct architecture sketch
idempotency_key = f"{order_id}:{customer_id}"  # stable, deterministic
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,  # gateway deduplicates on its side
)
# Only reach DB write if charge returned a charge_id
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The idempotency check must also verify against the **payment gateway**, not only the local DB.

### 🔴 CRITICAL — Lock TTL vs. Payment Retry Budget Mismatch

Quantified above in Scenario C, but worth isolating as its own finding:

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Worst-case payment client wall time: `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF`
= `3 × 30 + 2 × 5` = **100 seconds**

The lock TTL is 60 seconds. These two constants are defined in separate files with no cross-reference, no enforcement, and no comment linking them. This is a latent race condition that will manifest under load when the gateway is slow.

The lock TTL must be set to strictly greater than the maximum possible payment client execution time, with that constraint made explicit and enforced:

```python
# These must be kept consistent — lock must outlive payment client worst case
MAX_PAYMENT_WALL_TIME = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
ORDER_LOCK_TTL = MAX_PAYMENT_WALL_TIME + 30  # 30s safety margin
```

### 🔴 CRITICAL — Unhandled `PaymentError` Escapes Lock Context Manager

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError
        self.db.record_charge(...)
```

`PaymentError` is not caught anywhere in `process()`. It propagates up through the `_acquire_lock` context manager (which correctly releases the lock in `finally`), but then **escapes `process()` entirely** as an unhandled exception.

The caller receives an uncaught exception rather than an `OrderResult`. The PR's own return type annotation is `OrderResult` — this contract is violated. Depending on the caller (e.g., an HTTP handler), this could produce a 500 with a raw stack trace leaking internal infrastructure details (`https://api.payments.internal/v1/charges`, internal charge IDs, etc.).

`PaymentError` must be caught and mapped to a structured `OrderResult`:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

### 🟡 HIGH — Idempotency Check Has a Read-After-Lock Race on DB Replica Lag

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        return ...
```

If `self.db` routes reads to a replica (common in production DB setups), a charge committed by a previous worker may not yet be visible due to replication lag. The idempotency check returns `None`, the new worker proceeds to charge, and a duplicate charge occurs despite the lock.

The idempotency read **must** target the primary/leader node, or use a `SELECT FOR UPDATE` / `READ YOUR OWN WRITES` consistency guarantee. This must be documented as an explicit architectural constraint even if the current DB client happens to read from primary.

### 🟡 HIGH — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., too long, unexpected characters), its value may appear in the log output. Payment tokens are sensitive credentials. This is a PCI-DSS relevant finding.

The log message must not include raw field values:

```python
logger.warning("Invalid order request fields: %s", [e["loc"] for e in e.errors()])
```

---

### 🟡 MEDIUM — `payment_token` Present in `OrderRequest` Model Without Sanitization Boundary

`payment_token` flows from the API boundary through Pydantic into `process()` and then directly into `payment.charge()`. There is no masking, no truncation in logs, and the field has only a length constraint. If any logging middleware serializes the `OrderRequest` object (e.g., request tracing, audit logs), the raw token will appear in logs. The token field should be marked with a custom `__repr__`/`__str__` that masks the value.

---

### 🟡 MEDIUM — Lock Retry Backoff Does Not Include Jitter

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — deterministic
```

Under concurrent load, multiple workers contending on the same order will sleep for identical durations and retry simultaneously, causing a thundering herd on lock re-acquisition. Standard practice is additive or multiplicative jitter:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.5)
```

---

### 🟡 MEDIUM — `requests.exceptions.ConnectionError` Not Handled in `PaymentClient`

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
```

`requests.exceptions.ConnectionError` (DNS failure, TCP reset, refused connection) is not caught. It propagates as an unhandled exception, bypassing the retry logic entirely. A transient network partition would not be retried. This should be caught alongside `Timeout`:

```python
except (requests.exceptions.Timeout, requests.exceptions.ConnectionError):
    last_error = "connection error"
```

### 🟢 LOW — Test Coverage Gaps

The test suite validates the happy path and basic contention, but is missing coverage for the critical failure modes identified above:

| Missing Test Case | Why It Matters |
|---|---|
| `PaymentError` raised inside lock | Verifies exception doesn't escape `process()` |
| Lock TTL expiry during payment retry | Validates the race condition is detected |
| `payment.charge()` succeeds but `record_charge()` raises | Verifies no silent double-charge on retry |
| `get_charge()` returns stale `None` then charge is attempted | Replica lag simulation |
| `ValidationError` with sensitive token value | Verifies token not leaked to logs |

The existing test `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` — this is a valid negative assertion but does not verify the Lua script was called with the **correct token**, only that it was called once. A test that verifies token binding would be stronger.
