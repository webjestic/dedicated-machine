# Pull Request Review

## `services/order_processor.py`

### Critical: Charge Outside Transaction Boundary — At-Most-Once Guarantee Is False

The PR's central safety claim is that a crash between charge and record "cannot produce an unrecorded charge." That claim is incorrect.

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← external network call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an HTTP call to an external payment gateway. It is not a participant in the database transaction. The `db.transaction()` context manager controls only the database connection. The payment gateway has no rollback hook. The scenarios that break the guarantee:

**Scenario A — Charge succeeds, DB commit fails:**
The gateway debits the customer. The database transaction rolls back. `record_charge` is never persisted. The order appears unprocessed. The next worker acquires the lock, passes the idempotency check (no record exists), and charges again. The customer is double-charged.

**Scenario B — Charge succeeds, process crashes before DB commit:**
Same outcome as A.

**Scenario C — Gateway timeout on retry:**
`PaymentClient.charge()` retries up to 3 times on timeout. The first attempt may have succeeded at the gateway but the response was lost in transit. The retry issues a second charge request with the same token. Whether this produces a duplicate charge depends entirely on whether the payment gateway enforces idempotency on its end — there is no idempotency key being sent in the request body, so there is no contractual guarantee.

The correct architecture separates these concerns:

```
1. Charge the gateway → receive charge_id
2. Exit the lock-protected section only after DB record is committed
3. If DB commit fails after a successful charge, a reconciliation/recovery
   job must detect the orphaned charge via the gateway's own record
```

Or, if the gateway supports idempotency keys:

```python
requests.post(..., json={
    "token": token,
    "amount": amount_cents,
    "idempotency_key": order_id,   # gateway deduplicates on its side
})
```

Neither pattern is present here.

### Critical: Lock TTL vs. Payment Retry Window — Lock Can Expire During Processing

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5        # seconds between retries
```

Worst-case payment client execution time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. A payment call that exhausts all retries takes up to 100 seconds. The lock expires at second 60. A second worker acquires the lock at second 61, passes the idempotency check (nothing is recorded yet), and begins charging. The first worker may still be mid-retry. Two charges are now in flight simultaneously against the same order.

The TTL must be strictly greater than the maximum possible payment client execution time, with margin. Either:

- Raise `ORDER_LOCK_TTL` to at minimum 120 seconds (preferably 150+), or
- Implement lock extension (heartbeat/refresh) during long-running operations, or
- Reduce `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES - 1)` to fit within TTL

### Moderate: Lock Released Before Caller Observes Result

```python
with self._acquire_lock(req.order_id):
    ...
    return OrderResult(processed=True, ...)
# lock is released here, in __exit__, AFTER the return value is constructed
# but BEFORE the caller's stack frame receives it
```

This is actually fine in Python's execution model — `finally` runs before the caller receives the return value — but the lock is released immediately after the DB transaction commits and before the HTTP response is sent to the upstream caller. If the upstream caller retries on a dropped connection, the second request will find no lock and no DB record in the window between lock release and the upstream receiving the first response. This is a narrow but real TOCTOU window in systems where the caller cannot guarantee exactly-once delivery. Worth documenting explicitly.

---

### Moderate: `LockError` Is Silently Swallowed on Unexpected Exceptions

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Any other exception raised inside `_acquire_lock` or `process` — `redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`, a DB exception, a `PaymentError` — propagates uncaught to the caller. There is no top-level handler. The caller receives an unhandled exception rather than a structured `OrderResult`. This is inconsistent with the stated contract of the method and will produce unformatted 500 errors in a web context. At minimum, unexpected exceptions should be caught, logged with full traceback, and returned as a structured failure result.

---

### Minor: `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is accepted as input and passed directly to `self.payment.charge()`. Confirm that neither `PaymentClient` nor any middleware logs the raw token. The Pydantic model does not mark it as a secret type. In production, `payment_token` should be treated as a credential.

## `services/payment_client.py`

### Moderate: No Idempotency Key on Retry

As noted above, retrying `POST /v1/charges` with the same body but no idempotency key is not safe unless the gateway guarantees idempotency by token+amount. This is an assumption that must be verified and documented, or an idempotency key must be added.

```python
# Current — unsafe if gateway does not deduplicate by token
json={"token": token, "amount": amount_cents}

# Safer — pass caller-controlled idempotency key
json={"token": token, "amount": amount_cents, "idempotency_key": idempotency_key}
```

The `charge()` method signature would need to accept `idempotency_key` as a parameter, supplied by the caller with the `order_id`.

---

### Minor: `requests.exceptions.ConnectionError` Not Handled

`requests.post()` can raise `requests.exceptions.ConnectionError` (DNS failure, refused connection, etc.). This is not caught. It will propagate out of `charge()` as an untyped exception rather than a `PaymentError`, breaking the stated contract of the method.

## `tests/test_order_processor.py`

### The Tests Do Not Cover the Critical Failure Modes

The test suite validates the happy path and basic contention, but none of the critical scenarios identified above are tested:

| Scenario | Test Present |
|---|---|
| DB commit fails after successful charge | No |
| Lock expires mid-payment (TTL race) | No |
| Payment timeout triggers retry → duplicate charge | No |
| `redis.ConnectionError` during lock acquisition | No |
| `PaymentError` propagates uncaught | No |
| `ConnectionError` from `requests` in payment client | No |

The existing tests mock at too high a level to catch the transaction boundary flaw — `db.transaction()` is mocked as a no-op context manager, so the test cannot distinguish between "charge inside transaction" and "charge outside transaction."

---

## Summary of Findings

| Severity | Issue |
|---|---|
| Critical | Charge occurs inside `db.transaction()` but is not transactional — DB rollback after successful charge produces unrecorded charge and enables double-charge |
| Critical | Lock TTL (60s) shorter than maximum payment retry window (100s) — lock expires during processing, second worker can acquire lock and charge concurrently |
| Moderate | Payment client retries without idempotency key — gateway-level deduplication is assumed but not enforced |
| Moderate | Non-`LockError` exceptions propagate unhandled — inconsistent error contract |
| Moderate | `ConnectionError` not caught in `PaymentClient.charge()` |
| Minor | Test suite does not cover any of the critical failure scenarios |

---

**Decision: Request Changes**

**Summary:** The at-most-once payment guarantee stated in the PR description does not hold. The payment gateway call is not a participant in the database transaction, so a DB rollback after a successful charge leaves an unrecorded debit and the next worker will charge again. Independently, the lock TTL of 60 seconds is shorter than the maximum payment client execution time of 100 seconds, creating a window where a second worker acquires the lock on the same order while the first is still mid-retry. Both issues must be resolved before this can be approved.
