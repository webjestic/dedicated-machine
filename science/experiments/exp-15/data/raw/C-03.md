Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical atomicity flaw that undermines the at-most-once guarantee: `payment.charge()` is called *inside* `db.transaction()`, which means a successful charge that is followed by a DB commit failure will produce an unrecorded, unrecoverable charge against the customer. Additionally, there are several secondary issues around error handling, observability, and test coverage that need to be addressed before this is production-ready.

---

## Critical Issue

### 1. Payment call inside DB transaction breaks atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The docstring claims *"crash between charge and record cannot produce an unrecorded charge"* — this is false. If `record_charge` or `mark_order_complete` raises, or if the DB commit itself fails, the charge has already been submitted to the payment gateway and cannot be rolled back. You now have a charged customer with no internal record.

The correct pattern is to separate the external call from the DB write:

```python
# Correct — charge first, then record durably
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# If this block fails, the charge_id is logged above and can be reconciled
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "CHARGE_UNRECORDED order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

This does not eliminate the failure window entirely — distributed systems cannot guarantee that — but it makes the failure mode *visible and recoverable* rather than silent. The idempotency check on re-entry will also need to handle the case where a charge exists in the gateway but not yet in the DB (a reconciliation record or a pending state would help here).

## Secondary Issues

### 2. `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` and any unexpected exception escape the `_acquire_lock` context manager but are never caught in `process()`. The lock *will* be released correctly (the `finally` block handles that), but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract implied by the return type.

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed")
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, order_id=req.order_id, message="Internal error")
```

### 3. Timeout on payment gateway is too long relative to the Redis lock TTL

`ORDER_LOCK_TTL = 60s` and `REQUEST_TIMEOUT = 30s` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5s`. Worst-case payment duration: 3 × 30s + 2 × 5s = **100 seconds**, which exceeds the lock TTL by 40 seconds. A second worker can acquire the lock after 60 seconds and begin processing the same order while the first worker is still waiting on a payment response.

Either increase `ORDER_LOCK_TTL` to safely exceed the worst-case payment duration (e.g., 120s), or reduce the payment client's total retry budget. Document whichever bound you choose and why.

### 4. `payment_token` is logged at INFO level in `PaymentClient`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine, but verify that `token` is never interpolated into any log message in the payment client. A quick scan shows it is not currently, but the `PaymentError` message includes the HTTP status code which is acceptable. Add a note or a linter rule to keep the token out of logs — it is a sensitive credential.

### 5. `requests` session is not reused

`PaymentClient.charge` calls `requests.post` directly on each invocation, which means a new TCP connection (and TLS handshake) is established per call and per retry. Use a `requests.Session` configured at construction time. This also allows proper connection pooling and makes it straightforward to inject auth headers or mTLS certificates in one place.

```python
class PaymentClient:
    def __init__(self):
        self._session = requests.Session()
        # configure headers, certs, auth here
```

### 6. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()  # does not check args
```

The entire point of the Lua script is token verification. The test should assert that the script is called with the correct `keys` and `args`:

```python
call_kwargs = processor._release_script.call_args
assert call_kwargs.kwargs["keys"] == ["order:lock:order-001"]
# token is a uuid generated internally; at minimum assert args[0] is a valid uuid
assert len(call_kwargs.kwargs["args"][0]) == 36
```

### 7. No test for `PaymentError` or DB commit failure

The test suite covers the happy path, idempotency, lock contention, and input validation. It does not cover:

- `payment.charge` raising `PaymentError`
- `db.record_charge` raising after a successful charge (the critical failure mode from issue #1)
- `db.get_charge` raising (Redis or DB unavailable at idempotency check time)

These are the cases most likely to produce incorrect billing behavior and must be tested explicitly.

## Minor Notes

- `customer_id` has no pattern constraint unlike `order_id`. If it is user-supplied, add one.
- The `CHARGE_ENDPOINT` hardcoded in `PaymentClient` should come from configuration/environment, not a class-level constant, to support environment promotion without code changes.
- Consider adding a structured log field (e.g., `extra={"order_id": ..., "customer_id": ...}`) rather than interpolating into the message string, to make log aggregation and alerting easier.
