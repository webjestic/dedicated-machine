Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check are all sound. However, there is a critical atomicity flaw at the seam between the payment gateway call and the database transaction, plus several reliability and security issues that must be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — timeout risk is severe

```python
# Current — dangerous
with self.db.transaction():
    charge_id = self.payment.charge(...)   # up to 90s (30s × 3 attempts)
    self.db.record_charge(...)
```

The payment call can block for up to **90 seconds** (30s timeout × 3 retries) while holding an open database transaction. This will exhaust your connection pool under any real load and cause cascading failures across unrelated operations.

**Fix:** Charge outside the transaction, then record inside it. Accept that a crash between the two is possible and handle it via a reconciliation job or by checking for unrecorded charges on startup. The idempotency check already partially addresses this — extend it to cover orphaned gateway charges.

```python
# Charge outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically — if this fails, reconciliation catches the orphan
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

---

### 2. `PaymentError` and all other exceptions escape `process()` unhandled

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # PaymentError propagates uncaught
```

Only `LockError` and `ValidationError` are caught. A `PaymentError`, `redis.RedisError`, database exception, or `KeyError` from `response.json()["charge_id"]` will bubble up as an unhandled exception to the caller. For a service that touches payment data, callers must not receive raw internal exceptions.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

---

### 3. Redis connection failure silently bypasses the lock

If `self.redis.set(...)` raises `redis.RedisError` (network partition, Redis restart), the exception propagates out of `_acquire_lock` before `acquired` is set. The `contextmanager` will not yield, so processing won't proceed — but the caller gets an unhandled exception rather than a safe `OrderResult`. More importantly, if Redis becomes unavailable mid-deployment, you need a deliberate policy decision: **fail closed** (block all processing) or **fail open** (process without the lock). Right now the behavior is undefined. Make it explicit.

---

## Significant Issues

### 4. Lock TTL of 60 seconds is too short given the retry math

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds`. The lock can expire while a worker is still legitimately processing, allowing a second worker to acquire it and potentially double-charge. Either raise the TTL above the maximum possible payment duration or enforce a hard timeout on the payment call that is guaranteed to be under the TTL.

```python
ORDER_LOCK_TTL = 120   # must exceed max payment client duration
```

---

### 5. `payment_token` logged indirectly via `charge_id` — verify no token leakage in payment client

The `payment_token` field is not logged directly here, which is correct. However, `PaymentClient` logs `charge_id` and status codes but does not guard against logging the full request body on error. Confirm that `requests` exceptions do not serialize the POST body (they do not by default, but this should be explicitly verified and documented, especially if a debug logging handler is ever added).

---

### 6. `response.json()["charge_id"]` raises `KeyError` on malformed gateway response

```python
charge_id = response.json()["charge_id"]
```

A 200 response with an unexpected body (missing key, non-JSON content type) raises `KeyError` or `JSONDecodeError`, neither of which is caught. This will surface as an unhandled exception in `process()`.

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError("Gateway returned success but no charge_id")
```

---

### 7. Test for lock release verifies mechanism but not correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms `delete` is not called and the script is invoked, but it does not verify that the correct `lock_key` and `token` are passed to the script. A regression that passes the wrong key would not be caught. Assert on `call_args`.

---

### 8. No test for lock release on payment failure

There is no test verifying that the lock is released when `payment.charge` raises an exception. The `finally` block in `_acquire_lock` should handle this, but it is not exercised. This is the most important path to cover given issue #2 above.

```python
def test_lock_released_on_payment_failure(processor, payment, redis_client):
    payment.charge.side_effect = Exception("gateway down")
    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    processor._release_script.assert_called_once()
```

---

## Minor Issues

- `customer_id` has no pattern constraint, unlike `order_id`. If it is used in any query or log context, apply the same alphanumeric pattern.
- `RETRY_BACKOFF = 5` is a flat value with no jitter. Under concurrent worker restarts this will produce synchronized retry storms against the payment gateway. Add jitter.
- The `ORDER_LOCK_TTL` and `MAX_RETRIES` constants live in separate classes with no enforced relationship. Document the dependency explicitly or derive one from the other.
