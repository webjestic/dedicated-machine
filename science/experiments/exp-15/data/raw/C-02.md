Decision: Request Changes

Summary: The core locking strategy and payment flow contain a critical atomicity flaw that directly undermines the at-most-once guarantee this PR is designed to provide. Several additional reliability and security issues compound this. The code is well-structured and the intent is sound, but these problems must be resolved before this touches production.

---

## Critical Issues

### 1. Payment call is outside the database transaction — the central guarantee is broken

This is the most serious flaw. The PR description explicitly claims:

> "Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"

That claim is false. Look at the actual code:

```python
with self.db.transaction():
    charge_id = self.payment.charge(      # ← external network call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a payment gateway. Two failure modes immediately break the guarantee:

**Scenario A — Charge succeeds, process crashes before `record_charge`:**
The payment gateway has charged the customer. The database transaction rolls back. `record_charge` never runs. The idempotency check will find no existing charge on retry, and the customer gets charged again.

**Scenario B — Holding an open DB transaction across a 30-second network call:**
`PaymentClient.REQUEST_TIMEOUT = 30` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` means this transaction can be held open for over 90 seconds. This will exhaust your DB connection pool under any real load and cause cascading failures across unrelated services.

**The correct pattern** is to separate the charge from the record:

```python
# Outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB writes inside the transaction
with self.db.transaction():
    # Re-check idempotency inside the transaction under a DB-level lock
    if self.db.get_charge_for_update(req.order_id):
        # Charge already recorded — issue a refund for the duplicate charge
        self.payment.refund(charge_id)
        return OrderResult(processed=False, ...)
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The crash-between-charge-and-record problem is a known hard problem. The standard mitigations are: (a) a reconciliation/recovery job that scans for charges without DB records, (b) idempotency keys on the payment gateway side so retries don't double-charge, or (c) a pending charge state written to the DB *before* calling the gateway. Pick one and document it explicitly.

---

### 2. Redis lock TTL does not cover the actual critical section duration

```python
ORDER_LOCK_TTL = 60  # seconds
```

As established above, the payment call alone can take 90+ seconds. If the lock expires mid-processing while the payment is in-flight, a second worker acquires the lock, passes the idempotency check (charge not yet recorded), and issues a duplicate charge. The Redis lock provides no protection in this scenario.

The TTL must be set to a value that is strictly greater than the maximum possible execution time of the critical section. Either reduce the payment client's total retry budget well below 60 seconds, or increase the TTL accordingly — and document the relationship explicitly so they don't drift apart again.

---

### 3. Lock release silently swallows all exceptions

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script call raises (Redis connection lost, script evicted), the exception is silently suppressed by the `finally` block. The caller has no idea the lock was not released. It will hold until TTL expiry. This is acceptable behavior, but it should be *explicit*:

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error("Lock release returned 0 — token mismatch or key expired order_id=%s", order_id)
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except redis.RedisError:
        logger.error("Failed to release lock order_id=%s — will expire at TTL", order_id, exc_info=True)
```

The `released == 0` case is also worth logging as an error. It means either the TTL expired mid-processing (the race described above just happened) or there is a token bug.

## Moderate Issues

### 4. `payment_token` is logged indirectly and stored in the Pydantic model in plaintext

`payment_token` flows through `OrderRequest`, gets passed to `payment.charge()`, and the entire `req` object lives in memory for the duration of processing. Ensure this field is excluded from any `__repr__`, `dict()`, or `model_dump()` calls that might reach log output:

```python
payment_token: str = Field(..., min_length=1, max_length=256)

class Config:
    # or use a custom __repr__ that masks sensitive fields
```

Pydantic v2 does not redact fields by default. A single `logger.debug("Processing request: %s", req)` anywhere in this call stack leaks the token to log aggregation.

### 5. `PaymentClient` has no authentication on outbound requests

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, no mTLS configuration, no API key. Even on an internal network, this should be authenticated. If the endpoint is on an internal network that is later compromised or misconfigured, unauthenticated charge requests become a significant risk surface.

### 6. Timeout on gateway response does not account for retry on timeout

On a `Timeout` exception, the code retries. But a timeout means the gateway *may have processed the charge* and simply failed to respond in time. Retrying without an idempotency key on the gateway request risks a duplicate charge at the gateway level, independent of the Redis lock. The payment gateway API should be called with a stable idempotency key (e.g., `order_id`) on every attempt.

## Minor Issues

### 7. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms the Lua script is called but does not assert that the correct token is passed. The entire point of the token-verified release is that the token matches. Add `assert_called_once_with(keys=[...], args=[<expected_token>])` or at minimum verify the `args` kwarg is present and non-empty.

### 8. `LOCK_RETRY_ATTEMPTS = 3` with `backoff = 0.5 * (attempt + 1)` means up to 3 seconds of blocking

Under high contention this will block the calling thread for up to 3 seconds before returning a failure. If this runs in a synchronous web worker (Gunicorn, uWSGI), that is 3 seconds of a worker thread held. Document the threading model assumption or make the retry count and backoff configurable at construction time.

---

## What Needs to Change Before Approval

1. Move `payment.charge()` outside the database transaction and document the crash-recovery strategy explicitly.
2. Align `ORDER_LOCK_TTL` with the actual maximum critical section duration and enforce that relationship in code or configuration validation.
3. Add explicit error handling and logging in the lock release `finally` block, including the `released == 0` case.
4. Redact `payment_token` from Pydantic model serialization.
5. Add authentication to outbound payment gateway requests.
6. Pass a stable idempotency key to the payment gateway on each attempt.
