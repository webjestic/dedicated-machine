Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw that undermines the at-most-once guarantee: `payment.charge()` is called *inside* the database transaction, meaning a successful charge that is followed by a DB commit failure will produce an unrecorded, unrecoverable charge with no compensation path. There are also several secondary issues around error handling gaps, a timeout risk that can exceed the lock TTL, and missing test coverage for important failure modes that need to be addressed before this is merged.

---

## Detailed Findings

### 🔴 Critical — Charge Inside DB Transaction Breaks At-Most-Once Guarantee

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. The DB transaction wrapping an external HTTP call does not make the charge atomic with the record. If `charge()` succeeds but the DB commit fails (network partition, DB crash, deadlock), the charge is real money out of the customer's account with no record in your system. You cannot roll back an external payment gateway call by rolling back a DB transaction.

**Required fix — charge first, then record durably:**

```python
# Correct pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# From this point forward, we MUST record. Any failure here needs
# an alert/dead-letter queue — do not silently swallow exceptions.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "UNRECORDED CHARGE — manual reconciliation required "
        "order_id=%s charge_id=%s amount_cents=%d",
        req.order_id, charge_id, req.amount_cents,
    )
    raise  # or route to dead-letter queue
```

The idempotency check + lock together prevent double-charging on retry. The charge must happen *before* the transaction, and a DB failure after a successful charge must be loud and traceable.

### 🔴 Critical — Total Retry Duration Can Exceed Lock TTL

**File:** `services/payment_client.py`

```
MAX_RETRIES = 3
REQUEST_TIMEOUT = 30s per attempt
RETRY_BACKOFF = 5s between retries
```

Worst-case payment client duration: `3 × 30s + 2 × 5s = 100 seconds`

**File:** `services/order_processor.py`

```python
ORDER_LOCK_TTL = 60  # seconds
```

The lock can expire while payment is still in-flight. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. This is exactly the double-charge scenario the lock is supposed to prevent.

**Required fix:** Lock TTL must exceed the maximum possible payment duration with margin, or the payment client's total timeout ceiling must be reduced to fit within the TTL. These two constants need to be co-owned and documented together:

```python
# These must be set in concert. Lock TTL must exceed max payment duration.
# Max payment duration = MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES-1) * RETRY_BACKOFF
# = 3*30 + 2*5 = 100s. Lock TTL must be > 100s.
ORDER_LOCK_TTL = 120   # seconds — covers max payment round-trip with margin
```

### 🟡 Major — `PaymentError` Is Silently Swallowed

**File:** `services/order_processor.py`

`PaymentError` (and any other unexpected exception from `self.payment.charge()` or `self.db.*`) is not caught inside the lock context. It will propagate up through the `_acquire_lock` context manager's `finally` block, which correctly releases the lock, and then bubble up to the caller as an unhandled exception. There is no `except` clause converting it to an `OrderResult`.

This means callers get a raw exception instead of a structured `OrderResult`, breaking the stated return contract of the method. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    raise  # don't silently swallow unknown failures
```

### 🟡 Major — `payment_token` Logged or Leaked Risk

**File:** `services/order_processor.py` and `services/payment_client.py`

`payment_token` flows through `OrderRequest` and is passed directly to the payment client. Ensure it is never included in log output. The current logging statements do not log it directly, but the `ValidationError` log on line:

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors can include field values in their output depending on version and configuration. A `payment_token` that fails length validation would appear in the log. Mask sensitive fields:

```python
class OrderRequest(BaseModel):
    model_config = {"hide_input_in_errors": True}  # Pydantic v2
    payment_token: str = Field(..., min_length=1, max_length=256)
```

---

### 🟡 Major — `requests` Response Body Parsed Without Validation

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

This will raise an unhandled `KeyError` if the gateway returns a 200 with an unexpected body shape, and a `JSONDecodeError` if the body is not valid JSON. Both will propagate as unhandled exceptions rather than a clean `PaymentError`. Wrap this:

```python
try:
    body = response.json()
    charge_id = body["charge_id"]
except (ValueError, KeyError) as e:
    raise PaymentError(f"Unexpected gateway response format: {e}") from e
```

---

### 🟡 Major — Lock Retry Backoff Is Linear, Not Exponential as Described

**File:** `services/order_processor.py`

The PR description says "retry with backoff" which implies exponential. The implementation is linear (`0.5 * (attempt + 1)` → 0.5s, 1.0s, 1.5s). This is a minor behavioral issue but the description is misleading. Either fix the implementation to be exponential or correct the description. More importantly, with only 3 attempts and linear backoff, the total wait before giving up is 1.5 seconds — this may be too aggressive for lock contention under normal load. Consider making these configurable.

---

### 🟢 Minor — Missing Test Coverage for Key Failure Paths

**File:** `tests/test_order_processor.py`

The following scenarios have no test coverage and should be added:

- `PaymentError` raised by `payment.charge()` — verifies structured return and no DB write
- DB transaction failure after successful charge — this is the critical path from the first finding
- Lock TTL expiry scenario (harder to unit test, but worth a note)
- `payment_token` does not appear in log output on validation failure

---

### 🟢 Minor — `CHARGE_ENDPOINT` Should Not Be Hardcoded

**File:** `services/payment_client.py`

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs make environment promotion (staging → production) require code changes. Accept this via constructor parameter or environment variable.
