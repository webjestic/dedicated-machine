Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat scoping, Pydantic validation, and structured logging are all directionally correct. However, there are several architectural flaws and correctness issues that range from a subtle but critical TOCTOU race in the reservation logic, to a heartbeat timing bug that can cause premature lock expiry, to test coverage that doesn't actually validate the behaviors it claims to. None of these are cosmetic — they represent real failure modes under production flash-sale load.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Is Outside the Database Transaction

**Severity: Critical**

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    return ReservationResult(...)

new_stock = current_stock - req.quantity
order_id = str(uuid.uuid4())

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes access across *service instances*, but it provides zero protection against a database read that is not part of the same atomic transaction. If the DB supports any isolation level below serializable — or if the lock is ever lost and re-acquired between the read and write — the stock check is stale. The correct pattern is to move `get_stock` inside the transaction and use a `SELECT FOR UPDATE` (or equivalent pessimistic lock at the DB layer), or use a compare-and-swap update (`UPDATE inventory SET stock = stock - qty WHERE item_id = ? AND stock >= qty`) and check affected rows. The current design gives a false sense of safety.

### 2. Heartbeat Interval Allows Lock Expiry Window

**Severity: High**

```python
LOCK_TTL = 30          # seconds
HEARTBEAT_INTERVAL = 8  # seconds — fires at TTL/3
```

The comment claims the heartbeat fires at TTL/3, which is correct for the *first* beat. But `threading.Event.wait(timeout=N)` begins counting from the moment `wait()` is called — not from when the lock was acquired. Consider the timeline:

- T=0: Lock acquired (TTL=30s)
- T=0: Heartbeat thread starts, calls `wait(timeout=8)`
- T=8: First extension fires → TTL reset to 30s ✓
- T=8: `wait(timeout=8)` called again
- T=16: Second extension fires ✓

This looks fine in isolation, but the thread startup itself has non-zero latency, and more importantly: if the extension Redis call blocks (network partition, Redis slowdown), the next `wait()` call is delayed. Under a 30-second TTL with an 8-second interval, you have roughly 3 heartbeat windows before expiry. A single blocked extension call can consume an entire window. The safer design is to use `TTL/2` as the interval (15s here), giving a full extra window of margin, or to track wall-clock time and fire based on elapsed time since last successful extension rather than a fixed sleep.

### 3. Heartbeat Thread Leak on Lock Acquisition Failure Path

**Severity: High**

```python
t = threading.Thread(target=_beat, daemon=True)
t.start()

try:
    yield token
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The `finally` block correctly clears the event and joins the thread. However, if `self._release_script(...)` raises an exception *after* the join, the thread is already cleaned up — that part is fine. The real issue is subtler: `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` is `join(timeout=9)`. If the heartbeat thread is blocked on a Redis call that hangs for longer than 9 seconds, `join()` returns without the thread actually having exited. The thread is a daemon, so it won't prevent process exit, but it *will* continue running and attempting Redis operations against a lock key that the caller believes has been released. This can cause spurious `"Heartbeat lost lock ownership"` log noise at best, and interferes with lock state reasoning at worst. The join timeout should be bounded by a hard deadline, and if the join times out, that condition should be logged explicitly as an error.

### 4. `get_stock` Return Value Not Defensively Typed

**Severity: Medium**

```python
current_stock = self.db.get_stock(req.item_id)

if current_stock is None:
    return ReservationResult(...)

if current_stock < req.quantity:
```

There is no guard against `get_stock` returning a non-integer (e.g., a float, a string from a misconfigured ORM, or a negative value from a data integrity issue). A negative stock value would pass the `< req.quantity` check and produce a negative `new_stock` written back to the database. At minimum, assert `isinstance(current_stock, int)` and `current_stock >= 0`, or enforce this at the DB layer with a check constraint and handle the resulting exception here.

## Moderate Issues

### 5. Pydantic Validation Errors Are Silently Swallowed

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If `item_id` or `quantity` fails Pydantic validation, a `ValidationError` is raised. This is not caught anywhere in `reserve_item`. The `except LockAcquisitionError` block won't catch it. The exception will propagate to the caller as an unhandled `ValidationError`, which is a different contract than the `ReservationResult` the method signature promises. Either catch `ValidationError` explicitly and return a structured failure result, or document that the method can raise and update callers accordingly. The tests cover this path but only because Pydantic raises before the lock is ever acquired — the test passes for the wrong reason if the caller doesn't expect an exception.

```python
def test_input_validation_rejects_invalid_item_id(service):
    result = service.reserve_item("../../admin", quantity=1)
    assert result.success is False  # This will actually raise, not return
```

This test is incorrect. Run it against the actual code and it will fail with `ValidationError`, not return a `ReservationResult`. The checklist item "Tests pass" is therefore suspect.

---

### 6. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, so colon injection is not possible here — that's good. However, there is no environment or tenant namespace prefix. In a multi-tenant deployment or in a shared Redis instance used across staging and production (a common ops mistake), lock keys from different environments will collide. Recommend parameterizing a namespace prefix at `InventoryService` construction time.

---

### 7. `order_id` Generated Outside the Transaction

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This is low-risk but worth noting: if the transaction rolls back and is retried by a higher-level framework, a new `order_id` will not be generated — the same UUID will be reused. Depending on DB constraints, this could cause a duplicate key error on retry. Generate `order_id` inside the transaction or ensure retry logic regenerates it.

## Test Coverage Gaps

### 8. Heartbeat Behavior Is Not Tested

The heartbeat thread is the most operationally novel part of this implementation and has zero test coverage. There are no tests for:
- Extension being called at the expected interval
- Heartbeat stopping when the lock is lost (`result == 0`)
- Thread being joined before lock release
- Behavior when `_extend_script` raises a Redis connection error

### 9. `test_lock_release_is_atomic` Does Not Test What It Claims

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is a registered script object on the mock Redis client. `register_script` returns a callable mock, and asserting it was called once does not verify the keys or args passed to it. This test would pass even if the script were called with the wrong key or a blank token. Assert on the actual call arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, verify format if needed
)
```

### 10. No Test for DB Transaction Rollback Behavior

There is no test covering what happens if `db.update_stock` or `db.create_order` raises inside the transaction. Does the lock still get released? Does the heartbeat thread still get joined? The `finally` block in `_acquire_lock` should handle this, but it is not exercised.

## Minor Issues

- `retry_attempts=3` with backoffs of 0.1s, 0.2s, 0.4s means a caller waits up to 700ms before failure. For a flash sale endpoint this may be acceptable, but it should be documented and configurable, as it directly affects P99 latency under contention.
- The `HEARTBEAT_INTERVAL` constant is documented as "fires at TTL/3" but this relationship is implicit. If `LOCK_TTL` is ever changed without updating `HEARTBEAT_INTERVAL`, the invariant silently breaks. Derive it: `HEARTBEAT_INTERVAL = LOCK_TTL // 3`.
- No Redis connection error handling. If `self.redis.set(...)` raises `redis.RedisError`, it propagates as an unhandled exception rather than a structured `ReservationResult`.
