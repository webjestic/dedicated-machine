Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped lock state are all appropriate choices for this domain. However, there are several correctness flaws, a critical race condition in the heartbeat shutdown sequence, a stock-read/transaction gap that undermines the entire locking guarantee, and test coverage that is largely illusory due to mock misconfiguration. None of these are cosmetic — they represent real failure modes under production load.

## Critical Issues

### 1. Stock Read Outside the Database Transaction (Correctness Flaw)

This is the most serious architectural issue in the file.

```python
# Inside the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes computed value
    self.db.create_order(...)
```

The stock is read outside the database transaction. If the database uses `READ COMMITTED` isolation (the default in PostgreSQL and MySQL), another connection — one that is *not* participating in the Redis lock, such as an admin correction, a return/refund flow, a bulk import job, or a second service instance that bypassed the lock due to a Redis failover — can commit a stock change between the `get_stock` call and the `update_stock` call. The transaction then blindly writes a stale computed value.

The correct pattern is a conditional atomic update inside the transaction:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        # rollback / return early
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

`SELECT FOR UPDATE` acquires a row-level lock inside the transaction, making the read-modify-write atomic at the database level. The Redis lock handles cross-service coordination; the database lock handles correctness within a single transaction. Both are needed. As written, the Redis lock is doing all the work but the database provides no safety net when the lock assumption breaks.

### 2. Heartbeat Race Condition on Shutdown

```python
finally:
    heartbeat_running.clear()          # (A) signal thread to stop
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # (B) wait up to 9s
    self._release_script(keys=[lock_key], args=[token])  # (C) release
```

There is a window between `(A)` and `(C)` where the heartbeat thread may be mid-execution of `_extend_script`. The sequence:

1. Main thread clears the event at `(A)`.
2. Heartbeat thread is currently inside `_extend_script` — it has already passed the `heartbeat_running.wait()` check and is executing the Redis call.
3. Main thread calls `t.join()` and blocks.
4. Heartbeat thread finishes `_extend_script`, sees the event is cleared on the *next* iteration, and exits.
5. Main thread unblocks from `join()` and calls `_release_script` at `(C)`.

This specific sequence is actually *safe* because `join()` ensures the thread has fully exited before release. However, there is a subtler issue: if `t.join(timeout=...)` **times out** — which it can, since the heartbeat thread could be blocked on a slow Redis call — the main thread proceeds to `_release_script` while the heartbeat thread is still alive and may still call `_extend_script`. This is a benign race in practice (both use the same token), but it means the lock TTL could be silently extended *after* the Lua release script runs, which cannot happen with the current Lua script (release deletes the key), but it is a fragile assumption. More critically, if the join times out, you are leaking a thread. Daemon threads will be killed on process exit, but in a long-running service this accumulates.

The join timeout should be logged as an error, not silently ignored:

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.error("Heartbeat thread did not exit cleanly key=%s — possible thread leak", lock_key)
    self._release_script(keys=[lock_key], args=[token])
```

### 3. Heartbeat `wait()` Logic Is Inverted

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`threading.Event.wait(timeout)` returns `True` if the event is **set** and `False` if the timeout expired. The intent here is to sleep for `HEARTBEAT_INTERVAL` seconds between renewals and stop when the event is cleared. But the current logic does the opposite: it runs the renewal body *immediately* when the event is set (which it always is at thread start), then blocks until the event is cleared or the timeout fires. The first heartbeat fires at `t=0`, not at `t=HEARTBEAT_INTERVAL`. Under a 30-second TTL this is not catastrophic, but it is incorrect and will cause unnecessary Redis calls.

The correct idiom for a cancellable sleep is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout expired, event still set → extend
        result = self._extend_script(...)
        ...
    # wait() returned True → event was cleared → exit
```

Or more explicitly using a stop event:

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        self._extend_script(...)

# In finally:
stop_event.set()
t.join(...)
```

### 4. Pydantic Validation Exceptions Are Not Caught

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
    ...
    try:
        with self._acquire_lock(lock_key):
            ...
    except LockAcquisitionError as e:
        ...
```

The `ReservationRequest(...)` call is outside the `try/except` block. A `pydantic.ValidationError` will propagate uncaught to the caller. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but the actual behavior is an unhandled exception. These tests will fail against the real code unless the test framework is catching exceptions somewhere not shown.

Fix:

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

## Significant Issues

### 5. Tests Are Misconfigured and Provide False Confidence

**Mock `register_script` is not wired correctly.**

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1   # ← this attribute is never called
    return mock
```

`InventoryService.__init__` calls `redis_client.register_script(...)` twice, which on a `MagicMock` returns a new `MagicMock` callable. That callable (the registered script) returns another `MagicMock` when invoked — not `1`. The heartbeat's `if result == 0` check will never be `True` because `MagicMock() == 0` is `False`. The lock release script similarly returns a `MagicMock`. The tests are not testing the actual conditional branches in the heartbeat or release paths.

**`test_lock_release_is_atomic` is testing the wrong object.**

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()  # ← correct target
    redis_client.delete.assert_not_called()        # ← correct negative assertion
```

`service._release_script` is the return value of `redis_client.register_script(RELEASE_LOCK_SCRIPT)`. On a `MagicMock`, `register_script` returns a consistent child mock, so `service._release_script` and `redis_client.register_script.return_value` refer to the same object. This test will pass, but only because `MagicMock` auto-creates attributes — it does not verify the Lua script content was actually registered or called with the right arguments.

**`test_reserve_concurrent_quantity` is not testing concurrency.**

The test name implies concurrent access but it is a single-threaded sequential call. A meaningful concurrency test would use `threading.Barrier` or `concurrent.futures.ThreadPoolExecutor` to verify that two simultaneous reservations of the last 3 units result in exactly one success and one failure.

### 6. No Handling of Redis Connectivity Failures

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

If Redis is unavailable, `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` propagates uncaught through `_acquire_lock` and through `reserve_item`. The caller receives an unhandled exception rather than a `ReservationResult`. For a flash sale service, Redis downtime is a real operational scenario and should be handled explicitly — either by returning a degraded result or by re-raising a typed service exception that the API layer can translate to a 503.
