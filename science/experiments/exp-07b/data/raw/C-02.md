Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical race condition in the heartbeat/release sequencing, architectural gaps around failure handling, and test suite deficiencies that would make this unsafe to merge into a production flash-sale system where inventory correctness is paramount.

## Critical Issues

### 1. Race Condition: Heartbeat Can Extend a Lock After Release Begins

This is the most serious flaw in the PR.

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`heartbeat_running.clear()` signals the heartbeat to stop, but `threading.Event.wait(timeout=N)` will not wake immediately — it will sleep until the current `HEARTBEAT_INTERVAL` window expires (up to 8 seconds). During that window, the heartbeat thread may fire one final `_extend_script` call **after** the `finally` block has already decided to release. The sequence becomes:

1. `heartbeat_running.clear()` is called
2. Heartbeat thread is mid-sleep in `wait(timeout=8)`
3. `t.join(timeout=9)` blocks
4. Heartbeat wakes, checks `heartbeat_running.wait(...)` — `wait` on a cleared Event with a timeout returns `False` immediately, so the loop exits correctly

Actually, re-examining: `heartbeat_running.wait(timeout=8)` returns `False` when the event is cleared, so the `while` loop exits. This is correct behavior for the loop exit. **However**, the real race is the inverse: the heartbeat thread could be in the middle of executing `_extend_script` at the exact moment `_release_script` fires. The sequence:

1. Heartbeat calls `_extend_script` → extends TTL to 30s
2. `finally` block calls `_release_script` → deletes key
3. A **different process** acquires the lock on the now-deleted key
4. Heartbeat's `_extend_script` call returns 0 (key gone), which is fine — but if the heartbeat had extended *after* the release script ran and a new owner existed, the extend would fail safely due to the token check

The token-guarded Lua scripts do protect against extending another owner's lock. But the `t.join(timeout=...)` can silently time out — if the heartbeat thread is hung (e.g., Redis network stall), the join returns and `_release_script` fires anyway. This is acceptable, but it must be documented. More critically:

**The join timeout is `HEARTBEAT_INTERVAL + 1 = 9s`. If the heartbeat thread is blocked on a slow Redis call, the join times out, and `_release_script` fires while the heartbeat's `_extend_script` is still in-flight.** The Lua token guard makes this safe for *other* lock holders, but the extend completing after the release means the lock briefly reappears with a fresh TTL under the old token — which no one holds. This is a ghost lock that will persist for up to 30 seconds, blocking all reservations for that item.

**Fix:** Use a separate `threading.Event` to signal the heartbeat to stop, and a second event for the heartbeat to signal completion. Do not rely solely on `join` with a timeout.

```python
stop_event = threading.Event()
done_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
    done_event.set()

# In finally:
stop_event.set()
done_event.wait(timeout=self.HEARTBEAT_INTERVAL + 2)
self._release_script(...)
```

### 2. `get_stock` Is Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens outside the DB transaction. If the database supports snapshot isolation or read-committed isolation, and if the DB crashes between the read and the write, the transaction rolls back but the lock is released — this is acceptable. However, the more dangerous scenario is: **what guarantees does `db.get_stock` provide?** If it reads from a replica with replication lag, you can read stale stock, decrement it, and oversell. The read must either:

- Be inside the transaction with `SELECT ... FOR UPDATE` (or equivalent pessimistic locking), or
- Be explicitly documented as requiring a strongly-consistent read from the primary

The distributed lock protects against concurrent *service instances* racing, but it does not protect against the DB layer returning stale data. Move `get_stock` inside the transaction boundary.

### 3. No Handling of Exceptions from DB Operations

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `update_stock` or `create_order` raises an exception, it will propagate up through `_acquire_lock`'s `finally` block, which will correctly release the lock. But the exception is **not caught** in `reserve_item` — it will bubble up to the caller as an unhandled exception rather than returning a `ReservationResult`. For a service method with a defined return type contract, this is a broken interface. The caller has no way to distinguish a DB failure from a programming error.

Add explicit exception handling:

```python
except (DatabaseError, SomeOtherExpectedException) as e:
    logger.error("DB operation failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

### 4. Pydantic Validation Errors Are Not Caught

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If `item_id` or `quantity` fails Pydantic validation, a `ValidationError` is raised. This is not caught anywhere in `reserve_item`. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but the actual behavior is an uncaught exception — **these tests will fail or pass for the wrong reason depending on how the test runner handles unhandled exceptions.** The tests are asserting a contract that the code does not implement.

Fix:

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters.")
```

## Significant Issues

### 5. Heartbeat Interval Is Too Close to Lock TTL

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 8`. The comment says "fires at TTL/3" which is `30/3 = 10`, but 8 ≠ 10. More importantly, the heartbeat fires every 8 seconds, but the first fire happens 8 seconds after lock acquisition. If the DB operation takes longer than 30 seconds before the first heartbeat fires, the lock expires. The heartbeat should fire at `TTL / 2` or sooner, and the first extension should happen well before expiry. At minimum, fix the comment to match the actual value, and reconsider whether 8 seconds provides sufficient margin given network jitter and thread scheduling delays.

### 6. `t.join(timeout=...)` Return Value Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`Thread.join()` returns `None` whether the thread finished or timed out. There is no way to know if the join timed out. If it did time out, the heartbeat thread is still running and may interact with the lock after release. This should be:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly for key=%s", lock_key)
    # Do not proceed silently — this is a dangerous state
```

### 7. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which prevents injection into the key. However, there is no namespace isolation between environments (dev/staging/prod). If Redis is shared across environments (common in cost-constrained government deployments), a staging flash sale will block production reservations for the same item ID. Add an environment prefix or require it to be injected at construction time.

### 8. No Redis Connection Failure Handling

If `self.redis.set(...)` raises a `redis.RedisError` (connection refused, timeout, etc.), the exception propagates uncaught from `_acquire_lock`, bypassing the `LockAcquisitionError` path. The caller receives an unhandled `RedisError` instead of a graceful `ReservationResult`. Redis failures should be caught and converted to `LockAcquisitionError` or handled explicitly.

## Test Suite Issues

### 9. Validation Tests Assert Wrong Behavior

As noted in issue #4, `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will raise `ValidationError`, not return `ReservationResult(success=False)`. These tests are testing a contract the code doesn't fulfill. They may appear to pass if pytest catches the exception and the assertion is never reached, or they may error — either way they are not testing what they claim to test.
