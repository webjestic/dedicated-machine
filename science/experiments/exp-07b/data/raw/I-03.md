Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a meaningful race condition, a TOCTOU vulnerability, and test suite gaps that collectively make this unfit for a high-concurrency flash sale context. The issues range from subtle threading bugs to a stock-read that occurs outside the database transaction boundary, which undermines the entire locking guarantee.

---

## Critical Issues

### 1. Stock Read Occurs Outside the Database Transaction (TOCTOU)

This is the most serious architectural flaw. The Redis lock serializes access correctly, but the stock read and the stock write are not atomic at the database level:

```python
# PROBLEM: get_stock is OUTSIDE the db.transaction() block
current_stock = self.db.get_stock(req.item_id)  # ← read happens here

# ... stock check logic ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write happens here
    self.db.create_order(...)
```

If the database supports any form of read committed isolation or if the Redis lock is ever lost between the read and the write (e.g., Redis failover, network partition, heartbeat race — see issue #2), the stock value used for the decision is stale by the time the write commits. The read **must** occur inside the transaction, and ideally the `UPDATE` should use a conditional clause (`WHERE stock >= quantity`) as a final safety net at the database layer. Redis is a performance optimization here, not a correctness guarantee — the database must enforce the invariant.

```python
# Correct structure
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

### 2. Heartbeat Thread Race Condition on Lock Release

The heartbeat thread and the `finally` block have a race window:

```python
finally:
    heartbeat_running.clear()          # (1) signal stop
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # (2) wait up to 9s
    self._release_script(keys=[lock_key], args=[token])  # (3) release
```

The `threading.Event.wait(timeout=N)` call in `_beat` can return `False` (timeout expired) even after `heartbeat_running.clear()` is called, because `wait` with a timeout does not re-check the event state atomically after waking. More critically: if `t.join()` times out (e.g., the thread is blocked on a slow Redis call to `_extend_script`), execution falls through to `_release_script` while the heartbeat thread may **still be running and attempting to extend the lock TTL**. This creates a window where the lock is released and potentially re-acquired by another process, and then the zombie heartbeat extends the new owner's lock with the old token — which will silently fail due to the token check, but the timing window is real and the join timeout provides false safety assurance.

The join timeout should either be removed (block until the thread actually stops) or the thread should be structured to exit cleanly before the join is reached. A `threading.Event` used as a stop signal is correct, but the join timeout masks the underlying problem.

### 3. Validation Exceptions Are Not Caught

Pydantic `ValidationError` is raised but never caught in `reserve_item`:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # ← can raise ValidationError
```

The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but the actual behavior is an **unhandled exception propagating to the caller**. These tests are passing for the wrong reason or are not actually running against this code path correctly. The `except` block only catches `LockAcquisitionError`. A `ValidationError` catch block must be added and return a structured `ReservationResult`.

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    return ReservationResult(success=False, message=f"Invalid input: {e}")
```

## Moderate Issues

### 4. `heartbeat_running.wait(timeout=...)` Semantics Are Inverted

```python
while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
```

`Event.wait(timeout)` returns `True` if the event is **set** and `False` if the timeout expired. The intent here is to sleep for `HEARTBEAT_INTERVAL` seconds and then extend the lock. But the loop condition means: *keep running while the event is set*. When `heartbeat_running.clear()` is called, `wait` returns `False` immediately and the loop exits — which is the correct stop behavior. However, the sleep-then-act pattern is inverted: the thread extends the lock immediately on the first iteration if the event is already set (which it always is at start), then waits. This means the **first heartbeat fires immediately** at thread start rather than after `HEARTBEAT_INTERVAL` seconds. This is harmless but wasteful and indicates the logic was not carefully reasoned through. Use `time.sleep` or `Event.wait` with a proper initial delay.

---

### 5. Retry Backoff Does Not Include Jitter

```python
backoff = (2 ** attempt) * 0.1  # 0.1s, 0.2s, 0.4s
```

Under a flash sale load scenario with hundreds of concurrent waiters, pure exponential backoff without jitter causes thundering herd behavior — all contending threads wake at approximately the same time and hammer the lock simultaneously. Add random jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

### 6. `order_id` Logged Before Transaction Commits

```python
order_id = str(uuid.uuid4())

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)

logger.info("Reserved item_id=%s ... order_id=%s ...", ...)
```

This is acceptable only if the log line is after the `with` block (which it is). However, if `db.transaction().__exit__` raises on commit failure, the log line is never reached — which is correct. But there is no handling of transaction commit failure. A failed commit should return a `ReservationResult(success=False, ...)`, not propagate an unhandled exception. Wrap the transaction block in a `try/except`.

## Test Suite Issues

### 7. Tests Assert Behavior That Does Not Match Implementation

As noted in issue #3, `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will raise `ValidationError`, not return `ReservationResult(success=False)`. If these tests are actually passing, the test fixture or mock is masking the real code path. This needs to be verified and corrected alongside the fix in the service.

### 8. No Test for Heartbeat Thread Lifecycle

There is no test verifying that the heartbeat thread is started, that it calls `_extend_script`, or that it is properly joined and stopped on lock release. Given that the heartbeat is a core feature of this PR, its behavior under normal and abnormal conditions (lock loss, slow Redis) must be explicitly tested.

### 9. No Test for Database Transaction Failure

There is no test covering the case where `db.transaction().__exit__` raises (simulating a commit failure). This is a critical path in a financial reservation system.

### 10. `redis_client.execute_script` Mock Is Unused

```python
mock.execute_script.return_value = 1  # in redis_client fixture
```

`execute_script` is not called anywhere in the implementation. The scripts are invoked via `register_script` return values. This mock does nothing and suggests the test fixture was written without verifying it against the actual call sites.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is a reasonable ratio, but the comment says "TTL/3" — 30/3 is 10, not 8. Update the comment or the constant.
- The `lock_key` format `inventory:lock:{item_id}` is fine, but since `item_id` is already validated by the pattern `^[a-zA-Z0-9_\-]+$`, there is no injection risk. Worth a brief comment confirming this is intentional.
- Consider whether `retry_attempts=3` should be configurable at the service level rather than per-call, for operational consistency across deployments.
