Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped lock state are all directionally correct. However, there are several critical correctness flaws, a significant distributed systems race condition, and test suite gaps that collectively make this unsafe to merge into a production flash-sale path. The issues range from a stock-read/transaction split that defeats the purpose of the lock, to a heartbeat loop logic error that can cause premature termination, to tests that don't actually validate the behaviors they claim to test.

---

## Critical Issues

### 1. Stock Read Outside the Database Transaction (Race / Correctness)

```python
# PROBLEM: stock is read OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes inside
    self.db.create_order(...)
```

The Redis lock serializes *Redis-level* access, but `get_stock` executes at the database's default isolation level, outside the transaction boundary. If the database uses `READ COMMITTED` (the default in PostgreSQL and MySQL), another committed write between the read and the `update_stock` call can produce a phantom read — you write a stale `new_stock` value. The read **must** be inside the transaction, and ideally use a `SELECT FOR UPDATE` or equivalent pessimistic row lock to guarantee the value doesn't change between read and write:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

This is the single most important fix. Without it, the Redis lock provides a false sense of safety.

### 2. Heartbeat Loop Terminates Prematurely on First Interval

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            heartbeat_running.clear()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

`threading.Event.wait(timeout=N)` returns `True` if the event is **set** and `False` on timeout. Since `heartbeat_running` is set at the start and never cleared during normal operation, `wait()` returns `True` immediately on every iteration — the heartbeat fires in a tight loop with no sleep, hammering Redis continuously until the lock is released. The intent was clearly to sleep for `HEARTBEAT_INTERVAL` seconds between extensions, but the logic is inverted.

The correct pattern for a periodic heartbeat using an Event as a stop signal is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout elapsed, event still not set → extend
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
    # wait() returned True → event was set → stop signal received, exit cleanly
```

Or use a separate stop event with clearer semantics:

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...

# In finally:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

As written, this bug causes unbounded Redis load under any lock held longer than a millisecond.

### 3. Lock Release After Heartbeat Reports Loss

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread detected that the lock was stolen or expired (`result == 0`), the lock no longer belongs to this process. The Lua release script correctly guards against this (it checks the token), so the release call is a no-op in that case — that part is safe. However, the code continues executing the critical section after the heartbeat has already signaled loss of ownership. There is no mechanism to abort the in-progress `reserve_item` operation when the heartbeat loses the lock. This means:

- The lock expired mid-operation.
- Another process acquired the lock and may have already modified stock.
- This process continues and commits its transaction anyway.

You need a cooperative cancellation path. One approach is to have the heartbeat set a shared flag that the main thread checks at key points, or raise into the main thread via a future/exception mechanism.

## Significant Issues

### 4. Pydantic Validation Errors Not Caught

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
    ...
    except LockAcquisitionError as e:
        ...
```

`pydantic.ValidationError` is not caught anywhere. An invalid `item_id` or `quantity=0` raises an unhandled exception at the call site rather than returning a `ReservationResult`. The test `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` are almost certainly passing for the wrong reason — they likely raise an exception that pytest catches as a test failure, or the test is asserting `result.success is False` on an exception object, which would itself fail. These tests need to be verified against actual behavior.

The fix:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

### 5. `t.join()` Timeout Is Silently Ignored

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` does not raise if the thread is still alive after the timeout — it returns silently. If the heartbeat thread is stuck (e.g., blocked on a Redis call that's hanging), the lock release proceeds anyway while the heartbeat thread is still running and may extend the lock after release, potentially re-extending a lock that has already been legitimately acquired by another process. Add an explicit liveness check:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not proceed silently — alert or raise
```

### 6. Retry Backoff Is Insufficient Under Real Flash-Sale Load

```python
backoff = (2 ** attempt) * 0.1  # 0.1s, 0.2s, 0.4s — 3 attempts max
```

With a 30-second lock TTL and a heartbeat keeping it alive indefinitely, a caller that fails to acquire the lock after 0.7 seconds total wait time gets a hard failure. Under flash-sale conditions with hundreds of concurrent requests, nearly all of them will fail immediately. Consider a configurable retry budget with jitter, or expose retry parameters to the caller. At minimum, document this as an intentional design choice with the expected concurrency profile.

## Test Suite Issues

### 7. Tests Do Not Validate Heartbeat Behavior

The heartbeat is the most complex and novel part of this implementation, and there is zero test coverage for it. Given the bug identified in Issue #2, this is a significant gap. Tests should cover:

- Heartbeat fires at the correct interval (not immediately).
- Heartbeat stops when the lock context exits.
- Heartbeat stops when the extension returns 0.
- The main thread is not blocked by a stuck heartbeat beyond the join timeout.

### 8. `test_lock_release_is_atomic` Does Not Test What It Claims

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is the registered script object returned by `redis_client.register_script()`. Since `redis_client` is a `MagicMock`, `register_script()` returns another `MagicMock`. The test is asserting that a mock was called, not that the Lua script was actually executed with the correct keys and args. This test would pass even if the release logic were completely broken. Assert on the actual call arguments:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.ANY]  # token is a UUID, verify it's a string at minimum
)
```

### 9. `test_input_validation_rejects_*` Tests Are Likely Broken

As noted in Issue #4, these tests assert `result.success is False`, but `reserve_item` will raise `pydantic.ValidationError` rather than returning a result. The tests should either expect the exception or the code should be fixed to catch it and return a result — but both the code and the tests cannot be correct simultaneously in their current state.
