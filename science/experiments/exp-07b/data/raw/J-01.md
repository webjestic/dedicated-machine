Decision: Request Changes

Summary: The core distributed locking architecture is solid — atomic Lua scripts, scoped heartbeat state, exponential backoff, and Pydantic validation are all well-executed. However, there are several correctness and security issues that must be addressed before this merges: a race condition between stock read and the database transaction, a critical flaw in the heartbeat logic that can silently allow the lock to expire, and test coverage gaps that mask real failure modes. The issues below range from data integrity bugs to logic errors that would surface under production load.

---

## Issues

### 🔴 Critical — Data Integrity

**1. Stock read occurs outside the database transaction**

```python
# Current — read is outside the transaction boundary
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside tx
        self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` runs outside the database transaction. If the database supports any concurrent readers (e.g., a reporting query, a different service path, or a future code path that bypasses the lock), the stock value read can be stale by the time `update_stock` commits. The read must be inside the transaction, and `update_stock` should use an optimistic or pessimistic row-level lock at the database layer (e.g., `SELECT FOR UPDATE`) rather than relying solely on the Redis lock.

```python
# Recommended
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 🔴 Critical — Logic / Correctness

**2. Heartbeat `threading.Event.wait()` misuse — lock can silently expire**

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`threading.Event.wait(timeout=N)` returns `True` if the event is **set** and `False` on timeout. Since `heartbeat_running` is set (`.set()`) before the thread starts, `wait()` returns `True` immediately on every iteration without sleeping. The heartbeat fires in a tight loop, hammering Redis continuously rather than sleeping for `HEARTBEAT_INTERVAL` seconds between renewals.

The intent is to sleep between beats. Use `threading.Event.wait()` as a cancellation signal with `time.sleep`, or invert the event semantics:

```python
# Option A — use a stop event (clear = running, set = stop)
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait returns False on timeout (keep running), True if set (stop)
        result = self._extend_script(...)
        ...

# In finally:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 🟠 High — Security

**3. Pydantic validation errors are not caught — unvalidated input reaches callers as an unhandled exception**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
```

If `item_id` or `quantity` fails Pydantic validation, a `ValidationError` is raised and propagates uncaught to the caller. The test `test_input_validation_rejects_invalid_item_id` asserts `result.success is False`, but this test will actually **pass for the wrong reason** — it passes because the mock Redis lock is acquired and the mock DB returns stock, but the Pydantic model raises before that. The exception propagates rather than returning a `ReservationResult`. Catch `ValidationError` explicitly:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

**4. Item ID reflected into log messages without sanitization**

```python
logger.info("Reserved item_id=%s ...", req.item_id, ...)
return ReservationResult(..., message=f"Item {req.item_id} not found")
```

Even with the regex pattern on `item_id`, embedding user-supplied input directly into structured log fields and API response messages is a log injection risk if the logging backend or downstream consumer doesn't escape output. The Pydantic pattern `r'^[a-zA-Z0-9_\-]+$'` does constrain the character set, which mitigates the worst cases, but the message field in `ReservationResult` should use a generic string rather than interpolating the item ID directly into user-facing responses.

### 🟡 Medium — Reliability

**5. `t.join()` timeout is too short and join result is not checked**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is blocked on a slow Redis call, `join()` can time out and the thread continues running after `_release_script` executes. The thread may then attempt to extend a lock that has already been released and reacquired by another holder, corrupting the new holder's TTL. Check the join result:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    # Do not release — the lock state is uncertain; alert and fail safe
```

**6. `db.transaction()` exceptions are swallowed by the outer `LockAcquisitionError` handler**

The `except LockAcquisitionError` block is the only exception handler. If `db.transaction()`, `db.update_stock()`, or `db.create_order()` raises (e.g., a deadlock or connection error), the exception propagates out of `reserve_item` entirely — no `ReservationResult` is returned, and the lock is still released correctly (via `finally` in `_acquire_lock`), but the caller receives an unhandled exception instead of a structured result. Add a handler:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

### 🟡 Medium — Test Coverage

**7. Tests mock away the exact behavior being tested**

`test_lock_release_is_atomic` asserts `service._release_script.assert_called_once()`, but `_release_script` is a registered Redis script set up in `__init__`. The mock `redis_client` returns a `MagicMock` for `register_script`, so `_release_script` is already a mock — this test verifies that a mock was called, not that the Lua script path executes correctly. This test provides false confidence.

Additionally, `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will fail or behave incorrectly once issue #3 is fixed (they currently rely on the exception propagating through mock infrastructure accidentally). These tests need to be rewritten against the explicit `ValidationError` catch.

**8. No test for heartbeat thread behavior or lock extension**

Given that the heartbeat is a core reliability feature, there should be at least one test verifying that `_extend_script` is called after the interval elapses and that the thread stops cleanly when the context exits.

---

### 🔵 Minor — Style / Maintainability

**9. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants but not configurable**

For a government-grade service under varying load profiles, these should be injectable via constructor parameters with the class constants as defaults. Hardcoded timing values make tuning for different deployment environments require a code change.

**10. `retry_attempts` default is not exposed at the service level**

The `reserve_item` public API has no way to configure retry behavior. `retry_attempts=3` is hardcoded in `_acquire_lock`. This should either be a constructor parameter or a `reserve_item` argument.
