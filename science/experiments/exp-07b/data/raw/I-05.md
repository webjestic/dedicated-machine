Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a latent race condition in the heartbeat loop, a critical TOCTOU gap between the stock read and the database write, and test coverage that doesn't actually exercise the real code paths it claims to validate. None of these are cosmetic; in a flash sale context under real concurrency, they represent failure modes that will manifest.

---

## Findings

### 1. CRITICAL — TOCTOU Gap: Stock Read Occurs Outside the Database Transaction

**File:** `services/inventory.py`, `reserve_item()`

```python
# CURRENT — stock read is OUTSIDE the transaction
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # read here
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # write here
        self.db.create_order(...)
```

The distributed lock serializes callers at the application layer, but `get_stock` executes outside the database transaction boundary. If the database uses any isolation level below `SERIALIZABLE`, a concurrent actor operating outside this lock (a manual adjustment, a bulk import, a different service) can modify stock between the read and the transactional write. The lock protects against other instances of *this service*, but not the broader system.

**Required fix:** Pull `get_stock` inside the transaction so the read and write are atomic at the database level as well.

```python
with self.db.transaction():
    current_stock = self.db.get_stock(req.item_id)
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    order_id = str(uuid.uuid4())
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

Early returns inside a context manager require care — ensure the transaction context manager handles rollback on non-exceptional exits correctly, or restructure with explicit commit/rollback.

### 2. HIGH — Heartbeat Race: `threading.Event.wait()` Misused as Loop Gate

**File:** `services/inventory.py`, `_beat()`

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            heartbeat_running.clear()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

`threading.Event.wait(timeout=N)` returns `True` if the event is *set* and `False` if the timeout expires. Because `heartbeat_running` is set at construction and only cleared on shutdown or stolen lock, `wait()` returns `True` immediately on every iteration — the thread does not sleep at all. It hammers Redis in a tight loop until the lock is released.

The intent was almost certainly `time.sleep(HEARTBEAT_INTERVAL)` followed by a check, or use of `Event.wait()` inverted — waiting on a *stop* event rather than a *running* event.

**Required fix:**

```python
stop_event = threading.Event()  # cleared = running, set = stop

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In finally block:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

Now `wait()` blocks for `HEARTBEAT_INTERVAL` seconds (returns `False` on timeout, loop continues) or returns `True` immediately when `stop_event` is set (loop exits).

### 3. HIGH — Unhandled Exception in `reserve_item` Leaks Lock State and Crashes Silently

**File:** `services/inventory.py`, `reserve_item()`

Only `LockAcquisitionError` is caught. Any exception from `db.get_stock`, `db.update_stock`, `db.create_order`, or the transaction context manager propagates uncaught out of `reserve_item`. The `_acquire_lock` `finally` block will still release the Redis lock, which is correct, but the caller receives an unhandled exception rather than a `ReservationResult`. In an API context this likely surfaces as a 500 with a stack trace. The contract of the method — always returning a `ReservationResult` — is broken.

```python
except LockAcquisitionError as e:
    ...
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="An internal error occurred.")
```

### 4. MEDIUM — Pydantic Validation Exceptions Not Caught at the Service Boundary

**File:** `services/inventory.py`, `reserve_item()`

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

`ReservationRequest(...)` raises `pydantic.ValidationError` on invalid input. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, implying the method returns a `ReservationResult`. It does not — it raises. The tests pass only because the mock infrastructure swallows the exception or the test framework is misconfigured.

**Required fix:**

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters.")
```

### 5. MEDIUM — Heartbeat Thread Not Joined Before Lock Release on Stolen Lock Path

**File:** `services/inventory.py`, `_beat()`

When the heartbeat detects a stolen lock (`result == 0`), it calls `heartbeat_running.clear()` and returns. The `finally` block then calls `heartbeat_running.clear()` again (harmless) and `t.join(timeout=...)`. This path is fine. However, the `finally` block unconditionally calls `self._release_script(...)` after the join. If the lock was stolen, this script correctly returns `0` because the token no longer matches — the Lua guard holds. This is safe but should be explicitly documented, as the silent `0` return could be misread as a bug during incident response.

---

### 6. MEDIUM — `retry_attempts` Default Hardcoded; No Configurable Backoff Cap

**File:** `services/inventory.py`, `_acquire_lock()`

```python
backoff = (2 ** attempt) * 0.1
```

With `retry_attempts=3`, the maximum backoff is `0.4s` — acceptable. But `retry_attempts` is an exposed parameter with no documented upper bound. A caller passing `retry_attempts=20` produces a `52-second` blocking wait in the worst case on a synchronous request thread. This should either be capped internally or documented explicitly.

---

### 7. LOW — Tests Do Not Actually Validate the Heartbeat Behavior

**File:** `tests/test_inventory.py`

There is no test that:
- Confirms the heartbeat fires at the correct interval
- Confirms the heartbeat stops after lock release
- Confirms the heartbeat thread does not outlive the context manager

Given finding #2 (the heartbeat is currently a tight loop), this gap allowed the defect to pass review. A test using `threading` inspection or a mock `time.sleep` / `Event` would have caught it.

---

### 8. LOW — `test_lock_release_is_atomic` Tests a Mock, Not Behavior

**File:** `tests/test_inventory.py`

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is a `MagicMock` because `redis_client.register_script` returns a mock. This test asserts that a mock was called — it provides no guarantee about the actual Lua script content or that the real Redis client would behave atomically. This is a test of mock wiring, not of correctness. It should be supplemented with an integration test against a real or embedded Redis instance (e.g., `fakeredis`).
