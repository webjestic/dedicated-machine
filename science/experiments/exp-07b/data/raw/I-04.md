Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices for this use case. However, there are several correctness flaws, a meaningful architectural gap between the lock boundary and the database transaction, and test coverage that does not actually exercise the code paths it claims to validate. None of these are cosmetic; they represent real failure modes under production load.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and Database Transaction

This is the most significant flaw in the design. The stock read happens *inside* the lock boundary, which is correct, but the database transaction is a *separate* operation that does not hold a database-level row lock on the inventory record. If the database is shared with any other writer that does not participate in this Redis lock protocol — a backoffice tool, a migration script, a second service — the stock check and the write are not atomic at the database layer.

```python
# Inside the lock — good
current_stock = self.db.get_stock(req.item_id)

# ... logic ...

# Separate transaction — not protected from non-lock-aware writers
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read should be a `SELECT ... FOR UPDATE` (or equivalent pessimistic row lock) *inside* the database transaction, or the update should be expressed as a conditional decrement (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?`) with the affected row count checked. The Redis lock serializes callers through *this service*, but it provides zero protection against writers that bypass it.

---

### 2. Heartbeat `threading.Event.wait` Semantics Are Inverted

`threading.Event.wait(timeout)` returns `True` if the event is *set* before the timeout, and `False` if the timeout expires. The heartbeat loop uses `.set()` to mean "keep running" and `.clear()` to mean "stop." This means the loop body executes immediately every time the event is set — which is always, until `.clear()` is called. The intent is to sleep for `HEARTBEAT_INTERVAL` seconds between extensions, but the actual behavior is a tight busy-loop that hammers Redis with `EXPIRE` calls continuously for the entire duration the lock is held.

```python
# Current — fires continuously because the event is always set
while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
    ...
```

The correct pattern for a periodic background thread is either `time.sleep` with a stop flag checked afterward, or using `Event.wait` with the semantics inverted — set means "stop," clear means "keep running" — so that `wait` blocks for the full interval unless interrupted.

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In finally:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

This is a correctness bug that also has operational consequences: under any meaningful load, this will saturate the Redis connection pool.

### 3. Pydantic Validation Exceptions Are Not Caught

`ReservationRequest(item_id=item_id, quantity=quantity)` raises `pydantic.ValidationError` on invalid input. That exception is not caught anywhere in `reserve_item`. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but in reality these calls will raise an unhandled exception and propagate to the caller. The tests pass only because the mock infrastructure swallows the exception or the test framework is not configured to catch it — this needs to be verified, but the production code path is clearly wrong.

```python
# This raises ValidationError, it does not return a ReservationResult
req = ReservationRequest(item_id="../../admin", quantity=1)
```

Wrap the construction in a `try/except ValidationError` block and return an appropriate `ReservationResult(success=False, message="Invalid input")`.

## Significant Issues

### 4. `get_stock` Is Outside the Database Transaction

The stock read occurs before `with self.db.transaction()`. This means the read is not part of the atomic unit of work. Even within the Redis lock boundary, if the database has autocommit behavior or read-committed isolation, the value read by `get_stock` may not reflect the state that `update_stock` will act on. Move `get_stock` inside the transaction block, or use the conditional update pattern described in issue 1.

---

### 5. Heartbeat Thread Is Joined *Before* Lock Release, Creating a Delay Window

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)       # waits up to 9 seconds
    self._release_script(keys=[lock_key], args=[token]) # release happens after
```

The lock is held for up to `HEARTBEAT_INTERVAL + 1` seconds (9 seconds) after the critical section completes while waiting for the heartbeat thread to finish its current sleep. The lock should be released first, then the thread joined. The heartbeat thread must also be written to tolerate the key already being gone when it wakes.

```python
finally:
    stop_event.set()
    self._release_script(keys=[lock_key], args=[token])  # release immediately
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)          # then clean up thread
```

---

### 6. No Handling of Redis Connectivity Failures

`self.redis.set(...)`, `self._extend_script(...)`, and `self._release_script(...)` can all raise `redis.RedisError`. None of these are caught. A Redis outage during a flash sale will surface as an unhandled exception rather than a graceful degradation. At minimum, Redis errors during lock acquisition should be caught and converted to `LockAcquisitionError`. Errors during release should be logged and not re-raised, since the operation has already completed.

---

### 7. `retry_attempts=3` Produces Only 3 Attempts, Not 3 Retries

```python
for attempt in range(retry_attempts):  # 0, 1, 2
```

The parameter is named `retry_attempts` which implies retries after the first try, but `range(3)` gives three total attempts including the first. The naming and the error message ("after 3 attempts") are consistent with each other but the parameter name is misleading. This is minor but worth correcting for API clarity — either rename to `max_attempts` or adjust the range to `range(retry_attempts + 1)`.

## Test Coverage Gaps

### `test_lock_release_is_atomic` Does Not Test What It Claims

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is a mock registered via `redis_client.register_script(...)`. The mock returns another mock by default, so `assert_called_once()` will pass regardless of whether the release script was actually invoked with correct arguments. This test provides no meaningful assurance. It should assert the script was called with the correct `keys` and `args`, and should verify the token passed to release matches the token used during acquisition.

### Heartbeat Behavior Is Completely Untested

There are no tests for: the heartbeat thread extending the TTL, the heartbeat stopping when the lock is lost, the thread being joined before the context manager exits, or the behavior when `_extend_script` returns 0. Given that the heartbeat contains a correctness bug (issue 2), the absence of these tests allowed the bug to go undetected.

### No Test for Redis Failure During Acquisition or Release

Given that Redis connectivity is a single point of failure for this entire service, there should be explicit tests for `redis.RedisError` being raised during `set`, during `_extend_script`, and during `_release_script`.

### `test_reserve_concurrent_quantity` Is Not Actually Concurrent

The test name implies concurrency but it is a single-threaded sequential call. A meaningful concurrency test would use `threading.Barrier` or `concurrent.futures.ThreadPoolExecutor` to verify that two simultaneous reservations of the last 3 units do not both succeed.

## Minor Notes

- The `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 8` constants are class-level but not configurable at construction time. For a service deployed across environments with different Redis latency profiles, these should be constructor parameters with defaults.
- `order_id` is generated with `uuid.uuid4()` before the transaction opens. If the transaction fails and is retried externally, a new call will generate a new `order_id`, which is correct. However, if the caller retries `reserve_item` after a failure, there is no idempotency key mechanism to prevent double-reservation. This is worth a design conversation for the flash sale use case specifically.
- The `db` abstraction is entirely undefined in this PR. The contract assumed by `InventoryService` (`get_stock`, `update_stock`, `create_order`, `transaction`) should be expressed as a Protocol or abstract base class so the interface is explicit and testable.
