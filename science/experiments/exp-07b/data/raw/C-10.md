Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical race condition, architectural gaps that would cause silent data corruption or lock safety violations under production load, and a test suite that does not actually validate the behaviors it claims to test. None of these are cosmetic — they represent real failure modes in a high-concurrency flash sale context.

---

## Critical Issues

### 1. Race Condition: Stock Read Outside the Database Transaction

**File:** `services/inventory.py`, `reserve_item()`

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # writes based on stale read
    self.db.create_order(...)
```

The stock read happens outside the database transaction. The Redis lock serializes callers *through this service*, but it provides zero protection against:

- Direct database writes from other services, admin tooling, batch jobs, or other application nodes that do not participate in this lock protocol.
- Any scenario where the Redis lock is lost between the read and the write (e.g., Redis failover, network partition, or a GC pause that causes the heartbeat to miss its window).

The correct pattern is to move the read inside the transaction and use a pessimistic database-level lock (`SELECT ... FOR UPDATE`) or an optimistic CAS update (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?` and check rows affected). The Redis lock alone is not a substitute for database-level consistency guarantees.

### 2. Heartbeat Thread Termination Does Not Prevent Lock Release

**File:** `services/inventory.py`, `_acquire_lock()`

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            heartbeat_running.clear()
            return  # <-- heartbeat stops here
```

When the heartbeat detects that the lock has been lost (result == 0), it clears the event and returns. This is correct. However, the `finally` block in the context manager then unconditionally calls `self._release_script(keys=[lock_key], args=[token])`. At this point, the lock has already expired and may have been re-acquired by another holder with a different token. The Lua release script correctly guards against this with the token check, so the delete will be a no-op — but the `yield` continues executing, meaning the critical section proceeds *after lock expiry* with no signal to the caller that the lock was lost mid-operation. The database write will still be committed against data that another process may now also be modifying.

The heartbeat must have a mechanism to signal the critical section to abort. One approach is to raise an exception from the context manager or set a shared flag that `reserve_item` checks before committing.

### 3. Heartbeat `threading.Event.wait()` Behavior Is Misused

**File:** `services/inventory.py`, `_acquire_lock()`

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`threading.Event.wait(timeout)` returns `True` if the event is *set* within the timeout, and `False` if the timeout expires with the event still clear. The event is initialized with `.set()`, so on every iteration `wait()` returns `True` immediately — it does not sleep for `HEARTBEAT_INTERVAL` seconds. The heartbeat fires in a tight loop, hammering Redis with extension calls continuously until `heartbeat_running.clear()` is called.

The intended pattern here is `threading.Event` used as a cancellation token, with `time.sleep()` or a separate wait mechanism for the interval. The correct implementation would be:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock
        ...
```

Where `stop_event` is initially *clear* and gets `.set()` to signal shutdown. This way `wait()` blocks for the interval, returns `False` on timeout (continue), and returns `True` when set (stop).

### 4. Unhandled Exceptions in `reserve_item` Leak Lock State

**File:** `services/inventory.py`, `reserve_item()`

```python
try:
    with self._acquire_lock(lock_key):
        ...
        with self.db.transaction():
            self.db.update_stock(...)
            self.db.create_order(...)
except LockAcquisitionError as e:
    ...
```

Only `LockAcquisitionError` is caught. Any exception from `db.get_stock()`, `db.update_stock()`, `db.create_order()`, or the transaction context manager itself (network timeout, deadlock, constraint violation) will propagate uncaught to the caller. The `_acquire_lock` context manager's `finally` block will release the Redis lock, which is correct, but the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the stated return contract of the method and will cause 500-level errors in whatever API layer calls this. At minimum, a broad exception handler should catch database errors and return a structured failure result, and the exception should be logged with full context.

## Significant Issues

### 5. `t.join(timeout=...)` Does Not Guarantee Thread Termination

**File:** `services/inventory.py`, `_acquire_lock()`

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`t.join(timeout=...)` returns after the timeout regardless of whether the thread has actually stopped. If the heartbeat thread is blocked on a slow Redis call (e.g., Redis is under load or experiencing a network hiccup), the join will return, and the release script will execute while the heartbeat thread may still be alive and about to call `_extend_script`. This creates a narrow but real window where the lock is released and then re-extended by the stale heartbeat — potentially extending a lock that now belongs to a different holder. The thread should be checked with `t.is_alive()` after the join, and the situation should be logged as an error if the thread did not terminate cleanly.

---

### 6. No Redis Connection Error Handling

**File:** `services/inventory.py`

Redis calls — `self.redis.set()`, `self._extend_script()`, `self._release_script()` — can all raise `redis.RedisError` (connection refused, timeout, READONLY error during failover, etc.). None of these are caught anywhere. A Redis outage during a flash sale will produce unhandled exceptions that bypass the structured `ReservationResult` return type entirely. The lock acquisition path should catch `redis.RedisError` and raise `LockAcquisitionError` or return a structured failure. The heartbeat should catch and log Redis errors without crashing the thread silently.

---

### 7. Lock Granularity Is Correct but Undocumented as a Constraint

**File:** `services/inventory.py`

The lock key is `inventory:lock:{item_id}`, which is appropriate — per-item locking avoids a global bottleneck. However, there is no documentation or enforcement of the assumption that all nodes in the fleet use this same key scheme. If any caller path acquires the lock under a different key format, the mutual exclusion guarantee silently breaks. This should be enforced through a private static method or constant, not an f-string inline in `reserve_item`.

---

### 8. Redlock Is Not Implemented — Single Redis Node Is a SPOF

**File:** `services/inventory.py`

The PR uses a single Redis client. For a government-grade flash sale system handling high-value inventory, a single Redis node is a single point of failure. If Redis goes down, all reservations fail. If Redis experiences a failover (e.g., Sentinel promoting a replica), locks that existed on the primary may not have been replicated, meaning multiple clients can acquire the same lock simultaneously. The [Redlock algorithm](https://redis.io/docs/manual/patterns/distributed-locks/) exists precisely for this scenario. This should at minimum be called out as a known limitation with a tracking issue, not silently shipped as production-safe.

## Test Suite Issues

### 9. Tests Do Not Actually Exercise the Code They Claim to Test

**File:** `tests/test_inventory.py`

**`test_lock_release_is_atomic`:**
```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```
`service._release_script` is a registered script object returned by `redis_client.register_script()`. Since `redis_client` is a `MagicMock`, `register_script()` returns another `MagicMock`. The assertion `service._release_script.assert_called_once()` verifies that the mock was called, which it will be — but this does not validate that the Lua script logic is correct, that the token is passed correctly, or that the atomic behavior holds. This test would pass even if the implementation called `redis_client.delete()` directly, because `redis_client.delete` is also a mock and `assert_not_called()` only works if the code explicitly calls `redis_client.delete` — which it doesn't, so the assertion is trivially true regardless of implementation.

**`test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity`:**
```python
def test_input_validation_rejects_invalid_item_id(service):
    result = service.reserve_item("../../admin", quantity=1)
    assert result.success is False
```
These tests assert `result.success is False` but do not assert the failure reason or that a `ValidationError` was raised and caught. More importantly, the service does not catch `pydantic.ValidationError` — it will propagate as an unhandled exception
