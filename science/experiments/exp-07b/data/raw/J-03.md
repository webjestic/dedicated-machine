Decision: Request Changes

Summary: The core locking and reservation logic is well-structured — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and exponential backoff are all solid choices. However, there are several correctness, resilience, and security issues that must be addressed before this merges, ranging from a race condition in the heartbeat teardown to missing error handling around Redis and database failures. The test suite also has fixture wiring problems that mean key behaviors are not actually being exercised.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU gap between `get_stock` read and `update_stock` write**

The stock read happens outside the database transaction:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # inside transaction
```

If the database layer does not enforce a row-level lock during `get_stock`, a concurrent process that bypassed Redis (e.g., a non-flash-sale code path, a direct DB write, or a Redis failover) could read stale stock between these two calls. The read must either be inside the transaction with a `SELECT FOR UPDATE` / pessimistic lock, or `update_stock` must use a conditional update (`WHERE stock >= quantity`) and return affected rows so the caller can detect a lost race.

---

**2. Heartbeat thread can extend a lock the caller has already logically abandoned**

The `finally` block clears the event and then joins the thread, but there is a window:

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If the join times out (the thread is blocked on a slow Redis call), `_release_script` is called while the heartbeat thread may still be alive and attempting another `_extend_script`. The thread could then re-extend the lock after the Lua release script has already deleted it, meaning a new owner's lock gets silently extended with the old token — which will fail the ownership check, but the window itself is a correctness hazard. The join timeout should be shorter than `HEARTBEAT_INTERVAL`, and if the join times out, that condition must be logged as an error and the release must still proceed with a best-effort attempt.

---

**3. No exception handling around Redis operations**

`redis.exceptions.RedisError`, `ConnectionError`, and `TimeoutError` are never caught. A Redis blip during `_acquire_lock`, the heartbeat `_extend_script`, or `_release_script` will propagate as an unhandled exception. In the lock-release path specifically, an uncaught exception in `finally` will suppress the original exception from the `try` block, making failures very difficult to diagnose. Wrap Redis calls in the `finally` block with a bare `except Exception` at minimum, log the failure, and continue.

---

**4. No exception handling around database operations**

If `db.get_stock`, `db.update_stock`, or `db.create_order` raises, the exception propagates out of the `with self._acquire_lock` block, which is fine for lock release, but the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the contract of the public method. Database errors should be caught, logged with full context, and returned as a failed `ReservationResult` with a safe user-facing message. Do not leak internal exception messages to callers.

---

### 🟡 Significant

**5. `ValidationError` from Pydantic is not caught**

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails, `pydantic.ValidationError` is raised, not caught, and propagates to the caller. The test `test_input_validation_rejects_invalid_item_id` passes only because the test checks `result.success is False` — but with the current code, no `ReservationResult` is returned; an exception is raised. Either catch `ValidationError` and return a failed result, or document that callers are responsible for catching it (and update the tests accordingly).

**Verify this yourself:**
```python
# This will raise, not return a ReservationResult
service.reserve_item("../../admin", quantity=1)
```

---

**6. `HEARTBEAT_INTERVAL` is not derived from `LOCK_TTL`**

Both constants are hardcoded independently:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8   # comment says TTL/3, but 30/3 = 10, not 8
```

The comment claims `TTL/3` but the value is `8`, not `10`. If `LOCK_TTL` is changed, `HEARTBEAT_INTERVAL` will silently drift out of ratio. Define it as:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

---

**7. `retry_attempts` default is not configurable at the service level**

The retry count is a magic default buried in `_acquire_lock`'s signature. For a government-grade service under flash sale load, this should be a constructor parameter or a class constant, so it can be tuned per deployment without changing internal method signatures.

---

### 🟠 Test Suite Issues

**8. `redis_client` fixture does not wire `register_script` correctly**

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1   # ← this attribute is never called
    return mock
```

`register_script` returns a callable script object. `MagicMock` will auto-create it, but the returned callable's return value defaults to a `MagicMock`, not `1`. This means `_extend_script(...)` returns a `MagicMock`, and the heartbeat's `if result == 0` check will never be `True` — the heartbeat loss path is never tested. Explicitly configure:

```python
script_mock = MagicMock(return_value=1)
mock.register_script.return_value = script_mock
```

**9. `test_lock_release_is_atomic` asserts on `service._release_script` but the fixture wires it incorrectly**

Because `register_script` returns a new `MagicMock` each call, `service._release_script` is not the same object as `redis_client.register_script.return_value` unless explicitly set. The assertion `service._release_script.assert_called_once()` will pass vacuously on a fresh `MagicMock` even if the script was never called. Fix the fixture as described above and assert on the shared mock reference.

**10. No test for the heartbeat-lost code path**

There is no test that simulates `_extend_script` returning `0` mid-operation to verify the heartbeat thread stops cleanly and the lock is still released. This is a critical operational path.

**11. No test for Redis failure during lock acquisition or release**

Given finding #3, there should be tests that inject `redis.exceptions.RedisError` to confirm the service degrades gracefully.

---

### 🔵 Minor / Hardening

**12. `order_id` is generated before the transaction**

```python
order_id = str(uuid.uuid4())
with self.db.transaction():
    self.db.update_stock(...)
    self.db.create_order(order_id, ...)
```

This is fine functionally, but if the transaction is retried internally by the DB layer, the same `order_id` will be reused. Consider generating the ID inside the transaction, or ensure the DB layer does not perform transparent retries.

**13. Log message in `reserve_item` logs `item_id` from `req.item_id` (validated), which is correct — keep this**

This is a positive note: logging the validated value rather than the raw input is the right call. No change needed.
