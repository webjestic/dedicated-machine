Decision: Request Changes

Summary: The core distributed locking design is solid — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all well-executed. However, there are several correctness and reliability issues that must be resolved before this merges: a race condition between the DB read and transaction, a critical flaw in the heartbeat loop logic, and test coverage gaps that would let real bugs pass silently. None of these are cosmetic — they represent failure modes under the exact high-concurrency conditions this service is designed to handle.

---

## Detailed Review

### 🔴 Critical — Must Fix

**1. Stock read occurs outside the database transaction (TOCTOU)**

```python
# services/inventory.py
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
        self.db.create_order(...)
```

The `get_stock` call happens outside the database transaction boundary. If the DB supports snapshot isolation or if a concurrent process (e.g., a manual admin adjustment, a different code path not using this lock) modifies stock between the read and the `update_stock` write, you can silently oversell. The Redis lock protects against concurrent callers of *this* code path, but it does not protect against anything else touching the database.

**Fix:** Move `get_stock` inside the transaction, or use a `SELECT ... FOR UPDATE` / optimistic locking check at write time. At minimum, the transaction should be the outermost boundary:

```python
with self.db.transaction():
    current_stock = self.db.get_stock(req.item_id, for_update=True)
    # ... validation ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

**2. Heartbeat thread silently stops renewing if the lock is lost, but the caller continues executing**

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            heartbeat_running.clear()
            return   # ← thread exits, but the main thread keeps going
```

When `result == 0`, the heartbeat thread logs a warning and exits. The main thread is still inside the `with self._acquire_lock(lock_key):` block, operating under the assumption it holds the lock — but it no longer does. Another process may have acquired the lock and begun its own reservation. This is a silent correctness failure: you will proceed to write to the database without holding the lock, defeating the entire purpose of the locking mechanism.

**Fix:** Signal the main thread. Use a shared `threading.Event` for lock loss, check it at yield, or raise from the context manager. One clean approach:

```python
lock_lost = threading.Event()

def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.error("Heartbeat lost lock ownership key=%s — aborting", lock_key)
            lock_lost.set()
            heartbeat_running.clear()
            return

# In the context manager, after yield, or wrap yield:
try:
    yield token
    if lock_lost.is_set():
        raise LockAcquisitionError(f"Lock lost during operation for {lock_key}")
finally:
    ...
```

Ideally the check happens *before* the DB write commits, not after. Consider yielding a guard object the caller can check, or restructuring so the lock loss raises into the caller's context.

### 🟡 Significant — Should Fix

**3. Heartbeat `wait()` logic is inverted**

```python
while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
```

`threading.Event.wait(timeout)` returns `True` if the event is *set* (i.e., `heartbeat_running` is set), and `False` on timeout. The intent here appears to be: "sleep for `HEARTBEAT_INTERVAL` seconds, then renew." But the loop body runs *immediately* on the first iteration because the event is already set — there is no initial delay. The heartbeat fires at t=0, t=8, t=16, etc., rather than t=8, t=16, t=24. This means the first renewal happens before any meaningful elapsed time, wasting a Redis round-trip, and more importantly, the loop condition is semantically backwards from what a reader would expect.

**Fix:** Use `time.sleep` for the interval and check the event separately, which is clearer and correct:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout elapsed, event not set → still running
        result = self._extend_script(...)
        if result == 0:
            logger.error("Heartbeat lost lock ownership key=%s", lock_key)
            return
```

Or invert the event semantics: use a `stop_event` that starts unset, and loop `while not stop_event.wait(timeout=INTERVAL)`.

**4. Pydantic `ValidationError` is not caught in `reserve_item`**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise
```

If `item_id` or `quantity` fails validation, Pydantic raises `ValidationError`, which is not caught. The caller receives an unhandled exception rather than a `ReservationResult`. The test `test_input_validation_rejects_invalid_item_id` actually passes today only because Pydantic raises and pytest catches it as a non-`success` path — but the contract of this function (always return `ReservationResult`) is violated.

**Fix:**

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation input item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

### 🟠 Test Coverage Gaps

**5. `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` are testing an exception, not a result**

As noted above, these tests assert `result.success is False`, but the function currently raises `ValidationError` before returning anything. These tests are passing for the wrong reason. After fixing issue #4, re-verify these tests actually exercise the new catch block.

**6. Heartbeat behavior is entirely untested**

There is no test for:
- Lock loss mid-operation (heartbeat `result == 0` path)
- Heartbeat thread being joined before release
- Behavior when `_extend_script` raises a Redis connection error

Given that the heartbeat is the most operationally novel part of this PR, it needs direct test coverage.

**7. `test_lock_release_is_atomic` is testing mock internals, not behavior**

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

This test asserts that `_release_script` was called, but `_release_script` is assigned during `__init__` via `redis_client.register_script(...)`, which returns a mock. The test is verifying mock call counts on an object that doesn't execute the Lua script at all. This gives false confidence. Consider an integration test against a real Redis instance (e.g., `fakeredis`) to validate the actual Lua script semantics.

### 🔵 Minor / Informational

**8. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is reasonable, but undocumented**

The comment says "fires at TTL/3" (30/3 = 10, not 8). Either the constant or the comment is wrong. This is minor but will confuse the next engineer.

**9. No Redis connection error handling**

If Redis is unavailable during `_acquire_lock`, `redis.set(...)` will raise `redis.exceptions.ConnectionError`. This will propagate as an unhandled exception rather than a graceful `ReservationResult`. Consider wrapping the lock acquisition in a broader exception handler or letting it propagate explicitly with documentation that callers must handle it.

**10. `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` does not verify the thread actually stopped**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the thread is still alive after the join timeout (e.g., blocked on a slow Redis call), the lock is released while the heartbeat thread may still attempt to extend it. Add `if t.is_alive(): logger.error(...)` after the join.
