Decision: Request Changes

Summary: The core locking and reservation logic is well-structured — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and exponential backoff are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges, ranging from a race condition between the DB read and transaction to unhandled exceptions that could leave the system in an inconsistent state. The test suite also has coverage gaps and fixture wiring problems that mean the checklist claim of "tests pass" is suspect.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU race between `get_stock` and `update_stock`**

```python
# Inside the lock — but the DB read is outside the transaction
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # ← overwrites, doesn't CAS
    self.db.create_order(...)
```

`get_stock` is called outside the database transaction. If the DB supports snapshot isolation or if another process modifies stock between the read and the transactional write (e.g., a manual admin adjustment, a refund, or a second Redis cluster node during a failover), the write will silently corrupt stock. The Redis lock serializes *this service's* writers, but it does not serialize external DB writers.

**Recommendation:** Either (a) move `get_stock` inside the transaction and use a `SELECT ... FOR UPDATE` / optimistic CAS update (`UPDATE items SET stock = stock - ? WHERE id = ? AND stock >= ?` returning rows affected), or (b) document explicitly that this DB layer must enforce the constraint and add an assertion on rows-affected inside the transaction.

---

**2. Unhandled exceptions inside the lock leave the lock held longer than necessary and swallow errors**

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # can throw
    ...
    with self.db.transaction():                        # can throw
        self.db.update_stock(...)                      # can throw
```

Any unexpected exception (DB connection drop, timeout, etc.) propagates out of `reserve_item` as an unhandled exception. The `_acquire_lock` context manager's `finally` block *will* release the lock, which is correct, but `reserve_item` itself has no `except Exception` guard, so the caller receives a raw exception instead of a `ReservationResult`. For a service boundary this is a contract violation — callers expect a typed result, not a surprise stack trace.

**Recommendation:** Add a broad `except Exception` catch inside `reserve_item` (outside the lock context) that logs and returns a safe `ReservationResult(success=False, message="Internal error")`.

---

### 🟠 High

**3. Heartbeat thread can outlive the lock release on `t.join` timeout**

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # = 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is blocked (e.g., Redis is slow), `t.join` times out silently and `_release_script` is called anyway. The thread is a daemon so it will eventually die, but there is a window where the thread wakes up *after* `_release_script` has run, calls `_extend_script` on a key that now belongs to a different owner, and the Lua script correctly returns `0` — but the log line `"Heartbeat lost lock ownership"` will fire spuriously and could mask a real problem. More importantly, if the join times out, you have no signal that something is wrong.

**Recommendation:** Log a `logger.error` (not just silently continue) if `t.is_alive()` after `join`. Consider a shorter join timeout equal to `HEARTBEAT_INTERVAL` and treat a live thread post-join as a critical alert.

---

**4. `ValidationError` from Pydantic is not caught in `reserve_item`**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # raises ValidationError
```

The docstring and return type both promise a `ReservationResult`, but invalid input raises `pydantic.ValidationError` instead. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will currently *fail* because they assert `result.success is False` but will actually receive an exception.

**Recommendation:**
```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    return ReservationResult(success=False, message=f"Invalid input: {e}")
```

---

**5. `LOCK_TTL = 30` seconds is dangerously long for a flash sale hot path**

A 30-second lock on a per-item key means a single slow DB operation (or a GC pause, or a network hiccup) blocks *all* buyers for that item for up to 30 seconds. For a flash sale this is a denial-of-service vector — one unlucky request starves the queue.

**Recommendation:** Reduce `LOCK_TTL` to 5–10 seconds. The heartbeat exists precisely to extend it when needed, so the initial TTL is a safety ceiling, not an expected duration. Document the reasoning.

### 🟡 Medium

**6. Test fixture `redis_client` is not wired correctly — `register_script` returns a mock, not the scripts used in production**

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1   # ← this attribute is never called
    return mock
```

`InventoryService.__init__` calls `redis_client.register_script(...)` twice, which on a `MagicMock` returns a new `MagicMock` each time. Those returned mocks are stored as `self._release_script` and `self._extend_script`. The test `test_lock_release_is_atomic` then asserts `service._release_script.assert_called_once()` — but `service._release_script` is the mock returned by `register_script`, not `redis_client._release_script`. This test likely passes vacuously because `MagicMock` auto-creates attributes, but it is not testing what it claims to test.

**Recommendation:** Explicitly configure `register_script` return values and assert on the correct mock objects:
```python
mock_release = MagicMock(return_value=1)
mock_extend = MagicMock(return_value=1)
redis_client.register_script.side_effect = [mock_release, mock_extend]
```

**7. No test for heartbeat renewal behavior**

The heartbeat is the most novel and failure-prone component. There is no test that verifies the extend script is called during a long-running operation, or that the heartbeat stops after lock release.

**8. No test for DB exception during transaction**

There is no test covering the case where `db.update_stock` or `db.create_order` raises — which is the most important failure mode for data integrity.

---

### 🔵 Low / Nits

**9. `quantity` default of `1` in `reserve_item` signature is inconsistent with `ReservationRequest` having no default**

Minor API inconsistency. Not a bug, but worth aligning.

**10. Log line on successful reservation logs `item_id` from `req.item_id` (validated) — good. But the `LockAcquisitionError` log uses the pre-validation `item_id` indirectly via `req.item_id` which is already validated at that point — this is fine, just worth noting the validation happens before the lock attempt.**

**11. `HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` means the first renewal fires at 8s, well within the 30s window — correct. But if `LOCK_TTL` is reduced per finding #5, verify the ratio is maintained (TTL/3 or better).**
