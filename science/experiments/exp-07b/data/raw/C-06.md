Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical race condition, architectural concerns, and test suite deficiencies that must be addressed before this can be approved for a high-concurrency flash sale context.

---

## Critical Issues

### 1. Race Condition: Stock Read Outside the Database Transaction

This is the most serious flaw in the implementation.

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

if current_stock < req.quantity:
    return ReservationResult(success=False, ...)

new_stock = current_stock - req.quantity

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The Redis lock serializes access *across service instances*, but it does nothing to protect against the database itself being modified between the read and the write. If the database supports concurrent connections (and it almost certainly does), another process, a background job, an admin correction, or a different code path not using this lock can modify stock between `get_stock` and `update_stock`. The read **must** occur inside the transaction, and ideally the `UPDATE` should use a conditional `WHERE stock >= quantity` with row-level locking (`SELECT FOR UPDATE`) to make the decrement atomic at the database level. The Redis lock alone is not sufficient.

```python
# Correct pattern:
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(...)
```

### 2. Heartbeat Thread `wait()` Semantics Are Inverted — Heartbeat Fires Immediately on Stop

```python
heartbeat_running = threading.Event()
heartbeat_running.set()  # Event is SET = True

def _beat():
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...
```

`Event.wait(timeout)` returns `True` if the event is **set**, and returns `False` only after the timeout elapses with the event **cleared**. Since the event starts set, `wait()` returns `True` immediately on every iteration — the heartbeat fires in a tight loop with no delay whatsoever until the event is cleared. This will hammer Redis continuously for the entire duration of the lock.

The correct pattern for a periodic heartbeat with clean shutdown is:

```python
stop_event = threading.Event()  # starts cleared

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...
    # stop_event was set — exit cleanly

# To stop:
stop_event.set()
```

### 3. Unhandled Exceptions in `reserve_item` — DB Errors Propagate Uncaught

Only `LockAcquisitionError` is caught. Any exception from `db.get_stock`, `db.update_stock`, `db.create_order`, or `db.transaction()` — including network timeouts, deadlocks, or serialization failures — will propagate out of `reserve_item` as an unhandled exception. In a flash sale context this will surface as a 500 to the caller and, more importantly, the lock will still be released correctly (the `finally` block handles that), but the caller has no graceful error response. A `except Exception` catch with appropriate logging and a `ReservationResult(success=False, message="Internal error")` return is required.

## Significant Issues

### 4. `threading.Event.wait()` Used as a Sleep — Thread Join Timeout Is Too Tight

Related to issue #2, but separately: the `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` gives the thread only 9 seconds to stop. If the `_extend_script` call to Redis is slow or hanging (network partition, Redis overload), the join will time out and the lock release proceeds with the heartbeat thread still running. That thread now holds a reference to `token` and `lock_key` and will continue attempting to extend a lock that has already been released and potentially re-acquired by another holder. This is a correctness violation. The thread must be made interruptible and the join failure must be logged as a critical error.

### 5. Lock TTL and Heartbeat Interval Are Module-Level Constants With No Validation

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at TTL/3
```

These are hardcoded class attributes with no enforcement of the invariant `HEARTBEAT_INTERVAL < LOCK_TTL / 2`. If someone adjusts these values (and they will), the heartbeat could fire after the lock has already expired. These should be constructor parameters with an assertion:

```python
assert self.heartbeat_interval < self.lock_ttl / 2, \
    "Heartbeat interval must be less than half the lock TTL"
```

### 6. `_acquire_lock` Yields `token` But `reserve_item` Ignores It

```python
with self._acquire_lock(lock_key):  # token yielded but discarded
```

This is harmless today, but it means the caller has no way to verify lock ownership if the design ever evolves. It also signals the yield was an afterthought. Either use it or don't yield it.

### 7. No Redis Connection Error Handling at Lock Acquisition

If Redis is unavailable, `self.redis.set(...)` raises `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`. These are not caught and will propagate as unhandled exceptions rather than being converted to a `LockAcquisitionError`. In a flash sale scenario, a Redis blip should return a graceful "service unavailable" to the caller, not a stack trace.

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.exceptions.RedisError as e:
    logger.error("Redis error during lock acquisition: %s", e)
    raise LockAcquisitionError(f"Redis unavailable: {e}") from e
```

## Test Suite Issues

### 8. Tests Do Not Validate the Heartbeat Behavior At All

The heartbeat is the most operationally complex part of this implementation and has a critical bug (issue #2). There is zero test coverage for it. At minimum there should be tests verifying:
- The heartbeat calls `_extend_script` at the correct interval
- The heartbeat stops when the event is cleared
- The heartbeat stops when the lock is lost (`result == 0`)

### 9. `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` Will Raise, Not Return `False`

Pydantic `ValidationError` is raised by the `ReservationRequest` constructor. It is not caught in `reserve_item`, so these tests will fail with an unhandled exception rather than returning `ReservationResult(success=False)`. Either the service must catch `ValidationError` and return a structured result, or the tests must use `pytest.raises`. Currently neither is done, so the checklist item "Tests pass" is incorrect for these cases.

### 10. `redis_client` Fixture Has a Phantom `execute_script` Mock

```python
mock.execute_script.return_value = 1
```

`execute_script` is not a method on `redis.Redis`. The registered scripts are called as callables. This mock does nothing and gives false confidence. The `_release_script` and `_extend_script` attributes on the service instance need to be mocked directly after construction, as `test_lock_release_is_atomic` partially attempts but inconsistently.

### 11. Concurrency Is Not Tested

For a service explicitly designed for high-concurrency flash sales, there are no tests that spin up multiple threads and verify that stock never goes negative. This is the primary correctness guarantee of the entire design and it is completely untested.

---

## Minor Issues

### 12. `new_stock` Can Go Negative If Stock Is Modified Concurrently

Directly related to issue #1. If the read-modify-write is not atomic at the DB level, `new_stock` can be written as a negative value. A database-level `CHECK (stock >= 0)` constraint should be the last line of defense, and its violation should be handled gracefully.

### 13. `logger.info("Lock released")` Fires Even When Release Script Returns 0

If the Lua release script returns `0` (lock was already expired or stolen), the code logs "Lock released" anyway. This is misleading. The return value of `_release_script` should be checked and logged appropriately.

```python
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock release no-op — lock was already expired or stolen key=%s", lock_key)
```
