Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all well-executed. However, there are several correctness and reliability issues that must be addressed before this merges, ranging from a race condition between the DB read and transaction to a critical flaw in the heartbeat loop logic and gaps in test coverage that would allow those bugs to ship undetected.

---

## Detailed Findings

### 🔴 Critical

**1. Heartbeat loop exits immediately on first iteration (logic error)**

```python
# Current — BROKEN
while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
```

`threading.Event.wait(timeout=N)` returns `True` if the event is *set* (i.e., the flag is `True`) and `False` on timeout. Since `heartbeat_running` starts *set*, `wait()` returns `True` immediately on every call — the thread never sleeps. It will spin-extend the lock in a tight loop, hammering Redis until the lock is released.

The intent is to sleep for `HEARTBEAT_INTERVAL` seconds between extensions. The correct pattern uses a *stop* event (cleared to signal shutdown):

```python
stop_heartbeat = threading.Event()  # starts cleared

def _beat() -> None:
    while not stop_heartbeat.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In finally:
stop_heartbeat.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`wait(timeout=N)` on a cleared event blocks for N seconds and returns `False`, driving the sleep. Setting the event causes it to return `True`, exiting the loop cleanly.

**2. Stock read is outside the database transaction (TOCTOU window)**

```python
# Lock is held, but the read is outside the transaction boundary
current_stock = self.db.get_stock(req.item_id)
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # writes stale value
    self.db.create_order(...)
```

The Redis lock serializes *this service*, but if any other writer (a different service, a manual admin update, a migration) touches stock between `get_stock` and `update_stock`, you will silently write a stale value. The stock read must be inside the transaction and should use a pessimistic row lock (`SELECT ... FOR UPDATE`) to be safe:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

This also makes the Redis lock a performance optimization (reducing DB contention) rather than a correctness dependency — a much safer architecture.

### 🟡 Significant

**3. Unhandled exceptions inside the lock context leak the lock**

If `self.db.get_stock()` or the transaction raises an unexpected exception, it propagates through `reserve_item` and is not caught. The `finally` block in `_acquire_lock` *will* run and release the lock correctly — that part is fine. The problem is that `reserve_item` has no outer `except Exception` handler, so the caller receives a raw exception rather than a `ReservationResult`. For a service method returning a typed result, all failure modes should return a result object:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

---

**4. Pydantic `ValidationError` is silently unhandled**

`ReservationRequest(item_id=item_id, quantity=quantity)` raises `pydantic.ValidationError` on bad input. The test `test_input_validation_rejects_invalid_item_id` passes today only because the exception propagates and the test framework catches it — but the assertion `result.success is False` would actually fail if you run it, because no `ReservationResult` is returned. Add explicit handling:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation input: %s", e)
    return ReservationResult(success=False, message="Invalid input.")
```

---

**5. `t.join()` timeout is silently ignored**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

If the heartbeat thread is still running after the join timeout, execution continues and the lock is released anyway — but the thread is still alive and will attempt to extend a lock that no longer exists (or worse, one that has been re-acquired by another process with a different token). The Lua script guards against the wrong-token case, but a log warning here would at minimum surface the problem:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

### 🟠 Test Coverage Gaps

**6. The critical heartbeat bug has no test**

There is no test that verifies the heartbeat sleeps between extensions. A simple test using `time.sleep` mocking or checking call count after a short wall-clock delay would have caught the spin-loop immediately.

**7. `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` are incorrect**

As noted above, these tests assert `result.success is False`, but the current code raises `ValidationError` rather than returning a result. These tests either pass for the wrong reason or fail outright depending on the test runner configuration. They need to be updated once the `ValidationError` handler is added.

**8. No test for the TOCTOU scenario**

There is no test simulating a stock value changing between the read and the write. Once the fix in finding #2 is in place, a test should verify that `get_stock_for_update` is called inside the transaction.

**9. `test_lock_release_is_atomic` asserts on internal implementation detail incorrectly**

```python
service._release_script.assert_called_once()
```

`_release_script` is set during `__init__` via `register_script`, which returns a callable object on the mock. This assertion may pass vacuously. The test should verify the script was called with the correct `keys` and `args` to be meaningful.

---

### 🔵 Minor / Informational

**10. `retry_attempts` default is not configurable at the service level**

The default of `3` is hardcoded in `_acquire_lock`'s signature. For a flash sale scenario with high contention, operators may want to tune this without redeploying. Consider exposing it in `__init__` or via a config object.

**11. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants**

These should be instance-level or injected via config to allow per-environment tuning and easier testing without monkeypatching class attributes.

**12. Lock key namespace collision**

`f"inventory:lock:{req.item_id}"` — if `item_id` values are shared across environments (dev/staging/prod hitting the same Redis), locks will collide. Consider namespacing by environment: `f"{env}:inventory:lock:{req.item_id}"`.
