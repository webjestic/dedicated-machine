Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices for this use case. However, there are several correctness and architectural issues that need to be addressed before this can be approved, ranging from a race condition in the heartbeat termination sequence to a TOCTOU flaw between stock read and database write, to test coverage gaps that would allow these bugs to go undetected in CI.

---

## Critical Issues

### 1. Race Condition: Heartbeat Can Fire After Lock Release

**File:** `services/inventory.py`, `_acquire_lock` — `finally` block

```python
# Current
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

The sequence is: clear the event → join the thread → release the lock. The problem is that `heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL)` can return `False` (timeout expired) even after the event is cleared, and the thread will then attempt to call `_extend_script` *after* the `finally` block has already moved past `join` but before `_release_script` executes. More critically, if the join times out — which is possible under thread scheduling pressure — `_release_script` runs while the heartbeat thread is still alive and may still be calling `_extend_script`. This is a lock ownership integrity violation.

The fix is to release the lock *before* joining, or use a stop event that the heartbeat checks *before* its extend call, not after:

```python
finally:
    heartbeat_running.clear()
    # Release the lock first — token is now invalid, so a racing
    # extend call will get result == 0 and self-terminate cleanly.
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Lock released key=%s", lock_key)
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.warning("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

Once the lock is released, the token is gone. Any in-flight `_extend_script` call will return `0` and the thread will self-terminate. This makes the race benign rather than a correctness hazard.

### 2. TOCTOU: Stock Read Is Outside the Database Transaction

**File:** `services/inventory.py`, `reserve_item`

```python
# Outside the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens outside `self.db.transaction()`. The distributed lock protects against concurrent *Redis-aware* callers, but it does not protect against:

- Direct database writes that bypass this service (administrative tools, migrations, other services)
- Database replicas with replication lag if `get_stock` reads from a replica
- Any future code path that modifies stock without acquiring this lock

The stock read must be inside the transaction, and ideally should use a pessimistic row lock (`SELECT ... FOR UPDATE`) to make the check-and-decrement atomic at the database level. The distributed lock and the DB-level lock should be treated as complementary layers, not alternatives:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 3. Heartbeat `wait()` Semantics Are Inverted

**File:** `services/inventory.py`, `_beat`

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
```

`threading.Event.wait(timeout)` returns `True` if the event is *set* and `False` if the timeout expired. Since `heartbeat_running` is set at the start and cleared on shutdown, this loop will only execute the extend call if the event is set — meaning it fires *immediately* on the first iteration (the event is already set), then fires again immediately on the second iteration, and so on. There is no actual sleep between heartbeat pulses.

The intended pattern is to sleep for the interval and then check whether to continue. The correct idiom is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout elapsed, event still set → extend
        result = self._extend_script(
            keys=[lock_key],
            args=[token, str(self.LOCK_TTL)]
        )
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
    # wait() returned True → event was cleared → shutdown requested
```

Wait returns `False` on timeout (interval elapsed, keep going) and `True` when the event is set by `clear()` — except `clear()` unsets the event, so `wait()` will block until timeout. Let me be precise: the correct pattern for a cancellable sleep loop is:

```python
# stop_event is initially clear(); set it to signal shutdown
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # timeout elapsed without stop being signaled — do work
        ...

# To stop:
stop_event.set()
```

The current code uses a *set-to-run, clear-to-stop* event, which causes the `wait()` to return `True` immediately on every iteration rather than sleeping. This means the heartbeat is hammering Redis with extend calls in a tight loop rather than at `HEARTBEAT_INTERVAL` cadence. This is a functional bug and a potential Redis DoS vector under load.

## Moderate Issues

### 4. Unhandled Exceptions Inside the Lock Boundary

**File:** `services/inventory.py`, `reserve_item`

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unexpected exception, it will propagate through the `with self._acquire_lock(lock_key)` context manager. The `finally` block will correctly release the lock, which is good. However, the exception will then propagate out of `reserve_item` entirely — there is no catch for general `Exception`, meaning the caller receives an unhandled exception rather than a structured `ReservationResult`. For a service boundary this should return a controlled failure:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error during reservation")
```

### 5. Pydantic Validation Errors Are Not Caught

**File:** `services/inventory.py`, `reserve_item`

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails, Pydantic raises `ValidationError`. This is not caught. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but the actual behavior is an uncaught exception propagating to the caller. These tests are currently passing for the wrong reason or are not actually exercising the code path they claim to test. Validation errors must be caught explicitly:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

## Test Coverage Gaps

### 6. The Heartbeat Bug Is Not Caught by Tests

There is no test that verifies the heartbeat fires at the correct interval and not in a tight loop. A test using `time.sleep` and asserting the number of `_extend_script` calls within a bounded window would catch the inverted `wait()` semantics described in Issue 3.

### 7. `test_lock_release_is_atomic` Does Not Test What It Claims

**File:** `tests/test_inventory.py`

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is a registered script object on the mock Redis client. `register_script` on a `MagicMock` returns another `MagicMock`. Calling it records the call, but `assert_called_once()` on a child mock of a `MagicMock` will pass regardless of whether the actual Lua path was exercised. This test provides false confidence. Use a more explicit mock setup that verifies the script was registered with the correct Lua source and called with the expected `keys` and `args`.

### 8. No Test for Database Transaction Rollback Behavior

There is no test verifying that if `create_order` raises an exception after `update_stock` has been called, the transaction is rolled back and stock is not decremented. This is the most important correctness property of the transaction boundary and it has zero coverage.
