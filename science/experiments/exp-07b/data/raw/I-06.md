Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all commendable. However, there are several correctness flaws, a meaningful architectural gap, and test coverage holes that must be addressed before this can be approved for a high-concurrency flash sale context.

---

## Critical Issues

### 1. TOCTOU Race: Stock Read Occurs Outside the Database Transaction

This is the most significant flaw in the implementation.

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

# INSIDE the transaction:
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The Redis lock serializes callers *within this service*, but it provides no protection against:

- Direct database writes from other services, admin tooling, or batch jobs that do not participate in the Redis lock protocol.
- A database failover or Redis partition between the read and the write.

The read **must** occur inside the transaction, and ideally the `UPDATE` should use an optimistic or pessimistic row-level lock (e.g., `SELECT ... FOR UPDATE` in SQL) with a conditional predicate:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id AND stock >= :quantity
RETURNING stock;
```

This makes the check-and-decrement atomic at the database layer, which is the correct invariant. The Redis lock can remain as a first-pass contention reducer, but correctness must not depend on it exclusively.

### 2. Heartbeat Thread Can Silently Outlive the Lock Release Window

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

The `t.join(timeout=...)` call can return `False` — meaning the thread is still alive — and the code proceeds to release the lock anyway with no indication that the join timed out. A still-running heartbeat thread will then attempt to extend a lock that has already been released, and if a new holder has acquired the same key with a new token, the `EXTEND_LOCK_SCRIPT` will correctly return `0` (token mismatch), but this is a silent near-miss that should be logged explicitly. More importantly, the join timeout of `HEARTBEAT_INTERVAL + 1` (9s) is far too long for a `finally` block in a web request context. The heartbeat interval should be shortened or the join timeout should be a small fixed bound (e.g., 1–2s), relying on the daemon flag and the `heartbeat_running.clear()` signal for cooperative shutdown.

```python
finally:
    heartbeat_running.clear()
    joined = t.join(timeout=2)
    if t.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
    self._release_script(keys=[lock_key], args=[token])
```

### 3. `threading.Event.wait()` Semantics Invert the Intended Heartbeat Logic

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`Event.wait(timeout=N)` returns `True` if the event is **set** (i.e., signaled), and returns `False` only on timeout. Since `heartbeat_running` is set at construction and cleared on shutdown, `wait()` will return `True` *immediately* on every iteration until the event is cleared — it does not sleep for `HEARTBEAT_INTERVAL` seconds between beats. The thread will spin in a tight loop extending the lock continuously until `heartbeat_running.clear()` is called.

The correct pattern for a periodic heartbeat with a clean shutdown signal is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False = timeout elapsed, event still not set = keep running
        result = self._extend_script(...)
        ...
    # wait() returned True = event was set = shutdown requested
```

Or use a separate stop event with inverted semantics:

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...

# In finally:
stop_event.set()
```

This is a correctness bug. Under the current implementation the heartbeat fires in a tight loop, generating excessive Redis traffic and log noise.

## Moderate Issues

### 4. No Exception Handling Around the Database Transaction

If `self.db.update_stock` or `self.db.create_order` raises an exception, it will propagate out of `reserve_item` as an unhandled exception. The lock will still be released (the `finally` block in `_acquire_lock` handles that), but the caller receives a raw exception rather than a structured `ReservationResult`. For a public-facing service this is a reliability gap:

```python
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error("DB transaction failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed due to internal error.")
```

---

### 5. Pydantic Validation Exceptions Are Not Caught

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If `item_id` or `quantity` fails Pydantic validation, a `ValidationError` is raised and propagates to the caller unhandled. The test `test_input_validation_rejects_invalid_item_id` passes only because the test framework catches it — it does not verify that the service returns a graceful `ReservationResult`. Wrap this in a try/except:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Invalid request parameters.")
```

---

### 6. Lock Key Namespace Is Too Broad for Multi-Tenant or Multi-Environment Deployments

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared across environments (staging, production) or tenants, lock keys will collide. The key scheme should include an environment or tenant prefix, injected at construction time:

```python
lock_key = f"{self.key_prefix}:inventory:lock:{req.item_id}"
```

## Test Coverage Gaps

### 7. The Heartbeat Bug Is Not Caught by Any Test

No test exercises the heartbeat thread's behavior over time. A test that mocks `time` or uses a short `HEARTBEAT_INTERVAL` and asserts that `_extend_script` is called *at most once per interval* (not in a tight loop) would have caught issue #3.

### 8. `test_input_validation_rejects_invalid_item_id` Does Not Assert the Correct Contract

```python
def test_input_validation_rejects_invalid_item_id(service):
    result = service.reserve_item("../../admin", quantity=1)
    assert result.success is False
```

This test will currently **fail or error** because the `ValidationError` is unhandled and propagates as an exception rather than returning a `ReservationResult`. The test asserts `result.success is False`, which implies a graceful return — but the code does not do that. This test is asserting a contract the implementation does not fulfill.

### 9. No Test for Database Transaction Failure

There is no test covering the scenario where `db.update_stock` or `db.create_order` raises an exception mid-transaction. This is a critical path for data integrity.

### 10. `test_lock_release_is_atomic` Is Structurally Weak

```python
service._release_script.assert_called_once()
redis_client.delete.assert_not_called()
```

`_release_script` is assigned during `__init__` via `redis_client.register_script(...)`. Since `redis_client` is a `MagicMock`, `register_script` returns another `MagicMock`, and `_release_script` is that mock. Asserting it was called once is valid, but the test does not verify the correct `keys` and `args` were passed. A stronger assertion:

```python
service._release_script.assert_called_once_with(keys=[expected_lock_key], args=[expected_token])
```

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 8` is a reasonable ratio, but these should be configurable at construction time rather than class-level constants, particularly for testing.
- The `retry_attempts=3` default with exponential backoff tops out at 400ms total wait. For a flash sale this may be too aggressive — consider making this configurable and documenting the expected SLA.
- Consider registering the Lua scripts once at module level or caching the SHA via `EVALSHA` to avoid re-sending script bodies on every invocation under high load.
