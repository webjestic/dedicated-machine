Decision: Request Changes

Summary: The core distributed locking architecture is well-structured — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and exponential backoff are all solid choices. However, there are several correctness flaws, a meaningful race condition, and test gaps that must be addressed before this merges into a government-facing production system. The issues range from a stock-read/transaction split that undermines the lock's guarantee, to a heartbeat logic bug that can silently drop lock ownership without surfacing the failure to the caller.

---

## Findings

### 🔴 Critical

**1. Stock read is outside the database transaction (TOCTOU within the lock)**

```python
# services/inventory.py — reserve_item()
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
        self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` runs outside the database transaction. If the DB has any read-committed or weaker isolation, a concurrent writer (a refund, an admin correction, a different service node that doesn't participate in this lock) can mutate stock between the read and the `update_stock` write. The read must be inside the transaction, and `update_stock` should use a conditional decrement (e.g., `UPDATE ... WHERE stock >= quantity`) so the DB itself enforces the invariant:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None: ...
    if current_stock < req.quantity: ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

**2. Heartbeat loss is silent — caller never knows the lock was stolen**

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            heartbeat_running.clear()
            return   # ← thread exits, but the main flow continues unaware
```

When `result == 0`, the lock has expired or been taken by another process. The heartbeat thread logs a warning and exits, but `yield token` in the context manager has already returned control to `reserve_item`, which continues executing the DB transaction under the false belief it still holds the lock. This is a silent correctness failure — two processes can now both believe they own the reservation window.

The heartbeat needs a shared flag or exception channel back to the context manager. One clean pattern:

```python
lock_stolen = threading.Event()

def _beat():
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        if self._extend_script(...) == 0:
            lock_stolen.set()
            heartbeat_running.clear()
            return

# Inside the context manager, after yield:
# Check lock_stolen before committing, or poll it during long operations.
# At minimum, check before the DB transaction begins:
if lock_stolen.is_set():
    raise LockAcquisitionError("Lock lost during operation")
```

### 🟡 Moderate

**3. `ValidationError` from Pydantic is unhandled — will propagate as a 500**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
```

`test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but `ReservationRequest(...)` raises `pydantic.ValidationError`, it does not return a result. Those tests are currently passing only because the exception propagates and pytest catches it — they are not testing what the checklist claims. Add explicit handling:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

---

**4. Lock TTL / heartbeat interval relationship is fragile and undocumented**

`LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 8` are module-level constants with a comment saying "fires at TTL/3," but there is no enforcement of that relationship. If someone changes `LOCK_TTL` to 10 without updating `HEARTBEAT_INTERVAL`, the heartbeat fires after the lock has already expired. Derive one from the other or add an `__init__` assertion:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
    "HEARTBEAT_INTERVAL must be less than half LOCK_TTL"
)
```

---

**5. `t.join(timeout=...)` result is not checked — thread may still be running**

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`Thread.join()` with a timeout returns `None` whether or not the thread actually stopped. If the thread is still alive (e.g., blocked on a slow Redis call), the lock release races with the heartbeat extension. Check `t.is_alive()` after the join and log a warning if the thread did not terminate cleanly.

### 🟠 Test Coverage Gaps

**6. Validation tests assert wrong behavior and will give false confidence**

As noted in finding #3, `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will raise `ValidationError`, not return `success=False`. These tests need to be rewritten after the `ValidationError` catch is added to confirm the graceful return path.

**7. No test for heartbeat loss / lock-stolen scenario**

The most dangerous failure mode (finding #2) has zero test coverage. A test should mock `_extend_script` to return `0` mid-operation and assert the reservation fails rather than silently succeeding.

**8. `test_lock_release_is_atomic` asserts on a mock that was never wired correctly**

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
```

`service._release_script` is assigned in `__init__` via `redis_client.register_script(...)`. In the fixture, `register_script` returns a `MagicMock`, so `_release_script` is already a mock — but the assertion only checks it was called once, not that it was called with the correct `keys` and `args`. Tighten this:

```python
service._release_script.assert_called_once_with(
    keys=[f"inventory:lock:item_001"],
    args=[pytest.any_string]  # or capture the token
)
redis_client.delete.assert_not_called()
```

## What Is Done Well

- Lua scripts for atomic acquire, extend, and release are correct and appropriate.
- Scoping heartbeat state to the context manager (not instance state) correctly handles concurrent calls on the same service instance.
- `SET NX EX` is the right primitive — no separate `SET` + `EXPIRE` race.
- Pydantic model with regex pattern on `item_id` is the right place to enforce input shape.
- Structured logging with key/value pairs is consistent and grep-friendly.
- `LockAcquisitionError` as a typed exception is cleaner than `RuntimeError`.
- Exponential backoff is correctly scoped to the retry loop.
