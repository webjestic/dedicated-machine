Decision: Request Changes

Summary: The core locking and transaction logic is well-structured — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and exponential backoff are all solid choices. However, there are several correctness, reliability, and security issues that must be addressed before this merges, ranging from a TOCTOU race between the stock read and the DB transaction, to unhandled exceptions leaking internal state, to test fixtures that don't actually exercise the code paths they claim to cover.

---

## Findings

### 🔴 Critical

**1. TOCTOU Race: Stock Read Is Outside the Database Transaction**

```python
# services/inventory.py

# Stock is read BEFORE the transaction opens
current_stock = self.db.get_stock(req.item_id)   # ← outside transaction

# ...

with self.db.transaction():                        # ← transaction opens here
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes callers *in this service*, but it provides no protection against:
- Direct DB writes from other services or admin tooling
- A DB replica lag scenario if `get_stock` hits a read replica
- Any future code path that bypasses this lock

The stock read must be inside the transaction boundary, and ideally the `UPDATE` should use an optimistic or pessimistic row-level lock (`SELECT ... FOR UPDATE` / conditional `UPDATE ... WHERE stock >= quantity`). The current pattern is a classic check-then-act race.

```python
# Recommended pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        # handle and return — transaction rolls back cleanly
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

---

**2. Unhandled Exceptions Leak Internal Error Details**

Only `LockAcquisitionError` is caught. Any exception from `db.get_stock`, `db.update_stock`, `db.create_order`, or `db.transaction()` will propagate uncaught to the caller. In a web context this typically means a 500 with a stack trace. At minimum, a broad `except Exception` guard should log the error and return a safe `ReservationResult`, and the DB exception type should never surface raw to an API boundary.

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

---

**3. Pydantic Validation Errors Are Not Caught**

`ReservationRequest(item_id=item_id, quantity=quantity)` will raise `pydantic.ValidationError` for invalid inputs. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but the actual code will raise an unhandled exception, not return a `ReservationResult`. These tests will pass only if the test runner catches the exception at the assertion boundary — which it won't; they will error, not fail gracefully. Wrap the Pydantic construction:

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters.")
```

### 🟡 Significant

**4. Heartbeat Thread Leak on Lock Acquisition Failure Path**

If `self.redis.set` raises a `redis.RedisError` mid-retry loop, the exception propagates before the heartbeat thread is started — that's fine. But if the lock *is* acquired and then `yield token` raises an unexpected exception, the `finally` block does run. However, if `t.join(timeout=...)` times out silently (the heartbeat is stuck in a slow Redis call), the lock release via `_release_script` still fires — but the thread is left running as a daemon. Daemon threads are acceptable here, but the silent join timeout should be logged:

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

**5. `retry_attempts=3` With Exponential Backoff Has No Jitter**

Under high concurrency (flash sale scenario — exactly the stated use case), all contending threads will back off on the same schedule: 0.1s, 0.2s, 0.4s. This creates a thundering herd on retry. Add jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

**6. `LOCK_TTL = 30` Is Dangerously Long for a Flash Sale**

A 30-second lock on a hot inventory item during a flash sale will cause severe head-of-line blocking. If a DB operation hangs (deadlock, slow query), every other request for that item queues behind it for up to 30 seconds. Consider a much shorter TTL (3–5s) with the heartbeat extending it, which is the correct pattern — the heartbeat exists precisely so the TTL can be short. The current values invert this: long TTL, heartbeat at TTL/3 = 10s.

### 🟠 Test Quality Issues

**7. Test Fixtures Don't Wire the Lua Scripts Correctly**

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1   # ← this attribute is never called
    return mock
```

`register_script` returns a callable script object. `MagicMock()` will auto-create `register_script` as a mock returning another `MagicMock`, so `_release_script` and `_extend_script` are callable mocks — that part works by accident. But `execute_script` is never invoked; the fixture comment is misleading and suggests the author doesn't fully understand the mock wiring. The heartbeat's `_extend_script` call returning a truthy mock means the heartbeat will *never* detect a stolen lock in tests. Add explicit script mock setup:

```python
script_mock = MagicMock(return_value=1)
mock.register_script.return_value = script_mock
```

**8. `test_lock_release_is_atomic` Doesn't Assert What It Claims**

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is the result of `redis_client.register_script(...)` at construction time — it's a mock child of the mock `redis_client`. Asserting it was called once doesn't verify atomicity; it only verifies the mock was invoked. This test would pass even if the implementation called `GET` + `DELETE` separately, as long as it also called the script. The test needs to verify the *script content* or at minimum that no separate `GET`/`DELETE` calls were made.

**9. No Test for Heartbeat Behavior or Lock Extension**

Given that the heartbeat is a meaningful piece of the design, there is no test that:
- Verifies the heartbeat calls `_extend_script` at the expected interval
- Verifies the heartbeat stops when the lock is released
- Verifies behavior when `_extend_script` returns 0 (lock stolen)

### 🔵 Minor / Hygiene

**10. `remaining_stock` Exposed on Failure Responses**

```python
return ReservationResult(
    success=False,
    remaining_stock=current_stock,   # ← exact stock count returned to caller
    message="Insufficient stock"
)
```

Returning exact inventory counts to clients on failure leaks business-sensitive data and could be used to time attacks. Return a boolean or a bucketed value (`"low"`, `"out_of_stock"`) depending on the threat model for this deployment.

**11. `order_id` Generated Before Transaction Commits**

`order_id = str(uuid.uuid4())` is generated before the `with self.db.transaction()` block. If the transaction rolls back, the `order_id` is discarded — that's fine. But if the caller receives a success response and the transaction silently fails (swallowed exception), there's an inconsistency. Move `order_id` generation inside the transaction, or ensure transaction exceptions always propagate.

---

## Required Before Approval

1. Move `get_stock` inside the transaction; use row-level locking (`SELECT FOR UPDATE` or equivalent)
2. Catch `ValidationError` from Pydantic and return a safe `ReservationResult`
3. Add a broad exception handler for DB errors
4. Fix test fixtures to correctly mock `register_script` return values
5. Add tests for heartbeat extension and stolen-lock detection
6. Log (don't silently ignore) a heartbeat thread that fails to join
