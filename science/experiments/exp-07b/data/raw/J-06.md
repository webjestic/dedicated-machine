Decision: Request Changes

Summary: The core locking architecture is well-structured — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and exponential backoff are all solid choices. However, there are several correctness and reliability issues that must be resolved before this merges: a race condition between the DB read and the transaction, a heartbeat threading bug that can cause premature termination, and test fixtures that don't accurately reflect the real code paths. These are not cosmetic — they represent real failure modes under the concurrent load this service is explicitly designed to handle.

---

## Detailed Findings

### 🔴 Critical

**1. TOCTOU race between `get_stock` and `update_stock`**

```python
# Inside the lock — looks safe, but isn't
current_stock = self.db.get_stock(req.item_id)   # READ outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # WRITE inside transaction
    self.db.create_order(...)
```

The stock read happens *outside* the database transaction. If the DB read and the transaction are not on the same connection/session, another process (or a direct DB write bypassing Redis) can modify stock between the read and the write. The distributed lock protects against concurrent callers *through this service*, but it does not protect against direct DB writes, admin tooling, other services, or connection pool routing differences.

The fix is to move `get_stock` inside the transaction boundary, and ideally use a `SELECT ... FOR UPDATE` or equivalent pessimistic lock at the DB layer as a defense-in-depth measure:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        ...
    if current_stock < req.quantity:
        ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

**2. Heartbeat thread terminates immediately on first iteration**

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`threading.Event.wait(timeout=N)` returns `True` if the event is *set* before the timeout, and `False` if it times out. Since `heartbeat_running` is `.set()` at creation, `wait()` returns `True` immediately on every call — the heartbeat fires continuously in a tight loop rather than sleeping for `HEARTBEAT_INTERVAL` seconds between extensions. Under load, this will hammer Redis with extend calls and burn CPU.

The correct pattern is to use a separate stop event and `time.sleep`, or invert the event semantics:

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)

# In finally:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

`stop_event` starts unset; `wait(timeout=N)` blocks for N seconds, returns `False` on timeout (loop continues), returns `True` when set (loop exits). This is the correct idiom.

### 🟡 Moderate

**3. Unhandled exceptions inside the lock context are silently swallowed**

If `self.db.get_stock()` or the transaction raises an unexpected exception, it propagates through the `with self._acquire_lock(...)` block, which is correct — the `finally` clause will release the lock. However, `reserve_item` has no outer `except` for general DB errors, meaning an unhandled exception will bubble up to the caller as a 500 with no `ReservationResult`. Given this is a service method, callers likely expect a typed return. Add a catch for general exceptions:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error during reservation")
```

---

**4. Pydantic `ValidationError` is not caught in `reserve_item`**

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails (e.g., `item_id="../../admin"`), Pydantic raises `ValidationError`, not a handled exception. The test `test_input_validation_rejects_invalid_item_id` asserts `result.success is False`, but the actual behavior is an unhandled exception. This test is currently passing for the wrong reason or would fail in practice.

Wrap construction:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation input: %s", e)
    return ReservationResult(success=False, message="Invalid input")
```

---

**5. `retry_attempts` is not configurable at the service level**

The default of 3 retries with exponential backoff is hardcoded in `_acquire_lock`. Under a flash sale, contention will be high and 3 attempts may be insufficient or too aggressive depending on deployment. This should be a constructor parameter or at minimum a class-level constant alongside `LOCK_TTL`.

---

### 🟠 Test Quality Issues

**6. `redis_client` fixture does not reflect actual script registration**

```python
mock.execute_script.return_value = 1  # This method is never called
```

`register_script` returns a callable script object. The mock's `register_script` returns a `MagicMock`, which when called returns another `MagicMock` (truthy). This means `_extend_script` and `_release_script` are never actually validated — the tests pass vacuously. The fixture should be structured to return a mock script callable that returns `1`:

```python
mock_script = MagicMock(return_value=1)
mock.register_script.return_value = mock_script
```

Then assertions like `service._release_script.assert_called_once()` in `test_lock_release_is_atomic` will be meaningful.

---

**7. Heartbeat thread is never waited on in tests**

Because tests run fast and the heartbeat fires immediately (per bug #2), there is a real risk of the heartbeat thread outliving the test and interfering with subsequent tests or producing spurious log output. With the corrected stop-event pattern, this resolves naturally, but it should be explicitly validated.

---

### 🔵 Minor / Suggestions

- **`lock_key` format**: `inventory:lock:{item_id}` — since `item_id` is validated to `[a-zA-Z0-9_\-]+`, this is safe. Good.
- **`order_id` generated before transaction**: If the transaction rolls back, the `order_id` is orphaned in memory but never persisted — this is fine, but worth a comment.
- **`LOCK_TTL = 30` seconds**: For a flash sale service, 30 seconds is long. If the DB is slow, the heartbeat compensates, but a 30-second lock on a hot item under contention will queue requests for a long time. Consider documenting the rationale or making it configurable.
- **No Redis connection error handling**: A `redis.exceptions.ConnectionError` during lock acquisition or heartbeat will propagate as an unhandled exception. Should be caught and surfaced as `LockAcquisitionError` or `ReservationResult(success=False, ...)`.
