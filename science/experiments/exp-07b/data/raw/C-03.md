Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped lock state are all reasonable choices. However, there are several correctness defects, a critical race condition in the heartbeat/release sequencing, architectural concerns around the Redis-DB consistency boundary, and test suite gaps that collectively make this unsafe to merge for a high-stakes flash sale workload. The issues range from a subtle but exploitable TOCTOU window to daemon thread behavior that can silently corrupt state under process shutdown.

## Review

### 🔴 Critical — Must Fix

---

**1. TOCTOU gap between stock read and database transaction**

```python
# Lock is held here — good
current_stock = self.db.get_stock(req.item_id)   # read outside transaction
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # write inside transaction
    self.db.create_order(...)
```

`get_stock` is called **outside** the database transaction. If the database uses `READ COMMITTED` isolation (the default in PostgreSQL and MySQL), another committed write can land between the read and the `update_stock` call — even while the Redis lock is held — if any code path ever bypasses the lock (admin tooling, migrations, a bug in another service). The stock read and the conditional update must be atomic at the database level, not just at the Redis level. The correct pattern is a conditional `UPDATE ... WHERE stock >= quantity RETURNING stock`, or at minimum, moving `get_stock` inside the transaction and using `SELECT ... FOR UPDATE`.

```python
# Safer pattern
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ...  # rollback implicit
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

**2. Heartbeat thread race condition on lock release**

```python
finally:
    heartbeat_running.clear()          # (1) signal stop
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)   # (2) wait up to 9s
    self._release_script(keys=[lock_key], args=[token])  # (3) release
```

There is a window between steps (1) and (3) where the heartbeat thread may be mid-execution of `_extend_script` — it read `heartbeat_running` as set, entered the Redis call, and has not yet returned. The `clear()` does not interrupt an in-flight network call. This means:

- The heartbeat extends the TTL **after** `clear()` is called.
- `join()` completes.
- `_release_script` runs and successfully deletes the key.
- ✅ This specific sequence is actually fine.

The more dangerous inverse race: the heartbeat fires, the lock has already expired (e.g., Redis was slow), the heartbeat gets `result == 0`, clears itself, and **returns** — but the main thread has not yet noticed. The main thread then calls `_release_script` on a key that may now be owned by a **different** process. The Lua script guards against deletion of a foreign token, so data corruption is prevented — but the main thread has no signal that it lost the lock mid-operation. The database transaction may have already committed. **This is a silent partial failure.** The heartbeat's `clear()` on loss of ownership must propagate back to the main thread so the transaction can be aborted.

```python
# Suggested: use an Event to signal lock loss to the caller
lock_lost = threading.Event()

def _beat():
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            lock_lost.set()       # signal the main thread
            heartbeat_running.clear()
            return

# In the main thread, check after DB operations:
if lock_lost.is_set():
    raise LockLostError("Lock expired during transaction — result unreliable")
```

**3. `join(timeout=...)` does not guarantee thread termination**

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=N)` returns after N seconds **regardless of whether the thread has stopped**. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis overload), `join` returns, and `_release_script` fires while `_extend_script` is still in-flight. This is a benign race due to the Lua guard, but the thread is now orphaned — it will eventually attempt to extend a key that no longer exists, log a spurious warning, and exit. Under high concurrency this creates thread accumulation. You must check `t.is_alive()` after join and handle accordingly, or enforce a hard deadline on the Redis calls themselves via socket timeout.

**4. Pydantic validation exceptions are not caught in `reserve_item`**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
```

`ReservationRequest(...)` raises `pydantic.ValidationError` on bad input. The test `test_input_validation_rejects_invalid_item_id` passes only because the test framework catches it — but the actual return type contract (`ReservationResult`) is violated. The caller gets an unhandled exception, not a `ReservationResult(success=False, ...)`. Wrap the construction in a try/except:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    return ReservationResult(success=False, message=f"Invalid input: {e}")
```

The tests for input validation are therefore **false positives** — they assert `result.success is False` but the function never returns a result at all. This is a test suite integrity failure.

---

**5. No Redis connection error handling**

Every Redis call — `set`, `_extend_script`, `_release_script` — can raise `redis.RedisError` (connection refused, timeout, READONLY replica, etc.). None of these are caught. A Redis failure during lock acquisition will propagate as an unhandled exception through `reserve_item`, bypassing the `LockAcquisitionError` handler and exposing raw infrastructure errors to callers. Wrap Redis operations:

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
except redis.RedisError as e:
    logger.error("Redis error during lock acquisition: %s", e)
    raise LockAcquisitionError(f"Redis unavailable: {e}") from e
```

**6. Daemon thread behavior under process shutdown**

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed abruptly when the main thread exits. Under a graceful shutdown (SIGTERM → gunicorn worker exit), a heartbeat thread may be killed mid-`_extend_script`, leaving the lock to expire naturally — which is acceptable. However, if the main thread is inside `with self.db.transaction()` when shutdown begins, the transaction may be abandoned without rollback, depending on the DB driver. This is an edge case but worth documenting explicitly and potentially handling with a shutdown hook.

### 🟡 Medium — Should Address

**7. `HEARTBEAT_INTERVAL` is too close to `LOCK_TTL`**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8   # comment says TTL/3, which is correct
```

The comment says `TTL/3` which is 10, but the value is 8. Minor inconsistency, but more importantly: with a 30s TTL and 8s heartbeat, there are approximately 3 heartbeat opportunities before expiry. Under a Redis network hiccup of >8s, the heartbeat misses, and the next fires at 16s elapsed — still within the 30s window. This is acceptable, but the relationship between these constants should be enforced programmatically, not by convention:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

---

**8. `retry_attempts` backoff reaches only 400ms total**

```python
for attempt in range(retry_attempts):  # default 3
    backoff = (2 ** attempt) * 0.1     # 0.1, 0.2, 0.4 seconds
```

With 3 attempts, total wait is 0.7s. For a flash sale with a 30s lock TTL, a legitimate lock holder may hold the lock for several seconds. Callers will fail fast and return "unavailable" when they could succeed with slightly more patience. Consider making `retry_attempts` and the backoff base configurable, and consider jitter to prevent thundering herd:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

---

**9. No idempotency key / duplicate reservation guard**

A client that receives a network timeout may retry `reserve_item`. Without an idempotency key, the same customer can be charged twice and two orders created. For a flash sale service this is a business-critical gap. The `order_id` is generated server-side with `uuid4()` and is not tied to any client-provided request ID.
