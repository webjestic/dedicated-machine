Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all well-executed. However, there are several correctness, reliability, and test-fidelity issues that must be addressed before this merges into a government-grade production system. The most critical are a TOCTOU race between the stock read and the database transaction, a heartbeat logic bug that can cause premature lock abandonment, and test fixtures that don't actually exercise the code paths they claim to cover.

---

## Detailed Findings

### 1. TOCTOU Race — Stock Read Outside the Database Transaction (Critical)

**File:** `services/inventory.py`, `reserve_item`

```python
# PROBLEM: stock is read OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)   # ← read here
...
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # ← write here
    self.db.create_order(...)
```

The Redis lock serializes callers *at the application layer*, but it provides zero protection against:

- A direct database write from another service, a migration script, or an admin tool that bypasses the lock.
- A Redis failover (Sentinel promotion, cluster resharding) that silently drops the lock between the read and the write.

The stock read must occur **inside** the same database transaction as the write, and the `UPDATE` should use an optimistic or pessimistic row-level lock depending on your DB engine:

```python
with self.db.transaction():
    # Pessimistic: SELECT ... FOR UPDATE (Postgres/MySQL)
    current_stock = self.db.get_stock_for_update(req.item_id)
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock still has value for reducing DB contention under flash-sale load, but the database must be the authoritative guard.

### 2. Heartbeat Thread Logic Bug — Premature Lock Abandonment (High)

**File:** `services/inventory.py`, `_beat`

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            heartbeat_running.clear()
            return   # ← thread exits, but the caller still holds `yield token`
```

`threading.Event.wait(timeout=N)` returns `True` if the event is **set** and `False` on timeout. The loop condition `while heartbeat_running.wait(...)` therefore:

- Fires the extension immediately when the event is set (i.e., on every iteration without waiting).
- Exits the loop only when the event is cleared *or* the timeout elapses — which is the opposite of the intended behavior.

The correct pattern is to wait for the interval, then check whether the event is still set:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout elapsed, event still clear-ish
        # Re-check explicitly:
        if not heartbeat_running.is_set():
            return
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Or more idiomatically, use `time.sleep` with a stop-event check:

```python
def _beat() -> None:
    while True:
        # Sleep for the interval, but wake immediately if stop is signaled
        if heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL) is True:
            # Event was set (stop requested) before timeout
            return
        # Timeout elapsed — extend
        result = self._extend_script(keys=[lock_key], args=[token, str(self.LOCK_TTL)])
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Wait — `heartbeat_running` starts **set** and is cleared to stop. So `wait()` on a set event returns `True` immediately, meaning the current code hammers Redis in a tight loop from the moment the lock is acquired. This is a functional bug that will flood Redis under any real load.

The event semantics need to be inverted: use a `stop_event` that starts **clear** and is **set** to signal shutdown:

```python
stop_event = threading.Event()  # starts clear

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # timeout elapsed without stop being signaled — extend
        ...

# In finally:
stop_event.set()
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 3. Unhandled Exception Path Leaks the Lock (High)

**File:** `services/inventory.py`, `reserve_item`

```python
try:
    with self._acquire_lock(lock_key):
        ...
        with self.db.transaction():
            self.db.update_stock(...)
            self.db.create_order(...)
except LockAcquisitionError as e:
    ...
```

Only `LockAcquisitionError` is caught. Any other exception — `redis.RedisError`, a DB connection failure, an unexpected `ValueError` — propagates up to the caller **without returning a `ReservationResult`**. The `_acquire_lock` context manager's `finally` block will still release the lock (good), but the caller receives an unhandled exception rather than a graceful `ReservationResult(success=False, ...)`.

Add a broad catch for operational errors:

```python
except (redis.RedisError, DatabaseError, Exception) as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="An internal error occurred.")
```

Scope this carefully — you don't want to swallow `SystemExit` or `KeyboardInterrupt`.

### 4. Pydantic Validation Errors Not Caught (Medium)

**File:** `services/inventory.py`, `reserve_item`

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails, Pydantic raises `ValidationError`. This is not caught anywhere in `reserve_item`, so the caller receives a raw exception instead of a `ReservationResult`. The test `test_input_validation_rejects_invalid_item_id` passes only because the test framework catches the exception and the assertion `result.success is False` is never actually reached — it would raise before the return.

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation input: %s", e)
    return ReservationResult(success=False, message="Invalid input.")
```

### 5. Test Fixture Does Not Wire the Lua Scripts Correctly (Medium)

**File:** `tests/test_inventory.py`

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1  # ← this attribute is never called
    return mock
```

`InventoryService.__init__` calls `redis_client.register_script(...)` twice and stores the results as `self._release_script` and `self._extend_script`. The mock's `register_script` returns a new `MagicMock` by default, which is callable and returns another `MagicMock` (truthy, not `1`). The heartbeat's `if result == 0` check therefore never triggers correctly in tests.

`test_lock_release_is_atomic` asserts `service._release_script.assert_called_once()`, but `_release_script` is the return value of `mock.register_script(...)`, not `redis_client` itself — this assertion is on the right object only by coincidence of how `MagicMock` chains work. Make this explicit:

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    release_mock = MagicMock(return_value=1)
    extend_mock = MagicMock(return_value=1)
    mock.register_script.side_effect = [release_mock, extend_mock]
    return mock
```

### 6. Lock Key Namespace Collision Risk (Low)

**File:** `services/inventory.py`

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared across environments (staging keys bleeding into production is a known ops failure mode in shared-cluster setups), there is no environment or tenant prefix. Recommend:

```python
lock_key = f"{self.key_prefix}:inventory:lock:{req.item_id}"
```

where `key_prefix` is injected at construction time (e.g., `"prod"`, `"staging"`).
