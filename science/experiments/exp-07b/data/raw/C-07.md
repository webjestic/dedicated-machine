Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical race condition in the heartbeat/release sequencing, missing exception handling around the database layer, and test suite deficiencies that together make this unsafe to merge into a production inventory system, particularly under the high-concurrency flash sale workload this code is explicitly designed to serve.

---

## Critical Issues

### 1. Race Condition: Heartbeat Can Extend a Lock After Release Begins

This is the most serious flaw. The `finally` block sequences as:

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`heartbeat_running.clear()` signals the thread to stop, but `threading.Event.wait(timeout=N)` will not unblock immediately — it will sleep until the current interval expires. The `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` call sets a ceiling of 9 seconds, but during that window the heartbeat thread may fire one final extension *after* the business logic has completed and *before* `_release_script` runs. This is not catastrophic on its own, but under a GC pause or network hiccup the thread could extend a lock that the owning process has already logically relinquished, delaying another waiter by up to 30 seconds. More importantly, if `t.join()` times out (which it can — it's not guaranteed to complete), `_release_script` still runs, but you've lost the guarantee that the heartbeat is dead. Use an interruptible sleep pattern instead:

```python
# In _beat: replace heartbeat_running.wait(timeout=...) with a
# short-polling loop or use threading.Event with a stop event
# that is set *after* the lock is released, not before.
```

The correct sequencing is: release the lock first (atomically), *then* stop the heartbeat. Stopping the heartbeat first creates the window described above. Alternatively, use a dedicated `stop_event` separate from the "I lost the lock" signal so the two conditions don't share state.

### 2. `get_stock` Read Is Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read happens outside the database transaction boundary. The Redis lock serializes callers *through this service*, but it provides no protection against:

- Direct database writes from other services or admin tooling
- Database replicas with replication lag if `get_stock` reads from a replica
- Any future code path that bypasses this service

The read must be inside the transaction, and ideally use a `SELECT ... FOR UPDATE` (or equivalent pessimistic lock at the DB layer) so the stock value is consistent with the write. The distributed lock and the DB transaction should be treated as complementary, not redundant — the DB transaction is the true atomicity boundary.

### 3. No Exception Handling Around Database Operations

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises, the exception propagates out of `reserve_item` entirely uncaught (past the `LockAcquisitionError` handler which only catches that specific type). The caller receives an unhandled exception rather than a structured `ReservationResult`. For a flash sale endpoint this likely means a 500 to the end user and a lock that may or may not have been released depending on where in the `finally` chain the failure occurred. Add explicit handling:

```python
except (DatabaseError, redis.RedisError) as e:
    logger.error("Reservation failed item_id=%s: %s", req.item_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

## Significant Issues

### 4. `t.join()` Timeout Is Not Verified

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=...)` returns `None` whether the thread stopped or timed out. There is no check of `t.is_alive()` after the join. If the heartbeat thread is stuck (e.g., blocked on a Redis network call), the code silently proceeds to release the lock while the thread is still running. The thread then wakes up, calls `_extend_script`, gets result `0` (lock is gone), and logs a spurious "Heartbeat lost lock ownership" warning — a misleading signal. At minimum, log a warning if `t.is_alive()` after join.

### 5. Pydantic Validation Errors Are Not Caught

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
```

`ReservationRequest(...)` raises `pydantic.ValidationError` on invalid input. The test `test_input_validation_rejects_invalid_item_id` asserts `result.success is False`, but the actual behavior is an unhandled `ValidationError` exception — the test is wrong and would fail against the real code unless the test framework is catching it silently. This needs a `try/except ValidationError` block that returns a structured failure result.

### 6. Heartbeat Interval Arithmetic Is Fragile

`HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` means the heartbeat fires at roughly TTL/3.75. The comment says "fires at TTL/3" which would be 10 seconds — this is inconsistent. More importantly, these two constants are defined independently with no enforced relationship. If someone changes `LOCK_TTL` to 15 seconds without updating `HEARTBEAT_INTERVAL`, the heartbeat fires at TTL/1.875 — still safe — but if `LOCK_TTL` is reduced to 5 seconds, the heartbeat at 8 seconds fires *after* the lock has already expired. Derive one from the other:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

## Test Suite Issues

### 7. Tests Do Not Validate Actual Behavior of Pydantic Validation

```python
def test_input_validation_rejects_invalid_item_id(service):
    result = service.reserve_item("../../admin", quantity=1)
    assert result.success is False
```

As noted above, this test asserts a `ReservationResult` is returned, but the actual code raises `ValidationError`. The test passes only if the test runner catches the exception and the assertion is never evaluated — or it fails with an unexpected exception. Either way, it is not testing what it claims to test.

### 8. Heartbeat Thread Is Never Tested

There is no test that verifies the heartbeat actually extends the lock TTL, that it stops when the lock is released, or that it handles the "lock stolen" case (`result == 0`) correctly. For a component whose entire purpose is preventing lock expiry under load, this is a significant gap.

### 9. `test_lock_release_is_atomic` Tests the Wrong Thing

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is a registered script object on the mock — this assertion verifies that the mock's method was called, not that the Lua script was actually registered or executed correctly. This test provides false confidence. A meaningful test would verify the script content or use an integration test against a real Redis instance (e.g., via `fakeredis`).

### 10. No Test for Database Transaction Failure / Rollback

There is no test verifying that if `db.update_stock` or `db.create_order` raises, the transaction is rolled back and the lock is still released. This is a critical correctness property for an inventory system.

## Minor Issues

### 11. `HEARTBEAT_INTERVAL + 1` Magic Number

The join timeout of `self.HEARTBEAT_INTERVAL + 1` (9 seconds) is unexplained. Document why this specific value was chosen and what the failure mode is if it's exceeded.

### 12. Lock Key Namespace Collision

`f"inventory:lock:{req.item_id}"` — if `item_id` values are shared across environments (staging keys leaking into prod via a misconfigured Redis), this could cause spurious lock contention. Consider namespacing by environment: `f"{env}:inventory:lock:{req.item_id}"`.

### 13. No Observability on Lock Wait Time

For a flash sale system, lock acquisition latency is a key operational metric. The retry loop has no timing instrumentation. Add a metric or structured log entry recording total time spent acquiring the lock.
