Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all well-executed. However, there are several correctness and reliability issues that must be addressed before this merges, ranging from a race condition between stock read and the DB transaction, to a heartbeat logic bug that can cause premature lock abandonment, to test fixtures that don't accurately reflect the real code paths being exercised.

---

## Detailed Findings

### 🔴 Critical

**1. Stock read occurs outside the database transaction (TOCTOU within the lock)**

```python
# Current — read is outside the transaction boundary
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside tx
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside tx
        self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but `get_stock` runs outside the DB transaction. If the DB supports any form of concurrent access (read replicas, async writes, or if the lock is ever lost mid-flight), the stock value read is not guaranteed to be the same value the `UPDATE` operates against. The read and write must be atomic at the DB layer too.

```python
# Recommended — read inside the transaction, use a conditional update
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None or current_stock < req.quantity:
        ...  # early return / rollback
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

A `SELECT FOR UPDATE` (or equivalent pessimistic lock) inside the transaction makes the DB itself the final arbiter of consistency, and the Redis lock becomes a performance optimization (reducing DB lock contention) rather than a correctness dependency.

**2. Heartbeat `threading.Event.wait()` returns `False` on timeout, causing immediate self-termination**

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...
```

`threading.Event.wait(timeout=N)` returns `True` if the event is set, and `False` if the timeout expires without the event being set. The event starts set (`heartbeat_running.set()`), so on the first call `wait()` returns `True` immediately — the heartbeat fires at time 0, not at `HEARTBEAT_INTERVAL`. On subsequent iterations, `wait()` again returns `True` immediately because the event is still set, so the loop spins continuously, hammering Redis with extend calls in a tight loop.

The intent is to use the event as a cancellation signal, not a timer. The correct pattern is:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False → timeout elapsed, event not set → extend
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
    # wait() returned True → stop_event was set → exit cleanly

stop_event = threading.Event()  # starts cleared
```

This is a significant correctness bug. Under the current code the heartbeat either spins at 100% or never fires, depending on subtle timing.

### 🟡 Major

**3. Unhandled exceptions inside the DB transaction leak the lock and leave state undefined**

```python
with self._acquire_lock(lock_key):
    ...
    with self.db.transaction():
        self.db.update_stock(...)   # what if this raises?
        self.db.create_order(...)
```

If `update_stock` or `create_order` raises an unexpected exception (network timeout, constraint violation, etc.), the `transaction()` context manager will roll back the DB work, but the exception will propagate up through `_acquire_lock`'s `try/finally`, which will correctly release the Redis lock. That part is fine. What's missing is explicit handling and logging of this failure path — the caller receives an unhandled exception rather than a graceful `ReservationResult`. Add a broad `except Exception` around the transaction block with appropriate logging and a safe return value.

---

**4. Pydantic `ValidationError` is not caught — invalid input raises, not returns**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
```

The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, but `ReservationRequest(...)` will raise `pydantic.ValidationError`, not return a result. These tests will currently pass only if the test runner catches the exception and the assertion is never reached — or they will error. Catch `ValidationError` explicitly:

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

### 🟠 Moderate

**5. Test fixtures don't wire up the Lua script mocks correctly**

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1  # ← this attribute is never called
    return mock
```

`register_script` returns a callable script object. `MagicMock()` will auto-create `register_script` as a mock that returns another `MagicMock`, and calling that mock returns yet another `MagicMock` (truthy). This means `_release_script` and `_extend_script` are called against unconfigured mocks. The heartbeat's `result == 0` check will never be `True` (a `MagicMock` is not `0`), which incidentally masks bug #2 in tests. The fixtures need to explicitly configure script return values:

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    script_mock = MagicMock(return_value=1)
    mock.register_script.return_value = script_mock
    return mock
```

**6. `test_lock_release_is_atomic` asserts on `service._release_script` but that's the registered script object, not a method on the service**

The test checks `service._release_script.assert_called_once()`, which will pass trivially because `_release_script` is a `MagicMock` and `assert_called_once()` on a mock that was called (even with wrong args) passes. The test should verify the script was called with the correct `keys` and `args` to be meaningful.

### 🔵 Minor / Informational

**7. `HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` leaves a narrow safety margin**

At TTL/3.75, a single missed heartbeat (Redis hiccup, GIL contention) leaves ~22 seconds before expiry. Consider TTL/4 or TTL/5 (6–7s interval) or making both values configurable via constructor parameters so callers can tune for their environment.

**8. Lock key namespace could collide across environments**

`inventory:lock:{item_id}` has no tenant, environment, or service prefix. If this Redis instance is shared (staging/prod on the same cluster, or multiple services), keys will collide. Consider `{env}:inventory:lock:{item_id}` or inject a namespace prefix.

**9. Daemon thread suppresses clean shutdown signals**

`daemon=True` means the heartbeat thread is killed abruptly on interpreter exit. This is acceptable for a heartbeat, but worth a comment explaining the intentional choice so future maintainers don't change it to a non-daemon thread (which could block shutdown).
