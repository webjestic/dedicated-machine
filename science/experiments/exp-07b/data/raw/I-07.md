Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices for this problem domain. However, there are several correctness flaws, a meaningful architectural gap, and test coverage holes that must be addressed before this is safe to merge into a high-concurrency flash sale path.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and Transaction

This is the most dangerous flaw in the implementation. The stock read happens **outside** the database transaction:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # <-- outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # <-- inside transaction
        self.db.create_order(...)
```

The Redis lock serializes callers at the application layer, but it provides **zero protection** against:

- A direct database write bypassing this service entirely (a DBA hotfix, a migration script, another microservice, a legacy code path)
- A Redis failover or network partition that causes the lock to be silently lost between the read and the write — the heartbeat detects this but does not propagate the failure back to the reservation path

The stock read must occur **inside** the same database transaction as the write, and the `UPDATE` should use an optimistic or pessimistic row-level lock depending on your DB engine. The canonical safe pattern is:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT ... FOR UPDATE
    # validate, compute new_stock
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock can still serve a useful purpose here — it reduces DB-level lock contention and provides a fast rejection path — but it cannot be the **sole** correctness guarantee.

### 2. Heartbeat Thread Failure Is Silently Swallowed

When the heartbeat detects lock loss, it logs a warning and clears the event, but the critical section continues executing without interruption:

```python
if result == 0:
    logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
    heartbeat_running.clear()
    return  # <-- heartbeat exits, but reserve_item keeps running
```

This means a process that has lost its lock will continue to read stock, decrement it, and commit an order — potentially racing against another process that has legitimately acquired the same lock. The heartbeat thread needs a signaling mechanism back to the main thread. A `threading.Event` for abort, checked at yield boundaries, or a flag inspected before the DB transaction is committed, would both be acceptable approaches.

---

### 3. Validation Exceptions Are Not Caught

Pydantic `ValidationError` is raised inside `reserve_item` but there is no handler for it:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
```

The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` both assert `result.success is False`, but the actual behavior is an **unhandled exception propagating to the caller**. These tests are passing against incorrect behavior — or they are not actually running the real validation path, which means the mock setup is masking the gap. Either way this needs a `except ValidationError` block that returns a well-formed `ReservationResult`.

## Moderate Issues

### 4. Heartbeat `wait()` Semantics Are Inverted for Shutdown

```python
while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
```

`threading.Event.wait()` returns `True` if the event is **set** and `False` if it times out. The intent here is to sleep for `HEARTBEAT_INTERVAL` seconds between renewals, but the loop condition means the heartbeat will **stop immediately** when `heartbeat_running.clear()` is called from the finally block — which is the desired shutdown behavior. However, the loop will also stop if the event is cleared due to lock loss, which is correct. The logic is technically sound but semantically inverted from what a reader expects. The conventional pattern for a cancellable sleep is `stop_event.wait(timeout=interval)` where the event starts **cleared** and is **set** to signal shutdown. The current approach works but will confuse maintainers and is one refactor away from a subtle bug. Recommend inverting to a `stop_event` pattern.

### 5. No Timeout on Lock Acquisition Backoff Under Load

With `retry_attempts=3` and exponential backoff of `0.1, 0.2, 0.4` seconds, the maximum wait before failure is 700ms. Under a genuine flash sale spike this is likely acceptable, but the retry count and backoff base are hardcoded magic numbers with no configuration surface. These should be constructor parameters or at minimum named constants, especially since `LOCK_TTL` and `HEARTBEAT_INTERVAL` are already promoted to class-level constants.

### 6. `t.join(timeout=...)` Result Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join()` with a timeout returns silently whether or not the thread actually terminated. If the heartbeat thread is blocked on a slow Redis call, the lock release proceeds with the thread still running. The thread is daemonic so it won't prevent process exit, but it can still call `_extend_script` after the lock has been released and potentially re-acquired by another process, extending a lock it no longer owns. Check `t.is_alive()` after join and log a warning if the thread did not terminate cleanly.

## Test Coverage Gaps

### 7. Validation Tests Are Asserting Against Unhandled Exceptions

As noted in issue 3, these tests are structurally incorrect. They need to be verified against actual behavior, not assumed behavior.

### 8. No Test for Heartbeat Lock Loss Scenario

There is no test covering the case where the heartbeat detects lock loss mid-operation. This is precisely the failure mode that the heartbeat exists to handle, and it has no test coverage at all.

### 9. No Test for Database Transaction Rollback

There is no test verifying that a failure in `create_order` after `update_stock` results in a rollback and does not leave stock in a decremented state with no corresponding order.

### 10. `test_lock_release_is_atomic` Is Testing the Mock, Not the Code

```python
service._release_script.assert_called_once()
redis_client.delete.assert_not_called()
```

`_release_script` is assigned in `__init__` via `redis_client.register_script(...)`. In the test fixture, `register_script` returns a `MagicMock`, so `_release_script` is already a mock. This test confirms that a mock was called, which is trivially true. It does not verify that the Lua script body is correct or that the release is actually conditional on token ownership. Consider a test that runs the Lua script against a real or embedded Redis instance, or at minimum verify the script source is passed to `register_script`.

---

## Minor Notes

- The `order_id` is generated before the transaction opens. If the transaction is retried by the DB layer, a new `order_id` should be generated per attempt, or idempotency needs to be addressed explicitly.
- Log line in `reserve_item` logs `item_id` directly from `req.item_id` which is validated, but the raw `item_id` parameter is also used in the `LockAcquisitionError` handler log before validation has run — this is a low-severity inconsistency but worth aligning.
- Consider whether `remaining_stock` should be omitted from the response on a successful reservation in a flash sale context, as broadcasting exact remaining stock can itself drive demand spikes.
