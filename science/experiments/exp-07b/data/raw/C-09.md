Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a critical race condition, architectural gaps, and test suite deficiencies that must be addressed before this is mergeable into a production inventory system, particularly one operating under flash sale load.

---

## Critical Issues

### 1. Race Condition: Stock Read Outside the Database Transaction

This is the most serious flaw in the implementation.

```python
# OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # based on stale read
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes access across service instances, but it does **not** protect against the database itself. If the DB read and the DB write are not in the same transaction with an appropriate isolation level (e.g., `SELECT FOR UPDATE` or `REPEATABLE READ`), a concurrent writer — a refund process, an admin correction, a different code path that doesn't respect this lock — can modify stock between the read and the write. The stock read must be inside the transaction boundary, and ideally should use a pessimistic row lock (`SELECT ... FOR UPDATE`). The current structure gives a false sense of safety.

---

### 2. Heartbeat Thread Silently Swallows Lock Loss — No Propagation Mechanism

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            heartbeat_running.clear()
            return  # ← thread exits, but the main thread has no idea
```

When the heartbeat detects lock loss, it logs a warning and exits. The main thread continues executing the database transaction **without holding the lock**. This is the exact scenario distributed locking is supposed to prevent. Another process may have acquired the lock and begun its own reservation. You now have two processes simultaneously modifying the same inventory row.

The heartbeat must have a way to signal the main thread to abort. A `threading.Event` for "lock lost" that the main thread checks at critical points, or raising into the main thread via a shared flag checked before the DB write, is required. This is a correctness failure, not a style issue.

---

### 3. `t.join()` Before Lock Release Creates a Timing Vulnerability Window

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # waits up to 9 seconds
    self._release_script(keys=[lock_key], args=[token])
```

The lock is held for up to `HEARTBEAT_INTERVAL + 1` (9 seconds) after the critical section completes, purely waiting for the heartbeat thread to drain. During this window the lock is still held but no useful work is being done, blocking all other waiters. The heartbeat thread should be structured to exit promptly when `heartbeat_running` is cleared — `threading.Event.wait(timeout=...)` will unblock when the event is cleared, so the thread *should* exit within one poll cycle, but the `join(timeout=9)` suggests the author isn't confident in this. If `join` times out, the script is called anyway with no indication that the thread is still running, which could produce a double-release attempt. This needs explicit handling.

## Significant Issues

### 4. No Exception Handling Around the Database Operations

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)
    # ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

If `get_stock`, `update_stock`, or `create_order` raises any exception, it propagates out of `_acquire_lock`'s `try/finally`, which correctly releases the lock — that part is fine. But the exception then propagates out of `reserve_item` entirely, unhandled. The caller receives an unhandled exception rather than a `ReservationResult`. Given that this is a service method with a defined return type contract, DB errors should be caught, logged with full context, and returned as a structured failure result. Letting raw DB exceptions surface to callers is an information leak and breaks the API contract.

### 5. Heartbeat Interval Arithmetic Is Fragile

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # seconds — fires at TTL/3
```

`30 / 3 = 10`, not `8`. The comment is wrong, and more importantly, the values are not derived from each other — they're magic numbers that can drift independently. If someone changes `LOCK_TTL` to `15` and forgets to update `HEARTBEAT_INTERVAL`, the heartbeat fires at TTL/1.875, which is dangerously close to expiry. This should be:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

### 6. Pydantic Validation Errors Are Not Caught

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails, Pydantic raises `ValidationError`. The test `test_input_validation_rejects_invalid_item_id` passes only because the test framework catches the exception and the test checks `result.success is False` — but `reserve_item` never returns a `ReservationResult` in this case; it raises. Either the test is wrong (it would actually fail if run correctly), or the exception is being swallowed somewhere not shown. This must be wrapped in a `try/except ValidationError` block that returns a structured failure result.

## Architectural Concerns

### 7. Single Redis Node — No Redlock

The implementation uses a single Redis client with no mention of Redlock or Redis Sentinel/Cluster awareness. For a flash sale system described as handling "high-demand" concurrent load, a single Redis node is a single point of failure. If Redis goes down or partitions, `redis.set()` raises an exception that is not caught, causing the entire reservation path to throw rather than degrade gracefully. At minimum, document this limitation explicitly. For production, Redlock across 3–5 nodes or a Redis Cluster setup should be considered.

### 8. No Lock Acquisition Timeout — Only Retry Count

```python
for attempt in range(retry_attempts):
    backoff = (2 ** attempt) * 0.1
    time.sleep(backoff)
```

With `retry_attempts=3`, total wait time is `0.1 + 0.2 = 0.3` seconds before giving up. Under flash sale load this may be appropriate, but the retry count and backoff are not configurable at call sites. More critically, `time.sleep` blocks the calling thread. In an async or thread-pool-based web server this will exhaust worker threads under contention. This should be documented, and an async variant or at minimum a configurable timeout should be considered.

### 9. Daemon Thread Leak on Process Shutdown

```python
t = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed abruptly on process exit. If the process is shutting down mid-reservation, the lock may not be released (the `finally` block may not execute), and the heartbeat thread will be killed without the Lua release script running. The lock will then hold for up to `LOCK_TTL` seconds. For a 30-second TTL this is an operational nuisance but not catastrophic — however, it should be documented and a shutdown hook or signal handler considered.

## Test Suite Issues

### 10. Tests Do Not Actually Validate the Heartbeat Behavior

The test suite has zero coverage of the heartbeat thread. There is no test for:
- Lock extension being called on the correct interval
- Behavior when the heartbeat detects lock loss (the most critical failure mode identified above)
- Thread cleanup on normal exit
- Thread cleanup on exception in the critical section

### 11. `test_lock_release_is_atomic` Is Testing the Wrong Thing

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

This test asserts that `_release_script` was called and `delete` was not. But `_release_script` is registered at construction time via `redis_client.register_script(...)`, which on the mock returns another `MagicMock`. The test is asserting that a mock was called — it is not verifying that the Lua script content is correct or that it was called with the right `keys` and `args`. This test would pass even if the script body were empty. Assert on the `keys` and `args` passed to the script call.

### 12. Input Validation Tests Will Fail as Written

As noted in issue #6, `reserve_item` does not catch `ValidationError`, so `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` will raise rather than returning a result with `success=False`. These tests are asserting behavior that the implementation does not actually provide. If the tests are passing in CI, the test runner is misconfigured or the exception is being caught at a higher layer not shown here.

## Minor Issues

- No type annotation on `db` parameter — the DB interface should be a Protocol or ABC so the type checker can enforce the contract.
- `remaining_stock` is not returned on the "item not found" path, which is consistent, but the "insufficient stock" path returns `remaining_stock=current_stock` while "item not found" returns `remaining_stock=None` — callers need to handle both, and this asymmetry should be documented.
- Log message in `reserve_item` uses `item_id=` but logs `req.item_id` — fine, but be consistent about whether you log the raw input or the validated value. Always log the validated value.
