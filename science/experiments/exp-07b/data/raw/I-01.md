Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all directionally correct. However, there are several architectural flaws, a race condition in the heartbeat logic, a critical TOCTOU gap between the stock read and the database transaction, and test coverage that is largely theatrical rather than substantive. This code is not ready for a high-concurrency flash sale environment and should not be merged in its current state.

---

## Detailed Review

### 🔴 Critical Issues

---

#### 1. TOCTOU Race: Stock Read Lives Outside the Database Transaction

This is the most serious flaw in the entire design.

```python
# Inside the lock, but OUTSIDE the transaction:
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # Writes a value derived from the stale read
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are not atomic at the database level. The distributed lock serializes Redis-aware callers, but it provides zero protection against:

- Direct database writes from other services, admin tools, batch jobs, or migration scripts that do not participate in the Redis locking protocol.
- A database replica lag scenario where `get_stock` reads from a follower and `update_stock` writes to the primary.
- Any future code path that bypasses the lock entirely.

The read **must** occur inside the same transaction as the write, and ideally the `UPDATE` should use an optimistic or pessimistic database-level lock (e.g., `SELECT ... FOR UPDATE` or a conditional `UPDATE stock SET quantity = quantity - $1 WHERE item_id = $2 AND quantity >= $1 RETURNING quantity`). The distributed lock should be treated as a performance optimization to reduce DB contention, not as the sole correctness mechanism.

```python
# Correct pattern — read and write are atomic at the DB level:
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock < req.quantity:
        return ReservationResult(success=False, ...)
    self.db.update_stock(req.item_id, current_stock - req.quantity)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

#### 2. Heartbeat Race Condition: `threading.Event.wait()` Misused as a Sleep

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
```

`threading.Event.wait(timeout=N)` returns `True` if the event is **set** within the timeout, and `False` if the timeout expires. Since `heartbeat_running` is set before the thread starts and is only cleared on shutdown, `wait()` returns `True` immediately on every iteration — the heartbeat fires in a tight loop with no delay whatsoever. This will hammer Redis with `EXPIRE` calls continuously for the entire duration of every lock hold.

The intent was clearly to sleep for `HEARTBEAT_INTERVAL` seconds between extensions. The correct idiom is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False, meaning the timeout elapsed without the event being set
        result = self._extend_script(...)
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
    # Event was set (shutdown signal) — exit cleanly
```

This is a silent, severe correctness bug. The existing tests do not catch it because they mock the Redis client and never observe the thread's actual behavior.

---

#### 3. Lock Release After Heartbeat Clear Is Not Safe

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

There is a window between `heartbeat_running.clear()` and `self._release_script(...)` where the heartbeat thread may execute one final `_extend_script` call (because the thread may be mid-iteration when the event is cleared). This is a minor race but worth noting. More importantly, if `t.join()` times out — which can happen if the heartbeat thread is blocked on a slow Redis call — the lock release still proceeds with no indication that the thread is still running. A daemon thread that outlives the lock context and later calls `_extend_script` on a key that has been re-acquired by another caller would silently extend a lock it no longer owns. The Lua script guards against an incorrect extension (token mismatch), but the thread should be more aggressively terminated or the join timeout should be logged as an error.

### 🟠 Significant Issues

---

#### 4. Validation Errors Are Swallowed Silently and Return Ambiguous Results

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If Pydantic raises a `ValidationError`, it propagates up uncaught. The test `test_input_validation_rejects_invalid_item_id` asserts `result.success is False`, but in the actual implementation this would raise an unhandled exception, not return a `ReservationResult`. The test passes only because the mock infrastructure absorbs the exception differently than production would. This needs an explicit `except ValidationError` block that returns a well-formed `ReservationResult` with an appropriate message, or the caller must be responsible for validation before invoking the service.

---

#### 5. `retry_attempts` Is Not Exposed and the Backoff Ceiling Is Dangerously Low

```python
for attempt in range(retry_attempts):  # default: 3
    backoff = (2 ** attempt) * 0.1     # 0.1s, 0.2s, 0.4s
```

Under flash sale load — the exact scenario this service is designed for — three attempts with a maximum backoff of 400ms is almost certainly insufficient. More importantly, `retry_attempts` is a parameter to `_acquire_lock` but is hardcoded at the call site in `reserve_item`. The caller has no ability to tune this. Jitter should also be added to the backoff to prevent thundering herd behavior when many callers retry simultaneously:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

---

#### 6. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Module-Level Constants With No Validation

If someone subclasses `InventoryService` or modifies these values, there is no guard ensuring `HEARTBEAT_INTERVAL < LOCK_TTL`. If `HEARTBEAT_INTERVAL >= LOCK_TTL`, the lock will expire before the first extension attempt. This should be enforced in `__init__`:

```python
if self.HEARTBEAT_INTERVAL >= self.LOCK_TTL:
    raise ValueError("HEARTBEAT_INTERVAL must be less than LOCK_TTL")
```

---

#### 7. `db.transaction()` Exception Handling Is Absent

```python
with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

If `update_stock` succeeds but `create_order` raises, the transaction should roll back — but there is no `except` block to catch database exceptions and return a meaningful `ReservationResult`. Currently, any DB exception propagates out of `reserve_item` entirely, bypassing the lock release in the `finally` block... wait, actually the lock is in a `with` statement so it will release, but the caller receives an unhandled exception rather than a `ReservationResult`. This breaks the contract of the method signature.

### 🟡 Minor Issues

---

#### 8. Tests Are Largely Theatrical

The test suite has structural problems that undermine its value:

- **`test_lock_release_is_atomic`** asserts `service._release_script.assert_called_once()`. But `_release_script` is set in `__init__` via `redis_client.register_script(...)`, which returns a mock. The test is asserting that a mock was called, not that the Lua script logic is correct. The actual Lua script is never executed in any test.

- **The heartbeat thread is never tested.** There is no test that verifies the heartbeat fires, that it stops on lock release, or that it handles a lost lock (result == 0) correctly. Given the bug identified in point #2, this gap is particularly costly.

- **`test_reserve_concurrent_quantity`** is not a concurrency test. It is a sequential test with `quantity=3`. True concurrency testing requires multiple threads or processes.

- **`redis_client.execute_script`** in the fixture is never called by the implementation. It is a dead mock attribute that gives false confidence.

---

#### 9. Lock Key Namespace Could Collide in Multi-Tenant Deployments

```python
lock_key = f"inventory:lock:{req.item_id}"
```

If this Redis instance is shared across environments or tenants, there is no namespace isolation. Consider injecting a configurable prefix:

```python
lock_key = f"{self.key_prefix}:inventory:lock:{req.item_id}"
```

---

#### 10. No Redis Connection Error Handling

A `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` during lock acquisition or heartbeat extension will propagate as an unhandled exception. These should be caught and surfaced as `LockAcquisitionError` or logged and handled gracefully.
