Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped lock state are all directionally correct. However, there are several correctness flaws, a critical race condition in the heartbeat/release sequencing, a fundamental distributed systems flaw (Redis alone cannot safely guarantee mutual exclusion under failure), and test coverage that is largely illusory due to mock misconfiguration. None of these are acceptable for a production flash-sale system under concurrent load.

---

## Critical Issues

### 1. Race Condition: Heartbeat Can Extend a Lock After Release Has Been Decided

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    self._release_script(keys=[lock_key], args=[token])
```

`heartbeat_running.clear()` signals the thread to stop, but `threading.Event.wait(timeout=N)` will return `False` after the timeout — it does **not** wake the thread immediately. The heartbeat thread may be mid-sleep for up to `HEARTBEAT_INTERVAL` (8 seconds) when `clear()` is called. The `join(timeout=9)` then blocks the release for up to 9 seconds. This is the opposite of the intended behavior.

More critically: if the heartbeat thread is currently executing `_extend_script` when `clear()` is called, the sequence becomes:

1. `heartbeat_running.clear()` — sets the stop flag
2. Heartbeat thread completes `_extend_script`, re-extending the TTL
3. Main thread calls `_release_script` — releases the lock correctly

This specific ordering is safe, but the 8-second stall on `join` is not. Under load, this serializes every lock release behind an 8-second wall.

**Fix:** Use a dedicated stop event separate from the wait event, and use `threading.Event` for immediate wakeup:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...

# In finally:
stop_event.set()   # wakes the sleeping thread immediately
t.join(timeout=2)  # should be near-instant now
self._release_script(...)
```

### 2. Distributed Systems Flaw: Single Redis Node Is Not Safe for Mutual Exclusion

The PR uses a single `redis.Redis` client. Under the following real-world scenario, mutual exclusion is violated:

1. Process A acquires lock on Redis primary.
2. Redis primary fails before replicating to replica.
3. Redis replica is promoted.
4. Process B acquires the same lock on the new primary — **two processes now hold the lock simultaneously.**

For a flash-sale inventory system (a correctness-critical financial operation), this is not acceptable. The standard mitigation is [Redlock](https://redis.io/docs/manual/patterns/distributed-locks/) across an odd number (≥3) of independent Redis nodes, or — preferably for strong consistency — a CP system like etcd or ZooKeeper.

At minimum, this architectural constraint must be documented as a known limitation with a ticket to address it. Deploying this as-is implies a safety guarantee that does not exist.

### 3. Stock Read Occurs Outside the Database Transaction

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)  # outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
```

The stock read and the stock write are in different transactional scopes. If the database supports MVCC (PostgreSQL, MySQL InnoDB), another connection could modify stock between the `get_stock` call and the `update_stock` call, even with the Redis lock held — because the Redis lock only coordinates processes using *this service*, not direct DB writers, DB migrations, admin tools, or other services.

The correct pattern is to move the read inside the transaction and use a pessimistic lock at the DB level as a defense-in-depth measure:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    # validate and update within the same transaction
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock and the DB-level lock serve complementary purposes and should both be present.

### 4. `ValidationError` From Pydantic Is Not Caught

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)  # can raise ValidationError
    ...
    try:
        with self._acquire_lock(lock_key):
            ...
    except LockAcquisitionError as e:
        ...
```

`ReservationRequest(...)` is called **before** the `try` block. A `pydantic.ValidationError` will propagate uncaught to the caller as an unhandled exception, not as a `ReservationResult(success=False, ...)`. The tests for input validation (`test_input_validation_rejects_invalid_item_id`, `test_input_validation_rejects_zero_quantity`) are therefore testing that an exception is raised and checking `result.success` — which means they cannot be passing as claimed, or the test harness is swallowing exceptions silently.

**Fix:**

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters.")
```

## Significant Issues

### 5. Tests Are Largely Non-Functional Due to Mock Misconfiguration

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1  # this method does not exist
    return mock
```

`redis_client.register_script(...)` returns a callable script object. `MagicMock()` will return a new `MagicMock` for `register_script(...)`, and calling that mock will return another `MagicMock` — not `1`. The heartbeat's `if result == 0` check will never trigger correctly in tests. The `_release_script` and `_extend_script` are never properly mocked.

`test_lock_release_is_atomic` calls `service._release_script.assert_called_once()` — but `_release_script` is the return value of `register_script(...)` on a `MagicMock`, so this assertion will always pass regardless of whether the Lua script was actually invoked. This test proves nothing.

The tests claim to pass, but they are not testing the actual behavior of the system.

### 6. Heartbeat Thread Leak on `_extend_script` Exception

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = self._extend_script(...)
```

If `self._extend_script(...)` raises (Redis connection error, network partition), the exception propagates out of `_beat`, the thread terminates silently, and the lock will expire at `LOCK_TTL` without renewal. The main thread has no visibility into this failure. The lock will then expire mid-operation, potentially allowing another process to acquire it while the current operation is still running.

**Fix:** Catch exceptions in `_beat`, log them, and signal the main thread or set a shared failure flag.

### 7. `t.join(timeout=...)` Result Is Not Checked

```python
t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
self._release_script(keys=[lock_key], args=[token])
```

`Thread.join(timeout=N)` does not raise if the thread is still alive after the timeout — it returns silently. If the heartbeat thread is stuck (e.g., blocked on a Redis call during a network partition), the release script runs while the heartbeat thread may still be alive and will attempt to extend the lock after it has been released, potentially re-creating the key if the Lua script has a bug.

```python
t.join(timeout=2)
if t.is_alive():
    logger.error("Heartbeat thread did not terminate cleanly key=%s", lock_key)
```

### 8. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `[a-zA-Z0-9_\-]+`, which is correct. However, there is no environment or tenant namespacing. In a multi-tenant or multi-environment deployment (staging keys bleeding into production via a shared Redis), `inventory:lock:item_001` in staging and production are the same key. Add environment and tenant dimensions:

```python
lock_key = f"{self.env}:inventory:lock:{tenant_id}:{req.item_id}"
```

## Minor Issues

### 9. `HEARTBEAT_INTERVAL` Relationship to `LOCK_TTL` Is a Magic Number

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # seconds — fires at TTL/3
```

The comment says "fires at TTL/3" but `30/3 = 10`, not `8`. This is either a stale comment or an off-by-one in the design. Either way, the relationship should be derived, not hardcoded separately:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3  # renew at 1/3 of TTL
```
