Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, heartbeat renewal, Pydantic validation, and scoped lock state are all legitimate design choices. However, there are several correctness bugs, a critical race condition in the heartbeat/release sequence, architectural weaknesses around the Redis-DB consistency boundary, and test suite gaps that would make this unsafe to deploy in a high-stakes flash sale context. The issues range from a daemon thread that can silently swallow errors to a stock read that occurs outside the database transaction, creating a TOCTOU window that the distributed lock alone does not fully close.

---

## Detailed Review

### 1. Critical: TOCTOU Between Stock Read and Transaction

```python
# OUTSIDE the transaction
current_stock = self.db.get_stock(req.item_id)

# ...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)  # INSIDE the transaction
    self.db.create_order(...)
```

The stock read happens outside the database transaction. The distributed lock serializes callers *through this service*, but it provides zero protection against:

- A direct database write from another service, admin tool, or migration script that bypasses this lock entirely.
- A database failover or replica promotion between the read and the write, where the replica may have stale data.
- Any future code path that touches inventory without acquiring this lock.

The correct pattern is to move the read inside the transaction **and** use a pessimistic row-level lock (`SELECT ... FOR UPDATE`) or an optimistic CAS update (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ?`) so the database itself enforces the invariant. The distributed lock should be a *performance optimization* (reducing DB contention), not the *sole correctness mechanism*.

**Recommended fix:**

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    if current_stock is None:
        return ReservationResult(success=False, message=f"Item {req.item_id} not found")
    if current_stock < req.quantity:
        return ReservationResult(success=False, remaining_stock=current_stock, message="Insufficient stock")
    new_stock = current_stock - req.quantity
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(order_id, req.item_id, req.quantity)
```

### 2. Critical: Race Condition in Heartbeat Shutdown Sequence

```python
finally:
    heartbeat_running.clear()          # (1) signal heartbeat to stop
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)  # (2) wait up to 9 seconds
    self._release_script(...)          # (3) release the lock
```

There is a window between steps (1) and (3) where:

- The heartbeat thread has been signaled to stop but has not yet exited.
- The heartbeat thread is mid-execution of `_extend_script`, actively renewing the TTL.
- The main thread proceeds to `_release_script` and deletes the key.
- A new caller acquires the lock.
- The heartbeat thread's in-flight `expire` call completes — but this is now a benign no-op since the key was deleted, so this specific sub-case is safe.

The more dangerous direction: `t.join(timeout=...)` can time out silently. If the heartbeat thread is blocked on a slow Redis call (network partition, Redis slowlog), `join` returns after 9 seconds and the main thread releases the lock — but the heartbeat thread is still alive, still holding a reference to `token` and `lock_key`, and may successfully extend the TTL of a key that is now owned by a different caller. This breaks mutual exclusion.

**Recommended fix:** After `t.join(timeout=...)`, check `t.is_alive()` and log a critical error (or raise) if the thread did not terminate. Do not proceed with lock release if the heartbeat thread is still running.

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if t.is_alive():
        logger.critical(
            "Heartbeat thread did not terminate before lock release key=%s — "
            "mutual exclusion may be violated", lock_key
        )
        # Policy decision: raise here, or accept the risk and release anyway.
        # For a government contracting context, raising is the safer default.
        raise RuntimeError(f"Heartbeat thread failed to terminate for key={lock_key}")
    self._release_script(keys=[lock_key], args=[token])
```

### 3. High: Heartbeat Uses `threading.Event.wait()` Incorrectly as a Timer

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
```

`Event.wait(timeout)` returns `True` if the event is *set* (which it is, because `heartbeat_running.set()` was called before the thread started) and returns immediately — it does not sleep for `HEARTBEAT_INTERVAL` seconds. This means the heartbeat fires in a tight loop, hammering Redis with `expire` calls continuously until `heartbeat_running.clear()` is called, at which point `wait()` returns `False` and the loop exits.

This is a correctness bug masquerading as a design feature. The intent was clearly to fire every 8 seconds. The correct pattern is:

```python
def _beat() -> None:
    while not heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait() returned False (timed out) — time to extend
        result = self._extend_script(...)
        ...
    # wait() returned True — event was set (shutdown signal), exit cleanly
```

Or, using a separate stop event with inverted semantics:

```python
stop_event = threading.Event()

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        self._extend_script(...)

# In finally:
stop_event.set()
t.join(...)
```

### 4. High: Daemon Thread Silently Swallows Exceptions

```python
t = threading.Thread(target=_beat, daemon=True)
```

Any exception inside `_beat` (Redis connection error, serialization error, unexpected `None` return) will be silently discarded. The main thread has no visibility into heartbeat failures beyond the `result == 0` check, which only handles the "lock was stolen" case. A Redis `ConnectionError` mid-heartbeat will cause the thread to exit silently, the lock TTL will expire, another caller will acquire it, and the original caller will continue executing inside what it believes is a protected critical section.

**Recommended fix:** Wrap the heartbeat body in a try/except, propagate failures via a `threading.Event` or a shared exception container, and check that container in the main thread before proceeding with DB operations.

### 5. Medium: No Handling of `ValidationError` from Pydantic

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If `item_id` or `quantity` fails Pydantic validation, a `pydantic.ValidationError` is raised and propagates uncaught to the caller. The tests `test_input_validation_rejects_invalid_item_id` and `test_input_validation_rejects_zero_quantity` assert `result.success is False`, implying they expect a `ReservationResult` to be returned — but the current code raises instead. Either the tests are wrong (they pass only because `ValidationError` is being caught somewhere upstream in the test harness), or the code needs an explicit `try/except ValidationError` block that returns a `ReservationResult`.

```python
from pydantic import ValidationError

try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation request: %s", e)
    return ReservationResult(success=False, message="Invalid request parameters")
```

### 6. Medium: Lock Granularity Is Per-Item, But Stock Reads Are Not Isolated

If two concurrent requests for *different* items arrive simultaneously, they acquire different locks and proceed in parallel — which is correct and intentional. However, if `db.get_stock` reads from a shared database connection pool without read-committed isolation or better, phantom reads between concurrent transactions for different items could produce inconsistent aggregate views. This is less of a code bug and more of an architecture assumption that needs to be documented and validated against the actual database isolation level.

---

### 7. Medium: `retry_attempts` Backoff Does Not Jitter

```python
backoff = (2 ** attempt) * 0.1
```

Pure exponential backoff without jitter causes thundering herd behavior: all contending callers wake up at nearly the same time after each backoff interval and hammer the lock simultaneously. Under flash sale load (the exact scenario this code is designed for), this will amplify contention rather than reduce it.

**Recommended fix:**

```python
import random
backoff = (2 ** attempt) * 0.1 * (0.5 + random.random())
```

### 8. Low: `HEARTBEAT_INTERVAL` Is Not Safely Proportional to `LOCK_TTL`

The comment says "fires at TTL/3" but `30 / 8 = 3.75`, not `3`. More importantly, these are independent constants — if `LOCK_TTL` is changed without updating `HEARTBEAT_INTERVAL`, the heartbeat may fire after the lock has already expired. These should be derived from each other:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```
