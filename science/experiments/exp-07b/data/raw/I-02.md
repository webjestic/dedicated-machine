Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all appropriate choices. However, there are several correctness flaws, a meaningful race condition, and test gaps that must be addressed before this can be approved for a high-concurrency flash sale workload in a government contracting context.

---

## Critical Issues

### 1. TOCTOU Race Between Stock Read and Transaction

This is the most serious flaw. The stock read happens **outside** the database transaction:

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(req.item_id)   # ← outside transaction
    ...
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)  # ← inside transaction
        self.db.create_order(...)
```

The distributed lock serializes Redis-level access, but if `get_stock` reads from a replica with replication lag, or if the DB connection uses `READ COMMITTED` isolation, another process — one not going through this lock (a manual adjustment, an admin tool, a different code path) — could modify stock between the read and the write. The stock read must be inside the transaction, and ideally should use a pessimistic row lock (`SELECT ... FOR UPDATE`) to be safe:

```python
with self.db.transaction():
    current_stock = self.db.get_stock_for_update(req.item_id)  # SELECT FOR UPDATE
    ...
    self.db.update_stock(req.item_id, new_stock)
    self.db.create_order(...)
```

The distributed lock and the DB-level lock should be treated as complementary layers, not substitutes for one another.

### 2. Heartbeat `threading.Event.wait()` Misuse — Incorrect Termination Behavior

```python
def _beat() -> None:
    while heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...
        heartbeat_running.clear()  # ← clears the event, next wait() returns False immediately
        return
```

`threading.Event.wait(timeout)` returns `True` if the event **is set** when it wakes, and `False` on timeout. The intent is to sleep for `HEARTBEAT_INTERVAL` seconds between beats, but the event starts **set** (`heartbeat_running.set()` is called before the thread starts). This means `wait()` returns `True` immediately on the first call, firing the heartbeat extension at near-zero delay rather than after `HEARTBEAT_INTERVAL` seconds. The lock gets extended almost instantly after acquisition, which is wasteful and masks the actual interval logic.

The correct pattern for a periodic heartbeat with a clean shutdown signal is:

```python
stop_event = threading.Event()  # starts unset

def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # wait returns False on timeout (interval elapsed) → extend
        # wait returns True if stop_event is set → exit
        result = self._extend_script(...)
        ...

# To stop:
stop_event.set()
```

This is a behavioral correctness bug, not just a style issue.

### 3. Unhandled Exception in `reserve_item` Leaves Lock Held Longer Than Necessary

The `finally` block in `_acquire_lock` does handle cleanup, but `reserve_item` only catches `LockAcquisitionError`. Any unexpected exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught out of `reserve_item`, surfacing a raw exception to the caller rather than a `ReservationResult`. For a service method, this is an API contract violation. Add a broad catch in `reserve_item`:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error during reservation.")
```

The lock will still be released by the `finally` block, but the caller should receive a structured response.

## Moderate Issues

### 4. Pydantic Validation Exceptions Not Caught at the Right Layer

```python
req = ReservationRequest(item_id=item_id, quantity=quantity)
```

If validation fails, Pydantic raises `ValidationError`. This is currently not caught, so invalid input causes an unhandled exception rather than a clean `ReservationResult(success=False, ...)`. The test `test_input_validation_rejects_invalid_item_id` likely passes only because the test framework is catching it at a higher level, or the mock is absorbing it. Wrap this:

```python
try:
    req = ReservationRequest(item_id=item_id, quantity=quantity)
except ValidationError as e:
    logger.warning("Invalid reservation input: %s", e)
    return ReservationResult(success=False, message="Invalid input.")
```

---

### 5. Lock Key Namespace Collision Risk

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `^[a-zA-Z0-9_\-]+$`, which is good. However, there is no tenant, environment, or service namespace prefix. In a shared Redis instance (common in government multi-tenant deployments), `inventory:lock:item_001` from staging and production are indistinguishable. The key scheme should include at minimum an environment discriminator, and ideally a service namespace:

```python
lock_key = f"{self.namespace}:inventory:lock:{req.item_id}"
```

where `namespace` is injected at construction time.

---

### 6. Retry Backoff Does Not Include Jitter

```python
backoff = (2 ** attempt) * 0.1  # 0.1s, 0.2s, 0.4s
```

Under flash sale load, many concurrent callers will hit the same backoff windows simultaneously, causing retry thundering herd. Add jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

This is a well-documented requirement for distributed retry logic (see AWS Architecture Blog, "Exponential Backoff and Jitter").

## Test Quality Issues

### 7. Tests Do Not Validate the Heartbeat Behavioral Bug

There is no test that verifies the heartbeat fires **after** the interval, not immediately. Given the `Event.wait()` bug described above, a test like the following would have caught it:

```python
def test_heartbeat_does_not_fire_immediately(service, redis_client):
    with patch("time.sleep"):
        with service._acquire_lock("test:lock"):
            # extend_script should NOT have been called yet
            service._extend_script.assert_not_called()
```

### 8. `test_lock_release_is_atomic` Is Structurally Weak

```python
service._release_script.assert_called_once()
```

This asserts the script was called, but does not assert it was called with the correct `keys` and `args` (specifically, that the token passed to release matches the token used during acquisition). A compromised or regressed implementation could call the script with a hardcoded or empty token and this test would still pass.

### 9. No Test for Partial Transaction Failure

There is no test covering the scenario where `db.update_stock` succeeds but `db.create_order` raises an exception. The transaction rollback behavior is untested, and it is not clear from the `db` mock whether `__exit__` with an exception re-raises or suppresses. This should be explicitly tested.

## Minor / Informational

- `HEARTBEAT_INTERVAL = 8` with `LOCK_TTL = 30` gives a ratio of ~3.75x, which is acceptable, but the comment says "TTL/3" — 30/3 is 10, not 8. Either the constant or the comment is wrong. Fix the comment or the value for clarity.
- `t.join(timeout=self.HEARTBEAT_INTERVAL + 1)` — if the heartbeat thread is blocked on a slow Redis call when `clear()` is called, the join may time out and the thread becomes a background zombie. Consider adding a hard timeout on the Redis extend call itself.
- The `db` abstraction is entirely implicit. There is no interface, ABC, or protocol definition for `get_stock`, `update_stock`, `create_order`, and `transaction`. For a government-grade service, the contract should be explicit and typed.
