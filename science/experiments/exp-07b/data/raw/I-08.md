Decision: Request Changes

Summary: The implementation demonstrates solid foundational instincts — atomic Lua scripts, scoped heartbeat state, Pydantic validation, and structured logging are all directionally correct. However, there are several correctness flaws, a meaningful race condition, a TOCTOU gap that undermines the entire locking strategy, and test coverage that does not actually exercise the code paths it claims to validate. None of these are cosmetic — they represent real failure modes under the exact concurrent load this service is designed to handle.

---

## Critical Issues

### 1. TOCTOU Gap: Stock Read Is Outside the Database Transaction

This is the most significant architectural flaw in the PR.

```python
# stock read happens here — outside the transaction
current_stock = self.db.get_stock(req.item_id)

...

with self.db.transaction():
    self.db.update_stock(req.item_id, new_stock)   # write happens here
    self.db.create_order(order_id, req.item_id, req.quantity)
```

The Redis lock serializes access across service instances, but it provides no protection against the database itself. If the underlying database uses `READ COMMITTED` isolation (the default for PostgreSQL and MySQL), a concurrent transaction from a non-lock-aware code path — a manual adjustment, an admin tool, a different service, a migration script — can modify stock between the `get_stock` read and the `update_stock` write. The lock guards the Redis layer, not the database layer.

The stock read must be inside the transaction, and the `UPDATE` should use an optimistic or pessimistic guard at the SQL level:

```sql
-- Pessimistic: row-level lock
SELECT stock FROM inventory WHERE item_id = %s FOR UPDATE;

-- Or optimistic: conditional update
UPDATE inventory SET stock = stock - %s
WHERE item_id = %s AND stock >= %s
RETURNING stock;
```

Without this, overselling is possible even with the distributed lock in place. This is a correctness bug that directly undermines the stated purpose of the service.

### 2. Race Condition: Heartbeat Thread Can Extend a Lock After Intentional Release

```python
finally:
    heartbeat_running.clear()
    t.join(timeout=self.HEARTBEAT_INTERVAL + 1)          # waits up to 9s
    self._release_script(keys=[lock_key], args=[token])  # release happens after join
```

`heartbeat_running.clear()` signals the thread to stop, but the thread is mid-sleep at `heartbeat_running.wait(timeout=self.HEARTBEAT_INTERVAL)`. The `join` call blocks for up to `HEARTBEAT_INTERVAL + 1 = 9` seconds waiting for that sleep to drain. During those 9 seconds, the lock has not yet been released. This is the opposite of the intended behavior — the heartbeat exists to keep the lock alive during work, not during teardown.

The fix is to use an explicit stop event separate from the wait mechanism, or to interrupt the sleep:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock...

# In finally:
stop_event.set()   # wake the thread immediately
t.join(timeout=2)
self._release_script(...)
```

With the current implementation, under the worst case, lock release is delayed by nearly the full heartbeat interval on every successful reservation. Under high throughput this becomes a meaningful bottleneck.

### 3. Heartbeat Thread Is Not Joined on Exception Paths Within the Lock Body

If `self.db.get_stock()`, `self.db.update_stock()`, or `self.db.create_order()` raises an unhandled exception, the `finally` block in `_acquire_lock` does execute — but the exception propagates up through `reserve_item`, which only catches `LockAcquisitionError`. Any other database exception will surface as an unhandled error to the caller with no `ReservationResult` returned.

```python
try:
    with self._acquire_lock(lock_key):
        ...
except LockAcquisitionError as e:
    ...
    return ReservationResult(success=False, message="Service temporarily unavailable...")
# No handler for redis.RedisError, db connection errors, etc.
```

At minimum, a broad exception handler should be added to guarantee a `ReservationResult` is always returned, and database exceptions should be logged with enough context to diagnose failures:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", req.item_id)
    return ReservationResult(success=False, message="Internal error. Please try again.")
```

## Moderate Issues

### 4. `register_script` Return Value Is Not a Callable in All Redis Client Versions

```python
self._release_script = redis_client.register_script(RELEASE_LOCK_SCRIPT)
self._extend_script = redis_client.register_script(EXTEND_LOCK_SCRIPT)
```

`register_script` returns a `Script` object. Calling it as `self._release_script(keys=[...], args=[...])` works with `redis-py` but is not universally portable and is not the idiomatic modern approach. The preferred pattern uses `redis.client.Script` explicitly or `redis.execute_command("EVAL", ...)` for clarity. More importantly, the mock in the test fixture sets `mock.execute_script.return_value = 1`, which is not the method being called — the tests are not actually validating script execution behavior at all (see test issues below).

---

### 5. Lock Key Namespace Collision Is Possible

```python
lock_key = f"inventory:lock:{req.item_id}"
```

`item_id` is validated to `^[a-zA-Z0-9_\-]+$`, which is good. However, if this Redis instance is shared across environments (staging keys bleeding into production, or multiple tenants), there is no environment or tenant prefix. This is a deployment concern but worth flagging at review time for a government contracting context where environment separation is often a compliance requirement.

---

### 6. `LOCK_TTL = 30` With `HEARTBEAT_INTERVAL = 8` Is Aggressive for Cold Start

The comment says "fires at TTL/3" which is `30/3 = 10`, but the actual interval is `8`. This is a minor inconsistency, but more importantly: on a cold Redis connection or under GIL pressure in CPython, the first heartbeat fires at `t+8s`. If the DB operation takes longer than `30s` before the first heartbeat fires (e.g., a slow query on first acquisition), the lock expires before the heartbeat can renew it. The heartbeat interval should be `TTL / 3` enforced programmatically, not as a magic constant:

```python
HEARTBEAT_INTERVAL = LOCK_TTL // 3
```

## Test Issues

### 7. The Mock Setup Does Not Reflect How `register_script` Works

```python
@pytest.fixture
def redis_client():
    mock = MagicMock()
    mock.set.return_value = True
    mock.execute_script.return_value = 1   # ← this method is never called
    return mock
```

`InventoryService.__init__` calls `redis_client.register_script(...)` twice. `MagicMock` will return a new `MagicMock` for each call, which is callable and returns another `MagicMock` by default. This means `_release_script` and `_extend_script` are mock callables that return truthy values — the tests pass not because the logic is correct, but because `MagicMock` silently absorbs every call. The heartbeat's `if result == 0` check will never trigger in tests because `MagicMock() == 0` is `False`.

The fixtures need to explicitly configure the script mock return values:

```python
script_mock = MagicMock(return_value=1)
mock.register_script.return_value = script_mock
```

### 8. `test_lock_release_is_atomic` Does Not Test What It Claims

```python
def test_lock_release_is_atomic(service, redis_client):
    service.reserve_item("item_001")
    service._release_script.assert_called_once()
    redis_client.delete.assert_not_called()
```

`service._release_script` is the return value of `register_script(RELEASE_LOCK_SCRIPT)`, which on the mock is a `MagicMock`. Asserting it was called once does not verify the Lua script content, the keys passed, or the token argument. The test would pass even if the implementation called `redis_client.delete` directly and the assertion on `redis_client.delete` was the only real guard — but `MagicMock` absorbs `delete` calls silently unless configured otherwise. This test provides false confidence.

### 9. No Test for Heartbeat Behavior Under Lock Loss

There is no test that simulates the heartbeat detecting a stolen lock (`result == 0` from `_extend_script`) and verifying the thread exits cleanly. This is one of the more complex code paths and it is entirely untested.

### 10. No Test for Database Exception Propagation

There is no test verifying behavior when `db.get_stock`, `db.update_stock`, or `db.create_order` raises an exception. Given the missing broad exception handler noted in issue 3, this path currently raises uncaught exceptions to the caller.
