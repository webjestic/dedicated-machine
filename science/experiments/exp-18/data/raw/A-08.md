Decision: Request Changes

Summary: The lock TTL is 60 seconds, but the worst-case execution time while holding the lock can reach approximately 135 seconds — meaning the lock expires and becomes re-acquirable by a competing worker while the first worker is still mid-flight, before the idempotency record has been written. This is a critical double-charge vulnerability. Several additional issues compound the risk and must be resolved before this can be approved.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the primary rejection reason. Trace the full execution path while the lock is held:

```
ORDER_LOCK_TTL = 60 seconds

Worst-case execution while holding the lock:

  payment.charge() — one attempt:
    REQUEST_TIMEOUT = 30s (timeout fires)
    → last_error set, loop continues

  sleep(RETRY_BACKOFF) between attempts:
    RETRY_BACKOFF = 5s
    → sleeps between attempt 1→2, and 2→3

  Three full attempts:
    30s + 5s + 30s + 5s + 30s = 100s  ← payment alone

  db.get_charge()          ~1–5s (under load)
  db.transaction() commit  ~1–5s (under load)
  db.record_charge()       ~included in txn
  db.mark_order_complete() ~included in txn

  Realistic worst case: 100s + 10s overhead = ~110–135s
```

**At T=60s, the lock expires.** A second worker can now acquire the lock on the same `order_id`. At that moment, `db.record_charge()` has not yet been called — the idempotency check `db.get_charge()` returns `None` for the second worker. Both workers proceed to call `payment.charge()`. The customer is charged twice.

The fix requires at minimum one of the following, and ideally both:

**Option A — Reduce worst-case duration below TTL.** Cut `MAX_RETRIES` and `REQUEST_TIMEOUT` so the math closes. For example, `REQUEST_TIMEOUT=10`, `MAX_RETRIES=2`, `RETRY_BACKOFF=2` yields a worst case of roughly `10 + 2 + 10 = 22s`, safely under a 60s TTL with headroom. This is the simpler fix but it constrains your retry policy.

**Option B — Implement lock extension (the correct production fix).** Run a background heartbeat that calls `EXPIRE order:lock:{order_id} 60` every 20–30 seconds while the worker is still alive and holds the token. Only extend if the current value matches the worker's token (use a Lua script, same pattern as release). This keeps the TTL from expiring under a legitimately slow but still-alive worker.

Both options should be combined with a hard per-attempt timeout enforced at the `process()` call site so a runaway worker cannot hold a lock indefinitely.

## High: `payment.charge()` Is Called Inside the Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network I/O, up to 100s worst case
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction should hold a connection from the pool for its entire duration. Placing a network call with up to 100 seconds of worst-case latency inside the transaction means:

- The DB connection is held for the full duration of all payment retries.
- Under load this exhausts the connection pool, causing cascading failures across all DB operations in the service.
- If the payment gateway returns success but the subsequent `record_charge()` fails and the transaction rolls back, you have a charge with no record — a financial reconciliation gap with no recovery path in this code.

The correct structure is:

```python
# Outside the transaction:
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# Only then open the transaction for the fast DB writes:
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This does introduce the charge-succeeds/record-fails failure mode, which must be handled explicitly — see the `PaymentError` handling note below.

## High: `PaymentError` Is Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # can raise PaymentError
        ...
```

`PaymentError` propagates uncaught out of `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. The lock is still released correctly by the `finally` block, but the caller has no way to distinguish a payment failure from a programming error without catching the raw exception. Add an explicit `except PaymentError` block that returns `OrderResult(processed=False, message="Payment failed")` and logs the error with the order ID.

## Medium: Lock Retry Backoff Adds to Worst-Case Lock-Wait Time, Not Lock-Hold Time — But Is Still Misconfigured

The lock acquisition retry loop:

```python
LOCK_RETRY_ATTEMPTS = 3
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s
```

This is not a lock-hold issue, but the retry logic is inverted. On the last attempt (`attempt == LOCK_RETRY_ATTEMPTS - 1`), the code sleeps and then exits the loop without a final acquisition attempt. The sleep on the last attempt is wasted. The standard pattern is to sleep only when there will be another attempt:

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    if attempt < self.LOCK_RETRY_ATTEMPTS - 1:
        backoff = 0.5 * (attempt + 1)
        time.sleep(backoff)
```

The payment client has the same inverted pattern and correctly guards with `if attempt < self.MAX_RETRIES - 1`, so the fix is already demonstrated in the same PR.

## Medium: No Handling for Charge-Succeeds / Record-Fails

Once `payment.charge()` is moved outside the transaction (as required above), a new failure window opens: the charge succeeds at the gateway, but `record_charge()` or `mark_order_complete()` raises an exception and the transaction rolls back. The order now has a real charge with no local record. On retry, `get_charge()` returns `None`, and the customer is charged again.

This requires an outbox or compensation pattern. At minimum, log the `charge_id` durably before opening the transaction (e.g., write to a `pending_charges` table outside the main transaction, or use a structured log that feeds a reconciliation job). A full outbox implementation is the correct long-term fix.

## Low: `payment_token` Is Logged at INFO Level on Success

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is fine, but the `payment_token` field from `OrderRequest` flows into `payment.charge(token=req.payment_token, ...)`. Verify that no log line anywhere in the call stack emits the raw token. The `OrderRequest` object should not be logged wholesale. The current code does not appear to log the token directly, but this should be confirmed and a `__repr__` override or `model_config` with `hide_input_in_errors=True` added to `OrderRequest` to prevent accidental exposure in validation error messages.

---

## Low: Tests Do Not Cover the TTL Expiry Race

The test suite covers happy path, idempotency, lock contention, and input validation. There is no test that simulates the lock expiring mid-execution and a second worker acquiring it. This race is the primary failure mode of the design. A test using `threading` or mocked time that verifies the second worker's idempotency check fires correctly (or, after the fix, that the heartbeat extends the TTL) should be added.

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| Critical | Lock TTL (60s) shorter than worst-case execution time (~135s) — double-charge risk |
| High | `payment.charge()` inside DB transaction — connection pool exhaustion and unrecoverable charge/record split |
| High | `PaymentError` uncaught in `process()` — unstructured failure surface |
| Medium | Lock retry sleeps on final attempt — wasted wait |
| Medium | No compensation path for charge-succeeds/record-fails after transaction restructure |
| Low | Ensure `payment_token` cannot appear in logs or validation error output |
| Low | No test coverage for the TTL expiry race condition |
