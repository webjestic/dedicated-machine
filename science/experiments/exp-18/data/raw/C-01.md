Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals â€” atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there is a critical architectural flaw that undermines the at-most-once payment guarantee the PR claims to provide, along with several secondary reliability and security concerns that must be addressed before this handles real payment data.

---

## Critical Issue

### Payment charge occurs outside the database transaction â€” at-most-once guarantee is broken

This is the most serious bug in the PR. The code places `self.payment.charge(...)` *inside* the `with self.db.transaction():` block, but the payment gateway call is a **non-transactional external side effect**. Database transactions provide atomicity over database operations only. The payment gateway cannot participate in a database rollback.

```python
# Current code â€” THIS IS WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External call â€” NOT rolled back on DB failure
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Failure scenario:** The charge succeeds at the gateway, then `record_charge` or `mark_order_complete` raises an exception. The database transaction rolls back, leaving no record of the charge. On retry, `get_charge` returns `None`, the idempotency check passes, and the customer is charged a second time. The distributed lock does not protect against this because the lock is released before the retry.

**Required fix:** Charge first, then record atomically. If the DB write fails after a successful charge, you need a reconciliation path â€” not a silent retry.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Charge succeeded. Now persist atomically. If this fails, the charge
# exists at the gateway but has no DB record â€” requires reconciliation,
# NOT a transparent retry that would re-charge.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

The broader point is that this service needs a defined reconciliation strategy for the window between a successful gateway charge and a failed DB write. That strategy should be documented and ideally implemented (e.g., a dead-letter queue or a pending-charges table checked on startup).

## High Severity

### `PaymentError` is silently swallowed

`PaymentError` is never caught in `order_processor.py`. If the payment client raises it, the exception propagates out of `process()` uncaught, bypassing the structured `OrderResult` return contract and potentially crashing the caller. At minimum it needs to be caught and returned as a structured failure. More importantly, the caller needs to distinguish "charge definitely did not happen" from "charge status unknown" â€” these require different handling.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### Timeout on payment charge is longer than the Redis lock TTL under retry

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient.REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst-case payment client execution time: `3 Ă— 30 + 2 Ă— 5 = 100 seconds`. The Redis lock expires at 60 seconds, meaning a second worker can acquire the lock and begin processing while the first worker is still waiting on a payment response. The idempotency check will pass for the second worker because no charge has been recorded yet. This is a direct path to double-charging.

Either extend `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime with margin, or implement lock extension (Redlock-style keepalive) during the payment call.

## Medium Severity

### `payment_token` is logged at INFO level on success

```python
# In payment_client.py
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is fine, but `order_processor.py` logs `req.order_id` and `charge_id`. Verify that `payment_token` (a raw payment credential) never enters any log statement. Currently it does not appear to, but the `OrderRequest` object containing it is in scope throughout `process()`. Add a note or a `__repr__` override on `OrderRequest` that redacts sensitive fields to prevent accidental future logging.

### `requests.exceptions.ConnectionError` is not handled in `PaymentClient`

Network-level failures (DNS failure, connection refused, socket reset) raise `requests.exceptions.ConnectionError`, which is not caught. It will propagate as an unhandled exception rather than triggering the retry logic. Add it alongside the `Timeout` handler.

```python
except (requests.exceptions.Timeout, requests.exceptions.ConnectionError):
    last_error = "connection error"
    ...
```

### No test covers the payment-failure-after-lock-acquired path

The test suite covers lock contention and idempotency but has no test for `PaymentError` being raised inside the lock. Given that this is an unhandled exception path (see above), this gap allowed the bug to ship with a passing test suite.

## Low Severity / Nits

- `charge_id` is extracted from the response JSON without a `KeyError` guard (`response.json()["charge_id"]`). A malformed but 200-OK gateway response will raise an unhandled `KeyError`. Use `.get("charge_id")` with an explicit check.
- `CHARGE_ENDPOINT` is hardcoded as a class attribute. This should be injected via configuration or environment variable to support staging/production environments without code changes.
- The `LOCK_RETRY_ATTEMPTS = 3` with `0.5s` backoff gives a maximum wait of 3 seconds before failing. For a 60-second lock TTL this seems intentionally short. Document whether this is intentional â€” if a worker holds the lock for the full TTL, callers will fail rather than wait. That may be acceptable, but it should be explicit.
