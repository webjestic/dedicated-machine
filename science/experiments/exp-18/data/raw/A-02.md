Decision: Request Changes

Summary: The implementation contains a critical lock-expiry vulnerability: the worst-case duration of work performed while the lock is held can substantially exceed the 60-second TTL, meaning a second worker will acquire the lock, pass the idempotency check on an in-flight (not yet recorded) charge, and issue a duplicate payment. Several secondary issues compound the risk. The details follow.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the primary rejection reason.

**TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case wall-clock time while the lock is held**, tracing every blocking call and every retry/backoff cycle:

```
payment_client.charge() — one attempt:
  requests.post(..., timeout=30)  →  up to 30 s per attempt

PaymentClient constants:
  MAX_RETRIES   = 3
  RETRY_BACKOFF = 5 s   (sleep between attempts)

Worst-case charge() duration:
  attempt 1: 30 s timeout  →  sleep 5 s
  attempt 2: 30 s timeout  →  sleep 5 s
  attempt 3: 30 s timeout  →  no trailing sleep
  Total: (30 + 5) + (30 + 5) + 30 = 100 s
```

**100 s > 60 s TTL.**

At the 60-second mark Redis expires the key. A second worker calls `SET NX EX`, acquires the lock, calls `db.get_charge()`, finds nothing recorded (the first worker is still waiting on attempt 3 of its HTTP call), and proceeds to charge the same payment token. Both workers then record a charge, resulting in a double charge.

The Lua release script does not help here — it only prevents a worker from releasing a lock it does not own. It does not prevent the lock from expiring while the worker is still running.

**Required fix — choose one or combine:**

1. **Extend the TTL** to safely exceed worst-case execution time. With the current retry parameters the minimum safe TTL is `ceil(100 s) + safety margin`, so at least 120–150 s. This is the minimum viable fix but does not eliminate the race; it only widens the window before it becomes exploitable.

2. **Reduce `REQUEST_TIMEOUT` and `RETRY_BACKOFF`** so that `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` is well under the TTL. For example: timeout=8 s, backoff=2 s, 3 retries → worst case 30 s, giving comfortable headroom under a 60 s TTL.

3. **Implement lock extension (pexpire heartbeat).** Run a background thread or async task that refreshes the lock TTL every `TTL/2` seconds while the worker is alive. Release the heartbeat in the `finally` block. This is the production-grade solution.

4. **Record a pending/in-flight sentinel before charging.** Before calling `payment.charge()`, write a `pending` record to the database (within its own transaction). The idempotency check must treat `pending` as "already in flight" and reject concurrent workers. This makes the idempotency check durable across lock expiry.

## Secondary Issues

### 1. `payment.charge()` is called inside `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call, up to 100 s
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

An external network call inside a database transaction holds a DB connection (and potentially row locks) for the full duration of the HTTP call including all retries. Under load this exhausts the connection pool and causes cascading failures unrelated to the order being processed. The charge call must be moved **outside and before** the transaction:

```python
charge_id = self.payment.charge(...)   # outside transaction
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This does introduce a narrow window where a charge succeeds but the DB write fails. That is handled by the pending-sentinel pattern mentioned above, or by a reconciliation job, not by wrapping the HTTP call in the transaction.

### 2. Timeout on a successful charge is treated as a transient failure and retried

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the gateway received the request but the response was lost in transit. The charge may already have been applied. Retrying without a gateway-side idempotency key will issue a second charge. The `charge()` call must include a stable, caller-generated idempotency key (e.g., derived from `order_id`) sent as a request header or body field, so the gateway deduplicates retries server-side.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": idempotency_key},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

### 3. Lock retry backoff accumulates before the lock is acquired

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)   # 0.5 s, 1.0 s
    time.sleep(backoff)
```

Worst-case lock acquisition time: 0.5 + 1.0 = 1.5 s. This is minor but it is wall-clock time that eats into the TTL before any work begins. It should be accounted for in the TTL calculation.

### 4. `PaymentError` is not caught in `process()`

If `payment.charge()` raises `PaymentError` (permanent decline or exhausted retries), the exception propagates out of `process()` uncaught. The caller receives an unhandled exception rather than a structured `OrderResult`. Add an explicit handler:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 5. Test suite does not cover the TTL expiry race

The tests mock Redis and do not exercise the scenario where the lock expires mid-flight. There is no test for `PaymentError` propagation, no test for timeout-then-retry producing a duplicate charge, and no test asserting that `payment.charge()` is called with an idempotency key. These gaps allowed the critical flaw to pass the checklist undetected.

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| **Critical** | Worst-case execution time (100 s) exceeds lock TTL (60 s) — duplicate charge possible |
| **Critical** | Timeout retries without idempotency key — duplicate charge on lost response |
| **High** | External HTTP call inside DB transaction — connection pool exhaustion |
| **High** | `PaymentError` uncaught in `process()` — unhandled exception to caller |
| **Medium** | No test coverage for TTL expiry race, PaymentError, or idempotency key assertion |
