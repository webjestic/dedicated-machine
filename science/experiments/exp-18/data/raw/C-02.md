Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check inside the lock are all correct. However, there is a critical architectural flaw that breaks the at-most-once payment guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles payment data in production.

---

## Critical Issue

### Payment charge happens outside the database transaction — atomicity guarantee is false

The PR description states: *"charges the payment method and records the transaction within a single atomic database transaction."* This is incorrect and the flaw is dangerous.

```python
# services/order_processor.py
with self.db.transaction():
    charge_id = self.payment.charge(       # ← external HTTP call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an external HTTP call to a payment gateway. Wrapping it in a database transaction does not make it atomic with the database writes. The database transaction provides no rollback capability over the external charge. This creates two real failure modes:

**Scenario A — Charge succeeds, DB write fails:**
The customer is charged. The `record_charge` or `mark_order_complete` call throws. The database transaction rolls back. The order now has no charge record. The next worker acquires the lock, sees no existing charge, and charges the customer again. Double charge.

**Scenario B — DB connection drops mid-transaction:**
Same outcome. The charge went through at the gateway but the record never committed.

The correct pattern is to separate the concerns:

```python
# Correct structure
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        return OrderResult(processed=False, message="Order already processed")

    # Step 1: record intent / pending state before charging
    with self.db.transaction():
        self.db.mark_order_pending(req.order_id)

    # Step 2: charge externally — outside any db transaction
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )

    # Step 3: record the result atomically
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
```

Even this requires a reconciliation job for orders stuck in `pending` state after a crash between steps 2 and 3, but it eliminates the double-charge path. The payment gateway's own idempotency key support should also be used if available.

## High Severity

### Timeout exception does not break the retry loop in `PaymentClient`

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the request may or may not have been processed by the gateway — the response was simply not received in time. Retrying blindly on a timeout can cause the same charge to be submitted multiple times if the first request was received and processed. This is especially dangerous here because the payment client has no idempotency key on the request payload.

```python
# Minimum fix: include an idempotency key per logical charge attempt
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # caller must supply this
    timeout=self.REQUEST_TIMEOUT,
)
```

The `order_id` is a natural idempotency key and should be threaded through from `OrderProcessor` into `PaymentClient.charge()`.

### `PaymentError` is not caught in `OrderProcessor.process()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # can raise PaymentError
    self.db.record_charge(...)
```

If `PaymentError` is raised, it propagates out of `process()` as an unhandled exception. The caller gets a 500-level error with no structured `OrderResult`. This also means the lock is released (correctly, via `finally`), but the caller has no way to distinguish a payment decline from a server error without catching the raw exception type. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Medium Severity

### Lock TTL is too short relative to payment client timeout

`ORDER_LOCK_TTL = 60` seconds. `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst-case payment client runtime: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock while the first is still mid-retry, defeating the entire concurrency guarantee. Either extend the TTL or implement lock heartbeat renewal.

### `charge_id` key access on gateway response is unguarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected response body, this raises a `KeyError` or `JSONDecodeError` that propagates uncaught. Use `.get()` with validation:

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned success but no charge_id: {data}")
```

## Low Severity / Test Coverage Gaps

- There is no test covering the `PaymentError` path. Given that this is the most operationally likely failure mode, it needs coverage.
- There is no test verifying that a payment timeout does not result in a double charge (this is partially a design gap, but it should be documented at minimum).
- `customer_id` has no pattern validation in `OrderRequest`. Depending on how it flows into the database layer, this could be a vector for injection if the DB abstraction is not parameterized. Worth confirming.
- `CHARGE_ENDPOINT` is hardcoded as a class attribute. This should be injected via configuration for environment portability and testability.
