Decision: Request Changes

Summary: The PR's core safety claim — that the distributed lock prevents duplicate charges — is broken by a TTL/worst-case-duration mismatch that allows a second worker to acquire the lock while the first is still inside the critical section. A secondary but equally serious issue is that `payment.charge()` is called inside a database transaction, creating a window where a successful charge can be permanently lost with no recovery path. These two flaws together mean the system can both double-charge customers and silently drop confirmed charges, which is the opposite of the at-most-once guarantee the PR advertises.

---

## Critical Findings

### 1. Lock TTL Can Expire Before the Critical Section Completes (Duplicate Charge Risk)

This is the central flaw. The PR claims the lock guarantees at-most-once processing, but the numbers do not support that claim.

**Lock budget:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case time consumed before the lock is even acquired:**

```python
# _acquire_lock retry loop — worst case: 3 attempts, all failing except the last
# attempt=0: sleep 0.5s
# attempt=1: sleep 1.0s
# attempt=2: lock acquired — no sleep
# Total pre-acquisition sleep: 1.5s (minor, but real)
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)
```

The TTL clock starts at the moment `SET NX EX` succeeds. Everything after that point burns down the 60-second budget.

**Worst-case time consumed inside the critical section:**

```python
# PaymentClient.charge() — worst case: 3 attempts × (30s timeout + 5s backoff)
# attempt 0: 30s timeout  → sleep 5s
# attempt 1: 30s timeout  → sleep 5s
# attempt 2: 30s timeout  → no sleep (last attempt)
# Total: 90s + 10s = 100s
REQUEST_TIMEOUT = 30
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

**100 seconds of payment retries alone exceeds the 60-second lock TTL by 40 seconds.**

During those 40 seconds, the lock has expired. A second worker can call `SET NX EX` on the same `order:lock:{order_id}`, succeed, read `db.get_charge()` (which returns `None` because the first worker has not yet committed), and proceed to charge the customer a second time. Both workers then attempt to record the charge, resulting in a duplicate charge and a race on the database write.

The Lua release script does not help here. It correctly prevents a worker from releasing a lock it does not own, but it does not prevent the lock from expiring naturally while the worker is still running.

**Required fix:** The TTL must be strictly greater than the worst-case duration of every operation inside the critical section. Given the current payment client parameters, the minimum safe TTL is:

```
worst_case_payment = MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
                   = 3 × 30 + 2 × 5 = 100s

# Add DB operation headroom (e.g., 10s)
minimum_safe_ttl = 100 + 10 = 110s
```

Either raise `ORDER_LOCK_TTL` to at least 110 seconds, or — better — reduce the payment client's worst-case duration to fit within the TTL. The cleanest solution is a lock-refresh (fencing) mechanism or a background heartbeat that extends the TTL while the worker is still alive. Alternatively, reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` so the math closes:

```python
# Example: REQUEST_TIMEOUT=10, MAX_RETRIES=2, RETRY_BACKOFF=2
# worst case: 2×10 + 1×2 = 22s → fits comfortably inside 60s TTL
```

Whatever values are chosen, the constraint `ORDER_LOCK_TTL > worst_case_critical_section_duration` must be explicitly documented and enforced, ideally with an assertion or a unit test that computes both values.

### 2. `payment.charge()` Is Called Inside a Database Transaction (Lost Charge Risk)

```python
with self.db.transaction():
    charge_id = self.payment.charge(      # ← network call to external system
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is a classic two-phase commit problem. The payment gateway and the database are two independent systems with no shared transaction coordinator.

**Failure scenario:**

1. `payment.charge()` succeeds. The customer's card is charged. The gateway returns `charge_id`.
2. `self.db.record_charge()` or `self.db.mark_order_complete()` raises an exception (deadlock, connection drop, disk full, etc.).
3. The database transaction rolls back. `charge_id` is never persisted.
4. The lock expires or is released. The idempotency check (`db.get_charge()`) now returns `None`.
5. The next worker acquires the lock, sees no existing charge, and charges the customer again.

The customer is charged twice, and the system has no record of either charge being the duplicate.

**Additionally**, holding a database transaction open across a network call to an external payment gateway (which can take up to 30 seconds per attempt, 100 seconds worst-case) is a resource exhaustion vector. Database connections and transaction locks are held for the entire duration of the payment retry loop, which will cause connection pool starvation under any meaningful load.

**Required fix:** Separate the payment call from the database transaction entirely.

```python
# Correct ordering:
# 1. Check idempotency (inside lock, outside transaction)
existing = self.db.get_charge(req.order_id)
if existing:
    return already_processed_result

# 2. Persist intent before charging (outbox / pending record)
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

# 3. Charge outside the transaction
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 4. Confirm the charge in a separate transaction
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

With a pending record written before the charge, the idempotency check in step 1 will catch re-entrant attempts even if the confirmation transaction has not yet committed. A background reconciliation job can detect pending records that never received a confirmation and resolve them against the gateway's idempotency key.

## Moderate Findings

### 3. No Idempotency Key Passed to the Payment Gateway

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # ← no idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

On a timeout, the client cannot know whether the charge succeeded or failed at the gateway. The retry will send a second request with no idempotency key, so the gateway treats it as a new charge. This is the standard cause of duplicate charges at the network layer, independent of the distributed lock issue.

**Required fix:** Pass `order_id` (or a deterministic derivative of it) as an idempotency key in the request header or body, per the gateway's API contract.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": order_id},
    timeout=self.REQUEST_TIMEOUT,
)
```

### 4. `PaymentError` Is Silently Swallowed in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # can raise PaymentError
        ...
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which means the `finally` clause will run (correctly releasing the lock), but the caller receives an unhandled exception rather than a structured `OrderResult`. The PR's own return type annotation promises `OrderResult` in all cases.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 5. Test Coverage Does Not Exercise the Failure Modes That Matter

The tests validate the happy path and basic contention, but none of them test the scenarios that the lock and transaction logic are supposed to protect against:

- No test for lock expiry mid-operation (the critical race condition identified above).
- No test for `PaymentError` propagation and the missing `except` branch.
- No test for database transaction rollback after a successful charge.
- No test verifying that the payment client passes an idempotency key.
- No test for timeout behavior on the payment gateway call.

These are the exact scenarios where the bugs live. Tests that only cover the happy path provide false confidence for a payment system.
