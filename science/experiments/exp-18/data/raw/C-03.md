Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic NX/EX acquisition, token-verified Lua release, and the idempotency check are all sound. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide, plus several reliability and correctness issues that must be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge executes *inside* the database transaction — violates at-most-once guarantee

This is the most serious bug in the PR. The `payment.charge()` call is made inside `self.db.transaction()`:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect inside a DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `record_charge` or `mark_order_complete` raises after the charge succeeds, the DB transaction rolls back, but **the money has already moved**. On retry, the idempotency check finds no recorded charge and charges the customer again. This is a double-charge bug.

The payment gateway call must happen *before* opening the DB transaction, and the charge ID must be persisted atomically only after a confirmed successful charge:

```python
# Correct ordering
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    # Re-check idempotency inside the transaction to close the TOCTOU window
    if self.db.get_charge(req.order_id):
        # charge succeeded but was already recorded by a concurrent worker
        # handle or void the duplicate charge here
        ...
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even with this fix, you need a compensating action (void/refund) if DB persistence fails after a successful charge. That path must be explicitly handled and logged, not silently swallowed.

---

### 2. `PaymentError` is not caught in `order_processor.py`

`payment.charge()` can raise `PaymentError`, but `OrderProcessor.process()` has no handler for it. This exception will propagate uncaught to the caller with no `OrderResult` returned, breaking the contract of the method and potentially crashing the worker.

```python
# Missing in process()
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Significant Issues

### 3. TOCTOU race on idempotency check

The `get_charge` check happens *outside* the DB transaction. Two workers can both pass the check, both acquire the lock sequentially, and both charge. The lock helps but does not eliminate this if the lock TTL expires mid-flight or Redis has a failover. The idempotency check must be inside the DB transaction with a row-level lock or unique constraint on `order_id` as the true safety net. The Redis lock is a performance optimization, not a correctness guarantee.

### 4. Lock TTL of 60 seconds is dangerously short given retry behavior

`PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105 seconds` in the worst case. The lock TTL is 60 seconds. The lock will expire and another worker can acquire it while the first worker is still mid-charge. This directly enables double-charging. Either increase the TTL significantly, implement lock extension (watchdog pattern), or reduce the payment client's maximum latency to fit within the TTL with margin.

### 5. Timeout exception handling is incomplete in `payment_client.py`

`requests.exceptions.ConnectionError` is not caught. A DNS failure or refused connection will propagate as an unhandled exception rather than being retried or converted to a `PaymentError`. Add `requests.exceptions.ConnectionError` to the retry logic alongside `Timeout`.

### 6. `response.json()["charge_id"]` has no error handling

If the payment gateway returns a 200 with a malformed body or missing `charge_id` key, this raises a bare `KeyError` or `json.JSONDecodeError` that is not caught and not retried. This should be wrapped:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

---

## Minor Issues

### 7. Logging `payment_token` risk

Confirm that `payment_token` is never passed to any log statement anywhere in the call chain. It is not logged here directly, but it flows through to `payment.charge()` and any exception messages from the gateway response could echo it back. Scrub or truncate tokens in any error context before logging.

### 8. Test coverage gaps

- No test for `PaymentError` being raised by the payment client.
- No test for the DB transaction rolling back after a successful charge (the critical failure path from issue #1).
- No test verifying lock TTL expiry behavior or that lock release is still called when `payment.charge()` raises.
- The `test_lock_release_uses_lua_script` test asserts `redis_client.delete` is not called, which is correct, but does not assert the Lua script was called with the correct token — it only checks call count.

### 9. `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as class-level constants

These should be instance-level configuration (constructor parameters with defaults) so they can be tuned per deployment environment without subclassing.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct.
- Token-verified Lua release script is the right approach and correctly implemented.
- Pydantic validation with pattern enforcement on `order_id` is solid.
- Input validation rejecting path traversal is confirmed by test.
- Structured logging with contextual fields throughout.
- The `contextmanager` lock pattern ensures release in the `finally` block.
