Decision: Request Changes

Summary: The PR's lock TTL (60 s) is structurally incompatible with the worst-case duration of the work performed while the lock is held. Beyond that critical race condition, the payment call executes *inside* the database transaction, creating a distributed-atomicity violation that can produce phantom charges with no database record. Several secondary issues compound the risk.

---

## Critical Issues

### 1. Lock TTL vs. Worst-Case Execution Time — Duplicate Charge Race

This is the primary defect and the reason the PR's core safety guarantee does not hold.

**TTL budget:** `ORDER_LOCK_TTL = 60 s`

**Worst-case time consumed while the lock is held:**

| Step | Worst-case duration |
|---|---|
| Lock retry backoff (already spent *before* yield, but sets context) | — |
| `db.get_charge()` — slow replica or network hiccup | ~5 s |
| `payment.charge()` — `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES-1)` | `30×3 + 5×2 = 100 s` |
| `db.record_charge()` + `db.mark_order_complete()` | ~5 s |
| **Total worst-case** | **~110 s** |

At t=60 s the lock expires. Worker B acquires it, passes the idempotency check (the charge has not yet been recorded because Worker A's payment call is still in flight or the DB write hasn't committed), and issues a second charge. The customer is billed twice.

**Fix:** Either (a) reduce `REQUEST_TIMEOUT` and `MAX_RETRIES` so that `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES-1) < ORDER_LOCK_TTL` with meaningful headroom, or (b) implement lock extension (Redlock-style `PEXPIRE` refresh) during the payment call, or (c) move the idempotency record to a *pre-charge* "pending" state written inside the lock before the payment call begins, so any concurrent worker sees the in-progress marker and backs off.

The simplest safe configuration given the current constants would be:

```python
# Ensure: REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1) + DB_OVERHEAD < ORDER_LOCK_TTL
# e.g. timeout=5, retries=3, backoff=2 → 5*3 + 2*2 = 19 s, well under a 60 s TTL
REQUEST_TIMEOUT = 5
RETRY_BACKOFF = 2
```

But the right architectural fix is the pre-charge pending record described in (c).

---

### 2. Payment Call Inside the Database Transaction — Distributed Atomicity Violation

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network I/O inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is architecturally incorrect for two reasons:

**a) Phantom charge on DB failure.** If `self.payment.charge()` succeeds (money moves) but `record_charge()` or `mark_order_complete()` raises an exception, the DB transaction rolls back. The charge exists at the payment gateway with no local record. The next attempt will charge again because the idempotency check finds nothing.

**b) Long-held DB connection.** The database connection/transaction is held open for the entire duration of the HTTP call to the payment gateway (up to 30 s per attempt). Under any load this exhausts the connection pool.

**Fix:** Perform the payment call *outside and before* the transaction. Record a "pending" marker before charging, then confirm it after:

```python
# 1. Write a pending record inside a short transaction (idempotency anchor)
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# 2. Call the payment gateway outside any DB transaction
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Confirm the charge in a second short transaction
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

## Significant Issues

### 3. Timeout on Gateway Response Does Not Guarantee the Charge Did Not Execute

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A `Timeout` from `requests` means the *client* stopped waiting — it does not mean the gateway did not process the charge. Retrying after a timeout without a gateway-side idempotency key can result in multiple successful charges for a single logical request.

**Fix:** Pass a stable, per-order idempotency key in the request header:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # e.g. order_id
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge` method signature must accept and forward this key.

### 4. Lock Retry Backoff Consumed Before the Lock Is Held — Wasted TTL on Contention Path

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)  # 0.5 s + 1.0 s = 1.5 s before giving up
```

This is not a TTL-safety issue on its own, but the backoff sleep (up to 1.5 s total) occurs before the lock is acquired and therefore does not consume the TTL. This is noted as correct behavior, but the `LOCK_RETRY_ATTEMPTS = 3` with these backoffs means a worker can wait up to 1.5 s before raising `LockError`. Ensure this is acceptable for the latency SLA of the endpoint calling `process()`.

---

### 5. `PaymentError` Is Silently Swallowed in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` (and any other unexpected exception) is not caught in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which correctly triggers the `finally` clause and releases the lock. However, the caller receives an unhandled exception rather than a structured `OrderResult`. This is inconsistent with the method's contract and will produce 500-level errors at the API layer with no actionable message.

**Fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

### 6. Single-Node Redis Lock Is Not Fault-Tolerant

The implementation uses a single `redis.Redis` client. If the Redis node fails after `SET NX EX` returns `True` but before the lock is released, the lock is held until TTL expiry (acceptable). However, if Redis fails *during* `SET NX EX` (network partition between the `SET` and the `True` acknowledgment), the lock may or may not have been written, and the worker proceeds without a lock. For a government payment system, this risk profile should be explicitly acknowledged and either accepted with documented rationale or mitigated with Redlock across ≥3 nodes.

## Minor Issues

### 7. Test for Lock Release Verifies Mechanism, Not Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms the Lua script is called but does not verify that the correct `lock_key` and `token` are passed as arguments. A regression that passes the wrong key would not be caught.

### 8. No Test for Payment Timeout / Retry Exhaustion Path

There is no test verifying behavior when `payment.charge` raises `PaymentError` after retries. Given issue 5 above, this path is entirely untested and will surface as an unhandled exception in production.

### 9. `customer_id` Has No Pattern Constraint

`order_id` enforces `^[a-zA-Z0-9_\-]+$` but `customer_id` accepts any string up to 64 characters. If `customer_id` is used in database queries or log aggregation, this is an inconsistency worth closing.
