Decision: Request Changes

Summary: The pull request contains a critical race condition that directly contradicts its stated at-most-once guarantee: the worst-case duration of operations held under the lock substantially exceeds the lock TTL, meaning a concurrent worker can acquire the lock, pass the idempotency check, and issue a duplicate charge against a live payment gateway. Several secondary issues compound the risk. All must be resolved before this can be approved.

---

## Critical: Lock TTL vs. Worst-Case Held Duration

This is the primary rejection reason. Trace the full lifecycle carefully.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration of work performed while the lock is held:**

Inside `_acquire_lock`'s `yield`, `process()` executes the following sequentially:

1. `db.get_charge(req.order_id)` — database read, assume up to ~5 s under load
2. `self.payment.charge(...)` — called inside `with self.db.transaction():`

Now expand `payment.charge()` at worst case across all retry attempts:

| Attempt | `REQUEST_TIMEOUT` | `RETRY_BACKOFF` sleep (before next attempt) |
|---------|-------------------|----------------------------------------------|
| 1       | 30 s              | 5 s                                          |
| 2       | 30 s              | 5 s                                          |
| 3       | 30 s              | 0 s (last attempt, no sleep)                 |
| **Total** | **90 s**        | **+10 s = 100 s**                            |

3. `db.record_charge(...)` — database write, ~2 s
4. `db.mark_order_complete(...)` — database write, ~2 s

**Worst-case total: ~109 seconds.**

The lock TTL is 60 seconds. The lock expires roughly 49 seconds before Worker A finishes. Worker B then:

1. Successfully acquires the lock (`SET NX EX` succeeds — the key is gone)
2. Calls `db.get_charge()` — Worker A has not yet committed its transaction, so no record exists
3. Calls `payment.charge()` — **duplicate charge issued**

The idempotency check provides zero protection here because the charge record is only written *after* the payment succeeds, inside a transaction that has not yet committed when Worker B reads. This is a textbook time-of-check/time-of-use (TOCTOU) race.

**Required fix — choose one or combine:**

- **Extend the TTL** to safely exceed worst-case held duration. With 109 s worst case, a TTL of at least 180 s with margin is needed. This is a blunt instrument and delays detection of dead workers.
- **Reduce worst-case held duration** by moving `payment.charge()` *outside* the lock, using a two-phase approach: reserve the order under the lock (write a `PENDING` record), release the lock, charge, then finalize. The lock only needs to protect the reservation write.
- **Write an in-flight sentinel before charging.** Before calling `payment.charge()`, write a `PENDING` record to the database *within the transaction* so that any concurrent worker that acquires the lock after TTL expiry will see the in-flight state and abort. This is the most robust fix.
- **Use Redis lock extension (pexpire refresh).** Have a background heartbeat thread extend the lock TTL while the worker is still alive. This is complex and introduces its own failure modes but is the standard approach for long-held distributed locks.

## Secondary Issue: `payment.charge()` Is Called Inside `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call, up to 100 s
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Holding a database transaction open across an external HTTP call with up to 100 seconds of worst-case latency will:

- Exhaust the database connection pool under any meaningful concurrency
- Hold row-level locks for the duration, causing cascading contention
- Increase the probability of transaction timeout errors from the database itself

The external call must be moved outside the transaction. The correct pattern is:

```python
charge_id = self.payment.charge(...)   # outside transaction, outside or inside lock depending on chosen fix

with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

## Secondary Issue: Timeout on `payment.charge()` Does Not Guarantee the Charge Did Not Land

When `requests.exceptions.Timeout` is caught, the code retries. A timeout means the *client* stopped waiting — it does not mean the server did not process the request. The payment gateway may have successfully charged the customer, and the retry will charge them again.

The retry loop must pass a deterministic idempotency key to the payment gateway on every attempt for the same order:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": order_id},
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, retries on timeout are not safe for payment operations regardless of what the distributed lock does.

## Secondary Issue: Lock Retry Backoff Adds to Pre-Acquisition Delay

```python
LOCK_RETRY_ATTEMPTS = 3
# backoff: 0.5, 1.0, 1.5 seconds between attempts = up to 2.0 s before acquisition
```

This is minor in isolation, but it means the clock on the 60-second TTL starts up to ~2 seconds later than the caller expects, and this delay is not accounted for in any TTL calculation. Not a blocker on its own, but worth noting when recalculating margins after the TTL fix.

## Secondary Issue: Test Coverage Does Not Cover the Failure Modes That Matter

The tests mock `payment.charge` as an instant success. None of the following are tested:

- Payment timeout followed by retry — verifies the retry loop behaves correctly
- Lock expiry during a long payment call — this is the exact failure mode described above; it should be a regression test once fixed
- `PaymentError` raised from `payment.charge` — `process()` does not catch `PaymentError`, so it will propagate as an unhandled exception to the caller; this is likely unintentional and is untested
- Database transaction rollback when `record_charge` raises after a successful charge — the charge has landed but the record is not written; the idempotency check will not find it on retry

The last point is a separate at-most-once violation: if `record_charge` or `mark_order_complete` raises after `payment.charge` succeeds, the transaction rolls back, the lock is released, and the next call will charge again. This requires either compensating logic (void the charge on DB failure) or writing the charge record before calling the gateway (see the sentinel approach above).
