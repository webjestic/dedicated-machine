Decision: Request Changes

Summary: The implementation contains a critical lock-expiry race condition that directly violates the at-most-once payment guarantee the PR claims to provide. The worst-case duration of operations held under the lock can substantially exceed the 60-second TTL, meaning a second worker can acquire the lock, pass the idempotency check on an in-flight order, and issue a duplicate charge. Several secondary issues compound the risk and must be resolved before this code touches any payment path.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the core defect. Work through the math explicitly.

**TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration of work held under the lock:**

```
payment.charge() worst case
  = MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
  = 3 × 30s + 2 × 5s
  = 90s + 10s
  = 100 seconds
```

The payment call alone can run for **100 seconds** — 40 seconds past the TTL. The lock expires at second 60. A second worker then:

1. Acquires the lock (the key is gone).
2. Calls `db.get_charge()` — returns `None` because the first worker has not yet committed (it is still blocked in `payment.charge()` on attempt 2 or 3).
3. Calls `payment.charge()` — **duplicate charge issued**.

The first worker eventually succeeds, commits its record, and releases a lock it no longer holds (the Lua script correctly no-ops, but the damage is done). The idempotency check and the distributed lock both fail simultaneously, which is the worst possible outcome for a payment processor.

**Required fix:** The lock TTL must be strictly greater than the worst-case execution time of everything inside the `with self._acquire_lock(...)` block, or the payment call must be moved outside the lock with a two-phase commit pattern (reserve → charge → confirm). The simplest safe fix is:

```python
# Minimum safe TTL = MAX_RETRIES * REQUEST_TIMEOUT
#                  + (MAX_RETRIES - 1) * RETRY_BACKOFF
#                  + db operation headroom
# = 100s + ~10s headroom = 110s minimum
ORDER_LOCK_TTL = 120  # seconds — document the derivation explicitly
```

But raising the TTL alone is fragile; it couples two separate modules invisibly. The correct architectural fix is to make `PaymentClient` expose its worst-case bound and have `OrderProcessor` assert at construction time that `ORDER_LOCK_TTL > payment_client.worst_case_seconds()`. Even better: use a lock-refresh (watchdog) pattern or move to a saga/reservation model.

---

## High: Lock Retry Backoff Adds to Pre-Acquisition Delay

The retry loop in `_acquire_lock` sleeps before the lock is acquired, not while it is held, so it does not directly extend the held duration. However, it does mean the wall-clock time from `process()` entry to lock acquisition can be up to:

```
LOCK_RETRY_ATTEMPTS retries × max backoff = 3 × 1.5s = 4.5s
```

This is acceptable on its own, but it means the caller's timeout budget must account for it. More importantly, the retry loop has an off-by-one: on the final attempt (`attempt == LOCK_RETRY_ATTEMPTS - 1`), the code sleeps and then exits the loop without a final acquisition attempt. Trace it:

```python
for attempt in range(3):       # attempts 0, 1, 2
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)        # sleeps after attempt 2, then loop ends
```

After attempt 2 fails, the code sleeps 1.5 seconds and then falls through to `if not acquired: raise LockError`. That final sleep is wasted — the lock is not retried afterward. This is a minor efficiency bug but it means the effective retry count is `LOCK_RETRY_ATTEMPTS` attempts with `LOCK_RETRY_ATTEMPTS` sleeps, the last of which is pointless. Fix:

```python
if attempt < self.LOCK_RETRY_ATTEMPTS - 1:
    time.sleep(backoff)
```

## High: `payment.charge()` Is Called Inside a Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # network call, up to 100s
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A network call to an external payment gateway is inside a database transaction. This holds a database connection and any row-level locks for the entire duration of the payment call (up to 100 seconds in the worst case). Under any meaningful concurrency this will exhaust the connection pool and cause cascading failures across the entire service, not just this order. The charge call must be moved outside the transaction:

```python
# Outside the db transaction:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only fast, local operations inside the transaction:
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This introduces a new failure mode (charge succeeds, DB write fails), which must be handled: the idempotency record should be written first in a separate step, or the system must be able to reconcile against the payment gateway on startup.

## High: Timeout on Requests Is Connect+Read Combined, Not Per-Attempt

```python
response = requests.post(..., timeout=self.REQUEST_TIMEOUT)
```

`requests` interprets a scalar timeout as the socket read timeout, not a total request timeout. A slow server that sends headers immediately but streams the body slowly can hold the connection open far longer than 30 seconds. Use a tuple:

```python
timeout=(5, 30)  # (connect_timeout, read_timeout)
```

Also, `requests.exceptions.ConnectionError` is not caught. A DNS failure or refused connection raises `ConnectionError`, not `Timeout`, and will propagate as an unhandled exception out of `process()`, bypassing the `LockError` handler and surfacing a raw exception to the caller. Add it to the retry path.

---

## Medium: Idempotency Check Is Not Re-Verified After Charge

If `payment.charge()` succeeds but `self.db.transaction()` fails (deadlock, connection loss, etc.), the charge has been issued but not recorded. On retry, `db.get_charge()` returns `None` (nothing was committed), the idempotency check passes, and a second charge is issued. This is the classic "charge succeeded, record failed" problem.

The standard mitigation is to write a pending/in-flight record to the database *before* calling the payment gateway, then update it to complete after. That way a retry sees the in-flight record and can reconcile rather than re-charge.

---

## Medium: `PaymentError` Is Silently Swallowed

`payment.charge()` raises `PaymentError` on permanent failure or retry exhaustion. This exception is not caught anywhere in `process()`. It will propagate out of the `with self._acquire_lock(...)` block, which correctly triggers the `finally` clause and releases the lock, but then it surfaces as an unhandled exception to the caller. The PR claims structured error handling; this is a gap. Add:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

## Medium: No Test Covers the Lock-Expiry Race

The test suite does not test the scenario where the payment call outlasts the TTL. This is the most dangerous failure mode and it is entirely absent from coverage. A test should mock `payment.charge` with a delay (or simulate a slow response) and verify that a concurrent `process()` call on the same order does not result in two charges. Without this test the regression surface for the critical bug is zero.

---

## Low: `requests` Session Is Not Reused

`PaymentClient.charge()` creates a new `requests` session (via the module-level `requests.post`) on every call. For a payment service that may be called at high frequency, this means a new TCP connection and TLS handshake on every charge. Use a `requests.Session` stored on the instance, with appropriate connection pool sizing and retry configuration at the transport layer.

---

## Low: Logger Name Is Module-Level but Class Is Instantiated Multiple Times

`logger = logging.getLogger(__name__)` is fine, but if multiple `OrderProcessor` instances exist (e.g., different worker pools for different order types), log lines are indistinguishable. Consider passing a logger name or tagging log records with an instance identifier.
