Decision: Request Changes

Summary: The implementation contains a critical race condition where the worst-case execution time under the lock can substantially exceed the lock TTL, allowing a concurrent worker to acquire the lock, pass the idempotency check, and issue a duplicate charge. Beyond that primary flaw, there are several secondary issues around error handling, observability, and test coverage that must be addressed before this is safe to merge into a payment-critical path.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the blocking issue. Trace the full lifecycle of everything that executes while the lock is held.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration while the lock is held:**

```
PaymentClient.charge():
  - Per attempt: REQUEST_TIMEOUT = 30s
  - Between attempts: RETRY_BACKOFF = 5s
  - MAX_RETRIES = 3

  Worst case: (30s × 3 attempts) + (5s × 2 inter-attempt sleeps)
            = 90s + 10s
            = 100s
```

The payment call alone can take **100 seconds** in the worst case. The lock TTL is **60 seconds**. The lock expires 40 seconds before the payment call can finish.

The failure sequence is:

1. Worker A acquires the lock, starts `payment.charge()`.
2. At T=60s the lock expires. Worker B acquires the lock on the same `order_id`.
3. Worker B calls `db.get_charge()` — the charge has not been recorded yet because Worker A is still inside `payment.charge()` and has not reached `db.record_charge()`.
4. Worker B passes the idempotency check and calls `payment.charge()` itself.
5. Worker A eventually succeeds and records its charge. Worker B records a second charge.
6. The customer is billed twice.

The idempotency check is only meaningful if the lock is still held when it runs. Once the lock expires, it provides no protection.

**Required fix:** The lock TTL must be strictly greater than the worst-case execution time of everything inside the lock, with margin. Either:

- Increase `ORDER_LOCK_TTL` to at least `(MAX_RETRIES × REQUEST_TIMEOUT) + ((MAX_RETRIES - 1) × RETRY_BACKOFF) + db_margin`, e.g. 120–150s, **and** document this coupling explicitly so that changing either constant requires re-evaluating the other; or
- Implement lock extension (a background thread or async task that refreshes the TTL while the lock is held and the worker is still alive); or
- Move the payment call outside the lock and use a two-phase commit pattern: record a "pending charge" row before charging, then confirm or void it after. The idempotency check then detects the pending row and the lock window shrinks to only the database operations.

The third option is the most robust for a payment system because it survives worker crashes mid-charge without relying on TTL accuracy.

## High: Charge Issued Outside the Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)          # network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`payment.charge()` is a network call to an external service placed inside a database transaction. This is dangerous for two reasons:

1. **Connection exhaustion.** The database connection is held open for the full duration of the payment call (up to 100s worst case). Under any meaningful concurrency this will exhaust the connection pool.
2. **Uncommitted visibility.** If the payment succeeds but `record_charge` or `mark_order_complete` fail and the transaction rolls back, the charge exists in the payment gateway but not in the database. The next retry will pass the idempotency check (no record found) and charge again.

**Required fix:** Charge outside the transaction. Record the charge inside the transaction. Use the gateway-returned `charge_id` as the idempotency key. If the transaction fails after a successful charge, the retry must detect the existing charge via the gateway's own idempotency key mechanism, not just the local DB check.

---

## High: `payment.charge()` Is Not Idempotent on Retry

`PaymentClient.charge()` retries on timeout and 5xx errors with no idempotency key sent to the gateway. A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without an idempotency key will issue a second charge.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # no idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

**Required fix:** Generate a stable idempotency key (e.g., derived from `order_id`) and pass it as a header or body field on every attempt. The gateway deduplicates on its side. This is standard practice for payment APIs.

```python
json={
    "token": token,
    "amount": amount_cents,
    "idempotency_key": f"order-{order_id}",  # passed through from caller
}
```

`order_id` must be threaded into `PaymentClient.charge()` for this to work.

## Medium: Silent Lock Release Failure

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

The return value of the Lua script is discarded. A return value of `0` means the lock was not held by this worker at release time — i.e., the TTL expired and another worker acquired it. This is exactly the race condition described above, and it is happening silently. The log line says "released" regardless of whether the release succeeded.

**Required fix:**

```python
finally:
    released = self._release_script(keys=[lock_key], args=[token])
    if not released:
        logger.error(
            "Lock release failed — TTL may have expired while lock was held "
            "order_id=%s. Possible duplicate charge. Investigate immediately.",
            order_id,
        )
    else:
        logger.info("Order lock released order_id=%s", order_id)
```

This also serves as a runtime signal that the TTL is misconfigured.

## Medium: Lock Retry Backoff Adds to Worst-Case Time

The lock acquisition retry loop is separate from the TTL issue but worth noting:

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    ...
    backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s
    time.sleep(backoff)
```

Worst-case acquisition time: `0.5 + 1.0 = 1.5s`. This is small relative to the TTL issue but should be included in any TTL budget calculation.

## Medium: `PaymentError` Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentClient.charge()` raises `PaymentError` on permanent failure or after exhausting retries. This exception is not caught in `process()`. It will propagate up to the caller as an unhandled exception rather than returning a structured `OrderResult`. The PR checklist claims structured error handling but this path is missing.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

Database exceptions from `db.get_charge()`, `db.record_charge()`, and `db.mark_order_complete()` are similarly unhandled.

## Low: Test Coverage Gaps

The tests mock away the components with the most dangerous failure modes and do not test the scenarios that matter most for a payment system.

| Missing test | Why it matters |
|---|---|
| `payment.charge()` raises `PaymentError` | Verifies structured error return, not unhandled exception |
| `payment.charge()` succeeds but `db.record_charge()` raises | Verifies the charge-without-record scenario is detected |
| Lock release returns `0` (expired TTL) | Verifies the error log fires; currently the return value is ignored |
| `payment.charge()` called with an idempotency key | Once the fix is in, this must be enforced |
| Lock acquired on second attempt after first fails | The retry loop is untested |

The existing `test_lock_release_uses_lua_script` test only asserts that `redis.delete` was not called and that the script was called once. It does not assert the token value passed to the script, so it would pass even if the wrong token were used.

---

## Low: Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoding a production endpoint in source is a deployment hazard. This should be injected via configuration or environment variable, both for environment parity (dev/staging/prod) and to avoid accidental live charges during testing.
