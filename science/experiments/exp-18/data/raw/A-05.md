Decision: Request Changes

Summary: The critical flaw is a lock TTL exhaustion vulnerability: the Redis lock is granted a 60-second TTL, but the worst-case duration of work performed while holding that lock can exceed 60 seconds, allowing a second worker to acquire the lock, pass the idempotency check on an in-flight charge, and issue a duplicate charge. Additionally, the payment gateway call is made *inside* the database transaction, creating a distributed transaction hazard where a successful charge can be orphaned by a database rollback. These two issues together undermine the at-most-once guarantee the PR claims to provide.

---

## Detailed Findings

### 1. Lock TTL Exhaustion — Duplicate Charge Risk (Critical)

**The math the PR did not do.**

While the lock is held, the following operations execute sequentially:

| Operation | Worst-case duration |
|---|---|
| `db.get_charge()` (idempotency read) | network latency, assume ~1 s |
| `payment.charge()` — attempt 1 timeout | `REQUEST_TIMEOUT = 30 s` |
| backoff between attempts 1→2 | `RETRY_BACKOFF = 5 s` |
| `payment.charge()` — attempt 2 timeout | 30 s |
| backoff between attempts 2→3 | 5 s |
| `payment.charge()` — attempt 3 timeout | 30 s |
| `db.record_charge()` + `db.mark_order_complete()` | ~1 s |
| **Total** | **~102 s** |

`ORDER_LOCK_TTL = 60 s`.

At T=60 s the lock expires in Redis. A second worker can now acquire the lock, call `db.get_charge()`, find nothing recorded (because Worker 1 is still on its second or third payment attempt and has not yet committed), and proceed to charge the same payment token. Both workers can now hold an in-flight charge simultaneously. This is the exact double-charge scenario the lock is supposed to prevent.

**Fix:** The lock TTL must be strictly greater than the worst-case critical-section duration. Given the current retry parameters:

```
worst_case = MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF
           = 3 * 30 + 2 * 5 = 100 s   (payment alone)
```

Either raise `ORDER_LOCK_TTL` well above 102 s with margin, or — the better architectural choice — move the payment call *outside* the lock's critical section and use a two-phase approach: record *intent* to charge under the lock, release the lock, charge, then record the result. The idempotency check then gates on the intent record, not the completed charge.

A lock-extension (watchdog/heartbeat) pattern is another valid mitigation: a background thread periodically calls `PEXPIRE` on the lock key while the worker is still alive, and stops extending if the worker crashes, allowing natural TTL expiry.

### 2. Payment Call Inside Database Transaction — Orphaned Charge Risk (Critical)

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side-effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`payment.charge()` is a non-transactional external call made inside a database transaction. If the payment gateway succeeds but either `record_charge` or `mark_order_complete` raises an exception, the database transaction rolls back. The charge has been collected from the customer; there is no record of it anywhere in the system. This is an orphaned charge: money is taken, the order shows as unprocessed, and a retry will attempt to charge again.

**Fix:** Charge first, outside the transaction. Record the result inside the transaction. On database failure after a successful charge, the retry path must detect the existing charge via the payment gateway's own idempotency key before re-charging.

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-level dedup
)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 3. No Idempotency Key Passed to Payment Gateway (High)

`payment.charge()` is called with only `token` and `amount_cents`. There is no `idempotency_key` (e.g., `order_id`) sent to the gateway. If the request times out and is retried, the gateway has no way to deduplicate it and will issue a second charge. This is independent of the Redis lock: it is a gateway-level duplicate charge on retries of the *same* worker.

```python
# Current — no idempotency key
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

**Fix:** Pass `order_id` as an idempotency key in the request header or body, per the gateway's API contract. This is standard practice for any payment gateway that supports it (Stripe `Idempotency-Key`, etc.).

---

### 4. Timeout Response Retried Without Charge-State Verification (High)

When `payment.charge()` catches a `requests.exceptions.Timeout`, it logs and retries immediately. A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without first querying the gateway for the status of the previous attempt (using the idempotency key from finding 3) can result in a double charge even within a single worker's retry loop.

---

### 5. Lock Retry Backoff Adds to Critical-Section Duration (Medium)

The lock acquisition retry loop sleeps *before* acquiring the lock:

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)
```

This backoff (up to 1.5 s total) happens outside the lock and is not a safety issue by itself, but it means a worker can wait up to 1.5 s before giving up, during which it holds no lock. This is acceptable, but the `LOCK_RETRY_ATTEMPTS = 3` with these delays means a caller can block for 1.5 s before receiving a contention error. Document this as an SLA consideration.

---

### 6. Test Suite Does Not Cover the TTL Exhaustion Scenario (Medium)

No test simulates a slow payment gateway causing lock expiry and a concurrent worker acquiring the lock. The test for lock contention (`test_process_lock_contention`) only tests the case where `redis.set` returns `False` on the first call — it does not test the race between lock expiry and an in-flight charge. These tests pass while the critical vulnerability is present.

A meaningful test would mock the payment client to sleep longer than `ORDER_LOCK_TTL`, then assert that a second processor instance cannot acquire the lock and charge during that window — or, after the fix is applied, assert that the watchdog correctly extends the TTL.

---

### 7. `PaymentError` Not Caught in `process()` (Low)

If `payment.charge()` raises `PaymentError` after exhausting retries, the exception propagates out of `process()` uncaught. The caller receives an unhandled exception rather than a structured `OrderResult`. The lock is still released correctly via `finally`, but the API contract of `process()` — which returns `OrderResult` in all other failure cases — is violated.

```python
# Add to process()
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```
