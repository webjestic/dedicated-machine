Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical and deterministic lock-expiry race condition: the worst-case duration of operations executed while holding the lock can exceed the 60-second TTL, meaning a second worker will acquire the lock, pass the idempotency check on the in-flight order, and issue a duplicate charge. Several secondary issues compound the risk. No changes should be merged until the timing flaw is resolved.

---

## Critical Issue — Lock TTL vs. Worst-Case Held Duration

This is the primary blocker. The math is straightforward.

### Worst-case time the lock is held

```
payment_client.charge() worst-case duration
  = (REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES − 1))
  = (30s × 3)  + (5s × 2)
  = 90s + 10s
  = 100 seconds
```

The lock TTL is `ORDER_LOCK_TTL = 60` seconds.

A second worker can acquire the lock at `t = 60s`. At that moment, Worker 1 is still inside `payment_client.charge()` on attempt 2 or 3. The idempotency check (`db.get_charge`) has not yet recorded anything because `record_charge` is called only after `charge()` returns successfully. Worker 2 passes the idempotency check and calls `charge()` itself. Both workers complete successfully and the customer is billed twice.

```
t=0s   Worker 1 acquires lock, passes idempotency check, calls charge() attempt 1
t=30s  Worker 1 timeout → sleeps 5s
t=35s  Worker 1 calls charge() attempt 2
t=60s  Lock TTL expires ← Worker 2 acquires lock
t=60s  Worker 2 passes idempotency check (nothing recorded yet), calls charge()
t=65s  Worker 1 timeout → sleeps 5s
t=65s  Worker 2 charge() succeeds → records charge → releases lock  ← DUPLICATE CHARGE 1
t=70s  Worker 1 calls charge() attempt 3
t=100s Worker 1 charge() succeeds → records charge                  ← DUPLICATE CHARGE 2
```

### Required fix

The TTL must be strictly greater than the worst-case held duration, with margin. Additionally, the worker must detect lock expiry before performing irreversible operations.

```python
# In OrderProcessor
ORDER_LOCK_TTL = 180          # > worst-case held duration with margin

# In PaymentClient — tighten so the math closes
REQUEST_TIMEOUT = 15          # seconds per attempt
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5

# Worst-case with tightened values:
# (15 × 3) + (5 × 2) = 45 + 10 = 55s  <  180s TTL  ✓
```

TTL inflation alone is necessary but not sufficient. The worker must also verify it still holds the lock before calling `charge()`, because a GC pause or network partition can still cause silent expiry mid-execution.

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    acquired = False

    for attempt in range(self.LOCK_RETRY_ATTEMPTS):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
        if acquired:
            break
        backoff = 0.5 * (attempt + 1)
        time.sleep(backoff)

    if not acquired:
        raise LockError(f"Could not acquire lock for order {order_id}")

    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])

def _assert_lock_held(self, lock_key: str, token: str) -> None:
    """Raise LockError if this worker no longer holds the lock."""
    current = self.redis.get(lock_key)
    if current is None:
        raise LockError("Lock expired before irreversible operation")
    if current.decode() != token:
        raise LockError("Lock was acquired by another worker")
```

```python
# In process(), before calling charge():
lock_key = f"order:lock:{req.order_id}"
self._assert_lock_held(lock_key, token)   # guard before irreversible step
charge_id = self.payment.charge(...)
```

Note that `_assert_lock_held` reduces but does not eliminate the TOCTOU window — a pause between the check and the charge call can still cause expiry. The TTL fix is the primary control; the pre-charge check is a defense-in-depth layer.

## Secondary Issues

### 1. `payment.charge()` is called inside `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # network I/O inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction holds a connection and potentially row-level locks for its entire duration. Placing a retrying HTTP call (up to 100 seconds worst-case) inside the transaction will exhaust the connection pool under any meaningful concurrency and can cause cascading failures across unrelated orders.

**Required fix:** Charge outside the transaction, then open the transaction only for the database writes.

```python
# Charge outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB writes inside the transaction
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

### 2. `PaymentError` is silently swallowed

`payment.charge()` raises `PaymentError` on permanent failure or retry exhaustion. There is no `except PaymentError` block in `process()`. The exception will propagate out of the `with self.db.transaction()` context manager. Whether the transaction is rolled back depends entirely on the `db` implementation — it is not guaranteed. The caller receives an unhandled exception rather than a structured `OrderResult`.

**Required fix:**

```python
try:
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment failed",
    )
```

### 3. Timeout on `requests.get` response body is not bounded

`requests.post(..., timeout=30)` sets the connection and read timeout to 30 seconds per the `requests` library contract, but only for the initial response headers. If the gateway streams a response body slowly, the call can block indefinitely beyond the 30-second window. Use a tuple timeout.

```python
timeout=(10, 30)   # (connect_timeout, read_timeout)
```

### 4. Lock retry backoff adds unaccounted time to the held duration

The lock acquisition retry loop in `_acquire_lock` sleeps up to `0.5 + 1.0 = 1.5` seconds before the lock is acquired. This time is outside the lock and is not a problem. However, if the intent is ever to retry inside the lock, that time must be included in the TTL budget calculation.

### 5. Test coverage gaps

The test suite does not cover:

- Lock expiry mid-execution (the primary failure mode identified above).
- `PaymentError` propagation from `payment.charge()`.
- Retry behavior in `PaymentClient` (timeout path, 5xx path, backoff timing).
- The pre-charge lock-held assertion once that guard is added.

These are not merge blockers on their own, but they should be addressed alongside the fixes above.
