Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical architectural flaw that breaks the at-most-once payment guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issue

### Payment charge occurs outside the database transaction — atomicity guarantee is false

This is the most serious bug. The PR summary states:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But the code does this:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)        # External HTTP call — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction wraps only database operations. `self.payment.charge()` is an external HTTP call to a payment gateway. It is not part of the database transaction and cannot be rolled back if the subsequent `record_charge` or `mark_order_complete` calls fail. The real failure modes are:

1. `payment.charge()` succeeds → customer is charged
2. `record_charge()` raises an exception → DB transaction rolls back
3. The charge is now unrecorded. The idempotency check (`get_charge`) will find nothing on retry, and the customer will be charged again.

The lock does not protect against this because the lock is released after the `with` block exits, and the failure happens inside it. This is a textbook example of the dual-write problem.

**Required fix:** The payment call must happen *before* the database transaction, and the charge ID must be persisted in a way that survives a crash between charge and record. A standard pattern is:

```python
# 1. First, check idempotency (already done — good)
# 2. Call payment gateway OUTSIDE the transaction, capture charge_id
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Persist within a transaction. If this fails, you have a charge_id to reconcile.
try:
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.error(
        "RECONCILIATION REQUIRED: charge succeeded but record failed "
        "order_id=%s charge_id=%s", req.order_id, charge_id
    )
    raise
```

This does not fully eliminate the problem — nothing can make an HTTP call and a DB write truly atomic — but it makes the failure mode recoverable via a reconciliation process rather than silently causing double charges.

## High Priority Issues

### Unhandled `PaymentError` propagates through the lock context manager silently

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # Can raise PaymentError
```

`PaymentError` is not caught anywhere in `process()`. It will propagate up through `_acquire_lock`, which correctly releases the lock in `finally`, but the caller receives an unhandled exception with no `OrderResult`. The contract of `process()` appears to be that it always returns an `OrderResult`. This should be caught explicitly:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, str(e))
    return OrderResult(processed=False, message="Payment processing failed")
```

### `payment_token` logged indirectly via `payment_client.py`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine on its own, but `payment_client.py` logs `e.response.status_code` on decline. Depending on how the gateway formats error responses, sensitive card data could appear in exception messages. The `PaymentError` raised on non-5xx errors includes `e.response.status_code` only, which is acceptable, but this should be explicitly reviewed and the team should confirm that `requests` exception messages never include request body content in this gateway's error responses.

### Redis connection failures are not handled

If `self.redis.set()` raises a `redis.RedisError` (connection refused, timeout, etc.), it propagates uncaught through `process()`. For a payment processor, a Redis outage should produce a clear, structured failure response, not an unhandled exception. Add a `redis.RedisError` catch around the lock acquisition.

## Secondary Issues

### `RETRY_BACKOFF = 5` seconds is a fixed sleep, not a backoff

The variable is named `RETRY_BACKOFF` and the docstring says "retry with backoff," but the implementation sleeps a flat 5 seconds on every attempt. Either rename it `RETRY_DELAY` and update the docstring, or implement actual exponential backoff: `time.sleep(self.RETRY_BACKOFF * (2 ** attempt))`.

### Lock TTL of 60 seconds may be too short given payment retry logic

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can retry up to 3 times with a 5-second sleep between attempts and a 30-second timeout per attempt. Worst case: `3 × 30s timeout + 2 × 5s sleep = 100 seconds`. The Redis lock will expire before the payment client finishes, allowing a second worker to acquire the lock and attempt a duplicate charge. The TTL must exceed the maximum possible duration of the payment operation. Set it to at least 120–150 seconds, or make it configurable.

### Test coverage gaps

The tests do not cover:

- `PaymentError` raised by `payment.charge()` — currently this would cause an unhandled exception, which the tests would catch as a failure, but there is no explicit test asserting the correct behavior
- Redis connection failure during lock acquisition
- The scenario where `record_charge` fails after a successful charge (the reconciliation path)
- Retry behavior in `PaymentClient` (no tests exist for `payment_client.py` at all)

The `test_lock_release_uses_lua_script` test asserts `processor._release_script.assert_called_once()` but `_release_script` is set in `__init__` from `register_script`'s return value. The mock setup correctly captures this, but the test is asserting on the mock fixture's return value, not on the actual script object held by the processor instance. Verify this test actually fails when the Lua script is bypassed.

## Minor Notes

- `customer_id` has no pattern validation, only length. If it feeds into any query or log aggregation, consider adding a pattern constraint consistent with `order_id`.
- `CHARGE_ENDPOINT` is hardcoded as a class variable. This should be injected via configuration or environment variable to support environment-specific deployments without code changes.
- The `OrderResult` fields `order_id` and `charge_id` are `Optional` but are omitted on failure paths. This is fine, but consider whether downstream consumers expect these fields to always be present and default to a sentinel value instead.
