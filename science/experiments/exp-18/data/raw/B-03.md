Decision: Request Changes

Summary: The distributed lock TTL is 60 seconds, but the worst-case duration of operations held under that lock can reach approximately 135 seconds — more than double the TTL — meaning a second worker will acquire the lock, pass the idempotency check on an in-flight order, and issue a duplicate charge. Beyond the TTL race, the payment call runs inside the database transaction, creating a long-held transaction that couples network latency to database connection exhaustion. Several secondary issues compound the risk.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the blocking issue. Trace the full timeline of work performed while the lock is held:

```
ORDER_LOCK_TTL = 60 seconds

PaymentClient.charge() worst-case:
  Attempt 1: REQUEST_TIMEOUT (30s) → Timeout exception
  Sleep:      RETRY_BACKOFF (5s)
  Attempt 2: REQUEST_TIMEOUT (30s) → Timeout exception
  Sleep:      RETRY_BACKOFF (5s)
  Attempt 3: REQUEST_TIMEOUT (30s) → Timeout exception
  ─────────────────────────────────────────────────────
  Total:      30 + 5 + 30 + 5 + 30 = 100 seconds

db.get_charge():         ~1s (network + query)
db.transaction() open:   ~1s
db.record_charge():      ~1s
db.mark_order_complete(): ~1s
─────────────────────────────────────────────────────
Estimated total:         ~104 seconds (conservative)
```

**104 seconds > 60-second TTL.**

At T=60 the Redis key expires. Worker B calls `SET NX EX` and acquires the lock. Worker B then calls `db.get_charge()` — but Worker A has not yet called `db.record_charge()` because it is still blocked on a payment gateway timeout. Worker B's idempotency check returns `None`. Worker B issues a second charge. Worker A eventually succeeds and also records a charge. The customer is billed twice.

The Lua release script does not help here. It prevents Worker A from releasing Worker B's lock, but it does not prevent Worker B from acquiring the lock while Worker A is still executing inside it.

**Required fix:** The TTL must be strictly greater than the worst-case execution time of everything inside the lock, or the payment retry budget must be reduced to fit inside the TTL with margin. One concrete option:

```python
# Worst-case payment time: 3 attempts × 30s timeout + 2 × 5s backoff = 100s
# Add DB overhead and safety margin: +20s
ORDER_LOCK_TTL = 120  # seconds — must exceed worst-case operation time

# Or reduce payment budget to fit:
REQUEST_TIMEOUT = 10   # 3×10 + 2×5 = 40s → fits in 60s TTL with margin
MAX_RETRIES = 2        # 2×10 + 1×5 = 25s → safer
```

Pick one approach and document the derivation in a constant or comment so future changes to either value trigger a conscious review of the other.

## Critical: Payment Call Inside Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # up to 100s network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction holds a connection (and potentially row locks) for the entire duration of the payment call. Under load this will exhaust the connection pool and cause cascading failures across the service. The payment call must be moved outside the transaction:

```python
# Correct structure:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only fast DB writes inside the transaction:
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This also reduces the window during which a crash leaves the system in an inconsistent state.

---

## High: Charge-Then-Record Leaves a Silent Inconsistency Window

Even with the payment call moved outside the transaction, there is a window between a successful `payment.charge()` and the `db.transaction()` commit where a process crash produces a charge with no database record. This is a known hard problem in distributed payments, but the PR claims an "at-most-once payment guarantee" without addressing it. The guarantee is not met.

The standard mitigations are:

1. **Idempotency key on the payment gateway call** — pass `order_id` as the idempotency key so a retry of the same order returns the same `charge_id` rather than issuing a new charge.
2. **Pending-charge record before charging** — write a `PENDING` record to the database before calling the gateway. On restart, scan for pending records and reconcile against the gateway.

At minimum the PR description should not claim at-most-once semantics it cannot currently deliver.

## Medium: `PaymentError` Is Not Caught in `process()`

```python
# In process():
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError
        ...
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller receives a raw exception instead of an `OrderResult`. This is inconsistent with the method's contract and will cause `500` errors at the API layer. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

Similarly, `db.get_charge()`, `db.record_charge()`, and `db.mark_order_complete()` can raise, and none of those exceptions are caught.

---

## Medium: Lock Retry Backoff Delays Compound Inside the Lock Acquisition Phase

The lock retry loop in `_acquire_lock` sleeps before the lock is acquired, so this delay does not consume TTL. That part is fine. However, the retry loop has a subtle issue: with `LOCK_RETRY_ATTEMPTS = 3` and backoffs of 0.5s, 1.0s, the total wait before raising `LockError` is 1.5 seconds. This is acceptable, but the values are not documented relative to any expected lock hold time, making future tuning blind.

---

## Low: `requests.exceptions.ConnectionError` Is Not Retried

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    if e.response.status_code >= 500:
        ...
```

`ConnectionError` (DNS failure, TCP reset, connection refused) is not caught and will propagate immediately as an unhandled exception rather than being retried or converted to `PaymentError`. Use `requests.exceptions.RequestException` as a base catch for transient network errors, or enumerate `ConnectionError` explicitly.

---

## Low: Test Coverage Gaps

The tests mock away the exact behavior that matters most for correctness:

- No test verifies that `PaymentError` returns an `OrderResult` rather than raising (because the exception is currently unhandled, a test for this would fail, which is itself evidence of the bug).
- No test exercises the TTL/timing relationship.
- No test verifies that a crash between `payment.charge()` succeeding and `db.record_charge()` committing is detectable or recoverable.
- `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` but does not assert that the script was called with the correct token, so it would pass even if the release were a no-op.
