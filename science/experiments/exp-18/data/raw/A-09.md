Decision: Request Changes

Summary: The lock TTL is 60 seconds, but the worst-case duration of work performed while the lock is held can far exceed that ceiling, creating a window where a second worker acquires the lock, passes the idempotency check on an in-flight order, and issues a duplicate charge. Beyond the TTL race, the payment call executes inside a database transaction, which holds database-level locks for the entire duration of potentially multiple HTTP round-trips, and a gateway timeout during a retry cycle is mishandled in a way that can produce an unrecorded charge. These issues are detailed below.

---

## Critical: Lock TTL vs. Worst-Case Execution Time

This is the primary rejection reason.

### The math

```
ORDER_LOCK_TTL = 60 seconds

Worst-case execution while the lock is held:
  payment_client.charge() — up to MAX_RETRIES attempts
    Per attempt:
      REQUEST_TIMEOUT          = 30 s   (full timeout, not connect+read split)
      RETRY_BACKOFF sleep      =  5 s   (between attempts)

    Attempt 1: 30 s timeout + 5 s backoff  = 35 s
    Attempt 2: 30 s timeout + 5 s backoff  = 35 s
    Attempt 3: 30 s timeout (no backoff)   = 30 s
                                    Total  = 100 s

Lock TTL                                   =  60 s
Overflow                                   = +40 s  ← lock expires while worker 1 is still running
```

At second 60, the lock key vanishes from Redis. Worker 2 acquires the lock, calls `db.get_charge()`, finds nothing recorded (worker 1 has not committed yet), and proceeds to charge the same payment token. Both charges succeed. Worker 1 eventually commits; worker 2 commits. The customer is billed twice.

### Fix options (choose one or combine)

**Option A — extend the TTL to safely exceed worst-case duration**

```python
# Worst case = MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + DB overhead + margin
ORDER_LOCK_TTL = 180  # seconds — covers 100 s payment + DB time + safety margin
```

**Option B — enforce a hard deadline and abort before the TTL expires**

```python
import signal

LOCK_SAFETY_MARGIN = 10  # seconds before TTL expiry to abort

@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    acquired = False

    for attempt in range(self.LOCK_RETRY_ATTEMPTS):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
        if acquired:
            break
        time.sleep(0.5 * (attempt + 1))

    if not acquired:
        raise LockError(f"Could not acquire lock for order {order_id}")

    deadline = time.monotonic() + self.ORDER_LOCK_TTL - LOCK_SAFETY_MARGIN

    try:
        yield token, deadline  # caller checks deadline before each expensive step
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

**Option C — cap payment client timeouts so they fit inside the TTL**

```python
# If TTL = 60 and you need 10 s margin + DB time (~5 s):
# Budget per attempt ≈ (60 - 15) / 3 = 15 s
REQUEST_TIMEOUT = (5, 10)   # (connect_timeout, read_timeout) — never use a single int
MAX_RETRIES     = 3
RETRY_BACKOFF   = 2
# Worst case: 3 * (10 + 2) = 36 s — fits inside 60 s TTL with margin
```

All three options should be applied together for defense in depth.

## Critical: Payment Call Inside a Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← up to 100 s HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The database transaction is opened before the HTTP call and held open for its entire duration, including all retry attempts and backoff sleeps. This:

1. Holds row-level or table-level locks in the database for up to 100 seconds, blocking every other query that touches those rows.
2. Keeps a database connection checked out from the pool for the same duration, exhausting the pool under moderate concurrency.
3. On most databases, a 100-second open transaction will be killed by the server's idle-in-transaction timeout, causing the commit to fail silently after a successful charge — the charge goes unrecorded.

**Fix:** Charge first, then open the transaction only for the database writes.

```python
# Outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB writes inside the transaction — milliseconds, not seconds
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This requires the payment client to be idempotent on retry (it should be regardless), and the charge recording must handle the case where the charge succeeded but the DB write failed (see the unrecorded charge issue below).

## Critical: Unrecorded Charge on Gateway Timeout

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# loop continues to next attempt
```

A `Timeout` exception from `requests` fires when the client-side timer expires. It says nothing about whether the server processed the request. The gateway may have received the charge request, authorized it, and sent a response that arrived after the timeout. The client discards that response and retries, potentially issuing a second authorization on the same payment token.

**Fix:** The payment gateway must support idempotency keys, and every retry must reuse the same key for the same logical charge.

```python
def charge(self, token: str, amount_cents: int) -> str:
    idempotency_key = str(uuid.uuid4())  # generated once per logical charge, passed by caller

    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=(5, 10),
        )
```

The `idempotency_key` must be stable across retries for the same order. The simplest approach is to derive it from `order_id` or pass it in from `OrderProcessor`.

## Serious: Single Integer Timeout Hides Hung Connections

```python
REQUEST_TIMEOUT = 30
requests.post(..., timeout=self.REQUEST_TIMEOUT)
```

Passing a single integer to `requests` sets only the read timeout. The connect phase is unlimited. A TCP connection to a degraded gateway can hang in SYN-SENT indefinitely, holding the thread and the lock forever — the TTL does not help because the thread is blocked in the kernel.

**Fix:**

```python
REQUEST_TIMEOUT = (5, 25)  # (connect_timeout_s, read_timeout_s)
```

---

## Serious: Lock Retry Backoff Adds to Lock-Held Duration Accounting

The lock acquisition retry loop sleeps up to `0.5 + 1.0 + 1.5 = 3 seconds` before the lock is acquired. This is not inside the lock and does not consume TTL, which is correct. However, the comment and checklist describe this as "retry with backoff on transient payment gateway failures" — that description belongs to `PaymentClient`. The lock retry backoff is separate and should be documented clearly to avoid confusion during future TTL budget calculations.

## Moderate: `PaymentError` Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # can raise PaymentError
    ...
```

`PaymentError` is not caught anywhere in `process()`. It will propagate up to the caller as an unhandled exception rather than returning a structured `OrderResult`. The checklist claims structured error handling, but this case is missing.

**Fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

## Moderate: Lock Retry Attempts Are Too Few Given Backoff

```python
LOCK_RETRY_ATTEMPTS = 3
# sleeps: 0.5 s, 1.0 s — total wait before giving up: 1.5 s
```

If a legitimate worker holds the lock for up to 100 seconds (current worst case), a competing worker gives up after 1.5 seconds and returns "Order currently being processed by another worker" to the caller. The caller has no way to distinguish "try again in 90 seconds" from "this order is permanently unprocessable." The retry count and backoff are mismatched with the actual lock duration.

---

## Minor: Test Coverage Gaps

The tests do not cover:

1. `PaymentError` raised by `payment.charge()` — no test verifies the result or that no charge is recorded.
2. Timeout scenario where the gateway times out on attempt 1 but succeeds on attempt 2.
3. Lock expiry race — no test simulates the lock expiring mid-execution.
4. `db.transaction().__exit__` raising an exception after a successful charge (unrecorded charge path).
