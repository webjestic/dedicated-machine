Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical race condition between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, and several secondary issues need to be addressed before this is safe to merge.

---

## Critical Issues

### 1. Payment charge happens inside the DB transaction — charge can succeed but not be recorded

This is the most serious bug. The current flow is:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `self.payment.charge()` succeeds but the DB transaction rolls back (network blip, constraint violation, DB crash), the customer has been charged and there is **no record of it**. The next worker will pass the idempotency check (`get_charge` returns `None`) and charge again. This is a double-charge scenario.

The correct pattern is to separate the charge from the DB write and handle the partial-success case explicitly:

```python
# 1. Charge outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record durably — if this fails, you have a charge_id to reconcile
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "CHARGE_RECORDED_FAILURE order_id=%s charge_id=%s — "
        "manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

Even with this fix, you need a reconciliation/recovery path for the case where the DB write fails after a successful charge. That is an operational requirement, not just a code requirement.

### 2. Redis lock TTL is too short relative to payment retry window

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire while payment is still in flight. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge concurrently.

Either increase the TTL to safely exceed the maximum payment duration (e.g., 180s with margin), or implement lock extension (heartbeat). At minimum, document the constraint and enforce it with a constant:

```python
MAX_PAYMENT_DURATION = PaymentClient.MAX_RETRIES * (
    PaymentClient.REQUEST_TIMEOUT + PaymentClient.RETRY_BACKOFF
)
ORDER_LOCK_TTL = MAX_PAYMENT_DURATION + 30  # 30s safety margin
```

## Significant Issues

### 3. Unhandled `PaymentError` propagates out of `process()`

`PaymentError` is not caught in `process()`. It will propagate as an unhandled exception to the caller. The PR claims structured error returns via `OrderResult`, but this path breaks that contract. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

### 4. `payment_token` is logged at INFO level via `charge_id` path — verify no token leakage

`payment_token` is stored in `req.payment_token` and passed to `payment.charge()`. Confirm that `PaymentClient.charge()` never logs the token value. Currently it does not, but this is worth a comment or an explicit assertion in tests given the sensitivity of the field. Consider masking it in `OrderRequest.__repr__` as a defense-in-depth measure.

### 5. `customer_id` has no pattern validation

`order_id` correctly enforces `r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. If `customer_id` is used in DB queries or log output downstream, this is an inconsistency worth closing.

---

## Minor Issues

### 6. Lock retry backoff is linear, not exponential

The PR checklist says "retry with backoff" but the implementation is linear (`0.5 * (attempt + 1)` → 0.5s, 1.0s, 1.5s). This is fine functionally but the checklist language implies exponential. Either correct the checklist or use true exponential backoff (`0.5 * 2**attempt`) to reduce thundering herd under high contention.

### 7. `PaymentClient` has no circuit breaker or per-instance state

`PaymentClient` is a plain class with no connection pooling or session reuse. For a service making repeated outbound calls, a `requests.Session` should be used. This is a performance and reliability concern, not a correctness issue, but worth addressing before production load.

### 8. Test coverage gaps

- No test for `PaymentError` propagation from `process()` (which is currently unhandled — see issue 3).
- No test for lock TTL expiry during payment (the race in issue 2).
- No test for DB transaction rollback after a successful charge (the race in issue 1).
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, which is correct, but does not assert the Lua script was called with the correct token — the token is internal to `_acquire_lock`, so this test would pass even if the script call was removed entirely given the mock setup.
