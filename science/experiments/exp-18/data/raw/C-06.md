Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — at-most-once guarantee is broken

This is the most serious bug. The PR summary explicitly claims:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But the code does not do that. `self.payment.charge(...)` is a network call to an external payment gateway. It cannot participate in a database transaction. What the code actually does is call the payment gateway, get a `charge_id` back, and *then* record it in the DB — all wrapped in a `db.transaction()` block that provides no atomicity guarantee over the external call.

```python
# order_processor.py lines inside with self.db.transaction():
charge_id = self.payment.charge(...)        # external network call — NOT transactional
self.db.record_charge(...)                  # DB write
self.db.mark_order_complete(...)            # DB write
```

**Failure scenario:** The payment gateway charges the customer successfully and returns `charge_id`. Before `db.record_charge` executes, the process crashes (OOM, pod eviction, network drop). The Redis lock TTL expires. A second worker acquires the lock, finds no existing charge in the DB, and charges the customer again. The customer is double-charged.

**Required fix:** Move the payment call *outside and before* the DB transaction. The DB transaction should only wrap the two write operations. More importantly, the idempotency check must be made robust against this crash window. The standard pattern is:

1. Acquire lock.
2. Check for existing charge (idempotency).
3. Write a "pending" / "in-flight" record to the DB *before* calling the gateway, inside a transaction.
4. Call the payment gateway.
5. Update the record to "complete" with the `charge_id`, inside a new transaction.

With a "pending" row written before the gateway call, a restarting worker will see the in-flight record and can decide to query the gateway for the outcome rather than re-charging blindly.

### 2. `PaymentError` is silently swallowed — charge outcome is unknown to the caller

```python
# order_processor.py — process() method
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

`PaymentError` (and any other unexpected exception) is not caught inside `process()`. It will propagate up through the `_acquire_lock` context manager, which will correctly release the lock, and then bubble to the caller as an unhandled exception. The caller receives no `OrderResult` — they get a raw exception. Given this is described as a service method, callers likely expect a structured return value in all cases. Add an explicit `except PaymentError` block and return a structured `OrderResult(processed=False, ...)`.

## Secondary Issues

### 3. Timeout on payment gateway is dangerously long relative to the lock TTL

`PaymentClient.REQUEST_TIMEOUT = 30` seconds, `MAX_RETRIES = 3`, `RETRY_BACKOFF = 5` seconds.

Worst-case payment client execution time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`.

`OrderProcessor.ORDER_LOCK_TTL = 60` seconds.

The lock can expire while the payment client is still retrying. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet), and attempt a second charge while the first worker is still waiting on the gateway. This is a race condition baked into the constants. Either reduce the payment client's worst-case time to well under 60 seconds, or extend the lock TTL with a heartbeat, or both. These values need to be explicitly reconciled and documented.

### 4. `requests` response body is accessed without validation

```python
# payment_client.py
charge_id = response.json()["charge_id"]
```

If the gateway returns a 2xx response with a malformed or missing `charge_id` field, this raises an unhandled `KeyError` or `json.JSONDecodeError`. Both will propagate as unexpected exceptions rather than a clean `PaymentError`. Wrap this in a try/except and raise `PaymentError` with a descriptive message.

### 5. Lock retry backoff is too short and the retry count is too low for a 60-second TTL

```python
LOCK_RETRY_ATTEMPTS = 3
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s — total wait ~3s
```

If a worker holds the lock for up to 60 seconds (or 100 seconds per issue #3), three attempts over three seconds will almost always fail for a legitimately in-progress order. The caller then gets a "currently being processed" response, which may cause the upstream system to retry and create more contention. Consider a longer retry window with jitter, or make these values configurable via constructor injection so they can be tuned per deployment.

### 6. Test coverage gap — no test for `PaymentError` propagation

There is no test covering the case where `payment.charge` raises `PaymentError`. Given the unhandled exception issue in #2, this is a meaningful gap. A test should assert that a `PaymentError` results in a structured `OrderResult(processed=False)` and that the lock is still released (i.e., the Lua script is still called).

### 7. Minor — `payment_token` logged at INFO level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is present in the `OrderRequest` object and is a sensitive credential. Ensure no log statement anywhere in the call path serializes the full `req` object. Consider overriding `__repr__` or `__str__` on `OrderRequest` to redact `payment_token`, as a defense-in-depth measure against accidental future logging.

---

## What Is Done Well

- Lua script for atomic lock release is correct and the token-verification pattern is right.
- `SET NX EX` is the correct atomic acquisition primitive — no SETNX/EXPIRE race.
- Pydantic validation with pattern enforcement on `order_id` is appropriate.
- The test for path traversal on `order_id` is a good explicit check.
- `register_script` pre-registration is the right way to use Lua scripts in redis-py.
- Retry-on-5xx with no retry on 4xx in the payment client is the correct distinction.
