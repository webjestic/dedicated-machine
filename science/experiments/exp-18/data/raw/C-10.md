Decision: Request Changes

Summary: The implementation has solid fundamentals — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there is a critical race condition between the payment charge and the database transaction that creates a real double-charge risk, and several other reliability and security issues that must be addressed before this handles financial transactions in production.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — charge-without-record risk

```python
# Current code
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect INSIDE transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious bug. If `self.payment.charge()` succeeds but the database transaction rolls back (network blip, constraint violation, anything), the customer is charged with no record of it. The charge is an irreversible external side effect — it must not be inside a database transaction boundary that can silently discard it.

**Required fix:** Charge first, outside the transaction, then record atomically. If the DB write fails after a successful charge, your retry/recovery path needs to handle that explicitly (e.g., a pending-charge staging record written before the external call).

```python
# Safer pattern
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(processed=False, message="Order already processed")

# Write intent to DB before charging externally
self.db.record_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

---

### 2. Lock expiry can outlast processing — lock TTL is not a safety net here

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient.REQUEST_TIMEOUT = 30` and `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5`. Worst-case payment client runtime is approximately `3 × 30 + 2 × 5 = 100 seconds` — well beyond the 60-second lock TTL. A second worker can acquire the lock while the first is still mid-charge, defeating the entire concurrency guarantee.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed worst-case processing time (e.g., 150s), or implement lock extension (Redis `EXPIRE` refresh) during long operations. Document the relationship between these constants explicitly.

## Significant Issues

### 3. `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
    ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` (and any other exception) propagates out of `process()` as an unhandled exception. The caller gets a raw exception instead of a structured `OrderResult`. This is inconsistent with the method's contract and will cause unhandled 500s upstream.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

---

### 4. Timeout on payment retry does not account for already-charged state

In `PaymentClient.charge()`, a `Timeout` exception on attempt N does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying on timeout can result in duplicate charges at the gateway level. This needs an idempotency key passed to the gateway per-order, not per-attempt.

```python
# Pass a stable idempotency key derived from order_id, not regenerated per retry
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be stable across retries
    timeout=self.REQUEST_TIMEOUT,
)
```

---

### 5. `payment_token` logged indirectly via structured fields — audit this

`payment_token` is stored in `req` and passed directly to `self.payment.charge()`. Confirm that no logging path (including any DB layer or payment client debug logging) serializes the full `OrderRequest` or raw token value. PCI-DSS scope requires tokens never appear in logs. Consider a `__repr__`/`__str__` override on `OrderRequest` that masks `payment_token`.

---

## Minor Issues

### 6. Test for lock release is implementation-coupled and fragile

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This accesses a private attribute and asserts on internal mechanism rather than observable behavior. If the implementation changes to use a different atomic method, this test breaks without any real regression. Test the outcome (lock is not held after processing) rather than the mechanism.

---

### 7. `requests` has no connection timeout, only read timeout

```python
timeout=self.REQUEST_TIMEOUT  # single value = read timeout only
```

A single scalar timeout in `requests` sets the read timeout but not the connection timeout. A hung DNS resolution or TCP handshake will block indefinitely.

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

---

### 8. Missing `order_id` in `OrderResult` on `LockError` path

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`order_id` is not included in this result, making it harder for callers to correlate the failure. Pass `order_id=req.order_id` here (it's available in scope after validation).
