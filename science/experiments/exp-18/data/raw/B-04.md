Decision: Request Changes

Summary: The PR introduces a critical race condition where the worst-case duration of operations held under the lock substantially exceeds the lock TTL, allowing a concurrent worker to acquire the lock, pass the idempotency check, and issue a duplicate charge against the same order. Beyond the TTL breach, the payment call executes *inside* the database transaction, creating a non-atomic boundary between an external side effect and its local record that no rollback can undo. Several secondary issues compound the risk. All must be resolved before this code touches production.

---

## Critical Finding 1 — Lock TTL Exhaustion Enables Duplicate Charges

### The math

Trace every second that can elapse while Worker A holds the lock.

**Lock acquisition retries (before the lock is held):**
These consume time *before* the lock is acquired, so they do not count against the TTL. Noted for completeness; they do not affect the analysis below.

**Operations executed while the lock is held (`ORDER_LOCK_TTL = 60 s`):**

| Step | Worst-case duration |
|---|---|
| `db.get_charge()` — network I/O, no timeout configured | unbounded; assume ≤ 5 s |
| `payment.charge()` — up to 3 attempts | see below |
| &nbsp;&nbsp;• Each HTTP call: `REQUEST_TIMEOUT = 30 s` | 30 s |
| &nbsp;&nbsp;• Backoff between attempts: `RETRY_BACKOFF = 5 s`, fired on attempts 1 and 2 | 10 s |
| &nbsp;&nbsp;• Total for 3 attempts: `3 × 30 + 2 × 5` | **100 s** |
| `db.record_charge()` + `db.mark_order_complete()` | ≤ 5 s |
| **Worst-case total while lock is held** | **≥ 110 s** |

`110 s >> 60 s TTL`.

### The exploit path

1. Worker A acquires the lock on `order-123` (TTL = 60 s).
2. Worker A calls `db.get_charge()` → `None`. Idempotency check passes.
3. Worker A calls `payment.charge()`. The gateway times out twice (2 × 30 s = 60 s). The lock expires at exactly this moment.
4. Worker B acquires the lock on `order-123` (fresh TTL).
5. Worker B calls `db.get_charge()` → still `None`, because Worker A has not yet written the record (it is mid-retry inside `payment.charge()`).
6. Worker B charges the customer a second time and records the transaction.
7. Worker A's third attempt succeeds. Worker A records a second transaction for the same charge.

The customer is charged twice. The Lua release script does not prevent this — it only prevents Worker A from *releasing* Worker B's lock, but the damage is already done.

### Required fix

The lock TTL must be strictly greater than the worst-case duration of all work performed under it, **or** the work must be restructured so that the slow external call happens outside the lock and only a fast, bounded commit happens inside it. The cleanest approach for a payment system is the two-phase pattern:

```python
# Pseudocode — illustrative, not a drop-in replacement
ORDER_LOCK_TTL = 180  # must exceed worst-case with margin

with self._acquire_lock(order_id):
    # Fast idempotency check — still inside lock
    if self.db.get_charge(order_id):
        return already_processed_result()

    # Reserve a slot atomically before releasing the lock
    pending_id = self.db.create_pending_charge(order_id)
    # Lock is released here; pending row prevents a second worker from
    # passing the idempotency check even if the lock has expired.

# Slow external call is now outside the lock
charge_id = self.payment.charge(token=token, amount_cents=amount_cents)

# Fast commit — could acquire a short-lived lock again if needed
with self.db.transaction():
    self.db.finalize_charge(pending_id, charge_id)
    self.db.mark_order_complete(order_id)
```

The `pending` row acts as a durable idempotency marker that survives lock expiry. Alternatively, raise `ORDER_LOCK_TTL` to at least `3 × REQUEST_TIMEOUT + 2 × RETRY_BACKOFF + DB_margin = 3×30 + 2×5 + 10 = 110 s`, add a generous safety factor (2×), and enforce a hard deadline inside the lock with a `threading.Timer` or `signal.alarm` that aborts the operation and raises before the TTL is reached.

## Critical Finding 2 — External Side Effect Inside Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network call to payment gateway
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`payment.charge()` is a non-transactional external call. If it succeeds but `record_charge()` or `mark_order_complete()` subsequently raises and the transaction rolls back, the charge has been collected from the customer but no record exists in the database. The system believes the order was never charged; any retry will charge the customer again.

The database transaction must wrap only database operations. The charge call must precede the transaction:

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This is the standard pattern: charge first, record second. If the record fails you have a charge without a record (recoverable via reconciliation); if you record first and the charge fails you have a record without a charge (also recoverable). The inverse — charge inside a transaction that can roll back — is the only unrecoverable case.

## High Finding 3 — Lock Retry Backoff Consumes Time Before Acquisition, Masking True Contention Duration

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)   # 0.5 s, 1.0 s
    time.sleep(backoff)
```

Worst-case acquisition time: `0.5 + 1.0 = 1.5 s`. This is not the primary issue, but it means a caller can block for 1.5 s before receiving a `LockError`. More importantly, `time.sleep()` on a worker thread blocks the entire thread. In any async or thread-pool deployment this degrades throughput under contention. Use `asyncio.sleep` in async contexts or a non-blocking retry strategy.

## Medium Finding 4 — `PaymentError` Is Not Caught; Unhandled Exception Leaks to Caller

`payment.charge()` raises `PaymentError` on permanent failure. `process()` catches `LockError` but has no handler for `PaymentError` or any other exception raised inside the lock context. The exception propagates to the caller with no structured `OrderResult`, and the lock is released correctly (the `finally` block runs), but the caller receives a raw exception rather than the documented return type. Add an explicit handler:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

## Medium Finding 5 — No Timeout on Database Calls

`db.get_charge()`, `db.record_charge()`, and `db.mark_order_complete()` have no configured timeout. A slow or hung database connection will hold the lock indefinitely (until the TTL expires), which is exactly the condition that triggers the race described in Finding 1. All database calls must have explicit statement or connection timeouts shorter than the lock TTL.

## Low Finding 6 — Test Suite Does Not Cover the TTL Race

The tests mock Redis and the payment client but never simulate a payment call that takes longer than `ORDER_LOCK_TTL`. The most important failure mode — lock expiry mid-operation — has zero test coverage. Add an integration-style test that:

1. Sets `ORDER_LOCK_TTL` to a very small value (e.g., 1 s).
2. Makes `payment.charge()` sleep for longer than the TTL.
3. Asserts that a second worker calling `process()` on the same order does not issue a second charge.

This test will fail with the current implementation, which is the point — it must fail before the fix and pass after.

## Low Finding 7 — Structured Logging Includes Raw `payment_token`

`payment_token` is passed through `OrderRequest` and is available in scope throughout `process()`. Verify that no log statement, exception message, or Pydantic validation error (which serializes field values by default) emits the token. Currently `logger.warning("Invalid order request: %s", e)` will include the value of `payment_token` in the Pydantic `ValidationError` string if that field fails validation. Mask sensitive fields:

```python
class OrderRequest(BaseModel):
    payment_token: str = Field(..., min_length=1, max_length=256)

    model_config = {"json_schema_extra": {}}

    def __repr__(self):
        return f"OrderRequest(order_id={self.order_id!r}, ...)"
```

Or catch `ValidationError`, log only the field names that failed, and never log the raw error object.
