Decision: Request Changes

Summary: The PR's lock TTL is 60 seconds, but the worst-case duration of work performed while the lock is held can substantially exceed that ceiling: the payment client alone can consume up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 × (30 + 5) = 105 seconds` before raising `PaymentError`, and that figure does not yet include database round-trips or the lock-acquisition backoff time already spent before entering the critical section. When the TTL expires mid-flight, a second worker will acquire the lock, pass the idempotency check (because `record_charge` has not yet committed), and issue a duplicate charge — which is precisely the failure mode this PR claims to prevent. A second, independent issue is that `payment.charge()` is called **inside** `db.transaction()`, meaning a successful charge that is followed by a database failure will leave a real charge with no local record and no compensating rollback of the external payment, creating an unrecoverable inconsistency. These two flaws are architectural and must be resolved before merge.

---

## Detailed Findings

### 1. Lock TTL vs. Worst-Case Critical-Section Duration (Critical)

**Location:** `services/order_processor.py` `ORDER_LOCK_TTL = 60` vs. `services/payment_client.py` `REQUEST_TIMEOUT = 30`, `MAX_RETRIES = 3`, `RETRY_BACKOFF = 5`

**The arithmetic:**

```
Per attempt ceiling  = REQUEST_TIMEOUT + RETRY_BACKOFF
                     = 30s + 5s = 35s

Worst-case charge()  = MAX_RETRIES × per-attempt ceiling
                     = 3 × 35s = 105s          ← exceeds TTL alone
```

The lock expires at **60 s**. The payment call can run for **105 s**. The database writes follow the payment call. The lock-acquisition retry loop can also consume up to `0.5 + 1.0 + 1.5 = 3 s` *before* the lock is even held, but that is minor compared to the payment window.

**Failure sequence:**

```
T=0s    Worker A acquires lock, passes idempotency check (no record exists)
T=60s   Redis TTL expires — lock is gone
T=61s   Worker B acquires lock, passes idempotency check (still no record)
T=61s   Worker B issues charge #2  ← duplicate charge
T=105s  Worker A's charge() finally returns charge #1
T=105s  Worker A commits record_charge(charge #1)
T=105s  Worker B commits record_charge(charge #2)
        Two charges, two records, one order.
```

**Required fix — choose one or both:**

```python
# Option A: raise TTL to safely exceed worst-case critical-section duration
ORDER_LOCK_TTL = 180   # 105s payment + 30s DB headroom + margin

# Option B: reduce payment client ceiling to fit inside the TTL
REQUEST_TIMEOUT = 8
MAX_RETRIES = 2
RETRY_BACKOFF = 2
# worst-case = 2 × (8 + 2) = 20s  →  fits inside a 60s TTL with room for DB

# Option C (belt-and-suspenders): extend the lock from within the critical
# section using a background heartbeat thread that calls
#   redis.expire(lock_key, ORDER_LOCK_TTL)
# at ORDER_LOCK_TTL/2 intervals, and only while the worker still holds the token.
```

Whatever ceiling is chosen, it must be documented as a constant with an explicit comment showing the arithmetic, so future changes to retry parameters are not made in isolation.

### 2. External Payment Call Inside the Database Transaction (Critical)

**Location:** `services/order_processor.py` lines inside `with self.db.transaction():`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network call to external system
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction holds a connection (and potentially row-level locks) for its entire duration. Placing a synchronous HTTP call — with a 30-second timeout and three retries — inside that transaction causes two distinct problems:

**2a. Connection exhaustion under load.** Every in-flight order ties up a database connection for up to 105 seconds. Under modest concurrency this drains the connection pool and stalls unrelated operations across the service.

**2b. Unrecoverable charge-without-record on DB failure.** If `payment.charge()` succeeds but the subsequent `record_charge` or `mark_order_complete` raises an exception, the transaction rolls back. The charge has already been captured by the payment gateway; there is no compensating call to reverse it. The customer is billed, the order is not marked complete, and the idempotency record is absent — so a retry will charge again.

**Required fix:**

```python
# Charge OUTSIDE the transaction; record INSIDE.
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only open the DB transaction after the external call succeeds.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

The remaining risk (charge succeeds, DB transaction fails permanently) should be handled by a dead-letter / reconciliation queue that detects charges with no matching DB record and either voids them or records them out-of-band. That is an operational concern beyond this PR, but the immediate fix of moving the network call outside the transaction is mandatory.

### 3. Lock-Acquisition Retry Backoff Is Additive to TTL Pressure (Minor, but track it)

**Location:** `_acquire_lock` retry loop

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    ...
    backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s → up to 3s total
    time.sleep(backoff)
```

This delay occurs *before* the lock is acquired and so does not consume TTL, but it does mean a caller can block for up to 3 seconds before receiving a `LockError`. Under high concurrency this can cascade. Consider capping `LOCK_RETRY_ATTEMPTS` at 2 and documenting the intent.

### 4. `PaymentError` Is Not Caught in `process()` (Minor)

**Location:** `services/order_processor.py` `process()` method

`self.payment.charge()` raises `PaymentError` on permanent failure. `process()` catches `LockError` but has no handler for `PaymentError` or the general `Exception`. An unhandled `PaymentError` will propagate to the caller as an unstructured exception rather than an `OrderResult`, breaking the stated return contract of the method.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 5. Test Coverage Gaps (Minor)

The test suite does not cover:

| Missing scenario | Risk |
|---|---|
| `payment.charge()` raises `PaymentError` | Verifies finding #4 is caught |
| `db.transaction()` raises after a successful charge | Verifies charge-without-record path |
| Lock expires mid-flight (TTL simulation) | Verifies the core concurrency guarantee |
| `payment.charge()` returns a response missing `charge_id` key | `KeyError` propagates unhandled |

The existing tests use `MagicMock` for the DB transaction context manager correctly, but none of them exercise failure paths inside the critical section.
