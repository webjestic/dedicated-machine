Decision: Request Changes

Summary: The PR introduces a critical lock-expiry race condition: the worst-case duration of all work performed under the lock can substantially exceed the 60-second TTL, allowing a second worker to acquire the lock, pass the idempotency check, and issue a duplicate charge against the same order. Additionally, the payment gateway call is made inside the database transaction, coupling network I/O to a held transaction and creating a secondary failure mode. These two issues together undermine the at-most-once guarantee the PR claims to provide.

---

## Critical Issues

### 1. Lock TTL vs. Worst-Case Held Duration (Duplicate Charge Risk)

This is the primary defect. Trace the full execution path under the lock:

```
_acquire_lock TTL = 60 seconds

Work performed while lock is held:
  db.get_charge()                              ~variable, assume fast
  payment.charge() — worst case:
    attempt 1: REQUEST_TIMEOUT = 30s timeout  → 30s
    RETRY_BACKOFF sleep                        →  5s
    attempt 2: REQUEST_TIMEOUT = 30s timeout  → 30s
    RETRY_BACKOFF sleep                        →  5s
    attempt 3: REQUEST_TIMEOUT = 30s timeout  → 30s
                                              --------
    payment.charge() alone                    = 100s worst case

  db.transaction() + record_charge()          ~variable
```

**Worst-case duration under lock: ~100 seconds.**
**Lock TTL: 60 seconds.**

At the 60-second mark, the lock expires in Redis. A second worker can now call `SET NX EX` and succeed. That worker calls `db.get_charge()` — the first worker has not yet committed its transaction, so no charge record exists. The second worker proceeds to call `payment.charge()`. Both workers charge the customer. The first worker's `_release_script` then executes but finds a different token and silently returns 0, which is handled correctly — but the duplicate charge has already occurred.

**Required fix:** The lock TTL must be strictly greater than the worst-case held duration, or the held duration must be strictly bounded below the TTL. Concretely:

```python
# Worst-case held duration:
# MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF
# = 3 * 30 + 2 * 5 = 100 seconds

# TTL must exceed this with margin:
ORDER_LOCK_TTL = 120  # minimum; 150 recommended for DB overhead + clock skew
```

Or, preferably, reduce the payment client's worst-case duration to fit within the TTL:

```python
# If TTL must stay at 60s, then:
# MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF < 60
# With MAX_RETRIES=2, REQUEST_TIMEOUT=10, RETRY_BACKOFF=5: 2*10 + 1*5 = 25s — fits with margin
```

Either approach requires that `ORDER_LOCK_TTL`, `MAX_RETRIES`, `REQUEST_TIMEOUT`, and `RETRY_BACKOFF` are defined together with an explicit invariant assertion, not in separate files with no cross-reference:

```python
# Enforce the invariant at import time so a future config change cannot silently break it
_PAYMENT_WORST_CASE_SECONDS = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
assert OrderProcessor.ORDER_LOCK_TTL > _PAYMENT_WORST_CASE_SECONDS + DB_OPERATION_MARGIN, (
    f"Lock TTL ({OrderProcessor.ORDER_LOCK_TTL}s) must exceed payment worst-case "
    f"({_PAYMENT_WORST_CASE_SECONDS}s) plus DB margin"
)
```

### 2. Payment Gateway Call Inside the Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network I/O inside held transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction holds a connection (and potentially row locks) for its entire duration. Placing a network call with up to 100 seconds of worst-case latency inside that transaction means:

- The DB connection is held for the full payment retry duration, exhausting the connection pool under load.
- If the payment gateway returns success but the subsequent `record_charge()` or `mark_order_complete()` fails and the transaction rolls back, the charge has been collected with no record. This is a financial data integrity failure.
- If the DB connection drops mid-transaction, the payment may have already been captured.

**Required fix:** Perform the charge outside and before the transaction, then record the result atomically:

```python
# Outside the transaction: charge the payment method
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Inside the transaction: record the already-completed charge
with self.db.transaction():
    # Re-check idempotency inside the transaction to close the TOCTOU window
    if self.db.get_charge(req.order_id):
        # Charge already recorded by a concurrent worker; attempt reversal
        self._attempt_reversal(charge_id)
        return OrderResult(processed=False, order_id=req.order_id, message="Order already processed")
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Note that moving the charge outside the transaction introduces the need for a reversal path when a race is detected on the inner idempotency check. This is the correct trade-off: a rare reversal is preferable to holding DB connections open for 100 seconds.

## Moderate Issues

### 3. `PaymentError` Is Not Caught in `process()`

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
```

`PaymentError` and any other exception from `payment.charge()`, `db.record_charge()`, or `db.mark_order_complete()` propagate uncaught out of `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. This is an API contract violation — callers cannot distinguish a payment failure from a programming error without catching the raw exception.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 4. Timeout on `requests.post` Does Not Bound Total Time

```python
timeout=self.REQUEST_TIMEOUT  # 30 seconds
```

`requests` interprets a scalar timeout as the socket read timeout, not a total request duration cap. A slow server that sends response bytes intermittently can hold the connection open indefinitely while each individual read completes within 30 seconds. Use a tuple:

```python
timeout=(5, 30)  # (connect_timeout, read_timeout)
```

This does not change the worst-case arithmetic above materially, but it closes a separate availability hole.

### 5. `response.json()["charge_id"]` Has No Error Handling

```python
charge_id = response.json()["charge_id"]
```

A malformed or unexpected gateway response raises `KeyError` or `json.JSONDecodeError`, both of which propagate uncaught and are not retried. This should be wrapped:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

## Minor Issues

### 6. Lock Retry Backoff Adds to Worst-Case Held Duration Calculation

The lock acquisition retry loop sleeps up to `0.5 + 1.0 = 1.5` seconds before giving up. This does not affect the TTL race (the lock is not yet held during acquisition retries), but it does mean `process()` can block for 1.5 seconds before returning a contention error. This is acceptable but should be documented.

### 7. Test Coverage Does Not Cover the TTL Race

No test exercises the scenario where the lock expires mid-processing. Given that this is the central correctness claim of the PR, a test that simulates lock expiry (mock `redis.set` to return `True` but advance a mock clock past TTL before `record_charge` is called) should be required.

### 8. `payment_token` Logged Indirectly via `payment_token` Field

`payment_token` is present in `OrderRequest` and passed directly to `payment.charge()`. Confirm that no logging path in `PaymentClient` logs the token value. The current code does not, but this should be called out explicitly in a code comment given the sensitivity of the field.
