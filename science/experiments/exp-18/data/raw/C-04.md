Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical race condition between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data safely.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — charge can succeed while the record is lost

This is the most serious bug in the PR. The `payment.charge()` call is made *inside* `self.db.transaction()`:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `record_charge` or `mark_order_complete` raises after the payment gateway has already returned a `charge_id`, the database transaction rolls back. The customer has been charged. The system has no record of it. On retry, the idempotency check (`get_charge`) finds nothing and charges again.

The payment call must happen **before** the transaction opens, and the `charge_id` must be passed into the transaction:

```python
# Charge outside the transaction — this is an irreversible external side effect
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only record after the charge is confirmed
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This doesn't fully eliminate the window (the process can still die between charge and record), but it removes the rollback-induced double-charge path and is the correct pattern for at-most-once semantics with an external payment gateway.

### 2. `PaymentError` is not caught in `process()` — unhandled exceptions escape to the caller while the lock is still held

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # raises PaymentError on failure
```

`PaymentError` propagates out of `_acquire_lock`'s `try/finally`, so the lock *is* released correctly (the `finally` block runs). However, the exception then propagates out of `process()` entirely, giving the caller a raw exception instead of a structured `OrderResult`. This is inconsistent with how `LockError` and `ValidationError` are handled. Add a `PaymentError` catch:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

## Significant Issues

### 3. Timeout on payment gateway creates an ambiguous charge state that is silently retried

In `payment_client.py`, a `Timeout` exception is caught and retried:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request *may have succeeded* on the gateway side — the response was simply not received. Retrying unconditionally can result in double charges. This requires idempotency keys on the payment gateway request. The `order_id` is the natural key:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # pass order_id through
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

`PaymentClient.charge()` should accept an `idempotency_key` parameter, and `OrderProcessor` should pass `req.order_id`.

### 4. Redis connection failures are not handled — an unhandled `redis.RedisError` bypasses all error handling

If the Redis client raises during `SET` or during the Lua script release, neither `LockError` nor any other structured handler catches it. The raw exception escapes `process()`. At minimum, wrap Redis calls and return a structured error result, or let it propagate as a clearly documented contract. Right now it's neither.

### 5. Lock TTL of 60 seconds is shorter than the maximum possible processing time

`ORDER_LOCK_TTL = 60` but `PaymentClient` can take up to `REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1)` = `30 * 3 + 5 * 2` = **100 seconds** in the worst case. The lock expires while the first worker is still processing, a second worker acquires the lock, passes the idempotency check (no record yet), and issues a second charge. Either increase the TTL to at least 120 seconds with margin, or reduce the payment client's maximum retry window.

## Minor Issues

### 6. `payment_token` is logged indirectly — verify it never reaches log output

`req.payment_token` is stored on the validated model. Confirm that no log call anywhere serializes the full `OrderRequest` object. The current code appears safe (only `order_id` and `charge_id` are logged), but this should be explicitly noted and a `__repr__`/`__str__` override on `OrderRequest` that masks `payment_token` would make it structurally safe.

### 7. Test for lock release verifies the mock was called but not the arguments

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This passes even if the script is called with the wrong key or a hardcoded token. Assert on `call_args` to verify the correct `lock_key` is passed and that the token is a non-empty string.

### 8. No test covers the payment failure path or the charge-outside-transaction behavior

Once issue #1 is fixed, a test should verify that a `PaymentError` returns a structured `OrderResult` and that `record_charge` is never called if the charge fails.
