Decision: Request Changes

Summary: The implementation contains a critical lock-lifetime violation that directly undermines the at-most-once payment guarantee it claims to provide. Beyond that, there are several secondary architectural and correctness issues that must be resolved before this handles real payment data.

---

## Critical: Lock TTL Can Expire Before Work Completes

This is the primary rejection reason. The lock TTL is 60 seconds (`ORDER_LOCK_TTL = 60`). The worst-case duration of the work performed *while holding the lock* must be calculated against that budget.

**Worst-case execution path inside `_acquire_lock` context:**

```
PaymentClient.charge() — one attempt:
  requests.post(..., timeout=30)  →  30s (timeout fires)
  time.sleep(RETRY_BACKOFF=5)     →   5s

  requests.post(..., timeout=30)  →  30s (timeout fires)
  time.sleep(RETRY_BACKOFF=5)     →   5s

  requests.post(..., timeout=30)  →  30s (timeout fires)
  # no sleep after final attempt

Total payment.charge() worst-case: 3×30 + 2×5 = 100 seconds
```

**100 seconds of payment work inside a 60-second lock.** At the 60-second mark, Redis expires the key. A second worker acquires the lock, passes the idempotency check (because `record_charge` has not been called yet — the first worker is still waiting on a gateway timeout), and issues a second charge. Both workers then record their charges, and the customer is billed twice.

The checklist item "Distributed lock with atomic acquisition" is checked, but the lock does not actually protect the critical section for its full duration. This is the exact failure mode the PR claims to prevent.

**Required fix — choose one or combine:**

1. Raise `ORDER_LOCK_TTL` to comfortably exceed worst-case work duration. With the current payment client: `3 × 30 + 2 × 5 + db_overhead + margin` → minimum 120s, recommend 180s.
2. Reduce `PaymentClient` worst-case: lower `REQUEST_TIMEOUT` and/or `MAX_RETRIES × RETRY_BACKOFF` so the product fits inside the TTL with margin.
3. Implement lock extension (a background thread or async task that calls `EXPIRE` to refresh the TTL while the worker is still alive). This is the most robust option for variable-duration operations.

Option 3 is strongly preferred for production payment infrastructure because it keeps the TTL tight for the common case (fast expiry on dead workers) while guaranteeing coverage for legitimately slow operations.

## High: `payment.charge()` Is Called Inside `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call, up to 100s
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

An external network call is being made while a database transaction is open. This holds a database connection and potentially row-level locks for the full duration of the payment gateway round-trip (up to 100 seconds worst-case). Under any meaningful concurrency this will exhaust the connection pool and cause cascading failures across the service. The charge call must be moved *before* the transaction opens:

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This also requires handling the case where `charge` succeeds but the subsequent `db.transaction()` fails — that is a real partial-failure scenario that needs a reconciliation path (e.g., a dead-letter record of the `charge_id` written before the transaction, or a compensating refund on DB failure).

---

## High: Timeout Does Not Bound Total `requests.post` Duration

`requests.post(..., timeout=30)` sets a socket inactivity timeout, not a total request duration cap. If the server accepts the connection and streams a response byte every 29 seconds, this call will never time out regardless of the configured value. For a payment gateway call this is a correctness concern, not just a performance concern, because it means the lock-expiry calculation above is actually a *lower* bound. Use `requests` with an explicit total timeout or `httpx` with `timeout=httpx.Timeout(connect=5, read=30, write=10, pool=5)`.

---

## Medium: Transient vs. Idempotent Retry Is Not Distinguished

`PaymentClient.charge()` retries on 5xx and timeout. If the gateway received and processed the first request but returned a 500 (a common pattern — charge succeeded, response failed), retrying will issue a second charge against the same `payment_token`. The retry logic must either:

- Use a stable, per-attempt idempotency key sent as a request header (e.g., `Idempotency-Key: <order_id>-<attempt>`), or
- Only retry on connection-level errors and explicit gateway-documented retryable codes, not all 5xx.

Without this, the payment client itself can double-charge independently of the Redis lock.

---

## Medium: Lock Acquisition Retry Adds Unbounded Pre-Lock Delay

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    ...
    backoff = 0.5 * (attempt + 1)                 # 0.5s, 1.0s, 1.5s
    time.sleep(backoff)
```

Worst-case pre-lock wait is 0.5 + 1.0 + 1.5 = 3 seconds. This is acceptable in isolation, but it is additive with the in-lock duration for any caller reasoning about total request latency. More importantly, this retry loop is blocking — in any async or thread-pool context it ties up the worker thread. This should be documented as a synchronous-only design constraint, or the sleep should use `asyncio.sleep` if the service ever moves to an async runtime.

---

## Low: `PaymentError` Is Silently Swallowed

`process()` catches `LockError` explicitly but does not catch `PaymentError` or any other exception from `payment.charge()` or the database calls. An unhandled `PaymentError` will propagate as an uncaught exception to the caller. This may be intentional (let the framework handle it), but if so it should be documented. If the intent is to return a structured `OrderResult` on all failure paths, add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

## Low: Test Coverage Does Not Exercise the Failure Modes That Matter

The tests mock `payment.charge` as an instant success. None of the following are tested:

- Lock expires mid-flight (the critical bug above)
- `payment.charge` raises `PaymentError` — verifies structured error return
- `db.transaction()` raises after a successful charge — verifies no silent data loss
- Retry behavior in `PaymentClient` — verifies backoff timing and retry count
- Concurrent acquisition by two workers on the same `order_id`

The existing tests verify happy-path wiring. They would pass even if the TTL were set to 1 second.

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| Critical | Lock TTL (60s) < worst-case work duration (100s+); concurrent worker can acquire lock and duplicate charge |
| High | External HTTP call inside open DB transaction; will exhaust connection pool under load |
| High | `requests` timeout is inactivity-only, not total-duration; actual worst case is unbounded |
| Medium | Payment retry on 5xx does not use idempotency key; gateway-side duplicate charge possible |
| Medium | Blocking sleep in lock retry; must be documented or made async-safe |
| Low | `PaymentError` unhandled in `process()`; breaks structured error contract |
| Low | Tests do not cover TTL expiry, payment failure, or post-charge DB failure paths |
