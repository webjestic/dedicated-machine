Decision: Request Changes

Summary: The implementation contains a critical lock-expiry race condition in which the worst-case duration of operations held under the lock substantially exceeds the 60-second TTL, allowing a concurrent worker to acquire the lock, pass the idempotency check, and issue a duplicate charge against the same order. Two secondary issues compound the risk: the payment gateway call sits inside the database transaction, and a timeout on the payment gateway produces an ambiguous outcome that the retry logic does not handle safely. These flaws directly contradict the at-most-once guarantee the PR claims to provide.

---

## Detailed Findings

### 1. (Critical) Lock TTL vs. Worst-Case Held Duration — Duplicate Charge Window

This is the primary defect. Trace the full lock lifecycle against the worst-case execution path.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration while the lock is held:**

| Step | Worst-case time |
|---|---|
| `db.get_charge()` — idempotency read | ~2 s (network + slow replica) |
| `payment.charge()` attempt 1 — timeout | 30 s (`REQUEST_TIMEOUT`) |
| `RETRY_BACKOFF` sleep between attempts 1→2 | 5 s |
| `payment.charge()` attempt 2 — timeout | 30 s |
| `RETRY_BACKOFF` sleep between attempts 2→3 | 5 s |
| `payment.charge()` attempt 3 — timeout | 30 s |
| `db.record_charge()` + `db.mark_order_complete()` | ~2 s |
| **Total** | **~104 s** |

The worst-case path is **~104 seconds**. The lock expires after **60 seconds**.

**Exploitation sequence:**

```
t=0   Worker A acquires lock, reads idempotency → no prior charge
t=60  Redis expires the lock (Worker A is mid-retry on payment attempt 2)
t=60  Worker B acquires the lock (it is now free)
t=60  Worker B reads idempotency → still no prior charge (A hasn't written yet)
t=60  Worker B calls payment.charge() → second charge issued ← DUPLICATE
t=90  Worker A's payment.charge() succeeds → first charge recorded
t=90  Worker B's payment.charge() succeeds → second charge recorded
```

The idempotency check is not atomic with the charge; it is only a read under a lock that can expire. Once the lock expires, the check provides no protection.

**Fix — two changes are both required:**

```python
# 1. TTL must exceed the mathematical worst-case, with margin.
#    Worst case: (MAX_RETRIES * REQUEST_TIMEOUT) + ((MAX_RETRIES - 1) * RETRY_BACKOFF) + db overhead
#    = (3 * 30) + (2 * 5) + 5 = 105 s  →  set TTL to at least 120–180 s.
ORDER_LOCK_TTL = 180  # seconds

# 2. Re-validate the lock is still held before writing.
#    After payment.charge() returns, confirm the token is still present in Redis
#    before committing. If the token is gone, the lock expired; abort and compensate.
@contextmanager
def _acquire_lock(self, order_id: str):
    token = str(uuid.uuid4())
    lock_key = f"order:lock:{order_id}"
    ...
    try:
        yield token, lock_key          # expose both to the caller
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

```python
# In process(), after charge() returns and before committing:
with self.db.transaction():
    # Guard: confirm this worker still holds the lock before writing.
    current = self.redis.get(lock_key)
    if current is None or current.decode() != token:
        # Lock expired while we were charging. The charge may have gone through.
        # Compensate (void/refund) and surface an error — do not record.
        self.payment.void_or_refund(charge_id)
        raise LockError("Lock expired during payment; charge voided")
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Note: the re-validation check itself has a narrow TOCTOU window (the lock could expire between the `GET` and the `record_charge` commit), but this window is milliseconds rather than tens of seconds, and it eliminates the dominant risk. A stronger alternative is to use a database-level idempotency key with a unique constraint so the `INSERT` itself is the atomic guard, making the lock a performance optimization rather than a correctness mechanism.

### 2. (High) Payment Gateway Call Inside the Database Transaction

```python
# Current code
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network I/O inside tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This holds a database transaction open for up to 105 seconds (the full payment retry budget). Consequences:

- **Connection pool exhaustion.** Under load, all connections are blocked waiting on the payment gateway.
- **Lock contention in the database.** Any row-level locks held by the transaction block other readers and writers for the same duration.
- **Deadlock risk.** Long-held transactions dramatically increase the probability of deadlock with other concurrent transactions.

**Fix:** Perform the charge outside the transaction, then open the transaction only for the database writes.

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Transaction is now short — only two fast DB writes.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

### 3. (High) Timeout ≠ Failure — Unsafe Retry on Ambiguous Gateway Response

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` means the client did not receive a response within 30 seconds. It does **not** mean the charge was not processed. The payment gateway may have received the request, authorized the charge, and sent a response that never arrived. Retrying unconditionally in this case issues a second charge attempt for a payment that may already have succeeded.

**Fix:** The payment client must use idempotency keys on every request so the gateway can deduplicate retries server-side. The key must be stable across retries for the same logical charge (i.e., derived from `order_id`, not generated per-attempt).

```python
def charge(self, token: str, amount_cents: int, order_id: str) -> str:
    idempotency_key = f"charge-{order_id}"   # stable across retries

    for attempt in range(self.MAX_RETRIES):
        try:
            response = requests.post(
                self.CHARGE_ENDPOINT,
                json={"token": token, "amount": amount_cents},
                headers={"Idempotency-Key": idempotency_key},
                timeout=self.REQUEST_TIMEOUT,
            )
            ...
```

Without this, the retry loop is unsafe regardless of the lock behavior.

### 4. (Medium) Lock Acquisition Retry Backoff Also Consumes TTL Budget

The lock acquisition retry loop sleeps up to `0.5 + 1.0 + 1.5 = 3 seconds` before giving up. This time is not accounted for in the TTL analysis above (it occurs before the lock is held, so it does not reduce the held duration), but it does mean a worker can wait up to 3 seconds before discovering it cannot acquire the lock. This is acceptable in isolation but should be documented so future changes to `LOCK_RETRY_ATTEMPTS` or backoff coefficients are evaluated against the TTL budget.

---

### 5. (Low) Missing `PaymentError` Handling in `process()`

`payment.charge()` raises `PaymentError` on permanent failure. This exception is not caught in `process()`, so it propagates to the caller as an unhandled exception rather than returning a structured `OrderResult`. The lock is still released (the `finally` block runs), but the caller receives an exception instead of the documented return type.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 6. (Low) Test Coverage Gaps

The tests do not cover the scenarios that matter most for correctness:

- No test for lock expiry mid-operation (the critical race).
- No test for `payment.charge()` raising `PaymentError` — the unhandled exception path.
- No test asserting that `payment.charge()` is called **outside** the database transaction (once finding 2 is fixed).
- No test for the timeout-then-retry path in `PaymentClient`.
- `test_lock_release_uses_lua_script` asserts the Lua script was called but does not assert the correct `token` was passed as `ARGV[1]`, which is the entire point of the token-verified release.

---

## Required Changes Before Approval

1. **Increase `ORDER_LOCK_TTL`** to exceed the mathematical worst-case held duration with margin (`≥ 180 s` given current retry parameters), and add a comment tying the value to the formula so it is updated when retry parameters change.
2. **Add post-charge lock validation** before the database commit, with compensation (void/refund) if the lock has expired.
3. **Move `payment.charge()` outside the database transaction.**
4. **Add idempotency keys** to all payment gateway requests, derived from `order_id` and stable across retries.
5. **Catch `PaymentError`** in `process()` and return a structured `OrderResult`.
