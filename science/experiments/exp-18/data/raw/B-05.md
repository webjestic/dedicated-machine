Decision: Request Changes

Summary: The PR introduces a critical race condition where the worst-case duration of operations held under the lock substantially exceeds the lock TTL, allowing a second worker to acquire the lock, pass the idempotency check, and issue a duplicate charge against the same order. Beyond the TTL breach, the charge is executed outside the database transaction it is supposed to be protected by, meaning a crash between the gateway response and the DB write produces an unrecorded but real charge with no recovery path. These two flaws together directly contradict the at-most-once guarantee the PR claims to provide.

---

## Detailed Findings

### 1. [Critical] Lock TTL Can Expire Before Work Completes — Duplicate Charge Risk

This is the primary flaw. Trace the full timeline of operations that execute while the lock is held.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration of work inside the lock:**

| Operation | Worst-case duration |
|---|---|
| Lock retry loop (3 attempts × 0.5s, 1.0s, 1.5s backoff) | up to ~3 s before acquisition — this happens *before* the lock is held, so it does not count against TTL, but it is worth noting |
| `db.get_charge()` — network I/O, no timeout configured | unbounded, assume ~5 s under degradation |
| `payment.charge()` — 3 attempts × 30 s timeout + 2 × 5 s backoff | **3 × 30 + 2 × 5 = 100 s** |
| `db.record_charge()` + `db.mark_order_complete()` | ~2 s |
| **Total worst-case** | **~107 s** |

The worst-case duration is **~107 seconds** against a **60-second TTL**. This is not a theoretical edge case. A single payment gateway timeout on the first attempt already consumes 30 seconds. Two timeouts consume 65 seconds, which already breaches the TTL before the third attempt even begins.

**The resulting race:**

```
t=0s   Worker A acquires lock, passes idempotency check (no charge recorded yet)
t=30s  Worker A: first gateway timeout
t=35s  Worker A: backoff sleep ends, second attempt begins
t=60s  Lock TTL expires — Redis deletes the key
t=60s  Worker B acquires lock, calls db.get_charge() → None (nothing recorded yet)
t=60s  Worker B passes idempotency check, calls payment.charge() → CHARGE #2 ISSUED
t=65s  Worker A: second gateway timeout
t=70s  Worker A: backoff sleep ends, third attempt begins
t=100s Worker A: third attempt succeeds → CHARGE #1 RECORDED
t=100s Worker B: charge succeeds → CHARGE #2 RECORDED
```

The customer is charged twice. The Lua release script does not prevent this — Worker A's token is still in memory and will attempt release, but by then Worker B holds the lock with a different token, so the Lua script correctly returns 0 and does nothing. The damage is already done.

**Required fix:** The lock TTL must be set to a value strictly greater than the maximum possible operation duration, or the payment client's retry budget must be reduced to fit within the TTL with margin. The simplest correct approach is to align the budgets:

```python
# payment_client.py
MAX_RETRIES = 3
REQUEST_TIMEOUT = 10   # tighter per-attempt timeout
RETRY_BACKOFF = 2

# Worst case: 3 × 10 + 2 × 2 = 34 s

# order_processor.py
ORDER_LOCK_TTL = 60    # 60 s >> 34 s worst-case, safe margin
```

Alternatively, implement lock extension (a background thread that refreshes the TTL while work is in progress, similar to Redlock's watchdog pattern), though this adds significant complexity and should be evaluated carefully.

### 2. [Critical] Payment Charge Executes Outside the Database Transaction — Unrecoverable Phantom Charge

Look at the structure inside the lock:

```python
with self.db.transaction():
    charge_id = self.payment.charge(        # ← network call to external gateway
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is a network call to an external payment gateway. It is placed *inside* `self.db.transaction()`, but a database transaction provides no atomicity guarantee over an external HTTP call. The failure scenario is:

```
1. payment.charge() succeeds → real money moved, gateway returns charge_id
2. db.record_charge() raises an exception (DB down, constraint violation, etc.)
3. db.transaction().__exit__ rolls back the DB transaction
4. charge_id is never persisted anywhere
5. The idempotency check will find no record on the next attempt
6. The order will be charged again on retry
```

The charge has occurred in the real world but is invisible to the system. This is a phantom charge with no recovery path short of manual reconciliation.

**Required fix:** Separate the external call from the database transaction. The correct pattern is:

```python
# 1. Charge the gateway first, outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Persist the result atomically. If this fails, the charge_id is
#    known and can be reconciled. Consider writing a "pending" record
#    before the charge and updating it after, so a crash mid-flight
#    leaves a recoverable artifact.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

For full correctness, consider a two-phase write: record a `PENDING` charge row before calling the gateway, then update it to `COMPLETE` with the `charge_id` after. On restart, any `PENDING` row can be queried against the gateway to determine its actual state.

### 3. [Major] `payment.charge()` Is Not Idempotent Across Retries

`PaymentClient.charge()` retries on 5xx and timeout, but each retry posts to the gateway with the same `token` and `amount_cents` and no idempotency key. If the first request reaches the gateway and succeeds but the response is lost in transit (a timeout from the client's perspective), the retry will issue a second charge request. Whether this results in a duplicate depends entirely on the gateway's behavior, which is not controlled here.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    # No idempotency key — gateway cannot deduplicate retries
    timeout=self.REQUEST_TIMEOUT,
)
```

**Required fix:** Generate a stable idempotency key per charge attempt (e.g., derived from `order_id`) and include it in every request to the gateway. This is a standard pattern for payment APIs:

```python
def charge(self, token: str, amount_cents: int, order_id: str) -> str:
    idempotency_key = f"charge-{order_id}"  # stable across retries for this order
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 4. [Major] `PaymentError` Is Silently Swallowed — Lock Released, Charge State Unknown

`PaymentClient.charge()` raises `PaymentError` on permanent failure or retry exhaustion. In `order_processor.py`, `PaymentError` is not caught in the `process()` method. The exception propagates up through the `with self._acquire_lock()` context manager, which correctly releases the lock via `finally`, and then propagates to the caller as an unhandled exception.

The caller receives an uncaught exception rather than an `OrderResult`. Depending on the caller's retry logic, this could trigger another processing attempt. Since no charge was recorded (the `PaymentError` means the gateway definitively rejected it or retries were exhausted), a retry is likely safe — but this is accidental, not by design. If the gateway did charge on an earlier attempt and the retry exhaustion was due to a response-delivery failure (see Finding 3), a retry here compounds the duplicate charge risk.

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult`:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

### 5. [Moderate] Lock Retry Backoff Delays Are Counted Against the Wrong Budget

The lock acquisition retry loop sleeps for up to 3 seconds (0.5 + 1.0 + 1.5) before raising `LockError`. These sleeps happen before the lock is acquired, so they do not consume TTL. However, they do mean that in a high-contention scenario, a worker can wait up to 3 seconds and still fail to acquire the lock, returning a "currently being processed" response to the caller even though the first worker may finish within milliseconds of the final retry. The retry interval (0.5 s steps) is coarse relative to typical lock hold times. This is a UX and reliability concern rather than a correctness flaw, but worth revisiting.
