Decision: Request Changes

Summary: The implementation has a critical race condition where the worst-case operation duration under the lock can substantially exceed the lock TTL, allowing a concurrent worker to acquire the lock, pass the idempotency check, and issue a duplicate charge. Beyond that timing flaw, there are several secondary issues around error handling, transaction safety, and test coverage that must be addressed before this can be approved.

---

## Critical: Lock TTL vs. Worst-Case Held Duration

This is the primary blocker. Walk the math explicitly.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration while the lock is held** (all operations sequential, all retries exhausted):

| Step | Worst-case time |
|---|---|
| `db.get_charge()` network call | ~5s (unbound, no timeout set) |
| `payment.charge()` attempt 1: timeout | 30s (`REQUEST_TIMEOUT`) |
| Backoff between attempt 1 and 2 | 5s (`RETRY_BACKOFF`) |
| `payment.charge()` attempt 2: timeout | 30s |
| Backoff between attempt 2 and 3 | 5s |
| `payment.charge()` attempt 3: timeout | 30s |
| `db.record_charge()` + `db.mark_order_complete()` | ~5s (unbound) |
| **Total** | **~110s** |

`110s >> 60s TTL`. A second worker acquires the lock at T=60. At that moment, Worker 1 is still mid-charge (likely on retry 2 or 3). Worker 2 calls `db.get_charge()` â€” which returns `None` because Worker 1 has not yet committed â€” and proceeds to charge the same payment token. **This is a double-charge.**

The idempotency check is not a guard here; it is inside the lock, and the lock has already expired.

**Required fix:** The lock TTL must be set to a value strictly greater than the worst-case held duration, or â€” preferably â€” the worst-case held duration must be reduced to fit within a reasonable TTL. The two levers are:

1. Reduce `REQUEST_TIMEOUT` and `MAX_RETRIES`/`RETRY_BACKOFF` in `PaymentClient` so that the total payment operation is bounded well below the TTL.
2. Extend `ORDER_LOCK_TTL` to cover the realistic worst case, with documented justification.
3. Implement lock extension (e.g., a watchdog thread that refreshes the TTL while the worker is still alive), which is the correct long-term solution for operations with variable duration.

A concrete safe bound for the current retry config: `MAX_RETRIES=3`, `REQUEST_TIMEOUT=30`, `RETRY_BACKOFF=5` gives a payment-only worst case of `3*30 + 2*5 = 100s`. The TTL must exceed `100s + DB overhead + margin`. Either cut the payment retry budget or raise and document the TTL. Both numbers must be co-owned.

## High: Payment Call Inside Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A network call to an external payment gateway is inside a database transaction. This holds a DB connection and potentially row-level locks for the full duration of the payment call (up to 100s worst-case). Under load this will exhaust the connection pool and cause cascading failures across the service. It also creates a correctness hazard: if the payment gateway returns success but the DB commit fails, you have a charge with no record.

**Required fix:** Restructure to:

```python
# Outside the transaction:
charge_id = self.payment.charge(...)

# Then atomically record:
with self.db.transaction():
    # Re-check idempotency inside the transaction to close the window
    if self.db.get_charge(req.order_id):
        # charge already recorded â€” void/refund the new charge_id here
        ...
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The re-check inside the transaction is necessary because the window between the first `get_charge` and the `record_charge` commit is exactly where a duplicate slips through if the lock has expired.

## High: `PaymentError` Not Caught in `process()`

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        charge_id = self.payment.charge(...)   # can raise PaymentError
        ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` is not caught. It propagates to the caller as an unhandled exception. The caller receives no `OrderResult`, gets no structured error, and â€” depending on the caller's retry behavior â€” may retry the entire `process()` call, which could result in a second charge attempt if the first charge actually succeeded at the gateway but raised on the response parse.

**Required fix:** Catch `PaymentError` explicitly and return a structured `OrderResult(processed=False, ...)`. Also consider distinguishing between "gateway declined" (permanent, do not retry) and "gateway error" (transient, already retried internally).

## Medium: Retry Backoff in Lock Acquisition Adds Latency Before Failure

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):   # 3 attempts
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s
    time.sleep(backoff)
```

Total blocking time before raising `LockError`: up to `0.5 + 1.0 = 1.5s`. This is a synchronous sleep on whatever thread is serving the request. Under high contention this is acceptable, but the retry-on-lock-contention pattern implies the caller expects the lock to be released soon â€” which contradicts the 60s TTL. If a worker holds the lock for the full TTL, retrying after 0.5s is pointless. Either remove the retries (fail fast on contention, since the lock is long-lived) or document why retries are expected to succeed within the backoff window.

## Medium: Unhandled `RedisError` in Lock Acquisition and Release

If Redis is unavailable, `self.redis.set(...)` raises `redis.exceptions.RedisError`. This is not caught anywhere in `_acquire_lock` or `process()`. The exception propagates as an unhandled error. For a payment service, Redis unavailability should be a handled, logged, structured failure â€” not an uncaught exception that may expose a stack trace to the caller.

Similarly, `self._release_script(...)` in the `finally` block can raise. An exception in `finally` suppresses the original exception from the `try` block, making debugging very difficult.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
    except redis.exceptions.RedisError:
        logger.error("Failed to release lock order_id=%s â€” will expire at TTL", order_id)
```

## Medium: `charge_id` Scope Leak on Exception Path

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
    self.db.mark_order_complete(...)

logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

If `self.payment.charge()` raises, `charge_id` is never assigned, and the `logger.info` line below the transaction block would raise `UnboundLocalError`. This is currently masked because `PaymentError` is uncaught and the logger line is never reached â€” but it is a latent bug waiting for a refactor to expose it. Initialize `charge_id = None` before the transaction block.

## Low: Test Coverage Gaps

The tests mock away the exact behavior that matters most.

1. **No test for the TTL race.** There is no test that verifies behavior when the lock expires mid-operation. This is the critical failure mode.
2. **No test for `PaymentError` propagation.** `payment.charge.side_effect = PaymentError(...)` is never exercised. The test suite would pass even if `PaymentError` crashes the service.
3. **No test for Redis unavailability.** `redis_client.set.side_effect = redis.exceptions.RedisError(...)` is not covered.
4. **`test_lock_release_uses_lua_script` is fragile.** It asserts `redis_client.delete.assert_not_called()` as a proxy for "Lua script was used." This passes even if the Lua script call is also a no-op. Assert on the script's arguments instead.
5. **Lock retry backoff is not tested.** The `time.sleep` calls in `_acquire_lock` are not mocked, so `test_process_lock_contention` runs `1.5s` of real wall-clock sleep. Patch `time.sleep`.
