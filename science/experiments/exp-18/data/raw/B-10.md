Decision: Request Changes

Summary: The implementation contains a critical lock-expiry race condition that directly contradicts the PR's stated at-most-once guarantee: the worst-case duration of operations held under the lock substantially exceeds the lock TTL, meaning a second worker can acquire the lock, pass the idempotency check, and issue a duplicate charge while the first worker's payment call is still in flight. Several secondary issues compound the risk. This code must not be merged into any payment path.

---

## Critical: Lock TTL vs. Worst-Case Held Duration

This is the primary blocker. Trace the full lifecycle carefully.

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds.

**Worst-case duration of work performed while the lock is held:**

```
payment_client.charge() worst case
  = MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
  = 3 × 30s + 2 × 5s
  = 90s + 10s
  = 100 seconds
```

The lock expires in 60 seconds. The payment call alone can run for 100 seconds. The database writes (`record_charge`, `mark_order_complete`) add additional time on top of that.

**The race this creates:**

```
t=0s   Worker A acquires lock, reads get_charge() → None (no prior charge)
t=60s  Redis TTL expires; lock is gone
t=60s  Worker B acquires the same lock, reads get_charge() → still None
         (Worker A has not yet committed — it is still waiting on a gateway timeout)
t=60s  Worker B charges the customer → charge_id_B issued
t=60s  Worker B commits record_charge(charge_id_B) → idempotency row written
t=90s  Worker A's gateway call finally returns → charge_id_A issued
t=90s  Worker A attempts to commit record_charge(charge_id_A)
```

At this point the customer has been charged twice. Whether Worker A's commit succeeds or fails depends entirely on whether the DB schema has a unique constraint on `order_id` in the charges table — which is not shown and cannot be assumed. Even if it does fail, the charge against the payment method has already been captured by the gateway; a duplicate charge has occurred and must be reversed out-of-band.

**Fix required:** The lock TTL must be set to a value strictly greater than the worst-case held duration, with margin. Given the current payment client parameters that is a minimum of ~120 seconds. Alternatively — and preferably — reduce `REQUEST_TIMEOUT` and `MAX_RETRIES`/`RETRY_BACKOFF` so that the worst case fits well inside a reasonable TTL, then set the TTL to 1.5–2× that worst case. Both parameters must be owned together and documented as a coupled invariant.

```python
# These three constants are coupled. The invariant that must hold:
#   ORDER_LOCK_TTL > MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF
# Current values violate it: 60 < 100. Suggested values below restore it.

# In PaymentClient:
REQUEST_TIMEOUT = 10   # seconds
MAX_RETRIES     = 3
RETRY_BACKOFF   = 2    # seconds
# Worst case: 3*10 + 2*2 = 34 seconds

# In OrderProcessor:
ORDER_LOCK_TTL  = 60   # seconds — now safely exceeds 34s with margin
```

Add a startup assertion or a unit test that encodes this invariant explicitly so it cannot silently regress:

```python
assert OrderProcessor.ORDER_LOCK_TTL > (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
), "Lock TTL must exceed worst-case payment client duration"
```

## High: No Database-Level Idempotency Guarantee

The idempotency check is a read followed by a write, not an atomic upsert. Even with the lock working correctly, if the DB schema does not enforce `UNIQUE(order_id)` on the charges table, a crash between the lock expiry and the commit leaves no record, and the next worker will re-charge. The code assumes `get_charge()` is authoritative, but it is only authoritative if the schema makes it impossible to insert a second row for the same `order_id`.

This must be verified and documented. If the schema does not have that constraint, add it. The application-level check is a performance optimization on top of the constraint, not a replacement for it.

---

## High: Payment Call Inside Database Transaction

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external network call — up to 100s
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Holding a database transaction open across an external HTTP call is a serious resource leak. Database connections and row locks are held for the full duration of the payment call (up to 100 seconds under current parameters). Under any meaningful concurrency this will exhaust the connection pool and stall or crash the service.

The correct pattern is charge-then-record, with the DB transaction wrapping only the DB writes:

```python
# Outside any DB transaction:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only DB work inside the transaction:
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This does introduce a narrow window where a charge is captured but not yet recorded (if the process dies between the two lines). That window must be closed by a reconciliation job that detects captured-but-unrecorded charges from the gateway and writes them retroactively — standard practice in payment systems.

## Medium: Lock Retry Backoff Counted Against TTL

The lock acquisition retry loop itself consumes TTL before any work begins:

```python
LOCK_RETRY_ATTEMPTS = 3
# backoff on attempt 0: 0.5s, attempt 1: 1.0s — up to 1.5s before acquisition
```

This is minor given the larger TTL issue, but the worst-case held duration calculation above does not include this pre-acquisition time. Once the TTL is corrected, confirm the margin still holds after accounting for retry delays.

---

## Medium: `PaymentError` Not Caught in `process()`

`payment.charge()` raises `PaymentError` on non-retryable failures and after retry exhaustion. `process()` catches only `LockError`. An uncaught `PaymentError` will propagate out of `process()` as an unhandled exception, bypassing the `OrderResult` return contract the caller expects. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

---

## Medium: Timeout Not Set on Redis Operations

`redis.set()` and the Lua script release call have no explicit socket timeout configured. A hung Redis connection will block the worker indefinitely, holding the (already-expired) lock slot and consuming a thread. Configure `socket_timeout` and `socket_connect_timeout` on the Redis client at construction time.

---

## Low: Lock Retry Loop Sleeps on the Final Attempt

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)   # sleeps even on the last attempt before raising LockError
```

The `if attempt < self.MAX_RETRIES - 1` guard present in `PaymentClient` is absent here. The final failed attempt sleeps 1.5 seconds unnecessarily before raising `LockError`. Minor, but fix for correctness.

---

## Test Coverage Gaps

The test suite does not cover:

1. **The TTL race itself** — no test verifies behavior when the lock expires mid-operation.
2. **`PaymentError` propagation** — no test for gateway failure paths; the unhandled exception noted above would not be caught by the current suite.
3. **Lock release on exception** — no test verifies the `finally` block releases the lock when `payment.charge()` raises.
4. **Retry backoff timing** — `test_process_lock_contention` mocks `redis.set` to always return `False` but does not assert on the number of attempts or that backoff was applied.
