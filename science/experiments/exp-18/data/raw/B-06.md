Decision: Request Changes

Summary: The lock TTL is 60 seconds, but the worst-case duration of operations executed while the lock is held can reach **~135 seconds**, meaning the lock expires more than twice over before the critical section completes. A second worker will then acquire the lock, pass the idempotency check (the charge has not yet been recorded), and issue a duplicate charge. This is a critical correctness flaw in a payment system. Several secondary issues also need to be addressed before this is mergeable.

---

## Critical: Lock TTL vs. Worst-Case Held Duration

This is the primary rejection reason. Trace the full lifecycle:

**Lock TTL:** `ORDER_LOCK_TTL = 60` seconds

**Operations inside the lock:**

| Step | Worst-case time |
|---|---|
| `db.get_charge()` (network I/O, slow DB) | ~5 s |
| `payment.charge()` — attempt 1: timeout | 30 s |
| Backoff sleep between attempt 1 → 2 | 5 s |
| `payment.charge()` — attempt 2: timeout | 30 s |
| Backoff sleep between attempt 2 → 3 | 5 s |
| `payment.charge()` — attempt 3: timeout | 30 s |
| `db.record_charge()` + `db.mark_order_complete()` (slow DB) | ~5 s |
| **Total** | **~110–135 s** |

The lock expires at **t=60**. At **t=61**, a concurrent worker acquires the lock, calls `db.get_charge()`, finds nothing (the first worker has not yet committed), and proceeds to charge the customer a second time. Both workers eventually commit, producing a duplicate charge. The Lua-script release and atomic acquisition are both correct in isolation — the flaw is purely in the TTL-vs-duration math.

**Required fix:** The lock TTL must be set to a value strictly greater than the worst-case held duration, with margin. Given the numbers above, a minimum safe TTL is on the order of 150–180 seconds. Alternatively — and preferably — decouple the payment call from the lock:

1. Acquire lock.
2. Check idempotency.
3. Record an **in-flight sentinel** (e.g., `status=PENDING`) in the DB *within the lock*, then release the lock.
4. Call `payment.charge()` outside the lock.
5. Re-acquire lock (or use a DB-level compare-and-swap) to transition `PENDING → COMPLETE` and record the `charge_id`.

This bounds the held duration to two short DB writes rather than the full payment round-trip.

## High: Retry Backoff Is Applied After the Final Attempt

```python
# payment_client.py
if attempt < self.MAX_RETRIES - 1:
    time.sleep(self.RETRY_BACKOFF)
```

This is correct and does not sleep after the last attempt. However, the timeout and backoff values fed into the TTL analysis above are still accurate for attempts 0 and 1. No code change needed here, but the values must be accounted for in the TTL.

---

## High: `payment.charge()` Is Called Inside `db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

An external HTTP call with a 30-second timeout (and up to 3 retries) is being made while a database transaction is open. This holds a DB connection and potentially row-level locks for the full duration of the payment call. Under load this will exhaust the connection pool and cause cascading failures. The charge call must be moved outside the transaction:

```python
charge_id = self.payment.charge(...)   # outside transaction
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This does introduce a window where the charge succeeds but the DB write fails — which is exactly why the in-flight sentinel pattern described above is the correct long-term solution.

---

## Medium: Idempotency Check Does Not Cover the `PENDING` State

If a worker crashes after charging but before committing, `db.get_charge()` returns `None` on the next attempt, and the customer is charged again. The idempotency record must be written *before* the charge is attempted, not after. A `status` column (`PENDING` / `COMPLETE` / `FAILED`) on the charge record is the standard approach. The check should treat `PENDING` as "already in flight" and either return a retryable error or wait, not proceed.

## Medium: Lock Contention Retry Adds to Pre-Lock Latency, Not TTL, but the Backoff Is Misleading

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)
```

Worst-case pre-lock wait: `0.5 + 1.0 = 1.5 s`. This is outside the lock and does not affect TTL math, so it is not a correctness issue. However, three attempts with a maximum 1-second backoff means a worker gives up after 1.5 seconds of waiting on a lock that may be held for over two minutes. A caller will receive "Order currently being processed by another worker" almost immediately, which is a poor user experience and likely to trigger upstream retries that worsen contention. Consider a longer retry window or an async queue.

---

## Medium: `PaymentError` Is Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # can raise PaymentError
```

`PaymentError` propagates uncaught out of `process()`. The caller receives an unhandled exception rather than a structured `OrderResult`. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

## Low: Test Coverage Gaps

The test suite validates the happy path and a few negative cases, but does not cover:

- **Lock expiry during payment** — the core flaw identified above has no regression test.
- **`PaymentError` propagation** — no test asserts that a payment failure returns a structured result rather than raising.
- **`PENDING` idempotency** — no test for the crash-between-charge-and-record scenario.
- **`db.transaction()` rollback** — no test verifies that a DB failure after a successful charge does not leave a dangling committed charge with no record.
- **Lua script receives correct token** — `test_lock_release_uses_lua_script` only checks that the script was called; it does not assert the token argument matches the one used at acquisition.

---

## Low: Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs in source are an operational risk (environment promotion, secrets scanning false positives, inability to inject test doubles without monkeypatching). Move to configuration injection via environment variable or a settings object.

---

## Summary of Required Changes Before Approval

1. **Fix the TTL / held-duration race** — either raise `ORDER_LOCK_TTL` well above worst-case held duration, or restructure to hold the lock only around DB sentinel writes and not around the payment call.
2. **Move `payment.charge()` outside `db.transaction()`.**
3. **Add a `PENDING` sentinel written before the charge attempt** to close the crash-between-charge-and-record gap.
4. **Catch `PaymentError`** in `process()` and return a structured `OrderResult`.
5. **Add regression tests** for the scenarios listed above.
