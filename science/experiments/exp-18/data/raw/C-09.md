Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check inside the lock are all correct. However, there is a critical architectural flaw that breaks the at-most-once payment guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issue

### Payment charge occurs outside the database transaction — at-most-once guarantee is broken

This is the most important issue in the PR. The claim in the summary and docstring is:

> charges the payment method and records the transaction within a single atomic database transaction

But that is not what the code does. Look carefully:

```python
with self.db.transaction():
    charge_id = self.payment.charge(   # ← external network call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an external HTTP call to a payment gateway. Placing it inside a database transaction does not make it atomic with the database writes — it only holds the database connection and transaction open for the full duration of the network call (up to 30 seconds per the timeout, times 3 retries = up to 90 seconds). This creates two distinct failure scenarios:

**Scenario A — charge succeeds, DB write fails:**
The payment gateway charges the customer. The DB transaction then fails or rolls back. `record_charge` never persists. The idempotency check on retry finds no existing charge and charges the customer again. This is a double-charge.

**Scenario B — DB transaction held open during retries:**
Each payment retry holds the DB connection open across the `RETRY_BACKOFF = 5` second sleep. Under load this will exhaust the connection pool.

The correct pattern for this class of problem is:

```python
# 1. Charge first, outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Then persist atomically — if this fails, the lock prevents
#    a re-entry and the charge_id is lost, but no double-charge occurs.
#    A dead-letter / reconciliation job handles the orphaned charge.
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Even this pattern requires a reconciliation path for the case where the charge succeeds but the DB write fails. That is expected and acceptable; it is operationally recoverable. A double-charge is not.

## High Severity

### Timeout on payment gateway is dangerously long and interacts with lock TTL

```python
REQUEST_TIMEOUT = 30
MAX_RETRIES = 3
RETRY_BACKOFF = 5
ORDER_LOCK_TTL = 60
```

Worst-case payment client runtime: 3 timeouts × 30s + 2 sleeps × 5s = **100 seconds**. The Redis lock TTL is 60 seconds. The lock will expire before the payment client finishes, allowing a second worker to acquire the lock and begin processing the same order while the first worker is still mid-flight. The lock TTL must be greater than the maximum possible payment client runtime, or the payment client timeout and retry budget must be reduced to fit within the TTL with margin.

### `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError` is not caught in `process()`. If the payment client raises `PaymentError` after exhausting retries, the exception propagates out of `process()` entirely uncaught. The caller receives an unhandled exception rather than a structured `OrderResult`. This should be caught and returned as a failed `OrderResult` with appropriate logging.

## Medium Severity

### Lock retry backoff is blocking — `time.sleep` in a potentially async context

```python
time.sleep(backoff)
```

If this service is ever run under an async framework (FastAPI, asyncio workers), `time.sleep` will block the event loop. This should be noted in documentation at minimum, and `asyncio.sleep` considered if async support is anticipated.

### `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value may appear in logs. `payment_token` is a credential and must not be logged. Sanitize the error output or log only field names, not values.

### No test for `PaymentError` propagation

The test suite has no case covering what happens when `payment.charge` raises `PaymentError`. Given that this is an unhandled exception path (see above), this gap directly obscures the bug.

### No test for lock TTL expiry race condition

There is no test verifying behavior when the lock expires mid-processing. This scenario is currently unsafe (see lock TTL issue above) and should have explicit test coverage once the TTL math is corrected.

---

## Low Severity / Nits

### `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Endpoint should be injected via configuration or environment variable. Hardcoding it makes environment promotion (staging → production) error-prone.

### `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary characters up to 64 bytes. Given this is used in DB writes, it should have equivalent sanitization.

### Lock contention log level is `WARNING` on expected behavior

Lock contention during retry is operational noise, not a warning condition. Consider `INFO` for retry attempts and reserve `WARNING` for when all retries are exhausted.

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| Critical | Move `payment.charge()` outside `db.transaction()` and add reconciliation path |
| High | Fix lock TTL to exceed maximum payment client runtime |
| High | Catch `PaymentError` in `process()` and return structured `OrderResult` |
| Medium | Sanitize `ValidationError` log output to exclude `payment_token` value |
| Medium | Add tests for `PaymentError` and lock expiry race |
