Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical TOCTOU race condition where the payment charge happens *outside* the database transaction's atomicity guarantee in a way that can cause real double-charges, and there are several additional reliability and security issues that must be addressed before this handles financial transactions in production.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — double-charge risk

This is the most serious bug. The Redis lock prevents *concurrent* duplicate charges, but it does **not** protect against a crash between the charge succeeding and the DB write completing.

```python
# Current code — BROKEN
with self.db.transaction():
    charge_id = self.payment.charge(...)        # External side effect — cannot be rolled back
    self.db.record_charge(...)                  # If this throws, charge already happened
    self.db.mark_order_complete(...)
```

If `record_charge` or `mark_order_complete` raises after `payment.charge` succeeds, the DB transaction rolls back, the charge record is lost, and the next retry will charge the customer again. The lock has already been released by then.

**Required fix:** Charge *before* opening the transaction, then record inside it. More importantly, implement idempotent charge creation — pass a deterministic idempotency key to the payment gateway so that a retry of the same charge returns the existing `charge_id` rather than creating a new charge.

```python
# Safer pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # Gateway deduplicates on this
)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Without gateway-level idempotency keys, no amount of locking makes this truly safe.

---

### 2. `PaymentError` is silently swallowed — charge succeeds but order is never recorded

In `order_processor.py`, only `LockError` and `ValidationError` are caught. `PaymentError` and any DB exception propagate up as unhandled exceptions. That is actually acceptable for the payment failure case, but the real problem is that if the payment *succeeds* and then the DB write throws, the exception escapes with no record of the charge, no structured return, and no way for the caller to distinguish "payment failed" from "payment succeeded but recording failed." This needs explicit handling and logging.

```python
# These are not caught — caller gets a raw exception with no charge_id context
except PaymentError:
    ...  # missing
except Exception:
    ...  # missing — charge_id may be lost here
```

---

### 3. Lock TTL is dangerously short relative to retry timeout

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient.REQUEST_TIMEOUT = 30` and `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5`. Worst case payment time: `3 × 30 + 2 × 5 = 100 seconds`. The lock expires before the payment attempt completes, allowing a second worker to acquire the lock and begin processing the same order while the first is still mid-charge.

**Required fix:** Either increase `ORDER_LOCK_TTL` to at least 150 seconds with margin, or implement lock extension (Redlock-style keepalive), or reduce payment retry parameters to fit within the TTL.

---

## Significant Issues

### 4. `payment_token` logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., too long), the raw token value will appear in logs. Payment tokens must never appear in log output.

**Required fix:** Sanitize or suppress field values from validation error output before logging, or log only field names and error types.

```python
safe_errors = [{"field": e["loc"], "type": e["type"]} for e in exc.errors()]
logger.warning("Invalid order request fields=%s", safe_errors)
```

---

### 5. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected response body, this raises a `KeyError` or `JSONDecodeError` that is not caught, bypasses the retry logic, and surfaces as an unhandled exception. For a financial operation this is unacceptable.

```python
try:
    data = response.json()
    charge_id = data["charge_id"]
except (ValueError, KeyError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

---

### 6. Retry-on-timeout is unsafe without gateway idempotency keys

In `PaymentClient.charge`, a `Timeout` exception means the request *may have succeeded* on the gateway side — the response was simply not received. Retrying without an idempotency key can create duplicate charges at the gateway level, entirely bypassing the Redis lock protection.

This reinforces the fix in issue #1: every charge call must include a stable idempotency key.

---

### 7. `requests` has no connection-level timeout

```python
timeout=self.REQUEST_TIMEOUT  # This is only the read timeout
```

`requests` interprets a scalar timeout as the read timeout only. A connection that hangs during TCP handshake will block indefinitely. Use a tuple:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

---

## Minor Issues

### 8. Lock retry backoff is linear, not exponential

The PR description says "retry with backoff" but the implementation is linear (`0.5 * (attempt + 1)`). For a distributed system under contention, exponential backoff with jitter is strongly preferred to avoid thundering herd. This is minor but the description is misleading.

---

### 9. Test coverage gaps

- No test for the lock-expires-during-payment scenario (issue #3)
- No test for `PaymentError` propagation
- No test verifying `payment_token` does not appear in log output
- No test for malformed gateway response (issue #5)
- The `test_lock_release_uses_lua_script` test asserts the script was called but does not verify the correct `order_id`-derived key and token were passed — the assertion is too weak for a security-critical operation
