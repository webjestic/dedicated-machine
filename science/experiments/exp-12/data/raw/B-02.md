Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several critical correctness flaws and one significant security concern that must be resolved before this can be approved: the most serious being a payment-outside-transaction race that directly undermines the stated at-most-once guarantee, and a timeout-induced double-charge vulnerability in the payment client that is architecturally incompatible with the locking strategy.

---

## Critical Issues

### 1. Payment Call Is Outside the Database Transaction — At-Most-Once Guarantee Is Broken

**File:** `services/order_processor.py`, lines ~90–97

```python
with self.db.transaction():
    charge_id = self.payment.charge(      # ← external side-effect INSIDE db tx
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most critical flaw. The PR comment claims *"crash between charge and record cannot produce an unrecorded charge"* because the charge is inside the transaction. That reasoning is incorrect and the guarantee does not hold.

**Why it fails:**

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an external HTTP call to a payment gateway. The database engine has no knowledge of it and cannot roll it back. If the process crashes, loses network connectivity, or the database commit fails *after* `payment.charge()` returns a `charge_id` but *before* the transaction commits, the money has moved but no record exists. The Redis lock has also been released by the `finally` block, so a retry will pass the idempotency check and charge again.

**The correct pattern** is to separate the concerns:

```
1. Acquire lock
2. Idempotency check (read)
3. Create a PENDING charge record in the DB (within a transaction)
4. Perform the external payment call
5. Update the record to COMPLETED/FAILED (within a transaction)
6. Release lock
```

This way, a crash after step 3 but before step 5 leaves a PENDING record. On retry, the idempotency check finds the PENDING record and can either resume or surface it for reconciliation. No double-charge is possible.

### 2. Timeout Retry in Payment Client Creates Double-Charge Window

**File:** `services/payment_client.py`, lines ~30–45

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry loop
```

A network timeout means the request *may have succeeded* at the gateway — the response was simply lost in transit. Retrying unconditionally on timeout will submit a second charge for an order that may have already been charged. This is a direct financial loss vector.

**Required fix:** The payment gateway must support idempotency keys. Pass a stable, per-order idempotency key (e.g., derived from `order_id`) on every attempt so the gateway deduplicates on its side:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # ← required
    timeout=self.REQUEST_TIMEOUT,
)
```

If the gateway does not support idempotency keys, retrying on timeout must be removed entirely and the timeout treated as a permanent failure requiring out-of-band reconciliation.

### 3. Lock TTL Is Insufficient to Cover Payment Retry Duration

**File:** `services/order_processor.py`, line 14 and `services/payment_client.py`, lines 10–12

```python
ORDER_LOCK_TTL = 60   # order_processor.py

REQUEST_TIMEOUT = 30  # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still in its third payment retry, pass the idempotency check (no record exists yet), and issue a duplicate charge. The lock TTL must exceed the maximum possible critical-section duration with margin. Either raise `ORDER_LOCK_TTL` to at minimum `120`–`150` seconds, or reduce the payment client's maximum wall time to well under 60 seconds. Both values must be derived from a shared constant or configuration to prevent them from drifting apart again.

## Security Issue

### 4. Payment Token Logged on Validation Failure

**File:** `services/order_processor.py`, lines ~72–74

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value will appear in the log line. Payment tokens are credential-equivalent material and must never appear in logs.

**Fix:**

```python
except ValidationError as e:
    # Log field names and error types only — never field values
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
    return OrderResult(processed=False, message="Invalid order parameters")
```

## Moderate Issues

### 5. Unhandled Exceptions Inside the Lock Are Silently Swallowed

**File:** `services/order_processor.py`, lines ~80–115

`PaymentError`, database exceptions, and any unexpected runtime errors raised inside `with self._acquire_lock(...)` are not caught. They will propagate out of `process()` as unhandled exceptions. The `finally` block in `_acquire_lock` will correctly release the lock, but the caller receives an exception rather than an `OrderResult`. The method signature promises `OrderResult` — callers depending on that contract will break.

Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 6. Lock Retry Backoff Is Linear, Not Exponential

**File:** `services/order_processor.py`, lines ~47–53

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — linear
```

The docstring and PR checklist both claim exponential backoff. This is linear. Under high contention, linear backoff provides minimal thundering-herd relief. Use `0.5 * (2 ** attempt)` for true exponential backoff, and add jitter.

### 7. Test for Lua Script Release Is Fragile

**File:** `tests/test_order_processor.py`, lines ~68–72

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test accesses `processor._release_script` directly, which is a private attribute. More importantly, it does not verify that the correct `lock_key` and `token` were passed to the script. A call with wrong arguments would pass this test. Assert on the call arguments.
