# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Outside Database Transaction — At-Most-Once Guarantee is False

The PR summary and inline comment both claim:

> *"crash between charge and record cannot produce an unrecorded charge"*

This is **incorrect**. The `payment.charge()` call is placed **inside** `with self.db.transaction()`, but a database transaction provides atomicity over *database operations only*. The payment gateway call is a network side effect that is **not part of the database transaction's rollback domain**.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call; NOT rolled back on DB failure
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Consider the following crash sequence:

1. `payment.charge()` succeeds — money is debited from the customer.
2. `self.db.record_charge()` raises an exception (deadlock, constraint violation, connection drop).
3. The database transaction rolls back.
4. The charge is **permanently unrecorded**. The customer has been charged with no order record.

The correct pattern for at-most-once semantics is:

1. **Persist intent first** — write a `PENDING` charge record inside a DB transaction *before* calling the gateway.
2. **Then call the gateway** — outside or after the DB write is committed.
3. **Confirm/reconcile** — update the record to `COMPLETED` with the returned `charge_id`, or use a background reconciliation job for the gap.

Alternatively, the `charge_id` must be obtained *before* the transaction opens, and the transaction must only contain idempotent DB writes. Either way, the current structure provides **no protection** against the unrecorded-charge failure mode it explicitly claims to prevent.

### 🔴 Critical: Redis Lock Does Not Protect Against the Actual Race Condition

The distributed lock prevents two workers from *entering* the critical section simultaneously, but the idempotency check and the payment are **not atomic with respect to the lock boundary**:

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # reads committed data
    if existing:
        return ...                                  # safe path
    # ← lock held here, but the DB transaction below can still fail
    #   leaving no record, and the lock expires or is released
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

If the lock TTL (`ORDER_LOCK_TTL = 60s`) expires during a slow payment gateway call (which has `REQUEST_TIMEOUT = 30s` × `MAX_RETRIES = 3` = up to **~105 seconds** of blocking time including backoff), a second worker can acquire the lock, pass the idempotency check (no record exists yet), and issue a duplicate charge. The lock TTL is **shorter than the maximum payment client execution time**.

```
Max payment client wall time:
  3 attempts × 30s timeout + 2 × 5s backoff = 100s > 60s TTL
```

This must be resolved. Options include:

- Extending the TTL to safely exceed the maximum payment client wall time with margin.
- Implementing lock extension (Redlock-style watchdog/renewal) during the payment call.
- Writing a `PENDING` record to the DB *before* releasing the lock, so the idempotency check catches it even if the lock has expired.

### 🟡 Significant: `PaymentError` Is Silently Swallowed

`payment.charge()` raises `PaymentError` on permanent decline or retry exhaustion. This exception is **not caught** in `process()`. It will propagate up to the caller as an unhandled exception, bypassing the structured `OrderResult` return contract. The lock will still be released (the `finally` block is correct), but the caller receives a raw exception rather than a typed result. This should either be caught and mapped to an `OrderResult`, or the method signature should be documented as raising `PaymentError`.

---

### 🟡 Significant: `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials. The log message should either suppress field values or explicitly redact the `payment_token` field before logging.

---

### 🟡 Significant: No Authentication on Payment Gateway Request

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no API credentials, bearer tokens, HMAC signatures, or mTLS configuration present. Even for an internal endpoint, unauthenticated payment calls are a material security gap. Any process on the internal network could replay or forge requests to this endpoint.

---

### 🟡 Significant: `requests` Has No Connection Timeout — Only Read Timeout

`timeout=self.REQUEST_TIMEOUT` passes a single scalar to `requests.post`, which in the `requests` library applies only to the **read timeout**, not the connection establishment timeout. A hung TCP handshake to a misconfigured or unreachable host will block indefinitely. This should be:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

---

### 🟠 Moderate: Lock Retry Backoff Is Linear and Unbound for Contention Scenarios

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

With only 3 attempts and linear backoff, a worker under high contention will fail fast and return an error to the caller. This may be intentional, but the total wait time (3 seconds) is very short relative to the 60-second lock TTL. A caller retrying at the application layer could produce a thundering herd. Consider exponential backoff with jitter, or document that callers are expected to retry externally.

---

### 🟠 Moderate: `order_id` Not Included in `OrderResult` on Successful Idempotency Short-Circuit

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,   # ← present here, correct
        message="Order already processed",
    )
```

This is actually correct, but the `charge_id` from `existing` is not returned. A caller receiving `processed=False` with `"already processed"` has no way to retrieve the original `charge_id` without a separate lookup. This is an API ergonomics issue that could lead to callers incorrectly treating a successful idempotent replay as a failure.

---

### 🟢 What Is Done Well

- `SET NX EX` is atomic — correct acquisition pattern.
- Lua script for token-verified release is correct and necessary.
- `uuid4` token per lock acquisition is appropriate.
- `finally` block guarantees lock release on exception.
- Pydantic pattern enforcement on `order_id` blocks path traversal and injection attempts.
- `nx=True` prevents overwrite of an existing lock.
- Structured logging with contextual fields throughout.
- `PaymentError` correctly distinguishes permanent (4xx) from transient (5xx/timeout) failures.

---

## Test Coverage Gaps

- No test for `PaymentError` propagation from `process()`.
- No test for lock TTL expiry during a slow payment call (the core race condition identified above).
- No test for DB transaction rollback after a successful `payment.charge()` call.
- `test_lock_release_uses_lua_script` verifies the script is called but does not assert the correct `keys` and `args` are passed — a regression could call it with wrong arguments and this test would still pass.

---

## Decision: Request Changes

**Summary:** The PR contains two critical correctness flaws that directly contradict its stated safety guarantees: the payment gateway call is not protected by the database transaction it is nested inside, meaning a DB failure after a successful charge produces an unrecorded debit; and the lock TTL (60s) is shorter than the maximum payment client execution time (~100s), allowing a second worker to acquire the lock and issue a duplicate charge before the first worker's DB record is committed. These must be resolved before merge. Additionally, `PaymentError` is unhandled in the public `process()` method, breaking the structured result contract, and the payment client sends unauthenticated requests to the charge endpoint.
