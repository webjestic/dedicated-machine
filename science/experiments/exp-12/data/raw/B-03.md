Decision: Request Changes

Summary: The PR demonstrates solid fundamentals — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several critical architectural flaws that create real double-charge risk and operational blind spots that must be resolved before this handles production payment traffic. The issues range from a classic TOCTOU race on the idempotency check to unhandled exceptions that silently swallow payment state.

---

## Critical Issues

### 1. Payment Outside the Database Transaction — Charge Can Be Lost

This is the most serious flaw. The comment in the code claims crash safety, but the implementation does not deliver it.

```python
# CURRENT — BROKEN
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external side effect INSIDE tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is a network call to an external payment gateway. Database transactions do not wrap external HTTP calls. If the process crashes, the network drops, or the DB commit fails *after* the charge succeeds, you have charged the customer with no record. The charge is gone. The idempotency check will not find it on retry, so the next attempt charges again.

**Required fix:** Charge outside and before the transaction. Record inside. Handle the case where charge succeeds but record fails.

```python
# Charge first, outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically — if this fails, charge_id is logged and can be reconciled
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "CHARGE_UNRECORDED order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

This does not fully solve the two-generals problem, but it makes the failure mode visible and recoverable rather than silent and duplicating.

### 2. TOCTOU Race on Idempotency Check

The idempotency check reads from the database *after* acquiring the lock. This is correct in the single-Redis-instance model. However, the lock TTL is 60 seconds and the payment client can take up to `30s × 3 attempts + 10s backoff = 100 seconds` to exhaust retries. The lock can expire while the payment is in flight.

```
Worker A: acquires lock, starts payment (slow gateway)
          lock expires at t=60s
Worker B: acquires lock (A still charging), idempotency check passes (no record yet)
Worker B: charges — DUPLICATE
Worker A: charge completes, records
Worker B: charge completes, records — DB constraint may catch this, but charge already fired
```

**Required fixes:**
- `ORDER_LOCK_TTL` must be strictly greater than the maximum possible payment duration. With `MAX_RETRIES=3`, `REQUEST_TIMEOUT=30`, and `RETRY_BACKOFF=5`, worst-case is `(30+5)×3 = 105s`. TTL of 60s is insufficient.
- Add lock extension (Redlock watchdog pattern) or reduce payment client max duration to fit within TTL with margin.
- The database `record_charge` should have a unique constraint on `order_id` as a hard backstop. A DB-level unique constraint is the only guarantee that survives lock expiry.

---

### 3. Unhandled `PaymentError` and Generic Exceptions Escape `process()`

`PaymentError` and any unexpected exception from `self.db` are not caught in `process()`. They propagate to the caller with no structured response and no log context. In a web handler this becomes a 500 with no audit trail of what order was being processed.

```python
# Nothing catches this:
charge_id = self.payment.charge(...)   # raises PaymentError
# or
self.db.record_charge(...)             # raises DB exception
```

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    raise  # re-raise — don't swallow unknown failures silently
```

## Significant Issues

### 4. Redis Failure Mode Is Undefined

If `self.redis.set()` raises a `redis.RedisError` (connection refused, timeout, cluster failover), the exception propagates uncaught through `process()`. Depending on your deployment, this could mean all orders fail open or fail closed with no structured response.

Decide and enforce a policy explicitly:

```python
except redis.RedisError as e:
    logger.error("Redis unavailable order_id=%s error=%s", req.order_id, e)
    # Policy decision: fail closed on payment operations
    return OrderResult(processed=False, message="Service temporarily unavailable")
```

---

### 5. Lock Retry Backoff Does Not Jitter — Thundering Herd

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — deterministic
```

Under burst load, all workers contending on the same order retry at identical intervals and collide repeatedly. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

---

### 6. `payment_token` Logged Indirectly via `payment_token` Field

`payment_token` is accepted as a Pydantic field with `max_length=256` and no format validation. Depending on your payment provider, this token may be sensitive. Verify it is never interpolated into log messages. A `__repr__` override or a custom Pydantic type that masks the value in string output is advisable.

---

### 7. Test Coverage Gaps

The tests validate the happy path and basic contention, but miss the failure modes introduced above:

- No test for `PaymentError` during charge — what does `process()` return?
- No test for lock expiry during a slow payment (TTL race)
- No test for Redis connection failure
- No test for DB transaction failure after successful charge (the critical unrecorded-charge path)
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, but does not assert the Lua script was called with the *correct token* — the core security property of token-verified release is untested

---

## Minor Issues

### 8. `ORDER_LOCK_TTL` and `MAX_RETRIES` Are Magic Numbers Without Coordination

`ORDER_LOCK_TTL = 60` in `OrderProcessor` and `MAX_RETRIES = 3`, `REQUEST_TIMEOUT = 30`, `RETRY_BACKOFF = 5` in `PaymentClient` are defined independently with no enforced relationship. As noted above, they are currently inconsistent. These should either be injected via config with a validation assertion at startup, or co-located with a comment that makes the dependency explicit.

```python
# At startup or in __init__:
max_payment_duration = (
    PaymentClient.REQUEST_TIMEOUT + PaymentClient.RETRY_BACKOFF
) * PaymentClient.MAX_RETRIES
assert OrderProcessor.ORDER_LOCK_TTL > max_payment_duration + 15, (
    "Lock TTL must exceed max payment duration with margin"
)
```

Items 1, 2, and 3 are blocking. The architecture as written does not deliver the at-most-once guarantee it claims.
