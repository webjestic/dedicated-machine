Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness flaws that would allow double-charges in production under realistic failure conditions, a timing vulnerability that undermines the lock's safety guarantee, and a dangerous retry pattern in the payment client that can cause the exact duplicate charges this system is designed to prevent. None of these are edge cases; they are reachable under normal distributed failure modes.

---

## Critical Issues

### 1. Payment charge occurs inside `db.transaction()` — this is the most dangerous flaw

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)        # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations only. `self.payment.charge()` is an external HTTP call to a payment gateway. If the charge succeeds but the subsequent `db.record_charge()` raises an exception, the database transaction rolls back — but the charge has already been made and cannot be rolled back. The next worker to acquire the lock will find no existing charge record and will charge again.

The correct pattern is to perform the external charge **before** opening the transaction, then record the result inside the transaction:

```python
# Outside transaction — external side effect first
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Inside transaction — record what already happened
with self.db.transaction():
    # Re-check idempotency inside the transaction to close the TOCTOU window
    if self.db.get_charge(req.order_id):
        # Charge already recorded by a concurrent worker — handle the orphaned charge
        self.payment.refund(charge_id)
        return OrderResult(processed=False, order_id=req.order_id, message="Order already processed")
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This also requires a compensating refund path for the race where two workers both charge before either records. That path is currently entirely absent.

### 2. Lock TTL vs. total payment client timeout — the lock can expire while the charge is in-flight

```python
ORDER_LOCK_TTL = 60        # seconds
```

```python
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client wall time: `(30 × 3) + (5 × 2) = 100 seconds`.

The lock TTL is 60 seconds. A second worker can acquire the lock at second 61, pass the idempotency check (no charge recorded yet because the first worker is still waiting on a retry), and issue a second charge. The lock provides no protection in this scenario.

Fix: either extend `ORDER_LOCK_TTL` to comfortably exceed the maximum payment client wall time with margin (≥ 120 seconds), or implement lock heartbeat/renewal during the payment call. The TTL must be a deliberate function of the downstream timeout budget, not an independent constant.

---

### 3. Payment client retries on timeout are unsafe without gateway-side idempotency keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded at the gateway but the response was lost in transit. Retrying without an idempotency key will issue a second charge request that the gateway may process as a new charge. This is a standard distributed systems hazard.

Fix: generate a per-`charge()` call idempotency key (e.g., derived from `order_id`) and pass it as a header on every attempt:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `order_id` is the natural idempotency key here and should be threaded through from `OrderProcessor.process()` to `PaymentClient.charge()`.

### 4. TOCTOU window between idempotency check and charge

```python
existing = self.db.get_charge(req.order_id)   # read
if existing:
    return ...
# ... time passes, lock could expire here ...
charge_id = self.payment.charge(...)           # write
```

The idempotency check and the charge are not atomic. If the lock expires between the check and the charge (see issue 2), a second worker can pass the same check and both proceed to charge. The idempotency check must be re-evaluated inside the database transaction after the charge completes, as shown in the fix for issue 1.

## Moderate Issues

### 5. Unhandled exceptions leak through `process()` without releasing meaningful state

`PaymentError` and any database exception raised inside `_acquire_lock`'s `try` block are not caught in `process()`. They will propagate as unhandled exceptions to the caller. The lock will be released by the `finally` block, which is correct, but the caller receives an exception rather than an `OrderResult`. The method signature promises `OrderResult` in all cases. Either catch these explicitly and return structured results, or document the exception contract clearly and adjust the return type annotation.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

### 6. Lock retry backoff is blocking — ties up a worker thread

```python
time.sleep(backoff)   # 0.5s, 1.0s, 1.5s
```

In a synchronous worker pool this is acceptable, but it should be noted that this blocks the thread entirely. If the codebase moves toward async workers, this must become `await asyncio.sleep(backoff)`. Flag this for the team as a future migration concern.

---

### 7. Test suite does not cover the critical failure paths

The tests cover the happy path and basic contention, but are missing:

- Lock expires mid-payment (TTL race)
- Payment succeeds but `db.record_charge()` raises — verifies no double-charge
- `PaymentError` propagation behavior
- Timeout retry with idempotency key presence
- The compensating refund path (once added)

The existing tests pass because they mock away all the interesting failure modes. Tests should be written to verify the system's behavior under the failure conditions that motivated the design.
