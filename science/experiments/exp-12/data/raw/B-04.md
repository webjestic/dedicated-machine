Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness flaws and one significant security concern that must be resolved before this handles real payment traffic. The most severe issue is a charge/record atomicity gap that directly contradicts the PR's own safety guarantee.

---

## Critical Issues

### 1. Payment Charge Occurs Inside the Database Transaction — Atomicity is Illusory

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR claims: *"crash between charge and record cannot produce an unrecorded charge."* This is **false**.

`self.payment.charge()` is an external HTTP call to a payment gateway. Database transactions do not span external services. If `payment.charge()` succeeds but the process crashes before `record_charge()` commits, or if the DB transaction rolls back for any reason after a successful charge, you have a real charge with no local record. The DB transaction wrapping an external call provides no atomicity guarantee whatsoever — it only holds a DB connection open unnecessarily during a 30-second potential network wait, increasing connection pool pressure under load.

**Required fix:** Charge outside and before the DB transaction. Use the returned `charge_id` as the idempotency anchor. Record first in a `PENDING` state if needed, then confirm.

```python
# Correct structure:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only open the DB transaction after the external call succeeds
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even this requires the payment gateway to support idempotency keys on retries, which leads to issue #2.

### 2. Payment Retry on Timeout Creates Double-Charge Risk

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries unconditionally
```

A timeout means the request outcome is **unknown** — the gateway may have processed the charge successfully and the response was lost in transit. Retrying unconditionally on timeout without an idempotency key sent to the gateway is a double-charge vector. This is a well-known failure mode in distributed payment systems.

**Required fix:** Generate a stable idempotency key per logical charge attempt (e.g., derived from `order_id`) and pass it to the gateway on every retry. The gateway deduplicates on its end.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 3. Redis Lock TTL Is Insufficient Under Retry + Timeout Conditions

**File:** `services/order_processor.py` and `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. A slow-path payment attempt will outlive the lock. When the lock expires, a second worker can acquire it, pass the idempotency check (charge not yet recorded), and issue a second charge — exactly the race condition this system is designed to prevent.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed maximum payment client wall time (≥ 120s with margin), or implement lock heartbeat/extension. At minimum, enforce the constraint in code or configuration with a comment making the dependency explicit.

### 4. `PaymentError` Is Unhandled in `process()` — Uncontrolled Exception Propagation

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # raises PaymentError — not caught
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller receives no `OrderResult` — they receive a raw exception. This is inconsistent with the method's declared return type and contract. Depending on the caller (e.g., an HTTP handler), this may result in a 500 with stack trace exposure.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

## Security Concern

### 5. `payment_token` Reflected Into Logs on Validation Failure

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions and configurations. If `payment_token` fails validation, its value (a sensitive credential) may appear in log output. Payment tokens must never appear in logs.

**Required fix:** Log only field names and error types, not values. Explicitly scrub sensitive fields:

```python
safe_errors = [
    {"field": e["loc"], "type": e["type"]}
    for e in exc.errors()
]
logger.warning("Invalid order request fields=%s", safe_errors)
```

## Minor Issues

### 6. Lock Retry Backoff Does Not Account for Lock TTL Semantics

Three retries at 0.5s, 1.0s, 1.5s total ≈ 3 seconds of wait. If the competing worker is mid-payment (potentially 30–90 seconds), all retries exhaust immediately and the caller gets a contention failure. This is a UX/reliability concern rather than a correctness bug, but the retry window should be sized relative to expected lock hold time or documented as intentionally short-circuit.

### 7. Test Coverage Gaps

- No test for `PaymentError` propagation from `process()` (which is currently a bug, per issue #4)
- No test for lock TTL expiry race (lock expires mid-payment, second worker proceeds)
- No test for timeout-then-retry in `PaymentClient` verifying idempotency key behavior
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called but does not assert the Lua script was called with the correct token — the token verification property is untested

## What Is Done Well

- `SET NX EX` is atomic and correct — no SETNX/EXPIRE race
- Lua script for token-verified release is the right approach and correctly implemented
- `order_id` pattern validation prevents injection into Redis key namespace
- `LockError` is properly scoped and caught
- Structured logging with contextual fields throughout
- `requests_exceptions.HTTPError` correctly distinguishes 4xx (permanent) from 5xx (transient)
