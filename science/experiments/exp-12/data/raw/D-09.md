Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the idempotency check pattern are all correct. However, there is a critical atomicity flaw at the intersection of the payment call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data safely.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* but that claim is false. A database transaction does not wrap a network call to an external payment gateway. What `with self.db.transaction()` actually provides is atomicity across the database writes inside it. The `self.payment.charge(...)` call goes out over the network *before* `record_charge` is committed. If the process crashes, the Redis lock TTL expires, or the DB transaction rolls back after a successful charge response, you have charged the customer with no record of it.

```python
# Current — charge is NOT protected by the DB transaction
with self.db.transaction():
    charge_id = self.payment.charge(...)   # network call — not transactional
    self.db.record_charge(...)             # only this and below are atomic
    self.db.mark_order_complete(...)
```

The correct pattern for at-most-once payment is:

```python
# 1. Charge the gateway OUTSIDE the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record atomically — if this fails, you have a charge with no record,
#    which your reconciliation process must handle, but you will NOT
#    double-charge on retry because the idempotency check runs first
#    and the lock prevents concurrent workers
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

You still need a reconciliation path for the crash-after-charge-before-record window, but at least the semantics are honest. The current code gives a false sense of safety.

### 2. Timeout on payment charge can cause a double-charge

`PaymentClient` retries on `requests.exceptions.Timeout`. A timeout means the request *may have succeeded* on the gateway side — the response was simply lost in transit. Retrying a timed-out charge without a gateway-level idempotency key will issue a second charge.

```python
# payment_client.py — retrying a timeout is dangerous without idempotency
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry — may charge twice
```

The fix is to pass a stable idempotency key derived from `order_id` in the request headers, so the gateway deduplicates on its side:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # e.g. order_id
    timeout=self.REQUEST_TIMEOUT,
)
```

The `order_id` must flow from `OrderProcessor` into `PaymentClient.charge()` for this to work. The current signature `charge(token, amount_cents)` has no idempotency key parameter.

## Significant Issues

### 3. Unhandled `PaymentError` propagates through the lock context manager

If `self.payment.charge(...)` raises `PaymentError`, it is not caught anywhere in `process()`. The exception will propagate up through `_acquire_lock`'s `finally` block (which correctly releases the lock), but then bubble out of `process()` entirely, giving the caller an unhandled exception instead of an `OrderResult`. The method signature promises `OrderResult` — it should return one.

```python
# Add to process():
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

### 4. `payment_token` is logged at INFO level on success

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` appears in the `OrderRequest` model and is passed around freely. Verify it never enters a log statement. Payment tokens are sensitive — if any future log line interpolates the full request object, you will be writing PCI-relevant data to log files. Consider marking the field with a custom `__repr__`/`__str__` that masks it, or use a dedicated secrets type.

### 5. Lock TTL of 60 seconds is too short given retry configuration

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire before the payment client finishes, allowing a second worker to acquire the lock and pass the idempotency check while the first worker is still mid-flight. Either increase the TTL to at least 120 seconds with margin, or restructure so the lock TTL is derived from the payment client's maximum possible duration.

### 6. `requests` connection errors are not retried

`requests.exceptions.ConnectionError` (DNS failure, TCP reset, etc.) is not caught in `PaymentClient.charge()`. It will propagate immediately without retry, which is inconsistent with the stated intent of retrying transient failures. Add it alongside `Timeout`.

## Minor Issues

### 7. Test for lock release does not verify the token

`test_lock_release_uses_lua_script` asserts the Lua script was called but does not assert it was called with the correct token and key. A regression that passes the wrong args would not be caught.

### 8. No test for `PaymentError` handling

There is no test covering what happens when `payment.charge` raises. Given issue #3 above, this is currently an unhandled exception path.

### 9. `customer_id` has no pattern validation

`order_id` has a strict alphanumeric pattern. `customer_id` does not. If `customer_id` is used in any downstream query construction, this inconsistency is a risk. Apply the same or equivalent validation.
