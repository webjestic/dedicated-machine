Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several serious correctness and security flaws that must be resolved before this can be approved, primarily around the charge-inside-transaction anti-pattern, timeout-induced double-charge exposure, missing exception handling, and a lock TTL that is dangerously short relative to the retry window.

---

## Critical Findings

### 1. External Side Effect Inside Database Transaction — Double-Charge Risk

**File:** `services/order_processor.py`, lines within `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious flaw in the PR. The comment claims this provides atomicity against "crash between charge and record," but it does not — and it introduces a worse failure mode.

**The actual failure modes:**

- If `payment.charge()` succeeds but the database transaction rolls back (deadlock, constraint violation, connection drop), the charge is real and unrecorded. The idempotency check will never find it. The next retry will charge again. **This is exactly the double-charge scenario the PR claims to prevent.**
- Database transactions hold locks for their entire duration. Placing a 30-second network call (with up to 3 retries × 5s backoff = 75 seconds worst case) inside a transaction holds those locks for that entire window, creating contention and potential deadlock cascades under load.
- The `payment.charge()` call is not transactional. You cannot roll back a completed charge by rolling back a DB transaction.

**Required fix:** Perform the charge *outside* the transaction. Use a two-phase write with a pending state:

```python
# Phase 1: Record intent (within transaction, short-lived)
with self.db.transaction():
    self.db.create_charge_record(req.order_id, req.customer_id,
                                  req.amount_cents, status="pending")

# Phase 2: External call (outside transaction, no DB locks held)
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Phase 3: Finalize (within transaction)
with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

A reconciliation job should sweep `pending` records older than a threshold and resolve them against the payment gateway. This is the standard pattern for at-most-once payment guarantees.

### 2. Lock TTL vs. Retry Window — Lock Expiry During Active Processing

**File:** `services/order_processor.py` and `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60        # order_processor.py
MAX_RETRIES = 3            # payment_client.py
RETRY_BACKOFF = 5          # payment_client.py
REQUEST_TIMEOUT = 30       # payment_client.py
```

Worst-case payment client execution time:

```
3 attempts × 30s timeout + 2 inter-retry sleeps × 5s = 100 seconds
```

The Redis lock TTL is 60 seconds. The lock **will expire** before the payment client exhausts its retries. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The distributed lock provides no protection in this scenario.

**Required fix:** Either extend `ORDER_LOCK_TTL` to safely exceed the worst-case payment duration (≥120s with margin), or implement lock extension (Redlock-style `PEXPIRE` refresh) during long operations. The TTL and client timeout/retry parameters must be defined together with an explicit invariant enforced in tests:

```python
assert ORDER_LOCK_TTL > (MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF)
```

### 3. Timeout Response Is Retried — Idempotency Violation

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge failed. It means the client did not receive a response within the timeout window. The gateway may have processed the charge successfully. Retrying on timeout without a gateway-side idempotency key will submit a second charge attempt for a potentially already-completed charge.

**Required fix:** Generate a per-order idempotency key and pass it in the request header. Most payment gateways (Stripe, Braintree, etc.) support `Idempotency-Key`. On timeout, the retry with the same key will either return the original result or safely deduplicate:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    headers = {"Idempotency-Key": idempotency_key}
    response = requests.post(..., headers=headers, ...)
```

The `idempotency_key` should be derived from `order_id` and passed through from `OrderProcessor`.

### 4. Unhandled Exceptions Escape `process()` — Lock Leak and Caller Surprise

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

Only `LockError` and `ValidationError` are caught. Any other exception — `PaymentError`, database exceptions, `redis.RedisError`, `KeyError` on `response.json()["charge_id"]` — propagates uncaught to the caller. The `_acquire_lock` context manager's `finally` block will release the lock, which is correct, but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the stated contract of the method signature.

Additionally, if `self.redis.set()` itself raises (Redis unavailable), the exception propagates before `acquired` is set, which is fine, but there is no structured handling — the caller gets a raw `redis.RedisError`.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

## Significant Findings

### 5. `payment_token` Logged Indirectly — PCI-DSS Exposure Risk

**File:** `services/order_processor.py`

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their default string representation. If `payment_token` fails validation, its value may appear in the log output. Payment tokens are sensitive credentials. Logging them violates PCI-DSS requirements and creates a credential exposure surface in log aggregation systems.

**Required fix:** Sanitize validation error output before logging. Log field names and error types only, never values for sensitive fields:

```python
safe_errors = [{"field": e["loc"], "type": e["type"]} for e in exc.errors()]
logger.warning("Invalid order request: %s", safe_errors)
```

---

### 6. `response.json()["charge_id"]` — Unguarded Key Access

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 2xx response with a malformed or unexpected body (missing `charge_id`, non-JSON content type), this raises `KeyError` or `requests.exceptions.JSONDecodeError`. These exceptions are not caught and will propagate as unexpected types through `OrderProcessor.process()`, bypassing the `PaymentError` handler.

**Required fix:**

```python
try:
    body = response.json()
    charge_id = body["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

---

### 7. Lock Retry Backoff Holds Thread — Scalability Concern

**File:** `services/order_processor.py`

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

Synchronous sleep in lock retry blocks the worker thread. Under high contention with many workers, this creates thread starvation. In an async context this would be a blocking call on the event loop. This is a design-level concern rather than a correctness bug, but it should be noted for the deployment context.

---

### 8. Test Coverage Gaps

**File:** `tests/test_order_processor.py`

The following scenarios are not tested and represent the highest-risk paths:

| Missing Test | Risk |
|---|---|
| `PaymentError` raised during charge | Verifies lock is released and `OrderResult` is returned, not exception propagated |
| Lock TTL expiry during payment | Verifies the race condition in Finding #2 is caught |
| Database transaction rollback after successful charge | Verifies the double-charge scenario in Finding #1 |
| Timeout on payment with retry | Verifies idempotency key is consistent across retries |
| Redis unavailable at lock acquisition | Verifies structured failure, not raw exception |
| Malformed gateway response (missing `charge_id`) | Verifies Finding #6 |

The existing tests mock at too high a level to catch the interaction bugs identified above.
