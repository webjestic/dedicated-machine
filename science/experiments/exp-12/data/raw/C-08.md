Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic NX/EX acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several critical flaws that would produce exactly the failure modes this code claims to prevent: a timeout arithmetic gap that creates a live double-charge window, a charge-outside-transaction bug that makes the "atomic" guarantee false, and a retry loop in the payment client that will fire on ambiguous timeout responses and produce duplicate charges with no deduplication anchor passed to the gateway. These are not edge cases; they are the primary failure paths in a payment processor.

---

## Critical Issues

### 1. Charge Occurs Outside the Database Transaction — The "Atomic" Guarantee Is False

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
# What the code does:
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← network call to external system
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations. It provides **zero atomicity** over an external network call. The `payment.charge()` call crosses a trust boundary to an external payment gateway that has no knowledge of your database transaction. Three failure scenarios that all produce unrecorded charges:

- **Scenario A:** `payment.charge()` succeeds and returns `charge_id`. The process crashes before `record_charge()`. The DB transaction rolls back. The charge exists at the gateway. No record exists locally. Money is taken, order is unfulfilled.
- **Scenario B:** `payment.charge()` succeeds. `record_charge()` succeeds. `mark_order_complete()` raises. The DB rolls back. Same outcome as Scenario A.
- **Scenario C:** The gateway times out. `PaymentError` is raised. The DB transaction rolls back. But the charge may have landed — the gateway processed it before the TCP timeout fired.

The comment in the PR summary — *"crash between charge and record cannot produce an unrecorded charge"* — is **factually incorrect**. This is the most dangerous misunderstanding in the PR because it will give operators false confidence.

**Required fix:** Move `payment.charge()` outside and before the database transaction. The correct pattern for external payment + local record is:

```python
# Charge first, outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # see Issue #3
)

# Then record durably in DB
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

If `record_charge` fails after a successful charge, the retry path will hit the idempotency check at the gateway (via `idempotency_key`) and return the same `charge_id` rather than charging again.

### 2. Timeout Arithmetic Creates a Live Double-Charge Window

**Files:** `order_processor.py` and `payment_client.py`

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Maximum time the payment client can consume:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100 seconds
```

The Redis lock TTL is **60 seconds**. The payment client can run for **100 seconds**. This means:

1. Worker A acquires the lock (TTL = 60s).
2. Worker A's payment call stalls — gateway is slow.
3. At T=60s, the lock **expires automatically**.
4. Worker B acquires the lock on the same order (lock is gone).
5. Worker B passes the idempotency check (no charge recorded yet).
6. Worker B charges the customer.
7. Worker A's payment call eventually succeeds.
8. Worker A records its charge.
9. Customer is charged **twice**.

The distributed lock has been completely bypassed by its own TTL. This is a well-known failure mode in distributed locking and must be addressed.

**Required fix — choose one or combine:**

- Set `ORDER_LOCK_TTL` to at least `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + safety_margin`, e.g., 120–150s. Document this dependency explicitly with a constant or assertion so future changes to either value don't silently re-introduce the gap.
- Implement lock heartbeat/renewal: a background thread that extends the lock TTL while the payment call is in progress.
- Reduce `PaymentClient` max wall time to well under 60s (e.g., 2 retries × 10s timeout + 1 × 5s backoff = 25s), then set `ORDER_LOCK_TTL = 45s`.

At minimum, add a compile-time or startup assertion:

```python
import math
MAX_PAYMENT_WALL_TIME = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
assert OrderProcessor.ORDER_LOCK_TTL > MAX_PAYMENT_WALL_TIME, (
    f"Lock TTL ({OrderProcessor.ORDER_LOCK_TTL}s) must exceed max payment "
    f"wall time ({MAX_PAYMENT_WALL_TIME}s)"
)
```

### 3. Payment Client Retries on Timeout Without an Idempotency Key — Guaranteed Duplicate Charges

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry
```

A `Timeout` exception from `requests` means the **client-side TCP socket timed out waiting for a response**. It does not mean the gateway did not process the request. The gateway may have:

- Received the request, processed the charge, and sent a response that arrived after the timeout.
- Received the request and is still processing.

When the retry fires, it sends a second charge request with the same `token` and `amount_cents` but **no idempotency key**. The gateway has no way to deduplicate this from the first request. The customer is charged again.

This is not theoretical. Payment gateway timeout + retry = duplicate charge is one of the most common real-world billing bugs.

**Required fix:**

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},  # gateway deduplicates
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `idempotency_key` should be the `order_id` (or a stable derivative). The gateway will return the same `charge_id` for repeated requests with the same key, making retries safe. Verify that the internal payment gateway supports idempotency keys — if it does not, that is a separate blocker that must be resolved before this service can safely retry on timeouts.

## Moderate Issues

### 4. `PaymentError` Is Not Caught — Unhandled Exception Escapes `process()`

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    # ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError — not caught
```

`PaymentError` is not caught anywhere in `process()`. It will propagate up to the caller as an unhandled exception. Depending on the caller (HTTP handler, queue consumer, etc.), this may result in a 500 with no structured response, a worker crash, or a message being requeued and retried — potentially triggering another charge attempt without the lock.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

### 5. Lock Release Silently Swallows Errors in `finally` Block

**File:** `services/order_processor.py`

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection lost, script error), the exception is silently swallowed because there is no try/except inside the `finally`. Worse, the log line says "lock released" even if the release failed. The lock will then hold until TTL expiry, blocking all workers on that order for up to 60 seconds.

**Required fix:**

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.warning("Lock release returned 0 — token mismatch or already expired order_id=%s", order_id)
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release order lock order_id=%s — will expire at TTL", order_id)
```
