Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw between the payment charge and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge is inside the DB transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* but that claim is false. A DB transaction does not wrap a network call to an external payment gateway. These are two independent systems.

```python
# Current code — the comment is misleading and the guarantee does not hold
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)             # If this fails, charge_id is already captured by gateway
    self.db.mark_order_complete(...)
```

**What actually happens on failure:**
- `payment.charge()` succeeds → gateway has captured funds
- `db.record_charge()` raises an exception → DB transaction rolls back
- The charge is now unrecorded. The idempotency check (`db.get_charge`) will find nothing on the next attempt and charge the customer again.

**Fix — two-phase write:**

```python
# Correct approach: record intent before charging, then confirm
with self.db.transaction():
    self.db.create_charge_record(
        req.order_id, req.customer_id, req.amount_cents, status="pending"
    )

# Charge outside the DB transaction — idempotency is now anchored to the pending record
try:
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
except PaymentError:
    with self.db.transaction():
        self.db.mark_charge_failed(req.order_id)
    raise

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also be updated to treat a `pending` record as "in progress" and a `failed` record as retryable, rather than treating any existing record as "already processed."

### 2. Timeout on payment charge is ambiguous — retry can double-charge

In `payment_client.py`, a `Timeout` exception is retried unconditionally:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have been processed by the gateway. Retrying without an idempotency key sent to the gateway can result in the customer being charged multiple times for a single order.

**Fix:** The payment gateway API must support idempotency keys. Pass the `order_id` (or a deterministic derivative) as an idempotency key on every attempt so the gateway deduplicates on its side.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # Required
    timeout=self.REQUEST_TIMEOUT,
)
```

If the gateway does not support idempotency keys, the retry logic on timeouts must be removed entirely. This needs to be clarified before merge.

## Moderate Issues

### 3. Lock TTL can expire before the payment round-trip completes

`ORDER_LOCK_TTL = 60` seconds and `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst case payment client runtime: `3 × 30 + 2 × 5 = 100 seconds`. The lock expires before the payment client finishes, allowing a second worker to acquire the lock and begin processing the same order concurrently.

Either increase the TTL to safely exceed the worst-case payment duration (e.g., 120–150 seconds with margin), or implement lock extension (a periodic background heartbeat that resets the TTL while the lock is held).

### 4. Unhandled exceptions inside the lock context leak silently

If `self.payment.charge()` raises `PaymentError` or any unexpected exception, it propagates out of the `with self.db.transaction()` block and then out of the `with self._acquire_lock()` block. The lock is correctly released by the `finally` clause, but the exception is not caught in `process()`. The caller receives an unhandled exception rather than an `OrderResult`. Add a top-level exception handler:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 5. `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, the raw token value may appear in log output. Scrub sensitive fields before logging or log only field names and error types.

## Minor Issues

### 6. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute was called, which is a brittle implementation detail test. It will break on any internal refactor. Test the observable behavior instead: confirm the lock key is absent in Redis after processing, or use an integration test with a real Redis instance (e.g., `fakeredis`).

### 7. No test for the payment failure path

There is no test covering what happens when `payment.charge()` raises `PaymentError`. Given issue #4 above, this path currently raises an unhandled exception. This case must be tested.

### 8. `CHARGE_ENDPOINT` hardcoded in `PaymentClient`

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Endpoint should be injected via constructor or environment variable. Hardcoding it makes the client untestable without monkeypatching and makes environment-specific configuration impossible.
