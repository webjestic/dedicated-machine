Decision: Request Changes

Summary: The implementation demonstrates solid foundational patterns — atomic `SET NX EX` acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several critical correctness and security flaws that must be resolved before this can be approved, the most severe being a payment-inside-transaction anti-pattern that can cause unrecorded charges at scale, and a timeout race that can produce double-charges under retry. These are not edge cases; they are expected failure modes in distributed payment systems.

---

## Critical Findings

### 1. Payment Call Inside Database Transaction — Unrecorded Charge Risk

**Severity: Critical**
**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
# CURRENT — DANGEROUS
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external I/O inside DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The payment gateway call is inside the database transaction. If the database commit fails after `payment.charge()` returns a `charge_id`, the charge has occurred but is never recorded. The comment in the PR ("crash between charge and record cannot produce an unrecorded charge") is **incorrect** — a DB commit failure, deadlock, or connection drop after the charge but before the commit produces exactly that outcome.

Additionally, holding a DB transaction open across a 30-second (potentially 90-second with retries) external HTTP call will exhaust your connection pool under any real load.

**Required fix:** Charge outside the transaction, then record inside it. Use a two-phase approach with a pending record:

```python
# Write a pending record first — crash-safe audit trail
self.db.record_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# Charge outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Confirm atomically — only this can fail without a charge having occurred
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

### 2. Timeout Retry Creates Double-Charge Window

**Severity: Critical**
**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A network timeout does not mean the charge failed. The gateway may have processed the charge and the response was lost in transit. Retrying on timeout without an idempotency key will submit a second charge for the same amount. This is a well-known failure mode in payment systems.

**Required fix:** Pass a stable idempotency key derived from `order_id` to the gateway on every attempt:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # e.g., order_id
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge()` method signature must accept and forward this key. Without it, retries on timeout are unsafe.

### 3. Redis Lock TTL Does Not Cover Payment Retry Duration

**Severity: High**
**File:** `services/order_processor.py` and `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60   # order_processor.py
REQUEST_TIMEOUT = 30  # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. If the payment client reaches its maximum retry duration, the lock will have expired before processing completes. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The distributed lock provides no protection in this scenario.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at least 120 seconds with documented justification, or implement lock extension (e.g., a background thread that refreshes the TTL while the payment is in-flight). The values across both files must be treated as a coupled constraint and enforced together.

## Significant Findings

### 4. `payment_token` Logged Indirectly via Pydantic Validation Errors

**Severity: Medium**
**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in their output. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in the log message. Payment tokens must never appear in logs.

**Required fix:** Scrub or redact sensitive fields before logging, or log only field names and error types without values:

```python
except ValidationError as e:
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

### 5. Lock Not Released on `PaymentError` or Unhandled Exceptions

**Severity: Medium**
**File:** `services/order_processor.py`

`PaymentError` and any exception from `db.transaction()` are not caught in `process()`. They will propagate out of the `with self._acquire_lock(...)` block, which does correctly release the lock via `finally`. However, the caller receives an unhandled exception rather than an `OrderResult`, breaking the stated return contract of the method.

**Required fix:** Add explicit exception handling for `PaymentError` and database errors within the lock context:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 6. `response.json()["charge_id"]` — Unguarded Key Access

**Severity: Medium**
**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected response body, this raises an unhandled `KeyError` or `json.JSONDecodeError`. In a payment context, a missing `charge_id` on a nominally successful response is a critical data integrity issue and must be handled explicitly, not allowed to propagate as an uncaught exception.

**Required fix:**

```python
try:
    body = response.json()
    charge_id = body["charge_id"]
    if not isinstance(charge_id, str) or not charge_id:
        raise PaymentError("Gateway returned success with missing or invalid charge_id")
except (ValueError, KeyError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

## Minor Findings

### 7. Test Coverage Gaps

**File:** `tests/test_order_processor.py`

The following scenarios are not tested and represent real failure paths:

- Lock TTL expiry during payment (lock expires, second worker acquires, charge races)
- `PaymentError` raised during charge — verifies no partial DB state
- DB transaction commit failure after charge succeeds
- Timeout retry behavior in `PaymentClient` — specifically that retries do not re-raise on timeout until `MAX_RETRIES` is exhausted
- Lock release confirmed even when an exception is raised inside the lock context

These are not blocking this PR alone, but they should be tracked and addressed before this service handles production traffic.

---

### 8. `CHARGE_ENDPOINT` Hardcoded as Class Variable

**File:** `services/payment_client.py`

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoints in class bodies cannot be overridden per environment without subclassing or monkey-patching. This will cause test isolation issues and makes environment-specific configuration (staging vs. production) fragile.

**Required fix:** Accept endpoint and configuration via constructor injection or environment variable with a documented default.
