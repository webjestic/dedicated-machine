Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness and security flaws that must be resolved before this can be approved: a race condition that breaks the at-most-once guarantee, a timeout arithmetic bug that makes the lock window exploitable, and a payment client retry loop that can produce duplicate charges under the exact failure mode this system is designed to prevent.

---

## Critical Issues

### 1. Payment Retry Inside the Lock Creates Duplicate Charge Risk

**File:** `services/order_processor.py` + `services/payment_client.py`

This is the most severe flaw. The PR claims an at-most-once payment guarantee, but the retry logic in `PaymentClient.charge()` directly contradicts it.

```python
# payment_client.py
for attempt in range(self.MAX_RETRIES):
    response = requests.post(...)  # attempt 1: request sent, network drops
    # Timeout fires — did the gateway process it? Unknown.
    # Retry fires — second charge attempt on same token
```

On a `Timeout` exception, the gateway may have already processed the charge. The client has no way to know. Retrying unconditionally on timeout means a charge can be executed 2–3 times for a single `process()` call, all within the same lock window. The distributed lock protects against *concurrent workers*, not against the payment client itself issuing multiple charges.

**Required fix:** The payment client must use idempotency keys on every request. The `order_id` (or a deterministic derivative) must be sent as an idempotency key in the request header or body so the gateway can deduplicate retries.

```python
# Correct pattern
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be passed in, not generated per-attempt
    timeout=self.REQUEST_TIMEOUT,
)
```

The `order_id` must be threaded through from `OrderProcessor.process()` to `payment.charge()`. Generating a new key per attempt defeats the purpose entirely.

### 2. Lock TTL Is Shorter Than Maximum Possible Processing Time

**File:** `services/order_processor.py`

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
# payment_client.py
REQUEST_TIMEOUT = 30   # per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # between retries
```

Maximum elapsed time inside the lock:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 90s + 10s = 100s
```

The lock expires at 60 seconds. A second worker can acquire the lock at t=61s while the first worker is still inside `payment.charge()` on attempt 3. Both workers now hold the lock simultaneously from their own perspective, and both will proceed to the idempotency check. If the DB write from worker 1 hasn't committed yet, worker 2 sees no existing charge and issues a second payment.

**Required fix:** Either extend `ORDER_LOCK_TTL` to safely exceed the maximum processing window (≥120s with margin), or implement lock heartbeat/extension. Also enforce that `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF < ORDER_LOCK_TTL` as an assertion or configuration validation at startup.

### 3. Race Condition Between Idempotency Check and Charge (TOCTOU)

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)  # read
    if existing:
        return ...  # guard

    with self.db.transaction():
        charge_id = self.payment.charge(...)      # external side effect — NOT in DB transaction
        self.db.record_charge(...)
```

The idempotency check reads from the database, but the payment call happens *before* `record_charge` commits. If the lock expires mid-flight (see issue #2), a second worker passes the idempotency check against a DB that still has no record, and both workers charge the payment method. The distributed lock is the only guard, and it has already been shown to be defeatable via timeout arithmetic.

Additionally, `self.payment.charge()` is inside `self.db.transaction()`, but a database transaction provides no atomicity guarantee over an external HTTP call. If `record_charge` fails after `charge()` succeeds, the charge is real but unrecorded. The comment in the code acknowledges this but the implementation does not actually solve it — a database transaction rollback cannot undo a completed HTTP charge.

**Required fix:** The correct pattern is charge-then-record with idempotency keys on the payment side, plus a reconciliation/recovery mechanism for the unrecorded-charge case. At minimum, the code comment must not claim atomicity it does not have.

## Moderate Issues

### 4. `LockError` and `PaymentError` Are Not Caught Symmetrically

**File:** `services/order_processor.py`

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
```

`PaymentError` and any database exceptions are not caught. These will propagate as unhandled exceptions from `process()`, which the caller has no contract for. The method signature returns `OrderResult` but can raise. This is a contract violation that will cause unhandled 500s in any HTTP layer wrapping this.

**Required fix:** Catch `PaymentError` and DB exceptions explicitly, log them, and return a structured `OrderResult(processed=False, ...)` with appropriate messaging. Do not leak internal error details to the caller.

---

### 5. Lock Release Silently Swallows Failures

**File:** `services/order_processor.py`

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If `_release_script` raises (Redis connection failure, script error), the exception propagates out of the `finally` block and suppresses any exception already in flight from the `try` block. Additionally, a return value of `0` from the Lua script (meaning the lock was already expired or stolen) is silently ignored. The log line says "released" even when the lock was not held.

**Required fix:**

```python
finally:
    try:
        result = self._release_script(keys=[lock_key], args=[token])
        if result == 0:
            logger.error("Lock token mismatch or TTL expiry on release order_id=%s — possible concurrent execution", order_id)
        else:
            logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s", order_id)
```

### 6. Test Coverage Does Not Cover the Failure Modes That Matter

**File:** `tests/test_order_processor.py`

The tests cover the happy path and basic contention, but are entirely absent for the scenarios this system is specifically designed to handle:

- No test for lock expiry during payment (TTL race)
- No test for `PaymentError` propagation behavior
- No test for DB transaction failure after successful charge
- No test for lock release returning `0` (stolen/expired lock)
- No test for payment timeout retry producing a duplicate charge
- `test_lock_release_uses_lua_script` asserts the script was called but does not assert correct `keys` and `args` — a bug in key construction would pass this test

## Minor Issues

### 7. `payment_token` Logged Indirectly via Pydantic Validation Error

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, the token value may appear in logs. Payment tokens are sensitive credentials.

**Required fix:** Either sanitize the error output or log only field names and error types, not values.

---

### 8. `customer_id` Has No Pattern Constraint

**File:** `services/order_processor.py`

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has a strict alphanumeric pattern enforced. `customer_id` does not. If `customer_id` is used in any downstream query construction, log formatting, or key generation, this is an injection surface. Apply the same pattern discipline as `order_id`.
