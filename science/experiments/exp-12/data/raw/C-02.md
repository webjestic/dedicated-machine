Decision: Request Changes

Summary: The PR demonstrates solid foundational intent — atomic lock acquisition, Lua-script release, Pydantic validation, and idempotency checks are all directionally correct. However, there are several critical architectural flaws that create real double-charge risk, a lock TTL timing vulnerability, and a charge-outside-transaction problem that directly contradict the safety guarantees claimed in the PR description. These must be resolved before merge.

---

## Critical Issues

### 1. Payment Charge Executes Outside the Database Transaction — Contradicts Core Safety Claim

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

The PR description states: *"crash between charge and record cannot produce an unrecorded charge."* This is false as written.

```python
# What the code actually does:
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← External HTTP call INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Placing an external HTTP call inside a database transaction is an anti-pattern with two severe consequences:

**a) The claim is still wrong.** If `payment.charge()` succeeds but the network drops before the response is received, the payment client raises a `Timeout` exception. The database transaction rolls back. The charge happened. It is now unrecorded. The lock releases. The next retry sees no existing charge and charges again. **Double charge.**

**b) Database connection held open during a 30-second HTTP call (× 3 retries = 90 seconds max).** Under any load, this exhausts the connection pool and causes cascading failures across the service.

**Required fix:** Charge outside the transaction. Record inside. Accept that you need a reconciliation/recovery path for the gap — which is the correct architecture for payment systems:

```python
# Correct structure:
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        return OrderResult(processed=False, ...)

    # Charge OUTSIDE the transaction
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )

    # Record INSIDE the transaction — if this fails, a reconciliation
    # job must detect the unrecorded charge_id and recover
    with self.db.transaction():
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

This doesn't fully solve the gap problem — nothing does without a two-phase protocol or idempotency keys at the gateway level — but it stops holding database connections open across network I/O and is honest about the failure mode.

### 2. Lock TTL vs. Payment Retry Budget — Arithmetic Creates a Race Window

**File:** `services/order_processor.py` and `services/payment_client.py`

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Maximum payment client wall-clock time:

```
(30s timeout × 3 attempts) + (5s backoff × 2 gaps) = 100 seconds
```

The lock TTL is 60 seconds. The payment operation can take 100 seconds. **The lock expires while payment is in-flight.** A second worker acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. Both workers then attempt to record. **Double charge.**

**Required fix:** Either:
- Raise `ORDER_LOCK_TTL` to at minimum `(REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES - 1)) + safety_margin` — currently that's ≥ 120 seconds, and make this a computed constant so it stays in sync:

```python
from services.payment_client import PaymentClient

ORDER_LOCK_TTL = (
    PaymentClient.REQUEST_TIMEOUT * PaymentClient.MAX_RETRIES
    + PaymentClient.RETRY_BACKOFF * (PaymentClient.MAX_RETRIES - 1)
    + 30  # safety margin
)
```

- Or implement lock extension (heartbeat) during the payment call.
- Or enforce that `ORDER_LOCK_TTL` is a hard-coded invariant validated by a test that fails if payment client constants change.

### 3. `PaymentError` Is Not Caught — Unhandled Exception Escapes the Lock Context

**File:** `services/order_processor.py`

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        with self.db.transaction():
            charge_id = self.payment.charge(...)  # raises PaymentError
            ...
except LockError:
    return OrderResult(...)
# PaymentError propagates uncaught to the caller
```

`PaymentError` (and any other exception from `self.db`) propagates out of `process()` entirely. The lock is released correctly by the `finally` block — that part is fine — but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract of the method signature and will cause 500s at whatever API layer calls this.

**Required fix:**

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 4. Timeout Retry on Payment Is Unsafe Without Gateway-Side Idempotency Keys

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded at the gateway — the response was simply lost in transit. Retrying without an idempotency key sends a second charge request that the gateway treats as a new charge. This is a **double-charge vector** independent of the Redis lock.

**Required fix:** Pass a stable idempotency key derived from `order_id` through to `PaymentClient.charge()` and include it in the request headers:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

This requires the payment gateway to support idempotency keys, which any production-grade gateway (Stripe, Braintree, etc.) does. If the internal gateway does not support them, that is a separate critical gap that must be documented and tracked.

## Moderate Issues

### 5. `db.get_charge()` Executes Outside the Database Transaction

The idempotency check reads from the database before the transaction opens. Under Read Committed isolation (the default for PostgreSQL and most databases), a concurrent worker that committed between the `get_charge` read and the `transaction()` open will not be seen. This is a TOCTOU (time-of-check/time-of-use) race.

The Redis lock should prevent this in the normal case, but if the lock expires (see Issue 2), two workers can both pass the idempotency check. Moving `get_charge` inside the transaction and using `SELECT FOR UPDATE` on the order row provides a defense-in-depth layer:

```python
with self.db.transaction():
    existing = self.db.get_charge_for_update(req.order_id)  # SELECT FOR UPDATE
    if existing:
        return OrderResult(processed=False, ...)
    # proceed with charge
```

### 6. Lock Retry Backoff Is Synchronous — Blocks Worker Thread

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

In any async or thread-pool-based service, `time.sleep` blocks the OS thread. Under contention, this causes thread starvation. If this service runs under an async framework (FastAPI, aiohttp), this must use `asyncio.sleep`. If it's synchronous WSGI, this should be documented as a known throughput constraint.

---

### 7. Test Coverage Gaps

The test suite validates the happy path and basic contention but misses the failure modes introduced by the issues above:

- No test for `PaymentError` propagation (exposes Issue 3)
- No test for lock expiry during payment (exposes Issue 2)
- No test for timeout retry behavior in `PaymentClient`
- No test asserting that `payment.charge()` is called with an idempotency key (would enforce Issue 4)
- No test for `db.transaction()` rollback behavior when `record_charge` raises

## Minor Issues

### 8. `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage in Error Paths

The `payment_token` field is never logged directly, which is correct. Confirm that `PaymentError` messages from the gateway do not echo back the token in their body, and that `response.json()` is not logged on error paths.

### 9. `ORDER_LOCK_TTL` and Payment Client Constants Are Decoupled Magic Numbers

As noted in Issue 2, these constants must be kept in sync but are defined in separate files with no enforcement. A module-level assertion or a CI test that validates the relationship would prevent silent regression.
