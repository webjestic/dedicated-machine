Decision: Request Changes

Summary: The PR demonstrates solid foundational intent — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical correctness flaws, one significant security concern, and a few architectural gaps that must be addressed before this handles real payment traffic. The most severe issue is a TOCTOU race on the idempotency check combined with a non-atomic charge/record pattern that can produce duplicate charges under specific failure modes, directly contradicting the PR's stated at-most-once guarantee.

---

## Critical Issues

### 1. TOCTOU: Idempotency Check Is Outside the Database Transaction

**Severity: Critical — can produce duplicate charges**

```python
# CURRENT — check is OUTSIDE the transaction
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # read outside tx
    if existing:
        return ...

    with self.db.transaction():                    # tx starts here
        charge_id = self.payment.charge(...)       # charge happens inside tx
        self.db.record_charge(...)
```

The idempotency check reads from the database *before* the transaction opens. If two workers race past the lock (e.g., Redis failover, lock TTL expiry under a slow payment gateway, or a Redis partition), both can read `existing = None`, both proceed, and both charge. The check must be inside the transaction with a `SELECT FOR UPDATE` or equivalent row-level lock on the order record to be meaningful.

```python
# REQUIRED pattern
with self.db.transaction():
    existing = self.db.get_charge_for_update(req.order_id)  # row-locked read
    if existing:
        return ...
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

### 2. Charge Happens Inside the Database Transaction — Distributed Transaction Anti-Pattern

**Severity: Critical — produces unrecoverable inconsistency**

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external I/O inside DB tx
    self.db.record_charge(...)
```

Holding a database transaction open across an external HTTP call (up to 30 seconds per attempt × 3 retries = 90 seconds) causes:

- Long-held row locks starving other DB operations
- Connection pool exhaustion under load
- If the DB transaction rolls back *after* a successful charge, the charge is real but unrecorded — the exact failure mode this is meant to prevent

The correct pattern is **charge first, then record in a transaction with idempotency enforcement**:

```python
# Charge outside the transaction
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# Record atomically — if this fails, the charge_id is known and can be reconciled
try:
    with self.db.transaction():
        existing = self.db.get_charge_for_update(req.order_id)
        if existing:
            # Charge succeeded but record already exists — idempotent, return existing
            return OrderResult(processed=False, order_id=req.order_id,
                               charge_id=existing["charge_id"], message="Already processed")
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.error("CHARGE_UNRECORDED order_id=%s charge_id=%s — requires reconciliation",
                 req.order_id, charge_id)
    raise
```

### 3. Lock TTL Can Expire During Payment Retry Loop

**Severity: Critical — invalidates the entire locking guarantee**

```python
ORDER_LOCK_TTL = 60   # seconds

# PaymentClient
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment duration: `(30s timeout + 5s backoff) × 3 attempts = 105 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still in its retry loop, breaking mutual exclusion entirely. Either:

- Extend `ORDER_LOCK_TTL` to safely exceed max payment duration (e.g., 180s with margin), **or**
- Implement lock refresh (extend TTL via a background heartbeat while the lock is held)

This must be an explicit, documented, and tested invariant: `ORDER_LOCK_TTL > MAX_PAYMENT_DURATION`.

## Security Issues

### 4. Payment Token Logged at `INFO` Level

**Severity: High — credential exposure in log aggregation systems**

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

The `payment_token` is passed through the call chain and is a payment credential. While it is not directly logged in this line, the `OrderRequest` model and the `charge()` call arguments are both candidates for accidental structured logging (e.g., via exception serialization, request tracing middleware, or debug-level dumps). More directly, `payment_token` is accepted as a raw string in `OrderRequest` with no masking. Any exception that serializes the request object will expose the token.

Recommendations:
- Add a `__repr__`/`__str__` override to `OrderRequest` that masks `payment_token`
- Ensure `payment_token` is never passed to any logger, even indirectly via exception context
- Consider accepting a tokenized reference rather than the raw token at the service boundary

## Architectural Issues

### 5. `PaymentError` Is Silently Swallowed — Unrecorded Charges Are Invisible

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        with self.db.transaction():
            charge_id = self.payment.charge(...)
            ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` is not caught here. It will propagate up. That is actually correct behavior — but the lock will still be released by the `finally` block, and the next caller will find no recorded charge and attempt to charge again. If the payment gateway returned a charge but the client raised `PaymentError` (e.g., on a response parse failure after a successful 200), the order will be double-charged on retry. There must be a dead-letter or pending-charge state to handle the ambiguous "charged but unrecorded" case.

### 6. Redis Failover / Sentinel / Cluster Not Addressed

The PR uses a single `redis.Redis` client. In a multi-worker production deployment, Redis failover (Sentinel promotion or Cluster resharding) can cause a lock to be lost silently. The comment says this prevents duplicate charges, but that guarantee only holds if Redis is the single source of truth for the lock — which it cannot be under failover. The database-level `SELECT FOR UPDATE` (Issue #1) is the correct backstop. Document that the Redis lock is a *performance optimization* (reduces DB contention), not the *correctness guarantee* — the DB constraint is.

### 7. No Unique Constraint Enforcement Documented or Tested

There is no evidence that `db.record_charge` enforces a unique constraint on `order_id` at the database level. If it does not, all the locking logic above is defense-in-depth that can be bypassed by a bug. The database must have a `UNIQUE` constraint on `(order_id)` in the charges table as the final safety net. This should be called out explicitly and tested with a constraint violation scenario.

### 8. `requests` Used Without SSL Verification Configuration

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

No explicit `verify=True` is set (it defaults to `True`, but this is a payment endpoint — it should be explicit and enforced). More importantly, the certificate authority bundle and hostname validation behavior should be documented and pinned for an internal payment gateway. If this ever runs in an environment where `REQUESTS_CA_BUNDLE` or `CURL_CA_BUNDLE` is misconfigured, SSL verification silently degrades.

## Minor Issues

### 9. Test Coverage Gaps

- No test for lock TTL expiry during payment (Issue #3)
- No test for `PaymentError` propagation and re-attempt behavior (Issue #5)
- No test asserting that `payment_token` does not appear in any log output
- No test for the ambiguous timeout case (payment may or may not have succeeded)
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, but does not assert the Lua script was called with the correct token — a token mismatch would silently fail to release the lock

### 10. `backoff` Calculation Is Linear, Not Exponential

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

The docstring and PR description say "retry with backoff" which conventionally implies exponential backoff with jitter. Linear backoff under high contention will cause thundering herd. Recommend `0.1 * (2 ** attempt) + random.uniform(0, 0.1)`.
