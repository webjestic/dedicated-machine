Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the payment/database boundary that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge happens inside the DB transaction — this does not provide atomicity

The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

This claim is **incorrect**. A database transaction does not wrap the external HTTP call to the payment gateway. What actually happens:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)             # DB write
    self.db.mark_order_complete(...)       # DB write
```

If `self.payment.charge()` succeeds but the process crashes before `record_charge` commits, the customer is charged with no record. The DB transaction only protects the two DB writes from each other — it provides zero rollback capability over the already-completed payment. This is a **double-charge / silent charge risk** and the misleading comment makes it worse because a future engineer will trust the guarantee that isn't there.

**Required fix:** The payment call must happen *before* the DB transaction opens, and the idempotency record (in a pending/reserved state) should be written *before* the charge is attempted. A safe pattern:

```python
# 1. Write a pending charge record under the lock (idempotency anchor)
with self.db.transaction():
    self.db.reserve_charge(req.order_id, req.customer_id, req.amount_cents)

# 2. Attempt payment — if this succeeds but we crash, the pending record
#    allows a recovery job to reconcile with the gateway
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Finalize the record
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

A reconciliation/recovery path for the pending state is also needed, but at minimum the misleading comment and the false atomicity claim must be removed.

### 2. Timeout retry in `PaymentClient` can cause duplicate charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request may have **already been processed** by the gateway — the response simply didn't arrive. Retrying on timeout without an idempotency key sent to the gateway is a classic duplicate-charge vector. This is especially dangerous combined with issue #1.

**Required fix:** Pass a stable idempotency key (e.g., derived from `order_id`) to the payment gateway on every attempt for the same order:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # Must be stable across retries
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge()` signature needs to accept this key, and `OrderProcessor` must pass `req.order_id` (or a derivative) as that key.

## Significant Issues

### 3. Lock TTL is too short and not defended against long payment calls

`ORDER_LOCK_TTL = 60` seconds, and `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The lock will expire before the payment completes, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge.

**Required fix:** Either extend `ORDER_LOCK_TTL` beyond the maximum possible payment duration (at least 120–150s with current settings), or implement lock extension (Redlock-style `PEXPIRE` refresh), or reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` so the math closes. The values must be documented together so this relationship is visible.

---

### 4. `PaymentError` is silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
```

`PaymentError` is not caught in `process()`. It will propagate out of `_acquire_lock`'s `try/finally`, which will correctly release the lock, but the caller receives an unhandled exception rather than an `OrderResult`. This breaks the stated return contract of the method.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 5. `payment_token` logged or exposed in error paths

`payment_token` is a sensitive credential. Confirm it is never interpolated into log messages or exception strings anywhere in the call chain. The current code appears clean, but `ValidationError` from Pydantic can include field values in its string representation — verify this is not leaking the token via the `logger.warning("Invalid order request: %s", e)` line.

**Required fix:** Replace with a safe log that does not forward the raw exception:

```python
logger.warning("Invalid order request order_id=%s: %s", order_id, e.error_count())
```

## Minor Issues

### 6. `CHARGE_ENDPOINT` hardcoded as a string literal

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This should come from configuration/environment, not be hardcoded in the class body. This is a deployment hygiene issue — different environments (staging, prod) will need different values and this pattern requires a code change to rotate.

---

### 7. Test for lock release is testing mock wiring, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that the mock was called, which only confirms the mock is wired correctly — it does not test that the Lua script is called with the correct token or that release happens on exception paths. Add tests for:

- Lock is released when `PaymentError` is raised mid-processing
- Lock release is called with the token that was used to acquire it

---

### 8. No test for the lock TTL / payment duration relationship

Given issue #3, a test or at least an assertion that `ORDER_LOCK_TTL > max_possible_payment_duration` would catch this class of misconfiguration at the unit level.
