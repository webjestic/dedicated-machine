Decision: Request Changes

Summary: The architecture and intent here are solid — distributed locking, Lua-based atomic release, idempotency checks, Pydantic validation, and retry logic are all the right instincts. However, there is a critical correctness flaw in the payment/transaction boundary that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issue: Payment Charge Occurs Inside the Database Transaction

This is the most serious problem in the PR.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side-effect inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

This is **incorrect**, and the reasoning is backwards. Wrapping an external HTTP call inside a database transaction does not make them atomic. These are two entirely independent systems with no shared coordinator.

**What actually happens:**

| Scenario | Outcome |
|---|---|
| `payment.charge()` succeeds, then DB transaction commits | ✅ Correct |
| `payment.charge()` succeeds, then DB transaction **rolls back** (e.g., DB error, exception) | ❌ Charge occurred, nothing recorded — **silent double-charge on retry** |
| `payment.charge()` times out after the gateway processes it | ❌ Same problem — charge may have landed, `PaymentError` is raised, DB never written |
| `payment.charge()` raises, DB transaction rolls back | ✅ Consistent, but only because the charge genuinely failed |

The DB transaction provides atomicity across `record_charge` and `mark_order_complete`, but it provides **zero protection** for the external charge call. The comment in the code actively misleads future maintainers about the safety guarantee.

**Required fix:** Charge outside the transaction, then record inside it. If recording fails after a successful charge, that must be treated as a special case — log aggressively, alert, and do **not** retry the charge blindly.

```python
# Charge outside the transaction — external side-effect cannot be rolled back
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the result atomically in the DB
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Charge succeeded but recording failed.
    # Do NOT retry the charge. Alert and surface for manual reconciliation.
    logger.critical(
        "CHARGE_RECORD_FAILURE order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

The idempotency check helps on retry, but only if the charge was recorded. If recording failed, the idempotency check will miss it and the next attempt will charge again.

## Secondary Issues

### 1. `PaymentError` Is Not Caught in `process()`

```python
# In process():
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # Can raise PaymentError
    ...
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller gets a raw exception instead of an `OrderResult`. This is inconsistent with how `LockError` and `ValidationError` are handled. Add an explicit `except PaymentError` block and return a structured `OrderResult(processed=False, ...)`.

---

### 2. Lock TTL Is Dangerously Tight Given Retry Configuration

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
# PaymentClient
MAX_RETRIES = 3
REQUEST_TIMEOUT = 30   # seconds per attempt
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock while the first is still mid-retry on the payment gateway, defeating the entire purpose of the lock. Either increase `ORDER_LOCK_TTL` to at least 120 seconds, or reduce the payment client's worst-case ceiling to fit safely within it. These two constants need to be explicitly coupled in code or documentation.

---

### 3. `payment_token` Is Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their output. If `payment_token` fails validation, its value may appear in log output. Payment tokens are sensitive credentials. At minimum, scrub or summarize the error before logging, or log only field names and error types.

---

### 4. Lock Contention Retry Sleeps on the Last Attempt

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    ...
    time.sleep(backoff)  # ← sleeps even on the final failed attempt
```

On the last attempt, the loop falls through to `if not acquired: raise LockError(...)`, but the sleep still runs before that check. This adds up to 1.5 seconds of unnecessary latency on every lock contention failure. Move the sleep inside a condition: `if attempt < self.LOCK_RETRY_ATTEMPTS - 1`.

---

### 5. Test for Lock Release Does Not Verify Token Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test verifies that the Lua script is called, but does not verify that the correct token is passed as `ARGV[1]`. The entire point of the token-verified release is that the token matches. Add an assertion that the `args` passed to the script match the token that was set.

---

### 6. No Test for the Charge-Then-Record Failure Path

There is no test covering the case where `payment.charge()` succeeds but `db.record_charge()` raises. This is the most dangerous failure mode in the system and should have explicit test coverage once the transaction boundary is corrected.

---

## Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| 🔴 Critical | Move `payment.charge()` outside the DB transaction; handle recording failure explicitly |
| 🔴 Critical | Catch `PaymentError` in `process()` and return structured `OrderResult` |
| 🟠 High | Align `ORDER_LOCK_TTL` with worst-case payment client duration |
| 🟡 Medium | Scrub `payment_token` from validation error logs |
| 🟡 Medium | Fix unnecessary sleep on final lock retry attempt |
| 🟢 Low | Strengthen lock release test to verify token value |
| 🟢 Low | Add test for charge-succeeds / record-fails failure path |
