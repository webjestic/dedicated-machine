Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical flaws spanning distributed systems correctness, timeout arithmetic, and fault handling that would produce silent double-charges or undetected data loss in production. These must be resolved before merge.

---

## Critical Issues

### 1. Timeout Budget Arithmetic — Lock Expiry Can Race Payment Execution

**File:** `services/order_processor.py` and `services/payment_client.py`

**Severity:** Critical — Silent double-charge under normal failure conditions

The Redis lock TTL is 60 seconds. The payment client can consume up to:

```
MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
= 3 × 30 + 2 × 5 = 100 seconds
```

The lock expires after 60 seconds. Worker A's lock expires at t=60 while it is still waiting on a payment gateway response. Worker B acquires the lock at t=61, passes the idempotency check (no charge recorded yet), and begins charging. Worker A's charge completes at t=90 and records. Worker B's charge also completes. Two charges are issued against the same order.

The lock TTL must be strictly greater than the maximum possible payment execution time, including all retries and backoff, with margin. Either:

- Raise `ORDER_LOCK_TTL` to at least `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF + safety_margin` (e.g., 120–150 seconds), or
- Implement lock extension (Redlock-style watchdog / `EXPIRE` refresh) during payment execution, or
- Cap total payment client wall time to well under the lock TTL and enforce it with a hard deadline.

This is the most severe defect in the PR.

### 2. Payment Timeout Is Not a Safe Retry Signal — Idempotency Key Required

**File:** `services/payment_client.py`

**Severity:** Critical — Silent double-charge on timeout retry

A `requests.exceptions.Timeout` means the response was not received — it does **not** mean the charge did not execute. The gateway may have processed the charge and the response was lost in transit. Retrying on timeout without a stable idempotency key will issue a second charge for the same order.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ← Retries unconditionally. Charge may already be complete on the gateway.
```

The fix requires:
- Generating a per-attempt idempotency key (e.g., `f"{order_id}:{attempt}"` or a single stable key per order) and passing it in the request headers (`Idempotency-Key`).
- The payment gateway must honor this key to deduplicate. If the gateway does not support idempotency keys, timeout must be treated as a permanent failure requiring manual reconciliation, not a retry signal.

### 3. `payment.charge()` Is Called Inside `db.transaction()` — External Side Effect in Transaction Scope

**File:** `services/order_processor.py`, lines within `with self.db.transaction():`

**Severity:** High — Unrecoverable inconsistency on transaction rollback

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← External network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `record_charge` or `mark_order_complete` raises after `payment.charge()` succeeds, the DB transaction rolls back. The charge has been collected from the customer but no record exists in the database. This is an undetected revenue/liability event with no recovery path in this code.

The correct pattern is:

```python
charge_id = self.payment.charge(...)   # Outside the transaction

with self.db.transaction():
    # Re-check idempotency inside the transaction under a DB-level lock
    if self.db.get_charge(req.order_id):
        # Charge already recorded — this is a race; handle or compensate
        ...
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The charge must happen before the transaction, and the transaction must be used only for the atomic database writes. A second idempotency check inside the transaction (under a database-level row lock on the order) closes the window where two workers both pass the Redis-level check before either records.

### 4. `LockError` and `PaymentError` Are Not Caught Separately — All Exceptions Silently Swallowed

**File:** `services/order_processor.py`

**Severity:** High — Payment failures are invisible to callers

The `process()` method only catches `LockError` explicitly. `PaymentError`, database exceptions, and any other runtime errors propagate uncaught from within the `with self._acquire_lock(...)` block. This means:

- The lock **is released** in the `finally` block (correct).
- But the exception propagates to the caller with no `OrderResult` returned.
- The caller receives an unhandled exception rather than a structured failure result.

This is inconsistent with the method's documented contract of returning `OrderResult` in all cases. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 5. Lock Retry Backoff Is Blocking — Contention Stalls the Worker Thread

**File:** `services/order_processor.py`

**Severity:** Medium — Latency and throughput degradation under contention

```python
backoff = 0.5 * (attempt + 1)
time.sleep(backoff)
```

With 3 attempts, the maximum blocking time before raising `LockError` is 0.5 + 1.0 = 1.5 seconds of synchronous sleep. In a threaded or async worker pool this blocks the thread entirely. If many orders contend simultaneously, the thread pool saturates. Use async sleep if running under an async framework, or add jitter to prevent thundering herd:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.2)
```

### 6. `db.get_charge()` Idempotency Check Is Outside the Database Transaction — TOCTOU Window

**File:** `services/order_processor.py`

**Severity:** Medium — Race condition between check and write

The idempotency check and the charge recording are not atomic at the database level. Two workers can both acquire the Redis lock sequentially (one after the other's lock expires), both call `get_charge()` and receive `None`, and both proceed to charge. The Redis lock reduces this window significantly but does not eliminate it if the lock expires mid-execution (see Issue 1).

The database must enforce a unique constraint on `order_id` in the charges table, and `record_charge` must propagate the resulting `IntegrityError` (or equivalent) as a detectable signal, not a crash. This is the correct last-line-of-defense against double-charge even if all other guards fail.

## Test Coverage Gaps

| Missing Test | Risk |
|---|---|
| Payment timeout followed by retry — verifies idempotency key behavior | Masks double-charge on timeout |
| Lock TTL expiry mid-payment — second worker acquires lock before first records | Core race condition untested |
| `payment.charge()` succeeds but `record_charge` raises — verifies charge is not silently lost | Data loss scenario |
| `PaymentError` raised — verifies `OrderResult(processed=False)` returned, not exception propagation | Contract violation |
| DB unique constraint violation on `record_charge` — verifies graceful handling | Last-resort guard |

The existing tests are well-structured and cover the happy path and basic contention, but none of the failure-mode tests exercise the critical paths identified above.

## Minor Notes

- `payment_token` is logged at `INFO` level indirectly through `charge_id` correlation — confirm the token itself never appears in log output at any level. Verify the payment client does not log the raw `token` field from the request body.
- `ORDER_LOCK_TTL` and payment client timeout constants should be configurable via environment or config injection rather than hardcoded class attributes, to allow environment-specific tuning without code changes.
- `CHARGE_ENDPOINT` hardcoded in `PaymentClient` should be injected at construction time to support test and staging environments without subclassing.
