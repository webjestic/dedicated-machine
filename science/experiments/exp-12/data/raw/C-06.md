Decision: Request Changes

Summary: The architecture demonstrates solid foundational intent — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several critical flaws spanning distributed systems correctness, timeout arithmetic, and unhandled failure modes that would allow duplicate charges or silent data loss in production under realistic failure conditions. These must be resolved before merge.

---

## Critical Issues

### 1. Timeout Arithmetic Violation — Lock Can Expire Mid-Transaction

**File:** `services/order_processor.py` + `services/payment_client.py`

**Severity:** Critical — enables double-charge

The Redis lock TTL is 60 seconds. The payment client alone can consume up to:

```
MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
= 3 × 30 + 2 × 5 = 100 seconds
```

The lock expires at second 60. A second worker can acquire the lock at second 61, pass the idempotency check (the first worker hasn't committed yet), and issue a second charge. The entire guarantee the PR claims to provide collapses under this arithmetic.

**Required fix:** Either enforce `ORDER_LOCK_TTL > max_payment_duration` with a hard assertion at startup, or implement lock extension (e.g., a heartbeat thread that refreshes `EX` while the lock is held). The constants must be co-located with a documented invariant, not silently divergent across files.

```python
# Minimum required — enforce at class definition or __init__
MAX_PAYMENT_DURATION = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
assert OrderProcessor.ORDER_LOCK_TTL > MAX_PAYMENT_DURATION, (
    f"Lock TTL {OrderProcessor.ORDER_LOCK_TTL}s must exceed max payment "
    f"duration {MAX_PAYMENT_DURATION}s"
)
```

### 2. Payment Charge Outside Database Transaction — Unrecorded Charge Window

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

**Severity:** Critical — silent financial loss

The PR comment claims "crash between charge and record cannot produce an unrecorded charge." This is false. `self.payment.charge(...)` is a network call to an external system. It is **not** part of the database transaction. If the charge succeeds at the gateway but the database transaction rolls back (network partition, DB crash, constraint violation), the charge is real money taken with no record.

```python
# Current — charge happens inside transaction block but is NOT transactional
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect — cannot be rolled back
    self.db.record_charge(...)             # if this fails, charge is orphaned
```

The database transaction provides atomicity only over database operations. The payment call is an irreversible external side effect. The comment is actively misleading and will cause future engineers to trust a guarantee that does not exist.

**Required fix:** Implement a proper outbox pattern or two-phase approach:

1. Record a `PENDING` charge intent in the DB transaction before calling the gateway.
2. Call the gateway.
3. Update the record to `COMPLETED` with the returned `charge_id`.
4. A reconciliation job handles `PENDING` records older than a threshold.

Alternatively, if the payment gateway supports idempotency keys, pass a deterministic key derived from `order_id` so retries are safe and a separate reconciliation pass can confirm state.

### 3. Unhandled `PaymentError` Propagates Through Lock Release — Charge Orphaned

**File:** `services/order_processor.py`

**Severity:** High

If `self.payment.charge()` raises `PaymentError`, the exception propagates up through the `with self.db.transaction()` block (rolling back the DB transaction) and then through the `with self._acquire_lock()` context manager (correctly releasing the lock). However, the `process()` method has no `except PaymentError` handler. The exception surfaces to the caller as an unhandled exception with no `OrderResult` returned.

The caller receives a raw exception instead of a structured `OrderResult`, breaking the stated return contract of the method. Any upstream handler that expects `OrderResult` will crash.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

### 4. Lock Retry Backoff Uses Blocking `time.sleep` — Worker Thread Starvation

**File:** `services/order_processor.py`

**Severity:** Medium — operational / performance

```python
backoff = 0.5 * (attempt + 1)
time.sleep(backoff)
```

In any async or thread-pool deployment model, blocking `time.sleep` in a retry loop holds the thread for up to 1.5 seconds across three attempts. Under high concurrency this causes thread starvation. Additionally, deterministic backoff without jitter causes thundering herd: all contending workers wake simultaneously and re-contend on the same lock.

**Required fix:** Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.3)
```

For async deployments, replace with `asyncio.sleep`.

### 5. `db.get_charge()` Idempotency Check Is Outside the Lock's Consistency Scope

**File:** `services/order_processor.py`

**Severity:** Medium — subtle TOCTOU

The idempotency check reads from the database after acquiring the lock. This is correct for preventing concurrent workers from double-charging. However, the check is only as strong as the database read's isolation level. If `db.get_charge()` executes under `READ COMMITTED` and a concurrent transaction has charged but not yet committed, the check returns `None` and proceeds to charge again.

**Required fix:** Document the required transaction isolation level explicitly, or execute the idempotency check inside the same database transaction as the write with a `SELECT FOR UPDATE` or equivalent to guarantee the read sees committed state and serializes against concurrent writers.

### 6. `payment_token` Logged Indirectly — Verify No Accidental Exposure

**File:** `services/order_processor.py`, `services/payment_client.py`

**Severity:** Medium — compliance / PCI-DSS

`payment_token` flows through `OrderRequest` and is passed directly to `payment_client.charge()`. The payment client logs `charge_id` but not the token — that is correct. However, if any exception handler or debug logger serializes the full `OrderRequest` object or the `requests` response body, the token may appear in logs. Pydantic models serialize all fields by default.

**Required fix:** Mark `payment_token` as a secret field so it is redacted in any default serialization:

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

Audit all log call sites to confirm no `req.model_dump()` or `str(req)` is emitted.

## Test Coverage Gaps

### Missing: Lock Expiry During Payment

No test simulates the lock expiring (TTL elapsed) while `payment.charge()` is executing. This is the exact scenario that produces a double charge. A test should mock `redis.set` to return `True` for two sequential calls with the same key (simulating expiry and re-acquisition by a second worker) and assert that `record_charge` is called at most once.

### Missing: PaymentError Propagation

No test covers `payment.charge()` raising `PaymentError`. Given that this currently produces an unhandled exception (Issue 3), this gap is directly masking a known bug.

### Missing: DB Transaction Rollback After Successful Charge

No test covers the scenario where `payment.charge()` succeeds but `db.record_charge()` raises. This is the orphaned charge scenario from Issue 2.

### Missing: Concurrent Worker Idempotency Under Race

The idempotency test (`test_process_already_completed`) only tests the sequential case. No test exercises two processor instances racing through the lock acquisition and idempotency check simultaneously.

## Minor Issues

- `CHARGE_ENDPOINT` is hardcoded as a class attribute string. This should be injected via configuration or environment variable. Hardcoded internal URLs are a deployment and secret management concern.
- `customer_id` has no pattern constraint in `OrderRequest`, unlike `order_id`. If `customer_id` is used in any database query or log correlation, it should be similarly constrained.
- The `finally` block in `_acquire_lock` calls `self._release_script(...)` unconditionally. If `acquired` is `False` and the exception path somehow reaches `finally` (it cannot in current code due to the raise, but defensive coding matters), it would attempt to release a lock never acquired. The `finally` should guard: `if acquired: self._release_script(...)`. Currently safe but fragile.
