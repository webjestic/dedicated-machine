Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge happens inside the DB transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* because both operations are inside `self.db.transaction()`. This is incorrect and the claim is dangerous.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an HTTP call to an external payment gateway. It is **not enrolled in the database transaction**. If the process crashes after `charge()` succeeds but before `record_charge()` commits, you have charged the customer with no record. The DB transaction provides no protection here. The correct pattern is:

1. Persist a `PENDING` charge record to the DB first (inside a transaction).
2. Call the payment gateway.
3. Update the record to `COMPLETED` or `FAILED` in a follow-up transaction.
4. Use a reconciliation job to catch any `PENDING` records that never resolved.

This is a real-money double-charge / lost-charge risk and must be fixed.

---

### 2. Timeout retry in `PaymentClient` can cause double charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request may have already been processed by the gateway — the response just didn't arrive. Retrying on timeout without an idempotency key sent to the gateway can result in the customer being charged multiple times. The `charge()` call must include a stable idempotency key (e.g., the `order_id`) in the request headers or body, and the gateway must support it.

```python
# Required
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # Must be passed in
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 3. Lock TTL is too short and has no grace-period awareness

`ORDER_LOCK_TTL = 60` seconds. The payment client alone can take up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds` in the worst case. The lock can expire while the payment is still in flight, allowing a second worker to acquire the lock and enter the idempotency check before the first worker has written its charge record. The TTL must be set to comfortably exceed the maximum possible processing time.

```python
# 30s timeout × 3 attempts + 5s backoff × 2 + DB overhead
ORDER_LOCK_TTL = 120  # minimum; consider 180s
```

### 4. Exceptions inside the lock are silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
except LockError:
    return OrderResult(...)
```

Only `LockError` is caught. Any other exception — `PaymentError`, a DB connection error, an unexpected `KeyError` from `response.json()` — will propagate out of the `with` block, which is fine for the lock release (the `finally` handles it), but the caller gets an unhandled exception with no structured `OrderResult`. Add a broad catch at the outer level to return a structured failure and log the full traceback.

### 5. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises a `KeyError` or `JSONDecodeError` that is not caught. This should be wrapped with explicit error handling and a clear log message.

---

## Minor Issues

### 6. `payment_token` is logged indirectly via `charge_id` — verify no token leakage

The `payment_token` field is not logged directly, which is correct. Confirm that the `payment_client` also never logs the raw token value. A quick grep of the codebase for `token` in log statements is warranted before merge.

### 7. Test for lock TTL expiry during processing is missing

There is no test covering the scenario where the lock expires mid-processing. Given this is the primary failure mode for the at-most-once guarantee, it should be explicitly tested.

### 8. `RETRY_BACKOFF` is a flat value, not exponential

The order processor uses exponential backoff (`0.5 * (attempt + 1)`), but the payment client uses a flat 5-second sleep. These should be consistent. Flat backoff under sustained gateway degradation will hammer the endpoint. Use exponential backoff with jitter.
