Decision: Request Changes

Summary: The architecture and security fundamentals here are solid — atomic lock acquisition, token-verified Lua release, idempotency check, Pydantic validation, and structured retry logic are all implemented correctly. However, there are several reliability and correctness issues that must be addressed before this can be approved, the most critical being a TOCTOU race between the payment charge and the database transaction that can produce unrecorded charges in production.

---

## Critical Issues

### 1. Payment charge outside the database transaction (TOCTOU / data integrity)

This is the most serious bug. The PR comment claims *"crash between charge and record cannot produce an unrecorded charge"* — that claim is **false**.

```python
# CURRENT — charge happens INSIDE the db.transaction() block
with self.db.transaction():
    charge_id = self.payment.charge(...)        # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

At first glance this looks correct, but `self.payment.charge()` is an **external HTTP call to a payment gateway**. Wrapping it in a database transaction does not make it atomic with the database writes. The database transaction only protects the two DB operations against each other. If the process crashes after `charge()` returns but before `record_charge()` commits, the charge exists at the gateway with no record in your database. The comment is misleading and will give future maintainers false confidence.

The correct pattern for at-most-once payment semantics is:

```python
# Option A — charge first, then open a short-lived DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# If we crash here, the Redis lock TTL expires, the next worker hits
# the idempotency check (get_charge), finds nothing, and re-charges.
# You need a pre-authorization / reservation step to prevent this.

# Option B (preferred) — write a PENDING record before charging
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

Option B means the idempotency check (`get_charge`) will find the PENDING record on retry and the second worker will not re-charge. This is the standard two-phase approach used in payment systems. The current code does not implement this.

---

### 2. Timeout during payment retry can exceed the Redis lock TTL

```python
ORDER_LOCK_TTL = 60        # seconds
# payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. A second worker can acquire the lock while the first worker is still mid-charge on attempt 2 or 3, breaking the mutual exclusion guarantee entirely. This directly enables the double-charge scenario the service is designed to prevent.

**Fix:** Either extend `ORDER_LOCK_TTL` to safely exceed worst-case payment duration (e.g., 120–150s with margin), or implement lock heartbeat renewal, or reduce `MAX_RETRIES × REQUEST_TIMEOUT` to fit within the TTL. Document the relationship explicitly so it does not silently regress.

## Moderate Issues

### 3. Unhandled exceptions inside the lock context leak the lock state silently

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

If `self.db.get_charge()`, `self.db.transaction()`, or `self.db.record_charge()` raises an unexpected exception, the `contextmanager` `finally` block will correctly release the lock — that part is fine. However, the exception propagates all the way out of `process()` uncaught, returning nothing to the caller instead of an `OrderResult`. The caller contract is broken; every code path should return an `OrderResult`.

```python
except (DatabaseError, PaymentError) as e:
    logger.error("Order processing failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Processing error")
```

### 4. `PaymentError` is not imported or handled in `order_processor.py`

`payment_client.py` defines and raises `PaymentError`, but `order_processor.py` never imports it and has no `except PaymentError` handler. A payment decline or exhausted retries will raise an unhandled exception that escapes `process()` entirely, violating the `OrderResult` return contract and potentially crashing the worker.

```python
# Missing in order_processor.py
from services.payment_client import PaymentClient, PaymentError
```

### 5. `payment_token` logged or exposed in `OrderResult`

`payment_token` is a sensitive credential. Confirm it is never written to logs. Currently it is not, but `req` is a Pydantic model that could be accidentally logged in future debug statements. Consider marking the field with a custom `__repr__`/`__str__` that redacts the value, or storing it as a `SecretStr`.

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

## Minor Issues

### 6. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that `redis.delete` was not called and the script mock was called once, but it does not verify the script was called with the correct `keys` and `args` (i.e., the right lock key and the matching token). A refactor could satisfy this test while breaking the actual token-verification guarantee. Assert on the call arguments.

### 7. No test covering the lock TTL expiry / payment timeout race

Given issue #2 above, there should be a test (or at minimum a documented integration test requirement) that validates behavior when the lock expires mid-payment. This is the highest-risk scenario for the service's core guarantee.

### 8. `customer_id` has no pattern validation

`order_id` has a strict alphanumeric pattern enforced. `customer_id` does not. If `customer_id` is used in database queries downstream, this is a potential injection surface depending on the DB layer. Apply consistent validation.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | **Critical** | Implement two-phase charge (PENDING record) to prevent unrecorded charges on crash |
| 2 | **Critical** | Align lock TTL with worst-case payment client timeout or add heartbeat |
| 3 | Moderate | Catch and return `OrderResult` for all exception paths in `process()` |
| 4 | Moderate | Import and handle `PaymentError` in `order_processor.py` |
| 5 | Moderate | Use `SecretStr` for `payment_token` |
| 6 | Minor | Strengthen lock release test to assert on call arguments |
| 7 | Minor | Add test or documented scenario for lock expiry during payment |
| 8 | Minor | Add pattern validation to `customer_id` |

The bones of this service are good and the locking approach is sound. Items 1 and 2 are blocking — they directly undermine the at-most-once guarantee that is the entire point of this service.
