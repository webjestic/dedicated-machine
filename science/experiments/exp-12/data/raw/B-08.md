Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several critical architectural flaws that would allow double-charges in production under realistic failure conditions, plus a non-trivial security vulnerability in the payment client. These must be resolved before merge.

---

## Critical Findings

### 1. TOCTOU Race: External Payment Call Inside the Lock Does Not Prevent Double-Charge

**Severity: Critical — Financial Impact**

This is the most dangerous flaw in the PR. The comment in the code claims:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
```

This claim is **false**. The `payment.charge()` call is an external side effect that occurs *outside* the database transaction's atomicity boundary. The database transaction only makes `record_charge` and `mark_order_complete` atomic with each other — it provides zero atomicity guarantee over the external payment gateway call.

**Attack / Failure Scenario:**

```
Worker A: acquires lock, idempotency check passes (no existing charge)
Worker A: calls payment.charge() → gateway receives and processes charge
Worker A: gateway returns charge_id
Worker A: db.transaction() begins
Worker A: process crashes / pod is OOM-killed / network drops
Worker A: db.transaction() never commits → charge_id never recorded
Worker A: lock expires (TTL=60s) or was never released due to crash

Worker B: acquires lock (A is dead), idempotency check passes (nothing in DB)
Worker B: calls payment.charge() → SECOND CHARGE ISSUED
```

The Redis lock TTL of 60 seconds does not help here because the lock is released (or expires) before Worker B runs, and the DB has no record of the first charge.

**Required Fix:** Implement a two-phase commit pattern or an outbox pattern. Before calling the payment gateway, write a `PENDING` charge record to the database within a transaction. After the gateway responds, update the record to `COMPLETED`. The idempotency check must treat `PENDING` records as "already in flight" and block re-entry. This ensures a crash between charge and record leaves a `PENDING` tombstone that prevents re-charge.

```python
with self.db.transaction():
    # Write intent before external call — crash-safe tombstone
    self.db.create_charge_intent(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must then be updated:

```python
existing = self.db.get_charge(req.order_id)
if existing and existing.get("status") in ("pending", "completed"):
    return OrderResult(processed=False, message="Order already processed or in flight")
```

### 2. Lock TTL Is Shorter Than the Maximum Possible Processing Time

**Severity: High — Enables Double-Charge**

```python
ORDER_LOCK_TTL = 60  # seconds
```

```python
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5  # seconds between retries
```

Maximum payment client wall time: `(30s × 3) + (5s × 2) = 100 seconds`

The lock TTL is **60 seconds**. The payment client can hold the lock for up to **100 seconds**. This means the lock can expire while a worker is still executing a retry loop, allowing a second worker to acquire the lock and pass the idempotency check while the first worker is mid-flight.

**Required Fix:** Either increase `ORDER_LOCK_TTL` to at minimum `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + safety_margin` (e.g., 120–150s), or implement lock extension (Redlock-style heartbeat / `EXPIRE` refresh) for long-running operations. The two values must be co-owned and validated together — a comment or assertion in code enforcing this relationship is required.

### 3. Payment Client Retries on Timeout Are Unsafe Without Idempotency Keys

**Severity: High — Financial Impact**

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retries unconditionally
```

A timeout does **not** mean the charge failed. The gateway may have received and processed the charge successfully before the connection dropped. Retrying on timeout without an idempotency key sent to the gateway will issue a second charge for the same amount.

**Required Fix:** Generate a stable idempotency key per order (e.g., derived from `order_id`) and include it in every request to the payment gateway. Most payment APIs (Stripe, Braintree, etc.) honor `Idempotency-Key` headers to deduplicate retried requests.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 4. SSRF Vulnerability — Hardcoded Internal Endpoint with No Validation

**Severity: Medium-High — Security**

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

The endpoint is hardcoded as a class attribute, meaning it can be overridden at instantiation or monkey-patched. More critically, if this value is ever sourced from configuration (environment variables, config files, service discovery), there is no validation that the resolved URL points to an expected host. An attacker with config injection access (e.g., via a compromised secrets manager or environment variable injection) could redirect payment traffic to an attacker-controlled server.

**Required Fix:** Validate the resolved endpoint against an allowlist of expected hostnames at startup, not at call time. Fail hard on startup if the endpoint does not match.

```python
from urllib.parse import urlparse

ALLOWED_PAYMENT_HOSTS = {"api.payments.internal"}

def __init__(self):
    parsed = urlparse(self.CHARGE_ENDPOINT)
    if parsed.hostname not in ALLOWED_PAYMENT_HOSTS:
        raise ValueError(f"Untrusted payment endpoint host: {parsed.hostname}")
```

### 5. Unhandled Exceptions Escape the Lock Context Manager

**Severity: Medium — Reliability / Lock Leak**

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

If `self.payment.charge()` raises `PaymentError`, or if `self.db.record_charge()` raises a database exception, the exception propagates up through `process()` entirely uncaught. The lock *will* be released correctly by the `finally` block in `_acquire_lock` — that part is fine. However, the caller receives an unhandled exception rather than an `OrderResult`, breaking the contract that `process()` always returns an `OrderResult`. This can cause upstream failures in the agentic pipeline.

**Required Fix:** Catch `PaymentError` and database exceptions within `process()` and return a structured `OrderResult(processed=False, ...)`.

---

### 6. Test Coverage Does Not Test the Critical Failure Path

**Severity: Medium — Test Quality**

The test suite does not include a test for the crash-between-charge-and-record scenario, which is the most important correctness property claimed by this PR. There is no test verifying that a `PENDING` record blocks re-entry, no test for lock TTL expiry during payment retry, and no test for `PaymentError` propagation behavior. The existing tests validate the happy path and basic contention, but do not exercise the failure modes that motivated this PR's existence.

## Minor Issues

- `customer_id` has no pattern constraint in the Pydantic model, unlike `order_id`. If `customer_id` is used in any DB query construction downstream, this is an injection surface.
- `payment_token` is logged indirectly via `OrderRequest` — confirm that `__repr__` or `__str__` on the model does not leak token values into log output. Consider marking `payment_token` as a `SecretStr` field.
- The `LOCK_RETRY_ATTEMPTS = 3` with `backoff = 0.5 * (attempt + 1)` produces a maximum wait of 3 seconds before giving up. For a 60-second lock TTL, this means a second worker will fail fast and return an error to the caller rather than waiting for the first worker to finish. Depending on the caller's retry behavior, this could result in lost orders rather than queued ones. This is a design decision that should be explicitly documented.
