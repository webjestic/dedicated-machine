Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several critical architectural flaws that would allow double-charges in production under realistic failure conditions, plus secondary issues in the payment client and test coverage that must be resolved before this handles real financial transactions.

---

## Critical: Charge Outside Database Transaction — Double-Charge Window

This is the most severe flaw in the PR. The comment in the code claims crash safety, but the implementation does not deliver it.

```python
# As written — UNSAFE
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External side-effect INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Placing `payment.charge()` inside a database transaction does not make the payment atomic with the database write. The payment gateway is an external system that has no participation in your database's two-phase commit. This creates two distinct failure windows:

**Window 1 — Charge succeeds, DB commit fails.** The payment gateway has captured funds. The database transaction rolls back. `record_charge` is never persisted. The idempotency check on retry finds no existing charge and issues a second charge.

**Window 2 — Charge succeeds, process crashes before DB commit.** Same outcome as above.

The correct pattern for at-most-once payment semantics is an outbox or pre-authorization record:

```python
# Safe pattern — record intent before charging
with self.db.transaction():
    self.db.create_charge_intent(req.order_id, req.customer_id, req.amount_cents)

# Now charge externally — if this crashes, the intent record exists
# and a recovery job can reconcile or void
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the result — if this crashes, the intent record allows reconciliation
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

The intent record must be checked in the idempotency path. A charge in `INTENT` state should block re-entry and trigger reconciliation, not allow a new charge.

## Critical: Lock TTL Does Not Cover Payment Retry Duration

```python
ORDER_LOCK_TTL = 60   # seconds
```

```python
# payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
REQUEST_TIMEOUT = 30  # seconds per attempt
```

Worst-case payment client execution time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock at second 61 while the first worker is still in its third payment attempt. Both workers pass the idempotency check, both call `payment.charge()`. This is a direct double-charge path.

The TTL must be set to a value strictly greater than the maximum possible payment client execution time, with margin. Either increase `ORDER_LOCK_TTL` to at minimum 120 seconds, or — better — enforce that payment client maximum execution time is a named constant shared between both modules and validated at startup.

## Critical: Timeout Exceptions Are Not Retried Correctly

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to sleep and next attempt — looks correct
```

This appears correct on first read, but the problem is that a `Timeout` on a POST to a payment gateway is **not a safe retry condition**. The request may have been received and processed by the gateway before the timeout occurred on the client side. Retrying a timed-out charge without first querying the gateway for the status of the previous attempt will double-charge.

The payment client must implement an idempotency key per logical charge attempt, passed as a header or request field, so that gateway-side deduplication applies on retry. The idempotency key should be derived from `order_id`, not generated fresh per HTTP attempt.

```python
# Correct — stable idempotency key for the logical charge
idempotency_key = f"charge-{order_id}"

response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

## High: Lock Released Before Caller Returns — Race Window on Idempotency Check

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    ...
    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

The lock is released in `finally` before `process()` returns. The idempotency check and the charge both occur inside the `with self._acquire_lock(...)` block, which is correct. However, the `record_charge` database write must be fully committed and visible to other workers before the lock is released. If the database uses read-committed isolation and the commit is not flushed before the `with` block exits, a second worker acquiring the lock immediately after release may read stale data and not see the completed charge.

Verify that `self.db.transaction().__exit__` guarantees a synchronous, durable commit before control returns to the `with` block body. If the DB client uses async commit or connection pooling with deferred writes, this is a real race.

---

## Medium: `payment_token` Logged Indirectly via `charge_id` — Verify No Token Leakage

The `payment_token` field is accepted as input and passed directly to the payment client. Confirm that no logging path in `PaymentClient.charge()` or any middleware logs the raw request body, which would include the token. The current code does not log it explicitly, but `requests` libraries and HTTP middleware can log full request bodies in debug mode. Add a note or assertion in the logging configuration that `DEBUG` level is prohibited in production for this service.

---

## Medium: `requests.exceptions.ConnectionError` Not Handled

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
```

`requests.exceptions.ConnectionError` (DNS failure, TCP reset, refused connection) is not caught. It will propagate as an unhandled exception, bypass the retry loop entirely, and surface to the caller as a 500 with no retry. Given the same timeout-ambiguity concern above, a connection error mid-request is also not a safe unconditional retry without idempotency keys. At minimum, catch it and apply the same retry logic.

---

## Low: Test for Lock Release Verifies Mechanism, Not Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test verifies that `redis.delete` was not called and the script was called once. It does not verify that the script was called with the correct key and the correct token that was generated during acquisition. A regression that passes a hardcoded empty string as the token would pass this test. Assert on `call_args` to verify key and token structure.

There is also no test for the failure path where `payment.charge()` raises `PaymentError`. What does `process()` return? What is the lock state after? This should be an explicit test case.
