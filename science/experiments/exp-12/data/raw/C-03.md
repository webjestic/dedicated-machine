Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical flaws that would allow double-charges in production, create silent data corruption under failure, and expose the system to resource exhaustion. These must be resolved before merge.

---

## Critical Issues

### 1. Payment Charge Occurs Inside the Database Transaction — TOCTOU / Dual-Write Violation

This is the most severe flaw in the PR.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims this provides atomicity, but it does not. A database transaction cannot roll back an already-executed external HTTP call to the payment gateway. The failure modes are:

- `payment.charge()` succeeds → gateway debits the customer → `record_charge()` raises → DB rolls back → charge is real but unrecorded. The idempotency check will never find it. The next retry will charge again.
- `payment.charge()` times out after the gateway has already processed the charge → `PaymentClient` raises `PaymentError` → DB rolls back → same double-charge scenario.

**Required fix:** Charge outside and before the transaction. Record only after a confirmed charge ID is returned. The transaction should wrap only the DB writes.

```python
# Charge OUTSIDE the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically only after charge is confirmed
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even then, a crash between charge and `record_charge` is still possible. The correct long-term solution is an outbox/saga pattern or idempotency keys submitted to the payment gateway before charging, so a re-query can recover the charge ID without re-charging.

### 2. Lock TTL vs. Total Payment Retry Duration — Lock Expires While Work Is In-Flight

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client duration:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. A second worker can acquire the lock after 60 seconds while the first worker is still mid-charge. The idempotency check will pass for the second worker because the first worker hasn't written the record yet. Both workers charge the customer.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at least `(MAX_RETRIES × REQUEST_TIMEOUT) + (MAX_RETRIES × RETRY_BACKOFF) + safety_margin`, or implement lock extension (heartbeat renewal) during the payment call. The constants must be co-located or derived from one another with an explicit assertion:

```python
assert ORDER_LOCK_TTL > (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT +
    PaymentClient.MAX_RETRIES * PaymentClient.RETRY_BACKOFF
), "Lock TTL must exceed maximum payment client duration"
```

### 3. PaymentError Is Not Caught — Unhandled Exception Escapes `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # raises PaymentError
```

`PaymentError` is not caught anywhere in `process()`. It will propagate out of the context manager, but the lock will still be released correctly via `finally`. However, the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract of the method, which promises to return `OrderResult` in all cases.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 4. Retry on Timeout Is Unsafe Without Gateway Idempotency Keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

On a timeout, the gateway may have already processed the charge. Retrying without an idempotency key will submit a duplicate charge request. The gateway has no way to deduplicate it.

**Required fix:** Generate a per-order idempotency key before the retry loop and submit it as a header on every attempt:

```python
idempotency_key = f"order-{order_id}-{uuid.uuid4()}"  # generated once, passed in
# then on each attempt:
headers={"Idempotency-Key": idempotency_key}
```

The key must be stable across retries for the same logical charge attempt. This is the standard mechanism used by Stripe, Adyen, and similar gateways.

## Significant Issues

### 5. Lock Retry Backoff Is Blocking — Entire Worker Thread Stalls

```python
time.sleep(backoff)  # 0.5s, 1.0s, 1.5s
```

In any async or thread-pool worker context, `time.sleep()` blocks the thread for the full backoff duration. With `LOCK_RETRY_ATTEMPTS = 3`, a worker can stall for 3 seconds before giving up. Under high concurrency this degrades throughput significantly.

**Required fix:** If this runs in an async context, use `asyncio.sleep()`. If threaded, document the blocking behavior explicitly and ensure the thread pool is sized to account for it.

---

### 6. `db.get_charge()` Is Not Inside the Lock's Critical Section Correctly — Read Is Stale

The idempotency check reads from the database after acquiring the lock, which is correct. However, `db.get_charge()` must use a read that is consistent with the transaction isolation level. If the DB uses `READ COMMITTED` and another transaction has not yet committed, `get_charge()` may return `None` even though a charge is in-flight. This is a narrow but real race if the lock is somehow not functioning (e.g., Redis failover, lock TTL expiry per issue #2).

**Recommendation:** The idempotency check should be re-evaluated inside the DB transaction with a `SELECT FOR UPDATE` or equivalent row-level lock on the order record, providing a defense-in-depth layer independent of Redis.

---

### 7. No Handling of Redis Connectivity Failures

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If Redis is unavailable, `redis.set()` raises `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`. These are not caught. The exception will propagate out of `process()` as an unhandled error, again breaking the `OrderResult` return contract and potentially crashing the worker.

**Required fix:** Catch Redis exceptions explicitly. Decide on policy: fail open (process without lock, risky) or fail closed (return error to caller, safe). For a payment system, fail closed is required.

```python
except (redis.exceptions.ConnectionError, redis.exceptions.TimeoutError) as e:
    logger.error("Redis unavailable, refusing to process order_id=%s: %s", order_id, e)
    return OrderResult(processed=False, message="Lock service unavailable")
```

## Minor Issues

### 8. Test Coverage Gaps

- No test for `PaymentError` being raised during charge — currently would surface as an unhandled exception in the test runner, masking the bug.
- No test for Redis connectivity failure.
- No test for lock TTL expiry mid-payment (requires mocking time or injecting a sleep).
- `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` but does not assert the correct `keys` and `args` were passed to the Lua script, so a broken release would still pass the test.

### 9. `payment_token` Logged Indirectly via `charge_id`

Low risk here, but `payment_token` passes through to the payment client as a raw string. Confirm it is never written to logs at the `DEBUG` level by the `requests` library (e.g., via `requests` debug logging or a middleware logger). Token values in logs are a PCI-DSS finding.
