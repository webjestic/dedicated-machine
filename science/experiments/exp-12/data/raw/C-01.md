Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several critical flaws that create real double-charge risk, timeout arithmetic violations, and unhandled failure paths that would cause silent data loss or incorrect state in production. These must be resolved before merge.

---

## Critical Issues

### 1. Payment Call Outside the Database Transaction — Double-Charge Risk

This is the most severe flaw in the PR. The comment in the code claims crash safety, but the implementation does not deliver it.

```python
# CURRENT — payment.charge() is inside db.transaction() block
with self.db.transaction():
    charge_id = self.payment.charge(...)        # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations only. It has no rollback authority over an already-submitted external HTTP call to the payment gateway. If `record_charge` or `mark_order_complete` raises after `charge()` succeeds, the database transaction rolls back, but the charge has already been captured at the gateway. The order now has no record of the charge. On retry, the idempotency check (`get_charge`) finds nothing and charges again.

**Required fix:** Charge outside and before the transaction. Record inside the transaction. Accept that a crash between charge and record requires a reconciliation job, not a transaction boundary.

```python
# Correct pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

A dead-letter / reconciliation mechanism is still needed for the narrow crash window between charge and record, but this structure at least prevents the retry path from double-charging.

### 2. Lock TTL vs. Payment Timeout Arithmetic — Lock Expires Under Live Payment

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5
```

Worst-case payment client wall time:

```
3 attempts × 30s timeout = 90s
2 inter-attempt sleeps   × 5s = 10s
Total                    = 100s
```

The lock TTL is 60 seconds. A second worker can acquire the lock at T+60 while the first worker is still in its second or third payment retry. The idempotency check runs after lock acquisition, but if the first worker has not yet written the charge record (because it is still waiting on the gateway), the second worker sees no existing charge and proceeds to charge again.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at least `(MAX_RETRIES × REQUEST_TIMEOUT) + (MAX_RETRIES × RETRY_BACKOFF) + safety_margin` (≥ 120s recommended), or implement lock extension (Redlock-style keepalive) during payment processing. The constants must be co-located or derived from one another so this relationship cannot drift silently.

```python
ORDER_LOCK_TTL = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + PaymentClient.MAX_RETRIES * PaymentClient.RETRY_BACKOFF
    + 30  # safety margin
)
```

### 3. `PaymentError` and Unhandled Exceptions Escape the Lock Context — Lock Released, State Unknown

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError on failure
        ...
# PaymentError propagates out — lock is released by finally block
# process() has no except for PaymentError
```

`PaymentError` is not caught in `process()`. It propagates to the caller as an unhandled exception. The lock is correctly released by the `finally` block, but the caller receives a raw exception rather than an `OrderResult`. Any caller that does not explicitly catch `PaymentError` will crash. More importantly, a transient payment failure that exhausts retries leaves the order in an ambiguous state with no structured result returned.

Similarly, `db.get_charge()`, `db.record_charge()`, and `db.mark_order_complete()` can raise database exceptions that are also unhandled, producing the same raw exception escape.

**Required fix:** Catch `PaymentError` and database exceptions within `process()` and return structured `OrderResult` failures.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 4. Timeout on Payment Retry Is a Gateway Ambiguity — Retry May Double-Charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge was not captured. The request may have reached the gateway and been processed; only the response was lost in transit. Retrying on timeout without a gateway-side idempotency key will submit a second charge attempt that the gateway treats as a new, independent charge.

**Required fix:** The `charge()` call must include a caller-generated idempotency key passed as a request header (e.g., `Idempotency-Key: <uuid>`). This key must be stable across retries for the same logical charge attempt. Without this, the retry logic actively creates double-charge risk at the network layer.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    ...
    response = requests.post(
        self.CHARGE_ENDPOINT,
        json={"token": token, "amount": amount_cents},
        headers={"Idempotency-Key": idempotency_key},
        timeout=self.REQUEST_TIMEOUT,
    )
```

## Moderate Issues

### 5. Redis Unavailability Is Not Handled — Hard Crash on Infrastructure Failure

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If Redis is unavailable, `redis.set()` raises `redis.exceptions.ConnectionError` or `redis.exceptions.RedisError`. This propagates uncaught through `_acquire_lock` and `process()`, producing a raw exception to the caller. In a degraded infrastructure scenario, every order request crashes instead of returning a structured failure.

**Required fix:** Catch `redis.exceptions.RedisError` in `_acquire_lock` or `process()` and return a structured result or raise `LockError` with a clear message.

---

### 6. Lock Retry Backoff Is Additive, Not Exponential — Thundering Herd Under Contention

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — linear
```

Under high concurrency, all contending workers retry on the same linear schedule, increasing collision probability. Standard practice is exponential backoff with jitter.

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.1)
```

---

### 7. `charge_id` Key Access on Payment Response Is Unguarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id` key, or non-JSON content type), this raises `KeyError` or `JSONDecodeError` unhandled. The retry loop does not catch these, so they propagate as raw exceptions.

**Required fix:** Validate the response structure before key access.

```python
data = response.json()
if "charge_id" not in data:
    raise PaymentError(f"Malformed gateway response: missing charge_id")
charge_id = data["charge_id"]
```

## Test Coverage Gaps

### 8. No Test for Lock TTL Expiry During Payment

The test suite does not cover the scenario where the lock expires mid-payment. This is the exact race condition that Issue 2 describes and it should have a regression test.

### 9. No Test for `PaymentError` Propagation

There is no test verifying that a `PaymentError` from the payment client results in a structured `OrderResult` rather than an unhandled exception. Given Issue 3, this test would currently fail (the exception escapes).

### 10. No Test for Timeout Retry Behavior with Idempotency Key

The payment client retry-on-timeout path has no test asserting that the same idempotency key is used across retry attempts.

### 11. `test_lock_release_uses_lua_script` Is Fragile

```python
processor._release_script.assert_called_once()
```

This asserts the script was called but does not assert it was called with the correct `keys` and `args`. A regression that passes the wrong token would not be caught.

```python
processor._release_script.assert_called_once_with(
    keys=[f"order:lock:order-001"], args=[ANY]
)
```
