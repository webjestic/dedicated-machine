Decision: Request Changes

Summary: The architecture demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several non-trivial correctness and security issues that must be resolved before this can be approved, the most critical being a charge-before-commit race that directly undermines the at-most-once guarantee the PR claims to provide. The issues range from a fundamental distributed systems correctness flaw to missing security controls on a payment-sensitive code path.

---

## Findings

### 🔴 Critical — Charge Occurs Outside the Database Transaction (Correctness Violation)

This is the most serious issue in the PR. The docstring explicitly claims:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

That claim is **false** as written.

```python
# order_processor.py lines ~85-92
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External network call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Problem 1 — The charge is not atomic with the DB write.** The payment gateway is an external system. Wrapping `self.payment.charge()` inside `self.db.transaction()` does not make them atomic. If the charge succeeds but the DB transaction rolls back (deadlock, constraint violation, connection drop), the charge is real and unrecorded. The database transaction boundary provides no protection over a network call to a third-party system.

**Problem 2 — DB connection held open during a 30-second network call.** `REQUEST_TIMEOUT = 30` seconds, with up to 3 retries at 5-second backoff intervals. That is potentially **~105 seconds** of holding an open database transaction and connection. Under any meaningful concurrency this will exhaust your connection pool and cause cascading failures.

**The correct pattern** is to perform the charge *outside and before* the transaction, then record it atomically:

```python
# Charge first, outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record atomically — if this fails, the idempotency check
# on retry will catch the missing record and you handle reconciliation
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.error(
        "RECONCILIATION REQUIRED: charge succeeded but record failed "
        "order_id=%s charge_id=%s", req.order_id, charge_id
    )
    raise
```

This also requires a reconciliation strategy and an alert path — the comment in the except block above is not sufficient on its own. There needs to be a durable dead-letter or reconciliation queue for this failure mode.

### 🔴 Critical — Lock TTL Does Not Cover Actual Processing Time

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

The payment client can take up to `30s timeout × 3 retries + 5s backoff × 2 gaps = 100s` in the worst case. The lock expires in 60 seconds. A second worker can acquire the lock on the same order while the first worker is still mid-charge, directly enabling the double-charge the lock is designed to prevent.

Either the TTL must be set conservatively above the maximum possible processing time, or the lock must be extended (refreshed) during long operations. The TTL comment saying it "covers typical payment round-trip" is not a safe design basis for a financial system — you must design for the worst case.

---

### 🟠 High — `payment_token` Logged Indirectly via Pydantic ValidationError

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions and configurations. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in the log output. Payment tokens are sensitive credentials. This must be sanitized:

```python
except ValidationError as e:
    # Enumerate only field names and error types — never log field values
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

---

### 🟠 High — No Authentication or Authorization on the Payment Endpoint

`PaymentClient` posts to `https://api.payments.internal/v1/charges` with no authentication header, no mTLS configuration, and no API key. Even on an internal network, unauthenticated calls to a payment endpoint are not acceptable. This needs at minimum an injected credential and a clear note on how that credential is provisioned and rotated.

---

### 🟠 High — `payment_token` Exposed in Payment Gateway Request Body Without Masking in Logs

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This specific line is fine, but the `requests.post` call sends `{"token": token, ...}` and any request-level debug logging (e.g., from `requests` internals, middleware, or an HTTP proxy) will expose the raw token. The client should explicitly document that debug-level HTTP logging must be suppressed in production, or use a wrapper that masks the token field.

---

### 🟡 Medium — Lock Retry Backoff Is Linear and Blocks the Worker Thread

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep` in a synchronous worker is acceptable in isolation, but the retry window totals 3 seconds of blocking. If this service runs in a threaded or async context, this is a thread-starvation vector. Exponential backoff with jitter is the standard practice for distributed lock contention and would reduce thundering herd behavior under high concurrency:

```python
import random
backoff = min(0.1 * (2 ** attempt) + random.uniform(0, 0.1), 2.0)
```

---

### 🟡 Medium — Idempotency Check Has a TOCTOU Window Without Compensating Control

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(processed=False, ...)

with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

The Redis lock is the primary guard here, which is correct. However, if the Redis lock is lost mid-flight (TTL expiry, Redis failover), a second worker can pass the idempotency check concurrently. The database `record_charge` should enforce a **unique constraint on `order_id`** at the schema level so that even if two workers race past the application-layer check, only one insert succeeds. This is a defense-in-depth requirement for financial data, not optional.

---

### 🟡 Medium — `PaymentError` Is Not Caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # Can raise PaymentError
```

`PaymentError` is not caught in `process()`. It will propagate as an unhandled exception to the caller with no `OrderResult` returned, breaking the method's return contract. It should be caught and returned as a structured `OrderResult`:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, str(e))
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

### 🟡 Medium — Test Coverage Gaps on Critical Failure Paths

The tests do not cover:

- Lock TTL expiry mid-processing (lock lost before release)
- `PaymentError` raised during charge (currently causes unhandled exception per above)
- DB transaction rollback after successful charge (the unrecorded charge scenario)
- `requests.exceptions.ConnectionError` in `PaymentClient` (not retried, will propagate uncaught)
- Redis unavailability during lock acquisition or release

These are exactly the failure modes that matter in a financial processing service. Tests for the happy path and basic contention are necessary but not sufficient.

---

### 🔵 Low — `ConnectionError` Not Handled in `PaymentClient`

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
```

`requests.exceptions.ConnectionError` (DNS failure, refused connection, network partition) is not caught and will propagate uncaught through the retry loop. It should be treated as a transient failure and retried under the same logic as `Timeout`.

---

### 🔵 Low — Hardcoded Internal Endpoint URL

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs in source are an operational and security concern — they cannot be rotated, environment-overridden, or validated at startup. This should be injected via configuration or environment variable with a startup assertion that it is set.
