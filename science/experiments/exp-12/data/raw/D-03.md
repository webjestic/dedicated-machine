Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that precedes a database commit failure will result in a real charge with no recorded transaction, directly contradicting the comment claiming crash safety. Additionally, there are several reliability and correctness gaps that need to be addressed before this handles real financial data.

---

## Critical Issues

### 1. Payment call inside DB transaction violates the atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is false. If `record_charge` or `mark_order_complete` raises, or the DB commit fails, the payment gateway has already been charged and the money is gone with no record. A DB transaction cannot roll back an external HTTP call.

**The correct pattern** is to charge first, then record in the DB. The idempotency check plus the distributed lock already prevent double-charging on retry. If the DB write fails after a successful charge, the operator needs a reconciliation path, not a false guarantee:

```python
# Charge outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically — if this fails, charge_id exists but is unrecorded.
# The idempotency check on retry will NOT catch this because get_charge
# queries the DB, which has no record yet. See reconciliation note below.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "UNRECORDED_CHARGE order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

This is the fundamental two-generals problem with payment systems. The PR needs to either document the accepted failure mode explicitly, or implement a reconciliation/outbox pattern.

### 2. Timeout on payment gateway can produce an ambiguous charge state

`requests.exceptions.Timeout` is retried silently, but a timeout does **not** mean the charge failed — the gateway may have processed it and the response was lost. Retrying on timeout without checking for an existing charge at the gateway level can double-charge the customer.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # This retry is unsafe — the charge may have succeeded
```

At minimum, the payment client needs to send an idempotency key to the gateway on every attempt so the gateway deduplicates, and the same key must be reused across retries for the same order:

```python
# In OrderProcessor.process(), generate once and pass through
idempotency_key = f"{req.order_id}:{req.customer_id}"

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,
)
```

```python
# In PaymentClient.charge()
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},  # reused across retries
            timeout=self.REQUEST_TIMEOUT,
        )
```

## Moderate Issues

### 3. Lock TTL is too short and not justified

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` alone can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire while a worker is still mid-payment, allowing a second worker to acquire the lock and potentially charge again.

Either extend the TTL to safely exceed the maximum payment client duration (e.g., 180s), or implement lock heartbeating. At minimum, add a comment documenting the calculation.

```python
# MAX payment duration: MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * 35 = 105s
# TTL must exceed this. Set to 180s with margin.
ORDER_LOCK_TTL = 180
```

### 4. `payment_token` is logged at INFO level indirectly via structured fields

Verify that `charge_id` and `payment_token` never appear in log output. The current code logs `charge_id` at INFO which is fine, but `payment_token` is stored in `req.payment_token` and if any exception path ever logs the full request object, PCI-DSS relevant data leaks into logs. This is low risk as written but worth a defensive scrub.

### 5. `PaymentError` is silently swallowed

`PaymentError` is not caught in `OrderProcessor.process()`. It will propagate as an unhandled exception to the caller. This may be intentional, but it is inconsistent with the design of returning `OrderResult` for all other failure modes. Either catch it and return a structured result, or document that callers must handle it:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, str(e))
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

## Minor Issues

### 6. Test for lock release does not verify the token

`test_lock_release_uses_lua_script` only asserts the script was called once. It does not verify that the correct token was passed as `args`. A future refactor could break token verification silently:

```python
def test_lock_release_uses_correct_token(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    call_args = processor._release_script.call_args
    keys = call_args.kwargs.get("keys") or call_args[1].get("keys")
    args = call_args.kwargs.get("args") or call_args[1].get("args")
    assert keys == ["order:lock:order-001"]
    assert len(args) == 1 and len(args[0]) == 36  # UUID format
```

### 7. No test for payment gateway timeout or retry behavior

`PaymentClient` has meaningful retry logic but `tests/test_order_processor.py` has zero coverage of it. Add tests for: timeout triggers retry, 5xx triggers retry, 4xx does not retry, exhausted retries raise `PaymentError`.
