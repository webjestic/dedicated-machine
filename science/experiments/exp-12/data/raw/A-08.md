# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Charge Outside Database Transaction — At-Most-Once Guarantee is False

The PR summary and inline comment both claim:

> *"crash between charge and record cannot produce an unrecorded charge"*

This is **incorrect**. The `payment.charge(...)` call is made **inside** `with self.db.transaction()`, but a database transaction provides atomicity over *database operations only*. The payment gateway call is a remote HTTP side effect — it is **not enrolled in the database transaction**. If the process crashes, the network drops, or the DB commit fails after `payment.charge()` returns a `charge_id` but before `db.record_charge()` commits, you have a real charge with no record. The comment is actively misleading and will give future maintainers false confidence.

The correct pattern for at-most-once semantics here is one of:

1. **Write-first idempotency key**: Insert a `pending` charge record with a client-generated idempotency key *before* calling the gateway, then update it to `complete` after. On recovery, check for `pending` records and reconcile.
2. **Outbox pattern**: Write intent to DB first, process via a separate reconciliation worker.
3. **Gateway idempotency key**: Pass a stable, order-scoped idempotency key to the payment gateway so that a retry of the same charge is safe and returns the same `charge_id`.

None of these are present. The current code has a silent money-leak window.

### 🔴 Critical: `PaymentError` Is Silently Swallowed Inside the Lock

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError
        self.db.record_charge(...)
```

`PaymentError` (and any other unexpected exception) is **not caught** inside `process()`. It will propagate out through the `_acquire_lock` context manager's `finally` block (lock is correctly released — good), but then it surfaces as an unhandled exception to the caller. There is no `OrderResult` returned for this path. Depending on the caller, this either crashes a worker or produces an unstructured 500. A `PaymentError` should be caught and returned as a structured `OrderResult(processed=False, ...)`.

---

### 🔴 Critical: Redis Timeout on Gateway Call Can Expire the Lock Mid-Flight

`ORDER_LOCK_TTL = 60s` and `PaymentClient.REQUEST_TIMEOUT = 30s` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5s`. Worst-case payment duration:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

This **exceeds the 60-second lock TTL**. A second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still mid-flight. The distributed lock provides no protection in this scenario. Either the TTL must be extended to safely exceed the maximum payment duration, or a lock-extension (watchdog/renewal) mechanism must be implemented.

---

### 🟡 Significant: Idempotency Check Has a Read-Your-Writes Hazard

```python
existing = self.db.get_charge(req.order_id)
```

If the database uses a read replica or has replication lag, this read may not reflect a charge committed milliseconds earlier by another worker on a different connection. The lock *should* serialize this, but only if the lock TTL concern above is resolved. Even with a correct TTL, if `get_charge` hits a replica, the check is unreliable. This should explicitly target the primary/leader node.

---

### 🟡 Significant: `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation (e.g., too long), its value or a truncated form may appear in the log output. Payment tokens are sensitive credentials. The log message should be sanitized or the error should be serialized with field names only, not values.

---

### 🟡 Significant: Lock Retry Backoff Is Blocking — Ties Up a Worker Thread

```python
time.sleep(backoff)
```

In any async or thread-pool worker context, a blocking `time.sleep` inside lock acquisition holds the thread for up to `0.5 + 1.0 = 1.5s` before giving up. Under high concurrency this degrades throughput. This should use `asyncio.sleep` in async contexts or be documented explicitly as synchronous-only with a note that async callers must wrap in an executor.

## `services/payment_client.py`

### 🟡 Significant: Timeout Retry Semantics Are Ambiguous — Risk of Double Charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retry
```

A timeout means the request may have been received and processed by the gateway — the response was simply not received in time. Retrying on timeout **without a stable idempotency key** can result in the customer being charged multiple times. This is the same root issue as above but surfaced here: all charge attempts must carry a stable, order-scoped idempotency key in the request headers/body so the gateway can deduplicate.

---

### 🟠 Minor: No Authentication on Payment Gateway Request

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no API credentials, bearer tokens, or mTLS configuration present. Even for an internal endpoint, this is missing. This may be handled at the infrastructure layer (service mesh, mTLS sidecar), but if so, that assumption should be documented. As written, it is indistinguishable from a misconfiguration.

---

### 🟠 Minor: `response.json()["charge_id"]` — No `KeyError` / Malformed Response Handling

If the gateway returns a 200 with a body that does not contain `charge_id`, this raises an unhandled `KeyError`. This should be wrapped with an explicit check and raised as a `PaymentError` with a descriptive message.

## `tests/test_order_processor.py`

### 🟠 Minor: No Test for `PaymentError` Propagation Path

There is no test covering what happens when `payment.charge()` raises `PaymentError`. Given that this is an unhandled exception path in `process()`, the absence of a test means the failure mode is untested and the crash behavior is invisible in CI.

---

### 🟠 Minor: Lock Contention Test Does Not Exercise Backoff Timing

`test_process_lock_contention` sets `redis.set.return_value = False` but does not assert that `time.sleep` was called with the correct backoff values across the retry attempts. The retry-with-backoff logic is untested.

---

### 🟠 Minor: Transaction Rollback on `PaymentError` Is Not Tested

There is no test verifying that if `payment.charge()` raises mid-transaction, `db.record_charge` and `db.mark_order_complete` are **not** called and the transaction is rolled back. This is the most important correctness property of the transaction block and it has no coverage.

---

**Decision: Request Changes**

**Summary:** This PR contains three critical correctness flaws that directly undermine its stated guarantees: the charge-outside-transaction window creates a real money-leak path that the inline comment falsely claims is closed; unhandled `PaymentError` propagation leaves the caller with no structured failure result; and the lock TTL is shorter than the worst-case payment duration, meaning the distributed lock provides no protection under retry exhaustion. The idempotency and at-most-once guarantees advertised in the PR description are not actually implemented. All three critical issues must be resolved before this merges.
