Decision: Request Changes

Summary: The core locking and idempotency architecture is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check are all solid. However, there is a critical atomicity flaw that directly undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issue

### Payment charge occurs inside the database transaction — but the DB cannot roll back an external charge

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. If `self.payment.charge()` succeeds and then the process crashes before `record_charge` commits, the charge exists at the gateway but not in your database. The DB transaction wrapping this provides zero protection against that scenario because the payment gateway call is not a DB operation and cannot be rolled back by the DB.

The standard pattern here is:

1. Write a `PENDING` charge record to the DB **before** calling the gateway (inside a transaction)
2. Call the gateway
3. Update the record to `COMPLETED` with the returned `charge_id`
4. Handle the crash-between-2-and-3 case with a reconciliation job that checks pending charges against the gateway

Alternatively, use an idempotency key on the gateway call so a retry of the same logical charge returns the same `charge_id` rather than creating a duplicate, then record that result.

## Significant Issues

### Timeout on payment gateway creates a dangerous ambiguity

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout does not mean the charge failed — it means you do not know. The gateway may have processed the charge and the response was lost in transit. Retrying after a timeout without an idempotency key on the request can produce duplicate charges. The `charge` call needs to include a stable idempotency key (e.g., derived from `order_id`) in the request headers or body so the gateway deduplicates on its end.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # ← required
    timeout=self.REQUEST_TIMEOUT,
)
```

### Lock TTL is too short for the current retry configuration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient.REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds between retries. Worst case: 3 × 30s timeouts + 2 × 5s backoffs = 100 seconds. The lock expires before the payment client gives up, which means a second worker can acquire the lock and begin processing while the first worker is still mid-flight. Either increase the TTL to safely exceed the maximum payment client duration (e.g., 120–150s) or reduce the payment client's maximum total duration.

## Moderate Issues

### `payment_token` is logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some configurations. `payment_token` is a sensitive credential. Ensure the log output is scrubbed or that Pydantic is configured to hide this field's value in error representations. At minimum, add `repr=False` to the `payment_token` field definition.

```python
payment_token: str = Field(..., min_length=1, max_length=256, repr=False)
```

### `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$` but `customer_id` has no equivalent constraint. If `customer_id` is used in DB queries or log output downstream, this is an inconsistency worth closing.

### Lock retry backoff sleeps on the final attempt

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)  # ← sleeps even after the last failed attempt
```

The `if attempt < self.MAX_RETRIES - 1` guard present in `PaymentClient` is missing here. The final failed attempt sleeps 1.5 seconds before raising `LockError`. Minor, but wasteful.

## Minor Issues

### Test for lock release does not verify the token passed to the script

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This confirms the script was called but does not verify the correct token was passed as `args`. A test that asserts `args=[<expected_token>]` would actually validate the token-verification contract.

### No test covering the payment gateway timeout/retry path

The retry logic in `PaymentClient` is untested. Given the atomicity concerns above, this path needs coverage, particularly the timeout-then-succeed scenario.

### `charge_id` absent from `OrderResult` on already-processed path

```python
return OrderResult(
    processed=False,
    order_id=req.order_id,
    message="Order already processed",
)
```

The existing charge record is fetched (`existing = self.db.get_charge(...)`) but the `charge_id` from that record is not returned to the caller. Callers handling idempotent retries likely need the original `charge_id` to confirm what happened.
