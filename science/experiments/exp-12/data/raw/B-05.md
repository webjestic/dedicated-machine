Decision: Request Changes

Summary: The PR demonstrates solid fundamentals — atomic SET NX EX acquisition, token-verified Lua release, Pydantic validation, and idempotency intent are all present and correctly structured. However, there are several critical correctness flaws and one significant security issue that must be resolved before this can be approved: the payment charge operation escapes the database transaction boundary creating an unrecorded-charge window, the idempotency check has a TOCTOU race under Redis failure scenarios, and the payment client logs and propagates raw gateway response data without sanitization.

---

## Detailed Findings

### CRITICAL — Payment Charge Occurs Inside `db.transaction()` But Cannot Be Rolled Back

**File:** `services/order_processor.py`, lines within `with self.db.transaction():`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side effect — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is an external HTTP side effect. It is not part of the database transaction. If `record_charge` or `mark_order_complete` raise after the charge succeeds, the database rolls back but the charge has already been submitted to the payment gateway. The comment in the code ("crash between charge and record cannot produce an unrecorded charge") is **factually incorrect** — this is exactly the scenario that produces an unrecorded charge.

**Required fix:** The charge must occur *before* the transaction opens, or you must implement a two-phase approach: record a *pending* charge intent inside the transaction first, then execute the external charge, then finalize. The canonical pattern is:

```python
# 1. Record pending intent atomically — if we crash here, no charge was attempted
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

# 2. Execute external charge — idempotency key ties this to the pending record
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-level dedup
)

# 3. Finalize — if we crash here, a reconciliation job can recover from the pending record
with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This also requires the payment gateway to support idempotency keys, which the current `PaymentClient` does not pass. See finding below.

### CRITICAL — Missing Idempotency Key on Payment Gateway Calls

**File:** `services/payment_client.py`

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

The retry loop re-submits the charge on timeout with no idempotency key. A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without an idempotency key will produce duplicate charges. This is a billing correctness vulnerability.

**Required fix:** Pass `order_id` as an idempotency key in the request (header or body per gateway spec). The `charge()` method signature must accept it:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    ...
    headers = {"Idempotency-Key": idempotency_key}
    response = requests.post(..., headers=headers, ...)
```

### HIGH — Redis Lock TTL Is Insufficient Under Payment Retry Scenarios

**File:** `services/order_processor.py` and `services/payment_client.py`

```python
ORDER_LOCK_TTL = 60        # order_processor.py
REQUEST_TIMEOUT = 30       # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Maximum payment client wall time: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock TTL is 60 seconds. A slow payment gateway can cause the lock to expire while the worker is still inside the critical section, allowing a second worker to acquire the lock and begin processing the same order concurrently. The idempotency check will not save you here because the second worker runs `get_charge` before the first worker has committed.

**Required fix:** Either raise `ORDER_LOCK_TTL` to safely exceed maximum payment client wall time (minimum 120s with margin), or implement lock extension (Redlock-style watchdog). Also consider reducing `REQUEST_TIMEOUT` and `MAX_RETRIES` to fit within the TTL budget with explicit documentation of the constraint.

### HIGH — Unhandled Exception Paths Leave Lock Held Until TTL Expiry (Acceptable) But Suppress Errors Silently (Not Acceptable)

**File:** `services/order_processor.py`

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Any exception *other than* `LockError` raised inside the `with self._acquire_lock(...)` block — including `PaymentError`, database errors, or unexpected runtime exceptions — propagates out of `process()` uncaught. The lock is correctly released by the `finally` block, so that part is fine. However, the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the stated contract of the method and may expose internal stack traces to upstream callers depending on the web framework in use.

**Required fix:** Add a broad exception handler that returns a structured error result and logs the exception with full context:

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

### MEDIUM — Security: Raw Gateway Response Data Logged and Propagated Without Sanitization

**File:** `services/payment_client.py`

```python
raise PaymentError(f"Payment declined: {e.response.status_code}") from e
```

```python
logger.warning("Payment gateway error attempt=%d status=%d", attempt + 1, e.response.status_code)
```

The response body from the payment gateway is not examined or sanitized before logging. If the gateway returns error detail in the body (card numbers in error messages, PII in decline reasons), those will be written to logs. Additionally, `PaymentError` messages may propagate to API responses depending on how the caller handles them.

**Required fix:** Never log or propagate raw gateway response bodies. Log only status codes and safe structured fields. Ensure `PaymentError` messages exposed to callers contain no gateway-sourced content.

### MEDIUM — `response.json()["charge_id"]` Will Raise `KeyError` on Malformed Gateway Response

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

A `200 OK` response from the gateway with a missing or renamed `charge_id` field raises an unhandled `KeyError`, which will propagate as an unexpected exception type rather than a `PaymentError`. This breaks the exception contract of the method.

**Required fix:**

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned 200 but no charge_id in response")
```

### LOW — Test Coverage Gap: Lock TTL Expiry During Processing Not Tested

**File:** `tests/test_order_processor.py`

There is no test covering the scenario where the Redis lock expires mid-processing (TTL fires while the worker is inside the critical section). This is the most dangerous race condition in the design and should have an explicit test demonstrating the behavior, even if the current code cannot fully prevent it.

---

### LOW — `customer_id` Field Has No Pattern Constraint

**File:** `services/order_processor.py`

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. If `customer_id` is used in database queries or log output, it should have equivalent constraints.

| High | Unhandled exceptions escape `process()` without structured response |
| Medium | Raw gateway response body may be logged or propagated — PCI/PII risk |
