Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that is followed by a DB commit failure will produce an unrecorded, unrefundable charge despite the comment claiming otherwise. Additionally, there are several reliability and correctness gaps that need to be addressed before this touches production payment infrastructure.

---

## Critical Issues

### 1. Payment call inside DB transaction breaks the atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is false. A DB transaction does not wrap an HTTP call. If `record_charge` or `mark_order_complete` raises, or the DB commit fails, the charge has already been submitted to the payment gateway and cannot be rolled back. The DB transaction only protects the two DB writes from each other.

**Fix:** Charge first, outside the transaction, then record inside it. On any post-charge DB failure, log the orphaned `charge_id` immediately and alert — do not silently swallow it.

```python
# Correct pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# If we crash here, the lock TTL expires and the idempotency check on retry
# will miss this charge — see issue #2 below.

try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "UNRECORDED CHARGE — manual reconciliation required "
        "order_id=%s charge_id=%s amount_cents=%d",
        req.order_id, charge_id, req.amount_cents,
    )
    raise
```

---

### 2. Idempotency check does not cover in-flight or post-charge/pre-record state

The idempotency check queries `db.get_charge()` before charging. If a worker crashes after charging but before recording, the lock expires, a second worker acquires it, finds no DB record, and charges again. This is a double-charge scenario.

**Fix:** Introduce a `pending` state written to the DB *before* the charge is submitted. The idempotency check must treat `pending` records as blocking, not just completed ones. On retry, a `pending` record older than a safe threshold should trigger a gateway lookup to determine whether the prior charge succeeded before proceeding.

---

### 3. `PaymentError` is not caught in `process()`

```python
# In process() — no handler for PaymentError
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)
```

An unhandled `PaymentError` propagates out of `_acquire_lock`'s `try` block, the `finally` correctly releases the lock, but the caller receives an unhandled exception rather than a structured `OrderResult`. Given this is a service boundary, all known failure modes should return a typed result.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

## Moderate Issues

### 4. Timeout on payment gateway is dangerously long relative to lock TTL

`REQUEST_TIMEOUT = 30s`, `MAX_RETRIES = 3`, `RETRY_BACKOFF = 5s`. Worst case: ~105 seconds of payment client activity. `ORDER_LOCK_TTL = 60s`. The lock can expire while a payment attempt is still in progress, allowing a second worker to acquire the lock and begin processing the same order concurrently. The lock TTL must exceed the maximum possible payment client runtime, or the payment client's total budget must be strictly bounded below the TTL.

```
max_payment_time = MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF
                 = 3 * 30 + 2 * 5 = 100 seconds  >  ORDER_LOCK_TTL (60s)
```

Either reduce the payment client budget or increase the lock TTL with a documented justification.

---

### 5. Retry on timeout is unsafe without idempotency key

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # retries unconditionally
```

A timeout does not mean the charge was not processed — the gateway may have received and executed the request. Retrying without an idempotency key sent to the gateway can produce duplicate charges at the gateway level, independent of the Redis lock. The `charge()` call must include a stable idempotency key (e.g., derived from `order_id`) in the request headers.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # pass order_id-derived key
    timeout=self.REQUEST_TIMEOUT,
)
```

---

### 6. `payment_token` logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation, its value may appear in logs. Payment tokens must never appear in log output. Sanitize or explicitly exclude sensitive fields from the logged error.

## Minor Issues

### 7. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that `redis.delete` was not called, which is an implementation detail. More importantly, there is no test verifying that the lock is released when `payment.charge()` raises — the `finally` block is the most critical path and it has no coverage.

```python
def test_lock_released_on_payment_failure(processor, redis_client, payment):
    payment.charge.side_effect = Exception("gateway down")
    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    processor._release_script.assert_called_once()
```

### 8. `charge_id` not included in `OrderResult` on already-processed path

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id missing — caller cannot retrieve the prior charge
    )
```

Callers receiving a duplicate-detection response have no way to retrieve the existing `charge_id` for reconciliation or display. Return it from `existing`.
