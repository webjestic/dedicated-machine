# Pull Request Review

**PR:** Add distributed order processor with Redis-backed exclusive locking
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/order-processor`

---

## Detailed Findings

### 🔴 Critical — Charge Outside Database Transaction Boundary

This is the most severe architectural defect in the PR and directly contradicts the safety claim in the docstring.

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(        # ← EXTERNAL SIDE EFFECT
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims atomicity, but `self.payment.charge()` is an **external HTTP call to a third-party payment gateway**. It is entirely outside the database transaction boundary and cannot be rolled back. If `record_charge` or `mark_order_complete` fail after a successful charge, the database transaction rolls back, leaving a real charge with no record. The idempotency check on the next attempt will find no existing charge and **charge the customer a second time**. This is a double-charge vulnerability. The payment call must be moved **before** the transaction block, and the transaction should only record an already-completed charge. The architecture needs to be restructured as: charge → open transaction → record → commit, with compensating logic (refund) on record failure.

### 🔴 Critical — Timeout During Payment Creates Undetectable Duplicate Charge Risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# ... falls through to retry
```

When a timeout occurs, the payment gateway may have already processed the charge — the response simply didn't arrive in time. The client then **retries the same token**, potentially issuing a second charge. There is no idempotency key being sent to the payment gateway on the outbound request. The `charge()` call must include a stable, per-attempt or per-order idempotency key in the request headers (e.g., `Idempotency-Key: <order_id>`) so the gateway can deduplicate retries on its end. Without this, the retry logic is actively dangerous.

```python
# Current — no idempotency key
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)

# Required
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # stable per order
    timeout=self.REQUEST_TIMEOUT,
)
```

### 🔴 Critical — Lock TTL Does Not Cover Full Operation Duration

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

The lock TTL is 60 seconds. The payment client alone can consume up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The lock will expire mid-operation while a worker is still inside the critical section, allowing a second worker to acquire the lock and begin processing the same order concurrently. The TTL must exceed the maximum possible operation duration, or the system must implement lock extension (heartbeat/renewal) for long-running operations. At minimum, `ORDER_LOCK_TTL` must be raised to at least 120–150 seconds, and the relationship between these constants must be enforced programmatically, not left as independent magic numbers.

### 🔴 Critical — `payment_token` Logged or Exposed via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic `ValidationError` string representations include the field values that failed validation. If a `payment_token` is malformed and triggers a validation error, the raw token value will be written to the log. Payment tokens are sensitive credentials. The log statement must redact or exclude field values, logging only field names and error types.

### 🟠 High — Unhandled Exceptions Inside Lock Context Silently Leak the Lock to TTL Expiry

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

Any unhandled exception from `db.get_charge`, `db.transaction`, `db.record_charge`, `db.mark_order_complete`, or `payment.charge` propagates out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` does release the lock in this case — that part is correct. However, the `process()` method has **no outer exception handler** for these paths. The exception will propagate to the caller with no `OrderResult` returned, no structured logging of the failure, and no indication of whether the charge succeeded before the failure. The caller contract (always returns `OrderResult`) is violated. A catch-all handler with appropriate logging and a safe `OrderResult(processed=False, ...)` return is required, with careful distinction between pre-charge and post-charge failure states.

### 🟠 High — `PaymentError` Is Not Caught in `process()`

```python
# In payment_client.py
raise PaymentError(f"Payment declined: {e.response.status_code}") from e
raise PaymentError(f"Payment failed after {self.MAX_RETRIES} attempts: {last_error}")
```

```python
# In order_processor.py — only catches LockError
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

`PaymentError` is never caught in `process()`. A payment decline or gateway failure will raise an unhandled exception through the lock context manager (the lock will be released by `finally`, which is correct), but the caller receives an uncaught exception instead of an `OrderResult`. This breaks the method's documented return contract and likely causes a 500-level error at the API layer with no structured response to the client.

---

### 🟠 High — Redis Failure Is Not Handled

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

If the Redis connection is unavailable, `redis.set()` raises a `redis.exceptions.ConnectionError` or `redis.exceptions.RedisError`. This is not caught anywhere. In a distributed system, Redis unavailability is a real operational scenario. The failure mode here is that the entire order processing pipeline throws an unhandled exception. The system should have an explicit policy for Redis unavailability — either fail closed (reject the order with a clear error) or, if a fallback is acceptable, document and implement it. Currently it fails in an uncontrolled manner.

---

### 🟡 Medium — Lock Retry Backoff Is Deterministic and Collision-Prone

```python
backoff = 0.5 * (attempt + 1)
```

Under high concurrency, all workers contending on the same lock will sleep for identical durations (0.5s, 1.0s, 1.5s) and wake up simultaneously, creating a thundering herd on lock re-acquisition. Standard practice is exponential backoff with jitter: `backoff = min(cap, base * 2**attempt) + random.uniform(0, jitter)`.

---

### 🟡 Medium — `charge_id` Returned from `payment.charge()` Is Not Validated

```python
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
```

If `payment.charge()` returns `None`, an empty string, or a malformed value (e.g., due to an unexpected gateway response shape), it is passed directly into `db.record_charge()` and returned in `OrderResult`. There is no validation that `charge_id` is a non-empty string before it is persisted. A null or empty `charge_id` in the database is a silent data integrity failure.

---

### 🟡 Medium — Test Suite Does Not Cover the Critical Failure Paths

The tests cover the happy path, already-processed idempotency, lock contention, and input validation. The following critical scenarios have **zero test coverage**:

- Payment succeeds but `record_charge` raises — does the charge get double-billed on retry?
- `PaymentError` raised — does `process()` return an `OrderResult` or throw?
- Redis connection failure during lock acquisition
- Lock TTL expiry mid-operation (second worker acquires lock while first is still in payment)
- Timeout on payment with retry — is the same token reused, risking double charge?
- `payment.charge()` returns `None` or empty string

These are precisely the scenarios that matter most for a payment processing system and they are all absent.
