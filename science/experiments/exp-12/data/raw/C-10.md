Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly motivated. However, there are several critical flaws that would allow double-charges in production, silent data corruption on payment timeouts, and a lock TTL that is arithmetically guaranteed to expire before the operation it protects can complete. These are not edge cases; they are deterministic failure paths under normal operating conditions.

---

## Critical Findings

### 1. Lock TTL vs. Total Operation Time — Arithmetic Failure (Critical)

The lock TTL is 60 seconds. The payment client alone can consume up to **135 seconds** before raising `PaymentError`:

```
MAX_RETRIES=3 attempts × REQUEST_TIMEOUT=30s = 90s of blocking I/O
+ RETRY_BACKOFF=5s × 2 inter-attempt sleeps = 10s
+ lock acquisition retries: LOCK_RETRY_ATTEMPTS=3 × 0.5+1.0+1.5s = 3s
─────────────────────────────────────────────────────────────────────
Worst-case time inside the lock: ~103s (payment alone)
Lock TTL:                          60s
```

The lock **expires while the payment is in flight**. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. Both workers then attempt to record — one silently wins, one silently loses, and the customer is double-charged with no alert.

**Required fix:** TTL must be strictly greater than the maximum possible time spent inside the lock, with margin. Either raise `ORDER_LOCK_TTL` to ≥ 180s, or — better — make `PaymentClient` expose its worst-case duration and derive TTL from it programmatically so they cannot drift independently:

```python
ORDER_LOCK_TTL = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
    + LOCK_SAFETY_MARGIN_SECONDS  # e.g., 30s
)
```

### 2. Payment Timeout Is Not Idempotent — Silent Double-Charge on Retry (Critical)

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# ... falls through to retry
```

A `Timeout` means the **request was delivered but the response was not received**. The charge may have already been executed by the gateway. Retrying unconditionally issues a second charge request with no idempotency key, producing a duplicate charge with no detection mechanism.

**Required fix:** Every charge request must carry a stable idempotency key derived from `order_id` (not a random UUID per attempt), sent as a header or body field that the gateway deduplicates on:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": f"order-{order_id}"},  # stable per order
    timeout=self.REQUEST_TIMEOUT,
)
```

`order_id` must be threaded into `PaymentClient.charge()`. Without this, retry-on-timeout is strictly more dangerous than not retrying.

### 3. Charge Occurs Inside `db.transaction()` — Distributed Transaction Violation (Critical)

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations only. Placing an external HTTP call inside it creates a two-phase commit problem with no coordinator:

- If `payment.charge()` succeeds but the DB transaction rolls back (deadlock, constraint violation, connection loss), the charge is unrecorded and unrecoverable without manual reconciliation.
- The DB transaction holds locks for the full duration of the payment network call (up to 30s per attempt × 3), causing lock contention and connection pool exhaustion under load.

**Required fix:** Charge outside the transaction, record inside it. Use the idempotency check as the guard:

```python
# Outside transaction — charge first
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# Inside transaction — record atomically
with self.db.transaction():
    # Re-check idempotency inside the transaction under a row-level lock
    if self.db.get_charge_for_update(req.order_id):
        # Charge already recorded by a concurrent worker; void if gateway supports it
        self.payment.void_if_possible(charge_id)
        return OrderResult(processed=False, order_id=req.order_id, message="Already processed")
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 4. Idempotency Check Is Not Under a Database Row Lock (High)

```python
existing = self.db.get_charge(req.order_id)   # plain SELECT
if existing:
    return ...
# ... charge happens
```

The Redis lock prevents concurrent workers from reaching this point simultaneously — **as long as the lock TTL holds**. But given finding #1, the TTL can expire. When it does, two workers can both pass this check and both charge. Even with a corrected TTL, defense-in-depth requires the DB-level idempotency check to use a `SELECT ... FOR UPDATE` or equivalent row-level lock so the database itself enforces the invariant independently of Redis.

---

### 5. `payment_token` Logged Indirectly via Pydantic Validation Errors (Medium)

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include the field value in their string representation. If `payment_token` fails validation (e.g., exceeds 256 characters), the raw token value appears in log output. Payment tokens are credentials; they must not appear in logs.

**Required fix:** Sanitize or suppress field values for sensitive fields before logging, or catch field-specific errors and log only field names:

```python
SENSITIVE_FIELDS = {"payment_token"}
safe_errors = [
    {"field": e["loc"], "msg": e["msg"]}
    if e["loc"][-1] not in SENSITIVE_FIELDS
    else {"field": e["loc"], "msg": "value redacted"}
    for e in exc.errors()
]
logger.warning("Invalid order request: %s", safe_errors)
```

### 6. Lock Retry Backoff Consumes Lock TTL (Medium)

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)  # 0.5s, 1.0s, 1.5s = 3s total
```

This sleep occurs **before** the lock is acquired, so it does not consume the TTL of the held lock. This is not a bug in isolation, but combined with finding #1, it means a worker that acquires the lock on attempt 3 has already waited 3 seconds, reducing the effective time budget inside the lock further. Document this interaction explicitly, and account for it in the TTL derivation.

---

### 7. Test Coverage Gaps (Medium)

The tests do not cover the following paths, all of which correspond to the critical findings above:

| Missing Test | Finding |
|---|---|
| Payment timeout followed by retry — assert single charge issued | #2 |
| Lock TTL expiry mid-payment — assert second worker blocked at DB level | #1, #4 |
| DB transaction rollback after successful charge — assert charge is recoverable | #3 |
| `PaymentError` raised inside `db.transaction()` — assert transaction rolled back | #3 |
| Validation error on `payment_token` — assert token not present in log output | #5 |

The existing tests mock at too high a level to catch the distributed failure modes. Consider adding integration-style tests with a real Redis instance (e.g., `fakeredis`) and a DB mock that can simulate rollback.

## Minor Notes

- `requests.post` with no session reuse creates a new TCP connection per call. Under retry load this is wasteful; use a `requests.Session` with a `HTTPAdapter` configured for retry transport.
- `CHARGE_ENDPOINT` hardcoded as a class attribute makes environment-specific configuration impossible without subclassing. Move to constructor parameter or environment variable.
- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as bare class attributes with no enforcement of their relationship to `PaymentClient` constants is the root cause of finding #1. These constants need to live in a shared configuration object that validates their consistency at startup.

---

## What to Do Before Re-Review

1. Fix TTL arithmetic — derive `ORDER_LOCK_TTL` from `PaymentClient` timing constants with explicit margin.
2. Add stable idempotency key to every payment charge request; thread `order_id` into `PaymentClient.charge()`.
3. Move `payment.charge()` outside `db.transaction()`; add `SELECT FOR UPDATE` inside the transaction as a second guard.
4. Redact `payment_token` from validation error logs.
5. Add tests for timeout retry idempotency, post-charge DB rollback recovery, and lock expiry behavior.
