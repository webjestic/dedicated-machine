Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check are all solid. However, there is a critical atomicity flaw in the payment flow that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is safe to merge.

---

## Critical Issues

### 1. Payment charge occurs inside a database transaction — this does not make the operation atomic

The PR comment says:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

This claim is incorrect and the misunderstanding is dangerous.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External HTTP call — NOT part of the DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over **database operations only**. The `payment.charge()` call is an external HTTP request to a payment gateway. If the charge succeeds but the process crashes before `record_charge` commits, you have charged the customer with no record of it. Wrapping the HTTP call in a `db.transaction()` block does not give you any rollback capability over the external charge.

The standard mitigation for this class of problem is the **outbox pattern** or a **two-phase approach**:

1. Write a `PENDING` charge record to the DB within a transaction *before* calling the payment gateway.
2. Call the payment gateway.
3. Update the record to `COMPLETED` with the returned `charge_id`.
4. A reconciliation job handles records stuck in `PENDING` (e.g., process crashed mid-flight).

Alternatively, if the payment gateway supports idempotency keys (Stripe does, for example), pass a deterministic idempotency key derived from `order_id` so that retrying the same charge is safe. This does not fully solve the recording problem but significantly reduces the blast radius.

### 2. Timeout on payment gateway can cause a double-charge

In `payment_client.py`:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded — the gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout will charge the customer again. This is only safe if idempotency keys are used on every attempt. The current implementation sends no idempotency key.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # No idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

Fix: generate a stable idempotency key (e.g., `order_id`) and include it as a header on every attempt so the gateway deduplicates.

## Moderate Issues

### 3. `PaymentError` is silently swallowed

If `payment.charge()` raises `PaymentError`, it propagates out of the `with self.db.transaction()` block and out of `with self._acquire_lock()`. The lock is correctly released via `finally`, but the exception is never caught in `process()`. This will surface as an unhandled exception to the caller rather than returning a structured `OrderResult`. Add an explicit `except PaymentError` handler.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

### 4. Lock TTL may be insufficient under retry conditions

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire before the payment attempt completes, allowing a second worker to acquire the lock and begin processing the same order concurrently. Either extend the TTL or implement lock refresh (heartbeat).

### 5. `response.json()["charge_id"]` is not guarded

```python
charge_id = response.json()["charge_id"]
```

A `KeyError` here will propagate as an unhandled exception. A malformed but HTTP-200 response from the gateway will crash the worker. Add defensive handling:

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError("Gateway returned success but no charge_id")
```

## Minor Issues

### 6. Test for lock release is too shallow

`test_lock_release_uses_lua_script` only asserts the script was called once. It does not assert the correct `keys` and `args` were passed. A refactor that calls the script with the wrong token would pass this test.

### 7. No test for `PaymentError` path

There is no test covering what happens when `payment.charge()` raises. Given the current code swallows it silently (see issue 3), this gap allowed the bug to go unnoticed.

### 8. `customer_id` has no format validation

`order_id` enforces an alphanumeric pattern. `customer_id` does not. If `customer_id` is used in any downstream query or log aggregation, this inconsistency is worth closing.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct.
- Lua script for token-verified release is the right approach and correctly registered at init.
- Pydantic validation with pattern enforcement on `order_id` is solid.
- Retry with backoff in the payment client is structured correctly for non-timeout cases.
- Logging is consistent and includes relevant identifiers throughout.
- The idempotency check under the lock is the right place for it.
