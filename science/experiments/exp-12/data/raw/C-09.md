Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several critical architectural flaws that would allow double-charges in production under realistic failure conditions, a timeout arithmetic error that guarantees lock expiry during normal operation, and missing error handling paths that leave the system in undefined states. These are not cosmetic issues; they are correctness failures in the core safety guarantee this service is supposed to provide.

---

## Critical Issues

### 1. Lock TTL Is Shorter Than Maximum Possible Charge Operation Duration — Guaranteed Expiry Under Load

This is the most dangerous bug. The math does not close.

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Maximum possible time inside the lock, before the database transaction even begins:

```
3 attempts × 30s timeout = 90s
+ 2 inter-retry sleeps × 5s = 10s
= 100s maximum payment phase alone
```

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. This means the Redis lock **expires while the payment is still in flight**. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge — a double-charge — while the first worker is still waiting on the gateway. The entire safety guarantee of this service collapses under this condition.

**Required fix:** Either reduce `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF` to well under `ORDER_LOCK_TTL` with margin, or increase `ORDER_LOCK_TTL` to at minimum `(MAX_RETRIES × REQUEST_TIMEOUT) + ((MAX_RETRIES - 1) × RETRY_BACKOFF) + db_transaction_budget + safety_margin`. A concrete safe value given current payment client parameters would be 120–150 seconds minimum. The lock TTL and payment client timeout budget must be defined in a shared configuration and validated at startup so they cannot drift independently again.

### 2. `PaymentError` Is Not Caught — Unhandled Exception Escapes the Lock Context With an Ambiguous Charge State

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # raises PaymentError — not caught
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`PaymentError` is defined in `payment_client.py` but never imported or caught in `order_processor.py`. When the payment gateway returns a permanent 4xx or exhausts retries, `PaymentError` propagates out of the `with self._acquire_lock(...)` block unhandled. The `finally` clause in `_acquire_lock` correctly releases the lock, but the caller of `process()` receives an unhandled exception rather than an `OrderResult`. More importantly, on a timeout specifically, the charge may have succeeded at the gateway but the response was not received — the system now has an unknown charge state with no record and no retry protection.

**Required fix:** Catch `PaymentError` explicitly inside `process()` and return a structured `OrderResult(processed=False, ...)`. Separately, timeout responses from the payment gateway require a reconciliation path — the system must be able to query the gateway by idempotency key to determine whether a charge actually landed before deciding whether to retry or fail.

---

### 3. Payment Gateway Retries Without Idempotency Key — Retries Can Produce Multiple Charges at the Gateway Level

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

No idempotency key is sent with the charge request. On a timeout, the gateway may have processed the charge and simply failed to return the response in time. The retry loop then sends an identical request with no mechanism for the gateway to deduplicate it. This can produce multiple charges at the payment processor level even when the Redis lock and database idempotency check are both functioning correctly. The Redis lock only protects against concurrent workers — it does not protect against this worker retrying a request that already succeeded.

**Required fix:** Generate a stable idempotency key per order (e.g., derived from `order_id`) and include it as a request header (e.g., `Idempotency-Key`) on every attempt. The key must be stable across retries for the same order so the gateway can deduplicate. This is a standard capability of payment gateways and must be used here.

### 4. Charge Occurs Inside `db.transaction()` — Distributed Transaction Across a Payment Gateway and a Database Is Not Atomic

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment in the PR says "crash between charge and record cannot produce an unrecorded charge" — this is incorrect. `db.transaction()` is a database transaction. It provides atomicity over database operations only. It has no control over the external payment gateway call. If `self.payment.charge()` succeeds and then the process crashes before `self.db.record_charge()` commits, the database transaction rolls back but the charge at the gateway is real and permanent. The charge is now unrecorded. The comment describes a guarantee that the code does not actually provide.

**Required fix:** The charge must happen **before** the database transaction opens, and the resulting `charge_id` must be recorded in a durable outbox or written to the database in a way that survives a crash. A standard pattern: (1) write a `PENDING` charge record with the idempotency key before calling the gateway, (2) call the gateway, (3) update the record to `COMPLETE` with the returned `charge_id`. On restart, any `PENDING` records can be reconciled against the gateway using the idempotency key. Moving the network call inside the database transaction does not make it atomic — it makes the transaction longer and the problem harder to see.

## Significant Issues

### 5. Lock Retry Sleep Blocks the Worker Thread — No Async Support, Retry Budget Not Accounted in Lock TTL

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)
```

`time.sleep()` blocks the calling thread. In any async or thread-pool deployment context this is a latency hazard. Additionally, the retry sleep time (up to 1.5 seconds across 3 attempts) is not accounted for in the lock TTL budget analysis above — it makes the gap slightly worse. The retry count of 3 with 0.5s backoff increments is also low; under genuine contention a legitimate retry may need longer to wait for the holding worker to finish.

---

### 6. `db.get_charge()` Idempotency Check Is Outside the Database Transaction — TOCTOU Race Remains Possible

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # read
    if existing:
        return ...

    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)                 # write
```

The idempotency read and the charge write are in separate database transactions. The Redis lock is the intended protection here, and under normal operation it does prevent concurrent execution. However, if the Redis lock expires mid-operation (see Issue 1), two workers can both pass the `get_charge` check before either has written the record. Moving `get_charge` inside `db.transaction()` with an appropriate isolation level (e.g., `SERIALIZABLE` or a `SELECT FOR UPDATE`) would make the idempotency guarantee database-enforced rather than Redis-dependent.

---

### 7. `response.json()["charge_id"]` — No Key Guard, Malformed Gateway Response Raises `KeyError`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body, this raises `KeyError`, which is not caught and is not a `PaymentError`. It will propagate as an unhandled exception. The response body should be validated before key access, and a missing `charge_id` should raise `PaymentError` with a descriptive message.

---

### 8. Test Suite Does Not Cover the Critical Failure Paths

The tests cover the happy path, the already-processed path, lock contention, and input validation. They do not cover:

- `PaymentError` raised during charge — what does the caller receive?
- Lock expiry mid-operation — is a second worker blocked or does it proceed?
- Gateway timeout followed by retry — is a duplicate charge issued?
- `db.transaction()` commit failure after a successful charge — is the charge recorded?
- `db.get_charge()` raising an exception — does the lock release correctly?

The absence of these tests is consistent with the architectural issues above — the failure modes were not modeled, so they were not tested.

---

## Minor Issues

- `payment_token` is logged indirectly via `OrderRequest` in warning output on validation failure. Confirm that Pydantic's `ValidationError` string representation does not include the token value in log output. If it does, that is a credential leak to log infrastructure.
- `CHARGE_ENDPOINT` is a hardcoded string constant. This should come from configuration/environment so it can differ between environments without a code change.
- `ORDER_LOCK_TTL`, `MAX_RETRIES`, `REQUEST_TIMEOUT`, and `RETRY_BACKOFF` are defined in two separate files with no enforced relationship. As noted in Issue 1, this is how the TTL arithmetic bug was introduced. These should be co-located or validated against each other at startup.
