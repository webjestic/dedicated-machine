Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and concurrency issues that must be resolved before this is suitable for a production government contracting environment where inventory integrity is critical.

---

## Critical Issues

### 1. Race Condition in Heartbeat Token Check (TOCTOU)

The heartbeat performs a `GET` then `EXPIRE` as two separate commands. Between those two operations, the lock can expire and be acquired by another process, meaning you extend a lock you no longer own.

```python
# UNSAFE — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

**Fix:** Use a Lua script to atomically check ownership and extend TTL.

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_lua = self.redis.register_script(HEARTBEAT_SCRIPT)

# In heartbeat loop
self._heartbeat_lua(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same atomic pattern applies to lock release — the current `GET` + `DELETE` in the `finally` block has the same TOCTOU flaw and must also use a Lua script.

---

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`, which means a single `InventoryService` instance cannot safely handle concurrent calls. A second call to `reserve_item` will stomp the heartbeat state of an in-flight first call.

```python
# Both calls share this — last writer wins, first call's heartbeat is killed
self._heartbeat_active = False
self._heartbeat_thread = ...
```

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass a mutable container or use a threading `Event`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into heartbeat thread instead of self._heartbeat_active
```

### 3. No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no guard against negative quantities, empty strings, or malformed `item_id` values (e.g., injection-style strings used as Redis key components).

```python
# Nothing stops this
service.reserve_item("../../etc/passwd", quantity=-999)
```

**Fix:** Validate at the boundary.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.isalnum() or len(item_id) > 128:
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1 or quantity > 1000:
        return ReservationResult(success=False, message="Invalid quantity")
```

The Redis key `f"inventory:lock:{item_id}"` is constructed directly from user input. Without sanitization this is a Redis key injection vector.

## Moderate Issues

### 4. `db.update_stock` and `db.create_order` Are Not Atomic

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record — a silent inventory leak with no rollback path.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # crash here = lost inventory
```

**Fix:** These two operations must be wrapped in a database transaction, or the design must be reconsidered (e.g., write the order first, then decrement stock, with compensating logic).

---

### 5. Heartbeat Thread Leaks on Lock Acquisition Failure

If `_start_heartbeat` is called and then an exception is raised before `yield`, `_stop_heartbeat` is never called because it lives in the `finally` block — which only executes after `yield`. In the current flow `_start_heartbeat` is called before `yield`, so a failure between those two points orphans the thread.

Review the ordering carefully and ensure `_stop_heartbeat` is always reachable via a nested `try/finally` around the heartbeat start.

---

### 6. Daemon Thread Suppresses Errors Silently

The heartbeat thread is a daemon thread with no error handling. If Redis becomes unavailable mid-operation, the heartbeat silently stops extending the lock. The main thread has no visibility into this failure and will proceed under a lock that may have expired.

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        # Redis error here is swallowed entirely
```

**Fix:** At minimum, catch and log exceptions. Consider a signaling mechanism back to the main thread.

---

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — including the TOCTOU race — is completely untested. The most complex and failure-prone code in this PR has zero test coverage.

**Required:** At minimum one integration-style test that exercises the actual heartbeat loop with a mock Redis client, verifying the Lua script is called with the correct token.

### 8. No Test for Partial DB Failure

There is no test covering the scenario where `update_stock` succeeds and `create_order` raises. This is the most dangerous failure mode in the service.

---

## Minor Issues

- `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10` are class-level constants but should be constructor parameters to allow environment-specific tuning without subclassing.
- No maximum cap on `retry_attempts` — a caller could pass an arbitrarily large value and block a thread for an extended period.
- `RuntimeError` is too broad for a lock failure. A custom `LockAcquisitionError` would allow callers to distinguish this condition from unexpected runtime errors.

---

## What Is Done Well

- Lock ownership verification via UUID token is correct in principle.
- `nx=True, ex=self.LOCK_TTL` on `SET` is the right atomic primitive.
- Structured logging with key/value pairs is clean and consistent.
- Returning `ReservationResult` rather than raising exceptions at the service boundary is a good API design choice.
- The test for expired lock token (`test_lock_not_released_if_expired`) shows good defensive thinking.
