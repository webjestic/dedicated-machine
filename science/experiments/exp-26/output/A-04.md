Decision: Request Changes

Summary: The heartbeat mechanism contains a race condition that directly violates the core safety guarantee this PR is meant to provide. Beyond that, the service has instance-level state management that makes it unsafe for concurrent use, and the DB operations lack atomicity guarantees that the lock is intended to protect. These are not minor issues — they represent the exact failure mode (stale writes after lock expiry) that this service must prevent.

---

## Critical Issues

### 1. Heartbeat Race Condition — Core Safety Violation

This is the primary rejection reason. The heartbeat's check-then-extend sequence is **not atomic**:

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # Step 1: read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2: extend TTL
```

Between Step 1 and Step 2, the lock can expire and be acquired by another process. When Step 2 then executes, it extends the TTL of a lock now owned by a different holder. Process A and Process B are now both inside the critical section simultaneously. This is precisely the stale write scenario this service exists to prevent.

**Fix:** Replace the get-then-expire sequence with a Lua script to make the ownership check and TTL extension atomic:

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_lock = self.redis.register_script(HEARTBEAT_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
                if not result:
                    logger.error(
                        "Heartbeat failed — lock lost or expired key=%s", lock_key
                    )
                    self._heartbeat_active = False
                    # Signal the main thread that the lock is gone
                    # so it can abort rather than complete a stale write

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomic-check issue exists in the lock release path, though it is less severe since the Lua-based release is a well-known pattern:

```python
# Also replace the release logic with a Lua script
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
```

### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the instance. If two threads call `reserve_item()` on the same `InventoryService` instance concurrently — which is the expected use case in a flash sale — they will overwrite each other's heartbeat state. Thread A's `_stop_heartbeat()` will kill Thread B's heartbeat, and vice versa. The lock for one reservation will silently stop being renewed.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through closures or a per-call dataclass:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # Pass heartbeat_active into _start_heartbeat, stop by clearing the event
    ...
```

### 3. DB Operations Are Not Atomic — Lock Boundary Is Insufficient

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `update_stock` succeeds and `create_order` fails, stock is decremented with no corresponding order. If the process crashes between the two calls, inventory is permanently lost. The distributed lock prevents concurrent access, but it does not protect against partial failure within the lock.

**Fix:** These two operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 4. Heartbeat Does Not Signal Lock Loss to the Main Thread

When the heartbeat detects that the lock is gone (token mismatch), it currently just logs and exits silently. The main thread continues executing the critical section — reading stock, decrementing, writing orders — without a valid lock. The heartbeat needs a signaling mechanism (e.g., a `threading.Event`) to notify the main thread so it can abort.

### 5. No Input Validation on `item_id` or `quantity`

The PR checklist claims Pydantic validation, but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string. Negative quantity would silently increase stock. At minimum:

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

### 6. Heartbeat Interval Is Too Close to Lock TTL

`HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` gives a three-renewal window, which is reasonable in isolation. However, under GC pressure or a heavily loaded event loop, the heartbeat thread itself can be suspended. The interval should be no more than one-third of the TTL, and the code should log a warning if a heartbeat fires late.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture mocks the heartbeat entirely. This means the race condition in Issue 1 is completely untested. There should be at least one integration-level test that runs the real heartbeat against a real or fake Redis and verifies that the lock is actually extended and that lock loss is detected and handled.

### 8. No Test for Concurrent Reservation

There is no test verifying that two concurrent calls to `reserve_item` for the same item do not both succeed when only one unit of stock remains. This is the primary correctness guarantee of the entire service and it has no test coverage.
