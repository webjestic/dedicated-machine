Decision: Request Changes

Summary: The heartbeat design contains a race condition that directly violates the core safety guarantee this service is meant to provide. Beyond that, there are several architectural and correctness issues that must be resolved before this code is safe to merge into a high-stakes flash sale path.

---

## Critical Issues

### 1. Heartbeat Race — The Core Flaw

This is the issue the persona instruction specifically flags, and the code does not actually solve it.

```python
# _beat() in _start_heartbeat:
current = self.redis.get(lock_key)          # Step A: read token
if current and current.decode() == token:   # Step B: compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step C: extend
```

Between Step A and Step C, the lock can expire and be acquired by another process. The `GET` + `EXPIRE` sequence is **not atomic**. If the original lock expires between those two calls, a second process acquires it with a new token, and then this heartbeat extends *that* process's lock — or worse, extends a lock the current process no longer owns. The token check provides no safety here because the check and the extend are two separate round trips.

**Required fix:** Use a Lua script to make the check-and-extend atomic:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init:
self._heartbeat_sha = self.redis.script_load(HEARTBEAT_SCRIPT)

# In the beat loop:
extended = self.redis.evalsha(
    self._heartbeat_sha, 1, lock_key, token, str(self.LOCK_TTL)
)
if not extended:
    logger.error("Heartbeat lost lock ownership key=%s — halting", lock_key)
    self._heartbeat_active = False
    # Signal the main thread that the lock is gone
```

The same atomicity problem exists in the release path:

```python
# Also non-atomic — same two-step race:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This must also be a Lua script (this is the standard Redlock release pattern):

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. Heartbeat Loss Is Silently Ignored

If the heartbeat detects that the lock is no longer owned (token mismatch), it simply stops looping. The main thread has no idea the lock was lost and continues executing the critical section — reading stock, writing stock, creating orders — without holding the lock. This is exactly the stale write scenario the lock is supposed to prevent.

```python
# Main thread continues here even if lock expired:
new_stock = current_stock - quantity
self.db.update_stock(item_id, new_stock)   # ← stale write, no lock held
self.db.create_order(order_id, item_id, quantity)
```

**Required fix:** The heartbeat must have a signaling mechanism back to the main thread. A `threading.Event` is the appropriate primitive:

```python
self._lock_lost_event = threading.Event()

# In beat loop, on token mismatch:
self._lock_lost_event.set()

# In reserve_item critical section, before each DB operation:
if self._lock_lost_event.is_set():
    raise LockLostError("Distributed lock expired during critical section")
```

---

### 3. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `InventoryService` is used as a singleton (the typical pattern with a shared Redis client), two concurrent calls to `reserve_item` will overwrite each other's `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat.

**Required fix:** Heartbeat state must be scoped to the lock context, not the instance. Pass state through local variables or a small context object within `_acquire_lock`.

## Significant Issues

### 4. No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `item_id` is non-empty, no length bound, no check that `quantity >= 1`. A caller passing `quantity=0` or `quantity=-5` will silently succeed and corrupt stock values. An empty `item_id` produces a lock key of `inventory:lock:` which is a collision vector.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")
```

### 5. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The reservation is lost with no recovery path. These two operations must be wrapped in a database transaction. If the DB layer does not support transactions, that is an architectural gap that must be addressed before this service handles real money.

### 6. Heartbeat Interval Relative to TTL Is Marginal

`HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` means the heartbeat fires at 10s, 20s, 30s... The lock expires at exactly 30s. A single delayed heartbeat (GC pause, thread scheduling jitter, Redis latency spike) causes the lock to expire. The standard recommendation is to renew at `TTL / 3` or less, with explicit monitoring for renewal failures. At minimum, reduce `HEARTBEAT_INTERVAL` to `8` or lower, and treat a failed renewal as a fatal error for the operation.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test that uses the `service` fixture completely bypasses the heartbeat. The most important behavior — lock expiry during a slow DB call, heartbeat renewal failure, lock-lost signaling — is entirely untested. There must be integration-level tests that use a real or fake Redis and simulate lock expiry mid-operation.

### 8. The "Lock Not Released If Expired" Test Proves the Bug

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test verifies that `DELETE` is not called when the token mismatches. That is correct. But it does not assert that the reservation itself *failed*. In the current code, `result.success` will be `True` here — the order was created without holding the lock. This test is accidentally documenting the bug rather than catching it.
