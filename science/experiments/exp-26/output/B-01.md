Decision: Request Changes

Summary: The heartbeat implementation contains a race condition that directly contradicts the PR's core safety claim — a GC pause or thread suspension between the token check and the `expire` call can allow the lock to expire and be re-acquired by another process, after which this process extends a lock it no longer owns. Beyond that critical flaw, the service has an instance-level heartbeat state design that makes concurrent calls mutually destructive, and the test suite mocks away the exact behavior it needs to verify, providing false confidence.

---

## Critical Issues

### 1. Heartbeat Race Condition — Check-Then-Act on Lock Ownership

This is the central flaw the PR claims to prevent.

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:   # ← lock appears valid here
    # ↑ GC pause, thread suspension, or network delay can occur here
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← extends a lock we may no longer own
```

Between `GET` and `EXPIRE`, the lock TTL can expire naturally. Another process then acquires the lock with a new token. When this thread resumes, it calls `EXPIRE` unconditionally — extending a lock owned by the other process. That other process now believes it holds an exclusive lock, but so does this process, which will proceed to write stale inventory data.

**Required fix:** Replace the two-command sequence with a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL * 1000])
            if result == 0:
                logger.error(
                    "Heartbeat lost lock ownership key=%s — halting extension", lock_key
                )
                self._heartbeat_active = False
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same race exists in the lock release path and must also be made atomic:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. Instance-Level Heartbeat State Breaks Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance attributes, not scoped to individual lock acquisitions.

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False       # shared across all calls
        self._heartbeat_thread = None        # shared across all calls
```

If two threads call `reserve_item` concurrently on the same service instance:

- Thread A sets `_heartbeat_active = True`, starts heartbeat for lock A
- Thread B sets `_heartbeat_active = True`, overwrites `_heartbeat_thread`
- Thread A's `_stop_heartbeat()` sets `_heartbeat_active = False`, killing Thread B's heartbeat
- Thread B's lock now has no renewal and will expire mid-operation

**Required fix:** Scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 5.0)
        logger.warning("Lock contention key=%s attempt=%d retrying in %.2fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)
    release_lock = self.redis.register_script(RELEASE_LOCK_SCRIPT)

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL * 1000])
            if result == 0:
                logger.error("Lost lock ownership during heartbeat key=%s", lock_key)
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        result = release_lock(keys=[lock_key], args=[token])
        if result:
            logger.info("Lock released key=%s", lock_key)
        else:
            logger.warning("Lock already expired or stolen at release key=%s", lock_key)
```

### 3. Heartbeat Interval Allows Lock Expiry Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires every 10 seconds, but the TTL is 30 seconds. This provides a 20-second buffer, which sounds adequate. However, `time.sleep()` is not precise under load, and the current implementation sleeps *before* checking, meaning the first extension attempt happens at T+10s. If the DB operation takes 25 seconds and the heartbeat thread is also delayed by OS scheduling, the lock can expire before renewal.

**Required fix:** Use millisecond-precision TTL (`PEXPIRE`/`PX`) and set `HEARTBEAT_INTERVAL` to no more than `LOCK_TTL / 3`:

```python
LOCK_TTL_MS = 30_000
HEARTBEAT_INTERVAL = 8  # renew at ~27% of TTL remaining, with margin
```

## Significant Issues

### 4. No Input Validation on `item_id` or `quantity`

The PR checklist marks "Input validation via Pydantic" but the public method accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

An empty string `item_id` produces a valid lock key `inventory:lock:` that collides across all items with empty IDs. A negative `quantity` passes the stock check and increments inventory. A zero `quantity` silently succeeds and creates an order for nothing.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id and req.quantity below
```

### 5. Database Operations Not Atomic — Partial Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not help here — this is a database atomicity problem.

**Required fix:** Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support transactions, that is a separate architectural gap that must be addressed before this service ships.

---

### 6. Backoff Has No Jitter and No Cap

```python
backoff = (2 ** attempt) * 0.1
```

With multiple processes retrying simultaneously, identical backoff schedules cause retry storms — all waiters wake at the same time and hammer the lock acquisition. There is also no cap, though with only 3 attempts the maximum is 0.4s, which is low enough that the missing cap is not dangerous today but will become one if `retry_attempts` is ever increased.

**Required fix:**

```python
import random

backoff = min((2 ** attempt) * 0.1, 5.0) + random.uniform(0, 0.05)
```
