Decision: Request Changes

Summary: The heartbeat mechanism contains a non-atomic check-then-extend race that directly contradicts the PR's core safety claim, and the `InventoryService` instance carries shared mutable heartbeat state that makes concurrent `reserve_item` calls mutually destructive. Additionally, the database operations inside the lock boundary lack transactional atomicity, meaning a crash between `update_stock` and `create_order` produces silent inventory corruption that the lock provides no protection against. These are not style concerns — they are correctness and data-integrity failures in the critical path.

---

## Detailed Findings

### 1. CRITICAL — Heartbeat Race Condition (Non-Atomic Check-Then-Extend)

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # Step A: read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step B: extend TTL
```

Between Step A and Step B, the lock TTL can expire naturally and a second process can acquire it with a new token. This thread then calls `expire` on a lock it no longer owns, silently extending a lock held by a different process. This is the exact TOCTOU hazard the PR claims to prevent.

**Required fix:** Replace the two-round-trip sequence with a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = redis.register_script("""
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
""")

extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    self._heartbeat_active = False  # we lost the lock; stop beating
    logger.error("Heartbeat detected lost lock key=%s", lock_key)
```

The heartbeat must also signal the main thread when it detects lock loss so the critical section can be aborted rather than completing a stale write.

### 2. CRITICAL — Instance-Level Heartbeat State Breaks Concurrent Calls

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared across all callers
        self._heartbeat_thread: Optional[...] = None  # overwritten on each call
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance simultaneously:

- Call A sets `_heartbeat_active = True` and stores its thread in `_heartbeat_thread`.
- Call B immediately overwrites both fields.
- Call A's `_stop_heartbeat()` sets the shared flag to `False`, killing Call B's heartbeat.
- Call B's `_stop_heartbeat()` does the same to Call A.

The lock is now unprotected for whichever call is still in its critical section.

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.error("Lost lock during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire, start thread, yield, finally: heartbeat_active.clear()
```

### 3. CRITICAL — No Transactional Atomicity Between DB Writes

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # crashes or times out
```

The distributed lock serializes access but provides zero durability guarantee. A process crash, DB timeout, or network partition after `update_stock` and before `create_order` leaves stock decremented with no corresponding order. This is silent inventory loss with no compensating mechanism.

**Required fix:** Both writes must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not expose transactions, that is a prerequisite gap that must be resolved before this service ships.

### 4. MODERATE — Input Validation Is Declared But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, or excessively large quantities. A caller passing `quantity=0` silently creates an order for zero units and decrements stock by zero, which is a logic error. A caller passing `quantity=-5` increases stock.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. MODERATE — Heartbeat Thread Survives Lock Release on Exception Path

```python
finally:
    self._stop_heartbeat()              # sets flag to False
    current = self.redis.get(lock_key)  # heartbeat thread may still be mid-beat here
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

`_stop_heartbeat` sets a flag but does not join the thread. The heartbeat thread can execute one final `expire` call after `delete` has run, resurrecting a deleted lock key with a 30-second TTL. This leaves a ghost lock that blocks all other processes for up to 30 seconds.

**Required fix:** Join the thread with a bounded timeout in the `finally` block:

```python
finally:
    heartbeat_active.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # then release lock
```

### 6. LOW — Tests Mock Away the Exact Components Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat, which is the primary correctness mechanism being introduced, is mocked out in every test except `test_lock_not_released_if_expired`. The test suite therefore provides no coverage for the race conditions identified above. Integration tests against a real or Fakeredis instance with simulated TTL expiry are required.
