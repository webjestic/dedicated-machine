Decision: Request Changes

Summary: The core locking logic contains a race condition in the heartbeat and release path — the token ownership check is not atomic, which means a lock could be read, expire, be acquired by another process, and then be deleted by this instance. Beyond that, the `InventoryService` class carries instance-level heartbeat state (`_heartbeat_active`, `_heartbeat_thread`), making it unsafe for concurrent use across multiple threads or requests sharing the same service instance. Several additional issues in the tests and input validation also need to be addressed before this is production-ready.

---

## Detailed Findings

### 🔴 Critical — Race Condition in Lock Release (Non-Atomic Check-and-Delete)

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process. This instance then deletes a lock it no longer owns, corrupting the mutual exclusion guarantee entirely.

**Fix:** Use a Lua script to make this atomic. This is the industry-standard approach:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat's `GET` + `EXPIRE` sequence and must receive the same treatment:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

**File:** `services/inventory.py`, `InventoryService.__init__`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the instance. If a single `InventoryService` instance handles two concurrent requests (the normal case in any web framework with a shared service layer), the second call to `_start_heartbeat` overwrites `_heartbeat_active` and `_heartbeat_thread`, and a call to `_stop_heartbeat` from one request will kill the heartbeat of the other.

**Fix:** Heartbeat state must be scoped to the lock context, not the instance. Pass state through local variables or a small context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_script(keys=[lock_key], args=[token])

def _start_heartbeat(
    self, lock_key: str, token: str, active: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not active.is_set():
                break
            self._extend_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 🔴 Critical — No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model wrapping the inputs. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a path-traversal-style string as `item_id`.

**Fix:** Introduce a request model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

The `pattern` constraint on `item_id` is important here — that value is interpolated directly into the Redis key (`inventory:lock:{item_id}`). An unvalidated `item_id` is a key injection vector.

### 🟡 Major — Heartbeat Thread Is Never Joined

**File:** `services/inventory.py`, `_stop_heartbeat` and `_acquire_lock` finally block

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread may still be mid-sleep when the lock is deleted. If it wakes up after deletion and the lock has been re-acquired by another process, the now-stale heartbeat will extend a lock it has no business touching — until the atomic Lua fix above is also applied. Even with the Lua fix, leaving daemon threads running without joining is a resource leak.

**Fix:** Join the thread with a bounded timeout in the `finally` block (shown in the thread-safety fix above).

### 🟡 Major — `db.update_stock` and `db.create_order` Are Not Transactional

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The reservation is silently lost. The distributed lock only protects against concurrent access — it does not provide atomicity across two separate DB writes.

**Fix:** Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction should expose a transaction context manager, or the two operations should be combined into a single atomic stored procedure or SQL transaction.

### 🟡 Major — Tests Mock Away the Exact Code Under Test

**File:** `tests/test_inventory.py`, `service` fixture

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most operationally dangerous part of this service. Mocking it out means the tests provide zero coverage for the thread-safety issues identified above. The lock release path in `_acquire_lock`'s `finally` block is also not meaningfully exercised because the token comparison uses a hardcoded `b"test-token"` that will never match the `uuid4()` generated inside `_acquire_lock`, so `redis.delete` will never be called in the happy path tests — yet `test_reserve_success` passes, which means the tests are not asserting lock release at all.

**Fix:** Write at least one integration-style test that uses a real or `fakeredis`-backed instance and validates the full lock lifecycle, including heartbeat behavior and lock release.

### 🟡 Major — Bare `RuntimeError` Is Too Broad in Exception Handling

**File:** `services/inventory.py`, `reserve_item`

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

`RuntimeError` from `_acquire_lock` is intentional, but any other `RuntimeError` raised by `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` will be silently swallowed and returned as "temporarily unavailable," masking real data-layer failures. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

### 🟠 Minor — LOCK_TTL / HEARTBEAT_INTERVAL Ratio Is Fragile

With `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10`, the heartbeat fires at the two-thirds mark. That is acceptable, but these are bare class-level constants with no enforcement of the invariant `HEARTBEAT_INTERVAL < LOCK_TTL / 2`. If someone changes one without the other, the lock can expire before the first heartbeat fires. Assert the invariant in `__init__`:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, (
    "HEARTBEAT_INTERVAL must be less than half of LOCK_TTL"
)
```

---

### 🟠 Minor — `quantity` Default of `1` Should Be Explicit in the Model

A default of `1` is reasonable, but it should live in the validated request model, not as a raw function default, once the Pydantic input model is introduced.
