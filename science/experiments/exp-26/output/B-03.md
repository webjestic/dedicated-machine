Decision: Request Changes

Summary: The heartbeat design contains a race condition that directly contradicts the PR's core safety claim — a GC pause or thread suspension between the token check and the `expire` call can allow the lock to expire, be acquired by another process, and then have its TTL silently extended by the stale heartbeat, enabling concurrent writes to the same inventory record. Beyond that critical flaw, the service has several additional architectural and correctness issues that must be resolved before this handles real flash sale traffic.

---

## Critical: Heartbeat Race Condition (TOCTOU on Lock Ownership)

This is the primary blocker. The heartbeat's check-then-act pattern is not atomic:

```python
# _beat() — NOT atomic
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend ← race window here
```

Between steps (2) and (3), the following sequence is possible:

1. Process A's heartbeat reads the token — it matches.
2. Process A's thread is suspended (GC, OS scheduler, network hiccup).
3. The lock TTL expires naturally.
4. Process B acquires the lock with a new token and begins its critical section.
5. Process A's thread resumes and calls `expire` — it does **not** check ownership again, it just resets the TTL on a lock now owned by Process B.
6. Both processes now believe they hold the lock. Inventory is double-decremented.

The fix is a Lua script that makes the check-and-extend atomic:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_script(
                keys=[lock_key],
                args=[token, self.LOCK_TTL * 1000]  # milliseconds
            )
            if result == 0:
                # Lock was lost — signal the main thread
                logger.error(
                    "Heartbeat lost lock ownership key=%s — flagging for abort",
                    lock_key
                )
                self._heartbeat_active = False
                # Set a flag the main thread can observe
                self._lock_lost = True

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomic-Lua pattern must also be applied to the release path (see below).

## High: Lock Release Is Also a TOCTOU Race

```python
# _acquire_lock finally block — also NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # ← another process may acquire between get and delete
```

Between `get` and `delete`, another process can acquire the lock. The `delete` then destroys a lock you no longer own. This is a well-known Redis distributed lock pitfall. The canonical fix is another Lua script:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""
```

## High: Heartbeat Loss Is Not Propagated to the Critical Section

Even after fixing the atomic extension, if the heartbeat detects that lock ownership was lost (extension returned 0), the `reserve_item` critical section continues executing and will commit writes it no longer has the right to make. The main thread must be notified and must abort:

```python
# In reserve_item, after each DB operation:
if getattr(self, '_lock_lost', False):
    raise LockLostError("Lock expired during critical section — aborting reservation")
```

Consider using a `threading.Event` rather than a bare boolean flag to avoid visibility issues across threads without a memory barrier.

---

## High: Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active`, `_heartbeat_thread`, and `_lock_lost` are stored on `self`. A single `InventoryService` instance is almost certainly shared across threads (e.g., injected as a singleton). Two concurrent calls to `reserve_item` on different items will corrupt each other's heartbeat state:

- Thread A sets `_heartbeat_active = True`.
- Thread B calls `_stop_heartbeat()`, setting `_heartbeat_active = False`.
- Thread A's heartbeat stops extending a lock it still holds.

All heartbeat state must be scoped to the individual lock acquisition, not the instance. Pass state through closures or a small dataclass:

```python
@dataclass
class _LockState:
    active: threading.Event = field(default_factory=threading.Event)
    lost: threading.Event = field(default_factory=threading.Event)
```

## Medium: No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model on the input path. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:baz"` or an empty string will produce a malformed or colliding key. Negative or zero `quantity` values are not rejected. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Medium: Database Operations Are Not Atomic and Have No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. The reservation is silently lost. These two operations must be wrapped in a database transaction, or the service must implement compensating logic:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

## Medium: TTL/Heartbeat Interval Ratio Is Unsafe

```
LOCK_TTL = 30 seconds
HEARTBEAT_INTERVAL = 10 seconds
```

The heartbeat fires at T=10s and T=20s. If the heartbeat thread is delayed by even a few seconds (GC pause, thread starvation), the extension at T=20s may arrive after the lock has already expired at T=30s. The standard guidance is to renew at `TTL / 3` or less, and to use millisecond-precision TTLs (`PX` flag) to reduce the expiry window. With the Lua script fix above, use `PEXPIRE` with milliseconds.

## Low: Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The tests provide zero coverage of the heartbeat logic, the atomic extension, or the lock-lost propagation path — which are the highest-risk components. The test suite should include:

- A test that verifies the Lua extension script is called with the correct token.
- A test that verifies a `lock_lost` event aborts the critical section before writing.
- A test that verifies concurrent calls on the same instance do not corrupt each other's heartbeat state.
- An integration test (even with a local Redis via `fakeredis`) that exercises the full lock lifecycle without mocking internals.
