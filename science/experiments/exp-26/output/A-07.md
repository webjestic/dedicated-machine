Decision: Request Changes

Summary: The heartbeat design contains a **race condition that directly violates the core safety invariant this PR is meant to enforce.** There are also two additional architectural defects — a shared-state concurrency bug and missing input validation — that must be resolved before this code is safe to merge.

---

## Critical Defect — Heartbeat Race Window (Stale Write Risk)

This is the primary rejection reason. The heartbeat thread performs a non-atomic check-then-extend sequence:

```python
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2: extend
```

Between step 1 and step 2, the lock TTL can expire. Another process can acquire the lock between those two calls. When step 2 then executes `EXPIRE`, it extends a lock now owned by a different process. Both processes now believe they hold the lock simultaneously, and the stale write this entire service was designed to prevent becomes possible.

The fix is to replace the two-call sequence with an atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

This must be a single round-trip with no observable gap. There is no safe way to implement this correctly using separate `GET` and `EXPIRE` calls.

The same non-atomic pattern appears in the lock release path in `_acquire_lock` and carries the same risk — a token match followed by a separate `DELETE` can delete a lock owned by another process if the TTL expires between the two calls. This should also be replaced with an atomic Lua script or the standard `redis.delete` conditional pattern via a pipeline.

---

## Shared-State Concurrency Bug — Instance-Level Heartbeat Fields

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the expected production scenario — one call's `_stop_heartbeat()` will set `_heartbeat_active = False` and kill the heartbeat belonging to a *different* concurrent call. The lock for that other call will then expire undefended.

The heartbeat state must be scoped to each individual lock acquisition, not to the service instance. This can be done with a local flag object passed into the thread closure, or by encapsulating each lock's lifecycle in its own class.

---

## Missing Input Validation

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `item_id: str` and `quantity: int` with no validation at the call site. There is nothing preventing:

- `quantity <= 0` (reserving zero or negative units, which would silently *increase* stock via `current_stock - quantity`)
- An empty string `item_id`
- An `item_id` that is excessively long or contains characters that could affect the Redis key namespace

A `quantity <= 0` input is a direct inventory integrity bug. At minimum:

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
```

---

## Test Coverage Gaps

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat race condition described above is completely untested. The tests verify the happy path business logic but provide no coverage of the actual distributed locking behavior. Tests should be added that exercise the Lua script paths and verify that concurrent acquisition attempts produce correct mutual exclusion.
