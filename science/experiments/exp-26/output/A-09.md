Decision: Request Changes

Summary: The heartbeat design contains a race condition that directly violates the safety guarantee this PR claims to provide. Beyond that, there are several architectural and correctness issues that must be resolved before this code is safe to merge into a high-concurrency flash sale path.

---

## Critical Issues

### 1. Race Condition in Heartbeat — Check-Then-Act Is Not Atomic

This is the most serious flaw. The heartbeat performs a non-atomic read-then-extend:

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # Step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2: extend
```

Between Step 1 and Step 2, the lock can expire and be acquired by another process. When Step 2 then executes, it extends a lock now owned by a different holder. This is the exact scenario the system prompt requires rejection on — an unbounded GC pause or thread suspension between those two lines allows a stale write to proceed under a stolen lock extension.

**Required fix:** Replace with an atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

---

### 2. Instance-Level Heartbeat State Breaks Concurrent Usage

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`, meaning a single `InventoryService` instance cannot safely serve concurrent requests. If two callers invoke `reserve_item` simultaneously, the second call's `_stop_heartbeat()` will kill the first call's heartbeat thread mid-operation.

```python
self._heartbeat_active = False  # kills ANY active heartbeat on this instance
```

**Required fix:** Pass a `threading.Event` into the heartbeat closure and scope it locally to each lock acquisition, not to the instance.

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # atomic extend via Lua script
        ...

# In finally:
stop_event.set()
```

---

### 3. Heartbeat Interval Allows Lock Expiry Window

`LOCK_TTL = 30s`, `HEARTBEAT_INTERVAL = 10s`. The heartbeat sleeps *first*, then extends. If the lock is acquired at T=0, the first extension attempt is at T=10s. That is fine under normal conditions, but if the heartbeat thread is delayed by scheduler jitter or GC, the extension could arrive at T=31s — after the lock has already expired. The interval should be materially less than TTL/2, and the sleep should use a cancellable wait, not `time.sleep`.

---

### 4. `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The heartbeat thread may still be mid-execution (inside the Lua call) when the `finally` block proceeds to delete the lock. There is no `self._heartbeat_thread.join(timeout=...)` call. This creates a window where the heartbeat extends a lock that the main thread has already deleted, potentially extending a lock now held by a different process.

**Required fix:**

```python
def _stop_heartbeat(self, thread: threading.Thread, stop_event: threading.Event) -> None:
    stop_event.set()
    thread.join(timeout=5)
```

---

### 5. Lock Release Is Also Not Atomic

The same check-then-act problem exists in the `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Another process can acquire the lock between `get` and `delete`, causing deletion of a lock you no longer own. This must also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

## Moderate Issues

### 6. No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is used directly in a Redis key (`inventory:lock:{item_id}`) and passed to the database. A caller passing `item_id = "foo:bar"` or `quantity = -5` will produce incorrect behavior. Pydantic is imported but not applied here.

**Required fix:** Validate at the boundary:

```python
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    raise ValueError("Invalid item_id")
if quantity < 1:
    raise ValueError("quantity must be >= 1")
```

Or introduce a proper request model.

---

### 7. Non-Atomic Stock Decrement + Order Creation

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes Redis access. These two operations must be wrapped in a database transaction.

---

### 8. Retry Loop Sleeps on the Hot Path

```python
time.sleep(backoff)  # up to 0.4s blocking the calling thread
```

Under flash sale load, this will exhaust thread pool capacity quickly. Consider async-compatible retry or at minimum document that this service requires its own thread pool isolated from the HTTP worker pool.

---

## Test Coverage Gaps

### 9. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test that uses the `service` fixture completely bypasses the heartbeat. The most critical code path — the one this PR's safety argument depends on — has zero test coverage. There should be at least one integration-level test that runs the real heartbeat against a real or fake Redis and verifies atomic extension behavior.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

The test verifies that `delete` is not called when the token mismatches, which is correct behavior. However, it does not test that the heartbeat correctly stops extending in this case, nor does it test the race between heartbeat extension and lock release. The test name implies safety coverage that is not actually present.
