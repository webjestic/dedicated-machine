Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several race conditions and architectural flaws that could cause silent data corruption or lock leakage under real concurrent load. The instance-level heartbeat state makes the service non-reentrant and unsafe for concurrent callers, and the check-then-act lock release is not atomic. These are not cosmetic issues — in a flash sale context with high concurrency, they represent genuine correctness failures.

---

## Detailed Review

### 🔴 Critical — Correctness / Safety

**1. Non-atomic lock release (TOCTOU race)**

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock after the TTL expires. You then delete a lock you no longer own. This is the canonical mistake in distributed locking. The fix is a Lua script that makes the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat's `GET` + `expire` pair — it should also be a Lua script or use `SET ... XX` with the token embedded in the value checked atomically.

---

**2. Instance-level heartbeat state breaks concurrent use**

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of this service — one call's `_stop_heartbeat()` will kill the other call's heartbeat thread. The second lock will then expire mid-operation, silently.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # Lua-based atomic extend here
            ...

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_script(keys=[lock_key], args=[token])
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which is less responsive to shutdown.

---

**3. Heartbeat thread is never joined**

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread could still be mid-execution — specifically mid-`expire` — after the lock has already been deleted and re-acquired by another process. You are extending a lock you no longer own. Always join the thread before releasing:

```python
heartbeat_active.clear()
thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
# then release
```

### 🟡 Significant — Architecture / Reliability

**4. No input validation on `item_id` or `quantity`**

The checklist claims "Input validation via Pydantic" but there is none on the public method signature. `item_id` is passed directly into a Redis key and a DB query with no sanitization. A caller passing `item_id = "../../admin"` or an empty string will produce a malformed lock key. Quantity has no lower bound check.

```python
# Minimum viable guard
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not item_id.replace("-", "").replace("_", "").isalnum():
    raise ValueError("item_id contains invalid characters")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

Or define a proper request model and validate at the boundary.

---

**5. `db.update_stock` and `db.create_order` are not atomic**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the lock expires between these two calls, stock is decremented but no order exists. The Redis lock does not protect against application crashes. These two writes must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The DB abstraction should expose this. If it does not, that is a separate architectural gap that needs to be addressed before this service is safe.

---

**6. Heartbeat interval is too close to TTL**

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 10`. The heartbeat fires at 10s and 20s — fine under normal conditions. But `time.sleep` is not precise, Redis round-trips add latency, and under GIL pressure the thread may be delayed. A safer ratio is `HEARTBEAT_INTERVAL = TTL / 3` at most, and the TTL itself should be configurable per deployment environment, not hardcoded.

### 🟠 Test Coverage Gaps

**7. The most important scenario is not tested**

There is no test for concurrent access — two threads calling `reserve_item` on the same instance simultaneously. Given flaw #2 above, this would fail. A test using `threading.Thread` or `concurrent.futures.ThreadPoolExecutor` should be included to validate the heartbeat isolation.

**8. Heartbeat is mocked out entirely in most tests**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the tests do not exercise the actual locking path at all. There should be at least one integration-style test (using `fakeredis`) that runs the real heartbeat logic. The current test suite would pass even if the heartbeat contained a syntax error.

**9. `test_lock_not_released_if_expired` does not test what it claims**

The test patches `redis.get` to return a different token, but because `_start_heartbeat` and `_stop_heartbeat` are mocked, the lock release path being tested is the `finally` block in `_acquire_lock`. The test is valid for that path, but it does not test the heartbeat's token check, which is where the real race lives.

---

### 🔵 Minor

**10. `RuntimeError` is too broad for lock failure**

Callers cannot distinguish a lock contention timeout from an unexpected Redis error. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

**11. Daemon thread suppresses shutdown errors**

`daemon=True` means the heartbeat thread is killed hard on interpreter exit. If the process is shutting down mid-reservation, the lock will not be released cleanly. This is acceptable if TTL-based expiry is the fallback, but it should be a documented assumption, not an implicit one.
