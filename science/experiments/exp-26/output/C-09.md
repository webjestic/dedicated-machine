Decision: Request Changes

Summary: The core locking logic contains a race condition in the heartbeat and release flow that could cause lock theft or double-release under concurrent load, and the `InventoryService` instance is not safe for concurrent use because heartbeat state is stored on `self` rather than scoped to each lock acquisition. Additionally, the database operations inside the lock lack transactional guarantees, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory state. These are not cosmetic issues — in a flash sale context they are exploitable failure modes.

---

## Detailed Findings

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
# Current — stored on self, shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or shared instance. If two coroutines or threads call `reserve_item` concurrently, the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, causing the first heartbeat to stop silently. The first lock then expires while its operation is still in progress, allowing another caller to acquire it — a classic TOCTOU violation.

**Fix:** Scope all heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock was lost, stop beating

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread, yield, finally: heartbeat_active.clear()
```

### 🔴 Critical — Heartbeat Check-Then-Extend Is a Race Condition

```python
# In _beat():
current = self.redis.get(lock_key)          # step 1
if current and current.decode() == token:   # step 2
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 3 — non-atomic
```

Between steps 2 and 3, the lock can expire and be acquired by another process. You then extend a lock you no longer own. This is a well-known Redlock pitfall.

**Fix:** Use a Lua script to make the check-and-extend atomic.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once on __init__:
self._extend_lock = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat:
extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Lost lock during heartbeat key=%s", lock_key)
    break
```

The same pattern applies to lock release — the current `get` + conditional `delete` is also non-atomic and should use the same Lua approach.

### 🔴 Critical — No Transactional Guarantee Between DB Writes

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, loses its DB connection, or hits an exception after `update_stock` but before `create_order`, stock is decremented with no corresponding order. This is silent data corruption. The Redis lock provides no protection here because the failure is inside the lock boundary.

**Fix:** These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is a blocking architectural gap that must be resolved before this merges.

### 🟡 Moderate — Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` is passed directly into a Redis key and a DB query without sanitization. A caller passing `item_id = "lock:admin:*"` or an empty string will produce malformed keys.

**Fix:** Introduce a validated request model and enforce it at the boundary.

```python
from pydantic import BaseModel, Field, conint

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: conint(ge=1, le=1000) = 1

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 🟡 Moderate — Retry Loop Sleeps on the Calling Thread

```python
time.sleep(backoff)  # inside _acquire_lock, called from reserve_item
```

In any async or thread-pool context this blocks the worker for up to `0.1 + 0.2 + 0.4 = 0.7s` per call during contention. Under flash sale load this will exhaust the thread pool. If the service is or will be async, this must use `asyncio.sleep`. If it is synchronous, the retry budget and backoff ceiling should be documented and load-tested.

---

### 🟡 Moderate — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # <-- stop never runs
    return svc
```

The heartbeat thread — the most operationally complex and failure-prone component — is mocked out in every test except `test_lock_not_released_if_expired`, which itself does not actually start a thread. The race conditions described above would not be caught by this suite. Integration tests exercising real Redis (e.g., via `fakeredis` or a test container) with actual thread execution are required.

---

### 🟢 Minor — `HEARTBEAT_INTERVAL` Does Not Guarantee Safety Margin

With `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10`, the heartbeat fires at 10s, 20s, 30s. The 30s fire is a race against expiry. A safer convention is `HEARTBEAT_INTERVAL = LOCK_TTL // 3` enforced programmatically, not as two independent constants that can drift.

---

## What Is Done Well

- Ownership token (`uuid4`) on lock acquisition is correct practice
- Exponential backoff structure is sound
- `ReservationResult` as a typed return value is clean
- Structured logging with key context is good
- Daemon thread prevents process hang on unclean shutdown

---

## Required Before Approval

1. Scope heartbeat state to the lock context, not the instance
2. Make heartbeat extend and lock release atomic via Lua scripts
3. Wrap the two DB writes in a transaction
4. Implement actual input validation at the `reserve_item` boundary
5. Add integration-level tests that exercise the real heartbeat thread against a mock Redis that respects TTL (e.g., `fakeredis`)
