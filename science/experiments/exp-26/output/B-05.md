Decision: Request Changes

Summary: The heartbeat mechanism contains a non-atomic check-then-extend race that directly violates the safety guarantee this PR claims to provide, and the `InventoryService` instance is structured as a singleton-style object whose shared mutable heartbeat state makes it unsafe for concurrent use across threads or requests. Additionally, the database operations lack transactional atomicity, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory state even with the lock held. These are not minor issues — they represent the exact failure modes a distributed lock is meant to prevent.

---

## Detailed Findings

### Critical

**1. Heartbeat race condition — non-atomic GET + EXPIRE**

```python
# _beat() in _start_heartbeat:
current = self.redis.get(lock_key)          # ← read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You then extend a lock you no longer own. This is the exact scenario the PR claims to prevent. The fix is a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)
    self._heartbeat_active = True

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(
                    keys=[lock_key],
                    args=[token, self.LOCK_TTL * 1000]  # milliseconds
                )
                if not result:
                    logger.error(
                        "Heartbeat lost lock ownership key=%s — "
                        "halting extension", lock_key
                    )
                    self._heartbeat_active = False

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomic Lua pattern should also be applied to lock release in `_acquire_lock`'s `finally` block. The current GET + DELETE sequence there has the identical race.

**2. Shared mutable heartbeat state on the instance**

```python
self._heartbeat_active = False          # shared across all callers
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the normal deployment model — one thread's `_stop_heartbeat()` call sets `_heartbeat_active = False` and silently kills the other thread's heartbeat. The lock for the still-running operation then expires undefended.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script here
```

Note also that `_stop_heartbeat` currently does not join the thread. The heartbeat thread can execute one final extension cycle after the lock is released and after another process has legitimately acquired it.

**3. Non-atomic database operations — no transaction boundary**

```python
self.db.update_stock(item_id, new_stock)   # ← crash here
self.db.create_order(order_id, item_id, quantity)  # ← never runs
```

A process crash, network partition, or unhandled exception between these two calls leaves stock decremented with no corresponding order record. The distributed lock only serializes access — it does not provide atomicity across the two writes. This requires either a database transaction or a compensating rollback:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error(
        "DB transaction failed item_id=%s order_id=%s — "
        "reservation rolled back", item_id, order_id
    )
    raise
```

If the database layer does not support transactions, an explicit compensating write (`update_stock` back to `current_stock`) is required in the exception handler, with a dead-letter or audit log for failures of the compensation itself.

### Moderate

**4. Input validation is declared but not implemented**

The checklist marks input validation via Pydantic, but `reserve_item` accepts arbitrary `item_id` strings and any integer `quantity` with no validation. A negative `quantity` would increment stock. An empty or injection-crafted `item_id` would produce a Redis key like `inventory:lock:` or worse. Add a validated request model:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(default=1, ge=1, le=1000)
```

---

**5. TTL arithmetic does not account for heartbeat startup latency**

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 10`. The heartbeat fires at 10s, 20s, 30s. If the heartbeat thread is scheduled late by the OS (entirely possible under GC pressure or CPU contention — the exact scenario this lock guards against), the first renewal may arrive after the 30-second TTL has already elapsed. Use millisecond-precision TTLs and set the heartbeat interval to no more than one-third of the TTL, with the first beat at `TTL / 3`, not `TTL / 3` after an initial sleep:

```python
LOCK_TTL_MS = 30_000
HEARTBEAT_INTERVAL_S = 8  # well within TTL/3, leaves margin for scheduling jitter
```

---

**6. `RuntimeError` swallowed silently on lock failure**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Lock acquisition failure and an unexpected `RuntimeError` from inside the critical section are both caught here. A bug inside `reserve_item` (e.g., a programming error raising `RuntimeError`) will be silently returned as a user-facing "try again" message with no stack trace and no alerting signal. Use a dedicated exception type:

```python
class LockAcquisitionError(Exception):
    pass
```

---

### Test Coverage Gaps

The test suite mocks out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat race condition and the thread-safety issue described above are completely invisible to the test suite. The following scenarios have no coverage:

| Scenario | Risk |
|---|---|
| Heartbeat fires after lock expiry and extends a foreign lock | Data corruption |
| Two concurrent `reserve_item` calls on the same instance | Heartbeat cancellation, stale write |
| Crash between `update_stock` and `create_order` | Inventory inconsistency |
| `quantity <= 0` or negative | Stock increment |
| `item_id` with Redis special characters | Key collision or injection |
| Lock release after heartbeat thread has already extended a stolen lock | Deletes another owner's lock |
