Decision: Request Changes

Summary: The heartbeat mechanism contains a non-atomic check-then-extend race that directly contradicts the PR's core safety claim, and the `InventoryService` instance carries shared mutable heartbeat state that makes concurrent calls on the same service object unsafe. Several additional issues — ranging from a missing atomic release script to absent input validation — need to be resolved before this is safe to merge into a flash-sale critical path.

---

## Detailed Findings

### 1. CRITICAL — Heartbeat Race: Non-Atomic Check-Then-Extend

**File:** `services/inventory.py`, `_start_heartbeat` → `_beat()`

```python
# Current — NOT atomic
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write — separate round-trip
```

Between the `GET` and the `EXPIRE` call, the lock can expire and be acquired by another process. You then extend a lock you no longer own, directly enabling the stale-write scenario the PR claims to prevent.

**Required fix:** Use a Lua script so the check and extend are atomic on the Redis side.

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at __init__
self._extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In _beat()
result = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Heartbeat: lock no longer owned, stopping. key=%s", lock_key)
    self._heartbeat_active = False
    break
```

### 2. CRITICAL — Instance-Level Heartbeat State Breaks Concurrent Calls

**File:** `services/inventory.py`, class-level fields

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` is a single boolean on the instance. If two threads call `reserve_item` concurrently on the same `InventoryService` object, one call's `_stop_heartbeat()` will silence the other call's heartbeat thread, leaving that lock unrenewed and vulnerable to expiry.

**Required fix:** Scope heartbeat state to the lock acquisition, not the instance. Pass a `threading.Event` into `_start_heartbeat` and return it; the caller owns it.

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning("Heartbeat lost lock key=%s", lock_key)
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event

# In _acquire_lock finally block:
stop_event.set()
```

### 3. HIGH — Lock Release Is Also Non-Atomic

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same class of race as finding #1. Between `GET` and `DELETE`, the lock can expire and be re-acquired. You then delete another process's lock.

**Required fix:** Atomic Lua release script — this is the standard Redlock pattern.

```python
RELEASE_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""
```

### 4. HIGH — No Input Validation on `reserve_item`

The PR checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` parameters with no validation. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:../../admin"` or an empty string reaches Redis and the database without any sanitization.

**Required fix:** Introduce a `ReservationRequest` model and validate at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. MEDIUM — Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no order record. The Redis lock does not protect against partial database state — that requires a database transaction.

**Required fix:** Wrap both calls in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. MEDIUM — Heartbeat Interval Allows Near-Expiry Window

`LOCK_TTL = 30s`, `HEARTBEAT_INTERVAL = 10s`. The heartbeat fires at 10 s, 20 s, 30 s. At exactly 30 s, the lock expires before the third heartbeat fires. Under GC pressure or thread scheduling delays, the interval can slip past the TTL.

**Required fix:** Renew at `TTL / 3` or better, and confirm the renewal succeeded (addressed by finding #1's return-value check).

---

### 7. LOW — `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The heartbeat thread may still be mid-execution (inside the `GET`/`EXPIRE` window) when the lock is released. There is no join or synchronization point.

**Required fix:** After setting the stop event, join with a bounded timeout.

```python
stop_event.set()
if self._heartbeat_thread:
    self._heartbeat_thread.join(timeout=2.0)
```

---

### 8. LOW — Tests Mock Away the Exact Code Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the primary safety mechanism being introduced. Mocking it means none of the critical paths in findings #1, #2, or #7 are exercised. Integration-level tests against a real or `fakeredis`-backed instance are required.
