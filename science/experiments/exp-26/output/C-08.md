Decision: Request Changes

Summary: The core locking logic contains a race condition in the heartbeat and release path — the token ownership check (`GET` then `DEL`) is non-atomic, meaning another process could steal the lock between the two operations, leading to incorrect lock release or phantom ownership. Additionally, the `InventoryService` stores heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`), which makes the class unsafe for concurrent use across multiple threads or requests sharing the same service instance. There are also missing input validation guards, a test fixture bug that silently bypasses the code under test, and no transactional safety around the DB writes.

---

## Detailed Findings

### 🔴 Critical — Non-Atomic Lock Release (Race Condition)

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)          # ← window opens here
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← another owner possible by now
```

Between `GET` and `DEL`, another process can acquire the lock after yours expires. You would then delete a lock you no longer own, corrupting the mutual exclusion guarantee entirely. This is a well-documented Redis anti-pattern.

**Required fix:** Use a Lua script to make the check-and-delete atomic.

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# Register once at __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat's `GET` + `expire` sequence and must be fixed identically.

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False       # shared across all callers
        self._heartbeat_thread = None        # overwritten on each call
```

If a single `InventoryService` instance is shared (standard in DI containers, Flask app context, etc.), two concurrent calls to `reserve_item` will clobber each other's `_heartbeat_active` flag. The first call's `_stop_heartbeat()` will silence the second call's heartbeat thread prematurely, allowing the second lock to expire mid-operation.

**Required fix:** Scope heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # atomic Lua extend here
            ...

    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release
```

Using `threading.Event` also eliminates the `time.sleep` polling loop, which currently cannot be interrupted cleanly.

### 🔴 Critical — No Transactional Integrity on DB Writes

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record. This is a silent inventory leak. The distributed lock does not protect against partial writes within the same holder.

**Required fix:** Wrap both operations in a database transaction or use a compensating rollback.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 High — Missing Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation at the entry point.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing prevents item_id="" or quantity=0 or quantity=-5
```

A caller passing `quantity=0` would succeed and create a zero-quantity order. A negative quantity would increase stock. An empty `item_id` would produce a lock key of `inventory:lock:` shared across all such calls.

**Required fix:**

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

Or introduce a proper request model validated at the boundary.

### 🟡 High — Test Fixture Silently Bypasses Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← replaces real method
    svc._stop_heartbeat = MagicMock()    # ← replaces real method
    return svc
```

Every test using this fixture never exercises the heartbeat or lock release path. The test `test_lock_not_released_if_expired` is supposed to verify atomic release behavior, but the fixture mock means `_stop_heartbeat` is never called through real code. The tests pass but prove almost nothing about the locking logic.

**Required fix:** Test the heartbeat and release paths directly with a real or tightly controlled mock that does not wholesale replace the methods. The lock release Lua path in particular needs an integration-style test.

### 🟡 Medium — Heartbeat Can Extend a Lock After Logical Release

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps FIRST
        if self._heartbeat_active:            # checks AFTER sleep
            ...
            self.redis.expire(lock_key, self.LOCK_TTL)
```

If `_stop_heartbeat()` is called while the thread is sleeping, the thread wakes up, passes the `if self._heartbeat_active` check (which is now `False`), and does nothing — that part is fine. However, if the sleep completes and `_heartbeat_active` is set to `False` between the `while` check and the `if` check, the extend still fires. This is a minor window but demonstrates the need for the `threading.Event` approach above.

---

### 🟢 Low — Daemon Thread Lifecycle Is Unobservable

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are silently killed on interpreter shutdown. If the process is terminated mid-reservation, there is no log, no cleanup, and no signal that a lock may have been left orphaned (relying solely on TTL expiry). This is acceptable for a daemon thread but should be documented explicitly, and the TTL should be treated as the true safety net, not the heartbeat.
