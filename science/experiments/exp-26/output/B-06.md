Decision: Request Changes

Summary: The heartbeat implementation contains a classic TOCTOU race that directly contradicts the PR's stated safety guarantee — the check-then-extend sequence in `_beat()` is non-atomic, meaning a lock can expire between the `GET` and `EXPIRE` calls, allowing another process to acquire it while the heartbeat falsely believes it still holds ownership. Beyond that critical flaw, the service carries several additional correctness, safety, and reliability problems detailed below.

---

## Critical: Non-Atomic Heartbeat Extension (TOCTOU Race)

This is the core flaw the PR claims to prevent.

```python
# services/inventory.py — _beat() inside _start_heartbeat()
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the following sequence is possible:

1. Heartbeat reads the key — token matches.
2. Redis TTL expires (network hiccup, GC pause, slow heartbeat thread scheduling).
3. Process B acquires the lock and writes its own token.
4. Heartbeat calls `EXPIRE`, extending **Process B's lock**.
5. Both processes now believe they hold the lock. Stale write occurs.

**Required fix:** Replace the two-command sequence with a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            ttl_ms = self.LOCK_TTL * 1000
            result = extend_lock(keys=[lock_key], args=[token, ttl_ms])
            if result == 0:
                logger.error(
                    "Heartbeat lost lock ownership — lock expired or stolen key=%s",
                    lock_key
                )
                self._heartbeat_active = False
                # Signal the owning thread that it must abort
                self._lock_lost.set()

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomic pattern must be applied to lock release (see below).

## Critical: Non-Atomic Lock Release

```python
# services/inventory.py — _acquire_lock() finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)   # another process can acquire between GET and DELETE
```

Same TOCTOU window. Use a standard Lua release script:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

This is the canonical Redlock release pattern and must not be split across two round-trips.

## Critical: Heartbeat Loss Is Silently Ignored

When the heartbeat detects that the lock has been stolen or has expired, it currently just stops looping. The business logic in `reserve_item()` continues executing `db.update_stock()` and `db.create_order()` with no awareness that lock ownership was lost. This is the stale write the PR is supposed to prevent.

**Required:** Introduce a `threading.Event` (`_lock_lost`) shared between the heartbeat and the critical section. After each DB operation, check the event:

```python
with self._acquire_lock(lock_key) as token:
    current_stock = self.db.get_stock(item_id)
    if self._lock_lost.is_set():
        raise LockLostError("Lock lost after stock read")

    self.db.update_stock(item_id, new_stock)
    if self._lock_lost.is_set():
        raise LockLostError("Lock lost after stock update — requires compensating transaction")

    self.db.create_order(order_id, item_id, quantity)
```

## High: Instance-Level Heartbeat State Prevents Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped. If two threads call `reserve_item()` on the same `InventoryService` instance simultaneously (entirely plausible in a web worker pool), one thread's `_stop_heartbeat()` will cancel the other thread's heartbeat, or the second `_start_heartbeat()` call will orphan the first thread. The heartbeat state must be encapsulated per lock acquisition, not per service instance.

```python
@dataclass
class _LockContext:
    token: str
    lock_key: str
    heartbeat_active: threading.Event
    lock_lost: threading.Event
    thread: threading.Thread
```

## High: `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the lock is lost between these two calls, stock is decremented but no order exists — inventory is silently destroyed. These two operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## High: No Input Validation on `item_id` or `quantity`

The PR checklist claims "Input validation via Pydantic," but `reserve_item()` accepts raw `str` and `int` with no validation. There is no Pydantic model wrapping the inputs. A caller can pass `quantity=0`, `quantity=-5`, or an empty/injected `item_id` string.

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...

## Medium: Heartbeat Interval Allows a Full TTL Expiry Window

With `LOCK_TTL = 30s` and `HEARTBEAT_INTERVAL = 10s`, the heartbeat extends the lock with 20 seconds remaining — acceptable in isolation. However, the heartbeat thread is a daemon thread subject to GIL contention and OS scheduling delays. Under a GC pause or CPU saturation, the 10-second sleep can drift significantly. The interval should be a fraction of the TTL with explicit drift tolerance documented:

```python
LOCK_TTL_MS = 30_000
HEARTBEAT_INTERVAL = 8      # extend at ~27% of TTL remaining, not 33%
HEARTBEAT_DRIFT_BUDGET = 2  # seconds of acceptable scheduler delay
# Invariant: HEARTBEAT_INTERVAL + HEARTBEAT_DRIFT_BUDGET < LOCK_TTL / 2
```

## Medium: Test Suite Mocks Out the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture exercises zero heartbeat logic. The test `test_lock_not_released_if_expired` is the only test that touches Redis mock behavior, but it also uses a mocked heartbeat. There are no tests for:

- Heartbeat detecting a stolen lock and signaling the critical section.
- Concurrent calls on the same instance corrupting heartbeat state.
- The atomic Lua extension returning `0` mid-operation.
- `db.update_stock` succeeding but `db.create_order` raising — verifying rollback.

## Low: `RuntimeError` Is Too Broad for Lock Failure

```python
raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")
```

Catching `RuntimeError` in `reserve_item()` will also swallow unrelated runtime errors from inside the lock body. Define explicit exception types:

```python
class LockAcquisitionError(Exception): ...
class LockLostError(Exception): ...
```
