Decision: Request Changes

Summary: The heartbeat implementation contains a race condition that directly contradicts the PR's core safety claim — a GC pause or thread suspension between the token check and the `expire` call can allow the lock to expire, be acquired by another process, and then have its TTL silently extended by the stale heartbeat, enabling concurrent writes to the same inventory record. Beyond this critical flaw, the service carries several additional architectural and correctness issues that must be resolved before this handles real flash sale load.

---

## Critical: Heartbeat Race Condition (TOCTOU on Lock Ownership)

This is the central flaw. The heartbeat performs a non-atomic check-then-act:

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend — NOT atomic with (1)+(2)
```

Between steps (1) and (3), the original lock can expire and be acquired by Process B with a different token. The heartbeat thread then calls `expire` on Process B's lock, extending a lock it no longer owns. The result is exactly the scenario the PR claims to prevent: two processes believe they hold the lock simultaneously, and both proceed to write inventory.

**Required fix:** Replace the get/compare/expire sequence with a single atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same atomic pattern must also be applied to the lock release in `_acquire_lock`. The current release has the identical TOCTOU window:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can acquire between get and delete
```

Both operations must be atomic via Lua or a library that guarantees this (e.g., `redis-py-lock`).

## High: Shared Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance, the second call's `_start_heartbeat` overwrites `_heartbeat_active` and `_heartbeat_thread`. When either call finishes and invokes `_stop_heartbeat`, it sets `_heartbeat_active = False`, which terminates the heartbeat for **both** locks. The surviving lock then expires undefended.

**Required fix:** Heartbeat state must be scoped to the lock context, not the instance. Use a local `threading.Event` passed into the heartbeat closure, or encapsulate lock state in a dedicated object per acquisition.

---

## High: No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative units would pass the stock check and corrupt inventory)
- `item_id` being an empty string (produces a valid but meaningless lock key `inventory:lock:`)
- `quantity` being a non-integer passed by a dynamic caller

**Required fix:** Validate inputs at the boundary. A Pydantic model for the request, or explicit guards at the top of `reserve_item`, are both acceptable — but one must exist.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

## High: Database Operations Are Not Atomic

Within the lock boundary, two separate database calls are made with no transaction:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is killed, or the DB connection drops after `update_stock` but before `create_order`, stock is decremented with no corresponding order record. This is a silent data integrity failure — inventory is consumed but no order exists.

**Required fix:** Both operations must execute within a single database transaction, or the service must implement a compensating rollback on failure:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB transaction failed, reservation rolled back")
    raise
```

## Medium: Heartbeat Does Not Signal Lock Loss Back to the Critical Section

If the heartbeat detects that the token no longer matches (the lock was lost), it silently stops extending. The main thread has no way to know the lock was lost and continues executing the critical section — reading stale stock and writing against data another process may already be modifying.

**Required fix:** Use a `threading.Event` or similar mechanism to signal lock loss to the main thread, which should then abort the operation:

```python
lock_lost_event = threading.Event()

def _beat():
    while not stop_event.is_set():
        time.sleep(self.HEARTBEAT_INTERVAL)
        owned = self._extend_lock_atomic(lock_key, token)
        if not owned:
            lock_lost_event.set()
            return

# In the critical section, check periodically or after each DB call:
if lock_lost_event.is_set():
    raise RuntimeError("Distributed lock lost during operation")
```

---

## Medium: `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The heartbeat thread is sleeping for up to `HEARTBEAT_INTERVAL` (10 seconds) when this flag is set. The `finally` block in `_acquire_lock` proceeds immediately to delete the lock key, but the heartbeat thread may wake up within that window and call `expire` on a key that has already been deleted and potentially re-acquired by another process.

**Required fix:** Join the thread with a timeout after setting the stop flag:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

## Low: Tests Mock Away the Behavior Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat is never exercised
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the primary safety mechanism this PR introduces, and it is mocked out in every test except `test_lock_not_released_if_expired`. The tests validate the happy path of the business logic but provide no coverage for the concurrent safety properties. Tests should exist for: heartbeat extending a valid lock, heartbeat skipping extension on a stolen lock, and the main thread aborting when lock loss is detected.
