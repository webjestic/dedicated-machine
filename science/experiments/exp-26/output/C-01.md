Decision: Request Changes

Summary: The core locking logic is well-structured and the ownership-verification pattern on release is correct, but there are several meaningful flaws that need to be addressed before this is safe to merge into a production inventory system. The most critical issues are a race condition in the heartbeat, a shared-state concurrency bug that makes the service non-reentrant, and missing input validation that the checklist incorrectly claims is present.

---

## Detailed Review

### 🔴 Critical — Must Fix

**1. Shared heartbeat state makes `InventoryService` non-reentrant**

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two calls to `reserve_item` are made concurrently on the same `InventoryService` instance (which is the normal case for a shared service object), they will clobber each other's heartbeat state. One call stopping its heartbeat will silently kill the other's lock renewal.

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to the individual lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat thread and close over it locally in `_acquire_lock`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquisition logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

**2. Race condition between heartbeat check and TTL extension**

The heartbeat does a `GET` followed by an `EXPIRE` as two separate commands. Between those two operations, the lock can expire and be acquired by another process. You would then extend a lock you no longer own.

```python
# Vulnerable — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # another owner by now?
```

This must be a single atomic operation. Use a Lua script:

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In the heartbeat thread
result = self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
    stop_event.set()
```

**3. Input validation is claimed but not implemented**

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` with no validation whatsoever. A negative quantity, an empty string `item_id`, or a non-integer quantity will pass through silently and corrupt inventory data.

```python
# item_id="" and quantity=-5 both reach db.update_stock unchecked
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

Add a request model and validate at the entry point:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

### 🟡 Significant — Should Fix

**4. Heartbeat thread is never joined**

`_stop_heartbeat` sets the flag and returns immediately. The daemon thread may still be running when the lock is deleted, and on a slow iteration it could attempt to extend a lock that now belongs to another process. Even with the Lua fix above, the thread should be joined before the lock is released.

**5. `db.update_stock` and `db.create_order` are not atomic**

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order exists. The distributed lock protects against concurrent reads, but it does not protect against a partial write leaving the system in an inconsistent state.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # failure here = lost inventory
```

These two writes must be wrapped in a database transaction, or the stock decrement must be rolled back on failure. This is an inventory integrity issue.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

**6. Bare `RuntimeError` is too broad in the caller**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` from inside the `with` block — including bugs in `db.update_stock` — will be silently swallowed and returned to the caller as "temporarily unavailable." Use a dedicated exception class for lock failures.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError in reserve_item
```

---

### 🟢 Minor — Nice to Fix

**7. Tests mock away the exact behavior under test**

`test_lock_not_released_if_expired` is the most important test in the file, but the service fixture globally mocks `_start_heartbeat` and `_stop_heartbeat`, meaning the heartbeat is never exercised in any test. There is no test that verifies the Lua atomicity, no test for concurrent calls on the same instance, and no test for the partial-write failure scenario. Consider adding integration-level tests with a real Redis instance (e.g., via `fakeredis`) for the locking path.

**8. `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` leaves a narrow window**

If the heartbeat thread is delayed by GIL pressure or system load, there are only two renewal opportunities before expiry. Consider making these configurable via constructor parameters so operators can tune them per deployment environment.
