Decision: Request Changes

Summary: The core locking logic contains a race condition in the heartbeat and release path — the check-then-act on `redis.get` followed by `redis.expire` or `redis.delete` is non-atomic, meaning a lock could be stolen between the ownership check and the operation. Beyond that critical flaw, the `InventoryService` stores heartbeat state as instance variables, making the class unsafe for concurrent use, and the DB operations lack transactional guarantees. The tests mock away the heartbeat entirely and do not validate the actual atomic behavior being claimed.

---

## Detailed Findings

### 🔴 Critical — Non-Atomic Lock Operations (Race Condition)

Both the heartbeat renewal and the lock release perform a **check-then-act** sequence that is not atomic:

```python
# Heartbeat — NOT atomic
current = self.redis.get(lock_key)          # 1. read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # 2. extend — lock could expire between 1 and 2
```

```python
# Release — NOT atomic
current = self.redis.get(lock_key)          # 1. read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # 2. delete — another owner could acquire between 1 and 2
```

Between the `GET` and the `EXPIRE`/`DEL`, the lock can expire and be acquired by another process. You would then extend or delete a lock you no longer own. This is a well-known failure mode. The canonical fix is a **Lua script** that makes the check and the action a single atomic operation:

```lua
-- Atomic release
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```lua
-- Atomic renewal
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

These should be registered as constants and called via `redis.eval()` or `redis.register_script()`.

### 🔴 Critical — Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the instance, not scoped to a single lock acquisition. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously:

- Thread A sets `_heartbeat_active = True` and starts a thread
- Thread B sets `_heartbeat_active = False` via `_stop_heartbeat()`, killing Thread A's heartbeat prematurely
- Thread A's lock now expires under load — exactly the failure this feature claims to prevent

Heartbeat state must be scoped to the lock context, not the service instance. Pass a `threading.Event` into the heartbeat function:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

### 🔴 Critical — No Database Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping them. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. This is a data integrity violation — inventory is silently lost. These must execute within a single atomic transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction needs to expose a transaction context manager, and the tests need to verify rollback behavior on partial failure.

---

### 🟡 Major — Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative items would silently succeed or corrupt stock)
- `item_id` being an empty string or containing injection-relevant characters
- Extremely large `quantity` values

A Pydantic model should be used as the input contract:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 🟡 Major — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread could still be mid-execution — reading the key, about to extend it — when the lock is released. The release path should join the thread with a bounded timeout before executing the delete:

```python
stop_event.set()
self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 🟡 Major — Tests Do Not Test What They Claim

The `service` fixture mocks out the heartbeat entirely:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means none of the tests exercise the actual locking or heartbeat behavior. `test_lock_not_released_if_expired` claims to test expiry behavior but is only testing a mocked token comparison — the Lua atomicity concern is completely untested. Tests should be added for:

- Concurrent reservation attempts using `threading.Thread` to verify only one succeeds
- Heartbeat actually extending TTL under a slow DB mock
- Partial DB failure triggering rollback and leaving stock unchanged

---

### 🟢 Minor — Backoff Does Not Include Jitter

```python
backoff = (2 ** attempt) * 0.1
```

Under high concurrency, deterministic backoff causes all contending threads to retry at the same moment, creating thundering herd behavior. Add jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

---

### 🟢 Minor — `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These should be configurable at construction time to support different environments and SLAs, not hardcoded on the class. Pass them as constructor arguments with sensible defaults.
