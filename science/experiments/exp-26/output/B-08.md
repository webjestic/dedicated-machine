Decision: Request Changes

Summary: The heartbeat mechanism contains a non-atomic check-then-extend race that directly undermines the lock safety guarantee this PR is designed to provide. Beyond that, the service carries several additional concurrency, correctness, and operational flaws that must be resolved before this code is safe to deploy in a high-contention flash sale context.

---

## Critical Issues

### 1. Heartbeat Race Condition — Non-Atomic Check-Then-Extend

This is the core flaw. The heartbeat does this:

```python
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2: extend
```

Between step 1 and step 2, the lock can expire and be acquired by a competing process. At that point, `expire` extends a lock you no longer own. This is the exact scenario the PR claims to prevent. The fix is a single atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same atomic Lua pattern must also be applied to lock release in `_acquire_lock`. The current release has the identical TOCTOU flaw:

```python
current = self.redis.get(lock_key)           # race window here
if current and current.decode() == token:
    self.redis.delete(lock_key)              # may delete a lock you no longer own
```

---

### 2. Shared Mutable Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — one call's `_stop_heartbeat()` will cancel the other call's heartbeat, leaving its lock unprotected. The heartbeat state must be scoped to each individual lock acquisition, not to the instance. Pass a `threading.Event` into `_start_heartbeat` and return it, keeping all state local to the `_acquire_lock` context.

---

### 3. Database Operations Are Outside Transaction Boundaries

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate calls with no transaction wrapping them. If the process dies, is paused by GC, or hits a network error between them, stock is decremented but no order exists — inventory is silently lost. This must be wrapped in a single atomic database transaction. The distributed lock serializes access across processes, but it does not provide atomicity for the database writes themselves.

---

### 4. Input Validation Is Declared But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` parameters with no validation whatsoever. `item_id` is passed directly into a Redis key and a database query without sanitization. A caller can pass `quantity=0`, `quantity=-5`, or an empty string. Pydantic models are defined for the *output* only. A request model must validate inputs at the boundary.

---

## Significant Issues

### 5. Heartbeat Thread Leaks on Lock Acquisition Failure Path

If `_start_heartbeat` is called and then an exception is raised before `yield`, the `finally` block calls `_stop_heartbeat()`, which sets `_heartbeat_active = False`. However, the thread is still running and will complete its current `sleep(HEARTBEAT_INTERVAL)` before observing the flag. During that window, it may extend a lock that has already been released or that belongs to another process. The heartbeat thread should be started *after* the `try` block opens, not before, and the stop should join the thread with a timeout to confirm it has actually exited.

### 6. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In a flash sale scenario with high concurrency, this blocks a thread for up to 700ms total across retries. Under load, this will exhaust thread pools. Retry logic in a high-concurrency context should use async I/O or at minimum be clearly documented as unsuitable for synchronous thread-pool deployment without a concurrency budget.

### 7. Redis Connection Errors Are Silently Swallowed

Only `RuntimeError` is caught. `redis.exceptions.RedisError`, `ConnectionError`, and `TimeoutError` propagate uncaught and will surface as 500s with no structured error response. All Redis operations should be wrapped to handle connection failures gracefully with the same `ReservationResult` error path.

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Out Entirely in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The most dangerous code path in this PR — the heartbeat — is completely untested. There are no tests for the atomic extension behavior, the concurrent instance state collision, or the thread lifecycle. These must be tested with a real or fake Redis (e.g., `fakeredis`) rather than mocked away.

### 9. No Test for Partial Database Failure

There is no test covering the case where `update_stock` succeeds and `create_order` raises. This is the data-loss scenario described in issue 3 and it must have explicit test coverage.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test confirms that `delete` is not called when the token doesn't match, but it does so after a successful reservation. It does not test the scenario where the lock expires mid-operation and is acquired by a second process, which is the actual threat model. The test should simulate concurrent acquisition and verify that the second holder's lock is not deleted by the first holder's release path.
