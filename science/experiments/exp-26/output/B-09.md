Decision: Request Changes

Summary: The heartbeat implementation contains a non-atomic check-then-extend race that directly violates the core safety guarantee this PR is trying to provide. Beyond that critical flaw, there are several additional correctness, concurrency, and operational issues that must be resolved before this code is safe to merge into a high-contention flash sale path.

---

## Critical: Heartbeat Race Condition (TOCTOU)

This is the primary blocker. The heartbeat performs a non-atomic read-then-extend:

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # Step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2: extend
```

Between Step 1 and Step 2, the lock can expire and be acquired by a competing process. The `expire` call then silently extends a lock this process no longer owns. This is precisely the stale-write scenario the PR claims to prevent.

**Required fix:** Replace the two-step operation with an atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.error(
                    "Heartbeat lost lock ownership — lock expired or stolen key=%s",
                    lock_key
                )
                self._heartbeat_active = False
                # Signal the owning thread that it must abort
                break

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

## Critical: Lock Release Is Also Non-Atomic

The same TOCTOU flaw exists in the `finally` block:

```python
current = self.redis.get(lock_key)           # Step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)              # Step 2 — wrong lock may be deleted
```

If the lock expires between Step 1 and Step 2, this deletes a lock owned by another process. Use the standard atomic Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

## Serious: Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (entirely plausible under flash sale load), they share and corrupt each other's heartbeat state. Thread A stopping its heartbeat will kill Thread B's renewal.

**Required fix:** Heartbeat state must be scoped to the lock acquisition context, not the instance. Use a local `threading.Event` per call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

## Serious: Heartbeat Loss Is Silently Ignored

When the heartbeat detects it no longer owns the lock (even with the current broken check), it logs and exits quietly. The business logic thread continues executing `db.update_stock` and `db.create_order` against data it no longer holds a lock on. The heartbeat thread must have a signaling path back to the critical section to force an abort.

The `threading.Event` approach above enables this:

```python
def _beat(stop_event, lost_event):
    while not stop_event.is_set():
        stop_event.wait(timeout=HEARTBEAT_INTERVAL)
        if stop_event.is_set():
            break
        result = extend_script(keys=[lock_key], args=[token, LOCK_TTL])
        if result == 0:
            logger.error("Lock lost key=%s", lock_key)
            lost_event.set()  # signal the main thread
            break
```

The critical section then checks `lost_event` at safe points, or the context manager raises on exit if the event is set.

## Moderate: `quantity` Has No Input Validation

`reserve_item` accepts `quantity: int = 1` but never validates it. A caller passing `quantity=0` silently succeeds and creates a zero-quantity order. A negative value would *increase* stock. Pydantic is imported but unused for this path.

```python
if quantity <= 0:
    return ReservationResult(
        success=False,
        message="Quantity must be a positive integer"
    )
```

---

## Moderate: `db.update_stock` and `db.create_order` Are Not Atomic

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. There is no rollback path. The database layer must wrap both operations in a transaction, or the service layer must handle partial failure explicitly:

```python
try:
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
except Exception:
    # attempt compensating write or re-raise with alarm
    logger.critical(
        "Partial write failure item_id=%s order_id=%s — manual reconciliation required",
        item_id, order_id
    )
    raise
```

---

## Moderate: Heartbeat Interval Timing Is Unsafe Under GC Pressure

`HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` gives a 20-second window before the first renewal. A GC pause or thread scheduling delay that exceeds 20 seconds (not unreasonable on a JVM-adjacent host or under memory pressure) will cause the lock to expire before the first heartbeat fires. The interval should be at most `LOCK_TTL / 3`, and the first extension should fire at `LOCK_TTL / 3` from acquisition, not from the last sleep boundary.

---

## Minor: Tests Mock Away the Code Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture replaces the heartbeat entirely. The tests confirm business logic branching but provide zero coverage of the concurrency mechanism that is the entire point of this PR. Integration tests using a real or fake Redis (e.g., `fakeredis`) with actual thread scheduling are required to validate the lock lifecycle.

---

## Minor: `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has exited before the lock is released. The thread could wake from `time.sleep`, see `_heartbeat_active = False`, but the main thread has already deleted the lock and returned. In the corrected design using `threading.Event`, call `thread.join()` in the `finally` block before the atomic release.
