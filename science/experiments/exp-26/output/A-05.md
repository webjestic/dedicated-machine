Decision: Request Changes

Summary: The heartbeat mechanism contains a race condition that directly violates the core safety guarantee this PR is intended to provide. Beyond that, there are several architectural and correctness issues that must be resolved before this code is safe to merge into a high-stakes flash sale reservation path.

---

## Critical Issues

### 1. Heartbeat Race Condition — Stale Write Window Still Exists

This is the primary flaw. The heartbeat does not close the race; it narrows it while leaving the fundamental vulnerability intact.

```python
# The check-then-extend sequence is not atomic
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend — NOT atomic with (1)+(2)
```

Between steps (1) and (3), the lock can expire and be acquired by another process. You are then extending a lock you no longer own, giving two processes simultaneous ownership. The correct approach is a Lua script that makes the check-and-extend atomic:

```lua
-- extend.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    extend_script = self.redis.register_script(EXTEND_SCRIPT)
    self._heartbeat_active = True

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if not result:
                    logger.error(
                        "Heartbeat lost lock ownership — lock expired or stolen. "
                        "key=%s", lock_key
                    )
                    # Signal the main thread that the lock is gone
                    self._heartbeat_active = False

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomicity problem applies to the lock release in `_acquire_lock`:

```python
# Also not atomic — same race
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This must also use a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. Heartbeat Loss Is Silently Ignored

When the heartbeat detects it has lost ownership (even with the current non-atomic version, `current != token`), it logs a debug message and continues. The main thread has no idea the lock is gone and proceeds to write to the database. This is the exact stale write scenario this PR claims to prevent.

The heartbeat thread must have a signaling mechanism back to the main thread — an `threading.Event`, a shared flag checked inside the lock body, or an exception raised on the worker. The `reserve_item` critical section must abort if lock ownership is lost mid-operation.

### 3. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — two concurrent reservations will clobber each other's `_heartbeat_active` and `_heartbeat_thread`. The second call stopping its heartbeat will stop the first call's heartbeat as well.

Heartbeat state must be scoped to each individual lock acquisition, not to the service instance. Use a local object or closure per acquisition context.

## Significant Issues

### 4. No Input Validation on Public Entry Points

The PR checklist claims Pydantic validation, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, negative quantities, empty `item_id` strings, or excessively large quantities. A caller passing `quantity=0` or `quantity=-5` would pass the stock check and write a corrupt order record. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access. These two operations must be wrapped in a database transaction. If the `db` abstraction does not support transactions, that is a design gap that must be addressed before this code is production-ready.

### 6. Heartbeat Interval Is Too Close to TTL

`LOCK_TTL = 30` seconds, `HEARTBEAT_INTERVAL = 10` seconds. The heartbeat fires at 10s, 20s, 30s. At exactly 30s the lock expires before the third heartbeat fires, depending on scheduling jitter and GC pauses. The interval should be no more than `TTL / 3`, and ideally `TTL / 4` to provide a meaningful safety margin. With a 30s TTL, use a 7s interval.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out in Every Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the central safety mechanism of this PR. Mocking it out means none of the tests exercise the behavior being claimed. There must be at least one integration-level test that runs the actual heartbeat thread and verifies that lock extension is called, and that lock loss is detected and handled correctly.

### 8. The Atomicity Flaw Is Not Tested

There is no test covering the window between `GET` and `EXPIRE`/`DEL` where another process could acquire the lock. This scenario should be tested with a mock that simulates the lock changing ownership between those two calls.

---

## Minor Issues

- `_stop_heartbeat` sets the flag but does not join the thread. The daemon thread may still be running when the lock has already been released, creating a brief window where it could attempt to extend a lock owned by another process.
- No timeout on `_heartbeat_thread.join()` means a misbehaving thread could block shutdown indefinitely.
- The `RuntimeError` catch in `reserve_item` is overly broad. A `RuntimeError` raised inside the lock body (e.g., from `db.get_stock`) would be caught and returned as "service temporarily unavailable" rather than propagating as an unexpected error.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat extend and lock release must use atomic Lua scripts |
| 2 | Critical | Heartbeat loss must abort the critical section, not be silently ignored |
| 3 | Critical | Heartbeat state must be per-acquisition, not per-instance |
| 4 | High | Input validation must be enforced at the service boundary |
| 5 | High | DB operations must be wrapped in a transaction |
| 6 | Medium | Heartbeat interval must provide adequate margin below TTL |
| 7 | Medium | Heartbeat must not be mocked out in tests |
| 8 | Medium | Atomicity race must have test coverage |
