Decision: Request Changes

Summary: The heartbeat mechanism contains a race condition that directly violates the core safety guarantee this PR is meant to provide. Beyond that, the service has an instance-level state design flaw that makes it unsafe for concurrent use, and the DB operations lack transactional atomicity, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory. These are not minor issues — they represent the exact failure modes a distributed lock is supposed to prevent.

---

## Critical Issues

### 1. Heartbeat Race Condition — Lock Can Still Expire

This is the primary concern given my review mandate.

```python
# UNSAFE — gap between check and extend
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps 10s
        if self._heartbeat_active:
            current = self.redis.get(lock_key)         # read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

**The problem:** `GET` and `EXPIRE` are two separate commands with no atomicity guarantee. Between the `GET` and the `EXPIRE`, the TTL can reach zero, the lock expires, a second process acquires it, and then this thread calls `EXPIRE` on the new owner's lock — silently extending a lock this process no longer holds. The heartbeat then falsely believes the lock is still owned, the critical section continues, and you have two writers.

**Fix:** Replace the `GET`/`EXPIRE` pair with a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)
    self._heartbeat_active = True

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL * 1000])
            if result == 0:
                logger.error(
                    "Heartbeat lost lock ownership — lock expired or stolen key=%s",
                    lock_key
                )
                self._heartbeat_active = False
                # Signal the main thread that the lock is gone
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The heartbeat must also signal the main thread when it detects lock loss. Right now it silently stops — the critical section keeps running with no lock.

### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
# On the class instance — shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance simultaneously, the second call overwrites `_heartbeat_active` and `_heartbeat_thread`, silently killing the first caller's heartbeat. This is a direct path to lock expiry under concurrent load — exactly the scenario this service is designed for.

**Fix:** Scope heartbeat state to each lock acquisition, not to the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

### 3. Non-Atomic DB Operations — Inventory Corruption on Partial Failure

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # crashes
```

If the process dies, receives SIGKILL, or the DB connection drops between these two calls, stock is decremented but no order exists. The item is gone with no record. This is a silent data loss bug.

**Fix:** Wrap both operations in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If your DB layer does not expose transactions, that is a prerequisite that must be addressed before this service ships.

## Moderate Issues

### 4. No Input Validation on `item_id` or `quantity`

The Pydantic checklist item is misleading — `ReservationResult` uses Pydantic, but the inputs to `reserve_item` do not.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a string containing Redis key injection characters. The lock key `f"inventory:lock:{item_id}"` is constructed directly from untrusted input.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is not joined
```

The `finally` block sets the flag and immediately proceeds to release the lock. The heartbeat thread may still be mid-execution — between its `GET` and `EXPIRE` — when the lock is deleted and potentially re-acquired by another process. The stale `EXPIRE` then lands on the new owner's lock.

**Fix:** Join the thread with a bounded timeout before releasing:

```python
def _stop_heartbeat(self, heartbeat_thread: threading.Thread) -> None:
    self._heartbeat_active = False
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if heartbeat_thread.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly")
```

### 6. Lock Release Has the Same Non-Atomic GET/DELETE Problem

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same race as the heartbeat: between `GET` and `DELETE`, the lock can expire and be acquired by another process. The `DELETE` then removes the new owner's lock.

**Fix:** Use the standard Redlock release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

## Test Coverage Gaps

### Heartbeat Is Fully Mocked Out

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # bypassed entirely
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most complex and failure-prone component in this PR. Mocking it out means zero test coverage for the mechanism that is supposed to prevent lock expiry. Tests should exist for:

- Heartbeat correctly extends TTL using the atomic Lua path
- Heartbeat detects lock loss and signals the main thread
- Heartbeat thread terminates cleanly when the lock is released
- Concurrent calls to `reserve_item` on the same instance do not interfere with each other's heartbeat state

| # | Issue | Severity |
|---|-------|----------|
| 1 | Non-atomic GET/EXPIRE in heartbeat — stale extend possible | **Blocking** |
| 2 | Instance-level heartbeat state — concurrent calls corrupt each other | **Blocking** |
| 3 | Non-atomic DB operations — inventory corruption on partial failure | **Blocking** |
| 4 | Heartbeat not joined before lock release — stale EXPIRE on new owner | **Blocking** |
| 5 | Non-atomic GET/DELETE on lock release | **Blocking** |
| 6 | No input validation on `item_id` or `quantity` | **High** |
| 7 | Heartbeat mocked in tests — core mechanism untested | **High** |

The architecture is sound in intent — distributed lock with heartbeat renewal is the right pattern here. The implementation just needs to follow through on the atomicity guarantees that make the pattern safe.
