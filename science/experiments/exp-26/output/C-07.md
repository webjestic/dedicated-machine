Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — distributed lock ownership verification, heartbeat renewal, exponential backoff, and structured logging are all present and correctly motivated. However, there are several correctness, safety, and architectural issues that must be addressed before this handles real flash sale load in a government-grade system. The most critical problems are a race condition in the heartbeat's check-then-extend sequence, a shared-state concurrency bug that makes the service non-reentrant, and missing input validation despite the checklist claiming otherwise.

---

## Detailed Findings

### 🔴 Critical

**1. Heartbeat has a TOCTOU race condition (`services/inventory.py`, `_start_heartbeat`)**

```python
# Current — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock with a new token. The heartbeat would then extend a lock it no longer owns. This must be an atomic operation. Use a Lua script:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at __init__
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership, stopping. key=%s", lock_key)
    self._heartbeat_active = False
```

The same Lua pattern should also be applied to lock release in `_acquire_lock`.

**2. Instance-level heartbeat state makes the service non-reentrant (`services/inventory.py`)**

```python
# These are instance attributes — shared across all concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat. The lock for the first reservation will then expire mid-operation.

Move heartbeat state into the lock context so each acquisition is fully isolated:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into _start_heartbeat, stop via heartbeat_active.clear()
    ...
```

### 🟠 High

**3. Input validation is not actually implemented (checklist item is incorrect)**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is nothing preventing:

```python
service.reserve_item("", quantity=-5)
service.reserve_item("item_001 OR 1=1", quantity=0)
service.reserve_item("a" * 10_000, quantity=999999)
```

A negative quantity would silently *increase* stock. An empty `item_id` would produce a nonsensical lock key. At minimum:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation input: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    ...
```

**4. `db.update_stock` and `db.create_order` are not atomic with respect to each other**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the DB connection drops between these two calls, stock is decremented but no order record exists. The Redis lock does not protect against this — it only serializes access. This requires either:

- A database transaction wrapping both operations, or
- An outbox/saga pattern if the DB and order service are separate

At minimum, wrap in a transaction and handle the failure case explicitly:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Medium

**5. Heartbeat interval does not guarantee renewal before expiry under all conditions**

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The ratio is fine in the happy path, but `time.sleep(self.HEARTBEAT_INTERVAL)` is not clock-accurate under GIL pressure or system load. If the heartbeat thread is starved, the 30-second TTL may expire before renewal. Consider:

- Reducing the ratio (e.g., renew at TTL/3 = 10s is already at that boundary — tighten to TTL/4)
- Using `threading.Event.wait(timeout)` instead of `time.sleep` so the thread can be woken for clean shutdown

```python
stop_event = threading.Event()
stop_event.wait(timeout=self.HEARTBEAT_INTERVAL)  # interruptible
```

---

**6. `RuntimeError` is too broad an exception to catch**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` will also catch unexpected programming errors from inside the `with` block (e.g., a `RuntimeError` raised by a third-party DB driver for a completely unrelated reason), masking real bugs. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

### 🔵 Low / Test Quality

**7. Tests mock out the heartbeat entirely, so the most critical code path is never exercised**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the TOCTOU race and the non-reentrant state bugs described above would never be caught by this test suite. Add at least:

- An integration-style test that lets the real heartbeat run and verifies TTL is extended
- A concurrency test using `threading.Thread` to confirm two simultaneous reservations on the same instance do not corrupt each other's heartbeat state

**8. `test_lock_not_released_if_expired` does not test what it claims**

The test mocks `redis.get` to return `b"different-token"` globally, but the token generated inside `_acquire_lock` is a real `uuid4` that will never equal `b"different-token"`. The assertion passes for the wrong reason. The test should capture the actual token used and inject it properly, or use a seeded UUID mock.
