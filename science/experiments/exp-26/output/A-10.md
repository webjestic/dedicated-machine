Decision: Request Changes

Summary: The heartbeat mechanism contains a classic check-then-act race condition: it reads the token with `GET` and then calls `EXPIRE` as two separate, non-atomic operations, meaning the lock can expire and be acquired by another process in the window between those two calls. Beyond that race, the service carries a shared-state concurrency defect that makes it unsafe under any real concurrent load, and the database operations lack transactional atomicity guarantees. These issues together defeat the core safety guarantee the PR is trying to provide.

---

## Detailed Findings

### 1. Non-Atomic Heartbeat Renewal (Critical — Safety Violation)

This is the most serious flaw and directly triggers the rejection criterion in my review mandate.

```python
# UNSAFE — two separate round-trips, not atomic
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the following can happen:

1. The lock TTL expires naturally (e.g., during a GC pause or network hiccup).
2. Process B acquires the lock and writes its own token.
3. This heartbeat thread wakes up, reads Process B's token, finds it does **not** match, and skips the extend — which is correct *in that case*. **But** if the timing is slightly different and Process B hasn't written yet when step (1) executes, this thread will extend a lock it no longer rightfully owns, effectively stealing it back from Process B mid-operation.

The fix is a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            ttl_ms = self.LOCK_TTL * 1000
            result = extend_script(keys=[lock_key], args=[token, ttl_ms])
            if result == 0:
                logger.error(
                    "Heartbeat lost lock ownership — lock expired or stolen key=%s",
                    lock_key
                )
                self._heartbeat_active = False
                # Signal the main thread that the lock is gone
                # (see finding #3 for how to act on this)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

Similarly, the lock release in `_acquire_lock` has the same GET-then-DELETE race and must also use a Lua script:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. Shared Mutable State Makes the Service Non-Reentrant and Unsafe Under Concurrency (High)

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level** fields. If two threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the exact scenario this code is designed for), they will race on `_heartbeat_active` and `_heartbeat_thread`. One thread stopping its heartbeat will silently kill the other thread's lock renewal.

The fix is to make heartbeat state local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()   # local to this acquisition
    acquired = False

    # ... acquire logic unchanged ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... atomic release via Lua script ...
```

```python
def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            ttl_ms = self.LOCK_TTL * 1000
            result = extend_script(keys=[lock_key], args=[token, ttl_ms])
            if result == 0:
                logger.error("Lost lock ownership key=%s", lock_key)
                return

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note also that `_stop_heartbeat` currently just sets a flag and returns immediately. The main thread can proceed to the finally block and attempt lock release before the heartbeat thread has actually stopped, creating a brief window where both the heartbeat and the main thread are touching Redis state. Using `stop_event` + `join()` eliminates this.

### 3. No Signal Path When the Heartbeat Detects Lock Loss (High)

If the heartbeat determines it no longer owns the lock (finding #1 above, after the Lua fix), it currently just logs and returns. The main thread continues executing `db.update_stock` and `db.create_order` without a lock, which is the exact stale-write scenario this architecture is meant to prevent.

The heartbeat needs a way to signal the main thread to abort. A simple approach:

```python
lock_lost_event = threading.Event()

# In heartbeat:
if result == 0:
    logger.error("Lock lost key=%s — signaling main thread", lock_key)
    lock_lost_event.set()
    return

# In reserve_item, at critical checkpoints:
if lock_lost_event.is_set():
    raise LockLostError(f"Distributed lock expired during operation for {item_id}")
```

Check `lock_lost_event` before each DB call (`get_stock`, `update_stock`, `create_order`).

---

### 4. Database Operations Are Not Transactional (Medium)

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. This is a data integrity issue independent of the locking logic. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the atomicity requirement is non-negotiable for an inventory system.

---

### 5. Input Validation Is Incomplete (Low)

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation at the service boundary. `quantity` can be zero or negative, and `item_id` is unconstrained. Add a Pydantic request model:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(default=1, ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

---

### 6. Test Coverage Gaps (Low)

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic itself has zero test coverage. The most important behavior — lock loss detection, atomic renewal, and the signal path to the main thread — is completely untested. At minimum, add:

- A test that verifies the Lua renewal script is called with the correct token.
- A test that verifies `reserve_item` aborts and does not call `update_stock` when `lock_lost_event` is set mid-operation.
- A test for `quantity <= 0` and oversized `item_id` inputs.
