Decision: Request Changes

Summary: The heartbeat mechanism contains a race condition that directly violates the core safety guarantee this PR is trying to provide. Beyond that, the service has an instance-level state design flaw that makes it unsafe under any concurrent workload, which is precisely the scenario this code is meant to handle. Several secondary issues compound the risk enough that this cannot be approved in its current form.

---

## Critical Issues

### 1. Race Condition in Heartbeat: Check-Then-Act Is Not Atomic

This is the primary rejection reason. The heartbeat's ownership check and TTL extension are two separate Redis operations with no atomicity guarantee:

```python
# _start_heartbeat — NOT atomic
current = self.redis.get(lock_key)          # Step 1: read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Step 2: extend TTL
```

Between Step 1 and Step 2, the lock can expire and be acquired by a competing process. The heartbeat then blindly extends the TTL on a lock it no longer owns, keeping a foreign process's lock alive and defeating the entire mutual exclusion guarantee. This is the exact failure mode the system prompt requires rejection for.

**Required fix:** Replace with a Lua script that atomically checks ownership and extends in a single round-trip:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)
    self._heartbeat_active = True

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if result == 0:
                    logger.error(
                        "Heartbeat lost lock ownership key=%s — halting extension",
                        lock_key
                    )
                    self._heartbeat_active = False

        self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
        self._heartbeat_thread.start()
```

### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance, not scoped to the lock context:

```python
self._heartbeat_active = False          # shared across all callers
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (the normal deployment model), one thread's `_stop_heartbeat()` will cancel the other thread's heartbeat. This silently drops TTL renewal for an active lock, allowing it to expire mid-operation and enabling a competing process to acquire the lock and perform a stale write.

**Required fix:** Scope heartbeat state to the lock context using a local `threading.Event` and pass it through closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... retry logic unchanged ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua (see issue #1)
        ...

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.error("Lost lock ownership key=%s", lock_key)
                return

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 3. Lock Release Is Also Non-Atomic

The same check-then-act pattern appears in the `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be re-acquired. The `DELETE` then removes the new owner's lock, not yours. This is the canonical Redis distributed lock mistake. The fix is the same pattern — a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

## Moderate Issues

### 4. No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, or pathologically long strings that could be used to construct oversized Redis keys. A negative quantity would silently *increase* stock, which is a correctness and potential abuse vector.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
if len(item_id) > 256:
    return ReservationResult(success=False, message="item_id exceeds maximum length")
```

### 5. Heartbeat Interval Is Too Close to Lock TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10s, 20s, and 30s. At exactly 30s, the lock has already expired before the third heartbeat fires. Under any scheduler jitter, GC pause, or network hiccup, the heartbeat at t=29.9s may arrive after the TTL has elapsed. The interval should be no more than one-third of TTL with explicit margin:

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # fires at 8s, 16s, 24s — clear margin before 30s expiry
```

### 6. Daemon Thread Is Abandoned on Process Suspension

`daemon=True` means the heartbeat thread is silently killed if the main thread blocks (e.g., GC pause, OS scheduling). There is no mechanism to detect that the heartbeat stopped and abort the critical section. After the fixes above, the heartbeat thread should signal back to the main context if it loses ownership, and the main thread should check that signal before committing writes.

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Away in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test that uses the `service` fixture bypasses the heartbeat entirely. The most critical behavior — atomic extension, ownership loss detection, concurrent instance safety — has zero test coverage. At minimum, add:

- A test verifying that the Lua extension script is called with the correct token
- A test verifying that a failed extension (returns 0) halts the heartbeat and signals the caller
- A test verifying two concurrent `reserve_item` calls on the same instance do not interfere with each other's heartbeat state

### 8. `test_lock_not_released_if_expired` Does Not Test the Real Release Path

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test validates the non-atomic `GET`/`DELETE` pattern that is itself a bug. Once the release is replaced with a Lua script, this test needs to be rewritten to verify the script is called with the correct arguments and that a return value of `0` is handled gracefully.
