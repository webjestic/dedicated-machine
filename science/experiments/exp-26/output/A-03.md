Decision: Request Changes

Summary: The heartbeat mechanism contains a **race condition that directly violates the core safety invariant this PR is trying to enforce**. Beyond that, the service has an instance-level state design flaw that makes it unsafe under concurrent call patterns, and the test suite mocks away the heartbeat entirely — meaning the most critical and novel piece of logic has zero verified coverage. These issues collectively mean the lock expiry protection the PR advertises cannot be relied upon.

---

## Critical Issues

### 1. Heartbeat Race: Check-Then-Act Is Not Atomic

```python
# _beat() in _start_heartbeat:
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend — NOT atomic with (1)+(2)
```

Between steps (1) and (3), the lock can expire and be acquired by a second process. The heartbeat then extends a lock it no longer owns, giving the original process a false sense of continued ownership while the second process is already inside the critical section. **This is the exact failure mode the PR claims to prevent.**

The correct fix is a Lua script that atomically checks ownership and extends in a single round-trip:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The heartbeat must also act on a return value of `0` — it means the lock was lost and the critical section must be aborted, not silently continued.

---

### 2. Heartbeat Provides No Signal Back to the Critical Section

Even if the atomic extension is implemented, the current design has no mechanism to notify `reserve_item` that the lock was lost mid-operation. If the heartbeat detects loss, the DB writes proceed anyway:

```python
self.db.update_stock(item_id, new_stock)   # executes even if lock is gone
self.db.create_order(order_id, item_id, quantity)
```

A threading `Event` or similar cancellation token must be threaded through so the critical section can check and abort before committing writes.

---

### 3. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance fields, not call-scoped locals. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — a completely normal scenario in any web framework — one call's `_stop_heartbeat()` will cancel the other call's heartbeat thread. The second lock will silently stop renewing with no error raised.

Each lock acquisition must carry its own isolated cancellation state, scoped to the `_acquire_lock` context manager, not stored on `self`.

---

## Significant Issues

### 4. Lock Release Is Also Not Atomic

```python
# In _acquire_lock finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same check-then-act problem as the heartbeat. Between `GET` and `DELETE`, the lock can expire and be re-acquired. The `DELETE` then removes a lock owned by another process. This must also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

---

### 5. No Input Validation on Public Entry Point

The PR checks the Pydantic box, but `reserve_item` accepts raw `str` and `int` with no validation before the lock is acquired:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is interpolated directly into `f"inventory:lock:{item_id}"` — a Redis key. Malformed or adversarially crafted IDs (e.g., containing `:` or whitespace) can collide with or corrupt the key namespace.
- `quantity <= 0` is never rejected, meaning a caller can pass `quantity=0` and succeed with a no-op reservation, or pass a negative value and increment stock.

---

### 6. Test Suite Mocks Away the Feature Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

Every test that uses the `service` fixture exercises zero heartbeat logic. The atomic extension race, the lost-lock signal path, and the instance-state concurrency bug are all invisible to this suite. A dedicated test must construct a real `InventoryService`, drive the heartbeat's internal `_beat` function directly with controlled token states, and assert on the Lua script call and the cancellation signal.

---

## Minor Issues

### 7. Daemon Thread Leak on Unhandled Exceptions

If `self.db.get_stock` raises an unexpected exception, `_stop_heartbeat` is still called in `finally`, but the heartbeat thread is set as `daemon=True` and is never `.join()`ed. Under high request volume this accumulates leaked threads until process exit. The `finally` block should call `self._heartbeat_thread.join(timeout=...)` after stopping.

### 8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Magic Class Constants

These are not configurable at construction time. Different deployment environments (e.g., a slow government network path to the DB vs. a co-located instance) will need different values. Accept them as constructor parameters with the current values as defaults.

---

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Heartbeat extension must be a single atomic Lua script |
| 2 | Critical | Lost-lock must signal and abort the critical section |
| 3 | Critical | Heartbeat state must be call-scoped, not instance-scoped |
| 4 | High | Lock release must be a single atomic Lua script |
| 5 | High | Validate `item_id` format and `quantity > 0` before lock acquisition |
| 6 | High | Tests must cover real heartbeat behavior, not mock it out |
| 7 | Low | Join heartbeat thread in `finally` to prevent leak |
