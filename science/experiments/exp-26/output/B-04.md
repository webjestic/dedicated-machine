Decision: Request Changes

Summary: The heartbeat implementation contains a non-atomic check-then-extend race that can allow a competing process to acquire the lock between the `GET` and `EXPIRE` calls, directly violating the stated safety guarantee. Beyond that critical flaw, the service has several additional correctness, concurrency, and reliability problems that must be resolved before this handles real flash-sale traffic.

---

## Critical: Heartbeat Race Condition (Non-Atomic Check-Then-Extend)

This is the core flaw the PR claims to solve, but does not.

```python
# _start_heartbeat — UNSAFE
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the lock can expire and be acquired by Process B. Process A then extends Process B's lock, giving Process A an implicit belief it still holds the lock while Process B also believes it holds the lock. Both processes now operate inside what they believe is an exclusive critical section. This is the exact stale-write scenario the PR is supposed to prevent.

**Required fix:** Replace the two-command sequence with a single atomic Lua script.

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                # Lock was lost; signal the owning thread
                logger.error("Heartbeat lost lock key=%s — signaling abort", lock_key)
                self._heartbeat_active = False  # or set a shared abort event
    ...
```

The same non-atomic pattern exists in the lock release path and must also be fixed with a Lua script (see below).

## High: Non-Atomic Lock Release

```python
# _acquire_lock finally block — UNSAFE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)   # another process may have acquired between GET and DEL
```

This is a well-documented Redis distributed lock anti-pattern. Use the canonical atomic release script instead.

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""
```

## High: Shared Mutable Heartbeat State Across Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously, they share and overwrite each other's heartbeat state.

```python
# Thread A sets this...
self._heartbeat_active = True
# Thread B immediately overwrites it for a different lock
self._heartbeat_active = True
# Thread A calls _stop_heartbeat(), killing Thread B's heartbeat too
self._heartbeat_active = False
```

**Required fix:** Heartbeat state must be scoped to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

## High: Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. The distributed lock provides no protection against this partial write.

**Required fix:** These two operations must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is a blocker that must be resolved at the interface level before this service is safe to ship.

## Medium: Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # _heartbeat_thread is never joined
```

The thread may still be executing `redis.expire` after the lock has been released and potentially re-acquired by another process. The `finally` block must join the thread before proceeding with lock release.

```python
stop_event.set()
if heartbeat_thread.is_alive():
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
# Only now is it safe to release the lock
```

## Medium: Heartbeat Does Not Signal Lock Loss to the Owning Thread

If the heartbeat detects that the token no longer matches (lock was lost), it silently stops. The business logic in `reserve_item` continues executing, reads stale stock, and writes to the database without holding the lock.

**Required fix:** Use a shared `threading.Event` as an abort signal. The critical section must check this event at meaningful points, or the lock context manager must raise on exit if the lock was lost mid-operation.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    ...
    lock_lost_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(
        lock_key, token, stop_event, lock_lost_event
    )
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=...)
        if lock_lost_event.is_set():
            raise LockLostError(f"Lock expired mid-operation for key={lock_key}")
        # proceed with atomic release
```

## Medium: Input Validation Is Incomplete

The PR claims Pydantic input validation, but `reserve_item` accepts raw primitives with no validation at the entry point.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`item_id` is used directly in a Redis key (`inventory:lock:{item_id}`) with no sanitization. A caller passing `item_id = "foo:bar:baz"` or an empty string will silently produce a malformed or colliding lock key.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

## Low: Retry Backoff Does Not Respect Total Timeout Budget

The retry loop can block for up to `0.1 + 0.2 + 0.4 = 0.7s` regardless of system load or caller SLA requirements. For a flash sale under high contention this is both too short (lock TTL is 30s) and not configurable.

**Recommendation:** Accept a `timeout` parameter and use it as the retry budget ceiling rather than a fixed attempt count.

## Test Coverage Gaps

| Gap | Risk |
|---|---|
| Heartbeat is mocked out entirely in all tests | The critical race condition in `_beat()` has zero test coverage |
| No test for `lock_lost_event` signaling | Lock-loss-mid-operation path is untested |
| No test for partial DB write failure | Transaction rollback behavior is untested |
| `test_lock_not_released_if_expired` does not verify the lock was actually expired mid-operation | It only checks the release path, not the write-after-expiry scenario |

The heartbeat mock in the fixture is particularly concerning because it means the most important code path in this PR — the one that justifies the entire design — has never been executed by the test suite.

## Summary of Required Changes Before Approval

1. Replace heartbeat `GET` + `EXPIRE` with an atomic Lua script
2. Replace lock release `GET` + `DEL` with an atomic Lua script
3. Scope heartbeat state to the lock context, not the instance
4. Join the heartbeat thread before releasing the lock
5. Signal lock loss to the owning thread and abort the operation
6. Wrap `update_stock` and `create_order` in a database transaction
7. Validate and sanitize `item_id` before constructing Redis keys
8. Add tests that exercise the actual heartbeat thread behavior
