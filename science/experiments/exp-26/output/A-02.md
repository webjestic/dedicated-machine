Decision: Request Changes

Summary: The heartbeat design contains a race condition that directly violates the safety guarantee this PR claims to provide. Beyond that flaw, there are several additional architectural and correctness issues that must be resolved before this code is safe to merge into a high-concurrency flash sale path.

---

## Critical: Heartbeat Race Condition (TOCTOU)

This is the primary rejection reason. The heartbeat performs a non-atomic check-then-extend:

```python
# _beat() in _start_heartbeat
current = self.redis.get(lock_key)          # 1. read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # 2. extend TTL — NOT atomic with step 1
```

Between steps 1 and 2, the lock can expire and be acquired by another process. The heartbeat then extends the TTL of a lock it no longer owns, giving the current holder a false sense of safety while a second process is already inside the critical section. This is exactly the stale-write scenario the persona mandate prohibits.

**Required fix:** Replace the get/expire pair with a single atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The heartbeat must act on the return value: if `0` is returned, the lock was lost and the heartbeat must signal the main thread to abort the operation, not silently continue.

---

## Critical: Heartbeat Does Not Signal Lock Loss to Main Thread

Even if the atomic extension is fixed, the current heartbeat has no channel back to the calling thread. If the lock is lost mid-operation (e.g., Redis restart, network partition, GC pause that outlasts the TTL), `reserve_item` continues executing `update_stock` and `create_order` without a valid lock. A second process may be doing the same.

**Required fix:** Use a `threading.Event` or similar mechanism so the heartbeat can signal loss of lock ownership, and the main thread must check that signal before each DB mutation.

---

## Critical: Instance-Level Heartbeat State Is Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two concurrent calls to `reserve_item` hit the same `InventoryService` instance (which is the normal deployment pattern for a shared service object), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, causing the first heartbeat to stop prematurely and the first lock to expire unrenewed.

**Required fix:** Heartbeat state must be local to each lock acquisition, not stored on `self`. Pass state through closure variables or a dedicated lock-context object.

---

## Moderate: Lock Release Is Also Non-Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same TOCTOU pattern as the heartbeat. Between `GET` and `DELETE`, the lock can expire and be re-acquired by another process, which then gets its lock deleted by this thread.

**Required fix:** Use the standard atomic Lua release script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

---

## Moderate: No Input Validation on `reserve_item` Parameters

The PR checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar"` or an empty string produces a malformed or colliding key. `quantity` has no lower-bound check, so `quantity=0` or `quantity=-5` would pass the stock check and silently create a zero or negative reservation.

**Required fix:** Validate `item_id` is a non-empty alphanumeric/slug string and that `quantity >= 1` before entering the lock path. Pydantic is already a dependency; use it here.

---

## Moderate: DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not help here because the failure is in the database layer, not in lock contention.

**Required fix:** Both operations must execute within a single database transaction, or a compensating rollback must be implemented with explicit handling for partial failure.

---

## Minor: Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat is never exercised
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat, which is the central safety mechanism of this PR, is mocked out in the majority of tests. The race conditions described above would never surface in this test suite. `test_lock_not_released_if_expired` tests the release guard but does so against a mock that cannot reproduce real timing behavior.

**Required fix:** Add integration-level tests using a real or fakeredis-backed instance that exercise the heartbeat path, lock expiry during a simulated slow DB call, and concurrent acquisition attempts.

|---|---|

| Critical | No signal path from heartbeat to main thread on lock loss |
| Critical | Heartbeat state on `self` breaks concurrent use of same instance |
| Critical | Lock release is non-atomic (TOCTOU) |
| Moderate | No input validation on `item_id` or `quantity` |
| Moderate | DB operations not wrapped in a transaction |
| Minor | Tests mock out the heartbeat; race conditions are untestable as written |
