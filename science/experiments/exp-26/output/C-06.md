Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a government-grade production system.

---

## Critical Issues

### 1. Race Condition in Heartbeat Lock Extension

The heartbeat performs a non-atomic check-then-extend operation:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← window between get and expire
```

Between the `GET` and `EXPIRE` calls, the lock could expire and be acquired by another process. You would then extend a lock you no longer own. This must be an atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 2. Race Condition in Lock Release

The same TOCTOU flaw exists in `_acquire_lock`'s `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # ← another process could acquire between get and delete
```

This is a well-known pattern that **must** use a Lua script for atomic compare-and-delete. This is not optional — it is the canonical correct implementation:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 3. Instance-Level Heartbeat State Breaks Concurrency

`_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`. If a single service instance handles concurrent requests (e.g., in a thread pool or async framework), two simultaneous calls to `reserve_item` will clobber each other's heartbeat state. The heartbeat context must be scoped to the individual lock acquisition, not the service instance. Pass state through a local closure or a dedicated lock context object.

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` parameters with no validation. There is nothing preventing:

- `item_id = ""` or `item_id = "../../../../etc/passwd"` (relevant if item_id flows into file paths or shell commands elsewhere)
- `quantity = 0`, `quantity = -5`, or `quantity = 999999`

A zero or negative quantity would silently succeed and *increase* stock. Add a Pydantic input model or explicit guards:

```python
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

### 5. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises an exception, stock has already been decremented and the reservation is lost with no rollback. These must execute within a single database transaction. The Redis lock protects against concurrent access but does not protect against partial failure within the critical section.

## Moderate Issues

### 6. Heartbeat Stop Is Not Joined Before Lock Release

```python
self._stop_heartbeat()
current = self.redis.get(lock_key)  # heartbeat thread may still be running here
```

`_stop_heartbeat` sets a flag but does not join the thread. The heartbeat thread could execute one final `expire` call *after* the lock has been released and re-acquired by another process, silently extending a lock it no longer owns. Call `self._heartbeat_thread.join(timeout=...)` after setting the flag.

### 7. Retry Backoff Is Insufficient for Flash Sale Load

With `retry_attempts=3` and backoffs of `0.1s, 0.2s`, total wait time under contention is approximately 300ms. For a flash sale scenario with hundreds of concurrent requests, this will result in a cascade of `RuntimeError` failures and a flood of "service unavailable" responses. Consider:

- Increasing retry attempts (5–10)
- Adding jitter to prevent thundering herd: `backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)`
- Making `retry_attempts` a constructor parameter for tunability

### 8. Silent Exception Swallowing in `reserve_item`

Only `RuntimeError` is caught. Any other exception from `self.db.get_stock`, `self.db.update_stock`, or `self.db.create_order` (e.g., `ConnectionError`, `TimeoutError`, `OperationalError`) will propagate as an unhandled exception, bypassing the structured error response. Either broaden the catch or add a separate handler for database-layer exceptions with appropriate logging.

## Minor Issues

### 9. Test Coverage Gaps

- No test exercises the actual heartbeat thread behavior (it is mocked out entirely in the fixture, meaning the real `_start_heartbeat` code path is never exercised)
- No test covers the partial failure scenario (stock decremented, order creation fails)
- No test validates the behavior of concurrent calls on the same instance
- `test_lock_not_released_if_expired` does not assert the *result* of the reservation, only the lock behavior

### 10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Should Not Be Class Constants

These values are environment-dependent and should be injected via constructor parameters or configuration, not hardcoded as class-level constants. A slow database in staging vs. production will have very different requirements.

Items 1–5 are blockers. The architecture is sound in concept, but the implementation has correctness gaps that would cause data integrity failures under the exact conditions this service is designed to handle.
