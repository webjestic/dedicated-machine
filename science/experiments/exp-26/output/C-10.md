Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real flash sale load in a government-grade system.

---

## Critical Issues

### 1. Race Condition in Heartbeat Token Check (TOCTOU)

The heartbeat performs a `GET` then `EXPIRE` as two separate commands. Between those two operations, the lock can expire and be acquired by another process with a different token, which the heartbeat then extends — handing lock ownership to a thread that no longer holds it.

```python
# UNSAFE — two-step check/extend is not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

**Fix:** Use a Lua script to atomically check ownership and extend TTL in a single Redis round-trip.

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if result == 0:
                    logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
                    self._heartbeat_active = False
```

The same TOCTOU issue exists in `_acquire_lock`'s `finally` block and must receive the same treatment:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. `InventoryService` Is Not Concurrency-Safe for Multiple Callers

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (common in a web worker pool), they will stomp on each other's heartbeat state, causing premature lock releases or runaway heartbeat threads.

```python
# Both callers share this — will collide
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Move heartbeat state into the `_acquire_lock` context manager as local variables, or use a per-call `threading.Event` for cancellation.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        # atomic release via Lua script
```

### 3. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction boundary. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost with no compensating action.

```python
# Stock decremented here
self.db.update_stock(item_id, new_stock)
# If this throws, stock is gone and no order was created
self.db.create_order(order_id, item_id, quantity)
```

**Fix:** Wrap both operations in a database transaction. The exact API depends on your DB layer, but the intent must be explicit:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 4. Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` could be an empty string, a SQL injection fragment, or arbitrarily long. `quantity` could be zero or negative.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # No validation occurs here
```

**Fix:** Define a Pydantic input model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

### 5. Heartbeat Thread Leaks on Lock Acquisition Failure

If `_start_heartbeat` were ever called before the `if not acquired` guard (it isn't currently, but the structure is fragile), or if an exception is raised between `_start_heartbeat` and the `try` block, the thread runs forever. More concretely, after fixing issue #2 with a `stop_event`, ensure the event is always set in a `finally` block regardless of the code path.

---

### 6. Bare `RuntimeError` Is Too Broad

Catching `RuntimeError` in `reserve_item` will silently swallow unexpected runtime errors from the DB layer or Redis client that are not lock-related, masking real failures as "service temporarily unavailable."

```python
except RuntimeError as e:
    # This catches DB errors too
```

**Fix:** Define a specific exception class:

```python
class LockAcquisitionError(RuntimeError):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but document explicitly that heartbeat must fire at least twice before TTL expiry (i.e., `HEARTBEAT_INTERVAL < LOCK_TTL / 2`) and add an assertion or config validation to enforce this invariant.
- Consider whether `daemon=True` on the heartbeat thread is correct for your deployment. Daemon threads are killed abruptly on process exit, which is acceptable here since the lock will expire naturally, but it should be a conscious decision noted in a comment.
