Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check are all sound. However, there is a critical atomicity flaw that undermines the entire at-most-once guarantee: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that is followed by a database failure will result in a real charge with no recorded transaction, and the retry path will then attempt to charge again. Additionally, there are several reliability and correctness issues that need to be addressed before this can be approved.

---

## Critical Issues

### 1. Payment call inside DB transaction breaks atomicity guarantee

This is the most serious flaw. The PR summary explicitly claims:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is false. The current structure is:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)             # DB write
    self.db.mark_order_complete(...)       # DB write
```

If `payment.charge()` succeeds but the DB transaction rolls back (network blip, constraint violation, DB timeout), you have a real charge with no record. The next call will pass the idempotency check (`get_charge` returns `None`) and charge again. The DB transaction provides no rollback capability over an external HTTP call — wrapping it here creates a false sense of safety.

**Required fix:** Separate the payment call from the DB transaction. Charge first, then open the DB transaction to record. Handle the narrow window where the charge succeeds but recording fails as a distinct reconciliation problem (e.g., a separate recovery queue or idempotency key passed to the payment gateway).

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

You still need a recovery path for the window between a successful charge and a failed DB write, but at minimum this stops the retry path from double-charging.

---

### 2. Unhandled exceptions inside the lock context manager will silently swallow errors

The `process()` method has no `except` clause for exceptions raised inside `_acquire_lock`. If `self.payment.charge()` raises `PaymentError`, or if `self.db.record_charge()` raises a DB exception, the exception propagates up uncaught, the lock is released via `finally` (correct), but the caller receives an unhandled exception rather than a structured `OrderResult`. The public interface contract of returning `OrderResult` is broken.

```python
# Any exception inside the lock block bubbles out of process()
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)  # PaymentError raised here
    ...
    # caller gets an exception, not an OrderResult
```

**Required fix:** Add explicit exception handling for `PaymentError` and database exceptions inside the lock block, returning structured `OrderResult` failures.

---

### 3. `PaymentClient` retries on timeout are unsafe without idempotency keys

In `payment_client.py`, a `Timeout` exception is retried unconditionally:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have succeeded at the gateway. Retrying without an idempotency key passed to the payment provider can result in duplicate charges at the gateway level, independent of the Redis lock. This is a well-known failure mode in payment systems.

**Required fix:** Generate a per-call idempotency key (e.g., derived from `order_id`) and pass it as a request header (e.g., `Idempotency-Key`) to the payment gateway on every attempt in the retry loop. Do not generate a new key per attempt.

## Moderate Issues

### 4. Lock TTL is too short for the retry window it must cover

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105` seconds in the worst case. The Redis lock will expire before the payment operation completes, allowing a second worker to acquire the lock and proceed past the idempotency check while the first worker is still mid-flight.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client duration (e.g., 120–150 seconds with margin), or reduce the payment client's maximum total duration to fit within the lock TTL. Document the relationship explicitly.

### 5. `get_charge` is called outside the DB transaction

The idempotency check reads from the database before the transaction is opened:

```python
existing = self.db.get_charge(req.order_id)  # read outside transaction
if existing:
    ...
with self.db.transaction():                   # transaction opens here
    ...
```

Under read-committed isolation, a concurrent write could be committed between the read and the transaction open. The Redis lock should prevent two workers from reaching this point simultaneously for the same `order_id`, so this is partially mitigated — but it is a fragile dependency. If the lock ever fails to protect (e.g., Redis failover, clock skew on TTL), this becomes a race condition.

**Required fix:** Move `get_charge` inside the DB transaction, or add a unique constraint on `order_id` in the database schema as a hard backstop. A DB-level unique constraint is the correct defense-in-depth here regardless.

---

## Minor Issues

### 6. `payment_token` logged indirectly via `charge_id` — verify no token leakage in payment client

The `payment_token` field is passed to `payment_client.charge()` and serialized into the POST body. Confirm that the `requests` library or any middleware is not logging the full request body, which would expose payment tokens in log files. Consider masking or truncating `payment_token` in any log output.

### 7. Test for lock release verifies mock call but not token correctness

`test_lock_release_uses_lua_script` confirms the Lua script is called but does not assert that the token passed to the script matches the token set in Redis. The token-verification property — the main security guarantee of the lock release — is not tested.

### 8. No test for the payment failure path

There is no test covering `PaymentError` being raised during `payment.charge()`. Given that this is the most operationally sensitive path, it should be explicitly covered.
