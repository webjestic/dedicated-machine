Decision: Request Changes

Summary: The distributed locking strategy contains a fundamental atomicity flaw that undermines the at-most-once guarantee the PR is designed to provide. Several additional issues across the payment client, error handling, and test coverage require resolution before this is safe to merge into a production payment pipeline.

---

## Critical Issues

### 1. Payment Charge Occurs Inside the Database Transaction — Atomicity Guarantee is False

This is the most serious architectural flaw in the PR. The docstring claims:

> "Crash between charge and record cannot produce an unrecorded charge"

That claim is incorrect. The `payment.charge()` call is placed **inside** `self.db.transaction()`, but a payment gateway call is a **non-transactional external side effect**. Database transactions provide atomicity over database operations only. They have no rollback authority over a charge that has already been submitted to the payment gateway.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External call — NOT rolled back on DB failure
    self.db.record_charge(...)             # If this raises, the charge is already settled
    self.db.mark_order_complete(...)
```

**Failure scenario:** `payment.charge()` succeeds and money moves. `record_charge()` raises a database exception. The transaction rolls back. The charge is now unrecorded and the idempotency check will never find it. The customer has been charged with no record. This is a financial data integrity violation.

**Correct pattern:** Charge outside the transaction, then open the transaction only for the database writes. If the DB write fails after a successful charge, a reconciliation or recovery path is needed — not a rollback that silently discards the charge record.

```python
# Charge first, outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record atomically in the DB
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Even with this fix, a crash between the charge and the DB write requires a reconciliation mechanism. That is a known hard problem in distributed systems and should be documented explicitly.

### 2. Redis Lock Does Not Protect Against the Idempotency Check Race

The idempotency check reads from the database **inside** the lock, which is correct. However, the lock TTL is 60 seconds and the payment gateway timeout is 30 seconds with up to 3 retries and 5-second backoffs between them. The worst-case payment attempt duration is:

```
30s (attempt 1) + 5s (backoff) + 30s (attempt 2) + 5s (backoff) + 30s (attempt 3) = 100 seconds
```

This **exceeds** the `ORDER_LOCK_TTL` of 60 seconds. Redis will expire and release the lock while a worker is still actively processing. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. The lock TTL must be greater than the maximum possible processing time, or the payment client's retry budget must be reduced to fit within the TTL with margin.

---

### 3. Timeout Retry in `PaymentClient` Is Unsafe Without Idempotency Keys

On a gateway timeout, the client retries the charge. A timeout does not mean the charge failed — it means the response was not received. The gateway may have processed the charge successfully. Retrying without an idempotency key submitted to the gateway can result in double charges at the gateway level, independent of the Redis lock.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # Retries here without knowing if the previous attempt settled
```

The `charge()` call must include a stable idempotency key (e.g., derived from `order_id`) in the request headers or body, as supported by the payment gateway API, so that retries are deduplicated on the gateway side.

## Significant Issues

### 4. Unhandled Exceptions Leak the Lock Silently

The `_acquire_lock` context manager's `finally` block always releases the lock. However, if `payment.charge()` raises `PaymentError` or any other uncaught exception inside the `with self._acquire_lock(...)` block, the exception propagates up through `process()` with no handler. The lock is released correctly in this case, but the caller receives an unhandled exception rather than an `OrderResult`. This breaks the contract of `process()`, which is documented to always return an `OrderResult`.

```python
# PaymentError is never caught in process()
with self.db.transaction():
    charge_id = self.payment.charge(...)  # Can raise PaymentError
```

Add a `except (PaymentError, Exception)` handler in `process()` that returns a structured `OrderResult` on failure.

---

### 5. `payment_token` Is Logged at INFO Level via `charge_id` Correlation

The `payment_token` field is accepted as input and passed directly to the payment gateway. Confirm that no logging path emits the raw token. Currently the code does not log it directly, but the `OrderRequest` model should mark `payment_token` with a custom `__repr__` or be excluded from any future debug serialization. This is a PCI-DSS relevant concern and should be documented.

---

### 6. `customer_id` Has No Pattern Validation

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. If `customer_id` is used in database queries or log output, it should have equivalent sanitization.

---

## Test Coverage Gaps

### 7. No Test for Lock Expiry During Processing

There is no test simulating the case where the lock TTL expires mid-processing. This is the failure mode described in Issue 2 and should have a regression test once the TTL is corrected.

### 8. No Test for `PaymentError` Propagation

There is no test covering the case where `payment.charge()` raises `PaymentError`. Given Issue 4, this path currently causes `process()` to raise rather than return an `OrderResult`, and there is no test catching that.

### 9. `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

The test confirms the Lua script was called but does not verify the correct token was passed as `args`. A future regression where the wrong token is passed would not be caught.

---

## Minor Notes

- The `LOCK_RETRY_ATTEMPTS = 3` with `0.5s * (attempt + 1)` backoff means a maximum wait of 3 seconds before giving up. Under high contention this may be too aggressive for a 60-second lock, but this is a tuning concern rather than a correctness issue.
- `requests` is not pinned in any visible dependency file. Confirm it appears in `requirements.txt` or `pyproject.toml`.
- The internal endpoint `https://api.payments.internal/v1/charges` should be injected via configuration rather than hardcoded as a class attribute, to support environment-specific deployment without code changes.
