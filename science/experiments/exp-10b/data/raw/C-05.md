Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, which means a successful charge that is followed by a DB commit failure will produce an unrecorded, non-refunded charge — directly contradicting the comment claiming crash safety. There are also several secondary issues around error handling, observability, and test coverage gaps that need to be addressed before this touches production.

---

## Critical Issues

### 1. Payment call inside DB transaction breaks the atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious bug. The comment says *"crash between charge and record cannot produce an unrecorded charge"* — that claim is false. A DB transaction does not wrap the payment gateway call. If `charge()` succeeds but the DB commit fails (network partition, constraint violation, DB crash), you have charged the customer with no record of it. You also now hold an open DB connection/transaction for the full duration of an external HTTP call (up to 30s × 3 retries = 90s), which will exhaust your connection pool under any real load.

The correct pattern is charge-then-record with idempotent recovery:

```python
# Correct approach
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Payment is now committed externally. Record it durably.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # CRITICAL: charge succeeded but record failed.
    # Log with full context for manual reconciliation or async retry.
    logger.critical(
        "UNRECORDED_CHARGE order_id=%s charge_id=%s amount_cents=%d — "
        "manual reconciliation required",
        req.order_id, charge_id, req.amount_cents,
        exc_info=True,
    )
    raise
```

This doesn't make the problem disappear — distributed systems cannot achieve true atomicity across a payment gateway and a DB — but it makes the failure mode explicit, logged, and recoverable rather than silent.

### 2. `PaymentError` and DB exceptions escape `process()` uncaught

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # raises PaymentError — uncaught
        self.db.record_charge(...)            # raises arbitrary DB exception — uncaught
```

`PaymentError` and any DB exception propagate out of `process()` as unhandled exceptions. The lock *will* be released (the `finally` block is correct), but the caller gets a raw exception instead of an `OrderResult`. This breaks the stated contract of the method. At minimum these need to be caught and returned as structured results, and they need distinct log levels (a declined card is not the same severity as a DB outage).

```python
except PaymentError as e:
    logger.warning("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.error("Unexpected error processing order_id=%s", req.order_id, exc_info=True)
    return OrderResult(processed=False, message="Internal processing error")
```

## Significant Issues

### 3. Lock TTL is too short for the actual worst-case duration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `30s × 3 attempts + 5s × 2 sleeps = 100 seconds` in the worst case. If the lock expires while a worker is still in the retry loop, a second worker can acquire the lock, pass the idempotency check (no charge recorded yet), and issue a duplicate charge. The TTL must exceed the maximum possible processing time with margin.

```python
# PaymentClient worst case: 30s timeout × 3 attempts + 5s backoff × 2 gaps = 100s
# Add DB overhead and safety margin
ORDER_LOCK_TTL = 120  # minimum; consider 180s
```

---

### 4. `payment_token` logged at INFO level is a latent PCI/security risk

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is present in the `OrderRequest` object and in local scope throughout `process()`. One careless future log line (e.g., logging the full `req` object during debugging) exposes a live payment token. The token should be masked at the model level:

```python
class OrderRequest(BaseModel):
    ...
    payment_token: str = Field(..., min_length=1, max_length=256)

    def __repr__(self) -> str:
        return (
            f"OrderRequest(order_id={self.order_id!r}, "
            f"customer_id={self.customer_id!r}, "
            f"amount_cents={self.amount_cents!r}, "
            f"payment_token='***')"
        )
```

---

### 5. `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` accepts arbitrary strings up to 64 characters. If `customer_id` is used in DB queries or log aggregation downstream, this is an injection surface. Apply the same or equivalent pattern constraint.

---

### 6. `PaymentClient` has no authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no `Authorization` header, API key, or mTLS configuration. Even for an internal endpoint this is a significant gap — internal network compromise or SSRF would allow unauthenticated charges. At minimum there should be a hook for injecting credentials, even if the implementation is deferred.

---

## Minor Issues

### 7. `response.json()["charge_id"]` will raise `KeyError` on malformed gateway response

```python
charge_id = response.json()["charge_id"]
```

A `200 OK` with an unexpected body (e.g., `{}`, HTML error page with wrong `Content-Type`) raises an unhandled `KeyError` or `JSONDecodeError`. Wrap this:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

---

### 8. Test coverage gaps

The tests cover the happy path and basic contention, but are missing:

- **`PaymentError` raised during charge** — verify `process()` returns a structured result, not a raw exception
- **DB transaction failure after successful charge** — the most critical failure mode given issue #1
- **Lock TTL expiry during processing** — simulate lock expiring mid-flight
- **`PaymentClient` retry behavior** — verify it retries on 5xx but not on 4xx
- **Malformed gateway response** — `KeyError` on missing `charge_id`

The existing `test_lock_release_uses_lua_script` test asserts `redis_client.delete.assert_not_called()` which is a reasonable negative assertion, but it does not verify the token passed to the Lua script matches the one used during acquisition — the most important property of the token-verified release.
