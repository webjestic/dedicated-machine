Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction, plus several secondary issues in error handling, observability, and test coverage that need to be addressed before this is safe to ship to a production environment handling financial transactions.

---

## Critical Issues

### 1. Payment charge occurs inside the DB transaction — crash safety is broken

This is the most serious flaw. The PR comment claims:

> *"Crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**. The payment gateway call happens *inside* `self.db.transaction()`:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Problems:**

- If the process crashes after `payment.charge()` succeeds but before `record_charge()` commits, the customer is charged with no record. The DB transaction rollback does nothing — the charge already happened externally.
- Holding a DB transaction open across a 30-second (×3 retry) network call to the payment gateway will exhaust your DB connection pool under any real load.
- This is a classic distributed systems mistake: you cannot make an external side effect atomic with a local transaction.

**Required fix:** Charge *outside and before* the transaction. Record the charge ID *inside* the transaction. Use the idempotency check (which is already there) as your recovery path. The correct sequence is:

```python
# Outside the DB transaction
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# Inside the DB transaction — purely local writes
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

If the process crashes between the charge and the record, the Redis lock will expire, a retry will re-enter, the idempotency check will find no record, and you'll attempt to charge again. You need a compensating mechanism for this window — at minimum a `pending_charge` record written *before* the gateway call, inside a transaction, so a recovery path can detect and reconcile orphaned charges. This is a known hard problem and the current code does not solve it despite claiming to.

### 2. `PaymentError` is silently swallowed — charge failures are invisible

In `order_processor.py`, `PaymentError` is never caught. If `self.payment.charge()` raises, the exception propagates out of the `with self._acquire_lock(...)` block. The `finally` clause correctly releases the lock, but the exception then propagates to the caller with no structured `OrderResult` returned. The caller gets an unhandled exception rather than a clean failure response.

```python
# This is missing:
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

There is also no test covering payment failure, which is why this was missed.

## Significant Issues

### 3. Lock TTL is too short relative to retry window

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can spend up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds` on retries alone, before any DB time. The lock will expire mid-processing, allowing a second worker to acquire it and potentially double-charge. Either increase the TTL to cover the realistic worst case (≥ 120s with margin), or reduce the payment client's retry budget to fit within it. Document the relationship explicitly.

### 4. `payment_token` logged or exposed in `PaymentError` messages

`PaymentClient` logs `status_code` values which is fine, but `PaymentError` messages include raw gateway status codes that may propagate into caller logs. More importantly, `payment_token` flows through as a plain string with no masking anywhere in the logging path. If any logger is configured at DEBUG level and logs request bodies, tokens will appear in log files. Add a `__repr__`/`__str__` mask on `OrderRequest.payment_token` or explicitly redact it before logging.

### 5. `requests.post` has no connection timeout, only a read timeout

```python
timeout=self.REQUEST_TIMEOUT  # This is a single value — applies to read, not connect
```

Per the `requests` documentation, passing a single integer sets the *read* timeout only. A hung TCP connection during the connect phase will block indefinitely. Use a tuple:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

### 6. Response JSON is accessed without validation

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected body, this raises a `KeyError` or `JSONDecodeError` that is not caught and will not be retried. Wrap this in a try/except and treat it as a transient error or raise a clear `PaymentError`.

## Minor Issues

### 7. Test coverage gaps

The tests do not cover:
- `PaymentError` raised by the payment client (the silent swallow bug above would be caught here)
- Lock TTL expiry during processing
- DB transaction rollback behavior
- `requests.exceptions.ConnectionError` in `PaymentClient` (currently unhandled — will propagate as an uncaught exception, not retried)

### 8. `ConnectionError` not retried in `PaymentClient`

`requests.exceptions.ConnectionError` (DNS failure, refused connection) is not caught in the retry loop. It will propagate immediately as an unhandled exception. Add it alongside `Timeout` in the retry logic.

### 9. Lock retry backoff is linear, not exponential, and very short

`0.5s, 1.0s, 1.5s` — this is linear and the total wait is only 2 seconds across 3 attempts. Under genuine contention from a slow worker holding the lock for up to 60 seconds, these retries will all fail immediately and the caller gets a rejection. Consider whether the retry is providing real value here or whether you should just fail fast and let the caller retry at a higher level with proper exponential backoff.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct — no SETNX/EXPIRE race condition.
- Lua script for token-verified release is the right approach.
- Pydantic validation with pattern enforcement on `order_id` is solid.
- The idempotency check placement (inside the lock) is correct.
- Structured logging with consistent field names is good practice.
- The `register_script` pre-compilation is a nice touch.
