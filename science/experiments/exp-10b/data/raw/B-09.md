Decision: Request Changes

Summary: The architecture is sound in its intent — token-verified Lua release, `SET NX EX`, Pydantic validation, and idempotency checks are all correct patterns. However, there are several correctness bugs and one significant safety flaw that must be resolved before this touches production. The issues range from a race-prone timeout interaction that can produce double charges, to a silent exception swallow that will mask payment failures in the database.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — timeout race can produce an unrecorded charge

**File:** `services/order_processor.py`, lines inside `with self.db.transaction()`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external side effect inside tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims this is atomic and crash-safe, but it is not. `self.payment.charge()` is a network call to an external system. If the charge succeeds but the database transaction subsequently fails to commit (network partition, DB crash, deadlock), the charge is real money out of the customer's account with no record in your system. You cannot roll back an external payment gateway call by rolling back a local DB transaction — those are two entirely separate systems with no two-phase commit coordinator between them.

The correct pattern is:

1. Charge the payment gateway **outside** the transaction and capture the `charge_id`.
2. Open the DB transaction.
3. Re-check idempotency inside the transaction (the lock prevents races, but defense in depth matters).
4. Record the charge and mark complete.
5. Commit.

If step 4/5 fails after a successful charge, you need a reconciliation / dead-letter mechanism (e.g., write the raw `charge_id` to a recovery queue before the DB transaction) so the charge can be recorded on retry rather than silently lost.

### 2. `PaymentError` is not caught in `process()` — it propagates as an unhandled exception

**File:** `services/order_processor.py`, `process()` method

`LockError` is caught and converted to a structured `OrderResult`. `PaymentError` is not. If the payment gateway returns a 4xx or exhausts retries, the exception will propagate out of `process()` entirely, bypassing the `OrderResult` contract the caller expects. Any caller not wrapping this in its own try/except will crash. This is especially dangerous because the lock **will be released** in the `finally` block, meaning a second worker could immediately retry and potentially double-charge if the first charge actually succeeded but raised a timeout exception.

```python
# Missing:
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

---

### 3. Timeout on payment call can produce a charge with no `charge_id` — retry then double-charges

**File:** `services/payment_client.py`, `charge()` method

A `Timeout` exception means the request may or may not have been processed by the gateway. The code retries unconditionally on timeout. If the first attempt succeeded but the response was slow, the retry will issue a second charge. The payment client must send an idempotency key (a stable, per-order UUID) in the request headers so the gateway can deduplicate:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # ← required
    timeout=self.REQUEST_TIMEOUT,
)
```

The `idempotency_key` should be derived from `order_id` (e.g., `str(uuid.uuid5(NAMESPACE, order_id))`) and passed in from `OrderProcessor` so it is stable across retries.

## Moderate Issues

### 4. Lock TTL of 60 seconds does not account for retry duration

**File:** `services/order_processor.py`, `ORDER_LOCK_TTL = 60`

`PaymentClient` can take up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 × (30 + 5)` = **105 seconds** in the worst case. The lock expires after 60 seconds, allowing a second worker to acquire it and enter the idempotency check before the first worker has finished. The TTL must be set to at least the maximum possible processing time, or the lock must be extended (via a background heartbeat / `PEXPIRE` refresh) for long-running operations.

---

### 5. `_acquire_lock` retry backoff is too aggressive and blocks the event loop

**File:** `services/order_processor.py`, `_acquire_lock()`

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — linear, not exponential
time.sleep(backoff)
```

Two problems: (a) linear backoff does not reduce thundering herd under high contention — use exponential with jitter; (b) `time.sleep()` is a blocking call that will stall any async or threaded worker. If this ever runs in an async context, use `asyncio.sleep`. Even in a threaded context, consider whether holding a thread for up to 1.5 seconds per order is acceptable under load.

---

### 6. `response.json()["charge_id"]` will raise `KeyError` on unexpected gateway response shape

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises an unhandled `KeyError` or `JSONDecodeError`. Wrap this in explicit error handling:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Unexpected gateway response: {response.text[:200]}") from e
```

## Minor Issues

### 7. Tests do not cover the `PaymentError` propagation path

**File:** `tests/test_order_processor.py`

There is no test asserting behavior when `payment.charge` raises `PaymentError`. Given issue #2 above, this is a gap that would have caught the missing exception handler. Add:

```python
def test_payment_failure_returns_structured_result(processor, payment):
    from services.payment_client import PaymentError
    payment.charge.side_effect = PaymentError("declined")
    result = processor.process("order-001", "cust-001", 5000, "tok_visa")
    assert result.processed is False
    assert "Payment failed" in result.message
```

### 8. `customer_id` field has no pattern constraint

**File:** `services/order_processor.py`, `OrderRequest`

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. If `customer_id` is used in any downstream query or log aggregation, apply the same or equivalent pattern constraint.

---

## What Is Done Well

- `SET NX EX` is the correct atomic acquisition primitive — no SETNX + EXPIRE race.
- The Lua release script correctly prevents a worker from releasing a lock it does not hold.
- Pydantic validation with pattern enforcement on `order_id` is appropriate.
- Structured logging with consistent field names is good practice.
- The idempotency check before charging is the right guard — it just needs to survive the transaction restructuring noted in issue #1.
