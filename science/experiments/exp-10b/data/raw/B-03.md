Decision: Request Changes

Summary: The architecture contains a critical atomicity flaw: the payment charge occurs *inside* the database transaction, meaning a successful charge at the gateway can be committed to the database while the Redis lock is still held — but more dangerously, a database rollback after a successful charge leaves an unrecorded, unrecoverable charge with no compensation path. Additionally, the retry logic in the payment client can produce duplicate charges on timeout, since a timed-out request may have already been processed by the gateway. A secondary concern is that unhandled exceptions inside `process()` (e.g., database errors, `PaymentError`) will propagate out of the `_acquire_lock` context manager uncaught, which is correct for lock release but means the caller receives an unhandled exception rather than a structured `OrderResult`, breaking the contract the method signature implies.

---

## Detailed Findings

### 1. Critical — Charge-Before-Record Atomicity Is Not Actually Atomic

The PR description claims:

> *"Crash between charge and record cannot produce an unrecorded charge"*

This is false. The sequence inside `db.transaction()` is:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External network call — NOT part of the DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over database operations only. `self.payment.charge()` is an external HTTP call to a payment gateway. If the gateway succeeds but the subsequent `record_charge()` raises (deadlock, constraint violation, connection drop), the DB transaction rolls back and the charge is permanently lost — money has moved, but there is no record of it. There is no compensation, no reconciliation hook, and no alerting path for this case.

**The correct pattern for this class of problem is the outbox/saga pattern:**

1. Within a single DB transaction, record a `PENDING` charge intent.
2. Commit that intent.
3. Call the payment gateway.
4. On success, update the record to `COMPLETED` (idempotently, using the gateway's `charge_id`).
5. On failure, mark it `FAILED` and surface for reconciliation.

This separates the durable record of intent from the external side effect, and makes every state transition recoverable.

---

### 2. Critical — Payment Retry Logic Will Produce Duplicate Charges on Timeout

In `payment_client.py`:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# falls through to retry
```

A timeout means the request may have completed at the gateway — the client simply did not receive the response in time. Retrying unconditionally on timeout submits a second charge request with the same token and amount. If the gateway does not enforce idempotency on its end (and this code does not send an idempotency key), this **will** double-charge customers.

**Fix:** Generate a stable idempotency key (e.g., derived from `order_id`) and pass it as a header on every attempt:

```python
headers = {"Idempotency-Key": f"order-{order_id}"}
response = requests.post(..., headers=headers)
```

This requires threading `order_id` through to `PaymentClient.charge()`, which is a small interface change but a necessary one.

### 3. Moderate — Unhandled Exceptions Break the `OrderResult` Contract

`process()` has a return type of `OrderResult`, but several exception types are not caught and will propagate to the caller:

- `PaymentError` — raised by the payment client on permanent failure or exhausted retries
- Any database exception from `get_charge()`, `record_charge()`, `mark_order_complete()`
- `requests.exceptions.ConnectionError` from the payment client (not caught at all in `payment_client.py`)

The lock *will* be released correctly (the `finally` block in `_acquire_lock` handles that), but the caller receives a raw exception instead of a structured result. Either the method signature should be `-> OrderResult | raises PaymentError` and documented accordingly, or all expected failure modes should be caught and returned as structured results. Mixing the two is the current state and it is inconsistent.

---

### 4. Moderate — Redis Unavailability Is a Silent Hard Failure

If `self.redis.set()` raises (Redis is down, network partition), the exception propagates uncaught through `_acquire_lock` and then through `process()`. There is no structured handling, no circuit breaker, and no fallback. For a payment processing path, this should be an explicit, observable failure mode — not an unhandled exception that surfaces as a 500 to the caller with no context.

---

### 5. Minor — Lock TTL Is Insufficient Given Retry Timing

`ORDER_LOCK_TTL = 60` seconds, but the payment client can take up to:

```
MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF
= 3 × 30 + 2 × 5 = 100 seconds
```

The lock will expire before the payment client finishes its retry cycle. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and submit a duplicate charge. The TTL must be set to at least the maximum possible payment client duration, with margin. Either increase `ORDER_LOCK_TTL` to ≥120 seconds or reduce the payment client's maximum duration.

---

### 6. Minor — Test Coverage Gaps

The tests do not cover:

- `PaymentError` being raised inside the lock — verifies the lock is still released
- Database exception after a successful charge — the most dangerous failure mode identified above
- Redis raising on `set()` — verifies the caller gets a structured error
- Timeout retry producing a duplicate charge (this would expose finding #2 as a failing test)

The existing tests are well-structured and the fixture design is clean, but they test the happy path and one contention case. The failure modes that matter most for a payment processor are not exercised.
