## Detailed Findings

### ЁЯФ┤ Critical тАФ Charge Outside Database Transaction Atomicity Boundary

This is the most dangerous flaw in the entire PR and it directly contradicts the safety claim made in the docstring.

```python
# Charge and record atomically тАФ crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(          # тЖР EXTERNAL SIDE EFFECT
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `self.payment.charge(...)` call is a **network call to an external payment gateway**. It is not a participant in `self.db.transaction()`. Wrapping it inside a `with self.db.transaction()` block does **not** make it atomic with the database writes. A database transaction provides atomicity over database operations only.

**The actual failure scenario:**

1. `payment.charge()` succeeds тАФ money is captured from the customer.
2. `db.record_charge()` or `db.mark_order_complete()` raises an exception.
3. The database transaction rolls back.
4. The order now has **no recorded charge** in the database.
5. The Redis lock is released.
6. A subsequent retry sees no existing charge, passes the idempotency check, and **charges the customer a second time**.

This is the exact double-charge scenario the system was designed to prevent. The docstring comment `"crash between charge and record cannot produce an unrecorded charge"` is factually incorrect and dangerously misleading. This is a textbook **dual-write problem** and it is not solved here.

**Required remediation:** The payment charge must happen *outside and before* the database transaction. The correct pattern is:

1. Acquire lock.
2. Idempotency check.
3. Call `payment.charge()` тАФ capture the `charge_id`.
4. *Then* open `db.transaction()` and record the charge and mark complete.
5. If step 4 fails, you have an unrecorded charge тАФ this must be handled by a reconciliation/recovery path, not by pretending the database transaction covers the external call.

Alternatively, implement a two-phase approach: persist a `PENDING` charge record *before* calling the payment gateway, then update it to `COMPLETE` after. This gives you a recovery anchor.

### ЁЯФ┤ Critical тАФ Timeout Duration Exceeds Lock TTL, Enabling Race Window

```python
ORDER_LOCK_TTL = 60        # seconds тАФ covers typical payment round-trip
```

```python
REQUEST_TIMEOUT = 30      # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5         # seconds between retries on transient failure
```

Maximum possible `PaymentClient.charge()` wall-clock time:

```
3 attempts ├Ч 30s timeout + 2 inter-retry sleeps ├Ч 5s = 90s + 10s = 100 seconds
```

The lock TTL is **60 seconds**. The payment operation can take up to **100 seconds**. This means the Redis lock can expire and be acquired by a second worker while the first worker is still mid-charge. The idempotency check in the second worker will find no recorded charge (because the first worker hasn't committed yet), and both workers will proceed to charge.

This is a genuine race condition that can produce duplicate charges in production under degraded gateway conditions тАФ exactly the scenario most likely to cause retries.

**Required remediation:** Either:
- Reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` so the total maximum payment duration is well under `ORDER_LOCK_TTL` with margin, or
- Implement lock extension (refresh the TTL mid-operation via a heartbeat thread), or
- Significantly increase `ORDER_LOCK_TTL` to cover worst-case payment duration plus margin (e.g., 120тАУ180s).

### ЁЯФ┤ Critical тАФ `PaymentError` Not Caught in `process()`, Unhandled Exception Leaks

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

`PaymentClient.charge()` raises `PaymentError` on permanent failure or after exhausting retries. The `process()` method catches `LockError` but has **no handler for `PaymentError`** or any other exception from the payment or database layer. An unhandled `PaymentError` will propagate up the call stack as an uncaught exception.

Depending on the caller, this could:
- Crash a worker process.
- Leave the Redis lock unreleased if the exception somehow bypasses the `finally` block (it won't in this case due to `contextmanager`, but the lock *will* be released while the exception propagates тАФ the caller gets no structured `OrderResult`).
- Expose internal error details to upstream systems.

The `process()` method's return type is `OrderResult`. It should never raise. All domain exceptions must be caught and converted to a structured result.

**Required remediation:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### ЁЯЯб Major тАФ Idempotency Check Is Not Recheck-Safe After Lock Expiry Race

This is a consequence of the TTL/timeout race described above, but worth stating independently. The idempotency check:

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(...)
```

This check is only valid as long as the lock is held. If the lock has already expired by the time this check runs (possible under high latency), two workers can both pass this check simultaneously. The database transaction does not include a `SELECT FOR UPDATE` or equivalent pessimistic lock on the order row, so there is no database-level guard preventing concurrent inserts.

**Required remediation:** The database idempotency check should use a unique constraint on `order_id` in the charges table, so that a concurrent `record_charge` will fail with a constraint violation rather than silently inserting a duplicate. This is defense-in-depth that does not rely solely on the distributed lock.

### ЁЯЯб Major тАФ `payment_token` Logged Indirectly via Pydantic `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic `ValidationError` string representations include field values in their output. If `payment_token` fails validation (e.g., it is too long or malformed), the raw token value may appear in the log output. Payment tokens are sensitive credentials. Logging them violates PCI-DSS requirements and represents a credential exposure risk.

**Required remediation:** Log only the field names and error types, never the values, for any field that could contain payment credentials:

```python
logger.warning(
    "Invalid order request order_id=%s fields=%s",
    order_id,
    [e["loc"] for e in exc.errors()]
)
```

### ЁЯЯб Major тАФ Retry Backoff in `_acquire_lock` Uses Blocking `time.sleep`

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

This is a synchronous blocking sleep in what is presumably a worker thread or async context. In a high-concurrency deployment this ties up a thread for up to 1.5 seconds total across retries per contended order. Under load this can exhaust thread pools. Additionally, the backoff is linear with no jitter, which means all workers contending on the same lock will retry in synchronized waves тАФ a textbook thundering herd.

**Required remediation:** Add jitter (`random.uniform(0, backoff)`) and document the threading model. If this runs in an async framework, use `asyncio.sleep`.

### ЁЯЯб Major тАФ `requests` Used Without Session, No Connection Pooling, No Auth

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

- `requests.post()` at the module level creates a new TCP connection on every call. Under load this is inefficient and can exhaust ephemeral ports.
- There is no authentication header sent to the payment gateway. An internal payment API accepting unauthenticated POSTs is a significant security concern. Even on an internal network, this should use mTLS or a bearer token.
- The endpoint is hardcoded as a class attribute string. This should be injected via configuration.

### ЁЯЯа Moderate тАФ Lock Retry Count of 3 Is Likely Insufficient for Production Contention

```python
LOCK_RETRY_ATTEMPTS = 3
```

With 3 attempts and 0.5s/1.0s/1.5s cumulative wait, the system gives up after ~1.5 seconds of contention. The lock TTL is 60 seconds. If a worker is legitimately processing an order, any concurrent worker will fail after 1.5 seconds and return `"Order currently being processed by another worker"` to the caller. Depending on the upstream retry strategy, this could result in the caller retrying and eventually succeeding, or it could result in a lost order if the caller treats this as a permanent failure.

This is a design decision that needs explicit documentation and alignment with the upstream SLA.

---

### ЁЯЯа Moderate тАФ No Observability on Lock Expiry or Stale Lock Detection

There is no mechanism to detect or alert when a lock expires before the operation completes. Silent lock expiry is one of the hardest failure modes to diagnose in distributed systems. At minimum, the lock acquisition timestamp should be recorded and compared against the TTL at release time, with a warning logged if the operation approached or exceeded the TTL boundary.
