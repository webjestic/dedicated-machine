Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly structured. However, there are several correctness and architectural issues that must be resolved before this can be approved, the most critical being a race condition that breaks the at-most-once guarantee the PR explicitly claims to provide, and a payment retry loop that can produce duplicate charges.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — timeout retry creates duplicate charges

This is the most dangerous bug in the PR. The `PaymentClient.charge()` call is made *inside* `self.db.transaction()`. On a gateway timeout, `PaymentClient` retries up to `MAX_RETRIES` times. But a gateway timeout does **not** mean the charge failed — the payment may have been processed and the response lost in transit. Retrying blindly will issue duplicate charges against the customer's payment method.

```python
# services/order_processor.py — current, broken
with self.db.transaction():
    charge_id = self.payment.charge(   # ← retries on timeout; charge may already exist
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
```

The payment gateway call must be moved **outside** the database transaction. The correct pattern is:

1. Call the payment gateway (outside any DB transaction).
2. On success, open a DB transaction and record the result.
3. On timeout/ambiguous failure, query the gateway for the charge status by idempotency key before retrying.

The `PaymentClient` must also accept and forward a caller-supplied idempotency key (e.g., `order_id`) so the gateway can deduplicate on its side.

```python
# Sketch of corrected flow
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway deduplicates
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

### 2. Idempotency check is outside the database transaction — TOCTOU race

The `db.get_charge()` check happens **before** `db.transaction()` opens. Two workers can both pass the check simultaneously, both proceed to charge, and both record — defeating the entire purpose of the lock. The Redis lock *should* prevent this, but the lock is a best-effort coordination mechanism, not a database-level guarantee. If Redis is unavailable or the lock TTL expires during a slow payment call, the race is open.

The idempotency check must be performed **inside** the same database transaction as the write, ideally as a conditional insert or a `SELECT FOR UPDATE` on the order row. The database's own ACID guarantees are the correct backstop here, not Redis alone.

```python
with self.db.transaction():
    existing = self.db.get_charge_for_update(req.order_id)  # SELECT FOR UPDATE
    if existing:
        return OrderResult(processed=False, message="Order already processed")
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

---

### 3. Lock TTL is shorter than the maximum payment round-trip time

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `REQUEST_TIMEOUT × MAX_RETRIES + RETRY_BACKOFF × (MAX_RETRIES - 1)` = `30 × 3 + 5 × 2` = **100 seconds** in the worst case. The lock expires before the payment completes, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge.

Either:
- Increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client runtime (≥ 120 seconds with margin), or
- Implement lock extension (a background thread that refreshes the TTL while the payment is in flight), or
- Reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` so the total payment budget fits within the TTL.

All three values must be documented together so future changes to one don't silently invalidate the others.

## Moderate Issues

### 4. `payment_token` is logged at INFO level on payment success

```python
# services/payment_client.py
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

The token itself is not logged here, but `payment_token` is present in the `OrderRequest` model and flows through the call stack. Verify that no logging call anywhere serializes the full request object. More importantly, `payment_token` should be marked as a sensitive field and explicitly excluded from any `__repr__`, `__str__`, or model serialization. Pydantic v2 supports `Field(..., repr=False)` for this purpose. A PCI-scope finding here would be serious.

### 5. `requests.exceptions.ConnectionError` is not caught in `PaymentClient`

Network-level failures (DNS resolution failure, connection refused, TCP reset) raise `requests.exceptions.ConnectionError`, which is not a subclass of `HTTPError` or `Timeout`. These will propagate as unhandled exceptions, bypassing the retry logic entirely and surfacing as a 500-class error to the caller with no retry attempt.

```python
except requests.exceptions.ConnectionError:
    last_error = "connection error"
    logger.warning("Payment connection error attempt=%d", attempt + 1)
```

### 6. Lock release swallows exceptions silently

The `finally` block in `_acquire_lock` calls `self._release_script(...)` with no exception handling. If the Lua script call raises (Redis connection lost, script evicted), the exception will propagate and mask the original exception from inside the `with` block. Wrap the release in a `try/except` that logs and continues.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
    except Exception:
        logger.exception("Failed to release order lock order_id=%s — lock will expire naturally", order_id)
```

## Minor Issues

### 7. Test for lock release does not assert the correct key or token

`test_lock_release_uses_lua_script` asserts only that `_release_script` was called once. It does not assert that the correct `lock_key` and `token` were passed. A regression that passes the wrong key would not be caught.

### 8. No test for lock TTL expiry during payment

There is no test covering the scenario where the lock expires mid-payment and a second worker acquires it. This is the exact failure mode described in issue #3 and should be a documented test case even if the fix is architectural.

### 9. `CHARGE_ENDPOINT` is hardcoded

`"https://api.payments.internal/v1/charges"` should be injected via configuration or environment variable. Hardcoded internal endpoint URLs are a deployment and testability problem.
