# Pull Request Review

**PR Title:** Add distributed order processor with Redis-backed exclusive locking
**Author:** dev-team
**Branch:** feature/order-processor

---

## Detailed Analysis

### Critical Flaw — The Charge/Record "Atomic Transaction" Is Not Atomic

This is the most dangerous issue in the entire PR and it directly undermines the core safety guarantee being advertised.

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← external network call
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR comment states:

> *"crash between charge and record cannot produce an unrecorded charge"*

This is **false**. A database transaction provides atomicity over database operations only. `self.payment.charge(...)` is an external HTTP call to a third-party payment gateway. It is not enrolled in the database transaction. If the process crashes, the network drops, or the DB write fails after `charge()` returns a `charge_id`, the customer has been charged and no record exists. The database transaction wrapper gives a false sense of safety here and is arguably more dangerous than no wrapper at all, because it misleads the reader into believing a guarantee that does not exist.

The correct pattern for this class of problem is a **two-phase approach**:

1. Write a `PENDING` charge record to the database *before* calling the payment gateway, using the `order_id` as an idempotency key.
2. Call the payment gateway.
3. Update the record to `COMPLETED` with the returned `charge_id`.
4. On recovery, inspect `PENDING` records and reconcile against the gateway using its idempotency key support.

Without this, you have an unrecoverable phantom charge scenario under any crash or network partition between the gateway call and the DB write.

### Significant Flaw — Timeout Overlap Creates a Double-Charge Window

```python
ORDER_LOCK_TTL = 60  # seconds
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5  # seconds between retries
```

Worst-case payment client execution time:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The Redis lock TTL is 60 seconds. The payment client can legally run for 100 seconds. If the lock expires at second 61, a second worker acquires the lock, passes the idempotency check (no charge recorded yet because the first worker is still waiting on a retry), and issues a second charge. The idempotency check is only meaningful if the lock is held for the entire duration of the operation. These two constants are in separate files with no enforced relationship, and there is no documentation or assertion binding them together.

At minimum this requires:

```python
assert ORDER_LOCK_TTL > (MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF)
```

Ideally the `OrderProcessor` should own or be injected with the `PaymentClient` configuration so the TTL can be derived programmatically rather than maintained manually across two files.

### Significant Flaw — Timeout Retry on Payment Gateway Is Unsafe Without Idempotency Keys

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout does not mean the charge failed. The gateway may have received and processed the request; the response simply did not arrive in time. Retrying a timed-out charge request without supplying a stable idempotency key to the gateway is a direct path to duplicate charges. This is a well-known failure mode in payment systems.

The fix is to generate a per-attempt or per-order idempotency key and pass it as a header (`Idempotency-Key`) on every attempt for the same logical charge. Most payment gateways (Stripe, Braintree, etc.) support this. Without it, retry-on-timeout is unsafe by design.

### Moderate Flaw — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in the log output. Payment tokens are sensitive credentials. They must be explicitly excluded from log output. The validation error should be sanitized or only field names should be logged, not values.

### Moderate Flaw — Lock Release Silently Swallows Failure

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

The return value of the Lua script is `0` when the token does not match (i.e., the lock expired and was acquired by another worker). This case is silently ignored. The log message unconditionally says "lock released" even when the lock was not held at release time. This is a correctness and observability problem. A return value of `0` should be logged as a warning indicating lock expiry during processing, which is a signal that the TTL is too short and a potential double-processing event may have occurred.

```python
finally:
    released = self._release_script(keys=[lock_key], args=[token])
    if released:
        logger.info("Order lock released order_id=%s", order_id)
    else:
        logger.error(
            "Lock expired before release — possible concurrent processing order_id=%s",
            order_id,
        )
```

### Moderate Flaw — Unhandled Exceptions Leak Stack Traces Without Structured Context

Neither `process()` nor `_acquire_lock()` catches general exceptions. If `self.db.get_charge()`, `self.db.transaction()`, `self.db.record_charge()`, or `self.payment.charge()` raises an unexpected exception, it propagates uncaught to the caller. In a worker deployment this likely means the lock is released (correctly, via `finally`), but the caller receives a raw exception with no structured `OrderResult`. The public contract of `process()` implies it returns an `OrderResult` in all cases. Either the docstring must be corrected or a top-level catch must be added.

---

### Minor Flaw — Test for Lua Script Release Does Not Verify Token Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test only verifies that the script was called and that `delete` was not called directly. It does not verify that the token passed to the script matches the token used to acquire the lock. The entire security property of the token-verified release is untested. A test should capture the token at acquisition time and assert it is the same token passed to the release script.

---

### Minor — `customer_id` Has No Pattern Constraint

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has a strict alphanumeric pattern. `customer_id` does not. If `customer_id` is used in any downstream query, log interpolation, or external API call, this is an inconsistency that could introduce injection surface. Apply the same or equivalent pattern constraint.

---

### Minor — Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs in source are an operational and security concern. This should be injected via configuration or environment variable, especially in a government contracting context where environments (dev, staging, production, classified enclave) differ and accidental cross-environment charges are a real risk.

**Decision:** Request Changes

**Summary:** This PR contains two independent critical flaws that each independently produce a viable double-charge scenario: the payment call is placed inside a database transaction context that provides no protection over external network calls, and the Redis lock TTL is shorter than the maximum possible payment client execution time. Additionally, retrying on gateway timeouts without idempotency keys is an unsafe pattern for any payment system. None of these are edge cases — they are reachable under normal transient failure conditions in a concurrent worker deployment. The PR must not be merged until the charge persistence model is redesigned around a pre-write/reconcile pattern, the TTL and timeout bounds are formally constrained, and gateway idempotency keys are implemented.
