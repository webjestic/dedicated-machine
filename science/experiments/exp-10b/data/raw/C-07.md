Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, which means a successful charge that is followed by a DB failure will result in an unrecorded charge that the idempotency check will never catch, directly contradicting the comment claiming crash-safety. There are also several secondary issues around error handling, observability, and test coverage gaps that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment call inside DB transaction breaks the atomicity guarantee

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — that is false. If `record_charge` or `mark_order_complete` raises after `payment.charge()` succeeds, the DB transaction rolls back, the charge is lost from your records, and the idempotency check will never block a retry. The customer gets charged twice.

The correct pattern is charge-then-record with a two-phase approach:

```python
# Correct approach
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Only open the DB transaction after the external call succeeds.
# If this block fails, the charge_id is known and can be reconciled
# via a separate recovery path or a pending-charge record written before charging.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Log the charge_id explicitly so ops can reconcile manually or via job.
    logger.critical(
        "CHARGE_RECORDED_FAILED order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

The truly robust solution is to write a `pending_charge` record *before* calling the gateway, then confirm it after. That pattern is out of scope for this PR but should be tracked as a follow-up.

### 2. Timeout on payment gateway is dangerously ambiguous under retry

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying blindly on timeout can produce duplicate charges if the gateway is not idempotent on your end. At minimum, an idempotency key must be passed per-attempt *group* (not per-attempt):

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},  # same key on every retry
            timeout=self.REQUEST_TIMEOUT,
        )
```

The `order_id` is a natural idempotency key to pass here. Without this, the retry loop is a liability, not a safety net.

## Moderate Issues

### 3. Unhandled exceptions inside the lock context leak silently

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        ...
```

Any exception other than `LockError` — a DB connection error, an unexpected `PaymentError`, a `KeyError` on the response JSON — propagates out of `process()` uncaught. The lock *will* be released (the `finally` block is correct), but the caller gets a raw exception instead of an `OrderResult`. This breaks the contract of the method, which promises to always return an `OrderResult`. Add a catch-all:

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

### 4. `response.json()["charge_id"]` will raise `KeyError` on malformed gateway response

```python
charge_id = response.json()["charge_id"]
```

A 200 response with an unexpected body (e.g., `{}`, or non-JSON) raises `KeyError` or `JSONDecodeError`, which propagates as an untyped exception rather than a `PaymentError`. Wrap it:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {response.text[:200]}") from e
```

### 5. Lock TTL may be insufficient under retry + payment timeout

`ORDER_LOCK_TTL = 60s` with `MAX_RETRIES = 3` and `REQUEST_TIMEOUT = 30s` per attempt plus `RETRY_BACKOFF = 5s` between retries gives a worst-case payment duration of `3 * 30 + 2 * 5 = 100 seconds`. The lock expires before the payment completes, allowing a second worker to acquire the lock and potentially double-charge. Either increase the TTL to at least 120s with a comment explaining the math, or implement lock extension (EXPIRE refresh) during the payment call.

## Minor Issues

### 6. `payment_token` is logged indirectly through `OrderRequest` validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages can include field values. If `payment_token` fails validation, its value may appear in logs. Scrub sensitive fields before logging or log only field names:

```python
logger.warning("Invalid order request fields: %s", [e["loc"] for e in exc.errors()])
```

### 7. Test for lock release does not verify the token passed to the script

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms the Lua script is called but does not assert that the correct token and key are passed. A regression that passes a hardcoded or empty token would pass this test. Assert on the call arguments.

### 8. No test for the DB-failure-after-charge scenario

There is no test covering the case where `payment.charge()` succeeds but `db.record_charge()` raises. This is precisely the failure mode described in Critical Issue #1 and should have explicit coverage regardless of how the fix is implemented.

### 9. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Endpoint should be injected via constructor or environment variable to support staging environments and testing without monkey-patching the class.
