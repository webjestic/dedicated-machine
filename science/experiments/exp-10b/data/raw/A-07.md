# Pull Request Review

**PR:** Add distributed order processor with Redis-backed exclusive locking
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/order-processor`

---

## Detailed Findings

### 🔴 Critical — Charge Outside Transaction Boundary (Double-Charge Risk)

This is the most serious architectural flaw in the PR, and it directly contradicts the safety guarantee stated in the docstring.

```python
# From order_processor.py
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External side effect INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary claims: *"crash between charge and record cannot produce an unrecorded charge."* This is **false**. The `payment.charge()` call is an external HTTP call to a third-party gateway. Database transactions provide atomicity over database operations only. They offer **zero rollback capability** over an already-dispatched HTTP request. The scenarios that break this:

1. `payment.charge()` succeeds and returns `charge_id`. The gateway has money.
2. `db.record_charge()` or `db.mark_order_complete()` throws. The DB transaction rolls back.
3. The lock is released. The idempotency check (`db.get_charge`) now finds nothing.
4. The next worker re-enters and charges the customer again.

The correct pattern is: **charge outside and before the transaction**, then record inside. If the record fails, you have an unrecorded charge to reconcile — but you do not double-charge. Alternatively, implement a two-phase approach with a `pending` charge state written to the DB *before* calling the gateway, so any crash leaves a recoverable audit trail. The current architecture inverts this and creates a silent double-charge window.

### 🔴 Critical — Payment Timeout Retry Creates Duplicate Charge Exposure

```python
# From payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
# ... falls through to retry
```

A `Timeout` exception means the client did not receive a response — it does **not** mean the charge did not execute. The gateway may have processed the charge and the response was lost in transit. Retrying on timeout without an idempotency key submitted to the gateway will submit a second charge attempt against the same payment token. `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` means up to three charge attempts can be dispatched to the gateway for a single order. This is compounded by the lock retry loop in `_acquire_lock` — if a worker times out and another acquires the lock, the idempotency check at the DB layer may not protect against a gateway-level duplicate if the first charge landed silently.

The fix requires passing a stable, per-order idempotency key in the request headers to the payment gateway (e.g., `Idempotency-Key: {order_id}`) so that retries are deduplicated at the gateway level.

---

### 🔴 Critical — Lock TTL Does Not Account for Payment Retry Duration

```python
ORDER_LOCK_TTL = 60   # seconds
# vs.
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Maximum payment client execution time: `(30s × 3 attempts) + (5s × 2 gaps) = 100 seconds`.

The lock TTL is 60 seconds. The payment client can run for up to 100 seconds. If the payment client is still executing when the lock expires at T+60, Redis will release the lock automatically. A second worker can then acquire the lock, pass the idempotency check (nothing recorded yet), and begin charging. This is a textbook lock expiry race condition. The TTL must be set conservatively above the worst-case critical section duration, or the lock must be refreshed (fenced) during long operations.

### 🟡 High — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic `ValidationError` string representations include field values in their output. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value will appear in the log line. Payment tokens are sensitive credentials. The log output should be sanitized or field-level error messages should be extracted without including raw values.

---

### 🟡 High — Unhandled Exceptions Escape `process()` Without Lock Release Guarantee

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
```

`PaymentError`, `db` exceptions, and any unexpected runtime error are not caught inside `process()`. They will propagate up through the `_acquire_lock` context manager's `finally` block, which does correctly release the lock — that part is sound. However, the caller receives an unhandled exception rather than an `OrderResult`. The method signature promises `OrderResult` as its return type. Callers in a worker loop will crash unless they independently catch all exceptions. At minimum, a broad `except Exception` should be added to return a structured `OrderResult(processed=False, message="Internal processing error")` and log the full traceback, while still allowing the `finally` in `_acquire_lock` to execute.

### 🟡 Medium — Idempotency Check Has a TOCTOU Gap Against the Lock

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # read
    if existing:
        return ...
    with self.db.transaction():
        ...                                         # write
```

The lock is meant to serialize this check-then-act sequence, which is correct in principle. However, if the lock TTL expires between the `get_charge` read and the `record_charge` write (see the TTL finding above), a second worker can enter, pass the same idempotency check, and proceed to charge. The idempotency check is only as strong as the lock holding it. This is a dependent finding on the TTL issue but worth calling out explicitly as a compound failure mode.

---

### 🟡 Medium — `requests.post` Has No Connection Timeout Separation

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,    # scalar — applies to both connect and read
)
```

A scalar `timeout=30` in `requests` applies to both the connection establishment and the read. A hung connection waiting 30 seconds to connect, followed by 30 seconds waiting for a response, can block for up to 60 seconds per attempt. This should be a tuple: `timeout=(connect_timeout, read_timeout)` — e.g., `timeout=(5, 30)` — to fail fast on connection issues while allowing adequate time for the gateway to respond.

---

### 🟡 Medium — No Observability on Lock Expiry or Stale Lock Detection

There is no mechanism to detect or alert when a lock TTL expires before the critical section completes. In production, this failure mode is silent — the system will appear to function normally while occasionally double-charging. A lock heartbeat, a fencing token, or at minimum a post-hoc TTL validation check should be considered.

---

### 🟢 Low — `CHARGE_ENDPOINT` Hardcoded as Class Attribute

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Endpoint configuration should be injected via constructor or environment variable. Hardcoded URLs in source make environment promotion (dev → staging → prod) error-prone and require code changes for endpoint rotation.

---

### 🟢 Low — Test Suite Does Not Cover Critical Failure Paths

The tests cover the happy path, already-processed, lock contention, and input validation. Missing coverage:

- Payment gateway timeout followed by retry — verifies no duplicate charge
- DB transaction rollback after successful charge — verifies the double-charge scenario
- Lock TTL expiry mid-execution
- `PaymentError` propagation behavior from `process()`
- Lua script receiving wrong token (verifies it does not release a foreign lock)

The existing tests would pass against the broken architecture without detecting the critical double-charge flaw.

**Decision: Request Changes**

**Summary:** This PR contains three critical flaws that collectively guarantee double-charge scenarios under realistic failure conditions: the payment gateway call is placed inside a database transaction that can roll back after money has moved, the payment client retries on timeouts without a gateway-level idempotency key, and the lock TTL is shorter than the maximum payment client execution time. The distributed lock architecture is structurally sound and the Lua-based atomic release is correctly implemented, but the safety guarantees advertised in the PR description do not hold under
