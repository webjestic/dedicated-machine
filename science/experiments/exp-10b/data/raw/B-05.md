Decision: Request Changes

Summary: The PR demonstrates solid foundational thinking — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several correctness and architectural flaws that must be addressed before this can be approved, the most critical being a charge-before-commit race condition that directly undermines the at-most-once guarantee the PR claims to provide, and a timeout-induced double-charge risk in the payment client that is not handled.

---

## Critical Issues

### 1. Payment charge occurs inside a database transaction — but the charge is not reversible on rollback

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge()` is an external, non-transactional side effect. If `record_charge` or `mark_order_complete` raise an exception, the database transaction rolls back, but **the payment has already been captured**. The customer is charged with no record in the database. On retry, `get_charge` returns `None` (no record exists), the idempotency check passes, and the customer is charged again.

**Required fix:** Charge outside and after the database transaction, or — preferably — use a two-phase approach: write a `PENDING` record inside the transaction first, then charge, then update to `COMPLETE` in a second transaction. The payment gateway call must be the last external side effect, not the first.

```python
# Correct ordering sketch
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This way, a crash after charge but before finalization leaves a `PENDING` record, which can be reconciled. The idempotency check must also inspect `PENDING` records to avoid re-entry.

### 2. Timeout ambiguity in the payment client creates a double-charge vector

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge failed. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout will submit a duplicate charge to the gateway. This is a financial correctness bug.

**Required fix:** The payment client must send a caller-supplied idempotency key (e.g., `order_id` or a stable UUID derived from it) in every request. The gateway will deduplicate on its side. Without this, retries on timeout are unsafe regardless of what the application layer does.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

## Significant Issues

### 3. Lock TTL does not account for payment retry duration

```python
ORDER_LOCK_TTL = 60   # seconds
# PaymentClient: MAX_RETRIES=3, RETRY_BACKOFF=5, REQUEST_TIMEOUT=30
# Worst case: 3 * 30 + 2 * 5 = 100 seconds
```

The maximum payment client execution time (3 attempts × 30s timeout + 2 × 5s backoff) is **100 seconds**, which exceeds the 60-second lock TTL. If the lock expires while a worker is waiting on a payment retry, a second worker can acquire the lock, pass the idempotency check (no record yet), and issue a second charge. The TTL must be set to at least `MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF` plus margin, or the payment client's total timeout must be bounded below the lock TTL.

---

### 4. `PaymentError` is not caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    charge_id = self.payment.charge(...)   # can raise PaymentError
```

`PaymentError` is not caught anywhere in `process()`. It will propagate as an unhandled exception to the caller. The lock will still be released (the `finally` block in `_acquire_lock` handles that correctly), but the caller receives a raw exception rather than a structured `OrderResult`. This breaks the contract of the method, which promises to return `OrderResult`.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### 5. `response.json()["charge_id"]` is not guarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 2xx response with a malformed or missing `charge_id`, this raises a `KeyError` or `json.JSONDecodeError`, neither of which is caught. These will propagate as unhandled exceptions through `process()` (compounding issue 4). The response shape must be validated before accessing fields.

## Minor Issues

### 6. Lock retry backoff is linear and short

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Under high contention with many workers, linear backoff with no jitter causes retry storms. Exponential backoff with jitter is the standard pattern here. This is a performance concern, not a correctness bug, but worth addressing.

---

### 7. Test coverage gaps

- No test for `PaymentError` propagation from `process()` (which is currently broken per issue 4).
- No test for lock expiry during processing.
- No test for gateway timeout followed by retry.
- `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` but does not verify the Lua script was called with the correct token — it only checks call count, not arguments.
