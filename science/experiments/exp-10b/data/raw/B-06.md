Decision: Request Changes

Summary: The architecture has a critical atomicity flaw: the payment gateway charge is executed *inside* the database transaction, meaning a successful charge followed by a database commit failure leaves an unrecorded, unrecoverable charge with no compensation path — the opposite of the at-most-once guarantee claimed in the docstring. Additionally, the Redis lock TTL of 60 seconds is dangerously mismatched against a payment client that can run up to 90 seconds (3 attempts × 30s timeout), meaning the lock can expire and be acquired by a competing worker while the original worker is still mid-charge. There are several additional issues that must be resolved before this can be approved.

---

## Detailed Findings

### Critical — Must Fix

**1. Payment charge inside the database transaction violates at-most-once semantics**

```python
# Current — charge happens inside db.transaction()
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `self.payment.charge()` succeeds but the database commit fails (network partition, DB crash, deadlock), the charge is real and the money is gone, but nothing is recorded. The system has no way to reconcile this. The comment "crash between charge and record cannot produce an unrecorded charge" is factually wrong — a DB commit failure after a successful charge does exactly that.

The correct pattern for at-most-once is to record *intent* before charging, then confirm after:

```python
# Correct approach
with self.db.transaction():
    self.db.record_charge_intent(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This way the idempotency check on re-entry sees the intent record and can either confirm or void the charge via the gateway's idempotency key. The payment client must also send a stable idempotency key (e.g., derived from `order_id`) so the gateway deduplicates retries.

---

**2. Lock TTL shorter than maximum payment client runtime**

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient: MAX_RETRIES=3, REQUEST_TIMEOUT=30, RETRY_BACKOFF=5
# Worst case: 30 + 5 + 30 + 5 + 30 = 100 seconds
```

The lock can expire after 60 seconds while the payment is still in flight. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. This is a double-charge bug that the lock was specifically designed to prevent. Either raise `ORDER_LOCK_TTL` above the maximum possible payment duration with margin, or implement lock extension (Redlock-style keepalive). At minimum:

```python
ORDER_LOCK_TTL = 120  # must exceed MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)
```

### Significant — Must Fix

**3. Unhandled exceptions inside the lock context leak silently**

`process()` catches `LockError` but nothing else. If `self.payment.charge()` raises `PaymentError`, or the DB raises any exception, it propagates out of the `with self._acquire_lock(...)` block. The lock *is* released correctly by the `finally` block, but the caller receives an unhandled exception rather than a structured `OrderResult`. This breaks the contract of the method signature and will produce 500 errors at the API layer with no useful response body.

```python
# Add inside the lock context
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, order_id=req.order_id, message="Internal error")
```

---

**4. Payment client retries on timeout without idempotency key**

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without a stable idempotency key will issue a second charge. The gateway's idempotency key must be sent on every attempt and must be stable across retries:

```python
# charge() must accept and forward an idempotency key
headers = {"Idempotency-Key": idempotency_key}
response = requests.post(..., headers=headers)
```

The `order_id` is the natural idempotency key and should be threaded through from `process()` to `payment.charge()`.

---

**5. Redis connection errors are not handled**

If Redis is unavailable, `self.redis.set(...)` raises `redis.exceptions.ConnectionError` or `redis.exceptions.RedisError`. This propagates as an unhandled exception. The lock acquisition should catch Redis-specific errors and either raise `LockError` (fail closed, safe) or surface a structured result. Failing closed is strongly preferred for a payment system:

```python
try:
    acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
except redis.exceptions.RedisError as e:
    logger.error("Redis unavailable, refusing to process order_id=%s: %s", order_id, e)
    raise LockError(f"Lock backend unavailable for order {order_id}") from e
```

---

### Minor — Should Fix

**6. Lock retry backoff is linear and too short**

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Linear backoff with no jitter means all contending workers retry in lockstep. Use exponential backoff with jitter:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

---

**7. `customer_id` has no pattern constraint**

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. If `customer_id` is used in log output or database queries anywhere downstream, this is a potential injection vector. Apply the same or equivalent constraint.

---

**8. Test for lock release does not verify the token**

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test verifies the Lua script was called but does not assert that the correct token was passed as `args`. A regression that passes the wrong token or no token would not be caught. Assert on the call arguments:

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] == <expected_token>
# or capture the token via a spy on uuid.uuid4
```

---

**9. `PaymentClient` is not injected with configuration**

`CHARGE_ENDPOINT`, `REQUEST_TIMEOUT`, `MAX_RETRIES`, and `RETRY_BACKOFF` are hardcoded class-level constants. These should be injectable via constructor or configuration object to support environment-specific deployments and testing without monkey-patching.
