Decision: Request Changes

Summary: The core locking architecture contains a critical flaw that undermines the entire at-most-once guarantee the PR claims to provide: a payment gateway timeout during retry creates an undetected double-charge window that the current design cannot recover from. There are also several secondary issues spanning exception handling gaps, a missing `PaymentError` catch, and test coverage blind spots that need to be addressed before this handles real financial transactions.

---

## Critical Issues

### 1. Payment Timeout Creates Undetectable Double-Charge (Critical)

This is the most serious bug. In `payment_client.py`, a `Timeout` exception is treated as a safe retry signal, but a timeout does **not** mean the charge failed — it means you don't know whether it succeeded. The payment gateway may have processed the charge and the response was simply lost in transit.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # Falls through to retry — but the charge may have already landed
```

On the next attempt, you submit a second charge request with the same token and amount. If the gateway is not idempotent by default, the customer is charged twice. The Redis lock and the DB idempotency check both occur **before** the charge call, so they offer zero protection here.

**Required fix:** The payment gateway API must support idempotency keys. Generate a deterministic idempotency key (e.g., `HMAC(order_id + attempt_nonce)`) and pass it in a header on every attempt. On retry, reuse the same key so the gateway deduplicates. If the gateway does not support this, timeouts must be treated as permanent failures, not retryable ones, and a reconciliation job must be added.

---

### 2. `PaymentError` Is Never Caught in `process()` (Critical)

`OrderProcessor.process()` catches `LockError` but has no handler for `PaymentError` or any other exception raised inside the lock context. A payment failure or any unexpected exception will propagate out of the `with self._acquire_lock()` block, which means the `finally` clause in `_acquire_lock` will still release the lock correctly — but the caller receives an unhandled exception rather than a structured `OrderResult`. For a service boundary this is unacceptable.

```python
# In process() — only LockError is caught
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
# PaymentError, requests.ConnectionError, db errors, KeyError on response.json() — all uncaught
```

**Required fix:** Add a broad exception handler that returns a structured failure `OrderResult` and logs the full traceback. Distinguish between payment-specific failures and unexpected errors in the log.

---

### 3. `charge_id = response.json()["charge_id"]` — Unguarded Key Access

If the payment gateway returns a 200 with a malformed or unexpected response body, `response.json()["charge_id"]` raises a `KeyError` (or `json.JSONDecodeError`). This propagates as an unhandled exception. Given issue #2 above, this is especially dangerous: the charge may have succeeded but the `charge_id` was never captured, and the DB record is never written.

```python
charge_id = response.json()["charge_id"]  # KeyError if schema changes
```

**Required fix:** Validate the response schema explicitly. Log the raw response body on failure before raising.

## Architectural Issues

### 4. Lock TTL vs. Payment Retry Window Mismatch

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock will expire while the payment is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (no DB record yet), and issue a concurrent charge.

```python
ORDER_LOCK_TTL = 60        # seconds
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
# Worst case: 3 * (30 + 5) = 105 seconds > 60 second TTL
```

**Required fix:** Either raise `ORDER_LOCK_TTL` to safely exceed the maximum payment client execution time with margin (e.g., 120–150s), or implement lock extension (a background thread that refreshes the TTL while the operation is in progress, similar to Redlock's watchdog pattern).

---

### 5. Single Redis Node — No Fault Tolerance

The implementation uses a single `redis.Redis` client. If that node fails after the lock is acquired but before the DB transaction commits, the lock is gone and another worker can proceed. For a financial processing service this is a known risk that should be explicitly documented or mitigated.

**Required fix:** Either use `rediscluster` / Redis Sentinel with appropriate failover configuration, or document clearly in the module that this implementation assumes Redis availability and is not suitable for deployments where Redis uptime cannot be guaranteed. At minimum, a comment in the code acknowledging this tradeoff is required.

---

### 6. `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include the field value in their string representation in some configurations. If `payment_token` fails validation, its value (a sensitive credential) may appear in log output. This is a data exposure risk.

**Required fix:** Either sanitize the validation error before logging (exclude sensitive fields by name) or log only the field names and error types, not the values.

## Test Coverage Gaps

### 7. No Test for Payment Timeout / Retry Behavior

`test_order_processor.py` has no test that exercises the retry logic in `PaymentClient`, and specifically no test that verifies behavior when a timeout occurs on attempt 1 but succeeds on attempt 2. Given that this is the exact scenario that produces double charges (issue #1), it needs a test.

### 8. No Test for Lock Expiry During Payment

There is no test simulating the lock expiring mid-payment (issue #4). This scenario is testable by mocking time or by injecting a short TTL.

### 9. `test_lock_release_uses_lua_script` Does Not Verify Token Correctness

The test confirms the Lua script was called but does not assert that the correct token was passed as `args[0]`. A refactor could accidentally pass the wrong value and this test would still pass.

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
    # Missing: assert the token in args matches what was SET
```

## Minor Issues

- `CHARGE_ENDPOINT` is hardcoded as a class attribute. This should be injected via configuration or environment variable.
- No timeout is set on the DB transaction operations. A hung DB call inside the lock context will hold the lock until TTL expiry.
- `order_id` is not included in the `OrderResult` on the lock contention path, which makes client-side correlation harder.
