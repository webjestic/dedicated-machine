**PR:** Add distributed order processor with Redis-backed exclusive locking
**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/order-processor`

## Detailed Findings

### 🔴 Critical — Charge Outside Transaction Scope (Double-Charge Risk)

This is the most serious flaw in the PR, and it directly contradicts the stated safety guarantee in the docstring.

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(          # ← EXTERNAL SIDE EFFECT
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is an **external HTTP call to a third-party payment gateway**. It is not part of the database transaction. Database transactions provide atomicity over database operations only. They have no rollback authority over a charge that has already been submitted to and accepted by the payment processor.

The failure scenario is straightforward and realistic:

1. `payment.charge()` succeeds — money is taken from the customer.
2. `db.record_charge()` raises an exception — network blip, constraint violation, anything.
3. The `db.transaction()` context manager rolls back.
4. The system has no record of the charge, but the charge exists at the gateway.
5. The idempotency check on retry finds nothing and charges again.

The docstring comment — *"crash between charge and record cannot produce an unrecorded charge"* — is factually incorrect and dangerous. This is not a theoretical edge case. This is a well-understood failure mode in distributed payment systems.

**Correct pattern:** The charge must be recorded *before* it is submitted, using an idempotency key that is also sent to the gateway. The flow should be:

1. Write a `PENDING` charge record with a stable idempotency key inside a transaction.
2. Submit the charge to the gateway, passing the idempotency key.
3. Update the record to `COMPLETED` with the returned `charge_id`.

If the process crashes after step 2 but before step 3, a reconciliation job or retry can query the gateway using the idempotency key to determine the true outcome. The gateway's own idempotency key support prevents a re-submission from firing a second charge.

### 🔴 Critical — Payment Retry Amplifies the Double-Charge Risk

`PaymentClient.charge()` retries up to three times on timeout:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout does not mean the charge failed. The request may have been received and processed by the gateway before the connection dropped. Retrying a timed-out charge without a gateway-side idempotency key will submit a second, third, or fourth independent charge request. There is no idempotency key being passed in the current payload:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # ← no idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

This is a compounding problem. The Redis lock prevents concurrent workers from racing, but it does not protect against this client-side retry loop issuing duplicate charges against the same token within a single lock acquisition. These two issues together represent a credible path to billing a customer multiple times.

---

### 🔴 Critical — Lock TTL Does Not Cover Payment Retry Duration

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

```python
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment client execution time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`.

The lock TTL is 60 seconds. The payment operation can legally exceed it by 40 seconds. After the TTL expires, another worker can acquire the lock, pass the idempotency check (charge not yet recorded), and begin a second charge. The lock's entire safety guarantee evaporates in this scenario. Either the TTL must exceed the maximum possible payment operation duration with margin, or the lock must be actively refreshed (fenced with a monotonic token or a watchdog extension loop).

### 🟡 High — `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), its value — a live payment credential — will appear in the log output. Depending on log aggregation infrastructure, this could persist in plaintext in a SIEM, log shipper, or stdout capture. The log statement should be sanitized to exclude field values, or at minimum should explicitly redact sensitive fields before logging.

---

### 🟡 High — Redis Failure Is Silently Treated as a Lock Acquisition Failure

If the Redis client raises a `ConnectionError` or `TimeoutError` during `self.redis.set(...)`, the exception will propagate out of `_acquire_lock` uncaught, bypassing the `LockError` path entirely. The caller in `process()` only catches `LockError`:

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

A Redis infrastructure failure will surface as an unhandled exception to whatever is calling `process()`. Depending on the calling context this may be acceptable, but the behavior is undocumented and inconsistent with the structured error handling pattern the rest of the code establishes. More importantly, if Redis is unavailable and the exception is swallowed somewhere upstream, the system could fall through to processing without a lock. The intended behavior on Redis failure — fail open or fail closed — must be an explicit, documented, and tested decision.

### 🟡 High — `db.get_charge()` Is Outside the Transaction (TOCTOU)

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← read outside transaction
    if existing:
        return OrderResult(...)

    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

The idempotency read and the subsequent write are not in the same transaction. Under the Redis lock this is less dangerous than it would be otherwise, but if the lock expires mid-execution (see TTL issue above), two workers could both read `None` from `get_charge`, both proceed past the check, and both enter the charge path. The idempotency check should be inside the transaction, and the `record_charge` should use a unique constraint at the database level as the final enforcement mechanism — not just an application-level read.

---

### 🟠 Medium — Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep()` blocks the thread. In any async or thread-pool-based worker environment this ties up a worker thread for up to 1.5 seconds of cumulative sleep per contended order. The retry count is also hardcoded at 3 with no configuration surface. If this service runs under a thread-pool executor or an async framework, this needs to be revisited. Jitter should also be added to prevent retry storms when many workers are contending on the same key simultaneously.

---

### 🟠 Medium — `payment_token` Appears in `OrderRequest` and Persists in Memory

`payment_token` is stored on the validated `OrderRequest` Pydantic model. Pydantic models are not zeroed on deletion. In long-lived worker processes, this value will persist in heap memory until garbage collected. For a government contracting context this warrants at minimum a note in the security documentation, and ideally a `SecretStr` type from Pydantic which prevents the value from appearing in `repr()` and `str()` outputs.

---

### 🟠 Medium — No Authentication or Authorization on `process()`

The `process()` method accepts a `customer_id` as a plain string parameter with no verification that the caller is authorized to submit charges on behalf of that customer. There is no evidence of an auth layer above this in the PR. This may be handled at a higher layer, but it is not visible here and should be explicitly documented or enforced.

---

### 🟢 Low — Test Coverage Gaps

The test suite covers the happy path and basic contention, but is missing coverage for:

- Lock TTL expiry mid-execution.
- Redis connection failure behavior.
- Payment timeout followed by a successful retry (to catch idempotency key absence).
- `db.record_charge()` failure after a successful `payment.charge()` — the exact scenario that produces an unrecorded charge.
- Validation error logging to confirm `payment_token` is not leaked.

The existing tests also mock at too high a level to catch the transaction-scope issue. `db.transaction().__enter__` and `__exit__` are mocked to succeed unconditionally, which means no test exercises what happens when the transaction rolls back after a successful charge.

---

### 🟢 Low — Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoints belong in configuration, not source code. This prevents environment-specific overrides (staging, sandbox, test) without code changes and makes the class untestable without network mocking.
