Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful external charge followed by any DB failure will leave an unrecorded charge with no recovery path, directly contradicting the comment claiming crash safety. There are also several secondary issues around error handling gaps, a timeout hazard, and missing test coverage that need to be addressed before this is mergeable.

---

## Critical Issues

### 1. External Side Effect Inside DB Transaction (Payment Integrity Flaw)

This is the most serious issue. The PR comment says:

> *"Crash between charge and record cannot produce an unrecorded charge"*

That claim is false. The current code:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External HTTP call — cannot be rolled back
    self.db.record_charge(...)             # If this raises, charge is already gone
    self.db.mark_order_complete(...)
```

A DB transaction provides atomicity over *database* operations only. `payment.charge()` is an external HTTP call — it is not transactional and cannot be rolled back if the subsequent DB writes fail. If `record_charge` or `mark_order_complete` raises after the charge succeeds, you have taken money from the customer with no record of it. The comment actively misleads future maintainers about the safety guarantee.

**Required fix:** Charge outside the transaction, then open the transaction only for the DB writes. If the DB write fails after a successful charge, you need a reconciliation/recovery path (e.g., a pending-charges table, an outbox pattern, or at minimum an alert). The lock TTL gives you a window, but silent failure here is unacceptable.

```python
# Correct structure:
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# If we crash here, the idempotency check on retry must detect the orphaned charge.
# Consider writing a "charge_pending" record before calling payment.charge().

with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The most robust pattern is an **outbox/pre-insert**: write a `charge_pending` row inside a transaction *before* calling the payment gateway, then update it to `complete` after. This gives you a recovery audit trail.

## Significant Issues

### 2. Unhandled Exceptions Escape `process()` — Lock Is Released But Caller Gets an Unstructured Exception

`PaymentError`, `db.transaction()` failures, `db.get_charge()` failures, and network errors from the payment client all propagate uncaught out of `process()`. The method signature promises an `OrderResult` return, but callers will receive raw exceptions in these paths. This breaks the contract and will crash any caller that doesn't wrap the call defensively.

```python
# Missing: catch PaymentError, DB errors, and unexpected exceptions
except LockError:
    return OrderResult(processed=False, message="Order currently being processed")
# PaymentError, DB errors → uncaught, propagate as exceptions
```

At minimum, add:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 3. Lock TTL vs. Payment Timeout Math Is Unsafe

```python
ORDER_LOCK_TTL = 60        # seconds
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client execution time: `(30s timeout + 5s backoff) × 3 attempts = 105 seconds`. This exceeds the 60-second lock TTL by 45 seconds. The lock will expire while a payment attempt is still in flight, allowing a second worker to acquire the lock and potentially issue a duplicate charge — defeating the entire purpose of the lock.

**Required fix:** Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client wall time (e.g., 120–150s with margin), or implement lock extension (re-`SET` with a fresh TTL mid-flight), or reduce the payment client's maximum total time budget to fit within the lock TTL.

### 4. `payment_token` Logged or Exposed in Error Paths

`PaymentError` raised in `payment_client.py` includes the raw HTTP status, which is fine, but `req.payment_token` is passed directly into `payment.charge()`. Confirm that `payment_token` is never interpolated into log messages anywhere in the call chain. Currently the `OrderRequest` model holds it as a plain `str` with no masking. If any future logging of the full `req` object is added, this will leak a payment credential. Consider a custom `SecretStr` type or a `__repr__` override.

## Minor Issues

### 5. `requests` Used Without Connection Pooling or Session Reuse

`PaymentClient` calls `requests.post(...)` directly on every invocation, which creates a new TCP connection and TLS handshake each time. For a service under load this is wasteful and increases per-call latency. Use a `requests.Session` (ideally with `HTTPAdapter` retry/pool settings), instantiated once in `__init__`.

### 6. `response.json()["charge_id"]` — No `KeyError` Guard

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id`), this raises a bare `KeyError` that propagates as an unhandled exception. Wrap with a `.get()` check and raise a descriptive `PaymentError` if the field is absent.

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned success but no charge_id: {data}")
```

### 7. Test Coverage Gaps

The tests validate the happy path and a few contention/validation cases, but are missing:

- **`PaymentError` propagation** — verify that a payment failure returns a structured `OrderResult` (currently it would raise, not return)
- **Lock TTL expiry during payment** — no test for the race condition identified in issue #3
- **DB transaction failure after successful charge** — the most dangerous failure mode has zero test coverage
- **`charge_id` missing from gateway response** — `KeyError` path in `payment_client.py`
- **Lock release called on exception paths** — verify the `finally` block fires when `payment.charge()` raises

### 8. `LOCK_RETRY_ATTEMPTS = 3` With No Jitter

The retry backoff is deterministic (`0.5 * (attempt + 1)`). Under high concurrency, multiple workers hitting the same order will all retry on the same schedule, creating a thundering herd. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

## What Is Done Well

- `SET NX EX` is the correct atomic acquisition primitive — no SETNX/EXPIRE race
- Lua script for token-verified release is correct and necessary
- Pydantic validation with regex pattern on `order_id` is the right approach and correctly blocks path traversal
- `ValidationError` is caught and returns a structured result
- The idempotency check placement (inside the lock) is correct — checking outside would be a TOCTOU race
- Structured logging with consistent field naming throughout
