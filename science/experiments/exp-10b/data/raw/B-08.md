Decision: Request Changes

Summary: The core locking and idempotency architecture contains a critical race condition that undermines the stated at-most-once payment guarantee, and the payment client has a dangerous retry pattern that can produce duplicate charges under timeout conditions. Several secondary issues compound the risk. None of these are cosmetic — they represent real double-charge exposure in a concurrent production environment.

---

## Critical Issues

### 1. Payment Charge Outside the Database Transaction Does Not Provide Atomicity — It Inverts the Risk

The comment in the code claims:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
```

This is incorrect. `self.payment.charge(...)` is a **network call to an external payment gateway**. It is not a database operation and cannot participate in a database transaction. What the `with self.db.transaction()` block actually guarantees is that `record_charge` and `mark_order_complete` are atomic with respect to each other — not that the charge and the record are atomic together.

If the process crashes, is killed, or loses network connectivity after `payment.charge()` returns a `charge_id` but before `self.db.record_charge(...)` commits, you have a real charge with no record. The database transaction does not help here. This is the classic two-phase commit problem and it is not solved.

**Correct approach:** Record a `PENDING` charge row *before* calling the payment gateway (within a transaction), then update it to `COMPLETE` with the returned `charge_id` after the gateway responds. The idempotency check must query for both states. A separate reconciliation job should handle rows stuck in `PENDING`.

---

### 2. Payment Retry on Timeout Will Produce Duplicate Charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A `requests.exceptions.Timeout` means the client did not receive a response within 30 seconds. It does **not** mean the charge was not processed. The gateway may have received the request, authorized the card, and sent a response that never arrived. Retrying unconditionally on timeout will charge the customer again.

This is the most dangerous bug in the PR. Combined with the missing `PENDING` record pattern above, there is no recovery path.

**Correct approach:** The payment gateway must support idempotency keys. Generate a stable idempotency key (e.g., derived from `order_id`) and include it in every attempt for a given order. The gateway will deduplicate retries server-side. Without this, retry-on-timeout is not safe.

---

### 3. Redis Lock Does Not Prevent the Race It Claims To

The idempotency check reads:

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return ...
```

This check occurs *inside* the lock, which is correct. However, the lock TTL is 60 seconds and the payment gateway timeout is 30 seconds with up to 3 retries at 5-second backoff. Worst case, a single `process()` call can take approximately `3 × 30 + 2 × 5 = 100 seconds` — well beyond the 60-second lock TTL.

If the lock expires mid-processing, a second worker can acquire the lock, pass the idempotency check (the charge has not been recorded yet because the first worker is still waiting on the gateway), and issue a second charge. The lock provides no protection in this scenario.

**Correct approach:** Either (a) ensure `ORDER_LOCK_TTL` is strictly greater than the maximum possible processing time with margin, or (b) implement lock extension (heartbeat refresh) while the lock is held, or (c) rely on the `PENDING` record pattern from Issue 1 as the true idempotency barrier rather than the lock alone.

## Significant Issues

### 4. `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., too long), the token value may appear in log output. Payment tokens must never appear in logs. At minimum, scrub or summarize the error before logging, or log only field names without values.

---

### 5. Lock Release Silently Swallows Failures

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

The return value of `_release_script` is not checked. A return value of `0` means the lock was not released — either it expired or another worker holds it. The log line says "released" regardless. This makes debugging lock expiry invisible and gives false confidence. The return value should be checked and a warning logged when release fails.

---

### 6. `test_lock_release_uses_lua_script` Does Not Assert Correct Arguments

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test only verifies that the script was called once. It does not verify that the correct `keys` and `args` were passed — specifically, that the token used to acquire the lock is the same token passed to the release script. The entire security property of the token-verified release goes untested. Use `assert_called_once_with(keys=[...], args=[...])` with the expected token value.

---

### 7. No Handling of Unexpected Exceptions in `process()`

If `self.db.get_charge()`, `self.db.transaction()`, or any database call raises an unexpected exception, it propagates out of `process()` uncaught. The lock will still be released via `finally`, which is correct, but the caller receives an unhandled exception rather than a structured `OrderResult`. For a service boundary, all exception paths should return a typed result or be explicitly documented as raising.

---

### 8. `requests` Has No Connection Timeout

```python
timeout=self.REQUEST_TIMEOUT,
```

When passed a scalar, `requests` applies this as the read timeout only, not the connection timeout. A hung DNS resolution or TCP handshake can block indefinitely. Use `timeout=(connect_timeout, read_timeout)` as a tuple, e.g., `timeout=(5, 30)`.
