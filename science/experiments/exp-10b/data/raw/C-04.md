Decision: Request Changes

Summary: The architecture is sound and the security fundamentals are solid — atomic lock acquisition, token-verified Lua release, idempotency check, and input validation are all correctly implemented. However, there is a critical reliability flaw in the payment-within-transaction pattern: a successful external charge that precedes a database failure will produce an unrecorded, unrefundable charge with no recovery path, which is the exact double-charge/lost-charge scenario this service is meant to prevent. Several secondary issues also need to be addressed before this is production-ready.

---

## Critical Issues

### 1. External Payment Call Inside Database Transaction (Blocking Flaw)

```python
# services/order_processor.py — lines inside with self.db.transaction():
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**The problem:** The comment claims "crash between charge and record cannot produce an unrecorded charge," but this is incorrect. The database transaction provides atomicity over database operations only. If `payment.charge()` succeeds and returns a `charge_id`, but the subsequent `record_charge()` or `mark_order_complete()` raises (DB connection drop, constraint violation, OOM, process kill), the transaction rolls back and the charge is lost from your records — but the customer has been charged. The Redis lock is also released in `finally`, so a retry will hit the idempotency check, find nothing, and charge again.

Additionally, holding a database transaction open across a 30-second (potentially 90-second with retries) external HTTP call will exhaust your connection pool under any real load.

**Required fix:** Use a two-phase approach with a pending-charge record:

```python
# Correct ordering — no external call inside the DB transaction
with self.db.transaction():
    self.db.record_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# External call outside the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Finalize — idempotency key is the pending record created above
with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

If the process dies after `payment.charge()` but before `finalize_charge()`, a reconciliation job can detect the pending record and resolve it against the payment gateway. This is the standard pattern for at-most-once payment guarantees.

## Significant Issues

### 2. Timeout Retry Creates a Charge Ambiguity Window

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying on timeout without an idempotency key passed to the gateway will create duplicate charges. The payment client must send a stable idempotency key (e.g., the `order_id`) as a request header on every attempt for the same order.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # required
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge()` method signature needs to accept this key, and `OrderProcessor` should pass `req.order_id`.

### 3. Lock TTL Is Insufficient Given Payment Retry Duration

```python
ORDER_LOCK_TTL = 60  # seconds
# PaymentClient: MAX_RETRIES=3, REQUEST_TIMEOUT=30, RETRY_BACKOFF=5
# Worst case: 3 * 30 + 2 * 5 = 100 seconds
```

The payment client can take up to 100 seconds in the worst case. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still processing, defeating the entire concurrency guarantee. Either raise `ORDER_LOCK_TTL` to at least 120 seconds with a comment explaining the calculation, or implement lock extension (renewal heartbeat). The TTL value must be derived from the payment client's configuration, not hardcoded independently.

### 4. Unhandled Exceptions Leak Sensitive Data

```python
def process(self, ...):
    try:
        with self._acquire_lock(req.order_id):
            ...
    except LockError:
        return OrderResult(...)
    # PaymentError, DB exceptions, redis.RedisError — all uncaught
```

Any exception other than `LockError` propagates to the caller with a raw stack trace that may contain payment tokens, customer IDs, or internal hostnames. Add a top-level catch that logs the full exception internally and returns a sanitized `OrderResult`.

```python
except Exception:
    logger.exception("Unexpected error processing order order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

### 5. Redis Failure Causes Silent Order Drop

If `self.redis.set()` raises `redis.RedisError` (connection failure, cluster failover), the exception propagates uncaught. Depending on your deployment, this may be acceptable (fail closed), but the behavior should be explicit and the caller should receive a structured response, not an unhandled exception. Document the intended behavior and add the appropriate catch.

## Minor Issues

### 6. `payment_token` Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., too long), its value may appear in logs. Mask sensitive fields or log only field names and error types, not values.

### 7. Test Coverage Gaps

The test suite covers the happy path and basic contention, but is missing:

- **Lock TTL expiry during payment** — verify behavior when the lock expires mid-processing
- **`PaymentError` propagation** — what does the caller receive when the gateway permanently declines?
- **DB transaction rollback** — verify no charge is recorded if `record_charge` raises
- **Timeout retry ambiguity** — verify idempotency key is passed on retries (once fix #2 is applied)
- **Redis connection failure** — verify structured response, not unhandled exception

### 8. `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as Class-Level Constants

These should be instance-level configuration (constructor parameters with defaults) so they can be overridden in tests and different deployment environments without subclassing.

## What Is Done Well

- Lua script for atomic token-verified lock release is correct and necessary
- `SET NX EX` for atomic lock acquisition is correct
- Pydantic pattern enforcement on `order_id` correctly blocks path traversal
- Idempotency check placement (inside the lock) is correct — avoids TOCTOU
- Structured logging with contextual fields throughout
- `PaymentError` correctly distinguishes permanent (4xx) from transient (5xx/timeout) failures
- Test for Lua script usage verifying `redis.delete` is not called directly is a good negative assertion
