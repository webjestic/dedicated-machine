Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all solid patterns. However, there is a critical atomicity flaw at the seam between the payment gateway call and the database transaction that breaks the "at-most-once" guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment charge happens *inside* the DB transaction — this does not make it atomic

The comment says:
```
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
```

This is incorrect and the reasoning is dangerously misleading. The `payment.charge()` call is a **network call to an external system**. It is not a DB operation and cannot participate in the DB transaction. If the process crashes after `payment.charge()` returns a `charge_id` but before `db.record_charge()` commits, the customer has been charged and there is no record of it. The DB transaction provides atomicity between `record_charge` and `mark_order_complete`, but it provides **zero protection** for the gap between the gateway response and the DB write.

**Required fix:** Separate the charge call from the DB transaction. Record a "charge pending" row *before* calling the gateway (within a transaction), then call the gateway, then update the row to "completed" in a second transaction. On recovery/retry, detect the pending row and reconcile against the gateway using an idempotency key. This is the standard two-phase approach for external payment calls.

```python
# Sketch of correct structure:
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id,
                                  req.amount_cents, idempotency_key)

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=idempotency_key,   # <-- gateway-side dedup
)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

### 2. Timeout retry in `PaymentClient` can produce duplicate charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request may have **already been processed** by the gateway — the response was simply lost in transit. Retrying without a gateway-side idempotency key will attempt a second charge. This is a direct path to double-billing.

**Required fix:** Generate an idempotency key per logical charge attempt (not per HTTP retry) and pass it as a header or body field to the gateway on every retry of the same charge.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

## Significant Issues

### 3. Lock TTL does not account for retry duration

```python
ORDER_LOCK_TTL = 60        # seconds
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client duration: `(30s timeout + 5s backoff) × 3 retries = 105 seconds`. The lock TTL is 60 seconds. The lock can expire while a payment is in-flight, allowing a second worker to acquire it and begin a duplicate charge. Either increase `ORDER_LOCK_TTL` to safely exceed the maximum payment client duration with margin, or enforce that the payment client's total budget is strictly less than the lock TTL.

### 4. Unhandled exceptions inside the lock context leave no audit trail

If `self.payment.charge()` or `self.db.*` raise an unexpected exception, it propagates out of `_acquire_lock`, the lock is released (correctly), but there is no catch, no logging of the failure, and no returned `OrderResult`. The caller gets an unhandled exception. At minimum, wrap the body in a broad except, log the error with the `order_id`, and return a structured failure result.

### 5. `payment_token` is logged at INFO level in `charge_id` log line context

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is present on `req` and could easily be added to a log statement during future debugging. The `OrderRequest` model should override `__repr__`/`__str__` or use Pydantic's `model_config` to exclude `payment_token` from serialization to prevent accidental token leakage into logs.

```python
class OrderRequest(BaseModel):
    model_config = ConfigDict(json_schema_extra={})

    payment_token: str = Field(..., exclude=True, min_length=1, max_length=256)
```

## Minor Issues

### 6. Test for lock release is testing mock wiring, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that the mock was called, not that the Lua script actually enforces token ownership. The Lua script itself should have an integration test against a real (or `fakeredis`) Redis instance that verifies a foreign token cannot release the lock.

### 7. `RETRY_BACKOFF` is a flat constant, not exponential

The PR description says "retry with backoff" but `PaymentClient` uses a flat 5-second sleep. The lock contention retry in `_acquire_lock` does use linear backoff (`0.5 * (attempt + 1)`). These should be consistent and documented. For a payment gateway, exponential backoff with jitter is the standard.

### 8. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

A `KeyError` here (malformed gateway response) will propagate as an unhandled exception rather than a `PaymentError`. Wrap this in a try/except and raise `PaymentError` with context.
