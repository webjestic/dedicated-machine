Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* `db.transaction()`, meaning a successful charge that is followed by a DB commit failure will produce an unrecorded, unrecoverable charge against the customer — the exact double-charge/lost-charge scenario this service is supposed to prevent. Additionally, there are several secondary issues around error handling, timeout safety, and test coverage gaps that need to be addressed before this merges.

---

## Critical Issue

### Payment call inside DB transaction creates an unrecoverable charge-without-record scenario

```python
# Current code — BROKEN
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)             # if this or commit fails, charge is lost
    self.db.mark_order_complete(...)
```

The comment in the PR says *"crash between charge and record cannot produce an unrecorded charge"* — this is **incorrect**. A DB transaction rollback does not roll back the external payment gateway call. If `record_charge`, `mark_order_complete`, or the commit itself raises an exception, the customer has been charged but your system has no record of it. On the next retry, the idempotency check finds no existing charge and charges them again.

**Required fix:** Charge outside and before the transaction, then use the transaction only for the DB writes. Accept that you need a reconciliation path for the gap between charge success and DB write failure — a common pattern is a pre-write "pending charge" record, or a separate idempotency key passed to the payment gateway so the gateway itself deduplicates.

```python
# Safer pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# DB write failure here is recoverable via reconciliation against the gateway
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

Even better: pass `order_id` as an idempotency key to the payment gateway so a retry of the charge call returns the same `charge_id` rather than creating a second charge.

## High Severity

### Lock TTL can expire during a slow payment call, allowing a second worker to enter

`ORDER_LOCK_TTL = 60s` and `PaymentClient.REQUEST_TIMEOUT = 30s` with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5s`. Worst-case payment duration: `3 × 30s + 2 × 5s = 100s`, which exceeds the 60-second lock TTL by 40 seconds. A second worker can acquire the lock while the first is still mid-charge.

**Required fix:** Either extend the lock TTL to safely exceed the worst-case payment duration with margin (e.g., 120–150s), or implement lock heartbeat renewal. Also document the relationship between these constants explicitly.

### Unhandled exceptions inside the lock are silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, DB exceptions, and unexpected runtime errors are not caught. They will propagate out of `process()` as unhandled exceptions. The caller gets no `OrderResult` — they get a raw exception. Given this is described as a service method, it should return a structured result or the exception contract needs to be explicitly documented and tested.

## Medium Severity

### `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value may appear in logs. Payment tokens are sensitive — they should never appear in log output.

**Fix:** Either sanitize the error before logging, or log only field names and error types, not values.

### `PaymentClient` has no authentication on the internal gateway call

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no auth headers, no mTLS configuration, no API key. Even for an internal endpoint, this is a gap — especially in a government contracting context where defense-in-depth is expected. At minimum, inject credentials via constructor and document the auth mechanism.

### Retry on timeout is unsafe without gateway-side idempotency

Retrying a timed-out POST to the payment gateway can result in duplicate charges if the first request succeeded but the response was lost in transit. This is a well-known hazard. The fix is to send a stable idempotency key (e.g., `order_id`) in the request headers so the gateway deduplicates.

## Low Severity / Test Gaps

### Tests do not cover the lock-expiry-during-payment scenario

There is no test verifying behavior when the lock TTL is shorter than the payment duration. This is the scenario most likely to cause a production incident.

### Tests do not cover `PaymentError` propagation

No test exercises what happens when `payment.charge()` raises `PaymentError`. The caller behavior is currently undefined (raw exception vs. structured result).

### `test_lock_release_uses_lua_script` is fragile

```python
processor._release_script.assert_called_once()
```

This asserts the script was called but does not verify it was called with the correct `keys` and `args`. A refactor that calls the script with wrong arguments would pass this test.

```python
# Better assertion
processor._release_script.assert_called_once_with(
    keys=[f"order:lock:order-001"], args=[ANY]
)
```

## Minor Notes

- `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` as class-level constants are fine, but consider making them constructor parameters to allow per-deployment tuning without code changes.
- The `charge_id` field on `OrderResult` is `Optional[str]` but is never `None` on the success path. Consider making it required when `processed=True` — a discriminated union or separate result types would make this contract explicit.
- `CHARGE_ENDPOINT` hardcoded as a class variable makes the client untestable against a real test server without monkeypatching. Inject it via constructor.
