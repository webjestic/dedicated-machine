Decision: Request Changes

Summary: The implementation has a critical architectural flaw that undermines the core at-most-once guarantee it claims to provide: the payment charge occurs *inside* the database transaction, meaning a gateway timeout on retry can produce an unrecorded external charge that the idempotency check will never catch. Additionally, the lock TTL of 60 seconds is dangerously short given that the payment client alone can consume up to 90 seconds (`MAX_RETRIES=3` × `REQUEST_TIMEOUT=30s`), creating a window where the lock expires mid-processing and a second worker acquires it and double-charges. These two issues combine to make the duplicate-charge prevention guarantee false under realistic failure conditions.

---

## Detailed Findings

### 1. CRITICAL — Lock TTL Shorter Than Maximum Payment Duration

**File:** `services/order_processor.py` / `services/payment_client.py`

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
REQUEST_TIMEOUT = 30   # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5      # seconds between retries
```

Worst-case payment duration: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still mid-charge, defeating the entire mutual exclusion guarantee. This is not a theoretical edge case — it is the exact scenario that happens when the payment gateway is degraded, which is precisely when retries are triggered.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at minimum `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + safety_margin` (≥120s recommended), or implement lock extension (a periodic Lua-based TTL refresh while the worker is alive). The TTL and payment client constants must be co-owned or derived from a shared config so they cannot drift independently.

### 2. CRITICAL — Charge Occurs Inside DB Transaction, Breaking at-most-once Guarantee

**File:** `services/order_processor.py`

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External side effect inside transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. If the payment gateway returns success but the subsequent `record_charge` raises (DB constraint, network partition, OOM), the DB transaction rolls back. The charge has already settled externally. On the next attempt, `get_charge` returns `None`, so the idempotency check passes, and the customer is charged a second time.

The correct pattern is to separate the external call from the DB transaction:

```python
# Charge OUTSIDE the transaction — it is an external, non-rollbackable side effect
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically AFTER the charge succeeds
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This still does not fully solve the crash-between-charge-and-record problem, but it makes the failure mode explicit and recoverable: a crashed worker leaves a charge with no DB record, which can be reconciled via the payment gateway's webhook or a reconciliation job. Hiding the charge inside a rollbackable transaction creates silent double-charges with no audit trail.

### 3. HIGH — `payment_token` Logged Indirectly via Pydantic Validation Error

**File:** `services/order_processor.py`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in the log output. Payment tokens are PCI-DSS sensitive material and must never appear in logs.

**Required fix:**

```python
except ValidationError as e:
    # Log field names and error types only — never field values
    safe_errors = [{"field": err["loc"], "type": err["type"]} for err in e.errors()]
    logger.warning("Invalid order request fields=%s", safe_errors)
```

### 4. HIGH — Timeout Exceptions Not Retried in Payment Client

**File:** `services/payment_client.py`

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the request outcome is *unknown* — the gateway may have processed the charge and the response was lost in transit. Retrying blindly on timeout is exactly the scenario that produces duplicate charges at the gateway level. This requires idempotency keys sent with each request so the gateway can deduplicate retries.

**Required fix:** Generate a per-order idempotency key (e.g., derived from `order_id`) before the retry loop and include it in every request to the gateway:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 5. MEDIUM — `response.json()["charge_id"]` Raises Unhandled `KeyError`

**File:** `services/payment_client.py`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body, this raises `KeyError`, which propagates uncaught through `OrderProcessor.process()` as an unhandled exception. The order ends up in an indeterminate state: the charge may have succeeded but no `OrderResult` is returned.

**Required fix:**

```python
body = response.json()
charge_id = body.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned success but missing charge_id: {body}")
```

### 6. MEDIUM — Unhandled Exceptions Inside Lock Context Silently Suppress Errors

**File:** `services/order_processor.py`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

Any exception from `self.payment.charge()` or `self.db.*` that is not `LockError` propagates out of `process()` entirely uncaught. The caller receives an unhandled exception rather than an `OrderResult`. Given that this is a service boundary, all non-lock exceptions should be caught and returned as structured failures, or at minimum documented as intentional raises.

### 7. LOW — Tests Do Not Cover the Lock Expiry / Double-Charge Scenario

**File:** `tests/test_order_processor.py`

There is no test that simulates a slow payment causing lock expiry and a concurrent second worker succeeding. This is the primary failure mode the system is designed to prevent. The test suite validates the happy path and basic contention but does not stress the timing invariants.

---

### 8. LOW — `customer_id` Has No Pattern Constraint

**File:** `services/order_processor.py`

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'` but `customer_id` does not. If `customer_id` is used in DB queries or log aggregation, this is an inconsistency that could allow injection characters through validation.
