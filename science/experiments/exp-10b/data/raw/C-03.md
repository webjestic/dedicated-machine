Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured, and the Lua-based token-verified release, Pydantic validation, and retry patterns are all solid work. However, there is a critical atomicity flaw that undermines the entire at-most-once guarantee the PR claims to provide, plus several secondary reliability and security issues that must be addressed before this can be approved.

---

## Critical Issue

### Payment call sits outside the database transaction — the at-most-once guarantee is broken

The PR summary states: *"charges the payment method and records the transaction within a single atomic database transaction."* The checklist confirms: *"Charge and database record within a single atomic transaction."* Neither is true.

```python
# services/order_processor.py, inside process()
with self.db.transaction():
    charge_id = self.payment.charge(        # ← external HTTP call inside DB transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Problem 1 — The charge is not atomic with the DB record.** A database transaction provides atomicity over database operations. An external HTTP call to the payment gateway is not a database operation. If the process crashes after `payment.charge()` returns a `charge_id` but before `db.record_charge()` commits, the customer has been charged and there is no record of it. The DB transaction does not roll back the charge. This is the exact failure mode the PR claims to prevent.

**Problem 2 — Holding a DB connection open during an external HTTP call.** `PaymentClient.REQUEST_TIMEOUT` is 30 seconds, with up to 3 retries and 5-second backoffs — a worst-case of ~100 seconds. Wrapping that in a DB transaction holds a connection (and potentially row locks) for the full duration. Under any real load this will exhaust the connection pool.

**Recommended fix:** Charge first, outside the transaction, then open the transaction only to record the result. Pair this with a compensating refund path if the DB write fails, and ensure the idempotency check is re-evaluated after the charge if there is any possibility of a race on the record step.

```python
# Preferred structure
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)
try:
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Attempt compensating refund; log for manual reconciliation if refund also fails
    ...
    raise
```

## High Severity

### Timeout on payment gateway request is not enforced during retries

`PaymentClient` catches `requests.exceptions.Timeout` and retries. However, `requests.post(..., timeout=30)` sets a per-attempt timeout, which is correct. What is missing is any ceiling on total elapsed time. With `MAX_RETRIES=3` and `RETRY_BACKOFF=5`, plus 30-second timeouts per attempt, the worst-case wall time is approximately 105 seconds. The Redis lock TTL is 60 seconds (`ORDER_LOCK_TTL = 60`). The payment client can outlive the lock, meaning another worker can acquire the lock and re-enter the idempotency check while a charge is still in flight. The lock TTL must be greater than the maximum possible payment client duration, or the payment client must be bounded to fit within the lock TTL.

### `payment_token` is logged at INFO level on success

```python
# payment_client.py
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This specific line is fine, but `order_processor.py` logs `req.order_id` and `charge_id` — review whether any downstream log aggregator or exception handler could surface `payment_token`. More importantly, `payment_token` should be explicitly excluded from any structured log serialization of `OrderRequest`. Consider a `__repr__` override or a redacted field on the model. For a government contracting context this is a compliance concern, not just a best practice.

## Medium Severity

### `PaymentError` is silently swallowed inside `process()`

`process()` catches `LockError` explicitly but has no handler for `PaymentError` or any other exception raised inside the lock context. An unhandled `PaymentError` will propagate out of `process()` as an uncaught exception, bypassing the `OrderResult` return contract the method signature promises. Callers expecting an `OrderResult` will get a crash instead. Either catch `PaymentError` and return a structured `OrderResult`, or document explicitly that callers must handle it — but the current code does neither.

### Lock retry backoff is linear and very short

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Under high concurrency, linear half-second backoffs with only 3 attempts will cause most workers to give up quickly and return a failure to the caller. Consider exponential backoff with jitter. More importantly, `LOCK_RETRY_ATTEMPTS` and backoff values are hardcoded class attributes with no path for per-deployment configuration. These should be injectable via constructor or environment config.

### `db.get_charge()` is not covered by the lock scope in tests

The test `test_process_already_completed` mocks `get_charge` to return an existing charge, which is correct. However, there is no test covering the race window between `get_charge` returning `None` and the lock being held — specifically, no test verifies that a second concurrent call with the same `order_id` while the first is inside the lock will block or fail gracefully. This is the primary scenario the lock is supposed to protect and it is not tested.

## Low Severity / Nits

- `OrderResult.order_id` is `Optional[str]` but is omitted in the `LockError` return path and the validation failure path. Callers cannot distinguish "lock contention on order X" from "lock contention on an unknown order" without parsing the message string. Consider always populating `order_id` when it is known.

- `CHARGE_ENDPOINT` is a hardcoded string in `PaymentClient`. For an internal service URL this should come from environment configuration or a settings object, not a class-level constant. This will cause friction in test, staging, and production environment management.

- The `test_lock_release_uses_lua_script` test asserts `redis_client.delete.assert_not_called()` as a proxy for "Lua script was used." This is a negative assertion on an implementation detail. It would be more robust to assert the script was called with the correct `keys` and `args` values, which would also catch a regression where the script is called with the wrong token.
