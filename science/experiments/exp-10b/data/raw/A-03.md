**Reviewer:** Distributed Systems / Security Review
**Branch:** `feature/order-processor`

## Detailed Findings

### 🔴 Critical — Charge-Outside-Transaction Violates At-Most-Once Guarantee

This is the most severe architectural flaw in the submission, and it directly contradicts the PR's own stated guarantee.

```python
# From order_processor.py
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External side-effect INSIDE DB transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is a non-transactional, external HTTP side-effect placed inside a database transaction block. This is a textbook violation and introduces two distinct failure modes:

**Scenario A — Double Charge:**
1. Worker A acquires lock, enters transaction, calls `payment.charge()` → succeeds, returns `charge_id`
2. Worker A's DB transaction fails or times out on `record_charge()` → DB rolls back
3. The lock is released (charge was never recorded)
4. Worker B acquires lock, `get_charge()` returns `None` (nothing was recorded), charges again
5. Customer is billed twice

**Scenario B — Phantom Charge:**
1. Worker A charges successfully, DB transaction commits
2. Lock TTL expires mid-execution (60s is tight if payment gateway takes 30s × 3 retries = 90s possible)
3. Worker B acquires lock during Worker A's still-running execution
4. Both workers believe they are the sole processor

The correct pattern is **charge-then-record with idempotency at the gateway level**, or an **outbox/saga pattern** where the charge intent is durably persisted before the external call is made. The payment call must live *outside* the DB transaction, with the DB transaction used only for the record step:

```python
# Correct structure
existing = self.db.get_charge(req.order_id)
if existing:
    return already_processed_result

charge_id = self.payment.charge(...)          # External call OUTSIDE DB transaction

with self.db.transaction():                   # Only DB writes inside transaction
    # Re-check idempotency inside transaction to close the TOCTOU gap
    if self.db.get_charge(req.order_id):
        # Charge already recorded by a concurrent worker — must void/refund charge_id
        return already_processed_result
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Even this requires a compensating action if the DB commit fails after a successful charge.

### 🔴 Critical — Lock TTL Can Expire Before Payment Retries Complete

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment duration: `(30s timeout × 3 attempts) + (5s backoff × 2 gaps)` = **100 seconds**.

The lock TTL is 60 seconds. The lock can expire and be acquired by another worker while the original worker is still mid-execution, mid-retry. The atomic token-verified Lua release is a good implementation detail, but it cannot protect against the window where the lock has already expired and been re-acquired. Worker A will complete, call the Lua release, and silently release Worker B's lock (`get` will return Worker B's token, so the script returns `0` — but Worker A's charge may have already gone through). This is a silent failure with no alerting.

Either the lock TTL must be set to a value strictly greater than the maximum possible payment duration with margin, or the lock must be extended (using a watchdog/renewal pattern) during long-running operations.

### 🔴 Critical — TOCTOU Race on Idempotency Check

```python
existing = self.db.get_charge(req.order_id)   # Read
if existing:
    return ...
# ... time passes ...
with self.db.transaction():
    charge_id = self.payment.charge(...)       # Write
```

The idempotency check (`get_charge`) and the charge execution are not atomic with respect to each other, even under the Redis lock. If the Redis lock expires between the check and the charge (see above), two workers can both pass the `get_charge` check and both proceed to charge. The lock is the only guard here, and it has a demonstrated expiry vulnerability. A second idempotency check *inside* the database transaction (using `SELECT FOR UPDATE` or an equivalent upsert with a unique constraint on `order_id`) is required as a defense-in-depth layer.

### 🟡 Major — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic `ValidationError` string representations include field values in their output. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in the log output. Payment tokens are sensitive credentials. The log statement should either log `e.errors()` with field names only (stripping values), or explicitly exclude the `payment_token` field from any logged representation.

### 🟡 Major — `PaymentClient` Has No Authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There are no authentication headers, no mTLS configuration, no API key injection. Even for an internal endpoint (`api.payments.internal`), this is unacceptable in a government contracting context. Any process on the internal network can call this endpoint as-is. The client must accept and inject credentials at construction time, and the credential source must be documented.

---

### 🟡 Major — Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded infrastructure URLs are a deployment and security concern. This must be injected via configuration or environment variable. Hardcoded values cannot be rotated, environment-scoped, or audited through standard configuration management.

### 🟡 Major — Unhandled `PaymentError` and Generic Exceptions in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # Can raise PaymentError
        self.db.record_charge(...)             # Can raise DB exceptions
```

`PaymentError` is not caught in `process()`. It will propagate as an unhandled exception to the caller. There is no structured `OrderResult` returned for payment failures, no logging of the failure with order context, and no guarantee that the lock is cleanly released with appropriate error metadata. The `contextmanager` `finally` block will release the lock, but the caller receives a raw exception rather than a structured result, which breaks the contract established by the return type annotation.

Similarly, database exceptions from `record_charge` or `mark_order_complete` are unhandled. A charge may succeed while the DB write fails, and the exception propagates with no compensating action documented or implemented.

---

### 🟡 Major — Lock Retry Backoff Is Deterministic (Thundering Herd)

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Fixed linear backoff means all contending workers retry at identical intervals. Under high concurrency, this creates synchronized retry storms. This must use jittered backoff (e.g., `random.uniform(0, 0.5 * (attempt + 1))`).

---

### 🟠 Minor — `requests.exceptions.ConnectionError` Not Handled

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
```

`ConnectionError` (DNS failure, connection refused, network partition) is not caught and will propagate as an unhandled exception rather than being treated as a transient failure eligible for retry. This is a gap in the stated "retry on transient failures" guarantee.

---

### 🟠 Minor — Test Coverage Does Not Exercise the Critical Failure Paths

The test suite covers the happy path and basic contention, but entirely omits:

- Payment success followed by DB transaction failure (the double-charge scenario)
- Lock TTL expiry mid-execution
- `PaymentError` propagation behavior
- `ConnectionError` from the payment gateway
- Behavior when `record_charge` raises after a successful charge

These are precisely the failure modes that matter most for a payment processor. Tests that only cover success and simple contention provide false confidence against the real risk surface.

---

### 🟠 Minor — `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` Are Class-Level Constants

These values are not injectable at construction time, making it impossible to tune them per deployment environment or test them with different values without monkeypatching. They should be constructor parameters with defaults.

---

### 🟠 Minor — `charge_id` Leaks into `OrderResult` on Already-Processed Path

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
    )
```

This is actually correct — `charge_id` is not included. However, the existing charge's `charge_id` is not returned either, which means the caller has no way to retrieve the idempotent result. For a true idempotent API, the already-processed result should return the original `charge_id` from `existing` so callers can confirm the outcome without a separate lookup.
