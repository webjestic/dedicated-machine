Decision: Request Changes

Summary: The implementation has solid bones — the Lua-based token-verified lock release, Pydantic input validation, and idempotency check are all correct in principle. However, there is a critical atomicity flaw at the intersection of the payment call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that must be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this does not make it atomic

This is the most serious architectural flaw in the PR.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external HTTP call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge."* That claim is false. A database transaction provides atomicity over database operations only. It has no authority over an external payment gateway call. The scenarios that break this:

- **Gateway charges successfully, then the process crashes before `record_charge` commits.** The DB transaction rolls back. The charge is real. It is now unrecorded. The idempotency check on retry will find no existing charge and will charge again.
- **Gateway charges successfully, `record_charge` succeeds, but the DB transaction fails to commit (deadlock, connection drop, etc.).** Same outcome.
- **The gateway times out.** The charge may or may not have gone through. The code treats a timeout as a retriable transient failure and will retry — potentially double-charging.

The correct pattern for at-most-once semantics with an external payment system is:

1. Write a **pending charge record** to the database *before* calling the gateway, inside a transaction, with a unique idempotency key.
2. Call the gateway with that idempotency key so the gateway itself deduplicates.
3. Update the record to confirmed on success, or failed on permanent error.
4. On retry, detect the pending record and use the same idempotency key to resume rather than issuing a new charge.

The Redis lock alone cannot close this gap. The lock TTL is 60 seconds, but `MAX_RETRIES=3` with `RETRY_BACKOFF=5s` and `REQUEST_TIMEOUT=30s` means a worst-case payment call can take `3 × (30 + 5) = 105 seconds` — exceeding the lock TTL. The lock can expire and another worker can enter while the first is still mid-retry.

---

### 2. Lock TTL can expire before payment completes

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Worst-case payment duration: `3 × (30 + 5) − 5 = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. The lock TTL must exceed the maximum possible critical section duration, or the lock must be extended (fenced) during execution.

---

### 3. Timeout responses from the gateway are retried without idempotency — double-charge risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded on the gateway side. Retrying without sending a stable idempotency key to the gateway can result in two successful charges for one order. The payment client must send a per-order idempotency key (e.g., derived from `order_id`) in every request to the gateway, so the gateway deduplicates on its end.

## Significant Issues

### 4. Unhandled exceptions inside the lock context leak the lock correctly, but payment exceptions are not caught

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

If `self.payment.charge()` raises `PaymentError`, it propagates out of `process()` entirely — not caught, not returned as a structured `OrderResult`. The caller receives an unhandled exception. The lock is released correctly by the `finally` block, but the caller contract (always returns `OrderResult`) is violated. `PaymentError` and general `Exception` must be caught and mapped to a failed `OrderResult`.

### 5. `db.get_charge` is called outside the database transaction — TOCTOU window

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← outside transaction
    if existing:
        return ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

The idempotency read and the subsequent write are not in the same transaction. With the Redis lock held this is partially mitigated, but if the lock expires mid-execution (see issue 2), two workers can both read `existing = None` and both proceed to charge. The read should be inside the same transaction as the write, or the write should use a unique constraint on `order_id` as a hard database-level guard.

### 6. `payment_token` is logged indirectly and passed in plain text through the call stack

`payment_token` is accepted as a plain string, stored in the validated model, and passed to `payment.charge()`. Ensure it is never emitted to logs. Currently the `OrderRequest` repr (which Pydantic will produce on `logger.warning("Invalid order request: %s", e)`) may not include it, but this should be explicitly verified. Consider a custom `SecretStr` type from Pydantic to prevent accidental logging.

## Minor Issues

### 7. Lock retry backoff is linear, not exponential, and very short

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

For a 60-second lock TTL, retrying at 0.5–1.5 second intervals is unlikely to succeed if another worker is mid-payment. This wastes cycles. Exponential backoff with jitter is the standard approach.

### 8. Tests do not cover the critical failure paths

The test suite is missing coverage for:
- Payment gateway timeout followed by retry (and whether idempotency keys are sent).
- Lock expiry during payment processing.
- `PaymentError` propagation (currently unhandled — would cause test failures against the real interface).
- Database transaction rollback after a successful charge.

### 9. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This should be injected via configuration, not hardcoded in the class body. Hardcoded endpoints make environment promotion (staging vs. production) error-prone and untestable without monkeypatching.
