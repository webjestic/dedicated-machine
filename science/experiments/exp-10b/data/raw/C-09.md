Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw in the payment flow: `payment.charge()` is called *inside* the database transaction, meaning a successful charge that is followed by a DB commit failure will produce an unrecorded, real charge with no recovery path. The comment in the code even claims this is safe ("crash between charge and record cannot produce an unrecorded charge"), which is incorrect and dangerous. Additionally, there are several secondary issues that need to be addressed before this can be approved.

---

### Critical: Payment charge inside DB transaction does not provide atomicity

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect, not rollbackable
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The DB transaction only provides atomicity over DB operations. The payment gateway call is an external, non-rollbackable side effect. If `record_charge` or `mark_order_complete` raise after `charge()` succeeds, the charge has been collected but is invisible to the system. The next call will pass the idempotency check and charge again.

**Required fix:** Charge first, outside the transaction, then open the transaction only for the DB writes. On any DB failure after a successful charge, log a critical alert with the `charge_id` for manual reconciliation and do not allow a retry to re-charge.

```python
# Correct pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "CHARGE_UNRECORDED order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

### High: Timeout on payment gateway is ambiguous — retry will double-charge

In `payment_client.py`, a `Timeout` exception is retried unconditionally:

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have been processed by the gateway. Retrying blindly can result in a duplicate charge. The payment client must either use an idempotency key per logical charge attempt (not per retry), or the caller must be informed that the result is unknown and must not retry without confirmation.

```python
# Pass a stable idempotency key tied to the order, not the attempt
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # e.g., order_id
    timeout=self.REQUEST_TIMEOUT,
)
```

### Medium: `PaymentError` is not caught in `process()`

`process()` catches `LockError` but lets `PaymentError` propagate uncaught to the caller. This will surface as an unhandled exception rather than a structured `OrderResult`. Either catch it explicitly or document that callers are expected to handle it — but the current code structure implies all outcomes are returned as `OrderResult`.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

---

### Medium: Lock TTL does not account for retry duration

```python
ORDER_LOCK_TTL = 60        # seconds
# ...
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock TTL of 60 seconds can expire before the payment completes, allowing a second worker to acquire the lock and enter the idempotency check while the first worker is still mid-charge. The TTL must be set to at least `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + margin`.

---

### Medium: `payment_token` logged indirectly via `charge_id` — verify no token leakage

The `payment_token` field is accepted in `OrderRequest` and passed to `payment.charge()`. Confirm that no log line anywhere serializes the full `OrderRequest` or `payment_token` value. The current code appears clean on this, but the `ValidationError` log line at the top of `process()` warrants a second look — Pydantic validation errors can include field values in their string representation.

```python
# Potentially leaks payment_token value in validation error output
logger.warning("Invalid order request: %s", e)
```

Replace with a sanitized message or explicitly exclude sensitive fields from the error output.

---

### Low: Tests do not cover the post-charge DB failure scenario

Given that the critical bug above involves a post-charge DB failure, there is no test that exercises this path. Once the fix is in place, a test must be added:

```python
def test_db_failure_after_charge_logs_critical_and_raises(processor, db, payment):
    db.record_charge.side_effect = Exception("DB unavailable")
    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")
    # charge_id should have been logged at CRITICAL level
```

---

### Low: `customer_id` has no pattern constraint

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` accepts arbitrary strings up to 64 characters. If `customer_id` is used in any DB query or log aggregation downstream, it should have the same or equivalent constraint applied.
