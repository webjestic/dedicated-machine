Decision: Request Changes
Summary: The pull request implements a conceptually sound distributed locking mechanism with a heartbeat, which is appropriate for the problem space. However, a critical architectural flaw exists in the heartbeat implementation: it uses instance-level state, making the service class non-reentrant and unsafe for concurrent use from multiple threads within the same process. This flaw could lead to premature lock releases and data corruption under the very high-load scenarios this service is designed to handle.

---

### High-Priority Architectural Concern

**1. Heartbeat State Management Prevents Concurrent Use**

The current implementation of the heartbeat mechanism stores its state (`_heartbeat_active`, `_heartbeat_thread`) directly on the `InventoryService` instance. This creates a race condition if a single `InventoryService` instance is used to reserve different items concurrently (e.g., in a multi-threaded web server environment).

**Scenario:**
1.  Thread 1 calls `reserve_item("item-A")`. It acquires a lock and starts a heartbeat thread, storing it in `self._heartbeat_thread`.
2.  Before Thread 1 finishes, Thread 2 calls `reserve_item("item-B")` on the *same service instance*. It acquires a different lock and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread`.
3.  The original heartbeat thread for "item-A" is now orphaned and will never be stopped by the `_stop_heartbeat` method.
4.  When Thread 1's `_acquire_lock` context exits, its `finally` block will call `_stop_heartbeat()`, which will incorrectly stop the heartbeat for "item-B".

This will cause the lock for "item-B" to expire prematurely, defeating the entire purpose of the heartbeat and re-introducing the risk of data corruption.

**Recommendation:**
The state for the heartbeat (the thread object and the active flag) must be managed within the scope of the `_acquire_lock` context manager itself, not at the instance level. The `_acquire_lock` method should create, start, and ultimately stop the heartbeat thread for its specific execution context.

Example of a revised structure:

```python
# In services/inventory.py

@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    # ... (lock acquisition logic) ...

    if not acquired:
        raise RuntimeError(...)

    # --- Heartbeat state is now local to the context ---
    heartbeat_active = {"flag": True}
    
    def _beat(stop_signal: dict) -> None:
        while stop_signal["flag"]:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if stop_signal["flag"]:
                # ... (logic to extend lock if owned) ...

    heartbeat_thread = threading.Thread(target=_beat, args=(heartbeat_active,), daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        # --- Stop the specific heartbeat thread for this context ---
        heartbeat_active["flag"] = False
        heartbeat_thread.join(timeout=1) # Optional: wait briefly for thread to exit
        
        # ... (lock release logic) ...
```

### Security & Robustness Suggestion

**2. Atomicity of Lock Release**

The current lock release logic is:
1.  `GET` the lock key from Redis.
2.  Check if the token matches.
3.  `DELETE` the lock key.

This is a "check-then-act" pattern. While the window for a race condition is extremely small, it is not truly atomic. A more robust and standard approach is to use a Lua script executed by Redis's `EVAL` command. This guarantees that the check and delete operations are performed atomically on the Redis server, eliminating any possibility of a race condition during lock release.

**Recommendation:**
Implement the lock release using a Lua script.

```python
# In services/inventory.py, inside _acquire_lock's finally block

# LUA script for atomic delete
LUA_UNLOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
# ...
        finally:
            # ... stop heartbeat ...
            try:
                # Use a Lua script for atomic check-and-delete
                self.redis.eval(LUA_UNLOCK_SCRIPT, 1, lock_key, token)
                logger.info("Lock released key=%s", lock_key)
            except redis.exceptions.RedisError as e:
                logger.error("Failed to release lock with Lua script key=%s: %s", lock_key, e)

```
*Note: The original code is not incorrect here, but using a Lua script is best practice for atomicity and is expected in a high-stakes environment like ours.*

### Testing Gap

The unit tests correctly mock out the heartbeat functionality to simplify testing the business logic. However, this means the identified architectural flaw in the heartbeat was not caught. Once the heartbeat logic is corrected, an integration test that uses a real Redis instance and does not mock `time.sleep` should be added to verify that the lock TTL is correctly extended under a simulated workload.
