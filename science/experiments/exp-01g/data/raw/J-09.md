Decision: Request Changes
Summary: The pull request introduces a well-intentioned distributed locking mechanism with a heartbeat, but it contains critical architectural flaws that make it unsafe for concurrent use. The service instance is not thread-safe due to shared state for the heartbeat thread, which will cause lock expirations and race conditions under concurrent requests. Additionally, the tests explicitly mock out the entire heartbeat logic, failing to catch these severe defects.

---

### Detailed Review

This is a solid attempt at a complex problem, and I appreciate the attention to detail with the retry logic, ownership checks, and use of a context manager. However, there are two critical flaws in the current implementation of the heartbeat thread that must be addressed before this can be approved.

#### 1. CRITICAL: Service Instance is Not Thread-Safe

The `InventoryService` class uses instance attributes (`self._heartbeat_active`, `self._heartbeat_thread`) to manage the state of a single heartbeat thread. If a single instance of `InventoryService` is used by multiple worker threads (which is standard in any web framework like Flask or FastAPI), this will lead to a race condition.

**Scenario:**

1.  **Request A** calls `reserve_item("item-1")`. It acquires a lock and starts a heartbeat thread via `_start_heartbeat`, which populates `self._heartbeat_thread` and sets `self._heartbeat_active = True`. This thread is now correctly extending the lock for `item-1`.
2.  **Request B** calls `reserve_item("item-2")` on the *same service instance* before Request A completes. It acquires a lock for `item-2` and also calls `_start_heartbeat`.
3.  This second call **overwrites** `self._heartbeat_thread` with a new thread that extends the lock for `item-2`.
4.  The original heartbeat thread for `item-1` is now orphaned and garbage collected. The lock for `item-1` is no longer being extended and will expire after `LOCK_TTL` seconds, even if Request A's database operation is still in progress.

This will cause the exact data corruption (e.g., double-selling an item) that this locking mechanism is meant to prevent.

**Recommendation:**
The state and lifecycle of the heartbeat thread must be scoped to a single lock acquisition, not to the service instance. The thread object should be created, managed, and stopped entirely within the `_acquire_lock` context manager as a local variable, not stored on `self`.

#### 2. CRITICAL: Incomplete Test Coverage Hides Flaws

The test suite explicitly disables the entire heartbeat mechanism by mocking out `_start_heartbeat` and `_stop_heartbeat`:

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    # Disable heartbeat thread in tests
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

This is a significant testing gap. The most complex and novel part of this service is the heartbeat logic, and it is completely untested. This is precisely why the critical thread-safety issue above was not caught. While unit testing threads can be complex, this component's correctness absolutely depends on it.

**Recommendation:**
The tests must be updated to validate the behavior of the heartbeat thread. You can achieve this by:
*   Allowing the thread to run in tests.
*   Using `time.sleep()` to simulate the passage of time.
*   Asserting that the mock `redis_client.expire` method was called by the background thread.

#### 3. Minor: Potential Race on Heartbeat Shutdown

In `_acquire_lock`'s `finally` block, `_stop_heartbeat()` is called, which sets `self._heartbeat_active = False`. The code then immediately proceeds to release the lock. The background thread, however, might still be in its `time.sleep()` interval.

While the ownership check (`if current.decode() == token`) prevents this thread from later deleting a lock acquired by another process, it's cleaner to ensure the thread has fully terminated before proceeding.

**Recommendation:**
After signaling the thread to stop, you should `join()` it to wait for its completion. This ensures a clean and predictable shutdown sequence before the lock is released. This should be part of the refactor to make the heartbeat management local to the `_acquire_lock` function.

---

This code cannot be deployed in its current state. Please refactor the heartbeat management to be thread-safe and add comprehensive tests that validate its behavior under concurrency. The pattern is promising, but the implementation details are critical for a system that handles inventory.
