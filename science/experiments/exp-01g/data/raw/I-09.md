Decision: Request Changes
Summary: This pull request introduces a sophisticated distributed locking mechanism, but it contains a critical race condition vulnerability. The design does not account for process pauses (e.g., stop-the-world GC), which can cause the lock to expire and lead to data corruption by allowing stale data to be written. The fix requires implementing optimistic concurrency control at the database layer to ensure data consistency.

Team,

Thank you for this submission. The implementation of the Redis distributed lock shows a strong understanding of common patterns, including the use of a unique ownership token, atomic acquisition (`SET NX EX`), and a heartbeat mechanism to handle long-running operations. These are all excellent practices and demonstrate careful thought.

However, I have identified a critical architectural flaw that prevents me from approving this PR in its current state. The system is vulnerable to a race condition caused by process pauses, which can lead to data corruption—specifically, overselling items during a flash sale.

### The Failure Scenario: The Stale Lock Holder

The heartbeat thread is designed to prevent the lock from expiring while the holding process is actively working. The unstated assumption is that the process remains scheduled and running. This assumption is violated during events like a stop-the-world garbage collection pause, a container/VM migration, or extreme OS-level scheduling delays.

Consider the following sequence of events:

1.  **Worker A** acquires the lock for `item_001` with a 30-second TTL.
2.  **Worker A** reads the current stock from the database (e.g., `stock = 10`).
3.  **Worker A's process is frozen** by a long GC pause (e.g., for 35 seconds). The heartbeat thread, being part of the same process, is also frozen and cannot extend the lock's TTL.
4.  After 30 seconds, the lock in Redis expires.
5.  **Worker B** now successfully acquires a new lock for `item_001`.
6.  **Worker B** reads stock (10), decrements it to 9, writes the new value to the database, and releases its lock. The inventory is now correctly 9.
7.  **Worker A's** GC pause finishes. Its code resumes executing from where it was paused, completely unaware that it lost the lock.
8.  **Worker A** is still operating on its stale, in-memory data: `current_stock = 10`. It calculates `new_stock = 9`.
9.  **Worker A** executes `self.db.update_stock("item_001", 9)`. This write succeeds, overwriting the correct value in the database. The stock level is now 9, but two separate orders have been processed. One unit of inventory has been sold twice.

The safety check in the `finally` block correctly prevents Worker A from deleting a lock it no longer owns, but the damage to the business data has already been done within the critical section.

### Recommended Solution: Fencing at the Data Layer

A distributed lock is necessary to prevent multiple workers from starting the read-modify-write cycle simultaneously, but it is not sufficient to guarantee correctness in the face of these timing hazards. The ultimate source of truth is the database, and the final write operation must be conditional to protect against stale state.

We must implement an optimistic concurrency control mechanism. The `update_stock` operation must verify that the state of the data has not changed since it was read.

The simplest way to achieve this is to include the value you read in the `WHERE` clause of your `UPDATE` statement.

**Instead of:**
```sql
UPDATE inventory SET stock = ? WHERE item_id = ?
```

**It must be:**
```sql
UPDATE inventory SET stock = <new_stock> WHERE item_id = ? AND stock = <current_stock_read_earlier>
```

Your application code must then check the number of rows affected by this `UPDATE` statement. If it returns 0, it means another process modified the stock in the interim. In this case, the reservation must fail, and the entire operation should be aborted. A more robust pattern involves using a dedicated version number (a fencing token) on the database row, but a conditional check on the stock level is a sufficient first step here.

### Action Items

1.  Modify the `db.update_stock` method and the corresponding application logic in `reserve_item` to implement optimistic concurrency control. The update must be conditional, and the code must handle the case where the update affects zero rows, treating it as a failure.
2.  Consider adding a test case that mocks the database driver to simulate a failed optimistic lock update (e.g., by having the mock `update` method return 0 rows affected) and verify that the `reserve_item` function correctly reports a failure.

This is a subtle but severe issue endemic to distributed systems. I am happy to discuss this further if anything is unclear.
