Decision: Request Changes
Summary: This pull request introduces a well-intentioned distributed locking mechanism for inventory management, but it contains a critical thread-safety issue in its current implementation. The service instance is not re-entrant, as the heartbeat thread state is shared across all calls, which will lead to race conditions under concurrent load. Additionally, the most complex part of this feature—the heartbeat logic—is not covered by tests, and input validation for the reservation quantity is missing.

---

### Detailed Review

Thank you for this contribution. The goal of creating a robust, distributed reservation system for flash sales is a critical one, and the use of a Redis lock with a heartbeat is the right architectural direction. However, I've identified a few issues that must be addressed before this can be approved.

#### 1. **Critical: `InventoryService` is not Thread-Safe**

The current implementation of the heartbeat mechanism makes the `InventoryService` instance stateful and not safe for concurrent use. The `_heartbeat_active` flag and `_heartbeat_thread` are instance-level attributes.

**Scenario:**
1.  Request A calls `reserve_item("item-123")`. It acquires a lock and starts a heartbeat thread, setting `self._heartbeat_thread` and `self._heartbeat_active = True`.
2.  Before Request A finishes, Request B calls `reserve_item("item-456")` on the *same service instance*. It acquires a different lock and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread`.
3.  When Request A's `_acquire_lock` context manager exits, it calls `self._stop_heartbeat()`, which sets `self._heartbeat_active = False`.
4.  This will prematurely stop the heartbeat loop for Request B's lock, as it's checking the same shared `self._heartbeat_active` flag. Request B's lock could then expire unexpectedly during a long database operation.

**Recommendation:**
The lifecycle of the heartbeat thread must be tied to a specific lock acquisition, not the service instance. A better approach would be to manage the thread's state within the `_acquire_lock` context.

A common pattern for this is using a `threading.Event` to signal termination:

```python
# In _acquire_lock
...
stop_event = threading.Event()
heartbeat_thread = threading.Thread(target=_beat, args=(lock_key, token, stop_event))
heartbeat_thread.start()

try:
    yield token
finally:
    stop_event.set() # Signal the thread to stop
    heartbeat_thread.join(timeout=1) # Optionally wait for it to exit cleanly
    # ... release lock logic ...

# The _beat function would be modified to accept the event
def _beat(lock_key, token, stop_event):
    while not stop_event.wait(self.HEARTBEAT_INTERVAL):
        # ... extend lock logic ...
```
This encapsulates the state and ensures each lock has its own independent heartbeat mechanism.

#### 2. **Major: Heartbeat Logic is Untested**

The test suite explicitly mocks out the heartbeat functionality:
```python
# tests/test_inventory.py
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```
The heartbeat is the most complex and novel part of this PR. It's essential that its behavior is tested. We need to verify that it actually extends the lock's TTL and that it stops correctly when the lock is released. While testing threads can be complex, it's not impossible. You could, for example, allow the thread to run once and assert that `redis.expire` was called.

#### 3. **Security/Logic: Missing Input Validation**

The `reserve_item` method does not validate the `quantity` parameter. A malicious or buggy client could pass `quantity=0` or `quantity=-1`.
*   `quantity=0`: This would likely succeed, creating an order for zero items, which is probably not desired.
*   `quantity=-1`: This would *increase* the stock (`new_stock = current_stock - (-1)`), which is a critical flaw that could be exploited to corrupt inventory data.

**Recommendation:**
Add a validation check at the beginning of the method:
```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if quantity <= 0:
        return ReservationResult(
            success=False,
            message="Quantity must be a positive integer."
        )
    ...
```

#### 4. **Minor: Misleading Checklist Item**

The PR summary checklist states "[x] Input validation via Pydantic". However, Pydantic is only used for the `ReservationResult` output model. The input parameters (`item_id`, `quantity`) are not validated using Pydantic. This should be corrected or the claim removed.

Please address these points, with the thread-safety issue being the highest priority. Once these are resolved, I'll be happy to review again.
