Decision: Request Changes
Summary: This pull request introduces a distributed locking mechanism for inventory reservation, which is a necessary feature for flash sales. However, the current implementation contains a critical architectural flaw that makes the service stateful and unsafe for concurrent use. Additionally, it is missing essential input validation and the test suite fails to cover the complex heartbeat logic, which must be addressed before this can be approved.

---

### Detailed Review

Thank you for this contribution. The use of a distributed lock with a heartbeat is the correct approach for this problem. However, there are several critical issues that prevent me from approving this PR in its current state.

#### 1. Critical Flaw: Stateful Service Design

The `InventoryService` class is designed in a stateful way that makes it non-reentrant and unsafe for any concurrent environment (e.g., a multi-threaded web server). The `_heartbeat_active` flag and `_heartbeat_thread` are stored as instance attributes.

**Problem Scenario:**
If a single `InventoryService` instance is used to handle two concurrent requests, they will conflict:
1.  Request A calls `reserve_item`, acquires a lock, and starts a heartbeat thread, setting `self._heartbeat_thread` and `self._heartbeat_active = True`.
2.  Request B calls `reserve_item` for a *different* item, acquires its lock, and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread` from Request A.
3.  When Request A finishes, its call to `_stop_heartbeat()` will set `self._heartbeat_active = False`, which will incorrectly terminate the heartbeat for Request B's lock. This could cause Request B's lock to expire prematurely, leading to data corruption.

**Required Action:**
The state related to a single lock acquisition (the heartbeat thread and its control flag) must be managed entirely within the scope of that acquisition, not on the shared service instance. Please refactor the `_acquire_lock` context manager to manage the heartbeat thread's lifecycle internally without modifying instance state (`self`). A `threading.Event` object created within the context manager would be a safer way to signal the thread to stop.

#### 2. Missing Input Validation

The `reserve_item` method does not validate its inputs. A malicious or malformed request could pass a `quantity` of `0` or a negative number, which would lead to incorrect stock calculations and data corruption (e.g., `current_stock - (-1)` would incorrectly increase stock).

**Required Action:**
Add validation at the beginning of the `reserve_item` method to ensure `quantity` is a positive integer.

```python
# services/inventory.py

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not isinstance(quantity, int) or quantity <= 0:
        return ReservationResult(
            success=False,
            message="Quantity must be a positive integer."
        )
    # ... rest of the method
```

#### 3. Incomplete Test Coverage

The test suite completely mocks out the heartbeat mechanism (`_start_heartbeat` and `_stop_heartbeat`). This means a critical and complex part of the implementation—the logic designed to prevent lock expiry under load—is entirely untested. This is a significant gap in test coverage.

**Required Action:**
Please add tests that validate the behavior of the heartbeat thread. This will likely require a more advanced test setup that allows you to inspect the calls made from the background thread (e.g., asserting that `redis.expire` is called periodically), but it is essential for ensuring the robustness of this service.

#### 4. Best Practice: Atomic Lock Release

The current lock release logic in the `finally` block is functionally correct but requires two separate network calls (`GET` and `DELETE`). This can be made more efficient and robust by using a Lua script to perform the check-and-delete operation atomically on the Redis server in a single round trip. This is the standard, recommended practice for releasing distributed locks and avoids potential race conditions in more complex scenarios.

**Recommendation:**
Refactor the lock release to use a Lua script via `redis-py`'s `eval` or `register_script` methods.

**Example Lua script:**
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

Please address the critical issues (1, 2, 3) before resubmitting this for review. The fourth point is a strong recommendation for improving robustness and performance.
