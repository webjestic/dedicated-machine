Decision: Request Changes
Summary: The pull request introduces a sound general approach for distributed inventory management using a Redis lock. However, the heartbeat mechanism contains a critical architectural flaw that introduces a race condition and risks resource exhaustion under load. The test suite also needs improvement, as it currently mocks the flawed heartbeat logic and lacks true concurrency tests.

---

### Detailed Review

This is a solid first draft for a critical piece of infrastructure. The use of a Redis distributed lock with a unique token for ownership is the correct pattern, and I appreciate the attention to detail with the context manager and exponential backoff.

However, I've identified a critical flaw in the heartbeat implementation that must be addressed before this can be approved.

#### 1. Critical: Race Condition in Heartbeat State

The `_heartbeat_active` and `_heartbeat_thread` attributes are stored on the service instance (`self`). If this `InventoryService` instance is shared and used to process concurrent requests for *different* items (a common pattern in web frameworks), a race condition will occur.

**Scenario:**

1.  Request A calls `reserve_item("item-1")`. It acquires a lock and starts a heartbeat thread. `self._heartbeat_active` is set to `True`.
2.  Request B calls `reserve_item("item-2")` on the *same service instance*. It acquires its own lock and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread`.
3.  Request B finishes its work quickly. Its `finally` block in `_acquire_lock` calls `_stop_heartbeat()`, setting `self._heartbeat_active` to `False`.
4.  Now, the heartbeat thread for Request A's lock will see `self._heartbeat_active` is `False` on its next check and will terminate prematurely. If Request A's database operation is still running, its lock will expire, allowing another process to acquire it and corrupt the inventory data.

**Recommendation:**

The state for the heartbeat must be scoped to the individual lock acquisition, not the service instance. A simple way to fix this is to manage the active state within the `_acquire_lock` context.

```python
# A potential fix to illustrate the point
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    # ... (lock acquisition logic) ...

    if not acquired:
        raise RuntimeError(...)

    # Use a mutable object to share state with the thread
    heartbeat_active = [True] 
    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)

    try:
        yield token
    finally:
        # Signal the thread to stop
        heartbeat_active[0] = False 
        if heartbeat_thread:
            heartbeat_thread.join(timeout=1) # Wait briefly for it to exit cleanly

        # ... (lock release logic) ...

# _start_heartbeat would need to be modified to accept and use the state object
def _start_heartbeat(self, lock_key: str, token: str, active_flag: list) -> threading.Thread:
    def _beat():
        while active_flag[0]:
            # ...
    
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return thread
```

#### 2. Architectural Concern: Thread Proliferation

The current implementation creates a new thread for every single lock acquisition. During a flash sale, this could mean creating hundreds or thousands of short-lived threads, leading to significant overhead and potential resource exhaustion (hitting thread limits, memory pressure, context-switching slowdown).

**Recommendation:**

Consider a more scalable approach. Instead of a dedicated thread per lock, a centralized "lock extender" service could manage all active locks. This could be a single background thread that periodically scans a Redis set of active lock keys and extends their TTLs. This would be far more efficient than the one-thread-per-lock model.

While a full refactor to that model may be out of scope for this PR, fixing the race condition from point #1 is non-negotiable.

#### 3. Incomplete Testing

The test suite is a good start, but it has two significant gaps:

1.  **Heartbeat Logic is Not Tested:** The tests explicitly mock `_start_heartbeat` and `_stop_heartbeat`. This is precisely why the race condition was not caught. The tests verify the business logic *assuming* the lock works, but they do not verify the lock implementation itself. The heartbeat mechanism must be tested.
2.  **Lack of Concurrency Tests:** The PR is intended to solve a concurrency problem, but there are no tests that simulate concurrent execution. A proper test would involve spinning up multiple threads that all try to call `reserve_item` simultaneously and then asserting that the final inventory count is correct and no race conditions occurred.

**Recommendation:**

*   Remove the mocks for the heartbeat methods and write tests that verify the lock's TTL is actually extended. You can use a library like `freezegun` to manipulate time and assert that `redis.expire` is called as expected.
*   Add a new test case using Python's `threading` or `concurrent.futures` module to simulate a flash sale scenario.

#### 4. Minor: Input Validation

The summary mentions "full input validation," but the `reserve_item` method does not validate that `quantity` is a positive integer. An order for `quantity=0` or `quantity=-1` should be rejected.

**Recommendation:**

Add a check at the beginning of `reserve_item`:
```python
if not isinstance(quantity, int) or quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be a positive integer.")
```

Please address these points, paying special attention to the heartbeat race condition and the test coverage. This is a critical service, and we need to ensure it is completely robust.
