Decision: Request Changes
Summary: This pull request introduces a sophisticated but critically flawed distributed locking mechanism. While the heartbeat renewal is a thoughtful addition for long-running operations, it fails to protect against process-wide pauses (e.g., stop-the-world GC), leading to a race condition where stale data can be written to the database, causing inventory corruption. The fix requires an additional safety check, such as an optimistic lock or fencing token, at the database write layer to reject these stale updates.

---

### Detailed Analysis

Thank you for this submission. The implementation of a distributed lock with a heartbeat renewal thread demonstrates a clear understanding of the challenges in managing state across a distributed system, particularly the problem of premature lock expiration during a valid, ongoing operation. The use of a unique token for lock ownership and exponential backoff for contention are both commendable best practices.

However, I have identified a severe architectural flaw that makes this implementation unsafe for production use. The system is vulnerable to a classic "zombie leader" or "stale write" failure mode, which can lead to data corruption, such as overselling flash sale items.

#### The Failure Scenario

The core issue is that the heartbeat thread provides no protection against process-wide pauses, such as a long garbage collection cycle, container resource throttling, or a VM migration event. If the entire process freezes, the heartbeat thread also freezes and cannot renew the lock's TTL in Redis.

Consider the following sequence of events:

1.  **Process A** acquires the lock for `item_001` (current stock: 10). It starts its heartbeat thread.
2.  **Process A** reads the current stock from the database: `current_stock = 10`.
3.  **Process A freezes** due to a stop-the-world GC pause for 31 seconds. The `LOCK_TTL` is 30 seconds.
4.  The lock in Redis expires. The heartbeat thread in Process A was also frozen and could not extend it.
5.  **Process B** now successfully acquires the lock for `item_001`.
6.  **Process B** reads stock (10), decrements it to 9, writes the new value to the database, and releases the lock. The database correctly reflects a stock of 9.
7.  **Process A unfreezes**. It is unaware that it lost the lock. It continues its operation from where it left off, holding a stale `current_stock` value of 10.
8.  **Process A** calculates `new_stock = 10 - 1 = 9` and calls `self.db.update_stock("item_001", 9)`. This write succeeds, overwriting the correct value with a stale one.
9.  The final stock in the database is 9, but two items were sold. The inventory is now incorrect by one unit.

#### Required Changes

The lock alone is insufficient. The atomicity of the read-modify-write cycle must be guaranteed at the data layer. The process holding the lock must be able to prove to the database that it is still the valid owner and is not operating on stale data.

The standard solution is to use a **fencing token** or an **optimistic concurrency control** mechanism.

Please modify the `db.update_stock` operation to be conditional. The update should only succeed if the stock level has not changed since it was read.

The database call should be modified from:
`self.db.update_stock(item_id, new_stock)`

To something like:
`rows_affected = self.db.update_stock(item_id, new_stock, current_stock)`

The underlying SQL query must then be changed to:
```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id AND stock = :expected_current_stock;
```

If `rows_affected` is 0, it means another process modified the stock after we read it. In this case, the entire `reserve_item` operation must fail, as our view of the world was stale. This change ensures that even if a process loses its lock and becomes a "zombie," its subsequent writes will be safely rejected by the database.

#### Additional Concerns

*   **Testing:** The heartbeat mechanism, a critical and complex component, is completely mocked out in the tests (`svc._start_heartbeat = MagicMock()`). While testing multithreaded code is difficult, this component should have dedicated tests to verify its behavior, such as ensuring `redis.expire` is called correctly over time.

This change is mandatory before I can approve this pull request. The current implementation poses a significant risk to data integrity.
