Decision: Request Changes
Summary: This pull request correctly identifies the need for a distributed lock with a heartbeat mechanism to handle flash sale inventory. However, the current implementation of the heartbeat thread introduces a critical thread-safety issue, making the service instance non-reentrant and unsafe for use in a multi-threaded web server environment. Additionally, the lock release mechanism is not atomic, and there is a lack of input validation on the reservation quantity.

---

### Detailed Review

Thank you for this contribution, dev-team. This is a critical piece of infrastructure for handling high-concurrency events like flash sales, and the overall approach is on the right track. The use of a distributed lock with a heartbeat is the correct pattern for this problem.

However, I've identified a few issues, one of which is critical, that must be addressed before this can be approved.

#### 1. **[Critical] `InventoryService` is not Thread-Safe**

The current implementation stores the heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) directly on the service instance (`self`). In a typical web application deployment (e.g., Gunicorn, uWSGI), a single instance of `InventoryService` will be shared across multiple worker threads, each handling a different incoming request.

This creates a race condition:

1.  **Thread A** calls `reserve_item()`, acquires a lock, and starts a heartbeat thread by setting `self._heartbeat_thread`.
2.  **Thread B** calls `reserve_item()` on the *same instance*, acquires a different lock, and starts its own heartbeat thread, overwriting `self._heartbeat_thread`. Thread A's heartbeat is now lost.
3.  When **Thread A** finishes, it calls `_stop_heartbeat()`, which sets `self._heartbeat_active = False`. This will prematurely terminate the heartbeat thread belonging to **Thread B**.

This will cause locks to expire unexpectedly under concurrent load, defeating the purpose of the heartbeat and leading to overselling.

**Recommendation:** The state for the heartbeat must not be stored on the shared `self`. It should be managed entirely within the scope of the `_acquire_lock` context manager. You can use a `threading.Event` object created within the context manager to signal the thread to stop.

Example of a safer pattern:

```python
# Inside _acquire_lock context manager
...
if acquired:
    stop_event = threading.Event()

    def _beat():
        while not stop_event.wait(self.HEARTBEAT_INTERVAL):
            # ... extend lock TTL ...

    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        # ... release lock logic ...
```

#### 2. **[Major] Non-Atomic Lock Release**

The `finally` block in `_acquire_lock` uses a `GET` followed by a `DELETE` to release the lock.

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

This is a classic race condition. It's possible for a process to be paused between the `GET` and the `DELETE`. While unlikely, in a high-contention scenario, a lock could expire, be acquired by another process, and then be incorrectly deleted by the original, delayed process.

**Recommendation:** The check-and-delete operation must be atomic. The standard way to achieve this in Redis is with a simple Lua script.

```lua
-- delifequal.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

You can then execute this script from your Python code, ensuring atomicity. Most Redis client libraries have a clean way to register and call Lua scripts.

#### 3. **[Minor] Missing Input Validation**

The `reserve_item` method does not validate the `quantity` parameter. A malicious or buggy client could pass `quantity=0` or `quantity=-1`. A negative quantity would lead to an *increase* in stock, which is a severe business logic flaw.

**Recommendation:** Add a validation check at the beginning of the `reserve_item` method.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if quantity <= 0:
        return ReservationResult(
            success=False,
            message="Quantity must be a positive integer."
        )
    # ... rest of the logic
```

#### 4. **[Code Style] Untested Heartbeat Logic**

The tests explicitly mock out `_start_heartbeat` and `_stop_heartbeat`. While I understand that testing multi-threaded code is complex, this means a core component of this feature's reliability is completely untested.

**Recommendation:** This is a lower-priority concern, but for future iterations, consider designing the heartbeat mechanism in a way that is more amenable to testing. For now, please focus on the critical thread-safety issue first.

Please address these points, paying special attention to the thread-safety of the service instance. I'll be ready to review again once the changes are pushed.
