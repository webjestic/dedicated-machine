Decision: Request Changes
Summary: This pull request introduces a well-intentioned distributed locking mechanism with a heartbeat, but the current implementation has critical architectural flaws that make it unsuitable for a high-concurrency environment. The heartbeat thread management introduces severe race conditions and will not scale under the load of a flash sale. Additionally, the code is missing crucial input validation, which presents a business logic vulnerability.

---

### Senior Engineer Review

Dev Team,

Thank you for putting this together. Tackling distributed inventory for a flash sale is a critical and challenging problem. The use of a Redis lock with a unique token for ownership is the right foundational pattern, and I appreciate the attention to detail in the retry logic and the context manager pattern.

However, I've identified several critical issues that must be addressed before this can be approved. The current implementation, particularly the heartbeat mechanism, would likely fail catastrophically under the exact "flash sale" conditions it's designed for.

Here are my findings:

#### 1. **Critical Flaw: Per-Request Thread Spawning**

The `_start_heartbeat` method creates a new thread for every single lock acquisition. During a flash sale, we could have thousands of concurrent requests attempting to reserve an item. Spawning thousands of OS threads will lead to thread-thrashing, excessive context switching, and will quickly exhaust system resources, bringing the service down. This design is not scalable.

#### 2. **Critical Flaw: Instance-Level State Race Condition**

The `_heartbeat_active` flag and `_heartbeat_thread` are stored as instance variables on the `InventoryService` object (`self`). In any production web framework (like Flask, FastAPI, etc.), this service is likely to be instantiated as a singleton and shared across many concurrent requests.

This creates a severe race condition:
1.  Request A calls `reserve_item()`, acquires a lock, and starts `self._heartbeat_thread`.
2.  Request B calls `reserve_item()` for a different item, acquires its lock, and *overwrites* `self._heartbeat_thread` from Request A.
3.  When Request A finishes, its `_stop_heartbeat()` call will set `self._heartbeat_active = False`, which will terminate the heartbeat thread belonging to Request B's lock.

This will cause legitimate, long-running operations to lose their locks prematurely.

#### 3. **Major Flaw: Non-Atomic Lock Renewal**

The heartbeat's `_beat` function performs a non-atomic `GET` followed by an `EXPIRE`.

```python
# Non-atomic and dangerous
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

There is a window between the `GET` and the `EXPIRE` where the lock could be released by the main thread and re-acquired by another process. The heartbeat would then incorrectly extend the TTL of a lock that no longer belongs to it, causing the new owner to be locked out for an extended period.

**Recommendation:** This check-and-extend operation *must* be atomic. This is a textbook use case for a simple Redis Lua script.

```lua
-- LUA script for atomic extension
if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("expire", KEYS[1], ARGV[2])
else
  return 0
end
```

#### 4. **Vulnerability: Missing Input Validation**

The `reserve_item` method does not validate the `quantity` argument. A malicious or buggy client could pass `quantity=0` or `quantity=-1`.
*   If `quantity=-1`, the code would subtract a negative number, *increasing* the stock (`new_stock = current_stock - (-1)`). This is a critical business logic vulnerability that could be used to artificially inflate inventory.
*   If `quantity=0`, it would successfully create an order for nothing, which is likely an undesirable state.

**Recommendation:** Add validation at the beginning of the method to ensure `quantity` is a positive integer (`quantity > 0`).

#### 5. **Concern: Incomplete Testing**

The tests mock out the entire heartbeat mechanism (`_start_heartbeat` and `_stop_heartbeat`). This means the most complex and fragile part of this new feature has zero test coverage. The passing tests provide a false sense of security, as they do not exercise the logic that is most likely to fail.

### Path Forward

1.  **Re-architect the Heartbeat:** The "thread-per-request" model must be replaced. I would strongly suggest first re-evaluating if the heartbeat is necessary at all. A sufficiently long `LOCK_TTL` (e.g., 60 seconds) combined with fast database operations is often simpler and more robust. If a heartbeat is non-negotiable, it needs a proper implementation, such as a single, shared background thread that manages a dictionary of all active locks and their tokens.

2.  **Fix Atomicity:** Implement the lock extension using a Lua script to prevent race conditions.

3.  **Add Validation:** Add strict validation for all inputs, especially `quantity`.

4.  **Write Real Tests:** The new implementation for the locking and heartbeat mechanism must be thoroughly tested. This might involve more complex integration tests that allow threads to run for a short time to verify their behavior.

Due to the severity of these issues, I cannot approve this PR. Please address the points above. My preference is to remove the heartbeat complexity entirely unless there's a compelling, proven reason that our database operations will take longer than a reasonable TTL.
