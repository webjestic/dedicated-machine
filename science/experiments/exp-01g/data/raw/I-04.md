Decision: Request Changes
Summary: This pull request introduces a sophisticated but flawed distributed locking mechanism. While the use of a heartbeat thread is a clever attempt to handle long-running operations, it fails to protect against process-level pauses (e.g., stop-the-world garbage collection), creating a critical race condition that can lead to data corruption. The atomicity of the read-modify-write cycle is not guaranteed, and an additional safety mechanism at the database layer is required before this can be approved.

---

Team,

I have completed my review of the proposed distributed inventory reservation service. The implementation shows a solid understanding of the basic principles of distributed locking using Redis, including the atomic `SET NX EX` pattern, ownership tokens, and safe release logic. The heartbeat concept is also a thoughtful addition for handling expected-to-be-long operations.

However, I am flagging a critical, non-obvious architectural flaw in this design that compromises the very data integrity it aims to protect.

### The "Paused Process" Race Condition

The core assumption here is that the heartbeat thread will reliably extend the lock's TTL. This assumption is violated if the entire worker process is paused. This can happen for various reasons in a virtualized or containerized environment, with the most common culprit being a stop-the-world Garbage Collection (GC) pause.

Consider the following failure scenario:

1.  **T=0s:** Worker A acquires the lock (`inventory:lock:item_001`) with a 30-second TTL. It begins its critical section.
2.  **T=1s:** Worker A successfully reads the current stock from the database (`current_stock = 10`).
3.  **T=2s:** A significant GC event occurs, pausing **all threads** in the Worker A process, including the main thread and your heartbeat thread.
4.  **T=30s:** The lock's TTL in Redis expires. The heartbeat thread, being frozen, has not extended it. The lock is now available.
5.  **T=31s:** Worker B, processing another request for the same item, successfully acquires the lock.
6.  **T=32s:** Worker B reads the stock (10), decrements it to 9, writes the new value to the database, and releases the lock. The database correctly reflects a stock of 9.
7.  **T=40s:** The GC pause on Worker A completes. Its threads resume execution.
8.  **T=40.1s:** Worker A's main thread, completely unaware that it lost the lock, continues its operation using its stale, in-memory `current_stock` value of 10. It calculates `new_stock = 9`.
9.  **T=40.2s:** Worker A executes `self.db.update_stock("item_001", 9)`, overwriting the correct value in the database. The stock is now incorrectly set back to 9, when it should be 8. **An item has been sold twice, and inventory is now corrupted.**

The `finally` block's check-and-delete logic works correctly—Worker A will notice it no longer holds the lock and won't try to delete it. But by then, the damage to the business data has already been done. The lock is meant to protect the *write*, and in this scenario, it has failed.

### Required Remediation: Fencing

To solve this, we cannot rely solely on the time-based lock. We need a mechanism to detect and reject writes from a worker that has lost its lease. This is typically done with a **fencing token**.

The most direct way to implement this is through optimistic concurrency control at the database layer. The `update_stock` operation must be made conditional.

Instead of this:
`UPDATE inventory SET stock = ? WHERE item_id = ?`

The operation must become:
`UPDATE inventory SET stock = ? WHERE item_id = ? AND stock = ?`

The `db.update_stock` method signature should be changed to `update_stock(item_id, new_stock, expected_current_stock)`. Your service logic will then check the number of rows affected by this update. If 0 rows are affected, it means the stock level changed between the read and the write. This indicates our worker lost the race, and the entire operation must be aborted and failed.

This change ensures that a "zombie" worker operating with stale data cannot corrupt the database state.

### Minor Concern: Thread Safety of `_heartbeat_active`

The `self._heartbeat_active` flag is written by the main thread and read by the heartbeat thread. While the GIL makes single assignments of primitive types like booleans atomic in CPython, relying on this is not a robust concurrency pattern. It's better to be explicit. Please replace this flag with a `threading.Event` object for signaling between threads. It is safer, more explicit, and a standard pattern for this use case.

This PR cannot be approved in its current state. The data integrity risk is too high for a critical system like inventory management. Please implement the optimistic locking pattern as described above.
