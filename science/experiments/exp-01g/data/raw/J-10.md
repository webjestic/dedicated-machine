Decision: Request Changes
Summary: The pull request introduces a distributed lock with a heartbeat, which is a solid pattern for this use case. However, the current implementation has a critical concurrency flaw where the heartbeat state is stored on the shared service instance, making it unsafe for concurrent requests. Additionally, the test suite completely bypasses the new heartbeat logic, leaving the most complex part of the implementation untested.

### Detailed Feedback

Thank you for this contribution. The use of a distributed lock with a heartbeat to manage high-contention inventory is the right approach. However, there are a few critical issues that must be addressed before this can be approved.

#### 1. **Major Architectural Flaw: Concurrency Unsafe State**

The `_heartbeat_active` flag and `_heartbeat_thread` are stored as instance attributes on `InventoryService`. In a typical production environment, a single instance of `InventoryService` will be used to handle many concurrent requests.

This design will lead to a race condition:
1.  Request A calls `reserve_item`, acquires a lock, and starts a heartbeat thread, storing it in `self._heartbeat_thread`.
2.  Request B, using the same service instance, calls `reserve_item` for a different item. It acquires its own lock and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread`.
3.  When Request A's `_acquire_lock` context exits, it calls `_stop_heartbeat()`, which sets `self._heartbeat_active = False`.
4.  This will cause the heartbeat loop for Request B's lock to terminate prematurely, as it checks the same shared flag.

**Recommendation:** The state for a single lock's heartbeat (the thread object and its control mechanism) must be managed entirely within the scope of the `_acquire_lock` context manager, not on the shared `self` object. The `_start_heartbeat` function could return the necessary state, which is then managed and cleaned up within the `try...finally` block of the context manager.

#### 2. **Critical Gap in Test Coverage**

The test suite explicitly mocks out the heartbeat mechanism:

```python
# tests/test_inventory.py
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the most complex, novel, and error-prone part of this feature is completely untested. We have no verification that the heartbeat thread starts, extends the lock TTL correctly, or terminates cleanly.

**Recommendation:** Please add tests that validate the heartbeat behavior. This will likely involve using `threading.Event` to coordinate between the test and the background thread, and mocking `time.sleep` to test the timing logic without making the tests slow. You should assert that `redis.expire` is called by the background thread as expected.

#### 3. **Potential Performance Concern: Thread Proliferation**

The current design creates a new thread for every lock acquisition. During a flash sale, this could result in the creation of hundreds or thousands of threads, which introduces significant system overhead (memory, context switching).

**Recommendation:** While the concurrency bug is the primary blocker, please consider a more scalable long-term solution. A better pattern would be a single, shared background task/thread that periodically renews all active locks held by the service instance, rather than one thread per lock. For now, please prioritize fixing the concurrency flaw.

This code cannot be merged in its current state due to the race condition and lack of testing. Please address the feedback above.
