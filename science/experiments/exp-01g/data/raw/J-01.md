Decision: Request Changes
Summary: This pull request introduces a conceptually sound approach for distributed inventory reservation using a Redis lock with a heartbeat. However, the current implementation contains a critical architectural flaw in its heartbeat mechanism that makes it non-thread-safe and will fail under concurrent load. Additionally, a severe security vulnerability exists due to missing input validation on the reservation quantity, and the complex heartbeat logic is not covered by the test suite.

---

### Detailed Review

Thank you for this submission. The use of a distributed lock with a renewal mechanism is the correct pattern for this high-contention problem. However, I've identified several critical issues that must be addressed before this can be approved.

#### 1. Critical: Thread-Safety Flaw in Heartbeat Implementation

The current implementation of the heartbeat uses instance variables (`self._heartbeat_active`, `self._heartbeat_thread`) to manage the state of the renewal thread. In a production environment, a single `InventoryService` instance will likely handle multiple requests concurrently in separate threads.

This design introduces a race condition:
1.  Request A calls `reserve_item`, acquires a lock, and starts a heartbeat thread, setting `self._heartbeat_thread`.
2.  Before Request A completes, Request B calls `reserve_item` on the *same service instance*. It acquires its own lock and starts a *new* heartbeat thread, overwriting `self._heartbeat_thread`.
3.  The heartbeat for Request A is now lost. If its database operation is slow, its lock will expire prematurely, allowing another process to acquire it and leading to data corruption.

**Recommendation:** The state for the heartbeat must be scoped to the individual lock acquisition, not the service instance. Consider passing a control object (like a `threading.Event`) to the thread function to signal termination, avoiding shared instance state entirely.

#### 2. Critical: Security Vulnerability via Input Validation

The `reserve_item` method does not validate the `quantity` parameter. It is currently possible to pass a non-positive value.

-   `quantity=0`: This would successfully create a zero-quantity order, which is likely undesirable.
-   `quantity=-1`: This would **increase** the `current_stock`, as the calculation would be `new_stock = current_stock - (-1)`. This is a severe vulnerability that allows for the artificial inflation of inventory.

**Recommendation:** Add input validation at the beginning of `reserve_item` to ensure `quantity` is a positive integer (i.e., `quantity > 0`).

#### 3. Major: Inadequate Test Coverage

The tests explicitly mock out and disable the heartbeat mechanism (`svc._start_heartbeat = MagicMock()`). This means the most complex and failure-prone part of this new service is completely untested. We have no verification that the lock is actually renewed or that the thread is managed correctly.

**Recommendation:**
-   The heartbeat logic must be tested. This may require more complex tests that inspect the calls to `redis.expire` over time, perhaps by patching `time.sleep`.
-   Add test cases to verify that invalid input for `quantity` (e.g., 0, -1) is properly rejected.

This code is a good first step, but the identified issues present significant risks to system stability and data integrity. Please address these points before resubmitting for another review.
