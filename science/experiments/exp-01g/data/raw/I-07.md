Decision: Request Changes
Summary: This pull request introduces a sophisticated but critically flawed distributed locking mechanism. The heartbeat renewal thread does not protect against process pauses (e.g., stop-the-world GC), creating a "zombie leader" scenario that can lead to data corruption and overselling of inventory. Furthermore, the current implementation of the heartbeat thread is not thread-safe, causing concurrent reservations on the same service instance to interfere with one another.

---

### Review

This is a thoughtful attempt at solving a genuinely difficult problem in distributed systems: maintaining correctness under high contention. The use of a lock with a random token for ownership verification and a heartbeat for TTL extension demonstrates a good grasp of the common patterns. However, there are two significant architectural flaws that prevent me from approving this PR.

#### 1. Critical Flaw: Vulnerability to Process Pauses and Data Corruption

The primary issue is a subtle but severe race condition that the heartbeat mechanism fails to prevent. The design is vulnerable to any event that causes the entire worker process to pause for a duration longer than the lock's Time-To-Live (TTL), such as a stop-the-world garbage collection pause, a VM migration, or extreme OS-level resource contention.

Here is the failure scenario:

1.  **Worker A** acquires the Redis lock (`LOCK_TTL = 30s`) and starts its heartbeat thread.
2.  **Worker A** successfully reads the current stock from the database (e.g., `current_stock = 10`).
3.  **The entire process for Worker A freezes.** Crucially, this freezes the main application thread *and* the heartbeat thread. The heartbeat cannot run to extend the lock's TTL.
4.  After 30 seconds, the lock expires in Redis.
5.  **Worker B** now successfully acquires the lock for the same item. It reads the stock (10), decrements it to 9, writes the new value to the database, and releases the lock. The inventory is now correctly 9.
6.  Sometime later, **Worker A** unfreezes. It is unaware that its lock is long gone. It resumes execution with its stale data: `current_stock = 10`.
7.  **Worker A** calculates `new_stock = 10 - 1 = 9` and overwrites the database with this value.
8.  The final stock in the database is 9, but it should be 8. Two customers were charged, but the inventory was only decremented once. **We have lost data and sold an item we do not have.**

The lock release logic correctly checks the token, so the stale Worker A will not improperly delete the lock if Worker B (or another worker) now holds it. However, the damage is already done by the unprotected database write.

**Recommendation:**

The atomicity of the read-modify-write cycle must be guaranteed at the data store itself. The lock in Redis is a good mechanism for reducing contention, but it cannot be the sole guarantor of correctness.

You must implement **optimistic concurrency control** for the database write. The `update_stock` operation should fail if the state of the row has changed since it was read.

A simple way to achieve this is to make the `UPDATE` conditional on the value you previously read:

```sql
-- The 'current_stock' value read at the start of the critical section
-- is used in the WHERE clause.
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id AND stock = :current_stock_read;
```

Your `db.update_stock` method must then check the number of rows affected by this query. If it is 0, it means another process modified the stock, and this transaction must be aborted and likely retried. This is a form of **fencing** that prevents stale writes from "zombie" workers.

#### 2. Implementation Flaw: Heartbeat State is Not Thread-Safe

There is a secondary, but still serious, bug in the implementation. The `_heartbeat_active` flag and `_heartbeat_thread` object are stored as instance attributes on `self`. This means an `InventoryService` instance is not re-entrant or thread-safe.

Consider what happens if two requests arrive at the same web server worker process and concurrently call `reserve_item()` on the same `InventoryService` instance for two *different* items:

1.  Thread 1 calls `reserve_item("item-A")`. It acquires its lock, sets `self._heartbeat_active = True`, and starts `self._heartbeat_thread` to manage the lock for "item-A".
2.  Thread 2 calls `reserve_item("item-B")`. It acquires its lock and *overwrites* `self._heartbeat_thread` with a new thread for "item-B". The original heartbeat thread for "item-A" is now orphaned.
3.  When Thread 1 finishes its work, its `finally` block calls `_stop_heartbeat()`, which sets `self._heartbeat_active = False`.
4.  This will prematurely terminate the heartbeat for "item-B", not "item-A", potentially causing the lock for "item-B" to expire during a long operation.

**Recommendation:**

The state for a single lock's heartbeat (the thread object, the stop-condition flag, etc.) must not be stored on the shared service instance. It should be encapsulated and managed entirely within the scope of the `_acquire_lock` context manager for each specific call.

---

Due to the critical risk of data corruption, this PR cannot be approved in its current state. Please address the fencing issue at the database layer and correct the thread-safety of the heartbeat implementation.
