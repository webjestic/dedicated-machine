### Review

This is a solid attempt at a complex problem, and I appreciate the attention to detail in the locking mechanism, particularly the atomic `set nx=True ex=...`, the ownership token, and the safe release check. The heartbeat concept is also a good defense against premature lock expiration during long-running database transactions.

However, I've identified one critical architectural issue and one minor validation gap that must be addressed before this can be approved.

#### **Blocker: `InventoryService` is not thread-safe**

The current implementation manages the heartbeat thread and its state (`_heartbeat_active`, `_heartbeat_thread`) as instance variables on the `InventoryService` class.

**Problem:** If this service is instantiated as a singleton or reused across multiple concurrent requests (as is common in web frameworks), a race condition will occur.

*   **Request A** calls `reserve_item`, acquires a lock, and starts a heartbeat thread, setting `self._heartbeat_thread`.
*   **Request B** (for a different item) calls `reserve_item` on the *same service instance* a moment later. It also acquires its lock and starts a new heartbeat thread, overwriting `self._heartbeat_thread`.
*   When **Request A** finishes, its call to `_stop_heartbeat` will signal the flag to stop, but it has lost the handle to its original thread. The thread from Request B might be stopped prematurely, or Request A's thread might be orphaned and continue running.

This makes the service unreliable and unsafe for concurrent use, which is the primary use case for a flash sale.

**Recommendation:**

The entire lifecycle of the heartbeat mechanism should be encapsulated within the `_acquire_lock` context manager and not rely on instance state.

1.  Create a `threading.Event` or a simple list/dict flag *inside* `_acquire_lock`.
2.  Pass this event/flag to the `_beat` function.
3.  The `finally` block in `_acquire_lock` will set the event/flag to signal the thread to stop.

This ensures that each call to `reserve_item` has its own isolated, independent heartbeat mechanism that cannot interfere with other concurrent calls.

#### **Minor: Missing Input Validation**

The `reserve_item` method does not validate the `quantity` parameter. A user could potentially request a quantity of `0` or `-1`. While this might not crash the system, it could lead to unexpected behavior or data corruption (e.g., *increasing* stock if a negative number is processed).

**Recommendation:**

Add a check at the beginning of `reserve_item`:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if quantity <= 0:
        return ReservationResult(
            success=False,
            message="Quantity must be a positive integer."
        )
    # ... rest of the function
```

*   **Correct Lock Release:** The check `if current and current.decode() == token:` before deleting the lock is textbook-perfect. This is a critical safety feature that prevents a process from deleting a lock it no longer owns.
*   **Test Coverage:** The test for the "stolen lock" scenario (`test_lock_not_released_if_expired`) is excellent and demonstrates a deep understanding of the potential failure modes.
*   **Clarity:** The code is well-structured, and the use of a context manager for the lock makes the business logic in `reserve_item` clean and easy to follow.

Please address the thread-safety issue and add the input validation. Once those changes are made, I'll be happy to approve this PR.
