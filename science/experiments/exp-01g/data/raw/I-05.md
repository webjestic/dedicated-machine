Decision: Request Changes
Summary: This pull request introduces a sophisticated but flawed implementation of a distributed lock. The heartbeat mechanism correctly handles slow operations but fails to protect against process pauses (e.g., stop-the-world GC), leaving a critical race condition that can lead to data corruption, such as selling the same item twice. The atomicity of the critical database write must be guaranteed with a fencing token or optimistic concurrency control at the database level.

---

### Detailed Review

This is a thoughtful attempt at a very difficult problem, and I appreciate the attention to detail regarding retry logic, ownership tokens, and the effort to prevent lock expiry with a heartbeat. However, the current implementation contains a critical architectural flaw that undermines the entire purpose of the lock.

#### **Critical Flaw: Vulnerability to Process Pauses (The "Zombie Worker" Problem)**

The core issue is a false sense of security provided by the heartbeat thread. While it protects against a *slow* critical section, it offers no protection against a *paused* critical section.

Consider this failure scenario:

1.  **Acquire & Read:** Worker A acquires the lock for `item_001` (stock=5) with a 30-second TTL. It reads the current stock from the database (`current_stock = 5`).
2.  **Process Pause:** Immediately after the read, Worker A's process is hit with a stop-the-world garbage collection pause (or a VM migration, etc.) that lasts for 31 seconds. The heartbeat thread is part of this process and is also frozen; it **cannot** extend the lock's TTL.
3.  **Lock Expiry & Takeover:** At T+30s, the lock expires in Redis. Worker B, seeing the lock is free, acquires it for the same `item_001`.
4.  **Worker B Completes:** Worker B reads the stock (5), decrements it to 4, writes the new value to the database, and releases the lock. The database correctly reflects a stock of 4.
5.  **Zombie Write:** At T+31s, Worker A's process resumes. It is completely unaware that its lock expired and was used by another worker. It continues its operation from where it left off, holding a stale `current_stock` value of 5. It calculates `new_stock = 4`, and overwrites the database with this value.
6.  **Data Corruption:** The final stock in the database is 4. However, two items were successfully sold (one by Worker A, one by Worker B). The correct stock should be 3. An item has been oversold, and our inventory state is now corrupt.

The check-before-delete logic in your `finally` block correctly prevents Worker A from releasing Worker B's lock, but by then the damage to the database has already been done.

**Required Change:**

The only robust solution is to enforce atomicity at the source of truth: the database. The lock should be seen as a performance optimization to prevent contention, not as the sole guarantor of correctness.

You must implement optimistic concurrency control for the database write. The `update_stock` operation needs to include a condition to ensure the data has not changed since it was read.

**Example (using stock value):**

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id AND stock = :current_stock_at_time_of_read;
```

Your application code must then verify that this `UPDATE` statement affected exactly one row. If it affected zero rows, it means another process modified the stock in the interim. The operation must be aborted and likely retried from the beginning (re-acquiring the lock, re-reading the stock).

A more canonical and robust solution is to use a dedicated **fencing token** (i.e., a version number) on the database row that is incremented with each update.

#### **Secondary Issues**

1.  **Instance-level Heartbeat State:** The `_heartbeat_active` flag and `_heartbeat_thread` are stored as instance attributes (`self.*`). If a single `InventoryService` instance is used in a multi-threaded environment (e.g., a web server), concurrent calls to `reserve_item` for different items will overwrite each other's heartbeat state, causing unpredictable behavior. The heartbeat's lifecycle and state must be managed per-lock-acquisition, not per-service-instance.

2.  **Non-Atomic Lock Release:** The lock release logic (`get` followed by `delete`) is not atomic. While the window for a race condition is small, it is non-zero. The correct way to implement this in Redis is with a Lua script that atomically checks the token and deletes the key if it matches.

    ```lua
    -- release_lock.lua
    if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])
    else
      return 0
    end
    ```

Please address the critical database write vulnerability before this can be considered for approval. The secondary issues should also be rectified for a production-ready system.
