Decision: Request Changes
Summary: This pull request introduces a sophisticated but critically flawed distributed locking mechanism for inventory management. While the heartbeat thread commendably attempts to prevent lock expiration during slow operations, it provides a false sense of security as it fails to protect against process pauses (e.g., GC, VM suspension). This creates a race condition that can lead to data corruption, such as selling the same item twice.

---

### Detailed Review

This is a thoughtful attempt at a very difficult problem, and I appreciate the attention to detail regarding lock contention, backoff, and safe lock release. The use of a heartbeat to extend the lock TTL is a known pattern, but its application here is incomplete and masks a more subtle and critical failure mode.

#### Critical Flaw: Vulnerability to Process Pauses (The "Zombie Worker" Problem)

The primary issue is that the lock's TTL and the heartbeat renewal mechanism are insufficient to guarantee correctness in a distributed environment. The system is vulnerable to data corruption if a worker process experiences a significant pause *after* acquiring the lock and reading from the database.

**Failure Scenario:**

1.  **Worker A** acquires the lock for `item_001` (TTL = 30s) and starts its heartbeat thread.
2.  **Worker A** reads the database: `current_stock = 10`.
3.  **A stop-the-world event occurs:** The entire process for Worker A, including its heartbeat thread, is paused by the operating system, a VM hypervisor performing a migration, or a long garbage collection cycle. The pause lasts longer than the lock's 30-second TTL.
4.  **The lock expires in Redis.** The heartbeat thread, being frozen, was unable to extend it.
5.  **Worker B** now successfully acquires the lock for `item_001`.
6.  **Worker B** reads `current_stock = 10`, decrements it to 9, writes the new value to the database, and releases its lock. The inventory is now correctly 9.
7.  **Worker A resumes execution.** It is completely unaware that its lock expired and was used by another worker. It continues its operation based on its stale, in-memory data: `current_stock = 10`.
8.  **Worker A** calculates `new_stock = 10 - 1 = 9` and writes this value to the database, overwriting the correct value.
9.  **Worker A** attempts to release its lock. The safe release logic correctly prevents it from deleting the lock (which no longer belongs to it), but the data corruption has already occurred.

**Result:** Two separate orders have been processed, but the stock has only been decremented by one. We have sold an item we do not have.

#### Recommended Solution: Fencing with Optimistic Locking

To solve this, we cannot rely on the lock alone. The write operation to the data store must be conditional, ensuring that the state has not changed since it was read. This is a form of **fencing**.

The most direct way to implement this is with **optimistic locking** at the database level. The `UPDATE` statement must check that the `stock` value has not changed since it was read.

The database update logic in `reserve_item` should be changed from a blind write:

```python
# Unsafe
self.db.update_stock(item_id, new_stock)
```

To a conditional update that verifies the state:

```python
# Safe
rows_affected = self.db.update_stock_conditional(item_id, new_stock, current_stock)

if rows_affected == 0:
    # The stock was modified by another process after we read it.
    # This reservation attempt has failed due to a race condition.
    logger.warning("Optimistic lock failed for item_id=%s. Aborting.", item_id)
    return ReservationResult(
        success=False,
        message="Inventory level changed. Please try again."
    )
```

The corresponding SQL would look something like this:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id AND stock = :expected_current_stock;
```

This change makes the critical write operation atomic and safe. If the "zombie" Worker A wakes up and attempts its write, the `WHERE stock = 10` clause will fail because the stock is now 9. The update will affect 0 rows, signaling that the operation must be aborted.

### Minor Observations

*   **Thread Management:** The service manages its own background thread. This can complicate application shutdown and resource management. Ensure that the lifecycle of this service and its thread is handled correctly, especially in server environments with multiple worker processes (e.g., Gunicorn). Using a `threading.Event` for shutdown signaling would be more robust than the current boolean flag.
*   **Test Coverage:** The tests correctly mock out the heartbeat thread for simplicity. However, this means the heartbeat logic itself is untested. Given the critical flaw identified above, this is a secondary concern, but something to note for the future.

Please implement optimistic locking in the database update to prevent this race condition. The heartbeat is a useful optimization to prevent lock expirations during *genuinely long-running but active* operations, but it cannot be the sole mechanism for ensuring correctness.
