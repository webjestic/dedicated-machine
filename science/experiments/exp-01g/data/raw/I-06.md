Decision: Request Changes
Summary: This pull request introduces a sophisticated Redis-based distributed lock with a heartbeat mechanism. However, it contains a critical architectural flaw that can lead to data corruption under concurrent load. The implementation is vulnerable to process pauses (e.g., stop-the-world GC), which can cause a stale write to the database, effectively violating the mutual exclusion guarantee and leading to oversold inventory.

---

### Detailed Review

This is a thoughtful attempt at a difficult problem, and I appreciate the attention to detail in the Redis lock implementation, including the use of an ownership token, atomic operations, and exponential backoff. The heartbeat thread is a clever addition to handle long-running database operations.

However, a critical failure mode has been overlooked. The current design does not protect against process pauses, which can lead to a race condition that undermines the entire purpose of the lock.

#### The "Zombie Leader" Failure Scenario

The core issue is that a process holding the lock can be paused for an extended period, for example by a stop-the-world garbage collection cycle, a container being de-scheduled, or a VM migration. During this pause, the main application thread and the heartbeat thread are both frozen.

Consider the following sequence of events:

1.  **Worker A** acquires the lock for `item_001` with a 30-second TTL.
2.  **Worker A** executes `self.db.get_stock(item_id)` and reads the current stock, which is `10`.
3.  **A stop-the-world GC pause freezes Worker A's entire process for 35 seconds.** The heartbeat thread is also frozen and cannot extend the lock's TTL.
4.  After 30 seconds, the lock expires in Redis.
5.  **Worker B** now successfully acquires the lock for `item_001`.
6.  **Worker B** reads the stock (`10`), decrements it to `9`, writes `9` to the database, and releases its lock. The state of the system is now correct.
7.  **Worker A's process resumes.** It is unaware that it was paused or that its lock expired and was claimed by another worker. It continues its operation from where it left off, holding the stale `current_stock` value of `10`.
8.  **Worker A** calculates `new_stock = 10 - 1 = 9` and executes `self.db.update_stock("item_001", 9)`. This write succeeds, overwriting the correct value (`9`) with its own stale, but identical, value (`9`).
9.  **Worker A** proceeds to create an order record.

The result is catastrophic: two orders have been created, but the stock has only been decremented once. An item has been sold twice. The `finally` block in `_acquire_lock` correctly prevents Worker A from deleting the lock (as it no longer owns it), but the data corruption has already occurred.

#### Required Changes: Fencing

To prevent this, the lock's ownership token must be used as a **fencing token** that extends all the way to the data store. The write operation itself must be made conditional to prevent stale writes from succeeding.

The canonical solution is to add a monotonically increasing version number to the inventory table.

1.  When reading the stock, also read the current `version` number from the database row.
2.  The `update_stock` call must be modified to be conditional on this version number. The pseudo-SQL would look like this:

    ```sql
    UPDATE inventory
    SET
      stock = :new_stock,
      version = :old_version + 1
    WHERE
      item_id = :item_id AND version = :old_version;
    ```

3.  After executing this query, you **must** check the number of affected rows.
    *   If `1` row was affected, the update was successful.
    *   If `0` rows were affected, it means another process updated the row first (i.e., our `version` was stale). In this case, the entire `reserve_item` operation must be aborted and failed, as our "lease" on the data was violated.

This change makes the database the final arbiter of truth and protects against writes from "zombie" lock holders. While the Redis lock is still valuable for reducing contention on the database, it cannot, by itself, guarantee correctness in the face of process pauses.

Please implement this fencing mechanism before this PR can be considered for approval.
