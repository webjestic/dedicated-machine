# Operational Design Specification: Production Webhook Receiver

**Service:** Webhook Receiver
**Runtime:** Node.js, single containerized service
**Revision:** 1.0
**Status:** Authoritative handoff document

---

## Preamble

This document covers every decision made in the design of this receiver and every operational requirement the deploying engineer must satisfy. Nothing here is advisory. An item listed as a requirement is a requirement. An item listed as a failure mode is a real failure mode that has caused production incidents. Read this before writing any configuration, not after the first page.

# Part I: Implementation Decisions

---

## 1. Request Ingestion

### 1.1 HTTP Method and Path

The receiver exposes exactly one route: `POST /webhook`. All other methods on that path return `405 Method Not Allowed`. All other paths return `404 Not Found`. These are not default framework behaviors to be left in place — they must be explicitly configured. A catch-all that returns `200` on `GET /webhook` is an information leak and a source of false-positive health check confusion.

### 1.2 Raw Body Preservation

HMAC-SHA256 signature validation operates on the raw request body as received over the wire. The body must be captured as a raw `Buffer` before any parsing occurs. Any middleware that parses the body into a JavaScript object — JSON middleware, body-parser with automatic decoding — must not run before signature validation, and must not mutate the raw bytes used for validation.

**Decision:** The raw body is captured once, stored as a `Buffer`, and passed to the signature validator. JSON parsing occurs only after signature validation passes, and only on the same buffer. The parsed object is never used for validation.

**Failure mode this prevents:** A middleware that normalizes Unicode, strips whitespace, or re-serializes the body will produce a buffer that does not match the original. The HMAC will fail on legitimate events. This is not a theoretical concern — it has caused production outages when framework upgrades changed default body handling.

### 1.3 Body Size Limit

The receiver enforces a maximum body size. The limit is set to a value appropriate for the upstream provider's documented maximum payload size, plus a small margin. Requests exceeding the limit are rejected with `413 Payload Too Large` before the body is fully read.

**Decision:** The limit is not left at a framework default. It is explicitly set and documented. The current value must appear in this document when finalized. A missing limit is an unbounded memory allocation vulnerability on a public endpoint.

### 1.4 Content-Type Enforcement

The receiver requires `Content-Type: application/json`. Requests with a missing or non-JSON content type are rejected with `415 Unsupported Media Type`. This is enforced before body reading.

**Rationale:** Legitimate webhook senders always set this header. Rejecting early reduces processing cost on malformed or probing requests.

## 2. Signature Validation

### 2.1 Algorithm

HMAC-SHA256 as specified by the upstream provider. The implementation uses Node.js's built-in `crypto` module. No third-party cryptographic library is introduced for this operation.

### 2.2 Secret Material

The signing secret is a single value injected at runtime via environment variable. It is never hardcoded, never committed to source control, and never logged. The environment variable name is `WEBHOOK_SECRET`.

**Decision:** The secret is read once at process startup and held in memory. It is not re-read on each request. If the secret must be rotated without a restart, see Section 2.3.

### 2.3 Secret Rotation

Secret rotation is a first-class operational event, not an edge case. The upstream provider may rotate secrets, or the operator may rotate them in response to a suspected compromise. The receiver must not require downtime to rotate.

**Decision:** The receiver supports two active secrets simultaneously: `WEBHOOK_SECRET` (current) and `WEBHOOK_SECRET_PREVIOUS` (outgoing). On each request, validation is attempted against the current secret first. If that fails, validation is attempted against the previous secret. If both fail, the request is rejected. The previous secret slot is cleared once the upstream provider confirms all in-flight events were signed with the new secret.

**Failure mode this prevents:** A hard cutover from old secret to new secret creates a window during which events signed with the old secret — already in transit — are rejected. Depending on the provider's retry behavior, these events may be lost or may trigger false alerts.

### 2.4 Timing-Safe Comparison

The HMAC comparison uses a constant-time comparison function. Node.js `crypto.timingSafeEqual` is the required implementation. String equality (`===`) is explicitly prohibited for this comparison.

**Rationale:** A timing oracle on a public endpoint is a real attack surface. The comparison must not short-circuit.

### 2.5 Signature Header Parsing

The upstream provider may encode the signature in a header that includes a prefix (e.g., Stripe's `v1=` scheme) or a version identifier. The parser must extract the raw hex digest before comparison. The parser must handle the case where the header is absent, malformed, or contains multiple comma-separated values. In all failure cases, the request is rejected with `401 Unauthorized` and a structured log entry is emitted.

### 2.6 Rejection Response

A request that fails signature validation returns `401 Unauthorized`. The response body contains no information about why validation failed — not which header was missing, not which secret was tried. The failure reason is recorded in the structured log, not in the response.

**Rationale:** Diagnostic information in rejection responses assists attackers in probing the validation logic.

## 3. Replay Attack Prevention

### 3.1 Timestamp Validation

Many providers include a request timestamp in the signature payload or in a dedicated header (e.g., Stripe's `Stripe-Signature` header includes `t=`). If the provider includes a timestamp, the receiver validates that the timestamp is within an acceptable window of the current server time.

**Decision:** The acceptable window is five minutes in either direction. Requests outside this window are rejected with `400 Bad Request`. The window is configurable via environment variable `WEBHOOK_TIMESTAMP_TOLERANCE_SECONDS` with a default of 300.

**Failure mode this prevents:** A valid signature captured from a legitimate request can be replayed indefinitely if no timestamp check is enforced. The attacker does not need the secret — only a previously observed request.

### 3.2 Idempotency Key Deduplication

Timestamp validation alone does not prevent replay within the tolerance window. The receiver deduplicates events by their provider-assigned event ID.

**Decision:** Each incoming event ID is checked against a short-lived deduplication store before being queued. If the ID is already present, the request returns `200 OK` immediately — not an error — without re-queuing the event. The deduplication store retains IDs for a duration equal to twice the timestamp tolerance window.

**Decision on store technology:** The deduplication store is Redis with TTL-based expiry. An in-process store is explicitly rejected because it does not survive restarts and does not work correctly if more than one container instance is running.

**Failure mode this prevents:** Providers retry on any non-`2xx` response and on network timeouts. A receiver that processes an event and then crashes before returning `200` will receive the same event again. Without deduplication, downstream systems process the event twice.

## 4. Async Processing and Queuing

### 4.1 Separation of Receipt from Processing

The receiver's only job is to validate, deduplicate, and enqueue. No business logic executes in the request path. No database calls, no downstream HTTP calls, no transformations beyond what is required to construct the queue message.

**Decision:** The request path is: receive → validate signature → check deduplication → enqueue → return `200`. Total time in the request path must be bounded and fast. Any operation that can block — including the enqueue itself — must have a timeout.

### 4.2 Queue Technology

The queue is an external durable message queue. Acceptable options are Redis Streams, RabbitMQ, or a cloud-native queue (SQS, Pub/Sub). An in-process queue (EventEmitter, array, in-memory queue library) is explicitly rejected.

**Rationale:** An in-process queue loses all unprocessed events on restart. A containerized service restarts. This is not a theoretical concern.

### 4.3 Enqueue Failure Handling

The enqueue operation can fail. The queue may be unavailable. The network between the receiver and the queue may be degraded. This failure mode must be handled explicitly.

**Decision:** If the enqueue operation fails after exhausting retries (see Section 4.4), the receiver returns `500 Internal Server Error` to the upstream provider. It does not return `200`. Returning `200` on a failed enqueue silently drops the event.

**Consequence:** The upstream provider will retry. The receiver must be idempotent with respect to retries. The deduplication layer (Section 3.2) handles this — but only if the event was successfully deduplicated before the enqueue failed. The deduplication write and the enqueue must be treated as a unit: if the enqueue fails, the deduplication entry should not be committed, or the entry should be written only after a successful enqueue. The ordering of these two operations is a critical implementation detail.

**Decision on ordering:** Write to the deduplication store only after a successful enqueue. If the enqueue fails, the deduplication store is not written. This allows the provider's retry to be processed normally.

### 4.4 Enqueue Retry Policy

The receiver retries the enqueue operation a small number of times with exponential backoff before giving up. The total retry budget must fit within the upstream provider's request timeout (typically 10–30 seconds). The retry count and backoff parameters are configurable via environment variables.

**Default values:** 3 attempts, 100ms initial delay, 2x multiplier. Total maximum retry time: approximately 700ms. These values must be validated against the provider's documented timeout.

### 4.5 Queue Message Schema

The message written to the queue contains: the raw event payload (as a string or buffer), the event ID, the event type (from the provider's header), the timestamp of receipt (server time, ISO 8601), and the provider identifier. It does not contain the signing secret, any internal credentials, or any data not present in the original request.

## 5. Response Behavior

### 5.1 Success Response

A successfully validated and enqueued event receives `200 OK`. The response body is a minimal JSON object: `{"status":"received"}`. No event data is echoed back. No internal identifiers are included.

### 5.2 Error Response Codes

| Condition | Status Code |
|---|---|
| Signature validation failure | 401 |
| Missing or malformed signature header | 401 |
| Timestamp outside tolerance window | 400 |
| Body too large | 413 |
| Unsupported content type | 415 |
| Wrong HTTP method | 405 |
| Enqueue failure after retries | 500 |
| Duplicate event (already processed) | 200 |

### 5.3 No Stack Traces in Responses

Error responses never include stack traces, internal error messages, or framework error objects. All such information is written to the structured log only.

## 6. Security Hardening

### 6.1 TLS Termination

TLS is terminated at the load balancer or ingress layer, not in the Node.js process. The receiver listens on plain HTTP internally. This is standard for containerized deployments. The requirement is that TLS termination exists and is enforced — the receiver must not be reachable over plain HTTP from outside the cluster.

### 6.2 Network Ingress Restriction

If the upstream provider publishes a list of source IP ranges for webhook delivery, the ingress layer restricts inbound connections to those ranges. This is enforced at the network layer, not in application code. Application-layer IP filtering is a defense-in-depth measure only and must not be the primary control.

### 6.3 HTTP Security Headers

The receiver sets the following response headers on all responses: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`. It does not set `Server` or `X-Powered-By` headers. Framework defaults that expose version information must be explicitly disabled.

### 6.4 Rate Limiting

The receiver enforces a rate limit per source IP at the ingress layer. The limit is set to a value that accommodates the upstream provider's maximum delivery rate with headroom, while blocking abusive request volumes. The specific limit depends on the provider and must be set before deployment.
