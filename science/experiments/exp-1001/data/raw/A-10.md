## 1. Scope and Constraints

This specification covers a single containerized Node.js service whose sole responsibility is to receive, authenticate, and enqueue inbound webhook events from a trusted third-party provider (GitHub, Stripe, or equivalent). It does not cover downstream consumers or the queue infrastructure itself, except where the receiver's behavior is directly coupled to queue contracts.

**Hard constraints that drive every decision below:**

- The raw request body must be preserved byte-for-byte for HMAC validation. Any middleware that parses or transforms the body before signature verification is a correctness defect.
- The 200 response must be returned before any downstream processing begins. Latency in the processing path must not propagate back to the provider.
- A signature validation failure is a security event, not a routine error. It must be treated differently from a malformed payload or a queue write failure.
- This endpoint is public. It will receive probes, scanners, and deliberate abuse. The design must account for this from the start, not as an afterthought.

## 2. Implementation Decisions

### 2.1 Ingestion Layer

**Framework:** Express or Fastify. Either is acceptable. The decision criterion is team familiarity and the ability to intercept the raw body before any parsing occurs.

**Body handling:** The raw request body must be captured as a `Buffer` before any JSON parsing. This is non-negotiable. HMAC is computed over the exact bytes the provider signed — if the body is parsed and re-serialized first, signature validation will produce false negatives on any payload where key ordering, whitespace, or Unicode normalization differs from the provider's serialization. The raw buffer is stored on the request object and passed directly to the validation function. JSON parsing happens after validation succeeds, and only if the downstream queue requires a structured object rather than raw bytes.

**Content-Type enforcement:** The receiver must reject requests that do not carry `application/json` (or the provider-specific content type) before reaching the validation step. This is a cheap filter that eliminates a class of malformed requests early.

**Request size limit:** A hard cap on request body size must be enforced at the ingestion layer — not in application code. The appropriate limit is determined by the provider's documented maximum payload size, with a small margin. Requests exceeding the limit are rejected with 413 before the body is read into memory. This is a denial-of-service control.

---

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. The shared secret is used as the HMAC key. The message is the raw request body buffer.

**Comparison:** The computed HMAC digest must be compared to the provider-supplied signature using a constant-time comparison function. Node.js provides `crypto.timingSafeEqual` for this purpose. String equality (`===`) is not acceptable — it is vulnerable to timing side-channel attacks that allow an attacker to iteratively determine the correct signature byte by byte.

**Header extraction:** The provider delivers the signature in a request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The receiver must extract this header before validation. If the header is absent, the request is rejected immediately with 401. The absence of the header is not an edge case to handle gracefully — it is either a misconfigured sender or an attacker.

**Signature format:** Providers typically prefix the signature with the algorithm name (e.g., `sha256=<hex_digest>`). The receiver must strip the prefix before comparison. Failure to do so will produce a permanent validation failure for all requests, which is a silent correctness defect if error logging is insufficient.

**Failure response:** A failed signature validation returns 401. The response body must contain no information beyond a generic rejection message. Do not echo back the received signature, the expected signature, or any detail that assists an attacker in diagnosing why validation failed.

**Validation is synchronous and blocking.** It must complete before the 200 is returned. There is no acceptable design in which a request is acknowledged before its signature is verified.

---

### 2.3 Async Decoupling and Queue Contract

**Mechanism:** After a request passes signature validation, the event is written to a queue. The HTTP response is returned immediately after the enqueue operation completes — not after the event is processed. The processing path is entirely outside the request lifecycle.

**Queue selection:** The queue must be a durable, persistent store. In-memory queues (including Node.js `EventEmitter`, in-process arrays, or any structure that does not survive a process restart) are not acceptable for production. Acceptable options include Redis Streams, BullMQ backed by Redis, RabbitMQ, or a managed queue service (SQS, Cloud Pub/Sub). The selection criterion is durability guarantees and operational familiarity.

**Enqueue failure handling:** If the enqueue operation fails, the receiver must return 500 to the provider. This is the correct behavior because it signals the provider to retry the event. Returning 200 on an enqueue failure would silently drop the event — this is the worst possible outcome. The failure must be logged as a high-severity event with the full event payload (subject to data classification constraints) to enable manual recovery.

**Queue write timeout:** The enqueue operation must have an explicit timeout. A queue that is slow but not failing must not hold the HTTP connection open indefinitely. If the write does not complete within the timeout, the receiver returns 500 and logs the timeout. The timeout value must be shorter than the provider's own request timeout to ensure the provider receives the 500 and can retry, rather than timing out on its side and treating the event as delivered.

**Payload written to queue:** The queue message must include: the raw event payload, the event type (from the provider's header, e.g., `X-GitHub-Event`), a delivery ID (from the provider's header, e.g., `X-GitHub-Delivery`), the timestamp at which the receiver accepted the event, and the source IP. The delivery ID is the primary key for idempotency checks downstream.

---

### 2.4 Payload Handling

**No transformation in the receiver.** The receiver's job is to validate and enqueue. It must not filter fields, normalize values, or make decisions about which events are relevant. Those decisions belong to the downstream consumer. A receiver that filters events is a receiver that silently drops events when filter logic has a bug.

**Event type routing:** If the queue infrastructure supports topic or channel routing by event type, the receiver may set a routing key based on the event type header. This is acceptable because it is a mechanical operation with no semantic content — it does not require parsing or understanding the payload.

---

### 2.5 Response Contract

| Condition | HTTP Status | Notes |
|---|---|---|
| Signature valid, enqueue succeeded | 200 | Body: minimal acknowledgment or empty |
| Missing signature header | 401 | Generic message only |
| Signature invalid | 401 | Generic message only |
| Body exceeds size limit | 413 | |
| Unsupported content type | 415 | |
| Enqueue failed or timed out | 500 | Provider will retry |
| Unhandled exception | 500 | Must be caught by global error handler |

**Do not return 4xx for enqueue failures.** A 4xx tells the provider the event was malformed and should not be retried. A 500 tells the provider to retry. These are not interchangeable.

---

### 2.6 Secret Management

**The HMAC secret is not an environment variable baked into the container image.** It is injected at runtime via a secrets manager (Vault, AWS Secrets Manager, Kubernetes Secrets mounted as files, or equivalent). The secret must not appear in logs, stack traces, error messages, or environment variable dumps.

**Multiple active secrets:** The receiver must support holding more than one active secret simultaneously. This is required for secret rotation without downtime (see Section 3.1). The validation logic must attempt verification against each active secret and accept the request if any one succeeds.

**Secret loading:** Secrets are loaded at process startup and cached in memory. The receiver must not make a secrets manager call on every request — this introduces latency and a dependency on the secrets manager in the hot path.

---

### 2.7 Dependency Decisions

**Node.js `crypto` module only for HMAC.** No third-party cryptography libraries. The standard library implementation is correct, maintained, and carries no additional supply chain risk.

**Minimal dependency surface overall.** Every production dependency is an attack surface and an operational burden. The receiver should require: a web framework, a queue client library, a structured logging library, and nothing else that cannot be justified by a specific, non-negotiable requirement.

## 3. Operational Layer

### 3.1 Secret Rotation

**The failure mode this addresses:** A rotated secret that the receiver does not yet know about causes every subsequent webhook delivery to fail with 401. Depending on the provider's retry policy, this can mean hours of event loss before the problem is detected. If the rotation was triggered by a suspected compromise, the window between rotation and receiver update is the highest-risk period.

**Design decision:** The receiver maintains a list of active secrets, not a single secret. During rotation, the new secret is added to the list before the old secret is revoked at the provider. The receiver will accept signatures from either secret. Once the provider has been updated and a verification delivery has succeeded with the new secret, the old secret is removed from the list.

**Rotation procedure (must be documented in runbook):**
1. Generate new secret.
2. Add new secret to secrets manager alongside existing secret.
3. Trigger a live reload in the receiver (via signal or a reload endpoint accessible only from within the cluster).
4. Verify the provider can be updated to use the new secret.
5. Update the provider's webhook configuration to the new secret.
6. Confirm a successful delivery is validated with the new secret (check logs for which secret index was used).
7. Remove the old secret from secrets manager.
8. Trigger another live reload.

**Live reload mechanism:** The receiver must support reloading secrets without a process restart. A SIGHUP handler or an internal admin endpoint (not exposed externally) that re-fetches secrets from the secrets manager is acceptable. A process restart is not acceptable as the primary rotation mechanism because it introduces a gap in availability.

**Rotation must be tested in staging before it is needed in production.** A rotation procedure that has never been exercised will fail at the worst possible time.

---

### 3.2 Replay Attack Prevention

**The failure mode this addresses:** A valid, previously delivered webhook event is captured in transit and re-submitted. Because the signature is valid, the receiver accepts it. The downstream consumer processes the same event twice, potentially triggering duplicate charges, duplicate notifications, or duplicate state transitions.
