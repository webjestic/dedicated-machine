# Operational Design Specification: Production Webhook Receiver (Node.js)

---

## Document Scope

This specification covers the decisions, failure modes, and operational requirements for a production webhook receiver deployed as a single containerized service. It does not contain code. It is the document you read before writing code and the document you reach for at 3am.

---

# Part I: Implementation Decisions

---

## 1. Signature Validation

### Decision: Validate Before Everything Else

Signature validation executes before body parsing is exposed to application logic, before any database call, before any queue interaction. An invalid signature terminates the request immediately with `400`. No logging of the body. No further processing.

### Decision: Raw Body Preservation

HMAC-SHA256 is computed against the raw request body as received on the wire. Any middleware that parses, re-serializes, or normalizes the body before validation will silently corrupt the signature check. The raw buffer must be captured at the transport layer and held separately from any parsed representation. This is a hard architectural constraint, not a preference.

### Decision: Constant-Time Comparison

String equality (`===`) is not used for signature comparison under any circumstances. Timing side-channels on signature comparison are a real attack class. A constant-time comparison function from a vetted cryptographic library is used exclusively.

### Decision: Signature Header Normalization

The signature header name and expected prefix (e.g., `sha256=`) are configuration values, not hardcoded strings. Different providers format this differently. The prefix is stripped before comparison. If the header is absent, the request is rejected with `400` before any computation occurs.

### Decision: Reject on Algorithm Ambiguity

If the provider sends an algorithm prefix (e.g., `sha1=` vs `sha256=`), the receiver explicitly checks the algorithm field and rejects anything other than the configured algorithm. Downgrade attacks via algorithm substitution are a real risk during provider-side migrations.

## 2. Secret Management

### Decision: Secrets Are Never in Environment Variables Directly in Production

Secrets are injected via a secrets manager (Vault, AWS Secrets Manager, or equivalent) at startup and held in memory. Environment variables are acceptable for local development only. The distinction is enforced at the deployment configuration level, not by convention.

### Decision: Support for Multiple Active Secrets Simultaneously

Secret rotation is a live operational event. The receiver must support validating against an ordered list of secrets — current and previous — during a rotation window. Validation succeeds if any secret in the list produces a matching signature. The rotation window is bounded and explicit; old secrets are evicted after a configured TTL following rotation confirmation.

This is not optional. A receiver that can only hold one secret will cause dropped events during every rotation, and rotations will be avoided, which means secrets will never actually be rotated.

### Decision: Secret Reload Without Restart

The receiver exposes an internal mechanism (signal handler or admin endpoint, not public) to trigger secret reload from the secrets manager. Restarting a container to rotate a secret is operationally unacceptable in production.

## 3. Async Processing Architecture

### Decision: Queue-Based Decoupling Is the Only Acceptable Pattern

The request handler does one thing after validation: enqueue the event. It does not call downstream services, write to a database, or perform any computation that can block or fail in ways that affect the HTTP response. The 200 is returned as soon as the enqueue operation succeeds.

### Decision: Queue Write Failure Is a Hard Error

If the enqueue operation fails, the receiver returns `503`, not `200`. Returning `200` when the event has not been durably queued is a silent data loss event. The provider will retry on `503`. The provider will not retry on `200`.

This is the most operationally consequential decision in this document.

### Decision: Queue Technology Selection Criteria

The queue must provide:

- **At-least-once delivery guarantees** — events may be delivered more than once; they must not be lost
- **Durable persistence** — events survive a queue service restart
- **Visibility timeout / message locking** — a consumer that crashes mid-processing does not drop the event
- **Dead-letter queue (DLQ) support** — events that repeatedly fail processing are captured, not silently discarded

Acceptable options: Amazon SQS, RabbitMQ with durable queues and publisher confirms, Redis Streams with consumer groups. Redis pub/sub is not acceptable — it is fire-and-forget with no durability.

### Decision: Idempotency Is the Consumer's Responsibility, Enforced by Design

Because the queue provides at-least-once delivery, duplicate events will occur. The receiver stamps each enqueued message with the provider's event ID (extracted from the payload or header) and a receiver-assigned UUID. Consumers are required to implement idempotency keyed on the provider event ID. The receiver does not deduplicate at ingestion time — that would require shared state and adds latency to the hot path.

### Decision: Payload Stored in Queue Message, Not by Reference

The full validated payload is written into the queue message, not a pointer to a storage location. Indirection through a secondary store (e.g., "write to S3, enqueue the S3 key") introduces a second failure point in the hot path and a consistency window. For payloads within queue message size limits (SQS: 256KB, RabbitMQ: configurable), direct embedding is correct. For payloads exceeding limits, the overflow strategy is defined explicitly in configuration, not discovered at runtime.

## 4. HTTP Layer Decisions

### Decision: Body Size Limit Is Enforced at the Framework Layer

A hard maximum body size is configured at the HTTP server level. Requests exceeding this limit are rejected before the body is read into memory. The limit is set based on the provider's documented maximum payload size plus a defined margin. Accepting unbounded bodies on a public endpoint is a denial-of-service vector.

### Decision: Only POST Is Accepted on the Receiver Endpoint

All other HTTP methods return `405`. The endpoint does not respond to GET, HEAD, or OPTIONS in ways that reveal implementation details.

### Decision: Response Timing Is Bounded

The receiver has a hard internal timeout on the enqueue operation. If the queue write does not complete within this bound (suggested: 2–3 seconds), the request returns `503`. The provider's retry behavior is known and documented; the timeout is set to keep the receiver's response time well within the provider's own timeout window.

### Decision: No Redirects on the Receiver Path

The receiver endpoint does not issue redirects. A redirect on a POST request has undefined behavior across HTTP clients and webhook senders. The endpoint is stable, and its path does not change without a versioning strategy.

---

## 5. Containerization Decisions

### Decision: Single Process Per Container, No Process Manager

The container runs one Node.js process. No PM2, no cluster mode inside the container. Horizontal scaling is handled at the orchestration layer (Kubernetes, ECS). Running multiple processes inside a single container obscures resource accounting and complicates signal handling.

### Decision: Graceful Shutdown Is Implemented and Tested

On `SIGTERM`, the receiver stops accepting new connections, allows in-flight requests to complete (bounded by a drain timeout), and exits cleanly. A container that is killed mid-enqueue without graceful shutdown produces the exact failure mode (silent event loss) that the queue architecture is designed to prevent. The drain timeout is shorter than the orchestrator's termination grace period.

### Decision: Health Check Endpoints Are Separated by Concern

- `/healthz` — liveness: is the process alive and able to handle requests
- `/readyz` — readiness: is the process ready to receive traffic (secrets loaded, queue connection established)

These endpoints are not the same check. A process that is alive but has lost its queue connection must fail readiness without failing liveness. Failing liveness causes a restart; failing readiness removes the instance from the load balancer pool. These are different operational responses.

### Decision: Non-Root User in Container

The process runs as a non-root user. The base image is pinned to a specific digest, not a mutable tag. The image is rebuilt on a defined schedule to incorporate upstream security patches, not only when application code changes.

# Part II: Operational Layer

---

## 6. Observability

### 6.1 Structured Logging

Every log line is JSON. No unstructured strings in production. Every log entry for a webhook request carries: a correlation ID assigned at receipt, the provider event ID (if extractable before validation), the event type, the validation outcome, the enqueue outcome, and the response code. Log levels are: `info` for successful flows, `warn` for rejected requests (invalid signature, malformed payload), `error` for internal failures (queue write failure, secret load failure).

The raw payload body is never logged. Ever. Webhook payloads contain PII, credentials, and financial data depending on the provider. This is a compliance requirement, not a preference.

### 6.2 Metrics

The following metrics are emitted and alarmed on:

| Metric | Type | Purpose |
|---|---|---|
| `webhook.requests.received` | Counter | Total inbound volume |
| `webhook.requests.signature_valid` | Counter | Validation pass rate |
| `webhook.requests.signature_invalid` | Counter | Validation failure rate — alarm if elevated |
| `webhook.requests.enqueue_success` | Counter | Successful queue writes |
| `webhook.requests.enqueue_failure` | Counter | Queue write failures — alarm on any nonzero value |
| `webhook.requests.rejected_body_size` | Counter | Body size limit hits |
| `webhook.enqueue_latency_ms` | Histogram | Queue write latency — alarm on p99 degradation |
| `webhook.request_duration_ms` | Histogram | End-to-end request latency |
| `webhook.secret_rotation_active` | Gauge | 1 when multiple secrets are active, 0 otherwise |

Metrics are emitted in a format compatible with the observability stack (Prometheus, Datadog, CloudWatch). The choice is made at deployment time, not hardcoded.

### 6.3 Distributed Tracing

Each inbound request generates a trace. The trace carries through to the queue write. The correlation ID is propagated as a message attribute on the queue message so that downstream consumers can link their traces to the inbound event. Tracing is not optional instrumentation — it is the primary debugging tool when an event is reported missing.

## 7. Alerting

### 7.1 Signature Failure Rate Alert

A sustained rate of signature validation failures above a defined threshold (e.g., >5% of requests over 5 minutes) triggers a page, not a ticket. This pattern indicates one of: a secret rotation that was not coordinated with the receiver, an active replay or forgery attempt, or a provider-side bug. All three require immediate human attention.

A single invalid signature does not alert. Scanners and misconfigured clients will hit the endpoint. The rate, not the count, is the signal.

### 7.2 Enqueue Failure Alert

Any enqueue failure triggers an immediate alert. There is no acceptable rate of enqueue failures. A single failure means an event was not queued and the provider was told to retry. A sustained failure means events are being lost. This alert has the same priority as a service outage.

### 7.3 Queue Depth Alert

The downstream queue depth is monitored independently of the receiver. A queue that is growing without being consumed means the consumer is down or degraded. The receiver will continue to accept and enqueue events — this is correct behavior — but the alert ensures the consumer failure does not go unnoticed.

### 7.4 Receiver Latency Alert

If p99 request latency approaches the provider's documented timeout window, an alert fires. A slow receiver causes the provider to time out and retry, which produces duplicate events and backpressure on the queue. Latency degradation is almost always caused by queue write latency; the enqueue latency histogram identifies this directly.

### 7.5 Dead-Letter Queue Alert

Any message arriving in the DLQ triggers an alert. A message in the DLQ means a consumer has failed to process an event repeatedly. The event is not lost, but it is not being processed. The alert is a prompt to investigate the consumer, not the receiver — but the receiver's operational runbook includes this alert because the receiver team owns the end-to-end event pipeline.
