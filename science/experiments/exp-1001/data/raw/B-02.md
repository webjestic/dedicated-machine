# Operational Design Specification: Production Webhook Receiver (Node.js)

**Version:** 1.0
**Status:** Authoritative handoff document
**Scope:** Single containerized webhook receiver service — implementation decisions and operational requirements

---

## Preamble

This document exists because the gap between "validates signatures" and "operates safely in production" is where incidents live. Every decision recorded here reflects a failure mode that was considered and either mitigated or explicitly accepted. Every operational requirement is a production constraint, not a suggestion. An engineer deploying this service should be able to answer any operational question by reading this document. If they cannot, the document has failed.

# Part I: Implementation Decisions

---

## 1. Ingestion Layer

### 1.1 HTTP Framework

Use a minimal Node.js HTTP framework — Express or Fastify — with no middleware that touches the raw request body before signature validation. The raw body must be preserved exactly as received from the sender. Any transformation — JSON parsing, body normalization, encoding conversion — applied before signature validation will cause validation to fail on legitimate requests and cannot be debugged quickly under pressure.

**Decision:** Raw body must be captured as a `Buffer` before any parsing occurs. Store it on the request object as `req.rawBody`. JSON parsing happens after signature validation succeeds, and only then.

**Rationale:** HMAC-SHA256 signatures are computed against the raw payload bytes. GitHub, Stripe, and equivalent providers sign the exact bytes they transmit. A single middleware that calls `JSON.parse` on the body before validation is a silent failure mode that surfaces as "all webhooks failing" after a dependency upgrade.

### 1.2 Body Size Limit

Enforce a hard body size limit before reading the body. Do not buffer an unbounded payload into memory.

**Decision:** Default limit of 5MB. This is intentionally generous for legitimate webhook payloads (which are typically under 100KB) while preventing a trivial memory exhaustion attack via a single large POST.

**Rationale:** A public endpoint with no body size limit is an invitation to send a 2GB payload and exhaust container memory. The limit must be enforced at the read stage, not after the body is fully buffered.

**Failure mode if omitted:** A single malformed or malicious request can exhaust container memory, triggering an OOM kill. The container restarts. Legitimate events queued in memory at the time of the kill are lost. The attacker has achieved denial of service with one HTTP request.

### 1.3 Accepted Methods and Content Types

Accept only `POST` requests. Return `405 Method Not Allowed` for all other methods. Do not return `404` — a `405` signals that the endpoint exists but the method is wrong, which is the accurate and operationally useful response.

Accept `application/json` and `application/x-www-form-urlencoded` only if the upstream provider requires it. Reject other content types with `415 Unsupported Media Type`. Do not attempt to parse payloads with unexpected content types.

## 2. Signature Validation

### 2.1 Algorithm

HMAC-SHA256. The upstream provider (GitHub, Stripe, or equivalent) dictates this. Do not negotiate or accept alternative algorithms. If a request arrives claiming a different algorithm in its signature header, reject it.

### 2.2 Signature Header Parsing

GitHub sends: `X-Hub-Signature-256: sha256=<hex_digest>`
Stripe sends: `Stripe-Signature: t=<timestamp>,v1=<hex_digest>`

**Decision:** Parse the signature header strictly. If the header is absent, malformed, or does not match the expected format, reject with `400 Bad Request`. Do not attempt to recover or infer intent from a malformed header.

**Decision:** Extract the raw hex digest from the header before comparison. Do not compare the full header string against a computed value — this creates format-coupling that breaks silently when the provider changes header formatting.

### 2.3 Constant-Time Comparison

**Decision:** Use Node.js `crypto.timingSafeEqual()` for all signature comparisons. No exceptions.

**Rationale:** String equality in JavaScript short-circuits on the first mismatched character. This creates a timing side channel that allows an attacker to iteratively determine the correct signature byte by byte. `timingSafeEqual` eliminates this by comparing all bytes in constant time regardless of where the mismatch occurs.

**Failure mode if omitted:** A timing oracle on a public endpoint. Exploitable in practice given sufficient request volume and network stability.

### 2.4 Secret Management

The HMAC secret is a runtime secret. It must never appear in source code, container images, or version control.

**Decision:** Inject the secret exclusively via environment variable at container runtime. The application reads it once at startup and holds it in memory. It is never logged, never included in error responses, and never written to disk.

**Decision:** The application must fail to start if the secret environment variable is absent or empty. A receiver that starts without a configured secret will either crash on the first request or, worse, validate all requests against an empty string — which is a valid HMAC key and will produce consistent but meaningless signatures.

### 2.5 Secret Rotation

Secret rotation is a planned operational event that must not require downtime.

**Decision:** Support dual-secret validation during rotation windows. When a rotation is in progress, the application must attempt validation against both the current secret and the incoming secret. A request is valid if it passes validation against either. The rotation window is closed by removing the old secret from configuration and redeploying.

**Rationale:** GitHub and Stripe both support a rotation period during which both old and new secrets are valid. If the receiver only accepts the new secret before the provider has fully rotated, legitimate events will be rejected. This is a production incident that looks like an attack but is self-inflicted.

**Operational requirement:** The rotation procedure must be documented and tested before it is needed. See Part II, Section 2.2.

### 2.6 Rejection Behavior

On signature validation failure, return `403 Forbidden`. Do not return `401` — that implies the request could succeed with different credentials in the same format. Do not return `400` — that implies the request is malformed. A `403` is accurate: the request was understood and rejected.

Do not include any detail in the response body on validation failure. "Invalid signature" is more information than an attacker needs to confirm they are targeting the right endpoint.

## 3. Replay Attack Mitigation

### 3.1 Timestamp Validation (Stripe-style providers)

Stripe includes a request timestamp in the signature header (`t=<unix_timestamp>`). This timestamp is included in the signed payload, meaning the signature covers both the body and the timestamp.

**Decision:** Reject requests where the timestamp is more than 5 minutes old or more than 60 seconds in the future. This window is a deliberate tradeoff between replay protection and tolerance for clock skew between the provider and the receiver.

**Rationale:** Without timestamp validation, a captured valid request can be replayed indefinitely. The signature will remain valid. The receiver will process it. Downstream systems will see duplicate events.

**Decision:** The tolerance window (5 minutes, 60 seconds) must be configurable via environment variable, not hardcoded. Operational conditions — processing backlogs, clock synchronization issues — may require temporary adjustment without a code deployment.

### 3.2 Idempotency Keys (GitHub-style providers)

GitHub does not include a timestamp in the signed payload. Replay protection requires a different mechanism.

**Decision:** Extract the event delivery ID from the `X-GitHub-Delivery` header (or provider equivalent). Before queuing the event, check this ID against a short-term deduplication store (Redis with TTL, or equivalent). If the ID is already present, return `200 OK` without re-queuing. If absent, store the ID and queue the event.

**Rationale:** Returning `200` on a duplicate is correct. The provider considers the event delivered. Re-queuing it would cause duplicate processing downstream. The deduplication store TTL should be set to the maximum window in which a replay is considered a threat — typically 24 hours.

**Failure mode if omitted:** A provider retry storm — which is a normal operational event when the receiver has downtime — will result in every event being processed multiple times. Downstream systems must be idempotent, but they should not be required to compensate for a receiver that does not deduplicate.

## 4. Async Processing and Queue

### 4.1 Queue Selection

**Decision:** Use an external durable queue — Redis with a list or stream, RabbitMQ, SQS, or equivalent. Do not use an in-process queue (EventEmitter, in-memory array, or similar).

**Rationale:** An in-process queue does not survive container restart. Any events buffered in memory at the time of an OOM kill, a deployment, or a crash are permanently lost. The receiver's contract is to accept and preserve events. An in-process queue cannot fulfill that contract.

**Decision:** The queue must be durable. For Redis, this means AOF or RDB persistence configured appropriately. For SQS, durability is provided by default. The specific configuration depends on the queue technology selected, but the requirement — events survive a queue service restart — is non-negotiable.

### 4.2 Enqueue Operation

**Decision:** The enqueue operation is the last step in the request path, after signature validation and deduplication. The response to the provider is sent only after the enqueue operation succeeds.

**Rationale:** Returning `200` before the event is durably queued creates a window where the receiver has acknowledged receipt but the event is not yet preserved. If the process crashes in that window, the event is lost. The provider believes it was delivered.

**Decision:** If the enqueue operation fails (queue unavailable, connection timeout), return `503 Service Unavailable`. The provider will retry. This is the correct behavior — it is better to force a retry than to acknowledge an event that was not preserved.

**Failure mode if omitted:** Returning `200` before enqueue means the receiver lies to the provider. Events are silently lost during queue outages. There is no retry. There is no recovery.

### 4.3 Enqueue Payload

**Decision:** The payload written to the queue must include:
- The parsed event body
- The raw event body (for downstream reprocessing if needed)
- The event type header (e.g., `X-GitHub-Event`, `Stripe-Signature` event type)
- The delivery ID / idempotency key
- The timestamp at which the receiver accepted the event
- The provider source identifier (if the receiver handles multiple providers)

**Rationale:** Downstream processors should not need to re-derive metadata from the raw payload. The receiver has already validated and parsed the event — that context should be preserved in the queue message.

### 4.4 Queue Backpressure

**Decision:** The receiver must enforce a maximum queue depth or apply backpressure when the queue is saturated. If the queue is full or the enqueue operation times out after a configurable deadline, return `503`. Do not block the request thread indefinitely waiting for queue capacity.

**Rationale:** A downstream processing backlog will cause the queue to fill. Without backpressure, the receiver will continue accepting events it cannot preserve, either blocking indefinitely or silently dropping them. A `503` forces the provider to retry later, which is the correct behavior when the system is overloaded.

## 5. Response Behavior

### 5.1 Response Codes

| Condition | Response Code | Notes |
|---|---|---|
| Event accepted and queued | `200 OK` | Empty body or minimal acknowledgment |
| Missing or malformed signature header | `400 Bad Request` | No detail in body |
| Signature validation failure | `403 Forbidden` | No detail in body |
| Unsupported HTTP method | `405 Method Not Allowed` | Include `Allow: POST` header |
| Unsupported content type | `415 Unsupported Media Type` | No detail in body |
| Duplicate event (idempotency) | `200 OK` | Indistinguishable from first receipt |
| Queue unavailable | `503 Service Unavailable` | Include `Retry-After` header |
| Payload too large | `413 Content Too Large` | No detail in body |
| Unhandled internal error | `500 Internal Server Error` | No detail in body |
