# Operational Design Specification: Production Webhook Receiver

**Service:** Webhook Receiver
**Runtime:** Node.js, containerized
**Classification:** Public-facing ingress, security boundary
**Status:** Pre-implementation handoff

---

## Part I: Implementation Decisions

### 1. Request Ingestion

**Decision: Preserve the raw request body before any parsing.**

HMAC-SHA256 signature validation operates on the exact bytes transmitted by the sender. Any middleware that parses, normalizes, or re-serializes the body before validation will corrupt the signature check. The raw buffer must be captured and held alongside the parsed representation. These are two separate artifacts with two separate purposes. Do not conflate them.

The raw body must be captured with a hard size limit. Requests exceeding that limit are rejected with `413` before signature validation runs. This is not a courtesy — it is a resource protection boundary. A reasonable ceiling is 5MB; calibrate against the maximum payload the upstream service can legitimately produce, then add no more than a 20% margin. Generosity here is a liability.

**Decision: Reject non-POST methods at the router boundary, before any body reading occurs.**

There is no legitimate reason for a GET, PUT, or OPTIONS request to reach signature validation. Fail fast, fail cheap.

---

### 2. Signature Validation

**Decision: Validate using a constant-time comparison function exclusively.**

Node.js provides `crypto.timingSafeEqual`. It must be used. String equality operators (`===`) are not acceptable for comparing HMAC digests. Timing side-channels on a public endpoint are a real attack surface, not a theoretical one.

**Decision: Compute the expected signature from the raw body buffer and the shared secret. Compare against the signature provided in the request header.**

The header name is provider-specific (`X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). This must be treated as configuration, not a hardcoded string, so the receiver can be adapted without code changes.

**Decision: Reject any request missing the signature header with `401`, not `400`.**

`400` signals a malformed request. A missing signature header is an authentication failure. The distinction matters for alerting and for downstream log analysis.

**Decision: Reject any request with an invalid signature with `401`. Return no diagnostic detail in the response body.**

The response body on a `401` must be empty or contain only a static, non-descriptive string. Do not echo back what was received. Do not indicate whether the header was present but invalid versus absent entirely. Any information returned on a validation failure is information available to an attacker probing the endpoint.

**Decision: The shared secret is never logged, never included in error messages, and never exposed in health check output.**

This is an absolute constraint, not a best-effort guideline.

---

### 3. Secret Management

**Decision: Secrets are injected via environment variables at runtime. They are never present in the container image, the repository, or any build artifact.**

The container image must be auditable without exposing secrets. If the image is pushed to a registry, the secret must not be in it.

**Decision: Support secret rotation without a service restart.**

The receiver must be capable of validating against two active secrets simultaneously during a rotation window: the outgoing secret and the incoming secret. Validation succeeds if either secret produces a matching signature. This is not optional — a rotation strategy that requires a maintenance window or accepts a validation gap is not a rotation strategy.

The rotation window is bounded. Both secrets are not valid indefinitely. The window closes when the upstream provider confirms all in-flight events were signed with the new secret, or after a defined maximum duration (e.g., 30 minutes), whichever comes first.

**Decision: Secret loading is handled by a dedicated module with a defined interface. The rest of the application does not read environment variables directly for secrets.**

This makes the rotation logic testable and the secret surface area auditable.

---

### 4. Async Processing and Queuing

**Decision: The HTTP response is sent before any downstream processing begins.**

The request handler does exactly three things in the request path: validate the signature, enqueue the raw event, return `200`. Nothing else. No database writes, no downstream HTTP calls, no business logic.

**Decision: The queue is an external, durable queue — not an in-process structure.**

An in-process queue (array, EventEmitter, etc.) does not survive a process crash, a container restart, or a deployment. Events in flight at the moment of a crash are lost silently. This is not acceptable for a production system. The queue must be external and must provide at-least-once delivery guarantees. Redis Streams, RabbitMQ, SQS, and equivalent systems are acceptable. An in-memory structure is not.

**Decision: The enqueue operation is the only I/O in the request path.**

If the enqueue operation fails (queue unavailable, timeout, connection error), the receiver returns `503`, not `200`. Returning `200` when the event has not been durably queued is a lie to the upstream provider. Most providers will not retry on `200`. A silent drop is worse than a visible failure.

**Decision: The enqueue operation has an explicit timeout.**

The enqueue call must not block the request indefinitely. A timeout of 2–3 seconds is appropriate. If the queue does not accept the message within that window, the request fails with `503`. The upstream provider retries. The event is not lost.

**Decision: The payload written to the queue includes the raw body, the parsed headers, the provider-assigned event ID (if present), a receiver-assigned UUID, and a receipt timestamp.**

The receiver-assigned UUID is the correlation handle for all downstream tracing. The receipt timestamp is the authoritative ingestion time. Neither is optional.

---

### 5. Idempotency

**Decision: The receiver records event IDs and rejects duplicates before enqueuing.**

Most providers guarantee at-least-once delivery. Replayed events are not a hypothetical — they occur during provider-side retries, during network instability, and during provider incidents. The receiver must deduplicate on the provider-assigned event ID using a short-lived store (Redis with a TTL keyed to the provider's maximum retry window is appropriate). Duplicate events receive `200` immediately — they are not errors from the provider's perspective — but they are not enqueued again.

If the provider does not supply a stable event ID, the receiver must construct a deterministic fingerprint from the payload and timestamp. This is a fallback, not a preference. Push back on any provider integration that lacks stable event IDs.

---

### 6. Error Handling and Response Codes

The response code contract must be defined explicitly and adhered to without exception:

| Condition | Response Code | Notes |
|---|---|---|
| Valid signature, successfully enqueued | `200` | The only success path |
| Missing or invalid signature | `401` | No body detail |
| Payload exceeds size limit | `413` | Before signature validation |
| Wrong HTTP method | `405` | Before body reading |
| Queue unavailable or enqueue timeout | `503` | Provider will retry |
| Unexpected internal error | `500` | Logged; no stack trace in response |

No other response codes are returned by this service. Any deviation from this contract is a bug.

---

### 7. Transport Security

**Decision: TLS termination occurs at the load balancer or ingress layer. The Node.js process listens on plain HTTP internally.**

The container does not manage TLS certificates. This is the responsibility of the infrastructure layer. The specification must explicitly state that the ingress layer must enforce TLS 1.2 minimum, and that plain HTTP connections from the public internet must be rejected, not redirected.

**Decision: The receiver does not validate the source IP of incoming requests as a primary security control.**

IP allowlisting is a defense-in-depth measure, not a replacement for signature validation. Some providers publish IP ranges; those ranges should be enforced at the network layer if available, but the application layer treats signature validation as the authoritative control regardless.

## Part II: Operational Layer

### 8. Health Checks

**8.1 Liveness Probe**

The liveness probe confirms the process is alive and the HTTP server is accepting connections. It returns `200` with a static body. It performs no I/O. It does not check queue connectivity. A liveness probe that performs I/O will cause the container to be killed and restarted during a queue outage — which is precisely when the container must remain alive to return `503` correctly and allow the provider to retry.

**8.2 Readiness Probe**

The readiness probe confirms the service is ready to accept and process traffic. It checks queue connectivity. If the queue is unreachable, the readiness probe returns a non-200 status, and the container is removed from the load balancer rotation. This is the correct behavior — a receiver that cannot enqueue events must not receive events.

The readiness probe must be implemented as a lightweight ping against the queue, not a full enqueue operation. It must have its own timeout, shorter than the probe interval, to prevent probe stacking.

**8.3 Separation is Non-Negotiable**

Liveness and readiness must be separate endpoints. Conflating them produces a system that either restarts unnecessarily during dependency outages or fails to remove itself from rotation when it cannot function. Both outcomes are worse than the complexity of maintaining two endpoints.

### 9. Structured Logging

**9.1 Log Format**

All logs are emitted as structured JSON to stdout. No unstructured text. No logs to files. The container runtime and log aggregation pipeline handle collection and routing.

**9.2 Required Fields on Every Log Line**

Every log entry must include: timestamp (ISO 8601, UTC), log level, service name, version, environment, and the receiver-assigned request UUID. These fields are not optional on any log line. A log line without a correlation ID is a log line that cannot be traced under pressure.

**9.3 What Is Logged**

- Every inbound request: method, path, content-length, provider event ID if present, receiver UUID
- Every validation outcome: pass or fail (not the signature value, not the secret)
- Every enqueue outcome: success, failure, timeout — with duration
- Every duplicate event detected: provider event ID, receiver UUID
- Every `503` or `500` response: full error detail including stack trace, queue error message if applicable

**9.4 What Is Never Logged**

- The shared secret, in whole or in part
- The raw signature header value
- The raw request body
- Any field from the payload that is classified as PII or sensitive by the data classification policy

This list is a constraint on the logging implementation, not a suggestion. It must be enforced in code review.

**9.5 Log Levels**

- `INFO`: successful ingestion, enqueue confirmation
- `WARN`: duplicate event detected, queue latency above threshold
- `ERROR`: validation failure, enqueue failure, unexpected exception

Do not use `DEBUG` level in production. If debug logging is needed during an incident, it is enabled via a runtime flag, not a deployment.

### 10. Metrics

**10.1 Required Metrics**

The following metrics must be emitted and must be queryable in the monitoring system before the service goes to production. "We'll add metrics later" is how you discover an incident three days after it started.

| Metric | Type | Labels |
|---|---|---|
| `webhook_requests_total` | Counter | `status` (200/401/413/503/500) |
| `webhook_signature_validation_duration_ms` | Histogram | `result` (pass/fail) |
| `webhook_enqueue_duration_ms` | Histogram | `result` (success/failure/timeout) |
| `webhook_enqueue_failures_total` | Counter | `reason` (timeout/unavailable/error) |
| `webhook_duplicate_events_total` | Counter | — |
| `webhook_payload_size_bytes` | Histogram | — |

**10.2 Metric Emission**

Metrics are emitted in Prometheus format via a `/metrics` endpoint. This endpoint is not exposed on the public-facing port. It is exposed on a separate internal port, accessible only within the cluster network. Exposing internal metrics to the public internet is an information disclosure vulnerability.

**10.3 Baseline Before Alerting**

Alerts must be calibrated against observed baseline traffic. An alert threshold set before baseline is established is a threshold set arbitrarily. Spend the first week in production collecting baseline data before committing alert thresholds to configuration.

### 11. Alerting

**11.1 Alert: Elevated Signature Validation Failure Rate**

Condition: Signature validation failure rate exceeds 1% of total requests over a 5-minute window.

Severity: High. Page immediately.

Rationale: A sustained failure rate above noise indicates one of three things — a secret rotation that was not coordinated with the receiver, a misconfiguration, or an active probing attack. All three
