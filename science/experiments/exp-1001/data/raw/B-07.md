# Operational Design Specification: Production Webhook Receiver (Node.js)

**Version:** 1.0
**Status:** Authoritative handoff document
**Scope:** Single containerized webhook receiver service — implementation decisions and full operational layer

---

## Preamble

This document specifies every decision and requirement necessary to deploy, operate, and debug a production webhook receiver safely. Any requirement absent from this document is absent from the system. Engineers implementing or operating this service are expected to treat this document as complete and binding. Where a decision is marked as a constraint rather than a recommendation, deviation requires explicit written justification.

---

# Part I: Implementation Decisions

---

## 1. Ingestion Layer

### 1.1 HTTP Framework

Use a minimal Node.js HTTP framework (Fastify or raw `http` module). Express is acceptable but must not be configured with any body-parsing middleware that consumes the raw request body before signature validation. This is a hard constraint.

**Rationale:** HMAC-SHA256 signature validation operates on the raw request body bytes exactly as transmitted. Any middleware that parses, re-serializes, or normalizes the body before validation will silently corrupt the signature check. The framework must expose the raw buffer to the validation layer before any transformation occurs.

### 1.2 Raw Body Capture

The raw request body must be captured as a `Buffer` before any parsing occurs. The size of this buffer must be bounded. Set a maximum request body size of **5MB** as a hard ceiling, rejecting anything larger with `413 Payload Too Large` before reading the full body.

**Rationale:** An unbounded body read is a denial-of-service vector on a public endpoint. The 5MB ceiling is deliberately generous relative to any realistic webhook payload; it exists to bound memory consumption per request under adversarial conditions, not to match expected payload sizes.

### 1.3 Accepted HTTP Methods

Accept only `POST`. Return `405 Method Not Allowed` for all other methods. Do not redirect. Do not explain.

### 1.4 Content-Type Handling

Do not reject requests based on `Content-Type` alone. Some providers send `application/json`, others send `application/x-www-form-urlencoded`, others send custom types. Signature validation operates on the raw body regardless. Parse the body only after signature validation passes, and parse it according to the `Content-Type` header at that point.

## 2. Signature Validation

### 2.1 Algorithm

HMAC-SHA256. No other algorithm is accepted. The signature header format follows the provider's convention (e.g., GitHub: `X-Hub-Signature-256: sha256=<hex>`, Stripe: `Stripe-Signature: t=<timestamp>,v1=<hex>`). The service must be configured for the specific provider's header format and prefix stripping logic before deployment.

### 2.2 Secret Management

The HMAC secret is injected as an environment variable at container startup. It is never logged, never included in error responses, never written to disk, and never present in any configuration file committed to source control.

The secret must be stored in a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, or equivalent) and injected into the container environment at runtime by the orchestration layer. The mechanism for injection must be documented in the deployment runbook. This document does not specify the secrets manager; the deployment engineer must select and document one.

### 2.3 Comparison Method

Signature comparison must use a constant-time comparison function. Node.js `crypto.timingSafeEqual` is the required implementation. String equality (`===`) is forbidden for signature comparison. This is a hard constraint.

**Rationale:** Timing attacks against naive string comparison are a known, practical attack class against HMAC validation on public endpoints. The performance cost of constant-time comparison is negligible. The security cost of not using it is unbounded.

### 2.4 Validation Failure Response

Return `403 Forbidden` on signature mismatch. Return no body, or a static body with no diagnostic detail. Do not indicate whether the failure was a missing header, an incorrect signature, or a malformed signature. All three conditions return the same response.

**Rationale:** Differentiating failure modes in the response leaks information to an attacker probing the validation logic.

### 2.5 Timestamp Validation (Provider-Dependent)

Where the provider includes a timestamp in the signed payload or signature header (Stripe's `t=` field is the canonical example), validate that the timestamp is within a configurable tolerance window of the current server time. The default tolerance is **five minutes**. Reject events outside this window with `403 Forbidden`.

**Rationale:** Timestamp validation defeats replay attacks. Without it, a valid captured webhook can be replayed indefinitely. Five minutes is the industry-standard tolerance; it must be tunable via environment variable to accommodate clock skew in constrained environments without requiring a code change.

### 2.6 Replay Protection Beyond Timestamp

Timestamp windows alone do not prevent replay within the window. If the downstream system is not idempotent by design, implement a short-lived deduplication store (Redis or equivalent) keyed on the provider's event ID. The TTL on deduplication keys must exceed the timestamp tolerance window. If the downstream system is idempotent, document that fact explicitly in the runbook and omit the deduplication store.

The choice between these two paths must be made before deployment and recorded in the runbook. Defaulting to "we'll handle it downstream" without documenting that assumption is not acceptable.

## 3. Response Behavior

### 3.1 Immediate 200 Acknowledgment

The service returns `200 OK` immediately upon successful signature validation and successful enqueue of the event. The response is returned before any downstream processing occurs. Processing is fully asynchronous and decoupled from the HTTP response path.

**Rationale:** Provider retry policies are triggered by non-200 responses or timeouts. Any processing in the request path that can fail or block will cause the provider to retry, producing duplicate deliveries. The receiver's only job in the request path is to validate and enqueue.

### 3.2 Response on Queue Failure

If the enqueue operation fails (queue unavailable, timeout, capacity exceeded), return `500 Internal Server Error`. Do not return `200`. The provider will retry. This is the correct behavior: a `200` that lies about successful enqueue produces silent data loss.

**Rationale:** This is the one case where the request path must block briefly — on the enqueue acknowledgment. The enqueue call must have a hard timeout (recommended: **2 seconds**). If the queue does not acknowledge within that window, return `500` and let the provider retry.

### 3.3 Response Body

Return a minimal, static response body on `200` (e.g., `{"ok":true}`). Return no diagnostic information in error responses. Error responses may include a static, non-descriptive message for operational convenience (e.g., `{"error":"bad request"}`), but must not include stack traces, internal error messages, or any information that characterizes the internal system state.

## 4. Async Processing and Queue Design

### 4.1 Queue Selection

The queue must be an external, durable message queue. In-process queues (arrays, EventEmitter, in-memory queues) are forbidden. Acceptable options include: AWS SQS, Google Cloud Pub/Sub, RabbitMQ (with durable queues and publisher confirms), Redis Streams (with consumer groups and persistence enabled), or equivalent.

**Constraint:** The queue must provide at-least-once delivery guarantees. Exactly-once delivery is preferred where available but must not be assumed. Downstream consumers must be designed for idempotency regardless of queue guarantees.

### 4.2 Message Envelope

Each message enqueued must include:

- The raw event payload (as received, before any parsing)
- The provider event type (from the provider's event header, e.g., `X-GitHub-Event`)
- The provider event ID (from the provider's delivery ID header, e.g., `X-GitHub-Delivery`, `Stripe-Signature`'s `id` field)
- The timestamp of receipt (server time, ISO 8601, UTC)
- The source IP of the request
- A service-generated trace ID (see Section 5.3)

**Rationale:** Downstream consumers must be able to process events without re-fetching from the provider. The envelope must contain everything a consumer needs. The receipt timestamp and source IP are required for forensic analysis of incidents.

### 4.3 Queue Capacity and Backpressure

The receiver does not implement backpressure against the provider. If the queue is full or unavailable, the receiver returns `500` and relies on provider retry. The receiver must not buffer events in memory to absorb queue unavailability. Memory buffering under queue failure converts a transient queue outage into a data loss event when the process restarts.

## 5. Observability

### 5.1 Structured Logging

All log output must be structured JSON. No plaintext log lines. Every log entry must include at minimum: timestamp (ISO 8601, UTC), log level, service name, service version, and trace ID. The trace ID must be generated per request and propagated through every log entry for that request's lifecycle.

Log levels must be configurable at runtime without restart (via environment variable or signal). Default level in production: `INFO`.

### 5.2 What Must Be Logged

**On every request:**
- Request received: method, path, provider event type header, provider event ID header, source IP, trace ID
- Signature validation result: pass or fail (never log the signature value or the secret)
- Enqueue result: success or failure, queue latency in milliseconds
- Response sent: status code, total request duration in milliseconds

**On validation failure:**
- Log at `WARN` level
- Include source IP and provider event type if available
- Do not log the received signature value

**On queue failure:**
- Log at `ERROR` level
- Include error class and message (not stack trace in structured field — stack trace to a separate debug field, omitted at INFO level)

**Never log:**
- The HMAC secret
- The raw request body (it may contain PII or sensitive financial data)
- Full signature header values

### 5.3 Trace ID Generation

Generate a UUID v4 per request at the point of first byte received. Attach to all log entries for that request. Include in the enqueued message envelope. Return in the response as `X-Trace-Id` header. This header is the primary debugging handle for correlating a provider delivery with internal processing.

### 5.4 Metrics

The service must emit the following metrics. The metrics backend (Prometheus, Datadog, CloudWatch, etc.) is not specified here and must be selected and documented in the deployment runbook.

| Metric | Type | Labels |
|---|---|---|
| `webhook_requests_total` | Counter | `status` (200, 403, 413, 500), `event_type` |
| `webhook_signature_failures_total` | Counter | `reason` (missing_header, mismatch, expired_timestamp) |
| `webhook_enqueue_duration_ms` | Histogram | `result` (success, failure) |
| `webhook_request_duration_ms` | Histogram | `status` |
| `webhook_queue_failures_total` | Counter | `error_class` |

Metrics must be exposed on a separate port from the webhook receiver endpoint. The metrics port must not be publicly accessible.

## 6. Security Hardening

### 6.1 TLS

The service must terminate TLS. In a containerized deployment behind a load balancer, TLS termination at the load balancer is acceptable, provided the internal network path from load balancer to container is trusted and documented. The public endpoint must not accept plaintext HTTP. This is a hard constraint.

### 6.2 IP Allowlisting

Where the provider publishes a list of source IP ranges for webhook delivery (GitHub does; Stripe does), configure allowlisting at the network layer (security group, firewall rule, or load balancer policy) as a defense-in-depth measure. This is not a substitute for signature validation. Both controls must be in place simultaneously.

Where the provider does not publish source IP ranges, document that fact and do not implement allowlisting. Do not implement allowlisting based on undocumented or inferred IP ranges.

### 6.3 Rate Limiting

Implement rate limiting at the endpoint level. The rate limit must be configurable via environment variable. A reasonable default is **1000 requests per minute per source IP**. Requests exceeding the rate limit receive `429 Too Many Requests`. Rate limit state must be stored externally (Redis or equivalent) if the service runs more than one replica.

**Rationale:** A public endpoint with no rate limiting is an open amplification surface. Provider retry storms and adversarial traffic both require a ceiling
