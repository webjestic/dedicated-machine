## 1. Scope and Boundaries

This specification covers the webhook receiver only — from the moment a POST request arrives at the service boundary to the moment a validated event is durably enqueued. It does not cover downstream consumers, event routing logic, or business processing of event payloads.

**What this service owns:**
- TLS termination boundary (upstream, via load balancer or ingress — not this service)
- Raw body capture and preservation
- HMAC-SHA256 signature validation
- Idempotency key extraction and deduplication
- Durable enqueue of validated events
- Immediate HTTP response

**What this service does not own:**
- Consumer processing guarantees
- Event schema validation beyond structural prerequisites for deduplication
- Retry logic for downstream consumers
- Business logic of any kind

Any requirement not listed under "what this service owns" is explicitly out of scope and must be addressed in a separate specification.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework:** Express or Fastify. Either is acceptable. The decision is binding — do not mix. Rationale: both have well-understood middleware ordering semantics. Middleware ordering is a correctness concern, not a style concern.

**Raw body preservation:** The HMAC signature is computed against the raw request body as transmitted by the sender. Any middleware that parses, transforms, or re-serializes the body before signature validation will silently corrupt the validation input. The raw body must be captured as a `Buffer` before any parsing occurs and stored on the request object for the validation step. Body parsing middleware must not run before raw body capture. This ordering must be enforced structurally, not by convention.

**Body size limit:** Enforce a maximum body size at the HTTP layer before any processing. The limit must be set to the maximum payload size documented by the third-party provider, plus a small fixed margin (e.g., provider max + 10%). Requests exceeding this limit must be rejected with `413` before signature validation runs. This prevents memory exhaustion from oversized payloads regardless of signature validity.

**TLS:** This service must not terminate TLS. TLS must be terminated upstream (load balancer, ingress controller, or sidecar). The service must be configured to trust the upstream termination boundary and must not be reachable directly on a public interface without TLS in front of it. If the deployment topology cannot guarantee this, this specification is incomplete.

**HTTP method enforcement:** Only `POST` is accepted on the webhook endpoint. All other methods return `405`. This is not a default — it must be explicitly enforced.

**Endpoint path:** The endpoint path must be non-guessable or at minimum non-default. A path of `/webhook` is not acceptable in production. A path that includes a service-specific or environment-specific token (e.g., `/hooks/github/rcv-a3f9`) raises the cost of endpoint discovery without replacing signature validation as the primary control.

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. The signature is computed over the raw request body using a shared secret. The computed signature is compared against the value provided in the request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).

**Comparison method:** Signature comparison must use a constant-time equality function. Node.js provides `crypto.timingSafeEqual`. Using string equality (`===`) is not acceptable. Timing attacks against naive string comparison are not theoretical — they are documented and exploitable against public endpoints.

**Failure behavior:** Any request that fails signature validation must be rejected with `401`. The response body must be minimal — do not echo back the received signature, the expected signature, or any detail that aids an attacker in understanding the validation mechanism. Log the failure internally with full context (see §3.1).

**Validation must be synchronous and complete before any other processing.** No event data is read, no queue interaction occurs, and no downstream system is contacted until signature validation passes. This is a correctness and security requirement.

**Header absence:** If the signature header is absent, reject with `401`. Do not fall through to processing. Absence of the header is not a malformed request — it is an unsigned request, which is treated identically to a failed validation.

**Algorithm agility:** The validation implementation must not hard-code the prefix parsing logic in a way that silently accepts a downgraded algorithm. For example, GitHub sends `sha256=<hex>`. The prefix must be validated explicitly. A payload with a header of `sha1=<value>` must not pass SHA-256 validation by accident or by omission.

### 2.3 Async Decoupling and Queue Contract

**Requirement:** The HTTP response must be returned before any downstream processing occurs. Processing must not block the response path. This is a correctness requirement, not a performance preference — third-party senders have response timeouts, and a blocked response path will cause the sender to retry, producing duplicate events.

**Queue technology:** Select one of: Redis Streams, RabbitMQ, AWS SQS, or Google Cloud Pub/Sub. The selection must be made before implementation and must account for the deployment environment. The queue must provide at-least-once delivery guarantees. Exactly-once delivery is not assumed and must not be relied upon by consumers.

**Enqueue operation:** The enqueue operation is in the request path — it occurs before the `200` is returned. This is intentional. The `200` response is a contract with the sender that the event has been accepted. Returning `200` before the event is durably enqueued breaks that contract and creates a window where the event is lost with no retry triggered. If the enqueue fails, return `500` to the sender. The sender will retry.

**Payload envelope:** The message placed on the queue must include:
- The raw event payload (as received, not re-serialized)
- The event type header value (e.g., `X-GitHub-Event`, `Stripe-Event`)
- The delivery ID / idempotency key extracted from the request headers
- The timestamp of receipt (server-side, not sender-provided)
- The source identifier (which provider, which endpoint)

Do not enqueue only the parsed payload. The raw body and headers are the authoritative record. Downstream consumers may need them for their own validation or auditing.

**Enqueue timeout:** The enqueue operation must have an explicit timeout. If the queue is unreachable and the timeout expires, return `503`. Do not hang the connection indefinitely. The timeout value must be set below the sender's documented retry threshold.

### 2.4 Idempotency and Deduplication

**Delivery IDs:** Most production webhook providers include a delivery ID in request headers (e.g., `X-GitHub-Delivery`, `Stripe-Idempotency-Key`). This ID uniquely identifies a delivery attempt. Extract this value on every request.

**Deduplication window:** Implement a deduplication check using the delivery ID against a short-lived store (Redis with TTL is the standard choice). The TTL must be set to at least the sender's maximum retry window — typically 24–72 hours depending on the provider. Consult the provider's documentation and set the TTL explicitly. Do not guess.

**Deduplication behavior:** If a delivery ID has already been processed (i.e., the key exists in the deduplication store), return `200` immediately without re-enqueuing. Do not return `409`. The sender does not need to know the event was a duplicate — returning `200` terminates the retry loop cleanly.

**Deduplication store failure:** If the deduplication store is unavailable, the decision is: fail open (allow potential duplicates through) or fail closed (return `503`). This decision must be made explicitly and documented. The default recommendation is fail open with a metric increment and alert. Consumers must be designed for at-least-once delivery regardless. Failing closed on a deduplication store outage means losing events during that outage — which is typically worse than processing duplicates.

**Absence of delivery ID:** If the provider does not supply a delivery ID, generate a deterministic ID from the request body hash and the receipt timestamp. Document this as a fallback — it is not equivalent to a provider-issued ID and deduplication guarantees are weaker.

### 2.5 Error Classification and Response Codes

The response code returned to the sender determines whether the sender retries. This mapping is not advisory — it is the retry control plane.

| Condition | Response Code | Sender Behavior |
|---|---|---|
| Valid signature, event enqueued | `200` | No retry |
| Missing or invalid signature | `401` | Sender should not retry (bad secret or attack) |
| Unsupported HTTP method | `405` | No retry |
| Body exceeds size limit | `413` | No retry |
| Duplicate delivery ID | `200` | No retry |
| Queue unavailable / enqueue failed | `503` | Sender retries |
| Deduplication store unavailable (fail closed) | `503` | Sender retries |
| Unhandled internal error | `500` | Sender retries |

**Do not return `200` for any failure condition.** A `200` to a failed enqueue is an event loss with no recovery path.

**Do not return `4xx` for transient infrastructure failures.** A `429` or `400` for a queue timeout will suppress sender retries and lose the event.

### 2.6 Secret Management

**Storage:** The HMAC secret must not be stored in the container image, in source control, or in environment variable definitions committed to source control. It must be injected at runtime via a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, Kubernetes Secrets with appropriate RBAC).

**Access at runtime:** The secret must be loaded at process startup and held in memory. It must not be re-fetched on every request — that introduces latency and a dependency on the secrets manager in the hot path.

**Multiple secrets:** The implementation must support holding more than one active secret simultaneously. This is required for secret rotation (see §3.5). The validation logic must attempt validation against all active secrets and pass if any succeeds.

**Secret logging:** The secret value must never appear in logs, error messages, stack traces, or metrics labels. This must be enforced structurally — the secret must not be passed as a string through any logging path.

## 3. Operational Layer

### 3.1 Observability: Structured Logging

Every request must produce exactly one structured log entry at the point of response. Log entries must be machine-parseable (JSON). Log entries must not be assembled from string concatenation.

**Required fields on every log entry:**

| Field | Description |
|---|---|
| `timestamp` | ISO 8601, UTC |
| `request_id` | Generated per-request, returned in response header `X-Request-ID` |
| `delivery_id` | Provider-issued delivery ID, or generated fallback |
| `event_type` | Value of the event type header |
| `source` | Provider identifier (e.g., `github`, `stripe`) |
| `method` | HTTP method |
| `path` | Request path |
| `status_code` | HTTP response code returned |
| `duration_ms` | Total request handling time in milliseconds |
| `validation_result` | `pass`, `fail`, `missing_header` |
| `enqueue_result` | `success
