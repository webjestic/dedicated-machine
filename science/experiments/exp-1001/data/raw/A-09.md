## Scope and Constraints

This specification covers a single containerized Node.js service that accepts inbound webhook POST requests from a third-party provider, validates each request cryptographically, and enqueues valid events for asynchronous downstream processing. The service does not process events itself. It is a trust boundary and a durable handoff point.

The receiver is a public endpoint. Every design decision must account for that fact.

**Out of scope:** downstream consumer design, event schema validation beyond what is required for safe enqueuing, and provider-specific retry behavior (though provider retry semantics inform several decisions below).

## Implementation Decisions

### 2.1 Signature Validation

**Algorithm:** HMAC-SHA256, as specified by GitHub, Stripe, and most production webhook providers. The signature is computed over the raw request body using a shared secret.

**Critical constraint: raw body preservation.** Express and similar frameworks parse the body before middleware runs. A JSON body parser will re-serialize the payload, altering whitespace and key ordering, which invalidates the HMAC. The raw body — the exact byte sequence received over the wire — must be captured before any transformation. This is done by consuming the request stream directly into a buffer and storing it on the request object. The JSON parser, if needed for downstream use, operates on that buffer, not the wire bytes.

**Comparison method:** The computed HMAC digest and the provider-supplied signature must be compared using a constant-time comparison function. String equality (`===`) short-circuits on the first differing byte, which leaks timing information sufficient for a remote timing attack against the secret. Node.js provides `crypto.timingSafeEqual` for this purpose. Any other comparison method is a defect.

**Signature header normalization:** Provider signature headers vary (`X-Hub-Signature-256`, `Stripe-Signature`, etc.). The header name and expected prefix format (e.g., `sha256=`) must be explicitly configured per provider and treated as part of the validation contract, not inferred at runtime.

**Validation must be synchronous and must complete before the 200 is returned.** There is no valid design in which the signature is validated asynchronously after the response is sent. A 200 response is a commitment that the event was received and accepted. Sending 200 before validation is complete means accepting unauthenticated events.

**Rejection behavior:** Any request that fails signature validation returns `400`. No body, no diagnostic detail. The failure is logged internally with full context. Nothing is returned to the caller that aids in debugging the attack surface.

### 2.2 Request Ingestion Pipeline

**Ordering of middleware:**

1. Raw body capture (stream to buffer, enforce size limit)
2. Signature validation (operates on buffer)
3. Minimal structural check (confirm Content-Type is `application/json` or provider-specified type; confirm body is non-empty)
4. Enqueue
5. Return `200`

This order is non-negotiable. Parsing before validation means operating on untrusted data. Enqueuing before validation means persisting unauthenticated events.

**Body size limit:** Enforced at the raw capture stage. A configurable maximum (default: 1MB, tuned per provider's documented maximum payload size) must be applied. Requests exceeding this limit return `413` immediately, before the body is fully read. This prevents memory exhaustion from a single oversized request.

**HTTP method enforcement:** Only `POST` is accepted at the webhook endpoint. All other methods return `405`. This is enforced before body capture.

**Content-Type enforcement:** Requests without the expected Content-Type are rejected with `415` before body capture. This is a shallow defense but eliminates a class of malformed requests early.

**Endpoint path:** The webhook path should be non-guessable (e.g., `/webhooks/github/events` is acceptable; `/webhook` is not). Path obscurity is not a security control, but it reduces noise in logs and reduces automated scanner surface.

### 2.3 Async Decoupling and Queue Contract

**The receiver's only job after validation is durable enqueue.** It must not call downstream services, apply business logic, or perform I/O beyond writing to the queue. Any of these operations in the request path reintroduce latency and failure modes that the async requirement exists to eliminate.

**Queue selection criteria:** The queue must provide:
- At-least-once delivery guarantee
- Durable persistence (survives process restart and container replacement)
- Acknowledgment-based consumption (the message is not removed until the consumer confirms processing)
- Sufficient throughput headroom above peak expected event rate

Suitable options: Redis Streams, RabbitMQ, AWS SQS, Google Pub/Sub. An in-process queue (e.g., an in-memory array or EventEmitter) is categorically unacceptable — it loses all unprocessed events on container restart, which is not a theoretical failure mode but a guaranteed one during deployments.

**Message envelope:** The enqueued message must include:
- The raw event payload (as received, not re-serialized)
- The event type (from provider header, e.g., `X-GitHub-Event`)
- The delivery ID (from provider header, e.g., `X-GitHub-Delivery`) — this is the idempotency key
- Receipt timestamp (UTC, ISO 8601) assigned by the receiver at enqueue time
- Provider identifier (if the receiver handles multiple providers)

The delivery ID must be stored. It is the only reliable mechanism for deduplication at the consumer and for correlating a received event to a provider's delivery log during incident investigation.

**Enqueue failure:** If the write to the queue fails, the receiver must return `500`, not `200`. A `200` is a commitment. If the event was not durably enqueued, that commitment cannot be made. The provider's retry logic will re-deliver. This is the correct behavior.

**Enqueue timeout:** The queue write must have an explicit timeout. If the queue is degraded and the write hangs, the request must not hang indefinitely. A timeout of 2–3 seconds is appropriate for most queue backends. On timeout, return `500`.

### 2.4 Failure Handling at the Receiver Boundary

**The receiver must not crash on malformed input.** All parsing and validation steps must be wrapped in error handling that produces a structured response and a structured log entry, not an unhandled exception that takes down the process.

**Uncaught exception and unhandled rejection handling:** Node.js will terminate the process on an unhandled rejection by default (and has done so since v15). A top-level handler for `uncaughtException` and `unhandledRejection` must log the error with full context and then allow the process to exit. The container orchestrator (Kubernetes, ECS, etc.) restarts the container. Attempting to recover from an uncaught exception in place is not safe.

**Graceful shutdown:** On `SIGTERM`, the process must:
1. Stop accepting new connections (close the HTTP server)
2. Allow in-flight requests to complete (with a timeout, e.g., 10 seconds)
3. Flush any buffered log output
4. Exit with code 0

Kubernetes sends `SIGTERM` before terminating a pod during rolling deployments. A receiver that does not handle this correctly drops in-flight requests during every deployment.

**Backpressure from the queue:** If the queue is unavailable or returning errors, the receiver will return `500` for every event. The provider will retry. This is acceptable and correct. The receiver must not implement a local buffer to absorb queue unavailability — that buffer is the in-process queue problem restated.

### 2.5 Secret Management

**The HMAC secret must not be in source code, environment variable files committed to version control, or container image layers.** It must be injected at runtime via a secrets management system (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets backed by an external store, etc.).

**The secret must not be logged.** This is obvious and still happens. Any log statement that serializes the full environment, the full configuration object, or the full request headers is a potential secret leak. Structured logging must explicitly allowlist fields, not blocklist them.

**Multiple secrets for rotation:** The validation logic must support holding two active secrets simultaneously — the current secret and the previous secret. During a rotation event, the receiver validates against both. This is covered in detail in section 3.5.

---

### 2.6 Replay Protection

**Provider-supplied timestamps:** Some providers (Stripe) include a timestamp in the signature header. The receiver must enforce a tolerance window (Stripe recommends 300 seconds). Events with a timestamp outside this window are rejected with `400`, regardless of whether the signature is valid. A valid signature on a replayed event is still a replayed event.

**Providers without timestamps:** GitHub does not include a timestamp in the signature. For these providers, replay protection depends on delivery ID deduplication at the consumer layer. The receiver must include the delivery ID in the message envelope (see 2.3) to make this possible.

**The receiver itself should not be the deduplication layer** unless the queue backend makes idempotent writes trivially safe. Deduplication at the receiver requires shared state (a seen-delivery-ID store), which introduces a dependency that can fail and a consistency problem under concurrent instances. Push deduplication to the consumer, which has the full processing context to make the decision correctly.

## Operational Layer

### 3.1 Observability: Logging

**Every request must produce a structured log entry.** Structured means machine-parseable (JSON). Free-text log lines are not acceptable in a production system where logs are the primary debugging tool under pressure.

**Mandatory fields on every log entry:**

| Field | Description |
|---|---|
| `timestamp` | UTC ISO 8601, millisecond precision |
| `request_id` | UUID generated per request by the receiver |
| `delivery_id` | Provider delivery ID from header (if present) |
| `event_type` | Provider event type from header |
| `provider` | Provider identifier |
| `outcome` | `accepted`, `rejected_signature`, `rejected_replay`, `rejected_malformed`, `enqueue_failed` |
| `http_status` | Status code returned |
| `duration_ms` | Total request duration |
| `queue_write_ms` | Time spent on queue write (separately instrumented) |
| `body_bytes` | Raw body size in bytes |
| `source_ip` | Originating IP (from `X-Forwarded-For` if behind a load balancer, with validation) |

**What must not be logged:** The raw body. The HMAC secret. Any field that contains PII or payment data. The signature header value.
