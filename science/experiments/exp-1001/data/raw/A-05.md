# Operational Design Specification: Production Webhook Receiver (Node.js)

---

## Preamble

This specification covers a containerized Node.js service that receives POST requests from a third-party provider, validates HMAC-SHA256 signatures, acknowledges immediately, and queues valid events for downstream processing. The scope is decisions and failure modes — not implementation. Every decision recorded here exists because the alternative has a known failure path.

---

# Part I: Implementation Decisions

---

## 1. Request Ingestion

### 1.1 Raw Body Preservation

The HMAC signature is computed over the raw request body as transmitted. Any middleware that parses the body before signature validation — JSON parsers, body transformers — will mutate the byte sequence and invalidate the comparison. The raw buffer must be captured before any parsing occurs.

**Decision:** Capture the raw body as a `Buffer` at the earliest middleware position. Parse to JSON only after signature validation passes. The raw buffer is the authoritative input for cryptographic verification; the parsed object is a convenience for downstream use only.

**Failure mode avoided:** A body parser that normalizes whitespace, re-serializes, or decodes encoding will produce a different byte sequence than what was signed. The signature check passes in development (where the test payload is already normalized) and fails silently in production when the provider sends compact or differently-encoded JSON.

### 1.2 Body Size Limits

A public endpoint with no size limit is an amplification vector. An attacker can POST arbitrarily large bodies, consuming memory and CPU before the signature check even runs.

**Decision:** Enforce a hard body size limit at the ingestion layer — before reading the body into memory. The limit must be set to a value that accommodates the largest legitimate payload from the provider with margin, not to an arbitrary large number. Reject oversized requests with `413` before buffering completes.

**Failure mode avoided:** Without a size limit, a single large POST can exhaust container memory. With a limit set too generously (e.g., 50MB "to be safe"), the protection is nominal.

## 2. Signature Validation

### 2.1 Algorithm

HMAC-SHA256 using the shared secret provided by the third-party provider. The provider transmits the signature in a request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The format varies by provider — some prefix with an algorithm identifier (`sha256=...`), some include a timestamp component.

**Decision:** Parse the signature header according to the provider's documented format. Strip any prefix before comparison. Do not infer the algorithm from the header value — treat it as fixed.

### 2.2 Timing-Safe Comparison

String equality comparison (`===`) short-circuits on the first differing byte. This leaks information about how many leading bytes of the signature match, enabling a timing oracle attack against the secret.

**Decision:** Use `crypto.timingSafeEqual()` for all signature comparisons. Both buffers must be the same length before comparison — if they are not, the comparison must still take constant time (compare against a dummy buffer of the correct length, then reject). Do not return early on length mismatch without performing a comparison.

**Failure mode avoided:** A timing oracle against a public endpoint, given sufficient requests, can recover the HMAC secret. The attack is not theoretical against a high-throughput receiver.

### 2.3 Timestamp Validation

Stripe-style signatures include a timestamp in the header (`t=<unix_timestamp>`). This enables replay attack prevention: the signature covers both the payload and the timestamp, so a captured valid request cannot be replayed after the timestamp window expires.

**Decision:** If the provider includes a timestamp in the signature scheme, validate that the timestamp falls within an acceptable window (±5 minutes is standard). Reject requests outside the window with `400`. If the provider does not include a timestamp (GitHub's `X-Hub-Signature-256` does not), document this gap explicitly and accept that replay prevention is not available at the signature layer — it must be handled at the idempotency layer.

**Failure mode avoided:** Without timestamp validation, a valid webhook captured in transit can be replayed indefinitely. An attacker who captures a "payment succeeded" event can trigger downstream processing repeatedly.

### 2.4 Secret Rotation

The signing secret will be rotated. The rotation may be uncoordinated — the provider rotates the secret and begins signing with the new key before the service has been updated. If the service only holds one secret, all events during the rotation window fail validation.

**Decision:** Support a list of active secrets, not a single secret. Validate the incoming signature against each secret in the list. Accept if any match. The list is ordered: new secret first, old secret last. Remove the old secret after the rotation window has closed (typically 24–48 hours after the provider confirms the new secret is active). Secrets are injected via environment variables or a secrets manager — never baked into the image.

**Failure mode avoided:** A hard cutover from old secret to new secret produces a validation outage window. During a rotation, every event fails, the queue drains, and downstream processing halts — silently, if alerting is not configured on validation failure rate.

### 2.5 Rejection Response

On validation failure, return `400`. Do not return `401` or `403` — these imply the request was understood and authenticated. Do not return `200` — this would acknowledge an invalid event as accepted. Do not include detail in the response body about why validation failed (which header was missing, what the expected format is). That information aids an attacker.

## 3. Immediate Acknowledgment

### 3.1 Response Timing

The third-party provider has a response timeout. GitHub's is 10 seconds. Stripe's is 30 seconds. Exceeding this timeout causes the provider to mark the delivery as failed and retry. If processing is synchronous and slow, the receiver will time out under load, triggering a retry storm that compounds the backlog.

**Decision:** The request handler does exactly three things: validate the signature, enqueue the event, return `200`. No database writes, no downstream HTTP calls, no business logic. The `200` is returned before the queue write completes only if the queue client is configured for fire-and-forget — which introduces its own failure mode (see Section 3.2).

### 3.2 Enqueue-Before-Acknowledge vs. Acknowledge-Before-Enqueue

Two options:

**Option A — Enqueue then acknowledge:** Write to the queue, await confirmation, then return `200`. If the queue write fails, return `500`. The provider will retry. No event is lost, but queue latency is in the request path.

**Option B — Acknowledge then enqueue:** Return `200` immediately, enqueue asynchronously. If the enqueue fails after acknowledgment, the event is silently dropped. The provider will not retry because it received `200`.

**Decision:** Option A. Enqueue synchronously, await confirmation, then return `200`. The queue write latency (typically <10ms for a local or co-located queue) is acceptable within the provider's timeout window. Silent event loss is not acceptable. If the queue is unavailable, returning `500` is correct — the provider retries, and the event is not lost.

**Failure mode avoided:** Option B produces silent data loss under queue pressure or transient queue failures. This failure is invisible: the provider reports successful delivery, the queue has no record, downstream processing never runs.

## 4. Queue Design

### 4.1 Queue Selection

**Decision:** Use a durable, persistent queue with at-least-once delivery guarantees. Redis with persistence enabled (AOF or RDB) is acceptable for moderate throughput. A dedicated message broker (RabbitMQ, SQS, Kafka) is preferred for production. An in-process queue (EventEmitter, in-memory array) is not acceptable — it does not survive container restarts.

**Failure mode avoided:** An in-memory queue loses all pending events on container restart, OOM kill, or deployment. Container restarts are not exceptional events — they are routine.

### 4.2 Queue Message Schema

The message written to the queue must contain everything needed for downstream processing without requiring the consumer to call back to the receiver or the provider.

**Decision:** Queue messages include: raw event payload (as received), parsed event type, provider-assigned event ID (if present), timestamp of receipt, and a receiver-assigned UUID. Do not queue only an event ID and expect the consumer to fetch the payload — this creates a dependency on provider API availability and introduces rate limit exposure.

### 4.3 Idempotency

Providers deliver at-least-once. The same event will arrive more than once — on retry after a `5xx`, on retry after a timeout, on provider-side redelivery for reliability. Downstream consumers must be idempotent. The receiver's responsibility is to make idempotency tractable.

**Decision:** Include the provider's event ID in the queue message. If the provider does not supply an event ID, generate a deterministic ID from the HMAC of the raw body (the same payload will always produce the same ID). Downstream consumers use this ID as an idempotency key. The receiver does not deduplicate at the ingestion layer — deduplication at the queue layer introduces state and latency in the request path.

**Failure mode avoided:** Without an idempotency key, a consumer that processes the same event twice has no mechanism to detect the duplicate. "Process once" logic without an idempotency key is not idempotency — it is optimism.

## 5. Container Configuration

### 5.1 Secret Injection

**Decision:** Signing secrets are injected via environment variables at runtime, sourced from a secrets manager (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets). They are not present in the container image, the Dockerfile, or any version-controlled file. The service reads secrets at startup and holds them in memory. Secret rotation requires a restart or a live reload mechanism (see Section 7.3 of the operational layer).

### 5.2 Process Model

**Decision:** Single Node.js process per container. Do not use the cluster module or worker threads for request handling — horizontal scaling is handled at the container orchestration layer. This simplifies the operational model and avoids shared-memory complexity. The process must handle `SIGTERM` gracefully (see Operational Section 2).

### 5.3 Port and Network Exposure

**Decision:** The container exposes a single HTTP port. TLS termination is handled by the load balancer or ingress layer — not by the Node.js process. The container does not bind to `0.0.0.0` in production without a network policy restricting inbound traffic to the load balancer only.

# Part II: Operational Layer

---

## 1. Observability

### 1.1 Structured Logging

Every request must produce a structured log entry regardless of outcome. Log fields must include: timestamp (ISO 8601, UTC), request ID (generated at ingestion), provider event ID (if extractable before validation), HTTP method, path, response status code, validation outcome (`valid` / `invalid_signature` / `missing_header` / `replay_rejected` / `size_exceeded`), queue write outcome (`enqueued` / `queue_error`), and total handler duration in milliseconds.

Do not log the raw signature header value. Do not log the signing secret. Do not log the full request body at INFO level — this produces secrets-in-logs exposure if the payload contains sensitive fields.

Log at WARN for validation failures. Log at ERROR for queue write failures. Log at INFO for successful processing. Log at DEBUG for full request detail — DEBUG must be disabled in production by default and enableable without restart.

### 1.2 Metrics

Expose the following metrics, labeled by outcome:

- `webhook_requests_total` — counter, labeled by `status` (valid, invalid, error)
- `webhook_validation_failures_total` — counter, labeled by `reason`
- `webhook_queue_write_duration_seconds` — histogram
- `webhook_handler_duration_seconds` — histogram
- `webhook_body_size_bytes` — histogram
- `webhook_queue_depth` — gauge (if the queue client exposes this)

These metrics are the primary signal for detecting degradation. A spike in `webhook_validation_failures_total` is either an attack or a secret rotation that was not coordinated. A spike in `webhook_queue_write_duration_seconds` predicts the queue becoming the bottleneck in the request path.

### 1.3 Distributed Tracing

Assign a unique request ID at the earliest middleware position. Propagate this ID in the queue message. Downstream consumers must log this ID when processing the event. Without this, correlating a failed downstream processing job to the original inbound request requires log archaeology.
