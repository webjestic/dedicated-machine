## 1. Scope and Boundaries

**In scope:**
- Receiving inbound POST requests from a third-party webhook provider
- Validating HMAC-SHA256 signatures on every request before any further processing
- Enqueuing valid events for downstream consumption
- Returning HTTP 200 to the provider immediately upon successful enqueue
- Operating as a single containerized service

**Out of scope:**
- Downstream event processing logic (consumers are a separate system)
- Provider-side webhook configuration
- Consumer scaling or consumer failure handling

**Explicit boundary:** This service is a receiver and a queue producer. It owns the event from the moment it arrives at the HTTP layer until the moment it is durably enqueued. Downstream failures are not this service's responsibility. Enqueue failures are.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework:** Use a minimal HTTP framework (Fastify preferred over Express) for its lower overhead and built-in schema validation primitives. The receiver is on the hot path for every inbound event; framework overhead compounds at volume.

**Port binding:** Bind to a single configurable port (default: `3000`). Do not bind to privileged ports inside the container. TLS termination is handled by the load balancer or ingress layer — not by this service.

**Accepted methods:** Accept `POST` only on the webhook endpoint. All other methods return `405 Method Not Allowed`. This is not a default behavior to rely on — it must be explicitly configured.

**Endpoint path:** The endpoint path must be configurable via environment variable, not hardcoded. A hardcoded path is a path that cannot be rotated without a redeploy.

**Request size limit:** Enforce a maximum request body size. Default ceiling: `1MB`. Requests exceeding this limit are rejected with `413 Payload Too Large` before signature validation runs. This is a denial-of-service control, not a convenience setting.

**Timeouts:** Configure a request timeout of no more than `10 seconds`. A provider that does not complete transmission within this window receives a `408` or the connection is dropped. Do not allow slow-loris style connections to hold workers.

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. This is non-negotiable and provider-specified. Do not generalize to support multiple algorithms — generalization here is an attack surface.

**Signature source:** Read the signature from the provider-specified header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name is configuration, not code.

**Comparison method:** Use a constant-time comparison function for the HMAC digest comparison. Node.js `crypto.timingSafeEqual` is the correct primitive. A naive string equality check (`===`) is a timing oracle and must not be used under any circumstances.

**Validation is synchronous and blocking within the request path.** Signature validation must complete before the 200 is returned and before anything is enqueued. An event that has not been validated is an event that must not be enqueued. There is no exception to this ordering.

**Validation failure response:** Return `401 Unauthorized` on signature mismatch. Do not return `400`. The distinction matters for provider retry logic and for distinguishing misconfiguration from malformed requests in logs.

**Validation failure behavior:** Log the failure with enough context to diagnose (timestamp, source IP, header presence/absence, payload size) but never log the raw signature value or the secret. Increment a validation failure counter metric.

**Missing signature header:** If the signature header is absent entirely, return `400 Bad Request`. This is a distinct failure mode from a present-but-invalid signature and must be tracked separately.

### 2.3 Request Body Handling

**Raw body requirement:** HMAC-SHA256 validation is performed against the raw, unmodified request body bytes. Any middleware that parses the body before validation runs — JSON parsing, URL decoding, compression decompression that alters byte representation — will silently corrupt the signature check. The raw body buffer must be captured before any transformation.

**Implementation constraint:** Buffer the raw body first. Parse it second. Validation runs against the buffer. Parsed representation is used only after validation succeeds.

**Content-Type:** Accept `application/json` and `application/x-www-form-urlencoded` as dictated by the provider. Reject other content types with `415 Unsupported Media Type`. Do not silently accept unknown content types and attempt to parse them.

**Encoding:** Treat the body as UTF-8. Reject requests with non-UTF-8 encoding declarations.

### 2.4 Async Processing and Queue Architecture

**Core contract:** The HTTP handler's only job after successful validation is to enqueue the event and return `200`. No business logic. No downstream HTTP calls. No database writes. No synchronous processing of any kind.

**Queue selection:** Use a durable, external queue. Redis with Streams or BullMQ, or a managed queue service (SQS, Cloud Tasks). An in-process queue (array, EventEmitter, in-memory queue) is not acceptable — it does not survive process restart, and this service will restart.

**Durability requirement:** The queue must persist events across process crashes and container restarts. If the queue is Redis-backed, Redis must be configured with persistence (AOF or RDB). A queue that loses events on restart is not a queue — it is a buffer with false guarantees.

**Enqueue failure handling:** If the enqueue operation fails, do not return `200`. Return `500 Internal Server Error`. The provider will retry. A `200` on a failed enqueue is a silent data loss event. This is the worst failure mode in the system.

**Enqueue timeout:** The enqueue operation must have an explicit timeout (recommended: `2 seconds`). A queue that is slow to accept writes must not hold the HTTP response open indefinitely. Timeout triggers a `503 Service Unavailable` response.

**Payload stored on queue:** Store the full raw event payload, the event type header, the timestamp of receipt, a generated trace/correlation ID, and the source IP. Do not store the signature or the secret. The correlation ID must be included in the `200` response and in all downstream log entries.

**Idempotency key:** If the provider supplies a delivery ID header (e.g., `X-GitHub-Delivery`, `Stripe-Signature` contains an ID component), extract it and include it in the queued message. Downstream consumers are responsible for idempotent processing, but this service must preserve the key that makes idempotency possible.

### 2.5 Error Handling Contract

| Condition | HTTP Response | Retry Expected | Notes |
|---|---|---|---|
| Missing signature header | 400 | No | Provider misconfiguration |
| Invalid signature | 401 | No | Possible attack or secret mismatch |
| Body too large | 413 | No | Provider sending oversized payload |
| Unsupported content type | 415 | No | Provider misconfiguration |
| Enqueue failure | 500 | Yes | Provider should retry |
| Enqueue timeout | 503 | Yes | Queue degraded |
| Request timeout | 408 / drop | Yes | Provider should retry |
| Valid event, enqueued | 200 | No | Normal path |

**No 2xx response is returned unless the event is durably enqueued.** This is the single most important invariant in the system.

### 2.6 Secret Management

**Storage:** The HMAC secret must never be stored in source code, committed to version control, or written to a log. It is injected at runtime via environment variable or a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager).

**Access pattern:** Read the secret once at startup into a module-scoped variable. Do not re-read from the environment on every request — environment variable access is not free and the value will not change at runtime without a restart (see Secret Rotation, Section 3.1).

**Multiple secrets:** The implementation must support holding two active secrets simultaneously. This is not an edge case — it is the operational requirement for zero-downtime secret rotation. If the system cannot hold two secrets, secret rotation requires downtime or a gap in validation.

**Secret length:** Enforce a minimum secret length at startup (recommended: 32 bytes / 256 bits). Reject startup with a fatal error if the secret does not meet the minimum. A short secret is a weak HMAC.

### 2.7 Dependency Decisions

**Node.js version:** Pin to a specific LTS version in the Dockerfile (`FROM node:20-alpine` not `FROM node:latest`). `latest` is a moving target that will silently change behavior on rebuild.

**Dependency pinning:** All `package.json` dependencies pinned to exact versions. `package-lock.json` committed and used in the Docker build (`npm ci`, not `npm install`). A build that pulls different dependency versions than were tested is not a reproducible build.

**Minimal dependency surface:** This service has a narrow function. Its dependency list should reflect that. Every dependency is an attack surface and a maintenance obligation. Prefer Node.js built-in `crypto` for HMAC over third-party cryptography libraries unless the provider's SDK is the specified integration path.

## 3. Operational Layer

### 3.1 Secret Rotation

**Requirement:** Secret rotation must be possible without dropping valid events and without service downtime.

**Mechanism:** The service must support two simultaneously active secrets — the current secret and the incoming secret. On each request, validate against both. If either matches, the event is valid. This window allows the provider to be reconfigured to the new secret while the old secret remains valid for in-flight deliveries.

**Rotation procedure:**
1. Generate new secret.
2. Load new secret into secrets manager.
3. Redeploy or signal service to reload secrets (see below). Service now holds both old and new secrets.
4. Update provider webhook configuration to use new secret.
5. Wait for provider to confirm all deliveries are using the new secret. This window is provider-dependent — for GitHub and Stripe, allow a minimum of 15 minutes.
6. Remove old secret from secrets manager.
7. Redeploy or signal service. Service now holds only the new secret.

**Secret reload without restart:** If the runtime environment supports it (Kubernetes secrets mounted as files, Vault agent sidecar), implement a SIGHUP handler or a polling interval to reload secrets without a full process restart. If this is not implemented, document explicitly that secret rotation requires a rolling redeploy and that the deployment pipeline must support this without dropping requests.

**Rotation failure mode:** If the service is redeployed with the new secret before the provider is updated, all incoming events will fail validation. This is a production outage. The rotation procedure above is ordered specifically to prevent this. Deviating from the order is not a judgment call.
