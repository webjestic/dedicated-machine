## 1. Scope and Boundaries

This specification covers a single containerized Node.js service that:

- Accepts inbound POST requests from one or more third-party webhook sources
- Validates request authenticity via HMAC-SHA256 before any other processing
- Enqueues validated events for downstream consumption
- Returns a response to the caller within the HTTP request lifecycle, before any downstream processing occurs

This service is a **receiver and forwarder only**. It does not process events. It does not apply business logic. It does not call external APIs. The boundary is: signature validated, event enqueued, 200 returned. Everything downstream is out of scope.

**What this service owns:**
- The public HTTP endpoint
- Signature validation correctness
- The enqueue operation and its durability guarantee
- The response contract with the upstream provider

**What this service does not own:**
- Event processing logic
- Consumer failures
- Queue infrastructure provisioning (it depends on it; it does not own it)

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Decision:** Use a minimal HTTP framework. Express or Fastify are both acceptable. The framework choice is less important than the constraints placed on it.

**Constraints:**

- The receiver endpoint must be a single route: `POST /webhook` (or provider-namespaced equivalent, e.g., `POST /webhooks/github`). No other routes are exposed on this port except health endpoints (see §3.8).
- If multiple providers are supported, each gets a distinct path. Signature validation logic is provider-specific and must not be shared across paths without explicit, tested isolation.
- HTTP/1.1 is sufficient. HTTP/2 is not required and introduces unnecessary complexity for this workload.
- TLS termination is handled by the infrastructure layer (load balancer, ingress). The container itself listens on plain HTTP internally. This is not a security gap if the network path from terminator to container is private. If it is not private, this decision must be revisited before deployment.
- Bind to `0.0.0.0` on a configurable port (default: 3000). Do not hardcode.
- Set explicit, conservative timeouts on the HTTP server: request timeout 10s, keep-alive timeout 65s (must exceed load balancer idle timeout to prevent race-condition connection resets).

**Rationale:** The attack surface of this service is a single public POST endpoint. Every additional route, middleware, or framework feature is additional attack surface and additional failure surface. Minimize both.

---

### 2.2 Signature Validation

**Decision:** HMAC-SHA256 validation using Node.js's built-in `crypto` module. No third-party library for the cryptographic primitive itself.

**Requirements, without exception:**

1. **Read the raw request body before any parsing.** The HMAC is computed over the raw bytes as received. Any transformation — JSON parsing, body normalization, charset conversion — invalidates the signature. The raw body must be captured as a `Buffer` before the signature check runs.

2. **Use `crypto.timingSafeEqual` for the comparison.** String equality (`===`) is not acceptable. Timing attacks against a public endpoint are a realistic threat, not a theoretical one. Both the received signature and the computed signature must be converted to `Buffer` before comparison. If the lengths differ, return 401 without calling `timingSafeEqual` — but do not short-circuit in a way that reveals which byte failed.

3. **Validate the signature before doing anything else with the request.** No logging of request body content, no parsing, no queue interaction, no business logic — until the signature passes. A failed signature means the request is unauthenticated. Treat it accordingly.

4. **The signature header name is provider-specific and must be configured, not hardcoded.** GitHub uses `X-Hub-Signature-256`. Stripe uses `Stripe-Signature` with a different format. The validation logic must be written against the actual provider's scheme, not a generic approximation of it.

5. **For providers that embed a timestamp in the signature (e.g., Stripe's `t=` prefix):** the timestamp must be extracted and validated as part of the signature check. This is not optional — see §3.2.

6. **A missing signature header is treated identically to an invalid signature.** Return 401. Do not return 400. The distinction leaks information about what the validator expects.

7. **Signature validation failures are never silent.** Every failure is logged and metered. See §3.3.

**What a valid signature proves:** The request body was signed by a party in possession of the shared secret at the time of signing. It does not prove the event is not a replay. It does not prove the event has not been processed before. Those are separate concerns addressed in §3.2 and §2.7.

### 2.3 Request Body Handling

**Decision:** Buffer the entire raw request body in memory before processing.

**Constraints:**

- Set a hard maximum body size. 5MB is a reasonable upper bound for webhook payloads from providers like GitHub or Stripe. Requests exceeding this limit are rejected with 413 before signature validation runs. This is not a signature failure — it is a size policy rejection.
- The body size limit must be enforced at the byte level on the incoming stream, not after buffering. Buffering an unbounded body before rejecting it defeats the purpose of the limit.
- The raw body `Buffer` is passed to the signature validator. A parsed representation (JSON object) may be derived from it after validation passes, but the raw buffer is the authoritative input to the HMAC.
- Do not log the raw body in production. Log the event type and delivery ID (from headers) only. Raw bodies may contain PII or sensitive payload data.

### 2.4 Async Decoupling and Queue Contract

**Decision:** The queue is the decoupling boundary. The HTTP request lifecycle ends when the enqueue operation completes (or fails). Processing happens downstream, out of scope for this service.

**Queue selection criteria:**

The queue must provide:
- **At-least-once delivery.** Exactly-once is preferred but not required from the queue itself — idempotency is the consumer's responsibility (§2.7).
- **Durable persistence.** Messages must survive a queue service restart. In-memory queues (e.g., a plain Node.js `EventEmitter`, an in-process array) are not acceptable. The receiver process will restart. Events must not be lost when it does.
- **Acknowledgement-based consumption.** Consumers must explicitly acknowledge successful processing. Unacknowledged messages must be redelivered.

Acceptable options: Redis Streams, RabbitMQ, AWS SQS, Google Cloud Pub/Sub, or equivalent. The specific choice is an infrastructure decision outside this document's scope, but the above properties are non-negotiable.

**Enqueue operation contract:**

- The enqueue call is in the request path. It must complete before the 200 is returned.
- If the enqueue call fails (queue unavailable, timeout, network error), the service must return 5xx to the caller. Do not return 200 for an event that was not durably enqueued. Returning 200 for a dropped event is a silent data loss event.
- The enqueue call must have an explicit timeout. Default client timeouts are not acceptable. Set a timeout that is shorter than the provider's own request timeout (GitHub: 10s, Stripe: 30s). A reasonable default is 3–5 seconds.
- The message written to the queue must include: raw event payload, event type (from headers), delivery ID (from headers), timestamp of receipt, source IP, and the provider identifier. Do not write only the parsed payload — include the metadata the consumer will need for debugging and idempotency.

**What "async" means here:** Processing is async relative to the HTTP response. The enqueue itself is synchronous within the request lifecycle. This is the correct design. "Return 200 and then enqueue" is not acceptable — it creates a window where the caller believes the event was received but it has not been durably stored.

### 2.5 Response Contract

**Decision:** Fixed, minimal response bodies. The caller is a machine.

| Condition | Status Code | Body |
|---|---|---|
| Signature valid, enqueue succeeded | 200 | `{"status":"ok"}` |
| Enqueue failed (queue unavailable) | 503 | `{"status":"unavailable"}` |
| Signature invalid or missing | 401 | `{"status":"unauthorized"}` |
| Body too large | 413 | `{"status":"payload_too_large"}` |
| Wrong method or path | 404 | `{"status":"not_found"}` |
| Unhandled exception | 500 | `{"status":"error"}` |

**Constraints:**

- Never include internal error messages, stack traces, or queue error details in response bodies. These are logged internally. The caller receives a status code and a stable status string.
- The `Content-Type` response header is always `application/json`.
- 5xx responses cause most webhook providers to retry. This is the intended behavior when the queue is unavailable. Ensure retry behavior is understood and accounted for in §3.2 and §2.7.

### 2.6 Secret Management

**Decision:** Secrets are injected as environment variables. They are never in source code, configuration files committed to version control, or container image layers.

**Requirements:**

- The HMAC secret is read from the environment at process startup. If the secret is absent, the process must fail to start. Starting without a secret and silently accepting all requests (or silently rejecting all requests) are both unacceptable failure modes.
- Secrets are not logged. Not at startup, not in debug output, not in error messages. The secret value must never appear in any log line, metric label, or trace attribute.
- The secret is stored in a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, or equivalent). The environment variable is populated at container startup from the secrets manager, not hardcoded in the deployment
