# Operational Design Specification: Production Webhook Receiver

**Service:** Inbound Webhook Receiver
**Runtime:** Node.js, single containerized deployment
**Classification:** Public-facing ingress, security boundary

---

## Part I — Implementation Decisions

### 1.1 Signature Validation

**Algorithm:** HMAC-SHA256. The shared secret is compared against the provider-supplied signature header using a timing-safe comparison function. Node's `crypto.timingSafeEqual` is the only acceptable comparator. String equality (`===`) is prohibited — it short-circuits on the first differing byte and is trivially exploitable via timing oracle.

**Raw body requirement:** Signature validation must operate on the raw, unparsed request body as received over the wire. Any middleware that parses JSON before the signature check is applied will mutate the buffer and invalidate the comparison. The raw body must be captured first, stored on the request context, and only parsed after the signature is confirmed valid. This ordering is a hard constraint, not a preference.

**Secret storage:** The HMAC secret is injected as an environment variable at runtime. It is never committed to source, never logged, never included in error responses. The secret must be loaded once at startup and held in memory. It is not re-read per request.

**Secret rotation handling:** Providers rotate secrets. The receiver must support a transition window where both the current and the incoming secret are valid simultaneously. This means the validator accepts a list of active secrets and attempts each in sequence, accepting the request if any match. The old secret is removed from the list after the rotation window closes. Failing to design for this means a rotation event causes a complete ingestion outage.

**Failure behavior:** A request that fails signature validation receives a `401` response with no body detail. The failure is logged with the request ID, source IP, and the header value that was presented — but never the expected value. The event is dropped. It is not queued, not partially processed, not retried.

### 1.2 Request Handling and the 200 Contract

The receiver's only job in the request path is: validate the signature, enqueue the raw payload, return `200`. Nothing else belongs in the request path.

**Response timing target:** The `200` must be returned within a bounded, short window — target under 100ms for the p99 case. Any downstream system that is slow or unavailable must not be visible to the sender. If the queue is unavailable, see section 1.4.

**Body size limit:** A hard cap on the inbound body size is enforced before any processing. The limit should be set to the maximum documented payload size for the provider plus a reasonable margin. Requests exceeding this limit receive a `413` and are dropped. This prevents memory exhaustion from oversized or malformed payloads.

**Idempotency key capture:** The provider's event ID (e.g., `X-GitHub-Delivery`, Stripe's `id` field in the payload) is extracted and stored alongside the queued event. This is the deduplication key. It is not used for deduplication at ingestion time — that is the downstream consumer's responsibility — but it must be present on every queued message.

### 1.3 Queue Architecture

**Queue placement:** The queue sits between the receiver and all downstream processing. The receiver writes to the queue and returns. Downstream processors read from the queue independently. The receiver has no knowledge of what consumers exist or whether they are running.

**Durability requirement:** The queue must provide durable, acknowledged delivery. An in-process queue (e.g., an in-memory array, an EventEmitter) is not acceptable. A process restart must not lose events. Acceptable options include a managed queue service (SQS, Cloud Tasks, Pub/Sub) or a self-hosted broker (RabbitMQ, Redis Streams with persistence enabled). The choice depends on operational constraints, but the durability guarantee is non-negotiable.

**Write failure handling:** If the queue write fails, the receiver must not return `200`. It must return `500`. This is the one case where the `200` contract is intentionally broken. Returning `200` when the event has not been durably enqueued is a silent data loss event. The provider will retry on `5xx`. That retry mechanism is the recovery path and must be allowed to function.

**Message envelope:** Every queued message includes: the raw payload bytes, the event type header, the provider's event ID, the receiver's internal request ID, the ingestion timestamp (UTC), and the source IP. The raw payload is preserved — downstream consumers deserialize it themselves. The receiver does not transform the payload.

### 1.4 Backpressure and Queue Unavailability

When the queue is unavailable or saturated, the receiver must shed load rather than absorb it. The receiver does not buffer events internally. It does not retry the queue write in the request path. It fails fast, returns `503`, and allows the provider's retry logic to hold the event until the queue recovers.

An in-process buffer as a fallback is explicitly rejected. It creates the illusion of availability while guaranteeing data loss on restart and masking the real problem.

### 1.5 Replay and Timestamp Validation

Providers include a timestamp in the signature payload or as a separate header (e.g., Stripe's `t=` field in `Stripe-Signature`). The receiver enforces a replay window: events with a timestamp older than a configurable threshold (default: 5 minutes) are rejected with `400`. This prevents replay attacks where a valid captured request is resubmitted.

The threshold must be configurable without a deployment. Clock skew between the provider and the receiver must be accounted for — the window should be wide enough to tolerate reasonable skew but narrow enough to be meaningful. Five minutes is the industry-standard starting point.

If the provider does not include a timestamp in the signed payload, this protection cannot be applied at the signature layer. In that case, deduplication by event ID at the consumer layer is the primary replay defense.

### 1.6 HTTP Framework and Middleware Ordering

Middleware ordering is a correctness constraint, not a style choice. The required order is:

1. Body size enforcement
2. Raw body capture (store bytes on request context, do not parse)
3. Signature validation (operate on raw bytes)
4. Timestamp/replay validation
5. Payload parsing (JSON deserialization of the now-validated body)
6. Queue write
7. Response

Any deviation from this order — particularly parsing before validation — is a defect.

## Part II — Operational Layer

### 2.1 Structured Logging

Every request produces a structured log entry. Fields are fixed and machine-parseable. The entry includes: request ID (generated at ingestion), source IP, event type, provider event ID, signature validation result, queue write result, response status code, and end-to-end latency in milliseconds.

Validation failures are logged at WARN. Queue write failures are logged at ERROR. Successful ingestion is logged at INFO.

What is never logged: the raw HMAC secret, the full signature header value on success, any PII present in the payload. Payload contents are not logged. If payload inspection is needed for debugging, it is done by reading from the queue, not from receiver logs.

Log volume at scale is a cost and a noise problem. Successful `200` responses may be sampled at a configurable rate. Failures are never sampled — every failure is logged.

### 2.2 Metrics and Instrumentation

The receiver exposes the following metrics, labeled by event type and outcome:

- **`webhook_requests_total`** — counter, labeled by status code bucket (2xx, 4xx, 5xx)
- **`webhook_signature_failures_total`** — counter; a sustained non-zero rate here is an active incident
- **`webhook_queue_write_failures_total`** — counter; any non-zero value warrants immediate investigation
- **`webhook_request_duration_ms`** — histogram, p50/p95/p99; degradation here indicates queue latency bleeding into the request path
- **`webhook_replay_rejections_total`** — counter; unexpected spikes indicate a provider misconfiguration or an active replay attempt
- **`webhook_body_size_bytes`** — histogram; useful for detecting payload size anomalies before they become a problem

These metrics are scraped by the monitoring system, not pushed. The `/metrics` endpoint is not exposed on the public interface — it is available only on an internal port or a separate listener.

### 2.3 Health and Readiness Endpoints

**`/healthz` (liveness):** Returns `200` if the process is alive. No dependency checks. Used by the container orchestrator to determine whether to restart the container.

**`/readyz` (readiness):** Returns `200` only if the queue connection is established and the receiver is capable of processing requests. Returns `503` if the queue is unreachable. Used by the load balancer to determine whether to route traffic. A receiver that cannot reach the queue must not receive traffic — it will only return `503` to the provider and trigger unnecessary retries.

These two endpoints are distinct for a reason. A process that is alive but cannot write to the queue should be removed from rotation without being restarted.

### 2.4 Alerting Thresholds

Alerts are defined on metrics, not on logs. The following conditions page immediately:

| Condition | Threshold | Rationale |
|---|---|---|
| Signature failure rate | > 1% of requests over 5 minutes | Either a rotation event or an active probe |
| Queue write failure rate | Any non-zero sustained rate | Silent data loss is occurring |
| p99 request latency | > 500ms | Queue latency is bleeding into the request path |
| `5xx` response rate | > 0.5% over 5 minutes | Provider retries are accumulating |
| Process restart | Any | Container instability |

Signature failure spikes that coincide with a known secret rotation event are expected and should be suppressible via a maintenance annotation, not by raising the threshold permanently.

### 2.5 Secret Rotation Procedure

Secret rotation is a planned operational event, not an emergency. The procedure is:

1. The new secret is provisioned in the secret manager and added to the receiver's active secret list via configuration update. The receiver is reloaded without a full restart (or restarted with zero downtime via rolling deployment). Both secrets are now active.
2. The provider is updated to use the new secret.
3. A validation window of sufficient duration passes — long enough to confirm no requests are arriving signed with the old secret. Monitor `webhook_signature_failures_total` during this window.
4. The old secret is removed from the active list. The receiver is reloaded.

Skipping step 1 — updating the provider before the receiver accepts the new secret — causes an ingestion outage for the duration of the propagation lag. This is the most common rotation mistake.

### 2.6 Replay and Duplicate Event Handling

The receiver enforces a replay window at ingestion (section 1.5). This is a first-line defense only. It does not guarantee exactly-once delivery. The queue itself may redeliver messages. The provider may retry a request that received a `200` due to a network timeout on their side.

Downstream consumers must treat every event as potentially duplicate. The idempotency key (provider event ID) captured at ingestion is the deduplication primitive. The receiver's responsibility ends at enqueuing the event with that key present. Deduplication logic does not belong in the receiver.

### 2.7 Rate Limiting and Abuse Mitigation

The receiver is a public endpoint. It will be probed. Rate limiting is applied at the infrastructure layer (load balancer or API gateway), not in the Node.js process. The Node.js process is not the right place to enforce per-IP rate limits at scale — it lacks the shared state required to be effective across multiple instances and adds latency to the hot path.

The infrastructure layer enforces:
- Per-IP request rate limits
- Global request rate cap sized to the maximum expected provider throughput plus a safety margin
- Automatic blocking of IPs that sustain high signature failure rates

Requests that exceed rate limits receive `429`. The receiver process never sees them.

Body size limits (section 1.2) are the one abuse mitigation that lives in the process, because they protect against memory exhaustion before the infrastructure layer can act.

### 2.8 Deployment and Rollout

The receiver is stateless. It holds no in-process state beyond the active secret list and the queue connection. Rolling deployments are safe. Blue-green deployments are safe.

**Graceful shutdown:** On `SIGTERM`, the receiver stops accepting new connections, allows in-flight requests to complete up to a configurable drain timeout (default: 10 seconds), then exits. Requests that cannot complete within the drain window receive a `503`. The orchestrator must be configured to send `SIGTERM` and wait for the drain window before forcibly terminating the container. A hard kill without a drain window will drop in-flight requests silently.
