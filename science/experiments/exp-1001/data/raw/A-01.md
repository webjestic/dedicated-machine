# Operational Design Specification: Production Webhook Receiver

**Service:** Webhook Receiver
**Runtime:** Node.js (containerized)
**Classification:** Public-facing ingress, security boundary

---

## Part I — Implementation Decisions

### 1. Signature Validation

**Decision:** HMAC-SHA256 validation is the first operation executed on every inbound request, before any parsing, routing, or business logic. No exception path bypasses it.

**Raw body requirement:** The HMAC is computed against the raw request bytes as received, not against a parsed representation. Express (or equivalent) must be configured to capture the raw body buffer before any JSON parsing occurs. Body parsing and signature validation are decoupled pipeline stages, with raw capture happening first. If the framework mutates the body before validation is possible, the framework configuration is wrong.

**Comparison:** The computed HMAC is compared to the provider-supplied signature using a constant-time comparison function (`crypto.timingSafeEqual`). String equality operators are prohibited. Timing attacks against HMAC comparison are a real class of vulnerability; this is not theoretical hardening.

**Signature header normalization:** The provider signature header is extracted and normalized before comparison. For providers that prefix the signature (e.g., `sha256=`), the prefix is stripped in a controlled, explicit step — not via a regex that could be manipulated. If the header is absent, malformed, or does not match the expected format, the request is rejected with `400` before any further processing.

**Failure behavior:** Any signature validation failure returns `400 Bad Request` with a static, non-descriptive body. The response body does not indicate whether the failure was a missing header, a malformed header, or a computed mismatch. Diagnostic detail belongs in internal logs, not in the response.

**Secret handling:** The HMAC secret is never embedded in source code or container images. It is injected at runtime via environment variable, sourced from a secrets manager (Vault, AWS Secrets Manager, or equivalent). The secret is read once at process startup and held in memory. It is never logged.

---

### 2. Secret Rotation

**Decision:** The receiver must support dual-secret validation during rotation windows without dropping valid events or requiring a deployment.

**Mechanism:** At startup, the receiver reads both a primary secret and an optional secondary secret from the environment. On each request, validation is attempted against the primary secret first. If that fails and a secondary secret is present, validation is attempted against the secondary. If both fail, the request is rejected. This window is explicitly time-bounded: the secondary secret is removed from configuration once rotation is confirmed complete.

**Rationale:** Providers rotate secrets with finite propagation delays. A receiver that accepts only one secret at a time will reject legitimate events during the rotation window. Silent rejection during rotation is operationally indistinguishable from an attack — both produce a spike in `400` responses. The dual-secret window makes the transition observable and controlled.

**Audit requirement:** Every request that validates against the secondary (old) secret is logged with a distinct marker. This creates an observable signal that the rotation window is still active and that the old secret is still in use.

---

### 3. Request Ingestion and Immediate Response

**Decision:** The receiver returns `200 OK` before any downstream processing occurs. The `200` response indicates only that the event was received and accepted for processing — not that it was processed.

**Constraint:** No database call, no downstream HTTP call, no file I/O, and no blocking computation occurs in the request path. The request path does exactly three things: validate the signature, enqueue the raw event, and return `200`.

**Timeout risk:** Third-party providers enforce delivery timeouts (typically 5–30 seconds). Any processing in the request path that can block — even briefly — risks a timeout, which the provider interprets as failure and retries. Retries compound backlog. The only safe design is an unconditionally fast response path.

**Payload size limit:** The raw body is capped at a configured maximum size (default: 1MB, tunable). Requests exceeding this limit are rejected with `413` before signature validation. This prevents memory exhaustion from oversized payloads and removes a denial-of-service vector.

---

### 4. Queue Design

**Decision:** Valid events are written to a durable, persistent queue before the `200` response is sent. In-memory queues (arrays, EventEmitter, etc.) are prohibited for this role.

**Durability requirement:** The queue must survive process crashes and container restarts without losing events. This means the write to the queue is the commit point. If the write fails, the receiver returns `500` — not `200`. Returning `200` for an event that was not durably enqueued is a silent data loss event.

**Queue technology:** Redis Streams, RabbitMQ, SQS, or equivalent persistent queue. The choice is driven by the operational environment, but the durability guarantee is non-negotiable. Redis without persistence (`appendonly no`) does not qualify.

**Enqueue failure handling:** If the queue write fails (connection error, timeout, capacity exceeded), the receiver returns `500`. The provider will retry. This is the correct behavior. A `200` on enqueue failure would permanently lose the event.

**Event envelope:** The message written to the queue is not the raw payload alone. It is an envelope containing: the raw payload bytes, the event type header (e.g., `X-GitHub-Event`), the delivery ID header (e.g., `X-GitHub-Delivery`), the provider-supplied signature header (for downstream audit), the receipt timestamp (UTC, milliseconds), and the source IP. The delivery ID is the idempotency key.

---

### 5. Idempotency

**Decision:** The receiver is not responsible for idempotent processing — that is the responsibility of downstream consumers. The receiver is responsible for providing the data necessary to implement idempotency downstream.

**Delivery ID:** The provider-supplied delivery ID is extracted, validated as present, and included in the queue envelope. Downstream consumers use this as their deduplication key. If the provider does not supply a delivery ID, the receiver generates a deterministic ID from the HMAC of the payload and the receipt timestamp.

**Rationale for boundary placement:** Implementing deduplication at the receiver requires a shared state store with the associated latency, failure modes, and operational complexity — all in the request path. This violates the fast-response constraint. The receiver's job is to accept and enqueue. Deduplication belongs where processing occurs.

---

### 6. Transport Security

**Decision:** The receiver terminates TLS at the load balancer or ingress layer. The Node.js process itself communicates over plain HTTP within the private network. TLS termination in Node.js application code is not used.

**TLS version:** TLS 1.2 minimum. TLS 1.3 preferred. Cipher suite configuration is the responsibility of the termination layer, not the application.

**Certificate management:** Certificates are managed by the infrastructure layer (ACM, cert-manager, or equivalent). The application has no certificate material.

**Provider IP allowlisting:** If the provider publishes a static list of source IP ranges (GitHub does; Stripe does), those ranges are enforced at the load balancer or WAF layer — not in application code. Application-layer IP filtering adds latency and is bypassed if the load balancer forwards without preserving the source IP correctly.

---

### 7. HTTP Layer Configuration

**Decision:** The receiver exposes exactly one route: `POST /webhook`. All other methods on that path return `405`. All other paths return `404`. There is no index route, no health check on the public path, and no admin surface on the public interface.

**Health check:** A separate `GET /healthz` endpoint is exposed on a different port (or restricted to internal network access only). This endpoint is not reachable from the public internet. It returns `200` if the process is alive and the queue connection is healthy, `503` otherwise.

**Request ID:** Every inbound request is assigned a unique request ID at the earliest possible point in the middleware chain. This ID is attached to all log entries for that request and returned in the response as `X-Request-ID`. If the provider sends a delivery ID header, that is logged alongside the internal request ID.

## Part II — Operational Layer

### 2.1 Observability: Structured Logging

Every log entry is structured JSON. Free-text log lines are not produced. Log entries are not assembled by string concatenation.

**Fields on every log entry:** timestamp (ISO 8601, UTC), log level, service name, instance ID, request ID, event type, delivery ID, source IP, HTTP method, path, response status, response latency (ms), queue write latency (ms), signature validation result (success / failure-missing / failure-mismatch / failure-secondary).

**What is never logged:** The raw payload body. The HMAC secret. Any field that could contain PII from the payload. The computed HMAC value.

**Log levels:** `INFO` for successful receipt and enqueue. `WARN` for signature failures (expected attack surface noise, but must be monitored). `ERROR` for queue write failures, process errors, and unexpected exceptions. `DEBUG` is disabled in production by default and can be enabled per-instance without a deployment.

**Rationale:** When paged at 3am for a signature failure spike, the first question is: is this a rotation event, an attack, or a misconfiguration? The `signature_validation_result` field with its distinct values answers that question from the log query alone, without reading code.

---

### 2.2 Observability: Metrics

Metrics are emitted in a format compatible with the existing monitoring stack (Prometheus exposition format preferred). The following metrics are non-negotiable:

| Metric | Type | Labels |
|---|---|---|
| `webhook_requests_total` | Counter | `status`, `event_type`, `validation_result` |
| `webhook_request_duration_ms` | Histogram | `status` |
| `webhook_queue_write_duration_ms` | Histogram | `result` |
| `webhook_queue_write_failures_total` | Counter | `error_type` |
| `webhook_signature_failures_total` | Counter | `failure_reason` |
| `webhook_payload_size_bytes` | Histogram | — |

**Alerting thresholds (defined at deployment, not at code time):**
- `webhook_signature_failures_total` rate > N/min → page (possible attack or misconfiguration)
- `webhook_queue_write_failures_total` > 0 → page (data loss risk)
- `webhook_request_duration_ms` p99 > 500ms → alert (approaching provider timeout)
- `webhook_requests_total{status="500"}` rate > 0 → page

---

### 2.3 Observability: Distributed Tracing

The receiver emits a trace span for each inbound request. Child spans are created for: signature validation, queue write. The trace context is propagated into the queue message envelope so that downstream consumers can continue the trace across the async boundary.

This is not optional instrumentation. Without it, correlating a processing failure downstream to the original inbound event requires manual log correlation by delivery ID — which is slow and error-prone under incident conditions.

---

### 2.4 Secret Rotation Procedure

Rotation is a documented, rehearsed operational procedure — not an ad hoc action.

**Steps:**
1. Generate new secret in the secrets manager.
2. Update the secondary secret environment variable to the current (old) value.
3. Update the primary secret environment variable to the new value.
4. Trigger a rolling restart of the receiver instances (no downtime).
5. Update the secret in the provider's webhook configuration.
6. Monitor `webhook_signature_failures_total` and the secondary-secret validation log marker. Both should trend to zero as the provider propagates the new secret.
7. Once zero for a confirmed window (minimum: 1 hour, or one full provider retry cycle), remove the secondary secret from configuration.
8. Trigger a second rolling restart to clear the secondary secret from process memory.
9. Document the rotation in the change log with timestamps.

**Failure mode:** If step 5 is performed before step 4, there is a window where the provider is sending events signed with the new secret but the receiver only knows the old secret. All events in that window are rejected with `400`. The procedure order above prevents this.

---

### 2.5 Deployment and Container Configuration

**Process model:** One Node.js process per container. Clustering inside the container is not used. Horizontal scaling is handled at the container orchestration layer. This simplifies memory accounting, crash isolation, and signal handling.

**Signal handling:** The process handles `SIGTERM` with a graceful shutdown sequence: stop accepting new connections, allow in-flight requests to complete (with a bounded timeout, default 10 seconds), close the queue connection, exit. Abrupt termination on `SIGTERM` risks returning `200` for a request whose queue write was in flight.
