# Operational Design Specification: Production Webhook Receiver (Node.js)

---

## Document Status

This document is the authoritative handoff specification. Gaps here are gaps in production. Every decision recorded below has a corresponding failure mode that motivated it.

---

## Part I: Implementation Decisions

---

### 1. Signature Validation

#### 1.1 Algorithm and Canonical Form

Use HMAC-SHA256. The signature is computed over the **raw request body bytes**, not a parsed representation. Body parsing must be deferred until after the raw buffer is captured. Any middleware that parses the body before the signature check — JSON body parsers, schema validators — must not run upstream of the validation step. The order of middleware registration is a security property, not a style preference.

The comparison string must be constructed in the exact format the upstream provider specifies. For GitHub this is `sha256=<hex digest>`. For Stripe it is the full `v1,<hex digest>` scheme with timestamp inclusion. Deviation from the provider's canonical form produces a validator that passes the wrong events and rejects the right ones.

#### 1.2 Constant-Time Comparison

Signature comparison must use a constant-time equality function (`crypto.timingSafeEqual` in Node.js). String equality operators short-circuit on the first differing byte. That timing difference is measurable and exploitable. This is not a theoretical concern — it is a documented attack class. There is no acceptable reason to use `===` for signature comparison.

#### 1.3 Signature Header Absence

A request with no signature header is not an unsigned request that should be processed with reduced trust. It is an invalid request. Return `400`. Do not queue it. Do not log the body. Log the absence of the header, the source IP, and the timestamp.

#### 1.4 Signature Mismatch

Return `403`. Do not return `401`. There is no authentication challenge to issue. Log the mismatch with enough context to reconstruct the failure: the received header value (truncated or hashed, not raw), the expected digest prefix, the content-type, the body size in bytes, and the source IP. Do not log the secret. Do not log the full raw body in production unless log storage is encrypted and access-controlled.

#### 1.5 Secret Management

The HMAC secret is not an application configuration value. It is a secret. It must be injected at runtime via environment variable or a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager). It must not appear in source code, Dockerfiles, docker-compose files, CI logs, or Kubernetes manifests committed to version control. The secret must be loaded once at process startup and held in memory. It must not be re-read from disk or environment on each request — that is a performance problem and an inconsistency risk during rotation.

#### 1.6 Multiple Secrets for Rotation

The validator must support holding two active secrets simultaneously: the current secret and the incoming secret during a rotation window. Validation succeeds if either secret produces a matching digest. This is not optional. A rotation procedure that requires coordinated cutover between the provider and the receiver within a zero-tolerance window is a rotation procedure that will fail in production. See Section II.4 for the operational rotation procedure.

### 2. Request Handling and the 200 Contract

#### 2.1 The Constraint

The upstream provider has a timeout. GitHub's default delivery timeout is 10 seconds. Stripe's is similar. If the receiver does not respond within that window, the provider marks the delivery as failed and will retry. The receiver must return `200` before doing any downstream work. This is an architectural constraint, not a performance optimization.

#### 2.2 What Happens in the Request Path

Exactly four things happen before the response is sent:

1. Raw body is captured.
2. Signature is validated.
3. Event is enqueued to the internal queue.
4. `200` is returned.

Nothing else. No database writes. No downstream HTTP calls. No schema validation beyond what is necessary to extract a queue routing key. No logging that blocks the response.

#### 2.3 What Does Not Happen in the Request Path

- No downstream service calls.
- No database reads or writes.
- No event deduplication (this happens in the consumer, not the receiver).
- No business logic.
- No transformation beyond extracting a minimal envelope.

If the queue enqueue operation fails, return `500`. The provider will retry. This is the correct behavior. A `200` on a failed enqueue is a silent data loss event.

#### 2.4 Response Codes

| Condition | Response |
|---|---|
| Valid signature, enqueue succeeded | `200` |
| Missing signature header | `400` |
| Invalid signature | `403` |
| Enqueue failed | `500` |
| Body too large | `413` |
| Method not POST | `405` |
| Any unhandled exception | `500` |

Do not return `202`. Some providers treat `202` differently from `200`. Use `200` unless the provider's documentation explicitly requires `202`.

### 3. Queueing Architecture

#### 3.1 Queue Selection Criteria

The queue must satisfy three properties:

1. **Durability**: Events written to the queue survive process restart and container replacement.
2. **At-least-once delivery**: The consumer must receive every event that was enqueued, even if the consumer was unavailable when the event arrived.
3. **Acknowledgement-based consumption**: The consumer must explicitly acknowledge processing. Unacknowledged events must be redelivered after a configurable timeout.

In-memory queues (EventEmitter, in-process arrays, BullMQ backed by a non-persistent Redis instance) do not satisfy property 1. They are not acceptable for production. Acceptable options: Redis Streams with persistence enabled and AOF fsync policy set to `always` or `everysec`, RabbitMQ with durable queues and persistent message delivery mode, AWS SQS, Google Cloud Pub/Sub.

#### 3.2 Envelope Structure

Every message written to the queue must include:

- **event_id**: The provider's event identifier (e.g., GitHub's `X-GitHub-Delivery`, Stripe's event ID). Used for deduplication in the consumer.
- **event_type**: The provider's event type header (e.g., `X-GitHub-Event`, Stripe's `type` field). Used for routing in the consumer.
- **received_at**: ISO 8601 timestamp at time of receipt by the receiver. Not the provider's timestamp. The receiver's timestamp.
- **payload**: The raw body as a UTF-8 string or base64-encoded bytes. Do not parse it. The consumer owns parsing.
- **source_ip**: The IP address of the inbound request.
- **delivery_attempt**: Set to `1` on initial enqueue. The consumer increments this on requeue.

#### 3.3 Queue Unavailability

If the queue is unavailable when an event arrives, the receiver must return `500`. It must not buffer events in memory and retry the enqueue asynchronously. An in-memory buffer during queue outage is an invisible accumulation of undelivered events that will be lost on the next process restart. The provider's retry mechanism is the correct buffer. Let it be used.

#### 3.4 Backpressure

The receiver must apply a maximum queue depth check before enqueuing. If the queue depth exceeds a configured threshold, return `503`. This signals to the provider that the receiver is saturated and the provider should back off. A receiver that accepts events it cannot process is not a functioning receiver — it is a data loss machine with a `200` response code.

### 4. Body Handling

#### 4.1 Size Limit

Enforce a maximum body size before reading the body into memory. The limit must be set before the raw body capture middleware runs. A missing size limit is an unbounded memory allocation vulnerability on a public endpoint. Set the limit to the maximum payload size the upstream provider can send, plus a small margin. GitHub's maximum webhook payload is 25MB. Stripe's is significantly smaller. Set the limit to the provider's documented maximum. Not higher.

#### 4.2 Raw Body Capture

The raw body must be captured as a `Buffer` before any parsing occurs. The signature is computed over these bytes. If the body is parsed and re-serialized before signature computation, key ordering, whitespace, and encoding differences will produce a digest mismatch on valid events. This is a common implementation error. It produces a receiver that rejects all valid events silently.

#### 4.3 Content-Type

Do not reject requests based on `Content-Type` alone. Some providers send `application/json`. Some send `application/x-www-form-urlencoded`. Some send `application/octet-stream`. The signature is the trust boundary. Content-Type is metadata. Validate the signature. Pass the raw body to the consumer. Let the consumer parse it.

### 5. Concurrency and Process Model

#### 5.1 Single Container, Multiple Workers

The service runs as a single containerized process. Node.js is single-threaded. The event loop must not be blocked. All I/O — queue writes, any logging that involves network I/O — must be non-blocking. If the queue client performs synchronous I/O, it is not acceptable.

#### 5.2 Graceful Shutdown

On `SIGTERM`, the process must:

1. Stop accepting new connections.
2. Allow in-flight requests to complete, up to a configurable drain timeout (recommend 10 seconds).
3. Flush any buffered log output.
4. Close queue connections cleanly.
5. Exit.

A process that exits immediately on `SIGTERM` during a rolling deployment will drop in-flight requests. Those requests will be retried by the provider. Depending on provider retry behavior, this may or may not result in duplicate processing. It will result in elevated error rates during deployments. This is avoidable.

#### 5.3 Uncaught Exception Handling

Register handlers for `uncaughtException` and `unhandledRejection`. Both handlers must:

1. Log the error with full stack trace.
2. Flush logs.
3. Exit the process with a non-zero exit code.

Do not attempt to recover from an uncaught exception. The process state is undefined. Exit and let the container orchestrator restart the process. A process that swallows uncaught exceptions and continues running is a process in an unknown state accepting production traffic.

---

### 6. Idempotency Contract

The receiver does not enforce idempotency. The receiver's contract is: if the signature is valid and the queue accepts the message, return `200`. The same event may be enqueued more than once if the provider retries a delivery that received a `200` (this happens — network conditions, provider bugs). Deduplication is the consumer's responsibility, using the `event_id` field in the envelope. This division of responsibility must be documented and communicated to the consumer team. It is not the receiver's problem to solve, but it is the receiver's responsibility to make solvable by including the event ID in every envelope.

## Part II: Operational Layer

---

### II.1 Health and Readiness Endpoints

#### Liveness Endpoint

`GET /healthz`

Returns `200` if the process is running. No dependency checks. This endpoint exists to tell the container orchestrator whether to restart the container. It must always return `200` as long as the event loop is not blocked. If this endpoint is slow, the event loop is blocked, and that is the signal.

#### Readiness Endpoint

`GET /readyz`

Returns `200` only if the service is ready to accept webhook traffic. Checks:

- Queue connection is established and responsive (a lightweight ping, not a full operation).
- Secret is loaded and non-empty.

Returns `503` if any check fails. The container orchestrator must use this endpoint to gate traffic routing. A container that fails readiness must not receive webhook traffic. This is not the same as liveness. A container can be alive but not ready — during startup, during queue reconnection after a transient failure. Conflating liveness and readiness is a common error that results in traffic being routed to containers that cannot process it.

#### Operational Note

Do not put business logic in health endpoints. Do not query databases. Do not call downstream services. Health endpoints that are slow or fail due to downstream dependencies produce cascading failures where a downstream outage causes the receiver to be marked unhealthy and removed from rotation, compounding the incident.

### II.2 Structured Logging

#### Format

Every log line must be valid JSON. No exceptions. Human-readable log formats are not parseable by log aggregation systems under pressure. When you are debugging a signature failure at 3am, you need to be able to filter by `event_type`, `source_ip`, and `status_code` simultaneously. You cannot do that with unstructured logs.
