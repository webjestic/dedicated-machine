## 1. Scope and Constraints

This specification covers a single containerized Node.js service whose sole responsibility is to receive, authenticate, and enqueue webhook events from one external provider. It does **not** cover downstream consumers, event schema validation beyond what is required for routing, or provider-specific business logic.

**Hard constraints driving every decision below:**

- The endpoint is public. Any IP on the internet can POST to it.
- The signing secret is the only trust anchor. If validation is wrong or bypassable, every downstream system is compromised.
- Processing must not block the HTTP response. A slow consumer must not cause the provider to retry, which would produce duplicates.
- The service is a single container. There is no horizontal scaling assumption baked into this design, but nothing here must *prevent* horizontal scaling either.

## 2. Implementation Decisions

### 2.1 Ingestion Layer

**HTTP framework:** Express or Fastify. The choice is operationally inconsequential at this scale; what matters is the constraint applied to it: this service exposes **exactly one route** — `POST /webhook`. No other routes exist except the health endpoints defined in §3.9. Every other path returns 404. This is not a general-purpose API server.

**Raw body requirement:** HMAC-SHA256 is computed over the raw request body bytes, not a parsed representation. The framework must be configured to capture the raw body buffer *before* any JSON parsing occurs. This is a hard sequencing constraint. If the body is parsed first and then re-serialized for signature computation, the signature will fail on any payload where key ordering, whitespace, or Unicode normalization differs from what the provider sent. The raw buffer must be preserved and passed to the validation step directly.

**Body size limit:** Enforced at the HTTP layer before any processing. A hard cap — 1MB is a reasonable default for most providers; confirm against provider documentation. Requests exceeding this limit are rejected with 413 before signature validation runs. This prevents memory exhaustion from oversized payloads and removes a trivial denial-of-service vector.

**TLS termination:** Handled upstream (load balancer, ingress controller, or reverse proxy). The container itself listens on plain HTTP internally. TLS configuration does not belong in application code.

---

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. The provider sends a signature in a request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name and signature format are provider-specific and must be documented explicitly — some providers include a prefix (`sha256=`), some include a timestamp component that is part of the signed payload.

**Comparison:** Signatures must be compared using a **constant-time equality function**. Node.js provides `crypto.timingSafeEqual`. Standard string equality (`===`) is categorically forbidden here. Timing attacks against naive string comparison are not theoretical; they are practical against a public endpoint with measurable response times.

**Failure behavior:** Any request that fails signature validation returns **401** immediately. The body is not parsed. The event is not queued. A structured log entry is emitted (see §3.4). The response body is minimal — no diagnostic information is returned to the caller. The reason for rejection is logged internally, not exposed externally.

**What is validated:** The signature header must be present. If absent, the request is rejected with 401, not 400. Treating a missing signature as a bad request rather than an authentication failure leaks information about the validation path. Treat it uniformly as an authentication failure.

**What is not validated here:** Payload schema, event type routing, and idempotency are downstream concerns. The receiver's validation contract is binary: the signature is valid or it is not.

---

### 2.3 Async Decoupling and Queue Contract

**The invariant:** Once a valid event is enqueued, the HTTP response is sent. The receiver's job is complete. What happens to the event after enqueue is outside this service's responsibility boundary.

**Queue technology decision:** This depends on operational context, but the decision must be made explicitly and not deferred. Three options, in order of operational preference for a single-container deployment:

| Option | When to choose it |
|---|---|
| **External durable queue** (Redis Streams, SQS, RabbitMQ) | Default choice. Survives container restarts. Decouples receiver from consumer. Allows independent scaling. |
| **In-process queue with persistence** (e.g., BullMQ backed by Redis) | Acceptable if Redis is already in the stack and operational overhead of a separate queue is not justified. |
| **In-memory queue** | Not acceptable for production. Events are lost on container restart, OOM kill, or deployment. Explicitly ruled out. |

**Enqueue failure handling:** If the enqueue operation fails (queue unavailable, timeout, connection error), the receiver must **not** return 200. It must return **503**. This is the critical contract: a 200 tells the provider the event was accepted. If the event was not durably queued, a 200 is a lie that results in silent data loss. The provider will retry on 5xx, which is the correct behavior.

**Enqueue timeout:** The enqueue call must have an explicit timeout. If the queue is slow, the receiver must not hold the HTTP connection open indefinitely. The timeout value must be shorter than the provider's own HTTP timeout. If the enqueue times out, return 503.

**At-least-once delivery:** The queue must provide at-least-once delivery guarantees. This means consumers must be idempotent. Idempotency is a consumer responsibility, but the receiver must emit a stable, unique event identifier with every queued message to make idempotency keys possible (see §2.5).

---

### 2.4 Response Contract

| Condition | HTTP Status | Notes |
|---|---|---|
| Signature valid, event enqueued | **200** | Body: empty or minimal JSON acknowledgment |
| Signature invalid or missing | **401** | No diagnostic detail in body |
| Body too large | **413** | Before signature validation |
| Enqueue failure | **503** | Provider will retry |
| Method not POST | **405** | |
| Unknown path | **404** | |
| Unhandled exception | **500** | Caught by global error handler; structured log emitted |

The 200 response is sent only after the enqueue call returns successfully. It is not sent on a fire-and-forget basis. "Async" means the downstream *processing* is async — not the enqueue acknowledgment.

---

### 2.5 Payload Handling

**Delivery ID:** Most providers include a unique delivery identifier in a request header (e.g., `X-GitHub-Delivery`, `Stripe-Signature` contains an `id` component). This identifier must be extracted and included in the queued message. It serves as the idempotency key for consumers and as the correlation identifier for log tracing.

**Event type:** The provider's event type header (e.g., `X-GitHub-Event`) must be extracted and included in the queued message. Consumers need this to route events without re-parsing the payload.

**Queued message envelope:** The message placed on the queue must be a structured envelope containing at minimum:
- The raw payload (as a string or buffer, not parsed JSON — let consumers own parsing)
- The delivery ID
- The event type
- The timestamp of receipt (ISO 8601, UTC)
- The provider identifier (hardcoded for this deployment)

Parsing the payload in the receiver and placing parsed JSON on the queue is a mistake. It couples the receiver to the payload schema. If the provider changes the schema, the receiver breaks. The receiver's job is transport, not transformation.

---

### 2.6 Dependency Decisions

**Node.js version:** Current LTS. Not latest. LTS has a defined support window and predictable security patch cadence. Pin the exact version in the container image — not a range, not `node:lts`.

**Dependencies:** Minimized deliberately. The receiver does not need an ORM, a validation library, or a templating engine. Every dependency is an attack surface and an upgrade burden. The dependency list should be auditable in under five minutes.

**`crypto` module:** Node.js built-in. No third-party HMAC library. The built-in is audited, maintained, and sufficient.

## 3. Operational Layer

### 3.1 Secret Rotation

**The problem:** The signing secret will be rotated. This is not a hypothetical — providers rotate secrets on security incidents, and operators rotate them on personnel changes. A rotation that requires a container redeployment with downtime is not acceptable. A rotation that causes a validation outage — where valid events are rejected because the service has the old secret — is a data loss event.

**Decision: Support dual-secret validation during rotation windows.**

The service must support loading multiple active secrets simultaneously. During a rotation:

1. The new secret is added to the secret store alongside the old secret.
2. The service reloads secrets without restart (see below).
3. Validation attempts the new secret first. If it fails, it attempts the old secret. If both fail, the request is rejected.
4. Once the provider has been updated to use the new secret and a verification period has passed, the old secret is removed.
5. The service reloads again.

This window is typically 15–60 minutes. The exact duration must be defined in the runbook.

**Secret storage:** Secrets are never in environment variables baked into the container image. They are injected at runtime via a secrets manager (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets mounted as files). The service reads secrets on startup and must support a reload signal (SIGHUP or a polling interval) to pick up rotated values without restart.

**Audit trail:** Every secret reload must emit a structured log entry indicating how many secrets are now active. This is the operator's confirmation that rotation took effect.

---

### 3.2 Replay Attack Prevention

**The problem:** A valid, signed webhook payload captured by a network adversary (or a provider retry storm) can be replayed. The signature will validate. The event will be enqueued. The consumer will process it again.

**Decision: Timestamp validation plus delivery ID deduplication.**

**Timestamp validation:** Stripe's signature format includes a timestamp (`t=`) in the header. GitHub does not. For providers that include a timestamp:
- Extract the timestamp from the signature header.
- Reject any request where the timestamp is more than N seconds old (5 minutes is the Stripe-recommended default; this must be configurable).
- Reject any request where the timestamp is in the future by more than a small tolerance (30 seconds) to account for clock skew.
