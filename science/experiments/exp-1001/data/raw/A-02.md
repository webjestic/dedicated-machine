# Operational Design Specification: Production Webhook Receiver (Node.js)

---

## Preamble

This document specifies the design decisions and operational requirements for a production webhook receiver. It is not a code guide. It is a record of decisions made, failure modes anticipated, and operational obligations accepted. Every decision here exists because the alternative has a known cost.

---

# Part I: Implementation Decisions

---

## 1. Request Ingestion

### 1.1 HTTP Method and Path Enforcement

Only `POST` to the designated webhook path is accepted. All other methods return `405`. All other paths return `404`. These are not soft rejections — they are hard gates enforced before any body parsing begins. This limits the attack surface and prevents accidental processing of malformed requests from misconfigured senders.

### 1.2 Body Handling

The raw request body must be preserved as a `Buffer` before any parsing occurs. This is non-negotiable. HMAC-SHA256 signature validation operates on the exact bytes the sender signed. Any framework middleware that parses JSON before the raw body is captured will silently corrupt signature validation. The raw buffer is used for signature verification; the parsed JSON is used for everything else. These are two separate operations on two separate representations of the same payload.

**Failure mode:** If the body is parsed first and re-serialized for hashing, key ordering, whitespace, and encoding differences will cause valid signatures to fail. This will manifest as a sudden spike in 401s after a framework upgrade or middleware reorder — not during development.

### 1.3 Content-Length and Body Size Limits

A hard maximum body size is enforced before reading begins. The limit is chosen based on the maximum documented payload size of the upstream provider, with a reasonable ceiling (e.g., 1MB for GitHub, 512KB for Stripe). Requests exceeding this limit are rejected with `413` before the body is fully read. This prevents memory exhaustion from oversized payloads and slow-read attacks.

## 2. Signature Validation

### 2.1 Algorithm

HMAC-SHA256. The signature is computed over the raw request body using a shared secret. The algorithm is fixed and not negotiable per-request. Any provider header claiming a different algorithm is rejected, not accommodated.

### 2.2 Comparison Method

Signature comparison uses a constant-time equality function. Standard string equality (`===`) short-circuits on the first mismatched byte, which creates a timing side-channel that can be exploited to brute-force signatures byte by byte. Constant-time comparison eliminates this. Node.js provides `crypto.timingSafeEqual` for this purpose. There is no acceptable alternative.

### 2.3 Signature Header Parsing

The signature header format varies by provider. GitHub sends `sha256=<hex>`. Stripe sends `t=<timestamp>,v1=<hex>`. The parser must extract only the relevant signature component and handle malformed or missing headers explicitly. A missing header returns `401`. A malformed header returns `400`. These are different failure modes and must be logged differently.

### 2.4 Timestamp Validation (Replay Prevention)

For providers that include a timestamp in the signed payload or header (Stripe's `t=` field is the canonical example), the timestamp is validated against the current server time. Events older than a defined tolerance window (typically 5 minutes) are rejected with `400`. This prevents replay attacks where a captured valid request is resubmitted.

**Failure mode:** If the receiver's system clock drifts significantly from the sender's clock, legitimate events will be rejected. Clock synchronization (NTP or equivalent) is an operational dependency, not an assumption.

For providers that do not include a timestamp in the signature (GitHub), replay prevention must be handled at the idempotency layer downstream, not here.

### 2.5 Secret Management

The signing secret is never hardcoded. It is injected as an environment variable at runtime and sourced from a secrets manager (Vault, AWS Secrets Manager, or equivalent). The secret is not logged, not included in error responses, and not written to disk. The in-memory representation is a `Buffer`, not a string, to avoid accidental serialization.

### 2.6 Secret Rotation

The receiver must support simultaneous validation against two secrets: the current secret and the previous secret. During a rotation, the provider may send events signed with either key depending on timing. The receiver attempts validation with the current secret first. If that fails, it attempts validation with the previous secret. If both fail, the request is rejected. This window is time-bounded — the previous secret is removed after a defined rotation grace period (e.g., 24 hours). Without this, a secret rotation causes a guaranteed outage window.

## 3. Response Behavior

### 3.1 Immediate 200

A `200 OK` is returned as soon as the signature is validated and the event is accepted into the queue. No downstream processing occurs in the request path. The response body is minimal — a static JSON acknowledgment or empty body. No processing status, no event ID echoed back, no database round-trips.

**Rationale:** Third-party providers have short timeout windows (GitHub: 10 seconds, Stripe: 30 seconds). If processing is synchronous and the downstream system is slow or unavailable, the provider will retry. Retries compound backlog. Backlog compounds latency. The result is a cascade that takes down the receiver under normal load during a downstream incident.

### 3.2 Failure Response Codes

| Condition | Response |
|---|---|
| Missing or invalid signature | 401 |
| Malformed header or body | 400 |
| Body too large | 413 |
| Queue unavailable | 503 |
| Unexpected internal error | 500 |

`503` is returned when the queue cannot accept the event. This signals the provider to retry. A `200` must never be returned if the event was not successfully enqueued. That would be a silent data loss.

## 4. Async Queue

### 4.1 Queue Selection

A durable, external queue is required. In-memory queues (arrays, EventEmitter, even in-process Redis clients without persistence) are not acceptable for production. Acceptable options include Redis Streams with persistence enabled, RabbitMQ with durable queues and publisher confirms, AWS SQS, or Google Cloud Pub/Sub. The choice depends on the existing infrastructure, but the durability requirement is fixed.

**Failure mode:** An in-memory queue loses all pending events on process restart, container crash, or deployment. This is not a theoretical risk — containers restart.

### 4.2 Enqueue Semantics

The enqueue operation must be synchronous from the receiver's perspective — the receiver awaits confirmation that the message was accepted by the queue before returning `200`. Fire-and-forget enqueue followed by `200` is equivalent to in-memory queuing from a durability standpoint.

### 4.3 Payload Stored in Queue

The queued message includes: the raw event payload, the event type (from the provider's header), a receiver-generated UUID for the event, the timestamp of receipt, and the provider's delivery ID if available. The signing secret is never included in the queued message.

### 4.4 Idempotency Key

The provider's delivery ID (GitHub's `X-GitHub-Delivery`, Stripe's event ID) is stored as the idempotency key on the queued message. Downstream consumers use this key to deduplicate redeliveries. The receiver does not deduplicate — it queues everything valid. Deduplication at the receiver would require shared state and adds latency to the hot path.

### 4.5 Queue Backpressure

If the queue is full or the connection is unavailable, the receiver returns `503`. It does not block, it does not retry internally, and it does not buffer in memory. The provider's retry mechanism is the correct backpressure signal. Internal buffering masks queue unavailability from operators and creates a secondary failure mode.

## 5. Containerization

### 5.1 Process Model

Single Node.js process per container. No cluster mode. Horizontal scaling is achieved by running multiple container instances behind a load balancer. Cluster mode inside a container complicates signal handling, port binding, and health check behavior without meaningful benefit in a containerized environment.

### 5.2 Signal Handling

The process handles `SIGTERM` with a graceful shutdown sequence: stop accepting new connections, allow in-flight requests to complete (with a timeout), close the queue connection cleanly, then exit. The timeout for in-flight request completion is set below the container orchestrator's termination grace period. If in-flight requests do not complete within the timeout, the process exits anyway. Hanging indefinitely on shutdown causes deployment stalls.

### 5.3 Health Endpoints

Two endpoints:

- **`/healthz` (liveness):** Returns `200` if the process is running. No dependency checks. Used by the orchestrator to determine if the container should be restarted.
- **`/readyz` (readiness):** Returns `200` only if the queue connection is established and the process is ready to accept traffic. Returns `503` otherwise. Used by the orchestrator to determine if the container should receive traffic. A container that cannot reach the queue must not receive webhook deliveries.

These endpoints are on a separate internal port or path that is not exposed through the public load balancer.

### 5.4 Resource Limits

CPU and memory limits are set on the container. The receiver is stateless and I/O-bound. Memory usage is predictable if body size limits are enforced. If memory grows unexpectedly, it indicates a leak or a missing body size limit — not a reason to increase the limit.

## 6. Secret Rotation Procedure

### Trigger

Secret rotation is triggered on a schedule (e.g., quarterly), on suspected compromise, or when a team member with access to the secret departs.

### Procedure

1. Generate the new secret in the secrets manager.
2. Deploy the receiver with both the current secret (now `WEBHOOK_SECRET_PREVIOUS`) and the new secret (`WEBHOOK_SECRET_CURRENT`). The receiver validates against both.
3. Update the secret in the provider's dashboard (GitHub webhook settings, Stripe dashboard, etc.).
4. Monitor the `signature_validation_failure` metric and the `secret_version` label on successful validations. Failures should not spike. Successful validations should shift from `previous` to `current` within minutes.
5. After the grace period (24 hours minimum, or until zero events validate against the previous secret), remove `WEBHOOK_SECRET_PREVIOUS` from the secrets manager and redeploy.

### Failure Mode

If step 4 is skipped and the previous secret is removed before the provider has fully transitioned, all in-flight events signed with the old secret will be rejected. The provider will retry. If the receiver is now only accepting the new secret, those retries will succeed. The gap is bounded by the provider's retry window. This is acceptable only if the retry window is known and the event volume is low. For high-volume providers, the grace period is mandatory.

## 7. Logging

### 7.1 What Is Logged

Every request is logged with a structured JSON record containing:

- Receiver-generated request ID (UUID)
- Provider delivery ID (from header, if present)
- Event type (from header)
- HTTP method and path
- Request body size in bytes
- Signature validation result: `valid`, `invalid`, `missing`, `malformed`
- Secret version used for successful validation: `current` or `previous`
- Queue enqueue result: `success`, `failure`, `timeout`
- HTTP response code returned
- Total request duration in milliseconds
- Container instance ID

### 7.2 What Is Never Logged

- The raw or computed HMAC signature
- The signing secret or any derivative of it
- The request body content (it may contain PII or sensitive business data)
- Any value from a request header beyond the specific fields listed above

### 7.3 Log Level Policy

Successful requests: `INFO`. Signature failures: `WARN` with the validation result and delivery ID. Queue failures: `ERROR`. Unexpected exceptions: `ERROR` with stack trace. Debug logging (body parsing steps, timing breakdowns) is off by default and enabled via environment variable without redeployment.

### 7.4 Log Correlation

The receiver-generated request ID is included in the queued message. Downstream consumers log this ID. This creates an end-to-end trace from the inbound HTTP request to the final processing outcome without requiring a distributed tracing system.
