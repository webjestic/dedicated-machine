## 1. Scope and Boundaries

This service is responsible for exactly three things:

1. Authenticating inbound webhook POST requests via HMAC-SHA256 signature validation.
2. Returning an HTTP 200 to the sender immediately upon successful validation and enqueue.
3. Placing the validated event payload on a durable queue for downstream consumption.

This service is **not** responsible for:

- Processing or acting on event payloads.
- Retrying failed downstream operations.
- Routing events to multiple consumers.
- Any business logic whatsoever.

The boundary is the queue. Everything upstream of the queue is this service's responsibility. Everything downstream is not. This line must not blur during implementation.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Decision:** Use a minimal HTTP framework. Express or Fastify are both acceptable. Fastify is preferred for its schema-based request handling and lower overhead under load, but the choice is secondary to the constraints imposed on it.

**Constraints that are not negotiable:**

- Only `POST` requests to the webhook endpoint are accepted. All other methods return `405 Method Not Allowed`.
- The endpoint path must not be guessable or enumerable. A path like `/webhook` is insufficient. The path must include a non-secret but non-obvious segment (e.g., `/hooks/v1/receive`). Security does not depend on path obscurity, but obscurity reduces noise in access logs.
- No body parsing middleware runs before signature validation. The raw request body must be preserved exactly as received. Any middleware that parses, transforms, or re-serializes the body before validation will silently corrupt the HMAC comparison. This is the most common implementation error in webhook receivers.
- Request body size must be capped. A hard limit of 5MB is the default. The actual limit must be set to the maximum documented payload size of the upstream provider, plus a small buffer. Requests exceeding the limit return `413 Payload Too Large` and are not processed further.
- TLS termination occurs at the load balancer or ingress layer, not in this service. The service listens on plain HTTP internally. This is standard containerized deployment practice and must be documented so no one "fixes" it.

### 2.2 Signature Validation

**Decision:** HMAC-SHA256, using the raw request body as the message and the configured secret as the key. The exact signature scheme must match the upstream provider's documented specification precisely — including header name, prefix format (e.g., `sha256=`), and encoding.

**Constraints:**

- Signature comparison **must** use a constant-time comparison function. Node.js provides `crypto.timingSafeEqual()`. Using string equality (`===`) is a timing oracle and is not acceptable under any circumstance, including tests, staging, or "temporary" shortcuts.
- The signature header must be present. A missing header is an immediate `401`. It is not treated as a misconfiguration or a soft failure.
- The raw body buffer used for HMAC computation must be the exact bytes received on the wire. It must not be parsed, decoded, re-encoded, trimmed, or touched in any way before the HMAC is computed. The validation step must receive the buffer, not a string, not a parsed object.
- Validation failures return `401 Unauthorized`. The response body must be minimal — no diagnostic information, no error codes that reveal internal state. The only acceptable response body on a 401 is a static string such as `"Unauthorized"`.
- Validation is synchronous and completes before any queue interaction. An event that fails validation never touches the queue.

**What validation does not protect against by itself:**

- Replay attacks (covered in §3.6).
- A compromised signing secret (covered in §3.5).
- Requests that are valid but semantically malformed (the downstream consumer's problem, not this service's).

### 2.3 Request Body Handling

**Decision:** Buffer the entire request body in memory before processing begins.

**Rationale:** HMAC validation requires the complete body. Streaming validation is possible but adds complexity that is not justified at the payload sizes typical of webhook providers. If payload sizes approach or exceed available memory, this decision must be revisited and documented as a known constraint.

**Constraints:**

- The body buffer is held in memory for the duration of the request only. It is not written to disk, not logged, not stored anywhere except the queue payload.
- The body is parsed (JSON or otherwise) only after signature validation succeeds. Parsing before validation is wasteful and potentially dangerous if the parser has vulnerabilities.
- If the body cannot be parsed as the expected content type after validation succeeds, the service returns `400 Bad Request` and does not enqueue. The event is logged with enough context to reconstruct what happened (see §3.1).

### 2.4 Async Processing Architecture

**Decision:** The request handler does exactly two things after validation: enqueue the event, then return 200. No processing logic runs in the request path. No exceptions.

**The enqueue operation itself is synchronous from the handler's perspective** — the handler awaits confirmation that the message has been accepted by the queue broker before returning 200. This is a deliberate tradeoff.

**Rationale for awaiting enqueue confirmation:**

If the handler returns 200 before the enqueue is confirmed, and the enqueue subsequently fails, the event is silently lost. The upstream provider believes delivery succeeded. There is no retry. The event is gone. This is worse than returning a 5xx and allowing the provider to retry.

**Implication:** The queue broker's write latency is in the critical path of the HTTP response. This is acceptable. The queue write must be fast (sub-100ms under normal conditions). If the queue is unavailable, the handler returns 5xx and the provider retries. This is the correct behavior.

**What does not run in the request path:**

- Payload transformation.
- Database lookups.
- Downstream HTTP calls.
- Any I/O other than the single queue write.

### 2.5 Queue Design

**Decision:** Use a durable, persistent message queue with at-least-once delivery guarantees. Acceptable options: Redis Streams, RabbitMQ, AWS SQS, Google Cloud Pub/Sub. An in-process queue (e.g., an array, EventEmitter, or Bull queue backed by a non-durable Redis instance) is not acceptable for production.

**Required queue properties:**

- **Durability:** Messages must survive a broker restart. In-memory-only queues are excluded.
- **At-least-once delivery:** The queue must guarantee that a message is not lost if the consumer crashes before acknowledging it. Exactly-once is preferable where available but not required from this service's side.
- **Persistence of the raw payload:** The queue message must contain the original raw event payload, the event type (from headers), a timestamp of receipt, a unique message ID generated by this service, and the source IP. Nothing more is required. Nothing less is acceptable.
- **No transformation before enqueue:** The payload enqueued is the validated raw body. Downstream consumers own parsing and transformation. This service does not make assumptions about what downstream needs.

**Queue message schema (fields, not format):**

| Field | Source | Purpose |
|---|---|---|
| `messageId` | Generated (UUID v4) at receipt | Deduplication, tracing |
| `receivedAt` | Server timestamp (ISO 8601, UTC) | Replay detection, ordering |
| `eventType` | Provider-specific header | Consumer routing |
| `deliveryId` | Provider-supplied event ID header | Upstream deduplication |
| `sourceIp` | Request remote address | Audit, abuse detection |
| `payload` | Raw body (string or buffer) | Downstream processing |

### 2.6 Idempotency and Deduplication

**Decision:** This service provides a best-effort deduplication layer using the provider's delivery ID. It does not guarantee exactly-once delivery to the queue. Downstream consumers must be idempotent.

**Mechanism:** On receipt of a valid event, check a short-lived cache (Redis with TTL) for the provider's delivery ID. If found, return 200 without re-enqueuing. If not found, enqueue and write the delivery ID to the cache with a TTL of 24 hours (or the provider's documented retry window, whichever is longer).

**Why this is best-effort and not a guarantee:** The check-then-enqueue sequence is not atomic. Under concurrent delivery of the same event (which providers do under retry conditions), two instances of this service may both pass the deduplication check before either writes to the cache. This is acceptable. Downstream consumers must handle it. This service's deduplication reduces noise; it does not eliminate the requirement for downstream idempotency.

**If no delivery ID is present in the request:** Log a warning. Enqueue the event. Do not reject it. The provider may not supply a delivery ID, and rejecting valid events because of a missing deduplication key is worse than allowing duplicates.

### 2.7 Secret Management

**Decision:** The HMAC signing secret is never hardcoded, never in environment variables committed to source control, and never logged under any circumstances.

**Injection mechanism:** The secret is injected at container startup via a secrets management system (e.g., AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets with appropriate RBAC). The exact mechanism depends on the deployment environment and must be specified before deployment.

**In-process handling:**

- The secret is loaded once at startup and held in memory as a Buffer, not a string, to reduce the risk of accidental serialization.
- The secret is never attached to a request object, never included in a log line, never returned in an error response.
- If the secret fails to load at startup, the process must exit immediately with a non-zero exit code. A service that starts without a valid secret is not a degraded service — it is a broken service that will silently reject all events.
