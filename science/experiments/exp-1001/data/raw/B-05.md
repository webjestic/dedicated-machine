# Operational Design Specification: Production Webhook Receiver (Node.js)

**Version:** 1.0
**Status:** Authoritative handoff document
**Scope:** Single containerized webhook receiver service — implementation decisions and operational layer

---

## Preamble

This document exists because the gap between "validates signatures" and "safe to operate in production" is where incidents live. Every decision recorded here has a failure mode attached to it. Every operational requirement is a production constraint, not a suggestion. An engineer deploying this service without reading this document in full is not ready to deploy this service.

---

# Part I: Implementation Decisions

---

## 1. Ingress and Transport

### 1.1 TLS Termination

TLS is terminated upstream at a load balancer or reverse proxy (e.g., nginx, AWS ALB). The Node.js process listens on plain HTTP internally. Rationale: Node.js TLS handling is not the right layer for certificate rotation, cipher suite management, or OCSP stapling under operational pressure.

**Constraint:** The internal port must not be exposed outside the container network boundary under any circumstances. Exposure of the plain HTTP port to the public internet is a critical misconfiguration.

### 1.2 Listening Interface

The process binds to `0.0.0.0` on a single configured port. The port is injected via environment variable with no hardcoded default in production builds.

### 1.3 Request Size Limit

Maximum raw body size is enforced at the HTTP layer before any parsing occurs. The limit is set to a value that accommodates the largest legitimate payload from the upstream provider with a defined margin — not an arbitrary large number. For GitHub and Stripe, 1MB is a defensible ceiling. Requests exceeding this limit are rejected with `413` before signature validation is attempted.

**Failure mode this prevents:** Memory exhaustion via oversized body delivery. A 413 before body buffering is the correct behavior. A 413 after buffering is not.

### 1.4 HTTP Method Enforcement

Only `POST` is accepted on the receiver endpoint. All other methods return `405 Method Not Allowed`. This is enforced at the routing layer, not the application logic layer.

## 2. Raw Body Preservation

### 2.1 Why This Decision Exists

HMAC-SHA256 signature validation is computed against the raw request body as received — the exact byte sequence. Any middleware that parses the body (JSON parsing, URL decoding, body normalization) before the signature check will mutate the byte sequence and cause validation to fail for legitimate requests, or worse, pass validation on a mutated payload.

### 2.2 Decision

The raw body is buffered as a `Buffer` (not a string) before any other processing. The raw buffer is used for signature computation. JSON parsing is performed separately, after signature validation passes, on the same buffer. The raw buffer is not retained beyond the request handler scope.

**Constraint:** No body-parsing middleware (e.g., `express.json()`) is applied globally. It is applied only after signature validation, scoped to the validated buffer. Global body-parsing middleware is a misconfiguration in this service.

## 3. Signature Validation

### 3.1 Algorithm

HMAC-SHA256. The secret is the shared webhook secret provisioned by the upstream provider. The computed HMAC is compared against the signature delivered in the request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).

### 3.2 Comparison Method

Signature comparison uses a constant-time equality function. String equality operators (`===`) are not used for this comparison under any circumstances.

**Failure mode this prevents:** Timing oracle attacks. A character-by-character short-circuit comparison leaks information about how many bytes of the signature are correct. At scale, this is exploitable.

### 3.3 Header Presence Enforcement

If the signature header is absent, the request is rejected with `401` immediately. The absence of a signature header is not treated as a degraded case or a configuration issue to log and continue. It is a rejection condition.

### 3.4 Signature Format Validation

The signature header value is validated for expected format before HMAC computation begins. For GitHub, the expected prefix is `sha256=`. A header value that does not match the expected format is rejected with `400`. This prevents malformed input from reaching the crypto layer.

### 3.5 Timestamp Validation (Provider-Dependent)

For providers that include a timestamp in the signature scheme (Stripe includes `t=<timestamp>` in `Stripe-Signature`), the timestamp is validated against the current time within a defined tolerance window. The tolerance window is a configured value, not hardcoded. The recommended default is 300 seconds (5 minutes), matching Stripe's own recommendation.

**Failure mode this prevents:** Replay attacks. A valid signature on a captured payload is permanently valid without timestamp enforcement. A 5-minute window limits the replay surface to the operational window of a captured request.

**Operational dependency:** This validation requires the service's system clock to be accurate. NTP synchronization is a hard operational requirement, not an assumption. See Section II.4.

### 3.6 Rejection Response Behavior

All rejection responses (401, 400, 413) return no body or a minimal static body. No diagnostic information about why validation failed is returned to the caller. Detailed failure reasons are logged internally only.

**Failure mode this prevents:** Information disclosure. A response that distinguishes "signature malformed" from "signature incorrect" from "timestamp expired" gives an attacker a validation oracle.

## 4. Async Processing Architecture

### 4.1 Core Constraint

The HTTP response is returned before any downstream processing begins. The request handler's only responsibilities are: validate the signature, enqueue the event, return `200`. Processing logic does not execute in the request path.

### 4.2 Queue Selection

The queue is a durable, external message queue — not an in-process queue, not a local array, not an EventEmitter. Candidates: Redis Streams, RabbitMQ, AWS SQS, Google Pub/Sub. The selection criterion is durability: if the container restarts after the `200` is returned but before processing completes, the event must not be lost.

**Failure mode this prevents:** Silent event loss. An in-process queue loses all unprocessed events on container restart. This is not a theoretical concern — containers restart during deployments, OOM kills, and health check failures.

### 4.3 Enqueue-Before-Respond Ordering

The sequence is: validate → enqueue → respond `200`. The `200` is not sent before the enqueue operation completes. If the enqueue operation fails, the response is `500`, not `200`.

**Failure mode this prevents:** Acknowledged-but-lost events. Returning `200` before a successful enqueue tells the upstream provider the event was received. If the enqueue then fails, the event is gone. The upstream provider will not retry an acknowledged event.

### 4.4 Enqueue Failure Behavior

If the enqueue operation fails (queue unavailable, timeout, network error), the service returns `500`. The upstream provider is expected to retry on `5xx`. This is the correct behavior. Swallowing the error and returning `200` is not acceptable.

**Operational dependency:** The retry behavior of the upstream provider must be confirmed and documented. GitHub retries on `5xx` for up to 72 hours. Stripe retries on `5xx` for up to 72 hours with exponential backoff. This retry window defines the maximum acceptable queue outage before events are permanently lost.

### 4.5 Enqueue Timeout

The enqueue operation has a hard timeout. If the queue does not acknowledge the enqueue within the timeout window, the operation is treated as a failure and `500` is returned. The timeout value is configured, not hardcoded. A reasonable starting value is 2–3 seconds. This prevents the request handler from hanging indefinitely on a degraded queue.

### 4.6 Payload Envelope

The message placed on the queue is not the raw body alone. It is an envelope containing:

- The parsed event payload
- The event type (from the provider's event-type header, e.g., `X-GitHub-Event`)
- The provider's delivery ID (e.g., `X-GitHub-Delivery`, `Stripe-Signature` request ID component) for idempotency tracking
- The receipt timestamp (UTC, ISO 8601) assigned by the receiver at validation time
- The source IP of the request (for audit purposes)

The delivery ID is the key field for downstream idempotency. Its presence must be validated before enqueue. If the upstream provider does not supply a delivery ID, the receiver generates a UUID and records this in logs.

## 5. Response Behavior

### 5.1 Success Response

`200 OK` with no body, or a minimal static body (e.g., `{"status":"accepted"}`). The response body is not dynamic and does not reflect any event content.

### 5.2 Rejection Codes

| Condition | Response Code |
|---|---|
| Signature header absent | 401 |
| Signature format invalid | 400 |
| Signature mismatch | 401 |
| Timestamp out of window | 401 |
| Body too large | 413 |
| Enqueue failure | 500 |
| Method not allowed | 405 |
| Unknown route | 404 |

### 5.3 No Retry-After Header on 429

Rate limiting (see Section II.6) returns `429` without a `Retry-After` header. Providing a `Retry-After` header tells an attacker exactly when to resume. Legitimate providers will back off on `429` without guidance.

## 6. Secret Management

### 6.1 Secret Injection

The webhook signing secret is injected via environment variable at runtime. It is not present in the container image, the source repository, or any configuration file committed to version control.

### 6.2 Secret Rotation Protocol

Secret rotation is a defined operational procedure, not an ad hoc action. The procedure is:

1. The new secret is provisioned in the secret store.
2. The service is configured to accept signatures computed with either the old secret or the new secret during a transition window.
3. The upstream provider is updated to use the new secret.
4. After confirming the upstream provider is using the new secret (via successful validation logs), the old secret is removed from the service configuration.
5. The transition window is time-bounded. A maximum of 24 hours is recommended.

**Failure mode this prevents:** A hard cutover — updating the upstream provider and the service simultaneously — creates a window where all events fail validation. At scale, this causes a backlog of failed deliveries and potential event loss if the provider's retry window is shorter than the recovery time.

### 6.3 Dual-Secret Validation Logic

When two secrets are active during rotation, both are attempted in sequence. The first match succeeds. The result (which secret matched) is logged at debug level for rotation confirmation. This log entry must not appear in production log aggregation at a level that exposes the secret value — only which slot matched (e.g., "primary" or "secondary").

## 7. Health and Readiness

### 7.1 Health Endpoint

A `/health` or `/healthz` endpoint returns `200` when the process is running. This endpoint does not validate queue connectivity. It is a liveness signal only.

### 7.2 Readiness Endpoint

A `/ready` endpoint returns `200` only when the service can successfully reach the queue. It returns `503` when the queue is unreachable. The container orchestrator uses this endpoint to gate traffic. A pod that cannot reach the queue must not receive webhook traffic — it will return `500` on every event.

**Failure mode this prevents:** Traffic routing to a degraded instance. Without a readiness check, a container that has lost queue connectivity continues to receive requests, returns `500` on all of them, and the upstream provider begins accumulating retry backlog against a service that cannot process events.

### 7.3 Health Endpoint Security

The health and readiness endpoints are not authenticated. They must not return any information about the service's internal state, configuration, queue depth, or secret status. They return only the status code.

## II.1 Structured Logging

### II.1.1 Format

All log output is JSON, one object per line, written to stdout. No plaintext log lines. No multi-line log entries. Rationale: log aggregation systems (Datadog, Splunk, CloudWatch Logs Insights) parse structured JSON. Plaintext logs require fragile regex parsing and break under pressure.
