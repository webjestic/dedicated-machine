# Operational Design Specification: Production Webhook Receiver (Node.js)

---

## Document Status

**Scope:** Implementation decisions and operational requirements for a containerized Node.js webhook receiver. No code. Decisions, rationale, and failure modes only.

---

## Part I: Implementation Decisions

---

### 1. Signature Validation

#### 1.1 Algorithm and Comparison

HMAC-SHA256 is the mandated algorithm. The signature is computed over the **raw request body** — not a parsed representation. This is non-negotiable. Any middleware that parses, transforms, or re-serializes the body before signature computation will produce a mismatch under normal conditions and silently pass tampered payloads under adversarial ones.

The raw body must be captured as a `Buffer` before any parsing occurs. The content-encoding header must be respected: if the sender compresses the body, decompression must happen before hashing, and the decompressed bytes are what get hashed. Confirm this against the provider's documentation explicitly — do not assume.

Comparison between the computed digest and the received signature **must use a constant-time comparison function**. A byte-by-byte short-circuit comparison leaks timing information that can be used to forge valid signatures incrementally. Node.js provides `crypto.timingSafeEqual` for this purpose. It must be used. There is no acceptable alternative.

#### 1.2 Header Parsing

The signature header format varies by provider. Stripe sends `Stripe-Signature` as a compound value containing a timestamp and one or more signature versions (`t=...,v1=...`). GitHub sends `X-Hub-Signature-256` as a single hex-prefixed value (`sha256=...`). The parsing logic must be written against the specific provider's documented format. Generic header parsing is a defect waiting to manifest.

The timestamp embedded in Stripe-style headers is a replay-prevention mechanism and must be validated as part of signature validation, not as a separate optional step. See Section 1.4.

#### 1.3 Secret Management

The signing secret is a credential. It must never appear in source code, environment variable definitions committed to version control, or log output. It is injected at runtime via a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager — the specific system is an infrastructure decision, but the requirement is a secrets manager, not a `.env` file).

The secret is loaded once at startup and held in memory. It is not re-fetched on every request. The implication of this decision is addressed in Section 2.3 (Secret Rotation).

The secret must not be logged. The computed HMAC must not be logged. The received signature header may be logged at debug level for diagnostic purposes only, and only if the logging pipeline itself is secured. In production, it should not be logged at all.

#### 1.4 Replay Attack Prevention

A valid signature proves the payload was signed with the correct secret. It does not prove the payload was not captured and retransmitted. Replay prevention requires a time bound.

For providers that embed a timestamp in the signature header (Stripe's `t=` field), the receiver must reject any event where the timestamp falls outside a defined tolerance window. Five minutes is the conventional value and is defensible. Ten minutes is the outer limit of acceptability. Beyond that, the replay window is operationally meaningless.

For providers that do not embed a timestamp (GitHub's `X-Hub-Signature-256`), the receiver must implement its own idempotency check using the event ID, if the provider supplies one. The event ID is stored in a short-lived cache (Redis with TTL, or equivalent) keyed by provider-event-ID. If the ID is already present, the request is rejected with 200 (see Section 1.6 on response codes). The TTL on the cache entry must exceed the maximum expected delivery retry window of the provider.

If the provider supplies neither a timestamp nor a stable event ID, the replay attack surface cannot be fully closed at the receiver layer. This must be documented as a known risk and mitigated downstream.

#### 1.5 Body Size Limits

The receiver must enforce an explicit maximum body size before signature validation begins. Accepting an unbounded body and then validating it allows a denial-of-service vector: an attacker sends a multi-gigabyte payload, exhausting memory before the signature check rejects it.

The limit must be set based on the provider's documented maximum payload size, with a reasonable margin. For most providers this is in the range of 1–25 MB. The limit is enforced at the HTTP layer, before the body is buffered. Requests exceeding the limit are rejected with 413 before any processing occurs.

#### 1.6 Response Codes

The receiver returns **200** for every request that passes signature validation, regardless of what happens downstream. This is the contract.

The rationale: the provider's retry logic is triggered by non-2xx responses. If the receiver returns 500 because the queue is temporarily unavailable, the provider will retry. If the queue recovers before the retry, the event will be processed twice. Downstream idempotency must handle this — but the receiver should not amplify the problem by making retries more likely than necessary.

The specific response codes:

| Condition | Response |
|---|---|
| Signature invalid | 401 |
| Body too large | 413 |
| Method not POST | 405 |
| Signature valid, event enqueued | 200 |
| Signature valid, queue unavailable | 200 (event dropped — see Section 1.8) |
| Duplicate event ID detected | 200 |
| Timestamp outside tolerance | 401 |

Returning 401 for signature failures is intentional. It is honest. It does not reveal whether the secret is wrong, the payload was tampered, or the timestamp expired — the response is identical in all cases.

#### 1.7 Async Processing and Queue Design

The receiver's only job after signature validation is to enqueue the event and return 200. No business logic. No database writes. No downstream HTTP calls. The request path ends at the queue boundary.

The queue must be a durable external queue: SQS, RabbitMQ with durable queues and publisher confirms, Redis Streams with consumer groups, or equivalent. An in-process queue (an array, an EventEmitter, a `p-queue` instance) is not acceptable. If the process crashes, in-process events are lost with no recovery path.

The enqueue operation must be synchronous from the receiver's perspective — the receiver awaits confirmation that the message was accepted by the queue before returning 200. If the enqueue call fails, the receiver has a decision to make (see Section 1.8).

The payload written to the queue must include:
- The raw event body (as received, not re-serialized)
- The event type header from the provider
- The event ID, if available
- The timestamp of receipt at the receiver
- The provider identity (relevant if the receiver handles multiple providers)

#### 1.8 Queue Unavailability

This is the hardest failure mode in the receiver. The signature is valid. The event is real. The queue is down. The receiver cannot process the event and cannot ask the provider to retry without risking a duplicate.

The decision: **return 200 and log the loss, or return 5xx and accept the duplicate risk.**

The correct answer depends on the downstream system's idempotency guarantees and the provider's retry behavior. If the downstream system is fully idempotent and the provider's retry window is long, returning 5xx is acceptable — the event will be retried and processed once. If the downstream system is not idempotent, returning 200 and logging the loss is the lesser harm.

This decision must be made explicitly and documented. It cannot be left to the default behavior of whatever HTTP framework is in use. The chosen behavior must be tested.

A secondary mitigation: implement a local dead-letter buffer (a bounded in-memory queue or a local file write) that captures events during queue outages and replays them when the queue recovers. This is operationally complex and introduces its own failure modes (process crash during outage loses the buffer). It is worth considering for high-value event streams. It is not a substitute for fixing the queue.

#### 1.9 Endpoint Security

The receiver is a public HTTP endpoint. Its attack surface is real.

- **TLS termination** is handled upstream (load balancer or ingress). The receiver itself listens on plain HTTP internally. TLS must be enforced at the ingress layer — the receiver must not be reachable over plain HTTP from outside the cluster.
- **IP allowlisting** must be implemented if the provider publishes a stable list of source IP ranges. GitHub and Stripe both publish these. The allowlist is enforced at the network layer (security group, ingress rule) — not in application code. Application-layer IP checks are a defense-in-depth measure, not a primary control.
- **Path obscurity** is not a security control. A non-guessable path (e.g., `/webhooks/github/a3f9c2b1`) provides marginal noise reduction against automated scanners. It is not a substitute for signature validation.
- **Rate limiting** is applied at the ingress layer. The receiver does not implement rate limiting in application code. The limit must be set high enough to accommodate legitimate burst delivery from the provider and low enough to limit abuse.

## Part II: Operational Layer

---

### 2.1 Structured Logging

Every request must produce a structured log entry. The log entry is the primary diagnostic tool when something goes wrong at 3am.

**Required fields on every request log entry:**

- `timestamp` (ISO 8601, UTC)
- `request_id` (generated at ingress, propagated through the system)
- `provider` (github, stripe, etc.)
- `event_type` (the event type header from the provider)
- `event_id` (if available)
- `signature_valid` (boolean)
- `validation_failure_reason` (if applicable: `invalid_signature`, `timestamp_expired`, `missing_header`, `body_too_large`)
- `enqueue_success` (boolean)
- `enqueue_duration_ms`
- `total_duration_ms`
- `http_status_returned`

**Fields that must never appear in logs:**

- The signing secret
- The computed HMAC digest
- The full signature header value (at production log level)
- Any PII present in the event payload

Log format is JSON. Human-readable log formats are not acceptable in production — they cannot be reliably parsed by log aggregation systems and they break under payloads that contain newlines.

Log level discipline:

- `INFO`: every valid event received and enqueued
- `WARN`: duplicate event detected, timestamp outside tolerance, queue latency elevated
- `ERROR`: signature validation failure, enqueue failure, queue unavailable
- `DEBUG`: raw header values, timing breakdown — disabled in production

### 2.2 Metrics and Alerting

The following metrics must be emitted. The specific system (Prometheus, Datadog, CloudWatch) is an infrastructure decision. The metrics are not.

**Counters:**

| Metric | Labels |
|---|---|
| `webhook_requests_total` | `provider`, `status` (valid/invalid/error) |
| `webhook_signature_failures_total` | `provider`, `reason` |
| `webhook_enqueue_failures_total` | `provider` |
| `webhook_duplicate_events_total` | `provider` |
| `webhook_replay_rejections_total` | `provider` |

**Histograms:**

| Metric | Labels |
|---|---|
| `webhook_validation_duration_ms` | `provider` |
| `webhook_enqueue_duration_ms` | `provider` |
| `webhook_request_duration_ms` | `provider`, `status` |

**Gauges:**

| Metric | Description |
|---|---|
| `webhook_queue_depth` | Current depth of the downstream queue |
| `webhook_secret_age_days` | Days since last secret rotation |

**Alert thresholds (starting points — tune after observing baseline):**

| Alert | Condition | Severity |
|---|---|---|
| Signature failure spike | `webhook_signature_failures_total` rate > 10/min for 2 min | Page |
| Enqueue failure | Any `webhook_enqueue_failures_total` increment | Page |
| Queue depth elevated | `webhook_queue_depth` > threshold for 5 min | Warn |
| No events received | Zero `webhook_requests_total` for 15 min during business hours | Warn |
| High request latency | p99 `webhook_request_duration_ms` > 500ms | Warn |

The signature failure spike alert is the most important. A sudden increase in signature failures is the primary signal for a secret rotation event that was not coordinated with the receiver, or an active attack. Both require immediate response.
