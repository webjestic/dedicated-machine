## 1. Scope and Constraints

This specification covers a single containerized Node.js service whose sole responsibility is to receive inbound webhook POST requests from a trusted third-party provider, validate their authenticity, and enqueue them for asynchronous downstream processing. The service is not responsible for processing event payloads. It is responsible for the integrity of the handoff.

**Hard constraints that shape every decision below:**

- The raw request body must be preserved byte-for-byte before any parsing. HMAC validation operates on the wire bytes, not on a re-serialized object.
- The 200 response must be returned before any downstream work begins. The third-party provider's retry logic must never be triggered by internal processing latency.
- Signature validation failure must be a hard rejection with no fallback path.
- The queue is the boundary of this service's responsibility. If the event is enqueued, this service has succeeded. If it cannot be enqueued, this service must not return 200.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework selection:** A minimal HTTP framework — Fastify or bare `node:http` — is preferred over Express. The rationale is predictable middleware ordering and explicit body handling. Express's body-parser middleware, if misconfigured, will consume and re-serialize the body before the signature validation step can access the raw bytes. This is a class of bug that has silently broken signature validation in production systems. Fastify's `addContentTypeParser` with `parseAs: 'buffer'` makes the raw body the explicit contract.

**TLS termination:** Handled upstream by a load balancer or ingress controller. The container itself listens on plain HTTP. Accepting TLS inside the container adds complexity with no security benefit when the network path from ingress to container is private.

**Listening interface:** Bind to `0.0.0.0` inside the container. The container network boundary provides isolation. Binding to `127.0.0.1` inside a container is a common misconfiguration that silently causes the service to be unreachable.

**HTTP method enforcement:** Only POST is accepted on the webhook endpoint. All other methods return 405. This is not a security control — it is noise reduction in logs and a signal that something is misconfigured upstream.

**Endpoint path:** A non-guessable path prefix (e.g., `/webhooks/github/inbound`) is preferred over `/webhook`. This provides no cryptographic security but reduces automated scanner noise in access logs, which matters when debugging under pressure.

---

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. The signature is computed over the raw request body bytes using the shared secret. The result is compared against the value provided in the request header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).

**Comparison method:** Timing-safe comparison using `crypto.timingSafeEqual` is mandatory. String equality (`===`) leaks timing information that can be used to reconstruct the expected signature byte by byte. This is not theoretical — it is a documented attack class against HMAC verification endpoints. There is no acceptable reason to use string equality here.

**Failure behavior:** A signature mismatch returns 401 immediately. The body is not parsed. The event is not enqueued. No partial processing occurs. The failure is logged with enough context to diagnose (see §3.1) but without logging the raw signature value or the secret.

**Header absence:** If the signature header is absent entirely, the request is rejected with 401, not 400. The distinction matters: 400 implies a malformed request that might be retried with correction. 401 implies an authentication failure. A missing signature header is an authentication failure.

**Provider-specific envelope parsing:** Some providers (Stripe) embed a timestamp and multiple signature versions in a single header value. The parsing of this envelope must be treated as part of the validation step, not a preprocessing step. If the envelope cannot be parsed, the request is rejected before any HMAC computation occurs.

---

### 2.3 Request Body Handling

**Raw buffer preservation:** The request body must be read as a `Buffer` before any other processing. This buffer is what the HMAC is computed over. After validation, the buffer is parsed as JSON for enqueuing. If JSON parsing fails after a valid signature, this is logged as an anomaly (a signed but malformed payload is unusual and worth investigating) and the request is rejected with 422.

**Size limit:** A hard body size limit is enforced at the HTTP layer before the body is read into memory. 5MB is a reasonable upper bound for webhook payloads; anything larger is almost certainly not a legitimate webhook event and is a potential memory exhaustion vector. The limit is configurable via environment variable to avoid a code change for tuning.

**Content-Type enforcement:** `Content-Type: application/json` is required. Requests with other content types are rejected with 415 before the body is read. This is a defense-in-depth measure, not a primary security control.

---

### 2.4 Async Dispatch and Queuing

**Queue technology decision:** The queue must be external to the process. An in-process queue (array, EventEmitter, etc.) does not survive a container restart, a crash, or a deployment. Events in an in-process queue at the moment of a crash are silently lost. This is not acceptable. The queue must be a durable external system: Redis Streams, RabbitMQ, SQS, or equivalent.

**Selection criteria for queue technology:**

| Criterion | Implication |
|---|---|
| At-least-once delivery | Required. The downstream processor must be idempotent. |
| Durability across restarts | Required. Eliminates in-process queues entirely. |
| Backpressure signaling | Required. The receiver must know when the queue is saturated (see §3.7). |
| Operational complexity | A managed service (SQS, Cloud Pub/Sub) is preferred over self-managed for a single containerized deployment. |

**Enqueue-then-respond contract:** The enqueue operation must complete successfully before the 200 is returned. The sequence is: validate signature → parse body → enqueue → return 200. If the enqueue fails, the service returns 503. The third-party provider will retry. This is the correct behavior. A 200 that lies about successful enqueuing is worse than a 503 that triggers a retry.

**Payload envelope:** The message placed on the queue must include: the raw JSON payload, the event type header (e.g., `X-GitHub-Event`), the delivery ID header (e.g., `X-GitHub-Delivery`), the timestamp of receipt, and the source IP. The delivery ID is the deduplication key. The source IP is for forensic use only.

**No business logic in the receiver:** The receiver does not inspect the event type to make routing decisions. It enqueues everything that passes validation. Routing is the downstream processor's responsibility. This keeps the receiver's failure surface minimal.

---

### 2.5 Secret Management

**Storage:** The HMAC secret is never hardcoded and never in source control. It is injected as an environment variable at runtime, sourced from a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager). The container image contains no secrets.

**Access pattern:** The secret is read once at startup and held in memory for the lifetime of the process. It is not re-read on every request. Re-reading on every request introduces latency and a dependency on the secrets manager in the hot path.

**Multiple active secrets:** The service must support holding more than one active secret simultaneously. This is required for secret rotation (see §3.5). The implementation validates the incoming signature against each active secret in sequence. The first match is a pass. This list is bounded (two secrets maximum during rotation) and does not introduce meaningful latency.

---

### 2.6 Idempotency and Deduplication

**Delivery IDs:** Third-party providers include a delivery ID on every request (e.g., `X-GitHub-Delivery`, Stripe's event ID). This ID is included in the queued message envelope.

**Deduplication responsibility:** The receiver does not deduplicate. Deduplication at the receiver requires shared state (a seen-IDs store), which adds latency and a dependency in the hot path. The downstream processor, which already requires idempotency for at-least-once queue delivery, is the correct place to deduplicate. The receiver's job is to get the event onto the queue exactly once per valid inbound request. If the provider retries and sends the same delivery ID twice, both will be enqueued. The downstream processor handles this.

**Exception:** If the queue technology provides native deduplication (SQS FIFO deduplication windows), it should be enabled. This is free deduplication with no receiver-side complexity.

---

### 2.7 Response Contract

**200 OK:** Returned if and only if the signature is valid and the event is successfully enqueued. No body is required. An empty body or `{"ok": true}` is acceptable. The response must be sent before any downstream processing begins — but after the enqueue completes.

**401 Unauthorized:** Signature validation failure (missing header, mismatch, unparseable envelope).

**400 Bad Request:** Malformed request structure before signature validation can be attempted (e.g., missing body entirely).

**415 Unsupported Media Type:** Wrong Content-Type.

**422 Unprocessable Entity:** Valid signature, invalid JSON body.

**503 Service Unavailable:** Queue is unavailable or enqueue failed. The `Retry-After` header should be set if the provider supports it.

**429 Too Many Requests:** Rate limit exceeded (see §3.10).

**No 5xx on signature failure:** A signature failure is never a 5xx. 5xx signals a transient server-side error and invites retry. A signature failure is a permanent client-side authentication failure.

## 3. Operational Layer

### 3.1 Observability: Structured Logging

Every log line is structured JSON. No unstructured log lines in the hot path. Log aggregation systems (Datadog, Splunk, CloudWatch Logs Insights) cannot reliably parse unstructured text under pressure, and under pressure is exactly when these logs will be queried.

**Required fields on every log line:**

| Field | Value |
|---|---|
| `timestamp` | ISO 8601, UTC |
| `level` | `info`, `warn`, `error` |
