## Root Cause

**The deduplication check and the Stripe charge are not atomic. There is a race condition in `process_payment` that allows multiple workers to charge the same payment concurrently.**

Here's the exact sequence that produces a duplicate charge:

```
Worker A (instance 1)          Worker B (instance 2)
─────────────────────          ─────────────────────
redis.get("processed:pay_X")   redis.get("processed:pay_X")
→ None (not set yet)           → None (not set yet)

stripe.Charge.create(...)      stripe.Charge.create(...)
                               ← charge succeeds ($25.00)
← charge succeeds ($25.00)     

db.INSERT payments...          db.INSERT payments...
db.commit()                    ← UniqueViolationError (or succeeds on race)

redis.setex("processed:...")   redis.setex("processed:...")
```

Both workers pass the Redis guard at the same moment because neither has written the key yet. Both call Stripe. The customer is charged twice. The `UniqueViolationError` you see in the logs is the database catching the second insert — but by then **Stripe has already processed both charges**.

---

## Why This Wasn't Caught

The tests mock Redis with a shared in-memory dict, but **they test workers in isolation, sequentially**. There is no test that runs two concurrent calls to `process_payment` with the same `payment_id`. The race condition only manifests under concurrent execution, which the test suite never exercises.

---

## Why It's Happening Now

Two changes deployed 6 days ago created the conditions for this race:

1. **`MaxInstances` increased from 3 → 12.** More worker instances means more concurrent consumers polling the same queue. The probability of two workers picking up the same message simultaneously went up significantly.

2. **`VisibilityTimeout` increased from 30s → 120s.** This was the right instinct — messages were timing out and being redelivered. But it didn't fix the underlying problem. Under load, a slow worker can still hold a message past 120s, SQS redelivers it, and a second worker picks it up while the first is mid-flight between the Redis check and the Redis write.

The promotion traffic spike is what pushed concurrent execution high enough to hit this consistently.

---

## The Fix

### Immediate: Use a Redis distributed lock

Replace the read-then-write guard with an atomic `SET NX` (set if not exists). This is the standard Redis primitive for this pattern.

```python
# worker/payment_worker.py

LOCK_TTL = 300  # 5 minutes — long enough to cover worst-case processing time

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]
    lock_key = f"lock:{payment_id}"
    processed_key = f"processed:{payment_id}"

    # Atomically acquire lock. Returns True only for the first caller.
    # nx=True means SET only if key does Not eXist.
    acquired = redis_client.set(lock_key, "1", nx=True, ex=LOCK_TTL)
    if not acquired:
        logger.info("Payment %s is already being processed or is complete. Skipping.", payment_id)
        return

    try:
        # Re-check: a prior run may have completed successfully
        if redis_client.get(processed_key):
            return

        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Write the permanent processed marker
        redis_client.setex(processed_key, DEDUP_TTL, "1")

    except Exception:
        # Release the lock on failure so a retry can proceed
        redis_client.delete(lock_key)
        raise
```

**Why this works:** `SET NX` is atomic at the Redis level. Only one caller can win it. Every other concurrent worker for the same `payment_id` exits immediately. The lock TTL is a safety valve — if the worker crashes without releasing, the lock expires and a retry can proceed.

---

### Also Required: Add a Stripe Idempotency Key

The lock protects against concurrent workers. Idempotency keys protect against Stripe being called twice if the lock logic itself has a failure mode (e.g., Redis is briefly unavailable). This is a defense-in-depth measure and costs nothing.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this key
)
```

With this in place, even if `stripe.Charge.create` is called twice with the same `payment_id`, Stripe returns the original charge object instead of creating a new one.

---

### Longer Term: Enforce Idempotency at the Database Layer Too

The `UniqueViolationError` you're already seeing means the DB constraint is doing its job. Make the worker handle that case explicitly so a duplicate DB insert is treated as a success rather than an unhandled exception:

```python
try:
    db.execute("INSERT INTO payments ...")
    db.commit()
except UniqueViolationError:
    logger.warning("Payment %s already recorded in DB. Treating as success.", payment_id)
    # Still mark processed in Redis so future checks short-circuit
    redis_client.setex(processed_key, DEDUP_TTL, "1")
    return
```
