# Root Cause: The Idempotency Key Is Set After the Charge, Not Before It

## What Is Actually Happening

You have a classic check-then-act race condition with an unbounded window. Here is the exact sequence that produces a duplicate charge:

```
T=0ms    Request 1 arrives at Gateway → redis.get("processed:pay_001") → None → publishes to SQS
T=50ms   Request 2 arrives at Gateway → redis.get("processed:pay_001") → None → publishes to SQS
          (Redis key still not set — Worker hasn't finished yet)

T=100ms  Worker A dequeues message 1 → redis.get("processed:pay_001") → None → stripe.Charge.create() ← CHARGE 1
T=150ms  Worker B dequeues message 2 → redis.get("processed:pay_001") → None → stripe.Charge.create() ← CHARGE 2

T=200ms  Worker A: db.execute(INSERT) → db.commit() → redis.setex("processed:pay_001")
T=210ms  Worker B: db.execute(INSERT) → UniqueViolationError ← this is the error in your logs
```

The Redis key is not set until after `stripe.Charge.create()`, `db.execute()`, and `db.commit()` all complete. Every request that arrives during that entire window — which at payment processing latencies is measured in hundreds of milliseconds to seconds, not microseconds — passes both the gateway check and the worker check. With `MaxInstances` raised from 3 to 12, you now have 12 workers that can race simultaneously. The promotion traffic made this happen constantly.

## Why the Tests Did Not Catch It

The tests are single-threaded and sequential. They never exercise two requests arriving before the first completes. Look at what the test suite actually proves:

```python
# This test proves the guard works when Redis is ALREADY set before the call.
# It does not prove the guard works when two calls arrive simultaneously.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"   # ← key pre-loaded, race never happens
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()

# This test explicitly confirms the broken ordering.
# It asserts setex is called — but says nothing about WHEN relative to the charge.
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment(...)
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

The test suite has 28 passing tests that validate the happy path in isolation. None of them model concurrency. The CI green light is meaningless for this class of bug.

## The Two Compounding Deployment Changes

The `VisibilityTimeout` increase from 30s to 120s and the `MaxInstances` increase from 3 to 12 did not introduce the bug — the bug was always there. They made it unavoidable:

- **More workers** means more concurrent consumers racing on the same message or on duplicate-published messages
- **Longer visibility timeout** means a message being processed by a slow worker stays invisible longer, but a *second published message* for the same `payment_id` (from a duplicate gateway request) is a completely separate SQS message with its own visibility — the timeout does nothing to prevent that race

## What Needs to Change

### Immediate Fix: Acquire the Lock Before the Queue Boundary

The idempotency key must be set atomically in the gateway, before the message is published, using a Redis SET with NX (set if not exists):

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. Only one caller wins.
    # NX = only set if key does not exist
    # EX = TTL in seconds
    # Returns True if key was set (this caller won), None if key already existed.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,
        ex=DEDUP_TTL,
    )

    if not acquired:
        # Key already exists — either pending or completed. Either way, do not publish.
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    try:
        sqs.send_message(
            QueueUrl=QUEUE_URL,
            MessageBody=json.dumps(data),
        )
    except Exception:
        # If we fail to publish, release the lock so the client can retry.
        redis_client.delete(f"processed:{payment_id}")
        raise

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker then updates the key from `"pending"` to `"1"` after confirmed processing:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Guard against redelivered SQS messages (legitimate redelivery after worker crash).
    # A value of "1" means fully committed. "pending" means in-flight — also skip,
    # because another worker is handling it or the gateway just published it.
    state = redis_client.get(f"processed:{payment_id}")
    if state == "1":
        return
    # Note: if state == "pending", we are the legitimate worker — proceed.
    # If state is None, the gateway lock was lost (Redis restart, etc.) — proceed
    # and let the DB unique constraint be the final safety net.

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Upgrade from "pending" to "1" — payment fully committed.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Defense in Depth: Stripe Idempotency Keys

This should have been there from the start. Stripe supports idempotency keys natively. Even if two workers race through to the Stripe call, Stripe will deduplicate on their side:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # ← Stripe deduplicates on this
)
```

This does not fix the race — you still get two DB inserts racing and one `UniqueViolationError` — but it means the customer is not charged twice even if your application logic fails. This is your last line of defense and it costs one argument.

### The Database Constraint Is Correctly Placed — Keep It

The `UniqueViolationError` you are seeing in the logs is the database constraint doing its job. It is the only thing that prevented every duplicate from becoming a confirmed double charge. Do not remove it. It is not the cause of the incident — it is the evidence of the incident and the thing stopping it from being worse.

## Summary for the Emergency Call

| | Current (broken) | Fixed |
|---|---|---|
| Where lock is acquired | Worker, after Stripe charge | Gateway, before SQS publish |
| Lock mechanism | `GET` then `SETEX` (two operations, not atomic) | `SET NX EX` (single atomic operation) |
| Race window | Entire payment processing time (hundreds of ms to seconds) | Zero — gateway rejects before message enters queue |
| Stripe deduplication | None | Idempotency key on every charge call |
| DB constraint | Present | Present (keep it) |

The fix is two lines in the gateway and one argument to the Stripe call. The architectural rule going forward: **the idempotency lock must be acquired before the message crosses the queue boundary, not after the work completes.**
