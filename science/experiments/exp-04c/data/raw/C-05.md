# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check happens **before** the message crosses the queue boundary. The window between those two events is the entire payment processing time — and under concurrent load, that window is wide open.

---

## What Is Actually Happening

Walk through the sequence for a single `payment_id` under concurrent load:

```
T+0ms    Request 1 arrives at gateway
T+0ms    Gateway checks Redis → key not set → PASS
T+0ms    Gateway publishes message to SQS

T+50ms   Request 2 arrives at gateway (retry, duplicate tab, network hiccup)
T+50ms   Gateway checks Redis → key STILL not set → PASS
T+50ms   Gateway publishes a second message to SQS

T+100ms  Worker A dequeues message 1
T+100ms  Worker A checks Redis → key not set → PASS
T+100ms  Worker A calls stripe.Charge.create() ...

T+110ms  Worker B dequeues message 2
T+110ms  Worker B checks Redis → key STILL not set → PASS
T+110ms  Worker B calls stripe.Charge.create() ...

T+2100ms Worker A Stripe call returns → charge succeeds
T+2100ms Worker A writes to DB → succeeds
T+2100ms Worker A sets Redis key ← TOO LATE

T+2150ms Worker B Stripe call returns → charge succeeds (second charge)
T+2150ms Worker B writes to DB → UniqueViolationError ← this is the log entry you're seeing
T+2150ms Worker B never sets Redis key (exception path)
```

The `UniqueViolationError` in your logs is not the cause — it is the **symptom that surfaces after the second charge has already gone through**. The database constraint is catching the duplicate record, not preventing the duplicate charge.

---

## Why the Tests Pass and Production Burns

The tests are not wrong. They are testing the wrong thing.

```python
# This test is sequential. It proves the guard works when Redis is already set.
# It does not prove the guard works when two workers race to check before either sets.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

There is no test that instantiates two concurrent calls to `process_payment` with the same `payment_id` and verifies only one Stripe charge occurs. The race condition is untestable with the current architecture because the fix requires an atomic operation, not sequential logic.

The two recent infrastructure changes made this dramatically worse:

| Change | Effect |
|--------|--------|
| `VisibilityTimeout` 30s → 120s | Messages that previously timed out and redelivered now stay in-flight longer, meaning two workers can hold the same message simultaneously for up to 2 minutes |
| `MaxInstances` 3 → 12 | Four times as many workers competing to process the same message during the visibility window; the probability of a race goes from low to near-certain under load |

These changes did not introduce the bug. They widened a window that was always there until it became impossible to miss.

---

## The Structural Problem

```
Gateway:  CHECK redis → (no lock held) → publish to SQS
                                          ↑
                              This is where the window opens.
                              It stays open until the worker
                              finishes processing and sets Redis.
                              Under your current load that is
                              2-3 seconds minimum.

Worker:   CHECK redis → stripe.Charge.create() → db.execute() → SET redis
                                                                  ↑
                                                    Window closes here.
```

Any request that arrives at the gateway during that window passes the check. Any redelivered message that a second worker picks up during that window passes the check. The check and the lock are in different places with unbounded time between them.

---

## What Needs to Change

### Fix 1: Atomic reservation before the queue boundary (required)

The idempotency key must be **set at the gateway**, before the message is published, using a Redis atomic operation that fails if the key already exists.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # SET NX is atomic. Only one caller wins. There is no window.
    # If the key already exists, set_nx returns False.
    reserved = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,       # Only set if key does not exist — atomic in Redis
        ex=DEDUP_TTL,
    )

    if not reserved:
        # Key already exists — either processed or in-flight
        current = redis_client.get(f"processed:{payment_id}")
        if current == "done":
            return jsonify({"status": "already_processed", "payment_id": payment_id}), 200
        else:
            # "pending" — a prior request is already in the queue
            return jsonify({"status": "already_accepted", "payment_id": payment_id}), 202

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker then updates the value from `"pending"` to `"done"` after confirmed processing:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Belt-and-suspenders check. The gateway should have set "pending".
    # If it is already "done", a prior worker completed this. Skip.
    current = redis_client.get(f"processed:{payment_id}")
    if current == "done":
        return

    # If "pending" is missing entirely (Redis eviction, cold start),
    # attempt to claim it atomically before proceeding.
    if current is None:
        claimed = redis_client.set(
            f"processed:{payment_id}", "pending", nx=True, ex=DEDUP_TTL
        )
        if not claimed:
            # Another worker just claimed it
            return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Update to "done" only after confirmed DB write
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "done")
```

### Fix 2: Use SQS message deduplication as a second layer (defense in depth)

If this is a Standard queue, migrate to a FIFO queue with `MessageDeduplicationId` set to the `payment_id`. SQS will reject duplicate messages within a 5-minute window at the queue level, before a worker ever sees them.

```python
sqs.send_message(
    QueueUrl=QUEUE_URL,
    MessageBody=json.dumps(data),
    MessageGroupId="payments",
    MessageDeduplicationId=payment_id,  # SQS deduplicates on this
)
```

This does not replace Fix 1. It is a second barrier. The Redis atomic set is still required because the SQS deduplication window is only 5 minutes and does not cover redelivery after that window.

### Fix 3: Make the Stripe call itself idempotent

Stripe supports idempotency keys natively. Pass the `payment_id` as the idempotency key on every charge attempt. If the same key is used twice, Stripe returns the original charge object instead of creating a new charge.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this
)
```

This is your last line of defense. If both Fix 1 and Fix 2 fail, Stripe will not charge the card twice. This should have been in place from day one.

### Fix 4: Add a database-level guard that fires before the charge

Move the `INSERT` to before `stripe.Charge.create()` using a status column, so the database unique constraint becomes a pre-charge guard rather than a post-charge error log entry.

```python
# Attempt to claim the payment row before touching Stripe
try:
    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, status)"
        " VALUES (?, ?, ?, 'pending')",
        (payment_id, data["customer_id"], data["amount"]),
    )
    db.commit()
except UniqueViolationError:
    # Another worker already claimed this payment_id. Do not charge.
    return

# Only reach Stripe if we won the DB insert race
charge = stripe.Charge.create(...)

db.execute(
    "UPDATE payments SET stripe_charge_id = ?, status = 'complete'"
    " WHERE payment_id = ?",
    (charge["id"], payment_id),
)
db.commit()
```
