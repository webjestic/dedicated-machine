## Root Cause

**The idempotency check is not atomic. There is a race condition between the Redis `GET` and the Stripe charge that allows multiple workers to process the same payment concurrently.**

Here is the exact sequence that produces a duplicate charge:

```
Worker A (instance 1)          Worker B (instance 2)
─────────────────────          ─────────────────────
GET processed:pay_001 → None
                               GET processed:pay_001 → None
stripe.Charge.create() ✓
                               stripe.Charge.create() ✓  ← duplicate charge
db.INSERT payments ...
                               db.INSERT payments ...     ← UniqueViolationError
redis.setex processed:pay_001
                               redis.setex processed:pay_001
```

Both workers pass the guard check before either one sets the Redis key. The `UniqueViolationError` you see in the logs is the database constraint catching the second insert — but by then Stripe has already been charged twice. The constraint saves the database record, not the customer's money.

---

## Why This Is Happening Now

The two recent changes combined to make this race condition fire regularly:

1. **`MaxInstances` increased from 3 → 12.** More concurrent workers means more parallel message processing. The race window that existed before is now hit far more often.

2. **`VisibilityTimeout` increased from 30s → 120s.** This was the right instinct — messages were timing out and being redelivered. But it only reduced one source of redelivery. It did nothing to protect against two workers picking up the same message within the visibility window, which SQS does not guarantee against under load.

The underlying bug was always there. The scaling changes made it a production incident.

---

## Why the Tests Missed This

The tests are single-threaded and sequential. `test_process_payment_skips_if_already_processed` validates the guard works when Redis already has the key — but no test exercises two workers racing through the guard simultaneously before either sets the key. The mocked Redis also has no locking semantics, so concurrency cannot be observed even if a test tried.

---

## The Fix

Replace the non-atomic `GET` → `charge` → `SET` sequence with an atomic Redis lock using `SET NX` (set if not exists). This collapses the check-and-reserve into a single atomic operation.

### `worker/payment_worker.py`

```python
def process_payment(data: dict) -> None:
    """Charge the customer and record the transaction."""
    payment_id = data["payment_id"]
    lock_key = f"processing_lock:{payment_id}"
    processed_key = f"processed:{payment_id}"

    # Atomic check: skip if already fully processed
    if redis_client.get(processed_key):
        return

    # Atomic acquire: only one worker proceeds past this point.
    # NX = only set if key does not exist (atomic test-and-set).
    # EX = auto-expire after 300s so a crashed worker doesn't
    #      permanently block reprocessing.
    lock_acquired = redis_client.set(lock_key, "1", nx=True, ex=300)
    if not lock_acquired:
        # Another worker is already processing this payment_id.
        # Leave the message in the queue; SQS will redeliver if
        # the other worker fails before deleting it.
        logger.info("Payment %s already being processed, skipping", payment_id)
        return

    try:
        # Re-check after acquiring the lock. The first worker to
        # acquire the lock on a redelivered message should still skip
        # if a previous run already completed successfully.
        if redis_client.get(processed_key):
            return

        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Mark fully processed. Use a pipeline so both commands are
        # sent in one round-trip and the lock is always released.
        pipe = redis_client.pipeline()
        pipe.setex(processed_key, DEDUP_TTL, "1")
        pipe.delete(lock_key)
        pipe.execute()

    except Exception:
        # Release the lock on failure so the message can be retried.
        redis_client.delete(lock_key)
        raise
```

The critical properties of this fix:

| Property | How it is achieved |
|---|---|
| Only one worker charges Stripe per `payment_id` | `SET NX` is atomic; exactly one caller gets `True` |
| A crashed worker does not block retries forever | Lock has a 300s TTL — tune this to exceed your p99 Stripe latency |
| A redelivered message after a successful run is skipped | The `processed:` key check runs both before and after lock acquisition |
| Lock is always released on failure | `finally`-style `delete` in the `except` block |

## Additional Hardening (After the Incident)

These are not required to stop the bleeding today, but should be scheduled:

**Use Stripe idempotency keys.** Pass `payment_id` as the `idempotency_key` on every `stripe.Charge.create` call. This gives you a second line of defense at the Stripe layer — even if the Redis lock somehow fails, Stripe will deduplicate the charge.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # add this
)
```

**Add a concurrency test.** The test suite has no coverage for the scenario that caused this incident. A test that runs `process_payment` from two threads simultaneously and asserts `stripe.Charge.create` was called exactly once would have caught this.

```python
def test_concurrent_process_payment_charges_stripe_once(mock_redis, mock_stripe):
    import threading
    results = []
    def run():
        process_payment({"payment_id": "pay_race", "amount": 500, "customer_id": "cust_x"})
        results.append(True)

    threads = [threading.Thread(target=run) for _ in range(5)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    assert mock_stripe.Charge.create.call_count == 1
```

## Summary for the Call

> The duplicate charges are caused by a race condition in the payment worker. Multiple worker instances pass the Redis idempotency check simultaneously before any of them sets the "processed" key, so all of them call Stripe. The database constraint catches the duplicate insert but the charge has already happened. The autoscaling increase from 3 to 12 instances made this race fire regularly. The fix is to replace the non-atomic get-then-set with an atomic Redis `SET NX` lock before touching Stripe. We also need to add Stripe idempotency keys as a second layer of protection. The fix is isolated to one function in the worker and is safe to deploy now.
