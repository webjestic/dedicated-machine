## Root Cause

**The idempotency check is not atomic. There is a race condition between the Redis check and the Stripe charge, and the Redis key is only written after the charge succeeds. Under concurrent workers, multiple workers can pass the guard and charge the same customer.**

This was always latent. The two recent changes turned it into a production incident:

- `MaxInstances` 3 → 12 means up to 12 workers can now receive and begin processing the same redelivered message concurrently
- `VisibilityTimeout` 120s means a message that is slow to process stays visible to other workers for longer before being deleted — wider window for the race

---

## The Race Condition, Step by Step

```
Worker 1                          Worker 2
────────────────────────────────────────────────────────
redis.get("processed:pay_001")
→ None (not yet set)
                                  redis.get("processed:pay_001")
                                  → None (not yet set)

stripe.Charge.create(...)         stripe.Charge.create(...)
→ ch_abc (charge succeeds)        → ch_xyz (duplicate charge succeeds)

INSERT INTO payments ...          INSERT INTO payments ...
→ ok                              → UniqueViolationError ← this is the log error

redis.setex("processed:pay_001")
                                  redis.setex("processed:pay_001")
```

This explains both symptoms exactly:

- **Customer charged twice, both recorded:** Worker 1 completes before Worker 2 hits the DB constraint
- **Customer charged twice, second not recorded:** Worker 2 hits `UniqueViolationError` on the INSERT, the exception propagates, the message is not deleted from SQS, and no Redis key is set for that worker's run — but the Stripe charge already went through

The tests pass because they mock Redis with a shared in-memory dict and run single-threaded. Concurrency is never exercised. The race is invisible to the test suite.

---

## What Needs to Change

### Fix 1 — Atomic idempotency gate with `SET NX` (immediate, required)

Replace the read-then-write pattern with an atomic "set if not exists". This is the core fix.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim: only one worker proceeds for this payment_id.
    # SET NX returns True if the key was set (this worker owns it),
    # False if it already existed (another worker beat us here).
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,  # TTL set atomically, no separate EXPIRE call needed
    )
    if not acquired:
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Processing failed after we claimed the key.
        # Release the lock so the message can be retried.
        redis_client.delete(f"processed:{payment_id}")
        raise
```

**Why this works:** `SET NX` is atomic at the Redis level. Only one worker wins the race. The loser returns immediately before touching Stripe.

**The exception handler is not optional.** If you claim the key and then fail before writing to the DB, you must release it. Otherwise the message will be retried, the key will already exist, and the payment will be silently dropped — a missed charge is as bad as a duplicate.

---

### Fix 2 — Stripe idempotency keys (defense in depth, required)

Even with Fix 1, if a worker crashes after charging Stripe but before releasing the lock correctly, you can still get a duplicate on retry. Stripe provides a first-class solution for this.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their end
)
```

Stripe will return the original charge object for any retry with the same `idempotency_key`. This makes the Stripe call itself safe to retry and eliminates the window between the charge and the DB write.

---

### Fix 3 — Handle the `UniqueViolationError` explicitly (stop silent data loss now)

The current code logs the DB constraint error and does not delete the message from SQS. The message will be redelivered. The Stripe charge already exists. This is how customers are getting charged without a record in the system.

```python
# worker/payment_worker.py — in poll_queue

for msg in response.get("Messages", []):
    try:
        process_payment(json.loads(msg["Body"]))
        sqs_client.delete_message(
            QueueUrl=queue_url,
            ReceiptHandle=msg["ReceiptHandle"],
        )
    except UniqueViolationError:
        # Payment is already in the DB. This message is safe to discard.
        logger.warning("Duplicate DB insert for message, discarding: %s", msg["MessageId"])
        sqs_client.delete_message(
            QueueUrl=queue_url,
            ReceiptHandle=msg["ReceiptHandle"],
        )
    except Exception as e:
        logger.error("Failed to process payment message: %s", e)
        # Do not delete — let SQS redeliver for retry
```

## Rollback Option

If deploying fixes is not possible in the next few minutes, the fastest way to reduce the blast radius is to revert `MaxInstances` from 12 back to 3. This does not eliminate the race but narrows the concurrency window significantly and will reduce the rate of new duplicates while the fix is prepared.

## Summary for the Call

| | |
|---|---|
| **Root cause** | Non-atomic check-then-act on the Redis idempotency key. Multiple workers pass the guard concurrently and all call Stripe. |
| **Why it surfaced now** | Worker autoscaling to 12 instances increased concurrency. The race always existed but was rare at 3 instances. |
| **Why tests missed it** | Tests are single-threaded and mock Redis with a shared dict. Concurrent access is never tested. |
| **Immediate mitigation** | Revert `MaxInstances` to 3 to reduce rate of new duplicates. |
| **Fix** | `SET NX` atomic lock in the worker + Stripe idempotency keys + explicit `UniqueViolationError` handling. |
| **Customer impact** | Any `UniqueViolationError` in the logs is a customer who was charged without a DB record. Those need to be reconciled against Stripe manually. |
