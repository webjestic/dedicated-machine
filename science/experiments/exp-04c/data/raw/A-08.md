# Root Cause: Check-Then-Act Race Condition Across a Queue Boundary

## What Is Happening Right Now

Two (or more) HTTP requests for the same `payment_id` arrive at the gateway during the same window. Both hit the Redis check. Neither finds the key. Both publish to SQS. The key does not exist in Redis until a worker finishes processing — which takes however long Stripe takes to respond plus the DB write. Every request that arrives during that window passes the check and enqueues a duplicate message. With 12 worker instances polling the same queue, multiple workers can then dequeue and begin processing those duplicate messages concurrently.

The `UniqueViolationError` you are seeing in the logs is the database constraint catching the *second* charge after it has already hit Stripe. That means some customers were charged twice and the second charge has no database record. That is your worst-case scenario.

---

## The Exact Sequence That Produces a Duplicate Charge

```
T=0ms   Request 1 arrives at gateway. Redis check: key absent. Publishes to SQS.
T=50ms  Request 2 arrives at gateway. Redis check: key still absent. Publishes to SQS.
        (Worker has not finished. Key is not set yet. Both messages are now in the queue.)

T=100ms Worker A dequeues message 1. Redis check in worker: key absent. Calls Stripe.
T=100ms Worker B dequeues message 2. Redis check in worker: key absent. Calls Stripe.
        (Both workers pass the in-worker guard simultaneously for the same reason.)

T=800ms Worker A: Stripe responds. DB insert succeeds. Redis key set.
T=801ms Worker B: Stripe responds. DB insert fails — UniqueViolationError.
        Card has already been charged twice. Second charge has no payment record.
```

The window is not milliseconds. It is the entire Stripe round-trip. Under the Q1 load increase, more concurrent requests arrive during that window. The scale-up to 12 worker instances made the worker-level race worse by increasing the probability that two workers dequeue the duplicate messages simultaneously.

---

## Why the Tests Did Not Catch This

The test suite is synchronous and single-threaded. `test_process_payment_skips_if_already_processed` patches Redis to return `"1"` before the call, which tests the happy-path guard but never exercises the race. No test sends two concurrent requests to the gateway. No test runs two workers against the same queue simultaneously. The tests prove the logic is correct in isolation. They say nothing about concurrent execution.

```python
# This test is not wrong — it just tests the wrong scenario
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"   # Key already exists before the call
    process_payment(...)                # Tests the guard, not the race
    mock_stripe.Charge.create.assert_not_called()
```

## What Needs to Change

### Fix 1: Atomic Reservation in the Gateway (Required, Immediate)

The Redis key must be set **before** the message crosses the queue boundary, not after processing completes. Use `SET NX` (set-if-not-exists) as an atomic reservation. If the reservation fails, the payment is already in flight and the request is rejected without publishing.

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic reservation. SET NX returns True only for the first caller.
    # All concurrent requests for the same payment_id fail here.
    reserved = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,  # TTL prevents orphaned keys if the worker crashes
    )

    if not reserved:
        # Another request already claimed this payment_id
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

This is atomic at the Redis level. Two concurrent requests cannot both get `True` from `SET NX`. One wins; the other is rejected before a message is ever published.

### Fix 2: Stripe Idempotency Keys (Required, Defense-in-Depth)

Even with Fix 1 in place, SQS delivers messages at-least-once. A worker crash after the Stripe call but before the DB write will cause the message to be redelivered. Pass `payment_id` as the Stripe idempotency key so that a redelivered message produces the same charge object, not a new charge.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this key
)
```

This is free, it is what the feature exists for, and it should have been there from the start.

### Fix 3: Worker Guard Becomes a Distributed Lock (Required)

The in-worker Redis check is still a check-then-act gap. Replace it with `SET NX` as well, so the first worker to pick up a message wins the lock and all others skip.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic lock acquisition. If the gateway set processing:{payment_id},
    # this key already exists and the worker skips. If somehow two workers
    # dequeue the same message, only one acquires the lock.
    acquired = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,
        ex=DEDUP_TTL,
    )

    if not acquired:
        logger.info("Payment %s already in flight or processed, skipping", payment_id)
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
        idempotency_key=payment_id,
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Overwrite with permanent processed marker
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Fix 4: SQS Message Deduplication (Defense-in-Depth)

If the queue is FIFO-capable, enable content-based deduplication or pass `MessageDeduplicationId=payment_id`. This prevents SQS itself from delivering the same logical payment twice within the 5-minute deduplication window. This does not replace the above fixes but adds another layer.

## What to Say on the Call in 3 Minutes

> The root cause is a check-then-act race. The gateway checks Redis and then publishes. The key is not set until the worker finishes. Any duplicate request that arrives before the worker completes passes the check and publishes a second message. Under concurrent load with 12 workers, two workers can dequeue those messages simultaneously and both charge Stripe before either sets the key. The database constraint catches some of these, but not before the second Stripe charge goes through.
>
> The fix is to make the Redis write atomic and move it to before the message is published, using SET NX in the gateway. We also need Stripe idempotency keys, which we are missing entirely.
>
> The scale-up to 12 workers and the VisibilityTimeout increase both made this worse by widening the window and increasing the probability of concurrent worker execution. Neither change introduced the bug. The bug was always there.

## Summary Table

| Layer | Current Behavior | Required Behavior |
|---|---|---|
| Gateway Redis check | `GET` then publish (two operations, not atomic) | `SET NX` before publish (atomic, one operation) |
| Worker Redis check | `GET` then charge (same race, smaller window) | `SET NX` before charge |
| Stripe call | No idempotency key | `idempotency_key=payment_id` |
| SQS | Standard queue, no deduplication | FIFO with `MessageDeduplicationId` or content-based dedup |
| Tests | Single-threaded, no concurrency coverage | Concurrent request tests required |
