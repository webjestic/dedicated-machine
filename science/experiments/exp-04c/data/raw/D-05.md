## Root Cause

**The idempotency check and the Stripe charge are not atomic.** There is a race condition window between the Redis `GET` and the Redis `SETEX` where multiple workers can simultaneously pass the deduplication check and each independently charge Stripe.

Here's the exact sequence that produces a duplicate charge:

```
Worker 1 (Instance A):  GET processed:pay_001  → nil   ✓ proceed
Worker 2 (Instance B):  GET processed:pay_001  → nil   ✓ proceed   ← wins the race
Worker 1:               stripe.Charge.create()          → ch_aaa
Worker 2:               stripe.Charge.create()          → ch_bbb    ← duplicate charge
Worker 1:               INSERT INTO payments ...         → ok
Worker 2:               INSERT INTO payments ...         → UniqueViolationError ← logged
Worker 1:               SETEX processed:pay_001
Worker 2:               (exception, SETEX never reached)
```

This is why you're seeing both symptoms:
- **Customer charged twice, one charge missing from DB:** Worker 2 hits Stripe successfully, then fails on the `INSERT` due to the unique constraint, rolls back, never sets the Redis key, and the exception is swallowed in `poll_queue`. The charge exists in Stripe but not in your system.
- **`UniqueViolationError` in logs:** That's the DB constraint catching the second insert — it's the only guard that actually fired.

### Why it's happening *now* specifically

The autoscale increase from 3 → 12 worker instances multiplied the concurrency surface by 4x. The `VisibilityTimeout` increase from 30s → 120s was the right call for slow payments, but it means redelivered messages stay in flight longer, extending the window where two workers can hold the same message. Under the Q1 promotion load, the race is being hit regularly.

### Why the tests didn't catch it

The tests mock Redis with a single shared in-memory dict and run single-threaded. There is no concurrency in the test suite, so the race window never opens. All 28 tests pass cleanly against a bug that only manifests under concurrent execution.

## What Needs to Change

### Fix 1 — Atomic distributed lock (stops the bleeding)

Replace the non-atomic `GET` → work → `SETEX` pattern with a `SET NX` lock *before* touching Stripe. This is the core fix.

```python
# worker/payment_worker.py

LOCK_TTL = 300  # 5 minutes — must exceed max Stripe response time
DEDUP_TTL = 86400

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]
    lock_key = f"lock:{payment_id}"
    processed_key = f"processed:{payment_id}"

    # Atomically acquire lock. SET NX is a single round-trip — no race window.
    acquired = redis_client.set(lock_key, "1", nx=True, ex=LOCK_TTL)
    if not acquired:
        # Another worker is processing or has processed this payment.
        logger.info("Lock not acquired for %s, skipping", payment_id)
        return

    try:
        # Re-check processed state after acquiring the lock.
        # Handles the case where a prior worker completed and released.
        if redis_client.get(processed_key):
            return

        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Persist completion state. This survives beyond the lock TTL.
        redis_client.setex(processed_key, DEDUP_TTL, "1")

    finally:
        # Always release the lock, success or failure.
        redis_client.delete(lock_key)
```

**Why `SET NX` works:** It is a single atomic Redis operation. Only one caller gets `True`. Every other concurrent caller gets `False` and exits immediately. The race window is eliminated.

### Fix 2 — Handle the orphaned Stripe charges (data integrity)

The current exception handler silently drops failures. A worker that charged Stripe but failed to write the DB will leave an unrecorded charge. Add a reconciliation path:

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]
    # ... lock acquisition as above ...

    charge = None
    try:
        charge = stripe.Charge.create(...)

        db.execute("INSERT INTO payments ...", (..., charge["id"]))
        db.commit()
        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")

    except Exception:
        if charge is not None:
            # We billed the customer but failed to record it.
            # Log everything needed for manual reconciliation or a retry.
            logger.critical(
                "UNRECORDED_CHARGE payment_id=%s stripe_charge_id=%s — "
                "manual reconciliation required",
                payment_id,
                charge["id"],
            )
        raise  # Re-raise so the message is not deleted and SQS retries
    finally:
        redis_client.delete(lock_key)
```

### Fix 3 — Don't delete the SQS message on unhandled exceptions

The current `poll_queue` deletes the message *only* on success, which is correct. But the exception is logged and the loop continues, so the message will redeliver after the `VisibilityTimeout`. That's the right behavior — but make the intent explicit and ensure the lock TTL is shorter than the visibility timeout so a redelivered message can be picked up:

```
LOCK_TTL (300s) < VisibilityTimeout (120s)
```

Wait — with the current `VisibilityTimeout` of 120s and a `LOCK_TTL` of 300s, a failed worker holds the lock past when SQS redelivers the message. Set `LOCK_TTL` to something like 90s (under the 120s visibility timeout) so a redelivered message can acquire the lock after a genuine failure. Tune this to your observed p99 Stripe latency.
