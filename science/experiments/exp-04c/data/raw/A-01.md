# Root Cause: Check-Then-Act Race Condition With No Atomic Lock

## The Short Answer

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is the entire payment processing time — Stripe round-trip plus DB write, typically 1–3 seconds, sometimes longer. Any duplicate request that arrives during that window passes the check clean and gets a second charge issued.

This is not a bug that shows up in tests. It is a concurrency property that only manifests under load. Your 28 passing tests are testing the wrong thing.

---

## What Is Actually Happening in Production

Walk through the exact sequence that produced your 23 tickets:

```
T+0ms    Request 1 arrives at Gateway. Redis check: key absent. Message published to SQS.
T+50ms   Request 2 arrives at Gateway (retry, network blip, or client double-tap).
         Redis check: key STILL absent — Worker hasn't finished yet.
         Message published to SQS. Two messages now in queue.

T+100ms  Worker A dequeues Message 1. Redis check: absent. Proceeds.
T+110ms  Worker B dequeues Message 2. Redis check: absent. Proceeds.
         (12 workers now running after the MaxInstances increase — this is routine)

T+1200ms Worker A: Stripe charge completes → ch_abc123
T+1210ms Worker B: Stripe charge completes → ch_def456   ← DUPLICATE CHARGE
T+1220ms Worker A: INSERT INTO payments ... → succeeds
T+1230ms Worker B: INSERT INTO payments ... → UniqueViolationError ← this is your log line
T+1240ms Worker A: redis.setex("processed:pay_001") ← too late for both
```

The `UniqueViolationError` in your logs is the **symptom**, not the root cause. The DB constraint caught the second write. The Stripe charge already went through before that constraint fired.

The customers who were charged twice: both workers succeeded at Stripe before either hit the DB. The customers who show a failed second charge with no payment recorded: one worker won the DB insert, the other got the `UniqueViolationError`, the exception handler logged it and moved on — but the Stripe charge for the losing worker already happened and is sitting uncaptured in your Stripe dashboard.

## Why the Recent Changes Made This Worse

Both changes you deployed 6 days ago directly widened the race window:

**`VisibilityTimeout` 30s → 120s**
This was the right fix for the timeout-redelivery problem, but it means a message that is being processed stays invisible for up to 120 seconds. Any *new* request for the same payment arriving during that window hits the gateway, passes the Redis check (key still not set), and publishes a second message. You traded one problem for a larger one.

**`MaxInstances` 3 → 12**
More workers means more concurrent consumers. The probability that two messages for the same payment are dequeued simultaneously went from low to near-certain under the Q1 promotion load. With 3 workers the race was possible. With 12 it is routine.

## Why the Tests Don't Catch This

The test suite is synchronous and single-threaded. Look at what the tests actually validate:

```python
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", "amount": 1200, "customer_id": "cust_pqr"})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

This test **asserts the broken behavior is present** and calls it correct. It verifies that `setex` is called after processing. That is exactly the bug. The test suite has been confirming the race condition is intact on every CI run.

There is no test that calls `process_payment` concurrently. There is no test that calls the gateway twice before the worker finishes. The concurrency window is untestable with the current architecture because the check and the act are in different services with a queue between them.

## The Fix

### Immediate Mitigation (do this now, before the call)

Set the Redis key **in the gateway, before publishing**, using `SET NX` (set-if-not-exists). This is atomic. Either you set it or you don't. There is no window.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim. If key already exists, SET NX returns False.
    # Use a shorter TTL here — this covers the processing window.
    # The worker will extend/confirm it on completion.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist — atomic
        ex=300,        # 5 minutes; covers worst-case processing time
    )

    if not claimed:
        # Either currently processing or already processed
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker keeps its own Redis check as a second line of defense against SQS redeliveries, and sets the durable `processed:` key on completion:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Second guard: catches SQS redeliveries after VisibilityTimeout
    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Durable completion marker. Overwrites the processing key.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Why `SET NX` Fixes It

`SET NX` is a single atomic Redis command. There is no gap between the check and the write. Two concurrent gateway requests for the same `payment_id` race to call `SET NX`. One wins and gets `True`. The other gets `False` and returns immediately. Only one message enters the queue. The race window is eliminated at the entry point.

```
T+0ms    Request 1: SET NX processing:pay_001 → True  → publish → 202
T+50ms   Request 2: SET NX processing:pay_001 → False → return  → 200
         Message never enters queue. Worker never sees it.
```

### What Else Needs to Happen (not today, but before next promotion)

**Stripe idempotency keys.** Pass `payment_id` as the Stripe idempotency key on every charge call. This is your last-resort backstop. If a duplicate ever reaches Stripe, Stripe deduplicates it server-side and returns the original charge object instead of creating a new one. This does not replace the Redis fix — it is defense in depth.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # add this
)
```

**Handle the `UniqueViolationError` correctly in the worker.** Right now a DB constraint violation causes the exception handler to log and continue. The message gets deleted from SQS. The Stripe charge that caused the violation is orphaned — money moved, no record in your system. The worker needs to catch `UniqueViolationError` specifically, check whether the existing DB row has a different `stripe_charge_id`, and if so, issue a Stripe refund before deleting the message.

**Fix the test that asserts the broken behavior.** `test_process_payment_sets_redis_key_after_processing` should be replaced with a test that verifies the gateway's `SET NX` call happens before `send_message`, and a concurrent integration test that fires two simultaneous gateway requests and asserts exactly one SQS message is published.

## Summary for the Call

| | Current (broken) | Fixed |
|---|---|---|
| **Where lock is set** | Worker, after Stripe charge | Gateway, before queue publish |
| **Lock mechanism** | `GET` then `SET` (two operations) | `SET NX` (one atomic operation) |
| **Race window** | Entire payment processing time (1–3s+) | Zero — atomic |
| **Concurrent requests during window** | Both pass, both charge | Second request rejected immediately |
| **SQS redelivery protection** | Worker Redis check (kept as second guard) | Worker Redis check (unchanged) |
| **Stripe-level protection** | None | Idempotency key (add this week) |

The lock has to be acquired before the message crosses the queue boundary. Right now it is acquired after. That is the root cause.
