## Root Cause

**The idempotency check is not atomic. There is a race condition between the Redis check and the Stripe charge, and the Redis key is only written after the charge succeeds. Under concurrent execution, multiple workers can pass the guard and charge the same customer.**

---

### What's Actually Happening

Walk through the timeline with two workers both picking up the same redelivered message — which is now far more likely because you scaled to 12 worker instances:

```
Worker 1:  redis.get("processed:pay_001")  → None   ✓ proceed
Worker 2:  redis.get("processed:pay_001")  → None   ✓ proceed   (race)

Worker 1:  stripe.Charge.create(...)       → ch_aaa  (first charge)
Worker 2:  stripe.Charge.create(...)       → ch_bbb  (duplicate charge)

Worker 1:  INSERT INTO payments ...        → OK
Worker 2:  INSERT INTO payments ...        → UniqueViolationError ← the log you're seeing

Worker 1:  redis.setex("processed:pay_001", ...)
Worker 2:  (exception — Redis key never set for this path)
```

This explains both symptoms exactly:
- **Customer charged twice** — both workers hit Stripe before either sets the Redis key
- **UniqueViolationError with no payment recorded** — Worker 2's Stripe charge succeeded, but the DB constraint fires, the exception is caught by the outer `poll_queue` handler, and the message is *not* deleted from SQS. The charge exists in Stripe but not in your database.

The two recent changes made this dramatically worse, but neither is the root cause:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | Reduced *some* redeliveries, but doesn't eliminate concurrent processing across 12 instances |
| `MaxInstances` 3 → 12 | 4× more workers competing for the same messages during the promotion traffic spike |

The tests pass because they test each worker in isolation. No test exercises two concurrent calls to `process_payment` with the same `payment_id`.

---

### The Broken Code

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # ❌ CHECK and SET are not atomic — window for concurrent workers to both pass
    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(...)   # ❌ Stripe called before the lock is held

    db.execute("INSERT INTO payments ...")
    db.commit()

    # ❌ Redis key written last — if anything above fails, key is never set
    #    and the message will be reprocessed
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

---

### The Fix

Replace the non-atomic GET/SET with a Redis `SET NX` (set-if-not-exists). This is a single atomic operation — only one worker can win it, and the rest bail out before touching Stripe.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim — only one worker wins this across all instances
    # SET key value EX ttl NX  →  returns True if key was set, None if it already existed
    lock_acquired = redis_client.set(
        f"processed:{payment_id}",
        "1",
        ex=DEDUP_TTL,
        nx=True,
    )
    if not lock_acquired:
        logger.info("Payment %s already claimed or processed, skipping", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Release the lock so the message can be retried by another worker
        redis_client.delete(f"processed:{payment_id}")
        raise
```

Two things changed and both matter:

1. **`SET NX` instead of GET + SET** — atomic claim, no race window
2. **Delete the key on failure** — if the Stripe call or DB write fails, the lock is released so the message can be retried. Previously a failure after `setex` would silently drop the payment forever; a failure before it would allow duplicate charges.

### Gateway Also Needs the Same Fix

The gateway has the same non-atomic pattern. It's less likely to cause duplicates today because it's a single check before enqueue rather than before a charge, but it should be consistent:

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Same atomic check as the worker
    if not redis_client.set(f"queued:{payment_id}", "1", ex=DEDUP_TTL, nx=True):
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(QueueUrl=QUEUE_URL, MessageBody=json.dumps(data))
    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

Note the separate `queued:` key prefix — the gateway dedup and the worker dedup serve different purposes and shouldn't share a key.

---

### Immediate Remediation Steps

1. **Deploy the `SET NX` fix to the worker now** — this stops new duplicates
2. **Scale `MaxInstances` back to 3** until the fix is deployed — reduces the concurrency window immediately
3. **Audit Stripe charges against your DB** for the 6-hour window — find every `stripe_charge_id` that has no corresponding row in `payments`. Those are the silent failures: real charges with no record. Refund or reconcile them manually.
4. **Add a Stripe idempotency key** as a second line of defense — pass `idempotency_key=payment_id` to `stripe.Charge.create()`. Stripe will deduplicate on their side even if your lock fails.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this
)
```
