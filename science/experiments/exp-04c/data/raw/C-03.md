# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check happens **before** the message crosses the queue boundary. Between those two events is the entire payment processing time — Stripe round-trip, database write, everything. Any duplicate request that arrives during that window passes the check clean and gets enqueued. With 12 workers now consuming the queue concurrently, the same message can also be picked up by two workers simultaneously, both passing the in-worker Redis check for the same reason.

You have the race condition in two places. Either one is sufficient to cause a duplicate charge. Together they are reliable.

---

## The Race Condition, Precisely

### Race 1: Gateway → Queue (the primary driver)

```
T+0ms    Request 1 arrives → redis.get("processed:pay_001") → None → enqueue ✓
T+50ms   Request 2 arrives → redis.get("processed:pay_001") → None → enqueue ✓
         (Worker hasn't finished yet. Redis key doesn't exist yet.)
T+2000ms Worker finishes → redis.setex("processed:pay_001", ...) ← too late
```

The Redis check in the gateway is checking for a key that won't exist until the worker finishes processing. The window is not milliseconds. It is however long Stripe takes plus however long your database takes. Under load, that is seconds. Any retry, any duplicate submission, any client that fires twice — all pass the check.

### Race 2: Worker → Worker (amplified by the scaling change)

```
T+0ms    Worker A receives message, redis.get("processed:pay_001") → None
T+0ms    Worker B receives same message, redis.get("processed:pay_001") → None
T+2000ms Worker A charges Stripe, writes DB, sets Redis key
T+2000ms Worker B charges Stripe → UniqueViolationError on DB insert
         But the Stripe charge already went through.
```

This is exactly what the `UniqueViolationError` in your logs is showing. Worker B hit Stripe successfully, then failed on the database insert. The customer was charged. The payment was not recorded. That is the "charged twice with no record" tickets in your support queue.

The `VisibilityTimeout` increase from 30s to 120s was the right call for the timeout problem, but it did not fix the underlying race — it just changed the timing. The `MaxInstances` increase from 3 to 12 made the worker-level race four times more likely to produce concurrent consumers on the same message.

---

## Why the Tests Don't Catch This

The tests are testing sequential execution. Look at `test_process_payment_skips_if_already_processed`:

```python
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"  # Key already exists before the call
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

This test only validates the happy path of the guard — when the key is already set. It never tests two concurrent calls where neither has set the key yet. There is no test that calls `process_payment` twice simultaneously with the same `payment_id`. The 28 passing tests are testing a single-threaded world. The bug lives in the gap between two threads.

The same problem exists for the gateway tests. `test_charge_returns_200_for_already_processed` pre-loads the Redis mock with `"1"`. It does not test two simultaneous requests racing through the gateway before the worker sets the key.

---

## What Needs to Change

### Fix 1: Set the lock before enqueuing, not after processing

The gateway must claim the idempotency key atomically before the message crosses the queue boundary. Use `SET NX` (set if not exists) as the gate. If the key is already claimed, return early. If the claim succeeds, enqueue.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. NX = only set if key does not exist.
    # This is the gate. If it returns None, someone else already claimed it.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Already claimed or already processed — check which
        if redis_client.get(f"processed:{payment_id}"):
            return jsonify({"status": "already_processed", "payment_id": payment_id}), 200
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 202

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` is atomic at the Redis level. Two concurrent requests racing through the gateway will both attempt the set. One will get `True`. One will get `None`. The one that gets `None` never enqueues. This eliminates Race 1.

### Fix 2: Use a distributed lock in the worker for redelivery protection

The worker-level guard needs the same atomic claim pattern, not a check-then-act:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim. If another worker is already processing this payment_id,
    # this returns None and we exit immediately.
    lock_key = f"lock:{payment_id}"
    claimed = redis_client.set(lock_key, "1", nx=True, ex=300)  # 5 min TTL

    if not claimed:
        logger.info("Payment %s already being processed, skipping", payment_id)
        return

    try:
        # Check completed state after acquiring lock
        if redis_client.get(f"processed:{payment_id}"):
            return

        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")

    finally:
        redis_client.delete(lock_key)
```

### Fix 3: Handle the Stripe-charged-but-DB-failed case

This is the silent money leak. Stripe was charged. The DB insert failed. The customer has no record. You need to handle this explicitly:

```python
        try:
            db.execute(
                "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
                " VALUES (?, ?, ?, ?)",
                (payment_id, data["customer_id"], data["amount"], charge["id"]),
            )
            db.commit()
        except UniqueViolationError:
            # Another worker already recorded this. The charge that just went through
            # on Stripe needs to be refunded immediately.
            logger.error(
                "Duplicate charge detected for payment_id=%s stripe_charge_id=%s — "
                "issuing refund",
                payment_id,
                charge["id"],
            )
            stripe.Refund.create(charge=charge["id"])
            raise
```

### Fix 4: Use Stripe's own idempotency keys

Stripe supports idempotency keys natively. Pass your `payment_id` as the idempotency key on every charge call. If the same key is used twice, Stripe returns the original charge object instead of creating a new one. This is a defense-in-depth layer, not a replacement for the above fixes:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their end
)
```

## Summary of Changes Required

| Location | Current behavior | Required behavior |
|---|---|---|
| Gateway Redis check | `GET` after the fact | `SET NX` before enqueue |
| Worker Redis check | `GET` then act | `SET NX` as distributed lock |
| Stripe call | No idempotency key | Pass `payment_id` as `idempotency_key` |
| DB insert failure | Exception propagates, charge unrecorded | Catch, refund, re-raise |
| Tests | Sequential only | Add concurrent execution tests |

---

## The Underlying Principle

The check and the act must be atomic, or the check is decorative. In this codebase the check is in the gateway and the act is in the worker. Nothing can make those two atomic across a queue boundary. The only correct solution is to move the lock acquisition to before the message is enqueued, so that the act of enqueuing *is* the claim. The worker lock is a second layer for redelivery protection, not a substitute for fixing the gateway.
