# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check happens **before** the message crosses the queue boundary. Between those two events is the entire payment processing time — Stripe API call, database write, network latency. Any duplicate request that arrives during that window passes the check clean and gets enqueued. With 12 worker instances and a 120-second visibility timeout, that window is now enormous.

---

## The Race Condition, Step by Step

```
Timeline (two concurrent POST /payments/charge for pay_001):

T+0ms    Request A arrives → Redis check → key not set → enqueued ✓
T+50ms   Request B arrives → Redis check → key STILL not set → enqueued ✓
         (Worker hasn't finished. Redis key doesn't exist yet.)

T+100ms  Worker 1 picks up message A → Stripe charge → DB write → Redis set
T+150ms  Worker 2 picks up message B → Redis check → key not set yet → Stripe charge → DUPLICATE
```

The Redis check in the gateway is not wrong — it correctly blocks *already-processed* payments. But it provides **zero protection** against concurrent in-flight requests. The key only exists after the worker finishes. The gateway check and the worker's write are separated by unbounded processing time with no lock held across that gap.

---

## Why the Tests Don't Catch This

The test suite is testing the right behaviors in the wrong order. Look at the mock setup:

```python
# tests/test_payment_flow.py

@pytest.fixture
def mock_redis(monkeypatch):
    cache = {}
    mock = MagicMock()
    mock.get.side_effect = lambda k: cache.get(k)
    mock.setex.side_effect = lambda k, ttl, v: cache.update({k: v})
    monkeypatch.setattr("gateway.charge_handler.redis_client", mock)
    monkeypatch.setattr("worker.payment_worker.redis_client", mock)
    return mock
```

The gateway and worker share a single in-process mock. Every test runs **synchronously and serially**. The gateway enqueues, the worker processes, Redis is updated — all in sequence, no concurrency, no queue latency. The race window literally cannot open in this test environment.

The test that looks most relevant:

```python
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"  # Key already set
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

This tests the *steady-state* deduplication path — a redelivered message arriving after the key is set. It does not test two requests arriving before the key is set. That scenario requires concurrent execution, which no test here exercises. All 28 tests pass because the bug only manifests under concurrent load.

---

## Why the Recent Changes Made This Worse

Both operational changes directly expanded the race window:

**`VisibilityTimeout` increased from 30s → 120s**

When a message is in-flight, SQS hides it from other consumers for the visibility timeout duration. Increasing this timeout means a slow payment can hold the window open for up to 2 minutes before SQS redelivers. But it also means the gap between "message enqueued" and "Redis key set" is now up to 120 seconds — 4x longer than before.

**Worker `MaxInstances` increased from 3 → 12**

More workers means more concurrent message processing. Before: 3 workers, limited parallelism, lower probability of two workers racing on the same payment. Now: 12 workers, 4x higher throughput, 4x more concurrent processing. The probability of two in-flight messages for the same `payment_id` being processed simultaneously increased substantially.

The `UniqueViolationError` you're seeing in the logs is the database constraint catching the second insert — which means Stripe has already been charged twice. The constraint is working as a last-resort safety net, not as prevention.

---

## What Needs to Change

### Immediate Fix: Acquire the Lock Before Publishing

The lock must be held **before the message crosses the queue boundary**. Use a Redis `SET NX` (set-if-not-exists) as an atomic claim in the gateway:

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. SET NX returns True only if the key
    # did not exist. If it returns False, another request already claimed it.
    # TTL must exceed the maximum possible end-to-end processing time.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=300,        # 5 minutes — must exceed max worker processing time
    )

    if not claimed:
        # Key already exists: either being processed or already done
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker then updates the key on completion:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Belt-and-suspenders check for redelivered messages
    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Mark permanently processed and release the processing lock
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
    redis_client.delete(f"processing:{payment_id}")
```

The critical property: `SET NX` is atomic. Two concurrent gateway requests racing on the same `payment_id` cannot both return `True`. One wins, one loses. The message is published exactly once.

### Secondary Fix: Use Stripe's Built-in Idempotency Keys

This is defense in depth. Stripe supports idempotency keys natively:

```python
# worker/payment_worker.py

charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their end
)
```

If a duplicate charge request reaches Stripe, Stripe returns the original charge object rather than creating a new one. This does not fix the root cause — you still get two DB inserts and two `UniqueViolationError`s — but it prevents the customer from being charged twice while you're remediating.

### Tertiary Fix: Make the DB Insert Idempotent

The `UniqueViolationError` is currently crashing the worker and leaving the message undeleted, which causes redelivery. Change the insert to handle conflicts gracefully:

```python
# worker/payment_worker.py — for PostgreSQL

db.execute(
    "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
    " VALUES (?, ?, ?, ?)"
    " ON CONFLICT (payment_id) DO NOTHING",
    (payment_id, data["customer_id"], data["amount"], charge["id"]),
)
db.commit()
```

This turns a crash into a silent no-op for duplicate inserts. The message gets deleted. The worker moves on.

---

## Defense in Depth: All Three Layers

| Layer | Mechanism | What it prevents |
|---|---|---|
| Gateway | Redis `SET NX` before publish | Two messages ever entering the queue for the same `payment_id` |
| Stripe | Idempotency key on charge creation | Customer charged twice even if two messages somehow reach workers |
| Database | `ON CONFLICT DO NOTHING` | Worker crash and message redelivery loop on duplicate insert |

No single layer should be the only line of defense. The gateway lock is the fix. The Stripe idempotency key and the DB upsert are the safety nets for when something unexpected gets through.

## What to Test That Currently Isn't Tested

The test suite needs concurrent execution tests. The existing tests are all sequential and will never catch this class of bug:

```python
import threading

def test_concurrent_requests_publish_exactly_one_message(client, mock_redis, mock_sqs):
    """Two simultaneous requests for the same payment_id must result in
    exactly one message published to the queue."""
    _, sent = mock_sqs
    results = []

    def post():
        rv = client.post(
            "/payments/charge",
            json={"payment_id": "pay_concurrent", "amount": 1000, "customer_id": "cust_x"},
        )
        results.append(rv.status_code)

    threads = [threading.Thread(target=post) for _ in range(5)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # Exactly one message must have been published
    assert len(sent) == 1
    # The rest must have been rejected as duplicates
    assert results.count(200) == 4
    assert results.count(202) == 1
```

This test would have caught the bug before it reached production. Add it to the suite and run it as part of CI.
