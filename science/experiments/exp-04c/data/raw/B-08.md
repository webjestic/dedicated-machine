# Root Cause Analysis — Duplicate Charges

## The Structural Cause

**The idempotency lock is set after the charge completes. The check-then-act window is not milliseconds — it is the entire Stripe API call plus the database write. Any duplicate request that arrives during that window passes the check and charges the card.**

This is the same class of failure every time. The guard looks correct in isolation. It fails under concurrent load because the check and the lock are not atomic, and the processing time between them is unbounded.

---

## The Failure Path

Here is exactly how a duplicate charge happens in this system:

```
Time →

Request 1 arrives at Gateway
  └─ redis.get("processed:pay_001") → nil          ← check passes
  └─ sqs.send_message(pay_001)                      ← message 1 enqueued

Request 2 arrives at Gateway (retry, or duplicate POST)
  └─ redis.get("processed:pay_001") → nil          ← check passes (key not set yet)
  └─ sqs.send_message(pay_001)                      ← message 2 enqueued

Worker A picks up message 1
  └─ redis.get("processed:pay_001") → nil          ← check passes
  └─ stripe.Charge.create(...)                      ← CHARGE 1 hits Stripe
  └─ db.execute(INSERT ...)
  └─ redis.setex("processed:pay_001", ...)          ← lock set HERE, too late

Worker B picks up message 2 (or same message redelivered)
  └─ redis.get("processed:pay_001") → nil          ← check passes (lock not set yet)
  └─ stripe.Charge.create(...)                      ← CHARGE 2 hits Stripe
  └─ db.execute(INSERT ...) → UniqueViolationError  ← this is the error you're seeing
```

The `UniqueViolationError` in your logs is the database constraint catching the second insert *after the second Stripe charge has already succeeded*. The database is not preventing the duplicate charge — it is only recording that one happened.

---

## Why the Recent Changes Made It Worse

The two deployment changes six days ago did not introduce the bug. They widened the window and increased the concurrency, which is why a latent structural flaw became a production incident.

**VisibilityTimeout 30s → 120s:** A message that takes longer than the visibility timeout becomes visible again and gets picked up by a second worker. Previously at 30s, slow payments would re-enter the queue and produce duplicates. Increasing to 120s was the right instinct — it reduced that specific redelivery path — but it did not close the fundamental race.

**MaxInstances 3 → 12:** Four times as many workers polling the same queue means four times the probability that two workers receive overlapping messages during a traffic spike, and four times the throughput of concurrent `process_payment` calls that can all pass the Redis check before any of them sets the lock.

The promotion traffic spike is what forced the race condition to manifest at scale. The bug existed before these changes.

---

## Why the Tests Don't Catch This

Every test in `test_payment_flow.py` is single-threaded and sequential. The mock Redis is a plain dict with no locking. There is no test that fires two concurrent requests at the gateway before the worker finishes. There is no test that delivers the same SQS message to two workers simultaneously.

The test `test_process_payment_sets_redis_key_after_processing` actually *asserts the broken behavior* — it confirms that `setex` is called after processing, which is precisely the ordering that causes the incident.

```python
# This test documents the bug, not the fix
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", "amount": 1200, "customer_id": "cust_pqr"})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

28 tests passing is not signal here. The failure mode is a concurrency race that the test suite has no coverage for.

## What Needs to Change

### Immediate Fix — Atomic Lock Before the Queue Boundary

The lock must be acquired in the gateway, atomically, before the message is published. Use `SET NX` (set if not exists) as an atomic test-and-set. If the lock is already held, reject or return early. Do not check and then set separately.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic: set the key only if it does not already exist.
    # This is the lock acquisition. If it returns False, another request
    # already claimed this payment_id. Do not publish.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not acquired:
        # Key already exists — either pending or completed
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker then updates the key value from `"pending"` to `"complete"` after processing, which lets you distinguish a payment in-flight from one that finished. The gateway check on re-entry sees a key present regardless of state and rejects.

```python
# worker/payment_worker.py — process_payment

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Belt-and-suspenders: if the key is already "complete", skip.
    # "pending" means we own this message and should process it.
    state = redis_client.get(f"processed:{payment_id}")
    if state == "complete":
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Transition from pending → complete
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "complete")
```

### The Database Constraint Is Your Last Line of Defense — Keep It

The `UNIQUE` constraint on `payments.payment_id` is the only thing currently preventing a silent double-charge from going undetected. Do not remove it. It is not the fix — the `UniqueViolationError` fires after Stripe has already charged — but it is the audit signal that tells you a duplicate charge occurred. Without it you would not have 23 support tickets, you would have 23 customers silently charged twice.

### Stripe Idempotency Keys

Pass `idempotency_key` on every Stripe API call. Stripe will deduplicate on their side for calls with the same key within 24 hours. This is a defense-in-depth layer, not a substitute for fixing the lock.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # Stripe deduplicates on this
)
```

### Add Concurrency Tests

The test suite needs at least one test that fires two simultaneous gateway requests for the same `payment_id` and asserts that exactly one Stripe charge is created. Use `threading.Thread` or `concurrent.futures`. A sequential mock will not find this class of bug.

---

## Summary

| | Current behavior | Required behavior |
|---|---|---|
| **Lock acquisition** | After Stripe charge completes | Before message crosses queue boundary |
| **Lock operation** | Separate `GET` then `SET` | Atomic `SET NX` |
| **Concurrency window** | Entire Stripe API + DB write time | Zero — atomic |
| **Worker scale impact** | Each new worker widens the race | No impact — lock held before enqueue |
| **Test coverage** | Sequential only | Needs concurrent request test |

The fix is one atomic Redis operation moved from the end of the worker to the beginning of the gateway. Everything else is defense in depth.
