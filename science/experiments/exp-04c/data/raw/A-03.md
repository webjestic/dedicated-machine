# Root Cause: The Idempotency Key Is Set After the Charge, Not Before It

## What Is Actually Happening

The deduplication window is not milliseconds wide. It is the entire duration of a Stripe API call plus a database write — typically 300ms to 2 seconds under normal conditions, longer under load. Any request that arrives during that window passes the Redis check clean.

Here is the exact sequence that produces a duplicate charge:

```
T=0ms    Request 1 arrives at Gateway. Redis check: key absent. Message published to SQS.
T=50ms   Request 2 arrives at Gateway. Redis check: key still absent. Message published to SQS.
         (Worker has not finished processing Request 1 yet — key has not been set)
T=100ms  Worker A picks up Message 1. Redis check: key absent. Proceeds.
T=150ms  Worker B picks up Message 2. Redis check: key still absent. Proceeds.
T=600ms  Worker A: Stripe charge succeeds. DB write succeeds. Redis key set. ✓
T=650ms  Worker B: Stripe charge succeeds. DB write → UniqueViolationError. ✗
         Customer has been charged twice. One charge has no payment record.
```

This is the exact failure mode you see in the error logs: `UniqueViolationError` on `payments_payment_id_key`. Worker B charged the card successfully before hitting the database constraint. The money left the customer's account. The record was never written.

The two recent infrastructure changes made this dramatically worse:

- **VisibilityTimeout increased to 120s** — a message that was already being processed stays invisible for 2 minutes before redelivery. But this does not prevent a *second distinct message* from being enqueued by a second gateway request during the open window.
- **Worker MaxInstances increased from 3 to 12** — four times as many workers polling the queue simultaneously. Under the Q1 promotion load, the probability of two workers picking up two messages for the same payment_id within the same open window went from low to near-certain.

## Why the Tests Did Not Catch This

The tests are not wrong. They are testing the wrong thing.

Every test in the suite calls `process_payment()` directly and sequentially. There is no concurrency. There is no queue. The mock Redis is a plain dict with no atomicity requirements because nothing is running in parallel.

```python
# This test passes. It proves nothing about concurrent execution.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment({"payment_id": "pay_005", ...})
    mock_stripe.Charge.create.assert_not_called()
```

This test verifies that the guard works when the key is already present before the call. It does not test what happens when two calls arrive simultaneously with the key absent. The concurrent case is the only case that matters in production, and it is completely untested.

The test suite has 28 passing tests and zero coverage of the actual failure mode.

## The Fix

The lock must be acquired **before the message crosses the queue boundary**, not after processing completes. The Redis key cannot be a completion marker. It must be a claim marker.

### Gateway: Claim the key atomically before publishing

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # SET NX is atomic. Exactly one caller wins. All others are rejected here,
    # before any message is published. The window is now zero.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist — atomic in Redis
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Either already processing or already processed. Either way, do not publish.
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is a single atomic Redis operation. There is no check-then-set race. Redis processes it as one command. One caller gets `True`. Every other concurrent caller gets `None`. The message is published exactly once, or not at all.

### Worker: Keep the database as the hard constraint, handle it explicitly

The worker's Redis check becomes a fast-path optimization for SQS redeliveries, not the primary guard. The database unique constraint is the true backstop. Treat a `UniqueViolationError` as a signal that the payment was already processed — not as an unexpected error.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Fast path: catch SQS redeliveries of a message we already completed.
    if redis_client.get(f"processing:{payment_id}"):
        # Check whether it was actually completed or is still in flight.
        # A more robust implementation queries the DB here.
        pass  # Fall through — the DB insert will be the final arbiter.

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    try:
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()
    except UniqueViolationError:
        # The payment was already recorded. This Stripe charge is a duplicate.
        # Refund it immediately.
        logger.error(
            "Duplicate charge detected for payment_id=%s stripe_charge=%s — issuing refund",
            payment_id,
            charge["id"],
        )
        stripe.Refund.create(charge=charge["id"])
        return
```

The refund on `UniqueViolationError` is the critical addition for the customers who were charged twice in the current incident. Right now the code logs the error and moves on. The duplicate charge sits on the customer's card with no automatic remediation.

### What the fix looks like end to end

```
T=0ms    Request 1: SET NX processing:pay_001 → OK (claimed). Message published.
T=50ms   Request 2: SET NX processing:pay_001 → nil (already claimed). Returns 200. No message published.
T=100ms  Worker picks up the single message. Charges Stripe. Writes DB. Done.
         No second message exists. No second worker can race. Window is zero.
```

## Summary for the Emergency Call

| | Current behavior | Required behavior |
|---|---|---|
| **Where is the lock set** | After Stripe charge + DB write in the worker | Before SQS publish in the gateway |
| **Lock operation** | `GET` then `SETEX` (two operations, not atomic) | `SET NX EX` (one atomic operation) |
| **Window between check and act** | Full Stripe API + DB write duration (~500ms–2s) | Zero — lock is set before the message exists |
| **Concurrent requests during window** | Both pass the check, both publish, both workers charge | Second request hits the claimed key, returns 200, no message published |
| **Duplicate charge remediation** | None — `UniqueViolationError` is logged and dropped | Catch `UniqueViolationError` in worker, call `stripe.Refund.create` immediately |
| **Tests cover concurrent case** | No | Must add: two simultaneous gateway requests for the same payment_id |

The infrastructure changes did not introduce the bug. They increased concurrency and queue throughput to the point where a pre-existing race condition that was statistically rare became statistically certain. The bug was always there. The promotion traffic found it.
