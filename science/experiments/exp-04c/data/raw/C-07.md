# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is the entire payment processing time — Stripe API call plus database write. Any duplicate request that arrives during that window passes the check clean and gets charged.

---

## What Is Actually Happening

Walk through the sequence under concurrent load:

```
T+0ms    Request 1 arrives at gateway
T+0ms    Gateway checks Redis → key absent → publishes to SQS
T+50ms   Request 2 arrives at gateway (retry, duplicate tab, network blip)
T+50ms   Gateway checks Redis → key STILL absent → publishes to SQS again
T+100ms  Worker A dequeues message 1, checks Redis → key absent → proceeds
T+150ms  Worker B dequeues message 2, checks Redis → key STILL absent → proceeds
T+600ms  Worker A calls stripe.Charge.create() → charge 1 succeeds
T+650ms  Worker B calls stripe.Charge.create() → charge 2 succeeds
T+700ms  Worker A writes to DB, sets Redis key
T+750ms  Worker B writes to DB → UniqueViolationError (or second charge lands)
```

The Redis check in the worker is not wrong. The Redis check in the gateway is not wrong. **Both checks are useless because neither one holds a lock across the operation it is supposed to protect.**

The check and the act are separated by:
1. Network transit to SQS
2. Queue dwell time
3. Worker poll interval
4. Stripe API latency (typically 300–800ms)
5. Database write time

That is not a race condition measured in microseconds. That is a race condition measured in **seconds**, which is why it was invisible in sequential testing and catastrophic under the autoscaled worker pool you just expanded from 3 to 12 instances.

---

## Why the Recent Changes Made This Worse

Both changes deployed 6 days ago directly widened the blast radius:

**`VisibilityTimeout` increased from 30s to 120s**

This was the right fix for the timeout problem, but it means a message that was already being processed by Worker A stays invisible to other workers for 120 seconds. That sounds protective. It is not. The duplicate messages were already published to the queue by the gateway before any worker touched them. The visibility timeout governs redelivery of the *same* message, not the arrival of *duplicate* messages. You now have a longer window during which two distinct messages for the same `payment_id` can both be in-flight simultaneously.

**`MaxInstances` increased from 3 to 12**

More workers means more concurrent consumers. The probability that two workers dequeue two duplicate messages in the same processing window went from low to near-certain during any traffic spike. You did not increase throughput of a safe system — you increased the concurrency of an unsafe one.

---

## Why the Tests Did Not Catch This

Look at the test suite. Every test is sequential. `test_process_payment_skips_if_already_processed` passes because it manually pre-sets the Redis mock to return `"1"` before calling `process_payment`. It never tests what happens when two calls to `process_payment` run concurrently against a shared Redis state where neither has set the key yet.

`test_process_payment_sets_redis_key_after_processing` actually **asserts the broken behavior** — it confirms that `setex` is called after processing completes, which is precisely the ordering that causes the bug.

There is no test that spins up two threads, fires them simultaneously at `process_payment` with the same `payment_id`, and asserts that Stripe is called exactly once. That test would have failed immediately.

---

## The Fix

### Immediate: Atomic Lock Before the Queue Boundary

The idempotency key must be claimed **before** the message is published, using an atomic operation that guarantees only one caller can claim it.

**`gateway/charge_handler.py`**

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # SET NX is atomic. Exactly one caller wins. All others are rejected here,
    # before the message ever crosses the queue boundary.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Key already exists — either processing or already processed
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` (set-if-not-exists) is a single atomic Redis command. There is no gap between check and set. Two concurrent requests for the same `payment_id` will race on this one instruction, and Redis guarantees exactly one of them gets `True`. The other is rejected before it touches the queue.

### The Worker Still Needs Its Guard

The worker check is now a second line of defense against SQS redelivery (the original problem the `VisibilityTimeout` increase was trying to solve). Keep it, but understand what it protects against: redelivered messages, not concurrent duplicates. Those are now stopped at the gateway.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Defense against SQS redelivery. The gateway SET NX prevents concurrent
    # duplicates. This catches the case where a message is redelivered after
    # the visibility timeout expires on a slow or failed worker.
    if redis_client.get(f"processing:{payment_id}"):
        # Check if it was already successfully completed
        if redis_client.get(f"processed:{payment_id}"):
            return
        # Still processing or failed mid-flight — handle accordingly
        # (see note on failure handling below)

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Mark as fully processed. This key is what the worker checks above.
        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")

    except Exception:
        # Do NOT delete the message. Let SQS redeliver after visibility timeout.
        # The worker guard above will handle the redelivery safely.
        raise
```

### Use Stripe's Idempotency Keys as the Final Backstop

Stripe supports idempotency keys natively. Pass your `payment_id` as the idempotency key on every charge call. If a duplicate charge request reaches Stripe — whether due to a bug in your system or a network retry — Stripe will return the original charge object instead of creating a new one. This does not replace the fix above; it is a backstop that limits the blast radius if your application-level deduplication has a gap.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Add this
)
```

## Summary of What Needs to Change

| Location | Current behavior | Required behavior |
|---|---|---|
| `gateway/charge_handler.py` | Checks Redis, then publishes | Atomically claims key with `SET NX`, then publishes. Reject if claim fails. |
| `worker/payment_worker.py` | Checks Redis, charges, then sets Redis | Keep check as redelivery guard. Add Stripe idempotency key. |
| `stripe.Charge.create()` | No idempotency key | Pass `payment_id` as `idempotency_key` |
| Test suite | Sequential only | Add concurrent test: two threads, same `payment_id`, assert Stripe called once |

The lock has to be acquired before the message crosses the queue boundary. Right now it is acquired after the worker finishes. That is the entire processing time of exposure, and under 12 concurrent workers during a promotion spike, every duplicate request that arrives during that window will result in a charge.
