# Root Cause Analysis — Duplicate Charges

## The Structural Cause

**The idempotency key is set after the charge completes. The deduplication check happens before the message is queued. The window between those two events is unbounded — it is the entire payment processing time. Any request that arrives during that window passes the check and publishes a second message.**

This is not a bug introduced by the recent changes. The recent changes increased the blast radius. The structural defect was always there.

---

## The Failure Path

Walk through exactly what happens with two concurrent requests for the same `payment_id`:

```
T+0ms    Request 1 arrives at Gateway
T+0ms    Gateway checks Redis → key not set → passes
T+0ms    Gateway publishes message 1 to SQS
T+0ms    Gateway returns 202

T+50ms   Request 2 arrives at Gateway (retry, client timeout, whatever)
T+50ms   Gateway checks Redis → key STILL not set → passes
T+50ms   Gateway publishes message 2 to SQS
T+50ms   Gateway returns 202

         ... Worker picks up message 1 ...
         ... stripe.Charge.create() executes ...
         ... db.execute() + db.commit() executes ...

T+3200ms Redis key finally set to "processed:pay_id"

         Worker picks up message 2
         Redis check → key now set → skips
```

In the happy path above, message 2 is deduplicated at the worker. **But that only works if message 2 arrives at the worker after message 1 has fully completed and set the Redis key.** With 12 worker instances polling the same queue, message 2 can be picked up by a second worker while message 1 is mid-flight through Stripe. Both workers pass the Redis check. Both charge the card.

```
T+0ms    Worker A picks up message 1 → Redis check passes → calls Stripe...
T+100ms  Worker B picks up message 2 → Redis check passes → calls Stripe...
T+3100ms Worker A sets Redis key, writes DB
T+3200ms Worker B sets Redis key, writes DB → UniqueViolationError on payment_id
```

This is exactly what the `UniqueViolationError` is telling you. The database constraint is the only thing catching it — after the second charge has already gone through.

---

## Why the Recent Changes Made It Worse

The defect existed before. The recent changes widened the window and increased the number of concurrent workers racing through it.

**VisibilityTimeout 30s → 120s:** A message that takes longer than the visibility timeout becomes visible again and gets redelivered to a second worker before the first has finished. Increasing the timeout was the right operational instinct, but it did not close the race — it just reduced one redelivery path while the concurrent-worker path remained open.

**MaxInstances 3 → 12:** Four times as many workers polling the same queue. Four times the probability that two workers pull the same logical payment through the window simultaneously. This is why the incident started 6 days ago and correlates directly with the deployment. More instances, more concurrent processing, more races completed before the Redis key is set.

**The tests pass because they are sequential.** Every test calls `process_payment()` directly, one at a time, with a shared mock. There is no queue, no concurrency, no network latency. The race condition cannot manifest. The test suite is not wrong — it is testing the wrong thing. It validates the logic of a single execution path. It does not validate the system under concurrent load.

---

## What Needs to Change

### Immediate Fix — Set the Lock Before Publishing

The idempotency key must be acquired in the gateway, atomically, before the message crosses the queue boundary. Use Redis `SET NX` (set if not exists) as the gate. If the key cannot be set, the payment is already in flight or already processed — do not publish.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic acquire: set key only if it does not exist.
    # This is the single moment of truth. If this fails, we do not publish.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "in_flight",
        nx=True,        # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not acquired:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The key is now set **before** the message is published. Any subsequent request for the same `payment_id` — whether it arrives 50ms later or 50 seconds later — hits the Redis check, finds the key, and is rejected. The window is closed.

The worker's Redis check becomes a defense-in-depth guard against SQS redelivery, not the primary deduplication mechanism. Keep it. It is still valuable. But it is no longer load-bearing.

```python
# worker/payment_worker.py — worker check stays, role changes

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard: catches SQS redelivery after a worker crash mid-flight.
    # Primary deduplication now happens at the gateway before publish.
    if redis_client.get(f"processed:{payment_id}") == "1":
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Update key from "in_flight" to "1" to signal completion.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### One Remaining Edge Case to Acknowledge

If the gateway sets the Redis key and then crashes before publishing to SQS, that `payment_id` is locked out for 24 hours with no payment recorded. This is the correct tradeoff — a missed charge is recoverable; a duplicate charge to a customer is not. The customer or system can retry with a new `payment_id`, or you can build a reconciliation job that detects `in_flight` keys older than a threshold with no corresponding DB record. Do not try to solve this in the same PR as the incident fix.

### Also Required — Stripe Idempotency Keys

Pass `idempotency_key=payment_id` to `stripe.Charge.create()`. This is a second independent layer. If a worker crashes after charging Stripe but before writing to the database, a retry will hit Stripe's own idempotency layer and return the original charge object rather than creating a new one.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,
)
```

This costs nothing and eliminates an entire class of partial-failure duplicates.

## Summary

| | Current | Fixed |
|---|---|---|
| **Where lock is acquired** | Worker, after charge completes | Gateway, before message published |
| **Window size** | Full payment processing time (seconds) | Zero — atomic NX set |
| **Concurrent workers racing** | Yes — all pass check during window | No — first publisher wins |
| **SQS redelivery protection** | Worker Redis check (correct) | Worker Redis check (retained) |
| **Stripe-level protection** | None | Idempotency key on every charge |

The database `UNIQUE` constraint on `payment_id` is the only thing preventing more duplicate charges from being recorded right now. It is not preventing the duplicate Stripe charges — those are already happening before the insert fails. The fix above closes the race. The Stripe idempotency key handles the partial-failure case the fix does not cover.
