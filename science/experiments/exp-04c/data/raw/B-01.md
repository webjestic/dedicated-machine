# Root Cause Analysis — Duplicate Charges

## The Structural Cause

**The idempotency lock is set after the charge completes. The check happens before the message crosses the queue boundary. The window between those two events is the entire payment processing time — network round-trip to Stripe, database write, then Redis write. Any duplicate request that arrives during that window passes the check and gets published.**

This is not a race condition in the traditional sense. It is a design gap. The check and the lock are in different services with unbounded time between them.

---

## The Failure Path — Traced Exactly

Here is what happens under concurrent load:

```
T+0ms    Request 1 arrives at Gateway
T+0ms    Gateway checks Redis → key "processed:pay_001" does not exist → PASS
T+1ms    Gateway publishes pay_001 to SQS
T+1ms    Gateway returns 202

T+50ms   Request 2 arrives at Gateway (retry, duplicate tab, network blip)
T+50ms   Gateway checks Redis → key "processed:pay_001" still does not exist
         (Worker has not finished. Redis has not been written. The key is not there.)
T+51ms   Gateway publishes pay_001 to SQS — SECOND MESSAGE IN QUEUE
T+51ms   Gateway returns 202

T+200ms  Worker A dequeues message 1, checks Redis → not set → proceeds
T+210ms  Worker B dequeues message 2, checks Redis → not set → proceeds
         (Both workers are now past their guards simultaneously)

T+800ms  Worker A calls stripe.Charge.create → charge succeeds → ch_aaa
T+801ms  Worker B calls stripe.Charge.create → charge succeeds → ch_bbb
         TWO CHARGES ON THE CARD

T+810ms  Worker A writes to DB → succeeds
T+811ms  Worker B writes to DB → UniqueViolationError on payment_id_key
T+812ms  Worker B logs error, does NOT delete SQS message
T+813ms  Worker B's message becomes visible again after VisibilityTimeout
T+933ms  Worker C dequeues the redelivered message, checks Redis → now set → skips
         But the second Stripe charge already happened. The money is gone.
```

The `UniqueViolationError` you are seeing in the logs is the database catching the second insert. It is not preventing the second charge. The charge already happened before the insert. The database constraint is a symptom reporter, not a guard.

---

## Why the Recent Changes Made It Worse

The two infrastructure changes did not introduce the bug. The bug was always there. They widened the window and increased the probability of hitting it.

**VisibilityTimeout 30s → 120s:** When a message is in flight and the worker is slow, SQS hides it from other workers for the duration of the timeout. At 30s, a slow payment would redeliver quickly and the duplicate would hit the worker-side Redis check (which by then might be set). At 120s, the redelivery window is wider, but more importantly this change signals that payments were already taking longer than 30s — meaning the check-to-lock window was already dangerously long.

**Worker MaxInstances 3 → 12:** More workers polling the queue means higher probability that two workers dequeue two copies of the same message within milliseconds of each other — before either has written to Redis. This directly increases the concurrency that exposes the gap.

Neither change is the root cause. Both are accelerants.

---

## Why the Tests Pass

The tests are sequential. They test the gateway and worker in isolation, never concurrently. The mock Redis is shared and synchronous — when `process_payment` calls `setex`, it is visible to the next call immediately. There is no queue between them. There is no time between check and lock.

```python
# This test proves nothing about concurrent safety
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"  # Key pre-set. Real scenario: key not yet set.
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

The test that would have caught this does not exist: two concurrent calls to `process_payment` with the same `payment_id`, neither Redis key pre-set, both racing through the guard.

## What Needs to Change

### Immediate Fix — Move the Lock Before the Queue Boundary

The idempotency key must be set in the Gateway, before the message is published. Use a Redis `SET NX` (set if not exists) as an atomic claim. If the claim succeeds, publish. If it fails, the payment is already in flight or complete.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim: SET key value NX EX ttl
    # Returns True if key was set (we own it), False if key already existed
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not claimed:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard: catches redelivered messages after a worker crash
    # At this point the key should already exist as "pending"
    current = redis_client.get(f"processed:{payment_id}")
    if current == "1":
        return  # Already completed

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Overwrite "pending" with "1" to mark completion
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

The `SET NX` is atomic at the Redis level. Two concurrent gateway requests for the same `payment_id` cannot both succeed. One gets `True`, one gets `False`. Only one message enters the queue.

### Secondary Fix — Stripe Idempotency Keys

This is a defense-in-depth measure, not a substitute for the above. Pass the `payment_id` as the Stripe idempotency key. If a duplicate charge call reaches Stripe, Stripe will return the original charge object instead of creating a new one.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Add this
)
```

This does not fix the architectural gap. It does mean that if a duplicate slips through, Stripe absorbs it rather than charging the customer twice. It is the last line of defense and it costs nothing to add.

### What to Do About the Pending Key on Worker Failure

The `SET NX` approach introduces a new failure mode: if the Gateway sets the key to `"pending"` and then the worker crashes before completing, the payment is stuck. The key exists, so the Gateway will reject retries, but no charge was made.

Handle this with a TTL and a status check. The worker sets the key to `"1"` on success. If the key exists as `"pending"` after a reasonable processing window, a separate reconciliation job can identify stuck payments and either requeue them or alert. Do not try to solve this inline in the hot path today — that is a follow-up. The immediate priority is stopping the duplicate charges.

---

## Summary

| | Current | Fixed |
|---|---|---|
| Lock location | Worker, after Stripe call | Gateway, before queue publish |
| Lock operation | `SETEX` (always writes) | `SET NX` (atomic, conditional) |
| Window of vulnerability | Entire Stripe processing time | Zero — lock is held before publish |
| Stripe idempotency key | Not set | Set to `payment_id` |
| Duplicate request behavior | Second request enters queue, charges card | Second request rejected at Gateway |

The fix is two lines in the Gateway and one line in the Stripe call. The tests need to be rewritten to test concurrent behavior, not just sequential behavior. The 28 passing tests are testing the wrong thing.
