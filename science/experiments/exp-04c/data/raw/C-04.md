# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check-then-act window is not milliseconds — it is the entire Stripe API call plus the database write. Every request that arrives during that window passes the check and gets charged.

---

## What Is Actually Happening

Walk through the sequence with two concurrent requests for the same `payment_id`:

```
T=0ms   Request 1 → Gateway: redis.get("processed:pay_001") → None  ✓ passes
T=5ms   Request 2 → Gateway: redis.get("processed:pay_001") → None  ✓ passes
T=5ms   Request 1 → SQS: message published
T=6ms   Request 2 → SQS: message published        ← two messages now in queue
T=6ms   Worker A picks up message 1
T=6ms   Worker B picks up message 2                ← 12 workers, this is expected
T=6ms   Worker A: redis.get("processed:pay_001") → None  ✓ passes
T=7ms   Worker B: redis.get("processed:pay_001") → None  ✓ passes
T=800ms Worker A: stripe.Charge.create() → ch_aaa  ← first charge
T=820ms Worker B: stripe.Charge.create() → ch_bbb  ← duplicate charge
T=821ms Worker A: INSERT INTO payments ...
T=822ms Worker B: INSERT INTO payments ... → UniqueViolationError
T=823ms Worker A: redis.setex("processed:pay_001") ← too late
```

The Redis check in the gateway is checking a key that only gets written **after processing completes**. The lock is checked in Service A but written in Service B, with unbounded latency between them. This is not a race condition in the traditional sense — it is a design gap. The guard and the act are in different services with no shared critical section.

The `UniqueViolationError` you are seeing in the logs is the database constraint catching the second insert **after** Stripe has already been charged. That means some customers got charged twice successfully (both workers finished before either hit the DB constraint) and some got charged once with no payment record (the second worker charged Stripe, then hit the constraint, then the message went back to the queue and was skipped by the Redis check that the first worker finally set).

---

## Why the Recent Changes Made It Worse

Both changes deployed 6 days ago directly widened the window and increased the number of concurrent workers racing through it:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | A message that times out now stays invisible for 2 minutes before redelivery. This does not cause duplicates by itself, but it means a slow payment holds the message longer, increasing the chance a second request arrives at the gateway before the key is set. |
| `MaxInstances` 3 → 12 | Four times as many workers polling the queue simultaneously. The probability that two workers pick up two copies of the same message in the same window went from low to near-certain under load. |

The promotion traffic spike is what turned a latent design flaw into a production incident.

---

## Why the Tests Did Not Catch This

The tests are sequential. They are testing the right behaviors in isolation but they cannot observe the race:

```python
# This test proves the guard works when Redis is already set.
# It does not prove the guard works when two requests arrive before Redis is set.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

There is also no test that calls `process_payment` twice concurrently with the same `payment_id` and asserts that Stripe is only called once. The test suite has 28 passing tests and zero of them exercise concurrent execution. This is the test gap that needs to be closed alongside the code fix.

## What Needs to Change

### Fix 1: Acquire the lock before the message crosses the queue boundary (required)

The gateway must set a reservation key in Redis **before** publishing to SQS, using `SET NX` (set if not exists) so that only one request can win the reservation. This closes the window entirely — if a second request arrives at the gateway, it hits the reservation key and is rejected before a second message is ever published.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically reserve this payment_id. SET NX returns True only for the first caller.
    # Use a separate key so we can distinguish "reserved but not yet processed"
    # from "fully processed" if that distinction matters for your status API.
    reserved = redis_client.set(
        f"reserved:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not reserved:
        # Either already reserved (in-flight) or already processed.
        # Check which one if you need to return different status codes.
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` operation is atomic in Redis. Two concurrent requests cannot both win. The first sets the key and publishes. The second finds the key already set and returns without publishing. There is now only one message in the queue per `payment_id`.

### Fix 2: Keep the worker guard, but treat it as a safety net (required)

The worker check stays. SQS delivers messages at-least-once. Even with Fix 1, a message can be redelivered if the worker crashes after charging but before deleting the message. The worker guard handles that case. But it must also check the reservation key:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Check both keys: reservation (set by gateway) and processed (set by worker).
    # A missing reservation key means the message arrived without going through
    # the gateway — treat that as an error worth alerting on.
    if redis_client.get(f"processed:{payment_id}"):
        logger.info("Payment %s already processed, skipping.", payment_id)
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    try:
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()
    except UniqueViolationError:
        # The DB constraint caught a duplicate. The charge already happened.
        # Log this as an alert — it means the reservation key did not prevent
        # a second message, which should not happen after Fix 1 is deployed.
        logger.error(
            "DUPLICATE CHARGE DETECTED: payment_id=%s stripe_charge_id=%s — "
            "investigate reservation key logic immediately.",
            payment_id, charge["id"],
        )
        raise

    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Fix 3: Use Stripe's idempotency key (defense in depth, strongly recommended)

Stripe supports idempotency keys natively. Pass the `payment_id` as the idempotency key on every charge call. If two requests reach Stripe with the same key, Stripe returns the result of the first call and does not charge again. This is your last line of defense if both Fix 1 and Fix 2 fail:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # Stripe deduplicates on their side
)
```

This costs nothing and eliminates the financial impact of any future race condition that gets past your guards. It does not prevent duplicate messages from being processed, but it prevents duplicate charges from reaching the customer.

## Layered Defense Summary

```
Request 1 ──► Gateway: SET NX "reserved:pay_001" ──► wins ──► publishes to SQS
Request 2 ──► Gateway: SET NX "reserved:pay_001" ──► loses ──► returns 200, no publish

SQS redelivery ──► Worker: GET "processed:pay_001" ──► already set ──► skips

Both workers somehow get the message ──► Stripe idempotency key ──► one charge

All three fail ──► DB unique constraint ──► catches it, logs alert, no silent duplicate
```

Each layer catches what the layer above it missed. The fix is not to make any single layer perfect — it is to ensure that no single failure mode results in a customer being charged twice.

## Immediate Actions

1. **Deploy Fix 1 now.** This stops new duplicates from entering the queue. It is a small change to the gateway with no worker dependency.
