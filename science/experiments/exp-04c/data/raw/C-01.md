# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check-then-act window is not milliseconds — it is the entire Stripe API call plus the database write. Every request that arrives during that window passes the Redis check and gets charged.

---

## What Is Actually Happening

### The Race Condition

```
Time →

Request 1 arrives → Redis check: key absent → publishes to SQS
Request 2 arrives → Redis check: key absent → publishes to SQS   ← arrives before Worker 1 sets Redis
                                                  ↑
                              This is the entire payment processing window.
                              Stripe p50 latency: ~300ms. p99: several seconds.
                              Under load, this window is wide open.

Worker 1 picks up msg 1 → Redis check: absent → stripe.Charge.create() → db.execute() → redis.setex()
Worker 2 picks up msg 2 → Redis check: absent → stripe.Charge.create() → db.execute() → BOOM: UniqueViolationError
                                                                                          (or second successful charge)
```

The Redis check in the gateway is checking a key that only gets written by the worker, **after the charge completes**. The lock is never held. It is only observed, after the fact, by requests that arrive after a full payment cycle has completed.

### Why the Recent Changes Made It Worse

The two deployment changes six days ago directly widened this window:

**`VisibilityTimeout` increase (30s → 120s):** A message that times out and gets redelivered now stays invisible for 120 seconds. During that window, the original message is still being processed (or has failed silently), and the redelivered copy is sitting in the queue waiting. When the timeout expires, both can be in-flight simultaneously across different workers.

**Worker autoscaling (MaxInstances 3 → 12):** Four times as many workers polling the same queue. The probability that two workers pick up two copies of the same message in the same window — or that two gateway requests for the same `payment_id` both pass the Redis check before either worker finishes — increased by roughly 4x. This is not a coincidence. The incident started six days ago.

### Why the Tests Pass

The tests are sequential. They call `process_payment()` directly and synchronously. There is no queue boundary, no network latency, no concurrent workers. The mock Redis is a plain dict with no locking. The tests prove the happy path works in isolation. They cannot observe a race condition that requires two threads executing simultaneously with real latency between the check and the write.

```python
# This test proves nothing about concurrency:
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", "amount": 1200, "customer_id": "cust_pqr"})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

It confirms `setex` is called. It does not test what happens when a second worker calls `process_payment` with the same `payment_id` before `setex` completes.

---

## The Fix

### Principle

The idempotency lock must be **acquired before the message crosses the queue boundary**, not after the charge completes. The lock must be atomic. Check-then-set is not atomic. `SET NX` is.

### Step 1: Claim the lock in the gateway before publishing

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id.
    # SET key value NX EX ttl — sets only if key does not exist.
    # Returns True if the key was set (we own it), None if it already existed.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if not exists — atomic in Redis
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Key already existed — either processing or already processed.
        # Check which state we're in to return the right response.
        if redis_client.get(f"processed:{payment_id}"):
            return jsonify({"status": "already_processed", "payment_id": payment_id}), 200
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 202

    # We hold the lock. Now publish.
    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` is a single atomic Redis command. There is no window between the check and the claim. Two concurrent requests for the same `payment_id` will race on this command. Exactly one will receive `True`. The other receives `None` and stops.

### Step 2: Keep the worker guard, but treat it as a safety net, not the primary lock

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard — catches redelivered SQS messages.
    # The gateway SET NX is the primary lock. This is defense in depth.
    if redis_client.get(f"processed:{payment_id}"):
        logger.info("Payment %s already processed, skipping redelivered message", payment_id)
        return

    # Charge via Stripe
    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    # Record in database
    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Mark as fully processed. This is what the gateway check reads on retry.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Step 3: Handle the SQS redelivery case with Stripe idempotency keys

The `VisibilityTimeout` increase means a message can be redelivered if a worker dies mid-flight. The worker guard above catches this if Redis is healthy. Add a Stripe idempotency key as a third layer — Stripe will deduplicate on their side even if both guards fail:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # Stripe deduplicates on this key for 24h
)
```

This costs nothing and means a redelivered message that somehow bypasses both Redis checks will not produce a second charge at the Stripe level.

---

## Immediate Mitigation (Before Deploy)

While the fix is being reviewed and deployed:

1. **Scale worker `MaxInstances` back to 3.** Fewer concurrent workers means fewer simultaneous races. This does not eliminate the bug but reduces the blast radius significantly.
2. **Revert `VisibilityTimeout` to 30s** or set it to match your actual p99 processing time with headroom. The 120s value was masking a different problem (payments timing out) that should be diagnosed separately — possibly a Stripe timeout issue or a slow DB write.
3. **Do not attempt to replay failed messages** from the queue until the fix is deployed. Replaying now will produce more duplicates.

---

## What the Postmortem Should Record

The deduplication guard was architecturally misplaced from the start. It was set in the component that processes, not the component that accepts. The check and the act were separated by a queue boundary and a full Stripe API call. The system appeared correct in all tests because tests are sequential and the race requires concurrency. The autoscaling change did not introduce the bug — it revealed a latent race condition that existed from the first deployment. Any concurrent load would have triggered it eventually.

The fix is not "add more checks." The fix is to make the claim atomic and to make it before the work is dispatched, not after it completes.
