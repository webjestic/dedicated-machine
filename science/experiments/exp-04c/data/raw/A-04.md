# Root Cause: The Idempotency Key Is Set After the Charge, Not Before It

## What Is Actually Happening

The deduplication window is not milliseconds wide. It is the entire Stripe API call plus the database write — easily 2–5 seconds under normal conditions, longer under load. Any duplicate request that arrives during that window passes both guards and results in a second charge.

There are two distinct failure paths producing the 23 tickets.

---

## Failure Path 1: Concurrent HTTP Requests to the Gateway

```
t=0ms   Request A arrives → redis.get("processed:pay_001") → None → publishes to SQS
t=50ms  Request B arrives → redis.get("processed:pay_001") → None → publishes to SQS
                                                              ↑
                                        Worker hasn't finished yet. Key not set.
                                        Both messages are now in the queue.

t=3000ms  Worker finishes Request A → redis.setex("processed:pay_001") ← too late
t=3050ms  Worker picks up Request B → redis.get → "1" → skips  (if lucky)
                                                OR
t=3050ms  Second worker picks up Request B → redis.get → "1" → skips  (if lucky)
```

The Redis check in the gateway is checking for a key that the worker has not set yet. The gateway has no way to know a message is already in flight. The check is structurally useless for concurrent requests.

---

## Failure Path 2: SQS Redelivery to Multiple Workers

This is the failure the `VisibilityTimeout` change was meant to fix, and it made things worse.

```
t=0s      Worker 1 receives pay_001, starts processing
t=0s      SQS hides message for 120s (VisibilityTimeout)
t=2s      Worker 1: stripe.Charge.create() → success
t=2s      Worker 1: db.execute INSERT → success  
t=2s      Worker 1: redis.setex("processed:pay_001") ← key finally set
t=2s      Worker 1: sqs.delete_message() ← message deleted

--- but if Worker 1 dies or stalls between the Stripe call and the delete ---

t=120s    SQS makes message visible again (VisibilityTimeout expires)
t=120s    Worker 2 receives pay_001
t=120s    Worker 2: redis.get("processed:pay_001") → None  ← key was never set
t=120s    Worker 2: stripe.Charge.create() → second charge
```

With `MaxInstances` raised from 3 to 12, there are now 12 workers competing to pick up redelivered messages. The probability of two workers processing the same message concurrently — or a redelivered message hitting a worker before the first worker sets the Redis key — increased by 4x.

---

## Why the Tests Don't Catch This

The tests are single-threaded and sequential. `process_payment` is called directly, not through the queue. There is no concurrency. The mock Redis is synchronous and shared in-process. The tests prove the happy path works. They cannot prove anything about concurrent execution.

```python
# This test proves nothing about concurrent workers:
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"   # ← manually pre-set, no race possible
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

The test for the Redis key being set after processing is actually documenting the bug:

```python
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", ...})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
    # ↑ This test name contains the words "after processing" and passes. That is the bug.
```

## The Fix

### Immediate: Set the lock before the Stripe call, not after

The idempotency key must be acquired atomically before any external action is taken. Use `SET NX` (set if not exists) as a distributed mutex. If the key is already set, another worker owns this payment.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]
    lock_key = f"processed:{payment_id}"

    # Atomically claim this payment. If we don't get the lock, another
    # worker is processing or has processed it. Either way, we stop here.
    acquired = redis_client.set(lock_key, "1", nx=True, ex=DEDUP_TTL)
    if not acquired:
        logger.info("Payment %s already claimed or processed, skipping", payment_id)
        return

    # Now we hold the lock. No other worker will pass the check above.
    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Processing failed. Release the lock so a redelivered message can retry.
        # Do NOT release if the Stripe charge succeeded but the DB write failed —
        # check Stripe first before releasing to avoid charging twice on retry.
        redis_client.delete(lock_key)
        raise
```

The critical property: `SET NX` is atomic at the Redis level. Two workers calling it simultaneously for the same key will get exactly one `True` and one `None`. There is no window between check and set.

### Also Required: Stripe Idempotency Keys

Redis alone is not sufficient. Redis can fail. The lock can expire. Add Stripe-native idempotency as a second layer:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # Stripe deduplicates on their end
)
```

With this in place, even if two workers both reach the Stripe call, Stripe will return the same charge object for the same idempotency key and will not charge the card twice.

### Gateway: Set a Pending Marker Before Publishing

The gateway check is also broken. Fix it by marking the payment as in-flight before publishing to the queue:

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim the slot. If another request beat us here, stop.
    acquired = redis_client.set(
        f"processed:{payment_id}", "pending", nx=True, ex=DEDUP_TTL
    )
    if not acquired:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(QueueUrl=QUEUE_URL, MessageBody=json.dumps(data))
    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker then overwrites `pending` with `1` on completion. If the gateway crashes after setting the key but before publishing, the message never enters the queue and the key expires after 24 hours — acceptable behavior.

## Summary for the Call

| | Current behavior | Required behavior |
|---|---|---|
| **Lock acquisition** | After Stripe + DB write | Before Stripe call |
| **Lock mechanism** | `GET` then `SETEX` (two operations, not atomic) | `SET NX EX` (one atomic operation) |
| **Stripe deduplication** | None | Idempotency key on every charge call |
| **Gateway guard** | Checks a key the worker hasn't set yet | Sets the key atomically before publishing |
| **Concurrent safety** | No | Yes |
| **Redelivery safety** | No | Yes |

The `VisibilityTimeout` increase and the autoscaling increase did not cause the bug. They widened the window and increased the worker count, which turned a latent race condition into a production incident. The bug was present before those changes. Those changes just made it visible.
