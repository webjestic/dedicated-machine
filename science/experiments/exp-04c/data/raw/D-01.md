## Root Cause

**The idempotency check and the Stripe charge are not atomic.** There is a race condition in `process_payment` between the Redis `GET` and the `setex` at the end. With 12 worker instances now running concurrently (up from 3), multiple workers can pass the guard check simultaneously for the same message before any of them writes the processed flag.

Here's the exact sequence that produces a duplicate charge:

```
Worker A: GET processed:pay_001  → nil  (no key, proceeds)
Worker B: GET processed:pay_001  → nil  (no key, proceeds — A hasn't finished yet)
Worker A: stripe.Charge.create() → ch_aaa  ✓
Worker B: stripe.Charge.create() → ch_bbb  ✓  ← duplicate charge
Worker A: INSERT payments ...    → ok
Worker B: INSERT payments ...    → UniqueViolationError  ← this is the error in your logs
Worker A: redis.setex processed  → set
Worker B: redis.setex processed  → set (too late)
```

This explains both symptoms exactly:
- Customer charged twice when both Stripe calls succeed before either DB write
- `UniqueViolationError` on the second DB insert (the DB constraint catches it, but the charge already happened)
- The second charge has no payment record in your system (the insert failed, the exception was swallowed, the Stripe charge is orphaned)

### Why it wasn't happening before

The `VisibilityTimeout` increase from 30s → 120s was the right call, but it masked the real problem. Previously, messages redelivered after 30s would hit the Redis guard on the *redelivered* message. The race condition existed before, but with only 3 workers the window was narrow. **Scaling to 12 workers increased concurrency 4x, which made the race window wide enough to hit regularly under Q1 promotion traffic.**

### Why tests didn't catch it

The tests mock Redis and run single-threaded. There is no test that exercises two concurrent calls to `process_payment` with the same `payment_id`. The 28 passing tests are testing the happy path and the already-processed guard in isolation — neither of which reveals a race condition.

---

## The Fix

The guard needs to be an atomic **set-if-not-exists** operation, not a get-then-set. Redis `SET NX` (set if not exists) is atomic and eliminates the race window entirely.

### `worker/payment_worker.py`

```python
def process_payment(data: dict) -> None:
    """Charge the customer and record the transaction."""
    payment_id = data["payment_id"]

    # Atomically claim this payment_id before doing any work.
    # SET NX returns True only for the first caller — all concurrent
    # workers racing on the same payment_id will get False and exit.
    lock_acquired = redis_client.set(
        f"processed:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,  # same 24h TTL as before
    )
    if not lock_acquired:
        logger.info("Payment %s already claimed or processed, skipping.", payment_id)
        return

    try:
        # Charge via Stripe
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        # Record in database
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Processing failed — release the lock so the redelivered message
        # can be retried by another worker.
        redis_client.delete(f"processed:{payment_id}")
        raise
```

The key changes:

1. **`SET NX` replaces `GET` + late `setex`** — the claim is now a single atomic operation. Redis guarantees only one caller gets `True`.
2. **The lock is set *before* calling Stripe** — no two workers can both reach `stripe.Charge.create` for the same `payment_id`.
3. **The lock is released on failure** — if the Stripe call or DB write fails, we delete the key so the redelivered SQS message can be retried. Without this, a transient failure would permanently block a legitimate payment.

## What to communicate on the call

- **Root cause:** Race condition in `process_payment`. The Redis idempotency check is a non-atomic read-then-write. Under concurrent workers, multiple instances pass the guard before any of them sets the flag.
- **Trigger:** Scaling from 3 → 12 worker instances widened the race window enough to hit consistently under promotion traffic.
- **Fix:** One-line logic change — replace `GET` + `setex` with atomic `SET NX`. Deploy is low risk and requires no infrastructure changes.
- **Immediate mitigation while the fix is staged:** Scale `MaxInstances` back to 3. This narrows the race window significantly and will reduce (not eliminate) duplicates until the fix ships.
- **Affected customers:** Any payment where the `UniqueViolationError` appears in logs has an orphaned Stripe charge with no DB record. That list is recoverable by diffing Stripe's charge log against the payments table for the last 6 hours.
