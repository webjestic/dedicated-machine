# Root Cause Analysis — Duplicate Charges

## The Structural Cause

**The idempotency lock is set after the charge completes. The check happens before the message crosses the queue boundary. The window between those two events is the entire payment processing time — and any request that arrives during that window passes the check clean.**

This is not a race condition in the traditional sense. It is a design gap. There is no moment in this system where a payment is simultaneously "claimed" and "not yet processed." The system goes directly from "unclaimed" to "processed," with no in-progress state. That gap is where the duplicates are entering.

---

## The Failure Path — Traced Exactly

Here is what happens under concurrent load:

```
T+0ms    Request 1 arrives at gateway for pay_001
T+0ms    Gateway checks Redis: processed:pay_001 → (nil) ✓ passes
T+0ms    Gateway publishes pay_001 to SQS
T+0ms    Gateway returns 202

T+50ms   Request 2 arrives at gateway for pay_001 (retry, duplicate POST, or concurrent client)
T+50ms   Gateway checks Redis: processed:pay_001 → (nil) ✓ passes   ← STILL EMPTY
T+50ms   Gateway publishes pay_001 to SQS again
T+50ms   Gateway returns 202

         [Two messages for pay_001 now sitting in the queue]

T+200ms  Worker A dequeues message 1, checks Redis: processed:pay_001 → (nil) ✓ passes
T+210ms  Worker B dequeues message 2, checks Redis: processed:pay_001 → (nil) ✓ passes
         [Both workers pass the guard simultaneously]

T+800ms  Worker A calls stripe.Charge.create() → ch_aaa ✓
T+810ms  Worker B calls stripe.Charge.create() → ch_bbb ✓  ← DUPLICATE CHARGE

T+900ms  Worker A inserts into payments table ✓
T+910ms  Worker B inserts into payments table → UniqueViolationError ✗
T+920ms  Worker A sets Redis: processed:pay_001 ✓
```

This explains both symptoms exactly:
- **Customer charged twice, no error recorded:** Worker A wins the DB insert. Worker B charges Stripe successfully but fails on the DB insert. The exception is caught, the message is not deleted, it becomes visible again, and the cycle can repeat.
- **`UniqueViolationError` in logs:** Worker B hitting the unique constraint on `payment_id` after Worker A already inserted.

---

## Why the Recent Changes Made This Worse

The two infrastructure changes from six days ago did not introduce the bug. They widened the window and increased the number of concurrent workers racing through it.

**`VisibilityTimeout` increase (30s → 120s):** A message that was previously redelivered after 30 seconds now stays invisible for 120 seconds. This actually reduces one class of redelivery — but it means a slow payment holds the message longer, and if the worker dies mid-flight, the message reappears after 2 minutes with no Redis key set and no DB record, fully eligible for reprocessing.

**`MaxInstances` increase (3 → 12):** This is the direct accelerant. You now have up to 12 workers polling the same queue. The probability that two workers dequeue two copies of the same message — or two separately-published messages for the same `payment_id` — within the same processing window went from low to near-certain under promotion load. The gap was always there. You now have 12 threads running through it simultaneously.

This test pre-seeds Redis with `"1"` before calling `process_payment`. It proves that the guard works when the key is already there. ## What Needs to Change

There are two problems to fix. They are separate and both are required.

### Problem 1: The Gateway Has No Claim Step

The gateway checks Redis and then publishes. Nothing between those two operations prevents a second request from also checking Redis and also publishing. The check and the publish are not atomic.

**Fix:** Set a Redis key with `SET NX` (set if not exists) *before* publishing to the queue. Use a distinct key namespace to distinguish "in-flight" from "processed."

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. NX = only set if key does not exist.
    # This key covers both in-flight and completed payments.
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "1",
        ex=DEDUP_TTL,
        nx=True,  # SET NX — atomic claim
    )

    if not claimed:
        # Key already existed — either in-flight or already processed
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is atomic at the Redis level. Two concurrent requests racing to claim the same `payment_id` will have exactly one winner. The second call gets `nil` back and returns `already_processed` without publishing.

**Critical implication:** If the gateway claims the key but SQS publish fails, the payment is permanently blocked. You need error handling that deletes the Redis key on publish failure:

```python
    claimed = redis_client.set(f"processed:{payment_id}", "1", ex=DEDUP_TTL, nx=True)
    if not claimed:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    try:
        sqs.send_message(QueueUrl=QUEUE_URL, MessageBody=json.dumps(data))
    except Exception:
        redis_client.delete(f"processed:{payment_id}")  # release the claim
        raise

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

### Problem 2: The Worker Guard Is Not Sufficient Against Concurrent Workers

Even after fixing the gateway, SQS does not guarantee exactly-once delivery. A message can be delivered to two workers simultaneously if visibility timeout is misconfigured or if a worker stalls. The worker's Redis check must also be atomic.

The current worker guard is:

```python
if redis_client.get(f"processed:{payment_id}"):
    return
# ... time passes, another worker runs the same check ...
stripe.Charge.create(...)
```

With the gateway fix in place, the Redis key will already exist when the worker picks up the message. The worker's check will now correctly short-circuit. But this only holds if the gateway and worker share the same key namespace — which they do in the current code (`processed:{payment_id}`). Confirm that is intentional and do not change it.

The worker should also use `SET NX` as a secondary defense against the SQS duplicate-delivery case:

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic guard: if key already exists (set by gateway or prior worker), skip.
    # This is defense-in-depth against SQS duplicate delivery.
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "1",
        ex=DEDUP_TTL,
        nx=True,
    )
    if not claimed:
        # Already claimed by gateway or already processed — safe to discard
        logger.info("Skipping already-claimed payment %s", payment_id)
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()
    # Redis key was already set above — do not set it again here
```

Note: with the gateway setting the key before publish, the worker's `SET NX` will return `nil` (not claimed) only in the case of a genuine SQS duplicate delivery where somehow the key expired. That is the correct behavior — it becomes a last-resort guard rather than the primary one.
