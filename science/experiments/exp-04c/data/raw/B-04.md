# Root Cause Analysis — Duplicate Charges Incident

## The Short Answer

The idempotency lock is set **after** the charge completes. The window between the Redis check and the Redis write is the entire payment processing time — Stripe round-trip plus database write. Any duplicate request that arrives during that window passes the check clean and proceeds to charge. With 12 worker instances under Q1 promotion load, you have 12 concurrent opportunities to enter that window simultaneously.

---

## Failure Path — Traced Exactly

Here is the sequence that produces a duplicate charge. This is not speculation; this is what the `UniqueViolationError` in your logs is telling you.

```
T+0ms    Worker 1 receives message for pay_001
T+0ms    Worker 1: redis.get("processed:pay_001") → None  ✓ passes guard
T+0ms    Worker 1: stripe.Charge.create(...)  ← begins, takes ~300-800ms

T+50ms   Worker 2 receives SAME message (redelivery or second enqueue)
T+50ms   Worker 2: redis.get("processed:pay_001") → None  ✓ passes guard
         ^^^^ Redis key does not exist yet. Worker 1 hasn't finished.

T+350ms  Worker 1: stripe charge returns ch_abc123
T+351ms  Worker 1: db.execute(INSERT ...) → succeeds
T+352ms  Worker 1: redis.setex("processed:pay_001", ...)  ← lock finally set

T+360ms  Worker 2: stripe charge returns ch_xyz789  ← SECOND CHARGE ISSUED
T+361ms  Worker 2: db.execute(INSERT ...) → UniqueViolationError
         ^^^^ This is the error in your logs.
T+362ms  Worker 2: redis.setex never reached — exception was thrown first
```

The `UniqueViolationError` is a **symptom**, not the cause. The database constraint caught the second write. It did not prevent the second Stripe charge. Some customers were charged twice because both Stripe calls completed. Others show a failed second charge with no payment recorded because the DB constraint fired — the charge went through but the record didn't land.

---

## Why the Tests Pass

The tests are single-threaded and sequential. `test_process_payment_skips_if_already_processed` works because the mock returns `"1"` synchronously before any processing begins. There is no test that calls `process_payment` concurrently with the same `payment_id` and no artificial latency between the Redis check and the Redis write. The race condition is structurally untestable with this test design.

```python
# This test proves the guard works when Redis already has the key.
# It proves nothing about what happens when two workers enter simultaneously.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"   # key already set — this case is fine
    process_payment(...)                 # concurrent entry is never tested
```

---

## Why the Recent Changes Made It Worse

The `VisibilityTimeout` increase and the autoscaling increase did not introduce the bug. The bug was always there. They widened the window and increased the number of concurrent workers entering it.

**VisibilityTimeout 30s → 120s:** A message that takes longer than the timeout becomes visible again and gets picked up by a second worker. At 30s this was already happening — that's why the timeout was increased. At 120s the window is larger, but the core problem is that the lock is never held during processing, so the timeout value is irrelevant to the race.

**MaxInstances 3 → 12:** Four times as many workers polling the same queue. Under promotion load, the probability that two workers receive or re-receive the same message within the same processing window went from low to near-certain.

---

## The Structural Problem

```python
# worker/payment_worker.py — current order of operations

if redis_client.get(f"processed:{payment_id}"):   # 1. CHECK
    return

charge = stripe.Charge.create(...)                 # 2. ACT  ← unprotected
db.execute("INSERT INTO payments ...")             # 3. RECORD
db.commit()

redis_client.setex(f"processed:{payment_id}", ...) # 4. LOCK SET
#                                                       ^^^^ too late
```

The lock is set at step 4. The charge happens at step 2. Steps 2, 3, and 4 take hundreds of milliseconds. Every request that arrives between step 1 and step 4 passes the check.

The lock must be **acquired before the charge**, not after it.

---

## What Needs to Change

### Fix 1 — Atomic lock acquisition before processing (required, implement immediately)

Replace the read-then-write pattern with an atomic `SET NX` (set if not exists). This is a single atomic Redis operation. There is no window between the check and the claim.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim: SET processed:{id} "processing" EX 300 NX
    # Returns True if this worker won the claim, False if another worker already holds it.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "processing",
        ex=300,        # 5 minutes — must exceed your worst-case processing time
        nx=True,       # only set if key does not exist
    )

    if not acquired:
        logger.info("Payment %s already claimed or processed, skipping", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Overwrite "processing" with "processed" and extend TTL to 24 hours
        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "processed")

    except stripe.error.StripeError as e:
        # Release the lock so the message can be retried
        redis_client.delete(f"processed:{payment_id}")
        raise
    except Exception as e:
        redis_client.delete(f"processed:{payment_id}")
        raise
```

Two things to note about the exception handling here. If Stripe returns an error, you delete the lock so the message can be retried — that is correct behavior. If the Stripe charge **succeeds** but the database write fails, you do **not** delete the lock. You have a charge in Stripe with no record in your system. That is a separate reconciliation problem, but it is better than charging the customer twice. Log it, alert on it, and reconcile it. Do not retry a successful Stripe charge.

### Fix 2 — Idempotency key on the Stripe call (required, defense in depth)

Stripe provides idempotency keys precisely for this scenario. Pass the `payment_id` as the idempotency key. If two requests reach Stripe with the same key, Stripe returns the result of the first and does not charge again.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # add this
)
```

This is not a substitute for Fix 1. It is a second layer. Fix 1 prevents the duplicate Stripe call. Fix 2 ensures that if a duplicate call reaches Stripe anyway — network retry, bug in your lock logic, anything — Stripe absorbs it.

### Fix 3 — Gateway must also use SET NX (required)

The gateway has the same structural problem. It checks Redis and then publishes. Two HTTP requests for the same `payment_id` arriving within milliseconds of each other both pass the check and both enqueue a message.

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim at the gateway boundary
    acquired = redis_client.set(
        f"queued:{payment_id}",
        "1",
        ex=DEDUP_TTL,
        nx=True,
    )

    if not acquired:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

Use a separate key namespace (`queued:` vs `processed:`) so the gateway lock and the worker lock are independent and do not interfere with each other.

### Fix 4 — Database constraint stays, but treat it as a circuit breaker, not a guard

The `UNIQUE` constraint on `payments.payment_id` is correct and must stay. But it is catching charges that have already gone to Stripe. Catching a `UniqueViolationError` means money has already moved. The constraint is your last line of defense, not your primary one. The fixes above move the guard to before the charge. The constraint remains as a hard stop on data integrity.

---

## What to Do Right Now

1. **Deploy Fix 1 and Fix 2 together.** Fix 1 closes the race. Fix 2 protects against Stripe-level duplicates. Neither is optional.
2. **Deploy Fix 3.** The gateway race is lower probability under normal load but it is the same pattern.
3. **Audit Stripe for charges without a corresponding `payments` row.** The customers who were charged twice need refunds. The customers who have a Stripe charge but no database record need their records created or their charges refunded, depending on whether the payment should be considered complete.
4. **Do not roll back the VisibilityTimeout or MaxInstances changes** until after the fix is deployed. Rolling them back reduces the collision rate
