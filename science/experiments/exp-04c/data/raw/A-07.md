# Root Cause: Check-Then-Act Race Condition with No Atomic Lock

## What Is Happening Right Now

Two (or more) requests for the same `payment_id` arrive at the gateway within the same processing window. Both hit the Redis check before either worker has finished and written the `processed:` key. Both pass. Both enqueue. Both workers charge Stripe. One of three outcomes follows:

1. Both Stripe charges succeed → customer charged twice, both DB inserts succeed
2. Both Stripe charges succeed → second DB insert hits the `UniqueViolationError` → payment recorded once, but customer still charged twice, and the second charge is now an orphan with no DB record
3. First Stripe charge succeeds, second fails → no duplicate charge, but the `UniqueViolationError` in the logs still fires if the DB insert races

The `UniqueViolationError` you are seeing in the logs is not the cause — it is the symptom, and it is the only thing that saved some customers from a second successful charge. The cases where you do *not* see that error are the cases where both charges went through cleanly.

---

## The Exact Window

```
T+0ms    Request 1 arrives at gateway
T+0ms    Redis GET processed:pay_001 → nil  ✓ passes
T+1ms    Request 1 enqueued to SQS

T+50ms   Request 2 arrives at gateway
T+50ms   Redis GET processed:pay_001 → nil  ✓ passes  ← STILL NIL
T+51ms   Request 2 enqueued to SQS

T+200ms  Worker A dequeues message 1
T+200ms  Redis GET processed:pay_001 → nil  ✓ passes
T+800ms  Stripe charges card          ← 600ms of network time
T+810ms  DB INSERT succeeds
T+815ms  Redis SETEX processed:pay_001  ← KEY FINALLY SET

T+220ms  Worker B dequeues message 2
T+220ms  Redis GET processed:pay_001 → nil  ✓ passes  ← BEFORE T+815ms
T+900ms  Stripe charges card AGAIN
T+910ms  DB INSERT → UniqueViolationError
```

The Redis key is set at `T+815ms`. Every request and every worker that checks before that moment passes the guard. The window is not milliseconds — it is the entire Stripe round-trip plus DB write time, which under load can be hundreds of milliseconds to seconds.

---

## Why the Recent Changes Made This Dramatically Worse

Both changes deployed 6 days ago directly widened and multiplied the race window:

**`MaxInstances` 3 → 12:** You now have four times as many workers polling the queue simultaneously. Where previously one duplicate message might have been picked up by one of three workers, it can now be raced by one of twelve. The probability of two workers holding the same message in their processing window at the same time went up by roughly an order of magnitude.

**`VisibilityTimeout` 30s → 120s:** This was the right fix for the timeout-before-completion problem, but it did not address the root cause. It means a message that is being processed stays invisible longer — but it does not prevent two *separate enqueues* of the same `payment_id` from both being in flight. It also means that if a worker crashes mid-processing (after Stripe charge, before Redis write), the message reappears after 120 seconds and gets processed again. The guard that was supposed to catch redeliveries (`redis_client.get` in the worker) cannot catch this because the key was never written.

---

## Why the Tests Pass and Always Will

The tests are sequential. They call `process_payment()` directly, one at a time, in a single thread, against a mock that resolves instantly. There is no queue, no network latency, no concurrent workers. The race condition requires real time to elapse between the check and the write. The test suite cannot reproduce it by design.

```python
# This test proves the guard works when Redis already has the key.
# It does not test what happens when two workers check before either writes.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

The guard is logically correct. The tests are testing the right thing. Neither fact matters because the guard is set too late.

## The Fix

### Immediate: Set the lock before the message crosses the queue boundary

The idempotency key must be written in the gateway, atomically, before the message is published. Use `SET NX` (set-if-not-exists) as a compare-and-swap. If the key already exists, the payment is either in-flight or completed — either way, do not enqueue.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id before touching the queue.
    # SET NX returns True only if the key did not exist.
    # Use a separate "in-flight" key so the worker can distinguish
    # "never seen" from "currently processing" from "completed".
    acquired = redis_client.set(
        f"inflight:{payment_id}",
        "1",
        nx=True,       # only set if not exists
        ex=DEDUP_TTL,  # TTL must exceed max possible processing time
    )

    if not acquired:
        # Key already existed — either in-flight or already processed.
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard for redelivered messages.
    # The gateway lock prevents concurrent enqueues.
    # This catches SQS redelivery after a worker crash.
    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Write the permanent completion key.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
    # The inflight key can now be removed or left to expire.
    redis_client.delete(f"inflight:{payment_id}")
```

The critical property: `SET NX` is atomic at the Redis level. Two concurrent gateway requests racing on the same `payment_id` cannot both get `True`. One wins, one gets `False` and returns immediately without enqueuing. The message never enters the queue twice.

### Also Required: Stripe Idempotency Keys

The lock above prevents duplicate enqueues. It does not protect against a worker crash after the Stripe charge but before the DB write, followed by SQS redelivery. Add a Stripe idempotency key so that even if the Stripe call is retried, Stripe deduplicates it on their side:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this key
)
```

This is a defense-in-depth measure. Stripe will return the original charge object for any retry with the same idempotency key within 24 hours. The DB insert will still raise `UniqueViolationError` on retry, so catch that specifically and treat it as success — the charge already exists.

### Also Required: Handle the DB-Stripe Inconsistency

You currently have orphaned Stripe charges — charges that succeeded but whose DB insert failed. Add a reconciliation job that queries Stripe for charges in the last 24 hours and cross-references against the payments table. Any Stripe charge with no corresponding DB row needs to either be recorded or refunded. This is not optional — those are real customer charges with no internal record.

## Summary for the Emergency Call

| | Current behavior | Required behavior |
|---|---|---|
| Where idempotency key is set | Worker, after Stripe charge completes | Gateway, before message is enqueued |
| Mechanism | `GET` then `SETEX` (non-atomic, split across services) | `SET NX` (atomic, single operation, single service) |
| Window where duplicate is possible | Entire Stripe processing time (hundreds of ms to seconds) | Zero — atomic operation closes the window |
| Protection against SQS redelivery | Redis check in worker (correct but insufficient alone) | Redis check in worker + Stripe idempotency key |

The lock has to be acquired before the message crosses the queue boundary. Right now it is acquired after. That is the root cause.
