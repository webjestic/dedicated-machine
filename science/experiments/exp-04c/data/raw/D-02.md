## Root Cause

**The idempotency check is not atomic. There is a race condition between the Redis `GET` and the Stripe charge, and the Redis key is only written *after* the charge succeeds. Under concurrent execution, multiple workers can pass the guard simultaneously and charge the customer multiple times.**

This was always latent. The two recent changes made it reliably reproducible:

- **MaxInstances 3 → 12:** More workers polling concurrently means more threads racing through the same unprotected window.
- **VisibilityTimeout 30s → 120s:** Intended to stop redelivery, but it only delays it. Any message that takes longer than 120s (or where a worker crashes mid-flight) gets redelivered to a second worker while the first is still processing.

---

## The Race Condition, Step by Step

```
Worker 1                          Worker 2
────────────────────────────────────────────────────────
GET processed:pay_001 → nil
                                  GET processed:pay_001 → nil
stripe.Charge.create(pay_001) ✓
                                  stripe.Charge.create(pay_001) ✓  ← duplicate charge
INSERT INTO payments ...
                                  INSERT INTO payments ...          ← UniqueViolationError
SETEX processed:pay_001
```

Worker 2 reads Redis *before* Worker 1 has written the key. Both see nil, both charge Stripe. Worker 1 commits the DB row. Worker 2 hits the unique constraint — which is why you see both symptoms:

- **Customer charged twice, one payment recorded:** Worker 1 committed, Worker 2's DB insert failed after the Stripe charge already went through.
- **Customer charged twice, no payment recorded:** Both DB inserts failed (or one worker crashed), neither committed, but both Stripe charges landed.

The `UniqueViolationError` in the logs is the DB constraint doing its job — it is *preventing* duplicate DB records, but the Stripe charge has already happened by that point. The constraint is not preventing the duplicate charge, it is just evidence of it.

## Why the Tests Don't Catch This

The tests are single-threaded and sequential. `test_process_payment_skips_if_already_processed` pre-seeds `mock_redis.get` to return `"1"` before the call, which tests the happy-path guard but never exercises concurrent execution. No test spins up two workers against the same payment ID simultaneously. 28 tests passing means nothing here.

## What Needs to Change

### 1. Make the idempotency check atomic — use `SET NX` as a distributed lock

Replace the `GET` → (work) → `SET` pattern with a single atomic `SET NX` (set if not exists). The key must be written *before* the Stripe call, not after.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim: only one worker proceeds. SET NX returns True if the key
    # was created (this worker won the race), False if it already existed.
    # TTL is set atomically on the same operation — no gap.
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,  # TTL set atomically — no window where key exists without TTL
    )

    if not claimed:
        logger.info("Payment %s already claimed, skipping.", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )
    except Exception:
        # Stripe call failed — release the lock so this payment can be retried.
        redis_client.delete(f"processed:{payment_id}")
        raise

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()
```

Key points:
- `SET NX EX` is a single atomic Redis operation. No two workers can both get `True` for the same key.
- The key is written **before** the Stripe charge. The window is closed before the dangerous work begins.
- If Stripe fails, the key is deleted so the message can be retried. If the DB commit fails after a successful Stripe charge, the key stays set — that case needs a reconciliation path (see note 3 below).

### 2. Add Stripe idempotency keys as a second line of defense

Even with the Redis lock fixed, pass a Stripe idempotency key. This makes Stripe itself deduplicate if the same charge is somehow submitted twice. Use the `payment_id` since it is already your idempotency unit.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this
)
```

This costs nothing and provides defense-in-depth at the Stripe layer.

### 3. Handle the Stripe-succeeds / DB-fails case explicitly

If Stripe succeeds but the DB commit fails, the current code (even after the fix) will delete the Redis key on the exception path and allow a retry — which will hit Stripe again. With Stripe idempotency keys in place, Stripe will return the original charge object rather than creating a new one, so the retry is safe. Without them, this is another duplicate charge vector. Both fixes together close it.

### 4. Revert or re-examine the VisibilityTimeout change

Increasing VisibilityTimeout to 120s was the right instinct but it only delays redelivery, it does not prevent it. Any worker that crashes or is killed mid-flight will still redeliver the message after 120s. The fix above handles redelivery correctly, but the VisibilityTimeout should be set to a value that gives a worker enough time to complete under normal conditions without being so long that a crashed worker causes a 2-minute delay before retry. 30s was too short; 120s may be too long. Set it to something like 45–60s and monitor.

## Immediate Mitigation for the Call

Before the code fix is deployed, you can reduce the blast radius right now:

1. **Scale MaxInstances back to 3.** Fewer concurrent workers means fewer races. It does not eliminate the bug but reduces its frequency significantly.
2. **Purge or pause the queue** if you can afford to stop processing briefly while the fix is deployed.
3. **Do not roll back the VisibilityTimeout change** — reverting to 30s will make redelivery-driven duplicates worse, not better.

The code fix (SET NX + Stripe idempotency key) is a small, targeted change with no schema migration required. It can be deployed and verified quickly.
