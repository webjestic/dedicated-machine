# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is the entire payment processing time — Stripe API call plus database write. Any duplicate request that arrives during that window passes the check clean and proceeds to charge the card.

---

## What Is Actually Happening

### The Race Condition

```
Time ──────────────────────────────────────────────────────────────────►

Request 1:  [Redis GET: miss] ──── [Stripe charge] ── [DB insert] ── [Redis SET]
                                         ▲
Request 2:                    [Redis GET: miss] ── [Stripe charge] ── [DB insert] ── [Redis SET]
                                                                                          ▲
                                                                              UniqueViolationError
                                                                              (or second charge lands)
```

The Redis check in the gateway and the Redis write in the worker are separated by:
1. SQS publish latency
2. Queue polling interval (up to 1 second + long poll)
3. Worker scheduling and message dispatch
4. The Stripe API call itself (typically 200–800ms, can be seconds)
5. The database write

That is not a race window measured in milliseconds. It is a race window measured in **seconds to tens of seconds**. Under any meaningful load, concurrent requests will pass the check before the first one finishes.

### Why the Recent Changes Made It Worse

The two changes deployed six days ago directly widened and multiplied the window:

**`VisibilityTimeout` increased from 30s to 120s:**
The original timeout was already a symptom — payments were taking longer than 30 seconds. Increasing the timeout hides the latency problem but does not fix it. It means a message can be in-flight for up to 120 seconds before SQS considers it lost. During that entire window, the Redis key does not exist. Any retry or duplicate request from the client passes the gateway check and enqueues a second message.

**Worker `MaxInstances` increased from 3 to 12:**
This is the multiplier. With 3 workers, the probability of two workers pulling the same payment ID simultaneously was low. With 12 workers, you have four times as many concurrent consumers racing against the same unset Redis keys. The `UniqueViolationError` appearing in logs is the database constraint catching the cases where both workers complete — the customers seeing two charges are the cases where both workers complete successfully before either sets the Redis key.

### Why All 28 Tests Pass

The tests are sequential. `test_process_payment_skips_if_already_processed` correctly validates the guard, but it sets `mock_redis.get.return_value = "1"` before calling `process_payment` — it tests the case where the key already exists, not the case where two calls arrive simultaneously before either sets the key. There is no test that calls `process_payment` twice concurrently with a shared Redis mock that has realistic write timing. The test suite has zero coverage of the concurrent execution path.

## The Code, Annotated

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    payment_id = data["payment_id"]

    # CHECK happens here, in Service A
    if redis_client.get(f"processed:{payment_id}"):
        return jsonify({"status": "already_processed"}), 200

    # Message crosses queue boundary here.
    # The lock has NOT been acquired. The key does NOT exist.
    # Any request arriving after this line and before the worker
    # sets the key will pass the check above.
    sqs.send_message(QueueUrl=QUEUE_URL, MessageBody=json.dumps(data))

    return jsonify({"status": "accepted"}), 202
```

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Second check — also reads a key that does not exist yet
    # for any concurrent worker processing the same payment_id
    if redis_client.get(f"processed:{payment_id}"):
        return

    # ↓ WINDOW OPENS HERE
    charge = stripe.Charge.create(...)   # 200ms–2s, can spike higher
    db.execute("INSERT INTO payments ...")
    db.commit()
    # ↑ WINDOW CLOSES HERE — but only for this worker instance.
    # A second worker that passed the GET above is already past this point.

    # SET happens here, after the charge. Too late.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

The check and the act are in different services, separated by a queue, with unbounded processing time between them. This is the pattern.

## What Needs to Change

### Fix 1: Acquire the Lock Before Publishing (Required)

The Redis key must be set **before** the message is published to SQS, not after processing completes. Use `SET NX` (set-if-not-exists) as an atomic claim operation.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id.
    # SET NX returns True only if the key did not exist.
    # This is the lock. It must be acquired here, before the message
    # crosses the queue boundary.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,  # 24-hour expiry — covers already-processed case too
    )

    if not claimed:
        # Key already exists: either in-flight or already processed.
        # Both cases should return the same response to the caller.
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` is atomic at the Redis level. Two concurrent requests racing to set the same key will have one succeed and one fail — no gap, no window.

### Fix 2: Keep the Worker Guard, But Treat It as a Safety Net

The worker check is still correct to have — it defends against SQS redelivery (which is the scenario the `VisibilityTimeout` change was trying to address). But it cannot be the primary guard because it reads a key that does not exist until after the charge completes.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # This now checks the key set by the gateway before publishing.
    # The key exists from the moment the message was enqueued.
    # This guard handles SQS redelivery after a worker crash mid-processing.
    if redis_client.get(f"processing:{payment_id}") is None:
        # Key missing means it expired or was never set — log and skip.
        logger.warning("No processing key found for %s — skipping", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Do NOT delete the Redis key on failure.
        # Let it expire naturally or implement explicit retry logic.
        raise
```

### Fix 3: Add Stripe Idempotency Keys (Defense in Depth)

This does not fix the architectural race, but it prevents Stripe from executing a second charge if a duplicate request reaches their API. It costs nothing and should always be present.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their side
)
```

### Fix 4: Address the `VisibilityTimeout` Symptom Properly

The original 30-second timeout was firing because processing was taking longer than 30 seconds. The right fix is not to increase the timeout — it is to understand why processing takes that long and either optimize it or set the timeout to a value with margin above the p99 processing time, not the p50. If p99 is 90 seconds, set it to 150 seconds. Monitor `ApproximateAgeOfOldestMessage` to detect redelivery storms.

## Summary of Changes

| Location | Current Behavior | Required Change |
|---|---|---|
| `gateway/charge_handler.py` | `GET` then publish | `SET NX` then publish — atomic claim before queue |
| `worker/payment_worker.py` | `GET` then charge then `SET` | Check key set by gateway; remove post-charge `SET` |
| `stripe.Charge.create()` | No idempotency key | Pass `payment_id` as `idempotency_key` |
| Test suite | Sequential only | Add concurrent execution tests |

The invariant to enforce: **the idempotency key must exist before the message is visible to any worker.** Right now it does not exist until after the last worker finishes. That is the entire incident.
