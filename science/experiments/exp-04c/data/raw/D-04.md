## Root Cause

**The idempotency check is not atomic. There is a race condition between the Redis check and the Stripe charge, and the Redis key is only written after the charge succeeds. Under concurrent execution, multiple workers can pass the guard simultaneously and all charge Stripe before any of them writes the key.**

This was always a latent bug. The two recent changes made it a production incident:

- **MaxInstances 3 → 12**: Up to 12 workers now poll the same queue concurrently. The race window that was rarely hit with 3 workers is now hit regularly.
- **VisibilityTimeout 30s → 120s**: Intended to prevent redelivery. It does not — it just delays it. If a worker takes longer than 120s (or crashes), the message becomes visible again and a second worker picks it up. The Redis key may not be written yet.

---

## The Race Condition, Step by Step

```
Worker 1                          Worker 2
────────────────────────────────────────────────────────
redis.get("processed:pay_123")
→ None (not processed)
                                  redis.get("processed:pay_123")
                                  → None (not processed)

stripe.Charge.create(...)         stripe.Charge.create(...)
→ ch_aaa  ✓                       → ch_bbb  ✓  ← DUPLICATE CHARGE

db.INSERT payment_id=pay_123      db.INSERT payment_id=pay_123
db.commit() ✓                     → UniqueViolationError ✗

redis.setex("processed:pay_123")
                                  # Redis key never written
                                  # Payment not recorded in DB
                                  # Customer charged, no record exists
```

This explains both failure modes reported in the incident:

- **Customer charged twice**: Both workers complete the Stripe charge before either writes the Redis key.
- **Stripe charge failed, no DB record**: Worker 2's DB insert throws `UniqueViolationError`. The exception is caught in `poll_queue`, the message is *not* deleted from SQS, and the Redis key is never written. The customer was charged by Stripe but the system has no record of it.

The tests pass because they test each worker in isolation. No test exercises two concurrent calls to `process_payment` with the same `payment_id`.

---

## The Fix

The solution is to make the idempotency check and the claim atomic using a Redis `SET NX` (set-if-not-exists). This is the standard pattern for distributed locks. Whichever worker wins the `SET NX` owns the payment. All others bail out before touching Stripe.

### `worker/payment_worker.py`

```python
def process_payment(data: dict) -> None:
    """Charge the customer and record the transaction."""
    payment_id = data["payment_id"]

    # Atomically claim this payment_id.
    # SET NX returns True only for the first caller — all concurrent
    # workers racing on the same payment_id get False and exit.
    # TTL ensures the key expires even if the worker crashes mid-flight.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=120,        # Match VisibilityTimeout — prevents permanent lock on crash
    )
    if not claimed:
        logger.info("Payment %s already claimed, skipping.", payment_id)
        return

    try:
        # Check the durable completion record (survives Redis eviction/restart)
        if db.query_one(
            "SELECT 1 FROM payments WHERE payment_id = ?", (payment_id,)
        ):
            logger.info("Payment %s already in DB, skipping.", payment_id)
            return

        # Charge via Stripe
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        # Record in database
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Mark durably processed in Redis (fast path for future checks)
        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")

    except Exception:
        # Release the lock so the message can be retried
        redis_client.delete(f"processing:{payment_id}")
        raise
```

Three things changed and each one matters:

1. **`SET NX` as the gate** — replaces the non-atomic `GET` + later `SET` pattern. Only one worker can set the `processing:` key. Every other worker for that `payment_id` returns immediately without calling Stripe.

2. **DB check as the durable fallback** — Redis can be evicted, restarted, or flushed. If the `processing:` key is gone but the DB row exists, we still skip. This closes the gap where a worker crashes after the Stripe charge but before writing Redis.

3. **Lock release on failure** — if anything after the claim throws, we delete the lock key and re-raise. `poll_queue` will not delete the SQS message, it will become visible again after the VisibilityTimeout, and a fresh worker will retry cleanly.
