# Root Cause: Check-Then-Act Race Condition Across a Queue Boundary

## The Short Answer

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is not milliseconds — it is the entire Stripe API call plus the database write. Any duplicate request that arrives during that window passes the check clean and gets charged.

---

## What Is Actually Happening

### The Race in the Gateway (Service A)

```python
# Service A — charge_handler.py
if redis_client.get(f"processed:{payment_id}"):   # <── check
    return jsonify({"status": "already_processed"}), 200

sqs.send_message(...)                              # <── act: publishes to queue
                                                   #     Redis key NOT set here
```

The gateway checks Redis and, if the key is absent, publishes to SQS. It never sets anything in Redis. The key only gets set by the worker, after the charge completes. Under concurrent load — which you now have because `MaxInstances` went from 3 to 12 — two or more HTTP requests for the same `payment_id` can arrive before any worker finishes. Both pass the Redis check. Both publish. You now have two messages on the queue for the same payment.

### The Race in the Worker (Service B)

```python
# Service B — payment_worker.py
if redis_client.get(f"processed:{payment_id}"):   # <── check
    return

charge = stripe.Charge.create(...)                # <── Stripe API call (~300-800ms)
db.execute("INSERT INTO payments ...")            # <── DB write
db.commit()

redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")  # <── key set HERE
```

Even if only one message is on the queue, the worker guard has the same structural problem. With 12 worker instances polling concurrently, two workers can dequeue the same message before either deletes it — especially now that `VisibilityTimeout` is 120 seconds. Both pass the Redis check. Both call Stripe. The second one hits the `UniqueViolationError` on the DB insert, which is why you see that error in the logs. Whether the customer gets charged once or twice depends on which worker wins the race to Stripe first and whether the DB constraint fires before or after the second Stripe call.

### Why the Tests Pass

The tests are sequential. `test_process_payment_skips_if_already_processed` pre-seeds the mock Redis key before calling `process_payment`. It never tests two concurrent calls to `process_payment` with the same `payment_id` and an empty cache. The race condition is invisible to a single-threaded test suite. The CI green light is not a signal here.

### Why the Recent Changes Made This Worse

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | A message stays invisible to other workers for 2 minutes instead of 30 seconds. This was the right fix for timeouts, but it means a slow worker holds the message longer while other workers can still receive it if the visibility lapses or a second message was already enqueued. |
| `MaxInstances` 3 → 12 | Four times as many workers polling concurrently. The probability of two workers pulling the same message in the same visibility window went up significantly. More importantly, four times as many gateway instances means four times the concurrent HTTP requests that can pass the Redis check before any worker finishes. |

The duplicate rate scaled with the worker count because the race window is proportional to Stripe latency, and more concurrent workers means more opportunities to enter that window simultaneously.

---

## What Needs to Change

### Immediate Fix: Set the Lock Before the Message Crosses the Queue Boundary

The lock must be acquired in the gateway, atomically, before publishing. Use Redis `SET NX` (set if not exists):

```python
# gateway/charge_handler.py — replace the existing check
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. If the key already exists, another
    # request already owns it. NX = only set if key does not exist.
    # Use a distinct namespace so "in-flight" and "processed" are explicit.
    claimed = redis_client.set(
        f"inflight:{payment_id}",
        "1",
        nx=True,        # only set if not exists — atomic
        ex=DEDUP_TTL,   # expire so a failed payment doesn't block forever
    )

    if not claimed:
        # Key already existed — either in-flight or processed
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is atomic at the Redis level. Two concurrent requests racing on the same `payment_id` will result in exactly one of them getting `claimed = True`. The other gets `None` and returns immediately. The message is published exactly once.

### Worker Fix: Use the Database as the Final Idempotency Authority

Redis is a cache. It can be flushed, restarted, or evicted. The database `UNIQUE` constraint on `payment_id` is the only guarantee that survives a Redis failure. Restructure the worker to treat the DB constraint as the authoritative guard and Stripe's idempotency key as the charge-level guard:

```python
# worker/payment_worker.py
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Fast path: Redis check to avoid hitting the DB on every redelivery
    if redis_client.get(f"inflight:{payment_id}") is None:
        # Key was never set or has expired — check DB directly
        existing = db.query_one(
            "SELECT payment_id FROM payments WHERE payment_id = ?",
            (payment_id,)
        )
        if existing:
            logger.info("Payment %s already recorded, skipping", payment_id)
            return

    # Use Stripe's idempotency key so that even if we call Stripe twice
    # for the same payment_id, Stripe deduplicates on their side.
    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
        idempotency_key=payment_id,   # <── critical
    )

    try:
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()
    except UniqueViolationError:
        # Another worker beat us to the DB insert. The charge already exists.
        # The Stripe idempotency key means we got back the same charge object.
        # This is safe to swallow — log it and move on.
        logger.warning(
            "Duplicate DB insert blocked for payment_id=%s, charge=%s",
            payment_id, charge["id"]
        )
        return

    # Update Redis so the gateway fast-path stays warm
    redis_client.setex(f"inflight:{payment_id}", DEDUP_TTL, "1")
```

The Stripe `idempotency_key` parameter is the critical addition. If two workers do reach Stripe concurrently with the same `payment_id`, Stripe will return the same charge object for both calls rather than creating two charges. This is your last line of defense before money leaves the customer's account.

### The Layered Defense That Needs to Be in Place

```
Layer 1 — Gateway Redis SET NX
  Prevents duplicate messages from entering the queue.
  Atomic. Fails closed.

Layer 2 — Stripe idempotency_key
  Prevents duplicate charges even if two workers reach Stripe.
  Stripe deduplicates on their side for 24 hours.

Layer 3 — Database UNIQUE constraint
  Prevents duplicate records even if Stripe returns two different charge IDs.
  Catch UniqueViolationError explicitly; do not let it surface as an unhandled exception.
```

No single layer is sufficient. Redis can be unavailable. The queue can redeliver. Stripe idempotency keys cover the charge but not your local record. The DB constraint is durable but fires after the money has moved. You need all three.

## What to Tell the Head of Payments in 9 Minutes

> The deduplication key is set after the charge completes. The window between the Redis check and the Redis write is the full Stripe API call — several hundred milliseconds to over a second. Under the concurrent load from the Q1 promotion, multiple requests arrive during that window, all pass the check, and all get charged. The worker count increase made this worse by increasing the probability of two workers processing the same message simultaneously. The fix is to set the lock atomically in the gateway before the message enters the queue, and to pass Stripe's idempotency key on every charge call so that even if we reach Stripe twice, they deduplicate it. The database constraint is already catching some of these — that is the `UniqueViolationError` in the logs — but by the time the DB fires, the Stripe charge has already been created.
