# Root Cause Analysis — Duplicate Charges in `payment-processor`

## The Structural Cause

**The idempotency lock is set after the charge completes. The check happens before the message crosses the queue boundary. The window between those two events is the entire payment processing time — and any request that arrives during that window passes the check clean.**

This is not a bug introduced by the recent changes. The recent changes expanded the blast radius of a pre-existing race condition until it became visible in production.

---

## The Failure Path, Step by Step

```
T+0ms    Request 1 arrives at Gateway
         redis.get("processed:pay_001") → None  ✓ passes check
         sqs.send_message(pay_001)               message is now in queue

T+50ms   Request 2 arrives at Gateway (retry, client timeout, or duplicate submission)
         redis.get("processed:pay_001") → None  ✓ passes check — worker hasn't finished
         sqs.send_message(pay_001)               second message is now in queue

T+100ms  Worker instance A picks up message 1
         redis.get("processed:pay_001") → None  ✓ passes check
         stripe.Charge.create(...)               💳 CHARGE 1 HITS STRIPE

T+110ms  Worker instance B picks up message 2   (12 instances now available post-scaling)
         redis.get("processed:pay_001") → None  ✓ passes check — A hasn't set Redis yet
         stripe.Charge.create(...)               💳 CHARGE 2 HITS STRIPE

T+120ms  Worker A: db.execute(INSERT ...)        succeeds
         Worker A: redis.setex("processed:pay_001", ...)

T+121ms  Worker B: db.execute(INSERT ...)        💥 UniqueViolationError
         Worker B: redis.setex never reached
```

This explains both customer outcomes you're seeing:

- **Charged twice, payment recorded once:** Worker A completes fully. Worker B charges Stripe successfully but fails on the DB insert. The customer's card is hit twice; your system only has one record.
- **Charged twice, no payment recorded:** If Worker A also fails after the Stripe call (network blip, DB timeout), neither insert lands. Two charges, zero records.

The `UniqueViolationError` in your logs is not the cause — it is the symptom, and it is the only thing currently preventing some of these from being completely silent double charges.

---

## Why the Recent Changes Made This Worse

Neither change introduced the race. Both changes widened the window and increased the probability of concurrent workers hitting the same message.

**VisibilityTimeout 30s → 120s:**
The original 30s timeout was causing legitimate redeliveries because processing was taking longer than 30 seconds. The fix extended the timeout. But the underlying problem — that two messages for the same `payment_id` can exist in the queue simultaneously from the gateway — was untouched. The timeout change is correct for the redelivery problem. It does not protect against two gateway requests publishing two messages.

**MaxInstances 3 → 12:**
With 3 workers, the probability of two workers pulling concurrent messages for the same `payment_id` was low. With 12 workers, it is not. This is the amplifier. The race condition that existed at low probability at 3 instances became a near-certainty under Q1 promotion load at 12 instances.

---

## Why the Tests Don't Catch This

Every test in `test_payment_flow.py` is single-threaded and sequential. The mock Redis is a shared dict that updates synchronously. When `process_payment` calls `redis.setex`, it lands immediately, and any subsequent call to `redis.get` sees it.

```python
# This test passes. It proves nothing about concurrent execution.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

This test hardcodes the Redis state to already-processed before the call. It validates the guard branch works in isolation. It does not test whether two concurrent calls to `process_payment` with the same `payment_id` both pass the guard before either sets the key. That scenario is not represented anywhere in the test suite.

The CI is green because the tests were written to validate the happy path and the already-processed path. They were not written to validate the concurrent path.

---

## What Needs to Change

### Immediate Fix — Atomic Lock Before Queue Publish

The idempotency key must be set **before** the message is published to the queue, not after processing completes. Use a Redis `SET NX` (set if not exists) as an atomic gate in the gateway.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic: set the key only if it does not already exist.
    # This is a single Redis operation — no check-then-set race.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "queued",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not acquired:
        # Key already existed — either queued or processed
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker's Redis check becomes a secondary safety net for SQS redeliveries, not the primary guard. It stays. The `setex` at the end of `process_payment` should update the key value from `"queued"` to `"processed"` so you can distinguish state if you need to.

**Why `SET NX` is correct here:** `GET` followed by `SET` is two operations. Between them, a second request can execute its own `GET` and see the same empty result. `SET NX` is a single atomic operation. Redis processes it without interruption. Only one caller gets `acquired = True`.

### Secondary Fix — Stripe Idempotency Keys

This is defense in depth. Pass the `payment_id` as the Stripe idempotency key. If two charges reach Stripe for the same `payment_id`, Stripe will deduplicate them and return the same charge object for both calls.

```python
# worker/payment_worker.py

charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this
)
```

This does not fix the race condition. It caps the damage if the race condition fires anyway. You want both layers.

### Test Coverage That Would Have Caught This

```python
import threading

def test_concurrent_requests_do_not_duplicate_charge(client, mock_redis, mock_sqs):
    """
    Two simultaneous requests for the same payment_id must result in
    exactly one message published to the queue.
    """
    _, sent = mock_sqs
    results = []

    def make_request():
        rv = client.post(
            "/payments/charge",
            json={"payment_id": "pay_concurrent", "amount": 2500, "customer_id": "cust_abc"},
        )
        results.append(rv.status_code)

    t1 = threading.Thread(target=make_request)
    t2 = threading.Thread(target=make_request)
    t1.start()
    t2.start()
    t1.join()
    t2.join()

    assert len(sent) == 1, f"Expected 1 queued message, got {len(sent)}"
    assert sorted(results) == [200, 202]
```

Note: this test will also fail against the current gateway code with a real Redis instance, which is the point. It would have caught this before the deploy.

## Summary

| | Current behavior | Required behavior |
|---|---|---|
| **Lock acquisition** | After Stripe charge completes | Before SQS publish |
| **Lock operation** | `GET` then `SET` (two operations, race window) | `SET NX` (one atomic operation) |
| **Window size** | Full payment processing time (seconds to tens of seconds) | Zero — atomic |
| **Stripe deduplication** | None | Idempotency key on every charge call |
| **Concurrent test coverage** | None | Required before this ships |

The fix is small. The gate has to move from the end of the worker to the front of the gateway, and it has to be atomic. Everything else is defense in depth.
