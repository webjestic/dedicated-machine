# Incident Review: Root Cause of Duplicate Charges

## The Core Problem

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is not milliseconds — it is the entire Stripe API call plus the database write. Any duplicate request that arrives during that window passes the check clean.

This is the exact failure mode. Here is the sequence that produces a duplicate charge:

```
T+0ms    Worker 1: redis.get("processed:pay_001") → nil   ✓ passes guard
T+0ms    Worker 2: redis.get("processed:pay_001") → nil   ✓ passes guard  (race)
T+800ms  Worker 1: stripe.Charge.create(...)              → ch_aaa
T+900ms  Worker 2: stripe.Charge.create(...)              → ch_bbb         💳 DUPLICATE
T+950ms  Worker 1: db.execute(INSERT ...)                 → ok
T+960ms  Worker 2: db.execute(INSERT ...)                 → UniqueViolationError 💥
T+970ms  Worker 1: redis.setex("processed:pay_001", ...)  → too late
```

This explains both symptoms exactly:
- **Customer charged twice, no error visible:** Worker 1 succeeds end-to-end. Worker 2 charges Stripe successfully but then crashes on the DB insert. The charge exists in Stripe. It does not exist in your database. Support can see it. Your system cannot.
- **`UniqueViolationError` in logs:** The DB constraint fires on Worker 2's insert because Worker 1 already committed. The constraint is doing its job — but it fires *after* the money has already moved.

## Why the Recent Changes Made It Worse

The two infrastructure changes six days ago did not introduce the bug. They widened the window until the bug became unavoidable.

**`VisibilityTimeout` 30s → 120s:** A message that times out becomes visible again and gets picked up by a second worker. With 30s, a slow payment might escape redelivery. With 120s, you gave the system more runway — but you also gave two workers a longer window to hold the same message simultaneously during a visibility overlap at the boundary.

**`MaxInstances` 3 → 12:** You went from 3 workers racing to 12. The probability of two workers pulling the same payment ID during the unguarded window went up roughly 4x. Under Q1 promotion load, this became a near-certainty.

## Why All 28 Tests Pass

The tests are sequential. They call `process_payment()` directly, one at a time, in a single thread. The race condition requires two concurrent executions. The test suite has no concurrent test, no sleep, no thread or process boundary. It is structurally incapable of catching this failure.

```python
# This test proves the guard works when Redis is already set.
# It does not test what happens when two workers check before either sets.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

The test is correct. The scenario it tests never occurs in production. The scenario that does occur in production has no test.

## The Fix

The guard must be atomic and it must be acquired **before** the message crosses the queue boundary — meaning in the gateway, before `send_message`, using a Redis operation that checks and sets in one atomic step.

### Step 1: Acquire the lock in the gateway atomically

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # SET NX is atomic. Exactly one caller wins. All others are rejected here,
    # before the message ever reaches the queue. The lock is acquired before
    # the message crosses the queue boundary, not after processing completes.
    acquired = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not acquired:
        # Either currently processing or already processed.
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is a single atomic Redis command. There is no gap between the check and the set. Two concurrent requests for the same `payment_id` hit Redis simultaneously — one gets `True`, one gets `None`. The one that gets `None` never reaches the queue.

### Step 2: Keep the worker guard, but treat it as a safety net

The worker check is still correct to have. SQS delivers at-least-once. A message can be redelivered. The worker must handle that. But now the worker guard is a backstop against queue redelivery, not the primary deduplication mechanism.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard: catches SQS redelivery after a worker crash.
    # The primary guard is in the gateway. This should rarely fire.
    if redis_client.get(f"processing:{payment_id}") is None:
        # Key expired or was never set — something is wrong upstream.
        # Log and skip rather than charge blind.
        logger.warning("payment_id %s has no processing key; skipping", payment_id)
        return

    # Check for completed processing (set at end of successful run).
    if redis_client.get(f"processed:{payment_id}"):
        logger.info("payment_id %s already processed; skipping", payment_id)
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Mark fully processed. Gateway key remains as the dedup lock.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### Step 3: Add the test that would have caught this

```python
import threading

def test_concurrent_requests_charge_stripe_once(client, mock_redis, mock_sqs, mock_stripe):
    """
    Two requests for the same payment_id arrive concurrently.
    Stripe must be charged exactly once.
    """
    results = []

    def make_request():
        rv = client.post(
            "/payments/charge",
            json={"payment_id": "pay_concurrent", "amount": 5000, "customer_id": "cust_zzz"},
        )
        results.append(rv.status_code)

    t1 = threading.Thread(target=make_request)
    t2 = threading.Thread(target=make_request)
    t1.start()
    t2.start()
    t1.join()
    t2.join()

    assert sorted(results) == [200, 202]  # One accepted, one rejected
    _, sent = mock_sqs
    assert len(sent) == 1  # Exactly one message queued
```

## Summary

| | Before | After |
|---|---|---|
| **Where guard lives** | Worker (after queue) | Gateway (before queue) |
| **Guard operation** | `GET` then `SETEX` (two operations, non-atomic) | `SET NX EX` (one atomic operation) |
| **Race window** | Entire Stripe call + DB write (~1s+) | Zero — atomicity eliminates the window |
| **Concurrent requests** | Both pass the check, both publish, both charge | One acquires the lock, one is rejected at the gateway |
| **SQS redelivery** | Worker guard catches it | Worker guard still catches it as a backstop |

The lock has to be acquired before the message crosses the queue boundary. The check and the set have to be one atomic operation. Those two things together close the window.
