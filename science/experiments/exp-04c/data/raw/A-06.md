# Root Cause: The Idempotency Key Is Set After the Act, Not Before It

## What Is Actually Happening

There are two separate race windows in this system. Both are open simultaneously. Together they explain every symptom in the incident.

---

## Race Window 1: Gateway → Queue (The Primary Cause of Duplicate Charges)

This is the classic check-then-act race. Here is the exact sequence that produces a duplicate charge:

```
T=0ms   Request 1 arrives at gateway for pay_001
T=0ms   Gateway checks Redis: processed:pay_001 → nil  ✓ passes
T=1ms   Gateway publishes pay_001 to SQS
T=2ms   Request 2 arrives at gateway for pay_001 (retry, mobile client, user double-tap)
T=2ms   Gateway checks Redis: processed:pay_001 → nil  ✓ passes  ← STILL NIL
T=3ms   Gateway publishes pay_001 to SQS again
        ...
T=4000ms  Worker 1 picks up message 1, charges Stripe → ch_aaa
T=4001ms  Worker 2 picks up message 2, charges Stripe → ch_bbb  ← DUPLICATE CHARGE
T=4002ms  Worker 1 sets processed:pay_001 in Redis
T=4003ms  Worker 2 sets processed:pay_001 in Redis (too late)
```

The Redis key is not set until the worker finishes processing. The worker takes seconds to complete — Stripe API call, database write, then finally the Redis write. Every request that arrives at the gateway during that entire window passes the check and publishes a second message. The lock window is not milliseconds. It is the full payment processing latency.

**The gateway check is not a guard. It is a hint that only works after the fact.**

---

## Race Window 2: Worker → Worker (The Cause of the UniqueViolationError)

Even without the gateway race, the worker has the same pattern internally:

```python
# Worker process_payment():
if redis_client.get(f"processed:{payment_id}"):   # check
    return

charge = stripe.Charge.create(...)                 # act  ← Stripe charged here
db.execute("INSERT INTO payments ...")             # act
db.commit()
redis_client.setex(f"processed:{payment_id}", ...) # mark  ← set here, too late
```

With `MaxInstances` increased from 3 to 12, you now have up to 12 workers polling concurrently. With `VisibilityTimeout` at 120s, a message that is slow to process stays invisible to other workers — but a second *identical* message (from the gateway race above) is a different SQS message entirely. Both are visible. Both workers pass the Redis check. Both charge Stripe. The first worker to finish the DB insert succeeds. The second hits `UniqueViolationError` on `payments_payment_id_key`.

This explains the split symptom exactly:
- Customers who see **two charges**: both DB inserts succeeded (race resolved before constraint check, or constraint is on a different column path)
- Customers who see **one charge with no DB record**: the second Stripe charge succeeded but the DB insert threw `UniqueViolationError`, the exception was caught, the message was not deleted, and the worker logged the error and moved on — leaving a Stripe charge with no payment record

## Why the Tests Pass

The tests are structurally incapable of catching this. Look at what they actually test:

```python
# test_charge_returns_200_for_already_processed
mock_redis.get.return_value = "1"   # Redis is pre-seeded as if worker already finished
```

```python
# test_process_payment_skips_if_already_processed
mock_redis.get.return_value = "1"   # Same assumption
```

Every test that exercises the deduplication path pre-seeds Redis as already set. No test sends two concurrent requests to the gateway before the worker runs. No test runs two workers against the same message simultaneously. The tests validate the happy path of the guard, not the timing of when the guard is armed.

The 28 passing tests are testing a system that does not exist under concurrent load.

---

## The Two Deployment Changes Made This Worse

Both changes deployed 6 days ago directly widened the race windows:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | A slow payment holds its message invisible for 2 minutes. But duplicate messages from the gateway race are separate SQS messages — they are always visible. More time for concurrent workers to pick them up. |
| `MaxInstances` 3 → 12 | Four times as many workers polling concurrently. Four times the probability that two workers pick up duplicate messages in the same window. This is why the incident started 6 days ago and correlates exactly with the promotion traffic spike. |

## What Needs to Change

### Immediate Mitigation (Do This Now)

Set the idempotency key in the gateway *before* publishing, using a Redis atomic set-if-not-exists:

```python
# gateway/charge_handler.py
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic: set key only if it does not exist
    # Returns True if key was set (first request), False if it already existed
    acquired = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # SET if Not eXists — atomic in Redis
        ex=DEDUP_TTL,
    )

    if not acquired:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is atomic at the Redis level. Two concurrent requests racing on the same `payment_id` will both execute this call. Redis will grant it to exactly one. The other gets `None` back and returns 200. Only one message ever enters the queue.

### Correct Fix (Do This in the Next Sprint)

The gateway Redis key is a best-effort gate, not a guarantee. The worker must also be hardened:

**Option A — Database as the authority (simplest, most durable):**

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    try:
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()
    except UniqueViolationError:
        # Another worker already recorded this payment.
        # The Stripe charge above is a duplicate — initiate refund.
        logger.error(
            "Duplicate charge detected for payment_id=%s stripe_charge=%s — refunding",
            payment_id,
            charge["id"],
        )
        stripe.Refund.create(charge=charge["id"])
        return

    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

**Option B — Use Stripe's own idempotency keys:**

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,   # Stripe deduplicates on their end
)
```

Stripe will return the original charge object for any retry with the same idempotency key within 24 hours. This does not prevent the duplicate DB insert, but it prevents the duplicate card charge. Use both options together.

**Option C — SQS FIFO queue with message deduplication ID:**

Replace the standard SQS queue with a FIFO queue and set `MessageDeduplicationId=payment_id`. SQS will deduplicate messages with the same ID within a 5-minute window. This prevents duplicate messages from ever reaching the workers. This is the architectural fix that removes the problem at the queue boundary.

## Summary for the Emergency Call

> The idempotency key is set after the payment is processed, not before. The window between the Redis check and the Redis set is the full payment processing time — multiple seconds. Any duplicate request that arrives during that window passes the check, publishes a second message, and a second worker charges the card. Scaling to 12 worker instances made it four times more likely that two workers process duplicate messages concurrently. The `UniqueViolationError` in the logs is the database catching what the application layer missed — but by then Stripe has already charged the card.
>
> Immediate fix: change the gateway to use `SET NX` before publishing. This makes the deduplication atomic and moves the lock to before the queue boundary. Deploy that now. The refund logic and Stripe idempotency keys follow as hardening.
