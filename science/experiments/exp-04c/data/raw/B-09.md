# Root Cause Analysis — Duplicate Charges Incident

## The Structural Cause

**The idempotency lock is set after the charge completes. The check happens before the message crosses the queue boundary. The window between those two events is unbounded — it is the entire payment processing time. Any duplicate request that arrives during that window passes the check and gets enqueued.**

This is not a race condition in the traditional sense. It is a design gap. The check and the lock live in different services, with an async queue between them. There is no atomic "check and claim" operation. There is only "check, then eventually claim, with unbounded time in between."

---

## The Failure Path — Step by Step

Here is exactly what happens under concurrent load:

```
T+0ms    Request 1 arrives at Gateway for pay_001
T+0ms    Gateway checks Redis → key does not exist → passes
T+1ms    Gateway publishes pay_001 to SQS
T+1ms    Gateway returns 202

T+50ms   Request 2 arrives at Gateway for pay_001 (retry, duplicate tab, network hiccup)
T+50ms   Gateway checks Redis → key STILL does not exist → passes
         (Worker has not finished yet. Redis is not set yet.)
T+51ms   Gateway publishes pay_001 to SQS again
T+51ms   Gateway returns 202

T+2000ms Worker 1 picks up message 1, charges Stripe → ch_abc
T+2001ms Worker 1 writes to DB → succeeds
T+2001ms Worker 1 sets Redis key processed:pay_001

T+2100ms Worker 2 picks up message 2, checks Redis → key exists → returns early
         (Lucky path — if Worker 2 ran before Worker 1 finished, it charges again)
```

The lucky path above is what your tests exercise. The unlucky path — which is what production is hitting — looks like this:

```
T+0ms    Request 1 arrives → Redis miss → enqueued
T+50ms   Request 2 arrives → Redis miss → enqueued (Worker 1 still processing)

T+2000ms Worker 1 picks up message 1 → Redis miss → charges Stripe → ch_abc
T+2001ms Worker 2 picks up message 2 → Redis miss → charges Stripe → ch_xyz
         (Worker 1 has not set Redis yet, or Workers 1 and 2 ran concurrently)

T+2002ms Worker 1 writes DB → succeeds
T+2003ms Worker 2 writes DB → UniqueViolationError on payment_id_key
         Customer is charged twice. Second charge has no DB record.
```

This explains both symptoms you are seeing:
- Customers charged twice → both workers hit Stripe before either sets Redis
- `UniqueViolationError` with no payment recorded → second worker charged Stripe, then failed on the DB insert, then the Redis key was never set for that execution path

---

## Why the Recent Changes Made This Worse

The two infrastructure changes did not introduce the bug. The bug was always there. They widened the window and increased the number of concurrent workers racing through it.

**`VisibilityTimeout` increased from 30s to 120s:**
This was the right operational fix for timeout-driven redeliveries, but it has a side effect. When a message is in flight and a worker is slow, SQS will not redeliver it for 120 seconds. That is fine. But it does nothing to prevent a second *identical* message that was enqueued by the Gateway from being picked up immediately by a different worker. The VisibilityTimeout protects against SQS redelivering the *same* message. It does not protect against two *different* messages with the same payment ID both being in the queue simultaneously.

**`MaxInstances` increased from 3 to 12:**
More workers means more concurrent consumers. The probability that two workers pick up two enqueued copies of the same payment in the same processing window goes from low to near-certain under any load spike. You went from 3 workers racing to 12.

---

## Why the Tests Did Not Catch This

Look at `test_process_payment_skips_if_already_processed`. It sets `mock_redis.get.return_value = "1"` before calling `process_payment`. That simulates the state *after* a prior complete run. It does not simulate two concurrent calls to `process_payment` with the same payment ID where neither has set Redis yet.

Every test in this suite is single-threaded and sequential. The Gateway tests and Worker tests never run together. The queue is mocked out entirely — `send_message` appends to a list, `process_payment` is called directly. The async boundary, which is where the gap lives, is invisible to the test suite.

The tests are testing the components in isolation. The bug lives in the interaction between them under concurrency. This is a structural gap in test coverage, not a test quality problem. You cannot catch this with unit tests.

---

## What Needs to Change

### Immediate Fix — Atomic Claim in the Gateway

The lock must be acquired *before* the message is published, not after it is processed. The Gateway needs to perform an atomic SET NX (set if not exists) operation and only enqueue if it wins the set.

**`gateway/charge_handler.py` — replace the current check:**

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim: SET NX acquires the lock or tells us someone else did.
    # This key means "this payment has been accepted for processing."
    # The worker will overwrite it with "processed" state or we handle
    # that distinction — but nothing gets enqueued twice.
    claimed = redis_client.set(
        f"accepted:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist — atomic
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Either already accepted or already processed
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is atomic at the Redis level. Two concurrent requests for the same `payment_id` cannot both win it. One gets `True`, one gets `None`. Only the winner publishes to the queue. The message enters the queue exactly once.

The Worker's Redis check remains in place as a second line of defense against SQS redeliveries of the same message (which can still happen). That check is correct for that purpose. It just cannot be the primary deduplication mechanism because it is set too late.

### Secondary Fix — Stripe Idempotency Keys

This is a defense-in-depth measure. Even if a message somehow reaches two workers simultaneously, Stripe will not charge the card twice if you pass the same idempotency key.

**`worker/payment_worker.py` — pass idempotency key to Stripe:**

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this
)
```

This does not fix the root cause. It does mean that if two workers race through the gap, Stripe returns the same charge object to both rather than creating two charges. The DB insert will still fail with a `UniqueViolationError` on the second worker, but the customer is not charged twice. This is the difference between a data consistency problem and a customer harm problem.

### Operational Fix — Handle the UniqueViolationError Explicitly

Right now, when the DB insert fails, the exception propagates up, the message is not deleted from SQS, and the worker logs an error. The message will be redelivered after the VisibilityTimeout. Add explicit handling:

```python
from psycopg2 import errors as pg_errors  # or your DB driver equivalent

try:
    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()
except pg_errors.UniqueViolation:
    # Payment already recorded. This message is a duplicate.
    # Log it, but delete from queue — do not redeliver.
    logger.warning(
        "Duplicate message for payment_id=%s, stripe_charge_id=%s. Discarding.",
        payment_id,
        charge["id"],
    )
    db.rollback()
    return  # Caller will delete from SQS
```

Without this, a `UniqueViolationError` causes the message to sit in the queue and retry until it hits the dead-letter queue. With Stripe idempotency keys in place, those retries will return the same charge object and hit the same DB error in a loop. Handle it explicitly.

## Priority Order

1. **Deploy the `SET NX` change in the Gateway now.** This closes the gap. No new duplicate messages enter the queue after this deploys.
2. **Deploy Stripe idempotency keys now.** This protects against any messages already in the queue that were enqueued before fix 1 deployed.
3. **Deploy the `UniqueViolationError` handler.** This stops the retry loop for messages already in flight.
4. **Audit the current queue depth.** Check how many messages are sitting in `payment-processing` right now with payment IDs that already have a `processed:` key in Redis or a row in the DB. Those are duplicates waiting to fire. You may want to purge or manually inspect them before they are picked up.

---

## Summary

The check is in Service A. The lock is set in Service B. The queue between them means the window between check and lock is not milliseconds — it is the full processing time of every payment. Under concurrent load with 12 workers, that window is hit constantly. The `SET NX` in the Gateway collapses that window to zero by making the claim atomic before the message ever crosses the queue boundary. Everything else is defense in depth.
