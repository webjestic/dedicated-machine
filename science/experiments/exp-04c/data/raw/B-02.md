# Root Cause Analysis — Duplicate Charges Incident

## The Structural Cause

**The idempotency lock is set after the charge completes. The check happens before the message crosses the queue boundary. The window between those two events is unbounded — it is the entire payment processing time. Any request that arrives during that window passes the check and publishes a second message.**

This is not a bug introduced by the recent changes. This is a design flaw that the recent changes exposed by widening the window and increasing the number of concurrent workers racing through it.

---

## The Failure Path

Walk through exactly what happens under concurrent load:

```
T+0ms    Request 1 arrives at Gateway. Redis check: key absent. Message published to SQS.
T+50ms   Request 2 arrives at Gateway. Redis check: key still absent.
         Worker has the message but has not finished charging. Message published to SQS.
T+100ms  Worker 1 calls stripe.Charge.create() for Request 1. Charge succeeds.
T+110ms  Worker 2 calls stripe.Charge.create() for Request 2. Charge succeeds.
         Customer is now charged twice.
T+120ms  Worker 1 executes INSERT INTO payments. Succeeds.
T+125ms  Worker 2 executes INSERT INTO payments. UniqueViolationError.
         Payment record is NOT written. Redis key is NOT set.
T+130ms  Worker 1 calls redis.setex(). Key is now set.
```

This explains both symptoms exactly:

- **23 customer tickets with confirmed double charges** — two workers both reached `stripe.Charge.create()` before either set the Redis key.
- **"Failed second charge with no payment recorded"** — the `UniqueViolationError` on the database insert killed the exception handler before `redis.setex()` was called, so the key was never set, the message was not deleted from SQS, and the payment has no record in the system despite a successful Stripe charge.

---

## Why the Recent Changes Made This Worse

The flaw existed before. The recent changes widened the race window and increased the number of workers competing inside it.

**VisibilityTimeout increase (30s → 120s):** The original intent was correct — payments were timing out before completion. But the effect is that a message now stays invisible for up to 120 seconds. Under any retry or redelivery scenario, the window in which a duplicate can be published from the gateway is now 4x longer.

**Worker autoscaling (MaxInstances 3 → 12):** Four times as many workers polling the same queue. When two messages for the same `payment_id` are sitting in the queue — which happens the moment the gateway publishes a duplicate — the probability that two workers pick them up concurrently went from low to near-certain. This is what turned a latent flaw into 23 customer tickets.

Neither change is wrong on its own. Both are correct responses to real operational problems. They did not cause the bug. They removed the conditions that were accidentally suppressing it.

---

## Why the Tests Did Not Catch This

The tests are not wrong. They test exactly what the code does. The problem is what they cannot test.

```python
# test_process_payment_skips_if_already_processed
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

This test validates that the guard works when the key is already present. It does not and cannot test what happens when two calls to `process_payment()` execute concurrently against a real Redis instance with real network latency between the check and the set. The mock Redis is synchronous and in-process. There is no concept of time between `get` and `setex`. The race condition is physically impossible to reproduce in this test harness.

```python
# test_charge_returns_200_for_already_processed
def test_charge_returns_200_for_already_processed(client, mock_redis, mock_sqs):
    mock_redis.get.return_value = "1"
    ...
```

Same issue. This test confirms the gateway respects the key. It cannot test two gateway requests arriving before the worker sets the key, because the mock makes the key appear to already exist before the request is even made.

All 28 tests pass. The tests are correct. The architecture is not.

## The Fix

### Immediate Mitigation (deploy now)

Move the lock acquisition to the gateway, **before** the message is published. Use a Redis `SET NX` (set-if-not-exists) as an atomic claim. If the claim fails, the payment is already in-flight or completed — return `already_processed`.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id before publishing.
    # SET NX returns True only if the key did not exist.
    # TTL must be long enough to cover max processing time plus margin.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,  # 24 hours — covers completed payments too
    )

    if not claimed:
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` is atomic at the Redis level. Two concurrent requests cannot both receive `True`. One wins the claim and publishes. The other sees the key exists and returns immediately. The message is never published twice.

The worker-side Redis check can remain as a secondary defense against SQS redelivery, but it is no longer the primary guard.

### Required Follow-On (before this week is out)

**1. Fix the exception handling in the worker so a failed DB insert does not leave a charged customer with no record.**

The current code has `stripe.Charge.create()` succeeding and then `db.execute()` throwing `UniqueViolationError`. The exception propagates, `redis.setex()` is never called, the message is not deleted, and you have a customer who was charged with no record in your system. That is a reconciliation nightmare.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    if redis_client.get(f"processed:{payment_id}"):
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    try:
        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()
    except UniqueViolationError:
        # Another worker already recorded this payment. This is a duplicate
        # delivery. Log it for reconciliation and exit cleanly.
        logger.warning(
            "Duplicate delivery detected for payment_id=%s stripe_charge_id=%s — "
            "recording for reconciliation",
            payment_id,
            charge["id"],
        )
        # If we got here, a Stripe charge was created that may be a duplicate.
        # Flag it for the reconciliation job.
        flag_for_reconciliation(payment_id, charge["id"])
        return

    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

**2. Use Stripe's built-in idempotency keys.**

Stripe supports idempotency natively. Pass your `payment_id` as the idempotency key on every charge call. If two workers somehow both reach `stripe.Charge.create()` for the same payment, Stripe will return the same charge object for the second call instead of creating a new charge. This is a hard backstop at the network boundary.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their side
)
```

This does not replace the gateway-side fix. It is a defense-in-depth layer. The gateway fix prevents duplicate messages. The Stripe idempotency key prevents duplicate charges if a duplicate message somehow reaches the worker anyway.

**3. Run a reconciliation pass against Stripe for the affected window.**

Pull all Stripe charges from the last 6 days. For any `payment_id` that appears more than once in Stripe but only once in your database, you have a customer who was charged twice with only one record. Those need to be identified and refunded before customer support closes those 23 tickets. Do not assume the database is the source of truth here — the `UniqueViolationError` means some charges succeeded in Stripe but were never written to the database.

## Summary

| | Current behavior | Required behavior |
|---|---|---|
| **Where lock is acquired** | Worker, after charge completes | Gateway, before message is published |
| **Lock mechanism** | `GET` then `SET` (two operations, not atomic) | `SET NX EX` (single atomic operation) |
| **Window of vulnerability** | Entire payment processing time (seconds to minutes) | Zero — atomic claim before publish |
| **Stripe idempotency** | Not used | Pass `payment_id` as `idempotency_key` |
| **DB error handling** | Exception silently drops the Redis set | Caught, logged, flagged for reconciliation |

The lock has to be acquired before the message crosses the queue boundary, not after. That is the only change that closes this window.
