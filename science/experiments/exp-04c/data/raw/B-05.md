# Root Cause Analysis — Duplicate Charges in `payment-processor`

## The Structural Defect

The deduplication lock is set **after** the charge completes. The check happens **before** the message crosses the queue boundary. Between those two events — which are separated by the full duration of Stripe API latency plus DB write time — the payment ID is invisible to any guard in the system. Any request that arrives during that window passes clean.

This is not a bug introduced by the recent changes. The recent changes **exposed** it by widening the window and increasing the number of concurrent workers racing through it.

---

## The Failure Path, Traced Exactly

### Vector 1: Concurrent HTTP Requests (Gateway Race)

```
T+0ms    Request A arrives at gateway. redis.get("processed:pay_001") → None. Publishes to SQS.
T+50ms   Request B arrives at gateway. redis.get("processed:pay_001") → None. Publishes to SQS.
         # Redis key does not exist yet. It won't exist until the worker finishes.
         # Both messages are now sitting in the queue.
T+200ms  Worker 1 picks up message A. redis.get("processed:pay_001") → None. Proceeds.
T+210ms  Worker 2 picks up message B. redis.get("processed:pay_001") → None. Proceeds.
         # Both workers pass the guard simultaneously.
T+800ms  Worker 1: stripe.Charge.create() → ch_abc. DB INSERT. redis.setex().
T+820ms  Worker 2: stripe.Charge.create() → ch_xyz. DB INSERT → UniqueViolationError.
         # Card charged twice. One charge has no DB record.
```

This is the path producing your two symptom classes simultaneously: customers charged twice *and* customers with a charge but no payment record. The `UniqueViolationError` you're seeing in the logs is the DB constraint catching Worker 2 **after** Stripe has already taken the money.

### Vector 2: SQS Redelivery (Worker Race)

This is the second path, made significantly worse by the `VisibilityTimeout` increase.

```
T+0s     Worker 1 picks up message for pay_001. Starts processing.
T+30s    [OLD BEHAVIOR] VisibilityTimeout expires. SQS redelivers. Worker 1 still running.
T+120s   [NEW BEHAVIOR] VisibilityTimeout expires. SQS redelivers. Worker 1 still running.
         # The timeout increase was correct in intent — payments legitimately take time.
         # But it means the redelivery window is now 120s of concurrent exposure
         # instead of 30s, and with 12 workers instead of 3, the probability of
         # a second worker picking up the redelivered message before Worker 1
         # sets the Redis key is dramatically higher.
```

The `MaxInstances` increase from 3 to 12 multiplied the number of workers racing through the same unguarded window. These two changes together didn't create the defect — they turned a low-probability race into a near-certainty under Q1 promotion load.

## Why the Tests Don't Catch This

Look at `test_process_payment_sets_redis_key_after_processing`. The test **asserts** that `setex` is called after processing. It is literally testing and verifying the broken behavior. The test suite has 28 passing tests that confirm the system works correctly in sequential, single-threaded execution. There is no test that calls `process_payment` concurrently, and no test that calls the gateway endpoint twice before the worker runs. The tests are correct for what they test. They test the wrong thing.

## What Needs to Change

### Immediate Fix: Set the Lock Before the Queue Boundary

The idempotency key must be claimed in the gateway, atomically, before the message is published. Use Redis `SET NX` (set-if-not-exists) as the gate. If the key cannot be claimed, the payment is already in flight or already processed — reject it.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim: SET processed:{payment_id} "pending" NX EX 86400
    # Returns True if key was set (we own it), False if it already existed.
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,        # Only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Key already exists — either "pending" (in flight) or "1" (complete).
        # Either way, do not publish again.
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The worker's Redis check remains as a secondary guard against SQS redelivery, but it now checks for either state:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Guard against redelivered messages.
    # Key will be "pending" (set by gateway) or "1" (set by a prior worker run).
    # In either case, if the key is "1", we're done.
    if redis_client.get(f"processed:{payment_id}") == "1":
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Overwrite "pending" with "1" to mark fully complete.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

The lock is now set **before** the message is published. The window between check and act is reduced to the atomic duration of a single Redis `SET NX` command. Concurrent gateway requests for the same `payment_id` will race on that command — exactly one will win, the rest will see the key and return `already_processed`.

### Secondary Fix: Stripe Idempotency Keys

This is a defense-in-depth measure, not a substitute for the above. Pass the `payment_id` as Stripe's idempotency key. If two charges somehow reach Stripe for the same key, Stripe will deduplicate them and return the same charge object.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Add this
)
```

This does not fix the root cause. It is a backstop. The `UniqueViolationError` path — where the second worker charges the card but cannot write to the DB — still results in a customer being charged with no record in your system. Stripe idempotency keys prevent the double charge at the Stripe layer, but your DB record is still missing. Fix the lock first.

### Tertiary Fix: SQS Message Deduplication

If the queue is FIFO-capable, enable SQS content-based deduplication with `payment_id` as the `MessageGroupId`. This prevents duplicate messages from entering the queue at all for a 5-minute deduplication window. This is a queue-level guard, not a substitute for application-level idempotency.

## What the Recent Changes Did

To be precise about this, because the postmortem will ask:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | Correct fix for the timeout problem. Side effect: redelivered messages now have a 120s window to race a still-running worker instead of 30s. |
| `MaxInstances` 3 → 12 | Correct fix for throughput. Side effect: 4× more workers available to pick up a redelivered message simultaneously. |

Neither change was wrong in isolation. Both were reasonable operational responses to real problems. Together, under Q1 load, they increased the concurrency pressure on a pre-existing race condition past the threshold where it became visible. The defect existed before these changes. These changes made it inevitable.

---

## Summary for the Team

**Root cause:** The idempotency lock is set after the charge completes, in a different service, across a queue boundary. The unguarded window is the full payment processing time — currently measured in seconds to tens of seconds. Any duplicate request arriving during that window passes every check and charges the card.

**Immediate action:** Deploy the `SET NX` change to the gateway. This closes the window to a single atomic Redis operation. The worker guard remains as a secondary defense against SQS redelivery.

**Do not roll back** the `VisibilityTimeout` or `MaxInstances` changes. They are solving real problems. Fix the lock.
