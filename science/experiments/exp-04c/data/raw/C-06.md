# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The check-then-act window is not milliseconds — it is the entire Stripe API call plus the database write. Every request that arrives during that window passes the Redis check clean and gets charged.

---

## What Is Actually Happening

### The Race Condition

```
Time ──────────────────────────────────────────────────────────────────►

Request 1 (Gateway):  [Redis GET: miss] ──► [SQS publish] ──────────────────────────────────────────────────────────────────────────────────────────────►
                                                           │
Request 2 (Gateway):                       [Redis GET: miss] ──► [SQS publish] ──────────────────────────────────────────────────────────────────────────►
                                                                               │
Worker 1:                                                          [Redis GET: miss] ──► [stripe.Charge.create ~800ms] ──► [db.INSERT] ──► [Redis SET] ✓
                                                                               │
Worker 2:                                                          [Redis GET: miss] ──► [stripe.Charge.create ~800ms] ──► [db.INSERT] ──► 💥 UniqueViolationError
                                                                                                                                           (Redis SET never reached)
```

The Redis key is not set until **after** `stripe.Charge.create` returns and the database write commits. The window between the check and the set is not a race condition measured in microseconds — it is the full duration of an external HTTP call to Stripe plus a database round-trip. Under your Q1 load, with 12 worker instances polling the same queue, multiple workers can dequeue and begin processing the same `payment_id` before any of them finishes.

### Why the Gateway Check Does Not Help

The gateway check is checking for a key that the worker has not set yet. A payment that is *in flight* — dequeued, being processed — is indistinguishable from a payment that has never been seen. Any retry, any duplicate API call from the client, any redelivered SQS message that arrives while Worker 1 is waiting on Stripe will pass the gateway check and enter the queue.

### Why the Worker Check Does Not Help Either

The worker check has the same problem at a smaller scale. Two workers can receive the same message (SQS delivers at-least-once), both call `redis_client.get(f"processed:{payment_id}")` before either calls `setex`, both get a miss, and both proceed to charge.

### Why the Recent Changes Made This Worse

The two deployment changes six days ago directly expanded the race window:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | SQS will now redeliver a message after 120s instead of 30s. This does not prevent redelivery — it delays it. Any payment that takes longer than 120s still gets redelivered to a second worker. |
| `MaxInstances` 3 → 12 | Four times as many workers polling the same queue. The probability that two workers dequeue the same message in the same window increased by roughly 4×. |

### Why the Tests Pass

The tests are synchronous and single-threaded. `process_payment` is called directly, not through SQS. There is no concurrent second call arriving while the first is in flight. The mock Redis is a plain dict with no locking. The tests verify that the code *does the right thing when it runs alone*. They say nothing about what happens when it runs twice at the same time.

```python
# This test proves the guard works when Redis already has the key.
# It does not test what happens when two calls arrive before either sets the key.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

## The Fix

### Principle

The lock must be acquired **before the message crosses the queue boundary**, not after processing completes. The idempotency key must represent *intent to process*, not *completion of processing*.

### Step 1: Claim the lock in the gateway with `SET NX` before publishing

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. SET NX returns True only if the key
    # did not exist. If two requests arrive concurrently, exactly one wins.
    # The TTL must be long enough to cover the full processing time plus margin.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if Not eXists — this is the atomic part
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Either already processing or already processed. Either way, do not
        # publish a second message.
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

`SET NX` is atomic at the Redis level. Two concurrent requests racing on the same `payment_id` will serialize here. One sets the key and publishes. The other sees the key already exists and returns early. The message enters the queue exactly once.

### Step 2: Keep the worker guard, but treat it as a safety net, not the primary lock

The worker check is still necessary because SQS is at-least-once. A message can be redelivered even if the gateway never sends it twice. But the worker check should now be checking the same key the gateway set, not waiting for a key that only appears after processing.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # The gateway already set processing:{payment_id} before publishing.
    # This check catches SQS redeliveries of messages that completed successfully.
    if redis_client.get(f"completed:{payment_id}"):
        logger.info("Payment %s already completed, skipping redelivery", payment_id)
        return

    # Charge via Stripe
    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    # Record in database
    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Mark completed. This is separate from the processing lock the gateway set.
    # The processing lock prevents duplicate publishes.
    # This key prevents duplicate processing of redelivered messages.
    redis_client.setex(f"completed:{payment_id}", DEDUP_TTL, "1")
```

### Step 3: Use Stripe's idempotency key as a hard backstop

Stripe supports idempotency keys natively. Pass `payment_id` as the idempotency key on every charge call. If your application logic fails and two charge requests reach Stripe, Stripe will deduplicate them and return the original charge result for the second call. This does not replace the fix above — it is a last line of defense.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Add this
)
```

### Step 4: Handle the failure case in the gateway

The current gateway has no handling for the case where the message is published but the worker fails before completing. The `processing` key is set, the worker crashes, and the payment never completes — but the key blocks any retry. You need a way to distinguish *in progress* from *completed*.

```python
# Two keys, two states:
# processing:{payment_id}  — set by gateway before publish, means "message is in the queue"
# completed:{payment_id}   — set by worker after commit, means "charge succeeded"

# On retry/resubmit endpoint, check completed first, then processing:
completed = redis_client.get(f"completed:{payment_id}")
if completed:
    return jsonify({"status": "already_processed"}), 200

in_flight = redis_client.get(f"processing:{payment_id}")
if in_flight:
    return jsonify({"status": "processing"}), 202  # Tell client to poll for result
```

## What to Do Right Now

1. **Deploy the `SET NX` change to the gateway immediately.** This is the fix for the primary race. It is a small change with no schema migration and no worker restart required.

2. **Add the Stripe idempotency key immediately.** Zero risk, immediate protection at the Stripe layer for anything that slips through.

3. **Do not reduce `MaxInstances` as a workaround.** That treats the symptom. The race exists at any concurrency level — it is just less likely to trigger with fewer workers.

4. **Do not rely on the `UniqueViolationError` as a deduplication mechanism.** The `UniqueViolationError` means the charge already went to Stripe. The customer has already been charged. The database constraint is catching the duplicate *after the money has moved*. It is a data integrity guard, not an idempotency guard.

5. **Audit the 23 open tickets.** For each one, check whether the second Stripe charge succeeded or failed. If it succeeded, the customer was charged twice and needs a refund. If it failed with a `UniqueViolationError` before the Stripe call, no refund is needed but the payment record may be missing and needs to be reconciled manually.
