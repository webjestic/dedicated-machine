# Root Cause: Check-Then-Act Race Condition With No Distributed Lock

## The Short Answer

The Redis key is set **after** the charge completes. The window between the check and the set is the entire Stripe API call plus the database write — easily 1–3 seconds under normal conditions, longer under load. Any duplicate request that arrives during that window passes the check clean. Both workers charge the card.

The `VisibilityTimeout` increase and the autoscaling change didn't introduce the bug. They widened the window and multiplied the concurrent workers. A latent race condition became a production incident.

---

## What Is Actually Happening

### The Race in Service A (Gateway)

```python
# Both requests execute this line. Both get None. Both proceed.
if redis_client.get(f"processed:{payment_id}"):
    return ...

# Both publish to SQS.
sqs.send_message(...)
```

Two HTTP requests for the same `payment_id` arrive within milliseconds of each other. There is no lock, no atomic reservation — just a read. Both reads return `None`. Both messages land on the queue.

### The Race in Service B (Worker)

Even if only one message is on the queue, the `VisibilityTimeout` change made this worse in a second way. Before the change: a slow payment hit 30s, SQS redelivered it, a second worker picked it up. After the change: 120s timeout means redelivery is less frequent — but with `MaxInstances` raised from 3 to 12, you now have up to 12 workers polling concurrently. Two workers can receive the same message in separate polling windows if the first worker's visibility timeout expires before it finishes (or if SQS delivers it twice during a brief consistency window). Both workers hit the Redis check before either sets the key:

```python
# Worker 1 reads: None. Proceeds to stripe.Charge.create().
# Worker 2 reads: None. Proceeds to stripe.Charge.create().
if redis_client.get(f"processed:{payment_id}"):
    return

charge = stripe.Charge.create(...)   # <-- both workers are here simultaneously

db.execute("INSERT INTO payments ...")  # Worker 2 hits UniqueViolationError here
db.commit()

redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")  # Too late
```

This explains both symptoms exactly:
- **Customer charged twice, both recorded:** Two workers completed before the DB constraint fired
- **Second charge failed, nothing recorded:** `UniqueViolationError` on the DB insert — but Stripe already charged the card. The exception is caught by the outer `try/except` in `poll_queue`, logged, and the message is **not deleted**, so it will be retried

### Why the Tests Pass

The tests are single-threaded and synchronous. `process_payment` is called directly — no queue, no concurrency, no network latency. The mock Redis is a plain dict with no race conditions possible. The tests prove the happy path works in isolation. They cannot prove anything about concurrent execution. This is the exact failure mode: the guard looks correct in serial testing and fails under concurrent load.

---

## The Fix

### Immediate Mitigation (Do This Now)

Replace the read-then-write pattern in the gateway with an atomic `SET NX` (set if not exists). This makes the reservation and the guard a single atomic operation:

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic: sets the key only if it does not exist.
    # Returns True if we acquired it, None/False if it already existed.
    acquired = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=300,        # 5-minute TTL — must exceed your p99 processing time
    )

    if not acquired:
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The lock is acquired **before** the message crosses the queue boundary. The window is closed.

The worker still needs its own guard for SQS redelivery. Same pattern:

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic lock acquisition in the worker as a second line of defense.
    acquired = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,
        ex=300,
    )
    if not acquired:
        logger.info("Payment %s already in-flight or processed, skipping", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

        # Permanent record of completion, separate from the in-flight lock.
        redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")

    except Exception:
        # Release the lock on failure so the message can be retried.
        redis_client.delete(f"processing:{payment_id}")
        raise
```

Two keys, two purposes:
- `processing:{payment_id}` — in-flight lock, acquired before work starts, released on failure
- `processed:{payment_id}` — permanent completion record, set only after successful commit

### Longer Term

| Problem | Fix |
|---|---|
| Stripe charges without a DB record on `UniqueViolationError` | Use Stripe idempotency keys: `stripe.Charge.create(..., idempotency_key=payment_id)`. A retry with the same key returns the original charge object instead of creating a new one. |
| Worker catches all exceptions and continues | Distinguish retryable errors (network timeout) from non-retryable ones (duplicate key). Dead-letter queue for the latter. |
| Tests have no concurrency coverage | Add a test that calls `process_payment` from two threads simultaneously with the same `payment_id` and asserts Stripe is called exactly once. |
| Redis is a single point of failure for the lock | Acceptable for now. If Redis goes down, fail the gateway request with 503 rather than bypassing the lock. |

---

## Summary for the Emergency Call

The deduplication check and the deduplication write are in different services with unbounded processing time between them. The lock was never actually held — it was checked, then set later. The `VisibilityTimeout` and autoscaling changes exposed a race condition that was always present. The fix is one line in the gateway: `SET NX` before publishing. The lock has to be acquired before the message crosses the queue boundary.
