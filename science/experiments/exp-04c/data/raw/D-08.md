## Root Cause

**The idempotency check and the Stripe charge are not atomic.** There is a race condition window between the Redis `GET` and the Redis `SETEX` at the end of `process_payment`. With 12 worker instances now running concurrently (up from 3), multiple workers can pass the guard check simultaneously for the same `payment_id` before any of them has written the processed flag.

```
Worker 1: GET processed:pay_001  → nil   ✓ proceeds
Worker 2: GET processed:pay_001  → nil   ✓ proceeds   ← both pass the guard
Worker 1: stripe.Charge.create() → ch_aaa
Worker 2: stripe.Charge.create() → ch_bbb             ← duplicate charge
Worker 1: INSERT INTO payments ...
Worker 2: INSERT INTO payments ... → UniqueViolationError (DB catches it)
Worker 1: SETEX processed:pay_001
Worker 2: SETEX processed:pay_001
```

This explains both symptoms exactly:

- **Customer charged twice, nothing recorded in the system:** Worker 2's `INSERT` throws `UniqueViolationError`. The exception propagates up through `process_payment`, the message is *not* deleted from SQS, and the error is logged — but the Stripe charge already happened. Money taken, no DB record.
- **Customer charged twice, both recorded:** Possible if the DB constraint wasn't the first collision point (e.g., two different `payment_id` values for the same logical payment, or a timing window before the constraint fires).

The increased `MaxInstances` (3 → 12) is what made this race condition fire at scale. The `VisibilityTimeout` increase (30s → 120s) was the right call for slow payments, but it masked the underlying problem rather than fixing it — messages were still being redelivered, just less frequently.

The tests pass because they run sequentially in a single process against a shared mock. The race condition is invisible to them.

## The Broken Code

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # CHECK  ← Worker 1 and Worker 2 both pass this
    if redis_client.get(f"processed:{payment_id}"):
        return

    # ACT  ← Both workers reach here before either sets the flag
    charge = stripe.Charge.create(...)

    db.execute("INSERT INTO payments ...")
    db.commit()

    # SET  ← Too late. The damage is done.
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

---

## The Fix

Use `SET NX` (set-if-not-exists) as an atomic claim lock *before* touching Stripe. This collapses the check-then-act into a single atomic operation.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Atomic claim: only one worker wins this for a given payment_id.
    # SET NX returns True if the key was set, False if it already existed.
    acquired = redis_client.set(
        f"processed:{payment_id}",
        "1",
        nx=True,       # Only set if key does not exist
        ex=DEDUP_TTL,  # TTL in seconds (atomic with the SET)
    )

    if not acquired:
        # Another worker already claimed this payment_id
        logger.info("Skipping already-claimed payment %s", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Processing failed after we claimed the lock.
        # Release it so the message can be retried by another worker.
        redis_client.delete(f"processed:{payment_id}")
        raise
```

Key points on this fix:

1. **`SET NX EX` is atomic at the Redis level.** No two workers can both get `True` back for the same key. This eliminates the race window entirely.
2. **The lock is released on failure.** If Stripe or the DB call fails after we claimed the lock, we delete the key so the SQS message can be retried. Without this, a failed payment would be permanently skipped.
3. **The `UniqueViolationError` becomes a last-resort safety net**, not the primary guard. It should never fire in normal operation after this fix, but keep the DB constraint — it's a good backstop.

## Secondary Hardening (Do After the Incident)

These don't fix the root cause but close adjacent gaps:

**Use a Stripe idempotency key.** Even with the Redis fix, if Redis itself has a failure between the `SET NX` and the Stripe call, you could still get a duplicate charge on retry. Pass `idempotency_key=payment_id` to Stripe so their side deduplicates as well:

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their end
)
```

**Check the DB before charging on retry.** If a worker crashes after the Stripe charge but before the DB write, the next retry will hit Stripe again. Querying the DB first adds another layer:

```python
existing = db.query_one(
    "SELECT stripe_charge_id FROM payments WHERE payment_id = ?",
    (payment_id,)
)
if existing:
    logger.info("Payment %s already in DB, skipping", payment_id)
    return
```

**Fix the tests.** The test suite gives false confidence here because it doesn't test concurrent execution. Add a concurrency test:

```python
def test_no_duplicate_charge_under_concurrent_workers(mock_stripe):
    from concurrent.futures import ThreadPoolExecutor
    payment = {"payment_id": "pay_concurrent", "amount": 500, "customer_id": "cust_x"}

    with ThreadPoolExecutor(max_workers=10) as pool:
        futures = [pool.submit(process_payment, payment) for _ in range(10)]
        for f in futures:
            f.result()

    # Stripe must be called exactly once regardless of concurrency
    assert mock_stripe.Charge.create.call_count == 1
```

| | Detail |
|---|---|
| **Root cause** | Race condition in `process_payment`: Redis GET and SETEX are not atomic, so concurrent workers all pass the idempotency check before any sets the flag |
| **Trigger** | Worker autoscale from 3 → 12 instances increased concurrent processing of the same message enough to make the race fire regularly |
| **Fix** | Replace `GET` + `SETEX` with atomic `SET NX EX` before calling Stripe |
