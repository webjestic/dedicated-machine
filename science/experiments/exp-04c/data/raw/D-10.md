## Root Cause

**The idempotency check and the Stripe charge are not atomic.** There is a race condition in `process_payment` that allows multiple workers to charge the same payment concurrently.

Here's the exact sequence that produces a duplicate charge:

```
Worker 1                              Worker 2
────────────────────────────────────────────────────────
redis.get("processed:pay_001") → None
                                      redis.get("processed:pay_001") → None
stripe.Charge.create(...)  ← charge 1
                                      stripe.Charge.create(...)  ← charge 2 (DUPLICATE)
db.INSERT payments ...
redis.setex("processed:pay_001")
                                      db.INSERT payments ...  ← UniqueViolationError
                                      redis.setex never reached
```

Both workers pass the Redis guard before either one sets the key. The customer gets charged twice. On the second worker the DB insert fails with `UniqueViolationError`, the exception is caught and logged, the message is **not deleted from SQS**, and no payment record is written — which explains the two distinct failure modes you're seeing:

- Customer charged twice, both charges recorded (both workers completed fully before the race resolved)
- Customer charged twice, only one charge recorded (second worker's DB insert threw, leaving the Stripe charge orphaned)

### Why it got worse six days ago

The two config changes combined to make this far more likely:

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | A message stays invisible longer, so the original trigger for redelivery is reduced — but it doesn't eliminate concurrent processing |
| `MaxInstances` 3 → 12 | Four times as many workers polling the same queue. Under the Q1 promotion load, multiple instances regularly receive and begin processing the same message within the same visibility window |

The race existed before, but with 3 workers it was rare. With 12 workers under high load it's happening constantly.

### Why the tests don't catch it

The tests are single-threaded and sequential. `mock_redis` is a shared in-memory dict, so the first call to `setex` is immediately visible to any subsequent call in the same thread. The race only manifests under concurrent execution, which no test exercises.

## The Fix

The guard needs to be an atomic **claim**, not a read. Replace the read-then-write with a Redis `SET NX` (set-if-not-exists) so only one worker can claim a payment_id and proceed. All others bail out immediately.

```python
# worker/payment_worker.py

def process_payment(data: dict) -> None:
    """Charge the customer and record the transaction."""
    payment_id = data["payment_id"]

    # Atomically claim this payment_id. Only one worker succeeds.
    # NX = only set if key does not exist.
    # EX = TTL in seconds.
    # Returns True if the key was set (this worker owns it), None if it already existed.
    claimed = redis_client.set(
        f"processed:{payment_id}",
        "1",
        nx=True,
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Another worker already claimed or completed this payment.
        logger.info("Payment %s already claimed or processed, skipping.", payment_id)
        return

    try:
        charge = stripe.Charge.create(
            amount=data["amount"],
            currency=data.get("currency", "usd"),
            customer=data["customer_id"],
        )

        db.execute(
            "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
            " VALUES (?, ?, ?, ?)",
            (payment_id, data["customer_id"], data["amount"], charge["id"]),
        )
        db.commit()

    except Exception:
        # Processing failed after we claimed the key. Release the claim
        # so the message can be retried when SQS redelivers it.
        redis_client.delete(f"processed:{payment_id}")
        raise
```

### Key points on this implementation

**The claim happens before the Stripe call.** Redis `SET NX` is atomic at the Redis level — only one caller gets `True`. Every other concurrent worker gets `None` and returns. The race window is eliminated.

**The claim is released on failure.** If the Stripe call or the DB write throws, we delete the key and re-raise. SQS will redeliver the message after the visibility timeout and a clean worker will retry it. Without the `delete` on failure, a transient error would permanently block the payment.

**The `poll_queue` exception handler stays intact.** When `process_payment` raises, the message is not deleted from SQS. That's correct — it should be retried.

## Additional Hardening (post-incident)

These don't need to block the hotfix but should be tracked:

1. **Stripe idempotency keys.** Pass `idempotency_key=payment_id` to `stripe.Charge.create`. This is a server-side backstop — if a duplicate request reaches Stripe anyway, Stripe deduplicates it and returns the original charge instead of creating a new one. Defense in depth.

2. **Unique constraint is working correctly.** The `UniqueViolationError` on `payments_payment_id_key` is the DB doing its job. Keep that constraint — it's catching the cases that slip through.

3. **Dead-letter queue.** Messages that fail repeatedly should route to a DLQ rather than retrying forever. This limits blast radius for future incidents.

4. **Concurrent test coverage.** Add a test that calls `process_payment` from two threads simultaneously with the same `payment_id` and asserts `stripe.Charge.create` was called exactly once.

## Summary for the call

> The duplicate charges are caused by a race condition in the payment worker. Multiple worker instances — increased from 3 to 12 six days ago — are simultaneously passing the Redis idempotency check for the same payment before any of them marks it processed. The fix is a one-line change: replace the Redis `GET` guard with an atomic `SET NX`, so only one worker can claim a payment and proceed to charge Stripe. The DB unique constraint is correctly catching some of these races, but by then Stripe has already been charged. The fix is ready to deploy.
