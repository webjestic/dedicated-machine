# Root Cause Analysis — Duplicate Charges Incident

## The Structural Cause

**The idempotency key is set after the charge completes. The deduplication check happens before the message crosses the queue boundary. The window between those two events is unbounded — it is the entire payment processing time. Any concurrent or redelivered request that arrives during that window passes the check and charges the card.**

This is not a race condition in the traditional sense. It is a design flaw. The lock is acquired in the wrong place, at the wrong time, by the wrong service.

---

## Tracing the Failure Path

Here is exactly what happens during a duplicate charge. No speculation — this is what the code does:

```
T+0ms    POST /payments/charge arrives for pay_001
T+0ms    Gateway checks Redis: processed:pay_001 → nil  ✓ passes
T+0ms    Gateway publishes pay_001 to SQS
T+202ms  Worker A receives pay_001, checks Redis: processed:pay_001 → nil  ✓ passes
T+202ms  Worker A calls stripe.Charge.create(...)  ← first charge begins

         [Stripe API call takes 800ms - 2000ms under load]

T+350ms  POST /payments/charge arrives again for pay_001
         (retry from client, or duplicate submission)
T+350ms  Gateway checks Redis: processed:pay_001 → nil  ✓ passes  ← KEY IS NOT SET YET
T+350ms  Gateway publishes pay_001 to SQS again

T+600ms  Worker B receives pay_001, checks Redis: processed:pay_001 → nil  ✓ passes
T+600ms  Worker B calls stripe.Charge.create(...)  ← SECOND CHARGE

T+1100ms Worker A receives Stripe response, inserts into DB, sets Redis key
T+1400ms Worker B receives Stripe response, attempts INSERT → UniqueViolationError
```

The `UniqueViolationError` in your logs is the DB constraint catching the second insert *after both charges have already been sent to Stripe*. The constraint is doing its job. The money has already left the customer's account twice.

The two workers do not need to be different instances. A single worker reprocessing a redelivered SQS message produces the same outcome. With `MaxInstances` at 12 and `VisibilityTimeout` at 120s, you now have 12 workers that can each hold a copy of the same message, all of which pass the Redis check during the processing window.

## Why the Tests Pass

The tests are correct for the behavior they test. They are not testing the failure mode because the failure mode requires concurrency and real processing time — neither of which exists in the test suite.

```python
# This test validates the guard works when Redis is already set.
# It does not test what happens when two calls arrive before Redis is set.
def test_process_payment_skips_if_already_processed(db, mock_redis, mock_stripe):
    mock_redis.get.return_value = "1"  # ← key is pre-populated
    process_payment(...)
    mock_stripe.Charge.create.assert_not_called()
```

The mock Redis is synchronous and shared in-process. There is no queue, no network latency, no concurrent workers. The test proves the guard logic is syntactically correct. It cannot prove the guard is structurally sound under concurrent load. This is why the CI is green and production is on fire.

---

## The Fix

The idempotency key must be set **before** the message crosses the queue boundary, not after processing completes. The check and the reservation must be a single atomic operation.

### Service A — Gateway: Use `SET NX` to atomically reserve before publishing

```python
@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically reserve this payment_id.
    # SET NX returns True only if the key did not exist.
    # If it returns False, either we already processed it or
    # another request is currently processing it — either way, do not publish.
    reserved = redis_client.set(
        f"processed:{payment_id}",
        "pending",
        nx=True,       # Only set if Not eXists
        ex=DEDUP_TTL,
    )

    if not reserved:
        # Key already exists — either "pending" or "1"
        # Return 200 regardless; the client does not need to retry
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX` is atomic at the Redis level. Two concurrent requests for the same `payment_id` cannot both return `True`. One wins, one gets `already_processed`. The message is published exactly once.

### Service B — Worker: Keep the guard, update the value on completion

The worker guard is still necessary — it defends against SQS redeliveries if the worker crashes after charging but before deleting the message. Keep it. Change what it writes on completion.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Guard against redelivered messages.
    # If the key is "1" (completed), skip entirely.
    # If the key is "pending" (in-flight from gateway), this worker owns it — proceed.
    # If the key is missing entirely, something deleted it — treat as safe to process
    # but this should not happen in normal operation.
    current = redis_client.get(f"processed:{payment_id}")
    if current == "1":
        logger.info("Payment %s already completed, skipping", payment_id)
        return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()

    # Upgrade from "pending" to "1" — payment is complete
    redis_client.setex(f"processed:{payment_id}", DEDUP_TTL, "1")
```

### The Stripe Idempotency Key — Add This Now

This is your safety net. If a bug in your code or a network partition causes two Stripe calls for the same payment, Stripe will deduplicate them if you pass the same idempotency key. This does not replace the fix above — it is defense in depth.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on this key for 24 hours
)
```

## What the Recent Changes Did

The `VisibilityTimeout` increase and the autoscaling increase did not introduce the bug. They widened the window in which the existing bug fires.

| Change | Effect |
|---|---|
| `VisibilityTimeout` 30s → 120s | Messages stay invisible longer before redelivery. Payments that take longer than 30s no longer redeliver during processing. This change is correct and should stay. |
| `MaxInstances` 3 → 12 | More workers polling concurrently. More workers available to receive a redelivered or duplicate-published message during the processing window. This multiplied the probability of concurrent execution hitting the race. |

The bug existed before these changes. These changes made it visible at the traffic volume of the Q1 promotion. The root cause is the deduplication architecture, not the deployment.

---

## Immediate Actions

1. **Deploy the `SET NX` change to the gateway now.** This closes the window for new duplicate charges. The worker fix and Stripe idempotency key can follow in the same deploy or the next one.

2. **Do not roll back the `VisibilityTimeout` or autoscaling changes.** The timeout increase is correct — it was preventing legitimate completions from being redelivered. Rolling it back reintroduces that problem and does not fix the actual cause.

3. **For affected customers:** The `UniqueViolationError` means the DB constraint caught the second insert. Pull the Stripe charge IDs for any `payment_id` with that error. If Stripe shows two successful charges, issue refunds. If the second charge failed at Stripe, no refund is needed but confirm no partial state exists.

4. **Add a test that covers the concurrent submission case** — two calls to `charge()` with the same `payment_id` before `process_payment` runs. That test should have existed before this architecture shipped.
