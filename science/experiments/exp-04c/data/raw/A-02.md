# Root Cause: Check-Then-Act Race Condition With No Atomic Lock

## What Is Happening Right Now

Two (or more) HTTP requests for the same `payment_id` arrive at the gateway within the same window. Both hit the Redis check before either worker finishes. Both pass. Both messages land on the queue. Both workers charge the card.

The Redis key that would block the second request is not set until **after** the Stripe charge completes and the database write commits — which, under normal conditions, takes several seconds. Under the Q1 promotion load, with 12 worker instances polling concurrently, that window is being hit constantly.

---

## The Exact Failure Sequence

```
T+0ms    Request 1 arrives → redis.get("processed:pay_001") → None → publish to SQS
T+50ms   Request 2 arrives → redis.get("processed:pay_001") → None → publish to SQS
                             ^^^ key still not set, worker hasn't finished yet

T+3000ms Worker A dequeues message 1 → redis.get → None → stripe.Charge.create() → db.insert → redis.setex ✓
T+3050ms Worker B dequeues message 2 → redis.get → None → stripe.Charge.create() → DUPLICATE CHARGE
                                        ^^^ or: db.insert → UniqueViolationError (customer charged, no record)
```

The `UniqueViolationError` you are seeing in the logs is the database constraint catching the second insert **after the second Stripe charge has already succeeded**. That is why you have customers who were charged twice with no payment record — the charge went through, the insert failed, the worker threw, the message was not deleted, and SQS redelivered it.

---

## Why the VisibilityTimeout Change Made It Worse

The original 30-second timeout meant that if a worker stalled, SQS would redeliver after 30 seconds. Increasing it to 120 seconds did not close the race window — it only delayed redelivery. The race is not caused by redelivery. It is caused by two separate messages for the same `payment_id` being enqueued before the lock is set. The timeout change is a red herring for this bug, but it does mean stalled messages now sit invisible for twice as long before you can intervene.

---

## Why All 28 Tests Pass

The tests serialize everything. `process_payment` is called directly and synchronously. The mock Redis is shared in-process. There is no queue, no network latency, no concurrent workers. The race condition requires two requests to be in-flight simultaneously with real processing time between them. The test suite cannot observe it because it never creates two concurrent requests.

```python
# This test proves the key is set — it does not prove it is set before a second request arrives
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", "amount": 1200, "customer_id": "cust_pqr"})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

The test is correct. The timing is wrong. The test suite has no way to know the difference.

## What Needs to Change

### Fix 1 — Set the lock in the gateway, before the message crosses the queue boundary (required)

The idempotency key must be claimed **atomically before publishing**. Use `SET NX` (set if not exists) with an expiry. If the key is already set, reject or return early. If the `SET NX` succeeds, the slot is claimed. No second request can pass the check until the key expires or is explicitly released.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomically claim the slot. SET NX returns True only if the key did not exist.
    # TTL covers the full processing window plus margin.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,  # 24 hours — same as before
    )

    if not claimed:
        # Key already exists: either processing is in flight or already completed
        return jsonify({"status": "already_processed", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

The `SET NX EX` is a single atomic Redis command. There is no window between the check and the set. Two concurrent requests for the same `payment_id` cannot both return `True`.

### Fix 2 — Keep the worker-side guard (required, but change its role)

The worker check is now a safety net for SQS redelivery, not the primary deduplication mechanism. Do not remove it. Do not rely on it as the first line of defense.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard: catches SQS redelivery after a worker crash post-charge.
    # The primary lock is held by the gateway before this message was published.
    if redis_client.get(f"processing:{payment_id}"):
        # Check whether this is already fully committed, not just in-flight
        existing = db.query_one(
            "SELECT stripe_charge_id FROM payments WHERE payment_id = ?",
            (payment_id,)
        )
        if existing:
            logger.info("Payment %s already recorded, skipping", payment_id)
            return

    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()
```

### Fix 3 — Pass a Stripe idempotency key (defense in depth, do this today)

Stripe supports idempotency keys natively. If the same key is sent twice, Stripe returns the original charge object without creating a new charge. This does not fix the root cause, but it caps the blast radius while the fix is deployed.

```python
charge = stripe.Charge.create(
    amount=data["amount"],
    currency=data.get("currency", "usd"),
    customer=data["customer_id"],
    idempotency_key=payment_id,  # Stripe deduplicates on their side
)
```

### Fix 4 — Add a concurrency test (so this cannot regress silently)

```python
import threading

def test_concurrent_requests_do_not_duplicate(client, mock_redis, mock_sqs):
    """Two simultaneous requests for the same payment_id must produce exactly one queue message."""
    _, sent = mock_sqs
    results = []

    def post():
        rv = client.post(
            "/payments/charge",
            json={"payment_id": "pay_concurrent", "amount": 1000, "customer_id": "cust_x"},
        )
        results.append(rv.status_code)

    t1 = threading.Thread(target=post)
    t2 = threading.Thread(target=post)
    t1.start(); t2.start()
    t1.join(); t2.join()

    assert len(sent) == 1, f"Expected 1 queued message, got {len(sent)}"
    assert sorted(results) == [200, 202]
```

## Summary for the Emergency Call

| | Current behavior | Required behavior |
|---|---|---|
| **Where lock is set** | Worker, after Stripe charge completes | Gateway, before message is published |
| **Lock mechanism** | `GET` then `SETEX` (two operations, not atomic) | `SET NX EX` (one atomic operation) |
| **Race window** | Entire payment processing time (seconds) | Zero — lock is held before queue publish |
| **Concurrent requests during processing** | Both pass the Redis check | Second request hits existing key, returns 200 |
| **SQS redelivery protection** | Worker Redis check (keep this) | Worker Redis check + DB existence check |
| **Stripe-level protection** | None | Idempotency key on every `Charge.create` call |

The fix is a one-line change to the Redis call in the gateway — `GET` + `SETEX` becomes `SET NX EX` — plus the Stripe idempotency key as immediate mitigation. The worker guard stays but is no longer load-bearing for the primary race.
