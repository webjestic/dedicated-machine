# Root Cause Analysis: Duplicate Charges in `payment-processor`

## The Short Answer

The idempotency key is set **after** the charge completes. The window between the Redis check and the Redis write is the entire payment processing time — Stripe round-trip plus database write. Any duplicate request that arrives during that window passes the check clean and proceeds to charge.

---

## What Is Actually Happening

### The Race Window

```
Time ──────────────────────────────────────────────────────────────────►

Request 1:  [Redis GET: miss] ──── [Stripe charge] ── [DB insert] ── [Redis SET]
                     ▲                                                      ▲
                     │◄──────────── entire window is unguarded ────────────►│

Request 2:           [Redis GET: miss] ── [Stripe charge] ── [DB insert] ── BOOM
                              │
                              └── arrives here, key not set yet, passes check
```

The Redis check in the gateway and the Redis write in the worker are in **different services**, separated by a queue boundary. The lock is never held. It is checked in one place and set in another, with unbounded time in between.

### Why the Recent Changes Made It Worse

Both operational changes deployed 6 days ago directly widened this window:

| Change | Effect on the Race Window |
|---|---|
| `VisibilityTimeout` 30s → 120s | A message that is slow to process now stays invisible for 2 minutes before redelivery, but the window for *new* duplicate requests arriving at the gateway is unchanged — it is still the full processing time |
| Worker `MaxInstances` 3 → 12 | Four times as many workers now poll the queue concurrently. A single message that lands in the queue multiple times (redelivery, retry) now has four times the chance of being picked up by two workers simultaneously |

The `VisibilityTimeout` increase was the right instinct for the redelivery problem, but it did not fix the root cause. It just changed which failure mode dominates. The autoscaling increase then amplified the concurrent-worker variant of the same bug.

### Why the Tests Pass

The tests are sequential and single-threaded. They test the happy path and the "already processed" path, but they never exercise two concurrent requests arriving before the worker finishes. The mock also shares state between gateway and worker in the same process, which hides the cross-service timing gap entirely.

```python
# This test proves the key gets set. It does not prove it gets set
# before a second request can pass the check.
def test_process_payment_sets_redis_key_after_processing(db, mock_redis, mock_stripe):
    process_payment({"payment_id": "pay_006", "amount": 1200, "customer_id": "cust_pqr"})
    mock_redis.setex.assert_called_once_with("processed:pay_006", 86400, "1")
```

28 tests passing. Zero tests for concurrent behavior. The test suite has a blind spot exactly where the production failure lives.

---

## The Fix

The idempotency key must be set **before** the message crosses the queue boundary, not after processing completes. The gateway must claim the payment atomically before publishing.

### Immediate Fix: Atomic Claim in the Gateway

Replace the read-then-publish pattern with a `SET NX` (set-if-not-exists) operation. This is a single atomic Redis command. There is no window between the check and the claim.

```python
# gateway/charge_handler.py

@app.route("/payments/charge", methods=["POST"])
def charge():
    data = request.get_json()
    payment_id = data["payment_id"]

    # Atomic claim: SET key value NX EX ttl
    # Returns True if the key was set (we claimed it), False if it already existed.
    claimed = redis_client.set(
        f"processing:{payment_id}",
        "1",
        nx=True,       # only set if key does not exist
        ex=DEDUP_TTL,
    )

    if not claimed:
        # Key already existed — either being processed or already done.
        return jsonify({"status": "already_accepted", "payment_id": payment_id}), 200

    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps(data),
    )

    return jsonify({"status": "accepted", "payment_id": payment_id}), 202
```

Two concurrent requests for the same `payment_id` now race on a single atomic Redis operation. One wins. One loses. The message is published exactly once.

### Worker: Keep the Guard, Treat It as Defense-in-Depth

The worker check is not the primary guard anymore, but keep it. It defends against SQS redeliveries (which the `VisibilityTimeout` increase was trying to solve) and any other path that bypasses the gateway.

```python
def process_payment(data: dict) -> None:
    payment_id = data["payment_id"]

    # Secondary guard: defense against SQS redelivery and any bypass path.
    # This is not the primary idempotency mechanism — the gateway SET NX is.
    if redis_client.get(f"processing:{payment_id}") is None:
        # Key is gone — TTL expired on a very old redelivery, or Redis was flushed.
        # Treat as a potential duplicate and reject rather than charge.
        logger.warning("processing key missing for %s, rejecting as safety measure", payment_id)
        return

    # Charge via Stripe using Stripe's own idempotency key as a second layer.
    charge = stripe.Charge.create(
        amount=data["amount"],
        currency=data.get("currency", "usd"),
        customer=data["customer_id"],
        idempotency_key=payment_id,   # <-- Stripe deduplicates on their side too
    )

    db.execute(
        "INSERT INTO payments (payment_id, customer_id, amount, stripe_charge_id)"
        " VALUES (?, ?, ?, ?)",
        (payment_id, data["customer_id"], data["amount"], charge["id"]),
    )
    db.commit()
    # Key is already set. No change needed here.
```

### Add Stripe Idempotency Keys as a Third Layer

Pass `idempotency_key=payment_id` to every `stripe.Charge.create` call. Stripe will deduplicate on their side for any request with the same key within 24 hours. This is the last line of defense if both the gateway and worker checks are bypassed.

This does not replace the Redis fix. It is a backstop. The `UniqueViolationError` you are seeing in the logs means the database constraint is catching some duplicates — that is also a backstop, not a fix. You want to stop the duplicate charge from being attempted at all, not catch it after the fact.

---

## Layered Defense Summary

```
Layer 1 — Gateway SET NX (PRIMARY):
  Atomic claim before queue publish.
  Stops duplicate messages from entering the queue.

Layer 2 — Worker Redis check (SECONDARY):
  Defends against SQS redelivery.
  Stops a redelivered message from being charged again.

Layer 3 — Stripe idempotency_key (TERTIARY):
  Stripe deduplicates on their side.
  Stops a duplicate charge even if layers 1 and 2 both fail.

Layer 4 — DB unique constraint (BACKSTOP):
  Raises UniqueViolationError if a duplicate INSERT is attempted.
  Does NOT prevent the duplicate Stripe charge — the card is already charged
  by the time the INSERT runs. This is not a fix, it is an alarm.
```

## What to Do Right Now

1. **Deploy the gateway `SET NX` change.** This stops new duplicates immediately.
2. **Add `idempotency_key=payment_id` to the Stripe call.** This is a one-line change that gives you Stripe-side deduplication as a backstop.
3. **Do not roll back the `VisibilityTimeout` increase yet.** The redelivery problem it was solving is real. Fix the root cause first, then evaluate.
4. **Audit the 23 open tickets against Stripe.** For customers with a successful second charge, issue refunds. For customers where the second charge failed but no payment was recorded, determine whether the first charge succeeded in Stripe but the DB write failed — those customers may have been charged without a record in your system.
5. **Write a concurrent integration test** that fires two simultaneous requests for the same `payment_id` and asserts exactly one charge was created. This test would have caught this before it reached production.
