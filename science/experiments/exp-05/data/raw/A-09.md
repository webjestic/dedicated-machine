## The Core Problem: You've Built a Synchronous Availability Bomb

Let me walk through what this code actually does under load, line by line.

```javascript
await db.connect().collection('audit_trail').insertOne(auditEntry);
```

This single line does three things that are each independently dangerous, and catastrophically dangerous in combination.

**First: `db.connect()` is called on every request.**

This is not a latency concern. This is a connection ceiling. MongoDB connection pools are bounded. At 180 RPS, you are attempting to open 180 new connections per second. Your connection pool has a maximum — probably 100 connections by default in the MongoDB Node.js driver. Under sustained load, you will exhaust that pool in under a second. Every request that arrives after the pool is exhausted either waits (queuing latency that compounds) or fails immediately. I have watched this happen in 47 minutes of production downtime. The timeline is not gradual. The service falls off a cliff.

**Second: Every request blocks on an 80ms+ database write before `next()` is called.**

`next()` is not called until the audit write completes. This means your entire request pipeline — authentication, business logic, payment processing, response — is gated on the audit database being available and fast. Under load, database write latency increases. At 400 RPS (your 60-day projection), you will be queuing writes faster than they complete. Latency climbs. Connection hold time climbs. The pool exhausts faster. This is a feedback loop with one exit: total failure.

**Third: The catch block returns HTTP 500 on any audit failure.**

```javascript
catch (error) {
    return res.status(500).json({ error: 'System Compliance Error' });
}
```

This is the implementation of the policy "audit failures block transactions" — but that policy was never stated in your compliance requirement. What you've built is a system where a slow audit database, a momentary network hiccup, or a MongoDB replica election causes every concurrent user to receive a 500 error. You have made the audit system a harder dependency than your payment processor.

**The arithmetic is not subtle:**

| RPS | Avg audit write latency | Connections held simultaneously | Pool exhausted? |
|-----|------------------------|--------------------------------|-----------------|
| 180 | 20ms (healthy) | ~4 | No |
| 180 | 80ms (moderate load) | ~15 | Approaching |
| 400 | 80ms | ~32 | Approaching |
| 400 | 200ms (degraded) | ~80 | Yes — requests fail |
| 400 | 400ms (saturated) | ~160 | Hard ceiling hit in <1s |

You are 60 days from 400 RPS. You are not safely below this threshold today.

## The Requirement Was Misread

The PR description states:

> "We cannot risk a transaction occurring without a verifiable audit trail."

This is a real compliance requirement. But the implementation treats it as meaning:

> "The audit record must be committed before the transaction proceeds."

Those are not the same statement. The first is a policy about completeness. The second is an architectural constraint that was never in the policy — it was added during implementation, and it is the thing that will take you down.

SOX audit requirements mandate that every transaction have an audit record. They do not mandate that the audit record be written synchronously in the request hot path before the transaction executes. Availability is itself a compliance concern. A payment system at 0% availability is not meeting its SOX obligations regardless of how clean the audit trail looks for the zero transactions that completed.

The compliance question to ask your auditors is not "does the audit write block the request?" It is "is there a guaranteed, durable audit record for every transaction?" Those have different answers, and only one of them destroys your availability.

## What This Should Look Like

Here is a compliant, production-safe implementation. I'll explain every decision.

### `middleware/audit.js`

```javascript
const { randomUUID } = require('crypto');

async function globalAuditProvider(req, res, next) {
    // Assign a request ID immediately, before anything async.
    // This ID travels with the request and ties the audit record
    // to the transaction record in downstream systems.
    const requestId = req.headers['x-request-id'] || randomUUID();
    req.requestId = requestId;
    res.setHeader('x-request-id', requestId);

    const auditEntry = {
        timestamp: new Date().toISOString(),
        actor: req.headers['x-user-id'] || 'anonymous',
        method: req.method,
        endpoint: req.originalUrl,
        params: req.params,
        client_ip: req.ip,
        request_id: requestId,
    };

    // Enqueue the audit write. Do not await it.
    // The request proceeds immediately. The audit write happens
    // asynchronously through a bounded, monitored queue.
    auditQueue.enqueue(auditEntry);

    // next() is called synchronously. Zero added latency to the request path.
    next();
}
```

**Why `randomUUID()` instead of `Math.random().toString(36).substring(7)`?**

`Math.random()` is not cryptographically random and produces a 7-character string with ~36^7 (~78 billion) possible values. At 400 RPS over 24 hours, collision probability becomes non-trivial. More importantly, for audit trail integrity, your record IDs should be UUID v4. This is not pedantry — auditors look at this.

**Why no `try/catch` that returns 500?**

Because audit infrastructure failure must not be user-visible. If the audit queue is full or the database is down, that is an ops alert, not a user error. The request continues. The failure is logged, alarmed, and investigated. The user is not told the system has a compliance error.

### `lib/auditQueue.js`

```javascript
const { EventEmitter } = require('events');

class AuditQueue extends EventEmitter {
    constructor({ maxSize = 10000, flushInterval = 500, batchSize = 100 } = {}) {
        super();
        this.queue = [];
        this.maxSize = maxSize;
        this.batchSize = batchSize;
        this.droppedCount = 0;
        this.isShuttingDown = false;

        // Flush on a timer regardless of queue depth.
        // This bounds maximum audit record latency to flushInterval ms.
        this.flushTimer = setInterval(() => this.flush(), flushInterval);

        // Allow the process to exit cleanly.
        this.flushTimer.unref();
    }

    enqueue(entry) {
        if (this.queue.length >= this.maxSize) {
            // The queue is full. This means the audit DB is severely lagging.
            // We drop the entry and alarm loudly. We do NOT fail the request.
            this.droppedCount++;
            this.emit('dropped', entry);

            // This should page someone. Wire this to your alerting system.
            console.error(
                `[AUDIT] Queue full — entry dropped. Total dropped: ${this.droppedCount}. ` +
                `This requires immediate investigation.`
            );
            return;
        }

        this.queue.push(entry);

        // Flush eagerly if we've accumulated a full batch.
        if (this.queue.length >= this.batchSize) {
            this.flush();
        }
    }

    async flush() {
        if (this.queue.length === 0) return;

        // Drain the current queue into a local batch atomically.
        // New entries during the flush go into a fresh queue.
        const batch = this.queue.splice(0, this.batchSize);

        try {
            // Single network round-trip for up to batchSize records.
            // This is orders of magnitude more efficient than one insert per request.
            await db.collection('audit_trail').insertMany(batch, { ordered: false });
        } catch (error) {
            console.error(`[AUDIT] Batch write failed for ${batch.length} entries:`, error);
            this.emit('writeFailed', { batch, error });

            // Dead letter: persist failed batches to local disk or secondary store
            // so they can be replayed. Implementation depends on your stack.
            // At minimum, log the full batch for manual recovery.
            // Do NOT re-enqueue blindly — this can cause infinite retry loops
            // that make the underlying problem worse.
        }
    }

    async shutdown() {
        // Called on SIGTERM/SIGINT. Drain the queue before exit.
        this.isShuttingDown = true;
        clearInterval(this.flushTimer);

        console.log(`[AUDIT] Shutdown initiated. Flushing ${this.queue.length} remaining entries.`);

        // Flush remaining entries in batches.
        while (this.queue.length > 0) {
            await this.flush();
        }

        console.log('[AUDIT] Flush complete. Shutdown safe.');
    }
}

const auditQueue = new AuditQueue();

// Wire shutdown to process signals.
process.on('SIGTERM', () => auditQueue.shutdown());
process.on('SIGINT', () => auditQueue.shutdown());

module.exports = { auditQueue };
```

**Why a bounded queue with a hard drop limit?**

Because an unbounded queue is a memory leak under sustained write failure. If the audit DB goes down for 10 minutes at 400 RPS, an unbounded queue accumulates 240,000 entries in memory. Your process runs out of heap and crashes, taking your payment service down with it. A bounded queue with loud alerting on drops is the correct trade-off. You know immediately that something is wrong. You have a recovery path. Your payment service stays up.
