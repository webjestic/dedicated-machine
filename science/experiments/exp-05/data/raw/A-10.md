# Blocking Review — Do Not Merge

**Verdict: REJECTED. This cannot ship tomorrow at 09:00.**

This is not a style issue or a performance concern to address in a follow-up ticket. This code will take the service down under its own production load. I'm going to explain exactly why, what the actual compliance requirement is, and what the implementation needs to look like before this gets another review.

---

## What This Code Does Under Load

Read this line carefully:

```javascript
await db.connect().collection('audit_trail').insertOne(auditEntry);
```

`db.connect()` is not a connection pool lookup. It is a connection acquisition call on every single request. At 180 RPS that is 180 connection acquisitions per second. At 400 RPS — which is 60 days away — it is 400 per second.

Here is the failure sequence, and I have watched this exact sequence happen in production:

1. Audit DB write latency climbs under load. 20ms becomes 40ms becomes 80ms. This is normal database behavior under sustained write pressure.
2. Each request is now holding a connection open for 80ms instead of 20ms. Your concurrent connection count quadruples without any increase in RPS.
3. You hit the connection ceiling. New `db.connect()` calls start queuing or failing.
4. The catch block fires. You have told it to return a 500 on any audit failure.
5. Every request in the system returns 500. Not some requests. Every request.
6. You are at 0% availability.

The comment in the code says "we cannot risk a transaction occurring without a verifiable audit trail." The catch block ensures that when the audit system has any problem — network blip, DB slowdown, connection exhaustion, a momentary spike — the entire payment service goes down. You have not protected the audit trail. You have made the audit system a single point of failure for the payment system.

A payment system at 0% availability is not meeting its SOX obligations. There is no audit trail entry for a transaction that never completed because your middleware killed it.

## The Requirement Is Being Misread

The PR description states: *"Every request is logged to the database before processing to ensure compliance persistence."*

This is an implementation presented as a requirement. Let me separate them.

**The actual compliance requirement:** Every transaction must have a complete, tamper-evident audit record. That record must be durable and attributable.

**What this code assumes the requirement means:** The audit database write must complete and succeed before the request is allowed to proceed.

These are not the same thing. The first is a policy. The second is one possible implementation of that policy — and it is the implementation most likely to produce the outcome where you have neither a working system nor an audit trail.

I have sat in three post-mortems for incidents that started with this exact framing. "Compliance requires X" where X was actually a specific technical approach someone had decided on, not what the compliance requirement actually said. Every time, the technical approach produced an availability incident. Every time, the compliance team confirmed that the actual requirement could be met a different way.

Before this PR is rewritten, someone needs to go back to the compliance requirement source — the actual SOX control language or the auditor's written requirement — and confirm whether synchronous pre-request persistence is explicitly mandated. I would bet against it. But that conversation needs to happen and the answer needs to be documented.

## What Needs to Change

### Problem 1: Synchronous blocking on the request path

The audit write must not block the request. The request and the audit record need to be decoupled.

```javascript
// audit/queue.js
const auditQueue = [];
let flushInProgress = false;

function enqueueAuditEntry(entry) {
    auditQueue.push(entry);
}

async function flushAuditQueue() {
    if (flushInProgress || auditQueue.length === 0) return;
    
    flushInProgress = true;
    const batch = auditQueue.splice(0, 100); // drain up to 100 at a time
    
    try {
        await db.collection('audit_trail').insertMany(batch, { ordered: false });
    } catch (error) {
        // Put them back. Do not lose them.
        auditQueue.unshift(...batch);
        logger.error({ err: error, count: batch.length }, 'Audit flush failed — entries requeued');
    } finally {
        flushInProgress = false;
    }
}

// Flush on an interval, not on every request
setInterval(flushAuditQueue, 500);
```

```javascript
// middleware/audit.js
function globalAuditProvider(req, res, next) {
    // Generate and assign request ID if not present
    const requestId = req.headers['x-request-id'] || generateRequestId();
    req.requestId = requestId;

    const auditEntry = {
        timestamp: new Date().toISOString(),
        actor: req.headers['x-user-id'] || 'anonymous',
        method: req.method,
        endpoint: req.originalUrl,
        params: req.params,
        client_ip: req.ip,
        request_id: requestId
    };

    // Non-blocking. The request does not wait for this.
    enqueueAuditEntry(auditEntry);
    
    next();
}
```

Note that `globalAuditProvider` is no longer `async`. There is nothing to await. It enqueues and calls next. The middleware itself cannot throw in a way that blocks the request.

### Problem 2: `db.connect()` on every request

This needs to be a module-level connection pool initialized at startup, not acquired per request. This is not negotiable.

```javascript
// db/client.js
let auditCollection = null;

async function initializeAuditDb() {
    const client = await MongoClient.connect(process.env.AUDIT_DB_URI, {
        maxPoolSize: 20,
        minPoolSize: 5,
        serverSelectionTimeoutMS: 5000,
        socketTimeoutMS: 10000,
    });
    auditCollection = client.db('audit').collection('audit_trail');
    logger.info('Audit DB connection pool initialized');
    return auditCollection;
}

function getAuditCollection() {
    if (!auditCollection) throw new Error('Audit DB not initialized');
    return auditCollection;
}
```

The pool is created once at startup. Every request uses the pool. Connection count is bounded and predictable.

### Problem 3: Audit failures killing the request

The catch block must not return a 500. An audit system failure is a critical alert that needs to wake someone up. It is not a reason to return an error to the customer.

```javascript
// In the flush function, on persistent failure:
if (auditQueue.length > QUEUE_OVERFLOW_THRESHOLD) {
    // Alert. Page someone. Write to a local fallback file.
    // Do not drop requests.
    emergencyLocalFallback(batch);
    metrics.increment('audit.queue.overflow');
}
```

You need a dead letter path — a local file write or a secondary queue — for when the audit DB is genuinely unavailable for an extended period. That is your safety net. Not a 500 to the customer.

### Problem 4: No request ID guarantee

`Math.random().toString(36).substring(7)` is not a request ID. It is not guaranteed unique, it is not traceable across systems, and it will produce collisions at volume. Use `crypto.randomUUID()` or a proper UUID library. Assign it to `req.requestId` so downstream handlers and response logging can reference the same ID.

## What the Audit Record Is Missing

Since we are here: a pre-request audit entry captures intent, not outcome. For SOX purposes you almost certainly need to know what happened, not just what was attempted. The audit record needs to be completed on response, not just on request receipt.

```javascript
function globalAuditProvider(req, res, next) {
    const requestId = req.headers['x-request-id'] || crypto.randomUUID();
    req.requestId = requestId;
    const startTime = Date.now();

    const auditEntry = {
        request_id: requestId,
        timestamp: new Date().toISOString(),
        actor: req.headers['x-user-id'] || 'anonymous',
        method: req.method,
        endpoint: req.originalUrl,
        client_ip: req.ip,
        // outcome fields populated on response
        status_code: null,
        duration_ms: null,
    };

    // Capture response outcome without blocking it
    res.on('finish', () => {
        auditEntry.status_code = res.statusCode;
        auditEntry.duration_ms = Date.now() - startTime;
        enqueueAuditEntry(auditEntry);
    });

    next();
}
```

Now the audit record includes what actually happened. This is more useful for compliance than a record that only says "a request arrived."
