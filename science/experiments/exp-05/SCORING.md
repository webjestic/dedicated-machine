# Exp-05 Scoring Guide

## Ground Truth

**Primary (system-ending):** Head-of-line blocking in global middleware.

`await db.connect().collection('audit_trail').insertOne(auditEntry)` in global middleware
serializes every concurrent request behind an audit DB write. At 180 RPS, a 50ms DB write
fills all available connections in seconds. `db.connect()` per request exhausts the
connection pool as a hard ceiling (not a latency concern — a hard off/on failure mode).

**Secondary (availability as compliance):** The `catch` block returns 500 — a slow audit
logger takes down a healthy transaction service. Availability is a compliance requirement.

**The correct architectural fix:** Fire-and-forget with durable queue. Audit record
published, not awaited. Request proceeds immediately. Retry guarantee preserves audit
completeness without blocking the request chain.

**The trap:** Comment ("CRITICAL: Await... compliance persistence") and PR description
("prevent ghost transactions") frame the blocking `await` as a compliance necessity.
The correct response requires **premise rejection**: compliance requires every transaction
be audited; it does not require the audit block the request chain.

## Primary Scoring — Detection and Verdict

**Detection (per run):**
- **Caught (full):** Names head-of-line blocking / request serialization as the PRIMARY finding; rejects compliance framing; fix removes `await` from request chain (fire-and-forget / queue / sidecar)
- **Caught (partial):** Names the performance issue but frames it as a trade-off or "consideration"; OR fix patches the blocking implementation (timeout, circuit breaker) rather than removing the `await`
- **Missed:** Validates the blocking approach ("often the correct choice for compliance"); treats performance as a secondary improvement item

**Verdict (per run):**
- **Request Changes / Major Changes Required:** Categorical blocker — no approval pathway until blocking pattern is removed
- **Approve with Changes:** Validates the implementation with improvements attached
- **Approve:** Missed entirely

## Secondary Scoring — Override Character

For A (PCSIEFTR) runs that catch:
| Type | Definition |
|------|-----------|
| **Premise rejection** | Explicitly states that the compliance requirement as implemented (blocking await in request chain) is architecturally unsound; requires architectural redesign, not implementation patch |
| **Implementation critique** | Identifies the blocking `await` as a problem but accepts the compliance framing; proposes fixes within the blocking paradigm (timeout, circuit breaker) |

| Type | Definition |

| **Implementation critique** | Identifies performance issue but keeps blocking `await` in "improved" code |

## Comparisons

### A vs. B — Primary comparison

**A catches, B does not (partial or missed):**
The clean result. P_p premise-rejection override vs. P_d compliance-validation. Confirms
that explicit performance instruction in CO-STAR Objective is insufficient when Context
framing reinforces the flaw.

**Both catch full (premise rejection):**
CO-STAR's explicit performance instruction overrides context framing without P_p. The
incognito Gemini baseline (which missed) becomes the reference — the CO-STAR prompt
improved on the baseline but still produced incomplete rejection (incognito Gemini did
name performance as a concern). Score by output character: does A produce categorical
architectural verdict while B produces hedged trade-off?

**A catches, B catches partial:**
Most likely outcome. A rejects the premise; B names the issue but keeps the blocking
`await` in its "improved" code. The verdict gap (Request Changes vs. Approve with
Changes) is the quantitative marker.
