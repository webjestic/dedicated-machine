# Root Cause Analysis — `IntegrityError: UNIQUE constraint failed: task_results.task_id`

## The Failure, Read Backward

The constraint violation tells us exactly what happened: two workers completed the same task and both attempted to `INSERT` into `task_results` for the same `task_id`. The unique constraint caught the second write. The first write succeeded — which is why the reporting pipeline shows **gaps**, not duplicates. The second worker's result was thrown away with an unhandled exception, and that task's final state in `task_queue` is now indeterminate.

This is a classic **check-then-act race condition**. The check and the act are not atomic.

---

## The Unprotected Window

```python
def acquire_task(worker_id: int):
    # ← Worker A reads this row. Status = 'pending'.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # ← Worker B also reads this same row before A's UPDATE commits.
    #   Both workers now hold the same task_id in memory.

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    # ← Both workers commit their UPDATE successfully.
    #   SQLite's row-level locking does not prevent this.
    #   Both return the same dict. Both call execute_task. Both call complete_task.
    return dict(row)
```

Between the `SELECT` and the `UPDATE`, there is an open window. With a shared connection pool — introduced four days ago — multiple workers share connections and can interleave reads before either write is committed. Worker A reads the pending row. Worker B reads the same pending row. Both update it to `running` (both succeed, last write wins, no error). Both execute the task. Both attempt to insert into `task_results`. The second insert hits the unique constraint.

**This did not happen before the migration** because per-worker SQLite connections with SQLite's default journal mode gave each worker an effectively serialized view. The shared pool removed that accidental serialization. The bug was always there. The infrastructure change exposed it.

---

## Why the Tests Don't Catch It

The tests are single-worker, sequential, and synchronous. There is no concurrency in the test suite. A race condition that requires two workers to be inside `acquire_task` simultaneously will never appear in a test that calls `acquire_task(worker_id=1)` once and awaits the result. The 34 passing tests are testing the happy path of a single-threaded world. They are not wrong — they are just not testing the thing that is broken.

---

## The Fix

The `SELECT` and the `UPDATE` must be a single atomic operation. The standard pattern for this is `UPDATE ... WHERE status = 'pending' RETURNING` (if your database supports it), or a `SELECT ... FOR UPDATE` inside a transaction. For SQLite specifically, the correct approach is to make the claim conditional on the status at write time and check whether anything was actually updated.

```python
def acquire_task(worker_id: int):
    # First, find the candidate task id.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt to claim it atomically. The WHERE clause is the guard.
    # If another worker claimed it between our SELECT and this UPDATE,
    # the status is no longer 'pending' and rows_affected will be 0.
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← the critical addition
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker claimed this task in the window between our
        # SELECT and our UPDATE. Yield and let the caller retry.
        return None

    return dict(row)
```

The `AND status = 'pending'` predicate turns the `UPDATE` into a compare-and-swap. Only one worker's `UPDATE` can match — the one that gets there first. The other worker's `UPDATE` matches zero rows, sees `rows_affected == 0`, and returns `None`. The worker loop retries on the next tick.

This works because the `UPDATE` itself is atomic at the database level. The race is not eliminated — both workers may still reach the `UPDATE` — but only one of them wins, and the loser knows it lost.

---

## What Else Needs to Happen

**`complete_task` also needs a guard.** Even after fixing `acquire_task`, `complete_task` should be defensive:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # ← guard against double-complete
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` is a last-resort safety net, not the primary fix. The primary fix is in `acquire_task`. But defense in depth means the constraint violation stops being an unhandled exception that drops results.

**The test suite needs a concurrency test.** Something that spawns two threads, has them both call `acquire_task` simultaneously against the same seeded task, and asserts that exactly one of them returns the task and the other returns `None`. That test would have caught this before the deploy.

---

## Summary

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: `SELECT` and `UPDATE` are separate operations with no isolation between them |
| **Trigger** | Shared connection pool introduced real concurrency between workers that previously had serialized database access |
| **Symptom** | Two workers claim the same task, both write to `task_results`, second write hits unique constraint |
| **Data impact** | Second worker's result is lost; task may be left in inconsistent state |
| **Fix** | Add `AND status = 'pending'` to the claim `UPDATE`; check `rows_affected`; add `INSERT OR IGNORE` as a backstop |
| **Test gap** | No concurrent acquisition test exists; all tests are single-worker sequential |
