# Root Cause Analysis

## The Short Answer

`acquire_task` has a classic **check-then-act race condition**. The SELECT and the UPDATE are two separate, unprotected operations. Between them, a second worker can read the same row. Both workers claim the same task, both call `complete_task`, and the second `INSERT INTO task_results` hits the unique constraint on `task_id`.

---

## The Failure Sequence

Here is exactly what the logs are showing you, reconstructed:

```
T=0ms  Worker-1: SELECT → sees task_id=42, status='pending'
T=0ms  Worker-2: SELECT → sees task_id=42, status='pending'   ← same row, before either UPDATE lands
T=1ms  Worker-1: UPDATE task_queue SET status='running' WHERE id=42
T=1ms  Worker-1: COMMIT
T=1ms  Worker-2: UPDATE task_queue SET status='running' WHERE id=42  ← succeeds, no guard
T=1ms  Worker-2: COMMIT
       ... both workers execute the task independently ...
T=Xms  Worker-1: INSERT INTO task_results (task_id=42, ...) → OK
T=Xms  Worker-2: INSERT INTO task_results (task_id=42, ...) → 💥 UNIQUE constraint failed
```

The unique constraint on `task_results.task_id` is doing its job. It is the last line of defense, and it is firing. The missing results in the reporting pipeline are the tasks where Worker-2 won the race — its result was never written because the INSERT failed, and there is no retry or error recovery in `complete_task`.

---

## Why This Wasn't Happening Before

The migration to a **shared connection pool** is what exposed this. With per-worker SQLite connections, each worker had its own connection and SQLite's file-level locking was serializing access as a side effect — not by design, but it was there. With a shared pool, multiple workers are now genuinely concurrent against the same database, and the window between SELECT and UPDATE is real.

The ORM upgrade is a distraction. The race condition was always in the code. The infrastructure change made it reproducible.

---

## Why the Tests Don't Catch It

Every test calls `acquire_task` exactly once, with a single worker, against an in-memory database with a single connection. There is no concurrency. The tests are testing the happy path of the logic, not the invariant that a task is claimed by exactly one worker. This is a gap in the test suite, not a green light.

---

## The Fix

The SELECT and the UPDATE must be a single atomic operation. There are two standard approaches:

### Option 1: Atomic UPDATE with RETURNING (preferred for PostgreSQL/SQLite 3.35+)

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

The subquery and the UPDATE execute as one statement. There is no window. A row either gets claimed by this worker or it doesn't — no other worker can observe the same `pending` row and also succeed on the UPDATE.

### Option 2: Optimistic locking with a WHERE guard

If `RETURNING` is not available on your database version:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Another worker claimed it first. Back off and try again.
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means the UPDATE is a no-op if another worker already transitioned the row. Checking `rowcount` tells you whether you actually won the claim. This is still two round trips, so Option 1 is cleaner, but this works correctly.

## What Also Needs to Change in `complete_task`

The error recovery is missing. Right now, if the INSERT fails, the task stays marked `running` in `task_queue` and the result is lost. Add explicit handling:

```python
def complete_task(task_id: int, result: list) -> None:
    try:
        db.execute(
            "UPDATE task_queue SET status = 'complete', completed_at = ?"
            " WHERE id = ?",
            (datetime.utcnow(), task_id),
        )
        db.execute(
            "INSERT INTO task_results (task_id, result, recorded_at)"
            " VALUES (?, ?, ?)",
            (task_id, json.dumps(result), datetime.utcnow()),
        )
        db.commit()
    except Exception:
        db.rollback()
        # Log, alert, and decide: requeue the task or dead-letter it.
        # Do not silently swallow this.
        raise
```

Without the rollback, you can end up with the queue row marked `complete` but no result row written, or vice versa, depending on which statement failed and what the connection pool does with the dirty transaction.

---

## The Test Gap to Close

Add a concurrency test that actually validates the invariant:

```python
def test_acquire_task_is_exclusive(db, seed_tasks):
    """Two workers racing for the same task must produce exactly one claim."""
    from concurrent.futures import ThreadPoolExecutor

    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(acquire_task, worker_id=i) for i in (1, 2)]
        results = [f.result() for f in futures]

    claimed = [r for r in results if r is not None]
    assert len(claimed) == 1, (
        f"Expected exactly one worker to claim the task, got {len(claimed)}"
    )
```

This test would have failed before the fix and will pass after it. That is the test that belongs in the suite.

## Summary for the Postmortem

| | |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: SELECT and UPDATE are separate statements with no transaction isolation between them |
| **Trigger** | Migration to shared connection pool removed the accidental serialization that per-worker connections provided |
| **Visible symptom** | `UNIQUE constraint failed: task_results.task_id` |
| **Silent symptom** | Missing results in reporting pipeline (failed INSERT, no recovery) |
| **Fix** | Collapse SELECT + UPDATE into a single atomic statement; add rollback on failure in `complete_task` |
| **Test gap** | No concurrent acquisition test existed; add one |

The constraint violation is the database telling you the truth. The race was always there.
