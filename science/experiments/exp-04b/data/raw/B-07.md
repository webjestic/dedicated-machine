## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. Multiple workers can read the same `pending` row before any of them updates it to `running`. Every one of those workers then executes the task and calls `complete_task`, and the second `INSERT INTO task_results` hits the unique constraint on `task_id`.**

---

### The Failure Sequence

Here is exactly what happens at the database level when two workers are running concurrently:

```
Time  Worker 1                                    Worker 2
────  ──────────────────────────────────────────  ──────────────────────────────────────────
T1    SELECT ... WHERE status = 'pending'
      → returns task #42
T2                                                SELECT ... WHERE status = 'pending'
                                                  → also returns task #42
                                                  (Worker 1's UPDATE hasn't committed yet)
T3    UPDATE task_queue SET status = 'running'
      WHERE id = 42
      COMMIT
T4                                                UPDATE task_queue SET status = 'running'
                                                  WHERE id = 42
                                                  COMMIT  ← succeeds, no guard here
T5    execute_task(task #42) → result_A
T6                                                execute_task(task #42) → result_B
T7    INSERT INTO task_results (task_id=42, ...)
      COMMIT  ← succeeds
T8                                                INSERT INTO task_results (task_id=42, ...)
                                                  COMMIT  ← 💥 UNIQUE constraint failed
```

The SELECT and the UPDATE are two separate statements with no transaction isolation between them. The shared connection pool introduced by the migration means workers are no longer serialized by process-level SQLite file locking. The window that was always there is now wide open.

---

### Why the Tests Don't Catch It

The tests run a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in the test environment and never will unless the tests are explicitly written to exercise concurrent acquisition. Passing CI tells you nothing about this class of failure.

---

### Why This Is Not the ORM Upgrade

The ORM version is a distraction. The logic was always unsafe. The connection pool change is what made the unsafe window reachable in production. But patching the pool configuration — adding a connection limit, serializing at the pool layer — would be treating the symptom. The code itself is wrong and needs to be fixed regardless of how connections are managed.

---

### The Fix

The SELECT and UPDATE must be atomic. The standard approach for SQLite is to use `UPDATE ... WHERE status = 'pending' ... RETURNING` in a single statement, or to use `SELECT ... FOR UPDATE` within an explicit transaction if the database supports it. For SQLite specifically, the correct pattern is:

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # Single atomic UPDATE with a subquery.
    # Only one writer can claim a given row because the WHERE clause
    # re-checks status = 'pending' at write time, not at read time.
    # RETURNING gives us the row we claimed without a second SELECT.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The critical property here is that the status check and the status update happen in the same write operation. SQLite serializes writes. If two workers race to this UPDATE, one of them will find `status = 'pending'` and claim the row; the other will find `status = 'running'` and update zero rows. `RETURNING` will return nothing for the loser. No second SELECT, no window between check and act.

`RETURNING` requires SQLite 3.35.0 (released March 2021). If you are on an older version, the equivalent is:

```python
def acquire_task(worker_id: int):
    cursor = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if cursor.rowcount == 0:
        return None

    # Safe to SELECT now — we own this row, status is 'running'
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at >= ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

The second SELECT in the fallback is safe because it is scoped to rows this specific worker just claimed. Another worker cannot steal it because status is no longer `pending`.

---

### What Needs to Happen Right Now

1. **Immediate mitigation:** Reduce the worker pool to a single worker. This closes the race window without a code change and stops the integrity errors. You will lose throughput. That is acceptable. The missing results in the reporting pipeline are from tasks that lost the race and got no result written — those task IDs need to be identified and re-queued.

2. **The fix:** Deploy the atomic UPDATE pattern above. This is the only correct resolution.

3. **The test that should have existed:** Add a test that spawns two threads, has both call `acquire_task` simultaneously against a shared database, and asserts that exactly one of them gets the task. That test would have caught this before it shipped.

The unique constraint violation is not the bug. It is the database correctly refusing to let you write the same task result twice. The bug is that two workers are reaching `complete_task` for the same task ID at all.
