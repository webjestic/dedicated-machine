## Root Cause

**`acquire_task` has a check-then-act race condition with no atomic protection around the claim window.**

Here is the exact sequence that produces the `IntegrityError`:

```
Worker A                                    Worker B
--------                                    --------
SELECT ... WHERE status = 'pending'
  → returns task_id = 42
                                            SELECT ... WHERE status = 'pending'
                                              → returns task_id = 42  (still 'pending')
UPDATE task_queue SET status = 'running'
  WHERE id = 42
db.commit()
execute_task(task_42)
complete_task(42, result)
  INSERT INTO task_results (task_id, ...)   UPDATE task_queue SET status = 'running'
  ← succeeds                                  WHERE id = 42
db.commit()                                 db.commit()
                                            execute_task(task_42)
                                            complete_task(42, result)
                                              INSERT INTO task_results (task_id, ...)
                                              ← IntegrityError: UNIQUE constraint failed
```

Between the `SELECT` and the `UPDATE`, both workers see the same `pending` task. The `UPDATE` does not protect against this — it will succeed for both workers because it only filters on `id`, not on `status`. Both workers claim the task, both execute it, and both attempt to insert into `task_results`. The second insert hits the unique constraint on `task_id`.

---

## Why Tests Did Not Catch This

Every test runs a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in that environment. The tests are correct for what they test. They simply do not test the thing that is broken.

The migration four days ago is the trigger, not the cause. Moving to a shared connection pool meant multiple workers are now executing concurrently against the same database state. The race condition was always latent in this code. The pool gave it the conditions it needed to fire.

---

## What Needs to Change

The fix must make the claim atomic. The `SELECT` and the `UPDATE` need to be a single operation so no second worker can observe the same `pending` row in the window between them.

**Option 1 — Add a `status` guard to the `UPDATE` and check rows affected (minimal change, ships fastest):**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard added
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount

    db.commit()

    if rows_affected == 0:
        # Another worker claimed it between our SELECT and this UPDATE
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means only one worker's `UPDATE` will match. The other gets `rowcount = 0` and backs off. This is safe because the `UPDATE` itself is atomic at the database level — two concurrent updates to the same row serialize, and only one will see `status = 'pending'` at the moment it executes.

**Option 2 — Single atomic `UPDATE ... RETURNING` (cleaner, removes the SELECT entirely):**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " ) AND status = 'pending'"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This collapses the check and the act into one statement. There is no window. This is the correct long-term shape of this function.

## The Missing Results

The gaps in the downstream reporting pipeline are the tasks that Worker A claimed and completed successfully before Worker B's duplicate attempt failed. Worker B's `complete_task` call raises before it reaches `db.commit()`, which means Worker B's `UPDATE task_queue SET status = 'complete'` also never commits. That task row stays in `status = 'running'` permanently unless there is a timeout/requeue mechanism. If there is not one, those tasks are stuck and their results are missing. That needs to be verified and those rows may need to be manually requeued.

---

## For the Sync Call

> The race is in `acquire_task`. Two workers can SELECT the same pending task before either UPDATE commits. The fix is one line: add `AND status = 'pending'` to the UPDATE and check `rowcount`. If it is zero, return None and let the worker retry. Option 1 can be written, reviewed, and deployed in under 30 minutes. The missing results in the report are tasks stuck in `running` status — we need to check how many there are and decide whether to requeue them manually.
