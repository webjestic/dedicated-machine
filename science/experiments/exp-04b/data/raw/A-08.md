## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. There is no atomic claim on the row. Two workers can read the same pending task in the window between the SELECT and the UPDATE, both proceed to execute it, and both attempt to INSERT into `task_results`. The second INSERT hits the UNIQUE constraint on `task_id` and throws the `IntegrityError`.**

---

## The Failure Sequence

```
Worker A                                    Worker B
--------                                    --------
SELECT ... WHERE status = 'pending'
                                            SELECT ... WHERE status = 'pending'
  → returns task_id = 42                      → returns task_id = 42
                                              (status is still 'pending')

UPDATE task_queue SET status = 'running'
  WHERE id = 42
                                            UPDATE task_queue SET status = 'running'
                                              WHERE id = 42
                                              (succeeds — no guard on current status)

execute_task(42) → result_A
                                            execute_task(42) → result_B

INSERT INTO task_results (task_id=42, ...)
  → OK
                                            INSERT INTO task_results (task_id=42, ...)
                                              → 💥 IntegrityError: UNIQUE constraint
                                                   failed: task_results.task_id
```

Both workers read `status = 'pending'` before either UPDATE commits. Both UPDATEs succeed because neither one guards against the current status value. The race window is the gap between the SELECT and the UPDATE commit — and under load with a connection pool, that window is hit constantly.

---

## Why Tests Don't Catch This

The tests run a single worker against an in-process database. There is no concurrency. The race window does not exist in the test environment. This is the exact class of bug that passes CI and waits for production scale to appear.

The connection pool migration four days ago is the trigger, not the cause. It increased concurrency to the point where the pre-existing race is now hit regularly. The bug was always there.

---

## Why Results Are Missing Downstream

When Worker B's INSERT throws, `complete_task` raises before the `db.commit()` call. The UPDATE that set `status = 'complete'` is also rolled back (or never committed, depending on pool transaction behavior). The task may be left in `running` status with no result row. The reporting pipeline sees a gap.

---

## What Needs to Change

### Fix: Make the claim atomic

The SELECT and UPDATE must be a single atomic operation. The standard pattern is to add a `WHERE status = 'pending'` guard on the UPDATE and check the affected row count. If the count is zero, another worker got there first — discard and move on.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard added here
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Another worker claimed this task between our SELECT and UPDATE
        return None

    return dict(row)
```

The `AND status = 'pending'` clause turns the UPDATE into the actual claim gate. Only one worker's UPDATE will match. The other gets `rowcount = 0` and backs off.

### Stronger fix if SQLite supports it in your version: `UPDATE ... RETURNING`

Eliminates the SELECT entirely. One round trip, one atomic operation:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

No separate SELECT. No window. The subquery and the UPDATE execute as one statement.
