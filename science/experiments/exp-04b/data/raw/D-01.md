## Root Cause

**Two workers are claiming the same task and both calling `complete_task` on it. The second `INSERT INTO task_results` hits the `UNIQUE` constraint on `task_id`.**

### Why it's happening

`acquire_task` has a classic **check-then-act race condition**:

```python
# Worker A and Worker B both execute this SELECT at nearly the same time
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    ...
)
# Both get the same row back — status is still 'pending' for both reads

# Both then UPDATE the same task to 'running'
db.execute("UPDATE task_queue SET status = 'running' ...")
db.commit()
```

With per-worker SQLite connections (the old setup), each worker had its own connection and SQLite's file-level locking made this collision unlikely in practice. **The migration to a shared connection pool removed that accidental serialization.** Multiple workers now read concurrently through pooled connections with no isolation between the SELECT and the UPDATE.

Both workers proceed to `execute_task` and then `complete_task`. The first one inserts into `task_results` successfully. The second one hits the `UNIQUE` constraint on `task_id` and throws — which also explains the **missing results**: the second worker's `complete_task` fails mid-function, so the `UPDATE task_queue SET status = 'complete'` that precedes the INSERT either doesn't commit or leaves the task in an inconsistent state depending on error handling.

### Why tests didn't catch it

Every test runs a single worker against an isolated in-memory DB. There is no concurrent worker test. The race is invisible under sequential execution.

---

## What Needs to Change

The fix is to make the claim atomic. There are two good options:

### Option 1 — Atomic UPDATE with a subquery (preferred, no schema change)

```python
def acquire_task(worker_id: int):
    """Atomically claim a task in a single statement."""
    # Single statement: no window between read and write
    result = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    row = result.fetchone()
    return dict(row) if row else None
```

`RETURNING` is supported in SQLite 3.35+ and most production-grade databases. The SELECT and UPDATE are a single atomic operation — if two workers race, one gets the row and the other gets zero rows back.

### Option 2 — Optimistic locking with a `WHERE status = 'pending'` guard on the UPDATE

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    result = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard added
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, rowcount is 0 — back off
    if result.rowcount == 0:
        return None

    return dict(row)
```

This is less clean than Option 1 (still two round-trips) but works on older SQLite versions that don't support `RETURNING`.

---

## Secondary Recommendation

Add a guard in `complete_task` as a safety net regardless of which fix is chosen:

```python
db.execute(
    "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
    " VALUES (?, ?, ?)",
    (task_id, json.dumps(result), datetime.utcnow()),
)
```

`INSERT OR IGNORE` won't fix the race — it just stops the crash and prevents the downstream gap. It's a backstop, not the fix.

|---|---|
| **Root cause** | `acquire_task` SELECT + UPDATE is not atomic; concurrent workers claim the same task |
| **Why it's new** | Shared connection pool removed the accidental serialization of per-worker SQLite connections |
| **Why tests missed it** | No concurrent worker tests exist |
| **Fix** | Make the claim a single atomic `UPDATE ... WHERE id = (SELECT ...) RETURNING` |
| **Risk of fix** | Low — localized to one function, no schema change required |
