## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. There is no atomic claim on the row. Multiple workers can read the same `pending` task in the window between the `SELECT` and the `UPDATE`, both proceed to `execute_task`, and both attempt to `INSERT` into `task_results` for the same `task_id`. The unique constraint catches the second write. The first write may or may not have committed by then, which is why you're also seeing gaps — the loser's `INSERT` fails and its result is lost entirely.**

---

### The Exact Sequence

```
Worker A                                Worker B
────────────────────────────────────────────────────────────────
SELECT ... WHERE status = 'pending'
  → returns task_id = 42
                                        SELECT ... WHERE status = 'pending'
                                          → returns task_id = 42  ← same row
UPDATE task_queue SET status = 'running'
  WHERE id = 42
db.commit()
                                        UPDATE task_queue SET status = 'running'
                                          WHERE id = 42
                                        db.commit()
execute_task(task_id=42) ...
                                        execute_task(task_id=42) ...
INSERT INTO task_results (task_id=42)
db.commit()  ← succeeds
                                        INSERT INTO task_results (task_id=42)
                                        db.commit()  ← IntegrityError ← this is your alert
```

The `UPDATE` in step two does not prevent the race. Both workers have already read the row before either update commits. SQLite's default isolation does not make the `SELECT` + `UPDATE` pair atomic. With the new shared connection pool, multiple workers are now hitting the same database concurrently in a way the old per-worker connections never allowed. The migration four days ago is what exposed this. The bug was always there.

---

### Why the Tests Don't Catch It

Every test runs a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in the test environment. All 34 tests pass and will continue to pass. This is expected. The test suite is not wrong — it is incomplete.

---

### What Needs to Change

The fix must make the claim atomic. There are two viable approaches depending on what you can deploy right now.

**Option 1 — Atomic claim with a conditional UPDATE (preferred, no schema change)**

Flip the logic. Don't select then update. Update first, targeting a `pending` row, and let the database tell you whether you got it.

```python
def acquire_task(worker_id: int):
    # Atomically claim a task by updating it directly.
    # Only one writer can win the row lock.
    claimed_at = datetime.utcnow()
    
    # SQLite supports UPDATE with ORDER BY + LIMIT via a subquery.
    rowcount = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, claimed_at),
    )
    db.commit()

    if rowcount == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
```

The subquery runs inside the `UPDATE` statement. SQLite serializes writes. The first worker to execute this statement locks the row. The second worker's `UPDATE` finds zero `pending` rows matching that `id` and returns `rowcount = 0`. No second worker proceeds.

**Option 2 — Serialize acquisition with an explicit advisory lock**

If you cannot change the query shape right now, wrap acquisition in a process-level mutex. This is a stopgap, not a permanent fix, and it only works if all workers are in the same process.

```python
import threading
_acquire_lock = threading.Lock()

def acquire_task(worker_id: int):
    with _acquire_lock:
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
        )
        if not row:
            return None
        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )
        db.commit()
    return dict(row)
```

Do not use this if workers are separate processes. The lock does not cross process boundaries.

### The Missing Gap Problem

The `IntegrityError` in `complete_task` is currently unhandled. When the second worker's `INSERT` fails, the exception propagates and the result is dropped. Regardless of which fix you ship for acquisition, add a guard here so a duplicate attempt does not silently discard data:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ?",
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` is a secondary defense. It is not the fix. The fix is making acquisition atomic. If you only ship `INSERT OR IGNORE`, you have silenced the alert while the race is still running — both workers are still doing redundant work, and whichever result wins is arbitrary.
