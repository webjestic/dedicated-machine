## Root Cause

**`acquire_task` has a check-then-act race condition with no atomic protection.**

Here is the exact sequence that produces the `IntegrityError`:

```
Worker A                              Worker B
--------                              --------
SELECT ... WHERE status = 'pending'
                                      SELECT ... WHERE status = 'pending'
-- both read the same row, e.g. id=42 --

UPDATE task_queue SET status='running'
WHERE id = 42
COMMIT

                                      UPDATE task_queue SET status='running'
                                      WHERE id = 42
                                      COMMIT

execute_task(task_id=42)
complete_task(42, result)
  INSERT INTO task_results (task_id=42) ✅

                                      execute_task(task_id=42)
                                      complete_task(42, result)
                                        INSERT INTO task_results (task_id=42) 💥
                                        IntegrityError: UNIQUE constraint failed
```

The `SELECT` and the `UPDATE` are two separate statements with no lock held between them. Under the shared connection pool introduced four days ago, two workers can read the same pending row in the gap before either one commits the status change. Both proceed to execute and attempt to write to `task_results`. The unique constraint on `task_id` stops the second write — which is why you see the error — but the first write succeeded, which is why results are present for some tasks and missing for others (the second worker's result is lost entirely).

**Why tests don't catch this:** Every test runs a single worker against an in-process database with no concurrency. The race window does not exist in that environment. All 34 tests pass and always will under those conditions.

**Why it started four days ago:** The previous per-worker SQLite connections were isolated by definition — each worker had its own connection and its own file lock, so two workers could not interleave reads on the same connection. The shared pool removed that accidental serialization and exposed the window that was always latent in the logic.

## What Needs to Change

The fix must make the claim operation atomic. There are two viable approaches:

### Option 1 — Atomic UPDATE with WHERE clause (preferred, no schema change)

Collapse the SELECT and UPDATE into a single statement. Use `RETURNING` to get the claimed row back in one round-trip.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

The subquery and the update execute as one atomic operation inside the database engine. If two workers race, one gets the row and the other gets zero rows back. There is no window.

> **Note:** `RETURNING` requires SQLite 3.35+ (released March 2021). Verify the production SQLite version before deploying. If it is older, use Option 2.

### Option 2 — Explicit transaction with SELECT ... WHERE and re-check

If `RETURNING` is not available, hold an explicit write transaction and re-check status inside it:

```python
def acquire_task(worker_id: int):
    with db.transaction():  # acquires write lock immediately
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
        )
        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ? AND status = 'pending'",  # re-check status
            (worker_id, datetime.utcnow(), row["id"]),
        )

        if db.rowcount() == 0:
            # Another worker claimed it between our read and write
            return None

    return dict(row)
```

The `AND status = 'pending'` guard on the UPDATE means a worker that loses the race updates zero rows and backs off cleanly instead of proceeding to execute a task it does not own.

## Secondary Issue Worth Flagging (Not the Fire, But Fix It)

`complete_task` updates the queue status and inserts the result in two separate `db.execute` calls under one `db.commit`. If the process dies between those two statements, the task is marked complete but no result is recorded — silent data loss. Wrap both in an explicit transaction so they commit together or not at all. This is not causing today's incident but it is the next one.

## For the Sync Call

> The shared connection pool introduced four days ago exposed a pre-existing race condition in `acquire_task`. Two workers can read the same pending task before either one claims it, both execute it, and both attempt to insert into `task_results`. The unique constraint catches the second insert and throws the `IntegrityError`. The fix is to make the claim atomic — one SQL statement that selects and updates in a single operation so the database engine serializes the race, not our application code. The tests pass because they run one worker at a time and never exercise the concurrent path.
