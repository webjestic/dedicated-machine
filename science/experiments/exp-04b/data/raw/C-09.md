# Root Cause Analysis — `IntegrityError: UNIQUE constraint failed: task_results.task_id`

## The Failure, Read Backward

The constraint violation tells us exactly what happened: two workers wrote a result row for the same `task_id`. The unique constraint on `task_results.task_id` is doing its job — it caught the double-write. The gaps in the reporting pipeline are the other side of that coin: one of those two writes was rolled back, so the result is gone.

The question is how two workers ended up owning the same task. The answer is in `acquire_task`.

---

## The Race Condition

`acquire_task` is a **check-then-act** sequence with no atomicity guarantee:

```python
# Step 1 — CHECK: read a pending task
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# <-- the window lives here -->

# Step 2 — ACT: claim it
db.execute(
    "UPDATE task_queue"
    " SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
```

Between the `SELECT` and the `UPDATE`, another worker can execute the same `SELECT` and read the same row. Both see `status = 'pending'`. Both proceed to `UPDATE`. Both claim the task. Both call `complete_task`. The second `INSERT INTO task_results` hits the unique constraint.

This is a textbook **TOCTOU** (time-of-check to time-of-use) race.

---

## Why It Wasn't Visible Until Now

Three things aligned to hide this in testing and in single-worker production:

**1. The tests run a single worker sequentially.**
Every test calls `acquire_task` once, with no concurrency. There is no window to race through. All 34 tests pass. The tests are not wrong — they are testing the wrong threat model.

**2. The migration to a shared connection pool is what opened the door.**
With per-worker SQLite connections, each worker had its own connection and its own implicit serialization. The shared pool from `db-pool 2.1` means multiple workers are now issuing reads and writes against the same database through the same pool. The SELECT from worker A and the SELECT from worker B can now interleave in a way they physically could not before.

**3. It's intermittent because timing-dependent.**
Under low load, workers finish tasks before the next worker polls. Under high traffic, the queue fills faster than workers drain it, polling frequency increases, and the window gets hit. That matches the symptom: errors appear during high-traffic windows.

The ORM upgrade to 3.2.1 is worth auditing for changed transaction isolation defaults, but it is not the root cause. The root cause was always there. The pool migration made it observable.

---

## What Needs to Change

### Fix 1 — Make the claim atomic (required)

The SELECT and UPDATE must be a single atomic operation. The standard approach is `UPDATE ... WHERE status = 'pending' ... RETURNING` or its equivalent, so the claim only succeeds if the row is still unclaimed at the moment of the write:

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    
    # SQLite supports RETURNING as of 3.35. Verify your version.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The subquery runs inside the `UPDATE`. The database evaluates the `WHERE status = 'pending'` condition at write time, under whatever isolation the engine provides. If two workers race, one gets the row and one gets zero rows affected. There is no window between the check and the act because there is no separate check.

If your SQLite version predates `RETURNING` support, the equivalent is:

```python
def acquire_task(worker_id: int):
    # Claim first, then read — never read then claim.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if db.rows_affected() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

### Fix 2 — Add a WHERE guard to `complete_task` (defense in depth)

Even after Fix 1, add a predicate to the result insert so a duplicate write fails loudly and early rather than corrupting state silently:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # only complete what we own
        (datetime.utcnow(), task_id),
    )
    if db.rows_affected() == 0:
        # Another worker already completed this task. Do not write the result.
        logger.error(
            "complete_task called for task %s but it is no longer running — "
            "possible double-claim, skipping result write",
            task_id,
        )
        db.rollback()
        return

    db.execute(
        "INSERT INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

### Fix 3 — Add a concurrency test (close the test gap)

The test suite has no coverage for the scenario that caused this incident. Add one:

```python
import threading

def test_acquire_task_only_one_worker_claims_under_concurrency(db, seed_tasks):
    """Two workers racing to acquire the same task — only one should win."""
    claimed = []
    errors = []

    def worker(wid):
        try:
            task = acquire_task(worker_id=wid)
            if task:
                claimed.append((wid, task["id"]))
        except Exception as e:
            errors.append(e)

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(1, 6)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    assert not errors, f"Workers raised exceptions: {errors}"

    task_ids = [task_id for _, task_id in claimed]
    assert len(task_ids) == len(set(task_ids)), (
        f"Same task claimed by multiple workers: {claimed}"
    )
```

This test would have caught the race before the migration shipped.

---

## Summary

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: SELECT then UPDATE with no atomicity guarantee |
| **Trigger** | Migration to shared connection pool allowed concurrent workers to interleave reads |
| **Symptom** | Two workers claim the same task → two writes to `task_results` → UNIQUE constraint violation → one result lost |
| **Why tests missed it** | No concurrent test coverage; single-threaded tests cannot expose a race |
| **Required fix** | Collapse SELECT + UPDATE into a single atomic `UPDATE ... WHERE id = (SELECT ...)` |
| **Defense in depth** | Guard `complete_task` with `WHERE status = 'running'` and skip the result write if the update claims zero rows |

The constraint violation is not the bug. It is the database correctly refusing to let the bug corrupt your data. Fix the window in `acquire_task` and the constraint will never be reached.
