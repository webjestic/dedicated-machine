## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. There is no atomic claim on the row. Two workers can read the same `pending` task in the window between the `SELECT` and the `UPDATE`, both proceed to `execute_task`, and both attempt to `INSERT` into `task_results` for the same `task_id`. The second insert hits the unique constraint and throws.**

---

## The Exact Sequence

```
Worker A                                    Worker B
--------                                    --------
SELECT ... WHERE status = 'pending'
                                            SELECT ... WHERE status = 'pending'
  → returns task_id = 42                      → returns task_id = 42 (same row)

UPDATE task_queue SET status = 'running'
  WHERE id = 42
db.commit()
                                            UPDATE task_queue SET status = 'running'
                                              WHERE id = 42
                                            db.commit()

execute_task(42) ...
                                            execute_task(42) ...

INSERT INTO task_results (task_id, ...)
  VALUES (42, ...)                          INSERT INTO task_results (task_id, ...)
db.commit() ✅                                VALUES (42, ...)
                                            db.commit() 💥 IntegrityError
```

Both workers see `status = 'pending'` because neither `SELECT` is part of the same atomic operation as the `UPDATE`. The commit on Worker A's claim does not help — by the time it lands, Worker B has already read the row.

---

## Why Tests Don't Catch It

The tests run a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in the test environment. All 34 tests pass and always will under these conditions. This is exactly the class of bug that survives CI.

---

## Why It Started Now

The migration to `db-pool 2.1` four days ago is the trigger. Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per connection, so the workers were accidentally protected by isolation. With a shared connection pool, multiple workers are now genuinely concurrent against the same database state. The race that was always latent in the logic is now reachable.

---

## What Needs to Change

There are two valid fixes. Pick one based on what the database supports.

### Fix 1 — Atomic claim with a `WHERE` guard (preferred, works in SQLite and Postgres)

Collapse the `SELECT` and `UPDATE` into a single statement. Use the `status = 'pending'` predicate on the `UPDATE` itself, then check whether a row was actually modified. Only one writer can win that race.

```python
def acquire_task(worker_id: int):
    # The WHERE clause is the lock. Only one UPDATE will match
    # for a given task_id. The SELECT is gone.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only the worker that actually changed a row gets the task.
    if db.rows_affected() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The key property: `rows_affected() == 0` means another worker got there first. No second insert attempt, no integrity error.

### Fix 2 — Explicit transaction with `SELECT FOR UPDATE` (Postgres only)

If you are on Postgres, wrap the select and update in a transaction and use `FOR UPDATE SKIP LOCKED`. This is the idiomatic queue-worker pattern on Postgres and handles the priority ordering cleanly.

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # Postgres only
        )
        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )
    return dict(row)
```

`SKIP LOCKED` means a worker will skip any row another worker has already locked, rather than blocking or reading stale state.

---

## Secondary Issue to Address After the Incident

`complete_task` also has no guard against a task that was claimed by a second worker. Even after fixing `acquire_task`, if a task takes long enough that it gets re-queued and re-claimed, you can still get a double insert. Add a `WHERE status = 'running' AND worker_id = ?` predicate to the `UPDATE` in `complete_task` and check `rows_affected()` before executing the `INSERT`. If the row is no longer owned by this worker, do not write the result.

---

## For the Call

> The race is in `acquire_task`. The `SELECT` and `UPDATE` are two separate statements with no lock between them. Two workers read the same pending task, both claim it, both finish it, and both try to insert into `task_results`. The second insert fails on the unique constraint. The migration to the shared connection pool four days ago is what made this reachable — before that, per-worker connections were serializing access by accident. The fix is to make the claim atomic: one `UPDATE` statement with the `pending` filter in the `WHERE` clause, check `rows_affected`, and only proceed if this worker actually won the row. Tests pass because they are single-worker and single-threaded. The race does not exist in that environment.
