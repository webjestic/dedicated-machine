## Root Cause

**Two workers are claiming the same task and both calling `complete_task` on it. The second `INSERT INTO task_results` hits the `UNIQUE` constraint on `task_id` and throws.**

---

### Why It's Happening

The `acquire_task` function has a **check-then-act race condition** — a classic TOCTOU bug:

```python
# Worker A and Worker B both execute this SELECT at nearly the same time.
# Both see the same row with status = 'pending'. Both get the same task id.
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Both then UPDATE that same row to 'running'.
# In SQLite this serializes, so both UPDATEs succeed — last one wins.
# Both workers now believe they own the task.
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
return dict(row)  # Both workers return the same task dict.
```

Both workers proceed to `execute_task` and then `complete_task`. The first worker's `INSERT INTO task_results` succeeds. The second hits the unique constraint and throws.

---

### Why It Only Appeared Now

This race always existed in theory, but it was **harmless before the migration** because each worker had its own SQLite connection. SQLite's file-level locking meant concurrent writes were effectively serialized and the window for the race was negligible.

The migration to a **shared connection pool** (`db-pool 2.1`) means multiple workers now share connections to the same database with real concurrency. The SELECT and UPDATE are no longer atomic from the database's perspective — the gap between them is now routinely exploited under load.

The tests pass because they run a single worker sequentially. There is no concurrent worker in any test, so the race is never triggered.

---

### What Needs to Change

The fix is to make the claim operation **atomic** — the SELECT and UPDATE must happen as a single unit that only one worker can win.

**Option 1 — Conditional UPDATE (recommended, minimal change):**

```python
def acquire_task(worker_id: int):
    # First, grab a candidate task id.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Only claim it if it's STILL pending. One worker wins; the rest update 0 rows.
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard added here
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_affected == 0:
        return None  # Another worker claimed it first. Loop will retry.

    return dict(row)
```

The `AND status = 'pending'` guard turns the UPDATE into a compare-and-swap. Exactly one worker will see `rowcount == 1`; all others see `0` and back off.

**Option 2 — Single atomic UPDATE with RETURNING (cleaner, requires SQLite 3.35+ or Postgres):**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This eliminates the SELECT/UPDATE gap entirely. One statement, one winner.

---

### Also Worth Fixing in `complete_task`

Add a guard here too so a late-arriving duplicate write fails gracefully rather than crashing:

```python
db.execute(
    "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
    " VALUES (?, ?, ?)",
    (task_id, json.dumps(result), datetime.utcnow()),
)
```

`INSERT OR IGNORE` is a **safety net**, not the fix. The real fix is in `acquire_task`. But this stops the error from surfacing as an unhandled exception while the deploy goes out.
