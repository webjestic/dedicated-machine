## Root Cause

**The `acquire_task` function contains a classic check-then-act race condition. There is no atomic claim on the row. Multiple workers can read the same `pending` task in the window between the `SELECT` and the `UPDATE`, both proceed to `complete_task`, and both attempt to `INSERT` into `task_results`. The second insert hits the unique constraint on `task_id` and throws. The first worker's result lands. The second worker's result is lost. That is your gap in the reporting pipeline.**

---

### The Sequence That Produces This Failure

```
Time →

Worker 1:  SELECT ... WHERE status = 'pending' → gets task #42
Worker 2:  SELECT ... WHERE status = 'pending' → gets task #42   ← same row, status still 'pending'
Worker 1:  UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 1:  COMMIT
Worker 2:  UPDATE task_queue SET status = 'running' WHERE id = 42  ← succeeds, row already 'running' but no guard
Worker 2:  COMMIT
Worker 1:  execute_task(#42) → result_A
Worker 2:  execute_task(#42) → result_B
Worker 1:  INSERT INTO task_results (task_id=42, ...) → OK
Worker 2:  INSERT INTO task_results (task_id=42, ...) → 💥 IntegrityError: UNIQUE constraint failed
```

The `UPDATE` in `acquire_task` does not check the current status. It will happily claim a row that is already `running`. Nothing in this function is atomic. The SELECT and UPDATE are two separate round-trips with no isolation between them.

---

### Why Tests Pass and Production Burns

The tests run a single worker against a single in-process connection. There is no concurrency. The race window does not exist in that environment. You can run those 34 tests ten thousand times and never see this failure. This is the exact class of bug that CI cannot catch without explicit concurrency tests — and almost nobody writes those.

The migration four days ago is directly relevant. Moving to a shared connection pool means multiple workers are now hitting the same database with overlapping transactions. Before the migration, per-worker SQLite connections with SQLite's default serialized mode may have accidentally serialized enough access to hide this. The pool removed that accidental protection and exposed the window that was always there.

---

### What Needs to Change

The fix must make the claim atomic. There are two standard approaches:

**Option 1 — Conditional UPDATE with row-count check (preferred for SQLite)**

Collapse the SELECT and the claim into a single statement. Do not select first and update second. Update with the condition inline and check whether you actually claimed anything.

```python
def acquire_task(worker_id: int):
    # Attempt to claim atomically. The WHERE clause on status ensures
    # only one worker wins even if multiple attempt simultaneously.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",  # guard re-checked at write time
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only return the task if this worker actually claimed it.
    if db.last_rowcount() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The `AND status = 'pending'` predicate on the `UPDATE` is the lock. SQLite evaluates it at write time under its write serialization. If two workers race, one UPDATE touches one row, the other touches zero rows. The one that gets zero rows returns `None` and moves on.

**Option 2 — Explicit transaction with SELECT FOR UPDATE (if moving to Postgres)**

If the database is or becomes Postgres, use `SELECT ... FOR UPDATE SKIP LOCKED`. This is the purpose-built primitive for exactly this queue pattern.

```sql
BEGIN;
SELECT id, type, payload
  FROM task_queue
 WHERE status = 'pending'
 ORDER BY priority DESC, created_at ASC
 LIMIT 1
   FOR UPDATE SKIP LOCKED;

UPDATE task_queue
   SET status = 'running', worker_id = $1, claimed_at = NOW()
 WHERE id = $2;
COMMIT;
```

`SKIP LOCKED` means a worker that loses the race does not block — it simply skips the locked row and finds the next available one. No contention, no waiting, no duplicate processing.

---

### The Secondary Issue Worth Noting

`complete_task` also has a latent version of this problem. If a task is somehow executed twice (which this race makes possible), the `UPDATE task_queue SET status = 'complete'` will silently succeed for both workers because it has no status guard either. Fix that too once the claim is atomic:

```python
db.execute(
    "UPDATE task_queue"
    " SET status = 'complete', completed_at = ?"
    " WHERE id = ? AND status = 'running'",  # do not complete what you did not run
    (datetime.utcnow(), task_id),
)
```

### Summary for the Team

| | |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`. SELECT and UPDATE are separate statements with no isolation. |
| **Trigger** | Migration to shared connection pool removed accidental serialization that previously hid the window. |
| **Symptom 1** | `IntegrityError` on the second worker's INSERT — the constraint is doing its job correctly. |
| **Symptom 2** | Missing results in reporting — the second worker's result is discarded on the exception. |
| **Fix** | Make the claim a single atomic UPDATE with status guard. Check rowcount to determine whether this worker won the claim. |
| **Tests** | Existing suite cannot catch this. Add a concurrency test that races two threads through `acquire_task` on the same pending task and asserts exactly one claim succeeds. |

The unique constraint on `task_results.task_id` is not the bug. It is the only thing currently preventing silent duplicate processing. Do not relax it.
