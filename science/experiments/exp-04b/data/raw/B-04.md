## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. Multiple workers can read the same `pending` row before any of them writes the `UPDATE`. Both proceed to `execute_task` and then both attempt `INSERT INTO task_results` for the same `task_id`. The unique constraint on `task_results.task_id` fires on the second insert. The first writer wins; the second is dropped. That is your missing data.**

---

### The Exact Sequence

```
Time →

Worker A:  SELECT ... WHERE status = 'pending' → gets task_id = 42
Worker B:  SELECT ... WHERE status = 'pending' → gets task_id = 42   ← same row, status still 'pending'

Worker A:  UPDATE task_queue SET status = 'running' WHERE id = 42
Worker A:  commit

Worker B:  UPDATE task_queue SET status = 'running' WHERE id = 42    ← succeeds, no guard
Worker B:  commit

Worker A:  execute_task(42) → result_A
Worker B:  execute_task(42) → result_B

Worker A:  INSERT INTO task_results (task_id=42, ...)  ← succeeds
Worker B:  INSERT INTO task_results (task_id=42, ...)  ← IntegrityError: UNIQUE constraint failed
```

The `SELECT` and the `UPDATE` are two separate statements with no atomicity guarantee between them. The connection pool makes this dramatically worse — under a single connection, the GIL and connection serialization were providing accidental mutual exclusion. The migration to `db-pool 2.1` removed that accidental protection. The bug was always there. The migration exposed it.

---

### Why the Tests Don't Catch It

Every test runs a single worker against an in-process database. There is no concurrency. The race window never opens. All 34 tests pass and will continue to pass indefinitely. The test suite is not wrong; it is simply not testing the property that is broken.

---

### Why the Other Things You're Looking At Are Not the Root Cause

The `execute_task` deduplication logic is inefficient (O(n²) scan on `summary`) but it is not causing the incident. The ORM version bump is not causing the incident. The payload parsing has no null guard on `payload["user_ids"]` but that would produce a `KeyError`, not an `IntegrityError`. Do not patch those and call it done.

---

### What Needs to Change

The fix must make the claim operation atomic. There are two sound approaches depending on what your database supports.

**Option 1 — Atomic UPDATE with a conditional WHERE clause (preferred for SQLite)**

Eliminate the `SELECT` entirely. Claim the row in a single statement and inspect what was actually claimed.

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""

    # SQLite processes this as a single atomic write.
    # Only one worker can match the subquery and flip the status in one operation.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Now read back what this worker actually claimed.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at >= ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

The readback is slightly awkward. A cleaner structure is to add a `claim_token` column (a UUID written at claim time) so the readback is unambiguous:

```python
import uuid

def acquire_task(worker_id: int):
    token = str(uuid.uuid4())

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?, claim_token = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow(), token),
    )
    db.commit()

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE claim_token = ?",
        (token,),
    )
    return dict(row) if row else None
```

If two workers race, only one `UPDATE` touches the row. The other gets zero rows affected. Each worker reads back by its own token. No collision is possible.

**Option 2 — Explicit locking (if you move to PostgreSQL)**

```sql
SELECT id, type, payload FROM task_queue
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED
```

`SKIP LOCKED` is purpose-built for job queues. SQLite does not support it. If you are on SQLite in production at this traffic level, that conversation also needs to happen, but it is a separate issue from the incident.

### What to Add to the Test Suite After This

```python
import threading

def test_acquire_task_is_exclusive_under_concurrency(db, seed_tasks):
    """Two workers racing must claim different tasks, never the same one."""
    claimed_ids = []
    lock = threading.Lock()

    def worker(wid):
        task = acquire_task(worker_id=wid)
        if task:
            with lock:
                claimed_ids.append(task["id"])

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(5)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # No task ID should appear more than once
    assert len(claimed_ids) == len(set(claimed_ids))
```

This test would have caught the bug before the migration shipped. Add it. Make it part of the required suite.
