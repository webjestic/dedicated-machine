## Root Cause

**`acquire_task` has a check-then-act race condition. Two workers can read the same `pending` row before either one writes `running`. Both claim the task. Both call `complete_task`. The second `INSERT INTO task_results` hits the unique constraint on `task_id`.**

---

### The Exact Sequence

```
Worker 1                                    Worker 2
--------                                    --------
SELECT ... WHERE status = 'pending'
                                            SELECT ... WHERE status = 'pending'
  → returns task #42                          → returns task #42 (same row)

UPDATE task_queue SET status = 'running'
  WHERE id = 42
db.commit()
                                            UPDATE task_queue SET status = 'running'
                                              WHERE id = 42
                                            db.commit()

execute_task(task #42)                      execute_task(task #42)

INSERT INTO task_results (task_id=42)
db.commit()  ← succeeds
                                            INSERT INTO task_results (task_id=42)
                                            db.commit()  ← 💥 IntegrityError
```

The SELECT and the UPDATE are two separate statements with no lock held between them. Under a shared connection pool, two workers can interleave freely in that gap. This is a classic TOCTOU — time-of-check to time-of-use.

---

### Why Tests Pass

Every test runs a single worker against a single in-process connection. There is no concurrency. The race window does not exist in the test environment and never has. The 34 green tests tell you nothing about what happens when two workers share a pool.

---

### Why It Started Four Days Ago

Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per connection, and separate connections meant separate file locks — workers were accidentally protected by isolation. The shared pool removed that accidental serialization. The race was always in the logic. The infrastructure change just made it reachable.

---

### What Needs to Change

There are two viable fixes. Pick one based on what you can ship safely in the next hour.

**Option A — Atomic claim with a conditional UPDATE (ship this now)**

Remove the SELECT entirely. Claim the task in a single statement that only succeeds if the row is still `pending`:

```python
def acquire_task(worker_id: int):
    # Single atomic operation — no window between check and claim
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only fetch what we just claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at >= ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

The subquery and the UPDATE execute as one atomic operation in SQLite. If two workers race, one UPDATE touches one row, the other touches zero rows. Check `db.rowcount` after the execute — if it is 0, no task was claimed, return `None`.

Cleaner version using rowcount:

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, claimed_at),
    )
    db.commit()

    if db.rowcount == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at = ?",
        (worker_id, claimed_at),
    )
    return dict(row) if row else None
```

**Option B — Explicit transaction with SELECT FOR UPDATE**

If you move off SQLite to Postgres, use `SELECT ... FOR UPDATE SKIP LOCKED`. That is the correct primitive for this pattern at scale. Do not do this in the next nine minutes.

---

### The Missing Results

The gaps in the reporting pipeline are the losing worker's results. Worker 2 throws `IntegrityError`. `complete_task` has no exception handling, so the `UPDATE task_queue SET status = 'complete'` that precedes the failing INSERT may or may not have committed depending on transaction boundaries in the pool. Either way, the result row is never written. The task may be left in an inconsistent state — marked complete in the queue but absent from `task_results`.

After the fix ships, you will need a reconciliation query to find tasks where `status = 'complete'` in `task_queue` but no corresponding row exists in `task_results`, and requeue them.

---

### One-Line Summary for the Sync Call

> Two workers are reading the same pending task before either one locks it. Both execute it. The second write to `task_results` fails the unique constraint. Fix is to make the claim a single atomic UPDATE with no preceding SELECT.
