# Root Cause Analysis

## The Short Answer

**Two workers can claim the same task simultaneously.** `acquire_task` is a check-then-act operation with no atomicity guarantee. Both workers read the same `pending` row, both update it to `running`, both call `complete_task`, and the second `INSERT INTO task_results` hits the unique constraint. The `UNIQUE constraint failed` error is the symptom. The race window in `acquire_task` is the cause.

---

## The Failure Sequence

Here is the exact interleaving that produces the incident:

```
Time →

Worker 1: SELECT ... WHERE status = 'pending' LIMIT 1  → returns task_id = 42
                                                Worker 2: SELECT ... WHERE status = 'pending' LIMIT 1  → returns task_id = 42
Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 1: COMMIT
                                                Worker 2: UPDATE task_queue SET status = 'running' WHERE id = 42
                                                Worker 2: COMMIT
Worker 1: execute_task(42) ...
                                                Worker 2: execute_task(42) ...
Worker 1: INSERT INTO task_results (task_id=42, ...)
Worker 1: COMMIT  ✓
                                                Worker 2: INSERT INTO task_results (task_id=42, ...)
                                                Worker 2: COMMIT  ✗  IntegrityError
```

Worker 2's `execute_task` result is lost. That is your reporting gap. The unique constraint is doing its job — it is the only thing preventing a duplicate result row. The application logic is not.

---

## Why the Tests Don't Catch This

The test suite runs one worker against one in-process database connection. There is no concurrency. The race window does not exist in that environment, so all 34 tests pass cleanly and will continue to pass. This is not a test quality problem you can solve by adding more assertions to the existing test structure. You need concurrent test execution against a shared connection to reproduce it, and that is a different kind of test entirely.

---

## Why the Recent Migration Made It Worse

This race almost certainly existed before the migration, but it was masked. Per-worker SQLite connections with WAL mode or serialized access would have naturally serialized these reads. The shared connection pool introduced by `db-pool 2.1` means multiple workers are now genuinely interleaving operations against the same connection state. The ORM upgrade to 3.2.1 may also have changed transaction isolation defaults — that is worth checking in the changelog, but it is secondary. The pool change is what opened the window wide enough to trigger consistently.

---

## The Fix

The SELECT and UPDATE must be atomic. There are two standard approaches depending on your database.

### Option A — Atomic UPDATE with RETURNING (preferred for PostgreSQL/SQLite 3.35+)

```python
def acquire_task(worker_id: int):
    """
    Claim a task atomically. The UPDATE with RETURNING eliminates the
    window between the SELECT and the UPDATE entirely. If two workers
    race, only one UPDATE will match the row — the other returns nothing.
    """
    row = db.query_one(
        """
        UPDATE task_queue
        SET status = 'running',
            worker_id = ?,
            claimed_at = ?
        WHERE id = (
            SELECT id FROM task_queue
            WHERE status = 'pending'
            ORDER BY priority DESC, created_at ASC
            LIMIT 1
        )
        RETURNING id, type, payload
        """,
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if not row:
        return None

    return dict(row)
```

The subquery and the UPDATE execute as a single statement. There is no gap. One worker gets the row; the other gets nothing and loops.

### Option B — Optimistic locking with a WHERE guard on status

If `RETURNING` is not available in your environment:

```python
def acquire_task(worker_id: int):
    """
    Read the candidate row, then UPDATE only if it is still pending.
    The WHERE status = 'pending' guard means the second writer's UPDATE
    matches zero rows. Check rowcount to confirm the claim succeeded.
    """
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- the guard
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_affected == 0:
        # Another worker claimed it between our SELECT and UPDATE.
        # Return None and let the caller retry on the next loop tick.
        return None

    return dict(row)
```

This is still two statements, so there is technically still a window — but the `AND status = 'pending'` guard makes the second writer's UPDATE a no-op. `rowcount == 0` means you lost the race; return `None` and retry. This is safe as long as your database provides read-committed isolation or better, which SQLite and PostgreSQL both do by default.

Option A is cleaner. Option B is the fallback if your ORM or SQLite version does not support `RETURNING`.

---

## What to Do Right Now

1. **Deploy Option A or B immediately.** The race is active in production on every high-traffic window.
2. **Audit `task_results` for duplicate `task_id` values** — the unique constraint blocked the second insert, but verify no duplicates slipped through before the constraint was in place or during any migration window.
3. **Cross-reference the reporting gaps** against the error log timestamps. Every `IntegrityError` corresponds to a task whose result was computed by Worker 2 and then discarded. Those tasks may need to be re-queued depending on whether Worker 1's result was valid.
4. **Check `db-pool 2.1` release notes** for default transaction isolation level. If it changed from serializable to read-committed, that is a contributing factor worth documenting in the postmortem even though the fix above makes it irrelevant.
5. **Do not patch `execute_task`'s deduplication logic or `complete_task` directly.** Those are not the problem. The unique constraint on `task_results.task_id` is correct and should stay. The insert should never be reached twice for the same task — fix the acquisition, not the insertion.

---

## Postmortem Note

The visible issue is the `IntegrityError`. The actual issue is that task ownership is never established atomically. The unique constraint is working exactly as designed — it is the last line of defense, and it held. The fix moves the defense to where it belongs: at the moment of acquisition.
