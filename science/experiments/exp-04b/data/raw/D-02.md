## Root Cause

**Two workers are claiming the same task and both calling `complete_task` on it. The second `INSERT INTO task_results` hits the `UNIQUE` constraint on `task_id` and throws.**

---

### Why This Is Happening

The `acquire_task` function has a **check-then-act race condition** — a classic TOCTOU bug:

```python
# Worker A and Worker B both execute this SELECT at nearly the same time
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Both see the same row with status = 'pending'
# Both then issue this UPDATE — nothing in the SELECT prevents it
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
```

Both workers read the same pending task, both update it to `running`, both execute it, and both call `complete_task`. The first `INSERT INTO task_results` succeeds. The second one violates the unique constraint on `task_id`.

---

### Why Tests Don't Catch It

The tests run **sequentially against a single connection** — there is no concurrency, so the race never occurs. The 34 tests passing is not evidence the code is correct under concurrent load. This is a concurrency bug that only surfaces when multiple workers run simultaneously, which is exactly what the shared connection pool migration enabled at scale.

---

### Why It Got Worse Four Days Ago

Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per-connection, and separate connections to the same file get write-level locking that incidentally slowed the race window down enough to make collisions rare or impossible in practice. The new shared connection pool removes that accidental serialization — workers now genuinely race through the SELECT/UPDATE gap.

---

### The Fix

Replace the SELECT + UPDATE with a **single atomic UPDATE...RETURNING** (or equivalent) so a task can only be claimed by one worker:

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )

    if not row:
        return None

    db.commit()
    return dict(row)
```

The subquery inside the `UPDATE` and the `RETURNING` clause execute as a single statement. The database's row-level lock on the `UPDATE` target means only one worker wins — the other gets back zero rows and returns `None`.

**If the ORM or SQLite version doesn't support `UPDATE...RETURNING`**, the fallback is an explicit transaction with a `SELECT ... FOR UPDATE` or a status field CAS (compare-and-swap):

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # skip rows another worker already locked
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ? AND status = 'pending'",  # AND status guard is a safety net
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

---

### The Missing Results

The gaps in the reporting pipeline are the tasks whose `complete_task` call was the *second* one — the `IntegrityError` aborted that transaction, so neither the `UPDATE task_queue SET status = 'complete'` nor the `INSERT INTO task_results` committed. Those tasks are currently stuck in `status = 'running'` with no result row. A one-time cleanup query will be needed after the fix deploys to reset them to `pending` so they re-run:

```sql
UPDATE task_queue
SET status = 'pending', worker_id = NULL, claimed_at = NULL
WHERE status = 'running'
  AND id NOT IN (SELECT task_id FROM task_results);
```

|---|---|
| **Root cause** | Non-atomic claim in `acquire_task` — SELECT then UPDATE is not a single operation, two workers can claim the same task |
