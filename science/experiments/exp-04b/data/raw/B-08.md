## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. Multiple workers can read the same pending task row before any of them writes the status update that would exclude it from subsequent reads.**

This is not a bug in `complete_task`. The `UNIQUE constraint` violation is the symptom. The wound is in `acquire_task`.

---

### The Failure Sequence

Here is exactly what is happening under concurrent load:

```
Time →        Worker 1                          Worker 2
──────────────────────────────────────────────────────────────────
T1            SELECT ... WHERE status='pending'
              → returns task_id=42

T2                                              SELECT ... WHERE status='pending'
                                                → returns task_id=42
                                                (status is still 'pending')

T3            UPDATE task_queue SET status='running'
              WHERE id=42
              COMMIT

T4                                              UPDATE task_queue SET status='running'
                                                WHERE id=42
                                                COMMIT

T5            execute_task(42) → result_A

T6                                              execute_task(42) → result_B

T7            INSERT INTO task_results (task_id=42, ...)
              COMMIT ✓

T8                                              INSERT INTO task_results (task_id=42, ...)
              💥 IntegrityError: UNIQUE constraint failed
```

Between T1 and T3 there is an unprotected window. The `SELECT` and the `UPDATE` are two separate statements with a commit between them. Any worker that executes its `SELECT` inside that window sees the same row. The `UPDATE` does not protect against this because SQLite's default isolation does not prevent two concurrent reads from both seeing `status = 'pending'` before either write lands.

---

### Why This Started Four Days Ago

Before the migration, each worker had its own SQLite connection. SQLite serializes writes per connection, and with isolated connections the workers were effectively serialized at the file lock level — they could not interleave reads and writes on the same row this way. It was accidental protection, not intentional design.

The shared connection pool removed that accidental serialization. Now multiple workers genuinely execute concurrently against the same database state. The race that was always latent in the logic became observable.

The ORM upgrade is a distractor. The tests pass because they run a single worker against a single connection with no concurrency. They cannot catch this. No unit test of `acquire_task` in isolation will catch it. You need a concurrent integration test to reproduce it.

---

### What Needs to Change

The fix is to make the claim atomic. The `SELECT` and the `UPDATE` must be a single operation, not two.

**Option 1 — Atomic UPDATE with RETURNING (preferred if your SQLite version supports it, ≥3.35)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

The subquery and the update execute as one atomic statement. A row either gets claimed by exactly one worker or it does not get claimed at all. There is no window.

**Option 2 — Optimistic locking with a WHERE guard on the UPDATE**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # guard here
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Another worker claimed it first. Back off and retry.
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means the update is a no-op if another worker already claimed the row. Checking `rowcount` tells you whether you won the race. This is less clean than Option 1 because it requires the caller to handle the miss, but it works on older SQLite versions.

---

### What Is Not the Root Cause

- The `UNIQUE constraint` on `task_results.task_id` is working correctly. Do not remove it. It is the only thing preventing silent duplicate results from reaching the reporting pipeline.
- The missing results in the downstream pipeline are caused by the worker that loses the constraint violation. Its `complete_task` call raises before the commit lands, so that worker's result is never written. The task may or may not be retried depending on your error handling, which I do not see in this code.
- The deduplication logic in `execute_task` is inefficient but it is not causing this incident. Leave it for a separate ticket.

---

### Immediate Action

Deploy Option 1 or Option 2. Option 1 is cleaner and eliminates the window entirely. After deploy, confirm the `IntegrityError` rate drops to zero and verify the reporting pipeline gaps stop accumulating. Then audit whether any tasks that failed mid-flight during the incident window need to be replayed.
