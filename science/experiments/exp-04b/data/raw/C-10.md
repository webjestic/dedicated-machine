# Root Cause Analysis — `IntegrityError: UNIQUE constraint failed: task_results.task_id`

## The Short Answer

`acquire_task` has a **check-then-act race condition**. Multiple workers can read the same `pending` task in the window between the `SELECT` and the `UPDATE`. Both proceed to execute it. Both attempt to write to `task_results`. The second write hits the unique constraint. This is a classic **TOCTOU (Time of Check to Time of Use)** bug.

---

## The Failure Sequence

Here is exactly what happens at load with multiple workers:

```
Time →

Worker 1: SELECT ... WHERE status = 'pending' → gets task_id = 42
Worker 2: SELECT ... WHERE status = 'pending' → gets task_id = 42   ← same row, status still 'pending'

Worker 1: UPDATE task_queue SET status = 'running' WHERE id = 42
Worker 1: COMMIT

Worker 2: UPDATE task_queue SET status = 'running' WHERE id = 42    ← succeeds silently, row already 'running'
Worker 2: COMMIT

Worker 1: execute_task(42) → result_A
Worker 2: execute_task(42) → result_B

Worker 1: INSERT INTO task_results (task_id=42, ...) → OK
Worker 2: INSERT INTO task_results (task_id=42, ...) → 💥 UNIQUE constraint failed
```

The `UPDATE` on Worker 2 does not fail — it finds the row, updates it, and commits. SQLite and most databases will not raise an error when you update a row that another session already updated. The claim step **looks atomic but is not**. The gap between the `SELECT` and the `UPDATE` is the unprotected window.

---

## Why the Tests Don't Catch It

The tests run a single worker against an in-process database with no concurrency. There is no second worker to race against. The test for `acquire_task_marks_task_running` verifies the state *after* the claim — it never tests what happens when two workers claim simultaneously. The CI badge is green and the bug is invisible.

This is also why it only appeared after the migration. The old per-worker SQLite connections were isolated by file or by process — effectively serialized. The shared connection pool introduced by `db-pool 2.1` means multiple workers are now genuinely concurrent against the same database state.

---

## Secondary Symptom Explained

> *"Some task results are missing from `task_results` entirely."*

When Worker 2 hits the `IntegrityError` inside `complete_task`, the exception unwinds before `db.commit()` is called. The `UPDATE task_queue SET status = 'complete'` that precedes the `INSERT` is part of the same uncommitted transaction — it rolls back too. The task ends up in an ambiguous state: `status = 'running'` with no result written and no worker actively holding it. Depending on whether your queue has a timeout/requeue mechanism, that task either hangs forever or gets picked up again later. That is your reporting gap.

---

## What Needs to Change

### Fix 1 — Make the Claim Atomic (Required)

The `SELECT` and `UPDATE` must be a single atomic operation. The standard approach is to push the condition into the `WHERE` clause of the `UPDATE` and check whether a row was actually modified.

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""
    # First, identify the candidate
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Claim it — but only if it is still pending.
    # If another worker claimed it between the SELECT and here,
    # rowcount will be 0 and we bail out.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",   # ← the critical guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        # Lost the race — another worker claimed this task first
        return None

    return dict(row)
```

The `AND status = 'pending'` predicate turns the update into a compare-and-swap. Exactly one worker will see `rowcount == 1`. Every other worker that raced to the same row sees `rowcount == 0` and backs off cleanly.

**Verify that your `db` wrapper surfaces `rowcount`** — some thin wrappers swallow the cursor. If it does not, you need to fix the wrapper or use a different signal (e.g., `SELECT` the row back and check `worker_id`).

---

### Fix 2 — Collapse to a Single Atomic Statement (Preferred for SQLite)

If your database supports `RETURNING` (SQLite ≥ 3.35, PostgreSQL), eliminate the `SELECT` entirely:

```python
def acquire_task(worker_id: int):
    """Claim the next pending task in a single atomic statement."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

There is no window here. The subquery and the update execute as one statement under the database's own locking. This is the cleanest solution.

---

### Fix 3 — Harden `complete_task` Against Duplicate Writes (Defense in Depth)

Even after fixing the claim, add a guard in `complete_task`. Belt and suspenders:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",   # ← don't stomp a completed task
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",                        # ← idempotent write
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` means a duplicate write is silently discarded rather than raising. This does not fix the race — it limits the blast radius while the real fix is deployed.

---

### Fix 4 — Add a Concurrency Test

The test suite needs a worker that actually races:

```python
import threading

def test_acquire_task_only_one_worker_claims(db, seed_tasks):
    """Two workers racing for the same task — exactly one should win."""
    results = []
    barrier = threading.Barrier(2)

    def worker(worker_id):
        barrier.wait()  # both threads start the SELECT at the same moment
        task = acquire_task(worker_id=worker_id)
        results.append(task)

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(2)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    claimed = [r for r in results if r is not None]
    assert len(claimed) == 1, (
        f"Expected exactly 1 worker to claim the task, got {len(claimed)}"
    )
```

This test would have caught the bug before the migration shipped. Add it now so the next change cannot regress silently.

---

## Deployment Order

1. **Ship Fix 3 immediately** (`INSERT OR IGNORE`, `AND status = 'running'`). Zero risk, stops the bleeding, no logic change.
2. **Ship Fix 1 or Fix 2** in the next deploy. This is the actual fix.
3. **Audit `task_queue` for rows stuck in `status = 'running'`** with no corresponding `task_results` entry — those are the tasks that were orphaned during the incident. Decide whether to requeue or mark them failed.
4. **Add Fix 4** to the test suite before closing the incident.

## Summary

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: `SELECT` then `UPDATE` with no guard on the `UPDATE` |
| **Trigger** | Shared connection pool introduced real concurrency; previously isolated per-worker connections serialized access by accident |
| **Why CI missed it** | All tests are single-worker, single-threaded; the race requires two concurrent callers |
| **Immediate mitigation** | `INSERT OR IGNORE` in `complete_task` |
| **Permanent fix** | `AND status = 'pending'` guard on the claim `UPDATE`, check `rowcount`, or use `UPDATE ... RETURNING` |
| **Secondary damage** | Tasks orphaned in `running` state with no result — needs manual audit |
