## Root Cause

**The `acquire_task` function has a race condition. Multiple workers can claim the same task simultaneously, causing `complete_task` to attempt duplicate inserts into `task_results`.**

---

### What's happening, step by step

```
Worker A                          Worker B
--------                          --------
SELECT ... WHERE status='pending'
                                  SELECT ... WHERE status='pending'
                                  -- sees same row, status still 'pending'
UPDATE ... SET status='running'
                                  UPDATE ... SET status='running'
                                  -- both workers now "own" the same task_id
execute_task(task)
                                  execute_task(task)
INSERT INTO task_results ...
                                  INSERT INTO task_results ...
                                  -- UNIQUE constraint fails on task_id
```

The SELECT and UPDATE in `acquire_task` are two separate operations with no lock between them. Under concurrent load, two workers race through the SELECT before either UPDATE commits. Both see `status = 'pending'`, both claim the task, both call `complete_task`, and the second INSERT explodes.

---

### Why tests pass and prod fails

The tests run a single worker against a single in-process database connection — there is no concurrency. The race condition is physically impossible to trigger in that environment. It only surfaces under real parallel workers.

---

### Why this started four days ago

The migration to a **shared connection pool** is what exposed this. Previously each worker had its own SQLite connection. SQLite serializes writes per-connection, so workers were accidentally protected from each other. With a shared pool, multiple workers now interleave reads and writes against the same database, and the pre-existing race condition became reachable.

The ORM upgrade is a red herring unless its changelog shows changed transaction isolation behavior — the logic bug was always there.

---

### What needs to change

**Fix: make claim atomic using a single `UPDATE ... WHERE status = 'pending'` and check whether a row was actually updated.**

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # Identify the candidate in a subquery, claim it in the same statement.
    # Only one writer can flip status from 'pending' to 'running' for a given row.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only the worker that actually flipped the row gets it back.
    if db.rows_affected() == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

**Additionally, add a guard in `complete_task` as a safety net:**

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ?",
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` stops the crash immediately and prevents the result gaps in the reporting pipeline. It is not a substitute for fixing `acquire_task` — duplicate work still runs — but it contains the blast radius right now.
