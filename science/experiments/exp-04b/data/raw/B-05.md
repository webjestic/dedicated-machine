## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. There is no atomic claim on the row. Multiple workers can read the same `pending` task in the window between the `SELECT` and the `UPDATE`, both proceed to execute it, and both attempt to `INSERT` into `task_results`. The second insert hits the unique constraint on `task_id` and throws. That is your `IntegrityError`. The missing results in the reporting pipeline are the tasks where the second worker's insert was the one that failed — the result was computed but never written.**

---

### The Sequence That Produces This Failure

```
Time →

Worker A:  SELECT ... WHERE status = 'pending' → gets task_id = 42
Worker B:  SELECT ... WHERE status = 'pending' → gets task_id = 42   ← same row, still 'pending'

Worker A:  UPDATE task_queue SET status = 'running' WHERE id = 42
Worker B:  UPDATE task_queue SET status = 'running' WHERE id = 42    ← overwrites A's claim

Worker A:  execute_task(42) ...
Worker B:  execute_task(42) ...

Worker A:  INSERT INTO task_results (task_id, ...) VALUES (42, ...)  ← succeeds
Worker B:  INSERT INTO task_results (task_id, ...) VALUES (42, ...)  ← IntegrityError
```

The window is small. That is why it only surfaces under load. At low concurrency the workers are rarely polling at the same millisecond. Scale up, and it happens constantly.

---

### Why the Tests Don't Catch It

The tests run a single worker against a single in-process database connection with no concurrency. The race window does not exist in that environment. All 34 tests pass. They will always pass. They are not testing the thing that is broken.

---

### Why the Recent Migration Made It Worse

Before the migration, each worker had its own SQLite connection. SQLite's file-level locking serialized concurrent writes by accident — not by design, but the effect was there. The shared connection pool introduced by `db-pool 2.1` removes that incidental serialization. Multiple workers now issue their `SELECT` statements through the pool concurrently with nothing to stop them from reading the same row before either has written back.

The ORM upgrade is not the cause. The connection pool change is what exposed the pre-existing race.

---

### What Needs to Change

The fix must make the claim atomic. There are two standard approaches depending on what your database supports.

**Option 1: Atomic UPDATE with a subquery (preferred, works in SQLite and Postgres)**

Eliminate the `SELECT` entirely. Claim the row in a single statement and return what you claimed.

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    # SQLite processes this UPDATE as a single atomic operation.
    # Only one writer can hold the write lock at the moment of the update,
    # so only one worker will match and claim any given row.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Now fetch what this worker just claimed.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

**Option 2: If your database supports `RETURNING` (Postgres, SQLite ≥ 3.35)**

Cleaner — claim and retrieve in one round trip, no ambiguity about which row this worker owns.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

### What to Add to `complete_task`

Even after fixing `acquire_task`, add a guard in `complete_task`. Defense in depth. If a duplicate somehow gets through — a retry, a bug in a future refactor — you want to know about it explicitly rather than silently overwrite or crash.

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # only complete what this worker owns
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"  # SQLite
        " VALUES (?, ?, ?)",                                                  # or use ON CONFLICT
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` is a safety net, not the fix. Do not treat it as the fix. If you only add this and ship it, you will suppress the `IntegrityError` and the duplicate execution will continue — you will just stop seeing the error while tasks keep running twice and the second result gets silently dropped. That is worse than the current state.

### What to Add to the Test Suite

The existing tests need a concurrency test. Without one this class of bug is invisible.

```python
import threading

def test_acquire_task_no_duplicate_claims(db, seed_tasks):
    """Two workers racing to acquire the same task must not both claim it."""
    claimed_ids = []
    lock = threading.Lock()

    def worker(wid):
        task = acquire_task(worker_id=wid)
        if task:
            with lock:
                claimed_ids.append(task["id"])

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(10)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # Each task ID should appear at most once across all workers.
    assert len(claimed_ids) == len(set(claimed_ids)), (
        f"Duplicate claims detected: {claimed_ids}"
    )
```

### Summary for the Incident Channel

> **Root cause:** `acquire_task` reads a pending task and then updates it in two separate operations. Under concurrent load, multiple workers read the same row before any of them writes back. Both execute the task. Both attempt to insert into `task_results`. The second insert fails with `IntegrityError` on the unique constraint. The connection pool migration removed the incidental SQLite file-lock serialization that was masking this. The race was always there.
>
> **Fix:** Make the claim atomic with a single `UPDATE ... WHERE id = (SELECT ...)`. The `SELECT` and `UPDATE` must be one statement, not two.
>
> **Do not:** Add `INSERT OR IGNORE` as the only change. That hides the symptom and leaves tasks executing twice.
