# Root Cause Analysis

## The Short Answer

`acquire_task` has a classic **check-then-act race condition**. The SELECT and the UPDATE are two separate, unprotected operations. Between them, a second worker can SELECT the same row. Both workers claim the same task, both execute it, and both attempt to INSERT into `task_results` for the same `task_id`. The second INSERT hits the UNIQUE constraint and dies. The result for that task is either duplicated in flight or missing entirely from the table — which matches both symptoms in the incident report.

---

## The Failure Sequence

Here is the exact interleaving that produces the error:

```
Time    Worker 1                                    Worker 2
────────────────────────────────────────────────────────────────────────
T1      SELECT ... WHERE status = 'pending'
        → returns task_id = 42

T2                                                  SELECT ... WHERE status = 'pending'
                                                    → also returns task_id = 42
                                                    (Worker 1's UPDATE hasn't committed yet)

T3      UPDATE task_queue SET status = 'running'
        WHERE id = 42
        COMMIT

T4                                                  UPDATE task_queue SET status = 'running'
                                                    WHERE id = 42
                                                    COMMIT
                                                    (succeeds — UPDATE on an already-running
                                                    row is not blocked by anything here)

T5      execute_task(task_id=42) → result_A

T6                                                  execute_task(task_id=42) → result_B

T7      INSERT INTO task_results (task_id=42, ...)
        COMMIT ✅

T8                                                  INSERT INTO task_results (task_id=42, ...)
                                                    💥 IntegrityError: UNIQUE constraint failed
```

The UPDATE at T4 silently succeeds because there is no guard in the WHERE clause that says "only claim this if it is still pending." The row is already `running`, but the UPDATE does not check for that — it matches on `id` alone.

---

## Why the Tests Don't Catch This

The tests run one worker against one database connection in a single thread. There is no concurrency. The race window does not exist in that environment. All 34 tests pass and always will, right up until you add a second worker.

This is the exact failure mode that a single-threaded test suite cannot see by design. The tests are not wrong — they are just testing the wrong threat model.

---

## Why the Recent Migration Made This Worse

Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes at the file level, which accidentally papered over this bug. With a shared connection pool, multiple workers can now issue reads and writes concurrently against the same database without that accidental serialization. The bug was always there. The migration removed the thing that was hiding it.

---

## What Needs to Change

There are two valid fixes. Pick one based on your database.

### Fix 1 — Atomic claim with a guarded UPDATE (preferred for SQLite and PostgreSQL)

Move the SELECT inside the UPDATE. Claim the task in a single statement that only succeeds if the row is still pending. Return the claimed row in the same operation.

**PostgreSQL:**
```sql
UPDATE task_queue
SET    status = 'running',
       worker_id = ?,
       claimed_at = ?
WHERE  id = (
    SELECT id
    FROM   task_queue
    WHERE  status = 'pending'
    ORDER  BY priority DESC, created_at ASC
    LIMIT  1
    FOR UPDATE SKIP LOCKED          -- the key line
)
RETURNING id, type, payload;
```

`FOR UPDATE SKIP LOCKED` is the correct primitive here. It locks the candidate row at SELECT time so no other worker can see it as a candidate. If two workers race, one gets the row and the other skips it and moves on. No retry logic needed.

**SQLite** (does not support `FOR UPDATE`):

Use a single UPDATE with a subquery and check `rowcount` afterward:

```python
def acquire_task(worker_id: int):
    now = datetime.utcnow()
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, now),
    )
    db.commit()

    if db.rowcount == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, now),
    )
```

The subquery and the UPDATE execute as one atomic write. Only one worker can win the UPDATE for a given row. The `rowcount` check tells you whether you got it.

### Fix 2 — Add the status guard to the UPDATE's WHERE clause (minimum viable patch)

If you cannot change the SELECT pattern right now, at least make the UPDATE conditional:

```python
db.execute(
    "UPDATE task_queue"
    " SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ? AND status = 'pending'",   # <── this line
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()

if db.rowcount == 0:
    return None   # another worker got there first, skip it
```

This is better than what is there now but it is still not fully atomic — the SELECT and UPDATE are still separate round trips and the window still exists. Under high concurrency you will still see collisions, just fewer of them. Treat this as a stopgap while you implement Fix 1.

---

## The Visible Bugs — Do Not Chase These First

There are two other issues in the code that are real but not the cause of this incident:

1. **`execute_task` deserializes `payload` as a raw dict** — if `payload` is stored as a JSON string in the database (likely, given the ORM upgrade), `task["payload"]` will be a string and `payload["user_ids"]` will raise a `TypeError`, not produce the symptom you are seeing.

2. **The deduplication loop in `execute_task` is O(n²)** — it scans the entire summary list for every row. For large payloads this will be slow. Replace it with a set.

Both are worth fixing. Neither is why you are getting paged tonight. Fix the race condition first.

---

## Immediate Mitigation

Until the fix is deployed, reduce the worker count to 1. This eliminates the concurrency and stops the bleeding. It is not a fix — the bug is still in the code — but it will stop the `IntegrityError` flood and give the reporting pipeline time to catch up.
