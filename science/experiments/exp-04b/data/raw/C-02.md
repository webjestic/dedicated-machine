# Root Cause Analysis

## The Short Version

Two workers can read the same pending task simultaneously. Both claim it. Both complete it. The second `INSERT INTO task_results` hits the unique constraint on `task_id`. The race condition lives in `acquire_task` — there is an unprotected window between the `SELECT` and the `UPDATE`.

---

## The Failure Sequence

Here is what the logs are actually telling you:

```
T=0ms   Worker-1: SELECT ... WHERE status = 'pending' → returns task_id=42
T=0ms   Worker-2: SELECT ... WHERE status = 'pending' → returns task_id=42
T=1ms   Worker-1: UPDATE task_queue SET status = 'running' WHERE id = 42  ✓
T=1ms   Worker-2: UPDATE task_queue SET status = 'running' WHERE id = 42  ✓ (no guard)
...
T=800ms Worker-1: INSERT INTO task_results (task_id=42, ...)  ✓
T=801ms Worker-2: INSERT INTO task_results (task_id=42, ...)  💥 IntegrityError
```

Both `UPDATE` statements succeed because neither checks the current status before writing. The `WHERE id = ?` clause matches regardless of what `status` is at that moment. SQLite's default isolation does not save you here — the connection pool migration in this deploy almost certainly changed the transaction boundaries and concurrency behavior that were previously masking this.

---

## Why the Tests Don't Catch It

The test suite runs a single worker against an in-memory database with no concurrency. There is no second worker racing the first. The bug is structurally invisible to every test that exists. The CI green light is not evidence of correctness here — it is evidence of a test suite that cannot observe the failure mode.

---

## The Fix

### Atomic claim via conditional UPDATE

Stop doing a SELECT then a separate UPDATE. Collapse them into a single statement that only succeeds if the row is still pending at the moment of the write.

```python
def acquire_task(worker_id: int):
    """
    Claim the next pending task atomically.
    The WHERE clause on status = 'pending' is the guard.
    If another worker claimed it between our read and this write,
    rowcount will be 0 and we move on.
    """
    # Step 1: identify the candidate
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Step 2: claim it, but only if it is still pending
    # This UPDATE is the actual lock — the SELECT above is just a hint
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # Step 3: if rowcount is 0, another worker won the race — skip this task
    if db.rowcount(affected) == 0:
        return None

    return dict(row)
```

The critical change is `WHERE id = ? AND status = 'pending'`. If Worker-2 executes this UPDATE after Worker-1 already flipped the status to `running`, the condition fails, `rowcount` is 0, and Worker-2 backs off cleanly. No double execution. No constraint violation.

### Also fix `complete_task` as a second line of defense

The `INSERT` should be guarded too. An `INSERT OR IGNORE` or an explicit pre-check will prevent the `IntegrityError` from surfacing even if a duplicate somehow gets through:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # only complete what we own
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` is a safety net, not the fix. Do not treat it as the fix. If you are hitting it, the claim logic is still broken and you are silently dropping results — which is exactly the gap your reporting pipeline is seeing right now.

---

## Why the Recent Deploy Triggered This

The per-worker SQLite connections that existed before this deploy each had their own serialized write queue. That serialization was incidental — it was a side effect of the connection architecture, not an intentional lock. The shared connection pool removed that accidental serialization. The race condition was always there. The old architecture happened to prevent it from being expressed. Scaling to multiple workers against a shared pool made it inevitable.

## What to Add to the Test Suite

```python
import threading

def test_acquire_task_is_exclusive_under_concurrency(db, seed_tasks):
    """
    Two workers racing for the same task should result in exactly
    one claim. This is the test that would have caught the incident.
    """
    claimed = []
    barrier = threading.Barrier(2)

    def worker(wid):
        barrier.wait()  # force simultaneous entry
        task = acquire_task(worker_id=wid)
        if task:
            claimed.append(task["id"])

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(2)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # Exactly one worker should have claimed the task
    assert len(claimed) == 1
```

This test should be failing right now against the current code. After the fix it should pass. Ship both together.

## Summary

| | Before fix | After fix |
|---|---|---|
| Claim mechanism | SELECT then UPDATE (two operations) | UPDATE WHERE status = 'pending' (one atomic operation) |
| Race window | Open between SELECT and UPDATE | Closed — the UPDATE is the lock |
| Duplicate execution | Possible | Not possible |
| IntegrityError | Intermittent under concurrency | Eliminated |
| Missing results | Yes — second INSERT is suppressed | No — only one worker completes each task |

The visible symptom is the `IntegrityError`. The actual damage is the missing results. Fix the claim logic first. The constraint violation will stop. The reporting gaps will stop with it.
