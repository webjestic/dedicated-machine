## Root Cause

**`acquire_task` has an unprotected check-then-act window. Multiple workers can read the same pending task before any of them writes the claim.**

---

### The Failure Sequence

```
Time →

Worker 1:  SELECT (pending, id=42) ──────────────────── UPDATE id=42 → running ── COMMIT
Worker 2:  SELECT (pending, id=42) ── UPDATE id=42 → running ── COMMIT

Both workers now hold task 42.

Worker 1:  execute_task(42) ─────────────────────────────────── complete_task(42) → INSERT task_results ── COMMIT ✓
Worker 2:  execute_task(42) ──────────────────────── complete_task(42) → INSERT task_results ── BOOM
                                                                          IntegrityError: UNIQUE constraint
                                                                          failed: task_results.task_id
```

The SELECT and the UPDATE are two separate statements with no lock between them. Under a shared connection pool — which is exactly what the migration four days ago introduced — two workers can execute the SELECT concurrently, both see `status = 'pending'`, both proceed to UPDATE and claim the task, and both attempt to INSERT into `task_results`. The unique constraint on `task_id` catches the second insert. The first worker's result is written. The second worker's result is lost. That is your reporting gap.

---

### Why Tests Pass

Every test runs a single worker against an in-process database with no concurrency. The race requires two workers executing the SELECT in the same window. That condition cannot occur in the test suite as written. The tests are correct for what they test. They do not test the invariant that matters here: **a task is claimed by exactly one worker**.

---

### Why This Appeared Now

The per-worker SQLite connections in the old architecture serialized access by construction — each worker had its own connection and its own database file or WAL segment. The shared pool removed that accidental serialization. This race always existed in the logic. The old infrastructure was hiding it.

---

### What Needs to Change

There is one fix and one fix only that closes the window: **make the check and the claim a single atomic statement**.

```python
# Replace the SELECT + UPDATE pair in acquire_task with this.
# The WHERE clause is the guard. The task is only claimed if it is
# still pending at the moment the UPDATE executes.

def acquire_task(worker_id: int):
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The subquery inside the UPDATE is evaluated and the row is locked within a single database operation. The second worker's UPDATE hits a row that is no longer `pending` and matches zero rows. One worker gets the task. The other gets nothing and loops.

---

### Secondary Issue to Fix Before Next Deploy

`complete_task` also has no guard. If a worker somehow reaches INSERT after another worker has already completed the same task, the IntegrityError surfaces there rather than at claim time. Add a WHERE clause to the status UPDATE and check rows affected before inserting:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # guard added
        (datetime.utcnow(), task_id),
    )
    # Only insert if this worker actually owns the completion.
    # Check rowcount or use INSERT OR IGNORE depending on your db layer.
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

This is defense in depth, not the primary fix. The primary fix is the atomic claim.

|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`. SELECT and UPDATE are separate statements. Multiple workers claim the same task. |
| **Trigger** | Migration to shared connection pool removed the accidental per-process serialization that was masking the race. |
| **Blast radius** | Duplicate work on affected tasks. One result written, one IntegrityError, one reporting gap per collision. |
| **Fix** | Collapse SELECT + UPDATE into a single atomic UPDATE with a subquery. No application-level lock needed. |
| **Test gap** | No concurrent-worker test exists. Add one that spawns two workers against the same task and asserts exactly one result row. |
