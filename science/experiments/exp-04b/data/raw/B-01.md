## Root Cause

**The `acquire_task` function has a check-then-act race condition. Multiple workers can read the same `pending` row before any of them writes the `running` status update. Both proceed to execute and complete the task, and the second `INSERT INTO task_results` hits the unique constraint.**

---

### The Failure Sequence

Here is exactly what the logs are showing you:

```
T=0ms  Worker-1: SELECT ... WHERE status = 'pending' → returns task_id=42
T=0ms  Worker-2: SELECT ... WHERE status = 'pending' → returns task_id=42
T=1ms  Worker-1: UPDATE task_queue SET status = 'running' WHERE id = 42
T=1ms  Worker-2: UPDATE task_queue SET status = 'running' WHERE id = 42
       [both commits succeed — UPDATE has no guard, it will update any status]
T=50ms Worker-1: INSERT INTO task_results (task_id=42, ...) → OK
T=50ms Worker-2: INSERT INTO task_results (task_id=42, ...) → IntegrityError ✗
```

The SELECT and the UPDATE are two separate statements with no transaction isolation between them. There is nothing in the UPDATE's WHERE clause that requires the row to still be `pending`. Both workers claim the same task, both execute it, and the second write to `task_results` explodes. The first write succeeds silently, which is why you also see gaps — the result that lands is whichever worker finished first, not necessarily the correct one, and the second result is lost entirely.

---

### Why the Tests Don't Catch This

The tests run a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in that environment. All 34 tests pass and always will, because the condition that causes the failure requires two workers reading simultaneously — something the test suite never exercises. This is not a test quality problem you fix by adding more assertions. You need a concurrency test.

---

### Why This Appeared Now

The migration to a shared connection pool is the proximate trigger, not the root cause. With per-worker SQLite connections, each worker had its own database file or connection-level serialization that incidentally prevented this. The shared pool removed that accidental protection and exposed the race that was always latent in the logic. The ORM upgrade is a distractor — do not chase it.

---

### What Needs to Change

The fix must make the claim atomic. The SELECT and the UPDATE must be collapsed into a single statement that only succeeds for one worker.

**Option 1 — Atomic UPDATE with a status guard (correct fix, minimal change):**

```python
def acquire_task(worker_id: int):
    # First, find a candidate without claiming it
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Claim it only if it is still pending — this is the atomic gate.
    # If another worker got here first, affected_rows will be 0.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if affected.rowcount == 0:
        return None  # Another worker claimed it first

    return dict(row)
```

The `AND status = 'pending'` predicate on the UPDATE is the critical change. The database executes the UPDATE atomically. Exactly one worker will see `rowcount == 1`. Every other worker sees `rowcount == 0` and backs off. This works because a single UPDATE statement is atomic at the row level in both SQLite (with WAL mode) and any standard RDBMS.

**Option 2 — SELECT FOR UPDATE (if your database and pool support it):**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # Postgres; omit SKIP LOCKED for others
        )

        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )

    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is the standard pattern for job queues on Postgres. SQLite does not support it. Check which database is actually running in production before reaching for this.

**Option 3 — Deduplicate at the write, not the read:**

You could also add a guard to `complete_task`:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` stops the crash, but this is not the fix — it is a safety net. The task still executes twice, consuming resources and potentially producing side effects depending on what `execute_task` does. Do not ship this alone. Ship Option 1, and add this as a defensive layer.

---

### What to Ship

1. **Option 1 immediately** — the `AND status = 'pending'` guard on the UPDATE. This closes the race window.
2. **`INSERT OR IGNORE` on `complete_task`** as a belt-and-suspenders layer.
3. **A concurrency integration test** that spins up two workers against the same database and asserts that a given task ID appears exactly once in `task_results`. Without this, the next person who touches this code will reopen the window.

The missing results in the reporting pipeline are tasks where Worker-2 lost the `IntegrityError` and the result was never recorded. After deploying the fix, you will need to identify those task IDs from the error log and decide whether to requeue them.
