## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. Multiple workers can read the same pending task row before any of them writes the status update that would hide it from the others.**

---

### The Failure Sequence

Here is exactly what is happening in production:

```
Time →

Worker A:  SELECT ... WHERE status = 'pending' → gets task_id=42
Worker B:  SELECT ... WHERE status = 'pending' → gets task_id=42   ← same row, A hasn't updated yet
Worker A:  UPDATE task_queue SET status = 'running' WHERE id = 42
Worker A:  commit
Worker B:  UPDATE task_queue SET status = 'running' WHERE id = 42  ← redundant, but succeeds
Worker B:  commit

Worker A:  execute_task(42) → result_a
Worker B:  execute_task(42) → result_b

Worker A:  INSERT INTO task_results (task_id=42, ...)              ← succeeds
Worker B:  INSERT INTO task_results (task_id=42, ...)              ← UNIQUE constraint violation
```

The `UPDATE` in `acquire_task` does not protect anything because it runs *after* the `SELECT` in a separate statement, with no transaction or lock holding the row between them. The window between the read and the write is where both workers land simultaneously under load.

This is why the error is intermittent and only appears during high-traffic windows — you need the timing to be tight enough for two workers to execute that `SELECT` concurrently before either `UPDATE` commits.

---

### Why the Tests Don't Catch It

The tests run a single worker against an in-process database. There is no concurrency. The race window never opens. All 34 tests pass, and every one of them is testing the wrong thing for this failure mode. This is not a test quality problem you need to solve right now — it is just why CI gave you no signal.

---

### Why the Recent Deploy Is the Trigger

Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per connection, and separate connections to the same file do get some natural serialization from file-level locking. It was still a race, but the lock contention masked it most of the time.

The shared connection pool removes that accidental serialization. Multiple workers now issue their `SELECT` statements through pool connections that are genuinely concurrent against the same database. The race that was always there is now reliably reachable.

The ORM upgrade is likely a red herring unless it changed transaction autocommit behavior. Check that, but do not chase it until the structural fix is in.

---

### What Needs to Change

The fix is to make the claim operation atomic. The `SELECT` and the `UPDATE` must be a single operation that no other worker can interleave with.

**Option 1 — `UPDATE ... RETURNING` (preferred if your database supports it)**

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This is one statement. The database executes the subquery and the update atomically under a write lock. There is no window. A row either gets claimed by exactly one worker or it does not get claimed at all.

SQLite supports `RETURNING` since version 3.35.0 (2021). PostgreSQL has supported it for years. Verify your SQLite version before deploying.

**Option 2 — Explicit transaction with `SELECT FOR UPDATE` (PostgreSQL)**

If you are on PostgreSQL:

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE SKIP LOCKED"  # SKIP LOCKED prevents queueing behind other workers
        )
        if not row:
            return None
        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )
    return dict(row)
```

`FOR UPDATE SKIP LOCKED` is the correct primitive for a job queue on PostgreSQL. It locks the row for the duration of the transaction and skips rows already locked by other workers rather than blocking on them.

**Option 3 — Optimistic concurrency (works on SQLite without `RETURNING`)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )
    if not row:
        return None

    updated = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← the guard
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, rowcount is 0 — back off and retry
    if updated.rowcount == 0:
        return None

    return dict(row)
```

The `AND status = 'pending'` guard means the update is a no-op if another worker already claimed the row. Check `rowcount`. If it is zero, this worker lost the race cleanly and returns `None`. The caller's loop will retry on the next cycle. This is safe on SQLite because the `UPDATE` itself is atomic — the check and the write happen in one statement.

### What to Do Right Now

1. **Deploy Option 1 or Option 3 immediately.** Option 1 is cleaner if your SQLite version supports `RETURNING`. Option 3 works on any SQLite version and is safe to ship now.
2. **Do not patch `complete_task`.** The `UNIQUE constraint` error is the symptom. The duplicate write attempt is the consequence. The cause is upstream in `acquire_task`. Catching the `IntegrityError` in `complete_task` would suppress the error log and make the gap in `task_results` invisible. That is worse than the current state.
3. **The missing results in the reporting pipeline** are tasks where Worker B got the `IntegrityError`, its `complete_task` call failed, and the result was never written. After you deploy the fix, you will need to identify tasks that are stuck in `status = 'running'` with no corresponding row in `task_results` and decide whether to requeue them.
4. **Add a test that runs two workers concurrently** against a shared in-memory database and asserts that a given task ID appears exactly once in `task_results`. Your current test suite will not catch a regression of this class.
