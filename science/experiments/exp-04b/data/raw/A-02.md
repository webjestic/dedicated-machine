## Root Cause

**`acquire_task` has an unprotected check-then-act window. Multiple workers can read the same pending task before any of them writes the status update. Both proceed to `complete_task`, and the second `INSERT INTO task_results` hits the unique constraint on `task_id`.**

---

### The Exact Sequence

```
Worker 1                                    Worker 2
────────────────────────────────────────────────────────────────
SELECT ... WHERE status = 'pending'
                                            SELECT ... WHERE status = 'pending'
                                            ← same row returned, status still 'pending'

UPDATE task_queue SET status = 'running'
  WHERE id = 42
db.commit()
                                            UPDATE task_queue SET status = 'running'
                                              WHERE id = 42
                                            db.commit()
                                            ← succeeds, no guard on the UPDATE

execute_task(task_id=42)
                                            execute_task(task_id=42)

INSERT INTO task_results (task_id=42, ...)
db.commit()
                                            INSERT INTO task_results (task_id=42, ...)
                                            ← UNIQUE constraint failed ✴
```

The SELECT and the UPDATE are two separate statements with no isolation between them. The connection pool migration made this worse: before the migration, per-worker SQLite connections serialized access by accident — each worker had its own connection and SQLite's file-level locking kept them from interleaving at this granularity. The shared pool removed that accidental serialization. The race was always there. The migration exposed it.

This also explains the gaps in the reporting pipeline. When Worker 2 hits the `IntegrityError`, the exception unwinds before `db.commit()`. The `UPDATE task_queue SET status = 'complete'` that preceded the failed INSERT is not committed either — so the task row stays in `running` or gets rolled back depending on the ORM's exception handling. Either way, no result row lands in `task_results`.

---

### Why the Tests Don't Catch It

Every test calls `acquire_task` once, with a single worker, against an in-memory or single-connection fixture. There is no concurrent access. The race requires two workers reading before either writes. The test suite cannot see it.

---

### What Needs to Change

The fix must make the claim atomic. There are two viable paths:

**Option A — Atomic UPDATE with a WHERE guard (preferred, no schema change)**

Replace the SELECT-then-UPDATE with a single statement that only claims the row if it is still pending. Use the `RETURNING` clause (SQLite ≥ 3.35, PostgreSQL native) to get the row back in one round trip.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

The subquery and the UPDATE execute as a single atomic operation inside the database engine. A second worker running the identical statement at the same moment will find `status = 'running'` in the subquery and update zero rows. Only one worker gets a row back.

**Option B — Optimistic locking with a version column**

Add a `version INTEGER DEFAULT 0` column to `task_queue`. SELECT the row including version, then UPDATE with `WHERE id = ? AND version = ? AND status = 'pending'`, incrementing version. Check `rowcount`. If it is 0, another worker got there first — return None. More schema work, same correctness guarantee.

**Option A is the immediate fix.** It requires no schema change and works in SQLite and PostgreSQL. Option B is worth considering if you need audit history on claims, but it is not the right call at 9 minutes to the sync.

---

### What to Tell the Pipeline Lead

> Workers are racing to claim the same task. Two workers read the same pending row before either marks it running. Both execute it, both try to insert a result row, the second one fails on the unique constraint, and that task's result is dropped. The shared connection pool removed the accidental serialization that was hiding this. The fix is a single-statement atomic claim — one database round trip instead of two. No schema change required.
