## Root Cause

**`acquire_task` has a check-then-act race condition. Two workers can read the same pending task before either one writes the status update. Both proceed to `complete_task`, and the second `INSERT INTO task_results` hits the unique constraint on `task_id`.**

---

### The Failure Sequence

```
Time →

Worker 1:  SELECT (gets task #42, status='pending')
Worker 2:  SELECT (gets task #42, status='pending')   ← same row, before W1's UPDATE commits
Worker 1:  UPDATE task_queue SET status='running' WHERE id=42
Worker 1:  COMMIT
Worker 2:  UPDATE task_queue SET status='running' WHERE id=42  ← overwrites, no guard
Worker 2:  COMMIT
Worker 1:  execute_task → complete_task → INSERT task_results(task_id=42) ✓
Worker 2:  execute_task → complete_task → INSERT task_results(task_id=42) ✗ IntegrityError
```

The SELECT and the UPDATE are two separate statements with no isolation between them. The window between those two operations is exactly where the collision happens. Both workers see `status = 'pending'`, both claim the task, both execute it, and the second write to `task_results` explodes.

---

### Why Tests Don't Catch It

Every test runs a single worker against an in-process database. There is no concurrency. The race window does not exist in the test environment. All 34 tests pass and always will, regardless of whether this bug is present.

---

### Why It Started Now

The migration to a shared connection pool four days ago is the trigger. Before that, each worker had its own SQLite connection. SQLite serializes writes per connection, so workers were accidentally protected by isolation at the connection level. With a shared pool, multiple workers share connections and the SELECT → UPDATE sequence across two pool checkouts is no longer serialized. The race that was always latent in the logic is now reachable.

---

### What Needs to Change

There are two valid fixes. Pick one based on what the database supports.

**Option 1 — Atomic claim with `UPDATE ... WHERE status = 'pending'` (preferred)**

Remove the SELECT entirely from the claim step. Attempt the UPDATE directly and check whether a row was actually modified. Only one writer can transition a row from `pending` to `running`.

```python
def acquire_task(worker_id: int):
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only the worker that actually changed a row gets the task.
    if db.rows_affected() == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
```

The database enforces that only one UPDATE wins for a given row. The check and the act are now a single atomic operation.

**Option 2 — Explicit locking (if the ORM/driver supports `SELECT FOR UPDATE`)**

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE",          # holds a write lock until commit
    )

    if not row:
        return None

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

Note: SQLite does not support `FOR UPDATE`. If the backend is SQLite in production, Option 1 is the only safe path.

---

### Secondary Issue to Fix After Stabilization

`complete_task` should also guard against double-writes. Even after fixing `acquire_task`, add a conflict guard on the insert as a defense-in-depth measure:

```python
db.execute(
    "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
    " VALUES (?, ?, ?)",
    (task_id, json.dumps(result), datetime.utcnow()),
)
```

This is not the root cause fix. Do not let it distract from shipping Option 1. But it closes the gap between "the race is harder to hit" and "the race cannot produce a crash."

---

### Summary for the Call

> The SELECT and UPDATE in `acquire_task` are not atomic. Two workers can read the same pending task before either one claims it. Both execute the task and both attempt to insert into `task_results`. The second insert fails with a unique constraint violation. This race was always in the logic but was masked by per-worker SQLite connections before the pool migration four days ago. Fix: collapse the SELECT and UPDATE into a single atomic UPDATE statement and check `rows_affected`. That is the only change needed to stop the errors.
