## Root Cause

**`acquire_task` has a check-then-act race condition with no atomic protection around the claim window.**

This is a classic TOCTOU (time-of-check to time-of-use) bug. Here is the exact sequence that produces the `IntegrityError`:

```
Worker A                              Worker B
--------                              --------
SELECT ... WHERE status = 'pending'
                                      SELECT ... WHERE status = 'pending'
-- both read the same row, same task_id --

UPDATE ... SET status = 'running'
  WHERE id = ?
db.commit()
                                      UPDATE ... SET status = 'running'
                                        WHERE id = ?
                                      db.commit()

execute_task(task)
                                      execute_task(task)

INSERT INTO task_results (task_id, ...)
db.commit()
                                      INSERT INTO task_results (task_id, ...)
                                      -- BOOM: UNIQUE constraint failed --
```

Both workers read the same pending task before either has committed the status update. Both proceed to execute it. Both attempt to write to `task_results`. The second `INSERT` hits the unique constraint on `task_id`. That explains the `IntegrityError`. The gaps in the reporting pipeline come from the exception path — the second worker's `complete_task` fails entirely, so neither the `task_queue` status update nor the `task_results` insert from that call lands cleanly, depending on how the exception is handled.

---

## Why the Tests Did Not Catch This

The tests run a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in that environment. All 34 tests pass and always will, regardless of this bug. The test suite is not wrong — it is just not testing the thing that broke.

The migration four days ago is directly relevant. Moving from per-worker SQLite connections to a shared connection pool means multiple workers are now operating against the same database concurrently in a way they were not before. The bug may have existed in the logic before the migration, but the per-worker connection model serialized access by accident. The pool removed that accidental serialization and exposed the unprotected window.

---

## What Needs to Change

The SELECT and the UPDATE that claims the task must be atomic. There are two correct approaches depending on your database.

**Option 1 — Atomic UPDATE with RETURNING (preferred for PostgreSQL, SQLite 3.35+)**

Collapse the SELECT and UPDATE into a single statement. There is no window between them because there is no separate read.

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    if not row:
        return None
    db.commit()
    return dict(row)
```

Only one worker can win the UPDATE for a given row. The database's row-level locking ensures that. The worker that wins gets the row back via RETURNING. Workers that lose get nothing back and move on.

**Option 2 — Explicit advisory lock or SELECT FOR UPDATE SKIP LOCKED (PostgreSQL)**

If you are on PostgreSQL and want the most explicit form:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
        " FOR UPDATE SKIP LOCKED"
    )
    if not row:
        return None
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

`FOR UPDATE SKIP LOCKED` locks the selected row for the duration of the transaction and skips any row another transaction has already locked. This is purpose-built for job queues.

---

## What Is Not the Root Cause

The `execute_task` deduplication logic is inefficient — the `any()` call inside the loop is O(n²) — but it is not causing the incident. The ORM upgrade from 2.8.4 to 3.2.1 may have introduced behavioral differences worth reviewing, but the race condition is structural and would exist regardless of ORM version. Do not ship a fix to either of those things and call the incident resolved. The unprotected claim window is what is causing the constraint violations. Fix that first.
