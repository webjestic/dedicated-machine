## Root Cause

**The `acquire_task` function has a race condition. Multiple workers can claim the same task simultaneously, leading to duplicate `INSERT` attempts into `task_results`.**

Here's the exact sequence:

```
Worker A: SELECT → finds task #42 (status = 'pending')
Worker B: SELECT → finds task #42 (status = 'pending')  ← same row, before A's UPDATE commits
Worker A: UPDATE task #42 SET status = 'running' → commits
Worker B: UPDATE task #42 SET status = 'running' → commits (overwrites A's claim)
Worker A: INSERT INTO task_results (task_id=42, ...) → succeeds
Worker B: INSERT INTO task_results (task_id=42, ...) → 💥 UNIQUE constraint failed
```

The SELECT and UPDATE are **two separate statements with no lock between them**. This is a classic check-then-act race.

---

## Why It Wasn't Happening Before

The previous setup used **per-worker SQLite connections**. SQLite's file-level locking and the fact that each worker had its own connection effectively serialized access — not by design, but by accident. The migration to a **shared connection pool** removed that accidental serialization. Multiple workers now genuinely execute concurrently against the same database rows.

---

## Why Tests Are Passing

Every test runs a **single worker** against an **isolated test database**. There is no concurrency in the test suite. The race condition requires two workers hitting the same row at the same millisecond — that never happens in tests.

---

## The Fix

Replace the SELECT + UPDATE pair with an **atomic claim** using a single `UPDATE ... WHERE status = 'pending'` and check whether a row was actually modified. This is the standard pattern for queue workers.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # First, identify the candidate task ID
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Atomically claim it — only one worker wins this UPDATE
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- guard condition is the fix
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, rowcount will be 0
    if db.rowcount == 0:
        return None

    # Re-fetch the full row now that we own it
    claimed = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],),
    )
    return dict(claimed)
```

The critical change is `WHERE id = ? AND status = 'pending'`. The database evaluates this atomically. Only one concurrent `UPDATE` can match — the second one finds `status = 'running'` and touches zero rows. The `db.rowcount == 0` check lets that worker back off cleanly and pick a different task on the next loop iteration.

## Secondary Recommendation (for after the incident)

Add a dead-letter / retry mechanism for the tasks that lost the race and ended up with `status = 'running'` but **no corresponding row in `task_results`**. Those are the gaps the reporting pipeline is seeing right now. A backfill query to reset them to `'pending'` will recover the missing results:

```sql
UPDATE task_queue
SET status = 'pending', worker_id = NULL, claimed_at = NULL
WHERE status = 'running'
  AND id NOT IN (SELECT task_id FROM task_results)
  AND claimed_at < datetime('now', '-5 minutes');
```

Run that once the fix is deployed to drain the stuck tasks.
