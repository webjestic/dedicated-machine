## Root Cause

**`acquire_task` has a check-then-act race condition. There is no atomic claim on the row. Two workers can read the same pending task in the window between the SELECT and the UPDATE, both believe they own it, both call `complete_task`, and the second INSERT into `task_results` hits the UNIQUE constraint on `task_id`.**

---

### The Exact Sequence

```
Time →

Worker 1:  SELECT (gets task #42, status='pending')
Worker 2:  SELECT (gets task #42, status='pending')  ← same row, before W1's UPDATE commits
Worker 1:  UPDATE task_queue SET status='running' WHERE id=42
Worker 1:  COMMIT
Worker 2:  UPDATE task_queue SET status='running' WHERE id=42  ← succeeds, no guard
Worker 2:  COMMIT

... both workers execute task #42 ...

Worker 1:  INSERT INTO task_results (task_id=42, ...)  ← succeeds
Worker 2:  INSERT INTO task_results (task_id=42, ...)  ← 💥 IntegrityError
```

The UPDATE in `acquire_task` has no condition that checks the current status. It will happily overwrite `running` with `running`. Neither worker sees an error. Both proceed to `complete_task`.

---

### Why Tests Don't Catch It

Every test runs a single worker against an in-process database. There is no concurrency. The race window does not exist in the test environment. All 34 tests pass. This is expected — the tests are not wrong, they are just not testing the thing that broke.

The migration four days ago is the trigger, not the cause. Moving to a shared connection pool meant multiple workers are now hitting the same database with real concurrency. The race was always latent. The pool made it reachable.

---

### Why Results Are Missing Downstream

The worker that loses the INSERT race gets an `IntegrityError`. Depending on how that exception is handled (and from this code, it is not handled at all), that worker's result is silently dropped. The task may also be left in an inconsistent state in `task_queue`. That is the gap in the reporting pipeline.

---

### What Needs to Change

**The fix is to make the claim atomic. The UPDATE must be the check, not a separate step after it.**

Replace the SELECT + UPDATE pair in `acquire_task` with a single statement that only claims the row if it is still pending:

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # Single atomic claim: only one writer can transition
    # status from 'pending' → 'running' for a given row.
    # The WHERE clause IS the lock. No separate SELECT needed.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",  # guard survives the subquery window
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only fetch the row this worker actually claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC"
        " LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The critical property: **`status = 'pending'` appears in the WHERE clause of the UPDATE itself.** SQLite (and every ANSI-compliant database) guarantees that only one writer can win that transition for a given row. The second worker's UPDATE matches zero rows. It gets nothing back. It returns `None`. It does not proceed to `execute_task`. It does not reach `complete_task`. The constraint is never tested.

---

### Secondary Hardening (Do This Too)

`complete_task` should also guard against writing a result for a task this worker no longer owns, in case a future timeout/retry mechanism is added:

```python
def complete_task(task_id: int, result: list, worker_id: int) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running' AND worker_id = ?",
        (datetime.utcnow(), task_id, worker_id),
    )
    # Check rows_affected here before inserting.
    # If this worker no longer owns the task, do not write the result.
    db.execute(
        "INSERT INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```
