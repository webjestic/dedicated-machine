## Root Cause

**Two workers are claiming the same task and both calling `complete_task` on it. The second `INSERT INTO task_results` hits the `UNIQUE` constraint on `task_id` and throws.**

---

### Why It's Happening

The `acquire_task` function has a classic **check-then-act race condition**:

```python
# Worker A and Worker B both execute this SELECT at nearly the same time.
# Both read the same row with status = 'pending'.
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Both then UPDATE that same task_id to 'running'.
db.execute(
    "UPDATE task_queue SET status = 'running' ... WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
```

With a **per-worker SQLite connection** (the old setup), SQLite's file-level locking made this unlikely — each worker serialized against the file. With a **shared connection pool**, multiple workers share connections and can interleave reads and writes. The SELECT and UPDATE are not atomic. Two workers race through the SELECT, both see the same pending task, both claim it, both execute it, and both call `complete_task`. The first INSERT into `task_results` succeeds. The second one fails with the `UNIQUE` constraint violation — and that task's result is then missing from the reporting pipeline, which explains the gaps.

---

### Why Tests Didn't Catch It

The tests run a single worker sequentially against an in-memory database. There is no concurrency, so the race never occurs. All 34 tests pass cleanly.

---

### What Needs to Change

The SELECT and UPDATE must be made **atomic**. The fix is to collapse them into a single statement using `UPDATE ... WHERE status = 'pending' ... RETURNING` (if the DB supports it), or to use `SELECT ... FOR UPDATE` with a proper transaction. For SQLite specifically, the most reliable approach is a single `UPDATE` with a subquery so no separate `SELECT` is needed:

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""
    # Single statement: find and claim in one shot.
    # No separate SELECT means no window for another worker to race in.
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

If `RETURNING` is not available in the ORM version in use, the equivalent is:

```python
def acquire_task(worker_id: int):
    claimed_at = datetime.utcnow()
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )",
        (worker_id, claimed_at),
    )
    db.commit()

    # Only this worker's UPDATE will have set claimed_at to this exact value.
    # Re-fetch the row we just claimed.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND claimed_at = ? AND status = 'running'",
        (worker_id, claimed_at),
    )
    return dict(row) if row else None
```

A secondary defensive measure — worth adding regardless — is to make `complete_task` use `INSERT OR IGNORE` so a duplicate attempt fails silently rather than raising and dropping the result:

```python
db.execute(
    "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
    " VALUES (?, ?, ?)",
    (task_id, json.dumps(result), datetime.utcnow()),
)
```

This is a safety net, not the real fix. The real fix is `acquire_task`.
