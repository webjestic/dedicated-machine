## Root Cause

**The `acquire_task` function has a race condition — it's a non-atomic read-then-write.**

```python
# These two operations are NOT atomic
row = db.query_one("SELECT id ... WHERE status = 'pending' ...")  # Worker A and Worker B both read the same row
db.execute("UPDATE task_queue SET status = 'running' ... WHERE id = ?", ...)  # Both then claim it
db.commit()
```

Under the old per-worker SQLite connections, each worker had an isolated connection so this race didn't manifest. After the migration to a **shared connection pool**, multiple workers now genuinely execute concurrently against the same database, and the window between `SELECT` and `UPDATE` is exposed.

**The failure sequence:**

1. Worker A and Worker B both query for the next pending task
2. Both read the same `task_id` (e.g. `id = 42`) — it's still `'pending'` for both reads
3. Both proceed through `execute_task` independently
4. Worker A reaches `complete_task` first — `INSERT INTO task_results (task_id=42, ...)` succeeds
5. Worker B reaches `complete_task` — `INSERT INTO task_results (task_id=42, ...)` hits the `UNIQUE` constraint → **`IntegrityError`**

This also explains the **missing results in the reporting pipeline**: Worker B's `complete_task` throws before the `UPDATE task_queue SET status = 'complete'` is committed, so that task either stays `'running'` forever or gets partially written depending on error handling — either way the result is lost.

**Why tests don't catch it:** Every test runs a single worker sequentially against an isolated test database. There is no concurrency in the test suite, so the race window never opens.

## What Needs to Change

### Fix 1 — Make the claim atomic (required, do this now)

Add a `WHERE status = 'pending'` guard to the `UPDATE` itself and check whether a row was actually modified before proceeding. This collapses the two-step read/write into a single atomic operation.

```python
def acquire_task(worker_id: int):
    # Atomically claim a task in one statement.
    # The subquery finds the candidate; the WHERE clause on the UPDATE
    # ensures only one worker wins if two race to the same row.
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " AND status = 'pending'",   # <-- guard: only wins if still pending
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only return the task if this worker actually claimed it
    if db.rows_affected() == 0:
        return None

    return db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
```

> **Note for the engineers standing by:** check what `db-pool 2.1`'s wrapper exposes for rows-affected — it may be `db.rows_affected()`, `db.cursor.rowcount`, or similar. Confirm before deploying.

### Fix 2 — Make `complete_task` idempotent (belt-and-suspenders)

Even with Fix 1 in place, protect the insert against any future double-delivery:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ?",
        (datetime.utcnow(), task_id),
    )
    db.execute(
        # INSERT OR IGNORE means a duplicate task_id is silently skipped
        # rather than raising an IntegrityError
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

This stops the bleeding immediately and is a safe one-line change while Fix 1 is being reviewed.

### Fix 3 — Add a concurrency test (after the incident)

The test suite gave a false green because it never runs two workers simultaneously. Add a test that does:

```python
def test_acquire_task_no_duplicate_claims(db, seed_tasks):
    """Two workers racing for the same task must not both claim it."""
    from concurrent.futures import ThreadPoolExecutor

    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(acquire_task, wid) for wid in (1, 2)]
        results = [f.result() for f in futures]

    claimed = [r for r in results if r is not None]
    task_ids = [r["id"] for r in claimed]
    assert len(task_ids) == len(set(task_ids)), "Same task claimed by two workers"
```
