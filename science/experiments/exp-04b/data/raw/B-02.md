## Root Cause

**The `acquire_task` function has a classic check-then-act race condition. There is no atomic claim on the row. Two workers can read the same `pending` task simultaneously, both pass the `WHERE status = 'pending'` filter, both update the row to `running`, and both proceed to call `complete_task` on the same `task_id`. The second `INSERT INTO task_results` hits the unique constraint and fails. That is your `IntegrityError`. That is also your missing results — the worker that loses the constraint violation has its result discarded entirely.**

---

## The Sequence That Produces This Output

```
T=0   Worker A: SELECT ... WHERE status = 'pending' → returns task #42
T=0   Worker B: SELECT ... WHERE status = 'pending' → returns task #42  ← same row, status still 'pending'
T=1   Worker A: UPDATE task_queue SET status = 'running' WHERE id = 42  ← succeeds
T=1   Worker B: UPDATE task_queue SET status = 'running' WHERE id = 42  ← also succeeds, no guard
T=2   Worker A: execute_task(42) ...
T=2   Worker B: execute_task(42) ...
T=3   Worker A: INSERT INTO task_results (task_id=42, ...) → OK
T=3   Worker B: INSERT INTO task_results (task_id=42, ...) → IntegrityError: UNIQUE constraint failed
```

The `SELECT` and the `UPDATE` are two separate statements with no lock between them. The shared connection pool introduced by the migration means workers that previously had isolated connections are now genuinely concurrent against the same database state. This race always existed in the logic. The migration made it reachable.

---

## Why the Tests Don't Catch It

Every test runs a single worker against a test database with no concurrency. The race requires two workers to interleave at the exact window between the `SELECT` and the `UPDATE`. Sequential single-worker tests will never reproduce it. The 34 green tests are not evidence the code is correct — they are evidence the test suite does not model the production execution environment.

---

## What Is Not the Root Cause

The `IntegrityError` is the symptom. The unique constraint on `task_results.task_id` is doing its job correctly — it is the only thing preventing silent duplicate processing. Do not remove or relax that constraint. Do not wrap `complete_task` in a try/except and swallow the error. That would hide the duplicate execution while the underlying problem continues.

The deduplication logic in `execute_task` is also not the issue. It is inefficient — an O(n²) list scan that should be a dict or set — but it is not causing this incident.

---

## What Needs to Change

The claim operation must be atomic. The `SELECT` and the `UPDATE` must happen as a single operation that only one worker can win.

### Option 1 — Atomic UPDATE with RETURNING (preferred for SQLite 3.35+)

```python
def acquire_task(worker_id: int):
    """Atomically claim the next pending task."""
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This is one statement. The database processes the subquery and the update atomically under its write lock. A second worker issuing the same statement concurrently will either claim a different row or get nothing back. There is no window.

### Option 2 — Optimistic locking with a status guard on the UPDATE

If `RETURNING` is not available, add a `WHERE status = 'pending'` guard to the `UPDATE` and check whether it actually modified a row:

```python
def acquire_task(worker_id: int):
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard added
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount

    db.commit()

    if affected == 0:
        return None  # another worker claimed it first

    return dict(row)
```

This is strictly weaker than Option 1. It reduces the window but does not eliminate it — two workers can still both read the row and both attempt the update, and SQLite's write serialization will cause one update to see `rowcount = 0`. It is correct under SQLite's serialized write model, but Option 1 is cleaner and should be preferred.

---

## Immediate Mitigation

If you cannot deploy right now: reduce worker concurrency to 1. That eliminates the race by removing the concurrency. It is not a fix. It is buying time. Ship the atomic claim as soon as you can get it through review.

---

## What to Add to the Postmortem

The test suite needs a concurrency integration test. Something like: spin up two threads, have both call `acquire_task` simultaneously against a shared database with one pending task, assert that exactly one of them returns a task and the other returns `None`. That test would have caught this before the migration shipped.
