# Root Cause Analysis — `IntegrityError: UNIQUE constraint failed: task_results.task_id`

## The Short Answer

`acquire_task` has a **check-then-act race condition**. Two workers can read the same pending task in the window between the `SELECT` and the `UPDATE`. Both claim it. Both execute it. Both try to insert into `task_results`. The second insert hits the unique constraint.

The tests pass because they run a single worker against an isolated database. The race only surfaces under concurrent load.

---

## The Failure Sequence

Here is exactly what is happening in production:

```
Time →

Worker 1:  SELECT (status='pending', id=42) ──────────────────────────────► UPDATE id=42 → 'running' → execute → INSERT task_results(42) ✅
                                              ↑
Worker 2:                    SELECT (status='pending', id=42) ──────────► UPDATE id=42 → 'running' → execute → INSERT task_results(42) 💥 UNIQUE constraint
```

Both workers read `id=42` with `status = 'pending'` before either write lands. The `UPDATE` that follows is not conditional — it does not check that the row is still `pending`. It will succeed for both workers. Both proceed to `execute_task`. Both proceed to `complete_task`. The first insert into `task_results` succeeds. The second one explodes.

This is a **TOCTOU** (time-of-check to time-of-use) bug. The check is the `SELECT WHERE status = 'pending'`. The act is the `UPDATE SET status = 'running'`. There is no lock held between them.

---

## Why This Appeared Now

The migration to a shared connection pool four days ago is the trigger. Before that, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per connection, so separate connections were effectively taking turns at the file level. That was masking the race.

With a shared pool and multiple workers hitting the same connection surface simultaneously, the window between `SELECT` and `UPDATE` is now real and exploitable. The ORM upgrade is likely irrelevant — the pool change is what exposed this.

The CI suite never catches it because every test runs one worker. There is no test that spawns two workers and has them race to claim the same task.

---

## What Needs to Change

### Fix: Make the claim atomic

The `SELECT` and the `UPDATE` must be a single atomic operation. The standard pattern is to push the condition into the `UPDATE` itself and verify that it actually claimed something:

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""
    # Subquery selects the candidate; WHERE clause on the UPDATE
    # ensures we only claim it if it is still pending.
    # If another worker got there first, affected_rows == 0.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    if affected.rowcount == 0:
        return None

    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running'"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

The key property: the `WHERE id = (subquery)` and the `SET status = 'running'` happen inside a single statement. The database engine serializes concurrent writes to the same row. Exactly one of two racing workers will see `rowcount == 1`. The other sees `rowcount == 0` and backs off cleanly.

---

### Also Fix: `complete_task` needs a guard

Even after fixing `acquire_task`, `complete_task` is still fragile. If a worker crashes mid-task and another worker re-claims the same task later (via a timeout/retry mechanism you may add), you can still get a duplicate insert. Make the insert idempotent:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # guard: only complete what we own
        (datetime.utcnow(), task_id),
    )
    db.execute(
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` means a duplicate insert fails silently rather than crashing the worker. This is a safety net, not the primary fix. The primary fix is in `acquire_task`.

---

### Add a Concurrent Acquisition Test

The test suite has no coverage for the scenario that is burning you. Add this:

```python
import threading

def test_acquire_task_only_claimed_once(db, seed_tasks):
    """Two workers racing to claim the same task — only one should win."""
    results = []
    barrier = threading.Barrier(2)

    def worker(wid):
        barrier.wait()  # force both threads to attempt acquisition simultaneously
        task = acquire_task(worker_id=wid)
        results.append(task)

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(2)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    claimed = [r for r in results if r is not None]
    assert len(claimed) == 1, (
        f"Expected exactly 1 worker to claim the task, got {len(claimed)}"
    )
```

This test would have caught the bug before the migration shipped. Add it now so it catches any regression.

---

## Summary

| | Detail |
|---|---|
| **Root cause** | TOCTOU race in `acquire_task`: `SELECT` and `UPDATE` are separate statements with no lock between them |
| **Trigger** | Migration to shared connection pool removed the accidental serialization that was hiding the race |
| **Symptom** | Two workers claim the same task; second `INSERT` into `task_results` violates unique constraint |
| **Missing results** | The worker that hits the `IntegrityError` likely rolls back or dies mid-`complete_task`, dropping the result |
| **Fix** | Collapse `SELECT` + `UPDATE` into a single atomic `UPDATE ... WHERE id = (subquery)` |
| **Safety net** | `INSERT OR IGNORE` in `complete_task`; `AND status = 'running'` guard on the status update |
| **Test gap** | No concurrent acquisition test exists; add one |

The visible symptom is the `IntegrityError`. The actual bug is the unprotected window in `acquire_task`. Fix that window first.
