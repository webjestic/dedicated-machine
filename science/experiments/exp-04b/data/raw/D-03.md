## Root Cause

**Two workers are claiming the same task and both calling `complete_task()` on it. The second `INSERT INTO task_results` fails because `task_id` is a unique key and a row already exists.**

---

### Why This Is Happening

The `acquire_task()` function has a **check-then-act race condition** — a classic TOCTOU bug:

```python
# Worker A and Worker B both execute this SELECT at nearly the same time
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Both see the same row with status = 'pending'
# Both then issue this UPDATE — last writer wins on the status column
db.execute(
    "UPDATE task_queue SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
# Both return the same task dict
```

The `UPDATE` doesn't protect you here. Both workers update the row (the second update just overwrites the first), both return the task, both execute it, and both call `complete_task()`. The first `INSERT INTO task_results` succeeds. The second one hits the unique constraint and throws.

This is also why you're seeing **gaps in the reporting pipeline** — the `IntegrityError` is thrown inside `complete_task()` *after* the `UPDATE task_queue SET status = 'complete'` has already executed. The task is marked complete but no result row exists.

---

### Why Tests Didn't Catch It

The tests run a single worker against a single in-process database connection. There is no concurrency, so the race never occurs. The 34 tests passing is completely consistent with this bug.

---

### Why It Started Four Days Ago

Before the migration, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per-connection, and separate connections to the same file would have serialized at the file lock level — effectively preventing two workers from interleaving their SELECT/UPDATE at the microsecond level. The shared connection pool removes that accidental serialization and lets multiple workers interleave reads and writes freely.

---

### What Needs to Change

**The fix is to make claim and acquire atomic.** There are two solid options:

**Option 1 — Conditional UPDATE (preferred, no schema change):**

```python
def acquire_task(worker_id: int):
    # Claim it in a single atomic statement — only one worker can win
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Only the worker that won the UPDATE will find this row
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'running' AND worker_id = ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id,),
    )
    return dict(row) if row else None
```

This works because a single `UPDATE` statement is atomic — the database evaluates and applies it as one operation. Only one worker's UPDATE will match the `WHERE status = 'pending'` condition for a given row.

**Option 2 — Explicit transaction with SELECT FOR UPDATE (if the connection pool / DB supports it):**

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE",  # locks the row for the duration of this transaction
        )
        if not row:
            return None
        db.execute(
            "UPDATE task_queue SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )
    return dict(row)
```

Note: `FOR UPDATE` is not supported in SQLite. If the production database is SQLite, use Option 1.

---

### Also Fix the Gap in `complete_task()`

Regardless of the above fix, the ordering in `complete_task()` is dangerous. The task is marked `complete` before the result is written. If the INSERT fails for any reason, the task is lost silently. Reverse the order and keep both statements in the same transaction:

```python
def complete_task(task_id: int, result: list) -> None:
    # INSERT first — if this fails, the task stays 'running' and can be retried
    db.execute(
        "INSERT INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ?",
        (datetime.utcnow(), task_id),
    )
    db.commit()
```
