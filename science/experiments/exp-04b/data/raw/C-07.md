# Root Cause Analysis — `IntegrityError: UNIQUE constraint failed: task_results.task_id`

---

## The Failure, Read Backward

The constraint violation tells us exactly what happened: two workers inserted a row into `task_results` for the same `task_id`. That table has a unique constraint on `task_id`, so one of them won and one of them exploded. The missing results in the reporting pipeline are the tasks where the *loser* threw the exception — `complete_task` failed partway through, the result was never committed, and the task queue row may have been left in an inconsistent state depending on execution order.

The question is not *what* happened. The question is *how two workers got the same task*.

---

## The Root Cause

**`acquire_task` contains a classic check-then-act race condition. The SELECT and the UPDATE are not atomic.**

```python
# Worker A and Worker B both execute this SELECT at nearly the same instant.
# Both see the same row with status = 'pending'. Both return it.
row = db.query_one(
    "SELECT id, type, payload FROM task_queue"
    " WHERE status = 'pending'"
    " ORDER BY priority DESC, created_at ASC"
    " LIMIT 1"
)

# Worker A reaches this first, claims the task.
# Worker B reaches this a millisecond later, claims the same task again.
# Both commits succeed because the UPDATE has no guard — it doesn't
# check that status is still 'pending'.
db.execute(
    "UPDATE task_queue"
    " SET status = 'running', worker_id = ?, claimed_at = ?"
    " WHERE id = ?",
    (worker_id, datetime.utcnow(), row["id"]),
)
db.commit()
```

Between the moment Worker A reads the row and the moment it writes `status = 'running'`, there is an unprotected window. Worker B can read the same row during that window. Both workers claim the same task. Both call `complete_task`. Both attempt to insert into `task_results`. The second insert hits the unique constraint.

---

## Why This Wasn't Happening Before

The incident started four days ago. The recent change was a migration from **per-worker SQLite connections to a shared connection pool**. This is the trigger, not a coincidence.

With per-worker SQLite connections, each worker had its own connection to its own file, or at minimum SQLite's file-level locking serialized writes in a way that masked the race. The window existed, but the serialization made it nearly impossible to hit.

With a shared connection pool, multiple workers are now executing against the same database with genuine concurrency. The window that was always there is now wide open and being hit regularly under load.

**The tests pass because they run a single worker against a single in-memory database with no concurrency.** They cannot see this failure mode. They never could. The CI green check is meaningless for this class of bug.

---

## The Secondary Symptom Explained

The gaps in the reporting pipeline — tasks with no result at all — are the workers that *lost* the race. Their `complete_task` call threw `IntegrityError` on the INSERT. Depending on whether the ORM wraps that in a transaction, the preceding `UPDATE task_queue SET status = 'complete'` may or may not have been rolled back. If it was committed before the INSERT failed, that task is now marked `complete` in the queue with no corresponding row in `task_results`. It will never be retried. That is your data loss.

---

## What Needs to Change

### Fix 1 — Make the claim atomic (required)

The SELECT and UPDATE must be a single atomic operation. The standard pattern is to push the condition into the UPDATE's WHERE clause and check whether any row was actually modified:

```python
def acquire_task(worker_id: int):
    """Claim the next pending task atomically."""

    # Find the candidate without claiming it.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Claim it only if it is still pending.
    # The WHERE status = 'pending' is the guard.
    # If another worker claimed it between our SELECT and this UPDATE,
    # rowcount will be 0 and we return None instead of double-claiming.
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",
        (worker_id, datetime.utcnow(), row["id"]),
    ).rowcount
    db.commit()

    if rows_affected == 0:
        # Lost the race. Another worker claimed this task.
        return None

    return dict(row)
```

The `AND status = 'pending'` in the UPDATE WHERE clause is the lock. Only one UPDATE can match and modify the row. The other worker's UPDATE silently affects zero rows. `rowcount` tells you which outcome you got.

If your database supports `RETURNING` (PostgreSQL does; SQLite 3.35+ does), you can collapse this into a single statement and eliminate the SELECT entirely:

```sql
UPDATE task_queue
   SET status = 'running', worker_id = ?, claimed_at = ?
 WHERE id = (
       SELECT id FROM task_queue
        WHERE status = 'pending'
        ORDER BY priority DESC, created_at ASC
        LIMIT 1
       )
   AND status = 'pending'
RETURNING id, type, payload
```

That is a single round-trip with no window between read and write.

---

### Fix 2 — Make `complete_task` safe against partial failure (required)

The UPDATE and INSERT in `complete_task` must be wrapped in a single transaction. If the INSERT fails, the UPDATE must roll back. Right now the ORM may be auto-committing the UPDATE before the INSERT runs.

```python
def complete_task(task_id: int, result: list) -> None:
    with db.transaction():  # explicit transaction boundary
        db.execute(
            "UPDATE task_queue"
            " SET status = 'complete', completed_at = ?"
            " WHERE id = ?",
            (datetime.utcnow(), task_id),
        )
        db.execute(
            "INSERT INTO task_results (task_id, result, recorded_at)"
            " VALUES (?, ?, ?)",
            (task_id, json.dumps(result), datetime.utcnow()),
        )
        # commit happens at the end of the with block
        # rollback happens automatically if either statement raises
```

This does not fix the double-claim, but it prevents the data loss scenario where the queue row is marked `complete` with no result written.

---

### Fix 3 — Add a concurrency test (required before next deploy)

The test suite has no coverage for the failure mode that just took down production. Add one:

```python
import threading

def test_acquire_task_is_exclusive_under_concurrency(db, seed_tasks):
    """Two workers racing to claim the same task must not both succeed."""
    claimed = []
    errors = []

    def worker(wid):
        try:
            task = acquire_task(worker_id=wid)
            if task is not None:
                claimed.append(task["id"])
        except Exception as e:
            errors.append(e)

    threads = [threading.Thread(target=worker, args=(i,)) for i in range(1, 6)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    assert not errors, f"Workers raised exceptions: {errors}"
    # Each task ID must appear at most once across all claims
    assert len(claimed) == len(set(claimed)), (
        f"Same task claimed by multiple workers: {claimed}"
    )
```

This test would have caught the bug before it reached production.

---

## Summary

| | Detail |
|---|---|
| **Root cause** | Non-atomic check-then-act in `acquire_task`: SELECT and UPDATE are separate statements with no guard, allowing multiple workers to claim the same task |
| **Trigger** | Migration to shared connection pool enabled real concurrency; per-worker connections were masking the race |
| **Why tests passed** | All tests are single-worker, single-threaded; the race condition is physically impossible to hit in that environment |
| **Data loss mechanism** | `complete_task` UPDATE and INSERT are not in a single transaction; INSERT failure leaves task marked complete with no result |
| **Fix** | Add `AND status = 'pending'` to the UPDATE WHERE clause and check `rowcount`; wrap `complete_task` in an explicit transaction; add a concurrency test |

The visible bug here is the missing `AND status = 'pending'`. Do not stop there. The transaction boundary in `complete_task` is the second failure waiting to surface, and it will surface the next time you get a transient INSERT failure for any reason.
