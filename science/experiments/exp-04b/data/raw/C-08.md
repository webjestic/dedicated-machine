# Root Cause Analysis

## The Short Answer

Two workers can read the same pending task simultaneously. Both claim it. Both complete it. The second `INSERT INTO task_results` hits the unique constraint on `task_id`. This is a **check-then-act race condition** — a classic TOCTOU (Time of Check to Time of Use) bug.

The migration to a shared connection pool four days ago is what made this visible. With per-worker connections, SQLite's file-level locking was providing accidental serialization. That protection is gone now.

---

## The Failure Sequence

Here is exactly what the logs are showing you:

```
T=0ms   Worker-1: SELECT ... WHERE status = 'pending' → returns task #42
T=0ms   Worker-2: SELECT ... WHERE status = 'pending' → returns task #42  ← same row
T=1ms   Worker-1: UPDATE task_queue SET status = 'running' WHERE id = 42
T=1ms   Worker-2: UPDATE task_queue SET status = 'running' WHERE id = 42
T=50ms  Worker-1: execute_task(42) → result_A
T=50ms  Worker-2: execute_task(42) → result_B
T=51ms  Worker-1: INSERT INTO task_results (task_id=42, ...) → OK
T=51ms  Worker-2: INSERT INTO task_results (task_id=42, ...) → 💥 UNIQUE constraint failed
```

The `IntegrityError` is not the bug. It is the **symptom that proves the race happened**. The unique constraint is doing its job — it is the only thing preventing silent duplicate results from landing in your reporting pipeline.

The missing results in the downstream pipeline are the other half of this: Worker-2's exception causes its entire `complete_task` call to roll back, including the `UPDATE task_queue SET status = 'complete'`. That task may now be stuck in `running` state permanently, depending on whether you have a reaper process. If you do not, that result is gone.

---

## Why the Tests Are Green

The tests pass because they run one worker at a time, sequentially, against an in-process database. They cannot reproduce this failure. They never could. The test suite has no coverage for concurrent acquisition — there is no test that spins up two workers and has them race for the same task. The CI signal is meaningless for this class of bug.

## The Actual Bug

```python
# acquire_task — this is the broken window
def acquire_task(worker_id: int):
    row = db.query_one(          # ← Worker-1 and Worker-2 both read here
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # ← The window is here. Nothing prevents another worker from
    #   executing the same SELECT before either UPDATE runs.

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",         # ← Both workers update the same row
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

The SELECT and the UPDATE are two separate statements with no lock held between them. Under a shared connection pool with multiple workers, this window is real and reproducible under load.

---

## What Needs to Change

### Fix 1: Atomic Claim with a Guarded UPDATE (Correct)

Eliminate the SELECT entirely. Make the UPDATE the claim operation, and verify it actually claimed something by checking rows affected:

```python
def acquire_task(worker_id: int):
    """
    Atomically claim a task by updating it in a single statement.
    No separate SELECT means no window for another worker to race through.
    """
    # Find the best candidate first — this read is now just advisory,
    # the UPDATE below is what actually claims it.
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # The WHERE clause includes status = 'pending' — if another worker
    # claimed this row between our SELECT and this UPDATE, rows_affected
    # will be 0 and we bail out cleanly.
    rows_affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← guard on status
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    if rows_affected == 0:
        # Another worker claimed it first. This is not an error.
        return None

    # Now safe to fetch the full row — we own it.
    claimed = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],)
    )
    return dict(claimed)
```

The key is `WHERE id = ? AND status = 'pending'`. If another worker claimed the row first, the UPDATE matches zero rows. `rows_affected == 0` is your signal to back off. No exception, no retry loop needed — just return `None` and the worker's main loop will poll again.

> **Note:** Verify that your `db.execute()` wrapper returns rows affected. If it does not expose that, check `db.connection.rowcount` or adjust the wrapper. This is worth confirming before you ship.

---

### Fix 2: Skip the Advisory SELECT Entirely (Cleaner, Preferred)

If your database supports `RETURNING` (SQLite 3.35+, PostgreSQL), you can collapse this into a single round trip with no race window at all:

```python
def acquire_task(worker_id: int):
    """
    Single-statement atomic claim. No window. No advisory read.
    Requires SQLite >= 3.35 or PostgreSQL.
    """
    row = db.query_one(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        " )"
        " RETURNING id, type, payload",
        (worker_id, datetime.utcnow()),
    )
    db.commit()
    return dict(row) if row else None
```

This is the cleanest solution. One statement, one round trip, no window. The subquery and the UPDATE execute atomically within the database engine.

---

### Fix 3: Harden `complete_task` as a Defense-in-Depth Layer

Regardless of which fix above you ship, add an idempotency guard to `complete_task`. The unique constraint is already protecting you, but swallowing a duplicate gracefully is better than propagating an exception:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue"
        " SET status = 'complete', completed_at = ?"
        " WHERE id = ? AND status = 'running'",  # ← only complete if we still own it
        (datetime.utcnow(), task_id),
    )
    db.execute(
        # INSERT OR IGNORE means a duplicate silently no-ops rather than raising.
        # The first writer wins. This is correct behavior.
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

`INSERT OR IGNORE` means if the row already exists, the second insert is a no-op. No exception propagates. No rollback. The first writer's result stands. This does not fix the race — Fix 1 or Fix 2 does that — but it stops the error from surfacing and prevents the result from being lost when the race does occur.

---

## What to Do Right Now

**Immediate:** Deploy Fix 1 or Fix 2 to stop the bleeding. Fix 2 is cleaner if your SQLite version supports `RETURNING` — check with `SELECT sqlite_version()`. Fix 1 works on any version.

**Alongside it:** Deploy Fix 3. It costs nothing and makes the system resilient to any future claim-race that slips through.

**After the incident:** Add a test that actually reproduces this. Two threads, one database, both calling `acquire_task` simultaneously, assert that each gets a different task ID. That test would have caught this before it shipped.

**Audit the stuck tasks:** Any task currently in `running` status that has no corresponding row in `task_results` is a casualty of this incident. Depending on your task types, you may need to reset those to `pending` and reprocess them.
