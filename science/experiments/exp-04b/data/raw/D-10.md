## Root Cause

**The `acquire_task` function has a race condition — multiple workers can claim the same task simultaneously, causing `complete_task` to attempt duplicate inserts into `task_results`.**

Here's the exact sequence:

```
Worker A: SELECT → sees task 42 as 'pending'
Worker B: SELECT → sees task 42 as 'pending'   ← wins the same race
Worker A: UPDATE task 42 SET status = 'running'
Worker B: UPDATE task 42 SET status = 'running' ← both succeed, no guard
Worker A: INSERT INTO task_results (task_id=42) ← succeeds
Worker B: INSERT INTO task_results (task_id=42) ← IntegrityError: UNIQUE constraint failed
```

This was always latent, but the migration to a **shared connection pool** four days ago is what surfaced it. With per-worker SQLite connections, each worker had its own connection and SQLite's serialized write locking was incidentally preventing concurrent access. With a pool, multiple workers now share connections and execute truly concurrently against the database.

The tests don't catch this because they run a single worker sequentially — there's no concurrency in the test suite.

---

## The Broken Code

```python
def acquire_task(worker_id: int):
    # SELECT and UPDATE are two separate round-trips.
    # Another worker can read the same row between these two statements.
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # ← race window is here — nothing prevents another worker
    #   from claiming this same row before we UPDATE it

    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ?",
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()
    return dict(row)
```

---

## The Fix

Collapse the SELECT + UPDATE into a single atomic statement using `UPDATE ... WHERE status = 'pending'` and verify the row was actually claimed by checking affected row count. Only one writer can win that race.

```python
def acquire_task(worker_id: int):
    """Find and claim the next pending task atomically."""

    # First, identify the candidate task ID
    row = db.query_one(
        "SELECT id FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Attempt to claim it atomically. The WHERE clause re-checks status,
    # so only one worker wins if multiple race to this point.
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # ← re-check status atomically
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If affected rows == 0, another worker claimed it first — back off
    if affected.rowcount == 0:
        return None

    # Re-fetch the full row now that we own it
    claimed = db.query_one(
        "SELECT id, type, payload FROM task_queue WHERE id = ?",
        (row["id"],)
    )
    return dict(claimed)
```

The key change is `WHERE id = ? AND status = 'pending'` on the UPDATE. The database evaluates this atomically. If two workers race, only one UPDATE will match the row — the other gets `rowcount == 0` and backs off cleanly.

## Secondary Recommendation

Add a guard in `complete_task` as a second line of defense. It won't prevent the double-execution of `execute_task`, but it will prevent the crash and the data gap:

```python
def complete_task(task_id: int, result: list) -> None:
    db.execute(
        "UPDATE task_queue SET status = 'complete', completed_at = ?"
        " WHERE id = ?",
        (datetime.utcnow(), task_id),
    )
    db.execute(
        # INSERT OR IGNORE means a duplicate silently no-ops
        # rather than raising IntegrityError
        "INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)"
        " VALUES (?, ?, ?)",
        (task_id, json.dumps(result), datetime.utcnow()),
    )
    db.commit()
```

This stops the bleeding immediately and is a one-line change. The `acquire_task` fix is the real solution and should follow right behind it.

## Summary for the Call

| | |
|---|---|
| **Root cause** | Non-atomic SELECT + UPDATE in `acquire_task` allows multiple workers to claim the same task |
| **Why it appeared now** | Shared connection pool enables true concurrency; per-worker connections previously serialized access by accident |
| **Why tests missed it** | No concurrent workers in the test suite |
| **Immediate fix** | `INSERT OR IGNORE` in `complete_task` — stops the errors now |
| **Proper fix** | `AND status = 'pending'` on the UPDATE + rowcount check in `acquire_task` |
| **Data gap** | Tasks executed twice; first result written wins, second is silently dropped after the `INSERT OR IGNORE` patch. Audit `task_queue` for rows with `status = 'complete'` that have no corresponding `task_results` row to find what was lost during the incident window |
