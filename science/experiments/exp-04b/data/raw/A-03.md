## Root Cause

**The `acquire_task` function has a check-then-act race condition. Two workers can read the same pending task row before either one writes the status update. Both proceed to `execute_task` and then both attempt `INSERT INTO task_results` for the same `task_id`, which violates the unique constraint.**

---

## The Exact Sequence

```
Time →

Worker 1:  SELECT (status='pending', id=42) ─────────────────── UPDATE id=42 ── commit ── execute ── INSERT ── commit
                                                    ↑
Worker 2:  ────────────────── SELECT (status='pending', id=42) ─────────────── UPDATE id=42 ── commit ── execute ── INSERT ── 💥 IntegrityError
```

Both SELECTs complete before either UPDATE commits. At the moment Worker 2 reads, the row still says `pending`. Both workers claim the same task. One INSERT wins. The other throws. The winning INSERT is also not guaranteed to be the more complete result — it's a race.

This also explains the **missing results in the reporting pipeline**: when Worker 2's INSERT throws, `complete_task` raises before the `UPDATE task_queue SET status='complete'` is committed (it's in the same transaction, and the commit never lands). That task row may be left in a partially-written or inconsistent state depending on how the ORM handles the exception.

---

## Why Tests Don't Catch It

The tests run a single worker against a single in-process database connection. There is no concurrency. The race window does not exist in the test environment. All 34 tests pass cleanly and always will under these conditions. This is the exact failure mode where a green CI gives false confidence.

---

## Why It Started Now

The migration to a shared connection pool four days ago is the trigger. Previously, each worker had its own SQLite connection. SQLite's default journal mode serializes writes per connection, and per-worker connections meant the SELECT and UPDATE for a given worker were effectively isolated from other workers at the connection level. With a shared pool, multiple workers are now issuing reads and writes through the same pool concurrently, and the window between SELECT and UPDATE is fully exposed.

---

## What Needs to Change

There are two viable fixes. Pick one based on what the team can ship fastest.

---

### Fix 1 — Atomic claim using `UPDATE ... WHERE status = 'pending'` (preferred, ship this)

Remove the SELECT entirely. Do the claim in a single UPDATE and then read back the claimed row. Only one writer can set `status = 'running'` for a given row.

```python
def acquire_task(worker_id: int):
    # Single atomic operation — no window between check and act
    db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ("
        "   SELECT id FROM task_queue"
        "   WHERE status = 'pending'"
        "   ORDER BY priority DESC, created_at ASC"
        "   LIMIT 1"
        ")",
        (worker_id, datetime.utcnow()),
    )
    db.commit()

    # Now read back only the row this worker just claimed
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE worker_id = ? AND status = 'running' AND claimed_at >= ?"
        " ORDER BY claimed_at DESC LIMIT 1",
        (worker_id, datetime.utcnow()),
    )
    return dict(row) if row else None
```

> **Note on the read-back:** The timestamp approach above is fragile under clock skew. A cleaner option if the ORM exposes it is `last_insert_rowid()` / `lastrowid` on the cursor, or adding a `claimed_by_run_id` UUID column. Flag this as a follow-up if you ship the timestamp version now.

---

### Fix 2 — Explicit `SELECT FOR UPDATE` / serialized lock (if ORM supports it)

If the ORM upgrade to 3.2.1 exposes `SELECT FOR UPDATE`, wrap the acquire in a transaction with a row-level lock. This is standard on Postgres. SQLite does not support `SELECT FOR UPDATE` — if the backend is SQLite, this option is not available and Fix 1 is the only path.

```python
def acquire_task(worker_id: int):
    with db.transaction():
        row = db.query_one(
            "SELECT id, type, payload FROM task_queue"
            " WHERE status = 'pending'"
            " ORDER BY priority DESC, created_at ASC"
            " LIMIT 1"
            " FOR UPDATE",          # holds a write lock until transaction commits
        )
        if not row:
            return None

        db.execute(
            "UPDATE task_queue"
            " SET status = 'running', worker_id = ?, claimed_at = ?"
            " WHERE id = ?",
            (worker_id, datetime.utcnow(), row["id"]),
        )
    # transaction commits here, lock released
    return dict(row)
```

## Secondary Issue to Fix After the Incident

`complete_task` updates `task_queue` and inserts into `task_results` in the same function but there is no explicit transaction wrapping both operations. If the INSERT throws (as it is now), the UPDATE may or may not have been flushed depending on pool and ORM autocommit behavior. Wrap both statements in a single explicit transaction so they succeed or fail together. This is not the cause of the current incident but it is the reason some tasks are showing as incomplete in the queue while also missing from results.

```python
def complete_task(task_id: int, result: list) -> None:
    with db.transaction():
        db.execute(
            "UPDATE task_queue SET status = 'complete', completed_at = ?"
            " WHERE id = ?",
            (datetime.utcnow(), task_id),
        )
        db.execute(
            "INSERT INTO task_results (task_id, result, recorded_at)"
            " VALUES (?, ?, ?)",
            (task_id, json.dumps(result), datetime.utcnow()),
        )
```

| | |
|---|---|
| **Root cause** | `acquire_task` reads then writes in two separate statements. Multiple workers claim the same task in the gap. |
| **Trigger** | Shared connection pool replaced per-worker connections, exposing the race that was previously hidden. |
| **Fix** | Collapse the SELECT + UPDATE into a single atomic UPDATE in `acquire_task`. |
| **Why tests missed it** | All tests are single-worker, single-connection. The race requires concurrency to manifest. |
| **Secondary cleanup** | Wrap `complete_task` in an explicit transaction. |
