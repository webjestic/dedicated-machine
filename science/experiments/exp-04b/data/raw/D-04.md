## Root Cause

**The `acquire_task` function has a race condition — multiple workers can claim the same task simultaneously, causing `complete_task` to attempt duplicate inserts into `task_results`.**

Here's the exact sequence:

```
Worker A: SELECT → finds task_id=42 (status='pending')
Worker B: SELECT → finds task_id=42 (status='pending')  ← same row, before A's UPDATE commits
Worker A: UPDATE task_id=42 SET status='running' → commits
Worker B: UPDATE task_id=42 SET status='running' → commits (overwrites A's claim)
Worker A: INSERT INTO task_results (task_id=42) → succeeds
Worker B: INSERT INTO task_results (task_id=42) → 💥 UNIQUE constraint failed
```

This is a classic **check-then-act** race. The SELECT and UPDATE are two separate operations with no lock held between them.

---

## Why Tests Don't Catch It

The tests run a single worker against a single-threaded in-memory SQLite database. There is no concurrency, so the race window never opens. All 34 tests pass cleanly and always will under these conditions.

---

## Why It Started Now

The migration to a **shared connection pool** (`db-pool 2.1`) is what exposed this. Previously each worker had its own SQLite connection. SQLite serializes writes per-connection, so workers were accidentally protected by isolation. With a shared pool, multiple workers now interleave reads and writes against the same connection pool with real concurrency, and the race is live.

---

## What Needs to Change

The fix is to make the SELECT and claim atomic. There are two solid options:

### Option 1 — Conditional UPDATE (recommended, minimal change)

Claim the task in a single statement by targeting `status='pending'` in the `WHERE` clause. Only one worker can win the UPDATE; the others update zero rows.

```python
def acquire_task(worker_id: int):
    # First, find a candidate (no lock held yet — that's fine)
    row = db.query_one(
        "SELECT id, type, payload FROM task_queue"
        " WHERE status = 'pending'"
        " ORDER BY priority DESC, created_at ASC"
        " LIMIT 1"
    )

    if not row:
        return None

    # Atomic claim: only succeeds if this worker gets there first
    affected = db.execute(
        "UPDATE task_queue"
        " SET status = 'running', worker_id = ?, claimed_at = ?"
        " WHERE id = ? AND status = 'pending'",  # <-- status guard added
        (worker_id, datetime.utcnow(), row["id"]),
    )
    db.commit()

    # If another worker claimed it first, affected rows == 0 — back off
    if affected.rowcount == 0:
        return None

    return dict(row)
```

The `AND status = 'pending'` guard turns the UPDATE into the actual claim. The first writer wins; every other concurrent writer sees `rowcount == 0` and returns `None`, letting the worker loop pick up the next available task on the next iteration.

### Option 2 — `SELECT ... FOR UPDATE` (if the connection pool / ORM supports it)

If `db-pool 2.1` exposes row-level locking, wrap the SELECT and UPDATE in an explicit transaction with `FOR UPDATE`. This is more idiomatic on PostgreSQL but SQLite doesn't support it, so check your actual database engine before going this route.

---

## Secondary Issue Worth Noting

`complete_task` also has no guard against inserting a result that already exists. Even after the acquire fix, consider adding a defensive `INSERT OR IGNORE` (SQLite) or `ON CONFLICT DO NOTHING` as a belt-and-suspenders measure:

```sql
INSERT OR IGNORE INTO task_results (task_id, result, recorded_at)
VALUES (?, ?, ?)
```

This won't fix the race but it will prevent the `IntegrityError` from crashing the worker and dropping the result on the floor — which is what's causing the gaps in the downstream reporting pipeline right now.
