# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** Current Sprint

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous HTTP client in an async endpoint**

```python
# CURRENT — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous call inside async function
```

Using `requests` inside an `async def` route is a significant architectural problem. This blocks the entire event loop for the duration of the HTTP call, defeating the purpose of async FastAPI and degrading performance under any real load. This should be `httpx.AsyncClient` or `aiohttp`.

```python
# RECOMMENDED
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

---

**2. No authentication or authorization on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

This endpoint is completely open. Any unauthenticated caller can trigger notifications to arbitrary `user_id` values with arbitrary `message` content. In a government contracting context, this is a non-starter. At minimum this needs:

- A dependency-injected auth check (API key, OAuth2 bearer token, or internal service token)
- Verification that the calling identity is authorized to notify the specified `user_id`

```python
# RECOMMENDED
from fastapi import Depends
from api.auth import require_internal_service_token  # your auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_service_token)
) -> NotificationResponse:
```

---

**3. `message` field is unbounded and unvalidated**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no constraints whatsoever
```

An open `message: str` with no length cap or content validation is a vector for payload injection into the downstream notification service. If that service renders HTML or passes content to a template engine, this could escalate. Pydantic makes this trivial to fix.

```python
from pydantic import BaseModel, Field, constr

class NotificationRequest(BaseModel):
    user_id: str = Field(..., pattern=r"^user_[a-zA-Z0-9]{1,64}$")
    order_id: str = Field(..., pattern=r"^order_[a-zA-Z0-9]{1,64}$")
    message: str = Field(..., min_length=1, max_length=500)
```

---

**4. Hardcoded internal service URL**

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

Hardcoded URLs are a configuration management and security problem. The URL should come from environment configuration, validated at startup. This also makes environment promotion (dev → staging → prod) impossible without code changes.

```python
# RECOMMENDED
import os
from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

### 🟡 Moderate Issues

**5. No outbound request authentication to the notification service**

```python
response = requests.post(
    "https://notifications.internal/send",
    json={...},
    timeout=5,
)
```

There are no headers, no bearer token, no HMAC signature — nothing authenticating this service to the downstream notification endpoint. If `notifications.internal` is ever reachable from outside the trust boundary (misconfigured network, SSRF, etc.), unauthenticated requests will be accepted. Service-to-service auth should be enforced.

---

**6. Sensitive data potentially leaking through logs**

```python
logger.error("Notification service error: %s", e)
```

Depending on what the `requests.RequestException` contains, this could log response bodies, internal URLs with tokens in query params, or other sensitive data. Exception messages from HTTP libraries can be verbose. The log statement should be scoped and sanitized.

```python
# RECOMMENDED
logger.error(
    "Notification service error for order_id=%s: %s",
    request.order_id,
    type(e).__name__  # log the exception type, not the full message
)
```

---

**7. `data["notification_id"]` will raise an unhandled `KeyError`**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if key is absent
)
```

If the downstream service returns a 200 with a malformed or unexpected payload, this raises an unhandled `KeyError` that will surface as a 500 with a raw traceback. This should be explicitly handled.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: missing notification_id")
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

### 🟠 Test Quality Issues

**8. Test catches `Exception`, not `requests.RequestException`**

```python
# test
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The production code catches `requests.RequestException`. The test raises a bare `Exception`. This test is not actually validating the error handling path — it's passing for the wrong reason. The test should raise `requests.exceptions.ConnectionError` or another subclass of `RequestException` to match the real failure mode.

```python
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.exceptions.ConnectionError("Connection refused")):
```

---

**9. No test coverage for the `KeyError` path or malformed response**

Given finding #7, there is zero test coverage for the case where the notification service returns 200 but with a missing or malformed `notification_id`. This is a realistic failure mode that needs a test.

---

**10. No rate limiting**

For a checkout-triggered endpoint with no auth, there is no rate limiting. This is an open amplification vector — an attacker could spam this endpoint to flood the notification service or enumerate valid `user_id`/`order_id` combinations via timing differences. Rate limiting should be applied at the router or middleware level.

**Decision:** Request Changes

**Summary:** This PR has four critical issues that block merge: a synchronous HTTP call inside an async route that will degrade performance under load, a completely unauthenticated public endpoint that allows arbitrary notification triggering, an unbounded message field with no content validation, and a hardcoded internal service URL. Beyond the critical items, there are moderate concerns around outbound auth, log hygiene, and an unhandled `KeyError` on response parsing, plus test coverage that is validating the wrong exception type. All critical issues must be resolved before this is reconsidered, and I'd strongly recommend a security review pass after the fixes are in.
