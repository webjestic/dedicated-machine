# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** 2025-07-11

---

## Detailed Findings

### 🔴 Critical Issues

**1. Blocking Synchronous HTTP Call in an Async Endpoint**

This is the most glaring architectural problem. The endpoint is declared `async` but calls `requests.post()`, which is a synchronous, blocking I/O call. This will block the entire event loop for the duration of the network call, defeating the purpose of async and degrading performance under any meaningful load.

```python
# CURRENT — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # ❌ blocking call inside async context
```

```python
# REQUIRED — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)  # ✅ non-blocking
```

Replace `requests` with `httpx` and use `await`. The `requests` library has no place in an async FastAPI service.

---

**2. Unauthenticated Endpoint — No Authorization Layer**

The `/notify` endpoint has zero authentication or authorization. Any caller — internal or external — can trigger notifications for any `user_id` and `order_id` combination. This is a serious concern in a government contracting context.

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    # ❌ No auth dependency whatsoever
```

At minimum this needs a dependency injection guard:

```python
from fastapi import Depends
from api.auth import require_internal_token  # or OAuth2, API key, etc.

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_token)  # ✅ enforce auth
) -> NotificationResponse:
```

---

**3. Hardcoded Internal Service URL**

The URL `https://notifications.internal/send` is hardcoded directly in the business logic. This is a deployment and security liability.

```python
# CURRENT — hardcoded, untestable, environment-unaware
response = requests.post(
    "https://notifications.internal/send",  # ❌
    ...
)
```

```python
# REQUIRED — pull from environment/config
import os

NOTIFICATION_SERVICE_URL = os.getenv("NOTIFICATION_SERVICE_URL")
if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL is not configured")

response = await client.post(NOTIFICATION_SERVICE_URL, ...)  # ✅
```

Hardcoded URLs make environment promotion (dev → staging → prod) fragile and create a maintenance burden. They also make it trivial for someone reading the code to map internal infrastructure.

### 🟡 Significant Concerns

**4. `message` Field is Unbounded and Unvalidated**

The `message` field on `NotificationRequest` accepts any arbitrary string with no length constraint. This opens the door to payload bloat, potential injection into downstream systems, and abuse.

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # ❌ no constraints
```

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)  # ✅ bounded
```

The `user_id` and `order_id` fields should also have format constraints to prevent unexpected values from being forwarded to the external service.

---

**5. Raw Exception Detail Logged — Potential Information Leakage**

```python
except requests.RequestException as e:
    logger.error("Notification service error: %s", e)  # ⚠️ leaks internal detail
    raise HTTPException(status_code=503, detail="Notification service unavailable")
```

The `HTTPException` detail is fine — it's generic. But `%s` formatting on a raw exception can expose internal hostnames, stack traces, or service topology in log aggregators. Prefer structured logging with controlled fields:

```python
logger.error(
    "Notification service error",
    extra={"error_type": type(e).__name__, "order_id": request.order_id}
)
```

Never log the full exception object where log output is forwarded to external SIEM or monitoring tools without scrubbing.

---

**6. No Retry or Circuit Breaker Logic**

A single `timeout=5` with no retry strategy means transient failures immediately surface as 503s to the end user. For a checkout-critical path, this is a reliability concern. At minimum, document that retry logic is intentionally deferred and tracked elsewhere. If it's not tracked, it needs to be added.

Consider `tenacity` for retry with exponential backoff, or offload to a task queue (Celery, ARQ) so notifications are decoupled from the checkout response path entirely.

### 🟠 Test Quality Issues

**7. Wrong Exception Type in Failure Test**

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # ❌ too broad
```

The production code catches `requests.RequestException`. The test raises a bare `Exception`, which is a different type. This test is not actually validating the error handling path — it's passing for the wrong reason. Once you migrate to `httpx`, use `httpx.RequestError` consistently.

```python
import httpx

with patch("api.notifications.requests.post",
           side_effect=httpx.RequestError("Connection refused")):  # ✅ correct type
```

---

**8. No Test for Auth Rejection**

Once authentication is added (see finding #2), there must be a corresponding test asserting that unauthenticated requests receive a `401` or `403`. This is non-negotiable for a government-adjacent system.

---

**9. `notification_id` Key Access Has No Guard**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # ❌ KeyError if key is absent
)
```

If the external service returns a malformed response, this raises an unhandled `KeyError` that will bubble up as a 500. Use `.get()` with a fallback and handle the missing key explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Missing notification_id in service response", extra={"order_id": request.order_id})
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

**Decision:** Request Changes

**Summary:** This PR has three blocking issues that prevent approval: a synchronous `requests` call inside an async endpoint that will stall the event loop under load, a completely unauthenticated endpoint that allows arbitrary notification triggering, and a hardcoded internal service URL that is both an operational and a mild infrastructure-disclosure risk. Beyond the blockers, the Pydantic models lack meaningful field constraints, the failure test is asserting against the wrong exception type and therefore not actually testing the intended code path, and there is no guard against a malformed response from the external service. None of these are minor — address all findings before resubmission.
