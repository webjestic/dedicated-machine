# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** Current Sprint

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous HTTP call inside an async endpoint**

```python
# CURRENT — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, will starve other coroutines
```

Using `requests` (synchronous) inside an `async` def is a serious architectural problem in FastAPI. This blocks the entire event loop for the duration of the network call. Under any real load, this will degrade performance across *all* endpoints, not just this one.

```python
# REQUIRED — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

---

**2. No authentication or authorization on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

This endpoint is completely open. Any unauthenticated caller can trigger notifications to arbitrary `user_id` values with arbitrary `message` content. Given this is a government contracting environment, this is a non-starter.

At minimum, require a dependency-injected auth check:

```python
from fastapi import Depends
from api.auth import require_internal_service  # your existing auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_service)
) -> NotificationResponse:
```

---

**3. `message` field is unconstrained — potential injection / abuse vector**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no length limit, no content validation
```

An unbounded `message` field passed directly to an external service is a risk. This could be used for payload stuffing, log injection, or abusing the downstream notification service. Constrain it:

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

### 🟡 Moderate Issues

**4. Hardcoded internal service URL**

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

This is not configurable across environments (dev, staging, prod) and makes secret rotation or service migration painful. Pull this from config:

```python
from api.config import settings  # or however your settings are structured

response = await client.post(
    settings.NOTIFICATION_SERVICE_URL,
    ...
)
```

---

**5. `data["notification_id"]` is accessed without key validation**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if upstream changes contract
)
```

If the external service returns an unexpected shape, this raises an unhandled `KeyError` that will surface as a 500 with a potentially revealing traceback. Validate the response:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**6. Error logging may leak sensitive data**

```python
logger.error("Notification service error: %s", e)
```

Depending on what the `requests.RequestException` captures, this could log internal service URLs, headers, or partial payloads to wherever your log sink is. Be explicit about what you log:

```python
logger.error("Notification service error for order_id=%s: %s", request.order_id, type(e).__name__)
```

---

### 🔵 Minor / Test Quality Issues

**7. Test catches the wrong exception type**

```python
# test_send_notification_service_unavailable
side_effect=Exception("Connection refused")  # too broad
```

The production code catches `requests.RequestException`. This test uses the base `Exception`, which means it's not actually testing the real error handling path. It passes, but for the wrong reason. It should use `requests.RequestException` or a specific subclass like `requests.ConnectionError`.

```python
import requests as req_lib

side_effect=req_lib.ConnectionError("Connection refused")
```

---

**8. No test for the `KeyError` / malformed upstream response path**

Once issue #5 is addressed, a test covering a malformed upstream payload should be added. This is a gap in coverage that will matter in production.

---

**9. No rate limiting consideration**

For a checkout notification endpoint, there is no mention of rate limiting. A misconfigured client or a replay attack could hammer both this service and the downstream notification provider. This should be addressed at the router or middleware level.

**Decision: Request Changes**

**Summary:** This PR introduces three critical issues that block approval: a synchronous HTTP call inside an async endpoint that will degrade system-wide performance under load, a completely unauthenticated endpoint that allows arbitrary actors to trigger notifications with unconstrained message content, and insufficient input validation on all three request fields. The test suite also contains a subtle but meaningful flaw in how the error path is exercised. All critical items must be resolved before this can be merged, and the moderate items should be addressed in the same pass given the security posture required at this organization.
