# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** Current Sprint

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous `requests` in an Async Endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

Using the synchronous `requests` library inside an `async` def is a serious architectural problem. This will block the entire event loop for the duration of the HTTP call, degrading performance under any meaningful load. This should be replaced with `httpx.AsyncClient` or `aiohttp`.

```python
# Recommended
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

---

**2. No Authentication or Authorization on the Endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

There is zero auth on this endpoint. Any unauthenticated caller can trigger notifications to arbitrary `user_id` and `order_id` values with arbitrary `message` content. In a government contracting context, this is a non-starter. At minimum this needs a dependency-injected auth guard.

```python
from fastapi import Depends
from api.auth import require_authenticated_user  # your internal auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    current_user=Depends(require_authenticated_user)
) -> NotificationResponse:
```

---

**3. No Ownership Validation — Broken Object Level Authorization (BOLA)**

Even if auth is added, there is no check that the authenticated user is actually allowed to trigger a notification for the given `user_id` and `order_id`. This is a textbook BOLA / IDOR vulnerability. A logged-in user could send notifications on behalf of any other user.

---

**4. Unvalidated `message` Field — Potential Injection Vector**

```python
message: str
```

The `message` field is a raw, unbounded string passed directly to an external service with no sanitization, length limits, or content constraints. This opens the door to:

- Notification spam / abuse
- Potential injection depending on how the downstream service processes the message

```python
from pydantic import Field

message: str = Field(..., max_length=500, strip_whitespace=True)
```

---

**5. Hardcoded Internal Service URL**

```python
"https://notifications.internal/send"
```

Hardcoded URLs are a configuration management problem and a deployment risk. This should be pulled from environment configuration, not baked into source code.

```python
import os
NOTIFICATION_SERVICE_URL = os.getenv("NOTIFICATION_SERVICE_URL")

if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL is not configured")
```

---

**6. Sensitive Data Leakage in Logs**

```python
logger.error("Notification service error: %s", e)
```

Depending on what the exception contains, this can log `user_id`, `order_id`, or internal service details into your log stream. In a government-adjacent system, PII in logs is a compliance concern (think NIST 800-53, FedRAMP). Sanitize what gets logged.

```python
logger.error("Notification service error for order processing. RequestException type: %s", type(e).__name__)
```

---

### 🟡 Moderate Issues

**7. `data["notification_id"]` — No KeyError Guard**

```python
notification_id=data["notification_id"]
```

If the external service returns a 200 but omits `notification_id`, this raises an unhandled `KeyError` that will bubble up as a 500. Use `.get()` with a fallback or validate the response shape explicitly.

```python
notification_id = data.get("notification_id")
if not notification_id:
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**8. No Rate Limiting**

Without auth and without rate limiting, this endpoint is trivially abusable for notification flooding. This needs rate limiting middleware or a dependency guard before going anywhere near production.

---

**9. Test Coverage Gaps**

The test suite only covers the happy path and a generic exception. Missing coverage includes:

- Auth rejection (401/403) — once auth is added
- BOLA attempt (user sending notification for another user's order)
- Malformed/oversized `message` field
- External service returns 200 but missing `notification_id`
- External service returns non-200 (e.g., 400, 500)

The `test_send_notification_service_unavailable` test is also patching `Exception` rather than `requests.RequestException`, which doesn't accurately reflect the production error handling path.

---

### 🟢 What's Done Well

- Pydantic models are used correctly for request/response shaping
- `timeout=5` is set on the external call — good practice
- `raise_for_status()` is used — correct
- Logger is instantiated with `__name__` — correct pattern
- Router is properly separated from the app instance

---

## Summary

**Decision: 🔴 Request Changes**

**Summary:** This PR has multiple blocking issues that prevent approval — most critically: an unauthenticated endpoint with no ownership validation (BOLA), a synchronous HTTP call blocking the async event loop, a hardcoded internal service URL, and an unbounded message field that could be abused as an injection or spam vector. Given our government contracting posture and compliance obligations, none of these are acceptable in their current state. The foundation is structurally reasonable, but this needs a full pass on auth, input constraints, async correctness, and configuration management before it can merge.
