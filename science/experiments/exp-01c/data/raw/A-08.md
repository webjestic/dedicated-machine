# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** Current Sprint

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous `requests` in an Async Endpoint**

```python
# CURRENT — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

Using the synchronous `requests` library inside an `async` function will **block the entire event loop** for the duration of the HTTP call. Under any real load, this will degrade performance across all concurrent requests. This should be using `httpx.AsyncClient` or similar.

```python
# RECOMMENDED
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

---

**2. No Authentication or Authorization on the Endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

This endpoint is completely unauthenticated. Any actor who can reach this service can trigger notifications to arbitrary users with arbitrary messages. In a government contracting context, this is a **serious access control failure**. At minimum, require a verified caller identity — API key, OAuth2 bearer token, or internal service-to-service auth.

```python
# RECOMMENDED
from fastapi import Depends, Security
from api.dependencies import verify_api_key  # or OAuth2 scheme

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(verify_api_key)
) -> NotificationResponse:
```

---

**3. `message` Field is Unbounded and Unvalidated**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no constraints whatsoever
```

The `message` field has no length limit, no content validation, and is passed **directly to an external service**. This opens the door to payload injection attacks and potential abuse of the downstream notification service. Apply constraints:

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

### 🟡 Significant Concerns

**4. Hardcoded Internal Service URL**

```python
response = requests.post(
    "https://notifications.internal/send",
    ...
)
```

Hardcoded URLs are a maintenance and security liability. The URL should be externalized to configuration — environment variable or a settings module — so it can be changed per environment without touching source code and won't accidentally leak internal topology in version control.

```python
# RECOMMENDED
import os
NOTIFICATION_SERVICE_URL = os.environ.get("NOTIFICATION_SERVICE_URL")
if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL is not configured")
```

---

**5. Blind Trust of External Service Response**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a 200 but omits `notification_id`, this raises an unhandled `KeyError` which will produce a raw 500. Validate the response structure explicitly:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**6. No Rate Limiting**

Given this endpoint is triggered at checkout and is unauthenticated (see finding #2), there is nothing preventing a caller from hammering it. Even after auth is added, rate limiting per user or per caller should be applied to prevent notification spam and downstream service abuse.

---

**7. No TLS/Certificate Verification Enforcement**

`requests.post` defaults to verifying TLS, which is good, but this is not explicit and there is no documented policy around it. In a government contracting environment, mTLS or explicit certificate pinning to the internal notification service should be considered and at minimum documented as a conscious decision.

---

### 🔵 Code Quality & Architecture Notes

**8. Test Catches `Exception`, Not `requests.RequestException`**

```python
# In test
side_effect=Exception("Connection refused")

# In production code
except requests.RequestException as e:
```

The test is patching with a base `Exception`, but the production handler only catches `requests.RequestException`. The test **does not actually validate the real error handling path**. This is a false positive — the test passes but doesn't prove the handler works correctly.

```python
# CORRECT
import requests as req_lib
side_effect=req_lib.exceptions.ConnectionError("Connection refused")
```

---

**9. Logger Not Configured with a Name Tied to Module Hierarchy**

Minor, but `logging.getLogger(__name__)` is correct here — no issue. However, there is no indication that log levels or handlers are configured at the application level. Ensure sensitive fields (`user_id`, `order_id`) are not inadvertently logged at DEBUG level elsewhere in the call chain.

---

**10. Router Not Prefixed**

```python
router = APIRouter()
```

No prefix is defined on the router. Depending on how this is mounted in the main app, the endpoint could end up at a root-level `/notify` with no versioning or namespace. Recommend:

```python
router = APIRouter(prefix="/v1/notifications", tags=["notifications"])
```

**Decision: Request Changes**

**Summary:** This PR introduces three critical issues — a blocking synchronous HTTP call inside an async handler, a completely unauthenticated public-facing endpoint that accepts arbitrary user-targeted messages, and no input constraints on the `message` field — any one of which would be unacceptable in a production government contracting environment. Beyond the critical items, the test suite contains a logic error that provides false confidence in the error handling path, and the hardcoded internal service URL is a configuration and security liability. This PR should not merge until all critical findings are resolved and the significant concerns are addressed or explicitly documented with justification.
