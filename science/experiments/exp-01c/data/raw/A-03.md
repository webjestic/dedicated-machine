# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** 2025-07-10

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous HTTP call inside an async endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocks entire thread
```

Using `requests` (a synchronous library) inside an `async` FastAPI route is a serious architectural mistake. This will block the entire event loop for the duration of the network call, effectively making your async server behave synchronously under load. Replace with `httpx` or `aiohttp`.

```python
# Recommended
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(...)
```

---

**2. No authentication or authorization on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

This endpoint is completely open. Any unauthenticated caller can trigger notifications to arbitrary `user_id` values. In a government contracting context, this is a non-starter. At minimum, require a verified JWT or API key dependency.

```python
from fastapi import Depends
from api.auth import require_authenticated_user  # your existing auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    current_user=Depends(require_authenticated_user)
) -> NotificationResponse:
```

---

**3. No authorization check — user can notify on behalf of any other user**

Even if authentication is added, the endpoint does not verify that `request.user_id` matches the authenticated caller. This is a broken object-level authorization (BOLA/IDOR) vulnerability. A logged-in user could supply any `user_id` and trigger notifications for other users.

```python
# After auth is added, enforce ownership
if request.user_id != current_user.id:
    raise HTTPException(status_code=403, detail="Forbidden")
```

---

**4. `message` field is unbounded and unvalidated**

```python
message: str
```

There are no length constraints on `message`. This opens the door to payload bloat being forwarded directly to the internal service, and potentially log injection if the message content is ever logged. Apply field constraints.

```python
from pydantic import Field

message: str = Field(..., min_length=1, max_length=500, strip_whitespace=True)
```

---

**5. Hardcoded internal service URL**

```python
response = requests.post(
    "https://notifications.internal/send",
    ...
)
```

Hardcoding infrastructure URLs is a configuration management violation. This cannot be rotated, environment-switched, or overridden without a code change and redeployment. Use environment-based configuration.

```python
from api.config import settings  # pydantic-settings or equivalent

response = await client.post(settings.notification_service_url, ...)
```

---

### 🟡 Moderate Issues

**6. No HTTP client authentication to the internal service**

The outbound `requests.post` call carries no API key, bearer token, or mTLS configuration. Internal services should not be implicitly trusted — defense in depth applies here, especially in a government-adjacent environment.

```python
headers = {"Authorization": f"Bearer {settings.notification_service_token}"}
async with httpx.AsyncClient(timeout=5.0) as client:
    response = await client.post(url, json=payload, headers=headers)
```

---

**7. Overly broad exception catch in the test**

```python
# In test_send_notification_service_unavailable
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`, but the test throws a bare `Exception`. These are not equivalent. The test is not actually validating the error handling path it claims to test. This means the real failure mode (a `requests.RequestException`) is untested.

```python
import requests as req_lib

side_effect=req_lib.RequestException("Connection refused")
```

---

**8. No retry logic or circuit breaker**

A single timeout of 5 seconds with no retry strategy means transient failures result in immediate 503s to the end user. For a checkout-critical path, consider at minimum a single retry with exponential backoff, or integration with a circuit breaker pattern (e.g., `tenacity`).

---

**9. `notification_id` key access is unguarded**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if upstream changes contract
)
```

If the upstream service changes its response shape, this raises an unhandled `KeyError` that will surface as an uncontrolled 500. Use `.get()` with a fallback or validate the response with a Pydantic model.

---

**10. Logger leaks exception details**

```python
logger.error("Notification service error: %s", e)
```

Depending on what `e` contains, this could log internal service URLs, upstream error messages, or stack traces to wherever your log sink lives. In a high-compliance environment, log content should be sanitized. At minimum, be deliberate about what you're emitting.

---

### 🔵 Minor / Style Issues

**11. No rate limiting**

An open, unauthenticated endpoint with no rate limiting is trivially abusable for notification spam or resource exhaustion against the internal service.

**12. Router has no prefix or tags**

```python
router = APIRouter()
```

Should be:

```python
router = APIRouter(prefix="/notifications", tags=["Notifications"])
```

This improves OpenAPI documentation and avoids route collisions as the codebase grows.

**13. Test client is module-level state**

```python
app = FastAPI()
app.include_router(router)
client = TestClient(app)
```

This is fine for small test files but does not scale. Move app/client construction into a fixture to allow proper setup/teardown and avoid shared state between test runs.

## Decision: ❌ Request Changes

**Summary:** This PR introduces a BOLA/IDOR vulnerability, a missing authentication layer, and a blocking synchronous HTTP call inside an async route — any one of these alone would be sufficient grounds for rejection. The hardcoded service URL and lack of outbound authentication to the internal service are additional blockers that do not meet the security baseline expected in this environment. The test suite also has a critical gap where the error handling path is tested with the wrong exception type, giving false confidence in coverage. This PR should not merge until all Critical and Moderate items are resolved and re-reviewed.
