# PR Review: `feature/notification-endpoint`

---

**Decision: Request Changes**

**Summary:** This PR has a reasonable skeleton but introduces several meaningful security, reliability, and architectural concerns that need to be addressed before merge. The checklist claims are partially misleading — input validation exists but is insufficiently constrained, and the test suite has a critical gap that masks a real failure mode. This is not ready for a government contracting environment.

---

## 🔴 Critical Issues

### 1. No Authentication or Authorization on the Endpoint

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

There is **zero auth** on this endpoint. Any unauthenticated caller can trigger notifications to arbitrary users with arbitrary messages. In a checkout context, this is a significant abuse vector — an attacker could spam users, spoof order confirmations, or probe internal user/order ID formats.

**Required:** Add authentication dependency (e.g., OAuth2, API key validation, or internal service token). At minimum:

```python
from fastapi import Depends
from api.auth import require_internal_token  # your existing auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_token)
) -> NotificationResponse:
```

---

### 2. Unbounded `message` Field — Injection and Abuse Risk

```python
message: str
```

No length constraint, no content validation. This field is passed directly to an external service. This opens the door to:

- **Payload bloat / DoS** against the internal notification service
- **Content injection** if the downstream service renders HTML or markdown
- **Log injection** since the message could theoretically surface in logs

**Required:**

```python
from pydantic import BaseModel, Field, constr

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

The `user_id` and `order_id` fields should also be pattern-constrained. If these are UUIDs, enforce UUID format explicitly.

---

### 3. Synchronous `requests` in an Async Endpoint — Blocking the Event Loop

```python
async def send_notification(...):
    response = requests.post(...)  # BLOCKING CALL
```

This is a well-known footgun. `requests` is a synchronous library. Calling it inside an `async def` FastAPI route **blocks the entire event loop** for the duration of the HTTP call (up to 5 seconds per the timeout). Under any real load, this will degrade the entire service.

**Required:** Replace with `httpx` async client:

```python
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

## 🟠 Significant Issues

### 4. Hardcoded Internal Service URL

```python
"https://notifications.internal/send"
```

URLs should never be hardcoded. This makes environment promotion (dev → staging → prod) impossible without code changes, and it's a maintenance liability.

**Required:** Pull from config/environment:

```python
import os
NOTIFICATION_SERVICE_URL = os.environ.get("NOTIFICATION_SERVICE_URL")

if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL is not configured")
```

Prefer a centralized settings module (e.g., `pydantic-settings`) if one exists in this codebase.

---

### 5. Unsafe Key Access on External Response

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a 200 but omits `notification_id` — due to a schema change, partial failure, or malformed response — this raises an unhandled `KeyError` that will surface as a 500 with a potentially revealing traceback.

**Required:**

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

### 6. Exception Logging May Leak Sensitive Data

```python
logger.error("Notification service error: %s", e)
```

Depending on what `requests.RequestException` captures, this could log full request URLs, headers, or response bodies — including any tokens or PII. In a government contracting context, this is a compliance concern.

**Required:** Be explicit about what you log. Do not log the raw exception object without scrubbing it:

```python
logger.error("Notification service error for order %s: %s", request.order_id, type(e).__name__)
```

## 🟡 Minor Issues

### 7. Test Suite Has a Critical Gap — Wrong Exception Type Patched

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The handler catches `requests.RequestException`, not bare `Exception`. This test is patching the wrong exception type — it passes, but for the wrong reason. It's actually testing the FastAPI default 500 handler, not your custom 503 logic.

**Required:**

```python
from requests.exceptions import RequestException

with patch("api.notifications.requests.post",
           side_effect=RequestException("Connection refused")):
    response = client.post(...)

assert response.status_code == 503
```

---

### 8. No Rate Limiting

This endpoint is callable by anyone (see Issue #1) with no rate limiting. Even after auth is added, consider whether per-user or per-order rate limiting is appropriate to prevent notification flooding.

---

### 9. No Test for Input Validation Boundaries

The test suite has two happy-path-adjacent tests but nothing covering:
- Empty strings
- Oversized `message` payloads
- Malformed `user_id` / `order_id`
- Missing required fields

These should be added once field constraints (Issue #2) are implemented.
