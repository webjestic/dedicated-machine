# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Repo:** obotix.one internal pipeline

---

## Decision: Request Changes

## Summary

This PR introduces several security and architectural concerns that need to be addressed before merging. The endpoint accepts a free-form `message` field with no length or content constraints, uses a synchronous HTTP client inside an async function, and lacks authentication entirely — meaning any unauthenticated caller can trigger notifications against arbitrary user and order IDs. The test suite also has a coverage gap that masks a real failure path.

---

## Detailed Findings

### 🔴 Critical — Security

**1. No Authentication or Authorization on the Endpoint**

This is the most serious issue. The `/notify` endpoint is completely open. Any caller — internal or external — can POST to it with any `user_id` and `order_id`. There is no verification that the caller has the right to trigger a notification for that user.

```python
# Current — no auth whatsoever
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

At minimum, this needs an API key dependency or a JWT/OAuth2 bearer token check. Given our government contracting context, this is a non-starter without access control.

```python
# Expected pattern
from fastapi import Depends
from api.auth import require_internal_service_token  # or equivalent

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_service_token)
) -> NotificationResponse:
```

---

**2. Unvalidated `message` Field — Potential Injection / Abuse Vector**

The `message` field is a raw, unbounded string. It is passed directly into the downstream service payload with no sanitization, length limit, or content validation. This opens the door to payload injection into the notification service and potential downstream abuse (e.g., phishing content, oversized payloads).

```python
# Current — no constraints
message: str
```

```python
# Recommended
from pydantic import Field

message: str = Field(..., min_length=1, max_length=500, strip_whitespace=True)
```

Similarly, `user_id` and `order_id` should be constrained to expected formats, ideally with a regex pattern:

```python
from pydantic import Field
import re

user_id: str = Field(..., pattern=r"^user_[a-zA-Z0-9]{1,64}$")
order_id: str = Field(..., pattern=r"^order_[a-zA-Z0-9]{1,64}$")
```

---

**3. Sensitive Data Leakage in Logs**

The error log captures the full exception object `e`, which depending on the `requests` library internals can include the full URL, headers, and response body from the downstream service. In a government contracting environment, that is a compliance risk.

```python
# Current — may log sensitive upstream details
logger.error("Notification service error: %s", e)
```

```python
# Safer — log a controlled message, keep detail out of the log stream
logger.error("Notification service error for order_id=%s: %s", request.order_id, type(e).__name__)
```

If full exception detail is needed for debugging, it should go to a secured, access-controlled log sink — not a general application log.

---

### 🟠 High — Architecture

**4. Synchronous `requests` Inside an `async` Function**

This is a classic async footgun. `requests.post(...)` is a blocking call. Inside an `async def` FastAPI route, this will block the entire event loop for the duration of the HTTP call — up to the 5-second timeout. Under any meaningful load, this will degrade the entire service.

```python
# Current — blocks the event loop
response = requests.post("https://notifications.internal/send", ...)
```

Replace with `httpx` which has a native async client:

```python
import httpx

# In the route
async with httpx.AsyncClient(timeout=5.0) as client:
    response = await client.post(
        "https://notifications.internal/send",
        json={...}
    )
    response.raise_for_status()
```

The `httpx` client should ideally be instantiated once at app startup and injected as a dependency rather than created per-request, but that's a secondary concern to fixing the blocking call first.

---

**5. Hardcoded Internal Service URL**

The notification service URL is hardcoded as a string literal in the route handler. This is an operational and security concern — it cannot be rotated, environment-scoped, or audited without a code change.

```python
# Current — hardcoded
"https://notifications.internal/send"
```

```python
# Expected — pull from config
from api.config import settings

settings.NOTIFICATION_SERVICE_URL  # loaded from env / secrets manager
```

This also makes it impossible to point at a staging service in non-production environments without modifying source code.

---

**6. No Retry or Circuit Breaker Logic**

The endpoint makes a single attempt to the downstream service and immediately returns a 503 on any failure. In a checkout flow, a transient network hiccup will surface directly to the user. At minimum, a simple retry with exponential backoff should be considered. A circuit breaker pattern (e.g., via `tenacity` or a service mesh policy) would be appropriate for a production checkout path.

---

### 🟡 Medium — Test Quality

**7. Test Patches Wrong Exception Type**

The failure test patches `requests.post` to raise a bare `Exception`. However, the production handler only catches `requests.RequestException`. A bare `Exception` would propagate uncaught in production, resulting in a 500 rather than a 503. The test passes because `TestClient` catches unhandled exceptions differently in test mode — it masks the real behavior.

```python
# Current test — masks the real failure path
side_effect=Exception("Connection refused")
```

```python
# Correct — test what production actually catches
import requests as req_lib
side_effect=req_lib.exceptions.ConnectionError("Connection refused")
```

This is a test that gives false confidence. It should be corrected.

---

**8. No Test for Input Validation**

The checklist claims input validation is implemented, but there are zero tests exercising invalid inputs — empty strings, oversized payloads, malformed IDs. If validation constraints are added (as recommended above), tests must cover rejection of invalid inputs.

---

**9. No Test for Malformed Downstream Response**

If the notification service returns a 200 but with a body that is missing `notification_id`, the current code will raise an unhandled `KeyError`. There is no test for this case and no defensive handling in the implementation.

```python
# Current — will KeyError if notification_id is absent
notification_id=data["notification_id"]
```

```python
# Safer
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned no notification_id for order_id=%s", request.order_id)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

## Required Before Merge

| # | Severity | Item |
|---|----------|------|
| 1 | 🔴 Critical | Add authentication/authorization to the endpoint |
| 2 | 🔴 Critical | Constrain and validate `message`, `user_id`, `order_id` fields |
| 3 | 🔴 Critical | Audit and sanitize what is written to logs |
| 4 | 🟠 High | Replace `requests` with `httpx` async client |
| 5 | 🟠 High | Move service URL to config/environment variable |
| 6 | 🟡 Medium | Fix test to catch correct exception type |
| 7 | 🟡 Medium | Add tests for invalid input, missing response fields |
| 8 | 🟡 Medium | Handle missing `notification_id` in downstream response defensively |

Happy to pair on the auth dependency pattern or the `httpx` client lifecycle if that would help move this forward. The bones of the implementation are reasonable — these are fixable issues, but several of them are blockers given our compliance posture.
