# PR Review: `feature/notification-endpoint`

**Reviewer:** Senior Backend Engineer
**Date:** 2025-07-11

---

## Detailed Findings

### 🔴 Critical Issues

**1. Synchronous HTTP call inside an async endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, will starve other coroutines
```

Using `requests` (a synchronous library) inside an `async` def is a serious architectural mistake in FastAPI. This blocks the entire event loop for the duration of the network call, degrading performance under any meaningful load. The fix is to use `httpx.AsyncClient` or `aiohttp`.

```python
# Recommended
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

---

**2. No authentication or authorization on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

There is zero access control here. Any unauthenticated caller — internal or external — can trigger notifications for **any** `user_id` and `order_id` combination. In a government contracting context this is a non-starter. At minimum this needs a dependency-injected auth check.

```python
from fastapi import Depends
from api.auth import require_authenticated_service  # your internal auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_authenticated_service),
) -> NotificationResponse:
```

---

**3. Hardcoded internal service URL**

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

Hardcoding service URLs is a deployment and security risk. This cannot be rotated, environment-scoped, or audited without a code change. It also leaks internal topology into source history.

```python
# Recommended
import os
from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

### 🟡 Moderate Issues

**4. `message` field is unbounded and unvalidated**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no constraints whatsoever
```

An unbounded `message` field passed directly into an outbound HTTP payload is a vector for prompt injection, payload stuffing, or abuse of the downstream notification service. Apply explicit constraints.

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

---

**5. External response is trusted and unparsed**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is absent
)
```

If the external service returns a malformed or unexpected payload, this raises an unhandled `KeyError` that will bubble up as a raw 500. The external service response should be validated before use.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**6. Logging potentially sensitive data**

```python
logger.error("Notification service error: %s", e)
```

Depending on what the `requests.RequestException` captures in its string representation, this could log internal URLs, headers, or partial payloads to whatever log sink is configured. In a cleared/government environment, log destinations and content must be carefully controlled.

```python
# Be explicit about what you log
logger.error(
    "Notification service error for order_id=%s: %s",
    request.order_id,
    type(e).__name__,  # log the exception type, not the full repr
)
```

### 🔵 Minor / Test Quality Issues

**7. Test catches `Exception` instead of `requests.RequestException`**

```python
# In test
side_effect=Exception("Connection refused")

# In production code
except requests.RequestException as e:
```

The test patches with a bare `Exception`, but the handler only catches `requests.RequestException`. This test is **not actually validating the error handling path** — it passes by accident because FastAPI catches unhandled exceptions and returns 500, but the test asserts 503. Once you migrate to `httpx`, this gap will become a real failure. Tests should be precise about what they simulate.

```python
import httpx

side_effect=httpx.RequestError("Connection refused", request=MagicMock())
```

---

**8. No rate limiting**

An unauthenticated, externally reachable notification endpoint with no rate limiting is trivially abusable for notification spam or service exhaustion. Even a simple `slowapi` integration would be a meaningful baseline.

**Decision:** ❌ Request Changes

**Summary:** This PR introduces three critical issues that individually would block approval — a synchronous HTTP call inside an async handler that will degrade performance under load, a completely unauthenticated endpoint that allows arbitrary notification triggering, and a hardcoded internal service URL that violates environment separation and auditability. The moderate findings around input validation and unsafe external response parsing compound the risk profile. Resolve all critical and moderate items before resubmission; the minor test issue must also be corrected as the current test suite does not accurately validate the error handling contract.
