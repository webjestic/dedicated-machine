Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-and-extend sequence, a class-level state management flaw that makes the service unsafe for concurrent use, and a zombie-write vulnerability that the lock mechanism does not actually prevent. The test suite mocks away the heartbeat entirely and does not exercise the most dangerous failure modes. None of these are cosmetic — they are correctness and safety defects in a service that will run under high-concurrency flash sale load.

---

## Detailed Review

### 1. Critical: Heartbeat Race Condition (TOCTOU on Token Check)

```python
# _start_heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are two separate commands. Between them, the lock can expire and be acquired by another process. You then extend a lock you no longer own, silently handing a longer lease to whoever now holds it — or worse, extending a key that was re-created by a different owner with a different token if the timing is extremely tight.

**Required fix:** Replace this with a Lua script that atomically checks ownership and extends in a single round-trip:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the release in `_acquire_lock`'s `finally` block — it has the identical TOCTOU flaw between `GET` and `DELETE`. Use the canonical Lua release script here too.

### 2. Critical: Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables on a shared `InventoryService` object. If two requests call `reserve_item` concurrently on the same service instance — which is the entire point of a flash sale scenario — they will clobber each other's `_heartbeat_active` flag. Thread A calling `_stop_heartbeat()` sets `_heartbeat_active = False`, which also kills Thread B's heartbeat. Thread B's lock then expires mid-operation.

**Required fix:** Heartbeat state must be scoped to the lock acquisition context, not the service instance. Pass a mutable container (e.g., a `threading.Event`) into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into heartbeat thread, not self._heartbeat_active
```

### 3. Architectural: Zombie-Write Vulnerability Is Not Mitigated

The PR description claims distributed locking protects inventory writes. It does not, and the heartbeat makes this worse, not better.

Consider this sequence:

1. Process A holds the lock. It calls `db.get_stock()` and `db.update_stock()`.
2. Process A experiences a GC pause or OS scheduling delay *after* the lock expires but *before* `update_stock` commits.
3. Process B acquires the lock, reads stock, begins its own write.
4. Process A resumes and its write lands — a zombie write from a process that no longer holds the lock.

The heartbeat reduces the probability of this but does not eliminate it. Under the Kleppmann critique of Redlock, the only correct mitigation is a **fencing token** — a monotonically increasing value issued at lock acquisition time, passed to the storage layer, and rejected by the storage layer if a higher token has already been seen.

**Required fix:** The storage layer (`db.update_stock`, `db.create_order`) must accept and enforce a fencing token. If the DB layer cannot be modified to support this, document the residual risk explicitly and obtain sign-off, because this service cannot provide the safety guarantee it implies.

### 4. Moderate: Single Redis Node Is a Single Point of Failure

```python
acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
```

This is a single-node Redis lock. If the Redis primary fails after granting the lock but before the replica replicates the key, a failover promotes the replica, which has no record of the lock. A second process acquires the same lock. You now have two lock holders simultaneously — exactly the split-brain scenario this lock is supposed to prevent.

Redlock across N independent nodes is the standard mitigation, though it carries its own known failure modes under clock drift and network partition. At minimum, the architecture decision here needs to be documented. If this is a single-node deployment, the service must not claim mutual exclusion guarantees under failover.

---

### 5. Moderate: `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` will pass the stock check (`current_stock < 0` is false), decrement nothing, and create a zero-quantity order. `quantity=-1` will *increase* stock. The Pydantic model is on `ReservationResult`, not on the input. Add a validated input model:

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = Field(ge=1, le=1000)  # enforce reasonable bounds
```

---

### 6. Minor: Test Suite Mocks Away the Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally significant and most dangerous part of this PR. Mocking it out means the tests provide no coverage of the code path that actually runs in production. Add integration-level tests that let the heartbeat thread run against a real or fake-time Redis mock and verify that TTL extension and stop behavior work correctly under concurrent acquisition.
