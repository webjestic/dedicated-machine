Decision: Request Changes

Summary: The implementation contains several serious architectural flaws that make it unsafe for production use in a high-concurrency flash sale context. The most critical issue is a race condition in the heartbeat's check-then-extend sequence, combined with a class-level heartbeat state design that makes the service non-reentrant and unsafe under concurrent use. Additionally, the lock release in the `finally` block contains a non-atomic check-and-delete that creates a zombie-write window, and the DB operations inside the lock boundary have no transactional guarantees, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory state.

## Detailed Review

### 🔴 Critical — Must Fix Before Merge

---

#### 1. Non-Atomic Lock Release (Zombie-Write / TOCTOU)

**File:** `services/inventory.py`, lines in `_acquire_lock` `finally` block

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic time-of-check/time-of-use (TOCTOU) race. Between the `GET` and the `DELETE`, the lock TTL can expire and be acquired by another process. You then delete a lock you no longer own, evicting a legitimate holder. This is precisely the zombie-write scenario described in Kleppmann's critique of naive Redis locking.

The fix is a well-known Lua script that makes the check-and-delete atomic at the Redis server:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In _acquire_lock finally block:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

#### 2. Heartbeat Race Condition — Non-Atomic Check-Then-Extend

**File:** `services/inventory.py`, `_beat` inner function

```python
# CURRENT — two separate round-trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by a different process. You then extend a lock you no longer own. This silently extends a competitor's lock, or worse, extends a lock that has already been legitimately re-acquired with a new token.

Fix this with the same Lua atomicity pattern:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In _beat:
result = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not result:
    logger.warning("Heartbeat: lock ownership lost key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The heartbeat should also stop itself and surface the ownership loss rather than silently continuing.

#### 3. Class-Level Heartbeat State — Service Is Not Reentrant

**File:** `services/inventory.py`, `__init__` and `_start_heartbeat`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not scoped to a single lock acquisition. If two calls to `reserve_item` overlap on the same `InventoryService` instance (which is the normal case with a shared service object or a thread pool), the second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa. The first call's lock will then silently expire mid-operation.

The fix is to scope heartbeat state to the lock context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
            if not result:
                logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return

    # ... acquire lock ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

Note also that the current `_stop_heartbeat` does not `join()` the thread. The heartbeat thread can execute one final extension after the lock has been released, extending a lock that now belongs to another process.

#### 4. No Transactional Guarantee Between DB Writes

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping them. If the process crashes, is OOM-killed, or the DB connection drops between these two lines, stock is decremented but no order record exists. This is a silent inventory leak. The distributed lock does not protect against this — it only serializes access, it does not provide atomicity.

The fix requires the DB layer to expose a transaction boundary:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support this, that is a separate architectural gap that needs to be resolved before this service can be considered safe.

#### 5. Missing Input Validation on Public Entry Point

**File:** `services/inventory.py`, `reserve_item`

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`item_id` is used directly in a Redis key (`inventory:lock:{item_id}`) with no sanitization. A caller passing `item_id = "foo:bar"` or `item_id = ""` will produce malformed or colliding lock keys. The checklist claims Pydantic validation is present, but `ReservationResult` is an output model. There is no input validation model.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)
```

---

#### 6. Heartbeat Interval vs. TTL Ratio Is Unsafe

`LOCK_TTL = 30`, `HEARTBEAT_INTERVAL = 10`. The heartbeat fires at 10s, but the TTL is 30s. This means the lock will be extended from ~20s remaining back to 30s — acceptable in the happy path. However, if the heartbeat thread is delayed by GIL contention, GC pause, or OS scheduling (all realistic in a Python service under load), the actual extension could fire at 15s or later, leaving only 15s of runway. Under a slow DB operation, this is a real expiry risk.

A safer ratio is TTL = 3× heartbeat interval at minimum, or better, TTL = 5× to absorb scheduling jitter. Consider `LOCK_TTL = 60`, `HEARTBEAT_INTERVAL = 10`.

---

#### 7. `time.sleep` in Lock Retry Blocks the Calling Thread

**File:** `services/inventory.py`, `_acquire_lock`

```python
time.sleep(backoff)
```

In a synchronous WSGI/threaded server this is acceptable but worth flagging. In any async context (FastAPI with `asyncio`, for example), this will block the event loop. If this service is ever adapted for async use, this becomes a serious throughput issue. At minimum, document the synchronous-only constraint, or implement an async variant with `asyncio.sleep`.

#### 8. Tests Mock Out the Exact Code Paths Being Tested

**File:** `tests/test_inventory.py`, fixture

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most operationally risky part of this implementation, and the tests entirely bypass it. There are no tests for:

- Heartbeat correctly extending TTL while a slow DB operation runs
- Heartbeat stopping cleanly after lock release
- Heartbeat detecting ownership loss and stopping itself
- Concurrent calls to `reserve_item` not cross-contaminating heartbeat state

The `test_lock_not_released_if_expired` test is also misleading — it tests that `delete` is not called when the token mismatches, but the real fix (Lua atomic release) makes this test irrelevant. The test should be updated to verify the Lua script is invoked.
