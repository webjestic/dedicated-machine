Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's token-check/extend sequence that creates a zombie-write window, and the `InventoryService` instance is structured as a singleton with shared mutable state, making it unsafe under concurrent request load. Additionally, the database operations lack atomicity guarantees at the storage layer, meaning a process crash between `update_stock` and `create_order` can produce silent inventory corruption that the lock cannot protect against.

---

## Detailed Review

### 🔴 Critical — Shared Mutable Heartbeat State (Concurrency Correctness)

```python
# In InventoryService.__init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped locals. If this service is instantiated once and shared across request handlers (which is the standard pattern for dependency injection in Flask, FastAPI, etc.), two concurrent calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat, potentially letting that lock expire mid-operation. This is a textbook shared-state concurrency bug.

**Fix:** Heartbeat state must be scoped to the lock acquisition call, not the instance. Use a local `threading.Event` passed into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=2)
        # release logic...
```

### 🔴 Critical — Heartbeat Check-Then-Extend Is a Race (Zombie Write Window)

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)          # (1) read token
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps `(1)` and `(2)`, the lock can expire and be acquired by a different process. You then extend the TTL of a lock you no longer own. This is precisely the zombie-write scenario Kleppmann describes: your process believes it holds the lock, continues writing, and a second legitimate lock holder is also writing. The token check does not make this atomic.

**Fix:** Replace the two-step read/extend with a Lua script that makes the check-and-extend atomic at the Redis layer:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _extend_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(
        EXTEND_SCRIPT, 1, lock_key, token, str(self.LOCK_TTL)
    )
    return result == 1
```

The same issue exists in the release path:

```python
# Also non-atomic:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This must also be a Lua script (this is the standard safe-release pattern and is well-documented in the Redis docs):

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
```

### 🔴 Critical — No Storage-Layer Atomicity Between `update_stock` and `create_order`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

A process crash, OOM kill, or unhandled exception between these two calls leaves the system in a state where stock has been decremented but no order record exists. The distributed lock provides mutual exclusion, but it provides **zero durability guarantees**. This is a silent inventory leak.

**Fix:** These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architectural gap that must be resolved before this service ships. Consider also whether an outbox pattern or idempotency key is appropriate here given the flash sale context.

### 🟡 Moderate — Single-Node Redis Is Not a Distributed Lock

The PR description says "distributed lock" but the implementation uses a single `redis.Redis` client. A single Redis node is a **single point of failure**. If Redis goes down or becomes partitioned, all lock acquisitions fail and the service returns 503 to every user. Under a network partition, a client may believe it holds a lock that has already expired and been re-acquired.

If this is a government-facing flash sale system, the failure mode matters. Either:

1. **Document the single-node assumption explicitly** and accept the SPOF tradeoff, or
2. **Use Redis Sentinel or Redis Cluster** for HA, or
3. **Implement Redlock across N ≥ 3 independent Redis nodes** if you need partition tolerance — but note that Kleppmann's critique of Redlock is well-founded: it does not provide safety under clock drift or GC pauses without fencing tokens at the storage layer.

The honest answer for most deployments is option 1 with a documented SLA, combined with the Lua-atomic operations above.

### 🟡 Moderate — `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no bounds check. A caller passing `quantity=0` or `quantity=-5` will pass the stock check (`current_stock >= 0`) and either do nothing or **increase** stock. The Pydantic model is on `ReservationResult`, not on the input.

```python
if quantity <= 0:
    return ReservationResult(
        success=False,
        message="Quantity must be a positive integer"
    )
```

Consider also an upper bound to prevent a single caller from reserving the entire flash sale inventory in one call.

---

### 🟡 Moderate — Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread may still be mid-execution (inside `time.sleep`) when the lock is released and deleted. In the worst case it wakes up, passes the token check (against a now-deleted or re-acquired key), and extends a lock owned by a different process. The `join()` call in the fix above addresses this.

### 🟡 Moderate — Tests Mock Away the Exact Code Under Scrutiny

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← mocked out
    svc._stop_heartbeat = MagicMock()    # ← mocked out
    return svc
```

The heartbeat is the most operationally risky component of this PR and the tests completely bypass it. `test_lock_not_released_if_expired` is the right idea but it also uses the fixture that mocks the heartbeat. There are no tests for:

- Concurrent calls on the same instance racing on `_heartbeat_active`
- The heartbeat extending a lock it no longer owns
- The non-atomic release path being interleaved with another acquisition
- Transaction rollback on `create_order` failure

At minimum, add an integration-style test that exercises the real heartbeat thread with a short TTL and verifies the lock is not extended after release.

### 🟢 Acknowledged as Good

- `nx=True, ex=self.LOCK_TTL` on `SET` is correct — atomic acquire-or-fail with TTL
- Exponential backoff on contention is appropriate
- Token-based ownership check is the right concept (implementation needs to be made atomic as noted above)
- Structured logging with contextual fields is clean
- `ReservationResult` as a typed return value is good API hygiene
- Daemon thread prevents process hang on unclean shutdown
