Decision: Request Changes

Summary: The heartbeat renewal pattern here is the core architectural problem — it defeats the purpose of a bounded TTL and reintroduces exactly the zombie-write failure mode the TTL was meant to prevent. Beyond that, there are several concurrency and correctness issues that make this unsafe for a flash sale workload where inventory accuracy is a hard requirement.

---

## Critical Issues

### 1. Heartbeat Renewal Undermines the Safety Guarantee of TTL

This is the fundamental design flaw. The entire point of a bounded TTL on a distributed lock is that the system can make progress after a process stalls — GC pause, network partition, thread scheduling delay. The moment you add a heartbeat that extends the TTL, you have converted a bounded lease into an unbounded one.

The failure scenario this enables:

```
T=0   Process A acquires lock, heartbeat starts
T=5   Process A stalls (GC pause, OS scheduling, slow DB call)
T=10  Heartbeat thread fires, extends TTL — lock is now held for another 30s
T=15  Process A resumes, reads stale stock value from before the stall
T=16  Process A writes decrement based on stale read
T=17  Stock is now negative or double-decremented
```

The heartbeat thread and the critical section run in different threads with no synchronization. The heartbeat can extend the TTL after the application thread has already observed a stale state. This is a classic zombie-write setup.

If operations genuinely need more than 30 seconds, the right answer is to increase the TTL to a defensible bound, not to extend it indefinitely from a background thread. If you cannot bound the operation time, the lock model is wrong for this workload.

### 2. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not per-invocation state. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of a flash sale scenario — two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. One call's `_stop_heartbeat()` will kill the other call's heartbeat, and the second call's `_start_heartbeat()` will overwrite the thread reference, leaking the first thread entirely.

The heartbeat state must be scoped to the lock acquisition context, not the service instance. A simple fix is a local `Event` object passed into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    def _beat():
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        stop_event.set()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 3. Lock Release Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a check-then-act across two separate Redis commands. Between the `GET` and the `DELETE`, another process can acquire the lock. You would then delete a lock you do not own, causing a reservation in progress by another process to run without a lock.

This must be a single atomic operation. The standard pattern is a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

The same non-atomic pattern exists in the heartbeat:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This also needs to be a Lua script or replaced with a conditional `SET ... XX EX` pattern.

### 4. No Fencing Token at the Storage Layer

The Redis lock provides mutual exclusion only as long as Redis is healthy and the TTL is respected. The database operations — `update_stock` and `create_order` — have no knowledge of the lock. If the lock expires (or is extended past a stale read), the DB will accept the write without any way to detect it is out of order.

For a flash sale inventory system, the DB layer needs its own protection. At minimum:

- Optimistic concurrency on `update_stock` using a version column or CAS semantics: `UPDATE inventory SET stock = %s, version = version + 1 WHERE item_id = %s AND version = %s AND stock >= %s`
- The application should verify the update affected exactly one row and retry or abort otherwise

This is the defense-in-depth layer that makes the system correct even when the lock fails.

## Moderate Issues

### 5. `quantity` Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `quantity > 0`. A caller passing `quantity=0` succeeds and creates an order for zero units. A caller passing `quantity=-5` would increase stock. The Pydantic model is on the result, not the input. Add explicit validation:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

### 6. `item_id` Is Not Validated Before Use in a Lock Key

`item_id` is interpolated directly into `f"inventory:lock:{item_id}"`. If `item_id` contains a colon or Redis-significant characters, this could produce unexpected key collisions. Validate that `item_id` matches an expected format (alphanumeric, bounded length) before use.

### 7. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to observe it. The thread sleeps for `HEARTBEAT_INTERVAL` (10 seconds) before checking the flag. During that window after `_stop_heartbeat()` returns, the thread can still fire and extend the TTL on a lock that the application believes it has released. The `finally` block deletes the key, but then the heartbeat thread wakes up and calls `expire` on a key that may now belong to a different process. Add `thread.join()` in the stop path.

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Out Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test stubs out the heartbeat. This means the concurrency behavior — the primary risk surface — is completely untested. There should be at least one integration-level test that exercises the actual heartbeat thread, verifies it extends the TTL, and verifies it stops cleanly on lock release.

### 9. Non-Atomic Release Is Not Tested

`test_lock_not_released_if_expired` sets `redis_client.get.return_value = b"different-token"` and asserts `delete` is not called. This test passes today but will still pass after introducing the Lua script fix — the mock needs to be updated to reflect atomic semantics, and the test should verify the script is called with the correct arguments.

### 10. Concurrent Access Is Not Tested

There is no test that instantiates one `InventoryService` and calls `reserve_item` from two threads simultaneously. Given that instance-level heartbeat state is the critical bug, this test is essential.
