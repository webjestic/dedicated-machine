Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a government-facing production system.

---

## Critical Issues

### 1. Race Condition in Heartbeat Token Check (TOCTOU)

The heartbeat performs a `GET` then `EXPIRE` as two separate commands. Between those two operations, the lock can expire and be acquired by another process, which then gets its TTL silently extended by *your* heartbeat thread.

```python
# UNSAFE — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # may extend a different owner's lock
```

**Fix:** Use a Lua script to atomically check ownership and extend TTL.

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._heartbeat_sha = self.redis.script_load(HEARTBEAT_SCRIPT)

# In heartbeat:
self.redis.evalsha(self._heartbeat_sha, 1, lock_key, token, self.LOCK_TTL)
```

---

### 2. Race Condition in Lock Release (TOCTOU)

Same class of bug. `GET` then `DELETE` is not atomic.

```python
# UNSAFE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another owner may have acquired by now
```

**Fix:** Atomic Lua script, which is the industry-standard pattern for this.

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

---

### 3. Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is a shared singleton (the typical pattern), two concurrent `reserve_item` calls will clobber each other's heartbeat state. The second call's `_stop_heartbeat()` will kill the first call's thread.

```python
# Both callers share this — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Make heartbeat state local to each lock acquisition, passed via closure or a small dataclass.

```python
@dataclass
class _HeartbeatHandle:
    active: bool = True

def _start_heartbeat(self, lock_key: str, token: str) -> _HeartbeatHandle:
    handle = _HeartbeatHandle()

    def _beat() -> None:
        while handle.active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if handle.active:
                self.redis.evalsha(self._heartbeat_sha, 1, lock_key, token, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return handle
```

### 4. No Input Validation on `reserve_item`

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string, all of which will silently produce incorrect behavior (negative stock, spurious orders).

```python
# Nothing stops this:
service.reserve_item("", quantity=-100)
```

**Fix:** Validate at the entry point.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item ID")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

Or define a `ReservationRequest` Pydantic model and validate through it.

---

### 5. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate DB calls with no transaction boundary. If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost with no compensating action.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # crash here = lost inventory
```

**Fix:** Wrap in a transaction or use a unit-of-work pattern. At minimum, catch the failure and attempt to restore stock, and alert loudly.

```python
try:
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
except Exception as e:
    logger.critical(
        "Order creation failed after stock decrement — manual reconciliation required "
        "item_id=%s order_id=%s quantity=%d error=%s",
        item_id, order_id, quantity, e
    )
    raise  # do not silently swallow
```

## Moderate Issues

### 6. Bare `Exception` From DB Is Silently Swallowed

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` propagates uncaught to the caller with no structured response and no log entry at the `reserve_item` level.

**Fix:** Add a broad catch with logging that returns a structured failure result.

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Internal error during reservation")
```

---

### 7. Heartbeat Thread Is Never Joined

`_stop_heartbeat` sets the flag but does not join the thread. The thread may still be alive and holding a reference to `lock_key` and `token` when the lock has already been released, creating a window where it could attempt to extend a lock that no longer belongs to this process.

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

With the corrected per-invocation handle design from issue #3, this becomes straightforward.

---

### 8. Test Coverage Gaps

- No test exercises the actual heartbeat thread behavior.
- No test covers the `db.update_stock` succeeding but `db.create_order` raising (the partial-failure scenario from issue #5).
- `test_lock_not_released_if_expired` mocks `_start_heartbeat` and `_stop_heartbeat` on the service fixture but then creates a *new* `svc` instance without those mocks — the test is inconsistent and may not be exercising what it claims.
- No test validates rejection of invalid inputs (`quantity=0`, empty `item_id`).

---

## Minor Issues

- `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10` are class-level constants but should be constructor parameters to allow environment-specific tuning without subclassing.
- Log message `"Lock released key=%s"` fires before confirming the `DELETE` succeeded. Log after confirming.
- The `token` yielded from `_acquire_lock` is not used by `reserve_item`. Yielding it creates a misleading API surface. Yield `None` or restructure.
