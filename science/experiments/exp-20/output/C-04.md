Decision: Request Changes

Summary: The core locking logic contains a critical race condition in the heartbeat's token check — it performs a non-atomic GET + EXPIRE, meaning another process could steal the lock between those two operations and have its TTL silently extended by this instance's heartbeat thread. More broadly, the `InventoryService` class carries instance-level heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) that makes it fundamentally unsafe for concurrent use from multiple threads or coroutines sharing the same service instance, which is the exact scenario this code is designed for. The test suite mocks away the heartbeat entirely and does not exercise the actual Redis Lua path, so the most dangerous behaviors are untested.

---

## Detailed Findings

### 🔴 Critical — Race Condition in Heartbeat Token Verification

```python
# services/inventory.py — _start_heartbeat(), lines ~40–44
current = self.redis.get(lock_key)          # ← non-atomic
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← separate command
```

Between `GET` and `EXPIRE`, another process can acquire the lock with a new token. This heartbeat will then extend *that process's* lock, not its own. The canonical fix is a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def __init__(self, redis_client, db):
    ...
    self._extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

def _beat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        extended = self._extend_script(
            keys=[lock_key], args=[token, self.LOCK_TTL]
        )
        if not extended:
            logger.warning("Heartbeat: lock lost or expired key=%s", lock_key)
            break
```

The same atomic-script requirement applies to lock release in `_acquire_lock`.

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
# services/inventory.py
self._heartbeat_active = False          # shared across all callers
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (standard in any web framework with a shared service layer), one thread's `_stop_heartbeat()` will cancel the other thread's heartbeat. This silently drops lock protection mid-operation with no error raised.

The fix is to make heartbeat state local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 2.0)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat = threading.Thread(
        target=self._beat,
        args=(lock_key, token, stop_event),
        daemon=True,
    )
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release via Lua
        self._release_script(keys=[lock_key], args=[token])
```

### 🔴 Critical — Lock Release Is Also Non-Atomic

```python
# services/inventory.py — _acquire_lock() finally block
current = self.redis.get(lock_key)          # ← non-atomic
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← separate command
```

Same class of race as the heartbeat. Another process can acquire the lock after the `GET` and before the `DELETE`, causing this code to delete a lock it no longer owns. Use a Lua script here as well:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 🟡 High — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` and the code will silently proceed, potentially creating a zero-quantity order or corrupting stock.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        return ReservationResult(success=False, message=f"Invalid input: {e}")
```

### 🟡 High — Database Operations Are Outside Any Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists — inventory is permanently lost with no audit trail. This needs to be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 🟡 Medium — Backoff Has No Cap

```python
backoff = (2 ** attempt) * 0.1
```

With `retry_attempts` left open to callers (it is a parameter), a high value produces unbounded sleep. Cap it:

```python
backoff = min((2 ** attempt) * 0.1, 2.0)
```

---

### 🟡 Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The daemon thread may still be running when the `finally` block completes. Under high concurrency this leaks threads. Call `self._heartbeat_thread.join(timeout=...)` after setting the stop signal.

### 🟠 Low — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()
    return svc
```

The test for `test_lock_not_released_if_expired` asserts `delete` is not called, but the mock token `b"test-token"` will never match the UUID generated inside `_acquire_lock`, so the assertion passes for the wrong reason. Integration tests against a real Redis instance (e.g., `fakeredis`) are needed to validate the actual locking semantics.
