Decision: Request Changes

Summary: The heartbeat renewal pattern here is architecturally unsound for a system handling flash sale inventory — it defeats the core safety guarantee that lock TTLs are supposed to provide, and the implementation has a race condition in the release path that makes zombie writes possible. The instance-level heartbeat state also makes this service non-reentrant and unsafe under any concurrent call pattern. These are not style issues; they are correctness failures in the critical path.

---

## Detailed Review

### 1. The Heartbeat Pattern Is the Wrong Tool Here

This is the central architectural problem. The PR treats lock expiry as an operational inconvenience to be engineered around. It is not. TTL expiry is the *safety mechanism*.

Kleppmann's critique of Redlock is directly applicable here. Consider this sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=5   Process A enters a GC pause / slow DB call
T=10  Heartbeat fires, checks token, extends TTL — lock stays alive
T=15  Process A resumes, reads stale stock value from DB
T=20  Process A writes decrement based on stale read
      Meanwhile, no other process could intervene because heartbeat kept the lock alive
```

The heartbeat does not make the critical section safer. It makes the window for a stale-read-then-write *longer* and harder to bound. If `db.get_stock` or `db.update_stock` blocks for 45 seconds, the heartbeat will keep renewing indefinitely. You have traded a bounded failure window for an unbounded one.

The correct architecture for this use case is one of:

- **Optimistic locking with a version counter** at the DB layer (`UPDATE inventory SET stock = stock - ? WHERE item_id = ? AND stock >= ? AND version = ?`), with retry on version conflict
- **A fencing token passed through to storage** so the DB can reject writes from a process whose lock has logically expired, even if the process doesn't know it yet
- **Lua scripting the entire read-modify-write in Redis** if Redis is the authoritative store, making the operation atomic without needing a separate lock at all

A heartbeat-extended Redis lock with a separate DB write gives you the operational complexity of distributed locking with none of the safety guarantees.

### 2. Race Condition in Lock Release (TOCTOU)

```python
# In _acquire_lock finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic check-then-act race. Between `GET` and `DELETE`, another process can acquire the lock. You then delete a lock you no longer own. This is a well-known failure mode and the standard fix is a Lua script that makes the comparison and deletion atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

The test `test_lock_not_released_if_expired` claims to verify this behavior, but it only checks the happy path of the race — it does not and cannot test the window between `GET` and `DELETE` where the race actually occurs. The test is giving false confidence.

### 3. Instance-Level Heartbeat State Is Not Reentrant

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal case in any web framework with a shared service layer — two concurrent calls will stomp on each other's `_heartbeat_active` flag. Call A stopping its heartbeat will also stop Call B's heartbeat, causing Call B's lock to expire unexpectedly.

The heartbeat state must be scoped to the lock acquisition, not the service instance. At minimum it needs to be local to `_acquire_lock`. In practice this is another reason the heartbeat approach should be abandoned entirely.

---

### 4. No Input Validation on `quantity`

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `int` parameters with no validation. A caller can pass `quantity=0` or `quantity=-5`. With `quantity=-5`:

```python
current_stock < quantity  # 10 < -5 → False, passes the check
new_stock = current_stock - quantity  # 10 - (-5) = 15
self.db.update_stock(item_id, 15)  # stock increases
```

This is a data integrity bug. Negative quantity reservations silently increase stock. Add a guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

### 5. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record. The lock is held during both calls, which prevents concurrent access, but it does not make the two DB operations atomic with respect to process failure. These need to be wrapped in a database transaction. The current code has no exception handling around the DB calls inside the lock boundary.

---

### 6. The `_stop_heartbeat` / `_heartbeat_thread.join()` Gap

`_stop_heartbeat` sets `_heartbeat_active = False` but does not join the thread. The heartbeat thread sleeps for 10 seconds before checking the flag. After `_stop_heartbeat` returns, the heartbeat thread may still be alive and may fire one more `redis.expire` call after the lock has already been deleted. This extends a key that may now be owned by a different process. Add `self._heartbeat_thread.join(timeout=...)` in `_stop_heartbeat`, or use a `threading.Event` for immediate wakeup.

---

### Summary of Required Changes

| Issue | Severity | Required Action |
|---|---|---|
| Heartbeat pattern defeats TTL safety guarantees | Critical | Replace with optimistic locking or atomic Lua script |
| TOCTOU race in lock release | High | Replace GET+DELETE with atomic Lua script |
| Instance-level heartbeat state, non-reentrant | High | Scope state to call, or remove heartbeat entirely |
| No validation on `quantity` parameter | Medium | Add positive integer guard before lock acquisition |
| Non-atomic DB operations inside lock | Medium | Wrap in database transaction with exception handling |
| Heartbeat thread not joined on stop | Low | Use `threading.Event` and join on stop |
