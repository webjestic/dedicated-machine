Decision: Request Changes

Summary: The core locking logic has a meaningful concurrency flaw — the heartbeat's get-then-expire sequence and the lock release's get-then-delete sequence are both non-atomic, creating race windows where the lock can be stolen or released incorrectly under real distributed load. Beyond the atomicity issues, the `InventoryService` instance is stateful in a way that makes it unsafe for concurrent use, and the test suite mocks away the heartbeat entirely, meaning the most critical operational behavior is never actually tested. These issues are significant enough to block merge given the high-contention flash sale context this code is explicitly designed for.

---

### Critical Issues

#### 1. Non-Atomic Lock Release (Race Condition)

```python
# CURRENT — not atomic, dangerous
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock. You delete a lock you no longer own. This is the canonical mistake in DIY Redis locking. The fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

Apply the same fix to the heartbeat renewal — the get-then-expire is equally non-atomic:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

#### 2. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# On the instance — shared across all callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the entire point of this service), `_heartbeat_active = False` from one call's `_stop_heartbeat` will kill the other call's heartbeat thread. The heartbeat state must be scoped to the individual lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not active.is_set():
                break
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        active.clear()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)  # atomic Lua version
```

#### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, null bytes, or a path-traversal-style string that becomes a malformed Redis key. `quantity` could be zero or negative, which would silently "succeed" and increment stock.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

#### 4. DB Operations Outside Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` fails, stock has already been decremented and there is no order to show for it. This is a data integrity issue independent of the Redis lock:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the atomicity requirement is non-negotiable for an inventory system.

### Moderate Issues

#### 5. Tests Mock Away the Thing Being Tested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most operationally novel and risky part of this PR. Mocking it out entirely means the tests give false confidence. At minimum, add an integration-style test that uses a real (or `fakeredis`) Redis instance and verifies the lock is actually extended and correctly released.

#### 6. `time.sleep` Blocks the Heartbeat Thread's Shutdown

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10s, ignores stop signal
```

When `_stop_heartbeat` sets `_heartbeat_active = False`, the thread can still be sleeping for up to 10 seconds before it notices. The `threading.Event` pattern shown in issue #2 above resolves this cleanly with `active.wait(timeout=...)`.

#### 7. Heartbeat Thread Is Never Joined

The daemon thread is started but `_stop_heartbeat` only sets a flag. There is no `join()` call, so there is no guarantee the heartbeat has actually stopped before the lock is released. A slow-waking heartbeat could extend a lock that was already intentionally released and re-acquired by another process.

---

### Minor Issues

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first renewal fires at 10s, leaving a 20s margin. That is fine, but it should be documented as an explicit design constraint, not just two magic numbers.
- The `retry_attempts=3` default with backoffs of 0.1s, 0.2s, 0.4s means maximum wait time before failure is 0.7s. For a flash sale this may be too aggressive — worth a comment explaining the intentional choice.
- Consider `redis.exceptions.RedisError` in the exception handling. A Redis connection failure currently surfaces as an unhandled exception rather than a graceful `ReservationResult(success=False, ...)`.
