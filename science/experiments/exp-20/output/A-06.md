Decision: Request Changes

Summary: The heartbeat renewal pattern here is the central architectural problem — it defeats the purpose of a TTL-based lock entirely. A lock whose TTL can be extended indefinitely by a thread that has no knowledge of whether the holding process is actually healthy or has entered a GC pause, network partition, or zombie state is not a bounded lease; it's an unbounded hold with extra steps. The storage operations (`update_stock`, `create_order`) happen outside any transactional boundary with the lock itself, meaning a process can hold the lock, extend it via heartbeat, complete the DB writes, and then die before releasing — leaving the lock held until the heartbeat thread also dies, which for a daemon thread means until the process exits, not until the operation is confirmed safe. There are also several concrete implementation flaws detailed below that compound the architectural risk.

---

## Detailed Review

### Critical — Architecture

**1. Heartbeat renewal inverts the safety guarantee of a TTL lock.**

The entire point of a bounded TTL on a distributed lock is that the system can make progress after a bounded wait, even if the lock holder dies, hangs, or is partitioned. The heartbeat thread removes that bound. Consider this sequence:

```
T=0   Process A acquires lock, starts heartbeat thread
T=5   Process A enters GC pause (or slow DB call)
T=10  Heartbeat thread wakes, checks token, extends TTL to T=40
T=15  Process A is still paused — DB is in an inconsistent intermediate state
T=25  Process B times out waiting, assumes lock is legitimately held
T=35  Process A resumes, completes writes with stale reads
```

The heartbeat thread has no signal from the application layer about whether the operation is proceeding correctly. It just blindly extends. This is exactly the zombie-write scenario Kleppmann describes: the process is alive enough to extend the lock but not alive enough to be making correct progress.

**2. The lock does not protect the storage layer — it only serializes Redis access.**

`update_stock` and `create_order` are two separate DB calls with no transactional wrapper and no fencing token passed to the storage layer. If the process dies between those two calls, you have decremented stock without a corresponding order record. The lock serializes entry into this critical section, but it provides no atomicity guarantee over the DB operations themselves. For a flash sale inventory system, this is a data integrity failure mode, not just a performance concern.

```python
# This is not atomic. A crash here leaves stock decremented, order missing.
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

**3. Instance-level heartbeat state means the service is not safely concurrent.**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not per-lock-acquisition variables. If two callers invoke `reserve_item` concurrently on the same `InventoryService` instance — which is the normal case for a shared service object — `_stop_heartbeat()` from one acquisition will set `_heartbeat_active = False` and kill the heartbeat for the other. The second caller's lock can then expire mid-operation with no renewal. This is a silent failure: no exception is raised, the heartbeat just stops.

### High — Race Conditions

**4. Check-then-act on lock ownership is not atomic.**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

And in the release path:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `EXPIRE`/`DELETE`, another process can acquire the lock. This should be a Lua script to make the check-and-act atomic:

```lua
-- Atomic release
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```lua
-- Atomic renewal
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

This is not a theoretical concern. Under the load profile of a flash sale, the window between `GET` and `DELETE` is exactly when contention is highest.

**5. Heartbeat thread can extend a lock that has already been legitimately acquired by another process.**

If the original lock expires (heartbeat missed a cycle, Redis hiccup), another process acquires it with a new token. The original heartbeat thread wakes up, calls `GET`, gets the new token, sees it doesn't match, and correctly skips the extend. That part is fine. But if there's a race where the original token is still present at the time of the `GET` but expires between `GET` and `EXPIRE`, the `EXPIRE` resets the TTL on a key that is now unowned or re-owned. The Lua script above fixes this.

### Medium — Implementation

**6. `quantity` has no validation.**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `quantity > 0`. A caller passing `quantity=0` will succeed, decrement stock by zero, and create an order for zero units. A caller passing `quantity=-5` will pass the stock check (`current_stock < -5` is false when stock is positive), decrement stock to `current_stock + 5`, and create an order — effectively adding inventory. This is a business logic vulnerability.

```python
# Missing guard
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

**7. `item_id` is used directly in a Redis key with no sanitization.**

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:`, `*`, `?`, or newline characters, this can cause key namespace collisions or, depending on the Redis client and command pipeline, unexpected behavior. Validate or encode `item_id` before interpolating it into a key.

**8. Daemon thread lifecycle is not deterministic on shutdown.**

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Daemon threads are killed when the main thread exits, but "killed" here means abruptly terminated — there is no guarantee the `_heartbeat_active = False` signal is processed, and more importantly, there is no guarantee the lock is released on process shutdown. Under a graceful shutdown with in-flight reservations, the lock will be held until TTL expiry. For a 30-second TTL with a heartbeat that just extended it, that could be up to 30 seconds of lock hold after the process is gone.

### Low — Test Coverage

**9. Tests mock out the exact behavior under review.**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the primary novel mechanism in this PR. Mocking it out means the tests provide no coverage of the actual concurrent behavior, the race conditions in the heartbeat loop, or the instance-variable collision described in finding #3. The tests verify the happy-path business logic, which is fine, but they do not test the distributed locking behavior at all.

**10. No test for the `quantity <= 0` case.**

Follows directly from finding #6.

## Recommended Path Forward

1. Remove the heartbeat. If operations genuinely take longer than the lock TTL, the right answer is to make the TTL longer and accept the worst-case hold time, or to restructure the operation so it completes within a bounded time. A heartbeat that can extend indefinitely is not a distributed lock — it's a mutex with extra network calls.

2. Wrap `update_stock` and `create_order` in a database transaction, or use a single atomic stored procedure. The lock serializes access; the DB transaction provides atomicity.

3. Move heartbeat state (if retained) to per-acquisition local scope, not instance variables.

4. Replace the GET/EXPIRE and GET/DELETE patterns with Lua scripts for atomic ownership-checked operations.

5. Add `quantity` validation before acquiring the lock.

6. Sanitize or validate `item_id` before key interpolation.

7. Write integration-level tests that exercise the actual locking path, including concurrent acquisition attempts.
