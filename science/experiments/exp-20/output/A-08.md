Decision: Request Changes

Summary: The heartbeat-based lock design introduces a classic zombie-write failure mode that the implementation does not mitigate — a GC pause or network partition can cause the process to believe it still holds the lock while the TTL has already expired and another holder has acquired it, meaning the token check in the release path is a necessary but not sufficient safety guard. Beyond that architectural concern, there are several concrete correctness, concurrency, and security issues that need to be addressed before this is safe to merge into a flash-sale context where overselling has real consequences.

---

## Critical Issues

### 1. Heartbeat Does Not Eliminate the Zombie-Write Window — It Narrows It

This is the core architectural problem. The PR description frames the heartbeat as a solution to lock expiry, but it only reduces the probability of expiry — it does not eliminate the failure mode Kleppmann describes.

Consider this sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat fires, reads token, calls EXPIRE — looks good
T=11  Process A enters a GC pause (or kernel scheduler preemption)
T=41  Lock TTL expires (30s from last extension)
T=42  Process B acquires the same lock — now two holders exist
T=43  Process A wakes up, heartbeat fires, reads token...
      but by now Process B has written its token, so the check fails
      — HOWEVER — Process A is already past the check and inside
      the critical section executing db.update_stock()
```

The heartbeat check and the DB write are not atomic. There is a window between "token still matches" and "DB write completes" where the process can be preempted. The code has no fencing token passed to the storage layer, so the DB cannot reject a stale write.

The correct mitigation is a monotonically increasing fencing token issued by the lock server, passed through to every storage operation, and enforced at the storage layer. Redis does not provide this natively. If you need true safety here, you need either:

- A storage layer that accepts and enforces a fencing token (e.g., a conditional write with a version/sequence number)
- An optimistic concurrency control approach at the DB level (compare-and-swap on stock version)
- Accepting that this lock provides *efficiency* (reduces contention) but not *correctness* (cannot prevent all double-writes), and adding compensating logic downstream

The current code provides no indication to callers or operators that the safety model is "best effort under normal conditions," which is dangerous in a government contracting context where auditability matters.

### 2. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal case in any web framework — two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa.

```python
# Caller 1 acquires lock, starts heartbeat, _heartbeat_active = True
# Caller 2 acquires lock, starts heartbeat, _heartbeat_active = True (same flag)
# Caller 1 finishes, calls _stop_heartbeat(), sets _heartbeat_active = False
# Caller 2's heartbeat thread now exits — lock for Caller 2 will expire
```

The heartbeat state needs to be scoped to the lock acquisition context, not the service instance. A simple fix is to pass a mutable container into the thread closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = {"value": True}  # closure-scoped, not instance-scoped

    def _beat():
        while heartbeat_active["value"]:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if heartbeat_active["value"]:
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active["value"] = False
        # optionally thread.join() with a short timeout
```

### 3. Race Condition Between Token Check and Lock Delete in Release Path

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)              # <-- DEL (non-atomic with GET)
```

Between the `GET` and the `DEL`, another process can acquire the lock. The correct implementation is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    script = self.redis.register_script(RELEASE_SCRIPT)
    result = script(keys=[lock_key], args=[token])
    return result == 1
```

This is a well-known pattern and its absence here is a correctness bug, not a style preference. The Redis documentation for distributed locks explicitly calls this out.

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation at all. Pydantic is used only for the return type. This means:

- `item_id` can be an empty string, a path traversal attempt, or arbitrarily long
- `quantity` can be zero, negative, or `sys.maxsize`
- The lock key `f"inventory:lock:{item_id}"` is constructed from unvalidated user input, which means a caller can target arbitrary Redis keys

```python
# An attacker can call:
service.reserve_item("../admin:lock:system", quantity=-1)
# Lock key becomes: inventory:lock:../admin:lock:system
# quantity=-1 passes the stock check: current_stock(10) < -1 is False
# but new_stock = 10 - (-1) = 11 — stock increases on "reservation"
```

The negative quantity issue is particularly bad: it passes the `current_stock < quantity` guard (any positive stock is greater than a negative number) and then *increases* stock via `update_stock`. Add explicit validation:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 5. Database Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record. This is a silent data integrity failure. These two operations need to be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock protects the read-modify-write cycle at the Redis layer, but it does not protect against partial failure at the DB layer.

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined
```

The daemon thread will continue sleeping for up to `HEARTBEAT_INTERVAL` seconds after the flag is cleared. In a high-throughput flash sale scenario, this means you can accumulate a large number of sleeping threads. The thread should be joined with a bounded timeout:

```python
def _stop_heartbeat(self, thread: threading.Thread) -> None:
    heartbeat_active["value"] = False
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In an async or thread-pool context, blocking the calling thread during backoff is wasteful. For a flash sale with high concurrency, this can exhaust the thread pool. This is a design-level concern — if the surrounding framework is async, this entire service needs to be async-aware or offloaded to a thread executor.

### 8. Test Coverage Mocks Out the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally significant part of this PR and it has zero test coverage. There is no test for the concurrent instance-level state race (Issue 2), no test for the non-atomic GET/DEL release (Issue 3), and no integration test that exercises the Lua release script. The test suite validates the happy-path business logic but gives false confidence about the locking behavior.
