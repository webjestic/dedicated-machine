Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-and-extend sequence, a class-level state management flaw that makes the service unsafe for concurrent use, and a zombie-write vulnerability that the heartbeat pattern itself introduces rather than solves. These are not stylistic concerns — they represent correctness failures under the exact high-concurrency flash sale conditions this service is designed to handle. The test suite mocks away the heartbeat entirely, meaning none of the most dangerous failure modes are exercised at all.

## Detailed Review

### 🔴 Critical: Heartbeat Creates the Zombie-Write Problem It Claims to Solve

This is the most important issue in the PR and it is architectural, not cosmetic.

The PR description frames the heartbeat as a safety feature. It is the opposite. The heartbeat is a mechanism for *extending* a lock that the process may no longer legitimately hold. Consider this sequence:

1. Process A acquires the lock and starts the heartbeat thread.
2. Process A's main thread stalls — GC pause, slow DB call, kernel scheduling delay.
3. The lock TTL expires naturally. Redis evicts it.
4. Process B acquires the lock and begins its reservation.
5. Process A's heartbeat thread wakes up, calls `redis.get(lock_key)`, and sees... nothing, because the key expired. So far so good — the heartbeat correctly does not extend.
6. **But**: if the timing is slightly different and B acquired the lock *just before* A's heartbeat fires, A's heartbeat sees B's token, correctly skips the extend. A's main thread then wakes up and continues executing `db.update_stock` and `db.create_order` — **without holding the lock** — because there is no re-validation of lock ownership before the DB writes.

The lock is checked at *release time*, not before the writes. The window between "lock expired" and "writes complete" is exactly the zombie-write scenario described by Kleppmann. The heartbeat does not close this window; it obscures it.

The correct mitigation is a **fencing token** — a monotonically increasing value issued by the lock server and passed through to the storage layer, which rejects writes carrying a stale token. Redis does not natively provide this, which is a known limitation of Redis-based locking for operations requiring true mutual exclusion at the storage layer.

```python
# What the code does — no re-validation before writes
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)
    # ... lock may have expired here under GC/network pause ...
    self.db.update_stock(item_id, new_stock)   # zombie write possible
    self.db.create_order(order_id, item_id, quantity)  # zombie write possible
```

```python
# Minimum mitigation — re-validate ownership before committing writes
# (still not sufficient without fencing tokens at the DB layer)
with self._acquire_lock(lock_key) as token:
    current_stock = self.db.get_stock(item_id)
    # ... business logic ...
    if not self._still_own_lock(lock_key, token):
        raise LockLostError("Lock expired before write commit")
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🔴 Critical: Heartbeat Race Condition (Check-Then-Act)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a non-atomic check-then-act. Between the `GET` and the `EXPIRE`, the key can expire and be acquired by another process. You then extend *their* lock, not yours. This is a classic TOCTOU race.

The fix is a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _extend_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(
        EXTEND_LOCK_SCRIPT, 1, lock_key, token, self.LOCK_TTL
    )
    return result == 1
```

The same issue exists in the lock release path:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This must also be an atomic Lua operation. This is well-documented in the Redis distributed lock pattern:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 🔴 Critical: Class-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level attributes, not call-level. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal usage pattern for a shared service object — the following happens:

- Call A sets `_heartbeat_active = True`, starts thread A.
- Call B sets `_heartbeat_active = True`, starts thread B.
- Call A's `finally` block calls `_stop_heartbeat()`, setting `_heartbeat_active = False`.
- Thread B is now also stopped, even though Call B is still inside the lock.

This is a silent correctness failure. The heartbeat for Call B stops, the lock TTL is no longer extended, and the lock expires while Call B is still executing writes.

The fix is to scope heartbeat state to the lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()
    thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 🟡 Significant: `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`. A caller passing `quantity=0` succeeds, creates an order record, and decrements stock by zero — a phantom order. A caller passing `quantity=-5` *increases* stock, which is a data integrity violation and a potential abuse vector in a flash sale context.

```python
if quantity <= 0:
    return ReservationResult(
        success=False,
        message="Quantity must be a positive integer"
    )
```

Similarly, `item_id` is not validated. An empty string or a string containing Redis key separator characters (`:`) could cause unexpected key collisions in `f"inventory:lock:{item_id}"`.

---

### 🟡 Significant: DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` fails, stock is decremented but no order exists. The item is effectively lost. This needs to be wrapped in a database transaction, or the DB layer needs to expose a transactional API for this operation.

---

### 🟡 Significant: Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread may still be mid-execution — specifically, between the `GET` and `EXPIRE` calls — when the lock is released and deleted. The thread then extends a key that either no longer exists or belongs to another process.

```python
def _stop_heartbeat(self, thread: threading.Thread) -> None:
    self._heartbeat_active = False
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 🟡 Significant: Single Redis Node Is a Single Point of Failure

The implementation uses a single `redis.Redis` client. Under a Redis primary failover, the replica that is promoted may not have received the lock key before promotion (async replication). A new process can acquire the lock on the new primary while the original holder still believes it holds the lock. This is the split-brain scenario.

For a flash sale — a high-stakes, high-contention operation — this should either use Redlock across multiple independent Redis nodes, or the team should explicitly document the accepted risk and the operational conditions (e.g., synchronous replication, no failover during sale window) under which the single-node approach is considered acceptable. Undocumented assumptions in distributed systems are operational liabilities.

---

### 🟠 Minor: Tests Mock Away All Dangerous Behavior

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture mocks out the heartbeat. This means the race conditions in the heartbeat, the thread lifecycle issues, and the lock extension behavior are completely untested. The test suite gives a false sense of coverage. At minimum, integration-level tests against a real Redis instance (e.g., using `pytest-redis` or a Docker fixture) should exercise the lock acquisition, extension, and release paths end-to-end.
