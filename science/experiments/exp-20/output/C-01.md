Decision: Request Changes

Summary: The core locking logic is well-structured and the ownership-verification pattern is correct, but there are several meaningful flaws — ranging from a thread-safety race condition in the heartbeat to missing input validation that the checklist incorrectly claims is present. These issues would cause real problems under the concurrent, high-load conditions this service is explicitly designed for.

---

## Detailed Review

### 🔴 Critical Issues

#### 1. Shared Heartbeat State Breaks Concurrent Usage

The `_heartbeat_active` and `_heartbeat_thread` flags are **instance-level attributes**, not scoped to individual lock acquisitions. If two calls to `reserve_item` overlap (e.g., two threads sharing the same `InventoryService` instance), the second call's `_stop_heartbeat()` will set `_heartbeat_active = False` and kill the first call's heartbeat thread prematurely — or vice versa.

```python
# Current — instance-level, shared across all concurrent callers
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Scope heartbeat state to the lock acquisition itself, not the instance. Use a local `threading.Event` passed into the thread closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        time.sleep((2 ** attempt) * 0.1)

    if not acquired:
        raise RuntimeError(...)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    try:
        yield token
    finally:
        stop_event.set()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note `stop_event.wait(timeout=...)` also replaces the `time.sleep` + flag check pattern, which has its own TOCTOU window.

#### 2. Heartbeat Has a TOCTOU Race on Token Check

In the current `_beat` loop:

```python
time.sleep(self.HEARTBEAT_INTERVAL)
if self._heartbeat_active:           # check
    current = self.redis.get(lock_key)
    if current and current.decode() == token:
        self.redis.expire(lock_key, self.LOCK_TTL)  # act
```

Between the `get` and the `expire`, another process could acquire the lock. The `expire` then extends a lock you no longer own. This should be an atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same atomic pattern should also be applied to the lock **release** in the `finally` block — the current `get` + `delete` sequence has the identical race.

### 🟡 Significant Issues

#### 3. Checklist Claims Input Validation via Pydantic — This Is False

`ReservationResult` is a Pydantic output model. There is **zero validation on inputs** to `reserve_item`. Nothing prevents:

```python
service.reserve_item("", quantity=-5)
service.reserve_item("item_001", quantity=0)
service.reserve_item("A" * 10000, quantity=999999)
```

A negative or zero quantity would pass the `current_stock < quantity` check incorrectly (e.g., `10 < -1` is `False`, so a reservation of `-1` items succeeds and *increases* stock). This is a correctness bug with potential for inventory corruption.

**Fix:** Add explicit guards at the top of `reserve_item`:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if not isinstance(quantity, int) or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

Or use a Pydantic input model and validate before entering the lock.

#### 4. DB Operations Are Not Atomic

`update_stock` and `create_order` are two separate calls with no transaction wrapping:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock has already been decremented and the order doesn't exist. The distributed lock protects against concurrent reads, but it does not protect against partial writes. These must be wrapped in a database transaction.

#### 5. `blocking_timeout` / Max Wait Time Is Unbounded for Callers

With `retry_attempts=3` and exponential backoff of `0.1, 0.2, 0.4` seconds, the maximum wait before failure is only ~0.7 seconds — acceptable. However, `retry_attempts` is a parameter with no upper bound enforced. A caller passing `retry_attempts=100` would block for over 200 seconds. Add a reasonable cap.

### 🟠 Test Coverage Gaps

#### 6. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the most complex and failure-prone component — the heartbeat thread — has **zero test coverage**. The test `test_lock_not_released_if_expired` claims to test lock expiry behavior but the heartbeat is mocked away, so it's only testing the `finally` block's token comparison. Add at least one integration-style test that lets the heartbeat run.

#### 7. No Test for Negative or Zero Quantity

Given the inventory corruption risk described in issue #3, this is a required test case.

#### 8. No Test for DB Failure Mid-Transaction

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the partial-write scenario from issue #4 and it needs explicit coverage once the transaction fix is in.

---

### 🟢 What's Done Well

- Lock ownership verification via UUID token is correct in concept.
- `nx=True, ex=self.LOCK_TTL` in a single `SET` call is the right atomic acquisition pattern.
- Structured logging with key/value pairs is clean and consistent.
- Returning a typed `ReservationResult` rather than raising exceptions for business-logic failures is a good API design.
- `daemon=True` on the heartbeat thread is appropriate.
