Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-and-extend sequence, a class-level heartbeat state design that makes the service unsafe for concurrent use, and a zombie-write vulnerability at the storage layer that the distributed lock alone cannot prevent. These are not cosmetic issues — they represent correctness failures under the exact high-concurrency flash sale conditions this service is designed to handle. The test suite mocks out the heartbeat entirely and does not exercise the failure modes that matter most.

---

## Detailed Review

### Critical Issues

#### 1. Heartbeat Race Condition — Non-Atomic Check-Then-Extend

```python
# _start_heartbeat — current implementation
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are two separate commands with no atomicity guarantee. Between the `GET` returning your token and the `EXPIRE` executing, the lock can expire naturally, be acquired by another process, and then your `EXPIRE` extends *their* lock. You have now handed an adversary a longer lease on a lock they legitimately acquired.

The fix is a Lua script, which Redis executes atomically:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if result == 0:
                    logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
                    self._heartbeat_active = False
                    break
                logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same atomic-release pattern must be applied to lock deletion in the `finally` block:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

#### 2. Class-Level Heartbeat State — Concurrent Calls Will Corrupt Each Other

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False        # shared across ALL calls
        self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are instance attributes, not call-scoped. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the normal case in any web framework — the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first call's `_stop_heartbeat` sets `_heartbeat_active = False` for both. The second lock will never be extended and may expire mid-operation.

The heartbeat state must be scoped to the individual lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = min((2 ** attempt) * 0.1, 2.0)
        logger.warning("Lock contention key=%s attempt=%d retrying in %.2fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
                heartbeat_active.clear()
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        release_script = self.redis.register_script(RELEASE_LOCK_SCRIPT)
        released = release_script(keys=[lock_key], args=[token])
        if released:
            logger.info("Lock released key=%s", lock_key)
        else:
            logger.warning("Lock already expired or stolen at release key=%s", lock_key)
```

Note also that `thread.join()` is now called in the `finally` block. The current code does not join the heartbeat thread, meaning a daemon thread can be mid-`EXPIRE` when the lock is deleted, creating a small window where the next owner's lock gets extended.

#### 3. Zombie-Write Vulnerability — No Fencing Token at the Storage Layer

This is the fundamental critique Kleppmann raised against Redlock-style implementations, and it applies here. The sequence is:

1. Process A holds the lock and is mid-operation.
2. A GC pause, OS scheduler delay, or network hiccup causes A to stall beyond the lock TTL (or the heartbeat fails silently).
3. The lock expires. Process B acquires it and begins its operation.
4. Process A resumes and completes `db.update_stock` and `db.create_order`.
5. Both A and B have now written to the database. Stock is corrupted.

The heartbeat mitigates this but does not eliminate it. The only correct solution is a **fencing token** — a monotonically increasing value issued with each lock acquisition that the storage layer checks and rejects if it is not the highest seen.

```python
# Acquire a fencing token from a Redis counter at lock acquisition time
fence_token = self.redis.incr("inventory:fence:counter")

# Pass it through to every storage operation
self.db.update_stock(item_id, new_stock, fence_token=fence_token)
self.db.create_order(order_id, item_id, quantity, fence_token=fence_token)
```

The database layer must then enforce:

```sql
UPDATE inventory
SET stock = %s, last_fence_token = %s
WHERE item_id = %s
  AND last_fence_token < %s   -- reject stale writes
```

Without this, the distributed lock provides a coordination hint, not a correctness guarantee.

### Moderate Issues

#### 4. Input Validation Is Declared But Not Enforced

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `item_id` is non-empty, no check that `quantity` is a positive integer, and no check that `quantity` does not exceed a per-request maximum. A caller passing `quantity=0` or `quantity=-5` will silently decrement stock incorrectly or produce a nonsensical reservation.

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int

    @field_validator("item_id")
    @classmethod
    def item_id_must_not_be_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("item_id must not be blank")
        return v

    @field_validator("quantity")
    @classmethod
    def quantity_must_be_positive(cls, v: int) -> int:
        if v < 1:
            raise ValueError("quantity must be at least 1")
        if v > 100:
            raise ValueError("quantity must not exceed 100 per reservation")
        return v
```

#### 5. Backoff Has No Jitter and No Cap

```python
backoff = (2 ** attempt) * 0.1
```

Under a thundering herd — which is the exact scenario during a flash sale — all contending processes will sleep for identical durations and retry simultaneously. This does not reduce contention; it synchronizes it. Add jitter and a ceiling:

```python
import random
backoff = min((2 ** attempt) * 0.1, 2.0) + random.uniform(0, 0.05)
```

#### 6. `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```
