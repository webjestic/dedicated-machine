Decision: Request Changes

Summary: The implementation contains several serious architectural flaws that make it unsafe for production use in a high-concurrency flash sale context. The most critical issues are a race condition in the heartbeat's check-then-act sequence, shared mutable state on the service instance that breaks concurrent use entirely, and a zombie-write window between heartbeat stop and lock release. The test suite mocks away the heartbeat entirely, meaning none of these failure modes are exercised. These are not cosmetic issues — they represent correctness failures under exactly the conditions this service is designed to handle.

---

## Detailed Review

### Critical: Shared Instance State Breaks Concurrency

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the service instance, not scoped to the lock context. If two threads call `reserve_item` concurrently on the same service instance — which is the entire point of this service — one thread's `_stop_heartbeat()` call will set `_heartbeat_active = False` and kill the *other* thread's heartbeat. This is a silent correctness failure. Thread A's lock will expire while it believes it still holds it.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
    ...
```

### Critical: Zombie-Write Window at Lock Release

```python
finally:
    self._stop_heartbeat()           # (1) heartbeat stops
    current = self.redis.get(lock_key)  # (2) network round-trip
    if current and current.decode() == token:
        self.redis.delete(lock_key)  # (3) delete
```

Between steps (1) and (3) there is an unprotected window. After the heartbeat stops, the lock TTL begins counting down. If the process stalls (GC pause, network delay) between steps (1) and (3), the lock can expire, be acquired by another process, and then be deleted by this process at step (3). This is the classic zombie-write scenario. The check-then-delete sequence is not atomic.

The correct fix is a Lua script that makes the ownership check and delete atomic at the Redis layer:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

This is not optional. Without it, the ownership verification provides no actual safety guarantee.

### Critical: Heartbeat Has the Same Race Condition

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)          # (1) read token
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

The get-then-expire sequence is not atomic. Between steps (1) and (2), the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns, effectively stealing it from the new legitimate holder. This should also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

---

### Architectural: Single Redis Node Is Not a Distributed Lock

The PR description says "distributed lock" but this is a single-node Redis lock. Under a network partition or Redis failover, this provides no safety guarantees. If Redis uses async replication (the default), a lock acquired on the primary can be lost on failover before it replicates to the replica, and two processes can hold the lock simultaneously.

If true distributed safety is required, the options are:

1. **Redlock across 5 independent nodes** — though be aware of Kleppmann's critique regarding clock drift and the absence of fencing tokens making it unsafe for storage operations regardless
2. **A fencing token pattern** — the lock grants a monotonically increasing token; the storage layer rejects writes with a token lower than the last seen value; this is the only approach that survives process pauses after lock expiry
3. **Accept single-node Redis** with documented failure modes and compensating controls (idempotency keys, reconciliation jobs)

For a flash sale inventory system at this firm's standards, the storage layer should be enforcing a fencing token or using a database-level transaction with optimistic concurrency (compare-and-swap on a version column) as the true safety boundary. The Redis lock should be treated as a performance optimization, not a correctness guarantee.

---

### Moderate: `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is not validated before the lock is acquired. A caller passing `quantity=0` or `quantity=-5` would silently succeed, potentially setting stock to an incorrect value. This should be validated at the entry point:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

The PR checklist claims "Input validation via Pydantic" but `ReservationResult` is an output model. There is no input model validating `item_id` or `quantity`.

---

### Moderate: `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes between them, stock is decremented but no order record exists. The Redis lock does not protect against this — it only serializes access, it does not provide atomicity. These calls must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### Minor: Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- mocked out
    svc._stop_heartbeat = MagicMock()   # <-- mocked out
    return svc
```

The heartbeat is the most complex and failure-prone component in this PR, and it is mocked in every test except `test_lock_not_released_if_expired`. There are no tests for the heartbeat's behavior under concurrent access, for what happens when the heartbeat fires after the lock has been released, or for the shared-state concurrency bug described above. The test suite provides false confidence.
