Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-then-extend sequence (non-atomic GET+EXPIRE), a zombie-write vulnerability from the unsynchronized heartbeat thread that can extend a lock the main thread has already decided to release, and a shared-instance concurrency flaw where `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables rather than per-call state — meaning concurrent `reserve_item` calls on the same service object will corrupt each other's heartbeat lifecycle. The test suite mocks out the heartbeat entirely, so none of these failure modes are exercised. These are not cosmetic issues; in a flash sale scenario under real concurrency they constitute correctness-breaking bugs that can produce oversells or phantom lock holds.

---

## Detailed Findings

### 1. Critical — Non-Atomic Heartbeat Extension (Zombie-Lock Risk)

**File:** `services/inventory.py`, `_start_heartbeat`, lines inside `_beat()`

```python
# CURRENT — two separate round-trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You then extend a lock you no longer own. This is the classic check-then-act race on a distributed resource. The fix is a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

class InventoryService:
    def __init__(self, redis_client, db):
        self.redis = redis_client
        self.db = db
        self._extend_lock = redis_client.register_script(EXTEND_LOCK_SCRIPT)
```

The same atomic pattern applies to the release path in `_acquire_lock`'s `finally` block — it has the identical GET-then-DELETE race. This is well-documented; use the standard Lua release script here too:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 2. Critical — Shared Instance State Breaks Concurrent Calls

**File:** `services/inventory.py`, `__init__` and `_start_heartbeat`

```python
# CURRENT — instance-level, not call-level
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the **same** `InventoryService` instance simultaneously (which is the normal case for a shared service object), the second call's `_start_heartbeat` overwrites `_heartbeat_active` and `_heartbeat_thread`. When either call invokes `_stop_heartbeat`, it stops *both* heartbeats. This is a straightforward shared-mutable-state bug.

The fix is to scope heartbeat state to the lock acquisition call, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # call-scoped, not instance-scoped

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_active.set()
    heartbeat_thread = threading.Thread(
        target=self._heartbeat_loop,
        args=(lock_key, token, heartbeat_active),
        daemon=True,
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)

def _heartbeat_loop(
    self,
    lock_key: str,
    token: str,
    active: threading.Event,
) -> None:
    while active.wait(timeout=self.HEARTBEAT_INTERVAL):
        extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL * 1000])
        if not extended:
            logger.error(
                "Heartbeat failed to extend lock — lock lost key=%s", lock_key
            )
            active.clear()
            return
        logger.debug("Lock TTL extended key=%s", lock_key)
```

Note the `heartbeat_thread.join()` in `finally` — without it, the heartbeat thread can race ahead and attempt to extend a lock that the main thread is simultaneously releasing.

### 3. High — Heartbeat Thread Can Extend After Release Decision Is Made

Even with the instance-state issue fixed, the current `_stop_heartbeat` / release sequence has a window:

```
main thread:  _stop_heartbeat()  →  GET(lock_key)  →  DELETE(lock_key)
heartbeat:                      →  GET(lock_key)  →  EXPIRE(lock_key)   ← extends before main deletes
```

The `join()` in the revised `_acquire_lock` above closes this window by ensuring the heartbeat thread has fully exited before the release attempt proceeds.

### 4. High — No Validation on `item_id` or `quantity` at the Service Layer

**File:** `services/inventory.py`, `reserve_item`

`ReservationResult` uses Pydantic, but the *inputs* to `reserve_item` are raw strings and integers with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an `item_id` of arbitrary length. The lock key is constructed directly from `item_id` with no sanitization:

```python
lock_key = f"inventory:lock:{item_id}"
```

A malformed or adversarially crafted `item_id` containing colons or whitespace will not cause a Redis error, but it can collide with other key namespaces or produce confusing log output. Add explicit input validation:

```python
from pydantic import BaseModel, Field, field_validator
import re

ITEM_ID_RE = re.compile(r'^[a-zA-Z0-9_-]{1,64}$')

class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=64)
    quantity: int = Field(..., ge=1, le=1000)

    @field_validator('item_id')
    @classmethod
    def item_id_safe(cls, v: str) -> str:
        if not ITEM_ID_RE.match(v):
            raise ValueError(f"item_id contains invalid characters: {v!r}")
        return v
```

### 5. Medium — Single-Node Redis Is Not a Distributed Lock

The PR description says "distributed lock" but the implementation targets a single `redis.Redis` instance. Under a Redis primary failover or network partition, the lock provides no safety guarantee — a new primary may not have the key, and two processes can both believe they hold the lock simultaneously. This is the core of Kleppmann's critique of Redlock, and it applies even more directly to single-node Redis.

For a government contracting context where correctness under failure matters, the options are:

- **Accept the single-node limitation explicitly** and document that this provides mutual exclusion only under normal operation, not under partition or failover. Pair it with a fencing token passed through to the database layer so that a stale write from a zombie process is rejected at the storage layer even if the lock is lost.
- **Use Redlock across N≥3 independent Redis nodes** if you need stronger guarantees, with the understanding that Redlock still does not provide safety under clock drift or GC pauses without fencing tokens.
- **Use a CP coordination service** (etcd, ZooKeeper) if the correctness requirement is hard.

The current code has no fencing token propagation to `db.update_stock` or `db.create_order`, so even if the lock is held correctly, a zombie process that had its lock expire mid-operation will complete its writes with no rejection at the storage layer.

### 6. Medium — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the lock expires between these two calls, stock is decremented but no order is created — or vice versa depending on failure timing. These two writes must be wrapped in a single database transaction. This is not a distributed locking concern; it is basic data integrity.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```
