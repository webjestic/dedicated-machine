Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real flash sale load in a government-grade system. The most critical is a race condition in the heartbeat's check-then-extend sequence and a concurrency flaw in the `InventoryService` instance design that makes it unsafe under concurrent use.

---

## Issues

### 🔴 Critical

**1. `InventoryService` instance is not thread-safe — heartbeat state is shared**

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If a single `InventoryService` instance handles concurrent requests (the normal case with a shared service object), two simultaneous calls to `reserve_item` will clobber each other's heartbeat state. The second call's `_stop_heartbeat()` will set `_heartbeat_active = False`, killing the first call's heartbeat thread.

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to each lock acquisition, not to the instance. Pass a mutable stop-signal (e.g., a `threading.Event`) into the heartbeat thread directly:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        # release logic...
```

```python
def _start_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    threading.Thread(target=_beat, daemon=True).start()
```

Using `stop_event.wait(timeout=...)` also eliminates the `time.sleep` + double-check pattern, which is a minor logic smell in the current `_beat`.

**2. Heartbeat has a TOCTOU race on lock extension**

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write — not atomic
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You would then extend a lock you no longer own. This must be a single atomic operation. Use a Lua script:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not result:
    logger.warning("Lost lock during heartbeat key=%s", lock_key)
    stop_event.set()
```

The same TOCTOU issue exists in the lock release in `_acquire_lock`'s `finally` block and must be fixed with the same pattern.

### 🟠 High

**3. No input validation on `item_id` or `quantity`**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or an arbitrarily long `item_id` that becomes a Redis key. Pydantic is imported but not used for this purpose.

```python
# quantity=0 passes the stock check and creates a zero-quantity order
if current_stock < quantity:  # 10 < 0 is False — passes
```

Add a request model and validate at the entry point:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

The `item_id` pattern constraint is important — without it, a crafted `item_id` containing `:` could manipulate the Redis key namespace (`inventory:lock:{item_id}`).

**4. Database operations are not atomic — partial failure leaves inconsistent state**

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # raises — stock is now wrong
```

If `create_order` raises, stock has been decremented but no order exists. The distributed lock does not protect against this — it only serializes access. The database operations must be wrapped in a transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architecture gap that must be addressed before this goes to production.

---

**5. Bare `Exception` not caught — unhandled DB errors bypass the lock release... actually the `finally` handles it, but errors are silently swallowed**

If `self.db.get_stock()` raises (network timeout, connection pool exhausted), the exception propagates out of `reserve_item` entirely uncaught past the `RuntimeError` handler. The lock *will* be released by `finally`, which is correct, but the caller receives an unhandled exception rather than a graceful `ReservationResult`. Add a broad catch in `reserve_item`:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable. Please try again.")
```

### 🟡 Medium

**6. Tests mock out the heartbeat entirely — the most critical code path is untested**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture mocks out the heartbeat. This means the TOCTOU race, the thread lifecycle, and the lock extension logic have zero test coverage. Add dedicated unit tests for the heartbeat using `threading.Event` and a fake Redis that tracks call sequences.

---

**7. `retry_attempts=3` with backoff of 0.1s, 0.2s, 0.4s is too aggressive for flash sale load**

Under a flash sale, hundreds of goroutines hitting the same lock will all retry at nearly the same intervals, producing a retry storm. Add jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

Also consider making `retry_attempts` and base backoff configurable at the service level rather than hardcoded.

---

**8. `LOCK_TTL = 30` seconds is very long**

A 30-second lock means a crashed process holds the lock for up to 30 seconds, blocking all reservations for that item. For a flash sale, this is a significant availability risk. Reconsider whether the heartbeat approach (short TTL + renewal) is preferable to a long static TTL. If using heartbeat, the base TTL can be much shorter (e.g., 5–10 seconds) since renewal keeps it alive for healthy processes.

---

### 🔵 Minor

**9. `lock_key` format should be documented and centralized**

The key format `inventory:lock:{item_id}` is constructed inline. Define it as a class-level method or constant so it cannot drift between callers:

```python
@staticmethod
def _lock_key(item_id: str) -> str:
    return f"inventory:lock:{item_id}"
```

**10. `logger.error` vs `logger.exception` in exception handlers**

In the `RuntimeError` catch, use `logger.exception` to capture the full traceback, or at minimum pass `exc_info=True`. `logger.error` with a formatted string loses the stack trace.

---

## What's Done Well

- Lock ownership verification via UUID token is correct in concept
- `nx=True, ex=self.LOCK_TTL` in a single `SET` call is the right atomic primitive
- Returning `ReservationResult` rather than raising on business logic failures is clean
- Structured logging with key=value pairs is consistent
- The `finally` block guaranteeing lock release is correctly placed
