Decision: Request Changes

Summary: The heartbeat renewal pattern here is architecturally unsound for a distributed lock — it defeats the core safety guarantee that TTL expiry provides, and the implementation has a race condition in the check-then-extend sequence that makes it worse, not better. Beyond the locking model, there are several concrete correctness and safety issues that need to be addressed before this is anywhere near production-ready for a flash sale workload.

---

## Critical Issues

### 1. Heartbeat Renewal Is the Wrong Model — and This One Is Broken

This is the most important issue in the PR, so I want to be precise about it.

The TTL on a distributed lock exists for a reason: it is your **only safety net** when a process dies, stalls, or gets partitioned. The moment you add a heartbeat that extends the TTL indefinitely, you have converted a bounded lease into an unbounded one. You have not made the lock safer — you have removed its primary failure-recovery mechanism.

Now layer on top of that the specific implementation here:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a non-atomic check-then-act. Between the `GET` and the `EXPIRE`, another process can acquire the lock (after the original TTL expires), and this heartbeat thread will then extend *their* lock using *their* token — wait, no, the token check prevents that specific case. But the window still exists where:

1. Lock expires naturally (process stalled for >30s)
2. Process B acquires the lock with a new token
3. Original heartbeat thread wakes up, does `GET`, gets Process B's token, sees it doesn't match, does nothing — *correct behavior here*, but...
4. The original process (A) is now a **zombie**: it has no lock, but it is still executing inside the `with self._acquire_lock(lock_key)` block, reading and writing stock

The heartbeat does not solve this. It papers over it. The zombie-write scenario is still fully reachable: if the process stalls for longer than the heartbeat interval but the heartbeat itself also stalls (GC pause, thread starvation, network hiccup to Redis), the lock expires, another process acquires it, and now two processes are concurrently executing the critical section.

**What to do instead:** If you genuinely need long-running critical sections, the correct approach is to use a **fencing token** — a monotonically increasing value from Redis (e.g., `INCR`) issued at lock acquisition time — and pass it through to the storage layer so the DB can reject writes from stale lock holders. That is the only mechanism that actually prevents zombie writes. If your DB does not support fencing, you need to redesign the operation to be short enough to fit within a single TTL without renewal.

### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not per-lock-acquisition variables. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — two concurrent acquisitions will stomp on each other's `_heartbeat_active` flag. Caller A sets it `True`, starts a thread. Caller B sets it `True`, starts a second thread. Caller A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`. Now Caller B's heartbeat has also been killed. Caller B's lock will expire mid-operation.

This is a silent correctness failure under the exact load profile this service is designed for.

---

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation at all. Pydantic is used only for the *output* model. There is no guard against:

- `quantity <= 0` — a caller passing `quantity=0` or `quantity=-5` will pass the stock check and create a nonsense order
- `item_id` containing Redis key injection characters (`:`, `*`, `?`, `[`) — the lock key is constructed as `f"inventory:lock:{item_id}"` with no sanitization, which is a real concern when `item_id` comes from user input in a flash sale context

At minimum:

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
```

And `item_id` should be validated against an allowlist pattern before being interpolated into a Redis key.

---

### 4. DB Operations Are Not Atomic — Stock Can Go Negative

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `update_stock` succeeds and `create_order` fails, you have decremented stock with no corresponding order. If this is a relational DB, these must be wrapped in a transaction. If it is not, you need compensating logic. As written, a partial failure here will silently corrupt inventory state — which in a flash sale context means phantom stock loss that you will not discover until reconciliation.

## Moderate Issues

### 5. Lock Release Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is another non-atomic check-then-delete. Between `GET` and `DELETE`, the lock can expire and be acquired by another process, whose lock you then delete. The correct implementation is a Lua script, which Redis executes atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is well-documented in the Redis distributed lock literature and should be considered a baseline requirement, not an optimization.

---

### 6. `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has stopped. The thread could be mid-sleep on its 10-second interval. After `_stop_heartbeat()` returns, the lock is deleted, but the heartbeat thread may wake up within the next 10 seconds, call `GET`, find a *new* lock with a different token (or nothing), and proceed. Given the TOCTOU issue above, this creates additional surface area for interference. The thread should be joined with a timeout:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Test Coverage Mocks Out the Exact Behavior Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test that exercises `reserve_item` mocks out the heartbeat entirely. This means the test suite provides zero coverage of the heartbeat's interaction with lock state, the concurrency safety of instance-level flags, or the behavior under lock expiry. The one test that is supposed to test lock expiry behavior (`test_lock_not_released_if_expired`) is testing the release path in isolation, not an integrated scenario. For a service whose primary design claim is "heartbeat prevents expiry under load," having no tests of that claim is a significant gap.

## Summary of Required Changes

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Heartbeat renewal defeats TTL safety; implement fencing tokens or remove renewal |
| 2 | Critical | Instance-level heartbeat state is not safe for concurrent callers |
| 3 | High | No input validation on `quantity` or `item_id`; injection risk in key construction |
| 4 | High | DB operations lack transaction boundary; partial failure corrupts stock |
| 5 | High | Lock release is non-atomic; use Lua script |
| 6 | Moderate | Heartbeat thread not joined on stop |
| 7 | Moderate | Tests mock out the core mechanism being reviewed |

The fundamental design question that needs an answer before any of the implementation details matter: **does your storage layer support fencing tokens?** If not, the architecture of this service needs to be reconsidered from the ground up, because no amount of heartbeat tuning will make a Redis-only distributed lock safe for inventory mutation under partition or process stall conditions.
