Decision: Request Changes

Summary: The core locking logic is well-structured and the ownership-verification pattern (token comparison before release) is correct, but there are several meaningful flaws — one of them a race condition that undermines the entire safety guarantee of the distributed lock. The test suite also has a critical gap that masks the most important concurrent-access scenario. These must be addressed before this handles real flash-sale traffic.

---

## Critical Issues

### 1. Race Condition in Lock Release (TOCTOU)

The check-then-delete in the `finally` block is not atomic:

```python
# UNSAFE — another process can steal the lock between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock can expire and be acquired by a different worker. You would then delete a lock you no longer own, causing a double-reservation. This is a well-known failure mode and the standard fix is a Lua script that makes the comparison and deletion atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The exact same race exists in `_start_heartbeat` — the GET/EXPIRE pair must also be atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 2. Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not lock-scoped. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously (which is the entire point of this service), one thread's `_stop_heartbeat()` will kill the other thread's heartbeat. Heartbeat state must be scoped to each lock acquisition, not to the service instance. Pass a `threading.Event` into `_start_heartbeat` and return it, keeping all state local to the `_acquire_lock` context:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return stop_event

# In _acquire_lock:
stop_event = self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    stop_event.set()
    ...
```

Note that `stop_event.wait(timeout=...)` is also preferable to `time.sleep` + flag check because it responds to the stop signal immediately rather than waiting out the sleep interval.

## Moderate Issues

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but neither `item_id` nor `quantity` is validated before use. A caller passing `quantity=0`, `quantity=-5`, or an `item_id` containing shell-unsafe characters or path traversal sequences (relevant if `item_id` is ever used in a filesystem or subprocess context downstream) will pass through silently. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 4. `db.update_stock` and `db.create_order` Are Not Transactional

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record — inventory is silently lost. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architectural gap that must be resolved before this goes to production.

### 5. Bare `Exception` Not Caught

Only `RuntimeError` is caught. If `self.db.get_stock`, `self.db.update_stock`, or the Redis client raises any other exception (network timeout, connection reset, etc.), it will propagate uncaught out of `reserve_item`, bypassing the structured error response contract the caller expects. At minimum:

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Test Coverage Gaps

### 6. The Concurrent Access Test Is Mocked Away

The most important correctness property — that two simultaneous reservations against stock of 1 result in exactly one success — is completely absent. The heartbeat is also mocked out in the fixture, so no test exercises the actual heartbeat thread. Add:

- A threading-based concurrent reservation test using a real or fake in-memory backend
- A test that verifies the heartbeat actually extends the TTL (without mocking `_start_heartbeat`)
- A test for the atomic release path (Lua script called, not GET+DELETE)

### 7. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test validates the current unsafe GET+DELETE behavior. Once the Lua script is in place, this test should be rewritten to verify that the script is called with the correct keys and args, and that a non-matching token results in no deletion.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but both should be configurable via constructor parameters or environment config rather than class-level constants, especially for a government-facing service where operational tuning without a redeploy matters.
- Log lines should include a correlation/request ID for traceability across distributed components. Consider accepting an optional `request_id` parameter and threading it through all log calls.
- The `ReservationResult` model is clean. No issues there.
