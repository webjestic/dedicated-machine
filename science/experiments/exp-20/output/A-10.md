**Decision: Request Changes**

**Summary:** The heartbeat design contains a race condition that makes the lock release non-atomic, which is a textbook zombie-write vector. Beyond that critical flaw, the service has shared mutable state that breaks under concurrent callers, and the database operations inside the lock boundary have no transactional guarantee, meaning a crash between `update_stock` and `create_order` produces a silent inconsistency. These are not cosmetic issues — they are failure modes that will manifest under exactly the flash-sale load this service is designed to handle.

---

## Detailed Findings

### 1. Non-Atomic Lock Release (Critical)

The `finally` block performs a read-then-delete as two separate Redis commands:

```python
current = self.redis.get(lock_key)          # command 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # command 2
```

Between those two commands, the lock can expire and be acquired by another process. You then delete a lock you no longer own. This is the exact zombie-write failure mode Kleppmann describes. The canonical fix is a Lua script that makes the check-and-delete atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

The heartbeat has the same problem — `get` followed by `expire` is two commands with a gap between them. Both must be made atomic via Lua or replaced with a single `SET ... XX PX` renewal pattern.

---

### 2. Shared Mutable State Breaks Concurrent Callers (Critical)

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of this service — the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and `_stop_heartbeat` sets `_heartbeat_active = False` for both. One lock loses its heartbeat silently and expires mid-operation. Heartbeat state must be local to each lock acquisition, not stored on `self`. Pass it through a closure or a small dataclass scoped to the `_acquire_lock` context.

---

### 3. No Transactional Guarantee on Database Writes (High)

Inside the lock boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process dies, the network drops, or `create_order` raises after `update_stock` succeeds, stock is decremented with no corresponding order. The lock does not protect you from this — it only serializes access. These two writes need to be wrapped in a database transaction, or the operation needs to be designed for idempotent recovery. At minimum, the `except` block in `reserve_item` should not silently swallow exceptions that originate from inside the lock after a partial write.

---

### 4. Input Validation Is Declared But Not Enforced (Medium)

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation at the call site. `item_id` is passed directly into a Redis key and a database query without sanitization. A caller passing `item_id = "lock:admin:*"` or an empty string produces a malformed lock key. The Pydantic model is only used for the *output*. Add a request model for inputs or add explicit guards:

```python
if not item_id or not item_id.isalnum():
    raise ValueError(f"Invalid item_id: {item_id!r}")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

---

### 5. Heartbeat Can Extend a Lock After the Critical Section Exits (Medium)

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` seconds before checking `_heartbeat_active`. If `_stop_heartbeat` is called right after the thread wakes and reads `_heartbeat_active = True` but before it calls `redis.expire`, the extension fires after the lock has already been released or has expired. This is a narrow window but it is real, and under GC pause or thread scheduling delay it widens. The `_heartbeat_active` flag needs to be a `threading.Event` with proper coordination, not a bare boolean.

---

### 6. Test Coverage Mocks Away the Code Under Review (Medium)

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # stop never runs
    return svc
```

The heartbeat thread — the most operationally novel and failure-prone part of this PR — is mocked out in every test that uses the `service` fixture. The tests pass because they are not testing the code that was written. Add at least one integration-level test that lets the heartbeat thread actually run and verifies TTL extension behavior against a real or fakeredis instance.

---

### 7. Daemon Thread Suppresses Errors Silently (Low)

The heartbeat thread is a daemon thread with no exception handling inside `_beat`. If `redis.expire` raises a `ConnectionError` during a network partition, the thread dies silently, the lock stops being renewed, and the operation continues unaware. Add a try/except inside `_beat` that logs the failure and, on repeated failures, sets a flag the main thread can observe.
