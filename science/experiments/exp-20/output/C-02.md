Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw: `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on the `InventoryService` object, meaning concurrent calls to `reserve_item` on the same service instance will race on those shared fields — one call's heartbeat can be killed by another call's `_stop_heartbeat()`, or two heartbeats can stomp each other's state entirely. This is a correctness and reliability defect that directly undermines the lock safety guarantee the PR is designed to provide. Additionally, the heartbeat thread has a TOCTOU race between the `get` and `expire` calls that could allow a stolen lock to be silently extended, and the tests mock out the heartbeat entirely, meaning the most dangerous concurrent behavior is never actually exercised.

---

### Detailed Findings

#### 🔴 Critical — Heartbeat State Is Not Thread-Safe (Instance-Level Shared State)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are shared across all concurrent invocations of `reserve_item` on the same instance. In a web server handling concurrent requests (gunicorn, uvicorn, etc.), two threads calling `reserve_item` simultaneously will race:

- Thread A starts heartbeat → sets `_heartbeat_active = True`
- Thread B calls `_stop_heartbeat()` → sets `_heartbeat_active = False`
- Thread A's lock silently stops being renewed and expires mid-operation

The fix is to make heartbeat state local to each lock acquisition, not the instance. Pass a mutable container or use a `threading.Event` scoped to the `_acquire_lock` context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition only
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Release only if we still own the lock
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
            logger.info("Lock released key=%s", lock_key)
```

```python
def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning(
                    "Heartbeat: lock ownership lost key=%s, stopping", lock_key
                )
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note `stop_event.wait(timeout=...)` replaces `time.sleep` + flag check — it wakes immediately when `set()` is called, eliminating the up-to-`HEARTBEAT_INTERVAL`-second delay on shutdown.

---

#### 🔴 Critical — Heartbeat Thread Is Never Joined

The current `_stop_heartbeat` just sets a flag. The heartbeat thread may still be sleeping for up to `HEARTBEAT_INTERVAL` seconds after the lock is released, and could wake up and extend a lock that has already been legitimately acquired by another process. The fix above addresses this with `join(timeout=...)`.

---

#### 🟠 High — TOCTOU Race in Heartbeat Ownership Check

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.lock_TTL)  # write
```

Between `GET` and `EXPIRE`, another process could acquire the lock. This should be an atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat
extended = self._extend_lock_script(
    keys=[lock_key], args=[token, self.LOCK_TTL]
)
if not extended:
    logger.warning("Heartbeat: lost lock ownership key=%s", lock_key)
    break
```

The same TOCTOU concern applies to the lock release in `_acquire_lock`'s `finally` block — that should also use a Lua script (this is the standard Redlock pattern).

---

#### 🟠 High — No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model wrapping the inputs. A caller can pass `quantity=0`, `quantity=-5`, or an empty/malformed `item_id`. At minimum:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

---

#### 🟡 Medium — DB Operations Outside a Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists — a silent inventory leak. These should be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### 🟡 Medium — Tests Mock Out the Entire Heartbeat

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means zero test coverage of the most critical concurrent behavior. At minimum, add an integration-style test that verifies the heartbeat actually extends the TTL and stops cleanly after lock release. The `test_lock_not_released_if_expired` test is also misleading — it patches the token comparison at the mock level but doesn't test the actual `_acquire_lock` finally block behavior.

---

#### 🟢 Minor — `LOCK_TTL` and `HEARTBEAT_INTERVAL` Should Be Configurable

Hardcoded class-level constants make tuning for different deployment environments require a code change. Accept these as constructor parameters with defaults.
