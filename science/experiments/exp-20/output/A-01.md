Decision: Request Changes

Summary: The heartbeat-renewal pattern here is the canonical zombie-write setup that Kleppmann's critique specifically calls out — a GC pause, network stall, or OS scheduler delay can suspend the process *after* the heartbeat confirms ownership but *before* the storage write completes, meaning the lock has expired and been re-acquired by another process while this one is still executing inside the critical section. Beyond that architectural flaw, there are several concrete implementation bugs that would cause data corruption or silent failures under production load.

## Detailed Review

### 1. Critical: Heartbeat Renewal Does Not Prevent the Zombie-Write Failure Mode

This is the core issue and the reason I can't approve this as-is.

The heartbeat gives a *false sense of safety*. Consider this sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat extends TTL — lock looks healthy
T=11  Process A enters GC pause / OS scheduler delay / network stall
T=41  Lock expires (no heartbeat fired during pause)
T=42  Process B acquires the same lock
T=43  Process A resumes from pause, still inside `with self._acquire_lock()`
T=43  Process A calls db.update_stock() — now TWO processes own the critical section
```

The heartbeat thread is on a *different OS thread* from the application thread. The application thread can be paused independently. The heartbeat firing at T=10 gives you no guarantee the application thread is still making forward progress. This is not a theoretical concern — JVM/CPython GC, Linux CFS scheduler jitter, and network socket timeouts all produce exactly this pause profile.

**What actually solves this:** Fencing tokens at the storage layer. The lock server issues a monotonically increasing token on each acquisition. The storage backend rejects any write carrying a token older than the last committed write. Redis alone cannot provide this — you need the database to participate.

```python
# Sketch of what the DB layer needs to enforce:
# db.update_stock(item_id, new_stock, fence=token)
# Storage rejects if fence <= last_committed_fence[item_id]
```

If fencing tokens are out of scope for this iteration, the PR description needs to explicitly document the residual risk and the conditions under which a zombie write can occur. Shipping this without that acknowledgment is the problem.

### 2. Critical: Shared Mutable Heartbeat State Breaks Under Concurrent Use

```python
# InventoryService instance fields:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are *instance-level* fields. If a single `InventoryService` instance handles two concurrent `reserve_item` calls — which is the normal case in any threaded or async web framework — the second call overwrites `_heartbeat_active` and `_heartbeat_thread` while the first call's heartbeat is still running.

```
Thread 1: _heartbeat_active = True,  _heartbeat_thread = thread_A
Thread 2: _heartbeat_active = True,  _heartbeat_thread = thread_B  ← overwrites
Thread 1: finally: _stop_heartbeat() sets _heartbeat_active = False
           ← this also kills Thread 2's heartbeat
Thread 2: lock now expires because its heartbeat was silently stopped
```

The fix is to scope heartbeat state to the lock acquisition context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

Note also that `_stop_heartbeat` sets the flag but never joins the thread. The heartbeat thread can fire *after* the lock has been released and re-acquired by another process, extending a lock it no longer owns. The `get`/`expire` check is not atomic — there is a TOCTOU window between them.

### 3. High: `get` + `expire` in the Heartbeat Is Not Atomic

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be re-acquired by another process. You then extend a lock you no longer own. This should be a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

extend_lock = self.redis.register_script(EXTEND_LOCK_SCRIPT)

def _beat():
    result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
    if result == 0:
        logger.warning("Heartbeat lost lock ownership key=%s", lock_key)
        heartbeat_active.clear()
```

The same atomicity requirement applies to the release path — the existing `get`/`delete` pair has the same race. Use the standard atomic release script there too.

### 4. High: No Input Validation on Public Entry Point

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` — a caller passes `quantity=0` or `quantity=-5`, stock check passes (`10 < -5` is False... wait, actually `10 < -5` is False so it *would* pass the stock check with negative quantity, incrementing stock)
- `item_id` containing Redis key injection characters (`:`, `*`, `?`, `[`)
- `item_id` of unbounded length

```python
# Minimum required:
if not isinstance(item_id, str) or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
# Enforce a reasonable length bound
if len(item_id) > 128:
    raise ValueError("item_id exceeds maximum length")
```

The `current_stock < quantity` check with a negative `quantity` would pass and call `db.update_stock(item_id, current_stock - (-5))` — silently *adding* stock during a reservation call.

### 5. Medium: `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is killed, or the DB connection drops between these two calls, you have decremented stock with no corresponding order record. The distributed lock provides no protection here — this is a database transaction boundary problem. Both operations must be wrapped in a single DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Medium: Test Suite Mocks Away the Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()    # ← stop entirely bypassed
    return svc
```

The heartbeat thread — the most operationally complex and failure-prone component — is completely mocked out in every test except `test_lock_not_released_if_expired`. None of the concurrency behavior is tested. The tests validate the happy-path business logic but provide no coverage for the failure modes this PR is supposed to handle.

Minimum additional tests needed:
- Concurrent `reserve_item` calls on the same item from multiple threads, asserting final stock equals `initial - sum(reservations)` with no oversell
- Heartbeat fires and extends TTL before the lock expires
- Heartbeat detects lost ownership and stops extending
- Lock release is skipped when token has changed (this test exists but uses the mocked heartbeat, so it doesn't test the real release path)

---

### 7. Low: `LOCK_TTL = 30` With `HEARTBEAT_INTERVAL = 10` Is a Tight Margin

With a 10-second heartbeat interval and a 30-second TTL, you have at most two missed heartbeats before the lock expires. Under load — connection pool exhaustion, Redis latency spike, GC pause — missing two consecutive heartbeats is not rare. A more conservative ratio is TTL = 5× heartbeat interval. If slow DB operations are the stated concern, profile those operations and set TTL accordingly rather than using a round number.
