Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-and-extend sequence that creates a zombie-write window, and the `InventoryService` class uses instance-level heartbeat state that makes it unsafe for concurrent use — both are disqualifying flaws for a flash sale reservation system operating under high concurrency. Additionally, the storage layer lacks the atomicity guarantees necessary to prevent oversell, and the test suite mocks away the exact behaviors that need to be verified.

## Detailed Review

### 🔴 Critical — Must Fix Before Merge

---

#### 1. Heartbeat TOCTOU Race Creates Zombie-Write Window

```python
# _beat() in _start_heartbeat:
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the lock can expire and be acquired by a different process. The heartbeat then extends the TTL of a lock it no longer owns, giving the original (now-zombie) process a live lock alongside the new legitimate holder. Both processes now believe they hold the lock and will proceed to write inventory. This is a textbook zombie-write failure mode.

**Fix:** Replace the get-then-expire sequence with a single atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not result:
                logger.warning("Heartbeat: lock lost or expired key=%s", lock_key)
                self._heartbeat_active = False
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same TOCTOU race exists in the lock release path and must also be made atomic:

```python
# Current — also racy:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

**Fix:** Use the standard atomic release script:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

#### 2. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared instance state
        self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of a flash sale service — the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat and leaving its lock unrenewed. The first reservation will then lose its lock mid-operation.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    # ... acquire lock ...

    def _beat() -> None:
        extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not result:
                logger.warning("Heartbeat: lock lost key=%s", lock_key)
                heartbeat_active.clear()
                break

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script here
```

Note also that `_stop_heartbeat` sets a flag but does not `join()` the thread. The heartbeat thread can continue running after the lock is released, racing against the next acquisition on the same key.

#### 3. Non-Atomic Stock Decrement Allows Oversell

```python
current_stock = self.db.get_stock(item_id)   # read
# ...
self.db.update_stock(item_id, new_stock)      # write
self.db.create_order(order_id, item_id, quantity)
```

The Redis lock protects against concurrent Python processes, but only if every writer uses the same lock key and the lock is never lost. If the lock expires between the read and the write (e.g., heartbeat failed silently, GC pause exceeded TTL, Redis failover), the write proceeds against stale data. The storage layer has no independent guard.

**Fix:** The database write must be conditional. Use an optimistic concurrency check or a database-level atomic decrement with a floor constraint:

```python
# Option A: conditional update (SQL example)
rows_affected = self.db.decrement_stock_if_sufficient(
    item_id, quantity, expected_version=current_version
)
if rows_affected == 0:
    return ReservationResult(success=False, message="Stock changed, please retry")
```

```python
# Option B: atomic decrement with floor check (Redis-backed stock)
new_stock = self.redis.eval("""
    local stock = tonumber(redis.call('GET', KEYS[1]))
    if stock == nil or stock < tonumber(ARGV[1]) then return -1 end
    return redis.call('DECRBY', KEYS[1], ARGV[1])
""", 1, f"stock:{item_id}", quantity)
if new_stock < 0:
    return ReservationResult(success=False, message="Insufficient stock")
```

The distributed lock should be a performance optimization (reducing DB contention), not the sole correctness mechanism.

#### 4. Input Validation Is Declared But Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no check that `item_id` is non-empty, no maximum length, no check that `quantity > 0`. A caller passing `quantity=0` or `quantity=-5` will silently "succeed" and corrupt stock values.

```python
# Minimum required:
if not item_id or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
if quantity > MAX_RESERVATION_QUANTITY:
    raise ValueError(f"quantity exceeds maximum allowed ({MAX_RESERVATION_QUANTITY})")
```

If Pydantic is the intended mechanism, define a `ReservationRequest` model and validate at the boundary.

---

#### 5. Heartbeat Interval Exceeds Safe Renewal Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at 10 seconds and extends by 30 seconds. This is fine in the happy path. However, if a heartbeat fires at T=10 and the next fires at T=20, the lock expires at T=30 — leaving only a 10-second window before expiry. Under a GC pause, thread starvation, or Redis latency spike, that window can be missed entirely. The standard guidance is to renew at one-third of TTL, which this does achieve, but the TTL itself should be tuned to the actual p99 operation time, not set to a "generous" 30 seconds. A 30-second lock is an extremely long hold for an inventory read-modify-write.

---

#### 6. `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The item is effectively lost. These two writes must be wrapped in a single database transaction.
