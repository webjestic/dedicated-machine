Decision: Request Changes

Summary: The core locking pattern and heartbeat concept are sound, but there are several concurrency and correctness flaws that would be dangerous in a high-load flash sale scenario. The most critical issue is that the `InventoryService` instance stores heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`), making the service non-reentrant and unsafe for concurrent use. Beyond that, the lock release in the `finally` block contains a TOCTOU race, the input validation advertised in the checklist is largely absent, and the test suite mocks away the very behaviors it should be verifying.

---

## Detailed Findings

### 🔴 Critical — Instance-Level Heartbeat State (Concurrency Corruption)

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If two threads call `reserve_item` concurrently on the same `InventoryService` object (which is the entire point of this service), one thread's `_stop_heartbeat()` call will kill the other thread's heartbeat, potentially allowing its lock to expire mid-operation. This is a silent data-corruption bug under the exact load conditions this service is designed for.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local scope, not self.*

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff,
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_active.set()
    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, heartbeat_active),
        daemon=True,
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)

def _run_heartbeat(
    self,
    lock_key: str,
    token: str,
    active: threading.Event,
) -> None:
    while active.wait(timeout=self.HEARTBEAT_INTERVAL):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.LOCK_TTL)
            logger.debug("Lock TTL extended key=%s", lock_key)
        else:
            logger.warning("Lost lock ownership key=%s — stopping heartbeat", lock_key)
            break
```

### 🔴 Critical — TOCTOU Race in Lock Release

```python
# Current — two separate Redis round-trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE` another process could acquire the lock after the original TTL expires, and this code would then delete the new owner's lock. This is a well-known failure mode; the Redis documentation explicitly prescribes a Lua script for atomic check-and-delete.

**Fix:**

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock not released — already expired or stolen key=%s", lock_key
        )
```

### 🔴 Critical — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

Setting the flag does not guarantee the thread has stopped before the lock is deleted. The heartbeat thread could wake up after `_stop_heartbeat()` returns, observe the token still matches (because deletion hasn't happened yet or the thread read the flag late), and call `redis.expire`, extending a lock that should be released. Always join the thread before releasing.

### 🟠 High — No Input Validation Despite Checklist Claim

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts arbitrary `item_id` strings and `quantity` integers with zero validation.

```python
# item_id could be empty string, path-traversal fragment, or absurdly long
# quantity could be 0, negative, or INT_MAX
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

**Fix:** Introduce a request model and validate at the boundary.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

    @field_validator("item_id")
    @classmethod
    def no_whitespace(cls, v: str) -> str:
        if v != v.strip():
            raise ValueError("item_id must not have leading/trailing whitespace")
        return v
```

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValueError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")

    lock_key = f"inventory:lock:{req.item_id}"
    ...
```

### 🟠 High — `time.sleep` in Heartbeat Misses Cancellation

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10s, ignores stop signal
```

`time.sleep` is not interruptible. If the operation completes in 1 second, the heartbeat thread sleeps for the remaining 9 seconds before it can observe `_heartbeat_active = False`. During that window the lock has already been deleted and the thread will wake up and attempt to extend a dead or re-owned lock. The `threading.Event` approach in the fix above resolves this because `event.wait(timeout=N)` returns immediately when the event is cleared.

### 🟡 Medium — Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost. The distributed lock protects against concurrent access but does not protect against partial failure. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Medium — Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # stop never runs
    return svc
```

Every test that uses the `service` fixture is testing a version of the class that has no heartbeat at all. The tests pass, but they provide no confidence in the actual production code path. The lock contention test also does not verify retry count or backoff behavior.

**Recommended additions:**
- An integration-style test using `fakeredis` that exercises the real lock acquire/release/heartbeat path.
- A test that verifies `db.update_stock` is rolled back when `db.create_order` raises.
- A test that confirms concurrent calls on the same `item_id` serialize correctly and do not double-decrement stock.

---

### 🟢 Minor — Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
```

`RuntimeError` will also catch unexpected exceptions from inside the `with` block that happen to be `RuntimeError` subclasses. Define a specific exception type.

```python
class LockAcquisitionError(RuntimeError):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```
