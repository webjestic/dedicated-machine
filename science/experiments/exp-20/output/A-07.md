Decision: Request Changes

Summary: The heartbeat renewal pattern here is the canonical zombie-write setup that Kleppmann's critique specifically warns against — a GC-paused or network-partitioned process will keep renewing its lock while another process has already acquired it at the storage layer, and the ownership check (`get` + compare) is a non-atomic TOCTOU race that cannot be fixed without Lua. Beyond the correctness issues, the service has a shared-state threading bug that makes it unsafe for any concurrent call pattern, and the database operations lack transactional integrity, meaning a crash between `update_stock` and `create_order` produces silent data corruption. The test suite mocks away the heartbeat entirely, so none of the failure modes it's supposed to guard against are actually exercised.

---

### Critical Issues

#### 1. Heartbeat renewal defeats the purpose of lock expiry (zombie-write vector)

This is the core architectural flaw. The heartbeat thread renews the lock TTL while the holding process is alive — but liveness of the process is not the same as validity of the critical section. Consider:

```
T=0   Process A acquires lock, heartbeat starts
T=15  Process A enters GC pause / network partition
T=25  Lock TTL would have expired — but heartbeat renewed it at T=20
T=30  Process A resumes, believes it holds the lock
T=30  Process B has been waiting; storage layer has no way to evict A
```

The entire point of a finite TTL is to bound the blast radius of a dead or partitioned holder. Renewal eliminates that bound. If you need operations that take longer than your TTL, the correct answer is fencing tokens at the storage layer, not TTL extension. The DB operations (`update_stock`, `create_order`) need to accept and enforce a monotonically increasing fencing token so that a zombie write from an expired session is rejected even if it arrives late.

#### 2. Lock release is a TOCTOU race — needs atomic Lua CAS

```python
# Non-atomic: another process can acquire between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a classic check-then-act race. Between the `GET` and the `DELETE`, a concurrent process can acquire the lock (if it expired between those two calls), and you then delete a lock you no longer own. The standard fix is a Lua script that makes the comparison and deletion atomic:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

The same race exists in the heartbeat's `get` + `expire` sequence.

#### 3. Shared instance state makes concurrent calls mutually destructive

```python
self._heartbeat_active = False          # instance-level, not call-level
self._heartbeat_thread: Optional[...]   # same
```

`InventoryService` is almost certainly a singleton or long-lived object. Two concurrent calls to `reserve_item` on the same instance will share `_heartbeat_active` and `_heartbeat_thread`. Call A's `_stop_heartbeat()` will set `_heartbeat_active = False`, killing Call B's heartbeat thread mid-operation. This needs to be scoped to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread, stop via event.clear()
    ...
```

#### 4. No transactional boundary around the two DB writes

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

A crash, exception, or timeout between these two calls leaves stock decremented with no corresponding order record — or vice versa depending on ordering. This is silent data corruption. These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer doesn't expose transactions, that's a separate issue that needs to be resolved before this service can be considered safe.

### Moderate Issues

#### 5. Input validation is declared but not implemented

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is passed directly into a Redis key and a DB query with no sanitization. At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

A Pydantic `ReservationRequest` model should be the entry point if the checklist claim is to be accurate.

#### 6. Retry loop sleeps on the main thread under load

```python
time.sleep(backoff)  # blocks the calling thread for up to 0.3s per attempt
```

Under flash sale load this will exhaust thread pool capacity quickly. Callers should be informed of contention via a specific exception type (e.g., `LockContentionError`) so the API layer can return `503` with a `Retry-After` header rather than holding the connection open.

### Test Suite Issues

#### 7. Heartbeat is mocked away — the primary failure mode is untested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture stubs out the heartbeat. There are zero tests for: TTL extension behavior, the shared-state concurrency bug, heartbeat stopping when the lock is stolen, or what happens when `redis.expire` fails inside the thread. The test suite gives false confidence.

#### 8. `test_lock_not_released_if_expired` tests the wrong thing

This test verifies that `delete` is not called when the token doesn't match — but it doesn't test the TOCTOU window described in issue #2, and it doesn't test that a foreign lock is not deleted. It's testing the happy path of the ownership check, not the race.
