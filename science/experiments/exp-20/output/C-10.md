Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and architecture issues that must be addressed before this handles real flash sale load in a government contracting context where auditability and correctness are non-negotiable.

---

## Critical Issues

### 1. Race Condition in Heartbeat Lock Extension

The heartbeat performs a non-atomic check-then-extend, which is a TOCTOU race:

```python
# UNSAFE — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could be stolen between these
```

Between the `GET` and `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be an atomic Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In heartbeat:
extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    self._heartbeat_active = False  # we lost the lock, stop beating
    logger.warning("Heartbeat: lock ownership lost key=%s", lock_key)
```

### 2. Non-Atomic Lock Release (Same TOCTOU Problem)

The same race exists in the `finally` block:

```python
# UNSAFE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another owner could be deleted here
```

This is a well-known Redis pattern that requires a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""

released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
if not released:
    logger.warning("Lock already expired or stolen, skipping delete key=%s", lock_key)
```

---

### 3. Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`, meaning a single `InventoryService` instance cannot safely handle concurrent calls:

```python
self._heartbeat_active = False      # shared across all callers
self._heartbeat_thread = None       # last writer wins
```

If two threads call `reserve_item` simultaneously on the same instance, one `_stop_heartbeat()` call will kill the other's heartbeat. Fix by scoping heartbeat state to the lock context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, heartbeat_active)
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 4. Database Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. This is a phantom reservation — inventory is gone with no record. These must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction needs to expose a transaction context manager. This is not optional for financial or inventory operations.

---

### 5. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, a SQL injection fragment, or a multi-megabyte payload. `quantity` could be zero or negative:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing validates these before use
```

Add explicit guards:

```python
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1 or quantity > 1000:
    return ReservationResult(success=False, message="Invalid quantity")
```

Or introduce a proper request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat` sets a flag but does not wait for the thread to finish:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread may still be sleeping for up to HEARTBEAT_INTERVAL seconds
```

The thread could attempt to extend a lock after it has already been released. Always join with a bounded timeout:

```python
heartbeat_active.clear()
if heartbeat_thread and heartbeat_thread.is_alive():
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Broad Exception Handling Silently Swallows DB Errors

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught and unlogged through the lock context manager. The `finally` block will still run, but the caller gets an unhandled exception rather than a structured response. Add a broader catch inside the lock scope:

```python
with self._acquire_lock(lock_key):
    try:
        # ... db operations ...
    except Exception as e:
        logger.error("DB operation failed item_id=%s: %s", item_id, e, exc_info=True)
        return ReservationResult(success=False, message="Reservation failed due to internal error")
```

---

### 8. Test Suite Mocks Out the Exact Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()    # stop entirely bypassed
    return svc
```

The heartbeat is the most complex and failure-prone part of this implementation, and it is never exercised by the test suite. The critical TOCTOU bugs identified above would pass all these tests. At minimum, add integration-style tests that run the real heartbeat against a mock Redis client with controlled timing.

---

## Minor Issues

### 9. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These should be instance-level or injected via constructor to allow environment-specific tuning (e.g., staging vs. production DB latency profiles) without subclassing.

### 10. `token` Is Yielded From `_acquire_lock` But Never Used by the Caller

```python
with self._acquire_lock(lock_key):   # token is yielded but discarded
```

The yield is vestigial. Either use it or remove it to avoid confusion.

The architecture is sound and the intent is correct. Issues 1–4 are blocking and would cause data corruption or lock theft under real flash sale concurrency. Please resolve those before resubmission.
