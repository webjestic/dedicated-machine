Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `nx=True`, ownership-verified release, heartbeat renewal, and structured logging are all sound patterns. However, there are several correctness, safety, and architecture issues that must be addressed before this is production-ready in a high-stakes government contracting environment.

---

## Critical Issues

### 1. Race Condition in Heartbeat: Non-Atomic Check-and-Extend

```python
# CURRENT — two separate commands, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, another process could acquire the lock. You are then extending a lock you no longer own. This must be a single atomic operation using a Lua script:

```lua
-- Atomic heartbeat renewal
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In _start_heartbeat:
result = self.redis.eval(HEARTBEAT_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if result == 0:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
    self._heartbeat_active = False
```

### 2. Race Condition in Lock Release: Non-Atomic Check-and-Delete

Same class of bug. The release in `_acquire_lock` has the same GET-then-DELETE gap:

```python
# CURRENT — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a well-known Redis distributed lock anti-pattern. Use the canonical Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""

# In finally block:
released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 3. InventoryService Is Not Safe for Concurrent Use

The heartbeat state is stored as instance variables:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of this service — they will corrupt each other's heartbeat state. `_heartbeat_active = False` from one thread will kill the other thread's heartbeat.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through a mutable container or a dedicated lock-scoped object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # ... pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

## Significant Issues

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used on the *output* model. A caller can pass `quantity=-5`, `quantity=0`, or an `item_id` that is an empty string or contains injection-relevant characters.

```python
# Add at the top of reserve_item:
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1 or quantity > 10_000:
    return ReservationResult(success=False, message="Invalid quantity")
```

Or better, introduce a `ReservationRequest` Pydantic model as the method parameter.

---

### 5. `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. Inventory is silently lost. These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The distributed lock protects Redis-level contention, but it does not substitute for ACID guarantees on the database side.

---

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` raised inside the lock — including from `db.update_stock` or `db.create_order` — will be silently swallowed and returned as "service unavailable." Introduce a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass

# Raise it explicitly in _acquire_lock
raise LockAcquisitionError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

# Catch it specifically
except LockAcquisitionError as e:
    ...
```

## Test Coverage Gaps

### 7. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and most dangerous part of this implementation. It has zero test coverage. At minimum, add:

- A test verifying the heartbeat calls `EXPIRE` only when the token still matches
- A test verifying the heartbeat stops when `_heartbeat_active` is cleared
- A test verifying the heartbeat does *not* extend a lock it no longer owns

### 8. Lock Contention Test Does Not Validate Retry Behavior

`test_reserve_lock_contention` sets `redis_client.set.return_value = False` and checks for failure, but never asserts that retry was attempted the expected number of times (`redis_client.set.assert_called_n_times(3)`). The retry logic is untested.

### 9. No Test for the Partial DB Failure Scenario

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the most financially dangerous failure mode in the service.

---

## Minor Notes

- `LOCK_TTL = 30` seconds is generous for what should be a sub-second operation. Consider documenting the expected p99 DB latency that justifies this value, or reducing it with a comment explaining the reasoning.
- The `token` yielded from `_acquire_lock` is never used by the caller. The `yield token` is dead interface surface — either use it or `yield` nothing.
- Consider `redis.exceptions.RedisError` handling separately from application logic errors so Redis connectivity failures are observable and alertable distinctly from business logic failures.
