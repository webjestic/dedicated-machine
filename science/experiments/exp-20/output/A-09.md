Decision: Request Changes

Summary: The heartbeat renewal pattern is the central architectural flaw here — it directly undermines the safety guarantee that TTL-based locks are supposed to provide, and the implementation compounds this with a race condition in the check-then-extend sequence that makes zombie writes a realistic failure mode rather than a theoretical one. Beyond the locking model, the service has a non-atomic DB operation pair that can corrupt inventory state even when the lock behaves correctly, and the instance-level heartbeat state makes this class unsafe for any concurrent use. These issues need to be resolved before this touches a production flash sale.

## Detailed Review

### Critical: Heartbeat Renewal Defeats the Purpose of TTL

This is the core problem, and it is architectural, not cosmetic.

The entire value proposition of a TTL-based distributed lock is that the storage layer will *automatically reclaim the lock* if the holder dies, stalls, or gets garbage collected. That guarantee is what makes the lock safe under partial failure. The moment you add a heartbeat that continuously resets the TTL, you have converted a lease into a session lock — but without the session-layer failure detection that would make a session lock safe.

```python
# This pattern is the problem
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # TTL reset indefinitely
```

Consider the failure scenario: the process holding the lock stalls — GC pause, slow DB call, network hiccup. The heartbeat thread is in `time.sleep(10)`. The main thread is blocked. If the stall exceeds `LOCK_TTL` (30s), the lock expires and another process correctly acquires it and begins its critical section. When the original process resumes, it will check the token, find it has changed, and skip the delete — that part is correct. But the DB operations (`update_stock`, `create_order`) have *already executed* against shared state. You now have two processes that both believe they hold the lock, both writing to inventory. This is the zombie write failure mode.

The heartbeat makes this *more likely*, not less, because it extends the window during which the original process believes it is the lock holder while the storage layer has already moved on.

The correct mental model here: **the lock TTL is a hard upper bound on how long a critical section is allowed to run**. If your DB operations might take longer than that bound, the answer is to make the operations faster or to accept that the lock will expire and design your storage layer to be safe under that condition (e.g., conditional writes, optimistic concurrency, fencing tokens at the DB layer).

### Critical: Race Condition in Heartbeat Check-Then-Extend

Even setting aside the architectural problem above, the heartbeat implementation has a TOCTOU race:

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write — not atomic with the read
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. You then extend *their* TTL, not yours. This should be a single atomic operation. The standard pattern is a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same race exists in the lock release path:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process could acquire between GET and DELETE
```

This is a well-known pattern and the fix is identical — a Lua script that atomically checks ownership and deletes. This is not optional; it is the standard safe release pattern for Redis locks.

### Critical: Non-Atomic DB Operations Allow Inventory Corruption

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If the process dies, loses its DB connection, or hits an exception between these two lines, you will have decremented stock without a corresponding order record, or vice versa. This is a data integrity issue that exists independently of the locking behavior.

These operations must be wrapped in a single atomic transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the requirement is that both writes either commit together or neither commits.

### Critical: Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False
        self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` on the same `InventoryService` instance concurrently — which is the entire point of a service class in a web application — they will clobber each other's heartbeat state. Thread A's `_stop_heartbeat()` call will terminate Thread B's heartbeat, and vice versa.

The heartbeat state must be scoped to the lock acquisition context, not to the instance. The simplest fix is to pass the active flag as a mutable container into the thread closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = {"value": True}  # scoped to this acquisition

    def _beat():
        while heartbeat_active["value"]:
            ...

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active["value"] = False
        ...
```

### Moderate: No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:baz"` or an empty string will silently produce a malformed key. A negative `quantity` will silently decrement stock in the wrong direction. Define a `ReservationRequest` Pydantic model and validate at the entry point:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### Moderate: Tests Mock Out the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the heartbeat thread, the TOCTOU race in the release path, or the concurrency behavior of the instance-level state. The test suite passes precisely because it does not test the code paths that are broken. At minimum, add integration-style tests that use a real or fake Redis (e.g., `fakeredis`) and do not mock the locking internals.

---

### Minor: Daemon Thread Behavior on Process Exit

Daemon threads are silently killed when the main thread exits. If the process is shutting down while a reservation is in flight, the heartbeat thread will be killed mid-operation without any cleanup. This is probably acceptable for a heartbeat, but it should be a documented and conscious choice, not an implicit one.

---

## What Needs to Change Before Approval

1. **Remove the heartbeat renewal pattern** or replace it with a design that does not extend the lock indefinitely. If operations genuinely cannot complete within a fixed TTL, the correct solution is fencing tokens at the storage layer, not unbounded TTL extension.
2. **Replace the check-then-extend and check-then-delete patterns** with atomic Lua scripts.
3. **Wrap `update_stock` and `create_order` in a single DB transaction.**
4. **Move heartbeat state into the lock acquisition scope**, not the instance.
5. **Add input validation** on `item_id` and `quantity` via a Pydantic request model.
6. **Add tests that exercise the actual locking code** using `fakeredis` or equivalent, without mocking `_start_heartbeat` and `_stop_heartbeat`.
