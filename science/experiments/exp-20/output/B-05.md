Decision: Request Changes

Summary: The implementation contains a critical race condition in the heartbeat's check-and-extend sequence that can produce zombie writes under network partition or GC pause — the exact failure mode Kleppmann's Redlock critique identifies as unresolvable without fencing tokens. Beyond that core flaw, there are several additional correctness, concurrency, and architectural issues that must be addressed before this is safe to merge into a flash-sale reservation path, where an inventory double-spend is a direct business and compliance liability.

---

## Critical Issues

### 1. No Fencing Token — Zombie Writes Are Possible

This is the most serious flaw. The heartbeat thread does not fix the fundamental problem with Redis-based distributed locks: a process can believe it holds the lock while the lock has already expired and been re-acquired by another process.

**Scenario:**
1. Process A acquires the lock.
2. A GC pause, kernel preemption, or network stall causes A to stop executing for longer than `LOCK_TTL`.
3. The lock expires. Process B acquires it and begins its reservation.
4. Process A resumes, its heartbeat thread has also been paused, and it now calls `db.update_stock()` and `db.create_order()` — **inside what it still believes is a valid lock boundary**.
5. Both A and B commit writes. Inventory is double-spent.

The heartbeat does not prevent this. It only helps in the steady-state case where the process is alive and running. It cannot protect against a pause that outlasts the TTL window between heartbeat intervals.

The standard mitigation is a **monotonically increasing fencing token** issued at lock acquisition time, passed through to the storage layer, and rejected by the storage layer if a higher token has already been seen. This implementation has no such mechanism.

```python
# What is needed at the storage layer:
def update_stock(item_id: str, new_stock: int, fence: int) -> bool:
    # Reject if a higher fence token has already written to this record
    ...
```

Without this, no amount of heartbeat tuning makes this safe for a high-value reservation path.

### 2. Heartbeat Has a Non-Atomic Check-Then-Act Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # <-- window here
```

Between the `GET` and the `EXPIRE`, the lock can expire and be re-acquired by another process. The heartbeat then extends a lock it no longer owns. This must be a single atomic operation using a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same non-atomic pattern exists in the lock release:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # <-- another process's lock can be deleted here
```

This is a well-known bug. Use the standard Lua script for atomic release:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 3. Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal case in any threaded web server (Flask with threaded=True, gunicorn with threads, FastAPI with a thread pool executor) — two concurrent calls will clobber each other's `_heartbeat_active` and `_heartbeat_thread`. One call stopping its heartbeat will stop the other's. This is a silent correctness failure under load, which is exactly when flash sales run.

Each lock acquisition must carry its own isolated heartbeat state, not share it on the service instance.

```python
# Correct approach: heartbeat state is local to the context manager
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
```

### 4. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is killed, or the DB connection drops between them, stock is decremented but no order is created. This is a silent inventory leak. Both calls must be wrapped in a single database transaction that commits or rolls back atomically.

## Significant Issues

### 5. `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`. A caller passing `quantity=0` acquires the lock, passes the stock check (`0 < 10`), and creates an order for zero units. A caller passing `quantity=-5` would increase stock. The checklist claims input validation via Pydantic, but `quantity` is a bare `int` with no constraints. Add a validated request model:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128)
    quantity: int = Field(ge=1, le=1000)
```

---

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread may still be mid-execution, holding a reference to the Redis client and the lock key, after the `finally` block has already deleted the lock. Under high concurrency this produces a thread leak and unpredictable post-release Redis calls. At minimum:

```python
def _stop_heartbeat(self, thread: threading.Thread) -> None:
    heartbeat_active.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 7. Single Redis Node Is a Single Point of Failure

The implementation uses a single `redis.Redis` client. If that node fails, all lock acquisitions fail and the service returns "temporarily unavailable" for every request. For a flash sale this is an unacceptable availability profile. The PR description does not acknowledge this. If Redlock (multi-node quorum) is out of scope, the architectural decision to accept this SPOF must be explicitly documented and signed off on, not left implicit.

## Test Coverage Gaps

### 8. Heartbeat Is Mocked Away Entirely

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the primary novel mechanism this PR introduces. Mocking it out means zero test coverage of the most failure-prone component. There should be at least:
- A test that the heartbeat calls `expire` with the correct token.
- A test that the heartbeat does **not** extend a lock when the token has changed.
- A test that the heartbeat stops after `_stop_heartbeat` is called.

### 9. The Concurrency Case Is Not Tested

There is no test that runs two concurrent `reserve_item` calls against shared state and asserts that exactly one succeeds and stock is not double-decremented. For a flash sale reservation service, this is the primary correctness invariant and it is completely untested.

---

## Minor Issues

### 10. `RuntimeError` Is Too Broad for Lock Failure

Raising and catching `RuntimeError` for lock contention conflates infrastructure errors with expected contention behavior. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

### 11. `item_id` Is Not Sanitized Before Use as a Redis Key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:`, spaces, or newline characters, this can produce malformed or colliding key names. Validate and sanitize `item_id` before constructing the key.
