Decision: Request Changes

Summary: The heartbeat renewal pattern here is the canonical zombie-write setup that Kleppmann's critique describes — a process that believes it holds a lock after a GC pause or network partition can have its TTL renewed by a background thread that survived the pause, allowing it to continue writing against storage that another process now legitimately owns. Beyond that architectural flaw, there are several concrete race conditions and shared-state bugs that would cause silent data corruption under the exact concurrent load this service is designed to handle.

---

## Critical Issues

### 1. Heartbeat Renewal Defeats the Purpose of TTL (Architecture)

This is the core problem. The TTL on a distributed lock is your **safety net** — it bounds the window during which a crashed or paused process can block forward progress. The heartbeat thread eliminates that bound entirely.

Consider this sequence:

1. Process A acquires the lock, starts the heartbeat thread.
2. Process A's main thread enters a GC pause or gets descheduled for 25 seconds.
3. The heartbeat thread, running independently, keeps renewing the TTL every 10 seconds.
4. Process B times out waiting, fails its reservation.
5. Process A resumes, now operating on stale stock data it read before the pause, and commits a write.

The heartbeat thread has no visibility into whether the **work** is still valid — it only knows the thread is alive. The lock is being renewed based on process liveness, not operation liveness. This is the zombie-write failure mode.

**What to do instead:** Size your TTL to be genuinely larger than your worst-case operation time (include p99.9 DB latency, not just average). If your operations routinely approach the TTL, that is a signal to fix the operation, not extend the lock. If you need session-based locks, use a system designed for it (e.g., Zookeeper ephemeral nodes, or a fencing token pattern against your storage layer).

---

### 2. Shared Instance State Makes the Service Non-Reentrant and Thread-Unsafe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped locals. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern — one service instance, many concurrent requests), they will race on `_heartbeat_active` and `_heartbeat_thread`.

Concrete failure: Thread A sets `_heartbeat_active = True` and starts its heartbeat. Thread B calls `_stop_heartbeat()`, setting `_heartbeat_active = False`. Thread A's heartbeat loop exits. Thread A's lock now expires mid-operation. Thread A writes stale data.

**Fix:** Scope heartbeat state to the lock acquisition call, not the instance. Pass state through a closure or a local object.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into the thread closure
```

### 3. Check-Then-Act Race on Lock Release

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a non-atomic check-then-delete. Between the `GET` and the `DELETE`, another process can acquire the lock (after your TTL expires) and you will delete their lock, not yours. This is a well-known failure mode and the fix is a standard Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

The same race exists in the heartbeat's `GET` → `EXPIRE` sequence. Both must be atomic.

---

### 4. No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

- `item_id` is used directly in a Redis key: `f"inventory:lock:{item_id}"`. A caller passing `item_id = "foo:bar:baz"` will construct a malformed key. A caller passing an empty string constructs `inventory:lock:` which is a valid Redis key that will collide across all items with empty IDs.
- `quantity` has no lower bound check. `quantity=0` will succeed and create an order for zero units. `quantity=-5` will *increase* stock.

**Fix:**

```python
if not item_id or not item_id.isalnum():
    raise ValueError(f"Invalid item_id: {item_id!r}")
if quantity < 1:
    raise ValueError(f"quantity must be >= 1, got {quantity}")
```

---

### 5. DB Operations Are Not Atomic Relative to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order exists. The Redis lock does not protect against this — it only serializes access to the read-modify-write cycle. You need a database transaction wrapping both operations, or a compensating write on failure.

## Moderate Issues

### 6. Test Suite Mocks Out the Exact Behavior Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is mocked in every test. This means the tests provide zero coverage of the actual concurrency behavior — the thing most likely to fail in production. `test_lock_not_released_if_expired` claims to test lock expiry behavior but the heartbeat that would prevent expiry is mocked away. These tests validate the happy-path business logic only.

**Add:** Integration-level tests with a real or fake Redis (e.g., `fakeredis`) that exercise actual lock acquisition, TTL expiry, and concurrent reservation attempts.

### 7. Daemon Thread Lifetime Is Unbounded on Slow Operations

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

`daemon=True` means the thread will be killed when the main process exits, but it does not bound the thread's lifetime within a normal request. If `_stop_heartbeat` is never called (e.g., due to an unhandled exception type that is not `RuntimeError`), the thread runs until process exit, holding the lock open indefinitely.

The `finally` block in `_acquire_lock` does call `_stop_heartbeat`, so this is partially mitigated — but only for exceptions that propagate through the context manager. The shared-state bug in issue #2 can still cause `_stop_heartbeat` to be a no-op for the wrong thread's heartbeat.
