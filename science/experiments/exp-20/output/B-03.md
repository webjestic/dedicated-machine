Decision: Request Changes

Summary: The implementation contains a critical zombie-write vulnerability rooted in a non-atomic check-then-act lock release pattern, compounded by a race-prone heartbeat design that can silently extend a lock the process no longer legitimately owns. The service also carries a structural concurrency flaw — shared mutable heartbeat state on the instance — that makes it unsafe under any concurrent call pattern. These are not cosmetic issues; they represent correctness failures in the exact scenario this service is designed to handle.

---

## Detailed Review

### 🔴 Critical — Zombie-Write / Fencing Token Absent

```python
# In _acquire_lock finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a non-atomic check-then-act. Between the `GET` and the `DELETE`, another process can acquire the lock. This is the textbook zombie-write scenario: your process believes it still owns the lock, deletes the new owner's lock, and the next writer proceeds concurrently with your still-executing finally block.

The correct fix is a Lua script that atomically checks ownership and deletes:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

The same non-atomic pattern exists in the heartbeat:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # another process can slip in here
```

The heartbeat extension must also be atomic. Use a Lua script or `SET key token XX EX ttl` — but note that `SET ... XX` does not verify the existing value matches your token, so Lua is required here too.

### 🔴 Critical — Shared Mutable State Makes Concurrent Calls Unsafe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal usage pattern for a shared service object — two calls will stomp each other's `_heartbeat_active` flag. Call A stopping its heartbeat will also stop Call B's heartbeat, causing Call B's lock to expire mid-operation.

The fix is to scope heartbeat state to the lock acquisition context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into the thread, never touch self._heartbeat_active
```

### 🔴 Critical — Heartbeat Interval Exceeds Safe Renewal Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires every 10 seconds and renews for 30 seconds. This means the lock can expire up to 10 seconds before renewal fires. Under GC pause, thread starvation, or a slow Redis round-trip, the lock expires and is acquired by another process before the heartbeat runs. The heartbeat then checks the token, finds it still matches (race), and extends a lock now owned by someone else.

A safer ratio is to renew at `TTL / 3` or less, and the TTL itself should reflect your worst-case operation time with margin, not a "generous" guess. For a DB operation, 30 seconds is plausible but the heartbeat at 10s is too coarse.

### 🟡 High — No Fencing Token Propagated to Storage Layer

The lock token (`uuid4`) is acquired and yielded, but `reserve_item` ignores it:

```python
with self._acquire_lock(lock_key):  # token is yielded but discarded
    self.db.update_stock(item_id, new_stock)
```

Without a monotonically increasing fencing token passed through to the storage layer, a zombie process that survived a GC pause can write stale data after its lock expired, even if the lock release is fixed. This is the core of Kleppmann's critique of Redlock. If `db` supports conditional writes (optimistic locking, CAS, version columns), the token or a sequence number should be threaded through.

At minimum, the design should document this limitation explicitly. If the DB layer cannot enforce fencing, that constraint must be surfaced to callers.

---

### 🟡 High — No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, empty `item_id`, or excessively large quantities. A caller passing `quantity=0` or `quantity=-5` will silently pass the stock check (`current_stock < quantity` is false for negative quantities) and write a corrupted order. This is a data integrity issue, not just a style concern.

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

### 🟡 Medium — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally risky component in this PR. Mocking it out means zero test coverage of the renewal logic, the race conditions in the beat loop, or the stop/start lifecycle. The tests pass precisely because the dangerous code is not running.

There should be at least one integration-style test that runs the real heartbeat against a mock Redis and asserts renewal behavior, and a test that verifies the heartbeat stops cleanly on context exit.

---

### 🟡 Medium — `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has stopped. The thread may be mid-sleep when the flag is set, and will execute one more Redis call after `_stop_heartbeat` returns. In the finally block, this means the heartbeat can extend the lock *after* the lock has been released, which is a correctness violation. The fix:

```python
self._heartbeat_active = False
if self._heartbeat_thread:
    self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 🟢 Acknowledged Positives

- Lock ownership verification via token comparison is present (though non-atomic).
- Exponential backoff on contention is correctly implemented.
- `nx=True, ex=TTL` on `SET` is the correct atomic acquire primitive.
- Structured logging with contextual fields is consistent.
- `ReservationResult` as a typed return is clean.
- Daemon thread prevents process hang on unclean shutdown.
