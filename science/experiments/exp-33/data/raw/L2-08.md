**Layer 1 coverage:** Correctness review confirmed two violations — silent heartbeat exit allowing zombie writes after lock loss, and non-atomic idempotency check creating a TOCTOU window — and confirmed both require fixes outside the locking code itself.

---

**Infrastructure failure modes**

**1. Clock skew between worker host and Redis server invalidates TTL assumptions**

*Trigger:* Redis uses its own internal clock to expire keys. The worker sets `CLAIM_TTL = 60` and renews every 15 seconds. If the Redis server's clock is running fast relative to the worker host — or if the Redis server is under memory pressure and performing active expiry with jitter — the key can expire before the worker's renewal interval fires. This is not a theoretical edge case; Redis's active expiry cycle runs in batches and can lag under load, and NTP corrections on either host can produce sudden clock jumps.

*Failure path:* Worker A holds the claim. Redis clock runs ahead by more than the renewal slack. The key expires at wall time T+58 on the Redis side while the worker believes it has until T+60. Worker B acquires the claim at T+58. Worker A's renewal fires at T+60, but the key now belongs to B — the Lua script sees a token mismatch, returns 0, and the heartbeat thread exits silently. Worker A's main thread continues executing `_dispatch()` and proceeds to the database write. Both workers complete the job.

*Required protection:* The write layer — `db.record_execution` and `db.mark_job_complete` — must enforce mutual exclusion independently of the Redis clock. A fencing token (monotonic counter issued at lock acquisition, stored with the job record, and checked with a compare-and-swap at write time) ensures that only the write from the current lock holder is accepted. This cannot live inside the locking code because the locking code has already lost its guarantee by the time the write executes; the locking code cannot retroactively verify what the database accepted.

---

**2. Network partition between worker and Redis causes false lock-hold belief**

*Trigger:* A network partition isolates the worker from Redis for a duration longer than `CLAIM_TTL`. The worker's renewal calls fail — but depending on the Redis client configuration, they may raise exceptions, block, or return stale data from a connection pool. If the renewal call raises an exception rather than returning 0, the heartbeat thread's `result == 0` branch never fires. The thread continues looping. The lock has expired on Redis. Another worker acquires it.

*Failure path:* Worker A is partitioned from Redis at T+0. The renewal call at T+15 raises `redis.exceptions.ConnectionError`. The `_renew()` function has no exception handler — the exception propagates out of `_renew_script(...)`, unwinds the `while` loop, and the thread exits. But the exit is silent — no signal to the main thread. Worker A's main thread is mid-execution in `_dispatch()`. The partition heals. Worker B, which acquired the lock during the partition, is also executing. Both reach the database write. The job executes twice.

*Required protection:* Same fencing token at the write layer. Additionally, the renewal thread must treat connection exceptions as lock-loss events — not as retriable transient errors — because a worker that cannot reach Redis cannot know whether its lock is still held. The renewal thread's exception handling must be explicit: any failure to confirm lock ownership must be treated as loss of ownership. But even with correct thread signaling, the main thread cannot abort an already-in-flight external call (the `_dispatch` side effect). The database write is the last enforceable gate, and it must be the authoritative one.

---

**3. Redis replica promotion during cluster failover invalidates lock state**

*Trigger:* In a Redis Sentinel or Cluster deployment, a primary failure triggers promotion of a replica. Replication in Redis is asynchronous. A lock written to the primary may not have propagated to the replica before the primary fails. After promotion, the new primary has no record of the lock.

*Failure path:* Worker A acquires the claim on the primary. The primary fails 200ms later. The replica — which never received the `SET NX EX` — is promoted. Worker B attempts acquisition on the new primary, succeeds (the key does not exist), and begins executing. Worker A's heartbeat now targets the new primary through Sentinel's transparent redirection. The key exists on the new primary (Worker B's token). Worker A's renewal returns 0, heartbeat exits silently. Worker A's main thread continues. Both workers execute the job.

*Required protection:* This is the Redlock problem. Single-node Redis cannot provide the guarantee this code assumes under failover. The Layer 1 review explicitly deferred Redlock correctness — that deferral must be resolved before production deployment on any topology that includes replication or automatic failover. The protection cannot live in the locking code against this topology; it requires either Redlock across independent nodes, a CP store (etcd, ZooKeeper), or — again — a fencing token at the write layer that makes the lock advisory rather than authoritative.

---

**4. GC pause or slow external call exceeds CLAIM_TTL before first renewal**

*Trigger:* The worker process experiences a stop-the-world GC pause, or `_dispatch()` makes an external call (HTTP, downstream service) that blocks for longer than `CLAIM_TTL = 60` seconds. The renewal thread is also paused during GC. The lock expires. Another worker acquires it.

*Failure path:* This is the canonical zombie-write scenario from the Layer 1 review, now traced at the infrastructure layer. The lock expiry is not a code bug — it is a real-time guarantee that the infrastructure enforces regardless of the code's intent. The heartbeat thread cannot renew a lock it is not scheduled to run. The main thread cannot detect the pause it is experiencing. When execution resumes, the lock is gone, another worker holds it, and the main thread has no mechanism to discover this before writing.

*Required protection:* CLAIM_TTL must be calibrated against observed p99 job durations plus GC pause budgets — but calibration alone is not sufficient, because tail latencies are unbounded in practice. The authoritative protection is a fencing token at the write layer. TTL tuning reduces the probability of the race; it does not eliminate the failure mode. This is explicitly noted as out of scope in the Layer 1 review but must be resolved operationally before this system handles non-idempotent jobs.

---

**5. `_dispatch()` side effects are not bounded by the lock**

*Trigger:* `_dispatch()` calls an external system — sends an alert, charges a payment, triggers a webhook. This call is not atomic with the database write. The lock covers the window from acquisition to release, but the external call happens inside `_dispatch()`, before the database transaction, with no mechanism to roll it back if the subsequent write fails or if lock ownership has been lost by the time the write executes.

*Failure path:* Worker A dispatches the job — the external call succeeds (alert sent, payment charged). Before the database write completes, Worker A's process crashes, or the database transaction fails. The job is not marked complete. On retry, Worker A or Worker B acquires the claim, passes the idempotency check (no completion record exists), and dispatches again. The external action executes twice. The Redis lock provided no protection here because the non-idempotent side effect occurred before the record that would prevent re-execution.

*Required protection:* External side effects that are not idempotent must be made idempotent at the external system boundary — idempotency keys on payment APIs, deduplication tokens on notification services — or the job must be structured so the side effect is the last operation, after the completion record is written. The ordering in this code (`_dispatch()` → `record_execution()` → `mark_job_complete()`) is the wrong order for non-idempotent jobs. This cannot be fixed inside the locking code; it is an architectural ordering constraint.

---

**6. Renewal thread join timeout allows release-before-stop**

*Trigger:* In the `finally` block, `stop_event.set()` signals the renewal thread, then `t.join(timeout=self.RENEWAL_INTERVAL + 1)` waits up to 16 seconds. If the renewal thread is blocked on a slow Redis call — connection timeout not configured, or a Redis server under load — the join times out. The code then proceeds to `_release_script(...)` without confirming the thread has stopped.

*Failure path:* The release script executes and removes the claim key. The renewal thread unblocks from its Redis call, finds the key gone (or finds a new token from a new worker), and returns 0 — silent exit. This is benign in isolation. But if the join timeout fires while the renewal thread is mid-call and the release executes, there is a brief window where the release has fired but the thread has not confirmed stop. In a subsequent re-acquisition scenario on the same key, a delayed renewal call from the zombie thread could theoretically interact with a new token — though the Lua token check prevents actual renewal. The real risk is that the join timeout is set to `RENEWAL_INTERVAL + 1 = 16` seconds, meaning the `finally` block can block for 16 seconds on every execution path, including failure paths. Under load, this creates a thread exhaustion vector.

*Required protection:* Redis client connection timeouts must be configured shorter than `RENEWAL_INTERVAL` to bound the join wait. This is an operational configuration requirement that lives outside the application code — the application cannot enforce it from within.

---

**7. Test suite uses mocked Redis with `return_value=1` — infrastructure failure modes are untested**

*Trigger:* Every test fixture mocks `register_script` to return a mock with `return_value=1`. The renewal path returning 0 is never exercised. Network partition behavior, clock skew, and failover are not represented. The test suite confirms the happy path and the contention path; it does not confirm behavior under any of the failure modes above.

*Failure path:* The gap between tested behavior and production behavior is precisely where infrastructure failure modes live. A test suite that cannot represent `result == 0` from the renewal script cannot catch the silent-exit behavior. This is not a test quality complaint — it is a statement about what the test suite can and cannot certify.

*Required protection:* Integration tests against a real Redis instance (or a Redis test container) with injected TTL expiry, simulated partition via connection drop, and explicit assertions on main-thread behavior when renewal returns 0. These tests cannot be written against mocks because the failure modes are properties of the infrastructure, not the code.

---

**Open gaps**

1. **Fencing token at the write layer** — not present. The database write (`record_execution`, `mark_job_complete`) has no mechanism to reject a write from a worker whose lock has been superseded. This is the primary gap. All of failure modes 1, 2, 3, and 4 converge here.

2. **Redlock / topology contract** — not resolved. The code assumes single-node Redis lock semantics. The deployment topology (Sentinel, Cluster, standalone) is not specified. If the topology includes replication and automatic failover, the lock guarantee is void under failover. This must be explicitly resolved — either by constraining the deployment topology or by adopting a multi-node locking protocol.

3. **Dispatch ordering for non-idempotent jobs** — not resolved. External side effects execute before the completion record is written. For non-idempotent jobs, this ordering guarantees at-least-once execution of the side effect on any failure-and-retry path.

4. **Redis client timeout configuration** — not specified in the application. Connection timeouts, socket timeouts, and retry behavior are not configured in `JobExecutor.__init__`. The renewal thread's behavior under network partition depends entirely on client defaults, which vary by deployment environment.

5. **No integration test coverage for infrastructure failure modes** — the test suite cannot certify behavior under TTL expiry, partition, or failover.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system correctly implements the mechanical structure of a distributed lock — atomic acquisition, token-verified renewal, token-verified release, scoped heartbeat thread. Under nominal conditions with a stable Redis instance and jobs that complete well within TTL, it will prevent concurrent duplicate execution. That is the limit of what it protects.

What it does not protect against is the class of failures where the lock believes it is valid but the infrastructure has already invalidated it: clock skew expiring the key early, a GC pause holding the process past TTL, a network partition preventing renewal, or a Redis failover promoting a replica that never received the write. In every one of these cases, the lock expires or is superseded, a second worker acquires it, and both workers reach the database write with no mechanism at the write layer to determine which one holds current authority. The job executes twice. For a job that sends an alert, charges a payment, or triggers an irreversible external action, this is a production incident.

The fix is not in the locking code. The locking code is structurally correct. The fix is a fencing token — a monotonic counter issued at lock acquisition, stored with the job record, and enforced with a compare-and-swap at write time — so that the database itself rejects the write from any worker whose lock has been superseded, regardless of what the locking layer believes. Until that protection exists at the write layer, the Redis lock is advisory, and this system has no authoritative enforcement against duplicate execution under infrastructure failure.

Additionally: the deployment topology must be specified and the lock protocol must match it; dispatch ordering must be corrected for non-idempotent jobs; Redis client timeouts must be explicitly configured; and integration tests must cover the failure modes that mocks cannot represent.

This does not ship to production handling non-idempotent jobs until the fencing token is in place.
