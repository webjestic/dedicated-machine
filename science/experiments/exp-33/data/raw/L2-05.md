**Layer 1 coverage:** Correctness review confirmed the zombie-write path where the heartbeat thread exits silently on lock loss (`result == 0`) without signaling the main thread, allowing concurrent execution of the same job; secondary findings covered unguarded expiry windows, unhandled Redis exceptions in the renewal thread, and absence of atomic conditional writes at the DB layer.

---

**Infrastructure failure modes**

**1. Redis instance failover during active claim (Sentinel / Cluster promotion)**

*Trigger:* The worker acquires a lock on Redis primary node A. Before the heartbeat next fires, node A fails. Sentinel or Cluster promotes replica B. Because Redis replication is asynchronous, the lock key may not have replicated to B before A died.

*Failure path:* Worker 1 holds a lock that does not exist on the new primary. Worker 2 calls `SET NX EX` against the promoted primary, succeeds — the key is absent — and acquires what it believes is a clean lock. Both workers now hold valid tokens against what they each believe is the authoritative primary. Both proceed through `_dispatch` and the DB transaction. The job executes twice.

*Why the locking code cannot protect against this:* Worker 1's token is real and was legitimately issued. Worker 1's heartbeat will either succeed (if the key happened to replicate) or return 0 (if it didn't), but by the time the heartbeat fires, Worker 2 may already be deep into execution. The failure occurs in the interval between lock acquisition and the next heartbeat cycle — up to 15 seconds. The locking code has no visibility into replication lag or topology change. It cannot distinguish "I still hold the lock" from "I held a lock that no longer exists on the authoritative node."

*Required protection:* Redlock across N independent Redis instances (majority quorum acquisition) is the prescribed mitigation for this class. More practically: if the deployment uses a single Redis primary with Sentinel, the system must document and accept this failure mode explicitly, and the DB layer must provide the final serialization backstop — a unique constraint or conditional write that makes the second `record_execution` / `mark_job_complete` fail atomically. That protection cannot live in the locking code because the locking code's guarantee has already been invalidated by an event it cannot observe.

---

**2. Clock skew between worker and Redis causing premature effective expiry**

*Trigger:* The worker process clock and the Redis server clock diverge. This is not hypothetical — containerized environments, VM live migration, NTP corrections, and leap second handling all produce real clock discontinuities. The `ex=CLAIM_TTL` parameter is evaluated by Redis using Redis's own clock. The worker's renewal interval is driven by `time.sleep` / `threading.Event.wait`, which uses the worker's local clock.

*Failure path:* If the worker's clock runs slow relative to Redis (or Redis's clock jumps forward — e.g., after NTP step correction), the effective TTL on the Redis key expires before the worker's renewal interval elapses. The worker believes it has 15 seconds until the next renewal. Redis has already expired the key after 10 real seconds. Another worker acquires the lock. The original worker's next heartbeat returns 0 and exits — but the main thread is already mid-execution. Concurrent execution proceeds.

*Secondary path:* If the worker's clock runs fast, the renewal fires more frequently than intended — this is benign. The dangerous direction is always worker clock slow / Redis clock fast.

*Why the locking code cannot protect against this:* The renewal interval is computed locally. The code has no mechanism to query Redis's current TTL on the key and compare it against elapsed local time. Even if it did, the comparison would use two clocks that are not synchronized.

*Required protection:* The renewal interval must be set to a fraction of CLAIM_TTL that provides margin for clock skew — the current 15s renewal against 60s TTL provides 4x headroom, which is reasonable but not documented as a deliberate skew budget. More importantly, this reinforces the requirement for DB-layer conditional writes: the locking layer provides probabilistic exclusion; the DB layer must provide deterministic serialization. The protection cannot live in the locking code because the locking code's timing assumptions are built on a clock it does not control.

---

**3. Network partition between worker and Redis (worker-side isolation)**

*Trigger:* A network partition isolates the worker from Redis. The worker cannot reach Redis; Redis cannot receive heartbeat renewals. The lock TTL counts down and expires on the Redis side. Another worker, which can reach Redis, acquires the lock and begins execution.

*Failure path:* Worker 1 is mid-execution inside `_dispatch` — a slow external HTTP call, a large DB query. The network partition lasts longer than CLAIM_TTL (60 seconds). Worker 2 acquires the lock and begins. The partition heals. Worker 1's heartbeat thread attempts renewal — the key now belongs to Worker 2's token. `RENEW_CLAIM_SCRIPT` returns 0. The heartbeat thread exits silently (this is the Layer 1 finding). Worker 1's main thread completes `_dispatch`, enters the DB transaction, and writes. Worker 2 does the same. Duplicate execution.

*Failure path variant — partition shorter than TTL:* The partition lasts 20 seconds. The lock has not yet expired. Worker 1's heartbeat fires after the partition heals, renewal succeeds, execution continues normally. This is the safe case. The dangerous case is specifically partition duration > CLAIM_TTL.

*Why the locking code cannot protect against this:* The worker cannot distinguish "Redis is unreachable" from "Redis is slow." A network timeout on the renewal call is not the same as a renewal returning 0 — but the current code does not handle Redis exceptions in the renewal thread at all (noted in Layer 1). Even with exception handling added, the correct response to a renewal timeout is ambiguous: the worker cannot know whether the key still exists. Conservatively, it should treat a renewal failure as lock loss — but that means any transient network hiccup aborts a legitimate execution. The tradeoff cannot be resolved inside the locking code.

*Required protection:* DB-layer unique constraint on `(job_id, execution_id)` or a conditional `INSERT WHERE NOT EXISTS` on `mark_job_complete`. The locking layer is the first line; the DB constraint is the last line. The partition scenario is precisely the case where the first line fails and the last line must hold.

---

**4. Process suspension longer than CLAIM_TTL (the Layer 1 scenario, infrastructure instantiation)**

*Trigger:* The Layer 1 review identified GC pause as the trigger. The infrastructure-layer instantiation is broader: container live migration (Kubernetes node drain, VM migration), cgroup memory pressure causing the process to be throttled or swapped, CPU starvation under noisy-neighbor conditions on a shared host, or a debugger/strace attach that stops the process.

*Failure path:* Identical to the Layer 1 zombie-write path. The process is suspended for > CLAIM_TTL. The lock expires. Another worker acquires it. The suspended process resumes. Its heartbeat fires, returns 0, exits silently. The main thread continues. Duplicate execution.

*Infrastructure specificity:* This is not a code bug — it is a property of the execution environment. A Kubernetes node drain with a 30-second grace period against a 60-second CLAIM_TTL creates a window. A JVM-style GC pause is the canonical example, but Python's GIL does not prevent OS-level process suspension. `SIGSTOP`, cgroup CPU throttling, and VM migration all produce the same effect from the process's perspective: time passed that the process did not experience.

*Why the locking code cannot protect against this:* The process cannot measure its own suspension. It cannot distinguish "I was running continuously" from "I was suspended for 90 seconds and just resumed." The heartbeat thread's `stop_event.wait(timeout=15)` measures wall-clock time — but only when the process is scheduled. A suspended process does not accumulate wall-clock time in its own timers; it wakes up and the timer fires immediately, but the Redis key has already expired.

*Required protection:* This is the core architectural gap. The fix is a fencing token — a monotonically increasing counter issued by Redis (or a separate sequencer) at lock acquisition time, passed through every downstream write, and enforced at the DB layer via a conditional write that rejects any write carrying a token older than the last accepted token. The token makes the DB write order-dependent on lock acquisition order, not on execution order. A zombie writer carrying an old token is rejected at the DB layer regardless of what the locking layer believes. This protection cannot live in the locking code because the locking code is the component whose guarantee has been violated.

---

**5. Redis memory pressure causing key eviction under maxmemory policy**

*Trigger:* Redis is configured with a `maxmemory` policy that permits eviction of keys (e.g., `allkeys-lru`, `allkeys-random`, `volatile-lru` if the lock key has an expiry). Under memory pressure, Redis evicts the lock key before its TTL expires.

*Failure path:* Worker 1 holds the lock. Redis evicts the key due to memory pressure. Worker 2 calls `SET NX EX`, finds no key, acquires the lock. Both workers are now executing the same job concurrently. Worker 1's next heartbeat returns 0 (key absent — `GET` returns nil, token comparison fails). Heartbeat exits silently. Main thread continues. Duplicate execution.

*Why the locking code cannot protect against this:* The eviction is invisible to the worker. The heartbeat will eventually detect it, but the detection window is up to RENEWAL_INTERVAL (15 seconds). In that window, Worker 2 has acquired the lock and may be well into execution.

*Required protection:* Redis must be configured with `maxmemory-policy noeviction` for any instance used as a locking backend. This is an operational constraint that cannot be enforced from within the application code. It must be part of the deployment specification. The application code has no way to detect that it is running against a Redis instance with an eviction policy that invalidates its correctness assumptions. Additionally, the DB-layer fencing token provides the backstop if eviction occurs despite the operational constraint.

---

**6. Split-brain between DB transaction and Redis claim state at commit time**

*Trigger:* The DB transaction commits successfully (`record_execution` + `mark_job_complete`), but the subsequent `_release_script` call fails (Redis unreachable, timeout, connection reset). Alternatively: the DB transaction fails after `_dispatch` has already produced side effects (external API call, email sent, charge initiated inside the handler).

*Failure path — release failure:* The lock key remains in Redis until TTL expiry. For up to 60 seconds, no other worker can claim the job. This is not a duplicate-execution scenario — it is a availability gap. Acceptable if the job is already marked complete in the DB (the idempotency check will catch it on retry). The `finally` block in `_claim_job` calls `_release_script` but does not handle exceptions from it. A Redis exception here is silently swallowed (Python will suppress exceptions in `finally` if the `try` block completed normally — actually it will propagate, but it will surface as an unhandled exception from the context manager, not as a structured error).

*Failure path — side effects before DB commit:* `_dispatch` executes the handler. The handler sends an alert, charges a payment, calls an external API. The DB transaction then fails (deadlock, constraint violation, connection loss). The `execute` method raises an exception. The job is not marked complete in the DB. On retry, `get_execution` returns None, the job is dispatched again. The external side effect fires twice. This is a correctness violation that is entirely outside the locking layer's scope — but it is a real production failure mode for this system.

*Why the locking code cannot protect against this:* The locking code governs Redis state. It has no visibility into DB transaction outcomes or external side effect atomicity. The gap between "handler executed" and "DB committed" is a window that the lock does not close.

*Required protection:* Idempotency keys on all external calls made within `_dispatch`. The handler must be safe to retry — either by checking prior state before acting, or by passing a stable idempotency key (the `execution_id` is generated before `_dispatch` is called and is available for this purpose, but is not currently threaded into the handler). This protection cannot live in the locking code because it concerns the atomicity of operations that span systems the locking code does not control.

---

**7. Thundering herd on lock expiry under high worker concurrency**

*Trigger:* A long-running job holds the lock for an extended period. The lock expires (due to any of the above failure modes). Multiple workers are waiting. All of them call `SET NX EX` within the same scheduling window.

*Failure path:* The exponential backoff with jitter is applied only on failed acquisition attempts — not on the initial attempt. All workers make their first attempt simultaneously. Only one succeeds, but the contention spike is real. Under high worker counts, this produces a Redis request burst at the moment of expiry. This is a performance concern, not a correctness concern — but under sufficient load, it can cause Redis latency spikes that affect the renewal path for other locks, widening the expiry window for those locks and increasing the probability of failure mode 2 and 4.

*Required protection:* Jitter on initial claim attempt when workers are known to start simultaneously (e.g., triggered by a queue event). This is an operational tuning concern, not a blocking correctness issue.

---

**Open gaps**

1. **No DB-layer fencing token / conditional write.** The most critical unprotected gap. The DB transaction executes unconditionally once `_dispatch` returns. There is no mechanism to reject a write from a worker whose lock was invalidated before the write occurred. This is the gap that makes failure modes 1, 3, 4, and 5 produce incorrect results rather than merely failed operations. A unique constraint on job completion or a compare-and-swap write keyed on a monotonic lock generation counter is required.

2. **No fencing token issued at lock acquisition.** The current token is a UUID — it provides identity, not ordering. A fencing token requires a monotonically increasing value (Redis `INCR` on a generation key, or a DB sequence) so that the DB layer can enforce "only accept writes from the current lock generation." UUID tokens cannot provide this guarantee.

3. **Redis operational constraints not enforced or documented.** The correctness of this system depends on Redis being configured with `maxmemory-policy noeviction` and on replication guarantees that are not specified. These constraints exist outside the application code and will not be caught by any application-layer test. They must be in the deployment specification and validated at startup (e.g., `CONFIG GET maxmemory-policy` check on initialization).

4. **External side effects in `_dispatch` not idempotent by construction.** The system provides no mechanism to pass a stable idempotency key into handlers, and does not document the requirement that handlers be idempotent. For job types that produce external side effects (the PR mentions alerts and payments in its design notes), a DB transaction failure after dispatch produces a duplicate side effect on retry.

5. **Redis exception handling in renewal thread absent.** Noted in Layer 1 but relevant here: a network partition to Redis during renewal produces an unhandled exception in the daemon thread. The thread dies. The main thread continues with no heartbeat and no awareness that renewal has stopped. The lock will expire at TTL. This is failure mode 3 with a faster trigger.

6. **No startup validation of Redis configuration.** The `JobExecutor.__init__` accepts any Redis client without verifying that the backing instance is configured appropriately for use as a locking backend. A misconfigured Redis (wrong eviction policy, replication-only instance, Cluster mode without Redlock) will silently produce incorrect behavior.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system provides correct distributed locking mechanics under nominal conditions — the Lua scripts are sound, the acquisition pattern is correct, and the retry logic is reasonable. It will prevent duplicate execution in the common case where workers are healthy, Redis is stable, and no process is suspended longer than the lock TTL.

It does not protect against the failure modes that matter most in production: Redis failover with replication lag, process suspension beyond TTL (container migration, cgroup throttling, GC), network partition lasting longer than TTL, or Redis key eviction under memory pressure. In every one of these cases, the locking layer's guarantee is silently invalidated by an infrastructure event, the heartbeat thread eventually detects the loss and exits without signaling, and the main thread completes execution and writes to the database. The DB write is unconditional. There is no final serialization backstop.
