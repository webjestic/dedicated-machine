**Layer 1 coverage:** Correctness review confirmed lock acquisition, heartbeat renewal, release atomicity, error handling, idempotency check, and public interface contract — and identified that the heartbeat thread does not signal the main thread on lock loss, and that a fencing token / monotonic counter is required to close the residual race window at the database write.

---

**Infrastructure failure modes**

**1. Process pause exceeds CLAIM_TTL — lock expires mid-execution, second worker acquires, both complete**

*Trigger:* Worker A acquires the claim and begins `_dispatch()`. A GC pause, JVM safepoint, OS scheduler preemption, or slow external I/O call holds the process for longer than `CLAIM_TTL` (60 seconds). The claim key expires in Redis. Worker B calls `SET NX EX` on the same key, succeeds, and begins executing the same job. Worker A resumes. A's heartbeat fires — `RENEW_CLAIM_SCRIPT` evaluates `GET job:claim:{job_id}`, finds B's token, returns 0, and the renewal thread exits silently. A's main thread has no visibility into this. A completes `_dispatch()`, enters `self.db.transaction()`, writes `record_execution` and `mark_job_complete`. B does the same. The job executes twice. For a payment charge, alert dispatch, or external API call, this is a correctness violation.

*Why the protection cannot live inside the locking code:* The locking code operates on the assumption that time inside the process is bounded. It is not. The process can be paused by the runtime or OS for an arbitrary duration that no application-level timer can observe or prevent. By the time the heartbeat fires and detects token mismatch, the damage — the second worker acquiring and beginning execution — has already occurred. The heartbeat is a detection mechanism, not a prevention mechanism. Even with Layer 1's `stop_event` fix applied, A's main thread can only be stopped *after* the pause ends and the heartbeat fires. If A's pause ends and A reaches `db.transaction()` before the next heartbeat interval, A writes without any check.

*Required protection:* The database write must carry a fencing token — a monotonic, globally ordered value (Redis INCR on acquisition, or a database sequence) that is written into the execution record and checked under a database-level constraint or compare-and-swap. The database, not the lock, is the final arbiter of which write wins. This is the only layer that can enforce the invariant after the lock guarantee has been silently invalidated by time. The database constraint must be: "record this execution only if no execution record for this job_id already exists" — enforced atomically at the database level, not as a pre-check read followed by a write.

---

**2. Redis primary failure during the claim window — failover promotes a replica that has not replicated the claim key**

*Trigger:* Worker A acquires the claim via `SET NX EX` against the Redis primary. The write is acknowledged to A. Before the claim key replicates to any replica, the primary fails. Redis Sentinel or Cluster promotes a replica. The promoted replica has no record of the claim key. Worker B now calls `SET NX EX` on the promoted primary, succeeds, and acquires what it believes is a legitimate claim. Both A and B hold tokens they believe are valid. Both proceed to execute.

*Why the protection cannot live inside the locking code:* Redis replication is asynchronous by default. `SET NX EX` returning success guarantees the write reached the primary's memory — it does not guarantee the write has been replicated. The locking code has no mechanism to observe replication lag or failover events. The claim it holds is a promise from a node that may no longer exist. The code cannot distinguish "I hold a valid claim" from "I held a valid claim on a node that is now gone."

*Required protection:* At the Redis layer: enable `WAIT numreplicas timeout` after acquisition to confirm replication before proceeding, accepting the latency cost and the possibility of blocking on replica unavailability. Alternatively, use Redlock across independent Redis instances — though Redlock has its own documented failure modes under clock drift and network partition that must be evaluated against this workload's tolerance. At the database layer: the same fencing token / unique constraint protection from failure mode 1 applies here and closes this window regardless of which Redis topology decision is made. The database write is the last line of defense.

---

**3. Clock skew between worker and Redis — TTL expires earlier than the worker expects**

*Trigger:* `CLAIM_TTL` is 60 seconds. `RENEWAL_INTERVAL` is 15 seconds. The design assumes the renewal fires with enough margin before expiry. If the worker's system clock is running fast relative to Redis's internal TTL countdown — or if the worker process is scheduled with high latency by the OS — the effective window before expiry is shorter than 15 seconds from the worker's perspective. Under sustained CPU contention or a cgroup CPU throttle event, the renewal thread may not be scheduled for 20, 30, or more seconds. The claim expires. The failure path from here is identical to failure mode 1.

*Why the protection cannot live inside the locking code:* The renewal thread uses `stop_event.wait(timeout=RENEWAL_INTERVAL)` — a wall-clock sleep from the worker's perspective. The worker cannot observe how much of the Redis TTL has actually elapsed. It cannot observe its own scheduling latency. It cannot know that 15 seconds of `wait()` consumed 25 seconds of Redis TTL. The gap between "time the worker believes has passed" and "time Redis has counted against the TTL" is invisible to the application.

*Required protection:* Reduce the ratio of `RENEWAL_INTERVAL` to `CLAIM_TTL` — a 4:1 ratio (15s interval on a 60s TTL) is marginal under CPU contention. A 6:1 or 8:1 ratio provides more margin. More importantly: after each renewal, the worker can call `TTL job:claim:{job_id}` to verify the remaining TTL is within expected bounds. Any anomaly — TTL missing, TTL significantly below expected — should be treated as lock loss. This is a defense-in-depth measure. The database-layer fencing token remains the required protection.

---

**4. Network partition between worker and Redis — renewal succeeds from the worker's perspective but the command never reached Redis**

*Trigger:* Worker A holds the claim. A transient network partition between the worker host and Redis causes the `RENEW_CLAIM_SCRIPT` call to block, then raise a connection exception — or, in a more dangerous variant, to appear to succeed (the TCP connection is accepted by a load balancer that then drops the backend connection) while the command never executes. If the exception path is not handled in `_renew()`, the thread may exit. If the command silently fails, the TTL is not extended and the claim expires. The failure path is identical to failure mode 1.

*Why the protection cannot live inside the locking code:* The renewal thread in this implementation has no exception handling around `self._renew_script(...)`. A `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError` will propagate uncaught, terminate the thread, and leave the main thread executing without renewal — which is the silent exit failure mode identified in Layer 1. Even with Layer 1's exception handling fix applied, the worker cannot distinguish "renewal failed because the network is partitioned" from "renewal failed because the lock was stolen." Both are lock loss from the worker's perspective, and both require the same response: halt execution before the next side-effecting operation.

*Required protection:* Layer 1's required fix — exception handling in the renewal thread with `stop_event` signaling — is the correct response at the locking layer. The database-layer fencing token closes the residual window. Additionally: the Redis client must be configured with explicit socket timeouts (`socket_timeout`, `socket_connect_timeout`) so that network hangs produce timely exceptions rather than indefinite blocking that delays renewal thread exit and main thread notification.

---

**5. Redis command timeout on claim release — release silently fails, claim persists until TTL expiry**

*Trigger:* In the `finally` block, `self._release_script(keys=[claim_key], args=[token])` is called. If the Redis connection is unavailable or the command times out, the exception propagates out of the `finally` block — or is swallowed depending on exception handling — and the claim key remains in Redis until `CLAIM_TTL` expires. For 60 seconds, no other worker can claim the job. This is an availability failure, not a correctness failure, but in a job processing system with SLA requirements it is operationally significant.

*Why the protection cannot live inside the locking code:* The `finally` block executes after the job has completed successfully. The work is done. The release failure does not produce a duplicate execution — it produces a claim that outlives its usefulness. The locking code cannot force Redis to accept the delete. It can only retry.

*Required protection:* Retry logic on release with a short timeout and a bounded retry count. Log the failure with the claim key and TTL so operators can observe stuck claims. This is an operational concern that must be addressed at the deployment layer: monitoring on claim key age, alerting when claim keys persist beyond `2 * CLAIM_TTL`, and runbook procedures for manual claim release. The 60-second TTL is the backstop — it must be treated as a known recovery window, not an edge case.

---

**6. Worker process killed between `_dispatch()` completion and `db.transaction()` commit — job executed but not recorded**

*Trigger:* Worker A completes `_dispatch()` — the external action has occurred (payment charged, alert sent). Before `self.db.transaction()` commits, the worker process is killed by OOM killer, a deployment rollout, or a SIGKILL. The job has executed but `mark_job_complete` was never written. The claim key expires. Worker B acquires the claim, reads `get_execution()`, finds no record, and executes the job again.

*Why the protection cannot live inside the locking code:* The locking code cannot survive its own process death. The claim key will expire regardless. The idempotency check reads from the database — and the database has no record of the first execution because the transaction never committed.

*Required protection:* The idempotency guarantee must be pushed into `_dispatch()` itself, or into the external systems that `_dispatch()` calls. If the job sends a payment, the payment provider must support idempotency keys. If the job sends an alert, the alert system must deduplicate on job_id. The database record is a record of the framework's knowledge — it is not a guarantee that the external action did not occur. This is an architectural constraint that must be documented and enforced at the handler registration layer: every handler registered via `db.get_handlers()` must be idempotent with respect to its external effects. This cannot be enforced by the executor. It must be enforced by contract.

---

**Open gaps**

1. **No fencing token on the database write.** The `db.record_execution()` and `db.mark_job_complete()` calls carry no monotonic counter or globally ordered token from the claim acquisition. There is no database-level constraint that can distinguish a legitimate write from a write made after lock expiry. This is the primary open gap. It is not closeable from within the locking code.

2. **No exception handling in the renewal thread.** `_renew_script(...)` is called without a try/except. Any Redis exception terminates the thread silently. Layer 1 identified this; it is not yet fixed in this artifact.

3. **No `stop_event` signaling from the renewal thread to the main thread.** When `result == 0`, the thread returns without setting `stop_event`. The main thread continues executing. Layer 1 identified this; it is not yet fixed in this artifact.

4. **Redis client has no configured socket timeouts.** The constructor accepts a `redis.Redis` instance with no enforcement of `socket_timeout` or `socket_connect_timeout`. A hung Redis connection will block the renewal thread indefinitely, delaying detection of lock loss.

5. **No replication confirmation on claim acquisition.** `SET NX EX` success is taken as a sufficient guarantee. Under async replication with failover, this guarantee does not hold.

6. **Handler idempotency is unenforced.** The executor assumes that if `get_execution()` returns None, it is safe to execute. This assumption fails if the process was killed after dispatch but before the transaction committed. There is no mechanism to enforce or verify that registered handlers are idempotent with respect to their external effects.

7. **No operational monitoring for stuck claims.** There is no instrumentation, alerting, or runbook coverage for claim keys that persist beyond their expected lifetime due to release failures.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system implements a recognizable distributed locking pattern and gets several things right: atomic acquisition, token-verified release, Lua-based renewal, and a database transaction wrapping the completion record. Under nominal conditions — no pauses, no failures, no topology events — it will prevent duplicate execution. The problem is that nominal conditions are not the conditions that produce incidents.

The two gaps identified in Layer 1 are present and unresolved: the renewal thread exits silently on lock loss without signaling the main thread, and there is no fencing token at the database write. These are not theoretical. A GC pause, a CPU throttle event, or a slow external call in `_dispatch()` will hold the process past `CLAIM_TTL`, the lock will expire, a second worker will acquire it, and both workers will write to the database. The database transaction provides no protection against this because both workers will find `get_execution()` returning None — the check is a pre-condition read, not an atomic compare-and-swap.

Before this ships, the following must be in place: the renewal thread must signal `stop_event` on both `result == 0` and on any Redis exception, with exception handling added around the renewal call; the main thread must check `stop_event` before `_dispatch()` and before `db.transaction()`; and the database write must be protected by a unique constraint on `job_id` in the execution record table, enforced at the database level, so that the second writer's transaction fails rather than succeeds silently. The fencing token approach — incrementing a counter on claim acquisition and writing it into the execution record — provides the ordering guarantee that makes the database constraint meaningful.

The Redis replication gap and the handler idempotency gap are architectural constraints that must be explicitly accepted or addressed at the deployment and integration layer before this system handles jobs with non-idempotent external effects. They cannot be resolved in this code. They must be resolved in the systems this code depends on, and that resolution must be verified before this executor is used for payments, alerts, or any action that cannot be safely repeated.
