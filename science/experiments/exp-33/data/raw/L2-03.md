**Layer 1 coverage:** Correctness review confirmed lock acquisition/renewal/release atomicity, heartbeat-to-main-thread signaling, idempotency scope, Redis unavailability handling, and test coverage — with the central finding that silent heartbeat exit plus absent fencing on database writes permits duplicate execution under process pause; threading signals alone are insufficient.

---

**Infrastructure failure modes**

**1. Redis Cluster Failover During Active Claim**

*Trigger:* The Redis primary fails mid-execution. A replica is promoted. The promotion takes 10–30 seconds under default Sentinel or Cluster configuration. During that window, all Redis operations fail. After promotion, the replica's view of key state may be behind the primary's — any key written in the last replication cycle before failure may not exist on the new primary.

*Failure path:* Process A holds a claim. Primary fails. The new primary has no record of the claim key (replication lag ate it). Process B calls `SET NX EX` against the new primary, succeeds, and begins executing the same job. Process A's heartbeat is failing against a dead primary — it may be retrying, sleeping, or throwing exceptions depending on the Redis client's retry configuration. Process A's main thread is unaffected; it continues executing. Both processes complete the job. The database transaction has no awareness that the claim was invalidated by a topology event. Both writes succeed if the idempotency check races or if the DB uniqueness constraint is absent.

*Required protection:* The database layer must enforce a unique constraint on `job_id` in the execution record table, making one of the two concurrent writes fail at commit. This cannot live in the locking code because the locking code's guarantee — exclusive access — was destroyed by an infrastructure event the lock manager has no visibility into. The lock code believes it holds a valid claim. It is wrong. Only the database, which is not subject to the Redis failover, can enforce the final exclusivity guarantee. Additionally, if this system uses Redlock across multiple Redis nodes, that introduces its own set of clock-dependent failure modes (see item 3); if it uses a single Redis instance, that instance is a single point of failure for claim exclusivity with no fallback.

---

**2. GC Pause / OS Scheduler Preemption Exceeding CLAIM_TTL**

*Trigger:* Process A is paused — JVM-style GC, Python GIL contention under load, cgroup CPU throttling in a Kubernetes pod, or a slow external call inside `_dispatch` — for longer than 60 seconds. This is not a theoretical bound. A Python process under memory pressure with a large heap, or a pod being throttled by a noisy neighbor, can pause for minutes.

*Failure path:* Process A holds the claim. The 60-second TTL expires. Process B acquires the claim, begins executing. Process A resumes. Process A's heartbeat fires — the renewal script checks the token, finds a different token (Process B's), returns 0, and the heartbeat thread exits silently. Process A's main thread has no visibility into this. It proceeds to `_dispatch`, completes, and enters the database transaction. The idempotency check (`db.get_execution`) runs — if Process B has not yet committed, it returns None. Both processes pass the idempotency check. Both execute the job. Both attempt to write. Without a database-level unique constraint, both writes succeed.

*Required protection:* Two independent protections are required and neither can substitute for the other. First, the heartbeat thread must signal the main thread on token mismatch — a `threading.Event` or equivalent — so the main thread can abort before reaching `_dispatch`. This is the Layer 1 finding. Second, and independently necessary: the database write must carry a fencing token (the claim token, a monotonic counter, or an optimistic concurrency version) as a condition of the write. The database rejects any write whose token does not match the last known valid claim for that job. This protection cannot live in the locking code because by the time the main thread reaches the database write, the locking code's guarantee has already been silently invalidated. The main thread does not know this. The database write is the last point where the system can enforce "only the current legitimate holder of the claim may record completion." Without this, the threading signal is necessary but not sufficient — a race between the signal and the write, or a failure of the signal path itself, still produces duplicate execution.

---

**3. Clock Skew Between Workers Affecting TTL Semantics**

*Trigger:* Worker nodes have divergent system clocks. This is routine in cloud environments without enforced NTP discipline — 100–500ms of drift is common; multi-second drift occurs after NTP corrections, VM live migrations, or hibernation/resume cycles. Redis TTL expiry is computed against Redis's own clock, but the workers' decisions about when to renew are computed against their local clocks.

*Failure path:* Process A is on a worker whose clock is 20 seconds behind Redis's clock. CLAIM_TTL is 60 seconds. Process A's renewal interval is 15 seconds. From Process A's perspective, it renews at t=15, t=30, t=45 — well within the 60-second window. From Redis's perspective, the key was set at t=0 (Redis time), expires at t=60 (Redis time). Process A's first renewal fires at what Process A believes is t=15, but Redis's clock reads t=35. The second renewal fires at what Process A believes is t=30, but Redis reads t=50. The third renewal fires at what Process A believes is t=45, but Redis reads t=65 — the key has already expired. The renewal returns 0. The heartbeat exits. The main thread continues. Process B acquires the claim. Duplicate execution follows the same path as failure mode 2.

*Required protection:* The renewal interval must be set as a fraction of TTL that accounts for maximum expected clock skew plus network round-trip time to Redis, not just a comfortable fraction of TTL. More importantly, this is another instance where the database fencing token is the only reliable backstop — because the clock skew that caused the premature expiry is invisible to the locking code. The locking code believes it renewed successfully on the previous interval. It has no way to know that Redis's clock disagreed. The protection must live at the database write layer for the same reason as failure mode 2.

---

**4. Network Partition Between Worker and Redis (Partial)**

*Trigger:* A partial network partition — the worker can reach the database but not Redis. This is not symmetric. Cloud network failures are frequently asymmetric: a misconfigured security group change, a routing table update, or a flapping NIC can isolate a worker from Redis while leaving its database connection intact.

*Failure path:* Process A holds a claim. A network partition isolates Process A from Redis. The heartbeat fires, the renewal call blocks, times out (depending on Redis client socket timeout configuration — which is not set in this code), or raises a connection error. The current `_renew` function has no exception handling. An unhandled exception in the heartbeat thread causes the thread to exit. The main thread continues. Meanwhile, the claim TTL has expired (Process A has been unable to renew). Process B acquires the claim and begins executing. Process A completes `_dispatch` and attempts the database write. The database is reachable. The write succeeds. Process B also completes and writes. Duplicate execution.

*Additionally:* The Redis client socket timeout is not configured in this code. Under a partition, renewal calls may block indefinitely, causing the heartbeat thread to hang rather than exit — which means `stop_event.set()` in the finally block will block on `t.join(timeout=self.RENEWAL_INTERVAL + 1)` for 16 seconds before the join times out. The release script call then fires against a partitioned Redis — it may also block. The finally block is not a clean exit under partition conditions.

*Required protection:* Redis client must be initialized with explicit socket connect timeout and socket timeout values. The heartbeat thread must catch Redis exceptions and treat them as loss-of-claim events (conservative: if you cannot renew, assume the claim is lost). The main thread must receive the abort signal. And again, the database fencing token is the terminal protection — because the partition that prevented renewal is invisible to the main thread's execution path.

**5. Rolling Deployment — Old and New Worker Versions Running Concurrently**

*Trigger:* A deployment is in progress. Kubernetes rolling update, ECS service update, or any blue/green cutover where both old and new worker versions are running simultaneously against the same Redis instance and the same database.

*Failure path:* If a schema migration changes the claim key format, the token format, the Lua script behavior, or the database schema for execution records, old and new workers may have incompatible views of claim state. Specifically: if the new version changes `job:claim:{job_id}` to a different key structure, old workers and new workers are not competing for the same key — they each believe they hold an exclusive claim because they are looking at different keys. Both execute the same job. The database uniqueness constraint (if present) catches this. If it is absent, both writes succeed.

*Additionally:* If the new version changes the Lua scripts, the registered script hash changes. Old workers have the old script registered; new workers have the new script registered. This is generally safe for Lua scripts since Redis evaluates them by hash — but if a script is evicted from Redis's script cache (possible under memory pressure with `SCRIPT FLUSH` or cache eviction), the `register_script` call in `__init__` pre-loads the script, but the script object calls `EVALSHA` at runtime. If the script is not in cache, Redis returns a `NOSCRIPT` error. This is not handled. The renewal call raises an unhandled exception, the heartbeat thread exits, and the failure path from mode 4 applies.

*Required protection:* Deployment runbooks must define the overlap window and verify claim key format compatibility across versions before rollout. The NOSCRIPT error must be caught and handled — either by falling back to `EVAL` or by treating it as a renewal failure. This cannot be handled inside the locking code's current exception model because the current code has no exception handling in the heartbeat thread at all.

---

**6. Redis Memory Pressure — Key Eviction**

*Trigger:* Redis is configured with a `maxmemory` policy that permits key eviction (e.g., `allkeys-lru`, `volatile-lru`, `volatile-ttl`). Under memory pressure, Redis may evict the claim key before its TTL expires.

*Failure path:* Process A holds a claim. Redis evicts `job:claim:{job_id}` under memory pressure. Process B calls `SET NX EX`, finds no key, acquires the claim. Process A's heartbeat fires — the renewal script calls `GET`, finds no key, returns 0 (the `if` condition fails because `GET` returns nil, which does not equal the token). The heartbeat exits silently. The main thread continues. Duplicate execution follows.

*Required protection:* Redis must be configured with `noeviction` or a policy that does not evict keys with TTLs (`volatile-*` policies only evict keys with TTLs set — which includes claim keys — so `volatile-lru` is not safe here). This is an operational configuration requirement that cannot be enforced from within the application code. The database fencing token remains the terminal protection. This must be documented as a hard Redis configuration requirement for this service.

---

**7. Idempotency Check — Read-Your-Writes Consistency Gap**

*Trigger:* The idempotency check (`db.get_execution`) runs against a database read replica, or against a primary that has not yet flushed a concurrent commit from Process B due to transaction isolation level.

*Failure path:* Process B completes the job and commits `record_execution` + `mark_job_complete`. Process A, running concurrently after a lock race (any of the above failure modes), calls `db.get_execution`. If this read hits a replica with replication lag, or if the database isolation level is READ COMMITTED and Process B's transaction has not yet been visible, `get_execution` returns None. Process A concludes the job has not been executed. Process A executes the job and writes. Duplicate execution.

*Required protection:* The idempotency check must read from the primary with a consistency level that guarantees visibility of any committed write. Read replicas must not be used for this check. The database transaction that writes execution records must use a serializable isolation level or an explicit row-level lock on the job record to prevent two concurrent transactions from both passing the idempotency check and both committing. This cannot be enforced from within the locking code — it is a database transaction design requirement.

**Open gaps**

1. **No fencing token on database writes.** The database write carries no condition that verifies the writing process still holds a valid claim at write time. This is the primary unresolved gap from the Layer 1 review, and every infrastructure failure mode above converges on it as the terminal protection point. It is not present.

2. **No exception handling in the heartbeat thread.** Redis connection errors, timeout errors, and NOSCRIPT errors all cause unhandled exceptions that silently exit the thread. The main thread receives no signal. This is distinct from the `result == 0` path — it is the exception path, and it is completely unhandled.

3. **No Redis client timeout configuration.** Socket connect timeout and socket timeout are not set. Under partition or Redis unavailability, blocking calls in the heartbeat thread and in the finally block will hang for the OS default TCP timeout (minutes). This makes the finally block unreliable as a cleanup path.

4. **No threading abort signal from heartbeat to main thread.** The Layer 1 finding. Not addressed in this artifact. The `stop_event` flows only from main thread to heartbeat — not the reverse.

5. **Redis configuration requirements not documented or enforced.** `noeviction` or a non-volatile eviction policy is required. Single-instance Redis is a single point of failure for claim exclusivity. These are hard operational requirements with no enforcement mechanism and no documentation in the artifact.

6. **Idempotency check does not guarantee read-your-writes consistency.** The `db.get_execution` call has no specified consistency requirement. If the database layer routes reads to replicas, this check is unreliable.

7. **No unique constraint enforcement visible in the artifact.** `db.record_execution` and `db.mark_job_complete` are abstracted behind a mock. There is no evidence of a unique constraint on `job_id` in the execution table. Without it, concurrent writes from two workers both pass silently.

8. **Rolling deployment compatibility is unspecified.** No deployment runbook, no version compatibility contract, no claim key format versioning.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system implements the correct shape of a distributed job lock — atomic acquisition, token-verified renewal and release, scoped heartbeat, database-level idempotency check, transactional write. Under nominal conditions, with a stable Redis instance, no process pauses, and no concurrent failures, it will behave correctly. That is not the production condition.

What this system does not protect against is the class of failure where the locking mechanism believes it holds a valid claim but the underlying guarantee has been silently invalidated — by TTL expiry during a process pause, by Redis failover, by clock skew, by key eviction, by a network partition that the heartbeat thread cannot survive. In every one of these cases, the main thread continues executing with no knowledge that its claim is gone, and the database write succeeds because nothing at the database layer checks whether the writing process is still the legitimate claim holder. The Layer 1 review named this gap. This infrastructure review confirms it is load-bearing across at least six independent failure triggers, not one.

Before this ships, the following must be in place: a fencing token carried through to the database write as an optimistic concurrency condition (the database rejects writes from a process whose claim token is no longer current); exception handling in the heartbeat thread with abort signaling to the main thread on any failure, not just `result == 0`; Redis client socket timeouts configured explicitly; a unique constraint on `job_id` in the execution table as a final backstop; documented and enforced Redis configuration requirements (`noeviction`, replication topology, failover behavior); and a read consistency guarantee on the idempotency check. The threading abort signal is necessary but remains insufficient without the database fencing condition — a race between the signal and the write, or any failure of the signal path, still produces the duplicate execution this system was designed to prevent.
