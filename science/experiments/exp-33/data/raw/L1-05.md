**Findings**

---

**1. Silent Heartbeat Exit Does Not Stop Job Execution (Critical — Zombie Write)**

**Issue name:** Undetected lock loss — main thread continues after heartbeat silently exits on token mismatch.

**Failure path:**

1. Process A acquires `job:claim:{job_id}` with token `T1`, starts execution, heartbeat thread starts.
2. A GC pause, slow external call in `_dispatch`, or slow DB operation holds Process A for longer than `CLAIM_TTL` (60s).
3. The lock expires. Process B calls `SET NX EX`, acquires the same key with token `T2`, and begins executing the same job.
4. Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns `T2`, which does not match `T1`. The script returns `0`.
5. `_renew()` sees `result == 0` and executes `return` — the thread exits silently. No signal is sent to the main thread.
6. Process A's main thread is still inside `with self._claim_job(...)`. It has no visibility into the heartbeat thread's exit. It proceeds through `_dispatch`, into `self.db.transaction()`, calls `record_execution` and `mark_job_complete`.
7. Process B does the same. The job executes twice. For any non-idempotent side effect — payment charge, alert send, external API call — this is a correctness violation.

The checklist entry "Atomic claim renewal via Lua script (token-verified)" is accurate but incomplete: it describes the mechanism, not the behavior on failure. The mechanism works; the failure path is that the result is not acted upon.

**Required fix — minimum (Tier 0.5):** Signal the main thread on lock loss. Add a `lost_event = threading.Event()` alongside `stop_event`. In `_renew`, on `result == 0`, set `lost_event` before returning. In the `try` block inside `_claim_job`, after `yield`, check `lost_event` and raise a `ClaimError` or a dedicated `LockLostError`. This is necessary but not sufficient — see Finding 2.

**Required fix — complete (Tier 1.0):** `threading.Event` signaling closes the gap only for the period *after* the heartbeat fires. There is a window between lock expiry and the next heartbeat tick (up to `RENEWAL_INTERVAL = 15s`) during which Process A's main thread continues executing with no lock and no signal. The correct architectural fix is a **fencing token / monotonic counter**: Redis returns a version number at lock acquisition time; every write to the database or external system includes that token; the DB or downstream system rejects writes with a stale token. This makes the writes themselves safe regardless of when lock loss is detected. Alternatively, use optimistic concurrency control on the job record (e.g., a `version` column with a conditional update) so that the second writer's `mark_job_complete` fails if the first already committed. The `threading.Event` approach is a detection improvement; fencing is the correctness guarantee.

---

**2. Lock Loss Window Between Expiry and Next Heartbeat Tick**

**Issue name:** Unguarded execution window — lock can expire and be re-acquired by another worker up to `RENEWAL_INTERVAL` seconds before the heartbeat detects it.

**Failure path:** Even if Finding 1 is fixed with `lost_event` signaling, the heartbeat only fires every 15 seconds. If the lock expires at T=0 and Process B acquires it at T=1, Process A's heartbeat does not fire until T=15. For 14 seconds, both processes are executing the job concurrently with no detection. The `threading.Event` fix reduces the blast radius after detection; it does not close this window.

**Required fix:** Same as Finding 1 Tier 1.0 — fencing token or optimistic concurrency on the write path. The DB transaction in `execute` must be conditioned on the job not already being marked complete by another writer, with the condition enforced atomically inside the transaction (e.g., `UPDATE jobs SET status='complete', version=version+1 WHERE job_id=? AND status='pending' AND version=?`). If the update affects zero rows, the transaction should roll back and the executor should treat this as a duplicate execution.

---

**3. Claim Release Executes After Lock Has Already Expired**

**Issue name:** Stale release — `_release_script` in `finally` block may attempt to delete a key owned by another worker.

**Failure path:** After lock expiry and re-acquisition by Process B (token `T2`), Process A's `finally` block runs `_release_script(keys=[claim_key], args=[T1])`. The Lua script correctly checks the token: `GET` returns `T2`, mismatch, returns `0`, no delete. This is handled correctly by the existing script.

**Verdict on this finding:** The Lua script's token check prevents the unconditional delete. This specific sub-issue is correctly implemented. It does not mitigate Findings 1 and 2 — the release is safe, but the execution that preceded it was not.

---

**4. Heartbeat Thread Exception Handling**

**Issue name:** Unhandled exception in `_renew` silently terminates the heartbeat.

**Failure path:** If `self._renew_script(...)` raises (e.g., `redis.exceptions.ConnectionError`, `redis.exceptions.TimeoutError`), the exception propagates out of `_renew`, the thread terminates, and the claim is no longer being renewed. The main thread continues executing. The lock will expire at the next TTL boundary. This produces the same zombie-write scenario as Finding 1, triggered by Redis unavailability rather than a GC pause.

**Required fix:** Wrap the script call in `_renew` in a `try/except` block. On transient Redis errors, log and continue the renewal loop (the lock has not necessarily been lost). On persistent failure or on detecting `result == 0`, signal the main thread via `lost_event` (per Finding 1 fix). Distinguish between "Redis unreachable" (uncertain lock state) and "token mismatch" (lock definitively lost).

---

**5. `_dispatch` Exception Bypasses `mark_job_complete` but Does Not Release Cleanly**

**Issue name:** Exception in `_dispatch` propagates through `execute` uncaught (except for `ClaimError`), leaving the job in an indeterminate state.

**Failure path:** `_dispatch` raises `ValueError` for unknown job types. The test `test_execute_unknown_job_type` asserts `pytest.raises(ValueError)` — confirming this propagates to the caller. The `finally` block in `_claim_job` will still release the claim (correctly). However, `mark_job_complete` is never called, so the job remains in its prior state. On retry, `get_execution` returns `None`, and the job will be re-attempted. This is the correct behavior for a failed job, but it is not documented and the caller receives an unhandled exception rather than a `JobResult`. Depending on the contract expected by callers, this may be a correctness issue.

**Required fix (conditional):** If `execute` is expected to always return a `JobResult`, catch `ValueError` (and other dispatch exceptions) inside `execute`, log them, and return `JobResult(claimed=False, message="Job execution failed: ...")`. If raising is intentional, document it explicitly in the docstring and ensure callers handle it. The current docstring does not mention this behavior.

---

**6. `get_execution` / `mark_job_complete` Not Within the Same Atomic Scope**

**Issue name:** TOCTOU on idempotency check — the "already completed" check and the write are not atomic.

**Failure path:** Two workers both pass `get_execution` returning `None` (neither has committed yet), both proceed to `_dispatch`, both enter `self.db.transaction()`. The database transaction and `mark_job_complete` provide the final serialization point — but only if the DB enforces a uniqueness or conditional constraint on `job_id`. If `mark_job_complete` is an unconditional `UPDATE` or `INSERT`, both transactions may commit. The Redis lock is the primary guard, but as established in Findings 1 and 2, the lock can be held by two processes simultaneously under pause conditions.

**Required fix:** `mark_job_complete` (or `record_execution`) must use a conditional write — unique constraint on `job_id` in `record_execution`, or a conditional update in `mark_job_complete` that fails if the job is already marked complete. The DB transaction must check the affected row count and treat zero rows as a duplicate. This is the same architectural requirement as Finding 2 — the DB write must be the authoritative serialization point.

---

**Correctness verdict: NOT APPROVED**

**Handoff summary for production readiness review:**

This review traced the full lock acquisition, heartbeat renewal, and release path under concurrent execution, with specific attention to process-pause and GC-pause scenarios. The critical finding is that the heartbeat thread exits silently on lock loss (`result == 0` from `RENEW_CLAIM_SCRIPT`) without signaling the main thread, allowing two workers to execute the same job concurrently; this is a confirmed zombie-write path. Secondary findings cover the unguarded execution window between lock expiry and heartbeat detection, unhandled Redis exceptions in the renewal thread, and the absence of atomic conditional writes at the DB layer to serve as a final serialization backstop. The Lua scripts for release and renewal are themselves correctly implemented. This review does not cover infrastructure reliability, Redis cluster failover behavior, or operational deployment concerns.
