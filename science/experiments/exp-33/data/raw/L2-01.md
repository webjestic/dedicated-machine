**Layer 1 coverage:** Correctness review confirmed the structural lock-loss gap (heartbeat exits silently, main thread has no signal path), required threading.Event notification plus fencing token for downstream writes, and flagged unhandled Redis exceptions at acquisition/renewal and missing lock-loss test coverage.

**Infrastructure failure modes**

**1. Clock skew between worker and Redis causes premature lock expiration**

*Trigger:* Redis uses its own internal clock to evaluate TTLs. The worker sets `ex=CLAIM_TTL` (60s) at acquisition time, but the wall-clock elapsed time the worker experiences is not the same as what Redis measures. In a VM or container environment, the guest clock can drift relative to the Redis host clock — particularly after a live migration, a hypervisor scheduling stall, or NTP correction. A 60-second TTL on Redis may expire in 55 seconds of worker-perceived time, or the worker may believe it has 10 seconds remaining when Redis has already evicted the key.

*Failure path:* Worker A acquires the lock. Redis clock runs ahead of worker A's clock by several seconds (or worker A's clock is corrected backward by NTP mid-execution). Worker A's heartbeat fires at 15-second intervals measured by `stop_event.wait(timeout=15)` — but that timer is driven by the worker's own clock. If the worker clock is slow relative to Redis, the 60-second TTL expires before the 15-second heartbeat fires. Redis evicts the key. Worker B acquires the lock. Worker A's next heartbeat fires, `RENEW_CLAIM_SCRIPT` returns 0 (key gone or token mismatch), heartbeat exits silently. Worker A continues executing. Both workers complete the job.

*Why the protection cannot live inside the locking code:* The locking code has no access to Redis's internal clock. It cannot observe the delta between its own `time.sleep` and Redis TTL consumption. The worker cannot distinguish "I have 10 seconds left" from "Redis already expired this 2 seconds ago." The gap is in the measurement instrument, not the logic.

*Required protection:* The renewal interval must be set to a fraction of TTL that accounts for worst-case clock drift — not just network latency. A renewal interval of TTL/4 (15s for a 60s TTL) is marginal; TTL/4 with an explicit safety margin for known drift bounds is the minimum. More importantly: the downstream write path must carry a fencing token (the monotonic Redis counter or a versioned conditional write) so that even if two workers reach the write, only the one holding the current lock generation can commit. This protection lives at the storage layer because that is the only layer with a consistent view of ordering.

---

**2. Process suspension longer than CLAIM_TTL causes silent lock loss with no detection**

*Trigger:* The worker process is suspended for longer than 60 seconds by the operating system. Sources: JVM/Python GC stop-the-world pause (less common in CPython but possible under memory pressure with a lot of finalizers), Linux CFS scheduler starvation under CPU contention on a shared host, container CPU throttling when the cgroup CPU quota is exhausted, a `SIGSTOP` from an operator or monitoring tool, a VM live migration that pauses the guest for tens of seconds, or a kernel page fault on a memory-mapped file that blocks on slow storage.

*Failure path:* Worker A acquires the lock, begins executing. The process is suspended for 65 seconds. During suspension, the heartbeat thread is also suspended — it cannot fire. The 60-second TTL expires. Worker B acquires the lock and begins executing the same job. Worker A resumes. Its heartbeat fires; `RENEW_CLAIM_SCRIPT` returns 0. The heartbeat thread exits. Worker A's main thread has no signal path from the heartbeat thread (the `stop_event` is set by the main thread to stop the heartbeat, not by the heartbeat to signal the main thread). Worker A continues through `_dispatch`, through `db.transaction()`, records execution, marks job complete. Worker B does the same. The job executes twice.

*Why the protection cannot live inside the locking code:* The process cannot observe its own suspension. From inside the process, the suspension is a zero-duration event — the clock resumes where it stopped. There is no syscall that returns "you were paused for N seconds." The locking code cannot distinguish "I just acquired this lock and it is valid" from "I acquired this lock 90 seconds ago and it expired while I was suspended." The only observer with a consistent timeline is Redis.

*Required protection:* Two independent protections are both required, because either alone is insufficient:

- **Signal path (necessary but not sufficient):** The heartbeat thread must signal the main thread on lock loss (threading.Event from heartbeat to main, as identified in Layer 1). This catches the case where the process resumes before completing the job. It does not catch the case where the process resumes after completing the job — the signal arrives too late.

- **Fencing token at the write layer (required for correctness guarantee):** Every downstream write that must not execute twice must carry the lock generation token, and the storage layer must reject writes from a generation that is no longer current. This is the only protection that holds when the signal arrives after the damage is done. The token is acquired at lock acquisition time; the write layer checks it at commit time. This check cannot live in the locking code because the locking code has already exited (or is about to) by the time the write executes.

**3. Redis connection loss between acquisition and renewal produces an undetected split-brain window**

*Trigger:* The network path between the worker and Redis becomes unavailable after lock acquisition but before the first renewal. Sources: a Redis failover (Sentinel or Cluster promoting a replica), a network partition isolating the worker from Redis, a Redis process restart, or a TCP connection reset from a load balancer idle timeout.

*Failure path:* Worker A acquires the lock successfully. The network to Redis is lost. Worker A's heartbeat fires at t=15s; `self._renew_script(...)` raises a Redis connection exception. The current code has no exception handling in `_renew()`. An unhandled exception in a daemon thread causes the thread to exit silently — Python does not propagate thread exceptions to the main thread. The main thread has no visibility into this. Meanwhile, if Redis failed over to a replica that did not receive the lock key (replication is asynchronous), Worker B can acquire the same lock on the new primary. Both workers proceed. Worker A's subsequent heartbeat attempts also fail silently. Worker A completes the job. Worker B completes the job.

*Why the protection cannot live inside the locking code:* The locking code cannot distinguish a transient network hiccup (where the lock is still held on Redis) from a failover (where the lock key was lost on promotion of a replica that hadn't replicated it). Both look identical from inside the worker: a failed Redis call. The correct response to these two cases is different, and the locking code cannot determine which case it is in. Additionally, even if the locking code caught the exception and signaled the main thread, the main thread may have already passed the point of no return in `_dispatch`.

*Required protection:* Exception handling in `_renew()` is necessary (Layer 1 finding) but insufficient. The renewal exception must be treated as a lock-loss event and must signal the main thread to abort. However, the abort signal races with job completion — if `_dispatch` has already returned and the transaction is in flight, the signal may not prevent the write. The fencing token at the write layer is again the terminal protection. Additionally: if Redis is operating in a replicated topology without WAIT or without Redlock, the lock guarantee is fundamentally weaker than the code assumes — this is an architectural assumption that must be documented and accepted explicitly before production.

**4. The idempotency check and the job execution are not atomic with respect to the lock**

*Trigger:* The lock expires (for any reason in items 1–3 above) between the idempotency check and the completion of `db.transaction()`. This is not a single-point failure — it is a window that exists in every execution.

*Failure path:* Worker A acquires the lock. `db.get_execution(req.job_id)` returns None — job has not been completed. Worker A calls `_dispatch`. The lock expires mid-dispatch (GC pause, clock skew, network partition). Worker B acquires the lock. Worker B calls `db.get_execution(req.job_id)` — still returns None, because Worker A has not yet written the completion record. Worker B calls `_dispatch`. Both workers call `db.transaction()` and attempt to write. Without a fencing token or conditional write at the database layer, both writes succeed. The idempotency check is not a guard — it is a read that is only valid for the instant it executes, and its validity is not preserved by the lock if the lock can expire.

*Why the protection cannot live inside the locking code:* The lock is a coordination mechanism, not a transaction. It serializes entry, not completion. The window between the idempotency check and the completion write is unbounded in duration — it spans `_dispatch`, which can call external services, wait on I/O, or block for arbitrary time. No lock with a finite TTL can close this window. The only mechanism that can close it is a conditional write at the database layer: `INSERT ... WHERE NOT EXISTS` or an optimistic version check that atomically verifies the job is not complete and records completion in a single operation. This must live at the database layer because the database is the system of record, and only the database can make the check-and-write atomic.

**5. The release script executes after lock expiration, releasing a lock held by another worker**

*Trigger:* The `finally` block in `_claim_job` always executes `self._release_script(keys=[claim_key], args=[token])`. The Lua script is token-verified — it checks that the stored value matches the token before deleting. This is correct behavior. However: if the lock expired and Worker B acquired it, Worker B's token is now stored. Worker A's release script will find a token mismatch and return 0 without deleting — this is correct. The risk is a specific timing window: the lock expires, Worker B acquires it and stores its token, Worker A's release script fires in the window between Worker B's acquisition and Worker B's first renewal. The script correctly does nothing. *This specific path is safe.*

The actual risk is the inverse: Worker A holds the lock legitimately, `_dispatch` raises an unhandled exception, the `finally` block runs, the release script fires — and releases the lock before Worker A has confirmed the job did not partially execute. If `_dispatch` has side effects that are not transactional (external API calls, emails sent, charges initiated), releasing the lock allows Worker B to re-enter and attempt the job again, potentially duplicating those side effects.

*Failure path:* Worker A acquires lock. `_dispatch` calls an external payment API. The API call succeeds (payment charged) but the response is lost due to a network timeout. `_dispatch` raises a timeout exception. The `finally` block releases the lock. Worker B acquires the lock. The idempotency check finds no completion record (Worker A never wrote one). Worker B dispatches the same job. The payment is charged twice.

*Why the protection cannot live inside the locking code:* The locking code cannot know whether `_dispatch` had partial side effects. It cannot roll back an external API call. The lock release is correct — holding the lock longer does not help, because the lock does not prevent the external system from having already processed the request.

*Required protection:* Idempotency keys on all external API calls made within `_dispatch`. This is an application-layer concern that must be enforced at the point of each external call, not at the locking layer. The locking code cannot substitute for this.

**6. Daemon thread suppresses all exceptions, masking infrastructure failures silently**

*Trigger:* `_renew()` has no exception handling. Any exception — Redis connection error, Redis timeout, Redis authentication failure, Redis READONLY error (replica promotion), network reset — causes the thread to exit without logging, without signaling, and without any observable effect on the main thread. The thread is a daemon thread, so Python will not report its exit.

*Failure path:* This is the delivery mechanism for failure modes 2 and 3 above. The specific additional risk here is operational: when this system fails in production, the failure will be invisible in logs. The operator will see duplicate job executions with no corresponding error in the job executor logs. The post-incident investigation will have no signal from the component that failed. Mean time to diagnosis increases significantly.

*Required protection:* `_renew()` must catch all exceptions, log them at ERROR level with full context (job_id, attempt count, exception type), and treat any exception as a lock-loss event that signals the main thread. The logging must happen before the signal so that the signal does not race with log flushing. This protection lives in the heartbeat thread because that is the only place where the exception is observable.

**7. Single Redis instance is a single point of failure for the entire locking guarantee**

*Trigger:* The code assumes a single Redis instance. Redis replication is asynchronous. If the primary fails and a replica is promoted, any lock that was written to the primary but not yet replicated to the replica is silently lost. The new primary has no record of the lock. Any worker can acquire it.

*Failure path:* Worker A acquires the lock on the primary. The primary fails before replicating the key. The replica is promoted. Worker B connects to the new primary, finds no lock key, acquires the lock. Worker A continues executing (its connection to the old primary may still be alive, or it may have cached the lock state). Both workers execute the job.

*Why the protection cannot live inside the locking code:* The locking code has no visibility into replication lag or failover events. It cannot know that the Redis instance it is talking to has changed. It cannot know that its lock was not replicated.

*Required protection:* Either (a) accept this risk explicitly and document it as a known gap with the understanding that Redis failover is rare and the fencing token at the write layer is the terminal protection, or (b) implement Redlock across an odd number of independent Redis instances (as specified in the Redis documentation for this use case). This is an infrastructure topology decision that must be made before production. The code as written makes no provision for it and does not document the assumption.

**Open gaps**

1. **No fencing token passed to downstream writes.** The `yield token` in `_claim_job` makes the token available, but `execute()` ignores it — `with self._claim_job(req.job_id):` discards the yielded value. The database write path has no mechanism to reject a write from a worker whose lock generation has expired. This is the terminal protection for failure modes 1, 2, 3, and 4, and it is absent.

2. **No exception handling in `_renew()`.** Any Redis exception causes silent thread exit. No logging, no signal to main thread. This is the delivery mechanism for failure modes 3 and 6 and must be resolved before production.

3. **No signal path from heartbeat to main thread on lock loss.** The `stop_event` is unidirectional — main thread to heartbeat only. The heartbeat cannot abort the main thread's execution. This is the Layer 1 finding and remains open.

4. **Idempotency check is not atomic with the completion write.** `db.get_execution()` and `db.record_execution()` / `db.mark_job_complete()` are separate operations. The window between them is the full duration of `_dispatch`. Without a conditional write at the database layer, the check provides no guarantee. This gap is not closeable from within the locking code.

5. **External side effects in `_dispatch` are not idempotent.** The locking code cannot protect against duplicate external API calls if `_dispatch` raises after a partial side effect. Idempotency keys on external calls are required and are not present.

6. **Single Redis instance topology assumption is undocumented and unprotected.** The code assumes a single Redis primary with no provision for replication loss on failover. This assumption must be either accepted explicitly with documented risk or addressed with a multi-node topology.

7. **No test for lock-loss path.** The heartbeat returning 0 (token mismatch) is not tested. The behavior of the main thread when the heartbeat exits is not tested. This was a Layer 1 finding and remains open.

---

**Production readiness verdict: NOT APPROVED**

This system implements the visible mechanics of distributed locking correctly — atomic acquisition, token-verified release, Lua-based renewal, exponential backoff, structured logging. A surface reading finds nothing wrong. The failure modes are not in the code. They are in the gap between what the code assumes about the environment and what the environment actually guarantees.
