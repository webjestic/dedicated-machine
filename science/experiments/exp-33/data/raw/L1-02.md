**Findings**

**1. Heartbeat Silent Exit Does Not Signal Main Thread — Zombie Write Under Process Pause**

*Issue name:* Lock-loss invisibility / zombie write

*Failure path:*

Process A acquires `job:claim:{job_id}` with token T1 and begins executing. A GC pause, slow external call in `_dispatch`, or slow database operation holds Process A for longer than `CLAIM_TTL` (60s). The key expires in Redis. Process B calls `SET NX EX`, acquires the same key with token T2, and begins executing the same job.

Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs: `GET job:claim:{job_id}` returns T2, which does not match T1. The script returns 0. The `_renew` function hits `if result == 0: return` and exits. The `threading.Event` loop terminates. The thread is done.

Process A's main thread has no visibility into this. `stop_event` is not set by the heartbeat — it is only set by the `finally` block in `_claim_job`, which does not run until after `yield` returns. The main thread continues through `_dispatch`, through `self.db.transaction()`, through `record_execution` and `mark_job_complete`. Both Process A and Process B complete the job. For any non-idempotent job — payment charge, alert send, external API call — this is a correctness violation.

The checklist entry "Atomic claim renewal via Lua script (token-verified)" is accurate as far as it goes. The script correctly detects token mismatch. The defect is that the detection result is discarded: the heartbeat thread exits without communicating lock loss to the execution thread.

*Immediate fix — threading.Event signal:*

Add a `lock_lost_event = threading.Event()` alongside `stop_event`. In `_renew`, replace `return` with:

```python
if result == 0:
    lock_lost_event.set()
    return
```

In `_claim_job`, after `yield` returns (or before dispatching, via a cooperative check), raise if `lock_lost_event.is_set()`. The minimal version checks in the `finally` block before releasing:

```python
try:
    yield token
finally:
    stop_event.set()
    t.join(timeout=self.RENEWAL_INTERVAL + 1)
    if lock_lost_event.is_set():
        raise ClaimError(f"Claim lost during execution for job {job_id}")
    self._release_script(keys=[claim_key], args=[token])
```

This prevents the release attempt from running on a stale token and surfaces the failure to the caller.

*Why this is insufficient alone — architectural gap:*

The `threading.Event` fix prevents the *release* from running on a stale token and raises an error after the fact. It does not prevent the zombie write, because `_dispatch` and the database transaction complete before `finally` runs. The check comes too late.

The correct architectural fix is to make the critical section itself aware of lock state at the point of commit. This requires one of:

- **Fencing token / monotonic counter:** Redis returns a monotonically increasing token on lock acquisition (or a version counter is maintained). The database write includes a `WHERE fencing_token = N` predicate or uses optimistic concurrency control. If Process A's token is stale at commit time, the write is rejected by the database, not by the application.
- **Optimistic concurrency on job state:** `mark_job_complete` uses a conditional update — `UPDATE jobs SET status='complete', completed_by=? WHERE id=? AND status='pending'`. If Process B has already written, Process A's update affects zero rows. The application checks the row count and raises. This makes the database the authority on completion, not the Redis lock.

Without one of these, the window between lock expiry and the `finally` check remains exploitable. The `threading.Event` approach is a partial mitigation, not a correctness fix.

**2. Lock Loss Between `yield` and `finally` — Unchecked Execution Window**

*Issue name:* Execution proceeds after lock expiry with no mid-execution check

*Failure path:*

Even with the `lock_lost_event` check added to `finally`, the following sequence is valid: lock expires at T=61s, heartbeat detects loss at T=75s (next renewal interval), `_dispatch` completes at T=74s, `db.transaction()` commits at T=74s, `finally` runs at T=75s. The `lock_lost_event` is set, the error is raised — but the write already committed. The error is raised into `execute`, which has no handler for `ClaimError` raised from within the `with` block's `finally`, so it propagates uncaught past the `except ClaimError` block (which only catches the acquisition error) and surfaces as an unhandled exception to the caller, while the side effect is already done.

This is a variant of Finding 1. The root cause is the same. The fix is the same: fencing token or optimistic concurrency at the database layer.

---

**3. `ClaimError` Raised in `finally` Is Not Caught**

*Issue name:* Exception handling gap for post-execution lock loss

*Failure path:*

If Finding 1's partial fix is applied and `ClaimError` is raised in `finally`, it propagates out of the `with self._claim_job(...)` block. The `except ClaimError` in `execute` does catch it — but the `JobResult` returned will say "Job currently claimed by another worker" when in fact the job *did* execute (and may have committed). This is a misleading result. The caller receives `claimed=False` and may retry, causing a second execution.

*Fix:* Distinguish acquisition failure from mid-execution lock loss. Use a separate exception class (e.g., `ClaimLostError`) for lock loss detected post-acquisition. Handle it separately in `execute` and return a result that accurately reflects the ambiguous state — execution may have completed, retry is unsafe without idempotency guarantees.

---

**4. No Mid-Execution Cooperative Check**

*Issue name:* No liveness check between lock expiry and job completion

*Failure path:*

`_dispatch` may run for an arbitrarily long time (it calls `handler(payload)` which is user-supplied). The heartbeat thread and main thread share no channel other than `stop_event`. If `lock_lost_event` is added, it is only checked in `finally`. For long-running dispatches, the zombie window is the full remaining execution time after lock expiry.

*Fix:* For jobs where `_dispatch` is decomposable, pass `lock_lost_event` into the handler and check it at yield points. For opaque handlers, this is not possible without architectural changes. The fencing token approach at the database layer is the correct solution for opaque handlers.

---

**5. Test Suite Does Not Cover the Failure Mode**

*Issue name:* No test for heartbeat-detects-loss / main-thread-continues scenario

*Failure path:*

`test_claim_release_uses_lua_script` verifies the release script is called. No test simulates `_renew_script` returning 0 and verifies that the main thread does not complete the job. The test suite would pass on the current code and on the broken partial fix.

*Fix:* Add a test that configures `_renew_script` to return 0, runs `execute`, and asserts that `db.record_execution` is not called (or that a `ClaimLostError` is raised). This test will fail on the current code and pass only after a correct fix is in place.

---

**6. Redis Unavailability During Renewal Is Silently Swallowed**

*Issue name:* Network error in heartbeat treated as lock loss

*Failure path:*

If `self._renew_script(...)` raises a `redis.RedisError` (connection timeout, network partition), the exception propagates out of `_renew` uncaught, the thread dies, and `stop_event` is never set by the heartbeat. The main thread continues. This is indistinguishable from a clean lock loss from the main thread's perspective — except that in this case the lock may still be held (the key may still exist with the correct token). The renewal failure is not logged at a level that would alert on-call.

*Fix:* Wrap the `_renew_script` call in a `try/except redis.RedisError`. Log at ERROR level. Decide on policy: either treat Redis unavailability as lock loss (conservative, safe) or retry for a bounded number of attempts before treating as loss. Do not let the exception silently kill the thread.

---

**Correctness verdict:** NOT APPROVED

The primary defect — heartbeat detects lock loss and exits without signaling the main thread, allowing both the executing process and a concurrent process to complete the same job — is a correctness violation for any non-idempotent job. The threading.Event partial fix is a necessary but insufficient step. The architectural fix (fencing token or optimistic concurrency at the database write) is required before this code is correct under the stated concurrent load.

**Handoff summary for production readiness review:**

This review traced the full lock acquisition, renewal, and release path under concurrent execution, including the GC-pause / process-pause scenario where lock TTL expires mid-execution. The central finding is that the heartbeat thread detects token mismatch (lock loss) and exits silently, with no signal to the main execution thread; the main thread completes `_dispatch` and commits the database transaction with no awareness that the lock was lost and a second worker may be executing the same job — a zombie write for any non-idempotent job. A `threading.Event` signal from heartbeat to main thread is a necessary partial fix but does not close the window because the check occurs in `finally` after the database write; the correct fix is a fencing token or optimistic concurrency predicate at the database layer so the commit itself is conditional on lock validity. Secondary findings include a missing exception class distinction for post-execution lock loss (misleading retry semantics), unhandled Redis errors in the renewal thread, and absence of test coverage for the failure scenario. This review does not cover infrastructure failure modes, Redis cluster topology, deployment configuration, or operational runbook requirements.
