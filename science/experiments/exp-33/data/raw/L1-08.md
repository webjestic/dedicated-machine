**Findings**

---

**1. Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Under Process Pause**

**Issue name:** Undetected lock loss; main thread continues executing after claim expiry.

**Failure path:**

Process A acquires `job:claim:{job_id}` with token T1. A GC pause, slow external call in `_dispatch`, or slow database transaction holds Process A for longer than `CLAIM_TTL` (60s). The Redis key expires. Process B calls `SET NX EX`, acquires the same key with token T2, and begins executing the same job.

Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` executes `GET job:claim:{job_id}` and finds T2, not T1. The script returns 0. The `_renew` function checks `if result == 0: return` — and exits silently. No exception is raised. No event is set. No flag is written.

Process A's main thread is still inside the `with self._claim_job(req.job_id):` block. It has no mechanism to observe that the heartbeat exited. It completes `_dispatch`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`, and returns `JobResult(claimed=True, ...)`.

Both Process A and Process B complete the job. For any non-idempotent side effect — sending a notification, charging a payment, triggering an external API — the job runs twice. The database idempotency check (`get_execution`) does not protect against this because both processes pass it before either writes the completion record; the check is not inside the same atomic unit as the write.

**The checklist entry "Atomic claim renewal via Lua script (token-verified)" is accurate but incomplete. Token verification on renewal is only useful if a failed renewal terminates the work. Here it does not.**

**Required fix — two layers:**

*Layer 1 (necessary, not sufficient): Signal the main thread on lock loss.*

```python
stop_event = threading.Event()
lock_lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
        except redis.RedisError:
            lock_lost_event.set()
            return
        if result == 0:
            lock_lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

The main thread must check `lock_lost_event` at meaningful checkpoints — before and after `_dispatch`, before committing the transaction — and abort with an exception if it is set. This is the minimum required change to stop the zombie write in the common case.

*Layer 2 (architectural fix — required for correctness under arbitrary pause):* Threading events do not close the window; they narrow it. A process can pass the `lock_lost_event` check, then pause, then write — the event check and the write are not atomic with respect to the Redis lock state.

The correct fix is a **fencing token** (monotonic counter or Redis version): Redis returns a monotonically increasing integer when the lock is acquired (e.g., via `INCR` on a version key, or Redlock's recommended fencing approach). That token is passed through to every downstream write. The database and any external system reject writes carrying a token lower than the highest token they have seen. This makes stale writes detectable and rejectable at the point of effect, regardless of when the process paused.

Without a fencing token, the `lock_lost_event` approach reduces the probability of a zombie write but does not eliminate it.

**2. Idempotency Check Is Not Atomic With the Completion Write**

**Issue name:** TOCTOU on `get_execution` / `record_execution`.

**Failure path:**

Two workers both pass the `get_execution` check (both see `None`) before either calls `record_execution`. Both proceed to `_dispatch` and both write to the database. The database transaction wraps `record_execution` + `mark_job_complete` but does not include the `get_execution` check. The check and the write are not a single atomic operation.

This is a distinct issue from Finding 1. Even if the heartbeat correctly signals lock loss, this race exists whenever two workers acquire the lock in sequence fast enough that the first has not yet written its completion record when the second checks.

**Required fix:**

The idempotency check must be inside the same database transaction as the write, with a unique constraint or `INSERT ... WHERE NOT EXISTS` / `SELECT FOR UPDATE` pattern that makes the check-and-write atomic at the database level. The Redis lock serializes acquisition but does not make the DB check + DB write atomic.

---

**3. Lock Release Proceeds After Heartbeat Has Already Confirmed Token Loss**

**Issue name:** Spurious release attempt on a lock this process no longer owns.

**Failure path:**

After Finding 1's scenario, Process A's `finally` block calls `self._release_script(keys=[claim_key], args=[token])`. At this point the key holds Process B's token. The Lua script correctly returns 0 and does not delete the key. No data corruption results from this specific call.

However, if the `lock_lost_event` fix from Finding 1 is implemented and the main thread aborts early, the `finally` block still runs the release script. This is safe due to the token check, but it should be explicit: the release script's no-op on token mismatch is load-bearing for safety here, and that must be preserved. Any future refactor that replaces the Lua release with an unconditional `DEL` would reintroduce lock theft. This is a latent correctness dependency that is not documented in the code.

**Required fix:** Add a comment at the release call site stating that the Lua token check is the safety guarantee against releasing another worker's lock, and that unconditional `DEL` must never replace it.

---

**4. Redis Unavailability During Renewal Is Not Handled**

**Issue name:** `RedisError` in `_renew` propagates uncaught and terminates the heartbeat thread without signaling lock loss.

**Failure path:**

If Redis becomes unavailable mid-execution, `self._renew_script(...)` raises `redis.RedisError`. The exception propagates out of `_renew`, the thread terminates, and the main thread has no signal. The claim TTL will expire naturally (60s), but the main thread continues executing and may write after expiry — same zombie-write outcome as Finding 1, triggered by Redis unavailability rather than process pause.

**Required fix:**

Wrap the `_renew_script` call in a `try/except redis.RedisError` block. On exception, set `lock_lost_event` and return. This is shown in the corrected `_renew` above.

---

**5. Test Coverage Does Not Exercise the Failure Mode**

**Issue name:** No test for heartbeat-exit-without-main-thread-signal.

**Failure path:**

The test suite mocks `register_script` to return a mock that always returns 1. There is no test that sets the renewal script mock to return 0 and then asserts that `execute` raises or returns `claimed=False`. The gap in Finding 1 is untested and would not be caught by the existing suite after a fix is applied.

**Required fix:**

Add a test that configures the renew script mock to return 0 on first call, then asserts that `execute` does not call `record_execution` — verifying that lock loss aborts execution. This test should fail against the current code and pass after Finding 1 is fixed.

---

**Correctness verdict: NOT APPROVED**

Finding 1 is a production correctness violation. Under a GC pause or slow external call exceeding 60 seconds, two workers will complete the same job. The heartbeat's token-verified renewal is implemented correctly in isolation but is wired to no abort path. The fix requires both the `threading.Event` signal and a fencing token at the write layer to close the window fully.

---

**Handoff summary for production readiness review:**

This review traced the full lock acquisition, renewal, and release path under concurrent execution, including process-pause and Redis-unavailability scenarios. Two correctness violations were found: (1) the heartbeat thread exits silently on lock loss with no signal to the main thread, allowing zombie writes after a GC pause or slow dispatch — the architectural fix requires a fencing token at the write layer, not only a threading event; (2) the idempotency check and completion write are not atomic at the database level, leaving a TOCTOU window between two workers that both pass the check before either writes. Both issues must be resolved before this code handles non-idempotent jobs in production. This review does not cover infrastructure failure modes, Redis cluster topology, Redlock correctness under network partition, or operational concerns such as TTL tuning for observed job durations.
