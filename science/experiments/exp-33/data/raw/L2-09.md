**Layer 1 coverage:** Correctness review confirmed Lua script atomicity, heartbeat/main-thread communication, and Redis error handling — and identified the critical defect: `_renew` exits silently on lock loss with no signal to the main thread, requiring at minimum `threading.Event` signaling and a fencing token passed to downstream writes for full correctness.

---

**Infrastructure failure modes**

**1. Clock skew between worker and Redis server causes premature lock expiry**

*Trigger:* Redis TTL is set by the worker issuing `SET ... EX 60`. The expiry clock runs on the Redis server. If the worker's system clock and the Redis server's clock diverge — or if Redis is under memory pressure and key eviction fires before TTL expiry — the lock expires before the worker believes it should. This is distinct from a GC pause: the worker is running normally, the heartbeat is firing on schedule, but the renewal arrives after Redis has already expired the key. The renewal script sees no key, returns 0, and the heartbeat thread exits silently. The main thread continues executing.

*Failure path:* Worker A holds lock. Redis expires key early due to clock skew or eviction pressure. Worker B acquires lock. Worker A's heartbeat fires, renewal returns 0, heartbeat exits. Worker A's main thread has no signal. Both workers reach `_dispatch` and `db.transaction()`. Both write. Job executes twice.

*Required protection:* Fencing token (monotonic counter from Redis `INCR`, or a Redis-native lock version) passed through to the database write. The database layer must reject writes from a token that has been superseded. This cannot live in the locking code because by the time the locking code could detect the problem, the main thread is already past the check — the only enforcement point that is always reached before the write commits is the database transaction itself. Redis `WAIT` can reduce replication lag risk but does not solve clock skew. The protection must be at the write layer.

---

**2. Redis replica promotion during lock hold creates a lock that never existed on the new primary**

*Trigger:* Redis replication is asynchronous by default. Worker A acquires the lock on the primary. The primary fails before the write replicates. A replica is promoted. The new primary has no record of the lock. Worker B acquires the same lock on the new primary. Worker A continues executing against what it believes is a valid lock — its heartbeat will now either fail to connect (if the old primary is gone) or succeed against the new primary with a fresh key it did not set, depending on topology and client configuration.

*Failure path:* Worker A and Worker B both hold what each believes is the exclusive lock. Both execute the job. Both write. This is the canonical split-brain scenario for Redis-backed locks. It is not hypothetical — it occurs on any unplanned primary failure in a replicated Redis deployment.

*Required protection:* Redlock across independent Redis nodes (N≥3, majority quorum acquisition) eliminates the single-point replication gap, but introduces its own clock-dependency failure modes under NTP drift. The minimum viable protection for a single-node or sentinel deployment is the fencing token at the database layer — the same token required in failure mode 1. The locking layer cannot protect against this because the locking layer's own state has been invalidated by an infrastructure event it has no visibility into. The database write is the only serialization point that survives a Redis topology change.

---

**3. Network partition between worker and Redis causes false lock-loss on renewal, then reconnection causes dual execution**

*Trigger:* A transient network partition isolates the worker from Redis for longer than `RENEWAL_INTERVAL`. The renewal call fails or times out. Depending on the Redis client's timeout configuration and exception handling in `_renew`, the heartbeat thread may exit (if the exception propagates past the result check) or may loop and retry. Meanwhile, if the partition lasts longer than `CLAIM_TTL`, the lock expires. Worker B acquires the lock. The partition heals. Worker A's subsequent renewal either finds its key gone (returns 0, thread exits) or — if the partition healed before TTL expiry — successfully renews a lock that Worker B also holds if B acquired during the partition window.

*Failure path:* The current `_renew` implementation has no exception handling. A Redis connection error during renewal raises an exception inside the thread. The thread dies. The main thread is not notified. The main thread continues executing. If the partition outlasted the TTL, Worker B has also acquired the lock. Dual execution follows.

*Required protection:* Two layers. First, `_renew` must catch Redis exceptions and signal the main thread on unrecoverable renewal failure — this is the Layer 1 fix already identified. Second, and independently, the database write must carry a fencing token that the database layer enforces. The network partition scenario is particularly dangerous because the locking layer's failure is silent and the main thread has no in-band signal from the infrastructure that anything has changed.

---

**4. GC pause or slow external call in `_dispatch` exceeds `CLAIM_TTL` before first renewal**

*Trigger:* `CLAIM_TTL` is 60 seconds. `RENEWAL_INTERVAL` is 15 seconds. The first renewal fires at T+15. If the process pauses — GC, slow DNS, slow external HTTP call inside a handler — for longer than 60 seconds before T+15, the lock expires before the first renewal attempt. Worker B acquires the lock at T+60. Worker A resumes at T+61. Worker A's heartbeat fires at T+75, finds the key gone (or finds Worker B's token), returns 0, exits silently. Worker A's main thread is already past `_dispatch` and writing to the database.

*Failure path:* This is the canonical scenario from the Layer 1 review. It is worth restating here because the infrastructure dimension is that the pause is imposed externally — the JVM or CPython GC, the OS scheduler, a slow NFS mount, a slow database call in `get_execution` — and the locking code has no mechanism to detect that it has been paused. The code cannot defend against a pause it cannot observe.

*Required protection:* The fencing token passed to the database write is the only protection that survives an arbitrary pause. The database must enforce: "this write is only accepted if no write with a higher token for this job_id has been committed." This requires the database layer to store and compare tokens, not just job completion state. The current `mark_job_complete` / `record_execution` interface has no token parameter and therefore cannot enforce this. The idempotency check (`get_execution` before dispatch) does not protect here because both workers pass the check before either writes.

---

**5. Worker process receives SIGKILL after `_dispatch` completes but before `db.transaction()` commits**

*Trigger:* The OS kills the worker process — OOM killer, deployment rollout, spot instance reclaim — after the handler has executed its side effects (sent the alert, charged the payment, called the external API) but before `db.transaction()` commits. The job is not marked complete. The lock is released (or expires). Worker B acquires the lock, passes the idempotency check (no completion record exists), and executes the job again.

*Failure path:* The external side effect has already occurred. It occurs again. For idempotent handlers this is acceptable. For non-idempotent handlers — payment charge, alert send, webhook delivery — this is a correctness violation. The locking layer cannot prevent this because the process is dead.

*Required protection:* Handler idempotency enforced at the handler level, not the executor level. Each handler must be able to detect and safely skip a duplicate invocation. This requires either idempotency keys passed to external systems, or a pre-commit record of intent written to the database before `_dispatch` is called (a "job started" record that is distinct from "job completed"), so that on retry the handler can detect the prior attempt. The executor's current design calls `_dispatch` before writing any record, which means there is no recovery path for a crash between dispatch and commit.

---

**6. Redis client connection pool exhaustion causes `SET NX` to block or fail, masking availability as a contention signal**

*Trigger:* Under load, the Redis client connection pool is exhausted. The `redis.set()` call blocks waiting for a connection, or raises a connection pool timeout exception. The retry loop in `_claim_job` interprets a failed `set` call (returning `None` or raising) as contention and applies exponential backoff, then raises `ClaimError`. The job is not executed. No worker executes it.

*Failure path:* This is a liveness failure rather than a correctness failure — the job is dropped, not duplicated. However, the failure is silent: the caller receives `JobResult(claimed=False, message="Job currently claimed by another worker")` which is indistinguishable from genuine contention. Operators have no signal that Redis availability is the cause. Under sustained pool exhaustion, all jobs stop executing while appearing to be in contention.

*Required protection:* Distinguish Redis exceptions from `SET NX` returning `False`. The current retry loop does not catch exceptions — a connection error will propagate up through `_claim_job` as an unhandled exception, bypassing `ClaimError` handling and surfacing as an unhandled exception to the caller. This needs explicit exception handling with a distinct error path and metric emission. The protection must live at the infrastructure layer: connection pool sizing, Redis availability monitoring, and alerting on error-rate-vs-contention-rate divergence.

---

**7. `t.join(timeout=RENEWAL_INTERVAL + 1)` in `finally` block silently allows release before thread exits**

*Trigger:* The heartbeat thread is blocked on a slow Redis call — network hiccup, Redis latency spike — when the main thread enters the `finally` block. `stop_event.set()` is called. `t.join(timeout=16)` waits 16 seconds and then returns regardless of whether the thread has exited. `_release_script` is then called. If the thread subsequently unblocks and its renewal call completes after the release, the thread attempts to renew a key that has been released — or, if Worker B has acquired the lock in the window between release and the thread's renewal attempt, the renewal succeeds against Worker B's lock only if Worker B used the same token, which is impossible. The renewal will return 0 and the thread will exit. This specific path does not cause dual execution, but it means the join timeout is a silent swallow: the thread's final state is unknown at release time.

*Failure path:* Lower severity than the above. The primary risk is that a thread leak under sustained latency causes thread accumulation. The secondary risk is that in a future refactor, the assumption that the thread is done when `join` returns is wrong, and code added after the join that depends on thread state will be incorrect.

*Required protection:* Log a warning when `join` times out without thread exit. Emit a metric. The thread should be instrumented so that its exit — normal or timeout — is observable. This is an operational gap, not a correctness gap in the current code, but it is a trap for future maintainers.

---

**Open gaps**

1. **No fencing token passed to database writes.** The database layer has no mechanism to reject a write from a worker whose lock has been superseded. This is the primary unresolved correctness gap. It is not fixable inside the locking code.

2. **No pre-dispatch intent record.** `_dispatch` is called before any database record is written. A process crash after dispatch but before commit leaves the job in an undetectable partially-executed state. Recovery requires handler-level idempotency, which is not enforced or documented.

3. **No exception handling in `_renew`.** Redis connection errors inside the heartbeat thread kill the thread silently. The main thread has no signal. This is the Layer 1 gap carried forward; it also has infrastructure consequences under network partition.

4. **Redis replica promotion is unprotected.** A single Redis primary with async replication provides no durability guarantee for lock acquisition. The lock can be lost on failover with no signal to any worker. No Redlock or equivalent quorum mechanism is in place.

5. **Connection pool exhaustion is indistinguishable from contention.** No metric or log distinguishes a Redis availability failure from genuine job contention. Operators cannot tell the difference.

6. **Handler non-idempotency is unmitigated.** The executor makes no guarantee about handler idempotency and provides no framework for handlers to detect duplicate invocations. For any handler with non-idempotent external effects, the system provides no correctness guarantee under crash-recovery scenarios.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system correctly acquires and releases Redis locks under the happy path. The Lua scripts are individually correct. The retry logic and input validation are sound. Under nominal conditions with no infrastructure events, it will prevent duplicate execution.

It will not prevent duplicate execution under the conditions that actually occur in production: GC pauses, network partitions, Redis replica promotions, and process crashes. The root cause is architectural: the system uses Redis lock acquisition as a proxy for execution correctness, but Redis lock state and execution correctness are only coupled at the moment of acquisition. Any infrastructure event that invalidates the lock after acquisition — and before the database write commits — breaks that coupling invisibly. The main thread has no mechanism to detect that the coupling has broken, and the database write has no mechanism to reject a write from a worker whose authority has lapsed.

Before this ships, the following must be in place. First, the heartbeat thread must signal the main thread on lock loss or renewal failure, and the main thread must abort before the database write — this is the Layer 1 fix and it is a prerequisite, not a substitute for what follows. Second, a fencing token — a monotonic counter incremented on each lock acquisition, stored in Redis alongside the token, returned to the caller, and passed through to the database write — must be enforced at the database layer. The database must reject any write whose token is not the current maximum for that job_id. Third, either a pre-dispatch intent record must be written to the database before `_dispatch` is called, or every handler must be documented and enforced as idempotent with an explicit idempotency key passed to all external calls. Fourth, the Redis deployment topology must be documented and the replication failure mode must be explicitly accepted or mitigated — if this runs against a single primary with async replicas, the replica promotion gap must be called out as an accepted risk with a named owner.

The system as written will produce duplicate executions under conditions that are not edge cases. They are scheduled maintenance windows, spot reclaims, GC pressure under load, and network blips. I have been on the call when each of these triggered exactly this failure mode. I am not signing this off.
