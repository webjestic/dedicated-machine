**Layer 1 coverage:** Correctness review confirmed the structural gap — heartbeat exits silently on lock loss, main thread continues executing without visibility into that exit, producing duplicate job execution; fix requires fencing token or optimistic concurrency at the database write layer, not a threading signal.

---

**Infrastructure failure modes**

**1. Clock skew between worker host and Redis server causes premature lock expiry**

*Trigger:* Redis TTL is enforced by the Redis server's internal clock. The worker's renewal interval (15s) is calibrated against the worker's local clock. If the worker host clock runs slow relative to the Redis server — or if NTP correction causes a forward jump on the Redis host — the effective TTL from Redis's perspective elapses faster than the worker expects. A 60s TTL with a 15s renewal interval gives four renewal windows. A modest clock skew or NTP step-correction of even a few seconds on the Redis host compresses those windows. Combined with any renewal latency (network round-trip, Redis processing under load), the lock can expire before the next renewal fires.

*Failure path:* Worker A holds the lock. Redis clock advances faster than Worker A's clock. Lock expires at Redis before Worker A's renewal fires. Worker B acquires the lock and begins executing the same job. Worker A's next renewal returns 0 — heartbeat exits. Worker A's main thread continues. Both workers reach `db.record_execution` and `db.mark_job_complete`. If the database transaction has no uniqueness constraint or optimistic concurrency check on `job_id`, both writes succeed. Job executes twice.

*Required protection:* Fencing token (monotonic lock version counter) written by Redis at acquisition time, carried through to the database write, enforced by a uniqueness or compare-and-swap constraint at the database layer. The database must reject the second write based on the token, not on application-layer logic. This cannot live in the locking code because by the time the locking code runs its finally block, the execution has already completed. The protection must be at the point of state mutation.

---

**2. Redis network partition or transient unavailability during renewal causes silent lock abandonment**

*Trigger:* A network partition, Redis restart, or transient TCP failure causes `_renew_script` to raise an exception rather than return 0. The current `_renew()` implementation has no exception handling around the script call. An unhandled exception in the renewal thread causes the thread to exit. From the thread's perspective, this is identical to a token mismatch — silent exit. The main thread has no visibility into whether the heartbeat thread exited due to lock loss, network error, or normal shutdown.

*Failure path:* Worker A holds the lock. Network partition isolates Worker A from Redis for longer than CLAIM_TTL. The lock expires at Redis. Worker B, which can reach Redis, acquires the lock and begins executing. Meanwhile Worker A's renewal thread has been throwing exceptions and has exited. Worker A's main thread, unaware, completes execution and writes to the database. Same duplicate execution outcome as failure mode 1, but triggered by infrastructure partition rather than clock skew.

*Secondary path:* Even if the partition is brief — shorter than CLAIM_TTL — the renewal thread crashes on the exception and does not recover. Subsequent renewal attempts never fire. The lock expires at the next TTL boundary with no further renewals. Worker A continues executing past that expiry.

*Required protection:* Two distinct protections required. First, the renewal thread must catch Redis exceptions and handle them explicitly — either retry with backoff or signal lock loss to the main thread. This is a correctness fix, not infrastructure protection, but its absence makes the infrastructure failure mode worse. Second, and independently: the database write must carry a fencing token that Redis issued at acquisition time. The database enforces that only the holder of the current lock generation can write. This is the only protection that holds when the locking layer itself has been invalidated by infrastructure.

---

**3. Redis Sentinel or Cluster failover during the acquisition-to-renewal window invalidates the lock entirely**

*Trigger:* This implementation uses a single Redis client. In a Redis Sentinel or Cluster deployment, a primary failure triggers failover to a replica. Replication in Redis is asynchronous. A lock written to the primary may not have replicated to the replica before the primary fails. After failover, the new primary has no record of the lock. Any worker that attempts `SET NX EX` on the new primary will succeed — including a second worker that arrives after Worker A already holds the lock on the (now-dead) primary.

*Failure path:* Worker A acquires lock on primary. Primary fails before replication. Failover promotes replica. Worker B acquires the same lock key on the new primary — `SET NX EX` succeeds because the key does not exist on the new primary. Both Worker A and Worker B now believe they hold the lock. Both execute the job. Both write to the database. The locking mechanism believes it is operating correctly — both workers have valid tokens, both tokens match their respective Redis instances. The guarantee was invalidated by the infrastructure event, not by any code error.

*Required protection:* For single-node Redis: document the deployment constraint explicitly — this locking scheme is only safe against a single Redis instance, not a Sentinel or Cluster topology with async replication. For multi-node deployments: Redlock across independent Redis nodes (with all the known caveats about clock drift and Redlock's own failure modes under network partition). In either case: the fencing token at the database layer is the only protection that survives a split-brain scenario, because it enforces mutual exclusion at the point of state mutation regardless of what the locking layer believes.

---

**4. GC pause or process stall exceeding CLAIM_TTL invalidates the lock before renewal fires**

*Trigger:* This is the primary failure mode from Layer 1, but it has an infrastructure dimension that is distinct from the threading concern. A JVM-style stop-the-world GC pause, a Python GIL-induced stall, a container memory pressure event causing the process to be swapped out, or a kernel scheduling preemption can suspend the worker process for longer than CLAIM_TTL. The renewal thread is also suspended — it cannot fire during the pause. The lock expires. Another worker acquires it.

*Failure path:* Worker A acquires lock. Process is suspended for >60s (CLAIM_TTL). Lock expires. Worker B acquires lock, begins executing. Worker A resumes. Worker A's renewal thread fires, gets 0, exits. Worker A's main thread continues executing — it has been running the entire time from its own perspective, it just experienced no elapsed time. Worker A completes `_dispatch`, enters `db.transaction()`, writes. Worker B also completes and writes. Duplicate execution.

*Infrastructure dimension:* The 60s TTL is a configuration parameter. The actual duration of GC pauses, container scheduling jitter, and kernel preemption events in production is not bounded by application configuration. A 60s TTL that seems generous in testing is not generous against a container that gets OOM-killed and restarted, or a host under memory pressure that starts swapping. The TTL is not a safety bound — it is a best-effort parameter that the infrastructure can exceed.

*Required protection:* The TTL and renewal interval cannot be tuned to safety. The only protection is at the database write layer: a fencing token or monotonic version counter that the database enforces with a conditional write. The database must reject any write from a worker that does not hold the current lock generation. This is the same fix as failure modes 1–3 — all roads lead to the same missing protection.

---

**5. `db.get_execution` check and `db.record_execution` write are not atomic — TOCTOU race**

*Trigger:* The idempotency check (`existing = self.db.get_execution(req.job_id)`) and the write (`db.record_execution` + `db.mark_job_complete`) are two separate database operations separated by `_dispatch` execution time. The Redis lock is intended to serialize this sequence, but as established in failure modes 1–4, the lock can be held by two workers simultaneously due to infrastructure events. When that happens, both workers pass the `get_execution` check (neither has written yet), both execute `_dispatch`, and both attempt the write.

*Failure path:* Worker A and Worker B both hold the lock (due to any of the above failure modes). Worker A checks `get_execution` — returns None. Worker B checks `get_execution` — returns None. Worker A executes `_dispatch`. Worker B executes `_dispatch`. Worker A writes. Worker B writes. If the database has no uniqueness constraint on `job_id` in the execution table, both writes succeed. The idempotency check provides no protection because it is not atomic with the write.

*Required protection:* The database must enforce a unique constraint on `job_id` in the execution table (or equivalent — a unique index, a conditional insert, a compare-and-swap). The application-layer check is a performance optimization to avoid unnecessary dispatch work; it is not a correctness guarantee. The correctness guarantee must be enforced by the database at write time. This is a database schema requirement, not an application code requirement, and it cannot be satisfied by any change to the locking code.

---

**6. Release script fires after lock has already expired and been reacquired by another worker**

*Trigger:* In the `finally` block, after `stop_event.set()` and `t.join()`, the code calls `self._release_script(keys=[claim_key], args=[token])`. If the lock expired and was reacquired by Worker B before this point, Worker A's token no longer matches. The Lua script correctly returns 0 and does not delete the key. This is the intended behavior. However: if Worker A's process is suspended between `t.join()` returning and the release script executing — even briefly — and Worker B has acquired and *released* the lock in that window, and Worker C has then acquired the lock, Worker A's release script will again return 0 (token mismatch). This is correct. But the log line `"Claim released job_id=%s"` fires unconditionally regardless of the script's return value.

*Failure path:* Operational, not a correctness violation in isolation. But: the unconditional log creates false confidence in monitoring. An operator observing logs will see "Claim released" for every execution, including executions where the lock had already expired and been reacquired. This masks the infrastructure failure event. When duplicate execution occurs, the log trail will not show a failed release — it will show a normal release. The failure mode is invisible in the operational record.

*Required protection:* Log the return value of the release script. If the script returns 0, log at WARNING or ERROR level: the claim was not held at release time, which means the lock expired during execution. This is a signal that either the TTL is too short, the job is too slow, or a process stall occurred. This does not prevent duplicate execution — that requires the fencing token — but it makes the failure mode visible in production observability rather than silent.

---

**7. Renewal thread join timeout does not prevent release script from firing against an active lock held by another worker**

*Trigger:* `t.join(timeout=self.RENEWAL_INTERVAL + 1)` — 16 seconds. If the renewal thread is blocked on a Redis call that is itself blocked (Redis under extreme load, network half-open connection), `join` returns after 16 seconds regardless of whether the thread has actually exited. The code then proceeds to call the release script. If the lock has expired and Worker B holds it, the release script returns 0 (correct — token mismatch protects Worker B's lock). But the thread is still running in the background as a daemon thread.

*Failure path:* The daemon thread will eventually unblock, call `_renew_script`, and get 0 (Worker B's token is in Redis). Thread exits. No correctness violation from the release script (token mismatch protects it). But: the daemon thread holds a reference to `claim_key` and `token` from Worker A's execution. If Worker A is in a tight execution loop (processing many jobs), multiple orphaned renewal threads can accumulate, each making Redis calls with stale tokens. Under high job throughput, this creates a background load of failed renewal calls that is invisible to the application but visible to Redis.

*Required protection:* Operational concern rather than correctness violation, given the token-verified scripts. Document the bounded thread accumulation risk. Consider a hard timeout on the renewal thread's Redis calls to ensure the thread can always exit within a bounded time after `stop_event` is set.

---

**Open gaps**

1. **No fencing token at the database write layer.** This is the central unresolved gap. Every infrastructure failure mode that causes two workers to simultaneously hold the lock — clock skew, network partition, Redis failover, process stall — results in duplicate execution because the database write has no mechanism to reject the second writer. The application-layer idempotency check is not atomic with the write and provides no protection under concurrent execution. This gap cannot be closed by any change to the locking code. It requires a database schema change (unique constraint or conditional write on `job_id`) and a fencing token carried from lock acquisition through to the write.

2. **No exception handling in the renewal thread.** Redis errors during renewal cause the thread to exit silently via unhandled exception, which is operationally identical to lock loss. The main thread cannot distinguish between "renewal failed because lock was lost" and "renewal failed because Redis is unavailable." Both result in silent thread exit and continued execution. Requires explicit exception handling in `_renew()` with either retry logic or explicit signaling.

3. **Redis topology assumption is undocumented and unenforced.** The locking scheme is only safe against a single Redis primary. Sentinel or Cluster deployments with async replication can produce split-brain lock state after failover. This constraint is not documented anywhere in the code, the PR description, or the checklist. A deployment engineer has no signal that this matters.

4. **Release script return value is not checked.** A return value of 0 from the release script indicates the lock was not held at release time — a signal of lock expiry during execution. This is logged as a normal release. The failure mode is operationally invisible.

5. **CLAIM_TTL has no documented relationship to observed job durations.** The 60s TTL is described as "generous for slow jobs" with no basis in measurement. In production, job duration distributions are not known until the system runs. If any job type's p99 duration exceeds 60s, the renewal mechanism is the only protection — and renewal can be defeated by process stall. TTL tuning requires operational data that does not yet exist.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system implements a structurally sound distributed locking pattern — atomic acquisition, token-verified renewal and release, scoped heartbeat thread, input validation, database transaction wrapping. Under normal operating conditions, with a healthy single Redis instance and jobs that complete well within the TTL, it will behave correctly.

It does not protect against the class of failures where the infrastructure invalidates the lock guarantee while the application believes the guarantee still holds. A process stall exceeding CLAIM_TTL, a Redis failover with async replication lag, or clock skew compressing the effective TTL window will all produce a state where two workers simultaneously believe they hold the lock. When that happens, both workers will pass the idempotency check, both will execute the job handler, and both will write to the database. The database has no mechanism to reject the second write. For any non-idempotent job — payment processing, alert dispatch, external API calls — this is a correctness violation that the locking layer cannot prevent from within itself.

The required fix is a fencing token: Redis issues a monotonic counter at lock acquisition, the worker carries that token through to the database write, and the database enforces a conditional write that rejects any token that is not the current generation for that job. This is a database schema change and an application change to the write path. It is not optional for non-idempotent handlers.

Additionally required before ship: exception handling in the renewal thread (failure mode 2), documentation of the single-primary Redis topology constraint (failure mode 3), and logging of the release script return value (failure mode 6). The fencing token is the blocking item. The others are required but do not independently block.

This system will not be approved until the database write layer enforces mutual exclusion independently of the locking layer.
