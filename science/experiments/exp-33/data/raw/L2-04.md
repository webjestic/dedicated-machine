**Layer 1 coverage:** Correctness review confirmed the zombie-write vulnerability (heartbeat exit not signaled to main thread), identified fencing token at the storage layer as the required fix, and flagged Redis unavailability handling and observability gaps — infrastructure-level failure modes explicitly out of scope.

---

**Infrastructure failure modes**

**1. Clock skew between worker and Redis server causes premature lock expiration**

*Trigger:* Redis uses its own internal clock to evaluate TTL. The worker sets `ex=60` at the moment of acquisition, but the expiry countdown begins on the Redis server's clock. If the worker's system clock and the Redis server's clock are skewed — or if the Redis server is under memory pressure and key expiry is delayed or, conversely, accelerated by a clock adjustment (NTP step, leap second, VM clock slew after live migration) — the actual wall-clock duration of the lock as experienced by the worker diverges from CLAIM_TTL.

*Failure path:* Worker A acquires the lock with a 60-second TTL. The Redis host's clock steps forward 45 seconds due to an NTP correction. The key expires after 15 wall-clock seconds from Worker A's perspective. Worker B acquires the lock. Worker A's heartbeat fires at t=15 (its clock), finds `result == 0`, exits silently. Worker A's main thread continues executing — it has no signal. Both workers complete the job. The lock appeared valid to Worker A throughout; the expiry was imposed externally by a clock event neither the code nor the worker could observe.

*Required protection:* Fencing token / monotonic write counter at the storage layer. The database write that records job completion must include a `WHERE last_writer_token = current_token` predicate (optimistic concurrency). A worker whose token has been superseded cannot commit. This protection cannot live in the locking code because the locking code's view of lock validity is derived from the same Redis TTL that was already invalidated by the clock event — it is reasoning from a corrupted premise.

---

**2. Redis replica promotion during Redlock-style single-node operation produces a second valid lock**

*Trigger:* This implementation uses a single Redis instance (or treats a cluster as a single logical endpoint). Under Redis Sentinel or Cluster failover, the following sequence is possible: Worker A writes the lock to the primary. The primary acknowledges. Before the write is replicated, the primary fails. Sentinel promotes a replica. The replica has no record of the lock. Worker B connects to the new primary, successfully acquires the lock with `SET NX EX`.

*Failure path:* Worker A holds what it believes is a valid lock (it received `True` from `SET NX EX`). Worker B holds a lock on the same key on the new primary. Both heartbeat threads renew against the same key on the same (new) primary — one of them will eventually lose, but in the window between failover completion and the first renewal cycle (up to RENEWAL_INTERVAL = 15 seconds), both workers execute concurrently. Both pass the `db.get_execution()` idempotency check if the database write has not yet committed. Both dispatch. Both attempt to write. Without a fencing token predicate on the database write, both succeed.

*Required protection:* Again, fencing token at the database write layer. Additionally, the operational runbook must document that this implementation does not provide the Redlock guarantee and that single-node Redis failover is a known correctness window. The Redlock algorithm itself requires quorum writes across independent nodes with no replication — this code does not implement that. The gap must be named explicitly in the deployment documentation, not just in code review. The locking code cannot protect against this because the lock it holds is real and valid from its perspective; the invalidation happened at the infrastructure layer before it had any information.

---

**3. Network partition between worker and Redis allows lock expiry to proceed undetected while execution continues**

*Trigger:* A network partition isolates the worker from Redis after lock acquisition. The lock TTL continues counting down on the Redis server. The worker's heartbeat thread attempts renewal, receives a connection error or timeout, and — per the current implementation — silently exits (the `result == 0` branch is taken only on token mismatch; a network exception is unhandled and will propagate out of `_renew_script(...)`, terminating the thread via unhandled exception). Meanwhile, the main thread is executing the job. The partition heals after CLAIM_TTL seconds. The lock has expired. Another worker has acquired it.

*Failure path:* Worker A is mid-execution during a 90-second partition. The lock expires at t=60. Worker B acquires the lock at t=61. Worker B passes the idempotency check (Worker A has not committed). Worker B dispatches. The partition heals at t=90. Worker A commits. Worker B commits. Duplicate execution.

*Required protection:* Two layers. First, the heartbeat thread must treat a network exception as lock loss — it must signal the main thread to abort, not silently exit. This is the threading.Event fix from Layer 1, which is necessary but not sufficient. Second, and again, fencing token at the database write. The reason the threading.Event fix alone is insufficient: the signal reaches the main thread only if the main thread is in a position to check it. If the main thread is blocked in `_dispatch()` on a slow external call, the signal is not checked until dispatch returns — by which time Worker B may have already committed. The database write predicate is the only layer that can enforce mutual exclusion at the moment of commitment.

---

**4. Process pause longer than CLAIM_TTL causes lock expiry mid-execution with no in-process signal**

*Trigger:* GC pause, OS scheduler starvation, container CPU throttling (cgroups), or a slow external call in `_dispatch()` suspends the worker process for longer than CLAIM_TTL (60 seconds). This is the canonical scenario from the Layer 1 review, but the infrastructure dimension is distinct: container environments with CPU limits routinely produce pauses of this duration under load. A worker running in a Kubernetes pod with a 0.5 CPU limit and a co-located noisy neighbor can be throttled for minutes.

*Failure path:* Worker A acquires the lock. CPU throttling pauses the process for 75 seconds. The lock expires at t=60. Worker B acquires the lock at t=61. Worker A resumes at t=75. Worker A's heartbeat thread fires, finds `result == 0`, exits. Worker A's main thread has no visibility into this. Worker A completes `_dispatch()`, enters the database transaction, and commits. Worker B does the same. Duplicate execution.

*Required protection:* Fencing token at the database write. The protection cannot live in the locking code because the process was not running during the pause — it had no opportunity to observe the expiry or respond to it. Any in-process mechanism (threading.Event, stop_event, flag) is inert during a process pause. The only layer that can enforce correctness is the one that remains active during the pause: the database, enforcing a monotonic write counter predicate.

---

**5. RENEWAL_INTERVAL / CLAIM_TTL ratio allows expiry between heartbeat cycles under normal operation**

*Trigger:* CLAIM_TTL is 60 seconds. RENEWAL_INTERVAL is 15 seconds. Under normal operation, the heartbeat renews at t=15, t=30, t=45 — the lock never expires. But `stop_event.wait(timeout=15)` is a wall-clock wait that can be extended by OS scheduler jitter, thread starvation, or GC. In a JVM-style GC environment this is well-understood; in CPython, the GIL and GC can delay thread scheduling by seconds under load. If the renewal thread is delayed by more than 60 seconds from the last renewal, the lock expires before the next heartbeat.

*Failure path:* Same as failure mode 4, but triggered by thread scheduling delay rather than process pause. The renewal thread is runnable but not scheduled. The lock expires. Another worker acquires it. The main thread continues.

*Required protection:* The ratio of RENEWAL_INTERVAL to CLAIM_TTL provides a safety margin of 4x, which is reasonable but not a guarantee under adversarial scheduling. The operational mitigation is to set CLAIM_TTL conservatively high relative to observed p99 job execution times, and to monitor renewal latency. The architectural protection remains the fencing token — the ratio buys time but does not eliminate the race.

---

**6. Redis key eviction under memory pressure silently invalidates a held lock**

*Trigger:* Redis is configured with `maxmemory-policy allkeys-lru` or `allkeys-random` (common in shared Redis deployments). Under memory pressure, Redis evicts keys regardless of TTL. A held lock key is evicted. The next `SET NX EX` from another worker succeeds.

*Failure path:* Worker A holds the lock. Redis evicts `job:claim:{job_id}` under memory pressure. Worker B acquires the lock. Worker A's next heartbeat finds `result == 0` (key does not exist, token mismatch), exits silently. Worker A continues executing. Duplicate execution.

*Required protection:* Redis must be configured with `maxmemory-policy noeviction` for any namespace used for distributed locking. This is an operational configuration requirement, not enforceable from within the application code. The application cannot detect eviction from acquisition success alone. This must be enforced at the Redis deployment layer and verified in the deployment checklist. The fencing token at the database layer remains the backstop if this configuration requirement is not met.

---

**7. Lua script execution is not atomic with respect to Redis Cluster key slot boundaries**

*Trigger:* If this deployment uses Redis Cluster (not standalone), Lua scripts that reference multiple keys require all keys to reside in the same hash slot. The current scripts reference only one key (`KEYS[1]`), so this is not a current vulnerability. However, if the key schema is extended (e.g., adding a fencing counter key alongside the claim key), a naive implementation would break atomicity in Cluster mode unless both keys use the same hash tag.

*Failure path:* Not triggered by current code. Named here because the fencing token fix — the required remediation for every failure mode above — introduces a second key. If that key is added without hash tag alignment, the Lua script will fail in Cluster mode, and the fix that was supposed to close the zombie-write gap will itself be broken silently or with a CROSSSLOT error.

*Required protection:* When the fencing token key is added, both the claim key and the counter key must use Redis hash tags to force co-location: `{job:claim:job_id}:lock` and `{job:claim:job_id}:fence`. This must be a design requirement on the fencing token implementation, not an afterthought.

---

**Open gaps**

1. **No fencing token / monotonic write counter at the database layer.** Every failure mode above converges on this gap. The database write (`record_execution`, `mark_job_complete`) has no predicate that enforces "this write is only valid if no other worker has written since I acquired the lock." Without this, any scenario that produces two valid lock holders — clock skew, failover, partition, process pause, eviction — produces a duplicate execution that commits successfully. This is the primary open gap. It is not addressable within the locking code.

2. **Heartbeat thread exception handling on network error.** A Redis connection error during renewal terminates the heartbeat thread via unhandled exception. The main thread receives no signal. This is distinct from the `result == 0` path and is not covered by the current implementation. The thread exits; the main thread continues; the lock has effectively been abandoned without notification.

3. **Redis memory policy not enforced.** The application assumes Redis will not evict lock keys under memory pressure. This assumption is not verified at startup and not documented as a deployment requirement.

4. **Failover window not documented.** The implementation does not document that Redis primary failover produces a correctness window. Operators deploying this against a Sentinel or Cluster topology will not know this gap exists.

5. **No fencing counter key co-location design.** The fencing token fix has not been designed yet. When it is, Cluster-mode key slot alignment must be a stated requirement.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system implements a well-structured distributed lock with correct token-verified acquisition, renewal, and release. The Lua scripts are correct. The retry logic is reasonable. The idempotency check and database transaction are present. None of that is the problem.

The problem is that every guarantee this lock provides is contingent on an assumption the infrastructure does not honor unconditionally: that the lock, once acquired, remains valid until explicitly released or until the heartbeat fails to renew it in a detectable way. Under clock skew, Redis failover, network partition, process pause, or key eviction, the lock can be invalidated by an event the worker never observes. When that happens, the worker continues executing with full confidence that it holds an exclusive claim. It does not. A second worker holds the same claim. Both complete the job.

The idempotency check (`db.get_execution()`) does not close this gap. It is evaluated once, before dispatch, under the assumption that the lock is still valid at that moment. It provides no protection against a second worker that acquires the lock after the check passes and before the database write commits.

The required fix is a fencing token — a monotonic counter incremented on every lock acquisition, embedded in the lock value, and enforced as a predicate on the database write. The write must fail if the counter in the database is greater than or equal to the counter the worker holds. This is the only protection that remains active during a process pause, a network partition, or a failover — because it lives in the database, which is not subject to the same availability events as the lock store.

This system must not ship to production in its current form for any job type where duplicate execution produces an incorrect outcome. The conditions for approval are: (1) fencing token implemented at the database write layer with a `WHERE` predicate on the monotonic counter; (2) heartbeat thread exception handling on network error, with signal to the main thread; (3) Redis memory eviction policy documented and enforced as a deployment requirement; (4) failover window documented in the operational runbook; (5) fencing counter key designed for hash-slot co-location if Redis Cluster is in the topology.
