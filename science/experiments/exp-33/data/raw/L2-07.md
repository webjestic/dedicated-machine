**Layer 1 coverage:** Correctness review confirmed the zombie-write vulnerability — heartbeat exits silently on lock loss while the main thread continues executing — and identified fencing token on writes as the required architectural fix; secondary findings covered swallowed exceptions, unobservable release, API contract violation, and absent lock-loss test coverage.

---

**Infrastructure failure modes**

**1. Process pause exceeds CLAIM_TTL before heartbeat fires**

*Trigger:* A GC pause, JVM safepoint equivalent (Python stop-the-world during GC in CPython is brief but not zero; more critically, a slow external call in `_dispatch()` — network I/O, a downstream service stall, a large payload deserialization — holds Process A past 60 seconds before the heartbeat thread has had any opportunity to renew. The heartbeat runs every 15 seconds, but if the pause spans the full TTL without a renewal completing, the lock expires naturally.*

*Failure path:* Process A holds the lock. A pause or stall longer than CLAIM_TTL elapses. Redis expires the key by TTL — no notification to Process A, no signal to the heartbeat thread, no signal to the main thread. Process B calls `SET NX EX` on the same key, succeeds, acquires the lock legitimately. Process B passes the idempotency check (`get_execution` returns None — no execution has been recorded yet), dispatches the job, and begins writing. Meanwhile Process A's pause ends. Process A's main thread resumes inside `_dispatch()` or proceeds to the database transaction — it has no knowledge the lock expired. The heartbeat fires, `RENEW_CLAIM_SCRIPT` returns 0 (token mismatch), the heartbeat thread exits silently. Process A's main thread is not informed. Process A completes `_dispatch()`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`. Process B does the same. Both writes commit. The job executes twice. For any non-idempotent job — payment charge, alert dispatch, external API call — this is a correctness violation.*

*Why the protection cannot live inside the locking code:* The lock expiry is an event that happens inside Redis, asynchronously, with no channel back to the process that held the lock. The locking code can detect the loss only at the next heartbeat interval — up to 15 seconds after the fact, by which time the main thread may have already passed the idempotency check and entered execution. Thread signaling (threading.Event) closes the window but does not eliminate it: there is a race between the heartbeat detecting loss and the main thread completing the write. The only protection that works regardless of timing is a fencing token — a monotonic counter or conditional write token — that makes the database write itself reject a stale claim. The write must be conditioned on the token still being valid at write time, not on the lock being held at execution start. This protection must live at the database write layer because that is the only operation that can be made atomic with the validity check.*

*Required protection:* Fencing token issued at lock acquisition time (e.g., Redis `INCR` on a monotonic counter per job_id, returned as the token), passed through to `record_execution` and `mark_job_complete`, with the database transaction conditioned on `WHERE fencing_token = :expected AND job_status = 'pending'`. If Process A's write arrives after Process B has already written, the conditional update finds either the wrong token or the wrong status and rejects it. This is the only mechanism that makes the write layer safe against a lock that has already been invalidated by the time the write executes.*

**2. Redis unavailability during heartbeat renewal**

*Trigger:* Redis becomes unreachable mid-execution — network partition, Redis node failover, connection pool exhaustion. The heartbeat thread attempts `_renew_script()` and receives a `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`.*

*Failure path:* The current `_renew()` implementation has no exception handling around `self._renew_script(...)`. A Redis exception propagates out of `_renew()`, terminates the heartbeat thread with an unhandled exception, and is silently swallowed by the daemon thread runtime. The main thread continues executing. The lock's TTL is not renewed. After CLAIM_TTL seconds, Redis expires the key. Process B acquires the lock. The failure path from here is identical to failure mode 1: both processes complete the job.*

*Compounding factor:* The main thread has no visibility into whether the heartbeat thread is alive. `t.join(timeout=self.RENEWAL_INTERVAL + 1)` in the `finally` block will return immediately if the thread has already exited — it does not distinguish between clean exit and crash exit. The release script then attempts to run against an unavailable Redis, which may also fail silently (no exception handling on `self._release_script(...)` in the `finally` block). The claim key may or may not be present; the outcome is undefined.*

*Required protection:* Heartbeat thread must catch Redis exceptions and surface them — at minimum by setting a shared error flag that the main thread checks before writing. More robustly: the fencing token protection from failure mode 1 makes this moot at the write layer regardless. But the exception must not be silently swallowed; it must be observable. This is an observability requirement independent of correctness: operators need to know when heartbeat renewal is failing, because it means the locking layer is degraded.*

---

**3. Redis cluster failover invalidates lock during execution (Redlock / replica promotion)**

*Trigger:* The deployment uses Redis Sentinel or Redis Cluster with replication. Process A writes the lock to the primary. Before the write is replicated, the primary fails. Sentinel promotes a replica. The replica does not have the lock key. Process B calls `SET NX EX` on the new primary, succeeds.*

*Failure path:* Both Process A and Process B believe they hold a valid lock simultaneously. Process A's heartbeat renews against the new primary — but the key it is renewing is the one Process B set, with Process B's token. The renewal returns 0 (token mismatch). The heartbeat exits. Process A's main thread continues. Both processes complete the job. This is the standard Redlock split-brain scenario. It is not a code defect; it is a property of Redis replication under failure.*

*Why the protection cannot live in the locking code:* No client-side locking code can detect a replica promotion that happened between the SET and the next read. The lock appears valid from the client's perspective. The only protection is either: (a) use `WAIT` to require synchronous replication before considering the lock acquired — which degrades availability and does not fully eliminate the window; or (b) accept that Redis-based locking does not provide a hard mutual exclusion guarantee under failover, and protect the write layer with a fencing token as described in failure mode 1. The fencing token must be stored in a system with stronger consistency guarantees than Redis if the guarantee must hold across Redis failover — typically the same database that records job completion.*

*Required protection:* Fencing token on writes (same as failure mode 1), sourced from the authoritative database rather than Redis, so that the token's validity is determined by a system that does not have the split-brain exposure.*

---

**4. Clock skew between workers causes premature TTL expiry**

*Trigger:* CLAIM_TTL is set in seconds and applied via Redis `EXPIRE`. Redis uses its own internal clock. If the Redis server clock is skewed forward relative to the acquiring worker's clock, the effective TTL from the worker's perspective is shorter than CLAIM_TTL. A worker that believes it has 60 seconds may find its lock expired after 45 seconds of wall-clock time on the worker.*

*Failure path:* The heartbeat fires at 15-second intervals measured by the worker's clock. If the Redis clock is running fast, the lock may expire between heartbeat intervals even though the worker believes the renewal is on schedule. The failure path from here is identical to failure mode 1.*

*Required protection:* This is a configuration and operational concern, not a code concern — NTP synchronization between all nodes including Redis hosts is a deployment requirement. The code cannot defend against this from within. The fencing token on writes remains the backstop.*

---

**5. Idempotency check and job dispatch are not atomic with lock validity**

*Trigger:* This is not a single-event trigger but a structural gap. The sequence inside `_claim_job` is: (1) acquire lock, (2) check `get_execution`, (3) dispatch job, (4) write results. Steps 2 through 4 are not atomic with respect to lock validity. The lock can be lost at any point between step 1 and step 4.*

*Failure path:* Process A acquires the lock and passes the idempotency check at step 2 — `get_execution` returns None. Process A enters `_dispatch()`. The lock expires (any of the triggers above). Process B acquires the lock, passes the idempotency check (still None — Process A has not written yet), and enters `_dispatch()`. Both processes reach step 4 and write. The idempotency check is not a guard against this scenario; it is only a guard against re-execution of a job that has already been fully committed. It provides no protection during the window between check and write.*

*This is the same failure mode as 1, stated from the structural perspective: the idempotency check creates a false sense of safety. It will catch the case where a job is submitted twice after the first run completes. It will not catch the case where two workers are executing the same job simultaneously because the lock was lost mid-execution.*

*Required protection:* Fencing token on the write, as above. The database write must be the serialization point, not the lock acquisition.*

---

**6. `finally` block release attempt after lock has already expired**

*Trigger:* Lock expires during execution (any trigger above). The `finally` block in `_claim_job` calls `self._release_script(keys=[claim_key], args=[token])`. At this point, either the key no longer exists, or it exists with Process B's token.*

*Failure path:* If the key does not exist, the release script's `GET` returns nil, the comparison fails, the script returns 0 — harmless. If the key exists with Process B's token, the comparison fails, the script returns 0 — harmless. This is correct behavior from the Lua script's perspective. However: if `self._release_script(...)` raises a Redis exception (Redis unavailable), the exception propagates out of the `finally` block and may suppress any exception that was already propagating from the `try` block. This can cause the caller to receive a misleading exception or no exception at all, masking the actual failure.*

*Required protection:* The release call in the `finally` block must be wrapped in a try/except that logs the failure and does not re-raise. The release is best-effort; the token-verification in the Lua script already ensures it cannot release another worker's lock. The exception must not escape the `finally` block.*

---

**7. Deployment: multiple versions running simultaneously during rolling deploy**

*Trigger:* A rolling deployment leaves old and new worker versions running simultaneously. If the new version changes CLAIM_TTL, RENEWAL_INTERVAL, the claim key format, or the token format, old and new workers may interact in undefined ways.*

*Failure path:* If the claim key format changes (e.g., `job:claim:{job_id}` becomes `job:lock:{job_id}`), old and new workers use different keys and do not see each other's claims. Both acquire their respective keys and execute the same job simultaneously. This is not hypothetical — key format changes happen during refactors, and the old key format is not documented as a deployment contract.*

*Required protection:* The claim key format must be treated as a versioned deployment contract. Changes to it require a coordinated cutover, not a rolling deploy. This must be documented and enforced at the deployment layer, not the code layer. The code cannot protect against this from within.*

---

**Open gaps**

1. **No fencing token on database writes.** The write layer (`record_execution`, `mark_job_complete`) has no mechanism to reject a write from a process whose lock has already been superseded. This is the primary unresolved gap. The idempotency check does not close it.

2. **Heartbeat exception handling absent.** Redis exceptions in `_renew()` are silently swallowed by the daemon thread. The main thread has no visibility into heartbeat health. No alerting surface exists for heartbeat failure.

3. **No exception guard on `finally` release.** A Redis exception during the release call in the `finally` block can suppress upstream exceptions and produce misleading error propagation.

4. **Redis replication guarantee not specified.** The deployment configuration does not specify whether Redis is standalone, Sentinel, or Cluster, and does not document the replication guarantee assumed by the locking implementation. If replication is asynchronous, the split-brain failure mode (failure mode 3) is unmitigated.

5. **Claim key format not documented as a deployment contract.** Rolling deploys that change the key format will silently disable mutual exclusion between old and new workers. This is not documented anywhere in the PR.

6. **CLAIM_TTL adequacy not validated against actual job duration distribution.** 60 seconds is asserted as "generous for slow jobs" with no supporting measurement. If any job type routinely exceeds 60 seconds and the heartbeat fails to renew (for any reason), the lock expires and the zombie-write path opens. The TTL must be validated against observed p99 job durations in production, with margin.

---

**Production readiness verdict: NOT APPROVED WITH CONDITIONS**

This system implements the structural components of distributed claim locking — atomic acquisition, token-verified renewal and release, background heartbeat, idempotency check, database transaction — and does so with visible care. Under nominal conditions, with a healthy Redis instance, no process pauses, and no concurrent failures, it will prevent duplicate execution. That is the easy case. The system is not approved because it does not protect against the cases that actually occur in production.

The central unresolved gap is this: the database write layer has no mechanism to reject a write from a process whose lock has already been superseded. The lock can be lost — through process pause, GC, slow external call, Redis TTL expiry, network partition, or replica failover — at any point between lock acquisition and the final write. The idempotency check does not protect against this; it only catches re-execution of already-completed jobs. During the window between check and write, two processes can be executing the same job simultaneously with no mechanism to detect it at write time. For any non-idempotent job, both writes commit and the job executes twice.

Before this ships, the following must be in place: a fencing token sourced from a monotonically increasing counter in the authoritative database, passed through to every write operation, with the write conditioned on the token being current and the job status being uncommitted. This makes the write the serialization point rather than the lock acquisition. The lock remains useful — it reduces contention and prevents redundant work under normal conditions — but it is no longer the sole correctness mechanism. Exception handling must be added to the heartbeat thread, and the release call in the `finally` block must be guarded. The Redis deployment topology must be documented and the replication guarantee must be explicit. The claim key format must be documented as a deployment contract with a coordinated cutover requirement. None of these are optional for a system that processes payments, sends alerts, or triggers any external action that cannot be undone.
