**Findings**

---

**1. Silent Heartbeat Exit Does Not Signal Main Thread — Zombie Write Enabled**

*Issue name:* Undetected lock loss; main thread continues executing after claim expiry.

*Failure path:*

Process A acquires `job:claim:{job_id}` with token T1. A GC pause, slow external call in `_dispatch`, or slow database operation holds Process A for longer than `CLAIM_TTL` (60s) without a heartbeat firing. The key expires. Process B calls `SET NX EX`, acquires the same key with token T2, and begins executing the same job. Process A's heartbeat fires next — `RENEW_CLAIM_SCRIPT` evaluates `GET job:claim:{job_id}` → T2 ≠ T1 → returns 0. The `_renew` thread hits `if result == 0: return` and exits. The `stop_event` is never set. The main thread in `execute()` has no reference to the heartbeat thread's exit condition and no mechanism to observe it. Process A continues through `_dispatch`, enters `self.db.transaction()`, calls `record_execution` and `mark_job_complete`. Process B does the same. The job executes twice. For any non-idempotent job — payment charge, alert send, external API call — this is a correctness violation.

The checklist entry "Atomic claim renewal via Lua script (token-verified)" is accurate but incomplete: it describes the mechanism, not the behavior on failure. The gap is not in the Lua script; it is in the absence of any signal from the heartbeat thread to the main thread when that script returns 0.

*Required fix:*

The heartbeat thread must signal the main thread on lock loss. The minimal fix is to set `stop_event` on `result == 0` and check it in the main thread before proceeding past lock-sensitive operations:

```python
# In _renew():
if result == 0:
    logger.error("Claim lost — token mismatch job_id=%s", job_id)
    stop_event.set()   # signal main thread
    return

# In _claim_job(), after yield returns or at a check point:
# The main thread must poll or the context manager must raise on exit.
```

However, `stop_event.set()` alone is insufficient as a correctness fix — it is a best-effort signal. The main thread checks nothing between `yield token` and the `finally` block. The signal arrives after the job has already executed.

*Architectural fix required:*

The correct fix is a **fencing token** (monotonic counter) rather than a UUID. Redis provides this via `INCR` or Redlock's lock version. The token issued at acquisition is a monotonically increasing integer. Every write to the database or external system includes the token. The database (or external system) rejects writes with a token lower than the last committed token. This means Process A's writes are rejected at the storage layer even if the main thread never learns the lock was lost. This is the only fix that closes the race without relying on in-process coordination between threads that share no synchronization boundary during job execution.

`threading.Event` signaling is a partial mitigation — it reduces the window but does not eliminate it, because the signal is asynchronous and the main thread can complete writes before the signal is observed.

**2. Lock Release Attempted After Lock Has Already Expired or Been Reacquired**

*Issue name:* Stale release attempt on expired or foreign claim.

*Failure path:*

This is a consequence of Finding 1. After Process A's job completes (having executed without a valid claim), the `finally` block calls `self._release_script(keys=[claim_key], args=[token])`. At this point the key either no longer exists (expired) or holds Process B's token. The Lua script correctly returns 0 in both cases and does not delete. This is safe — the script is correctly token-verified. However, if the key holds Process B's token and the TTL is near expiry, the no-op release is silent. No error is logged. The caller has no indication the release was a no-op versus a successful release. This is a secondary observability gap, not a new correctness violation beyond Finding 1, but it should be logged.

*Required fix:*

```python
released = self._release_script(keys=[claim_key], args=[token])
if not released:
    logger.error("Claim release was a no-op — token mismatch or key expired job_id=%s", job_id)
```

**3. Heartbeat Renewal Interval Relative to TTL Provides No Safety Margin Under Load**

*Issue name:* Renewal interval too close to TTL under thread starvation.

*Failure path:*

`CLAIM_TTL = 60`, `RENEWAL_INTERVAL = 15`. Under normal conditions this is a 4:1 ratio — adequate. However, `stop_event.wait(timeout=15)` is subject to Python's GIL and OS scheduling. Under GC pressure or CPU saturation, the heartbeat thread may not be scheduled for longer than 15 seconds. If the thread is starved for `>60s` from the last successful renewal, the key expires. This is the same trigger as Finding 1 but via thread starvation rather than main-thread pause. The ratio is not itself a bug, but the system has no defense against it other than the ratio — and the ratio is not documented as a design constraint.

*Required fix:*

This is mitigated by the architectural fix in Finding 1 (fencing tokens). Without that fix, document the ratio as a hard operational constraint and consider reducing `RENEWAL_INTERVAL` to `CLAIM_TTL / 5` or lower. This is a secondary concern; Finding 1 is the primary issue.

**4. Redis Unavailability During Acquisition Is Not Distinguished From Contention**

*Issue name:* Redis failure surfaces as `ClaimError`; caller cannot distinguish network failure from lock contention.

*Failure path:*

`self.redis.set(claim_key, token, nx=True, ex=self.CLAIM_TTL)` raises `redis.RedisError` if Redis is unavailable. This exception is not caught in `_claim_job`. It propagates through `execute()`, is not caught by the `except ClaimError` block, and surfaces as an unhandled exception to the caller. The `JobResult` contract is broken — the caller receives an exception rather than a structured result. Whether this is acceptable depends on the caller contract, but it is inconsistent with how `ClaimError` is handled.

*Required fix:*

Catch `redis.RedisError` explicitly in `_claim_job` or in `execute()` and return a structured `JobResult` with an appropriate message, or re-raise as a typed exception the caller can handle. At minimum, log it at ERROR level before propagation.

```python
except redis.RedisError as e:
    logger.error("Redis unavailable during claim acquisition job_id=%s: %s", job_id, e)
    return JobResult(claimed=False, message="Claim service unavailable")
```

**5. Redis Unavailability During Heartbeat Renewal Is Silently Treated as Lock Loss**

*Issue name:* Transient Redis failure causes heartbeat to exit as if lock was lost.

*Failure path:*

If `self._renew_script(...)` raises `redis.RedisError` (connection timeout, Redis restart), the exception propagates out of `_renew()` uncaught. The thread exits via exception. From the main thread's perspective this is identical to the lock-loss path in Finding 1 — the heartbeat is gone, the main thread continues. A transient Redis blip causes the heartbeat to abandon renewal permanently, after which the key will expire and Finding 1's failure path applies.

*Required fix:*

Wrap the renewal call in a try/except. On transient error, log and continue the renewal loop (do not exit). After N consecutive failures, treat as unrecoverable and signal the main thread.

```python
try:
    result = self._renew_script(keys=[claim_key], args=[token, str(self.CLAIM_TTL)])
except redis.RedisError as e:
    logger.warning("Renewal attempt failed job_id=%s: %s", job_id, e)
    consecutive_failures += 1
    if consecutive_failures >= MAX_RENEWAL_FAILURES:
        logger.error("Renewal unrecoverable job_id=%s", job_id)
        stop_event.set()
        return
    continue
consecutive_failures = 0
if result == 0:
    ...
```

**6. Test Suite Does Not Cover the Failure Mode**

*Issue name:* No test for heartbeat-exit / main-thread-continues scenario.

*Failure path:*

The test `test_claim_release_uses_lua_script` verifies the release script is called. No test simulates `_renew_script` returning 0 and verifies that the main thread does not proceed with job execution. The gap in the production code is untested, which means any fix must also add a test that would have caught this.

*Required fix:*

Add a test that configures `_renew_script` to return 0 on first call and verifies that `db.record_execution` is not called — i.e., the main thread halts when the heartbeat detects lock loss. This test will fail against the current code and pass only after Finding 1 is fixed.

---

**Correctness verdict:** NOT APPROVED

Finding 1 is a confirmed zombie-write vulnerability. Under a GC pause or slow external call exceeding `CLAIM_TTL`, two workers will execute the same job concurrently. The heartbeat correctly detects the loss and exits; the main thread has no mechanism to observe this. Findings 4 and 5 are additional correctness gaps in error handling. The code is well-structured and the locking primitives are correctly implemented in isolation — the failure is architectural: the heartbeat thread's exit state is not observable by the execution thread.

**Handoff summary for production readiness review:**

This review traced the full lock acquisition, renewal, and release path under concurrent execution, including GC-pause and thread-starvation scenarios. The central finding is a zombie-write vulnerability: heartbeat thread exit on lock loss is not signaled to the main execution thread, allowing two workers to complete the same job; the required fix is a fencing token / monotonic counter at the storage layer, not in-process thread signaling. Secondary findings cover Redis unavailability handling (unstructured exceptions on acquisition and silent heartbeat abandonment on transient failure), a stale-release observability gap, and absent test coverage for the failure path. This review does not cover infrastructure-level failure modes (Redis cluster failover, network partition behavior, or Redlock correctness under clock skew).
