**Findings**

---

**1. Heartbeat exit does not signal main thread — zombie write on lock loss**

*Issue name:* Silent heartbeat exit / missing lock-loss signal

*Failure path:*

Process A acquires the job lock (token T1) and begins execution. A GC pause, slow external call, or scheduler delay holds Process A for longer than `CLAIM_TTL` (60 seconds). The lock expires in Redis. Process B calls `SET NX EX` on the same key, acquires it with token T2, and begins executing the same job.

Process A's heartbeat fires. `RENEW_CLAIM_SCRIPT` runs `GET job:claim:{job_id}` and finds T2, not T1. The script returns 0. `_renew()` hits `if result == 0: return` and exits — silently, with no signal to the main thread.

Process A's main thread is executing inside `with self._claim_job(req.job_id):`. It has no visibility into the heartbeat thread's exit. It proceeds through `_dispatch()`, enters `self.db.transaction()`, calls `record_execution()` and `mark_job_complete()`. Process B does the same. The job executes twice.

For a job that sends an alert, charges a payment, or triggers an irreversible external action, this is a correctness violation. The idempotency check (`db.get_execution`) does not protect against this: both processes pass it before either writes the completion record.

*Required fix — two layers:*

**Layer 1 (immediate / necessary):** `_renew()` must signal the main thread on lock loss. The `stop_event` is already in scope. On `result == 0`, call `stop_event.set()` before returning. The main thread must check this signal at a defined point — the cleanest approach is a second event (`_lost_event = threading.Event()`) that `_renew()` sets on loss, and the `finally` block (or the `yield` site) checks it and raises before proceeding to release or allowing the caller to continue.

Concretely:

```python
lost_event = threading.Event()

def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        result = self._renew_script(
            keys=[claim_key],
            args=[token, str(self.CLAIM_TTL)]
        )
        if result == 0:
            lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)

# After yield returns (in finally):
stop_event.set()
t.join(timeout=self.RENEWAL_INTERVAL + 1)
if lost_event.is_set():
    raise ClaimError(f"Claim lost during execution for job {job_id}")
```

The caller (`execute()`) must catch `ClaimError` from inside the `with` block, not just from acquisition. Currently `ClaimError` is only raised at acquisition time; the `with` block's body is not wrapped for it. The `finally` block raising after `yield` will propagate through the `with` block correctly — verify this is handled at the `execute()` call site.

**Layer 2 (architectural / required for correctness under adversarial timing):** Layer 1 closes the window for the common case but does not eliminate it. There is a residual race: Process A's lock expires, Process B acquires it, Process A's heartbeat fires and sets `lost_event`, but Process A's main thread is already past the point where it would check — it is mid-`_dispatch()` or mid-`db.transaction()`. The signal arrives too late.

The correct architectural fix is a **fencing token / monotonic counter**: Redis returns a monotonically increasing integer when the lock is acquired (e.g., via a Redis counter incremented atomically at acquisition, or Redlock's recommended fencing approach). This token is passed to every downstream operation — database writes, external calls. The database layer rejects writes carrying a token lower than the highest token it has seen for this job. This makes stale writers fail at the write boundary regardless of when they detect lock loss.

Without a fencing token, the heartbeat signal is a best-effort mitigation, not a correctness guarantee.

**2. Lock-loss exception not handled at execute() call site**

*Issue name:* Unhandled `ClaimError` from post-acquisition lock loss

*Failure path:*

After applying Finding 1 Layer 1, `_claim_job`'s `finally` block can raise `ClaimError`. This exception propagates out of the `with self._claim_job(...)` block in `execute()`. The current `except ClaimError` handler in `execute()` catches it — but only because it wraps the entire `with` block. Verify this remains true after the fix: the `raise` in `finally` must not be swallowed by any intermediate exception handling, and the returned `JobResult` must accurately reflect lock loss (not "claimed by another worker," which is misleading — the job may have partially executed).

*Required fix:* Distinguish acquisition failure from mid-execution lock loss in the `ClaimError` message or via a subclass. The caller needs to know whether the job was never started or was started and then lost the lock — these have different recovery semantics.

---

**3. Heartbeat thread not checked for Redis errors**

*Issue name:* Silent renewal failure on Redis exception

*Failure path:*

`_renew()` calls `self._renew_script(...)` with no exception handling. If Redis is unavailable during renewal, this raises `redis.RedisError` (or a subclass). The exception propagates out of `_renew()`, terminating the thread with an unhandled exception. The main thread continues executing with an expiring, unrenewed lock — functionally identical to the `result == 0` case, but without even the silent exit path. The `lost_event` (after Fix 1) is never set because the exception bypasses that branch.

*Required fix:*

```python
def _renew() -> None:
    while not stop_event.wait(timeout=self.RENEWAL_INTERVAL):
        try:
            result = self._renew_script(
                keys=[claim_key],
                args=[token, str(self.CLAIM_TTL)]
            )
        except redis.RedisError as e:
            logger.error("Claim renewal failed job_id=%s: %s", job_id, e)
            lost_event.set()
            return
        if result == 0:
            lost_event.set()
            return
        logger.debug("Claim renewed job_id=%s", job_id)
```

---

**4. Redis unavailability at acquisition not handled**

*Issue name:* Unhandled `RedisError` at lock acquisition

*Failure path:*

`self.redis.set(claim_key, token, nx=True, ex=self.CLAIM_TTL)` raises `redis.RedisError` if Redis is unavailable. This propagates out of `_claim_job` as an unhandled exception, bypassing the `ClaimError` handler in `execute()`, and surfaces as an unhandled exception to the caller.

*Required fix:* Wrap the `set` call in `_claim_job` (and the release script call in `finally`) with `redis.RedisError` handling. Convert to `ClaimError` or a dedicated `RedisUnavailableError` and handle at the `execute()` boundary. The `finally` block's release call has the same exposure — a `RedisError` there would suppress any in-flight exception from the `try` block.

---

**5. Test suite does not cover the lock-loss scenario**

*Issue name:* Missing test for heartbeat exit / main thread continuation

*Failure path:*

No test simulates `_renew_script` returning 0 and verifies that `execute()` does not complete the job. The mock fixture hardcodes `script_mock = MagicMock(return_value=1)` for all script calls. The critical failure mode is untested.

*Required fix:* Add a test that configures `_renew_script` to return 0 on first call, triggers a heartbeat cycle, and asserts that `record_execution` is not called and the returned `JobResult` reflects lock loss. This requires either exposing the `lost_event` for inspection or testing via timing/threading with appropriate synchronization.

---

**Correctness verdict:** NOT APPROVED

The heartbeat thread exits silently on lock loss (Finding 1) without signaling the main thread. The main thread continues executing and completes the job. This is the primary correctness violation. Findings 2–4 are required fixes before approval. Finding 5 must be addressed to give the fix meaningful test coverage. The architectural fix (fencing token) is required for a correctness guarantee; the threading.Event signal is a necessary but insufficient mitigation.

**Handoff summary for production readiness review:**

This review traced the full lock acquisition, renewal, and release path under concurrent execution, including GC-pause and Redis-unavailability scenarios. The primary finding is a structural correctness violation: the heartbeat thread exits silently on lock loss, and the main thread has no signal path to detect this, allowing two workers to complete the same job concurrently; the required fix is a `threading.Event` signal from heartbeat to main thread (necessary) combined with a fencing token / monotonic counter passed to downstream writes (required for a correctness guarantee under adversarial timing). Three secondary findings cover unhandled Redis exceptions at acquisition and renewal, and a missing test for the lock-loss path — all must be resolved before approval. This review does not cover infrastructure failure modes, Redlock/multi-node Redis topology, deployment configuration, or performance characteristics under load.
