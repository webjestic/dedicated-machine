**Layer 1 coverage:** Correctness review confirmed the zombie-write path — heartbeat exits silently on lock loss, main thread commits without awareness, no fencing at the database layer — and identified threading.Event signaling as insufficient without an optimistic concurrency predicate at commit time.

**Infrastructure failure modes**

**1. Redis clock skew / key expiry fires before TTL elapses from worker's perspective**

*Trigger:* Redis server time and worker wall clock diverge — common after NTP correction, VM live migration, or container host clock adjustment. Redis expires keys on server time. The worker calculates renewal intervals on local time. If server time runs ahead, the key expires before the worker's RENEWAL_INTERVAL fires. If server time runs behind, the worker believes the key is live when Redis has already evicted it.

*Failure path:* Worker A acquires the lock. Server clock jumps forward 20 seconds. The key expires at second 40 server-time while worker A's local clock reads second 38 — before the first renewal fires at second 45 local-time. Worker B acquires the lock. Worker A's heartbeat fires, calls RENEW_CLAIM_SCRIPT, gets 0 back, and exits silently. Worker A's main thread continues executing. Both workers reach `_dispatch` and commit. Duplicate execution.

*Required protection:* The protection cannot live in the locking code because the locking code has no authoritative view of server time — it can only observe the consequence (renewal returns 0) after the race has already opened. The protection must be a fencing token written into the database transaction predicate: the commit succeeds only if the token in the database matches the token this worker holds, and that token was written by a lock acquisition that is still valid at commit time. Clock skew is invisible to application code; the database predicate makes the commit self-invalidating when the race has occurred regardless of cause.

---

**2. Redis network partition — worker isolated from Redis mid-execution**

*Trigger:* Network partition between the worker host and Redis (firewall rule change, AZ connectivity event, Redis node failover) occurring after lock acquisition but before execution completes.

*Failure path:* Worker A holds the lock. Network partition isolates worker A from Redis. Renewal calls fail — but the failure mode here depends on the Redis client timeout configuration, which is not set in this code. If the client blocks indefinitely (default socket timeout is None in redis-py), the renewal thread hangs on the network call. stop_event.wait() is not blocking the renewal thread — the blocking call is the Redis command itself. The renewal thread is now stuck inside `_renew_script(...)`, not checking stop_event, not returning. Meanwhile the CLAIM_TTL expires on Redis. Worker B acquires the lock and begins execution. Worker A's main thread is executing `_dispatch` — it has no visibility into the partition. Both workers execute. When the partition heals, worker A's renewal call either returns 0 (lock gone) or raises a connection error — both cases result in the heartbeat thread exiting or crashing with no signal to the main thread. Worker A commits. Duplicate execution.

*Required protection:* Two layers required. First, the Redis client must be configured with explicit socket_connect_timeout and socket_timeout — this is deployment configuration that must be enforced outside the application code, because the application cannot set its own timeout after the client is injected. Second, the renewal thread must treat Redis exceptions as lock-loss events and signal accordingly — but even with that signal, the main thread cannot safely abort a partially-executed job. The durable protection is again the database-layer fencing predicate: the commit must be conditional on lock validity, and lock validity must be verifiable at commit time without relying on Redis being reachable. This means the fencing token must be written to the database at lock acquisition and validated at commit time within the same transaction, making the Redis partition irrelevant to commit correctness.

---

**3. Redis Cluster topology change — key migrates or slot moves mid-execution**

*Trigger:* Redis Cluster resharding, node failure with slot reassignment, or replica promotion during execution. The key `job:claim:{job_id}` lives in a specific slot. If that slot migrates, subsequent commands for that key may be redirected or fail.

*Failure path:* Worker A acquires the lock against node N1. Cluster resharding begins. The slot for `job:claim:{job_id}` migrates to N2. Worker A's renewal call is redirected by redis-py's cluster client — or, if using a non-cluster-aware client pointed at a single node, the call fails with MOVED error and the renewal thread crashes or exits. The key on N1 may have been deleted as part of migration; the key on N2 may not exist or may have been re-created by Worker B. Worker B acquires the lock on N2. Both workers execute. The Lua scripts provide no protection here because atomicity is per-node — the token check and the expire happen on whatever node currently owns the slot, which may not be the node that originally accepted the SET NX.

*Required protection:* Cluster topology events are invisible to the locking code. The redis-py client configuration — cluster-aware vs. single-node, MOVED handling, retry behavior — must be specified and validated in deployment configuration. More fundamentally, Redis Cluster does not provide the cross-slot atomicity guarantees that this locking pattern implicitly requires to be reliable. The operational runbook must document which Redis topology this code is validated against. The database-layer fencing predicate remains the only protection that survives topology events, because it does not depend on Redis remaining consistent across the event.

---

**4. Redis Sentinel failover — standby promoted, lock state lost or inconsistent**

*Trigger:* Redis Sentinel promotes a replica to primary during execution. The replica may be seconds behind the primary at the time of failover. The lock key written to the primary may not have replicated before the primary failed.

*Failure path:* Worker A acquires the lock on the primary. Primary fails. Sentinel promotes a replica that did not receive the SET NX before the primary crashed. The new primary has no record of the lock. Worker B calls SET NX, succeeds, acquires the lock, begins execution. Worker A's renewal call goes to the new primary, finds its token absent, gets 0 back, heartbeat exits silently. Worker A continues executing. Both workers execute. This is the Redlock problem in its canonical form — single-instance Redis with Sentinel does not provide the durability guarantee that this code assumes.

*Required protection:* This cannot be fixed in application code. The protection requires either: (a) Redis configured with `wait` for synchronous replication before acknowledging the SET, which has latency and availability tradeoffs that must be explicitly accepted; (b) Redlock across independent Redis instances, which has its own documented failure modes and requires careful implementation; or (c) accepting that the Redis lock is advisory only and the database-layer fencing predicate is the authoritative correctness mechanism. The deployment configuration must document which guarantee is being relied upon. The code as written assumes the lock is durable across failover — that assumption is not documented and is false for standard Sentinel deployments.

---

**5. Process pause after lock acquisition, before first renewal — TTL window**

*Trigger:* GC pause, OS scheduler preemption, container CPU throttling (cgroup CPU quota exhaustion), or VM steal time causes the worker process to pause for longer than CLAIM_TTL (60 seconds) before the first renewal fires.

*Failure path:* This is the canonical scenario from the Layer 1 review, but the infrastructure dimension is distinct: CPU throttling in containerized deployments is not a code-level event. A container hitting its CPU quota limit is paused by the kernel — the process has no visibility into this. CLAIM_TTL of 60 seconds is described as "generous for slow jobs" but provides no protection against a container that is throttled for 61 seconds, which is routine under aggressive CPU limits. Worker A is paused. Lock expires. Worker B acquires. Worker A resumes, heartbeat fires, gets 0, exits silently. Worker A executes. Duplicate execution.

*Required protection:* The CLAIM_TTL value must be set with knowledge of the container CPU limits and the expected worst-case pause duration — this is deployment configuration, not application configuration. The application cannot observe its own throttling. The RENEWAL_INTERVAL to CLAIM_TTL ratio (15s to 60s, giving 3 renewal windows) is reasonable but provides no protection if the pause exceeds CLAIM_TTL entirely. This is a configuration governance requirement: the deployment must document the CPU limits, the expected maximum pause duration, and the TTL chosen in relation to both. The database-layer fencing predicate is the only protection that survives a pause longer than CLAIM_TTL regardless of how TTL is tuned.

---

**6. Renewal thread silently swallows Redis exceptions — lock loss is undetectable**

*Trigger:* Any Redis error during renewal — connection timeout, READONLY error after failover, cluster MOVED error, OOM error — raises an exception inside `_renew_script(...)`. The `_renew()` function has no exception handler. An unhandled exception in a daemon thread is logged to stderr by the Python runtime and the thread terminates. The main thread receives no signal.

*Failure path:* Worker A's renewal thread encounters a Redis exception. The thread terminates. The main thread continues executing with no awareness that renewal has stopped. The lock expires at CLAIM_TTL. Worker B acquires the lock. Both workers execute. This failure mode is distinct from result == 0 (token mismatch) — it is a silent thread death from an infrastructure error, not a detected lock-loss condition. The existing code handles result == 0 by returning (still silently), but does not handle exceptions at all.

*Required protection:* Exception handling in the renewal thread is necessary but not sufficient — it can detect the failure but cannot safely abort `_dispatch` mid-execution. The renewal thread must catch Redis exceptions, treat them as lock-loss events equivalent to result == 0, and signal the main thread. But the main thread signal path has the same problem identified in Layer 1: checking the signal after `_dispatch` completes does not prevent the duplicate execution. The durable fix is the database-layer fencing predicate. The exception handling is a defense-in-depth requirement that must be added regardless.

---

**7. Worker process killed after `_dispatch` completes, before transaction commits**

*Trigger:* SIGKILL, OOM kill, hardware fault, or container eviction between the completion of `_dispatch` and the commit of `self.db.transaction()`.

*Failure path:* Worker A completes `_dispatch` — the job's external side effect has occurred (alert sent, payment charged, external API called). Worker A is killed before the database transaction commits. The database has no record of the execution. Worker B (or Worker A on restart) claims the job, checks `db.get_execution()`, finds no record, and executes the job again. The external side effect occurs twice. This is not a locking failure — the lock worked correctly. It is a gap between the external side effect and the durability record. The idempotency check (`get_execution`) is only effective if the execution record was committed before the side effect occurred, or if the side effect itself is idempotent.

*Required protection:* The execution record must be written to the database before the external side effect is triggered, not after. The current order is: acquire lock → check existing → dispatch (side effect) → record execution. The correct order for non-idempotent jobs is: acquire lock → check existing → record execution intent → dispatch → record completion. The intent record allows a restarted worker to detect that execution was in progress and either skip or resume rather than re-execute. This is an ordering requirement that cannot be enforced by the locking mechanism — it is an application design requirement that must be specified in the job handler contract. The deployment runbook must document which job types are idempotent and which require the intent-record pattern.

---

**8. `db.get_execution()` read is not linearizable with lock acquisition**

*Trigger:* Database read replica lag, or a database that does not provide read-your-writes consistency on the execution check.

*Failure path:* Worker A completes a job and commits the execution record to the primary. The lock expires. Worker B acquires the lock and calls `db.get_execution()` — if this read is served from a replica that has not yet received the commit, it returns None. Worker B proceeds to execute the job again. The lock provided mutual exclusion during execution but the idempotency check is defeated by read replica lag. This is a database topology assumption that is not documented in the code.

*Required protection:* The `get_execution` read must be directed to the primary, or the database must provide read-your-writes consistency for this call. This is a database client configuration requirement. The code makes no distinction between reads and writes in its `db` interface — the deployment must enforce that the execution check is not served from a lagging replica. This must be documented in the operational runbook and enforced in the database client configuration injected at construction time.

---

**9. Daemon thread survives process shutdown — release script not called**

*Trigger:* Process receives SIGTERM during job execution. Python's default SIGTERM handler initiates interpreter shutdown. Daemon threads are killed without cleanup. The `finally` block in `_claim_job` may not execute.

*Failure path:* Worker A is executing. Deployment rolls out a new version. Orchestrator sends SIGTERM. Python begins shutdown. The `finally` block in `_claim_job` — which calls `_release_script` — may not execute if the main thread is interrupted before reaching it, or if the interpreter shuts down before the finally block completes. The lock remains in Redis until CLAIM_TTL expires. Other workers are blocked from claiming the job for up to 60 seconds. If the job was partially executed and the lock is not released, the job may be retried after TTL expiry with the same duplicate-execution risk.

*Required protection:* SIGTERM handling must be implemented explicitly. The process must register a signal handler that sets a shutdown flag, allows the current job to complete or abort cleanly, and ensures the finally block executes before the process exits. This is a deployment lifecycle requirement — Kubernetes and similar orchestrators send SIGTERM before SIGKILL with a configurable grace period. The grace period must exceed the maximum job execution time, or the job must be checkpointed. Neither is addressed in this code. The deployment configuration must specify terminationGracePeriodSeconds in relation to CLAIM_TTL and expected job duration.

**Open gaps**

1. **No database-layer fencing predicate.** The commit is unconditional on lock validity. This is the central gap from Layer 1 that infrastructure failure modes 1, 2, 3, 4, and 5 all converge on. No infrastructure protection closes this gap — it must be fixed in the application.

2. **No Redis client timeout configuration.** socket_timeout and socket_connect_timeout are not set. The renewal thread can block indefinitely on a network call, preventing stop_event from being checked and allowing CLAIM_TTL to expire undetected.

3. **No exception handling in the renewal thread.** Redis errors kill the thread silently. The main thread has no signal.

4. **No documented Redis topology guarantee.** The code assumes lock durability across failover. This assumption is false for Sentinel without synchronous replication and for Cluster during resharding. The deployment must document which topology is in use and what durability guarantee it provides.

5. **Execution record written after side effect.** For non-idempotent jobs, a kill between `_dispatch` and commit produces a re-executable job with no record of the prior side effect.

6. **`get_execution` read consistency not enforced.** Read replica lag can defeat the idempotency check. The database client configuration must direct this read to the primary.

7. **No SIGTERM handler.** Graceful shutdown is not implemented. The finally block is not guaranteed to execute on process termination. Lock release and clean shutdown are not coordinated with the deployment lifecycle.

8. **CLAIM_TTL not governed against deployment constraints.** Container CPU limits, expected pause durations, and TTL value are not documented or validated against each other. The TTL is a configuration value with correctness implications that is currently set without reference to the environment it runs in.

---

**Production readiness verdict: NOT APPROVED**

This system provides meaningful protection against the common case — two workers racing to claim the same job under normal operating conditions. The Lua scripts are correct, the token verification is sound, the backoff is reasonable, and the database transaction wraps the write correctly. Under normal conditions, with a healthy Redis instance, no pauses, and no topology events, this works.

It does not protect against the conditions that actually cause incidents. The heartbeat mechanism detects lock loss only in the result == 0 path and only if the Redis call succeeds — it does not handle exceptions, does not signal the main thread, and does not prevent the commit regardless. The database transaction has no predicate that makes it conditional on lock validity at commit time; any infrastructure event that invalidates the lock between acquisition and commit — clock skew, partition, failover, process pause exceeding TTL — produces a duplicate execution with no detection and no prevention.
