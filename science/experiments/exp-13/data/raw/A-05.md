Decision: Request Changes

Summary: The architecture demonstrates solid foundational thinking — atomic lock acquisition via `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there are several non-trivial correctness and security flaws that must be resolved before this can be approved, the most critical being a charge-before-commit race that directly undermines the at-most-once guarantee the PR claims to provide. The issues below are ordered by severity.

---

## Critical

### 1. Payment charge occurs inside a database transaction — external side effect is not rollback-safe

This is the central architectural flaw and it directly contradicts the comment in the code that reads *"crash between charge and record cannot produce an unrecorded charge."*

```python
# services/order_processor.py, lines inside process()
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The database transaction provides atomicity over database operations only. `self.payment.charge()` is an external HTTP call to a payment gateway. If the charge succeeds but the subsequent `db.record_charge()` or `db.mark_order_complete()` raises an exception — or the process crashes — the database transaction rolls back, leaving no record of the charge. The next worker that picks up this order will pass the idempotency check (`get_charge` returns `None`) and charge the customer again. This is a double-charge vulnerability.

**Required fix:** The charge must be recorded before it is committed as "complete," and the design must account for the possibility that the charge succeeded but the record write failed. The standard pattern is:

1. Write a "charge pending" record with the `order_id` and a generated idempotency key *before* calling the gateway.
2. Call the payment gateway using that idempotency key.
3. Update the record to "charge complete" with the returned `charge_id`.
4. Mark the order complete.

Steps 3 and 4 can be wrapped in a transaction. Step 1 must be committed independently so it survives a crash. On retry, the presence of the pending record prevents a second charge attempt and allows reconciliation.

### 2. Timeout during payment charge produces an ambiguous outcome that is not handled

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

On a timeout, the gateway may or may not have processed the charge. The client retries unconditionally. Without an idempotency key passed to the gateway on each attempt, this retry can produce a second charge for the same request. The PR checklist claims retry safety but the implementation does not pass any idempotency key to the gateway endpoint.

**Required fix:** Generate a stable idempotency key (e.g., derived from `order_id`) before the retry loop and pass it in every request, typically as a header (`Idempotency-Key`). This allows the gateway to deduplicate retries server-side.

```python
# Sketch only — adapt to actual gateway API
headers = {"Idempotency-Key": f"order-{order_id}"}
response = requests.post(..., headers=headers, ...)
```

## High

### 3. Lock TTL of 60 seconds is dangerously short given the retry configuration

```python
ORDER_LOCK_TTL = 60   # services/order_processor.py
MAX_RETRIES = 3       # services/payment_client.py
REQUEST_TIMEOUT = 30  # services/payment_client.py
RETRY_BACKOFF = 5     # services/payment_client.py
```

Worst-case payment client execution time: 3 attempts × 30s timeout + 2 gaps × 5s backoff = **100 seconds**. The lock TTL is 60 seconds. A second worker can acquire the lock while the first worker is still inside `process()`, executing a payment charge. The idempotency check will pass for the second worker because the first worker has not yet committed. This is another double-charge path.

**Required fix:** Either raise `ORDER_LOCK_TTL` to comfortably exceed worst-case payment execution time (≥ 120s with margin), or implement lock extension (Redlock-style watchdog). At minimum, document the constraint explicitly and enforce it with a startup assertion.

---

### 4. `payment_token` is logged indirectly and may appear in exception traces

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value may be written to logs. Payment tokens are sensitive credentials.

**Required fix:** Do not log the raw `ValidationError` object. Log a sanitized message or enumerate only the failing field names without values.

```python
logger.warning(
    "Invalid order request order_id=%s fields=%s",
    order_id,
    [str(err["loc"]) for err in e.errors()],
)
```

## Medium

### 5. Unhandled exceptions inside the lock context silently release the lock but do not surface the error

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    ...
    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

If `self.db.transaction()` raises an unexpected exception (database connection loss, constraint violation, etc.), the lock is released correctly, but the exception propagates out of `process()` with no catch clause. The caller receives an unhandled exception rather than an `OrderResult`. This is inconsistent with the contract of `process()`, which returns `OrderResult` for all handled failure modes.

**Required fix:** Add a broad exception handler in `process()` that catches unexpected errors, logs them with full traceback, and returns a structured `OrderResult(processed=False, message="Internal processing error")`. Do not swallow the exception without logging.

---

### 6. Lock retry backoff is linear and blocks the worker thread

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

`time.sleep` in a synchronous worker blocks the thread entirely. In a multi-threaded or async deployment this degrades throughput. Additionally, deterministic backoff without jitter causes thundering herd behavior when multiple workers contend on the same order simultaneously — they will all retry at the same intervals.

**Required fix:** Add jitter to the backoff calculation.

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.3)
```

### 7. `get_charge` idempotency check is not inside the same transaction as `record_charge`

```python
existing = self.db.get_charge(req.order_id)   # read outside transaction
if existing:
    return ...

with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

The read and the write are not atomic with respect to the database. Under the distributed lock this is largely safe, but if the lock is lost (TTL expiry per issue #3) and two workers proceed past the idempotency check before either commits, both will attempt to charge. This is a defense-in-depth gap.

**Required fix:** Move `get_charge` inside the transaction and use a `SELECT FOR UPDATE` or equivalent row-level lock on the order record as a secondary guard independent of Redis.

## Low / Informational

### 8. Test for lock release verifies the script was called but not that the correct token was passed

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()   # ← no argument assertion
```

This test would pass even if the token argument were empty or incorrect. The token-verification property is the entire security guarantee of the Lua script.

**Required fix:**

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["keys"] == ["order:lock:order-001"]
assert len(call_args.kwargs["args"][0]) == 36   # UUID format
```

---

### 9. No test covers the payment gateway timeout + retry path

The `PaymentClient` retry logic is entirely untested. Given that this is the path most likely to produce duplicate charges (see issues #2 and #3), it warrants explicit coverage with a mock that raises `Timeout` on the first attempt and succeeds on the second.

---

### 10. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoints are a deployment portability and secret-rotation concern. This should be injected via configuration or environment variable.
