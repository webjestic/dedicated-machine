Decision: Request Changes

Summary: The core locking and idempotency logic has a critical architectural flaw that directly undermines the duplicate-charge guarantee this PR is designed to provide. There are also secondary reliability and correctness issues in the payment client and test suite that need to be addressed before this is production-ready.

---

## Critical Issue: Payment Call Outside the Database Transaction

This is the most serious problem in the PR. The comment in the code claims:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
with self.db.transaction():
    charge_id = self.payment.charge(...)       # ← external network call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This claim is **false**. A database transaction provides atomicity over *database operations only*. The `payment.charge()` call is an external HTTP request to a third-party gateway. If the process crashes, the container is killed, or the network drops *after* the charge succeeds but *before* `db.record_charge()` commits, you have:

- A real charge on the customer's card
- No record of it in your database
- The idempotency check will see no existing charge on retry
- The customer gets charged again

The correct pattern is to move the payment call **outside and before** the transaction, then use the transaction solely to record the result with a guard against double-recording:

```python
# 1. Charge first, outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record atomically, idempotently
with self.db.transaction():
    # Re-check inside the transaction to close the TOCTOU window
    if self.db.get_charge(req.order_id):
        # Charge already recorded by a concurrent worker — do not double-record
        # Consider issuing a refund for the charge we just made
        ...
        return OrderResult(processed=False, ...)
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

You also need a reconciliation/recovery path for the case where the charge succeeds but the DB write fails even after this fix — that is a business process requirement, not just a code fix.

## Significant Issue: TOCTOU Race on the Idempotency Check

The idempotency check happens *before* the database transaction opens:

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # check outside transaction
    if existing:
        ...
    with self.db.transaction():                    # transaction opens here
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

The Redis lock reduces the likelihood of a race, but it does not eliminate it. Redis locks are advisory and can be lost (TTL expiry under a slow worker, Redis failover in non-RedLock configurations). The idempotency check must be performed *inside* the database transaction with appropriate row-level locking (e.g., `SELECT ... FOR UPDATE` on the order row) to be a hard guarantee. The check outside the transaction is fine as an early-exit optimization, but it cannot be the sole guard.

## Issue: Timeout in Payment Client Is Not Retried Correctly

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means you do not know whether the charge succeeded or not. Blindly retrying on timeout can itself cause a duplicate charge if the gateway processed the request but the response was lost in transit. At minimum, the payment client needs to send a stable idempotency key (e.g., the `order_id`) with each request so the gateway can deduplicate retries:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be passed in
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, the retry logic in `PaymentClient` is potentially more dangerous than no retry at all.

## Issue: `PaymentError` Is Silently Swallowed

In `order_processor.py`, only `LockError` is caught:

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

If `self.payment.charge()` raises `PaymentError`, or if `self.db.record_charge()` raises any exception, the exception propagates uncaught to the caller. This may be intentional, but it is inconsistent with the method's documented contract of returning an `OrderResult`. The caller has no way to distinguish a payment failure from a programming error. At minimum, `PaymentError` should be caught and mapped to a structured result, and unexpected exceptions should be logged before re-raising.

## Minor Issue: Lock TTL vs. Payment Retry Duration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient.MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds and `REQUEST_TIMEOUT = 30` seconds per attempt. In the worst case the payment client runs for `3 * (30 + 5) = 105` seconds, which exceeds the lock TTL by 45 seconds. A second worker can acquire the lock while the first is still in its retry loop, defeating the concurrency protection entirely. Either increase the TTL to safely exceed the maximum payment client runtime, or pass a deadline into the payment client so it aborts before the lock expires.

---

## Test Coverage Gaps

The tests do not cover the scenarios that matter most for this service:

1. **No test for payment success followed by DB write failure.** This is the exact crash scenario the transaction is supposed to protect against, and as noted above, it does not actually protect against it.
2. **No test for lock TTL expiry mid-processing.** The lock can expire while the payment is in flight.
3. **No test for `PaymentError` propagation.** The caller behavior on payment failure is untested.
4. **`test_lock_release_uses_lua_script` is too shallow.** It only asserts the script was called once. It does not verify the correct key and token were passed, which is the entire point of the token-verified release.

---

The distributed lock mechanics (`SET NX EX`, Lua-script release, token verification) are implemented correctly and the Pydantic validation is solid. The structural intent is right. But the critical flaw around the payment call placement means this cannot go to production as written — it does not actually provide the at-most-once guarantee it advertises.
