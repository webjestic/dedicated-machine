Decision: Request Changes

Summary: The core locking and idempotency logic contains a critical atomicity flaw that directly undermines the duplicate-charge guarantee this PR is designed to provide. Several additional reliability and security issues compound that concern. The code is well-structured and the intent is sound, but these problems must be resolved before this touches production.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — the central guarantee is broken

This is the most serious bug in the PR. The docstring and checklist both claim the charge and record happen within a single atomic database transaction, but the code does not deliver that.

```python
# What the code actually does:
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← External HTTP call INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is an external HTTP request to a payment gateway. Wrapping it in a `db.transaction()` context manager does **not** make it atomic with the database writes. The database transaction only governs the DB operations. The payment gateway has no knowledge of your database transaction state.

The failure scenario this creates:

1. `payment.charge()` succeeds — money is taken from the customer
2. `db.record_charge()` raises an exception
3. The database transaction rolls back
4. `db.get_charge()` now returns `None` for this order
5. The next worker acquires the lock and charges the customer again

The correct pattern is to perform the external charge **before** opening the database transaction, then record the result durably:

```python
# Charge outside the transaction first
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then record atomically in the DB — if this fails, charge_id is still known
# and can be reconciled. The idempotency check should also store pending state.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

Even better: write a `PENDING` record to the database *before* calling the payment gateway, so a crash mid-flight is detectable and recoverable rather than silent.

### 2. Idempotency check has a race window — TOCTOU between check and charge

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return ...  # already processed

with self.db.transaction():
    charge_id = self.payment.charge(...)
    self.db.record_charge(...)
```

The Redis lock is meant to serialize this, but the lock TTL is 60 seconds and the payment client can take up to `30s × 3 retries + 10s backoff = ~100 seconds` in the worst case. The lock can expire while a payment is in flight, a second worker acquires the lock, passes the idempotency check (charge not yet recorded), and issues a duplicate charge.

The lock TTL must exceed the maximum possible processing time, or the idempotency check must be enforced at the database layer with a unique constraint on `order_id` in the charges table — which is the only reliable backstop regardless of lock behavior.

---

### 3. Unhandled exceptions inside the lock context will silently suppress errors

```python
try:
    with self._acquire_lock(req.order_id):
        ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Only `LockError` is caught. Any other exception raised inside the lock — `PaymentError`, a database error, a network error — will propagate up to the caller as an unhandled exception rather than returning a structured `OrderResult`. The caller has no way to distinguish "lock contention" from "payment failed" from "database is down." These should be caught and handled explicitly with appropriate logging and result codes.

## Significant Issues

### 4. `payment_token` is logged at INFO level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is present in the `OrderRequest` object and flows through this code. One misplaced log line — easy to add during debugging — exposes a live payment credential. The `payment_token` field should be marked explicitly as sensitive. Consider a custom Pydantic type or `__repr__`/`__str__` override that masks the value, so it cannot accidentally appear in logs or tracebacks.

```python
class SensitiveStr(str):
    def __repr__(self):
        return "'[REDACTED]'"
```

### 5. Payment client does not validate the response body before accessing `charge_id`

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises an unhandled `KeyError` or `json.JSONDecodeError` that bypasses the retry logic and propagates as an uncaught exception. The response shape should be validated before use.

### 6. Lock TTL is a class-level constant with no configurability

`ORDER_LOCK_TTL = 60` is hardcoded. The payment client can demonstrably exceed this under worst-case retry conditions (see issue 2). This value should be injected or at minimum derived from the payment client's configured timeout and retry parameters so the relationship is explicit and enforced.

---

## Test Coverage Gaps

- No test for the case where `payment.charge()` succeeds but `db.record_charge()` raises — this is exactly the double-charge scenario and it has no coverage.
- No test for `PaymentError` propagation — what does the caller receive?
- No test verifying lock TTL exceeds maximum payment client execution time.
- `test_lock_release_uses_lua_script` asserts the script was called but does not verify the correct `token` was passed, which is the entire point of the token-verified release.

---

## Minor Notes

- `RETRY_BACKOFF = 5` with `MAX_RETRIES = 3` means up to 10 seconds of sleep on top of 90 seconds of timeout. Document this ceiling explicitly.
- `requests.post` should use a `Session` for connection pooling if this runs under any meaningful load.
- The `charge_id` field on `OrderResult` is `Optional[str]` but is only populated on success. Consider a discriminated union or separate result types to make the success/failure structure explicit at the type level.
