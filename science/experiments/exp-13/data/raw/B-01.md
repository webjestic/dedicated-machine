Decision: Request Changes

Summary: The distributed locking strategy contains a critical atomicity flaw — the payment charge occurs *inside* the database transaction but *outside* any durable, rollback-safe boundary that spans both systems, meaning a gateway timeout on retry can produce an unrecorded charge that the idempotency check will never catch. The lock TTL of 60 seconds is dangerously short relative to the combined payment retry window (up to ~75 seconds with `MAX_RETRIES=3` and `RETRY_BACKOFF=5`), creating a window where the lock expires mid-processing, a second worker acquires it, passes the idempotency check (charge not yet recorded), and issues a duplicate charge. Additional structural issues are detailed below.

---

## Critical Issues

### 1. Lock TTL Can Expire Before Payment + DB Write Completes

```python
# order_processor.py
ORDER_LOCK_TTL = 60  # seconds

# payment_client.py
MAX_RETRIES = 3
REQUEST_TIMEOUT = 30  # per attempt
RETRY_BACKOFF = 5     # between retries
```

**Worst-case payment client runtime:**
```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL is 60 seconds. A slow-but-eventually-successful payment call can take longer than the lock lives. When the lock expires mid-flight, a second worker can acquire it, pass the idempotency check (nothing is recorded yet), and issue a second charge. This directly breaks the at-most-once guarantee the PR claims to provide.

**Fix — lock TTL must exceed the maximum possible critical section duration with margin:**

```python
# payment_client.py
MAX_RETRIES = 3
REQUEST_TIMEOUT = 10   # tighten per-attempt timeout
RETRY_BACKOFF = 2

# Worst case: 3×10 + 2×2 = 34 seconds

# order_processor.py
ORDER_LOCK_TTL = 90    # 34s payment worst-case + DB write margin + safety buffer
```

Or implement lock extension (heartbeat renewal) if the critical section duration is genuinely unbounded.

### 2. Payment Charge Occurs Inside a Database Transaction — Cross-System Atomicity Illusion

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external HTTP call inside DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is a well-known anti-pattern. The database transaction provides no rollback capability over the external payment call. The scenarios that break this:

| Scenario | Outcome |
|---|---|
| `payment.charge()` succeeds, DB write fails and rolls back | Charge issued, no record — unrecorded charge, idempotency check will never block a retry |
| `payment.charge()` times out on attempt 3 but the gateway processed it | `PaymentError` raised, DB rolls back, retry re-charges |
| DB connection lost after charge, before commit | Same as above |

**Fix — separate the charge from the DB transaction:**

```python
# 1. Charge first, outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record durably in a separate, tight DB transaction
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # Charge succeeded but record failed — log for manual reconciliation,
    # do NOT retry the charge. Alert on this condition.
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but record failed "
        "order_id=%s charge_id=%s", req.order_id, charge_id
    )
    raise
```

The idempotency check must also be moved to check for an *in-flight* state, not just a completed charge, to handle the gap between charge and record.

### 3. Gateway Timeout Is Ambiguous — Retrying on Timeout Can Double-Charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have been processed by the gateway. Retrying unconditionally on timeout with the same payment token assumes the gateway is idempotent on `token`. This assumption is not stated, not tested, and not enforced. If the gateway is not idempotent, every timeout retry is a potential duplicate charge.

**Fix:**
- Document the idempotency contract with the payment gateway explicitly.
- Use a per-request idempotency key (e.g., `order_id + attempt_number` or a stable UUID per order) passed in the request body or header, and verify the gateway honors it.
- If the gateway cannot guarantee idempotency on timeout, timeouts must *not* be retried automatically — they must be escalated for manual resolution.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # caller must supply this
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 4. `payment_token` Logged Indirectly via `payment_client` — Audit Risk

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine, but the `token` field is passed directly in the JSON body and is visible in any request-level debug logging, network traces, or error message captures. Ensure the logging configuration does not capture request bodies at DEBUG level in production. This is a PCI-DSS concern if `payment_token` is a raw card token rather than a gateway-issued opaque token.

---

### 5. Lock Release Silently Swallows Failure

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Lua script call raises (Redis connection lost, script evicted), the exception is swallowed by the `finally` block's implicit suppression, and the log line still prints "released" — which is false. The lock will then expire naturally via TTL, but the false log is a correctness and observability problem.

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if released:
            logger.info("Order lock released order_id=%s", order_id)
        else:
            logger.warning(
                "Order lock not released (token mismatch or already expired) "
                "order_id=%s", order_id
            )
    except Exception:
        logger.exception(
            "Failed to release order lock order_id=%s — will expire via TTL",
            order_id,
        )
```

---

### 6. Retry Backoff Is Linear and Blocks the Worker Thread

```python
backoff = 0.5 * (attempt + 1)
time.sleep(backoff)
```

`time.sleep` blocks the thread. In an async or thread-pool worker environment this degrades throughput. Additionally, without jitter, all contending workers retry at the same intervals and collide repeatedly (thundering herd).

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.2)
```

For async environments, replace with `asyncio.sleep`.

---

### 7. Test Coverage Gaps

The tests mock the lock mechanism entirely, meaning the most critical failure modes are untested:

- **No test for lock expiry during processing** — the core race condition identified above.
- **No test for payment success followed by DB write failure** — the unrecorded charge scenario.
- **No test for `PaymentError` propagation** — what does `process()` return when the payment client raises? Currently it would be an unhandled exception, not an `OrderResult`.
- **No test for lock release failure** — verifies the `finally` path behaves correctly on Redis error.

The `PaymentError` gap is also a code defect: `process()` has no `except PaymentError` handler, so a payment failure propagates as an uncaught exception rather than returning a structured `OrderResult`.

```python
# Missing in process():
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```
