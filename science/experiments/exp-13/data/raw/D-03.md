Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapper are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs *inside* the database transaction — this does not make it atomic

The PR comment says:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

This claim is false, and the misunderstanding is dangerous.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside a DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations*. It provides zero atomicity guarantee over an external HTTP call to a payment gateway. The scenarios that break this:

- **Charge succeeds, DB write fails or connection drops** → money is taken, no record exists, the idempotency check will never find it, and the next retry will charge again.
- **Charge succeeds, process crashes before `record_charge`** → same outcome.
- **Long-held DB transaction** → the payment gateway call can take up to 30 seconds × 3 retries = 90 seconds. Holding a DB connection/transaction open for 90 seconds under load is a serious resource and deadlock risk.

The correct pattern for at-most-once payment processing is:

```python
# 1. Write a PENDING record first, inside a transaction, before touching the gateway
with self.db.transaction():
    self.db.create_pending_charge(req.order_id, req.customer_id, req.amount_cents)

# 2. Call the gateway OUTSIDE any DB transaction
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Finalize the record in a separate transaction
with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

If the process crashes between steps 2 and 3, a reconciliation job can detect PENDING records older than N minutes and query the gateway for their status. This is standard practice for payment systems.

### 2. `PaymentError` is not caught in `process()` — unhandled exception leaks to caller

```python
# In process(), only LockError is caught:
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

If `self.payment.charge()` raises `PaymentError` (permanent decline, exhausted retries), or if `self.db.*` raises any exception, the exception propagates uncaught out of `process()`. The caller receives a raw exception instead of an `OrderResult`. This breaks the contract of the method, which is documented to always return an `OrderResult`.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

### 3. Timeout on payment retry is dangerously long relative to the Redis lock TTL

```
ORDER_LOCK_TTL = 60 seconds
REQUEST_TIMEOUT = 30 seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5 seconds between retries
```

Worst-case payment duration: `(30s × 3) + (5s × 2) = 100 seconds`

The Redis lock expires after 60 seconds. A second worker can acquire the lock at second 61 while the first worker is still waiting on a payment response. The idempotency check is now racing. Either reduce the payment timeout + retries to fit within the lock TTL with margin, or extend the lock TTL significantly and document the tradeoff.

## Moderate Issues

### 4. `payment_token` is logged indirectly via `charge_id` — verify no token leakage

The `payment_token` field is not directly logged, which is good. However, confirm that `self.payment.charge()` does not log the token internally, and that the `requests` library debug logging is disabled in production. Payment tokens are PCI-DSS sensitive. Consider adding a note or assertion in the client.

### 5. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body, this raises `KeyError` or `json.JSONDecodeError`, which is not caught and will not be retried. Wrap this:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response") from e
```

### 6. Lock retry backoff is linear and very short

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Under high contention, all workers will retry in near-lockstep. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.3)
```

## Minor Issues

### 7. Test for lock release verifies the mock was called but not the arguments

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This does not verify that the correct `keys` and `args` (the token) were passed. A regression that passes the wrong token would not be caught. Assert the call arguments explicitly.

### 8. No test covering `PaymentError` propagation

There is no test for what happens when `payment.charge` raises `PaymentError`. Once issue #2 is fixed and the exception is caught, a test should be added to verify the returned `OrderResult` is correct.

### 9. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This should be injected via configuration or environment variable, not hardcoded in the class body. Hardcoded endpoints make environment promotion (staging → production) error-prone.

---

## What Is Done Well

- Lua script for atomic lock release is correct and the token-verification logic is sound.
- `SET NX EX` in a single command is the right approach — no TOCTOU on lock acquisition.
- Pydantic validation with pattern enforcement on `order_id` is appropriate.
- The idempotency check placement (inside the lock, before charging) is correct in principle.
- Payment client correctly distinguishes retryable (5xx, timeout) from non-retryable (4xx) errors.
- Structured logging is consistent throughout.
