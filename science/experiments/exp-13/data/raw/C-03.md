Decision: Request Changes

Summary: The architecture demonstrates solid fundamentals — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical distributed systems flaw that undermines the at-most-once guarantee the PR claims to provide: `payment.charge()` is called *inside* a database transaction, meaning a successful charge that precedes a database commit failure will result in an unrecorded, unrecoverable charge with no compensating mechanism. Additionally, there are several secondary issues across reliability, security, and test coverage that must be addressed before this handles real payment data.

---

## Critical Flaw — Charge Inside Database Transaction

```python
# services/order_processor.py

with self.db.transaction():
    charge_id = self.payment.charge(       # ← network call inside DB transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is the most serious issue in the PR. The comment in the code claims:

> *"crash between charge and record cannot produce an unrecorded charge"*

**This claim is false.** A database transaction does not wrap the payment gateway call. If `self.payment.charge()` succeeds but the subsequent `record_charge()` or `mark_order_complete()` raises — or if the DB commit itself fails — money has left the customer's account with no record in your system. The distributed lock is released in `finally`, so the idempotency check on retry will find nothing and charge again.

The correct pattern separates the external side effect from the DB write:

```python
# Correct approach: charge first, outside the transaction,
# then record idempotently. The charge_id itself becomes the
# idempotency anchor.

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# If this block fails, the lock is still held (or retried),
# the idempotency check finds nothing, but you can detect
# the orphaned charge via charge_id reconciliation.
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "UNRECORDED CHARGE — manual reconciliation required "
        "order_id=%s charge_id=%s amount_cents=%d",
        req.order_id, charge_id, req.amount_cents,
    )
    raise
```

Even this is not a complete solution — you need a reconciliation job or an outbox pattern for full safety. But it is strictly better than the current code and makes the failure mode visible rather than silent.

## High — Lock TTL vs. Payment Retry Window

```python
ORDER_LOCK_TTL = 60        # seconds
# services/payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
REQUEST_TIMEOUT = 30       # seconds per attempt
```

Maximum payment time: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`.

The lock TTL is 60 seconds. The payment client can outlive the lock. If the lock expires mid-payment, a second worker acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. The distributed lock provides no protection in this scenario.

```python
# Enforce the invariant at the class level
ORDER_LOCK_TTL = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
    + 30  # safety margin
)
```

Or reduce the payment client's maximum wall-clock time to fit within the TTL. Either way, the relationship must be explicit and enforced, not coincidental.

## High — `payment_token` Logged on Validation Failure

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
    return OrderResult(processed=False, message="Invalid order parameters")
```

Pydantic's `ValidationError` string representation includes the field values that failed validation. If `payment_token` is malformed, it will appear in the log output. Payment tokens are sensitive credentials. This will likely violate PCI-DSS requirements depending on token type.

```python
except ValidationError as e:
    # Log field names and error types only — never values
    safe_errors = [
        {"field": err["loc"], "type": err["type"]}
        for err in e.errors()
    ]
    logger.warning("Invalid order request fields=%s", safe_errors)
    return OrderResult(processed=False, message="Invalid order parameters")
```

## Medium — Timeout on `requests.post` Does Not Cover All Failure Modes

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

`requests` `timeout` applies per socket operation, not to the total request duration. A slow server that sends headers immediately but streams the body slowly can hold a connection open indefinitely beyond `REQUEST_TIMEOUT`. Use a tuple timeout:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

Also, `requests.exceptions.ConnectionError` is not caught. A DNS failure or refused connection will propagate as an unhandled exception through the lock context manager, which will correctly release the lock, but the caller will receive an unhandled exception rather than an `OrderResult`. Add it to the retry logic alongside `Timeout`.

## Medium — Lock Retry Backoff Is Deterministic

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

Under concurrent load, all contending workers sleep for identical durations and wake simultaneously, creating a retry thundering herd. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) + random.uniform(0, 0.3)
```

## Medium — `charge_id` Not Validated from Gateway Response

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a response without `charge_id`, this raises a bare `KeyError` that propagates through the transaction and lock context managers. The lock releases correctly, but the exception is unhandled at the `process()` call site. At minimum:

```python
data = response.json()
charge_id = data.get("charge_id")
if not charge_id:
    raise PaymentError(f"Gateway returned no charge_id: {data}")
```

## Low — Tests Do Not Cover the Critical Failure Path

There is no test for the scenario where `record_charge()` raises after a successful `payment.charge()`. This is exactly the failure mode that the critical flaw above creates. A test like the following should be required:

```python
def test_charge_success_but_record_fails_does_not_silently_swallow(
    processor, db, payment
):
    db.record_charge.side_effect = Exception("DB write failed")

    with pytest.raises(Exception):
        processor.process("order-001", "cust-001", 5000, "tok_visa")

    # Charge was issued — this must be detectable
    payment.charge.assert_called_once()
    # Verify the critical log was emitted (use caplog fixture)
```

---

## Low — `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` Are Not Configurable

These are hardcoded class-level constants with no injection path. In a government contracting context where deployment environments vary, these should be injectable via constructor or environment configuration to avoid requiring code changes for operational tuning.
