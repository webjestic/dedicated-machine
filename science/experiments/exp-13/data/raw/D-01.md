Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured and the Lua-based token-verified release is correct, but there is a critical atomicity flaw: the payment charge happens *inside* the database transaction, meaning a successful charge that precedes a DB commit failure will result in a real charge with no recorded transaction — the DB transaction boundary does not span the external HTTP call, so rolling back the DB does not roll back the charge. Additionally, there are several reliability and correctness issues that need to be addressed before this is production-safe.

---

## Critical Issues

### 1. Payment charge inside DB transaction — unrecoverable money loss

This is the most serious bug in the PR. The comment in the code even claims the opposite of what the code does:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge   ← THIS COMMENT IS WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A DB transaction provides atomicity over *database operations only*. If `self.payment.charge()` succeeds but the DB commit fails (network blip, DB crash, constraint violation), you have charged the customer with no record of it. There is no rollback for an external payment gateway call.

**Required fix:** Charge first, outside the transaction, then open the DB transaction only to record the result. Use a "pending charge" state to handle the crash window:

```python
# Step 1: Mark order as in-flight BEFORE charging (crash-safe sentinel)
self.db.set_order_status(req.order_id, "pending_charge")

# Step 2: Charge outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Step 3: Record the result atomically
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

A recovery process should periodically reconcile orders stuck in `pending_charge` by querying the payment gateway for their status.

### 2. Timeout on payment charge creates ambiguous lock state

`PaymentClient.REQUEST_TIMEOUT = 30` seconds per attempt, with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds. Worst-case payment duration is `(30 * 3) + (5 * 2) = 100 seconds`. The Redis lock TTL is `ORDER_LOCK_TTL = 60` seconds.

The lock can expire while a payment attempt is still in flight. A second worker can then acquire the lock, pass the idempotency check (no charge recorded yet), and issue a second charge.

```
Worker A: acquires lock (TTL=60s) → payment attempt 1 times out (30s)
                                  → payment attempt 2 times out (30s)
                                  → lock expires at t=60s
Worker B:                                              acquires lock at t=61s
                                                       → idempotency check passes (no record)
                                                       → issues second charge ← DUPLICATE CHARGE
Worker A:                                  → payment attempt 3 succeeds at t=95s
```

**Required fix:** Either enforce `ORDER_LOCK_TTL > max_payment_duration` with margin, or implement lock extension (Redlock-style keepalive). The simplest safe fix is:

```python
ORDER_LOCK_TTL = 120   # must exceed MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)
```

And document the invariant explicitly so it does not drift:

```python
# Invariant: ORDER_LOCK_TTL must exceed PaymentClient's worst-case duration.
# PaymentClient worst case = MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES-1) * RETRY_BACKOFF
# Currently: 3*30 + 2*5 = 100s. TTL set to 120s for margin.
ORDER_LOCK_TTL = 120
```

## Moderate Issues

### 3. Unhandled exceptions inside the lock context manager leak the lock

If `self.db.get_charge()`, `self.db.transaction()`, or any DB operation raises an unexpected exception, it propagates out of `_acquire_lock`'s `try` block and the `finally` clause releases the lock — that part is fine. However, `process()` has no outer `except` clause for these cases, so the exception bubbles up to the caller uncaught. The lock *is* released, but the caller receives an unhandled exception rather than a structured `OrderResult`. This should return a structured error:

```python
except (LockError, Exception) as e:
    # Separate handling:
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

### 4. `payment_token` logged indirectly via structured fields

`payment_token` is passed through `OrderRequest` and used in `self.payment.charge(token=req.payment_token, ...)`. Confirm that no logging path in `PaymentClient` logs the full request body or token value. The current code does not appear to log it, but the `requests.post` payload contains the raw token. If debug-level request logging is ever enabled (e.g., `http.client` debug mode), the token will appear in logs. Add an explicit note or scrubbing guard.

### 5. `customer_id` has no pattern validation

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` does not. If `customer_id` is used in any DB query construction downstream, this is a latent injection risk. Apply the same pattern constraint or document why it is not needed.

## Minor Issues

### 6. Lock retry backoff sleeps on the final attempt

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    ...
    time.sleep(backoff)  # ← sleeps even on the last attempt before raising LockError
```

On the final failed attempt, the code sleeps before raising `LockError`. This adds unnecessary latency. Move the sleep inside a `if attempt < self.LOCK_RETRY_ATTEMPTS - 1` guard, consistent with how `PaymentClient` handles this correctly.

### 7. Test for lock release verifies mock call but not token correctness

`test_lock_release_uses_lua_script` confirms `_release_script` was called but does not assert that the correct token was passed as `args`. A wrong token would silently fail to release the lock in production. Consider capturing the token and asserting on `args`.

### 8. No test for the payment-failure path

There is no test covering what happens when `self.payment.charge()` raises `PaymentError`. The order should not be marked complete, no charge should be recorded, and the lock should be released. This is the most financially sensitive code path and it has zero test coverage.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct — no TOCTOU on lock acquisition.
- Lua script for token-verified release is the right approach and correctly implemented.
- Pydantic validation with pattern enforcement on `order_id` is solid.
- `PaymentClient` correctly distinguishes transient (5xx, timeout) from permanent (4xx) failures.
- Structured logging with consistent field names throughout.
- The idempotency check pattern is architecturally correct, just undermined by issue #2.
