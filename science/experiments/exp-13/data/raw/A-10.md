# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Charge Occurs Inside the Database Transaction — Atomicity Claim is False

The PR summary and inline comment both assert:

> *"Crash between charge and record cannot produce an unrecorded charge"*

This claim is **architecturally incorrect and dangerous.**

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External network call — NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge(...)` is a call to an **external payment gateway**. It is entirely outside the database's transaction scope. A database transaction provides atomicity over database operations only. It offers **zero rollback capability** over a completed external charge.

The failure scenarios this code silently introduces:

- **Scenario A — Charge succeeds, DB commit fails:** The customer is charged. The database rolls back. `get_charge()` returns `None` on retry. The customer is charged **again**.
- **Scenario B — Charge succeeds, process crashes before commit:** Identical outcome to Scenario A.
- **Scenario C — Timeout ambiguity:** `payment.charge()` times out after `REQUEST_TIMEOUT=30s`. The charge may or may not have been applied at the gateway. The code raises `PaymentError`, the transaction rolls back, and the system has **no record of the ambiguous charge**. Retry logic in `PaymentClient` will attempt the charge again with no coordination.

The correct architecture is to **record a pending charge intent before calling the gateway**, then confirm or void it after. The database record must precede the external call:

```python
with self.db.transaction():
    self.db.record_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

This pattern ensures that on crash-recovery, a pending record exists and can be reconciled against the gateway. Without it, the "at-most-once guarantee" advertised in the docstring is **not implemented**.

### 🔴 Critical: Redis Lock Does Not Protect Against the Failure Window It Claims To

The idempotency check reads:

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return ...
```

This check queries for a **committed** charge record. But under Scenario A and B above, no committed record ever exists. Two workers can therefore both pass this check on retry and both proceed to charge. The lock only prevents **concurrent** duplicate processing — it provides no protection against **sequential** retries following a partial failure. The checklist item *"Idempotency check — verifies no prior charge before processing"* is therefore only partially true and misleading in the context of the failure modes introduced.

### 🟡 Significant: Lock TTL vs. Payment Timeout Math is Wrong

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client wall-clock time: `(30s × 3 attempts) + (5s × 2 gaps)` = **100 seconds**.

The lock TTL is **60 seconds**. The lock will expire while the payment client is still retrying on its second or third attempt. A second worker can then acquire the lock, pass the idempotency check (no committed record exists yet), and issue a second charge. This directly undermines the stated concurrency guarantee.

`ORDER_LOCK_TTL` must be set to at least `(REQUEST_TIMEOUT × MAX_RETRIES) + (RETRY_BACKOFF × (MAX_RETRIES - 1))` plus headroom for DB operations. A value of **120–150 seconds** would be the minimum defensible floor given current `PaymentClient` parameters. Alternatively, lock extension (Redlock-style `PEXPIRE` refresh) should be considered for long-running operations.

### 🟡 Significant: `payment_token` Logged Indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic `ValidationError` string representations include field values in some versions and configurations. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the raw token value may appear in log output. Payment tokens are sensitive credentials. The log statement should either suppress field values entirely or explicitly redact the `payment_token` field before logging.

---

### 🟡 Significant: `payment_token` Present in `OrderRequest` Model — Exposure Surface

`payment_token` is stored on the validated `OrderRequest` Pydantic model. If this model is ever serialized (e.g., `.model_dump()`, `.json()`, audit logging, exception tracebacks), the raw token will be included. The field should be wrapped in a type that masks its `__repr__` and `__str__`:

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

---

### 🟡 Significant: `LockError` is Silently Swallowed on Retry Exhaustion — Caller Cannot Distinguish

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
```

Lock exhaustion after three attempts and a genuine concurrent worker are indistinguishable to the caller. A system under sustained load or a stuck lock will silently drop orders with no observable signal beyond a log line. At minimum, a metric counter or structured log field indicating retry exhaustion vs. first-attempt contention should be emitted. Ideally, the `OrderResult` should carry a status enum rather than a freeform string so callers can programmatically distinguish transient contention from exhaustion.

---

### 🟠 Minor: Lock Retry Backoff is Linear and Unjitered

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

Under thundering-herd conditions (many workers contending on the same order), linear deterministic backoff causes synchronized retry waves. Standard practice is exponential backoff with jitter:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

---

### 🟠 Minor: `LOCK_RETRY_ATTEMPTS = 3` is a Class-Level Constant Shared Across All Orders

There is no mechanism to vary retry behavior per order priority, SLA tier, or caller context. This is acceptable for an initial implementation but should be noted as a hardcoded policy that will need to be configurable before this service handles heterogeneous order types.

## `services/payment_client.py`

### 🟡 Significant: Timeout Retry Does Not Differentiate Ambiguous vs. Definitive Failure

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout on a POST to a payment gateway is **not a safe retry condition** without idempotency key support. The gateway may have received and processed the request before the timeout occurred. Retrying blindly can result in duplicate charges at the gateway level, independent of the Redis lock. The `charge` call must include an idempotency key (typically derived from `order_id`) in the request headers:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, the payment client's retry logic is itself a source of duplicate charges.

---

### 🟠 Minor: `CHARGE_ENDPOINT` Hardcoded as Class Attribute

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded endpoint URLs in class bodies are not injectable, not environment-configurable, and not testable without monkeypatching. This should be passed via constructor or environment variable.

## `tests/test_order_processor.py`

### 🟡 Significant: No Test for Lock Expiry During Payment Processing

The test suite has zero coverage for the scenario where the Redis lock TTL expires mid-payment. This is the most operationally dangerous failure mode given the TTL/timeout math issue identified above, and it is entirely absent from the test matrix.

### 🟡 Significant: No Test for Charge-Succeeds-But-DB-Commit-Fails

```python
# This scenario is untested:
# payment.charge() returns successfully
# db.transaction().__exit__ raises an exception
```

This is the exact scenario that breaks the at-most-once guarantee. Its absence from the test suite means the critical flaw identified above has no regression coverage.

### 🟠 Minor: `test_lock_release_uses_lua_script` Tests Mechanism, Not Behavior

The test asserts `redis_client.delete.assert_not_called()` and that the Lua script was invoked. It does not assert that the script was called with the **correct token** — meaning a bug that passes the wrong token to the release script would not be caught. The test should assert `args=[<expected_token>]`.
