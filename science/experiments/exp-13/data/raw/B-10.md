Decision: Request Changes

Summary: The architecture is directionally sound — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correct in principle. However, there are several correctness flaws that would produce the exact failure modes this service is designed to prevent: a charge-outside-transaction bug that breaks the atomicity guarantee, a timeout-on-retry flaw that creates silent double-charge risk, and missing exception handling that leaves the lock in a held state under payment failure. These must be resolved before merge.

---

## Critical Issues

### 1. Payment charge executes outside the database transaction — atomicity guarantee is broken

This is the most severe flaw. The PR description claims:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But `payment.charge()` is a network call to an external service. Database transactions do not wrap external HTTP calls. If `payment.charge()` succeeds and then `db.record_charge()` raises, the transaction rolls back but the charge has already been submitted to the gateway. You now have an unrecorded charge with no `charge_id` in the database — exactly the failure mode the design claims to prevent.

```python
# Current code — charge happens "inside" the db.transaction() block
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external side effect, NOT transactional
    self.db.record_charge(...)                # ← if this raises, charge is orphaned
    self.db.mark_order_complete(...)
```

**Required fix:** Charge outside the transaction, then open the transaction only for the database writes. If the DB write fails after a successful charge, the idempotency check on retry will catch the orphaned charge only if you also implement a reconciliation path. The correct pattern is:

```python
# 1. Charge the payment method (external, non-transactional)
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record atomically in the database only
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This is still not fully safe without a compensating reconciliation job, but it is honest about what the transaction boundary actually covers.

### 2. Timeout retry in `PaymentClient` creates silent double-charge risk

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # ← falls through to retry
```

A timeout means the request reached the gateway and may have been processed. The gateway may have charged the card and simply failed to return the response in time. Retrying unconditionally on timeout submits a second charge attempt with no idempotency key, which can result in a duplicate charge.

**Required fix:** The payment gateway call must include an idempotency key (typically the `order_id` or a deterministic derivative), and the gateway must support idempotent retries. Without this, timeout retries must not be performed, or must be gated on a gateway-side status check.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # ← required
    timeout=self.REQUEST_TIMEOUT,
)
```

### 3. Unhandled `PaymentError` leaves the Redis lock held until TTL expiry

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # ← raises PaymentError
        ...
# PaymentError propagates out of _acquire_lock's try block
# The finally clause does execute, so the lock IS released — see below
```

On re-reading, the `finally` in `_acquire_lock` does execute on exception, so the lock is released. However, `PaymentError` is not caught in `process()`, which means it propagates to the caller as an unhandled exception rather than returning a structured `OrderResult`. This is inconsistent with how `LockError` and `ValidationError` are handled, and will produce a 500-level error at the API boundary with no structured response.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
```

## Moderate Issues

### 4. `db.get_charge()` executes outside the database transaction — TOCTOU window

The idempotency check reads from the database before the transaction opens:

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← read outside transaction
    if existing:
        return ...
    with self.db.transaction():
        ...
```

The Redis lock serializes workers on this order, so this is not a race between two workers on the same order. However, if the lock is ever lost (e.g., Redis failover, TTL expiry under a slow payment gateway), two workers could pass the idempotency check before either commits. The idempotency check should be inside the transaction with a `SELECT FOR UPDATE` or equivalent to make it safe under lock loss.

---

### 5. Lock TTL of 60 seconds is shorter than the maximum possible processing time

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1)` = `30 * 3 + 5 * 2` = **100 seconds** in the worst case. The lock expires before processing completes, allowing a second worker to acquire the lock on the same order while the first is still in the payment retry loop.

**Required fix:** Either increase `ORDER_LOCK_TTL` to at least 120 seconds with a documented justification, or implement lock extension (Redlock-style `PEXPIRE` refresh) during long-running operations.

---

### 6. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation, its value may appear in log output. Payment tokens are sensitive credentials and must not appear in logs.

**Required fix:** Sanitize or suppress field values from validation error logs, or log only field names and error types.

## Minor Issues

### 7. Test for lock release is implementation-coupled and fragile

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts on the internal `_release_script` attribute, which is a private implementation detail. It will break on any internal refactor even if the behavior is correct. The test should assert on observable behavior (e.g., the lock key is not present after processing) rather than internal call paths.

### 8. No test coverage for `PaymentClient` retry behavior or timeout handling

Given that the retry logic contains the double-charge risk identified in issue #2, there is no test that exercises the timeout retry path, the 5xx retry path, or the idempotency key behavior. These are the highest-risk code paths in the payment client and must have explicit test coverage.

### 9. `CHARGE_ENDPOINT` is hardcoded

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded internal endpoints are not injectable for testing without monkeypatching and cannot be varied across environments. This should be an injected dependency or environment-configured value.
