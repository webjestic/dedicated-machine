Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issue — Payment Call Inside DB Transaction

This is the most serious problem in the PR.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is an external HTTP request that can take up to 30 seconds (per `REQUEST_TIMEOUT`) and can retry up to 3 times with 5-second backoffs — a worst-case of **~75 seconds** of open DB transaction. This creates several compounding problems:

1. **The DB transaction cannot make the charge atomic.** The payment gateway is not a participant in the DB transaction. If the DB commit fails after a successful charge, you have charged the customer with no record. The comment `"crash between charge and record cannot produce an unrecorded charge"` is factually incorrect and dangerously misleading.

2. **Connection pool exhaustion under load.** A long-held DB connection during payment retries will starve other workers. Under concurrent load this becomes a cascading failure.

3. **Lock TTL is 60 seconds; payment worst-case is ~75 seconds.** The Redis lock can expire while the payment is still in-flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge.

**Required fix:** Charge outside the transaction, then open a short-lived DB transaction only for the record writes. Accept that a charge-succeeded / record-failed scenario is possible and handle it explicitly — typically via a reconciliation job or by catching the DB failure and attempting a refund.

```python
# Correct structure:
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)
try:
    with self.db.transaction():
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
except Exception:
    logger.critical("Charge succeeded but record failed. charge_id=%s requires reconciliation.", charge_id)
    raise
```

## High — Lock TTL / Payment Timeout Mismatch

Even with the above fix, the numbers don't add up:

- `ORDER_LOCK_TTL = 60s`
- `REQUEST_TIMEOUT = 30s`, `MAX_RETRIES = 3`, `RETRY_BACKOFF = 5s` → worst case ≈ 75s

The lock can expire before processing completes. The constants live in two separate files with no enforced relationship. At minimum, add a validation assertion in `OrderProcessor.__init__` and document the dependency explicitly. Better: extend the lock TTL to a safe margin above the payment client's worst-case, or implement lock heartbeat renewal.

```python
def __init__(self, redis_client, db, payment_client):
    max_payment_time = (
        payment_client.REQUEST_TIMEOUT * payment_client.MAX_RETRIES
        + payment_client.RETRY_BACKOFF * (payment_client.MAX_RETRIES - 1)
    )
    assert self.ORDER_LOCK_TTL > max_payment_time, (
        f"Lock TTL {self.ORDER_LOCK_TTL}s must exceed max payment time {max_payment_time}s"
    )
```

## Medium — Unhandled Exceptions Leak Lock State

The `_acquire_lock` context manager releases the lock in `finally`, which is correct. But `process()` has no outer exception handler for anything other than `LockError`. If `self.db.get_charge()` raises, or if the DB transaction raises, the exception propagates to the caller with no `OrderResult` returned. The caller receives an unhandled exception rather than a structured failure response.

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
# Missing:
except Exception:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error processing order")
```

---

## Medium — `payment_token` Logged or Exposed in Tracebacks

`payment_token` is stored on the validated `req` object. If any exception is logged with `exc_info=True` or if the `OrderRequest` object is ever serialized in a log line, the raw payment token will appear in logs. The field should be marked as sensitive and excluded from any default serialization.

```python
payment_token: str = Field(..., min_length=1, max_length=256)

class Config:
    # Prevent accidental serialization of sensitive fields
    json_encoders = {}

def __repr__(self):
    return f"OrderRequest(order_id={self.order_id!r}, customer_id={self.customer_id!r}, amount_cents={self.amount_cents!r}, payment_token='[REDACTED]')"
```

## Medium — `payment_client.py` Missing Authentication

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

There is no API key, bearer token, mTLS, or any authentication on the payment gateway request. Even for an internal endpoint this is unacceptable for a payment service. This needs an auth mechanism injected at construction time.

---

## Low — Test Coverage Gaps

The tests cover the happy path and basic contention, but are missing:

1. **No test for payment failure.** What does `process()` return when `payment.charge()` raises `PaymentError`? Currently it would be an unhandled exception (see medium issue above).
2. **No test for lock expiry during processing.** The TTL/timeout mismatch is untested.
3. **No test for DB failure after successful charge.** This is the most dangerous failure mode and has zero coverage.
4. **`test_lock_release_uses_lua_script` is fragile.** It asserts `redis_client.delete` was not called, but this would pass even if the Lua script was never registered correctly. Assert on the script call arguments instead.

---

## Minor — Hardcoded Internal Endpoint

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

Hardcoded URLs in source are a deployment and security concern. This should be injected via configuration or environment variable.
