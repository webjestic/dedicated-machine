Decision: Request Changes

Summary: The architecture is sound and the security fundamentals are well-constructed — atomic lock acquisition, token-verified Lua release, Pydantic input validation, and idempotency checks are all correctly implemented. However, there is a critical distributed systems flaw that directly undermines the at-most-once payment guarantee the PR claims to provide, plus several secondary issues that must be addressed before this handles real payment data.

---

## Critical Issue

### Payment call is outside the database transaction — the at-most-once guarantee is broken

The PR summary states: *"charges the payment method and records the transaction within a single atomic database transaction."* The checklist confirms this. **This is factually incorrect and the most dangerous bug in the submission.**

```python
# services/order_processor.py  (lines inside _acquire_lock)

with self.db.transaction():
    charge_id = self.payment.charge(          # ← external HTTP call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This pattern has two compounding problems:

**Problem 1 — The payment call is not atomic with the DB write.** A database transaction provides atomicity over database operations only. The `payment.charge()` call is an external HTTP request to a third-party gateway. If the charge succeeds but the subsequent `db.record_charge()` raises (deadlock, connection loss, constraint violation), the transaction rolls back. The charge has already been captured by the payment processor. The order now has no recorded charge, the idempotency check will return `None` on the next attempt, and the customer will be charged again. This is a double-charge scenario.

**Problem 2 — Holding a DB transaction open across a network call is an anti-pattern.** `PaymentClient.REQUEST_TIMEOUT` is 30 seconds with up to 3 retries and 5-second backoffs — worst case ~105 seconds. Holding a database connection and transaction open for that duration under concurrent load will exhaust the connection pool and cause cascading failures across the service.

**The correct pattern** is the two-phase approach:

```python
# Phase 1: reserve the slot atomically in the DB before touching the payment gateway
with self.db.transaction():
    existing = self.db.get_charge(req.order_id)      # re-check inside transaction
    if existing:
        return already_processed_result
    self.db.reserve_order(req.order_id)              # write a PENDING record

# Phase 2: call the gateway outside any DB transaction
try:
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
except PaymentError:
    with self.db.transaction():
        self.db.mark_order_failed(req.order_id)
    raise

# Phase 3: finalize
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

The PENDING record in Phase 1 serves as the idempotency gate. Any concurrent worker that acquires the lock after a crash between Phase 1 and Phase 3 sees the PENDING state and can either wait or surface it for a reconciliation job rather than re-charging.

## High-Severity Issues

### Timeout on a successful charge is retried — potential double-charge

```python
# services/payment_client.py

except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry
```

A `Timeout` exception means the client did not receive a response within 30 seconds. It does **not** mean the charge failed. The gateway may have processed the charge and the response was lost in transit. Retrying unconditionally on timeout will submit a second charge request for the same payment token. This is a real-money double-charge risk.

The fix requires the payment gateway to support idempotency keys. Pass a stable, per-order idempotency key on every attempt:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},   # derived from order_id, not uuid
    timeout=self.REQUEST_TIMEOUT,
)
```

If the gateway does not support idempotency keys, timeouts must **not** be retried automatically — they must be routed to a reconciliation process.

### Idempotency check is outside the database transaction

```python
existing = self.db.get_charge(req.order_id)    # read
if existing:
    return ...
# ... gap ...
with self.db.transaction():                    # write
    charge_id = self.payment.charge(...)
```

The read and the write are not in the same transaction. Two workers can both read `None`, both proceed past the idempotency check, and both attempt to charge. The Redis lock is the only guard here, and Redis locks are advisory — they can fail silently under network partition, Redis restart, or clock skew. The database must be the authoritative source of truth. The idempotency check must be enforced with a unique constraint on `order_id` in the charges table, and the read must occur inside the same transaction as the write, or be replaced entirely by an `INSERT ... ON CONFLICT DO NOTHING` pattern.

## Medium-Severity Issues

### Lock TTL is shorter than the maximum payment retry window

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can run for up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds`. The lock will expire while the payment is still in flight. A second worker can then acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The TTL must be set to at least `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) + margin`, or the payment client's maximum runtime must be bounded below the lock TTL.

### `payment_token` is logged indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their string representation. If `payment_token` fails validation, its value may appear in log output. Payment tokens are sensitive credentials. Either scrub the error before logging or log only field names and error types, not values.

### `requests.exceptions.ConnectionError` is not handled

`payment_client.py` handles `Timeout` and `HTTPError` but not `ConnectionError` (DNS failure, refused connection, TLS error). An unhandled `ConnectionError` will propagate out of `PaymentClient.charge()`, exit the `db.transaction()` context manager, and surface as an unhandled exception in `process()`. The lock will still be released correctly, but the caller receives no structured `OrderResult`. All `requests` exceptions that represent transient network conditions should be caught explicitly.

## Low-Severity / Test Coverage Gaps

### No test for the timeout-retry double-charge scenario

There is no test that verifies behavior when `payment.charge()` raises `Timeout` on the first attempt and succeeds on the second. This is the highest-risk path in `PaymentClient` and it is entirely uncovered.

### No test for DB transaction rollback after a successful charge

The scenario where `payment.charge()` succeeds but `db.record_charge()` raises is not tested. This is the exact scenario that produces an unrecorded charge and should be a required test case.

### `test_lock_release_uses_lua_script` is testing mock wiring, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that the mock was called, not that the Lua script performs the correct token comparison. An integration test against a real Redis instance (or `fakeredis`) should verify that a worker cannot release a lock it does not hold.
