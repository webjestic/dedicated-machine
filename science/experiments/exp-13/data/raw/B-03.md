Decision: Request Changes

Summary: The architecture is sound in its broad strokes — atomic `SET NX EX`, token-verified Lua release, Pydantic validation, and idempotency check are all correct and well-implemented. However, there are several correctness flaws that individually could cause duplicate charges or silent data loss in production, and a few security and operational gaps that must be addressed before this handles real payment flows.

---

## Critical Issues

### 1. Payment charge executes outside the database transaction — the core guarantee is broken

This is the most serious flaw. The PR summary claims "crash between charge and record cannot produce an unrecorded charge," but the code does not deliver that guarantee.

```python
# Current code — charge happens INSIDE db.transaction()
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction wraps your local DB operations. It has no knowledge of the external payment gateway call. If `self.payment.charge()` succeeds and then the process crashes before `record_charge` commits, the charge is real and unrecorded. The DB transaction rolling back does not reverse the charge.

The correct pattern for at-most-once external side effects is:

```python
# 1. Record intent / pending state in DB first (within a transaction)
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

# 2. Execute the external call
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

# 3. Confirm / finalize in DB
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

A reconciliation job must then handle rows stuck in `pending` state (charge attempted, confirmation never written). This is the standard two-phase approach for payment systems. The current code provides a false sense of atomicity.

### 2. Lock TTL is shorter than the maximum operation time — lock can expire while work is in progress

```python
ORDER_LOCK_TTL = 60        # seconds
# payment client: MAX_RETRIES=3, RETRY_BACKOFF=5, REQUEST_TIMEOUT=30
# worst-case payment time: 3 * (30 + 5) = 105 seconds
```

The payment client can take up to 105 seconds in the worst case (3 attempts × (30s timeout + 5s backoff)). The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still mid-charge, defeating the entire duplicate-prevention mechanism. Either the TTL must exceed the maximum possible operation time with margin, or the lock must be extended (fenced/refreshed) during long operations.

```python
ORDER_LOCK_TTL = 180  # must exceed worst-case operation time
# or implement lock extension via periodic PEXPIRE renewal in a background thread
```

### 3. Timeout exception does not distinguish "charge may have succeeded" from "charge definitely failed"

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout on a POST to a payment gateway means the request may have been received and processed — the response was simply lost in transit. Retrying unconditionally can produce a duplicate charge. The payment client must either:

- Use an idempotency key on every request (pass `order_id` or a stable derived key as `X-Idempotency-Key` or in the request body), so the gateway deduplicates on its side, or
- Not retry on timeout without gateway-side idempotency support, and instead surface the ambiguous state for reconciliation.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents, "idempotency_key": idempotency_key},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 4. `payment_token` is logged indirectly and passed in plain-text structures

```python
class OrderRequest(BaseModel):
    payment_token: str = Field(..., min_length=1, max_length=256)
```

`payment_token` sits in the same model as fields that get logged (`order_id`, `customer_id`). If a future developer adds `logger.info("Processing %s", req)` or if an exception serializes the model, the token leaks into log aggregators. At minimum, override `__repr__`/`__str__` on `OrderRequest` to redact the token, or move it out of the loggable model.

```python
class OrderRequest(BaseModel):
    ...
    def __repr__(self):
        return (f"OrderRequest(order_id={self.order_id!r}, "
                f"customer_id={self.customer_id!r}, "
                f"amount_cents={self.amount_cents!r}, "
                f"payment_token='[REDACTED]')")
```

### 5. Lock contention retry uses blocking `time.sleep` with no jitter — thundering herd

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — deterministic
time.sleep(backoff)
```

Under concurrent load, all workers that miss the lock sleep for identical durations and retry simultaneously. Add jitter:

```python
import random
backoff = 0.5 * (attempt + 1) * (0.5 + random.random())
```

### 6. `PaymentError` is not caught in `process()` — unhandled exception escapes to caller

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError — not caught
```

`PaymentError` propagates out of `process()` as an unhandled exception. The method signature promises it returns `OrderResult`. Either catch it and return a structured failure result, or document the exception contract explicitly and update the return type annotation.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
```

### 7. Redis connection failure is not handled — any Redis error crashes the service

If Redis is unavailable, `self.redis.set(...)` raises `redis.exceptions.ConnectionError`. This is not caught anywhere. For a payment service, the correct behavior on Redis unavailability should be a deliberate policy decision (fail closed vs. fail open), not an uncaught exception.

## Minor Issues

### 8. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    processor._release_script.assert_called_once()   # does not check args
```

This test passes even if the script is called with the wrong token or no arguments. It should assert the correct `keys` and `args` are passed:

```python
processor._release_script.assert_called_once_with(
    keys=["order:lock:order-001"], args=[pytest.any(str)]
)
```

### 9. No test covers the lock-expiry-during-processing scenario

There is no test that verifies behavior when the lock TTL expires before the charge completes. This is the exact failure mode introduced by Issue #2 and should be explicitly tested once the TTL is corrected.

### 10. `charge_id` missing from `OrderResult` on already-processed path

```python
if existing:
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Order already processed",
        # charge_id from `existing` is not returned
    )
```

Callers checking idempotent responses cannot retrieve the original `charge_id` without a separate lookup. Return `existing["charge_id"]` (or equivalent) here.
