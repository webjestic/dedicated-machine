Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic `SET NX EX`, token-verified Lua release, and Pydantic input validation are all correctly implemented. However, there are several distributed-systems correctness flaws, a critical security gap in the payment client, and meaningful gaps in test coverage that must be addressed before this can be approved for a payment-handling service.

---

## Critical Issues

### 1. Payment charge occurs outside the database transaction — at-most-once guarantee is broken

This is the most serious architectural flaw. The PR summary claims "crash between charge and record cannot produce an unrecorded charge," but the code does not deliver that guarantee.

```python
# CURRENT — charge happens inside db.transaction() block, but the
# database transaction cannot roll back an already-submitted external HTTP call.
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect, NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `self.payment.charge()` succeeds and the process crashes before `record_charge` commits, the customer is charged with no record. The database transaction wrapper gives false confidence here — it cannot atomically coordinate with an external payment gateway.

**Required fix:** Adopt a two-phase pattern. Issue the charge first, then record it. On startup/recovery, implement a reconciliation pass that queries the payment gateway for any charge associated with an `order_id` that has no corresponding DB record and back-fills it. Alternatively, use an outbox pattern: write a "charge pending" record first, then charge, then update to "charge complete" — all with idempotency keys passed to the gateway so retries are safe.

```python
# Safer ordering — record intent before charging
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-level dedup
)

with self.db.transaction():
    self.db.record_charge_complete(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

### 2. Payment client retries on timeout without an idempotency key — duplicate charges are possible

```python
# payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... falls through to retry
```

A timeout means the request may have succeeded at the gateway but the response was lost in transit. Retrying without an idempotency key will submit a second charge for the same amount. This is a direct financial correctness bug.

**Required fix:** Accept and forward an `idempotency_key` parameter on every charge call, passed as a header (`Idempotency-Key`) to the gateway. The `order_id` is the natural key here.

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    headers = {"Idempotency-Key": idempotency_key}
    response = requests.post(..., headers=headers, ...)
```

### 3. Redis lock TTL is too short relative to payment retry window

```python
ORDER_LOCK_TTL = 60   # order_processor.py
REQUEST_TIMEOUT = 30  # payment_client.py
MAX_RETRIES = 3
RETRY_BACKOFF = 5
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock while the first is still in its third payment retry, defeating the entire concurrency guarantee.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at least `(MAX_RETRIES * REQUEST_TIMEOUT) + (MAX_RETRIES * RETRY_BACKOFF) + safety_margin` (≥ 120s), or implement lock extension (Redlock-style `PEXPIRE` refresh) during long-running operations. At minimum, enforce a configuration-time assertion:

```python
assert self.ORDER_LOCK_TTL > (
    payment_client.MAX_RETRIES * payment_client.REQUEST_TIMEOUT +
    payment_client.MAX_RETRIES * payment_client.RETRY_BACKOFF
), "Lock TTL must exceed maximum payment operation duration"
```

## Significant Issues

### 4. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` is submitted with an invalid value, it may appear in log output. Payment tokens are sensitive credentials.

**Required fix:** Either sanitize the error before logging, or log only field names without values for sensitive fields.

```python
failed_fields = [err["loc"][0] for err in e.errors()]
logger.warning("Invalid order request: failed fields=%s", failed_fields)
```

---

### 5. No exception handling for Redis failures — an unavailable cache crashes the service

```python
with self._acquire_lock(req.order_id):
```

If Redis is down, `redis.exceptions.ConnectionError` propagates uncaught to the caller. For a payment service, the correct behavior should be an explicit policy decision: fail closed (reject the order, safe) or fail open (process without lock, unsafe). Currently it fails open by accident through an unhandled exception.

**Required fix:** Catch `redis.exceptions.RedisError` explicitly and return a structured failure, or document and enforce the fail-closed policy.

```python
except redis.exceptions.RedisError as e:
    logger.error("Redis unavailable, rejecting order order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Service temporarily unavailable")
```

---

### 6. `payment_token` included in the `OrderRequest` model but `payment_token` validation is too permissive

The `payment_token` field accepts any string of 1–256 characters with no format constraint. Depending on the gateway, tokens have known formats (e.g., `tok_*`, `pm_*`). More importantly, there is no masking of this value anywhere in the codebase. If `OrderRequest` is ever serialized to a log or error report, the token is exposed.

**Required fix:** Add a custom `__repr__`/`__str__` override or a Pydantic field serializer that masks the token value.

## Test Coverage Gaps

### 7. No test for lock TTL expiry during payment — the core race condition is untested

The most dangerous scenario (lock expires mid-payment, second worker enters) has no test coverage. This should be tested with a mock that simulates a slow payment call and verifies the second worker's behavior.

### 8. No test for Redis connection failure

Given issue #5 above, there is no test verifying the service returns a structured error when Redis is unavailable rather than raising an unhandled exception.

### 9. No test for payment timeout retry behavior

`test_order_processor.py` does not test `PaymentClient` at all. The timeout-then-retry path — which is where the duplicate charge risk lives — has no test coverage.

### 10. `test_lock_release_uses_lua_script` does not verify the token passed to the script

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()  # does not verify args
```

This test confirms the script was called but does not verify the correct token was passed. A regression that passes a hardcoded or empty token would not be caught.

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] == expected_token  # verify token round-trips correctly
```

## Minor Notes

- `LOCK_RETRY_ATTEMPTS = 3` with `backoff = 0.5 * (attempt + 1)` means the total wait before giving up is 3 seconds. For a 60-second lock TTL this seems intentionally short — document whether this is a deliberate latency budget or an oversight.
- The `charge_id` field on `OrderResult` is `Optional[str]` but is `None` on the already-processed path. Callers should not need to null-check this; consider a discriminated union or separate result types for processed vs. skipped.
- `requests` is used without a session object, meaning each retry opens a new TCP connection. Use `requests.Session` with a retry adapter for connection reuse and consistent timeout behavior.
