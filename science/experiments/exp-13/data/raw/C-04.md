Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the idempotency check are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues around timeout handling, retry safety, and missing exception coverage that must be addressed before this touches production.

---

## Critical Issues

### 1. Payment charge occurs *inside* the database transaction — this does not make it atomic

The PR comment says:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

This claim is false, and the reasoning is backwards.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside a DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity *within the database*. It does not extend to an external HTTP call to a payment gateway. The scenarios that break the guarantee:

| Failure point | Outcome |
|---|---|
| `payment.charge()` succeeds, process crashes before `record_charge` commits | Money taken, no record — **unrecorded charge** |
| `payment.charge()` returns a charge ID, `record_charge` raises, transaction rolls back | Money taken, DB shows no charge — idempotency check will allow a second charge on retry |
| DB transaction holds an open connection for the full payment round-trip (up to 30 s × 3 retries = 90 s) | Connection pool exhaustion under load |

**Required fix:** Charge *outside and before* the transaction. Only open the transaction once you have a confirmed `charge_id`, then record it. Use the idempotency key on the payment gateway side (most gateways support this) to make the charge itself idempotent.

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway-side dedup
)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 2. Timeout on `payment.charge()` is retried — this is unsafe without gateway-side idempotency

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have been processed by the gateway. Retrying a timed-out charge without an idempotency key on the gateway request can produce duplicate charges. This is a direct financial risk.

**Required fix:** Pass a stable idempotency key (e.g., `order_id`) in the request headers or body on every attempt, and confirm the gateway honors it. If the gateway does not support idempotency keys, timeouts must *not* be retried silently — they must surface as an ambiguous state requiring manual reconciliation.

## Significant Issues

### 3. `PaymentError` and general exceptions are not caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)  # can raise PaymentError
        ...
```

`PaymentError`, database exceptions, and any unexpected `Exception` propagate out of `process()` uncaught. The lock is released correctly via `finally`, but the caller receives a raw exception instead of an `OrderResult`. The public interface contract is broken — callers must handle both return values and exceptions.

**Required fix:**

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 4. Lock TTL of 60 seconds can be violated by the payment client

`ORDER_LOCK_TTL = 60` seconds. `PaymentClient` can take up to `30s × 3 attempts + 5s × 2 backoffs = 100 seconds`. The lock expires before the payment completes, allowing a second worker to acquire the lock and pass the idempotency check while the first worker is still mid-charge.

**Required fix:** Either raise `ORDER_LOCK_TTL` above the maximum possible payment duration (120+ seconds with margin), or implement lock extension (Redlock-style keepalive) during the payment call.

### 5. `payment_token` is logged at INFO level indirectly via charge ID correlation

This is a minor concern now, but `payment_token` flows through `OrderRequest` and is passed directly to `payment.charge()`. Confirm that no logging path serializes the full `req` object. The `ValidationError` log at WARNING does not redact fields — if Pydantic includes field values in its error output (it does by default), a validation error on `payment_token` will log the token value.

**Required fix:** Override `__repr__` or use `model_config = ConfigDict(hide_input_in_errors=True)` (Pydantic v2) on `OrderRequest`.

## Minor Issues

### 6. `CHARGE_ENDPOINT` is a hardcoded string constant

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This should come from configuration/environment, not be hardcoded in source. Rotating the endpoint or pointing at a staging gateway requires a code change.

### 7. Test for lock release correctness is shallow

`test_lock_release_uses_lua_script` only asserts that the script was called once and that `redis.delete` was not called directly. It does not assert that the correct `lock_key` and `token` were passed to the script. A regression that passes the wrong key would not be caught.

### 8. No test covers the `PaymentError` path or the lock-expiry-during-payment scenario

Given that these are the two most dangerous failure modes, they need explicit test coverage.

---

## What Is Done Well

- `SET NX EX` acquisition is correct — single atomic operation, no SETNX/EXPIRE race.
- Lua script for token-verified release is correct and properly registered at init time.
- Pydantic pattern enforcement on `order_id` blocks path traversal and injection attempts.
- Retry backoff structure in `PaymentClient` is reasonable for the non-timeout case.
- Structured logging with consistent field names throughout.
- Test fixture design is clean and the mock boundaries are appropriate.

---

The atomicity flaw (#1) and the unsafe timeout retry (#2) are blocking. Both represent direct financial risk — duplicate charges or unrecorded charges — which is unacceptable in a payment processing path. Address those two, fix the lock TTL ceiling (#4), add exception handling (#3), and this will be in good shape.
