# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Charge Occurs Inside DB Transaction — External Side Effect Cannot Be Rolled Back

This is the most serious architectural flaw in the submission.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External, non-transactional side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims this is "atomic," but that is **false**. A database transaction provides atomicity over *database operations only*. `self.payment.charge(...)` is an external HTTP call to a payment gateway. If the charge succeeds but the subsequent `db.record_charge(...)` or `db.mark_order_complete(...)` raises an exception, the DB transaction rolls back — but **the charge has already been captured**. You now have a real charge with no record of it. The idempotency check on re-entry will find nothing and charge the customer again.

The correct pattern is to perform the charge *outside and before* the transaction, then open the transaction only to record the result:

```python
# Charge first — outside any DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record atomically only after charge succeeds
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

This still has a narrow crash window between charge and record, which is why idempotency keys at the payment gateway level are the correct long-term solution — but at minimum the current structure must be inverted.

### 🔴 Critical: `PaymentError` Is Uncaught — Lock Is Released, Retry Will Double-Charge

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)
    if existing:
        ...

    with self.db.transaction():
        charge_id = self.payment.charge(...)   # Can raise PaymentError
        ...
```

`PaymentError` is not caught anywhere in `process()`. If the payment gateway returns a 5xx after exhausting retries, `PaymentError` propagates up through the lock context manager, the lock is released, and the exception surfaces to the caller. So far, so good. But if the gateway actually *did* charge before returning the error (a common gateway behavior on timeout — the charge may have landed), the next retry by the caller will pass the idempotency check (no record exists) and attempt to charge again.

This is a **duplicate charge risk on ambiguous gateway failures**. At minimum, `PaymentError` should be caught, the order should be flagged in a `PENDING_RECONCILIATION` state, and no retry should be permitted without manual or automated reconciliation. The current code has no such state.

---

### 🟡 Significant: Lock TTL Is Dangerously Coupled to Payment Timeout Math

```python
ORDER_LOCK_TTL = 60        # seconds
# ...
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client wall time: `(30s × 3) + (5s × 2) = 100 seconds`.

The lock TTL is 60 seconds. A slow-but-legitimate payment attempt will **expire the lock before the charge completes**. A second worker can then acquire the lock, pass the idempotency check (no record yet), and issue a second charge — while the first worker is still waiting on the gateway. These two constants are defined in separate files with no enforced relationship, no comment cross-referencing them, and no test covering this scenario. Either the TTL must exceed the maximum possible payment duration with margin, or the payment timeout envelope must be strictly bounded below the TTL. This needs to be explicit and documented as a contract.

---

### 🟡 Significant: Idempotency Check Is Not Re-Entrant Safe Against Its Own Lock Window

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # Read from DB
    if existing:
        return ...
    # ... charge and record
```

The idempotency check reads from the database *after* acquiring the lock. This is correct for the concurrent-worker case. However, if the DB read (`get_charge`) and the charge/record sequence are not within the same serializable transaction or snapshot, a read-committed DB could return stale data depending on isolation level. This is a lower-probability concern but is worth documenting. The isolation level assumption should be explicit.

---

### 🟡 Significant: `_acquire_lock` Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)
time.sleep(backoff)
```

`time.sleep()` in a worker that may be handling concurrent orders blocks the entire thread. In a synchronous deployment this may be acceptable, but in an async or thread-pool context this is a latency hazard. More importantly, with only 3 attempts at 0.5s, 1.0s, and 1.5s, the total wait is 3 seconds — which is very short for a 60-second lock TTL. A worker that crashes mid-processing will hold the lock for up to 60 seconds, and the retry window here will be exhausted almost immediately. The retry budget and TTL are mismatched.

---

### 🟠 Minor: `payment_token` Is Logged Indirectly via Pydantic Validation Error

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field names and the offending values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Payment tokens are sensitive credentials. The log message should either redact field values or use a sanitized error summary.

## `services/payment_client.py`

### 🟡 Significant: Timeout Retry Does Not Distinguish Ambiguous Charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
```

A timeout means the request may or may not have been processed by the gateway. Retrying unconditionally on timeout is a classic duplicate-charge vector. The payment gateway must support idempotency keys, and each retry for the *same logical charge* must reuse the same idempotency key. Without this, every timeout retry is a potential duplicate charge. There is no idempotency key being sent in the current implementation:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},   # No idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

This is a **financial correctness issue**, not just a reliability concern.

---

### 🟠 Minor: No Circuit Breaker or Backpressure on Gateway

Three retries with a fixed 5-second backoff provides no protection against a degraded gateway causing worker pile-up. In a multi-worker deployment, all workers will simultaneously retry against a failing gateway, amplifying load. A circuit breaker pattern should be considered.

## `tests/test_order_processor.py`

### 🔴 Critical: No Test for Lock TTL Expiry During Payment

There is no test simulating the scenario where the payment call takes longer than `ORDER_LOCK_TTL`. This is the most dangerous race condition in the design and it has zero test coverage.

### 🔴 Critical: No Test for PaymentError Propagation

```python
# This scenario is completely untested:
payment.charge.side_effect = PaymentError("gateway 503")
```

There is no test verifying behavior when `payment.charge()` raises. Given that this is the primary failure mode that leads to duplicate charges, its absence is a significant gap.

### 🟡 Significant: Transaction Atomicity Test Is Structural, Not Behavioral

```python
def test_charge_recorded_within_transaction(processor, db):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    db.transaction.assert_called_once()
    db.record_charge.assert_called_once()
```

This test only verifies that `transaction()` was called, not that `record_charge` was called *within* the transaction context. A refactor that moves `record_charge` outside the `with` block would not be caught. The test should verify ordering or use a mock that tracks call sequence relative to context entry/exit.

### 🟠 Minor: Lock Contention Test Does Not Exercise Backoff

```python
def test_process_lock_contention(redis_client, db, payment):
    redis_client.set.return_value = False
```

`set` always returns `False`, so all three attempts fail immediately. The backoff sleep is not mocked, meaning this test actually sleeps for `0.5 + 1.0 + 1.5 = 3` real seconds on every run. `time.sleep` should be patched.

**Decision: Request Changes**

**Summary:** The PR contains multiple critical financial correctness defects that must be resolved before merge. The charge occurring inside the database transaction creates a false atomicity guarantee that can produce unrecorded charges on DB failure. The `PaymentError` propagation path has no reconciliation state, and the lock TTL is shorter than the maximum payment client wall time, creating a concrete duplicate-charge window under slow gateway conditions. The test suite does not cover any of these failure paths. This code must not be deployed to a payment-processing environment in its current state.
