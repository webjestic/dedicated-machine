# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Charge Occurs Inside DB Transaction — External Side Effect Cannot Be Rolled Back

This is the most severe architectural flaw in the submission.

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)       # External, non-transactional side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment claims this is atomic and crash-safe. It is not. `self.payment.charge()` is a call to an external payment gateway. It is **not** enrolled in the database transaction. If `charge()` succeeds but `record_charge()` or `mark_order_complete()` subsequently raises, the DB transaction rolls back — but the charge has already been captured by the payment processor. You now have a real charge with no internal record of it. The idempotency check on re-entry (`db.get_charge`) will return `None`, and the next worker will charge the customer again.

The correct pattern is to perform the external charge **before** opening the DB transaction, capture the `charge_id`, then open the transaction to record it. If the DB write fails after a successful charge, a reconciliation mechanism (e.g., a dead-letter queue or idempotency key passed to the payment gateway) must handle recovery — not a DB rollback that silently discards evidence of the charge.

```python
# Correct ordering
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)
try:
    with self.db.transaction():
        self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
        self.db.mark_order_complete(req.order_id)
except Exception:
    # charge_id must be logged durably or enqueued for reconciliation
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but DB record failed "
        "order_id=%s charge_id=%s", req.order_id, charge_id
    )
    raise
```

### 🔴 Critical: Unhandled Exceptions Inside Lock Context Silently Suppress Errors

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, `DatabaseError`, or any unexpected exception raised inside the lock context is **not caught**. It will propagate out of `process()` as an unhandled exception. The caller receives no `OrderResult` — they receive a raw exception. The PR checklist claims structured error handling, but the only caught exception at the outer level is `LockError`. Any payment failure or DB failure becomes an uncontrolled crash path.

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed by another worker")
# PaymentError? DatabaseError? KeyError from response.json()? All uncaught.
```

---

### 🟡 Significant: Redis Lock Does Not Protect Against Clock Skew or Lock Expiry Mid-Execution

`ORDER_LOCK_TTL = 60` seconds is described as covering "typical payment round-trip." The payment client has `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds and `REQUEST_TIMEOUT = 30` seconds per attempt. Worst-case payment client execution time is `3 × (30 + 5) = 105 seconds`, which **exceeds the lock TTL by 45 seconds**. A second worker can acquire the lock while the first is still mid-charge, defeating the entire mutual exclusion guarantee.

The TTL must be set to a value strictly greater than the maximum possible execution time of the critical section, or a lock extension/watchdog mechanism must be employed.

---

### 🟡 Significant: Idempotency Check Is Not Re-Entrant Safe Under Lock Expiry

The idempotency check reads from the DB before charging:

```python
existing = self.db.get_charge(req.order_id)
if existing:
    return OrderResult(processed=False, ...)
```

If the lock expires mid-execution (as described above), a second worker passes this check while the first is still charging. Both workers will proceed to charge. This is a direct consequence of the TTL issue above, but worth calling out independently because the idempotency check creates a false sense of safety.

---

### 🟡 Significant: `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error representations include field values in some configurations. `payment_token` is a sensitive credential. If a malformed token is submitted, its value may appear in logs. The `payment_token` field should be marked with `repr=False` and the log message should not include the raw exception object without scrubbing.

```python
payment_token: str = Field(..., min_length=1, max_length=256, repr=False)
```

---

### 🟡 Significant: Lock Retry Backoff Is Linear and Blocking

```python
backoff = 0.5 * (attempt + 1)
time.sleep(backoff)
```

`time.sleep()` in a synchronous worker blocks the thread entirely. With `LOCK_RETRY_ATTEMPTS = 3`, a worker can block for up to `0.5 + 1.0 = 1.5` seconds before failing. In a high-throughput async or threaded deployment this is a latency and throughput hazard. If this service is ever ported to `asyncio`, this becomes a correctness issue. Exponential backoff with jitter is the standard recommendation for distributed lock retry.

## `services/payment_client.py`

### 🟡 Significant: Timeout Exceptions Are Retried But Are Potentially Non-Idempotent

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A `Timeout` on a `POST /charges` request does not mean the charge was not processed. The gateway may have received and processed the request before the timeout fired on the client side. Retrying blindly on timeout can result in duplicate charges at the gateway level. The payment client must pass a client-generated idempotency key in the request headers (e.g., `Idempotency-Key: <uuid>`) that is stable across retries for the same logical charge attempt.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # Required
    timeout=self.REQUEST_TIMEOUT,
)
```

---

### 🟡 Significant: `response.json()["charge_id"]` Has No Error Handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 2xx response with a malformed body, missing `charge_id`, or non-JSON content type, this raises `KeyError` or `JSONDecodeError`. These exceptions are not caught and will propagate as unhandled errors. The caller in `order_processor.py` does not catch them either (as noted above).

---

### 🟠 Minor: `requests` Session Not Reused

A new TCP connection is established on every call. For a high-frequency service, a persistent `requests.Session` with connection pooling should be used.

## `tests/test_order_processor.py`

### 🟡 Significant: No Test for the Critical Failure Path — Charge Succeeds, DB Write Fails

The most dangerous failure mode in this entire PR — successful charge followed by failed DB record — has zero test coverage. This is the scenario that produces unrecorded charges and potential double-billing.

---

### 🟡 Significant: No Test for Lock Expiry During Execution

There is no test verifying behavior when the lock TTL expires mid-processing. Given that the TTL can be exceeded by the payment client (as noted above), this is a critical untested path.

---

### 🟠 Minor: `test_lock_release_uses_lua_script` Does Not Verify Token Correctness

```python
processor._release_script.assert_called_once()
```

This only verifies the script was called. It does not verify that the correct token was passed as `args=[token]`. A regression that passes the wrong token or no token would not be caught.

**Decision: Request Changes**

**Summary:** This PR contains two critical correctness flaws that directly contradict its stated guarantees: the payment charge occurs inside a database transaction creating a false atomicity claim that will produce unrecorded charges on DB failure, and the lock TTL of 60 seconds is shorter than the worst-case payment client execution time of 105 seconds, rendering the mutual exclusion guarantee defeatable. Additionally, the payment client retries on timeout without an idempotency key, which can produce duplicate charges at the gateway level independent of the distributed lock. All three issues must be resolved before this code handles any real payment data.
