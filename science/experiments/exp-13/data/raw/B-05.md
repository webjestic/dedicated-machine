Decision: Request Changes

Summary: The core locking and idempotency design is well-structured — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all sound. However, there is a critical distributed systems flaw that directly undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that must be addressed before this handles real payment data.

---

## Critical: Payment Call Lives Outside the Database Transaction

This is the most serious flaw. The comment in the code claims:

```python
# Charge and record atomically — crash between charge and record
# cannot produce an unrecorded charge
```

That claim is false. The `payment.charge()` call is made **inside** `with self.db.transaction()`, but a database transaction provides atomicity over *database operations only*. The payment gateway call is a network side effect that the database cannot roll back.

```python
# Current code — the comment is misleading and the guarantee does not hold
with self.db.transaction():
    charge_id = self.payment.charge(...)   # <-- external network call, NOT transactional
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

**Failure scenario:** `payment.charge()` succeeds and money moves. The process crashes before `record_charge()` commits, or the DB transaction rolls back due to a constraint violation. The charge is now unrecorded. The idempotency check will find nothing on retry, and the customer gets charged again.

**Required fix:** Separate the payment call from the DB transaction. Charge first, then open the DB transaction to record. Use an idempotency key on the payment gateway call itself (most gateways support this). Handle the partial-failure case explicitly:

```python
# Charge outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,  # gateway-level dedup
)

# Record the result atomically in the DB
with self.db.transaction():
    # Re-check: another worker may have recorded while we were charging
    existing = self.db.get_charge(req.order_id)
    if existing:
        # We have a charge_id from the gateway — reconcile, do not abandon it
        return OrderResult(processed=False, order_id=req.order_id,
                           charge_id=charge_id, message="Order already processed")
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

## High: Lock TTL vs. Payment Retry Window Creates a Race

```python
ORDER_LOCK_TTL = 60   # seconds
# PaymentClient:
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
REQUEST_TIMEOUT = 30  # seconds per attempt
```

Worst-case payment duration: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`. The lock TTL is 60 seconds. A second worker can acquire the lock while the first is still mid-charge on attempt 3. The idempotency check will pass for the second worker because no charge has been recorded yet. You now have two concurrent charges in flight.

**Required fix:** Either raise `ORDER_LOCK_TTL` above the maximum possible payment duration (≥ 120s with margin), or implement lock extension (Redlock-style keepalive). Document the relationship explicitly with a constant or assertion:

```python
MAX_PAYMENT_DURATION = (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
)
ORDER_LOCK_TTL = MAX_PAYMENT_DURATION + 30  # 30s safety margin
```

## High: `payment_token` Logged or Exposed via Exception Paths

`payment_token` flows into `OrderRequest` and is passed directly to `payment.charge()`. Verify it is never interpolated into log messages or exception strings anywhere in the call chain. The `PaymentError` raised in `payment_client.py` does not expose it, but the validation error path in `order_processor.py` logs the full Pydantic error object:

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors can echo back field values. If `payment_token` fails validation (e.g., exceeds `max_length=256`), the token value may appear in logs. Add a scrubbing step or log only field names, not values:

```python
logger.warning("Invalid order request fields: %s", [e["loc"] for e in e.errors()])
```

## Medium: Unhandled Exceptions Inside the Lock Are Silently Swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, database exceptions, and any unexpected runtime errors raised inside the lock are not caught. They will propagate up through `_acquire_lock`'s `finally` block (which correctly releases the lock), but they will then surface as unhandled exceptions to the caller. The `process()` method has no outer `except` clause for these cases, so the caller receives a raw exception rather than an `OrderResult`. This breaks the stated contract of the method signature.

Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

## Medium: Idempotency Check Is Not Atomic with Lock Acquisition

The idempotency check (`db.get_charge`) happens *after* lock acquisition, which is correct. However, the check and the subsequent write are not in the same database transaction. A gap exists between the read and the `with self.db.transaction()` block. On a single-primary DB this is low risk given the Redis lock, but if the Redis lock is lost (TTL expiry per the issue above), two workers can pass the idempotency check before either writes. Move `get_charge` inside the DB transaction:

```python
with self.db.transaction():
    existing = self.db.get_charge(req.order_id)  # read inside the transaction
    if existing:
        ...
    self.db.record_charge(...)
```

## Low: Test Coverage Gaps

The tests cover the happy path and basic contention but miss the failure modes that matter most for a payment processor:

- No test for `PaymentError` propagation (currently unhandled — see above)
- No test for DB transaction rollback behavior
- No test for lock TTL expiry mid-payment
- No test that `payment_token` does not appear in log output
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, but does not assert the Lua script was called with the correct token — it only checks call count

---

## Low: `payment_token` Field Name Collision Risk

`OrderRequest.payment_token` and the `payment_token` parameter to `process()` share a name with the internal field passed to `payment.charge(token=req.payment_token)`. This is not a bug today, but rename the gateway parameter to `charge_token` or similar to make the data flow explicit and reduce the chance of a future substitution error.
