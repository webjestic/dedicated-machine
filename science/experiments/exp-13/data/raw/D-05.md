Decision: Request Changes

Summary: The core locking and idempotency logic contains a critical atomicity flaw that directly undermines the duplicate-charge guarantee this PR is designed to provide. There are also several secondary reliability and security concerns that need to be addressed before this is production-ready.

---

## Critical Issue — Charge Inside a DB Transaction Breaks the Atomicity Guarantee

The PR summary claims "crash between charge and record cannot produce an unrecorded charge," but the implementation does not deliver this. The `payment.charge()` call is placed *inside* `self.db.transaction()`:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Database transactions provide atomicity over *database operations only*. The payment gateway call is an external HTTP side effect. If the process crashes after `charge()` succeeds but before `record_charge()` commits, the DB transaction rolls back, the charge record is lost, and the idempotency check on retry will find nothing and charge the customer again. This is the exact failure mode the lock and idempotency check are supposed to prevent.

**Required fix:** Charge outside and before the transaction. Record the result inside it. If the DB write fails after a successful charge, you need a reconciliation path (e.g., a pending/unreconciled charge table written before the gateway call, or an outbox pattern).

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Even this simpler fix is not fully crash-safe, but it is strictly better than the current code and does not make false claims about its guarantees.

## Significant Issue — Redis Lock Does Not Protect Against the Actual Race

The idempotency check and the charge are separated by time inside the lock window, but the real gap is that the Redis lock TTL is 60 seconds and the payment client can take up to `30s × 3 attempts + 10s backoff = ~100 seconds` in the worst case. The lock will expire mid-payment, another worker can acquire it, pass the idempotency check (charge not yet recorded), and issue a second charge. The lock TTL must be greater than the maximum possible payment duration.

```python
ORDER_LOCK_TTL = 60   # can expire before payment completes
REQUEST_TIMEOUT = 30  # 3 attempts × 30s + backoffs = ~100s worst case
```

Either reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` so the total fits inside the TTL with margin, or extend the TTL, or implement lock extension (Redlock-style keepalive). At minimum, document the constraint explicitly and add an assertion or configuration validation that catches a misconfigured pairing.

## Significant Issue — `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error output includes field values in some versions. If `payment_token` fails validation (e.g., too long), the raw token value may appear in logs. Payment tokens are sensitive credentials. Scrub or redact the token field before logging, or log only field names and error types, not values.

---

## Moderate Issue — `payment_token` Passed in Request Body Without Masking in Logs

In `payment_client.py`, the token is passed directly in the JSON body. If `requests` debug logging is enabled at the framework or middleware level, the full token will appear in logs. This is a PCI-adjacent concern for a government contractor. At minimum, add a note in the client that debug-level HTTP logging must be suppressed in production, and consider masking the token in any client-side error messages.

---

## Moderate Issue — `charge_id` Extraction Has No Error Handling

```python
charge_id = response.json()["charge_id"]
```

If the payment gateway returns a 200 with a malformed or unexpected body (missing `charge_id` key, non-JSON content type), this raises an unhandled `KeyError` or `JSONDecodeError`. That exception will propagate out of the DB transaction context manager and out of the lock context manager. The lock will still be released correctly due to `finally`, but the order will be left in an indeterminate state with no structured error returned to the caller. Wrap this in explicit error handling and raise a `PaymentError` with a clear message.

---

## Minor Issues

**Lock retry backoff is blocking.** `time.sleep()` in `_acquire_lock` blocks the worker thread for up to 1.5 seconds. In an async or thread-pool deployment this is a latency problem. If this service runs under async workers, use `asyncio.sleep`. If threaded, this is acceptable but worth noting.

**Test for lock release is fragile.** `test_lock_release_uses_lua_script` asserts `redis_client.delete.assert_not_called()` as a proxy for "Lua script was used." This will pass even if the script call is broken or a no-op. Assert on the script's arguments instead:

```python
processor._release_script.assert_called_once_with(
    keys=[f"order:lock:order-001"], args=[ANY]
)
```

**No test covers the TTL-expiry / payment-duration race described above.** A test that simulates a slow payment gateway (mock `charge` to sleep past the TTL) and verifies the lock is still held would catch this class of bug at the unit level.

**`OrderResult.order_id` is `None` on lock contention.** The caller has no way to correlate the failure back to the order they submitted without re-parsing the message string. Consider always populating `order_id` in the result, including on failure paths.
