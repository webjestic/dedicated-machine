Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical architectural flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issue

### Payment call is outside the database transaction — at-most-once guarantee is broken

The PR description says:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But the code does this:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external network call INSIDE db transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

This is actually worse than it looks, and the comment in the code is misleading. There are two distinct problems here:

**Problem 1 — The payment call is inside the DB transaction, holding a DB connection open during a 30-second (×3 retry) network call.** Under any load this will exhaust your connection pool. The DB transaction should wrap only DB operations.

**Problem 2 — There is no handling of the ambiguous timeout case.** If `payment.charge()` times out, the exception propagates, the DB transaction rolls back, and the lock is released. The next worker (or retry) will pass the idempotency check (no charge recorded) and attempt to charge again. But the first charge may have succeeded at the gateway — the timeout was on the response, not the charge itself. This is the classic dual-write problem and it **will** produce duplicate charges in production.

The correct pattern is:

```python
# 1. Charge outside the DB transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# 2. Record atomically — if this fails, you have an unrecorded charge
#    that must be reconciled via idempotency key lookup at the gateway
try:
    with self.db.transaction():
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
except Exception:
    logger.critical(
        "UNRECORDED CHARGE: charge_id=%s order_id=%s — requires reconciliation",
        charge_id, req.order_id,
    )
    raise
```

Even this is not fully safe without a gateway-side idempotency key. The payment client should be sending a stable idempotency key (e.g., the `order_id`) on every attempt so that retries hit the same charge object at the gateway rather than creating new ones.

## Secondary Issues

### 1. No idempotency key sent to the payment gateway

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

There is no `Idempotency-Key` header or equivalent field in the request body. Without this, every retry to the gateway creates a new charge. This is a payment correctness bug independent of the locking logic.

**Fix:** Pass `order_id` through to `PaymentClient.charge()` and include it as an idempotency key:

```python
headers={"Idempotency-Key": order_id}
```

### 2. Lock TTL is too short relative to the retry window

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can take up to `MAX_RETRIES × (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 × (30 + 5) = 105` seconds. The lock can expire while a worker is still inside `_acquire_lock`'s `try` block, allowing a second worker to acquire the lock and proceed to the idempotency check before the first worker has recorded the charge.

**Fix:** Either raise `ORDER_LOCK_TTL` above the maximum possible payment duration (≥120s with margin), or implement lock extension (Redlock-style keepalive).

### 3. `LockError` is swallowed silently — no logging

```python
except LockError:
    return OrderResult(
        processed=False,
        message="Order currently being processed by another worker",
    )
```

There is no log line here. Lock contention after all retries is operationally significant and should be logged at WARNING or ERROR so it surfaces in alerting.

### 4. `payment_token` is logged indirectly via Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include field values in some configurations. If `payment_token` fails validation (e.g., length), the raw token value could appear in logs. Sanitize or explicitly exclude sensitive fields from the error message before logging.

### 5. Test coverage gaps

- No test for the lock-expiry-during-payment scenario described in issue #2 above.
- No test for `PaymentError` propagation — what does `process()` return if `payment.charge()` raises? Currently it would be an unhandled exception bubbling out of `process()`, which is inconsistent with the structured `OrderResult` return contract. There should be explicit handling and a corresponding test.
- No test verifying the idempotency key is sent to the gateway.

## Minor Notes

- `charge_id` referenced in the `logger.info` call after the `with self.db.transaction()` block is only in scope if the transaction succeeded, which is correct — but if the transaction raises, the exception propagates uncaught from `process()`. Add a top-level `except` for unexpected errors that returns a structured `OrderResult` rather than letting raw exceptions escape the public interface.
- `RELEASE_LOCK_SCRIPT` is a module-level constant, which is fine, but consider moving it inside the class or to a dedicated `lua_scripts.py` to keep the module boundary clean as this grows.
