Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the payment/lock boundary that directly undermines the duplicate-charge guarantee this PR is built around, plus several secondary issues that need to be addressed before this touches production.

---

## Critical Issue

### 1. Payment call lives inside the DB transaction — charge can succeed while the record is lost

```python
# Current code
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side-effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If `self.db.record_charge` or `self.db.mark_order_complete` raises *after* the payment gateway has already returned a `charge_id`, the DB transaction rolls back. The lock is then released. The next worker (or a retry) hits `get_charge` → `None`, and charges again. The comment in the PR ("crash between charge and record cannot produce an unrecorded charge") is **incorrect** — a DB exception, not just a crash, produces exactly that state.

The payment call must happen *outside and before* the DB transaction. The DB transaction should only record an already-confirmed charge:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Payment is now real money in motion. Record it unconditionally.
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This also means the idempotency check in `get_charge` needs to handle a partially-recorded state (charge succeeded, DB write failed). Consider a `pending_charge` column or a separate reconciliation path for that edge case, but at minimum get the ordering right.

---

## High Severity

### 2. Timeout on payment is longer than the lock TTL

`ORDER_LOCK_TTL = 60` seconds, but `REQUEST_TIMEOUT = 30` seconds with `MAX_RETRIES = 3` and `RETRY_BACKOFF = 5` seconds between retries. Worst case payment duration: `3 × 30s + 2 × 5s = 100 seconds`. The lock expires at 60 seconds, a second worker acquires it, passes the idempotency check (charge not yet recorded), and issues a duplicate charge. Either extend the TTL well beyond the worst-case payment duration, or reduce/align the payment client parameters.

### 3. `PaymentError` is not caught in `process()`

```python
# order_processor.py — only LockError is caught
except LockError:
    return OrderResult(...)
```

An unhandled `PaymentError` (or `requests.RequestException`) propagates to the caller as an uncaught exception while the lock is still held (the `finally` block releases it, but the caller gets a raw exception rather than a structured `OrderResult`). Catch `PaymentError` explicitly and return a structured result.

---

## Medium Severity

### 4. `payment_token` is logged at INFO level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `req.payment_token` is held in the validated model and is one errant log statement away from appearing in log aggregators. The `payment_token` field should be marked with a custom `__repr__`/`__str__` that redacts it, or use a `SecretStr` type from Pydantic:

```python
from pydantic import SecretStr

payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

### 5. `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`, but `customer_id` accepts arbitrary Unicode up to 64 characters. If `customer_id` is used in any downstream query construction or log aggregation, this is a gap. Apply the same or a similarly restrictive pattern.

### 6. Lock retry backoff is linear and blocks the worker thread

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
time.sleep(backoff)
```

Three blocking sleeps totaling up to 3 seconds per contended order ties up a worker thread. If this runs in an async or thread-pool context, consider jitter (`random.uniform(0, backoff)`) to avoid thundering herd when many workers contend on the same order simultaneously.

---

## Low Severity / Test Coverage Gaps

### 7. No test for `PaymentError` propagation

There is no test asserting that a payment failure returns a structured `OrderResult(processed=False, ...)` rather than raising. Once issue #3 is fixed, a test should be added.

### 8. No test for the lock-TTL/payment-timeout race (issue #2)

This is an integration-level concern, but at minimum a unit test simulating a slow payment call (mock `time.sleep` or inject a delay) documents the expected behavior.

### 9. `test_lock_release_uses_lua_script` asserts on internal implementation detail

```python
processor._release_script.assert_called_once()
```

Asserting on `_release_script` directly couples the test to the private implementation. Test the observable behavior instead: verify the lock key is absent after processing (using a real or fake Redis, or asserting the script was called with the correct `keys` and `args`).
