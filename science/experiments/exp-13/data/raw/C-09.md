Decision: Request Changes

Summary: The architecture is sound and the security fundamentals are well-considered — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the external payment call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, along with several secondary issues that must be resolved before this handles real financial transactions.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this does not provide atomicity, it inverts the risk

```python
# Current code — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)      # External network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. Wrapping an external HTTP call in a database transaction does not make them atomic. The database transaction only governs the DB operations. What actually happens:

- `payment.charge()` succeeds and money moves
- The DB write fails, or the process crashes, or the DB transaction rolls back for any reason
- The charge is now unrecorded — exactly the failure mode the comment claims is prevented

The correct pattern is **charge outside the transaction, record inside it**, with a compensating reconciliation path:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# charge_id is now a real external fact — record it durably
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    logger.critical(
        "UNRECORDED CHARGE — manual reconciliation required "
        "order_id=%s charge_id=%s amount_cents=%d",
        req.order_id, charge_id, req.amount_cents,
    )
    raise
```

This does not fully solve the problem — nothing can make an external call and a DB write truly atomic — but it correctly identifies which side is the source of truth (the payment gateway) and ensures the DB record reflects reality rather than potentially suppressing a real charge.

### 2. Payment timeout on retry is ambiguous — retrying may double-charge

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may have succeeded — the response was simply not received. Retrying unconditionally on timeout without an idempotency key sent to the gateway can result in two successful charges for the same order. The gateway must receive a stable idempotency key per order so that retried requests are deduplicated on the gateway side:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": order_id},  # stable per order, not per attempt
    timeout=self.REQUEST_TIMEOUT,
)
```

Without this, the retry logic in `PaymentClient` actively creates the double-charge scenario the distributed lock is designed to prevent.

## Significant Issues

### 3. Redis failure is silently treated as lock acquisition failure

If `self.redis.set()` raises a connection error, the exception propagates uncaught through `_acquire_lock` and is not handled in `process()`. This means a Redis outage causes unhandled exceptions rather than a clean `OrderResult`. More importantly, the system should define an explicit policy: fail closed (reject orders when Redis is unavailable) or fail open (process without locking). Currently it fails open by accident because the exception would bubble past the `LockError` handler. Add explicit handling:

```python
except (redis.RedisError, ConnectionError) as e:
    logger.error("Redis unavailable, cannot acquire lock order_id=%s", order_id)
    return OrderResult(processed=False, message="Service temporarily unavailable")
```

### 4. Lock TTL of 60 seconds is too short given the retry configuration

`ORDER_LOCK_TTL = 60` seconds, but `PaymentClient` can run for up to `MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)` = `3 * (30 + 5)` = **105 seconds** in the worst case. The lock expires before the payment completes, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The TTL must exceed the maximum possible payment client execution time with margin:

```python
ORDER_LOCK_TTL = 180  # Must exceed max payment client wall time
```

### 5. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., length violation), the token value may appear in log output. Payment tokens must never appear in logs. Scrub sensitive fields before logging, or log only field names and error types.

## Minor Issues

### 6. `requests` has no connection timeout, only a read timeout

```python
timeout=self.REQUEST_TIMEOUT,  # scalar — this is the read timeout only
```

A scalar timeout in `requests` sets the read timeout but not the connection timeout. A hung DNS resolution or TCP handshake will block indefinitely. Use a tuple:

```python
timeout=(10, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```

### 7. Test for lock release verifies mock call but not token correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This test confirms the script was called but does not assert that the correct token was passed as `args`. A bug where the wrong token is passed would not be caught. Assert on the arguments:

```python
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] == expected_token
```

### 8. No test covers the lock TTL expiry / payment duration mismatch scenario described in issue 4

Given that this is a financial processing service, a test simulating lock expiry mid-payment should be included.

---

## What Is Done Well

The Lua script for atomic lock release is correctly implemented and prevents a worker from releasing a lock it does not hold. The `SET NX EX` acquisition is correct. The `order_id` pattern validation blocks path traversal. The structured retry with backoff in `PaymentClient` correctly distinguishes permanent failures (4xx) from transient ones (5xx). The idempotency check placement inside the lock is correct — checking before acquiring would be a TOCTOU race. The test fixture structure is clean and the coverage of the happy path and contention path is reasonable.
