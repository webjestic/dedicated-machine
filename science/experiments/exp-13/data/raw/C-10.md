Decision: Request Changes

Summary: The core locking and idempotency architecture is well-structured — atomic SET NX EX, token-verified Lua release, and the idempotency check pattern are all sound. However, there is a critical distributed systems flaw that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that must be addressed before this touches a payment path in any environment.

---

## Critical Issues

### 1. Payment call is outside the database transaction — at-most-once guarantee is broken

This is the most serious flaw. The PR summary states *"charges the payment method and records the transaction within a single atomic database transaction"* and the comment in the code repeats this claim. **It is false.**

```python
# What the code actually does:
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← network call to external gateway
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`self.payment.charge(...)` is a network call to an external payment gateway. It is not part of the database transaction. The database transaction provides atomicity only over the two `db.*` calls. The payment call cannot be rolled back if the subsequent DB write fails, and the DB transaction cannot prevent the charge from settling if the process crashes after `charge()` returns but before `record_charge()` commits.

The comment `# crash between charge and record cannot produce an unrecorded charge` is **incorrect** and dangerous — it will mislead every future maintainer.

**The correct pattern for at-most-once payment semantics requires one of:**

- **Idempotency key on the gateway call**: Generate a deterministic idempotency key (e.g., `sha256(order_id + customer_id)`) and pass it to the payment gateway. On retry or crash-recovery, the gateway deduplicates and returns the same `charge_id`. Then record locally. This is the standard approach used by Stripe and others.
- **Two-phase with a pending state**: Write a `PENDING` charge record to the DB *before* calling the gateway (within a transaction), call the gateway, then update the record to `SETTLED`. On startup, a reconciliation job resolves any stuck `PENDING` records by querying the gateway.

Without one of these, a crash window exists between `payment.charge()` returning and `db.record_charge()` committing where money is taken but no record exists.

### 2. Lock TTL is shorter than the maximum operation duration

```python
ORDER_LOCK_TTL = 60        # seconds
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Maximum payment client wall time: `(30s × 3 attempts) + (5s × 2 gaps)` = **100 seconds**.

The lock expires at 60 seconds. A second worker can acquire the lock at second 61 while the first worker is still mid-charge on attempt 3. The idempotency check passes for the second worker because no charge has been recorded yet. Both workers charge the customer.

**Fix**: Either raise `ORDER_LOCK_TTL` to a value safely above the maximum payment client duration (≥ 120s with margin), or implement lock extension (Redlock-style `PEXPIRE` refresh) during long operations. The values across both files must be designed together and documented as a constraint.

## Significant Issues

### 3. Timeout on gateway call does not guarantee the charge did not settle

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the *client* gave up waiting — it does not mean the gateway did not process the charge. Retrying after a timeout without an idempotency key can result in double charges. This is a well-known hazard in payment integrations. The fix is the same as issue #1: use gateway-side idempotency keys.

### 4. `payment_token` is logged at INFO level on success

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is fine, but the `OrderRequest` model holds `payment_token` and the `OrderProcessor.process()` signature accepts it as a plain argument. Verify that no log line anywhere in the call stack serializes the full request object or kwargs. In particular, any framework-level request logging middleware must be confirmed to not echo `payment_token`. This is a PCI-DSS concern. Add a `__repr__`/`__str__` override to `OrderRequest` that masks `payment_token`, or use a dedicated secrets type.

### 5. `requests` is used without certificate verification confirmation

```python
response = requests.post(self.CHARGE_ENDPOINT, json={...}, timeout=self.REQUEST_TIMEOUT)
```

`requests` verifies TLS by default, which is correct. However, `CHARGE_ENDPOINT` is a hardcoded string with no configuration path. In test or staging environments there is a strong temptation to patch this to an HTTP endpoint or pass `verify=False`. Make TLS verification explicit (`verify=True`) and non-overridable from configuration, and externalize the endpoint to environment config rather than a class-level constant.

### 6. Redis connection failures are not handled

If `self.redis.set(...)` raises a `redis.exceptions.ConnectionError` or `redis.exceptions.TimeoutError`, the exception propagates uncaught out of `process()`. The caller receives an unhandled exception rather than an `OrderResult`. Given this is a payment path, all Redis failures should be caught and return a structured `OrderResult(processed=False, message="...")` or re-raise as a typed domain exception that the caller is expected to handle. Currently the contract is inconsistent: lock contention returns an `OrderResult`, but Redis being down raises.

### 7. Lock retry backoff sleeps on the final attempt

```python
for attempt in range(self.LOCK_RETRY_ATTEMPTS):
    acquired = self.redis.set(...)
    if acquired:
        break
    backoff = 0.5 * (attempt + 1)
    time.sleep(backoff)  # ← sleeps even when attempt == LOCK_RETRY_ATTEMPTS - 1
```

The `if attempt < self.MAX_RETRIES - 1` guard present in `PaymentClient` is absent here. On the final failed attempt the worker sleeps 1.5 seconds before raising `LockError`. Minor, but it adds unnecessary latency on the hot path.

## Test Coverage Gaps

### 8. No test for the lock TTL / payment duration race condition

There is no test asserting that `ORDER_LOCK_TTL` is greater than the maximum possible `PaymentClient` wall time. This constraint should be encoded as a test so it cannot be broken by independently changing either constant.

### 9. No test for Redis connection failure behavior

The inconsistent exception vs. `OrderResult` contract noted in issue #6 is untested. A test that patches `redis_client.set` to raise `redis.exceptions.ConnectionError` would immediately expose the gap.

### 10. No test for payment timeout followed by retry

The scenario where `payment.charge` times out on attempt 1 and succeeds on attempt 2 is not covered. More critically, the scenario where it times out on all attempts and then a second call to `process()` with the same `order_id` is made is not tested — this is the double-charge scenario from issue #3.

---

## Minor Notes

- `ORDER_LOCK_TTL` and `PaymentClient` retry parameters live in separate files with no documented relationship. Consider a shared `config.py` or at minimum a comment cross-referencing the constraint.
- `CHARGE_ENDPOINT` should be injected via environment variable or constructor parameter, not a class-level constant, to support environment-specific deployment without code changes.
- `customer_id` has no pattern validation in `OrderRequest` while `order_id` does. If `customer_id` is used in any downstream query or log aggregation, apply equivalent constraints.
