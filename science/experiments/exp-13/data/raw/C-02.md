Decision: Request Changes

Summary: The architecture is sound and the checklist items are largely implemented correctly — the Lua-based token-verified lock release, Pydantic validation, and idempotency check are all solid. However, there are several correctness flaws that collectively undermine the at-most-once guarantee the PR claims to provide, plus a few secondary concerns that must be addressed before this touches a payment path.

---

## Critical Issues

### 1. Payment call is inside the database transaction — this is the core flaw

```python
# Current (broken)
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect inside TX
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — but that is only true if the charge call is **inside** the transaction. The actual problem is the opposite: if the payment gateway call succeeds but the database transaction rolls back (deadlock, constraint violation, network partition to DB), the charge has been captured but is never recorded. You now have a real charge with no internal record of it. This is the exact failure mode the design claims to prevent.

The correct pattern is:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only open the DB transaction after the external call succeeds
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

If the DB write fails after a successful charge, the retry path will hit the idempotency check on the **payment gateway side** (which requires the gateway to support idempotency keys — see issue 2). The charge itself must be the point-of-no-return, not a step inside a rollbackable transaction.

---

### 2. No idempotency key on the payment gateway call

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

The retry loop in `PaymentClient.charge` will re-submit the charge on timeout or 5xx with no idempotency key. A timeout does not mean the charge failed — the gateway may have processed it and the response was lost in transit. Retrying without an idempotency key will double-charge the customer.

Fix: pass a stable idempotency key derived from `order_id` (not generated per-attempt):

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": order_id},
    timeout=self.REQUEST_TIMEOUT,
)
```

This requires `order_id` to be threaded into `PaymentClient.charge`, which is a small but necessary interface change.

---

### 3. Lock TTL is shorter than the maximum possible processing time

```
ORDER_LOCK_TTL = 60        # seconds
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
```

Worst-case payment client execution time: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock, pass the idempotency check (no DB record yet), and issue a second charge while the first worker is still in its retry loop. The lock TTL must exceed the maximum possible critical-section duration with margin. Either raise `ORDER_LOCK_TTL` to at minimum 120–150 seconds, or reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` so the math closes.

---

### 4. `payment_token` is logged at INFO level on success

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This line is fine, but trace through the full call: `payment_token` is passed into `charge()` and the full request body `{"token": token, "amount": amount_cents}` may be emitted by `requests` debug logging or any middleware that logs request bodies. More directly, `payment_token` is a field in `OrderRequest` — ensure that `OrderRequest` (or any exception serialization of it) never reaches a log sink. Add `model_config = ConfigDict(hide_input_in_errors=True)` in Pydantic v2, or override `__repr__`/`__str__` to redact `payment_token`.

## Secondary Issues

### 5. Unhandled exceptions inside the lock are silently swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, database exceptions, and any unexpected exception propagate out of the `with self._acquire_lock` block. The `finally` clause in `_acquire_lock` correctly releases the lock, which is good. However, none of these exceptions are caught in `process()` — they will surface as unhandled 500s to the caller with no `OrderResult` returned, breaking the stated return contract. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

---

### 6. `customer_id` has no format validation

```python
customer_id: str = Field(..., min_length=1, max_length=64)
```

`order_id` has a strict alphanumeric pattern. `customer_id` does not. If `customer_id` is used in any downstream query or log aggregation, this is an inconsistency worth closing. Apply the same or a comparable pattern constraint.

---

### 7. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that `redis.delete` was not called and the script mock was called — but it is asserting implementation details of the mock fixture, not that the Lua script was actually invoked with the correct key and token. Add assertions on the call arguments:

```python
processor._release_script.assert_called_once_with(
    keys=["order:lock:order-001"],
    args=[unittest.mock.ANY],  # token is a uuid, verify it's a string
)
```

Also missing: a test that verifies the lock is released even when an exception is raised inside the critical section (the `finally` path).

---

### 8. No test covering the lock-TTL-shorter-than-processing-time scenario

There is no test or documented assertion that `ORDER_LOCK_TTL > max_payment_client_duration`. This should be enforced with either a unit test or a startup assertion so the constants cannot drift apart silently:

```python
assert OrderProcessor.ORDER_LOCK_TTL > (
    PaymentClient.MAX_RETRIES * PaymentClient.REQUEST_TIMEOUT
    + (PaymentClient.MAX_RETRIES - 1) * PaymentClient.RETRY_BACKOFF
), "Lock TTL must exceed maximum payment client execution time"
```
