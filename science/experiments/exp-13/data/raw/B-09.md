Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB-level idempotency check are all correctly implemented. However, there is a critical atomicity flaw at the intersection of the payment call and the distributed lock boundary: `payment.charge()` is invoked *inside* `db.transaction()`, meaning a gateway timeout on retry can produce an externally-committed charge that the DB never records, and the retry logic in `PaymentClient` will re-fire against an already-charged token with no idempotency key. There are also several secondary issues across the locking model, error handling, and test coverage that need to be addressed before this is safe to merge into a production payment path.

---

## Critical Issues

### 1. Payment charge called inside DB transaction — timeout window creates phantom charges

```python
# services/order_processor.py  (current — WRONG)
with self.db.transaction():
    charge_id = self.payment.charge(...)      # ← external I/O inside DB txn
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — that claim is false. If `payment.charge()` succeeds at the gateway but the response is lost (network partition, pod eviction, `requests.Timeout` on the read side), the DB transaction is rolled back and `record_charge` never commits. The order remains in an unprocessed state, so the next worker will pass the idempotency check and charge again.

The correct pattern is:

```python
# Charge OUTSIDE the transaction — get the charge_id first
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # see issue #2
)

# Then record atomically — if this crashes, the charge_id is known
# and the idempotency check on retry will find it via a pending-charge record
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

You also need a *pending charge* record written inside the lock *before* calling the gateway, so that a crash mid-flight is detectable on re-entry:

```python
with self.db.transaction():
    self.db.record_pending_charge(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents,
                                 idempotency_key=req.order_id)

with self.db.transaction():
    self.db.finalize_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check at the top must also inspect pending records and handle the in-flight case explicitly.

### 2. `PaymentClient.charge()` has no idempotency key — retries will double-charge

```python
# services/payment_client.py  (current — WRONG)
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},   # ← no idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

On a 5xx or timeout, `PaymentClient` retries up to three times. Without an idempotency key passed to the gateway, each retry is a new charge request. The `order_id` must be forwarded as an idempotency key:

```python
def charge(self, token: str, amount_cents: int, idempotency_key: str) -> str:
    for attempt in range(self.MAX_RETRIES):
        response = requests.post(
            self.CHARGE_ENDPOINT,
            json={"token": token, "amount": amount_cents},
            headers={"Idempotency-Key": idempotency_key},
            timeout=self.REQUEST_TIMEOUT,
        )
```

### 3. Lock TTL is shorter than the maximum payment round-trip time

```python
ORDER_LOCK_TTL = 60   # seconds
# vs.
REQUEST_TIMEOUT = 30  # seconds × MAX_RETRIES = 3 × RETRY_BACKOFF = 5s
# worst case: 3 × 30s + 2 × 5s = 100 seconds
```

The lock can expire while the payment client is still retrying. A second worker then acquires the lock, passes the idempotency check (no committed charge yet), and fires a second charge. The lock TTL must exceed the maximum possible payment client wall-clock time:

```python
# Maximum payment wall-clock: MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES-1) * RETRY_BACKOFF
# = 3*30 + 2*5 = 100s  →  set TTL to at least 120s with margin
ORDER_LOCK_TTL = 120
```

Or, better, make `OrderProcessor` aware of `PaymentClient`'s timing constants and assert the invariant at construction time.

## Significant Issues

### 4. `payment_token` is logged indirectly — verify it never reaches log sinks

```python
logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field values in their default string representation. If `payment_token` fails validation, its value (a bearer credential) will appear in the log line. Scrub sensitive fields before logging:

```python
safe_errors = [
    {"field": err["loc"], "msg": err["msg"]}
    for err in e.errors()
]
logger.warning("Invalid order request: %s", safe_errors)
```

Also audit that `payment_token` is never passed to `logger.info/warning` elsewhere.

---

### 5. Unhandled exceptions inside the lock context manager silently release the lock

```python
@contextmanager
def _acquire_lock(self, order_id: str):
    ...
    try:
        yield token
    finally:
        self._release_script(keys=[lock_key], args=[token])
```

If `db.transaction()` raises an unexpected exception (DB connection loss, ORM error), the `finally` block releases the lock and the exception propagates uncaught out of `process()`. The caller gets an unhandled exception with no `OrderResult`, and no log entry explains what order was in flight. At minimum, catch and log at the `process()` level:

```python
except Exception as e:
    logger.exception(
        "Unexpected error processing order order_id=%s", req.order_id
    )
    return OrderResult(processed=False, message="Internal error — order not charged")
```

Decide explicitly whether the lock should be held or released on a DB failure (holding it until TTL expiry may be safer than releasing and allowing a retry that hits a half-written DB state).

---

### 6. `requests.exceptions.ConnectionError` is not caught in `PaymentClient`

```python
except requests.exceptions.Timeout:
    ...
except requests.exceptions.HTTPError as e:
    ...
```

A `ConnectionError` (DNS failure, TCP reset, TLS handshake failure) is not a subclass of either of these. It will propagate uncaught, bypassing the retry loop entirely, and will not be wrapped in `PaymentError`. This means the caller sees a raw `requests` exception rather than the abstracted `PaymentError`, breaking the error contract. Add:

```python
except requests.exceptions.ConnectionError:
    last_error = "connection error"
    logger.warning("Payment connection error attempt=%d", attempt + 1)
```

## Test Coverage Gaps

### 7. No test for lock expiry during payment

There is no test simulating the lock TTL expiring while `payment.charge()` is in progress. This is the exact scenario that causes the double-charge described in issue #3. Add a test that mocks `time.sleep` and verifies behavior when the lock TTL would have elapsed.

### 8. No test for `PaymentError` propagation

There is no test verifying that a `PaymentError` raised by the payment client is handled gracefully and returns a structured `OrderResult` rather than an unhandled exception. This is currently a gap in both the test suite and the production code (see issue #5).

### 9. No test for the pending-charge / in-flight crash recovery path

Once issue #1 is resolved and a pending-charge record is introduced, there must be a test that verifies: a worker re-entering a lock after a crash finds the pending record and does not re-charge.

---

## Minor Issues

### 10. `RETRY_BACKOFF` is a flat constant — should be exponential to avoid thundering herd

```python
RETRY_BACKOFF = 5  # flat
```

Under sustained gateway degradation, all retrying workers will hammer the gateway in synchronized 5-second waves. Use exponential backoff with jitter:

```python
backoff = self.RETRY_BACKOFF * (2 ** attempt) + random.uniform(0, 1)
time.sleep(backoff)
```

### 11. `ORDER_LOCK_TTL` and `PaymentClient` timing constants are defined in separate classes with no enforced relationship

Document the invariant or add a constructor assertion. This is a maintainability hazard — someone increasing `MAX_RETRIES` or `REQUEST_TIMEOUT` will not know they have invalidated the lock TTL.

---

## What Is Done Well

- Atomic `SET NX EX` acquisition is correct — no SETNX/EXPIRE race.
- Token-verified Lua release script is correct and prevents cross-worker lock theft.
- Pydantic pattern enforcement on `order_id` correctly blocks path traversal inputs.
- `customer_id` and `amount_cents` bounds are reasonable.
- The retry-with-backoff structure in `PaymentClient` is the right shape; it just needs the fixes above.
- Test fixture design is clean and the existing cases cover the happy path and basic contention correctly.
