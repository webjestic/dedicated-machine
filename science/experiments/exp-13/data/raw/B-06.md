Decision: Request Changes

Summary: The distributed locking strategy contains a critical TOCTOU flaw that undermines the at-most-once guarantee it claims to provide: the payment charge occurs *inside* a `db.transaction()` block but *outside* any rollback-safe boundary with respect to the external payment gateway, meaning a crash or exception after `payment.charge()` succeeds but before `db.record_charge()` commits will produce an unrecorded charge that the idempotency check will never detect — directly contradicting the inline comment claiming this is handled. Additionally, the lock TTL of 60 seconds is shorter than the cumulative worst-case payment client execution time (up to 3 attempts × 30s timeout + 2 × 5s backoff = 100 seconds), creating a window where the lock expires mid-flight and a second worker can acquire it, re-enter the idempotency check, find no record, and issue a duplicate charge. A third structural concern is that `PaymentError` is not caught in `process()`, meaning gateway failures propagate as unhandled exceptions rather than returning a structured `OrderResult`, which breaks the contract the return type advertises.

---

## Detailed Findings

### Critical — Charge Outside Recoverable Boundary (TOCTOU / Phantom Charge)

```python
# Current code — FLAWED
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side-effect, not rollbackable
    self.db.record_charge(...)                # if this raises, charge is orphaned
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. A `db.transaction()` can only roll back database writes. It cannot undo an already-confirmed external HTTP call to the payment gateway. If `record_charge` raises (deadlock, constraint violation, OOM), the DB transaction rolls back, the Redis lock is released, and the next worker finds no existing charge record and charges again.

The correct pattern is to charge first, capture the `charge_id`, and then make the DB write the *only* thing inside the transaction. The idempotency key must be written before the lock is released, and the charge must be attempted only once per lock acquisition with the result persisted immediately:

```python
# Recommended structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# DB write is now the only thing that can fail; charge already happened
try:
    with self.db.transaction():
        self.db.record_charge(
            req.order_id, req.customer_id, charge_id, req.amount_cents
        )
        self.db.mark_order_complete(req.order_id)
except Exception:
    # charge_id is known — log it prominently for manual reconciliation
    logger.critical(
        "CHARGE_RECORDED_FAILURE order_id=%s charge_id=%s — manual reconciliation required",
        req.order_id, charge_id,
    )
    raise
```

This does not fully solve the phantom charge problem (no distributed system can without two-phase commit or an idempotency key on the payment gateway side), but it eliminates the false safety claim and makes the failure mode explicit and observable.

### Critical — Lock TTL Shorter Than Worst-Case Payment Duration

```python
ORDER_LOCK_TTL = 60        # seconds
# PaymentClient:
REQUEST_TIMEOUT = 30       # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5          # seconds between retries
# Worst case: 3×30 + 2×5 = 100 seconds
```

When the lock expires at second 60 and the payment call is still in flight (second 61–100), a second worker acquires the lock, passes the idempotency check (no record yet), and issues a second charge. This is the exact failure mode the lock is supposed to prevent.

Required fix — either raise the TTL with margin above worst-case or, preferably, implement lock extension (a background heartbeat that refreshes the TTL while the critical section is active). The TTL must be an *upper bound* on the critical section, not just a "typical" estimate:

```python
ORDER_LOCK_TTL = 180  # 100s worst-case + 80s margin; document the derivation
```

Document the derivation explicitly so future changes to `PaymentClient` timeouts trigger a review of this constant.

### High — `PaymentError` Not Caught in `process()`

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
# PaymentError propagates uncaught — caller receives an exception, not an OrderResult
```

The method signature promises `OrderResult`. Callers depending on that contract will get an unhandled exception on gateway failures. Add explicit handling:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, str(e))
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment processing failed",
    )
```

### Medium — `payment_token` Logged Indirectly via `payment_token` Field in `OrderRequest`

`payment_token` is accepted up to 256 characters and stored in a Pydantic model. Confirm that no logging path serializes the full `OrderRequest` object. The current code does not log it directly, but `logger.warning("Invalid order request: %s", e)` on a `ValidationError` may include field values in the Pydantic error output depending on version. Mask or exclude `payment_token` from error serialization:

```python
class OrderRequest(BaseModel):
    model_config = {"hide_input_in_errors": True}  # Pydantic v2
    payment_token: str = Field(..., min_length=1, max_length=256)
```

---

### Medium — Retry Backoff in Lock Acquisition Blocks the Worker Thread

```python
time.sleep(backoff)  # 0.5s, 1.0s — blocks thread in a synchronous service
```

In any async or thread-pool context this is a latency hazard. If this service runs under an async framework (FastAPI, etc.), this must be `await asyncio.sleep(backoff)`. If synchronous, document the threading model assumption explicitly.

---

### Low — Test for Lock Release Verifies Mechanism, Not Correctness

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test confirms `delete` was not called and the script was called once, but does not verify the script was called with the correct key and token. A regression that passes the wrong key would not be caught. Assert the arguments:

```python
processor._release_script.assert_called_once_with(
    keys=["order:lock:order-001"],
    args=[pytest.ANY],  # token is a UUID generated internally
)
```

---

### Low — No Test for Lock Expiry During Payment

There is no test covering the scenario where the lock TTL expires while the payment call is in flight. This is the highest-risk race condition in the design and should have at least a documented test case, even if it requires a time-mock to exercise.
