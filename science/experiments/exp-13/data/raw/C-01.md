Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and Pydantic input validation are all correctly implemented. However, there are several correctness flaws and one significant architectural hazard that must be resolved before this can be approved. The payment-inside-transaction pattern introduces a critical dual-write problem that directly threatens the at-most-once guarantee the PR claims to provide.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this inverts the safety guarantee

```python
# Current code — DANGEROUS
with self.db.transaction():
    charge_id = self.payment.charge(...)      # external side effect
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — but that is wrong. If `payment.charge()` succeeds and then the database transaction rolls back for any reason (deadlock, constraint violation, network partition to DB, OOM), you have charged the customer with no record of it. The charge is an external, non-transactional side effect. Wrapping it in a DB transaction provides zero atomicity guarantee across the two systems.

The correct pattern is to charge first, then record durably, with the idempotency check as your safety net on retry:

```python
# Correct ordering
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Only open the DB transaction after the external call succeeds
with self.db.transaction():
    # Re-check idempotency inside the transaction to close the
    # TOCTOU window between the outer check and the write
    if self.db.get_charge(req.order_id):
        # charge succeeded but was already recorded — void/refund charge_id
        # and return already-processed. Requires compensating logic.
        ...
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This also requires a compensating action (refund or void) if the DB write fails after a successful charge, or an idempotency key passed to the payment gateway so the same charge attempt is deduplicated on retry.

### 2. TOCTOU race on the idempotency check

```python
existing = self.db.get_charge(req.order_id)   # read
if existing:
    return ...
# ... time passes ...
with self.db.transaction():                    # write
    charge_id = self.payment.charge(...)
```

The check and the write are not atomic at the database level. Two workers could both pass the `get_charge` check before either commits. The Redis lock is your primary guard here, but the lock TTL is 60 seconds and payment retries can take up to `3 × (30 + 5) = 105 seconds` — meaning the lock can expire while a payment attempt is still in flight, allowing a second worker to acquire the lock and pass the idempotency check before the first worker's transaction commits.

**Required fix:** Either reduce `ORDER_LOCK_TTL` to be strictly greater than the maximum payment round-trip time (currently it is not), or move the idempotency check inside the DB transaction with a `SELECT FOR UPDATE` / unique constraint as the true enforcement mechanism, with the Redis lock as a fast-path optimization only.

```python
# Lock TTL must exceed worst-case payment duration
MAX_PAYMENT_DURATION = MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF)  # = 105s
ORDER_LOCK_TTL = 120  # must be > 105
```

### 3. `payment_token` is logged indirectly and passed in plain-text JSON

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    ...
)
```

The `payment_token` field flows from `OrderRequest` into `PaymentClient.charge()`. If request/response logging is enabled at the `requests` level (common in debug environments or via middleware), the raw token will appear in logs. Payment tokens are sensitive credentials. At minimum, the token must never appear in any log statement, and the field should be marked as secret in the Pydantic model.

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

## Moderate Issues

### 4. Unhandled exceptions inside the lock context manager will silently swallow errors

```python
try:
    with self._acquire_lock(req.order_id):
        ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed")
```

Only `LockError` is caught. If `self.payment.charge()` raises `PaymentError`, or if `self.db.transaction()` raises any exception, it propagates out of `process()` entirely — no `OrderResult` is returned, the caller gets an unhandled exception, and depending on the framework this may produce a 500 with no structured error. This needs an explicit catch for `PaymentError` and a general fallback.

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

### 5. Lock release failure is silently ignored

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
```

If the Redis call raises (connection dropped, Redis restart), the exception propagates out of the `finally` block and may mask the original exception from inside the `try` block. The lock will eventually expire via TTL, but the caller will see a confusing Redis error instead of the actual business error.

```python
finally:
    try:
        self._release_script(keys=[lock_key], args=[token])
        logger.info("Order lock released order_id=%s", order_id)
    except Exception:
        logger.exception(
            "Failed to release lock for order_id=%s — will expire via TTL in %ds",
            order_id, self.ORDER_LOCK_TTL,
        )
```

## Minor Issues

### 6. Test for lock release does not verify the token argument

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()  # no assertion on args
```

This test verifies the script was called but not that the correct token was passed. A regression that passes a hardcoded or empty token would not be caught. Assert on `call_args`.

### 7. No test for lock TTL expiry during payment

There is no test covering the scenario where the lock expires mid-payment and a second worker processes the same order. This is the highest-risk scenario for duplicate charges and should have an explicit integration-level test or at minimum a documented known limitation.

### 8. `requests` library has no connection timeout, only a read timeout

```python
timeout=self.REQUEST_TIMEOUT  # applies to read, not connect
```

Pass a tuple to set both:

```python
timeout=(5, self.REQUEST_TIMEOUT)  # (connect_timeout, read_timeout)
```
