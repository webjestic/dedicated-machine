Decision: Request Changes

Summary: The core locking and idempotency logic is well-structured — atomic `SET NX EX`, token-verified Lua release, and the idempotency check are all sound. However, there is a critical atomicity flaw that defeats the duplicate-charge guarantee, a silent exception swallowing issue that will cause unrecorded charges in production, and several secondary concerns that need to be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment call is outside the database transaction — the core guarantee is broken

This is the most serious flaw. The PR summary claims:

> *"charges the payment method and records the transaction within a single atomic database transaction"*

But that is not what the code does:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← network call to external system
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity over *database operations only*. `self.payment.charge()` is an HTTP call to an external payment gateway — it is not part of the database transaction. If `record_charge` or `mark_order_complete` raise after `charge` succeeds, the DB transaction rolls back, but the charge has already been captured. The order now has a real charge with no record of it. On the next retry, the idempotency check finds no existing charge and charges again. This is exactly the double-charge scenario the lock is supposed to prevent.

The standard pattern for this class of problem is to record a *pending* charge intent to the database *before* calling the payment gateway, then update the record to *confirmed* after the gateway responds. The idempotency check must also detect the pending state and treat it as "in-flight, do not re-charge." This way a crash after the gateway call but before the DB update is recoverable via a reconciliation job, not a silent double-charge.

```python
# Sketch of the correct ordering:
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)
# DB record exists now — any concurrent worker will see it and bail

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

### 2. `PaymentError` and all other exceptions inside the lock are silently swallowed

```python
try:
    with self._acquire_lock(req.order_id):
        ...
        with self.db.transaction():
            charge_id = self.payment.charge(...)
            ...
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

The `except LockError` block is the *only* handler. Any `PaymentError`, `requests.RequestException`, database exception, or unexpected `RuntimeError` raised inside the lock will propagate up the call stack uncaught. The caller gets an unhandled exception rather than an `OrderResult`. This is a contract violation — the method signature promises to return `OrderResult` in all cases. Add explicit handlers for `PaymentError` and a broad fallback:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, order_id=req.order_id, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

## Significant Issues

### 3. Timeout on payment gateway is dangerously long relative to the lock TTL

`REQUEST_TIMEOUT = 30` seconds per attempt, `MAX_RETRIES = 3`, `RETRY_BACKOFF = 5` seconds. Worst-case payment client execution time: `3 × 30 + 2 × 5 = 100 seconds`. The lock TTL is 60 seconds. The lock will expire mid-processing, another worker can acquire it, pass the idempotency check (charge not yet recorded), and issue a second charge while the first worker is still waiting on a retry. Either reduce the payment timeout, reduce retries, or increase the lock TTL to comfortably exceed the maximum payment client runtime with margin. These constants need to be co-designed, not set independently.

### 4. Timeout exceptions are retried but the charge may have already been captured

In `payment_client.py`, a `requests.exceptions.Timeout` triggers a retry. A timeout means the *client* gave up waiting — the server may have processed the charge successfully. Retrying without an idempotency key on the gateway request risks charging twice at the gateway level. The `charge` call should include a stable idempotency key (e.g., derived from `order_id`) in the request headers so the gateway can deduplicate:

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},
    timeout=self.REQUEST_TIMEOUT,
)
```

### 5. `payment_token` is logged at INFO level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` flows through `OrderRequest` and could easily end up in a log line during debugging or error reporting. `payment_token` is a PCI-scoped credential. It must never appear in logs. Add a `__repr__`/`__str__` override or a custom Pydantic field serializer that masks it, and add a note in the model:

```python
payment_token: str = Field(..., min_length=1, max_length=256)

    @field_serializer('payment_token')
    def mask_token(self, v: str) -> str:
        return '***'
```

## Minor Issues

### 6. `customer_id` has no pattern constraint

`order_id` correctly enforces `pattern=r'^[a-zA-Z0-9_\-]+'`. `customer_id` has only length bounds. Apply the same or a similarly restrictive pattern to prevent unexpected characters from reaching the database layer.

### 7. Lock retry backoff is linear and very short

`backoff = 0.5 * (attempt + 1)` gives 0.5s, 1.0s, 1.5s. For a 60-second lock TTL, this means all retries are exhausted in under 4 seconds. If the holding worker is mid-payment (which can take up to 30s), the retrying worker will always fail. Consider exponential backoff with jitter, or increase `LOCK_RETRY_ATTEMPTS` to be meaningful relative to the lock TTL.

### 8. Test for lock release verifies the mock was called, not the arguments

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This does not verify that the correct `keys` and `args` (specifically the token) were passed. A regression that passes the wrong token would not be caught. Assert on `call_args`:

```python
call_kwargs = processor._release_script.call_args
assert call_kwargs.kwargs['args'][0] == expected_token  # or inspect keys
```

### 9. `charge_id` missing from `OrderResult` on already-processed path

When an order is already processed, the existing `charge_id` is available from `existing` but is not returned. The caller has no way to retrieve the idempotent result. Return it:

```python
return OrderResult(
    processed=False,
    order_id=req.order_id,
    charge_id=existing.get("charge_id"),
    message="Order already processed",
)
```
