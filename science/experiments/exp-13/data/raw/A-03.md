# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Outside Database Transaction Boundary

This is the most severe flaw in the entire PR. The comment in the code claims:

> *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**, and dangerously so.

```python
with self.db.transaction():
    charge_id = self.payment.charge(          # ← External HTTP call INSIDE db transaction
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The `payment.charge()` call is an **external HTTP call to a payment gateway** wrapped inside a database transaction. This is a textbook distributed systems violation:

1. **Database connections are held open for the full duration of the HTTP round-trip** (up to 30 seconds per `PaymentClient.REQUEST_TIMEOUT`, multiplied by `MAX_RETRIES=3` — potentially 90+ seconds). Under any meaningful load this will exhaust your connection pool and collapse the service.

2. **A database transaction cannot span an external service call.** If the payment gateway succeeds but the database `COMMIT` fails, you have charged the customer with no record. If the database write succeeds but the connection drops before the caller receives confirmation, you have an inconsistent state. The comment's atomicity guarantee simply does not exist.

3. **Rollback semantics are broken.** If `record_charge` raises after a successful `payment.charge()`, the database transaction rolls back, but the charge has already been captured at the gateway. You now have a real charge with no database record — exactly the failure mode the architecture claims to prevent.

The correct pattern is to perform the charge **outside and before** the transaction, then record the result inside it:

```python
# Charge OUTSIDE the transaction — external side effect
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Record the result atomically in the database
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This still requires a compensating refund path if the database write fails after a successful charge, but it is at least honest about the failure modes and does not hold a database connection hostage to an HTTP call.

### 🔴 Critical: Redis Lock Does Not Cover the Idempotency Check Race Window

The lock is acquired correctly, but the idempotency check reads from the database **inside** the lock and **outside** the database transaction:

```python
with self._acquire_lock(req.order_id):
    existing = self.db.get_charge(req.order_id)   # ← Read outside transaction
    if existing:
        return ...

    with self.db.transaction():                    # ← Write begins here
        charge_id = self.payment.charge(...)
        self.db.record_charge(...)
```

If the database read uses a different connection or isolation level than the write transaction, a concurrent worker that somehow bypasses the lock (Redis failure, TTL expiry, network partition) could pass the idempotency check before the first worker's transaction commits. The idempotency check and the write must be atomic with respect to each other. The `get_charge` call should be inside the transaction, and `record_charge` should enforce a unique constraint at the database level as the final backstop. The Redis lock is a performance optimization, not a correctness guarantee — the database must be the authority.

---

### 🔴 Critical: Lock TTL Is Insufficient for the Actual Worst-Case Duration

```python
ORDER_LOCK_TTL = 60  # seconds — covers typical payment round-trip
```

`PaymentClient` can retry up to `MAX_RETRIES=3` times with `RETRY_BACKOFF=5` seconds between attempts and `REQUEST_TIMEOUT=30` seconds per attempt. Worst-case payment duration:

```
3 attempts × 30s timeout + 2 gaps × 5s backoff = 100 seconds
```

The lock TTL of 60 seconds **expires before the payment client can exhaust its retries**. A second worker will acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge while the first worker is still waiting on a retry. The TTL must be set to at least the maximum observable duration of everything inside the lock, with margin. Either increase `ORDER_LOCK_TTL` to ≥ 120 seconds or — better — align the payment client's total timeout budget with the lock TTL by design and enforce that relationship in code.

---

### 🟡 Significant: Unhandled Exceptions Leak the Lock Silently

```python
try:
    with self._acquire_lock(req.order_id):
        ...
except LockError:
    return OrderResult(...)
```

Only `LockError` is caught at the outer level. If `self.db.get_charge()`, `self.payment.charge()`, `self.db.record_charge()`, or `self.db.mark_order_complete()` raises any other exception — `DatabaseError`, `PaymentError`, `ConnectionError`, an unexpected `KeyError` from `response.json()["charge_id"]` — the exception propagates out of `process()` entirely unhandled. The `finally` block in `_acquire_lock` will still release the lock, which is correct, but the caller receives a raw exception rather than a structured `OrderResult`. This breaks the contract of the method signature and will cause unhandled exceptions in whatever is calling `process()`. All anticipated exception types must be caught and mapped to `OrderResult` responses.

---

### 🟡 Significant: `payment_token` Logged Indirectly via Pydantic Validation Errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation error messages include the field value that failed validation. If `payment_token` fails validation (e.g., too long, wrong type), the raw token value may appear in log output. Payment tokens are sensitive credentials. Validation error logging must either redact sensitive fields or log only field names and error types, never values.

## `services/payment_client.py`

### 🟡 Significant: Timeout Retry Logic Charges on Unknown State

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    logger.warning("Payment timeout attempt=%d", attempt + 1)
    # falls through to retry
```

On a timeout, the gateway may have received and processed the request — the response simply did not arrive in time. Retrying without an idempotency key sent to the gateway means the client may issue a second charge for a request that already succeeded. This is a well-known problem in payment systems. Every charge request must include a client-generated idempotency key (typically passed as a header like `Idempotency-Key`) so that the gateway can deduplicate retries. Without this, the retry logic actively creates the double-charge scenario the broader system is trying to prevent.

---

### 🟡 Significant: `response.json()["charge_id"]` Is Unguarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 2xx response with a malformed body, missing `charge_id`, or non-JSON content type, this line raises `KeyError` or `JSONDecodeError`. These exceptions are not caught anywhere in `PaymentClient.charge()` or in `OrderProcessor.process()`. A successful charge at the gateway would have no recorded `charge_id` and no path to recovery. This must be wrapped with explicit error handling and a clear exception type.

## `tests/test_order_processor.py`

### 🟠 Moderate: Tests Do Not Cover the Most Critical Failure Modes

The test suite validates the happy path and basic contention, but is entirely absent on the scenarios that matter most for a payment processor:

- No test for `PaymentError` raised during `payment.charge()` — does `process()` return a structured result or raise?
- No test for database failure after a successful charge — the most dangerous failure mode given the architectural flaw above.
- No test verifying that the lock is **released** when an exception occurs mid-processing (the `finally` path).
- No test for `response.json()` missing `charge_id` in `PaymentClient`.
- No test for lock TTL expiry during processing.
- `test_lock_release_uses_lua_script` asserts `_release_script` was called once but does not assert it was called with the correct token, which is the entire point of the token-verified release.

---

### 🟠 Moderate: `test_process_lock_contention` Does Not Actually Test Retry Behavior

```python
def test_process_lock_contention(redis_client, db, payment):
    redis_client.set.return_value = False
```

`LOCK_RETRY_ATTEMPTS = 3`, but this test does not assert that `redis.set` was called exactly 3 times before giving up. The retry loop could be broken (e.g., someone changes `range(self.LOCK_RETRY_ATTEMPTS)` to `range(1)`) and this test would still pass.
