Decision: Request Changes

Summary: The architecture demonstrates solid foundational instincts — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all present and correctly structured. However, there are several non-trivial correctness and security issues that must be resolved before this handles real payment workloads, the most critical being a charge-before-transaction pattern that breaks the at-most-once guarantee the PR explicitly claims to provide, and a timeout race that can cause the lock to silently expire mid-operation.

---

## Critical Issues

### 1. Payment charge is executed *inside* the database transaction — this does not make it atomic

This is the most dangerous flaw in the PR, and it directly contradicts the stated guarantee.

```python
# services/order_processor.py, lines ~90–96
with self.db.transaction():
    charge_id = self.payment.charge(       # <-- external network call INSIDE db txn
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction does not wrap an external HTTP call. These are two entirely separate systems with no shared commit protocol. The scenarios this creates:

- **Charge succeeds, DB commit fails** → customer is charged, no record exists, idempotency check on retry returns `None`, order gets charged *again*. This is a double-charge.
- **Charge succeeds, process crashes before `record_charge`** → same outcome as above.
- **DB transaction held open during the full payment gateway round-trip** → up to 90 seconds of lock held on a DB connection, connection pool exhaustion under load.

The correct pattern is **charge-then-record**, outside the transaction for the charge, with the DB write being the commit of truth:

```python
# Charge FIRST, outside any db transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)

# Then atomically record — if this fails, the idempotency check
# must be able to detect the charge via a pre-insert pending record
# or the payment gateway's own idempotency key
with self.db.transaction():
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

Even this is not fully safe without an **idempotency key passed to the payment gateway** so that a retry of a timed-out charge does not create a second charge. The `PaymentClient.charge()` method sends no idempotency key whatsoever.

---

### 2. Redis lock TTL can expire before the operation completes — silent correctness violation

```python
ORDER_LOCK_TTL = 60   # seconds
REQUEST_TIMEOUT = 30  # seconds per attempt
MAX_RETRIES = 3
RETRY_BACKOFF = 5     # seconds between retries
```

Worst-case payment client execution time: `3 attempts × 30s timeout + 2 gaps × 5s = 100 seconds`. The lock TTL is 60 seconds. The lock expires while the worker is still inside the critical section. A second worker can now acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The lock provides no protection in this scenario.

Options:
- Reduce `MAX_RETRIES` and `REQUEST_TIMEOUT` to fit within the TTL with margin.
- Implement a lock heartbeat/refresh thread that extends the TTL while the operation is in progress.
- Use Redlock if multi-node Redis HA is required (though this introduces its own tradeoffs).

At minimum, enforce a documented invariant: `MAX_RETRIES × REQUEST_TIMEOUT + (MAX_RETRIES - 1) × RETRY_BACKOFF << ORDER_LOCK_TTL`, and add an assertion or startup check.

### 3. Payment client retries on timeout without an idempotency key — guaranteed double-charge path

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout does not mean the charge failed. The gateway may have processed the charge and the response was lost in transit. Retrying without an idempotency key issues a second charge for the same amount. This is a financial correctness bug.

The fix is to generate a stable idempotency key per order (e.g., derived from `order_id`) and pass it as a header on every attempt:

```python
headers = {"Idempotency-Key": f"order-{order_id}"}
response = requests.post(..., headers=headers)
```

The key must be stable across retries for the same order, not regenerated per attempt.

## Significant Issues

### 4. Lock release silently swallows all exceptions

```python
finally:
    self._release_script(keys=[lock_key], args=[token])
    logger.info("Order lock released order_id=%s", order_id)
```

If the Redis call raises (connection loss, Redis restart), the exception is suppressed entirely. The lock will eventually expire via TTL, but the caller receives no signal that release failed. At minimum, log at `ERROR` level. Consider whether the calling layer needs to know.

```python
finally:
    try:
        released = self._release_script(keys=[lock_key], args=[token])
        if not released:
            logger.error("Lock release returned 0 — token mismatch or key expired order_id=%s", order_id)
    except Exception:
        logger.exception("Failed to release lock order_id=%s — will expire via TTL", order_id)
```

### 5. Idempotency check is not inside the lock scope for the full duration

The check is correctly inside the lock, but the `db.get_charge()` read and the subsequent write are not in the same database transaction. A read-then-write with a gap is a TOCTOU window. Under normal operation the Redis lock closes this window, but if the lock expires (see issue 2), two workers can both read `None` and both proceed to charge. The DB transaction should include the idempotency check as a `SELECT FOR UPDATE` or equivalent to provide a second layer of defense.

### 6. `PaymentError` is not caught in `process()`

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)   # raises PaymentError — uncaught
```

`PaymentError` propagates out of `process()` as an unhandled exception. The caller receives a raw exception rather than an `OrderResult`. This breaks the stated contract of the method. Catch it and return a structured `OrderResult(processed=False, message=...)`.

### 7. `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises `KeyError` or `JSONDecodeError`, which propagates as an unhandled exception through the retry loop and out of `charge()`. Wrap this in a `try/except` and raise a `PaymentError` with a descriptive message.

## Minor Issues

### 8. Lock retry backoff is linear and short

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s
```

Three attempts with linear backoff totaling 3 seconds is very aggressive under genuine contention. Consider exponential backoff with jitter to reduce thundering herd behavior across workers.

### 9. Test coverage gaps

- No test for `PaymentError` propagating out of `process()` (which is currently unhandled — see issue 6).
- No test for lock TTL expiry mid-operation.
- No test for the payment client timeout-then-retry path.
- No test verifying that the idempotency key is passed to the payment gateway (once added).
- `test_lock_release_uses_lua_script` asserts `redis_client.delete` is not called, but does not assert the Lua script was called with the correct token, making it a weak correctness assertion.

### 10. `payment_token` logged indirectly via `charge_id`

Not a current issue, but ensure that `payment_token` is never interpolated into log messages in future changes. The structured logging here is clean; flag it as a policy note for reviewers.
