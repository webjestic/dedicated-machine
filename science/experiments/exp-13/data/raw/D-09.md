Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the DB transaction wrapping are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues that need to be addressed before this is production-ready.

---

## Critical Issues

### 1. Payment call is inside the DB transaction — this does not make the operation atomic, it makes it worse

```python
# Current code
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — this is incorrect. The DB transaction provides no atomicity guarantee over an external HTTP call. What actually happens:

- `payment.charge()` succeeds and money moves
- The DB transaction rolls back (network partition, DB error, process crash, timeout)
- The charge is now unrecorded — exactly the failure mode the comment claims is prevented

Worse, holding an open DB transaction for the full duration of a payment gateway round-trip (up to `REQUEST_TIMEOUT * MAX_RETRIES = 90 seconds`) pins a connection and holds locks. Under any real load this will exhaust your connection pool.

**Fix:** Call the payment gateway *before* opening the DB transaction. Accept that you need a reconciliation/recovery path for the gap between charge success and record failure — that is the correct design for this problem, not wrapping an HTTP call in a DB transaction.

```python
# Correct structure
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Gateway call is done. Now record durably.
try:
    with self.db.transaction():
        self.db.record_charge(...)
        self.db.mark_order_complete(...)
except Exception:
    # charge succeeded but record failed — log with full context for reconciliation
    logger.critical(
        "RECONCILIATION REQUIRED: charge succeeded but record failed "
        "order_id=%s charge_id=%s amount_cents=%s customer_id=%s",
        req.order_id, charge_id, req.amount_cents, req.customer_id,
    )
    raise
```

### 2. Unhandled exceptions inside the lock context silently swallow errors

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, DB exceptions, and any unexpected exception propagate out of the `with self._acquire_lock` block. The `finally` in `_acquire_lock` correctly releases the lock, but the exception then propagates out of `process()` entirely — it is not caught, not logged at the call site, and not converted to an `OrderResult`. The caller gets an unhandled exception instead of a structured result.

```python
# Missing handler — add this around the lock block
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal error")
```

## Significant Issues

### 3. Redis failure is not handled — it takes down the entire service

If `self.redis.set(...)` raises (connection refused, timeout, cluster failover), the exception propagates uncaught. For a payment processor this is a hard availability problem. Redis connectivity issues should produce a structured `OrderResult`, not an unhandled exception that crashes the worker.

---

### 4. The lock TTL does not account for actual worst-case processing time

```python
ORDER_LOCK_TTL = 60  # seconds
```

`PaymentClient` can take up to `REQUEST_TIMEOUT * MAX_RETRIES + RETRY_BACKOFF * (MAX_RETRIES - 1) = 30 * 3 + 5 * 2 = 100 seconds` in the worst case. The lock expires after 60 seconds. A second worker can then acquire the lock while the first is still mid-charge, defeating the entire purpose of the lock. The TTL must be greater than the maximum possible processing time, or you need lock extension (Redlock-style keepalive).

---

### 5. `payment_token` is logged at INFO level in `PaymentClient`

```python
logger.info("Payment successful charge_id=%s attempt=%d", charge_id, attempt + 1)
```

This is fine, but review the full request payload — `json={"token": token, "amount": amount_cents}` — and confirm your HTTP client library is not logging request bodies at DEBUG. Payment tokens in logs is a PCI DSS finding. Add explicit scrubbing or confirm your log configuration.

---

### 6. Test coverage gaps

The test suite does not cover:

- Payment gateway raises `PaymentError` — what does `process()` return? (Currently: unhandled exception)
- DB transaction rollback after successful charge — the reconciliation path
- Lock TTL expiry mid-processing (the race condition in issue #4)
- `PaymentClient` retry behavior: verify that a 5xx followed by a 200 returns the correct `charge_id` and that a 4xx does not retry

---

### 7. Minor: `response.json()["charge_id"]` has no error handling

```python
charge_id = response.json()["charge_id"]
```

A `KeyError` here (malformed gateway response) raises an untyped exception that bypasses the retry logic and surfaces as an internal error. Wrap this:

```python
data = response.json()
if "charge_id" not in data:
    raise PaymentError(f"Malformed gateway response: missing charge_id")
charge_id = data["charge_id"]
```

## What Is Done Well

- `SET NX EX` for atomic lock acquisition is correct
- Lua script for token-verified release is correct and necessary
- Pydantic validation with pattern enforcement on `order_id` is solid
- Retry with backoff on 5xx is the right behavior
- Not retrying on 4xx (declined) is correct
- Structured logging with consistent field names throughout
- Test for path traversal on `order_id` is a good catch
