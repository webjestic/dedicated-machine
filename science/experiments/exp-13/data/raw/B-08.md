Decision: Request Changes

Summary: The core locking and idempotency design is solid — atomic SET NX EX, token-verified Lua release, and the charge-within-transaction pattern are all correct. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that undermines the at-most-once guarantee the PR claims to provide, plus several secondary issues in the payment client and test coverage that need to be addressed before this is production-safe.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* and that the charge and record happen *"atomically."* This is incorrect and the misunderstanding is dangerous.

```python
# services/order_processor.py  (lines inside with self.db.transaction())
with self.db.transaction():
    charge_id = self.payment.charge(...)   # ← external HTTP call, NOT part of DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

A database transaction provides atomicity only over database operations. `self.payment.charge()` is an outbound HTTP call to an external payment gateway. If the process crashes, the network drops, or the DB commit fails **after** the charge succeeds but **before** the commit lands, you have charged the customer with no record. Wrapping an external I/O call inside a DB transaction does not roll it back.

**Required fix — use a two-phase write pattern:**

```python
# Phase 1: persist intent before charging
with self.db.transaction():
    self.db.record_charge_pending(
        req.order_id, req.customer_id, req.amount_cents
    )

# Phase 2: charge externally
try:
    charge_id = self.payment.charge(
        token=req.payment_token,
        amount_cents=req.amount_cents,
    )
except PaymentError:
    with self.db.transaction():
        self.db.mark_charge_failed(req.order_id)
    raise

# Phase 3: confirm
with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must also handle the `pending` state — a worker recovering from a crash should detect a pending record and either complete or void it, not re-charge.

### 2. Lock TTL does not account for payment retry duration

```python
ORDER_LOCK_TTL = 60   # services/order_processor.py
```

```python
REQUEST_TIMEOUT = 30  # services/payment_client.py
MAX_RETRIES     = 3
RETRY_BACKOFF   = 5
```

Worst-case payment client runtime: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock expires at 60 seconds. A second worker can acquire the lock, pass the idempotency check (no charge recorded yet), and issue a duplicate charge. These two constants are in separate files with no enforced relationship — this will silently break under load.

**Required fix:** Either derive one from the other, or add a startup assertion:

```python
# In OrderProcessor.__init__
assert self.ORDER_LOCK_TTL > (
    self.payment.MAX_RETRIES * self.payment.REQUEST_TIMEOUT
    + (self.payment.MAX_RETRIES - 1) * self.payment.RETRY_BACKOFF
), "Lock TTL too short to cover maximum payment client runtime"
```

## Significant Issues

### 3. Timeout exception on retry does not distinguish idempotent from non-idempotent failure

```python
# services/payment_client.py
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # falls through to retry
```

A timeout means the request may or may not have been received and processed by the gateway. Retrying unconditionally on timeout can produce duplicate charges at the gateway level if the gateway processed the first request but the response was lost in transit. The payment client needs to either pass a caller-supplied idempotency key to the gateway on every attempt, or treat timeouts as terminal and require the caller to reconcile via a separate status check endpoint.

```python
# Recommended
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # passed in, stable across retries
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

### 4. `charge_id` key access is unguarded

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected body, this raises an unhandled `KeyError` inside the lock context. That exception will propagate up through `process()`, bypass the `LockError` handler, and surface as an unhandled exception to the caller with no structured result. Add explicit handling:

```python
data = response.json()
if "charge_id" not in data:
    raise PaymentError(f"Gateway response missing charge_id: {data}")
charge_id = data["charge_id"]
```

### 5. Lock contention retry backoff is too aggressive and blocks the thread

```python
backoff = 0.5 * (attempt + 1)   # 0.5s, 1.0s, 1.5s — linear, not exponential
time.sleep(backoff)
```

Linear backoff with `time.sleep` blocks the worker thread. Under high concurrency this degrades throughput across all orders, not just the contested one. Use exponential backoff with jitter, and document that this service is not suitable for async/event-loop contexts without converting to `asyncio.sleep`.

## Test Coverage Gaps

### 6. No test for the lock-expiry / double-charge race condition

There is no test that verifies behavior when the payment call outlasts the lock TTL. This is the most dangerous failure mode and it is entirely untested.

### 7. No test for payment timeout retry behavior

The retry logic in `PaymentClient` is not tested at all. There are no tests for: timeout followed by success, timeout exhaustion, 5xx followed by success, or 4xx immediate failure.

### 8. `test_lock_release_uses_lua_script` does not verify the token

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test only confirms the script was called — it does not verify the correct token was passed as `args`. A bug that passes the wrong token or an empty args list would pass this test.

```python
# Should assert something like:
call_args = processor._release_script.call_args
assert call_args.kwargs["args"][0] == <the token used during acquisition>
```

## Minor Issues

- `payment_token` is logged indirectly via `payment_token` field presence in `OrderRequest`. Confirm it never appears in log output at `DEBUG` level if debug logging is enabled in any environment.
- `CHARGE_ENDPOINT` is a hardcoded string constant. This should be injected via configuration to support environment-specific routing and to avoid the internal hostname appearing in source control.
- `requests.post` has no connection timeout separate from the read timeout. Use `timeout=(connect_timeout, read_timeout)` to avoid indefinite hangs on connection establishment.
