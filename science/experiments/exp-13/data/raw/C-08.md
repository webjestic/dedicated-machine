Decision: Request Changes

Summary: The architecture is well-structured and the checklist items are largely implemented correctly — the Lua-based token-verified lock release, Pydantic validation, and idempotency check are all sound. However, there are several correctness and reliability flaws that must be addressed before this can be approved, the most critical being a charge-before-commit race that breaks the at-most-once guarantee the PR claims to provide. Additional gaps in error handling, observability, and test coverage also need to be resolved.

---

## Critical Issues

### 1. Payment charge occurs inside a database transaction — violates at-most-once guarantee

This is the most serious flaw. The PR comment says:

> *"crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**. The charge happens inside `self.db.transaction()`, but the payment gateway call is a **non-transactional, external side effect**. If the process crashes or the DB connection drops after `self.payment.charge()` succeeds but before `self.db.record_charge()` commits, the customer has been charged with no record in the database. The Redis lock will eventually expire and the idempotency check (`db.get_charge`) will find nothing, allowing a **second charge** on retry.

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side effect — cannot be rolled back
    self.db.record_charge(...)             # if this or commit fails, charge is unrecorded
```

**Required fix:** Use a two-phase approach. Record a pending/in-flight state *before* charging, then charge, then mark complete. The idempotency check must treat an in-flight record as "already processing" and the recovery path must handle the case where a charge was initiated but the outcome is unknown.

```python
# Correct pattern
with self.db.transaction():
    self.db.record_charge_pending(req.order_id, req.customer_id, req.amount_cents)

charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)

with self.db.transaction():
    self.db.record_charge_complete(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

A recovery/reconciliation job should also be defined for orders stuck in `pending` state beyond a threshold.

### 2. Timeout during payment charge creates an unresolvable ambiguity

In `payment_client.py`, a `requests.exceptions.Timeout` is caught and retried. However, a timeout means the request **may or may not have been processed** by the gateway. Retrying after a timeout can result in a double charge if the first request succeeded but the response was lost in transit.

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # Retrying here may charge the customer twice
```

**Required fix:** Either (a) require the payment gateway to support idempotency keys and pass a stable per-order idempotency key on every attempt, or (b) do not retry on timeout — surface the ambiguity to the caller for manual reconciliation. Most production payment APIs (Stripe, etc.) support idempotency keys for exactly this reason.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # must be passed in
    timeout=self.REQUEST_TIMEOUT,
)
```

## Significant Issues

### 3. Lock TTL is too short and not justified for the actual critical section

`ORDER_LOCK_TTL = 60` seconds covers a single payment round-trip, but the actual worst case is: lock acquisition + DB read + DB transaction open + payment charge (up to `30s × 3 retries + 5s × 2 backoffs = 100s`) + DB commit. The maximum payment client execution time **exceeds the lock TTL**, meaning the lock can expire while the payment is in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge.

**Required fix:** Either extend `ORDER_LOCK_TTL` to safely exceed the maximum end-to-end processing time (≥ 120–150s with margin), or implement lock extension (Redlock-style `PEXPIRE` refresh) during long operations.

---

### 4. Redis unavailability silently allows concurrent processing

If `self.redis.set(...)` raises a `redis.RedisError` (connection failure, timeout), the exception propagates uncaught through `_acquire_lock` and then through `process()`, returning an unhandled exception to the caller rather than a safe `OrderResult`. More critically, there is no explicit policy decision here: **should Redis unavailability block all order processing or allow a degraded fallback?** That decision must be made explicitly and documented.

```python
# Current: redis.RedisError propagates uncaught
acquired = self.redis.set(lock_key, token, nx=True, ex=self.ORDER_LOCK_TTL)
```

**Required fix:** Catch `redis.RedisError` explicitly in `_acquire_lock`, log it, and either raise a distinct `LockError` (safe — blocks processing) or document a deliberate fallback policy. Do not let infrastructure exceptions leak as untyped exceptions.

---

### 5. `payment_token` is logged indirectly and may appear in tracebacks

`payment_token` is stored on the validated `OrderRequest` object. If any exception is raised and logged with `%s` formatting against the request object (e.g., in a future log line or unhandled exception handler), the token will appear in logs. Payment tokens must be treated as secrets.

**Required fix:** Either exclude `payment_token` from `__repr__`/`__str__` on `OrderRequest` using Pydantic's `model_config` with a custom `__repr__`, or use a dedicated secret type.

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
    
    model_config = ConfigDict(str_strip_whitespace=True)
```

## Minor Issues

### 6. `CHARGE_ENDPOINT` is a hardcoded string — not environment-configurable

```python
CHARGE_ENDPOINT = "https://api.payments.internal/v1/charges"
```

This must come from configuration/environment variables, not be hardcoded in source. This is a basic twelve-factor requirement and a security concern for a government contracting context.

---

### 7. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute was called — it is testing implementation, not contract. It will break on any internal refactor and provides no behavioral guarantee. Replace with a test that verifies the lock key is absent after processing, or that the Lua script is called with the correct key and token arguments.

---

### 8. No test coverage for Redis failure, payment failure, or DB transaction rollback

The test suite covers the happy path, duplicate detection, lock contention, and input validation — but misses the following cases that are critical for a payment processor:

- Redis raises `RedisError` during lock acquisition
- `payment.charge()` raises `PaymentError` — does the DB transaction roll back cleanly?
- `db.record_charge()` raises an exception after a successful charge — is the charge orphaned?
- Lock TTL expiry mid-processing (simulate with a mock that returns `None` on `get`)

---

### 9. `amount_cents` upper bound of 1,000,000 ($10,000) may be a business logic constraint that belongs in domain configuration

Hard-coding financial limits in a validation model is a maintenance risk. If this limit changes, it requires a code change and redeployment. Consider making it a configurable constant or pulling it from a policy configuration layer, especially in a government contracting context where contract values may vary significantly by agency.
