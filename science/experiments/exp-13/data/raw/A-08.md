Decision: Request Changes

Summary: The architecture demonstrates solid foundational intent — token-verified Lua release, idempotent checks, and Pydantic validation are all directionally correct. However, there are several non-trivial correctness flaws, one critical safety gap around payment atomicity, and a handful of operational concerns that must be addressed before this touches production. The issues range from a genuine at-most-once violation to a silent exception swallow that could mask data integrity failures.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — this does not provide the atomicity guarantee the comment claims

This is the most serious flaw in the PR. The docstring and inline comment both assert:

> *"Crash between charge and record cannot produce an unrecorded charge"*

That claim is **false**. A database transaction does not wrap the payment gateway call. The sequence is:

```
BEGIN TRANSACTION
  → payment.charge()       # external side effect — not rolled back on DB failure
  → db.record_charge()
  → db.mark_order_complete()
COMMIT
```

If `db.record_charge()` or `db.mark_order_complete()` raises, or if the process crashes after `payment.charge()` returns but before `COMMIT`, the charge has been collected from the customer and **no record exists in the database**. The Redis lock has also been released by the `finally` block, meaning a retry will pass the idempotency check and charge again.

The standard mitigation here is the **outbox / two-phase approach**: record a `PENDING` charge row *before* calling the gateway, then mark it `COMPLETE` after. The idempotency check must look for `PENDING` rows as well. Alternatively, the payment gateway must support idempotency keys so a retry of the same logical operation is safe.

```python
# Suggested skeleton
with self.db.transaction():
    self.db.reserve_charge(req.order_id, req.customer_id, req.amount_cents)
    # COMMIT — crash here means no charge was attempted

charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # gateway deduplicates on its side
)

with self.db.transaction():
    self.db.confirm_charge(req.order_id, charge_id)
    self.db.mark_order_complete(req.order_id)
```

The idempotency check must then also handle the `PENDING` state:

```python
existing = self.db.get_charge(req.order_id)
if existing and existing["status"] in ("pending", "complete"):
    # decide whether to resume or reject
```

### 2. `PaymentError` is not caught in `process()` — unhandled exception escapes the public interface

`payment.charge()` raises `PaymentError` on permanent failure or retry exhaustion. `process()` catches only `LockError`. Any `PaymentError` (or `requests` exception that leaks through) propagates to the caller as an unhandled exception, bypassing the structured `OrderResult` return contract entirely.

```python
# Missing in process():
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment failed")
```

This also means the lock `finally` block *will* still fire and release the lock, but the caller receives an exception instead of an `OrderResult`, breaking the stated interface.

## Significant Issues

### 3. Lock TTL of 60 seconds is dangerously short given the retry configuration

`ORDER_LOCK_TTL = 60` seconds, but the payment client can take up to:

```
MAX_RETRIES * (REQUEST_TIMEOUT + RETRY_BACKOFF) = 3 * (30 + 5) = 105 seconds
```

The lock can expire while a worker is still inside the critical section, mid-retry on a slow gateway. A second worker then acquires the lock, passes the idempotency check (no charge recorded yet), and issues a second charge. The first worker's Lua release will then silently return `0` (token mismatch) — the lock was already held by worker 2, which is now also charging. This is a direct double-charge path.

Either raise `ORDER_LOCK_TTL` well above the maximum possible payment duration (e.g., 180–300 seconds with margin), or implement lock extension (Redlock-style keepalive), or reduce the payment client's maximum wall-clock time.

---

### 4. Retry backoff in `_acquire_lock` is linear and cumulative — total wait can exceed caller expectations

```python
backoff = 0.5 * (attempt + 1)  # 0.5s, 1.0s, 1.5s
```

With `LOCK_RETRY_ATTEMPTS = 3`, total blocking time is up to 3 seconds before raising `LockError`. This is synchronous blocking on a thread. In any async or thread-pool context this will stall the worker. This should be documented explicitly, and the retry strategy should use jitter to prevent thundering herd under high contention:

```python
import random
backoff = min(0.5 * (2 ** attempt) + random.uniform(0, 0.1), 5.0)
```

---

### 5. `payment_token` is logged indirectly through Pydantic validation errors

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic validation errors include field names and the submitted values in their string representation. If `payment_token` fails validation (e.g., too long), the raw token value may appear in log output. Payment tokens are sensitive credentials. Either scrub the error before logging or log only field names:

```python
logger.warning("Invalid order request fields: %s", [e["loc"] for e in exc.errors()])
```

---

### 6. `requests.exceptions.ConnectionError` and other network exceptions are not caught in `PaymentClient.charge()`

The retry loop handles `Timeout` and `HTTPError`, but `requests.exceptions.ConnectionError` (DNS failure, TCP reset, connection refused) is not caught and will propagate immediately without retry. In a microservices environment this is a common transient failure mode.

```python
except (requests.exceptions.Timeout, requests.exceptions.ConnectionError):
    last_error = "connection error"
    logger.warning("Payment connection error attempt=%d", attempt + 1)
```

## Minor Issues

### 7. `charge_id` extraction from response has no error handling

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed body, or the `charge_id` key is absent, this raises `KeyError` or `JSONDecodeError` — neither of which is caught. This should be wrapped:

```python
try:
    charge_id = response.json()["charge_id"]
except (KeyError, ValueError) as e:
    raise PaymentError(f"Malformed gateway response: {e}") from e
```

---

### 8. Test for lock release verifies the mock was called but not the arguments

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor._release_script.assert_called_once()
```

This passes even if the script is called with the wrong key or a hardcoded token. The test should assert `keys` and that `args` contains a valid UUID:

```python
call_kwargs = processor._release_script.call_args
assert call_kwargs.kwargs["keys"] == ["order:lock:order-001"]
assert len(call_kwargs.kwargs["args"][0]) == 36  # UUID length
```

---

### 9. No test covers the lock-expiry-during-payment scenario (the double-charge path from issue 3)

The test suite has no coverage for the case where the lock TTL expires mid-payment. This is the highest-risk scenario and should have an explicit integration-level test or at minimum a documented known limitation.

---

### 10. `ORDER_LOCK_TTL` and `LOCK_RETRY_ATTEMPTS` are class-level constants with no override path

These are hardcoded on the class. Any environment that needs different tuning (staging vs. production, different payment SLA tiers) requires a code change. These should be constructor parameters with defaults:

```python
def __init__(self, redis_client, db, payment_client,
             lock_ttl: int = 60, lock_retry_attempts: int = 3):
    self.ORDER_LOCK_TTL = lock_ttl
    self.LOCK_RETRY_ATTEMPTS = lock_retry_attempts
```
