# Pull Request Review

## `services/order_processor.py`

### 🔴 Critical: Payment Charge Occurs Inside the Database Transaction — Unsafe External Side Effect

This is the most serious architectural flaw in this PR. The payment gateway call is made *inside* `self.db.transaction()`:

```python
with self.db.transaction():
    charge_id = self.payment.charge(...)   # External network call inside DB transaction
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The PR summary claims *"crash between charge and record cannot produce an unrecorded charge"* — this is **incorrect reasoning** and the design is actively dangerous.

**Problem 1 — DB transaction held open during a 30-second network call.** The payment client has a `REQUEST_TIMEOUT = 30` seconds and `MAX_RETRIES = 3` with `RETRY_BACKOFF = 5` seconds. Under worst-case retry behavior, the database transaction can be held open for over **two minutes**. This will exhaust your DB connection pool under any meaningful concurrency, cause lock contention on DB rows, and degrade the entire system.

**Problem 2 — The atomicity claim is false.** If `self.payment.charge()` succeeds but the DB transaction subsequently fails to commit (network partition to DB, deadlock, disk error), the charge has been executed against the payment gateway but is **never recorded**. The customer is charged with no record. This is the exact double-charge/lost-charge scenario the design claims to prevent. External I/O cannot participate in a database transaction's atomicity guarantees.

**Problem 3 — Retry amplification risk.** The payment client retries on 5xx and timeouts. If a charge request reaches the gateway and succeeds, but the response is lost in transit (timeout), the client will retry and potentially issue a second charge. There is no idempotency key being sent to the payment gateway. This is a duplicate charge vector that exists entirely independently of the Redis lock.

**Correct pattern:** Charge outside the transaction, then open a short-lived DB transaction only for the record operations. Use an idempotency key on the gateway request. Implement a reconciliation/recovery path for the case where charge succeeds but DB write fails.

```python
# Correct structure (sketch):
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # Gateway-level dedup
)
# DB transaction is now short and contains no external I/O
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

### 🔴 Critical: Redis Lock Does Not Cover the Idempotency Check Window — TOCTOU Race

The idempotency check reads from the database, then charges, then writes back to the database. The Redis lock is meant to serialize this sequence, and it does — *under normal operation*. However:

**The Redis lock TTL is 60 seconds. The maximum time inside the lock can exceed 60 seconds** (payment retries alone can take ~75 seconds as noted above). If the lock expires mid-operation, a second worker can acquire the lock, pass the idempotency check (charge not yet recorded), and issue a second charge. The lock TTL must be strictly greater than the maximum possible execution time inside the critical section, and that maximum is not bounded or enforced anywhere in this code.

This is a fundamental property of TTL-based distributed locks: the TTL is not a safety guarantee, it is a liveness guarantee. The code treats it as a safety guarantee.

---

### 🟡 Significant: `payment_token` Logged Indirectly via `ValidationError`

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)
```

Pydantic's `ValidationError` string representation includes field values in some versions and configurations. If `payment_token` fails validation, its value (or a truncated form) may appear in log output. Payment tokens must never appear in logs. At minimum, scrub or explicitly format the error message to exclude sensitive field values.

---

### 🟡 Significant: Unhandled Exceptions Inside the Lock Are Silently Swallowed

```python
with self._acquire_lock(req.order_id):
    ...
    with self.db.transaction():
        charge_id = self.payment.charge(...)
        ...
```

`PaymentError`, database exceptions, and any unexpected runtime errors raised inside the lock are **not caught**. They will propagate up through the `_acquire_lock` context manager (which correctly releases the lock in `finally`), but `process()` has no outer handler for them. The caller receives an unhandled exception rather than an `OrderResult`. The method signature promises `OrderResult` as a return type — this contract is broken on any non-`LockError` failure path. These should be caught and returned as structured failure results, and the exception details should be logged without exposing sensitive fields.

---

### 🟡 Significant: Lock Retry Backoff Is Blocking — Threads Stall

```python
time.sleep(backoff)  # 0.5s, 1.0s
```

In any threaded or async worker environment, `time.sleep()` in the lock acquisition path blocks the entire thread for up to 1.5 seconds cumulative. Under high contention this compounds. If this runs in an async context (FastAPI, asyncio workers), this is a correctness issue, not just a performance one — it will block the event loop. The backoff strategy should use `asyncio.sleep` in async contexts or be documented as sync-only with explicit guidance.

---

### 🟡 Significant: `payment_token` Passed Through as Raw String With No Masking in Logs

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This specific line is fine, but `payment_token` is stored on `req` and accessible throughout the method scope. There is no masking or clearing of the token after the charge call completes. In environments with heap dumps, exception traceback logging, or debug-level logging of local variables (e.g., Sentry with local variable capture), the token can leak. Consider clearing or masking it after use.

## `services/payment_client.py`

### 🔴 Critical: No Idempotency Key on Charge Requests

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    timeout=self.REQUEST_TIMEOUT,
)
```

As noted above, the retry loop will re-POST on timeout with no idempotency key. If the first request reached the gateway and was processed, the retry issues a second charge. This is a financial correctness bug. Every charge request must include a stable idempotency key (the `order_id` is the natural candidate) that the gateway uses to deduplicate.

---

### 🟡 Significant: `requests.exceptions.ConnectionError` Is Not Handled

The retry loop catches `Timeout` and `HTTPError`. A `ConnectionError` (DNS failure, TCP reset, refused connection) is not caught and will propagate as an unhandled exception rather than triggering the retry path. Given this is described as an internal gateway, transient connectivity issues are plausible and should be retried under the same logic.

---

### 🟡 Significant: Response Structure Is Not Validated Before Access

```python
charge_id = response.json()["charge_id"]
```

If the gateway returns a 200 with a malformed or unexpected JSON body (missing `charge_id`, non-JSON content type), this raises a `KeyError` or `JSONDecodeError` that bypasses the retry logic and propagates as an unhandled exception. The response schema should be validated before accessing fields.

---

### 🟡 Minor: `CHARGE_ENDPOINT` Hardcoded as Class Attribute

`"https://api.payments.internal/v1/charges"` is hardcoded. This should be injected via configuration or environment variable to support environment-specific deployments and to avoid requiring code changes for endpoint rotation.

---

## `tests/test_order_processor.py`

### 🟡 Significant: No Test for Payment Failure Inside Lock — Lock Release Not Verified on Exception

There is no test that verifies the lock is released when `payment.charge()` raises an exception. The `finally` block in `_acquire_lock` should handle this, but it is not tested. Given the critical nature of lock release for system liveness, this path must be explicitly covered.

---

### 🟡 Significant: No Test for the TOCTOU / Lock Expiry Scenario

The race condition between lock TTL expiry and long-running payment operations is not tested or even acknowledged in the test suite.

---

### 🟡 Minor: `test_lock_release_uses_lua_script` Does Not Assert Token Correctness

```python
processor._release_script.assert_called_once()
```

This only asserts the script was called. It does not verify that the token passed to the release script matches the token that was set during acquisition. The token-verification property — the stated security guarantee of the Lua script — is not actually tested.

**Decision: Request Changes**

**Summary:** This PR contains multiple critical correctness and financial safety issues that must be resolved before merge. The most severe is placing the external payment charge call inside a database transaction, which is architecturally unsound — it holds DB connections open for the full payment retry window and provides no actual atomicity guarantee between the charge and its record, directly contradicting the stated design goal. The absence of an idempotency key on payment gateway requests means the retry logic is a duplicate charge vector. The Redis lock TTL is shorter than the maximum possible critical section duration, invalidating the lock's safety guarantee under slow payment retries.
