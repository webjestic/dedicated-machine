Decision: Request Changes

Summary: The core locking and validation primitives are well-constructed — atomic `SET NX EX`, token-verified Lua release, Pydantic pattern enforcement, and idempotency gating are all sound. However, there are several distributed-systems correctness flaws, one critical security gap, and a few reliability issues that must be addressed before this can be approved for a payment-critical path.

---

## Critical Issues

### 1. Payment charge lives outside the database transaction — at-most-once guarantee is broken

This is the most serious correctness flaw. The comment says *"Charge and record atomically — crash between charge and record cannot produce an unrecorded charge"*, but that claim is false.

```python
# CURRENT — payment.charge() is INSIDE db.transaction()
with self.db.transaction():
    charge_id = self.payment.charge(...)        # external HTTP call
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

`payment.charge()` is an external HTTP call to a payment gateway. Database transactions do not span external network calls. If the process crashes after `payment.charge()` returns a `charge_id` but before `db.record_charge()` commits, the customer has been charged and the system has no record of it. The comment creates a false sense of safety that is worse than silence.

**Required fix:** Move the payment call *outside and before* the DB transaction. Record the returned `charge_id` first, then open the transaction to persist it. Accept that a crash between charge and record is a reconciliation problem, not a transactional one — and implement a reconciliation/recovery path or an outbox pattern.

```python
# Correct structure
charge_id = self.payment.charge(token=req.payment_token, amount_cents=req.amount_cents)
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

And document explicitly that a crash between those two steps requires reconciliation against the payment gateway.

### 2. Timeout on payment retry can produce duplicate charges

```python
except requests.exceptions.Timeout:
    last_error = "gateway timeout"
    # ... retries
```

A timeout means the request *may have succeeded* at the gateway — the response was simply lost in transit. Retrying unconditionally on timeout without an idempotency key sent to the gateway will cause duplicate charges. This is a well-known failure mode in payment systems.

**Required fix:** Generate a stable idempotency key (e.g., derived from `order_id`) and pass it as a header on every attempt to the payment gateway. Most payment APIs (Stripe, Braintree, etc.) support `Idempotency-Key`. Without this, the retry logic actively makes the system less safe.

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},
    headers={"Idempotency-Key": idempotency_key},  # stable per order
    timeout=self.REQUEST_TIMEOUT,
)
```

### 3. Redis lock TTL does not account for payment retry duration

```python
ORDER_LOCK_TTL = 60   # seconds
# PaymentClient: MAX_RETRIES=3, RETRY_BACKOFF=5s, REQUEST_TIMEOUT=30s
# Worst case: 3 * (30 + 5) = 105 seconds
```

The worst-case payment client execution time (3 attempts × (30s timeout + 5s backoff)) is **105 seconds**, which exceeds the 60-second lock TTL by 45 seconds. The lock will expire while payment processing is still in flight, allowing a second worker to acquire the lock, pass the idempotency check (charge not yet recorded), and issue a duplicate charge. The lock TTL must be greater than the maximum possible execution time of everything inside the lock.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at least 120–150 seconds, or reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` in the payment client, or implement lock extension (Redlock-style `EXPIRE` refresh). All three values must be documented together as a coupled constraint.

## Security Issue

### 4. `payment_token` is logged at INFO level

```python
logger.info(
    "Order processed order_id=%s charge_id=%s",
    req.order_id, charge_id,
)
```

This specific line is fine, but `payment_token` flows through `OrderRequest` and is passed directly to `payment.charge(token=req.payment_token, ...)`. Confirm that no log line anywhere in the call path emits `payment_token` or the raw `OrderRequest` object. The Pydantic model should mark `payment_token` as a `SecretStr` to prevent accidental serialization into logs or error messages.

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

Then call `.get_secret_value()` only at the point of use.

## Reliability and Design Issues

### 5. `LockError` is silently swallowed — no observability

```python
except LockError:
    return OrderResult(processed=False, message="Order currently being processed...")
```

Lock contention on a payment order is operationally significant. This should be logged at WARNING or ERROR before returning, otherwise sustained lock contention (e.g., a stuck worker holding the lock until TTL) is invisible in production.

---

### 6. Unhandled exceptions inside the lock context leak without structured handling

If `self.payment.charge()` raises `PaymentError` or any unexpected exception, it propagates out of `with self._acquire_lock(...)` — the `finally` block will correctly release the lock, which is good. However, the exception is completely unhandled at the `process()` level and will surface as an unstructured 500 to the caller. Add a top-level `except Exception` with logging and a structured `OrderResult` return.

---

### 7. Test for lock release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that a private attribute on the processor was called — it is coupled to the implementation detail of how the script object is stored. It also does not test the critical case: that the lock is released even when an exception is raised inside the lock body. Add a test that injects a failure inside the lock and asserts the release script is still called.

---

### 8. `customer_id` has no pattern validation

`order_id` correctly enforces `^[a-zA-Z0-9_\-]+$`. `customer_id` has only length bounds. If `customer_id` is used in database queries or log aggregation downstream, it should have equivalent pattern enforcement.
