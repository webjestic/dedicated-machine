Decision: Request Changes

Summary: The core locking and idempotency primitives are well-structured — atomic SET NX EX, token-verified Lua release, and Pydantic input validation are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the external payment call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues around timeout handling, error surface exposure, and test coverage gaps that must be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge is inside the database transaction — this does not make it atomic

The PR comment says *"crash between charge and record cannot produce an unrecorded charge"* but that claim is false. The `payment.charge()` call is a **network call to an external system** placed inside `self.db.transaction()`. Database transactions provide atomicity over database operations only. They have no rollback authority over an already-submitted external charge.

```python
# Current — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external side-effect — NOT rolled back on DB failure
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The failure modes this creates:

| Failure point | Outcome |
|---|---|
| DB write fails after `payment.charge()` succeeds | Customer charged, no record — silent double-charge risk on retry |
| Process crash after charge, before commit | Same as above |
| Transaction rollback for any reason | Charge already submitted, cannot be undone |

**Required fix:** Charge first, record the charge ID durably before doing anything else, then complete the order. If the record write fails, you have a charge ID to reconcile against. Consider a two-phase approach:

```python
# Safer pattern
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
)
# Record the charge ID immediately and independently
# so any subsequent failure is recoverable via reconciliation
with self.db.transaction():
    self.db.record_charge(
        req.order_id, req.customer_id, charge_id, req.amount_cents
    )
    self.db.mark_order_complete(req.order_id)
```

This does not fully eliminate the gap — nothing can make an external call and a DB write truly atomic — but it makes the failure recoverable rather than silent. The idempotency check on re-entry will catch the recorded charge. You should also add a dead-letter / reconciliation path for the case where `record_charge` itself fails after a successful charge.

### 2. Lock TTL is shorter than the total possible operation time

`ORDER_LOCK_TTL = 60` seconds, but:

- `PaymentClient.REQUEST_TIMEOUT = 30` seconds per attempt
- `PaymentClient.MAX_RETRIES = 3`
- `PaymentClient.RETRY_BACKOFF = 5` seconds between retries

Worst-case payment client runtime: `(30 × 3) + (5 × 2) = 100 seconds`

The lock expires at 60 seconds. A second worker can acquire the lock at second 61, pass the idempotency check (no charge recorded yet because the first worker is still waiting on a retry), and submit a duplicate charge. This directly defeats the stated duplicate-charge prevention guarantee.

**Required fix:** Either raise `ORDER_LOCK_TTL` above the maximum possible payment client runtime with margin (≥ 120s), or reduce `MAX_RETRIES`/`REQUEST_TIMEOUT` so the math closes. Document the invariant explicitly with a constant or assertion so future changes to either value don't silently reopen this window.

```python
# Make the invariant explicit and machine-checked
MAX_PAYMENT_DURATION = (
    PaymentClient.REQUEST_TIMEOUT * PaymentClient.MAX_RETRIES
    + PaymentClient.RETRY_BACKOFF * (PaymentClient.MAX_RETRIES - 1)
)
assert ORDER_LOCK_TTL > MAX_PAYMENT_DURATION, (
    f"Lock TTL {ORDER_LOCK_TTL}s must exceed max payment duration {MAX_PAYMENT_DURATION}s"
)
```

## Significant Issues

### 3. Timeout on gateway response does not guarantee the charge was not submitted

In `PaymentClient.charge()`, a `requests.exceptions.Timeout` is caught and retried. A timeout means the client did not receive a response — it does **not** mean the charge was not processed by the gateway. Retrying after a timeout can submit the charge a second time if the gateway processed the first request but the response was lost in transit.

The payment client needs to either:
- Use an idempotency key on every charge request (pass `order_id` or a deterministic UUID derived from it as a gateway-level idempotency key), or
- Treat timeouts as non-retryable and surface them for manual reconciliation

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # required
    },
    timeout=self.REQUEST_TIMEOUT,
)
```

### 4. `payment_token` is logged at INFO level

```python
logger.info("Order processed order_id=%s charge_id=%s", req.order_id, charge_id)
```

This line is fine, but `payment_token` is present in the `OrderRequest` object and could easily be swept into a debug log, a serialized exception, or a future structured logging call. Payment tokens are PCI-relevant credentials. Add an explicit `__repr__`/`__str__` override or use a `SecretStr` field type to prevent accidental exposure:

```python
from pydantic import SecretStr

class OrderRequest(BaseModel):
    payment_token: SecretStr = Field(..., min_length=1, max_length=256)
```

### 5. Unhandled exceptions from `payment.charge()` and `db` operations propagate uncaught

`process()` catches `LockError` and `ValidationError` but any other exception — `PaymentError`, database connection errors, unexpected gateway responses — propagates to the caller as an unhandled exception. For a service boundary this means the caller gets a raw stack trace that may contain internal details. Add a top-level catch with structured logging and a safe return:

```python
except PaymentError as e:
    logger.error("Payment failed order_id=%s: %s", req.order_id, e)
    return OrderResult(processed=False, message="Payment processing failed")
except Exception as e:
    logger.exception("Unexpected error processing order_id=%s", req.order_id)
    return OrderResult(processed=False, message="Internal processing error")
```

## Minor Issues

### 6. Test for lock TTL expiry / lock-lost scenario is absent

There is no test covering the case where the Redis lock expires mid-processing (the worker continues executing after TTL). This is the exact scenario that produces a double-charge. A test should verify the system's behavior — even if the current behavior is "undefined, needs a watchdog" — so the gap is visible.

### 7. `test_lock_release_uses_lua_script` asserts on internal implementation detail

```python
processor._release_script.assert_called_once()
```

This test is coupled to the private `_release_script` attribute name. It will break on any internal refactor without the behavior changing. Test the observable contract instead: that the lock key is absent in Redis after processing, using a fake Redis or `fakeredis`.

### 8. No health check or Redis connectivity validation at startup

If Redis is unavailable at startup, the first `process()` call will raise an unhandled connection error. A startup connectivity check or a clear `RedisUnavailableError` with a safe `OrderResult` return would make the failure mode explicit.
