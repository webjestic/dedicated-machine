Decision: Request Changes

Summary: The architecture is sound and the security fundamentals are well-considered — atomic lock acquisition, token-verified Lua release, Pydantic validation, and idempotency checks are all correctly implemented. However, there is a critical atomicity flaw at the boundary between the payment gateway call and the database transaction that breaks the at-most-once guarantee the PR claims to provide, plus several secondary issues that must be addressed before this handles real payment data.

---

## Critical Issues

### 1. Payment charge occurs inside the database transaction — this is the core design flaw

```python
# Current code — WRONG
with self.db.transaction():
    charge_id = self.payment.charge(...)   # external network call inside DB tx
    self.db.record_charge(...)
    self.db.mark_order_complete(...)
```

The comment says *"crash between charge and record cannot produce an unrecorded charge"* — but this is **false**. The sequence of failure modes is:

| Failure point | DB transaction state | Payment state | Outcome |
|---|---|---|---|
| `payment.charge()` raises after money moves but before returning `charge_id` | Rolled back | Charged | **Silent money loss — charge exists, no record** |
| `payment.charge()` times out (ambiguous) | Rolled back | Unknown | **Undetectable duplicate on retry** |
| Process crashes after `charge()` returns, before `record_charge()` commits | Rolled back | Charged | **Same as above** |

Wrapping an external network call in a database transaction provides no atomicity guarantee across the two systems. The database transaction only protects the DB operations from each other.

**Required fix:** Use the outbox pattern or a two-phase approach:

```python
# Correct approach
with self.db.transaction():
    # Reserve the order slot first — idempotency anchor
    self.db.reserve_order(req.order_id, req.customer_id, req.amount_cents)

# Payment happens OUTSIDE the transaction
charge_id = self.payment.charge(
    token=req.payment_token,
    amount_cents=req.amount_cents,
    idempotency_key=req.order_id,   # critical — see issue #2
)

# Record the result in a second transaction
with self.db.transaction():
    self.db.record_charge(req.order_id, req.customer_id, charge_id, req.amount_cents)
    self.db.mark_order_complete(req.order_id)
```

The reservation step in the first transaction is what prevents a second worker (after lock expiry) from re-entering. The idempotency key on the payment call is what makes a retry safe on timeout.

### 2. Payment client does not send an idempotency key

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={"token": token, "amount": amount_cents},  # no idempotency key
    timeout=self.REQUEST_TIMEOUT,
)
```

On a timeout, `PaymentError` is raised. The caller (inside the Redis lock) will then propagate the exception, the lock will release, and a retry of the entire order will issue a **second charge request with no way for the gateway to deduplicate it**. This is a double-charge vector independent of the locking logic.

**Required fix:**

```python
response = requests.post(
    self.CHARGE_ENDPOINT,
    json={
        "token": token,
        "amount": amount_cents,
        "idempotency_key": idempotency_key,  # pass order_id through
    },
    headers={"Idempotency-Key": idempotency_key},  # also as header per most gateway specs
    timeout=self.REQUEST_TIMEOUT,
)
```

The `charge()` method signature must accept `idempotency_key: str` as a required parameter.

### 3. Lock TTL is dangerously short given the retry logic inside the lock

```
ORDER_LOCK_TTL = 60 seconds
PaymentClient.REQUEST_TIMEOUT = 30 seconds
PaymentClient.MAX_RETRIES = 3
PaymentClient.RETRY_BACKOFF = 5 seconds
```

Worst-case payment time inside the lock: `3 × 30s timeout + 2 × 5s backoff = 100 seconds`. The lock expires at 60 seconds, meaning a second worker can acquire the lock and begin processing while the first worker is still in `payment.charge()`. The distributed lock provides no protection in this scenario.

**Required fix:** Either raise `ORDER_LOCK_TTL` to at minimum `120` seconds with a comment justifying the calculation, or reduce the payment client's retry budget. Document the relationship explicitly so they cannot drift independently.

```python
# Maximum time inside the lock:
# MAX_RETRIES * REQUEST_TIMEOUT + (MAX_RETRIES - 1) * RETRY_BACKOFF
# = 3 * 30 + 2 * 5 = 100s  →  TTL must exceed this with margin
ORDER_LOCK_TTL = 120
```

## Secondary Issues

### 4. `payment_token` is logged indirectly — verify it never reaches log output

The `payment_token` field passes through `ValidationError` formatting. If Pydantic ever includes field values in its error messages (it can, depending on version and configuration), the token will appear in logs.

```python
except ValidationError as e:
    logger.warning("Invalid order request: %s", e)  # may serialize field values
```

**Required fix:**

```python
except ValidationError as e:
    logger.warning("Invalid order request order_id=%s field_count=%d", order_id, len(e.errors()))
    return OrderResult(processed=False, message="Invalid order parameters")
```

### 5. `PaymentError` is not caught in `process()`

If `payment.charge()` raises `PaymentError` (permanent decline or exhausted retries), the exception propagates out of `process()` entirely — uncaught, unlogged at the processor level, and not converted to an `OrderResult`. The caller receives a raw exception rather than the structured return type the method signature promises.

```python
# Add to the except chain in process()
except PaymentError as e:
    logger.error("Payment failed order_id=%s error=%s", req.order_id, e)
    return OrderResult(
        processed=False,
        order_id=req.order_id,
        message="Payment failed",
    )
```

### 6. Test for Lua script release is testing mock internals, not behavior

```python
def test_lock_release_uses_lua_script(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    redis_client.delete.assert_not_called()
    processor._release_script.assert_called_once()
```

This test asserts that `_release_script` was called, but `_release_script` is a mock. It does not verify that the script receives the correct `keys` and `args` — specifically that the token passed to release matches the token used to acquire. The test would pass even if the wrong token were passed, which is the entire security property of the Lua script.

```python
def test_lock_release_passes_correct_token(processor, redis_client):
    processor.process("order-001", "cust-001", 5000, "tok_visa")
    call_args = processor._release_script.call_args
    keys = call_args.kwargs["keys"]
    args = call_args.kwargs["args"]
    assert keys == ["order:lock:order-001"]
    assert len(args) == 1
    assert len(args[0]) == 36  # UUID format
```

### 7. No test coverage for `PaymentError` propagation or timeout behavior

The test suite has no case for payment failure, which is now doubly important given issue #5. Add:

- `test_payment_permanent_failure_returns_result` — `payment.charge` raises `PaymentError`
- `test_payment_timeout_does_not_double_charge` — validates idempotency key is forwarded (once issue #2 is fixed)

---

## What Is Done Well

The `SET NX EX` acquisition is correct and atomic. The Lua release script correctly handles the token comparison and is registered once at construction rather than re-evaluated per call. The `order_id` pattern validation blocks path traversal as demonstrated by the test. The retry backoff in `_acquire_lock` is reasonable. The structured logging is consistent. The separation of `LockError` from payment errors is the right design — these are distinct failure modes that callers should handle differently.
