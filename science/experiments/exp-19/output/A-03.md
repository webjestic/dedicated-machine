**Decision: Request Changes**

**Summary:** The heartbeat thread reduces the probability of lock expiry but does not eliminate it — a full process pause (GC stop-the-world, OOM freeze, kernel scheduler preemption, container CPU throttle) can suspend all threads simultaneously, including the heartbeat, for longer than the remaining TTL. When the process resumes, the lock is gone, but `reserve_item` continues executing and writes to the database with no awareness that it has lost exclusive access. There is no fencing token or optimistic lock check at the database write layer, so `db.update_stock` and `db.create_order` can execute concurrently with another process that legitimately acquired the lock after expiry. The test suite validates the happy path and basic contention but does not test the process-pause scenario, and the heartbeat mock in the fixture silently hides the exact code path that would fail in production.

## Detailed Findings

### 1. Critical — No Fencing Token at the Write Layer (the core issue)

This is the architectural flaw that makes the lock insufficient for its stated purpose.

```python
# What the code does:
self.db.update_stock(item_id, new_stock)   # unconditional write
self.db.create_order(order_id, item_id, quantity)  # unconditional write
```

The Redis lock is advisory. It coordinates access only while all participants are alive and communicating. It does not prevent a write from executing after the lock has expired. The scenario:

1. Worker A acquires lock, heartbeat starts.
2. All threads in Worker A's process are paused for 35 seconds (container throttle, GC, kernel).
3. Lock TTL (30s) expires. Heartbeat never fired because it was also paused.
4. Worker B acquires the lock, reads stock = 10, writes stock = 9, commits.
5. Worker A resumes. It does not re-check lock ownership before writing.
6. Worker A reads stale `current_stock` (already captured before the pause as 10), writes stock = 9 again, creates a second order for the same unit.

The check in the `finally` block only governs whether the lock key is deleted. It does not gate the writes that already happened on lines above it.

**Required fix — Option A (fencing token with conditional DB write):**

```python
# The lock token becomes a fencing token passed to the DB layer.
# The DB enforces: only apply this write if no write with a higher
# fence has been applied since you read the stock.

with self._acquire_lock(lock_key) as token:
    current_stock, stock_version = self.db.get_stock_versioned(item_id)
    # ... availability check ...
    # Write is conditional on the version not having changed.
    # If another worker wrote in the interim, this raises StaleVersionError.
    self.db.update_stock_conditional(
        item_id,
        new_stock,
        expected_version=stock_version   # optimistic lock at DB level
    )
```

**Required fix — Option B (database-level atomic decrement with floor check):**

```python
# Atomically decrement only if stock >= quantity.
# Returns updated row or raises InsufficientStockError.
# The Redis lock becomes a performance optimization (reduce DB contention),
# not the sole correctness mechanism.
rows_updated = self.db.decrement_stock_if_available(item_id, quantity)
if rows_updated == 0:
    return ReservationResult(success=False, message="Insufficient stock")
```

The SQL behind Option B:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id
  AND stock >= :quantity
RETURNING stock;
```

Either option makes the write safe in isolation. Option B is simpler and removes the dependency on Redis for correctness (Redis then only throttles request rate, which is a legitimate use).

### 2. High — Heartbeat State Is Instance-Level, Not Lock-Scoped

```python
self._heartbeat_active = False   # instance variable, not per-lock
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or long-lived object. If two concurrent calls to `reserve_item` overlap (which is the entire point of a flash sale service), the second call's `_start_heartbeat` overwrites `self._heartbeat_active` and `self._heartbeat_thread`. When the first call's `finally` block runs `_stop_heartbeat`, it sets `self._heartbeat_active = False`, killing the second call's heartbeat. The second lock now has no renewal and will expire.

**Required fix:**

```python
@dataclass
class _HeartbeatHandle:
    active: bool
    thread: threading.Thread

def _start_heartbeat(self, lock_key: str, token: str) -> _HeartbeatHandle:
    handle = _HeartbeatHandle(active=True, thread=None)

    def _beat() -> None:
        while handle.active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if handle.active:
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    handle.thread = threading.Thread(target=_beat, daemon=True)
    handle.thread.start()
    return handle

# In _acquire_lock:
handle = self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    handle.active = False
```

### 3. High — Race Condition in Heartbeat's Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # non-atomic with the GET above
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. The heartbeat then extends a lock it no longer owns. This is a TOCTOU race. The fix is a Lua script that makes the check-and-extend atomic:

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init:
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In the heartbeat loop:
self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same race exists in the lock release in `_acquire_lock`'s `finally` block and should receive the same treatment:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

### 5. Medium — `db.update_stock` and `db.create_order` Are Not in a Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock has already been decremented. There is no rollback. The item is effectively lost. These two writes must be atomic at the database level:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. Low — Heartbeat Thread Leak on Unhandled Exception

If `_start_heartbeat` is called and then an exception propagates before `_stop_heartbeat` runs (e.g., during the `yield`), the daemon thread will continue running until the process exits. With the instance-level state bug in Finding 2, this also corrupts state for subsequent calls. The per-lock handle approach in Finding 2's fix resolves this when combined with the `finally` block.

### 7. Test Suite Gap — The Critical Scenario Is Not Tested

The test `test_lock_not_released_if_expired` checks that `redis.delete` is not called when the token mismatches. It does not verify that `db.update_stock` is also not called in that scenario — which is the actual correctness requirement. The test passes today because it cannot fail: the mock bypasses the real lock path entirely via `svc._start_heartbeat = MagicMock()`.
