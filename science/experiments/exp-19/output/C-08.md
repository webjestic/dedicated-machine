Decision: Request Changes

Summary: The core locking logic contains a critical TOCTOU race condition in the lock release path — the `GET` + `DELETE` sequence is non-atomic, meaning another process could acquire the lock between the check and the delete, causing an illegitimate release. Additionally, the heartbeat implementation has a thread-safety flaw where a single `InventoryService` instance cannot safely handle concurrent calls, and input validation is claimed but not actually enforced. These issues are significant enough to block merge in a high-concurrency flash sale context.

---

## Detailed Findings

### 🔴 Critical — Non-Atomic Lock Release (TOCTOU Race)

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock. You would then delete a lock you no longer own. This is a well-known failure mode for Redis distributed locks.

**Required fix:** Use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at __init__
self._release_lock_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
released = self._release_lock_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU issue exists in the heartbeat's `GET` + `expire` sequence and must be fixed there too:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 🔴 Critical — Heartbeat State Is Instance-Level, Not Call-Level

**File:** `services/inventory.py`, `_heartbeat_active` and `_heartbeat_thread`

```python
# These are shared across ALL concurrent calls on the same instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance, the second call's `_stop_heartbeat()` will set `_heartbeat_active = False` and kill the first call's heartbeat thread prematurely. In a flash sale scenario with a shared service instance (e.g., injected as a singleton), this is a near-certain failure path.

**Required fix:** Scope heartbeat state to the individual lock acquisition, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... retry logic unchanged ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... atomic release via Lua script ...

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            released = self._extend_lock_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            if not released:
                logger.warning("Heartbeat: lock lost key=%s", lock_key)
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note also that `_beat` currently calls `time.sleep()` first, then checks `_heartbeat_active`. If the lock is released and the thread is stopped, it still sleeps the full `HEARTBEAT_INTERVAL` before noticing. Using `threading.Event.wait()` makes the stop responsive.

### 🟡 High — Input Validation Is Claimed but Not Enforced

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing prevents item_id="" or quantity=-5 or quantity=0
```

A caller passing `quantity=0` or `quantity=-1` would silently succeed or produce nonsensical results. `item_id` being an empty string would acquire a lock on `inventory:lock:` and proceed against the database.

**Required fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id and req.quantity below
```

The `item_id` pattern constraint is particularly important — without it, a crafted `item_id` containing `:` could manipulate the Redis key namespace.

### 🟡 High — Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no order record. This is a partial write and will cause inventory drift over time.

**Required fix:** Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should require transaction support and this should be documented.

### 🟡 Medium — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # ← stop never runs
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the concurrency behavior, the TOCTOU release path, and the heartbeat logic are entirely untested. `test_lock_not_released_if_expired` asserts `delete` is not called, but because the token in the fixture is hardcoded to `b"test-token"` while the service generates a fresh `uuid4()`, this test passes for the wrong reason — it would pass even if the ownership check were removed entirely.

**Required fix:** Add integration-style tests using `fakeredis` to exercise the actual lock lifecycle:

```python
import fakeredis
```

### 🟢 Low — Minor Issues

- **`LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10`:** The first heartbeat fires at t=10s, which is fine, but the relationship between these constants is implicit. A comment or assertion (`assert HEARTBEAT_INTERVAL < LOCK_TTL / 2`) would prevent a future misconfiguration from silently causing lock expiry before the first renewal.
- **`daemon=True` on heartbeat thread:** Correct for preventing process hang on shutdown, but worth a comment explaining the intent since daemon threads can be surprising.
- **`retry_attempts=3` with backoff of 0.1, 0.2, 0.4s:** Total max wait is 700ms. Under genuine flash sale load this may be too aggressive. Consider making this configurable at construction time rather than per-call.
