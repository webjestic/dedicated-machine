**Decision: Request Changes**

**Summary:** The heartbeat mitigates *slow* operations but does not close the fundamental race this architecture is designed to prevent. The database writes at lines `self.db.update_stock` / `self.db.create_order` carry no fencing token or optimistic lock version check, meaning a process-wide pause (GC stop-the-world, VM live migration, kernel preemption) that outlasts the TTL — even with the heartbeat — leaves the storage layer with no mechanism to reject a stale write. Beyond that critical gap, there are several compounding issues that together make this unsafe for production at any concurrency level.

---

## Critical: No Fencing at the Storage Layer

This is the issue I look for before anything else.

```python
# The lock is held here. The heartbeat is running.
# A full process pause begins — all threads, including the heartbeat thread.

current_stock = self.db.get_stock(item_id)   # reads 5
# ... pause lasts 31 seconds ...
# Lock TTL expires. Another worker acquires the lock, reads 5, writes 4.
# This process resumes. Heartbeat has not fired. Lock is gone.
# Token check in the finally block will catch the *release*, but not this:

self.db.update_stock(item_id, new_stock)     # also writes 4 — stock is now wrong
self.db.create_order(order_id, item_id, quantity)  # phantom order created
```

The heartbeat thread is itself a Python thread. Python threads are subject to the same process-wide pauses as the main thread — OS scheduler preemption, hypervisor migration, `SIGSTOP`. A pause that exceeds `LOCK_TTL` (30 seconds) silences the heartbeat for the same duration. When the process resumes, the heartbeat may re-extend a lock that has already been acquired by another worker. The token check in `_beat` has a TOCTOU window: `GET` and `EXPIRE` are two separate commands, not atomic.

The lock protects the *read-modify* sequence in Redis. It does not protect the *write* in the database. The database has no way to reject the write from the zombie process.

**Required fix:** The database write must carry a condition the storage layer enforces atomically.

```python
# Option A — optimistic locking with a version counter
# db.update_stock must execute:
#   UPDATE inventory SET stock = %s, version = version + 1
#   WHERE item_id = %s AND version = %s AND stock = %s
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_version=stock_row.version,
    expected_stock=current_stock,
)
if rows_affected == 0:
    # Lost the race — another writer committed first
    raise StaleDataError(f"Concurrent modification detected for {item_id}")
```

```python
# Option B — monotonically increasing fencing token passed through to the DB write
# The lock server (or a separate sequence) issues a token that only increases.
# The DB rejects any write whose token is less than the last committed token.
# Redis INCR on a separate key at lock acquisition time gives you this cheaply.
```

Without one of these, the lock is advisory only. It reduces the probability of a collision under normal conditions. It does not prevent it.

## High: Heartbeat Has a TOCTOU Race

```python
current = self.redis.get(lock_key)          # step 1: GET
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2: EXPIRE
```

Between step 1 and step 2, the lock can expire and be acquired by another worker. The heartbeat then extends a lock it no longer owns. This must be a single atomic operation.

```python
# Correct approach — Lua script, atomic GET+EXPIRE
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at __init__
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In _beat:
result = self._heartbeat_script(
    keys=[lock_key],
    args=[token, self.LOCK_TTL]
)
if not result:
    logger.warning("Heartbeat found lock already lost key=%s", lock_key)
    self._heartbeat_active = False  # stop beating; we no longer own it
```

The same atomic script pattern applies to lock release in the `finally` block.

## High: Instance-Level Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not scoped to a single lock acquisition. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the normal deployment pattern for a service object — two callers share `_heartbeat_active`. The first caller's `_stop_heartbeat()` sets it to `False` and kills the second caller's heartbeat thread.

```python
# Fix: scope heartbeat state to the lock context, not the instance
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self._heartbeat_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            if not result:
                logger.warning("Lost lock during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return

    # ... acquire, start thread with _beat, yield, finally heartbeat_active.clear()
```

## Medium: `quantity` Has No Lower Bound Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the stock check (`current_stock < 0` is false), creates an order for zero units, and decrements stock by zero — a no-op order that still consumes a lock cycle and writes a record. `quantity=-5` passes the stock check and *increases* stock, which is an inventory injection vulnerability.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)  # enforce at the model boundary
```

The PR description claims input validation via Pydantic, but `reserve_item` accepts raw primitives. The `ReservationResult` model is output-only. There is no input model.

---

## Medium: `update_stock` and `create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock has already been decremented. The item is gone from inventory with no order to show for it. These two writes must be wrapped in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Low: `item_id` Is Used Directly in the Lock Key Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:` or whitespace, the key namespace can be manipulated. A value like `item_001:lock:other_item` produces `inventory:lock:item_001:lock:other_item`, which collides with a different lock key pattern. Validate and sanitize `item_id` before constructing Redis keys, or use a hash.

## Low: Test Suite Mocks Out the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # stop never runs

```

The heartbeat is the primary safety mechanism being introduced. Mocking it out means the tests provide no coverage of the behavior the PR is selling. `test_lock_not_released_if_expired` is the most important test in the file and it passes only because `redis_client.get` returns a hardcoded mismatched token — it does not exercise actual expiry, actual TTL behavior, or the TOCTOU window in the heartbeat. These tests verify the happy path of the business logic. They do not verify the distributed systems guarantees.
