Decision: Request Changes

Summary: The heartbeat thread addresses clock-drift-induced expiry under *slowness*, but it cannot protect against the scenario where the entire process is paused — GC stop-the-world, VM live migration, kernel preemption, OOM stall — because the heartbeat thread is paused alongside every other thread in the process. When execution resumes, the lock may have expired and been acquired by another worker; the database write then executes without any awareness that lock ownership was lost, producing a phantom write. There is no fencing token or optimistic lock check at the database write layer, so the Redis lock does not actually protect the critical section. There are also secondary issues in the concurrency model, test fidelity, and operational safety that must be addressed before this is production-ready.

---

## Critical: The Lock Does Not Protect the Write

### 1. No fencing token or database-level guard

This is the central flaw. The PR author clearly read the right literature — the heartbeat is a reasonable mitigation for *gradual* slowness — but the heartbeat assumption breaks completely under a full process pause.

**The failure sequence:**

```
Worker A                          Redis                    Database
--------                          -----                    --------
acquires lock (token=abc)
heartbeat running...
                                                           
[PROCESS PAUSED — GC / VM migration / kernel preemption]
                                                           
                                  TTL expires
                                  
Worker B acquires lock (token=xyz)
Worker B reads stock = 10
Worker B writes stock = 9
Worker B releases lock

[WORKER A RESUMES]

heartbeat fires:
  GET lock → returns "xyz"        ← token mismatch
  heartbeat does NOT extend        ← correct behavior
  
  BUT: the write path does not check this.
  
Worker A: db.update_stock(item_id, 8)   ← executes anyway
Worker A: db.create_order(...)          ← executes anyway
```

Stock is now 8. Two orders exist. One reservation was made without a valid lock. The heartbeat correctly detected the loss, but the write path was never told.

The heartbeat thread and the application thread share no synchronization primitive. `_heartbeat_active` is a plain Python `bool`. There is no `threading.Event`, no `threading.Lock`, no atomic flag that the write path consults before executing. Even if the heartbeat sets `_heartbeat_active = False` after detecting token mismatch — which it does not currently do — the write path would not check it.

**What is required:** A fencing token passed through to the database layer, with the database enforcing monotonicity:

```python
# Conceptual — requires DB schema support
# e.g., a `lock_version` column or a conditional UPDATE

def update_stock(self, item_id, new_stock, fence: int):
    rows_affected = self.db.execute(
        "UPDATE inventory SET stock = %s "
        "WHERE item_id = %s AND lock_version = %s",
        (new_stock, item_id, fence)
    )
    if rows_affected == 0:
        raise StaleWriteError("Write rejected: lock version mismatch")
```

Or, if the database supports it, an optimistic lock via a `version` or `updated_at` column checked in the `WHERE` clause of every write. The lock version read at the start of the critical section must be the same version present at write time, enforced by the database atomically, not by the application.

Without this, the Redis lock is advisory at best.

### 2. Heartbeat does not signal write path on token loss

Even setting aside the fencing token requirement, the heartbeat's token check is purely observational. When it detects that the token no longer matches, it logs and stops extending — but it does not raise, it does not set a shared abort flag, and the write path continues executing.

```python
# Current heartbeat — detects loss, does nothing actionable
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
    logger.debug("Lock TTL extended key=%s", lock_key)
# If token mismatch: silently falls through, loop continues
```

A minimal improvement (still insufficient without a DB-level fence, but at least internally consistent):

```python
def _beat(self) -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if not self._heartbeat_active:
            break
        current = self.redis.get(lock_key)
        if not current or current.decode() != token:
            logger.error(
                "Lock lost during operation key=%s — setting abort flag", lock_key
            )
            self._lock_lost.set()   # threading.Event
            self._heartbeat_active = False
            return
        self.redis.expire(lock_key, self.LOCK_TTL)
```

And the write path must check `self._lock_lost` before each database call. But again: this is a best-effort mitigation. A paused process cannot check anything. The database fence is the only reliable guard.

### 3. `_heartbeat_active` is not thread-safe

`_heartbeat_active` is a plain `bool` read and written from two threads with no memory barrier. In CPython this is incidentally safe due to the GIL, but it is not guaranteed by the language memory model, it is not safe in PyPy or other runtimes, and it signals to every future maintainer that thread safety was not considered. Use `threading.Event`.

```python
self._heartbeat_active = threading.Event()
# set: self._heartbeat_active.set()
# clear: self._heartbeat_active.clear()
# check: self._heartbeat_active.is_set()
# sleep with early exit: self._heartbeat_active.wait(timeout=HEARTBEAT_INTERVAL)
```

The `wait(timeout=...)` pattern also eliminates the race between `time.sleep` and `_heartbeat_active = False` where the heartbeat can sleep through a stop signal and fire one extra time after the lock is released.

### 4. Instance-level heartbeat state prevents concurrent use

`_heartbeat_active` and `_heartbeat_thread` are instance variables, not scoped to the lock acquisition. If `reserve_item` is ever called concurrently on the same `InventoryService` instance — which is the entire point of a service object — two concurrent reservations will share and corrupt each other's heartbeat state. The second acquisition will overwrite `_heartbeat_active` and `_heartbeat_thread` from the first.

```python
# Thread 1 acquires lock for item_001
self._heartbeat_active = True
self._heartbeat_thread = Thread(...)

# Thread 2 acquires lock for item_002 — same instance
self._heartbeat_active = True        # fine, same value
self._heartbeat_thread = Thread(...) # Thread 1's reference is now lost

# Thread 1 exits critical section
self._stop_heartbeat()               # sets _heartbeat_active = False
                                     # Thread 2's heartbeat is now also stopped
```

Heartbeat state must be scoped to the lock acquisition, not the instance. Pass it as local state into the context manager and the thread closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

## Secondary Issues

### 5. `update_stock` and `create_order` are not atomic

Even within a valid lock window, if `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order. There is no transaction boundary.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # if this raises: stock is gone
```

These two writes must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. `quantity` has no validated lower bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 10 is False, so it proceeds), creates an order for zero units, and decrements stock by zero. `quantity=-5` would *increase* stock. This must be validated before the lock is acquired.

```python
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

### 7. `item_id` is used directly in a Redis key without sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains `:`, `*`, `?`, `[`, or newline characters, it can produce unexpected key patterns or interfere with Redis key-space operations. Validate or encode `item_id` before interpolation.

### 8. Tests mock out the exact behavior under test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat is never tested
    svc._stop_heartbeat = MagicMock()    # ← stop is never tested
    return svc
```

The heartbeat is the mechanism the PR claims provides safety. Mocking it out means the tests provide no coverage of the behavior that is being sold as the correctness argument. `test_lock_not_released_if_expired` is the most important test in the file and it also runs against a service with a mocked heartbeat.

The tests verify the happy-path business logic correctly, but they do not test the distributed systems behavior at all. Integration tests against a real Redis instance (or `fakeredis`) with simulated lock expiry are needed.
