Decision: Request Changes

Summary: The core locking pattern and heartbeat concept are sound, but there are several meaningful flaws that make this unsafe to merge into a production government-facing system. The most critical issue is that the heartbeat uses instance-level state (`self._heartbeat_active`, `self._heartbeat_thread`), which means a single `InventoryService` instance cannot safely handle concurrent calls — a second reservation attempt will clobber the heartbeat state of an in-flight one. Beyond that, the lock release in `_acquire_lock` is not atomic, creating a TOCTOU window where the service can delete a lock it no longer owns. The test suite also has coverage gaps that mask real behavior.

---

## Detailed Findings

### 🔴 Critical — Concurrency Corruption via Instance-Level Heartbeat State

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not scoped to a single lock acquisition. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the entire point of this service), the second call's `_start_heartbeat` will overwrite `_heartbeat_active` and `_heartbeat_thread`, and the first call's `_stop_heartbeat` will kill the second call's heartbeat prematurely — or vice versa.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance. Pass a threading `Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning("Heartbeat lost ownership key=%s — stopping", lock_key)
                break

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

Note the use of `stop_event.wait(timeout=...)` instead of `time.sleep` + a flag check. This also means the heartbeat exits promptly on shutdown rather than sleeping through its interval.

### 🔴 Critical — Non-Atomic Lock Release (TOCTOU Race)

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock. You would then delete a lock you no longer own. This is a well-documented Redis anti-pattern. The canonical fix is a Lua script that makes the check-and-delete atomic.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_lock_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
released = self._release_lock_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 🟡 Significant — Input Validation Is Claimed but Not Implemented

The checklist says "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation whatsoever.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- Empty or whitespace-only `item_id` (which would produce a valid Redis key like `inventory:lock:   `)
- `quantity <= 0` (a caller passing `quantity=0` would always succeed and create a phantom order)
- Excessively long `item_id` strings that could bloat Redis keys

**Fix:** Introduce a request model and validate at the boundary.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128)
    quantity: int = Field(ge=1, le=10_000)

    @field_validator("item_id")
    @classmethod
    def no_whitespace_only(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("item_id must not be blank")
        return v.strip()
```

### 🟡 Significant — Database Errors Are Silently Unhandled

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock has been decremented with no order record — inventory is lost with no compensation or rollback. Only `RuntimeError` from lock acquisition is caught. Any other exception propagates uncaught to the caller with no structured response.

**Fix:** Wrap the DB block and handle partial failure explicitly.

```python
try:
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
except Exception as e:
    logger.error(
        "DB operation failed after stock decrement item_id=%s order_id=%s: %s",
        item_id, order_id, e
    )
    # Attempt compensating write
    try:
        self.db.update_stock(item_id, current_stock)
    except Exception as rollback_err:
        logger.critical(
            "ROLLBACK FAILED item_id=%s — stock inconsistency possible: %s",
            item_id, rollback_err
        )
    return ReservationResult(success=False, message="Reservation failed. Please retry.")
```

A more robust long-term solution would be a transactional outbox or saga pattern, but the compensating write is a minimum acceptable guard.

### 🟡 Significant — Heartbeat Interval Exceeds Safe Threshold Relative to TTL

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

With the current `time.sleep` implementation (not `Event.wait`), the heartbeat fires at `t=10`, `t=20`, `t=30`. At `t=30` the lock may already be expired before the third heartbeat executes, depending on scheduling jitter and GIL pressure. The general rule is to renew at no more than `TTL / 3`, which this technically meets, but the sleep-based implementation has no tolerance for delays. The `Event.wait` fix above resolves this, but the values should be documented with this constraint explicitly called out.

### 🟠 Minor — Test Suite Mocks Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # ← stop never runs
    return svc
```

The heartbeat — one of the two primary features of this PR — is mocked out in every test that uses the `service` fixture. The lock release path in `_acquire_lock`'s `finally` block is also never exercised with a real token comparison because `redis_client.get` returns a hardcoded `b"test-token"` that never matches the UUID generated inside the context manager.

**Required additions:**
1. An integration-style test using `fakeredis` that exercises the actual lock acquire/release/heartbeat cycle.
2. A test that verifies `db.update_stock` is rolled back when `db.create_order` raises.
3. A test that verifies `quantity=0` and blank `item_id` are rejected.

---

### 🟠 Minor — `_start_heartbeat` and `_stop_heartbeat` Remain as Public-ish Methods After Refactor

Once the heartbeat is scoped to the context manager (as recommended above), `_start_heartbeat` and `_stop_heartbeat` as standalone methods on the class become dead code and a test surface that encourages the exact mocking anti-pattern seen in the test suite. Remove them.
