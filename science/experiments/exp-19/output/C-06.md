Decision: Request Changes

Summary: The implementation demonstrates solid distributed systems thinking — the ownership-verified lock release, heartbeat pattern, and exponential backoff are all directionally correct. However, there are several correctness, safety, and architecture issues that must be resolved before this handles real inventory under concurrent load. The test suite also has a critical gap that masks the most dangerous race condition in the code.

---

## Critical Issues

### 1. Heartbeat Race Condition — Check-Then-Act is Not Atomic

```python
# CURRENT — two separate round trips; not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, another process can acquire the lock. You would then extend a lock you no longer own, silently evicting the legitimate holder.

**Fix:** Use a Lua script to make the check-and-extend atomic — the same pattern already used correctly at release time should be applied here.

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_lock_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In heartbeat
extended = self._extend_lock_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

### 2. Lock Release is Also Not Atomic

```python
# CURRENT — same TOCTOU flaw at release
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is the canonical example of why lock release must be a Lua script. The checklist claims "ownership verification" but the implementation does not actually guarantee it.

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```python
RELEASE_LOCK_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""
```

### 3. `InventoryService` is Not Safe for Concurrent Use

The heartbeat state is stored as instance variables:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` on the same service instance simultaneously, they will corrupt each other's heartbeat state. This is a silent data integrity failure, not a noisy crash.

**Fix:** Pass heartbeat state into the lock context rather than storing it on `self`. Encapsulate it in a dataclass or pass it through a closure.

```python
@dataclass
class _LockState:
    active: bool = True

@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    state = _LockState()
    # ... pass state into heartbeat thread, not self
```

### 4. Database Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented with no corresponding order. The Redis lock does not protect you from this — it only serializes access. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. If the underlying `db` object does not support transactions, that is an architecture gap that needs to be resolved before merge.

## Moderate Issues

### 5. No Input Validation on `reserve_item`

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation. `item_id` could be an empty string, whitespace, or a path-injection attempt. `quantity` could be zero or negative.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

A `quantity` of `0` currently passes the `current_stock < quantity` check and would create a zero-quantity order successfully.

---

### 6. Heartbeat Thread Leaks on Lock Acquisition Failure Path

```python
if not acquired:
    raise RuntimeError(...)

self._start_heartbeat(lock_key, token)  # only reached if acquired
```

This path is safe. However, if `_start_heartbeat` itself raises (e.g., thread creation fails under resource exhaustion), the lock is acquired but the heartbeat never starts and `_stop_heartbeat` is never called in a finally block scoped to that failure. Consider wrapping the heartbeat start in the try block with a compensating lock release.

---

### 7. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed ...")
    return ReservationResult(success=False, ...)
```

Any `RuntimeError` raised inside the lock — including from `db.update_stock` or `db.create_order` — will be silently swallowed and returned as "service temporarily unavailable." Introduce a specific exception type.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) instead of RuntimeError
# catch only LockAcquisitionError at the call site
```

## Test Coverage Gaps

### 8. The Most Critical Concurrent Scenario Is Not Tested

`test_lock_not_released_if_expired` mocks `redis.get` to return a different token, but the service fixture already mocks `_start_heartbeat` and `_stop_heartbeat`. The test does not exercise the real heartbeat or the real lock release path. There is no test that:

- Spawns two threads calling `reserve_item` simultaneously
- Verifies stock is decremented exactly once
- Verifies no negative stock results

This is the entire purpose of this service. It needs at least one integration-level concurrent test.

### 9. Zero and Negative Quantity Are Not Tested

```python
# These should both be rejected before any lock is acquired
service.reserve_item("item_001", quantity=0)
service.reserve_item("item_001", quantity=-5)
```

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the lock is renewed at 1/3 of TTL. This is a reasonable ratio, but document the invariant explicitly: heartbeat interval must always be less than `LOCK_TTL / 2`.
- The `token` yielded from `_acquire_lock` is not used by the caller. Either use it internally only and stop yielding it, or document why a caller might need it.
- `logger = logging.getLogger(__name__)` is correct. Confirm the application logging configuration emits `DEBUG` level in non-production environments only, given the heartbeat emits at `DEBUG`.
