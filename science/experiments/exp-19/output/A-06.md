**Decision: Request Changes**

**Summary:** The heartbeat thread reduces the probability of lock expiry but does not eliminate the fundamental race condition: a process pause (GC, OS scheduler, VM migration) can occur *after* the heartbeat last confirmed ownership and *before* or *during* the database writes, at which point the lock may expire, be acquired by another worker, and both workers will execute `update_stock` and `create_order` concurrently against an unguarded database. There is no fencing token or optimistic lock check at the database write level, so the lock does not actually protect the write. Additionally, the heartbeat implementation carries a shared-state concurrency bug and the `InventoryService` instance is not safe for concurrent use. These issues are detailed below.

---

## Critical: No Fencing at the Database Write Layer

This is the central flaw. The heartbeat is a mitigation, not a guarantee. Consider the following pause scenario:

```
Worker A                          Worker B
--------                          --------
acquires lock
heartbeat last ran at T=20s
                                  [Worker A: all threads paused — GC stop-the-world,
                                   hypervisor migration, OS scheduler starvation]
                                  lock TTL expires at T=30s
                                  Worker B acquires lock
                                  reads stock = 10
                                  writes stock = 9
                                  creates order B
[Worker A resumes]
reads stock = 10  ← stale read, already committed by B
writes stock = 9  ← duplicate decrement lost
creates order A   ← phantom order against already-decremented stock
```

The heartbeat cannot prevent this. The pause can be arbitrarily long and can happen between any two instructions. The only correct defense is a condition at the database write itself.

**Required fix — optimistic locking via version/ETag check:**

```python
# db.update_stock must reject the write if the version has changed
# since the read. This must be atomic at the database level.

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"

    try:
        with self._acquire_lock(lock_key):

            # Read returns both value and a version token
            # (row version, ETag, updated_at timestamp, or SELECT FOR UPDATE)
            stock_record = self.db.get_stock_with_version(item_id)

            if stock_record is None:
                return ReservationResult(success=False, message=f"Item {item_id} not found")

            current_stock, version = stock_record.stock, stock_record.version

            if current_stock < quantity:
                return ReservationResult(
                    success=False,
                    remaining_stock=current_stock,
                    message="Insufficient stock"
                )

            new_stock = current_stock - quantity
            order_id = str(uuid.uuid4())

            # This write MUST fail if version has changed since the read.
            # The database enforces this atomically — not the application layer.
            rows_updated = self.db.update_stock_if_version_matches(
                item_id, new_stock, expected_version=version
            )

            if rows_updated == 0:
                # Another writer committed between our read and write.
                # The lock was not sufficient protection.
                raise ConcurrentModificationError(
                    f"Stock version conflict for item {item_id}, retry required"
                )

            self.db.create_order(order_id, item_id, quantity)
            # ...
    except ConcurrentModificationError as e:
        logger.warning("Version conflict item_id=%s: %s", item_id, e)
        return ReservationResult(success=False, message="Conflict, please retry.")
```

The corresponding SQL must enforce this at the storage layer:

```sql
-- update_stock_if_version_matches
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND version = :expected_version
  AND stock >= :quantity;
-- rowcount == 0 means conflict or race; application must handle this
```

Without this, the Redis lock is a courtesy signal between cooperative processes, not a correctness guarantee.

## High: Shared Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are instance variables, not local to each lock acquisition. If two coroutines or threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the entire point of this service), they will race on these fields.

```python
# Thread A calls _start_heartbeat → sets self._heartbeat_active = True
# Thread B calls _stop_heartbeat  → sets self._heartbeat_active = False
# Thread A's heartbeat silently dies while A still holds the lock
# The lock TTL is no longer being renewed
# The scenario the heartbeat was meant to prevent now occurs
```

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    stop_event = threading.Event()  # local to this acquisition

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Release only if we still own the lock
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.warning(
                    "Heartbeat: lock no longer owned, stopping key=%s", lock_key
                )
                break

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note also: `stop_event.wait(timeout=N)` is preferable to `time.sleep(N)` because it allows the thread to exit promptly when the event is set, rather than sleeping through the full interval after the lock has already been released.

## High: Heartbeat Check-Then-Extend Is Not Atomic

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write — separate round trip
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. The heartbeat then extends a lock it no longer owns. This must be a single atomic operation:

```lua
-- heartbeat.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

class InventoryService:
    def __init__(self, redis_client, db):
        self.redis = redis_client
        self.db = db
        self._heartbeat_script = redis_client.register_script(HEARTBEAT_SCRIPT)

    def _extend_lock(self, lock_key: str, token: str) -> bool:
        result = self._heartbeat_script(
            keys=[lock_key], args=[token, self.LOCK_TTL]
        )
        return result == 1
```

The same atomicity requirement applies to the release in the `finally` block — that `GET` + conditional `DELETE` is also a TOCTOU race and should use the standard Lua release script.

## Medium: Input Validation Is Declared But Not Enforced

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation. `ReservationResult` is a Pydantic output model, not an input validator.

```python
# quantity=-5 passes silently and increments stock
# quantity=0 passes silently and creates a zero-quantity order
# item_id="" passes silently

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        raise ValueError("item_id must be a non-empty string")
    if quantity < 1:
        raise ValueError(f"quantity must be >= 1, got {quantity}")
    if quantity > MAX_RESERVATION_QUANTITY:  # define a sane upper bound
        raise ValueError(f"quantity exceeds maximum allowed per reservation")
```
