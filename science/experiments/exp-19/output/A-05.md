**Decision: Request Changes**

**Summary:** The heartbeat thread reduces the probability of lock expiry but does not eliminate it — a full process pause (GC stop-the-world, OS scheduler preemption, container CPU throttle) can suspend all threads simultaneously, including the heartbeat, allowing the TTL to lapse silently while `reserve_item` resumes and executes the database writes believing the lock is still held. There is no fencing token or optimistic concurrency check at the `db.update_stock` / `db.create_order` call sites, so the lock does not actually protect those writes. Additionally, the service has a single-instance heartbeat state design that makes it unsafe for concurrent callers.

---

## Detailed Findings

### Critical — The Lock Does Not Protect the Writes

This is the core issue and the reason I cannot approve this PR.

```python
# The heartbeat extends the TTL. But it cannot prevent this sequence:
#
# T=0      Worker A acquires lock, heartbeat starts
# T=10     Heartbeat fires, extends TTL to T=40
# T=11     OS/JVM/container pauses ALL threads in Worker A (GC, SIGSTOP, CPU throttle)
# T=41     Lock expires. Worker B acquires the lock.
# T=42     Worker B reads stock=10, decrements to 9, writes.
# T=43     Worker A resumes. It still holds `token` in local memory.
# T=43     Worker A reads stock=9 (stale read from before pause, or re-reads 9)
# T=43     Worker A decrements to 8, calls db.update_stock(item_id, 8)
# T=43     Worker A calls db.create_order(...)
#
# Two reservations have now been committed against the same stock unit.
# The heartbeat did not help because it was also paused.
```

The heartbeat is a latency optimization, not a safety guarantee. Kleppmann's critique of Redlock makes exactly this point: you cannot reason about lock validity from inside the process that holds the lock, because you cannot observe your own pause.

The fix requires a fencing token or an optimistic lock at the database layer:

```python
# Option A — Optimistic concurrency / conditional update at the DB layer
# The DB enforces: only decrement if current stock matches what we read.
# A second writer with a stale read will get a conflict error, not a silent overwrite.

rows_affected = self.db.update_stock_conditional(
    item_id,
    expected_stock=current_stock,   # the value we read
    new_stock=new_stock
)
if rows_affected == 0:
    # Another writer got there first. Abort.
    raise ConflictError(f"Stock changed under us for item {item_id}")

# Option B — Monotonic fencing token from Redis (INCR-based)
# Pass the token value through to the DB write. The DB rejects writes
# from a token older than the last committed token for that item.
# This requires DB-side token tracking but is the stronger guarantee.
```

Without one of these, the distributed lock is providing a false sense of safety. The checklist says "distributed lock with ownership verification" — the ownership verification only happens at lock *release*, not at the point of the write. That is too late.

### Critical — Single-Instance Heartbeat State Is Not Concurrency-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared instance state
        self._heartbeat_thread: Optional[...] = None  # shared instance state
```

If a single `InventoryService` instance handles two concurrent calls to `reserve_item` — which is the normal case in any web framework with a thread pool or async workers — the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and `_stop_heartbeat` sets `_heartbeat_active = False` for both. The first call's heartbeat is now orphaned or prematurely killed.

```python
# Thread 1: acquires lock A, starts heartbeat, _heartbeat_active = True
# Thread 2: acquires lock B, starts heartbeat, overwrites _heartbeat_thread
# Thread 1: finishes, calls _stop_heartbeat(), sets _heartbeat_active = False
# Thread 2's heartbeat loop exits on next iteration. Lock B is now unprotected.
```

The heartbeat state must be scoped to the lock acquisition, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread, yield, finally: heartbeat_active.clear()
```

### Moderate — TOCTOU Race in Lock Release

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)           # GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)              # DEL — not atomic with GET
```

Between the `GET` and the `DEL`, another process can acquire the lock. This is a classic TOCTOU. The standard fix is a Lua script that makes the check-and-delete atomic:

```lua
-- Standard safe release script
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### Moderate — `quantity` Has No Validated Lower Bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check for any non-negative stock value, creates an order for zero units, and decrements stock by zero. `quantity=-5` increases stock. The Pydantic model is on `ReservationResult`, not on the input. Add a validated input model:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128)
    quantity: int = Field(ge=1, le=1000)  # enforce domain bounds

    @field_validator('item_id')
    @classmethod
    def item_id_no_injection(cls, v: str) -> str:
        # Defensive: ensure item_id cannot carry Redis key injection characters
        if not v.replace('-', '').replace('_', '').isalnum():
            raise ValueError('item_id contains invalid characters')
        return v
```

### Minor — `item_id` Is Interpolated Directly Into the Redis Key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is caller-controlled and not validated, a value like `foo:lock:admin` or a key containing spaces or newlines could produce unexpected key collisions or, in some Redis client configurations, protocol confusion. The validator above addresses this, but it should be called explicitly before constructing the key.

### Minor — Tests Mock Away the Exact Behaviors Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat is never tested
    svc._stop_heartbeat = MagicMock()    # stop is never tested
    return svc
```

The concurrency-safety of the heartbeat — the primary novel mechanism this PR introduces — has zero test coverage. `test_lock_not_released_if_expired` tests the token comparison logic but does so with a mock that returns a hardcoded token, so it does not exercise the actual Lua-script-less TOCTOU window. Add integration-level tests using a real Redis instance (or `fakeredis`) that exercise the heartbeat thread lifecycle and the concurrent-caller scenario.
