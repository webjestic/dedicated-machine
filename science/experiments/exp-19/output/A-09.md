**Decision: Request Changes**

**Summary:** The heartbeat mitigates *slow* operations but does not close the fundamental race this architecture is designed to prevent. The database writes at lines `self.db.update_stock` / `self.db.create_order` carry no fencing token or optimistic-lock check, meaning a process that was fully paused (GC stop-the-world, VM live migration, kernel preemption) long enough for the lock to expire and be re-acquired by a competing worker will execute its stale writes unconditionally after resuming — the heartbeat cannot help a thread that is not scheduled to run. Beyond that core issue, there are several additional correctness and reliability defects.

---

## Critical: No Fencing at the Write Layer

This is the defect the entire PR is built around preventing, and it is not prevented.

```python
# Current — unguarded writes
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

The lock protects the *read-modify* phase. It does not protect the *write* phase unless the storage layer can reject a write from a worker whose lock generation is stale. The heartbeat keeps the lock alive for a *slow* worker, but a *paused* worker — one whose entire thread scheduler was suspended — sees no wall-clock time pass internally. It wakes up, the lock has expired, another worker has already decremented stock, and this worker overwrites that result with its own stale `new_stock` value.

**Required fix — choose one:**

**Option A — Optimistic locking in the database (preferred):**
```python
# Schema: stock table gains a `version` integer column.
# Read returns (stock, version). Write is conditional on version match.

current_stock, version = self.db.get_stock_versioned(item_id)

# ... availability check ...

rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=current_stock - quantity,
    expected_version=version,
)
if rows_affected == 0:
    # Another writer beat us. Abort — do not create the order.
    raise StaleStockError(f"Concurrent modification detected for {item_id}")
```

This makes the write itself safe regardless of what happened to the lock.

**Option B — Fencing token passed through to storage:**
```python
# The lock token becomes the fencing token.
# Storage layer rejects writes whose token is no longer current.
with self._acquire_lock(lock_key) as token:
    ...
    self.db.update_stock(item_id, new_stock, fence=token)
    # db checks: UPDATE ... WHERE item_id=? AND lock_token=?
```

Without one of these, the distributed lock is a coordination hint, not a correctness guarantee.

## High: Heartbeat Race on Lock Expiry Check

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps 10s
        if self._heartbeat_active:
            current = self.redis.get(lock_key)  # two-step check
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` + `EXPIRE` sequence is not atomic. Between the `GET` confirming ownership and the `EXPIRE` executing, the TTL can reach zero, Redis can evict the key, and a second worker can acquire the lock with a new token. The `EXPIRE` then resets the TTL on the *new* owner's lock, extending a lock this process no longer holds.

**Required fix — use a Lua script for atomic check-and-extend:**
```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```
```python
EXTEND_SCRIPT = redis.client.Script("""
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
""")

extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.error("Lost lock during heartbeat key=%s — aborting", lock_key)
    self._heartbeat_active = False
    # signal the main thread to abort the critical section
```

The same atomicity requirement applies to the release path — the existing `GET` + `DELETE` in the `finally` block has the same race and should also be a Lua script.

## High: Heartbeat Thread Silently Loses the Lock Without Signaling the Main Thread

```python
if not extended:
    logger.error("Lost lock during heartbeat ...")
    self._heartbeat_active = False
    # ← main thread is never told; it continues writing
```

If the heartbeat detects that the lock is gone, the main thread in `reserve_item` has no way to know. It will continue executing `update_stock` and `create_order` against a database it no longer has exclusive access to.

**Required fix:** Use a shared `threading.Event` to signal abort:

```python
def __init__(self, ...):
    ...
    self._lock_lost = threading.Event()

def _start_heartbeat(self, lock_key, token):
    self._lock_lost.clear()
    ...
    def _beat():
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                self._lock_lost.set()   # signal main thread
                return

# In reserve_item, before each DB call:
if self._lock_lost.is_set():
    raise LockLostError("Lock expired before write could complete")
```

## Medium: Instance-Level Heartbeat State Breaks Concurrent Calls

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False        # instance variable
        self._heartbeat_thread = None         # instance variable
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If two coroutines or threads call `reserve_item` concurrently on the same `InventoryService` instance — entirely plausible under an async framework or a thread pool — the second call's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first call's `_stop_heartbeat` sets `_heartbeat_active = False`, killing the second call's heartbeat.

**Required fix:** Scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key, retry_attempts=3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    lock_lost = threading.Event()

    def _beat():
        while heartbeat_active.is_set():
            time.sleep(self.HEARTBEAT_INTERVAL)
            extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                lock_lost.set()
                return

    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token, lock_lost
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

## Medium: Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is not joined
```

The thread is daemon=True so it will not block process exit, but it will continue executing for up to `HEARTBEAT_INTERVAL` seconds after the lock is released. During that window it can extend a lock that has already been legitimately acquired by another worker. The fix above (scoping to context + `thread.join`) resolves this.

## Medium: `quantity` Has No Validated Lower Bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` acquires the lock, reads stock, and creates an order for zero units — a spurious write under lock. `quantity=-5` would *increase* stock. Neither is caught before the lock is acquired.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(ge=1, le=1000)

# validate at service boundary, before acquiring the lock
```

Validating before lock acquisition also avoids holding the lock while discovering the input is invalid.

## Low: Tests Mock Away the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The test suite verifies the happy-path business logic but provides zero coverage of the lock lifecycle, the heartbeat atomicity, the lost-lock signal path, or the concurrent-instance issue. These are the scenarios most likely to fail in production.

**Required additions:**
- A test that verifies `update_stock` is *not* called when the lock is lost mid-operation (requires the `lock_lost` event to be wired up).
- A test for the atomic extend script receiving a mismatched token.
- A concurrency test using `threading.Barrier` to force two threads past the stock read simultaneously and assert only one write commits.
