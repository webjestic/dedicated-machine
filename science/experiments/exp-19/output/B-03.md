Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not eliminate the fundamental race condition: if the process is paused (GC, OS scheduling, container CPU throttle, network stall) *after* the heartbeat thread last extended the TTL but *before* the DB writes complete, the lock can expire, a second process can acquire it and begin its own read-modify-write cycle, and then the first process resumes and commits its stale writes — with no mechanism at the database layer to detect or reject the conflict. There is no fencing token, no optimistic lock version check, and no conditional write at the DB level, meaning the distributed lock is the *only* guard and it is not a hard guarantee. Additionally, the implementation carries several secondary defects in concurrency management, test correctness, and operational safety that must be addressed before this is suitable for a high-contention flash sale path.

---

## Detailed Findings

### 1. Critical — No Fencing Token or DB-Level Optimistic Lock (Data Integrity Failure)

This is the primary rejection reason.

The heartbeat thread runs on a *best-effort* basis inside the same process. Consider this exact sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat extends TTL to T+30
T=11  OS/container pauses all threads in Process A (GC, CPU throttle, cgroup freeze)
T=41  Lock TTL expires (30s elapsed since last extension at T=10)
T=42  Process B acquires the lock, reads stock=5, proceeds
T=43  Process B writes stock=4, creates order B
T=44  Process A resumes, still inside `with _acquire_lock(...)` block
T=44  Process A writes stock=4 (its own stale read), creates order A
      → Two orders created, stock decremented twice from the same baseline
```

The heartbeat cannot prevent this. A paused process cannot extend its own lock. The lock TTL is a wall-clock guarantee, not a process-liveness guarantee. This is a well-documented failure mode described by Martin Kleppmann in his analysis of Redlock and addressed by systems like ZooKeeper (ephemeral nodes + session fencing) or by database-level conditional writes.

**Required fix:** The database writes must carry a fencing token or version check that makes them conditional. Two acceptable patterns:

```python
# Option A — Optimistic locking with a version/generation column
# DB schema: stock_version INTEGER, updated atomically

rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_version=stock_version   # read alongside current_stock
)
if rows_affected == 0:
    raise StaleWriteError("Stock was modified by a concurrent writer; aborting")

# Option B — Fencing token passed through to DB layer
# Lock token (or a monotonic counter from Redis INCR) is written as a
# "last_writer" column; the UPDATE includes WHERE last_writer <= :token
# so a later writer always wins and an earlier stale writer is rejected.
```

Without one of these, the distributed lock is purely advisory and the critical section is not actually protected against the pause scenario.

### 2. Critical — `update_stock` and `create_order` Are Not Atomic

Even ignoring the lock expiry issue, the two DB writes are not wrapped in a transaction:

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # crashes
# → stock is decremented but no order exists; inventory is permanently lost
```

**Required fix:**

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 3. High — Heartbeat Thread Is a Shared Instance Variable (Concurrency Bug)

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is a singleton (the typical pattern for a service object), two concurrent calls to `reserve_item` will race on these fields:

```python
# Thread 1 sets self._heartbeat_active = True, starts heartbeat for lock A
# Thread 2 sets self._heartbeat_active = True, starts heartbeat for lock B
# Thread 1 calls self._stop_heartbeat() → sets self._heartbeat_active = False
# Thread 2's heartbeat is now silently killed mid-operation
```

**Required fix:** Heartbeat state must be scoped to the lock acquisition, not the service instance. Use a local `threading.Event` or encapsulate in a context-local object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=2)
        ...
```

### 4. High — Heartbeat Thread Is Not Joined on Stop

`_stop_heartbeat` sets the flag but does not join the thread. The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10s) between checks. After `_stop_heartbeat()` is called, the thread can remain alive for up to 10 more seconds, still holding a reference to `lock_key` and `token`, and can extend the TTL of a lock that has already been released and re-acquired by another process:

```
T=0   Process A releases lock (deletes Redis key)
T=0   Process B acquires the same lock with its own token
T=3   Process A's heartbeat thread wakes up, checks current == token (stale local var)
      → No, token won't match here, so this specific scenario is safe
```

Actually the token check in the heartbeat does protect against *extending* the wrong lock, but the thread is still a leaked resource. In a high-throughput flash sale, hundreds of these daemon threads can accumulate. The thread must be joined:

```python
finally:
    stop_event.set()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 5. Medium — Lock Release Is Not Atomic (TOCTOU on Delete)

The release sequence is:

```python
current = self.redis.get(lock_key)       # read
if current and current.decode() == token:
    self.redis.delete(lock_key)          # write
```

Between the `GET` and the `DELETE`, another process can acquire the lock. This is a classic TOCTOU. The standard fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 6. Medium — No Input Validation on `quantity`

The PR checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `int` parameters with no validation. A caller can pass `quantity=0` or `quantity=-5`:

- `quantity=0`: passes the stock check, creates a zero-quantity order, decrements stock by 0 — a no-op order that pollutes the order table.
- `quantity=-5`: `current_stock < -5` is false for any positive stock, so it passes, and `new_stock = current_stock - (-5)` *increases* stock.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = Field(gt=0, le=MAX_RESERVATION_QUANTITY)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 7. Low — Test Suite Has a Typo That Masks a Missing Assertion

In `test_reserve_lock_contention`:

```python
db.update_stack.assert_not_called()   # "stack" not "stock"
```

This assertion silently passes because `MagicMock` auto-creates any attribute. The test provides zero coverage of the property it claims to test. This should be:

```python
db.update_stock.assert_not_called()
db.create_order.assert_not_called()
```

---

### 8. Low — No Test Covers the Pause/Stale-Write Scenario

The test `test_lock_not_released_if_expired` checks that the lock is not *deleted* when the token has changed, but it does not assert that the DB writes are also aborted in that scenario. Once the fencing token is implemented (item 1), a test must verify that `update_stock` is not called when the write would be rejected as stale.

## Summary of Required Changes Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Add DB-level fencing token or optimistic version check on all writes |
| 2 | Critical | Wrap `update_stock` + `create_order` in a single DB transaction |
| 3 | High | Scope heartbeat state to lock context, not service instance |
| 4 | High | Join heartbeat thread on stop to prevent thread leak |
| 5 | Medium | Use Lua script for atomic lock release |
| 6 | Medium | Validate `quantity > 0` and enforce an upper bound via Pydantic |
| 7 | Low | Fix typo `update_stack` → `update_stock` in test |
