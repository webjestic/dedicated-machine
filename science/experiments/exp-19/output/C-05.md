Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `nx=True`, token-based ownership verification, and heartbeat renewal are all correctly structured. However, there are several correctness, safety, and concurrency issues that must be resolved before this can be merged into a government-facing production system.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrent Usage

The heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) is stored on the `InventoryService` instance, not scoped to the lock acquisition. If `InventoryService` is a shared singleton (the common pattern with dependency injection), two concurrent calls to `reserve_item` will race on these fields — one call's `_stop_heartbeat()` will kill the other's heartbeat thread.

```python
# Current — instance-level, not safe for concurrent use
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Scope heartbeat state to the lock context using a local `threading.Event` or a dedicated lock-context object.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquisition logic ...

    heartbeat_thread = threading.Thread(
        target=self._beat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

### 2. Heartbeat Check-Then-Act Is a Race Condition

The heartbeat has a TOCTOU flaw: it reads the token, checks ownership, then calls `expire` as two separate commands. Between the `GET` and `EXPIRE`, the lock can expire and be acquired by another process, which then gets its TTL silently extended by the old heartbeat.

```python
# Vulnerable — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

**Fix:** Use a Lua script to make the check-and-extend atomic.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _extend_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(
        EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL
    )
    return result == 1
```

---

### 3. Lock Release Is Also Non-Atomic

The same TOCTOU problem exists in the `finally` block of `_acquire_lock`.

```python
# Vulnerable — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

**Fix:** Use the standard Redlock release Lua script.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

### 4. No Input Validation on `reserve_item`

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string, all of which will silently produce incorrect behavior (negative stock, phantom orders).

```python
# No guard against this
service.reserve_item("", quantity=-1)
```

**Fix:** Add a request model or explicit guards.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

### 5. Database Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # failure here = lost stock
```

**Fix:** These must execute within a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat()` only sets a flag; it does not join the thread. The `finally` block can complete and release the lock while the heartbeat thread is mid-sleep and about to call `expire` on a key that now belongs to a different owner.

**Fix:** Call `heartbeat_thread.join(timeout=...)` in the `finally` block after setting the stop event (as shown in fix #1 above).

---

### 7. Test Suite Mocks Away the Code Under Test

The `service` fixture replaces `_start_heartbeat` and `_stop_heartbeat` with mocks, meaning the tests never exercise the heartbeat logic at all. The most critical concurrent behavior — the exact code that has the race conditions identified above — has zero test coverage.

**Fix:** Add integration-level tests using a real or fakeredis instance that verify heartbeat extension and atomic release behavior.

```python
import fakeredis

def test_heartbeat_extends_lock():
    r = fakeredis.FakeRedis()
    svc = InventoryService(r, db_mock)
    # verify TTL is extended before expiry
    ...
```

---

### 8. Retry Backoff Ceiling

With `retry_attempts=3`, the maximum backoff is `(2^2) * 0.1 = 0.4s`. Under genuine flash sale load this is reasonable, but the value is hardcoded with no jitter, which means all contending threads will retry at the same intervals and continue to collide (thundering herd).

**Fix:** Add jitter.

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

---

## Minor Issues

### 9. `item_id` Logged Directly

`item_id` is logged without sanitization. If `item_id` originates from user input, this is a log injection vector.

### 10. Redis Connection Errors Not Handled

A `redis.RedisError` during lock acquisition or release will propagate as an unhandled exception. The `except RuntimeError` block in `reserve_item` will not catch it.
