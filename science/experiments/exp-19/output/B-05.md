Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not eliminate the fundamental race condition it claims to solve. The database writes at the critical section have no fencing token or optimistic concurrency check, meaning a process pause between the heartbeat's last extension and the write operations can still result in stale data being committed under a lock held by a different process. Additionally, there is a secondary concurrency defect in the heartbeat implementation itself and a test that silently passes despite asserting against a nonexistent method.

---

## Critical Issues

### 1. No Fencing Token at the Database Write Layer — Data Integrity Failure

This is the core defect, and the heartbeat does not fix it.

The PR author's reasoning is: *"the heartbeat keeps the lock alive, so the lock won't expire."* This reasoning is incorrect in the general case. Consider this sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat fires, extends TTL to T=40
T=11  OS/GC/hypervisor pauses all threads in Process A for 35 seconds
T=46  Process A resumes — lock expired at T=40, Process B acquired it at T=41
T=46  Process A calls db.update_stock() and db.create_order() — writes proceed
T=46  Process B is also inside the lock, also writing
```

The heartbeat thread is paused along with every other thread. It cannot extend a lock it cannot run. The lock expires. Another process acquires it. When Process A resumes, the `with self._acquire_lock(lock_key)` context manager does not re-validate lock ownership before the writes execute. The writes proceed against stale read data under a lock now owned by someone else.

```python
# This check happens ONLY at lock release, not before writes
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The ownership check is only performed during cleanup. It is never consulted before `db.update_stock()` or `db.create_order()`. By the time the stale write is detected at release, the damage is done.

**Required fix:** The database layer must enforce a fencing token or optimistic lock. Two acceptable approaches:

**Option A — Fencing token passed to DB writes:**
```python
# Pass the lock token (or a monotonic version) into the write
self.db.update_stock(item_id, new_stock, expected_version=read_version)
self.db.create_order(order_id, item_id, quantity, lock_token=token)
# DB layer rejects the write if version has changed or token is stale
```

**Option B — Optimistic concurrency check before writes:**
```python
# Re-validate lock ownership immediately before the write, not just at release
current = self.redis.get(lock_key)
if not current or current.decode() != token:
    raise RuntimeError("Lock expired before write — aborting")

self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

Option B reduces the window but does not close it — there is still a gap between the check and the write. Option A with a DB-level version check is the only approach that actually provides a safety guarantee. Both options together are better than either alone.

### 2. Heartbeat Has a Race Condition on `_heartbeat_active`

The heartbeat state is stored as a plain instance variable with no synchronization primitive:

```python
self._heartbeat_active = False  # set in __init__
self._heartbeat_active = True   # set in _start_heartbeat
self._heartbeat_active = False  # set in _stop_heartbeat
```

This is a shared mutable boolean accessed from two threads with no lock, no `threading.Event`, and no memory barrier. In CPython this is unlikely to corrupt due to the GIL, but it is still formally a data race and will not be safe if the implementation ever moves to a GIL-free runtime (Python 3.13+ free-threaded mode, PyPy STM, etc.).

More practically: if `InventoryService` is reused across multiple calls (which it will be, given it is injected as a dependency), concurrent calls to `reserve_item` will share `_heartbeat_active` and `_heartbeat_thread`. The second call's `_start_heartbeat` will overwrite `_heartbeat_thread` while the first call's heartbeat is still running.

**Required fix:**
```python
# Use threading.Event for clean signaling
self._heartbeat_stop_event = threading.Event()

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_stop_event.clear()

    def _beat() -> None:
        while not self._heartbeat_stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()

def _stop_heartbeat(self) -> None:
    self._heartbeat_stop_event.set()
```

Additionally, `_heartbeat_thread` and `_heartbeat_stop_event` should be local to the `_acquire_lock` context rather than instance state, to avoid the shared-state problem across concurrent calls entirely.

### 3. Heartbeat Check-Then-Extend Is Not Atomic

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This should be a single atomic operation using a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.warning("Heartbeat could not extend lock — lock lost key=%s", lock_key)
    self._heartbeat_stop_event.set()  # signal the main thread
```

The same atomicity issue exists in the lock release code and should receive the same treatment. Redlock implementations use exactly this pattern.

## Moderate Issues

### 4. `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a raw `int` with no bounds checking. A caller can pass `quantity=0` (silently creates a zero-quantity order) or `quantity=-5` (increases stock). The Pydantic checklist item is misleading — Pydantic is only used for the response model, not for input validation.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

### 5. `item_id` Has No Validation

An empty string or a string containing Redis special characters could produce unexpected key behavior. At minimum:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
```

### 6. Non-Atomic Dual Write Without a Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. This is a partial write. These two operations must be wrapped in a database transaction or a compensating rollback must be implemented.

## Test Issues

### 7. `test_reserve_lock_contention` Asserts Against a Nonexistent Method

```python
db.update_stack.assert_not_called()  # typo: "stack" not "stock"
```

`MagicMock` will silently create `update_stack` as a new mock attribute rather than raising `AttributeError`. This test passes regardless of whether `update_stock` was called. The assertion is a no-op and provides zero coverage of the intended behavior.

**Fix:**
```python
db.update_stock.assert_not_called()
```

### 8. Tests Mock Away the Exact Behavior Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the primary safety mechanism being claimed in the PR. Mocking it out means no test exercises the actual lock lifecycle. There should be at least one integration-style test that runs the real heartbeat against a real or fakeredis instance and verifies lock extension behavior.

### 9. No Test for the Lock-Expired-Before-Write Scenario

There is a test named `test_lock_not_released_if_expired` but it only verifies that `delete` is not called when the token mismatches. It does not test whether the write was correctly aborted. This is the most important failure mode and it has no coverage.
