Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock with token ownership, heartbeat renewal, and exponential backoff are all appropriate patterns for this problem. However, there are several correctness, safety, and architecture issues that must be addressed before this is suitable for a production government contracting environment where inventory integrity is critical.

---

## Issues Requiring Changes

### 🔴 Critical

**1. Race condition in heartbeat: non-atomic GET + EXPIRE**

```python
# Current — vulnerable to TOCTOU race
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and `EXPIRE`, another process could acquire the lock. This must be atomic. Use a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_lua = self.redis.register_script(HEARTBEAT_SCRIPT)

# In heartbeat thread
self._heartbeat_lua(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same race exists in the lock release path and must be fixed there too.

---

**2. Non-atomic lock release: GET + DELETE**

```python
# Current — same TOCTOU problem
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is the canonical Redlock mistake. Use the standard atomic Lua release:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

---

**3. Instance-level heartbeat state breaks concurrent usage**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` is stored on the instance. If `reserve_item` is called concurrently (e.g., from a thread pool or async framework), one call's `_stop_heartbeat()` will cancel another call's heartbeat, silently dropping lock protection. Heartbeat state must be scoped to the lock context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            self._heartbeat_lua(keys=[lock_key], args=[token, self.LOCK_TTL])

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lua(keys=[lock_key], args=[token])
```

**4. Database operations are not transactional**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no order record — inventory is silently lost. These must execute in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. If the underlying DB layer doesn't support it, that must be resolved before this merges.

### 🟡 Significant

**5. No input validation on `item_id` or `quantity`**

The checklist claims "Input validation via Pydantic" but there is none. `item_id` is passed directly into a Redis key and a DB query without sanitization. A malformed `item_id` (e.g., containing `:`, newlines, or null bytes) could corrupt lock key namespacing or cause unexpected DB behavior. `quantity` has no lower bound check.

```python
# Minimum required
if not isinstance(item_id, str) or not item_id.strip():
    raise ValueError("item_id must be a non-empty string")
if not quantity or quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

A Pydantic request model at the call boundary would be cleaner and is what the checklist implies.

---

**6. Heartbeat thread is never joined**

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. The thread could still be executing a Redis call after the lock is released, potentially extending a lock that now belongs to another process. `thread.join()` must be called before the lock is deleted.

---

**7. Broad exception handling swallows DB errors**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught, bypassing the `finally` block's lock release... actually, `finally` will still run, but the caller receives an unhandled exception rather than a structured `ReservationResult`. Widen the handler or add a separate catch for DB exceptions with appropriate logging.

---

### 🔵 Minor / Test Quality

**8. Tests mock out the exact behavior under test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # ← stop never runs
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the actual heartbeat or lock release logic. `test_lock_not_released_if_expired` claims to test token mismatch behavior, but the lock release code path is bypassed by the mock. These are coverage theater. At minimum, one integration-style test should run the real lock path against a local Redis instance (or `fakeredis`).

**9. Missing test: partial DB failure / rollback**

There is no test for the scenario where `create_order` raises after `update_stock` succeeds. This is the most dangerous failure mode and must be covered.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | 🔴 | Heartbeat GET+EXPIRE is non-atomic — use Lua |
| 2 | 🔴 | Lock release GET+DELETE is non-atomic — use Lua |
| 3 | 🔴 | Heartbeat state on instance breaks concurrent calls |
| 4 | 🔴 | DB operations not wrapped in a transaction |
| 5 | 🟡 | No actual input validation despite checklist claim |
| 6 | 🟡 | Heartbeat thread not joined before lock release |
| 7 | 🟡 | DB exceptions not caught, return unstructured errors |
| 8 | 🔵 | Tests mock the code paths they claim to cover |
| 9 | 🔵 | No test for partial DB failure / rollback |

Items 1–4 are blockers. The implementation is structurally on the right track, but the race conditions in the atomic operations and the instance-level concurrency bug would cause real data integrity failures under the exact load conditions this service is designed to handle.
