Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not eliminate the fundamental race condition: if the process is paused (GC stop-the-world, OS scheduler preemption, container CPU throttle) *after* the heartbeat last confirmed ownership but *before* the DB writes complete, the lock can expire, be acquired by another process, and both processes will then execute `update_stock` and `create_order` against the same stale stock value with no database-level guard to reject the second writer. There is no fencing token or optimistic concurrency check at the DB write layer, meaning the lock is the sole correctness mechanism — and it is not a reliable one. Additionally, the service has a structural concurrency defect where a single `InventoryService` instance cannot safely serve concurrent requests because `_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not scoped to the lock context.

---

## Detailed Findings

### 1. Critical — No Fencing Token or Optimistic Lock at the Database Write Layer

This is the primary rejection reason.

The PR author is aware of the expiry risk and added a heartbeat to mitigate it. The heartbeat does reduce the *probability* of expiry, but it cannot eliminate it. The scenario that breaks correctness is not "the lock expires because we forgot to renew it" — it is "the process is paused for longer than the remaining TTL window between heartbeats."

Trace the full lifecycle:

```
T+0:   Process A acquires lock, heartbeat starts
T+10:  Heartbeat fires, lock extended to T+40
T+11:  Process A calls db.get_stock() → returns 5
T+11:  OS scheduler / GC / container throttle pauses all threads in Process A
T+41:  Lock expires (30s TTL from last extension at T+10)
T+41:  Process B acquires the same lock
T+41:  Process B calls db.get_stock() → still returns 5 (A hasn't written yet)
T+41:  Process B decrements to 4, writes update_stock(item, 4), creates order
T+42:  Process A resumes from pause
T+42:  Process A still holds its local variable current_stock = 5
T+42:  Process A decrements to 4, writes update_stock(item, 4), creates order
```

Both orders are created. Stock is decremented twice from the same baseline of 5, but the second write overwrites the first, leaving stock at 4 instead of 3. This is a classic lost-update anomaly. The heartbeat thread was also paused and could not fire.

The correct fix is to push the correctness guarantee down to the database layer, where it cannot be bypassed by process pauses. Two standard approaches:

**Option A — Optimistic concurrency with a version column:**

```python
# Schema: items table has a `version` INTEGER column

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    lock_key = f"inventory:lock:{item_id}"
    try:
        with self._acquire_lock(lock_key):
            row = self.db.get_stock_with_version(item_id)
            # returns {"stock": 5, "version": 42}

            if row is None:
                return ReservationResult(success=False, message=f"Item {item_id} not found")
            if row["stock"] < quantity:
                return ReservationResult(success=False, remaining_stock=row["stock"],
                                         message="Insufficient stock")

            new_stock = row["stock"] - quantity
            order_id = str(uuid.uuid4())

            # This UPDATE will affect 0 rows if another writer already
            # incremented the version, making the stale write a no-op.
            rows_affected = self.db.update_stock_versioned(
                item_id,
                new_stock,
                expected_version=row["version"]   # <-- fencing predicate
            )
            if rows_affected == 0:
                raise RuntimeError("Concurrent modification detected — version mismatch")

            self.db.create_order(order_id, item_id, quantity)
            return ReservationResult(success=True, order_id=order_id,
                                     remaining_stock=new_stock,
                                     message="Reservation successful")
    except RuntimeError as e:
        logger.error("Reservation failed item_id=%s: %s", item_id, e)
        return ReservationResult(success=False,
                                 message="Service temporarily unavailable. Please try again.")
```

The corresponding SQL for `update_stock_versioned`:

```sql
UPDATE items
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND version = :expected_version
  AND stock >= :quantity;   -- belt-and-suspenders guard against negative stock
```

**Option B — Atomic conditional decrement (no separate read needed):**

```sql
UPDATE items
SET stock = stock - :quantity
WHERE item_id = :item_id
  AND stock >= :quantity
RETURNING stock;
```

If this returns no rows, the reservation failed. This is safe without any application-level lock at all, provided the DB enforces `stock >= 0` as a constraint.

Either approach means a stale write is rejected at the storage layer regardless of what happened to the Redis lock.

### 2. High — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False          # shared across all callers
        self._heartbeat_thread: Optional[...] = None
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the normal pattern in any web framework with a shared service object), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat and leaving its lock unprotected.

The fix is to scope heartbeat state to the lock acquisition context, not to the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local to this acquisition

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

    def _beat():
        while not heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.set()   # signals the beat loop to exit
        t.join(timeout=2)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

### 3. Medium — Heartbeat Uses `get` + `expire` Instead of a Lua Atomic Check-and-Extend

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE` there is a window where the lock can expire and be acquired by another process. The `EXPIRE` then extends a lock that is no longer yours. This should be a single atomic Lua script:

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In _beat:
self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same issue exists in the lock release path and should receive the same treatment.

### 4. Medium — No Input Validation on `quantity`

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts raw `int` parameters with no validation. A caller passing `quantity=0` or `quantity=-5` will pass the `current_stock < quantity` check silently and either do nothing or *increase* stock.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not isinstance(item_id, str):
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

### 5. Low — Test Suite Has a Typo That Hides a Real Test Gap

```python
def test_reserve_lock_contention(redis_client, db):
    ...
    db.update_stack.assert_not_called()   # typo: 'stack' not 'stock'
```

`MagicMock` will silently create `update_stack` as a new attribute rather than raising `AttributeError`, so this assertion always passes and tests nothing. It should be `db.update_stock.assert_not_called()`. The checklist item "Tests pass" is technically true but misleading because this test is inert.

### 6. Low — `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has exited
