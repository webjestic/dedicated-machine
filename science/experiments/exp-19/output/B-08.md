Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not eliminate the core race condition: if the process is paused (GC stop-the-world, OS scheduler preemption, container CPU throttle) *after* the heartbeat last extended the TTL but *before* the DB writes complete, the lock can expire, a second process can acquire it and begin its own writes, and then the first process resumes and commits its stale read against a database that has already moved on. There is no fencing token or optimistic lock version check at the database write layer, so both processes can corrupt the same inventory record with no detection. Additionally, the service has a concurrency bug in its heartbeat state management that makes it unsafe for concurrent use. These issues must be resolved before this code touches production inventory data.

---

## Detailed Findings

### Critical: No Fencing Token at the Database Write Layer

This is the central flaw and the reason the heartbeat argument does not hold.

The lock lifecycle looks like this:

```
T=0   Process A acquires Redis lock, token=abc, TTL=30s
T=10  Heartbeat extends TTL to T=40
T=15  OS pauses Process A (scheduler, GC, cgroup CPU throttle, etc.)
T=40  Lock expires — Redis deletes it
T=41  Process B acquires the same lock, reads stock=10, begins its write
T=42  Process A resumes, still holding its in-memory read of stock=10
T=43  Process A calls db.update_stock(item_id, 9)   ← stale write
T=44  Process B calls db.update_stock(item_id, 9)   ← also stale write
```

Both processes read `stock=10`, both write `stock=9`. One unit of inventory has been sold twice. The heartbeat cannot prevent this because the pause can happen in the window *between* the last heartbeat extension and the DB write — a window that is always nonzero.

The fix is a fencing token or optimistic concurrency check at the write layer. The Redis lock should return a monotonically increasing token (Redlock calls this a fencing token), and the database write must be conditional:

```python
# Option A: optimistic lock with a version column
# db.update_stock must reject the write if version != expected_version
self.db.update_stock(item_id, new_stock, expected_version=stock_version)

# Option B: conditional SQL — the WHERE clause is the safety net
# UPDATE inventory SET stock = %s, version = version + 1
# WHERE item_id = %s AND stock = %s AND version = %s
rows_affected = self.db.update_stock_conditional(
    item_id, new_stock, expected_current_stock=current_stock
)
if rows_affected == 0:
    raise StaleDataError("Concurrent modification detected, aborting reservation")
```

Without one of these, the distributed lock is providing the *appearance* of safety while the actual write is unguarded.

### Critical: Heartbeat State Is Shared Instance State — Not Safe for Concurrent Calls

```python
# InventoryService instance fields:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are set on `self`, not scoped to the lock context. If two threads call `reserve_item` on the same `InventoryService` instance concurrently:

- Thread 1 sets `self._heartbeat_active = True`, starts its heartbeat thread
- Thread 2 sets `self._heartbeat_active = True`, starts a second heartbeat thread
- Thread 1 finishes, calls `self._stop_heartbeat()` → sets `self._heartbeat_active = False`
- Thread 2's heartbeat thread now sees `False` and exits, leaving Thread 2's lock unrenewed
- Thread 1's `self._heartbeat_thread` reference has been overwritten by Thread 2's thread, so join/cleanup is also broken

The heartbeat state must be local to each lock acquisition, not stored on the instance. Pass it through a closure or a small context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread, yield, stop via heartbeat_active.clear()
```

### Moderate: Heartbeat Check-Then-Act Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE` another process could acquire the lock. This should be a single atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The lock release in `_acquire_lock` has the same issue — `GET` then `DELETE` should also be a single Lua script (this is the standard Redlock release pattern).

---

### Moderate: `db.update_stock` and `db.create_order` Are Not in a Transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order record exists. The reservation is lost with no recovery path. These two writes must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### Minor: Test Suite Has a Typo That Masks a Real Assertion

```python
def test_reserve_lock_contention(redis_client, db):
    ...
    db.update_stack.assert_not_called()  # ← "stack" not "stock"
```

Because `MagicMock` auto-creates attributes on access, `db.update_stack` is a new mock that was never called, so this assertion always passes regardless of what the code does. The test provides no actual coverage of the invariant it claims to test. This should be `db.update_stock.assert_not_called()`.

---

### Minor: No Input Validation on `quantity`

`quantity` is typed as `int` but there is no guard against `quantity <= 0`. A caller passing `quantity=0` would succeed, decrement stock by zero, and create a zero-unit order record. A caller passing `quantity=-5` would *increase* stock. Add an explicit guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

### Minor: Lock Granularity May Serialize Too Broadly

A single lock per `item_id` means all reservation attempts for that item queue behind each other. For very high-demand flash sale items this will become a bottleneck. Consider whether stock can be pre-partitioned into buckets (e.g., `inventory:lock:{item_id}:{shard}`) to allow parallel reservations against separate stock pools. This is a design note rather than a blocker, but it should be addressed before this goes to a real flash sale load.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | **Critical** | No fencing token or conditional write at DB layer — lock expiry during process pause allows double-spend |
| 2 | **Critical** | Heartbeat state stored on instance — concurrent callers corrupt each other's heartbeat |
| 3 | Moderate | Heartbeat GET+EXPIRE and lock release GET+DELETE are not atomic |
| 4 | Moderate | `update_stock` and `create_order` not wrapped in a DB transaction |
| 5 | Minor | Typo `update_stack` in test masks the assertion entirely |
| 6 | Minor | No validation that `quantity > 0` |

Items 1 and 2 are blockers. The code must not be merged in its current state.
