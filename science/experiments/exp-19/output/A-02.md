**Decision: Request Changes**

**Summary:** The heartbeat thread reduces the probability of expiry during normal slowdowns, but it does not eliminate the failure mode it claims to prevent — and more critically, it introduces no compensating control at the database write layer. The lock protects a read-modify-write sequence across two separate database calls with no atomicity guarantee and no fencing token, meaning a process-wide pause (GC stop-the-world, VM live migration, OS scheduler starvation, container CPU throttle) that outlasts the heartbeat interval will silently produce a phantom write that the system cannot detect or reject. This is the exact scenario the heartbeat is supposed to prevent, and the heartbeat cannot prevent it.

---

## Critical Findings

### 1. No Fencing Token at the Write Layer — The Core Failure

This is the architectural defect that invalidates the lock's safety guarantee.

```python
# What the code does:
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)

# What the code needs to do:
self.db.update_stock(item_id, new_stock, if_version=stock_version)
# or equivalently:
# UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s
```

The scenario the heartbeat cannot prevent:

1. Worker A acquires lock, heartbeat starts.
2. Worker A reads `stock = 5`, computes `new_stock = 4`.
3. OS/JVM/container pauses **all threads in the process** — including the heartbeat thread — for longer than `LOCK_TTL` (30 seconds). This is not hypothetical; it happens during GC pauses on heap-heavy JVMs, during VM live migration, and under Linux CFS bandwidth throttling.
4. Lock expires. Worker B acquires the lock, reads `stock = 5`, writes `stock = 4`, releases lock.
5. Worker A's threads resume. The heartbeat wakes up, checks `redis.get(lock_key)` — the key now belongs to B's token, so the heartbeat correctly does not extend. But Worker A's `reserve_item` execution context **does not know this happened**. It proceeds directly to `self.db.update_stock(item_id, 4)`.
6. Stock is now `4` but two orders exist for the same unit. The lock provided no protection.

The heartbeat thread and the application thread share the same process. Any pause that freezes one freezes both. The heartbeat is useful for slow-but-running operations; it is structurally incapable of protecting against process-level pauses. Kleppmann's original Redlock critique documents this exact failure mode.

The fix is a conditional write at the database layer — either optimistic locking with a version/etag column, or a `WHERE stock = :expected_stock` predicate that causes the update to affect zero rows if another writer has intervened. The application must check the affected row count and treat zero as a conflict.

### 2. Non-Atomic Two-Phase Write With No Rollback Path

```python
self.db.update_stock(item_id, new_stock)   # write 1
self.db.create_order(order_id, item_id, quantity)  # write 2
```

These are two separate database operations with no enclosing transaction. If the process dies, loses its database connection, or throws between these two lines, the system is left in an inconsistent state: stock is decremented but no order record exists, or an order exists for stock that was never decremented. There is no retry path, no compensation, and no detection.

Both writes must be wrapped in a single database transaction with commit/rollback semantics.

---

### 3. Shared Heartbeat State Is Not Reentrant

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the expected usage pattern for a service object — two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will terminate the first call's heartbeat, and vice versa. The fix is to make heartbeat state local to each lock acquisition, passed as closure variables or encapsulated in a per-call object.

---

### 4. `quantity` Has No Validated Lower Bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no validation. A caller passing `quantity=0` will succeed and create an order record for zero units. A caller passing `quantity=-5` will increase stock. The Pydantic model is on the result, not the input. Add a validated input model:

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int = Field(default=1, ge=1, le=1000)  # enforce domain bounds
```

---

### 5. `item_id` Is Used Directly in a Redis Key Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

`item_id` is an unvalidated string interpolated directly into a Redis key. A value containing `:` or whitespace or a crafted namespace prefix could collide with unrelated keys or enable key-space manipulation. Validate that `item_id` matches an expected format (alphanumeric, bounded length) before constructing the key.

---

### 6. The Heartbeat Race on Check-Then-Extend

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a non-atomic check-then-act. The lock can expire between the `GET` and the `EXPIRE`, at which point a different owner holds the key and the heartbeat extends a lock it no longer owns. This should be a single Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the lock release in `_acquire_lock`.

---

### 7. Test Suite Does Not Test the Failure Mode It Claims to Test

```python
def test_lock_not_released_if_expired(service, redis_client):
    """Lock release is skipped if token no longer matches (lock expired or stolen)."""
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test verifies that `redis.delete` is not called when the token has changed. It does not verify — and cannot verify with this test design — that the database write was also suppressed. The test passes while the phantom write executes. A correct test would assert that `db.update_stock` was not called when the lock was lost mid-operation, which requires the implementation to actually check lock ownership before writing.

## Required Changes Before Approval

| # | Severity | Change Required |
|---|----------|-----------------|
| 1 | **Blocking** | Add optimistic lock / conditional write at `db.update_stock` — check version or expected stock value, abort on mismatch |
| 2 | **Blocking** | Wrap both DB writes in a single transaction with rollback on failure |
| 3 | **Blocking** | Make heartbeat state per-call, not per-instance |
| 4 | **Blocking** | Validate `quantity >= 1` and apply an upper bound |
| 5 | **High** | Validate and sanitize `item_id` before key construction |
| 6 | **High** | Replace check-then-extend and check-then-delete with atomic Lua scripts |
| 7 | **Medium** | Rewrite the expiry test to assert the DB write was suppressed, not just the Redis delete |
