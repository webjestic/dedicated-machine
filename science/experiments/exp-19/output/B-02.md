**Decision: Request Changes**

**Summary:** The heartbeat mechanism creates a false sense of safety but does not eliminate the fundamental fencing problem: if the Python process is paused (GC stop-the-world, OS scheduler preemption, VM live migration, container CPU throttling) *after* the heartbeat thread last extended the TTL but *before* the DB writes complete, the lock can expire and be acquired by another process, after which this process resumes and blindly executes `update_stock` and `create_order` against stale data with no database-level guard to reject the write. There is no fencing token or optimistic lock version check at the DB write layer. Additionally, the heartbeat implementation has a race condition and the test suite has a silent bug that masks a real failure path.

---

## Detailed Review

### Critical: No Fencing Token at the Database Write Layer

This is the core failure. The PR description frames the heartbeat as solving the lock expiry problem. It does not.

```python
# reserve_item — the critical window
current_stock = self.db.get_stock(item_id)   # reads stock = 10

# ← process pauses here for 35 seconds (GC, scheduler, VM migration)
# ← lock TTL = 30s, heartbeat interval = 10s
# ← heartbeat thread is also paused — it cannot extend the lock
# ← lock expires at t+30
# ← Process B acquires the lock, reads stock = 10, writes stock = 9, releases lock
# ← Process A resumes

new_stock = current_stock - quantity          # computes 10 - 1 = 9 (stale)
self.db.update_stock(item_id, new_stock)      # overwrites Process B's write: 9 → 9
self.db.create_order(order_id, item_id, quantity)  # duplicate reservation
```

The heartbeat thread runs in the *same process* as the application thread. Any pause that freezes the application thread also freezes the heartbeat thread. The lock expires. Another process acquires it and completes a valid write. The original process resumes with a stale `current_stock` value and no mechanism exists to detect or reject the now-invalid write.

The fix requires a database-level guard. Two standard approaches:

**Option A — Optimistic locking with a version/row version check:**
```python
# DB schema: stock table has a `version` column (integer, increments on each write)
current_stock, version = self.db.get_stock_with_version(item_id)

# ...

rows_updated = self.db.update_stock_if_version(
    item_id,
    new_stock=new_stock,
    expected_version=version
)
if rows_updated == 0:
    # Another writer modified the row — our read is stale, abort
    raise StaleDataError(f"Concurrent modification detected for item {item_id}")
```

**Option B — Fencing token passed through to the DB write:**

The Redis lock token (or a monotonic counter from Redis `INCR`) is stored alongside the reservation. The DB write is conditional: `UPDATE stock SET quantity = %s WHERE item_id = %s AND lock_token = %s`. This is harder to implement correctly across two separate stores and Option A is generally preferred.

Without one of these, the distributed lock provides ordering *hints* under normal conditions but provides **no correctness guarantee** under process pause scenarios. For a flash sale inventory system — where overselling has direct financial and contractual consequences — this is not acceptable.

### Significant: Heartbeat Race Condition

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps 10 seconds
        if self._heartbeat_active:            # re-checks flag
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The `get` + `expire` sequence is not atomic. Between the `GET` and the `EXPIRE` call, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns.

The fix is a Lua script executed atomically:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_LOCK_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In __init__:
self._extend_script = self.redis.register_script(EXTEND_LOCK_SCRIPT)

# In _beat:
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat could not extend lock — lock lost key=%s", lock_key)
    self._heartbeat_active = False  # signal to the main thread
```

The same non-atomic pattern exists in `_acquire_lock`'s finally block (`GET` then `DELETE`), which should also use a Lua script or the standard Redlock release pattern.

### Significant: Heartbeat Thread State is Instance-Level, Not Call-Level

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are stored on the instance, not scoped to a single `_acquire_lock` invocation. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the normal case for a shared service object), one call's `_stop_heartbeat()` will set `_heartbeat_active = False` and kill the heartbeat belonging to a *different concurrent call*.

```python
# Thread 1 acquires lock, starts heartbeat → _heartbeat_active = True
# Thread 2 acquires lock, starts heartbeat → _heartbeat_active = True (same flag)
# Thread 1 finishes → _stop_heartbeat() → _heartbeat_active = False
# Thread 2's heartbeat is now dead, lock will expire
```

The heartbeat state must be encapsulated per lock acquisition, not per service instance. The cleanest fix is a small dataclass or a closure that carries its own `active` flag:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

### Moderate: Test Suite Has a Silent Failure

```python
def test_reserve_lock_contention(redis_client, db):
    redis_client.set.return_value = False
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    result = svc.reserve_item("item_001")
    assert result.success is False
    assert "unavailable" in result.message
    db.update_stack.assert_not_called()   # ← typo: "update_stack" not "update_stock"
```

`db.update_stack` does not exist on the mock. `MagicMock` will silently create it and the assertion `assert_not_called()` will always pass regardless of what the code does. This test provides no actual safety net for the lock contention path.

```python
# Correct:
db.update_stock.assert_not_called()
```

### Minor: No Input Validation on `quantity`

The PR checklist claims input validation via Pydantic, but `quantity` is a raw `int` with no bounds check before it reaches the lock and DB layer.

```python
service.reserve_item("item_001", quantity=0)   # succeeds, creates a zero-quantity order
service.reserve_item("item_001", quantity=-5)  # succeeds, *increases* stock
```

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

### Minor: No Observability on Heartbeat Failure

If the heartbeat loses the lock (Redis unavailable, lock stolen), the main thread has no way to know. It will proceed to execute DB writes after the lock is gone. The heartbeat should signal back to the main thread — via a `threading.Event` or a shared flag — so the main thread can abort the operation before touching the database.
