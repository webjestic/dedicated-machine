# Code Review: `feature/flash-sale-inventory`

---

## Decision: Request Changes

**Summary:** The heartbeat renewal mechanism creates a false sense of safety but does not eliminate the fundamental race condition it is designed to prevent. The critical database writes — `update_stock` and `create_order` — have no fencing token or optimistic concurrency check at the storage layer, meaning a process pause that outlasts the heartbeat interval can cause the lock to expire, a second process to acquire it and commit writes, and then the first process to resume and overwrite with stale data. Additionally, the service has a stateful concurrency defect that makes it unsafe for concurrent use within a single process. These issues together represent a data integrity failure, not a performance concern.

---

## Critical Issues

### 1. No Fencing at the Database Write Layer (Data Integrity Failure)

This is the core defect. The review instructions ask specifically about this scenario, and the code does not survive it.

**The scenario:**

1. Process A acquires the lock. Token is `abc-123`. Heartbeat starts.
2. Process A reads `current_stock = 5`.
3. The OS pauses all threads in Process A for longer than `HEARTBEAT_INTERVAL` (10s). This can happen due to GC, a kernel preemption, a VM live migration, a container CPU throttle, or a debugger. The heartbeat thread is paused along with everything else — it cannot extend the lock.
4. The lock TTL of 30s elapses. Redis deletes the key.
5. Process B acquires the lock. Token is `xyz-789`. Process B reads `current_stock = 5`, decrements to 4, writes `update_stock(item_id, 4)`, creates an order. Lock released cleanly.
6. Process A resumes. It calls `self.redis.get(lock_key)` in the `finally` block. The key now holds `xyz-789`, not `abc-123`, so it correctly skips the delete. **But the damage is already done.** Process A already executed `self.db.update_stock(item_id, 4)` and `self.db.create_order(...)` with data computed from a stale read. Stock is now 3, but two orders exist for a total of 2 units — one of which may be a phantom.

The heartbeat does not protect against this. It only extends the lock while threads are running. A paused process cannot extend anything.

The token check in the `finally` block only prevents releasing a lock you no longer own. It does not prevent writing with stale data after the lock has expired. The writes have already happened before the `finally` block runs.

**Required fix:** The database layer must enforce correctness independently of the lock. Two standard approaches:

**Option A — Optimistic locking with a version/ETag:**
```python
# Read returns a version alongside the value
stock, version = self.db.get_stock_with_version(item_id)

# Write is conditional: only succeeds if version still matches
updated = self.db.update_stock_if_version(item_id, new_stock, expected_version=version)
if not updated:
    # Another writer committed between our read and write — abort
    raise StaleDataError("Concurrent modification detected, reservation aborted")
```

**Option B — Monotonic fencing token passed through to the DB:**
```python
# The lock token acts as a fencing token only if the DB enforces ordering.
# Redis alone cannot provide a monotonically increasing token, so a
# separate sequence (e.g., Redis INCR, or a DB sequence) is needed.
fence = self.redis.incr("inventory:fence:sequence")
# Pass fence to DB; DB rejects writes with a lower fence than last seen.
self.db.update_stock_fenced(item_id, new_stock, fence_token=fence)
```

Without one of these, the distributed lock is a coordination hint, not a correctness guarantee.

### 2. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not per-call variables. If two threads call `reserve_item` on the same `InventoryService` instance concurrently:

- Thread 1 calls `_start_heartbeat`, sets `self._heartbeat_active = True`.
- Thread 2 calls `_start_heartbeat`, overwrites `self._heartbeat_thread`.
- Thread 1's `finally` block calls `_stop_heartbeat`, sets `self._heartbeat_active = False`.
- Thread 2's heartbeat is now silently dead. Its lock will expire with no renewal.

For a flash sale service, concurrent calls on the same instance are the expected case, not an edge case. This design is fundamentally incompatible with that workload.

**Required fix:** Heartbeat state must be scoped to the individual lock acquisition, not the instance. Use a local flag per call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()
    thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... conditional delete ...
```

## Significant Issues

### 3. `update_stock` and `create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls. If the process crashes, is killed, or raises an exception between them, stock is decremented but no order exists — inventory is silently lost. This is a consistency gap independent of the locking issue.

**Required fix:** Both operations must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 4. `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no lower bound check. A caller passing `quantity=0` or `quantity=-5` will pass the stock check and either create a zero-unit order or increase stock. The `ReservationResult` model uses Pydantic but the method signature does not.

**Required fix:**

```python
from pydantic import validator

# Or validate explicitly at entry:
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

### 5. `item_id` Is Used Directly in a Redis Key Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is externally supplied and contains `:`, spaces, or newline characters, this can produce malformed or colliding Redis keys. In a government contracting context, input that influences key construction must be treated as untrusted.

**Required fix:** Validate or encode `item_id` before use:

```python
import re
if not re.fullmatch(r'[a-zA-Z0-9_\-]{1,128}', item_id):
    return ReservationResult(success=False, message="Invalid item identifier")
```

## Test Quality Issues

### 6. Heartbeat Is Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat logic — the primary novel mechanism in this PR — has zero test coverage. The tests verify the happy path of the business logic, not the behavior of the system under the conditions it was designed for.

**Required additions:**
- A test that verifies the heartbeat calls `redis.expire` with the correct key and token.
- A test that verifies the heartbeat stops after `_stop_heartbeat` is called.
- A test that verifies the heartbeat does **not** extend the lock when the token no longer matches.

### 7. Typo in Test Produces a Silent False Pass

```python
def test_reserve_lock_contention(redis_client, db):
    ...
    db.update_stack.assert_not_called()  # "stack" not "stock"
```

`MagicMock` will silently create a new attribute `update_stack` and the assertion will pass vacuously. This test does not verify what it claims to verify.

**Required fix:**

```python
db.update_stock.assert_not_called()
```

## Summary of Required Changes

| # | Severity | Issue |
|---|----------|-------|
| 1 | **Critical** | No fencing token or optimistic lock at DB write layer — stale writes possible after lock expiry |
| 2 | **Critical** | Heartbeat state is instance-scoped, not call-scoped — concurrent callers corrupt each other's lock state |
| 3 | **High** | `update_stock` and `create_order` not wrapped in a transaction — partial writes on crash |
| 4 | **Medium** | `quantity` not validated — zero or negative values accepted |
| 5 | **Medium** | `item_id` used raw in Redis key construction — no sanitization |
| 6 | **Medium** | Heartbeat has no test coverage despite being the central mechanism |
| 7 | **Low** | Typo `update_stack` in test causes silent false pass |

Items 1 and 2 must be resolved before this can be approved. The lock infrastructure as written does not provide the correctness guarantee the PR claims.
