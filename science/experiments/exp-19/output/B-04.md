**Decision: Request Changes**

**Summary:** The heartbeat mechanism creates a false sense of safety but does not close the fundamental race condition this code is designed to prevent. The database writes at the end of `reserve_item` are not protected by any fencing token or optimistic concurrency check at the storage layer, meaning a process pause between the heartbeat's last renewal and the write operations can allow stale data to be committed after the lock has expired and been acquired by another process. Additionally, there is a critical concurrency defect in the heartbeat implementation itself, and the test suite contains a silent bug that masks a test failure. Each issue is detailed below.

---

## Critical: No Fencing Token at the Write Layer

This is the core flaw. The entire correctness argument rests on the assumption that the lock is held continuously from read through write. That assumption is not guaranteed.

Consider this execution sequence:

```
Process A holds lock, heartbeat last renewed at T=20s
T=25s: Process A's thread is paused (GC, OS scheduler, network stall)
T=30s: Lock TTL expires
T=31s: Process B acquires the lock, reads stock=5, decrements to 4, writes stock=4
T=35s: Process A resumes, writes stock=4 (computed from its stale read of 5)
```

The heartbeat does not eliminate this window — it only reduces its probability. Any pause longer than `LOCK_TTL - last_heartbeat_offset` seconds is sufficient to open it. On a JVM this is a GC stop-the-world event. On CPython it is a sufficiently long I/O stall or OS preemption. The `LOCK_TTL` of 30 seconds is described as "generous for slow DB operations," which is precisely the scenario where this pause is most likely.

The fix is to enforce correctness at the database layer, not just at the lock layer. The write must be conditional. Two acceptable patterns:

**Option 1 — Optimistic locking with a version column:**
```python
# Schema: items table has a `version` integer column
# Read includes version
row = self.db.get_stock_with_version(item_id)
current_stock, version = row.stock, row.version

# Write is conditional on version not having changed
rows_affected = self.db.update_stock_if_version(
    item_id,
    new_stock=current_stock - quantity,
    expected_version=version
)
if rows_affected == 0:
    # Another writer beat us; the lock expired and was re-acquired
    raise ConcurrentModificationError(item_id)
```

**Option 2 — Atomic decrement with a floor check at the DB layer:**
```sql
UPDATE items
SET stock = stock - :quantity
WHERE item_id = :item_id
  AND stock >= :quantity
RETURNING stock;
```

If zero rows are returned, the reservation must be rejected and the order record must not be created. Either approach makes the write self-validating regardless of what happened to the lock.

## Critical: Heartbeat Has a Non-Atomic Check-Then-Act Race

The heartbeat's ownership check is not atomic:

```python
# services/inventory.py lines ~47-50
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend — NOT atomic with (1)
```

Between steps (1) and (3), the lock can expire and be acquired by Process B. The heartbeat then extends Process B's lock using Process B's key but under the assumption that it still belongs to Process A. The correct implementation uses a Lua script to make the check and extend atomic:

```python
HEARTBEAT_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    script = self.redis.register_script(HEARTBEAT_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if result == 0:
                    logger.warning(
                        "Heartbeat lost ownership of lock key=%s — lock expired",
                        lock_key
                    )
                    self._heartbeat_active = False
                    # Signal the main thread that the lock is gone
                else:
                    logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

The same non-atomic pattern exists in the lock release in `_acquire_lock`. The `get` followed by `delete` should also be a Lua script or use `redis.delete` with a check script to avoid deleting a lock owned by another process in a race:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""
```

## Moderate: Heartbeat Expiry Is Not Propagated to the Main Thread

When the heartbeat detects that it can no longer extend the lock (because the lock expired and was taken by another process), it currently just stops looping. The main thread continues executing the critical section without knowing the lock is gone. There is no signaling mechanism — no event, no flag checked by `reserve_item`, no exception raised. The main thread will proceed to call `db.update_stock` and `db.create_order` on data it read under a lock it no longer holds.

At minimum, add a threading event:

```python
def __init__(self, ...):
    ...
    self._lock_lost_event = threading.Event()

# In heartbeat, on failed renewal:
self._lock_lost_event.set()

# In reserve_item, before each DB write:
if self._lock_lost_event.is_set():
    raise LockExpiredError("Lock expired during operation")
```

This is still not a substitute for the fencing token fix above, but it reduces the blast radius.

---

## Moderate: Shared Heartbeat State Breaks Concurrent Usage

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to a single lock acquisition. If two threads call `reserve_item` concurrently on the same `InventoryService` instance, one thread's `_stop_heartbeat()` call will set `_heartbeat_active = False` and kill the other thread's heartbeat. The heartbeat state must be encapsulated per lock acquisition, not per service instance.

## Minor: Test Suite Has a Silent Failure

In `test_reserve_lock_contention`:

```python
db.update_stack.assert_not_called()  # typo: "stack" not "update_stock"
```

`MagicMock` auto-creates attributes on access, so `db.update_stack` is a new mock that was never called, and the assertion passes vacuously. The test does not actually verify that `db.update_stock` was not called. This should be:

```python
db.update_stock.assert_not_called()
```

---

## Minor: Input Validation Is Incomplete

The PR checklist claims input validation via Pydantic, but `reserve_item` accepts `item_id: str` and `quantity: int` as raw Python parameters with no validation. There is no guard against `quantity <= 0`, an empty `item_id`, or an excessively large quantity. A caller passing `quantity=0` would succeed and create an order for zero units. Consider a Pydantic model at the entry point:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128)
    quantity: int = Field(..., ge=1, le=1000)
```
