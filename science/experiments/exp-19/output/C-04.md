Decision: Request Changes

Summary: The implementation demonstrates solid intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, safety, and architecture issues that must be addressed before this is suitable for a production government contracting environment. The most critical is a race condition in the heartbeat that can corrupt lock ownership, and a concurrency flaw that makes the service unsafe under parallel load.

---

## Critical Issues

### 1. Heartbeat Race Condition — Check-Then-Act on Token Verification

```python
# CURRENT — NOT ATOMIC
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. You would then extend a lock you no longer own. This must be an atomic operation.

```python
# REQUIRED — use a Lua script for atomic check-and-extend
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at __init__
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
    self._heartbeat_active = False
```

The same atomic pattern must also be applied to the lock release in `_acquire_lock`. The current `GET` + `DELETE` sequence has the identical race condition.

---

### 2. Instance-Level Heartbeat State Is Not Thread-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables. If `InventoryService` is used as a singleton (the standard pattern with a DI container or application factory), two concurrent calls to `reserve_item` will overwrite each other's `_heartbeat_active` and `_heartbeat_thread`. The second caller stopping its heartbeat will silently kill the first caller's lock renewal.

The fix is to make heartbeat state local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
                heartbeat_active.clear()
                return

    # ... acquire lock, start thread with _beat, yield
    # In finally: heartbeat_active.clear(); thread.join(timeout=...)
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which has a latency window where the heartbeat loop checks `_heartbeat_active` after sleeping but the flag was already cleared.

### 3. Input Validation Is Claimed but Not Implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model gating the inputs. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string, all of which will silently produce incorrect behavior or corrupt stock values.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
    ...
```

The `item_id` pattern constraint is particularly important here because it is interpolated directly into a Redis key (`inventory:lock:{item_id}`). An unvalidated `item_id` containing `:` or whitespace can cause key collisions or unexpected Redis behavior.

## Moderate Issues

### 4. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If `create_order` raises, stock has already been decremented and the reservation is silently lost. This is a data integrity failure, not just an edge case.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the transactional boundary must exist.

---

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

The thread is daemon=True so it won't block process exit, but `_stop_heartbeat` returns immediately without waiting for the thread to finish. If the thread is mid-sleep when the lock is released, there is a window where it wakes up and attempts to extend a lock that has already been deleted and potentially re-acquired by another process. The `finally` block should join the thread with a timeout.

```python
finally:
    self._stop_heartbeat()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # ... then release lock
```

---

### 6. Retry Backoff Has No Jitter

```python
backoff = (2 ** attempt) * 0.1
```

Under a flash sale load spike, all competing threads will retry at the same intervals (0.1s, 0.2s, 0.4s). This produces synchronized retry storms. Add jitter:

```python
import random
backoff = (2 ** attempt) * 0.1 * (0.5 + random.random())
```

## Test Coverage Gaps

### 7. Heartbeat Is Mocked Out Entirely in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is a non-trivial concurrent component and is the source of the most critical bugs identified above. It has zero test coverage. At minimum, add:

- A test verifying the heartbeat stops when the lock is released
- A test verifying the heartbeat does not extend a lock it no longer owns
- A test verifying concurrent calls to `reserve_item` on the same instance do not interfere with each other's heartbeat state

### 8. No Test for Negative or Zero Quantity

Given the missing input validation, there is no test asserting that `reserve_item("item_001", quantity=0)` or `quantity=-1` is rejected. These would currently pass the stock check and corrupt inventory.

---

## Minor Notes

- `LOCK_TTL = 30` is a class-level constant but should be configurable at construction time for different deployment environments. Hard-coded values in a shared service class are a maintenance liability.
- The `RuntimeError` used for lock failure is too broad. Define a specific `LockAcquisitionError(Exception)` so callers can handle it precisely without catching unrelated runtime errors.
- Consider whether `reserve_item` should propagate unexpected exceptions (e.g., Redis connection failure) rather than catching only `RuntimeError`. A Redis outage currently surfaces as "Service temporarily unavailable" with no distinction from lock contention, which will make incident diagnosis harder.
