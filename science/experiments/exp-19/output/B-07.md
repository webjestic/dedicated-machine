Decision: Request Changes

Summary: The heartbeat renewal mechanism creates a false sense of safety but does not eliminate the fundamental race condition: if the Python process is paused (GC stop-the-world, OS scheduling, container CPU throttling) long enough for the lock to expire *between* the last heartbeat extension and the database writes, a second process can acquire the lock and begin its own read-modify-write cycle before the first process resumes and executes its stale writes. There is no fencing token or optimistic concurrency check at the database layer, so both processes can commit conflicting writes and the second writer's changes will be silently overwritten. Additionally, the `InventoryService` instance carries heartbeat state as instance-level fields, making it unsafe for concurrent use across multiple threads sharing the same service object.

---

## Detailed Review

### Critical: No Fencing Token or Database-Level Optimistic Lock

This is the primary rejection reason.

The PR's defense against lock expiry is the heartbeat thread. The assumption is: *if the heartbeat is running, the lock is held, therefore the writes are safe.* This reasoning is flawed in a well-understood way documented by Martin Kleppmann and others in the distributed systems literature.

Consider this exact sequence:

```
T=0   Process A acquires lock, token="abc", TTL=30s
T=10  Heartbeat extends TTL to T=40s
T=11  Process A's thread is paused (GC, OS scheduler, cgroup CPU burst limit)
      The heartbeat thread is also paused — it is in the same process
T=41  Lock expires in Redis
T=42  Process B acquires lock, token="xyz"
T=43  Process B reads stock=5, decrements to 4, writes to DB
T=44  Process A resumes
T=45  Process A writes stock=4 (its stale read from T=11) to DB
      Process A creates a second order for the same units
T=46  Process B's write is now gone. Inventory is wrong.
```

The heartbeat does not protect against this. A paused process cannot extend its own lock. The heartbeat thread is a thread inside the paused process. Both are paused together.

The fix requires a fencing token at the write layer. The lock server (Redis) must issue a monotonically increasing token on each acquisition. That token must be passed through to every database write. The database must reject any write carrying a token older than the last committed token for that resource.

```python
# What the DB layer needs to enforce — not present anywhere in this PR

def update_stock(self, item_id: str, new_stock: int, fence: int) -> bool:
    """
    Only apply the write if the fencing token is >= the last seen token
    for this item. Reject stale writers atomically.
    """
    rows_affected = self.db.execute(
        """
        UPDATE inventory
        SET stock = %s, last_fence = %s
        WHERE item_id = %s AND last_fence < %s
        """,
        (new_stock, fence, item_id, fence)
    )
    return rows_affected == 1
```

Without this, the distributed lock is providing the *appearance* of safety while the actual write path has no guard.

### Critical: Instance-Level Heartbeat State Is Not Thread-Safe

```python
# services/inventory.py

self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance fields, not call-scoped locals. If two threads call `reserve_item` on the same `InventoryService` instance concurrently:

- Thread 1 sets `_heartbeat_active = True`, starts its heartbeat thread
- Thread 2 calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- Thread 1's heartbeat loop exits, the lock for Thread 1's operation stops being renewed

The lock management state must be scoped to the individual lock acquisition, not to the service instance. A simple fix is to return a stop-event from `_start_heartbeat` and pass it into `_stop_heartbeat`:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event

# In _acquire_lock:
stop_event = self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    stop_event.set()
    # ... release logic
```

### Moderate: Heartbeat Check-Then-Extend Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

There is a TOCTOU window between `GET` and `EXPIRE`. The lock can expire and be acquired by another process in that gap. The correct approach is a Lua script, which Redis executes atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.error("Heartbeat lost ownership of lock key=%s", lock_key)
    # signal the main thread that the lock is gone
```

The same issue exists in the lock release path, which the PR does handle with a get-then-delete pattern, but that also needs to be a Lua script or use the standard Redlock release script.

### Moderate: No Input Validation on `item_id` or `quantity`

The PR claims input validation via Pydantic, but `reserve_item` accepts raw `str` and `int` parameters with no validation applied. Pydantic is only used for the return type. There is no guard against:

- `quantity <= 0` (reserving zero or negative units would pass the stock check and corrupt inventory)
- `item_id` being an empty string or containing injection characters if the DB layer is not parameterized

```python
# Minimum required
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item ID")
```

### Minor: Test Coverage Gaps and a Latent Bug in Tests

**`test_reserve_lock_contention` has a typo that masks a real assertion:**

```python
db.update_stack.assert_not_called()  # 'stack' not 'stock'
```

`update_stack` does not exist on the mock. `MagicMock` will silently create it and the assertion will pass vacuously. This test is not actually verifying that `update_stock` was not called.

**The heartbeat is mocked out in every test:**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat thread is never exercised by the test suite. The TOCTOU bug in the heartbeat, the instance-level state collision, and the lock extension failure path are all untested.

**There is no test for the scenario where `update_stock` succeeds but `create_order` raises an exception.** In that case, stock is decremented but no order is created. The code has no transaction rollback. This is a data consistency gap independent of the locking issue.
