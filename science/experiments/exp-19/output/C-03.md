Decision: Request Changes

Summary: The core locking logic and heartbeat concept are sound, but there are several architectural flaws that make this unsafe for production under concurrent load. The most critical issue is that `_heartbeat_active`, `_heartbeat_thread`, and related state are stored as instance-level attributes, meaning a single shared `InventoryService` instance — the typical deployment pattern — will have its heartbeat state corrupted across concurrent calls. Secondary concerns include a non-atomic lock release, missing input validation, and test coverage gaps that mask the concurrency bug entirely.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

This is the most serious flaw. The heartbeat state lives on `self`:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` simultaneously on the same service instance, the second call's `_stop_heartbeat()` will set `self._heartbeat_active = False`, killing the first call's heartbeat thread. This can cause the first call's lock to expire mid-operation — exactly the failure mode the heartbeat was designed to prevent. Under flash sale load, this will happen constantly.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state through a local object or closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # ...
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # release lock...
```

### 2. Non-Atomic Lock Release (TOCTOU Race)

The lock release is two separate Redis operations:

```python
current = self.redis.get(lock_key)          # operation 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # operation 2
```

Between `GET` and `DELETE`, another process could acquire the lock after TTL expiry. You would then delete a lock you no longer own. This must be atomic. Use a Lua script:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

This is a well-known requirement for correct distributed locking and is not optional.

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw, unvalidated arguments. There is no guard against:

- Negative or zero `quantity`
- Empty or malformed `item_id` (which directly interpolates into the Redis key `inventory:lock:{item_id}`, creating a key injection risk)
- Excessively large `quantity` values

```python
# Minimum required
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

A Pydantic model for the request would be the cleaner solution and would match what the checklist claims.

## Moderate Issues

### 4. Heartbeat Thread Is Never Joined

`_stop_heartbeat` only sets a flag. The thread is never joined:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is abandoned
```

Under high load this leaks threads. The context manager's `finally` block should join the thread with a bounded timeout before releasing the lock, otherwise the heartbeat could extend a lock that has already been logically released.

---

### 5. DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order record. This is a data integrity issue. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` from anywhere in the lock body — including unexpected failures in `db.update_stock` — will be silently swallowed and returned as "unavailable." Use a dedicated exception type:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError in reserve_item
```

## Test Coverage Gaps

### 7. Tests Mock Away the Bug

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking both heartbeat methods means the concurrency bug in issue #1 is invisible to the test suite. There should be at least one integration-level test that exercises the real heartbeat path.

### 8. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the scenario that would expose the missing transaction boundary.

### 9. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test validates that `delete` is not called when the token mismatches, but because `_start_heartbeat` and `_stop_heartbeat` are mocked, it does not actually exercise the real `finally` block path through `_acquire_lock`. The test passes for the wrong reasons.
