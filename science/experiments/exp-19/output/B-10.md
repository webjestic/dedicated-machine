Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not close the fundamental race condition this code is designed to prevent. The database writes at the critical section have no fencing token or optimistic concurrency check, meaning a process pause between heartbeats — or a heartbeat failure — allows stale writes to proceed after lock expiry. Additionally, there are several secondary issues in the implementation and test suite that must be addressed before this is production-ready for a flash sale workload.

---

## Critical: Fencing Token / Optimistic Lock Gap

This is the primary rejection reason.

The heartbeat thread extends the lock TTL, but it does not make the lock *safe*. Consider this sequence:

1. Process A acquires the lock and reads `current_stock = 5`.
2. All threads in Process A are paused — GC stop-the-world, OS scheduler preemption, container CPU throttle, etc.
3. The heartbeat thread is also paused. The lock TTL expires in Redis.
4. Process B acquires the lock, reads `current_stock = 5`, decrements to 4, writes successfully.
5. Process A resumes. Its heartbeat fires, sees the lock is now held by B's token, and does nothing. But the application thread in A does **not** check the lock again before writing.
6. Process A calls `self.db.update_stock(item_id, 4)` and `self.db.create_order(...)` — writing stale data over B's committed state.

```python
# This is the unsafe sequence — no guard between lock expiry and write
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # read happens here
    # ... pause of arbitrary length possible here ...
    self.db.update_stock(item_id, new_stock)      # write proceeds with stale data
    self.db.create_order(order_id, item_id, quantity)
```

The heartbeat only helps when the process is *running*. It provides zero protection during a pause. The lock and the database write must be coupled with a fencing mechanism.

**Required fix — choose one or both layers:**

**Option A — Fencing token passed to the database write:**
```python
# The lock token becomes a fencing token. The DB layer rejects writes
# from a token that is no longer current.
self.db.update_stock(item_id, new_stock, fence=token)
self.db.create_order(order_id, item_id, quantity, fence=token)
```
This requires the database layer to store and compare the last-seen token, rejecting any write whose token is older than the current one. This is the approach described in the Kleppmann distributed locks analysis and is the correct solution when the lock is advisory.

**Option B — Optimistic concurrency control at the DB write level:**
```python
# Read a version counter or expected stock value at read time,
# and use a conditional update that fails if the value has changed.
rows_affected = self.db.update_stock_if_version(
    item_id,
    new_stock=new_stock,
    expected_version=stock_version  # obtained alongside current_stock
)
if rows_affected == 0:
    raise StaleDataError("Stock was modified by another process")
```
SQL equivalent:
```sql
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id AND version = :expected_version
```
If `rows_affected == 0`, the reservation must be retried or rejected — not silently committed.

Without one of these, the distributed lock is decorative. It reduces the probability of a collision under normal load but does not prevent the data integrity failure under pause or partial failure conditions.

## Secondary Issue: Heartbeat Has a Non-Atomic Check-Then-Extend Race

```python
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2), the lock can expire and be acquired by another process. The heartbeat then extends the TTL of the *new owner's* lock, silently granting the original holder more time it no longer deserves. This should be a single atomic operation via a Lua script:

```lua
-- Atomic check-and-extend
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    script = self.redis.register_script(HEARTBEAT_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if self._heartbeat_active:
                result = script(keys=[lock_key], args=[token, self.LOCK_TTL])
                if result == 0:
                    logger.error(
                        "Heartbeat lost lock ownership key=%s — lock expired or stolen",
                        lock_key
                    )
                    self._heartbeat_active = False
                    # Signal the main thread that the lock is gone
```

Note the additional benefit here: when the heartbeat detects it has lost the lock, it should signal the main thread to abort rather than continuing toward the write. This is the correct integration point for a cooperative cancellation mechanism.

## Secondary Issue: Heartbeat Loss Is Silently Ignored

When the heartbeat detects `current.decode() != token`, it logs a debug message and exits the loop. The main application thread is never notified. It continues executing, reads stock, and writes to the database — all after the lock has been confirmed lost. The heartbeat thread and the application thread need a shared cancellation signal:

```python
self._lock_valid = threading.Event()
self._lock_valid.set()

# In heartbeat: self._lock_valid.clear() on loss

# In reserve_item, before each DB operation:
if not self._lock_valid.is_set():
    raise LockExpiredError("Lock was lost during operation")
```

---

## Secondary Issue: Instance-Level Heartbeat State Is Not Thread-Safe

`_heartbeat_active` and `_heartbeat_thread` are instance variables on `InventoryService`. If two threads call `reserve_item` concurrently on the same service instance, they will overwrite each other's heartbeat state. The heartbeat management must be scoped to the individual lock acquisition, not the service instance. Pass state through closure or a dedicated lock-context object.

---

## Secondary Issue: Lock Release Is Also Non-Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same check-then-act race as the heartbeat. Use the standard Lua script for safe release:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

## Test Suite Issues

**`test_reserve_lock_contention` has a typo that masks a real assertion:**
```python
db.update_stack.assert_not_called()  # 'stack' is not an attribute — this always passes
```
Should be `db.update_stock.assert_not_called()`. This test is currently providing false confidence.

**The heartbeat is mocked out in every meaningful test:**
```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```
This means the heartbeat logic — including the race condition described above — is completely untested. At minimum, add integration-style tests that exercise the real heartbeat path with a mock Redis that simulates lock expiry mid-operation.

**`test_lock_not_released_if_expired` does not test the scenario that matters:**
The test confirms that `redis.delete` is not called when the token has changed. It does not test whether the *database write* is suppressed when the lock is lost. That is the scenario that causes data corruption, and it is not covered anywhere in the test suite.

**No test for the concurrent-instance heartbeat collision described above.** Two simultaneous `reserve_item` calls on the same instance should be a test case.

## Minor Notes

- `quantity` has no validation guard against zero or negative values. `quantity=0` would succeed and create a zero-quantity order. Add a guard at the top of `reserve_item`.
- The `LOCK_TTL = 30` comment says "generous TTL for slow DB operations" — this framing is backwards. A longer TTL increases the window during which a paused process can hold a phantom lock. TTL should be as short as safely possible, with the heartbeat providing extension only while the process is confirmed alive.
- Consider whether `RuntimeError` is the right exception type for lock acquisition failure. A domain-specific exception class makes it easier for callers to distinguish lock contention from programming errors.
