Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness and safety issues that must be addressed before this can be approved, ranging from a race condition in the heartbeat logic to a critical concurrency flaw that undermines the entire locking strategy. The test suite also has gaps that mask these defects rather than catching them.

---

## Findings

### 🔴 Critical — Heartbeat Race Condition (Check-Then-Act)

**File:** `services/inventory.py`, `_start_heartbeat` → `_beat()`

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be a single atomic operation.

**Fix:** Use a Lua script to make this atomic:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register on __init__
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In _beat()
extended = self._extend_script(
    keys=[lock_key],
    args=[token, self.LOCK_TTL]
)
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same atomic pattern already used in `_acquire_lock`'s release path should be applied consistently here.

### 🔴 Critical — Instance-Level Heartbeat State Breaks Concurrency

**File:** `services/inventory.py`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If `InventoryService` is shared across threads (which is the normal deployment model for a service object), two concurrent calls to `reserve_item` will stomp on each other's heartbeat state. The second caller's `_stop_heartbeat()` will kill the first caller's heartbeat thread.

**Fix:** Move heartbeat state into `_acquire_lock` as local closure variables, scoped to the lock context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            extended = self._extend_script(
                keys=[lock_key], args=[token, self.LOCK_TTL]
            )
            if not extended:
                logger.warning("Lost lock ownership key=%s", lock_key)
                heartbeat_active.clear()
                break

    # ... acquire lock, start thread with _beat, yield, finally clear event
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which is less responsive to shutdown signals.

### 🟡 Medium — `db.update_stock` and `db.create_order` Are Not Atomic

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. The distributed lock does not protect against this — it only serializes access, it does not provide rollback.

**Fix:** These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the requirement is non-negotiable for inventory integrity.

### 🟡 Medium — Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

Nothing prevents `item_id=""`, `quantity=0`, `quantity=-5`, or a non-string `item_id` from reaching the lock and database layers.

**Fix:** Introduce a request model and validate at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

The `item_id` pattern constraint also prevents a potential Redis key injection where a crafted `item_id` like `inventory:lock:foo:bar` could collide with other key namespaces.

### 🟡 Medium — Bare `Exception` Not Caught; DB Errors Leak

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate as an unhandled exception to the caller. For a flash sale endpoint under load, this likely means a 500 to the user and a partially-applied state if the transaction fix above is not also applied.

**Fix:**

```python
except (RuntimeError, DatabaseError, redis.RedisError) as e:
    logger.error("Reservation failed item_id=%s: %s", item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

Use specific exception types. Do not use bare `except Exception` as a substitute for proper error handling design.

---

### 🟡 Medium — `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

Under high flash sale concurrency, many threads will be sleeping here simultaneously. This is acceptable if the service is async-aware, but if this runs in a synchronous WSGI context (e.g., gunicorn), sleeping threads consume worker slots and will cause request queuing to cascade.

This is an architectural concern that needs to be documented at minimum, and ideally addressed with an async-compatible lock implementation if the stack supports it.

### 🔵 Low — Test Suite Mocks Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # ← stop never runs
    return svc
```

The heartbeat logic — which contains the race condition identified above — is never exercised by the test suite. The tests validate the reservation business logic, which is good, but they provide false confidence about the locking subsystem.

**Fix:** Add integration-level tests using `fakeredis` that exercise the full lock lifecycle including heartbeat behavior:

```python
import fakeredis

def test_heartbeat_extends_ttl():
    r = fakeredis.FakeRedis()
    svc = InventoryService(r, mock_db)
    lock_key = "inventory:lock:test_item"
    token = "known-token"
    r.set(lock_key, token, ex=30)
    svc._start_heartbeat(lock_key, token)
    time.sleep(0.5)
    ttl = r.ttl(lock_key)
    svc._stop_heartbeat()
    assert ttl > 25  # TTL was refreshed
```

### 🔵 Low — Daemon Thread Lifecycle Is Not Documented

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Using `daemon=True` means the heartbeat thread will be killed abruptly on process exit without releasing the lock. This is acceptable for most cases since Redis TTL will eventually expire the lock, but it should be explicitly documented as a known behavior, and the TTL should be sized accordingly.
