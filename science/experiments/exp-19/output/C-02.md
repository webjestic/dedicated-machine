Decision: Request Changes

Summary: The implementation has a critical concurrency flaw ‚Äî the `InventoryService` instance stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance-level attributes, meaning a single shared service instance cannot safely handle concurrent reservations; a second call will stomp the heartbeat state of an in-flight first call. There are also meaningful gaps in atomicity guarantees, input validation, and test fidelity that need to be addressed before this is safe to merge into a government-facing production system.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Breaks Concurrency (High Severity)

This is the most dangerous flaw. The heartbeat is tracked on `self`:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item()` on the same `InventoryService` instance concurrently (which is the entire point of this service), the second call's `_start_heartbeat` will overwrite `_heartbeat_active`, and the first call's `_stop_heartbeat` will kill the second call's heartbeat ‚Äî or vice versa. This is a silent race condition that will cause lock expiry under the exact load this code is designed to handle.

**Fix:** Scope heartbeat state to the lock acquisition context, not the instance. Pass state via a local object or use a per-call flag:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

Note also that `_stop_heartbeat` does not `join()` the thread. The heartbeat thread can continue running briefly after the lock is released, potentially re-extending a lock that was just deleted. The `join()` call is mandatory.

### 2. Non-Atomic Check-Then-Act on DB Operations (Medium-High Severity)

The lock protects the Redis layer, but `db.update_stock` and `db.create_order` are two separate calls with no transaction boundary shown:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes, is killed, or the DB throws between these two calls, stock is decremented but no order is created ‚Äî inventory is silently lost. For a government contracting system handling flash sale inventory, this is an auditable data integrity failure.

**Fix:** These must execute in a single DB transaction. The interface contract for `db` needs to expose transactional semantics:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 3. Input Validation Is Declared But Not Implemented

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserving zero or negative items would pass the stock check)
- `item_id` being an empty string or containing injection-relevant characters
- `quantity` being unreasonably large

**Fix:** Introduce a request model and validate at the boundary:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Moderate Issues

### 4. Heartbeat Has a TOCTOU Race on Token Verification

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The GET and EXPIRE are not atomic. Between the GET and the EXPIRE, the lock can expire and be acquired by another process. The EXPIRE then extends a lock owned by someone else. This should use a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to lock release in the `finally` block (GET then DELETE should be a single atomic Lua script).

---

### 5. `RuntimeError` Is Too Broad an Exception Boundary

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` raised inside the `with self._acquire_lock` block ‚Äî including from `db.update_stock` or `db.create_order` ‚Äî will be silently swallowed and returned as "service unavailable." A failed DB write will look identical to a lock timeout. Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass
```

And raise/catch only that.

## Test Coverage Gaps

### 6. Tests Mock Away the Exact Behavior Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This is applied in the fixture for all tests. The heartbeat ‚Äî which is the primary novel mechanism in this PR ‚Äî is never actually exercised. The concurrency bug in issue #1 would not be caught by any test here.

**Required additions:**
- A test that calls `reserve_item` concurrently from multiple threads and asserts exactly one succeeds
- An integration-level test that exercises the actual heartbeat path
- A test verifying that a DB failure between `update_stock` and `create_order` does not corrupt stock state

### 7. Lock Expiry Test Asserts the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts that `delete` is not called when the token doesn't match, which is correct behavior. However, it does not assert what the `result` was, and it does not test that the reservation itself failed or succeeded in a defined way under this condition. The test is incomplete.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at 10s, leaving a 20s window before the first renewal. Under a slow DB, this is fine, but the comment "generous TTL" should be a documented operational parameter, not a magic number ‚Äî especially for a system that may face audit requirements.
- The `db` interface has no type annotation or abstract base class defined. For a service of this sensitivity, the contract should be explicit and mockable by interface, not by duck typing.
