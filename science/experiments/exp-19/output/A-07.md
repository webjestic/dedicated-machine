**Decision: Request Changes**

**Summary:** The heartbeat thread reduces the *probability* of lock expiry but does not eliminate it — a full process pause (GC stop-the-world, OOM freeze, VM live migration, kernel preemption) can suspend all threads simultaneously, including the heartbeat, for longer than the remaining TTL, after which the lock expires, another worker acquires it, and this worker resumes and executes `db.update_stock` and `db.create_order` against stock that has already been modified. There is no fencing token or optimistic/versioned check at the database write level, so the lock does not actually protect the write. Additionally, the service carries a concurrency design flaw where a single `InventoryService` instance cannot safely serve concurrent requests. These are not edge cases — they are the exact failure modes that distributed locks without database-level guards are known to produce in production.

## Critical Issues

### 1. No Fencing Token or Optimistic Lock at the Database Write Layer

This is the central defect. The entire correctness argument rests on the assumption that the Redis lock is held continuously from stock read through both writes. That assumption is not guaranteed by any mechanism this code controls.

**The failure scenario, step by step:**

```
Worker A acquires lock, reads stock = 5
Worker A's process is paused (GC, VM migration, kernel preemption)
  → All threads pause, including the heartbeat thread
  → 30 seconds elapse
  → Redis expires the lock
Worker B acquires the lock, reads stock = 5, writes stock = 4, commits order
Worker A resumes
  → Heartbeat thread wakes, checks token — token is gone, heartbeat goes silent
  → But reserve_item() is already past the lock check, inside the critical section
  → Worker A writes stock = 4 (again), creates a duplicate order
```

The heartbeat thread cannot prevent this. The process pause suspends the heartbeat along with everything else. When Worker A resumes, the code inside `with self._acquire_lock(lock_key):` continues executing unconditionally. There is no re-validation of lock ownership before the writes.

**What is required** is a database-level guard that makes the write conditional. Two standard approaches:

**Option A — Optimistic locking with a version/generation counter:**

```python
# Schema carries: stock INT, version BIGINT
# Read returns both values
current_stock, version = self.db.get_stock_with_version(item_id)

# Write is conditional — fails if version has changed since read
rows_affected = self.db.update_stock_if_version(
    item_id,
    new_stock=current_stock - quantity,
    expected_version=version
)
if rows_affected == 0:
    # Another worker modified stock between our read and write
    raise StaleDataError(f"Stock version conflict for {item_id}")
```

```sql
-- The actual SQL that makes this safe
UPDATE inventory
SET stock = :new_stock, version = version + 1
WHERE item_id = :item_id
  AND version = :expected_version;
-- Zero rows affected = someone else wrote first
```

**Option B — Monotonic fencing token passed through to the database:**

```python
# Lock token becomes the fencing token — must be threaded into the DB write
with self._acquire_lock(lock_key) as token:
    self.db.update_stock_with_fence(item_id, new_stock, fence=token)

# DB layer rejects writes carrying a token older than the last committed token
```

Option A is more practical with a standard RDBMS. Either way, the write must be able to detect and reject a stale operation at the storage layer, independent of what Redis says.

### 2. Single Instance Cannot Serve Concurrent Requests

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to a lock acquisition. If two requests call `reserve_item` concurrently on the same `InventoryService` instance (the normal case in any web framework with a shared service layer):

```python
# Request A calls _start_heartbeat → sets self._heartbeat_active = True
# Request B calls _start_heartbeat → overwrites self._heartbeat_thread
# Request A's finally block calls _stop_heartbeat → self._heartbeat_active = False
# Request B's heartbeat thread is now orphaned with _heartbeat_active = False
# Request B's lock will not be renewed
```

The heartbeat state must be local to each lock acquisition context, not shared across the instance.

**Fix — scope heartbeat state to the acquisition:**

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # local to this acquisition

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        time.sleep((2 ** attempt) * 0.1)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key}")

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        # Atomic check-and-delete via Lua
        self._release_lock(lock_key, token)
```

### 3. Lock Release Is Not Atomic

The release sequence is:

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:   # check
    self.redis.delete(lock_key)             # delete
```

Between the `GET` and the `DELETE`, another worker can acquire the lock. This is a classic TOCTOU race. The lock release must be a single atomic operation.

**Fix — standard Lua script for atomic check-and-delete:**

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(self.RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if not released:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU issue exists in the heartbeat's check-then-extend sequence, though the consequence there is less severe (a missed renewal rather than a deleted lock you no longer own).

### 4. `quantity` Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check and creates an order for zero units. `quantity=-5` increases stock. Neither is caught. The Pydantic model is on the result, not the input.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 5. Test Suite Does Not Test the Actual Failure Mode

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat is mocked out entirely
    svc._stop_heartbeat = MagicMock()
    return svc
```

The test named `test_lock_not_released_if_expired` verifies that `redis.delete` is not called when the token mismatches. That is correct behavior for the release path. But it does not test the scenario that matters: that `db.update_stock` is **not called** when the lock has expired before the write executes. That scenario is currently untestable because there is no guard to test. Once the database-level version check is added, the test suite must include:

```python
def test_stale_write_rejected_after_lock_expiry(service, db):
    """
    Simulate Worker A resuming after lock expiry and another worker
    has already modified the stock version.
    db.update_stock_if_version returns 0 rows affected.
    Reservation must fail, not silently double-write.
    """
    db.update_stock_if_version.return_value = 0  # version conflict
    result = service.reserve_item("item_001", quantity=1)
    assert result.success is False
    db.create_order.assert_not_called()
```

## Summary of Required Changes

| # | Issue | Severity |
|---|-------|----------|
| 1 | No database-level fencing token or optimistic version check | **Blocking** |
| 2 | Heartbeat state is instance-level, not acquisition-scoped | **Blocking** |
| 3 | Lock release is not atomic (TOCTOU on GET + DELETE) | **Blocking** |
| 4 | `quantity` parameter accepts zero and negative values | **Blocking** |
| 5 | Test suite mocks out heartbeat and does not test stale-write rejection | **Blocking** |

The Redis lock as written is a performance optimization and a contention reducer. It is not a correctness guarantee. For correctness under the full failure model, the database write must be the authority.
