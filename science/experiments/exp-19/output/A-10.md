**Decision: Request Changes**

**Summary:** The heartbeat thread addresses clock-based expiry under *gradual* slowdown, but the implementation contains no fencing token or optimistic lock check at the database write layer, meaning a process pause long enough to outlast even the heartbeat leaves `db.update_stock` and `db.create_order` completely unguarded. Beyond that critical architectural gap, there are several additional correctness and safety defects that must be resolved before this can be approved.

---

## Primary Defect: No Fencing at the Database Write Layer

This is the defect that makes the lock not actually protect the write.

The scenario: the process acquires the lock, the heartbeat starts, then the OS pauses *all threads in the process simultaneously* — GC stop-the-world, hypervisor preemption, `SIGSTOP`, kernel scheduler starvation. The heartbeat thread is also paused. The lock TTL expires. Another process acquires the lock and completes its reservation. The original process resumes. The heartbeat may or may not have time to re-extend before the write executes, but that is a race — and even if the heartbeat does re-extend, it does so *after* the lock was lost and re-acquired by someone else, meaning the token check in the heartbeat is now comparing against a *new* token it does not own.

The result: two processes both execute `db.update_stock` and `db.create_order` against the same item concurrently. Stock goes negative. Two orders are created for inventory that does not exist.

The lock provides *mutual exclusion in Redis*. It provides nothing at the database layer. The database layer must protect itself.

**Required fix:** The database write must include a condition that makes it fail if the world has changed since the read. Two acceptable approaches:

```python
# Option A: Optimistic concurrency — write only if stock matches what was read
rows_affected = self.db.update_stock_if_matches(
    item_id,
    expected_stock=current_stock,
    new_stock=new_stock
)
if rows_affected == 0:
    raise ConcurrentModificationError("Stock changed during reservation")

# Option B: Fencing token passed through to the database write
# Requires the DB layer to track and reject stale tokens — heavier but
# correct even against split-brain across multiple Redis nodes
self.db.update_stock(item_id, new_stock, fence=token)
```

Option A is the minimum acceptable bar. Without one of these, the distributed lock is a performance optimization, not a correctness guarantee.

## Secondary Defect: Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not call-scoped variables. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — two concurrent calls will stomp each other's `_heartbeat_active` flag. Call A sets it `True`, Call B sets it `True`, Call A finishes and sets it `False`, which stops Call B's heartbeat while Call B is still holding its lock.

**Required fix:** Heartbeat state must be local to each lock acquisition, not stored on `self`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    ...
    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            ...
    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

## Tertiary Defect: Heartbeat Check-Then-Extend Is a Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a non-atomic check-then-act. Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The `EXPIRE` then extends a lock now owned by someone else.

**Required fix:** Use a Lua script to make the check-and-extend atomic.

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same race exists in the lock release path and must be fixed there too — the existing `GET` + `DELETE` should already be a Lua script or use `redis.delete` only after an atomic compare.

## Additional Defects

**`db.update_stock` and `db.create_order` are not atomic with each other.** If `update_stock` succeeds and `create_order` throws, stock is decremented but no order exists. This requires either a database transaction wrapping both calls, or a compensating rollback on failure.

```python
# Minimum acceptable:
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

**`quantity` has no validation.** A caller passing `quantity=0` or `quantity=-5` will silently succeed. The stock check `current_stock < quantity` passes for zero or negative quantities, and `update_stock` will be called with a value equal to or greater than current stock.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

**`item_id` is passed directly to database calls without sanitization.** Pydantic is used for `ReservationResult` but not for validating the inputs to `reserve_item`. If `db.get_stock` constructs a query from `item_id`, this is a potential injection vector depending on the DB layer.

---

## Test Suite Defects

**The tests mock away the exact behavior under test.** The fixture does this:

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means no test exercises the heartbeat at all. The heartbeat's race condition and the instance-level state collision are both invisible to the test suite. Tests must exist that exercise the actual heartbeat thread, including the scenario where the token has changed by the time `expire` is called.

**`test_lock_not_released_if_expired` does not test the scenario it names.** It sets `redis.get` to return a different token, but because `_start_heartbeat` is mocked, the heartbeat never ran, and the token in the lock context is a real UUID that will never match `b"different-token"`. The test passes for the wrong reason and does not cover the actual expiry scenario.

**There is no test for concurrent calls on the same instance**, which is the scenario that exposes the instance-level heartbeat state collision.
