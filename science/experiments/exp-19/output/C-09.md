Decision: Request Changes

Summary: The implementation demonstrates solid intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness, concurrency safety, and architectural issues that must be addressed before this is suitable for a production government contracting environment where inventory integrity is a hard requirement.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, not scoped to individual lock acquisitions.

```python
# CURRENT — broken under concurrent calls on the same service instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the **same `InventoryService` instance** simultaneously (which is the entire point of a flash sale scenario), one call's `_stop_heartbeat()` will kill the other call's heartbeat thread. This can cause the lock to expire mid-operation, leading to a double-reservation and inventory going negative.

**Fix:** Scope heartbeat state to the lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                break  # Lock was lost; stop beating

    # ... acquire lock ...

    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 2. Heartbeat Check-Then-Extend Is a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # Lock could expire between these two calls
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The extend must be atomic.

**Fix:** Use a Lua script for atomic check-and-extend.

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once, e.g., in __init__
self._extend_lock = self.redis.register_script(EXTEND_SCRIPT)

# In heartbeat
extended = self._extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
    heartbeat_active.clear()
```

---

### 3. Lock Release Is Also a TOCTOU Race

The same pattern appears in the `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # Another process could acquire between GET and DELETE
```

**Fix:** Use the canonical Lua script for atomic compare-and-delete.

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

This is a well-known pattern documented in the Redis distributed locks specification. The current implementation is incorrect.

### 4. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but neither `item_id` nor `quantity` is validated before use. `item_id` is interpolated directly into a Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar:baz"` could manipulate the key namespace. A caller passing `quantity = -5` would *increase* stock silently.

**Fix:**

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(pattern=r'^[a-zA-Z0-9_-]{1,64}$')
    quantity: int = Field(default=1, ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

### 5. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists. This is a data integrity failure — inventory is lost with no audit trail.

**Fix:** These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` abstraction must expose a transaction context manager. If it does not, that is an architectural gap that needs to be resolved before this service can be merged.

## Moderate Issues

### 6. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish. If the thread is mid-sleep when the flag is cleared, it may attempt to extend a lock that has already been released, potentially extending a lock now owned by another process. The `join()` call in the fix above addresses this.

---

### 7. `time.sleep` in Retry Loop Blocks the Caller Thread

```python
time.sleep(backoff)
```

In an async or thread-pool-based web server, blocking the thread during backoff wastes a worker. This is acceptable if the service is explicitly synchronous, but it should be documented, and the maximum total wait time (0.1 + 0.2 + 0.4 = 0.7s minimum, longer with jitter) should be surfaced in configuration.

**Recommendation:** Add jitter to prevent thundering herd.

```python
import random
backoff = (2 ** attempt) * 0.1 + random.uniform(0, 0.05)
```

---

### 8. Test Suite Mocks Out the Components Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — which is the most novel and risky part of this implementation — is entirely bypassed in every test. The TOCTOU race in the lock release is also not exercised because `redis_client.get` is hardcoded to return `b"test-token"` unconditionally.

The tests verify the business logic path, which is fine, but there is zero coverage of the locking mechanism itself. At minimum, add integration-style tests using `fakeredis` that exercise the actual lock acquisition, heartbeat extension, and release paths.

---

## Minor Issues

### 9. Logger Is Module-Level but Service Is Instantiated

No issue with the logger itself, but log lines should include a correlation ID or request ID if this service is called in a web request context. Consider accepting an optional `request_id` parameter to thread through log statements.

### 10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Class-Level Constants

These should be constructor parameters with defaults, so they can be tuned per deployment without subclassing.

```python
def __init__(self, redis_client, db, lock_ttl: int = 30, heartbeat_interval: int = 10):
```
