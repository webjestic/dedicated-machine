Decision: Request Changes

Summary: The heartbeat mechanism creates a false sense of safety but does not eliminate the core race condition: if the process is paused (GC, OS scheduling, VM migration, swap) *after* the heartbeat last extended the TTL but *before* the DB writes complete, the lock can expire, a second process can acquire it and begin its own writes, and then the first process resumes and overwrites with stale data. There is no fencing token or optimistic lock version check at the database write layer, so the distributed lock is the only guard — and distributed locks alone cannot provide mutual exclusion guarantees when processes can pause arbitrarily. The test suite also contains a silent bug that masks a failure path, and the service has a concurrency defect in its heartbeat state management that makes it unsafe for concurrent use.

---

## Detailed Findings

### 1. Critical — No Fencing Token at the Write Layer (Data Integrity Failure)

This is the primary rejection reason.

The PR description correctly identifies the problem it is trying to solve, but the heartbeat does not solve it. Consider this exact sequence:

```
Process A acquires lock, heartbeat is running
Process A calls db.get_stock()  → reads stock = 5
  *** OS pauses Process A for 31+ seconds (GC, swap, VM live migration) ***
  *** Heartbeat thread is also paused — it cannot extend the TTL ***
  *** Lock TTL expires ***
Process B acquires the same lock
Process B reads stock = 5, decrements to 4, writes to DB
Process B releases lock
  *** OS resumes Process A ***
Process A continues from where it left off
Process A writes stock = 4 (its own stale decrement), overwriting Process B's write
Process A calls db.create_order() — a duplicate or phantom order now exists
```

The heartbeat thread runs inside the same process. When the OS or JVM pauses the process, **all threads pause**, including the heartbeat. The lock expires. Another process acquires it. When the original process resumes, it has no way of knowing the lock expired, because the `_heartbeat_active` flag is still `True` from its own perspective, and the code does not re-check lock ownership before executing the writes.

The only correct fix is a **fencing token** passed through to the database write operations. Redis's Redlock documentation and Martin Kleppmann's analysis of distributed locks both describe this requirement explicitly. The lock must increment a monotonic counter (or the DB layer must use an optimistic version check) so that a stale write from a previously-paused process is rejected by the database, not just by the lock manager.

```python
# What the DB layer needs to look like
def update_stock(self, item_id: str, new_stock: int, fence: int) -> bool:
    # Only write if the current fence token in the DB matches
    rows_affected = self.db.execute(
        "UPDATE inventory SET stock = %s, fence = %s "
        "WHERE item_id = %s AND fence = %s",
        (new_stock, fence + 1, item_id, fence)
    )
    return rows_affected == 1
```

Without this, the distributed lock provides only a probabilistic best-effort coordination hint, not a correctness guarantee. For a flash sale inventory system — where overselling has direct financial and contractual consequences — probabilistic is not acceptable.

### 2. Critical — Heartbeat State Is Instance-Level, Not Call-Level (Concurrency Defect)

```python
# In InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not scoped to a single `reserve_item` call. If the same `InventoryService` instance is used by two concurrent threads (which is the normal case in any WSGI/ASGI server), the following race exists:

```
Thread 1 calls _start_heartbeat → sets self._heartbeat_active = True
Thread 2 calls _stop_heartbeat  → sets self._heartbeat_active = False
Thread 1's heartbeat loop exits immediately, lock is no longer renewed
Thread 1's lock expires while Thread 1 is still inside the critical section
```

The heartbeat state must be encapsulated per-call, not per-instance. A simple fix is to pass a threading.Event into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ... acquire lock ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 3. High — `update_stock` and `create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction wrapping. If the process crashes, is killed, or the DB connection drops between these two lines, the stock is decremented but no order record exists. The item is effectively lost. This must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock, fence=current_fence)
    self.db.create_order(order_id, item_id, quantity)
```

### 4. Medium — Silent Test Bug Masks a Failure

```python
def test_reserve_lock_contention(redis_client, db):
    redis_client.set.return_value = False
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    result = svc.reserve_item("item_001")
    assert result.success is False
    assert "unavailable" in result.message
    db.update_stack.assert_not_called()  # <-- typo: 'update_stack' not 'update_stock'
```

`db.update_stack` does not exist on the mock. `MagicMock` will silently create it and return a new mock rather than raising `AttributeError`. This assertion always passes regardless of whether `update_stock` was called. The test is not testing what it claims to test.

```python
# Correct
db.update_stock.assert_not_called()
```

### 5. Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish its current `time.sleep(HEARTBEAT_INTERVAL)`. The thread can still be alive and holding a reference to `lock_key` and `token` when the lock has already been released and potentially re-acquired by another process. The `finally` block should join the thread before releasing the lock:

```python
finally:
    stop_event.set()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # now safe to release the lock
```

---

### 6. Low — `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`. A caller passing `quantity=0` would succeed and create an order for zero items. A caller passing `quantity=-5` would increase stock. The Pydantic model is on the return type, not the input. Add explicit validation:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

## What Needs to Change Before Approval

| # | Severity | Item |
|---|----------|------|
| 1 | Critical | Add fencing token or DB-level optimistic lock version check on all writes |
| 2 | Critical | Move heartbeat state to per-call scope, not per-instance |
| 3 | High | Wrap `update_stock` + `create_order` in a single DB transaction |
| 4 | Medium | Fix `update_stack` typo in test; verify assertion actually fires |
| 5 | Medium | Join heartbeat thread before releasing the lock |
| 6 | Low | Validate `quantity > 0` at entry |

Items 1 and 2 are blocking. The service cannot be deployed to a production flash sale environment without them.
