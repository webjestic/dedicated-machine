# Production Rate Limiter in Node.js

Let me design this completely before writing a line of code.

---

## Phase 1: Requirements Design

### What problem are we actually solving?

A rate limiter protects a system from being overwhelmed — by abuse, by bugs in clients, or by legitimate traffic spikes. Before choosing an algorithm, I need to nail down the requirements.

### Core Design Decisions

**1. What are we limiting by?**
- IP address (unauthenticated traffic)
- API key / user ID (authenticated traffic)
- Endpoint (some routes cost more than others)
- Combination of the above

**2. What algorithm?**

| Algorithm | Pros | Cons | Best For |
|---|---|---|---|
| Fixed Window | Simple, fast | Burst at window edges (2x traffic possible) | Coarse protection |
| Sliding Window Log | Precise | Memory-heavy (stores every request timestamp) | Low-volume, high-precision |
| Sliding Window Counter | Precise, memory-efficient | Approximate (weighted average) | General purpose |
| Token Bucket | Allows controlled bursting | State is harder to reason about | APIs that allow bursting |
| Leaky Bucket | Smooth output rate | Queuing complexity, latency | Egress rate smoothing |

**Decision:** Sliding Window Counter. It's precise enough, memory-efficient, and the most defensible choice for a general-purpose API rate limiter. I'll also implement Token Bucket as a second strategy since burst allowance is a real production need.

**3. Where does state live?**
- In-memory: Fast, but doesn't work across multiple instances
- Redis: The correct answer for any real deployment

**Decision:** Redis with an in-memory fallback that fails open (allows traffic) when Redis is unavailable. Failing closed during a Redis outage takes down your entire API — that's almost always the wrong tradeoff.

**4. What happens when the limit is hit?**
- Return `429 Too Many Requests`
- Include `Retry-After` header
- Include `X-RateLimit-*` headers on every response
- Log the event for observability

**5. Multi-tier limiting?**
Yes. Production systems need at least two tiers:
- **Per-second burst limit** — stops thundering herd
- **Per-minute sustained limit** — stops sustained abuse

## Phase 2: Architecture

```
Request
   │
   ▼
┌─────────────────────────────────────┐
│           Rate Limit Middleware      │
│                                     │
│  1. Identify key (IP, API key, etc) │
│  2. Determine applicable rules      │
│  3. Check all tiers atomically      │
│  4. Allow or deny                   │
│  5. Set response headers            │
└─────────────────────────────────────┘
         │              │
         ▼              ▼
    Redis Store    In-Memory Fallback
    (primary)      (when Redis down)
         │
         ▼
    Lua Script
    (atomic check-and-increment)
```

**Why a Lua script?** Redis executes Lua atomically. Without it, check-then-increment is a race condition — two requests can both read "0 requests used", both decide they're under the limit, and both increment. Under load, this race is not theoretical.

### `scripts/slidingWindow.lua`

This is the heart of the sliding window counter. It runs atomically in Redis.

```lua
--[[
  Sliding Window Counter Rate Limiter
  
  Keys:
    KEYS[1] - current window key  (e.g., "rl:user:123:1700000060")
    KEYS[2] - previous window key (e.g., "rl:user:123:1700000000")
  
  Args:
    ARGV[1] - current timestamp in milliseconds
    ARGV[2] - window size in milliseconds
    ARGV[3] - limit (max requests per window)
    ARGV[4] - TTL for keys in seconds (2x window to keep previous window alive)
  
  Returns:
    Array: [allowed, current_count, limit, reset_time_ms]
      allowed: 1 if request is allowed, 0 if denied
--]]

local current_key  = KEYS[1]
local previous_key = KEYS[2]

local now          = tonumber(ARGV[1])
local window_ms    = tonumber(ARGV[2])
local limit        = tonumber(ARGV[3])
local ttl_seconds  = tonumber(ARGV[4])

-- How far into the current window are we? (0.0 to 1.0)
local window_start    = math.floor(now / window_ms) * window_ms
local elapsed_in_window = now - window_start
local previous_weight = 1 - (elapsed_in_window / window_ms)

-- Get counts from both windows
local current_count  = tonumber(redis.call('GET', current_key)) or 0
local previous_count = tonumber(redis.call('GET', previous_key)) or 0

-- Weighted count: full current window + weighted fraction of previous window
-- This approximates a true sliding window without storing every timestamp
local weighted_count = current_count + (previous_count * previous_weight)

local reset_time = window_start + window_ms

if weighted_count >= limit then
  -- Denied — do not increment
  return {0, math.ceil(weighted_count), limit, reset_time}
end

-- Allowed — increment current window
local new_count = redis.call('INCR', current_key)

-- Set TTL on first write to current window
if new_count == 1 then
  redis.call('EXPIRE', current_key, ttl_seconds)
end

return {1, math.ceil(weighted_count + 1), limit, reset_time}
```

### `src/stores/redisStore.js`

```javascript
'use strict';

const Redis = require('ioredis');
const path = require('path');
const fs = require('fs');

class RedisStore {
  /**
   * @param {object} options
   * @param {string} options.host
   * @param {number} options.port
   * @param {string} [options.password]
   * @param {number} [options.db]
   * @param {string} [options.keyPrefix]
   * @param {object} [options.tls]
   */
  constructor(options = {}) {
    this.keyPrefix = options.keyPrefix || 'rl:';
    this.isAvailable = false;
    this.consecutiveFailures = 0;
    this.MAX_CONSECUTIVE_FAILURES = 5;

    // Load Lua script at construction time — fail fast if it's missing
    const luaPath = path.join(__dirname, '../../scripts/slidingWindow.lua');
    this.luaScript = fs.readFileSync(luaPath, 'utf8');

    this._initRedis(options);
  }

  _initRedis(options) {
    this.client = new Redis({
      host: options.host || '127.0.0.1',
      port: options.port || 6379,
      password: options.password,
      db: options.db || 0,
      tls: options.tls,
      keyPrefix: this.keyPrefix,

      // Retry strategy: exponential backoff, give up after 30s
      // This prevents connection storms on Redis restart
      retryStrategy: (times) => {
        if (times > 10) {
          return null; // Stop retrying — triggers 'error' event
        }
        return Math.min(times * 100, 3000);
      },

      // Don't queue commands when disconnected — fail fast
      // so we can fall back to in-memory immediately
      enableOfflineQueue: false,

      // Connection timeout
      connectTimeout: 5000,

      // Reconnect on error for specific error types only
      reconnectOnError: (err) => {
        const targetErrors = ['READONLY', 'ECONNRESET', 'ECONNREFUSED'];
        return targetErrors.some((e) => err.message.includes(e));
      },
    });

    this.client.on('connect', () => {
      this.isAvailable = true;
      this.consecutiveFailures = 0;
      console.info('[RateLimiter] Redis connected');
    });

    this.client.on('ready', () => {
      this.isAvailable = true;
      // Load Lua script into Redis script cache
      // SHA is deterministic — same script always produces same SHA
      this.client
        .script('load', this.luaScript)
        .then((sha) => {
          this.scriptSha = sha;
          console.info(`[RateLimiter] Lua script loaded, SHA: ${sha}`);
        })
        .catch((err) => {
          console.error('[RateLimiter] Failed to load Lua script:', err);
        });
    });

    this.client.on('error', (err) => {
      // Log but don't crash — we have a fallback
      console.error('[RateLimiter] Redis error:', err.message);
      this.consecutiveFailures++;

      if (this.consecutiveFailures >= this.MAX_CONSECUTIVE_FAILURES) {
        this.isAvailable = false;
      }
    });

    this.client.on('close', () => {
      this.isAvailable = false;
      console.warn('[RateLimiter] Redis connection closed');
    });
  }

  /**
   * Execute the sliding window Lua script.
   *
   * @param {string} identifier - Rate limit key (e.g., "user:123:endpoint:/api/data")
   * @param {number} windowMs - Window size in milliseconds
   * @param {number} limit - Max requests per window
   * @returns {Promise<{allowed: boolean, count: number, limit: number, resetTime: number}>}
   */
  async slidingWindowCheck(identifier, windowMs, limit) {
    const now = Date.now();
    const windowSeconds = Math.ceil(windowMs / 1000);

    // Current and previous window timestamps (floored to window boundary)
    const currentWindowStart = Math.floor(now / windowMs) * windowMs;
    const previousWindowStart = currentWindowStart - windowMs;

    const currentKey = `${identifier}:${currentWindowStart}`;
    const previousKey = `${identifier}:${previousWindowStart}`;

    // TTL = 2x window so the previous window key stays alive long enough
    // to be read during the next window's first requests
    const ttlSeconds = windowSeconds * 2;

    try {
      let result;

      if (this.scriptSha) {
        // Use cached SHA — faster than sending full script each time
        try {
          result = await this.client.evalsha(
            this.scriptSha,
            2,            // number of keys
            currentKey,
            previousKey,
            now,
            windowMs,
            limit,
            ttlSeconds
          );
        } catch (err) {
          if (err.message.includes('NOSCRIPT')) {
            // Script was evicted from Redis cache (e.g., Redis restart)
            // Fall back to EVAL with full script, then reload
            result = await this.client.eval(
              this.luaScript,
              2,
              currentKey,
              previousKey,
              now,
              windowMs,
              limit,
              ttlSeconds
            );
            // Reload for next time
            this.scriptSha = await this.client.script('load', this.luaScript);
          } else {
            throw err;
          }
        }
      } else {
        // Script not yet loaded (race condition on startup)
        result = await this.client.eval(
          this.luaScript,
          2,
          currentKey,
          previousKey,
          now,
          windowMs,
          limit,
          ttlSeconds
        );
      }

      this.consecutiveFailures = 0;

      return {
        allowed: result[0] === 1,
        count: result[1],
        limit: result[2],
        resetTime: result[3],
      };
    } catch (err) {
      this.consecutiveFailures++;
      if (this.consecutiveFailures >= this.MAX_CONSECUTIVE_FAILURES) {
        this.isAvailable = false;
      }
      throw err;
    }
  }

  async close() {
    await this.client.quit();
  }
}

module.exports = RedisStore;
```

### `src/stores/memoryStore.js`

'use strict';

/**
 * In-memory sliding window counter store.
 *
 * Used as a fallback when Redis is unavailable.
 *
 * IMPORTANT OPERATIONAL NOTES:
 * - State is NOT shared across Node.js processes or instances.
 * - If you run multiple instances behind a load balancer, each instance
 *   has its own counters. Effective limit per user = limit * instance_count.
 * - This is intentional: failing open (allowing more traffic) is safer
 *   than failing closed (blocking all traffic) during a Redis outage.
 * - Memory grows with unique keys. The cleanup interval prevents unbounded growth.
 */
class MemoryStore {
  constructor(options = {}) {
    // Map of key -> { currentCount, previousCount, windowStart, windowMs }
    this.windows = new Map();

    // Clean up expired entries every 5 minutes
    // Without this, the map grows forever with unique IP/user keys
    this.cleanupInterval = setInterval(
      () => this._cleanup(),
      options.cleanupIntervalMs || 5 * 60 * 1000
    );

    if (this.cleanupInterval.unref) {
      this.cleanupInterval.unref();
    }
  }
