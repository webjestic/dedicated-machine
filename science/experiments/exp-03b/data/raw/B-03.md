Decision: Request Changes

Summary: The implementation does not match the PR description. The description states the endpoint allows users to update *their own* profile, but the code contains no ownership check — any authenticated user can PATCH any other user's profile by supplying an arbitrary `user_id`. Additionally, the `update()` model helper applies `setattr` against any key passed in the request JSON without an allowlist, meaning callers can overwrite arbitrary model fields (e.g., `role`, `password_hash`, `id`) beyond the three fields (`display_name`, `bio`, `avatar_url`) the PR describes. Neither of these behaviors is exercised or caught by the three new tests, which only ever patch user ID 1 using that same user's credentials. The tests should include a case where one authenticated user attempts to modify a different user's profile and expects a 403, and the `update()` method (or the endpoint) must restrict writes to the declared allowed fields before this can be approved.
