**Decision: Request Changes**

**Summary:** This PR contains two compounding critical vulnerabilities that must be resolved before merge. The `update()` helper is a textbook mass assignment sink — `setattr(self, key, value)` over unvalidated kwargs means any field on the User model (`role`, `is_admin`, `password_hash`, `email`, or anything else) is writable via a JSON body, with no allowlist enforcing what "profile fields" actually means. On top of that, the endpoint performs no ownership check: `@login_required` only proves the caller is authenticated, not that `current_user.id == user_id`, so any logged-in user can send this request against any `user_id` and modify that account's fields — that is a direct IDOR. The tests cover only the happy path on a single user and provide zero coverage of either attack surface.

---

### Finding 1 — Mass Assignment (Critical)

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist, no filtering
```

Every field on the User model is now a writable API parameter. A request body of `{"role": "admin"}` or `{"is_staff": true}` or `{"password_hash": "<attacker-controlled>"}` will be applied without complaint. The reviewer note frames this as a feature — *"any future profile fields can be added to the model without touching the endpoint logic"* — but that is precisely the problem. The endpoint logic is the allowlist. Removing it means every column added to the model in the future is silently exposed.

**Required fix:** Replace the open `setattr` loop with an explicit allowlist.

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile_fields(self, data: dict) -> None:
    """Apply only permitted profile fields. Raises ValueError on unknown keys."""
    unknown = data.keys() - PROFILE_FIELDS
    if unknown:
        raise ValueError(f"Non-permitted fields: {unknown}")
    for key in PROFILE_FIELDS.intersection(data.keys()):
        setattr(self, key, data[key])
```

The allowlist should live on the model or in a schema layer (e.g., marshmallow, pydantic), not inferred from whatever keys arrive in the request. Rejecting unknown fields rather than silently ignoring them is preferable — silent ignoring masks client bugs and makes future audits harder.

### Finding 2 — IDOR / Missing Ownership Check (Critical)

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # no check that user_id == current_user.id
    user.update(**data)
```

Any authenticated session can target any `user_id`. Combined with the mass assignment above, this means an attacker can escalate privileges on an arbitrary account, not just their own. Even after the mass assignment is fixed, the IDOR remains a standalone vulnerability — a user should not be able to overwrite another user's display name or bio.

**Required fix:**

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403
    user = User.query.get_or_404(user_id)
    ...
```

If admins need to edit other users' profiles, that should be a separate endpoint gated by `@admin_required`, not a conditional inside this one. Mixing user-scoped and admin-scoped mutations in the same handler is how ownership checks get bypassed by logic errors.

### Finding 3 — Tests Cover the Happy Path, Not the Attack Path (Blocking)

The three new tests confirm that `display_name` and `bio` round-trip correctly and that an empty body returns 400. They do not test:

- `{"role": "admin"}` — should return 400 or 422, not 200
- `{"password_hash": "x"}` — same
- A request where the authenticated user's ID does not match `user_id` — should return 403

These tests need to exist and pass before this merges. A test suite that only exercises the intended inputs gives false confidence. The CI green check here is not evidence of safety.

```python
def test_update_profile_rejects_role_escalation(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code in (400, 422)
    # confirm the field was not written
    user = User.query.get(1)
    assert user.role != 'admin'


def test_update_profile_rejects_other_user(client, seed_users, user2_headers):
    # user2_headers authenticates as user 2, targeting user 1
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'Hijacked'},
        headers=user2_headers
    )
    assert rv.status_code == 403
```

### Secondary Note — `admin_required` Implementation

This is pre-existing and outside the scope of this PR, but worth flagging while it is visible:

```python
if not current_user.role == 'admin':
```

This should be `if current_user.role != 'admin':` — the `not` applied to the comparison rather than the result is functionally equivalent here but is a readability hazard that has caused real bugs in similar patterns. Log it for a follow-up.

### What Needs to Happen Before Merge

| # | Issue | Severity |
|---|-------|----------|
| 1 | Replace `setattr` loop with explicit field allowlist | Critical |
| 2 | Add `current_user.id == user_id` ownership check | Critical |
| 3 | Add tests for field rejection and cross-user access | Blocking |

The `update()` helper in its current form should not exist on the model. A helper that applies arbitrary kwargs via `setattr` is not a convenience — it is a footgun that will be called from other endpoints and cause the same class of bug there. If a generic update helper is wanted, it must require an explicit allowlist argument. The name `update()` with no qualifier suggests it is safe to call with any data, and that assumption will be wrong every time.
