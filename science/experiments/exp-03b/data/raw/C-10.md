**Decision: Request Changes**

**Summary:** This PR contains two critical security vulnerabilities that must be resolved before merge. The `update()` helper is a mass assignment vulnerability — it calls `setattr` in an unbounded loop over raw user input, meaning any field on the `User` model (`role`, `is_admin`, `password_hash`, `email`, etc.) is settable by sending the corresponding key in the JSON body. Compounding this, the endpoint performs no ownership check: any authenticated user can send a PATCH to `/api/users/<id>/profile` for any `user_id`, making this an IDOR that lets one user overwrite another user's fields — including privileged ones. The tests cover only the intended happy path and provide zero coverage of either attack surface.

---

### Issue 1 — Mass Assignment (Critical)

```python
# models.py — as written
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

This is the exact pattern that burned us before. The reviewer note frames the unbounded `setattr` loop as a feature ("any future profile fields can be added without touching the endpoint"). That is also a description of the attack: any future *or current* sensitive field is writable without touching the endpoint. The convenience is the vulnerability.

**Required fix — enforce an allowlist at the model boundary:**

```python
# models.py
PROFILE_UPDATABLE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    """Apply updates for user-controlled profile fields only."""
    unknown = kwargs.keys() - PROFILE_UPDATABLE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields rejected: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

The allowlist lives on the model, not the endpoint, so every future caller gets the same protection automatically. The `ValueError` on unknown fields means a misconfigured caller fails loudly rather than silently writing to a sensitive column.

### Issue 2 — IDOR / Missing Ownership Check (Critical)

```python
# api/users.py — as written
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # no check that user_id == current_user.id
    user.update(**data)
```

`@login_required` confirms the requester is authenticated. It says nothing about *which* user they are allowed to modify. An attacker authenticated as user 42 can send `PATCH /api/users/1/profile` and write to user 1's record. Combined with Issue 1, they can write `{"role": "admin"}` to any account in the system.

**Required fix — scope the operation to the requesting user:**

```python
# api/users.py
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    try:
        user.update_profile(**data)
    except ValueError as e:
        return jsonify({'error': str(e)}), 400

    db.session.commit()
    return jsonify(user.to_dict())
```

The ownership check comes before the database fetch. There is no reason to hit the database for a record the requester is not allowed to touch.

### Issue 3 — Test Coverage Does Not Cover the Attack Surface

The three new tests confirm that `display_name` and `bio` round-trip correctly and that an empty body returns 400. That is happy-path coverage. None of the following are tested:

| Scenario | Why it must be tested |
|---|---|
| `PATCH /api/users/2/profile` by user 1 | Verifies IDOR fix — must return 403 |
| `{"role": "admin"}` in the body | Verifies mass assignment fix — must return 400 |
| `{"password_hash": "..."}` in the body | Same |
| `{"email": "attacker@x.com"}` in the body | Same |
| Unknown field with valid fields mixed in | Verifies allowlist rejects the whole payload or strips cleanly |

These tests need to exist and pass before this merges. A test that only exercises the path you intended to build does not tell you whether the path you accidentally built is safe.

---

### On the Reviewer Note

> "The `update()` helper keeps the endpoint clean — any future profile fields can be added to the model without touching the endpoint logic."

This framing should be a flag during review, not a selling point. An endpoint that automatically exposes new model fields as they are added is not extensible — it is a standing invitation to accidentally publish sensitive columns. The correct design is the opposite: new fields must be explicitly opted into the allowlist before they become writable via the API. Friction here is the control.
