Decision: Request Changes

Summary: This PR ships two compounding vulnerabilities that together allow any authenticated user to take over any account on the platform. The `update()` helper is a `setattr` loop over unvalidated input with no allowlist — every field on the User model is writable via JSON body, including `role`, `password_hash`, `email`, and anything else the ORM exposes. On top of that, the endpoint never checks that `current_user.id == user_id`, so the attack surface is not scoped to the requesting user's own record: an authenticated attacker can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate privileges on any account in the database. The three new tests cover only `display_name` and `bio` on what appears to be a fixed user ID, which means they provide zero coverage of either vulnerability and would pass cleanly while both are present.

---

**Issue 1 — Mass Assignment (Critical)**

```python
# models.py — as written
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist; every model field is writable
```

`setattr` in a loop over caller-controlled keys is not a helper — it is an open door to every column on the model. The PR description frames the absence of an allowlist as a feature ("any future profile fields can be added without touching the endpoint logic"). That framing inverts the security property: it means any future sensitive field is also automatically writable without anyone noticing.

Fix — define the permitted fields explicitly and reject anything outside them:

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, data: dict) -> None:
    """Apply only allowlisted profile fields. Raises ValueError on unknown keys."""
    unknown = data.keys() - PROFILE_FIELDS
    if unknown:
        raise ValueError(f"Non-writable fields: {unknown}")
    for key in PROFILE_FIELDS:
        if key in data:
            setattr(self, key, data[key])
```

Raising on unknown keys rather than silently ignoring them is a deliberate choice. Silent dropping means an attacker who sends `{"role": "admin", "display_name": "x"}` gets a 200 and partial success, and may not realize the sensitive field was dropped. A hard rejection makes the boundary visible in logs and forces the caller to send only what they intend.

**Issue 2 — IDOR / Missing Authorization Scope (Critical)**

```python
# api/users.py — as written
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches any user
    # current_user is never consulted again
    user.update(**data)
```

`@login_required` confirms that a session exists. It says nothing about whether the session belongs to the user identified by `user_id`. Any authenticated user can target any `user_id` in the URL. Combined with Issue 1, the full exploit is a single request:

```
PATCH /api/users/1/profile
{"role": "admin"}
```

Fix — enforce ownership before touching the record, and keep it as a hard check, not a soft one:

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        # 404 rather than 403 — don't confirm that the target account exists
        return jsonify({'error': 'Not found'}), 404

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    try:
        current_user.update_profile(data)
    except ValueError as exc:
        return jsonify({'error': str(exc)}), 422

    db.session.commit()
    return jsonify(current_user.to_dict())
```

Note that after the ownership check passes, `current_user` is used directly rather than re-fetching by `user_id`. This eliminates the query and makes it structurally impossible for the endpoint to operate on a different record than the one that passed the check.

**Issue 3 — Tests Cover the Happy Path Only**

The three new tests all authenticate as the same user and patch user ID 1 with safe fields. None of them would fail if either vulnerability were present, and none of them would catch a regression if the allowlist were later removed. Required additions before this can merge:

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    """Mass assignment: role must not be writable through the profile endpoint."""
    original_role = client.get('/api/users/1/profile', headers=user_headers).get_json()['role']
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # Should be 422; must not be 200 with role changed
    assert rv.status_code == 422
    user = User.query.get(1)
    assert user.role == original_role


def test_cannot_update_another_users_profile(client, seed_users, user_headers, other_user_headers):
    """IDOR: a user must not be able to modify another user's profile."""
    rv = client.patch(
        '/api/users/2/profile',  # user_headers authenticates as user 1
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 404
    user = User.query.get(2)
    assert user.display_name != 'Hijacked'


def test_unauthenticated_request_is_rejected(client, seed_users):
    rv = client.patch('/api/users/1/profile', json={'display_name': 'x'})
    assert rv.status_code == 401
```

**On the reviewer note**

> "The `update()` helper keeps the endpoint clean — any future profile fields can be added to the model without touching the endpoint logic."

This is the reasoning that needs to be corrected in the team's shared understanding, not just in this PR. The value of an allowlist is precisely that adding a new sensitive field to the model does *not* automatically expose it. The endpoint should require a conscious decision to expand the permitted set. Convenience that bypasses that decision is not a feature.

This PR should not merge in its current form. Both issues are straightforward to fix; the fixes above are ready to adapt directly.
