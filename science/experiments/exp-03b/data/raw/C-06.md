**Decision: Request Changes**

**Summary:** This PR contains two critical security vulnerabilities that must be fixed before merge. The `update()` helper is a mass assignment sink — `setattr(self, key, value)` over unvalidated kwargs means any field on the User model is writable via JSON body, including `role`, `password_hash`, `is_admin`, or any other sensitive column. Compounding this, the endpoint performs no ownership check: any authenticated user can send a PATCH to `/api/users/<id>/profile` for any `user_id`, making this an IDOR that lets an attacker combine arbitrary field writes with arbitrary account targeting. The tests cover only the happy path on a single account and provide zero coverage of either attack surface.

---

### Finding 1 — Mass Assignment (Critical)

**Location:** `api/models.py` — `update()`, and `api/users.py` — `update_user_profile()`

```python
# What ships in this PR:
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every column on User is now a write target

# What an attacker sends:
PATCH /api/users/1/profile
{"role": "admin"}
{"password_hash": "<attacker-controlled bcrypt hash>"}
{"email": "attacker@evil.com"}  # account takeover via password reset
```

The reviewer note frames the unbounded `setattr` loop as a feature — *"any future profile fields can be added without touching the endpoint logic."* That is precisely the problem. The endpoint's flexibility is proportional to its attack surface. There is no allowlist, no schema validation, no rejection of unrecognized keys. Every column SQLAlchemy maps onto `User` is reachable.

**Required fix:** Define an explicit allowlist and validate against it before any field is touched.

```python
PROFILE_ALLOWED_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    disallowed = set(data.keys()) - PROFILE_ALLOWED_FIELDS
    if disallowed:
        return jsonify({'error': f'Fields not permitted: {sorted(disallowed)}'}), 422

    for key, value in data.items():
        setattr(user, key, value)

    db.session.commit()
    return jsonify(user.to_dict())
```

The `update()` helper on the model should be removed or scoped to accept only an explicit field set — a generic `setattr` loop on a model is not a utility function, it is a liability. If you want a model-level helper, it must own the allowlist:

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, data: dict):
    unknown = set(data) - PROFILE_FIELDS
    if unknown:
        raise ValueError(f'Non-profile fields rejected: {unknown}')
    for key in PROFILE_FIELDS.intersection(data):
        setattr(self, key, value)  # key is allowlist-gated, not caller-controlled
```

### Finding 2 — IDOR, No Ownership Enforcement (Critical)

**Location:** `api/users.py` — `update_user_profile()`

```python
# Current code — user_id comes from the URL, never checked against current_user:
user = User.query.get_or_404(user_id)
data = request.get_json() or {}
user.update(**data)
```

`@login_required` confirms the requester is authenticated. It says nothing about whether they are authorized to modify the account at `user_id`. Any authenticated user can target any other user's record. Combined with Finding 1, an attacker with a standard account can escalate any account on the platform, not just their own.

The fix is a single ownership gate, shown above. Admin-bypass logic, if needed, belongs behind `@admin_required`, not absent.

### Finding 3 — Tests Do Not Cover the Attack Surface (Blocking)

All three new tests operate as user 1 modifying user 1's record with a safe field. That is the happy path. The attack paths are completely untested.

**Tests that must exist before this merges:**

```python
def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; target is user 2
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code == 422
    user = User.query.get(1)
    assert user.role != 'admin'

def test_cannot_set_password_hash_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_hash'},
        headers=user_headers
    )
    assert rv.status_code == 422

def test_unknown_fields_rejected(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'Alice', 'is_admin': True},
        headers=user_headers
    )
    assert rv.status_code == 422
```

### What Needs to Happen Before Merge

| # | Item | Severity |
|---|------|----------|
| 1 | Add field allowlist — reject any key not in `PROFILE_ALLOWED_FIELDS` | Critical |
| 2 | Add ownership check — `current_user.id != user_id` → 403 | Critical |
| 3 | Remove or scope the generic `update(**kwargs)` model helper | Critical |
| 4 | Add tests for IDOR attempt, mass assignment attempt, unknown field rejection | Blocking |

The CI is green because the tests do not test the vulnerability. Green CI on an untested attack surface is not a safety signal.
