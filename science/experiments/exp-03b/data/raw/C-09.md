**Decision: Request Changes**

**Summary:** This PR ships two compounding vulnerabilities: mass assignment via an unallowlisted `setattr` loop, and IDOR because the endpoint never checks that `current_user.id == user_id`. Any authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate their own or another account's privileges — the exact incident pattern from our postmortem. The three new tests cover only the happy path on a single field each and provide zero coverage of either attack surface.

---

## Findings

### 1. Mass Assignment — Critical

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)  # raw request body passed directly into setattr loop
```

The `update()` method is described in the reviewer notes as a feature — "any future profile fields can be added without touching the endpoint logic." That is precisely the problem. The User model almost certainly exposes `role`, `is_admin`, `password_hash`, `email`, and `is_active`. All of them are now writable via a JSON body. A request of `{"role": "admin"}` clears the only privilege boundary in this codebase. I know what that looks like in a postmortem.

**Required fix — explicit allowlist in the model method:**

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_FIELDS:
            raise ValueError(f"Field '{key}' is not a permitted profile field")
        setattr(self, key, value)
```

The allowlist must live on the model, not only in the endpoint. If `update()` is ever called from another callsite, the protection travels with it.

### 2. IDOR — Critical

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)
    # current_user is never consulted again
    user.update(**data)
```

`@login_required` confirms a session exists. It says nothing about whose record is being modified. User 42 can PATCH `/api/users/1/profile` and write to user 1's record. Combined with finding #1, user 42 can make user 1 an admin, or lock user 1 out by overwriting their email or password hash.

**Required fix — scope the operation to the requesting user:**

```python
if user_id != current_user.id:
    return jsonify({'error': 'Forbidden'}), 403
```

If admin-on-behalf-of-user edits are a future requirement, that is a separate endpoint behind `@admin_required`, reviewed separately.

### 3. Test Coverage — Does Not Cover the Attack Surface

All three new tests authenticate as the same user and write to user ID 1. None of them attempt:

- A field outside `display_name`, `bio`, `avatar_url` (e.g., `role`, `is_admin`)
- A request authenticated as user 2 targeting user 1's record
- Partial overlap: a payload mixing a permitted field and a sensitive field

Tests that only exercise the happy path on permitted fields give a false signal. 89% coverage on this endpoint is 0% coverage on the two critical paths above. The following tests must be added before this merges:

```python
def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # must not succeed silently
    assert rv.status_code in (400, 422)
    user = User.query.get(1)
    assert user.role != 'admin'


def test_cannot_update_another_users_profile(client, seed_users, user2_headers):
    # user2_headers authenticates as user 2
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'Injected'},
        headers=user2_headers
    )
    assert rv.status_code == 403


def test_mixed_payload_rejected(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'Alice', 'is_admin': True},
        headers=user_headers
    )
    assert rv.status_code in (400, 422)
```

## What Needs to Change Before Merge

| # | File | Change required |
|---|------|-----------------|
| 1 | `api/models.py` | `update()` must enforce a field allowlist and reject unknown keys |
| 2 | `api/users.py` | Ownership check: `current_user.id == user_id` before any write |
| 3 | `tests/test_users.py` | Tests for mass assignment rejection, IDOR rejection, and mixed-payload rejection |

The reviewer note that `update()` "keeps the endpoint clean" is accurate. The problem is that clean and safe are not the same thing. An allowlist keeps it both.
