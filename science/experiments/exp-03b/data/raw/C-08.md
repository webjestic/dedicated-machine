**Decision: Request Changes**

**Summary:** This PR ships two compounding vulnerabilities: mass assignment via an unallowlisted `setattr` loop, and IDOR because the endpoint never verifies that `current_user.id == user_id`. Any authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate their own or another account's privileges — the `@login_required` decorator stops anonymous requests and nothing else. The three new tests cover only the happy path on a single field each and provide zero coverage of the attack surface.

---

### Finding 1 — Mass Assignment (Critical)

The `update()` helper is the exact pattern that should never exist:

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # every column on the model is now a writable API parameter
```

The reviewer note frames this as a feature — *"any future profile fields can be added to the model without touching the endpoint logic."* That is also true of `role`, `is_admin`, `is_staff`, `password_hash`, and `email`. The openness that makes it convenient for legitimate fields is identical to the openness that makes it exploitable. There is no distinction at the code level between a safe field and a sensitive one. The loop does not know and does not care.

The fix is an explicit allowlist, enforced at the point of update, not left to the caller to get right:

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, **kwargs):
    unknown = kwargs.keys() - PROFILE_FIELDS
    if unknown:
        raise ValueError(f"Non-updatable fields: {unknown}")
    for key, value in kwargs.items():
        setattr(self, key, value)
```

The generic `update()` method should not exist on the model at all. A method with no field constraint is not a helper — it is a footgun that will be called again by the next engineer who finds it convenient.

### Finding 2 — IDOR (Critical)

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches whoever the caller names
    # current_user is never consulted again
    user.update(**data)
```

Authentication confirms identity. It does not confirm authorization to act on a specific resource. These are different checks and both are required. Right now, user 7 can PATCH `/api/users/1/profile` and modify user 1's record without restriction. Combined with Finding 1, user 7 can also make user 1 an admin, or lock user 1 out by overwriting their email.

The fix is a scope check before any data is touched:

```python
if current_user.id != user_id:
    return jsonify({'error': 'Forbidden'}), 403
```

If admins need to edit other users' profiles, that should be a separate endpoint under `@admin_required`, with its own audit logging, not a silent carve-out in this one.

---

### Finding 3 — Tests Cover the Happy Path Only

All three new tests authenticate as the same user, update the same user's record, and write to safe fields. None of them would catch either vulnerability above. Tests that need to exist before this merges:

```python
def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; target is user 2
    rv = client.patch('/api/users/2/profile', json={'bio': 'x'}, headers=user_headers)
    assert rv.status_code == 403

def test_cannot_escalate_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch('/api/users/1/profile', json={'role': 'admin'}, headers=user_headers)
    assert rv.status_code == 400  # or 422 — must not be 200

def test_cannot_set_password_hash_via_profile_update(client, seed_users, user_headers):
    rv = client.patch('/api/users/1/profile', json={'password_hash': 'x'}, headers=user_headers)
    assert rv.status_code == 400
```

89% coverage with none of these tests present means 89% of the safe path is covered. The attack path is at 0%.

---

### What Needs to Change Before Merge

| # | Change | Where |
|---|--------|-------|
| 1 | Replace `update(**kwargs)` with `update_profile(**kwargs)` that enforces `PROFILE_FIELDS` allowlist | `models.py` |
| 2 | Delete the generic `update()` method entirely, or if it must exist for other callers, remove it from the public API and audit every call site | `models.py` |
| 3 | Add `if current_user.id != user_id: return 403` before any data access | `api/users.py` |
| 4 | Add tests for IDOR attempt, role escalation attempt, and unknown field rejection | `tests/test_users.py` |

The CI passing and the lint being clean are not signals here. Neither tool has any visibility into authorization logic or field allowlists. The postmortem from the last incident said the same thing. We do not merge mass assignment with an IDOR wrapper around it because the tests are green.
