**Decision: Request Changes**

**Summary:** This PR ships two compounding vulnerabilities: mass assignment via an unvalidated `setattr` loop, and IDOR because the endpoint never checks that `current_user.id == user_id`. Any authenticated user can send `{"role": "admin"}` to `/api/users/<any_id>/profile` and escalate privileges on any account in the system. The `update()` helper is not a convenience — it is the exact pattern that caused the incident documented in our postmortem, and the tests cover only the happy path, leaving the entire attack surface unexercised.

---

### Finding 1 — Mass Assignment (Critical)

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)  # raw request body passed directly into setattr loop
```

`kwargs` is the full JSON body. There is no allowlist. A User model that has `role`, `is_admin`, `password_hash`, `email` — any of those fields are now settable by sending the right key. The "clean endpoint" argument in the reviewer notes is the problem, not a feature. The endpoint being clean is downstream of the model method being dangerous.

**Required fix — enforce an allowlist before any field is touched:**

```python
PROFILE_ALLOWED_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update_profile(self, data: dict) -> None:
    unknown = data.keys() - PROFILE_ALLOWED_FIELDS
    if unknown:
        raise ValueError(f"Fields not permitted for update: {unknown}")
    for key, value in data.items():
        setattr(self, key, value)
```

The generic `update(**kwargs)` method should be removed entirely. A method that applies arbitrary field writes to a model has no safe use case on a request-handling path.

### Finding 2 — IDOR (Critical)

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)
    # current_user is never checked against user_id
    user.update(**data)
```

`@login_required` confirms the requester is authenticated. It says nothing about whether they are the owner of `user_id`. User 12 can patch user 7's profile. Combined with Finding 1, user 12 can escalate user 7 to admin, or lock user 7 out by overwriting their email or password hash.

**Required fix — scope the operation to the requesting user:**

```python
if user.id != current_user.id:
    return jsonify({'error': 'Forbidden'}), 403
```

This check belongs in the endpoint, before any data is read or applied. If there is a legitimate admin-override path needed in the future, it should be a separate endpoint under `admin_required`, not a conditional inside this one.

---

### Finding 3 — Tests cover the happy path only (Blocking)

The three new tests confirm that `display_name` and `bio` round-trip correctly and that an empty body returns 400. None of them test the attack surface.

**Tests that must be added before merge:**

```python
def test_cannot_escalate_role(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    # endpoint must reject unknown fields, not silently apply them
    assert rv.status_code == 400

def test_cannot_update_other_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; user 2 is a different account
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_overwrite_password_hash(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'aaaa'},
        headers=user_headers
    )
    assert rv.status_code == 400
```

### Additional note on the reviewer comment

> "any future profile fields can be added to the model without touching the endpoint logic"

This is the wrong property to optimize for. The correct property is: adding a sensitive field to the model must not silently make it writable via this endpoint. An explicit allowlist gives you that guarantee. A generic `setattr` loop gives you the opposite — every new model field is automatically exposed until someone remembers to go back and restrict it, and someone will forget.
