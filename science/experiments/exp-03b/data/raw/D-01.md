Decision: Request Changes

Summary: The implementation contains two critical vulnerabilities that block merge regardless of CI scope. First, `user.update(**data)` passes unvalidated request input directly into a `setattr` loop with no field allowlist — any field on the `User` model, including `role`, `is_admin`, `password_hash`, and `email`, is writable via the JSON body; this is the exact mass assignment pattern that produces privilege escalations. Second, the endpoint performs no ownership check: any authenticated user can send a PATCH to `/api/users/<id>/profile` for any `user_id`, making this a straightforward IDOR that compounds the first issue — an attacker can escalate privileges on accounts they do not own. The three new tests cover only `display_name` and `bio` on a fixed ID, so they do not exercise either attack surface and do not constitute meaningful coverage of this endpoint's security boundary.
