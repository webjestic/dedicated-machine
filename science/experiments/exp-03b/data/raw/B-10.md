Decision: Request Changes

Summary: The implementation does not match the PR description. The description states the endpoint updates specific fields — display name, bio, and avatar URL — but the `update()` helper uses an unrestricted `setattr` loop that will apply any key-value pair from the request body to the model, with no allowlist enforced at either the model or endpoint level. The three new tests only exercise the named profile fields and do not verify that arbitrary fields are rejected, so the test suite does not validate the behavior the PR claims to implement. Before merging, the endpoint (or the `update()` helper) must restrict writes to the declared fields (`display_name`, `bio`, `avatar_url`), and tests should be added to confirm that supplying an unlisted field (e.g., `role`, `password_hash`) results in a 400 or is silently ignored per policy.
