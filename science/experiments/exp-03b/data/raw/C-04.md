**Decision: Request Changes**

**Summary:** This PR ships two compounding vulnerabilities: mass assignment via an unallowlisted `setattr` loop, and an IDOR that lets any authenticated user overwrite any other user's record. The `update()` helper is not a convenience — it is an open write channel to every column on the User model, including `role`, `password_hash`, `email`, and anything else that gets added later. The three new tests cover only the happy path on intended fields and provide zero coverage of either attack surface.

---

## Findings

### 1. Mass Assignment — Critical

```python
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)
```

There is no allowlist. Every key in the request body becomes a column write. A user who sends:

```json
{"role": "admin"}
```

```json
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
{"email": "attacker@evil.com"}
```

gets those writes committed to the database. The endpoint logic calls `user.update(**data)` where `data` is the raw, unvalidated JSON body. The "any future profile fields can be added without touching the endpoint" note in the reviewer comments is describing the vulnerability as a feature.

**Required fix:** Define an explicit allowlist at the point of update. Nothing else is acceptable.

```python
PROFILE_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    filtered = {k: v for k, v in data.items() if k in PROFILE_FIELDS}
    if not filtered:
        return jsonify({'error': 'No valid fields provided'}), 400

    user = User.query.get_or_404(user_id)
    user.update(**filtered)
    db.session.commit()
    return jsonify(user.to_dict())
```

The `update()` method on the model is fine if and only if the caller has already filtered to safe keys. The filtering must happen at the boundary, not inside the model method, because the model method will be called from other places and you cannot guarantee what reaches it.

---

### 2. IDOR — Critical

```python
user = User.query.get_or_404(user_id)
# current_user is never checked against user_id
user.update(**data)
```

`@login_required` confirms the requester is authenticated. It says nothing about whether they own the record at `user_id`. Any authenticated user can PATCH `/api/users/2/profile`, `/api/users/3/profile`, or `/api/users/1/profile` — including the admin account — with full write access to every field on those records.

Combined with finding 1: an attacker authenticates as any ordinary user, sends `{"role": "admin"}` to `/api/users/1/profile` (or any user ID they want to escalate), and they are done.

**Required fix:** The ownership check must happen before the query is used, and it must be a hard return — not a log line, not a warning.

```python
if current_user.id != user_id:
    return jsonify({'error': 'Forbidden'}), 403
```

This belongs at the top of the function, before `get_or_404`. Querying the record and then checking ownership leaks whether a user ID exists to someone who does not own it.

### 3. Test Coverage — Does Not Test the Attack Surface

All three new tests authenticate as the same user and write to user ID 1 using only `display_name` and `bio`. The tests pass because the happy path works. They would also pass if the endpoint accepted `{"role": "admin"}` and committed it — nothing in the test suite would catch that.

**Tests that must be added before this merges:**

```python
def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; this targets user 2
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'hacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code in (200, 400)  # either is acceptable
    user = User.query.get(1)
    assert user.role != 'admin'

def test_cannot_set_password_hash_via_profile_update(client, seed_users, user_headers):
    original_hash = User.query.get(1).password_hash
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'attacker_hash'},
        headers=user_headers
    )
    assert User.query.get(1).password_hash == original_hash

def test_unauthenticated_profile_update_returns_401(client, seed_users):
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'no auth'}
    )
    assert rv.status_code == 401
```

### 4. Incidental — `admin_required` Implementation

Not introduced by this PR, but visible in the diff:

```python
if not current_user.role == 'admin':
```

This is `if current_user.role != 'admin'` with an extra negation that makes it harder to read and easier to misread during review. More importantly, if `role` is `None` or missing, behavior depends on how the comparison resolves. Recommend `if current_user.role != 'admin':` and a null check. Flag this for a follow-up; it is not a blocker for this PR specifically, but it is the guard on every admin route.

---

## What Needs to Happen Before Merge

1. Add the `PROFILE_FIELDS` allowlist and filter `data` against it before calling `update()`.
2. Add the `current_user.id != user_id` ownership check at the top of the handler, before `get_or_404`.
3. Add the four tests listed above. All four must pass.
4. Consider whether `update()` on the model should accept only known-safe keys by construction, or whether it should remain a general-purpose setter with the expectation that callers filter. Either is defensible; the current state — general-purpose setter, no caller filtering — is not.

The CI is green because the tests do not cover the attack path. Green CI on this PR is not a signal that the code is safe.
