**Decision: Request Changes**

**Summary:** This PR contains two critical security vulnerabilities that must be resolved before merge. The `update()` helper is a mass assignment vulnerability — it calls `setattr` in an unbounded loop over raw user input, meaning any field on the `User` model (`role`, `is_admin`, `password_hash`, `email`, etc.) is writable by sending the corresponding key in the JSON body. Compounding this, the endpoint performs no ownership check: `@login_required` only confirms the requester is authenticated, not that `current_user.id == user_id`, so any logged-in user can send a PATCH to `/api/users/<any_id>/profile` and modify another user's record — including escalating their privileges. The tests cover only the happy path on a single field each and provide zero coverage of either attack surface.

---

### Finding 1 — Mass Assignment (Critical)

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

```python
# endpoint
data = request.get_json() or {}
user.update(**data)  # raw request data passed directly into the loop
```

Any authenticated user can send:

```json
PATCH /api/users/1/profile
{"role": "admin"}
```

```json
PATCH /api/users/1/profile
{"password_hash": "<precomputed bcrypt hash of attacker's chosen password>"}
```

```json
PATCH /api/users/1/profile
{"is_staff": true, "email": "attacker@evil.com"}
```

The `update()` method's stated design goal — *"any future profile fields can be added to the model without touching the endpoint logic"* — is precisely what makes it dangerous. It means any future sensitive field is also automatically exposed. The convenience is the vulnerability.

**Required fix:** Replace the unbounded `setattr` loop with an explicit allowlist. The allowlist must live at the point of validation, not be inferred from what the model happens to have:

```python
# In the endpoint, not in a generic model method
PROFILE_UPDATABLE_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    user = User.query.get_or_404(user_id)
    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    unknown = set(data.keys()) - PROFILE_UPDATABLE_FIELDS
    if unknown:
        return jsonify({'error': f'Unknown fields: {sorted(unknown)}'}), 422

    for key in PROFILE_UPDATABLE_FIELDS:
        if key in data:
            setattr(user, key, data[key])

    db.session.commit()
    return jsonify(user.to_dict())
```

The generic `update(**kwargs)` method on the model should be removed entirely. A helper that applies arbitrary kwargs to a model object is not a utility — it is a footgun that will be reused.

### Finding 2 — IDOR / Missing Ownership Check (Critical)

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches whoever the URL names
    # current_user is never compared to user_id
```

`@login_required` answers one question: *is there a session?* It does not answer: *does this session own this resource?* Without the ownership check, every authenticated user is effectively an admin over every other user's profile. Combined with Finding 1, a standard user account is sufficient to escalate any account on the platform to admin.

**Required fix:** The ownership check shown above (`current_user.id != user_id → 403`) is the minimum. If admin users should be able to edit other profiles, that should be a separate, explicitly gated code path using the existing `admin_required` decorator — not the same endpoint with the same logic.

### Finding 3 — Test Coverage Does Not Cover the Attack Surface

All three new tests authenticate as the same user and update only `display_name` or `bio`. They confirm the happy path works. They do not confirm the attack path is blocked.

**Tests that must be added before this merges:**

```python
def test_cannot_update_another_users_profile(client, seed_users, user_headers):
    # user_headers authenticates as user 1; target is user 2
    rv = client.patch(
        '/api/users/2/profile',
        json={'display_name': 'Hijacked'},
        headers=user_headers
    )
    assert rv.status_code == 403

def test_cannot_set_role_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'role': 'admin'},
        headers=user_headers
    )
    assert rv.status_code in (400, 422)
    # Confirm the field was not written
    user = User.query.get(1)
    assert user.role != 'admin'

def test_cannot_set_password_hash_via_profile_update(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'password_hash': 'arbitrary'},
        headers=user_headers
    )
    assert rv.status_code in (400, 422)

def test_unknown_fields_rejected(client, seed_users, user_headers):
    rv = client.patch(
        '/api/users/1/profile',
        json={'display_name': 'Alice', 'is_staff': True},
        headers=user_headers
    )
    assert rv.status_code == 422
```

### On the Reviewer Note

> *"The `update()` helper keeps the endpoint clean — any future profile fields can be added to the model without touching the endpoint logic."*

This framing should be reconsidered as a design principle. An endpoint that automatically exposes new model fields without a code change is not clean — it means the security boundary is never explicitly reviewed when the model grows. The allowlist requiring a deliberate update is not friction; it is the review checkpoint. Every field that becomes user-writable should be a conscious decision, not a side effect of adding a column.
