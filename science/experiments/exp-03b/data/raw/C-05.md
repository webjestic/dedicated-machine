**Decision: Request Changes**

**Summary:** This PR contains two critical security vulnerabilities that must be fixed before merge. The `update()` helper is a mass assignment vulnerability — `setattr` in a loop over unvalidated user input means any field on the `User` model is writable via the request body, including `role`, `is_admin`, `password_hash`, and `email`. Compounding this, the endpoint never verifies that `current_user.id == user_id`, so any authenticated user can send these payloads against any other user's account — that is a textbook IDOR. The three new tests cover only the happy path on a single user's own fields and provide zero coverage of either attack surface.

---

## Findings

### 1. Mass Assignment — Critical

```python
# models.py
def update(self, **kwargs):
    for key, value in kwargs.items():
        setattr(self, key, value)  # no allowlist — every model field is writable
```

```python
# users.py
data = request.get_json() or {}
user.update(**data)  # caller-controlled keys flow directly into setattr
```

Any authenticated user can send:

```json
PATCH /api/users/1/profile
{"role": "admin"}
```

```json
PATCH /api/users/1/profile
{"password_hash": "$2b$12$attacker_controlled_hash"}
```

```json
PATCH /api/users/1/profile
{"email": "attacker@evil.com", "is_verified": true}
```

The reviewer note frames the absence of an allowlist as a feature — *"any future profile fields can be added to the model without touching the endpoint logic."* That is precisely the problem. The endpoint should be the enforcement point. An allowlist that never needs updating is an allowlist that doesn't exist.

**Fix:** Define permitted fields explicitly. Do not pass this responsibility to the model method.

```python
PROFILE_FIELDS = {'display_name', 'bio', 'avatar_url'}

@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    if current_user.id != user_id:
        return jsonify({'error': 'Forbidden'}), 403

    data = request.get_json() or {}
    if not data:
        return jsonify({'error': 'No data provided'}), 400

    unknown = set(data.keys()) - PROFILE_FIELDS
    if unknown:
        return jsonify({'error': f'Unknown fields: {sorted(unknown)}'}), 400

    user = User.query.get_or_404(user_id)
    user.update(**{k: v for k, v in data.items() if k in PROFILE_FIELDS})
    db.session.commit()
    return jsonify(user.to_dict())
```

And the model method should only accept known fields — it should not be a generic `setattr` dispatcher:

```python
PROFILE_FIELDS = frozenset({'display_name', 'bio', 'avatar_url'})

def update(self, **kwargs):
    for key, value in kwargs.items():
        if key not in PROFILE_FIELDS:
            raise ValueError(f'Field not updatable via profile endpoint: {key}')
        setattr(self, key, value)
```

Defense in depth: the endpoint filters first, the model method refuses anything that slips through.

### 2. IDOR — Critical

```python
@users_bp.route('/api/users/<int:user_id>/profile', methods=['PATCH'])
@login_required
def update_user_profile(user_id):
    user = User.query.get_or_404(user_id)  # fetches whoever the caller names
    # current_user is never checked against user_id
```

`@login_required` confirms the caller is authenticated. It says nothing about whether the caller is authorized to modify the target record. User 42 can send `PATCH /api/users/1/profile` and the endpoint will comply. Combined with finding #1, user 42 can escalate user 1 to admin, lock user 1 out by replacing their password hash, or take over their email address.

**Fix:** shown above — check `current_user.id != user_id` before the query. Return 403, not 404. Returning 404 for another user's record leaks whether the ID exists; 403 is the correct response when the caller is authenticated but not authorized.

---

### 3. Tests Do Not Cover the Attack Surface

All three new tests authenticate as the same user and update only `display_name` and `bio`. None of the following are tested:

| Scenario | Tested |
|---|---|
| Authenticated user modifies another user's profile | ✗ |
| `role` field rejected | ✗ |
| `is_admin` field rejected | ✗ |
| `password_hash` field rejected | ✗ |
| Unknown field returns 400 | ✗ |
| Unauthenticated request returns 401 | ✗ |

After the fixes above, these cases must have explicit tests. A test that sends `{"role": "admin"}` and asserts `403` or `400` is the test that would have caught the incident I wrote the postmortem on.

---

### 4. Minor — `admin_required` Implementation

Not introduced in this PR, but visible in the diff:

```python
if not current_user.role == 'admin':
```

This should be:

```python
if current_user.role != 'admin':
```

Operator precedence is correct here but `not x == y` is a code smell that obscures intent. Flag for cleanup.

---

## What Needs to Happen Before Merge

1. Add an explicit allowlist of writable profile fields — in the endpoint, enforced before the model method is called.
2. Add `current_user.id != user_id` authorization check at the top of the handler.
3. Harden the `update()` model method to reject fields outside the allowlist as a second layer.
4. Add tests for the IDOR case, the mass assignment rejection cases, and the unauthenticated case.

The CI passing and coverage staying at 89% are not signals of safety here. The tests were written against the happy path. The attack path is untested and, as written, succeeds.
