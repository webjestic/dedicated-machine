# Production Readiness Review: Rate Limiter

## Executive Summary

The core algorithm is sound. The Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT recovery pattern is appropriate. However, this implementation has **seven categories of gaps** that would cause operational problems in production. I'll address each explicitly, flag what requires architectural decisions beyond this codebase, and provide concrete fixes.

---

## Gap 1: Redis Failure Has No Fallback — Requests Will 500

**Severity: Critical**

When Redis is unavailable, `rateLimiter.check()` throws an unhandled exception. The middleware has no try/catch, so Express returns a 500 to the caller. Depending on your failure mode preference, you want either **fail-open** (allow traffic through) or **fail-closed** (reject traffic). Neither is implemented — you get a crash instead.

Additionally, the NOSCRIPT retry has a subtle race: if the second `evalsha` also fails with a non-NOSCRIPT error, it throws unhandled from inside the catch block, which can obscure the original error in some logging pipelines.

```javascript
// In RateLimiter class
async check(identifier) {
  const key = `${this.keyPrefix}:${identifier}`;
  const now = Date.now();
  const requestId = uuidv4();

  let result;
  try {
    if (!this._scriptSha) {
      await this._loadScript();
    }

    try {
      result = await this.redis.evalsha(
        this._scriptSha, 1, key,
        String(this.windowMs), String(this.limit), String(now), requestId
      );
    } catch (err) {
      if (err.message && err.message.includes('NOSCRIPT')) {
        await this._loadScript();
        result = await this.redis.evalsha(
          this._scriptSha, 1, key,
          String(this.windowMs), String(this.limit), String(now), requestId
        );
      } else {
        throw err; // re-throw non-NOSCRIPT Redis errors to outer catch
      }
    }
  } catch (err) {
    // Emit for observability before deciding behavior
    this.emit('error', err); // see Gap 5 — RateLimiter should extend EventEmitter

    if (this.failOpen) {
      // Fail-open: allow the request, log the degradation
      return {
        allowed: true,
        count: 0,
        remaining: this.limit,
        limit: this.limit,
        resetAt: new Date(now + this.windowMs).toISOString(),
        degraded: true, // caller can log/alert on this flag
      };
    }
    throw err; // fail-closed: let middleware handle it
  }

  const [allowed, count] = result;
  return {
    allowed: allowed === 1,
    count,
    remaining: Math.max(0, this.limit - count),
    limit: this.limit,
    resetAt: new Date(now + this.windowMs).toISOString(),
  };
}
```

```javascript
// In middleware — always handle errors regardless of fail mode
return async function rateLimitMiddleware(req, res, next) {
  try {
    const identifier = getIdentifier(req);
    const result = await rateLimiter.check(identifier);

    // Set standard rate limit headers on every response (see Gap 3)
    res.set({
      'X-RateLimit-Limit': result.limit,
      'X-RateLimit-Remaining': result.remaining,
      'X-RateLimit-Reset': result.resetAt,
    });

    if (!result.allowed) {
      return res.status(429).json({
        error: 'Too Many Requests',
        retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
      });
    }

    next();
  } catch (err) {
    // Redis is down and fail-closed is configured — return 503, not 500
    // 503 signals "temporarily unavailable" and is retryable by clients/proxies
    return res.status(503).json({ error: 'Rate limiter unavailable' });
  }
};
```

**Flag for requester:** The choice between fail-open and fail-closed is a **product/security decision**, not a technical one. Fail-open keeps your API available during Redis outages but allows unlimited traffic. Fail-closed protects against abuse during outages but takes down your API with Redis. This must be explicitly decided and documented.

## Gap 2: `req.ip` Is Untrustworthy Behind a Proxy — Spoofable by Default

**Severity: Critical**

`req.ip` in Express returns the value of `X-Forwarded-For` when `trust proxy` is enabled, or the raw socket IP when it isn't. Neither is safe by default:

- Without `trust proxy`: all requests appear to come from your load balancer's IP — one user can exhaust the limit for everyone.
- With `trust proxy` enabled naively: any client can send `X-Forwarded-For: 1.2.3.4` and spoof their IP, bypassing per-IP limits entirely.

The `keyBy: req => req.headers['x-user-id'] || req.ip` fallback compounds this — an unauthenticated request with a spoofed IP gets its own bucket with no rate limiting enforcement.

```javascript
// In bootstrap — configure trust proxy correctly
// Only trust the specific number of proxy hops in your infrastructure
app.set('trust proxy', 1); // trust one hop (your load balancer only)
// OR for multiple known proxies:
// app.set('trust proxy', ['loopback', '10.0.0.0/8']);

// In middleware — harden the identifier extraction
const getIdentifier = options.keyBy || function(req) {
  // req.ip is now the real client IP given correct trust proxy config
  const ip = req.ip;
  if (!ip) {
    // No IP means misconfigured proxy setup — fail loudly in dev, 
    // use a safe fallback in prod
    throw new Error('Could not determine client IP — check trust proxy config');
  }
  return ip;
};
```

**Flag for requester:** The correct `trust proxy` value depends entirely on your infrastructure topology (number of load balancer hops, whether you use Cloudflare/AWS ALB, etc.). This cannot be set correctly without knowing your deployment environment.

## Gap 3: No Standard Rate Limit Headers — Clients Cannot Back Off

**Severity: High**

The 429 response includes `retryAfter` but no standard headers on successful responses. Clients and API consumers have no way to know their current usage until they're already blocked. The `Retry-After` header on 429s is also using a non-standard JSON field name instead of the HTTP header.

The fix is shown in Gap 1's middleware code above. Additionally, the 429 response needs the HTTP header:

```javascript
if (!result.allowed) {
  res.set('Retry-After', Math.ceil(rateLimiter.windowMs / 1000));
  return res.status(429).json({
    error: 'Too Many Requests',
    retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
  });
}
```

The `resetAt` value in the current implementation is also misleading — it always returns `now + windowMs`, which is the far edge of the window, not when the *oldest request* in the window will expire. For a sliding window, the accurate reset time is when the earliest entry in the sorted set expires. Getting the true value requires an additional Redis call (`ZRANGE key 0 0 WITHSCORES`), which is a tradeoff between accuracy and latency. For most use cases, the current approximation is acceptable if documented.

## Gap 4: No Input Validation — Malformed Identifiers Corrupt Key Space

**Severity: High**

The `identifier` parameter is concatenated directly into a Redis key with no validation. This creates two problems:

1. **Key collision**: An identifier containing `:` (e.g., `"user:123"`) produces `rl:user:123`, which is indistinguishable from a prefix collision with another namespace.
2. **Empty/null identifiers**: If `keyBy` returns `undefined`, `null`, or an empty string, you get keys like `api:rl:undefined` — all unauthenticated requests share one bucket.

```javascript
async check(identifier) {
  if (!identifier || typeof identifier !== 'string') {
    throw new Error(
      `RateLimiter.check() requires a non-empty string identifier, got: ${JSON.stringify(identifier)}`
    );
  }
  if (identifier.length > 512) {
    throw new Error('Rate limiter identifier exceeds maximum length of 512 characters');
  }

  // Sanitize to prevent key injection — encode colons in the identifier
  const safeIdentifier = identifier.replace(/:/g, '%3A');
  const key = `${this.keyPrefix}:${safeIdentifier}`;
  // ...
}
```

## Gap 5: No Observability — You Cannot Operate What You Cannot See

**Severity: High**

There is no logging, no metrics emission, and no way to observe the rate limiter's behavior in production. You cannot answer:

- What percentage of requests are being rate limited?
- Which identifiers are hitting limits most frequently?
- Is Redis latency affecting API response times?
- Is the rate limiter in degraded mode?

```javascript
const { EventEmitter } = require('events');

class RateLimiter extends EventEmitter {
  constructor({ windowMs, limit, keyPrefix = 'rl', redisClient, failOpen = true }) {
    super();
    this.windowMs = windowMs;
    this.limit = limit;
    this.keyPrefix = keyPrefix;
    this.redis = redisClient;
    this.failOpen = failOpen;
    this._scriptSha = null;
  }

  async check(identifier) {
    const startTime = Date.now();
    // ... existing logic ...

    const duration = Date.now() - startTime;

    // Emit structured events — caller wires these to their metrics/logging system
    this.emit('check', {
      identifier,
      allowed: result.allowed,
      count: result.count,
      remaining: result.remaining,
      durationMs: duration,
      degraded: result.degraded || false,
    });

    if (!result.allowed) {
      this.emit('limited', { identifier, count: result.count });
    }

    return result;
  }
}
```

## Gap 6: Redis Connection Has No Production Configuration

**Severity: Medium**

The Redis client is created with bare defaults. In production this means:

- No reconnection strategy — a transient Redis blip disconnects the client permanently
- No connection timeout — requests hang indefinitely if Redis is unreachable
- No TLS — credentials and data transmitted in plaintext
- No error handler on the client — ioredis emits `'error'` events; without a handler, Node.js throws an unhandled exception and crashes the process

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,

  // TLS for production (required if Redis is not on localhost/private network)
  // tls: process.env.REDIS_TLS === 'true' ? {} : undefined,

  // Connection timeout
  connectTimeout: 5000,        // 5s to establish connection
  commandTimeout: 2000,        // 2s max per command — prevents hanging requests

  // Reconnection: exponential backoff, give up after 10 attempts
  retryStrategy(times) {
    if (times > 10) {
      return null; // stop retrying, emit error
    }
    return Math.min(times * 200, 3000); // 200ms, 400ms, ... up to 3s
  },

  // Required: prevents unhandled 'error' event from crashing Node process
  lazyConnect: false,
});

redisClient.on('error', (err) => {
  console.error('[redis] Connection error:', err);
  // yourMetrics.increment('redis.connection.error');
});

redisClient.on('reconnecting', () => {
  console.warn('[redis] Reconnecting...');
});
```

## Gap 7: No Graceful Shutdown — In-Flight Requests Dropped on Deploy

**Severity: Medium**

The server has no shutdown handler. On `SIGTERM` (sent by Kubernetes, systemd, or any process manager during deploy), Node exits immediately, dropping in-flight requests and leaving the Redis connection unclosed. This causes connection pool exhaustion on Redis if deploys are frequent.

```javascript
const server = app.listen(3000, () => {
  console.log('Server listening on port 3000');
});

async function shutdown(signal) {
  console.log(`[shutdown] Received ${signal}, starting graceful shutdown`);

  // Stop accepting new connections
  server.close(async () => {
    console.log('[shutdown] HTTP server closed');

    try {
      // Flush pending Redis commands and close connection cleanly
      await redisClient.quit();
      console.log('[shutdown] Redis connection closed');
    } catch (err) {
      console.error('[shutdown] Error closing Redis connection:', err);
      redisClient.disconnect(); // force disconnect if quit hangs
    }

    process.exit(0);
  });

  // Force exit if graceful shutdown takes too long
  setTimeout(() => {
    console.error('[shutdown] Graceful shutdown timed out, forcing exit');
    process.exit(1);
  }, 10_000);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT',  () => shutdown('SIGINT'));
```
