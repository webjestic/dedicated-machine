# Production Readiness Review: Rate Limiter

## Summary

The core algorithm is sound — the Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT retry is handled. However, there are several gaps across reliability, security, observability, and operability that would make this unsafe to run in production as-is.

---

## Critical Issues

### 1. Redis Failure Has No Fallback — Unhandled Errors Crash Requests

If Redis is unavailable, `check()` throws, and the middleware has no `try/catch`. This means a Redis outage takes down your entire API.

```javascript
// Current — unhandled rejection crashes the request (or worse, the process)
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const result = await rateLimiter.check(identifier); // throws if Redis is down
```

**Required:** Define an explicit failure policy and implement it consistently.

```javascript
// Option A: Fail open (allow traffic, log the error)
// Option B: Fail closed (block traffic)
// Option C: Degrade to in-process fallback limiter

return async function rateLimitMiddleware(req, res, next) {
  let result;
  try {
    result = await rateLimiter.check(identifier);
  } catch (err) {
    logger.error({ err, identifier }, 'Rate limiter check failed');
    metrics.increment('rate_limiter.redis_error');

    // Fail open: allow request, but signal degraded state
    return next();

    // OR fail closed:
    // return res.status(503).json({ error: 'Service temporarily unavailable' });
  }
  // ...
};
```

The choice between fail-open and fail-closed is a business decision, but it must be made explicitly and documented. Failing open is common for rate limiters (availability over protection during outages), but fail-closed may be required for security-sensitive endpoints.

### 2. `req.ip` Is Untrustworthy Behind a Proxy

```javascript
keyBy: req => req.headers['x-user-id'] || req.ip,
```

`req.ip` in Express returns the value of `X-Forwarded-For` only if `app.set('trust proxy', ...)` is configured. Without it, `req.ip` is the load balancer's IP, meaning all traffic shares one rate limit bucket. With it misconfigured, clients can spoof their IP by setting `X-Forwarded-For` themselves.

**Required:**

```javascript
// In app setup — set to the number of trusted proxy hops, or a CIDR range
app.set('trust proxy', 1); // trust one hop (your load balancer)

// In keyBy — validate the identifier is non-empty and well-formed
keyBy: req => {
  const userId = req.headers['x-user-id'];
  if (userId) {
    // Ensure this header is set by your auth layer, not by the client
    return `user:${userId}`;
  }
  return `ip:${req.ip}`;
}
```

**Separate concern:** `x-user-id` must be set by authenticated middleware, not passed directly from the client. If a client can set this header arbitrarily, they can impersonate any user ID or use a non-existent ID to bypass per-user limits.

### 3. No Connection Error Handling on Redis Client Startup

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
});
```

ioredis will silently retry connections by default. If Redis is unavailable at startup, the app starts anyway and all rate limit checks fail. There are no error event handlers.

**Required:**

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  maxRetriesPerRequest: 2,        // fail fast on individual commands
  enableReadyCheck: true,
  lazyConnect: true,              // connect explicitly so you can handle failure
});

redisClient.on('error', (err) => {
  logger.error({ err }, 'Redis client error');
  metrics.increment('redis.connection_error');
});

redisClient.on('reconnecting', () => {
  logger.warn('Redis client reconnecting');
});

// Explicit connect with startup failure handling
try {
  await redisClient.connect();
} catch (err) {
  logger.fatal({ err }, 'Failed to connect to Redis at startup');
  process.exit(1); // or start in degraded mode
}
```

## Security Issues

### 4. Key Injection via Identifier

```javascript
const key = `${this.keyPrefix}:${identifier}`;
```

If `identifier` contains `:` or other characters, it could collide with other keys or be used to craft unexpected key names. If `identifier` is empty or null, the key becomes `rl:` or `rl:undefined`.

**Required:**

```javascript
_buildKey(identifier) {
  if (!identifier || typeof identifier !== 'string') {
    throw new Error(`Invalid rate limit identifier: ${JSON.stringify(identifier)}`);
  }
  // Sanitize or hash the identifier
  const sanitized = identifier.replace(/[^a-zA-Z0-9._@-]/g, '_');
  return `${this.keyPrefix}:${sanitized}`;
}
```

For high-cardinality identifiers (long user IDs, tokens), consider hashing:

```javascript
const crypto = require('crypto');
const hashed = crypto.createHash('sha256').update(identifier).digest('hex');
return `${this.keyPrefix}:${hashed}`;
```

### 5. No Rate Limit Headers Returned on Allowed Requests

The result object contains `remaining`, `limit`, and `resetAt`, but these are never sent to the client. This is both a standards compliance issue (RFC 6585, draft `RateLimit` header spec) and an operability issue — clients cannot implement backoff without this information.

**Required:**

```javascript
return async function rateLimitMiddleware(req, res, next) {
  // ...
  
  // Set headers on all responses, not just rejections
  res.set({
    'X-RateLimit-Limit': result.limit,
    'X-RateLimit-Remaining': result.remaining,
    'X-RateLimit-Reset': result.resetAt,
  });

  if (!result.allowed) {
    res.set('Retry-After', Math.ceil(rateLimiter.windowMs / 1000));
    return res.status(429).json({
      error: 'Too Many Requests',
      retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
    });
  }

  next();
};
```

## Reliability Issues

### 6. `resetAt` Is Inaccurate

```javascript
resetAt: new Date(now + this.windowMs).toISOString(),
```

This returns the end of a fixed window from now, not the actual reset time for a sliding window. In a sliding window, the reset time is when the oldest request in the current window expires — which is `oldest_request_timestamp + windowMs`.

**Required:** Return the oldest entry's timestamp from the Lua script.

```lua
-- Modified Lua script
local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
local reset_at = 0
if #oldest > 0 then
  reset_at = tonumber(oldest[2]) + window_ms
end

if count < limit then
  redis.call('ZADD', key, now, request_id)
  redis.call('PEXPIRE', key, window_ms)
  return {1, count + 1, reset_at}
else
  return {0, count, reset_at}
end
```

```javascript
const [allowed, count, resetAtMs] = result;
return {
  allowed: allowed === 1,
  count,
  remaining: Math.max(0, this.limit - count),
  limit: this.limit,
  resetAt: resetAtMs > 0 ? new Date(resetAtMs).toISOString() : new Date(now + this.windowMs).toISOString(),
};
```

### 7. `_loadScript` Has a Race Condition

```javascript
async _loadScript() {
  this._scriptSha = await this.redis.script('LOAD', SLIDING_WINDOW_SCRIPT);
}

// In check():
if (!this._scriptSha) {
  await this._loadScript();
}
```

If two concurrent requests both see `_scriptSha === null`, they both call `_loadScript()` simultaneously. This is harmless in practice (both get the same SHA), but the pattern is fragile. More importantly, there is no initialization step — the script should be loaded once at startup, not lazily per-request.

**Required:**

```javascript
class RateLimiter {
  constructor(opts) {
    // ...
    this._scriptSha = null;
    this._loadingScript = null; // promise deduplication
  }

  async initialize() {
    await this._loadScript();
  }

  async _loadScript() {
    if (this._loadingScript) return this._loadingScript;
    this._loadingScript = this.redis.script('LOAD', SLIDING_WINDOW_SCRIPT)
      .then(sha => {
        this._scriptSha = sha;
        this._loadingScript = null;
      });
    return this._loadingScript;
  }
}

// At startup — fail fast if this doesn't work
await limiter.initialize();
```

### 8. No Timeout on Redis Commands

A slow Redis response will hold the request open indefinitely, exhausting the Node.js event loop and connection pool.

**Required:**

```javascript
const redisClient = new redis({
  // ...
  commandTimeout: 200,  // ms — fail fast, don't hold requests open
});
```

Or wrap individual calls:

```javascript
const withTimeout = (promise, ms) =>
  Promise.race([
    promise,
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error('Redis timeout')), ms)
    ),
  ]);

result = await withTimeout(
  this.redis.evalsha(...),
  200
);
```

## Observability Issues

### 9. No Logging or Metrics

There is no instrumentation. In production you need to know:

- Rate limit hit rate per identifier/endpoint
- Redis error rate and latency
- Whether the limiter is actually working

**Required:**

```javascript
return async function rateLimitMiddleware(req, res, next) {
  const start = Date.now();
  const identifier = getIdentifier(req);

  try {
    const result = await rateLimiter.check(identifier);
    const latency = Date.now() - start;

    metrics.histogram('rate_limiter.check_duration_ms', latency);

    if (!result.allowed) {
      metrics.increment('rate_limiter.rejected', { identifier_type: getIdentifierType(req) });
      logger.warn({ identifier, count: result.count, limit: result.limit }, 'Rate limit exceeded');
    }

    // ... set headers, call next()
  } catch (err) {
    metrics.increment('rate_limiter.error');
    logger.error({ err, identifier }, 'Rate limiter error');
    // apply failure policy
  }
};
```

## Operability Issues

### 11. No Graceful Shutdown

```javascript
app.listen(3000, () => {
  console.log('Server listening on port 3000');
});
```

When the process receives SIGTERM (e.g., during deployment), in-flight requests are dropped and the Redis connection is not cleanly closed.

**Required:**

```javascript
const server = app.listen(3000, () => {
  console.log('Server listening on port 3000');
});

async function shutdown(signal) {
  logger.info({ signal }, 'Shutdown signal received');
  
  server.close(async () => {
    logger.info('HTTP server closed');
    await redisClient.quit();
    logger.info('Redis connection closed');
    process.exit(0);
  });

  // Force exit if graceful shutdown takes too long
  setTimeout(() => {
    logger.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10_000);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

### 12. No Configuration Validation

```javascript
constructor({ windowMs, limit, keyPrefix = 'rl', redisClient }) {
  this.windowMs = windowMs;
  this.limit = limit;
  // ...
}
```

If `windowMs` or `limit` is undefined, zero, or negative, the Lua script will behave incorrectly and the error will be difficult to diagnose.

**Required:**

```javascript
constructor({ windowMs, limit, keyPrefix = 'rl', redisClient }) {
  if (!Number.isInteger(windowMs) || windowMs <= 0) {
    throw new Error(`windowMs must be a positive integer, got: ${windowMs}`);
  }
  if (!Number.isInteger(limit) || limit <= 0) {
    throw new Error(`limit must be a positive integer, got: ${limit}`);
  }
  if (!redisClient) {
    throw new Error('redisClient is required');
  }
  // ...
}
```
