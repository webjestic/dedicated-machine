# Production Readiness Review: Rate Limiter

## Executive Summary

The core algorithm is sound. The Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT retry pattern is handled. However, this implementation has **seven categories of gaps** that would cause operational problems in production. I'll address each explicitly, flag what requires external decisions, and provide concrete fixes.

---

## Gap 1: Redis Failure Has No Fallback — Requests Will Error 500

**Severity: Critical**

When Redis is unavailable, `check()` throws. The middleware has no try/catch, so unhandled promise rejections propagate to Express's default error handler, returning 500s to all clients. A Redis blip takes down your API.

You must make an explicit policy decision: **fail open** (allow requests when Redis is down) or **fail closed** (block requests). Neither is universally correct — it depends on whether your rate limiter is a security control or a capacity control.

```javascript
// In createRateLimitMiddleware — add explicit failure policy
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);

  let result;
  try {
    result = await rateLimiter.check(identifier);
  } catch (err) {
    // POLICY DECISION: documented and explicit, not accidental.
    // Fail open: Redis outage does not take down the API.
    // Change to fail closed if this limiter is a security boundary.
    console.error('[rate-limiter] Redis error, failing open:', err.message);
    
    // Emit to your metrics/alerting system here — see Gap 5
    return next();
  }

  // ... rest of middleware
};
```

**Flag for requester:** Which policy applies here? If this limiter protects against credential stuffing or scraping, fail-closed is likely correct. If it's purely capacity management, fail-open preserves availability.

## Gap 2: `req.ip` Is Untrustworthy Behind a Proxy — Identifier Poisoning

**Severity: Critical**

The default `keyBy` uses `req.ip`. Behind a load balancer or reverse proxy, `req.ip` is the proxy's IP, meaning all users share one rate limit bucket. Worse, if `X-Forwarded-For` is trusted without validation, a client can spoof it to use an arbitrary key — including an empty string or another user's IP.

```javascript
// In bootstrap — trust your proxy tier explicitly
const app = express();
app.set('trust proxy', 1); // Trust exactly one proxy hop (your load balancer)
// Use a specific number, not `true` — `true` trusts all hops, enabling spoofing

// In keyBy — validate the identifier before using it
app.use('/api', createRateLimitMiddleware(limiter, {
  keyBy: req => {
    const userId = req.headers['x-user-id'];
    
    // Validate user ID comes from your auth layer, not raw client input.
    // This should be set by authenticated middleware upstream, not trusted
    // directly from the request header if clients can set arbitrary headers.
    if (userId && /^[a-zA-Z0-9_-]{1,128}$/.test(userId)) {
      return `user:${userId}`;
    }
    
    // req.ip is now correct because trust proxy is set
    return `ip:${req.ip}`;
  },
}));
```

**Flag for requester:** Is `x-user-id` set by your authentication middleware (trustworthy) or passed directly from the client (not trustworthy)? If the latter, an unauthenticated user can claim any user ID and exhaust that user's quota.

## Gap 3: No Rate Limit Headers — Clients Cannot Self-Regulate

**Severity: High**

RFC 6585 and the draft `RateLimit` header spec (widely adopted) define headers that tell clients their current quota state. Without them, clients have no way to back off gracefully — they discover the limit only by hitting 429. Well-behaved API clients and SDKs depend on these headers.

```javascript
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);

  let result;
  try {
    result = await rateLimiter.check(identifier);
  } catch (err) {
    console.error('[rate-limiter] Redis error, failing open:', err.message);
    return next();
  }

  // Set headers on every response, not just 429s.
  // Clients use these to throttle proactively.
  res.set({
    'X-RateLimit-Limit':     result.limit,
    'X-RateLimit-Remaining': result.remaining,
    'X-RateLimit-Reset':     result.resetAt,
    // Retry-After is seconds, required on 429 by RFC 6585
    ...(result.allowed ? {} : { 'Retry-After': Math.ceil(rateLimiter.windowMs / 1000) }),
  });

  if (!result.allowed) {
    return res.status(429).json({
      error: 'Too Many Requests',
      retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
    });
  }

  next();
};
```

## Gap 4: `resetAt` Is Wrong — It Reports Window End, Not Quota Reset

**Severity: Medium**

The current `resetAt` is `now + windowMs`, which is the far edge of the sliding window. But in a sliding window, quota resets incrementally as old requests age out — not at a fixed future time. Reporting `now + windowMs` tells clients they must wait the full window duration, which is incorrect and causes unnecessary backoff.

The accurate value is when the *oldest request in the current window* will expire, which is when one slot opens up.

```javascript
// This requires knowing the oldest entry's timestamp.
// Add a second return value to the Lua script:

const SLIDING_WINDOW_SCRIPT = `
local key = KEYS[1]
local window_ms = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local request_id = ARGV[4]

redis.call('ZREMRANGEBYSCORE', key, 0, now - window_ms)
local count = redis.call('ZCARD', key)

if count < limit then
  redis.call('ZADD', key, now, request_id)
  redis.call('PEXPIRE', key, window_ms)
  return {1, count + 1, -1}
else
  -- Get the oldest entry's score (timestamp) so client knows when a slot opens
  local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
  local oldest_ts = oldest[2] and tonumber(oldest[2]) or now
  return {0, count, oldest_ts + window_ms}
end
`;

// In check():
const [allowed, count, resetTs] = result;
const resetAt = resetTs > 0
  ? new Date(resetTs).toISOString()
  : new Date(now + this.windowMs).toISOString();
```

## Gap 5: No Observability — You Cannot Operate What You Cannot See

**Severity: High**

There is no metrics emission, no structured logging, and no way to answer basic operational questions:

- What is the current 429 rate?
- Which identifiers are being throttled?
- Is Redis latency affecting request latency?
- Are there identifier distribution anomalies (one key consuming all quota)?

This is not optional for production. You need at minimum:

```javascript
// Structured logging on every check — use your existing logger (pino, winston, etc.)
// This example shows the shape; wire to your actual logger.

return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const start = Date.now();

  let result;
  try {
    result = await rateLimiter.check(identifier);
    
    const latencyMs = Date.now() - start;
    
    // Emit to your metrics system (Prometheus, Datadog, StatsD, etc.)
    // metrics.histogram('rate_limiter.check.latency_ms', latencyMs);
    // metrics.increment('rate_limiter.check.result', { allowed: result.allowed });
    
    // Structured log for throttled requests — these are operationally significant
    if (!result.allowed) {
      logger.warn({
        event: 'rate_limit_exceeded',
        identifier,          // Be careful: may be PII depending on your key strategy
        count: result.count,
        limit: result.limit,
        latencyMs,
        path: req.path,
        method: req.method,
      });
    }
  } catch (err) {
    // metrics.increment('rate_limiter.check.error');
    logger.error({ event: 'rate_limiter_redis_error', err: err.message });
    return next(); // fail open
  }

  // ... rest of middleware
};
```

**Flag for requester:** What metrics infrastructure exists? Prometheus scrape endpoint, Datadog agent, StatsD? The hooks need to be wired to your actual system.

## Gap 6: Redis Connection Has No Resilience Configuration

**Severity: High**

The Redis client is created with defaults. In production this means:

- No connection timeout — a hung connection blocks indefinitely
- Default retry strategy — may retry too aggressively or not enough
- No TLS — plaintext Redis traffic on your network
- No error event handler — unhandled `error` events crash Node.js processes

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  
  // Require TLS in production
  tls: process.env.NODE_ENV === 'production' ? {} : undefined,
  
  // Auth if your Redis requires it
  password: process.env.REDIS_PASSWORD,
  
  // Command timeout — don't let a hung Redis block requests indefinitely
  commandTimeout: 500, // ms; tune based on your p99 Redis latency + headroom
  
  // Connection timeout
  connectTimeout: 2000,
  
  // Retry strategy — exponential backoff with a ceiling
  retryStrategy(times) {
    if (times > 10) {
      // Stop retrying; let the circuit open
      return null;
    }
    return Math.min(times * 100, 2000); // ms
  },
  
  // Maximum reconnect attempts before giving up
  maxRetriesPerRequest: 1, // For rate limiting, fast failure is better than queuing
  
  // Prevent commands from queuing up during reconnection
  enableOfflineQueue: false,
});

// Required: handle connection errors explicitly
// Without this, Node.js will throw an uncaught exception and crash
redisClient.on('error', (err) => {
  logger.error({ event: 'redis_connection_error', err: err.message });
  // metrics.increment('redis.connection_error');
});

redisClient.on('reconnecting', () => {
  logger.warn({ event: 'redis_reconnecting' });
});
```

**Note on `enableOfflineQueue: false`:** With the fail-open pattern in Gap 1, this is the correct pairing. When Redis is down, commands throw immediately rather than queuing, so the catch block fires quickly and requests are not delayed.

## Gap 7: No Input Validation — Malformed Identifiers Corrupt Key Space

**Severity: Medium**

`check()` accepts any string as `identifier` and interpolates it directly into a Redis key. A null, undefined, empty string, or a string containing `:` can produce malformed keys or key collisions.

```javascript
async check(identifier) {
  // Validate before touching Redis
  if (!identifier || typeof identifier !== 'string') {
    throw new TypeError(
      `RateLimiter.check: identifier must be a non-empty string, got ${JSON.stringify(identifier)}`
    );
  }
  
  // Sanitize: strip characters that break Redis key conventions
  // or that could be used to escape the key namespace
  const safeIdentifier = identifier.trim().slice(0, 512); // Redis key limit is 512MB but keep sane
  if (!safeIdentifier) {
    throw new TypeError('RateLimiter.check: identifier is empty after sanitization');
  }
  
  const key = `${this.keyPrefix}:${safeIdentifier}`;
  // ...
}
```

## What Is Not Covered Here

These are real production concerns that require architectural decisions beyond this implementation:

- **Redis cluster / sentinel topology** — single Redis node is a single point of failure; ioredis supports both but requires explicit configuration
- **Distributed clock skew** — if multiple app servers have clock drift, sliding window boundaries are inconsistent; use Redis `TIME` command inside the Lua script if this matters
- **Key expiry under high cardinality** — millions of unique identifiers (e.g., per-IP) will accumulate keys; monitor Redis memory and set a `maxmemory-policy`
- **Burst allowance** — the current implementation has no burst headroom; a token bucket or leaky bucket may be more appropriate for some traffic patterns
- **Multi-tier limiting** — per-IP and per-user limits are often needed simultaneously; the current design applies one limit per middleware instance
