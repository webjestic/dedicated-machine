# Rate Limiter Production Readiness Review

## Summary

The core algorithm is sound — the Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT retry is handled. However, there are several gaps across reliability, security, and operability that need to be addressed before this is production-safe.

---

## Critical Issues

### 1. No Redis Failure Handling — Outage Blocks All Traffic

If Redis is unavailable, every request throws an unhandled exception. The middleware has no try/catch, so the Express error handler receives it. Depending on configuration, this either returns a 500 to the client or crashes the process.

**The missing decision:** what is the failure mode? The two options are *fail open* (allow requests through) or *fail closed* (block requests). This must be an explicit choice, not an accident.

```javascript
// Current middleware — no error handling
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const result = await rateLimiter.check(identifier); // throws on Redis failure
  // ...
};
```

```javascript
// Required: explicit failure policy
return async function rateLimitMiddleware(req, res, next) {
  let result;
  try {
    result = await rateLimiter.check(identifier);
  } catch (err) {
    // Log the error — this is an infrastructure failure, not a client error
    logger.error({ err, identifier }, 'Rate limiter check failed');

    if (options.failOpen ?? true) {
      // Fail open: allow the request, but signal degraded state
      res.set('X-RateLimit-Status', 'degraded');
      return next();
    } else {
      // Fail closed: safer for abuse prevention, worse for availability
      return res.status(503).json({ error: 'Service temporarily unavailable' });
    }
  }
  // ...
};
```

### 2. `req.ip` Is Untrustworthy Behind a Proxy

`req.ip` returns the value Express resolves, which depends on the `trust proxy` setting. Without it configured, `req.ip` is the load balancer's IP — every request shares the same rate limit bucket.

With `trust proxy` enabled but without validation, a client can spoof `X-Forwarded-For` to bypass limits entirely.

```javascript
// Attacker sends: X-Forwarded-For: 1.2.3.4
// Express reads req.ip as 1.2.3.4 — a different bucket each request
```

```javascript
// Required: configure trust proxy to match your infrastructure
// Only trust the specific number of proxy hops in your stack
app.set('trust proxy', 1); // trust one hop (e.g., one load balancer)

// For the keyBy function, validate the identifier is meaningful
keyBy: req => {
  const userId = req.headers['x-user-id'];
  // User IDs should come from verified auth tokens, not raw headers
  // A raw x-user-id header can be spoofed by any client
  return userId || req.ip;
}
```

---

### 3. `x-user-id` Header Is Caller-Controlled

The bootstrap code uses `req.headers['x-user-id']` directly as the rate limit key. Any client can set this header to any value, allowing them to rate-limit other users or use arbitrary identifiers to avoid their own limit.

```javascript
// Attacker sets x-user-id: victim@example.com
// Now the victim's account gets rate-limited, not the attacker's
```

The identifier must come from a verified source — a decoded JWT, a session, or a value injected by authenticated middleware — not from a raw request header.

```javascript
// Required: use identity established by auth middleware
keyBy: req => {
  // req.user is set by your auth middleware after token verification
  return req.user?.id || req.ip;
}
```

## Significant Issues

### 4. No Response Headers

Clients have no visibility into their rate limit state. This forces them to poll until they hit a 429, rather than backing off proactively. The standard headers (`RateLimit-*` from RFC 9110, or the older `X-RateLimit-*` convention) are expected by API clients and gateways.

```javascript
// Add to middleware after a successful check
res.set({
  'RateLimit-Limit':     result.limit,
  'RateLimit-Remaining': result.remaining,
  'RateLimit-Reset':     Math.ceil(new Date(result.resetAt).getTime() / 1000),
});

// On 429, add Retry-After (already partially done, but should match reset time)
res.set('Retry-After', Math.ceil(rateLimiter.windowMs / 1000));
```

---

### 5. Redis Connection Has No Error Handling or Reconnection Configuration

ioredis handles reconnection automatically, but without explicit configuration the defaults may not match production requirements. More importantly, connection errors are silently swallowed unless you attach a listener — Node.js will emit an unhandled `error` event and potentially crash.

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  // Missing:
  connectTimeout: 2000,
  commandTimeout: 500,       // don't let slow Redis block requests
  maxRetriesPerRequest: 1,   // fail fast rather than queue indefinitely
  retryStrategy(times) {
    if (times > 10) return null; // stop retrying after 10 attempts
    return Math.min(times * 100, 2000);
  },
  lazyConnect: true,         // don't connect until first command
});

// Required: attach error handler or the process will crash on connection errors
redisClient.on('error', (err) => {
  logger.error({ err }, 'Redis client error');
});
```

### 6. `_loadScript` Has a Race Condition Under Concurrent Load

If multiple requests arrive simultaneously before `_scriptSha` is set, each will call `_loadScript()` concurrently. This is mostly harmless (LOAD is idempotent) but wastes connections and can cause issues under high concurrency at startup.

```javascript
// Current: multiple concurrent calls possible
if (!this._scriptSha) {
  await this._loadScript(); // called N times concurrently
}
```

```javascript
// Required: serialize initialization
async _loadScript() {
  if (this._loadingPromise) return this._loadingPromise;
  this._loadingPromise = this.redis
    .script('LOAD', SLIDING_WINDOW_SCRIPT)
    .then(sha => { this._scriptSha = sha; });
  return this._loadingPromise;
}
```

### 7. `resetAt` Is Inaccurate

The current `resetAt` is `now + windowMs`, which is the time the *current* window expires — not when the *oldest entry* in the window expires. For a sliding window, the accurate reset time is when the earliest request in the window ages out. The current value overstates the wait time and misleads clients.

Fixing this accurately requires returning the oldest entry's score from the Lua script:

```lua
-- In the Lua script, return the oldest entry timestamp
local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
local reset_at = oldest[2] and (tonumber(oldest[2]) + window_ms) or (now + window_ms)
return {allowed, count, reset_at}
```

## Additional Production Requirements

### 8. No Structured Logging or Observability

There is no logging of rate limit decisions, no metrics emitted, and no way to answer operational questions like "which identifiers are being throttled?" or "is Redis latency affecting request handling?"

At minimum:
- Log 429 responses with the identifier and current count
- Emit metrics: requests allowed/denied per key prefix, Redis command latency, error rate
- Add a health check endpoint that verifies Redis connectivity

### 9. No Graceful Shutdown

The process has no SIGTERM handler. On deployment, in-flight requests may be cut off and the Redis connection is not cleanly closed.

```javascript
process.on('SIGTERM', async () => {
  logger.info('SIGTERM received, shutting down');
  server.close(async () => {
    await redisClient.quit();
    process.exit(0);
  });
});
```

### 10. Configuration Is Not Validated

If `windowMs` or `limit` are missing, undefined, or zero, the Lua script receives invalid arguments and will behave incorrectly. The constructor should validate its inputs.

```javascript
constructor({ windowMs, limit, keyPrefix = 'rl', redisClient }) {
  if (!Number.isInteger(windowMs) || windowMs <= 0)
    throw new TypeError('windowMs must be a positive integer');
  if (!Number.isInteger(limit) || limit <= 0)
    throw new TypeError('limit must be a positive integer');
  if (!redisClient)
    throw new TypeError('redisClient is required');
  // ...
}
```
