# Rate Limiter Production Readiness Review

## Summary

The core algorithm is sound — the Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT retry is handled. However, there are several gaps across reliability, security, observability, and correctness that would cause problems in production.

---

## 1. Redis Failure Handling — No Fallback Policy

**This is the most critical gap.** When Redis is unavailable, the middleware throws an unhandled error that crashes the request.

```javascript
// Current behavior: unhandled rejection propagates up
result = await this.redis.evalsha(...);
```

Production requires a deliberate policy choice:

```javascript
// In createRateLimitMiddleware:
return async function rateLimitMiddleware(req, res, next) {
  let result;
  try {
    const identifier = getIdentifier(req);
    result = await rateLimiter.check(identifier);
  } catch (err) {
    // Policy must be explicit and documented — neither option is universally correct.
    // FAIL OPEN: allow traffic, log the error, alert on-call
    logger.error({ err }, 'Rate limiter unavailable — failing open');
    metrics.increment('ratelimiter.redis_error');
    return next();

    // FAIL CLOSED: reject all traffic (protects backend, but causes outage)
    // return res.status(503).json({ error: 'Service temporarily unavailable' });
  }

  // ... rest of middleware
};
```

The choice depends on the threat model. Failing open is usually correct for rate limiting (it degrades gracefully), but it must be a conscious decision, not an accident.

## 2. Missing Rate Limit Headers

Clients need headers to implement backoff correctly. Without them, clients cannot distinguish a rate limit response from other errors and cannot schedule retries efficiently.

```javascript
// In createRateLimitMiddleware, set headers on every response:
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const result = await rateLimiter.check(identifier);

  // Standard headers (RateLimit header field draft-ietf-httpapi-ratelimit-headers)
  res.set({
    'RateLimit-Limit':     result.limit,
    'RateLimit-Remaining': result.remaining,
    'RateLimit-Reset':     Math.ceil(new Date(result.resetAt).getTime() / 1000), // Unix epoch
    'X-RateLimit-Limit':   result.limit,     // legacy clients
    'X-RateLimit-Remaining': result.remaining,
  });

  if (!result.allowed) {
    res.set('Retry-After', Math.ceil(rateLimiter.windowMs / 1000));
    return res.status(429).json({
      error: 'Too Many Requests',
      retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
    });
  }

  next();
};
```

## 3. `resetAt` Calculation Is Incorrect

The current implementation always reports `now + windowMs`, which is the far edge of the window — not when the client's oldest request expires.

```javascript
// Current — always tells the client to wait a full window, which is wrong
resetAt: new Date(now + this.windowMs).toISOString()
```

To fix this accurately, the Lua script must return the oldest entry's score:

```lua
-- Add to Lua script return values:
local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
local reset_at = oldest[2] and (tonumber(oldest[2]) + window_ms) or (now + window_ms)
return {allowed_flag, count, reset_at}
```

```javascript
// In check():
const [allowed, count, resetAtMs] = result;
return {
  // ...
  resetAt: new Date(resetAtMs).toISOString(),
};
```

## 4. Identifier Spoofing via `x-user-id` Header

Any unauthenticated client can set `x-user-id` to impersonate another user or bypass their own limit.

```javascript
// Current — trivially spoofable
keyBy: req => req.headers['x-user-id'] || req.ip,
```

The identifier must come from a verified source:

```javascript
// Correct: use the identity established by your authentication middleware
keyBy: req => {
  // req.user is set by a prior auth middleware that verified a JWT/session
  if (req.user?.id) {
    return `user:${req.user.id}`;
  }
  // Fall back to IP only if truly unauthenticated
  return `ip:${req.ip}`;
},
```

If `req.user` is not populated before the rate limiter runs, the middleware ordering is wrong and must be fixed.

## 5. `req.ip` Is Untrustworthy Behind a Proxy

```javascript
// req.ip will be the proxy's IP, not the client's — everyone shares one bucket
keyBy: req => req.ip
```

```javascript
// In Express bootstrap:
app.set('trust proxy', 1); // Trust first proxy (e.g., nginx, load balancer)
// Now req.ip correctly reflects X-Forwarded-For[0]

// For multiple proxy hops, be explicit:
app.set('trust proxy', 'loopback, linklocal, uniquelocal');
```

Note: `X-Forwarded-For` can also be spoofed by clients if not stripped at the edge. Ensure your infrastructure strips this header before it reaches the app.

## 6. No Observability

There is no way to know the system is working, degrading, or being abused.

```javascript
// Minimal instrumentation to add:

// In check():
async check(identifier) {
  const start = Date.now();
  try {
    const result = await this._doCheck(identifier);
    
    metrics.histogram('ratelimiter.check_duration_ms', Date.now() - start);
    metrics.increment('ratelimiter.decision', {
      allowed: result.allowed,
      key_prefix: this.keyPrefix,
    });

    if (!result.allowed) {
      logger.warn({ identifier, count: result.count }, 'Rate limit exceeded');
    }

    return result;
  } catch (err) {
    metrics.increment('ratelimiter.error');
    logger.error({ err, identifier }, 'Rate limiter check failed');
    throw err;
  }
}
```

Minimum alerts to configure:
- Rate of 429 responses (sudden spike = possible attack or misconfiguration)
- Redis error rate (any errors = degraded protection)
- p99 latency of `check()` (Redis slowness adds to every request)

## 7. Redis Connection Is Not Production-Configured

```javascript
// Current — no error handling, no reconnection tuning, no TLS
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
});
```

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST,
  port: Number(process.env.REDIS_PORT) || 6379,
  password: process.env.REDIS_PASSWORD,
  tls: process.env.REDIS_TLS === 'true' ? {} : undefined,

  // Prevent the app from hanging indefinitely on Redis unavailability
  connectTimeout: 2000,
  commandTimeout: 500,   // Fail fast; don't let slow Redis block requests

  // Reconnection — ioredis retries forever by default
  maxRetriesPerRequest: 1,
  retryStrategy(times) {
    if (times > 10) return null; // Stop retrying, surface the error
    return Math.min(times * 100, 2000);
  },
  
  lazyConnect: true, // Don't connect until first command; lets app start cleanly
});

redisClient.on('error', (err) => {
  logger.error({ err }, 'Redis connection error');
  metrics.increment('redis.connection_error');
});

redisClient.on('reconnecting', () => {
  logger.warn('Redis reconnecting');
});
```

## 8. Race Condition in Script Loading

The `_loadScript` method is not concurrency-safe. If two requests arrive simultaneously before the script is loaded, both will call `_loadScript` concurrently.

```javascript
// Current — two concurrent requests both call _loadScript()
if (!this._scriptSha) {
  await this._loadScript();
}
```

```javascript
// Fix: serialize with a promise
async _ensureScriptLoaded() {
  if (this._scriptSha) return;
  if (!this._loadingPromise) {
    this._loadingPromise = this.redis
      .script('LOAD', SLIDING_WINDOW_SCRIPT)
      .then(sha => {
        this._scriptSha = sha;
        this._loadingPromise = null;
      })
      .catch(err => {
        this._loadingPromise = null; // Allow retry on next call
        throw err;
      });
  }
  await this._loadingPromise;
}
```

Alternatively, load the script eagerly during application startup before accepting traffic, which is simpler and eliminates the problem entirely.

## 9. No Graceful Shutdown

The Redis connection is never closed, which can cause the process to hang on `SIGTERM` and delay deployments.

```javascript
async function shutdown(signal) {
  logger.info({ signal }, 'Shutting down');
  
  server.close(async () => {
    await redisClient.quit(); // Graceful: waits for in-flight commands
    logger.info('Shutdown complete');
    process.exit(0);
  });

  // Force exit if graceful shutdown takes too long
  setTimeout(() => {
    logger.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10_000);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT',  () => shutdown('SIGINT'));
```

## Priority Order

| Priority | Issue | Risk if unaddressed |
|---|---|---|
| P0 | Redis failure — no fallback | Unhandled rejections crash requests |
| P0 | Identifier spoofing | Rate limiting is trivially bypassed |
| P1 | `req.ip` proxy trust | All users share one rate limit bucket |
| P1 | Missing rate limit headers | Clients cannot implement correct backoff |
| P1 | No observability | Silent failures, attacks go undetected |
| P2 | `resetAt` calculation | Clients wait longer than necessary |
| P2 | Redis connection config | Hangs, no TLS, credentials in env only |
| P3 | Script loading race | Benign in practice but incorrect |
| P3 | Graceful shutdown | Deployment delays, potential data loss |
