# Rate Limiter Production Readiness Review

## Summary

The core algorithm is sound — the Lua script correctly implements an atomic sliding window, and the NOSCRIPT retry pattern is handled. However, the implementation has several gaps that would make it unsafe or operationally fragile in production.

---

## Critical Issues

### 1. Redis Failure Opens the Gate Entirely

When Redis is unavailable, `check()` throws, the middleware has no `try/catch`, and Express will either hang or return a 500. There is no defined failure policy.

```javascript
// Current middleware — unhandled rejection on Redis failure
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const result = await rateLimiter.check(identifier); // throws on Redis failure
  // ...
};
```

Production requires an explicit decision: **fail open** (allow traffic) or **fail closed** (block traffic). Neither is universally correct — it depends on the threat model — but the choice must be deliberate and the behavior must be consistent.

```javascript
return async function rateLimitMiddleware(req, res, next) {
  let result;
  try {
    result = await rateLimiter.check(identifier);
  } catch (err) {
    // Log the error with context before deciding policy
    logger.error('Rate limiter check failed', {
      identifier,
      error: err.message,
      stack: err.stack,
    });

    // FAIL OPEN: allow the request, accept the risk during degradation
    // Use this when availability is more important than strict limiting
    return next();

    // FAIL CLOSED: block the request, accept false positives during degradation
    // Use this when security is more important than availability
    // return res.status(503).json({ error: 'Service temporarily unavailable' });
  }

  // ... rest of middleware
};
```

### 2. Identifier Spoofing via `x-user-id` Header

The `keyBy` function trusts a client-supplied header without any validation. An unauthenticated caller can set `x-user-id: victim-user-123` to exhaust another user's quota, or set it to a value that bypasses limits entirely.

```javascript
// Current — fully attacker-controlled
keyBy: req => req.headers['x-user-id'] || req.ip,
```

The identifier must come from a verified source — the authenticated session, a validated JWT claim, or a value set by trusted upstream middleware — never directly from a request header.

```javascript
// The identifier should be set by auth middleware before rate limiting runs
app.use('/api', authMiddleware);        // sets req.user from verified JWT/session
app.use('/api', createRateLimitMiddleware(limiter, {
  keyBy: req => req.user?.id || req.ip, // req.user is server-controlled
}));
```

### 3. IP Address is Unreliable Behind a Proxy

`req.ip` returns the proxy's IP, not the client's IP, unless Express trust proxy is configured. Every request from behind a load balancer or CDN will share one rate limit bucket.

```javascript
// Without trust proxy, req.ip is the load balancer's IP
const app = express();
// app.set('trust proxy', ...) is never called
```

```javascript
// Set trust proxy to match your infrastructure topology
// A specific number means "trust N proxy hops"
app.set('trust proxy', 1); // trust one hop (e.g., a single load balancer)

// Or trust a specific IP range (your known proxy/CDN CIDR)
app.set('trust proxy', '10.0.0.0/8');

// Never use: app.set('trust proxy', true)
// That trusts the entire X-Forwarded-For chain, which is fully spoofable
```

### 4. No Timeout on Redis Operations

A slow or hung Redis connection will hold the request open indefinitely, exhausting the Node.js event loop and causing cascading failures.

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  // No timeouts configured
});
```

```javascript
const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  connectTimeout: 2000,       // fail fast if connection cannot be established
  commandTimeout: 500,        // abort individual commands after 500ms
  maxRetriesPerRequest: 1,    // don't retry indefinitely on failure
  enableOfflineQueue: false,  // reject commands immediately when disconnected
                              // rather than queuing them indefinitely
});
```

## Significant Issues

### 5. `resetAt` Is Inaccurate for a Sliding Window

The response claims `resetAt` is `now + windowMs`, which implies a fixed window. In a sliding window, the actual reset time is when the *oldest* request in the current window expires — which requires reading the sorted set minimum score. The current value misleads clients about when to retry.

```javascript
// Current — always reports a full window from now, regardless of actual state
resetAt: new Date(now + this.windowMs).toISOString(),
```

To return an accurate value, the Lua script must return the oldest timestamp:

```lua
-- In the Lua script, return the oldest entry score
local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
local reset_at = oldest[2] and (tonumber(oldest[2]) + window_ms) or (now + window_ms)
return {1, count + 1, reset_at}
```

```javascript
// In check(), use the value from Redis
const [allowed, count, resetAtMs] = result;
return {
  // ...
  resetAt: new Date(resetAtMs).toISOString(),
};
```

### 6. Standard Rate Limit Response Headers Are Missing

RFC 6585 and the emerging `RateLimit` header draft (used by most API gateways) define standard headers. Clients — including SDKs, monitoring tools, and browsers — expect these. Their absence makes the API harder to integrate with and harder to debug.

```javascript
// Add to middleware after a successful check
res.set({
  'RateLimit-Limit': result.limit,
  'RateLimit-Remaining': result.remaining,
  'RateLimit-Reset': result.resetAt,
});

if (!result.allowed) {
  res.set('Retry-After', Math.ceil(rateLimiter.windowMs / 1000));
  return res.status(429).json({ error: 'Too Many Requests' });
}
```

### 7. No Observability

There is no instrumentation. In production, you need to know: How often is the limiter triggered? Which identifiers are being limited? Is Redis slow? Are there errors?

```javascript
// Minimal instrumentation hooks needed on RateLimiter
async check(identifier) {
  const start = Date.now();
  try {
    const result = await this._check(identifier);
    
    metrics.increment('rate_limiter.check', {
      allowed: result.allowed,
      key_prefix: this.keyPrefix,
    });
    metrics.histogram('rate_limiter.redis_latency_ms', Date.now() - start);

    if (!result.allowed) {
      logger.warn('Rate limit exceeded', { identifier, count: result.count });
    }

    return result;
  } catch (err) {
    metrics.increment('rate_limiter.error', { key_prefix: this.keyPrefix });
    throw err;
  }
}
```

### 8. No Graceful Shutdown

The Redis connection is never explicitly closed. On process termination, in-flight commands may be dropped ungracefully, and the connection may not be returned to any pool.

```javascript
// Register shutdown handlers
async function shutdown(signal) {
  console.log(`Received ${signal}, shutting down gracefully`);
  
  server.close(async () => {
    await redisClient.quit(); // sends QUIT, waits for acknowledgment
    process.exit(0);
  });

  // Force exit if graceful shutdown takes too long
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10000);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

Invalid configuration produces confusing runtime errors far from the source.

### 10. `_loadScript` Has a Race Condition Under Concurrent Startup

If multiple requests arrive before `_scriptSha` is set, each will call `_loadScript()` concurrently. This is harmless with Redis (LOAD is idempotent) but wastes round trips. The standard fix is to memoize the in-flight promise.

```javascript
async _loadScript() {
  if (this._loadingPromise) return this._loadingPromise;
  this._loadingPromise = this.redis
    .script('LOAD', SLIDING_WINDOW_SCRIPT)
    .then(sha => {
      this._scriptSha = sha;
      this._loadingPromise = null;
    });
  return this._loadingPromise;
}
```
