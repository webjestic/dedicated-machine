# Production Readiness Review: Rate Limiter

## Executive Summary

The core algorithm is sound. The Lua script is atomic, the sliding window logic is correct, and the NOSCRIPT retry pattern is properly implemented. However, this implementation has **critical gaps** that would make it unsafe to operate in production. I've organized findings by severity and whether they're addressed here or flagged for the requester.

---

## Critical Gaps

### 1. No Redis Failure Handling — Requests Will 500

**The problem:** When Redis is unavailable, `check()` throws. The middleware has no try/catch, so Express returns a 500 to every caller. A Redis blip takes down your entire API.

**Production requirement:** A defined, explicit fail-open or fail-closed policy. Neither is universally correct — this is a product decision that must be made deliberately.

- **Fail-open** (allow requests): Correct for availability-critical APIs where false-blocking is worse than over-serving
- **Fail-closed** (block requests): Correct for fraud/abuse prevention where over-serving is unacceptable

```javascript
// In createRateLimitMiddleware — add explicit failure policy
function createRateLimitMiddleware(rateLimiter, options = {}) {
  const getIdentifier = options.keyBy || (req => req.ip);
  
  // This must be a deliberate choice, not a default that gets forgotten.
  // 'open'   = allow all requests when Redis is down
  // 'closed' = block all requests when Redis is down
  const failurePolicy = options.onRedisFailure || 'open';

  return async function rateLimitMiddleware(req, res, next) {
    const identifier = getIdentifier(req);

    let result;
    try {
      result = await rateLimiter.check(identifier);
    } catch (err) {
      // Emit the error so your monitoring sees it — never swallow silently
      req.log
        ? req.log.error({ err }, 'rate-limiter: Redis error')
        : console.error('rate-limiter: Redis error', err);

      if (failurePolicy === 'open') {
        // Degrade gracefully: allow the request, signal degraded state
        res.set('X-RateLimit-Degraded', 'true');
        return next();
      } else {
        return res.status(503).json({
          error: 'Service temporarily unavailable',
        });
      }
    }

    res.set({
      'X-RateLimit-Limit':     String(result.limit),
      'X-RateLimit-Remaining': String(result.remaining),
      'X-RateLimit-Reset':     result.resetAt,
    });

    if (!result.allowed) {
      return res.status(429).json({
        error: 'Too Many Requests',
        retryAfter: Math.ceil(rateLimiter.windowMs / 1000),
      });
    }

    next();
  };
}
```

**Flagged for requester:** Choose your failure policy before deploying. Document it. The default should not be accidental.

### 2. No Redis Connection Lifecycle Management

**The problem:** The Redis client is created at module load with no error handler, no reconnection strategy configuration, and no graceful shutdown. Node.js will print an unhandled `error` event warning; in some configurations this crashes the process.

```javascript
// Replace the bare redis instantiation in Bootstrap

const redisClient = new redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: Number(process.env.REDIS_PORT) || 6379,
  
  // Prevent ioredis from retrying forever during a hard outage.
  // Without this, the retry queue grows unbounded.
  maxRetriesPerRequest: 2,
  
  // Exponential backoff capped at 2s — avoids thundering herd on reconnect.
  retryStrategy(times) {
    if (times > 10) return null; // stop retrying, surface the error
    return Math.min(times * 100, 2000);
  },

  // Connection name appears in Redis CLIENT LIST — essential for debugging
  // which service is hammering Redis in a multi-tenant environment.
  connectionName: 'api-rate-limiter',
  
  enableOfflineQueue: false, // fail fast when disconnected; pairs with fail-open above
});

// ioredis requires an explicit error handler or Node emits an unhandled error event
redisClient.on('error', (err) => {
  // Replace with your structured logger (pino, winston, etc.)
  console.error('rate-limiter redis error:', err);
});

redisClient.on('reconnecting', () => {
  console.warn('rate-limiter redis: reconnecting');
});

// Graceful shutdown — allows in-flight requests to complete
// Wire this to your process manager (PM2, Kubernetes SIGTERM handler, etc.)
async function shutdown() {
  await redisClient.quit();
}

process.on('SIGTERM', shutdown);
process.on('SIGINT',  shutdown);
```

**Flagged for requester:** `enableOfflineQueue: false` is the right default here only if you've chosen fail-open. If fail-closed, you may want the queue — but bound its size. Decide explicitly.

### 3. `resetAt` Is Wrong — Clients Will Retry Too Early

**The problem:** `resetAt` is calculated as `now + windowMs`, which is always the far end of the window. For a sliding window, the correct reset time is when the *oldest entry* in the window expires — i.e., when the window will have room again. Returning the wrong value causes clients to retry too early and generate unnecessary 429s.

This requires returning the oldest entry timestamp from the Lua script:

```lua
-- Replace SLIDING_WINDOW_SCRIPT

local SLIDING_WINDOW_SCRIPT = `
local key        = KEYS[1]
local window_ms  = tonumber(ARGV[1])
local limit      = tonumber(ARGV[2])
local now        = tonumber(ARGV[3])
local request_id = ARGV[4]

redis.call('ZREMRANGEBYSCORE', key, 0, now - window_ms)
local count = redis.call('ZCARD', key)

if count < limit then
  redis.call('ZADD', key, now, request_id)
  redis.call('PEXPIRE', key, window_ms)
  -- No meaningful reset: window has capacity now
  return {1, count + 1, 0}
else
  -- Oldest entry score = the timestamp of the request that will expire soonest
  local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
  local oldest_ts = tonumber(oldest[2])
  -- Client should retry after this many ms
  local retry_after_ms = (oldest_ts + window_ms) - now
  return {0, count, retry_after_ms}
end
`;
```

```javascript
// Update check() to use the accurate reset time
const [allowed, count, retryAfterMs] = result;

const resetAt = allowed
  ? null  // window has capacity; no reset needed
  : new Date(now + retryAfterMs).toISOString();

return {
  allowed: allowed === 1,
  count,
  remaining: Math.max(0, this.limit - count),
  limit:     this.limit,
  resetAt,
  retryAfterMs: allowed ? 0 : retryAfterMs,
};
```

### 4. `x-user-id` Header Is Unauthenticated — Trivial Bypass

**The problem:**

```javascript
keyBy: req => req.headers['x-user-id'] || req.ip,
```

Any caller can set `x-user-id: victim-user-123` and exhaust another user's quota, or set `x-user-id: some-random-uuid` to get a fresh bucket on every request. This completely defeats per-user rate limiting.

**Production requirement:** The identifier used for rate limiting must come from a verified, authenticated source — a validated JWT claim, a session established by your auth middleware, or a verified API key.

```javascript
// Example: identifier sourced from verified auth context,
// not from an arbitrary request header.
keyBy: req => {
  // req.user must be set by your authentication middleware
  // before this rate limiter runs in the middleware chain.
  if (!req.user?.id) {
    // Fall back to IP only if you explicitly accept unauthenticated traffic.
    // Log this — it may indicate auth middleware misconfiguration.
    return `ip:${req.ip}`;
  }
  return `user:${req.user.id}`;
},
```

**Flagged for requester:** This is an architectural decision. Define who can access `/api` routes — authenticated users only, or also anonymous traffic — and ensure the middleware order enforces it. The rate limiter must run *after* auth middleware.

## Significant Gaps

### 5. No Rate-Limit Response Headers

Clients have no way to implement backoff without polling. The standard headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are missing. The `Retry-After` header on 429s is present but uses seconds-from-now rather than the accurate reset time. These are included in the failure-handling fix above (Gap 1).

---

### 6. No Timeout on Redis Operations

A slow Redis (network partition, overloaded instance) will hold Express worker slots open indefinitely. Under load, this exhausts the connection pool and hangs the server.

```javascript
// In check(), wrap the evalsha call with a timeout
async function withTimeout(promise, ms, label) {
  let timer;
  const timeout = new Promise((_, reject) => {
    timer = setTimeout(
      () => reject(new Error(`rate-limiter: ${label} timed out after ${ms}ms`)),
      ms
    );
  });
  try {
    return await Promise.race([promise, timeout]);
  } finally {
    clearTimeout(timer);
  }
}

// Inside check():
result = await withTimeout(
  this.redis.evalsha(this._scriptSha, 1, key,
    String(this.windowMs), String(this.limit), String(now), requestId),
  50,  // 50ms is a reasonable ceiling for a local/regional Redis call
  'evalsha'
);
```

### 7. `_loadScript` Has a Race Condition Under Concurrent Startup

If multiple requests arrive before `_scriptSha` is set, each will call `_loadScript()` concurrently. This is benign with ioredis (SCRIPT LOAD is idempotent) but wastes connections and adds latency on every cold-start request.

```javascript
async _loadScript() {
  // Serialize concurrent callers — only one SCRIPT LOAD in flight at a time
  if (this._loadingPromise) return this._loadingPromise;
  
  this._loadingPromise = this.redis
    .script('LOAD', SLIDING_WINDOW_SCRIPT)
    .then(sha => {
      this._scriptSha = sha;
      this._loadingPromise = null;
    })
    .catch(err => {
      this._loadingPromise = null; // allow retry on next request
      throw err;
    });

  return this._loadingPromise;
}
```

Better: call `_loadScript()` explicitly during application startup before the server begins accepting traffic, so the SHA is warm before the first request arrives.

### 8. No Observability Instrumentation

You cannot operate a rate limiter you cannot observe. At minimum, you need:

- **Metrics:** Request count by `{allowed, blocked}`, Redis operation latency histogram, Redis error rate
- **Structured logging:** Every block event with identifier, count, limit, and timestamp
- **Alerting thresholds:** Redis error rate spike (indicates outage), block rate spike (indicates attack or misconfiguration)

```javascript
// Minimal instrumentation hooks — wire to your metrics library (prom-client, DataDog, etc.)
return async function rateLimitMiddleware(req, res, next) {
  const identifier = getIdentifier(req);
  const start = Date.now();

  let result;
  try {
    result = await rateLimiter.check(identifier);
    
    metrics.histogram('ratelimit.redis.duration_ms', Date.now() - start);
    metrics.increment('ratelimit.check', { allowed: String(result.allowed) });

    if (!result.allowed) {
      // Structured log — queryable in your log aggregator
      logger.warn({
        event:      'rate_limit_exceeded',
        identifier,
        count:      result.count,
        limit:      result.limit,
        resetAt:    result.resetAt,
        path:       req.path,
        method:     req.method,
      });
    }
  } catch (err) {
    metrics.increment('ratelimit.redis.error');
    // ... failure policy handling
  }
};
```

## What Is Not Addressed Here (Requires Requester Decision)

Two gaps require architectural decisions that cannot be made unilaterally:

**Gap 4 — Authentication source for rate limit key:** The correct identifier depends on your auth model (JWT, session, API key). Define it, enforce middleware order, and ensure the rate limiter always receives a verified identity.

**Gap 8 — Observability infrastructure:** The instrumentation hooks above are stubs. Wire them to whatever metrics and logging stack you operate (Prometheus, DataDog, Honeycomb, etc.). Without this, you will not know when the rate limiter is degraded, misconfigured, or under attack.
