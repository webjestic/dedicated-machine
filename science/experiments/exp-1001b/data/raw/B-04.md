## 1. Scope and Boundaries

This service has one job: receive a POST request, validate it, enqueue it, and return 200. It does not process events. It does not query a database. It does not call downstream services synchronously. Any requirement that expands this scope is a scope change and must be explicitly acknowledged, because every addition to the request path is a new failure mode in the latency budget.

**In scope:**
- HTTP ingress
- HMAC-SHA256 signature validation
- Enqueuing valid payloads
- Returning appropriate HTTP status codes

**Out of scope:**
- Event processing logic
- Downstream consumer implementation
- Dead-letter queue consumer
- Any synchronous external call during request handling

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework:** Use a minimal HTTP framework (Fastify preferred over Express for its lower overhead and built-in schema validation infrastructure). The framework choice is not cosmetic — under burst load from a third-party sender, framework overhead compounds.

**HTTP version:** HTTP/1.1. Webhook senders from major providers (GitHub, Stripe) do not use HTTP/2. Do not introduce complexity for no gain.

**Port:** Configurable via environment variable. Default to 3000 internally. The container orchestrator maps external port. The service must not hardcode port assumptions.

**TLS termination:** Handled by the load balancer or ingress controller, not by this service. The service receives plain HTTP internally. This is a firm boundary — TLS management does not belong in application code.

**Timeouts:**
- Keep-alive timeout must be set explicitly and must be longer than the upstream load balancer's idle timeout. Failure to do this causes intermittent 502s under low traffic that are difficult to diagnose. A safe default is keep-alive at 65 seconds if the LB idle timeout is 60 seconds.
- Request timeout: set an upper bound (e.g., 10 seconds) to prevent slow-body attacks from holding connections open.

**Body size limit:** Enforce a hard maximum body size. Stripe payloads are typically under 100KB; GitHub webhook payloads are typically under 25KB. Set the limit at 512KB. Requests exceeding this limit must be rejected with 413 before signature validation. This is not a generosity decision — it is a resource protection decision.

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. This is not configurable. The algorithm is dictated by the third-party provider.

**Timing-safe comparison:** The computed HMAC digest and the digest supplied in the request header must be compared using a constant-time comparison function. A byte-by-byte short-circuit comparison leaks timing information that can be used to forge signatures. Node.js provides `crypto.timingSafeEqual`. Use it. There is no acceptable alternative.

**Signature header:** Provider-specific. GitHub uses `X-Hub-Signature-256`. Stripe uses `Stripe-Signature` with a timestamp component. The exact header name and format must be documented per provider and treated as a hard dependency. If the header is absent, the request is rejected with 401 immediately — no further processing.

**Stripe-specific consideration:** Stripe's signature includes a timestamp (`t=`) to prevent replay attacks. The receiver must extract this timestamp, verify it is within an acceptable window (Stripe recommends 300 seconds), and reject requests outside that window with 400. This timestamp check is not optional — without it, a captured valid request can be replayed indefinitely.

**GitHub-specific consideration:** GitHub sends `X-Hub-Signature-256: sha256=<hex>`. The `sha256=` prefix must be stripped before comparison.

**Raw body requirement:** HMAC validation must be performed against the raw, unparsed request body bytes — exactly as received. Any middleware that parses the body before validation is reached will invalidate the signature check. The raw body must be captured before any parsing occurs and made available to the validation step. This is the most common implementation error in webhook receivers. The order of operations is: receive raw bytes → validate signature → parse body. Not the reverse.

**Failure response:** Return 401 for missing or invalid signature. Do not return 403. Do not return a body that distinguishes between "signature missing" and "signature invalid" — that distinction is information the caller does not need and an attacker might use.

---

### 2.3 Request Body Handling

**Parsing:** JSON parsing occurs after signature validation succeeds. If JSON parsing fails, return 400. Do not enqueue malformed payloads.

**Content-Type enforcement:** Require `Content-Type: application/json`. Reject requests with incorrect content type with 415 before body parsing. This is a defense-in-depth measure, not a primary control.

**Event type extraction:** Extract the event type from the provider-specific header (e.g., `X-GitHub-Event`, Stripe's `type` field in the body). This value is included in the queue message as metadata. Downstream consumers use it for routing. If the event type cannot be determined, the message is still enqueued — the absence of a known event type is a downstream concern, not a receiver concern.

---

### 2.4 Async Offload and Queuing

**Queue technology:** Must be an external, durable message queue. Acceptable options: Redis Streams, AWS SQS, RabbitMQ (with durable queues and persistent messages), Google Cloud Pub/Sub. An in-process queue (e.g., an in-memory array or EventEmitter) is not acceptable. If the process dies, in-memory state is lost. This is not a theoretical concern — containers restart.

**Message envelope:** Each enqueued message must contain:
- Raw payload (string or buffer, not re-serialized)
- Event type
- Provider-identified event ID (e.g., `X-GitHub-Delivery`, Stripe's event `id`) — used for idempotency downstream
- Received-at timestamp (ISO 8601, UTC)
- Source IP (for audit)
- A receiver-generated UUID for internal tracing

**Enqueue operation in request path:** The enqueue call is the only I/O operation in the request path. It must have a timeout. If the queue is unavailable and the enqueue call blocks indefinitely, every inbound request will hang until the connection is dropped by the sender. The enqueue timeout must be set explicitly (recommended: 2–3 seconds). Behavior on enqueue timeout is covered in Section 3.11.

**No processing in receiver:** The receiver does not inspect event content beyond what is needed for routing metadata. It does not call any downstream service. It does not write to a database. Any logic that requires understanding the event payload belongs in a consumer, not here.

### 2.5 Response Contract

| Condition | HTTP Status | Notes |
|---|---|---|
| Valid signature, enqueued successfully | 200 | Empty body or `{"status":"ok"}` |
| Missing or invalid signature | 401 | No distinguishing detail in body |
| Body too large | 413 | |
| Malformed JSON (post-validation) | 400 | |
| Wrong Content-Type | 415 | |
| Queue unavailable | 503 | See Section 3.11 for posture decision |
| Any unhandled exception | 500 | Generic message only, no stack traces |

**On 200:** The 200 response means "received and queued." It does not mean "processed." This must be documented in the integration contract with the provider and with any internal consumers who might misinterpret it.

**Response body:** Keep it minimal. Do not reflect request data back in the response. Do not include internal identifiers that could aid enumeration.

### 2.6 Error Classification

Errors fall into three categories with different handling requirements:

**Client errors (4xx):** Log at WARN level with request metadata. Do not alert on individual occurrences. Alert on rate (see Section 3.5).

**Queue errors (503):** Log at ERROR level. Alert immediately. These represent potential event loss.

**Unhandled exceptions (500):** Log at ERROR level with full context (no stack trace in response, full stack trace in log). Alert immediately. These represent a code defect.

Unhandled promise rejections and uncaught exceptions must be caught at the process level and logged before the process exits or is restarted. Silent crashes are not acceptable.

## 3. Operational Layer

### 3.1 Secrets Management

**What is a secret:** The HMAC signing secret is a secret. It must never appear in logs, error messages, environment variable dumps, or container image layers.

**Injection method:** Inject via environment variable at runtime. The value must come from a secrets manager (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets backed by an external store). It must not be stored in the container image, in source control, or in plaintext configuration files committed to any repository.

**Rotation:** The signing secret must be rotatable without a service restart causing downtime. This requires the service to support a brief dual-validation window: during rotation, validate against both the old and new secret, accepting either. The window duration is bounded by the provider's rotation propagation time. Once rotation is confirmed complete, the old secret is removed. If the service cannot support dual-validation, the rotation procedure must document the acceptable downtime window and the procedure for re-delivery of any events rejected during that window.

**Audit:** Access to the secret in the secrets manager must be logged and auditable. This is an infrastructure requirement, not an application requirement, but it must be explicitly confirmed as in place before go-live.

---

### 3.2 Queue Durability and Failure Handling

**Durability requirements:** Messages must survive a queue node restart. This means:
- Redis Streams: AOF persistence enabled, or use a Redis cluster with replication
- RabbitMQ: durable queues, persistent message delivery mode
- SQS: durable by default; confirm the queue is not configured with a retention period too short for the consumer's processing SLA
- Pub/Sub: durable by default with subscription acknowledgement

**Acknowledgement model:** The receiver publishes to the queue. It does not consume. The enqueue operation is fire-and-confirm: the receiver waits for the queue to acknowledge receipt of the message before returning 200 to the caller. A publish-and-forget model (no ack wait) means a queue failure is invisible to the receiver and the event is silently lost.

**Dead-letter queue:** A DLQ must exist for the consumer side (messages that cannot be processed after N retries). This is a consumer-side concern, but the receiver team must confirm it exists before go-live. Events that cannot be processed must not be silently dropped.

**Queue depth monitoring:** Queue depth is a leading indicator of consumer failure. If the queue grows unboundedly, the receiver is still functioning but events are not being processed. This must be monitored independently of receiver health.

**Message retention:** Set a retention period long enough to cover the consumer's maximum expected outage plus recovery time. For most systems, 24–72 hours is a reasonable floor. The specific value must be a deliberate decision, not a default.
