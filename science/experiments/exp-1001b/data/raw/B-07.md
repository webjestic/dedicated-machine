# Operational Design Specification: Production Webhook Receiver

**Service:** Webhook Receiver
**Runtime:** Node.js (containerized)
**Version:** 1.0
**Status:** Authoritative handoff document

---

## Part I: Implementation Decisions

---

### 1. Request Ingestion

**Decision:** Accept only HTTP POST on a single dedicated route (e.g., `/webhook`). All other methods and paths return `405 Method Not Allowed` or `404 Not Found` respectively, with no body. No redirect behavior. No catch-all routing.

**Rationale:** Minimizing the attack surface. A receiver that accepts GET or exposes other routes is a receiver that will eventually be probed or misused.

**Body handling:** Raw body must be preserved as a `Buffer` before any parsing. HMAC validation operates on the raw bytes, not a re-serialized JSON object. If the framework (e.g., Express) parses the body before the signature check, the raw bytes are gone and the signature check is broken or spoofable. Use a `rawBody` middleware that captures the buffer and attaches it to the request object before any JSON parsing occurs. Do not use `express.json()` globally.

**Content-Type enforcement:** Reject requests that do not present `application/json` (or the provider-specific content type) with `415 Unsupported Media Type`. Do not attempt to process bodies of unknown type.

**Body size limit:** Enforce a hard maximum body size (recommended: 1 MB unless the provider documents larger payloads). Reject oversized requests with `413 Payload Too Large` before reading the full body. This is a denial-of-service control.

---

### 2. HMAC-SHA256 Signature Validation

**Decision:** Validate the signature on every inbound request, unconditionally. There are no exemptions, no debug bypass flags in production, no IP allowlist that substitutes for signature validation.

**Mechanism:**
- Extract the signature from the provider-specified header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).
- Compute `HMAC-SHA256(secret, rawBody)` using Node's built-in `crypto` module.
- Compare the computed digest against the extracted signature using `crypto.timingSafeEqual`. Do not use `===` or any string equality operator. Timing attacks against naive string comparison are real and documented.

**Secret management:** The HMAC secret is injected at runtime via environment variable. It is never hardcoded, never logged, never included in any diagnostic output. See Section II for secret rotation handling.

**Failure behavior:** A request that fails signature validation returns `401 Unauthorized` immediately. No event is queued. No downstream system is touched. The failure is logged with sufficient context for forensic review (timestamp, source IP, header names present, computed vs. received digest length — not the digest values themselves).

**Missing header behavior:** If the signature header is absent entirely, return `400 Bad Request`. Treat absence as distinct from mismatch; both are failures, but the distinction is operationally useful in logs.

---

### 3. Immediate 200 Response

**Decision:** The HTTP response is sent before any downstream work begins. The sequence is: validate signature → enqueue event → return `200 OK`. The enqueue operation must be fast enough that it does not materially delay the response. If the enqueue itself fails, see Section 3.1 below.

**Rationale:** Third-party providers (GitHub, Stripe) have short delivery timeout windows (typically 10–30 seconds). Any processing in the request path — database writes, downstream HTTP calls, business logic — risks timeout, retry storms, and duplicate delivery. The receiver's only job is to validate and hand off.

**Response body:** Return a minimal, static JSON body (e.g., `{"received": true}`). Do not echo back any request data. Do not include internal identifiers, queue positions, or processing status. The 200 is an acknowledgment of receipt, not a processing confirmation.

#### 3.1 Enqueue Failure Before 200

**Decision:** If the enqueue operation fails (queue unavailable, connection error), do **not** return `200`. Return `500 Internal Server Error`. The provider will retry. A `200` that silently drops an event is worse than a `500` that triggers a retry.

This is a deliberate tradeoff: a `500` may cause the provider to retry aggressively. This is acceptable. Silent data loss is not acceptable.

---

### 4. Event Queue

**Decision:** Use an external, durable message queue. Redis with a persistent list (RPUSH/BLPOP pattern) is acceptable for lower-stakes workloads. For production reliability, prefer a purpose-built queue: RabbitMQ, AWS SQS, or Google Pub/Sub depending on infrastructure context.

**Durability requirement:** The queue must persist messages across restarts of both the receiver and the queue service. In-memory queues (including Node's own EventEmitter, BullMQ without persistence, or any queue backed by non-persistent Redis) are not acceptable for production. An event that exists only in memory is an event that will be lost on the next deploy, OOM kill, or infrastructure event.

**Message envelope:** Each queued message must include:
- A unique event ID (either from the provider's headers, e.g., `X-GitHub-Delivery`, or generated by the receiver using `crypto.randomUUID()` if the provider does not supply one)
- The raw event payload (as a string or buffer)
- The event type (from provider headers, e.g., `X-GitHub-Event`)
- Ingestion timestamp (ISO 8601, UTC)
- Source IP of the delivering request

**Rationale for envelope fields:** Downstream processors need idempotency keys (event ID), routing information (event type), and audit fields (timestamp, source IP). Stripping these at ingestion forces downstream systems to reconstruct them from the payload, which is fragile and sometimes impossible.

**At-least-once delivery:** The queue must guarantee at-least-once delivery. Downstream processors must be idempotent. This is not a receiver responsibility to enforce, but it must be documented as a contract with the downstream team.

---

### 5. Concurrency and Process Model

**Decision:** Run the receiver as a single-process Node.js application. Do not use Node's `cluster` module or worker threads within the container. Horizontal scaling is handled at the container orchestration layer (see Section II). A single process per container simplifies signal handling, graceful shutdown, and health check semantics.

**Event loop protection:** The receiver must not perform any synchronous, CPU-bound work in the request path. HMAC computation on payloads up to 1 MB is fast enough to be synchronous without blocking the event loop. If payload sizes increase substantially, revisit this.

**Connection pooling:** Maintain a persistent connection pool to the queue backend. Do not open and close a new connection per request. Connection establishment latency will push the receiver past provider timeout windows under load.

---

### 6. Error Handling and Response Codes

| Condition | HTTP Response | Queue Action | Log Level |
|---|---|---|---|
| Valid signature, enqueue success | 200 | Enqueued | INFO |
| Missing signature header | 400 | None | WARN |
| Invalid signature | 401 | None | WARN |
| Unsupported Content-Type | 415 | None | WARN |
| Body too large | 413 | None | WARN |
| Queue unavailable | 500 | None | ERROR |
| Unhandled exception | 500 | None | ERROR |

No `4xx` response should ever result in an event being queued. No `5xx` response should ever return a `200`.

---

### 7. Logging

**Decision:** Structured JSON logs only. No unstructured strings. Every log line must include: timestamp (ISO 8601, UTC), log level, request ID (generated per-request, attached to all log lines for that request), event type, event ID, and outcome.

**What is never logged:** The raw HMAC secret. The computed or received HMAC digest. The full request body. PII that may appear in webhook payloads (card numbers, email addresses, etc.).

**Log verbosity:** INFO for successful ingestion. WARN for validation failures. ERROR for infrastructure failures. DEBUG level is permitted in non-production environments and must be disabled by default in production via environment variable.

## Part II: Operational Layer

---

### 8. Deployment and Container Configuration

**Decision:** The service runs as a single Docker container. The container image must be built from a minimal base (e.g., `node:20-alpine` or a distroless Node image). No development dependencies in the production image. Multi-stage build required.

**Process:** The container runs a single Node.js process as a non-root user. The image must not run as `root`. Define a dedicated user in the Dockerfile (`USER node` or equivalent). A container running as root that is compromised is a host that is compromised.

**Port:** Expose a single port (e.g., `3000`). TLS termination is handled upstream at the load balancer or ingress layer. The receiver does not terminate TLS. Do not expose port `443` from the container.

**Filesystem:** The container filesystem is read-only except for a defined `/tmp` scratch volume if needed. No writes to the container layer at runtime.

**Resource limits:** CPU and memory limits must be set at the orchestration layer. A receiver without resource limits will consume unbounded memory under a payload flood and take down co-located services. Recommended starting limits: 256 MB memory, 0.5 CPU. Tune based on observed load.

---

### 9. Environment Variables and Secret Management

The following environment variables are required at runtime. None have defaults that are safe for production. The service must fail to start if any required variable is absent.

| Variable | Purpose | Sensitivity |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC signing secret | **Secret** |
| `QUEUE_URL` | Connection string for queue backend | **Secret** |
| `PORT` | Listening port | Non-sensitive |
| `LOG_LEVEL` | Logging verbosity | Non-sensitive |
| `NODE_ENV` | Runtime environment designation | Non-sensitive |

**Secret injection:** Secrets are injected via a secrets manager (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets with external-secrets-operator, or equivalent). Secrets are never stored in environment files committed to source control, never baked into the container image, and never passed as Docker build arguments.

**Secret rotation:** When `WEBHOOK_SECRET` is rotated, there is a window during which the provider is delivering events signed with the new secret while the receiver still holds the old secret. This causes a validation failure spike. Two mitigations:

1. **Dual-secret validation:** During rotation, the receiver accepts signatures valid against either the old or new secret. This requires the receiver to support a secondary secret variable (e.g., `WEBHOOK_SECRET_PREVIOUS`) and attempt validation against both. The old secret variable is cleared after the rotation window closes (typically 24 hours or after confirming no delivery failures).
2. **Coordinated rotation:** Rotate the secret in the provider's dashboard and redeploy the receiver atomically. This is simpler but requires a deployment to be the rotation mechanism. Acceptable only if deployment time is under 60 seconds and the provider's retry window is long enough.

Choose one approach and document it. Undocumented rotation procedures become production incidents.

### 10. Health Checks

**Decision:** Expose two distinct health check endpoints.

**Liveness probe (`/healthz`):** Returns `200 OK` if the process is alive and the event loop is not blocked. This check must be trivially fast — no external calls. The orchestrator uses this to decide whether to restart the container. A liveness probe that calls the queue will cause cascading restarts when the queue is degraded.

**Readiness probe (`/readyz`):** Returns `200 OK` only if the service is ready to accept traffic — specifically, if the queue connection is established and healthy. Returns `503 Service Unavailable` if the queue is unreachable. The orchestrator uses this to decide whether to route traffic to this instance. A readiness probe that always returns `200` will route traffic to instances that cannot enqueue events, producing silent data loss.

**These two probes must not be the same endpoint.** Conflating liveness and readiness is one of the most common and damaging misconfigurations in containerized services.
