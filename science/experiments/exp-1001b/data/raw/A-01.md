# Operational Design Specification: Production Webhook Receiver

**Version:** 1.0
**Status:** Draft
**Scope:** Single containerized Node.js service receiving, validating, and queuing inbound webhook events from a third-party provider (GitHub, Stripe, or equivalent).

---

## Part I: Implementation Decisions

---

### 1. HTTP Layer

**Decision:** Use a minimal HTTP framework (Fastify preferred over Express) with no unnecessary middleware in the request path.

**Rationale:** The receiver is latency-sensitive at the ingress boundary. Every millisecond added to the request path increases the window during which the third-party provider may time out and retry. Fastify's lower per-request overhead is measurable under load.

**Constraints:**
- No session middleware, no cookie parsing, no view engine.
- Body parsing must preserve the raw request body as a `Buffer` before any JSON parsing occurs. HMAC validation is performed against the raw bytes. If the body is parsed first and re-serialized, byte-for-byte equivalence with the original payload is not guaranteed and signature validation will fail intermittently.
- The raw body buffer must be captured at the framework level, not reconstructed downstream.

**Binding:** The service binds to `0.0.0.0` on a configurable port. It does not terminate TLS directly; TLS termination is the responsibility of the load balancer or ingress layer.

---

### 2. HMAC-SHA256 Signature Validation

**Decision:** Validate the provider's HMAC-SHA256 signature on every inbound POST before any other processing. Requests that fail validation are rejected immediately with `400 Bad Request`. No event is queued for a request that has not passed validation.

**Implementation decisions:**

- **Constant-time comparison is mandatory.** Use `crypto.timingSafeEqual()` from Node's built-in `crypto` module. A naive string equality check (`===`) short-circuits on the first differing byte and is vulnerable to timing attacks that can be used to brute-force or forge signatures. This is not theoretical — it is a known attack class against webhook receivers.
- The shared secret is loaded from an environment variable at startup. It is never logged, never included in error responses, and never written to disk.
- The expected signature is computed fresh on each request from the raw body buffer and the secret. The provider-supplied signature is read from the designated header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).
- If the signature header is absent, the request is rejected with `400` before any HMAC computation is attempted. This avoids unnecessary CPU work on clearly invalid requests.
- Stripe's signature scheme includes a timestamp component to prevent replay attacks. If the provider is Stripe, the timestamp must be extracted, validated against a configurable tolerance window (default: 300 seconds), and included in the signed payload. Requests outside the tolerance window are rejected with `400`. For GitHub, replay protection is not built into the signature scheme and must be handled separately if required (see Section 4).

**Failure mode:** If the secret is not present in the environment at startup, the process must exit with a non-zero code. Operating without a secret is not a degraded mode — it is a broken configuration.

---

### 3. Immediate 200 Response

**Decision:** The `200 OK` response is sent as soon as signature validation passes and the event has been accepted into the queue. It is not sent after downstream processing completes.

**Rationale:** Third-party providers have short timeout windows (GitHub: 10 seconds, Stripe: 30 seconds). If processing is in the request path and a downstream dependency is slow or unavailable, the provider will retry. Retries under load create feedback loops. The contract with the provider ends at acknowledgment; the contract with the downstream system begins at queue acceptance.

**Constraints:**
- The response is sent after the enqueue operation completes (or is confirmed), not before. Sending `200` before the event is durably accepted into the queue creates a window where the event is lost if the process crashes between acknowledgment and enqueue.
- The response body is minimal. A JSON object with a single field (e.g., `{"status": "accepted"}`) is sufficient. No processing result, no event ID echoed back unless required by the provider's protocol.
- No response is sent on a timeout or connection drop from the provider. The service does not attempt to re-establish the connection.

---

### 4. Event Queuing

**Decision:** Valid events are written to a durable, external message queue before the `200` is returned. The queue is not in-process.

**Rationale:** An in-process queue (e.g., a Node.js `EventEmitter` or an array) does not survive process restart. A crash between receiving the event and completing downstream processing results in silent data loss. The queue must be external and durable.

**Queue selection criteria:**
- Must support at-least-once delivery guarantees.
- Must support message acknowledgment (consumer explicitly acks after successful processing; unacked messages are redelivered).
- Acceptable options: Redis Streams, RabbitMQ, AWS SQS, Google Pub/Sub, Kafka. The specific choice is deployment-context-dependent but must satisfy the durability requirement.
- In-memory queues, including BullMQ backed by a non-persistent Redis configuration, are not acceptable for production.

**Message envelope:** Each queued message includes:
- The raw event payload (as received, not re-serialized).
- The event type, extracted from the provider's header (e.g., `X-GitHub-Event`, `Stripe-Event`).
- A receiver-assigned UUID for the message (distinct from any provider-assigned event ID).
- The UTC timestamp of receipt.
- The provider-supplied event ID if present (for idempotency keying by the consumer).

**Enqueue failure handling:** If the enqueue operation fails (queue unavailable, timeout, network error), the service returns `503 Service Unavailable` to the provider. The provider is expected to retry. This is the correct behavior: the event has not been lost from the provider's perspective, and the service has not falsely acknowledged receipt. The failure must be logged and alerted on.

---

### 5. Concurrency and Process Model

**Decision:** The service runs as a single Node.js process per container instance. No worker threads are used for the request path. The event loop must not be blocked.

**Constraints:**
- HMAC computation is synchronous CPU work. For typical webhook payload sizes (under 1MB), this is acceptable on the event loop. If payload sizes are large or throughput is extremely high, this decision must be revisited and HMAC computation offloaded to a worker thread pool.
- All I/O (queue writes, any logging over the network) must be non-blocking and awaited correctly. Unhandled promise rejections must be treated as fatal.
- `process.on('unhandledRejection')` and `process.on('uncaughtException')` handlers must log the error and exit the process. Silent continuation after an unhandled exception is not acceptable in a production service.

---

### 6. Configuration

**Decision:** All configuration is supplied via environment variables. No configuration file is bundled in the container image.

**Required variables:**

| Variable | Purpose | Required |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC signing secret | Yes |
| `PORT` | HTTP listen port | Yes |
| `QUEUE_URL` / `QUEUE_DSN` | Queue connection string | Yes |
| `QUEUE_NAME` | Target queue or stream name | Yes |
| `REPLAY_TOLERANCE_SECONDS` | Timestamp replay window (Stripe) | Conditional |
| `LOG_LEVEL` | Logging verbosity | No (default: `info`) |

**Startup validation:** All required variables are validated at process startup. Missing or malformed required variables cause immediate process exit with a descriptive error message. The service does not start in a partially configured state.

## Part II: Operational Layer

---

### 7. Containerization

**Decision:** The service is packaged as a Docker image built from a minimal base (e.g., `node:20-alpine` or a distroless Node image).

**Requirements:**
- The container runs as a non-root user. Running as root inside a container is a security misconfiguration.
- No secrets are baked into the image. The image contains only the application code and its production dependencies (`npm ci --omit=dev`). Secrets are injected at runtime via the orchestrator's secret management mechanism (Kubernetes Secrets, AWS Secrets Manager, Vault, etc.).
- The image is built with a `.dockerignore` that excludes `node_modules`, `.env` files, test directories, and any local configuration.
- The image is tagged with the Git commit SHA, not `latest`. `latest` is not a version.
- A `HEALTHCHECK` instruction is defined in the Dockerfile pointing at the health endpoint (see Section 10).

---

### 8. Deployment and Scaling

**Decision:** The service is horizontally scalable. Multiple container instances can run concurrently behind a load balancer without coordination.

**Requirements:**
- Because all state is externalized to the queue, no instance-to-instance coordination is required. Each instance independently validates and enqueues.
- The load balancer must not use sticky sessions. Webhook requests from the provider are stateless; any instance can handle any request.
- Minimum instance count in production: 2. A single instance is a single point of failure.
- Autoscaling trigger: CPU utilization or request rate, not memory. Memory usage for this service is expected to be low and stable.
- Rolling deployments only. Blue-green or canary are acceptable. Hard cutover with downtime is not acceptable for a receiver that the provider will retry against.
- During rolling deployment, the old version must remain healthy until the new version passes health checks. Providers do not wait for deployments.

---

### 9. Graceful Shutdown

**Decision:** The service handles `SIGTERM` with a graceful shutdown sequence.

**Sequence:**
1. Stop accepting new connections immediately (close the HTTP server's listening socket).
2. Allow in-flight requests to complete, up to a configurable drain timeout (default: 10 seconds).
3. Close the queue connection cleanly.
4. Exit with code `0`.

**Rationale:** Container orchestrators (Kubernetes, ECS) send `SIGTERM` before `SIGKILL`. If the process exits immediately on `SIGTERM`, any request currently being processed — including the enqueue operation — is interrupted. The event may be lost or the provider may receive a connection reset and retry unnecessarily.

**Constraint:** The drain timeout must be shorter than the orchestrator's `terminationGracePeriodSeconds`. If the orchestrator kills the process before the drain completes, the graceful shutdown is moot.

---

### 10. Health Checks

**Decision:** The service exposes two distinct health endpoints. These endpoints are not authenticated.

**`GET /healthz` (Liveness):**
- Returns `200` if the process is alive and the event loop is not blocked.
- Returns `503` if the process is in a degraded state from which it cannot recover (e.g., the secret is missing, an unrecoverable internal error has occurred).
- This endpoint is used by the orchestrator to determine whether to restart the container. It must be extremely cheap — no external I/O.

**`GET /readyz` (Readiness):**
- Returns `200` if the service is ready to accept traffic: the HTTP server is listening, the queue connection is established, and the service has passed startup validation.
- Returns `503` if the queue connection is unavailable or the service is still initializing.
- This endpoint is used by the orchestrator to determine whether to route traffic to this instance. An instance that cannot reach the queue must not receive traffic, because it will return `503` to the provider on every request anyway.

**Constraint:** Health check endpoints must not be exposed to the public internet. They must be accessible only to the orchestrator and internal monitoring systems. This is enforced at the network/ingress layer, not in the application.

---

### 11. Logging

**Decision:** All logs are written to `stdout` as newline-delimited JSON. No log files. No log rotation in the container.

**Rationale:** The container runtime captures `stdout`. A centralized log aggregator (Datadog, CloudWatch, Loki, etc.) collects and indexes the output. Writing to files inside a container creates operational complexity (rotation, disk pressure, log loss on container restart) with no benefit.

**Required log fields on every log line:**

| Field | Value |
|---|---|
| `timestamp` | ISO 8601 UTC |
| `level` | `info`, `warn`, `error` |
| `service` | Service name constant |
| `version` | Git commit SHA |
| `request_id
