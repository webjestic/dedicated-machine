# Operational Design Specification: Production Webhook Receiver

**Version:** 1.0
**Status:** Draft
**Scope:** Single containerized Node.js service receiving, validating, and queuing third-party webhook events

---

## Part I: Implementation Decisions

---

### 1. HTTP Layer

**Decision:** Use a minimal HTTP framework (Fastify preferred over Express) with a single POST route mounted at a versioned path (e.g., `/v1/webhooks/:provider`).

**Rationale:** Fastify's schema validation pipeline and lower overhead are appropriate for a high-throughput, low-logic endpoint. The versioned path allows provider multiplexing and future non-breaking evolution.

**Constraints:**
- No middleware that parses the body before the raw buffer is captured. HMAC validation requires the raw request body exactly as received. Any JSON parsing that occurs before signature verification will corrupt the comparison surface.
- Body size must be capped. A hard limit of 1MB is appropriate for most providers; this must be enforced at the framework layer, not left to the OS or upstream proxy alone.
- Keep-alive connections should be enabled. Webhook senders frequently reuse connections; refusing keep-alive forces unnecessary TLS handshakes and increases sender-side error rates.

---

### 2. Signature Validation

**Decision:** Validate HMAC-SHA256 on every inbound request before any other processing. Reject without queuing on failure.

**Implementation decisions:**

- **Raw body capture:** The raw request body must be captured as a `Buffer` before any deserialization. The HMAC is computed over the exact bytes transmitted, not over a re-serialized object.
- **Constant-time comparison:** Use `crypto.timingSafeEqual` for the final comparison. String equality (`===`) is subject to early-exit behavior and is exploitable via timing oracle. This is non-negotiable.
- **Secret storage:** The signing secret must be sourced from an environment variable or a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager). It must never appear in source code, configuration files committed to version control, or log output.
- **Signature header normalization:** Provider signature headers vary (e.g., GitHub uses `X-Hub-Signature-256`, Stripe uses `Stripe-Signature` with a timestamp prefix). The validation logic must be provider-aware and must handle the specific header format for each provider, including timestamp tolerance checks where the provider embeds a timestamp in the signature payload (Stripe's scheme).
- **Timestamp tolerance (where applicable):** For providers that include a timestamp in the signed payload (Stripe), reject events where the timestamp is more than 5 minutes old. This prevents replay attacks using captured valid signatures.
- **Failure response:** Return `400 Bad Request` on signature mismatch. Do not return `401` or `403`; those imply credential-based authentication semantics. Do not include diagnostic detail in the response body — the response must not confirm whether the secret, the signature format, or the payload was the point of failure.

---

### 3. Immediate 200 Response

**Decision:** Acknowledge receipt with `200 OK` immediately after signature validation passes. All downstream work is deferred.

**Rationale:** Third-party webhook senders have aggressive timeout windows (GitHub: 10 seconds, Stripe: 30 seconds). Any synchronous downstream I/O in the request path — database writes, HTTP calls, heavy deserialization — risks timeout-induced retries, which produce duplicate events and complicate idempotency.

**Constraints:**
- The 200 must be sent before the enqueue operation completes. The enqueue is fire-and-forget from the HTTP handler's perspective.
- The response body should be minimal (e.g., `{"status":"accepted"}`). No processing status, no event ID echoed back, no downstream state reflected.
- If the enqueue itself fails after the 200 has been sent, that failure must be captured via an error handler and surfaced to the operational layer (logging, metrics, alerting). The sender will not be informed; the failure is internal.

---

### 4. Event Queue

**Decision:** Use an external durable queue. Redis with BullMQ, or a managed broker (AWS SQS, Google Pub/Sub, RabbitMQ) depending on infrastructure constraints.

**Queue selection criteria:**

| Criterion | BullMQ/Redis | SQS | RabbitMQ |
|---|---|---|---|
| Durability | Requires AOF/RDB persistence | Durable by default | Durable with mirrored queues |
| Operational burden | Moderate (Redis ops) | Low (managed) | High |
| At-least-once delivery | Yes | Yes | Yes |
| Visibility timeout / job locking | Yes | Yes | Yes |
| Dead-letter support | Yes | Yes (DLQ) | Yes |

**Decision rationale:** For a single containerized service without an existing broker, SQS is preferred if running on AWS — zero broker ops, durable, scales without configuration. BullMQ is preferred if Redis is already present in the stack. RabbitMQ is not recommended for a new deployment without existing operational expertise.

**Enqueue contract:**
- The enqueued payload must include: raw event body (as string or base64), provider identifier, event type header, received timestamp (UTC), and a generated idempotency key derived from the provider's event ID header where available.
- The enqueue call must have a timeout. If the broker is unreachable, the enqueue must fail fast (within 2–3 seconds) rather than blocking indefinitely. The 200 has already been sent; a hung enqueue ties up the Node.js event loop.
- Failed enqueues must be logged with full context and incremented on a dedicated metric counter.

---

### 5. Idempotency

**Decision:** Idempotency is enforced at the consumer layer, not the receiver layer.

**Rationale:** The receiver's responsibility ends at validated enqueue. Deduplication at the receiver would require a shared state store (Redis, database) in the hot path, adding latency and a new failure dependency. Providers guarantee at-least-once delivery; consumers must be written to handle duplicate events regardless of receiver-side deduplication.

**Receiver responsibility:** Include the provider's native event ID (e.g., GitHub's `X-GitHub-Delivery`, Stripe's event `id` field in the body) in the queued message so consumers can use it as a deduplication key.

---

### 6. Concurrency Model

**Decision:** Node.js single-threaded event loop is appropriate for this workload. No worker threads required.

**Rationale:** The receiver is I/O-bound. Signature validation is CPU-light (single HMAC computation per request). The only blocking risk is a synchronous body read; this must be handled via streaming with backpressure, not `readFileSync`-equivalent patterns.

**Connection handling:** Set explicit timeouts on the HTTP server — both `headersTimeout` and `requestTimeout` — to prevent slow-client attacks from holding connections open indefinitely.

---

### 7. Configuration

**Decision:** All configuration sourced from environment variables. No configuration files bundled in the container image.

**Required configuration surface:**
- `PORT` — listening port
- `WEBHOOK_SECRET_<PROVIDER>` — per-provider signing secret
- `QUEUE_DSN` or provider-specific queue configuration
- `LOG_LEVEL` — runtime log verbosity
- `BODY_SIZE_LIMIT` — override default if needed
- `TIMESTAMP_TOLERANCE_SECONDS` — replay window, defaulting to 300

## Part II: Operational Layer

---

### 8. Deployment and Containerization

**Decision:** Single container, single process. No process manager (PM2, forever) inside the container. The container runtime and orchestrator (Kubernetes, ECS) own process lifecycle.

**Image requirements:**
- Base image: `node:20-alpine` or distroless Node equivalent. Minimal attack surface.
- Run as a non-root user. The process must not have write access to the filesystem beyond `/tmp` if needed.
- No secrets baked into the image. Secrets injected at runtime via environment or mounted secret volumes.
- Image must pass a vulnerability scan (Trivy, Grype) as a gate in CI. Critical and high CVEs block promotion.

**Health endpoints:**
- `GET /healthz` — liveness probe. Returns 200 if the process is alive. No dependency checks.
- `GET /readyz` — readiness probe. Returns 200 only if the queue broker connection is verified reachable. Returns 503 otherwise. Kubernetes must not route traffic to a pod that cannot reach the queue.

**Startup behavior:** The service must not begin accepting webhook traffic until the readiness probe passes. Fail-fast on missing required environment variables at startup — do not start in a degraded state.

---

### 9. Logging

**Decision:** Structured JSON logging to stdout. No file-based logging. Log aggregation is the responsibility of the platform (Datadog, CloudWatch, Loki, Splunk).

**Log schema requirements:**

Every log line must include:
- `timestamp` (ISO 8601, UTC)
- `level` (error/warn/info/debug)
- `requestId` — generated per inbound request, included in all log lines for that request
- `provider` — derived from route parameter
- `eventType` — from provider-specific header
- `durationMs` — total handler duration

**What must be logged:**
- Every inbound request: provider, event type, response code, duration
- Every signature validation failure: provider, reason category (missing header / invalid format / mismatch / expired timestamp), requestId — no secret material, no raw signature value
- Every failed enqueue: provider, error message, queue target, requestId
- Startup and shutdown events

**What must never be logged:**
- Raw request body
- Signing secret or any derivative
- Full signature header values
- Any field that could constitute PII or PCI-scoped data depending on provider payload content

**Log level in production:** `info` by default. `debug` must be disableable without a deployment — controlled via `LOG_LEVEL` environment variable, readable at runtime if the framework supports it.

---

### 10. Metrics

**Decision:** Expose a Prometheus-compatible `/metrics` endpoint. Metrics must be scraped by the platform monitoring stack, not pushed.

**Required metrics:**

| Metric Name | Type | Labels | Description |
|---|---|---|---|
| `webhook_requests_total` | Counter | `provider`, `status_code` | All inbound requests by outcome |
| `webhook_signature_failures_total` | Counter | `provider`, `reason` | Validation failures by failure category |
| `webhook_enqueue_duration_ms` | Histogram | `provider`, `queue_target` | Time to enqueue after validation |
| `webhook_enqueue_failures_total` | Counter | `provider`, `queue_target` | Failed enqueue attempts |
| `webhook_handler_duration_ms` | Histogram | `provider` | End-to-end handler latency |
| `webhook_body_size_bytes` | Histogram | `provider` | Inbound payload size distribution |

**Cardinality discipline:** Labels must use bounded, known values only. Never use event IDs, request IDs, or free-form strings as label values. Unbounded cardinality will degrade the metrics store.

---

### 11. Alerting

**Decision:** Alerts are defined as code (Prometheus alerting rules or equivalent) and version-controlled alongside the service.

**Required alerts:**

- **High signature failure rate:** `webhook_signature_failures_total` rate over 5 minutes exceeds a threshold (e.g., >10/min). Indicates either a misconfigured secret rotation, a replay attack, or a provider-side bug. Severity: warning escalating to critical.
- **Enqueue failure rate:** Any non-zero sustained rate on `webhook_enqueue_failures_total`. A failed enqueue is a silent data loss event — the sender received a 200 but the event was not queued. Severity: critical.
- **Handler latency degradation:** p99 of `webhook_handler_duration_ms` exceeds 500ms. Indicates queue broker latency bleeding into the receiver. Severity: warning.
- **Pod readiness failures:** Readiness probe failing for more than 60 seconds. Indicates queue broker connectivity loss. Severity: critical.
- **Error rate spike:** 4xx or 5xx rate on `webhook_requests_total` exceeds baseline by 3 standard deviations. Severity: warning.

**Alert routing:** Enqueue failure and readiness failure alerts must page on-call immediately. Signature failure rate alerts must notify but not page unless sustained.
