## 1. Scope and Boundaries

This service receives inbound HTTP POST requests from a third-party event source (e.g., GitHub, Stripe), validates each request's authenticity via HMAC-SHA256 signature, and enqueues the raw payload for downstream processing. It does not process events. It does not call downstream services synchronously. It does not own the consumer side of the queue.

**In scope:**
- Ingress HTTP handling
- Signature validation
- Payload acceptance and enqueue
- Immediate 200 response to the sender

**Out of scope:**
- Downstream event processing logic
- Queue consumer implementation
- Business logic of any kind

Any ambiguity about what this service owns ends at the queue write. Everything after that is a different system.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework:** Express.js or Fastify. Either is acceptable. The decision must be made and fixed before implementation begins. Rationale for each:

- **Express:** Larger ecosystem, more operational precedent, middleware ecosystem is well-understood.
- **Fastify:** Lower overhead, built-in schema validation, better performance under high throughput.

**Decision driver:** If expected inbound volume exceeds ~500 req/s sustained, prefer Fastify. Otherwise Express is acceptable and carries less implementation risk.

**HTTP version:** HTTP/1.1 is sufficient. HTTP/2 is not required and adds operational complexity without benefit for this use case.

**Accepted methods:** POST only. All other methods return 405 immediately, before any body parsing or signature validation. This is not a courtesy — it prevents unnecessary computation on malformed or probing requests.

**Accepted content type:** `application/json` only, enforced at the framework level. Requests with any other `Content-Type` are rejected with 415 before body parsing.

**Endpoint path:** A single, fixed path (e.g., `/webhook`). No dynamic routing. The path must not be discoverable from any public documentation or error messages this service emits.

---

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256, as specified by the third-party provider.

**Signature location:** Read from the HTTP header specified by the provider (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name is a configuration value, not a hardcoded string, to allow provider-agnostic deployment.

**Comparison method:** Constant-time comparison using Node.js `crypto.timingSafeEqual`. Under no circumstances is string equality (`===`) used for signature comparison. This is non-negotiable. Timing attacks against HMAC validation are a known, documented attack class.

**Validation sequence:**
1. Confirm the signature header is present. If absent, reject 401.
2. Compute HMAC-SHA256 of the raw request body using the configured secret.
3. Compare computed value to the value in the header using `timingSafeEqual`.
4. If mismatch, reject 403.
5. If match, proceed.

**Validation must occur on the raw body bytes**, before any JSON parsing. This is a hard requirement. JSON parsing normalizes whitespace and key ordering, which will produce a different byte sequence than what the sender signed. The raw body must be captured before the JSON parser touches it.

**Timestamp validation (Stripe-style providers):** If the provider embeds a timestamp in the signature header (Stripe does), validate that the timestamp is within a configurable tolerance window (default: 300 seconds). Requests outside this window are rejected with 400. This prevents replay attacks. The tolerance window is a configuration value. It is not hardcoded.

**Failure behavior:** Signature failures are logged with the request ID, source IP, user agent, and the header value that was presented (truncated, not in full). They are never logged with the secret. Failures do not reveal to the caller whether the secret was wrong or the payload was tampered.

---

### 2.3 Request Body Handling

**Body size limit:** Enforced at the HTTP layer before any processing. Default limit: 1MB. This value is configurable via environment variable. Requests exceeding the limit are rejected with 413 before signature validation. Rationale: signature validation and body buffering are both CPU/memory operations. Do not perform them on arbitrarily large payloads.

**Body buffering:** The raw body must be buffered as a `Buffer` (not a string) before any other processing. The raw buffer is used for HMAC computation. The parsed JSON object is derived from the same buffer after validation passes.

**Encoding:** UTF-8 assumed. If the provider specifies a different encoding, this must be explicitly handled. Do not assume.

---

### 2.4 Async Decoupling and Queue

**Requirement:** The 200 response is sent before any downstream work begins. The queue write is not in the request path. This is the core architectural constraint.

**Queue technology decision:** This must be decided and fixed. Options and their tradeoffs:

| Option | Durability | Operational Complexity | Latency | Notes |
|---|---|---|---|---|
| Redis (BullMQ) | Moderate (AOF/RDB) | Low | Very low | Acceptable for most use cases. Redis persistence must be configured. |
| RabbitMQ | High | Moderate | Low | Better durability guarantees. More infrastructure to operate. |
| AWS SQS / GCP Pub/Sub | Very high | Low (managed) | Low–moderate | Preferred if cloud-native. No self-managed broker. |
| In-process queue (EventEmitter, etc.) | None | None | Lowest | **Not acceptable for production.** Process crash loses all queued events. |

**Decision:** In-process queuing is explicitly prohibited. Any queue technology chosen must survive a process restart without data loss. If the deployment environment is cloud-native, a managed queue service (SQS, Pub/Sub) is the preferred choice. If self-hosted, Redis with AOF persistence enabled or RabbitMQ with durable queues.

**Queue write failure handling:** If the queue write fails, the service must **not** return 200. It must return 500. The third-party sender will retry. Returning 200 when the event was not durably enqueued is a silent data loss event. This is the most dangerous failure mode in this system.

**Queue message format:** Each message must include:
- `event_id`: The unique event identifier from the provider's headers (e.g., `X-GitHub-Delivery`, `Stripe-Signature` event ID). If the provider does not supply one, generate a UUID at receipt time.
- `event_type`: The event type from the provider's headers (e.g., `X-GitHub-Event`).
- `received_at`: ISO 8601 timestamp of receipt (server time, UTC).
- `payload`: The raw JSON payload as a string, not a parsed object. Consumers deserialize. The receiver does not make parsing decisions for consumers.
- `source_ip`: The IP address of the sender.
- `headers`: A filtered subset of request headers relevant to the event type. Not all headers. Not zero headers.

**Queue message size:** Validate that the serialized message does not exceed the queue technology's per-message size limit before attempting the write. If it does, reject with 413 and log. Do not attempt to write an oversized message.

### 2.5 Response Contract

| Condition | HTTP Status | Body |
|---|---|---|
| Valid request, enqueued successfully | 200 | `{"status":"ok"}` |
| Missing or malformed signature header | 401 | `{"error":"unauthorized"}` |
| Signature mismatch | 403 | `{"error":"forbidden"}` |
| Wrong method | 405 | `{"error":"method not allowed"}` |
| Wrong content type | 415 | `{"error":"unsupported media type"}` |
| Body too large | 413 | `{"error":"payload too large"}` |
| Timestamp outside tolerance window | 400 | `{"error":"bad request"}` |
| Queue write failure | 500 | `{"error":"internal server error"}` |
| Any unhandled exception | 500 | `{"error":"internal server error"}` |

**Response bodies must not include internal error details, stack traces, queue error messages, or any information useful to an attacker.** All such detail goes to the structured log only.

**Response time target:** P99 < 100ms under normal operating conditions. The 200 path is: receive body → validate signature → write to queue → respond. All three operations are fast. If P99 exceeds 100ms, the queue write latency is the first place to investigate.

---

### 2.6 Error Handling and Rejection Policy

**Global exception handler:** Any unhandled exception that reaches the top of the call stack must be caught, logged with full context (request ID, error message, stack trace), and result in a 500 response. The process must not crash on a per-request error.

**Uncaught exception and unhandled rejection handlers:** `process.on('uncaughtException')` and `process.on('unhandledRejection')` must be registered. These handlers must log the error with maximum detail and then **exit the process**. The rationale: an uncaught exception indicates a programming error that left the process in an unknown state. Attempting to continue is more dangerous than restarting. The container orchestrator will restart the process. This is the correct behavior.

**Rejection policy is strict:** Any request that fails validation at any stage is rejected immediately. There is no partial acceptance, no best-effort processing, no fallback path that accepts an invalid request.

---

### 2.7 Payload Schema Validation

**Decision:** Lightweight structural validation after signature validation passes. This is not full JSON Schema validation of business fields — that is the consumer's responsibility. The receiver validates:
- The body is valid JSON (parse error → 400).
- The top-level structure is an object, not an array or scalar.

Nothing more. The receiver is not a business logic layer. It does not validate event-specific fields.

---

### 2.8 Idempotency

**The receiver is not responsible for idempotency.** It enqueues every valid, signed request it receives. If the third-party sender retries a delivery (which all well-behaved webhook senders do), the same event will be enqueued multiple times. Deduplication is the consumer's responsibility.

**The receiver must include the provider's event ID in every queue message** (see 2.4) precisely so that consumers can deduplicate. If the receiver discards the event ID, deduplication becomes impossible downstream.
