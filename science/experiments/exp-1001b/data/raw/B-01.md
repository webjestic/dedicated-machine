## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use a minimal HTTP framework (Express or Fastify). Do not use a full application framework. The surface area of this service is intentionally small: one POST route, one health route, one metrics route.

**Rationale:** A smaller dependency surface reduces attack surface and startup time. This service has no need for templating, session management, or routing complexity.

**Binding address:** The process binds to `0.0.0.0` on a configurable port (default `8080`). It does not bind to `443` directly. TLS termination is handled at the ingress layer (see §2.11).

**Accepted methods:** Only `POST` is accepted on the webhook endpoint. All other methods return `405 Method Not Allowed`. Requests to undefined routes return `404`. Neither response includes a body with implementation detail.

**Timeout:** An inbound request timeout of 10 seconds is enforced at the HTTP server level. A request that does not complete within this window is terminated and logged. This protects against slow-loris and stalled connections.

---

### 1.2 Signature Validation

**Decision:** HMAC-SHA256 validation is performed on every inbound request before any other processing. It is not optional, not feature-flagged, and not skippable in any environment including local development.

**Mechanism:**

1. Extract the signature from the provider-specified header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name and format are provider-specific and must be documented per integration.
2. Compute HMAC-SHA256 of the **raw request body bytes** using the shared secret.
3. Compare the computed digest against the extracted signature using a **timing-safe comparison** (`crypto.timingSafeEqual`). String equality (`===`) is never used for this comparison.
4. If the header is absent, malformed, or the comparison fails, return `401 Unauthorized` immediately. No further processing occurs. No body is returned beyond a static string. No internal detail is leaked.

**Critical constraint:** The raw body must be captured before any parsing occurs. If the body is parsed (e.g., `JSON.parse`) before the HMAC is computed, the signature will not match because whitespace and key ordering may differ from what the provider signed. The raw buffer must be preserved alongside the parsed payload.

**Secret rotation:** The service must support holding two active secrets simultaneously during a rotation window. Validation passes if either secret produces a matching digest. The old secret is removed after the rotation window closes. This prevents a hard cutover from causing dropped events.

---

### 1.3 Request Body Handling

**Decision:** Enforce a maximum request body size. The limit is set to 5 MB by default, configurable via environment variable. Requests exceeding this limit are rejected with `413 Payload Too Large` before the body is fully read.

**Rationale:** Without a body size limit, a malicious or misconfigured sender can exhaust memory. The limit must be applied at the stream level, not after buffering.

**Content-Type:** The service accepts `application/json` and `application/x-www-form-urlencoded` as dictated by the provider. Requests with unexpected content types are rejected with `415 Unsupported Media Type`. The raw body buffer is retained for HMAC computation regardless of content type.

---

### 1.4 Async Dispatch and Queuing

**Decision:** After a request passes signature validation, the event is enqueued to an external durable queue. The HTTP response is returned immediately after the enqueue call completes (or is acknowledged). Downstream processing does not occur in the request path under any circumstances.

**Queue technology:** A durable, persistent message queue is required. Acceptable choices: Amazon SQS, Google Cloud Pub/Sub, RabbitMQ with durable queues and publisher confirms, Redis Streams with consumer groups. An in-memory queue (e.g., a Node.js `EventEmitter` or array) is not acceptable. Loss of the process must not lose events.

**Enqueue failure handling:** If the enqueue operation fails (network error, queue unavailable), the service returns `503 Service Unavailable` to the caller. The third-party provider is expected to retry. The event is not silently dropped. This is the correct failure mode: it is better to return a non-200 and trigger a provider retry than to acknowledge receipt of an event that was not durably stored.

**Enqueue timeout:** The enqueue call has an explicit timeout (default: 3 seconds, configurable). If the queue does not acknowledge within this window, the call is treated as a failure and `503` is returned.

**No fire-and-forget:** The enqueue call is awaited. Returning `200` before the queue acknowledges the write means the service is lying about receipt. This is not acceptable.

---

### 1.5 Response Contract

**Decision:** The response matrix is fixed and must not deviate:

| Condition | HTTP Status |
|---|---|
| Valid signature, event enqueued | `200 OK` |
| Missing or invalid signature | `401 Unauthorized` |
| Body too large | `413 Payload Too Large` |
| Unsupported content type | `415 Unsupported Media Type` |
| Method not allowed | `405 Method Not Allowed` |
| Queue unavailable / enqueue failed | `503 Service Unavailable` |
| Unhandled internal error | `500 Internal Server Error` |

**Response body:** All responses return a minimal, static JSON body (e.g., `{"status":"ok"}` or `{"error":"unauthorized"}`). No stack traces, internal error messages, queue names, or implementation details are ever included in a response body.

**Response headers:** Remove or suppress default framework headers that disclose implementation (e.g., `X-Powered-By`). Do not add headers that reveal queue technology, internal service names, or version strings.

---

### 1.6 Idempotency

**Decision:** The receiver is responsible for deduplication at the point of enqueue. It does not assume the downstream consumer is idempotent.

**Mechanism:** Third-party providers typically include a unique delivery ID per event (e.g., `X-GitHub-Delivery`, Stripe's `id` field in the event payload). This ID is extracted and stored in a short-lived deduplication store (e.g., Redis with a TTL of 24 hours, or equivalent). Before enqueuing, the ID is checked. If it has been seen, the service returns `200` without re-enqueuing.

**TTL selection:** The deduplication TTL must exceed the provider's maximum retry window. For most providers this is 24–72 hours. The TTL is set conservatively.

**Failure mode:** If the deduplication store is unavailable, the service must decide between two failure modes and document the choice explicitly:
- **Fail closed:** Reject the request with `503`. No duplicate risk, but events may be dropped if the store is down.
- **Fail open:** Enqueue without deduplication. Downstream consumers must handle duplicates.

The default decision is **fail open with logging**. Deduplication store unavailability is a degraded state, not a reason to drop events. The downstream consumer must be designed to handle duplicates regardless.

---

### 1.7 Event Schema and Envelope

**Decision:** Events are not enqueued as raw provider payloads. Each event is wrapped in an internal envelope before being placed on the queue.

**Envelope fields (minimum required):**

| Field | Description |
|---|---|
| `envelope_id` | UUID generated by the receiver at ingest time |
| `provider` | String identifying the source (e.g., `"github"`, `"stripe"`) |
| `event_type` | Provider-specific event type extracted from headers or payload |
| `delivery_id` | Provider's own delivery/event ID, used for deduplication and tracing |
| `received_at` | ISO 8601 UTC timestamp of receipt |
| `payload` | The parsed event payload |
| `raw_body_hash` | SHA256 hash of the raw body, for downstream integrity verification if needed |

**Rationale:** Wrapping in an envelope decouples the downstream consumer from provider-specific wire formats. It also ensures traceability fields are present regardless of what the provider includes.

---

### 1.8 Configuration Management

**Decision:** All configuration is injected via environment variables. No configuration files are baked into the container image. No defaults that are safe only in development are acceptable in production.

**Required environment variables:**

| Variable | Purpose |
|---|---|
| `PORT` | HTTP bind port |
| `WEBHOOK_SECRET` | Primary HMAC signing secret |
| `WEBHOOK_SECRET_PREVIOUS` | Secondary secret during rotation (optional) |
| `QUEUE_URL` / `QUEUE_NAME` | Queue connection string or name |
| `QUEUE_TIMEOUT_MS` | Enqueue operation timeout |
| `DEDUP_STORE_URL` | Redis or equivalent connection string |
| `BODY_SIZE_LIMIT_BYTES` | Maximum accepted body size |
| `LOG_LEVEL` | Logging verbosity |
| `NODE_ENV` | Must be `production` in production |

**Validation at startup:** The service validates that all required environment variables are present and non-empty at startup. If any required variable is absent, the process exits with a non-zero code and a clear error message before binding to any port. It does not start in a degraded state.
