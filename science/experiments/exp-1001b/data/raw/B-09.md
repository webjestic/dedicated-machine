## 1. Scope and Boundaries

This service has exactly one job: receive an inbound HTTP POST, validate it, and enqueue it. It does not process events. It does not mutate state. It does not call downstream services synchronously. Any logic beyond receive-validate-enqueue is out of scope and belongs in a separate consumer service.

The boundary is the queue. What happens after enqueue is not this service's responsibility and must not be coupled to this service's availability.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Decision:** Use a minimal HTTP framework. Express is acceptable. Fastify is preferred for its lower overhead and built-in schema validation primitives. Raw `http` module is not acceptable — the operational cost of reimplementing routing, error handling, and middleware is not justified.

**Binding:** The service binds to `0.0.0.0` on a configurable port (default `3000`). It must not bind only to `localhost` inside a container.

**TLS termination:** TLS is terminated at the load balancer or ingress layer. The service itself speaks plain HTTP internally. This is non-negotiable for operational simplicity. Placing TLS inside the container creates certificate rotation complexity with no security benefit inside a private network.

**Accepted methods:** Only `POST` to the designated webhook path (e.g., `/webhook`). All other methods return `405`. All other paths return `404`. There are no other routes except health probes (see §3.4).

**Timeouts:** An inbound request timeout of 10 seconds is enforced at the HTTP server level. A sender that does not complete transmission within 10 seconds is dropped with a `408`. This prevents slow-loris style resource exhaustion.

**Body size limit:** Enforced at the framework middleware layer. Default limit: `1MB`. This must be tuned to the specific provider's documented maximum payload size. Requests exceeding the limit return `413` immediately, before any signature validation.

### 2.2 Signature Validation

**Decision:** HMAC-SHA256 validation using Node.js's built-in `crypto` module. No third-party library for this step. The standard library is sufficient and eliminates a supply chain dependency on the most security-critical path.

**Timing-safe comparison:** The computed HMAC digest and the digest provided in the request header **must** be compared using `crypto.timingSafeEqual`. String equality (`===`) is not acceptable. Timing attacks against naive string comparison are a known, practical attack vector. This is not a theoretical concern.

**Signature header:** The header name is provider-specific (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The header name is configurable, not hardcoded. The service must return `400` if the header is absent.

**Secret key:** Retrieved from environment at startup (see §3.1). If the secret is absent at startup, the process must not start. Failing open on missing configuration is not acceptable.

**Validation order:** Body size limit check → header presence check → HMAC computation → timing-safe comparison. Signature validation never runs on a request that has already been rejected for size. HMAC computation never runs before the full body is buffered.

**Raw body requirement:** Signature validation requires the raw, unmodified request body bytes. The raw body must be captured before any JSON parsing. If the body is parsed first and then re-serialized for HMAC computation, the signature will not match payloads with non-canonical JSON formatting. The raw buffer must be preserved and passed to the HMAC function directly.

**Failure response:** Any signature validation failure returns `401`. The response body contains no diagnostic detail. Do not indicate whether the failure was a missing header, wrong key, or corrupted payload. All three return the same `401`. Diagnostic detail in error responses aids attackers.

### 2.3 Request Body Handling

**Decision:** Buffer the complete raw body before any processing. Do not stream the body into the queue. The complete body must be in memory before signature validation begins.

**Encoding:** The raw body is treated as a `Buffer` (binary-safe). After successful validation, the body is parsed as JSON. If JSON parsing fails on a validated payload, the event is still enqueued with a parse-failure flag rather than rejected. The downstream consumer decides what to do with malformed-but-authenticated payloads. Rejecting them at the receiver would silently discard events the provider will not resend.

**Content-Type:** A `Content-Type: application/json` header is expected but its absence is not a hard rejection. Some providers do not set it correctly. The body is always parsed as JSON regardless.

### 2.4 Async Dispatch and Queuing

**Decision:** The queue is the durability boundary. Once the event is enqueued, the receiver's responsibility is complete. The receiver must not hold state in memory between the enqueue call and the downstream consumer's acknowledgment.

**Queue technology:** An external, persistent queue is required. In-memory queues (e.g., an in-process `EventEmitter`, a plain array) are not acceptable. They do not survive process restarts. Options in order of preference:

- **Redis with a stream or list** (Redis Streams preferred): Low operational overhead if Redis is already present in the stack. Provides persistence with AOF/RDB. Supports consumer groups for at-least-once delivery.
- **RabbitMQ**: Acceptable. Higher operational overhead. Justified if the organization already operates it.
- **AWS SQS / GCP Pub/Sub / Azure Service Bus**: Acceptable if the deployment target is cloud-native. Eliminates queue infrastructure management. Preferred for cloud deployments.
- **PostgreSQL with a LISTEN/NOTIFY or outbox table**: Acceptable as a last resort if no dedicated queue infrastructure exists. Not preferred. Adds write latency to the hot path.

**Enqueue operation:** The enqueue call is `await`-ed before the `200` is returned. This is a deliberate trade-off. The alternative — returning `200` before enqueue completes — risks acknowledging receipt of an event that was never durably stored. If the enqueue fails, the receiver returns `500` and the provider retries. This is the correct behavior.

**Enqueue timeout:** The enqueue operation has an explicit timeout (recommended: 2–3 seconds). If the queue is unreachable and the operation does not complete within the timeout, the call fails fast with a `500`. Without an explicit timeout, a hung queue connection will hold the request open until the inbound request timeout fires, degrading throughput.

**Payload envelope:** The enqueued message is not the raw body alone. It is an envelope containing:
- `receivedAt`: ISO 8601 timestamp (set by the receiver, not derived from the payload)
- `eventId`: A unique ID generated by the receiver (UUID v4) for tracing
- `providerEventId`: The event ID from the provider's header, if present (e.g., `X-GitHub-Delivery`)
- `eventType`: The event type from the provider's header, if present (e.g., `X-GitHub-Event`)
- `rawBody`: The raw body as a base64-encoded string or the parsed JSON object
- `headers`: A filtered subset of request headers (provider-specific headers only, not all headers)

Storing the raw body in the envelope ensures the downstream consumer can re-validate or re-parse without data loss.

### 2.5 Response Contract

| Condition | HTTP Status |
|---|---|
| Valid signature, enqueue succeeded | `200` |
| Body exceeds size limit | `413` |
| Signature header absent | `401` |
| Signature validation failed | `401` |
| Body is present but enqueue failed | `500` |
| Method not allowed | `405` |
| Path not found | `404` |
| Request timeout | `408` |

**Response body:** The `200` response body is minimal: `{"status":"accepted"}`. No event ID, no processing status, no queue depth. The receiver does not expose internal state in responses.

**Do not return `202`:** Some designs use `202 Accepted` to indicate async processing. This is semantically correct but operationally problematic. Several providers (including GitHub) treat any non-`200` as a soft failure and increase retry backoff. Use `200`.

### 2.6 Error Handling Strategy

**Unhandled exceptions:** A global uncaught exception handler and unhandled promise rejection handler must be registered. Both must log the error with full stack trace and then **exit the process**. Attempting to recover from an unknown state is more dangerous than restarting. The container orchestrator restarts the process. The provider retries the webhook.

**Queue connection errors:** Queue connection failures are caught and surfaced as `500` responses. They do not crash the process unless the connection cannot be re-established after the configured retry window (see §3.8).

**Validation errors:** All validation failures are caught, logged at `warn` level, and returned as the appropriate 4xx. They do not propagate to the global error handler.

### 2.7 Dependency Choices

| Concern | Choice | Rationale |
|---|---|---|
| HTTP framework | Fastify | Low overhead, built-in schema tooling |
| HMAC | Node.js `crypto` | No external dependency on security-critical path |
| Queue client | Provider-specific SDK (ioredis, amqplib, aws-sdk) | Use maintained official clients |
| Logging | `pino` | Structured JSON output, low overhead, async transport |
| Process management | Container restart policy | No PM2 or cluster inside container |
| Schema validation | Fastify's built-in (Ajv) | Request schema validation at the framework layer |

## 3. Operational Layer

### 3.1 Configuration and Secrets Management

**All configuration is injected via environment variables.** No configuration files committed to the repository. No defaults for security-sensitive values.

**Required environment variables:**

| Variable | Description | Behavior if absent |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC signing secret | Process refuses to start |
| `QUEUE_URL` or equivalent | Connection string for queue | Process refuses to start |
| `PORT` | Listening port | Default `3000` |
| `WEBHOOK_PATH` | URL path to receive webhooks | Default `/webhook` |
| `SIGNATURE_HEADER` | Header name for HMAC signature | Default to provider standard |
| `BODY_SIZE_LIMIT` | Max body size in bytes | Default `1048576` (1MB) |
| `ENQUEUE_TIMEOUT_MS` | Timeout for queue write | Default `3000` |
| `LOG_LEVEL` | Logging verbosity | Default `info` |

**Startup validation:** At process startup, before binding the HTTP port, the service validates that all required variables are present and non-empty. If any required variable is absent, the process logs the missing variable name and exits with code `1`. It does not start in a degraded state.
