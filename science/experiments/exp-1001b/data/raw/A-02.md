## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use a minimal HTTP framework — Fastify preferred over Express.

**Rationale:**
- Fastify's schema-based request handling and lower per-request overhead are appropriate for a high-throughput, low-logic receiver.
- The receiver has exactly one job: validate and enqueue. It does not need middleware ecosystems.
- Fastify's built-in JSON serialization and error handling reduce accidental complexity.

**Constraints:**
- The server must bind to a configurable port via environment variable. Default: `3000`.
- Only `POST /webhook` (or an equivalent provider-namespaced path) is exposed. All other routes return `404`. No root handler. No catch-all.
- HTTP/1.1 only. TLS termination is handled at the ingress layer, not inside the container.

---

### 1.2 Signature Validation

**Decision:** Validate HMAC-SHA256 signatures synchronously, before any other processing, using a constant-time comparison.

**Rationale:**
- Signature validation is the trust boundary. Nothing downstream executes unless this passes.
- Constant-time comparison (`crypto.timingSafeEqual`) is mandatory. String equality (`===`) is not acceptable — it is vulnerable to timing attacks.
- The raw request body must be used for signature computation, not a re-serialized parsed object. JSON serialization is not stable across implementations.

**Implementation constraints:**
- The raw body must be captured as a `Buffer` before any parsing occurs. Frameworks that parse JSON before exposing raw body require explicit configuration to preserve it.
- The `X-Hub-Signature-256` header (GitHub) or `Stripe-Signature` header (Stripe) must be present. Absence is an immediate `401`, not a `400`. The distinction is intentional: a missing signature is an authentication failure, not a malformed request.
- If the computed HMAC does not match, return `401`. Log the mismatch with enough context to diagnose (source IP, event type header if present, timestamp) but never log the raw body or the secret.
- Validation must complete within a bounded time. If the raw body exceeds the configured size limit, reject with `413` before attempting validation.

**Secret handling:**
- The HMAC secret is a single value per provider endpoint. If multi-tenancy or secret rotation is required, that is a future scope item and must not be retrofitted without a formal design. For now: one secret, one endpoint.

### 1.3 Request Handling Contract

**Decision:** The receiver returns `200 OK` with a minimal JSON body (`{"received": true}`) immediately after successful enqueue. It does not wait for downstream processing.

**Rationale:**
- Third-party providers (GitHub, Stripe) implement retry logic with exponential backoff triggered by non-2xx responses or timeouts. Holding the connection open for downstream processing introduces unnecessary coupling and timeout risk.
- The receiver's SLA is: validate the signature, enqueue the payload, return `200`. That is the complete contract.

**Response codes:**
| Condition | Response |
|---|---|
| Valid signature, enqueue succeeded | `200 OK` |
| Missing or malformed signature header | `401 Unauthorized` |
| Signature mismatch | `401 Unauthorized` |
| Body exceeds size limit | `413 Payload Too Large` |
| Malformed JSON (if content-type is JSON) | `400 Bad Request` |
| Enqueue failure (queue unavailable) | `503 Service Unavailable` |
| Any unhandled internal error | `500 Internal Server Error` |

**Critical:** `503` on enqueue failure is a deliberate choice. Returning `200` when the event was not enqueued would silently drop it. Returning `503` causes the provider to retry, which is the correct behavior.

### 1.4 Async Decoupling and Queue Integration

**Decision:** Valid events are published to a message queue immediately in the request path. The queue publish call is awaited — it is not fire-and-forget.

**Rationale:**
- Fire-and-forget publish removes the ability to return `503` on queue failure. The publish must complete (or fail) before the response is sent.
- "Async" in the requirements means downstream *processing* is async — the consuming workers are separate processes. The publish itself is synchronous within the request lifecycle.

**Queue selection:**
- Preferred: Redis Streams (via `ioredis`) or a managed queue (SQS, Cloud Pub/Sub, RabbitMQ). The choice depends on infrastructure constraints and is an environment-level decision, not a code-level one.
- The queue client must be initialized at startup and its connection health must be part of the readiness check.
- The queue client must not be instantiated per-request.

**Message envelope:**
Each enqueued message must include:
- `eventId`: derived from the provider's event ID header if available (e.g., `X-GitHub-Delivery`, Stripe's `id` field in the payload), or a generated UUID if absent.
- `eventType`: from the provider's event type header (e.g., `X-GitHub-Event`, Stripe's `type` field).
- `receivedAt`: ISO 8601 UTC timestamp assigned by the receiver at the moment of receipt.
- `source`: identifier for the provider (e.g., `"github"`, `"stripe"`).
- `payload`: the raw body as a UTF-8 string. Not re-parsed. Not modified.

**Rationale for raw payload:** Downstream consumers may need to re-validate the signature, perform their own parsing, or handle provider-specific quirks. Passing a re-serialized object introduces subtle bugs.

---

### 1.5 Payload Handling

**Decision:** Enforce a hard body size limit of 5MB. Default to 1MB unless the provider's documented maximum payload size requires otherwise.

**Rationale:**
- Unbounded body reads are a denial-of-service vector.
- GitHub webhooks are typically well under 25KB. Stripe events are similarly small. 1MB is generous. 5MB is the absolute ceiling.

**Content-Type handling:**
- Accept `application/json` and `application/x-www-form-urlencoded` (Stripe uses both depending on configuration).
- Reject other content types with `415 Unsupported Media Type`.
- Do not attempt to parse the body before signature validation. Parse after validation succeeds, if parsing is needed for envelope construction (e.g., extracting `eventType` from the payload body rather than a header).

---

### 1.6 Idempotency

**Decision:** The receiver itself does not enforce idempotency. It enqueues every valid, verified event.

**Rationale:**
- Idempotency enforcement requires state (a seen-event store). That state introduces latency, a new failure mode, and operational complexity in the request path.
- Providers guarantee at-least-once delivery, not exactly-once. Duplicate suppression belongs in the downstream consumer, not the receiver.
- The `eventId` field in the message envelope provides the key downstream consumers need to implement their own deduplication.

**Caveat:** If the queue itself provides deduplication (e.g., SQS FIFO with message deduplication ID), that mechanism should be used. The `eventId` from the envelope maps directly to the deduplication ID.

---

### 1.7 Configuration Management

**Decision:** All configuration is injected via environment variables. No configuration files are bundled in the image.

**Required environment variables:**

| Variable | Purpose | Required |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC signing secret | Yes |
| `QUEUE_URL` / `QUEUE_NAME` | Queue connection details | Yes |
| `PORT` | HTTP bind port | No (default: 3000) |
| `BODY_SIZE_LIMIT` | Max body size in bytes | No (default: 1MB) |
| `LOG_LEVEL` | Logging verbosity | No (default: `info`) |
| `NODE_ENV` | Runtime environment | Yes |

**Startup validation:** The service must validate that all required environment variables are present and non-empty at startup. If any are missing, the process must exit with a non-zero code and a clear error message before binding to the port. Silent startup with missing secrets is not acceptable.

---

### 1.8 Dependency Decisions

**Decision:** Minimize runtime dependencies. Every dependency is a surface area for vulnerabilities and operational incidents.

**Acceptable dependencies:**
- HTTP framework: Fastify
- Queue client: determined by queue choice (e.g., `ioredis`, `@aws-sdk/client-sqs`)
- Nothing else is required. `crypto` is a Node.js built-in.

**Explicitly excluded:**
- No ORM, no database client (the receiver has no persistent state of its own).
- No general-purpose HTTP client (the receiver does not make outbound calls).
- No validation library for the webhook payload (validation is the consumer's responsibility).
- No `lodash` or utility libraries.

## 2. Operational Layer

### 2.1 Containerization

**Base image:** `node:20-alpine` (or the current LTS equivalent). Alpine reduces attack surface and image size. Avoid `node:latest` — it is non-deterministic.

**Image construction constraints:**
- Run as a non-root user. Create a dedicated `node` user if the base image does not provide one. The process must not run as UID 0.
- Copy only production artifacts into the image. `node_modules` must be installed with `--omit=dev`. Source files not required at runtime (tests, local config, `.env` files) must not be present in the final image.
- Use a multi-stage build. Build stage installs all dependencies and compiles (if TypeScript is used). Production stage copies only the output and production `node_modules`.
- No secrets in the image. No `.env` files. No hardcoded credentials at any layer.
- The image must be tagged with the Git commit SHA, not just `latest`. `latest` is not a deployment tag.

**Filesystem:**
- The container filesystem should be read-only where possible. If the runtime requires a writable temp directory, mount a dedicated `tmpfs` volume. Do not write to the application directory at runtime.

---

### 2.2 Process Management and Graceful Shutdown

**Decision:** The Node.js process is PID 1 inside the container. It must handle `SIGTERM` and `SIGINT` explicitly.

**Shutdown sequence:**
1. On `SIGTERM`: stop accepting new connections immediately (close the HTTP server's listen socket).
2. Allow in-flight requests to complete. Set a hard drain timeout of 10 seconds. Any request not completed within that window is terminated.
3. Flush any buffered log output.
4. Close the queue client connection cleanly.
5. Exit with code `0`.
