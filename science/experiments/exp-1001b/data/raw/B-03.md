## 1. Scope and Boundaries

This service receives inbound POST requests from a third-party webhook provider, validates each request cryptographically, enqueues valid payloads for downstream processing, and returns a response to the caller. It does not process events. It does not own event outcomes. It owns exactly two things: **acceptance** and **enqueuing**.

Anything downstream of the queue is out of scope for this specification.

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Framework:** Use a minimal HTTP framework (Fastify preferred over Express for its lower overhead and built-in schema validation hooks). The receiver has one route. Framework weight should reflect that.

**Route:** `POST /webhook`

**Body parsing:** The raw request body — not a parsed JSON object — must be preserved and made available to the signature validation step. Many HMAC validation failures in production are caused by body parsers mutating or re-serializing the payload before the digest is computed. The raw buffer must be captured before any parsing occurs. Parse to JSON only after signature validation passes.

**Content-Type:** Accept `application/json`. Reject anything else with `415`. Do not attempt to process payloads of unknown content type.

**Request size limit:** Enforce an explicit maximum body size (recommend 1MB unless the provider's documented maximum is lower). Requests exceeding this limit must be rejected with `413` before signature validation is attempted. This prevents a class of denial-of-service where large bodies are fully buffered before rejection.

**HTTP method enforcement:** Any method other than POST to `/webhook` must return `405`. Do not return `404`. `405` communicates that the endpoint exists but the method is wrong, which matters for debugging misconfigured senders.

### 2.2 Signature Validation

**Algorithm:** HMAC-SHA256. The signature is computed over the raw request body using a shared secret. The provider-specific header name (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe) must be read and parsed per that provider's documented format.

**Comparison:** Use a constant-time comparison function for the computed digest against the received digest. Do not use `===` or any comparison that short-circuits on the first differing byte. Timing attacks against naive string comparison are a known and documented attack vector against webhook receivers.

**Failure response:** Return `401` on signature mismatch. Do not return `400`. `400` implies a malformed request; `401` implies a failed authentication check. The distinction matters for alerting and for the sender's retry logic.

**Timestamp validation (if provider supports it):** Stripe and some other providers include a timestamp in the signature header. If the provider supports it, validate that the timestamp is within an acceptable tolerance window (recommend ±5 minutes). Reject requests outside this window with `401`. This prevents replay attacks using previously captured valid requests. If the provider does not include a timestamp, document this gap explicitly and accept the replay risk as a known limitation.

**Secret rotation:** The validation logic must support reading the HMAC secret from an environment variable or secret store at startup. It must not be hardcoded. The mechanism for rotating this secret without downtime is addressed in the operational layer.

### 2.3 Request Handling and Async Decoupling

**Response timing:** The service must return `200` before any downstream operation — including queue enqueue — completes. The 200 is an acknowledgment of receipt, not an acknowledgment of processing.

**Implication:** If the queue enqueue fails, the caller has already received a `200`. This is an accepted trade-off. The alternative — holding the response until enqueue succeeds — reintroduces synchronous coupling to the queue and creates a failure mode where queue latency causes webhook timeouts and provider-side retries. The queue failure handling strategy (see Section 3.6) must compensate for this.

**Worker thread / event loop:** Enqueue operations must not block the event loop. If the queue client performs synchronous I/O, it must be wrapped appropriately. The receiver must remain responsive to new inbound requests while an enqueue is in flight.

### 2.4 Queue Integration

**Queue technology:** The queue must be an external, durable message broker — not an in-process queue, not a local file, not a database table used as a queue. Acceptable options: SQS, RabbitMQ, Redis Streams (with persistence enabled), Google Pub/Sub. The choice must reflect the deployment environment and the durability requirements. An in-process queue loses all buffered messages on process restart.

**Message envelope:** Each message published to the queue must include:
- The raw payload (or a reference to it if payloads are large)
- The event type header from the provider (e.g., `X-GitHub-Event`, `Stripe-Event`)
- A unique event ID from the provider header if available (e.g., `X-GitHub-Delivery`, `Stripe-Event-ID`)
- The receipt timestamp (time the webhook was received by this service, not the provider's timestamp)
- The provider source identifier (to support multi-provider deployments)

**Do not enrich the message further.** The receiver's job is delivery, not transformation.

**Queue client initialization:** The queue client must be initialized at service startup and validated before the HTTP server begins accepting connections. A service that starts accepting webhooks before its queue connection is established will silently drop events.

### 2.5 Error Classification

All errors must be classified before a response is returned. The classification determines the HTTP status code and the log level.

| Condition | HTTP Status | Log Level |
|---|---|---|
| Missing signature header | 401 | WARN |
| Signature mismatch | 401 | WARN |
| Timestamp out of window | 401 | WARN |
| Body too large | 413 | INFO |
| Wrong content-type | 415 | INFO |
| Wrong HTTP method | 405 | INFO |
| Malformed JSON (post-validation) | 400 | WARN |
| Queue enqueue failure | — (200 already sent) | ERROR |
| Unhandled exception | 500 | ERROR |

A `500` response to a webhook provider will typically trigger a retry. This is the correct behavior for a transient internal failure. Do not suppress 500s in an attempt to prevent retries — retries are the recovery mechanism.

### 2.6 Idempotency

The receiver does not guarantee exactly-once delivery to the queue. It guarantees at-least-once delivery on a best-effort basis. Downstream consumers must be idempotent. This must be documented in the handoff to downstream teams.

If the provider includes a unique event ID in the request headers, that ID must be included in the queue message envelope. Downstream consumers are responsible for deduplication using that ID.

The receiver itself does not perform deduplication. Adding a deduplication cache (e.g., Redis) to the receiver introduces a new dependency and a new failure mode. The cost is not justified at this layer.

## 3. Operational Layer

### 3.1 Secret Management

**Requirement:** The HMAC secret must never appear in source code, container images, or version control. It must be injected at runtime via environment variable or a secrets manager (e.g., AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets).

**Rotation procedure:** Secret rotation must be executable without service downtime. The rotation procedure is:

1. The new secret is provisioned in the secret store.
2. The service is redeployed or the secret is hot-reloaded (if hot-reload is implemented).
3. The provider is updated to use the new secret.
4. The old secret is revoked.

If hot-reload is not implemented, the rotation window — the period between step 2 and step 3 — will produce `401` errors for all inbound webhooks. This window must be minimized and the on-call team must be aware it will occur. If the provider supports dual-secret validation during rotation, implement it.

**Audit:** Access to the secret in the secret store must be logged and auditable. This is an operational control, not a code control.

### 3.2 Health and Readiness Probes

**Liveness probe:** `GET /health/live` — returns `200` if the process is running. No dependency checks. This probe determines whether the container should be restarted. It must not check queue connectivity; a queue outage should not cause the container to restart in a loop.

**Readiness probe:** `GET /health/ready` — returns `200` only if the service is ready to accept traffic. This probe must check queue client connectivity. If the queue is unreachable, this probe returns `503` and the container is removed from the load balancer rotation. This prevents the service from accepting webhooks it cannot enqueue.

**Distinction is mandatory.** Conflating liveness and readiness is a common misconfiguration that causes cascading restarts during dependency outages.

### 3.3 Graceful Shutdown

**Trigger:** `SIGTERM` signal from the container orchestrator.

**Shutdown sequence:**

1. Stop accepting new connections (close the HTTP server's listening socket).
2. Allow in-flight requests to complete. Set a hard deadline — recommend 10 seconds. Any request not completed within the deadline is terminated.
3. Flush any pending queue operations.
4. Close the queue client connection cleanly.
5. Exit with code `0`.

**If the process exits without completing this sequence** — for example, due to `SIGKILL` or an unhandled exception during shutdown — any in-flight requests at the moment of termination are lost. The provider will retry on timeout. This is acceptable. What is not acceptable is a shutdown that leaves the queue client in a state where messages are acknowledged but not delivered.

**`SIGKILL` grace period:** The container orchestrator must be configured to wait at least 15 seconds between `SIGTERM` and `SIGKILL`. This must be set explicitly. The default in Kubernetes (`terminationGracePeriodSeconds`) is 30 seconds, which is sufficient, but it must be verified — do not assume defaults.

### 3.4 Logging

**Format:** Structured JSON. Every log line must be machine-parseable. Free-text log lines are not acceptable in production.

**Required fields on every log line:**

- `timestamp` (ISO 8601, UTC)
- `level` (error / warn / info / debug)
- `service` (static: `webhook-receiver`)
- `requestId` (generated per request, included in all log lines for that request)
- `message`

**Required fields on request log lines:**

- `method`
- `path`
- `statusCode`
- `durationMs`
- `eventType` (from provider header, if present)
- `eventId` (from provider header, if present)
- `signatureValid` (boolean)

**What must not be logged:**

- The raw HMAC secret
- The full request body (payloads may contain PII or sensitive financial data)
- Full signature header values (partial logging — first 8 characters — is acceptable for debugging)

**Log destination:** Stdout only. The container runtime is responsible for log collection. The service must not write to files, must not manage log rotation, and must not ship logs directly to an external system. That is infrastructure's responsibility.
