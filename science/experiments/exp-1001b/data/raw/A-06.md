## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use Node.js `http` module directly or a minimal framework (Fastify preferred over Express).

**Rationale:**
- Fastify has lower per-request overhead and native schema validation. Express adds no meaningful value for a receiver with a single POST route.
- The service has exactly one functional route: `POST /webhook`. All other routes return `404`. A `GET /healthz` and `GET /readyz` are added for operational purposes only and are not part of the functional surface.
- HTTP/1.1 only. No HTTP/2 requirement exists; third-party senders do not require it.
- Listening port is configurable via environment variable, defaulting to `3000`. The container exposes this port; the host binding is `0.0.0.0`.
- No TLS termination at the application layer. TLS is terminated at the load balancer or ingress. The service assumes it receives plaintext internally.
- Request timeout: 10 seconds. Any sender that does not complete transmission within this window receives a connection reset. This prevents slow-loris-style resource exhaustion.
- Maximum request body size: 5 MB hard limit. Requests exceeding this are rejected with `413` before signature validation is attempted. This is enforced at the framework/body-parser layer, not in application logic.

---

### 1.2 Signature Validation

**Decision:** HMAC-SHA256 validation is mandatory, synchronous, and occurs before any other processing. A request that fails validation is dead. It is not queued, not retried, not logged with payload content.

**Mechanism:**

1. The raw request body must be read as a `Buffer` before any parsing. JSON parsing happens only after the signature is confirmed valid. This is non-negotiable: parsing before validation exposes the application to prototype pollution and other payload-based attacks.
2. The expected signature is derived by computing `HMAC-SHA256(secret, rawBody)` and encoding the result as a lowercase hex string.
3. The received signature is extracted from the provider-defined header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name and format are provider-specific and must be documented per integration.
4. Comparison uses `crypto.timingSafeEqual`. String equality (`===`) is forbidden. Timing-safe comparison is required to prevent timing oracle attacks.
5. If the header is absent, malformed, or the comparison fails, the service returns `401` immediately. No body is returned. No retry hint is given.
6. The HMAC secret is a single value per configured provider. Secret rotation is handled via the secret management layer (see §2.8); the application must support loading a new secret without a full restart.

**Failure behavior:**
- Signature failure is logged at WARN level with: timestamp, source IP, provider identifier, event type header (if present), and failure reason. The raw payload is never logged.
- Repeated signature failures from the same source IP are tracked as a metric. Alerting on this metric is defined in §2.7.

### 1.3 Request Handling Contract

**Decision:** The service returns `200 OK` to the sender as soon as the following conditions are satisfied and not one step later:

1. The request body has been fully received.
2. The HMAC signature has been validated.
3. The event has been durably enqueued (acknowledgment received from the queue, not merely handed to an in-process buffer).

**Rationale:** The `200` must not be sent before durable enqueue. Sending `200` before the event is safely in the queue creates a window where the event is lost if the process crashes. The sender will not retry because it received a `200`. This is the most common correctness mistake in webhook receiver implementations.

**Response codes:**
| Condition | Response |
|---|---|
| Valid signature, successfully enqueued | `200 OK` |
| Missing or invalid signature | `401 Unauthorized` |
| Body too large | `413 Payload Too Large` |
| Queue unavailable or enqueue timeout | `503 Service Unavailable` |
| Any unhandled internal error | `500 Internal Server Error` |
| Any route other than `POST /webhook` | `404 Not Found` |

**On `503`:** The sender is expected to retry. The response includes a `Retry-After` header set to a fixed value (e.g., 30 seconds). This is a best-effort hint; not all senders honor it.

**On `500`:** This indicates a bug or an unexpected state. It must trigger an alert (§2.7). The sender may or may not retry depending on its policy.

### 1.4 Async Decoupling and Queue Integration

**Decision:** Valid events are published to an external durable queue before `200` is returned. In-process queues (arrays, EventEmitter, worker threads) are explicitly rejected.

**Queue technology:** A managed queue service with at-least-once delivery guarantees. Candidates: AWS SQS, Google Cloud Pub/Sub, RabbitMQ with publisher confirms, Redis Streams with acknowledgment. The specific technology is an infrastructure decision outside this document, but the following properties are required:
- **Durability:** Messages must survive a broker restart.
- **At-least-once delivery:** The queue must not silently drop messages.
- **Publisher acknowledgment:** The client library must confirm that the broker has accepted the message before the application considers the enqueue successful. Fire-and-forget publish is not acceptable.
- **Backpressure signaling:** The queue client must surface errors when the broker is unavailable so the service can return `503` rather than silently dropping events.

**Message envelope:** The message published to the queue is not the raw payload alone. It is a structured envelope containing:
- `receivedAt`: ISO 8601 timestamp of receipt
- `provider`: string identifier (e.g., `"github"`, `"stripe"`)
- `eventType`: value from the provider's event type header
- `deliveryId`: provider-assigned delivery ID if present (used for idempotency downstream)
- `payload`: the validated, parsed JSON body
- `receiverVersion`: service version string

**Enqueue timeout:** The publish operation has a hard timeout of 5 seconds. If the queue does not acknowledge within this window, the service returns `503`. This timeout must be configurable.

---

### 1.5 Payload Handling

**Decision:** The raw body is retained as a `Buffer` through signature validation. JSON parsing occurs after validation succeeds. If the body is not valid JSON, the service returns `400 Bad Request`. The parsed object is used only to construct the message envelope; it is not further inspected by the receiver.

**Decision:** The receiver is payload-agnostic. It does not branch on event type, does not filter events, and does not apply business logic. All filtering and routing is the responsibility of downstream consumers. This keeps the receiver simple, stable, and independently deployable.

---

### 1.6 Idempotency

**Decision:** The receiver does not enforce idempotency. It accepts and enqueues every valid, signed request it receives, including duplicates. Idempotency is the responsibility of downstream consumers, who must use the `deliveryId` in the message envelope to deduplicate.

**Rationale:** Enforcing idempotency at the receiver requires shared state (a cache or database of seen delivery IDs), which introduces a dependency, a failure mode, and latency. The receiver's job is to validate and enqueue. Deduplication belongs at the processing layer.

---

### 1.7 Configuration Management

All configuration is supplied via environment variables. No configuration files are baked into the image. Required variables:

| Variable | Description | Required |
|---|---|---|
| `PORT` | Listening port | No (default: 3000) |
| `WEBHOOK_SECRET` | HMAC signing secret | Yes |
| `QUEUE_URL` | Queue endpoint or connection string | Yes |
| `QUEUE_PUBLISH_TIMEOUT_MS` | Enqueue timeout in milliseconds | No (default: 5000) |
| `PROVIDER` | Provider identifier string | Yes |
| `LOG_LEVEL` | Logging verbosity | No (default: `info`) |
| `SERVICE_VERSION` | Injected at build time | Yes |

The service validates the presence of all required variables at startup and exits with a non-zero code and a descriptive error message if any are missing. It does not start in a degraded state.

### 1.8 Dependency Decisions

| Concern | Decision | Rejected Alternative | Reason for Rejection |
|---|---|---|---|
| HTTP framework | Fastify | Express | Higher overhead, no advantage for single-route service |
| Signature comparison | `crypto.timingSafeEqual` (stdlib) | String equality | Timing oracle vulnerability |
| Body parsing | Raw buffer, manual JSON parse | Framework auto-parse | Must validate before parse |
| Queue client | Provider SDK with publisher confirms | In-process buffer | No durability guarantee |
| Logging | Structured JSON logger (pino) | `console.log` | Not parseable by log aggregators |
| Process manager | Container PID 1 / tini | pm2 | pm2 adds complexity; container orchestrator handles restarts |

## 2. Operational Layer

### 2.1 Containerization

**Base image:** Use an official Node.js slim image pinned to a specific digest, not a floating tag. `node:20-slim@sha256:<digest>`. Floating tags (`node:20`, `node:lts`) are forbidden in production because they change without notice and break reproducibility.

**Image construction:**
- Multi-stage build. Build stage installs all dependencies including `devDependencies`. Production stage copies only `node_modules` with production dependencies and the application source. The production image contains no build tooling.
- `.dockerignore` excludes: `.git`, `test/`, `*.md`, local `.env` files, and any credential files. These must be explicitly enumerated, not assumed.
- The application runs as a non-root user. A dedicated user (`node` or a custom `appuser`) is created in the Dockerfile with no shell and no home directory write access. Running as root inside a container is not acceptable.
- No secrets are baked into the image at any stage. The image is safe to push to a shared registry.

**Image versioning:** Images are tagged with the full Git commit SHA. `latest` is not used as a production tag. The `SERVICE_VERSION` environment variable is set at build time using the same commit SHA.

---

### 2.2 Process Management and Graceful Shutdown

**Decision:** The container runs a single Node.js process. The process is PID 1 via `tini` (or equivalent init shim) to ensure correct signal forwarding and zombie process reaping. Running Node.js as PID 1 without an init shim causes `SIGTERM` to be ignored in some container runtimes.

**Graceful shutdown sequence:**
1. The orchestrator sends `SIGTERM`.
2. The process stops accepting new connections immediately.
3. In-flight requests are allowed to complete up to a configurable drain timeout (default: 15 seconds).
4. After the drain timeout, any remaining connections are forcibly closed.
5.
