## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use a minimal HTTP framework — Fastify is preferred over Express for this workload.

**Rationale:**
- Fastify's request lifecycle is explicit and well-suited to middleware-light, high-throughput receivers.
- Its schema-based validation pipeline can be disabled selectively for the raw body route without framework fighting.
- Lower overhead per request matters when the service has a single, narrow responsibility.

**Constraints:**
- Expose exactly one route: `POST /webhook`.
- All other routes return `404`. Do not expose a catch-all.
- Bind only to the port specified by environment variable. Do not hardcode.
- Disable the `x-powered-by` equivalent header.

---

### 1.2 Signature Validation

**Decision:** Validate HMAC-SHA256 signatures using Node's built-in `crypto` module. Use `crypto.timingSafeEqual` for the comparison. Reject before any further processing.

**Rationale:**
- `timingSafeEqual` eliminates timing oracle attacks. A naive string comparison (`===`) leaks information about how many bytes match, which is exploitable.
- The built-in `crypto` module is audited, maintained, and carries no supply chain risk. No third-party HMAC library is justified.

**Mechanics:**
- Compute the expected HMAC over the raw request body bytes using the shared secret.
- Extract the provider's signature from the designated header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).
- Normalize the extracted value to strip any prefix (e.g., `sha256=`) before comparison.
- Both buffers passed to `timingSafeEqual` must be equal length. If lengths differ, the request is rejected immediately — do not pad or truncate.
- If the header is absent, reject with `400`. If the header is present but malformed, reject with `400`. If the signature does not match, reject with `401`.

**Failure mode:** If the secret is not loaded at startup, the service must not start. A missing secret is not a recoverable runtime condition.

---

### 1.3 Request Body Handling

**Decision:** Buffer the raw request body before any parsing. Enforce a hard size limit. Do not use streaming body parsing.

**Rationale:**
- HMAC validation requires the exact byte sequence as received. Any transformation — JSON parsing first, re-serialization — risks byte-level divergence and will cause signature mismatches.
- The raw buffer must be preserved and passed to the HMAC function directly.
- A hard size cap prevents memory exhaustion from oversized or malicious payloads.

**Constraints:**
- Maximum body size: `1MB`. This is configurable via environment variable but must have a hard ceiling enforced in code (e.g., `5MB` absolute maximum regardless of configuration).
- Requests exceeding the limit are rejected with `413` before signature validation.
- After signature validation, parse the buffered body as JSON. If parsing fails, reject with `400`.

---

### 1.4 Async Decoupling

**Decision:** The request handler returns `200` immediately after enqueuing the event. All downstream processing is outside the request path, entirely.

**Rationale:**
- Downstream processing latency must not affect the provider's retry behavior. If the receiver takes too long, the provider retries, producing duplicates.
- The receiver's responsibility ends at: validate, enqueue, acknowledge. Nothing else.

**Constraints:**
- The enqueue operation itself must be synchronous from the handler's perspective — meaning the handler awaits confirmation that the message was accepted by the queue, not that it was processed.
- If the enqueue call fails, the handler must not return `200`. It returns `500` or `503` depending on the failure type. The provider will retry. This is the correct behavior.
- Under no circumstances should processing logic — database writes, API calls, business logic — execute in the request handler.

---

### 1.5 Queue Interface

**Decision:** Use an external durable queue. Redis with a list-based queue (via BullMQ) or a managed queue service (SQS, Cloud Tasks) are both acceptable. An in-process queue (e.g., an in-memory array or EventEmitter) is not acceptable.

**Rationale:**
- In-process queues do not survive container restarts. Any event in memory at the time of a crash is lost permanently. This is not recoverable and not acceptable for a production webhook receiver.
- The queue must provide at-least-once delivery guarantees.

**Constraints:**
- The queue client must be initialized and its connection verified before the HTTP server begins accepting requests.
- Queue connection failures at startup are fatal. The container must exit.
- Queue connection failures at runtime are not immediately fatal — see Section 2.7 on degradation — but must be surfaced as alerts.
- Message payload written to the queue must include: raw event body (as string or base64), event type header value, provider delivery ID (if available), and receipt timestamp (ISO 8601, UTC).
- Do not write the raw HMAC secret or any derived key material to the queue payload.

---

### 1.6 Response Contract

**Decision:** The service returns a minimal, fixed response body on `200`. Error responses return a structured JSON body with a machine-readable error code.

| Condition | Status | Body |
|---|---|---|
| Valid, enqueued | `200` | `{"status":"accepted"}` |
| Missing/malformed signature header | `400` | `{"error":"invalid_signature_header"}` |
| Signature mismatch | `401` | `{"error":"signature_mismatch"}` |
| Body too large | `413` | `{"error":"payload_too_large"}` |
| Body not valid JSON | `400` | `{"error":"invalid_payload"}` |
| Queue unavailable | `503` | `{"error":"service_unavailable"}` |
| Unexpected internal error | `500` | `{"error":"internal_error"}` |

**Constraint:** Do not return stack traces, internal error messages, or queue error details in any response body. These are logged internally only.

---

### 1.7 Idempotency

**Decision:** The receiver does not enforce idempotency. It is the downstream consumer's responsibility.

**Rationale:**
- The receiver's contract is at-least-once delivery. Providers retry on non-`2xx` responses and on network failures. Deduplication at the receiver would require shared state (a seen-IDs store), which adds latency and a new failure dependency in the hot path.
- The delivery ID from the provider (e.g., `X-GitHub-Delivery`, `Stripe-Idempotency-Key`) must be forwarded in the queue message payload to enable downstream deduplication.

**Constraint:** Document this explicitly in the service README and in the consumer contract. Downstream consumers must be idempotent.

---

### 1.8 Configuration Management

**Decision:** All configuration is supplied via environment variables. No configuration files are read at runtime. No defaults are provided for security-sensitive values.

| Variable | Required | Description |
|---|---|---|
| `PORT` | Yes | HTTP bind port |
| `WEBHOOK_SECRET` | Yes | HMAC shared secret |
| `QUEUE_URL` | Yes | Queue connection string |
| `MAX_BODY_SIZE_BYTES` | No | Defaults to `1048576` (1MB) |
| `LOG_LEVEL` | No | Defaults to `info` |
| `NODE_ENV` | Yes | Must be `production` in production |

**Constraint:** At startup, validate that all required variables are present and non-empty. If any are missing, log the specific missing variable names and exit with code `1`. Do not start a partially configured service.

## 2. Operational Layer

### 2.1 Health Checks

**Two distinct endpoints are required.** They serve different purposes and must not be conflated.

#### Liveness: `GET /healthz`

- **Purpose:** Tells the container orchestrator whether the process is alive and should not be killed.
- **Logic:** Returns `200` if the Node.js event loop is responsive. No dependency checks.
- **Failure behavior:** If this endpoint is unresponsive, the orchestrator restarts the container. This is intentional.
- **Constraint:** This endpoint must never check queue connectivity. A queue outage must not cause a liveness failure and a container restart loop.

#### Readiness: `GET /readyz`

- **Purpose:** Tells the load balancer whether this instance should receive traffic.
- **Logic:** Returns `200` only if the queue connection is healthy. Returns `503` if the queue is unreachable.
- **Failure behavior:** On `503`, the load balancer removes this instance from rotation. Traffic is shed to healthy instances, not dropped.
- **Constraint:** The readiness check must use a lightweight queue ping (e.g., Redis `PING`), not a full enqueue operation. Latency of the readiness check must not exceed `200ms`.

---

### 2.2 Observability: Logging

**Decision:** Structured JSON logging only. No unstructured log lines in production.

**Required log events and their mandatory fields:**

| Event | Level | Required Fields |
|---|---|---|
| Service startup | `info` | `port`, `node_env`, `queue_url` (redacted credentials) |
| Request received | `info` | `request_id`, `method`, `path`, `content_length`, `remote_ip` |
| Signature validation failure | `warn` | `request_id`, `reason`, `provider_delivery_id` (if present) |
| Event enqueued | `info` | `request_id`, `event_type`, `provider_delivery_id`, `queue_message_id` |
| Enqueue failure | `error` | `request_id`, `error_code`, `error_message`, `queue_url` (redacted) |
| Unhandled exception | `error` | `request_id`, `stack`, `error_message` |
| Graceful shutdown initiated | `info` | `signal`, `timestamp` |

**Constraints:**
- `WEBHOOK_SECRET` must never appear in any log line. Enforce this with a startup assertion that the secret value is not present in the serialized config object logged at startup.
- `request_id` must be generated per-request (UUID v4 or equivalent) and propagated through all log lines for that request.
- Log to stdout only. Log shipping is the infrastructure's responsibility, not the application's.
- Do not log request bodies. They may contain PII or sensitive financial data.
