## 1. Scope and Boundaries

This service has exactly one job: receive an authenticated HTTP POST from a third-party provider, validate it, enqueue it, and return 200. It does not process events. It does not call downstream APIs. It does not write to a database. Any requirement that expands this scope is out of scope for this service and must be handled by a downstream consumer.

**In scope:**
- HTTP ingress on a single POST endpoint
- HMAC-SHA256 signature validation
- Enqueuing validated payloads to a durable queue
- Returning 200 (or appropriate 4xx/5xx) to the caller
- Operational observability of the above

**Out of scope:**
- Event processing logic
- Downstream consumer implementation
- Queue consumer dead-letter handling (owned by the consumer service)
- Provider-side retry policy configuration (owned by the provider relationship)

## 2. Implementation Decisions

### 2.1 HTTP Layer

**Decision:** Use Node.js with a minimal HTTP framework. Express or Fastify are both acceptable. Fastify is preferred for its schema validation primitives and lower overhead, but the choice must be made once and documented — do not mix.

**Endpoint:** A single route: `POST /webhook`. No other routes serve application logic. The path must not encode provider identity (e.g., not `/webhook/github`) unless multi-provider support is explicitly added as a versioned requirement. Encoding provider identity in the path creates operational coupling between routing infrastructure and provider configuration.

**Port:** The service listens on a single configurable port (default: 3000). TLS termination is handled by the ingress layer, not by this process. The service speaks plain HTTP internally.

**Method enforcement:** Any non-POST request to `/webhook` must return 405 Method Not Allowed. Any request to an unrecognized path must return 404. These are not errors; they are correct rejections.

**Payload size limit:** Enforce a hard maximum body size. 5MB is a reasonable upper bound for webhook payloads from providers like GitHub or Stripe. Requests exceeding this limit must be rejected with 413 before signature validation is attempted. This is a denial-of-service control, not a courtesy.

### 2.2 Signature Validation

**Decision:** Validate the HMAC-SHA256 signature on every inbound request, unconditionally. There is no bypass, no debug mode that skips validation, and no IP allowlist that substitutes for validation. Signature validation is the only authentication mechanism this service has.

**Algorithm:** HMAC-SHA256. The shared secret is provided by the third-party provider at webhook registration time and stored as a secret (see §4.4). The signature is computed over the raw request body bytes, not a parsed representation.

**Comparison:** Use a constant-time comparison function (`crypto.timingSafeEqual` in Node.js). A naive string equality check (`===`) is vulnerable to timing attacks. This is not theoretical — it is a known attack class against HMAC verification.

**Header location:** The provider dictates which header carries the signature. For GitHub this is `X-Hub-Signature-256`. For Stripe this is `Stripe-Signature`. The header name must be configurable, not hardcoded. If the header is absent, return 401 immediately. Do not attempt validation against an empty value.

**Failure response:** A signature mismatch returns 401 Unauthorized. The response body must be minimal — do not echo back any detail about what was expected or received. Log the failure internally (see §4.5).

**Timestamp validation (provider-dependent):** Stripe's signature scheme embeds a timestamp in the header and expects the receiver to reject payloads older than a configurable tolerance (default: 5 minutes). This replay protection must be implemented if the provider includes it. GitHub does not include a timestamp; replay protection for GitHub depends on idempotency handling (see §2.6). The implementation must not assume all providers behave identically.

### 2.3 Request Body Handling

**Decision:** The raw request body bytes must be captured before any parsing occurs. HMAC is computed over the raw bytes. If the framework parses the body before the signature check runs, the original byte sequence is lost and the signature cannot be verified correctly. This is a common implementation defect.

**Mechanism:** Buffer the raw body as a `Buffer` (not a string) at the middleware layer. Store it on the request object for use by the signature validator. Only after successful validation should the body be parsed as JSON for enqueuing.

**Content-Type:** Require `Content-Type: application/json`. Reject other content types with 415 Unsupported Media Type. If a provider sends a different content type (e.g., `application/x-www-form-urlencoded` for some Stripe events), this must be explicitly accommodated with a documented exception, not a permissive wildcard.

**Malformed JSON:** If the body passes signature validation but fails JSON parsing, return 400 Bad Request. Do not enqueue unparseable payloads. Log the raw body (truncated, see §4.5) for debugging.

### 2.4 Async Decoupling and Queue

**Decision:** After successful validation, the payload is enqueued and a 200 is returned. The enqueue operation is the only work done in the request path beyond validation. There is no processing, no enrichment, and no conditional logic applied to the payload before enqueuing.

**Queue technology:** A durable, persistent queue is required. Redis with a list or stream (via `LPUSH`/`XADD`) is acceptable for moderate throughput. RabbitMQ or a managed service (AWS SQS, Google Cloud Pub/Sub) is preferred for production workloads where message durability guarantees must survive broker restarts without operator intervention. The queue technology must be decided before deployment and must not be changed without a migration plan. In-memory queues (e.g., a Node.js `EventEmitter` or an in-process array) are explicitly prohibited — they do not survive process restarts.

**Enqueue failure handling:** If the enqueue operation fails (queue unavailable, timeout, connection error), the service must return 500 to the caller. It must not return 200. Returning 200 when the payload has not been durably queued is a data loss event. The provider's retry mechanism is the recovery path for this scenario, and it only activates on non-2xx responses.

**Enqueue timeout:** The enqueue call must have an explicit timeout (suggested: 2–3 seconds). A hung queue connection must not hold the HTTP response open indefinitely. If the timeout is exceeded, treat it as a queue failure and return 500.

**Message envelope:** The message placed on the queue must include:
- The parsed event payload
- The raw body (as a base64-encoded string or equivalent) for downstream re-validation if needed
- The event type header value from the provider (e.g., `X-GitHub-Event`, `Stripe-Event`)
- A receiver-assigned UUID for tracing
- The UTC timestamp of receipt
- The provider identifier (if multi-provider support is ever added, this field must exist from day one)

Do not strip fields from the provider payload. Downstream consumers decide what they need.

### 2.5 Response Contract

The response contract is fixed and must not deviate:

| Condition | HTTP Status | Body |
|---|---|---|
| Valid signature, enqueued successfully | 200 | `{"status":"ok"}` |
| Missing or malformed signature header | 401 | `{"error":"unauthorized"}` |
| Signature mismatch | 401 | `{"error":"unauthorized"}` |
| Body exceeds size limit | 413 | `{"error":"payload too large"}` |
| Unsupported Content-Type | 415 | `{"error":"unsupported media type"}` |
| Malformed JSON (post-validation) | 400 | `{"error":"bad request"}` |
| Queue unavailable | 500 | `{"error":"service unavailable"}` |
| Unhandled exception | 500 | `{"error":"internal error"}` |

The response body must always be JSON. The response must always include `Content-Type: application/json`. Error messages must not include stack traces, internal service names, queue addresses, or any detail that aids an adversary.

### 2.6 Idempotency

**Decision:** This service does not enforce idempotency. It enqueues every valid, authenticated request it receives. Idempotency is the responsibility of the downstream consumer, which has access to provider-supplied event IDs (e.g., GitHub's `X-GitHub-Delivery` header, Stripe's `id` field in the event object).

**Rationale:** Enforcing idempotency at the receiver requires state (a seen-event store), which introduces a dependency and a failure mode. The receiver's job is to accept and enqueue. Deduplication at the consumer is architecturally cleaner and does not block ingress.

**Required:** The provider-supplied event ID header must be extracted and included in the message envelope (see §2.4) to enable downstream deduplication.

### 2.7 Configuration Management

All configuration is injected via environment variables. There are no configuration files in the container image. There is no runtime configuration reload — a configuration change requires a redeployment.

**Required environment variables:**

| Variable | Description | Sensitive |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC shared secret from provider | Yes |
| `QUEUE_URL` | Connection string for queue backend | Yes |
| `QUEUE_NAME` | Queue or topic name | No |
| `PORT` | HTTP listen port | No |
| `SIGNATURE_HEADER` | Header name carrying the signature | No |
| `MAX_BODY_BYTES` | Maximum request body size in bytes | No |
| `ENQUEUE_TIMEOUT_MS` | Timeout for queue write operations | No |
| `LOG_LEVEL` | Logging verbosity | No |
| `NODE_ENV` | Runtime environment identifier | No |

The service must fail to start if any required sensitive variable is absent or empty. It must not start with a missing `WEBHOOK_SECRET` and fall back to a permissive mode. There is no permissive mode.

### 2.8 Dependency Choices

**HTTP framework:** Fastify (preferred) or Express. Chosen once. Documented.

**Queue client:** Determined by queue technology selection. Must support connection pooling, explicit timeouts, and reconnection with backoff.

**Logging library:** Pino. Structured JSON output. Not Winston, not console.log.

**No ORM, no database client, no authentication library beyond Node.js built-in `crypto`.** Every dependency added to this service is an attack surface and a failure mode. The dependency list must be minimal and audited.
