## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use Node.js with a minimal HTTP framework. Express or Fastify are both acceptable. Fastify is preferred for its schema-based request handling and lower overhead, but this decision is not load-sensitive at typical webhook volumes. The framework choice must not introduce middleware that buffers, transforms, or parses the raw request body before the signature validation step.

**Binding:** The service binds to `0.0.0.0` on a single configurable port (default `8080`). It does not terminate TLS directly. TLS termination is handled by the ingress layer or load balancer in front of it.

**Endpoint surface:** Exactly one route is exposed for inbound webhooks: `POST /webhook`. All other routes return `404`. A `/healthz` and `/readyz` route are also exposed (see Section 2.3). No other surface area exists.

**HTTP version:** HTTP/1.1 is sufficient. HTTP/2 is not required and adds no value for this use case.

---

### 1.2 Signature Validation

**Decision:** HMAC-SHA256 validation is performed on every inbound request without exception. There are no bypass paths, no IP allowlists that substitute for signature validation, and no environment flags that disable it. Signature validation is the security boundary. It must be unconditional.

**Algorithm:**
1. Read the raw request body as a `Buffer` before any parsing occurs.
2. Read the signature from the provider-specified header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name is provider-specific and must be documented in the deployment configuration.
3. Compute `HMAC-SHA256(secret, rawBody)` using Node.js's built-in `crypto` module. No third-party crypto library is used.
4. Compare the computed digest against the received signature using `crypto.timingSafeEqual`. String equality (`===`) is explicitly prohibited. Timing attacks against naive string comparison are a known exploit class against HMAC-validated webhooks.
5. If the comparison fails, return `401` immediately. Do not enqueue. Do not log the body. Do not log the received signature value.

**Signature format:** Some providers (Stripe) include a timestamp in the signature header and expect the timestamp to be incorporated into the HMAC input. If the provider does this, the timestamp must be validated against a configurable tolerance window (default: 5 minutes). Requests outside the tolerance window are rejected with `400`. This prevents replay attacks. The tolerance window is a configuration value, not a hardcoded constant.

**Secret storage:** The HMAC secret is never in source code, never in environment variables baked into the image, and never logged. It is injected at runtime via a secrets manager (see Section 2.1).

---

### 1.3 Request Body Handling

**Decision:** The raw request body must be captured as a `Buffer` before any framework middleware parses it. This is non-negotiable. JSON parsing middleware that runs before signature validation will produce a re-serialized string that does not byte-for-byte match the original payload, causing all signatures to fail.

**Implementation pattern:** Register a raw body capture middleware that reads the stream into a `Buffer` and attaches it to the request object. JSON parsing, if needed for downstream envelope construction, runs after signature validation and operates on the already-captured buffer.

**Size limit:** Enforce a maximum request body size. The limit must be set before the body is fully read into memory. A configurable limit of `1MB` is the default. Requests exceeding this limit are rejected with `413` before signature validation. This prevents memory exhaustion from oversized payloads. The limit must be validated against the maximum payload size documented by the third-party provider.

**Content-Type:** Accept `application/json` and `application/x-www-form-urlencoded` as required by the provider. Reject other content types with `415`. This is a defense-in-depth measure, not a primary security control.

---

### 1.4 Async Processing Architecture

**Decision:** The request handler does exactly three things in the request path: validate the signature, enqueue the event, and return `200`. No business logic executes in the request path. No database reads or writes occur in the request path. No downstream HTTP calls occur in the request path.

**Rationale:** The third-party provider has a response timeout. If the receiver does not respond within that timeout (typically 5–30 seconds depending on provider), the provider will retry. Retries create duplicate events. The only way to reliably prevent timeout-induced duplicates is to make the request path as fast as possible. The queue write is the only I/O permitted in the request path, and it must complete within a tight internal deadline (see Section 1.5).

**No in-process worker:** The receiver service does not contain a worker that processes events from the queue. Processing is the responsibility of a separate downstream consumer. This service is a receiver only. Coupling the receiver and the consumer in the same process creates a failure mode where consumer slowness or failure causes the receiver to back up and miss its response deadline.

---

### 1.5 Queue Integration

**Decision:** The queue is an external durable message broker. Acceptable choices, in order of operational preference: AWS SQS, Google Cloud Pub/Sub, RabbitMQ (with durable queues and publisher confirms), Redis Streams (with AOF persistence enabled). An in-memory queue (e.g., a Node.js `EventEmitter` or array) is explicitly prohibited. In-process queues do not survive process restarts and create silent data loss.

**Publish semantics:** The service publishes to the queue synchronously within the request path. The queue write must complete (or fail) before the `200` is returned. This is intentional: if the queue write fails, the service must return a non-`200` response so the provider retries. Returning `200` before confirming the enqueue would silently drop events.

**Queue write timeout:** The queue client must have a hard timeout on the publish call. Default: `3000ms`. If the queue does not acknowledge within this window, the publish is treated as failed. The request returns `503`. The provider will retry.

**At-least-once delivery:** The queue must be configured for at-least-once delivery. Exactly-once delivery is not assumed. Idempotency handling is the responsibility of the downstream consumer, not this service (see Section 1.8).

**Message contents:** The message placed on the queue is an envelope (see Section 1.7). The raw body is included in the envelope. The downstream consumer must not need to call back to this service or to the provider to reconstruct the event.

---

### 1.6 Response Contract

The response contract is fixed and must not be changed without provider coordination:

| Condition | HTTP Status | Body |
|---|---|---|
| Valid signature, enqueued successfully | `200` | `{"status":"ok"}` |
| Missing or malformed signature header | `401` | `{"error":"unauthorized"}` |
| Signature validation failure | `401` | `{"error":"unauthorized"}` |
| Timestamp outside tolerance window | `400` | `{"error":"bad_request"}` |
| Body exceeds size limit | `413` | `{"error":"payload_too_large"}` |
| Unsupported content type | `415` | `{"error":"unsupported_media_type"}` |
| Queue write failed or timed out | `503` | `{"error":"service_unavailable"}` |
| Unhandled internal error | `500` | `{"error":"internal_error"}` |

**Critical constraint:** `200` is returned if and only if the event has been durably enqueued. Any condition that prevents durable enqueue must return a non-`2xx` status. This is the contract that enables provider retry behavior to function correctly.

Error response bodies must not include stack traces, internal error messages, queue connection details, or any information useful to an attacker. They must not vary in structure. The structure above is fixed.

---

### 1.7 Event Schema and Envelope

**Decision:** The message placed on the queue is a structured envelope, not the raw body alone. The envelope includes:

- `event_id`: A unique identifier for this delivery attempt. Sourced from the provider's delivery ID header if available (e.g., `X-GitHub-Delivery`, `Stripe-Idempotency-Key`). If the provider does not supply one, generate a UUID v4 at receive time.
- `received_at`: ISO 8601 timestamp of when the receiver accepted the request, in UTC.
- `provider`: String identifying the webhook source (e.g., `"github"`, `"stripe"`). Configured at deploy time.
- `event_type`: The event type header value from the provider (e.g., `X-GitHub-Event`, Stripe's `type` field). Extracted before enqueue.
- `content_type`: The `Content-Type` header of the original request.
- `raw_body`: The raw body as a base64-encoded string. Base64 is used to avoid encoding ambiguity in queue message serialization.
- `headers`: A filtered subset of request headers. Include only headers documented as meaningful by the provider. Do not forward all headers. Do not forward `Authorization` headers, cookies, or any header not explicitly required by downstream consumers.

**Rationale for including raw body:** Downstream consumers may need the raw body for their own signature verification, audit logging, or parsing. Stripping it at the receiver forces consumers to depend on a secondary source of truth.

---
