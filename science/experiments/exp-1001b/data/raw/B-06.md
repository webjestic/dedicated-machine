## 1. Implementation Decisions

### 1.1 HTTP Server and Framework

**Decision:** Use Node.js with Express or Fastify. Either is acceptable. The choice must be made once and documented; do not mix.

**Rationale:** The receiver has a narrow surface area — one POST route, one health route, and one metrics route. Framework overhead is not a concern. What matters is that the chosen framework does not silently buffer or transform the raw request body before signature validation has access to it. This is a non-negotiable constraint that eliminates any middleware stack that processes the body before the validation layer sees the raw bytes.

**Constraints:**
- The HTTP server must listen on a single configurable port. Default: `3000`. Exposed via environment variable `PORT`.
- No session middleware, no cookie parser, no view engine. None of these belong in this service.
- The server must not expose framework version headers or `X-Powered-By` headers. These must be explicitly disabled.
- Maximum request body size must be enforced at the HTTP layer before any processing occurs. Default ceiling: `1MB`. This is configurable but must have a hard upper bound. A third-party provider sending a 500MB payload must be rejected before the process allocates memory to parse it.

---

### 1.2 Signature Validation

**Decision:** HMAC-SHA256 validation using the raw request body bytes and a shared secret. Validation is synchronous, happens before any other processing, and is the first thing that executes after the body is received.

**Mechanism:**
1. Extract the signature from the provider-specific header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The exact header name is provider-specific and must be documented per integration.
2. Read the raw body as a `Buffer`. The body must not be parsed, decoded, or transformed before this step.
3. Compute `HMAC-SHA256(secret, rawBody)` and encode as hex.
4. Compare the computed signature against the received signature using a **timing-safe comparison** (`crypto.timingSafeEqual`). Standard string equality (`===`) is not acceptable. Timing attacks against HMAC validation are a known, documented threat.
5. If the comparison fails, return `401` immediately. Do not enqueue. Do not log the received signature value. Log only that validation failed and the request metadata (timestamp, source IP, event type header if present).

**Failure modes to handle explicitly:**
- Missing signature header → `401`, logged as `SIGNATURE_MISSING`.
- Malformed signature (not valid hex, wrong length) → `401`, logged as `SIGNATURE_MALFORMED`. Do not attempt comparison.
- Signature present but invalid → `401`, logged as `SIGNATURE_INVALID`.
- Secret not loaded at startup → service must refuse to start. This is a startup-time assertion, not a runtime condition.

**Secret rotation:** The validation layer must support accepting two secrets simultaneously during a rotation window — the outgoing secret and the incoming secret. If either secret produces a valid signature, the request is accepted. This is not optional. Secret rotation without dual-validation support requires a maintenance window, which is operationally unacceptable for a production receiver.

---

### 1.3 Request Body Handling

**Decision:** The raw body must be captured as a `Buffer` before any parsing occurs. This is a hard constraint imposed by the HMAC validation requirement.

**Mechanism:** Use a raw body parser middleware that stores the `Buffer` on the request object before any JSON parsing. If JSON parsing is needed downstream (for event type routing or logging), it is performed on the already-validated buffer, not as a pre-validation step.

**Constraints:**
- Do not use `express.json()` or equivalent as the first middleware. It will consume the stream and return a parsed object, discarding the raw bytes needed for HMAC computation.
- The raw body must be stored on the request object under a consistent, documented property name (e.g., `req.rawBody`).
- If body reading fails (stream error, timeout, size limit exceeded), return `400`. Log the failure. Do not proceed.
- Body read timeout must be enforced. A provider that opens a connection and streams the body slowly must not be allowed to hold a server socket indefinitely. Default: `10 seconds`. Configurable.

---

### 1.4 Async Processing Architecture

**Decision:** The request handler returns `200` as soon as the event is successfully enqueued. No downstream processing occurs in the request path.

**Rationale:** Third-party providers (GitHub, Stripe) have documented retry policies. If the receiver takes too long to respond, the provider will retry the delivery. Retries create duplicate events. Duplicate events create downstream problems. The receiver's job is to acknowledge receipt as fast as possible and hand off to a queue. The receiver is not responsible for what happens to the event after it is enqueued.

**What happens in the request path:**
1. Body read and size enforcement
2. Signature validation
3. Minimal event metadata extraction (event type, delivery ID if present)
4. Enqueue operation (see §1.5)
5. Return `200`

**What does not happen in the request path:**
- Database writes
- HTTP calls to downstream services
- Business logic of any kind
- Event deduplication (this belongs in the consumer, not the receiver)

**Enqueue failure:** If the enqueue operation fails, the receiver must not return `200`. It must return `500`. The provider will retry. This is the correct behavior. Returning `200` on a failed enqueue silently drops the event. That is not acceptable.

---

### 1.5 Queue Integration

**Decision:** Events are written to a durable, external message queue. The queue is not in-process. An in-process queue (e.g., an array, an EventEmitter, a Node.js `worker_threads` queue) is not acceptable for production use because it does not survive process restarts.

**Acceptable queue backends:** Redis Streams, AWS SQS, RabbitMQ, Google Pub/Sub. The specific choice is deployment-environment-dependent and must be documented in the deployment configuration, not in this specification. What this specification requires is that the queue backend satisfies the following properties:

- **Durability:** Messages must survive a queue broker restart.
- **At-least-once delivery:** The queue must guarantee that a successfully enqueued message will be delivered to a consumer at least once.
- **Acknowledgment model:** The consumer must explicitly acknowledge processing. Unacknowledged messages must be redelivered after a configurable timeout.

**Message envelope:** Every message written to the queue must include:
- Raw event payload (the validated body, as bytes or base64-encoded string)
- Event type (from provider header, e.g., `X-GitHub-Event`)
- Delivery ID (from provider header, e.g., `X-GitHub-Delivery`, if present)
- Received timestamp (UTC, ISO 8601)
- Source IP of the inbound request
- Provider identifier (string constant identifying which provider sent this event)

**Connection management:** The queue client connection must be established at startup and reused across requests. A new connection must not be opened per request. Connection pooling behavior must be configured explicitly, not left to library defaults.

**Queue client failure at startup:** If the queue connection cannot be established at startup, the service must not start. It must exit with a non-zero code. A service that starts without a queue connection and then silently drops events is worse than a service that fails to start.

---

### 1.6 Idempotency

**Decision:** The receiver does not enforce idempotency. It is not the receiver's responsibility.

**Rationale:** The receiver's contract is: receive, validate, enqueue, acknowledge. Deduplication requires state. State requires storage. Storage in the receiver creates a dependency that can fail and block acknowledgment. The correct place for deduplication is the queue consumer, using the delivery ID as the idempotency key.

**Requirement on the queue consumer (out of scope for this service, but must be documented as a dependency):** The downstream consumer must treat the delivery ID as an idempotency key and handle duplicate deliveries. This specification does not implement that, but it does require that the message envelope (§1.5) always includes the delivery ID so the consumer can do so.

### 1.7 Response Contract

The receiver returns exactly four HTTP status codes. No others.

| Status | Condition |
|--------|-----------|
| `200 OK` | Event received, validated, and successfully enqueued |
| `400 Bad Request` | Malformed request — missing body, body too large, body read error, missing required headers |
| `401 Unauthorized` | Signature validation failed for any reason |
| `500 Internal Server Error` | Enqueue operation failed; provider should retry |

**Response body:** All responses return a minimal JSON body: `{"status": "ok"}` for `200`, `{"error": "<reason_code>"}` for all others. Reason codes are short, uppercase, underscore-separated strings (e.g., `SIGNATURE_INVALID`, `QUEUE_UNAVAILABLE`). They must not include stack traces, internal error messages, or any information that would assist an attacker in understanding the system's internals.

**No other routes return data.** The health check endpoint (§2.3) and metrics endpoint (§2.6) are the only other routes. They do not accept POST requests.

### 1.8 Configuration Management

**Decision:** All configuration is injected via environment variables. No configuration files are read at runtime. No configuration is hardcoded.

**Required environment variables:**

| Variable | Description | Required | Default |
|----------|-------------|----------|---------|
| `PORT` | HTTP listen port | No | `3000` |
| `WEBHOOK_SECRET` | HMAC signing secret (primary) | Yes | None — startup fails |
| `WEBHOOK_SECRET_PREVIOUS` | HMAC signing secret (rotation) | No | None |
| `QUEUE_URL` | Queue connection string | Yes | None — startup fails |
| `QUEUE_NAME` | Queue name / topic / stream key | Yes | None — startup fails |
