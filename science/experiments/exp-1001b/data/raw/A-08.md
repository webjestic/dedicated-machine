## 1. Implementation Decisions

### 1.1 Transport and Server

**Decision:** Use Node.js with a minimal HTTP framework (Fastify preferred over Express). A single POST route is exposed at a fixed path (e.g., `/webhook`). No other application routes exist on this service. TLS termination is handled upstream by a load balancer or ingress controller; the application listens on plain HTTP internally.

**Rationale:** Fastify's request lifecycle is explicit and its schema validation is fast. Keeping TLS at the edge is standard practice for containerized services and avoids certificate management inside the container.

**Failure modes addressed:**
- Oversized payloads are rejected before reaching application logic. A hard body size limit is enforced at the framework level (e.g., 1 MB). Requests exceeding this limit receive `413 Payload Too Large` and are not processed further.
- Requests with unsupported `Content-Type` (anything other than `application/json`) are rejected with `415 Unsupported Media Type` before signature validation runs.

---

### 1.2 Signature Validation

**Decision:** Signature validation is the first application-level operation after the raw body is captured. It runs synchronously, before any parsing or business logic. The implementation uses Node.js's built-in `crypto` module exclusively — no third-party signature library.

**Mechanism:**
- The raw request body is read as a `Buffer` (not a decoded string) to avoid encoding-related signature mismatches.
- The HMAC-SHA256 digest is computed over the raw buffer using the shared secret.
- The computed digest is compared to the value in the provider-specific signature header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe) using `crypto.timingSafeEqual`. Direct string comparison is prohibited.
- If the header is absent, malformed, or the digest does not match, the request is rejected with `401 Unauthorized`. No diagnostic information is included in the response body.

**Failure modes addressed:**
- Timing attacks: `timingSafeEqual` eliminates early-exit comparison vulnerabilities.
- Replay attacks: addressed separately in §1.8.
- Missing header: treated identically to a failed validation — same response, same log treatment — to avoid header-presence oracle.

**What is not validated here:** Payload schema. Schema validation, if applied, happens after signature validation and after enqueuing is confirmed unnecessary — see §1.6.

---

### 1.3 Request Body Handling

**Decision:** The raw body must be captured before any JSON parsing occurs. Framework-level body parsing is disabled on the webhook route. The raw `Buffer` is retained for signature validation. JSON parsing is performed separately, after signature validation passes.

**Rationale:** Frameworks that parse JSON before exposing the raw body make it impossible to reliably reproduce the exact byte sequence used to compute the HMAC. This is a common and silent source of signature validation failures (e.g., whitespace normalization, key reordering).

**Failure modes addressed:**
- JSON parse failure after a valid signature: the request receives `400 Bad Request`. The raw body is logged (truncated to a safe length) for debugging. This is not treated as a security event but is metered separately.
- Empty body with valid signature: treated as `400 Bad Request`. A zero-length payload is not a valid event.

---

### 1.4 Async Decoupling and Queue Contract

**Decision:** After signature validation and JSON parsing succeed, the event is published to a durable message queue before the HTTP response is sent. The HTTP response is not sent until the enqueue operation confirms durability (i.e., the broker acknowledges the write). This is the only synchronous operation in the processing path beyond validation.

**Queue selection:** A managed queue service (AWS SQS, Google Pub/Sub, or RabbitMQ with publisher confirms) is required. An in-process queue (e.g., an EventEmitter or a plain array) is explicitly prohibited. In-process queues do not survive process crashes.

**Envelope written to queue:** See §1.6.

**Failure modes addressed:**
- If the enqueue operation fails (broker unavailable, timeout, network error), the service returns `503 Service Unavailable` to the caller. The third-party provider is expected to retry. This is preferable to returning `200` and silently losing the event.
- If the enqueue succeeds but the process crashes before the `200` is sent, the provider will retry. The downstream consumer must be idempotent (see §1.8). This is an accepted trade-off.
- Downstream processing failures are entirely outside this service's responsibility. This service's contract ends at confirmed enqueue.

---

### 1.5 Response Contract

**Decision:** The service returns exactly two success-path responses and a small set of defined error responses. No response body is returned on success. Error responses return a minimal JSON body with a non-identifying error code.

| Condition | HTTP Status | Notes |
|---|---|---|
| Enqueue confirmed | `200 OK` | Empty body |
| Missing or invalid signature | `401 Unauthorized` | Generic body only |
| Unsupported Content-Type | `415` | |
| Body too large | `413` | |
| JSON parse failure (post-validation) | `400 Bad Request` | |
| Enqueue failure | `503 Service Unavailable` | Signals provider to retry |
| Unexpected internal error | `500 Internal Server Error` | Generic body only |

**Decision:** `200` is used rather than `202 Accepted`. While `202` is semantically more accurate for async processing, many third-party providers treat any non-`2xx` response as a failure and retry. `200` is the safe default unless the provider's documentation explicitly supports `202`.

---

### 1.6 Payload Schema and Envelope

**Decision:** The service does not perform deep schema validation of the event payload. It validates only that the body is parseable JSON and that a minimum set of fields required for routing exist (e.g., event type identifier). Full schema validation is the responsibility of downstream consumers.

**Rationale:** Schema validation in the receiver creates a coupling between the receiver and the provider's payload format. Provider schemas evolve. A receiver that rejects a valid, signed event because a new field was added is worse than a receiver that passes it through.

**Envelope written to queue:**

Each message written to the queue is a structured envelope, not the raw payload. The envelope contains:

- `receivedAt`: ISO 8601 timestamp of receipt (server clock)
- `eventId`: provider-supplied event identifier, extracted from the payload or header (used for idempotency — see §1.8)
- `eventType`: provider-supplied event type string
- `source`: identifier for the webhook source (e.g., `"github"`, `"stripe"`)
- `signature`: the raw signature header value (retained for downstream re-verification if required)
- `payload`: the parsed JSON event object

The envelope is serialized as JSON before being written to the queue.

---

### 1.7 Secret Management

**Decision:** The HMAC shared secret is never hardcoded, never present in environment variable definitions in source control, and never logged. It is injected at runtime via one of:

- A secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager), fetched at process startup and cached in memory.
- A Kubernetes Secret mounted as an environment variable, with the secret value managed outside the application repository.

**Multi-secret support:** The service must support holding more than one active secret simultaneously to enable rotation without downtime (see §2.7). Validation attempts each active secret in sequence. If any secret produces a matching digest, validation passes.

**Failure modes addressed:**
- Secret fetch failure at startup: the process must not start in a degraded state with no secret. If the secret cannot be loaded, the process exits with a non-zero code and does not begin accepting traffic.
- Secret present in logs: all log sanitization must explicitly exclude the secret value. Structured logging fields must never include the raw secret.

---

### 1.8 Idempotency

**Decision:** The receiver itself does not enforce idempotency. It will enqueue a duplicate event if the same event is delivered more than once (which is normal behavior for most providers under retry conditions).

**Responsibility assignment:** Idempotency is enforced by downstream consumers using the `eventId` field in the envelope. The receiver's responsibility is to include a stable, provider-supplied event identifier in every envelope.

**Rationale:** Implementing idempotency in the receiver requires a shared state store (Redis, DynamoDB, etc.), which introduces an additional dependency, additional latency in the request path, and an additional failure mode. The receiver's primary obligation is to be fast, simple, and available. Pushing idempotency downstream is the correct architectural boundary.

**Exception:** If the provider does not supply a stable event identifier, the receiver generates a deterministic identifier by hashing the raw body. This is documented as a degraded idempotency posture.

---

### 1.9 Dependency Choices

| Concern | Choice | Rationale |
|---|---|---|
| HTTP framework | Fastify | Explicit lifecycle, fast, minimal |
| Cryptography | Node.js `crypto` (built-in) | No external dependency for security-critical path |
| Queue client | Provider SDK (AWS SDK, etc.) | Managed, maintained, supports publisher confirms |
| Logging | `pino` | Structured JSON output, low overhead, async transport |
| Process management | Native Node.js signal handling | No PM2 or forever in container; the container orchestrator manages restarts |
| Configuration | Environment variables + secrets manager | Twelve-factor compliant |

**Prohibited dependencies:** Any library that wraps or abstracts HMAC validation in a way that obscures the comparison method. Any in-process queue abstraction. Any logging library that does not produce structured JSON by default.
