## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use Node.js with a minimal HTTP framework. Express or Fastify are both acceptable. Fastify is preferred for its built-in schema validation, lower overhead, and first-class async support.

**Rationale:** The receiver is not a general-purpose API. It has one route, one method, and a strict contract. Framework overhead must not introduce latency in the request path. Fastify's per-route raw body access is also cleaner than Express middleware ordering, which matters critically for HMAC validation.

**Binding:** The service binds to `0.0.0.0` on a configurable port (default `8080`). It does not terminate TLS. TLS termination is the responsibility of the upstream load balancer or ingress controller.

**Accepted methods:** `POST /webhook` only. All other methods and paths return `405` or `404` respectively, with no body. Do not expose information in error responses.

**Timeouts:** The server enforces a request timeout of 5 seconds. Any request that does not complete body transmission within this window is aborted with `408`. This prevents slow-loris style resource exhaustion.

---

### 1.2 Signature Validation

**Decision:** HMAC-SHA256 validation using Node.js's built-in `crypto` module. No third-party library for this step.

**Algorithm:**
1. Read the raw request body as a `Buffer` before any parsing.
2. Extract the signature from the provider-specific header (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).
3. Compute `HMAC-SHA256(secret, rawBody)`.
4. Compare the computed digest against the provided signature using `crypto.timingSafeEqual`.

**Critical constraint:** `timingSafeEqual` is non-negotiable. String equality comparison (`===`) is vulnerable to timing attacks and must never be used for secret comparison.

**Failure behavior:** Any request that fails signature validation returns `401` immediately. No event is queued. No body is parsed further. The failure is logged with enough context to diagnose misconfiguration (provider, header present/absent, timestamp if available) but the raw signature value is never logged.

**Secret rotation:** The validation logic must support accepting two secrets simultaneously during a rotation window — the outgoing secret and the incoming secret. If either validates, the request is accepted. The active secret used for validation is recorded in the structured log entry to aid rotation auditing.

---

### 1.3 Request Body Handling

**Decision:** The raw body must be captured as a `Buffer` before any JSON parsing. This is the most operationally significant ordering constraint in the entire service.

**Rationale:** HMAC is computed over the exact bytes transmitted. If the body is parsed to JSON and re-serialized, key ordering and whitespace may differ, invalidating the signature. The raw buffer must be retained and passed to the HMAC function. JSON parsing happens after validation succeeds, as a separate step.

**Body size limit:** Enforced at 1MB by default, configurable via environment variable. Requests exceeding this limit return `413` before signature validation occurs. This prevents memory exhaustion from oversized payloads.

**Content-Type enforcement:** The service accepts `application/json` only. Requests with other content types are rejected with `415` before body reading begins.

---

### 1.4 Async Decoupling and Queue Integration

**Decision:** Valid, parsed events are enqueued to an external message queue. The enqueue operation must complete before returning `200`. Processing of the event happens entirely outside the HTTP request lifecycle.

**Queue technology:** A managed queue service is required — AWS SQS, Google Cloud Pub/Sub, or RabbitMQ with a durable queue. The choice is environment-dependent. Redis-based queues (BullMQ) are acceptable only if Redis is deployed with persistence and replication. In-memory queues are explicitly prohibited.

**Rationale for completing enqueue before responding:** Returning `200` before the enqueue completes creates a window where the provider believes delivery succeeded but the event is lost. This is worse than returning a non-2xx, which will trigger provider retry logic. The enqueue must be synchronous from the request's perspective, even though downstream processing is not.

**Enqueue failure handling:** If the enqueue operation fails (queue unavailable, timeout, network error), the service returns `503` to the provider. Most webhook providers will retry on non-2xx responses. This is the correct behavior. The failure is logged as a critical error with full context.

**Enqueue timeout:** The enqueue call has its own timeout, set to 2 seconds by default. This is separate from the server request timeout. If the queue does not acknowledge within 2 seconds, the enqueue is treated as failed and `503` is returned.

**Message envelope:** The message placed on the queue includes: the parsed event payload, the event type header value, the delivery ID (provider-assigned, e.g., `X-GitHub-Delivery`), the timestamp of receipt, and the service instance identifier. The raw body is not enqueued — only the parsed payload. If downstream processors need the raw body for their own validation, that requirement must be addressed at the queue design level, not here.

---

### 1.5 Idempotency and Deduplication

**Decision:** The receiver does not perform deduplication. It is the responsibility of downstream consumers to handle duplicate events.

**Rationale:** Deduplication requires shared state (a cache or database). Introducing that dependency into the receiver increases its failure surface and couples it to infrastructure it does not otherwise need. The delivery ID from the provider is included in the message envelope precisely so downstream consumers can use it for idempotency checks.

**Exception:** If the queue technology supports native deduplication (e.g., SQS FIFO with a message deduplication ID), that feature should be enabled using the provider delivery ID as the deduplication key. This is a queue configuration decision, not a receiver code decision.

---

### 1.6 Error Handling Strategy

**Decision:** All unhandled exceptions at the route level are caught by a top-level error handler. The error handler logs the full error with stack trace and returns `500` with an empty body. No error details are returned to the caller.

**Uncaught exceptions and unhandled rejections:** The process registers handlers for `uncaughtException` and `unhandledRejection`. Both log the error as fatal and initiate graceful shutdown. The process does not continue running after an uncaught exception — this is a deliberate decision. A process in an unknown state is more dangerous than a restarting process.

**No retry logic in the receiver:** The receiver does not retry failed enqueue operations. It returns `503` and relies on provider retry behavior. Retry logic in the receiver would increase response latency and complicate failure reasoning.

---

### 1.7 Configuration Management

**Decision:** All configuration is injected via environment variables. No configuration files are read at runtime. No defaults are hardcoded for secrets.

**Required environment variables:**

| Variable | Description | Required |
|---|---|---|
| `WEBHOOK_SECRET` | HMAC signing secret (primary) | Yes |
| `WEBHOOK_SECRET_PREVIOUS` | HMAC signing secret (rotation) | No |
| `QUEUE_URL` | Queue endpoint or connection string | Yes |
| `PORT` | HTTP listen port | No (default: 8080) |
| `BODY_SIZE_LIMIT` | Max body size in bytes | No (default: 1048576) |
| `ENQUEUE_TIMEOUT_MS` | Queue operation timeout | No (default: 2000) |
| `LOG_LEVEL` | Logging verbosity | No (default: info) |
| `SERVICE_INSTANCE_ID` | Injected by orchestrator | No (default: hostname) |

**Startup validation:** On startup, the service validates that all required environment variables are present and non-empty. If validation fails, the process exits with code `1` and logs the missing variables. It does not start listening for connections.

---

### 1.8 Dependency Decisions

**Decision summary:**

| Concern | Decision |
|---|---|
| HTTP framework | Fastify (preferred) or Express |
| Signature validation | Node.js `crypto` (built-in) |
| Queue client | Provider SDK (AWS SDK, etc.) |
| Logging | `pino` — structured JSON, low overhead |
| Process management | Not managed by the application — delegated to the container orchestrator |

**Dependency minimization is a hard requirement.** Each additional dependency is an attack surface, a maintenance burden, and a potential source of supply chain risk. Dependencies that duplicate Node.js built-in capabilities are not permitted.

## 2. Operational Layer

### 2.1 Containerization

**Base image:** Use an official Node.js image pinned to a specific minor version (e.g., `node:20.14-alpine`). Do not use `latest`. Alpine is preferred for reduced attack surface. The image tag must be pinned in the Dockerfile and updated deliberately.

**Non-root user:** The application process must not run as root inside the container. A dedicated non-root user (e.g., `node`, uid 1000) must own the application files and run the process. This is enforced in the Dockerfile with `USER` directive.

**Read-only filesystem:** The container filesystem should be mounted read-only where possible. The application writes no files at runtime. Any writable paths (e.g., `/tmp`) are explicitly mounted as tmpfs volumes.

**Image layers:** Dependencies are installed in a separate layer from application code. This ensures that code changes do not invalidate the dependency cache layer, keeping image builds fast.

**No secrets in the image:** No secrets, credentials, or environment-specific configuration are baked into the image. The image is environment-agnostic. Secrets are injected at runtime by the orchestrator.

**Image scanning:** All images are scanned for known CVEs before promotion to any environment. Scanning is integrated into the CI pipeline. Images with critical-severity CVEs are blocked from deployment.

### 2.2 Health and Readiness Endpoints

**Two distinct endpoints are required.** They serve different purposes and must not be conflated.

**Liveness endpoint:** `GET /healthz`
- Returns `200` with a minimal JSON body (`{"status":"ok"}`) if the process is alive.
- Returns nothing if the process is dead (the orchestrator handles this case).
- Does not check external dependencies. A liveness check that fails due to a queue outage will cause the orchestrator to restart the pod, which does not fix the queue. This is the wrong behavior.
- Liveness failures trigger pod restart. The bar must be high.

-
