# Operational Design Specification: Production Webhook Receiver

**Version:** 1.0
**Status:** Draft
**Scope:** Single containerized Node.js service receiving, validating, and queuing inbound webhook events from a third-party provider (e.g., GitHub, Stripe).

---

## Part I: Implementation Decisions

---

### 1. HTTP Layer

**Decision:** Use a minimal HTTP framework (Fastify preferred over Express) with a single POST route exposed per provider integration.

**Rationale:** Fastify's lower overhead and built-in schema validation reduce latency in the request path, which matters because the receiver's only job is to validate and enqueue — not compute. Express is acceptable but carries unnecessary middleware surface area.

**Route structure:** `/webhooks/:provider` or a fixed `/webhooks` if single-provider. The route must be explicit; no catch-all routing.

**Port binding:** Bind to a non-privileged port (e.g., 3000). TLS termination is handled by the ingress layer, not the application.

---

### 2. Raw Body Preservation

**Decision:** The raw request body must be captured before any parsing occurs and held as a `Buffer` for the duration of signature validation.

**Rationale:** HMAC-SHA256 signature validation operates on the exact bytes transmitted by the sender. Any JSON parsing, whitespace normalization, or body transformation applied before validation will silently corrupt the signature check. This is the most common implementation defect in webhook receivers.

**Mechanism:** Register a raw body parser that captures the buffer and attaches it to the request object. JSON parsing occurs only after signature validation passes, and only if downstream processing requires it.

**Body size limit:** Enforce a hard cap (e.g., 5MB). Requests exceeding the limit are rejected with 413 before signature validation is attempted. This prevents memory exhaustion from oversized payloads.

---

### 3. HMAC-SHA256 Signature Validation

**Decision:** Validation is synchronous, performed using Node.js's built-in `crypto` module. No third-party libraries for this operation.

**Algorithm:**
1. Extract the signature from the request header (provider-specific header name, e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe).
2. Compute `HMAC-SHA256(secret, rawBody)` using the shared secret.
3. Compare the computed digest to the received signature using `crypto.timingSafeEqual`.

**Critical constraint:** Comparison must use `crypto.timingSafeEqual` exclusively. String equality (`===`) is categorically unacceptable due to timing oracle vulnerability.

**Failure behavior:** Any request that fails signature validation returns `401` immediately with a generic error body. No diagnostic information about why validation failed is returned to the caller. The failure is logged internally with full context.

**Secret management:** The HMAC secret is never hardcoded, never committed, and never logged. It is injected at runtime via environment variable and loaded once at startup. See Section II.5 for secret rotation handling.

**Replay protection:** Validate the timestamp embedded in the provider's signature header (where available, e.g., Stripe's `t=` field) against a configurable tolerance window (default: 300 seconds). Requests outside the window are rejected with `400`. This is not optional.

---

### 4. Immediate 200 Response

**Decision:** The HTTP response is sent before any queue interaction. The sequence is: validate signature → enqueue → respond 200. If enqueueing fails, the response is still 200 but the failure is logged and alerted.

**Rationale:** The third-party provider's retry behavior is triggered by non-2xx responses or timeouts. Holding the response open while waiting for queue acknowledgment introduces latency and couples the receiver's availability to the queue's availability. The tradeoff is accepted: we prefer to acknowledge receipt and handle queue failures operationally rather than force provider retries.

**Alternative considered and rejected:** Respond 200 before enqueueing. Rejected because a crash between response and enqueue would silently drop the event with no retry possible.

**Accepted risk:** If the enqueue call fails after the 200 is sent, the event is lost unless the provider retries independently. This is mitigated by queue reliability design (Section 5) and alerting (Section II.3).

---

### 5. Queue Design

**Decision:** Use a durable, persistent message queue. Redis with a reliable queue pattern (e.g., BullMQ) or a managed queue service (e.g., AWS SQS, Google Cloud Pub/Sub) depending on infrastructure constraints.

**Selection criteria:**

| Criterion | BullMQ/Redis | SQS/Pub/Sub |
|---|---|---|
| Operational complexity | Higher (Redis ops burden) | Lower (managed) |
| Delivery guarantee | At-least-once with persistence | At-least-once |
| Latency | Sub-millisecond local | Low single-digit ms |
| Cost | Infrastructure cost | Per-message cost |

**Recommendation:** SQS or equivalent managed queue for production. BullMQ is acceptable if Redis is already in the operational footprint and the team has Redis operational competency.

**Message envelope:** Each queued message includes:
- Raw payload (as string or base64-encoded buffer)
- Provider identifier
- Event type (extracted from header, not body, to avoid parsing before validation)
- Received timestamp (UTC, ISO 8601)
- A unique receiver-assigned event ID (UUID v4)
- The provider's own event ID if present in headers

**Idempotency:** The receiver does not deduplicate. Deduplication is the responsibility of downstream consumers using the provider event ID or the receiver-assigned ID. This boundary is explicit and documented.

---

### 6. Error Handling and Response Contract

**Decision:** The receiver's response surface is intentionally narrow.

| Condition | HTTP Status | Body |
|---|---|---|
| Valid signature, enqueued | 200 | `{"status":"accepted"}` |
| Missing or malformed signature header | 401 | `{"error":"unauthorized"}` |
| Signature mismatch | 401 | `{"error":"unauthorized"}` |
| Timestamp outside tolerance window | 400 | `{"error":"bad_request"}` |
| Body exceeds size limit | 413 | `{"error":"payload_too_large"}` |
| Queue unavailable (enqueue failed) | 200 | `{"status":"accepted"}` (logged, alerted) |
| Unhandled internal error | 500 | `{"error":"internal_error"}` |

No stack traces, internal error messages, or diagnostic detail are returned to the caller under any circumstances.

---

### 7. Concurrency and Process Model

**Decision:** Single Node.js process. No worker threads for the receiver itself. The event loop handles concurrent inbound connections natively.

**Rationale:** The receiver performs no CPU-bound work. HMAC computation on typical webhook payloads (< 100KB) is negligible. Worker threads would add complexity without benefit.

**Connection handling:** Rely on the Node.js HTTP server's default keep-alive behavior. Set explicit `keepAliveTimeout` and `headersTimeout` values above the load balancer's idle timeout to prevent race conditions on connection teardown.

**Graceful shutdown:** On `SIGTERM`, stop accepting new connections, allow in-flight requests to complete within a bounded drain window (e.g., 10 seconds), then exit. Requests that cannot complete within the drain window are terminated. The process exits with code 0 on clean shutdown, non-zero on forced termination.

---

### 8. Configuration

**Decision:** All configuration is environment-variable-driven. No configuration files bundled in the image.

**Required variables:**

- `WEBHOOK_SECRET` — HMAC signing secret
- `QUEUE_URL` or `QUEUE_CONNECTION_STRING` — Queue endpoint
- `PORT` — HTTP listen port
- `TIMESTAMP_TOLERANCE_SECONDS` — Replay window (default: 300)
- `MAX_BODY_SIZE_BYTES` — Body size cap (default: 5242880)
- `LOG_LEVEL` — Logging verbosity (default: `info`)

**Startup validation:** The application validates all required environment variables at startup and exits with a non-zero code and a clear error message if any are missing or malformed. It does not start in a degraded state.

## Part II: Operational Layer

---

### II.1 Containerization

**Base image:** Use an official Node.js LTS image on a minimal base (e.g., `node:20-alpine`). Pin the exact image digest, not just the tag. Tags are mutable; digests are not.

**Image build:**
- Multi-stage build. Build stage installs all dependencies; production stage copies only the compiled output and production `node_modules`.
- No development dependencies in the production image.
- No shell, package manager, or build tooling in the final image layer unless operationally required.
- Run as a non-root user. Create a dedicated `node` user with no login shell. The process must not run as UID 0 under any circumstance.

**Filesystem:** The container filesystem is read-only except for explicitly mounted volumes (none required for this service). This is enforced at the runtime level, not assumed.

**Image scanning:** Every image build is scanned for known CVEs before promotion to any environment. Builds with critical-severity vulnerabilities are blocked from deployment.

---

### II.2 Deployment and Availability

**Minimum replicas:** Two replicas at all times in production. Single-replica deployments are not acceptable for a service in the inbound critical path.

**Replica placement:** Anti-affinity rules ensure replicas do not co-locate on the same underlying node. A single node failure must not take the service offline.

**Rolling updates:** Deployments use a rolling update strategy. The old replica set is not terminated until new replicas pass health checks. Zero-downtime deployments are a hard requirement, not a goal.

**Health checks:**
- **Liveness probe:** HTTP GET to `/healthz`. Returns 200 if the process is alive. Does not check queue connectivity. A liveness failure triggers a container restart.
- **Readiness probe:** HTTP GET to `/readyz`. Returns 200 only if the queue connection is established and the service can accept traffic. Returns 503 otherwise. A readiness failure removes the replica from the load balancer pool without restarting it.

The distinction between liveness and readiness is intentional and must not be collapsed into a single endpoint.

**Startup probe:** Configured with a generous initial delay and failure threshold to allow the process to complete startup validation before liveness checks begin. Prevents premature restarts during cold start.

---

### II.3 Alerting

Alerting is not optional. The following conditions require paged alerts, not dashboard-only visibility.

**Alert: Signature validation failure rate elevated**
- Condition: More than N signature failures per minute sustained over a 5-minute window.
- Severity: High.
- Rationale: A sudden spike in validation failures indicates either a misconfigured secret rotation, a provider-side issue, or an active probing/attack attempt. All three require immediate investigation.

**Alert: Queue enqueue failure**
- Condition: Any single enqueue failure.
- Severity: Critical.
- Rationale: An enqueue failure after a 200 response means an event has been silently dropped. Every occurrence is a data loss event.

**Alert: HTTP 5xx rate elevated**
- Condition: Any sustained 500 responses from the receiver.
- Severity: High.
- Rationale: Unhandled errors in the receiver indicate a defect or an unexpected input condition.

**Alert: Replica count below minimum**
- Condition: Available replicas drop below 2.
- Severity: High.
- Rationale: Single-replica operation eliminates fault tolerance.

**Alert: Pod restart loop**
- Condition: Any replica restarts more than twice within 10 minutes.
- Severity: High.

**Alert: Queue depth exceeding threshold**
- Condition: Queue depth exceeds a defined high-water mark sustained over 10 minutes.
- Severity: Warning.
- Rationale: Indicates downstream consumers are falling behind. The receiver itself is not the problem, but the condition affects end-to-end event processing SLA.

---

### II.4 Logging

**Format:** Structured JSON. Every log line is a JSON object. No unstructured text logs in production.

**Required fields on every log line:**

- `timestamp` — ISO 8601 UTC
- `level` — `debug`, `info`, `warn`, `error`
- `service` — static identifier for this service
- `requestId` — UUID assigned per inbound request, propagated through all log lines for that request
- `provider`
