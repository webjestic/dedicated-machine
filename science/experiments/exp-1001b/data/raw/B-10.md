## 1. Implementation Decisions

### 1.1 HTTP Server and Framework

**Decision:** Use Express or Fastify. Either is acceptable. The choice is not architecturally significant here; what is significant is that the framework must not obscure or transform the raw request body before signature validation occurs.

**Constraints:**
- The server must listen on a single configurable port (default: `3000`), injected via environment variable.
- TLS termination is handled upstream (load balancer or ingress). The service itself speaks plain HTTP internally. Do not implement TLS in the application layer.
- Accept only `POST` requests on the webhook endpoint path. All other methods on that path return `405 Method Not Allowed`.
- The endpoint path must be configurable via environment variable, not hardcoded. This allows path rotation without a code change.
- `Content-Type` must be `application/json`. Requests with any other content type are rejected with `415 Unsupported Media Type` before any further processing.

**Rationale:** Keeping the service's responsibility narrow — receive, validate, enqueue — means the HTTP layer should do as little as possible. Middleware that silently transforms the body (e.g., JSON parsing before raw body capture) is a class of bug that produces intermittent signature failures that are difficult to diagnose.

---

### 1.2 Raw Body Preservation

**Decision:** The raw, unparsed request body buffer must be captured and held in memory for the duration of the request lifecycle. HMAC validation must be computed against this raw buffer. JSON parsing occurs only after successful signature validation.

**Constraints:**
- Enforce a hard maximum body size. **Default: 5MB.** This must be configurable via environment variable. Requests exceeding this limit are rejected with `413 Payload Too Large` before the body is fully read.
- The raw body must never be logged in full. Log only its byte length.
- After validation, the raw buffer must be explicitly dereferenced to allow GC. Do not hold it on the request object beyond the validation step.

**Rationale:** HMAC-SHA256 signatures are computed over the exact byte sequence the sender signed. Any middleware that parses, re-serializes, or normalizes the body before validation will produce a different byte sequence and cause valid requests to fail validation. This is a non-obvious failure mode that has caused production outages in systems where body parsing middleware was added without understanding this constraint.

---

### 1.3 HMAC-SHA256 Signature Validation

**Decision:** Validate the HMAC-SHA256 signature on every inbound request without exception. There is no bypass mode, no debug mode that skips validation, and no IP allowlist that substitutes for cryptographic validation.

**Constraints:**
- The signature is read from the request header specified by the provider (e.g., `X-Hub-Signature-256` for GitHub, `Stripe-Signature` for Stripe). The header name must be configurable.
- Signature comparison must use a **constant-time comparison function** (`crypto.timingSafeEqual`). String equality (`===`) is not acceptable. Timing attacks against naive string comparison are a real threat class.
- If the signature header is absent, return `400 Bad Request` immediately.
- If the signature header is present but malformed (wrong prefix, non-hex characters), return `400 Bad Request`.
- If the signature is present, well-formed, and does not match, return `401 Unauthorized`.
- All validation failures must be logged with: timestamp, remote IP, failure reason (absent/malformed/mismatch), and the first 8 characters of the received signature (never the full value, never the secret).
- Validation must complete within a bounded time. If body read exceeds a configurable timeout (default: 10 seconds), abort the connection and log the event.

**Rationale:** Constant-time comparison is mandatory. The attack surface is small but real: an adversary who can measure response latency can, in principle, determine how many bytes of a naive string comparison matched before short-circuiting. The fix is one function call. There is no reason not to use it.

---

### 1.4 Synchronous Response Contract

**Decision:** The service returns `200 OK` to the caller as soon as the event has been successfully enqueued. The response is sent before any downstream processing occurs. The response body is a minimal JSON acknowledgment.

**Constraints:**
- Response body: `{ "status": "accepted" }`. No event data echoed back. No processing status. No downstream system state.
- The `200` is a receipt acknowledgment, not a processing confirmation. This must be documented in the runbook to avoid confusion when downstream failures occur and the provider shows successful delivery.
- If enqueue fails (queue unavailable, write timeout), return `503 Service Unavailable`. Do not return `200` for an event that was not enqueued. This is the only condition under which a non-2xx response is returned for a valid, authenticated request.
- Response time target: **p99 < 500ms** under normal load. This is a design constraint, not a monitoring threshold. If enqueue latency threatens this, the queue client timeout must be set below this budget.

**Rationale:** Third-party providers have delivery retry logic tied to HTTP response codes. Returning `200` for an event that was silently dropped is the worst possible failure mode: the provider believes delivery succeeded, the event is lost, and no retry occurs. The `503` path ensures the provider retries.

---

### 1.5 Async Queue Architecture

**Decision:** After returning `200`, the service has no further responsibility. It does not process events. It does not call downstream services. It does not maintain state about what happened to an event after enqueue. The queue is the handoff boundary.

**Constraints:**
- Enqueue is the last operation in the request path. Nothing after enqueue is in the request path.
- The queue client must be initialized at startup and held as a module-level singleton. Do not create a new queue connection per request.
- Queue writes must be fire-and-confirm, not fire-and-forget. The service must receive confirmation that the message was accepted by the queue broker before returning `200`. Fire-and-forget is not acceptable.
- Enqueue operations must have a configurable timeout (default: 3 seconds). If the queue does not acknowledge within this window, the enqueue is treated as failed and `503` is returned.

**Rationale:** Fire-and-forget enqueue defeats the purpose of the `503` fallback. If the queue client buffers the message locally and returns immediately, the service has no way to know whether the message was durably received. The `200` guarantee becomes meaningless.

---

### 1.6 Queue Technology Selection

**Decision:** Use a durable, broker-backed message queue. Acceptable options: **Redis Streams**, **RabbitMQ**, **Amazon SQS**, **Google Pub/Sub**, or **Kafka**. In-process queues (arrays, EventEmitter, `p-queue`) are not acceptable for production use.

**Constraints:**
- The queue must provide **at-least-once delivery** guarantees. Exactly-once is preferred where available but not required; idempotency (see 1.8) handles duplicates.
- Messages must be **durably persisted** to disk or replicated before the broker acknowledges the write. In-memory-only queues are not acceptable.
- The queue client library must support connection pooling and reconnection with exponential backoff. A queue client that fails permanently on first disconnect is not acceptable.
- The queue name/topic/stream identifier must be configurable via environment variable.

**Rationale:** In-process queues are lost on process crash or container restart. The webhook receiver is stateless by design; state lives in the queue. If the queue is also in-process, there is no durable state anywhere, and events received during a crash window are silently lost.

---

### 1.7 Event Envelope and Schema

**Decision:** Events are not enqueued as raw provider payloads. Each message enqueued is a structured envelope that wraps the provider payload with receiver-side metadata.

**Required envelope fields:**

| Field | Type | Description |
|---|---|---|
| `envelope_id` | UUID v4 | Generated by the receiver at ingestion time |
| `received_at` | ISO 8601 UTC | Timestamp of receipt, set before enqueue |
| `provider` | string | Configured provider identifier (e.g., `"github"`, `"stripe"`) |
| `event_type` | string | Provider event type header value (e.g., `X-GitHub-Event`) |
| `provider_delivery_id` | string or null | Provider's own delivery ID header, if present |
| `payload` | object | Parsed JSON body of the provider request |
| `signature_validated` | boolean | Always `true`. Events that fail validation are never enqueued. |

**Constraints:**
- `envelope_id` is the idempotency key for downstream consumers.
- `received_at` is set by the receiver, not derived from any provider-supplied timestamp.
- `payload` is the parsed JSON object, not the raw string. Raw body is not stored in the envelope.
- The envelope schema is versioned. Include a `schema_version` field (initial value: `"1"`). This allows downstream consumers to handle schema evolution without breaking changes.

---

### 1.8 Idempotency

**Decision:** The receiver does not enforce idempotency itself. It enqueues every valid, authenticated request it receives, including duplicates. Idempotency is the responsibility of downstream consumers, using `envelope_id` and `provider_delivery_id` as deduplication keys.

**Constraints:**
- The receiver must not maintain a seen-event store. This would require shared state, introduce a new failure mode, and create a scaling constraint.
- The `provider_delivery_id` field must be preserved in the envelope precisely to enable downstream deduplication against provider-native delivery IDs.
