## 1. Implementation Decisions

### 1.1 HTTP Layer

**Decision:** Use a minimal HTTP framework (Fastify preferred over Express) with a single POST route exposed at a versioned path (e.g., `/webhooks/v1/:provider`).

**Rationale:**
- Fastify's schema-based pipeline and lower per-request overhead are appropriate for a high-throughput, latency-sensitive receiver.
- A versioned, provider-namespaced path allows future multi-provider support without breaking changes and makes log filtering unambiguous.
- No other routes are exposed. The service does one thing.

**Constraints:**
- TLS termination is handled upstream by a load balancer or ingress. The service itself listens on plain HTTP internally.
- Maximum request body size is capped at a fixed limit (e.g., 512 KB). Requests exceeding this are rejected with `413` before signature validation occurs. This prevents memory exhaustion from oversized payloads.
- Keep-alive timeouts must be set explicitly and must be longer than the upstream load balancer's idle timeout to avoid race-condition connection resets.

---

### 1.2 Signature Validation

**Decision:** Validate the HMAC-SHA256 signature on every inbound request, unconditionally, before any other processing.

**Mechanics:**
- The raw request body bytes — not a parsed representation — are used as the HMAC input. Body parsing must not occur before signature validation. The framework must be configured to buffer the raw body and make it available alongside any parsed form.
- The computed HMAC digest is compared against the provider-supplied signature header using a **constant-time comparison** function. Node's `crypto.timingSafeEqual` is used. String equality (`===`) is explicitly prohibited here.
- The expected signature format is provider-specific (e.g., GitHub prefixes with `sha256=`). The prefix is stripped before comparison.

**Failure behavior:**
- Any request that fails signature validation returns `401` immediately with no body or with a generic message. No diagnostic detail is returned to the caller.
- Validation failures are logged at WARN level with the provider, route, and a truncated (non-sensitive) request identifier. The secret itself is never logged.
- Validation failures increment a dedicated metric counter (see §2.3).

**Key management:** See §1.8.

---

### 1.3 Request Body Handling

**Decision:** The raw body buffer is captured once and reused for both signature validation and subsequent JSON parsing. The body is not re-read from the stream after the initial capture.

**Rationale:** Frameworks that parse the body before exposing it to middleware lose the original byte sequence, which breaks HMAC validation against the raw payload. The raw buffer is the authoritative input.

**Constraints:**
- JSON parsing occurs only after signature validation succeeds.
- A JSON parse failure after a valid signature returns `400`. This is a legitimate failure: the provider sent a signed but malformed payload. It is logged and counted separately from signature failures.
- No schema validation beyond JSON well-formedness is performed in the receiver. Schema enforcement is the responsibility of downstream consumers.

---

### 1.4 Async Decoupling

**Decision:** The receiver enqueues the validated event and returns `200` before any downstream processing begins. No downstream system is called in the request path.

**Rationale:** The receiver's SLA is defined by its ability to acknowledge the provider quickly and reliably. Downstream processing latency, failures, or slowdowns must not propagate back to the provider. Providers that do not receive a timely `2xx` will retry, creating duplicate delivery pressure.

**What is in the request path:**
1. Body buffering
2. Signature validation
3. JSON parsing
4. Enqueue operation (see §1.5)
5. Return `200`

**What is explicitly not in the request path:**
- Database writes
- External HTTP calls
- Business logic
- Downstream service calls

**Enqueue failure:** If the enqueue operation itself fails (queue unavailable), the service returns `500`. The provider will retry. This is the correct behavior: it is better to signal failure and receive a retry than to silently drop an event.

---

### 1.5 Queue Integration

**Decision:** Valid events are published to a durable message queue. The queue is the exclusive handoff point between the receiver and all downstream processing.

**Queue selection criteria (technology-agnostic):**
- Must support at-least-once delivery guarantees.
- Must support message durability (persistence to disk or equivalent; in-memory queues are not acceptable).
- Must support acknowledgment-based consumption so that downstream processors can signal processing failure without message loss.
- Must be reachable from the container with a connection latency that does not materially affect the receiver's response time under normal conditions.

**Message envelope:** Each enqueued message includes:
- The raw event payload (as received, post-validation)
- Provider identifier
- Event type (extracted from the provider's header, e.g., `X-GitHub-Event`)
- Delivery ID (from the provider's header, used for idempotency; see §1.9)
- Received timestamp (UTC, ISO 8601)
- Receiver service version

**Connection management:**
- The queue client connection is established at startup and maintained as a long-lived connection. Per-request connection establishment is not acceptable.
- Connection loss triggers a reconnection loop with exponential backoff and a cap. During reconnection, inbound requests that attempt to enqueue will fail fast and return `500` rather than blocking the request thread.

---

### 1.6 Response Contract

| Condition | HTTP Status | Notes |
|---|---|---|
| Valid signature, successfully enqueued | `200` | Empty body or minimal JSON acknowledgment |
| Invalid or missing signature | `401` | Generic message only |
| Body exceeds size limit | `413` | Before signature validation |
| Valid signature, malformed JSON | `400` | Logged and counted |
| Queue unavailable | `500` | Provider will retry |
| Method not POST | `405` | |
| Unknown route | `404` | |

**Decision:** `200` is returned, not `202`. Some providers treat `202` as a distinct semantic (accepted but not processed), which is accurate but introduces unnecessary ambiguity. `200` is universally understood as "received and acknowledged." Either is defensible; `200` is chosen for broad compatibility.

---

### 1.7 Error Classification

Errors are classified into three categories, which determines logging level, metric label, and operational response:

| Class | Examples | Logging Level | Action |
|---|---|---|---|
| **Client error** | Invalid signature, malformed JSON, oversized body | WARN | Log, count, return 4xx |
| **Infrastructure error** | Queue unavailable, connection timeout | ERROR | Log, count, alert, return 500 |
| **Internal error** | Unhandled exception, programming fault | ERROR | Log, count, alert, return 500 |

Unhandled exceptions must be caught at the framework level and classified as internal errors. They must never surface stack traces or internal details in HTTP responses.

---

### 1.8 Secrets Management

**Decision:** The HMAC signing secret is injected into the container as an environment variable at runtime. It is never baked into the container image.

**Operational requirements:**
- The secret is sourced from a secrets manager (e.g., AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets with appropriate RBAC) and injected at container startup by the orchestration layer.
- The secret is read once at startup and held in memory. It is not re-read on every request.
- Secret rotation procedure is covered in §2.9.
- The secret is never logged, included in error responses, or exposed via any diagnostic endpoint.

---

### 1.9 Idempotency

**Decision:** The receiver itself does not enforce idempotency. It enqueues every valid, well-formed event it receives, including retries and duplicates.

**Rationale:** Idempotency enforcement requires state (a store of seen delivery IDs). Maintaining that state in the receiver adds latency to the request path, introduces a new failure dependency, and complicates the receiver's operational profile. The receiver's contract is: validate and enqueue. Deduplication is the responsibility of downstream consumers, which have access to persistent storage and can make the correct domain-level decision about what "duplicate" means.

**Requirement placed on downstream:** Downstream consumers must treat delivery ID as an idempotency key and handle duplicate delivery gracefully.

## 2. Operational Layer

### 2.1 Health and Readiness

**Two distinct endpoints are required:**

**Liveness probe** (`GET /healthz`):
- Returns `200` if the process is running and the event loop is not blocked.
- Does not check external dependencies.
- If this endpoint fails, the container is restarted. It must therefore be cheap and always responsive.

**Readiness probe** (`GET /readyz`):
- Returns `200` only if the service is ready to accept and process webhook requests.
- Checks: queue connection is established and healthy.
- Returns `503` if the queue connection is not available. The orchestrator removes the instance from the load balancer pool until readiness is restored.
- This probe prevents traffic from being routed to an instance that cannot enqueue events.

**Failure mode:** If the readiness probe is too aggressive (e.g., checks a slow dependency), it will flap and cause unnecessary pool removal. The readiness check must be fast (sub-100ms) and must not perform network I/O beyond a lightweight ping to the queue client's internal connection state.

---

### 2.2 Observability: Logging

**Decision:** Structured JSON logs only. No unstructured text. Every log line is a JSON object.

**Required fields on every log line:**

| Field | Description |
|---|---|
| `timestamp` | UTC ISO 8601 |
| `level` | `debug`, `info`, `warn`, `error` |
| `service` | Service name constant |
| `version` | Deployed image/build version |
| `traceId` | Distributed trace ID (see §2.4) |
| `requestId` | Per-request unique ID |
| `provider` | Webhook provider identifier |
| `eventType` | Provider event type header value |
| `deliveryId` | Provider delivery ID header value |

**Additional fields by event type:**

- On signature failure: `reason: "invalid_signature"`, HTTP status
- On enqueue success: `queueMessageId`, latency in ms
- On infrastructure error: `errorCode`, `errorMessage` (sanitized)

**What is never logged:**
- The HMAC secret
- Full request headers (may contain tokens)
- Full request body (may contain PII or sensitive data)
- Stack traces in production log output (captured separately in error tracking)

**Log volume:** At high throughput, debug-level logging must be disabled in production or sampled. Info-level logs one line per successful request. Warn/error logs for every failure.
