## 1. Implementation Decisions

### 1.1 HTTP Layer

**Framework:** Express or Fastify. Either is acceptable. The decision criterion is team familiarity, not performance — at webhook-scale throughput, the difference is irrelevant. What matters is that the framework does not silently buffer or transform the raw request body before signature validation runs.

**Critical constraint:** The raw request body must be preserved byte-for-byte for HMAC validation. Any middleware that parses JSON before the signature check will operate on a re-serialized body that may differ from what was signed. The correct approach is to capture the raw buffer at the earliest middleware position and attach it to the request object. JSON parsing, if needed for downstream use, happens after validation is confirmed, against the already-captured buffer.

**Listening interface:** Bind to `0.0.0.0` inside the container. Port is injected via environment variable. Do not hardcode.

**TLS termination:** Handled by the load balancer or ingress layer, not by the Node.js process. The service receives plain HTTP internally.

**HTTP method enforcement:** Accept only `POST` on the webhook endpoint. All other methods return `405 Method Not Allowed`. This is not optional — allowing `GET` or `HEAD` on this route creates unnecessary surface area.

**Request size limit:** Enforce a hard maximum body size. For GitHub and Stripe, 5MB is a safe upper bound; most payloads are under 100KB. Requests exceeding the limit are rejected with `413 Payload Too Large` before signature validation runs. This prevents memory exhaustion from oversized inputs.

---

### 1.2 Signature Validation

**Algorithm:** HMAC-SHA256. This is the algorithm used by GitHub (`X-Hub-Signature-256`) and Stripe (`Stripe-Signature`). Do not support HMAC-SHA1 for new deployments. SHA1 is deprecated by GitHub and must not be treated as a fallback.

**Comparison:** Use a constant-time comparison function for the HMAC digest. Node.js provides `crypto.timingSafeEqual()`. Do not use `===` or any string equality operator. Timing attacks against naive string comparison are a real threat when the attacker controls the payload and can observe response latency.

**Signature header:** The header name is provider-specific and must be configurable. Do not hardcode `X-Hub-Signature-256`. A provider configuration map (provider name → header name, prefix format) is the correct abstraction when supporting multiple providers.

**Stripe-specific:** Stripe's signature scheme includes a timestamp in the `Stripe-Signature` header to prevent replay attacks. The timestamp must be validated against a configurable tolerance window (Stripe recommends 300 seconds). Requests outside the window are rejected with `400`. This logic must be implemented explicitly — it is not automatic.

**Failure behavior:** Any signature validation failure returns `400 Bad Request` with no body or a minimal body. Do not return diagnostic information (e.g., "expected X, got Y"). Log the failure internally with full context.

**Validation is synchronous and blocking in the request path.** This is intentional. There is no value in deferring it. The queue must never receive an unvalidated payload.

### 1.3 Request Lifecycle and Async Handoff

**Contract:** The service returns `200 OK` as soon as the event is durably enqueued. Processing happens downstream. The `200` does not mean the event has been processed — it means the event has been accepted and will be processed.

**Sequence:**
1. Receive raw body buffer.
2. Validate HMAC signature synchronously.
3. On failure: return `400`, log, stop.
4. On success: enqueue the event (see §1.4).
5. On successful enqueue: return `200`.
6. On enqueue failure: return `500`, log, do not return `200`.

**No processing logic belongs in the request path.** No database calls, no downstream HTTP calls, no business logic. The only I/O permitted in the request path is the enqueue operation.

**Response body on 200:** Minimal. A JSON object with a correlation ID is acceptable and useful for debugging. Do not echo back payload content.

---

### 1.4 Queue Integration

**Queue selection:** A durable, persistent queue is required. Redis with persistence (AOF + RDB) is acceptable for moderate durability requirements. For stronger guarantees, use a purpose-built message broker: RabbitMQ, Amazon SQS, or Google Cloud Pub/Sub. The choice must be made based on the organization's existing infrastructure and the acceptable window of data loss on broker failure.

**Redis (if chosen):** Use a list-based queue (LPUSH/BRPOP) or a stream (`XADD`). Streams are preferred because they support consumer groups, acknowledgment, and replay — none of which are available on plain lists. Persistence must be enabled. A Redis instance with no persistence configured is not acceptable for this use case.

**SQS (if chosen):** Standard queues provide at-least-once delivery. FIFO queues provide exactly-once delivery within a 5-minute deduplication window. For webhook events, at-least-once with idempotent downstream consumers is the standard approach. Do not assume FIFO ordering from a Standard queue.

**Enqueue payload:** The enqueued message must include:
- Raw event payload (as received, not re-serialized)
- Provider identifier
- Event type (from provider-specific header, e.g., `X-GitHub-Event`)
- Delivery ID (from provider-specific header, e.g., `X-GitHub-Delivery`)
- Received timestamp (UTC, ISO 8601)
- Correlation ID (generated by this service)

**Idempotency:** The delivery ID from the provider is the idempotency key. It must be included in the enqueued message. Downstream consumers are responsible for deduplication — this service does not deduplicate. Do not attempt to deduplicate at the receiver; the receiver's job is to accept and enqueue, not to reason about event history.

**Queue client connection:** Establish the connection at startup, not per-request. Connection failures at startup are fatal and must prevent the service from accepting traffic. A broken queue connection discovered mid-request must result in a `500` response, not a silent drop.

---

### 1.5 Payload Handling

**Storage of raw payload:** The raw buffer is enqueued, not a parsed representation. This preserves fidelity and allows downstream consumers to re-validate or re-parse without loss.

**Encoding:** Enqueue as base64 if the transport layer requires string values. Document the encoding explicitly in the queue message schema. Do not leave this implicit.

**No logging of payload content at INFO level.** Webhook payloads may contain sensitive data (customer IDs, financial events, PII in GitHub issue bodies). Payload content is logged only at DEBUG level, behind a feature flag that is off by default in production.

---

### 1.6 Error Taxonomy and Response Contract

| Condition | HTTP Status | Logged | Queued |
|---|---|---|---|
| Valid signature, enqueue success | 200 | Yes (INFO) | Yes |
| Invalid or missing signature | 400 | Yes (WARN) | No |
| Body exceeds size limit | 413 | Yes (WARN) | No |
| Replay attack (timestamp out of window) | 400 | Yes (WARN) | No |
| Queue unavailable | 500 | Yes (ERROR) | No |
| Unsupported HTTP method | 405 | Yes (WARN) | No |
| Unhandled exception | 500 | Yes (ERROR) | No |

**Do not return `200` for any failure condition.** Third-party providers use the HTTP status to determine whether to retry. Returning `200` on a queue failure causes silent data loss.

### 1.7 Configuration Management

All configuration is injected via environment variables. No configuration files are baked into the container image.

**Required variables:**
- `PORT` — listening port
- `WEBHOOK_SECRET` — HMAC signing secret (one per provider; use namespaced keys if multi-provider)
- `QUEUE_URL` or equivalent broker connection string
- `MAX_BODY_SIZE_BYTES` — enforced request size limit
- `REPLAY_TOLERANCE_SECONDS` — timestamp window for providers that include it
- `LOG_LEVEL` — runtime log verbosity
- `NODE_ENV` — must be `production` in production; affects error detail in responses

**Secrets must not be logged.** The startup log may confirm that a secret is present (truthy check), but must never log its value. This applies to queue connection strings that may embed credentials.

## 2. Operational Layer

### 2.1 Deployment and Container Design

**Base image:** Use an official Node.js LTS image on a minimal base (Alpine or Debian slim). Do not use `latest`. Pin to a specific digest in the Dockerfile, not just a tag. Tags are mutable; digests are not.

**Run as non-root:** The container process must not run as root. Create a dedicated system user in the Dockerfile and switch to it before the `CMD` instruction. This is a hard requirement, not a recommendation.

**Read-only filesystem:** Mount the container filesystem as read-only where possible. The application should not write to the local filesystem at runtime. Logs go to stdout/stderr. Temporary files, if any, use a dedicated tmpfs mount.

**No shell in production image:** Use the `exec` form of `CMD` (JSON array syntax), not the shell form. This ensures signals are delivered directly to the Node.js process, not to a shell wrapper.

**Single responsibility:** This container runs one process: the webhook receiver. It does not run a sidecar, a cron job, or a worker. If a worker is needed to consume from the queue, it is a separate container.

**Image scanning:** The container image is scanned for known CVEs as part of the CI pipeline. Images with critical or high-severity vulnerabilities are not promoted to production.

---

### 2.2 Process Management and Graceful Shutdown

**Signal handling:** The process must handle `SIGTERM` and `SIGINT`. On receipt of either signal:
1. Stop accepting new connections (close the HTTP server).
2. Allow in-flight requests to complete, up to a configurable drain timeout (default: 10 seconds).
3. Close the queue client connection cleanly.
4. Exit with code `0`.

**Drain timeout:** If in-flight requests do not complete within the drain timeout, exit with code `1`. Do not wait indefinitely. Kubernetes sends `SIGTERM` and then `SIGKILL` after `terminationGracePeriodSeconds`; the drain timeout must be shorter than that grace period.

**Unhandled rejections and uncaught exceptions:** Register handlers for both. Log the error with full stack trace. Exit with code `1`. Do not attempt to recover from an unknown state. The container orchestrator will restart the process.

**Cluster mode:** Do not use Node.js cluster mode in a containerized environment. Horizontal scaling is handled by running multiple container replicas. Cluster mode inside a container adds complexity with no benefit and complicates signal handling.
