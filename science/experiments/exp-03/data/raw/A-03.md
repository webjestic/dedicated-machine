**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution against the database. The auth gates are irrelevant here: insider threat, compromised admin credentials, or any future auth bypass all lead to the same place.

The test suite passing is not a defense. All three new tests use clean, well-formed strings. They test functionality. They do not test safety. 91% coverage on this code is 91% coverage of the happy path.

---

**Required before this can merge:**

Replace the interpolated query with a parameterized statement. The fix is not complicated:

```python
# What is in this PR — do not merge
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Minimum acceptable replacement
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%` wildcards belong in the **parameter value**, not in the query string. The query string must be a static literal. The database driver handles binding. This is not negotiable.

---

**Additional issues to address while you're in here:**

1. **`admin_required` references `current_user` without importing it.** This decorator will throw a `NameError` at runtime. It is not tested by the existing suite because the tests presumably inject auth state through `admin_headers` in a way that bypasses this code path, or the fixture is masking it. Either way, this is a latent bug.

2. **Prefer the ORM for this query.** The codebase already has SQLAlchemy and a `User` model. There is no reason to drop to raw SQL for a simple LIKE search. The ORM binding is correct by default:

   ```python
   users = User.query.filter(
       User.username.ilike(f"%{q}%")
   ).order_by(User.username).limit(50).all()
   ```

   `ilike` is case-insensitive and parameterized automatically. Use it.

3. **Add an injection test.** The test suite should include at least one case with a malicious input string. A passing test on `'; DROP TABLE users; --` does not prove the database is safe, but it documents the expectation and will catch a regression if someone reverts to interpolation:

   ```python
   def test_search_users_rejects_injection_attempt(client, seed_users, admin_headers):
       rv = client.get(
           "/api/users/search?q=' OR '1'='1",
           headers=admin_headers
       )
       assert rv.status_code == 200
       # Should return only users whose username literally contains that string
       # — not every user in the database
       data = rv.get_json()
       assert len(data) == 0
   ```
