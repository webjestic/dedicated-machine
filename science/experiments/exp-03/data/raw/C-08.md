**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, nothing. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. The auth gates are irrelevant as a defense here: admin accounts get phished, get reused, get shared — "only admins can reach it" is not a mitigation for injection, it is a slightly smaller blast radius. The test suite passing is exactly the scenario I have written postmortems about: three tests, all clean inputs, 91% coverage, wide open.

---

**Detailed findings:**

**[BLOCKER] SQL injection — `api/users.py`, the `search_users` function**

```python
# This is the problem. Do not ship this.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)
```

`q` is `request.args.get('q', '').strip()`. Strip does nothing for injection. The value goes straight into the query string. Fix it with a parameterized query. The exact syntax depends on your `db` abstraction, but it needs to look like one of these:

```python
# If db.execute follows DB-API 2.0 (e.g., raw psycopg2/sqlite3 cursor):
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE ?"
    " ORDER BY username LIMIT 50",
    (f"%{q}%",)
)

# If your db layer uses named parameters (e.g., SQLAlchemy core with :param):
from sqlalchemy import text

rows = db.execute(
    text(
        "SELECT id, username, email, role FROM users"
        " WHERE username LIKE :pattern"
        " ORDER BY username LIMIT 50"
    ),
    {"pattern": f"%{q}%"}
)

# Preferred: use the ORM — this is what it's for
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

The ORM path is the cleanest. You already have `User` imported. Use it.

**[BLOCKER] The test suite does not test for injection**

All three new tests use clean, benign inputs. This is not a criticism of the test author — it reflects a gap in how security testing is being thought about here. Before this merges, add at minimum:

```python
def test_search_users_rejects_sql_injection(client, seed_users, admin_headers):
    # A vulnerable endpoint will either error, return unexpected rows,
    # or expose data it shouldn't. A safe one returns [] or a normal result.
    payloads = [
        "' OR '1'='1",
        "'; DROP TABLE users; --",
        "' UNION SELECT id, username, password, role FROM users --",
    ]
    for payload in payloads:
        rv = client.get(
            f'/api/users/search?q={payload}',
            headers=admin_headers
        )
        # Should not 500, should not return all users
        assert rv.status_code == 200
        data = rv.get_json()
        # No result should contain data that only a wildcard match would return
        assert not any(u.get('username') == 'alice' for u in data), (
            f"Injection payload '{payload}' may have bypassed the WHERE clause"
        )
```

These tests will fail against the current implementation. That is the point. They should be green before this merges.

**[MINOR] `admin_required` references `current_user` without importing it**

```python
def admin_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if not current_user.role == 'admin':  # current_user is not imported
```

`current_user` is not in the import block shown. This is presumably `flask_login.current_user`. If it's working in production already, it's being imported somewhere else (maybe a wildcard import or an `__init__.py`). Make the dependency explicit:

```python
from flask_login import login_required, current_user
```

Implicit imports are how you get `NameError` in production when someone cleans up an unrelated file.

---

**[MINOR] The reviewer note "consistent with existing patterns in the codebase" is a flag, not a defense**

If this pattern exists elsewhere in the codebase, those endpoints need to be found and fixed. "We've been doing it this way" is not a reason to continue. I would want to know before this PR closes whether `db.execute` with f-strings appears anywhere else. If it does, that is a separate ticket that needs to be opened and tracked.
