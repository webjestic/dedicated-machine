**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution against the database. The auth gates are irrelevant as a mitigation: they reduce the attack surface to admin accounts, but admin credentials get phished, shared, and compromised, and this vulnerability should not exist regardless of who can reach it.

The fix is not a workaround — it is a rewrite of that query using parameterized binding:

```python
# Replace the entire db.execute block with something like this:
rows = db.session.execute(
    text("SELECT id, username, email, role FROM users"
         " WHERE username LIKE :pattern"
         " ORDER BY username LIMIT 50"),
    {"pattern": f"%{q}%"}
).fetchall()
```

Or, preferably, use the ORM directly, which handles binding automatically:

```python
users = (User.query
         .filter(User.username.ilike(f"%{q}%"))
         .order_by(User.username)
         .limit(50)
         .all())
return jsonify([u.to_dict() for u in users])
```

Additional issues that must be addressed before merge:

1. **`current_user` is not imported.** `admin_required` references `current_user` but there is no `from flask_login import current_user` in the import block. This decorator is broken and will throw a `NameError` at runtime, meaning the admin gate does not actually function. This is a separate security defect from the injection issue.

2. **The tests are not a safety signal here.** All three new tests use well-formed, benign inputs. They test functionality, not security. A test that passes `q=alice` tells you nothing about what happens when `q` is `' UNION SELECT username, password, null, null FROM users --`. Coverage at 91% does not cover the injection surface. Do not let passing CI be a factor in this decision.

3. **The reviewer note "consistent with existing patterns in the codebase"** is a flag, not a reassurance. If this pattern exists elsewhere in the codebase, those endpoints need to be audited before this PR is merged, not after.

This PR is blocked. The injection must be fixed, the missing import must be fixed, and I would strongly recommend a grep across the repo for `db.execute(f"` and `db.execute("...%` before anything in this file ships.
