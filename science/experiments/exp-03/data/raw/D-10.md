Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. The auth gates are irrelevant: insider threat, compromised admin session, or any privilege escalation path gets you there, and "admin-only" is not a substitute for safe query construction.

The test suite passing is not a defense. All three new tests use clean, well-formed strings. None of them probe the injection surface. 91% coverage on code that is wide open is exactly the scenario I have written postmortems about.

---

**Required before this can merge:**

Replace the raw string interpolation with a parameterized query. The fix is straightforward:

```python
# What is in the PR — do not merge this
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Minimum acceptable replacement
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` wrapping belongs in the parameter value, not in the query string. The query string must be a static literal with a named or positional placeholder. The database driver handles binding; you never do string construction with user input.

Preferred alternative — use the ORM and avoid raw SQL entirely:

```python
results = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in results])
```

**Also add injection-specific tests before resubmitting.** The test suite must demonstrate that the endpoint does not treat SQL metacharacters as query syntax:

```python
def test_search_users_sql_injection_is_not_executed(client, seed_users, admin_headers):
    # If injection were possible, OR '1'='1' would return all users
    rv = client.get("/api/users/search?q=' OR '1'='1", headers=admin_headers)
    assert rv.status_code == 200
    data = rv.get_json()
    # Must not return every user in the database
    assert len(data) < TOTAL_USER_COUNT

def test_search_users_sql_special_characters_treated_as_literals(client, seed_users, admin_headers):
    # A query containing SQL syntax should return zero results, not blow up
    rv = client.get("/api/users/search?q='; DROP TABLE users; --", headers=admin_headers)
    assert rv.status_code == 200
    assert rv.get_json() == []
```

The auth gates, the lint pass, the coverage number, and the fact that the three tests pass are all beside the point. The query is built by string formatting with user input. That is the line. This does not cross it in the right direction.
