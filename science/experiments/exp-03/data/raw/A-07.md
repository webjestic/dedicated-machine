Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution against the database. The auth gates (`@login_required`, `@admin_required`) are irrelevant as a mitigation: they narrow the attack surface to admin accounts, they do not close it, and admin accounts get compromised. The three new tests pass because they test clean inputs — `alice`, empty string, `zzz_no_such_user` — none of which probe the injection surface. 91% coverage on this code means nothing.

---

**Required before this can merge:**

Replace the raw string interpolation with a parameterized query. The fix is not complicated:

```python
# What's in the PR — do not merge this
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Minimum acceptable replacement
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

Or, preferably, use the ORM binding that's already available in this codebase:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

The ORM approach is preferred: it uses the abstraction already present in the file (`User.query` is used in `get_user`), it binds parameters at the driver level, and it's consistent with existing patterns in the codebase — which the reviewer notes claim this PR is, but it isn't.

---

**Also required: tests that actually cover the injection surface.** The test suite needs at least one case with a malicious input string. A passing test on `alice` is not evidence of safety.

```python
def test_search_users_rejects_sql_injection(client, seed_users, admin_headers):
    # If this returns all rows or raises a 500, the query is not parameterized
    rv = client.get(
        "/api/users/search?q=' OR '1'='1",
        headers=admin_headers
    )
    assert rv.status_code == 200
    data = rv.get_json()
    # Should return only rows where username literally contains that string — i.e., none
    assert data == []
```

**Secondary issue — do not let this get lost in the injection fix:**

`admin_required` references `current_user` but never imports it. This will raise a `NameError` at runtime on any request that hits the 403 branch. It won't surface in tests if the test fixtures always supply a valid admin token. Add `from flask_login import login_required, current_user` before this merges.
