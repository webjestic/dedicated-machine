**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, nothing. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. The auth gates are irrelevant here: insider threat, compromised admin session, or any privilege escalation upstream makes this the entry point. Do not merge.

---

**The specific problem, line by line:**

```python
# This is the vulnerability. Full stop.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"   # <-- q is raw user input
    f" ORDER BY username LIMIT 50"
)
```

`q` comes from `request.args.get('q', '')`. That is user-controlled. It goes directly into the query string via an f-string. There is no parameterization. There is no escaping. The database receives whatever the user sends.

---

**On the auth gates:**

I have seen this argument before. "It's admin-only, so the attack surface is limited." No. Auth is not a substitute for safe query construction. It is a separate control layer. These are not alternatives. Broken auth, session hijacking, a compromised admin account, a privilege escalation bug elsewhere in the codebase — any of these collapses the assumption that only trusted users reach this endpoint. The query must be safe regardless of who calls it.

---

**On the tests:**

The three new tests pass. They test `q=alice`, `q=`, and `q=zzz_no_such_user`. Clean inputs, expected outputs, green CI. This is exactly the scenario I described in my postmortem. A test suite that only exercises well-formed inputs tells you nothing about injection surface. 91% coverage on code that is wide open.

---

**Required fix:**

Use parameterized queries. The value goes in the parameter binding, not the string.

```python
# Option 1: Parameterized raw SQL (if db.execute supports it — check your driver)
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)

# Option 2: ORM-level — preferred, less surface area for this kind of mistake
users = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
return jsonify([u.to_dict() for u in users])
```

The ORM option is preferable here. SQLAlchemy's `.ilike()` binds the value as a parameter. You are not constructing a string. You are not trusting yourself to escape correctly. The driver handles it.

---

**Secondary issues — not blockers, but fix them while you're here:**

1. `current_user` is used in `admin_required` but never imported. This is a runtime `NameError` on every protected request. The existing code has this bug too, but this PR ships new functionality that depends on it.

2. The reviewer note says this is "consistent with existing patterns in the codebase." If raw string interpolation into `db.execute()` is an existing pattern, that is not a defense — that is a finding. I want to know where else this appears.

3. Add a test that sends a malicious input and asserts the query does not break. It will not catch everything, but it makes the contract explicit:

```python
def test_search_users_rejects_injection_attempt(client, seed_users, admin_headers):
    rv = client.get("/api/users/search?q=' OR '1'='1", headers=admin_headers)
    assert rv.status_code == 200
    # Should return only users whose username literally contains that string
    # not every user in the database
    data = rv.get_json()
    assert len(data) == 0
```

---

**What needs to happen before this merges:**

1. Replace the f-string interpolation with parameterized binding or ORM filtering.
2. Fix the `current_user` import.
3. Add at least one test with adversarial input.
4. If `db.execute` with f-strings is used elsewhere in this codebase, open a separate ticket. That is not this PR's problem to solve, but it needs to be on the board.

The feature is fine. The implementation is not. Resubmit after the query construction is fixed.
