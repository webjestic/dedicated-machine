**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is pulled directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution against the database. The auth gates (`@login_required`, `@admin_required`) are not a mitigating factor: they reduce the attack surface to admin accounts, they do not eliminate it, and admin credentials get phished, leaked, and reused. This does not merge.

---

### The specific problem

```python
# This is the line that cannot ship:
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"   # <-- user input, unbound, in the query string
    f" ORDER BY username LIMIT 50"
)
```

`q` is a string from `request.args`. It goes directly into the query. There is no parameterization. The f-string is doing string concatenation at the SQL level. That is the definition of SQL injection surface.

---

### The fix

Use a parameterized query. The exact syntax depends on your `db` driver, but the pattern is non-negotiable:

```python
# If db.execute() accepts DB-API 2.0 style bindings:
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

Or, given that this codebase already has SQLAlchemy models (`User.query`), use the ORM — it handles binding by construction:

```python
from sqlalchemy import or_

users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

The `ilike` call lets SQLAlchemy construct a parameterized LIKE expression. The `%` wildcards are in your application code, not in the query string. The user value never touches the SQL text.

---

### Why the test suite does not cover this

All three new tests pass clean, well-formed input: `alice`, an empty string, `zzz_no_such_user`. None of them probe the injection surface. A test that passes `' OR '1'='1` or `%'; SELECT * FROM users WHERE '1'='1` would fail against this implementation and pass against the parameterized version. The 91% coverage number is irrelevant here — coverage tells you which lines executed, not whether those lines are safe.

### Secondary issues to address before merge

These are not blockers on their own but should not be deferred:

1. **`current_user` is not imported.** `admin_required` references `current_user` but there is no `from flask_login import current_user` in the imports. This decorator will raise a `NameError` at runtime. The auth gate you are relying on is currently broken.

2. **`User.query.get_or_404` is deprecated** in SQLAlchemy 2.x. Not a security issue, but worth flagging while the file is open.

3. **Consider adding a test that validates the auth gates actually reject non-admin users.** The existing tests only confirm behavior for authenticated admins. A test asserting that a non-admin gets a 403 would confirm the decorator is functioning.

---

### What needs to happen before this can be approved

- Replace the f-string query with a parameterized query or ORM equivalent
- Fix the missing `current_user` import
- Add at least one test with adversarial input to confirm the parameterization holds

The auth scoping and the pagination limit are fine. The structure of the endpoint is fine. The single change that makes this unshippable is the string interpolation into the query. Fix that and this is close to ready.
