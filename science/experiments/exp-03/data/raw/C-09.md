**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, nothing. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` has full query manipulation capability. The auth gates do not mitigate this: admin accounts get compromised, get shared, get phished — "only admins can reach it" is not an acceptable substitute for not handing attackers a query string.

The three new tests are not security tests. They test clean alphanumeric inputs and confirm the happy path works. They would pass on code that is completely wide open, and they did. 91% coverage means nothing here.

---

**Required before this can merge:**

**1. Parameterize the query. This is the blocker.**

Replace the f-string interpolation with a bound parameter. The exact syntax depends on your DB driver, but the pattern is:

```python
# If db.execute() is SQLAlchemy's connection.execute():
from sqlalchemy import text

rows = db.execute(
    text("SELECT id, username, email, role FROM users"
         " WHERE username LIKE :pattern"
         " ORDER BY username LIMIT 50"),
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` wildcard expansion happens in Python, on a plain string, *before* it is handed to the parameter binding layer. The bound value is never interpreted as SQL. That is the correct boundary.

If the codebase has SQLAlchemy ORM available — and it does, `User.query` is right there in the same file — prefer the ORM and avoid raw SQL entirely:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

`ilike` is parameterized at the ORM level. This is the cleanest fix and is already consistent with how `get_user` works.

---

**2. Add a test that would have caught this.**

A passing test suite that cannot detect injection is not a safety signal, it is a false one. Add at minimum:

```python
def test_search_users_rejects_sql_injection(client, seed_users, admin_headers):
    # A naive injection attempt should return normal (empty or filtered) results,
    # not blow up, not return all rows, not expose data it shouldn't.
    payload = "' OR '1'='1"
    rv = client.get(f'/api/users/search?q={payload}', headers=admin_headers)
    assert rv.status_code == 200
    data = rv.get_json()
    # Should not return every user in the database
    assert len(data) < total_user_count  # however you want to assert this
```

This test would fail on the current code. After the fix it should pass. That is what a security test is for.

---

**3. Minor — secondary issues to address in the same pass:**

- `admin_required` references `current_user` but never imports it from `flask_login`. This is a runtime `NameError` on any request that hits the 403 branch. It did not surface in tests because the test suite presumably always passes admin credentials. Fix: `from flask_login import login_required, current_user`.
- Consider whether `email` and `role` should be returned in a search response at all. If this is a lookup-by-username feature, returning role and email for every result is a broader data exposure than the feature requires. Worth a deliberate decision, not an accidental one.
- Input length: `q` is currently unbounded. A `LIKE '%<10000 chars>%'` query is valid SQL and will be executed. Add a max-length guard before the query runs.

---

The auth gating is correct in intent. The structure is fine. The single issue that blocks merge is the injection surface, and it has a clean fix that is consistent with patterns already in this codebase. Fix the query, fix the import, add a test that would catch this class of bug, and resubmit.
