**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string with an f-string — `WHERE username LIKE '%{q}%'` — which means any authenticated admin can execute arbitrary SQL against the database. Auth gating does not mitigate this: it narrows the attack surface to admin accounts, but admin credentials get phished, stolen, and compromised, and the blast radius of this query running unparameterized is the entire database. The three new tests pass because they test clean inputs; they are not evidence of safety and should not be read as such.

---

**The specific problem:**

```python
# This is the line that cannot merge.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"   # <-- user input, unbound, directly in query string
    f" ORDER BY username LIMIT 50"
)
```

An attacker with admin credentials sends:

```
GET /api/users/search?q=%' OR '1'='1
GET /api/users/search?q=%'; DROP TABLE users; --
GET /api/users/search?q=%' UNION SELECT id, password_hash, email, role FROM users --
```

The first returns every row in the table. The third exfiltrates password hashes. The auth decorator does not change this.

---

**Required fix — parameterized query:**

```python
@users_bp.route('/api/users/search')
@login_required
@admin_required
def search_users():
    q = request.args.get('q', '').strip()
    if not q:
        return jsonify([])

    # Bind the value. Do not format it into the string.
    rows = db.execute(
        "SELECT id, username, email, role FROM users"
        " WHERE username LIKE :pattern"
        " ORDER BY username LIMIT 50",
        {"pattern": f"%{q}%"}   # value goes into the binding, not the query string
    )
    return jsonify([
        {'id': r['id'], 'username': r['username'],
         'email': r['email'], 'role': r['role']}
        for r in rows
    ])
```

If the codebase has SQLAlchemy models already (it does — `User` is imported), the preferred fix is to stay at the ORM level entirely and avoid raw SQL here:

```python
from sqlalchemy import or_

results = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
```

SQLAlchemy's `.ilike()` binds the value correctly. This is the approach most consistent with the existing `get_user` endpoint above it.

---

**Secondary issues to address before merge:**

1. **`current_user` is not imported.** The `admin_required` decorator references `current_user` but it is not in the import list. This means the decorator will raise a `NameError` at runtime on any non-admin request. Add `from flask_login import login_required, current_user`.

2. **The test suite needs an injection test.** Three tests covering happy path, empty string, and no-match are not sufficient for an endpoint that accepts free-text user input. Add at minimum:

```python
def test_search_users_rejects_sql_injection(client, seed_users, admin_headers):
    # A naive injection attempt should return normal results or empty,
    # not raise a 500 or return unintended rows.
    rv = client.get(
        "/api/users/search?q=' OR '1'='1",
        headers=admin_headers
    )
    assert rv.status_code == 200
    data = rv.get_json()
    # Should not return every user in the database
    assert len(data) < TOTAL_USER_COUNT
```

3. **Reviewer note "consistent with existing patterns" needs scrutiny.** If raw f-string SQL is an existing pattern in this codebase, that is not a reason to continue it — it is a reason to audit the existing endpoints. Do not let "we've always done it this way" carry weight in a security review.

---

**What needs to happen before this can merge:** Fix the query to use parameter binding or the ORM. Fix the missing `current_user` import. The tests can be merged as-is with the addition of an injection case. The auth structure is fine. The endpoint design is fine. The single line that formats user input into a query string is the blocker.
