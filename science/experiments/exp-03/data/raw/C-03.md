**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is pulled directly from user input and interpolated into a raw SQL string with an f-string — `WHERE username LIKE '%{q}%'` — with zero sanitization or parameterization. An authenticated admin submitting `q=` `%' OR '1'='1` dumps the table; `q='; DROP TABLE users; --` is a keystroke away from worse. The auth gates are irrelevant: admin accounts get phished, get reused passwords, get compromised — "only admins can reach it" is not a containment strategy for injection.

---

The specific line that blocks this merge:

```python
# This is the problem. Do not ship this.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"   # <-- user input directly in query string
    f" ORDER BY username LIMIT 50"
)
```

The fix is not complicated. Use a parameterized query:

```python
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}   # value bound separately, never interpolated
)
```

Or, since this codebase already has SQLAlchemy models, use the ORM and stay out of raw SQL entirely:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

---

**Additional issues that need to be addressed before merge:**

1. **`current_user` is not imported.** `admin_required` references `current_user` but there is no `from flask_login import current_user` in the import block. This decorator is broken for every route that uses it, including this one. The auth gate you're relying on does not actually work.

2. **The tests do not test for injection.** Three tests, all clean alphanumeric inputs. This is exactly the coverage situation I would flag in a postmortem: 91% coverage, all green, wide open. Add at minimum:
   - A test with `q` containing a single quote
   - A test with `q` containing a LIKE wildcard (`%`, `_`)
   - A test with a comment sequence (`--`, `/*`)

   These should pass silently (treated as literal characters) with a correct parameterized implementation.

3. **LIKE wildcards in the search term are unhandled even after fixing injection.** If a user submits `q=%`, a parameterized query will prevent injection but will still match every row. Decide whether that's acceptable behavior for your admin dashboard, and if not, escape `%` and `_` in the input before binding it.

---

**To the reviewer note that says "consistent with existing patterns in the codebase":** If raw f-string query construction is an existing pattern in this codebase, that is not a reason to approve this PR. That is a reason to open a separate ticket and audit every other endpoint. Consistency with a vulnerable pattern is not a defense.

Do not merge until the parameterization is in place and `current_user` is imported.
