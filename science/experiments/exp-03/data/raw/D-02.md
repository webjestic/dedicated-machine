Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution. The auth gates (`@login_required`, `@admin_required`) do not mitigate this; they narrow the attack surface to admin accounts, which are themselves targets for compromise. The test suite passes because it only exercises clean, well-formed inputs — this is exactly the failure mode where green CI is not evidence of safety.

---

**Required before this can merge:**

Replace the raw string interpolation with a parameterized query. The fix is straightforward:

```python
# DO NOT MERGE — current code
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Required replacement
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

If the codebase uses SQLAlchemy ORM consistently elsewhere, the preferred form is:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
```

Either approach binds the value at the driver level. Neither approach allows the input to be interpreted as SQL.

---

**Additional issues to address in the same pass:**

1. **Tests do not cover adversarial input.** The three new tests exercise normal strings. Add at minimum:
   - A query string containing a single quote: `q=alice'`
   - A query string containing a LIKE wildcard: `q=%`
   - A query string containing a comment sequence: `q=alice'--`
   These should return normal results or empty lists, not 500 errors or unexpected data. If the parameterized fix is in place, they will. If they are not in place, they will expose the vulnerability in CI where it belongs.

2. **`current_user` is not imported.** The `admin_required` decorator references `current_user` but it does not appear in the import list. This is a latent `NameError` that existing tests may not be hitting if the decorator path is not fully exercised under failure conditions. Import it: `from flask_login import login_required, current_user`.

3. **Reviewer note "consistent with existing patterns" needs scrutiny.** If raw `db.execute` with f-strings is an existing pattern in this codebase, that is not a justification — it is a finding. Flag it separately.

---

The auth gates are correctly structured and the endpoint design is otherwise fine. The fix is a one-line change. This should be a fast turnaround once the parameterization is in place and the adversarial test cases are added.
