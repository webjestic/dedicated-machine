Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability — the `q` parameter is interpolated directly into a raw query string with no parameterization whatsoever. An authenticated admin can trivially exfiltrate the entire database or, depending on the database driver, execute stacked queries; the auth gate does not mitigate this because the threat model for injection is not limited to external attackers, and admin accounts get compromised. The three new tests pass exclusively because they test clean inputs, which is exactly the failure mode I have written postmortems about — 91% coverage on code that is wide open.

---

**Blocking issue — SQL injection (critical)**

```python
# What is in the PR. Do not merge this.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)
```

`q` is a raw string from `request.args`. There is no escaping, no validation, no parameterization. An admin user — or anyone who has obtained an admin session — sends:

```
GET /api/users/search?q=%' UNION SELECT id,username,password_hash,role FROM users--%20
```

and gets password hashes back in the response JSON, formatted and paginated for them. Depending on the driver and database configuration, it gets worse from there.

The fix is not optional and is not a follow-up ticket:

```python
# Parameterized. The driver handles escaping. This is the only acceptable form.
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

If the codebase uses SQLAlchemy ORM consistently elsewhere, prefer that over raw SQL entirely:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

`.ilike()` binds the value as a parameter. The `%` wildcards are yours to control in application code; the user-supplied string never touches the query structure.

---

**The tests do not cover this and cannot cover this in their current form**

All three tests pass a well-formed alphabetic string. That is not a test of injection safety; it is a test that the feature works when nobody is trying to break it. Add at minimum:

```python
def test_search_users_rejects_sql_injection_attempt(client, seed_users, admin_headers):
    # Should return an empty list or normal results, never blow up or leak data.
    payloads = [
        "' OR '1'='1",
        "' UNION SELECT id,username,password_hash,role FROM users--",
        "%' OR 1=1--",
    ]
    for payload in payloads:
        rv = client.get(
            f'/api/users/search?q={payload}',
            headers=admin_headers
        )
        assert rv.status_code == 200
        data = rv.get_json()
        # Must not return every row in the table.
        assert len(data) < total_user_count
```

These tests will fail against the current implementation. That is the point. They should be added alongside the fix, not after.

---

**Secondary issue — `admin_required` references `current_user` without importing it**

```python
def admin_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if not current_user.role == 'admin':  # NameError at runtime
```

`current_user` is not imported anywhere in the diff. This is either a pre-existing bug masked by the existing test suite or a regression introduced here. Either way it means the auth gate may not be functioning, which makes the injection issue worse. Verify the import and add a test that confirms a non-admin authenticated user receives 403, not 200.

---

**What needs to happen before this can merge**

1. Replace the f-string query with a parameterized query or ORM filter. No exceptions.
2. Confirm `current_user` is imported and the `admin_required` decorator actually enforces the role check.
3. Add a test that sends malformed/injection input and asserts it does not produce a data leak.
4. Add a test that sends a request as an authenticated non-admin and asserts 403.

The CI passing and the coverage number are not relevant here. I have seen this exact pattern in production. It does not end well.
