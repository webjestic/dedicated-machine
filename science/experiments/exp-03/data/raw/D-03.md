Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no binding, no escaping. An authenticated admin sending `q=` `%' OR '1'='1` or `'; DROP TABLE users; --` hands the database to the attacker. The auth gates are irrelevant to this: admin credentials get stolen, get phished, get reused from a breach — "only admins can reach it" is not a mitigation for injection. The three new tests pass because they test clean inputs; they are not evidence of safety and should not be read as such.

---

**Required before this can merge:**

Replace the raw f-string query with a parameterized statement. The fix is straightforward:

```python
# UNSAFE — do not merge
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Required replacement — parameterized binding
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` wildcard expansion happens in Python before the value is bound — the bound parameter is the fully-formed pattern string, and the database driver treats it as a literal value, not executable SQL. Alternatively, use the ORM directly:

```python
users = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
```

SQLAlchemy's `ilike` binds the parameter for you and also gives you case-insensitive matching, which is almost certainly what the admin dashboard actually wants.

---

**Additional issues to address in the same pass:**

1. **`current_user` is not imported.** `admin_required` references `current_user` but it is not in the import list. This decorator is silently broken — depending on the runtime environment it either raises `NameError` or resolves to something unexpected. Add `from flask_login import login_required, current_user`. This needs a test that verifies a non-admin authenticated user receives 403, not 200.

2. **The three new tests do not cover the injection surface at all.** A test with a payload like `q=alice%' OR '1'='1' --` should be added and should confirm the response contains only legitimately matching users. This is not optional — if the parameterized fix regresses, the test suite needs to catch it.

3. **`db.execute` return type is unverified.** Depending on what `db` is in this codebase, `db.execute` may return a `CursorResult` that requires `.mappings()` or `.fetchall()` before iteration. Confirm this works correctly or switch to the ORM call above, which has no ambiguity.

---

The auth scoping, the pagination limit, the empty-query short-circuit — those are all fine. The single issue that blocks merge is the injection. Fix the query, fix the missing import, add a test that would have caught this, and I will approve the next revision.
