**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is pulled directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, no binding. An authenticated admin sending `q=' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query execution against the database. The auth gates are irrelevant to this assessment: admin accounts get phished, get reused, get compromised, and "only admins can reach it" has never been an acceptable substitute for not building an injection sink.

The three new tests pass because they test clean alphanumeric inputs. They do not test `' OR 1=1 --`, they do not test `'; DROP TABLE users; --`, they do not test `%27`, they do not test anything that would expose this vulnerability. 91% coverage on an injectable endpoint is not a safety signal. This is exactly the scenario where coverage metrics actively mislead — the line that matters is covered, the input that matters was never tried.

**Required before merge:**

Replace the raw string interpolation with a parameterized query. The fix is straightforward:

```python
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` wildcard expansion happens in the Python string that builds the *parameter value*, not in the query string itself. The query string contains only the placeholder `:pattern`. The database driver handles binding. That is the correct boundary.

**Also flag for follow-up (not merge blockers, but do not let these get lost):**

- `current_user` is referenced in `admin_required` but never imported. This decorator will raise a `NameError` at runtime on any protected route. The auth gate you are relying on is broken.
- The `LIMIT 50` is in the query but there is no pagination — the reviewer notes say "paginated list" but the response is a flat array with no page/cursor/total. Spec drift, sort it out before this ships to the dashboard team.
- Consider whether returning `email` and `role` for every result in a bulk search response is the right exposure level, even for admins. Scope that to what the dashboard actually needs.

The injection issue is a hard block. Do not merge.
