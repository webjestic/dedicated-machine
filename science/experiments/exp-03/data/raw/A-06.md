Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. The auth gates are irrelevant to this assessment: insider threat, compromised admin credentials, and session hijacking are all real threat vectors, and "only admins can reach it" has never been an acceptable substitute for writing the query correctly.

The three new tests pass because they test clean inputs. They do not test `' OR '1'='1`, `'; SELECT * FROM users; --`, or any other malformed value. Passing tests on clean inputs is not evidence of safety. 91% coverage on this code means 91% coverage on a vulnerability.

---

**Required before merge:**

Replace the raw string interpolation with a parameterized query. The fix is a one-line change:

```python
# REMOVE THIS — it is not acceptable in any form
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# USE THIS — parameter binding, not string formatting
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%` wildcards belong in the bound parameter value, not in the query string. The query string must be a static literal.

Alternatively, use the ORM, which handles binding for you:

```python
users = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
```

**Secondary issues to address in this PR or immediately after:**

- `admin_required` references `current_user` without importing it from `flask_login`. This will raise a `NameError` at runtime on any request that hits the decorator. The auth gate you are relying on is currently broken.
- The reviewer note says this is "consistent with existing patterns in the codebase." If raw f-string query construction is an existing pattern, that needs to be treated as a separate incident, not a precedent.

This PR is blocked on the parameterization fix. The broken `current_user` import is a hard block on its own. Neither issue is a style concern.
