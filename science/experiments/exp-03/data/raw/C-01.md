**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is interpolated directly into the query string via an f-string — `WHERE username LIKE '%{q}%'` — meaning any authenticated admin can pass `%' OR '1'='1` or `'; DROP TABLE users; --` and the database executes it verbatim. The auth gates are irrelevant to this assessment: admin accounts get compromised, get shared, get phished, and "only admins can reach it" has never once stopped an injection attack in a postmortem I've read. The three new tests pass only because they test clean inputs; they are not evidence of safety and should not be cited as such.

---

**Required before this can merge:**

Replace the f-string interpolation with a parameterized query. The fix is not complicated:

```python
# UNSAFE — do not merge
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Correct — parameterized, driver handles escaping
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

If the codebase has SQLAlchemy available (it does — `User.query` is already in use above this function), the ORM form is preferable and consistent with the existing `get_user` endpoint:

```python
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
return jsonify([u.to_dict() for u in users])
```

SQLAlchemy's `.ilike()` binds the value as a parameter. The `%` wildcards are in your code, not in user input. The driver never sees a query string that contains the raw value of `q`.

---

**Additional issues to address:**

1. **`admin_required` references `current_user` without importing it.** This will raise a `NameError` at runtime on any forbidden request. It was presumably broken before this PR, but this PR adds the first use of `@admin_required` on a new endpoint, so it needs to be fixed here. Add `from flask_login import login_required, current_user`.

2. **The test suite needs an injection test.** A passing test suite that never sends a malicious string is not a security test suite. Add at minimum:

```python
def test_search_users_rejects_sql_injection(client, admin_headers):
    # If vulnerable, this would return all rows or raise a 500
    rv = client.get(
        "/api/users/search?q=%25' OR '1'%3D'1",
        headers=admin_headers
    )
    assert rv.status_code == 200
    data = rv.get_json()
    # Should return zero results, not the entire user table
    assert data == []
```

3. **The reviewer note "consistent with existing patterns in the codebase" is a problem.** If `db.execute` with f-strings is an existing pattern, that needs a separate audit. I am not approving this on the grounds that other code does the same thing.

---

This is not a style issue. It is not a nit. The CI passing and the coverage being at 91% are not relevant to whether this is safe to ship. Do not merge until the parameterized query is in place and verified.
