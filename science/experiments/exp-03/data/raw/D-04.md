Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot merge. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. Auth gating is irrelevant here: the attack surface is any admin credential, which may be phished, stolen, or held by a malicious insider. The three new tests pass because they test clean inputs — they are not evidence of safety, they are evidence that the test suite does not cover the threat model.

**Required fix:** Replace the raw string interpolation with a parameterized query. Given that SQLAlchemy is already in the stack, the correct fix is one of the following:

**Option 1 — SQLAlchemy ORM (preferred, already imported):**
```python
users = User.query.filter(
    User.username.ilike(f'%{q}%')
).order_by(User.username).limit(50).all()
return jsonify([u.to_dict() for u in users])
```
SQLAlchemy binds the value; the `%` wildcards are part of the pattern passed to the driver, not concatenated into the query string.

**Option 2 — Parameterized raw SQL if raw SQL is required:**
```python
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```
The value is bound by the driver. The query string is static. This is acceptable.

**What is not acceptable:**
```python
f"WHERE username LIKE '%{q}%'"  # This is the current code. This is a blocker.
```

**Additional notes while I have the diff open:**

- `current_user` is referenced in `admin_required` but is not imported. This is a runtime `NameError` on any non-admin request — the auth gate is broken. Import `current_user` from `flask_login`.
- The test suite should be extended to include at least one adversarial input (e.g., a single quote, a LIKE wildcard character like `%` or `_`, a semicolon). These don't need to simulate a full attack; they need to confirm the parameterization holds and doesn't raise or return unexpected results.
- The reviewer note that this is "consistent with existing patterns in the codebase" is a concern, not a reassurance. If raw string interpolation is an existing pattern, that warrants a separate audit of `api/` before this merges.

This does not merge until the parameterization is in place and the `current_user` import is confirmed. Everything else here — the auth decorators, the pagination limit, the field allowlist in the response — is fine.
