Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is pulled directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker full query control. The auth gates on this endpoint are irrelevant to that risk: admin credentials get compromised, get phished, get shared — and even if they didn't, "only admins can trigger the injection" is not an acceptable threat model for unparameterized SQL.

---

The tests passing is not a defense here. All three new tests use clean, well-formed input strings. They test application logic. They do not test the injection surface at all. 91% coverage on this code means 91% coverage on a vulnerability.

**Required before this can merge:**

Replace the raw `db.execute` f-string with a parameterized query. The fix is straightforward:

```python
# What's in the PR — do not merge
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Minimum acceptable replacement
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` construction needs to happen in Python before it becomes the bound value — that's fine. What is not fine is any part of `q` touching the query string directly. The bound parameter approach is what makes the database treat the entire value as data rather than syntax, regardless of what characters it contains.

**Preferred fix:** If SQLAlchemy ORM is available (and it is — `User` is already an ORM model), use it:

```python
users = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
```

`ilike` with a bound value is parameterized automatically. This is also more consistent with `get_user` above, which uses `User.query` correctly.

**Additional issues to address while you're in here:**

- `current_user` is used in `admin_required` but never imported. This is a latent `NameError` that the tests apparently don't catch because the admin fixture presumably has the right role and the decorator short-circuits — but it will blow up on any non-admin request in production.
- Add at least one test with a malicious input string (e.g., a single quote, a LIKE wildcard, a semicolon) to verify the parameterized query handles it without error. Not because parameterization needs the test to work — it does — but because it documents the expectation and will catch any future regression if someone "simplifies" the query back to interpolation.

This is a blocker. The fix is a two-line change. Get it right and resubmit.
