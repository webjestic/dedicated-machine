**Decision: Request Changes**

**Summary:** This PR contains a textbook SQL injection vulnerability. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. Any authenticated admin (or an attacker who has compromised an admin session) can terminate the LIKE clause and append arbitrary SQL: exfiltrate any table, drop data, or pivot further depending on database permissions. The auth gates are irrelevant here — they narrow the attack surface slightly but do not protect against a malicious or compromised admin, and they do not change the classification of this as an injection vulnerability. The test suite passes because it only exercises clean alphanumeric inputs; 91% coverage tells us nothing about injection surface.

---

**The specific line that blocks this merge:**

```python
# This is not acceptable in any form.
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"   # <-- user input directly in query string
    f" ORDER BY username LIMIT 50"
)
```

A payload of `%' OR '1'='1` returns every row. A payload of `%'; DROP TABLE users; --` does what it says. These are not hypotheticals. This is the first thing an attacker tries.

---

**Required fix — parameterized query:**

The value must be bound as a parameter, never concatenated. The exact syntax depends on your database driver, but the pattern is:

```python
# Option 1: Raw SQL with a bound parameter (SQLAlchemy core / most drivers)
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)

# Option 2: ORM-level query — preferred if User is a mapped model
users = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
```

In both cases the database driver handles binding. The value of `q` is never interpreted as SQL syntax. This is the fix. There is no other acceptable path to merge.

---

**Secondary issues to address before merge:**

1. **`current_user` is not imported.** `admin_required` references `current_user` but it does not appear in the import list. This is a runtime `NameError` on the first 403 path. Add `from flask_login import login_required, current_user`.

2. **Input validation is too permissive.** Stripping whitespace is not enough. Consider a minimum length (1–2 characters), a maximum length, and optionally a character allowlist. This limits both injection surface and the cost of queries triggered by single-character wildcards against a large users table.

3. **The tests do not test for injection.** Tests that only pass clean inputs are not security tests. Add at minimum:

```python
def test_search_users_rejects_sql_injection(client, admin_headers):
    # A vulnerable endpoint returns data or throws a DB error.
    # A safe one returns an empty list or a 400.
    rv = client.get("/api/users/search?q=%' OR '1'='1", headers=admin_headers)
    assert rv.status_code == 200
    # Should not return all users
    assert rv.get_json() == []
```

4. **Reviewer note "consistent with existing patterns" is a concern.** If this pattern exists elsewhere in the codebase, those endpoints need to be found and fixed. I would want confirmation before this PR is revisited that `db.execute` with f-strings is not a recurring pattern. If it is, that is a separate audit item.

---

**To the reviewer who wrote "auth gates are in place":** Auth gates determine who can reach the vulnerability. They do not determine whether the vulnerability exists. An admin account gets phished, gets reused from a breach, gets handed to a contractor — the gates are open and the injection is still there. Defense in depth means the query is safe regardless of who is calling it.

This does not merge until the parameterization is in place and confirmed.
