Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is pulled directly from user input and interpolated into a raw SQL string — `WHERE username LIKE '%{q}%'` — with no parameterization, no escaping, and no binding. An authenticated admin sending `q=alice' OR '1'='1` or `q='; DROP TABLE users; --` hands an attacker direct query control. The auth gates are irrelevant here: the threat model for injection is not "unauthenticated outsider," it is "anyone who can reach this endpoint," and that includes compromised admin accounts, insider threats, and session hijacking.

The three new tests pass because they test clean inputs. They are not evidence of safety. 91% coverage on this code means nothing — the injection surface is not a branch condition that a coverage tool can see.

---

**Required before merge:**

Replace the raw SQL with parameterized execution. The simplest acceptable fix using the existing SQLAlchemy stack:

```python
# Option 1: ORM-level query (preferred — no raw SQL at all)
results = User.query.filter(
    User.username.ilike(f'%{q}%')
).order_by(User.username).limit(50).all()
return jsonify([u.to_dict() for u in results])
```

```python
# Option 2: If raw SQL is genuinely necessary, use bound parameters
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%` wildcards belong in the **parameter value**, not in the query string. The query string must be a static template. That is the entire point.

---

**Secondary issues to address in the same pass:**

- `current_user` is used inside `admin_required` but is never imported. This decorator will raise a `NameError` at runtime on any non-admin request, which means the auth gate is broken in a way the tests do not catch. Import it: `from flask_login import login_required, current_user`.
- Consider adding a minimum length check on `q` (e.g., require at least 2–3 characters) to prevent full-table scans on single-character queries against large user tables.
- The tests should be extended to include at least one adversarial input — a string containing a single quote, a `LIKE` wildcard, or a SQL keyword — and assert that it is handled safely and returns a normal response rather than an error or unexpected data. Tests that only cover clean inputs do not validate injection resistance.

Do not approve a re-submission until the parameterization fix is in place and `current_user` is imported. Everything else is secondary.
