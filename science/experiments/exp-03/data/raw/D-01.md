Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f" WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, and no binding. An authenticated admin submitting `q=` `%' OR '1'='1` or a UNION-based payload gets direct query manipulation; the admin-only gate does not change the attack surface, it just narrows who can exploit it. The three new tests pass exclusively because they use clean inputs — they are not evidence of safety, they are evidence that the test suite does not test for injection.

**Required fix before this can merge:**

Replace the raw f-string query with a parameterized statement:

```python
# REMOVE THIS — it is not acceptable in any form:
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"        # <-- injection point
    f" ORDER BY username LIMIT 50"
)

# REPLACE WITH — parameterized binding:
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}                  # value bound, never concatenated
)

# OR — preferred if the ORM is available everywhere else in this file:
results = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()
```

**The test suite also needs an injection case.** Tests that only exercise clean inputs give false confidence. Add at minimum:

```python
def test_search_users_rejects_sql_injection(client, seed_users, admin_headers):
    # A vulnerable endpoint returns all rows or raises a DB error.
    # A safe one returns an empty list or only literal matches.
    payload = "' OR '1'='1"
    rv = client.get(f'/api/users/search?q={payload}', headers=admin_headers)
    assert rv.status_code == 200
    data = rv.get_json()
    # Should match zero users — not the entire table
    assert all(payload in u['username'] for u in data)
```

**Secondary issue — `current_user` is not imported.** The `admin_required` decorator references `current_user` but it does not appear in the import list (`flask_login` import only brings in `login_required`). This will raise a `NameError` at runtime on any admin-gated route. Add `current_user` to the `flask_login` import. This is not the blocker — the injection is — but it must also be fixed.

The CI passing, the lint being clean, and the 91% coverage number are not relevant to this decision. The vulnerability is visible in a thirty-second read of the diff. Do not merge.
