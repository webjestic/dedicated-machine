Decision: Request Changes

Summary: This PR contains a textbook SQL injection vulnerability and cannot be merged. The `q` parameter is taken directly from user input and interpolated into a raw SQL string — `f"WHERE username LIKE '%{q}%'"` — with no parameterization, no escaping, and no binding. Admin authentication does not mitigate this; credential theft, session hijacking, or a malicious insider means that auth gate is not a control you can rely on to protect the query layer. The three new tests pass because they test clean inputs against clean inputs, which is exactly the failure mode I have written postmortems about.

The fix is not complicated. Replace the raw `db.execute` with a parameterized query:

```python
# Drop this entirely:
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# Use this instead — parameter binding, not string formatting:
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

Or, given that SQLAlchemy is already in this codebase via `User.query`, use the ORM and avoid raw SQL entirely:

```python
results = (
    User.query
    .filter(User.username.ilike(f"%{q}%"))
    .order_by(User.username)
    .limit(50)
    .all()
)
```

Two additional issues worth fixing before merge while you are in here:

1. **`current_user` is not imported.** `admin_required` references `current_user` but there is no `from flask_login import current_user` in the import block. This decorator is broken silently — it will raise a `NameError` at runtime, which depending on error handling may result in a 500 that bypasses the 403 gate entirely. That needs an import and a test that actually verifies a non-admin is rejected.

2. **The new tests do not cover the auth boundary at all.** There is no test asserting that an unauthenticated request returns 401, and no test asserting that a non-admin authenticated request returns 403. Auth gates that are never tested in CI are auth gates that quietly break.

The injection issue is a hard blocker. The missing import is a hard blocker. The auth test gaps are required before this goes anywhere near the admin dashboard.
