**Decision: Request Changes**

**Summary:** This PR has a textbook SQL injection vulnerability on lines 28–31 of `api/users.py`. The `q` parameter is pulled directly from user input and interpolated into the query string with an f-string — no parameterization, no escaping, no binding. An authenticated admin can send `q=' OR '1'='1` and dump the table, or worse depending on the database driver's multi-statement behavior. The auth gates do not mitigate this: they narrow the attack surface to admin accounts, but admin accounts get compromised, get phished, and get abused by insiders. "Authenticated" is not a substitute for "safe query construction."

---

The three new tests pass because they test clean inputs. They are not evidence of safety. A test suite that never sends `'; DROP TABLE users; --` as a query parameter is not testing the thing that matters here. 91% coverage on this code is 91% coverage of a vulnerability.

**Required fix — replace the raw string interpolation with a parameterized query:**

```python
# BEFORE — do not merge
rows = db.execute(
    f"SELECT id, username, email, role FROM users"
    f" WHERE username LIKE '%{q}%'"
    f" ORDER BY username LIMIT 50"
)

# AFTER — parameterized, safe
rows = db.execute(
    "SELECT id, username, email, role FROM users"
    " WHERE username LIKE :pattern"
    " ORDER BY username LIMIT 50",
    {"pattern": f"%{q}%"}
)
```

The `%{q}%` wildcard construction belongs in the **value**, assembled in Python before binding, not inside the query string. The query string must be a static template. The database driver handles quoting and escaping when you bind; it cannot do that when you hand it a pre-interpolated string.

**Preferred fix — use the ORM, which is already imported:**

```python
from sqlalchemy import or_

users = User.query.filter(
    User.username.ilike(f"%{q}%")
).order_by(User.username).limit(50).all()

return jsonify([
    {'id': u.id, 'username': u.username,
     'email': u.email, 'role': u.role}
    for u in users
])
```

SQLAlchemy's `.ilike()` binds the parameter correctly. This is the pattern the rest of the codebase appears to use (`User.query.get_or_404` is already here). Dropping to raw SQL while `User.query` is sitting right there is the wrong direction.

---

**Additional issues to address before merge:**

1. **`current_user` is not imported.** `admin_required` references `current_user` but it is not in the import list. This is a latent `NameError` waiting to happen — either it is already broken in production or there is a wildcard import somewhere that is masking it. Either way, add `from flask_login import login_required, current_user` explicitly.

2. **The auth decorator order matters and should be documented.** `@login_required` before `@admin_required` is correct — you want the login check to run first so `current_user` is populated before `admin_required` tries to read `.role`. This is right, but it is not obvious, and the reviewer notes describe this as "consistent with existing patterns" without confirming the pattern is correct. Worth a comment.

3. **Add injection-specific tests.** Before this merges, the test suite should include at minimum:
   ```python
   def test_search_users_rejects_sql_injection(client, admin_headers):
       rv = client.get(
           "/api/users/search?q=' OR '1'='1",
           headers=admin_headers
       )
       assert rv.status_code == 200
       # Should return only users whose username literally matches,
       # not the entire table
       data = rv.get_json()
       assert len(data) == 0  # or whatever the seeded data produces
   ```
   This test would have caught the vulnerability before review.

---

The reviewer notes say this is "consistent with existing patterns in the codebase." If raw `db.execute` with f-strings is an existing pattern in this codebase, that is a separate finding that needs to go into a follow-up ticket immediately. This PR does not get to inherit safety from a pattern that is itself unsafe.

Do not merge until the parameterization is in place and the missing import is resolved.
