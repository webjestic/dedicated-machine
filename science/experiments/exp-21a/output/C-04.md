Decision: Request Changes

Summary: The core locking pattern and business logic are well-structured, but there are several meaningful flaws that need to be addressed before this can be merged. The most critical is a race condition in the heartbeat's token check, a shared-state concurrency bug that makes this service unsafe under the exact load conditions it was designed for. Several secondary issues around input validation, atomicity guarantees, and test fidelity also need to be resolved.

---

## Review

### 🔴 Critical — Must Fix

**1. Heartbeat has a TOCTOU race condition (shared instance state)**

```python
# _start_heartbeat
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `get` and `expire` are not atomic. Between the check and the `expire`, the lock could expire and be acquired by another process. The correct approach is a Lua script that performs the check and extension atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same issue applies to the lock release in `_acquire_lock`'s `finally` block — use the same Lua pattern there too. Redis documentation explicitly recommends this for safe lock release.

**2. `InventoryService` is not safe for concurrent use — shared heartbeat state**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two calls to `reserve_item` run concurrently on the same `InventoryService` instance (e.g., in a thread pool), one call's `_stop_heartbeat()` will kill the other's heartbeat thread. In a flash sale scenario, this is exactly the failure mode you're trying to prevent. The heartbeat state must be scoped to each lock acquisition, not to the instance. Consider returning a handle object from `_start_heartbeat` or encapsulating lock state in a small dataclass.

### 🟠 High — Should Fix Before Merge

**3. No input validation on `reserve_item` parameters**

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no guard against:

- Empty or whitespace-only `item_id` (which would produce a valid but meaningless lock key like `inventory:lock:   `)
- `quantity <= 0` (a reservation for zero or negative units would succeed and corrupt stock)
- Excessively long `item_id` strings

At minimum:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

**4. `db.update_stock` and `db.create_order` are not atomic**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent access but does nothing to protect against partial writes. These two operations need to execute within a database transaction. The PR gives no indication this is handled at the `db` layer, and there's no exception handling around these calls.

**5. `RuntimeError` is too broad for lock failure signaling**

Catching `RuntimeError` in `reserve_item` will silently swallow unrelated runtime errors (e.g., a bug in `db.get_stock` that raises `RuntimeError`). Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

### 🟡 Medium — Address in This PR or Immediately After

**6. Test suite mocks out the exact behavior under test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

This means the heartbeat logic — the most novel and risky part of this PR — has zero test coverage. `test_lock_not_released_if_expired` is the most important test here, but it also uses this fixture, so it never exercises the real `finally` block behavior. Add at least one integration-style test that uses the real heartbeat path with a short TTL and a mocked `time.sleep`.

**7. Heartbeat `time.sleep` blocks clean shutdown**

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10s before checking flag
```

After `_stop_heartbeat()` sets the flag to `False`, the thread can still be alive for up to `HEARTBEAT_INTERVAL` seconds. For a daemon thread this is acceptable in production, but it makes tests non-deterministic and slow. Use `threading.Event` instead:

```python
self._stop_event = threading.Event()
# in _beat:
self._stop_event.wait(timeout=self.HEARTBEAT_INTERVAL)
if self._stop_event.is_set():
    break
```

### 🔵 Minor / Informational

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first extension fires at 10s, leaving a 20s margin. That's fine, but it should be documented as an invariant: `HEARTBEAT_INTERVAL` must always be less than `LOCK_TTL / 2`.
- The `token` yielded from `_acquire_lock` is unused at the call site in `reserve_item`. Either use it or don't yield it.
- Consider whether `retry_attempts=3` with backoffs of 0.1s, 0.2s, 0.4s is aggressive enough for a flash sale. Under extreme load, callers may want a configurable max wait.
