Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness and safety issues that must be addressed before this can be merged, ranging from a critical race condition in the heartbeat to a shared-state concurrency bug that makes this service unsafe under the exact load it was designed to handle.

---

## Critical Issues

### 1. Heartbeat Has a TOCTOU Race Condition

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could be stolen here
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. You would then extend a lock you no longer own. This must be an atomic operation. Use a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
HEARTBEAT_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once on __init__
self._heartbeat_sha = self.redis.script_load(HEARTBEAT_SCRIPT)

# In the heartbeat thread
self.redis.evalsha(self._heartbeat_sha, 1, lock_key, token, self.LOCK_TTL)
```

The same atomic pattern is already correctly applied at lock release — it must be consistent here too.

### 2. Shared Instance State Makes This Unsafe for Concurrent Use

```python
# On the class instance — not per-call
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two coroutines or threads call `reserve_item` on the same `InventoryService` instance simultaneously, they will clobber each other's `_heartbeat_active` flag. One call stopping its heartbeat will silently kill the other's. This is a correctness bug under exactly the concurrent load this service is built for.

Move heartbeat state into the lock context manager as local variables:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_thread = None
    acquired = False
    ...
    heartbeat_active.set()
    heartbeat_thread = threading.Thread(
        target=self._beat, args=(lock_key, token, heartbeat_active), daemon=True
    )
    heartbeat_thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        ...
```

Using a `threading.Event` also eliminates the double-check on `_heartbeat_active` inside `_beat`, which is a minor logic smell in the current code.

## Moderate Issues

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model gating the inputs. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a path-traversal-style string as `item_id`. At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

Or introduce a proper `ReservationRequest` Pydantic model and validate at the boundary.

### 4. `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect you from this — it only serializes access. These two operations need to be wrapped in a database transaction. The `db` abstraction should expose a transactional interface, or this method needs to handle the rollback explicitly.

### 5. Heartbeat Thread Is Not Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The daemon thread will be abandoned. In a high-throughput scenario this leaks threads until the process exits. After clearing the flag, join with a short timeout:

```python
heartbeat_active.clear()
if heartbeat_thread:
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

## Minor Issues

### 6. Test Coverage Gaps

- No test for the heartbeat logic itself (extend fires, does not extend after stop, does not extend if token mismatches).
- No test for the partial-failure case: `update_stock` succeeds, `create_order` raises.
- `test_reserve_lock_contention` mocks `set` to always return `False` but does not assert the retry count matches `retry_attempts=3`. The retry behavior is untested.
- The `test_lock_not_released_if_expired` test is correct in intent but will pass vacuously once the shared-state bug is fixed — it should be revisited.

### 7. `time.sleep` in Retry Loop Blocks the Calling Thread

For a service expected to handle flash sale load, blocking a thread for up to `0.1 + 0.2 + 0.4 = 0.7s` per failed reservation will exhaust thread pools quickly. Consider whether an async implementation with `asyncio.sleep` is more appropriate for this deployment context, or at minimum document the threading model assumption.

---

## What Is Done Well

- Lock release using atomic Lua script is correct and is the right pattern.
- `nx=True, ex=self.LOCK_TTL` on `SET` is the correct single-command atomic acquisition.
- `uuid4` tokens for ownership verification is appropriate.
- `ReservationResult` as a typed return value is clean.
- Structured logging with key/value pairs is good operational practice.
- Daemon thread prevents process hang on unclean shutdown.

---

The atomic heartbeat fix and the shared instance state bug are blockers. The missing DB transaction is a data integrity issue that must also be resolved before this handles real orders.
