Decision: Request Changes

Summary: The core locking logic and heartbeat concept are directionally sound, but there are several concrete concurrency and correctness flaws that must be resolved before this is safe to merge into a high-throughput flash sale path. The issues range from a race condition in the heartbeat's check-then-extend sequence, to instance-level state that breaks concurrent use, to a DB operation pair that is not atomic and has no rollback path. These are not style concerns — they are failure modes that will manifest under the exact load this service is designed to handle.

---

## Detailed Findings

### 1. Critical — Heartbeat Race Condition (TOCTOU on Lock Ownership)

**File:** `services/inventory.py`, `_start_heartbeat` → `_beat()`

```python
# Current — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process can acquire the lock. You are then extending a lock you no longer own, which is exactly the fencing token violation the PR claims to defend against. This must be a single atomic operation.

**Required fix:** Replace with a Lua script that atomically checks ownership and extends in one round-trip.

```lua
-- extend_if_owner.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In _beat():
result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Heartbeat lost ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same atomic pattern applies to the lock release in `_acquire_lock`'s `finally` block. The current `GET` then `DELETE` sequence has the same race. Use a Lua script there as well.

### 2. Critical — Instance-Level Heartbeat State Breaks Concurrent Use

**File:** `services/inventory.py`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If `InventoryService` is a singleton (the standard pattern for a service class), two concurrent calls to `reserve_item` will clobber each other's heartbeat state. The second caller's `_stop_heartbeat()` will kill the first caller's beat thread mid-operation.

**Required fix:** Move heartbeat state into the lock context itself, scoped to each acquisition. Pass state through a local object or closure rather than `self`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = {"running": False}
    beat_thread = None
    acquired = False

    # ... acquire logic ...

    active["running"] = True
    beat_thread = self._start_heartbeat(lock_key, token, active)

    try:
        yield token
    finally:
        active["running"] = False
        # ... release logic ...
```

### 3. High — Non-Atomic DB Operations With No Rollback

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary and no compensation logic. If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. You now have a silent inventory leak. The distributed lock does not protect you here — it only serializes entry, not outcome.

**Required fix:** Wrap both operations in a DB transaction, or implement explicit rollback on failure.

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB transaction failed item_id=%s order_id=%s", item_id, order_id)
    raise
```

If the `db` abstraction does not support transactions, that contract needs to be defined and enforced at the interface level before this merges.

### 4. Medium — `quantity` Has No Validation

**File:** `services/inventory.py`, `reserve_item`

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

The checklist claims Pydantic input validation, but `quantity` is a raw `int` with no bounds check. A caller passing `quantity=0` or `quantity=-5` will silently succeed or produce nonsensical stock values. `item_id` is also an unvalidated bare string — an empty string or a value containing Redis key separator characters (`:`) will corrupt the lock key namespace.

**Required fix:**

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(ge=1, le=1000)  # set le to a domain-appropriate ceiling
```

Validate at the entry point of `reserve_item` before any lock acquisition.

### 5. Medium — Heartbeat Interval vs. TTL Ratio Is Unsafe Under Jitter

**File:** `services/inventory.py`

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

`time.sleep(10)` in CPython is not guaranteed. Under GIL contention, thread scheduling delays, or a GC pause, the actual sleep can exceed the interval meaningfully. With a 10s interval and 30s TTL, you have two missed beats before expiry — a thin margin. The heartbeat should fire at `TTL / 3` or less, and the beat loop should measure elapsed time rather than relying solely on sleep accuracy.

### 6. Low — Tests Mock Away the Behavior Under Test

**File:** `tests/test_inventory.py`

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is stubbed out in every test except `test_lock_not_released_if_expired`. This means the concurrency behavior of the heartbeat — the highest-risk component — has zero test coverage. At minimum, add an integration-style test that runs the real `_beat` loop against a mock Redis and asserts the Lua extension script is called, and that it stops when ownership is lost.
