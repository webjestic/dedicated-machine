Decision: Request Changes

Summary: The implementation has a critical concurrency flaw that undermines the entire purpose of the distributed lock: `InventoryService` stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance-level attributes rather than per-invocation locals, meaning concurrent calls to `reserve_item` on the same service instance will race on those fields and can prematurely kill each other's heartbeat or leave a thread running after lock release. Beyond that, the lock release in the `finally` block contains a non-atomic check-then-delete race (GET + DELETE instead of a Lua CAS), and the input validation claimed in the checklist is not actually present — `item_id` and `quantity` are never validated before use.

---

### Detailed Findings

#### 1. Critical — Shared Heartbeat State (Instance-Level Race Condition)

```python
# CURRENT — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests call `reserve_item` concurrently on the same `InventoryService` instance, the second call's `_stop_heartbeat()` sets `self._heartbeat_active = False`, which kills the first call's heartbeat thread. The lock the first call is holding can then expire mid-operation — the exact failure mode this code claims to prevent.

**Fix:** Encapsulate heartbeat state in a per-call context object, not on `self`.

```python
@dataclass
class _HeartbeatContext:
    active: bool = True
    thread: Optional[threading.Thread] = None

def _start_heartbeat(self, lock_key: str, token: str) -> _HeartbeatContext:
    ctx = _HeartbeatContext()

    def _beat() -> None:
        while ctx.active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if ctx.active:
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    ctx.thread = threading.Thread(target=_beat, daemon=True)
    ctx.thread.start()
    return ctx
```

#### 2. High — Non-Atomic Lock Release (TOCTOU Race)

```python
# CURRENT — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock. You then delete a lock you no longer own. This is the fencing token violation the PR claims to guard against, but it is not actually guarded here.

**Fix:** Use the standard Lua CAS script, which is atomic at the Redis server:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(self.RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if not released:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

#### 3. High — No Input Validation Despite Checklist Claim

The checklist marks "Input validation via Pydantic" as complete, but neither `item_id` nor `quantity` is validated before use.

- A negative or zero `quantity` passes silently and decrements stock incorrectly.
- An empty or whitespace-only `item_id` produces a nonsensical lock key like `inventory:lock:   `.
- No maximum quantity bound is enforced, which is a business logic and DoS concern on a flash sale endpoint.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=100)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

#### 4. Medium — DB Operations Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented with no corresponding order — inventory is silently lost. The distributed lock protects against concurrent reads, but it does not protect against partial writes.

**Fix:** Wrap both operations in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### 5. Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to observe it. The thread can still be mid-execution — specifically mid-`redis.expire` — after `_stop_heartbeat` returns and the lock has already been released. This creates a window where a thread you believe is stopped extends a lock you have already released or that another process now owns.

**Fix:**

```python
def _stop_heartbeat(self, ctx: _HeartbeatContext) -> None:
    ctx.active = False
    if ctx.thread:
        ctx.thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

#### 6. Low — Test Coverage Gaps

- `test_lock_not_released_if_expired` mocks `_start_heartbeat` and `_stop_heartbeat` on the fixture-level service, so it never exercises the actual lock release path it claims to test. The assertion that `delete` is not called passes vacuously.
- No test covers the partial-write failure scenario (DB exception between `update_stock` and `create_order`).
- No test exercises the heartbeat thread's token verification logic in isolation.
