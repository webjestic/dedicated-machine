Decision: Request Changes

Summary: The implementation has a critical concurrency flaw: `InventoryService` stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance-level attributes, meaning a single shared service instance cannot safely handle concurrent `reserve_item` calls — one call's `_stop_heartbeat()` will clobber another call's active heartbeat mid-operation. Beyond this, the DB operations (`update_stock` + `create_order`) are not atomic and have no rollback path, the lock release in `finally` has a TOCTOU race between `GET` and `DELETE` (requires a Lua script for atomicity), and `quantity` has no validation guard against zero or negative values despite the Pydantic model being in place. The test suite also mocks out the heartbeat entirely, meaning the most dangerous code path — the threading behavior — has zero test coverage.

---

### Detailed Findings

#### 1. Critical — Instance-Level Heartbeat State Breaks Concurrency

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance, the second call's `_stop_heartbeat()` sets `_heartbeat_active = False` globally, killing the first call's heartbeat. The feature this PR exists to provide silently stops working under the exact load conditions it is designed for.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass a threading `Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    heartbeat.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

Note the `stop_event.wait(timeout=...)` pattern also replaces the `time.sleep` + flag check, which is a busy-poll anti-pattern.

#### 2. Critical — TOCTOU Race on Lock Release

```python
# Current — non-atomic check-then-delete
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process. You then delete a lock you no longer own. This is precisely the fencing token violation the PR claims to guard against. The fix is a Lua script, which Redis executes atomically.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init
self._release_lock_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally
self._release_lock_script(keys=[lock_key], args=[token])
```

#### 3. High — Non-Atomic DB Operations With No Rollback

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented with no corresponding order. Inventory is silently lost. This needs to be wrapped in a database transaction, or the service layer needs explicit compensating logic.

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB transaction failed, reservation rolled back item_id=%s", item_id)
    raise
```

#### 4. Medium — No Input Validation on `quantity`

The Pydantic model is on the result, not the input. Nothing prevents `quantity=0` or `quantity=-5`, both of which would pass the stock check and corrupt inventory silently.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="Invalid item_id")
    if quantity < 1:
        return ReservationResult(success=False, message="Quantity must be at least 1")
```

---

#### 5. Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The daemon thread may still be running when the lock is released, creating a window where it extends a lock you just deleted. The `join()` call in the scoped fix above resolves this.

---

#### 6. Low — Test Suite Mocks Out the Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # entire feature mocked
    svc._stop_heartbeat = MagicMock()
    return svc
```

The tests give a false confidence signal. The threading behavior, the heartbeat renewal, and the lock release race are entirely untested. At minimum, add an integration-style test using a real or fakeredis instance that exercises the full lock lifecycle under a simulated slow DB operation.
