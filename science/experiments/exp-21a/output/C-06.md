Decision: Request Changes

Summary: The core locking logic and heartbeat concept are sound, but there are several architectural flaws that make this unsafe for a concurrent production environment. Most critically, the heartbeat and lock state are stored as instance variables on a shared service object, meaning concurrent calls from multiple threads will race on `_heartbeat_active` and `_heartbeat_thread`, corrupting each other's lock state. Additionally, the lock release in the `finally` block contains a TOCTOU race condition, and input validation — despite being listed as complete in the checklist — is entirely absent.

---

## Detailed Findings

### 🔴 Critical — Shared Mutable State Breaks Concurrency (Architecture)

This is the most severe issue. `_heartbeat_active` and `_heartbeat_thread` are instance-level attributes:

```python
# services/inventory.py
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests hit `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern — one service object, many threads), they will stomp on each other's heartbeat state.

**Scenario:**
- Thread A acquires lock for `item_001`, sets `_heartbeat_active = True`, starts heartbeat.
- Thread B acquires lock for `item_002`, sets `_heartbeat_active = True`, starts a *second* heartbeat thread.
- Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`.
- Thread B's heartbeat thread now silently exits, leaving Thread B's lock unrenewed and vulnerable to expiry mid-operation.

The fix is to scope heartbeat state to the lock acquisition context, not the instance. Pass state through a local closure or a dedicated per-call object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()  # "active" while set

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
            else:
                break  # lock was lost, stop beating

    # ... acquire lock, start thread with heartbeat_active ...
    try:
        yield token
    finally:
        heartbeat_active.clear()  # signals the thread to stop
        # ... release lock ...
```

### 🔴 Critical — TOCTOU Race on Lock Release

The `finally` block performs a non-atomic check-then-delete:

```python
# services/inventory.py — _acquire_lock finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be re-acquired by another process. The `DELETE` then destroys a lock this instance does not own. This is a well-known Redis distributed lock pitfall. The canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

### 🟡 High — Input Validation Is Absent Despite Checklist Claiming Otherwise

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw, unvalidated arguments:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation on `item_id` (empty string, path traversal characters, excessive length) or `quantity` (zero, negative, absurdly large values). A call like `reserve_item("", -5)` would silently proceed into the lock and DB layer. A request model should gate entry:

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(default=1, ge=1, le=1000)
```

### 🟡 High — Heartbeat Does Not Handle Lock Loss

In the heartbeat thread, if the token check fails (lock expired or was stolen), the thread continues looping silently:

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                # else: nothing — keeps looping, polling Redis every 10s for no reason
```

When the token no longer matches, the thread should stop. Beyond wasting Redis connections, this masks a serious operational event — the lock was lost while a DB write may have been in flight — that warrants at minimum an error-level log and ideally a mechanism to abort the in-progress operation.

### 🟡 Medium — Tests Mock Away the Exact Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat completely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally complex part of this service, and the tests provide zero coverage of it. The concurrency bug described above would not be caught by any test here. At minimum, add integration-style tests that exercise `_acquire_lock` with real threading to verify the heartbeat starts, extends the TTL, and stops cleanly. The `test_lock_not_released_if_expired` test is also misleading — it validates behavior of the TOCTOU-vulnerable code path, not a correct atomic release.

---

### 🟢 Low — `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

In an async or thread-pool context this is a stall, not just a wait. This is acceptable if the service is always called from a dedicated thread pool with known concurrency limits, but that constraint is not documented. Consider noting the threading model assumption in the class docstring, or flagging this for async refactoring if the broader service is `asyncio`-based.
