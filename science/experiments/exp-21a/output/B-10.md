Decision: Request Changes

Summary: The core locking logic is well-structured and the ownership-check on release is correct, but there are several meaningful flaws that must be addressed before this merges. The most critical is a race condition in the heartbeat that can cause a lock to be extended *after* the owning process has already decided to release it, combined with an instance-level state design that makes the service non-reentrant and unsafe under concurrent use. Additional gaps in input validation, DB atomicity, and test fidelity need to be resolved given the sensitivity of inventory reservation in a flash-sale context.

---

## Detailed Findings

### 🔴 Critical — Non-Reentrant / Shared Mutable State on Instance

```python
# In InventoryService.__init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance. If a single `InventoryService` instance is shared across threads (the normal pattern for a service object), two concurrent calls to `reserve_item` will clobber each other's heartbeat state. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, or vice versa. This is a silent correctness failure under the exact load profile this code is designed to handle.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass a `threading.Event` into `_start_heartbeat` and return it, keeping all state local to the `_acquire_lock` call stack.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        # ... release logic ...
```

### 🔴 Critical — Heartbeat Race Condition on Release

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False

# In finally block:
self._stop_heartbeat()
current = self.redis.get(lock_key)          # <-- window here
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Setting `_heartbeat_active = False` does not synchronously stop the heartbeat thread. The thread may be mid-sleep at `time.sleep(HEARTBEAT_INTERVAL)` and will wake up, pass the `if self._heartbeat_active` check (which it reads *before* the flag is set in some interleavings), and call `redis.expire()` *after* the owning process has already confirmed ownership and is about to delete. More dangerously, the sequence is:

1. Main thread: `_stop_heartbeat()` → flag set to False  
2. Heartbeat thread: wakes, reads flag as False, exits — fine  
3. Main thread: `redis.get()` → token matches  
4. **[Context switch]** Another process acquires the lock with a new token  
5. Main thread: `redis.delete()` → deletes the *new* owner's lock  

The release itself should be an atomic Lua script (standard Redlock practice), not a `GET` + `DELETE`:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In finally:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### 🔴 Critical — No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but neither `item_id` nor `quantity` are validated before use. `item_id` is interpolated directly into the Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A caller passing `item_id = "foo:bar"` or `item_id = "../../../etc"` can manipulate the key namespace. A `quantity <= 0` would silently succeed and *increase* stock (`new_stock = current_stock - (-5)`).

**Fix:** Validate at the method boundary.

```python
from pydantic import BaseModel, Field, constr

class ReservationRequest(BaseModel):
    item_id: constr(pattern=r'^[a-zA-Z0-9_-]{1,64}$')
    quantity: int = Field(default=1, ge=1, le=1000)
```

### 🟡 Major — DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — the lock is a Redis construct and the DB is a separate system. These two operations must be wrapped in a DB-level transaction, or the service must implement a compensating rollback.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 🟡 Major — Heartbeat Can Extend a Lock Indefinitely

There is no maximum lock hold time. A thread that hangs (e.g., DB deadlock, GC pause) will have its lock renewed forever by the heartbeat. This can starve all other waiters permanently.

**Fix:** Add a `MAX_LOCK_HOLD` constant and track wall-clock acquisition time inside the heartbeat. If `time.time() - acquired_at > MAX_LOCK_HOLD`, stop renewing and log a critical alert.

### 🟡 Major — Tests Mock Away the Code Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # <-- heartbeat never runs
    svc._stop_heartbeat = MagicMock()    # <-- stop never runs
    return svc
```

The heartbeat — which is the primary novel risk surface in this PR — is mocked out in every test except `test_lock_not_released_if_expired`. The tests pass, but they do not test the code as it will run in production. The race conditions described above would not be caught by this suite. At minimum, add integration-style tests that allow the heartbeat to run with a short TTL and verify lock extension behavior.

---

### 🟡 Major — `test_lock_not_released_if_expired` Has Wrong Assertion

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts `delete` is never called, but the `service` fixture mocks `_start_heartbeat` and `_stop_heartbeat`. The `redis_client.get` is also called during the heartbeat's ownership check — since the heartbeat is mocked, the only `get` call is in the `finally` block. The test is structurally correct for what it tests, but it is testing a code path that cannot actually occur in isolation (a stolen lock implies the heartbeat failed, which this test cannot simulate). The test gives false confidence.

---

### 🟠 Minor — `quantity` Not Validated as Integer Type at Runtime

Pydantic is imported but `reserve_item` accepts a raw `int` parameter with no enforcement. If this service is called over a network boundary (e.g., FastAPI route), type coercion may not happen automatically depending on the integration layer. Use the `ReservationRequest` model at the entry point.

---

### 🟠 Minor — Logger Name Will Be `services.inventory`

Not a bug, but confirm the logging configuration in the deployment environment captures this namespace. If the root logger is not configured, all `logger.info` / `logger.warning` calls will be silently dropped.
