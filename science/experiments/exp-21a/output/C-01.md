Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes, meaning a single `InventoryService` instance cannot safely handle concurrent calls to `reserve_item` without one call's heartbeat corrupting another's state. Beyond that, the heartbeat's get-then-expire pattern has a TOCTOU race condition, the `quantity` parameter has no validation against negative or zero values, and the DB operations lack transactional atomicity. The tests mock out the heartbeat entirely, so they provide no coverage of the most complex and risky part of the implementation.

---

### Detailed Findings

#### 🔴 Critical — Instance-Level Heartbeat State (Thread Safety)

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If two threads call `reserve_item` concurrently on the same service instance, they share these fields.

```python
# Thread A sets this to True, starts its heartbeat
self._heartbeat_active = True

# Thread B then calls _stop_heartbeat(), killing Thread A's heartbeat
self._heartbeat_active = False
```

This silently kills a legitimate lock's renewal, allowing the lock to expire mid-operation — the exact failure mode this feature is designed to prevent. The fix is to scope heartbeat state to the lock context, not the instance.

```python
# Preferred: pass a mutable container scoped to each lock acquisition
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = {"value": True}  # scoped to this call

    def _beat():
        while heartbeat_active["value"]:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if heartbeat_active["value"]:
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    ...
    try:
        yield token
    finally:
        heartbeat_active["value"] = False
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

Note also that `_stop_heartbeat` sets the flag but never joins the thread. The daemon thread can continue running briefly after the lock is released, potentially extending a lock you no longer own.

#### 🔴 Critical — TOCTOU Race in Heartbeat Renewal

```python
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2: extend
```

Between steps 1 and 2, the lock can expire and be acquired by another process. You would then extend a lock you no longer own. This must be an atomic operation. Use a Lua script:

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at init
self._heartbeat_script = self.redis.register_script(HEARTBEAT_SCRIPT)

# In the heartbeat thread
self._heartbeat_script(keys=[lock_key], args=[token, self.LOCK_TTL])
```

The same atomic pattern should also be applied to lock release, which has the identical get-then-delete race.

#### 🔴 Critical — No Input Validation on `quantity`

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw `int` with no guard. A caller passing `quantity=0` or `quantity=-5` would pass the stock check and corrupt inventory silently.

```python
# Current — no guard
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:

# Required
if not isinstance(quantity, int) or quantity < 1:
    return ReservationResult(success=False, message="Quantity must be a positive integer")

if not item_id or not isinstance(item_id, str) or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
```

#### 🟡 High — DB Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. This is a silent data integrity failure. These must be wrapped in a transaction or compensating rollback:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception as e:
    logger.error("DB transaction failed item_id=%s order_id=%s: %s", item_id, order_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

#### 🟡 High — Tests Mock Out the Core Risk

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # entire heartbeat bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat and lock lifecycle are the most failure-prone parts of this PR. Mocking them entirely means the tests do not cover the actual behavior being shipped. At minimum, add integration-level tests using a real or fakeredis instance that exercise lock acquisition, expiry, and renewal end-to-end.

---

#### 🟡 Medium — Heartbeat Interval Allows Near-Expiry Window

With `LOCK_TTL = 30` and `HEARTBEAT_INTERVAL = 10`, the heartbeat fires at T+10 and T+20. If a DB operation starts at T+21 and takes 12 seconds, the lock expires at T+30 before the next heartbeat at T+30. The interval should be a fraction of the TTL with margin, e.g. `HEARTBEAT_INTERVAL = LOCK_TTL // 3` with a recommendation to document the safety margin explicitly.

---

#### 🟢 Low — Minor Issues

- `_stop_heartbeat` should be documented as non-blocking; callers may assume it waits for the thread to finish.
- The `token` yielded from `_acquire_lock` is not used by `reserve_item`. Either use it or don't yield it — the unused value is misleading.
- Log line in `reserve_item` logs `item_id` which could be user-supplied. Ensure it is sanitized before reaching log aggregators to prevent log injection.
