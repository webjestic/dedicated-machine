Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this problem. However, there are several correctness and safety issues that must be resolved before this can be approved, ranging from a race condition in the heartbeat to a critical concurrency flaw in the service instance model that would cause silent failures under real flash sale load.

---

## Issues

### 🔴 Critical

**1. Instance-level heartbeat state breaks concurrent usage**

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, not scoped to each lock acquisition. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the expected deployment model), one thread's `_stop_heartbeat()` will silently kill the other thread's heartbeat. This is the exact scenario this service is designed to handle, and it will fail silently under load.

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

The heartbeat state must be scoped to each lock acquisition. Use a local `threading.Event` passed into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... retry logic unchanged ...

    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)

def _run_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.LOCK_TTL)
```

Note the use of `stop_event.wait(timeout=...)` instead of `time.sleep` — this allows the heartbeat to exit promptly when the event is set rather than sleeping through the full interval before noticing the stop signal.

**2. Heartbeat check-then-extend is a race condition**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. Use a Lua script to make this atomic:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _run_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> None:
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
        if result == 0:
            logger.warning("Heartbeat lost lock ownership key=%s — stopping", lock_key)
            break
```

The same atomic pattern should be applied to the lock release in the `finally` block. The existing `redis-py` `redis.lock` implementation uses this exact approach and can serve as a reference.

### 🟡 Significant

**3. No input validation on `reserve_item`**

The checklist claims "Input validation via Pydantic" but the method signature accepts arbitrary values with no validation. A caller passing `quantity=0`, `quantity=-5`, or an empty `item_id` will either silently succeed or produce a confusing result. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

**4. `db.update_stock` and `db.create_order` are not transactional**

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record. The Redis lock does not protect against this — it only serializes access. The database operations need to be wrapped in a transaction, or the order must be created before stock is decremented with compensating logic on failure:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception as e:
    logger.error("DB transaction failed order_id=%s: %s", order_id, e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

**5. Broad exception handling masks real errors**

Only `RuntimeError` is caught from `_acquire_lock`. A Redis connection failure, a `db` exception, or any other unexpected error will propagate as an unhandled exception to the caller. Either catch `Exception` broadly with appropriate logging, or define a specific `LockAcquisitionError` and let DB exceptions surface to a higher-level handler that can return a proper HTTP 500.

### 🟠 Minor / Follow-up

**6. Tests mock out the components under test**

`_start_heartbeat` and `_stop_heartbeat` are mocked in the fixture, which means the tests do not exercise the heartbeat or lock release paths at all. `test_lock_not_released_if_expired` is the one test that attempts to verify lock behavior, but because the token is hardcoded as `b"different-token"` in the mock, it does not actually exercise the UUID comparison logic in the real code path. Add at least one integration-style test using `fakeredis` that exercises the full lock lifecycle without mocking internal methods.

**7. `heartbeat_thread.join()` is missing in the current implementation**

The `finally` block calls `_stop_heartbeat()` and immediately proceeds to check and delete the lock. The heartbeat thread may still be mid-execution and could race against the delete. The `join()` call in the fix above addresses this.

**8. HEARTBEAT_INTERVAL vs. LOCK_TTL ratio**

At 10s interval and 30s TTL, there is a 20s window on the first heartbeat. Under a GC pause or a slow DB call that exceeds 30s before the first heartbeat fires, the lock expires. Consider either reducing the interval to `LOCK_TTL / 3` or documenting the assumption that no single DB operation will exceed 30s.

---

## What to Do

1. Refactor heartbeat state to be lock-scoped using `threading.Event` (Critical #1)
2. Make heartbeat extension and lock release atomic via Lua script (Critical #2)
3. Add `ReservationRequest` Pydantic validation (Significant #3)
4. Wrap DB operations in a transaction (Significant #4)
5. Broaden exception handling (Significant #5)
6. Add `fakeredis`-based integration tests that do not mock internal methods (Minor #6)

The core design is sound and worth preserving. Address the critical items at minimum before re-review.
