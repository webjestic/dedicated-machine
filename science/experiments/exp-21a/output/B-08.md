Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for a high-concurrency flash sale environment. The most serious issues are a race condition in the heartbeat thread, a shared-state concurrency bug that makes the service non-reentrant, and a missing atomicity guarantee between the two database writes. These issues must be resolved before this code is merged.

---

## Critical Issues

### 1. Shared Heartbeat State — Non-Reentrant Service (High Severity)

`_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`. If two coroutines or threads call `reserve_item` concurrently on the **same service instance**, they will stomp on each other's heartbeat state. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, and vice versa.

```python
# Both calls share this — this is broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass a `threading.Event` into the heartbeat closure and keep it local to `_acquire_lock`.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition
    acquired = False
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 2. Race Condition in Heartbeat — Check-Then-Act Without Atomicity (High Severity)

The heartbeat does a `GET` followed by an `EXPIRE` as two separate commands. Between those two operations, the lock can expire and be acquired by another process. You would then extend a lock you no longer own.

```python
# NOT atomic — another process can acquire the lock between GET and EXPIRE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

**Fix:** Use a Lua script to make the check-and-extend atomic.

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.warning("Heartbeat: lock already lost key=%s", lock_key)
    stop_event.set()
```

The same atomic pattern should also be applied to the lock release in the `finally` block, which has the identical GET-then-DELETE race.

---

### 3. Non-Atomic Dual Database Write — Partial Failure Window (High Severity)

`update_stock` and `create_order` are two separate database calls with no transaction wrapping them. If the process crashes, is killed, or the DB times out between these two calls, stock is decremented but no order exists — inventory is silently lost.

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # crash here = lost inventory
```

**Fix:** These two operations must be wrapped in a single database transaction. The `db` abstraction layer should expose a transactional interface.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 4. No Input Validation on `reserve_item` Parameters

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation. `quantity=0` or `quantity=-5` would silently pass the stock check and create a nonsense order. `item_id` is not validated for format or length, which is a concern in a government contracting context where item IDs likely have a defined schema.

```python
# quantity=0 passes current_stock < quantity check silently
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

**Fix:** Define a `ReservationRequest` Pydantic model with constraints and validate at the entry point.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

---

### 5. Daemon Thread Leak on Unhandled Exceptions

If `yield token` raises an exception that is not a `RuntimeError`, the `finally` block runs `_stop_heartbeat()` — but with the current shared-state design, if the heartbeat thread is mid-sleep when the flag is set, it will continue running until the next sleep cycle completes. With a 10-second interval, that is a 10-second window of a running thread with a reference to a potentially invalid lock context. After fixing issue #1 with `threading.Event`, call `stop_event.set()` and `thread.join()` in the `finally` block to guarantee the thread is dead before exiting.

## Test Coverage Gaps

### 6. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most complex and failure-prone part of this PR. Mocking it entirely means the race condition in issue #2 and the thread leak in issue #5 are completely untested. At minimum there should be an integration-level test that runs the real heartbeat against a local Redis instance (e.g., `fakeredis`) and verifies TTL is extended and the thread terminates cleanly on context exit.

### 7. Lock Release Atomicity Test Is Incorrect

`test_lock_not_released_if_expired` patches `redis.get` to return a different token and asserts `delete` is not called. This is correct behavior to test, but it does not test the race condition — it only tests the happy-path of the check. The test needs to simulate the lock expiring *between* the GET and DELETE to validate that the atomic Lua script (once added) actually prevents the double-delete.
