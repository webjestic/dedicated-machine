Decision: Request Changes

Summary: The core locking logic and heartbeat design are directionally sound, but there are several architectural and correctness issues that must be resolved before this merges. Most critically, the `InventoryService` instance is not safe for concurrent use — shared mutable state on the instance (`_heartbeat_active`, `_heartbeat_thread`) will race destructively under simultaneous requests, which is exactly the high-contention scenario this service is designed for. Additionally, the DB operations lack transactional atomicity, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory state with no recovery path.

---

## Detailed Findings

### 🔴 Critical — Instance-Level Heartbeat State Is Not Thread-Safe

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped locals. If two requests hit the same `InventoryService` instance concurrently (standard in any WSGI/ASGI worker), the following race is trivially reproducible:

- Request A acquires lock, starts heartbeat → sets `_heartbeat_active = True`
- Request B acquires a different item's lock, calls `_stop_heartbeat()` → sets `_heartbeat_active = False`
- Request A's heartbeat thread silently dies mid-operation
- Lock A expires; a third process steals it; Request A writes on a stolen lock

This defeats the entire purpose of the heartbeat. The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    active.set()

    def _beat():
        while active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    thread = threading.Thread(target=_beat, daemon=True)
    # acquire lock, start thread, yield, finally: active.clear(), join thread
```

Using `threading.Event` also eliminates the `time.sleep` polling pattern, which is imprecise and non-interruptible.

### 🔴 Critical — No Transactional Atomicity Between DB Writes

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If the process crashes, is killed, or the DB throws after `update_stock` but before `create_order`, you have decremented stock with no corresponding order record. This is a silent data corruption path — inventory is consumed but the sale is lost, with no audit trail and no compensating mechanism.

At minimum this needs a transactional boundary:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer doesn't support transactions, that is itself a blocker that needs to be surfaced now.

---

### 🔴 Critical — Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` parameters with zero validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against:
- `quantity <= 0` (reserve zero or negative stock — passes the `< quantity` check silently when stock is 0)
- `item_id` being empty string, whitespace, or a path-traversal-style value that could affect the lock key namespace
- `quantity` being an absurdly large integer

Pydantic is imported but only used for the response model. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

### 🟡 Major — Heartbeat Has a Check-Then-Act Race on Expiry Extension

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You are then extending a lock you no longer own. This must be an atomic operation. Use a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the release logic in the `finally` block — that `GET` + conditional `DELETE` should also be a single atomic Lua script (this is the standard Redlock release pattern).

### 🟡 Major — Tests Mock Away the Exact Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most operationally risky component in this PR and it is mocked out in every test except `test_lock_not_released_if_expired`. There are no tests for:
- Heartbeat actually extending TTL
- Heartbeat stopping cleanly after lock release
- Concurrent calls on the same instance (which would expose the race described above)
- DB failure mid-transaction rolling back correctly

`test_lock_not_released_if_expired` also has a logical gap — it patches `redis.get` to return a different token, but because `_stop_heartbeat` is mocked, the heartbeat thread never ran anyway, so this test is not validating the scenario it claims to cover.

---

### 🟡 Major — `LOCK_TTL = 30` With No Justification or Upper Bound on DB Operations

A 30-second lock TTL is extremely long for an inventory reservation in a flash sale context. Under high concurrency, a single slow or hung DB connection holds the lock for up to 30 seconds, serializing all other requests for that item. There is no timeout enforced on the DB calls themselves, meaning a hung DB operation can hold the lock indefinitely (the heartbeat will keep renewing it forever).

DB calls must have explicit timeouts, and those timeouts should be shorter than `LOCK_TTL`. The heartbeat should have a maximum renewal count or wall-clock deadline.

---

### 🟠 Minor — `quantity` Defaults to 1 Silently

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

For an inventory reservation endpoint, a silent default quantity is a footgun. Callers that forget to pass quantity will silently reserve 1 unit. This should be a required parameter with no default, forcing the caller to be explicit.

---

### 🟠 Minor — Logger Name Will Be `services.inventory` in Production But Tests Use Module Path

Minor, but structured logging in a government contracting context should include a correlation/request ID so reservations can be traced end-to-end across services. The current log fields (`item_id`, `order_id`) are good, but there is no trace context threading through.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | 🔴 Critical | Heartbeat state is instance-scoped, not call-scoped — concurrent requests will race |
| 2 | 🔴 Critical | No DB transaction wrapping the two write operations |
| 3 | 🔴 Critical | Input validation is not implemented despite checklist claim |
| 4 | 🟡 Major | Heartbeat TTL extension and lock release are not atomic |
| 5 | 🟡 Major | Tests mock out the heartbeat entirely, leaving core behavior untested |
| 6 | 🟡 Major | No timeout on DB operations; heartbeat will renew indefinitely on hung connections |
| 7 | 🟠 Minor | `quantity` should be required, not defaulted |
| 8 | 🟠 Minor | No request correlation ID in structured logs |

Items 1–3 are blockers. Items 4–6 should be resolved in this PR given the stated design goals. Items 7–8 can be tracked as follow-on if there is a strong reason to defer, but should be documented.
