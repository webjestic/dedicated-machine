Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — distributed lock ownership verification, heartbeat renewal, and exponential backoff are all present and correctly motivated. However, there are several correctness, safety, and concurrency issues that must be addressed before this handles real flash sale load in a government-adjacent production environment. The most critical is a race condition in the heartbeat and a non-atomic lock release that can cause inventory corruption or lock theft under concurrent access.

---

## Critical Issues

### 1. Non-Atomic Lock Release (Race Condition)

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
# CURRENT — two separate operations, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock after yours expires. You then delete *their* lock. This is the canonical mistake with Redis distributed locks.

**Fix:** Use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 2. Heartbeat Has the Same Non-Atomic Race

**File:** `services/inventory.py`, `_beat` inner function

```python
# CURRENT — same GET + EXPIRE race
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

**Fix:** Lua script for atomic check-and-extend:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# In _beat:
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat: lock expired or stolen, stopping beat key=%s", lock_key)
    self._heartbeat_active = False
    break
```

### 3. Instance-Level Heartbeat State Breaks Concurrency

**File:** `services/inventory.py`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` stores heartbeat state on `self`. If this service instance handles concurrent calls (e.g., via a thread pool), two simultaneous `reserve_item` calls will stomp on each other's `_heartbeat_active` flag. One call stopping its heartbeat will kill the other's.

**Fix:** Move heartbeat state into the lock context itself, scoped to the call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    # ... acquire logic ...

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if not heartbeat_active.is_set():
                break
            extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Heartbeat lost lock key=%s", lock_key)
                break

    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_script(keys=[lock_key], args=[token])
```

Note the `join()` — without it, the daemon thread may attempt to extend a lock you just released.

### 4. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will silently process it.

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str
    quantity: int

    @field_validator("item_id")
    @classmethod
    def item_id_must_not_be_blank(cls, v: str) -> str:
        if not v or not v.strip():
            raise ValueError("item_id must not be blank")
        return v.strip()

    @field_validator("quantity")
    @classmethod
    def quantity_must_be_positive(cls, v: int) -> int:
        if v < 1:
            raise ValueError("quantity must be at least 1")
        return v

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        return ReservationResult(success=False, message=f"Invalid input: {e}")
    # use req.item_id, req.quantity from here
```

### 5. DB Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. This is a silent inventory leak. The Redis lock does not protect against application-layer partial failure.

**Fix:** Wrap in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction doesn't support transactions, that interface needs to be revisited before this service ships.

## Moderate Issues

### 6. Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

Any `RuntimeError` from inside the lock — including bugs in `db.update_stock` — will be silently swallowed and returned as "service unavailable." Define a specific exception:

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch only LockAcquisitionError in reserve_item
```

### 7. Test Suite Mocks Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # <-- mocked out
    svc._stop_heartbeat = MagicMock()   # <-- mocked out
    return svc
```

The heartbeat and lock release logic — the most failure-prone parts — are not exercised by any test. `test_lock_not_released_if_expired` is the one test that doesn't mock the heartbeat, but it doesn't actually verify the Lua script path either since the atomic release isn't implemented yet.

Add integration-style tests using `fakeredis` that exercise the full lock lifecycle, including expiry and concurrent acquisition.

---

### 8. `LOCK_TTL = 30` With No Justification or Cap

A 30-second lock on a flash sale item means one slow DB call can block all other buyers for 30 seconds. This needs a documented SLA budget and should be configurable at construction time, not hardcoded as a class variable.

```python
def __init__(
    self,
    redis_client: redis.Redis,
    db,
    lock_ttl: int = 10,
    heartbeat_interval: int = 3,
):
```

| # | Severity | Issue |
|---|----------|-------|
| 1 | Critical | Non-atomic lock release — use Lua script |
| 2 | Critical | Non-atomic heartbeat extend — use Lua script |
| 3 | Critical | Instance-level heartbeat state breaks concurrent use |
| 4 | High | No actual input validation on `reserve_item` |
| 5 | High | Non-transactional DB writes risk inventory leak |
| 6 | Moderate | Overly broad `RuntimeError` catch masks bugs |
| 7 | Moderate | Tests mock out the critical path being reviewed |

The architecture is sound and the intent is correct. Fix the atomicity issues and concurrency scope problems first — those are the ones that will cause real data corruption under flash sale load.
