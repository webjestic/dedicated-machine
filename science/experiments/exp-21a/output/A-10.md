Decision: Request Changes

Summary: The core locking logic is directionally sound — atomic `SET NX EX`, UUID ownership verification, and heartbeat renewal are all reasonable primitives — but there are several correctness and safety issues that must be resolved before this is production-ready. The most critical flaw is a shared-instance concurrency bug that makes the service unsafe under any real concurrent load, which directly contradicts the stated purpose of the PR.

---

## Detailed Findings

### 🔴 Critical — Shared Instance State Breaks Concurrent Safety

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are **instance-level fields**, not call-scoped locals. If a single `InventoryService` instance handles two concurrent `reserve_item` calls (which is the entire point of this service), the second call's `_start_heartbeat` will overwrite `_heartbeat_thread`, and either call's `_stop_heartbeat` will set `_heartbeat_active = False` for **both** threads simultaneously.

**Consequence:** Under concurrent load, one reservation can silently kill the heartbeat of another, allowing the lock to expire mid-operation — the exact zombie-write failure mode this PR claims to prevent.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass state via closure or a small dataclass:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    # ... acquire lock, start thread with heartbeat_active, yield, clear event
```

Using `threading.Event` also eliminates the busy-sleep pattern and gives you a clean, immediate shutdown signal.

### 🔴 Critical — Heartbeat Race on Lock Release

```python
finally:
    self._stop_heartbeat()                          # sets flag False
    current = self.redis.get(lock_key)              # heartbeat may still be running here
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

`_stop_heartbeat()` sets a flag but does **not join the thread**. The heartbeat can execute a `redis.expire` call between `_stop_heartbeat()` and `redis.delete()`, which is harmless in isolation, but more importantly the thread is not guaranteed to have observed the flag before the owning context exits. Under the call-scoped fix above, you must join or use an event with a timeout:

```python
finally:
    heartbeat_active.clear()
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    # then release lock
```

### 🔴 Critical — Lock Release Is Not Atomic

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can acquire between GET and DELETE
```

This is a classic TOCTOU. Between `GET` and `DELETE`, another process can acquire the lock after a legitimate expiry. You would then delete a lock you no longer own.

**Fix:** Use the canonical Lua script for atomic compare-and-delete, which is the Redis-documented correct approach:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```

### 🟡 Medium — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. Pydantic is only used on the **output** model. There is no guard against:

- `quantity <= 0` (reserve zero or negative items)
- `item_id` being an empty string or containing injection-relevant characters for your key namespace

**Fix:** Add a request model or explicit guards:

```python
if not item_id or not item_id.strip():
    return ReservationResult(success=False, message="Invalid item_id")
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

### 🟡 Medium — DB Operations Are Not Atomic With Respect to Each Other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The Redis lock does not protect against partial DB writes. This needs to be wrapped in a database transaction, or the failure path needs compensating logic:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 🟡 Medium — Heartbeat Interval Allows Near-Expiry Window

`LOCK_TTL = 30s`, `HEARTBEAT_INTERVAL = 10s`. The heartbeat fires at 10s, 20s, 30s. At exactly 30s the lock expires before the third heartbeat fires. The effective safe window is `LOCK_TTL - HEARTBEAT_INTERVAL`, not `LOCK_TTL`. This should be documented explicitly, and the ratio should be enforced:

```python
assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL / 2, \
    "Heartbeat interval must be less than half the lock TTL"
```

---

### 🟡 Medium — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # ← heartbeat logic entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat and lock lifecycle are the novel and risky parts of this PR. Mocking them out means the tests provide no coverage for the concurrency bugs identified above. At minimum, add integration-level tests using a real or fake Redis (e.g., `fakeredis`) that exercise the full lock path including heartbeat behavior.

---

### 🟢 Low — `daemon=True` Thread Suppresses Shutdown Errors Silently

Daemon threads are killed abruptly on process exit without running cleanup. If the process exits while a heartbeat thread holds a lock reference, the lock will simply expire rather than be released cleanly. This is acceptable behavior but should be documented, and the join-on-release fix above mitigates the in-process case.
