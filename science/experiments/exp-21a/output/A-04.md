Decision: Request Changes

Summary: The core locking logic is structurally sound — atomic `SET NX EX`, UUID ownership verification, and heartbeat renewal are all reasonable approaches — but there are several correctness, safety, and architectural issues that must be resolved before this merges. The most critical is a race condition in the heartbeat and a shared-state concurrency bug that directly undermines the guarantees this service is supposed to provide.

---

## Critical Issues

### 1. Shared Heartbeat State — Instance-Level Concurrency Bug

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on `InventoryService`, not scoped to individual lock acquisitions.

```python
# Current — shared across all concurrent calls on the same instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same service instance (which is the entire point of this service), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, silently killing the first call's heartbeat. The first lock will then expire mid-operation, and the second call's `_stop_heartbeat()` will also stop the second heartbeat prematurely.

**Fix:** Scope heartbeat state to each lock acquisition, not the instance. Pass a `threading.Event` or equivalent into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

### 2. Heartbeat Race Condition on Lock Release

In `_acquire_lock`, the `finally` block calls `_stop_heartbeat()` then immediately reads and deletes the key. The heartbeat thread is not joined before the release check executes.

```python
finally:
    self._stop_heartbeat()          # Sets flag to False
    current = self.redis.get(lock_key)   # Heartbeat may still be running here
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

The heartbeat thread could be mid-execution between the `get` and `expire` calls at exactly this moment. This is a TOCTOU window where the heartbeat extends the TTL of a lock you are in the process of releasing. With the `stop_event.set()` + `join()` pattern above, this is eliminated.

---

### 3. Lock Release Is Not Atomic

The release sequence is a non-atomic read-then-delete:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock (if TTL expired in that window), and you would then delete their lock. This is a well-known failure mode with Redis distributed locks. The standard fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

Same issue exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a Lua script there as well.

---

### 4. DB Operations Are Not Atomic — No Rollback Path

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, stock is decremented but no order exists. This is a silent data corruption path. These two operations must be wrapped in a database transaction, or a compensating rollback must be implemented.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Moderate Issues

### 5. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used for the return type. There is no guard against:

- `quantity <= 0` (reserving zero or negative items would succeed silently)
- Empty string `item_id`
- Excessively long `item_id` strings used as Redis key components

```python
if quantity <= 0:
    raise ValueError(f"quantity must be positive, got {quantity}")
if not item_id or not item_id.strip():
    raise ValueError("item_id must not be empty")
```

---

### 6. Heartbeat Interval Relative to TTL Is Risky Under Load

`HEARTBEAT_INTERVAL = 10` with `LOCK_TTL = 30` means the heartbeat fires at 1/3 of the TTL. Under a GIL-induced thread scheduling delay or a Redis network hiccup, the heartbeat could miss its window. A safer ratio is 1/5 or 1/6 of TTL, and the heartbeat should log a warning if the renewal fails rather than silently continuing.

---

### 7. Daemon Thread Leak on Process Shutdown

Daemon threads are acceptable here, but `_stop_heartbeat` without a `join()` means there is no guarantee the thread has actually stopped before the lock is released. As noted in issue #1, this is resolved by the `stop_event` + `join()` pattern.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test mocks the heartbeat, meaning the actual heartbeat code path — including the race condition in issue #1 and #2 — has zero test coverage. At minimum, one integration-style test should exercise the real heartbeat with a short TTL to verify renewal behavior.

### 9. DB Transaction Failure Is Not Tested

There is no test for the case where `create_order` raises after `update_stock` succeeds. Once issue #4 is fixed, this path must be covered.

### 10. `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
redis_client.get.return_value = b"different-token"
result = service.reserve_item("item_001")
redis_client.delete.assert_not_called()
```

This test passes `different-token` back from `get`, which means the mock simulates a stolen lock — but the reservation itself still succeeds because `reserve_item` does not check the token during the operation, only at release. The test name implies it is testing a safety property, but it is actually testing that `delete` is skipped on release, while a successful reservation result is returned. This is misleading and masks the fact that a stolen lock mid-operation is not detected at all.
