Decision: Request Changes

Summary: The implementation demonstrates solid architectural intent — distributed locking with ownership verification, heartbeat renewal, and exponential backoff are all appropriate patterns for this use case. However, there are several correctness and safety issues that must be resolved before this can be approved, ranging from a critical race condition in the heartbeat to a concurrency bug that makes the service unsafe for multi-threaded deployments.

---

## Critical Issues

### 1. Instance-level heartbeat state breaks concurrent usage

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes, not scoped to the lock context. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously — which is the entire point of this service — they will clobber each other's heartbeat state.

```python
# Thread A sets this to True, starts its heartbeat
self._heartbeat_active = True

# Thread B calls _stop_heartbeat(), kills Thread A's heartbeat
self._heartbeat_active = False
```

The fix is to scope heartbeat state to the lock context, not the instance. Use a local `threading.Event` passed into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

```python
def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 2. Race condition between heartbeat check and lock release

In `_acquire_lock`, the lock release is:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process, which you then delete. This is a classic TOCTOU bug. The canonical fix is a Lua script for atomic check-and-delete, which Redis guarantees executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same race exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a similar Lua script.

---

### 3. `db.update_stock` and `db.create_order` are not atomic

The lock protects the read-modify-write cycle, but if `update_stock` succeeds and `create_order` raises an exception, you have decremented stock with no corresponding order record. This is a data integrity failure.

```python
# Stock decremented here
self.db.update_stock(item_id, new_stock)

# If this throws, stock is gone but no order exists
self.db.create_order(order_id, item_id, quantity)
```

These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should be documented or enforced to expose a transaction context manager.

## Moderate Issues

### 4. Input validation is claimed but not implemented

The checklist marks "Input validation via Pydantic" but there is no validation on `item_id` or `quantity` in `reserve_item`. Pydantic is imported for the response model only. A negative or zero quantity, or an empty/malformed `item_id`, will pass through silently.

```python
# This currently succeeds with no error
service.reserve_item("", quantity=-5)
```

Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

---

### 5. Heartbeat uses `time.sleep` — not responsive to stop signal

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)  # sleeps 10s before checking stop flag
```

With a 10-second sleep, the heartbeat thread can linger up to 10 seconds after the lock is released. With the `threading.Event` fix above (`stop_event.wait(timeout=...)`), this is resolved cleanly and the thread exits promptly.

---

### 6. Bare `RuntimeError` is too broad in the exception handler

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

A `RuntimeError` raised inside `db.update_stock` or `db.create_order` would be silently swallowed and returned as "service unavailable" rather than surfaced as an error. Use a dedicated exception class:

```python
class LockAcquisitionError(Exception):
    pass

# Raise it specifically in _acquire_lock
raise LockAcquisitionError(f"Could not acquire lock for {lock_key}...")

# Catch only that
except LockAcquisitionError as e:
    ...
```

## Test Coverage Gaps

### 7. Heartbeat is fully mocked out — the most critical path is untested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most novel and risky part of this implementation. There should be at least one integration-level test that runs the real heartbeat logic against a mock Redis and verifies that `expire` is called and that the thread stops cleanly after lock release.

### 8. The TOCTOU lock release path is tested incorrectly

`test_lock_not_released_if_expired` mocks `redis.get` to return a different token, but since `_start_heartbeat` is also mocked, the test does not exercise the real `_acquire_lock` flow. Once the heartbeat is scoped locally (fix #1), this test will need to be rewritten to exercise the actual `finally` block.

### 9. No test for partial DB failure (fix #3 dependency)

Once transactions are added, there must be a test confirming that a failure in `create_order` rolls back the stock decrement.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is reasonable, but worth a comment explaining the relationship (heartbeat fires at 1/3 of TTL, giving two renewal windows before expiry).
- Consider whether `reserve_item` should propagate Redis connection errors rather than catching them silently. A Redis outage is operationally distinct from lock contention.
- The `db` interface has no type hints or abstract base class definition in this PR. For a service this critical, an explicit `Protocol` or `ABC` would make the contract auditable.
