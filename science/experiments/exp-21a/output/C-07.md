Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `NX+EX`, ownership-verified release, heartbeat renewal, and structured logging are all sound patterns. However, there are several correctness, safety, and architecture issues that must be addressed before this can be merged into a production government-facing system.

---

## Critical Issues

### 1. Race Condition in Heartbeat Token Check

The heartbeat performs a non-atomic `GET` + `EXPIRE` sequence:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, another process could acquire the lock. This extends a lock you no longer own. This **must** be replaced with a Lua script to make the check-and-extend atomic:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

### 2. Non-Atomic Lock Release

The same TOCTOU race exists in `_acquire_lock`'s `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a well-known pattern that **requires** a Lua script. Use the canonical implementation:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 3. Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is a shared singleton (the typical case with a DI container), concurrent calls to `reserve_item` will corrupt each other's heartbeat state. The heartbeat context must be scoped to the lock acquisition, not the instance:

```python
# Pass state via closure or a dedicated HeartbeatContext dataclass
# Do NOT use self._heartbeat_active
```

### 4. No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. There is no guard against:

- Empty string `item_id` (produces a valid but meaningless lock key)
- `quantity <= 0` (would allow negative stock)
- Excessively long `item_id` strings (potential key-space abuse in Redis)

Add a Pydantic input model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Moderate Issues

### 5. Database Operations Are Not Transactional

`update_stock` and `create_order` are two separate calls with no transaction wrapping:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises, stock is decremented but no order exists. This is a data integrity failure. These must execute within a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 6. Heartbeat Thread Is Never Joined

`_stop_heartbeat` only sets a flag. The thread is `daemon=True`, which prevents clean shutdown blocking, but there is no `join()` with a timeout. Under high throughput this leaks thread handles and makes shutdown behavior unpredictable. At minimum:

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._heartbeat_thread = None
```

### 7. Bare `RuntimeError` Is Too Broad

The `except RuntimeError` block will silently swallow any `RuntimeError` raised inside the lock body (e.g., from `db.update_stock`), returning a misleading "service unavailable" message. Define and raise a specific exception class:

```python
class LockAcquisitionError(Exception): ...
```

## Minor Issues

### 8. Test Coverage Gaps

- No test for the heartbeat Lua atomicity path
- No test for `quantity <= 0` or empty `item_id`
- `test_lock_not_released_if_expired` mocks the token comparison but does not assert the log warning was emitted — low signal
- The `service` fixture mocks `_start_heartbeat` and `_stop_heartbeat` entirely, meaning the heartbeat logic itself has **zero test coverage**

### 9. Lock Key Namespace Collision Risk

`f"inventory:lock:{item_id}"` — if `item_id` is user-supplied and not sanitized (see Issue 4), a crafted input like `../admin` or `lock:global` could produce unexpected key patterns. The regex validation in Issue 4 mitigates this, but it should be called out explicitly in a code comment.

### 10. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Should Be Configurable

Hardcoded class-level constants are not appropriate for a service that will run across environments with different Redis latency profiles. Accept these in `__init__` with sensible defaults.
