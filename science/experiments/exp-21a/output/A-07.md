Decision: Request Changes

Summary: The core locking logic and heartbeat design are directionally sound, but there are several concrete correctness and safety issues that must be addressed before this merges. The most critical is a shared-instance concurrency hazard in the heartbeat state, combined with a non-atomic lock release that can cause a race-condition data loss under the exact high-load scenario this service is designed to handle. Test coverage also has meaningful gaps that would let these bugs ship silently.

---

## Detailed Findings

### 🔴 Critical — Shared Heartbeat State Breaks Concurrent Instance Use

```python
# InventoryService stores heartbeat state as instance variables
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If a single `InventoryService` instance handles two concurrent calls to `reserve_item` (which is the entire point of this service), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, orphaning the first heartbeat and potentially stopping it prematurely. This is a silent correctness failure — no exception is raised, but the first lock loses its renewal.

**Fix:** Scope heartbeat state to the lock acquisition, not the instance. Pass a threading `Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...

def _start_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

### 🔴 Critical — Non-Atomic Lock Release Is a Race Condition

```python
# In the finally block:
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process can acquire the lock (if the TTL expired in that window). The `DELETE` then destroys a lock you no longer own. This is the fencing token violation the PR claims to defend against. The fix is a single atomic Lua script, which is the Redis-documented correct approach.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

### 🔴 Critical — DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access. These two operations must be wrapped in a database transaction. If the DB layer does not support transactions, this needs explicit compensating logic.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 🟡 Major — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # Thread is never joined
```

The daemon thread continues running for up to `HEARTBEAT_INTERVAL` seconds after the lock is released. During that window it can call `redis.expire` on a key now owned by a different process, silently extending a lock you no longer hold. Joining the thread in the `finally` block (as shown in the fix above) eliminates this.

### 🟡 Major — `quantity` Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller passing `quantity=0` or `quantity=-5` will pass the stock check and either create a zero-quantity order or *increase* stock (`new_stock = 10 - (-5) = 15`). Pydantic is listed in the checklist but is only applied to the result model, not the input.

```python
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

Or enforce it at the model boundary with a Pydantic input model.

---

### 🟡 Major — `item_id` Is Not Sanitized Before Use as a Redis Key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-controlled and contains `:`, `*`, `?`, or newline characters, this can produce unexpected key patterns or interact badly with Redis keyspace notifications and scanning. Validate `item_id` against an allowlist pattern (alphanumeric plus hyphen/underscore) before constructing the key.

---

### 🟡 Major — Heartbeat Interval Exceeds Half the Lock TTL Safety Margin

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The ratio here is acceptable (10s renewal against 30s TTL), but the heartbeat uses `time.sleep(HEARTBEAT_INTERVAL)` which means the *first* renewal does not happen until 10 seconds in. Under a GIL-induced or GC pause, the effective renewal gap can approach the TTL. The standard recommendation is to renew at `TTL / 3` or less, and to renew *before* sleeping, not after.

---

### 🟠 Minor — Tests Mock Away the Exact Code Under Review

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat is the most novel and risky part of this PR. Mocking it out entirely means the tests provide no coverage for the concurrency behavior being advertised. Add at least one integration-style test that lets the real heartbeat run and asserts `redis.expire` is called within the expected window.

---

### 🟠 Minor — `test_lock_not_released_if_expired` Tests the Wrong Thing

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts the current (racy GET-then-DELETE) behavior rather than the correct atomic behavior. Once the Lua script fix is applied, this test should verify that the script was called with the correct token, not that `delete` was skipped.
