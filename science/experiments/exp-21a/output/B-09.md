Decision: Request Changes

Summary: The implementation contains several critical architectural flaws that would make it unsafe for a high-concurrency flash sale environment. The most serious issues are a race condition in the heartbeat's check-then-extend sequence, a shared-instance concurrency bug that makes the service non-reentrant, and the absence of database transaction atomicity, which means the lock boundary does not actually protect against partial writes. These issues must be resolved before this code is merged.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

**1. Shared instance state breaks reentrancy and concurrent use**

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance-level attributes. If a single `InventoryService` instance is shared across threads (which is the normal pattern for a service object), two concurrent calls to `reserve_item` will clobber each other's heartbeat state. The second call's `_stop_heartbeat()` will set `_heartbeat_active = False`, silently killing the first call's heartbeat mid-operation, allowing that lock to expire without renewal.

The heartbeat state must be scoped to the lock context, not the instance. Pass a mutable flag or use a `threading.Event` local to `_acquire_lock`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into heartbeat thread instead of using self._heartbeat_active
    ...
```

---

**2. Heartbeat has a TOCTOU race on token verification**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The `GET` and `EXPIRE` are two separate round trips. Between them, the lock can expire and be acquired by another process. You would then extend a lock you no longer own, directly causing the fencing token violation the PR claims to prevent. This must be an atomic operation. Use a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same TOCTOU problem exists in the lock release path in `_acquire_lock`'s `finally` block. The `GET` + conditional `DELETE` must also be a single atomic Lua script or a Redis transaction.

---

**3. No database transaction wrapping the two DB writes**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent reads of stale stock, but it does not protect against a partial write leaving the database in an inconsistent state. This is a data integrity bug independent of the locking strategy.

At minimum, these must be wrapped in a database transaction. Ideally, the service should accept a transaction/session object or the DB layer should expose a transactional method.

---

### 🟡 Significant — Should Fix

---

**4. `quantity` has no lower-bound validation**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`. A caller passing `quantity=0` will always succeed (0 < any non-negative stock), write a zero-quantity order, and log a successful reservation. A negative quantity would *increase* stock. Input validation is listed as complete on the checklist, but this case is missing. Add an early guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

**5. `item_id` is not validated before being interpolated into the lock key**

```python
lock_key = f"inventory:lock:{item_id}"
```

An `item_id` containing a colon, whitespace, or Redis special characters could produce a malformed or colliding key. Since `item_id` comes from external input, it should be validated against an allowlist pattern (e.g., alphanumeric plus hyphen/underscore) before use.

---

**6. Daemon heartbeat thread is silently swallowed on Redis failure**

The heartbeat thread has no error handling. If the Redis connection drops mid-operation, the `redis.get()` or `redis.expire()` call will raise an exception that is silently swallowed inside the thread, the heartbeat stops without any signal to the main thread, and the lock expires. The main thread has no way to know the lock was lost and will proceed to write to the database believing it holds the lock. The heartbeat thread must catch exceptions, log them, and ideally signal the main thread to abort.

---

### 🔵 Minor — Recommended

---

**7. Test suite mocks away the components under test**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The fixture stubs out the heartbeat entirely. This means the tests do not exercise the actual lock lifecycle, the heartbeat thread behavior, or the `finally` release path in any realistic way. `test_lock_not_released_if_expired` in particular asserts behavior about the release path but the mock fixture means the real `_acquire_lock` context manager is not being tested. Integration-level tests against a real or embedded Redis instance (e.g., `fakeredis`) should be added to cover the actual lock acquire/release/extend cycle.

---

**8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are hardcoded class constants**

These should be configurable at construction time. Different deployment environments (test, staging, production) and different item categories may require different TTL tuning. Hardcoded class-level constants make this difficult to adjust without code changes.
