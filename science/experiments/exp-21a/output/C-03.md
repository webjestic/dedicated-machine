Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `nx=True`, token-based ownership verification, heartbeat renewal, and exponential backoff are all sound patterns. However, there are several correctness, safety, and concurrency issues that must be resolved before this handles real flash-sale load in a government-grade system. The most critical flaw is a race condition in the heartbeat's check-then-extend sequence and a shared-state concurrency bug that makes the service unsafe for multi-threaded use.

---

## Detailed Findings

### 🔴 Critical — Must Fix

**1. Heartbeat has a TOCTOU race condition (`_beat` function)**

```python
# Current — NOT atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock could expire between these two calls
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You then blindly extend a lock you no longer own. This must be an atomic operation. Use a Lua script:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register once at __init__
self._extend_script = self.redis.register_script(EXTEND_SCRIPT)

# In _beat:
extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.warning("Heartbeat lost lock ownership key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

Similarly, the lock release in `_acquire_lock`'s `finally` block has the same TOCTOU issue and should also use a Lua script (this is the canonical Redlock pattern):

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""
```

**2. Instance-level heartbeat state makes `InventoryService` non-reentrant and thread-unsafe**

```python
# These are instance variables — shared across ALL concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same service instance (entirely normal in a web server), one thread's `_stop_heartbeat()` will kill the other thread's heartbeat. This is a silent data-integrity bug under load.

Fix by scoping heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    acquired = False

    # ... acquire logic ...

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            extended = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not extended:
                logger.warning("Lost lock key=%s", lock_key)
                break

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_script(keys=[lock_key], args=[token])
```

Note the `thread.join()` — without it, a heartbeat thread can outlive the lock release and attempt a spurious extension.

### 🟡 Significant — Should Fix

**3. No input validation on `reserve_item`**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw, unvalidated arguments. `item_id` is passed directly into a Redis key and a DB query with no sanitization. A malformed `item_id` like `"item:lock:override"` could collide with the lock key namespace.

```python
# Minimum required
if not item_id or not isinstance(item_id, str) or len(item_id) > 128:
    return ReservationResult(success=False, message="Invalid item_id")
if not isinstance(quantity, int) or quantity < 1 or quantity > 1000:
    return ReservationResult(success=False, message="Invalid quantity")
```

Or define a `ReservationRequest(BaseModel)` and validate at the boundary — that would actually justify the Pydantic import.

---

**4. `db.update_stock` and `db.create_order` are not atomic**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # crash here = phantom stock decrement
```

If the process dies or the DB throws between these two calls, stock is decremented but no order exists. The distributed lock protects against concurrent access, not partial writes. These two operations must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

**5. Heartbeat interval is dangerously close to TTL**

`LOCK_TTL = 30s`, `HEARTBEAT_INTERVAL = 10s`. The first heartbeat fires at T+10s, which is fine. But `time.sleep()` is not precise under GIL pressure or OS scheduling jitter. Under load, the actual sleep can exceed 10s, and with only 30s TTL, you have limited margin. Either reduce the interval ratio or use a monotonic deadline check. A ratio of at least 3:1 (TTL to interval) with the `threading.Event` pattern above is safer.

### 🔵 Minor — Recommended

**6. Test suite mocks out the exact behavior under test**

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # heartbeat is never tested
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat logic — the most complex and failure-prone part of this PR — has zero test coverage. Add at minimum:
- A test that verifies the Lua extension script is called during a long-running operation
- A test that verifies the heartbeat stops cleanly when the lock is released
- A test that verifies a lost lock (extension returns 0) stops the heartbeat

**7. `RuntimeError` is too broad a catch**

```python
except RuntimeError as e:
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

A `RuntimeError` from inside `db.get_stock()` or `db.update_stock()` would be silently swallowed and reported as a lock failure. Define a specific `LockAcquisitionError` exception class so the catch is precise.
