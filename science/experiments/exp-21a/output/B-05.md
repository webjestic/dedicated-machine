Decision: Request Changes

Summary: The implementation has a critical concurrency flaw that undermines the entire purpose of the distributed lock: `InventoryService` stores heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) as instance-level attributes, meaning a single shared service instance handling concurrent requests will have threads stomping on each other's heartbeat lifecycle — one request's `_stop_heartbeat()` call will silently kill another request's renewal thread, exposing the lock to premature expiry under the exact high-concurrency flash sale load this code is designed to handle. There are also meaningful atomicity, input validation, and test coverage gaps that need to be addressed before this is production-safe. Detailed findings below.

---

## Critical

### 1. Instance-Level Heartbeat State Is Not Thread-Safe

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are shared across all concurrent calls on the same instance. Under concurrent load:

- Request A acquires lock, starts heartbeat, sets `_heartbeat_active = True`
- Request B acquires a different lock, calls `_start_heartbeat`, overwrites `_heartbeat_thread`
- Request A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`
- Request B's heartbeat thread silently dies; its lock expires mid-operation

The heartbeat state must be scoped to the individual lock acquisition, not the instance. The fix is to pass a mutable stop-signal (e.g., a `threading.Event`) into the heartbeat closure and return it from `_start_heartbeat`:

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller holds the stop handle
```

Then in `_acquire_lock`:

```python
stop_event = self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    stop_event.set()
    # ... release lock
```

Note also that `time.sleep(HEARTBEAT_INTERVAL)` followed by a separate `if self._heartbeat_active` check is a TOCTOU race. Using `threading.Event.wait(timeout=...)` as shown above eliminates that race and allows immediate clean shutdown.

### 2. Non-Atomic Read-Modify-Write Is Not Protected End-to-End

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

The Redis lock serializes access at the application layer, but if `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record. There is no transaction wrapping these two DB calls. This needs either:

- A database transaction (preferred), or
- Explicit compensating logic on failure

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

## Major

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. Pydantic is only used on the *output* model. A caller can pass:

- `quantity=0` or `quantity=-5` — passes the stock check, creates a fraudulent order
- `item_id=""` or `item_id="../../../../etc"` — flows directly into the lock key and DB query

At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity < 1:
    raise ValueError("quantity must be a positive integer")
```

Or introduce a Pydantic input model to match what the checklist actually claims.

### 4. Lock Release Has a TOCTOU Race in the `finally` Block

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process could acquire the lock (if it expired and was re-acquired). The correct pattern is an atomic Lua script, which is the Redis-documented standard for safe lock release:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

## Moderate

### 5. Heartbeat Can Silently Extend a Lock You No Longer Own

The heartbeat calls `redis.expire(lock_key, self.LOCK_TTL)` after verifying the token matches. But `GET` + `EXPIRE` is not atomic either. If the lock expires between the `GET` and the `EXPIRE`, a new owner acquires it and the heartbeat extends *their* lock. This should also be a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

### 6. Unhandled Redis Exceptions Propagate as Uncaught Errors

Only `RuntimeError` (lock contention) is caught. Any `redis.RedisError` (connection drop, timeout) during `get_stock`, `update_stock`, or `create_order` will bubble up as an unhandled exception. Given this is a flash sale service expected to run under load, Redis instability is a realistic scenario. Add explicit handling:

```python
except redis.RedisError as e:
    logger.error("Redis error item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Minor / Test Coverage Gaps

### 7. Tests Mock Away the Exact Behavior Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most novel and risky part of this PR. Mocking it out entirely means the thread lifecycle, the stop-signal race, and the token-check logic are completely untested. Add at least one integration-style test that lets the heartbeat thread run and verifies it extends the TTL and stops cleanly.

### 8. No Test for Partial DB Failure

There is no test covering the case where `update_stock` succeeds but `create_order` raises. This is the exact scenario that would cause silent stock corruption in production.

### 9. `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

These should be configurable at construction time to allow environment-specific tuning and easier testing without monkey-patching class attributes.
