Decision: Request Changes

Summary: The core locking logic and heartbeat design are directionally sound, but there are several concrete concurrency, correctness, and reliability defects that must be resolved before this merges. The issues range from a race condition that can cause double-writes under concurrent load, to an instance-level state design that breaks under any realistic multi-threaded deployment, to test coverage that mocks away the exact code paths most likely to fail in production.

## Detailed Review

### 🔴 Critical — Must Fix

---

#### 1. Instance-level heartbeat state is not thread-safe and breaks under concurrency

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the service instance, not scoped to the lock invocation. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the entire point of this service — one thread's `_stop_heartbeat()` call will kill the other thread's heartbeat. This is a silent failure: the lock will expire mid-operation with no exception raised, the heartbeat's purpose is defeated, and you are back to the zombie-write scenario this code was written to prevent.

**Fix:** Scope heartbeat state to the lock context, not the instance. Pass a mutable container (e.g., a `threading.Event`) into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    # ... acquire logic ...

    heartbeat_thread = threading.Thread(
        target=self._beat,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release logic ...
```

#### 2. Race condition between heartbeat check and lock release

In `_beat`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

And in the `finally` block:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Both the heartbeat thread and the main thread perform a non-atomic check-then-act on the same key. Between the `GET` and the `EXPIRE` or `DELETE`, another process can acquire the lock. This is a classic TOCTOU race. The PR description explicitly mentions fencing token violations — this pattern introduces exactly that.

**Fix:** Both the heartbeat renewal and the lock release must be atomic. Use a Lua script evaluated server-side.

```lua
-- Heartbeat renewal
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```lua
-- Lock release
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

This is well-established practice and is documented in the Redis distributed locks specification (Redlock).

#### 3. No input validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no validation. There is no Pydantic model gating the inputs. A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a string containing path traversal or injection characters depending on how `item_id` is used downstream in `db.get_stock`.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

**Fix:** Define and enforce an input model at the boundary.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, request: ReservationRequest) -> ReservationResult:
```

### 🟡 Significant — Should Fix Before Merge

---

#### 4. DB operations are not atomic with respect to each other

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `create_order` fails after `update_stock` succeeds, stock is decremented with no corresponding order. The distributed lock protects against concurrent reads of stale stock, but it does not protect against partial writes. This is a data integrity issue independent of the locking strategy.

**Fix:** Wrap both operations in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### 5. Heartbeat thread is daemon but is never joined

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
self._heartbeat_thread.start()
```

`_stop_heartbeat` sets the flag but does not join the thread. After `_stop_heartbeat()` returns in the `finally` block, the heartbeat thread may still be alive and may execute one more `expire` call after the lock has already been deleted and potentially re-acquired by another process. The `daemon=True` flag prevents interpreter shutdown from blocking, but it does not prevent this post-release extension.

**Fix:** Join the thread with a bounded timeout in the `finally` block (as shown in fix #1 above).

---

#### 6. `quantity` defaults to `1` but is never validated as a positive integer

A call to `reserve_item("item_001", quantity=0)` will pass the stock check (`current_stock < 0` is false for any non-negative stock), decrement stock by zero, and create an order for zero units. This is a logic error that produces a valid-looking `ReservationResult` with `success=True`.

### 🔵 Observations — Address in Follow-up

---

#### 7. Tests mock away the code under test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

The heartbeat and lock release paths are the highest-risk code in this PR. Mocking them out means the tests do not exercise the actual `_acquire_lock` context manager, the Lua script atomicity, or the thread lifecycle. `test_lock_not_released_if_expired` is the most important test in the suite and it asserts nothing about the heartbeat behavior.

Add integration-level tests using a real or fakeredis instance that exercise the full lock acquire → heartbeat → release cycle, including the token-mismatch branch.

---

#### 8. Lock key namespace is predictable and not tenant-scoped

```python
lock_key = f"inventory:lock:{item_id}"
```

If this Redis instance is shared across environments or tenants, lock keys from one context will collide with another. Consider namespacing by environment or tenant ID.

---

#### 9. `LOCK_TTL` and `HEARTBEAT_INTERVAL` are class-level constants with no relationship enforcement

If `HEARTBEAT_INTERVAL >= LOCK_TTL`, the heartbeat will never fire before the lock expires. This relationship is implicit and fragile. Add an assertion in `__init__` or enforce it structurally.

```python
def __init__(self, ...):
    assert self.HEARTBEAT_INTERVAL < self.LOCK_TTL, (
        "HEARTBEAT_INTERVAL must be less than LOCK_TTL"
    )
```
