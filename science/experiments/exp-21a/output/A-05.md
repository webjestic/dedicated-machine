Decision: Request Changes

Summary: The core locking logic and heartbeat concept are directionally sound, but there are several concrete concurrency, correctness, and operational flaws that must be resolved before this merges. The most critical issue is a race condition in the heartbeat that can produce the exact zombie-write failure mode the PR claims to prevent. Secondary issues around instance-level state, missing input validation, non-atomic lock release, and inadequate test coverage compound the risk enough that this cannot be approved for a high-throughput flash sale workload.

---

## Detailed Review

### 🔴 Critical — Race Condition in Heartbeat (Defeats the Core Safety Guarantee)

The heartbeat performs a **non-atomic check-then-extend**:

```python
current = self.redis.get(lock_key)          # step 1: read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # step 2: extend
```

Between step 1 and step 2, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns, silently handing a longer lease to the new owner while the original holder continues writing. This is precisely the fencing token violation the PR claims to guard against.

**Fix:** Replace with a Lua script to make the check-and-extend atomic:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True
    extend_lock = self.redis.register_script(EXTEND_SCRIPT)

    def _beat() -> None:
        while self._heartbeat_active:
            time.sleep(self.HEARTBEAT_INTERVAL)
            if not self._heartbeat_active:
                break
            result = extend_lock(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning("Heartbeat lost lock ownership key=%s — halting extension", lock_key)
                self._heartbeat_active = False
                break
            logger.debug("Lock TTL extended key=%s", lock_key)

    self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    self._heartbeat_thread.start()
```

### 🔴 Critical — Instance-Level Heartbeat State Breaks Concurrent Use

`_heartbeat_active` and `_heartbeat_thread` are stored on the instance, not scoped to the lock context:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the normal deployment pattern), one call's `_stop_heartbeat()` will cancel the other's heartbeat mid-operation. This is a silent data integrity failure with no exception raised.

**Fix:** Scope heartbeat state to the lock context using a local `threading.Event`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    stop_event = threading.Event()

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        # Atomic release via Lua
        release_lock = self.redis.register_script(RELEASE_SCRIPT)
        release_lock(keys=[lock_key], args=[token])
```

---

### 🔴 Critical — Non-Atomic Lock Release

The release in `finally` has the same check-then-act race as the heartbeat:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Another process can acquire the lock between `GET` and `DELETE`, causing deletion of a lock you no longer own.

**Fix:** Use a Lua script for atomic release:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 🟠 High — No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. There is no guard against:

- `quantity <= 0` (reserving zero or negative stock would silently increase inventory)
- `item_id` being an empty string or containing injection-relevant characters used in the lock key

**Fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

The `item_id` validation is particularly important because it is interpolated directly into the Redis key string.

### 🟠 High — DB Operations Not Atomic; No Rollback on Partial Failure

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented with no corresponding order. The lock provides mutual exclusion, but it does not provide durability. For a government contracting context, this is an audit and reconciliation liability.

**Fix:** Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not expose transactions, that is a gap in the `db` interface that must be addressed before this service is production-ready.

---

### 🟡 Medium — Heartbeat Thread Is Daemon but Never Joined

`daemon=True` means the thread will be killed abruptly on process exit, which is acceptable. However, `_stop_heartbeat()` only sets a flag — it does not join the thread. If the heartbeat is in the middle of a `time.sleep(10)` when the lock is released, it can wake up after release and attempt to extend a lock that is now owned by another process (mitigated but not eliminated by the atomic Lua fix above). Joining with a timeout makes the shutdown deterministic:

```python
stop_event.set()
if heartbeat_thread.is_alive():
    heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

---

### 🟡 Medium — Test Suite Stubs Out the Components Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat is fine for unit tests of the reservation logic, but there are **zero tests** for the heartbeat itself, the atomic lock extension, or the lock release path. Given that the PR's primary value proposition is correct distributed locking behavior, these are the most important paths to test.

Required additions:
- Test that heartbeat calls the Lua extension script with the correct token
- Test that heartbeat stops extending when token no longer matches (lock lost)
- Test that lock release uses the atomic Lua script
- Test that concurrent calls on the same instance do not interfere with each other's heartbeat state (integration-level)

---

### 🟡 Medium — `LOCK_TTL` and `HEARTBEAT_INTERVAL` Are Hardcoded Class Constants

These values are appropriate defaults, but they cannot be tuned per deployment without subclassing. For a service expected to handle variable DB latency profiles, these should be constructor parameters with defaults:

```python
def __init__(
    self,
    redis_client: redis.Redis,
    db,
    lock_ttl: int = 30,
    heartbeat_interval: int = 10,
):
```

---

### 🟢 Acknowledged Positives

- Lock acquisition uses `nx=True, ex=self.LOCK_TTL` in a single atomic `SET` command — correct.
- UUID token for ownership verification — correct pattern.
- Exponential backoff on contention — appropriate.
- `contextmanager` for lock scoping — clean design.
- Structured logging with key/value pairs — good operational hygiene.
- `ReservationResult` as a typed return value — good API design.
