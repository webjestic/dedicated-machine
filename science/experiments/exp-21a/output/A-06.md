Decision: Request Changes

Summary: The core locking logic and heartbeat concept are directionally sound, but there are several concrete concurrency, correctness, and safety defects that must be resolved before this can be merged into a production flash-sale path. The issues range from a race condition in the heartbeat that can cause a stale-lock extension after release, to an instance-level state model that breaks under any concurrent call pattern, to a non-atomic lock-release sequence that reintroduces the fencing violation the PR claims to prevent. The test suite also mocks away the heartbeat entirely, meaning the most critical concurrent behavior is untested.

---

## Detailed Findings

### 1. Critical — Instance-Level Heartbeat State Breaks Concurrent Calls

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the **instance**, not scoped to the lock context. If two calls to `reserve_item` overlap on the same `InventoryService` instance (which is the normal case behind a web framework), the second call's `_start_heartbeat` will overwrite `_heartbeat_active`, and the first call's `_stop_heartbeat` will kill the second call's heartbeat prematurely — or vice versa. This is a silent data-race with no exception raised.

**Required fix:** Scope heartbeat state to the lock context, not the instance. Pass a `threading.Event` into `_start_heartbeat` and return it; let `_stop_heartbeat` set that specific event.

```python
def _start_heartbeat(self, lock_key: str, token: str) -> threading.Event:
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return stop_event  # caller holds this reference
```

### 2. Critical — Non-Atomic Lock Release Reintroduces the Fencing Violation

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- GET
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # <-- DEL (separate round-trip)
```

Between `GET` and `DEL`, another process can acquire the lock. This is the classic check-then-act race. The PR description explicitly mentions fencing token violations, yet the release path contains one. The correct pattern is a Lua script that makes the check-and-delete atomic.

**Required fix:**

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_lock = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._stop_event.set()
self._release_lock(keys=[lock_key], args=[token])
```

### 3. High — Heartbeat Race After `_stop_heartbeat` / Before Lock Delete

```python
finally:
    self._stop_heartbeat()          # sets flag to False
    current = self.redis.get(lock_key)
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10 s) and checks `_heartbeat_active` **after** waking. There is a window where:

1. `_stop_heartbeat()` sets the flag.
2. The heartbeat thread wakes, reads the flag as `False`, and exits — fine.

But with the current `while self._heartbeat_active: time.sleep(...)` pattern, the flag is checked **before** sleeping, meaning a thread that just started its sleep will run one more `expire` call after the lock has already been deleted and potentially re-acquired by another owner. Using `threading.Event.wait(timeout=...)` (shown above) eliminates this window because the event can be set mid-sleep.

### 4. High — `db.update_stock` and `db.create_order` Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock does not protect against this; it only serializes access. This requires either a database transaction wrapping both calls, or a compensating rollback on failure.

**Required fix:**

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.exception("DB transaction failed item_id=%s order_id=%s", item_id, order_id)
    raise
```

### 5. Medium — Input Validation Is Claimed but Not Present

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` with no validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will silently decrement stock by zero or go negative.

**Required fix:**

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

Validate at the entry point of `reserve_item` and return a structured error on `ValidationError`.

---

### 6. Medium — `time.sleep` in Lock Retry Blocks the Calling Thread

```python
time.sleep(backoff)
```

In an async or thread-pool web server, sleeping the calling thread during retry holds a worker for up to `(2^2) * 0.1 = 0.4 s`. Under flash-sale load this will exhaust the thread pool. The retry logic should either be async-aware or the caller should be told to retry (return a 503 with `Retry-After`) rather than blocking.

---

### 7. Low — `_heartbeat_thread.join()` Is Never Called

The daemon thread will be abandoned silently. While daemon threads do not block process exit, not joining means there is no way to confirm the thread has actually stopped before the lock is released. Add a bounded `join` after setting the stop event.

```python
stop_event.set()
if self._heartbeat_thread:
    self._heartbeat_thread.join(timeout=2)
```

---

### 8. Low — Tests Mock Away the Behavior Under Review

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Every test fixture replaces the heartbeat with a no-op. The concurrent correctness properties — the primary justification for this PR's complexity — are completely untested. At minimum, add an integration-style test that runs `reserve_item` from two threads simultaneously against a real or fakeredis backend and asserts exactly one succeeds.
