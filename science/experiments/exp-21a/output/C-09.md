Decision: Request Changes

Summary: The implementation demonstrates solid fundamentals — atomic lock acquisition with `NX+EX`, token-verified release, heartbeat ownership check, and structured logging are all correct in principle. However, there are several architectural and correctness flaws that must be addressed before this can be merged into a system handling government-adjacent inventory operations where race conditions or silent failures carry real consequences.

---

## Critical Issues

### 1. Heartbeat Race Condition on Lock Release (TOCTOU)

The lock release in `_acquire_lock` and the heartbeat's renewal loop both perform a `GET` followed by a conditional `EXPIRE` or `DELETE`. These are not atomic, meaning a lock can expire between the `GET` and the `DELETE`/`EXPIRE`, allowing another process to acquire it and then have it deleted out from under them.

**Fix:** Use a Lua script for both operations to make them atomic.

```python
# Atomic release
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Atomic renewal
RENEW_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""

# Register on __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)
self._renew_script = self.redis.register_script(RENEW_SCRIPT)
```

---

### 2. Instance-Level Heartbeat State Breaks Concurrency

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If `InventoryService` is shared across threads (which it will be in any real WSGI/ASGI deployment), concurrent calls to `reserve_item` will corrupt each other's heartbeat state — one call stopping another's heartbeat mid-operation.

**Fix:** Make heartbeat state local to each lock acquisition context.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False
    # ... acquire logic ...

    heartbeat_thread = threading.Thread(
        target=self._heartbeat_loop,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_script(keys=[lock_key], args=[token])
```

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw `str` and `int` with no validation. A negative `quantity`, empty `item_id`, or a crafted `item_id` containing Redis key injection characters (e.g., `*`, spaces, or glob patterns) will pass through silently.

**Fix:** Validate at the service boundary.

```python
import re

_SAFE_ITEM_ID = re.compile(r'^[a-zA-Z0-9_\-]{1,64}$')

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not isinstance(item_id, str) or not _SAFE_ITEM_ID.match(item_id):
        return ReservationResult(success=False, message="Invalid item ID")
    if not isinstance(quantity, int) or quantity < 1 or quantity > 1000:
        return ReservationResult(success=False, message="Invalid quantity")
    # ...
```

### 4. DB Operations Outside Transaction Boundary

`update_stock` and `create_order` are two separate DB calls with no transaction wrapping. If `create_order` fails after `update_stock` succeeds, stock is decremented but no order exists — inventory is silently lost. This is a data integrity bug.

**Fix:** Wrap both in a single atomic transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. If the current `db` abstraction doesn't support transactions, that needs to be resolved before this service ships.

## Moderate Issues

### 5. Heartbeat Thread Is Never Joined

`_stop_heartbeat` sets a flag but never calls `join()` on the thread. The thread may still be running when the lock is deleted, and on a slow system it could attempt to renew a lock that has already been released and re-acquired by another process.

**Fix:** Join the thread in the `finally` block as shown in issue #2 above.

---

### 6. Silent Swallow of Non-RuntimeError Exceptions

Only `RuntimeError` is caught in `reserve_item`. Any other exception — Redis connection error, DB timeout, unexpected `TypeError` — propagates uncaught to the caller with no structured response. Given this is a service method returning a `ReservationResult`, callers expect that contract to hold.

**Fix:**

```python
except Exception as e:
    logger.exception("Unexpected error during reservation item_id=%s", item_id)
    return ReservationResult(
        success=False,
        message="Service temporarily unavailable. Please try again."
    )
```

Log with `exception` (not `error`) so the stack trace is captured.

---

### 7. Retry Backoff Blocks the Calling Thread

The exponential backoff uses `time.sleep()` on the calling thread. Under flash sale load, this means threads pile up waiting, exhausting the thread pool. Maximum wait is `0.1 + 0.2 + 0.4 = 0.7s` per request, which compounds badly at scale.

**Recommendation:** Either use async-compatible backoff, or document clearly that this service must not be called from a thread pool without a timeout budget enforced at the caller level. Consider adding a `max_wait` parameter.

## Test Coverage Gaps

### 8. Heartbeat Is Fully Mocked Out in All Tests

Every test fixture patches `_start_heartbeat` and `_stop_heartbeat` to no-ops. This means the heartbeat logic — the most novel and failure-prone part of this PR — has zero test coverage. The TOCTOU race in issue #1 would not be caught by these tests.

**Required additions:**
- Test that heartbeat calls `expire` with the correct token before TTL elapses
- Test that heartbeat does NOT extend a lock it no longer owns (token mismatch)
- Test that the heartbeat thread terminates cleanly after lock release

### 9. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test mocks `redis.get` to return `b"different-token"` globally, but the fixture also mocks `_start_heartbeat` and `_stop_heartbeat`. The test does not exercise the actual lock release path under realistic conditions and will pass even if the release logic is completely broken.

---

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is a reasonable ratio, but both should be configurable via constructor parameters rather than class-level constants, to support different environments (test vs. production).
- The `db` interface has no type annotation or abstract base class. Define an `AbstractInventoryDB` protocol so the service's dependencies are explicit and mockable with confidence.
- Log line in `reserve_item` logs `item_id` directly from user input. Ensure the logging pipeline sanitizes this before it reaches any log aggregation system.
