Decision: Request Changes

Summary: The implementation has a critical race condition in the heartbeat mechanism — `_heartbeat_active`, `_heartbeat_thread`, and the lock token are stored as instance-level attributes rather than being scoped to each lock acquisition, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will corrupt each other's heartbeat state and potentially extend or drop the wrong lock. Beyond that architectural flaw, the lock release in the `finally` block contains a non-atomic check-then-delete sequence that reintroduces the fencing token violation the PR claims to defend against, and the DB operations lack transactional guarantees that could leave stock and order records inconsistent on partial failure.

---

### Detailed Findings

#### 1. Critical — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are shared across all concurrent calls on the same instance. If two threads call `reserve_item` simultaneously:

- Thread A sets `_heartbeat_active = True` and starts its thread.
- Thread B sets `_heartbeat_active = True` and starts a *second* thread, orphaning Thread A's reference.
- When Thread B calls `_stop_heartbeat()`, it sets `_heartbeat_active = False`, killing *both* heartbeat threads.
- Thread A's lock now goes unrenewed and expires while Thread A's DB operation is still in flight — exactly the zombie-write scenario the PR claims to prevent.

**Fix:** Scope heartbeat state to each lock acquisition using a local `threading.Event` and pass the token explicitly into the thread closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    heartbeat_thread = threading.Thread(
        target=self._beat, args=(lock_key, token, stop_event), daemon=True
    )
    heartbeat_thread.start()
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

#### 2. High — Non-Atomic Lock Release Is a TOCTOU Race

```python
# In the finally block
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, another process can acquire the lock. This is a textbook time-of-check/time-of-use (TOCTOU) race. The standard fix is a Lua script that atomically checks ownership and deletes in a single round-trip:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def __init__(self, ...):
    ...
    self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
self._release_script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's `GET` → `expire` sequence and should receive the same treatment.

#### 3. High — No Transactional Boundary Around DB Writes

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order record exists. This is a silent data integrity failure — inventory is consumed with no corresponding order, and there is no compensating transaction or rollback path.

**Fix:** Wrap both operations in a database transaction, or implement a compensating rollback on exception:

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception:
    logger.error("DB write failed, reservation rolled back item_id=%s", item_id)
    raise
```

#### 4. Medium — Input Validation Is Claimed But Not Implemented

The checklist marks "Input validation via Pydantic" but there is no validation on `item_id` or `quantity` at the `reserve_item` boundary. Pydantic is imported and used for the *response* model only. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will silently proceed.

**Fix:** Add a request model or explicit guards:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[\w\-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

#### 5. Medium — Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to finish its current `sleep`. The thread may still be alive and attempting a Redis `expire` call after the lock has been released and potentially re-acquired by another process. `thread.join()` must be called after setting the stop signal.

---

#### 6. Low — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests never exercise the actual lock acquisition and release path through `_acquire_lock`. The `test_lock_not_released_if_expired` test in particular asserts behavior that is bypassed by the mock. Integration-level tests using a real or embedded Redis instance (e.g., `fakeredis`) are needed to validate the locking contract.

| # | Severity | Issue |
|---|----------|-------|

| 2 | High | Lock release must use atomic Lua script |
| 3 | High | DB writes must be wrapped in a transaction with rollback |
