Decision: Request Changes

Summary: The implementation contains several architectural flaws that make it unsafe for a high-concurrency flash sale environment. The most critical issue is that `InventoryService` stores heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`), meaning concurrent calls to `reserve_item` on the same service instance will race against each other's heartbeat lifecycle — one call's `_stop_heartbeat()` will silence another call's renewal thread, potentially allowing a live lock to expire mid-write. Additionally, the heartbeat's check-then-extend sequence (`GET` followed by `EXPIRE`) is a non-atomic TOCTOU race; another process could steal the lock between those two operations, and the heartbeat would still extend it, defeating the ownership guarantee the PR claims to provide. The test suite mocks out both `_start_heartbeat` and `_stop_heartbeat` universally in the shared `service` fixture, meaning the concurrency and heartbeat behaviors are never actually exercised by any test.

---

### Detailed Findings

#### Critical — Shared Mutable Heartbeat State (Instance-Level Concurrency Hazard)

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` on the same `InventoryService` instance simultaneously:

- Thread A sets `_heartbeat_active = True`, starts its beat thread.
- Thread B enters `_acquire_lock`, sets `_heartbeat_active = True` again (harmless so far).
- Thread A finishes, calls `_stop_heartbeat()` → sets `_heartbeat_active = False`.
- Thread B's heartbeat thread now exits silently. Thread B's lock will expire unrenewed during a slow DB write.

This is a silent data-integrity failure under the exact load profile this service is designed for. Each lock acquisition must carry its own isolated stop signal. The fix is to scope heartbeat state to the lock context, not the instance.

```python
# Suggested direction — pass a threading.Event per acquisition
def _start_heartbeat(self, lock_key: str, token: str, stop_event: threading.Event) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            # atomic extend below
            ...
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

#### Critical — Non-Atomic Heartbeat Extend (TOCTOU Race)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is precisely the fencing token violation the PR claims to guard against. The extend must be atomic. Use a Lua script:

```lua
-- extend_if_owner.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('EXPIRE', KEYS[1], ARGV[2])
else
    return 0
end
"""

extended = self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
if not extended:
    logger.warning("Heartbeat: lock no longer owned, stopping. key=%s", lock_key)
    stop_event.set()
```

The same atomic pattern should be applied to lock release (the PR's `GET` + `DELETE` pair carries the identical race).

#### High — DB Operations Not Atomic; Partial-Write Risk

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction boundary. If `create_order` raises, stock has already been decremented and no order exists — inventory is silently lost. This must be wrapped in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

#### High — Input Validation Not Actually Enforced

The checklist claims input validation via Pydantic, but `reserve_item` accepts raw primitives with no validation:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no guard against `quantity <= 0`, negative stock manipulation, or an empty/malicious `item_id` being interpolated into the lock key. A caller passing `quantity=-5` would increase stock under the lock. At minimum:

```python
if not item_id or not isinstance(item_id, str):
    raise ValueError("item_id must be a non-empty string")
if quantity <= 0:
    raise ValueError("quantity must be a positive integer")
```

Consider a Pydantic input model for the method signature if the checklist claim is to be accurate.

#### Medium — Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()  # ← heartbeat never runs
    svc._stop_heartbeat = MagicMock()   # ← stop never runs
    return svc
```

Every test using the `service` fixture silently bypasses all heartbeat and lock-lifecycle logic. The tests confirm the happy-path DB logic but provide no coverage for the locking mechanism itself. `test_lock_not_released_if_expired` is particularly misleading — it asserts on `delete` not being called, but the lock release path it claims to test is also bypassed by the mocked stop. Add integration-level tests that allow the real `_acquire_lock` context manager to execute.

---

#### Low — `time.sleep` in Heartbeat Thread Prevents Clean Shutdown

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)
```

A daemon thread sleeping for 10 seconds will not respond to a stop signal until the sleep completes. Under normal operation this is a 10-second lag before the thread exits. Use `threading.Event.wait(timeout=...)` instead, which wakes immediately on set.

---

#### Low — Logger Not Configured at Module Level

```python
logger = logging.getLogger(__name__)
```

This is fine as a pattern, but there is no `NullHandler` added, which will produce "No handlers could be found" warnings in environments that have not configured logging. Add `logging.getLogger(__name__).addHandler(logging.NullHandler())` per library best practice.
