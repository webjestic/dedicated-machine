Decision: Request Changes

Summary: The implementation contains several critical architectural and correctness flaws that make it unsafe for a high-concurrency flash sale environment. The most severe issue is a race condition in the heartbeat mechanism combined with a shared-state threading model that makes the service non-reentrant and unsafe for concurrent use. Additional concerns around atomicity, input validation gaps, and test fidelity need to be resolved before this can be approved.

---

## Critical Issues

### 1. Non-Reentrant Service — Shared Heartbeat State

This is the most dangerous bug in the PR. `_heartbeat_active` and `_heartbeat_thread` are instance-level attributes, not scoped to individual lock acquisitions.

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two calls to `reserve_item` overlap on the same `InventoryService` instance (which is the normal case in any web framework with a shared service layer), `_stop_heartbeat()` from the first call will kill the heartbeat for the second call mid-flight. The service is documented as solving zombie-write failure modes, but this bug *introduces* one.

**Fix:** Scope heartbeat state to each lock acquisition, not to the instance. Pass a `threading.Event` into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    heartbeat = threading.Thread(target=_beat, daemon=True)
    # ... acquire lock, start thread, yield, stop_event.set(), join thread
```

### 2. Non-Atomic Lock Release — TOCTOU Race

The lock release is a check-then-act sequence that is not atomic:

```python
# Two operations — not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and `DELETE`, another process could acquire the lock after expiry. You then delete a lock you no longer own, which is exactly the fencing token violation the PR claims to prevent. This must be a Lua script to be atomic.

```lua
-- release.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register on __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat's `GET` + `EXPIRE` sequence and needs the same treatment.

### 3. No Input Validation on `item_id` or `quantity`

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw strings and integers with zero validation. There is no Pydantic model on the input path.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller can pass `quantity=0`, `quantity=-5`, an empty string, or a path-traversal-style string as `item_id`. The lock key becomes `inventory:lock:` with an empty suffix, or worse, `inventory:lock:../admin`. For a government contracting context, this is unacceptable.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

---

### 4. DB Operations Are Not Atomic

`update_stock` and `create_order` are two separate DB calls with no transaction boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process dies between these two calls, stock is decremented but no order exists. The Redis lock provides no protection here — this requires a DB-level transaction. The interface needs to expose a transactional operation or these must be wrapped in one.

## Significant Issues

### 5. Heartbeat Uses `time.sleep` — Not Interruptible

```python
while self._heartbeat_active:
    time.sleep(self.HEARTBEAT_INTERVAL)  # blocks for 10 seconds, ignores stop signal
```

When `_stop_heartbeat()` is called, the thread can continue sleeping for up to `HEARTBEAT_INTERVAL` seconds before it checks the flag. With `threading.Event.wait(timeout=...)`, the thread wakes immediately on signal. This is a correctness and resource issue.

### 6. Heartbeat Interval vs. TTL Ratio Is Unsafe

`HEARTBEAT_INTERVAL = 10`, `LOCK_TTL = 30`. The heartbeat fires at 10s, but the first beat doesn't fire until 10 seconds have elapsed. If a slow DB operation starts at t=0 and the lock was acquired at t=0, the lock expires at t=30. The heartbeat fires at t=10, t=20 — fine in theory. But with `time.sleep` not being interruptible and subject to OS scheduling jitter, this margin is tighter than it appears. A ratio of at least 3:1 with a reliable interrupt mechanism is the minimum bar.

### 7. Exception Handling Swallows All Failures Silently

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` propagates uncaught and will surface as a 500 with no structured response. This needs a broader `except Exception` with logging and a safe return value, distinct from the lock contention path.

## Test Fidelity Issues

### 8. Tests Mock Away the Exact Code Under Test

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat — which is the primary novel mechanism in this PR — is mocked out in every test. No test exercises the actual threading behavior, the `time.sleep` loop, the token comparison, or the TTL extension call. These tests prove the business logic works in isolation, which has value, but they provide zero coverage for the distributed correctness claims made in the PR description.

Add at minimum:
- A test that verifies `redis.expire` is called by the heartbeat thread with the correct key and token.
- A test that verifies the heartbeat stops after `_stop_heartbeat` is called.
- A test that verifies the heartbeat does *not* extend a lock it no longer owns.

### 9. `test_lock_not_released_if_expired` Does Not Test What It Claims

```python
def test_lock_not_released_if_expired(service, redis_client):
    redis_client.get.return_value = b"different-token"
    result = service.reserve_item("item_001")
    redis_client.delete.assert_not_called()
```

This test asserts `delete` is never called, but `reserve_item` returns a `ReservationResult` — meaning the reservation *succeeded* while the lock was "stolen." That is the exact failure mode this code is supposed to prevent. The test is asserting on the wrong thing and masking a logic error.

## Minor Issues

- `logger = logging.getLogger(__name__)` at module level is correct, but there is no guidance on log level configuration. In a government context, `DEBUG` level logs containing lock keys and token values should be explicitly gated.
- The `LOCK_TTL` and `HEARTBEAT_INTERVAL` as class-level constants make them difficult to tune per-deployment without subclassing. These should be constructor parameters with documented defaults.
- `uuid4` for `order_id` generation inside the reservation service conflates concerns. Order ID generation should be delegated to the order/DB layer.
