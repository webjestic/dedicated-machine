Decision: Request Changes

Summary: The implementation has a critical race condition in the heartbeat mechanism — `_heartbeat_active`, `_heartbeat_thread`, and the check/extend sequence in `_beat()` are shared mutable state with no synchronization, meaning concurrent calls to `reserve_item()` on the same `InventoryService` instance will corrupt each other's heartbeat lifecycle. Beyond the race condition, the lock release in the `finally` block has a non-atomic check-then-delete that can cause a stolen-lock deletion, and the DB operations lack transactional guarantees, meaning a crash between `update_stock` and `create_order` will silently corrupt inventory. The test suite mocks out the heartbeat entirely, so none of these failure modes are exercised.

---

### Detailed Findings

#### 🔴 Critical — Shared Heartbeat State (Race Condition)

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item()` concurrently on the **same** `InventoryService` instance, the second call will overwrite `_heartbeat_active` and `_heartbeat_thread` before the first call's lock is released.

```python
# Thread A sets this to True, starts its heartbeat
self._heartbeat_active = True

# Thread B immediately overwrites it — Thread A's heartbeat is now uncontrolled
self._heartbeat_active = True
```

When Thread A calls `_stop_heartbeat()`, it sets `_heartbeat_active = False`, which also kills Thread B's heartbeat. Thread B's lock will then expire undefended. This is a zombie-write failure mode — exactly what the PR claims to prevent.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Use a local `threading.Event` per lock context.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = threading.Thread(
        target=self._beat, args=(lock_key, token, stop_event), daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release below
```

#### 🔴 Critical — Non-Atomic Lock Release (TOCTOU)

The `finally` block performs a check-then-delete as two separate Redis commands. Between the `GET` and `DELETE`, the lock can expire and be acquired by another process, which you then delete.

```python
# NOT atomic — another process can acquire between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

This is a well-known failure mode. The canonical fix is a Lua script, which Redis executes atomically.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally:
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU exists in the heartbeat's `_beat()` function — the `GET` then `EXPIRE` should also be a Lua script or replaced with `SET ... XX EX` to extend only if owned.

#### 🔴 Critical — No DB Transaction Boundary

`update_stock` and `create_order` are two separate DB calls with no transaction wrapping them. A process crash, network partition, or exception between them leaves the database in an inconsistent state: stock is decremented but no order exists, or vice versa.

```python
# If this succeeds but the next line raises, stock is gone with no order record
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # crash here = phantom decrement
```

**Fix:** Wrap in a transaction. The exact API depends on your DB layer, but the intent must be explicit.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support transactions, that is an architectural gap that must be resolved before this merges.

#### 🟡 Medium — Input Validation Is Claimed But Not Present

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` parameters with no validation. `item_id` is passed directly into a Redis key (`inventory:lock:{item_id}`) and into DB queries without sanitization. A caller passing `item_id = "foo:bar"` or `quantity = -5` or `quantity = 0` will produce silent misbehavior.

```python
# quantity = 0 passes the stock check and creates a zero-quantity order
# quantity = -5 increases stock — a critical inventory integrity bug
if current_stock < quantity:  # -5 < 10 is False, so this passes
```

**Fix:** Validate at the entry point.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        return ReservationResult(success=False, message=f"Invalid input: {e}")
    # proceed with req.item_id, req.quantity
```

---

#### 🟡 Medium — Test Suite Does Not Test What It Claims

The heartbeat is mocked out in every test via `svc._start_heartbeat = MagicMock()`. This means the tests provide zero coverage of the heartbeat logic, the race condition above, or the TOCTOU in lock release. `test_lock_not_released_if_expired` asserts `delete` is not called, but because the heartbeat is mocked, the token comparison logic is the only thing being tested — and even that path is not realistic because the mock bypasses the actual `_acquire_lock` flow.

Tests must exercise the real heartbeat path, including verifying that concurrent calls on the same instance do not corrupt each other's lock state.

---

#### 🟢 Minor — Heartbeat Thread Is Not Joined on Stop

`_stop_heartbeat()` sets the flag but does not join the thread. The daemon thread will sleep for up to `HEARTBEAT_INTERVAL` (10 seconds) after the lock is released before it exits. During that window it could still attempt a `GET`/`EXPIRE` on a key now owned by another process. The `join(timeout=...)` in the fix above addresses this.
