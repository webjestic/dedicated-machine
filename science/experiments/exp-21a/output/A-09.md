Decision: Request Changes

Summary: The implementation contains several architectural flaws that make it unsafe for production use under concurrent load. The most critical issue is that `_heartbeat_active`, `_heartbeat_thread`, and the heartbeat control methods are stored as instance-level state on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields — one call's `_stop_heartbeat()` will silently kill another call's heartbeat, or two heartbeats will stomp each other's `_heartbeat_active` flag. Additionally, the check-get-expire sequence in the heartbeat thread is non-atomic, creating a TOCTOU window where the lock can expire between the `GET` and `EXPIRE` calls, defeating the entire purpose of the renewal. There are also missing input validation guards, a non-atomic DB write pair, and test coverage gaps that mask real failure modes.

## Detailed Findings

### 🔴 Critical — Shared Mutable Heartbeat State (Race Condition)

**File:** `services/inventory.py`, `InventoryService.__init__`

```python
# Current — instance-level state shared across all concurrent calls
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly instantiated once and reused (standard DI pattern). Every concurrent call to `reserve_item` overwrites `_heartbeat_active` and `_heartbeat_thread` on the shared instance. Call A starts its heartbeat, Call B starts its own, then Call B's `_stop_heartbeat()` sets `_heartbeat_active = False`, killing Call A's renewal loop while Call A's DB write is still in flight. This is a textbook instance-state concurrency bug.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance. Pass a mutable container or use a `threading.Event` per call:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff,
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # Atomic release via Lua script (see below)
        self._release_lock(lock_key, token)
```

### 🔴 Critical — Non-Atomic Heartbeat Renewal (TOCTOU)

**File:** `services/inventory.py`, `_start_heartbeat._beat`

```python
# Non-atomic: lock can expire between GET and EXPIRE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

Between the `GET` and the `EXPIRE`, the lock TTL can elapse and another process can acquire it. You then extend a lock you no longer own, which is exactly the fencing token violation the PR claims to prevent. This must be a single atomic operation via a Lua script:

```python
HEARTBEAT_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _start_heartbeat(
    self,
    lock_key: str,
    token: str,
    stop_event: threading.Event,
) -> threading.Thread:
    script = self.redis.register_script(HEARTBEAT_SCRIPT)

    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.warning(
                    "Heartbeat lost lock ownership key=%s — stopping renewal",
                    lock_key,
                )
                return
            logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    return thread
```

### 🔴 Critical — Non-Atomic Lock Release (TOCTOU, Again)

**File:** `services/inventory.py`, `_acquire_lock` finally block

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Same class of bug as the heartbeat. Between `GET` and `DELETE`, the lock can expire and be acquired by another process, which you then delete. This is the canonical mistake in DIY Redis locking. Use the standard Lua script:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock release skipped — no longer owner key=%s", lock_key
        )
```

### 🟠 High — No Input Validation on `reserve_item`

**File:** `services/inventory.py`, `reserve_item`

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

The checklist claims "Input validation via Pydantic" but there is none at the entry point. `item_id` is interpolated directly into a Redis key with no sanitization. A caller passing `item_id = "foo:bar:../../admin"` or an empty string produces a malformed or unintended lock key. `quantity` has no lower bound check — a caller passing `quantity = 0` or `quantity = -5` would pass the stock check and write a corrupted stock value.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_\-]+$")
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValueError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # use req.item_id and req.quantity from here
```

### 🟠 High — Non-Atomic DB Write Pair Without Rollback

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises, stock is decremented but no order exists. The lock is held so no other process can correct this. There is no transaction boundary, no rollback, and no compensating action. This needs to be wrapped in a DB transaction or at minimum a try/except that attempts to restore stock on failure:

```python
try:
    with self.db.transaction():
        self.db.update_stock(req.item_id, new_stock)
        self.db.create_order(order_id, req.item_id, req.quantity)
except Exception as e:
    logger.error(
        "DB write failed item_id=%s order_id=%s: %s",
        req.item_id, order_id, e,
    )
    return ReservationResult(
        success=False,
        message="Reservation failed due to a database error.",
    )
```

### 🟡 Medium — Heartbeat Thread Leaks on `_start_heartbeat` Exception

**File:** `services/inventory.py`, `_acquire_lock`

If `_start_heartbeat` raises after the lock is acquired (e.g., Redis connection error during script registration), the `finally` block runs `_stop_heartbeat()` but the lock is never released because the `yield` was never reached. The lock will sit until TTL expiry. With the refactored approach using `stop_event`, this is naturally resolved because the thread reference is local and the finally block unconditionally calls `_release_lock`.

---

### 🟡 Medium — `time.sleep` in Retry Loop Blocks the Calling Thread

**File:** `services/inventory.py`, `_acquire_lock`

```python
time.sleep(backoff)
```

Under a flash sale load spike, hundreds of threads will pile up here sleeping. This is acceptable only if the service is explicitly designed as a thread-per-request model with bounded concurrency. If this runs in an async framework (FastAPI, aiohttp) or a thread pool with limited workers, this will cause starvation. The PR gives no indication of the deployment context. At minimum this should be documented; ideally the lock acquisition should be async-compatible or use a queue-based backpressure mechanism.
