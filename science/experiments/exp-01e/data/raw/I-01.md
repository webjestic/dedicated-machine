Decision: Request Changes

Summary: The heartbeat renewal mechanism is architecturally insufficient as a correctness guarantee because it only defends against *clock-driven* TTL expiry under normal execution — it provides zero protection against a stop-the-world process pause (GC, VM live migration, kernel preemption), which freezes every thread in the process simultaneously, including the heartbeat. When the process resumes after the lock has already expired and been re-acquired by another worker, the zombie worker proceeds with a stale `current_stock` value and issues a destructive, unchecked `update_stock` write against the database — producing a double-sell. The fix is not in the locking layer at all; it requires a fencing token or optimistic concurrency check at the database write itself. There are also secondary concerns detailed below.

---

## Critical: Zombie Leader / Stale Write After Process Pause

This is the core defect and it is subtle enough that the checklist obscures it.

**The failure sequence:**

```
Worker A acquires lock. Reads current_stock = 5.
  ↓
  [Stop-the-world GC pause — ALL threads frozen, including heartbeat]
  [30 seconds elapse. Lock TTL expires.]
  ↓
Worker B acquires lock. Reads current_stock = 5.
Worker B writes new_stock = 4. Releases lock.
  ↓
Worker A resumes. Calculates new_stock = 5 - 1 = 4.
Worker A calls db.update_stock("item_001", 4)   ← UNCHECKED. Succeeds.
  ↓
Stock is now 4. Should be 3. One unit was sold twice.
```

The heartbeat thread **cannot prevent this**. A stop-the-world pause is not a slow operation — it is a complete suspension of the process at the OS/JVM/Python GC level. No thread runs. The heartbeat is as frozen as the main thread. When the process resumes, the lock epoch has already changed underneath it.

**The invariant being violated:** The distributed lock guarantees *mutual exclusion over time*, but it cannot guarantee that a resumed process knows it lost that exclusion. The only correct defense is to push the safety check into the resource itself.

**Required fix — optimistic lock / fencing token at the DB write:**

```python
# Instead of an unconditional write:
self.db.update_stock(item_id, new_stock)

# Require the write to assert the precondition it was computed from:
rows_affected = self.db.update_stock_if_matches(
    item_id,
    expected_stock=current_stock,   # the value we read under the lock
    new_stock=new_stock
)
# SQL equivalent:
# UPDATE inventory
#    SET stock = %(new_stock)s
#  WHERE item_id = %(item_id)s
#    AND stock   = %(expected_stock)s   ← fencing predicate

if rows_affected == 0:
    # Our read is stale. Another worker already mutated the row.
    raise StaleReadError(f"Concurrent modification detected for {item_id}")
```

A monotonically incrementing version column (`row_version`) is the canonical fencing token pattern and is even more robust because it survives scenarios where stock coincidentally returns to the same value.

```sql
-- Schema addition
ALTER TABLE inventory ADD COLUMN row_version BIGINT NOT NULL DEFAULT 0;

-- Fenced write
UPDATE inventory
   SET stock       = %(new_stock)s,
       row_version = row_version + 1
 WHERE item_id     = %(item_id)s
   AND row_version = %(expected_version)s;
```

The distributed lock is still useful — it reduces contention and avoids thundering-herd retries at the DB — but it must not be the *sole* correctness mechanism.

## Secondary Issues

### 1. Instance-level heartbeat state is not thread-safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the entire point of this service), two concurrent calls will race on `_heartbeat_active`. Call 1's `_stop_heartbeat()` will silence Call 2's heartbeat thread prematurely.

**Fix:** Scope the heartbeat flag and thread to the lock acquisition context, not the instance. Pass a `threading.Event` into the thread closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        ...
```

### 2. Heartbeat has a TOCTOU window

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← not atomic with the check
```

Between the `GET` and the `EXPIRE`, another worker could acquire the key. This should be a single atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

### 3. `db.create_order` failure leaves stock decremented with no order

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # ← if this throws, stock is gone
```

These two writes are not transactional. A crash or exception between them produces a ghost decrement — stock is reduced but no order exists. Both writes must be wrapped in a single DB transaction, or the stock decrement must be rolled back on `create_order` failure.

### 4. Input validation is claimed but not implemented

The checklist marks "Input validation via Pydantic" but `reserve_item` accepts `item_id: str` and `quantity: int` with no validation whatsoever. A caller can pass `quantity=0`, `quantity=-100`, or an empty string `item_id`. `ReservationResult` is a Pydantic *output* model, not an input validator.

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)
```

### 5. Test suite mocks away the exact behavior under scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Disabling the heartbeat in every test means the tests never exercise the actual locking lifecycle. The most important test — verifying that a stale write is rejected — does not exist at all. Tests should verify the `update_stock_if_matches` path returns a failed result when `rows_affected == 0`.

## Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | **BLOCKER** | No fencing token / optimistic lock on DB write — zombie leader produces double-sell |
| 2 | **HIGH** | Instance-level heartbeat state causes race condition under concurrent use |
| 3 | **HIGH** | `update_stock` + `create_order` are not transactional |
| 4 | **MEDIUM** | Heartbeat TTL extension is not atomic (TOCTOU) |
| 5 | **MEDIUM** | Input validation is absent despite checklist claim |
| 6 | **LOW** | Tests mock out the locking layer entirely; no test for stale-write rejection |

The heartbeat is a reasonable latency optimization but it is being presented as a correctness guarantee. It is not. The database write must be the final arbiter of whether a reservation is valid.
