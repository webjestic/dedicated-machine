# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains several critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The heartbeat mechanism introduces a false sense of safety while masking the fundamental unsoundness of the locking model, and the non-atomic DB operations create a corruption window that no amount of Redis locking can close. None of these failure modes are exercised by the test suite, which is structured in a way that actively conceals them.

---

## Critical Issues (Blockers)

### 1. Instance-Level Heartbeat State — Concurrent Calls Will Corrupt Each Other

This is the most dangerous flaw in the codebase. The heartbeat state is stored on the *instance*, not scoped to the individual lock acquisition.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

```python
def _start_heartbeat(self, lock_key: str, token: str) -> None:
    self._heartbeat_active = True   # ← shared mutable state on self
    ...

def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False  # ← stomps on every in-flight heartbeat
```

**The failure scenario:**

1. Request A acquires `inventory:lock:item_001`, starts heartbeat, sets `self._heartbeat_active = True`.
2. Request B (different item, different thread) acquires `inventory:lock:item_002`, calls `_start_heartbeat`, which is fine — but then Request A's `finally` block calls `_stop_heartbeat()`, setting `self._heartbeat_active = False`.
3. Request B's heartbeat thread now silently stops. If Request B's DB operations take longer than `LOCK_TTL`, the lock expires.
4. Request C sees no lock, acquires it, reads stale stock, and a double-reservation occurs.

In a flash sale, where the whole point is high concurrency against a *shared service instance*, this is not a theoretical edge case — it is the common case. The PR's own stated goal ("prevent lock expiry under load") is self-defeating under the exact load conditions it targets.

**Fix:** Heartbeat state must be scoped to the lock, not the instance. Use a local `threading.Event` per acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = threading.Thread(
        target=self._heartbeat_loop,
        args=(lock_key, token, stop_event),
        daemon=True
    )
    heartbeat_thread.start()

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua script (see Issue #2)
```

### 2. Non-Atomic Lock Release — TOCTOU Race Condition

The lock release is a check-then-act sequence across two separate Redis round-trips with no atomicity guarantee:

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # round-trip 1
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # round-trip 2
```

**The failure scenario:**

1. The GC or OS scheduler causes a Stop-the-World pause between the `GET` and `DELETE`.
2. During that pause, the lock TTL expires (perhaps the heartbeat was also paused).
3. Another process acquires the lock with a new token.
4. The paused process resumes, the `if` condition was already evaluated as `True`, and it executes `DELETE`, releasing a lock it no longer owns.
5. A third process immediately acquires the lock. Now two processes hold the lock simultaneously.

This is a well-documented failure mode. The canonical fix is a Lua script, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(self.RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

### 3. Non-Atomic Database Operations — Inventory Corruption on Partial Failure

Within the lock boundary, two separate DB writes are performed with no transactional guarantee:

```python
self.db.update_stock(item_id, new_stock)   # write 1
self.db.create_order(order_id, item_id, quantity)  # write 2
```

**The failure scenario:**

1. `update_stock` succeeds. Stock is now decremented in the DB.
2. The process crashes, the network partitions, or the DB throws on `create_order`.
3. The lock is released (or expires). Stock is decremented but no order exists.
4. The item is effectively lost — phantom consumption with no fulfillment record.

The inverse is equally dangerous: if the order of operations were reversed, an order could exist with no corresponding stock decrement.

**Fix:** Both operations must execute within a single DB transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is an architectural gap that must be resolved before this service can be considered safe for production.

### 4. Heartbeat Race on Wakeup — The Lock Can Expire Before the First Extension

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)  # ← sleeps FIRST
        if self._heartbeat_active:
            ...
            self.redis.expire(lock_key, self.LOCK_TTL)
```

The thread sleeps for 10 seconds *before* the first extension. The lock TTL is 30 seconds. If the DB operations take between 20 and 30 seconds (which the PR explicitly acknowledges as a possibility by setting a "generous TTL"), the lock will expire before the *second* heartbeat fires. The sequence is: acquire (t=0), sleep (t=0–10), extend to t=40 (t=10), sleep (t=10–20), extend to t=50 (t=20) — this is actually fine in the two-heartbeat case, but consider a slow operation starting at t=25: the next extension fires at t=30, which is exactly when the lock expires. Under any scheduler jitter, this is a race.

More critically, `time.sleep()` is not immune to OS scheduling delays or GC pauses. A 10-second sleep can wake up at 10.5 or 11 seconds in a loaded JVM or CPython process under GC pressure. The safety margin is not as wide as the numbers suggest.

**Fix:** Sleep for `LOCK_TTL / 3` (i.e., 10 seconds for a 30-second TTL), which is what Redisson and other production-grade clients do. Also use `stop_event.wait(timeout=interval)` instead of `time.sleep()` so the heartbeat can be interrupted promptly on lock release rather than sleeping through the cleanup window:

```python
def _heartbeat_loop(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> None:
    interval = self.LOCK_TTL / 3
    while not stop_event.wait(timeout=interval):
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.expire(lock_key, self.LOCK_TTL)
            logger.debug("Lock TTL extended key=%s", lock_key)
        else:
            logger.error(
                "Heartbeat detected lost lock ownership key=%s — "
                "critical: in-flight operation may be unsafe",
                lock_key
            )
            break
```

Note the added logging when ownership is lost mid-operation. The calling code should also be notified of this condition, as any work done after lock expiry is potentially unsafe.

## Significant Issues (Must Address)

### 5. `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no lower-bound check. A caller passing `quantity=0` will succeed, decrement stock by zero, and create a real order record for a zero-unit reservation. A caller passing `quantity=-5` will *increase* stock (`new_stock = current_stock - (-5)`), effectively injecting phantom inventory. This is a business logic vulnerability, not just a bug.

```python
if quantity <= 0:
    return ReservationResult(
        success=False,
        message="Quantity must be a positive integer"
    )
```

Similarly, `item_id` should be validated as a non-empty string before the lock is even attempted.
