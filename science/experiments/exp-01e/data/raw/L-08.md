**Decision: Request Changes**

**Summary:** The heartbeat thread addresses lock expiry under *normal* conditions, but it cannot prevent expiry during a GIL pause, a network partition, or a process stall that occurs *after* the heartbeat's last successful extension and *before* the database writes complete. In that window, another process can acquire the lock and begin its own reservation while this process continues executing `update_stock` and `create_order` against a database that has no knowledge the lock was lost — the critical writes are entirely unguarded. This is the core fencing token problem: the lock protects entry into the critical section, but nothing protects the writes *within* it, so the lock's expiry guarantee is illusory.

## Detailed Review

### The Core Architectural Problem

The PR description says the heartbeat "ensures the lock does not expire during slow DB operations." That claim is only true when everything is working. It is false in the failure modes that matter.

Consider this sequence:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat extends lock (all good)
T=19  Process A calls db.update_stock() — DB call is slow
T=20  Heartbeat thread wakes, tries to extend lock
      → Redis is unreachable (brief network partition)
      → Extension silently fails (no exception raised, no flag set)
T=30  Lock expires
T=31  Process B acquires the same lock
T=31  Process B reads stock: still sees pre-decrement value
T=32  Process A's db.update_stock() completes
T=32  Process A calls db.create_order()
T=33  Process B calls db.update_stock() with the same pre-decrement value
```

You now have two orders created against stock that only existed for one of them. The heartbeat thread did not help because **it has no channel back to the main thread to abort the critical section when renewal fails.**

This is not a theoretical edge case. It is the exact scenario Kleppmann's critique of Redlock describes, and it applies equally to single-node Redis locks. The heartbeat makes the window smaller; it does not close it.

### Specific Code Issues

**1. Heartbeat failure is silent**

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← failure is ignored
```

If `redis.expire` raises or returns `False`, the thread logs nothing and loops again. The main thread has no idea the lock is no longer being renewed. There is no `threading.Event` to signal that the critical section must abort. The write proceeds regardless.

**2. The race between heartbeat check and extend**

```python
current = self.redis.get(lock_key)          # ← lock owned here
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← but maybe not here
```

Between `GET` and `EXPIRE`, another client can acquire the lock. This is a TOCTOU race. The correct implementation is a single atomic Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same race exists in the lock release path.

**3. `_heartbeat_active` is an instance variable, not per-lock**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are shared across all calls to `reserve_item` on the same `InventoryService` instance. If two threads call `reserve_item` concurrently (plausible under flash sale load), one thread's `_stop_heartbeat()` will kill the other thread's heartbeat. This is a straightforward concurrency bug that the tests do not catch because they mock the heartbeat entirely.

**4. The test suite validates the wrong thing**

```python
def test_lock_not_released_if_expired(service, redis_client):
    """Lock release is skipped if token no longer matches (lock expired or stolen)."""
    redis_client.get.return_value = b"different-token"

    result = service.reserve_item("item_001")

    redis_client.delete.assert_not_called()
```

This test confirms the lock is not *released* when expired. It says nothing about whether the *writes* were skipped. In the actual code, `update_stock` and `create_order` both complete before the lock release check happens. The test is asserting correct cleanup behavior while ignoring the data corruption that preceded it.

### What Would Actually Guard the Writes

There are two approaches that are honest about the guarantees they provide:

**Option A: Fencing tokens (correct for storage systems that support them)**

The lock server issues a monotonically increasing token on each acquisition. Every write to the database includes that token. The database rejects writes with a token lower than the highest it has seen. This means a stale lock holder's writes are rejected even if they execute after a newer holder has begun.

```python
# Conceptual — requires DB-side support
fence = acquire_lock_with_fencing_token(lock_key)
db.update_stock(item_id, new_stock, fence=fence)  # DB rejects if fence is stale
```

**Option B: Optimistic locking at the database layer**

Skip the distributed lock for the read. Use a conditional update that fails if the row has changed since it was read. Retry on conflict. This is often simpler and more correct than trying to make a distributed lock behave like a database transaction.

```python
# Pseudocode
stock, version = db.get_stock_with_version(item_id)
if stock < quantity:
    return insufficient()
rows_updated = db.update_stock_if_version(item_id, stock - quantity, expected_version=version)
if rows_updated == 0:
    raise ConflictError("stock changed, retry")
```

Option B is likely the right answer here. If your database supports transactions and conditional updates, the distributed lock is adding complexity without adding safety. The lock only helps if the database cannot enforce the invariant itself — and most databases can.

### Minor Issues

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` means the first heartbeat fires at T+10, leaving a 20-second window at acquisition time where a slow operation could expire the lock before the first renewal. Use millisecond precision (`PX` instead of `EX`) and a tighter ratio.
- No timeout on `db.get_stock`, `db.update_stock`, or `db.create_order`. A hung DB call will hold the lock until TTL expiry regardless of the heartbeat.
- The `InventoryService` instance is presumably shared across request threads. The instance-level heartbeat state makes it unsafe for concurrent use.

### Required Changes Before Approval

1. **Either** implement fencing tokens with database-side enforcement, **or** replace the distributed lock with optimistic locking at the database layer. The current design cannot prevent concurrent writes in the face of lock expiry.
2. Make heartbeat state per-lock-acquisition, not per-instance.
3. Make heartbeat failure observable to the main thread and abort the critical section on renewal failure.
4. Make the heartbeat's token check and TTL extension atomic via Lua script.
5. Add tests that verify write behavior (not just lock release behavior) when the lock has expired mid-operation.
