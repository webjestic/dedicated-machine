# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains several critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The heartbeat mechanism introduces a false sense of safety while masking the fundamental unsoundness of the lock design, and the non-atomic database operations create a split-brain window that no amount of locking can close. None of these failure modes are exercised by the test suite, which is structured to confirm happy-path behavior rather than probe the system's safety boundaries.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Concurrency-Safe

**Severity: Critical — Silent Data Corruption**

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes on a shared `InventoryService` object. In any realistic deployment, a single service instance handles concurrent requests — whether via a thread pool, async framework, or WSGI workers sharing an object.

Consider two concurrent calls:

```
Thread A: reserve_item("item_001") → acquires lock A → starts heartbeat → _heartbeat_active = True
Thread B: reserve_item("item_002") → acquires lock B → starts heartbeat → _heartbeat_active = True (overwrites)
Thread A: finishes → _stop_heartbeat() → _heartbeat_active = False  ← kills Thread B's heartbeat too
```

Thread B's lock now silently expires mid-operation. A third thread acquires the lock for `item_002` while Thread B believes it still holds it. Both threads now read and write stock simultaneously. You have oversold inventory with no exception raised, no log warning, and no test catching it.

The fix requires scoping heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # Pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
    ...
```

### 2. Non-Atomic Database Operations Create an Unrecoverable Inconsistency Window

**Severity: Critical — Inventory Corruption and Phantom Orders**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. The Redis lock protects against *concurrent reads of the same stock value*, but it does not protect against partial writes. If the process crashes, is OOM-killed, or receives a SIGTERM between these two lines:

- `update_stock` committed → stock is decremented
- `create_order` never ran → no order record exists

The item is now gone from inventory with no corresponding order. This is unrecoverable without a manual audit. Conversely, if the order of operations were reversed and `create_order` succeeded but `update_stock` failed, you have an order for stock that was never decremented — an oversell.

The Redis lock provides *mutual exclusion*, not *atomicity*. These are different guarantees. The fix requires wrapping both calls in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 3. The Heartbeat Check-Then-Extend Is a TOCTOU Race

**Severity: High — Lock Theft Goes Undetected**

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)      # ← read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is a classic TOCTOU (Time-of-Check-Time-of-Use) race. The correct implementation uses a Lua script to make the check-and-extend atomic:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
extend_script = self.redis.register_script("""
    if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('EXPIRE', KEYS[1], ARGV[2])
    else
        return 0
    end
""")
result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.error("Lost lock during heartbeat key=%s — ABORTING", lock_key)
    # Signal the main thread to abort the operation
```

The same TOCTOU race exists in the lock release in `_acquire_lock`:

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process could acquire between GET and DELETE
```

This also requires a Lua script or the standard Redlock release pattern.

### 4. A GC Pause or Stop-the-World Event Can Expire the Lock Mid-Operation

**Severity: High — Correctness Guarantee Is Illusory**

The heartbeat thread runs in the same Python process as the main operation thread. A JVM-style stop-the-world GC pause doesn't apply to CPython in the same way, but the equivalent exists: a long GC cycle, a `time.sleep` overshoot under system load, a Python GIL contention spike, or a host-level memory pressure event can pause *all* threads in the process simultaneously, including the heartbeat thread.

If the process is paused for longer than `LOCK_TTL` (30 seconds), the lock expires. When the process resumes, the main thread believes it holds the lock, the heartbeat thread resumes and may even re-extend a lock now owned by another process (see Issue 3), and two processes are now executing the critical section concurrently.

This is not hypothetical. It is the documented failure mode of any single-process heartbeat approach. The PR description calls the 30-second TTL "generous for slow DB operations" — but this framing inverts the risk. A longer TTL increases the blast radius when the process pauses or crashes. The correct mitigation is fencing tokens: the database operation must include a conditional write that rejects the update if a newer lock generation has been issued. Redis's `WAIT` command or a monotonic fence token from a service like ZooKeeper can provide this.

### 5. Heartbeat Thread Leaks on Unhandled Exceptions

**Severity: Medium — Resource Exhaustion Under Failure**

```python
self._start_heartbeat(lock_key, token)
try:
    yield token
finally:
    self._stop_heartbeat()
```

`_stop_heartbeat` only sets `self._heartbeat_active = False`. It does not join the thread. The thread continues executing its current `time.sleep(self.HEARTBEAT_INTERVAL)` — up to 10 more seconds — before checking the flag. Under sustained error conditions (repeated lock failures, DB timeouts), threads accumulate. Each holds a reference to `lock_key` and `token` and continues making Redis calls.

The fix requires joining with a timeout:

```python
def _stop_heartbeat(self, thread: threading.Thread, active_flag: threading.Event) -> None:
    active_flag.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
    if thread.is_alive():
        logger.error("Heartbeat thread did not terminate cleanly")
```

### 6. `quantity` Has No Validation Guard

**Severity: Medium — Negative Quantity Increases Stock**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that `quantity > 0`. A caller passing `quantity=0` passes the stock check (`current_stock < 0` is false), decrements by zero, and creates an order for nothing. A caller passing `quantity=-5` with a stock of 10 passes the check (`10 < -5` is false), sets `new_stock = 15`, and *increases* inventory while creating an order. The Pydantic model is on `ReservationResult`, not on the input. Add a guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

Or enforce it at the call boundary with a validated input model.

## Test Suite Deficiencies

The tests confirm the happy path but are structurally incapable of catching any of the issues above.

| Missing Test | Why It Matters |
|---|---|
| Concurrent `reserve_item` calls on the same instance | Would expose the shared `_heartbeat_active` corruption (Issue 1) |
| `update_stock` raises an exception | Would expose the missing transaction boundary (Issue 2) |
| Lock token changes between heartbeat GET and EXPIRE | Would expose the TOCTOU race (Issue 3) |
| `quantity <= 0` input | Would expose the negative quantity stock inflation (Issue 6) |
| `_heartbeat_thread.is_alive()` after `_stop_heartbeat` | Would expose the thread leak (Issue 5) |

The existing `test_lock_not_released_if_expired` test is well-intentioned but tests the wrong thing. It verifies that `delete` is not called when the token doesn't match — but it does not verify that the *operation was aborted* when the lock was lost. The code currently continues executing `update_stock` and `create_order` even after detecting the token mismatch, because the mismatch is only checked in the `finally` block during release, not during the operation itself.
