Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — storing heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`) makes `InventoryService` unsafe for concurrent use, which is exactly the scenario this service is designed to handle. There are also meaningful atomicity and input validation gaps that need to be addressed before this is production-ready for a high-demand flash sale context.

---

## Detailed Review

### 🔴 Critical — Instance-Level Heartbeat State (Thread Safety)

```python
# Current — breaks under concurrent requests
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If a single `InventoryService` instance handles two concurrent `reserve_item` calls (standard in any web framework with a shared service layer), the second call will overwrite `_heartbeat_active` and `_heartbeat_thread`, corrupting the first call's heartbeat. One lock could go unextended and expire mid-operation while stock is being written.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            if heartbeat_active.is_set():
                current = self.redis.get(lock_key)
                if current and current.decode() == token:
                    self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note also that `_stop_heartbeat` only sets the flag — it never joins the thread. The heartbeat thread can still fire once more after `_stop_heartbeat()` returns but before the lock is deleted, calling `expire` on a key we're about to delete. Harmless in isolation but sloppy, and the `join` above resolves it.

### 🔴 Critical — No Input Validation on Public Entry Point

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with no validation at all.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    # Nothing stops quantity=0, quantity=-5, or item_id=""
```

A caller passing `quantity=0` succeeds silently — stock is unchanged, an order record is created for zero units, and `success=True` is returned. A negative quantity would *increase* stock. This is a data integrity issue.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not isinstance(item_id, str):
        return ReservationResult(success=False, message="Invalid item_id")
    if not isinstance(quantity, int) or quantity < 1:
        return ReservationResult(success=False, message="Quantity must be a positive integer")
```

### 🟠 High — DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls with no transaction wrapping. If `create_order` throws, stock has already been decremented and there is no compensating rollback. The distributed lock protects against concurrent reads, but it does not protect against a partial write leaving the system in an inconsistent state.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on your DB layer, but the intent needs to be there. If the DB abstraction doesn't support transactions, that's a separate issue that needs to be surfaced before this merges.

### 🟠 High — Heartbeat Race on Lock Expiry Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

This is a check-then-act race. The lock can expire between `GET` and `EXPIRE`, allowing another process to acquire it — then we extend *their* lock. This should be an atomic Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

The same pattern applies to the release logic in `_acquire_lock`'s `finally` block.

---

### 🟡 Medium — Tests Mock Away the Thing Being Tested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat is reasonable for unit tests focused on reservation logic, but there are **zero tests** for the heartbeat and lock lifecycle themselves. Given that the heartbeat is the primary novel mechanism here, it needs its own test coverage — verify the thread starts, verify it extends TTL, verify it stops cleanly on context exit.

---

### 🟡 Medium — `time.sleep` in Retry Loop Blocks the Caller Thread

```python
time.sleep(backoff)  # up to 0.4s blocking in a web request thread
```

In an async web framework (FastAPI, Starlette) this blocks the event loop. Even in a threaded WSGI context this is a latency hit on a high-demand endpoint. At minimum document the threading assumption. If there's any chance this runs in an async context, use `asyncio.sleep`.

---

### 🟢 What's Done Well

- Lock ownership via UUID token is correct and necessary
- `nx=True, ex=self.LOCK_TTL` in a single `SET` call is the right approach — no SETNX/EXPIRE split
- Structured logging with key/value pairs is clean and consistent
- `ReservationResult` as a typed return value is the right pattern
- The lock-not-released-if-expired test (`test_lock_not_released_if_expired`) shows good defensive thinking
