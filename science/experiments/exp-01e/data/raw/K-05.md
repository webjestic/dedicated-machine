# Code Review: `feature/flash-sale-inventory`

---

## Decision: Request Changes

## Summary

This implementation contains several critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The distributed lock design has fundamental race conditions that no unit test in this suite can detect, the heartbeat mechanism introduces a dangerous TOCTOU vulnerability, and the database operations lack atomicity guarantees that the lock is implicitly assumed to provide. None of these issues will surface in CI — they will surface during your first flash sale.

---

## Critical Findings

### 1. 🔴 TOCTOU Race Condition in Lock Release — Silent Data Corruption

This is the most dangerous flaw in the codebase. The lock release in `_acquire_lock` is a non-atomic check-then-act sequence:

```python
# BROKEN: Two separate Redis round-trips. Not atomic.
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # Round-trip 1: read token
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # Round-trip 2: delete
```

**The failure scenario:**

```
Thread A (our process)         Redis                  Thread B (another node)
─────────────────────────────────────────────────────────────────────────────
GET lock_key  ──────────────►  returns token_A
                               [GC pause / network stall here]
                               TTL expires, key deleted
                                                       SET lock_key token_B ✓
                                                       [B reads stock = 5]
                                                       [B writes stock = 4]
DEL lock_key  ──────────────►  deletes token_B ← WRONG
                                                       [B's lock is now gone]
                                                       [C acquires lock]
                                                       [C reads stock = 4]
                                                       [C writes stock = 3]
```

Thread A deletes a lock it no longer owns. This is a classic distributed systems failure mode. The fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In the finally block:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The test `test_lock_not_released_if_expired` claims to cover this, but it only validates the happy path of the check. It cannot simulate the race between the GET and the DELETE because the mock is synchronous. **This test provides false confidence.**

### 2. 🔴 Heartbeat Thread Shares State Across Concurrent Calls — Guaranteed Corruption Under Load

The heartbeat state is stored as instance variables:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly instantiated once and shared (e.g., via dependency injection or a singleton). When two concurrent requests arrive:

```
Request A: _start_heartbeat() → sets self._heartbeat_active = True
Request B: _start_heartbeat() → sets self._heartbeat_active = True (same flag)

Request A completes → _stop_heartbeat() → sets self._heartbeat_active = False

Request B's heartbeat thread: reads self._heartbeat_active == False → exits
Request B's lock now expires mid-operation → another node acquires it
Request B continues writing to the database → two writers, no lock
```

This is a silent killer. Both requests succeed from the application's perspective. The database now has a corrupted reservation. Stock can go negative.

The fix is to make heartbeat state local to each lock acquisition context:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()  # start as active
    
    # ... acquire lock ...
    
    heartbeat_thread = threading.Thread(
        target=self._run_heartbeat,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    heartbeat_thread.start()
    
    try:
        yield token
    finally:
        heartbeat_active.clear()  # signal stop
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... atomic release via Lua ...
```

Note also that `_stop_heartbeat` does not join the thread. The heartbeat thread can still be running and extending the TTL *after* the lock has been released and re-acquired by another node. This compounds the TOCTOU issue above.

### 3. 🔴 Database Operations Are Not Atomic — The Lock Doesn't Protect What You Think It Does

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is OOM-killed, or receives SIGTERM between these two lines:

- Stock is decremented
- Order record is never created
- The reservation is lost with no trace
- The customer sees a failure but the inventory is permanently reduced

The Redis lock serializes access but provides zero durability guarantees for the database operations. This requires a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the underlying database doesn't support transactions (e.g., this is DynamoDB or a similar store), you need a compensating transaction pattern or a saga with explicit rollback logic.

### 4. 🟠 Heartbeat Timing Creates a Predictable Expiry Window

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at `t=10`, `t=20`, `t=30`. At `t=10`, the lock has 20 seconds remaining. This is fine. But consider a GC pause or network partition that begins at `t=9` and lasts 22 seconds. The heartbeat at `t=10` is blocked. The lock expires at `t=30`. Another node acquires the lock at `t=31`. The paused process resumes at `t=31`, the heartbeat fires, checks the token — and here is where the TOCTOU race in Finding #1 becomes exploitable.

The heartbeat interval should be at most `TTL / 3` to provide a safety margin, and the system must account for the fact that a heartbeat success does not mean the lock was *continuously* held. Redlock's fencing token pattern exists precisely for this reason. For true safety, consider returning a monotonically increasing fencing token from lock acquisition and requiring the database layer to reject writes with stale tokens.

### 5. 🟠 `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no lower bound check. A caller passing `quantity=0` will succeed, decrement stock by zero, and create a phantom order. A caller passing `quantity=-5` will *increase* stock (`new_stock = current_stock - (-5)`), silently corrupting inventory upward. This will not raise an exception anywhere in the current code.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

The PR description mentions "full input validation via Pydantic" but `quantity` is never passed through a Pydantic model. This checklist item is inaccurate.

---

### 6. 🟡 Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

Under a flash sale load, every thread waiting for a lock is a blocked thread consuming a connection from your database pool and a worker from your application server. With 1,000 concurrent requests and 3 retry attempts, you can exhaust your thread pool in milliseconds. The retry logic belongs in the caller with async primitives, or the service should fail fast and let the caller (e.g., a queue consumer) handle backoff externally.

---

### 7. 🟡 Test Suite Validates Mocks, Not Behavior

Beyond the issues already noted, the test suite has a structural problem: `_start_heartbeat` and `_stop_heartbeat` are mocked out in the fixture, which means the heartbeat threading bugs in Findings #2 and #4 are completely invisible to the test runner. The tests verify that the mock was called, not that the system behaves correctly. There are no integration tests against a real Redis instance (e.g., via `fakeredis` or a test container), no tests for partial database failure between `update_stock` and `create_order`, and no concurrency tests.
