Decision: Request Changes

Summary: The implementation demonstrates solid foundational thinking — atomic lock acquisition with `nx=True`, ownership verification before release, and heartbeat renewal are all correct patterns. However, there are several architectural and correctness flaws that must be addressed before this merges, ranging from a thread-safety race condition in the heartbeat to missing input validation that the checklist incorrectly claims is present.

---

## Critical Issues

### 1. Instance-Level Heartbeat State — Not Thread-Safe

This is the most serious flaw. `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables, not scoped to the lock context. If two requests hit the same `InventoryService` instance concurrently (which is the entire point of this service), they will stomp on each other's heartbeat state.

```python
# CURRENT — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

**Fix:** Scope the heartbeat flag to the lock acquisition context using a `threading.Event` passed into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        # Optionally join with a short timeout to avoid orphaned threads
        heartbeat_thread.join(timeout=2)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

### 2. Lock Release Is Not Atomic

The check-then-delete on lock release is a TOCTOU race:

```python
# CURRENT — two round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, the lock could expire and be acquired by another process. You would then delete a lock you no longer own, corrupting another request's critical section. This is a well-known Redis pattern that requires a Lua script to be atomic.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# Register once at init
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
released = self._release_script(keys=[lock_key], args=[token])
if released:
    logger.info("Lock released key=%s", lock_key)
else:
    logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same TOCTOU problem exists in the heartbeat's renewal check.

### 3. Input Validation Is Not Actually Present

The checklist marks input validation via Pydantic as complete, but `reserve_item` accepts raw, unvalidated parameters directly. There is no guard against negative quantities, empty strings, excessively long `item_id` values, or non-integer quantities. `ReservationResult` is an output model, not a validator for incoming data.

```python
# These all reach the lock and DB layer unchecked
service.reserve_item("", quantity=1)
service.reserve_item("item_001", quantity=-5)
service.reserve_item("item_001", quantity=0)
service.reserve_item("A" * 10000, quantity=1)
```

**Fix:** Add an input model and validate at the entry point.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation input: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")

    lock_key = f"inventory:lock:{req.item_id}"
    ...
```

## Moderate Issues

### 4. DB Operations Are Not Atomic

`update_stock` and `create_order` are two separate database calls with no transaction boundary. If `create_order` raises an exception after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent access, but it does not protect against partial failure.

```python
# CURRENT — stock can be lost here
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)  # <-- exception here = lost stock
```

These must be wrapped in a database transaction. The `db` abstraction should expose a transaction context manager, or this service should call a single atomic `db.reserve(item_id, order_id, quantity)` method.

---

### 5. Heartbeat Thread Leaks on Lock Acquisition Failure Path

If `_start_heartbeat` were called before the `if not acquired` check (it isn't currently, but the structure is fragile), a thread would leak. More concretely, if an exception is raised between `_start_heartbeat` and the `try` block, `_stop_heartbeat` is never called. The current code has exactly this gap:

```python
self._start_heartbeat(lock_key, token)  # thread starts

try:                                     # if something raises here before try...
    yield token
finally:
    self._stop_heartbeat()               # ...this never runs
```

The `threading.Event`-based fix in issue #1 resolves this if the `stop_event.set()` is placed in the `finally` block.

---

### 6. `time.sleep` Blocks the Calling Thread

The retry backoff uses `time.sleep` on the calling thread. In any async or thread-pool-based web framework (FastAPI, Gunicorn with workers), this will stall a worker for up to `0.1 + 0.2 = 0.3` seconds per call under contention. For a flash sale endpoint this will cascade quickly. If the service is running in an async context, this must use `asyncio.sleep`. If synchronous, document the threading model assumption explicitly.

## Minor Issues

### 7. Tests Mock Away the Behavior Being Tested

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat is acceptable for unit tests of the reservation logic, but there are zero tests covering the heartbeat thread itself — its renewal behavior, its stop behavior, or the race condition in issue #1. At minimum add integration-level tests for the lock lifecycle.

### 8. `test_lock_not_released_if_expired` Tests the Wrong Thing

This test verifies that `delete` is not called when the token doesn't match, which is correct behavior. However, once issue #2 (TOCTOU) is fixed with a Lua script, this test needs to be rewritten to verify the script is called with the correct arguments, not that `delete` is skipped.

### 9. `item_id` Is Interpolated Directly Into the Lock Key

```python
lock_key = f"inventory:lock:{item_id}"
```

Without the input validation fix in issue #3, a caller can inject arbitrary Redis key patterns. With validation in place this is fine, but it must not merge without it.
