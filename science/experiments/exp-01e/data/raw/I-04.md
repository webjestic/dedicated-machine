Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under slow I/O, but it contains a fundamental and silent safety gap that this PR does not address. The distributed lock pattern, as implemented, is not safe against **process-level pauses** — specifically stop-the-world GC events, VM live migrations, or OS-level scheduling preemptions — because such pauses freeze **every thread in the process simultaneously**, including the heartbeat thread. When the process resumes after the lock TTL has silently expired, the zombie worker continues executing with stale state and no awareness that another worker has already acquired the lock and committed a write. This is the classic **Zombie Leader / split-brain** failure mode in distributed locking, and the heartbeat provides zero protection against it.

---

## Detailed Findings

### 🔴 Critical — Zombie Leader / Fencing Token Gap

The failure scenario is concrete and reproducible under production load:

1. **Worker A** acquires the lock. Reads `current_stock = 5`.
2. A stop-the-world GC pause (or hypervisor migration) freezes the **entire Worker A process** — heartbeat thread included.
3. After 30 seconds, the lock TTL expires naturally. The heartbeat never fired because it was also frozen.
4. **Worker B** acquires the lock. Reads `current_stock = 5`. Writes `new_stock = 4`. Releases lock cleanly.
5. **Worker A** resumes. It has no idea time passed. It calculates `new_stock = 5 - 1 = 4` and calls `self.db.update_stock("item_001", 4)`.
6. The database now reads `4`, but the correct value is `3`. **One unit has been sold twice.**

The token check in the lock release (`current = self.redis.get(lock_key)`) only prevents a stale *delete* of the lock key. It does **not** prevent the stale `update_stock` write from landing in the database. The damage is done before the lock is ever released. The test `test_lock_not_released_if_expired` validates the wrong invariant — it confirms the lock key isn't deleted, but says nothing about whether the stale DB write was blocked.

**Required fix — optimistic concurrency / fencing token at the write layer:**

The database write must carry a version assertion. The canonical pattern is a conditional UPDATE:

```sql
UPDATE inventory
SET stock = %(new_stock)s, version = version + 1
WHERE item_id = %(item_id)s
  AND stock = %(expected_stock)s
  AND version = %(expected_version)s
```

If the UPDATE affects 0 rows, the operation must be treated as a conflict and aborted — not silently committed. This makes the zombie write fail at the database level regardless of what Redis believes about lock ownership. A monotonic `version` column (fencing token) is the canonical defense described in Kleppmann's *Designing Data-Intensive Applications*, §8.

In application code, this means `update_stock` must return an affected-row count, and `reserve_item` must check it:

```python
rows_affected = self.db.update_stock(item_id, new_stock, expected_stock=current_stock)
if rows_affected == 0:
    # Stale read — another writer already modified this row
    raise ConflictError(f"Concurrent modification detected for item_id={item_id}")
```

Without this, the Redis lock is providing only *advisory* mutual exclusion — it works when everything goes right, but provides no hard safety guarantee when processes pause.

### 🟡 Moderate — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**, not call-scoped. If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool), two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat thread prematurely. The heartbeat should carry its own cancellation event scoped to the lock context:

```python
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        ...

# In finally:
stop_event.set()
```

This also eliminates the `time.sleep` polling loop and makes shutdown deterministic.

### 🟡 Moderate — Heartbeat Has a TOCTOU Window on Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The GET and EXPIRE are not atomic. Between the GET and the EXPIRE, the lock could expire and be acquired by Worker B. The heartbeat would then extend Worker B's lock under Worker A's token check — except the token would no longer match, so this specific path is safe. However, the correct pattern is a Lua script to make this atomic:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

The same Lua atomicity concern applies to the lock release in `_acquire_lock`'s `finally` block.

### 🟡 Moderate — `quantity` Has No Lower-Bound Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller passing `quantity=0` or `quantity=-5` will pass the `current_stock < quantity` guard (since `10 < 0` is `False`) and successfully "reserve" a negative or zero quantity, potentially incrementing stock or creating a nonsense order. Pydantic is listed in the checklist but is only used for the response model, not for input validation. Add a guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

Or introduce a validated request model.

---

### 🟡 Moderate — `item_id` Is Not Sanitized

`item_id` is interpolated directly into the Redis key:

```python
lock_key = f"inventory:lock:{item_id}"
```

A malformed or adversarial `item_id` (e.g., containing `:`, newlines, or Redis command injection patterns) could collide with unrelated keys or cause unexpected behavior. Validate that `item_id` matches an expected format (alphanumeric + hyphens/underscores) before use.

---

### 🟠 Minor — `db.create_order` Failure Leaves Stock Decremented

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock has been decremented with no corresponding order record. These two writes should be wrapped in a database transaction, or the order should be created first and stock decremented only on order commit. This is a data integrity concern independent of the locking issue.

---

### 🟠 Minor — Test Suite Validates the Wrong Safety Property

`test_lock_not_released_if_expired` is presented as a correctness test for the expiry scenario, but it only verifies that `redis.delete` is not called when the token mismatches. It says nothing about whether `db.update_stock` was called with stale data — which is the actual danger. A correct test for this scenario should assert that `db.update_stock` is **not called** (or raises) when the lock has been lost mid-operation. As written, the test suite provides false confidence.

---

## Summary of Required Changes Before Approval

| Severity | Issue |
|---|---|
| 🔴 Critical | No fencing token / optimistic lock at `update_stock` — zombie writes are undetected |
| 🟡 Moderate | Heartbeat and lock release are not atomic (GET+EXPIRE, GET+DEL should be Lua scripts) |
| 🟡 Moderate | Instance-level heartbeat state is not safe for concurrent calls on the same instance |
| 🟡 Moderate | `quantity` accepts zero and negative values |
| 🟡 Moderate | `item_id` is not validated before use in Redis key construction |
| 🟠 Minor | `update_stock` + `create_order` are not transactional |
| 🟠 Minor | Test suite validates lock key cleanup, not write-layer safety |

The heartbeat is a legitimate optimization for slow I/O under a held lock. It is not, and cannot be, a substitute for write-layer fencing. The Redis lock provides advisory coordination between cooperative workers; the database write must enforce the invariant independently. Until a fencing token or optimistic concurrency check is in place at the `update_stock` call, this service will silently oversell inventory under GC pressure or VM migration — exactly the conditions most likely to occur during a high-traffic flash sale.
