# Code Review: `feature/flash-sale-inventory`

---

**Decision:** Request Changes

**Summary:** This implementation contains several critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The distributed locking strategy has fundamental race conditions that no unit test in this suite can detect, the heartbeat design introduces shared mutable state that breaks under concurrent use, and the database operations lack atomicity guarantees that the lock is falsely assumed to provide. None of these will surface in the current test suite because the mocks eliminate the exact timing windows where the bugs live.

---

## Critical Issues

### 1. 🔴 Non-Atomic Lock Release — TOCTOU Race Condition

This is the most dangerous bug in the codebase. The lock release in `_acquire_lock` is a classic **Time-of-Check to Time-of-Use (TOCTOU)** race:

```python
# BROKEN — two separate round trips to Redis
current = self.redis.get(lock_key)          # T1: we check ownership
if current and current.decode() == token:
    self.redis.delete(lock_key)             # T2: we delete
```

Between `T1` and `T2`, the following sequence is entirely possible:

1. Our lock TTL expires (e.g., during a GC pause or network hiccup between the two calls).
2. Another process acquires the lock and writes its own token.
3. We call `delete`, wiping out the lock we no longer own.
4. A third process now acquires the lock simultaneously with the second process.

**Two processes now hold the lock.** For a flash sale inventory system, this means overselling.

The canonical fix is a **Lua script**, which Redis executes atomically:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In _acquire_lock finally block:
self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

The same atomic-check-then-act problem exists in the heartbeat's `expire` call and must be fixed there too:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 2. 🔴 Shared Heartbeat State — Concurrent Request Corruption

`InventoryService` is almost certainly instantiated as a singleton (injected into a web framework, shared across request threads). The heartbeat state is stored as **instance-level attributes**:

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

Under concurrent requests for *different* items, this happens:

| Time | Thread A (item_001) | Thread B (item_002) |
|------|---------------------|---------------------|
| T1 | `_heartbeat_active = True` | |
| T2 | | `_heartbeat_active = True` (same attribute) |
| T3 | | `_heartbeat_active = False` (stop) |
| T4 | Thread A's heartbeat stops silently | |
| T5 | Thread A's lock expires mid-operation | |
| T6 | Another process steals the lock | |
| T7 | Thread A completes `update_stock` under no lock | **OVERSELL** |

The heartbeat state must be **scoped to the lock acquisition**, not the service instance. Pass it through a local closure or a dedicated lock context object:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()
    thread = threading.Thread(
        target=self._beat, 
        args=(lock_key, token, heartbeat_active), 
        daemon=True
    )
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

Using `threading.Event` also eliminates the non-atomic `while self._heartbeat_active` check in the heartbeat loop.

### 3. 🔴 Non-Atomic DB Operations — Lock Does Not Protect Against DB Failure

The lock serializes access, but the two database writes are not wrapped in a transaction:

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # crashes
```

If `create_order` raises an exception after `update_stock` succeeds, you have decremented stock with no corresponding order record. The customer sees a failure, retries, and the stock is now permanently phantom-decremented. At flash sale scale, this is a meaningful inventory leak.

The fix requires either a database transaction or a compensating rollback:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

Or at minimum, explicit compensation:

```python
self.db.update_stock(item_id, new_stock)
try:
    self.db.create_order(order_id, item_id, quantity)
except Exception:
    self.db.update_stock(item_id, current_stock)  # compensate
    raise
```

### 4. 🔴 Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The heartbeat thread sleeps for `HEARTBEAT_INTERVAL` (10 seconds). After `_stop_heartbeat()` is called, the thread can remain alive for up to 10 more seconds. During that window, the thread wakes up, checks `_heartbeat_active`, sees `False`... but due to the shared-state bug in Issue #2, it may be reading a *different* request's flag. The lock is already released, but the thread may still attempt to call `expire` on a key now owned by another process, silently extending a lock it no longer owns.

The thread must be joined before the lock is released:

```python
heartbeat_active.clear()
thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
# THEN release the lock
```

## Significant Issues

### 5. 🟠 `quantity` Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 0 is False), acquires the lock, and creates a zero-unit order. `quantity=-5` would *increase* stock. The PR claims Pydantic validation, but `reserve_item` is a plain method, not a Pydantic model. The validation doesn't exist.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

---

### 6. 🟠 `item_id` Is Not Sanitized Before Use as a Redis Key

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is user-supplied and contains characters like spaces, newlines, or Redis-special characters, this can cause unexpected key behavior or, in older Redis client versions, protocol injection. Validate or encode `item_id` before embedding it in a key:

```python
if not re.match(r'^[a-zA-Z0-9_\-]{1,128}$', item_id):
    raise ValueError(f"Invalid item_id format: {item_id!r}")
```

---

### 7. 🟠 Retry Backoff Is Blocking the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In an async framework (FastAPI, aiohttp) or a thread-pool-based WSGI server under load, this blocks the worker for up to 700ms. Under flash sale traffic, this will exhaust the thread pool. The retry logic should use `asyncio.sleep` in async contexts, or this should be documented as synchronous-only with explicit guidance on worker pool sizing.

## Test Suite Issues

### 8. 🟡 Tests Prove Nothing About the Critical Paths

The test suite mocks away every component that matters:

```python
svc._start_heartbeat = MagicMock()  # heartbeat never runs
svc._stop_heartbeat = MagicMock()   # stop never called
mock.set.return_value = True        # lock always acquired instantly
```

The tests verify that the *business logic conditional branches* work in isolation. They do not test:

- Concurrent access (the entire reason this code exists)
- Lock expiry during DB operations
- Heartbeat thread lifecycle
- The TOCTOU release race
- DB partial failure / rollback behavior

The `test_lock_not_released_if_expired` test is particularly misleading. It asserts `delete` is not called when the token doesn't match — but this is testing the broken non-atomic implementation. After the Lua fix, this test needs to be rewritten to verify the Lua script is invoked correctly.

At minimum, add integration tests using a real Redis instance (e.g., `pytest-redis` or a Docker fixture) and use `threading.Barrier` to force the race condition windows.
