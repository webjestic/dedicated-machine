**Decision: Request Changes**

**Summary:** The heartbeat thread addresses lock expiry under *normal* conditions, but it cannot prevent expiry in the scenarios that actually matter — GC pauses, network partitions, process stalls, or a heartbeat thread that silently dies. The database writes (`update_stock`, `create_order`) execute with no fencing mechanism whatsoever, meaning a process that resumes after its lock has expired will corrupt inventory state with no defense. This is not a lock implementation; it is a lock implementation with an optimistic hope attached.

---

## Detailed Findings

### Critical: No Fencing on the Write Path

This is the core defect. Consider the sequence:

```
Process A acquires lock
Process A stalls (GC pause, slow network, heartbeat thread starved)
Lock expires
Process B acquires lock, reads stock=5, writes stock=4
Process A resumes, writes stock=4 (its stale read was also 5)
```

The heartbeat *reduces* the probability of this window. It does not close it. The lock can expire between the last successful heartbeat and the write. After that point, nothing stops `update_stock` and `create_order` from executing on stale data while another process holds the lock legitimately.

```python
# This executes with no verification that the lock is still held.
# By the time we reach this line, the heartbeat may have already failed
# its last token check and the lock may be held by another process.
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

The test `test_lock_not_released_if_expired` demonstrates awareness of the expiry scenario but tests only the *release* path. It does not test — and cannot prevent — writes that execute after expiry.

**What is required:** The database layer must enforce the invariant independently of the Redis lock. The standard approaches are:

1. **Optimistic locking with a version/fence column.** Read a `version` value alongside stock. The `UPDATE` includes `WHERE version = :read_version AND stock >= :quantity`. Check `rowcount == 1`. If zero rows were updated, abort. The database serializes this correctly regardless of what Redis did.

2. **Database-level conditional update.** `UPDATE inventory SET stock = stock - :qty WHERE item_id = :id AND stock >= :qty`. The atomicity guarantee comes from the database, not from Redis.

3. **Fencing tokens passed to the storage layer.** A monotonically increasing token issued at lock acquisition time, verified by the storage backend before accepting any write. This is the approach Kleppmann describes and requires the storage layer to participate.

The Redis lock can still serve a useful purpose — reducing contention and database load — but it must be treated as a *performance optimization*, not a *correctness mechanism*. Correctness must live in the storage layer.

### Significant: Heartbeat Thread Failure Is Silent

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

If the `redis.get` or `redis.expire` call raises — connection reset, timeout, Redis failover — the exception is swallowed by the thread and the loop exits silently. The main thread has no signal that heartbeats have stopped. It continues executing under the assumption that the lock is held.

At minimum, the thread should set a shared flag on failure that the main thread checks before executing writes. More robustly, the thread should propagate the failure so the operation can be aborted.

```python
# Minimal mitigation — still not sufficient without write-path fencing
self._heartbeat_failed = threading.Event()

def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        try:
            ...
        except Exception as e:
            logger.error("Heartbeat failed, lock may have expired: %s", e)
            self._heartbeat_failed.set()
            return
```

### Significant: Instance-Level Heartbeat State Breaks Concurrency

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes, not per-lock-acquisition attributes. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — one call's `_stop_heartbeat()` will set `_heartbeat_active = False` and terminate the heartbeat for all concurrent acquisitions. The second lock's heartbeat stops. Its TTL is not renewed. It expires.

Each lock acquisition must carry its own isolated heartbeat state.

---

### Moderate: `update_stock` and `create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary shown. If the process dies, is killed, or the database connection drops between them, stock is decremented but no order exists. The item is lost. These operations must execute within a single transaction that either commits both or neither.

---

### Moderate: Tests Mock Away the Behavior Under Test

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking the heartbeat is reasonable for unit tests. The problem is that the tests then assert correctness of the reservation logic without any test covering the scenario that actually matters: what happens when the lock expires *before* the write executes. The test suite provides false confidence. A test that injects an expired lock state between the stock read and the stock write — and verifies that the write is rejected — does not exist and cannot exist with the current architecture, because there is nothing to reject it.

---

### Minor: LOCK_TTL and HEARTBEAT_INTERVAL Are Class-Level Constants

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

These should be constructor parameters with defaults. Different deployments, different database latency profiles, and different item types may require different values. Hardcoded class constants make this impossible to configure without modifying the class.

---

## What Approval Requires

1. **Write-path fencing is non-negotiable.** The database must enforce the stock invariant atomically, independent of whether the Redis lock is still held at write time. The specific mechanism — optimistic locking, conditional update, or storage-layer fencing tokens — is open for discussion, but something must guard the write itself.

2. **`update_stock` and `create_order` must be wrapped in a single database transaction.**

3. **Heartbeat state must be scoped per lock acquisition**, not per service instance.

4. **Heartbeat failures must be detectable** by the main thread, even if the response to detection is only to log and abort rather than to guarantee safety (since safety must come from point 1).

5. **Tests must cover the expired-lock-before-write scenario** and demonstrate that the storage layer rejects the stale write.
