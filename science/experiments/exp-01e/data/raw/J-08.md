Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — storing heartbeat state as instance variables (`self._heartbeat_active`, `self._heartbeat_thread`) makes the `InventoryService` class unsafe for concurrent use, which is exactly the scenario this code is designed to handle. Beyond that, there are several architectural and correctness issues that need to be resolved before this is anywhere near production-ready for a high-demand flash sale context.

---

## Critical Issues

### 1. Instance-Level Heartbeat State (Race Condition)

This is the most serious problem. The heartbeat state is stored on the instance, not scoped to the lock invocation:

```python
# BROKEN — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two threads call `reserve_item()` on the same `InventoryService` instance concurrently (which is the entire point of this service), one thread calling `_stop_heartbeat()` will set `self._heartbeat_active = False` and kill the *other* thread's heartbeat. This silently drops the distributed lock protection you're trying to provide, and it won't show up in any of the current tests because they mock out the heartbeat entirely.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
    ...
```

### 2. Non-Atomic DB Operations Inside the Lock

The lock protects the Redis key, but `update_stock` and `create_order` are two separate database calls with no transaction boundary:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes or the DB connection drops between these two calls, you will have decremented stock with no corresponding order record, or vice versa. For a government contracting context where auditability matters, this is not acceptable. These must be wrapped in a database transaction.

### 3. No Input Validation on `reserve_item`

The checklist claims "Input validation via Pydantic" but there is no validation on the actual public method inputs. `item_id` is an unvalidated string and `quantity` has no bounds check:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller passing `quantity=0`, `quantity=-5`, or an empty `item_id` will either silently succeed in a nonsensical way or produce confusing behavior. Pydantic is imported but only used for the *output* model. Add a request model:

```python
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(..., ge=1, le=1000)
```

## Moderate Issues

### 4. Heartbeat Race on Lock Release

There is a TOCTOU window in the lock release logic:

```python
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)          # <-- read
    if current and current.decode() == token:
        self.redis.delete(lock_key)             # <-- write
```

Between the `GET` and the `DELETE`, another process could acquire the lock. This should be a Lua script to make it atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    script(keys=[lock_key], args=[token])
```

The same TOCTOU issue exists in the heartbeat's renewal check.

### 5. Heartbeat Thread Leaks on Unhandled Exceptions

The heartbeat thread is daemon=True, which prevents hard leaks, but `_stop_heartbeat()` is only called in the `finally` block. If `_start_heartbeat` itself raises (e.g., thread limit hit), the lock is held but the heartbeat never starts, and the error handling path is unclear. The thread should also be joined with a timeout on stop to confirm clean shutdown:

```python
def _stop_heartbeat(self, heartbeat_active: threading.Event, thread: threading.Thread) -> None:
    heartbeat_active.clear()
    thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
```

### 6. Silent Swallowing of DB Exceptions

Only `RuntimeError` from lock acquisition is caught. Any exception from `db.get_stock`, `db.update_stock`, or `db.create_order` will propagate uncaught out of `reserve_item`, bypassing the structured error response contract. Add explicit handling:

```python
except (DatabaseError, ConnectionError) as e:
    logger.error("DB operation failed item_id=%s: %s", item_id, e, exc_info=True)
    return ReservationResult(success=False, message="Service temporarily unavailable.")
```

## Minor Issues

### 7. Test Coverage Gaps

- No test for the atomic lock release (Lua script path once fixed)
- No test for `quantity <= 0` or empty `item_id` once validation is added
- No test verifying that a DB exception mid-operation returns a clean error response
- `test_lock_not_released_if_expired` is testing the right behavior but the assertion is fragile — it relies on the mock returning `b"different-token"` which bypasses the actual token comparison logic since the service generates its own UUID token internally. The test should be restructured to properly simulate the expiry scenario

### 8. `LOCK_TTL` and `HEARTBEAT_INTERVAL` as Class-Level Constants

These should be configurable at instantiation time, not hardcoded as class variables. Flash sale load profiles vary. Pass them through `__init__` with sensible defaults.

---

## What's Done Well

The overall structure is sound. The context manager approach for lock lifecycle is the right pattern. Ownership verification via token before release is correct thinking. Exponential backoff is appropriate. Structured logging with key/value pairs is good practice. The Pydantic output model is clean.

This just needs the concurrency model fixed before it can be trusted in production.
