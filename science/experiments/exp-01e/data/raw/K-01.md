# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains multiple critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The heartbeat mechanism introduces a race condition that actively undermines the safety guarantee it claims to provide, the lock release is non-atomic and exploitable, and the entire design rests on a false premise — a Redis distributed lock does not provide the transactional safety this code assumes it does. None of these failure modes are exercised by the test suite, which is structured in a way that makes the most dangerous bugs invisible.

---

## Critical Issues

### 1. Non-Atomic Lock Release — TOCTOU Race Condition

```python
# _acquire_lock finally block
current = self.redis.get(lock_key)          # ← T1: we read the token
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← T2: we delete the key
```

There is a window between `GET` and `DELETE` where the lock can expire and be re-acquired by another process. At T1, we confirm we own the lock. Between T1 and T2, a Stop-the-World GC pause, a slow network round-trip, or a thread preemption can cause the lock to expire. A second process acquires it at T1.5. We then delete it at T2, releasing a lock we no longer own. The second process now believes it holds an exclusive lock that has been silently revoked.

**This is the canonical Redis distributed lock mistake.** The fix is a Lua script that makes the check-and-delete atomic:

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    script = self.redis.register_script(RELEASE_SCRIPT)
    return bool(script(keys=[lock_key], args=[token]))
```

### 2. The Heartbeat Thread Actively Creates the Race Condition It Claims to Prevent

The heartbeat is presented as a safety feature. It is, in fact, a safety hazard.

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # ← sleeps 10s
        if self._heartbeat_active:
            current = self.redis.get(lock_key)  # ← non-atomic read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← non-atomic extend
```

Consider this sequence:

1. Process A holds the lock. Its GC pauses for 31 seconds. The lock expires.
2. Process B acquires the lock with a new token.
3. Process A's heartbeat thread wakes up, reads the key, sees... Process B's token. It correctly skips the extend. So far so good.
4. **But what if the GC pause is exactly 29 seconds?** The lock has 1 second left. The heartbeat wakes up, reads Process A's token (still valid), and extends by 30 seconds. Process A's main thread resumes, now operating under a lock that was *nearly* expired, with no awareness that it may have been in an inconsistent state for 29 seconds.
5. More critically: the `GET` and `EXPIRE` are two separate commands. Between them, the lock can expire and be re-acquired by Process B. Process A's heartbeat then extends Process B's lock under Process A's token check — this is impossible if tokens differ, but the window for a race between `GET` and `EXPIRE` is real and non-atomic.

The atomic operation for heartbeat extension is `SET key token XX EX ttl` or a Lua script. The current implementation is not atomic.

Furthermore, there is a deeper conceptual problem: **a heartbeat that can extend a lock held by a process that has been paused for longer than the original TTL is not a safety mechanism — it is a liveness mechanism that trades away the safety guarantee.** The entire point of the TTL is that if a process is paused long enough, other processes can make progress. The heartbeat defeats this, but does so unreliably.

### 3. The Distributed Lock Does Not Protect the Database Operations

This is the most dangerous architectural flaw because it is invisible in testing.

```python
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # ← DB read
    # ... logic ...
    self.db.update_stock(item_id, new_stock)      # ← DB write
    self.db.create_order(order_id, item_id, quantity)  # ← DB write
```

The Redis lock serializes access among processes that *participate in the locking protocol*. It provides zero protection against:

- **Direct database writes** from any other service, migration script, admin tool, or database trigger that does not acquire this lock.
- **Database-level concurrent transactions** from any other code path in this same service that calls `db.update_stock` without going through `reserve_item`.
- **The lock expiring mid-operation** (even with the heartbeat, as shown above), after which the database writes proceed on a process that no longer holds the lock.

The stock decrement must be atomic at the database level regardless of the lock. The correct pattern is a conditional update:

```sql
UPDATE inventory
SET stock = stock - :quantity
WHERE item_id = :item_id
  AND stock >= :quantity
RETURNING stock;
```

If this returns zero rows, the reservation fails. This is the only correct way to prevent overselling. The Redis lock can remain as a performance optimization to reduce database contention, but it cannot be the *sole* correctness mechanism.

### 4. Non-Atomic `update_stock` + `create_order` — Partial Write Corruption

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, loses its database connection, or encounters an exception between these two lines, stock is decremented but no order is created. The inventory is permanently lost with no audit trail. This is a silent data corruption bug that will manifest as inventory shrinkage that cannot be reconciled.

These operations must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

### 5. Shared Heartbeat State Breaks Concurrent Usage

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False
        self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are instance-level attributes. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the normal usage pattern for any web framework with a thread pool — the following happens:

1. Thread A acquires lock for `item_001`, starts heartbeat, sets `_heartbeat_active = True`.
2. Thread B acquires lock for `item_002`, starts heartbeat, **overwrites** `_heartbeat_thread` with its own thread.
3. Thread A finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False`.
4. **Thread B's heartbeat is now dead.** Thread B's lock will expire during its database operations.
5. Thread A's `finally` block tries to release its lock. The heartbeat for lock A was already killed by Thread B's `_stop_heartbeat` call, but the lock release itself is still attempted — however, the lock may have already expired and been re-acquired.

The heartbeat state must be scoped to the lock context, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

## Significant Issues

### 6. No Input Validation on `quantity`

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is no validation that `quantity > 0`. A caller passing `quantity=0` will succeed, decrement stock by zero, and create an order for zero units. A caller passing `quantity=-5` will *increase* stock (`new_stock = current_stock - (-5)`), effectively creating inventory from nothing. This is a business logic vulnerability.

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

`item_id` should also be validated for empty strings.
