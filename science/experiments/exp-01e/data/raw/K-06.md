# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains multiple critical flaws that will cause silent data corruption and phantom inventory loss under real production conditions. The heartbeat mechanism introduces a race window that defeats the purpose of the distributed lock, the database operations are non-atomic and unguarded against partial failure, and the `InventoryService` instance is fundamentally unsafe for concurrent use. None of these failure modes are exercised by the test suite, which is structured to avoid the exact conditions that matter.

---

## Critical Issues

### 1. Instance-Level Heartbeat State Is Not Thread-Safe (Data Corruption)

This is the most dangerous flaw in the codebase. `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables, not as locals scoped to a single lock acquisition.

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If a single `InventoryService` instance is shared across threads — which is the standard deployment pattern for any web framework (Django, Flask, FastAPI) — two concurrent calls to `reserve_item` will race on these shared fields. Thread A's `_stop_heartbeat()` call will set `self._heartbeat_active = False`, which immediately kills Thread B's heartbeat. Thread B's lock is now silently unprotected and will expire, allowing a third process to acquire it while Thread B's database writes are still in flight. You now have two writers operating simultaneously under the assumption of mutual exclusion.

**The fix:** Heartbeat state must be local to each lock acquisition context, not stored on `self`. Use a local `threading.Event` passed into the heartbeat closure.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # scoped to this acquisition
    acquired = False
    # ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock
```

### 2. Non-Atomic Database Operations With No Rollback Path (Phantom Orders)

The two database writes are not wrapped in a transaction:

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` raises an exception, the stock is permanently decremented but no order record exists. The item is gone from inventory with no corresponding sale. Under flash sale load — where DB timeouts and transient errors are common — this is not a theoretical edge case. It will happen.

Separately, if `update_stock` itself raises, the exception propagates out of the `with self._acquire_lock(lock_key):` block. The `finally` clause in `_acquire_lock` will run and release the lock correctly, but the `except RuntimeError` in `reserve_item` will **not** catch it (it's not a `RuntimeError`), so the exception bubbles up unhandled and the caller receives a 500 rather than a clean `ReservationResult`. The error handling contract is broken.

**The fix:** Wrap both writes in a single database transaction. Catch all exceptions at the `reserve_item` boundary, not just `RuntimeError`.

```python
try:
    with self.db.transaction():
        self.db.update_stock(item_id, new_stock)
        self.db.create_order(order_id, item_id, quantity)
except Exception as e:
    logger.error("DB write failed, transaction rolled back: %s", e)
    return ReservationResult(success=False, message="Reservation failed. Please try again.")
```

### 3. TOCTOU Race in Lock Release (Lock Theft Window)

The lock release in the `finally` block is a two-step operation — a `GET` followed by a `DELETE` — with no atomicity guarantee:

```python
current = self.redis.get(lock_key)      # Step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)          # Step 2
```

Between Step 1 and Step 2, the following can happen: the lock TTL expires (perhaps the heartbeat thread was delayed by a GIL-induced pause or a JVM-style stop-the-world GC event), a second process acquires the lock and writes its own token, and then this process deletes that second process's lock. You have now released a lock you no longer own, and a third process can immediately acquire it, giving you three concurrent writers.

This is a well-known problem with a well-known solution: use a Lua script to make the check-and-delete atomic.

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

release = self.redis.register_script(RELEASE_SCRIPT)
release(keys=[lock_key], args=[token])
```

The same atomic-script pattern should be applied to the heartbeat's `expire` call.

### 4. Heartbeat Thread Is Not Joined on Exit (Resource Leak and Delayed Corruption)

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # No join()
```

Setting the flag to `False` is a signal, not a guarantee. The heartbeat thread may be mid-sleep (`time.sleep(10)`) when the flag is set. The `finally` block then proceeds to delete the lock key. Up to 10 seconds later, the heartbeat thread wakes up, checks the flag, sees it's `False`, and stops — but in the window between the lock deletion and the thread waking, if a new process acquired the lock, the sleeping thread could theoretically still call `expire` on a key it no longer owns (though the token check mitigates this partially, it's still a logic hazard and a resource leak).

More concretely: the lock is deleted in `finally`, but the thread is still alive and holding a reference to `lock_key` and `token`. In a high-throughput system, you will accumulate these zombie threads.

**The fix:** Always `join()` the heartbeat thread after setting the stop signal, with a bounded timeout.

## Significant Issues

### 5. `quantity` Parameter Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no validation. A caller passing `quantity=0` will succeed (stock check passes, `new_stock == current_stock`, an order is created for zero units). A caller passing `quantity=-5` will *increase* the stock level. The PR checklist claims "Input validation via Pydantic" but `ReservationResult` is the *output* model. There is no input validation model.

**The fix:**

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive.")
```

Or define a `ReservationRequest` Pydantic model with `quantity: int = Field(ge=1)`.

---

### 6. `item_id` Is Used Directly in a Redis Key Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is sourced from user input (which it almost certainly is in a flash sale context), a malicious or malformed value like `"item_001\ninventory:lock:item_002"` or an extremely long string could cause unintended behavior. At minimum, validate that `item_id` matches an expected format before constructing the key.

---

### 7. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)  # 0.1s, 0.2s, 0.4s
```

In an async framework (FastAPI with `asyncio`) or a thread-pool-based server under flash sale load, blocking the calling thread for up to 700ms during lock contention will exhaust the thread pool and cause cascading latency. This should use non-blocking sleep (`asyncio.sleep`) if the service is async, or the retry logic should be reconsidered entirely.

## Test Suite Deficiencies

The test suite is written to pass, not to find bugs. It explicitly avoids the failure modes that matter:

| Scenario | Tested? |
|---|---|
| Concurrent calls on the same instance racing on `_heartbeat_active` | ❌ |
| `update_stock` succeeds, `create_order` raises | ❌ |
| TOCTOU on lock release (token changes between GET and DELETE) | ❌ |
| `quantity=0` or `quantity=-1` | ❌ |
| Heartbeat thread still alive after lock release | ❌ |
| Redis connection failure mid-operation | ❌ |

`test_lock_not_released_if_expired` is the right idea but tests the wrong thing — it verifies the `delete` is skipped when the token mismatches, but does not test the race condition that causes the mismatch in the first place.

---

## Summary of Required Changes

| Priority | Issue | Required Action |
|---|---|---|
| P0 | Instance-level heartbeat state | Move to local `threading.Event` per acquisition |
| P0 | Non-atomic DB writes | Wrap in a single DB transaction with rollback |
| P0 | TOCTOU on lock release | Replace GET+DELETE with atomic Lua script |
| P1 | Heartbeat thread not joined | Add `thread.join(timeout=...)` before lock release |
| P1 | No input validation on `quantity` | Add guard for `quantity <= 0` |
| P1 | `item_id` unsanitized in Redis key | Validate format before key construction |
