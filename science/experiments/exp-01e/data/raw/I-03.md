Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under normal load, but it contains a fundamental and silent failure mode that this codebase does not address: **it provides zero protection against process-level pauses**. A stop-the-world GC event, VM live migration, or OS scheduler preemption freezes every thread in the process simultaneously — the heartbeat thread included. When the process resumes, the lock TTL may have already elapsed, been acquired by a competing worker, and that worker may have already committed a write. The resumed worker has no awareness of this; it holds a stale `current_stock` value and proceeds to call `db.update_stock()` with a now-incorrect new stock figure. The result is a silent double-sell: two workers each believe they successfully reserved the same unit, and the database reflects the last writer's value rather than the correct decremented total.

---

### The Specific Failure Sequence

```
Worker A acquires lock. Reads current_stock = 5.
  → [GC pause / VM migration freezes entire process — all threads halted]
  → Lock TTL (30s) expires. Redis evicts the key.
Worker B acquires lock. Reads current_stock = 5. Writes new_stock = 4. Releases lock.
  → [Worker A resumes, unaware any time has passed]
Worker A calculates new_stock = 5 - 1 = 4.
Worker A calls db.update_stock("item_001", 4).  ← STALE WRITE SUCCEEDS
```

**Outcome:** Stock reads `4`. Two customers hold confirmed reservations for the same unit. The heartbeat thread was also paused and could not extend the TTL. The ownership token check in the lock release correctly skips the `DELETE` (Worker A no longer owns the lock), but that is the *only* guard that fires — and it fires *after* the stale write has already been committed to the database.

This is the classical **Zombie Leader** problem described by Martin Kleppmann in *Designing Data-Intensive Applications* (§8, "The Truth Is Defined by the Majority"). The Redis lock provides mutual exclusion only under the assumption that the process holding the lock remains alive and responsive. That assumption is not safe in a JVM, CPython (GC), or containerized environment subject to live migration.

---

### Why the Existing Mitigations Are Insufficient

| Mitigation in PR | What it protects against | What it does NOT protect against |
|---|---|---|
| Heartbeat renewal thread | Lock expiry during slow-but-live DB operations | Process-wide pauses (GC, VM migration, SIGSTOP) |
| Token ownership check on release | Accidental release of a lock we no longer own | Stale writes that already occurred before release |
| Exponential backoff retry | Thundering herd on lock acquisition | Nothing about write correctness post-pause |

The token check at release is correct behavior, but it is a *release guard*, not a *write guard*. The dangerous window is not the release — it is the `db.update_stock()` call that precedes it.

---

### Required Fix: Fencing Token / Optimistic Lock at the Database Write

The canonical solution is to push the safety invariant down to the database layer, where it cannot be bypassed by a paused process. The `update_stock` call must include a conditional predicate on the value it read:

```python
# CURRENT — unconditional, vulnerable to stale writes
self.db.update_stock(item_id, new_stock)

# REQUIRED — optimistic lock / fencing token pattern
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_stock=current_stock   # or a row-level version/fence token
)
if rows_affected == 0:
    # Another writer modified the row while we were paused.
    # The reservation must be aborted and retried or failed cleanly.
    raise StaleDataError(f"Stock for {item_id} was modified concurrently; aborting reservation.")
```

The underlying SQL must be:

```sql
UPDATE inventory
SET stock = %(new_stock)s
WHERE item_id = %(item_id)s
  AND stock = %(expected_stock)s   -- optimistic lock predicate
```

A monotonically incrementing `version` column is the more robust canonical form (a fencing token), as it survives edge cases where stock happens to return to the same value:

```sql
UPDATE inventory
SET stock = %(new_stock)s, version = version + 1
WHERE item_id = %(item_id)s
  AND version = %(expected_version)s
```

If `rowcount == 0`, the write was rejected because the row changed underneath us. The zombie write fails at the database level, not at the application level, which is the only layer that can be trusted after a process pause.

### Secondary Concerns

**1. Instance-level heartbeat state is not concurrency-safe.**
`_heartbeat_active` and `_heartbeat_thread` are instance variables on `InventoryService`. If a single service instance is used across multiple concurrent calls (e.g., in a thread pool), two simultaneous `reserve_item` calls will race on `_heartbeat_active`, causing one heartbeat to silently kill the other's renewal loop. These should be local variables scoped to the lock context, not instance state.

```python
# Problematic: shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

Each lock acquisition needs its own independent heartbeat lifecycle, isolated via a local `threading.Event` or a closure-scoped flag.

**2. The heartbeat thread has a TOCTOU window on the token check.**
```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```
The `GET` and `EXPIRE` are not atomic. Between the `GET` and the `EXPIRE`, the lock could expire and be acquired by Worker B. The heartbeat then extends Worker B's lock using Worker B's key but Worker A's renewal logic. This should be a Lua script executed atomically:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

**3. Test suite mocks out the exact mechanism under scrutiny.**
`svc._start_heartbeat = MagicMock()` and `svc._stop_heartbeat = MagicMock()` disable the heartbeat entirely in tests. This is acceptable for unit isolation, but there are no integration-level tests that exercise the heartbeat under simulated contention or verify that the atomic Lua renewal behaves correctly. Given that the heartbeat is the PR's primary claimed safety mechanism, its absence from meaningful test coverage is a gap.

**4. `quantity` has no validation floor.**
A caller passing `quantity=0` or `quantity=-5` would pass the `current_stock < quantity` check (e.g., `10 < 0` is false), decrement stock to an incorrect value, and create a spurious order. A guard of `if quantity < 1: raise ValueError(...)` is missing. Pydantic is listed in the checklist but is only used for the response model, not for input validation on `reserve_item`.

---

### Summary of Required Changes Before Approval

| Priority | Issue |
|---|---|
| **Critical** | Add optimistic lock / fencing token predicate to `db.update_stock` — the heartbeat cannot protect against process-wide pauses |
| **High** | Move heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) out of instance scope into per-call local scope |
| **High** | Make heartbeat token renewal atomic via Lua script |
| **Medium** | Add `quantity >= 1` input validation |
| **Low** | Add integration test coverage for the heartbeat and concurrent write paths |

The core logic of the lock acquisition, backoff, ownership-verified release, and structured logging is sound. The architecture is not approvable in its current form solely because of the missing database-level write guard — a process pause of sufficient duration will produce silent inventory corruption with no error surfaced to the caller or the logs.
