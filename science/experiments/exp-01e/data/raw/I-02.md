Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard but contains a fundamental safety gap that makes the lock guarantee illusory under real-world failure conditions. The implementation is vulnerable to the **Zombie Leader problem**: a stop-the-world GC pause (or OS-level VM migration/live migration event) freezes the *entire process*, including the heartbeat thread, simultaneously. When the process resumes after the 30-second TTL has elapsed, Worker A holds a stale read of `current_stock` and proceeds to call `db.update_stock()` with a value computed before the pause — unaware that Worker B has already acquired the lock, performed its own reservation, and released cleanly. The result is a phantom reservation: stock is decremented to `4` twice when it should be `3`, meaning one unit is sold twice. No amount of heartbeat tuning resolves this because the failure is a process-level freeze, not a slow operation.

---

### Critical Flaw: Heartbeat Cannot Protect Against Process Pauses

The PR checklist states *"Heartbeat thread prevents lock expiry under load"* — this is only true for the case where the process is **running slowly**. It is categorically false for the case where the process is **not running at all**.

```
Worker A                          Redis TTL          Worker B
────────────────────────────────────────────────────────────────
acquires lock (token=A)           30s
reads current_stock = 5
                                                     
[GC / VM pause — ALL threads frozen, including heartbeat]
                                  TTL expires → 0
                                                     acquires lock (token=B)
                                                     reads current_stock = 5
                                                     writes new_stock = 4  ✓
                                                     releases lock
[process resumes]
computes new_stock = 5 - 1 = 4
redis.get(lock_key) → token=B ≠ token=A
lock release skipped ✓ (correct)
db.update_stock("item_001", 4)  ← ZOMBIE WRITE ✗
db.create_order(order_id, ...)  ← PHANTOM ORDER ✗

Final DB state: stock=4, but 2 orders exist for 1 unit
```

The lock ownership check in the `finally` block correctly avoids *releasing* the stolen lock — but it does **not** prevent the stale write from landing. By the time the token mismatch is detected, the damage is already done two lines earlier.

---

### Required Fix: Fencing Token / Optimistic Lock at the Database Write

The only correct mitigation is to make the **database write itself reject stale state**. The distributed lock provides coordination under normal conditions; the database must be the final arbiter of correctness under failure conditions.

**Option 1 — Optimistic concurrency (compare-and-swap on stock value):**

```python
# Instead of unconditional update:
self.db.update_stock(item_id, new_stock)

# Require the DB to confirm the precondition still holds:
rows_affected = self.db.update_stock_if_unchanged(
    item_id,
    expected_stock=current_stock,
    new_stock=new_stock
)
if rows_affected == 0:
    # Another writer landed between our read and write — abort
    raise StaleInventoryError(f"Concurrent modification detected for {item_id}")
```

```sql
-- Underlying query
UPDATE inventory
   SET stock = %(new_stock)s
 WHERE item_id = %(item_id)s
   AND stock   = %(expected_stock)s   -- fencing condition
```

**Option 2 — Monotonic fencing token (canonical Redlock-style defense):**

Add a `version` column to the inventory row. Read the version alongside stock, and include it in the `WHERE` clause of the update. Increment the version on every write. A zombie write from a stale reader will fail because its version is behind.

```sql
UPDATE inventory
   SET stock = %(new_stock)s, version = version + 1
 WHERE item_id = %(item_id)s
   AND version = %(expected_version)s
```

Either approach ensures that even if the distributed lock fails — for any reason, including GC pauses, clock skew, network partitions, or Redis failover — the database layer will reject the conflicting write with zero rows affected, and the caller can return a clean failure rather than silently corrupting inventory.

### Secondary Concerns (Non-Blocking but Should Be Addressed)

**1. Instance-level heartbeat state is not thread-safe for concurrent callers**

`_heartbeat_active` and `_heartbeat_thread` are instance variables. If two callers invoke `reserve_item` concurrently on the same `InventoryService` instance, one caller's `_stop_heartbeat()` will silence the other's heartbeat thread mid-operation. The heartbeat state should be scoped to the lock context, not the service instance.

```python
# Problematic — shared mutable state on `self`
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

Each `_acquire_lock` invocation should carry its own `heartbeat_active` flag, passed by closure or via a local object, not written to `self`.

**2. `db.update_stock` and `db.create_order` are not atomic**

If `update_stock` succeeds but `create_order` raises, stock has been decremented with no corresponding order. This should be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

**3. Test suite mocks away the exact behavior under scrutiny**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is fully stubbed in every test. There is no test that exercises the lock expiry / token mismatch path with an actual (or simulated) TTL expiry and concurrent write. `test_lock_not_released_if_expired` verifies that `delete` is not called when the token mismatches — but does not assert that the DB write is also suppressed. Once the fencing token fix is in place, a test must verify that a stale `update_stock_if_unchanged` returning `rows_affected=0` causes the reservation to fail cleanly.

**4. `quantity` has no lower-bound validation**

A caller passing `quantity=0` or `quantity=-5` would succeed the stock check and create a zero or negative-quantity order. Add explicit validation:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

### What the PR Gets Right

To be fair: the lock ownership verification via UUID token is correct. The atomic `SET NX EX` acquisition is correct. The exponential backoff is reasonable. The structured logging is clean. The Pydantic result model is a good interface choice. The bones of this implementation are solid — the critical gap is specifically the absence of a write-side safety net, which is a well-known and non-obvious requirement in distributed systems literature (see Kleppmann, *"How to do distributed locking"*, 2016).

**This must not be merged until the fencing token / optimistic lock is in place.** The current code will silently oversell inventory under GC pressure or VM-level scheduling events, both of which are routine in containerized and cloud-hosted environments.
