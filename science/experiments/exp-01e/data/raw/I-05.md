Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under normal load, but it contains a fundamental and silent correctness flaw that the test suite does not cover and that no amount of heartbeat tuning can fix. The distributed lock strategy as written is insufficient for a flash sale inventory system where overselling is a critical failure mode.

---

## Critical Flaw: The Zombie Leader / Process Pause Race Condition

The heartbeat thread protects against one failure mode — a *running* process that is simply slow. It does **not** protect against a stop-the-world process pause, which is the more dangerous scenario.

Here is the exact failure sequence:

```
Worker A acquires lock. Reads current_stock = 5.

>>> PROCESS PAUSE (GC, VM migration, OS scheduler starvation) <<<
    All threads in the process are frozen — including the heartbeat thread.
    30 seconds elapse. Lock TTL expires in Redis.

Worker B acquires the lock (legitimately).
Worker B reads current_stock = 5.
Worker B writes new_stock = 4. Releases lock.

>>> Worker A resumes <<<
Worker A calculates new_stock = 5 - 1 = 4.
Worker A calls db.update_stock("item_001", 4).  ← STALE WRITE SUCCEEDS
Worker A calls db.create_order(...)             ← ORDER IS CREATED

Result: stock = 4, but two orders exist for the same unit.
        One item has been sold twice.
```

The heartbeat thread is frozen alongside every other thread in the process during a stop-the-world pause. When the process resumes, it has no awareness that the lock expired and was re-acquired by another worker. The token check in the `finally` block catches the case where the lock was *stolen while the process was running*, but at that point the damage is already done — `update_stock` and `create_order` have already been called with stale data before the lock release is attempted.

```python
# This check is too late. The writes already happened above it.
finally:
    self._stop_heartbeat()
    current = self.redis.get(lock_key)
    if current and current.decode() == token:
        self.redis.delete(lock_key)
```

The token check only gates the *delete*, not the *writes*.

---

## Required Fix: Fencing Token / Optimistic Lock at the Database Layer

The Redis lock must be treated as a *best-effort advisory lock*, not a correctness guarantee. The correctness guarantee must live at the database write boundary. The canonical pattern is an optimistic lock using the stock value as a precondition, or a monotonic fencing token (row version number):

```sql
-- Option A: Optimistic lock on current stock value
UPDATE inventory
   SET stock = :new_stock
 WHERE item_id = :item_id
   AND stock = :expected_stock;   -- ← stale zombie write produces 0 rows affected

-- Option B: Monotonic fencing token (preferred)
UPDATE inventory
   SET stock = :new_stock,
       version = version + 1
 WHERE item_id = :item_id
   AND version = :read_version;   -- ← version was incremented by Worker B; zombie fails
```

If the UPDATE affects 0 rows, the reservation must be treated as failed and the order record must not be created. The application layer must check `rowcount` and roll back accordingly.

```python
# Sketch of the corrected write path
rows_affected = self.db.update_stock_conditional(
    item_id, new_stock, expected_current=current_stock
)
if rows_affected == 0:
    # Lost the race — another worker modified stock while we were paused
    raise StaleDataError(f"Concurrent modification detected for item {item_id}")

self.db.create_order(order_id, item_id, quantity)
```

Without this, the distributed lock provides a *performance optimization* (reducing contention) but not a *safety guarantee* (preventing overselling).

## Secondary Concerns

**Instance-level heartbeat state is not thread-safe for concurrent calls.**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool), two concurrent reservations on different `item_id`s will stomp each other's `_heartbeat_active` flag. One call stopping its heartbeat will silence the other's renewal thread. The heartbeat state must be keyed per lock token, not stored as a single instance field.

**`quantity` has no lower bound validation.**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

A caller passing `quantity=0` or `quantity=-5` would either succeed vacuously or produce a negative stock write. Pydantic is used for `ReservationResult` but not for input validation on `reserve_item` itself, which the checklist incorrectly marks as satisfied.

**The test suite mocks out the exact behavior under scrutiny.**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Disabling the heartbeat in tests is reasonable for isolation, but there is no test that exercises the fencing/optimistic lock behavior, no test for concurrent writes, and no test verifying that a stale write is rejected. The test for `test_lock_not_released_if_expired` checks that `delete` is not called, but does not verify that the stale `update_stock` and `create_order` calls are also suppressed — they are not, which is the bug.

---

## What Needs to Change Before Approval

1. **Implement a fencing token or optimistic lock condition on `db.update_stock`**, and verify `rowcount` before calling `db.create_order`. This is non-negotiable for a flash sale inventory system.
2. **Move heartbeat state into a per-lock-token scope** (a local dict or a small dataclass) to eliminate the instance-level race condition under concurrent use.
3. **Add input validation** on `item_id` (non-empty string) and `quantity` (positive integer) using Pydantic or explicit guards.
4. **Add a test** that simulates a stale write scenario — mock `update_stock_conditional` returning 0 rows and assert that no order is created and the result is a failure.
