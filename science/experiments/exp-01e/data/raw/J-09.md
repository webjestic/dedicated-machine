Decision: Request Changes

Summary: The implementation has several meaningful flaws that need to be addressed before this can merge. The most critical is a thread-safety issue with the heartbeat mechanism — instance-level state (`_heartbeat_active`, `_heartbeat_thread`) makes `InventoryService` unsafe for concurrent use, which is a serious problem in any high-throughput flash sale scenario. There are also missing input validation guards, a non-atomic lock release, and test gaps that leave real failure modes untested.

---

## Detailed Review

### 🔴 Critical — Must Fix

---

**1. Instance-level heartbeat state breaks concurrent use**

```python
# Current — shared across all calls on the same instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If two requests hit the same `InventoryService` instance simultaneously (which is the norm in any web framework with a shared service layer), `_heartbeat_active = False` in one call will kill the heartbeat thread belonging to the other. This is a silent correctness failure — the lock expires mid-operation with no error raised.

```python
# Fix — scope heartbeat state to each lock acquisition
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    # ... acquire lock ...

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

**2. Lock release is not atomic**

```python
# Current — TOCTOU race between GET and DELETE
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and `DELETE`, another process can acquire the lock after expiry and you will delete their lock, not yours. This is a well-known Redis locking pitfall. The canonical fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block
self._release_script(keys=[lock_key], args=[token])
```

The same race exists in the heartbeat's `GET` + `EXPIRE` sequence and should be addressed with a similar Lua script.

---

**3. No input validation on `reserve_item`**

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw `str` and `int` with no guards. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` and the code will silently create a zero-unit order or corrupt stock values.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="item_id must not be empty")
    if quantity < 1:
        return ReservationResult(success=False, message="quantity must be at least 1")
```

Pydantic is imported and `ReservationResult` is modeled correctly, but it is only used for the *output*. Consider a `ReservationRequest` model for the input as well.

### 🟡 Moderate — Should Fix

---

**4. `db.update_stock` and `db.create_order` are not transactional**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock protects against concurrent access but does nothing for partial failure. These two operations need to be wrapped in a database transaction, or the stock decrement must be rolled back on failure.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

**5. Heartbeat thread is never joined**

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
# Thread is never joined
```

Setting the flag to `False` signals the thread to stop, but the `finally` block proceeds immediately. The heartbeat thread can still be executing `redis.expire` after the lock has been deleted and re-acquired by another process. Always join with a bounded timeout before releasing the lock.

---

**6. Bare `RuntimeError` is too broad**

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

This catches *any* `RuntimeError` thrown inside the `with` block, including bugs in `db.update_stock` or `db.create_order`. Define a specific exception class:

```python
class LockAcquisitionError(Exception):
    pass
```

And raise/catch that instead.

---

### 🔵 Minor — Worth Noting

---

**7. Test coverage gaps**

The tests mock out the heartbeat entirely, which means the most dangerous concurrent behavior is never exercised. The following cases are untested:

- `db.create_order` raises after `db.update_stock` succeeds (partial failure)
- `quantity=0` or `quantity=-1` passed to `reserve_item`
- Empty or whitespace `item_id`
- Redis connection error mid-operation

`test_lock_not_released_if_expired` is a good test, but it passes a hardcoded `b"different-token"` without accounting for the UUID generated internally — it works by coincidence of mock setup, not by design. Consider using `unittest.mock.patch('uuid.uuid4', return_value=...)` to make the token deterministic.

---

**8. `LOCK_TTL = 30` with no justification tied to actual DB SLAs**

A 30-second lock TTL is generous enough to cause real queuing pain under flash sale load. This value should either be configurable at construction time or documented with a reference to measured DB operation latency. Hard-coding it as a class variable makes it invisible to operators.
