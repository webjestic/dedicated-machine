Decision: Request Changes

Summary: This implementation contains multiple critical flaws that will cause silent data corruption and availability failures under real distributed load. The heartbeat mechanism introduces a race condition that actively undermines the safety guarantee it was designed to provide, the instance-level state management makes the service non-reentrant and unsafe under concurrency, and the database operations lack atomicity guarantees that the distributed lock was presumably meant to protect. None of these failure modes are caught by the test suite, which is structurally incapable of detecting them.

---

## Critical Findings

### 1. 🔴 Non-Atomic Lock Release — TOCTOU Race Condition

**File:** `services/inventory.py`, lines in `_acquire_lock` finally block

```python
# BROKEN: check-then-act is not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # another process can steal the lock between these two calls
```

Between the `GET` and the `DELETE`, a Stop-the-World GC pause, OS thread scheduling delay, or network hiccup gives another process a window to:

1. Observe the lock as expired (if TTL lapses)
2. Acquire it with a new token
3. Begin a legitimate critical section

Your process then resumes, sees its old token still matches (it hasn't been overwritten yet in this narrow window), and deletes the lock out from under the new legitimate owner. **This is a classic TOCTOU race and it will cause inventory double-spend.**

The fix is well-established — use a Lua script to make the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    script(keys=[lock_key], args=[token])
```

The same non-atomic pattern exists in the heartbeat's `GET` → `EXPIRE` sequence (see finding #2).

### 2. 🔴 Heartbeat Renewal Is Non-Atomic and Can Extend a Stolen Lock

**File:** `services/inventory.py`, `_beat()` inner function

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # not atomic with the GET above
```

Consider this sequence:

1. Heartbeat thread calls `GET` — sees its own token. ✅
2. GC pause / thread sleep occurs.
3. Lock TTL expires naturally. A second service instance acquires the lock with a new token and begins its critical section.
4. Heartbeat thread resumes, calls `EXPIRE` — **extends the TTL of the lock now owned by the second instance.**

The second instance's lock now has an artificially extended TTL. If the second instance later crashes, the lock will be held for up to `LOCK_TTL` additional seconds beyond what was intended, blocking all reservations for that item. More dangerously, the first instance may still be executing its critical section concurrently with the second. The lock has provided no mutual exclusion.

Fix: Use a Lua script for atomic check-and-extend:

```python
EXTEND_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
"""
```

If the extension fails (returns 0), the heartbeat must signal the main thread to abort the operation immediately, since the lock has been lost. The current design has no such abort path.

### 3. 🔴 Instance-Level Heartbeat State — Service Is Not Reentrant or Thread-Safe

**File:** `services/inventory.py`, `__init__` and `_start_heartbeat`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored as instance attributes. This means:

- **Concurrent calls to `reserve_item` on the same instance** (e.g., from a thread pool or async framework) will have the second call overwrite `_heartbeat_active = True` and then set `_heartbeat_active = False` when *it* finishes, silently killing the heartbeat of the still-running first call.
- There is no `threading.Lock` protecting these fields.
- The `_stop_heartbeat` method sets a bare boolean flag with no memory barrier guarantee.

In any realistic deployment where a single service instance handles concurrent requests, this will cause intermittent lock expiry mid-operation, leading to inventory corruption that will be nearly impossible to reproduce in testing.

Fix: Heartbeat state must be scoped to the individual lock acquisition, not the service instance. Pass a `threading.Event` into the heartbeat closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ... acquire lock ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

### 4. 🔴 Database Operations Are Not Atomic — Lock Does Not Protect What It Claims To

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is killed, or the database connection drops between these two lines:

- Stock is decremented ✅
- Order record is never created ❌

The result is **phantom inventory loss** — stock is gone, but no order exists. There is no compensating transaction, no saga, no outbox pattern. This will silently corrupt inventory counts during any partial failure, which is precisely the scenario flash sales are most likely to trigger due to load spikes.

The distributed lock provides mutual exclusion between *processes*, but it provides zero atomicity guarantees for the *database operations within the lock*. These are orthogonal concerns and both must be addressed.

Fix: Wrap both operations in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

Or, if the DB abstraction doesn't support transactions, implement an outbox pattern or use a compare-and-swap update (`UPDATE inventory SET stock = %s WHERE item_id = %s AND stock = %s`) to at minimum make the stock update idempotent and detectable.

### 5. 🟡 Heartbeat Thread Is Never Joined — Daemon Thread Suppresses Errors Silently

**File:** `services/inventory.py`, `_stop_heartbeat`

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
    # thread is never joined
```

The heartbeat thread is created as `daemon=True`, which means:

- Exceptions raised inside `_beat()` (e.g., `redis.ConnectionError`) are silently swallowed with no logging and no propagation.
- There is no guarantee the thread has actually stopped before the lock is released. The thread may execute one final `EXPIRE` call *after* the lock has been deleted and re-acquired by another process (see finding #2 for consequences).

Fix: Join the thread with a timeout after setting the stop flag, and add exception handling inside `_beat()` that logs errors and exits cleanly:

```python
def _beat() -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            result = extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if result == 0:
                logger.error("Lost lock ownership during heartbeat key=%s", lock_key)
                stop_event.set()
                break
        except redis.RedisError as e:
            logger.error("Heartbeat Redis error key=%s: %s", lock_key, e)
```

### 6. 🟡 Test Suite Cannot Detect Any of the Above Failures

**File:** `tests/test_inventory.py`

The tests mock out the heartbeat entirely (`svc._start_heartbeat = MagicMock()`) and use a single-threaded, synchronous mock for Redis. This means:

- No test exercises the TOCTOU race in lock release.
- No test verifies that concurrent calls to `reserve_item` on the same instance don't corrupt each other's heartbeat state.
- No test simulates a crash between `update_stock` and `create_order`.
- `test_lock_not_released_if_expired` tests the *intent* of the ownership check but cannot detect that the check-then-delete is non-atomic.

At minimum, add:

```python
def test_concurrent_reservations_do_not_corrupt_state():
    """Two threads calling reserve_item simultaneously must not interfere."""
    # Use a real threading.Barrier to synchronize entry into the critical section
    # and verify both heartbeats remain active for their respective lock durations.

def test_partial_db_failure_does_not_leak_stock():
    """If create_order raises, update_stock must be rolled back."""
    db.create_order.side_effect = Exception("DB connection lost")
    result = service.reserve_item("item_001")
    db.update_stock.assert_not_called()  # or verify rollback was called
```
