**Decision: Request Changes**

**Summary:** The heartbeat thread addresses lock expiry under *normal* conditions, but it does not close the fundamental safety gap: if the lock expires between the heartbeat's last renewal and the execution of `update_stock`/`create_order` — due to a GIL pause, a slow DB call, a network hiccup to Redis, or a process stall — the writes proceed without any guarantee that the lock is still held. There are no fencing tokens, no conditional writes, and no version checks on the database side, so two processes can corrupt inventory simultaneously in exactly the scenario this code is designed to prevent. There is also a secondary concurrency bug in the heartbeat implementation that makes the situation worse.

---

## Detailed Findings

### 1. The Core Issue: The Lock Does Not Guard the Write

This is the architectural problem that blocks approval.

The heartbeat keeps the lock *probably* alive. It does not keep it *certainly* alive. Between any two lines of Python, the process can stall — GC pause, OS scheduler, slow `db.get_stock()` call that holds the GIL, network timeout to Redis that blocks the heartbeat thread. When the process resumes, the lock TTL may have elapsed, another process may have acquired it, and that other process may be mid-write.

```python
# At this point the lock *was* held. Is it still?
# Nothing here answers that question.
self.db.update_stock(item_id, new_stock)   # ← unguarded write
self.db.create_order(order_id, item_id, quantity)  # ← unguarded write
```

The heartbeat renewal is a probabilistic improvement, not a safety guarantee. The PR description frames it as solving the problem ("ensures the lock does not expire during slow DB operations"), but that framing is incorrect. It reduces the *window* of vulnerability; it does not close it.

**What is actually needed** is one of:

**Option A — Fencing tokens with server-side enforcement (preferred):**
The lock server issues a monotonically increasing token on each acquisition. Every write to the database carries that token. The database (or a compare-and-swap layer in front of it) rejects writes with a token lower than the last committed one. A stale process cannot corrupt state because its token is rejected, regardless of what its local lock state says.

```python
# Sketch — the DB layer must enforce the fencing check
token_version = self.redis.get_lock_version(lock_key)  # monotonic counter
self.db.update_stock(item_id, new_stock, fence=token_version)
# db raises if a higher fence has already been committed
```

**Option B — Optimistic locking / compare-and-swap in the database:**
Read a version number with the stock value. Write only if the version is unchanged. The distributed lock reduces contention; the CAS prevents corruption if the lock fails.

```python
stock, version = self.db.get_stock_with_version(item_id)
# ... check stock ...
updated = self.db.update_stock_if_version(item_id, new_stock, expected_version=version)
if not updated:
    # lost the race despite holding the lock — retry or fail clean
```

**Option C — Idempotent writes keyed on order_id:**
Generate `order_id` before acquiring the lock. Make `create_order` and `update_stock` idempotent on that key. A duplicate write from a stale process is a no-op rather than a corruption event.

Without one of these, the distributed lock is, as Kleppmann puts it, an optimisation, not a correctness mechanism.

### 2. Shared Heartbeat State Breaks Under Concurrent Calls

`_heartbeat_active` and `_heartbeat_thread` are instance variables, not scoped to a single lock acquisition. If `reserve_item` is ever called concurrently on the same `InventoryService` instance — which is the normal case for any service object held in a DI container or module-level singleton — the second call's `_start_heartbeat` overwrites `_heartbeat_active`, and the first call's `_stop_heartbeat` kills the second call's heartbeat.

```python
# Instance A calls reserve_item → _heartbeat_active = True, thread_1 starts
# Instance call B calls reserve_item → _heartbeat_active = True, thread_2 starts
# Instance A's finally block → _stop_heartbeat() sets _heartbeat_active = False
# thread_2 exits on next iteration — B's lock is now unrenewed
```

The fix is to scope heartbeat state to the lock acquisition context, not the service instance. Pass a `threading.Event` into the heartbeat thread and signal it on exit.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ... acquire ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        # ... release ...
```

### 3. Heartbeat Has a TOCTOU Race on Token Check

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← lock can expire or be stolen here
```

Between `GET` and `EXPIRE`, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This should be a single atomic operation via a Lua script:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

---

### 4. Lock Release Has the Same TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)  # ← another process can acquire between GET and DELETE
```

This is the canonical case for atomic release via Lua script. `redis-py` ships with `redis.lock.Lock` which handles this correctly. The same script pattern applies:

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 5. The Test Suite Does Not Test the Failure Mode That Matters

`test_lock_not_released_if_expired` tests that the lock is not *deleted* when the token has changed. That is correct behaviour for the release path. But there is no test for the scenario that actually causes data corruption: the lock expires *during* the write, a second process acquires it, and both processes call `update_stock` concurrently. The test suite validates the happy path and the lock-contention path; it does not validate the invariant the lock is supposed to enforce.

A meaningful test would mock the DB calls to simulate a stall, confirm that a concurrent reservation attempt either blocks or fails clean, and assert that `update_stock` is called exactly once with the correct value. Without fencing tokens or CAS, this test cannot be made to pass reliably — which is itself the signal that the architecture needs to change.

---

### 6. Minor: `quantity` Is Not Validated

`quantity` is accepted as a plain `int` with no lower bound check. A caller passing `quantity=0` or `quantity=-5` will silently succeed and *increase* stock. This should be caught at the service boundary:

```python
if quantity < 1:
    return ReservationResult(success=False, message="Quantity must be at least 1")
```

## Summary of Required Changes

| # | Severity | Issue |
|---|----------|-------|
| 1 | **Blocking** | No fencing tokens or CAS — writes are unguarded if lock expires |
| 2 | **Blocking** | Heartbeat and stop state is shared across concurrent calls on the same instance |
| 3 | High | Heartbeat token check and `EXPIRE` are not atomic |
| 4 | High | Lock release `GET`/`DEL` is not atomic |
| 5 | Medium | Test suite does not cover the concurrent write corruption scenario |
| 6 | Low | `quantity` accepts zero and negative values |

Items 1 and 2 must be resolved before this can be approved. Items 3 and 4 should be resolved in the same pass — the non-atomic operations undermine the ownership verification that the rest of the code depends on.
