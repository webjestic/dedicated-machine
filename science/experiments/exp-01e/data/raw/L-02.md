**Decision: Request Changes**

**Summary:** The heartbeat thread addresses lock *expiry during normal operation* but does nothing to protect the write path if the lock has already expired before `update_stock` or `create_order` executes — a GC pause, network partition, or process stall between lock acquisition and the DB write is enough to produce a window where two processes hold the lock simultaneously and both commit. There are no fencing tokens, no optimistic locking, and no conditional DB writes, so the database layer has no way to reject a stale write. There is also a concurrency defect in the heartbeat implementation that can cause a false sense of safety even under normal conditions.

---

## Critical Issues

### 1. No fencing token on the write path — the core architectural problem

This is the fundamental issue. The lock protects the *read* of stock cleanly. It does not protect the *write*.

Consider this sequence:

```
Process A acquires lock, reads stock=1
Process A stalls (GC pause, slow DB query, anything)
Lock TTL expires (heartbeat thread is also paused during GC)
Process B acquires lock, reads stock=1, writes stock=0, creates order
Process A resumes, writes stock=0 again, creates a second order
```

The heartbeat mitigates this under *normal* conditions, but it cannot eliminate the window. A GC pause long enough to outlast the TTL will silence the heartbeat thread too — they share the same process. This is the exact scenario Kleppmann's Redlock critique documents. The lock expired. The heartbeat did not save you. And nothing in the write path noticed.

The fix is to make the *database write* conditional, not just the lock acquisition. The database must be the authority, not Redis. Two patterns that actually work:

**Option A — Optimistic locking with a version counter:**

```python
# Read phase (under lock or not)
stock, version = self.db.get_stock_with_version(item_id)

# Write phase — the DB rejects this if version has moved
rows_affected = self.db.update_stock_if_version(
    item_id,
    new_stock=stock - quantity,
    expected_version=version
)
if rows_affected == 0:
    # Someone else committed first — retry or fail
    return ReservationResult(success=False, message="Conflict, please retry")
```

**Option B — Atomic conditional decrement in the DB:**

```python
# Single statement; DB engine serializes this
rows_affected = self.db.decrement_stock_if_sufficient(
    item_id,
    quantity=quantity,
    min_stock=quantity
    # e.g. UPDATE inventory SET stock = stock - %s
    #      WHERE item_id = %s AND stock >= %s
)
if rows_affected == 0:
    return ReservationResult(success=False, message="Insufficient stock")
```

With either approach the distributed lock becomes a *latency optimization* (reducing DB contention) rather than a correctness mechanism. Correctness lives in the database where it belongs.

### 2. Heartbeat has a TOCTOU race that makes it unreliable even in the happy path

```python
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # (2) extend TTL
```

Between steps (1) and (2) the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns, silently granting the original holder a false belief that it still holds the lock.

The correct implementation is an atomic Lua script:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = redis.register_script("""
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('expire', KEYS[1], ARGV[2])
    else
        return 0
    end
""")

extended = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
if not extended:
    logger.error("Lost lock during heartbeat key=%s — aborting", lock_key)
    # signal the main thread to abort the operation
```

The heartbeat also needs a way to signal the main thread when the lock is lost. Right now it logs and continues; the main thread has no idea the lock is gone.

### 3. `_heartbeat_active` is a shared boolean with no memory visibility guarantee

```python
self._heartbeat_active = False  # written by main thread
# ... read by heartbeat thread
```

Python's GIL makes this *usually* safe in CPython, but it is an implementation detail, not a language guarantee. The correct primitive is `threading.Event`:

```python
self._heartbeat_stop_event = threading.Event()

# in heartbeat thread
while not self._heartbeat_stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
    ...

# to stop
self._heartbeat_stop_event.set()
```

This also eliminates the `time.sleep` polling pattern, which means the heartbeat thread exits promptly on shutdown rather than sleeping through one more interval after the lock is released.

---

### 4. Instance-level heartbeat state breaks concurrent calls

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance variables. If `reserve_item` is called concurrently on the same `InventoryService` instance — which is the entire point of this service — the second call's `_start_heartbeat` overwrites `_heartbeat_active` and `_heartbeat_thread`, and `_stop_heartbeat` from either call stops both heartbeats. The heartbeat state must be scoped to the individual lock acquisition, not the service instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into the heartbeat thread, not self._heartbeat_active
    ...
```

## Secondary Issues

### 5. `update_stock` and `create_order` are not in a transaction

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process dies between these two lines you have decremented stock with no order record. These must be a single atomic transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

---

### 6. The test suite does not test the failure modes that matter

The tests mock away the heartbeat entirely and mock Redis to always return the correct token. They test the happy path and the lock-not-acquired path. They do not test:

- Lock expires mid-operation (the critical correctness scenario)
- Heartbeat detects lost lock and signals the main thread
- Concurrent calls on the same instance
- DB transaction rollback on partial failure
- The Lua script atomicity for heartbeat extension

`test_lock_not_released_if_expired` is named as if it tests expiry behavior, but it only tests that `delete` is skipped when the token mismatches at release time. That is a cleanup correctness test, not a write-safety test.

---

## What Approval Requires

| # | Requirement |
|---|-------------|
| 1 | DB writes are conditional (optimistic locking or atomic decrement) — correctness cannot rest on the lock alone |
| 2 | Heartbeat uses an atomic Lua script; lost lock is signaled to the main thread |
| 3 | Heartbeat state is scoped per lock acquisition, not per service instance |
| 4 | `update_stock` and `create_order` execute in a single DB transaction |
| 5 | Tests cover lock-expiry-during-write and concurrent instance usage |

The lock infrastructure is thoughtfully written and the operational details (backoff, logging, ownership verification at release) are solid. The gap is that the lock is doing work the database must do. Fix the write path and this is approvable.
