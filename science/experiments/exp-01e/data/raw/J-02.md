Decision: Request Changes

Summary: The core locking logic and heartbeat design contain several critical flaws that make this unsafe for production under concurrent load. The most serious issue is that `_heartbeat_active` and `_heartbeat_thread` are instance-level fields rather than call-scoped, meaning concurrent calls to `reserve_item` on the same `InventoryService` instance will race on those shared fields — one call's `_stop_heartbeat()` will silently kill another call's heartbeat, potentially allowing a live lock to expire mid-operation. Additionally, the lock release in the `finally` block is not atomic: the `GET` and `DELETE` are two separate Redis commands, creating a TOCTOU window where another process could acquire the lock between the check and the delete; this must be replaced with a Lua script or a Redis pipeline with a watch. There are also secondary concerns around missing input validation on `item_id` and `quantity` at the service boundary, the absence of database transaction guarantees between `update_stock` and `create_order`, and test coverage gaps that mock away the heartbeat entirely rather than verifying its correctness.

---

### Detailed Findings

#### 🔴 Critical — Shared Heartbeat State Breaks Concurrency

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance attributes. If two threads call `reserve_item` concurrently on the same service instance (which is the entire point of this service), they share `_heartbeat_active`. The second call's `_stop_heartbeat()` sets it to `False`, which terminates the first call's heartbeat thread. The first call's lock then expires while its DB operations are still running, allowing a third process to acquire the lock and read stale stock.

**Required fix:** Heartbeat state must be scoped to each individual lock acquisition, not to the instance.

```python
# Suggested approach — pass a stop event into the thread
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    stop_event = threading.Event()
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

#### 🔴 Critical — Non-Atomic Lock Release (TOCTOU Race)

```python
# Current code in _acquire_lock finally block
current = self.redis.get(lock_key)          # Step 1
if current and current.decode() == token:
    self.redis.delete(lock_key)             # Step 2 — gap here
```

Between `GET` and `DELETE`, another process can acquire the lock. This is a well-known distributed locking antipattern. The canonical fix is a Lua script executed atomically on the Redis server.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    released = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning(
            "Lock not released — already expired or taken key=%s", lock_key
        )
```

#### 🔴 Critical — No Database Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. The distributed lock does not protect against this — it only serializes access, it does not provide atomicity across DB writes. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the DB layer, but the requirement is non-negotiable for a financial/inventory operation.

#### 🟡 High — Missing Input Validation at Service Boundary

The checklist claims "Input validation via Pydantic," but `reserve_item` accepts raw primitives with no validation. Pydantic is only used on the *output* model. There is nothing preventing `quantity=0`, `quantity=-5`, or an `item_id` that is an empty string or a 10,000-character payload.

#### 🟡 High — Heartbeat Thread Is Fully Mocked Out in Tests

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

This means the heartbeat — the most novel and risky piece of this PR — has zero test coverage. At minimum there should be integration-level tests that verify the heartbeat extends the TTL and that it stops cleanly after the context manager exits.

---

#### 🟠 Medium — `time.sleep` in Retry Loop Blocks the Calling Thread

For a service expected to handle flash sale load, blocking a thread for up to `0.1 + 0.2 = 0.3` seconds per contended reservation is a throughput problem. If this service runs under an async framework (FastAPI, etc.), sleeping in a sync function will block the event loop. Consider whether this should be an async service with `asyncio.sleep`, or whether the retry should be offloaded.

---

#### 🟠 Medium — Daemon Thread Swallows Exceptions Silently

```python
self._heartbeat_thread = threading.Thread(target=_beat, daemon=True)
```

Any exception inside `_beat` (e.g., a Redis connection error) is silently discarded. The lock will then expire without renewal and without any alerting. The heartbeat should catch exceptions, log them at ERROR level, and ideally signal the parent context that the lock may be compromised.

```python
def _beat(stop_event: threading.Event) -> None:
    while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        try:
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)
            else:
                logger.error(
                    "Heartbeat lost lock ownership key=%s — aborting renewal",
                    lock_key
                )
                break
        except redis.RedisError as exc:
            logger.error("Heartbeat Redis error key=%s: %s", lock_key, exc)
```
