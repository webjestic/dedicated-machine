Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under slow I/O, but it contains a fundamental and silent failure mode that no amount of thread-level sophistication can close: **it provides zero protection against a stop-the-world process pause**. A JVM-style GC pause, a Python GIL-induced stall, a VM live-migration, or an OS scheduler preemption will freeze *every thread in the process simultaneously* — including the heartbeat thread. When the process resumes, the TTL has already elapsed, the lock has expired, a second worker has acquired it and committed its write, and the first worker continues executing against a stale read with no awareness that it has lost lock ownership. The result is a classic **Zombie Leader** scenario: two workers both believe they hold exclusive access during overlapping critical sections, and the database receives two conflicting writes that together produce a corrupted stock count — one item sold twice, silently, with no error surfaced to either caller.

---

### The Specific Failure Sequence

```
Worker A: acquires lock, reads current_stock = 5
          ← PROCESS PAUSE (GC / VM migration / OOM stall) →
          heartbeat thread also frozen — cannot extend TTL
          lock TTL expires after 30s
Worker B: acquires lock, reads current_stock = 5
          writes new_stock = 4, releases lock
Worker A: resumes, calculates new_stock = 5 - 1 = 4
          calls db.update_stock("item_001", 4)  ← stale write succeeds
Result:   stock = 4, but two reservations were issued against stock = 5
          one item has been double-sold
```

The heartbeat thread is not the problem — it is simply irrelevant to this failure class. The lock TTL mechanism as a whole cannot protect against pauses that are longer than the TTL, because the process cannot observe its own suspension.

---

### Root Cause

The critical section performs a **read-modify-write** sequence:

```python
current_stock = self.db.get_stock(item_id)   # read
new_stock = current_stock - quantity          # modify
self.db.update_stock(item_id, new_stock)      # write  ← no guard here
```

The `update_stock` call is unconditional. It does not verify that the database state still matches what was read. Any process pause between the read and the write invalidates the assumption of exclusivity, and the write will silently succeed against stale data.

---

### Required Fix: Fencing Token / Optimistic Locking at the Database Write

The canonical solution is to make the write **conditional on the value that was read**. The database becomes the final arbiter of correctness, not the distributed lock.

```python
# Require that stock hasn't changed since we read it
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_stock=current_stock   # fencing condition
)

if rows_affected == 0:
    # Another writer committed between our read and our write.
    # The lock was lost during a pause. Abort and surface the failure.
    raise StaleReadError(
        f"Stock for {item_id} changed between read and write — "
        "lock was likely lost during a process pause"
    )
```

The underlying SQL must use the conditional predicate:

```sql
UPDATE inventory
   SET stock = %(new_stock)s
 WHERE item_id = %(item_id)s
   AND stock   = %(expected_stock)s   -- fencing condition
```

If `rows_affected == 0`, the zombie write is rejected at the database level. No silent data corruption occurs. The caller receives a clean failure it can handle.

A monotonic **version column** (row-level fencing token) is the more robust variant, as it guards against the ABA problem where stock coincidentally returns to the same value:

```sql
ALTER TABLE inventory ADD COLUMN version BIGINT NOT NULL DEFAULT 0;

UPDATE inventory
   SET stock   = %(new_stock)s,
       version = version + 1
 WHERE item_id  = %(item_id)s
   AND version  = %(read_version)s   -- must match the version we observed
```

### Secondary Issues

| # | Location | Issue |
|---|----------|-------|
| 1 | `InventoryService.__init__` | `_heartbeat_active` and `_heartbeat_thread` are instance-level fields, not per-lock-acquisition fields. Concurrent calls to `reserve_item` on the same service instance will race on these shared fields — one call's `_stop_heartbeat()` will terminate the other's heartbeat. The heartbeat state must be scoped to the lock context, not the service instance. |
| 2 | `_acquire_lock` / `_stop_heartbeat` | `_stop_heartbeat` sets the flag but does not join the thread. The heartbeat thread can continue executing for up to one `HEARTBEAT_INTERVAL` (10s) after the lock is released, potentially extending a lock that has already been handed to another worker. |
| 3 | `_acquire_lock` finally block | The `redis.get` + `redis.delete` sequence is not atomic. A lock expiry between the `get` and the `delete` allows deletion of a lock owned by another worker. This should use a Lua script for atomic compare-and-delete — a well-known Redis pattern. |
| 4 | `tests/test_inventory.py` | The test suite mocks out `_start_heartbeat` and `_stop_heartbeat` entirely, which means the heartbeat logic has zero test coverage. The lock release race (issue 3) is also untested. |
| 5 | `quantity` parameter | No validation that `quantity > 0`. A caller passing `quantity=0` or `quantity=-5` will produce nonsensical results that pass the stock check silently. |

### Lua Script for Atomic Lock Release (Issue 3)

```lua
-- KEYS[1] = lock_key, ARGV[1] = token
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

### Summary of Required Changes Before Approval

1. **[Blocking]** Add a conditional `UPDATE ... WHERE stock = %(expected_stock)s` (or version-based fencing token) to `update_stock`. The distributed lock alone is insufficient to prevent double-writes across process pauses.
2. **[Blocking]** Scope heartbeat state (`_heartbeat_active`, `_heartbeat_thread`) to the lock acquisition context, not the service instance, to prevent concurrent-call interference.
3. **[Blocking]** Replace the non-atomic `get` + `delete` lock release with a Lua script.
4. **[Required]** Join the heartbeat thread in `_stop_heartbeat` to prevent post-release TTL extensions.
5. **[Required]** Add `quantity > 0` validation with a clear error result.
6. **[Required]** Add test coverage for the heartbeat thread behavior and the conditional write path.
