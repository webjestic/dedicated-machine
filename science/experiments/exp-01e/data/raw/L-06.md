**Decision: Request Changes**

**Summary:** The heartbeat prevents *normal* expiry, but it does not prevent the lock from expiring during a GC pause, network partition, or thread starvation event that occurs between the last successful heartbeat and the database writes. When that happens, `db.update_stock` and `db.create_order` execute against a resource this process no longer holds exclusively — the lock has become a suggestion, not a guarantee. There is no fencing token or conditional write mechanism guarding the database operations themselves, so two processes can concurrently commit conflicting stock decrements. The test suite validates the happy path and the lock-release skip, but it does not test the scenario that actually matters: writes proceeding after the lock has silently expired.

## Detailed Findings

### 1. The Core Safety Violation: Unguarded Writes (Critical)

This is the issue that would cause a production incident.

The heartbeat extends the lock TTL every 10 seconds. But consider this sequence:

1. Process A holds the lock. Heartbeat last fired at T=0, extending TTL to T=30.
2. At T=25, the Python process experiences a GC pause, thread starvation, or the heartbeat thread is delayed by OS scheduling.
3. At T=30, the lock expires in Redis.
4. At T=31, Process B acquires the lock. It reads `current_stock = 1`.
5. At T=32, Process A's pause ends. It is past the `if current_stock < quantity` check. It calls `db.update_stock(item_id, 0)` and `db.create_order(...)`.
6. At T=33, Process B calls `db.update_stock(item_id, 0)` and `db.create_order(...)`.

Both orders are created. Stock goes negative. The lock did not protect anything.

The heartbeat makes this *less likely*, not *impossible*. For flash sale inventory — where the entire point is that demand exceeds supply — "less likely" is not acceptable.

```python
# What happens today: the lock boundary is advisory
with self._acquire_lock(lock_key):
    current_stock = self.db.get_stock(item_id)   # reads under lock
    # ... lock may expire here, invisibly ...
    self.db.update_stock(item_id, new_stock)      # write is unguarded
    self.db.create_order(order_id, item_id, quantity)  # unguarded
```

The fix depends on what `self.db` is, but the principle is the same in all cases: **the storage layer must reject a stale write, not just the application layer**.

**Option A — Database-level optimistic concurrency (preferred):**

```python
# Pass the stock value we read; DB rejects the write if it has changed
rows_affected = self.db.update_stock_if_unchanged(
    item_id,
    expected_stock=current_stock,
    new_stock=new_stock,
)
if rows_affected == 0:
    # Someone else modified stock while we held (or lost) the lock
    raise StockConflictError("Stock changed during reservation; retry")
```

This is a standard optimistic lock / compare-and-swap at the DB level:

```sql
UPDATE inventory
SET stock = :new_stock
WHERE item_id = :item_id
  AND stock = :expected_stock  -- the value we read
```

**Option B — Monotonic fencing token passed to storage:**

```python
# The lock token (or a sequence number from Redis) is passed through
# to every write. The DB or storage layer rejects writes with a
# token older than the last committed token for that item.
with self._acquire_lock(lock_key) as token:
    self.db.update_stock(item_id, new_stock, fence=token)
```

This is the approach Kleppmann describes in his Redlock critique. It requires the storage layer to participate, but it is the only approach that is safe against clock skew and process pauses of arbitrary duration.

**Option C — Atomic Redis operations (if Redis is the system of record):**

```python
# Use a Lua script to make read-check-decrement atomic in Redis,
# then persist asynchronously with reconciliation
script = """
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil then return -1 end
if stock < tonumber(ARGV[1]) then return -2 end
redis.call('SET', KEYS[1], stock - tonumber(ARGV[1]))
return stock - tonumber(ARGV[1])
"""
```

### 2. Heartbeat Thread Has a Race on `_heartbeat_active` (Moderate)

`_heartbeat_active` is a plain Python `bool` shared between threads with no synchronization primitive.

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False   # written by main thread

def _beat() -> None:
    while self._heartbeat_active:    # read by heartbeat thread
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:   # read again — window between the two reads
            ...
            self.redis.expire(lock_key, self.LOCK_TTL)
```

The double-check pattern here does not eliminate the race: `_heartbeat_active` can be set to `False` (lock released, lock deleted from Redis) between the `while` check and the `self.redis.expire` call. The heartbeat then extends a lock that has already been intentionally released, potentially re-granting it to a process that no longer holds it logically.

```python
# Use threading.Event instead
self._stop_event = threading.Event()

def _beat() -> None:
    while not self._stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
        # Atomic ownership check + extend via Lua script
        extended = self.redis.eval(extend_script, 1, lock_key, token, self.LOCK_TTL)
        if not extended:
            logger.warning("Lost lock ownership during heartbeat key=%s", lock_key)
            self._stop_event.set()
            break

def _stop_heartbeat(self) -> None:
    self._stop_event.set()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=2)
```

Also note: the heartbeat's ownership check (`GET` then `EXPIRE`) is itself non-atomic. Use a Lua script:

```lua
-- extend_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

### 3. `InventoryService` Is Not Safe for Concurrent Use (Moderate)

`_heartbeat_active` and `_heartbeat_thread` are instance-level fields. If two threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the normal case for a web service), they share and overwrite these fields.

```python
# Thread 1 calls reserve_item("item_001") — sets _heartbeat_active = True
# Thread 2 calls reserve_item("item_002") — sets _heartbeat_active = True
# Thread 1 finishes — sets _heartbeat_active = False
# Thread 2's heartbeat is now dead, but Thread 2 doesn't know
```

The heartbeat state must be scoped to the lock acquisition, not to the service instance. Pass it through a local object or closure:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()   # local to this acquisition
    ...
    self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        ...
```

### 4. Lock Release Is Non-Atomic (Low–Moderate)

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between `GET` and `DELETE`, the lock can expire and be acquired by another process. You then delete their lock. This is the classic TOCTOU on lock release. Use the standard Lua script:

```lua
-- release_lock.lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

This is well-known enough that it should be in any Redis distributed lock implementation. The test `test_lock_not_released_if_expired` validates the intent but not the atomicity.

---

### 5. Test Coverage Misses the Failure Mode That Matters (Moderate)

The tests mock out the heartbeat entirely and never simulate lock expiry during a write. The test named `test_lock_not_released_if_expired` tests lock *release* behavior, not write safety after expiry. There is no test that verifies the system behaves correctly when the lock expires between `get_stock` and `update_stock`.

At minimum, add:

```python
def test_writes_rejected_if_lock_expired_mid_operation(service, db):
    """
    Simulate lock expiry between stock read and stock write.
    The update must be rejected or the conflict must be detected.
    Without a fencing mechanism, this test is expected to FAIL
    and documents the known safety gap.
    """
    # After get_stock returns, simulate another process taking the lock
    # and decrementing stock to 0
    call_count = 0
    def update_stock_side_effect(item_id, new_stock):
        # In a real scenario, the DB would enforce expected_stock
        # This test documents that currently there is no such check
        pass

    db.update_stock.side_effect = update_stock_side_effect
    # ... assert that double-spend is detected
```
