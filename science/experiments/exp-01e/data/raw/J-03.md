Decision: Request Changes

Summary: The core locking logic contains a critical race condition in the lock-release path — the `GET` then `DEL` sequence is non-atomic, meaning another process could steal the lock between those two operations, causing a legitimate owner to delete a lock it no longer holds. Additionally, the heartbeat implementation uses instance-level state (`_heartbeat_active`, `_heartbeat_thread`) rather than per-lock state, making the service non-reentrant and unsafe under any concurrent call pattern. Input validation is also absent at the `reserve_item` boundary despite the checklist claiming otherwise.

---

## Detailed Findings

### 🔴 Critical — Non-Atomic Lock Release (Race Condition)

```python
# services/inventory.py  lines in _acquire_lock finally block
current = self.redis.get(lock_key)          # <-- window opens here
if current and current.decode() == token:
    self.redis.delete(lock_key)             # <-- another owner possible by now
```

Between `GET` and `DEL`, a concurrent process can acquire the lock after TTL expiry. You would then delete their lock, not yours. This is a well-documented failure mode for Redis distributed locks. The canonical fix is a Lua script that makes the check-and-delete atomic:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block — replaces the GET/DEL pair
self._release_script(keys=[lock_key], args=[token])
logger.info("Lock released key=%s", lock_key)
```

The same race exists in the heartbeat's `GET` + `expire` sequence and needs the same treatment:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""
```

### 🔴 Critical — Heartbeat State is Instance-Level, Not Per-Lock

```python
# All of these are on self, not scoped to a single lock acquisition
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `reserve_item` is called concurrently (e.g., two threads hit the same service instance), the second call sets `_heartbeat_active = True`, overwriting the first. When either call finishes and sets `_heartbeat_active = False`, both heartbeats die. This is a silent correctness failure — the lock can expire mid-operation with no error raised.

Fix by scoping heartbeat state to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()          # per-acquisition, not per-instance
    acquired = False

    # ... retry loop unchanged ...

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)

def _start_heartbeat(
    self, lock_key: str, token: str, stop_event: threading.Event
) -> threading.Thread:
    def _beat() -> None:
        while not stop_event.wait(timeout=self.HEARTBEAT_INTERVAL):
            result = self._extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
            if not result:
                logger.warning("Heartbeat lost ownership key=%s", lock_key)
                break
    t = threading.Thread(target=_beat, daemon=True)
    t.start()
    return t
```

Note also that `stop_event.wait(timeout=...)` is preferable to `time.sleep` because it responds immediately to the stop signal rather than waiting out the full sleep interval.

### 🟡 High — No Input Validation on `reserve_item`

The checklist marks "Input validation via Pydantic" as complete, but `reserve_item` accepts raw `str` and `int` parameters with zero validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will silently attempt to decrement stock by a nonsensical amount.

```python
# Add a request model and validate at the boundary
class ReservationRequest(BaseModel):
    item_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    quantity: int = Field(..., ge=1, le=1000)

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    try:
        req = ReservationRequest(item_id=item_id, quantity=quantity)
    except ValidationError as e:
        logger.warning("Invalid reservation request: %s", e)
        return ReservationResult(success=False, message="Invalid request parameters")
    # proceed with req.item_id, req.quantity
```

### 🟡 High — DB Operations Outside Transaction Boundary

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds but `create_order` raises, stock is decremented with no corresponding order record. The Redis lock provides mutual exclusion but does not provide atomicity for the DB writes. These two calls must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not expose transactions, that interface needs to be extended before this service can be considered safe.

---

### 🟡 Medium — Test Coverage Gaps

The tests mock out the heartbeat entirely (`svc._start_heartbeat = MagicMock()`). That is acceptable for unit tests, but there is no integration-level test that verifies the heartbeat actually extends the lock, or that the Lua release script behaves correctly under a stolen-lock scenario. Given that the heartbeat and atomic release are the two most critical correctness properties of this service, they need dedicated tests.

The `test_lock_not_released_if_expired` test also has a logical gap — it mocks `redis.get` to return `b"different-token"` but the UUID generated inside `_acquire_lock` will never equal `"test-token"` anyway, so the test passes vacuously regardless of implementation.

---

### 🟢 Acknowledged as Good

- Daemon thread usage is correct — won't block process shutdown.
- `nx=True, ex=self.LOCK_TTL` on `SET` is the correct atomic acquire primitive.
- Exponential backoff is implemented correctly.
- `ReservationResult` as a Pydantic response model is clean.
- Structured logging with key/value pairs is consistent and useful.
- Lock key namespacing (`inventory:lock:{item_id}`) is appropriate.

---

## Required Before Approval

1. Replace `GET` + `DEL` with an atomic Lua release script.
2. Replace `GET` + `expire` in heartbeat with an atomic Lua extend script.
3. Scope heartbeat state per lock acquisition, not per instance.
4. Add `ReservationRequest` Pydantic validation at the `reserve_item` boundary.
5. Wrap the two DB writes in a transaction.
6. Fix or replace the vacuous `test_lock_not_released_if_expired` test.
