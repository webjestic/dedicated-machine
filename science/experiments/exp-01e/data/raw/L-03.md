**Decision: Request Changes**

**Summary:** The heartbeat thread addresses lock expiry under *normal* conditions, but it cannot prevent expiry caused by a GIL pause, a network partition, a slow Redis round-trip, or a process stall — and when expiry does happen, nothing stops the database writes from executing anyway. The critical write path (`update_stock` / `create_order`) carries no fencing token, so a lock that expired mid-operation is indistinguishable from a lock that is still held: both result in an unguarded write. There is also a secondary concurrency defect in the heartbeat implementation itself that makes the safety story worse than it first appears.

---

## The Core Problem: The Lock Does Not Guard the Write

The PR description frames the heartbeat as a safety mechanism. It is not — it is a liveness optimization. The distinction matters enormously.

A heartbeat answers the question: *will the lock expire during a normal operation?* It does not answer the question: *if the lock has already expired, will the write be blocked?* Nothing in this codebase answers that second question, and that is the question that determines whether this is a lock or a suggestion.

Consider the following sequence, which is not exotic — it can happen on any loaded JVM or CPython process:

```
t=0   Process A acquires lock, heartbeat starts
t=10  Heartbeat fires, extends TTL to t=40
t=15  Process A enters db.get_stock() — DB is slow under flash sale load
t=20  Heartbeat fires, extends TTL to t=50
t=25  Process A thread is descheduled (GIL contention, GC, kernel scheduler)
t=50  Lock expires — heartbeat is also paused because the process is stalled
t=51  Process B acquires the lock, reads stock=10
t=52  Process A resumes, also holds stock=10 in memory (read at t=15)
t=53  Process A calls db.update_stock(item_id, 9)   ← unguarded
t=54  Process B calls db.update_stock(item_id, 9)   ← also unguarded
```

Both writes succeed. Stock is decremented once. Two orders are created. This is the oversell scenario the lock was meant to prevent.

The heartbeat cannot close this window because the heartbeat thread is inside the same process that stalled. A thread cannot extend a lock that has already expired before the thread woke up.

---

## What the Fix Looks Like

The standard solution is a **fencing token** — a monotonically increasing integer issued by the lock server at acquisition time, passed through every write, and enforced by the storage layer.

```python
# Conceptual shape — implementation depends on your DB layer

# At lock acquisition:
token, fence = self._acquire_lock(lock_key)
# fence is a sequence number from Redis (e.g. via INCR on a fencing key)

# At write time:
self.db.update_stock(item_id, new_stock, fence=fence)
self.db.create_order(order_id, item_id, quantity, fence=fence)

# In the DB layer:
# UPDATE inventory SET stock = %s
# WHERE item_id = %s AND last_fence < %s   ← rejects stale writers
```

If Process A's fence is 42 and Process B subsequently acquires the lock with fence 43, then when Process A's stale write arrives, the DB rejects it because `42 < 43` is false — the condition `last_fence < 42` fails. Process B's write with fence 43 succeeds. Inventory is correct.

Redis does not natively issue fencing tokens, but you can implement them with a Lua-scripted `INCR` on a companion key at acquisition time, or use a system that does provide them (ZooKeeper epoch numbers, etcd revision numbers).

## Secondary Issue: The Heartbeat Has a Race Condition

The heartbeat implementation has a check-then-act race that makes it unsafe even on its own terms:

```python
# _beat() in _start_heartbeat:
current = self.redis.get(lock_key)          # (1) read token
if current and current.decode() == token:   # (2) compare
    self.redis.expire(lock_key, self.LOCK_TTL)  # (3) extend ← not atomic with (1)+(2)
```

Between steps (1) and (3), the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is exactly the scenario the ownership check was meant to prevent, and the non-atomic read-compare-extend does not prevent it.

The correct implementation uses a Lua script to make the ownership check and the extension atomic:

```lua
-- extend_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = redis.register_script("""
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('expire', KEYS[1], ARGV[2])
    else
        return 0
    end
""")

# In _beat():
result = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
if result == 0:
    logger.warning("Lost lock during heartbeat key=%s — stopping beat", lock_key)
    self._heartbeat_active = False
```

The same atomicity requirement applies to lock release. The current release code has the identical race:

```python
# _acquire_lock finally block:
current = self.redis.get(lock_key)              # (1)
if current and current.decode() == token:       # (2)
    self.redis.delete(lock_key)                 # (3) ← not atomic
```

This should also be a Lua script (this is the canonical pattern from the Redis documentation):

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

## Tertiary Issue: Shared Heartbeat State Breaks Concurrent Use

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` concurrently on the same `InventoryService` instance — which is the normal case for a web service — one thread's `_stop_heartbeat()` call will set `_heartbeat_active = False` and kill the other thread's heartbeat. The lock held by the second thread will then expire undefended.

The fix is to make heartbeat state local to each lock acquisition:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()
    # pass heartbeat_active into _start_heartbeat, not self._heartbeat_active
```

## Test Coverage Gap

The tests mock out the heartbeat entirely and mock Redis at the method level. No test exercises the scenario that matters: lock expires between read and write. The test named `test_lock_not_released_if_expired` verifies that `delete` is not called when the token has changed — it does not verify that the *write* is blocked in that scenario. That is the gap that corresponds directly to the production risk.

A meaningful test would look like:

```python
def test_write_blocked_when_lock_expires_mid_operation(service, db):
    """
    Simulates lock expiry between stock read and stock write.
    The update_stock call must not succeed (or must be rejected by storage).
    Without fencing tokens, this test will fail — which is the point.
    """
    def expire_lock_on_read(item_id):
        # Simulate: by the time we return, the lock has expired and
        # another process has taken it
        service.redis.get.return_value = b"new-owner-token"
        return 10

    db.get_stock.side_effect = expire_lock_on_read

    result = service.reserve_item("item_001", quantity=1)

    # Without fencing, this assertion will fail:
    db.update_stock.assert_not_called()
```

This test should be added and should pass before the PR is merged. Currently it would fail, which is the correct signal.

## Summary of Required Changes

| Issue | Severity | Required Before Merge |
|---|---|---|
| No fencing token on DB writes | Critical | Yes |
| Non-atomic heartbeat extend (TOCTOU) | High | Yes |
| Non-atomic lock release (TOCTOU) | High | Yes |
| Shared heartbeat state across concurrent callers | High | Yes |
| No test for stale-write scenario | Medium | Yes |

The heartbeat, the retry logic, the ownership token, and the structured logging are all reasonable. The problem is not what is here — it is what is absent. A lock that cannot enforce exclusivity at the write layer is not a correctness mechanism; it reduces collision probability under normal conditions, which is a different and weaker property than what a flash sale inventory system requires.
