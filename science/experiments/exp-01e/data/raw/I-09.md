Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under normal load, but it contains a fundamental and silent safety gap that makes it insufficient as the sole concurrency guard. The implementation is vulnerable to the **Zombie Leader failure mode**: a stop-the-world GC pause (or OS-level VM migration/suspension) freezes the *entire process* — every thread simultaneously, including the heartbeat thread — for a duration that can exceed `LOCK_TTL`. When the process resumes, the lock has already expired and been legitimately acquired by another worker; the original worker has no awareness of this and proceeds to commit a now-stale write. This is not a race condition in the traditional sense — it is a **fencing failure**, and no amount of heartbeat tuning closes it. The database layer has no mechanism to reject the zombie write.

---

### Critical Flaw: Missing Fencing Token / Optimistic Lock at the Write Layer

**The failure scenario, step by step:**

1. Worker A acquires the Redis lock. Reads `current_stock = 5`.
2. A stop-the-world GC pause freezes the entire Worker A process — heartbeat thread included.
3. After 30 seconds, the Redis TTL expires naturally. No thread in Worker A is alive to extend it.
4. Worker B acquires the lock. Reads `current_stock = 5`. Writes `new_stock = 4`. Releases lock. ✅
5. Worker A resumes from the pause. It has no idea time has passed. It calculates `new_stock = 5 - 1 = 4`.
6. Worker A calls `self.db.update_stock("item_001", 4)` — **this write succeeds silently**.
7. **Result:** Stock is `4` in the database, but two reservations were issued. One item was sold twice.

The lock ownership check in the `finally` block (`redis.get(lock_key) == token`) correctly detects the expiry and skips the Redis `delete` — but by then the **database writes have already committed**. The guard is on the wrong side of the damage.

```python
# This check is too late. The writes above it already executed.
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The token check needs to gate the **database write**, not just the lock release.

---

### Required Fix: Optimistic Locking at the Database Write

The canonical solution is a **fencing token** — a monotonic version number (or the stock value itself as a precondition) embedded in the `UPDATE` statement:

```sql
UPDATE inventory
SET stock = :new_stock, version = :new_version
WHERE item_id = :item_id
  AND version = :expected_version   -- or AND stock = :expected_stock
```

In Python, this surfaces as:

```python
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_stock=current_stock   # or expected_version=read_version
)

if rows_affected == 0:
    # Our read is stale — another worker already modified this row.
    # The lock was lost under us. Abort and surface a conflict error.
    raise StaleReservationError(f"Concurrent modification detected for item {item_id}")
```

This makes the zombie write **fail at the database level** regardless of what Redis says. Even if the Redis lock check were somehow bypassed, the DB enforces the invariant. Defense in depth.

### Secondary Issues Worth Addressing

**1. Instance-level heartbeat state is not concurrency-safe**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If a single `InventoryService` instance is shared across threads (common in web frameworks), two concurrent calls to `reserve_item` will race on `_heartbeat_active` and `_heartbeat_thread`. One call's `_stop_heartbeat()` will kill the other's heartbeat. The heartbeat state must be scoped to the lock acquisition context, not the instance.

```python
# Safer: pass a threading.Event into the heartbeat closure
stop_event = threading.Event()

def _beat():
    while not stop_event.wait(timeout=HEARTBEAT_INTERVAL):
        ...

# In finally:
stop_event.set()
```

**2. `quantity` has no validated lower bound**

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 0 is False) and creates a zero-unit order. `quantity=-5` would *increase* stock. Input validation should enforce `quantity >= 1` before the lock is even acquired — ideally via a Pydantic input model on this method's parameters, consistent with the stated use of Pydantic elsewhere.

**3. `db.update_stock` and `db.create_order` are not atomic**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `create_order` raises after `update_stock` succeeds, stock is decremented but no order record exists. These two writes should be wrapped in a single database transaction. The current implementation has no rollback path.

**4. Test suite mocks away the exact behavior under scrutiny**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Disabling the heartbeat in tests is reasonable for unit isolation, but there is no integration-level test that exercises the actual Redis interaction, the token comparison logic under expiry, or the thread lifecycle. The `test_lock_not_released_if_expired` test verifies that `delete` is not called — but does not verify that the *database write was also aborted*, which is the actual safety requirement.

---

### Summary of Required Changes Before Approval

| # | Severity | Issue |
|---|----------|-------|
| 1 | 🔴 Critical | No fencing token at DB write — zombie writes succeed silently after GC pause or VM migration causes lock expiry |
| 2 | 🔴 Critical | `update_stock` + `create_order` not wrapped in a transaction — partial failure leaves data inconsistent |
| 3 | 🟠 High | Instance-level heartbeat state is not safe for shared-instance concurrent use |
| 4 | 🟡 Medium | `quantity` not validated for `>= 1` before lock acquisition |
| 5 | 🟡 Medium | Test suite does not assert that stale DB writes are rejected — the most important invariant is untested |

The heartbeat is a legitimate optimization for keeping the lock alive under slow-but-running operations. It is not, and cannot be, a substitute for write-time fencing. These two mechanisms must coexist.
