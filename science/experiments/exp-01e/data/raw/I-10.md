Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard, but it contains a fundamental and silent correctness flaw that makes it insufficient as the sole concurrency guard. The distributed lock architecture here is vulnerable to the **Zombie Leader failure mode** — a stop-the-world GC pause (or OS-level VM migration/suspension) freezes the *entire process*, including the heartbeat thread, simultaneously. When the process resumes after the TTL has elapsed, it holds a stale belief that it owns the lock, but the lock has already expired and been re-acquired by another worker. The heartbeat cannot save you from a pause it is also frozen inside of. The tests do not cover this scenario, and the checklist item "Heartbeat thread prevents lock expiry under load" is technically true but dangerously misleading — it implies a level of safety guarantee that does not exist.

---

## Detailed Findings

### 🔴 Critical — Zombie Leader / Stale Write (No Fencing Token)

**File:** `services/inventory.py`, `reserve_item` and `_acquire_lock`

The failure sequence is deterministic and reproducible under real production conditions:

1. Worker A acquires the lock. `current_stock = 5`.
2. Worker A reads stock, begins DB operations.
3. A stop-the-world GC pause (Python's GC can do this; so can JVM, .NET, container memory pressure, or hypervisor live migration) freezes **every thread in the process** — the heartbeat thread included.
4. 30 seconds elapse. Redis expires the lock.
5. Worker B acquires the lock. Reads `current_stock = 5`. Writes `new_stock = 4`. Releases lock cleanly.
6. Worker A resumes from the pause. It has no awareness that time passed. It calls `self.db.update_stock("item_001", 4)` — overwriting Worker B's committed write.
7. **Result:** Two units were sold, but stock reads `4` instead of `3`. One item has been double-sold. No exception is raised. No log entry indicates anything went wrong.

The ownership check in the `finally` block (`redis.get(lock_key) == token`) **does not protect the DB write**. It only protects the Redis `DELETE`. The stale write to the database has already happened before the `finally` block is reached.

**The fix is a fencing token (optimistic lock) at the database write layer:**

```python
# Instead of:
self.db.update_stock(item_id, new_stock)

# Require a conditional update that fails if stock has changed:
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock,
    expected_stock=current_stock  # fencing condition
)
if rows_affected == 0:
    raise StaleInventoryError(
        f"Stock for {item_id} was modified concurrently. Reservation aborted."
    )
```

At the SQL level, this translates to:

```sql
UPDATE inventory
SET stock = %(new_stock)s
WHERE item_id = %(item_id)s
  AND stock = %(expected_stock)s
```

If `rows_affected == 0`, the zombie write is rejected at the database — the only layer that can authoritatively enforce this. A monotonic version/fencing token column (`version INT`) is the canonical pattern and is even more robust than a stock-value comparison (which could theoretically collide if stock returns to the same value through independent operations).

The distributed lock is still valuable — it reduces contention and avoids thundering herd — but it **cannot be the sole correctness mechanism**. The database write must be self-defending.

### 🟡 Moderate — Instance-Level State on a Shared Service Object

**File:** `services/inventory.py`, `_heartbeat_active` and `_heartbeat_thread`

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If `InventoryService` is instantiated as a singleton (common in DI frameworks and Flask/FastAPI app factories), concurrent calls to `reserve_item` will race on `_heartbeat_active`. Worker A's `_stop_heartbeat()` call will terminate Worker B's heartbeat thread mid-operation.

The heartbeat state must be scoped to the individual lock acquisition, not to the service instance. A simple fix is to pass a `threading.Event` into `_start_heartbeat` and return it, keeping the lifecycle entirely local to `_acquire_lock`:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # ... acquire lock ...
    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)
    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release lock ...
```

### 🟡 Moderate — Heartbeat Thread Is Never Joined

**File:** `services/inventory.py`, `_stop_heartbeat` and `_acquire_lock` finally block

`_stop_heartbeat()` sets `_heartbeat_active = False` and returns immediately. The heartbeat thread is sleeping for up to `HEARTBEAT_INTERVAL` (10 seconds) and will not observe the flag until it wakes. In the window between `_stop_heartbeat()` and the thread waking, the following sequence is possible:

1. `_stop_heartbeat()` is called.
2. Lock is deleted from Redis (`redis.delete(lock_key)`).
3. Worker B acquires the same lock with a new token.
4. Worker A's heartbeat thread wakes up, calls `redis.get(lock_key)` — gets Worker B's token — token mismatch, so no extend. Safe *this time*, but only because of the token check.

The token check saves you here, but the thread is still running for up to 10 seconds after the lock is released. Under high throughput with many goroutines/threads, this accumulates. The `finally` block should call `thread.join(timeout=...)` to ensure clean teardown.

### 🟡 Moderate — `quantity` Has No Validated Lower Bound

**File:** `services/inventory.py`, `reserve_item`

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 10 is False... wait — actually `10 < 0` is False, so it proceeds) and creates an order for zero units, decrementing stock by zero. `quantity=-5` would *increase* stock. Neither is caught. The Pydantic model on `ReservationResult` validates the output, not the input. Add a guard:

```python
if quantity < 1:
    return ReservationResult(
        success=False,
        message="Quantity must be at least 1"
    )
```

Or enforce this with a Pydantic input model on the request layer before it reaches this service.

---

### 🟡 Moderate — `db` Contract Is Implicit and Untested for Partial Failure

**File:** `services/inventory.py`, `reserve_item`

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls inside the lock boundary, but there is no transaction wrapping them. If `update_stock` succeeds and `create_order` raises, stock has been decremented but no order exists. The lock is released. The item is effectively lost. These two operations must be atomic — either both succeed or neither does. If the `db` abstraction does not expose transaction semantics, that is an architectural gap that needs to be resolved before this ships.

---

### 🟢 Minor — Lock Retry Backoff Is Synchronous and Blocks the Thread Pool

**File:** `services/inventory.py`, `_acquire_lock`

```python
time.sleep(backoff)
```

In an async framework (FastAPI with `asyncio`, for example), this will block the event loop thread. Even in a synchronous WSGI context, sleeping the worker thread under flash sale load is undesirable. This is a lower-priority concern depending on the deployment model, but worth flagging for the team to confirm the threading model is compatible.

## Required Before Approval

1. **Fencing token / optimistic lock on `update_stock`** — this is non-negotiable. The distributed lock alone does not provide safety guarantees against process pauses.
2. **Heartbeat state scoped to lock acquisition**, not to the service instance.
3. **Transactional atomicity** for `update_stock` + `create_order`.
4. **Input validation** for `quantity >= 1`.
5. **Test coverage** for the stale write scenario and partial DB failure.

The heartbeat implementation is not wrong — it is a reasonable defense against slow-but-alive operations. It is being asked to do a job it is architecturally incapable of doing alone. Defense in depth requires the database to be the final arbiter of correctness.
