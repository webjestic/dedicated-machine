# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains multiple critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The heartbeat pattern introduces a race condition that actively undermines the safety guarantee the lock is supposed to provide, and the non-atomic DB operations create a partial-failure window that no amount of locking can close. These are not edge cases — they are near-certainties under flash sale load.

---

## Critical Issues

### 🔴 Issue 1: Instance-Level Heartbeat State Breaks Concurrent Safety Entirely

This is the most dangerous flaw in the codebase. The heartbeat state is stored on the *instance*, not scoped to the lock invocation.

```python
# In __init__:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None

# In _start_heartbeat:
self._heartbeat_active = True  # ← shared instance state
```

**The failure scenario:**

If `InventoryService` is used as a singleton (the standard pattern in any DI container or Flask/FastAPI app), two concurrent requests will share `_heartbeat_active` and `_heartbeat_thread`. When Request A finishes and calls `_stop_heartbeat()`, it sets `self._heartbeat_active = False`, which immediately kills the heartbeat thread that is actively protecting Request B's lock. Request B's lock then expires mid-operation, another process steals it, and you now have two processes simultaneously decrementing stock — a textbook oversell.

**Fix:** Heartbeat state must be scoped to the lock acquisition, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # Pass event into the thread closure, not instance state
    ...
```

### 🔴 Issue 2: The Heartbeat Check-Then-Extend Is a Non-Atomic TOCTOU Race

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:
            current = self.redis.get(lock_key)          # ← read
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← write
```

Between the `GET` and the `EXPIRE`, the lock can expire and be acquired by another process. You then extend *their* lock, not yours. You have now gifted an adversarial process an extra 30 seconds of lock ownership while your own process continues executing under the false belief that it holds the lock.

This is not a theoretical concern. Under a GIL-induced pause, a GC pause, or any network hiccup to Redis, this window opens.

**Fix:** Replace the two-step GET+EXPIRE with a single atomic Lua script, which is the industry-standard pattern for Redlock-style heartbeats:

```lua
-- Atomic compare-and-extend
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _extend_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(
        EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL
    )
    return bool(result)
```

### 🔴 Issue 3: Non-Atomic DB Operations Create an Unrecoverable Partial-Failure Window

```python
self.db.update_stock(item_id, new_stock)   # ← succeeds
self.db.create_order(order_id, item_id, quantity)  # ← crashes
```

If the process dies, experiences a Stop-the-World GC pause that expires the lock, or if `create_order` throws, you now have decremented stock with no corresponding order record. This is silent inventory loss. There is no retry path that can recover this because `order_id` was generated in-memory and is now gone.

The lock provides *mutual exclusion*, not *atomicity*. These are different guarantees. The lock cannot save you here.

**Fix:** Both operations must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer doesn't support this, that is a prerequisite architectural requirement that must be resolved before this service can be considered safe.

### 🔴 Issue 4: Lock Release Is Also a Non-Atomic TOCTOU Race

```python
# In the finally block:
current = self.redis.get(lock_key)              # ← read
if current and current.decode() == token:
    self.redis.delete(lock_key)                 # ← write
```

This is the same class of bug as Issue 2. Between `GET` and `DELETE`, the lock can expire and be re-acquired by Process B. You then delete Process B's lock, leaving the critical section completely unprotected.

**Fix:** Use the canonical atomic Lua script for compare-and-delete, which is documented in the official Redis distributed lock specification:

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return bool(result)
```


### 🟠 Issue 6: Heartbeat Interval vs. TTL Ratio Is Unsafe Under Pause Conditions

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 10
```

The heartbeat fires at T=10 and T=20. If the process experiences a Stop-the-World GC pause, a Python GIL contention stall, or a Redis network hiccup of >10 seconds at T=19, the next heartbeat fires at T=29 — one second before expiry. Any jitter in the `time.sleep` call or the Redis round-trip and the lock expires before renewal.

The standard guidance (per Martin Kleppmann and the Redlock analysis) is to renew at ≤ TTL/3. At a 30-second TTL, the heartbeat interval should be no more than 10 seconds, and you should log a warning if the renewal itself takes more than a threshold duration.

```python
LOCK_TTL = 30
HEARTBEAT_INTERVAL = 8  # renew at ~TTL/3.75, with margin
```

Additionally, the heartbeat thread must detect a failed renewal and signal the main thread to abort, rather than silently continuing:

```python
if not self._extend_lock(lock_key, token):
    logger.critical(
        "Lost lock ownership during operation key=%s — aborting", lock_key
    )
    heartbeat_active.clear()
    # signal main thread via an event or flag
```

## Moderate Issues

### 🟡 Issue 7: Test Suite Validates Nothing About the Real Failure Modes

The tests mock out `_start_heartbeat` and `_stop_heartbeat` entirely, which means they cannot detect Issues 1, 2, or 4. The test for lock expiry (`test_lock_not_released_if_expired`) validates the wrong behavior — it confirms that `delete` is not called when the token doesn't match, but it uses a mocked `get` that returns a static value, not a token that expired mid-operation.

There are no tests for:
- Concurrent access (the entire reason this service exists)
- `db.create_order` failure after `db.update_stock` succeeds (Issue 3)
- Negative or zero `quantity` values (Issue 5)
- Heartbeat thread lifecycle under concurrent calls to the same instance

At minimum, a test using `threading.Barrier` to force a true concurrent reservation scenario is required before this can be considered tested.
