**Decision: Request Changes**

**Summary:** The heartbeat thread is well-implemented and the lock *acquisition* story is solid, but the critical write path — `update_stock` and `create_order` — has no fencing mechanism at the storage layer. If the lock expires between the heartbeat's last check and the DB writes (GC pause, network stall, slow DB call), a second process can acquire the lock and read stale stock, and both processes will then write conflicting updates to a database that has no awareness of the lock token and will accept both. The heartbeat answers "will the lock usually still be alive?" not "is this write still authorized?" — and for inventory in a flash sale, that distinction is the entire problem.

---

### The Core Issue: No Fencing at the Write Site

The PR's own docstring says:

> *The heartbeat thread ensures the lock does not expire during slow DB operations.*

That is the wrong guarantee to rely on. The question is not whether the lock *usually* survives — it is what happens to the write when it does not. The heartbeat closes the window; it does not close it to zero.

**The failure scenario:**

```
Process A holds lock, heartbeat running.

[A's heartbeat thread: sleeps for 10s]
[A's application thread: GC pause / slow network / slow DB — 11s]

Lock TTL expires.

Process B acquires the lock.
Process B reads stock = 10, proceeds.

[A's application thread resumes]
A calls db.update_stock(item_id, 9)   ← lock already gone, DB doesn't know
B calls db.update_stock(item_id, 9)   ← same read, same write

Result: stock decremented once, two orders created.
```

This is not a theoretical edge case. It is the exact scenario Kleppmann's fencing token paper demonstrates, and it is why Jepsen finds these bugs: they require only a pause longer than `HEARTBEAT_INTERVAL` at the wrong moment, which any production GC, network blip, or slow query can produce.

---

### What "Fencing" Actually Means Here

A fencing token is a monotonically increasing value issued with the lock. The storage layer rejects any write carrying a token older than the last accepted one. Redis's `WATCH`/`MULTI`/`EXEC` (optimistic locking) or a database-level conditional write (`UPDATE ... WHERE stock = :expected_stock AND version = :expected_version`) are the practical forms of this for inventory.

The database — not the lock client — must be the final arbiter of whether a write is still valid.

### Specific Issues

**1. `update_stock` and `create_order` are unconditional writes**

```python
# reserve_item(), inside the lock context — but the lock may be gone
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

Neither call carries any proof that the lock is still held. The database accepts them regardless. A conditional update is required:

```python
# Example: optimistic concurrency via version column
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_stock=current_stock,   # or a version/etag
)
if rows_affected == 0:
    # Another writer got there first — abort
    raise StaleDataError(f"Concurrent modification detected for {item_id}")
```

**2. `update_stock` and `create_order` are not atomic with each other**

```python
self.db.update_stock(item_id, new_stock)   # succeeds
self.db.create_order(order_id, item_id, quantity)  # fails / times out
```

Stock is decremented but no order exists. These two writes must be in a single database transaction, or the failure path must compensate. The lock does not help here — a crash between the two calls leaves the system inconsistent regardless of lock state.

**3. The heartbeat has a TOCTOU race in its own token check**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # lock can expire between these two lines
```

The `GET` + `EXPIRE` is not atomic. Use a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

**4. The `InventoryService` instance carries heartbeat state as instance variables**

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `reserve_item` is called concurrently on the same service instance (which is the entire point of this service), one call's `_stop_heartbeat()` sets `_heartbeat_active = False` and kills the other call's heartbeat thread. This is a straightforward data race. Heartbeat state must be scoped to each lock acquisition, not to the instance.

**5. The test suite does not test the scenario it names**

```python
def test_lock_not_released_if_expired(service, redis_client):
    """Lock release is skipped if token no longer matches (lock expired or stolen)."""
    redis_client.get.return_value = b"different-token"

    result = service.reserve_item("item_001")

    redis_client.delete.assert_not_called()
```

This test verifies the lock release is skipped — correct — but it does not assert that `update_stock` was also skipped. The writes happen before the release check. The test passes while the inventory corruption occurs. Add:

```python
    db.update_stock.assert_not_called()
    db.create_order.assert_not_called()
```

Those assertions will fail, which is the point: they expose that the writes are not guarded.

---

### What Needs to Change Before Approval

| # | Issue | Required Fix |
|---|-------|-------------|
| 1 | Unconditional writes after lock may have expired | Conditional/versioned writes at DB layer, or `WATCH`/`MULTI`/`EXEC` |
| 2 | `update_stock` + `create_order` not atomic | Wrap in a single DB transaction |
| 3 | Heartbeat `GET`+`EXPIRE` is not atomic | Replace with Lua script |
| 4 | Heartbeat state on instance, not per-lock | Move `_heartbeat_active` / `_heartbeat_thread` into `_acquire_lock` scope |
| 5 | Test does not assert writes are blocked on expired lock | Add `update_stock.assert_not_called()` to the expiry test |

The lock acquisition and release mechanics are thoughtful. The gap is that the lock is used to *coordinate* but the storage layer is not used to *enforce*. For a flash sale inventory system, that gap will be exercised.
