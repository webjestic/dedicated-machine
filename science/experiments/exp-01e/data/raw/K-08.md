# Code Review: `feature/flash-sale-inventory`

---

**Decision: Request Changes**

**Summary:** This implementation contains several critical architectural flaws that will cause silent data corruption and inventory overselling under real production conditions. The most severe issues involve a non-atomic lock release, a shared-state heartbeat design that breaks under concurrent calls, and a non-atomic DB write pair that creates a partial-failure window — none of which are caught by the test suite because the tests mock away the exact behaviors that would expose them. This code must not be merged into a service handling financial transactions.

---

## Critical Findings

### 🔴 CRITICAL-1: Non-Atomic Lock Release (TOCTOU Race → Inventory Theft)

```python
# _acquire_lock finally block
current = self.redis.get(lock_key)          # ← (1) read
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← (2) delete
```

This is a classic **check-then-act** race condition. Between the `GET` at step (1) and the `DELETE` at step (2), a GC pause, OS scheduler preemption, or network hiccup can cause the lock's TTL to expire. Another process then legitimately acquires the lock. When this process resumes at step (2), it deletes a lock it no longer owns, allowing a **third** process to also acquire the lock. You now have two concurrent holders of a mutex. For a flash sale, this means two threads can simultaneously read the same stock value and both successfully reserve the last unit.

The canonical fix is a Lua script executed atomically on the Redis server:

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

# In __init__:
self._release_script = self.redis.register_script(RELEASE_SCRIPT)

# In finally block, replacing the get+delete:
self._release_script(keys=[lock_key], args=[token])
```

This is not a theoretical concern. The Redis documentation explicitly warns against this pattern and mandates the Lua approach for exactly this reason.

### 🔴 CRITICAL-2: Instance-Level Heartbeat State Breaks All Concurrent Usage

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False        # ← shared instance state
        self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on the **instance**, not scoped to the lock invocation. If `InventoryService` is used as a singleton (the standard pattern in any DI container or Flask/FastAPI app), two concurrent calls to `reserve_item` will share this state. The sequence of failure is deterministic:

| Time | Thread A | Thread B |
|------|----------|----------|
| T1 | Acquires lock A, sets `_heartbeat_active = True` | — |
| T2 | — | Acquires lock B, sets `_heartbeat_active = True` (no-op, already True) |
| T3 | Finishes, calls `_stop_heartbeat()`, sets `_heartbeat_active = False` | — |
| T4 | — | Thread B's heartbeat loop exits. **Lock B stops being renewed.** |
| T5 | — | Lock B expires mid-operation. A third process steals it. |

Thread B is now operating under a lock it no longer holds, with no awareness of this fact. The heartbeat mechanism — the primary safety feature of this PR — silently stops working for any non-first concurrent caller.

The fix is to scope heartbeat state to the lock context, not the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()
    
    def _beat():
        while heartbeat_active.is_set():
            time.sleep(self.HEARTBEAT_INTERVAL)
            if heartbeat_active.is_set():
                # Lua script for atomic check-and-extend
                extend_script(keys=[lock_key], args=[token, self.LOCK_TTL])
    
    heartbeat_thread = threading.Thread(target=_beat, daemon=True)
    heartbeat_thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # atomic release via Lua
```

Note also that `_stop_heartbeat` sets the flag but does not `join()` the thread. The heartbeat thread can still be alive and attempting to extend the lock **after** the lock has been released, creating a window where it could extend a lock now owned by another process.

### 🔴 CRITICAL-3: Non-Atomic Dual Write Creates Guaranteed Partial-Failure Window

```python
self.db.update_stock(item_id, new_stock)    # ← write 1
self.db.create_order(order_id, item_id, quantity)  # ← write 2
```

These are two separate database calls with no transaction boundary. If the process crashes, is OOM-killed, or the DB connection drops between these two lines, you get **stock decremented with no order record**. The item is gone from inventory but there is no order to fulfill, ship, or refund. This is a silent data corruption event — the system believes the item was sold, but there is no record of to whom.

This must be wrapped in a database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The exact API depends on the DB layer, but the atomicity requirement is non-negotiable for any financial or inventory system.

## High Severity Findings

### 🟠 HIGH-1: Heartbeat Does Not Verify Lock Ownership Before Extending

```python
# In _beat():
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # ← not atomic with the check above
```

The heartbeat has the same TOCTOU problem as the release. The `GET` and `EXPIRE` are not atomic. More importantly, if the lock has already expired and been acquired by another process with a different token, the check correctly skips the extend — but this means the current operation is now running without a lock and is completely unaware of it. There is no mechanism to abort the in-progress `reserve_item` call when the lock is discovered to be lost.

A robust implementation needs a way to signal the main thread that lock ownership was lost mid-operation so it can abort rather than commit a write without holding the lock.

### 🟠 HIGH-2: `quantity` Has No Validated Lower Bound

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity` is a plain `int` with no validation. A caller passing `quantity=0` will always succeed (0 ≤ any stock), create a real order record, and decrement stock by zero — a no-op reservation that pollutes the order table. A caller passing `quantity=-5` will **increase** stock (`new_stock = current_stock - (-5)`), which is an inventory injection vulnerability. If this endpoint is externally reachable, this is a security issue.

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_-]+$')
    quantity: int = Field(ge=1, le=100)  # enforce sane bounds
```

## Moderate Findings

### 🟡 MOD-1: The Test Suite Validates Nothing Real

The test suite mocks `_start_heartbeat` and `_stop_heartbeat` at the fixture level, which means the heartbeat is never exercised. The lock release test (`test_lock_not_released_if_expired`) mocks `redis.get` to return a different token, which does verify the conditional delete — but it cannot catch CRITICAL-1 because the race condition exists between two Redis calls that are both mocked as instantaneous. No test covers concurrent execution, partial DB failure, or the instance-state collision in CRITICAL-2. The tests confirm the happy-path control flow but provide no safety guarantee for the failure modes this service was specifically designed to handle.

### 🟡 MOD-2: `RuntimeError` Is Too Broad an Exception Contract

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
    return ReservationResult(success=False, message="Service temporarily unavailable...")
```

`RuntimeError` is a broad base class. Any `RuntimeError` raised inside the `with self._acquire_lock(lock_key):` block — including from `db.update_stock` or `db.create_order` — will be silently swallowed and returned as "service temporarily unavailable," masking a real data error as a transient retry-able failure. Define a specific `LockAcquisitionError` and catch only that.

### 🟡 MOD-3: Daemon Thread Behavior on Interpreter Shutdown Is Unsafe

Daemon threads are killed abruptly when the main thread exits. If the application is shutting down mid-reservation, the heartbeat thread dies with no cleanup, the lock may or may not be released depending on timing, and the `finally` block in `_acquire_lock` may not execute. For a graceful shutdown scenario, the service needs a `close()` or `shutdown()` method that signals all active heartbeat threads to stop and waits for them to complete.
