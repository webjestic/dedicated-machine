Decision: Request Changes

Summary: This implementation contains several critical architectural flaws that will cause silent data corruption and availability failures under real flash-sale load conditions. The most severe issues involve non-atomic lock release, shared heartbeat state that breaks under concurrent calls, and a split database operation that has no transactional guarantee — meaning partial writes are not just possible, they are guaranteed to occur eventually. None of these failure modes are caught by the test suite, which is itself structurally unsound.

---

## Critical Findings

### 1. 🔴 Non-Atomic Lock Release — TOCTOU Race Condition

```python
# _acquire_lock finally block
current = self.redis.get(lock_key)          # ← moment A
if current and current.decode() == token:
    self.redis.delete(lock_key)             # ← moment B
```

Between moment A and moment B, the following sequence is entirely possible:

1. Your process reads the key and confirms it owns the lock.
2. A Stop-the-World GC pause (or network hiccup) freezes your process for 31+ seconds.
3. The lock TTL expires. Redis drops the key.
4. A second process acquires the lock and begins its critical section.
5. Your process resumes and calls `redis.delete`, destroying the second process's lock.
6. A third process now acquires the lock simultaneously with the second.

**Two processes now hold the lock concurrently.** Every safety guarantee this system is built on collapses at this point. This is not a theoretical edge case — GC pauses in Python are real, and flash-sale load is exactly the condition that triggers them.

The fix is a Lua script that makes the check-and-delete atomic:

```lua
-- release_lock.lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
    return result == 1
```

This is not optional. The same TOCTOU vulnerability exists in the heartbeat's `get`→`expire` sequence and must be fixed there as well.

### 2. 🔴 Instance-Level Heartbeat State — Breaks All Concurrent Usage

```python
# On the InventoryService instance:
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`InventoryService` is almost certainly a singleton or a long-lived object. Consider two concurrent calls:

```
Thread A: reserve_item("item_001") → _start_heartbeat() → sets self._heartbeat_active = True
Thread B: reserve_item("item_002") → _start_heartbeat() → sets self._heartbeat_active = True
Thread A: finishes → _stop_heartbeat() → sets self._heartbeat_active = False
```

Thread B's heartbeat is now silently dead. Thread B's lock will expire mid-operation. The stock read and the stock write will now execute in different lock tenancies, meaning another process can interleave between them. You get a classic lost-update: two processes both read stock=1, both decrement to 0, both write 0, and you've sold the same unit twice.

The heartbeat state must be scoped to the individual lock acquisition, not the service instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    # ... acquire lock ...
    heartbeat_active.set()
    thread = threading.Thread(
        target=self._heartbeat_loop,
        args=(lock_key, token, heartbeat_active),
        daemon=True
    )
    thread.start()
    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        self._release_lock(lock_key, token)
```

Note also that the current `_stop_heartbeat` does not join the thread. The heartbeat thread can continue executing for up to `HEARTBEAT_INTERVAL` seconds after the lock is released, extending a lock that may now be owned by a different process.

### 3. 🔴 Non-Transactional Database Operations — Guaranteed Partial Write Window

```python
self.db.update_stock(item_id, new_stock)   # ← succeeds
self.db.create_order(order_id, item_id, quantity)  # ← crashes
```

If the process dies, the network drops, or the DB throws between these two lines, stock is decremented but no order record exists. The item is gone from inventory with no corresponding sale. At flash-sale volumes, this will happen.

These two operations must be wrapped in a single database transaction:

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the `db` abstraction does not support transactions, that is itself a blocking architectural issue that must be resolved before this code ships.

### 4. 🟠 Heartbeat Race on Lock Expiry — Extends a Lock You No Longer Own

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # ← sleeps 10s
        if self._heartbeat_active:
            current = self.redis.get(lock_key)  # ← TOCTOU again
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

The `get`→`expire` sequence here has the same TOCTOU problem as the release path. Between the `get` and the `expire`, the lock can expire and be acquired by another process. Your heartbeat then extends the new owner's lock — which is benign in the extend case but demonstrates that the check provides no real safety. Use a Lua script here too:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

More importantly: if the heartbeat detects that the token no longer matches, it means the lock expired and was stolen. The critical section is no longer protected. The heartbeat must signal the main thread to abort the operation. Currently it logs nothing and lets the main thread continue writing to the database without lock protection.

### 5. 🟠 `quantity` Parameter Has No Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

`quantity=0` passes the `current_stock < quantity` check (0 < 10 is False, so it proceeds), decrements stock by 0, and creates a real order record for zero units. `quantity=-5` increases stock. This is a business logic vulnerability. Pydantic is imported but not used for the method signature. Add a guard:

```python
if quantity <= 0:
    return ReservationResult(success=False, message="Quantity must be positive")
```

### 6. 🟡 Test Suite Does Not Test the System Under Test

The test fixture patches out the heartbeat entirely and hardcodes the Redis token:

```python
mock.get.return_value = b"test-token"  # always matches any generated UUID
```

This means `test_lock_not_released_if_expired` does not actually test the real release path — it tests a mock. The TOCTOU race, the concurrent heartbeat corruption, and the partial write scenario are completely invisible to this suite. The tests confirm that the Python control flow is correct in isolation, not that the distributed system behaves correctly. Integration tests against a real Redis instance (e.g., using `pytest-redis` or a Docker fixture) are required for a system of this criticality.

## Summary of Required Changes Before Approval

| Severity | Issue | Required Fix |
|---|---|---|
| 🔴 Critical | Non-atomic lock release (TOCTOU) | Lua script for atomic check-and-delete |
| 🔴 Critical | Instance-level heartbeat state | Scope heartbeat state per lock acquisition |
| 🔴 Critical | Non-transactional DB writes | Wrap both writes in a single DB transaction |
| 🟠 High | Heartbeat extends stolen lock silently | Lua script for extend; abort signal to main thread on token mismatch |
| 🟠 High | Thread not joined before lock release | `thread.join()` before releasing lock |
| 🟠 High | No quantity validation | Guard against `quantity <= 0` |
| 🟡 Medium | Tests mock away all distributed behavior | Add integration tests against real Redis |
