Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields, causing one call's heartbeat to silently kill another's, or two heartbeat threads to stomp each other. Beyond that, the DB operations are not atomic and there is no input validation on the public method's arguments despite the checklist claiming otherwise. These issues need to be resolved before this can go anywhere near a production flash sale workload.

---

### Critical Issues

**1. Shared Heartbeat State — Race Condition (Blocker)**

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields. If two threads call `reserve_item` concurrently on the same `InventoryService` instance (which is the entire point of this service), they will share `_heartbeat_active`. The second caller's `_start_heartbeat` overwrites `_heartbeat_thread`, and the first caller's `_stop_heartbeat` sets `_heartbeat_active = False`, killing the second caller's heartbeat silently. The lock will then expire mid-operation with no error raised.

Fix: Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat() -> None:
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning("Lock contention key=%s attempt=%d retrying in %.1fs",
                       lock_key, attempt + 1, backoff)
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    t = threading.Thread(target=_beat, daemon=True)
    t.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        t.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

Note the added `t.join()` — without it the heartbeat thread can extend the lock *after* the finally block deletes it, creating a window where a new owner acquires the lock and then has it silently extended by the previous owner's zombie thread.

**2. Non-Atomic DB Operations — Inventory Leak Under Failure (Blocker)**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate calls with no transaction boundary. If `update_stock` succeeds and `create_order` throws, stock is decremented but no order exists. The item is effectively lost. The distributed lock only protects against concurrent reads — it does not substitute for database atomicity.

Fix: Wrap in a transaction or use a compensating rollback.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

If the DB layer does not support transactions, that is an architecture gap that needs to be addressed before this merges.

**3. No Input Validation on `reserve_item` (Checklist Claims Otherwise)**

The checklist marks input validation as complete, but `reserve_item` accepts arbitrary values for `item_id` and `quantity` with no guards. A caller can pass `quantity=0`, `quantity=-5`, or an empty string for `item_id`. The Pydantic model is only used on the *output*.

```python
# Current — no validation
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

Fix:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="item_id must not be empty")
    if quantity < 1:
        return ReservationResult(success=False, message="quantity must be at least 1")
```

Or use a Pydantic input model and validate at the boundary. Either way, the checklist should not have been marked complete.

### Moderate Issues

**4. Heartbeat Reads Lock Value Over the Network — TOCTOU Window**

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)
```

The get and expire are two separate commands. Between them, the lock can expire and be acquired by another process, which then has its TTL silently extended by this thread. This should be a single atomic operation using a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

self.redis.eval(EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL)
```

The same pattern already used correctly in the release path should be applied here too.

**5. `time.sleep` in Retry Loop Blocks the Calling Thread**

For a flash sale under high concurrency, blocking the thread for up to `0.1 + 0.2 + 0.4 = 0.7s` per failed reservation is a latency cliff. This is acceptable for a background worker but not for a user-facing request path. Consider whether the caller should receive an immediate 503 and retry client-side, or whether this service should be async.

### Test Coverage Gaps

**6. Concurrent Access Is Not Tested**

The most important property of this service — that two simultaneous reservations do not both succeed when only one unit remains — has no test. This should be a mandatory test given the stated purpose.

```python
def test_concurrent_reservation_only_one_succeeds():
    import threading
    # Use a real or fake Redis with actual locking semantics,
    # not a MagicMock that always returns True
    ...
```

**7. Heartbeat Is Fully Mocked Out**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat entirely means the thread-safety bug in issue #1 is invisible to the test suite. At minimum one integration-level test should run the real heartbeat path against a local Redis instance (or `fakeredis`) to catch regressions.

**8. DB Failure Mid-Transaction Is Not Tested**

There is no test for `create_order` raising an exception after `update_stock` succeeds. This is the exact failure mode described in issue #2 and it should be a test case.

---

### Minor Notes

- `LOCK_TTL = 30` is generous. For a flash sale, 30 seconds of lock hold time means one slow DB call can block every other buyer on that item for half a minute. Profile your DB operations and tighten this.
- The `RuntimeError` catch in `reserve_item` is too broad. If the DB raises a `RuntimeError` inside the lock, it will be silently swallowed and returned as "Service temporarily unavailable" with no indication of what actually failed. Catch a more specific lock exception type.
- `logger.error` on lock failure is correct, but DB errors inside the lock context are completely unhandled and will propagate as unhandled exceptions to the caller. Add an explicit except block with logging for DB-layer failures.
