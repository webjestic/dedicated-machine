**Decision: Request Changes**

**Summary:** The heartbeat thread is well-implemented and the lock *acquisition* story is solid, but the critical write path — `update_stock` and `create_order` — has no protection against executing after the lock has already expired. The heartbeat reduces the probability of expiry under normal conditions; it does not eliminate it, and it provides zero protection against the cases that actually cause data corruption (GC pause, network partition, process stall between the last heartbeat and the write). There is no fencing token passed to the database writes, so the database cannot reject a write from a stale lock holder, meaning two processes can both believe they hold the lock and both write concurrently. A second structural issue compounds this: the `_heartbeat_active` flag and `_heartbeat_thread` are instance-level state, so concurrent calls to `reserve_item` on the same `InventoryService` instance will race on that shared state and corrupt each other's heartbeat lifecycle.

---

### Issue 1 (Critical): No fencing on the write path

The heartbeat keeps the lock alive *probabilistically*. It does not make the writes *safe*. Consider this sequence:

1. Process A holds the lock, heartbeat is running.
2. Process A stalls (GC, slow syscall, scheduler delay) for 31 seconds.
3. The lock expires. Process B acquires it and begins its own reservation.
4. Process A's heartbeat thread wakes up, checks the token, finds it gone, stops extending. But Process A's main thread is still mid-stall.
5. Process A resumes. It is now past the `with self._acquire_lock(lock_key):` line. Nothing re-checks lock ownership before the writes.
6. `self.db.update_stock(item_id, new_stock)` executes concurrently with Process B's identical call.

The lock has expired. The heartbeat correctly detected this. But the critical section ran anyway. The heartbeat thread and the main thread share no synchronization on this condition — `_heartbeat_active` being set to `False` by the heartbeat thread does not stop the main thread from proceeding to the write.

**What is required:** A fencing token — a monotonically increasing integer issued by Redis (e.g., via `INCR` on a separate key, or using Redlock's clock-based approach) — must be passed through to the database layer, and the database must reject any write whose token is not the highest it has seen. Without this, the lock is, as Kleppmann puts it, a suggestion.

A minimal mitigation short of full fencing: re-check lock ownership *immediately before each write* inside the critical section, and abort if the token no longer matches. This does not close the TOCTOU window entirely but it dramatically narrows it:

```python
def _assert_lock_held(self, lock_key: str, token: str) -> None:
    current = self.redis.get(lock_key)
    if not current or current.decode() != token:
        raise LockExpiredError(f"Lock expired before write could complete: {lock_key}")

# Inside reserve_item, with self._acquire_lock(lock_key) as token:
self._assert_lock_held(lock_key, token)
self.db.update_stock(item_id, new_stock)
self._assert_lock_held(lock_key, token)
self.db.create_order(order_id, item_id, quantity)
```

Note that `yield token` already provides the token to the caller — it is not being used. That is the immediate gap to close.

---

### Issue 2 (Critical): Instance-level heartbeat state is not concurrency-safe

`_heartbeat_active` and `_heartbeat_thread` are attributes on `self`. If two requests call `reserve_item` concurrently on the same service instance:

- Request A calls `_start_heartbeat`, sets `self._heartbeat_active = True`, starts thread A.
- Request B calls `_start_heartbeat`, sets `self._heartbeat_active = True`, starts thread B — overwriting `self._heartbeat_thread`.
- Request A finishes, calls `_stop_heartbeat`, sets `self._heartbeat_active = False`.
- Thread B is now stopped. Request B's lock will expire undefended.
- Request B finishes, calls `_stop_heartbeat` again — no-op, but thread A's reference is lost.

In a flash sale context, concurrent calls are the entire workload. This will silently fail to renew locks under exactly the load it was designed for.

**Fix:** Move heartbeat state into the lock context, not onto the instance. A dataclass or closure scoped to each `_acquire_lock` invocation is the right shape:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    active = threading.Event()
    # ... acquire ...
    active.set()
    thread = threading.Thread(
        target=self._beat, args=(lock_key, token, active), daemon=True
    )
    thread.start()
    try:
        yield token
    finally:
        active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # ... release ...
```

Using a `threading.Event` also gives you a clean shutdown signal rather than a bare boolean flag, which has no memory visibility guarantees across threads in the general case.

---

### Issue 3 (Moderate): Heartbeat thread is not joined on exit

`_stop_heartbeat` sets the flag and returns immediately. The heartbeat thread may still be mid-sleep or mid-Redis-call when the lock is deleted. In the worst case, the thread wakes up after the lock has been released and re-extended by a new holder, then calls `redis.expire` on a lock it no longer owns — extending a lock that belongs to another process. The token check mitigates this, but only if the new holder has already written their token before the thread wakes. There is a window. Joining the thread before releasing the lock closes it.

---

### Issue 4 (Moderate): `db.update_stock` and `db.create_order` are not atomic

If `update_stock` succeeds and `create_order` fails, stock is decremented with no order record. The lock does not help here — this is a local atomicity problem. These two writes need to be wrapped in a database transaction, or the service needs compensating logic on `create_order` failure.

---

### Issue 5 (Minor): Tests mock away the components under test

`test_lock_not_released_if_expired` is the right instinct but it tests the wrong thing. It verifies that `redis.delete` is not called when the token mismatches — correct — but no test verifies that the *writes* are not called when the lock has expired. That is the actual safety property. A test like this is missing:

```python
def test_writes_aborted_if_lock_expires_before_write(service, db, redis_client):
    # Simulate lock expiring between acquisition and write
    redis_client.get.return_value = b"different-token"  # token mismatch at write time
    result = service.reserve_item("item_001")
    assert result.success is False
    db.update_stock.assert_not_called()
    db.create_order.assert_not_called()
```

This test cannot pass with the current implementation — which is precisely the point.

---

### What approval requires

1. Fencing token passed to DB writes, with DB-side enforcement; or at minimum, a synchronous lock ownership re-check immediately before each write with abort on mismatch.
2. Heartbeat state scoped per lock acquisition, not per service instance.
3. Heartbeat thread joined before lock release.
4. `update_stock` + `create_order` wrapped in a transaction.
5. Tests that assert write-path safety under lock expiry, not just lock release behavior.

The bones of this implementation are good. The retry logic, the token-verified release, the structured logging, the Pydantic validation — all fine. The gap is specifically that the lock guards entry into the critical section but nothing guards the writes within it. In a flash sale, that gap is where overselling happens.
