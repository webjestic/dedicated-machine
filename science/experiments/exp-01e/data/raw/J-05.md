Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — `_heartbeat_active` and `_heartbeat_thread` are stored as instance variables on a shared `InventoryService` object, meaning concurrent calls to `reserve_item` will race on those fields, causing one call's heartbeat to be silently killed by another's `_stop_heartbeat`. Beyond that, the DB operations are not atomic with respect to the lock in any durable sense, and input validation is claimed but not actually enforced. These issues need to be resolved before this touches a production flash-sale path.

---

## Detailed Findings

### 🔴 Critical — Thread-Safety: Shared Heartbeat State

This is the most severe issue in the PR.

```python
# InventoryService.__init__
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are **instance-level fields**. If two requests hit the same `InventoryService` instance concurrently (which is the entire point of this service), the following race occurs:

- Request A acquires lock, starts heartbeat → sets `_heartbeat_active = True`
- Request B acquires lock, calls `_start_heartbeat` → also sets `_heartbeat_active = True`, overwrites `_heartbeat_thread`
- Request A finishes, calls `_stop_heartbeat` → sets `_heartbeat_active = False`
- **Request B's heartbeat is now dead.** Its lock will expire mid-operation.
- Worse: Request A's `finally` block may now delete Request B's lock reference.

The heartbeat state must be scoped **per-call**, not per-instance. The fix is to pass a local `threading.Event` into the heartbeat thread:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()  # local to this call
    acquired = False

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(...)

    heartbeat_thread = self._start_heartbeat(lock_key, token, stop_event)

    try:
        yield token
    finally:
        stop_event.set()
        heartbeat_thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        # then release lock
```

### 🔴 Critical — Input Validation Is Claimed but Not Enforced

The checklist says "Input validation via Pydantic" but `reserve_item` accepts raw, unvalidated arguments:

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

There is **no validation** on:
- `item_id` being empty, whitespace, or a path-traversal/injection string
- `quantity` being zero, negative, or absurdly large

This is a government contracting context. Unvalidated `item_id` flowing directly into Redis key construction and DB queries is a meaningful attack surface.

```python
lock_key = f"inventory:lock:{item_id}"  # item_id="../../admin" or "lock:*" etc.
```

A request model should be defined and validated at the boundary:

```python
from pydantic import BaseModel, Field, field_validator

class ReservationRequest(BaseModel):
    item_id: str = Field(min_length=1, max_length=128, pattern=r'^[a-zA-Z0-9_\-]+$')
    quantity: int = Field(ge=1, le=1000)
```

### 🟠 High — DB Operations Are Not Transactional

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` throws, stock is decremented but no order exists. The Redis lock provides mutual exclusion but **not durability**. This needs either:

- A DB-level transaction wrapping both calls, or
- Explicit rollback/compensating logic in an `except` block

This is especially important given the heartbeat complexity — if the process crashes mid-lock, the heartbeat dies, the lock eventually expires, and another process can enter while the DB is in a partially updated state.

---

### 🟠 High — Heartbeat Thread Is Never Joined on Stop

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting a flag is not sufficient. The thread may be mid-sleep (`time.sleep(10)`) when the flag is set, meaning it can outlive the lock release and attempt to extend a lock that has already been deleted or re-acquired by another process. Use `threading.Event` with `.wait()` instead of `time.sleep()`, and `.join()` the thread before releasing the lock.

---

### 🟡 Medium — Lock Release Has a TOCTOU Race

```python
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

The `GET` + `DELETE` is not atomic. Between the `GET` and `DELETE`, the lock could expire and be re-acquired by another process, which you then delete. This is a well-known Redis distributed lock pitfall. The correct approach is a Lua script:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```python
RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

self.redis.eval(RELEASE_SCRIPT, 1, lock_key, token)
```

---

### 🟡 Medium — Tests Mock Away the Exact Code Under Scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

The heartbeat is the most operationally risky part of this PR, and the tests completely bypass it. There should be at least one integration-style test that lets the heartbeat thread run and verifies TTL extension behavior. The current test suite gives false confidence.

---

### 🟡 Medium — `quantity` Defaults to 1 Silently

Minor but worth noting: a caller passing `quantity=0` will receive a `success=False` with message `"Insufficient stock"` and `remaining_stock=10`, which is misleading. A zero or negative quantity should be a validation error, not a stock check failure.
