Decision: Request Changes

Summary: The heartbeat renewal mechanism is architecturally insufficient as a correctness guarantee because it cannot protect against process-level pauses — a stop-the-world GC event, VM live migration, or OS scheduling preemption freezes **every thread in the process simultaneously**, including the heartbeat thread. When the process resumes, the lock TTL has already expired and may have been acquired by another worker; the zombie process then proceeds to write stale state to the database with no awareness that it has lost authority. Additionally, the service instance is not safe for concurrent use — `_heartbeat_active` and `_heartbeat_thread` are instance-level fields, meaning two simultaneous calls to `reserve_item` on the same `InventoryService` instance will corrupt each other's heartbeat state. These are not edge cases to be dismissed — in a flash sale context under high load, GC pressure is elevated and the probability of a multi-hundred-millisecond pause is non-trivial.

---

### Critical Flaw: Zombie Leader / Process Pause Race

The specific failure sequence the heartbeat **cannot prevent**:

1. Worker A acquires the lock. Reads `current_stock = 5`.
2. A stop-the-world GC pause freezes the entire Worker A process — heartbeat thread included.
3. After 30 seconds, the lock TTL expires naturally.
4. Worker B acquires the lock. Reads `current_stock = 5`. Writes `new_stock = 4`. Releases lock.
5. Worker A resumes from the pause. It has no knowledge that time passed or that the lock expired. It calculates `new_stock = 5 - 1 = 4` and calls `db.update_stock("item_001", 4)`.
6. **Result:** Stock is `4` but should be `3`. One unit has been sold twice.

The heartbeat is a liveness optimization — it reduces the probability of unnecessary lock expiry under slow-but-running conditions. It provides **zero safety guarantee** against pauses, because safety requires that a stale writer be *prevented from committing*, not merely *less likely to lose the lock*. These are categorically different properties.

---

### Required Fix: Fencing Token / Optimistic Lock at the Write Layer

The database write must be made conditional on the state that was observed at read time. The canonical pattern is an optimistic concurrency check:

```python
# Instead of unconditional:
self.db.update_stock(item_id, new_stock)

# Require a conditional update that fails if stock has changed:
rows_affected = self.db.update_stock_conditional(
    item_id,
    new_stock=new_stock,
    expected_stock=current_stock   # fencing condition
)
if rows_affected == 0:
    # Another writer committed between our read and write — abort
    raise StaleInventoryError(f"Concurrent modification detected for {item_id}")
```

The underlying SQL must be:

```sql
UPDATE inventory
SET stock = %(new_stock)s
WHERE item_id = %(item_id)s
  AND stock = %(expected_stock)s
```

A monotonically incrementing version column (classic optimistic locking) is the more robust canonical form:

```sql
UPDATE inventory
SET stock = %(new_stock)s, version = version + 1
WHERE item_id = %(item_id)s
  AND version = %(observed_version)s
```

Zero rows affected means the fencing condition failed — the zombie write is rejected at the database level regardless of what the distributed lock says. This is the only layer that can provide a hard correctness guarantee, because the database is the single source of truth and its write serialization is not subject to process-level pauses in the application tier.

---

### Secondary Flaw: Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are shared mutable fields on the service instance. If two threads call `reserve_item` concurrently on the same instance (which is the entire point of this service), the second call's `_start_heartbeat` will overwrite `_heartbeat_thread`, and the first call's `_stop_heartbeat` will set `_heartbeat_active = False`, terminating the second call's heartbeat prematurely. The heartbeat state must be scoped to the lock acquisition context, not the service instance — pass it through a local closure or a per-call state object.

### Minor Issues

- **`quantity` is not validated to be positive.** `reserve_item("item_001", quantity=0)` would succeed and create a zero-unit order. `reserve_item("item_001", quantity=-5)` would *increase* stock. Add a guard: `if quantity < 1: raise ValueError(...)`.
- **`item_id` is not validated for format/length.** A malicious or buggy caller could pass an arbitrarily long string that becomes the Redis key and log entry. Enforce a max length and character allowlist.
- **`db.update_stock` and `db.create_order` are not atomic with respect to each other.** If `create_order` raises after `update_stock` succeeds, stock is decremented but no order exists. These two operations should be wrapped in a database transaction.
- **The test suite mocks out the heartbeat entirely**, which means the instance-level state collision described above is never exercised. Tests should also cover concurrent invocations on the same instance.
- **`RuntimeError` is too broad a catch.** If `db.get_stock` raises a `RuntimeError` for an unrelated reason inside the lock boundary, it will be silently swallowed and returned as "Service temporarily unavailable," masking the real error and leaving the lock to be cleaned up without logging the true cause.

---

The distributed lock is a reasonable first layer, but it must be paired with a fencing token at the write layer before this is safe to merge into any production path. The heartbeat is not a substitute for that guarantee and should not be presented as one.
