**Decision: Request Changes**

**Summary:** The heartbeat thread is well-implemented and the lock *acquisition* story is solid, but the critical write path — `update_stock` followed by `create_order` — executes with no fencing token passed to the database layer and no conditional-write guard at the storage level. This means that if the lock expires between the heartbeat's last check and the DB write (GC pause, network stall, slow disk flush), a second process can acquire the lock and read stale stock before the first process's write lands, resulting in oversell. The heartbeat reduces the probability of this window but does not close it; a lock that *might* have expired when the write executes is a suggestion, not a lock.

---

### The core problem

The heartbeat thread and the DB write run on different threads with no synchronization between them. The sequence that breaks this:

```
T=0   Process A acquires lock, heartbeat starts
T=10  Heartbeat extends TTL → lock good for another 30s
T=11  Process A hits a GC pause / slow DB connection
T=41  Lock expires (heartbeat was paused too, it lives in the same process)
T=42  Process B acquires lock, reads stock=10, writes stock=9
T=43  Process A resumes, reads stock=10 (stale read from its own earlier call),
      writes stock=9 again → oversell by 1
```

The heartbeat does not protect against this because it is in-process. A JVM/CPython GC pause, a `time.sleep` overrun, or a kernel scheduling delay pauses *both* the heartbeat thread and the worker thread simultaneously. The lock can expire without the worker knowing.

The PR's own test `test_lock_not_released_if_expired` acknowledges that the lock can expire mid-operation — but it only tests the *release* path. It does not test what happens to the write that already executed after the lock expired.

---

### What is missing: a fencing token at the write site

The fix is not a better heartbeat. The fix is to make the storage layer reject stale writes. The standard approach (Kleppmann §8, Chandra & Toueg):

**Option 1 — Monotonic fencing token written through to storage:**

```python
# Lock acquisition returns a fencing token (monotonically increasing integer,
# e.g. Redis INCR on a version key, or a Zookeeper epoch).
# The token is passed to every write.

def update_stock(self, item_id: str, new_stock: int, fence: int) -> bool:
    """
    Only applies the write if the stored fence < supplied fence.
    Implemented as a single atomic compare-and-write (SQL CHECK, Lua script,
    conditional DynamoDB write, etc.)
    """
    rows = self.db.execute(
        "UPDATE inventory SET stock = %s, fence = %s "
        "WHERE item_id = %s AND fence < %s",
        (new_stock, fence, item_id, fence)
    )
    return rows.rowcount == 1
```

**Option 2 — Optimistic locking / version column (no Redis dependency on the write path):**

```python
# Read returns (stock, version). Write is conditional on version unchanged.
stock, version = self.db.get_stock_versioned(item_id)
# ... check stock ...
updated = self.db.update_stock_if_version(item_id, new_stock, expected_version=version)
if not updated:
    # Another writer beat us; retry or surface conflict
    raise StaleWriteError(item_id)
```

Option 2 is often preferable because it makes the database — the authoritative store — the arbiter of correctness, rather than relying on Redis reachability.

### Secondary issues

**1. Instance-level heartbeat state is not re-entrant**

```python
self._heartbeat_active = False   # shared across all concurrent calls
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `reserve_item` is called concurrently on the same `InventoryService` instance (e.g., in a thread pool), the second call's `_stop_heartbeat()` will kill the first call's heartbeat. The heartbeat state must be scoped per lock acquisition, not per service instance.

```python
# Fix: pass a mutable stop-flag into _start_heartbeat
stop_event = threading.Event()
self._start_heartbeat(lock_key, token, stop_event)
# ...
stop_event.set()
```

**2. `update_stock` and `create_order` are not atomic**

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If the process crashes between these two lines, stock is decremented but no order exists — inventory is silently lost. These must be wrapped in a single database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

**3. The heartbeat check-then-extend is not atomic**

```python
current = self.redis.get(lock_key)          # read
if current and current.decode() == token:
    self.redis.expire(lock_key, self.LOCK_TTL)  # write
```

Between `GET` and `EXPIRE` the lock can expire and be acquired by another process. The extend must be a single atomic operation:

```lua
-- Lua script executed atomically in Redis
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("expire", KEYS[1], ARGV[2])
else
    return 0
end
```

**4. Tests mock away the exact behavior under review**

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Mocking the heartbeat is reasonable for unit tests, but there is no integration test that exercises lock expiry mid-write. The test suite passes precisely because it never exercises the failure mode this PR is supposed to handle.

---

### What I'd need to see before approving

| # | Required change |
|---|----------------|
| 1 | Fencing token or optimistic version check at the DB write site — the write must be rejected if it arrives after a newer writer has already committed |
| 2 | Per-acquisition heartbeat stop flag (not instance-level boolean) |
| 3 | `update_stock` + `create_order` wrapped in a single DB transaction |
| 4 | Atomic Lua script for heartbeat TTL extension |
| 5 | At least one test that simulates lock expiry *before the write* and asserts the write is rejected, not just that the lock is not deleted |

The heartbeat, the backoff, the ownership token on release — these are all good. The gap is that none of them protect the write itself.
