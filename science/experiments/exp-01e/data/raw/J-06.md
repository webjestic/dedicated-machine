Decision: Request Changes

Summary: The implementation has solid bones — the lock ownership check, heartbeat pattern, and exponential backoff are all reasonable starting points. However, there are several correctness, safety, and architecture issues that must be resolved before this touches production, particularly given the concurrent flash-sale workload this is explicitly designed for.

---

## Critical Issues

### 1. Instance-Level Heartbeat State is Not Thread-Safe

```python
# Current — shared mutable state on the instance
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

`_heartbeat_active` and `_heartbeat_thread` are stored on `self`. If two requests hit the same `InventoryService` instance concurrently (which is the entire point of this service), one call's `_stop_heartbeat()` will kill the other call's heartbeat thread mid-operation. The lock can then expire while a legitimate transaction is still running, causing a split-brain scenario where two callers believe they hold the lock.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    heartbeat_active = threading.Event()
    heartbeat_active.set()

    def _beat():
        while heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)

    acquired = False
    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(f"Could not acquire lock for {lock_key} after {retry_attempts} attempts")

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.clear()
        thread.join(timeout=self.HEARTBEAT_INTERVAL + 1)
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
```

### 2. Lock Release is Not Atomic

```python
# Current — two separate round trips, not atomic
current = self.redis.get(lock_key)
if current and current.decode() == token:
    self.redis.delete(lock_key)
```

Between the `GET` and the `DELETE`, another process could acquire the lock. You would then delete a lock you no longer own. This is a well-known Redis distributed lock pitfall. The canonical fix is a Lua script that makes check-and-delete atomic.

```python
RELEASE_SCRIPT = """
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
"""

def _release_lock(self, lock_key: str, token: str) -> None:
    script = self.redis.register_script(RELEASE_SCRIPT)
    released = script(keys=[lock_key], args=[token])
    if released:
        logger.info("Lock released key=%s", lock_key)
    else:
        logger.warning("Lock already expired or stolen key=%s", lock_key)
```

The same race exists in the heartbeat thread's `GET` + `expire` — that should also be a Lua script or replaced with a `SET ... XX` call.

### 3. Database Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

If `update_stock` succeeds and `create_order` throws, you have decremented stock with no corresponding order record. Inventory is permanently lost with no audit trail. These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract needs to expose a transaction context manager, and the tests need to verify rollback behavior on partial failure.

## Significant Issues

### 4. Input Validation Is Claimed But Not Present

The checklist says "Input validation via Pydantic" but neither `item_id` nor `quantity` are validated anywhere in `reserve_item`. Pydantic is imported and used only for the *output* model. There is no guard against:

- `quantity <= 0` (reserve zero or negative units)
- `item_id` being an empty string or containing injection-relevant characters
- `quantity` being a non-integer

class ReservationRequest(BaseModel):

def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    req = ReservationRequest(item_id=item_id, quantity=quantity)

### 5. Heartbeat Thread Is Never Joined

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not wait for the thread to actually stop. The thread could still be mid-execution — specifically mid-`expire` call — after the lock has been deleted and potentially re-acquired by another process. You need `thread.join()` before the lock release check, as shown in the fix for issue #1.

---

### 6. Retry Backoff Blocks the Calling Thread

```python
time.sleep(backoff)
```

Under flash-sale load, this means every waiting request is holding a thread for up to 300ms (attempt 2) or 700ms total across retries. At scale this will exhaust your thread pool. Consider making the retry behavior configurable or documenting the threading model assumption explicitly so the caller can make an informed decision about async alternatives.

## Test Coverage Gaps

### 7. The Atomicity Bug Is Not Tested

There is no test for the scenario where `create_order` raises after `update_stock` succeeds. This is the most dangerous failure mode and it has zero coverage.

```python
def test_reserve_partial_db_failure(service, db):
    db.update_stock.return_value = None
    db.create_order.side_effect = Exception("DB connection lost")

    with pytest.raises(Exception):
        service.reserve_item("item_001", quantity=1)

    # Stock must be rolled back — currently this will FAIL, exposing the bug
    db.update_stock.assert_called_with("item_001", 10)  # rollback expected
```

### 8. Concurrent Access Is Not Tested

The primary claim of this PR is correctness under concurrency, but there is no test that exercises two threads calling `reserve_item` simultaneously. Given that the instance-level heartbeat state bug (issue #1) only manifests under concurrent load, this is a meaningful gap.

### 9. Mock Bypasses the Token Comparison Logic

```python
redis_client.get.return_value = b"test-token"
```

The token generated inside `_acquire_lock` is a fresh `uuid4()`, so it will never equal `b"test-token"`. The test `test_lock_not_released_if_expired` accidentally passes for the wrong reason, and the normal release path in `test_reserve_success` is also never actually exercised. Use `unittest.mock.patch('uuid.uuid4', return_value=...)` to control the token value and make these assertions meaningful.

## Minor Notes

- `LOCK_TTL = 30` with `HEARTBEAT_INTERVAL = 10` is reasonable, but both should be configurable via constructor parameters or environment config, not hardcoded class attributes. Flash sale SLAs may require tuning.
- The `RuntimeError` used for lock failure is too broad a type. A domain-specific `LockAcquisitionError(Exception)` would allow callers to distinguish this case from unexpected runtime errors.
- Log line `"Lock released key=%s"` fires at `INFO` but the lock-stolen warning path is silent in the current code. Both paths should log at appropriate levels with the token suffix for traceability.
