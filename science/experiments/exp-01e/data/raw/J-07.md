Decision: Request Changes

Summary: The implementation has a critical thread-safety flaw — storing heartbeat state as instance variables (`_heartbeat_active`, `_heartbeat_thread`) makes `InventoryService` non-reentrant, meaning concurrent calls on a shared instance will corrupt each other's lock lifecycle. There are also meaningful architectural and correctness concerns around the DB operations lacking transactional guarantees, missing input validation on the public interface, and a race condition in the heartbeat's token check. These issues need to be resolved before this can be merged into a production path handling flash sale inventory.

---

### Detailed Findings

#### 🔴 Critical — Instance-Level Heartbeat State (Thread Safety)

```python
# Current — broken under concurrency
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

If `InventoryService` is instantiated once and shared (typical in web frameworks via DI containers, module-level singletons, etc.), two concurrent calls to `reserve_item` will race on `_heartbeat_active`. The second call's `_stop_heartbeat()` will kill the first call's heartbeat, potentially letting that lock expire mid-operation.

**Fix:** Scope heartbeat state to the lock context, not the instance.

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    acquired = False
    heartbeat_active = threading.Event()  # local scope, not self

    for attempt in range(retry_attempts):
        acquired = self.redis.set(lock_key, token, nx=True, ex=self.LOCK_TTL)
        if acquired:
            break
        backoff = (2 ** attempt) * 0.1
        logger.warning(
            "Lock contention key=%s attempt=%d retrying in %.1fs",
            lock_key, attempt + 1, backoff
        )
        time.sleep(backoff)

    if not acquired:
        raise RuntimeError(
            f"Could not acquire lock for {lock_key} after {retry_attempts} attempts"
        )

    def _beat() -> None:
        while not heartbeat_active.wait(timeout=self.HEARTBEAT_INTERVAL):
            # wait() returns False on timeout, True on set() — invert logic
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
                logger.debug("Lock TTL extended key=%s", lock_key)

    thread = threading.Thread(target=_beat, daemon=True)
    thread.start()

    try:
        yield token
    finally:
        heartbeat_active.set()   # signals the thread to stop
        thread.join(timeout=2)   # don't leave threads dangling
        current = self.redis.get(lock_key)
        if current and current.decode() == token:
            self.redis.delete(lock_key)
            logger.info("Lock released key=%s", lock_key)
```

Note the additional fix here: the original heartbeat uses `time.sleep()` which is not interruptible. Using `threading.Event.wait(timeout=...)` allows clean shutdown. Also, `thread.join()` is now called — the original code never joins the thread, meaning threads can outlive their context and pile up under load.

#### 🔴 Critical — No Input Validation on Public Interface

The checklist claims "Input validation via Pydantic" but `reserve_item` accepts raw primitives with zero validation. A caller can pass `quantity=0`, `quantity=-5`, or an empty `item_id` string and the code will happily acquire a distributed lock and attempt DB operations.

```python
# quantity=0 passes the stock check and creates a zero-unit order
# quantity=-1 would *increase* stock — silent data corruption
if current_stock < quantity:  # -1 < 10 is False, so this passes
    ...
new_stock = current_stock - quantity  # 10 - (-1) = 11
```

**Fix:** Validate at the boundary.

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
    if not item_id or not item_id.strip():
        return ReservationResult(success=False, message="item_id must not be empty")
    if quantity < 1:
        return ReservationResult(success=False, message="quantity must be >= 1")
```

#### 🟠 High — DB Operations Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate DB calls inside a Redis lock. If `update_stock` succeeds and `create_order` raises, you have decremented stock with no corresponding order record. The Redis lock does not protect against this — it only serializes access, it does not roll back.

**Fix:** These two operations must be wrapped in a database transaction.

```python
with self.db.transaction():
    self.db.update_stock(item_id, new_stock)
    self.db.create_order(order_id, item_id, quantity)
```

The `db` interface contract should enforce this. If the current `db` abstraction doesn't support transactions, that's a blocker that needs to be addressed at the interface level before this service can be considered safe.

#### 🟠 High — Heartbeat Race Condition on Token Check

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        if self._heartbeat_active:           # re-check after sleep
            current = self.redis.get(lock_key)
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)
```

There is a TOCTOU window between `redis.get(lock_key)` and `redis.expire(lock_key, ...)`. Another process could acquire the lock in that gap. The correct pattern here is a Lua script to make the check-and-extend atomic:

```python
EXTEND_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('expire', KEYS[1], ARGV[2])
else
    return 0
end
"""

def _extend_lock(self, lock_key: str, token: str) -> bool:
    result = self.redis.eval(
        EXTEND_SCRIPT, 1, lock_key, token, self.LOCK_TTL
    )
    return bool(result)
```

The same atomic pattern should also be applied to lock release (the `finally` block has the same TOCTOU issue).

#### 🟡 Medium — Tests Mock Away the Thing Being Tested

```python
@pytest.fixture
def service(redis_client, db):
    svc = InventoryService(redis_client, db)
    svc._start_heartbeat = MagicMock()   # heartbeat entirely bypassed
    svc._stop_heartbeat = MagicMock()
    return svc
```

Mocking `_start_heartbeat` and `_stop_heartbeat` means the tests provide zero coverage of the heartbeat lifecycle, the thread behavior, or the lock release path. The test `test_lock_not_released_if_expired` is the most important correctness test in the suite and it passes only because the mock sidesteps the actual code path.

**Fix:** Tests should mock at the Redis boundary, not at internal methods. Let the lock context manager run. Use `fakeredis` for deterministic integration-level tests of the locking behavior.

---

#### 🟡 Medium — `retry_attempts` Hardcoded, Not Configurable

```python
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
```

`LOCK_TTL` and `HEARTBEAT_INTERVAL` are class constants, but `retry_attempts` is a buried parameter default. For a service going into a government-facing production environment, retry policy should be a first-class configuration concern — injectable at construction time, not scattered as defaults on internal methods.

---

#### 🟡 Low — Bare `RuntimeError` Is Too Broad

```python
except RuntimeError as e:
    logger.error("Lock acquisition failed item_id=%s: %s", item_id, e)
```

Catching `RuntimeError` will swallow unrelated runtime errors from the DB layer or elsewhere. Define a specific exception type.

```python
class LockAcquisitionError(Exception):
    pass

# raise LockAcquisitionError(...) in _acquire_lock
# catch LockAcquisitionError in reserve_item
```
