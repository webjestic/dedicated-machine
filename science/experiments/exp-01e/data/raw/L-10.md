**Decision: Request Changes**

**Summary:** The heartbeat is a availability mechanism, not a safety mechanism — it reduces the probability of lock expiry but cannot eliminate it, and the database writes at the bottom of `reserve_item` have no fencing protection whatsoever. If the lock expires between the heartbeat's last extension and the `update_stock`/`create_order` calls (GC pause, network partition, slow DB, any reason), a second process can acquire the lock, read the same stock level, and both writes proceed — this is exactly the oversell scenario this PR is meant to prevent. There is also a secondary concurrency bug in the heartbeat implementation itself that makes the safety story worse than it appears.

---

## Critical Issues

### 1. No fencing on the write path

This is the core problem. The PR's safety argument is: *"the heartbeat keeps the lock alive, so the lock won't expire during the writes."* That is a liveness argument, not a safety argument. It says the bad thing is unlikely. It does not say the bad thing is impossible.

The sequence that breaks this:

```
Process A                          Process B
---------                          ---------
acquires lock
reads stock = 10
heartbeat extends TTL
                                   [A stalls — GC pause, slow network, anything]
                                   lock TTL expires
                                   acquires lock
                                   reads stock = 10
                                   update_stock(item, 9)   ← committed
                                   create_order(...)       ← committed
A resumes
update_stock(item, 9)   ← committed again, oversold
create_order(...)       ← committed again, double order
```

The heartbeat does not close this window. It narrows it. The window is any interval between the last heartbeat extension and the write completing. With a 10-second heartbeat interval and a 30-second TTL, that window is up to 20 seconds — easily long enough for a GC pause or a slow database call.

The fix is a fencing token. Redis's `WAIT` command and Lua scripts can help at the Redis layer, but the real protection has to be at the storage layer. The standard approaches:

**Option A — Monotonic fencing token checked by the database:**
```python
# Redis returns a monotonically increasing token (e.g. from a counter)
# The DB write includes a WHERE clause that rejects stale tokens
self.db.update_stock_fenced(
    item_id,
    new_stock,
    fencing_token=token,          # storage layer rejects if token is stale
    expected_version=stock_version # optimistic concurrency on the row
)
```

**Option B — Optimistic concurrency / compare-and-swap at the DB layer:**
```python
# Read stock with a version number
current_stock, version = self.db.get_stock_with_version(item_id)

# Write only succeeds if version hasn't changed
rows_updated = self.db.update_stock_if_version(item_id, new_stock, version)
if rows_updated == 0:
    raise ConcurrentModificationError("Stock modified by another process")
```

Option B is often more practical because it doesn't require the storage layer to know about the distributed lock. The lock becomes a contention-reduction mechanism (which is a legitimate use), and the DB transaction is the actual safety boundary.

### 2. Heartbeat has a TOCTOU race that makes expiry more likely than it looks

```python
def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)   # sleeps 10s
        if self._heartbeat_active:
            current = self.redis.get(lock_key)           # ← two round trips
            if current and current.decode() == token:
                self.redis.expire(lock_key, self.LOCK_TTL)  # ← not atomic
```

The `GET` and `EXPIRE` are two separate commands. Between them, the lock can expire and be acquired by another process. The heartbeat then extends a lock it no longer owns. This is exactly the scenario `_stop_heartbeat` is trying to prevent, but the check doesn't prevent it — it just makes it less likely.

This should be a single atomic Lua script:

```lua
-- Extend TTL only if we still own the lock
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

```python
EXTEND_SCRIPT = redis.register_script("""
    if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('EXPIRE', KEYS[1], ARGV[2])
    else
        return 0
    end
""")

def _beat() -> None:
    while self._heartbeat_active:
        time.sleep(self.HEARTBEAT_INTERVAL)
        result = EXTEND_SCRIPT(keys=[lock_key], args=[token, self.LOCK_TTL])
        if result == 0:
            logger.warning("Lost lock during heartbeat key=%s", lock_key)
            self._heartbeat_active = False
            break
```

The same atomicity problem exists in `_acquire_lock`'s release path, which already uses a Lua-equivalent pattern via `GET` + conditional `DELETE` — that should also be a single Lua script.

### 3. `_heartbeat_active` is a shared boolean with no memory visibility guarantee

```python
self._heartbeat_active = False  # written by main thread
                                # read by heartbeat thread
```

Python's GIL makes this safe in CPython in practice, but it is not guaranteed by the language memory model and it will not hold under alternative runtimes. Use `threading.Event`:

```python
self._heartbeat_stop = threading.Event()

def _beat():
    while not self._heartbeat_stop.wait(timeout=self.HEARTBEAT_INTERVAL):
        # extend lock ...

def _stop_heartbeat():
    self._heartbeat_stop.set()
    if self._heartbeat_thread:
        self._heartbeat_thread.join(timeout=5)
```

`Event.wait(timeout=...)` also eliminates the `time.sleep` + double-check pattern, which has its own race.

### 4. Instance-level heartbeat state breaks concurrent calls

```python
class InventoryService:
    def __init__(self, ...):
        self._heartbeat_active = False       # instance-level
        self._heartbeat_thread = None        # instance-level
```

If two threads call `reserve_item` on the same `InventoryService` instance concurrently (which is the entire point of this service), they share `_heartbeat_active` and `_heartbeat_thread`. The second call's `_start_heartbeat` overwrites the first call's thread reference. The first call's `_stop_heartbeat` sets `_heartbeat_active = False` and stops both heartbeats.

This state must be local to each lock acquisition, not to the instance:

```python
@contextmanager
def _acquire_lock(self, lock_key: str, retry_attempts: int = 3):
    token = str(uuid.uuid4())
    stop_event = threading.Event()
    # pass stop_event into the heartbeat thread, not self._heartbeat_active
```

## Test Coverage Gaps

The tests mock away the heartbeat entirely (`svc._start_heartbeat = MagicMock()`), which means none of the concurrency behavior is tested at all. The test named `test_lock_not_released_if_expired` tests the release path but not the write path after expiry — which is the actual failure mode.

The critical missing test:

```python
def test_no_oversell_if_lock_expires_before_write():
    """
    If the lock expires after the stock check but before the write,
    the write must fail or be rejected — not silently succeed.
    """
    # This test should fail with the current implementation,
    # demonstrating the gap that fencing is meant to close.
```

If you cannot write a passing test for this scenario, that is the proof that the safety guarantee is missing.

---

## What Approval Requires

1. **Fencing on the write path** — either optimistic concurrency at the DB layer (version check on `update_stock`) or a storage-layer fencing token. The heartbeat can stay as a latency optimization but cannot be the safety mechanism.
2. **Atomic heartbeat extension** via Lua script.
3. **Per-acquisition heartbeat state** moved off the instance.
4. **`threading.Event`** for stop signaling with `join` on teardown.
5. **A test that covers the expired-lock-before-write scenario** and demonstrates the fencing actually rejects the stale write.

The engineering here is solid in structure — the lock acquisition pattern, the ownership verification on release, the backoff logic, the result modeling. The gap is specifically that the lock protects the read but not the write, and in a flash sale oversell scenario, that is the gap that matters.
