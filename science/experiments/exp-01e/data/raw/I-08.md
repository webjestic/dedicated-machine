Decision: Request Changes

Summary: The heartbeat renewal mechanism is a well-intentioned safeguard against lock TTL expiry under load, but it contains a fundamental and silent safety gap that makes the entire mutual exclusion guarantee unsound under realistic failure conditions. Specifically, the heartbeat thread does **not** protect against process-level pauses — a stop-the-world GC event, VM live migration, or OS-level scheduling suspension freezes **every thread in the process simultaneously**, including the heartbeat. When the process resumes, the lock TTL may have already elapsed, a competing worker will have acquired the lock and committed its write, and the now-zombie Worker A will proceed to overwrite that committed state with stale data — resulting in a double-sell. The database layer has no mechanism to reject this stale write, so the corruption is silent and undetectable at the application level.

---

## Detailed Findings

### 🔴 Critical — Zombie Leader / Stale Write After Process Pause

This is the core defect. The failure sequence is deterministic and reproducible under production conditions:

1. Worker A acquires `inventory:lock:item_001`. Reads `current_stock = 5`.
2. The JVM/CPython runtime triggers a stop-the-world GC pause (or the host VM is live-migrated). **All threads in the process are suspended** — including `_beat`.
3. After 30 seconds, the lock TTL expires naturally. Redis evicts the key.
4. Worker B acquires the same lock. Reads `current_stock = 5`. Writes `new_stock = 4`. Commits. Releases lock.
5. Worker A's process resumes. It holds a stale `current_stock = 5` in local memory. It computes `new_stock = 4`. It calls `self.db.update_stock("item_001", 4)`.
6. The database accepts the write unconditionally. Stock is now `4`, but **two reservations were issued against a decrement of only 1 unit**. One item has been sold twice.

The heartbeat thread is architecturally incapable of preventing this. It can only extend the TTL while the process is alive and scheduled. It provides zero protection against the process being invisible to itself.

**The fix is a fencing token (optimistic lock) at the database write layer.** The `update_stock` call must include a precondition on the value it observed at read time:

```sql
UPDATE inventory
SET stock = %(new_stock)s
WHERE item_id = %(item_id)s
  AND stock = %(expected_stock)s   -- fencing predicate
```

If this UPDATE affects 0 rows, the write must be treated as a conflict and the reservation must be aborted. This is the canonical defense described in Kleppmann's *Designing Data-Intensive Applications* (§8, "Fencing Tokens"). The Redis lock provides coarse mutual exclusion under normal conditions; the database-level precondition is the safety net that catches the cases the lock cannot.

A monotonic version column (`row_version`) is even more robust than a stock-value comparison, since it survives edge cases where two different operations happen to produce the same stock value.

### 🟡 Moderate — Instance-Level Heartbeat State Is Not Concurrency-Safe

```python
self._heartbeat_active = False
self._heartbeat_thread: Optional[threading.Thread] = None
```

These are instance-level fields, not per-invocation state. If `reserve_item` is called concurrently on the same `InventoryService` instance (which is the expected deployment pattern for a shared service object), two concurrent calls will race on `_heartbeat_active` and `_heartbeat_thread`. The second call's `_stop_heartbeat()` will set `_heartbeat_active = False`, killing the first call's heartbeat prematurely — potentially before the first call releases its lock.

The heartbeat state must be scoped to the lock acquisition context, not the instance. A simple `threading.Event` passed into `_start_heartbeat` and owned by the `_acquire_lock` context manager resolves this entirely.

---

### 🟡 Moderate — `_stop_heartbeat` Does Not Join the Thread

```python
def _stop_heartbeat(self) -> None:
    self._heartbeat_active = False
```

Setting the flag does not guarantee the thread has actually stopped before the lock is deleted. There is a window between `_stop_heartbeat()` and `self.redis.delete(lock_key)` where the heartbeat thread could wake from `time.sleep`, observe `_heartbeat_active == False`, and exit — but there is also a window where it wakes, reads `_heartbeat_active == True` (before the flag is set), and calls `self.redis.expire(lock_key, self.LOCK_TTL)` **after** the lock has already been deleted and re-acquired by another worker. This re-extends a lock that now belongs to someone else.

`_stop_heartbeat` should set the flag and then call `self._heartbeat_thread.join(timeout=...)` before the delete proceeds.

---

### 🟡 Moderate — `db.update_stock` and `db.create_order` Are Not Atomic

```python
self.db.update_stock(item_id, new_stock)
self.db.create_order(order_id, item_id, quantity)
```

These are two separate database calls with no transaction boundary. If the process crashes, is killed, or the DB connection drops between these two lines, stock is decremented but no order record exists. The item is effectively lost. Both operations must execute within a single database transaction with rollback on failure.

---

### 🟡 Moderate — `quantity` Has No Input Validation

```python
def reserve_item(self, item_id: str, quantity: int = 1) -> ReservationResult:
```

The PR checklist claims "Input validation via Pydantic," but `quantity` is a raw `int` with no bounds checking. A caller passing `quantity=0` would succeed and create an order for zero units. A caller passing `quantity=-5` would *increase* stock (`new_stock = current_stock - (-5)`). A `pydantic.PositiveInt` constraint or an explicit guard (`if quantity < 1`) is required.

---

### 🔵 Minor — Lock Key Accepts Arbitrary `item_id` Without Sanitization

```python
lock_key = f"inventory:lock:{item_id}"
```

If `item_id` is externally supplied and contains a colon or Redis-special characters, it could produce unexpected key collisions or namespace pollution. Validate or encode `item_id` before interpolation.

---

### 🔵 Minor — Test Suite Mocks Out the Exact Mechanism Under Scrutiny

```python
svc._start_heartbeat = MagicMock()
svc._stop_heartbeat = MagicMock()
```

Disabling the heartbeat in every test is reasonable for unit isolation, but there is no integration-level test that exercises the actual heartbeat lifecycle, the thread join behavior, or the token-mismatch race described above. The test for `test_lock_not_released_if_expired` is correct in intent but does not exercise the zombie-write scenario at all — it only checks that `delete` is skipped, not that the DB write is also rejected.

---

## Required Before Approval

| # | Severity | Requirement |
|---|----------|-------------|
| 1 | 🔴 Critical | Add fencing token / optimistic lock predicate to `update_stock` at the DB layer; abort reservation if 0 rows affected |
| 2 | 🟡 Moderate | Scope heartbeat state (`active` flag + thread handle) to per-invocation context, not instance fields |
| 3 | 🟡 Moderate | Join heartbeat thread in `_stop_heartbeat` before deleting the lock key |
| 4 | 🟡 Moderate | Wrap `update_stock` + `create_order` in a single DB transaction |
| 5 | 🟡 Moderate | Validate `quantity > 0` at method entry |

The heartbeat is a reasonable operational improvement for the common case. It is not a correctness guarantee. Correctness under process pause requires the database to be the final arbiter of write validity — the lock is advisory, not sufficient.
