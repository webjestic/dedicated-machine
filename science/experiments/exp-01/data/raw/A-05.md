Decision: Needs Clarification

Summary: The tests pass and the caching logic is internally consistent, but this PR introduces a module-level mutable dictionary that is fundamentally unsafe in any multi-instance deployment — and I cannot tell from this PR whether that describes production. Before this merges, I need answers to the deployment questions below, and there is one unconditional correctness issue that needs to be fixed regardless of the answers.

---

## Issues

### 🔴 Blocking — Correctness regardless of deployment model

**No cache TTL or size bound.**

`_preferences_cache` grows without limit and never expires entries on its own. In a long-running process this is a memory leak. A user whose preferences are cached at startup will never see an updated value unless something explicitly calls `invalidate_preferences_cache` — and that call can be missed (exception path, a direct DB write from another service, a migration script, an admin tool). "Preferences rarely change" is not the same as "preferences never change," and silent staleness is worse than a cache miss.

This needs at minimum a TTL-based eviction strategy. `cachetools.TTLCache` is a one-import fix. Rolling your own with a timestamp dict is also fine, but it needs to exist.

```python
# Example — drop-in replacement, thread-safe, bounded, TTL-based
from cachetools import TTLCache
_preferences_cache = TTLCache(maxsize=10_000, ttl=300)
```

### 🔴 Blocking — Deployment topology (must answer before merge)

**Is this service deployed as more than one instance?**

If yes — and most services behind a load balancer are — this cache is not shared across instances. Instance A caches user_123's preferences. The user updates their preferences. The write goes to the DB. `invalidate_preferences_cache("user_123")` is called on Instance A. Instance B still holds the stale value and will serve it until its process restarts or the entry expires (which, as noted above, never happens today).

The PR description says "preferences rarely change." That is a product assumption, not a safety guarantee. The question I need answered:

1. **How many instances run in production?** Single process, single machine? Auto-scaling group? Kubernetes with multiple replicas?
2. **Is there a load balancer in front of this service?** Sticky sessions or round-robin?
3. **Where does the write path live?** Is `invalidate_preferences_cache` called in the same service that handles preference updates, or is the write path in a different service entirely?
4. **Is there any shared cache layer already in the stack** (Redis, Memcached)? If so, why is this not using it?

If the answer is "multiple instances, no sticky sessions, no shared cache," this implementation is not safe to ship. The fix is either: use the existing shared cache layer, add one, or scope the cache to a request context rather than the module.

---

### 🟡 Non-blocking but should be addressed

**Thread safety.**

A plain `dict` in CPython has incidental thread safety for individual `__getitem__` and `__setitem__` calls due to the GIL, but this is an implementation detail, not a guarantee, and it does not hold under alternative runtimes (PyPy, Jython) or if the service ever moves to `asyncio` with true concurrency. If you switch to `TTLCache` as suggested above, wrap access with a `threading.Lock` or use `cachetools.func.ttl_cache` which handles this for you.

**The tests do not test the thing that can go wrong.**

The existing tests verify the happy path in a single-process, single-thread context. They do not test staleness, they do not test concurrent access, and they cannot test cross-instance invalidation by design. I am not asking you to add a distributed systems integration test to a unit test suite — but the test file should have a comment acknowledging that cross-instance consistency is out of scope for these tests and is a deployment concern, so the next engineer does not read green tests and assume the problem is solved.

**`invalidate_preferences_cache` is not called anywhere in this diff.**

Where is it called? If it lives in the same module as the write path, show that. If it is the caller's responsibility, document it. An invalidation function that exists but is not wired up is not an invalidation strategy.

## What I need to approve this

1. Answers to the deployment topology questions above.
2. TTL and size bound on the cache (unconditional).
3. A clear answer on where `invalidate_preferences_cache` is called in the write path.

If the deployment answer is "single instance, no horizontal scaling planned," I will approve with the TTL fix. If the answer is "multiple instances," the design needs to change before this merges — not after.
