**Decision:** Request Changes / Needs Clarification

**Summary:** This PR introduces a module-level mutable cache with no TTL, no thread safety, no eviction policy, and no bounds on memory growth — and critically, it assumes a single-process deployment model that may not hold in production. Before any of the implementation concerns can be resolved, I need answers about the runtime environment. The tests validate the happy path but leave the most important failure modes completely uncovered.

---

## Review

### 🔴 Blocking — Deployment Model (Must Answer Before This Merges)

The central assumption baked into this entire approach is that there is one process, one cache, and one source of truth. I need to know:

1. **How is this service deployed?** Single instance? Multiple replicas behind a load balancer? Auto-scaling group? If there are N instances, there are N independent caches. A write that invalidates the cache on instance A leaves instances B through N serving stale data indefinitely.

2. **Is `invalidate_preferences_cache` called from within this service, or is it expected to be called by other services?** If a separate service updates preferences and calls invalidation over an API boundary, only the instance that received that call gets invalidated.

3. **Is there any shared storage layer — Redis, Memcached, a distributed cache — already in the platform?** If yes, this PR is reinventing a wheel that already exists and doing it unsafely. If no, that conversation needs to happen before a cache is introduced.

4. **What is the expected write frequency for preferences?** The PR summary says "rarely change." Rarely means different things. Once per session? Once per day? Once per account lifetime? This determines whether a TTL-based approach is acceptable or whether invalidation must be exact.

I cannot answer "is this safe at scale?" from the information provided. I will not approve until I can.

---

### 🔴 Blocking — No TTL / No Eviction

```python
_preferences_cache = {}
```

This cache grows without bound and never expires. In a long-running process:

- Every user who ever makes a request gets a permanent entry.
- Preferences updated in the database are never reflected until the process restarts or `invalidate_preferences_cache` is explicitly called — and as noted above, that call may not reach every instance.
- There is no mechanism to reclaim memory.

A cache without a TTL is not a cache. It is a memory leak with optimistic framing. At minimum, entries need an expiry. Something like `cachetools.TTLCache` gives you both a TTL and a maximum size in a few lines. If the platform already has Redis, use it — that solves the multi-instance problem at the same time.

---

### 🔴 Blocking — Thread Safety

```python
if user_id in _preferences_cache:
    return _preferences_cache[user_id]

_preferences_cache[user_id] = result
```

Python's GIL makes individual dict operations atomic, but the check-then-act sequence here is not atomic. Under a threaded server (gunicorn with threads, uWSGI, any async framework running in a thread pool), two concurrent requests for the same `user_id` can both miss the cache, both query the database, and both write back. This is a benign race in this specific case — the result is just a redundant query — but it signals that thread safety was not considered, and that assumption will not hold if the cache logic becomes more complex (e.g., negative caching, write-through, or reference counting).

If you are on a multi-threaded server, use `threading.Lock` or switch to a thread-safe structure. If you are fully async (asyncio, no threads), document that assumption explicitly so the next person does not deploy this under gunicorn and spend an afternoon debugging it.

### 🟡 Significant — No Test for Stale Data After Update

The tests confirm that the cache is hit on a second call and that explicit invalidation forces a re-query. They do not test the scenario that will actually cause a production incident:

```python
def test_preferences_not_stale_after_update():
    # If preferences are updated in the DB without invalidation being called,
    # the cache returns the old value. Is this tested? Is this acceptable?
    ...
```

Whether this is a bug or a known limitation depends on the answers to the deployment questions above. But it needs to be a deliberate, documented decision — not an untested assumption.

Additional missing test cases:

- `invalidate_preferences_cache` called for a user who was never cached (should not raise — the `pop` handles this, but it should be tested explicitly)
- Concurrent access behavior under threading (even a basic smoke test)
- Cache behavior when `db.query` raises an exception — does a failed query poison the cache?

### 🟡 Significant — Exception Handling

```python
result = db.query(...)
_preferences_cache[user_id] = result
```

If `db.query` raises, the exception propagates and nothing is cached — that part is fine. But what does `db.query` return on a user with no preferences? An empty dict? `None`? An empty list? If it returns `None` and the caller does not check, you will cache `None` and return it on subsequent calls, potentially causing `TypeError` downstream when the caller tries to iterate or access keys. The contract of `db.query` needs to be explicit, and the return type of `get_user_preferences` should be documented and validated.

---

### 🟡 Significant — Mutable Cache Values

```python
_preferences_cache[user_id] = result
return result
```

If `result` is a mutable dict (which it almost certainly is), the caller receives a direct reference to the cached object. Any mutation by the caller mutates the cache:

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # now the cache is wrong
```

Return a copy:

```python
return dict(_preferences_cache[user_id])
# or for nested structures:
import copy
return copy.deepcopy(_preferences_cache[user_id])
```

---

### 🟡 Minor — Missing Type Annotations

The existing codebase context is not visible, but the new functions should carry type annotations consistent with whatever standard the project uses:

```python
_preferences_cache: dict[str, dict] = {}

def get_user_preferences(user_id: str) -> dict:
    ...

def invalidate_preferences_cache(user_id: str) -> None:
    ...
```

The return type of `get_user_preferences` in particular matters for callers — especially if `None` is a possible return value (see exception handling note above).

- The invalidation function is a reasonable interface — `pop` with a default is the right call, it will not raise on a cache miss.
- The test structure is clean and the mock usage is correct.
- The PR summary is honest about the motivation and scope.
