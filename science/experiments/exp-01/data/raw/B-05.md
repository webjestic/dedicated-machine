**Decision: Request Changes / Needs Clarification**

**Summary:** This PR introduces a module-level mutable cache with no TTL, no size bound, and no thread safety — and the tests validate only the happy path in a single-process context. Before any of the implementation concerns can be fully evaluated, I need answers about the deployment model, because the correctness of this entire approach depends on topology. I am not approving this in its current form.

---

## Deployment Model — Blocking Questions

These must be answered before I can assess whether this approach is viable at all.

**1. How many instances of this service run in production?**

A module-level dict (`_preferences_cache = {}`) is process-local. If this service runs as more than one instance — behind a load balancer, in a container orchestrator, anywhere — then:

- Instance A caches user_123's preferences.
- User_123 updates their preferences. The write goes to Instance B.
- Instance B calls `invalidate_preferences_cache("user_123")`.
- Instance A's cache is untouched. It continues serving stale data indefinitely.

This is not a theoretical edge case. This is the default failure mode of process-local caches in horizontally scaled services. If you have more than one instance, this cache is a consistency bug, not a performance improvement.

**2. Is `invalidate_preferences_cache` called in the same process that handles the write?**

Even in a single-instance deployment, I need to see where invalidation is called. If the write path and the read path can be in different workers (e.g., multiple Gunicorn workers, multiple threads), the invalidation guarantee breaks down.

**3. Are multiple threads or async workers sharing this process?**

A plain `dict` in CPython has incidental thread safety for individual operations due to the GIL, but compound check-then-act patterns (`if user_id in cache / cache[user_id] = result`) are not atomic across a context switch. Under concurrent load, two threads can both miss the cache, both query the database, and both write back — the lesser of the two problems here, but still worth noting.

## Implementation Issues (Independent of Deployment)

### No TTL / No Expiry

```python
_preferences_cache[user_id] = result
```

This cache never expires. A user who updates their preferences will see stale data until the process restarts — unless `invalidate_preferences_cache` is called explicitly and reliably on every write path. That is a fragile contract. One missed call to invalidation (a direct DB update, a future code path, a bulk migration) and the cache silently serves wrong data with no indication anything is wrong.

A TTL is not optional for correctness here. It is the fallback that bounds how wrong you can be.

### Unbounded Memory Growth

```python
_preferences_cache = {}
```

There is no eviction policy and no size limit. In a long-running process serving a large user base, this dict grows without bound. This is a memory leak in slow motion. At minimum, use `functools.lru_cache` with a `maxsize`, or an explicit LRU structure. A plain dict is the wrong data structure for a cache.

### Cache Stores the Raw DB Result

```python
_preferences_cache[user_id] = result
```

If `result` is a mutable object (a list of rows, a dict), callers who mutate the returned value will silently corrupt the cache. The cache should store a copy, or the return value should be immutable. This is not hypothetical — it is a common source of subtle bugs that are very hard to reproduce.

```python
# Safer
_preferences_cache[user_id] = dict(result)  # or a deep copy, depending on structure
return _preferences_cache[user_id].copy()   # return a copy to the caller
```

### No Handling of DB Errors

```python
result = db.query(...)
_preferences_cache[user_id] = result
```

If `db.query` raises an exception, the cache is not populated — fine. But if it returns an empty result or `None` for a user that does not exist, that empty result gets cached. Subsequent calls return the cached empty result without re-querying. If the user is created later, they will see no preferences until the cache is invalidated. Is that the intended behavior? It is not documented and not tested.

## Test Coverage Issues

The tests confirm the mechanical behavior of the cache (hit count, invalidation) but do not test any of the failure modes that matter:

| Scenario | Tested |
|---|---|
| DB raises an exception | ✗ |
| DB returns empty / None | ✗ |
| Concurrent access | ✗ |
| Cache behavior after invalidation of a key that was never cached | ✗ |
| Caller mutates the returned dict | ✗ |

The existing tests also have a state isolation problem: `_preferences_cache` is module-level state. If tests run in sequence without resetting it, earlier tests can pollute later ones. There is no `setUp`/teardown clearing the cache between tests. This will produce intermittent failures as the test suite grows.

```python
# This should be in every test that touches the cache
def setUp(self):
    _preferences_cache.clear()
```

## Security Note

Preference data is user-scoped. The cache key is `user_id` passed in as a string. I do not see anything in this diff that validates or sanitizes `user_id` before using it as a cache key or a query parameter. The query uses a parameterized placeholder (`?`) which is correct for SQL injection prevention, but I want to confirm that `user_id` is validated as belonging to the authenticated caller before this function is invoked. That is likely handled upstream, but it should be confirmed — caching amplifies the impact of an authorization bypass.

## What I Need Before This Can Move Forward

1. **Deployment topology**: How many instances? How many workers per instance?
2. **Write path**: Show me where `invalidate_preferences_cache` is called. Is it called on every preference update, in every code path?
3. **Acceptable staleness**: What is the business requirement? Is serving 60-second-old preferences acceptable? That answer determines whether a TTL-based approach is viable.

If the answer to (1) is "more than one instance," the implementation needs to be reconsidered entirely — either move to a shared cache (Redis, Memcached) with TTL, or accept that this optimization is not safe for this service's deployment model.

The performance problem this PR is solving is real. The solution as written is not ready.
