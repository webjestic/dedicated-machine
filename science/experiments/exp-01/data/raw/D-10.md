Decision: Request Changes

Summary: The caching approach is functionally correct for a single-process environment, but the implementation has critical correctness, security, and operational problems that need to be resolved before merging. The cache is unbounded, never expires, and is shared across all users in process memory — making it unsuitable for production use without significant hardening.

---

## Detailed Findings

### 1. Correctness ❌

**Stale data / no TTL**

The cache has no expiration mechanism. Preferences updated by any path other than this service's own write path (a migration, admin tool, another service instance) will never be reflected.

```python
# Current: cached forever
_preferences_cache[user_id] = result

# Suggested: store with a timestamp and respect a TTL
import time

_CACHE_TTL_SECONDS = 300  # make this configurable

_preferences_cache: dict[str, tuple[dict, float]] = {}

def get_user_preferences(user_id: str) -> dict:
    entry = _preferences_cache.get(user_id)
    if entry is not None:
        value, cached_at = entry
        if time.monotonic() - cached_at < _CACHE_TTL_SECONDS:
            return value
    result = db.query(...)
    _preferences_cache[user_id] = (result, time.monotonic())
    return result
```

**Multi-instance deployments**

`invalidate_preferences_cache` only clears the cache in the current process. In any multi-worker or multi-instance deployment (gunicorn, k8s, etc.) other processes will continue serving stale data. This needs to be called out explicitly in the PR description, or the invalidation strategy needs to be moved to a shared layer (e.g., Redis).

**Mutable cache values**

If `db.query` returns a mutable `dict` and the caller modifies it, the cached reference is mutated too. Return a copy:

```python
_preferences_cache[user_id] = result
return dict(result)  # or copy.deepcopy(result) if nested
```

### 2. Test Coverage ❌

The tests verify the happy path only. Missing cases:

- Cache miss on a cold start (first call always hits DB)
- TTL expiry causes a re-query (once TTL is added)
- Invalidating a key that was never cached (should not raise)
- Concurrent access behaviour (see Performance below)
- `get_user_preferences` called with an unknown `user_id` — what does `db.query` return and is that value safe to cache?

Also, the tests share the module-level `_preferences_cache` dict across test runs. Without explicit cache clearing between tests, test ordering can cause false positives:

```python
def setup_function():
    # clear the cache before each test
    services.preferences._preferences_cache.clear()
```

---

### 3. Naming Conventions ✅

Names are clear and consistent. `invalidate_preferences_cache` follows the existing verb-noun pattern in the codebase. No issues here.

---

### 4. Error Handling ❌

There is no handling for a failed DB query. If `db.query` raises an exception, nothing is cached (correct), but the exception propagates uncaught. That is probably the right behaviour — but it should be a deliberate, documented decision rather than an accident. More importantly, consider whether a partial/empty result should ever be cached:

```python
result = db.query(...)
if result is None:
    # Don't cache a None result — it may indicate a transient failure
    return {}
_preferences_cache[user_id] = (result, time.monotonic())
return dict(result)
```

---

### 5. Performance ❌

**Unbounded memory growth**

Every distinct `user_id` ever seen accumulates in the dict for the lifetime of the process. For a service with many users this is a memory leak.

```python
# Minimum viable fix: use a bounded LRU cache
from functools import lru_cache

# Or use cachetools for TTL + LRU together:
from cachetools import TTLCache
_preferences_cache: TTLCache = TTLCache(maxsize=1024, ttl=300)
```

**Thread safety**

Python's GIL makes individual dict operations atomic, but the check-then-set pattern in `get_user_preferences` is not atomic under a threaded server. Two threads can both find a cache miss and both issue DB queries. For a read-heavy cache this is acceptable (thundering herd is bounded to one per user), but it should be documented. If strict single-query semantics are required, a lock per user_id is needed.

---

### 6. Security ⚠️

No new injection surface is introduced (the `user_id` was already being passed to `db.query`). However, one concern worth flagging:

**Cache poisoning via untrusted `user_id`**

If `user_id` is derived from user-controlled input (e.g., a JWT claim or query parameter) and is not validated before this function is called, an attacker could cause the cache to grow unboundedly by cycling through synthetic user IDs. This is a denial-of-service vector. Confirm that `user_id` is validated upstream, and add the bounded cache regardless.

This does not by itself block the PR, but the bounded cache fix (Performance section) is the mitigation and must be included.

---

### 7. Documentation ⚠️

`invalidate_preferences_cache` has a docstring — good. `get_user_preferences` has none. Given that the caching behaviour is now non-obvious, it needs one:

```python
def get_user_preferences(user_id: str) -> dict:
    """
    Return preferences for the given user.

    Results are cached in-process for up to CACHE_TTL_SECONDS to reduce
    database load. Call invalidate_preferences_cache(user_id) after any
    write to ensure consistency within this process. Note: cache is not
    shared across service instances.
    """
```
