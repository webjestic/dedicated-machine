Decision: Request Changes

Summary: The caching approach is functionally correct and the tests verify the intended behavior, but the implementation has significant production-readiness problems that need to be addressed before merging. The module-level dictionary is a process-wide, unbounded cache with no TTL or eviction policy, meaning it will grow indefinitely and — more critically — serve stale data across requests with no guarantee of consistency, particularly in multi-process or multi-worker deployments where `invalidate_preferences_cache` called in one worker will not propagate to others. There are no security vulnerabilities per se, but the design risks are serious enough to warrant changes.

---

**Specific issues to address:**

1. **Stale data / no TTL.** Preferences that change in the database will be served from cache indefinitely until explicit invalidation occurs. At minimum, cache entries should carry an expiry timestamp.

   ```python
   import time
   
   _CACHE_TTL_SECONDS = 300  # make this configurable
   _preferences_cache: dict[str, tuple[dict, float]] = {}
   
   def get_user_preferences(user_id: str) -> dict:
       entry = _preferences_cache.get(user_id)
       if entry is not None:
           value, expires_at = entry
           if time.monotonic() < expires_at:
               return value
       result = db.query(
           "SELECT key, value FROM preferences WHERE user_id = ?",
           (user_id,)
       )
       _preferences_cache[user_id] = (result, time.monotonic() + _CACHE_TTL_SECONDS)
       return result
   ```

2. **Unbounded memory growth.** Every user who ever makes a request will have an entry retained forever. Use `functools.lru_cache`, a fixed-size structure, or a library like `cachetools` with a max size.

3. **Multi-worker / multi-process inconsistency.** In a typical WSGI/ASGI deployment (gunicorn, uvicorn with multiple workers), each worker has its own `_preferences_cache`. Invalidation in one worker is invisible to others. If cross-process consistency matters, this cache should live in a shared store (e.g., Redis) rather than process memory. This should at least be documented as a known limitation.

4. **Thread safety.** Concurrent requests in a threaded server can race on reads and writes to the dict. Python's GIL provides some protection for simple dict operations, but compound check-then-set logic is not atomic. Consider a `threading.Lock` or switch to `cachetools.TTLCache` which can be wrapped with `RLock`.

5. **Test isolation.** The tests share module-level state. If test ordering changes, a populated cache from a prior test can cause false passes. Each test should clear the cache in setup/teardown:

   ```python
   def setup_function():
       _preferences_cache.clear()
   ```

6. **Minor — type annotation and docstring missing on `get_user_preferences`** now that it has non-trivial logic.
