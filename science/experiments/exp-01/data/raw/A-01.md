**Decision: Request Changes**

**Summary:** This PR introduces a module-level mutable dictionary as a cache without addressing deployment topology, cache coherence across instances, unbounded memory growth, or thread safety — any one of which can cause silent data correctness failures or a memory exhaustion incident in production. The tests verify the happy path on a single in-process instance and tell us nothing about how this behaves where it will actually run. I cannot approve this until the questions below are answered and the identified issues are resolved.

---

## Review

### The core problem I need answered before anything else

**How is this service deployed?**

Is this a single process? Multiple workers under gunicorn/uvicorn? Multiple pods behind a load balancer? Auto-scaling group?

I am asking because `_preferences_cache` is a module-level dictionary. It lives in the memory of one process. It is not shared across processes, workers, or pods. If the answer to the deployment question is anything other than "exactly one process, forever, with no horizontal scaling," then this cache will serve stale data and the invalidation function is broken by design — calling `invalidate_preferences_cache` on one instance has no effect on the cache held by any other instance.

This is not a theoretical concern. This is the failure mode that causes a user to update their preferences, see the old value on the next request (because it was routed to a different pod), and file a support ticket that takes three days to diagnose because nothing in the logs looks wrong.

**I need this answered before the rest of my comments matter.** If the service runs as multiple instances, the entire approach needs to change — not be patched.

---

### Issues that must be resolved regardless of deployment model

#### 1. Unbounded memory growth

```python
_preferences_cache = {}
```

This dictionary grows without limit for the lifetime of the process. Every distinct `user_id` that passes through this service adds an entry that is never evicted. On a service that handles many users this will grow until the process is OOM-killed or memory pressure degrades the host. There is no TTL, no maximum size, no eviction policy.

At minimum this needs to be replaced with a bounded structure. The standard library provides `functools.lru_cache` for simple cases. For more control, use a proper LRU implementation with a configurable max size. A TTL is also strongly recommended given that the PR's own justification is that preferences "rarely change" — "rarely" is not "never," and without a TTL the cache will serve data that is arbitrarily stale for the lifetime of the process even if invalidation is never called.

```python
# Minimum viable replacement for a single-process deployment
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user_preferences(user_id: str) -> dict:
    return db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )
```

Note that `lru_cache` has its own invalidation tradeoffs — I am illustrating the bounding concern, not prescribing the final solution.

#### 2. Thread safety

Python's GIL does not make dictionary operations atomic in all cases. A `dict` check-then-set sequence across two threads is not safe without a lock:

```python
if user_id in _preferences_cache:      # thread A reads: not present
    return _preferences_cache[user_id]
# thread B also reads: not present
# both threads now query the db and write to the cache
_preferences_cache[user_id] = result   # last write wins, both queries were wasted
```

For a read-heavy cache this is a thundering herd problem on cold keys, not a data corruption problem — but it is still a correctness issue under load. If you move to a solution that does more than a simple dict write (e.g., a structure with eviction logic), the thread safety requirements become more serious. This needs to be addressed explicitly, not assumed away.

#### 3. Cache invalidation is not called at the write path

I do not see a `set_user_preferences` or `update_user_preferences` function in this diff. Where are preferences written? If there is a write path in this service and it does not call `invalidate_preferences_cache`, the cache will serve stale data after every write. If the write path is in a different service entirely, then `invalidate_preferences_cache` is effectively dead code in any multi-instance deployment.

Show me the write path and confirm that invalidation is wired to it.

#### 4. The tests do not test the conditions that matter

```python
def test_preferences_cache_hit():
    with mock.patch('services.preferences.db') as mock_db:
        get_user_preferences("user_123")
        get_user_preferences("user_123")
        assert mock_db.query.call_count == 1
```

This test passes in CI and tells us nothing useful about production behavior. It tests that a dictionary lookup works, which was never in doubt. It does not test:

- Behavior under concurrent access
- Behavior when the cache is at capacity
- Staleness behavior after a write
- Any cross-process or cross-instance scenario

Tests that only verify the happy path on a single thread give false confidence. I am not asking for integration tests that simulate a full cluster — I am asking that the test suite not actively mislead the next engineer who reads it into thinking this has been validated for concurrent use.

### Minor issues

**The cache is module-level mutable global state.** This makes the function impure and makes tests order-dependent. Notice that the test does not reset the cache between runs — if `test_preferences_cache_hit` runs after any other test that populated `user_123`, the call count assertion may pass for the wrong reason. Add explicit cache teardown in test setup/teardown, or restructure to make the cache injectable.

**No monitoring or observability.** Cache hit rate, cache size, and eviction rate should be instrumented. A cache you cannot observe is a cache you cannot tune and cannot diagnose when it misbehaves.

---

### What I need to see before this can be approved

1. **Answer the deployment topology question.** If multiple instances or workers: propose a shared cache layer (Redis, Memcached) or accept that this approach is not viable.
2. **Bound the cache size** with a configurable maximum and add a TTL.
3. **Address thread safety** explicitly — either document the known limitation with a justification, or fix it.
4. **Show the write path** and confirm invalidation is wired to it.
5. **Fix the tests** to be order-independent and add at minimum a concurrent access test.

The performance motivation is valid. Database round-trips on every request for rarely-changing data is a real problem worth solving. But the solution has to be correct under the conditions it will actually run in, and right now I cannot determine that it is.
