Decision: Request Changes

Summary: The caching approach is directionally reasonable but the implementation has several significant problems that must be addressed before merging. The most critical issues are a shared mutable cache that will cause data leakage between users in a multi-worker environment, unbounded memory growth, and no cache invalidation on writes. The tests also have an isolation bug that could produce false positives in CI.

---

## Detailed Findings

### 1. Correctness ❌

**Cache is never invalidated on writes.** If preferences are updated elsewhere in the service, the cache will serve stale data indefinitely. `invalidate_preferences_cache` is provided but there is no evidence it is called from any write path (e.g., `set_user_preferences`, `delete_user_preferences`). This is the most likely source of silent correctness bugs in production.

```python
# Any write path must call this — where is that wired up?
def update_user_preferences(user_id: str, prefs: dict) -> None:
    db.execute(...)
    invalidate_preferences_cache(user_id)  # must be here
```

**No TTL.** Preferences that change in the database will never be reflected until the process restarts or an explicit invalidation occurs. Even if writes go through this service, a restart-free deployment or a direct DB change will leave the cache permanently stale.

---

### 2. Test Coverage ❌

**Cache state leaks between tests.** The module-level `_preferences_cache` dict is never cleared between test runs. If `test_preferences_cache_hit` runs before `test_preferences_cache_invalidation`, the second test starts with `user_123` already cached, so the first `get_user_preferences` call is a cache hit and `call_count` will be 1 instead of 2 — the test passes for the wrong reason. Add a fixture:

```python
@pytest.fixture(autouse=True)
def clear_cache():
    _preferences_cache.clear()
    yield
    _preferences_cache.clear()
```

**Missing test cases:**
- Cache miss (first call always hits the DB)
- Concurrent access behavior
- Invalidation of a key that does not exist (should not raise)
- Behavior when `db.query` raises an exception (should not cache the failure)

---

### 3. Naming Conventions ✅

Names are clear and consistent with the existing module style. `invalidate_preferences_cache` is appropriately descriptive.

---

### 4. Error Handling ❌

**Failed queries must not be cached.** If `db.query` raises an exception, the current code never reaches the cache-write line, so this is accidentally safe — but it is fragile. If the control flow is ever refactored, a failed result could be cached. Make the intent explicit:

```python
result = db.query(...)
_preferences_cache[user_id] = result  # only reached on success
return result
```

Consider wrapping with a try/except and logging the failure rather than letting the exception propagate raw to the caller, depending on the service's error-handling contract.

**No input validation.** A `None` or empty `user_id` would be cached under a falsy key and could mask bugs in callers.

```python
if not user_id:
    raise ValueError(f"user_id must be a non-empty string, got {user_id!r}")
```

---

### 5. Performance ⚠️

**Unbounded cache growth.** A module-level dict with no eviction policy will grow indefinitely for the lifetime of the process. In a service with many users this is a memory leak. Use `functools.lru_cache` with a `maxsize`, or a simple bounded structure:

```python
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user_preferences(user_id: str) -> dict:
    ...
```

Note that `lru_cache` returns the same dict object on cache hits. If callers mutate the returned dict, they corrupt the cache. Either return a copy or document that the return value is read-only.

**Multi-process / multi-worker deployments.** A module-level dict is per-process. In a typical WSGI/ASGI deployment with multiple workers, each worker has its own cache and its own invalidation state. An invalidation in worker A does not affect worker B. If this is a concern, a shared cache (Redis, Memcached) is the correct solution. At minimum, document this limitation explicitly.

---

### 6. Security ❌

**Cross-user data exposure risk.** Because the cache is a plain module-level dict keyed only by `user_id`, any code path that can influence `user_id` before the cache lookup could cause one user's preferences to be served to another. This is not a vulnerability in the current code, but the design has no defense-in-depth against it. Ensure all callers validate that `user_id` comes from a trusted, authenticated source before passing it here — and add a comment to that effect.

**Mutable cache values.** `db.query` returns a dict that is stored and returned by reference. A caller who mutates the returned dict silently mutates the cached entry:

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "hacked"  # now cached with the wrong value
```

Return a shallow copy to prevent this:

```python
_preferences_cache[user_id] = result
return dict(result)  # or result.copy()
```

---

### 7. Documentation ⚠️

`invalidate_preferences_cache` has a docstring — good. `get_user_preferences` has none. Given that the caching behavior is now non-obvious (especially the stale-data and multi-worker caveats), it needs one:

```python
def get_user_preferences(user_id: str) -> dict:
    """
    Return preferences for the given user, using an in-process cache.

    The cache is per-process and has no TTL. Callers must invoke
    invalidate_preferences_cache() after any write to ensure consistency.
    In multi-worker deployments, invalidation is not propagated across workers.
    The returned dict must not be mutated by the caller.
    """
```
