Decision: Request Changes

Summary: The caching approach is functionally correct and the tests validate the core hit/miss behavior, but the implementation has several serious problems that must be addressed before merging. The unbounded global dictionary will grow indefinitely and leak memory, the cache is process-local which will cause stale or inconsistent data in any multi-process or multi-worker deployment, and mutable cache values can be corrupted by callers. There are also missing tests for important edge cases.

---

## Detailed Findings

### 1. Correctness

**🔴 Multi-process / multi-worker inconsistency**

The cache is a module-level dictionary, meaning each worker process has its own independent copy. In a typical deployment (gunicorn, uvicorn with multiple workers, etc.), a write on worker A will not invalidate the cache on worker B. Callers will silently receive stale data.

```python
# Worker 1: user updates preferences → invalidate_preferences_cache called
# Worker 2: still serving the old cached value — never notified
```

If the service runs as a single process today, this is a latent bug waiting to surface at scale. The PR description should at minimum document this constraint explicitly; ideally the design should use a shared cache (Redis, memcached) or accept that this is only safe in single-worker deployments.

**🟡 Cache is never populated with a copy — mutable aliasing**

`db.query` presumably returns a dict or list. The cache stores the same object reference that is returned to the caller. If any caller mutates the returned value, the cached entry is silently corrupted for all subsequent callers.

```python
# Fix: store a defensive copy
_preferences_cache[user_id] = dict(result)  # or copy.deepcopy(result)
```

---

### 2. Test Coverage

**🔴 No test for cache isolation between users**

The tests only exercise `user_123`. There is no test confirming that invalidating one user's cache does not affect another user's cached entry.

**🔴 No test for cold-start / empty result**

There is no test covering what happens when `db.query` returns `None`, an empty dict, or raises an exception. If `db.query` returns `None`, `None` gets cached and subsequent calls return `None` without ever retrying the database.

```python
# This will cache None and never recover:
_preferences_cache["user_123"] = None
```

**🟡 No test for invalidating a user who was never cached**

`invalidate_preferences_cache` uses `.pop(key, None)` which is safe, but there is no test confirming this does not raise.

**🟡 Tests do not reset global state between runs**

Because `_preferences_cache` is module-level, test execution order can affect results. Tests should explicitly reset the cache or use a fixture to patch it.

```python
@pytest.fixture(autouse=True)
def clear_cache():
    _preferences_cache.clear()
    yield
    _preferences_cache.clear()
```

---

### 3. Performance

**🔴 Unbounded cache growth — memory leak**

There is no eviction policy, no maximum size, and no TTL. In a long-running service with many users, this dictionary will grow without bound and never release memory. At minimum, add a size cap or TTL. Consider using `functools.lru_cache`, `cachetools.TTLCache`, or similar.

```python
from cachetools import TTLCache
_preferences_cache = TTLCache(maxsize=1024, ttl=300)
```

**🟡 No TTL means stale data can persist indefinitely**

Even in a single-worker deployment, preferences updated via a path that does not call `invalidate_preferences_cache` (e.g., a direct DB write, a different service, a migration) will never be reflected. A TTL provides a safety net.

---

### 4. Security

**🟡 No input validation on `user_id`**

`user_id` is used directly as a cache key without validation. This is low severity here since it is not used in a query at this layer, but it is worth noting as a general hygiene issue, especially if `user_id` originates from an untrusted source upstream.

**🟡 Cache poisoning surface**

If `invalidate_preferences_cache` is exposed via an API endpoint (not shown here, but worth confirming), it must be access-controlled. An unauthenticated caller forcing cache invalidation is a denial-of-service vector.

---

### 5. Error Handling

**🔴 Database exceptions bypass the cache but are not handled**

If `db.query` raises an exception, the cache is not populated (correct), but the exception propagates uncaught. This is probably acceptable, but it should be a deliberate decision documented in the code, not an accident.

**🟡 No handling for `user_id` being `None` or empty string**

```python
get_user_preferences(None)   # caches under None key
get_user_preferences("")     # caches under "" key
```

These will cache successfully and may mask bugs in callers.

---

### 6. Naming & Documentation

**🟢 Naming is clear and consistent** — `get_user_preferences`, `invalidate_preferences_cache`, and `_preferences_cache` follow a logical convention. The leading underscore correctly signals module-private intent.

**🟡 Missing module-level comment explaining cache scope and limitations**

Given the significant operational constraints (single-process only, no TTL), a comment is warranted.

```python
# Module-level cache. NOTE: This is process-local and not shared across
# workers. Only safe in single-process deployments. Entries are never
# evicted unless explicitly invalidated via invalidate_preferences_cache().
_preferences_cache = {}
```

## Summary of Required Changes

| 🔴 Must fix | Caching `None` / error results — guard before storing |
| 🔴 Must fix | Mutable aliasing — store a copy, not the original reference |
| 🔴 Must fix | Tests do not reset global cache state between runs |
