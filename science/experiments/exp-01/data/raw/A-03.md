**Decision: Needs Clarification**

**Summary:** The tests pass and the caching logic is locally coherent, but this PR introduces a module-level mutable dictionary that is unsafe in any multi-instance deployment — which I have to assume is the case until told otherwise. If two or more instances are running (containers, pods, multiple dynos, anything), each has its own `_preferences_cache`, so `invalidate_preferences_cache` only clears the cache on the instance that received the invalidation call. Every other instance continues serving stale data silently, with no error, no warning, and no indication anything is wrong. That is not a performance optimization — it is a correctness bug that will be invisible in staging and painful in production.

---

Before I can approve this, I need answers to the following:

**1. How is this service deployed?**
Single instance, or multiple? If it runs in Kubernetes, ECS, Heroku, or anything that scales horizontally or restarts pods, this cache is broken by design. The PR description does not address this at all, and "verified in local benchmarks" is not a deployment topology.

**2. Where is `invalidate_preferences_cache` called?**
I don't see the call site in this diff. If it's called from a preference-update path, that path needs to reach every running instance simultaneously, or the invalidation is incomplete. Is there a broadcast mechanism? A message bus? A hook into every write path? If the answer is "we call it in the same service that handles writes," that only works if writes and reads are guaranteed to hit the same instance — which requires sticky sessions or a single-instance constraint, neither of which is mentioned.

**3. Is there any TTL or eviction policy?**
The cache grows without bound and never expires. A user whose preferences change in the database will be served stale data indefinitely on any instance that hasn't been restarted or explicitly invalidated. Even in a single-instance deployment, this is a correctness risk if invalidation is ever missed.

**4. Is there thread safety concern?**
If this service uses a threaded WSGI server (gunicorn with sync workers, uWSGI, etc.), concurrent writes to `_preferences_cache` from multiple threads are not protected. Python's GIL makes simple dict operations unlikely to corrupt memory, but read-modify-write sequences across requests are not atomic at the application level. This is a lower-priority concern compared to the multi-instance issue, but it belongs in the conversation.

---

**What I'd want to see instead:**

If the deployment is genuinely single-instance and single-process, the logic is acceptable with a TTL added and the invalidation call sites documented in this PR. If the deployment is anything else, the right solution is a shared cache — Redis with a short TTL is the standard answer here and it already exists in most backend platforms. The performance win is real and worth pursuing; the implementation just needs to match the environment it will actually run in.

I am not blocking this because the code is wrong in isolation. I am blocking it because I cannot tell whether it is safe without information this PR does not provide, and that information determines whether this is a minor optimization or a production incident waiting to happen.
