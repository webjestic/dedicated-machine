**Decision: Needs Clarification**

**Summary:** The tests pass and the caching logic is locally coherent, but this PR introduces a module-level mutable dictionary that is fundamentally unsafe in any multi-instance deployment — which I have to assume is the case until told otherwise. Before this can be approved, I need answers to the deployment topology questions below, and depending on those answers, the implementation may need to change substantially.

---

## Review

### What the PR does well

The intent is sound. Caching preferences that rarely change is a legitimate optimization. The invalidation function exists, which is better than nothing. The tests correctly verify cache-hit deduplication and invalidation behavior in isolation.

---

### Blocking concerns

#### 1. Deployment topology is unspecified — this is the central question

This is a module-level dictionary:

```python
_preferences_cache = {}
```

That means the cache lives in the memory of a single process. It is not shared across processes or hosts.

**Questions I need answered before this can proceed:**

- How many instances of this service run in production? Is this ever deployed with more than one replica?
- Is there a load balancer in front of this service? If so, are sessions sticky (requests from a given user always route to the same instance)?
- What triggers a preference update? Is `invalidate_preferences_cache` called from within this same service, or from a separate service, a background job, an admin tool, or a database trigger?

If the answer to any of the above is "multiple instances" or "invalidation can happen outside this process," then this cache will serve stale data with no mechanism to detect or correct it. Instance A caches user_123's preferences. User updates their preferences. Instance B handles the update and calls `invalidate_preferences_cache` — which clears Instance B's cache only. Instance A continues serving the old data indefinitely.

This is not a theoretical edge case. This is the default failure mode of process-local caches behind a load balancer, and it is invisible in staging if staging runs a single instance.

#### 2. No TTL — stale data has no expiry path

Even in a single-instance deployment, there is no time-based expiry. If invalidation is missed for any reason — a bug, a deployment, a crash-and-restart that loses cache state mid-session — stale preferences persist for the lifetime of the process. A cache without a TTL is a cache that requires perfect invalidation coverage forever. That is a high bar to guarantee.

```python
# What this cache needs at minimum:
import time

_preferences_cache = {}  # {user_id: (data, timestamp)}
CACHE_TTL_SECONDS = 300

def get_user_preferences(user_id: str) -> dict:
    if user_id in _preferences_cache:
        data, cached_at = _preferences_cache[user_id]
        if time.monotonic() - cached_at < CACHE_TTL_SECONDS:
            return data
        del _preferences_cache[user_id]
    ...
```

Even this is only safe in a single-process context.

#### 3. Unbounded memory growth

The cache has no eviction policy and no size limit. In a long-running service with a large user base, this dictionary grows without bound. If this service handles tens of thousands of users, the cache will eventually hold tens of thousands of entries with no mechanism to release them. This is a slow memory leak.

At minimum, use `functools.lru_cache` with a `maxsize`, or a proper bounded cache structure.

#### 4. No thread safety

Python's GIL makes individual dictionary operations atomic, but the check-then-write pattern here is not:

```python
if user_id in _preferences_cache:      # check
    return _preferences_cache[user_id] # read — GIL does not protect this sequence
...
_preferences_cache[user_id] = result   # write
```

Under a threaded WSGI server (gunicorn with threads, uWSGI, etc.), two concurrent requests for the same uncached user_id can both pass the `if` check, both query the database, and both write to the cache. This is a race condition. In this specific case the writes are idempotent so it will not corrupt data, but it does defeat the purpose of the cache for cold entries and indicates the threading model was not considered.

#### 5. The invalidation function is only as good as its call coverage

`invalidate_preferences_cache` is defined, but the PR does not show where it is called when preferences are actually updated. If the update path is in a different module or service and does not call this function, the cache is never invalidated in practice. I want to see the preference update code and confirm the invalidation is wired in.

---

### What I need before this moves forward

1. **Deployment topology:** How many instances run in production? Sticky sessions or not?
2. **Invalidation ownership:** Where does preference mutation happen, and is `invalidate_preferences_cache` called there?
3. **If multi-instance:** Replace this with a shared cache (Redis, Memcached) or accept that local caching is not appropriate for this data.
4. **If single-instance:** Add a TTL and a bounded size at minimum.

I am not blocking this because caching is wrong. I am blocking it because the safety of this implementation depends entirely on deployment assumptions that are not stated in this PR, and I have been in production incidents caused by exactly this pattern. The tests passing is not evidence of correctness here — they do not and cannot test the multi-instance case.
