Summary: The tests pass and the caching logic is internally consistent, but this PR introduces a module-level mutable dictionary that is almost certainly broken under the actual deployment conditions — and the PR says nothing about what those conditions are. Before this can be approved, I need answers to the questions below.

---

## What I Need Answered Before This Can Merge

### 1. How is this service deployed?

This is the blocking question. A module-level `_preferences_cache = {}` lives in the memory of a single process. Its behavior depends entirely on your deployment topology:

| Deployment model | Cache behavior |
|---|---|
| Single process, single instance | Works as described |
| Multiple workers (gunicorn, uvicorn with `--workers N`) | Each worker has its own independent cache — invalidation in worker A does nothing to workers B, C, D |
| Multiple instances (horizontal scaling, Kubernetes) | Same problem, worse — now you have N×W independent caches |
| Async single-process (asyncio) | Works, but see thread safety note below |

If you are running more than one worker process — which is the normal production configuration for any service handling real traffic — then `invalidate_preferences_cache` is **silently broken**. A user updates their preferences, the write path calls `invalidate_preferences_cache`, that call evicts the entry from one worker's cache, and every other worker continues serving stale data indefinitely. There is no error, no log line, no signal that anything is wrong. This is exactly the class of bug that passes all tests and fails in production.

### 2. What calls `invalidate_preferences_cache`, and from where?

The invalidation function is defined but I don't see the call sites in this PR. I need to know:

- Is it called in the same service, on the same write path that modifies preferences?
- Is it called from a different service over the network?
- Is there any case where preferences are updated without invalidation being called?

If another service owns the write path, this cache has no reliable invalidation mechanism at all.

### 3. Is there a TTL?

There is no expiry on cache entries. A preference loaded at startup will be served forever — or until the process restarts, or until explicit invalidation is called. If invalidation is the only eviction mechanism and it can be missed (see above), users can be stuck with stale preferences across sessions, deployments, or preference updates that fail partway through.

## Secondary Issues (Fixable, But Fix Them)

### Thread safety

`dict` operations in CPython are not atomically safe under concurrent writes. In a threaded server (threaded gunicorn workers, for example), two threads could race on the same `user_id` write. This is unlikely to cause data corruption in practice due to the GIL, but it is not something to rely on. If threading is in scope, use a lock.

### Cache grows without bound

`_preferences_cache` has no size limit and no eviction policy. In a long-running process with many distinct users, this will grow until the process is restarted. For a preferences service this may be acceptable — but it should be a conscious decision, not an oversight. At minimum, document it.

### Tests don't cover the multi-worker failure mode

The tests validate the happy path correctly. They do not and cannot catch the cross-process invalidation failure because they run in a single process. This is not a criticism of the tests as written — it is a reason why passing tests are not sufficient evidence that this is safe to ship.

---

## What a Safe Version Looks Like

If the service runs multiple workers or instances, the right fix is one of:

1. **Move the cache to shared storage** — Redis with a short TTL is the standard answer. Invalidation becomes a `DEL` call that all workers see.
2. **TTL-only caching, no explicit invalidation** — Accept bounded staleness (e.g., 60 seconds), remove the invalidation function entirely, and document the consistency contract. Simple and honest about its tradeoffs.
3. **Keep it in-process, but only if single-worker is a documented and enforced constraint** — Add a deployment guard, document the assumption, and make sure it cannot silently be violated when someone bumps the worker count.

Option 2 is probably the right starting point given the stated goal ("preferences that rarely change"). It is simpler than Redis, it removes the broken invalidation path, and it is honest about what it guarantees.

I am not approving this until I know the deployment model. If the answer is "single process, single worker, and that is enforced," I can work with that — but I need to see it stated explicitly, not assumed.
