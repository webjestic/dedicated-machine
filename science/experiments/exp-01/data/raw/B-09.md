**Decision: Request Changes / Needs Clarification**

**Summary:** This PR introduces a module-level mutable cache without addressing deployment topology, cache invalidation consistency across instances, or memory growth bounds — any one of which can cause silent data correctness failures or a memory exhaustion incident in production. The tests verify the happy path in isolation but do not cover the failure modes that matter. I cannot approve this without answers to the blocking questions below and resolution of the issues I've identified.

---

## Review

### 🔴 Blocking Issues

#### 1. Deployment topology is unspecified — this is the most important question

```python
_preferences_cache = {}
```

This is a process-local dictionary. If this service runs as more than one instance — behind a load balancer, in a container orchestration environment, as multiple gunicorn workers on the same host — each instance has its own independent copy of this cache.

The consequences:

- A user updates their preferences. The write goes to instance A, which calls `invalidate_preferences_cache`. Instances B, C, and D still hold stale data. The user's next request hits instance B and gets their old preferences. From their perspective, the update did not work.
- If sessions are not sticky (and relying on sticky sessions for correctness is itself a fragile assumption), this will happen on a significant fraction of requests.

**I need to know before this merges:**
- How many instances of this service run in production?
- How many worker processes per instance (gunicorn workers, uwsgi workers, etc.)?
- Is `invalidate_preferences_cache` called in the same process that handles the write, or is it called via an internal API?
- Are sticky sessions in use, and if so, is that load balancer configuration guaranteed and documented?

If the answer is "multiple instances, no sticky sessions," this implementation is incorrect as written. The fix is either a shared cache (Redis, Memcached) or accepting that the cache cannot be reliably invalidated and adding a TTL-based expiry instead.

#### 2. No TTL — stale data is guaranteed over time

Even in a single-instance deployment, preferences cached at process startup will never expire unless `invalidate_preferences_cache` is explicitly called. This means:

- Any write path that does not call `invalidate_preferences_cache` — including direct database operations, admin tooling, migrations, or a future code path someone adds without knowing about this cache — will result in stale reads indefinitely.
- The cache has no self-healing mechanism.

At minimum, entries need a TTL. A simple approach:

```python
import time

_CACHE_TTL_SECONDS = 300  # make this configurable

_preferences_cache: dict[str, tuple[dict, float]] = {}

def get_user_preferences(user_id: str) -> dict:
    if user_id in _preferences_cache:
        value, cached_at = _preferences_cache[user_id]
        if time.monotonic() - cached_at < _CACHE_TTL_SECONDS:
            return value
        del _preferences_cache[user_id]

    result = db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )
    _preferences_cache[user_id] = (result, time.monotonic())
    return result
```

This does not fix the multi-instance problem, but it bounds how wrong the cache can be.

#### 3. Unbounded memory growth

```python
_preferences_cache = {}
```

There is no eviction policy. In a long-running process serving a large user base, this dictionary will grow without bound. One entry per user, forever, until the process restarts or runs out of memory.

This needs either:
- A maximum size with an eviction policy (LRU is the standard choice — `functools.lru_cache` or `cachetools.LRUCache` are both available)
- Or a documented, justified bound on the number of users this service will ever serve, with a size limit enforced in code

```python
from cachetools import TTLCache
# maxsize should be tuned to your user base and memory budget
_preferences_cache = TTLCache(maxsize=10_000, ttl=300)
```

This also resolves issue #2 if you use `TTLCache`.

#### 4. No thread safety

Python's GIL makes individual dictionary operations (`__getitem__`, `__setitem__`, `pop`) effectively atomic for CPython, but the check-then-act pattern in `get_user_preferences` is not:

```python
if user_id in _preferences_cache:       # check
    return _preferences_cache[user_id]  # act — key could be gone here
```

Under concurrent requests with invalidation happening on another thread, this can raise a `KeyError`. This is unlikely to trigger in practice with CPython's GIL, but it is a latent bug that will surface if the runtime ever changes or if the cache implementation is swapped for one without GIL protection.

The fix is straightforward:

```python
cached = _preferences_cache.get(user_id)
if cached is not None:
    return cached
```

Note: this also introduces an ambiguity if `None` is a valid cache value. If `db.query` can return `None` on an empty result set, you need a sentinel:

```python
_MISSING = object()

cached = _preferences_cache.get(user_id, _MISSING)
if cached is not _MISSING:
    return cached
```

### 🟡 Significant Issues

#### 5. The cache is not isolated between tests

The tests share the module-level `_preferences_cache` dictionary. If tests run in the order `test_preferences_cache_hit` → `test_preferences_cache_invalidation`, the second test starts with `user_123` already in the cache from the first test, and `mock_db.query.call_count` will be wrong.

This passes today because `mock.patch` replaces `db` but does not reset `_preferences_cache`. You are relying on test ordering and the specific mock behavior to get the right answer. Add explicit cache teardown:

```python
def setup_function():
    _preferences_cache.clear()

# or per-test:
def test_preferences_cache_hit():
    _preferences_cache.clear()
    with mock.patch('services.preferences.db') as mock_db:
        ...
```

---

#### 6. No test for the multi-instance / cross-process invalidation gap

I understand you cannot easily test a multi-process scenario in a unit test, but there should be at least a documented test that verifies the *contract*: that a write followed by a read in a *different cache state* returns fresh data. The current tests only verify the single-process happy path.

---

#### 7. `db.query` error handling is absent

```python
result = db.query(
    "SELECT key, value FROM preferences WHERE user_id = ?",
    (user_id,)
)
_preferences_cache[user_id] = result
```

If `db.query` raises an exception, the exception propagates correctly — that part is fine. But if `db.query` returns an error sentinel, an empty result, or `None` for a user that does not exist, that value gets cached. A subsequent call for a non-existent user will return the cached empty/None result rather than hitting the database again. Depending on whether users can be created after their first lookup, this could cache a "user not found" state permanently.

What does `db.query` return for a user with no preferences? Document it, and decide explicitly whether that result should be cached.

---

### 🟢 What Is Done Well

- `invalidate_preferences_cache` exists and is documented. The intent to keep the cache consistent is there; the implementation just does not hold up at scale.
- The test structure is correct in principle — mock the database, verify call counts. The right instinct, wrong execution.
- The SQL query is parameterized. No injection risk introduced here.
