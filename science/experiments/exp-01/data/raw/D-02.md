Decision: Request Changes

Summary: The caching approach is functionally correct for a single-process environment and the tests validate the basic hit/miss behavior, but the implementation has several significant problems that must be addressed before merging. Most critically, the unbounded global cache will grow indefinitely and the cached data is mutable and shared by reference, creating a correctness hazard. In a multi-worker or multi-process deployment this cache will also silently serve stale data, which may be the dominant production configuration.

---

## Detailed Findings

### 1. Correctness

**🔴 Mutable return value is shared by reference**

```python
# Current — caller can mutate the cached dict
_preferences_cache[user_id] = result
return result

# If a caller does this, the cache is silently corrupted:
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"   # mutates the cached object
```

The cache should store and return a defensive copy:

```python
import copy

_preferences_cache[user_id] = copy.deepcopy(result)
return copy.deepcopy(_preferences_cache[user_id])
# or, if result is a shallow dict of scalars, dict(result) is sufficient
```

**🟡 Cache is not populated before first return**

This is a minor style point — the current code stores `result` in the cache and then returns `result`, which is fine, but it's worth being explicit that the stored and returned values are the same object (see the mutation issue above).

---

### 2. Test Coverage

**🟡 Tests do not reset global state between runs**

The module-level `_preferences_cache` persists across tests. If `test_preferences_cache_hit` runs before another test, `user_123` is already cached and the second test's first `get_user_preferences` call will never reach the mock DB.

```python
# Add to both tests, or use a shared fixture:
def test_preferences_cache_hit():
    _preferences_cache.clear()   # or mock/monkeypatch the dict
    with mock.patch('services.preferences.db') as mock_db:
        ...
```

A pytest fixture is cleaner:

```python
@pytest.fixture(autouse=True)
def clear_cache():
    _preferences_cache.clear()
    yield
    _preferences_cache.clear()
```

**🔴 No test for cache miss (user not found / empty result)**

What does `db.query` return when there are no rows? If it returns `None`, callers receiving a cached `None` will likely raise `TypeError` downstream. This path needs a test and explicit handling.

**🟡 No test for `invalidate_preferences_cache` on a key that was never cached**

The `pop(..., None)` default handles this safely, but it should be tested to document the contract.

---

### 3. Performance

**🔴 Unbounded cache growth**

```python
_preferences_cache = {}   # grows forever
```

In a long-running process serving many users this will consume unbounded memory. Use a size-limited structure:

```python
from functools import lru_cache
# or
from cachetools import TTLCache, LRUCache

_preferences_cache = TTLCache(maxsize=1024, ttl=300)  # evicts after 5 min
```

A TTL is especially important here because preferences *can* change — without expiry, a user who updates their preferences in another process or service will be served stale data indefinitely until the process restarts.

**🟡 No TTL / expiry mechanism**

Even if the cache is bounded by size, there is no time-based expiry. The PR summary says preferences "rarely change," but "rarely" is not "never." A TTL of a few minutes would bound the staleness window without meaningful overhead.

---

### 4. Security

**🟡 Cache key is the raw `user_id` string**

This is acceptable, but worth confirming that `user_id` is always a validated, canonical identifier before it reaches this function. If `user_id` can be caller-supplied and is not normalized (e.g., `"User_123"` vs `"user_123"`), two cache entries could exist for the same user, bypassing the cache and, more importantly, potentially leaking one user's preferences to a lookup under a different key variant.

**🟡 Multi-process / multi-worker staleness**

Most Python web servers (gunicorn, uWSGI) run multiple worker processes. Each process has its own `_preferences_cache`. A preference update in one worker calls `invalidate_preferences_cache`, but all other workers continue serving the old value. This is a correctness and potential security issue (e.g., a revoked permission that lives in a preference key). The PR description and any operational runbook should explicitly document this limitation, or the cache should be moved to a shared layer (Redis, Memcached).

---

### 5. Error Handling

**🔴 DB errors are not handled; a failed query could cache a bad result**

```python
result = db.query(...)          # raises on connection error?
_preferences_cache[user_id] = result   # only reached if no exception — OK
```

This is actually fine *if* `db.query` raises on failure, but if it returns `None` or an error sentinel on failure, that sentinel will be cached. Confirm the contract of `db.query` and add a guard:

```python
result = db.query(...)
if result is None:
    raise PreferencesNotFoundError(user_id)   # or return a default — document the choice
_preferences_cache[user_id] = result
```

---

### 6. Naming

**✅ Acceptable overall.** `_preferences_cache`, `get_user_preferences`, and `invalidate_preferences_cache` are clear and consistent. The leading underscore correctly signals that the cache dict is module-private.

---

### 7. Documentation

**🟡 No module-level docstring or comment explaining the cache's scope and limitations**

Given the non-obvious operational behavior (per-process, no TTL, unbounded), a short comment is warranted:

```python
# Module-level in-memory cache. Scoped to a single process — in multi-worker
# deployments, invalidation is local only. No TTL; entries live until
# invalidated or the process restarts. See invalidate_preferences_cache().
_preferences_cache = {}
```

## Summary of Required Changes

| Priority | Issue |

| 🔴 Must fix | Add cache size bound (LRU or similar) to prevent unbounded memory growth |
| 🔴 Must fix | Add TTL or document staleness behavior explicitly |
| 🔴 Must fix | Reset global cache state between tests |
| 🔴 Must fix | Test and handle `None` / empty result from `db.query` |
| 🟡 Should fix | Document multi-process limitation prominently |
| 🟡 Should fix | Validate/normalize `user_id` before use as cache key |
