**Decision: Needs Clarification**

**Summary:** This PR introduces a module-level mutable dictionary as a cache, which is process-local by definition. Before this can be approved, I need to understand the deployment topology — specifically, how many instances of this service run in production, whether they sit behind a load balancer without sticky sessions, and where preference mutations happen. If any of those answers point to a multi-instance deployment (which is the default assumption for any service described as part of a "larger backend platform"), this cache will silently serve stale data to users whose preferences were updated on a different instance, because `invalidate_preferences_cache` only clears the cache in the process that calls it. The tests pass precisely because they cannot see this failure mode — they test a single in-process execution path, which is exactly the scenario where this code works correctly and exactly not the scenario that matters.

---

### Detailed Notes

#### 1. The core problem: invalidation scope is process-local

```python
_preferences_cache = {}  # lives in one process, invisible to all others
```

Consider this sequence across two instances behind a round-robin load balancer:

```
Instance A                          Instance B
──────────────────────────────      ──────────────────────────────
GET /preferences/user_123           
  → cache miss, queries DB          
  → caches {theme: "dark"}          

                                    POST /preferences/user_123
                                      → writes {theme: "light"} to DB
                                      → calls invalidate_preferences_cache("user_123")
                                      → clears cache on Instance B only

GET /preferences/user_123           
  → cache hit on Instance A         
  → returns {theme: "dark"}  ← STALE
```

Instance A has no knowledge that Instance B invalidated the entry. There is no cross-process signaling. The user updated their preference and immediately sees the old value. This is not a theoretical edge case — it is the default behavior under any multi-instance deployment.

#### 2. The tests cannot catch this

The tests are correct for what they test. That is the problem. They validate single-process cache mechanics — hit, miss, invalidation — all within one Python interpreter. They provide no signal about the failure mode that will actually occur in production. A green test suite here is actively misleading.

```python
# This test proves the cache works in one process.
# It says nothing about whether invalidation reaches other processes.
def test_preferences_cache_invalidation():
    ...
    assert mock_db.query.call_count == 2  # passes, always, everywhere
```

#### 3. There is no TTL or eviction

Even in a single-instance deployment, the cache grows without bound and never expires. A preference updated directly in the database (migration, support tooling, admin panel) will never be reflected. This needs either a TTL or a clear documented contract that the only write path is through this service.

#### 4. No thread safety

`dict` operations in CPython have incidental thread safety due to the GIL, but that is an implementation detail, not a guarantee, and it does not hold under all WSGI/ASGI workers. If this service runs with multiple threads per process (Gunicorn with `--threads`, any async framework), concurrent writes to `_preferences_cache` are not safe without a lock.

#### 5. The `result` object is cached by reference

```python
_preferences_cache[user_id] = result
return result
```

If the caller mutates the returned dict, the cached value is mutated too. This should be `result.copy()` or the return type should be documented as immutable. Right now the cache is a shared mutable reference, not a snapshot.

---

### Questions that must be answered before this can proceed

1. **How many instances of this service run in production?** If the answer is more than one, the invalidation design is broken as written.
2. **Is there a load balancer in front of this service? Are sticky sessions configured?** Sticky sessions would reduce the blast radius but not eliminate it (instance restarts, failover).
3. **What are all the write paths for user preferences?** If anything other than this service can update the preferences table, invalidation via this function is insufficient regardless of instance count.
4. **What is the threading model of the WSGI/ASGI server in production?**

---

### What a safe version of this looks like

If the deployment is genuinely single-instance and single-threaded (worth confirming, not assuming):

```python
import time
from threading import Lock

_preferences_cache: dict[str, tuple[dict, float]] = {}
_cache_lock = Lock()
_CACHE_TTL_SECONDS = 300  # preferences are stale after 5 minutes at most

def get_user_preferences(user_id: str) -> dict:
    with _cache_lock:
        entry = _preferences_cache.get(user_id)
        if entry is not None:
            value, cached_at = entry
            if time.monotonic() - cached_at < _CACHE_TTL_SECONDS:
                return value.copy()  # return a copy, not the cached reference

    result = db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )
    with _cache_lock:
        _preferences_cache[user_id] = (result, time.monotonic())
    return result.copy()
```

If the deployment is multi-instance (the more likely scenario for a service described as part of a larger platform), the correct solution is a shared cache — Redis with an appropriate TTL — not a process-local dict. That is more work, but it is the work that actually solves the problem safely.

---

I am not blocking this because the approach is wrong in principle — caching preferences is a reasonable optimization. I am blocking it because the safety of this specific implementation depends entirely on deployment facts that are not stated in the PR, and the failure mode if those facts are unfavorable is silent stale data served to users. That is not a risk I can sign off on without answers.
