**Decision:** Request Changes / Needs Clarification

**Summary:** This PR introduces a module-level mutable cache with no TTL, no thread safety, no size bound, and no consideration of deployment topology — any of which can cause correctness failures or production incidents. Before this merges, I need answers to the deployment questions below, and several issues need to be resolved regardless of those answers.

---

## Blocking Issues

### 1. Deployment topology — cache correctness across instances

This is the question I need answered before anything else.

`_preferences_cache` is a Python module-level dictionary. It lives in the memory of a single process. If this service runs as more than one instance — behind a load balancer, in a Kubernetes deployment with multiple replicas, under gunicorn with multiple workers, under uwsgi, *anything* that is not a single process — this cache will silently serve stale data.

The scenario:

```
Instance A: user_123 loads preferences → cached in Instance A's dict
User updates preferences → write goes to DB, invalidate_preferences_cache("user_123") is called
                         → but which instance handles that request?
Instance B: next request for user_123 → still serving the old cached value
            (Instance B's cache was never populated, never invalidated)
```

This is not a theoretical concern. It is the exact class of bug that looks fine in staging (single instance, single worker) and breaks in production (multiple replicas).

**Questions I need answered:**

- How many instances of this service run in production?
- How many workers per instance (gunicorn/uwsgi worker count)?
- Is preference invalidation always handled by the same process that cached the value, or can writes and reads land on different instances?
- Is there a shared cache layer (Redis, Memcached) already in the platform that should be used here instead?

If the answer to any of these is "multiple instances" or "we don't know," this cache is not safe as implemented.

### 2. No TTL — unbounded staleness

There is no expiry on cached entries. A preference cached at process startup will be served forever, or until an explicit invalidation call is made. This assumes:

- Every write path calls `invalidate_preferences_cache`
- Every write path that could affect these preferences is in this codebase
- No direct DB writes happen from other services, migrations, or admin tooling

If any of those assumptions are wrong, users will see stale data with no recovery path short of a process restart.

**At minimum**, cached entries need a TTL. A simple approach:

```python
import time
from typing import NamedTuple

_CACHE_TTL_SECONDS = 300  # make this configurable

class _CacheEntry(NamedTuple):
    value: dict
    expires_at: float

_preferences_cache: dict[str, _CacheEntry] = {}

def get_user_preferences(user_id: str) -> dict:
    entry = _preferences_cache.get(user_id)
    if entry is not None and time.monotonic() < entry.expires_at:
        return entry.value

    result = db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )
    _preferences_cache[user_id] = _CacheEntry(
        value=result,
        expires_at=time.monotonic() + _CACHE_TTL_SECONDS,
    )
    return result
```

### 3. No thread safety

A plain `dict` in CPython has incidental thread safety for single operations due to the GIL, but compound check-then-act sequences are not atomic. More importantly, this code cannot assume CPython forever, and the pattern is wrong regardless:

```python
# Thread 1 and Thread 2 both reach this simultaneously for the same user_id
if user_id in _preferences_cache:   # both see False
    ...
# both proceed to query DB and write to cache — redundant at best
```

For a read-heavy cache this is mostly a performance concern rather than a correctness one, but invalidation under concurrent load is a real race:

```python
# Thread 1: invalidate_preferences_cache("user_123")
# Thread 2: get_user_preferences("user_123") — reads stale value between query and write
```

Use `threading.Lock` or switch to a purpose-built structure. If the platform already uses Redis, this entire problem goes away and you get cross-instance consistency for free.

---

### 4. No cache size bound — memory leak under load

`_preferences_cache` grows without bound. Every distinct `user_id` that ever makes a request adds an entry that is never evicted (absent explicit invalidation). On a service with many users this is an unbounded memory leak.

Use `functools.lru_cache` if the function signature allows it, or `cachetools.TTLCache` / `cachetools.LRUCache` which give you both size bounds and TTL in a thread-safe wrapper:

```python
from cachetools import TTLCache
from threading import Lock

_cache: TTLCache = TTLCache(maxsize=1024, ttl=300)
_cache_lock = Lock()
```

## Non-Blocking Issues (must be addressed before merge)

### 5. Cached value is a mutable reference

`_preferences_cache[user_id] = result` stores whatever `db.query` returns. If that is a dict or list, callers who mutate the returned value are mutating the cached copy:

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # silently corrupts the cache
```

Either document that callers must not mutate the return value (fragile, will be violated), or return a copy:

```python
return result.copy()  # shallow — use copy.deepcopy if values are nested
```

---

### 6. Test coverage gaps

The existing tests verify the happy path only. Missing coverage:

- **Concurrent access** — two threads requesting the same user_id simultaneously
- **Cache miss after TTL expiry** — once TTL is added, this must be tested
- **Invalidation of a key that was never cached** — currently passes (`dict.pop` with default), but should be an explicit test
- **Mutation isolation** — caller mutates returned dict, subsequent call returns uncorrupted data
- **Cache miss for unknown user** — what does `db.query` return for a user with no preferences? Does the cache store that? Should it?

Also: the tests do not reset `_preferences_cache` between runs. If test execution order changes, `test_preferences_cache_hit` can pass spuriously because a previous test populated the cache. Add explicit teardown:

```python
def setup_function():
    _preferences_cache.clear()
```

---

### 7. No documentation of cache scope or limitations

`invalidate_preferences_cache` has a docstring. `get_user_preferences` has none. At minimum, the function-level docstring should state:

- That results are cached
- The TTL (once added)
- That the cache is **process-local** (critical for anyone reading this later who is thinking about multi-instance deployments)

```python
def get_user_preferences(user_id: str) -> dict:
    """
    Return preferences for the given user.

    Results are cached in process-local memory for up to CACHE_TTL_SECONDS.
    This cache is NOT shared across instances or workers. Callers must not
    mutate the returned dict.
    """
```

---

### 8. `invalidate_preferences_cache` is not called at the write path

I do not see a `set_user_preferences` or equivalent in this diff. If it exists elsewhere, I need to confirm it calls `invalidate_preferences_cache`. If it does not exist yet, that is a missing piece — the cache has no invalidation path from writes, only from explicit external calls.
