Decision: Request Changes

Summary: The caching approach will improve performance in the happy path, but the implementation has several correctness, security, and reliability problems that need to be resolved before merging. The cache is unbounded, never expires, is not thread-safe, and stores mutable data without isolation — any one of these issues could cause subtle production bugs. The test suite also doesn't adequately cover the failure modes introduced by this change.

---

## Detailed Findings

### 1. Correctness ❌

**Stale data / no TTL.** The cache has no expiration mechanism. Preferences updated by another service, a background job, or a direct database write will never be reflected until the process restarts or `invalidate_preferences_cache` is explicitly called. This is a correctness bug disguised as a performance feature.

```python
# Preferences changed in DB by another process — this still returns stale data
get_user_preferences("user_123")  # returns cached, outdated dict
```

**Mutable cache values.** `db.query` likely returns a mutable object (list or dict). The caller can mutate the returned value and silently corrupt the cache entry for every subsequent caller.

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # mutates the cached object — next caller sees "dark" too
```

Fix: return a deep copy, or store an immutable representation.

```python
import copy

_preferences_cache[user_id] = copy.deepcopy(result)
return copy.deepcopy(result)
```

**`invalidate_preferences_cache` is not called on writes.** If there's a `set_user_preferences` or `delete_user_preferences` function elsewhere in the service, it almost certainly doesn't call the new invalidation function. The PR doesn't show those call sites being updated.

---

### 2. Test Coverage ❌

The two tests cover only the ideal path. Missing cases include:

- **Cache isolation between users** — does caching `user_123` affect `user_456`?
- **Mutation safety** — mutating the returned dict should not affect a subsequent cache hit.
- **Thread safety** — concurrent calls for the same uncached user should not result in a race condition (see below).
- **`db.query` raises an exception** — the cache should not be populated with a failed/partial result.
- **`invalidate_preferences_cache` on a key that was never cached** — this works correctly today (`dict.pop` with a default), but it should be an explicit regression test.
- **Cache growth** — no test verifies that the cache doesn't grow without bound.

---

### 3. Naming Conventions ✅

Names are clear and consistent with the existing module style. `_preferences_cache` is correctly prefixed with an underscore to signal module-private scope. No issues here.

---

### 4. Error Handling ❌

If `db.query` raises an exception, the current code is safe because the cache write never executes. However, there is no explicit test for this, and a future refactor could easily break that ordering. Make the intent explicit:

```python
def get_user_preferences(user_id: str) -> dict:
    if user_id in _preferences_cache:
        return copy.deepcopy(_preferences_cache[user_id])

    result = db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )
    # Only cache on success; db.query raises on failure
    _preferences_cache[user_id] = copy.deepcopy(result)
    return copy.deepcopy(result)
```

---

### 5. Performance ❌

**Unbounded memory growth.** `_preferences_cache` is a plain `dict` with no eviction policy and no size cap. In a long-running process serving many users, this will grow indefinitely. Use `functools.lru_cache` with a `maxsize`, or a purpose-built structure like `cachetools.TTLCache`.

```python
from cachetools import TTLCache
_preferences_cache = TTLCache(maxsize=1024, ttl=300)  # 5-minute TTL, max 1 K entries
```

**Thread safety.** Python's GIL does not make dict operations atomic at the application level. A TOCTOU (time-of-check/time-of-use) race exists between the `in` check and the `__setitem__` write. Under a multi-threaded WSGI server (gunicorn with threads, uWSGI, etc.) two threads can both miss the cache and both write, which is benign here but signals the code was not designed with concurrency in mind. If the cache is later replaced with something more complex, this will become a real bug. Add a `threading.Lock` now.

```python
import threading
_cache_lock = threading.Lock()

def get_user_preferences(user_id: str) -> dict:
    with _cache_lock:
        if user_id in _preferences_cache:
            return copy.deepcopy(_preferences_cache[user_id])

    result = db.query(...)

    with _cache_lock:
        _preferences_cache[user_id] = copy.deepcopy(result)

    return copy.deepcopy(result)
```

Note: `cachetools.TTLCache` is also not thread-safe by default; wrap it with `cachetools.cached` using a `Lock`.

---

### 6. Security ⚠️

**Cross-user data leakage risk.** Because the cached value is mutable and shared, a bug in calling code that mutates the returned dict could cause one user's preferences to bleed into another user's view. The deep-copy fix above mitigates this, but it's worth calling out explicitly given that this is a user-data service.

**No concern with the SQL query itself** — parameterized queries are already used correctly. No new injection surface introduced.

**Process-local cache in a multi-process deployment.** If the service runs under multiple worker processes (common with gunicorn), each process has its own `_preferences_cache`. An invalidation call routed to process A will not invalidate the cache in process B. This is a correctness issue but also a potential security concern if a user revokes a permission or changes a sensitive preference and expects it to take effect immediately. The PR description should explicitly acknowledge this limitation, or the implementation should use a shared cache (e.g., Redis).

---

### 7. Documentation ⚠️

`invalidate_preferences_cache` has a docstring — good. `get_user_preferences` has none, and now has non-trivial caching behavior that warrants a comment explaining:

- that results are cached per process,
- that the cache has no TTL (or whatever TTL is added),
- that callers must invoke `invalidate_preferences_cache` after writes.
