Decision: Request Changes

Summary: The caching approach is functionally correct for a single-process environment and the tests verify the intended behaviour, but the implementation has several significant problems that must be addressed before merging. Most critically, the module-level dict is an unbounded, process-local cache that will cause correctness failures in multi-process deployments and memory exhaustion under load. There are also security, error-handling, and observability gaps that make this unsuitable for a production service.

---

## Detailed Findings

### 1. Correctness

**[Critical] Cache is process-local — incorrect in multi-process/multi-instance deployments**

Most Python web servers (gunicorn, uWSGI) fork multiple worker processes. Each process has its own `_preferences_cache`. A write to preferences in one worker will not invalidate the cache in any other worker, meaning users will see stale data.

```python
# This call in worker A invalidates only worker A's cache.
# Workers B, C, D continue serving stale preferences.
invalidate_preferences_cache(user_id)
```

If the service runs as more than one instance (containers, horizontal scaling), the problem is even worse. The PR summary says preferences "rarely change" but does not eliminate the possibility of change, and silent staleness is a correctness bug, not a trade-off.

**Recommendation:** Use a shared cache (Redis, Memcached) that all workers and instances can read and invalidate, or document explicitly that this service is guaranteed to run as a single process with a single worker — and enforce that with a startup assertion or deployment guard.

---

**[Moderate] No TTL — cache entries live forever**

Even in a single-process deployment, there is no expiry. If preferences are updated via a path that does not call `invalidate_preferences_cache` (a direct DB write, a migration, another service), the cache will never reflect reality.

```python
# Suggested shape — even a simple TTL entry helps:
import time
_preferences_cache: dict[str, tuple[dict, float]] = {}
CACHE_TTL_SECONDS = 300

def get_user_preferences(user_id: str) -> dict:
    entry = _preferences_cache.get(user_id)
    if entry is not None:
        value, stored_at = entry
        if time.monotonic() - stored_at < CACHE_TTL_SECONDS:
            return value
    result = db.query(...)
    _preferences_cache[user_id] = (result, time.monotonic())
    return result
```

### 2. Security

**[Critical] Mutable cache values returned by reference**

`db.query` presumably returns a `dict`. That same object is stored in the cache and returned directly to callers. Any caller that mutates the returned dict will silently corrupt the cached value for all subsequent callers.

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "hacked"          # mutates the cached object
get_user_preferences("user_123")   # now returns the mutated value
```

This is a data-integrity issue and, depending on how preferences are used downstream, could be a security issue (e.g., a preference controlling access or feature flags).

**Fix:** Return a copy.

```python
return dict(_preferences_cache[user_id])   # shallow copy
# or, for nested structures:
import copy
return copy.deepcopy(_preferences_cache[user_id])
```

---

**[Moderate] No input validation on `user_id`**

`user_id` is used as a dict key without any validation. If callers can pass attacker-controlled values, an adversary could probe cache occupancy (timing side-channel) or, in a worst case, pass a value that causes unexpected behaviour in downstream code. At minimum, validate that `user_id` is a non-empty string of expected format before using it.

```python
import re
_USER_ID_RE = re.compile(r'^[a-zA-Z0-9_-]{1,64}$')

def _validate_user_id(user_id: str) -> None:
    if not _USER_ID_RE.match(user_id):
        raise ValueError(f"Invalid user_id format: {user_id!r}")
```

### 3. Performance

**[Critical] Unbounded cache — memory leak**

The cache grows without limit. In a long-running process serving many distinct users, this will exhaust memory.

**Fix:** Use `functools.lru_cache` (if TTL is not needed), or a proper bounded structure:

```python
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user_preferences(user_id: str) -> dict:
    ...
```

Note that `lru_cache` has its own caveats (returns the same object — see the mutability issue above), but at least it is bounded. For TTL support, consider `cachetools.TTLCache`.

---

**[Minor] No cache metrics or logging**

There is no way to observe cache hit rate, size, or eviction behaviour in production. This makes it impossible to validate the claimed performance improvement or detect problems.

```python
import logging
log = logging.getLogger(__name__)

# inside get_user_preferences:
if user_id in _preferences_cache:
    log.debug("preferences cache hit", extra={"user_id": user_id})
    return ...
log.debug("preferences cache miss", extra={"user_id": user_id})
```

### 4. Error Handling

**[Moderate] DB errors are not handled; a failed query is not cached, but the code path is fragile**

If `db.query` raises an exception, the cache is not populated (correct), but there is no explicit handling. More importantly, if `db.query` returns `None` or an empty result for a user that genuinely has no preferences, that `None`/empty result is cached and will be returned on subsequent calls — which is probably fine, but should be a deliberate decision with a comment.

```python
result = db.query(...)
if result is None:
    # User has no preferences; cache the empty state explicitly
    # to avoid repeated DB hits for non-existent users.
    result = {}
_preferences_cache[user_id] = result
```

### 5. Test Coverage

**[Moderate] Tests do not cover multi-process staleness, mutation, or error paths**

The two tests cover the happy path only. Missing cases:

| Scenario | Covered? |
|---|---|
| Cache hit returns correct data | ✗ (only checks call count) |
| Mutating returned dict does not corrupt cache | ✗ |
| `invalidate_preferences_cache` on unknown user | ✗ |
| `db.query` raises an exception | ✗ |
| Cache state is isolated between tests | ✗ |

The last point is particularly important: because `_preferences_cache` is module-level state, tests can bleed into each other. Each test should clear the cache in a `setUp`/`tearDown` or use a fixture.

```python
@pytest.fixture(autouse=True)
def clear_cache():
    _preferences_cache.clear()
    yield
    _preferences_cache.clear()
```

---

### 6. Naming and Documentation

**[Minor] `_preferences_cache` is fine; consider a comment explaining the scope limitation**

The name is clear. However, given the significant operational constraints of a process-local cache, a module-level docstring or comment is warranted:

```python
# WARNING: This cache is process-local. It is only correct when the service
# runs as a single process with a single worker. In multi-worker or
# multi-instance deployments, use a shared cache (e.g. Redis) instead.
_preferences_cache: dict[str, dict] = {}
```
