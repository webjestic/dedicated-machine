Decision: Request Changes

Summary: The caching approach will improve database query frequency, but the implementation has several significant problems that must be addressed before merging. The most critical issues are a memory leak from unbounded cache growth, cross-request cache pollution from mutable shared state, and the complete absence of cache expiry — all of which make this unsafe for a production web service. The test suite also has structural flaws that mean it isn't actually validating the behavior it claims to cover.

---

## Detailed Findings

### 🔴 Security / Correctness — Cross-Request Data Leakage

**Severity: Critical**

`_preferences_cache` is a module-level dict. In any multi-threaded or async server (Gunicorn workers sharing state, asyncio, etc.), this dict is shared across all requests. If `db.query` returns a mutable dict or list and a caller modifies the returned object, every subsequent cache hit for that user will return the corrupted data. Worse, if there is any bug elsewhere that writes to the wrong user's result, the cached value persists and could be served to other requests.

```python
# Current — caller receives a direct reference to the cached object
_preferences_cache[user_id] = result
return result

# Fix — return a shallow copy so callers cannot mutate cached state
import copy
_preferences_cache[user_id] = result
return copy.copy(result)  # or copy.deepcopy if values are nested
```

---

### 🔴 Performance / Reliability — Unbounded Memory Growth

**Severity: Critical**

The cache has no eviction policy, no size limit, and no TTL. In a service that handles many users, this will grow without bound for the lifetime of the process. A single deployment serving 100k users will hold all 100k preference dicts in memory indefinitely.

At minimum, add a maximum size and a TTL:

```python
from functools import lru_cache
from datetime import datetime, timedelta
import threading

_CACHE_TTL = timedelta(minutes=5)
_MAX_CACHE_SIZE = 10_000

# Option A — use functools.lru_cache with a TTL wrapper (simple)
# Option B — use a proper cache library such as cachetools

from cachetools import TTLCache
_preferences_cache = TTLCache(maxsize=_MAX_CACHE_SIZE, ttl=300)
_cache_lock = threading.Lock()
```

`cachetools.TTLCache` is a well-tested, thread-safe-compatible drop-in that handles both concerns. If adding a dependency is not acceptable, at minimum document the known limitation and file a follow-up ticket.

---

### 🔴 Correctness — No Thread Safety

**Severity: High**

A plain `dict` in CPython has some incidental thread safety due to the GIL for individual operations, but the check-then-set pattern across two statements is not atomic:

```python
if user_id in _preferences_cache:   # thread A reads: not present
    return _preferences_cache[user_id]
# thread B also reads: not present — both now query the DB
_preferences_cache[user_id] = result  # both write; one result is dropped
```

This is a race condition that causes redundant DB queries (defeating the purpose) and could cause issues if the cache is later replaced with a non-GIL-safe structure. Add a lock, or use `cachetools` which is designed for this.

```python
_cache_lock = threading.Lock()

def get_user_preferences(user_id: str) -> dict:
    with _cache_lock:
        if user_id in _preferences_cache:
            return copy.copy(_preferences_cache[user_id])
    
    result = db.query(...)  # do not hold the lock during I/O
    
    with _cache_lock:
        _preferences_cache[user_id] = result
    return copy.copy(result)
```

---

### 🟡 Correctness — Cache Is Never Invalidated on Write

**Severity: High**

`invalidate_preferences_cache` exists, but there is no evidence it is called when preferences are updated or deleted. If `update_user_preferences` or any equivalent write path exists in this service (or elsewhere in the platform), it must call invalidation. This PR should either:

1. Show the write path and confirm it calls `invalidate_preferences_cache`, or
2. Explicitly document that callers are responsible for invalidation and add a note to the write path.

Without this, the cache will silently serve stale data after writes.

---

### 🟡 Test Quality — Tests Are Not Isolated

**Severity: Medium**

Both tests operate against the same module-level `_preferences_cache`. If `test_preferences_cache_hit` runs before `test_preferences_cache_invalidation`, the cache already contains `"user_123"` from the first test, making `call_count` assertions unreliable. The tests need to reset cache state:

```python
import services.preferences as prefs

def setup_function():
    prefs._preferences_cache.clear()  # or use importlib.reload

def test_preferences_cache_hit():
    with mock.patch('services.preferences.db') as mock_db:
        mock_db.query.return_value = {"theme": "dark"}
        prefs.get_user_preferences("user_123")
        prefs.get_user_preferences("user_123")
        assert mock_db.query.call_count == 1
```

---

### 🟡 Test Coverage — Missing Cases

**Severity: Medium**

The following scenarios are not tested and should be:

| Scenario | Why It Matters |
|---|---|
| Two different users do not share cache entries | Guards against key collision bugs |
| DB query raises an exception | Should the cache store the error? It currently would not, but this should be explicit |
| Invalidating a user not in the cache | `pop` handles this, but it should be a test |
| Concurrent access (if threading is added) | Validates the lock logic |

---

### 🟢 Naming and Documentation — Acceptable

The function and variable names are clear and consistent with what appears to be the existing codebase style. The docstring on `invalidate_preferences_cache` is appropriate. No issues here.

---

### 🟢 Correctness — Cache Hit Logic Is Sound

The basic `if user_id in _preferences_cache` pattern is correct in intent. The `pop(user_id, None)` in invalidation is the right idiom to avoid a `KeyError`. These are fine.

## Summary of Required Changes

| Priority | Issue |
|---|---|
| 🔴 Must fix | Add TTL and size cap to prevent memory exhaustion |
| 🔴 Must fix | Return copies of cached values to prevent mutation |
| 🔴 Must fix | Add thread safety (lock or thread-safe cache structure) |
| 🔴 Must fix | Confirm write paths call invalidation, or document the contract |
| 🟡 Should fix | Isolate tests with cache teardown |
| 🟡 Should fix | Add missing test cases |

The performance motivation is valid and worth pursuing — the implementation just needs these issues resolved first. I'd suggest adopting `cachetools.TTLCache` as it addresses memory, TTL, and provides a clean interface, reducing the amount of custom infrastructure needed.
