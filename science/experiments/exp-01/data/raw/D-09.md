Decision: Request Changes

Summary: The caching approach is functionally correct and the tests validate the core hit/miss behavior, but the implementation has several significant problems that must be addressed before merging. Most critically, the unbounded global cache will grow indefinitely, the cached data is mutable and shared by reference, and the design is unsafe in any multi-process or multi-worker deployment. These issues collectively make this change unsuitable for a production web service.

---

## Detailed Findings

### 1. Correctness

**🔴 Mutable cache values are shared by reference**

```python
_preferences_cache[user_id] = result
return result
```

If `db.query()` returns a mutable object (e.g., a `dict` or `list`), callers can mutate the cached value in place, corrupting subsequent cache hits for the same user. The fix is to store and return a defensive copy:

```python
import copy

_preferences_cache[user_id] = copy.deepcopy(result)
return copy.deepcopy(_preferences_cache[user_id])
```

Or, if the result type allows it, prefer an immutable representation (e.g., a frozen dataclass or `types.MappingProxyType`).

**🟡 No handling of `db.query()` exceptions**

If the database call raises, nothing is cached — that part is fine — but the exception propagates silently with no log or context. Consider at minimum logging the failure before re-raising so operators can correlate cache misses with DB errors.

---

### 2. Test Coverage

**🟡 Tests do not reset global cache state between runs**

The global `_preferences_cache` dict persists across tests. If `test_preferences_cache_hit` runs before another test that expects a cold cache, it will produce a spurious hit. Each test should explicitly clear the cache:

```python
def setup_function():
    _preferences_cache.clear()
```

Or expose a `clear_preferences_cache()` helper and call it in `setUp`/`teardown`.

**🟡 No test for concurrent access, TTL expiry, or cache-after-update behavior**

The test suite only covers the happy path. Missing cases include:
- Verifying that updating preferences and then calling `get_user_preferences` returns fresh data (i.e., that the write path calls `invalidate_preferences_cache`)
- Behavior when `user_id` is `None` or an empty string
- Behavior when `db.query()` raises

---

### 3. Naming Conventions

**🟢 Acceptable overall.** `_preferences_cache`, `get_user_preferences`, and `invalidate_preferences_cache` are clear and consistent. The leading underscore correctly signals module-private intent.

---

### 4. Error Handling

**🔴 No TTL — stale data is served indefinitely**

This is the most operationally dangerous aspect of the design. Preferences that change via any path other than this module (direct DB writes, admin tooling, other services, other processes) will never be reflected. Even within this service, if `invalidate_preferences_cache` is not called on every write path, users will see stale preferences until the process restarts.

Required: either add a TTL or document explicitly (with a code comment and in the PR) that callers are contractually required to call `invalidate_preferences_cache` on every write, and verify that all write paths do so.

**🔴 Cache is process-local — unsafe in multi-worker deployments**

A typical Python web service runs under Gunicorn, uWSGI, or similar with multiple worker processes. Each process has its own `_preferences_cache`. An invalidation in worker A is invisible to workers B, C, and D. This means the cache provides correctness guarantees only in single-process deployments, which is rarely the production topology.

Options in increasing order of robustness:
1. Document the limitation clearly and restrict use to single-worker deployments
2. Use a shared cache (Redis, Memcached) instead of a module-level dict
3. Accept eventual consistency and rely on TTL to bound staleness

---

### 5. Performance

**🔴 Unbounded cache growth**

```python
_preferences_cache = {}
```

There is no eviction policy. In a service with many users, this dict will grow until the process runs out of memory or is restarted. Use `functools.lru_cache` with a `maxsize`, or a proper LRU structure:

```python
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user_preferences(user_id: str) -> dict:
    ...
```

Note: `lru_cache` does not support manual invalidation out of the box, so a custom LRU with TTL (e.g., `cachetools.TTLCache`) is likely the right tool here:

```python
from cachetools import TTLCache
from threading import Lock

_preferences_cache: TTLCache = TTLCache(maxsize=1024, ttl=300)
_cache_lock = Lock()
```

**🟡 No thread safety**

The standard CPython GIL provides some incidental protection for dict operations, but relying on it is fragile and non-portable. If the service uses threading (e.g., threaded Flask, Django with threads), concurrent writes to `_preferences_cache` should be protected by a `threading.Lock`.

---

### 6. Security

**🟡 No input validation on `user_id`**

`user_id` is used directly as a cache key. If `user_id` can be caller-controlled and is not validated upstream, a malicious or buggy caller could use a crafted key to retrieve another user's cached preferences (e.g., by guessing or enumerating keys). This is not introduced by this PR, but the cache makes the blast radius of such a bug larger. Confirm that `user_id` is always derived from a verified authentication token before reaching this function.

---

### 7. Documentation

**🟡 Missing module-level comment explaining the caching contract**

The `invalidate_preferences_cache` docstring is good, but there is no comment explaining:
- That the cache is process-local
- That callers must invalidate on write
- The absence of a TTL and its implications

A short block comment above `_preferences_cache` would suffice:

```python
# Process-local in-memory cache for user preferences.
# IMPORTANT: This cache is not shared across worker processes.
# Callers MUST invoke invalidate_preferences_cache() after any write
# to avoid serving stale data. No TTL is applied; entries persist
# until explicitly invalidated or the process restarts.
_preferences_cache = {}
```
