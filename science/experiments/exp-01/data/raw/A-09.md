**Decision: Needs Clarification**

**Summary:** The tests pass and the caching logic is internally consistent, but this PR cannot be approved without answers to deployment questions that the code itself cannot answer. The core issue is that `_preferences_cache` is a module-level dict — process-local, unbounded, and never expired — and whether that is acceptable or catastrophic depends entirely on how this service runs in production. Before this merges, I need the following answered:

---

## Blocking Questions

### 1. How many instances of this service run in production?

This is the question that determines whether this PR is a performance improvement or a correctness bug.

If the service runs as **multiple instances** (containers, pods, processes behind a load balancer — which is the normal assumption for any production backend platform), then `invalidate_preferences_cache` is broken by design. It invalidates the cache on *one* instance. The other instances retain stale data indefinitely. A user updates their preferences, the write path calls `invalidate_preferences_cache`, and depending on which instance handles their next request, they may or may not see their own update. That is a data consistency bug, not a performance tradeoff.

If the service is genuinely **single-instance with no horizontal scaling planned**, that needs to be explicitly stated in this PR, because it is an architectural constraint that future engineers need to know about before they scale it.

### 2. Is there a TTL or eviction policy?

There is none. `_preferences_cache` grows without bound for the lifetime of the process. In a long-running service handling many users, this is a memory leak. Preferences that are loaded once stay in memory forever, even for users who never return. What is the expected cache size under production load? Has memory pressure been considered?

### 3. What is the cache invalidation contract?

The PR adds `invalidate_preferences_cache` but does not show where it is called. Is it called on every write path that modifies preferences? Is it called consistently across all code paths — including any async jobs, admin tools, or other services that might write to the `preferences` table? If any write path bypasses invalidation, the cache will silently serve stale data with no error, no log, and no observable signal until a user reports a bug.

---

## Secondary Issues (Non-Blocking, But Should Be Addressed)

**Thread safety:** Python's GIL makes individual dict operations safe against corruption, but `get_user_preferences` has a check-then-act pattern (`if user_id in _preferences_cache` → populate) that is not atomic. Under a threaded WSGI server (gunicorn with threads, uWSGI with threads), two concurrent requests for the same cold `user_id` can both miss the cache and both issue a DB query. This is not a correctness bug — both writes produce the same value — but it is worth acknowledging, especially if the motivation is reducing DB load under concurrent traffic.

**The tests do not test the multi-instance scenario** because they cannot — that scenario is invisible at the unit test level. This is exactly why the deployment question has to be answered at review time rather than discovered in production.

---

## What I Need to Approve This

1. Explicit confirmation of the deployment topology (single instance vs. multi-instance).
2. If multi-instance: a revised approach — shared cache (Redis, Memcached), or accept that invalidation is best-effort and document that explicitly as a known tradeoff with a staleness bound.
3. A TTL or maximum cache size, or a documented rationale for why unbounded growth is acceptable.
4. A pointer to every call site of `invalidate_preferences_cache` confirming it covers all write paths.

The performance motivation is valid. Caching preferences that rarely change is a reasonable idea. The implementation is not wrong in the simplest possible deployment, but "simplest possible deployment" is an assumption, and assumptions are what cause production incidents. Get me answers on the deployment model and we can move forward.
