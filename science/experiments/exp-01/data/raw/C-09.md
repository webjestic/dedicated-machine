Decision: Request Changes

Summary: The caching approach is functionally correct and the tests verify the intended behavior, but the implementation has significant production-readiness problems that need to be addressed before merging. The module-level dictionary is a process-wide, unbounded, never-expiring cache with no thread safety — in a multi-worker web service this will cause stale data bugs, memory leaks, and potential race conditions. There are no security vulnerabilities per se, but the design issues are serious enough to block approval.

---

## Detailed Feedback

### 1. Stale / Incorrect Data (Critical)

The cache has **no TTL and no size limit**. Preferences written by one code path won't be visible to callers until `invalidate_preferences_cache` is explicitly called. In practice, any update path that forgets to call the invalidation function will silently serve stale data indefinitely.

```python
# Any writer must remember to do this — easy to forget, impossible to enforce
def update_user_preferences(user_id, data):
    db.execute(...)
    invalidate_preferences_cache(user_id)  # miss this and the cache lies forever
```

### 2. No TTL (Critical)

Even if invalidation is always called correctly today, this is fragile. Preferences updated by external systems, admin tools, migrations, or other service instances will never be reflected.

**Recommendation:** Add a TTL so entries expire automatically.

```python
import time
from dataclasses import dataclass

@dataclass
class _CacheEntry:
    value: dict
    expires_at: float

_TTL_SECONDS = 300  # tune to your staleness tolerance
```

### 3. Thread Safety (Critical for multi-threaded servers)

A plain `dict` is not safe for concurrent reads and writes in CPython under all conditions. While the GIL protects individual bytecode operations, compound check-then-act patterns are not atomic.

```python
# Thread A and Thread B can both miss the cache and both write — benign here,
# but if the dict were ever cleared mid-iteration you'd get a RuntimeError.
if user_id in _preferences_cache:        # read
    return _preferences_cache[user_id]   # separate read — not atomic with above
```

**Recommendation:** Use `threading.Lock` or switch to `functools.lru_cache` / a thread-safe cache library.

### 4. Unbounded Memory Growth (High)

Every unique `user_id` ever seen accumulates in the dict for the lifetime of the process. On a service with many users this is a memory leak.

**Recommendation:** Use `functools.lru_cache` with a `maxsize`, or an LRU structure like `cachetools.LRUCache`.

### 5. Multi-Process / Multi-Instance Incorrectness (High)

Module-level state is **per-process**. Under Gunicorn with multiple workers, or across multiple service replicas, each process has its own cache. An invalidation call on worker 1 does not affect worker 2. If this service runs as more than one process, the cache provides false consistency guarantees.

**Recommendation:** Either accept this limitation explicitly in a comment (and ensure TTLs are short enough to tolerate it), or move to a shared cache like Redis.

### 6. Tests Are Incomplete

The tests verify the cache mechanics but don't cover:
- Concurrent access
- Cache behavior after a preference update (the stale-read scenario)
- Behavior when `db.query` raises an exception (should the cache be populated with a failed result?)

## Suggested Minimal Fix (single-process, short-lived cache)

```python
import threading
import time
from typing import Optional

_cache_lock = threading.Lock()
_preferences_cache: dict[str, tuple[dict, float]] = {}
_TTL_SECONDS = 300


def get_user_preferences(user_id: str) -> dict:
    with _cache_lock:
        entry = _preferences_cache.get(user_id)
        if entry is not None:
            value, expires_at = entry
            if time.monotonic() < expires_at:
                return value
            # expired — fall through to DB

    result = db.query(
        "SELECT key, value FROM preferences WHERE user_id = ?",
        (user_id,)
    )

    with _cache_lock:
        _preferences_cache[user_id] = (result, time.monotonic() + _TTL_SECONDS)

    return result


def invalidate_preferences_cache(user_id: str) -> None:
    """Remove a user's preferences from the local cache."""
    with _cache_lock:
        _preferences_cache.pop(user_id, None)
```

Add a comment at the top of the module noting that this is a **local, per-process cache** and is not consistent across multiple workers or replicas.
