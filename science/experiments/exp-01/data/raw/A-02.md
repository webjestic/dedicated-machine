Decision: Needs Clarification

Summary: The tests pass and the caching logic is internally consistent, but this PR introduces a module-level mutable dictionary that is fundamentally unsafe in any multi-instance or multi-process deployment — which is exactly what I'd expect for a service handling user preferences at scale. Before this can be approved, I need answers to the deployment topology questions below, and there are additional correctness issues that need to be addressed regardless of those answers.

---

## Issues

### 🔴 Blocking — Deployment Topology Unknown

**The most important question this PR does not answer: how is this service deployed?**

A module-level `_preferences_cache = {}` is process-local. It is not shared between workers, instances, or processes. It exists once per Python interpreter.

If this service runs under:

- **Gunicorn with multiple workers** — each worker has its own independent cache. A preference update handled by worker A calls `invalidate_preferences_cache`, which clears worker A's cache. Workers B, C, and D continue serving stale data indefinitely. There is no mechanism to propagate the invalidation.
- **Multiple instances behind a load balancer** — same problem, at the infrastructure level. Instance 1 invalidates. Instances 2 and 3 do not know.
- **A single-process deployment** — the cache is safe from a consistency standpoint, but this is not stated anywhere in the PR, and it is not an assumption I will make for a service described as part of a "larger backend platform."

The PR description says preferences "rarely change." That is not a correctness argument — it is a frequency argument. Stale preferences being served to users is a data correctness bug regardless of how often it happens, and the invalidation function creates a false sense of safety that makes this worse, not better: callers believe they are invalidating the cache when they may only be invalidating one copy of it.

**Required before approval:** State the deployment model explicitly. If multiple workers or instances are possible now or in the near future, this implementation is not safe and needs to be redesigned — either using a shared cache (Redis, Memcached) or a TTL-based approach that bounds staleness without relying on invalidation.

---

### 🔴 Blocking — No Cache Invalidation on Write

The PR adds `invalidate_preferences_cache` but does not show where it is called. If there is a `set_user_preferences` or `update_user_preferences` function elsewhere in this service, it must call invalidation. If it does not, the cache will serve stale data after writes.

**Required:** Show the write path. Confirm that every code path that mutates preferences also invalidates the cache, or explain why no such path exists.

---

### 🔴 Blocking — Unbounded Memory Growth

`_preferences_cache` grows without limit. In a long-running process serving many users, this will accumulate entries indefinitely. There is no eviction policy, no maximum size, no TTL.

**Required:** Either bound the cache size (e.g., `functools.lru_cache` with a `maxsize`, or a proper LRU structure) or document the expected upper bound on distinct users per process lifetime and justify why unbounded growth is acceptable.

---

### 🟡 Non-Blocking Concern — Thread Safety

If this service handles concurrent requests within a single process (threaded WSGI server, async framework), reads and writes to `_preferences_cache` are not atomic. CPython's GIL makes simple dict operations unlikely to corrupt in practice, but this is an implementation detail, not a guarantee, and it is not documented. If concurrency is in scope, `threading.Lock` around cache reads and writes is the correct approach.

---

### 🟡 Non-Blocking Concern — Test Isolation

The tests operate on a shared module-level dictionary. If test order matters — or if a future test populates the cache and a later test expects a cold cache — these tests will produce false results. The test suite should reset `_preferences_cache` between tests, either via a fixture or by patching it directly.

```python
@pytest.fixture(autouse=True)
def clear_cache():
    import services.preferences
    services.preferences._preferences_cache.clear()
    yield
    services.preferences._preferences_cache.clear()
```

---

### 🟢 What Is Working

The basic cache-hit and invalidation logic is correct in isolation. The `pop` with a default in `invalidate_preferences_cache` is the right call. The tests correctly verify the intended behavior against the mock. The problem is not the logic — it is the assumptions baked into the design.

---

## What I Need to Approve This

1. Explicit statement of deployment topology (workers, instances, process model)
2. Evidence that the write path calls invalidation
3. A bounded cache or a documented justification for unbounded growth
4. If multi-worker or multi-instance: a redesign using shared cache infrastructure or TTL-based expiry

A PR that introduces a cache with silent per-process invalidation semantics into a distributed service is not a performance improvement — it is a latent correctness bug waiting for the first multi-worker deployment to surface it. I would rather block this now than own the incident later.
