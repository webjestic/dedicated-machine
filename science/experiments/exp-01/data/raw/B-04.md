**Decision:** Request Changes / Needs Clarification

**Summary:** This PR introduces a module-level mutable cache with no TTL, no thread safety, no size bound, and no consideration of deployment topology — any one of these is a production incident waiting to happen, and together they make this unsafe to merge. Before this goes further, I need answers about the runtime environment, and several structural issues need to be addressed regardless of those answers.

---

## Review

### 🔴 Blocking — Deployment Topology (Must Clarify Before Anything Else)

The most important question this PR does not answer: **how is this service deployed?**

`_preferences_cache` is a module-level dictionary. That means it lives in the memory of a single process. If this service runs as more than one instance — behind a load balancer, in a container orchestrator, as multiple gunicorn workers, *anything* — then:

- A preference update that calls `invalidate_preferences_cache` on instance A leaves the stale entry alive on instances B, C, and D.
- Users will read stale preferences with no indication that anything is wrong.
- The bug will be intermittent and load-dependent, which makes it extremely hard to reproduce and diagnose.

This is not a theoretical concern. This is the exact class of failure that causes production incidents that take hours to root-cause because everything looks fine in staging (single instance, single worker).

**I need to know:**
1. How many instances of this service run in production?
2. How many worker processes per instance (gunicorn workers, uwsgi processes, etc.)?
3. Is there a shared cache layer (Redis, Memcached) already in the platform that this should be using instead?
4. When preferences are updated, what code path is responsible for calling `invalidate_preferences_cache`? Is that call guaranteed to reach every process that might hold a cached copy?

If the answer to (1) and (2) is genuinely "always exactly one process," that needs to be documented explicitly — both in this PR and in the code — because the next person who scales this service will not know the cache is there.

---

### 🔴 Blocking — No TTL / Unbounded Staleness

Even in a single-process deployment, the cache has no expiry. A preference written to the database is served stale indefinitely unless something explicitly calls `invalidate_preferences_cache`. That means:

- If the invalidation call is missed anywhere — a direct database write, a bug in a calling service, a future code path that forgets — users see stale data forever, until the process restarts.
- There is no self-healing behavior.

A TTL (e.g., 60 seconds) is not a perfect solution, but it bounds the blast radius. Without it, a cache poisoning event is permanent within a process lifetime.

---

### 🔴 Blocking — No Thread Safety

```python
_preferences_cache = {}
```

Python's GIL protects individual bytecode operations, but it does not make compound read-modify-write sequences atomic. Under a threaded server (threaded Flask, Django with threading, any ASGI server), two concurrent requests for the same uncached `user_id` will both miss the cache, both query the database, and both write to the dict. This is a benign race in this specific case — the writes are idempotent — but:

- It means the "only one DB query" guarantee the tests assert is not actually guaranteed under concurrent load.
- More importantly, it establishes a pattern. The next person who extends this cache (adding metadata, reference counts, anything stateful) will inherit a structure that looks thread-safe but is not.

At minimum, use `threading.Lock` around cache reads and writes, or use a structure designed for concurrent access.

---

### 🔴 Blocking — No Bound on Cache Size

```python
_preferences_cache = {}
```

This grows without limit. A service that handles N distinct users will accumulate N entries and never release them. In a long-running process serving a large user base, this is a memory leak. It will not show up in local benchmarks.

Use `functools.lru_cache` with a `maxsize`, or `cachetools.TTLCache`, or any structure with an eviction policy.

---

### 🟡 Significant — Cache Stores a Mutable Reference

```python
_preferences_cache[user_id] = result
return result
```

If `db.query` returns a mutable object (a list, a dict), the caller receives a reference to the same object stored in the cache. A caller that mutates the returned value mutates the cached entry. Subsequent callers get the mutated version, not the database version.

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # silently corrupts the cache
```

Return a copy:

```python
_preferences_cache[user_id] = result
return copy.deepcopy(result)  # or copy.copy() if the structure is shallow
```

---

### 🟡 Significant — Test Isolation Is Broken

The tests share module-level state. `_preferences_cache` persists between tests. If `test_preferences_cache_hit` runs before `test_preferences_cache_invalidation`, the second test starts with `user_123` already in the cache, and the first `db.query` call never happens, so `call_count == 2` fails.

This is a latent ordering dependency. It may pass today because of test execution order, and silently break when a test runner parallelizes or reorders.

Each test must clear the cache in setup:

```python
def setup_function():
    _preferences_cache.clear()
```

Or the cache should be injectable (see naming note below) so tests can pass a fresh one.

---

### 🟡 Significant — No Test for Concurrent Access or Multi-Instance Behavior

The tests cover the happy path: single-threaded, single-process, sequential calls. There is no test for:

- Two concurrent requests for the same user (race on cache population)
- Behavior when `db.query` raises an exception (does a partial write land in the cache?)
- Invalidation called for a user not in the cache (this works — `pop` with default — but it is not tested)

---

### 🟡 Significant — Exception Behavior Is Undefined

```python
result = db.query(...)
_preferences_cache[user_id] = result
return result
```

If `db.query` raises, nothing is cached and the exception propagates — that part is fine. But what does `db.query` return on an empty result set? An empty dict? `None`? An empty list? If it returns `None` and the caller does not check, you get `TypeError` downstream. If it returns `None` and you cache it, subsequent calls return `None` without hitting the database, which may mask a transient database error as a permanent empty result.

The return type of `db.query` should be documented, and the cache-write path should handle `None` or empty results explicitly.

---

### 🟠 Minor — Naming

`_preferences_cache` is acceptable but the leading underscore signals "module-private," which is correct. No issue there.

`invalidate_preferences_cache` is clear. No issue.

If the cache is ever made injectable for testability, consider naming the parameter `cache` or `store` rather than `_cache` to avoid the private-signal confusion at call sites.

---

### 🟠 Minor — No Documentation on Invalidation Contract

The docstring on `invalidate_preferences_cache` says what it does, not when it must be called. The contract — "this must be called whenever preferences are written" — is the critical invariant, and it is not documented anywhere. Someone reading this code in six months will not know that omitting this call produces stale reads.

Add a module-level comment explaining the invalidation contract and its limitations.

---

## What I Would Want to See Before Approving

1. **Answers to the deployment topology questions above.** If multi-instance, this implementation is wrong and the right answer is a shared cache (Redis) or no cache at all.
2. **TTL-based eviction.** `cachetools.TTLCache` is one import and solves unbounded growth and staleness simultaneously.
3. **Thread safety.** A lock, or a thread-safe cache structure.
4. **Test isolation.** Cache cleared between tests.
5. **Defensive copy on return.** Callers must not be able to mutate cached state.
6. **Documented invalidation contract.** In code, not just in the PR description.
7. **Handling of `db.query` return type.** What does it return on no results, and is that handled?

The performance motivation is valid — avoiding redundant database queries on hot paths is a legitimate goal. The implementation just needs to be correct under the conditions it will actually run in.
