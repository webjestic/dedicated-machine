**Decision:** Request Changes / Needs Clarification

**Summary:** This PR introduces a module-level mutable cache with no TTL, no size bound, and no thread safety — and does so without documenting any assumptions about the deployment topology it expects to run under. The performance win is real, but the implementation as written is unsafe in the most common production configurations and will silently serve stale data in ways that are extremely difficult to debug after the fact. I cannot approve this until the deployment questions are answered and the implementation is hardened accordingly.

---

## Review

### 🔴 Blocking — Deployment Topology (Must Clarify Before Merge)

The most important question this PR does not answer: **how is this service deployed?**

```python
_preferences_cache = {}  # module-level, process-local
```

This cache lives in a single Python process. If the service runs as more than one instance — behind a load balancer, in a Kubernetes deployment with multiple replicas, under gunicorn with multiple workers — each instance has its own independent copy of this dict. That means:

- A preference update hits one instance, which calls `invalidate_preferences_cache`. The other three instances never hear about it. They continue serving the old value indefinitely.
- There is no cross-process invalidation mechanism. None. The PR does not mention this limitation anywhere.

If this is a single-process, single-instance deployment, this is a known and acceptable trade-off. But that needs to be **stated explicitly**, because the service is described as part of a larger backend platform, which typically implies horizontal scaling. I will not assume single-instance. I am asking.

**Required before merge:** Confirm the deployment model in writing. If multiple instances or workers are possible, this approach needs to be redesigned — either using a shared cache (Redis, Memcached) or accepting that invalidation is best-effort and documenting that explicitly as a product decision.

### 🔴 Blocking — No Cache Invalidation on Write

The PR adds a cache and an invalidation function, but I do not see any call to `invalidate_preferences_cache` in the diff. Where is the write path? If `update_user_preferences` (or equivalent) exists elsewhere in the codebase and does not call invalidation, this cache will serve stale data from the moment a user updates their preferences until the process restarts.

```python
# What does this look like? Where is it?
def update_user_preferences(user_id: str, prefs: dict) -> None:
    db.execute(...)
    invalidate_preferences_cache(user_id)  # Is this happening?
```

If the write path is out of scope for this PR, that is a problem — you should not ship the cache without shipping the invalidation hookup at the same time. Shipping them separately creates a window where the cache is live but writes do not invalidate it.

### 🔴 Blocking — No TTL, Unbounded Memory Growth

```python
_preferences_cache = {}
```

This dict grows forever. In a long-running process serving thousands of users, this is a memory leak. There is no eviction policy, no maximum size, no expiry. Preferences that were cached for users who never return will sit in memory indefinitely.

At minimum, this needs one of:

```python
# Option A: TTL-based expiry (e.g., using cachetools)
from cachetools import TTLCache
_preferences_cache = TTLCache(maxsize=1024, ttl=300)

# Option B: LRU with a size cap
from functools import lru_cache
# (requires refactoring to a function-level decorator)

# Option C: Manual timestamping (more work, less recommended)
_preferences_cache = {}  # {user_id: (value, cached_at)}
CACHE_TTL_SECONDS = 300
```

A TTL also partially mitigates the cross-instance staleness problem — stale data has a bounded lifetime rather than an indefinite one. It does not eliminate the problem, but it makes it survivable.

### 🔴 Blocking — Thread Safety

Python's GIL protects individual bytecode operations, but it does not make compound read-modify-write sequences atomic. Under a threaded server (threaded Flask, threaded Django, any WSGI server with `--threads > 1`):

```python
# Thread A                          # Thread B
if user_id in _preferences_cache:   # concurrent read
    return _preferences_cache[...]  # key deleted between check and read
                                    # → KeyError
```

The `pop` in `invalidate_preferences_cache` and the assignment in `get_user_preferences` are not coordinated. This is a race condition. The fix is straightforward:

```python
import threading
_cache_lock = threading.Lock()

def get_user_preferences(user_id: str) -> dict:
    with _cache_lock:
        if user_id in _preferences_cache:
            return _preferences_cache[user_id]
    # fetch outside lock to avoid holding it during I/O
    result = db.query(...)
    with _cache_lock:
        _preferences_cache[user_id] = result
    return result
```

Note: if you switch to `cachetools.TTLCache`, it is also **not thread-safe by default** and requires the same locking pattern or use of `cachetools.LRUCache` wrapped with `cachetools.cached` and a lock argument.

### 🟡 Significant Concern — Cache Returns Mutable Reference

```python
_preferences_cache[user_id] = result
return _preferences_cache[user_id]  # effectively returns the cached dict directly
```

The caller receives a reference to the same dict object that is stored in the cache. If any caller mutates the returned dict — even accidentally, even in test code — the cached value is silently corrupted for all subsequent callers in the same process.

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # caller thinks they're working on a local copy
                          # they just mutated the cache
```

Return a copy:

```python
return dict(_preferences_cache[user_id])
# or for nested structures:
import copy
return copy.deepcopy(_preferences_cache[user_id])
```

### 🟡 Significant Concern — Test Coverage Gaps

The two tests cover the happy path only. Missing:

| Scenario | Why It Matters |
|---|---|
| Concurrent access | The race condition described above will not be caught without threaded tests |
| Cache miss after TTL expiry | Once TTL is added, this needs a test |
| `invalidate_preferences_cache` on a key that was never cached | Already handled by `pop(..., None)` — but worth an explicit test to document the contract |
| Cache behavior when `db.query` raises an exception | Does a failed query leave a partial entry in the cache? |
| Memory behavior under high user volume | Not a unit test, but worth a note |

The db exception case is worth calling out specifically:

```python
# Current code — if db.query raises, no cache entry is written. Good.
# But this is implicit. A test would make it explicit and prevent regression.
def test_cache_not_populated_on_db_error():
    with mock.patch('services.preferences.db') as mock_db:
        mock_db.query.side_effect = DatabaseError("connection refused")
        with pytest.raises(DatabaseError):
            get_user_preferences("user_123")
        # Cache should be empty — a subsequent call should retry the DB
        mock_db.query.side_effect = None
        mock_db.query.return_value = {"theme": "light"}
        get_user_preferences("user_123")
        assert mock_db.query.call_count == 2
```

### 🟡 Minor — No Documentation of Cache Semantics

The module-level cache has no docstring, no comment explaining its scope, and no documentation of its consistency guarantees (or lack thereof). Anyone reading this code in six months will not know whether the absence of a TTL is intentional or an oversight.

```python
# Minimum acceptable documentation:
# Process-local cache for user preferences.
# WARNING: This cache is not shared across processes or instances.
# Invalidation is best-effort. Do not use in multi-instance deployments
# without a shared backing store.
# Max size and TTL are enforced via TTLCache to bound memory growth.
_preferences_cache: TTLCache = TTLCache(maxsize=1024, ttl=300)
```

### ✅ What Is Good

- The intent is correct. Caching preferences that rarely change is a reasonable optimization.
- `invalidate_preferences_cache` using `pop(..., None)` is the right pattern — no KeyError on a cache miss.
- The test structure (mock db, assert call count) is the right approach for this kind of test.
- The PR description is honest about what it does and why.
