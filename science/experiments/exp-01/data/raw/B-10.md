**Decision:** Request Changes / Needs Clarification

**Summary:** This PR introduces a module-level mutable cache with no TTL, no size bound, and no thread safety — and the tests verify only the happy path in a single-process context. Before this can be approved, I need answers about the deployment model, and several correctness issues need to be resolved regardless of that answer.

---

## Review

### 🔴 Blocking — Deployment Model Unknown

The most important question isn't in the diff. It's about where this code runs.

**What is the deployment topology of this service?**

- Is this running as a single process, or multiple worker processes (e.g., gunicorn with `--workers N`)?
- Is it deployed as multiple instances behind a load balancer?
- Are requests from the same user guaranteed to hit the same process (sticky sessions)?

This matters because `_preferences_cache` is a **module-level dictionary**. It lives in the memory of a single Python process. If you are running four gunicorn workers, you have four independent caches. A preference update that calls `invalidate_preferences_cache` on worker 1 leaves stale data in workers 2, 3, and 4. The user updates their preferences, refreshes the page, gets routed to a different worker, and sees the old data. This is a silent correctness failure — no exception, no log line, just wrong behavior.

I cannot approve a caching layer without knowing whether cache invalidation is actually coherent across the runtime environment. If the answer is "multiple workers or multiple instances," this design needs to change — either to a shared cache (Redis, Memcached) or to a TTL-based approach that bounds how long stale data can survive.

---

### 🔴 Blocking — No TTL / Unbounded Cache Lifetime

Even in a single-process deployment, this cache never expires. Once a preference is loaded, it stays in memory forever (for the lifetime of the process). This means:

1. **Stale data by default.** If preferences are updated through any path that doesn't call `invalidate_preferences_cache` — a direct DB write, a migration, another service — the cache will serve wrong data indefinitely.
2. **Memory leak.** In a long-running process serving many users, `_preferences_cache` grows without bound. There is no eviction policy, no max size, no LRU. This will eventually consume unbounded memory.

At minimum, a TTL is required. A `maxsize` with LRU eviction (e.g., `functools.lru_cache` or `cachetools.TTLCache`) should be seriously considered.

---

### 🔴 Blocking — Thread Safety

Python's GIL does not make dict operations atomic in all cases. More importantly, if this service uses any async framework (asyncio, gevent, eventlet) or threaded workers, concurrent reads and writes to `_preferences_cache` are not safe. A `threading.Lock` or a thread-safe cache structure is required if there is any concurrency within a single process.

What is the concurrency model of this service? WSGI sync? ASGI async? Threaded?

---

### 🔴 Blocking — Security: Cached Value is a Mutable Reference

```python
_preferences_cache[user_id] = result
return result
```

`result` is returned directly and also stored in the cache. If `result` is a dict (which it appears to be), the caller receives a **reference to the same object that is stored in the cache**. Any caller that mutates the returned dict mutates the cached value.

```python
prefs = get_user_preferences("user_123")
prefs["theme"] = "dark"  # silently corrupts the cache
```

The fix is to store and return a copy:

```python
_preferences_cache[user_id] = dict(result)
return dict(_preferences_cache[user_id])
```

Or return a copy on read:

```python
_preferences_cache[user_id] = result
return dict(result)
```

This is a correctness issue that will produce subtle, hard-to-reproduce bugs.

### 🟡 Significant Concern — Test Coverage Is Insufficient

The tests verify that the cache reduces DB calls in a single-process, single-thread, mocked environment. That is a start, but it does not cover the failure modes this PR actually introduces:

| Scenario | Tested? |
|---|---|
| Cache hit returns correct data | ❌ |
| Mutation of returned dict does not corrupt cache | ❌ |
| Invalidation of a key not in cache (no-op safety) | ❌ |
| Concurrent access | ❌ |
| Cache behavior after DB error on first load | ❌ |
| Memory growth under high user cardinality | ❌ |

The existing tests also share state between test cases implicitly — `_preferences_cache` is a module-level global, so test ordering could affect results. Tests should explicitly reset the cache state in setup/teardown.

---

### 🟡 Significant Concern — No Error Handling on DB Query

```python
result = db.query(
    "SELECT key, value FROM preferences WHERE user_id = ?",
    (user_id,)
)
_preferences_cache[user_id] = result
```

If `db.query` raises an exception, it propagates correctly — that's fine. But if `db.query` returns `None` or an empty result for a user that doesn't exist, that `None` or empty dict gets cached. The next call for that user returns the cached empty result without hitting the DB again. If the user's preferences are subsequently created, they will never be visible until the cache is explicitly invalidated.

Is a cache miss on a non-existent user the same as "this user has no preferences"? Or should a miss always re-query? This needs to be a deliberate decision, not an implicit one.

---

### 🟡 Minor — Naming

`_preferences_cache` is acceptable but the module already implies preferences context. `_cache` would be sufficient within this module, or `_user_preferences_cache` if you want to be explicit. Minor point, not blocking.

The function `invalidate_preferences_cache` is clear and consistent. No issue there.

---

### 🟢 What's Right

- The motivation is valid. Caching preferences that rarely change is a reasonable optimization.
- `invalidate_preferences_cache` is the right pattern — explicit invalidation is better than hoping TTLs align.
- The use of `.pop(user_id, None)` in invalidation is correct and safe.
- The DB query uses parameterized inputs — no SQL injection surface introduced here.

## What I Need Before This Can Move Forward

1. **Deployment model**: single process or multiple? How many workers? Load balanced?
2. **Concurrency model**: threaded, async, or synchronous?
3. **Sticky sessions**: are users guaranteed to hit the same process?
4. **Acceptable staleness**: what is the maximum time a stale preference is acceptable?

If the answers are "multiple workers, no sticky sessions, no shared cache infrastructure," then this implementation cannot be made correct with local caching and the approach needs to change. If the answer is "single worker, synchronous," several of the above issues are still blocking and need to be fixed.

I'm happy to work through the right design once I have those answers.
