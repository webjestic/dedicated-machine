**Decision: Needs Clarification**

**Summary:** The tests pass and the caching logic is internally consistent, but this PR introduces a module-level mutable dictionary that is only correct under a single-instance deployment model. Before this can be approved, I need answers to the deployment topology questions below — because if this service runs as multiple instances or processes (which is the norm for any production web service under load), this cache will silently serve stale data and the invalidation function will be functionally broken.

---

## Review

### What the PR does well

The intent is sound. Preference data is a legitimate caching candidate — high read frequency, low write frequency, user-scoped. The invalidation hook exists, which shows awareness that the cache needs to be managed. The tests correctly verify the hit and invalidation behavior in isolation.

---

### Blocking Issues

#### 1. Deployment topology is unspecified — and the answer determines whether this is correct or broken

`_preferences_cache` is a Python module-level dictionary. It lives in the memory of a single process. In any multi-instance or multi-process deployment — which includes virtually every production web service running under gunicorn, uvicorn with multiple workers, or any horizontally scaled container setup — the following is true:

- Each instance has its own independent copy of `_preferences_cache`.
- A write to user preferences triggers `invalidate_preferences_cache` on **one** instance.
- Every other instance continues serving the stale cached value indefinitely.
- There is no TTL, no expiry, no cross-instance coordination.

This is not a theoretical concern. This is the exact class of bug that causes production incidents where a user updates their preferences and sees the old value for minutes or hours depending on which instance handles their subsequent requests.

**I need to know:**
- How many instances of this service run in production?
- How many worker processes per instance (gunicorn workers, uvicorn workers)?
- Is there sticky session routing, or are requests load-balanced across instances?
- Where is `invalidate_preferences_cache` called from? Is it called in this same service, or from a different service entirely?

If the answer is "single process, single instance, forever" — that is an unusual constraint that needs to be documented explicitly in the code, because the next engineer will not know it and will deploy it differently.

#### 2. The invalidation function cannot fulfill its contract at scale

Even if the deployment is currently single-instance, this function is exported as a public API with a docstring that implies it works. If any caller — in this service or another — calls `invalidate_preferences_cache("user_123")` expecting that subsequent reads will reflect updated data, that guarantee only holds within the same process. The function's interface makes a promise the implementation cannot keep in any distributed context. That is a correctness problem that will compound as the system grows.

#### 3. The cache has no upper bound on memory consumption

`_preferences_cache` grows without limit for the lifetime of the process. In a long-running service handling many users, this is a memory leak by another name. There is no eviction policy, no maximum size, no TTL. This needs either a bounded structure (`functools.lru_cache`, `cachetools.TTLCache`) or an explicit documented rationale for why unbounded growth is acceptable.

---

### Security Note

The cache stores raw query results keyed by `user_id`. If `user_id` is ever derived from user-controlled input without validation upstream, there is a potential for cache poisoning — one user's data stored under a key that resolves to another user's ID on a subsequent lookup. I cannot assess this fully without seeing how `user_id` is sourced and validated before reaching this function. If that validation is solid upstream, this is low risk; if it is not, it is a data isolation vulnerability.

---

### What I'd want to see before approving

1. **Answers to the deployment questions above.** If multi-instance: this approach needs to be replaced with a shared cache (Redis, Memcached) or a short TTL-based local cache where stale reads are an acceptable and documented tradeoff.

2. **A TTL or size bound on the cache**, regardless of deployment model.

3. **A note in the docstring or a code comment** that explicitly states the scope of `invalidate_preferences_cache` — that it only affects the local process — so the next engineer does not assume cross-instance invalidation.

4. **Confirmation of where `user_id` originates** and that it is validated before reaching this layer.

---

The tests are not wrong — they test what the code does. The problem is that what the code does is only correct under conditions that have not been stated and may not hold. That is the review working correctly by surfacing this now rather than in a production incident.
