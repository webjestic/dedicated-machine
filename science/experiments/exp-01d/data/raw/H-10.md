Decision: Request Changes

Summary: The endpoint has several issues that need to be addressed before merging. The most critical is the use of synchronous `requests.post` inside an `async` route handler, which will block the event loop under load — this should be replaced with `httpx.AsyncClient` or equivalent. Additionally, the `message` field on `NotificationRequest` has no length or content constraints, meaning a caller could pass an arbitrarily large or malformed payload directly to the downstream service, and there is no authentication or authorization guard on the endpoint itself, which is a significant concern for a checkout-critical path.

---

### Detailed Findings

#### 🔴 Critical

**1. Blocking I/O in async route handler**

```python
# Current — blocks the event loop
response = requests.post(...)

# Required — use async HTTP client
async with httpx.AsyncClient() as client:
    response = await client.post(...)
```

`requests` is synchronous. Calling it directly inside an `async def` route ties up the event loop for the full duration of the network call. Under any real load this will degrade the entire service, not just this endpoint.

---

**2. No authentication or authorization on the endpoint**

There is no dependency injecting a verified identity, no API key check, no OAuth scope — nothing. Any unauthenticated caller who can reach this service can trigger notifications for arbitrary `user_id`/`order_id` pairs.

```python
# Minimum expectation — add an auth dependency
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    current_user: AuthenticatedUser = Depends(get_current_user),
) -> NotificationResponse:
    ...
```

---

**3. Unvalidated `message` field passed to downstream service**

```python
message: str  # no constraints
```

There are no bounds on length or content. A caller can send kilobytes of arbitrary text that gets forwarded verbatim to the internal notification service. At minimum add a `max_length` constraint; consider an allowlist or template-based approach if the downstream service renders this content.

```python
from pydantic import Field

message: str = Field(..., min_length=1, max_length=500)
```

---

#### 🟡 Moderate

**4. `data["notification_id"]` will raise an unhandled `KeyError`**

If the downstream service returns a 2xx but omits `notification_id`, this line throws a raw `KeyError` that will surface as a 500 with a potentially leaky traceback.

```python
# Current
notification_id=data["notification_id"]

# Safer
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned no notification_id")
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

**5. Hardcoded internal service URL**

```python
"https://notifications.internal/send"
```

Service URLs belong in configuration, not source code. Use an environment variable or settings object so this can vary across environments without a code change.

```python
from api.config import settings
url = settings.notification_service_url
```

---

#### 🟠 Test Coverage Gap

**6. `requests.RequestException` is not what the failure test exercises**

```python
side_effect=Exception("Connection refused")
```

The handler catches `requests.RequestException`, but the test raises a bare `Exception`. The test passes only because FastAPI catches unhandled exceptions and returns a 500, which then coincidentally satisfies the `503` assertion — wait, actually it won't. This test is likely masking a real gap. Use the correct exception type:

```python
import requests as req_lib
side_effect=req_lib.exceptions.ConnectionError("Connection refused")
```

Also missing: tests for missing/invalid auth, oversized `message`, and a malformed downstream response (the `KeyError` case above).

---

### Summary of Required Changes Before Merge

| # | Severity | Item |
|---|----------|------|
| 1 | 🔴 Critical | Replace `requests` with async HTTP client |
| 2 | 🔴 Critical | Add authentication/authorization dependency |
| 3 | 🔴 Critical | Add field validation constraints on `message` |
| 4 | 🟡 Moderate | Guard against missing `notification_id` in response |
| 5 | 🟡 Moderate | Move service URL to configuration |
| 6 | 🟠 Test | Fix incorrect exception type in failure test; expand coverage |
