Decision: Request Changes

Summary: The endpoint has several issues that need to be addressed before merging. The most critical concern is the use of synchronous `requests.post` inside an `async` route handler — this will block the event loop under load and is a hard no in an async FastAPI service; replace with `httpx.AsyncClient` and `await`. Additionally, the `message` field on `NotificationRequest` is unbounded and unvalidated, meaning a caller can inject arbitrarily large or malformed payloads that get forwarded directly to the internal service, which is a data integrity and potential abuse risk — apply length constraints and sanitization via Pydantic field validators. Finally, the failure test patches the base `Exception` class rather than `requests.RequestException`, meaning it does not actually validate the intended error-handling path and would pass even if the except clause were removed entirely; tighten that mock to match the production code's exception handling contract.

---

### Detailed Findings

#### 🔴 Critical — Blocking I/O in Async Context

```python
# Current — blocks the event loop
response = requests.post(...)
```

```python
# Required — non-blocking
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        try:
            response = await client.post(
                "https://notifications.internal/send",
                json={...},
                timeout=5.0,
            )
            response.raise_for_status()
            ...
        except httpx.HTTPStatusError as e:
            logger.error("Notification service returned error: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
        except httpx.RequestError as e:
            logger.error("Notification service unreachable: %s", e)
            raise HTTPException(status_code=503, detail="Notification service unavailable")
```

---

#### 🔴 Critical — Unvalidated Free-Text Field Forwarded to Internal Service

The `message` field has no constraints. An unbounded string forwarded to an internal service is an injection and abuse vector.

```python
# Current
message: str

# Required
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

#### 🟡 Medium — Test Does Not Exercise the Actual Error Path

```python
# Current — patches base Exception, not the caught type
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

```python
# Required — matches the production except clause
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.RequestException("Connection refused")):
```

After switching to `httpx`, update this to `httpx.RequestError` accordingly.

---

#### 🟡 Medium — External Service URL is Hardcoded

`"https://notifications.internal/send"` is hardcoded. This makes environment promotion (dev → staging → prod) error-prone and untestable without patching.

```python
# Required
import os

NOTIFICATION_SERVICE_URL = os.environ.get(
    "NOTIFICATION_SERVICE_URL",
    "https://notifications.internal/send"
)
```

---

#### 🟡 Medium — `data["notification_id"]` is Accessed Without Guard

If the external service returns a 200 but omits `notification_id`, this raises an unhandled `KeyError` that will surface as a 500.

```python
# Required
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service response missing notification_id: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

#### 🟢 Minor — Authentication to Internal Service

There is no auth header or mTLS configuration shown for the call to `notifications.internal`. If this is an internal service boundary, even internal calls should be authenticated. Confirm whether this is handled at the infrastructure layer or needs to be explicit here.
