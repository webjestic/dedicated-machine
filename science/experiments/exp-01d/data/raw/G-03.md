Decision: Request Changes

Summary: The endpoint has several meaningful issues that need to be addressed before merge. The most pressing are a blocking I/O call inside an async route, an unauthenticated and open-input endpoint that could be abused, and a hardcoded internal service URL with no configuration layer. The test suite also has a gap that masks a real failure mode.

---

## Detailed Review

### 🔴 Critical Issues

**1. Blocking `requests.post` inside an `async` route**

```python
# Current — blocks the entire event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

`requests` is a synchronous library. Calling it inside an `async` def will block the event loop for the full duration of the network call (up to your 5s timeout), starving every other coroutine. Use `httpx` with its async client instead.

```python
# Recommended
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            settings.notification_service_url,
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

If you must keep `requests` for now, convert the route to a plain `def` so FastAPI runs it in a thread pool — but the async approach is the right fix.

---

**2. No authentication or authorization on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

Any unauthenticated caller can hit this endpoint and trigger notifications to arbitrary `user_id`/`order_id` pairs. There is nothing verifying that the caller owns the `user_id` or that the `order_id` is real. This is a significant abuse vector — notification spam, enumeration, or worse depending on what the downstream service does.

At minimum, require a verified identity via a dependency:

```python
from fastapi import Depends
from api.auth import get_current_user  # your existing auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    current_user: User = Depends(get_current_user),
) -> NotificationResponse:
    # Enforce that current_user.id matches request.user_id
    if current_user.id != request.user_id:
        raise HTTPException(status_code=403, detail="Forbidden")
```

**3. Hardcoded internal service URL**

```python
response = requests.post(
    "https://notifications.internal/send",  # hardcoded
    ...
)
```

This couples the code to a single environment, makes it untestable without network mocking, and means a URL change requires a code change and redeploy. Pull this into configuration:

```python
# config.py
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

settings = Settings()
```

```python
# notifications.py
from api.config import settings

response = await client.post(settings.notification_service_url, ...)
```

### 🟡 Moderate Issues

**4. `message` field is unbounded and unvalidated**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no constraints
```

A caller can send a multi-megabyte string as `message`, which gets forwarded directly to the downstream service. Add length constraints and consider whether `message` should even be caller-supplied at all — for an order confirmation, the message content should probably be generated server-side from the `order_id`, not accepted from the client.

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

---

**5. Swallowing the error detail in the 503 response is fine, but the log is too coarse**

```python
logger.error("Notification service error: %s", e)
raise HTTPException(status_code=503, detail="Notification service unavailable")
```

This is acceptable for not leaking internals to the client, but you should log the `order_id` and `user_id` (or a correlation ID) alongside the error so you can actually debug failures in production without cross-referencing other systems.

```python
logger.error(
    "Notification service error for user=%s order=%s: %s",
    request.user_id,
    request.order_id,
    e,
    exc_info=True,
)
```

---

**6. No retry or circuit-breaker strategy**

A single transient failure immediately returns a 503 to the user during checkout. For a notification service, at least one retry with exponential backoff is reasonable, or offload this to a task queue (Celery, ARQ, etc.) so checkout is not synchronously coupled to notification delivery at all. The latter is the more robust architecture.

### 🟠 Test Issues

**7. The failure test catches the wrong exception type**

```python
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # bare Exception
```

The production handler catches `requests.RequestException`, not bare `Exception`. This test passes today only because `Exception` is the base class, but it does not verify that your specific `except requests.RequestException` clause is doing the right thing. A `ValueError` or any other unrelated exception would also pass this test and would *not* be caught by the handler in production — it would bubble up as an unhandled 500.

```python
import requests as requests_lib

with patch(
    "api.notifications.requests.post",
    side_effect=requests_lib.ConnectionError("Connection refused"),
):
```

**8. No test for the `KeyError` path on `data["notification_id"]`**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # unguarded key access
)
```

If the downstream service returns a 200 but with a malformed body, this raises an unhandled `KeyError` which becomes a 500. Add a test for this case and guard the access:

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Missing notification_id in response for order=%s", request.order_id)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

- `timeout=5` is set — good habit
- `raise_for_status()` is called — correct
- Error is not leaked to the client — correct
- Logger uses `%s` formatting rather than f-strings — correct
