Decision: Request Changes

Summary: This PR has several issues that need to be addressed before it can be merged. The most significant concerns are a blocking synchronous HTTP call inside an async endpoint, missing authentication/authorization on a sensitive endpoint, and insufficient input validation on user-controlled fields. Additionally, the test suite has a gap where it catches a broad `Exception` rather than the specific `requests.RequestException` the handler is designed for, which masks a real bug.

---

## Detailed Review

### 🔴 Critical Issues

**1. Blocking I/O in an Async Endpoint**

```python
# Current — blocks the entire event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

Using the synchronous `requests` library inside an `async` def will block the event loop for the full duration of the HTTP call (up to the 5-second timeout). Under load, this will stall all other requests being handled by the same worker.

```python
# Required fix — use httpx or run in a thread pool
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
        )
```

---

**2. No Authentication or Authorization**

The `/notify` endpoint accepts a `user_id` and `order_id` from any caller with zero verification. There is nothing preventing an unauthenticated actor from:

- Spamming arbitrary users with messages
- Probing valid `user_id`/`order_id` pairs
- Injecting arbitrary `message` content to any user

At minimum, this endpoint must require a valid session token or API key, and the authenticated identity should be validated against the supplied `user_id`.

```python
from fastapi import Depends
from api.auth import get_current_user  # your existing auth dependency

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    current_user: AuthenticatedUser = Depends(get_current_user),
) -> NotificationResponse:
    if current_user.user_id != request.user_id:
        raise HTTPException(status_code=403, detail="Forbidden")
    ...
```

**3. Unvalidated `message` Field**

The `message` field is a free-form `str` with no length cap or content constraints. This field is forwarded directly to an external service and ultimately delivered to an end user, making it a vector for:

- Notification spam / abuse
- Downstream injection if the notification service renders HTML or markdown

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

### 🟡 Significant Issues

**4. Test Catches `Exception`, Not `requests.RequestException` — Masks a Real Bug**

```python
# test_notifications.py
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):  # ← too broad
```

The handler in `notifications.py` only catches `requests.RequestException`. A plain `Exception` would propagate unhandled and return a 500, not a 503. The test as written passes today only because `TestClient` catches unhandled exceptions and converts them — it does not accurately reflect production behavior. This is a test that gives false confidence.

```python
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.RequestException("Connection refused")):
    response = client.post("/notify", json={...})

assert response.status_code == 503
```

---

**5. Hardcoded Internal Service URL**

```python
"https://notifications.internal/send"
```

Service URLs must come from configuration, not be hardcoded in source. This makes the service impossible to point at a staging environment without a code change, and it leaks internal topology into version history.

```python
# config.py
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

settings = Settings()

# notifications.py
response = await client.post(
    f"{settings.notification_service_url}/send",
    ...
)
```

---

**6. `notification_id` Key Access Is Not Guarded**

```python
notification_id=data["notification_id"]
```

If the external service returns a malformed or unexpected payload, this raises an unhandled `KeyError`, which will surface as a 500 with a raw traceback. Validate the response explicitly.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

### 🔵 Minor / Hygiene

**7. Logger Error Leaks Exception Detail to Log Aggregators**

```python
logger.error("Notification service error: %s", e)
```

Depending on what `e` contains (URLs, tokens in query params, internal hostnames), this may leak sensitive infrastructure details into log aggregators. Consider logging a sanitized message and reserving full exception detail for `logger.exception` calls gated behind a debug flag.
