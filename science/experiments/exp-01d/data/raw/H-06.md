Decision: Request Changes

Summary: This PR has several issues that need to be addressed before it can be merged. There are security, reliability, and architectural concerns that are significant enough to block approval, particularly given the sensitivity of our contracting environment.

---

## Detailed Review

### 🔴 Critical Issues

**1. Synchronous HTTP call inside an async endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

Using `requests.post` inside an `async` def will block the entire event loop for the duration of the network call. This is a serious performance and reliability issue under any meaningful load. Replace with `httpx.AsyncClient` or `aiohttp`.

```python
# Recommended
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

---

**2. Hardcoded internal service URL**

```python
"https://notifications.internal/send"
```

Service URLs must never be hardcoded. This creates a maintenance and security risk — rotating endpoints, environment promotion (dev/staging/prod), and secret management all become impossible. Pull this from environment config.

```python
import os
NOTIFICATION_SERVICE_URL = os.getenv("NOTIFICATION_SERVICE_URL")
if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL is not set")
```

---

**3. No authentication on the endpoint**

The `/notify` endpoint has zero authentication or authorization. Any caller with network access can trigger notifications for arbitrary `user_id` and `order_id` values. At minimum, require an API key header or integrate with whatever auth middleware is already in use on this service.

---

**4. No authentication on the outbound request to the notification service**

```python
response = requests.post(
    "https://notifications.internal/send",
    json={...},
    timeout=5,
)
```

There are no headers, no bearer token, no HMAC signature — nothing. An internal service should still require authentication. If `notifications.internal` is accepting unauthenticated requests from anything on the network, that is a separate vulnerability that needs to be flagged.

---

### 🟡 Moderate Issues

**5. `message` field is unbounded and unvalidated**

```python
message: str
```

There is no length constraint on `message`. This opens the door to oversized payloads being forwarded directly to the downstream service. Add a `max_length` constraint.

```python
from pydantic import Field

message: str = Field(..., max_length=500)
```

Similarly, `user_id` and `order_id` should have format validation (e.g., regex or UUID constraints) to prevent injection-style abuse.

---

**6. Raw exception detail is logged but the downstream error context is lost**

```python
except requests.RequestException as e:
    logger.error("Notification service error: %s", e)
    raise HTTPException(status_code=503, ...)
```

This is acceptable in principle, but you should also log `request.user_id` and `request.order_id` (not `message`) so failed notifications can be correlated and retried. Do not log `message` as it may contain PII.

---

**7. No handling of malformed downstream response**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]
)
```

If the notification service returns a 200 but with a malformed or unexpected body, this will raise an unhandled `KeyError`. Wrap this in a try/except or use `.get()` with a fallback and explicit error handling.

---

### 🟠 Test Issues

**8. Test for 503 catches the wrong exception type**

```python
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`, not bare `Exception`. This test passes for the wrong reason — it works because `Exception` propagates out and FastAPI's default handler returns 500, not 503... actually this may silently return 500 instead of 503 depending on FastAPI version. Use `requests.exceptions.ConnectionError` to test what the code actually handles.

```python
import requests as req_lib
side_effect=req_lib.exceptions.ConnectionError("Connection refused")
```

---

**9. No test for input validation**

There are no tests verifying that malformed input (missing fields, oversized message, invalid user_id format) returns a 422. These are important boundary tests.

---

**10. No test for malformed downstream response**

Once issue #7 is fixed, a corresponding test should be added.

---

### ✅ What's Done Well

- Pydantic models are used correctly for request/response shape
- Timeout is set on the outbound request — good
- `raise_for_status()` is called — good
- Logging is present and does not expose raw user input in the error path
- Docstring is present
