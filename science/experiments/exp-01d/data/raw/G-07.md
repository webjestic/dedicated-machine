Decision: Request Changes

Summary: This PR has a reasonable foundation — Pydantic validation, error handling, and test coverage are all present — but there are several meaningful issues spanning security, reliability, and architecture that need to be addressed before this is production-ready. The use of synchronous `requests` in an async endpoint is the most immediately damaging problem, but the security and design gaps are equally concerning.

---

## Review

### 🔴 Critical Issues

**1. Synchronous `requests` inside an `async` endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

This is a classic async footgun. Calling the synchronous `requests` library inside an `async` def will block the entire event loop for the duration of the HTTP call, which under any real load will tank your throughput and can cause cascading timeouts across unrelated endpoints.

```python
# Fix — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)
```

Replace `requests` with `httpx` and use `await`. If you need to keep `requests` for some reason, wrap the call with `asyncio.to_thread()`, but that's a workaround, not a solution.

---

**2. Hardcoded internal service URL**

```python
"https://notifications.internal/send",
```

This is a hardcoded URL sitting directly in business logic. This is a configuration management failure — it makes the service non-portable across environments (dev, staging, prod), untestable without mocking at the network level, and a maintenance hazard.

```python
# Fix — pull from settings
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

settings = Settings()

# Then in the endpoint:
response = await client.post(f"{settings.notification_service_url}/send", ...)
```

---

**3. No authentication on the endpoint**

```python
@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
```

There is zero authentication or authorization on this endpoint. Anyone who can reach this service can trigger notifications to arbitrary users with arbitrary messages. That's a direct vector for spam, phishing, and abuse.

```python
# Fix — at minimum, require a verified caller identity
from fastapi import Depends
from api.dependencies import require_internal_service  # your auth layer

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    _: None = Depends(require_internal_service)
) -> NotificationResponse:
```

The exact mechanism depends on your auth architecture — API key, JWT with scope check, mTLS — but something must be here.

### 🟡 Significant Issues

**4. No validation constraints on `message` field**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # completely unbounded
```

An unbounded `message` field is a problem. A caller can send a 10MB string and you'll happily forward it to your internal service. Add length constraints at minimum.

```python
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=1000)
```

The regex patterns on `user_id` and `order_id` also prevent injection-style values from being forwarded downstream.

---

**5. Leaking upstream error detail into logs**

```python
logger.error("Notification service error: %s", e)
```

Depending on what the `requests.RequestException` contains, you may be logging sensitive internal details — response bodies, internal hostnames, auth headers embedded in URLs. Be deliberate about what you log.

```python
logger.error(
    "Notification service error for order_id=%s: %s",
    request.order_id,
    type(e).__name__,  # log the exception type, not the full repr
)
```

Include structured context (order ID is fine, user ID is debatable depending on your data classification policy) but scrub the raw exception string.

---

**6. Unhandled `KeyError` on `data["notification_id"]`**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # raises KeyError if missing
)
```

If the upstream service returns a 200 but with a malformed or unexpected body, this raises an unhandled `KeyError` which will bubble up as a 500 with a stack trace. Validate the response explicitly.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: keys=%s", list(data.keys()))
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

Note the status code distinction: `503` is "service unavailable", `502` is "bad gateway / upstream returned garbage" — use the right one.

### 🟠 Test Quality Issues

**7. Test patches `Exception`, not `requests.RequestException`**

```python
# In the test
side_effect=Exception("Connection refused")

# But the handler only catches:
except requests.RequestException as e:
```

This test is lying to you. A bare `Exception` will **not** be caught by `except requests.RequestException`, so you're actually testing the FastAPI unhandled exception path, not your error handler. This test would pass even if you deleted your `except` block entirely.

```python
# Fix
import requests as requests_lib

side_effect=requests_lib.RequestException("Connection refused")
```

---

**8. No test for the `notification_id` missing from response**

Given the `KeyError` risk called out above, there should be a test covering a malformed upstream response:

```python
def test_send_notification_malformed_response():
    mock_response = MagicMock()
    mock_response.json.return_value = {}  # missing notification_id
    mock_response.raise_for_status.return_value = None

    with patch("api.notifications.requests.post", return_value=mock_response):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed."
        })

    assert response.status_code == 502
```

---

**9. No timeout configuration and no retry strategy**

```python
timeout=5,
```

A hardcoded 5-second timeout with no retry logic means a single transient blip in the notification service will fail the user's checkout. For a non-critical notification, consider:

- A short timeout (1–2s) with 1–2 retries on connection errors only (not on 4xx)
- Or making the notification fire-and-forget via a background task so it doesn't block checkout completion

```python
from fastapi import BackgroundTasks

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    background_tasks: BackgroundTasks,
) -> NotificationResponse:
    background_tasks.add_task(_dispatch_notification, request)
    return NotificationResponse(success=True, notification_id="pending")
```

Whether this is appropriate depends on your product requirements — if the caller needs a real `notification_id` synchronously, you can't do this. Clarify the contract.
