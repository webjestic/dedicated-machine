Decision: Request Changes

Summary: This PR has several issues that need to be addressed before it can be merged. The most significant concerns are a blocking synchronous HTTP call inside an async endpoint, missing authentication/authorization on a sensitive endpoint, and insufficient input validation that could allow prompt injection or oversized payloads through the `message` field. The test coverage also has a gap where the failure case patches the wrong exception type, meaning the 503 error path isn't actually being tested correctly.

---

## Detailed Review

### 🔴 Critical Issues

**1. Blocking I/O in an Async Endpoint**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, will stall other requests
```

Using `requests` (synchronous) inside an `async` def will block the entire event loop for the duration of the network call. At scale this will cause serious throughput degradation.

```python
# Fix — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://notifications.internal/send",
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
```

---

**2. No Authentication or Authorization**

This endpoint triggers external service calls and is exposed with zero auth. Anyone who can reach the service can spam the notification system or enumerate user/order IDs.

```python
# Fix — require a valid token at minimum
from fastapi import Depends, Security
from fastapi.security import HTTPBearer

security = HTTPBearer()

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(
    request: NotificationRequest,
    token: str = Security(security),  # or your internal auth dependency
) -> NotificationResponse:
```

Internal-facing does not mean unauthenticated, especially in a government contracting context.

---

**3. Unvalidated `message` Field**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # no length limit, no sanitization
```

An unbounded free-text field being forwarded directly to an external service is a risk vector — oversized payloads, injection attempts, and downstream abuse. Constrain it.

```python
from pydantic import BaseModel, Field, constr

class NotificationRequest(BaseModel):
    user_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    order_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    message: str = Field(..., min_length=1, max_length=500)
```

The `user_id` and `order_id` fields should also be constrained to known safe patterns, not arbitrary strings.

### 🟡 Moderate Issues

**4. Hardcoded Internal Service URL**

```python
"https://notifications.internal/send"
```

Service URLs should never be hardcoded. This makes environment promotion (dev → staging → prod) impossible without code changes and creates a maintenance liability.

```python
# Fix — pull from config
import os
from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

---

**5. Raw Exception Data Logged Without Sanitization**

```python
logger.error("Notification service error: %s", e)
```

Exception messages from external services can contain sensitive data — URLs with tokens, internal hostnames, response bodies. Log a controlled message and consider structured logging.

```python
logger.error(
    "Notification service error for order_id=%s: %s",
    request.order_id,
    type(e).__name__,  # log the exception type, not the full message
)
```

---

**6. No Rate Limiting**

This endpoint calls an external service on every hit. Without rate limiting, a single bad actor or runaway client can exhaust your external service quota or trigger downstream costs.

Consider a `slowapi` limiter or an upstream API gateway rule.

### 🟠 Test Issues

**7. Failure Test Patches the Wrong Exception**

```python
# Current — patches with a bare Exception, but the handler only catches RequestException
with patch("api.notifications.requests.post",
           side_effect=Exception("Connection refused")):
```

The production handler catches `requests.RequestException`. A bare `Exception` will bubble up uncaught and the test is passing for the wrong reason — it's hitting FastAPI's default 500 handler, not your 503 branch.

```python
# Fix
import requests as req_lib

with patch("api.notifications.requests.post",
           side_effect=req_lib.RequestException("Connection refused")):
    response = client.post("/notify", json={...})

assert response.status_code == 503
assert response.json()["detail"] == "Notification service unavailable"
```

---

**8. Missing Test Cases**

The following scenarios have no coverage:

- Malformed/missing fields (expect 422)
- `message` exceeding max length once validation is added
- External service returns a 200 but with a missing `notification_id` key — this will currently raise an unhandled `KeyError`
- Unauthenticated request once auth is added (expect 401/403)

### 🔵 Minor Notes

- The `notification_id` extraction has no guard:
  ```python
  notification_id=data["notification_id"]  # KeyError if key is absent
  ```
  Use `data.get("notification_id")` and validate before returning.

- Consider whether `success: bool` in the response model is meaningful — if the endpoint returns 200, success is implied. If it returns non-200, the model is never reached. The field adds noise.
