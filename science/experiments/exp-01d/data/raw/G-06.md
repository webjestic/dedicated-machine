Decision: Request Changes

Summary: The endpoint has several meaningful issues that need to be addressed before merging. The most critical are a blocking synchronous HTTP call inside an async route, an unconstrained `message` field that's a vector for injection/abuse, and a hardcoded internal service URL with no authentication. The test suite also has a gap that masks a real failure mode.

---

## Detailed Review

### 🔴 Critical Issues

**1. Blocking `requests.post` inside an async route**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)
```

Using the synchronous `requests` library inside an `async def` FastAPI route will block the entire event loop for the duration of the HTTP call. Under any real load this will strangle your server's concurrency.

```python
# Fix — use httpx with async support
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            settings.notification_service_url,
            json={...},
            timeout=5.0,
        )
```

---

**2. Hardcoded service URL with no authentication**

```python
# Current — hardcoded, no auth
response = requests.post(
    "https://notifications.internal/send",
    ...
)
```

Hardcoded URLs make environment promotion (dev → staging → prod) impossible without code changes, and there's no token or credential being sent to the downstream service. Any service-to-service call in a real system needs authentication.

```python
# Fix — externalize config, inject auth header
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str
    notification_service_api_key: str

    class Config:
        env_file = ".env"

settings = Settings()

# In the route:
response = await client.post(
    settings.notification_service_url,
    headers={"Authorization": f"Bearer {settings.notification_service_api_key}"},
    json={...},
    timeout=5.0,
)
```

---

**3. Unconstrained `message` field**

```python
# Current — accepts anything, any length
message: str
```

The `message` field is passed directly into an outbound request payload with zero validation. There's no length cap, no character allowlist, nothing. This is a prompt injection risk if the notification service renders content, and a payload abuse vector in general.

```python
# Fix — constrain it
from pydantic import BaseModel, Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    order_id: str = Field(..., pattern=r"^[a-zA-Z0-9_-]{1,64}$")
    message: str = Field(..., min_length=1, max_length=500)
```

Apply the same regex constraints to `user_id` and `order_id` — right now those accept arbitrary strings too.

### 🟡 Significant Issues

**4. Swallowing `KeyError` on `data["notification_id"]`**

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # unguarded
)
```

If the external service returns a 200 but with a malformed or unexpected payload, this raises an unhandled `KeyError` that will bubble up as a 500 with a raw traceback. You need to handle this explicitly.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**5. Logging the raw exception object is a liability**

```python
logger.error("Notification service error: %s", e)
```

Depending on the exception type, `str(e)` can contain internal hostnames, connection strings, or stack details. These shouldn't be written to logs that might be aggregated externally. Log a controlled message and use `exc_info=True` only in environments where that's appropriate.

```python
logger.error("Notification service request failed for order_id=%s", request.order_id, exc_info=True)
```

---

### 🟠 Test Suite Issues

**6. `test_send_notification_service_unavailable` patches the wrong exception**

```python
# Current — patches with a bare Exception
side_effect=Exception("Connection refused")
```

The production code catches `requests.RequestException`, not `Exception`. This test is accidentally passing because `Exception` is a parent class, but it's not testing what you think it's testing. If someone later narrows the except clause, this test will still pass while production breaks.

```python
# Fix — use the specific exception the handler actually catches
import requests as requests_lib

side_effect=requests_lib.exceptions.ConnectionError("Connection refused")
```

**7. No test for the malformed response payload case**

There's no test covering the scenario where the external service returns 200 but without `notification_id` in the body. Once you add the guard in point 4, add a corresponding test.

```python
def test_send_notification_malformed_response():
    mock_response = MagicMock()
    mock_response.json.return_value = {}  # missing notification_id
    mock_response.raise_for_status.return_value = None

    with patch("api.notifications.requests.post", return_value=mock_response):
        response = client.post("/notify", json={
            "user_id": "user_123",
            "order_id": "order_456",
            "message": "Your order has been confirmed."
        })

    assert response.status_code == 502
```

### 🔵 Minor / Architectural Notes

**8. No rate limiting or authentication on the endpoint itself**

This endpoint triggers an outbound notification and is presumably called during checkout. There's nothing stopping an unauthenticated caller from hammering it and generating notification spam or driving up costs on the downstream service. Even a simple dependency for verifying an internal service token would be appropriate here.

**9. No retry logic**

A single attempt with a 5-second timeout and an immediate 503 is brittle. Consider whether a single retry with exponential backoff (e.g., via `tenacity`) is appropriate before surfacing the error to the caller.
