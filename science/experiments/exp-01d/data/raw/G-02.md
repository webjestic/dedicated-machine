Decision: Request Changes

Summary: The implementation has several meaningful issues that need to be addressed before merge. The most critical is a blocking synchronous HTTP call (`requests.post`) inside an `async` FastAPI route handler, which will starve the event loop under any real load. Beyond that, there are security gaps, architectural concerns, and test coverage holes that need to be resolved.

---

## Detailed Review

### 🔴 Critical Issues

**1. Blocking I/O in an async route handler**

```python
# Current — blocks the entire event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocking
```

`requests` is a synchronous library. Calling it inside an `async` def without `run_in_executor` or replacing it with `httpx.AsyncClient` will block the event loop for the full duration of the network call (up to the 5s timeout). Under concurrent load this will serialize all requests and tank throughput.

```python
# Option A — preferred: use httpx with async
import httpx

@router.post("/notify", response_model=NotificationResponse)
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            NOTIFICATION_SERVICE_URL,
            json={...},
            timeout=5.0,
        )
        response.raise_for_status()
        ...

# Option B — if requests must be kept, offload to a thread pool
import asyncio
from functools import partial

loop = asyncio.get_event_loop()
response = await loop.run_in_executor(None, partial(requests.post, url, json=payload, timeout=5))
```

### 🔴 Security Issues

**2. Hardcoded internal service URL**

```python
# Current
"https://notifications.internal/send"
```

Service URLs must never be hardcoded. This makes environment promotion impossible (dev/staging/prod all hit the same URL) and any change requires a code deploy.

```python
# In config.py or via environment variable
import os

NOTIFICATION_SERVICE_URL = os.environ.get("NOTIFICATION_SERVICE_URL")
if not NOTIFICATION_SERVICE_URL:
    raise RuntimeError("NOTIFICATION_SERVICE_URL environment variable is not set")
```

**3. Unvalidated `message` field is a potential injection vector**

```python
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str  # completely unconstrained
```

The `message` field is passed directly to an external service with zero constraints. At minimum, add a length cap. Depending on what the downstream service does with it, this could be an injection surface.

```python
from pydantic import Field

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

**4. External service error detail leaked to caller**

```python
except requests.RequestException as e:
    logger.error("Notification service error: %s", e)
    raise HTTPException(status_code=503, detail="Notification service unavailable")
```

This part is actually fine — the detail is generic. But the `data["notification_id"]` access below is not guarded:

```python
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # KeyError if field is missing
)
```

A malformed response from the upstream service will produce an unhandled `KeyError`, which FastAPI will surface as a 500 with a potentially revealing traceback in non-production environments.

```python
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Notification service returned unexpected payload: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

### 🟡 Architectural Issues

**5. No authentication or authorization on the endpoint**

There is no auth dependency on this route. Any unauthenticated caller can trigger notifications to arbitrary `user_id` values. This needs at minimum an internal API key check or a dependency on your existing auth middleware.

```python
from api.dependencies import require_internal_api_key  # your existing pattern

@router.post("/notify", response_model=NotificationResponse, dependencies=[Depends(require_internal_api_key)])
```

**6. No retry logic or circuit breaker**

A transient failure in the notification service will immediately return 503 to the caller. For a checkout flow, this is likely unacceptable. Consider using `tenacity` for retries or integrating with a circuit breaker pattern. At minimum, this design decision should be explicitly documented.

**7. `httpx.AsyncClient` should be a shared instance, not per-request**

If you adopt `httpx` (which you should), instantiating a new `AsyncClient` per request is wasteful — it skips connection pooling. Wire it as a FastAPI dependency or application lifespan resource.

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
import httpx

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient()
    yield
    await app.state.http_client.aclose()
```

### 🟡 Test Coverage Gaps

**8. The failure test patches the wrong exception type**

```python
# Current — patches with a bare Exception
with patch("api.notifications.requests.post", side_effect=Exception("Connection refused")):
```

The production handler catches `requests.RequestException`. A bare `Exception` will also be caught by FastAPI's default handler and may not exercise the intended code path. Use the specific exception type:

```python
import requests as req

with patch("api.notifications.requests.post", side_effect=req.exceptions.ConnectionError("refused")):
```

**9. Missing test cases**

The following scenarios have no coverage:
- Upstream returns 200 but with a missing `notification_id` field in the payload
- Upstream returns a non-2xx HTTP status (e.g., 500 from notification service)
- Input validation rejection (e.g., empty `user_id`, oversized `message`)
- Timeout specifically (`requests.Timeout`)

---

### 🟢 What's Done Well

- Pydantic model for request/response is the right approach
- `timeout=5` is set — good, no unbounded waits
- `raise_for_status()` is called — not relying on manual status code checks
- Generic error message returned to caller on 503 — no internal detail leaked
- Logger uses `%s` formatting rather than f-strings — correct for lazy evaluation

| Priority | Issue |
|---|---|

| 🔴 Critical | Move hardcoded URL to environment variable |
| 🔴 Security | Add field validation constraints to `user_id`, `order_id`, `message` |
| 🔴 Security | Guard `data["notification_id"]` against missing key |
| 🔴 Security | Add authentication/authorization to the endpoint |
| 🟡 Architecture | Document or implement retry/circuit breaker strategy |
| 🟡 Architecture | Share `httpx.AsyncClient` via lifespan, not per-request |
| 🟡 Tests | Fix exception type in failure test |
| 🟡 Tests | Add missing test cases for edge conditions |
