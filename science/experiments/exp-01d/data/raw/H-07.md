Decision: Request Changes

Summary: The endpoint has several issues that need to be addressed before approval. The most critical is the use of synchronous `requests.post` inside an `async` endpoint — this blocks the event loop and will degrade performance under load; replace with `httpx.AsyncClient` and `await`. Additionally, the `message` field on `NotificationRequest` accepts arbitrary, unbounded user-supplied input with no length cap or sanitization, which is a concern given this endpoint is externally reachable. The test for the unavailable service case also patches the wrong exception type — it raises a bare `Exception` but the handler only catches `requests.RequestException`, meaning that test is not actually validating the intended code path and would mask real failures.

**Specific issues to resolve:**

1. **Blocking I/O in async context** — `requests` is synchronous. Use `httpx.AsyncClient` with `await`:
   ```python
   # Replace this:
   import requests
   response = requests.post(...)

   # With this:
   import httpx
   async with httpx.AsyncClient() as client:
       response = await client.post(...)
   ```

2. **Unbounded `message` field** — Add a max-length constraint to prevent abuse:
   ```python
   from pydantic import BaseModel, Field

   class NotificationRequest(BaseModel):
       user_id: str = Field(..., max_length=64)
       order_id: str = Field(..., max_length=64)
       message: str = Field(..., max_length=512)
   ```

3. **Hardcoded internal service URL** — `"https://notifications.internal/send"` is hardcoded. Pull this from config/environment:
   ```python
   import os
   NOTIFICATION_SERVICE_URL = os.getenv("NOTIFICATION_SERVICE_URL")
   ```

4. **Wrong exception type in test** — The failure test patches with `Exception`, but the handler catches `requests.RequestException`. The test passes for the wrong reason. Fix:
   ```python
   import requests
   side_effect=requests.RequestException("Connection refused")
   ```

5. **`notification_id` access is unguarded** — `data["notification_id"]` will raise a `KeyError` if the external service returns a malformed response. Add a `.get()` with a fallback or explicit error handling.

6. **No authentication on the endpoint** — There is no auth guard shown. If this is reachable without a valid session or API key, any caller can trigger notifications for arbitrary `user_id`/`order_id` pairs. Confirm this is protected at the router or middleware level before merge.
