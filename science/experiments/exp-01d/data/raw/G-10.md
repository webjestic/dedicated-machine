Decision: Request Changes

Summary: The implementation has several meaningful issues that need to be addressed before merge. The most critical is a **blocking synchronous HTTP call** (`requests.post`) inside an `async` FastAPI route handler — this will block the event loop under load and is a fundamental async correctness issue. Beyond that, there are security gaps, missing input validation, and a test suite that doesn't actually cover what it claims to.

---

## Detailed Findings

### 🔴 Critical

**1. Blocking I/O in an async route handler**

```python
# Current — blocks the event loop
async def send_notification(request: NotificationRequest) -> NotificationResponse:
    response = requests.post(...)  # synchronous, blocks entire event loop
```

`requests` is a synchronous library. Calling it directly inside an `async` def will block every other request being handled by the server while waiting on the network. This is a correctness issue, not just a style preference.

```python
# Option A — use httpx with async support (preferred)
import httpx

async def send_notification(request: NotificationRequest) -> NotificationResponse:
    async with httpx.AsyncClient() as client:
        response = await client.post(...)

# Option B — if requests must be kept, offload to a thread pool
from fastapi.concurrency import run_in_threadpool

response = await run_in_threadpool(requests.post, ...)
```

---

**2. Hardcoded internal service URL**

```python
"https://notifications.internal/send"  # hardcoded
```

Service URLs must not be hardcoded. This makes the service impossible to configure across environments (dev, staging, prod) without a code change, and it's a deployment/security smell.

```python
# Correct approach — pull from config/environment
import os
from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    notification_service_url: str

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

### 🟠 High

**3. No input validation beyond type checking**

Pydantic is used, but the fields have zero constraints. `message` in particular is a free-form string being forwarded to an external service — that's a vector for injection or abuse.

```python
# Current — no constraints at all
class NotificationRequest(BaseModel):
    user_id: str
    order_id: str
    message: str
```

```python
# Better — enforce reasonable bounds
from pydantic import BaseModel, Field
import re

class NotificationRequest(BaseModel):
    user_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    order_id: str = Field(..., min_length=1, max_length=64, pattern=r"^[a-zA-Z0-9_-]+$")
    message: str = Field(..., min_length=1, max_length=500)
```

---

**4. `data["notification_id"]` is accessed without any guard**

If the external service returns a malformed or unexpected response, this will raise an unhandled `KeyError` that bubbles up as a 500 with a potentially leaky traceback.

```python
# Current — will raise KeyError on unexpected response shape
data = response.json()
return NotificationResponse(
    success=True,
    notification_id=data["notification_id"]  # unguarded
)
```

```python
# Better
data = response.json()
notification_id = data.get("notification_id")
if not notification_id:
    logger.error("Unexpected response from notification service: %s", data)
    raise HTTPException(status_code=502, detail="Invalid response from notification service")
```

---

**5. Error logging may leak sensitive data**

```python
logger.error("Notification service error: %s", e)
```

Depending on what `requests.RequestException` surfaces, this could log internal URLs, auth headers, or request payloads to wherever your log sink goes. Be deliberate about what you log.

```python
logger.error(
    "Notification service error for order_id=%s: %s",
    request.order_id,
    type(e).__name__  # log the exception type, not the full exception
)
```

---

### 🟡 Medium

**6. No authentication or authorization on the endpoint**

```python
@router.post("/notify")
async def send_notification(request: NotificationRequest):
    ...
```

Any caller can trigger notifications for any `user_id`/`order_id` combination. There's no verification that the caller is authorized to send notifications on behalf of that user. At minimum, this endpoint should require a service-level auth token (e.g., `Depends(verify_internal_token)`), and ideally the `user_id` should be derived from the authenticated session rather than accepted as raw input.

---

**7. No retry logic or circuit breaker**

A single `timeout=5` is the only resilience mechanism. For a checkout-critical path, a transient failure in the notification service will immediately surface as a 503 to the user. Consider whether this should be fire-and-forget (async task queue), retried with backoff, or whether a notification failure should actually block checkout completion.

---

### 🔵 Low / Test Quality

**8. The failure test patches the wrong exception type**

```python
# Current — patches with a generic Exception
with patch("api.notifications.requests.post", side_effect=Exception("Connection refused")):
    ...
```

The handler only catches `requests.RequestException`. A bare `Exception` will **not** be caught by that handler and will propagate as an unhandled 500, not a 503. The test passes for the wrong reason — it's testing the wrong code path.

```python
# Correct
import requests as req_lib

with patch(
    "api.notifications.requests.post",
    side_effect=req_lib.RequestException("Connection refused")
):
    ...
```

**9. Missing test cases**

- Malformed response from external service (missing `notification_id`)
- Validation errors (empty strings, oversized message, invalid characters)
- HTTP 4xx/5xx from the external service (`raise_for_status` path)
- No test for unauthorized access once auth is added

## Summary of Required Changes Before Merge

| Priority | Issue |
|---|---|
| 🔴 Critical | Replace `requests` with `httpx` async or offload to threadpool |
| 🔴 Critical | Move service URL to environment/config |
| 🟠 High | Add field-level validation constraints to `NotificationRequest` |
| 🟠 High | Guard `data["notification_id"]` access |
| 🟠 High | Scope what gets logged in the error handler |
| 🟡 Medium | Add endpoint authentication |
| 🟡 Medium | Define failure behavior for checkout path |
| 🔵 Low | Fix exception type in failure test |
| 🔵 Low | Expand test coverage |
